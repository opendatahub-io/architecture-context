## Purpose

**Short**: Out-of-tree Llama Stack evaluation provider that implements Ragas (Retrieval Augmented Generation Assessment) evaluation capabilities, supporting both inline (in-process) and remote (Kubeflow Pipelines) execution modes.

**Detailed**: This component implements the [Ragas](https://github.com/explodinggradients/ragas) evaluation framework as an external provider for [Meta's Llama Stack](https://github.com/meta-llama/llama-stack) platform. It plugs into Llama Stack's `Eval` API to provide RAG quality assessment metrics — including answer relevancy, context precision, faithfulness, and context recall — against datasets registered through Llama Stack's DatasetIO API.

The provider ships in two execution modes. The **inline** provider runs Ragas evaluation in the same process as the Llama Stack server, wrapping Llama Stack's Inference API as LangChain-compatible LLM and embeddings adapters for Ragas consumption. The **remote** provider offloads evaluation to Kubeflow Pipelines (Data Science Pipelines), constructing a two-step KFP pipeline that retrieves datasets from Llama Stack and runs Ragas evaluation in a separate container pod, storing results to S3. This remote mode enables scalable, asynchronous evaluation on OpenShift clusters.

The component is part of the TrustyAI ecosystem within RHOAI, providing model evaluation capabilities for the AI/ML platform. It is built as a Python package (`llama-stack-provider-ragas`) and also shipped as a container image (`odh-trustyai-ragas-lls-provider-dsp-rhel9`) for use as the base image in Kubeflow Pipeline component execution. Note: as of the latest commit on this branch (`f401c54`), the component has been offboarded from RHOAI 3.2.

## Architectural Analysis

The llama-stack-provider-ragas component follows the Llama Stack out-of-tree provider pattern, implementing the `Eval` and `BenchmarksProtocolPrivate` interfaces. This design allows it to be loaded dynamically by any Llama Stack distribution server without modifying the core Llama Stack codebase. The provider declares API dependencies on `Inference`, `DatasetIO`, `Files`, and `Benchmarks`, which Llama Stack's dependency injection system resolves and passes at initialization. This clean separation of concerns means the provider doesn't need to manage model serving or data storage — it focuses purely on evaluation orchestration.

The dual inline/remote architecture addresses two distinct deployment scenarios. The inline provider wraps Llama Stack's Inference API behind LangChain-compatible `BaseRagasLLM` and `BaseRagasEmbeddings` adapters, enabling Ragas to make LLM calls through async/sync bridging patterns (notably using `asyncio.to_thread()` to run Ragas' synchronous `evaluate()` in a background thread, avoiding event loop blocking). The remote provider takes a fundamentally different approach: it constructs a KFP pipeline with two components (`retrieve_data_from_llama_stack` and `run_ragas_evaluation`), submits it to a Data Science Pipelines server, and later polls for completion and fetches results from S3. The remote wrappers use `llama_stack_client` (the HTTP SDK) rather than the in-process Inference API, since KFP pods run in separate containers.

A notable design pattern is the base image resolution chain for KFP components. The `get_base_image()` function in `components.py` follows a three-tier fallback: first checking the `KUBEFLOW_BASE_IMAGE` environment variable, then reading from a `trustyai-service-operator-config` ConfigMap in candidate namespaces (`redhat-ods-applications`, `opendatahub`), and finally falling back to a hardcoded default image. This integration with the TrustyAI operator's ConfigMap allows the platform operator to control which container image KFP pods use without requiring changes to the provider code, though it creates a runtime dependency on the Kubernetes API at import time (the `@dsl.component` decorator evaluates `get_base_image()` at module load).

The compatibility layer (`compat.py`) handles the upstream Llama Stack package restructuring from `llama_stack` to `llama_stack_api`, using try/except import fallbacks across all API types. This allows the provider to work with both old and new Llama Stack versions simultaneously, though it adds import complexity. Job management is entirely in-memory (both inline and remote providers use Python dicts to track evaluation jobs), meaning job state is lost on server restart. The remote provider partially mitigates this by storing results in S3, but the job ID mapping itself is ephemeral.

As of the latest commit on this branch (`f401c54 offboard ragas from rhoai-3.2`), this component has been offboarded from the RHOAI 3.2 release. The offboarding suggests the component may be reintroduced in a future RHOAI version or maintained only in the upstream TrustyAI project.
