## Purpose

**Short**: TrustyAI is a Java-based Quarkus service providing AI fairness metrics, model drift detection, and explainability algorithms for ML models served via ModelMesh and KServe on OpenShift AI.

**Detailed**: TrustyAI Explainability is a multi-module Java library and REST service that provides Explainable AI (XAI) capabilities for the Red Hat OpenShift AI platform. The core library (`explainability-core`) implements fairness metrics (Statistical Parity Difference, Disparate Impact Ratio), drift detection algorithms (Meanshift, FourierMMD, KS Test), and local/global explainability methods (LIME, SHAP, Counterfactual, TSSaliency, Partial Dependence Plots).

The service module (`explainability-service`) wraps the core library into a Quarkus-based REST API that integrates with ML inference platforms. It consumes inference payloads from ModelMesh (via direct HTTP with protobuf-encoded data) and KServe (via Knative CloudEvents), reconciles input/output pairs, stores them in configurable backends (PVC, MinIO, MariaDB/MySQL database, or in-memory), and exposes metric computation and explainability endpoints. Computed fairness and drift metrics are published to Prometheus via Micrometer for monitoring and alerting.

The connectors module (`explainability-connectors`) provides gRPC and HTTP clients implementing the KServe V2 Inference Protocol, enabling TrustyAI to call back to model servers for explainability computations. An Arrow module (`explainability-arrow`) facilitates data interchange between the Java service and the TrustyAI Python library via Apache Arrow IPC format.

## Architectural Analysis

### Multi-Module Library Architecture

TrustyAI follows a clean multi-module Maven architecture where `explainability-core` is a pure Java library with no framework dependencies, making it reusable outside the Quarkus service context. The core module contains the actual ML fairness, drift, and explainability algorithms with OptaPlanner as the only significant dependency (for Counterfactual constraint solving). The `explainability-connectors` module adds KServe V1/V2 protocol support via gRPC and HTTP, with generated protobuf code from the `grpc_predict_v2.proto` definition. The `explainability-arrow` module provides Apache Arrow IPC serialization for efficient data interchange with the TrustyAI Python library, enabling hybrid Java-Python workflows.

### Dual Inference Platform Support

A key architectural decision is the dual support for both ModelMesh and KServe inference platforms, with fundamentally different payload ingestion patterns. ModelMesh sends base64-encoded protobuf payloads via direct HTTP POST to `/consumer/kserve/v2`, while KServe uses Knative CloudEvents with JSON-encoded payloads. Both paths converge through `InferencePayloadReconciler` subclasses that buffer partial payloads (request/response arrive separately), match them by request ID, convert to internal `Dataframe` format, and persist to the configured storage backend. The `ModelMeshInferencePayloadReconciler` additionally handles synthetic data detection via the `BIAS_IGNORE_PARAM` parameter and strips the `__isvc` suffix from model IDs.

### Storage Backend Flexibility

The service supports four storage backends (PVC flat-file, MinIO S3, MariaDB/MySQL database, in-memory) behind a clean abstraction layer. The `StorageInterface` generic interface allows the `DataSource` implementation to be swapped via configuration (`service.storage-format`). Notably, the database backend includes an automated migration path from PVC storage, implemented as an event-driven async process in `HibernateMigrator` that batches data transfer with 300-second transaction timeouts. This migration capability reflects the platform's evolution from PVC-based storage to database-backed persistence for better scalability.

### Profile-Based Feature Gating

The RHOAI distribution disables all explainability endpoints (LIME, SHAP, Counterfactual, TSSaliency, PDP) via the `@EndpointDisabled` annotation mechanism, leaving only fairness metrics (SPD, DIR) and drift detection active. This is a deliberate product decision — explainability algorithms require callback calls to model servers (potentially expensive and latency-sensitive), while fairness and drift metrics operate purely on stored inference data. The ODH profile enables all features. This pattern uses Quarkus profile-based configuration (`%rhoai.*` properties) rather than build-time exclusion, meaning the code is present but endpoints return 404 when disabled.

### Init Container for ModelMesh Integration

The base deployment manifest includes an init container that uses `oc` CLI to create/update a `model-serving-config` ConfigMap. This ConfigMap registers TrustyAI's `/consumer/kserve/v2` endpoint as a `payloadProcessor` for ModelMesh, enabling automatic inference payload forwarding. This is a namespace-aware self-registration pattern — the init container reads the current namespace from the service account token and constructs the internal service URL dynamically.

### Metrics Publication Architecture

Fairness and drift metrics are computed on a configurable schedule (default 5s) by `PrometheusScheduler`, which maintains a concurrent registry of scheduled metric requests. The scheduler optimizes batch reads — it determines the maximum batch size needed across all pending requests to minimize storage I/O. Computed values are published as Prometheus gauges with the `trustyai_` prefix via `PrometheusPublisher`, which dynamically creates gauges with model-specific tags. The `ServiceMonitor` CR configures Prometheus to scrape these metrics, with metric relabeling to keep only `trustyai_*` series.
