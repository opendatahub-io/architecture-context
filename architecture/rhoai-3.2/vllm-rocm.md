## Purpose

**Short**: GPU-accelerated vLLM inference runtime container image built for AMD ROCm, serving LLM inference via OpenAI-compatible HTTP and TGIS-compatible gRPC APIs.

**Detailed**: The vllm-rocm repository is a thin container image packaging layer that customizes the upstream Red Hat AI Infrastructure Services (RHAIIS) vLLM ROCm base image (`registry.redhat.io/rhaiis/vllm-rocm-rhel9`) for deployment within the RHOAI platform. It contains no application source code — all vLLM runtime code, the vllm-tgis-adapter, ROCm GPU libraries, and Python dependencies come from the RHAIIS base image.

The repository's sole purpose is to configure the container entrypoint, set environment variables (gRPC port, HTTP port, speculative decoding behavior), apply Red Hat container labels, and set the non-root user context. The resulting image provides high-performance LLM inference using AMD ROCm GPUs with PagedAttention memory management. It is deployed as a serving runtime within KServe InferenceService CRs on RHOAI, specifically targeting AMD Instinct GPU hardware (MI250X, MI300X, etc.) as an alternative to the NVIDIA CUDA-based vLLM image.

The entrypoint runs the `vllm_tgis_adapter` module, which wraps the core vLLM engine and exposes both an OpenAI-compatible REST API (port 8000) and a TGIS-compatible gRPC API (port 8033), enabling compatibility with both the KServe v2 inference protocol and Red Hat's TGIS protocol used by downstream consumers.

## Architectural Analysis

The vllm-rocm repository is an extremely thin packaging layer — it consists of a single Dockerfile that sets four environment variables, a non-root user, an entrypoint, and container labels on top of the RHAIIS base image. This architecture reflects the AIPCC (AI Platform Common Components) integration pattern introduced in RHOAI 3.2, where inference runtime images are sourced from the RHAIIS product stream rather than built from upstream vLLM source. The commit history confirms this transition: commit `7c2119552` ("Integrate AIPCC to consume vLLM ROCm from RHAIIS releases") replaced the previous multi-stage build with a single `FROM` line pulling the pre-built RHAIIS image.

The entry point `python3 -m vllm_tgis_adapter --uvicorn-log-level=warning` runs the TGIS adapter, which is a wrapper around vLLM's AsyncLLMEngine that exposes both OpenAI-compatible REST endpoints (port 8000) and TGIS-compatible gRPC endpoints (port 8033). This dual-protocol exposure is architecturally significant: the OpenAI API provides broad compatibility with client libraries and tools, while the TGIS gRPC API maintains backward compatibility with Red Hat's existing inference pipeline that predates the adoption of vLLM. The `DISABLE_LOGPROBS_DURING_SPEC_DECODING=false` environment variable overrides vLLM's default optimization to ensure log probabilities remain available even when speculative decoding is active — important for hosted model serving where clients may depend on logprob output.

The non-hermetic build (`hermetic: false` in the Tekton PipelineRun) is appropriate here because the Dockerfile performs no package installation. However, this means all supply chain security guarantees — RPM provenance, Python package pinning, ROCm library versions — are entirely deferred to the RHAIIS base image build. Any vulnerability or FIPS compliance gap in the base image propagates directly to this component. The Tekton pipeline also references `Dockerfile.rocm.ubi` (line 37) for the PR pipeline rather than `Dockerfile.konflux.rocm`, suggesting there may be a separate Dockerfile for local/PR builds that is not present on this branch (possibly on upstream/main).

The container runs as UID 2000 (non-root), which aligns with OpenShift security requirements and RHOAI's SecurityContextConstraints. GPU access is provided through device plugins at the Kubernetes level rather than requiring root privileges.
