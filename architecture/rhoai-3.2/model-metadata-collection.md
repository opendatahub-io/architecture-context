## Purpose

**Short**: CLI toolchain and data container that extracts, enriches, and catalogs AI model metadata from OCI container registries and HuggingFace, producing YAML catalog files shipped as a static data image for the RHOAI dashboard.

**Detailed**: The model-metadata-collection component serves as the metadata pipeline for Red Hat AI validated models. It contains two CLI tools (`model-extractor` and `metadata-report`) and a data container image that packages pre-generated model catalog YAML files. The `model-extractor` tool connects to OCI container registries (primarily `registry.redhat.io`) to pull model container images (modelcars), extract model cards from annotated container layers, parse metadata from YAML frontmatter and markdown content, and enrich the extracted data with information from the HuggingFace API. The resulting `models-catalog.yaml` and `validated-models-catalog.yaml` files provide structured metadata (name, provider, description, license, tasks, languages, OCI artifacts) consumed by the RHOAI dashboard to display model information to users.

The Konflux-built container image (`Dockerfile.konflux`) does not run any Go binary at runtime. Instead, it packages the pre-generated `data/` directory contents into a `ubi9-minimal` container image that exposes catalog data via volume mounts at `/app/data` and benchmark data at `/app/benchmarks`. The container runs `sleep infinity` as its command, acting as a data sidecar that other pods mount to access catalog files. A separate GitHub Actions workflow builds the upstream (ODH) image using the standard `Dockerfile` with a `ubi9-micro` base. The CI/CD pipeline (Tekton via Konflux) triggers on pushes to `rhoai-3.2` and rebuilds the data image, ensuring catalog updates are automatically shipped.

## Architectural Analysis

The model-metadata-collection component has an unusual architecture for a platform component: it is a **data pipeline masquerading as a container image**. The Go source code in the repository is tooling that runs during CI/CD to produce YAML artifacts, but the shipped container image contains zero compiled binaries — only static YAML files and a `sleep infinity` command. This means the component's security surface at runtime is minimal (a sleeping UBI container), while the build-time surface involves network calls to external APIs (HuggingFace, OCI registries).

The metadata extraction pipeline implements a sophisticated multi-source enrichment strategy with explicit data provenance tracking. Each metadata field (name, provider, description, license, tasks, language, etc.) is tagged with its source: `modelcard.yaml` (YAML frontmatter in container model cards), `modelcard.regex` (regex extraction from markdown), `huggingface.yaml` (YAML frontmatter from HuggingFace READMEs), `huggingface.api` (HuggingFace API response), `huggingface.tags` (parsed from repository tags), `huggingface.regex` (regex extraction from HuggingFace README text), or `generated` (synthesized from model names). There is a clear priority hierarchy: HuggingFace YAML frontmatter > HuggingFace API > modelcard YAML > modelcard regex > generated fallback. This provenance tracking is written to separate `enrichment.yaml` files alongside each model's `metadata.yaml`, enabling the `metadata-report` tool to audit data quality.

The OCI layer scanning uses the `containers/image/v5` library to pull manifests without downloading full images. It specifically looks for layers annotated with `io.opendatahub.modelcar.layer.type=modelcard`, extracts tar/gzip archives, and parses the contained markdown files. When no model card is found in the container, the tool falls back to fetching the HuggingFace README for the best-matching model (using a token-based similarity algorithm with a 0.5 threshold). Cross-family matching is prevented by `extractModelFamily()` which identifies known model families (llama, granite, mistral, etc.) to avoid incorrect enrichment.

The model deduplication logic in `catalog.go` consolidates models with the same name (case-insensitive) by merging their OCI artifacts, taking the earliest creation timestamp and latest update timestamp, and merging metadata fields with priority to the first occurrence. This handles cases where the same model appears in multiple processing runs or with multiple quantization variants.

A notable design decision is the split between two Dockerfiles: `Dockerfile.konflux` (production, `ubi9-minimal` base, data-only) and `Dockerfile` (upstream/ODH, `ubi9-micro` base, includes `sample-data/` benchmarks). The Konflux file was migrated from `ubi9-micro` to `ubi9-minimal` specifically for FIPS compliance (commit 4e4b922), as `ubi9-micro` lacks the OpenSSL libraries required by check-payload validation.
