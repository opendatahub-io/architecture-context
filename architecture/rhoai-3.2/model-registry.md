## Purpose

**Short**: A central repository for recording, managing, and retrieving metadata associated with ML models, versions, artifacts, and serving deployments.

**Detailed**: Model Registry is a multi-component service that provides a REST API for managing ML model metadata throughout the model lifecycle. It stores metadata about registered models, model versions, model artifacts, serving environments, inference services, experiments, and experiment runs. The primary service (`model-registry proxy`) exposes an OpenAPI v1alpha3 REST interface backed by an embedded metadata store (EmbedMD) that supports both MySQL and PostgreSQL as database backends.

The repository also contains a Kubernetes controller that synchronizes KServe InferenceService CRDs with Model Registry entities, automatically tracking deployed models and their lifecycle states. A CSI storage initializer enables KServe to download model artifacts directly using `model-registry://` URIs. A model catalog service provides a separate REST API (v1alpha1) for browsing curated model catalogs sourced from HuggingFace and other providers. An async upload job image provides a Kubernetes Job for copying models between storage backends and registering them in the registry.

Supporting clients include a Python SDK (`model-registry` package) and a web UI with a Go BFF (backend-for-frontend). The platform operator (rhods-operator) deploys and manages Model Registry instances, while the components in this repository define the application logic and deployment manifests consumed by the operator.

## Architectural Analysis

The Model Registry follows a modular, multi-component architecture within a single repository. The primary service is a REST API proxy built with `go-chi` and OpenAPI code generation, providing a strongly-typed interface for ML metadata CRUD operations. The proxy uses a "dynamic router" pattern where the HTTP handler is swapped atomically once the database connection is established, allowing the server to start accepting health probes immediately while the database connects asynchronously in a background goroutine. This design prevents pod restart loops when database startup is slow.

The database layer uses GORM with an "EmbedMD" abstraction that supports both MySQL and PostgreSQL through separate connect/migrate packages. Database schema migrations are handled automatically at startup via `golang-migrate/migrate`, and the health check system distinguishes between startup readiness (`/readyz/isDirty` — checks schema migration state) and general health (`/readyz/health` — includes full service liveness). The TLS configuration for database connections is thorough, supporting custom CA bundles, client certificates, configurable cipher suites with explicit insecure cipher rejection, and a minimum TLS version of 1.2.

The Kubernetes controller is conditionally enabled via the `INFERENCE_SERVICE_CONTROLLER=managed` environment variable and uses a finalizer pattern for graceful deletion. When a KServe InferenceService is deleted, the controller first sets the Model Registry entity's desired state to `UNDEPLOYED` before allowing the Kubernetes resource to be removed. The controller discovers Model Registry service endpoints dynamically by listing Kubernetes Services with the `component: model-registry` label or by reading explicit URL annotations. This service discovery pattern allows multiple Model Registry instances to coexist in different namespaces, with each InferenceService selecting its registry via labels.

The CSI storage initializer implements the KServe `Provider` interface, acting as a URI resolver rather than a full storage provider. It parses `model-registry://` URIs to extract model and version names, queries the Model Registry API to find the associated model artifact's actual storage URI (S3, GCS, etc.), then delegates the download to the appropriate KServe built-in storage provider. This indirection layer decouples model serving from storage locations — models can be moved between storage backends without updating InferenceService manifests.

A notable security concern is that the Istio AuthorizationPolicy for the model-registry-service uses empty rules (`rules: - {}`) which effectively allows all traffic. In production RHOAI deployments, the platform operator (rhods-operator) overrides this with more restrictive policies. The base manifests in this repository are oriented toward Kubeflow/ODH development environments where broader access is acceptable.
