## Platform Overview

Red Hat OpenShift AI (RHOAI) 3.2 is an enterprise AI/ML platform deployed on OpenShift Container Platform via the Operator Lifecycle Manager (OLM). The platform operator (`rhods-operator`) orchestrates the lifecycle of 40 components spanning model serving, pipeline orchestration, distributed training, model registry, interactive notebooks, AI safety guardrails, and observability. The operator uses a registry-based plugin architecture where each component self-registers via Go `init()` functions, enabling modular enablement through a `DataScienceCluster` custom resource. Components are deployed primarily into the `redhat-ods-applications` namespace for platform services and user-selected namespaces for workloads.

RHOAI 3.2 introduces a three-layer Gateway API ingress architecture replacing the previous direct OpenShift Route model. External traffic flows through an Istio-based Envoy proxy (managed via a GatewayClass and Gateway resource), then through `kube-auth-proxy` for centralized authentication (supporting both OpenShift OAuth and external OIDC providers), and finally to backend services. Components register themselves as HTTPRoutes referencing the platform's shared `data-science-gateway`. This architecture provides unified TLS termination, authentication, and traffic management for all platform services — a significant evolution from the per-component Route approach in RHOAI 2.x.

The security posture is defense-in-depth: FIPS-compliant Go binaries compiled with `GOEXPERIMENT=strictfipsruntime` and CGO dynamic linking against UBI 9 OpenSSL; kube-rbac-proxy sidecars performing Kubernetes SubjectAccessReview authorization; NetworkPolicies isolating inter-component traffic; OpenShift service-ca signed TLS certificates for pod-to-pod encryption; and Kuadrant AuthPolicies enforcing per-service authorization in the Gateway API model. Container images are built on UBI 9 base images and shipped through Konflux with hermetic builds and digest pinning.

### Central Components

**rhods-operator** is the root orchestrator with the most reverse dependencies — every platform component depends on it for deployment and lifecycle management. It manages 184 controller watches and 21 CRDs, deploying all other operators and services.

**KServe** is the second most connected component, serving as the universal model serving layer. It is referenced by odh-model-controller (ingress and auth extensions), TrustyAI (inference logging and payload processing), Model Registry (deployment tracking via CRD watch), guardrails detectors (InferenceService deployment), vLLM variants (serving runtime), OpenVINO (serving runtime), and caikit-nlp (model serving). KServe manages 11 CRDs and 41 controller watches.

**odh-dashboard** is the primary user-facing entry point, integrating with nearly every platform capability: notebooks, model serving (KServe), pipelines (DSP), model registry, TrustyAI metrics, LlamaStack, feature stores, and hardware profiles. Its Module Federation architecture with Go BFF sidecars enables independent feature-team velocity.

**kube-auth-proxy** is the platform's shared authentication layer, deployed as a sidecar by rhods-operator into the Gateway Envoy proxy. It enforces authentication for all platform services accessed through the Gateway API ingress.

**Kueue** serves as the cross-cutting workload admission controller, integrating with KubeRay (Ray workloads), Kubeflow Training Operator (training jobs), Kubeflow Trainer (v2 training), and TrustyAI (LMEval job scheduling).

## Deployment Architecture

### Deployment Topology

The platform follows a three-tier namespace model. The **operator tier** runs in `redhat-ods-operator` (OLM-managed) and contains the rhods-operator pod. The **platform services tier** in `redhat-ods-applications` hosts all operator controllers, the dashboard, and shared infrastructure (Gateway controller, monitoring). The **ingress tier** in `openshift-ingress` runs the Gateway Envoy proxy and kube-auth-proxy for centralized authentication. **User namespaces** contain workload-specific resources: notebook pods, InferenceService pods, pipeline runs, model registry instances, and TrustyAI service instances. A **monitoring tier** in `redhat-ods-monitoring` runs the observability stack (Prometheus via MonitoringStack CRD, Tempo for tracing, OpenTelemetry Collector, and Perses dashboards).

#### Manifest Consumption Pipeline

Component manifests flow from individual component repositories (where they are maintained alongside source code) through the `get_all_manifests.sh` script into the rhods-operator repository's `prefetched-manifests/` directory. At operator build time, these manifests are embedded into the operator binary or referenced from the container filesystem. At reconcile time, the operator applies kustomize overlays (selecting ODH or RHOAI variants), renders Go templates with instance-specific parameters, and deploys resources via Kubernetes Server-Side Apply (SSA). This pipeline ensures that component teams own their manifests while the platform operator controls deployment ordering and parameterization.

### Disconnected Support

RHOAI supports air-gapped (disconnected) deployments through the RELATED_IMAGE pattern. All container image references are declared as `RELATED_IMAGE_*` environment variables in the operator's ClusterServiceVersion (CSV), enabling OLM to catalog them for mirroring. Administrators use `oc-mirror` to create a local mirror registry containing all 91 container images. All image references within the operator use SHA-256 digests rather than mutable tags, ensuring reproducible deployments. The `odh-trusted-ca-bundle` ConfigMap mechanism allows injection of custom CA certificates into all platform components for environments with enterprise TLS inspection proxies.

## Platform Architectural Analysis

### Ingress Architecture Transition

The most significant architectural change in RHOAI 3.2 is the migration from per-component OpenShift Routes to a centralized Gateway API architecture. The new model deploys an Istio-based Envoy proxy in `openshift-ingress` with a kube-auth-proxy sidecar for centralized authentication. An EnvoyFilter injects ext_authz and Lua filters into the proxy — the Lua filter strips OAuth2 cookies from upstream requests and injects Bearer tokens. This is architecturally elegant but creates a significant blast radius: if the Gateway Envoy or kube-auth-proxy fails, all platform services become inaccessible simultaneously. The previous per-component Route model was less efficient but more resilient to individual component failures.

The transition is incomplete in RHOAI 3.2 — DSPO, TrustyAI, and Model Registry still use the RHOAI 2.x pattern of OpenShift Routes with kube-rbac-proxy sidecars, while dashboard, notebooks, KServe, KubeRay, and MLflow use the new Gateway API model. This creates two parallel authentication paths that must both be maintained, increasing operational complexity and security audit surface.

### Authentication Architecture Fragmentation

RHOAI 3.2 exhibits at least five distinct authentication patterns depending on the component and access path: (1) Gateway ext_authz via kube-auth-proxy for browser-based access; (2) kube-rbac-proxy sidecars for Route-based services; (3) Kuadrant AuthPolicies with Authorino for LLMInferenceService endpoints; (4) application-level Bearer token validation (DSP, Feast); and (5) direct Kubernetes API authentication for operator controllers. While each pattern serves a valid purpose, the lack of a single unified auth model increases the difficulty of security auditing and makes it harder to reason about the overall platform authentication posture. The odh-model-controller's CRD-gated feature activation means the same InferenceService may have full Kuadrant auth enforcement or no auth at all depending on whether Kuadrant CRDs are installed — a significant behavioral variance that is not visible at the API level.

### Template-Driven vs Programmatic Resource Management

A recurring architectural tension exists between components that use template-driven resource generation (DSPO, Model Registry Operator, TrustyAI Operator, MLflow Operator) and those that construct resources programmatically in Go (KServe, rhods-operator, odh-model-controller). Template-driven operators are easier to maintain for DevOps teams and clearly separate resource shape from controller logic, but they make it difficult to determine the exact Kubernetes resources created without reading both Go code and YAML templates. Programmatic operators offer better IDE support and type safety but tend to produce longer, harder-to-review controller code. The platform does not enforce a single pattern, resulting in heterogeneous operator architectures that require different debugging approaches.

### Guardrails Architecture Layering

The AI safety stack in RHOAI 3.2 has notable architectural depth: vLLM Orchestrator Gateway (Rust) → FMS Guardrails Orchestrator (Rust) → Guardrails Detectors (Python FastAPI microservices) → LLM backends (vLLM). The TrustyAI Operator's autoConfig feature — discovering labeled InferenceServices and dynamically generating orchestrator configuration — is a powerful declarative pattern that reduces manual configuration. However, the chain involves three separate Rust/Python services between the user request and the LLM backend, each with its own TLS configuration, health checks, and failure modes. The system also manages NeMo Guardrails as a separate controller within the TrustyAI Operator binary, providing an alternative guardrails approach with a Colang DSL. The coexistence of two guardrails frameworks (FMS and NeMo) within the same operator reflects the platform's need to support multiple AI safety paradigms.

### Observability Stack Complexity

The rhods-operator's Monitoring controller deploys an entire observability platform: Prometheus (via MonitoringStack CRD from Cluster Observability Operator), Tempo (distributed tracing), OpenTelemetry Collector (telemetry collection), and Perses (dashboarding). It dynamically generates PrometheusRules based on which components are enabled and validates dependent operator CRDs before deployment. Twenty-three of 40 components expose Prometheus `/metrics` endpoints, and 31 implement health probe endpoints. This is a mature observability foundation, but the dependency on three external operators (Cluster Observability Operator for Prometheus, Tempo Operator, and OpenTelemetry Operator) creates a complex dependency chain for the monitoring subsystem alone.

### LLM Serving Evolution

KServe's codebase reflects its evolution from traditional ML model serving to LLM workloads. The new LLMInferenceService controller manages InferencePool and InferenceModel CRDs for intelligent request routing via the EPP (Endpoint Picker Protocol) scheduler, supports prefill/decode disaggregation through separate Deployments, and handles self-signed TLS certificates for pod-to-pod encryption. The llm-d-inference-scheduler implements ext_proc integration with Istio for request-level routing based on model cache state, while the llm-d-routing-sidecar enables disaggregated prefill/decode architectures. This represents a significant investment in LLM-specific infrastructure that goes well beyond traditional model serving patterns, positioning RHOAI for high-performance multi-tenant LLM deployments.

### Cross-Namespace Certificate Management

Several components implement cross-namespace certificate propagation patterns that warrant attention at scale. The odh-model-controller propagates Ray TLS CA certificates from the controller namespace to all user namespaces containing multi-node ServingRuntimes. The rhods-operator distributes `odh-trusted-ca-bundle` ConfigMaps across namespaces. The DSPO relies on OpenShift service-ca for per-namespace TLS certificate generation. These patterns work well at moderate scale but create O(N) reconciliation work as namespace count grows, and certificate rotation requires coordinated updates across all namespaces.
