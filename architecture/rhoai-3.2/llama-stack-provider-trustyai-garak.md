## Purpose

**Short**: An out-of-tree Llama Stack evaluation provider that integrates NVIDIA's Garak LLM red-teaming framework for automated vulnerability scanning and security assessment of large language models.

**Detailed**: The llama-stack-provider-trustyai-garak component implements the Llama Stack `Eval` API to provide automated LLM red-teaming capabilities using Garak. It enables security teams and ML engineers to run vulnerability assessments against deployed LLMs, covering OWASP LLM Top 10, AVID security/ethics/performance taxonomies, and custom probe sets. The provider supports prompt injection, jailbreak, toxicity, and bias testing through Garak's extensive probe library.

The component operates in two distinct execution modes: **inline mode**, where Garak scans run locally within the Llama Stack server process as async subprocesses, and **remote mode**, where scans are dispatched as Kubeflow Pipeline (KFP) runs on OpenShift AI / Data Science Pipelines. Remote mode submits a three-step pipeline (validate → scan → parse results) that runs inside a container image built from this repository. Both modes share a common base class (`GarakEvalBase`) for benchmark management, command construction, and result parsing.

The provider also supports **shield testing** — measuring the effectiveness of LLM guardrails (e.g., Prompt-Guard-86M) by routing Garak attack probes through Llama Stack's Safety API before and after the LLM. Results are produced in Garak's native JSONL format, converted to AVID (AI Vulnerability Database) taxonomy reports, and stored via the Llama Stack Files API with HTML visual reports.

## Architectural Analysis

The provider follows a clean **strategy pattern** with a shared base class (`GarakEvalBase`) that encapsulates benchmark management, command construction, sampling parameter extraction, and result parsing, while two concrete adapters (`GarakInlineEvalAdapter` and `GarakRemoteEvalAdapter`) implement mode-specific execution, job lifecycle management, and resource cleanup. This design allows the same benchmark definitions and garak command-line arguments to work identically whether scans run locally or on Kubeflow Pipelines, with the remote adapter overriding key methods like `_ensure_garak_installed()` (no-op since garak runs in-container) and `_resolve_framework_to_probes()` (returns `['all']` with `probe_tags` for runtime resolution in the KFP pod).

The **shield testing architecture** is particularly notable. The `SimpleShieldOrchestrator` class acts as a callable that Garak invokes as a `function.Single` generator. Instead of directly calling the LLM, it first passes each attack prompt through configured input shields (e.g., Prompt-Guard-86M) via the Llama Stack Safety API. If no violation is detected, it forwards the prompt to the LLM, then optionally passes the LLM's response through output shields. This enables measuring guardrail effectiveness — comparing scan results with and without shields reveals how many attacks each shield blocks. The orchestrator is a module-level singleton (`simple_shield_orchestrator`) that Garak references by its fully-qualified Python name (`llama_stack_provider_trustyai_garak.shield_scan#simple_shield_orchestrator`).

The **compatibility layer** (`compat.py`) is a significant architectural choice — it abstracts the Llama Stack API import paths to support both the legacy `llama_stack` package layout and the newer `llama_stack_api` package. This allows the provider to work across Llama Stack versions without code changes, using a try/except import pattern that first attempts the new package structure and falls back to the legacy one.

The **Hermeto/Konflux build story** is unusually complex for a Python project. The `Cargo.toml` and `Cargo.lock` at the repository root are not for a Rust application — they exist solely as a workaround for a Hermeto bug that fails to prefetch Rust build dependencies of Python packages (like `fancy-regex` required by `garak`). Similarly, the `.konflux/` directory contains architecture-specific requirements files and a prep script to work around a Hermeto bug where sdists are not fetched when binary wheels are unavailable for certain architectures (s390x, ppc64le). The `.konflux/main.rs` file is a one-line dummy Rust binary referenced by `Cargo.toml` — it compiles to nothing but satisfies Hermeto's requirement for a valid Rust crate. This is well-documented in `.konflux/README.md`.

The **KFP pipeline** (`garak_scan_pipeline`) demonstrates a robust three-step pattern: input validation (Llama Stack connectivity, garak installation, command safety), scan execution with retry logic and GPU/CPU branching, and result parsing with AVID report generation. The pipeline resolves its base image dynamically — first from the `KUBEFLOW_BASE_IMAGE` environment variable, then from the `trustyai-service-operator-config` ConfigMap in `redhat-ods-applications` or `opendatahub` namespaces, and finally falling back to a default image. The scan results flow back to the Llama Stack server through a predictable filename pattern (`{job_id}_mapping.json`) uploaded to the Files API, which the server-side adapter polls for after KFP run completion. This avoids requiring direct communication from the KFP pod back to the adapter — instead, both sides use the Files API as a shared state store.
