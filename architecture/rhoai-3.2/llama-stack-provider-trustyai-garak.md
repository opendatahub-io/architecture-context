## Purpose
**Short**: Out-of-tree Llama Stack evaluation provider that performs automated LLM red-teaming and vulnerability scanning using NVIDIA's Garak framework.

**Detailed**: This component implements an external evaluation provider for the Llama Stack framework, integrating NVIDIA's Garak vulnerability scanner to perform automated red-teaming of Large Language Models. It enables security assessments including prompt injection, jailbreak, toxicity, and bias detection, aligned with compliance frameworks such as OWASP LLM Top 10 and the AVID (AI Vulnerability and Incident Database) taxonomy.

The provider ships two execution modes: an **inline mode** that runs Garak scans locally within the Llama Stack server process, and a **remote mode** that submits scans as Kubeflow Pipelines (KFP) jobs on OpenShift AI / Data Science Pipelines. The remote mode is the production deployment model, where each scan runs as a containerized KFP pipeline task using the provider's container image. The provider also supports shield testing, allowing users to measure the effectiveness of guardrail models (e.g., Prompt-Guard-86M) against adversarial attacks by routing probes through input/output shields before and after LLM inference.

Results are produced as JSONL reports, AVID-format compliance reports, and HTML visualizations, all uploaded to the Llama Stack Files API for retrieval. The provider is packaged as a Python library (`pip install llama-stack-provider-trustyai-garak`) and as a container image (`odh-trustyai-garak-lls-provider-dsp-rhel9`) built via Konflux.

### Network Flow: Remote Scan Execution

```
User → Llama Stack Server (8321) → GarakRemoteEvalAdapter
  → KFP API (submit pipeline)
  → KFP Pod created:
    → validate_inputs (check Llama Stack connectivity, garak installed)
    → garak_scan (run garak CLI, upload results to Llama Stack Files API)
    → parse_results (parse JSONL/AVID, upload combined results, log KFP metrics)
  → User polls job_status → GarakRemoteEvalAdapter → KFP API (get run state)
  → User gets results → GarakRemoteEvalAdapter → Llama Stack Files API
```

### Network Flow: Inline Scan Execution

```
User → Llama Stack Server (8321) → GarakInlineEvalAdapter
  → asyncio subprocess (garak CLI)
  → garak → vLLM (OpenAI-compatible API)
  → Results written to scan_dir (/tmp/.cache/llama_stack_garak_scans/)
  → Results uploaded to Llama Stack Files API
  → User polls job_status (in-memory state)
  → User gets results from Files API
```

## Security

### TLS Configuration

The provider supports configurable TLS verification via the `tls_verify` configuration parameter:
- `true` (default): Verify TLS certificates for all outbound connections
- `false`: Disable TLS verification (development only)
- `/path/to/ca.pem`: Custom CA certificate file path

TLS settings are propagated to:
- Llama Stack client connections (`httpx.Client(verify=...)`)
- KFP client connections (`Client(verify_ssl=..., ssl_ca_cert=...)`)
- KFP pipeline pod environment (`GARAK_TLS_VERIFY` env var)
