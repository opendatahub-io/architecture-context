## Purpose

**Short**: Kubernetes operator managing TrustyAI explainability services, LM evaluation jobs, guardrails orchestration, and NEMO guardrails within Red Hat OpenShift AI.

**Detailed**: The TrustyAI Service Operator is a multi-controller Go operator built on controller-runtime that manages four distinct custom resource types within the RHOAI platform. It operates as a single binary (`/manager`) with selectable controller services enabled via the `--enable-services` flag (TAS, LMES, GORCH, NEMO_GUARDRAILS, JOB_MGR).

The TrustyAI Service (TAS) controller deploys AI explainability service instances with kube-rbac-proxy sidecars, Prometheus monitoring, and optional Istio service mesh integration (VirtualService, DestinationRule). It integrates deeply with KServe by patching InferenceService resources with payload processor endpoints (ModelMesh) or inference loggers (KServe Raw/Serverless).

The LM Evaluation Service (LMES) controller manages evaluation job pods that execute the `lm-evaluation-harness` Python framework against language models, with support for Kueue-based workload scheduling, offline/S3/OCI storage, and configurable security controls (AllowOnline, AllowCodeExecution). The GuardrailsOrchestrator (GORCH) controller deploys a multi-container guardrails inference pipeline with auto-configuration that discovers KServe InferenceServices and generates orchestrator/gateway configs dynamically. The NemoGuardrails controller deploys NVIDIA NeMo Guardrails server instances with CA bundle aggregation.

## Architectural Analysis

The TrustyAI Service Operator implements a modular multi-controller architecture where four independent controllers share a single binary. The service registration pattern in `controllers/controllers.go` allows selective enabling of controllers via the `--enable-services` flag, providing deployment flexibility. Each controller registers itself via `registerService()` calls in separate files (`controllers/tas.go`, `controllers/lmes.go`, `controllers/gorch.go`, `controllers/nemo_guardrails.go`), with `SetupControllers()` iterating through enabled services. This design allows the same operator binary to be deployed with different controller subsets for ODH (all services), RHOAI (all services), or specialized modes (LMES-only overlay).

The authentication architecture follows the RHOAI 2.x pattern consistently across all controllers — using OpenShift Routes with kube-rbac-proxy sidecars rather than the RHOAI 3.x Gateway API pattern. The `security.opendatahub.io/enable-auth` annotation on each CR triggers conditional deployment of kube-rbac-proxy containers, ConfigMaps, ServiceAccounts, and ClusterRoleBindings. All kube-rbac-proxy configurations enforce TLS 1.2 with an explicit FIPS 140-2 compliant cipher suite list. The auth proxy performs SubjectAccessReview checks against the Kubernetes API to validate bearer tokens before proxying requests to the upstream application.

The GuardrailsOrchestrator controller's auto-configuration feature is architecturally notable. Rather than requiring users to manually specify detector and generator endpoints, the controller watches for InferenceService resources in the namespace, discovers services labeled with `trustyai/guardrails: true`, and dynamically generates orchestrator and gateway ConfigMaps containing complete routing configuration. Configuration changes are tracked via a SHA-based hash in the status field, and the controller triggers redeployment when the hash changes. This enables a declarative workflow where users deploy InferenceServices and create a GuardrailsOrchestrator CR with `autoConfig`, and the controller automatically wires everything together.

The LMEval controller implements a state machine pattern for job lifecycle management (New → Scheduled → Running → Complete/Cancelled/Suspended) with security hardening for the evaluation pod. The RHOAI overlay disables online access and remote code execution by default (`lmes-allow-online=false`, `lmes-allow-code-execution=false`), while ODH enables both. The DSC ConfigMap override mechanism (`trustyai-dsc-config`) allows platform-level policy enforcement that overrides per-job settings. Input validation is rigorous — the controller validates model names, task names, and arguments against strict regex patterns and blocks shell metacharacters (`; & | $ \` ( ) < >`) to prevent injection attacks in the Python evaluation subprocess.

The TAS controller's KServe integration demonstrates two distinct integration patterns. For ModelMesh deployments, it patches the `MM_PAYLOAD_PROCESSORS` environment variable in the ModelMesh serving container with the TrustyAI service URL (`https://{name}/consumer/kserve/v2`). For KServe Raw deployments, it sets an InferenceLogger spec pointing to the TrustyAI endpoint. This dual integration pattern exists because ModelMesh and KServe use fundamentally different mechanisms for request interception — ModelMesh uses payload processors as middleware, while KServe uses inference loggers as asynchronous sinks.
