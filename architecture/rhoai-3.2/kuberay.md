## Purpose

**Short**: KubeRay is a Kubernetes operator that manages the lifecycle of Ray distributed computing clusters, jobs, and serving deployments.

**Detailed**: The KubeRay operator provides three Custom Resource Definitions -- RayCluster, RayJob, and RayService -- that allow users to declaratively manage Ray distributed computing infrastructure on Kubernetes. The operator handles cluster provisioning (head and worker pods), autoscaling, job submission and monitoring, and zero-downtime serving upgrades via a blue-green deployment strategy.

In the RHOAI distribution, the operator is extended with OpenShift-specific features including OAuth/OIDC authentication via kube-rbac-proxy sidecars, Gateway API integration (HTTPRoute-based ingress), mTLS encryption for inter-node communication via cert-manager, NetworkPolicy enforcement for network segmentation, and SecurityContextConstraints for pod security. The operator integrates with the RHOAI platform's Gateway API infrastructure by creating HTTPRoutes in the platform namespace and ReferenceGrants for cross-namespace service access.

The operator also supports batch scheduling integrations with Volcano, YuniKorn, and the Kubernetes scheduler-plugins project for gang scheduling of Ray workloads, and provides GCS fault tolerance with Redis-backed external storage for cluster state persistence.

## Architectural Analysis

The KubeRay operator follows a hierarchical CRD ownership model: RayService owns RayCluster instances (enabling blue-green serving upgrades), and RayJob optionally owns a RayCluster (enabling ephemeral job clusters). This layered ownership simplifies lifecycle management -- deleting a RayService cascades to its RayClusters, which cascade to pods and services.

The authentication architecture represents a RHOAI 3.x pattern where both OAuth and OIDC modes converge on the same implementation: a kube-rbac-proxy sidecar injected into head pods, combined with an HTTPRoute created in the platform namespace (e.g., `redhat-ods-applications`) that references the RHOAI data-science-gateway. The `ray.io/enable-oauth` and `ray.io/enable-oidc` annotations on the RayCluster CR both trigger the same OIDC-based flow (as noted in the code: "Both OAuth and OIDC modes now use the same OIDC sidecar"). The authentication controller creates a cross-namespace ReferenceGrant to allow the Gateway in the gateway namespace (default: `openshift-ingress`) to reference the backend service in the user's namespace. This is a notable security design -- each namespace gets a single ReferenceGrant shared across all RayClusters in that namespace.

The mTLS and NetworkPolicy controllers are activated via the annotation `ray.io/enable-secure-trusted-network` (aliased as `odh.ray.io/secure-trusted-network`). When enabled, the mTLS controller provisions a full PKI chain via cert-manager (self-signed root issuer -> CA issuer -> head cert + worker cert), and the NetworkPolicy controller creates per-cluster ingress policies that restrict traffic to head pods (ports 8265, 10001, 8443, 8080) and allow intra-cluster worker communication. The mutating webhook automatically sets this annotation to `true` on OpenShift clusters, making security-by-default the norm in RHOAI deployments.

The operator's batch scheduler integration is pluggable via the `BatchScheduler` configuration field, supporting Volcano (PodGroup CRD with queue and priority), YuniKorn (pod annotations with task group definitions), and scheduler-plugins (PodGroup CRD). Each scheduler plugin implements the `BatchScheduler` interface, which is called during cluster submission and pod creation to add appropriate scheduling metadata. This design cleanly separates scheduling concerns from core cluster management.

The OpenShift kustomize overlay introduces several RHOAI-specific customizations: a SecurityContextConstraints resource (`run-as-ray-user`) that enforces UID 1000, drops all capabilities, and restricts seccomp profiles; OpenShift serving-cert integration for webhook TLS (avoiding cert-manager dependency for the webhook itself); and environment variable patches that configure the Gateway API namespace for HTTPRoute placement. The params.env file provides the operator image and namespace as configurable parameters consumed by the platform operator (rhods-operator/opendatahub-operator) during component deployment.
