## Purpose
**Short**: KubeRay Operator manages the lifecycle of Ray clusters, jobs, and services on Kubernetes/OpenShift.

**Detailed**: The KubeRay Operator is a controller-runtime based Kubernetes operator that manages Ray distributed computing infrastructure. It provides three custom resources - RayCluster, RayJob, and RayService - enabling users to deploy and manage Ray clusters, submit batch jobs, and operate Ray Serve applications declaratively on Kubernetes.

In the RHOAI 3.2 distribution, the operator includes significant OpenShift-specific enhancements: an authentication controller that configures kube-rbac-proxy sidecars and Gateway API HTTPRoute resources for secure external access to Ray dashboards, a network policy controller that creates fine-grained NetworkPolicies for cluster isolation, and an mTLS controller that uses cert-manager to provision TLS certificates for intra-cluster encryption. The operator integrates with the RHOAI platform's data-science-gateway for ingress, supporting cross-namespace HTTPRoute routing with ReferenceGrant-based trust.

The operator is built with FIPS-compliant Go (strictfipsruntime), runs as a non-root container on UBI 9 Minimal, and is deployed via kustomize overlays managed by the rhods-operator. It supports batch scheduling integration with Volcano and Yunikorn, autoscaling with the Ray autoscaler, and GCS fault tolerance with Redis-backed storage.

**Controller-Managed Ingress (RHOAI 3.x Pattern)**:

The Authentication Controller dynamically creates the following resources per RayCluster when the `odh.ray.io/secure-trusted-network: "true"` annotation is set:

1. **HTTPRoute** (in platform namespace, e.g., `opendatahub`) referencing `data-science-gateway` in `openshift-ingress` namespace
   - Path: `/ray/{namespace}/{cluster}` with URL rewrite to `/`
   - Cross-namespace backend reference to the head service in the cluster's namespace
2. **ReferenceGrant** (in cluster namespace) allowing HTTPRoutes from the platform namespace to reference Services
   - Shared per namespace, reference-counted cleanup
3. **kube-rbac-proxy sidecar** (8443/TCP) injected into Ray head pods
   - Proxies to Ray Dashboard at `127.0.0.1:8265`
   - Enforces Kubernetes RBAC (SubjectAccessReview) before proxying
   - Config: checks "get" verb on "services" resource for the head service
4. **ServiceAccount** per RayCluster for authentication
5. **ConfigMap** with kube-rbac-proxy authorization config
