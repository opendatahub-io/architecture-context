## Purpose

**Short**: NeMo Guardrails is an open-source toolkit for adding programmable safety guardrails to LLM-based conversational systems, providing input/output filtering, content safety, jailbreak detection, and topic control.

**Detailed**: NeMo Guardrails (originally by NVIDIA) provides a framework for controlling the behavior of large language model (LLM) applications. It uses a domain-specific language called Colang (supporting versions 1.0 and 2.x) to define conversational flows and safety rails that intercept, validate, and filter both user inputs and LLM outputs.

The server exposes an OpenAI-compatible chat completions API (FastAPI/Uvicorn on port 8000) that processes messages through a configurable pipeline of input rails, dialog rails, retrieval rails, and output rails. Each rail type can invoke built-in guardrail modules (24+ included, such as content safety, jailbreak detection, hallucination detection, sensitive data detection, and injection detection) or custom user-defined actions.

In the RHOAI distribution, NeMo Guardrails is built as a container image (`odh-nemo-guardrails`) using Konflux CI/CD. The image pre-downloads required ML models (spaCy, sentence-transformers, FastEmbed) at build time and serves guardrail configurations mounted via ConfigMap at runtime. It integrates with external LLM services (OpenAI, NVIDIA NIM, or any LangChain-compatible provider) and can optionally use Redis for session persistence and embeddings caching.

## Architectural Analysis

NeMo Guardrails follows a **pipeline architecture** where user messages flow through a sequence of configurable rail stages (input → dialog → retrieval → output), each able to block, modify, or annotate the message. The core design pattern is an **event-driven state machine** implemented in the Colang 2.x runtime, which tracks multiple concurrent "heads" (execution pointers) across flow definitions. This enables complex conversational patterns like parallel rail evaluation, conditional branching, and multi-turn dialog management without blocking the async event loop.

The **Colang DSL** is a significant architectural element — it provides a declarative way to define guardrail behavior without writing Python code. The dual-version support (Colang 1.0 via a custom parser and Colang 2.x via a Lark EBNF grammar) adds complexity but ensures backward compatibility. The Colang 2.x runtime (`statemachine.py` at ~95KB) is the most complex single module, implementing flow lifecycle management, conflict resolution between competing flows, and head-based multi-flow execution with fork/merge semantics.

The **guardrail library** follows a plugin architecture with 24+ built-in modules, each implementing the `@action()` decorator interface. Guardrails are selected at build time via the `GUARDRAILS_PROFILE` build argument (`opensource` vs `all`), which controls whether closed-source provider integrations (ActiveFence, CleanLab, Clavata, PrivateAI, Fiddler, PatronusAI, Prompt Security, GCP Moderate Text) are included in the image. The RHOAI build uses the `opensource` profile, excluding these proprietary integrations. Each guardrail module follows a consistent pattern: define an async action function that calls the LLM task manager with a specialized prompt template, parse the LLM's classification response, and return allow/block/modify decisions.

The **embedding and knowledge base subsystem** uses Annoy for approximate nearest-neighbor search with a pluggable provider model (FastEmbed, SentenceTransformers, OpenAI, Cohere, NIM, etc.). Embedding computation results are cached using a configurable cache backend (in-memory, filesystem, or Redis) with pluggable key generators (hash, MD5, SHA-256). The MD5 key generator has a FIPS compatibility gap — it does not implement the graceful fallback present in `utils.py`, so it will fail if selected on a FIPS-enabled host. This is a minor risk since the default key generator is `hash` (Python's built-in hash function), not MD5.

The **server architecture** is intentionally minimal — a thin FastAPI wrapper around the Colang runtime with no built-in authentication, authorization, or TLS. This is by design: in RHOAI deployments, the platform operator (rhods-operator) provides these capabilities via kube-rbac-proxy sidecars and Gateway API HTTPRoutes. The server supports an extensibility hook via `config.py` where custom initialization (including auth middleware) can be injected at startup. The auto-reload feature (via watchdog file monitoring) enables dynamic configuration updates without server restart, which is useful for development but should be disabled in production.
