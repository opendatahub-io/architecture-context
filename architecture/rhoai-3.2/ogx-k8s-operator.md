## Purpose

**Short**: Kubernetes operator that manages the lifecycle of LlamaStack distribution servers — AI inference endpoints running the Meta Llama Stack framework.

**Detailed**: The LlamaStack K8s Operator (internally `ogx-k8s-operator`, upstream `llama-stack-k8s-operator`) is a controller-runtime-based Kubernetes operator that watches `LlamaStackDistribution` custom resources and reconciles them into fully managed deployments of the LlamaStack inference server. When a user creates a `LlamaStackDistribution` CR, the operator provisions a complete set of Kubernetes resources: a Deployment running the specified LlamaStack distribution image, a ClusterIP Service for intra-cluster access, a ServiceAccount with an OpenShift SCC RoleBinding for `anyuid`, optional PersistentVolumeClaims for model/data storage, optional NetworkPolicies, PodDisruptionBudgets, and HorizontalPodAutoscalers.

The operator supports multiple LlamaStack distribution images (starter, remote-vllm, meta-reference-gpu, postgres-demo) that are embedded at build time in `distributions.json`, with the RHOAI distribution overridden via the `RELATED_IMAGE_RH_DISTRIBUTION` environment variable. It provides advanced configuration capabilities including user-supplied ConfigMaps for custom `run.yaml` server configuration, CA bundle management (both explicit and auto-detected from the ODH trusted CA bundle ConfigMap), TLS certificate validation with PEM parsing, content-based hash annotations for automatic pod restarts on config changes, image mapping overrides via operator-level ConfigMaps, and feature flags (e.g., NetworkPolicy toggle). The operator uses kustomize internally to render base manifests and applies Go-based transformation plugins (namespace setter, name prefix, field mutator) to customize resources per CR instance.

The operator fits into the RHOAI platform as the component responsible for deploying and managing LlamaStack-based AI inference servers, complementing other inference-serving components like KServe and vLLM by providing a Meta Llama Stack-native deployment path.

## Architectural Analysis

The LlamaStack K8s Operator follows a clean, single-controller architecture built on the Kubebuilder scaffold with controller-runtime v0.22.4. The operator manages exactly one CRD (`LlamaStackDistribution`) and generates all child resources through a kustomize-based rendering pipeline rather than imperative Go code. Base manifests in `controllers/manifests/base/` define template resources (Deployment, Service, PVC, ServiceAccount, RoleBinding, NetworkPolicy, HPA, PDB), which are rendered at reconciliation time through kustomize and then transformed by three Go-based plugins: a namespace setter, a name prefix transformer (which excludes Deployments for backward compatibility), and a field mutator that uses JSON Pointer paths to inject instance-specific values like replica counts, port numbers, storage sizes, and label selectors. This approach keeps manifest generation declarative and testable.

A notable design pattern is the operator's ConfigMap-driven architecture. The operator watches three categories of ConfigMaps: user-supplied configuration (`userConfig.configMapName` referenced in the CR spec for custom `run.yaml` files), CA bundle ConfigMaps (both explicit via `tlsConfig.caBundle` and auto-detected via the ODH well-known `odh-trusted-ca-bundle` ConfigMap), and the operator's own configuration ConfigMap (`llama-stack-operator-config`) containing feature flags and image mapping overrides. Content-based hashing of ConfigMap data drives pod annotations that trigger rolling restarts when configuration changes. The CA bundle management is particularly thorough: the operator validates PEM structure and X.509 certificate format, enforces size limits (10MB, 1000 certificates), and concatenates all certificate sources into a single operator-managed ConfigMap mounted at `/etc/ssl/certs/ca-bundle/ca-bundle.crt` with `SSL_CERT_FILE` automatically set.

The operator demonstrates good upgrade-safety practices. It performs one-time cleanup of legacy ClusterRoleBindings from previous versions that used cluster-scoped bindings (migrated to namespace-scoped RoleBindings). It preserves immutable Deployment selectors during updates by copying them from the existing resource. It uses server-side apply (SSA) with `ForceOwnership` for Deployment patches and owner-reference checks before patching any resource to prevent "stealing" resources from other controllers. PVCs are explicitly skipped during patching since they're immutable after creation.

The deployment topology is straightforward: a single operator pod with leader election (using Lease resources) manages `LlamaStackDistribution` instances across all namespaces. Each CR produces a Deployment running one of several LlamaStack distribution images (starter, remote-vllm, meta-reference-gpu, postgres-demo, or a custom image). The operator actively health-checks managed servers by making HTTP requests to `/v1/health`, `/v1/providers`, and `/v1/version` endpoints, populating the CR status with provider information, server version, and phase (Pending → Initializing → Ready/Failed). The operator supports HA features including HorizontalPodAutoscaler integration (with automatic replica field removal from Deployment manifests when HPA is enabled), PodDisruptionBudgets, topology spread constraints across regions/zones/hosts, and pod anti-affinity for multi-replica deployments.

The container startup logic is interesting: the operator injects a shell script as the container command that detects the installed `llama-stack` Python package version and selects the appropriate module path or uvicorn CLI command, ensuring compatibility across llama-stack versions 0.2.x through 0.3.x+. Worker count configuration is passed through `LLS_WORKERS` environment variable to uvicorn for horizontal scaling within a single pod.
