## Purpose

**Short**: A Kubernetes controller that extends KServe model serving with OpenShift integration, NVIDIA NIM account management, LLM inference service authorization, and observability features.

**Detailed**: The odh-model-controller watches KServe InferenceService, ServingRuntime, InferenceGraph, and LLMInferenceService custom resources and reconciles supplementary Kubernetes resources required for the RHOAI model serving platform. For standard InferenceService resources, it manages OpenShift Routes for external access, Prometheus ServiceMonitors/PodMonitors for metrics collection, KEDA TriggerAuthentications for autoscaling, ClusterRoleBindings for authentication, NetworkPolicies for network isolation, and data connection secrets aggregated into storage configuration.

For LLMInferenceService resources (the newer Model-as-a-Service API), the controller manages Kuadrant AuthPolicies for Gateway API authentication/authorization, Istio EnvoyFilters for TLS connectivity to the Authorino authorization service, and tier-based RBAC Roles/RoleBindings for multi-tenant access control. It also manages NVIDIA NIM Account resources by validating API keys, fetching model metadata from the NIM API, creating Docker pull secrets for nvcr.io, and generating OpenShift Templates containing ServingRuntime definitions.

The controller integrates with several platform components including the Gateway API infrastructure (Gateways, HTTPRoutes), Kuadrant/Authorino for API authentication, Istio for service mesh traffic management, Prometheus for observability, KEDA for autoscaling, the Kubeflow Model Registry for model metadata tracking, and Ray for multi-node distributed inference TLS certificate management.

## Architectural Analysis

The odh-model-controller follows a hierarchical composition pattern where top-level controllers delegate to sub-reconcilers. The InferenceServiceReconciler aggregates six sub-reconcilers (Route, ClusterRoleBinding, MetricsService, ServiceMonitor, MetricsDashboard, KEDA) into a single orchestration unit. Similarly, the LLMInferenceServiceReconciler composes four sub-reconcilers (Role, RoleBinding, AuthPolicy, EnvoyFilter). This pattern provides modularity—each sub-reconciler handles exactly one resource type—and error isolation through `multierror` aggregation. However, the hierarchical depth (manager → controller → orchestrator → sub-reconciler) adds complexity when debugging reconciliation failures.

A notable design decision is the use of **CRD-gated feature activation**. The LLM controller conditionally watches Kuadrant AuthPolicies, Authorino CRs, and Istio EnvoyFilters only if their respective CRDs are installed in the cluster. This allows the controller to run on clusters without the full Kuadrant/Istio stack, degrading gracefully rather than crashing. The tradeoff is that the reconciliation behavior varies significantly depending on cluster configuration—the same LLMInferenceService will get full auth enforcement on a Kuadrant-enabled cluster but no auth at all on a bare cluster.

The controller implements two distinct ingress patterns for model serving. For standard InferenceServices, it uses OpenShift Routes with optional TLS reencrypt and auth via ClusterRoleBindings (the RHOAI 2.x pattern). For LLMInferenceServices (MaaS), it uses Gateway API with Kuadrant AuthPolicies targeting either the Gateway (shared, UserDefined auth) or HTTPRoutes (per-service, Anonymous auth), with Istio EnvoyFilters configuring TLS upstream connectivity to Authorino (the RHOAI 3.x pattern). Both patterns coexist in the same controller, which must be understood when analyzing the codebase.

The NIM Account subsystem uses a handler chain pattern (Validation → ConfigMap → Template → PullSecret) where each handler returns a `HandleResponse` controlling flow (continue/requeue/error). The handlers integrate with the NVIDIA NIM external API for model metadata, introducing an external dependency that requires rate-limited validation (default 24h) and time-based config refresh. The `sync.Once` pattern in the APIKeyManager prevents redundant secret fetches within a reconciliation cycle.

The Ray TLS certificate management spans both the Pod webhook (mutating) and the Pod/ServingRuntime controllers. The webhook injects init containers and volume mounts at pod creation time, while the controllers manage the CA certificate lifecycle and per-pod TLS certificate generation based on pod IP addresses. Certificate data flows from the controller namespace (ray-ca-tls) to user namespaces (ray-tls), with the controller propagating CA updates across all namespaces containing multi-node ServingRuntimes. This cross-namespace certificate propagation is a potential scalability concern in clusters with many multi-node inference deployments.
