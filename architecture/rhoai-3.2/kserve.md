## Purpose

**Short**: KServe is a Kubernetes-native model serving platform that provides CRDs and controllers for deploying, scaling, and managing machine learning inference services, inference graphs, and LLM workloads.

**Detailed**: KServe defines and reconciles a family of Custom Resource Definitions in the `serving.kserve.io` API group that cover the full lifecycle of ML model serving on Kubernetes. The core `InferenceService` CRD (v1beta1) manages single-model and multi-model serving with support for predictor, transformer, and explainer components across three deployment modes: Serverless (Knative), RawDeployment (plain Kubernetes Deployments), and ModelMesh (multi-model routing). Each mode creates the appropriate compute resources (Knative Services, Deployments, HPAs) and ingress resources (Istio VirtualServices, Gateway API HTTPRoutes, or Kubernetes Ingress objects) depending on the platform configuration.

The newer `LLMInferenceService` CRD (v1alpha1) extends KServe with LLM-specific capabilities: multi-node inference via LeaderWorkerSet, prefill/decode disaggregation for optimized token generation, Gateway API Inference Extension integration (InferencePool, InferenceModel) for intelligent request routing, self-signed TLS certificates for pod-to-pod encryption, Istio DestinationRule management for mTLS, and a scheduler (EPP — Endpoint Picker Placement) for prefix-cache-aware and load-aware request scheduling. Additional CRDs support inference graphs (`InferenceGraph`), multi-model serving (`TrainedModel`), serving runtime definitions (`ServingRuntime`, `ClusterServingRuntime`), local model caching (`LocalModelCache`, `LocalModelNode`, `LocalModelNodeGroup`), and pluggable storage backends (`ClusterStorageContainer`).

The repository produces six container images via Konflux: the controller manager (Go operator), the inference agent (Go sidecar for model management/logging/batching), the inference router (Go service for InferenceGraph traversal), the storage initializer (Python init container for model download), and two local model managers (Go controllers for cluster-wide and per-node model caching). A Python SDK provides the model serving framework (FastAPI/gRPC server, V1/V2 inference protocols) and a Kubernetes client library. On RHOAI, KServe integrates with the platform's Gateway API ingress, OpenShift service-ca for TLS, and Istio service mesh for mTLS.

## Architectural Analysis

KServe's architecture reflects its evolution from a Knative-centric serving framework (KFServing) into a multi-modal platform supporting traditional ML models, inference graphs, and LLM workloads. The codebase has three distinct controller families that coexist in a single binary: the mature v1beta1 InferenceService controller, the v1alpha1 controllers (InferenceGraph, TrainedModel, LocalModel), and the newer LLMInferenceService controller. Each family has its own reconciliation patterns and resource management strategies, creating a broad but sometimes inconsistent API surface.

The InferenceService controller implements a plugin-style architecture where deployment mode (Serverless/RawDeployment/ModelMesh) and ingress mode (Istio VirtualService / Gateway API HTTPRoute / Kubernetes Ingress) are selected at reconciliation time based on ConfigMap configuration and cluster CRD availability. This allows the same InferenceService CR to work across different cluster configurations without changes. The component model (predictor, transformer, explainer) is compositional — each component independently creates its compute resources and the ingress reconciler wires them together with routing rules.

The LLMInferenceService controller represents a significant architectural shift toward Gateway API Inference Extension primitives. It manages InferencePool and InferenceModel CRs for intelligent request routing via the EPP scheduler, supports prefill/decode disaggregation through separate Deployments or LeaderWorkerSets, and handles self-signed TLS certificates for pod-to-pod encryption. The configuration merging system (`config_merge.go`) uses a BaseRefs pattern with well-known defaults and Go template variable replacement, allowing platform operators to ship versioned configuration templates (e.g., `kserve-config-llm-template`) that users can extend. The fail-secure auth pattern in `router_validation_ocp.go` — which refuses to create HTTPRoutes if AuthPolicy CRDs are unavailable — demonstrates defense-in-depth thinking for multi-tenant LLM serving.

A notable FIPS compliance gap exists in the localmodel and localmodel-agent images, which are built with `CGO_ENABLED=0` and use distroless base images without OpenSSL. These will fail check-payload validation and likely represent upstream-only images not shipped in RHOAI. The three primary RHOAI images (controller, agent, router) correctly use `GOEXPERIMENT=strictfipsruntime` with CGO dynamic linking on UBI 9 base images, and the self-signed certificate generation uses FIPS-approved algorithms (RSA 4096, SHA-256, PKCS#8).

The webhook system is central to KServe's storage initialization architecture — the `/mutate-pods` webhook intercepts pod creation, detects model storage URIs from annotations, and injects the storage-initializer init container along with credential volumes and CA bundle mounts. This pattern keeps the model download logic separate from the inference containers, enabling support for 9+ storage backends (S3, GCS, Azure, HDFS, HTTP, Hugging Face, OCI, PVC, file) without modifying individual serving runtimes.
