## Purpose

**Short**: Builds and publishes container images for interactive data science workbenches (Jupyter, RStudio, Code Server) and pipeline runtime environments used by the RHOAI platform.

**Detailed**: The notebooks repository produces the complete set of workbench container images that data scientists interact with directly in Red Hat OpenShift AI. It is not an operator or running service itself -- it is a build system that produces ~22 distinct container images across three IDE families (JupyterLab, RStudio Server, VS Code/code-server), three compute backends (CPU, NVIDIA CUDA, AMD ROCm), and four architectures (x86_64, aarch64, ppc64le, s390x).

Each workbench image is a self-contained development environment with Python, ML frameworks (PyTorch, TensorFlow, LLMCompressor), data science libraries, and an IDE frontend proxied through NGINX. The images are published as OpenShift ImageStreams via kustomize manifests, enabling the ODH Notebook Controller to spawn user workbenches as StatefulSets. Additionally, the repository produces lightweight "runtime" images used by Elyra for executing pipeline nodes in Kubeflow Pipelines (KFP) workflows.

The build system uses a layered image hierarchy: base images (cpu-base, cuda-base, rocm-base) provide common OS packages and Python tooling, intermediate stages add ML frameworks, and final stages add IDE-specific tooling. Downstream RHOAI builds use Konflux/Tekton pipelines with Dockerfile.konflux variants, while upstream ODH uses GitHub Actions. Dependencies are locked via per-image `pylock.toml` files with hash verification.

## Architectural Analysis

The notebooks repository exemplifies a sophisticated multi-variant container image build system designed to serve the diverse needs of data science practitioners across multiple hardware platforms and IDE preferences. Its architecture reflects several key design decisions that balance developer experience, operational complexity, and enterprise requirements.

**Layered Image Hierarchy**: The most notable architectural pattern is the compute-type base layer system. Three foundation stages -- `cpu-base`, `cuda-base`, and `rocm-base` -- establish the common runtime environment (Python 3.12, uv, micropipenv, OpenShift CLI), while application-specific stages build incrementally: `cpu-base` -> `jupyter-minimal` -> `jupyter-datascience` -> `jupyter-pytorch`. This layered approach minimizes image size duplication and ensures consistent tooling across variants. The CUDA and ROCm base images inherit NVIDIA/AMD drivers from external base images, keeping GPU driver management separate from application concerns.

**Cross-Architecture Build Complexity**: Supporting four CPU architectures (x86_64, aarch64, ppc64le, s390x) introduces significant build complexity, particularly for ppc64le and s390x where pre-built Python wheels are scarce. The Code Server Dockerfile is the most complex example, with dedicated `rpm-base` and `whl-cache` stages that compile code-server from source for non-x86 architectures and pre-build Python wheels respectively. This pattern of architecture-conditional build stages (using `uname -m` checks and dedicated builder stages) appears throughout the repository and represents a considerable maintenance burden.

**Dual CI/CD Pipeline Model**: The repository maintains two parallel build systems: Konflux/Tekton for RHOAI enterprise builds (60+ pipeline definitions in `.tekton/`) and GitHub Actions for upstream ODH development (25+ workflows in `.github/workflows/`). Konflux Dockerfiles (`Dockerfile.konflux.*`) exist only for Code Server and RStudio (3 files total), while Jupyter and runtime images use standard Dockerfiles that Konflux builds directly. This asymmetry suggests that Code Server and RStudio require downstream-specific build modifications (e.g., rpm2cpio for unsigned RPM handling, cache mount strategies) that differ from upstream builds.

**Two-Tier Kustomize Architecture**: The deployment manifests implement a clear separation between image registration (20 ImageStreams in `manifests/base/` managed by the platform operator) and workload deployment (32 per-notebook kustomize directories defining StatefulSets/Pods). The ConfigMap-driven replacement system in `manifests/base/kustomization.yaml` is particularly elegant -- over 100 replacement entries dynamically inject SHA256-pinned image references from `params.env` into ImageStream tags, enabling atomic image version updates without YAML template modifications. The 7-version retention strategy (N through N-6) ensures backward compatibility while the `opendatahub.io/notebook-build-commit` annotation provides build-to-source traceability.

**Security Model**: Workbench images follow an "unauthenticated in-pod" pattern -- JupyterLab tokens are disabled, code-server passwords are empty, and RStudio uses nss_wrapper for user mapping. This is by design: authentication is enforced at the ingress layer by kube-rbac-proxy sidecars injected by the ODH Notebook Controller. The images themselves run as UID 1001 with GID 0 for OpenShift SCC compatibility. One notable security finding is that no FIPS-specific build flags exist in any Dockerfile -- FIPS compliance relies entirely on the UBI9 base image's OpenSSL.
