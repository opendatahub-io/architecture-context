## Purpose

**Short**: A collection of detector algorithm microservices for the FMS Guardrails Orchestrator, providing text content analysis for safety, PII detection, file-type validation, custom detection logic, and LLM-as-a-judge evaluation.

**Detailed**: The guardrails-detectors repository provides three independently deployable Python FastAPI microservices that integrate with the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator). Each detector exposes a standardized REST API (`/api/v1/text/contents`) conforming to the IBM Detector API specification, allowing the orchestrator to invoke content analysis on text generation input and output.

The repository produces three detector types: (1) a **built-in detector** that bundles lightweight regex-based PII detection (email, credit card, SSN, phone numbers, IP addresses), file-type validation (JSON, XML, YAML with optional schema validation), and user-defined custom detection functions; (2) a **HuggingFace detector** that loads any `AutoModelForSequenceClassification` or `GraniteForCausalLM` model for ML-based content classification (toxicity, jailbreak, bias, etc.); and (3) an **LLM Judge detector** that uses the `vllm_judge` library to perform LLM-as-a-judge evaluation against a remote vLLM-compatible inference server.

These detectors are designed for deployment as KServe InferenceServices on Red Hat OpenShift AI, using the KServe `ServingRuntime` and `InferenceService` custom resources. The HuggingFace and LLM Judge detectors each define sample `ServingRuntime` and `InferenceService` manifests. The built-in detector is typically deployed as a sidecar alongside the Guardrails Orchestrator. All detectors expose Prometheus metrics for observability.

## Architectural Analysis

The guardrails-detectors repository follows a clean microservice architecture where each detector type is independently deployable as a container image. All three detectors share a common Python library (`detectors/common/`) that provides the FastAPI base class (`DetectorBaseAPI`), Pydantic request/response schemas conforming to the IBM Detector API specification, and an instrumented detector base class (`InstrumentedDetector`) for consistent Prometheus metrics collection. This shared foundation ensures all detectors expose the same `/api/v1/text/contents` endpoint interface and metrics format, allowing the FMS Guardrails Orchestrator to treat them uniformly.

The built-in detector's architecture is notable for its extensibility through the `BaseDetectorRegistry` pattern. Three registries (regex, file-type, custom) are loaded at startup via the FastAPI lifespan context manager. The custom detector registry (`CustomDetectorRegistry`) provides a particularly interesting capability: it dynamically loads user-defined Python functions from a `custom_detectors/custom_detectors.py` module, performing static code analysis to block forbidden imports (`os`, `subprocess`, `sys`, `shutil`) and function calls (`eval`, `exec`, `open`, `compile`, `input`) before loading. This sandboxing is best-effort -- it uses AST analysis to detect dangerous patterns but cannot prevent all possible attacks in dynamically loaded Python code. The custom detector framework also supports non-blocking background execution via a decorator pattern and user-defined Prometheus metrics.

The HuggingFace detector supports two distinct model architectures: `AutoModelForSequenceClassification` (multi-class classification with threshold-based filtering and configurable "safe labels") and `GraniteForCausalLM` (IBM's Granite Guardian causal language model with safe/unsafe token probability analysis). The model type is auto-detected from the model config at initialization. CUDA support is opportunistic -- the detector falls back to CPU if CUDA is unavailable. The Konflux Dockerfile for the HuggingFace detector includes complex multi-architecture support (amd64, ppc64le, s390x), building PyTorch from source for Power and z-series architectures and OpenBLAS from source for Power. This makes it the most complex build in the repository with 5 Docker stages.

A significant architectural observation is that none of the detectors implement authentication or authorization at the application level. Security is entirely delegated to the deployment infrastructure: Istio sidecar injection for mTLS, and the `security.opendatahub.io/enable-auth: 'true'` annotation for platform-level auth enforcement via kube-rbac-proxy. The LLM Judge detector's connection to the external vLLM server is also unauthenticated and unencrypted by default, which could be a security concern in multi-tenant environments.

The LLM Judge detector does not have a Konflux Dockerfile, indicating it may not yet be shipped as part of the RHOAI 3.2 release. The `Dockerfile.judge` exists only in upstream form. The Tekton pipelines under `.tekton/` only cover the built-in detector and huggingface runtime, confirming the LLM Judge is not part of the Konflux build pipeline for this release.
