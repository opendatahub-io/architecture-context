## Purpose

**Short**: Kubernetes operator that manages MLflow tracking server deployments on OpenShift via a cluster-scoped `MLflow` custom resource.

**Detailed**: The MLflow Operator watches cluster-scoped `MLflow` custom resources (API group `mlflow.opendatahub.io/v1`) and reconciles them into fully configured MLflow tracking server deployments. Rather than generating raw Kubernetes manifests in Go code, the operator embeds a Helm chart (`charts/mlflow/`) and renders it at reconciliation time using the Go Helm SDK, translating the MLflow CR spec fields into Helm values. This approach keeps the deployment templates maintainable and separable from the operator logic.

The operator manages the full lifecycle of MLflow deployments including: Deployment, Service, ServiceAccount, PersistentVolumeClaim (optional), NetworkPolicy, ClusterRole, and ClusterRoleBinding. Beyond the Helm-rendered resources, the controller directly reconciles two additional resources: an OpenShift `ConsoleLink` (to add MLflow to the OpenShift console's application menu) and a Gateway API `HTTPRoute` (to expose MLflow through the platform's shared ingress gateway). Both of these are conditionally created based on CRD availability detected via the Kubernetes discovery API at startup.

The operator supports multiple storage backends (SQLite/file-based for development, PostgreSQL/S3/GCS for production) and integrates with OpenShift's service-ca for automatic TLS certificate provisioning. The MLflow server itself runs with TLS enabled natively via uvicorn, serving on port 8443/TCP with certificates from the OpenShift service-ca operator.

## Architectural Analysis

The MLflow Operator follows a hybrid Helm-operator pattern, where the operator uses controller-runtime for its reconciliation loop but delegates resource rendering to the Go Helm SDK. This is an unusual but effective design choice: the Helm chart (`charts/mlflow/`) is embedded directly into the operator binary (via `COPY --from=builder /workspace/charts/mlflow charts/mlflow` in the Dockerfile), and at reconciliation time the operator translates the MLflow CR spec into Helm values, renders the templates in-memory, and applies the resulting unstructured objects via Server-Side Apply. This gives the operator the flexibility of Helm templates (conditional PVC creation, deployment strategy switching based on storage mode) while retaining the event-driven reconciliation model of a native operator.

The operator creates two resources outside the Helm chart: `ConsoleLink` and `HTTPRoute`. Both are conditionally created based on CRD availability detected via the Kubernetes discovery API at startup (not at reconciliation time). This means the operator can run on non-OpenShift clusters (no ConsoleLink) or clusters without Gateway API (no HTTPRoute) without failing. The HTTPRoute configuration routes traffic through a shared platform gateway (`data-science-gateway` in `openshift-ingress` namespace), with URL rewriting rules that strip the `/mlflow/api` prefix to `/api` before forwarding to the backend. This aligns with the RHOAI 3.x ingress pattern of using Gateway API for component exposure.

A notable design pattern is the shared ClusterRole: all MLflow instances share a single `mlflow` ClusterRole (for namespace listing, required by the workspaces feature), with each MLflow CR adding itself as a non-controller owner reference. This avoids creating duplicate ClusterRoles while ensuring garbage collection when all MLflow instances are deleted. The `clusterRoleToMLflowRequests` watch handler manually maps ClusterRole events back to all owning MLflow CRs since standard `Owns()` only works with controller owner references.

Security is handled at multiple layers: the MLflow server itself runs with TLS (service-ca certificates via uvicorn's `--ssl-keyfile`/`--ssl-certfile`), authentication uses Kubernetes SubjectAccessReview via the `--app-name=kubernetes-auth` MLflow plugin, and a NetworkPolicy restricts ingress to port 8443/TCP while allowing all egress (necessary for external database and storage connections). The operator disables HTTP/2 on its metrics endpoint as a security hardening measure. FIPS compliance is enforced at build time via `GOEXPERIMENT=strictfipsruntime` and `CGO_ENABLED=1`.

The MLflow CR is cluster-scoped with a validation rule requiring the name to be exactly `"mlflow"`, which effectively makes it a singleton pattern. This simplifies resource naming (suffix is empty for the default name) and aligns with the RHOAI model where each component has a single instance per cluster. The CRD includes extensive CEL validation rules to enforce consistent storage configuration (e.g., file-based backends require PVC, file-based artifacts require `serveArtifacts` to be enabled).
