## Purpose

**Short**: A Kubernetes operator that manages the lifecycle of MLflow tracking server instances on OpenShift/Kubernetes clusters as part of Red Hat OpenShift AI (RHOAI).

**Detailed**: The MLflow Operator watches for `MLflow` custom resources (cluster-scoped) and reconciles the desired state by rendering an embedded Helm chart to produce Kubernetes resources (Deployment, Service, ServiceAccount, PVC, NetworkPolicy, ClusterRole, ClusterRoleBinding). The operator uses server-side apply (SSA) to manage these resources. Each MLflow instance runs the MLflow tracking server with Kubernetes-native authentication (`kubernetes-auth` app) and TLS termination handled directly by uvicorn using OpenShift service-ca generated certificates.

The operator supports RHOAI 3.x ingress patterns by conditionally creating Gateway API `HTTPRoute` resources that reference the platform `data-science-gateway` in `openshift-ingress` namespace. It also conditionally creates OpenShift `ConsoleLink` resources for integration with the OpenShift web console application menu. Both features are runtime-detected via the Kubernetes discovery API, allowing the operator to work on clusters with or without Gateway API or OpenShift console support.

The operator is built with FIPS-compliant Go runtime (`strictfipsruntime`), deployed via kustomize overlays managed by the `rhods-operator`, and supports multiple MLflow instances through resource-name suffixing based on the CR name.

### Ingress

| Name | Type | Hosts | Port | Protocol | Encryption | TLS Mode | Exposure |
|------|------|-------|------|----------|------------|----------|----------|
| mlflow{suffix} | HTTPRoute (Gateway API) | Via data-science-gateway | 8443/TCP | HTTPS | TLS (passthrough to service) | N/A (backend handles TLS) | External (via Gateway) |

The operator dynamically creates HTTPRoute resources (Gateway API v1) when the HTTPRoute CRD is detected in the cluster. Each HTTPRoute references the `data-science-gateway` Gateway in the `openshift-ingress` namespace. Two route rules are created:
1. `/mlflow{suffix}/api` path prefix - rewrites to `/api` on the backend service (port 8443)
2. `/mlflow{suffix}` path prefix - forwards directly to the backend service (port 8443)
