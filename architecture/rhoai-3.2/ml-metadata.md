## Purpose
**Short**: ML Metadata is a gRPC server and library for recording and retrieving metadata associated with ML workflows, tracking artifacts, executions, and their lineage.

**Detailed**: ML Metadata (MLMD) is a metadata storage system originally developed by Google as part of TensorFlow Extended (TFX). It provides a structured way to record, query, and manage metadata generated by ML pipelines — including information about datasets, models, training runs, and pipeline executions. The system uses a relational database backend (MySQL, PostgreSQL, or SQLite) to persist metadata and exposes a gRPC API for clients to interact with the store.

In the RHOAI (Red Hat OpenShift AI) distribution, MLMD is shipped as a containerized gRPC server (`odh-mlmd-grpc-server`) built on UBI 9 (Red Hat Universal Base Image). It serves as the metadata backbone for Kubeflow Pipelines, enabling pipeline components to record artifacts, executions, contexts, events, and lineage information. The server supports MySQL, PostgreSQL, and SQLite backends, with configurable SSL/TLS for both gRPC transport and database connections. It runs as a non-root user (UID 65534) and is built via Konflux CI pipelines for multi-architecture support (amd64, arm64, ppc64le).

The Python client library (`ml-metadata` PyPI package) provides a high-level API for interacting with the gRPC server or directly with local database backends. The library includes protobuf-based message types for artifacts, executions, contexts, and their relationships.

The server supports configurable SSL/TLS via the `MetadataStoreServerConfig.SSLConfig` protobuf:

| Config Field | Purpose |
|-------------|---------|
| server_key | PEM-encoded private key for server TLS |
| server_cert | PEM-encoded server certificate |
| custom_ca | PEM-encoded custom CA certificate(s) |
| client_verify | When true, requires and verifies client certificates (mTLS) |

Without SSL configuration, the server uses insecure (plaintext) gRPC credentials.

### Protobuf Schema

The metadata model is defined in protobuf and includes these core domain types:

| Proto Type | Purpose |
|------------|---------|
| Artifact | Represents data artifacts (datasets, models) with typed/custom properties, state, and URI |
| Execution | Represents pipeline component executions with typed/custom properties and state |
| Context | Groups artifacts and executions; supports parent-child hierarchies |
| Event | Links artifacts to executions (INPUT/OUTPUT/DECLARED_INPUT/DECLARED_OUTPUT/INTERNAL_INPUT/INTERNAL_OUTPUT/PENDING_OUTPUT) |
| Attribution | Links artifacts to contexts |
| Association | Links executions to contexts |
| ArtifactType / ExecutionType / ContextType | Schema definitions for typed properties |
| Value | Property value (int64, double, string, struct, proto, bool) |
| ConnectionConfig | Database connection configuration (MySQL, PostgreSQL, SQLite, Fake) |
| MetadataStoreServerConfig | Server configuration including connection config, migration options, and SSL |
