## Purpose

**Short**: ML Metadata (MLMD) is a gRPC server and client library that records and retrieves metadata associated with ML workflows, tracking artifacts, executions, contexts, and their lineage relationships.

**Detailed**: ML Metadata is an open-source library originally developed by Google as part of the TensorFlow Extended (TFX) ecosystem. It provides a metadata store that tracks the lifecycle of ML artifacts (datasets, models, metrics), executions (training runs, transformations, evaluations), and contexts (pipelines, experiments) along with the events that connect them (input/output relationships). The core data model uses a typed property system where ArtifactTypes, ExecutionTypes, and ContextTypes define schemas, and instances carry both typed properties and arbitrary custom properties.

In the RHOAI platform, MLMD is deployed as the `odh-mlmd-grpc-server` container — a C++ gRPC server binary built via Bazel that exposes the `MetadataStoreService` gRPC API on port 8080. It serves as the persistence layer for ML pipeline metadata, enabling Kubeflow Pipelines and Data Science Pipelines (DSP) to store and query provenance information. The server supports MySQL, PostgreSQL, and SQLite backends for metadata storage, with schema migration capabilities for version upgrades.

The repository also provides Python and Go client libraries that can connect to the gRPC server or directly to a database backend. The Python client (`ml_metadata.metadata_store.MetadataStore`) supports both direct database connections (via a C++ pybind11 extension) and gRPC remote connections with optional TLS/SSL. The Go client provides similar functionality through SWIG-generated bindings to the C++ library.

## Architectural Analysis

ML Metadata follows a classic three-tier architecture: gRPC API layer → business logic → database backend. The C++ server binary is the only deployable artifact from this repository; the Python and Go libraries are client-side SDKs. The server is built using Bazel, which manages all C++ dependencies (abseil, gRPC, protobuf, BoringSSL, ZetaSQL, database client libraries) as pinned `http_archive` rules. This provides strong reproducibility for the C++ dependency tree but operates outside the typical Go/Python package management ecosystem.

The data model is centered on three core entity types (Artifact, Execution, Context), three edge types (Event, Attribution, Association), and a type system (ArtifactType, ExecutionType, ContextType) that defines property schemas. The type system supports backward-compatible evolution — new properties can be added but existing property types cannot be changed. This design enables schema flexibility while preserving data integrity across pipeline versions. The `PutExecution` and `PutLineageSubgraph` RPCs provide atomic multi-entity operations that are critical for consistent lineage recording in concurrent pipeline environments, with optimistic concurrency control (OCC) available via `abort_if_latest_updated_time_changed`.

The gRPC API supports rich query capabilities through `ListOperationOptions.filter_query`, which accepts SQL-like boolean expressions for filtering on node attributes, properties, custom properties, connected contexts, parent/child contexts, and events. This query language is powered by ZetaSQL parsing at the server side. Pagination is handled via `next_page_token` with a server-enforced maximum result size of 100 per page.

The RHOAI-specific modifications in this fork are minimal — primarily downstream container image management (Dockerfile.konflux for Konflux builds, Dockerfile.redhat for RHOAI), ZetaSQL compatibility patches for UBI9, and Tekton pipeline definitions for multi-architecture builds (amd64, arm64, ppc64le). The core MLMD codebase remains closely aligned with the upstream Google repository. The container runs as a non-root user (UID 65534) and uses UBI9-minimal as the runtime base image, keeping the attack surface small.

A notable architectural concern is the bundling of BoringSSL via Bazel rather than using the system OpenSSL from the UBI9 base image. While BoringSSL is well-maintained by Google, it complicates FIPS compliance validation because the FIPS certification status of the specific bundled BoringSSL commit may differ from Red Hat's validated OpenSSL. The non-hermetic Konflux build (`hermetic: false`) means the build environment can pull dependencies at build time, which is a supply chain consideration for security-sensitive deployments.
