## Purpose

**Short**: A FIPS-compliant authentication and authorization reverse proxy that provides OIDC, OpenShift OAuth, and Kubernetes RBAC enforcement as a sidecar for RHOAI platform components.

**Detailed**: kube-auth-proxy is a composite authentication/authorization proxy designed specifically for Red Hat OpenShift AI (RHOAI) environments. It combines two proxy binaries—an OAuth/OIDC authentication proxy (derived from oauth2-proxy) and a Kubernetes RBAC authorization proxy (derived from brancz/kube-rbac-proxy)—into a single container image with a mode-switching entrypoint. This design allows the RHOAI platform operator to deploy a single image in either "auth" mode (OIDC/OAuth authentication) or "rbac" mode (Kubernetes SubjectAccessReview authorization) depending on the component's security requirements.

The kube-auth-proxy component serves as a drop-in replacement for the legacy oauth-proxy sidecar used in earlier RHOAI versions. It is designed for the RHOAI 3.x architecture where components are exposed via Gateway API (HTTPRoute) with kube-rbac-proxy or kube-auth-proxy sidecars providing per-component authentication and authorization. The proxy intercepts incoming requests, validates user identity through OIDC or OpenShift OAuth, optionally checks Kubernetes RBAC permissions via SubjectAccessReview, and then forwards authenticated/authorized requests to the upstream application with identity headers.

FIPS compliance is a primary design requirement. All binaries are built with `GOEXPERIMENT=strictfipsruntime` and `CGO_ENABLED=1` to use Go's FIPS-validated cryptographic module. The project maintains argument compatibility with the legacy oauth-proxy to enable seamless migration of existing sidecar configurations.

## Architectural Analysis

**Dual-mode proxy architecture.** The most distinctive design decision in kube-auth-proxy is packaging two fundamentally different proxy implementations (OAuth2/OIDC authentication and Kubernetes RBAC authorization) into a single container image with a mode-switching entrypoint. This simplifies container image management for the RHOAI platform—the operator only needs to reference one image regardless of whether a component needs authentication (BYOIDC, OpenShift OAuth) or authorization (SubjectAccessReview). The entrypoint binary (`cmd/entrypoint/main.go`) is a 87-line Go program that uses `os/exec` to delegate to the selected binary, preserving stdio and exit codes transparently.

**Go workspace for multi-module monorepo.** The repository uses Go workspaces (`go.work`) to manage two separate Go modules—the root module (`github.com/opendatahub-io/kube-auth-proxy/v1`) and the kube-rbac-proxy submodule (`github.com/brancz/kube-rbac-proxy`). This reflects the component's origins: kube-auth-proxy is a streamlined fork of oauth2-proxy, while kube-rbac-proxy is a fork of brancz/kube-rbac-proxy. Keeping them as separate modules minimizes merge conflicts when pulling upstream changes while allowing a unified build.

**Session management design.** The auth-mode proxy implements a sophisticated session management system with two storage backends. The default cookie store (`pkg/sessions/cookie/session_store.go`) encrypts sessions with AES-CFB, signs them with HMAC-SHA256, and automatically splits oversized sessions across multiple cookies (4000-byte limit per cookie). For high-traffic deployments, a Redis store (`pkg/sessions/redis/redis_store.go`) uses a ticket-based architecture—an encrypted ticket is stored in the cookie, and the actual session data lives in Redis with configurable TTL and distributed locking via `redislock`. The persistence layer (`pkg/sessions/persistence/manager.go`) abstracts the underlying store, making the choice transparent to the authentication middleware.

**Hardcoded Prometheus authorization in kube-rbac-proxy.** The kube-rbac-proxy includes a hardcoded authorizer (`pkg/hardcodedauthorizer/metrics.go`) that always allows `system:serviceaccount:openshift-monitoring:prometheus-k8s` to GET /metrics. This is marked with a TODO for removal once the Cluster Monitoring Operator (CMO) supports static authorization configuration. This pattern exists because without it, the monitoring stack cannot scrape metrics from kube-rbac-proxy-protected endpoints without additional RBAC configuration.

**FIPS enforcement strategy.** Both proxy binaries are built with `CGO_ENABLED=1` and `GOEXPERIMENT=strictfipsruntime`, which forces Go to use the host's OpenSSL library (provided by UBI 9) for all cryptographic operations. The encryption layer (`pkg/encryption/cipher.go`) uses both AES-GCM (preferred) and AES-CFB (legacy compatibility) modes—both are FIPS-approved symmetric ciphers. No deprecated algorithms (MD5, RC4, DES) were detected in the codebase. The `InsecureSkipVerify` TLS option is available but defaults to off, and the OpenShift provider enforces a minimum TLS 1.2 version when constructing HTTP clients.
