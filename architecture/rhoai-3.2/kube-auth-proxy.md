## Purpose
**Short**: A FIPS-compliant authentication and authorization proxy that bundles kube-auth-proxy (OIDC/OpenShift OAuth authentication) and kube-rbac-proxy (Kubernetes RBAC authorization via SubjectAccessReview) into a single container image for use as a sidecar in RHOAI deployments.

**Detailed**: kube-auth-proxy is a focused, FIPS-compliant authentication proxy derived from oauth2-proxy, specifically designed for OpenShift Data Hub (ODH) and Red Hat OpenShift AI (RHOAI) environments. It provides secure authentication through external OIDC providers and OpenShift's built-in OAuth service, serving as a drop-in replacement for existing oauth-proxy sidecars. All non-essential providers (Google, GitHub, Azure, etc.) have been intentionally removed to minimize attack surface.

The container image ships three binaries: `kube-auth-proxy` (authentication proxy), `kube-rbac-proxy` (RBAC authorization proxy based on brancz/kube-rbac-proxy v0.19.1), and `entrypoint` (a mode-switching wrapper). The entrypoint binary selects which proxy to run based on the `PROXY_MODE` environment variable or command-line argument (`auth` or `rbac`, defaulting to `rbac`). This dual-mode design enables RHOAI components to use a single container image for both authentication-only (OIDC/OAuth flows) and authorization-only (Kubernetes SubjectAccessReview) use cases.

kube-auth-proxy supports Envoy ext_authz compatibility, enabling integration with Gateway API and Envoy-based ingress patterns used in RHOAI 3.x. The project emphasizes FIPS compliance through strictfipsruntime build tags and CGO-enabled compilation.

This component runs as a sidecar container. Ingress to the sidecar is managed by the parent component's Service/Route/HTTPRoute. In RHOAI 3.x, the common pattern is:
- **Gateway API + kube-rbac-proxy**: Components use kube-rbac-proxy sidecars (RBAC mode, port 8443/TCP) fronting application containers, with HTTPRoute CRs directing traffic through the platform Gateway.
- **OpenShift Route + kube-auth-proxy**: Components may use auth mode (port 4180/TCP) behind OpenShift Routes with edge TLS termination.
- **Envoy ext_authz**: The `/oauth2/auth` endpoint returns 200/401 for Envoy external authorization, supporting Gateway API patterns.
