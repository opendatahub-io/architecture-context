## Purpose

**Short**: Python library providing NLP capabilities (text generation, embeddings, reranking, classification) as pluggable modules for the Caikit AI runtime framework.

**Detailed**: Caikit-NLP is a Python library that implements the "task" and "module" abstractions from the Caikit framework to deliver various Natural Language Processing capabilities. It does not run as a standalone service — instead, it is loaded as a runtime library (`RUNTIME_LIBRARY=caikit_nlp`) by the Caikit serving runtime, which exposes the modules via both gRPC (port 8085) and HTTP (port 8080) APIs.

The library provides nine NLP modules organized into five task categories: text generation (local and TGIS-backed, with PEFT prompt tuning support), text embeddings and semantic similarity (via sentence-transformers), document reranking (bi-encoder and cross-encoder), text classification, and token classification. Models can be bootstrapped from HuggingFace Hub, trained locally with distributed PyTorch support, or served remotely via the Text Generation Inference Server (TGIS) backend.

In the RHOAI platform context, caikit-nlp is packaged as a container image (`odh-caikit-nlp-rhel9`) built on UBI 9. It is deployed as part of the model serving stack, loaded by the Caikit runtime within KServe InferenceService pods to handle NLP inference workloads.

## Architectural Analysis

Caikit-NLP follows the Caikit framework's module/task architecture pattern. Each NLP capability is implemented as a "module" (a Python class decorated with `@module`) that registers itself for one or more "tasks" (abstract interfaces like `TextGenerationTask` or `EmbeddingTask`). The Caikit runtime discovers these modules at startup and automatically exposes them via both gRPC and HTTP endpoints. This design means caikit-nlp itself has no network code — all serving infrastructure is inherited from the `caikit` core framework.

The text generation modules demonstrate a clean separation between local and remote inference strategies. `TextGeneration` and `PeftPromptTuning` run models entirely in-process using HuggingFace Transformers, while `TextGenerationTGIS` and `PeftPromptTuningTGIS` delegate inference to an external TGIS server via gRPC. The `TGISAutoFinder` model management component bridges these two worlds — it probes TGIS connectivity at model load time and transparently routes models to the remote backend when available, falling back to local inference otherwise. This pattern allows the same model artifacts to work in both standalone and distributed deployment topologies.

The embedding subsystem is notable for its thread-safety and optimization layers. The `SentenceTransformerWithTruncate` wrapper (in `embedding.py`, lines 903-1227) implements per-thread tokenizer copies to avoid Rust tokenizer borrow conflicts under concurrent inference. It also layers optional optimizations — Intel IPEX, PyTorch 2 compile (`torch.compile`), and autocast (bfloat16) — that can be toggled via configuration. This is architecturally significant because embedding models are typically deployed for high-throughput retrieval augmented generation (RAG) workloads where concurrency and latency matter.

The training infrastructure leverages HuggingFace's `Trainer` and `Accelerator` abstractions for distributed training support. PEFT prompt tuning is the primary training use case, supporting both vanilla prompt tuning and multi-task prompt tuning (MPT). Training metadata (loss curves) is persisted as JSONL files alongside model artifacts, enabling training monitoring. The `torch_run.py` utility configures PyTorch elastic launch with automatic GPU/CPU detection and NCCL/gloo backend selection, supporting single-node multi-GPU training scenarios.

A security-relevant observation is that model downloads from HuggingFace Hub are disabled by default (`allow_downloads: false`), which prevents arbitrary code execution from untrusted model repositories. Similarly, `trust_remote_code: false` is the default for embedding models, preventing execution of custom model code. These defaults align with the security posture expected for a RHOAI component, though the settings can be overridden via configuration. The TGIS backend connection supports mTLS (via `ca_cert_file`, `client_cert_file`, `client_key_file` configuration), but these are null by default and must be explicitly configured for production deployments.
