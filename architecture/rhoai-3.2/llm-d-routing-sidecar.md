## Purpose

**Short**: A reverse proxy sidecar that routes LLM inference requests between prefill and decode workers in a disaggregated prefill/decode (P/D) architecture.

**Detailed**: The llm-d-routing-sidecar is a Go-based HTTP reverse proxy that runs as a sidecar container alongside a vLLM decoder instance. It implements the disaggregated prefill/decode pattern for large language model serving, where the prefill (prompt processing) and decode (token generation) phases are separated across different GPU workers to optimize GPU utilization and throughput.

When a request arrives with an `x-prefiller-host-port` header, the sidecar intercepts it and orchestrates a two-phase protocol: first forwarding the request to the designated prefill worker (with modified parameters like `max_tokens=1` and KV transfer metadata), then forwarding the decode request (enriched with KV cache transfer parameters from the prefill response) to the co-located vLLM decoder. Requests without the prefill header are passed through directly to the local decoder. The sidecar supports three connector protocols (NIXL v1, NIXL v2, and LMCache), though only NIXL v2 is actively maintained.

The sidecar includes an optional SSRF (Server-Side Request Forgery) protection mechanism that watches Kubernetes InferencePool resources (from the Gateway API Inference Extension) and maintains a dynamic allowlist of valid prefill targets based on pod IPs matching the InferencePool selector. This prevents request forgery attacks where a malicious `x-prefiller-host-port` header could redirect traffic to arbitrary hosts. The project is noted as experimental and slated for removal in a future iteration of the llm-d architecture.

## Architectural Analysis

The llm-d-routing-sidecar implements a sidecar proxy pattern purpose-built for the disaggregated prefill/decode (P/D) architecture in LLM serving. Rather than treating prefill and decode as phases within a single inference server, this architecture splits them across separate GPU workers, allowing independent scaling and GPU memory optimization. The sidecar sits alongside each decoder pod and orchestrates the two-phase protocol transparently to the client.

The proxy architecture is deliberately minimal. It uses Go's standard `net/http/httputil.ReverseProxy` for proxying, with a small LRU cache (16 entries) to avoid recreating proxy handlers for frequently-accessed prefill workers. The buffered response writer (`status_response_writer.go`) captures the entire prefill response in memory to extract KV transfer parameters before constructing the decode request. This means the sidecar must hold the full prefill response in memory, which could be a concern for extremely large responses, though prefill responses are typically small (the prefill request is capped at `max_tokens=1`).

The SSRF protection mechanism is architecturally interesting. It uses Kubernetes informers to watch InferencePool CRDs (from the Gateway API Inference Extension, `inference.networking.x-k8s.io/v1alpha2`) and builds a dynamic allowlist of pod IPs and names. This prevents a malicious actor from setting the `x-prefiller-host-port` header to an arbitrary internal host. The allowlist tracks hosts/IPs only (not ports), so any port on an allowed host is permitted. The 30-second resync period (`resyncPeriod = 30 * time.Second`) balances responsiveness with API server load. Notably, the SSRF protection is optional and disabled by default for backward compatibility, which means deployments that don't enable it are vulnerable to SSRF via the prefill header.

The TLS configuration is thorough: the sidecar supports TLS on its own listener (with self-signed cert generation for testing), optional TLS for upstream connections to both prefiller and decoder, and explicit cipher suite restrictions (ECDHE+AES-GCM and ChaCha20-Poly1305 only, TLS 1.2 minimum). The `InsecureSkipVerify` flags for prefiller and decoder connections are available but disabled by default, which is appropriate for development/testing scenarios. The HTTP server configuration intentionally omits `ReadTimeout` and `WriteTimeout` because LLM inference can take hours for large contexts, while still setting `ReadHeaderTimeout` (30s) and `IdleTimeout` (5m) to prevent resource exhaustion.

The project is explicitly marked as experimental and scheduled for removal in a future iteration of the llm-d architecture, as noted in the README. The recent commit history (commit `4548428`) indicates that llm-d-routing-sidecar functionality is being absorbed into the `llm-d-inference-scheduler` project. The deployment manifests use a StatefulSet (rather than Deployment) pattern, which is somewhat unusual for a stateless proxy but may be driven by stable network identities needed for pod-to-pod communication in the P/D architecture.
