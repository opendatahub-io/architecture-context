## Purpose

**Short**: Downstream Konflux packaging repository that produces the RHOAI vLLM CUDA container image by re-layering the RHAIIS (Red Hat AI Inference Server) vLLM CUDA base image with the TGIS gRPC adapter entrypoint.

**Detailed**: This repository does not contain vLLM source code. It is a minimal downstream packaging repo in the `red-hat-data-services` GitHub organization that produces the `odh-vllm-cuda-rhel9` container image for Red Hat OpenShift AI (RHOAI). The image is built by Konflux CI/CD and is based on the `registry.redhat.io/rhaiis/vllm-cuda-rhel9` image from the Red Hat AI Inference Server (RHAIIS) product.

The image adds a TGIS (Text Generation Inference Server) gRPC adapter layer on top of the upstream vLLM HTTP server. This adapter provides a gRPC interface (port 8033) that is compatible with the KServe/ServingRuntime inference protocol used in RHOAI, while the standard vLLM OpenAI-compatible HTTP API remains available on port 8000. The container runs as non-root user (UID 2000) and is built for both amd64 and arm64 architectures.

The actual vLLM engine, TGIS adapter code, CUDA libraries, and all Python dependencies are provided by the RHAIIS base image. This repository's role is limited to: (1) pinning the RHAIIS version, (2) setting environment variables and the entrypoint, (3) adding container labels, and (4) defining the Konflux/Tekton build pipeline.

## Architectural Analysis

This repository is architecturally trivial — it is a thin re-layering wrapper around the RHAIIS (Red Hat AI Inference Server) vLLM CUDA image. The entire vLLM inference engine, the TGIS gRPC adapter, CUDA libraries, Python runtime, and all dependencies are provided by the base image at `registry.redhat.io/rhaiis/vllm-cuda-rhel9:3.2.2`. This repository's contribution is limited to setting the entrypoint, environment variables, container labels, and build pipeline configuration.

The key architectural decision visible in this repository is the TGIS adapter integration. The entrypoint (`python3 -m vllm_tgis_adapter --uvicorn-log-level=warning`) launches the TGIS adapter module, which wraps the vLLM engine with a gRPC interface on port 8033 alongside the standard vLLM OpenAI-compatible HTTP API on port 8000. The `DISABLE_LOGPROBS_DURING_SPEC_DECODING=false` environment variable overrides a vLLM optimization that disables logprobs during speculative decoding, ensuring hosted models behave predictably regardless of internal optimization strategies.

The non-hermetic build (`hermetic: false` in Tekton pipeline) is notable from a supply chain perspective. This is intentional — since the Dockerfile performs no package installation and simply re-layers an already-built RHAIIS image, hermeticity at this layer would add complexity without security benefit. The actual supply chain security boundary is at the RHAIIS image build. However, this means that FIPS compliance, dependency pinning, and vulnerability scanning for the application layer must all be validated at the RHAIIS image level, not here.

The RHOAI 3.2 release (this branch) represents a significant shift from earlier versions where vLLM was built from source in this repository. The commit history shows the transition: earlier commits reference `argfile.conf`, TGIS adapter environment variables, and direct vLLM version pinning. The September 2025 commit `e927dff02` ("Integrate AIPCC to consume vLLM CUDA from RHAIIS releases") marks the pivot to consuming the pre-built RHAIIS image. This architectural change simplifies the downstream build but centralizes all supply chain responsibility in the RHAIIS product team.
