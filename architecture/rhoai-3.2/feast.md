## Purpose

**Short**: Feast is an open-source feature store for machine learning that manages feature definitions, storage, and serving for both online (low-latency) and offline (batch) feature retrieval.

**Detailed**: Feast (Feature Store) provides a complete feature management platform for ML/AI workflows on Red Hat OpenShift AI. The repository contains three primary components: a Go-based Kubernetes operator that manages the lifecycle of Feast deployments via a `FeatureStore` CRD, a Python SDK that implements multiple feature serving protocols (REST, gRPC, Apache Arrow Flight), and a Go-based online feature server optimized for low-latency serving with Apache Arrow columnar data handling.

The Feast operator reconciles `FeatureStore` custom resources to deploy and configure the feature serving infrastructure — including offline stores (for historical feature retrieval), online stores (for real-time serving), a feature registry (for metadata management), and an optional web UI. The operator handles TLS certificate management (with OpenShift service serving certificate integration), Kubernetes RBAC and OIDC-based authorization, PVC-backed persistence, CronJob-based scheduled operations, and integration with OpenDataHub/RHOAI notebooks via automatic ConfigMap injection.

The Python feature server exposes a FastAPI-based REST API for online feature retrieval and materialization, a gRPC registry server for CRUD operations on feature definitions, and an Apache Arrow Flight server for efficient offline feature access. The Go feature server provides an alternative high-performance serving path with gRPC and HTTP endpoints, backed by Redis, DynamoDB, or SQLite online stores.

## Architectural Analysis

The Feast repository is a multi-language monorepo that ships two distinct Konflux images — a Go operator and a Python feature server — each serving a fundamentally different role. The operator is a standard controller-runtime operator managing a single CRD (`FeatureStore`), while the Python image bundles three independent server types (REST, gRPC, Arrow Flight) into a single container. This design means the feature server pod runs multiple protocols simultaneously, configured by the `feature_store.yaml` that the operator generates and mounts as a ConfigMap.

The TLS architecture is particularly well-integrated with OpenShift. When running on OpenShift, the operator automatically detects the platform and configures TLS via service serving certificate annotations — no manual certificate management required. For the registry service, which exposes both gRPC (port 6570) and REST (port 6572) endpoints, the operator creates a single TLS certificate with both service hostnames as Subject Alternative Names, avoiding certificate duplication. The operator also integrates with the ODH trusted CA bundle ConfigMap (`odh-trusted-ca-bundle`) for custom CA chain support, mounting it into all Feast pods.

The authorization model is dual-mode: Kubernetes RBAC or OIDC. When Kubernetes auth is enabled, the operator creates a complete RBAC hierarchy — ServiceAccount, Role, RoleBinding, ClusterRole, and ClusterRoleBinding — and the feature server validates tokens via the Kubernetes TokenReview API. The Python permission system is remarkably fine-grained, defining 8 distinct actions (CREATE, DESCRIBE, UPDATE, DELETE, READ_ONLINE, READ_OFFLINE, WRITE_ONLINE, WRITE_OFFLINE) with regex-based resource name matching and tag-based filtering. This allows operators to define permissions like "user X can read online features from feature views matching `prod-*` but cannot write."

The Go feature server (`go/main.go`) is architecturally separate from the operator — it shares the repository but is a standalone binary that serves online features with Apache Arrow columnar data handling. It supports Redis, DynamoDB, and SQLite online store backends, with OpenTelemetry tracing integration. The Go server is primarily used as an embedded component or alternative high-performance serving path, while the Python server is the default deployment target for the operator.

The NotebookConfigMapReconciler represents an interesting integration pattern with OpenDataHub. When a Notebook CR is labeled with `opendatahub.io/feast-integration=true` and annotated with `opendatahub.io/feast-config` listing comma-separated project names, the controller automatically creates a ConfigMap containing the Feast client configuration for those projects. This enables zero-configuration Feast access from Jupyter notebooks without manual YAML copying.
