## Purpose

**Short**: Kubernetes-native operator for managing distributed machine learning training jobs across multiple frameworks (PyTorch, MPI, DeepSpeed, JAX, TorchTune, MLX).

**Detailed**: Kubeflow Trainer is a Kubernetes operator that manages the lifecycle of distributed ML training workloads. It provides a declarative API (TrainJob, TrainingRuntime, ClusterTrainingRuntime) for defining, scheduling, and executing training jobs across multiple ML frameworks. Users create a TrainJob referencing a TrainingRuntime that defines the infrastructure blueprint (framework, scheduling, node count), and the operator translates this into JobSet resources with appropriate environment variables, networking, and scheduling constraints.

The RHOAI distribution extends upstream Kubeflow Trainer with Red Hat-specific features including RHAI progression tracking (real-time training metrics polling), FIPS-compliant builds, RHOAI-specific ClusterTrainingRuntime definitions (torch-distributed, training-hub, ROCm variants), and Prometheus PodMonitor integration. The operator is deployed in the `opendatahub` namespace and is managed by the platform operator (rhods-operator) as a component of the RHOAI training stack.

The operator's plugin-based architecture supports extensible ML runtimes: the Torch plugin configures distributed PyTorch training (torchrun, TorchTune), the MPI plugin manages SSH key distribution and hostfile generation for MPI workloads, and co-scheduling plugins (Kueue scheduler-plugins, Volcano) enforce gang scheduling. A data-cache subsystem provides distributed Iceberg-based dataset caching via a Rust head/worker architecture using LeaderWorkerSets.

## Architectural Analysis

The Kubeflow Trainer operator follows a clean plugin-based architecture where the core controller framework is decoupled from ML-framework-specific logic. The TrainJob controller delegates resource construction to a chain of plugins: first the ML policy plugin (Torch, MPI, or PlainML) configures framework-specific parameters, then the PodGroup policy plugin (CoScheduling or Volcano) adds scheduling constraints, followed by pod network configuration, and finally the component builder (JobSet plugin) assembles the Kubernetes objects. This pipeline ensures that adding a new ML framework requires only implementing a new plugin without modifying the core reconciliation logic.

The RHAI progression tracking feature is a notable RHOAI-specific extension that adds real-time training observability without modifying the upstream API. It uses annotations (`trainer.opendatahub.io/*`) to opt in and store metrics, avoiding CRD schema changes. The implementation polls training pods via HTTP (default port 28080) during reconciliation, with a three-priority fallback for final state capture: (1) container termination messages, (2) HTTP polling, (3) last-known annotation state. The controller uses `APIReader` (direct API calls) rather than cached informers for pod lookups, which avoids setting up pod watches across all namespaces — a deliberate trade-off of slightly higher API server load per poll versus dramatically lower memory consumption.

The operator's security model relies on Kubernetes-native mechanisms: validation webhooks enforce immutability of critical fields (runtimeRef, managedBy), self-managed TLS certificates avoid external cert-manager dependencies, and HTTP/2 is disabled by default to mitigate CVE-2023-44487 and CVE-2023-39325. The MPI plugin generates ECDSA P521 SSH key pairs per TrainJob as immutable Secrets, preventing key rotation during training (which would break MPI communication). The FIPS compliance posture is strong at the build level (CGO_ENABLED=1, GOEXPERIMENT=strictfipsruntime on UBI9) and the application avoids non-FIPS crypto patterns, though the plaintext HTTP polling for progression tracking is a design choice accepting that pod-to-controller communication occurs within the cluster network.

The kustomize manifest structure cleanly separates base resources from distribution-specific overlays. The RHOAI overlay (`manifests/rhoai/`) uses `configMapGenerator` + `replacements` (not legacy `vars`) for image parameterization, which is the modern kustomize pattern. It adds five RHOAI-specific ClusterTrainingRuntimes covering CUDA, ROCm (AMD GPU), and Training Hub variants. The `params.env` file provides the image substitution values consumed by the platform operator during deployment.

The data-cache subsystem is architecturally distinct from the main operator — it's a Rust service using Apache Arrow Flight for distributed query execution over Iceberg tables, deployed via LeaderWorkerSet. This is an optional component (enabled via the data-cache overlay) that provides dataset pre-caching for training jobs, reducing I/O bottlenecks during distributed training. Its integration with the main operator is loose: the cache initializer runs as a separate replicated job in the JobSet with a dependency ordering (initializer completes before trainer nodes start).
