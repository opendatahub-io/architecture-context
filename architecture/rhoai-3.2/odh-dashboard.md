## Purpose
**Short**: The ODH Dashboard is the web-based management UI for Red Hat OpenShift AI, providing a central interface for managing data science workloads, notebooks, model serving, model registries, and AI/ML platform configuration.

**Detailed**: The ODH Dashboard (deployed as `rhods-dashboard` in RHOAI) is a full-stack web application consisting of a Fastify-based Node.js backend that serves a React/PatternFly frontend and proxies Kubernetes API requests. The application provides a unified management interface for the OpenShift AI platform, enabling users to launch Jupyter notebooks, manage model serving runtimes and inference services, configure data connections, manage hardware profiles, and interact with model registries.

The dashboard uses a modular federation architecture where specialized UI plugins (model-registry-ui and gen-ai-ui) run as separate containers within the same Pod, each serving federated JavaScript modules that are dynamically loaded by the main dashboard frontend. These federated modules communicate through their own HTTPS-secured ports and are proxied through the main backend for API traffic.

In RHOAI 3.x, the dashboard adopts the Gateway API ingress pattern, using an HTTPRoute CR that references the platform `data-science-gateway` Gateway in `openshift-ingress`, with a `kube-rbac-proxy` sidecar handling authentication and authorization via SubjectAccessReview before proxying to the main application container. The backend serves as a BFF (Backend-for-Frontend) that proxies Kubernetes API calls and WebSocket watch connections on behalf of the authenticated user.

### Ingress

| Name | Type | Hosts | Port | Protocol | Encryption | TLS Mode | Exposure |
|------|------|-------|------|----------|------------|----------|----------|
| rhods-dashboard | HTTPRoute (Gateway API) | Auto-assigned by Gateway | 8443/TCP | HTTPS | TLS (Gateway-terminated + re-encrypt to backend) | PASSTHROUGH to kube-rbac-proxy | External |

**Ingress Details (RHOAI 3.x Pattern)**:
- The dashboard creates an `HTTPRoute` CR (Gateway API v1) referencing the `data-science-gateway` Gateway in `openshift-ingress` namespace
- The HTTPRoute matches all paths (`/`) and routes to the `rhods-dashboard` Service on port 8443
- Traffic flows: Client → Gateway (openshift-ingress) → HTTPRoute → Service:8443 → kube-rbac-proxy container
- kube-rbac-proxy performs SubjectAccessReview (checking `project.openshift.io/v1/projects` list permission) before proxying to the dashboard backend on localhost:8080
- kube-rbac-proxy injects `X-Auth-Request-User` and `X-Auth-Request-Groups` headers for the backend

**Legacy Route (still present in manifests but NOT deployed via RHOAI onprem overlay)**:
- A Route CR exists in `core-bases/base/routes.yaml` with reencrypt TLS termination, but it is NOT included in the `core-bases/base/kustomization.yaml` resources list
- The Route is effectively unused in the current RHOAI 3.x deployment

### Flow 1: User Accesses Dashboard UI

| Step | Source | Destination | Port | Protocol | Encryption | Auth |
