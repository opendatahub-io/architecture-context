## Purpose

**Short**: Web dashboard UI and API backend for Red Hat OpenShift AI, providing a unified management interface for notebooks, model serving, pipelines, model registry, and platform configuration.

**Detailed**: The odh-dashboard component is the primary user-facing web application for the Red Hat OpenShift AI (RHOAI) platform. It implements a modular architecture consisting of a React/PatternFly frontend served by a Node.js (Fastify) backend, with optional Go-based Backend-for-Frontend (BFF) sidecar modules deployed via Webpack Module Federation.

The Fastify backend acts as the main BFF layer — it proxies authenticated requests to the Kubernetes API, Data Science Pipelines, Model Registry, TrustyAI, MLMD, and Prometheus/Thanos services. It manages dashboard configuration via the `OdhDashboardConfig` CRD, watches platform resources for real-time state updates, and enforces authorization through kube-rbac-proxy and namespace-scoped RBAC checks.

The modular architecture enables feature modules (Gen AI, Model Registry, MaaS) to be developed independently as Go+React packages, each producing a separate container image that runs as a sidecar within the dashboard pod. These sidecar BFFs communicate with their respective upstream services (LlamaStack, Model Registry API, Kubernetes API) and serve federated React micro-frontends that are dynamically loaded by the host application at runtime. This architecture allows RHOAI to ship feature-specific UI and API capabilities as independently versioned container images while sharing the PatternFly design system, React runtime, and authentication infrastructure.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| Frontend (React/PatternFly) | React SPA | Host application providing the dashboard UI with module federation support for loading plugin remotes |
| Backend (Fastify/Node.js) | Node.js BFF | Main API backend serving the frontend, proxying K8s API calls, managing dashboard CRDs, and providing WebSocket connections |
| Gen AI BFF | Go HTTP Service | Sidecar BFF for generative AI features — proxies to LlamaStack, manages vector stores, MCP tool integration, and AI response streaming |
| Model Registry BFF | Go HTTP Service | Sidecar BFF for model registry features — proxies to Model Registry REST API, manages registered models/versions/artifacts, and model catalog access |
| MaaS BFF | Go HTTP Service | Sidecar BFF for Model-as-a-Service features — manages tier configuration via Kubernetes ConfigMaps |
| kube-rbac-proxy | Auth Sidecar | Authentication/authorization proxy fronting the dashboard on port 8443, enforcing OpenShift project list RBAC before proxying to the application |

## Architectural Analysis

The odh-dashboard implements a sophisticated multi-tier architecture that has evolved from a simple React+Node.js application into a modular platform with Webpack Module Federation at its core. The primary architectural decision — running feature-specific Go BFF sidecars alongside the main Node.js backend within a single pod — provides strong isolation between feature domains while sharing authentication, TLS, and network identity. Each BFF module (Gen AI, Model Registry, MaaS) is independently versioned and built via Konflux, enabling teams to ship UI+API changes without coordinating dashboard releases.

The authentication flow is notable for its depth: external requests pass through the platform Gateway (Envoy), then kube-rbac-proxy which validates that the user can list OpenShift projects (a baseline authorization check), then the Fastify backend which extracts the user identity from X-Auth-Request-User/Groups headers and performs namespace-scoped ownership checks. For BFF sidecars, the user's access token is forwarded via the x-forwarded-access-token header, and each Go BFF uses it as a Kubernetes bearer token for all downstream API calls — ensuring that authorization is always evaluated against the requesting user's permissions, never the service account's.

The Fastify backend acts as a universal proxy layer — it can proxy to any upstream service by discovering it through Kubernetes service resources or DSPipeline/TrustyAI CR status fields. This dynamic service discovery pattern (implemented in `backend/src/utils/proxy.ts`) means the dashboard doesn't require hardcoded service URLs; it uses the K8s API to find services at runtime. The backend also maintains in-memory caches of watched resources (OdhDashboardConfig, applications, documents, quick starts, console links) with real-time updates via Kubernetes watch API, avoiding redundant API calls.

The Module Federation architecture uses a ConfigMap (`federation-config`) to define which remote modules are available, their service endpoints, and their proxy configurations. The host application (frontend) dynamically loads `remoteEntry.js` from each sidecar, while the Fastify backend registers proxy routes for each module's API path prefix (e.g., `/model-registry/api` → sidecar port 8043, `/gen-ai/api` → sidecar port 8143). Shared singletons (React, ReactDOM, PatternFly, plugin-core) ensure consistent UI behavior across federated boundaries.

The NetworkPolicy applied in the modular-architecture overlay explicitly allows ingress to ports 8443 (kube-rbac-proxy), 8043 (Model Registry BFF), and 8143 (Gen AI BFF) from all sources, while the pod's anti-affinity rule ensures dashboard replicas are spread across availability zones. The deployment runs 2 replicas with significant resource allocations (1 CPU / 2GB per container × 4 containers = 4 CPU / 8GB total per replica), reflecting the dashboard's role as a critical user-facing service.
