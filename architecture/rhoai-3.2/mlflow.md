## Purpose

**Short**: MLflow is an open-source ML lifecycle management platform providing experiment tracking, model versioning, LLM observability, and artifact management, adapted for Red Hat OpenShift AI with Kubernetes-native workspace isolation and RBAC.

**Detailed**: MLflow serves as the experiment tracking and model registry component for the RHOAI platform. It provides a hybrid Flask/FastAPI server that exposes REST and GraphQL APIs for managing ML experiments, runs, metrics, model versions, scorers, and artifacts. The server also serves a React-based web UI for interactive exploration.

In the RHOAI distribution, MLflow is extended with a Kubernetes workspace provider plugin (`mlflow-kubernetes-workspace-provider`) that maps MLflow workspaces 1:1 to Kubernetes namespaces and enforces authorization via Kubernetes RBAC (SelfSubjectAccessReview or SubjectAccessReview APIs). This enables multi-tenant workspace isolation where each namespace represents a separate MLflow workspace, with access control delegated to the Kubernetes API server.

The platform is built as a single container image (`odh-mlflow-rhel9`) using a multi-stage Dockerfile: a Node.js stage builds the React frontend, a Python stage builds the MLflow and workspace provider wheels, and a final UBI9 Python 3.11 runtime stage installs everything and exposes the server on port 5000.

## Architectural Analysis

MLflow for RHOAI represents a significant adaptation of the upstream open-source project. The core MLflow server is a monolithic Python application built on a hybrid Flask/FastAPI architecture — Flask handles the majority of REST endpoints and serves the React UI, while FastAPI wraps the Flask app as a WSGI sub-application and provides native async routes for OTLP trace ingestion and job management. This layered approach allows the team to incrementally migrate from Flask to FastAPI without breaking backward compatibility.

The most architecturally significant RHOAI addition is the `kubernetes-workspace-provider` plugin, which implements multi-tenant workspace isolation by mapping MLflow workspaces 1:1 to Kubernetes namespaces. This plugin operates at two layers: (1) a `WorkspaceProvider` that discovers namespaces via a background watch loop and exposes them as workspaces, filtering by label selectors and glob exclusion patterns; and (2) an authorization middleware that intercepts every API request, resolves the target workspace/namespace, and validates permissions via Kubernetes RBAC (SubjectAccessReview or SelfSubjectAccessReview). The authorization decisions are cached with a 5-minute TTL using a custom read-write lock to avoid excessive K8s API calls. This design is elegant — it delegates all access control to the Kubernetes control plane, meaning MLflow inherits whatever RBAC policies are already defined for namespaces without maintaining a parallel permission system.

The auth plugin supports two authorization modes: `SelfSubjectAccessReview` (default), where users present Bearer tokens and the K8s API checks their own permissions; and `SubjectAccessReview`, where a proxy (such as kube-rbac-proxy) injects identity via `x-remote-user`/`x-remote-groups` headers and MLflow checks on their behalf. The SubjectAccessReview mode is the expected pattern in RHOAI 3.x, where kube-rbac-proxy sits in front of the application as a sidecar.

A notable build concern is the non-hermetic Tekton pipeline — `hermetic: false` is explicitly set, meaning Python packages are fetched from PyPI during the build. While the `uv.lock` file provides reproducibility, the lack of Hermeto/cachi2 prefetch means the supply chain is not fully locked. Additionally, no `rpms.lock.yaml` exists on this branch for OS package pinning. These are expected gaps for a component this early in its RHOAI integration lifecycle, but they represent supply chain risks that should be addressed before GA.

The frontend is a React 18 single-page application built with CRACO (Create React App Configuration Override), Apollo Client for GraphQL, and AG Grid for data tables. It is compiled at image build time by a separate Node.js 20 builder stage and served as static files by the Python backend with aggressive cache headers (28-day max-age with hash-based busting).
