## Purpose
**Short**: REST API orchestrator that coordinates AI text generation with content safety guardrails by invoking detector, chunker, and LLM services.

**Detailed**: The FMS Guardrails Orchestrator (fms-guardrails-orchestr8) is a Rust-based middleware service that sits between clients and language models to enforce content safety. It receives text generation requests, coordinates with configurable detector services to identify harmful content, manages text chunking for efficient detection, and forwards approved requests to LLM backends (TGIS or caikit-nlp via gRPC, or OpenAI-compatible APIs via HTTP).

The orchestrator supports both unary and streaming request patterns, including Server-Sent Events (SSE) for streaming responses and ND-JSON for streaming content detection. It provides a comprehensive v1 and v2 API surface covering classification-with-generation, standalone text detection (content, chat, context documents, generated text), and OpenAI-compatible chat completions and completions endpoints with integrated detection.

The service is built with Axum (HTTP framework), Tonic (gRPC client), and Tokio (async runtime). It supports TLS and mTLS for both inbound server connections and outbound client connections to detectors, chunkers, and generation services. OpenTelemetry integration provides observability through traces and metrics export via gRPC or HTTP protocols.

## Security

### TLS Configuration

The orchestrator supports comprehensive TLS configuration for both server-side and client-side connections:

**Server-side TLS** (inbound connections on port 8033):
- Configured via `TLS_KEY_PATH` and `TLS_CERT_PATH` environment variables
- mTLS enabled by additionally providing `TLS_CLIENT_CA_CERT_PATH`
- Uses rustls with ring crypto provider
- Supports TLS 1.2+

**Client-side TLS** (outbound connections to backends):
- Per-service TLS configuration in `config.yaml`
- Named TLS configurations for reuse across services
- Supports cert paths, key paths, CA cert paths
- Insecure mode available (skip server verification) for development

**Note**: The health server (port 8034) always runs without TLS on a separate listener.
