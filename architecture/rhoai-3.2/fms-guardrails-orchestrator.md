## Purpose

**Short**: A REST API orchestrator that coordinates AI text generation with content safety guardrails by managing detectors, chunkers, and LLM backends.

**Detailed**: The FMS Guardrails Orchestrator is a high-performance Rust service that acts as middleware between clients and language models, enforcing content safety through configurable detection pipelines. It receives text generation or detection requests via HTTP REST APIs and orchestrates calls to multiple downstream services: text generation backends (TGIS or caikit-nlp via gRPC, or OpenAI-compatible via HTTP), content safety detector services (via HTTP), and text chunker services (via gRPC).

The orchestrator supports both unary and streaming request patterns. For streaming, it implements a sophisticated detection pipeline using an actor-based batching system that processes detection results from multiple detectors concurrently. Input text can be checked for harmful content before generation, and generated output can be checked during or after generation. When harmful content is detected above configurable thresholds, the orchestrator returns warnings or blocks the content.

The service is designed for deployment in Red Hat OpenShift AI (RHOAI) as part of the TrustyAI stack. It supports TLS and mTLS for all connections, OpenTelemetry-based distributed tracing and metrics, configurable header passthrough for authentication token propagation, and graceful shutdown handling.

## Architectural Analysis

The FMS Guardrails Orchestrator follows a classic middleware/gateway pattern implemented in Rust for maximum throughput and low latency. The dual-server architecture separates health endpoints (port 8034) from the main API surface (port 8033), enabling independent health monitoring without impacting API traffic. This is a common cloud-native pattern that allows Kubernetes readiness/liveness probes to operate on a separate port.

The orchestration layer uses a handler trait pattern (`Handle<Task>`) where each API endpoint maps to a specific handler implementation. Handlers follow a consistent three-phase pattern: optional input detection (with early exit on violations), generation (for endpoints that include LLM interaction), and optional output detection. This separation of concerns makes the detection pipeline composable and extensible. The streaming handlers use Tokio's async concurrency primitives (mpsc channels, `tokio::spawn`, `DashMap`) to process detection results from multiple detectors in parallel, with a custom actor-based batching system (`DetectionBatcherManager`) that synchronizes results before returning them to clients.

A notable architectural decision is the use of `ring` as the exclusive cryptographic provider for rustls, rather than the aws-lc-rs backend or OpenSSL. While this simplifies the build (no C dependencies for crypto), it means the application does not use FIPS-validated cryptography despite OpenSSL being installed in the container image. The `insecure` TLS configuration option (which disables certificate verification via a custom `NoVerifier` implementation) is available for development use but represents a security risk if enabled in production.

The client layer abstracts over two fundamentally different generation backends: TGIS (IBM's Text Generation Inference Server) via gRPC, and OpenAI-compatible APIs (such as vLLM) via HTTP. The `GenerationClient` adapter pattern wraps both with a unified interface including retry logic with exponential backoff for transient failures (status codes 502-506). The gRPC clients use `ginepro`'s `LoadBalancedChannel` for DNS-based service discovery with configurable eager/lazy resolution strategies, enabling transparent failover across replicas.

The header passthrough mechanism is architecturally significant: the orchestrator can forward configured HTTP headers (including rewriting `X-Forwarded-Access-Token` to `Authorization: Bearer`) to downstream services. This enables token-based authentication propagation through the orchestration layer without the orchestrator itself performing authentication, delegating that responsibility to the backend services or to an external authentication proxy (e.g., kube-rbac-proxy) in the RHOAI deployment.
