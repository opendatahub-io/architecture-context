## Purpose

**Short**: An HTTP gateway that provides OpenAI-compatible chat completion endpoints with configurable detector-based content filtering, routing requests to the FMS Guardrails Orchestrator.

**Detailed**: The vLLM Orchestrator Gateway is a Rust-based HTTP service that sits between clients and the FMS Guardrails Orchestrator, enabling controlled interaction through configurable content detection pipelines. It exposes multiple OpenAI-compatible `/v1/chat/completions` endpoints — one per configured route — each with a fixed set of content detectors (e.g., PII regex detection). This allows users to drop in a guardrailed endpoint as a replacement for an unguardrailed chat completions model.

The gateway reads a YAML configuration file that defines the upstream orchestrator connection, available detectors (with input/output classification and parameters), and named routes that map detector sets to unique HTTP paths. When a request arrives at a route endpoint, the gateway enriches the payload with the route's detector configuration and forwards it to the orchestrator's `/api/v2/chat/completions-detection` endpoint. If detections are found in the response, the gateway can substitute a configurable fallback message. Both streaming (SSE) and non-streaming response modes are supported.

The service supports mTLS for secure communication with the orchestrator, loading TLS certificates from Kubernetes-mounted paths (`/etc/tls/private/` for client certs, `/etc/tls/ca/` for CA certificates). It forwards `Authorization` and `X-Forwarded-*` headers from incoming requests to the orchestrator, preserving authentication context.

## Architectural Analysis

The vLLM Orchestrator Gateway follows a straightforward reverse proxy pattern with request enrichment. Its core value proposition is translating the concept of "named detector pipelines" into OpenAI-compatible endpoints that clients can use without understanding the underlying guardrails infrastructure. Each route defined in `config.yaml` becomes a `POST /{route_name}/v1/chat/completions` endpoint, making it trivial to swap between guardrailed and unguardrailed model access by simply changing the base URL.

The routing architecture is dynamic — routes are constructed from config at startup via a loop in `main.rs` (lines 84-109), with each route capturing its own detector list, fallback message, and a shared `Arc<reqwest::Client>` for orchestrator communication. This is a clean design that avoids a centralized routing table and leverages Rust's ownership model to ensure each route handler has isolated, immutable copies of its configuration. However, the config is read once at startup with no reload mechanism — configuration changes require a pod restart.

The TLS implementation (`build_orchestrator_client`, lines 351-406) is noteworthy for its approach: it uses OpenSSL's PEM parsing to load certificates, then converts them to a PKCS#12 archive in-memory (with no password) so they can be consumed by reqwest's `Identity` API. This indirect path exists because reqwest's native-tls backend expects PKCS#12 identities, not separate PEM cert/key pairs. The code also includes a `danger_accept_invalid_hostnames(true)` flag when the orchestrator hostname is `localhost`, which is appropriate for local development but should not fire in production since the orchestrator's Kubernetes Service DNS name would be used.

The gateway has a notable lack of health check endpoints — there is no `/healthz`, `/readyz`, or similar endpoint. This means Kubernetes liveness/readiness probes would need to be configured as TCP checks against port 8090 rather than HTTP checks. Additionally, there is no metrics endpoint (e.g., Prometheus `/metrics`), which limits observability in production. The service also has no rate limiting, circuit breaking, or retry logic for orchestrator communication — a single failed orchestrator request results in an immediate 500 error to the client.

The streaming implementation (lines 259-348) parses SSE chunks from the orchestrator, applies fallback message substitution if detections are found, and re-emits them as SSE events to the client. This works for the detection use case but the SSE parsing is fragile — it assumes each `bytes_stream()` chunk maps cleanly to SSE frames, which may not hold under network fragmentation. The `[DONE]` sentinel is stripped but empty data lines are passed through as empty strings.
