## Purpose
**Short**: The RHOAI (Red Hat OpenShift AI) operator manages the lifecycle of all AI/ML platform components on OpenShift, including Gateway API ingress infrastructure, authentication, monitoring, and component deployment.

**Detailed**: The rhods-operator is the central platform operator for Red Hat OpenShift AI (RHOAI) 3.x. It orchestrates the complete AI/ML platform lifecycle through two primary custom resources: DSCInitialization (DSCI) for platform-level initialization (monitoring, CA bundles, namespace setup) and DataScienceCluster (DSC) for managing individual AI/ML components such as Dashboard, KServe, Workbenches, Model Registry, Data Science Pipelines, Ray, Kueue, TrustyAI, Training Operator, Feast Operator, LlamaStack Operator, and MLflow Operator.

In RHOAI 3.x, the operator deploys critical Gateway API ingress infrastructure via its GatewayConfig controller. This creates a `data-science-gateway` Gateway resource in the `openshift-ingress` namespace, along with a `kube-auth-proxy` deployment for centralized OAuth2/OIDC authentication, EnvoyFilter for request interception, and HTTPRoute CRs for routing. Component controllers (in downstream components like notebook-controller) create HTTPRoute CRs that reference this gateway. The operator supports two ingress modes: OcpRoute (ClusterIP + OpenShift Routes) and LoadBalancer.

The operator also manages platform-wide Auth (RBAC group management), Monitoring (MonitoringStack/Prometheus/Tempo integration), and certificate management. It uses a plugin-based architecture where components and services register themselves via init() functions and are reconciled independently.

**Gateway API Architecture (RHOAI 3.x)**:
- The Gateway controller creates a `data-science-gateway` Gateway CR in `openshift-ingress` namespace
- A `data-science-gateway-class` GatewayClass uses the `openshift.io/gateway-controller/v1` controller
- The Gateway terminates TLS on port 443 using certificates from OpenShift default ingress, self-signed, or user-provided
- An `EnvoyFilter` (`data-science-authn-filter`) intercepts all requests through the Gateway and routes them to `kube-auth-proxy` for authentication via ext_authz
- A Lua filter in the EnvoyFilter strips OAuth2 cookies and converts them to Bearer tokens for upstream services
- A `DestinationRule` (`data-science-tls-rule`) configures mTLS for backend service connections on ports 8443 and 443
- Component controllers in downstream repos (e.g., notebook-controller) create HTTPRoute CRs referencing this gateway as parent
- The kube-auth-proxy supports two authentication modes: OpenShift OAuth (IntegratedOAuth) and external OIDC
- **OcpRoute ingress mode**: Gateway uses ClusterIP service, and an OpenShift Route is created in front with reencrypt TLS termination
- **LoadBalancer ingress mode**: Gateway uses LoadBalancer service type, hostname set directly on Gateway listener
