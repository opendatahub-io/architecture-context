## Purpose

**Short**: The RHOAI platform operator that orchestrates the entire Red Hat OpenShift AI platform — managing component lifecycle, ingress infrastructure, authentication, monitoring, and RBAC across the cluster.

**Detailed**: The rhods-operator (also known as opendatahub-operator) is the central platform operator for Red Hat OpenShift AI (RHOAI). It serves as the single control plane for deploying, configuring, and managing the complete AI/ML platform on OpenShift. The operator manages two primary custom resources: `DSCInitialization` (cluster-wide initialization including namespaces, monitoring, auth, gateway, and trusted CA bundles) and `DataScienceCluster` (component lifecycle management for 14 AI/ML components).

The operator deploys a sophisticated ingress stack using Kubernetes Gateway API with an Envoy-based data plane, kube-auth-proxy for OAuth2/OIDC authentication, and EnvoyFilter-based traffic processing. It manages the full lifecycle of platform components including Dashboard, Workbenches (Jupyter notebooks), Data Science Pipelines, KServe (model serving), Kueue (job queuing), Ray, TrustyAI, Model Registry, Training Operator, Trainer, Feast Operator, LlamaStack Operator, and MLflow Operator. Each component is deployed via kustomize manifests stored in `prefetched-manifests/` and rendered dynamically based on platform type (Managed RHOAI, Self-Managed RHOAI, or ODH).

The operator also provides platform services: an Auth service for RBAC group management, a Monitoring service for Prometheus/Tempo/OpenTelemetry observability stack deployment, a CertConfigMapGenerator for trusted CA bundle distribution, and a Setup service for operator lifecycle management. All controllers use a registry-based plugin architecture where components self-register via Go `init()` functions and are discovered at startup.

## Architectural Analysis

### Registry-Based Plugin Architecture

The operator uses a sophisticated registry pattern where each component and service self-registers at Go startup time via `init()` functions. The component registry (`internal/controller/components/registry/registry.go`) and service registry (`internal/controller/services/registry/registry.go`) maintain ordered lists of `ComponentHandler` and `ServiceHandler` interfaces. Each handler implements `Init()`, `GetName()`, `NewCRObject()`, `NewComponentReconciler()`, `UpdateDSCStatus()`, and `IsEnabled()`. This pattern enables adding new components with minimal boilerplate — a new component only needs to implement the handler interface and call `cr.Add()` in its `init()` function. The DSC controller iterates the registry to create component CRs when components are enabled, and the main entrypoint iterates the registry to create reconcilers. This is an effective plugin architecture that keeps the core operator logic decoupled from individual component logic.

### Multi-Layer Ingress Architecture (RHOAI 3.x)

The Gateway controller deploys one of the most sophisticated ingress stacks in the Kubernetes ecosystem. The architecture has three layers: (1) **Gateway API layer** — a GatewayClass and Gateway resource managed by the OpenShift Gateway Controller (Istio-based), which deploys an Envoy proxy pod that terminates TLS on port 443; (2) **Authentication layer** — a kube-auth-proxy Deployment running OAuth2 Proxy, called by Envoy's ext_authz filter before allowing any request through, supporting both OpenShift Integrated OAuth and external OIDC providers; (3) **Traffic processing layer** — an EnvoyFilter CR injecting ext_authz and Lua filters into the Envoy proxy, plus a DestinationRule for mTLS configuration. The Lua filter strips OAuth2 proxy cookies from upstream requests and injects Bearer tokens. This three-layer architecture provides centralized authentication for all RHOAI components. The controller supports two ingress modes: LoadBalancer (cloud-native, external LB) and OcpRoute (OpenShift-native, using Routes with reencrypt TLS termination), with automatic detection of existing infrastructure.

### Component Lifecycle Management via Server-Side Apply

All resources are deployed using Kubernetes Server-Side Apply (SSA), which enables declarative, conflict-free resource management. The operator owns fields it manages and doesn't interfere with fields managed by other controllers. This is critical for a platform operator that creates resources across many namespaces and interacts with resources owned by other operators (e.g., OLM-managed CSVs, Istio-managed Envoy sidecars). The garbage collection action (`gc`) ensures resources that are no longer part of the desired state are cleaned up automatically based on ownership labels.

### Platform-Aware Build System

The operator binary is compiled with build tags (`-tags rhoai`) that conditionally include platform-specific API type definitions. The `DSCInitializationSpec` has different fields for ODH vs RHOAI (e.g., RHOAI defaults `ApplicationsNamespace` to `redhat-ods-applications`). This compile-time platform selection avoids runtime branching for core API types while allowing the same codebase to support both ODH and RHOAI distributions. The Konflux Dockerfile uses `GOEXPERIMENT=strictfipsruntime` for FIPS compliance, dynamically linking against OpenSSL via `CGO_ENABLED=1`.

### Observability Stack Orchestration

The Monitoring controller is remarkably comprehensive — it deploys an entire observability platform including Prometheus (via MonitoringStack CRD from Cluster Observability Operator), Tempo (distributed tracing), OpenTelemetry Collector (telemetry collection), Perses (dashboarding), and custom PrometheusRules for each component. It dynamically generates Prometheus alerting rules based on which components are enabled, validates dependent operator CRDs atomically before attempting deployment, and supports custom exporters (OTLP, Prometheus Remote Write) for integration with external observability platforms. The monitoring controller's complexity rivals the gateway controller, managing dozens of resources across multiple operator dependencies.

### Security Considerations

The FIPS compliance annotation in the CSV is set to `"false"` despite the binary being compiled with `strictfipsruntime`. This may indicate the annotation hasn't been updated to reflect the actual build-time FIPS enforcement. The DestinationRule for the ingress stack uses `insecureSkipVerify: true` for both port 443 and 8443 traffic within the cluster, which is acceptable for cluster-internal mTLS but should be noted. The kube-auth-proxy credentials secret contains auto-generated values (32-byte random cookie secret) that are not automatically rotated — secret rotation requires deleting the secret and letting the controller regenerate it. The `InsecureSkipVerify` field in the gateway controller actions is configurable via `GatewayConfig.Spec.VerifyProviderCertificate`, defaulting to `true` (verify), which is the secure default.
