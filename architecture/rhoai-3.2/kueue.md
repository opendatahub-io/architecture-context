## Purpose

**Short**: Kueue is a Kubernetes-native job queueing system that manages when workloads should be admitted to start based on quota, priority, and fair sharing policies.

**Detailed**: Kueue provides a set of APIs and controllers for job-level resource management in Kubernetes. It acts as a quota and admission control layer that decides when jobs should be admitted (allowed to create pods) and when they should be suspended or preempted. Kueue introduces the concept of ClusterQueues (cluster-scoped resource pools with quotas), LocalQueues (namespace-scoped queue endpoints for job submission), ResourceFlavors (hardware abstractions like GPU types or node pools), and Workloads (internal representations of jobs being managed).

The system supports advanced scheduling features including cohort-based resource sharing and borrowing between queues, fair sharing with weighted distribution, topology-aware scheduling for placing pods on nearby nodes, multi-cluster job distribution (MultiKueue), and integration with cluster autoscaler via Provisioning Requests. Kueue integrates with multiple workload frameworks including Kubernetes batch/v1 Jobs, Kubeflow training jobs (PyTorchJob, TFJob, MPIJob, XGBoostJob, PaddleJob), Ray clusters/jobs, JobSets, AppWrappers, and plain Pods/Deployments/StatefulSets. In the RHOAI platform, Kueue provides the resource quota management layer for AI/ML workload scheduling.

## Architectural Analysis

Kueue implements a sophisticated job queueing and admission control system using the Kubernetes operator pattern with controller-runtime. The architecture centers on a scheduler loop that periodically takes a snapshot of the cluster state, calculates resource assignments for pending workloads, and admits or preempts workloads based on quota, priority, and fair sharing policies. This snapshot-based approach provides consistency guarantees during scheduling cycles — each cycle operates on an immutable view of the cluster, preventing race conditions between concurrent reconcilers.

The controller hierarchy follows an observer/watcher pattern where state changes propagate through channels. When a ResourceFlavor is updated, it notifies the ClusterQueueReconciler which re-evaluates all dependent queues. When a Workload changes state, the LocalQueueReconciler and ClusterQueueReconciler are notified to update counters and status. This event-driven design avoids polling and ensures responsive scheduling. The controllers use a leader-aware reconciler decorator that allows non-leader replicas to requeue events with a lease-duration delay, preventing event loss during leader election transitions while keeping scheduling decisions centralized.

The fair sharing algorithm uses a tournament-based iterator that computes Dominant Resource Share (DRS) at each level of the cohort hierarchy. The DRS divides resource usage above nominal quota by a configurable weight, with workloads from ClusterQueues having the lowest DRS being admitted first. This naturally balances resource consumption across competing teams. Preemption integrates with fair sharing through two configurable strategies: LessThanOrEqualToFinalShare (preempt if the preemptor's post-admission share would not exceed the preemptee's post-preemption share) and LessThanInitialShare (preempt only if the preemptor's share would remain strictly below the preemptee's current share). These strategies prevent oscillation and ensure monotonic progress toward fairness.

The webhook architecture is notably comprehensive — Kueue registers 17 mutating and 17 validating webhooks covering all supported workload types. All webhooks use `failurePolicy: Fail`, meaning the API server rejects requests if the webhook is unavailable. This strict policy ensures that no unmanaged workloads slip through, but it also means Kueue is a critical path component — its unavailability blocks job creation cluster-wide. The RHOAI deployment adds a NetworkPolicy restricting webhook ingress to port 9443, providing defense-in-depth.

The MultiKueue feature enables multi-cluster job distribution by maintaining kubeconfig-based connections to remote worker clusters. The management cluster's Kueue instance creates workloads on worker clusters and monitors their status, implementing a hub-and-spoke pattern. The garbage collector runs periodically to clean up orphaned remote workloads. This architecture has implications for secret management — each worker cluster connection requires a kubeconfig stored as a Kubernetes Secret in the Kueue namespace.

The Topology-Aware Scheduling (TAS) feature, still in alpha, adds pod placement optimization by reading node topology labels (e.g., rack, block) and scheduling pods within the same topology domain. This is critical for AI/ML training workloads that benefit from low-latency inter-node communication, such as distributed PyTorch training using NCCL. TAS injects scheduling gates into pods to delay scheduling until topology assignments are computed, then adds node selectors matching the assigned topology domain.
