## Purpose

**Short**: A high-throughput, memory-efficient LLM inference and serving engine optimized for Intel Gaudi (Habana) accelerators, providing OpenAI-compatible REST API and gRPC endpoints.

**Detailed**: vllm-gaudi is a fork of the upstream vLLM project, adapted for Intel Gaudi HPU accelerators. It serves as the primary LLM inference runtime for RHOAI when deployed on Intel Gaudi hardware (Gaudi2/Gaudi3). The component provides an OpenAI-compatible HTTP API server for chat completions, text completions, embeddings, scoring, reranking, and audio transcription, as well as a gRPC adapter for Text Generation Inference Service (TGIS) protocol compatibility.

The repository produces a single container image (`odh-vllm-gaudi-rhel9`) built on RHEL 9.6 UBI with the complete Habana software stack including PyTorch with HPU support, OpenMPI for distributed inference, and RDMA/InfiniBand drivers for high-speed inter-node communication. The image supports two entrypoint modes: the vLLM OpenAI-compatible API server (port 8000) and the TGIS gRPC adapter (port 8033), selected at container startup via the entrypoint script.

vllm-gaudi integrates with the RHOAI model serving stack. It is deployed as a runtime container within KServe InferenceService pods, receiving inference requests routed through the platform's ingress infrastructure (Gateway API / HTTPRoute). The engine supports tensor parallelism, pipeline parallelism, and data parallelism for distributing large models across multiple Gaudi cards, with MPI-based multi-worker coordination via SSH.

## Architectural Analysis

### Intel Gaudi HPU Backend Architecture

The vllm-gaudi HPU backend implements a sophisticated execution strategy built on Habana's lazy evaluation model. Unlike CUDA's eager execution, HPU operations are accumulated into graphs that are compiled and executed as units. The `HPUModelRunnerBase` manages this complexity through a bucketing system (`HPUBucketingManager`) that pre-defines batch size and sequence length combinations, each mapping to a pre-compiled HPU graph. This amortizes compilation cost across inference requests at the expense of some padding overhead. The `wrap_in_hpu_graph()` mechanism and `mark_step()` synchronization points provide fine-grained control over graph boundaries, balancing compilation granularity against execution efficiency.

### Dual-Protocol Serving Architecture

The container image supports two distinct serving protocols through a shared engine: the OpenAI-compatible REST API (FastAPI/Uvicorn on port 8000) and the TGIS gRPC adapter (on port 8033). Both share the same underlying vLLM engine and HPU worker infrastructure. The TGIS adapter is selected by using the `tgis-entrypoint.sh` script instead of `vllm-entrypoint.sh`. Both entrypoints start an SSH daemon on port 3022 for MPI-based distributed inference before launching the respective serving process under the `vllm` user. This dual-protocol design allows the same container image to serve in both KServe (REST) and ModelMesh (gRPC) deployment modes.

### Distributed Inference via MPI and torch.distributed

Multi-card and multi-node inference uses a hybrid approach: MPI (OpenMPI 4.1.6) handles process launch and initial coordination via SSH, while `torch.distributed` (with HPU-specific `torch-ccl` or `gloo` backends) handles runtime tensor communication. The SSH daemon runs on port 3022 with relaxed host checking to support dynamic pod scheduling in Kubernetes. RDMA/InfiniBand drivers are bundled for high-bandwidth inter-node tensor transfers. The `HPUWorker` initializes distributed groups for tensor parallelism, pipeline parallelism, and optional KV cache transfer for disaggregated prefill scenarios.

### Observability and Production Readiness

The serving stack implements comprehensive observability: Prometheus metrics (request counts, latency histograms, queue depth, GPU utilization) are exposed at `/metrics` with multiprocess support for distributed deployments. OpenTelemetry tracing with GenAI semantic conventions tracks per-request latency breakdown (queue time, scheduling, forward pass, end-to-end). Health endpoints (`/health`, `/ping`) provide liveness probing compatible with both Kubernetes and SageMaker. The engine includes a watchdog task that monitors backend health every 5 seconds and triggers graceful shutdown on engine failure.

### Container Security Posture

The image runs as a non-root user (UID 2000, GID 0) for OpenShift compatibility, with the entrypoint scripts starting as root only to launch the SSH daemon before dropping to the `vllm` user. The explicit removal of `openssl-fips-provider-so` (line 39 of Dockerfile.konflux.gaudi) to resolve a RHEL 9.5/9.6 version conflict is a notable FIPS concern — this removes the OpenSSL FIPS provider, potentially breaking FIPS mode. The `hashlib.md5` usage throughout the codebase is exclusively for non-security purposes (configuration hashing with `usedforsecurity=False`), which is FIPS-compliant usage.
