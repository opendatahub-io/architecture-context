## Purpose

**Short**: Intelligent inference request scheduler that routes LLM inference requests to optimal model-serving pods via the Gateway API Inference Extension (GIE) framework.

**Detailed**: The llm-d-inference-scheduler provides two deployable components for the llm-d inference framework: an Endpoint Picker (EPP) service and a Prefill/Decode (P/D) routing sidecar. The EPP implements the GIE ext-proc callback mechanism to make real-time routing decisions for incoming inference requests based on a pluggable system of filters, scorers, and profile handlers. It evaluates candidate pods using criteria including KV-cache locality, queue depth, session affinity, and active request counts.

The P/D routing sidecar runs alongside vLLM decoder pods and implements disaggregated prefill/decode inference. It intercepts OpenAI-compatible chat completion requests and orchestrates the two-phase inference pipeline: sending the prompt to a prefiller pod for KV-cache population, then forwarding the request with KV transfer metadata to the local decoder for token generation. The sidecar supports three P/D protocols (NIXL v2, LMCache, SGLang) and includes SSRF protection via InferencePool-based allowlisting.

Both components build on the upstream Gateway API Inference Extension project (`sigs.k8s.io/gateway-api-inference-extension`), extending it with llm-d-specific features such as disaggregated P/D scheduling, precise prefix cache scoring via the llm-d-kv-cache-manager, data-parallel request distribution, and custom filter/scorer plugins.

## Architectural Analysis

The llm-d-inference-scheduler demonstrates a clean separation between the scheduling decision engine (EPP) and the request execution engine (P/D sidecar). The EPP operates purely as a decision-maker via the GIE ext-proc protocol -- it never handles actual inference traffic, only metadata. This design means the EPP is not in the data path for large request/response payloads, reducing latency impact and enabling the Gateway's Envoy proxy to handle the heavy lifting of request forwarding. The sidecar, conversely, is fully in the data path but only for P/D scenarios where two-phase orchestration is required.

The plugin architecture is the most architecturally significant design decision. All 12 custom plugins follow a consistent factory pattern with JSON-based configuration, enabling runtime composition via EndpointPickerConfig YAML. The 9 pre-built configuration variants in `deploy/config/` demonstrate the flexibility: from simple single-profile scheduling (epp-config.yaml) to complex P/D disaggregation with prefix cache scoring (pd-epp-config.yaml) to precise KV-cache tracking (epp-precise-prefix-cache-config.yaml). Notably, the PdProfileHandler implements a threshold-based decision algorithm (lines 136-170 of pd_profile_handler.go) that dynamically chooses between decode-only and prefill+decode paths based on prefix cache hit percentage -- this is the core intelligence that makes P/D disaggregation efficient rather than always splitting.

The SSRF protection in the sidecar (allowlist.go) is a notable security feature. Rather than relying on network policies alone, the sidecar watches InferencePool resources to build a dynamic allowlist of valid prefiller endpoints. When a request arrives with an `x-prefiller-host-port` header, the sidecar validates the target against this allowlist before forwarding. This prevents a compromised request from using the sidecar as an open proxy to arbitrary cluster endpoints. The implementation uses Kubernetes informers with a 30-second resync and handles both `inference.networking.k8s.io/v1` and the legacy `inference.networking.x-k8s.io/v1alpha2` API groups.

The build system reveals an interesting multi-language dependency chain. The EPP binary requires CGO for three reasons: Python 3.12 embedding (for Jinja2 template rendering via llm-d-kv-cache-manager), Rust libtokenizers (for HuggingFace tokenizer bindings), and ZeroMQ (for KV-cache event subscription). The Konflux Dockerfile builds libtokenizers from source using vendored Cargo dependencies, ensuring build hermeticity. The Python embedding is used specifically for the PrecisePrefixCacheScorer's `preprocessing.RenderJinjaTemplateRequest` function, which renders chat completion templates to extract raw prompts for prefix matching. This multi-language approach adds complexity but enables precise KV-cache locality scoring that would be difficult to achieve in pure Go.

One risk worth noting is the TLS configuration in the sidecar. While the cipher suite selection is strong (6 modern ECDHE-AEAD suites, TLS 1.2 minimum), the `InsecureSkipVerify` flags for both prefiller and decoder connections default to false but are exposed as CLI flags. In production P/D deployments, if operators enable these flags to avoid certificate management complexity, they would lose TLS verification. Additionally, the self-signed certificate generation uses RSA-4096 with a 10-year validity period -- appropriate for ephemeral sidecar use but the long validity could be a concern if certificates are persisted or reused.
