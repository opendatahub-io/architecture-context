## Purpose

**Short**: Kubernetes operator that manages the lifecycle of Model Registry and Model Catalog instances on OpenShift, provisioning REST services, databases, auth proxies, and associated RBAC resources.

**Detailed**: The Model Registry Operator is a controller-runtime based Kubernetes operator that reconciles `ModelRegistry` custom resources (CRD group `modelregistry.opendatahub.io`) in both `v1alpha1` (deprecated) and `v1beta1` (storage version) API versions. For each ModelRegistry CR, the operator creates and manages a complete stack of Kubernetes resources: a Deployment running the model-registry REST service, a Service, a ServiceAccount, RBAC Roles/RoleBindings, optional kube-rbac-proxy sidecar for authentication, optional OpenShift Routes for external access, NetworkPolicies, and optionally an auto-provisioned PostgreSQL database (Deployment, PVC, Service, Secret).

The operator also includes a dedicated `ModelCatalogReconciler` that manages a singleton "model-catalog" deployment in the operator's namespace. The catalog deployment is a separate model-registry instance pre-loaded with curated model metadata, backed by its own PostgreSQL database. The catalog reconciler fetches admin groups from the platform Auth CR (`services.platform.opendatahub.io/v1alpha1/Auth`) and creates admin RBAC accordingly.

The operator supports automatic migration from OAuth proxy to kube-rbac-proxy (RHOAI 2.x to 3.x transition), CRD storage version migration (v1alpha1 to v1beta1) via both Kubernetes StorageVersionMigration API and manual re-read/re-write fallback, and runtime platform detection to support traditional OpenShift, BYOIDC-enabled OpenShift 4.20+, and plain Kubernetes clusters.

## Architectural Analysis

The Model Registry Operator follows a standard kubebuilder v4 pattern with two controllers sharing a common template engine. The `ModelRegistryReconciler` handles per-instance ModelRegistry CRs using Go templates to generate Kubernetes resources (Deployments, Services, Routes, etc.), while the `ModelCatalogReconciler` manages a singleton catalog instance. Both controllers share a `TemplateApplier` and `ResourceManager` for consistent resource lifecycle management with patch-based updates via the `k8s-objectmatcher` library.

A significant architectural decision is the OAuth-to-kube-rbac-proxy migration path. The operator automatically migrates `OAuthProxy` configurations to `KubeRBACProxy` in both the mutating webhook (`Default()`) and the reconcile loop itself (line 180-191 of `modelregistry_controller.go`). This dual-migration approach ensures registries created before the webhook was enabled are also migrated. The v1beta1 API drops Istio gateway support entirely (present in v1alpha1), replacing it with a simpler kube-rbac-proxy + Route pattern. The mutual exclusion validation (`Cannot use both oauthProxy and kubeRBACProxy`) ensures clean transitions.

The platform capability detection system (`ClusterCapabilities`) is noteworthy for its graceful handling of BYOIDC-enabled OpenShift 4.20+ clusters. Rather than a simple "is OpenShift" check, it separately tracks `IsOpenShift` (route API), `HasUserAPI` (user API), and `HasConfigAPI` (config API). This allows the operator to skip `user.openshift.io/Group` creation in BYOIDC mode while still creating Routes and reading the cluster ingress domain. The detection logic handles partial API discovery failures (expected when `user.openshift.io` is intentionally disabled) without treating them as errors.

The CRD storage version migration system uses a strategy pattern with two implementations: `SVMStrategy` (using the Kubernetes StorageVersionMigration API) as the preferred approach, and `ManualStrategy` (re-reading and re-writing all resources to trigger conversion) as a fallback. The migration runs as a background goroutine that starts 30 seconds after the manager boots, checks the CRD's `status.storedVersions`, and migrates if the source version is still present. This is essential for the v1alpha1-to-v1beta1 transition.

The template-based resource generation is a double-edged architectural choice. It enables clean separation between resource shapes and controller logic but makes it harder to trace exactly what resources are created without reading both the Go code and the YAML templates. The kube-rbac-proxy configuration is particularly interesting: it uses a ConfigMap with a `resourceAttributes` authorization policy that grants access based on `services/get` verb on the registry's Service resource, effectively delegating access control to standard Kubernetes RBAC rather than requiring custom authorization logic.
