## Purpose

**Short**: Argo Workflows is a Kubernetes-native workflow orchestration engine that executes multi-step workflows as directed acyclic graphs (DAGs) of containerized tasks, used as the execution backend for Data Science Pipelines in RHOAI.

**Detailed**: Argo Workflows provides a workflow engine that runs on Kubernetes, defining workflows as custom resources (CRDs) in the `argoproj.io` API group. The system consists of three main binaries: a workflow controller that watches Workflow CRDs and orchestrates pod creation for each workflow step, an executor sidecar (argoexec) that runs inside workflow pods to manage artifact I/O and step coordination, and an API server (argo-server) that provides gRPC/REST APIs and a web UI for workflow management.

In the RHOAI platform, Argo Workflows serves as the execution engine for Data Science Pipelines (DSP). The DSP operator deploys the workflow controller and argoexec images as part of the pipeline infrastructure. Workflows represent ML pipeline runs, with each step executing as a Kubernetes pod. The controller manages the full lifecycle including scheduling, artifact storage, retry logic, TTL-based cleanup, and optional archival to PostgreSQL or MySQL databases.

The component supports pluggable artifact storage (S3, GCS, Azure Blob, HDFS, Artifactory), configurable parallelism controls at both workflow and namespace levels, CronWorkflow scheduling, and SSO/OIDC authentication for the API server. For RHOAI, the images are built with FIPS-compliant Go crypto (GOEXPERIMENT=strictfipsruntime) on UBI9 base images.

## Architectural Analysis

Argo Workflows implements a classic Kubernetes operator pattern but with a distinctive two-process execution model. The workflow-controller acts as the orchestrator, running as a long-lived Deployment with leader election, while the argoexec binary runs ephemerally inside each workflow pod as a sidecar. Communication between the main container and argoexec happens entirely through the `/var/run/argo/` shared EmptyDir volume -- the "emissary protocol." This design avoids the need for a container runtime socket (Docker/CRI) and works on any Kubernetes distribution, which is critical for OpenShift compatibility. The init container copies the argoexec binary and template JSON into the shared volume, the main container writes outputs to well-known paths, and argoexec monitors those paths for completion signals and exit codes.

The controller uses a multi-queue architecture with five separate rate-limited work queues: workflow reconciliation (32 workers by default), TTL cleanup (4), pod garbage collection (4), CronWorkflow scheduling (8), and workflow archival (8). This separation prevents slow database archival from blocking workflow reconciliation. The controller also implements workflow-level parallelism controls, namespace-level parallelism, and a synchronization manager for semaphore-based resource locking. Pod creation includes retry affinity support (scheduling retried pods on different nodes) and a comprehensive pod GC strategy system with five cleanup policies.

For the RHOAI distribution, the build produces dual binaries for argoexec: a FIPS-compliant dynamically-linked binary (`argoexec-fips` with `CGO_ENABLED=1, GOEXPERIMENT=strictfipsruntime`) and a standard statically-linked binary (`argoexec` with `CGO_ENABLED=0`). The workflow-controller Konflux build only produces the FIPS binary. Both use UBI9 minimal as the runtime base image. The presence of `InsecureSkipVerify=true` in several code paths (SSO client, executor agent, self-signed TLS) represents a potential FIPS compliance gap at the application level, though these are typically configurable and can be disabled in production.

Notably, in RHOAI's Data Science Pipelines deployment model, only the workflow-controller and argoexec images are used. The argo-server (API server with web UI) is not deployed -- DSP provides its own API and uses the Kubernetes API directly to submit Workflow CRDs. This means the extensive SSO/OIDC authentication, gRPC service layer, and web UI code in the repository are present but unused in RHOAI DSP deployments. The CRDs are also typically installed by the DSP operator rather than from the manifests in this repository.
