Security Considerations:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. Controller Security:
   - Runs as non-root user (runAsNonRoot: true)
   - No privilege escalation (allowPrivilegeEscalation: false)
   - Minimal capabilities (drop ALL)
   - Seccomp profile: RuntimeDefault

2. Webhook Security:
   - TLS 1.2+ enforced for webhook communication
   - mTLS authentication from Kubernetes API server
   - Certificate rotation should be automated (cert-manager recommended)

3. Training Pod Security:
   - Pods run in user namespaces with user-controlled ServiceAccounts
   - S3 credentials stored in Kubernetes secrets (should use encryption at rest)
   - Consider using IRSA (IAM Roles for Service Accounts) for AWS authentication
   - GPU access requires privileged capabilities (security vs. functionality trade-off)

4. Network Security:
   - Pod-to-pod training traffic is unencrypted (performance optimization)
   - Trust boundary: assume pod network is trusted
   - Isolate training namespaces from sensitive workloads

5. API Security:
   - Alpha API (v1alpha1) - use with caution in production
   - Webhook validation prevents malformed TrainJob submissions
   - RBAC limits who can create TrainJob resources

6. Observability Security:
   - Metrics endpoint uses HTTPS with bearer token authentication (RHOAI)
   - Logs may contain sensitive training parameters (review log retention policies)
