Authentication Flows:
═══════════════════════════════════════════════════════════════════════════

MLflow API Authentication (kubernetes-auth):
1. Client sends request with Bearer Token (Kubernetes ServiceAccount token)
2. MLflow receives request at /api/* or / endpoints
3. MLflow extracts Bearer token from Authorization header
4. MLflow sends self_subject_access_review to Kubernetes API (6443/TCP HTTPS)
   - Uses client's Bearer token (not MLflow's ServiceAccount token)
5. Kubernetes validates token and returns SubjectAccessReview result
6. MLflow authorizes request if token is valid (no special RBAC required)
7. MLflow processes request and returns response

Operator API Authentication:
1. Prometheus sends request with Bearer Token (ServiceAccount)
2. kube-rbac-proxy or ServiceMonitor validates token
3. Token must have metrics-reader role
4. Operator returns Prometheus metrics

TLS Certificate Provisioning (OpenShift):
1. Operator annotates MLflow Service with:
   service.beta.openshift.io/serving-cert-secret-name: mlflow-tls
2. service-ca-operator watches Service annotation
3. service-ca-operator generates TLS certificate and private key
4. service-ca-operator creates mlflow-tls Secret with cert/key
5. MLflow Deployment mounts mlflow-tls Secret as volume
6. MLflow uvicorn server uses cert/key for TLS 1.3 termination
7. service-ca-operator auto-rotates certificate every 90 days
