Authentication Patterns
============================================================

1. Gateway API + kube-auth-proxy:
   Components: Dashboard, Notebooks, MLflow, HTTPRoute-based services
   Flow: Envoy ext_authz → kube-auth-proxy (OAuth2/OIDC → Bearer token)

2. kube-rbac-proxy sidecar:
   Components: Model Registry, TrustyAI, Guardrails Orchestrator, NeMo, Dashboard
   Flow: SubjectAccessReview against K8s RBAC

3. oauth-proxy sidecar:
   Components: ModelMesh pods, RayClusters (via CodeFlare)
   Flow: OpenShift OAuth proxy on port 8443

4. Kubernetes TokenReview:
   Components: Models-as-a-Service (Kuadrant AuthPolicy)
   Flow: Gateway API AuthPolicy → K8s TokenReview

5. ServiceAccount tokens:
   Components: Argo Workflows, DSP API, KFP components
   Flow: K8s API bound service account tokens

============================================================
TLS Patterns
============================================================

- OpenShift service-ca:    Dashboard, kube-auth-proxy, operator webhooks,
                           model-registry-operator
- Self-managed certs:      Training Operator, Trainer (webhook certs)
- cert-manager:            KubeRay (mTLS, auth), Kueue (webhook)
- Istio mTLS:              Model Registry (in-mesh), data-science-gateway backend
- Optional user-provided:  vLLM, OVMS, TGIS, MLMD, llm-d-routing-sidecar
- FIPS-compliant:          All Go binaries (GOEXPERIMENT=strictfipsruntime)

============================================================
Network Policies
============================================================

- kube-auth-proxy:         Gateway pods only on 8443; monitoring on 9000
- notebook-controller:     Per-notebook: ingress from Gateway + monitoring only
- codeflare-operator:      Per-RayCluster: inter-pod + OAuth proxy access
- llama-stack-k8s-operator: Per-LlamaStackDistribution
- model-registry-operator: Per-ModelRegistry instance
- kserve:                  Per-InferenceService pod isolation

============================================================
Security Context Constraints (SCCs)
============================================================

- kserve-scc:   KServe model serving pods (GPU device access)
- anyuid:       Various runtime containers (non-root UID)

============================================================
Secrets (Common Patterns)
============================================================

- storage-config:          Per-backend JSON (S3, Azure, GCS) — ModelMesh
- aws-credentials:         AWS ACCESS_KEY_ID, SECRET_ACCESS_KEY — DSP, ilab, ragas
- database credentials:    DB connection strings — MLMD, model-registry, DSP, mlflow, feast
- OAuth client secrets:    OAuth2 client creds — kube-auth-proxy, dashboard
- model serving secrets:   Inference URL, API token, TLS — odh-model-controller
- image pull secrets:      Registry credentials — odh-model-controller (NIM)
