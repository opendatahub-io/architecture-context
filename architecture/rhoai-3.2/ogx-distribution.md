## Purpose

**Short**: A container image distribution of Meta's Llama Stack framework, packaged for Red Hat OpenShift AI with pre-configured inference providers, vector databases, evaluation, and agent capabilities.

**Detailed**: ogx-distribution builds and ships a containerized Llama Stack server image that serves as the AI inference orchestration layer in RHOAI. Llama Stack is Meta's open framework for building AI applications, providing a unified API surface across inference, agents, safety, evaluation, vector I/O, tool runtime, and file storage. This distribution customizes the upstream Llama Stack for the Red Hat ecosystem by packaging it with Red Hat-specific provider plugins (TrustyAI for safety/evaluation, KFP for pipelines), pre-installing dependencies into a UBI9 base image, and shipping a `run.yaml` configuration file that wires up all providers.

The distribution does not contain significant application source code of its own — it is primarily a build and packaging system. The `distribution/build.py` script invokes the `llama stack list-deps` CLI command to compute all Python dependencies for the configured providers, then renders a `Containerfile` from a template (`Containerfile.in`). The `run.yaml` file is the core runtime artifact: it defines all API providers (inference via vLLM/Bedrock/WatsonX/Azure/VertexAI/OpenAI, vector I/O via Milvus/FAISS/pgvector, safety via TrustyAI, evaluation via lm-eval and RAGAS, etc.) with environment-variable-driven conditional enablement. The Llama Stack server listens on port 8321 and exposes OpenAI-compatible REST APIs for inference, embeddings, agents, and more.

For RHOAI, the image is built by Konflux from `Dockerfile.konflux`, which uses a pre-built AIPCC base image containing wheel releases rather than installing from PyPI. This ensures reproducibility and supply-chain integrity for the productized build. The upstream community build uses `distribution/Containerfile` and installs from the opendatahub-io fork of llama-stack on GitHub.

## Architectural Analysis

The ogx-distribution repository is architecturally unusual within the RHOAI platform because it contains almost no application source code. It is fundamentally a **build and packaging system** for the upstream Llama Stack framework. The actual runtime code lives in the opendatahub-io/llama-stack fork (version v0.3.5+rhai0), and this repository's value lies in: (1) the `run.yaml` configuration that wires up all providers for the Red Hat ecosystem, (2) the `build.py` script that auto-generates the Containerfile with correct dependency resolution, and (3) the Konflux build pipeline that produces the productized image.

The **provider architecture** is highly extensible and environment-variable-driven. The `run.yaml` uses Llama Stack's conditional provider syntax (`${env.VAR:+provider_id}`) to enable providers only when specific environment variables are set. This means the same container image can serve as a simple vLLM frontend (just set `VLLM_URL` and `INFERENCE_MODEL`) or as a full-featured AI platform with RAG, safety, evaluation, multi-cloud inference, and agent capabilities. The default configuration enables Milvus (inline/local), TrustyAI safety, lm-eval evaluation, basic scoring, web search tools, and local file storage. Cloud providers (Bedrock, WatsonX, Azure, VertexAI, OpenAI) are all opt-in.

The **storage architecture** mandates PostgreSQL as the primary persistence layer (commit 12cd263a). Two storage backends are configured: `kv_postgres` for key-value storage (metadata, agent state, vector IO persistence) and `sql_postgres` for relational data (inference logs, file metadata, agent responses, OpenAI conversations). This is a significant architectural decision — earlier versions likely supported SQLite or in-memory storage, but the current design requires a PostgreSQL instance as an external dependency. The smoke tests validate that PostgreSQL tables (`llamastack_kvstore`, `inference_store`) are created and populated after inference.

The **dual-build system** reflects the ODH/RHOAI split. The community build (`distribution/Containerfile`) installs from PyPI and the opendatahub-io GitHub fork, using `uv pip install` with mostly unpinned dependencies. The Konflux build (`Dockerfile.konflux`) uses a fundamentally different approach: it starts from an AIPCC base image (`quay.io/aipcc/base-images/cpu:3.0-*`) that has been pre-built with wheel releases, then runs `install-wheel-release.sh` to install pre-compiled, versioned wheel packages. This provides better reproducibility and supply chain control for the productized image, but the `hermetic: false` flag in the Tekton PipelineRun indicates that full build hermeticity has not yet been achieved — the wheel installation still requires network access to a private GitLab registry (authenticated via `BOT_PAT`).

The **security model** is notable for what it delegates. The Llama Stack server exposes all APIs on port 8321 without any authentication or encryption. This is by design — security is expected to be provided by the RHOAI platform's ingress layer (kube-rbac-proxy sidecars for component auth, Gateway API for TLS termination). However, the egress security profile is rich: connections to cloud providers use API keys or IAM credentials over TLS, Milvus supports mTLS with client certificates, and TrustyAI FMS supports custom CA certificates. The embedded granite-embedding-125m-english model (`huggingface-cli download` in the Dockerfile) means the image can perform embeddings locally without any external service dependency.
