## Purpose

**Short**: CPU-only build of vLLM, a high-throughput and memory-efficient inference and serving engine for large language models (LLMs).

**Detailed**: vLLM CPU is the CPU-optimized variant of the vLLM inference server, packaged as a container image for Red Hat OpenShift AI (RHOAI). It provides OpenAI-compatible REST API endpoints for LLM inference (chat completions, text completions, embeddings, etc.) without requiring GPU hardware, targeting ppc64le and s390x architectures that lack GPU support.

The component ships as a single Konflux-built container image (`odh-vllm-cpu-rhel9`) with two distinct entrypoints: the primary OpenAI-compatible HTTP API server (`vllm.entrypoints.openai.api_server`) and a gRPC adapter (`vllm_tgis_adapter`) that exposes TGIS (Text Generation Inference Server) gRPC protocol compatibility. The gRPC adapter is built as a second Dockerfile stage atop the OpenAI server image, adding the `vllm-tgis-adapter` package from the opendatahub-io project.

Within RHOAI, vLLM CPU is deployed as an InferenceService runtime via KServe. Users select this runtime when deploying models on CPU-only nodes. The server loads transformer models from HuggingFace Hub or local storage, runs inference using PyTorch with CPU-optimized backends (Intel Extension for PyTorch on x86_64, OpenBLAS on ppc64le/s390x), and serves predictions through the standardized APIs.

## Architectural Analysis

The vLLM CPU repository is a downstream fork of the upstream [vllm-project/vllm](https://github.com/vllm-project/vllm) repository, rebased to version 0.11.2 with Red Hat-specific patches for CPU-only builds on ppc64le and s390x architectures. The primary architectural distinction from the upstream project is the removal of GPU dependencies and the addition of platform-specific build scripts (`build_vllm_ppc64le.sh`, `build_vllm_s390x.sh`) that compile system libraries (OpenBLAS, LLVM, numactl) from source for architectures where pre-built packages are unavailable.

The Dockerfile.konflux.cpu uses a multi-stage build pattern with a notable design: a single Dockerfile produces two logical components through Docker stage inheritance. The `vllm-openai` stage is the primary server, while `vllm-grpc-adapter` extends it by installing the `vllm-tgis-adapter` package and changing the entrypoint. This means the same image can serve both HTTP and gRPC protocols depending on the entrypoint command. The TGIS adapter is cloned from the `opendatahub-io/vllm-tgis-adapter` repository at build time (version 0.10.0), creating a tight coupling between the vLLM version and the adapter version.

The authentication architecture has an interesting layering: vLLM implements its own `AuthenticationMiddleware` (ASGI middleware) that validates Bearer tokens against SHA-256 hashes of configured API keys, but only for `/v1/*` paths. Health checks, metrics, and non-v1 API paths are intentionally unauthenticated. In a RHOAI deployment, the platform operator injects a `kube-rbac-proxy` sidecar that provides a second authentication layer at the pod level, meaning the vLLM-native auth is typically redundant but provides defense-in-depth.

The build hermeticity is notably weak for this component — the Tekton PipelineRun explicitly sets `hermetic: false`, and the multi-architecture build requires fetching hundreds of Python packages and compiling system libraries from source during the build. This is a pragmatic tradeoff: achieving hermeticity for source-compiled C libraries across ppc64le and s390x would require significant infrastructure investment. The absence of language-level lock files (no `uv.lock` or `requirements.lock`) means builds are not reproducible across time, though the pinned versions in `requirements/common.txt` and `requirements/cpu.txt` provide some version stability.

The server exposes a remarkably broad API surface — 20+ HTTP endpoints covering OpenAI, Anthropic, SageMaker, and custom protocols. The Anthropic Messages API compatibility (`/v1/messages`) and the Responses API (`/v1/responses`) demonstrate vLLM's ambition to be protocol-agnostic. The SageMaker compatibility layer (`/ping`, `/invocations`) allows the same image to run in AWS SageMaker environments, though this is less relevant for RHOAI deployments.
