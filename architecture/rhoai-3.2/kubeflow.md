## Purpose

**Short**: Manages the lifecycle of Jupyter Notebook workbenches on OpenShift, providing per-notebook ingress routing, authentication, network isolation, and idle culling.

**Detailed**: This repository is a downstream fork of the upstream Kubeflow kubeflow repository, tailored for Red Hat OpenShift AI (RHOAI). It contains two cooperating Go-based Kubernetes controllers that together manage Jupyter Notebook workbench instances:

1. **notebook-controller** (upstream Kubeflow): The foundational controller that reconciles `Notebook` CRs into StatefulSets, Services, and optionally Istio VirtualServices. It also provides an idle-culling controller that monitors Jupyter kernel/terminal activity and scales down idle notebooks to zero replicas.

2. **odh-notebook-controller** (ODH/RHOAI extension): An enterprise-grade extension controller that layers RHOAI-specific capabilities on top of upstream Notebook CRs. It manages Gateway API HTTPRoutes for per-notebook ingress, injects kube-rbac-proxy authentication sidecars, creates per-notebook NetworkPolicies, handles certificate trust bundles, integrates with Data Science Pipelines (Elyra), syncs runtime images from ImageStreams, and manages cross-namespace ReferenceGrants for Gateway API routing. This controller is the primary differentiator between upstream Kubeflow notebooks and RHOAI workbenches.

Both controllers are built with controller-runtime, compiled with FIPS-compliant crypto (strictfipsruntime), and deployed as separate container images via Konflux. They run in the same namespace (typically `opendatahub`) but operate on different aspects of the Notebook CR lifecycle.

## Architectural Analysis

The kubeflow repository implements a **dual-controller architecture** where the upstream Kubeflow notebook-controller handles the foundational Kubernetes primitives (StatefulSet, Service) while the odh-notebook-controller layers enterprise-grade networking, security, and integration features on top. This separation preserves upstream compatibility while enabling RHOAI-specific capabilities. Both controllers watch the same `Notebook` CRD but reconcile different aspects of the resource, coordinated through annotations and the Kubernetes API server's watch mechanism.

The **central namespace routing pattern** is a key architectural decision. HTTPRoutes are created in the controller's namespace (typically `opendatahub`) rather than the user's namespace, which prevents users from creating or modifying their own ingress routes. This requires Gateway API ReferenceGrants in each user namespace to authorize cross-namespace Service references. The shared ReferenceGrant model (one per namespace, reference-counted by active notebooks) is efficient but requires careful cleanup via finalizers since owner references don't work cross-namespace.

The **kube-rbac-proxy sidecar injection** follows the RHOAI 3.x authentication pattern, replacing the RHOAI 2.x oauth-proxy approach. The webhook mutates Notebook CRs at admission time to inject the sidecar container, while the reconciler creates the supporting infrastructure (Service, ConfigMap, ClusterRoleBinding). The RBAC policy is specific: users must have `get` permission on the Notebook CR itself, which integrates naturally with Kubernetes RBAC. The authentication flow (TokenReview → SubjectAccessReview → proxy) is robust but adds latency on every request. The controller still contains OAuthClient cleanup code for 2.x → 3.x migration, showing the evolutionary nature of the auth model.

The **webhook update guard** is a sophisticated mechanism to prevent unnecessary pod restarts. When the webhook mutates a running notebook (e.g., because the kube-rbac-proxy image reference changed), it compares the old and new specs to determine if the change was user-initiated or webhook-initiated. Webhook-only changes are blocked with an `update-pending` annotation, deferring the update until the user manually stops and restarts the notebook. This prevents disruption to active Jupyter sessions.

The **idle culling** implementation polls Jupyter's REST API (`/api/kernels`, `/api/terminals`) rather than relying on Kubernetes-level metrics like CPU or network activity. This provides accurate application-level idle detection — a notebook with a running kernel is never culled even if it's not consuming CPU. The polling interval (default 1 minute) and idle timeout (default 24 hours) are configurable via ConfigMap. The culling action is annotation-driven: setting `kubeflow-resource-stopped` triggers the upstream controller to scale the StatefulSet to zero replicas, and a reconciliation lock annotation prevents race conditions between the culling controller and the ODH controller during initial setup.
