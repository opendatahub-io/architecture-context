## Purpose
**Short**: ODH Notebook Controller extends the upstream Kubeflow Notebook Controller to manage OpenShift-specific lifecycle operations for Jupyter notebook workbenches, including Gateway API ingress, kube-rbac-proxy authentication sidecars, network policies, certificate management, and pipeline integration.

**Detailed**: The ODH Notebook Controller is a controller-runtime based Go operator that watches `Notebook` custom resources (CRD defined by the upstream Kubeflow Notebook Controller at `kubeflow.org/v1`) and reconciles OpenShift-specific resources for each notebook instance. It operates as a companion controller alongside the upstream Kubeflow Notebook Controller.

In RHOAI 3.x, the controller implements the Gateway API ingress pattern: for each notebook, it creates an `HTTPRoute` in a central protected namespace that references the platform's `data-science-gateway` Gateway. When authentication is enabled (via annotation `notebooks.opendatahub.io/inject-auth`), the controller provisions a kube-rbac-proxy sidecar, a dedicated Service for the proxy, RBAC resources (ServiceAccount, ClusterRoleBinding for auth-delegator), and a ConfigMap with authorization policy. It also creates `ReferenceGrant` resources to allow cross-namespace backend references from HTTPRoutes to notebook Services.

Additionally, the controller manages trusted CA certificate bundles for notebooks, NetworkPolicy resources for pod isolation, Elyra pipeline runtime configuration secrets (integrating with DataSciencePipelinesApplication CRs), pipeline runtime image ConfigMaps from ImageStreams, Feast feature store configuration mounting, and legacy OAuthClient cleanup for migrations from RHOAI 2.x. A mutating webhook intercepts Notebook CREATE and UPDATE operations to inject the kube-rbac-proxy sidecar container, mount CA certificates, resolve ImageStream references to container image digests, and manage reconciliation locks to prevent race conditions during pod startup.

| notebook-httproute-access (per user namespace) | ReferenceGrant (Gateway API) | N/A | N/A | N/A | N/A | N/A | Allows cross-namespace HTTPRoute backend refs |

**Gateway API Pattern (RHOAI 3.x)**: The controller creates HTTPRoute CRs in its own central namespace (not the user's namespace) that reference the `data-science-gateway` Gateway in `openshift-ingress` namespace. The HTTPRoute path is `/notebook/{namespace}/{notebook-name}`. Cross-namespace backend references to the notebook Service in the user's namespace are authorized by a ReferenceGrant created per user namespace. When kube-rbac-proxy authentication is enabled, the HTTPRoute targets the kube-rbac-proxy Service (port 8443) instead of the notebook Service directly (port 8888).
