## Purpose

**Short**: Data Science Pipelines provides a Kubernetes-native platform for defining, running, scheduling, and tracking ML/AI pipelines as Argo Workflows with artifact management, caching, and metadata tracking.

**Detailed**: Data Science Pipelines (DSP) is the RHOAI distribution of Kubeflow Pipelines v2. It enables data scientists and ML engineers to define reproducible ML workflows as directed acyclic graphs (DAGs), compile them to Argo Workflow CRDs, and execute them on OpenShift clusters. The platform provides a REST/gRPC API server for pipeline management, a persistence agent that synchronizes workflow state to a relational database, a scheduled workflow controller for cron/periodic pipeline execution, and a driver/launcher system that orchestrates individual pipeline steps with input/output resolution, caching, and artifact management.

The system integrates with ML Metadata (MLMD) for experiment tracking and lineage, object stores (S3/MinIO/GCS) for artifact persistence, and relational databases (MySQL/PostgreSQL) for pipeline and run metadata storage. In RHOAI deployments, the platform operator (rhods-operator via the DataSciencePipelinesApplication CR) deploys and configures all DSP components. The driver and launcher binaries run as init/sidecar containers within Argo Workflow pods, handling parameter resolution, cache lookups, artifact downloads/uploads, and MLMD registration for each pipeline step.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| API Server | Go Service | REST/gRPC API for pipeline, run, experiment, and artifact management with webhook support |
| Driver | Go CLI (Argo step) | Runs before user containers to resolve inputs, check cache, generate pod spec patches |
| Launcher | Go CLI (Argo step) | Runs inside user containers to download artifacts, execute commands, upload outputs, record to MLMD |
| Persistence Agent | Go Service | Watches Workflow/ScheduledWorkflow CRDs and syncs state to API server database via gRPC |
| Scheduled Workflow Controller | Go Controller | Watches ScheduledWorkflow CRDs and creates Argo Workflows or API runs on schedule |

## Architectural Analysis

Data Science Pipelines follows a multi-process architecture where each binary has a distinct responsibility in the pipeline execution lifecycle. The API server acts as the central control plane, handling all CRUD operations and serving as the database gateway. The scheduled workflow controller and persistence agent are satellite processes that communicate with the API server via gRPC, using projected service account tokens for authentication. This separation allows each component to be independently scaled and restarted without affecting others.

The driver/launcher split is particularly noteworthy. The driver runs as an Argo Workflow init step before each user container, resolving inputs from upstream tasks, checking the cache, and generating a pod spec patch that rewrites the user container's entrypoint to run through the launcher. The launcher then handles artifact download, user command execution, artifact upload, and MLMD recording. This two-phase design isolates platform concerns (input resolution, caching) from execution concerns (artifact I/O, user code) and enables caching at the step level — if a cache hit occurs in the driver, the launcher and user code are never executed.

The caching mechanism uses a fingerprint-based approach: the driver computes a SHA256 hash of the step's inputs, outputs, container image, and command, then queries the API server's TaskService for matching cache entries. On a hit, outputs are copied from the cached MLMD execution. On a miss, the launcher creates a cache entry after successful execution. This design achieves deterministic caching without requiring content-addressable storage — the fingerprint captures the full execution context.

The launcher image contains both FIPS and non-FIPS binaries (`/bin/launcher-v2-fips` and `/bin/launcher-v2`), with the platform operator selecting the appropriate variant at deployment time. This dual-binary approach avoids the overhead of FIPS-compliant crypto for pipeline steps that don't require it, while maintaining compliance for regulated environments. The other four components (API server, driver, persistence agent, scheduled workflow controller) are built exclusively with FIPS flags.

For RHOAI deployments, the DSP components are not deployed directly from these kustomize manifests. Instead, the rhods-operator's DataSciencePipelinesApplication (DSPA) controller reads the user's DSPA CR and generates deployments with RHOAI-specific image references, TLS configuration, and object store settings. The upstream kustomize manifests serve as the reference for resource definitions, RBAC, and CRDs, but the actual deployment is templated by the platform operator. This indirection means that analyzing DSP's deployment behavior requires examining both this repository (for source code and CRD definitions) and the rhods-operator repository (for deployment logic).
