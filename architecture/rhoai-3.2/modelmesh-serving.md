## Purpose
**Short**: ModelMesh Serving is a Kubernetes operator that manages the lifecycle of multi-model serving runtimes using the ModelMesh distributed model serving framework, enabling efficient loading and serving of ML models at scale.

**Detailed**: ModelMesh Serving provides a controller that watches ServingRuntime, ClusterServingRuntime, Predictor, and InferenceService custom resources to orchestrate model inference workloads. When a ServingRuntime is created, the controller generates a Deployment containing ModelMesh sidecar containers (model-mesh Java process, storage helper/puller, optional REST proxy, and oauth-proxy) alongside the user-specified model server container(s). ModelMesh handles intelligent model placement, caching, and load balancing across runtime pods using etcd as a coordination store.

The controller supports both namespace-scoped and cluster-scoped operation modes. In cluster-scope mode, namespaces with the `modelmesh-enabled=true` label are managed. The Predictor controller reconciles model loading via the ModelMesh gRPC management API (VModel operations), supporting scale-to-zero with configurable grace periods. The ServingRuntime controller manages per-runtime Deployments with HPA autoscaling support. A Service controller creates and maintains Kubernetes Services for inference endpoints, and optionally creates ServiceMonitor resources for Prometheus integration.

The RHOAI build uses Konflux with FIPS-compliant Go runtime (`GOEXPERIMENT=strictfipsruntime`). Runtime pods include an `oauth-proxy` sidecar on port 8443 for OpenShift authentication, fronting the REST proxy endpoint on port 8008 with TLS termination using OpenShift serving certificates.
