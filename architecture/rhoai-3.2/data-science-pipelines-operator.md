## Purpose

**Short**: Manages the lifecycle of Data Science Pipelines (Kubeflow Pipelines v2) application instances on OpenShift, deploying and reconciling all required sub-components per namespace.

**Detailed**: The Data Science Pipelines Operator (DSPO) is a Kubernetes operator built with controller-runtime that watches `DataSciencePipelinesApplication` (DSPA) custom resources and deploys the full Kubeflow Pipelines v2 stack per-namespace. Each DSPA instance triggers the operator to deploy up to seven sub-components: an API Server (the main KFP backend), a Persistence Agent, a Scheduled Workflow controller, an Argo Workflow Controller, an ML Metadata (MLMD) gRPC server, an MLMD Envoy proxy, and optionally a MariaDB database and/or Minio object storage.

The operator uses Go templates (manifestival) to render Kubernetes manifests at reconcile time, injecting per-instance configuration such as database credentials, object storage endpoints, TLS certificates, and container images. It performs health checks against the database and object store before deploying pipeline components, supports pod-to-pod TLS via OpenShift service-ca signed certificates, and integrates kube-rbac-proxy sidecars for API authentication on exposed Routes. The operator also manages the optional Kubernetes-backed pipeline store with admission webhooks (validating and mutating) for Pipeline and PipelineVersion CRDs.

DSPO is deployed by the platform operator (rhods-operator) as a managed component within RHOAI. It operates within the `redhat-ods-applications` namespace (RHOAI overlay) and reconciles DSPA resources across all namespaces. The operator exposes Prometheus metrics for component readiness monitoring and supports FIPS-compliant builds via `GOEXPERIMENT=strictfipsruntime`.

## Architectural Analysis

The Data Science Pipelines Operator follows a template-driven reconciliation pattern rather than the more common programmatic resource construction. All sub-component manifests live in `config/internal/` as Go templates (`.yaml.tmpl` files) and are rendered at reconcile time using the manifestival library. This approach provides significant flexibility — the operator can inject per-instance configuration (names, namespaces, credentials, images, TLS settings) into any manifest field — but it also means that the actual Kubernetes resources created by the operator cannot be determined from static manifest analysis alone. You must understand both the template files AND the Go controller code that selects and parameterizes them.

The operator's reconcile loop in `dspipeline_controller.go` follows a strict ordering: Database → Storage → Common (NetworkPolicies) → Webhook (conditional) → API Server → Persistence Agent → Scheduled Workflow → Workflow Controller → MLMD. This ordering is intentional — the API Server requires a healthy database and object store before it can start, and MLMD is last because it depends on a TLS secret that may not exist until the MLMD gRPC Service (with its `service.beta.openshift.io/serving-cert-secret-name` annotation) is created and processed by OpenShift's service-ca controller. If the TLS secret is missing, the reconcile returns an error and requeues, creating a natural retry loop for certificate provisioning.

The security model implements defense in depth. External traffic enters via OpenShift Routes with TLS Reencrypt termination, hitting kube-rbac-proxy sidecars that perform SubjectAccessReview (SAR) authorization before proxying to the application containers. Internal pod-to-pod communication uses TLS certificates signed by OpenShift's service-ca (enabled by default via `PodToPodTLS: true`). NetworkPolicies restrict which pods can communicate directly with each service — for example, MariaDB only accepts connections from the DSPO operator (for health checks), the API Server, and the MLMD gRPC server. The operator also watches for changes to the `odh-trusted-ca-bundle` ConfigMap, triggering re-reconciliation to inject updated CA certificates into pipeline components.

The operator exposes a wide RBAC surface because pipeline execution pods need to create diverse Kubernetes resources. The `manager-role` ClusterRole includes permissions for KServe InferenceServices, Ray clusters, CodeFlare AppWrappers, and Seldon deployments — reflecting the ML ecosystem's diverse workload types. The `pipeline-runner` ServiceAccount (created per-DSPA instance) receives a subset of these permissions scoped to the namespace, allowing pipeline steps to interact with these ML frameworks.

One notable architectural decision is the dual-mode pipeline store. By default, pipelines are stored in the MariaDB database (`pipelineStore: database`). When `pipelineStore: kubernetes` is specified, pipelines are stored as Pipeline and PipelineVersion CRDs, and the operator deploys admission webhooks (both validating and mutating) in the DSPO namespace. The webhook deployment is shared across all DSPA instances using Kubernetes-backed storage, and the operator carefully manages webhook lifecycle — cleaning up webhook resources only when no DSPA instances with Kubernetes storage remain.
