## Purpose
**Short**: Kubernetes-native operator for managing distributed training jobs across multiple ML frameworks including PyTorch, TensorFlow, XGBoost, MPI, PaddlePaddle, and JAX.

**Detailed**: The Kubeflow Training Operator (KFTO) provides a unified Kubernetes-native interface for running distributed machine learning training jobs. It defines Custom Resource Definitions (CRDs) for six ML frameworks: PyTorchJob, TFJob, XGBoostJob, MPIJob, PaddleJob, and JAXJob. Each CRD allows users to declaratively specify distributed training topologies with replica types (e.g., master, worker, parameter server) and the operator manages the complete lifecycle of the training workload.

The operator creates and manages Pods, headless Services (for inter-pod communication), NetworkPolicies (for network isolation), and optionally HorizontalPodAutoscalers (for PyTorch elastic training). It supports gang scheduling via Volcano or scheduler-plugins for coordinated pod scheduling. The operator also includes a validating webhook server with self-managed TLS certificates for admission control of all job CRDs.

In RHOAI, the operator is deployed in the `opendatahub` namespace with the `kubeflow-` name prefix. The operator image is injected by the rhods-operator at deploy time via kustomize overlay parameter substitution.

|---------------|-----------|---------|

| HorizontalPodAutoscaler | PyTorchJob controller | Elastic training auto-scaling (when ElasticPolicy.Metrics is specified) |
| PodGroup (Volcano) | All job controllers (optional) | Gang scheduling coordination via Volcano scheduler |
