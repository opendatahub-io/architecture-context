## Purpose

**Short**: Kubernetes operator that manages distributed machine learning training jobs across multiple frameworks (PyTorch, TensorFlow, XGBoost, MPI, PaddlePaddle, JAX).

**Detailed**: The Kubeflow Training Operator (KFTO) is a controller-runtime based Kubernetes operator that provides Custom Resource Definitions (CRDs) for declaratively running distributed training workloads on Kubernetes. It manages the full lifecycle of training jobs — creating worker pods, headless services for inter-pod discovery, gang scheduling integration, elastic scaling, and failure recovery — across six ML frameworks.

Each framework has its own CRD (PyTorchJob, TFJob, XGBoostJob, MPIJob, PaddleJob, JAXJob) and a dedicated controller that understands the framework's distributed training topology. The common controller framework abstracts shared patterns (pod/service lifecycle, status management, gang scheduling) while framework-specific controllers handle environment variable injection, init containers, rendezvous coordination, and completion detection unique to each framework.

In RHOAI, the training operator is deployed via kustomize manifests in the `manifests/rhoai/` overlay, with namespace `opendatahub` and name prefix `kubeflow-`. It is built as a FIPS-compliant container image using UBI9 base images with `GOEXPERIMENT=strictfipsruntime` and `CGO_ENABLED=1`.

## Architectural Analysis

The Kubeflow Training Operator follows a clean plugin architecture: a shared common framework (`pkg/controller.v1/common/`) provides the reconciliation lifecycle, pod/service management, gang scheduling integration, and status management, while six framework-specific controllers implement the `ControllerInterface` to handle framework-particular concerns. This design makes adding new ML frameworks straightforward — a new controller only needs to implement environment variable injection (`SetClusterSpec`), completion detection (`IsMasterRole`, `UpdateJobStatus`), and optionally init containers or elastic scaling.

The MPI controller is notably more complex than the others. While most frameworks use headless services for inter-pod DNS discovery and inject environment variables for coordination, MPI requires a full launcher infrastructure: a dedicated ServiceAccount, Role, RoleBinding, and ConfigMap with hostfile and kubexec.sh scripts. The launcher pod uses `kubectl exec` to run mpirun commands on worker pods, which means the controller must create per-job RBAC granting `pods/exec` permission — a privilege escalation pattern worth noting from a security perspective. The MPI controller also overrides `ReconcileServices` to skip service creation entirely, as MPI communication occurs via kubectl exec rather than direct network connections.

PyTorch's elastic training support is the most sophisticated feature in the operator. When an `ElasticPolicy` is configured, the controller switches from static master/worker topology to dynamic worker scaling. It injects `PET_*` environment variables for PyTorch Elastic's rendezvous protocol (defaulting to C10D backend with Worker-0 as the rendezvous host), creates HPA resources when custom metrics are defined, and changes job completion semantics from "all workers succeed" to "any worker succeeds." The controller also creates NetworkPolicy resources for PyTorchJobs to restrict pod-to-pod communication within the same job, which is the only framework that does this.

The webhook certificate management deserves attention. Rather than relying on cert-manager (a common Kubernetes pattern), the operator uses OPA's cert-controller library to self-manage certificates. This creates a self-contained deployment with no external cert-manager dependency, but it means the operator needs RBAC access to Secrets and ValidatingWebhookConfigurations. The readiness probe is deliberately tied to webhook server startup — the operator reports not-ready until certificates are generated and the webhook server is listening, preventing premature webhook failures during startup. The `certsReady` channel coordinates between the cert generation goroutine and the controller setup goroutine, ensuring controllers only start after the webhook infrastructure is ready.

The RHOAI overlay introduces several customizations beyond namespace and naming: it removes the metrics port from the ClusterIP Service (relying instead on PodMonitor for direct pod scraping), generates an empty webhook-cert Secret (which cert-controller populates at runtime), and reuses the controller image as the PyTorch init container image (via `--pytorch-init-container-image` flag set to the same image as the controller). The aggregated ClusterRoles (`training-edit`, `training-view`) use standard Kubernetes aggregation labels (`rbac.authorization.k8s.io/aggregate-to-edit` and `aggregate-to-admin`) rather than Kubeflow-specific labels, enabling integration with OpenShift's default role hierarchy.
