## Purpose

**Short**: A framework for evaluating language models against standardized benchmarks, packaged as a batch job container image for RHOAI's LMEval (TrustyAI) evaluation pipeline.

**Detailed**: lm-evaluation-harness is a fork of [EleutherAI's lm-evaluation-harness](https://github.com/EleutherAI/lm-evaluation-harness) (v0.4.8), customized and packaged for Red Hat OpenShift AI. It provides a standardized runtime for running language model evaluations against 150+ benchmark task suites (6,500+ individual task YAML configurations). The framework supports multiple model backends including HuggingFace Transformers, vLLM, OpenAI-compatible APIs, Anthropic, IBM WatsonX AI, GGUF, and others.

In the RHOAI platform, this component is deployed as a Kubernetes batch Job container (`odh-ta-lmes-job-rhel9`). The TrustyAI operator (lm-eval-operator) creates Job resources that run this container to evaluate models. The container includes PyTorch 2.6.0, the Unitxt evaluation framework, and support for custom tasks and offline metrics. Results are written to a configurable output directory, and can optionally be pushed to OCI registries as artifacts using the included `oci.py` script (via the `olot` library and `skopeo`).

The container is built for four architectures (x86_64, arm64, ppc64le, s390x) using Konflux, with multi-stage Dockerfiles that build PyTorch and Apache Arrow from source for non-x86 architectures.

## Architectural Analysis

The lm-evaluation-harness follows a modular, registry-based architecture. The core design uses Python decorator-based registries (`MODEL_REGISTRY`, `TASK_REGISTRY`, `METRIC_REGISTRY`, `FILTER_REGISTRY`) that allow models, tasks, metrics, and output filters to be plugged in via simple registration. This pattern enables the 20+ model backends and 150+ task suites to coexist without tight coupling — each model adapter only needs to implement three abstract methods (`loglikelihood`, `loglikelihood_rolling`, `generate_until`) from the `LM` base class.

The task system is particularly sophisticated. Tasks are defined via YAML configuration files (6,500+ across 154 benchmark categories), with each YAML referencing a dataset path, prompt template, scoring metric, and output type. The `ConfigurableTask` class processes these YAMLs at runtime, while specialized Python task classes (like `Unitxt`) handle more complex evaluation patterns. The `TaskManager` dynamically indexes all YAML configs from the `lm_eval/tasks/` directory tree at startup, supporting hierarchical task groups, tags, and glob-pattern task selection.

The RHOAI packaging adds several notable customizations to the upstream EleutherAI project. The offline metrics system (`HF_EVALUATE_OFFLINE` environment variable, `LMEVAL_METRICS_PREFIX`) enables air-gapped operation by loading metric implementations from a local `metrics/` directory rather than downloading them from HuggingFace. The `oci.py` script integrates with the `olot` library to package evaluation outputs as OCI artifacts, enabling result traceability in the Konflux/RHOAI supply chain. The container pre-configures Unitxt catalogs and custom task directories, reflecting the RHOAI pattern of enabling user-defined benchmarks alongside the standard suite.

The multi-architecture build pipeline is noteworthy — the Konflux Dockerfile uses a 5-stage multi-stage build to compile PyTorch and Apache Arrow from source on s390x and ppc64le, while using pre-built wheels on x86_64 and arm64. This adds significant build complexity (4-hour build timeout in the Tekton PipelineRun) but ensures the same evaluation framework runs consistently across all RHOAI-supported architectures. The s390x build requires a custom patch (`patches/s390x/parquet-support.patch`) for Apache Arrow Parquet support.

The container runs as a non-root user (UID 65532) with writable directories for model cache (`hf_home`), evaluation output, custom tasks, and Unitxt catalogs. This follows OpenShift security best practices while still allowing the evaluation process to download and cache models as needed.
