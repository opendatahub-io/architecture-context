## Purpose

**Short**: High-performance AI model inference server built on Intel OpenVINO toolkit, serving models via gRPC, REST, and C API with support for LLM continuous batching, embeddings, reranking, and image generation.

**Detailed**: OpenVINO Model Server (OVMS) is a C++-based inference serving system optimized for deployment on Intel architectures. It loads machine learning models from local filesystems or cloud object storage (S3, Azure Blob, Google Cloud Storage, HuggingFace Hub) and exposes them through multiple API protocols: TensorFlow Serving API (v1), KServe v2 Inference Protocol, OpenAI-compatible API for LLM/embeddings/reranking, and a C shared library API.

The server supports advanced serving features including model versioning with automatic version discovery, dynamic model configuration reloading, DAG (Directed Acyclic Graph) pipelines for multi-model inference workflows, MediaPipe graph-based processing, and custom Python nodes. For large language models, it provides continuous batching with the OpenVINO GenAI library, streaming responses via Server-Sent Events, and tool/function calling support with model-specific parsers for Llama3, Mistral, Hermes3, Phi4, and Qwen3.

In the RHOAI platform, OVMS is deployed as the inference runtime for KServe InferenceService CRs. It serves as the container image that KServe's model serving infrastructure uses to run inference workloads on OpenVINO-optimized models, providing a high-performance alternative to other serving runtimes.

## Architectural Analysis

OpenVINO Model Server employs a modular architecture where server capabilities are decomposed into independent modules (gRPC, HTTP, Metrics, Model Manager, Python, Profiler) that are created by a factory pattern and started in a defined order. The server uses a singleton pattern for its core `Server` instance, with modules registered and managed through a lifecycle that ensures proper startup ordering — metrics first, then servable manager creation, then network listeners, then servable manager start. This ordering ensures the server can respond to health probes immediately while models are still loading.

The dual HTTP server implementation (Drogon framework vs. TensorFlow Serving's net_http) reflects an ongoing transition. Drogon provides superior performance for streaming use cases (LLM token-by-token output via SSE) with separate thread pools for unary and streaming requests. The streaming pool routes `/completions` traffic to MediaPipe graph executors that maintain long-lived connections. CORS support is implemented in the Drogon path with configurable allowed origins, methods, and headers.

The model serving architecture is notable for its storage abstraction layer. The `FileSystem` base class provides a common interface for S3, Azure Blob, GCS, and local filesystems, with credential handling via environment variables. The `ModelManager` runs a watcher thread that polls for configuration changes and model version updates at configurable intervals, enabling hot-reloading without server restarts. Models are versioned by directory structure (model/1/, model/2/), and version policies control which versions are served.

For LLM serving, OVMS integrates deeply with the OpenVINO GenAI library's `ContinuousBatchingPipeline`, which implements paged attention and dynamic batching. The LLM pipeline is exposed via a MediaPipe graph calculator (`HttpLlmCalculator`) rather than a direct code path, allowing the graph framework to manage streaming lifecycle. Model-specific generation configuration builders (Llama3, Mistral, Hermes3, Phi4, Qwen3) handle chat template formatting, tool call parsing, and output structure extraction, making the server compatible with various LLM architectures.

A security consideration is that OVMS itself provides no authentication or authorization — gRPC uses `InsecureServerCredentials()` and REST endpoints have no auth middleware. In RHOAI deployments, authentication is delegated to the `kube-rbac-proxy` sidecar injected by the platform operator, which handles TLS termination and RBAC enforcement before proxying to the OVMS container. The use of `openssl/md5.h` for filesystem checksums is a minor FIPS concern, though MD5 is used here for integrity checking rather than cryptographic security.
