SERVICE MESH CONFIGURATION
════════════════════════════════════════════════════════════════════════════

Operator Pods:
  - Annotation: sidecar.istio.io/inject: "false"
  - Rationale: Operator must communicate with K8s API without mesh interference
  - No Envoy sidecar injected

Training Worker Pods:
  - Mesh injection: User-configurable (depends on namespace labels)
  - If enabled: Pod-to-Pod mTLS via Istio/Linkerd
  - Benefits: Encrypted inter-worker communication, observability
  - Considerations: May add latency to training communication

SECRETS

1. webhook-server-cert
   - Type: kubernetes.io/tls
   - Purpose: TLS certificate for admission webhook server (port 9443)
   - Provisioned by: cert-manager or manual creation
   - Auto-rotation: Depends on provisioner (cert-manager: yes)
   - Mounted to: /tmp/k8s-webhook-server/serving-certs/

2. Image Pull Secrets (user-provided)
   - Type: kubernetes.io/dockerconfigjson
   - Purpose: Authenticate to private container registries
   - Scope: Per namespace
   - Referenced by: Training job pods (spec.template.spec.imagePullSecrets)

3. MPI Job SSH Keys (auto-generated per job)
   - Type: Opaque
   - Purpose: SSH authentication for MPI launcher to workers
   - Lifecycle: Created on job start, deleted on job completion
   - Keys: ssh-privatekey, ssh-publickey

════════════════════════════════════════════════════════════════════════════
SECURITY CONSIDERATIONS
════════════════════════════════════════════════════════════════════════════

1. Pod Security:
   - Operator runs as non-root (UID: 65532, GID: 65532)
   - securityContext.allowPrivilegeEscalation: false
   - No privileged containers

2. Network Policies:
   - Operator needs egress to Kubernetes API (6443/TCP)
   - Training pods need ingress from other pods in same job
   - Prometheus needs ingress to operator (8080/TCP)

3. FIPS Compliance:
   - RHOAI builds use '-tags strictfipsruntime' for FIPS mode
   - TLS 1.2+ enforced for all external communication

4. Authentication:
   - Kubernetes API: ServiceAccount token (JWT)
   - Webhook: Kubernetes API authentication
   - Metrics: No auth (ClusterIP service, internal only)
   - Health probes: No auth (liveness/readiness only)

5. Authorization:
   - RBAC enforced via ClusterRole/ClusterRoleBinding
   - Principle of least privilege (fine-grained permissions)
   - training-edit/training-view roles for user access control

6. Audit Logging:
   - All API operations logged by Kubernetes API server
   - Operator generates Kubernetes Events for job lifecycle

7. Multi-tenancy:
   - Namespace isolation for training jobs
   - Gang scheduler prevents resource contention
   - Optional: Per-namespace operator instances for stronger isolation
