## Purpose
**Short**: ModelMesh is a distributed LRU cache and intelligent routing layer for serving machine learning models at high scale and density.

**Detailed**: ModelMesh is a mature, general-purpose model serving management and routing framework designed for high-scale, high-density, and frequently-changing model use cases. It runs as a sidecar container alongside model runtime containers within Kubernetes Deployments, acting as a distributed LRU cache that manages the lifecycle of models across a cluster of pods.

The framework handles model registration, loading, unloading, and intelligent request routing without requiring awareness of the underlying model format. It communicates with colocated model runtime containers via gRPC (either over localhost TCP or Unix Domain Sockets) using a well-defined `ModelRuntime` service interface. External clients interact with ModelMesh through a gRPC `ModelMesh` management API and through proxied custom inference gRPC interfaces, with model selection indicated via `mm-model-id` or `mm-vmodel-id` gRPC metadata headers.

ModelMesh includes built-in support for model auto-scaling (scaling copies across pods based on request rate or latency), virtual models (vmodels) for blue-green model updates, TLS/mTLS for external and inter-pod communication, Prometheus metrics export, and integration with etcd or ZooKeeper for distributed state management. It is deployed as part of the RHOAI ModelMesh Serving stack, managed by the modelmesh-serving controller.

### gRPC Services

#### ModelMesh Management API (External)

| Service | Port | Protocol | Encryption | Auth | Purpose |
|---------|------|----------|------------|------|---------|
| mmesh.ModelMesh | 8033/TCP | gRPC | TLS (configurable) | mTLS (optional, configurable) | Model lifecycle management and inference request routing |

**RPC Methods:**
- `registerModel` - Register a trained model with the cluster
- `unregisterModel` - Remove a model from the cluster
- `getModelStatus` - Get current status of a model (NOT_FOUND, NOT_LOADED, LOADING, LOADED, LOADING_FAILED)
- `ensureLoaded` - Ensure a model is loaded in the cluster
- `setVModel` - Create or update a virtual model mapping
- `deleteVModel` - Delete a virtual model
- `getVModelStatus` - Get status of a virtual model

#### ModelRuntime API (Internal, Sidecar)

| Service | Port | Protocol | Encryption | Auth | Purpose |
|---------|------|----------|------------|------|---------|
| mmesh.ModelRuntime | 8085/TCP or UDS | gRPC | plaintext | None | Communication with colocated model runtime container |

**RPC Methods (implemented by model runtimes):**
- `loadModel` - Load a model into memory
- `unloadModel` - Unload a model from memory
- `predictModelSize` - Predict memory consumption of unloaded model
- `modelSize` - Calculate memory consumption of loaded model
- `runtimeStatus` - Report runtime status, capacity, and configuration

The startup script (`start.sh`) explicitly disables Java FIPS mode (`-Dcom.redhat.fips=false`) to avoid TLS breakage. This is required for the current TLS implementation using Netty's native SSL support (BoringSSL/OpenSSL).
