================================================================================
  AUTHENTICATION MECHANISMS
================================================================================

1. OAuth2 JWT (JWKS):
   - Token: Bearer JWT in Authorization header
   - Validation: JWKS endpoint public key verification
   - Claims: sub, groups, team, project, tenant
   - Maps to: roles, teams, projects, namespaces (access_attributes)

2. OAuth2 Introspection:
   - Token: Bearer token in Authorization header
   - Validation: RFC 7662 POST to introspection endpoint
   - Auth: Client ID + Client Secret
   - Maps to: access_attributes from introspection response

3. Custom Endpoint:
   - Token: Bearer token in Authorization header
   - Validation: POST to custom auth URL with API key
   - Response: AuthResponse with principal + access_attributes

4. Kubernetes TokenReview:
   - Token: Bearer token (ServiceAccount token)
   - Validation: Kubernetes TokenReview API
   - Maps to: K8s user identity

================================================================================
  ACCESS CONTROL (ABAC)
================================================================================

- Policy Type: Attribute-Based Access Control (Cedar-like)
- Actions: create, read, update, delete
- Resource Types: model, vector_db, shield, dataset, benchmark, tool_group
- Conditions: when/unless clauses with attribute matching
- Evaluation: First match wins, default DENY
- Principals: User identifiers from authentication provider

================================================================================
  TLS CONFIGURATION
================================================================================

- Server TLS: Optional (tls_certfile + tls_keyfile)
- Mutual TLS: Supported (tls_cafile enables ssl.CERT_REQUIRED)
- No Kubernetes Ingress/Gateway defined (externally managed)
- Egress TLS: Varies by destination (see individual service entries)
