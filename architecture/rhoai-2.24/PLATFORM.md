### Platform Description

Red Hat OpenShift AI (RHOAI) 2.24 is a comprehensive AI/ML platform that provides an end-to-end environment for developing, training, serving, and monitoring machine learning models on OpenShift. The platform is orchestrated by a central operator (`rhods-operator`) that manages 45 components spanning:

- **Development Environments**: JupyterLab, Code Server, and RStudio workbench images with GPU support (CUDA, ROCm, Gaudi)
- **Model Serving**: KServe (serverless + raw deployment), ModelMesh (multi-model), vLLM (multiple accelerator variants), OpenVINO, TGIS, and Caikit-NLP runtimes
- **LLM Inference Optimization**: llm-d framework with Gateway API Inference Extension, disaggregated prefill/decode, KV-cache-aware routing
- **Pipeline Orchestration**: Kubeflow Pipelines v2 with Argo Workflows execution backend
- **Distributed Training**: Kubeflow Training Operator (PyTorchJob, TFJob, etc.), KubeRay, CodeFlare, Kueue job queuing
- **Model Management**: Model Registry, ML Metadata, Feast Feature Store
- **AI Safety**: FMS Guardrails Orchestrator with detector pipelines, TrustyAI fairness/bias monitoring, LM Evaluation Harness
- **LLM Application Framework**: Llama Stack API server with pluggable providers
- **Fine-Tuning**: InstructLab pipeline for LLM fine-tuning on OpenShift

## Data Flows

### Flow 1: Model Inference (KServe Serverless)

```
Client → OpenShift Route (443/HTTPS)
  → Knative Ingress Gateway (Istio)
    → Istio Sidecar (mTLS)
      → Inference Runtime (vllm/OVMS/TGIS/caikit)
        → GPU/CPU (inference)
      → Response back through chain
```

### Flow 2: Model Inference (KServe RawDeployment with Gateway API)

```
Client → Gateway (HTTPRoute, 80/HTTP)
  → Envoy ext-proc → llm-d EPP (9002/gRPC)
    → EPP selects optimal pod (KV-cache aware)
  → Envoy → selected vLLM pod (8000/HTTP)
    → [Optional] llm-d-routing-sidecar (P/D disaggregation)
      → Remote Prefiller (prefill phase)
      → Local Decoder (decode phase)
    → Response
```

### Flow 3: Model Inference (ModelMesh Multi-Model)

```
Client → Service (8008/REST or 8033/gRPC)
  → rest-proxy (8008→8033 translation, if REST)
  → ModelMesh router (selects pod with model loaded)
    → modelmesh-runtime-adapter (8085/gRPC)
      → Inference Runtime (OVMS/MLServer/Triton/TorchServe)
    → Response
  etcd ← ModelMesh (state synchronization)
```

### Flow 4: Pipeline Execution (DSP)

```
User → Dashboard / API (8888/REST)
  → data-science-pipelines API Server
    → MariaDB (3306) [pipeline metadata]
    → ml-metadata (8080/gRPC) [artifact lineage]
    → argo-workflows controller
      → Workflow pods (pipeline steps)
        → S3/MinIO (artifact storage)
        → Model Serving endpoints (inference)
```

### Flow 5: Guardrailed Inference

```
Client → vllm-orchestrator-gateway (8090/HTTP)
  → FMS Guardrails Orchestrator (8033/HTTP)
    → Input Detectors (8080/8000/HTTP) [PII, toxicity, etc.]
    → LLM Backend (vLLM/TGIS/Caikit) [generation]
    → Output Detectors [safety checks]
  → Response (with detection metadata or fallback message)
```

### Flow 6: InstructLab Fine-Tuning Pipeline

```
User → DSP API (pipeline submission)
  → SDG Component → Teacher Model (KServe InferenceService)
    → Synthetic data generation
  → Training Component → PyTorchJob (Training Operator)
    → Distributed GPU training (multi-node)
  → Eval Component → Judge Model (KServe InferenceService)
    → MT-Bench + MMLU evaluation
  → Output → OCI Registry / Model Registry
```

### Flow 7: Workbench Access

```
User Browser → Gateway/Route (443/HTTPS)
  → kube-rbac-proxy sidecar (8443/HTTPS) [TokenReview]
    → JupyterLab/Code Server/RStudio (8888/8787/HTTP)
```

## Deployment Architecture

### OLM Deployment

The platform is deployed via OLM (Operator Lifecycle Manager):

1. **File-Based Catalog (FBC)** published to registry containing operator bundle
2. **CatalogSource** references the FBC catalog image
3. **Subscription** triggers OLM to install the `rhods-operator`
4. **ClusterServiceVersion (CSV)** contains 73 `RELATED_IMAGE` environment variables for disconnected support
5. **rhods-operator** deploys all components via kustomize overlays based on `DataScienceCluster` CR

## Next Steps

### Recommended Actions

1. **Network Policy Audit**: Review and standardize NetworkPolicy creation across components (currently fragmented across DSP operator, codeflare, llama-stack operator, training-operator)
2. **TLS Standardization**: Align TLS configuration patterns — some components use OpenShift service-ca (feast), some use cert-manager, some use self-signed
3. **Auth Consolidation**: Multiple auth patterns coexist (OAuth proxy, kube-rbac-proxy, Istio mTLS, Bearer token, JWT) — consider consolidating where possible
4. **Gateway API Migration**: Track migration from Knative/Istio ingress to Gateway API (GIE) for model serving
5. **Observability Gaps**: Not all components export OpenTelemetry traces — extend OTEL coverage to DSP, ModelMesh, training operators

### Known Architectural Considerations

- **llm-d framework** introduces a separate Istio control plane (`istiod-llm-d-gateway`) which adds operational complexity
- **ModelMesh etcd** dependency requires careful stateful management and backup
- **vllm multi-variant images** (CUDA, CPU, ROCm, Gaudi) share most code but require separate build pipelines
- **TGIS deprecation** path requires ensuring vllm-tgis-adapter covers all TGIS protocol features
- **Gateway API Inference Extension** is pre-GA upstream (v1alpha2) — monitor for breaking changes
