TRUST BOUNDARIES
═══════════════════════════════════════════════════════════════════════════

1. External → Service Layer
   - Trust: Untrusted
   - Encryption: None (plaintext gRPC/HTTP on 8033/8008)
   - Auth: Optional application-level
   - Recommendation: Use OpenShift Routes with TLS termination or Service Mesh

2. Service Layer → Runtime Pods
   - Trust: Cluster-internal
   - Encryption: None (plaintext)
   - Auth: None
   - Recommendation: Enable Service Mesh mTLS for pod-to-pod encryption

3. Runtime Pods → ETCD
   - Trust: Cluster-internal or external
   - Encryption: Optional TLS (configurable)
   - Auth: Optional mTLS or username/password
   - Recommendation: Enable TLS and mTLS for production

4. Runtime Pods → S3 Storage
   - Trust: External service
   - Encryption: TLS 1.2+ (mandatory)
   - Auth: AWS Signature V4 / IAM credentials
   - Secure: Yes (industry standard)

5. Controller → Kubernetes API
   - Trust: Cluster-internal (trusted)
   - Encryption: TLS 1.2+ (mandatory)
   - Auth: ServiceAccount token
   - Secure: Yes

6. Kubernetes API → Webhook Server
   - Trust: Cluster-internal (trusted)
   - Encryption: TLS 1.2+ (mandatory)
   - Auth: Kubernetes API Server client certificate
   - Secure: Yes
