Data Flow Security:
═══════════════════

External User → Model Registry (OAuth Mode):
1. User → OpenShift Router: HTTPS/443 TLS1.2+ (no auth)
2. Router → Route: Edge Termination
3. Route → Service: HTTPS/8443 TLS1.2+
4. Service → oauth-proxy: HTTPS/8443 TLS1.2+ Bearer Token
5. oauth-proxy → OpenShift OAuth: HTTPS/443 TLS1.2+ (token validation)
6. oauth-proxy → rest-container: HTTP/8080 (localhost, no encryption)
7. rest-container → Database: PostgreSQL/5432 or MySQL/3306, TLS optional, username/password

Internal Pod → Model Registry (Non-OAuth Mode):
1. Pod → Service: HTTP/8080 (no encryption, no auth)
2. Service → rest-container: HTTP/8080
3. rest-container → Database: PostgreSQL/5432 or MySQL/3306, TLS optional, username/password

Operator Reconciliation:
1. controller-manager → Kubernetes API: HTTPS/6443 TLS1.2+ ServiceAccount Token
2. Kubernetes API → webhook-service: HTTPS/443 TLS Webhook Client Certificate

Metrics Collection:
1. Prometheus → controller-manager-metrics-service: HTTPS/8443 TLS Bearer Token

Security Best Practices:
════════════════════════

1. Use OAuth mode for external access (spec.oauthProxy.serviceRoute=enabled)
2. Enable database TLS (spec.postgres.sslMode=require or verify-full)
3. Use OpenShift Service CA for automatic certificate rotation
4. Restrict RBAC permissions per ModelRegistry instance
5. Network policies restrict ingress to OpenShift Router only
6. All containers run as non-root with seccomp=RuntimeDefault
7. Deployments use restricted-v2 SCC (OpenShift)
8. Database credentials stored in secrets, not plaintext
9. OAuth proxy session cookies are encrypted
10. Webhook server uses TLS with CA validation
