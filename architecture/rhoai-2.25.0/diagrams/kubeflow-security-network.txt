Security Considerations:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  ⚠️  CRITICAL LIMITATIONS:

  1. Culling Feature Security:
     - Requires UNAUTHENTICATED access to /api/kernels endpoint on notebook pods
     - No mutual TLS or token validation for culling checks
     - Risk: Any pod with network access can query kernel status
     - Mitigation: Network policies, service mesh policies

  2. Metrics Endpoint:
     - Exposed over unencrypted HTTP (8080/TCP)
     - No authentication required
     - Mitigation: ClusterIP service (internal only), network policies

  3. Health Probes:
     - Exposed over unencrypted HTTP (8081/TCP)
     - No authentication required
     - Mitigation: ClusterIP service (internal only)

  4. Notebook Pod Security:
     - User-controlled pod specifications (can run as root if configured)
     - No built-in PodSecurityPolicy enforcement by controller
     - Mitigation: Cluster-level PodSecurityStandards, admission controllers

  5. Data Persistence:
     - No built-in backup/restore for notebook data
     - Relies on user-configured PVCs
     - Risk: Data loss if PVC not configured properly

  ✅ SECURITY STRENGTHS:

  1. ServiceAccount Token Authentication:
     - All Kubernetes API operations use JWT tokens
     - RBAC enforced for controller operations

  2. TLS for API Communication:
     - All controller → Kubernetes API calls use TLS 1.2+
     - Certificate validation enabled

  3. Non-root Container:
     - Controller runs as user 1001 (non-root)
     - FIPS compliance with strictfipsruntime

  4. Least Privilege RBAC:
     - Separate ClusterRoles for controller, leader election, user access
     - Granular permissions for each role

  5. Leader Election:
     - Prevents split-brain in HA deployments
     - ConfigMap-based locking
