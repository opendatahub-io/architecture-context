DATA FLOWS - SECURITY DETAILS
═══════════════════════════════════════════════════════════════════════════

Flow 1: RayCluster Creation
  User → K8s API (443/TCP HTTPS TLS1.2+ Bearer Token)
  K8s API → Webhook (9443/TCP HTTPS TLS1.2+ mTLS)
  Operator → K8s API (443/TCP HTTPS TLS1.2+ ServiceAccount Token)
  Workers → Head GCS (6379/TCP TCP None Optional Redis Password)
  Prometheus → Operator (8080/TCP HTTP None None)

Flow 2: RayJob Submission
  User → K8s API (443/TCP HTTPS TLS1.2+ Bearer Token)
  Operator → K8s API (443/TCP HTTPS TLS1.2+ ServiceAccount Token)
  Submitter → Dashboard (8265/TCP HTTP None Optional Token)

Flow 3: RayService Serving
  Client → Ray Serve (8000/TCP HTTP None Application-level)
  Operator → Head (8265/TCP HTTP None Optional Token)
  Operator → K8s API (443/TCP HTTPS TLS1.2+ ServiceAccount Token)

Flow 4: Autoscaling
  Ray Autoscaler → K8s API (443/TCP HTTPS TLS1.2+ Ray SA Token)
  New Worker → GCS (6379/TCP TCP None Optional Redis Password)

═══════════════════════════════════════════════════════════════════════════
SECURITY RECOMMENDATIONS
═══════════════════════════════════════════════════════════════════════════

1. Enable Ray Dashboard authentication (--auth-token)
2. Use external Redis with password for GCS fault tolerance
3. Enable TLS on OpenShift Routes/Ingress (Edge or Passthrough)
4. Implement NetworkPolicies for pod-to-pod traffic isolation
5. Use ImagePullSecrets for private container registries
6. Enable cert-manager for automatic webhook certificate rotation
7. Monitor operator and Ray cluster metrics for security events
8. Apply resource quotas to limit Ray cluster resource consumption
9. Use RBAC to restrict which users can create RayCluster/Job/Service CRs
10. Audit Ray Dashboard access logs for unauthorized access attempts
