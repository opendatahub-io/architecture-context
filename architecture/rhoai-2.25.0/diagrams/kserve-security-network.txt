═══════════════════════════════════════════════════════════════════════════
NETWORK POLICIES
═══════════════════════════════════════════════════════════════════════════

Note: KServe relies on service mesh (Istio) for network security rather than
traditional Kubernetes NetworkPolicies. All traffic within the mesh is mTLS
encrypted with service identity-based authorization.

Ingress Rules (enforced by Istio):
├─ External → Istio Gateway: 443/TCP HTTPS (TLS 1.2+)
├─ Gateway → Knative Services: 80/TCP HTTP (mTLS STRICT)
├─ Activator → Predictor Pods: 80/TCP HTTP (mTLS STRICT)
└─ Prometheus → Metrics endpoints: 8080/TCP HTTP (ClusterIP only)

Egress Rules (enforced by Istio):
├─ Storage Initializer → S3/GCS/Azure: 443/TCP HTTPS (TLS 1.2+)
├─ Controller → Kubernetes API: 6443/TCP HTTPS (TLS 1.2+)
├─ Agent → Event Sink: 80/443 HTTP/HTTPS
└─ Allow all egress (default) - can be restricted with Istio ServiceEntry

═══════════════════════════════════════════════════════════════════════════
SECURITY CONTROLS
═══════════════════════════════════════════════════════════════════════════

1. Transport Encryption:
   - External traffic: TLS 1.2+ (443/TCP)
   - Service mesh: mTLS STRICT (all pod-to-pod)
   - Storage download: TLS 1.2+ (S3/GCS/Azure)
   - Kubernetes API: TLS 1.2+ (6443/TCP)
   - Localhost: Plaintext (agent ↔ model-server)

2. Authentication:
   - External clients: Optional Bearer Token (JWT)
   - Service mesh: mTLS certificates (service identity)
   - Storage: AWS IAM, GCP Service Account, Azure credentials
   - Kubernetes API: Service Account tokens
   - Webhooks: Kubernetes API Server certificates

3. Authorization:
   - RBAC: ClusterRole-based permissions for controller
   - Istio AuthorizationPolicy: Path and method-based access control
   - Storage IAM: Cloud provider IAM policies

4. Recent Security Fixes (RHOAI 2.25):
   - CVE mitigation: Path traversal vulnerability fixes in storage downloader
   - Hardening: Root path rejection in HTTPS storage component
   - Image optimization: Removal of unnecessary files from runtime containers

═══════════════════════════════════════════════════════════════════════════
DEPLOYMENT MODES
═══════════════════════════════════════════════════════════════════════════

1. Serverless Mode (default with Knative):
   - Autoscaling: Scale-to-zero, Knative Autoscaler (KPA)
   - Traffic: Istio Gateway → VirtualService → Knative Service → Activator → Pod
   - Benefits: Resource efficiency, automatic scaling

2. RawDeployment Mode (without Knative):
   - Autoscaling: Kubernetes HPA or KEDA ScaledObject
   - Traffic: Istio Gateway → VirtualService → Deployment → Pod
   - Benefits: Simpler dependencies, no Knative required

3. LLM Multi-Node Mode (LeaderWorkerSet):
   - Autoscaling: Manual or custom scaling
   - Traffic: Gateway → Leader Pod → Worker Pods (Ray cluster)
   - Benefits: Distributed inference for large models (vLLM, TensorRT-LLM)
