Service Mesh / mTLS Configuration:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

Pod-to-Pod TLS (podToPodTLS):
  - Default: Enabled (true) for DSP v2 on OpenShift
  - Mechanism: TLS between DSPA components (API Server ↔ MLMD gRPC)
  - Certificate: Managed by DSPO (ds-pipeline-metadata-grpc-tls-config-{name})
  - Scope: Application-level TLS, not service mesh

Service Mesh Integration (Optional):
  - If OSSM/Istio is deployed, DSP components can be included in service mesh
  - Provides mTLS for all pod-to-pod traffic automatically
  - Envoy sidecar proxies handle TLS termination and authentication
  - Compatible with podToPodTLS setting

Trust Boundaries:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. External → Ingress: TLS 1.2+ (OpenShift Routes), OAuth authentication
2. Ingress → Application: TLS 1.2+ (Reencrypt mode), OAuth proxy validation
3. Application Internal: Optional TLS (podToPodTLS), bearer tokens, DB/S3 auth
4. Application → Control Plane: TLS 1.2+ (Kubernetes API), service account tokens
5. Application → External Services: TLS 1.2+ (S3, external DB), credentials
