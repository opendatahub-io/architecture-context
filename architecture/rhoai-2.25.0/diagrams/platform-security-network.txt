SERVICE MESH CONFIGURATION
═══════════════════════════════════════════════════════════════════════════════════════

mTLS Modes:
- PERMISSIVE (default): Accepts both plaintext and mTLS connections
- STRICT (opt-in): Requires mTLS for all connections within namespace

PeerAuthentication:
- Applied per namespace for InferenceService predictor pods
- Istio CA issues X.509 certificates with 90-day rotation
- SPIFFE identity: cluster.local/ns/{namespace}/sa/{serviceaccount}

AuthorizationPolicies (when enabled):
- JWT validation via Authorino or Kuadrant AuthPolicy
- Principal-based access control (user must have namespace access)
- Path-based rules: /v1/models/:predict, /v2/models/:infer, /metrics

Traffic Management:
- Load balancing: Round Robin (default)
- Circuit breaking: Configurable per InferenceService
- Timeout: 300s default for inference requests
- Retry: 3 attempts default with exponential backoff

AUTHENTICATION MECHANISMS
═══════════════════════════════════════════════════════════════════════════════════════

1. OpenShift OAuth (User-Facing Services):
   - Components: Dashboard, Model Registry UI, DSPA UI, TrustyAI API, Ray Dashboard, Notebooks
   - Flow: User → OpenShift OAuth Server → OAuth Proxy Sidecar → Application
   - Token: Bearer JWT token (OpenShift-issued)
   - Enforcement: OAuth Proxy validates token via TokenReview API

2. Kubernetes RBAC (ServiceAccount Tokens):
   - Components: All operators and workloads
   - Flow: Pod → Kubernetes API Server with ServiceAccount JWT
   - Token: Projected ServiceAccount token (auto-mounted)
   - Enforcement: Kubernetes API Server validates and checks RBAC

3. Istio mTLS (Service Mesh):
   - Components: KServe predictors, ModelMesh pods (when service mesh enabled)
   - Flow: Client pod → Istio sidecar → Server Istio sidecar → Server pod
   - Certificate: X.509 client certificate (Istio CA-issued)
   - Enforcement: Istio sidecar validates peer certificate (SPIFFE identity)

4. Webhook mTLS (Admission Control):
   - Components: All operator admission webhooks
   - Flow: Kubernetes API Server → Operator webhook endpoint
   - Certificate: API Server client certificate + Operator server certificate
   - Enforcement: Mutual TLS validation

5. AWS IAM (S3 Access):
   - Components: KServe, DSPA, Training jobs, notebooks
   - Methods: Static credentials (AWS_ACCESS_KEY_ID/SECRET) or IRSA (IAM Roles for Service Accounts)
   - Flow: Pod → AWS STS (AssumeRoleWithWebIdentity) → S3 API
   - Enforcement: AWS S3 validates signature

6. Database Authentication:
   - Components: Model Registry, DSPA, Feast
   - Methods: Username/password or client certificates (PostgreSQL/MySQL)
   - Enforcement: Database server validates credentials

7. Bearer Tokens (API Calls):
   - Components: Dashboard → Kubernetes API, external clients → KServe inference
   - Flow: Client → Service with Authorization: Bearer {token}
   - Enforcement: Token validation (JWT signature, expiry, claims)

8. None / Network Isolation (Internal Services):
   - Components: Inter-component communication within cluster (e.g., operator → webhook)
   - Enforcement: Kubernetes NetworkPolicies, namespace isolation

NETWORK SECURITY ZONES
═══════════════════════════════════════════════════════════════════════════════════════

Zone 1 - External (Untrusted):
  - Threat Level: HIGH
  - Trust: Zero trust, all traffic considered malicious until verified
  - Enforcement: OpenShift Router TLS termination, OAuth authentication

Zone 2 - Ingress/DMZ:
  - Threat Level: MEDIUM-HIGH
  - Trust: TLS-terminated traffic, OAuth-validated users
  - Enforcement: Istio AuthorizationPolicy, OpenShift RBAC

Zone 3 - Platform Control Plane:
  - Threat Level: MEDIUM
  - Trust: Operator ServiceAccounts with ClusterRole permissions
  - Enforcement: Kubernetes RBAC, admission webhooks

Zone 4 - User Workload Namespaces:
  - Threat Level: MEDIUM
  - Trust: User workloads with namespace-scoped permissions
  - Enforcement: Namespace isolation, ResourceQuotas, NetworkPolicies

Zone 5 - Service Mesh (Internal):
  - Threat Level: LOW
  - Trust: mTLS-authenticated service-to-service communication
  - Enforcement: Istio PeerAuthentication (STRICT mode)

Zone 6 - Shared Infrastructure:
  - Threat Level: LOW
  - Trust: Platform infrastructure components (Istio, Knative, Prometheus)
  - Enforcement: ClusterRole permissions, namespace isolation

Zone 7 - External Services (Egress):
  - Threat Level: MEDIUM
  - Trust: TLS-encrypted, credential-authenticated external services
  - Enforcement: AWS IAM, cloud provider auth, registry credentials

═══════════════════════════════════════════════════════════════════════════════════════
COMPLIANCE & SECURITY POSTURE
═══════════════════════════════════════════════════════════════════════════════════════

FIPS Compliance:
- All operators built with FIPS-compliant Go runtime
- Crypto operations use crypto/rand (not math/rand)
- TLS 1.2+ enforced (TLS 1.0/1.1 disabled)

Container Security:
- All containers run as non-root (UID > 1000)
- Read-only root filesystems where possible
- Dropped capabilities (CAP_NET_RAW, CAP_SYS_ADMIN)
- Security Context Constraints (SCCs) enforced

Network Security:
- Default deny NetworkPolicies (optional)
- mTLS for service mesh communication (STRICT mode available)
- TLS termination at ingress (OpenShift Router)
- No plaintext protocols exposed externally

Secret Management:
- Secrets encrypted at rest (etcd encryption)
- Kubernetes Secrets for credentials (base64 encoded)
- ServiceAccount token projection (limited scope, expiry)
- cert-manager for certificate lifecycle (90-day rotation)

Audit & Logging:
- Kubernetes audit logs (API server events)
- Operator event logging (reconciliation failures, warnings)
- Istio access logs (request/response metadata)
- Prometheus metrics (ServiceMonitor for all operators)

Vulnerability Management:
- Regular base image updates (UBI9 security patches)
- Go version updates for CVE fixes (1.25+ in RHOAI 2.25.0)
- govulncheck integration in CI/CD pipelines
- Dependency scanning (renovate, Dependabot)

Multi-Tenancy Isolation:
- Namespace-based isolation (ResourceQuotas, LimitRanges)
- RBAC separation (users cannot access other namespaces)
- Workload queueing (Kueue ClusterQueue quotas)
- Network isolation (NetworkPolicies, service mesh boundaries)
