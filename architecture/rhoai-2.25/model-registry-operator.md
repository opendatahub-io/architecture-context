## Purpose

**Short**: Manages the lifecycle of Model Registry instances on OpenShift, deploying and configuring REST and gRPC services backed by PostgreSQL or MySQL with optional OAuth proxy authentication.

**Detailed**: The Model Registry Operator is a Kubernetes operator built with controller-runtime that watches `ModelRegistry` custom resources and reconciles the desired state into a set of dynamically-created Kubernetes resources: Deployments (running the model-registry REST proxy and MLMD gRPC server containers), Services, ServiceAccounts, Roles, RoleBindings, OpenShift Routes, NetworkPolicies, and Secrets. Each ModelRegistry instance represents a distinct model registry deployment with its own database connection, optional OAuth proxy sidecar for authentication, and optional OpenShift Route exposure.

The operator also includes an optional Model Catalog controller (enabled via the `ENABLE_MODEL_CATALOG` environment variable) that deploys a singleton catalog service providing a browsable index of ML models. The catalog uses an init container to load model metadata from a curated data image, and supports the same OAuth proxy pattern as the main registry.

A storage version migration subsystem handles the CRD version transition from v1alpha1 (deprecated) to v1beta1 (current storage version). The migration uses a strategy chain: it first attempts the Kubernetes-native `StorageVersionMigration` API, and falls back to a manual re-read/re-write strategy that triggers conversion webhooks. The operator supports both OpenShift and vanilla Kubernetes clusters, detecting OpenShift at startup via API group discovery and conditionally enabling Route management and OpenShift-specific auth features.

## Architectural Analysis

The Model Registry Operator follows a clean template-driven resource creation pattern. Rather than constructing Kubernetes resources programmatically in Go structs, the operator embeds YAML templates (19 total `.yaml.tmpl` files) and executes them with Go's `text/template` engine, passing the ModelRegistry spec as template parameters. This approach provides clear separation between resource definitions and controller logic, and makes it straightforward to see exactly what resources will be created by reading the templates. The `TemplateApplier` utility handles template execution and YAML unmarshaling, while the `CreateOrUpdate` pattern uses `k8s-objectmatcher` for deterministic diffing to avoid unnecessary updates.

The OAuth proxy integration is implemented as a conditional sidecar pattern. When `spec.oauthProxy` is configured, the operator injects an `oauth-proxy` container into the ModelRegistry Deployment alongside the REST container, creates a ClusterRoleBinding for `system:auth-delegator` to enable token review, generates a random cookie secret, and creates an HTTPS Route with `reencrypt` TLS termination. The OAuth proxy forwards authenticated requests to the REST container on localhost. On OpenShift, TLS certificates are auto-provisioned via the serving cert controller annotation; on non-OpenShift clusters, users must provide a TLS secret manually. When OAuth is disabled, the operator cleans up all auth-related resources (ClusterRoleBinding, Route, NetworkPolicy) via explicit delete operations in the reconciler.

The CRD version migration from v1alpha1 to v1beta1 is a notable architectural feature. The v1alpha1 API supported Istio service mesh integration (with Gateway, DestinationRule, and AuthConfig resources), while v1beta1 removes Istio support entirely in favor of the simpler OAuth proxy pattern. The conversion webhook automatically replaces Istio configuration with equivalent OAuthProxy defaults (port 8443, route enabled). The storage migration manager runs as a background goroutine, waits 30 seconds for the operator to initialize, then checks if v1alpha1 is still in the CRD's `status.storedVersions`. If so, it attempts migration via the Kubernetes-native `StorageVersionMigration` API first, and falls back to manually re-reading/re-writing all ModelRegistry resources in the target version. This dual-strategy approach ensures migration works even on clusters without the SVM API.

The Model Catalog controller uses a singleton pattern — regardless of how many ModelRegistry instances exist, only one `model-catalog` Deployment is created. The controller maps all watched resources to a single reconciliation request (`model-catalog/{targetNamespace}`) to prevent reconcile storms. Its ownership model is unusual: the ServiceAccount and Deployment are owned by the platform-level `default-modelregistry` CR (from `components.platform.opendatahub.io`), while downstream resources (Service, Role, Route, NetworkPolicy) are owned by the Deployment for cascading deletion. ClusterRoleBindings and Secrets have no owner reference and require manual cleanup.

The operator's error detection includes a distinctive pod log scraping mechanism. During status reconciliation, the controller reads recent log lines from the gRPC (MLMD) and REST containers of the managed Deployment, applying regex patterns to detect database connection failures, schema migration errors, and alert markers. This provides actionable error messages in the ModelRegistry status conditions rather than generic "unavailable" states. While unconventional compared to structured health checks, this approach catches configuration errors (wrong database credentials, incompatible schema versions) that health probes alone cannot surface.
