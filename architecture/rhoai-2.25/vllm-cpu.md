## Purpose

**Short**: CPU-only build of vLLM, a high-throughput and memory-efficient inference and serving engine for large language models, packaged for Red Hat OpenShift AI.

**Detailed**: vllm-cpu is a downstream fork of the open-source [vLLM project](https://github.com/vllm-project/vllm), purpose-built for CPU-only inference on OpenShift AI. It provides an OpenAI-compatible REST API server for serving LLM completions, chat, embeddings, classification, scoring, reranking, tokenization, and audio transcription — all without GPU hardware. The CPU variant is optimized for x86_64, ppc64le, s390x, and aarch64 architectures, using platform-specific acceleration libraries (Intel Extension for PyTorch on x86_64, OpenBLAS on ppc64le/s390x).

The Konflux Dockerfile produces two image stages: a primary **OpenAI-compatible HTTP server** (`vllm-openai`) and a **gRPC adapter** (`vllm-grpc-adapter`) that wraps vLLM with the [vllm-tgis-adapter](https://github.com/IBM/vllm-tgis-adapter) for TGIS (Text Generation Inference Service) protocol compatibility. Both images run as non-root user (UID 2000) on UBI 9 minimal base. The server is typically deployed as a KServe `InferenceService` runtime within the RHOAI model serving stack.

### FIPS Compliance

FIPS compliance has two layers: **build-time enforcement** (check-payload validation in Konflux) and **application-level crypto correctness** (what the code actually does with cryptography). Both must be documented.

#### Build-Time FIPS (check-payload gate)

Konflux validates images via [check-payload](https://github.com/openshift/check-payload), which enforces:
- Go binaries must be **dynamically linked** (CGO_ENABLED=1, no static linking)
- **OpenSSL must be installed** in the container image
- No statically linked Go crypto (pure-Go crypto / BoringCrypto static gets rejected)

check-payload is the bare minimum — it validates build artifacts but not runtime behavior. Document what the application actually does with cryptography:

| Aspect | Value | Source |
|--------|-------|--------|
| **TLS configuration** | Uses Python `ssl.SSLContext` via Uvicorn; TLS is optional and configured via CLI flags (`--ssl-keyfile`, `--ssl-certfile`); default is `ssl.CERT_NONE` for client cert validation | `vllm/entrypoints/openai/cli_args.py:119`, `vllm/entrypoints/ssl.py:1-76` |

| **Non-FIPS crypto risks** | Uses `hashlib.md5(usedforsecurity=False)` extensively for cache key hashing in compilation and config modules — acceptable as MD5 is used for non-security checksums only | `vllm/compilation/backends.py:514,523`, `vllm/envs.py:1269`, `vllm/config/__init__.py:1824,1872,2004,2439,2549` |

_The Python runtime inherits FIPS-compliant OpenSSL from the UBI 9 base image. When FIPS mode is enabled at the OS level, Python's `ssl` and `hashlib` modules automatically use FIPS-validated crypto. The `hashlib.md5(..., usedforsecurity=False)` calls are compliant because Python 3.9+ allows non-security MD5 usage in FIPS mode._

## Architectural Analysis

The vllm-cpu repository is a downstream fork of the vLLM project, targeting CPU-only inference for Red Hat OpenShift AI. The key architectural decision is the dual-entrypoint Dockerfile: a single `Dockerfile.konflux.cpu` builds two container stages — the primary OpenAI-compatible server (`vllm-openai`) and a TGIS gRPC adapter (`vllm-grpc-adapter`). This avoids duplicating the base image build while supporting two distinct serving protocols. The TGIS adapter stage merely installs the `vllm-tgis-adapter` package on top of the OpenAI stage and overrides the entrypoint.

The build system is complex and multi-architecture. The Dockerfile uses architecture-specific build scripts (`build_vllm_ppc64le.sh`, `build_vllm_s390x.sh`) that compile dependencies from source — including OpenBLAS, LAPACK, and numactl — to optimize for each CPU platform. On x86_64, Intel Extension for PyTorch provides acceleration. The builder stage caches compiled wheels and system libraries, which the final stage consumes via `--mount=type=cache` for efficient layer reuse.

The vLLM engine supports two execution paths: V0 (legacy) and V1 (new). The image defaults to `VLLM_USE_V1=1`, which uses the new `AsyncLLM` engine with improved multiprocessing and memory management. The V0 path supports the older `MQLLMEngineClient` with ZeroMQ IPC for multi-process coordination. Both paths expose the same FastAPI HTTP API surface.

Security is configured but optional at the application level. TLS is disabled by default (`ssl.CERT_NONE`) but can be enabled via `--ssl-keyfile` and `--ssl-certfile` flags, with an innovative `SSLCertRefresher` that uses the `watchfiles` library to monitor certificate files and hot-reload them when they change (supporting cert-manager rotation scenarios). API key authentication uses constant-time SHA-256 hash comparison to prevent timing attacks. In production RHOAI deployments, TLS termination and authentication are typically handled by the KServe/Istio ingress layer rather than vLLM itself.

A notable FIPS consideration: while the UBI 9 base image provides FIPS-validated OpenSSL, the code extensively uses `hashlib.md5(usedforsecurity=False)` for cache key computation in compilation and configuration modules. This is compliant with Python's FIPS-aware hashlib (Python 3.9+), but developers modifying this code should be aware that removing the `usedforsecurity=False` flag would break FIPS mode.
