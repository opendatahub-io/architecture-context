## Purpose

**Short**: ML Metadata (MLMD) is a gRPC service for recording, retrieving, and querying metadata (artifacts, executions, contexts, and their lineage) associated with ML workflows.

**Detailed**: ML Metadata provides a structured metadata store that tracks the provenance and lineage of machine learning artifacts, executions, and their relationships. It is a core component of the Kubeflow Pipelines ecosystem, used by pipeline orchestrators and ML platforms to record what artifacts were produced, what executions ran, and how they relate through events and contexts.

The component is shipped as a standalone gRPC server (`metadata_store_server`) built from C++ using Bazel. It supports multiple database backends: MySQL/MariaDB, PostgreSQL, and SQLite (including an in-memory "fake" database for testing). The server exposes the `MetadataStoreService` gRPC API, which provides CRUD operations for artifacts, executions, contexts, types, events, attributions, associations, parent contexts, and lineage graph queries. The Python client library (`ml-metadata` PyPI package) provides both a direct database connection mode (via pybind11 C++ bindings) and a gRPC client mode.

In RHOAI, this component is deployed as the `odh-mlmd-grpc-server` container image, providing the metadata backend for Data Science Pipelines. It typically connects to a MariaDB or PostgreSQL database provisioned by the platform operator.

## Architectural Analysis

ML Metadata follows a clean separation between its gRPC service layer, metadata store logic, and database backends. The `MetadataStoreServiceImpl` creates a new database connection per RPC call (via `ConnectMetadataStore`), which provides isolation but may have performance implications under high concurrency. Each gRPC method follows the same pattern: connect to the database, call the corresponding `MetadataStore` method, convert the status, and return. There is no connection pooling in the service implementation itself — connection management is delegated to the underlying database drivers.

The component's build system is entirely Bazel-based, which is unusual in the RHOAI ecosystem where most components use Go modules, pip, or npm. The WORKSPACE file pins all external dependencies with SHA256 checksums, providing strong reproducibility at the source level. However, the Konflux build is explicitly non-hermetic (`hermetic: false`), meaning it fetches dependencies from the internet during container image builds. The build requires a UBI9 builder image with `bazel5` from a COPR repository, `gcc`, `clang`, `cmake`, and `openssl-devel`. The final runtime image is UBI9 `ubi-minimal`, containing only the compiled binary and third-party license files (MariaDB Connector/C).

The ZetaSQL dependency is notable — it provides SQL-like filter query parsing for the `ListOperationOptions.filter_query` field, enabling clients to filter artifacts, executions, and contexts using SQL-like boolean expressions. The WORKSPACE file has a version-specific commit pinned for UBI9/Fedora compatibility (`f764f4e986ac`), with a commented-out older commit for UBI8/Fedora39, indicating active maintenance of platform compatibility.

The Python client library (`ml_metadata`) offers dual connectivity modes: direct database access via C++ pybind11 bindings (no network hop) and remote gRPC client access. The gRPC client mode supports SSL/TLS with custom CA, client key, and server certificate. The direct mode is used for testing and local development, while the gRPC mode is the production pattern in RHOAI deployments. The Python client includes retry logic with exponential backoff for Aborted transactions (default 5 retries), providing resilience against transient database conflicts.

Security is minimal at the application layer — the server supports optional TLS and mTLS via protobuf configuration, but has no built-in authentication, authorization, or rate limiting. In RHOAI deployments, access control relies on Kubernetes network policies and the DSP operator's deployment configuration rather than MLMD's own security features. The server runs as non-root (UID 65534) in the container.
