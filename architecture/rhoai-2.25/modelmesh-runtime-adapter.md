## Purpose

**Short**: A multi-purpose sidecar image providing model storage pulling and runtime-specific adapters that translate the ModelMesh ModelRuntime gRPC API into native model server protocols.

**Detailed**: The modelmesh-runtime-adapter repository produces a single container image containing multiple binaries that run as sidecar containers alongside model serving runtimes in ModelMesh Serving Pods. The image serves two primary roles:

1. **Model Serving Puller** (`/opt/app/puller`): A gRPC proxy server that intercepts ModelRuntime API calls from ModelMesh. On `loadModel` requests, it first downloads model artifacts from remote storage (S3, GCS, Azure Blob, HTTP, PVC) to local disk before forwarding the request to the actual runtime adapter. On `unloadModel`, it cleans up local model files after the runtime unloads the model. The puller supports multiple storage backends through the PullMan library.

2. **Runtime Adapters** (`/opt/app/triton-adapter`, `/opt/app/ovms-adapter`, `/opt/app/mlserver-adapter`, `/opt/app/torchserve-adapter`): Each adapter implements the ModelMesh ModelRuntime gRPC API and translates calls into the native protocol of a specific model serving runtime (NVIDIA Triton, OpenVINO Model Server, Seldon MLServer, TorchServe). The adapters handle model layout adaptation (restructuring downloaded model files into the format expected by each runtime), model loading/unloading via the runtime's native API, memory capacity reporting, and model size estimation.

The container is not an operator and does not define CRDs. It is deployed by the `modelmesh-serving` controller as part of ServingRuntime Pod specifications. All communication is via localhost gRPC between containers in the same Pod.
