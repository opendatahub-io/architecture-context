## Purpose

**Short**: Sidecar container that adapts ModelMesh model lifecycle operations to multiple third-party inference runtime backends and pulls models from cloud storage.

**Detailed**: The ModelMesh Runtime Adapter is a multi-purpose container image that runs as a sidecar in ModelMesh Serving model server pods. It contains five binaries: four runtime-specific adapters (Triton, MLServer, OpenVINO Model Server, TorchServe) and a model-serving puller. Each adapter implements the `mmesh.ModelRuntime` gRPC interface, translating ModelMesh's model load/unload/status operations into backend-specific gRPC or REST calls against the colocated inference server container.

The puller component downloads model artifacts from cloud storage backends (S3/IBM COS, Google Cloud Storage, Azure Blob Storage, HTTP, and Kubernetes PVCs) and stages them on the local filesystem. It can run as a standalone gRPC service or be embedded within each adapter. The adapters then transform the downloaded model files into the directory layout expected by each inference runtime — creating symlinks, generating configuration files (config.pbtxt, model-settings.json, model_config_list.json), and optionally converting model formats (e.g., Keras .h5 to TensorFlow SavedModel).

This component is upstream from kserve/modelmesh-runtime-adapter (release-0.11.0) and is shipped as a single container image (`odh-modelmesh-runtime-adapter`) in RHOAI. The modelmesh-controller configures which binary to invoke as the entrypoint for each model serving pod based on the selected runtime.

## Architectural Analysis

The ModelMesh Runtime Adapter follows a clean adapter pattern — all four runtime adapters implement the identical `mmesh.ModelRuntime` gRPC interface, differing only in how they translate load/unload operations to backend-specific protocols. This uniformity means the ModelMesh orchestrator treats all backends identically, with the adapter serving as the sole abstraction boundary. The adapters bind exclusively to localhost, operating within the trust boundary of the pod, which eliminates the need for inter-container authentication or TLS.

The model layout adaptation layer is the most complex part of each adapter. Each inference runtime expects models organized in a different directory structure with different configuration formats: Triton uses versioned directories with `config.pbtxt` (protobuf text format), MLServer expects `model-settings.json`, OVMS maintains a centralized `model_config_list.json`, and TorchServe works with MAR archive files. The adapters use symlinks extensively to avoid duplicating model data on disk, which is critical since model files can be multi-gigabyte. The Keras-to-TensorFlow conversion in the Triton adapter is noteworthy — it shells out to a Python script (`tf_pb.py`) to perform the conversion, which is why the container image includes a full Python 3.11 + TensorFlow installation despite being primarily a Go service.

The OVMS adapter is architecturally distinct from the others in two ways. First, it communicates with its runtime via HTTP REST API rather than gRPC, reflecting OVMS's configuration reload mechanism. Second, it implements an actor-based batch processing pattern (`modelmanager.go`) that accumulates load/unload requests over a configurable time window (100ms to 3s) before issuing a single configuration reload. This design reduces the overhead of frequent config reloads when multiple models are loaded in rapid succession. The batch processing uses a gather-and-flush pattern with short and long timers to balance latency against efficiency.

The Pullman storage framework demonstrates a well-structured provider plugin pattern. Each storage backend (S3, GCS, Azure, HTTP, PVC) registers itself via `init()` functions and implements a common `StorageProvider` interface. The framework includes a TTL-based client cache (1-hour TTL, 4-hour cleanup) that avoids recreating storage clients for the same credentials. The PVC provider is notably different from the cloud providers — it creates symlinks rather than downloading files, since PVCs are already locally mounted at `/pvc_mounts/{pvcName}/`. The HTTP provider supports mTLS via configurable client certificates, making it suitable for authenticated model registries.

A design tension exists in the embedded puller pattern. When `USE_EMBEDDED_PULLER=true`, each adapter instantiates its own Pullman client and downloads models directly, bypassing the standalone puller binary. This eliminates an extra container but means storage credential configuration must be passed to each adapter. The standalone puller mode is used when a separate puller init-container or sidecar handles downloads, leaving the adapter to focus solely on model layout adaptation and runtime management.
