## Purpose

**Short**: A gRPC-Gateway reverse proxy that translates KServe V2 REST inference requests into gRPC calls for ModelMesh Serving.

**Detailed**: The rest-proxy component implements a reverse proxy server using the [grpc-gateway](https://github.com/grpc-ecosystem/grpc-gateway) library. It accepts HTTP/REST requests conforming to the [KServe V2 REST Predict Protocol](https://github.com/kserve/kserve/blob/master/docs/predict-api/v2/required_api.md#httprest) and translates them into gRPC calls following the [KServe V2 gRPC Predict Protocol](https://github.com/kserve/kserve/blob/master/docs/predict-api/v2/required_api.md#grpc). This enables clients that prefer REST/JSON to interact with inference servers (ModelMesh) that natively speak gRPC.

The proxy is deployed as a sidecar container within ModelMesh Serving pods, configured by the `restProxy.image` setting in the `model-serving-config` ConfigMap. It handles custom JSON marshaling/unmarshaling for tensor data types (BOOL, INT8-64, UINT8-64, FP16-64, BYTES), including support for multidimensional nested arrays, base64-encoded binary data, and raw tensor byte contents. The proxy runs on a configurable listen port (default 8008) and forwards to a localhost gRPC endpoint (default port 8033).

## Architectural Analysis

The rest-proxy is a focused, single-purpose component that solves the protocol impedance mismatch between REST/JSON clients and the gRPC-native ModelMesh inference serving infrastructure. Its architecture is minimal by design — a stateless reverse proxy with no persistent storage, no Kubernetes API interaction, and no CRDs. The entire application logic fits in ~550 lines of hand-written Go code plus ~590 lines of generated gRPC gateway stubs.

The most architecturally significant code is in the custom JSON marshaler (`proxy/marshaler.go`) and request transformer (`proxy/request.go`). The gRPC-Gateway library handles the HTTP-to-gRPC translation at the protocol level, but the KServe V2 predict protocol requires non-trivial tensor data transformations. The proxy must handle: multidimensional nested JSON arrays for tensor data (flattening arbitrarily nested arrays by stripping intermediate brackets), base64-encoded binary data for BYTES tensors, raw byte tensor contents split by 4-byte size delimiters, and bidirectional conversion between REST JSON parameter maps and gRPC InferParameter messages. FP16 tensor support is explicitly unimplemented and returns an error.

The proxy supports optional TLS on both the client-facing REST endpoint (via `MM_TLS_KEY_CERT_PATH` / `MM_TLS_PRIVATE_KEY_PATH` environment variables) and the backend gRPC connection (via `REST_PROXY_USE_TLS`). The `InsecureSkipVerify` option for the gRPC backend connection is configurable — this is intentional for in-pod localhost communication where certificate CN validation may not match. There is no authentication or authorization at the proxy level; this is delegated to the ModelMesh Serving infrastructure that encloses the sidecar.

The FIPS story is notable: the Konflux Dockerfile (`Dockerfile.konflux`) uses `CGO_ENABLED=1`, `GOEXPERIMENT=strictfipsruntime`, and `-tags strictfipsruntime` for FIPS-compliant builds, while the upstream community Dockerfile uses `CGO_ENABLED=0` (statically linked, non-FIPS). The upstream Dockerfile also uses `ubi9/ubi-micro:9.5` as the runtime base (smaller image), while the Konflux version uses `ubi9/ubi-minimal:latest` (includes OpenSSL for FIPS). This bifurcation is the standard RHOAI pattern for downstream FIPS compliance. The controller-runtime dependency is used solely for its `zap` structured logger — no Kubernetes controller machinery is invoked.
