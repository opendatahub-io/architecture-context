RBAC Summary:
- This component does NOT define ClusterRoles or RoleBindings
- RBAC is managed by ODH Notebook Controller which launches workbench pods
- Workbench pods run with ServiceAccount created by ODH Notebook Controller
- ServiceAccount has permissions to:
  * Read secrets (for S3 credentials, pull secrets)
  * Read configmaps (for notebook configuration)
  * Create/update pods (for the workbench itself)

Authentication & Authorization:
- OAuth Proxy Sidecar:
  * Enforces OpenShift OAuth authentication
  * Only authenticated OpenShift users can access their own notebooks
  * Validates Bearer tokens from OpenShift OAuth server
  * Policy: User must be the owner of the notebook namespace or have explicit RBAC permissions

- Container Registry:
  * Image pulls require pull secrets
  * Pull secrets are automatically mounted by Kubelet
  * Auth mechanism: kubernetes.io/dockerconfigjson secret type

Secrets:
- Pull Secrets (kubernetes.io/dockerconfigjson):
  * Purpose: Authenticate to Quay.io and registry.redhat.io
  * Provisioned By: OpenShift Installation / Cluster Admin
  * Rotation: Manual (no auto-rotation)
  * Scope: Cluster-wide or namespace-scoped

- User Credentials (Opaque):
  * Purpose: S3 access keys, cloud credentials, API tokens
  * Provisioned By: End users (mounted via ODH Dashboard)
  * Rotation: User-managed (no auto-rotation)
  * Scope: Namespace-scoped

Network Policies:
- Workbench pods do NOT have NetworkPolicies by default
- Users can apply custom NetworkPolicies to restrict egress/ingress
- Default behavior: All egress allowed, ingress allowed from OAuth proxy only

Service Mesh:
- NOT USED by this component
- Workbenches do not participate in Istio/OpenShift Service Mesh
- TLS termination occurs at OpenShift Router (Route-level)
- Internal pod communication uses localhost HTTP (OAuth proxy → workbench container)
