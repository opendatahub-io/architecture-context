═══════════════════════════════════════════════════════════════════════════
AUTHENTICATION & AUTHORIZATION
═══════════════════════════════════════════════════════════════════════════

External Access (via OpenShift Route):
  Endpoint: /apis/v2beta1/* (DSP API)
  Auth Flow:
    1. User → Route (443/TCP) - OAuth Token in Authorization header
    2. Route → OAuth Proxy (8443/TCP) - TLS re-encryption
    3. OAuth Proxy validates token with OpenShift OAuth server
    4. OAuth Proxy → API Server (8888/TCP) - Bearer token (K8s SA)
    5. API Server enforces Kubernetes RBAC

Internal Access (within cluster):
  Endpoint: /apis/v2beta1/* (direct to Service)
  Auth: Kubernetes Bearer Token (Service Account)
  Enforcement: API Server validates SA token, enforces RBAC

MLMD Access:
  External: OAuth Proxy → Envoy → MLMD gRPC (with optional mTLS)
  Internal: Direct gRPC with mTLS (if podToPodTLS=true)

Webhook Access:
  Caller: Kubernetes API Server
  Auth: Mutual TLS (API server certificates)
  Enforcement: Kubernetes admission control

Database Access:
  Auth: MySQL username/password (stored in secrets)
  Connections: API Server, MLMD gRPC, Persistence Agent

Storage Access:
  S3/Minio: S3 access/secret keys (stored in secrets)
  Clients: API Server, Pipeline Step Pods

═══════════════════════════════════════════════════════════════════════════
TRUST BOUNDARIES
═══════════════════════════════════════════════════════════════════════════

1. EXTERNAL → INGRESS
   - Trust Boundary: OpenShift Route
   - Validation: OAuth token validation
   - Encryption: TLS 1.3 (external) → TLS 1.2+ (re-encrypt)

2. INGRESS → APPLICATION
   - Trust Boundary: OAuth Proxy
   - Validation: OpenShift OAuth + K8s RBAC
   - Encryption: TLS 1.2+ → HTTP/HTTPS (podToPod TLS optional)

3. APPLICATION → EXTERNAL SERVICES
   - Trust Boundary: Network egress
   - Validation: S3 credentials, DB credentials, Registry auth
   - Encryption: TLS 1.2+

4. OPERATOR → APPLICATION
   - Trust Boundary: Namespace isolation
   - Validation: K8s RBAC (ClusterRole permissions)
   - Encryption: TLS 1.2+ (K8s API)

═══════════════════════════════════════════════════════════════════════════
SECURITY CONSIDERATIONS
═══════════════════════════════════════════════════════════════════════════

1. Pod-to-Pod TLS: Enabled by default in DSP v2 on OpenShift. Provides mTLS
   encryption between DSP components (API Server ↔ MLMD gRPC, Pipeline Pods ↔
   MLMD gRPC). Configurable via DSPA spec (podToPodTLS field).

2. Minio (Dev/Test Only): The built-in Minio deployment is NOT supported for
   production use. Use external S3-compatible storage with proper access
   controls and encryption for production deployments.

3. Artifact Signed URLs: API Server generates time-limited signed URLs (60s
   default TTL) for artifact downloads. Prevents unauthorized access but
   requires timely consumption of URLs.

4. Database Encryption: MariaDB connections support optional TLS encryption.
   External database should enforce TLS for production.

5. Service Mesh Integration: Optional integration with OpenShift Service Mesh
   (Istio) for additional mTLS and traffic management. Configure via
   NetworkPolicies and ServiceMesh configuration.

6. Webhook Security: Webhook server uses mutual TLS authentication with K8s
   API server. Certificate auto-rotation via OpenShift service annotations.

7. RBAC Least Privilege: Each component has dedicated ServiceAccount with
   minimal required permissions. Pipeline step pods use pipeline-runner SA
   with read-only access to secrets/workflows.

8. Network Isolation: NetworkPolicies restrict access to MariaDB and MLMD
   gRPC to authorized components only. Prevents lateral movement.
