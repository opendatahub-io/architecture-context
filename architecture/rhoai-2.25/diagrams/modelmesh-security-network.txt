Kubernetes Service:
- Service: model-mesh (headless ClusterIP)
  - Port: 8033/TCP → 8033/TCP (grpc)
  - Exposure: Internal only
  - No Ingress resources defined (external access via modelmesh-serving)

NetworkPolicy:
- Defined in config/base/networkpolicy.yaml
- Restricts ingress/egress at pod level

RBAC Summary:
- None — ModelMesh does not define RBAC resources
- All RBAC is managed by the modelmesh-serving controller

Secrets:
- model-mesh-tls-secret (kubernetes.io/tls)
  - Purpose: TLS cert/key for external gRPC and internal Litelinks communication
  - Provisioned By: Manual / cert-manager
  - Auto-Rotate: No

- model-mesh-icd-etcd (Opaque)
  - Purpose: etcd connection configuration (JSON)
  - Provisioned By: Manual
  - Auto-Rotate: No

Authentication & Authorization:
- gRPC API (8033): mTLS client certificates (optional)
  - Enforcement: Netty SSL handler (ModelMeshApi)
  - Policy: Configurable via MM_TLS_CLIENT_AUTH (NONE/REQUIRE/OPTIONAL)

- Litelinks inter-pod (8080): mTLS (optional)
  - Enforcement: Litelinks SSL config
  - Policy: Configurable via MM_INTRACLUSTER_CLIENTAUTH (true/false)

- Health probes (8089): Open (no auth)
- Prometheus metrics (2112): Open (HTTPS with self-signed cert)

FIPS Compliance:
- ⚠️  Java FIPS explicitly DISABLED (-Dcom.redhat.fips=false)
- SunPKCS11 security provider reconfigured
- Netty tcnative-boringssl-static (native TLS, NOT FIPS-validated)
- Known gap for RHOAI deployments on FIPS-enabled clusters

TLS Implementation:
- External gRPC: Netty GrpcSslContexts with tcnative-boringssl-static
- Inter-pod Litelinks: Litelinks SSL framework
- Prometheus HTTPS: Self-signed cert generated via BouncyCastle 1.78
- Custom JKS truststore built at startup from Kubernetes CA cert + provided CA certs
