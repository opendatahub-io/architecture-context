## How to Use

### Mermaid Diagrams (.mmd files)
- **In GitHub/GitLab**: Paste into markdown with ` ```mermaid ` code blocks
- **Render to PNG locally**:
   ```bash
   PUPPETEER_EXECUTABLE_PATH=/usr/bin/google-chrome mmdc -i diagram.mmd -o diagram.png -s 3
   ```
- **Online**: View at https://mermaid.live by pasting `.mmd` file contents

### ASCII Diagrams (.txt files)
- View in any text editor or terminal
- Include directly in security documentation
- Perfect for SAR (Security Architecture Review) submissions

## Diagram Descriptions

### Component Dependency Graph
Shows how all 43 platform components depend on each other. Components are grouped into: Core Operator (rhods-operator), Model Serving Stack (KServe, ModelMesh, OVMS), Disaggregated LLM Stack (llm-d, new in 2.25), User Interface (odh-dashboard), ML Lifecycle (pipelines, registry, TrustyAI, Feast), Distributed Computing (CodeFlare, KubeRay, Kueue, Training Operator), and External Dependencies (Knative, Istio, Gateway API, OLM). Central components (rhods-operator, KServe, odh-dashboard, Kueue, odh-model-controller) are highlighted with thicker borders.

### Platform Network Topology
Complete network architecture showing the 5-tier namespace topology: Operator (redhat-ods-operator), Platform (redhat-ods-applications), Monitoring (redhat-ods-monitoring), User (per-project), and optional Mesh (istio-system). Includes all ingress points with exact port numbers, protocols, TLS configuration, and authentication mechanisms. Shows 6 ingress paths (Dashboard, KServe, LLMInferenceService Gateway, Notebooks, Ray Dashboard, Model Registry), internal controller-to-workload communication, and 5 egress destinations.

**Two formats provided**:
- **Mermaid (.mmd)**: Visual diagram with color-coded trust zones. Great for presentations and architecture discussions.
- **ASCII (.txt)**: Precise text format with no ambiguity. Required for SAR (Security Architecture Review) submissions and compliance documentation.

Both formats contain the same information, just different representations.

### Cross-Component Workflows
Five sequence diagrams showing end-to-end user workflows:
1. **Model Serving (KServe Serverless)**: Dashboard → webhook mutation → KServe → Knative → storage-initializer → odh-model-controller → TrustyAI → production inference
2. **LLM Inference (llm-d P/D)** [New in 2.25]: LLMInferenceService → Gateway API → Envoy ext-proc → EPP plugin pipeline → prefill/decode split → NIXL KV-cache transfer
3. **ML Pipeline Execution**: Dashboard → DSP API → Argo Workflows → driver/launcher → MLMD → persistence agent
4. **Distributed Training with Quota**: CodeFlare SDK → Kueue admission → KubeRay/Training Operator → Ray mTLS cluster
5. **Model Registry to Serving**: Registry UI → model registration → CSI storage initializer → `model-registry://` URI resolution → KServe deployment

### Platform Security Overview
Visual representation of the full security architecture including:
- **8 authentication mechanisms**: OAuth, Bearer passthrough, Authorino, kube-rbac-proxy, SAR, Istio mTLS, Ray mTLS, API key validation
- **RBAC**: ServiceAccount → ClusterRole bindings for all major operators
- **Broad access grants**: system:authenticated → LMEvalJob (security concern highlighted)
- **Container security**: Non-root, UBI 9, FIPS builds, capability drops, digest pinning, hermetic builds
- **FIPS compliance gaps**: localmodel CGO_ENABLED=0, Python services, DSP root containers
- **Elevated privileges**: LLM SCC (IPC_LOCK, SYS_RAWIO), multi-node SCC, DSP root containers
- **Service mesh policies**: PeerAuthentication, AuthorizationPolicy, VirtualService, NetworkPolicy
- **Webhook security**: 30+ webhook configurations across 6 operators

### Platform Observability
Maps all monitoring and observability endpoints:
- **Prometheus scrape targets**: 8 operator/controller endpoints (2 protected via kube-rbac-proxy, 6 standard), 4 workload metric sources
- **Health probes**: 12 operator liveness/readiness probes (all on :8081), 2 service probes, workload-specific probes
- **Distributed tracing**: OTEL Collector → Tempo with KServe, Pipeline, and llm-d instrumentation
- **Alerting**: PrometheusRules for Kueue, KServe, and platform health

### Platform Maturity Dashboard
High-level metrics organized into 8 categories:
- **Components**: 43 total, 14 operator-based, 5 new in 2.25, 64+ container images
- **Serving modes**: 4 modalities (KServe Serverless, KServe Raw, LLMInferenceService, ModelMesh)
- **Security posture**: FIPS enabled (Go), 3 FIPS gaps identified, digest-pinned images
- **API maturity**: v1 (6 CRDs stable), v1beta1 (5 maturing), v1alpha1 (12+ experimental)
- **Deployment readiness**: HA via leader election, disconnected support, 4 OCP versions
- **Dependencies**: 4 external platform, 4 external services, 16 manifest repos
- **Observability**: MonitoringStack, Tempo, OTEL Collector
- **Architectural concerns**: 3 serving stacks, 6+ auth patterns, llm-d complexity, broad RBAC grants
