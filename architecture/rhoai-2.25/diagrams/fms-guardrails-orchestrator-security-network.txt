RBAC Summary:
- No Kubernetes RBAC required (stateless service, no K8s API interaction)

Secrets:
- TLS server cert/key (kubernetes.io/tls) - Server TLS for guardrails endpoint
  - Provisioned via: --tls-cert-path, --tls-key-path (manual)
  - Auto-Rotate: No
- TLS client CA cert (CA bundle) - mTLS client verification
  - Provisioned via: --tls-client-ca-cert-path (manual)
  - Auto-Rotate: No
- Per-service TLS certs (kubernetes.io/tls) - Client TLS to downstream services
  - Provisioned via: config.yaml tls section (manual)
  - Auto-Rotate: No

Authentication & Authorization:
- /api/v1/* and /api/v2/*:
  - Auth: Header passthrough (X-Forwarded-Access-Token → Bearer token rewrite)
  - Enforcement: Delegated to downstream services
  - Config: passthrough_headers + rewrite_forwarded_access_header in config.yaml
- /health, /info:
  - Auth: None (unauthenticated)

FIPS Compliance Warning:
- TLS Implementation: rustls with ring crypto backend
- ring is NOT FIPS-certified
- OpenSSL is installed in container (UBI base) but NOT used for TLS
- Significant FIPS gap if FIPS-validated cryptography is required end-to-end

Security Considerations:
- NoVerifier struct in src/utils/tls.rs disables certificate verification
  when insecure=true is set per-service in config.yaml
  - Risk: Man-in-the-middle attacks if used in production
  - Mitigation: Only enable for development environments
- Auth is passthrough-only — orchestrator does NOT enforce authentication
  - Risk: If downstream services don't validate tokens, requests are unauthenticated
  - Mitigation: Ensure all downstream services enforce their own auth
