RBAC Summary
================================================================================

Cluster-Scoped ClusterRoles:
- rhods-operator:           Full lifecycle: CRDs, Deployments, Services, RBAC, Routes,
                            OAuth, monitoring, mesh, serving, scheduling
- odh-model-controller:     InferenceServices, Routes, Gateways, AuthConfigs,
                            NetworkPolicies, ServiceMonitors, NIM accounts
- odh-dashboard:            Nodes, StorageClasses, Notebooks, InferenceServices,
                            ModelRegistries, ConfigMaps/Secrets across namespaces
- kserve:                   InferenceServices, ServingRuntimes, Knative Services,
                            Gateway API resources
- training-operator:        Training CRDs (PyTorchJob, TFJob, etc.), Pods, Services,
                            NetworkPolicies, PodGroups
- trustyai-service-operator: TrustyAI/LMEval/Orchestrator CRDs, InferenceServices,
                            Routes, Kueue workloads
- kueue:                    ClusterQueues, ResourceFlavors, Workloads, scheduling

Namespace-Scoped Roles:
- DSP operator:             Per-namespace pipeline stack management
- model-registry-operator:  Per-namespace ModelRegistry deployment

Aggregated ClusterRoles:
- training-edit/view:       Grants training job access to users with edit/admin/view roles

================================================================================
Service Mesh Configuration
================================================================================

- ServiceMeshControlPlane:   Managed by rhods-operator (istio-system or configurable)
- ServiceMeshMember:         Per InferenceService namespace enrollment
- PeerAuthentication:        mTLS STRICT per InferenceService namespace
- Gateway (kserve-local-gw): Per-ISVC TLS termination on port 8445
- Telemetry:                 Per-namespace metrics collection
- AuthConfig (Authorino):    Per-InferenceService authorization
- DestinationRule:           mTLS for TrustyAI (conditional)
- VirtualService:            TrustyAI traffic routing (conditional)

================================================================================
Network Policies
================================================================================

- redhat-ods-applications:   Allow traffic on ports 8443, 8080, 8081, 5432, 8082, 8099, 8181, 9443
- redhat-ods-monitoring:     Allow from openshift-monitoring, operator NS
- redhat-ods-operator:       Allow from monitoring, console, operators namespaces
- model-mesh (per NS):       Intra-deployment mesh + client inference + Prometheus
- rhods-dashboard:           Allow on 8443, 8043
- allow-metrics-traffic:     Prometheus scraping of odh-model-controller
- odh-model-controller:      Webhook from API server
- PyTorchJob (dynamic):      Intra-job + metrics from openshift-monitoring

================================================================================
Container Security Patterns
================================================================================

- Non-root execution:        All components (UIDs: 1001, 2000, 5000, 65532)
- UBI9 base images:          All components (ubi9, ubi9-minimal, ubi9-micro, openjdk-21)
- FIPS compliance:           Go (GOEXPERIMENT=strictfipsruntime), Java (BouncyCastle)
- Image digest pinning:      All 76 images by SHA256 in RELATED_IMAGE env vars
- Read-only filesystem:      Recommended for OVMS (with tmpfs /tmp)
- Drop all capabilities:     TGIS, training-operator, others
- SeccompProfile:            RuntimeDefault across operator deployments

================================================================================
SSRF Protection
================================================================================

- modelmesh (RemotePayloadProcessor): Validates URIs, rejects private/internal addresses
- llm-d-routing-sidecar:              SSRF protection via InferencePool target filtering
