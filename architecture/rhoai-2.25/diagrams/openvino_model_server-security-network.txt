RBAC Summary:
- OVMS is a containerized inference service, NOT a Kubernetes operator
- OVMS does NOT define any ClusterRoles, Roles, or RoleBindings
- RBAC is managed entirely by the deploying operator (KServe/ModelMesh)
- ServiceAccount is created by the deploying operator

Authentication & Authorization:
- OVMS implements NO authentication or authorization
- gRPC server uses grpc::InsecureServerCredentials() explicitly
- REST server has no TLS configuration
- Auth enforcement delegated to platform:
  - RHOAI 3.x: kube-rbac-proxy sidecar (SubjectAccessReview)
  - RHOAI 2.x: oauth-proxy sidecar (OAuth/OIDC)
  - Service mesh (Istio) provides mTLS between services
- WARNING: Direct exposure of OVMS ports without auth proxy = unauthenticated access

Secrets:
- Storage credentials (Opaque) - S3 (AWS_ACCESS_KEY_ID/AWS_SECRET_ACCESS_KEY), Azure (connection string), GCS (credentials JSON)
  - Provisioned by: User/Operator
  - Auto-Rotate: No
  - Mounted as: Environment variables or files
- HuggingFace Hub token (Opaque) - Optional model download authentication
  - Provisioned by: User/Operator
  - Auto-Rotate: No
- OVMS does NOT create or manage Kubernetes Secret resources directly

FIPS Compliance Notes:
- C++ binary (not Go) - Go FIPS flags do not apply
- Uses OpenSSL from UBI 9 base image
- MD5 used for file checksumming (NOT FIPS-approved, but not security-critical)
- No server-side TLS - FIPS TLS compliance delegated to infrastructure
- FIPS compliance depends on: UBI 9 OpenSSL + service mesh TLS + auth proxy TLS

Build Hermeticity:
- Tekton pipeline: hermetic=false (explicit)
- Build downloads deps from internet at build time (Boost, OpenVINO, Azure SDK, etc.)
- Dependencies pinned via git commit hashes (reproducibility, not hermeticity)
- Supply chain risk: significant
