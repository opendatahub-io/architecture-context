## Purpose
**Short**: A reverse proxy sidecar that orchestrates disaggregated prefill/decode (P/D) inference by routing requests between prefill and decode vLLM workers.

**Detailed**: The llm-d-routing-sidecar is a Go-based HTTP/HTTPS reverse proxy deployed as a sidecar container alongside vLLM inference servers in a disaggregated prefill/decode (P/D) architecture. It intercepts OpenAI-compatible chat completion requests and coordinates the two-phase inference process: first forwarding the prompt to a remote prefiller pod for KV cache computation, then forwarding the augmented request to the local decoder pod for token generation.

The sidecar supports three connector protocols for KV cache transfer coordination: NIXL v2 (current default), NIXL v1 (deprecated), and LMCache (deprecated). It determines the prefiller target from the `x-prefiller-host-port` HTTP request header, which is typically set by the upstream llm-d inference scheduler. When no prefiller header is present, requests pass through directly to the local decoder.

The sidecar includes SSRF (Server-Side Request Forgery) protection that can be enabled via feature flag. When active, it watches Kubernetes `InferencePool` custom resources (from the Gateway API Inference Extension) and maintains a dynamic allowlist of valid prefill targets based on pod IPs matching the InferencePool selectors, rejecting requests to unauthorized targets with HTTP 403.

### SSRF Protection

When `--enable-ssrf-protection` is enabled:
1. The sidecar watches a specific `InferencePool` CR via Kubernetes dynamic client
2. It extracts the pod selector from the InferencePool spec
3. It creates a pod informer matching that selector to track pod IPs
4. On each chat completion request with `x-prefiller-host-port`, the target host is validated against the allowlist
5. Unauthorized targets receive HTTP 403 Forbidden
6. The allowlist auto-updates as pods are added/removed (30-second resync period)
