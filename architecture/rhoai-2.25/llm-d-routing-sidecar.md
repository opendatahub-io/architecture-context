## Purpose

**Short**: A reverse proxy sidecar that orchestrates disaggregated prefill/decode (P/D) inference by routing requests between prefiller pods and the local decoder (vLLM) instance.

**Detailed**: The llm-d-routing-sidecar is a Go-based HTTP reverse proxy designed to run as a sidecar container alongside a vLLM decoder instance. It intercepts incoming OpenAI-compatible chat completion and legacy completion requests, and when a prefiller host is specified via the `x-prefiller-host-port` HTTP header, it orchestrates a two-phase disaggregated inference flow: first sending the prompt to a remote prefiller pod for KV cache computation, then forwarding the enriched request (with KV transfer parameters) to the local decoder for token generation.

The sidecar supports three connector protocols for prefill/decode coordination: NIXL v2 (default, current), NIXL v1 (deprecated), and LMCache (deprecated). Each protocol implements a different strategy for passing KV cache transfer metadata between the prefiller and decoder stages. Requests without a prefiller header are transparently proxied to the local decoder, making the sidecar invisible to non-disaggregated workloads.

The component includes an optional SSRF (Server-Side Request Forgery) protection feature that watches Kubernetes `InferencePool` resources (from the Gateway API Inference Extension) to dynamically maintain an allowlist of valid prefiller targets. When enabled, only pod IPs/hostnames matching the InferencePool's selector are permitted as prefill destinations, preventing request forgery attacks in multi-tenant environments.

## Architectural Analysis

The llm-d-routing-sidecar implements a clean sidecar proxy pattern for disaggregated LLM inference. Its core design decouples the prefill/decode orchestration from the vLLM engine itself, allowing the inference framework to remain unaware of the distributed scheduling layer. The sidecar intercepts only two specific API paths (`/v1/chat/completions` and `/v1/completions`) while transparently proxying all other traffic to the local decoder, which means existing vLLM APIs (models, tokenizers, health checks) remain accessible without modification.

The three connector protocols (NIXL v2, NIXL v1, LMCache) reflect the rapid evolution of the P/D disaggregation architecture. NIXL v2 is the current standard, encapsulating all KV transfer metadata within a single `kv_transfer_params` JSON object. NIXL v1 used flat top-level fields (`remote_block_ids`, `remote_engine_id`, `remote_host`, `remote_port`), which was less extensible. LMCache took a simpler approach — sending the full prompt to the prefiller with `max_tokens=1` to warm the cache, then forwarding the original request to the decoder unchanged. Both NIXL v1 and LMCache are marked deprecated. The code path selection happens at server initialization time (not per-request), so all requests within a sidecar instance use the same protocol.

The SSRF protection mechanism is architecturally notable. Rather than using a static configuration file, it dynamically watches the same `InferencePool` Custom Resource (from the Gateway API Inference Extension, group `inference.networking.x-k8s.io`) that the routing scheduler uses to define inference pools. This means the allowlist automatically stays in sync with the actual pod membership. The implementation uses Kubernetes dynamic client with shared informers for both InferencePool and Pod resources, rebuilding the full allowlist on every pod add/update/delete event. The allowlist tracks only host IPs and pod names (not ports), which is a deliberate simplification — any port on an allowed host is permitted. This trades port-level granularity for operational simplicity, as vLLM pods may expose multiple ports.

The TLS architecture provides defense in depth with three independent TLS configurations: the proxy server's own TLS (enabled by default with self-signed certs as fallback), the decoder connection TLS (opt-in via `--decoder-use-tls`), and the prefiller connection TLS (opt-in via `--prefiller-use-tls`). All three use the same hardcoded cipher suite list targeting AES-GCM and CHACHA20-POLY1305 with ECDHE key exchange, and enforce TLS 1.2 minimum. The `InsecureSkipVerify` flags for prefiller and decoder connections are available for development/testing but disabled by default. The Konflux Dockerfile uses `CGO_ENABLED=1` for FIPS-compatible dynamic linking, though the explicit `GOEXPERIMENT=strictfipsruntime` flag is absent.

The prefiller proxy instances are cached in an LRU cache of size 16 (`hashicorp/golang-lru`), keyed by host:port string. This means the sidecar can efficiently handle requests routed to up to 16 distinct prefiller pods without creating new reverse proxy instances on each request. Beyond 16, the least-recently-used proxy is evicted and recreated on demand. This design is appropriate for typical inference pool sizes. The project is described as experimental and slated for removal in a future iteration of the llm-d P/D architecture, suggesting the orchestration logic may eventually move into the EPP or vLLM engine itself.
