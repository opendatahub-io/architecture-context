## Purpose

**Short**: The rhods-operator is the central platform operator for Red Hat OpenShift AI (RHOAI), responsible for deploying, configuring, and lifecycle-managing all data science components and platform services on OpenShift clusters.

**Detailed**: The rhods-operator orchestrates the entire RHOAI platform through a two-tier architecture. At the infrastructure level, the `DSCInitialization` controller creates namespaces, sets up networking policies, configures monitoring, initializes service mesh integration with Authorino-based authorization, manages CA certificate bundles, generates secrets, and configures RBAC. At the application level, the `DataScienceCluster` controller provisions and manages 14 data science component controllers (Dashboard, Workbenches, KServe, ModelMesh, DataSciencePipelines, Kueue, CodeFlare, Ray, TrustyAI, ModelRegistry, TrainingOperator, FeastOperator, LlamaStackOperator, and ModelController) each of which deploys its respective component via kustomize-based manifests fetched from downstream repositories.

The operator supports three platform modes: Self-Managed RHOAI, Managed RHOAI (addon on OSD/ROSA), and Open Data Hub (ODH). Each mode affects default namespaces, monitoring configuration, and component deployment. The operator includes 10 admission webhooks (5 mutating, 5 validating) that enforce singleton constraints, inject hardware profiles into workloads, manage platform connections for InferenceServices and Notebooks, and validate Kueue queue labels. The operator manages service mesh integration via Maistra/Istio, including ServiceMeshControlPlane creation, Authorino deployment for authorization, and EnvoyFilter configuration for KServe inference services.

## Architectural Analysis

### Two-Tier Controller Architecture

The rhods-operator implements a sophisticated two-tier architecture that cleanly separates infrastructure concerns from application-level component management. The `DSCInitialization` controller handles all platform prerequisites — namespace creation, network policies, monitoring setup, service mesh initialization, CA certificate distribution, and RBAC group management — while the `DataScienceCluster` controller manages the lifecycle of 14 data science components through a registry-based plugin system. Each component registers itself via Go `init()` functions, allowing the operator to dynamically discover and manage components without hardcoded dependencies in the main controller. This design enables components to be added or removed without modifying core controller code.

### Registry-Based Component System

The component registry (`internal/controller/components/registry/`) and service registry (`internal/controller/services/registry/`) implement a plugin pattern where each component/service registers a `ComponentHandler` or `ServiceHandler` that provides Init, GetName, GetManagementState, and NewReconciler methods. The `DataScienceCluster` controller iterates the component registry to provision component CRs, then each component's dedicated controller reconciles its own CR by rendering kustomize manifests from the `prefetched-manifests/` directory. This indirection means the DSC controller never directly deploys component resources — it only creates the component CR, and the component controller handles the actual deployment. The pattern supports three management states: Managed (active deployment), Unmanaged (manual control), and Removed (cleanup).

### Manifest Supply Chain

Component manifests are fetched from 16 downstream repositories via `get_all_manifests.sh` at development time, then baked into the container image at build time via `prefetched-manifests/` (pre-fetched copies) and `opt/manifests/` (runtime path). The `Dockerfile.konflux` copies both `opt/manifests/` and `prefetched-manifests/` into `/opt/manifests/`, which becomes the `DEFAULT_MANIFESTS_PATH` at runtime. Each component controller reads its manifests from this path, applies kustomize overlays (which differ between RHOAI and ODH), and deploys via server-side apply. The `RELATED_IMAGE_*` environment variables injected by OLM allow air-gapped installations by overriding container image references in the rendered manifests.

### Service Mesh and Authorization Architecture

When ServiceMesh is enabled in the DSCI spec, the ServiceMesh controller deploys a full Istio control plane via Maistra's `ServiceMeshControlPlane` CR, configures Authorino as an external authorization provider (patching the SMCP with an external authz extension), and manages `ServiceMeshMember` CRs to bring namespaces into the mesh. The KServe component controller additionally creates `EnvoyFilter` CRs for inference service routing (activator EnvoyFilter, inference graph EnvoyFilter) and `AuthorizationPolicy` CRs for per-service authorization. This creates a layered security model: Istio provides mTLS between services, Authorino provides fine-grained authorization, and individual component controllers configure service-specific policies.

### Monitoring and Observability Stack

The Monitoring service controller is one of the most complex service controllers, deploying an entire observability stack: `MonitoringStack` (via Cluster Observability Operator), `TempoMonolithic` or `TempoStack` (via Tempo Operator for traces), `OpenTelemetryCollector` and `Instrumentation` (via OpenTelemetry Operator for metrics/traces collection), `ThanosQuerier` (for federated metrics queries), and per-component `PrometheusRule` CRs that are dynamically enabled/disabled based on which components are active in the DataScienceCluster. The controller detects single-node clusters and adjusts replica counts accordingly. For managed RHOAI, additional monitoring includes Deadmansnitch heartbeats and PagerDuty alerting integration.

### Webhook Architecture

The 10 webhooks serve distinct purposes: (1) singleton enforcement for DSC and DSCI prevents multiple instances, (2) the DSC defaulter sets ModelRegistry namespace defaults, (3) the DSCI validator prevents deletion while DSCs exist, (4) HardwareProfile injectors mutate Notebooks and InferenceServices to apply GPU/accelerator resource specifications from HardwareProfile CRs, (5) platform connection webhooks inject S3/OCI/URI storage credentials into InferenceServices and environment variables into Notebooks, and (6) Kueue validators enforce queue-name labels on 7 workload types (Notebooks, PyTorchJobs, RayJobs, RayClusters, InferenceServices) in Kueue-managed namespaces. All webhooks use a build tag (`!nowebhook`) allowing webhook-free builds for testing.
