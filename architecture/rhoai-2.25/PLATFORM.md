## 1. Platform Overview

Red Hat OpenShift AI (RHOAI) 2.25 is an enterprise AI/ML platform deployed on OpenShift Container Platform. It provides a comprehensive, integrated environment for data scientists, ML engineers, and platform administrators to develop, train, serve, monitor, and govern AI/ML models at scale.

The platform is managed by a single top-level OLM operator (`rhods-operator`) that orchestrates the lifecycle of 15+ component sub-systems through two primary Custom Resources: `DSCInitialization` (platform initialization) and `DataScienceCluster` (component enablement). Each component is independently manageable (enabled/disabled/removed) and deployed via kustomize overlays with pinned container image digests for reproducibility and disconnected/air-gapped support.

## 7. Data Flows

### 7.1 User → Dashboard → Kubernetes API

```
Browser → HTTPS (443) → OpenShift Router → Reencrypt → oauth-proxy (8443/TLS)
  → x-forwarded-access-token → odh-dashboard backend (8080/HTTP)
    → Kubernetes API (6443/HTTPS, user's Bearer Token)
```

### 7.2 Model Inference (KServe Serverless)

```
Client → HTTPS (443) → OpenShift Router → TLS Passthrough
  → Istio Ingress Gateway (8445/HTTPS, serving cert)
    → Authorino AuthConfig validation
      → KServe predictor Pod (8012/HTTP, mTLS via Istio)
        → Inference Runtime (vLLM/OVMS/TGIS, internal)
```

### 7.3 Model Inference (ModelMesh)

```
Client → HTTPS (443) → OpenShift Route
  → ModelMesh gRPC (8033/TLS optional)
    → LRU routing → local or remote ModelMesh pod (8080/Thrift)
      → modelmesh-runtime-adapter (8085/gRPC)
        → Inference Runtime (OVMS/TGIS)
```

### 7.4 Pipeline Execution

```
User creates Pipeline Run → DSP API Server (8888)
  → Argo Workflow Controller creates Workflow
    → Argo creates executor Pods
      → Pods execute pipeline steps using runtime images
        → Artifacts stored to S3/MinIO
        → Metadata recorded to ML Metadata (8080/gRPC)
        → Status persisted to MariaDB (3306)
```

### 7.5 TrustyAI Model Monitoring

```
ModelMesh/KServe InferenceService sends payloads:
  [ModelMesh] → POST /consumer/kserve/v2 (HTTP, port 80)
  [KServe] → CloudEvents (org.kubeflow.serving.inference.*)

TrustyAI Service:
  → Reconciles payloads → Stores to PVC/DB/MinIO
  → PrometheusScheduler (cron) → Computes fairness/drift metrics
  → Publishes to /q/metrics → Prometheus scrapes
```

### 7.6 Guardrailed LLM Request

```
Client → vllm-orchestrator-gateway (8090/HTTP)
  → Injects detector config per route
    → FMS Guardrails Orchestrator (8033/HTTP)
      → Input detectors (regex/HF/LLM-judge, parallel)
      → Generation service (vLLM InferenceService)
      → Output detectors (parallel)
    → Gateway checks detections
      → If flagged: return fallback_message
      → If clean: return generated response
```

### 7.7 Distributed Training

```
User creates PyTorchJob → Training Operator webhook (9443/HTTPS)
  → Training Operator creates:
    - Master/Worker Pods (with headless Service for DNS discovery)
    - NetworkPolicy (intra-job + monitoring access)
    - Optional: PodGroup (Volcano/scheduler-plugins gang scheduling)
    - Optional: HPA (elastic training)
  → Workers communicate via NCCL/Gloo (distributed training)
  → Prometheus scrapes metrics from worker pods (8080/TCP)
```

### 8.5 Disconnected / Air-Gapped Support

- All 76 container images are declared as `RELATED_IMAGE_*` environment variables in the operator CSV
- OLM uses `relatedImages` for automatic image mirroring via `oc-mirror`
- `ImageDigestMirrorSet` resources defined in RHOAI-Build-Config map `registry.redhat.io` → `quay.io/rhoai`
- All images referenced by SHA256 digest (not tags) for deterministic deployment
- `features.operators.openshift.io/disconnected: "true"` annotation on CSV
