## Platform Overview

Red Hat OpenShift AI (RHOAI) 2.25 is a comprehensive machine learning platform deployed on OpenShift Container Platform via the Operator Lifecycle Manager (OLM). The platform spans the full ML lifecycle — from interactive development environments (Jupyter notebooks, VS Code, RStudio) through distributed training (PyTorch, TensorFlow, MPI, JAX), pipeline orchestration (Kubeflow Pipelines with Argo Workflows), feature management (Feast), model registry, and multi-modal inference serving (KServe serverless, KServe raw, ModelMesh multi-model, and the new llm-d disaggregated LLM serving stack). The platform is managed by a single central operator (`rhods-operator`) that uses a two-tier CRD architecture: a `DSCInitialization` CR for infrastructure prerequisites and a `DataScienceCluster` CR that drives a registry-based component system to deploy 14 individual component operators and services.

The deployment model follows a hub-and-spoke pattern. The `rhods-operator` runs in the `redhat-ods-operator` namespace (installed via OLM), deploys platform components into the `redhat-ods-applications` namespace, and user workloads (notebooks, pipelines, inference services) run in user-created namespaces. Component manifests are pre-fetched from 16 downstream repositories at build time and baked into the operator image, with kustomize overlays providing RHOAI-specific configuration (distinct from the upstream Open Data Hub distribution). Air-gapped installations are supported through `RELATED_IMAGE_*` environment variables injected by OLM, enabling image mirroring without manifest modification.

The security posture is layered: OpenShift OAuth provides platform authentication, optional Istio service mesh provides mTLS between services, Authorino provides fine-grained authorization for inference endpoints, kube-rbac-proxy and oauth-proxy sidecars protect individual component APIs, and NetworkPolicies enforce namespace isolation. FIPS compliance is achieved through Go binaries built with `GOEXPERIMENT=strictfipsruntime` on UBI 9 base images with `CGO_ENABLED=1`, although not all components (notably KServe's local model caching) have completed FIPS enablement. The platform supports multi-architecture builds (amd64, ppc64le, s390x) with varying levels of completeness across components.

### Central Components

**rhods-operator** is the platform root — it deploys and lifecycle-manages all 14 component operators and 6 platform services. Every component has a direct or transitive dependency on it. It owns 20+ CRDs spanning component lifecycle, platform services, infrastructure, and RBAC.

**KServe** is the most-connected serving component. It is depended upon by odh-model-controller (networking/auth), ModelMesh Serving (InferenceService CRD), Model Registry (ISVC label reconciliation), TrustyAI (ISVC patching), odh-dashboard (serving UI), caikit-tgis-serving (ServingRuntime), and OVMS (KServe v2 protocol). KServe also owns the new LLMInferenceService CRD that integrates with the llm-d stack via Gateway API Inference Extensions.

**odh-dashboard** is the primary user-facing entry point, proxying requests to 7+ backend services (Data Science Pipelines, Feast, KServe, Model Registry, TrustyAI, LlamaStack, MLMD) and integrating with Notebooks, CodeFlare SDK, and platform RBAC.

**Kueue** is the cross-cutting resource governance layer, providing quota-aware admission for RayJobs, RayClusters, PyTorchJobs, Notebooks, InferenceServices, and AppWrappers. The rhods-operator enforces Kueue queue-name labels via admission webhooks on 7 workload types.

**odh-model-controller** is the model serving integration hub, managing Routes, NetworkPolicies, ServiceMonitors, AuthConfigs, and RBAC for KServe InferenceServices across three deployment modes (serverless, raw, ModelMesh), plus NVIDIA NIM account management and Ray TLS certificate issuance.

## Deployment Architecture

### Deployment Topology

The platform uses a three-tier namespace topology. The **operator tier** (`redhat-ods-operator`) contains only the OLM-managed `rhods-operator` Deployment. The **platform tier** (`redhat-ods-applications`) hosts all component operators and shared infrastructure — approximately 14 Deployments for component controllers, the odh-dashboard pod (with oauth-proxy and BFF sidecars), and webhook servers. The **user tier** spans multiple user-created namespaces, each containing workloads like Notebooks (StatefulSets), DSPA instances (Deployments for API server, persistence agent, scheduler, Argo controller, MLMD, MariaDB, optional MinIO), InferenceServices (Deployments or Knative Services with model runtime pods), RayClusters (head/worker pods), FeatureStores (Deployments), Model Registry instances (Deployments with REST proxy), and TrustyAI instances (Deployments). An optional **mesh tier** (`istio-system`) hosts the ServiceMeshControlPlane and Authorino when service mesh is enabled. The **monitoring tier** (`redhat-ods-monitoring`) hosts the observability stack (MonitoringStack, Tempo, OpenTelemetry Collector).

#### Manifest Consumption Pipeline

Component manifests flow through a multi-stage pipeline: source repositories maintain upstream manifests with kustomize structure → `get_all_manifests.sh` in rhods-operator fetches specific branches/commits from 16 downstream forks → manifests are stored in `opt/manifests/` and `prefetched-manifests/` directories → the Konflux Dockerfile copies both into `/opt/manifests/` in the container image → at runtime, each component controller reads its manifests from `DEFAULT_MANIFESTS_PATH`, applies the distribution-appropriate kustomize overlay (selected based on `platform` label), and deploys via Kubernetes server-side apply. The `RELATED_IMAGE_*` environment variables from OLM's ClusterServiceVersion enable the overlay system to substitute image references for air-gapped deployments without modifying the stored manifests.

### Disconnected Support

RHOAI supports air-gapped (disconnected) deployments through the OLM `RELATED_IMAGE_*` pattern. The ClusterServiceVersion (CSV) declares all container image references in its `relatedImages` section, enabling `oc-mirror` to discover and mirror the complete image set to a disconnected registry. At runtime, OLM injects these references as environment variables (`RELATED_IMAGE_<name>=<digest>`), which the rhods-operator's kustomize overlay system substitutes into rendered manifests. All production Konflux-built images use SHA256 digest pinning (e.g., `@sha256:b9b10f42d...`) rather than floating tags, ensuring reproducibility in disconnected environments. The `oc-mirror` tool handles the ImageContentSourcePolicy/ImageDigestMirrorSet configuration that redirects the cluster's container runtime to pull from the mirrored registry.

## Platform Architectural Analysis

RHOAI 2.25 exhibits a platform in architectural transition, with several notable cross-cutting patterns and tensions emerging from the analysis of all 43 components.

**Dual Serving Architecture**: The most significant architectural development is the emergence of two parallel model serving stacks. The established KServe + Knative path (InferenceService → Knative Service → Istio routing) coexists with the new KServe + Gateway API path (LLMInferenceService → Deployment/LeaderWorkerSet → Gateway HTTPRoute → ext-proc EPP). These stacks share CRD types but have fundamentally different networking, scaling, and operational models. The odh-model-controller must maintain mode-specific sub-reconciler composites (12 sub-reconcilers for serverless, 6 for raw, 3 for ModelMesh) to bridge this gap, making it the most integration-sensitive component on the platform. Additionally, ModelMesh provides a third serving modality with its own distributed cache and routing system. This three-path model serving architecture increases the operational surface significantly.

**Inconsistent Authentication Patterns**: Authentication enforcement varies across components in ways that may confuse operators and create security blind spots. The dashboard uses oauth-proxy → Node.js backend → K8s API proxy. TrustyAI uses oauth-proxy → direct service access. KServe uses Authorino via Istio AuthorizationPolicy (mesh-dependent). Model Registry uses oauth-proxy with BFF SubjectAccessReview. Ray clusters use self-signed mTLS with oauth-proxy for dashboard access. The llm-d stack relies on Envoy/Gateway API without explicit auth integration. This heterogeneity means there is no single "how is this endpoint authenticated?" answer for the platform — each component must be evaluated individually.

**FIPS Compliance Gaps**: While Go operators consistently use `GOEXPERIMENT=strictfipsruntime`, several gaps exist. KServe's `localmodel.Dockerfile.konflux` uses `CGO_ENABLED=0` with a distroless base, meaning its binary would fail FIPS check-payload validation. Python services (caikit-nlp, caikit-tgis-serving, vLLM, Feast feature server, guardrails detectors) rely on UBI 9's system OpenSSL but don't have explicit FIPS validation for their Python cryptography stacks. The Data Science Pipelines driver, launcher, persistence agent, and scheduled workflow controller all run as root, which complicates FIPS certification. Several Dockerfiles in data-science-pipelines and data-science-pipelines-tekton use non-UBI base images (alpine, ubuntu:16.04, tensorflow, python) for upstream-inherited components that are not shipped in RHOAI but exist in the source tree.

**Root Container Execution**: Multiple pipeline components (DSP driver, launcher, persistence agent, scheduled workflow controller) run as root. While these run within user namespaces with OpenShift's SCC enforcement, the explicit `USER root` directive in production Konflux Dockerfiles is a security concern that should be addressed for defense-in-depth. KServe's LLM SCC grants elevated capabilities (IPC_LOCK, SYS_RAWIO) for GPU memory operations, which is necessary but represents a privilege escalation that should be tightly scoped.

**Kustomize Overlay Complexity**: The manifest supply chain — 16 repos → get_all_manifests.sh → prefetched-manifests/ → kustomize overlays → server-side apply — is powerful but creates a long dependency chain where a bug in any overlay can affect production. Each component maintains its own overlay structure with varying conventions (params.env, replacements, patches), and the rhods-operator must correctly compose these at runtime. Version skew between the manifest snapshot in the operator image and the actual component operator version could cause drift, though the build pipeline is designed to prevent this.

**llm-d Stack Operational Complexity**: The disaggregated LLM serving stack (llm-d, llm-d-inference-scheduler, llm-d-kv-cache, llm-d-routing-sidecar) introduces significant operational complexity. The EPP requires ZeroMQ libraries for real-time KV-cache event streaming, Rust-compiled tokenizer binaries, and HuggingFace model access for tokenization. The routing sidecar implements three P/D protocols (NIXL v2, NIXL v1 deprecated, LMCache deprecated) with an SSRF protection layer that watches InferencePool resources. This multi-component data plane requires coordinated deployment and monitoring across 4+ container images per inference pod.

**TrustyAI Broad Access Grants**: The `default-lmeval-user-rolebinding` ClusterRoleBinding grants all `system:authenticated` users full CRUD access to LMEvalJob resources cluster-wide. Similarly, the odh-dashboard grants `system:authenticated` read access to accelerator profiles, hardware profiles, NIM accounts, and serving runtime templates. While convenient for onboarding, these broad grants mean any authenticated cluster user can create GPU-consuming evaluation jobs or view platform configuration, which may not align with multi-tenant environments.

**Module Federation for Extensibility**: The odh-dashboard's adoption of Webpack Module Federation for federated UI modules (model-registry-ui, gen-ai) is a forward-looking architectural decision that enables independent deployment of feature modules. However, the gen-ai BFF contains a hardcoded `InsecureSkipVerify=true` on its outbound HTTP transport, which must be addressed before production deployment. The model-registry BFF handles this correctly, highlighting inconsistent security practices across federated modules.

**Controller Concurrency Design**: Components show thoughtful concurrency configuration. Kueue uses concurrency=5 for high-throughput resources (Job, Pod, Workload) but concurrency=1 for quota-affecting control plane resources (ClusterQueue, LocalQueue), ensuring serialized reconciliation of quota changes. The rhods-operator's component controllers each run independently with their own leader election, and the DSC controller only creates component CRs — never directly deploying resources — preventing bottlenecks in the central operator.
