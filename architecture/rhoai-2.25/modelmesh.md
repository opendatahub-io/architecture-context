## Purpose

**Short**: ModelMesh is a distributed LRU cache and intelligent routing layer for high-scale, high-density model serving that runs as a sidecar alongside model runtime containers.

**Detailed**: ModelMesh is a mature, general-purpose model serving management/routing framework originally developed by IBM and contributed to the KServe project. It acts as a distributed LRU cache for serving runtime models, providing automatic model placement, scaling, and lifecycle management across a cluster of model-serving pods. Each ModelMesh instance runs as a sidecar container alongside a model runtime container (e.g., Triton, MLServer, custom runtimes), communicating with it via a local gRPC connection (TCP or Unix Domain Socket).

The framework provides two primary APIs: an external management API (`mmesh.ModelMesh` gRPC service) for registering, unregistering, and querying model status, and a dataplane API that transparently routes inference requests to the correct instance where the requested model is loaded. ModelMesh coordinates model placement across cluster instances using a distributed key-value store (etcd or ZooKeeper) and the Litelinks service discovery framework. It supports virtual models (vmodels) for zero-downtime model updates, type-based placement constraints, payload processing/logging, and comprehensive Prometheus and StatsD metrics. In the RHOAI platform, ModelMesh is deployed by the `modelmesh-serving` controller as part of `ServingRuntime` pod deployments.

## Architectural Analysis

ModelMesh's architecture follows the sidecar pattern, where the framework runs alongside model runtime containers in the same pod. This design provides transparent model caching, routing, and lifecycle management without requiring changes to the model runtime itself. The separation of concerns is clean: ModelMesh handles the distributed systems complexity (consensus, placement, routing, caching), while the model runtime focuses solely on model loading and inference via the simple `ModelRuntime` gRPC interface.

The distributed coordination layer is built on Litelinks (IBM's service discovery framework) and etcd (or ZooKeeper). Each ModelMesh instance registers itself in the KV store with its capacity, loaded models, and health status. A leader election mechanism ensures that only one instance coordinates global operations like model placement decisions. The `ConcurrentLinkedHashMap` provides an efficient LRU cache implementation, and models are evicted based on least-recently-used semantics when capacity is exceeded. The unload retry mechanism (up to 90 retries over ~15 minutes) provides resilience against transient model runtime failures.

The dataplane API is notable for its transparency: inference requests arrive on port 8033 with a `mm-model-id` header, and ModelMesh routes them to the correct instance and forwards them to the model runtime without deserializing the protobuf payload. The `ProtoSplicer` class performs efficient model-id injection into protobuf messages without full deserialization. This zero-copy approach, combined with Netty's direct buffer management, provides high-performance request forwarding. The `DataplaneApiConfig` allows runtime configuration of which RPCs are permitted and how model IDs are extracted from request payloads.

The VModel (virtual model) subsystem enables zero-downtime model updates by maintaining a mapping from a stable vmodel ID to a concrete model ID. When a new model version is deployed, the vmodel transitions atomically from the old to the new version only after the new version is fully loaded, preventing serving disruptions. This is particularly valuable for production model serving scenarios where model updates must not impact inference availability.

A security concern worth noting is that ModelMesh explicitly disables Java FIPS mode (`-Dcom.redhat.fips=false`) in its startup script, and reconfigures the SunPKCS11 security provider in the JDK security configuration. This is done to avoid compatibility issues when TLS is enabled, but it means the component does not operate in FIPS-compliant mode. The Netty tcnative-boringssl-static dependency provides native TLS but is not FIPS-validated. For RHOAI deployments on FIPS-enabled OpenShift clusters, this represents a known gap.
