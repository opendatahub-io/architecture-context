## Purpose

**Short**: Caikit-TGIS-Serving is a container image that provides a Caikit runtime frontend for Large Language Model (LLM) inference, backed by the Text Generation Inference Server (TGIS).

**Detailed**: Caikit-TGIS-Serving packages the Caikit AI runtime with the caikit-nlp and caikit-tgis-backend libraries into a container image that serves as a KServe ServingRuntime for LLM inference on Red Hat OpenShift AI. The image runs `python -m caikit.runtime`, which exposes both HTTP (port 8080) and gRPC (port 8085) APIs for model inference. It communicates with a co-located TGIS container (on localhost:8033) that loads and runs the actual model weights.

The repository itself contains no application source code — it is a packaging repository. The Python runtime, NLP modules, and TGIS backend integration are all pulled in as pip dependencies (`caikit==0.28.1`, `caikit-nlp==0.5.14`, `caikit-tgis-backend==0.1.39`). The repo provides the Dockerfile, Caikit runtime configuration (`caikit.yml`), a model conversion utility, demo manifests for deploying on KServe with Knative/Istio, and CI/CD pipeline definitions for Konflux builds. In production, this image runs as the `transformer-container` alongside a TGIS `kserve-container` within a KServe InferenceService pod, forming a two-container serving stack where Caikit handles API routing, model management, and health probing while TGIS provides the GPU-accelerated inference engine.

## Architectural Analysis

Caikit-TGIS-Serving is a thin packaging layer rather than a substantial application. The repository contains zero lines of application code — the entire runtime is pulled in via three Python packages (`caikit`, `caikit-nlp`, `caikit-tgis-backend`). The only custom file is `caikit.yml`, a 25-line configuration that wires the TGIS backend to the Caikit runtime. This design makes the repository a "glue" component: its architectural significance lies in how it composes upstream libraries and configures them for the RHOAI platform, not in any novel logic.

The two-container pod architecture (Caikit as `transformer-container` + TGIS as `kserve-container`) is a deliberate KServe pattern. Caikit serves as a protocol adapter and model management layer — it handles HTTP/gRPC API routing, health probing, model format detection (TGIS-AUTO finder), and lazy model loading. TGIS handles the computationally expensive inference work. Communication between them is via gRPC on localhost:8033, avoiding network encryption overhead for the high-throughput inference path. This localhost-only communication is a security feature: the TGIS gRPC port is never exposed outside the pod boundary.

The Dockerfile management reveals an interesting CVE remediation pattern. The Konflux Dockerfile (`Dockerfile.konflux`) uses poetry to install locked dependencies, while the upstream Dockerfile (`Dockerfile`) layers additional `pip install` commands on top to pin `urllib3>=2.6.0`, `fastapi==0.123.7`, `starlette>=0.49.1,<0.51.0`, and `aiohttp>=3.13.3,<4.0.0` as CVE fixes. These overlays exist because updating the Poetry lock file requires a full dependency resolution cycle, while direct pip pins provide faster CVE response. This dual-dockerfile approach — Konflux for production builds, upstream Dockerfile for development/CI — is common across RHOAI components.

The build pipeline is non-hermetic (`hermetic: false` in Tekton PipelineRun), which is a notable supply chain observation. The RHOAI version is `2.25.7` (from the Tekton push pipeline), and builds produce multi-arch images for `linux/x86_64` and `linux/arm64`. The pipeline uses `konflux-central` for its build definition, indicating centralized build infrastructure management across RHOAI components.
