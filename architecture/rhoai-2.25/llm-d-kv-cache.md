## Purpose

**Short**: A pluggable KV-cache indexing and management suite that enables KV-cache-aware routing, storage offloading, and cache eviction for LLM inference platforms.

**Detailed**: The llm-d-kv-cache repository provides the core KV-cache management capabilities for the llm-d distributed LLM inference platform. Its primary component is the **KV-Cache Indexer**, a high-performance Go library that maintains a near-real-time global view of KV-cache block locality across a fleet of vLLM or SGLang model-server pods. The indexer subscribes to KV-Events (emitted by model servers via ZeroMQ) and exposes a scoring API that ranks pods by how much of a request's prefix they already have cached, enabling intelligent KV-cache-aware routing decisions.

The library is designed to be embedded in a host process (llm-d's Envoy Processing Plugin / EPP is the reference consumer). It supports pluggable index backends (in-memory LRU, cost-aware Ristretto, Redis/Valkey), multiple engine adapters (vLLM and SGLang), and Kubernetes pod auto-discovery for dynamic ZMQ subscriber management. Beyond the core indexer, the repository ships three additional sub-components: a **UDS Tokenizer Service** (gRPC-based tokenization sidecar), an **FS Backend** connector (vLLM KV-cache offloading to filesystem or S3 object storage), and a **PVC Evictor** daemon (automatic disk space management for KV-cache PVCs).

## Architectural Analysis

The llm-d-kv-cache repository follows a **library-first architecture** — the core KV-Cache Indexer is designed as an embeddable Go library, not a standalone service. The reference deployment is as a linked dependency within llm-d's Envoy Processing Plugin (EPP), which calls `Indexer.ScoreTokens` directly in-process. The repository also ships a standalone binary (built from `examples/kv_events/online/main.go`) and a gRPC wrapper (from `examples/kv_cache_index_service/`) for deployments that prefer network-accessed scoring. This dual-mode design gives platform architects flexibility: embed for lowest latency, or deploy as a sidecar/service for looser coupling.

The **dual-key design** is the most architecturally significant pattern in the indexer. Model servers identify KV-cache blocks using engine-internal hashes (engine keys), while the indexer computes its own hash chain from prompt tokens (request keys). These two key spaces are independent — the indexer bridges them on ingestion by recording `engine_key → request_key` mappings when `BlockStored` events arrive. This decoupling means the indexer never needs to query a model server to score a prompt; it can compute request keys from tokens alone. The tradeoff is a second hash computation per block and a secondary index that must be maintained, but this enables the critical property of scoring without round-trips.

The **event ingestion pipeline** uses a sharded worker pool with FNV-1a hashing on pod ID to guarantee in-order processing per pod. This is essential because KV-cache events are causally ordered — a `BlockRemoved` referencing engine key E1 must be processed after the `BlockStored` that created the `E1 → R1` mapping. The pool supports both centralized ZMQ (single subscriber binds, all pods connect) and pod-discovery mode (Kubernetes label-selector watch creates per-pod subscribers). Pod-discovery mode is production-recommended because it supports multi-replica active-active deployments where each indexer replica independently converges to the same state.

The **Python sub-components** (UDS tokenizer, FS backend, PVC evictor) form a cohesive storage management stack. The FS backend offloads KV-cache blocks from GPU memory to shared storage (POSIX filesystem or S3 via NVIDIA NIXL), the PVC evictor manages disk space with threshold-based hysteresis (85% trigger / 70% target), and the UDS tokenizer provides efficient sidecar tokenization over Unix Domain Socket. The PVC evictor's multi-process N+2 architecture (N crawlers + 1 activator + 1 deleter) with hex-modulo load balancing is well-suited for multi-TB volumes with millions of small files. Notably, the FS backend uses `hashlib.md5` for non-cryptographic S3 object key sharding (`obj_backend.py`), which could be a FIPS compliance concern in environments with strict FIPS enforcement.

The deployment manifests are incomplete on this upstream branch — the kustomize structure references OpenShift routes and RBAC resources that don't yet exist. The Helm chart (`vllm-setup-helm/`) provides a more complete deployment story, packaging vLLM, the KV-cache manager, an optional external tokenizer sidecar, and optional Redis/LMCache. The PVC evictor has its own Helm chart with OpenShift-aware security context constraints (configurable fsGroup, SELinux level, runAsUser). For RHOAI, deployment is managed by the platform operator (rhods-operator) rather than these raw manifests.
