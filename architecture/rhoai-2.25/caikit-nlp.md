## Purpose

**Short**: Caikit-NLP is a Python library providing NLP capabilities (text generation, embeddings, reranking, classification, tokenization) built on top of the Caikit AI framework, designed for use with the RHOAI model serving stack.

**Detailed**: Caikit-NLP implements a suite of NLP modules that plug into the Caikit runtime framework. It is not a standalone service but a library that the Caikit runtime loads to provide NLP inference and training capabilities. The library wraps HuggingFace Transformers, sentence-transformers, and PEFT models behind Caikit's module interface, enabling them to be served via Caikit's gRPC and HTTP runtime servers.

The library supports two execution backends: **local** (in-process PyTorch inference) and **remote TGIS** (Text Generation Inference Server). For text generation, models can run locally using HuggingFace Transformers or remotely via gRPC to a TGIS backend. For embeddings and reranking, models run locally using sentence-transformers and cross-encoders. The TGIS backend integration enables prompt tuning vectors trained by caikit-nlp to be loaded into a remote TGIS server for efficient inference.

In the RHOAI platform, caikit-nlp is packaged as a container image (`odh-caikit-nlp-rhel9`) built on UBI 9. The image installs the caikit-nlp wheel into a Python virtualenv and is used as a base for model serving containers. It integrates with `caikit-tgis-backend` for remote text generation and relies on the Caikit runtime framework for gRPC/HTTP server lifecycle, model management, and health checking.

## Architectural Analysis

Caikit-NLP follows a modular architecture where each NLP capability is implemented as a Caikit "module" — a class decorated with `@module()` that declares its tasks, version, and unique ID. This design enables the Caikit runtime to dynamically discover and serve any module through a uniform gRPC/HTTP interface without module-specific server code. The module IDs (UUIDs) are stable identifiers used for model serialization and loading, ensuring forward compatibility.

The library implements a **dual-backend pattern** for text generation. Local modules (`TextGeneration`, `PeftPromptTuning`) load HuggingFace models directly into the Python process, while TGIS modules (`TextGenerationTGIS`, `PeftPromptTuningTGIS`) delegate inference to a remote TGIS server via gRPC. The TGIS modules are registered as "backend modules" through the `@module(backend_type=TGISBackend.backend_type, base_module=...)` decorator, allowing the Caikit runtime to automatically select the appropriate module based on available backends. The `TGISAutoFinder` model finder further automates this by probing TGIS connections and configuring modules transparently.

The embedding and reranking modules (`EmbeddingModule`, `CrossEncoderModule`) take a different approach — they always run locally using sentence-transformers. These modules implement custom truncation handling via `SentenceTransformerWithTruncate` and `CrossEncoderWithTruncate` subclasses that provide thread-safe tokenization (using per-thread tokenizer copies to avoid "Already borrowed" errors from Rust-backed fast tokenizers) and explicit token count tracking. The truncation logic is sophisticated: it supports configurable `truncate_input_tokens` behavior where 0 means "error on truncation", negative means "allow implicit truncation", and positive values specify an explicit token limit.

The PEFT prompt tuning module is architecturally notable for its training pipeline: it uses HuggingFace Accelerate for mixed-precision training, supports both vanilla prompt tuning and multitask prompt tuning (MPT), and exports trained prompt vectors as `.pt` files compatible with TGIS's prompt caching system. The `PeftPromptTuningTGIS` module then loads these vectors into a remote TGIS server for production inference. This decoupling of training (in caikit-nlp) from serving (in TGIS) is a key architectural pattern that enables efficient resource utilization.

The container image (`Dockerfile.konflux`) uses a multi-stage build: a builder stage compiles the wheel via tox, and a deploy stage installs it into a Python virtualenv at `/opt/caikit/`. The image runs as a non-root `caikit` user (UID 1001) and exposes a `/caikit/` volume for model and configuration files. Notably, the image's `CMD` is just `python` — the actual runtime startup command is expected to be provided by the orchestration layer (KServe/ModelMesh), typically `python -m caikit.runtime.grpc_server`.
