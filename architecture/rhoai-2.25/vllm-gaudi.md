## Purpose

**Short**: Intel Gaudi (Habana) hardware plugin for vLLM that enables high-performance large language model inference on Gaudi2 and Gaudi3 accelerators.

**Detailed**: vllm-gaudi is a hardware plugin package for the vLLM inference engine that integrates Intel Gaudi AI accelerators (formerly Habana Labs) with the vLLM serving framework. It provides HPU (Habana Processing Unit) specific implementations for attention backends, model runners, workers, distributed communication, and quantization operators, enabling vLLM's OpenAI-compatible API server to run efficiently on Gaudi hardware.

The plugin follows vLLM's pluggable hardware architecture (RFC #11162 and #19161), registering itself through Python entry points for platform, utility, ops, and model plugins. It provides custom implementations of core vLLM components optimized for HPU, including paged attention with 128-token block alignment, HPU graph compilation, FP8/INC/AWQ/GPTQ quantization, LoRA adapters, speculative decoding (EAGLE), KV cache transfer via NIXL, and a multi-model API server supporting dynamic model switching on a single HPU device. The component ships as a container image (`odh-vllm-gaudi-rhel9`) built on RHEL 9.6 UBI with Habana SynapseAI 1.24.0 and PyTorch 2.10.0, designed to be deployed as an inference serving runtime within RHOAI's model serving infrastructure (typically via KServe ServingRuntime).

## Architectural Analysis

The vllm-gaudi plugin is a substantial codebase (110+ Python files in the `vllm_gaudi/` package) that deeply integrates with vLLM's internal architecture. Rather than wrapping a minimal set of device operations, it provides comprehensive replacements for attention backends, model runners, workers, schedulers, and even the engine core itself. The plugin uses vLLM's entry point system (`vllm.platform_plugins` and `vllm.general_plugins`) for clean registration, but internally employs extensive monkey-patching — for example, replacing `torch.Tensor.data_ptr` globally to support HPU memory addressing, overriding `AsyncScheduler` with `HPUAsyncScheduler`, and patching `EngineCore` with `gaudi_reconfigure_engine` for multi-model support.

The v1 engine architecture is the primary target. The code contains both a `vllm_gaudi/attention/` (legacy v0) and `vllm_gaudi/v1/attention/` directory, but the platform class (`HpuPlatform`) directs all attention to the v1 backend (`HPUAttentionBackendV1`). The HPU attention implementation uses a unique 128-token block size (vs. GPU's typical 16-32), which is critical for Gaudi's memory access patterns. Hybrid models (Mamba/GDN + attention) require careful block size alignment logic in `HpuPlatform.check_and_update_config()`, with special handling for Qwen3.5 and GraniteMoE hybrids.

The multi-model switching feature (`MultiModelAsyncLLM`) is notable and HPU-specific — it enables a single Gaudi device to serve multiple LLMs sequentially by sleeping/waking models (moving weights between HPU and CPU memory), stashing compiled graph caches, and reconfiguring the engine core in-place via cloudpickle serialization. This is exposed through the non-standard `/v1/models/switch` endpoint, gated behind `VLLM_SERVER_DEV_MODE=1`. The implementation includes error recovery with model rollback if reconfiguration fails.

The Dockerfile build is notably complex — a 3-stage multi-layer build that installs Habana SynapseAI drivers, PyTorch for Gaudi, and vLLM from source, all targeting RHEL 9.6 UBI. The explicit removal of `openssl-fips-provider-so` is a significant security architecture decision that prevents FIPS-compliant operation. The container runs as non-root (UID 2000, GID 0) for OpenShift compatibility. The build is non-hermetic (`hermetic: false` in Tekton), fetching dependencies from multiple external sources (PyPI, Habana Artifactory, CentOS mirrors, EPEL), which is a supply chain consideration but expected given the hardware-specific driver requirements.

The quantization subsystem supports multiple schemes: INC (Intel Neural Compressor), compressed-tensors, FP8 (with dynamic KV cache quantization), AWQ, GPTQ, and ModelOpt. Each has a dedicated ops module in `vllm_gaudi/ops/`. The calibration pipeline (`calibration/`) provides a multi-step workflow for FP8 quantization: device detection → dataset preparation → scale measurement → postprocessing → scale quantization → unification → expansion. This is critical for production deployment, as quantized models significantly reduce HPU memory requirements.
