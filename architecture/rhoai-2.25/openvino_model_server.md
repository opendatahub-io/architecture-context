## Purpose

**Short**: OpenVINO Model Server is a high-performance C++ inference server that serves AI/ML models via gRPC, REST, and OpenAI-compatible APIs using the Intel OpenVINO runtime.

**Detailed**: OpenVINO Model Server (OVMS) is a production-grade inference serving platform developed by Intel and integrated into Red Hat OpenShift AI (RHOAI). It provides model serving capabilities through multiple API protocols — KServe v2, TensorFlow Serving, and OpenAI-compatible generative AI APIs — enabling clients to perform remote inference on models optimized for Intel architectures.

The server supports a wide range of AI workloads: traditional computer vision and NLP models via KServe/TFS APIs, large language model (LLM) text generation with continuous batching via OpenAI-compatible chat/completions endpoints, text embeddings via OpenAI-compatible embeddings API, document reranking via Cohere-compatible API, and image generation via OpenAI-compatible image generation API. Models can be loaded from local filesystem, cloud object storage (S3, Azure Blob, GCS), or pulled directly from HuggingFace Hub.

OVMS runs as a standalone container image built on UBI 9 and is deployed within RHOAI as part of the model serving infrastructure. It uses Bazel as its build system, compiles OpenVINO from source with LTO optimizations for the RHOAI distribution, and supports hardware acceleration on Intel CPUs, GPUs, and NPUs. The server also provides a C API (libovms_shared.so) for embedding inference capabilities into other applications, and supports MediaPipe graph-based pipelines and DAG scheduling for complex multi-model inference workflows.

## Architectural Analysis

OpenVINO Model Server follows a modular C++ architecture with a clear separation between protocol frontends, model management, and inference execution. The server binary (`ovms`) initializes modules in a specific order: signal handlers → profiler → metrics → Python interpreter → servable manager → C API → gRPC → HTTP. This ordered startup ensures liveness probes respond quickly (HTTP/gRPC modules start before the servable manager finishes loading models). Shutdown follows the reverse order, with a 5-second deadline for gRPC server draining.

The server's multi-protocol support is architecturally interesting. Rather than implementing each API independently, the REST endpoints internally convert to gRPC request format and delegate to shared inference service implementations. This means the KServe v2 REST API, TFS REST API, and their gRPC counterparts share the same inference codepath — the protocol frontier is thin. The OpenAI-compatible v3 APIs take a different path, routing through MediaPipe graph calculators (LLM calculator, embeddings calculator, rerank calculator) which encapsulate the model-specific inference logic. This design allows the generative AI APIs to leverage MediaPipe's streaming and graph-composition capabilities.

The continuous batching architecture for LLM serving is the most sophisticated module. It uses OpenVINO GenAI's `ContinuousBatchingPipeline` with configurable `SchedulerConfig` to manage dynamic batching of multiple concurrent requests. Key parameters include `max_num_batched_tokens` (default 256), `cache_size` (default 8GB for KV-cache), and `max_num_seqs` (default 256). The pipeline supports speculative/assisted decoding with draft models, prefix caching for prompt reuse optimization, and streaming token generation. An `LLMExecutorWrapper` manages a dedicated executor thread that processes batched generation asynchronously, with results streamed back to clients via Server-Sent Events (SSE) for the REST API or bidirectional streaming for gRPC.

A notable security consideration is that OVMS itself provides NO authentication, authorization, or TLS termination. The gRPC server explicitly uses `grpc::InsecureServerCredentials()`, and the REST server has no TLS configuration. This is by design — in RHOAI deployments, the platform operator injects kube-rbac-proxy or oauth-proxy sidecars to handle authentication, and the service mesh (Istio) provides mTLS between services. However, this means that a misconfigured deployment (e.g., directly exposing OVMS ports without a proxy) would serve inference requests without any authentication. The `extras/nginx-mtls-auth/` directory provides a standalone nginx wrapper with mutual TLS for non-platform deployments, but this is not used in the RHOAI distribution.

The build process for RHOAI is notable for its complexity and lack of hermeticity. The Dockerfile.konflux is a multi-stage build that compiles OpenVINO from source (not using pre-built binaries), builds Boost, Azure SDK, OpenCV, and numerous other dependencies from source, then builds OVMS itself via Bazel. The Tekton pipeline explicitly sets `hermetic: false`, and the build downloads dependencies from multiple external sources (GitHub, SourceForge, Fedora repos, CentOS repos). Dependencies are pinned via git commit hashes (e.g., `ov_source_branch=c01cd93e...`) which provides reproducibility but not isolation from upstream compromise. The `.konflux/build-args.conf` configures the RHOAI-specific build with LTO optimization (`-flto=auto -ffat-lto-objects -march=haswell`), building from the `opendatahub-io` fork of OpenVINO with the `releases/2025/3` branch.
