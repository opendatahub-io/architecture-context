## Purpose

**Short**: Manages the lifecycle of RayCluster resources and optionally AppWrapper resources, providing OAuth proxy injection, mTLS certificate generation, network policies, and ingress for Ray Dashboard access on OpenShift.

**Detailed**: The CodeFlare Operator is a Kubernetes operator built with controller-runtime that reconciles `RayCluster` custom resources (owned by the KubeRay project, `ray.io/v1`) and optionally `AppWrapper` resources (`workload.codeflare.dev/v1beta2`). It does not define its own primary CRD but instead watches and enhances RayClusters created by the KubeRay operator.

When a RayCluster is created, the operator's mutating webhook injects an OAuth proxy sidecar container and mTLS init containers into the head and worker pod specs. The controller then reconciles supporting resources: OpenShift Routes for dashboard and Ray client access, OAuth secrets, ServiceAccounts with OAuth redirect annotations, ClusterRoleBindings for auth delegation, and NetworkPolicies that restrict traffic to the Ray head and worker pods. On vanilla Kubernetes (non-OpenShift), it creates Ingress resources instead of Routes.

The operator also embeds the AppWrapper controller (from `github.com/project-codeflare/appwrapper`) which integrates with Kueue for batch workload scheduling. When enabled via configuration, AppWrappers wrap Kubernetes resources (Deployments, Jobs, RayClusters, PyTorchJobs) and manage their lifecycle with quota-aware scheduling. The operator loads its configuration from a ConfigMap (`codeflare-operator-config`) and uses cert-controller for webhook certificate rotation.

## Architectural Analysis

The CodeFlare Operator follows a distinctive "enhancer" pattern rather than the typical operator model of managing its own CRD. Instead of owning a primary CRD, it watches `RayCluster` resources owned by the KubeRay operator and augments them with enterprise security features (OAuth, mTLS, network isolation). This architectural choice keeps the operator lightweight — its core controller code is contained in a single file (`raycluster_controller.go`, ~714 lines) with webhook logic in `raycluster_webhook.go` (~468 lines).

The operator's webhook-based injection strategy is notable: the mutating webhook intercepts RayCluster creates/updates to inject an `oauth-proxy` sidecar container, mTLS init containers (`create-cert`), TLS environment variables, and certificate volumes. The validating webhook then enforces immutability of these injected components, preventing users from removing or modifying security-critical configuration. This pattern ensures security is applied declaratively without requiring users to configure it manually, but creates a tight coupling between the operator version and the injected container images (`OAUTH_PROXY_IMAGE` and `CERT_GENERATOR_IMAGE`).

The mTLS implementation uses a per-cluster CA approach: the controller generates a self-signed RSA 2048-bit CA certificate for each RayCluster and stores it in a Secret. Init containers on both head and worker pods use this CA to generate per-pod server certificates via `openssl` commands at pod startup. The certificates include pod-specific SANs (pod IP, localhost, service DNS, route hostname). While functional, this approach has a notable gap: the CA certificates have a 1-year validity and are not automatically rotated (unlike the webhook certificates which use cert-controller auto-rotation). The use of SHA1 for OAuth cookie secret generation (`sha1.New()` in raycluster_controller.go:468) is a minor FIPS concern, though it's used only for cookie generation and not for cryptographic verification.

The AppWrapper functionality is embedded directly from the `github.com/project-codeflare/appwrapper` library rather than deployed as a separate controller. This design reduces operational complexity (one deployment instead of two) but means the AppWrapper controller's lifecycle is tied to the operator's. The AppWrapper is disabled by default and must be explicitly enabled in the `codeflare-operator-config` ConfigMap. When enabled, it registers its own controllers and webhooks alongside the RayCluster controller, sharing the same webhook certificate and manager process.

Network isolation is implemented through dynamically-created NetworkPolicies per RayCluster. The head node policy allows traffic from the KubeRay operator namespace (auto-discovered), the OpenShift monitoring namespace (for metrics scraping), and same-namespace pods on specific ports. Worker pods are fully isolated to intra-cluster communication only. The operator dynamically discovers the KubeRay operator's namespace by listing pods with the `app.kubernetes.io/component=kuberay-operator` label, with fallback to well-known namespaces (`ray-system`, `opendatahub`, `redhat-ods-applications`). This is more resilient than hardcoding namespaces but introduces a runtime dependency on pod discovery.
