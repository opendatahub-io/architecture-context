## Purpose

**Short**: Feast is a feature store for machine learning that provides a Kubernetes operator to deploy and manage feature serving infrastructure, including online/offline stores, registries, and feature transformation services.

**Detailed**: Feast (Feature Store) is an open-source feature store for machine learning, providing a central repository for storing, managing, and serving ML features. The RHOAI distribution includes two primary deployable components: the Feast Operator and the Feature Server.

The Feast Operator is a Kubernetes controller-runtime-based operator that manages `FeatureStore` custom resources. When a user creates a FeatureStore CR, the operator reconciles it by deploying a complete feature serving stack — including Deployments for registry, online serving, offline serving, and UI services, along with Services, ConfigMaps, PersistentVolumeClaims, ServiceAccounts, RBAC resources, CronJobs for materialization, and optionally OpenShift Routes for external access. The operator supports multiple authorization models (Kubernetes RBAC and OIDC), automatic TLS via OpenShift service serving certificates, and cross-namespace feature store discovery via a namespace registry.

The Feature Server is a Python-based FastAPI application that serves as the primary data plane, providing HTTP endpoints for online feature retrieval, feature pushing, materialization, document retrieval, and a chat interface. It also includes a Go-based embedded feature server that can serve features via gRPC or HTTP with support for multiple online store backends (Redis, DynamoDB, SQLite). The Python server supports Prometheus metrics, MCP (Model Context Protocol) integration, and pluggable authorization via OIDC or Kubernetes RBAC.

## Architectural Analysis

The Feast component follows a layered architecture with clear separation between the control plane (operator) and the data plane (feature server). The operator is a standard controller-runtime operator that watches a single CRD (`FeatureStore`) and reconciles it into a full-stack deployment. This design means a single FeatureStore CR produces approximately 10-15 Kubernetes resources — a Deployment with multiple containers (one per service mode), Services, ConfigMaps, PVCs, RBAC resources, and optionally Routes and CronJobs. The operator uses owner references for garbage collection, so deleting the FeatureStore CR cleanly removes all child resources.

A notable architectural pattern is the "namespace registry" — a global ConfigMap (`feast-configs-registry`) maintained in the operator's system namespace that tracks all deployed FeatureStore instances across the cluster. This enables cross-namespace discovery: a FeatureStore in namespace A can reference a remote registry in namespace B via `spec.services.registry.remote.feastRef`. The operator watches for changes to remote registries and re-reconciles dependent FeatureStores, creating a dependency graph across namespaces. The namespace registry ConfigMap grants read access to all authenticated users via a dedicated Role/RoleBinding.

The TLS architecture is tightly coupled to OpenShift. When running on OpenShift (detected at startup by probing the `route.openshift.io` API group), the operator automatically provisions TLS via service-serving certificate annotations. Each feast service gets a TLS secret (e.g., `feast-{name}-registry-tls`), and services switch to HTTPS ports. The operator also handles Subject Alternative Names (SANs) for services that expose multiple endpoints (e.g., registry gRPC + REST). On non-OpenShift Kubernetes, TLS is not automatically provisioned — users must provide their own certificates or operate without TLS. Additionally, the operator supports custom CA bundles via the `odh-trusted-ca-bundle` ConfigMap (label-driven injection), which is critical for RHOAI deployments with internal CAs.

The authorization model supports three modes: no auth (default), Kubernetes RBAC, and OIDC. The Kubernetes RBAC mode creates empty Roles that administrators populate with rules to control feature access, while the OIDC mode stores provider configuration in a Kubernetes Secret referenced by the FeatureStore CR. The Python feature server enforces permissions at the endpoint level using FastAPI dependency injection (`inject_user_details`), with each endpoint checking specific `AuthzedAction` permissions (READ_ONLINE, WRITE_ONLINE, WRITE_OFFLINE). The Go embedded server does not implement the same permission model — it provides a lower-level serving interface without authorization enforcement.

The CronJob architecture is noteworthy: rather than running feast CLI commands in a separate container with direct database access, the CronJob uses `kubectl exec` into the existing feast Deployment pod to run `feast apply` and `feast materialize-incremental`. This avoids duplicating database credentials and configuration, but requires the CronJob ServiceAccount to have `pods/exec` permission — a relatively privileged operation. The CronJob schedule defaults to `@yearly` (effectively never), requiring users to set an appropriate schedule for their materialization needs.
