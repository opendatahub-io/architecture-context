## Purpose

**Short**: Builds and ships container images that provide interactive data science workbench environments (JupyterLab, code-server, RStudio) and headless Elyra pipeline runtime images for Red Hat OpenShift AI.

**Detailed**: The notebooks repository is a multi-image build factory that produces all user-facing workbench container images for the RHOAI platform. These images are not operators or services — they are the environments that data scientists launch via the RHOAI Dashboard. When a user creates a "Workbench" in the Dashboard, the platform's notebook controller (odh-notebook-controller) creates a StatefulSet running one of these images as a pod.

The repository produces three categories of images: (1) **Workbench images** — full interactive IDE environments including JupyterLab notebooks (minimal, datascience, PyTorch, TensorFlow, TrustYAI, PyTorch+LLMCompressor variants), code-server (VS Code in browser), and RStudio Server; (2) **Runtime images** — headless Python environments used by the Elyra pipeline editor to execute individual notebook nodes as Kubernetes jobs in data science pipelines (Kubeflow Pipelines / Tekton); (3) **Base images** — CUDA and ROCm GPU base layers consumed by the workbench and runtime images.

All images share a layered build architecture: a `BASE_IMAGE` (AIPCC — AI Platform Container Collection) provides the OS + Python foundation, then each Dockerfile adds framework-specific libraries (PyTorch, TensorFlow, etc.) and IDE tooling (JupyterLab, code-server, RStudio). Images are built for multiple architectures (amd64, ppc64le, s390x) and GPU accelerators (NVIDIA CUDA, AMD ROCm). The kustomize manifests in `manifests/` produce OpenShift ImageStream resources that the platform operator deploys to make these images selectable in the Dashboard.

_Python dependencies are locked per-image via PEP 751 `pylock.toml` files with hash verification (`--verify-hashes`). The `uv pip install` commands in Dockerfiles enforce strict hash checking via `--verify-hashes --compile-bytecode`. RPM-level locking is absent — OS package versions float with the UBI9 base image updates. No Hermeto/cachi2 prefetch integration is present, meaning builds download Python packages from PyPI at build time rather than from a pre-fetched cache._

## Architectural Analysis

The notebooks repository follows a **layered image build architecture** where each workbench variant extends a simpler base. The CPU build chain is: `AIPCC base → cpu-base → jupyter-minimal → jupyter-datascience → {pytorch, tensorflow, trustyai, llmcompressor}`. The CUDA chain mirrors this: `AIPCC CUDA base → cuda-base → cuda-jupyter-minimal → cuda-jupyter-datascience → {cuda-jupyter-pytorch, cuda-jupyter-tensorflow}`. This layering ensures that all images share common OS packages, security patches, the `oc` CLI, and Python tooling, while each layer adds only its specific framework dependencies. The layering is visible in the multi-stage Dockerfiles where intermediate stages are reused via `FROM ... AS` directives.

**Multi-architecture support** is a significant engineering investment in this repository. Every image must build on amd64 (primary), ppc64le (IBM Power), and s390x (IBM Z). Since many Python packages (pyarrow, ONNX, OpenBLAS) lack pre-built wheels for Power and Z, the Dockerfiles include elaborate multi-stage builds that compile these packages from source. The `TARGETARCH` build argument gates architecture-specific logic: on amd64 packages install from PyPI wheels, while on ppc64le and s390x entire compilation toolchains (gcc-toolset-13, cmake, Rust, etc.) are installed to build packages from source. This makes the Konflux Dockerfiles substantially more complex than typical application Dockerfiles.

**Dependency management** has migrated from Pipfile.lock to PEP 751 `pylock.toml` files, with `uv` replacing `pipenv` as the package installer. All installs use `--verify-hashes --compile-bytecode --no-deps --strict` flags for supply chain security. The root-level `uv.lock` file locks test/CI dependencies (Python 3.14 for the test suite), while per-image `pylock.toml` files lock production Python dependencies (Python 3.12 for workbench images, Python 3.11 for RStudio). The separation ensures test tooling doesn't contaminate production images.

**Code-server and RStudio images use NGINX as a reverse proxy**, unlike JupyterLab which directly listens on :8888. These images run multiple processes: the IDE (code-server on :9999 or RStudio on :8787), NGINX on :8787 for proxying and health checks, and httpd on :8080 for CGI scripts that validate kernel access. The NGINX proxy enables the notebook controller's health probes to succeed even when the IDE is still initializing — NGINX returns a CGI-based probe response while the IDE starts up. This architectural choice adds complexity but improves reliability in the Kubernetes pod lifecycle.

**CVE management is active and ongoing**, as seen in the recent commit history: CVE-2026-2492 (TensorFlow HDF5 plugin privilege escalation mitigated via `HDF5_PLUGIN_PATH=disable`), CVE-2026-32274 (black library update), CVE-2026-27904/CVE-2026-26996 (minimatch update), and proactive `dnf upgrade` at the top of every Dockerfile to pull in UBI9 security patches. The `.syft.yaml` configuration disables source scanning to prevent false-positive CVEs from cross-image contamination in the SBOM generation. The RStudio image includes a dedicated `cve_remediation.sh` script for npm-level CVE fixes.
