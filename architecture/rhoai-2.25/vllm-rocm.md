## Purpose

**Short**: GPU-accelerated vLLM inference runtime for AMD ROCm, repackaged from RHAIIS (Red Hat AI Inference Server) for RHOAI distribution.

**Detailed**: This repository is a downstream packaging repository that produces the `odh-vllm-rocm-rhel9` container image for Red Hat OpenShift AI (RHOAI). It contains no application source code — instead, it builds FROM the upstream Red Hat AI Inference Server (RHAIIS) vllm-rocm image (`registry.redhat.io/rhaiis/vllm-rocm-rhel9`) and applies RHOAI-specific configuration: environment variables, labels, and the entry point command.

The resulting container runs vLLM with the TGIS (Text Generation Inference Server) adapter, exposing both an OpenAI-compatible HTTP API on port 8000 and a gRPC endpoint on port 8033. This enables the image to serve as a ServingRuntime in KServe-based model serving pipelines, providing high-performance LLM inference on AMD GPU hardware using the ROCm software stack. The TGIS adapter layer provides backward compatibility with the TGIS gRPC protocol used by earlier RHOAI model serving implementations.

The image is built by Konflux CI/CD pipelines and published to `quay.io/rhoai/odh-vllm-rocm-rhel9`. All source code, model loading logic, inference engine, and the TGIS adapter are inherited from the RHAIIS base image — this repo's sole responsibility is the final image composition and CI pipeline configuration.

## Architectural Analysis

The vllm-rocm repository represents the thinnest possible downstream packaging pattern: a single-stage Dockerfile that re-layers an upstream product image (RHAIIS vllm-rocm-rhel9) with only environment variables, an entry point, and Red Hat container labels. No source code is compiled, no dependencies are installed, and no configuration files are added. This pattern minimizes maintenance burden — the repository's sole purpose is to provide a Konflux-buildable entry point for the RHOAI distribution pipeline.

The TGIS adapter entry point (`python3 -m vllm_tgis_adapter`) is architecturally significant. It wraps vLLM's native OpenAI-compatible HTTP server with a gRPC layer implementing the TGIS (Text Generation Inference Server) protocol. This dual-protocol design allows the same container to serve both the modern OpenAI API (port 8000) and the legacy TGIS gRPC protocol (port 8033), enabling backward compatibility with older RHOAI model serving configurations. The `DISABLE_LOGPROBS_DURING_SPEC_DECODING=false` environment variable override is a deliberate product decision — vLLM disables logprobs during speculative decoding for performance, but RHOAI re-enables them because hosted model users expect consistent behavior regardless of whether speculative decoding is configured.

The build pipeline configuration reveals important operational details. The Tekton pipeline uses the `multi-arch-container-build.yaml` pipeline from `konflux-central`, builds only for `linux/x86_64` (ARM64 is commented out with a Slack thread reference), has a 9-hour total pipeline timeout with 6-hour task timeout (reflecting the large image size), and runs ecosystem certification preflight checks and Clair vulnerability scans with elevated resource limits (16Gi memory, 8 CPU). The `hermetic: false` setting is intentional since the Dockerfile performs no network-dependent operations — it only re-layers a pre-built image.

The RHAIIS version pinning (`RHAIIS_VERSION=3.2.2-1772160625`) is the critical dependency management mechanism for this repo. Upstream RHAIIS image updates flow through PRs that bump this version (visible in the git log: "Update RHAIIS_VERSION in Dockerfile", "consume rhaiis-3.2.2.post1"). This creates a clear supply chain: upstream vLLM -> RHAIIS vllm-rocm-rhel9 -> RHOAI odh-vllm-rocm-rhel9.
