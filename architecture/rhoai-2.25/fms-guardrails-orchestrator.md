## Purpose
**Short**: A REST API orchestrator that coordinates AI text generation with content safety guardrails by routing requests through configurable detector and chunker services.

**Detailed**: The FMS Guardrails Orchestrator serves as middleware between clients and language models, providing content safety detection, filtering, and secure text generation. It manages multiple detector services (for content analysis, chat detection, context document detection, and generation detection), text chunking services, and LLM backends (TGIS, caikit-nlp, OpenAI-compatible) through unified HTTP APIs.

The orchestrator exposes two HTTP servers: a main guardrails API server (default port 8033) and a health check server (default port 8034). It supports both unary and streaming request/response patterns, including SSE (Server-Sent Events) for streaming responses and NDJSON for streaming content detection. The service supports TLS (including mTLS) for both inbound connections and outbound connections to backend services. OpenTelemetry tracing and metrics export is built in via OTLP (gRPC or HTTP).

Configuration is YAML-driven, allowing operators to define generation providers, chunker services, detector services, TLS configurations, and header passthrough rules. The orchestrator validates all configurations at startup and optionally performs health checks on all configured backend services.

### TLS Configuration

The orchestrator supports comprehensive TLS configuration:

**Server-side TLS** (for incoming connections):
- Configured via `--tls-cert-path`, `--tls-key-path` CLI args
- Optional mTLS via `--tls-client-ca-cert-path`
- Uses rustls with ring crypto provider
- Supports TLS 1.2 and TLS 1.3

**Client-side TLS** (for outbound connections):
- Configured per-service in config.yaml via named TLS configs
- Supports cert/key/CA per backend service
- Supports `insecure: true` to skip server cert verification
- gRPC clients use tonic TLS; HTTP clients use hyper-rustls
