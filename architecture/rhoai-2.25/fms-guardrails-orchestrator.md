## Purpose

**Short**: A REST API orchestrator that coordinates AI text generation with safety guardrails by routing requests through configurable detector, chunker, and LLM backend services.

**Detailed**: The FMS Guardrails Orchestrator (`fms-guardrails-orchestr8`) acts as middleware between clients and language models to provide content safety detection, filtering, and secure text generation. It receives user requests via HTTP REST endpoints, optionally generates text through upstream LLM backends (TGIS via gRPC or OpenAI-compatible via HTTP), and runs the input and/or output text through configurable detector services for content analysis (e.g., hate speech, PII, toxicity). Text is chunked via chunker services before being sent to detectors.

The orchestrator supports multiple API versions (v1 and v2), both unary and streaming response patterns, and OpenAI-compatible chat completions with detection. It manages TLS connections to all downstream services, provides health checking for all configured backends, exports OpenTelemetry traces and metrics, and supports header passthrough for authentication token forwarding. The service is designed for deployment as a sidecar or standalone service in RHOAI inference pipelines, providing a unified guardrails layer across different model serving backends.

## Architectural Analysis

The FMS Guardrails Orchestrator follows a clean middleware architecture pattern where it sits between API clients and multiple backend services (LLMs, detectors, chunkers). The core design uses a trait-based handler dispatch system (`Handle<Task>`) where each API endpoint creates a typed task struct, and the `Orchestrator` dispatches it to the appropriate handler implementation. This pattern provides compile-time type safety across the 12 distinct handler implementations while maintaining a unified orchestration interface.

The client abstraction layer is particularly well-designed: a heterogeneous `ClientMap` stores different client types (gRPC-based TgisClient/NlpClient/ChunkerClient, HTTP-based DetectorClient/OpenAiClient) behind a sealed `Client` trait with type-safe downcast support. The `GenerationClient` wrapper unifies TGIS and NLP backends behind a common interface with configurable retry logic (exponential backoff for 502-506 status codes). All clients support configurable TLS with per-service certificate configuration, DNS-based load balancing for gRPC (via ginepro), and comprehensive health checking.

Streaming is a first-class concern throughout the architecture. The orchestrator supports four streaming patterns: SSE (Server-Sent Events) for v1 classification streaming and OpenAI-compatible endpoints, ND-JSON (Newline-Delimited JSON) for v2 streaming content detection with bidirectional input, and gRPC bidirectional streaming for chunker communication. The detection batcher system (`CompletionBatcher` for choice-indexed detection, `MaxProcessedIndexBatcher` for chunk-indexed detection) ensures in-order aggregation of detection results across concurrent detector invocations during streaming, with broadcast fan-out for multi-detector subscriptions.

A notable architectural risk is the TLS implementation: the service uses rustls with the ring crypto backend rather than OpenSSL. While OpenSSL is installed in the container image (for UBI base compatibility and potentially other runtime dependencies), the application itself performs all TLS operations through ring, which is not FIPS-certified. The `NoVerifier` struct in `src/utils/tls.rs` that disables certificate verification when `insecure: true` is configured represents another security consideration — while useful for development, it should be carefully controlled in production deployments. The header passthrough mechanism (`passthrough_headers` with optional `rewrite_forwarded_access_header` to convert X-Forwarded-Access-Token into Bearer tokens) delegates authentication to downstream services rather than enforcing it at the orchestrator level.

The observability stack is comprehensive: OpenTelemetry integration provides W3C Trace Context propagation across all HTTP and gRPC calls, with custom tower layers (`OtelGrpcLayer` for gRPC, `ClientMakeSpan`/`ClientOnRequest`/`ClientOnResponse` for HTTP) that emit request counts, latency histograms, and error rate counters. The tracing configuration supports multiple log formats (Full, Compact, Pretty, JSON) and configurable OTLP export endpoints for both traces and metrics.
