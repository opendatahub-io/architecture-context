## Purpose

**Short**: A collection of detector algorithm microservices for the FMS Guardrails Orchestrator, providing text content analysis for safety, PII detection, file-type validation, and LLM-as-a-judge evaluation.

**Detailed**: Guardrails Detectors is a repository that produces multiple independent detector microservices designed to work with the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator). Each detector is a standalone FastAPI-based Python service that implements the IBM Detector API specification, exposing REST endpoints for text content analysis.

The repository contains three detector types: (1) a **built-in detector** that bundles lightweight heuristic detection functions (regex-based PII detection and file-type validation) into a single service requiring no external model; (2) a **HuggingFace detector** that loads transformer models (AutoModelForSequenceClassification or GraniteForCausalLM) from a local directory to perform ML-based content classification such as hate/abuse/profanity detection and safety guardrailing; and (3) an **LLM Judge detector** that delegates evaluation to an external OpenAI-compatible LLM server (e.g., vLLM) using the `vllm_judge` library for flexible metric-based content assessment.

Both the HuggingFace and LLM Judge detectors are designed to be deployed as KServe InferenceServices on RHOAI, using ServingRuntime definitions that integrate with the platform's model serving infrastructure. The built-in detector runs as a standalone pod. All detectors share a common library (`detectors/common/`) providing the FastAPI base application class, Pydantic request/response schemas, Prometheus metrics instrumentation, and health check endpoints.

## Architectural Analysis

The guardrails-detectors repository follows a clean microservice architecture where each detector type is an independent deployable unit sharing a common library layer. The `detectors/common/` package provides a well-designed `DetectorBaseAPI` base class that standardizes health checks, error handling, exception formatting, and detector lifecycle management (registration, retrieval, cleanup) across all detector types. This shared base ensures consistent API behavior and error responses regardless of which detector is deployed.

The built-in detector uses an extensible registry pattern (`BaseDetectorRegistry` abstract class) that allows new detector types to be added via subclassing. The `RegexDetectorRegistry` and `FileTypeDetectorRegistry` implementations are instantiated at application startup and stored in the FastAPI application state. Notably, the built-in detector supports custom regex patterns — clients can pass arbitrary regex strings that aren't in the predefined registry, which is powerful but could be a security consideration (ReDoS attacks via malicious regex patterns). The credit card detector implements Luhn checksum validation beyond simple regex matching, showing attention to detection accuracy.

The HuggingFace detector supports two model architectures: `AutoModelForSequenceClassification` (multi-label classification with configurable safe labels and thresholds) and `GraniteForCausalLM` (IBM Granite Guardian causal models that generate safe/unsafe tokens). The `SAFE_LABELS` environment variable allows operators to configure which model output labels should be excluded from detection results, providing flexibility across different model label schemes. CUDA/GPU support is automatic — the detector moves the model to GPU if available and falls back to CPU. The Konflux Dockerfile includes multi-architecture support with special build stages for `ppc64le` (building PyTorch and OpenBLAS from source) and `s390x`, reflecting RHOAI's multi-architecture requirements.

The LLM Judge detector is architecturally distinct from the other two — it performs no local inference but instead delegates all evaluation to an external vLLM server via the `vllm_judge` library. This makes it lightweight (no model loading, no GPU needed) but adds a runtime dependency on a functioning LLM server. It exposes an additional `/api/v1/text/generation` endpoint not present in the other detectors, allowing evaluation of prompt+response pairs rather than just standalone text. The `vllm_judge` library provides a rich set of built-in metrics (safety, toxicity, accuracy, helpfulness, clarity, etc.) and supports custom criteria via natural language descriptions.

A notable gap is that the LLM Judge detector has no Konflux Dockerfile — only `Dockerfile.judge` exists. This means it is not currently built through the Konflux/Tekton CI pipeline for RHOAI releases, and only the built-in and HuggingFace detectors have Konflux-based builds with multi-architecture support. The `.tekton/` directory confirms only `odh-built-in-detector` and `odh-guardrails-detector-huggingface-runtime` have pipeline definitions. This may indicate the LLM Judge is in an earlier maturity stage or is not yet shipped as part of RHOAI 2.25.

All three detectors run as plaintext HTTP services, relying on the Istio service mesh sidecar (annotated via `sidecar.istio.io/inject: 'true'` in the KServe manifests) for transport encryption. The detectors themselves have no TLS termination, authentication, or authorization — platform-level security is entirely delegated to the KServe/Istio/RHOAI infrastructure layer via the `security.opendatahub.io/enable-auth: 'true'` annotation.
