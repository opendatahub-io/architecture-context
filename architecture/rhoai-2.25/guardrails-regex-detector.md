## Purpose

**Short**: A lightweight Rust HTTP service that detects PII and custom patterns in text using regular expressions, serving as a detector backend for the FMS Guardrails Orchestrator.

**Detailed**: The guardrails-regex-detector is a stateless HTTP microservice built in Rust using the Axum web framework. It exposes a single detection endpoint that accepts text content and regex parameters, then scans the text for matches and returns structured detection results with character offsets, detection types, and confidence scores.

The service ships with three built-in PII detection patterns — email addresses, Social Security numbers (SSN), and credit card numbers — which clients can invoke by name. Additionally, clients can pass arbitrary regular expressions for custom pattern matching. All detections return a uniform response schema with start/end character positions, matched text, detection type classification, and a score of 1.0 (deterministic regex match).

This component is designed to be deployed as a sidecar or companion service alongside the FMS Guardrails Orchestrator, which routes text content through multiple detector backends (including this one) to enforce content safety guardrails on LLM inputs and outputs within the RHOAI TrustyAI stack.

## Architectural Analysis

The guardrails-regex-detector is an intentionally minimal service with a very small codebase (~210 lines of Rust across two source files). Its architecture follows a simple request-response pattern with no persistent state, no database connections, and no external service calls. This makes it highly suitable for horizontal scaling and sidecar deployment patterns.

The detection system uses a two-tier matching approach: built-in named detectors (email, SSN, credit card) are registered in a HashMap and invoked by keyword, while arbitrary user-supplied regex patterns are compiled and executed on the fly. All regex patterns are compiled per-request using the `regex` crate, which means there is no pattern caching across requests. For high-throughput scenarios, this could become a performance bottleneck if clients repeatedly send the same custom patterns — precompiling and caching regex patterns would be an optimization opportunity.

The service has no authentication, no TLS, and no authorization mechanism. Security relies entirely on network-level isolation — it is expected to run within the same Kubernetes namespace as the Guardrails Orchestrator and be accessible only via cluster-internal networking. There are no Kubernetes manifests, Helm charts, or kustomize overlays in this repository; deployment configuration is likely managed externally by the platform operator or the orchestrator's own deployment manifests.

Notably, this branch (rhoai-2.25) has no Konflux Dockerfile, which is unusual for an RHOAI downstream component. The existing Dockerfile uses a standard `rust:1.84.0` builder image from Docker Hub rather than a Brew/OSBS/Konflux-managed builder, and the runtime image tag defaults to `latest`. This suggests the component may be in an early integration stage for RHOAI or that Konflux build infrastructure is managed in a separate branch or repository. The Dockerfile does include test, lint, and format stages, providing build-time quality gates even without Konflux integration.

The response format wraps detections in a double-nested array (`[[...]]`), which is the format expected by the FMS Guardrails Orchestrator's detector protocol. Each detection includes character-level offsets (`start`, `end`), the matched text, a detection type classification (`pii` for built-in detectors, `custom` for user-supplied patterns), and a score of 1.0 (since regex matching is deterministic, there is no probabilistic scoring).
