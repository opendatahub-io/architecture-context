## Purpose

**Short**: A Kubernetes operator that automates the lifecycle management of model serving infrastructure for KServe InferenceServices, InferenceGraphs, LLMInferenceServices, and NVIDIA NIM accounts on Red Hat OpenShift AI.

**Detailed**: The odh-model-controller bridges the gap between KServe's core model serving capabilities and the RHOAI platform's requirements for enterprise-grade networking, security, observability, and multi-tenancy. When users deploy InferenceService resources, this controller automatically provisions the surrounding infrastructure: OpenShift Routes for external access, Istio service mesh integration (PeerAuthentication, Telemetry, Gateways, VirtualServices), Authorino AuthConfigs for request-level authentication, NetworkPolicies for namespace isolation, Prometheus ServiceMonitors and PodMonitors for observability, and KEDA TriggerAuthentications for autoscaling.

The controller supports three KServe deployment modes — Serverless (Knative-based with full mesh integration), RawDeployment (direct pod deployment with Routes), and ModelMesh (multi-model serving) — each with its own set of reconciled sub-resources. It also manages NVIDIA NIM (NVIDIA Inference Microservices) accounts, handling NGC API key validation, model catalog synchronization, pull secret generation, and ServingRuntime template creation. A comprehensive webhook system validates and mutates Pods (Ray TLS injection), InferenceServices, InferenceGraphs, Knative Services (mesh membership verification), and NIM Accounts (singleton enforcement).

The controller also integrates with Kuadrant AuthPolicies for LLMInferenceService resources, the Kubeflow Model Registry for model metadata tracking, and manages Ray TLS certificates for distributed multi-node inference workloads. It is deployed by the RHOAI platform operator (rhods-operator) as a single-replica Deployment in the operator namespace.

## Architectural Analysis

The odh-model-controller follows a **modular sub-reconciler architecture** that cleanly separates concerns across KServe's three deployment modes. The main `InferenceServiceReconciler` delegates to mode-specific composite reconcilers — `KserveServerlessInferenceServiceReconciler` (12 sub-reconcilers), `KserveRawInferenceServiceReconciler` (6 sub-reconcilers), and `ModelMeshInferenceServiceReconciler` (3 sub-reconcilers) — each of which composes a different set of `SubResourceReconciler` implementations. This design allows adding new reconciled resources without modifying the main controller, and the `GenericSubResourceReconciler[T]` interface enables the same reconciler patterns to be reused across `InferenceService` and `LLMInferenceService` types.

The delta processing pattern (`deltaProcessor.go`) is central to the reconciliation strategy: every sub-reconciler computes a desired state, fetches the existing state, and uses a type-specific comparator to determine whether a create, update, or delete is needed. This approach avoids unnecessary API server writes and produces idempotent reconciliation. The comparator system (`internal/controller/comparators/`) has 17 implementations, one per managed resource type, each checking only semantically significant fields (ignoring metadata like resourceVersion or UID). This is more precise than generic equality checks but requires maintenance when CRD schemas evolve.

The NVIDIA NIM integration (`internal/controller/nim/`) follows a handler pipeline pattern rather than the sub-reconciler pattern used for KServe. The `AccountReconciler` chains five handlers (Validation → ConfigMap → Template → PullSecret → Status) in sequence, where each handler can return `Continue`, `Requeue`, or `Error`. This pipeline design cleanly separates NGC API communication (validation, catalog fetching) from Kubernetes resource management (ConfigMap, Template, Secret creation). The singleton enforcement via validating webhook ensures only one NIM Account exists per namespace, simplifying the reconciler's invariants.

A notable security pattern is the **Ray TLS certificate management** for multi-node inference. The `ServingRuntimeReconciler` creates a self-signed CA (`ray-ca-tls`) in the controller namespace, while the `PodReconciler` generates per-pod certificates keyed by pod IP. The Pod webhook injects init containers that copy certificates from a Kubernetes Secret volume mount to an emptyDir, enabling Ray workers to establish mTLS connections. This design trades simplicity for security — the CA is long-lived (5 years) and not rotated, but individual pod certificates are created on-demand and cleaned up with pods. The retry loop in the init container (24 attempts × 10 seconds) handles the race condition where the pod gets an IP before the controller generates its certificate.

The controller's integration surface is exceptionally broad — it touches 15+ external API groups and creates resources across Istio, OpenShift, Kubernetes, Prometheus, KEDA, Authorino, Kuadrant, and Maistra APIs. This makes the controller a critical integration hub but also a potential failure point: if any of these external CRDs are unavailable, conditional setup logic in `cmd/main.go` skips those watchers (e.g., `AuthConfig`, `TriggerAuthentication`), but runtime failures in reconcilers are handled per-reconciler with error aggregation. The `MESH_DISABLED` environment variable provides a global escape hatch to skip all mesh-related reconciliation.
