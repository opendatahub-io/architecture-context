## Purpose

**Short**: Data Science Pipelines provides a platform for defining, deploying, and managing reproducible machine learning workflows on Kubernetes using Argo Workflows as the execution engine.

**Detailed**: Data Science Pipelines (DSP) is the RHOAI distribution of Kubeflow Pipelines v2. It provides a comprehensive ML pipeline orchestration platform that enables data scientists to define multi-step ML workflows as directed acyclic graphs (DAGs), schedule recurring executions, track experiment metadata, manage pipeline artifacts, and cache intermediate results for efficiency.

The system consists of five shippable backend components: an API server (serving both REST and gRPC endpoints for pipeline/run/experiment management), a persistence agent (syncing Kubernetes workflow state to a relational database), a scheduled workflow controller (managing cron and periodic pipeline triggers), a driver (orchestrating individual pipeline steps within Argo Workflows), and a launcher (executing user code within pipeline step containers). These components interact with external services including a relational database (MySQL or PostgreSQL), S3-compatible object storage (MinIO), ML Metadata (MLMD) for artifact lineage tracking, and the Argo Workflows execution engine.

The platform supports multi-user operation with Kubernetes-native RBAC, namespace-scoped isolation, and token-based authentication. Pipeline definitions can be stored either in the relational database or as Kubernetes CRDs (Pipeline/PipelineVersion). The system provides four CRDs: ScheduledWorkflow and Viewer (kubeflow.org/v1beta1), and Pipeline and PipelineVersion (pipelines.kubeflow.org/v2beta1).

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| API Server | Go Service | REST/gRPC API for pipeline, run, experiment, and artifact management |
| Persistence Agent | Go Service | Watches Argo Workflows and ScheduledWorkflows, persists state to database |
| Scheduled Workflow Controller | Go Controller | Manages cron/periodic pipeline execution via ScheduledWorkflow CRD |
| Driver | Go Binary (Argo step) | Orchestrates pipeline step execution: resolves inputs, checks cache, generates pod specs |
| Launcher | Go Binary (Argo step) | Executes user code within pipeline step containers, publishes results to MLMD |

## Architectural Analysis

Data Science Pipelines follows a **multi-binary, shared-codebase** architecture where five distinct binaries are built from the same Go module. The API server is the central hub — it serves REST (via gRPC-Gateway), gRPC, and webhook endpoints from a single process. The gRPC-Gateway pattern means all REST endpoints are auto-generated from protobuf service definitions, ensuring API consistency between HTTP and gRPC consumers. The API server uses an internal loopback gRPC connection (with `InsecureSkipVerify=true`) for the HTTP→gRPC translation, which is a standard gRPC-Gateway pattern but worth noting from a security audit perspective.

The **driver/launcher split** is architecturally notable. Rather than implementing a custom workflow engine, DSP delegates execution orchestration to Argo Workflows and injects its own driver and launcher binaries as step containers. The driver runs first to resolve inputs (from MLMD), evaluate CEL conditions, check the cache service, and generate pod spec patches. The launcher then wraps the user's actual code execution, capturing outputs and publishing them to MLMD and object storage. This design enables DSP to layer pipeline-specific semantics (caching, metadata tracking, artifact management) on top of Argo's proven workflow orchestration without reimplementing scheduling or container lifecycle management.

The **dual storage mode** for pipeline definitions (database vs Kubernetes CRDs) is a significant architectural decision. The CRD-based storage mode (`pipelinesStoreKubernetes=true`) stores Pipeline and PipelineVersion as native Kubernetes resources, enabling GitOps workflows and Kubernetes-native tooling. When this mode is enabled, the API server also runs admission webhooks to validate pipeline specs and add owner references. The database mode remains the default and is more scalable for large numbers of pipelines. The RHOAI operator (via DataSciencePipelinesApplication CR) typically manages which mode is used.

The **launcher's FIPS dual-binary** approach in `Dockerfile.konflux.launcher` is distinctive — it builds both a statically-linked non-FIPS binary (`CGO_ENABLED=0`, `/bin/launcher-v2`) and a dynamically-linked FIPS binary (`CGO_ENABLED=1 GOEXPERIMENT=strictfipsruntime`, `/bin/launcher-v2-fips`). The RHOAI operator selects which binary to use based on the cluster's FIPS mode. This pattern is unique among the DSP components — the other four binaries are always built with FIPS flags.

The **cache system** operates at two levels: the cache server (a separate deployment with its own database tables) handles cache-related admission webhooks for Argo Workflow pods, while the API server's TaskService provides gRPC endpoints for cache fingerprint lookup and recording. The driver generates SHA256 fingerprints from normalized task inputs/outputs/images/commands and queries the TaskService to find matching previous executions. This two-tier design separates the webhook admission path (mutating pods) from the data query path (fingerprint matching).
