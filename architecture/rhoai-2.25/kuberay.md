## Purpose

**Short**: Manages the lifecycle of Ray clusters, Ray jobs, and Ray services on Kubernetes, including support for OpenShift Routes and batch scheduling integrations.

**Detailed**: The KubeRay operator is a Kubernetes operator built with controller-runtime that provides three Custom Resource Definitions (CRDs) — RayCluster, RayJob, and RayService — to declaratively manage the Ray distributed computing framework on Kubernetes and OpenShift. The operator automates the provisioning of Ray head nodes, worker nodes, and associated Kubernetes resources (Services, Ingress, Routes, RBAC), enabling data scientists and ML engineers to run distributed training, batch inference, and online serving workloads without managing Ray infrastructure directly.

The operator integrates with the broader RHOAI platform as a component deployed by the platform operator (rhods-operator/opendatahub-operator). On OpenShift, it creates Routes for Ray dashboard exposure and applies SecurityContextConstraints for Ray pod security. The operator supports pluggable batch schedulers (Volcano, YuniKorn) for gang scheduling of distributed training jobs, GCS fault tolerance via Redis for cluster state persistence, zero-downtime upgrades for RayService via blue-green cluster promotion, and Kueue integration for workload queuing. The RHOAI build uses FIPS-compliant Go compilation flags (GOEXPERIMENT=strictfipsruntime, CGO_ENABLED=1) and ships on UBI9-minimal base images.

## Architectural Analysis

The KubeRay operator follows a clean three-controller architecture, with each controller (RayCluster, RayJob, RayService) managing a single CRD type while sharing common resource-building utilities. The RayCluster controller is the foundational layer: it manages the full lifecycle of Ray head and worker pods, services, ingress/routes, and RBAC. RayJob and RayService are higher-level abstractions that create and manage RayCluster resources internally, adding job submission and zero-downtime upgrade semantics respectively. This layered design means RayJob and RayService delegate all cluster-level concerns to the RayCluster controller via owned resource watches.

The operator exhibits a notable dual-platform pattern for network exposure. On OpenShift (detected at startup), the operator creates OpenShift Routes for Ray dashboard access; on vanilla Kubernetes, it creates standard Ingress resources. The `USE_INGRESS_ON_OPENSHIFT` environment variable allows overriding this behavior. The head service defaults to a Headless service (`ClusterIP=None`) since a RayCluster should have at most one head pod, though this can be changed to a ClusterIP service via `ENABLE_RAY_HEAD_CLUSTER_IP_SERVICE`. The serve service is conditionally created based on annotations (for RayCluster) or always created (for RayService), and selects only pods with the `ray.io/serve=true` label — which is set by the RayService controller after verifying serve proxy health via HTTP.

A significant design consideration is the operator's integration with Ray's dashboard HTTP API for runtime operations. The RayJob controller submits jobs and polls status via the dashboard API, and the RayService controller uses it to deploy serve applications. This creates a runtime dependency on the Ray head pod being healthy with dashboard port 8265 accessible — the operator is not purely declarative but also acts as an orchestrator that makes HTTP calls to the managed workload. The reconcile loops are aggressive: RayJob requeues every 3 seconds, RayService every 2 seconds, and RayCluster every 300 seconds (configurable). The batch scheduler integration is cleanly abstracted through a pluggable interface (`BatchSchedulerInterface`) with implementations for Volcano, YuniKorn, and a default no-op, allowing the operator to work with various scheduling backends without code changes.

The RHOAI-specific configuration in `config/openshift` adds a SecurityContextConstraints resource (`run-as-ray-user`) that enforces UID 1000 for operator pods, drops all capabilities, and requires the `runtime/default` seccomp profile. The OpenShift overlay also replaces the upstream image with the ODH-branded image and sets the deployment namespace to `opendatahub`. The webhook configuration is opt-in (controlled by `ENABLE_WEBHOOKS` env var and requires the `default-with-webhooks` overlay) and currently only performs basic validation on RayCluster resources — name format regex and worker group name uniqueness. The feature gate system supports gradual rollout of new features: `RayClusterStatusConditions` is Beta (default enabled) and `RayJobDeletionPolicy` is Alpha (default disabled).
