## Purpose
**Short**: KubeRay Operator manages the lifecycle of Ray clusters, jobs, and services on Kubernetes/OpenShift.

**Detailed**: KubeRay is a Kubernetes operator that simplifies the deployment and management of Ray applications on Kubernetes. It provides three Custom Resource Definitions (CRDs) -- RayCluster, RayJob, and RayService -- that allow users to declaratively manage Ray cluster infrastructure, submit batch jobs to Ray clusters, and deploy Ray Serve applications with zero-downtime upgrades.

The operator is built using the controller-runtime framework and implements reconciliation loops for each CRD. On OpenShift, it automatically detects the platform via API discovery and creates OpenShift Routes (instead of Kubernetes Ingresses) for exposing the Ray dashboard. It supports batch scheduling integrations with Volcano and Yunikorn, GCS fault tolerance with Redis cleanup, autoscaler service account management, and Prometheus metrics emission. The RHOAI build uses FIPS-compliant compilation (`GOEXPERIMENT=strictfipsruntime`) and runs on UBI9 base images.

The operator dynamically creates Pods (head and worker), Services (head service, serve service, headless worker service), Ingresses or Routes (for dashboard exposure), ServiceAccounts, Roles, RoleBindings (for autoscaler RBAC), and batch Jobs (for Redis cleanup during GCS fault tolerance teardown).

**Note**: On OpenShift (RHOAI), the operator auto-detects the platform via API discovery (`route.openshift.io` API group) and creates OpenShift Routes instead of Kubernetes Ingresses for the Ray dashboard. Ingress creation is opt-in via `spec.headGroupSpec.enableIngress: true` on the RayCluster CR. The `USE_INGRESS_ON_OPENSHIFT` environment variable can override this behavior to use Kubernetes Ingresses on OpenShift.
