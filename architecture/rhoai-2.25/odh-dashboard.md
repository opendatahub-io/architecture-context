## Purpose

**Short**: Web dashboard providing the primary user interface for Red Hat OpenShift AI (RHOAI), enabling data scientists and administrators to manage notebooks, model serving, pipelines, model registries, and platform settings.

**Detailed**: The odh-dashboard is a monorepo application consisting of a React/TypeScript frontend, a Node.js/Fastify backend (serving as an API gateway and Kubernetes proxy), and multiple federated plugin modules with Go Backend-for-Frontend (BFF) services. The frontend uses PatternFly 6 components and the OpenShift Dynamic Plugin SDK to provide a plugin-extensible UI. The backend proxies all Kubernetes API calls, service-to-service communication, and WebSocket connections, enforcing authentication and authorization at each layer.

The application employs Webpack Module Federation to dynamically load plugin modules (currently model-registry and gen-ai) at runtime without redeploying the host application. Each federated module ships as an independent container image containing both a React frontend and a Go BFF service. The BFF services run as sidecar containers in the same pod as the main dashboard, communicating with upstream platform services (Model Registry API, LlamaStack, MCP servers) while the frontend modules are loaded via the Module Federation runtime. The dashboard manages a wide range of platform resources including notebooks, accelerator profiles, hardware profiles, serving runtimes, connection types, model registries, and platform-level configuration through the OdhDashboardConfig CRD.

## Architectural Analysis

The odh-dashboard employs a layered architecture that cleanly separates concerns across the browser, API gateway, and Kubernetes API layers. The Node.js backend acts as a critical security boundary: it enforces authentication via OpenShift OAuth (through the oauth-proxy sidecar), performs authorization checks (`secureRoute`/`secureAdminRoute` wrappers), and proxies all upstream service calls with proper token forwarding. No direct browser-to-Kubernetes communication is possible; all K8s API calls are mediated through the `/api/k8s/*` proxy, and all service-to-service calls go through `/api/service/*` proxy routes.

The Module Federation architecture is a significant design decision that enables independent deployment of feature modules (model-registry, gen-ai) without redeploying the host dashboard. Each federated module ships as a sidecar container in the same pod, running a Go BFF service that handles upstream API calls while the React frontend is loaded dynamically via Webpack Module Federation runtime. The `federation-config` ConfigMap controls which remotes are available, and the host application discovers and loads them at startup. This pattern enables the platform to add new features (like gen-ai) without modifying the core dashboard codebase. The current deployment includes the model-registry-ui sidecar; the gen-ai module exists as source code but does not yet have its own Konflux Dockerfile for production deployment.

The authentication flow is multi-layered: the oauth-proxy sidecar handles OpenShift OAuth, passes the access token via `x-forwarded-access-token` header to the Node.js backend, which then either uses it directly for K8s API calls or forwards it to BFF sidecars. The BFF sidecars perform their own authorization via Kubernetes SubjectAccessReview (SAR/SSAR) checks before proxying to upstream services. This layered auth approach ensures defense-in-depth but introduces complexity: each BFF must independently validate access, and token propagation through the proxy chain must be carefully maintained. The WebSocket authentication uses an unusual but effective pattern where the bearer token is base64url-encoded in the WebSocket subprotocol header.

A notable security observation is the gen-ai BFF's hardcoded `InsecureSkipVerify=true` on its outbound HTTP transport (`packages/gen-ai/bff/internal/integrations/http.go`), which disables TLS certificate verification for all outbound HTTP calls. While the gen-ai BFF is not yet deployed via Konflux, this should be addressed before production deployment. The model-registry BFF handles this correctly with a configurable flag defaulting to `false` and enforcing TLS 1.3 minimum. The main dashboard backend also loads CA bundles for proper certificate verification.

The RBAC footprint is extensive: the dashboard ClusterRole grants broad read access across the platform (nodes, machine autoscalers, cluster versions, CSV subscriptions, users, groups, pods, services, events) and write access to several critical resources (secrets, configmaps, PVCs, notebooks, model registries, RBAC bindings). Several RoleBindings grant `system:authenticated` users read access to accelerator profiles, hardware profiles, NIM accounts, and serving runtime templates, which enables the dashboard to display these resources to any authenticated user. The `auth-delegator` ClusterRoleBinding allows the dashboard service account to perform token review, supporting the backend's user validation logic.
