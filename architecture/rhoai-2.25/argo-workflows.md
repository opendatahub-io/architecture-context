## Purpose

**Short**: Argo Workflows is a Kubernetes-native workflow engine that orchestrates parallel jobs as DAGs and step-based workflows, used as the execution backend for Data Science Pipelines in RHOAI.

**Detailed**: Argo Workflows provides the core workflow execution engine for Red Hat OpenShift AI's Data Science Pipelines component. It defines Custom Resource Definitions (CRDs) for workflow specifications (Workflow, WorkflowTemplate, ClusterWorkflowTemplate, CronWorkflow) and manages their lifecycle through a controller that creates and monitors Kubernetes Pods for each workflow step.

The RHOAI distribution ships two container images from this repository: the **workflow-controller** (which reconciles Workflow CRDs and creates execution Pods) and **argoexec** (which runs as a sidecar/init container within workflow Pods to manage artifact upload/download, log capture, and step execution). The Argo Server (API/UI component) and the `argo` CLI are present in the upstream source but are NOT shipped as separate Konflux images in RHOAI — they are consumed indirectly through the Data Science Pipelines operator.

The workflow controller uses a leader-elected, multi-worker architecture with configurable parallelism, pod cleanup, TTL-based garbage collection, and optional persistence to SQL databases for workflow archival and large workflow node status offloading. Artifact storage integrates with S3-compatible stores (MinIO, AWS S3, GCS) for pipeline artifact management.

## Architectural Analysis

Argo Workflows in RHOAI functions as a dedicated workflow execution engine embedded within the Data Science Pipelines stack rather than as a standalone platform component. The RHOAI distribution intentionally ships only two of the three upstream components — the workflow-controller and argoexec — omitting the argo-server (API/UI). This means the controller operates headlessly, receiving workflow CRs created by the DSP API server rather than through Argo's own REST/gRPC API. The Argo Server code remains in the repository and is analyzed here for completeness (it exposes port 2746/TCP with gRPC and HTTP connection multiplexing via cmux), but it should not be considered part of the RHOAI deployment topology.

The controller's reconciliation architecture is designed for high throughput: 32 parallel workflow workers by default, with separate worker pools for TTL expiry (4), pod cleanup (4), and CronWorkflow scheduling (8). Each workflow reconciliation is bounded to 30 seconds (configurable via MAX_OPERATION_TIME), and the controller uses a rate-limited work queue with a custom fixed-interval rate limiter. Leader election via the Kubernetes Lease API ensures only one controller instance reconciles at a time, with non-leaders running a "dummy" metrics server for Prometheus scraping continuity. The health check at `/healthz` on port 6060 monitors reconciliation freshness — if any workflow remains unreconciled for more than 5 minutes (configurable via HEALTHZ_AGE), the controller reports unhealthy.

A noteworthy FIPS consideration exists in the argoexec image: the Konflux Dockerfile builds **two separate binaries** — `argoexec-fips` (dynamically linked with `CGO_ENABLED=1` and `GOEXPERIMENT=strictfipsruntime`) and `argoexec` (statically linked with `CGO_ENABLED=0`, non-FIPS). The container ENTRYPOINT defaults to the non-FIPS `/bin/argoexec`. For FIPS compliance in RHOAI, the DSP operator or controller configuration must ensure the FIPS binary is invoked. The workflow-controller image, by contrast, only builds the FIPS variant. This asymmetry is architecturally significant — the argoexec binary runs in customer workflow Pods and is the most widely deployed artifact from this repository.

The controller dynamically creates several Kubernetes resource types at runtime beyond Pods: PersistentVolumeClaims for artifact and data exchange between workflow steps, PodDisruptionBudgets to protect running workflows from voluntary disruptions, ConfigMaps for memoization caching of step results, and Leases for leader election. The artifact storage subsystem is highly pluggable, with support for S3 (including MinIO), GCS, Azure Blob, Alibaba OSS, HDFS, and Artifactory — though in RHOAI the DSP operator typically configures MinIO as the default artifact backend. Optional SQL persistence (PostgreSQL or MySQL) enables workflow archival and node status offloading for workflows with very large step counts (100k+ nodes).

The security model relies entirely on Kubernetes RBAC — the controller runs under the `argo` ServiceAccount with broad permissions (create/delete Pods, PVCs, PDBs, and all Argo CRDs), while workflow Pods run under user-specified ServiceAccounts. The controller's access to Secrets is narrowly scoped to a single named secret (`argo-workflows-agent-ca-certificates`). Network-level security is minimal: no NetworkPolicies are included in the manifests, and the controller's metrics (9090/TCP) and health (6060/TCP) endpoints are unauthenticated HTTP, relying on network isolation at the namespace/cluster level for protection.
