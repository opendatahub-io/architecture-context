## Purpose
**Short**: An HTTP gateway that provides multiple OpenAI-compatible chat completion endpoints with configurable guardrail detector pipelines via the FMS Guardrails Orchestrator.

**Detailed**: The vllm-orchestrator-gateway is a Rust-based HTTP service that acts as a routing and filtering layer between clients and the FMS Guardrails Orchestrator. It enables controlled interaction with the orchestrator by exposing multiple distinct `/v1/chat/completions` endpoints, each configured with a specific set of content safety detectors (e.g., PII detection, content filtering).

When a client sends a standard OpenAI-compatible chat completion request, the gateway intercepts it, injects the appropriate detector configurations based on the route, and forwards the enriched request to the orchestrator's `/api/v2/chat/completions-detection` endpoint. If any detectors flag content in the orchestrator's response, the gateway can substitute a configurable fallback message. This allows organizations to deploy guardrailed LLM endpoints as drop-in replacements for unguardrailed ones, with route-specific safety policies.

The gateway supports mTLS communication with the orchestrator via certificate-based identity, and forwards authorization headers from incoming requests to the backend orchestrator service.

### TLS Configuration

The gateway supports optional mTLS for orchestrator communication:

| Path | Type | Purpose |
|------|------|---------|
| `/etc/tls/private/tls.crt` | Client Certificate | mTLS client identity certificate |
| `/etc/tls/private/tls.key` | Client Private Key | mTLS client identity key |
| `/etc/tls/ca/service-ca.crt` | CA Certificate | Custom CA for verifying orchestrator's server certificate |

- When TLS cert and key are present, the client uses HTTPS with mTLS
- When only the CA cert is present, the client adds it as a trusted root but uses HTTP
- When no TLS files exist, the client uses plain HTTP
- For `localhost` orchestrator hosts, hostname verification is relaxed
