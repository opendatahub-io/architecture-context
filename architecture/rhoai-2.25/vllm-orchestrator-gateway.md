## Purpose

**Short**: An OpenAI-compatible HTTP gateway that routes chat completion requests through configurable detector-based content filtering pipelines backed by the FMS Guardrails Orchestrator.

**Detailed**: The vLLM Orchestrator Gateway is a lightweight Rust HTTP service that sits between clients and the FMS Guardrails Orchestrator, providing controlled access to guardrailed LLM inference. It reads a YAML configuration defining detector pipelines (e.g., PII regex detection) and dynamically exposes per-route `/v1/chat/completions` endpoints. Each route maps to a specific combination of input/output detectors with configurable fallback messages.

When a client sends an OpenAI-compatible chat completion request to a route endpoint (e.g., `/{route_name}/v1/chat/completions`), the gateway injects the route's detector configuration into the payload and forwards it to the FMS Guardrails Orchestrator's `/api/v2/chat/completions-detection` endpoint. If the orchestrator detects unsuitable content (based on the route's detectors), the gateway replaces the response with a configured fallback message. The gateway supports mTLS for secure communication with the orchestrator when TLS certificates are mounted, and forwards `Authorization` and `X-Forwarded-*` headers from clients.

This component is part of the TrustyAI guardrails stack in RHOAI, enabling platform administrators to define content safety policies as named routes, each with different detector combinations and fallback behaviors, while presenting a standard OpenAI-compatible API to consumers.

## Architectural Analysis

The vllm-orchestrator-gateway follows a straightforward reverse-proxy gateway pattern implemented in Rust using the axum framework. Its primary architectural contribution is the config-driven route multiplication: a single YAML config file generates N OpenAI-compatible endpoints, each bound to a different detector pipeline. This turns the complex orchestrator API (which requires detector specifications in each request) into a simple drop-in replacement for any OpenAI-compatible client — consumers just point at the appropriate route endpoint and get guardrailed inference without needing to know about detectors.

The mTLS implementation (`build_orchestrator_client` in `src/main.rs:201-256`) demonstrates thoughtful certificate handling: it reads PEM-formatted certs and keys, converts them to PKCS#12 via the `openssl` crate (necessary because `reqwest`'s `native-tls` backend requires PKCS#12 for client identity), and conditionally enables HTTPS only when certificates are present. The fallback to plaintext HTTP when no certs are found makes local development straightforward. The `danger_accept_invalid_hostnames(true)` call when connecting to `localhost` is a pragmatic concession for local testing where the orchestrator's certificate won't match `localhost`.

A notable design choice is the header forwarding strategy (`src/main.rs:273-283`): only `Authorization` and `X-Forwarded-*` headers are forwarded to the orchestrator. This is a security-positive pattern that prevents leaking arbitrary client headers to the backend while ensuring authentication tokens and proxy chain information propagate correctly. However, the gateway itself performs no authentication — it fully delegates auth enforcement to the orchestrator or to whatever ingress layer sits in front of it.

The config validation (`config.rs:39-56`) panics at startup if a route references a detector name not defined in the config's `detectors` list. This fail-fast approach prevents runtime errors from misconfiguration but means the gateway process will crash on bad config rather than serving partial functionality. There is no runtime config reload capability — config changes require a container restart.
