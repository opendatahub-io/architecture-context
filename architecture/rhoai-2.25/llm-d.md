## Purpose

**Short**: llm-d is a Kubernetes-native distributed inference serving stack that provides optimized container images, deployment guides, and Helm/Kustomize recipes for production LLM serving across multiple hardware accelerators.

**Detailed**: llm-d is a CNCF sandbox project founded by Red Hat, Google Cloud, IBM Research, CoreWeave, and NVIDIA. It provides the fastest "time to state-of-the-art (SOTA) performance" for large language models across most hardware accelerators with well-tested guides and real-world benchmarks.

The repository itself does not contain application source code (no Go operators, no Python services). Instead, it serves as the **integration and build layer** for the llm-d inference stack: it builds optimized container images packaging vLLM with high-performance networking libraries (NIXL, UCX, NVSHMEM, GDRCopy), GPU compute kernels (FlashInfer, DeepEP, DeepGEMM), and KV cache management (LMCache, Infinistore). The repository also provides comprehensive deployment recipes via Helm charts and Kustomize overlays that compose a complete inference serving topology: Gateway API ingress, llm-d Router (Endpoint Picker), and model server replicas.

The deployment patterns range from a simple "optimized baseline" with prefix-cache-aware routing to advanced configurations including prefill/decode disaggregation with RDMA KV transfer, wide expert-parallelism for Mixture-of-Experts models (e.g., DeepSeek-R1), tiered KV cache offloading to CPU/SSD/remote storage, and workload-aware autoscaling. All patterns deploy on Kubernetes using the Gateway API Inference Extension as the control plane.

## Architectural Analysis

The llm-d repository occupies a unique position in the RHOAI ecosystem: it is neither a traditional operator nor a service, but rather an **integration layer** that assembles a complete inference serving stack from upstream components. The repository's primary artifacts are container images and deployment recipes, not application code. This design reflects the project's principle of "respecting upstreams" — vLLM, Kubernetes Gateway API, and NIXL are where code changes start, and llm-d composes them into production-ready configurations.

The container image build system is notably sophisticated. The CUDA Dockerfile (`docker/Dockerfile.cuda`) uses a multi-stage build with over 20 build arguments, supporting multiple OS bases (RHEL UBI 9, Ubuntu), multiple architectures (x86_64, aarch64), and multiple CUDA generations (12.x, 13.x). The build pipeline compiles C/C++ networking libraries (UCX, GDRCopy) from source, builds GPU compute kernels (FlashInfer, DeepEP, DeepGEMM) as Python wheels, and installs vLLM in one of three modes: full editable install (for development), precompiled binaries (faster builds), or production wheels. A notable pattern is the GB200/Blackwell build target (`runtime-gb200`) which swaps in architecture-specific DeepEP and NVSHMEM variants — this shows the team actively supporting bleeding-edge hardware without forking the main build.

The deployment architecture uses a layered Kustomize composition pattern. Base recipes in `guides/recipes/` define the fundamental resources (Deployment, ServiceAccount, PodMonitor), while guide-specific overlays in `guides/optimized-baseline/`, `guides/pd-disaggregation/`, etc. patch in model-specific, hardware-specific, and topology-specific configurations. The Helm chart for the Router (EPP) is consumed from the separate `llm-d-inference-scheduler` repository. This separation of concerns means a deployment requires coordinating three independently-versioned components: the container image (this repo), the Router Helm chart (llm-d-inference-scheduler), and the Gateway controller (Envoy/Istio/AgentGateway).

A significant architectural consideration for RHOAI integration is that llm-d currently has **no Konflux Dockerfiles** on the main branch. The images are built via GitHub Actions and pushed to ghcr.io. For RHOAI productization, Konflux Dockerfiles with hermetic builds (rpms.lock.yaml, cachi2/Hermeto prefetch) would need to be added on downstream release branches. The current `docker/common-versions` file provides partial reproducibility by pinning vLLM and dependency versions to specific git commits, but the build is not fully hermetic — it fetches dependencies from the internet at build time.

The networking architecture is designed for high-performance GPU-to-GPU communication. The NIXL library, compiled into every GPU container image, supports multiple transport backends: UCX (InfiniBand, RoCE), UCCL (InfiniBand, RoCE, TCP-X, EFA), and libfabric (EFA). This multi-backend support enables the same container image to run on diverse infrastructure — from on-premises InfiniBand clusters to AWS EFA-equipped instances. The EFA support is conditionally compiled (`ENABLE_EFA=true` build arg) and currently limited to RHEL-based builds. For RHOAI deployments on OpenShift, this RDMA capability is critical for prefill/decode disaggregation where KV cache tensors are transferred directly between GPU memories without CPU involvement.
