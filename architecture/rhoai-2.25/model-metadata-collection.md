## Purpose

**Short**: A Go CLI tool that extracts, enriches, and catalogs AI/ML model metadata from OCI container registries and HuggingFace, packaging the results into a data-only container image for consumption by the RHOAI platform.

**Detailed**: The Model Metadata Collection component is a build-time data pipeline that processes Red Hat AI validated model definitions from an index file (`data/models-index.yaml`), pulls OCI container images from `registry.redhat.io`, extracts model card metadata from image layers, and enriches that metadata with information from the HuggingFace API. The enriched metadata is compiled into a unified catalog (`data/models-catalog.yaml`) that describes all validated models with their names, descriptions, licenses, languages, tasks, timestamps, and OCI artifact references.

The component does not run as a service in the RHOAI cluster. Instead, it runs during CI/CD to produce pre-generated catalog files that are packaged into a data-only container image (`odh-model-metadata-collection-rhel9`). This container uses `ubi9-micro` as its base, copies the catalog files in, and runs `sleep infinity` so other platform components can mount or copy the catalog data at runtime. The container serves as a sidecar data volume providing model metadata to the RHOAI dashboard and model registry components.

The metadata enrichment pipeline follows a priority hierarchy: HuggingFace YAML frontmatter > HuggingFace API > modelcard YAML > modelcard regex extraction > generated defaults. This ensures the highest-quality metadata source is preferred while still providing fallback values for models with incomplete upstream documentation.

## Architectural Analysis

### Data-Only Container Pattern

This component uses an unconventional deployment pattern: it ships a container image that contains no executable application code. The `Dockerfile.konflux` copies two pre-generated YAML files into a `ubi9-micro` image and sets `CMD ["sleep", "infinity"]`. The container exists solely to make catalog data available to other pods via volume mounts or init container copies. This pattern decouples data generation (CI/CD time) from data consumption (runtime), allowing catalog updates without redeploying consuming components.

### Metadata Enrichment Priority Hierarchy

The enrichment pipeline implements a carefully designed priority system across five data sources:
1. **HuggingFace YAML frontmatter** (highest priority) - Structured metadata from HuggingFace model card YAML headers
2. **HuggingFace API** - Model details from the HuggingFace REST API (downloads, likes, tags)
3. **Modelcard YAML** - Structured metadata embedded in OCI image layers (`io.opendatahub.modelcar.layer.type=modelcard`)
4. **Modelcard regex** - Regex-based extraction from unstructured model card text (provider names, dates, descriptions)
5. **Generated defaults** (lowest priority) - Algorithmically generated values from model names

Each metadata field tracks its source via `enrichment.yaml` files, enabling the `metadata-report` tool to audit data provenance.

### Family-Aware Model Matching

The enrichment system includes a family-aware matching algorithm (`isCompatibleModelFamily`) that prevents cross-family contamination when matching OCI registry models to HuggingFace models. For example, it prevents a `modelcar-llama-3-3-70b-instruct` container from being matched to a Granite HuggingFace model, even if token-level similarity scores might suggest a match. This is critical for maintaining metadata accuracy across the 45+ models in the index.

### Concurrent Processing with Bounded Parallelism

The model-extractor processes OCI images concurrently using goroutines with a semaphore pattern (configurable via `--max-concurrent`, default 5). This balances throughput against registry rate limits and local resource consumption. Results are collected via a channel and sorted deterministically for reproducible catalog output.

### Static Catalog Merging

The pipeline supports merging pre-defined static catalog entries (`input/supplemental-catalog.yaml`) with dynamically extracted metadata. This handles legacy models (e.g., `granite-7b-redhat-lab`, `prometheus-8x7b-v2-0`) that may not have OCI images in the current registry but still need catalog representation. Deduplication by model name ensures no duplicates in the final catalog.

### Metadata Format Migration

The codebase includes migration logic (`internal/metadata/migration.go`) that handles evolution of the metadata schema. It converts legacy string-based artifact lists to structured `OCIArtifact` objects with timestamps and custom properties, and normalizes timestamp formats across different metadata sources.

### Multi-Architecture Build

The Konflux pipeline builds multi-architecture images for x86_64, ppc64le, s390x, and arm64. Since the container contains only YAML data files (no compiled binaries), the multi-architecture build ensures the data container can be pulled on any supported platform, matching RHOAI's multi-architecture support requirements.
