## Purpose

**Short**: A central repository for storing and managing ML model metadata, versions, and artifacts with a REST API interface.

**Detailed**: Model Registry provides a metadata management service for machine learning workflows on the Red Hat OpenShift AI platform. It stores and retrieves metadata associated with ML developer and data scientist workflows — including registered models, model versions, model artifacts, serving environments, inference services, experiments, and experiment runs. The service exposes a RESTful OpenAPI v1alpha3 interface for CRUD operations on these entities, backed by either MySQL or PostgreSQL databases through an embedded metadata (EmbedMD) storage layer with GORM-based ORM and golang-migrate schema management.

The repository produces multiple deployable components: (1) the core Model Registry REST API server that serves the OpenAPI-generated endpoints, (2) a Model Catalog service providing a read-only interface to browse model catalogs from YAML files and HuggingFace Hub, (3) a Kubernetes controller that reconciles KServe InferenceService CRDs with Model Registry metadata, (4) a CSI storage initializer that downloads model artifacts from the registry using the `model-registry://` protocol scheme, and (5) an async-upload Python job that copies models between storage backends (S3, OCI registries, HuggingFace) and registers them in the registry.

## Architectural Analysis

The Model Registry follows a modular, multi-binary architecture where a single Go module produces four distinct deployment artifacts — the REST API server, a catalog service, a Kubernetes controller, and a CSI storage initializer — plus a separate Python-based async job. This design allows each component to scale and deploy independently while sharing core library code (OpenAPI models, converters). The REST API server and catalog service share a binary (invoked via different Cobra subcommands), while the controller and CSI initializer have their own `cmd/` entry points with dedicated Dockerfiles.

The data model is rooted in the MLMD (ML Metadata) schema, adapted through a Go-native EmbedMD layer that replaces the upstream C++ gRPC MLMD service with direct database access via GORM. This eliminates the gRPC dependency entirely — the API server talks directly to MySQL or PostgreSQL. The schema is managed through embedded SQL migrations (20+ migration files per database type) using golang-migrate, and entity types are seeded during startup. The database layer uses a connector/migrator factory pattern (`internal/db/`) that abstracts database-specific connection details while supporting TLS for both MySQL and PostgreSQL with configurable cipher suites.

The InferenceService controller implements a reconciliation pattern that bridges the Kubernetes-native KServe world with Model Registry metadata. When a KServe InferenceService is annotated with model registry labels, the controller creates corresponding ServingEnvironment and InferenceService records in the registry, maintaining bidirectional synchronization. The controller discovers Model Registry endpoints via Kubernetes Service lookups and uses bearer token authentication. A notable architectural decision is the use of finalizers for cleanup — when an InferenceService is deleted, the controller marks the registry record as UNDEPLOYED before removing the finalizer. HTTP/2 is disabled by default in the controller to mitigate known CVEs (GHSA-qppj-fm5r-hxr3, GHSA-4374-p667-p6c8).

The async upload job introduces a Python layer for model synchronization between heterogeneous storage backends. Its three-intent architecture (create_model, create_version, update_artifact) covers the full model lifecycle — from initial model registration through version updates. The job uses skopeo for OCI registry operations and the model-registry Python client for API interactions. The Pydantic-based configuration system supports both CLI arguments and environment variables (with `MODEL_SYNC_` prefix), making it flexible for both interactive use and Kubernetes Job specifications. The RHOAI Konflux build uses cachi2 for hermetic dependency prefetch, ensuring reproducible builds.

Security-wise, the application itself does not implement authentication — it relies on the platform's Istio service mesh for mutual TLS and network-level access control. The Istio AuthorizationPolicy currently allows all traffic to model-registry pods, which is appropriate for an internal service where authorization is handled upstream (by the Dashboard or other clients). The controller's metrics endpoint is the exception, using Kubernetes TokenReview and SubjectAccessReview for authentication. The main FIPS concern is that only the Konflux-built REST API server image uses FIPS-compliant build flags; the controller and CSI Dockerfiles in this branch use CGO_ENABLED=0, which would not pass check-payload validation. These components likely have separate Konflux build configurations on their respective downstream branches.
