## Purpose

**Short**: Python SDK for managing Ray clusters and Ray jobs on Kubernetes/OpenShift, providing a simplified interface for data scientists to create, configure, and interact with distributed compute resources on Red Hat OpenShift AI.

**Detailed**: The CodeFlare SDK is a Python client library that enables data scientists to programmatically manage Ray clusters and submit Ray jobs on Kubernetes and OpenShift clusters. It abstracts the complexity of Kubernetes Custom Resources (RayCluster, RayJob) behind a Pythonic API, handling cluster lifecycle management (create, apply, status, delete), TLS certificate generation for mTLS connections, Kueue queue integration for resource management, and Jupyter notebook widget-based cluster management.

The SDK operates as a client-side library — it does not deploy any server-side components. Instead, it communicates with the Kubernetes API server to create and manage RayCluster and RayJob custom resources, which are then reconciled by the KubeRay operator running on the cluster. The SDK supports two primary workflows: interactive cluster management (create a RayCluster, wait for it to be ready, submit jobs via the Ray Job Submission Client) and declarative job management (submit a RayJob CR that optionally creates its own ephemeral cluster). It integrates with RHOAI platform features including Gateway API-based HTTPRoute ingress (RHOAI v3.0+), OpenShift Route-based ingress (pre-v3.0), Kueue workload queuing, and kube-authkit for authentication.

The SDK vendors the KubeRay Python client for RayJob API operations and includes Jupyter notebook widgets (ipywidgets) for interactive cluster management within notebook environments. It supports GPU/accelerator scheduling across NVIDIA, Intel, AMD, AWS Neuron, Google TPU, Habana Gaudi, and Huawei Ascend devices.

## Architectural Analysis

The CodeFlare SDK follows a client-side abstraction pattern where the SDK generates Kubernetes Custom Resource YAML and submits it to the API server, relying on server-side operators (KubeRay, Kueue) for reconciliation. This design means the SDK has zero server-side footprint — it is purely a library consumed by data scientists in Jupyter notebooks or Python scripts. The `Cluster` class uses server-side apply (`api_instance.server_side_apply`) with a field manager `codeflare-sdk`, enabling safe concurrent modifications without conflicting with the operator's field ownership.

The SDK exhibits a clear migration pattern from RHOAI v2.x to v3.0+. Dashboard URL discovery (`cluster_dashboard_uri`, `_get_dashboard_url_from_httproute`) implements a cascading fallback: first try HTTPRoute (Gateway API for v3.0+), then fall back to OpenShift Routes (v2.x), then fall back to vanilla Kubernetes Ingresses (Kind/non-OpenShift). This allows a single SDK version to work across all platform versions and deployment targets. The HTTPRoute discovery uses label selectors (`ray.io/cluster-name`, `ray.io/cluster-namespace`) and traverses the Gateway reference chain to extract the hostname, constructing URLs in the format `https://{hostname}/ray/{namespace}/{cluster-name}`.

The TLS certificate management is architecturally notable. The SDK generates client-side mTLS certificates signed by the CA secret created by KubeRay when the Ray head pod starts. Certificates are stored in `~/.local/share/codeflare/tls/{cluster}-{namespace}/` with 0600 permissions, following XDG Base Directory conventions. The certificates include SAN entries for both localhost addresses and the Kubernetes service DNS names (`{cluster}-head-svc.{namespace}.svc.cluster.local`), enabling connections from both inside and outside the cluster. The `apply()` method polls for the CA secret before generating certificates, with a configurable timeout. Certificate lifecycle management includes refresh, cleanup, expiration detection, and automatic cleanup on cluster deletion.

The RayJob workflow introduces a second architectural path that wraps the vendored KubeRay Python client. The `RayJob` class builds a RayJob CR with either a `clusterSelector` (for existing clusters) or an embedded `rayClusterSpec` (for ephemeral clusters). File handling for job submission uses a creative pattern: local Python files or working directories are zipped, base64-encoded, stored in a Kubernetes Secret with an owner reference to the RayJob (for automatic cleanup), and mounted into the submitter pod. An init container unzips the working directory before the Ray job submitter runs. This avoids the need for external storage (S3, GCS) for simple job submissions.

The vendored KubeRay Python client (`src/codeflare_sdk/vendored/python_client/`) is a fork of the upstream kuberay Python client, maintained separately to allow SDK-specific modifications. This vendoring approach provides stability but creates a maintenance burden — changes to the upstream KubeRay API require manual synchronization. The vendored client uses a separate Kubernetes API client initialization path from the main SDK (it calls `config.load_kube_config()` directly rather than using the SDK's `config_check()` / `get_api_client()` pattern), which could cause authentication inconsistencies in environments where the SDK's kube-authkit integration provides the API client.
