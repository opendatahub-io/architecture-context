## Purpose

**Short**: Manages the lifecycle of Jupyter Notebook workspaces (workbenches) on OpenShift, including pod orchestration, OAuth authentication, network isolation, and idle resource reclamation.

**Detailed**: This repository is a downstream fork of the upstream Kubeflow `kubeflow/kubeflow` repository, scoped to the `components/` directory which contains notebook controller logic. It produces two cooperating Go operators that together manage notebook workbenches in RHOAI.

The first operator, **kf-notebook-controller** (upstream Kubeflow notebook-controller), is the core lifecycle manager. It watches `Notebook` CRs (kubeflow.org/v1) and reconciles a StatefulSet (replicas=1), a ClusterIP Service, and optionally an Istio VirtualService for each notebook. It also runs a culling controller that monitors Jupyter kernel/terminal activity via HTTP API calls and automatically stops idle notebooks by setting replicas to 0 after a configurable idle timeout.

The second operator, **odh-notebook-controller**, is an OpenShift-specific extension that watches the same `Notebook` CRs and adds platform integration. It injects OAuth proxy sidecar containers via a mutating webhook, creates OpenShift Routes with TLS termination, manages OAuthClient resources for authentication, provisions NetworkPolicies for workload isolation, aggregates CA certificate bundles for TLS trust, and integrates with Data Science Pipelines (DSPA) for Elyra notebook pipeline support. Together these controllers form the complete notebook workbench management stack for RHOAI.

## Architectural Analysis

The kubeflow notebook controllers follow a **dual-operator pattern** where the upstream Kubeflow controller handles core Kubernetes lifecycle management (StatefulSet, Service, culling) while the downstream ODH controller adds OpenShift-specific platform integration (OAuth, Routes, NetworkPolicy, CA bundles). This separation preserves upstream compatibility — the kf-notebook-controller code closely tracks the upstream Kubeflow project, and the odh-notebook-controller is a pure downstream addition. Both controllers watch the same `kubeflow.org/v1 Notebook` CR, which creates a collaborative reconciliation model where each controller is responsible for its own subset of managed resources.

The **authentication architecture** uses the RHOAI 2.x pattern: OpenShift Routes with oauth-proxy sidecars. The odh-notebook-controller's mutating webhook intercepts Notebook CREATE/UPDATE operations and injects an oauth-proxy container with `--provider=openshift`, SAR-based authorization (`verb=get, resource=notebooks`), and auto-generated OAuth credentials. The OAuthClient is a cluster-scoped resource that cannot use owner references, so the controller manages it with a finalizer (`notebook-oauth-client-finalizer.opendatahub.io`) to ensure cleanup on notebook deletion. TLS certificates for the OAuth service are auto-provisioned by OpenShift's Service CA operator via the `service.beta.openshift.io/serving-cert-secret-name` annotation.

The **culling system** is notable for its in-band activity detection approach. Rather than relying on metrics or pod-level signals, the CullingReconciler makes direct HTTP calls to each notebook's Jupyter API (`/api/kernels`, `/api/terminals`) to check kernel execution state and terminal activity timestamps. This provides accurate idle detection but creates a network dependency between the controller and every running notebook pod on port 8888. The culling logic uses annotation-based signaling: it sets `kubeflow-resource-stopped` on the Notebook CR, which the NotebookReconciler reads to set StatefulSet replicas to 0.

The **webhook mutation logic** in odh-notebook-controller implements an update-blocking pattern to prevent disruption to running notebooks. When a Notebook update would change the pod spec while the notebook is running, the webhook sets a `notebooks.opendatahub.io/update-pending` annotation instead of applying the change, deferring it until the notebook is stopped or restarted. This is a deliberate UX trade-off — preventing data loss from unexpected restarts at the cost of delayed configuration updates.

The **network isolation model** creates two NetworkPolicies per notebook: one allowing the controller to reach the notebook on port 8888 (required for culling activity checks), and one allowing unrestricted ingress on port 8443 (the OAuth proxy port, since authentication is handled by the proxy itself). This provides workload-level microsegmentation while still allowing the OAuth proxy to accept traffic from the OpenShift router.
