## Purpose

**Short**: KServe is a Kubernetes-native platform for deploying, managing, and serving machine learning models with support for serverless inference, LLM serving, and local model caching.

**Detailed**: KServe provides a set of Kubernetes Custom Resource Definitions (CRDs) and controllers that orchestrate the complete lifecycle of ML model serving infrastructure. The core InferenceService CRD supports three deployment modes — Serverless (via Knative Serving), Raw Deployment (standard Kubernetes Deployments), and ModelMesh (shared serving infrastructure) — and manages predictor, transformer, and explainer components as a unified serving pipeline.

The platform has expanded significantly with LLM-specific capabilities through the LLMInferenceService CRD, which supports disaggregated prefill/decode architectures, multi-node distributed inference via LeaderWorkerSet, and integration with Gateway API Inference Extension for advanced request scheduling. The local model caching subsystem (LocalModelCache, LocalModelNode, LocalModelNodeGroup) enables pre-positioning models on cluster nodes for reduced cold-start latency.

In the RHOAI distribution, KServe integrates with OpenShift certificate management (replacing cert-manager), OpenShift SecurityContextConstraints for LLM workloads, and the platform's ingress infrastructure via the openshift-ai-inference gateway. The ODH overlay removes upstream ClusterServingRuntime definitions (managed separately by the platform operator), disables automatic service account token mounting, and configures storage initializer with Modelcar support.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| kserve-controller-manager | Go Operator | Main controller manager running InferenceService, InferenceGraph, TrainedModel, and LLMInferenceService controllers with admission webhooks |
| kserve-agent | Go Sidecar | Pod sidecar handling request proxying, model pulling, logging, batching, and metrics collection |
| kserve-router | Go Sidecar | InferenceGraph routing sidecar that traverses DAG nodes and forwards inference requests between services |

| kserve-localmodelnode-agent | Go DaemonSet Agent | Runs on each node to manage local model downloads via batch Jobs using storage-initializer |

## Architectural Analysis

KServe's architecture follows a multi-controller pattern where a single repository produces six distinct container images, each serving a different role in the model serving lifecycle. The main controller manager (`cmd/manager`) runs four controllers (InferenceService, InferenceGraph, TrainedModel, LLMInferenceService) in a single process with shared leader election, while the local model caching subsystem is split into two separate binaries (`cmd/localmodel` and `cmd/localmodelnode`) that run as independent controller managers. This separation allows the caching subsystem to be deployed as a DaemonSet on individual nodes without the overhead of the full controller manager.

The most architecturally significant development is the LLMInferenceService controller, which introduces a fundamentally different serving pattern from the original InferenceService. While InferenceService follows a Knative-first approach (create a Knative Service, let Knative handle scaling and networking), LLMInferenceService bypasses Knative entirely and directly manages Deployments, LeaderWorkerSets, and Gateway API resources. The LLM controller also integrates with the Gateway API Inference Extension (InferencePool/InferenceModel CRDs) for request-aware load balancing, and supports disaggregated serving where prefill and decode workloads run as separate deployments with different resource profiles. This dual-path architecture means the codebase must maintain two largely independent networking stacks — one Knative-based (InferenceService) and one Gateway API-native (LLMInferenceService).

The RHOAI overlay (`config/overlays/odh/`) makes several security-significant changes: it replaces cert-manager with OpenShift's built-in certificate injection, disables automatic service account token mounting (a defense-in-depth measure), and adds SecurityContextConstraints for LLM workloads that grant capabilities like IPC_LOCK and SYS_RAWIO needed for high-performance GPU memory operations. The multi-node SCC variant additionally allows hostPath volumes for shared storage across leader-worker pods. The overlay also removes the upstream ClusterServingRuntime definitions, as these are managed by the platform operator in RHOAI deployments.

A notable FIPS compliance gap exists in the local model caching components: the `localmodel.Dockerfile.konflux` uses `CGO_ENABLED=0` and a distroless base image without FIPS build flags, while the main controller, agent, and router all properly use `CGO_ENABLED=1` with `GOEXPERIMENT=strictfipsruntime` on UBI 9 bases. This means the local model controller binary would fail check-payload validation if it were subject to FIPS requirements. The `localmodel-agent.Dockerfile.konflux` also uses distroless but does integrate Hermeto/cachi2 prefetch for hermetic builds.

The webhook architecture is comprehensive, covering both mutation and validation across the resource lifecycle. The pod mutating webhook (`/mutate-pods`) is particularly critical — it intercepts pod creation for any pod with a `serving.kserve.io/inferenceservice` label and injects up to five sidecar/init containers (storage-initializer, agent, batcher, metrics aggregator, accelerator). This webhook-driven injection pattern means that the KServe controller doesn't directly create pods with sidecars; instead, it creates Deployments or Knative Services, and the webhook mutates the resulting pods at creation time. This loose coupling enables sidecars to be added independently of the controller reconciliation logic but creates an ordering dependency: the webhook configuration must be deployed before any InferenceService pods are created.
