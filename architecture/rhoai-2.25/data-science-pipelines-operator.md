## Purpose

**Short**: Manages the lifecycle of Data Science Pipelines (Kubeflow Pipelines) deployments on OpenShift by reconciling DataSciencePipelinesApplication custom resources into a suite of interconnected microservices.

**Detailed**: The Data Science Pipelines Operator (DSPO) is a Kubernetes operator built with controller-runtime that watches `DataSciencePipelinesApplication` (DSPA) custom resources and deploys a complete Kubeflow Pipelines V2 stack within each namespace. For each DSPA CR, the operator provisions up to eight sub-components: an API server, persistence agent, scheduled workflow controller, Argo workflow controller, ML Metadata gRPC server, ML Metadata Envoy proxy, ML Pipelines UI, and a pipeline version webhook.

The operator manages both infrastructure dependencies (MariaDB for metadata storage, MinIO for artifact storage) and the pipeline execution components. It supports external database and object storage backends as alternatives to the built-in MariaDB and MinIO deployments. The operator performs health checks against database and object storage before deploying pipeline components, implements pod-to-pod TLS via OpenShift service-ca certificate injection, and exposes Prometheus metrics for monitoring pipeline stack health.

DSPO integrates deeply with the OpenShift platform, using Routes for external access with OAuth proxy sidecars for authentication, service-ca for automatic TLS certificate provisioning, and trusted CA bundle propagation from the platform's `odh-trusted-ca-bundle` ConfigMap. It also supports FIPS-compliant builds and InstructLab managed pipeline imports for RHELAI integration.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| DSPO Controller Manager | Go Operator (controller-runtime) | Reconciles DSPA CRs, orchestrates deployment of all sub-components with dependency ordering |
| DS Pipeline API Server | Deployed Service (Go) | Kubeflow Pipelines API server providing REST (8888/TCP) and gRPC (8887/TCP) endpoints for pipeline CRUD and execution |
| Persistence Agent | Deployed Service (Go) | Syncs Argo Workflow status back to the API server, tracking pipeline run progress |
| Scheduled Workflow Controller | Deployed Service (Go) | Manages recurring pipeline runs based on cron schedules |
| Argo Workflow Controller | Deployed Service (Go) | Executes pipeline DAGs as Argo Workflows, manages workflow lifecycle |
| ML Metadata gRPC Server | Deployed Service (C++) | Stores ML metadata (artifacts, executions, contexts) in MariaDB, serves via gRPC |
| ML Metadata Envoy Proxy | Deployed Service (Envoy) | Reverse proxy for MLMD gRPC, provides HTTP-to-gRPC translation and OAuth authentication |
| ML Pipelines UI | Deployed Service (Node.js) | Web UI for viewing pipeline runs, artifacts, and experiment results |
| Pipeline Version Webhook | Deployed Service (Go) | Validating and mutating admission webhook for PipelineVersion resources (Kubernetes pipeline store mode) |
| MariaDB | Deployed Database | Default relational database for pipeline metadata; optional, replaceable with external DB |
| MinIO | Deployed Object Store | Default S3-compatible artifact storage; optional, replaceable with external storage |

## Architectural Analysis

The Data Science Pipelines Operator follows a template-driven architecture where each sub-component is defined as a set of Go templates in `config/internal/`. The operator renders these templates with parameters extracted from the DSPA CR spec and applies them using the manifestival library, which provides declarative reconciliation with automatic diff-based updates. This approach cleanly separates the deployment configuration (YAML templates) from the orchestration logic (Go controllers), making it straightforward to modify deployed resources without changing controller code.

The reconciliation flow implements a strict dependency ordering: database and object storage are provisioned and health-checked first, followed by common resources, then pipeline components in a specific order (API server, persistence agent, scheduled workflow, UI, workflow controller), with MLMD deployed last because it depends on TLS certificates that are only available after the gRPC service is created and OpenShift service-ca has injected certificates. This ordering is critical for correctness and is implemented as a sequential chain in the `Reconcile()` method rather than a parallel deployment. If any prerequisite fails, the controller sets degraded status conditions and requeues after a configurable interval (default 20 seconds).

The operator's TLS architecture has two distinct modes controlled by the `podToPodTLS` field (default: true). When enabled, all inter-component communication uses TLS certificates provisioned by OpenShift's service-ca operator. This includes API server to MariaDB, API server to MLMD gRPC, and persistence agent to API server. The operator also manages a sophisticated CA bundle aggregation pipeline: it collects CA certificates from three sources (the platform's `odh-trusted-ca-bundle` ConfigMap, user-provided CABundle, and service-ca certificates), merges them into a single `dsp-trusted-ca-{name}` ConfigMap, and mounts this into every component that needs to validate TLS connections. The `SSL_CERT_DIR` environment variable is set to include both the system cert directory and the service account cert path.

The webhook component (`pipelineversions.pipelines.kubeflow.org`) is notably unique in the architecture: it runs in the operator's namespace (not the DSPA namespace) and is shared across all DSPAs that use `PipelineStore=kubernetes`. The operator implements careful lifecycle management for this shared resource, including checking whether other DSPAs still need the webhook before cleaning it up during DSPA deletion. The webhook deployment reuses the API server image but runs with webhook-specific flags, which is an unusual but pragmatic design choice that avoids maintaining a separate webhook binary.

The operator exposes eight Prometheus gauge metrics that provide a comprehensive view of pipeline stack health, each labeled by DSPA name and namespace. These metrics track the availability of database, object store, and the readiness of every deployed component. Combined with the condition-based status reporting on the DSPA CR (using standard `metav1.Condition` types), this provides both pull-based (Prometheus) and push-based (CR status) observability of the pipeline deployment lifecycle.
