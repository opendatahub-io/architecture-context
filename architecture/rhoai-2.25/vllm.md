## Purpose

**Short**: GPU-accelerated LLM inference runtime container image built on vLLM with a TGIS gRPC adapter for serving large language models in RHOAI.

**Detailed**: This repository is a downstream RHOAI container image wrapper for the vLLM inference engine. It contains no application source code — it consists solely of a single Konflux Dockerfile (`Dockerfile.konflux.cuda`) that extends the Red Hat AI Inference Server (RHAIIS) vLLM CUDA base image (`registry.redhat.io/rhaiis/vllm-cuda-rhel9`) by configuring environment variables and setting the entrypoint to the `vllm_tgis_adapter` Python module.

The vLLM TGIS adapter provides a gRPC interface (port 8033) compatible with the Text Generation Inference Server (TGIS) protocol, alongside vLLM's native OpenAI-compatible HTTP API (port 8000). This allows the component to be used as a model serving runtime within KServe's InferenceService framework, where models are deployed as ServingRuntime pods. The adapter bridges the gap between KServe's gRPC-based model serving protocol and vLLM's high-performance inference engine, enabling tensor-parallel GPU inference with PagedAttention, continuous batching, and speculative decoding.

All application code, Python dependencies, and GPU runtime libraries (CUDA) are inherited from the RHAIIS base image. This repo's role is to provide the RHOAI-specific container image configuration, Konflux build pipeline integration, and multi-architecture build support (x86_64, arm64).

## Architectural Analysis

This repository follows the "thin downstream wrapper" pattern common in RHOAI's ML runtime components. Rather than containing application source code, it serves as a Konflux build definition that layers RHOAI-specific configuration (container labels, environment variables, entrypoint) on top of the Red Hat AI Inference Server (RHAIIS) base image. The RHAIIS image (`registry.redhat.io/rhaiis/vllm-cuda-rhel9`) contains the complete vLLM installation, CUDA runtime, Python dependencies, and the vllm_tgis_adapter module — this repo merely configures the final container entry point.

The architectural significance of this pattern is that security, dependency, and FIPS analysis must be performed on the RHAIIS base image source rather than this repository. The only build-time decision made here is the RHAIIS version pinning (`RHAIIS_VERSION=3.2.2-1772160593`), which determines the exact vLLM version, CUDA toolkit version, and Python dependency tree. Recent commits show active version management — multiple commits update the RHAIIS version and switch between TGIS adapter endpoints, indicating this repo serves as the version control point for which RHAIIS build ships with RHOAI 2.25.

The dual-protocol design (gRPC on 8033, HTTP on 8000) is notable: the TGIS gRPC adapter provides backward compatibility with KServe's legacy TGIS-based model serving protocol, while the HTTP port exposes vLLM's native OpenAI-compatible API. The `DISABLE_LOGPROBS_DURING_SPEC_DECODING=false` environment variable override is a deliberate UX decision — vLLM normally disables logprobs during speculative decoding as an optimization, but for a hosted serving platform, users expect consistent API behavior regardless of internal optimization strategies. This is a concrete example of the RHOAI wrapper adding platform-appropriate defaults.

The Konflux build pipeline supports multi-architecture builds (x86_64 and arm64), with push builds targeting `quay.io/rhoai/odh-vllm-cuda-rhel9` and pull request builds using ephemeral images with 5-day expiry. The pipeline configuration is centrally managed via the `konflux-central` repository and synced to this repo automatically, as documented in the `.tekton/README.md`. Build timeouts are set to 10 hours for the pipeline and 8 hours for individual tasks, reflecting the large image size typical of CUDA-based ML containers.
