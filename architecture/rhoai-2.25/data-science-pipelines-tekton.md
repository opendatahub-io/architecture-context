## Purpose

**Short**: Backend platform for running Kubeflow ML pipelines with Tekton as the workflow execution engine, providing API management, scheduling, caching, and artifact persistence.

**Detailed**: Data Science Pipelines Tekton (DSPT) integrates Kubeflow Pipelines with Tekton Pipelines to provide a complete ML pipeline management platform. It replaces Argo Workflows with Tekton as the execution backend, enabling users to define, compile, upload, schedule, and execute machine learning workflows on Kubernetes using Tekton PipelineRuns. The platform is based on Kubeflow Pipelines 1.8.4 and Tekton >= 0.41.0.

The component consists of multiple cooperating services: an API server that exposes gRPC and REST endpoints for pipeline CRUD operations, a persistence agent that watches Tekton PipelineRuns and syncs their status to a MySQL database, a scheduled workflow controller that manages cron and periodic scheduling of pipeline executions, a cache server (mutating admission webhook) that intercepts pod creation to serve cached results for repeated pipeline steps, and supporting tools for loop execution, task sequencing, and kubectl resource management. Within RHOAI, this component is deployed and managed by the Data Science Pipelines Operator (data-science-pipelines-operator), not directly from these manifests.

The frontend (React/TypeScript web UI) is included in the repository but is separately deployed. A Python SDK (kfp-tekton) enables users to author pipelines programmatically and compile them to Tekton YAML. The Tekton Catalog directory provides custom Tekton Tasks and a PipelineLoop controller for iterative pipeline execution.

## Architectural Analysis

The Data Science Pipelines Tekton repository is a fork of the upstream Kubeflow Pipelines project, modified to replace Argo Workflows with Tekton Pipelines as the execution engine. The architecture retains the Kubeflow Pipelines API surface (v1beta1) while substituting Tekton PipelineRuns for Argo Workflows at the execution layer. This is visible in the RBAC rules, which include both `argoproj.io` (Argo) and `tekton.dev` (Tekton) API groups — the Argo rules are vestigial from the upstream codebase and are maintained for backward compatibility, but Tekton is the active execution engine (evidenced by `PIPELINE_RUNTIME=tekton` in the deployment manifests).

The cache server implements a clever optimization pattern: a mutating admission webhook intercepts pod creation for Tekton TaskRun pods, computes a SHA256 cache key from the TaskRun spec plus pipeline metadata, and queries a MySQL `execution_caches` table. On cache hits, the webhook replaces the task container with a minimal UBI image that echoes cached outputs, dramatically reducing CPU/memory to 0.01 CPU / 16Mi memory. A companion pod watcher monitors completed pods and writes their outputs back to the cache. This design is transparent to the pipeline author — caching is enabled via labels (`pipelines.kubeflow.org/cache_enabled=true`) and requires no code changes. The cache staleness is configurable per-task via RFC3339 duration annotations.

The PipelineLoop controller in `tekton-catalog/` is a Tekton Custom Task that extends Tekton's execution model with iterative (loop) semantics. It uses the Knative controller framework rather than controller-runtime, creating one PipelineRun per loop iteration and managing parallelism up to a configurable limit. The controller integrates with both the cache system (via a shared MySQL database) and an optional S3-compatible object store for log persistence. This is a critical piece of the KFP-Tekton architecture — without it, Kubeflow Pipelines' `dsl.ParallelFor` and loop constructs cannot be expressed in Tekton.

A notable security consideration is the `pipeline-runner` Role, which grants wildcard (`*`) permissions on many resource types including all `kubeflow.org` resources, Seldon deployments, core pods/services/deployments, and batch jobs. This is intentional — the pipeline-runner ServiceAccount is used by the actual pipeline execution pods, which may need to create arbitrary Kubernetes resources as part of ML workflows. However, this makes the pipeline-runner ServiceAccount a privileged principal whose tokens should be carefully guarded. The API server, by contrast, follows least-privilege with targeted RBAC rules and delegates authorization checks to SubjectAccessReview in multi-user mode.

The persistence agent operates at cluster scope (ClusterRole) to monitor PipelineRuns across all namespaces. It uses Kubernetes informers with a 30-second resync interval and reports status changes to the API server via gRPC. After a configurable TTL (default 86400 seconds / 1 day), it deletes completed PipelineRun resources to prevent cluster resource exhaustion. This TTL is separate from the pipeline run metadata in MySQL, which is retained indefinitely.
