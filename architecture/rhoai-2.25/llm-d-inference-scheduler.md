## Purpose

**Short**: Inference request scheduler (Endpoint Picker / EPP) that makes optimized routing decisions for LLM inference requests via a Kubernetes Gateway API-based inference gateway.

**Detailed**: The llm-d-inference-scheduler is the core scheduling component of the llm-d inference framework. It implements an Endpoint Picker (EPP) that extends the upstream Gateway API Inference Extension (GIE) project to provide intelligent, pluggable routing of inference requests to model-serving pods (e.g., vLLM instances). The EPP runs as an external processing (ext-proc) gRPC server that Envoy proxy calls back to when it receives inference requests, enabling the scheduler to choose the optimal backend pod for each request.

The scheduler supports multiple routing strategies through a plugin architecture with configurable filters, scorers, and profile handlers. Key capabilities include prefix-cache-aware scoring (both estimation-based and real-time KV-cache tracking via ZMQ events), load-aware scoring, session affinity, and disaggregated Prefill/Decode (P/D) scheduling where prefill and decode stages can be routed to specialized worker pods. The plugin configuration is driven by YAML-based `EndpointPickerConfig` resources that define which plugins are instantiated and how scheduling profiles compose them.

Within RHOAI, this component is deployed as a StatefulSet that works alongside an Envoy-based Gateway (Gateway API) and InferencePool/InferenceModel custom resources from the GIE project. It watches InferencePool and InferenceModel CRDs and Pod/EndpointSlice resources to maintain an up-to-date view of available model-serving endpoints, then uses its plugin pipeline to select the best endpoint for each incoming inference request.

## Architectural Analysis

The llm-d-inference-scheduler is architecturally a thin extension layer on top of the upstream Gateway API Inference Extension (GIE) project. The `cmd/epp/main.go` entry point is remarkably minimal — it registers GIE's built-in plugins, then registers the llm-d-specific plugins, and delegates the entire server lifecycle to GIE's `runner.NewRunner().Run(ctx)`. This design means the heavy lifting of gRPC server setup, Envoy ext-proc protocol handling, InferencePool/InferenceModel reconciliation, and the core scheduling framework all come from the GIE dependency. The llm-d-inference-scheduler's value-add is purely in its custom plugins.

The plugin architecture follows GIE's factory pattern: each plugin type registers a factory function via `plugins.Register(type, factory)`, and factories receive a name, raw JSON parameters, and a handle providing context. This allows the same plugin type to be instantiated multiple times with different configurations (e.g., multiple `by-label` filters with different label criteria). The scheduling pipeline is composed via `EndpointPickerConfig` YAML that defines plugin instances and scheduling profiles — a declarative composition model that avoids hard-coded routing logic. Nine plugin types are registered: `by-label`, `by-label-selector`, `decode-filter`, `prefill-filter`, `prefill-header-handler`, `pd-profile-handler`, `prefix-cache-scorer` (with two modes), `load-aware-scorer`, and `session-affinity-scorer`.

The disaggregated Prefill/Decode (P/D) scheduling is the most architecturally significant feature. The `PdProfileHandler` implements a multi-profile scheduling strategy: it first runs the decode profile, then conditionally runs the prefill profile based on a prompt length threshold and prefix cache hit rate. If the non-cached suffix of the prompt is shorter than the threshold, P/D is skipped and the request is handled entirely by the decode worker. This threshold-based decision is informed by the `PrefixCachePlugin`'s cycle state, creating a cross-plugin data dependency through the `CycleState` mechanism. The `PrefillHeaderHandler` pre-request plugin then wires the prefill worker's address into an `x-prefiller-host-port` header that the decode worker's sidecar uses to coordinate the remote prefill.

The `PrefixCacheTrackingScorer` (`cache_tracking` mode) is notable for its use of ZeroMQ for real-time event streaming from vLLM instances. It initializes a `kvevents.Pool` that subscribes to KV-cache block events and feeds them into a `kvcache.Indexer` running in a separate goroutine. This gives the scorer a near-real-time view of which prefix blocks are cached on which pods, enabling far more accurate routing than the estimation-based alternative. The trade-off is operational complexity: it requires the ZMQ library (libzmq) at runtime, HuggingFace tokenizer models, and a tokenizer submodule compiled from Rust source during the container build. The Konflux Dockerfile handles this by installing Rust/Cargo from Red Hat repos and building `libtokenizers.a` from a git submodule with vendored cargo dependencies via cachi2.

The deployment architecture is interesting in that the EPP can be deployed either as a StatefulSet (OpenShift base overlay) or a Deployment (inference-gateway component). The StatefulSet approach provides stable pod identity, which may be relevant for session affinity or ZMQ connection stability. The contracts/schemas directory contains JSON schemas for Istio CRDs (AuthorizationPolicy, DestinationRule, EnvoyFilter, Gateway, VirtualService, etc.), suggesting the component is designed to work within an Istio service mesh environment, although the EPP itself does not create or manage these resources.
