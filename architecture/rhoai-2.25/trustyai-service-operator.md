## Purpose

**Short**: Kubernetes operator that manages TrustyAI explainability services, language model evaluation jobs, and guardrails orchestration within the Red Hat OpenShift AI platform.

**Detailed**: The TrustyAI Service Operator is a multi-controller operator built with controller-runtime that manages three distinct CRDs: TrustyAIService (bias/explainability monitoring for ML models), LMEvalJob (language model evaluation using lm-evaluation-harness), and GuardrailsOrchestrator (guardrails orchestration for inference services with content detectors). The operator uses a pluggable architecture where controllers are enabled via `--enable-services` flags (TAS, LMES, GORCH, JOB_MGR), allowing the same binary to be deployed in different configurations.

The operator integrates deeply with KServe InferenceServices (patching them to route inference payloads through TrustyAI for monitoring), Kueue (for fair-share scheduling of LMEval jobs), Prometheus (for metrics exposure), and the OpenShift OAuth proxy (for authentication). It creates deployments, services, routes, service monitors, and Istio service mesh resources (DestinationRules, VirtualServices) dynamically through controller reconciliation logic. The RHOAI distribution uses restricted defaults (online access and code execution disabled for LMEval) compared to the ODH distribution.

## Architectural Analysis

The TrustyAI Service Operator demonstrates a modular multi-controller architecture where a single Go binary houses four independent controller loops (TAS, LMES, GORCH, JOB_MGR), selectively enabled via command-line flags. This allows the same operator to be deployed in different configurations — the LMES-only overlay, for example, deploys just the LMEvalJob controller. This pattern reduces image proliferation but means the operator binary includes all CRD types and dependencies regardless of which controllers are active. The main.go entry point disables the default controller-runtime caching for ConfigMap, Secret, Pod, and Service kinds to avoid cluster-wide informer flooding, a pragmatic choice given the operator's need to work across multiple namespaces.

The TrustyAIService controller follows the RHOAI 2.x ingress pattern exclusively: OpenShift Routes with oauth-proxy sidecars for authentication. It creates an internal ClusterIP service (port 80/HTTP and 443/HTTPS) alongside a TLS service (port 443 targeting oauth-proxy on 8443), with the Route using reencrypt termination. The controller also creates Istio DestinationRules and VirtualServices conditionally — only when it detects InferenceServices with `sidecar.istio.io/inject: true` annotation, indicating service mesh is in use. This dual networking approach (Route + optional Istio mesh) adds complexity but ensures compatibility across mesh and non-mesh deployments.

The LMEvalJob controller uses a pod-per-job model with a distinctive driver sidecar pattern. An init container copies the driver binary from the driver image into a shared EmptyDir volume. The main container runs lm-eval while the driver monitors progress by parsing tqdm output patterns. The controller polls the driver's HTTP endpoint (port 18080) for status updates rather than relying on pod conditions alone. Security is enforced at the pod spec level (RunAsNonRoot, drop ALL capabilities, SeccompProfile RuntimeDefault) and at the application level through "protected" environment variables that the controller owns — users cannot override HF_HUB_DISABLE_TELEMETRY, TRUST_REMOTE_CODE, and online/offline mode flags. The RHOAI distribution defaults to restrictive settings (no online access, no code execution) with the DSC ConfigMap providing an override path.

The GuardrailsOrchestrator controller implements an auto-configuration pattern: it discovers KServe InferenceServices and ServingRuntimes, extracts their endpoints and TLS configuration, and generates orchestrator YAML configuration dynamically. It supports an optional sidecar gateway (a vllm-orchestrator-gateway container running alongside the orchestrator) and optional built-in regex detectors. The generated ConfigMap includes TLS certificate paths when HTTPS services are detected, and a SHA256 hash of the configuration is tracked in status to detect drift. The OAuth integration follows the same openshift-oauth-proxy pattern as TAS but adds a second OAuth proxy for the gateway sidecar, each with its own downstream/upstream port pair.

A notable architectural concern is the `default-lmeval-user-rolebinding` ClusterRoleBinding, which grants all authenticated users (`system:authenticated` group) full CRUD access to LMEvalJob resources cluster-wide. While this simplifies onboarding, it means any authenticated user can create LMEval pods — potentially consuming significant GPU resources. The fix in commit d4909b4 (RHOAIENG-34218) addressed a related issue where TrustyAI was granting pod listing across namespaces. Resource governance depends on Kueue integration (JOB_MGR controller) when enabled, providing quota enforcement and fair-share scheduling.
