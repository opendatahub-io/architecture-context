## Purpose
**Short**: Kubernetes-native operator for managing distributed training jobs across multiple ML/AI frameworks (PyTorch, TensorFlow, XGBoost, MPI, PaddlePaddle, JAX).

**Detailed**: The Kubeflow Training Operator (KFTO) is a controller-runtime based Kubernetes operator that manages the lifecycle of distributed training jobs. It provides CRDs for six training frameworks: PyTorchJob, TFJob, XGBoostJob, MPIJob, PaddleJob, and JAXJob. Each CRD allows users to declare distributed training workloads with replica specifications (masters, workers, parameter servers), and the operator handles pod creation, service creation, status tracking, and cleanup.

The operator supports optional gang scheduling integration via Volcano or scheduler-plugins, enabling coordinated scheduling of all replicas in a training job. It also manages webhook-based validation for job submissions with self-managed TLS certificate rotation. For PyTorchJob specifically, the operator dynamically creates NetworkPolicy resources to isolate training job pods and allow only intra-job communication plus metrics scraping from the openshift-monitoring namespace.

In RHOAI, the operator is deployed into the `opendatahub` namespace with the `kubeflow-` name prefix via kustomize overlays managed by the rhods-operator. The PyTorch init container image is set to the same operator image for FIPS compliance.

### Dynamically Created Resources (per training job)

The operator dynamically creates the following Kubernetes resources for each training job:

| Resource Type | Created By | Purpose |
|---------------|------------|---------|
| Pod | All controllers | Worker/master/PS replica pods for training jobs |
| Service (headless) | All controllers | Inter-pod communication for distributed training (DNS-based discovery) |
| NetworkPolicy | PyTorchJob controller only | Isolates PyTorch training pods; allows intra-job communication and metrics scraping from openshift-monitoring |
| HorizontalPodAutoscaler | PyTorchJob controller only | Optional HPA for elastic PyTorch training |
| PodGroup (volcano.sh) | All controllers (optional) | Gang scheduling coordination when Volcano is configured |
| PodGroup (scheduler-plugins) | All controllers (optional) | Gang scheduling coordination when scheduler-plugins is configured |

### Network Policies (Dynamic)

For each PyTorchJob, the controller creates a NetworkPolicy that:
- **Allows ingress** from pods with matching `training.kubeflow.org/job-name` label (intra-job communication)
- **Allows ingress** on port 8080/TCP from pods in the `openshift-monitoring` namespace (Prometheus metrics scraping)
- **Selects** pods with the `training.kubeflow.org/job-name` label matching the job name

### Ingress

No ingress resources (Routes, HTTPRoutes, Gateways) are created. This is a cluster-internal operator with no external-facing endpoints.
