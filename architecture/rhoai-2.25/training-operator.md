## Purpose

**Short**: Kubernetes-native operator for managing distributed training jobs across multiple ML frameworks (PyTorch, TensorFlow, XGBoost, MPI, JAX, PaddlePaddle).

**Detailed**: The Kubeflow Training Operator (KFTO) is a controller-runtime-based Kubernetes operator that reconciles six custom resource definitions representing distributed training workloads. Each CRD maps to a specific ML framework: PyTorchJob, TFJob, XGBoostJob, MPIJob, JAXJob, and PaddleJob. The operator manages the full lifecycle of these jobs, including creating and monitoring worker/master pods, establishing headless services for inter-pod communication, injecting framework-specific environment variables for distributed coordination, and cleaning up resources upon job completion or failure.

The operator supports advanced features including elastic training (PyTorch HPA-based autoscaling of workers), gang scheduling integration with both Volcano and Kubernetes Scheduler-Plugins, job suspension/resumption, configurable restart policies with exit code-based retry logic, and MPI jobs using a kubectl-exec pattern instead of SSH for remote execution. In the RHOAI distribution, the operator is built with FIPS-compliant crypto (strictfipsruntime), runs as a non-root user, and integrates with OpenShift monitoring via PodMonitor.

The operator deploys as a single Deployment with one container that runs all six framework controllers and a validating webhook server. It uses self-managed TLS certificates (via OPA cert-controller) for the webhook endpoint and exposes Prometheus metrics on port 8080.

## Architectural Analysis

The Kubeflow Training Operator follows a well-structured extensible controller pattern where a common `JobController` base handles generic distributed job lifecycle management (pod/service creation, status tracking, gang scheduling, cleanup policies) while framework-specific reconcilers implement the `ControllerInterface` to customize cluster topology, environment variable injection, and completion semantics. This design allows all six frameworks to share ~2,800 lines of common infrastructure code while each framework adds only 400-1,600 lines of framework-specific logic.

The MPI controller stands out architecturally as it departs significantly from the common pattern. Unlike other frameworks that use headless services for pod discovery, MPI jobs use a Launcher/Worker topology where the launcher pod executes commands on worker pods via `kubectl exec` (through the `kubexec.sh` script). This SSH-less approach is more secure than traditional MPI SSH-based communication because RBAC scopes the launcher's `pods/exec` permission to only its own worker pods by generating job-specific Roles with `resourceNames` restrictions. The MPI controller also creates its own ConfigMaps (hostfile, kubexec.sh, discover_hosts.sh), ServiceAccounts, Roles, and RoleBindings - resources that no other framework controller creates.

The PyTorch controller is the most feature-rich, with unique capabilities including: elastic training via `ElasticPolicy` (with HPA integration for autoscaling workers), init containers that wait for master DNS resolution before workers start, and automatic NetworkPolicy creation that allows intra-job traffic plus OpenShift monitoring access on port 8080. The elastic training configuration supports multiple rendezvous backends (c10d, etcd, etcd-v2) and generates `PET_*` environment variables compatible with `torchrun`/`torch.distributed.elastic`.

A notable security pattern is the webhook certificate management. Rather than depending on cert-manager (which may not be available), the operator uses the OPA cert-controller library to self-manage its TLS certificates. The cert CA is stored in a namespace-scoped Secret (accessed via a Role, not a ClusterRole), and the readiness probe gates on certificate availability, ensuring no traffic is served before TLS is ready. The RHOAI distribution builds with `CGO_ENABLED=1 GOEXPERIMENT=strictfipsruntime` for FIPS compliance and runs as non-root user 65532.

The RHOAI overlay adds several operational enhancements: a `kubeflow-` name prefix for all resources (avoiding naming conflicts), a PodMonitor for Prometheus integration, aggregated ClusterRoles (`training-edit`, `training-view`) that integrate with OpenShift's built-in `edit`/`view`/`admin` ClusterRoles via label-based aggregation, and image parameterization through `configMapGenerator` with `replacements`. The RHOAI overlay also patches the Service to remove the metrics port (keeping only the webhook port), since metrics are scraped directly from the pod via PodMonitor rather than through a Service.
