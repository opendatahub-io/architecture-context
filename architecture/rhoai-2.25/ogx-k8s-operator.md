## Purpose

**Short**: Kubernetes operator that manages the lifecycle of Llama Stack AI inference server distributions on OpenShift/Kubernetes clusters.

**Detailed**: The Llama Stack K8s Operator (internally `ogx-k8s-operator`, upstream `llama-stack-k8s-operator`) provides a declarative interface for deploying and managing Llama Stack distribution servers via a custom resource called `LlamaStackDistribution` (short name `llsd`). Users specify a distribution name (e.g., `remote-vllm`, `ollama`, `tgi`) or a direct container image, and the operator reconciles a complete deployment stack including a Deployment, Service, ServiceAccount, PersistentVolumeClaim, NetworkPolicy, and optional Ingress for external access.

The operator embeds a mapping of supported distribution names to container images via `distributions.json`, which is compiled into the binary. For RHOAI deployments, a single Red Hat-certified distribution image is injected via the `RELATED_IMAGE_RH_DISTRIBUTION` environment variable, overriding the embedded upstream images. The operator features a kustomize-based manifest rendering pipeline with custom Go plugins for name prefixing, namespace injection, field mutation, and NetworkPolicy transformation. It also provides CA bundle auto-detection for ODH/RHOAI trusted CA bundles and integrates content-based hash annotations to trigger rolling restarts when referenced ConfigMaps change.

The operator runs as a single-replica Deployment managed by the platform operator (`rhods-operator` for RHOAI, `opendatahub-operator` for ODH). It uses leader election via Kubernetes Leases to support high-availability configurations. Health and version information are actively polled from the deployed Llama Stack server's `/v1/health`, `/v1/providers`, and `/v1/version` endpoints and surfaced in the CR status.

## Architectural Analysis

The Llama Stack K8s Operator follows a conventional controller-runtime operator pattern but with a distinctive twist: instead of directly constructing Kubernetes resource objects in Go code, it uses an embedded kustomize pipeline with custom Go transformer plugins to render manifests at reconciliation time. Base manifests live in `controllers/manifests/base/` and are transformed through four plugin stages: name prefixing (appending instance name to resource names, excluding Deployment), namespace injection, field mutation (setting ports, labels, storage sizes, replica counts via JSON Pointer paths), and NetworkPolicy transformation (dynamically building ingress rules from the CR spec). This approach keeps resource templates declarative and human-readable while still allowing fine-grained per-instance customization.

The operator manages a single CRD (`LlamaStackDistribution`) that encapsulates the full deployment lifecycle of a Llama Stack inference server. Each CR instance results in up to seven Kubernetes resources: Deployment, Service, ServiceAccount, PersistentVolumeClaim (optional), NetworkPolicy (feature-flagged), Ingress (optional, when `spec.network.exposeRoute=true`), and a ClusterRoleBinding for OpenShift's `anyuid` SCC. The ClusterRoleBinding creation includes a safety check — it verifies the referenced ClusterRole exists before creating the binding, gracefully skipping if the SCC is not available (e.g., on vanilla Kubernetes).

The ConfigMap watching mechanism is notably sophisticated. The reconciler sets up field indexers for efficient lookup of LlamaStackDistribution CRs that reference specific ConfigMaps (both user-config and CA bundle), with a manual fallback for older Kubernetes versions that don't support custom field labels. When a referenced ConfigMap changes, the operator recalculates a content-based hash and updates the Deployment's pod template annotations, triggering a rolling restart. The CA bundle handling includes PEM validation, multi-key concatenation via init containers, and auto-detection of the platform's well-known `odh-trusted-ca-bundle` ConfigMap.

A notable FIPS compliance gap exists in the Konflux Dockerfile: while `CGO_ENABLED=1` is set (ensuring dynamic linking), the `GOEXPERIMENT=strictfipsruntime` flag and `-tags strictfipsruntime,openssl` build tags present in the non-Konflux Dockerfile are missing from `Dockerfile.konflux`. This means RHOAI builds produced via Konflux will pass check-payload's binary linking check but may not use the FIPS-validated cryptographic backend. The operator itself performs minimal cryptographic operations (no custom TLS configuration, no direct crypto usage beyond stdlib), so the practical impact is limited to the Go runtime's internal crypto usage for the HTTP client used for health polling.

The distribution image resolution mechanism has a clean two-tier design: an embedded `distributions.json` provides upstream community images mapped by distribution name (starter, ollama, bedrock, remote-vllm, tgi, together, vllm-gpu), while the `RELATED_IMAGE_RH_DISTRIBUTION` environment variable overrides these with a single Red Hat-certified image when set. Users can also specify a direct image reference, bypassing the distribution name mapping entirely. This design supports both upstream development (multiple distribution options) and downstream RHOAI deployment (single certified image).
