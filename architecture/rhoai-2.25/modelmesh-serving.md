## Purpose

**Short**: Kubernetes controller that orchestrates multi-model inference serving by managing ModelMesh runtime deployments, model lifecycle, and inference service endpoints.

**Detailed**: ModelMesh Serving Controller is a controller-runtime based Go operator that manages the lifecycle of ML model serving on Kubernetes. It watches Predictor and InferenceService custom resources to register models with the ModelMesh inference engine, and watches ServingRuntime/ClusterServingRuntime custom resources to create and manage Deployments containing ModelMesh containers alongside runtime-specific inference server containers (OVMS, Triton, MLServer, TorchServe).

The controller constructs complex pod specifications dynamically, adding ModelMesh containers, optional REST proxy sidecars, model puller sidecars, and built-in adapter containers alongside user-defined runtime containers. It manages the full lifecycle: deployment creation, autoscaling (HPA), scale-to-zero, TLS configuration, etcd connectivity for model state tracking, and Prometheus metrics integration.

The component bridges the Kubernetes declarative model (CRDs) with the ModelMesh gRPC-based model management API, translating Predictor/InferenceService specifications into gRPC SetVModel/DeleteVModel calls and reflecting model loading status back to CR status fields. It supports both namespace-scoped and cluster-scoped operation modes, with namespace isolation enforced through etcd key prefixing.

## Architectural Analysis

ModelMesh Serving Controller implements a multi-controller orchestration pattern with three primary reconciliation loops that coordinate through shared state. The **ServiceReconciler** manages the cluster-internal inference Service and Prometheus ServiceMonitor, establishing the gRPC connection to ModelMesh. The **ServingRuntimeReconciler** handles the most complex logic: it constructs multi-container Deployment specs through a 14-step transformation pipeline that builds pod specifications from a Go template (`config/internal/base/deployment.yaml.tmpl`), injecting ModelMesh containers, optional REST proxy, optional puller/adapter sidecars, TLS volumes, etcd configuration, and user-defined runtime containers. The **PredictorReconciler** bridges the Kubernetes declarative world with the ModelMesh imperative gRPC API, translating Predictor CR specs into SetVModel calls and polling for model loading status with configurable requeue intervals (1s during loading, 3s while waiting for runtime).

The architecture employs a plugin system for predictor sources through the `PredictorSource` interface, enabling multiple CR types to drive model serving. The built-in sources — `PredictorCRRegistry` for native Predictor CRs and `InferenceServiceRegistry` for KServe InferenceServices — are backed by a `cachedPredictorSource` that maintains a local cache with version tracking and a min-heap deletion queue (20-second grace period) to handle race conditions between status updates and deletion events. The `InferenceServiceRegistry` performs comprehensive storage URI parsing, supporting S3, GCS, Azure Blob, PVC, and HTTP schemes with automatic parameter extraction.

A notable design decision is the use of etcd as a sideband communication channel between the controller and ModelMesh runtime pods. The `ModelMeshEventStream` watches etcd vmodel and model registries for changes, translating them into Kubernetes controller-runtime GenericEvents that trigger PredictorReconciler reconciliation. This enables the controller to react to model loading status changes without polling the ModelMesh gRPC API. The etcd connection configuration is propagated from the controller namespace to application namespaces via Secret copies with namespace-specific root prefix transformation (`{prefix}/mm_ns/{namespace}`), providing multi-tenant isolation within a shared etcd cluster.

The deployment template (`deployment.yaml.tmpl`) reveals an interesting security architecture: the base template includes an `oauth-proxy` sidecar container that fronts the REST proxy on port 8443 with OpenShift OAuth authentication and SAR-based authorization. This provides endpoint-level access control for REST inference requests, checking that the caller has `services/get` permission in the serving namespace. The gRPC path (port 8033) uses a different authentication model based on optional mTLS client certificates configured through the TLS Secret. This dual-auth design reflects the different consumption patterns: REST for interactive users (OAuth) and gRPC for service-to-service communication (mTLS).

The scale-to-zero feature implements a grace-period-based idle detection mechanism in the ServingRuntimeReconciler. When a runtime has no matching Predictors, the controller waits for a configurable grace period (default 60 seconds) before scaling the Deployment to zero replicas. The controller tracks runtime-to-predictor mappings by scanning model type labels against Predictor specs, and uses requeue-based polling to check if the grace period has elapsed. When combined with HPA autoscaling, the controller coordinates between the two scaling mechanisms: it sets minimum replicas to 0 when scale-to-zero is active, and delegates to HPA for scale-up based on CPU utilization metrics.
