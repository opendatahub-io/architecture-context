## Purpose

**Short**: TrustyAI is a Quarkus-based REST service providing AI model fairness metrics, explainability algorithms, and data drift detection for models served via KServe and ModelMesh on OpenShift AI.

**Detailed**: TrustyAI Explainability is a Java library and microservice that delivers Explainable AI (XAI) capabilities to the Red Hat OpenShift AI platform. It intercepts inference payloads from KServe and ModelMesh model serving infrastructure, stores them in configurable backends (PVC flat-file, MinIO object storage, or MariaDB/MySQL database via Hibernate), and computes fairness metrics (Statistical Parity Difference, Disparate Impact Ratio), data drift metrics (KS Test, Fourier MMD, Meanshift), and local/global explainability results (LIME, SHAP, Counterfactual, Partial Dependence Plot, Time-Series Saliency).

The service exposes a comprehensive REST API for on-demand metric computation, scheduled metric publishing to Prometheus, inference data upload/download, and model metadata management. It integrates with the KServe V2 Inference Protocol via gRPC for model invocation during explainability computations. Scheduled metric results are published as Prometheus gauges with the `trustyai_` prefix, scraped by a ServiceMonitor at 4-second intervals. The service supports Knative CloudEvents for receiving KServe inference payloads and a direct REST endpoint for ModelMesh partial payloads.

The codebase is organized as a Maven multi-module project with four modules: `explainability-core` (fairness metrics, XAI algorithms, data structures), `explainability-connectors` (KServe/ModelMesh gRPC integration via protobuf), `explainability-arrow` (Apache Arrow interop for Python bridge), and `explainability-service` (Quarkus REST service with storage, scheduling, and Prometheus integration).

## Architectural Analysis

TrustyAI's architecture follows a clean separation between its XAI algorithm library (`explainability-core`), protocol adapters (`explainability-connectors`), and the service layer (`explainability-service`). This modularity allows the core algorithms to be used independently as a Java library (published to Maven Central) while the service wraps them in a Quarkus microservice with storage, scheduling, and observability. The `explainability-arrow` module exists specifically to enable interop with the TrustyAI Python library via Apache Arrow IPC format, though it is excluded from the RHOAI service build profile (`service-minimal` profile omits it).

The storage architecture is notable for its pluggable backend design using Quarkus `@LookupIfProperty` CDI qualifiers. The three backends — PVC flat-file (CSV), MinIO object storage, and Hibernate/database — share a common `StorageInterface` but have vastly different capabilities. PVC and MinIO backends use CSV serialization with a parser layer, while the database backend uses JPA entities (`DataframeRow`) with direct SQL queries. The MinIO backend explicitly does not support batch reads or tag/ID filtering (throwing `StorageReadException`), which is an important limitation. The database backend is the most capable, supporting batch reads, range queries, tag filtering, and efficient counting operations. A migration path exists (`HibernateMigrator`) for transitioning from PVC to database storage.

The inference payload reconciliation pattern is architecturally significant. Both KServe and ModelMesh payloads arrive as separate input/output halves that must be matched by inference ID. The `InferencePayloadReconciler` abstract class implements a two-phase pattern: each half is stored in a `PayloadStorage` (memory or database-backed), and when both halves for an ID arrive, they are reconciled into a complete `Dataframe` and persisted. The ModelMesh reconciler also performs model ID normalization (stripping `__isvc` suffixes) and synthetic data detection (via `BIAS_IGNORE_PARAM`), which are important for accurate metric computation.

The Prometheus integration is tightly coupled to the scheduled metric computation loop. `PrometheusScheduler` runs on a configurable cron (default 5s), iterates over all registered metric requests, loads the relevant organic dataframe, computes the metric, and publishes the result as a Micrometer gauge. This means metric freshness is bounded by the schedule interval and the batch size. The `trustyai_*` gauge names include tags for model, request UUID, and optional subcategories, enabling per-model per-metric Prometheus queries. The ServiceMonitor scrapes at 4s intervals, tighter than the default 5s computation schedule, ensuring minimal lag.

Authentication is entirely delegated to infrastructure. The service itself implements no auth — all endpoints are open at the application level. In production, the TrustyAI Operator injects a `kube-rbac-proxy` or `oauth-proxy` sidecar that intercepts traffic before it reaches the Quarkus application. This is a deliberate architectural choice that keeps the service code clean but means the service is completely unprotected if deployed without the operator's sidecar injection. The application properties configure TLS certificates mounted at `/etc/tls/internal/` on port 4443, suggesting the operator provisions internal TLS for the sidecar-to-application channel.

The Konflux build pipeline on this downstream branch is fully hermetic. Unlike the upstream Dockerfile which compiles from source using `mvn clean package`, the `Dockerfile.konflux` extracts pre-built JARs from PNC (Product Newcastle) archives that are prefetched by Hermeto. This means the downstream RHOAI build never executes `mvn` — it simply layers pre-validated, pre-compiled artifacts into the UBI9 runtime image. RPM packages, Java artifacts, and generic dependencies are all locked with checksums (`rpms.lock.yaml`, `artifacts.lock.yaml`), providing full build reproducibility across the four supported architectures (x86_64, aarch64, ppc64le, s390x).
