## Purpose

**Short**: Kueue is a Kubernetes-native job queueing system that manages when workloads should be admitted to start based on quota, priority, and fair sharing policies.

**Detailed**: Kueue provides a set of APIs and controllers for job-level resource management in Kubernetes clusters. It introduces the concept of ResourceFlavors (representing different hardware types or pricing models), ClusterQueues (cluster-scoped quota pools), and LocalQueues (namespace-scoped submission points) to control when jobs should be admitted to run based on available resources, priority, and fair sharing policies.

Kueue operates as a controller-runtime based operator that watches multiple job types (Kubernetes batch Jobs, Kubeflow training jobs, Ray jobs, JobSets, CodeFlare AppWrappers, Pods, Deployments, StatefulSets, and LeaderWorkerSets) and creates Workload objects to represent their resource requirements. The scheduler component evaluates pending Workloads against ClusterQueue quotas and admission checks before admitting them. Kueue supports advanced features including quota borrowing/lending between cohorts, preemption policies, partial admission, topology-aware scheduling (TAS), multi-cluster workload distribution (MultiKueue), and provisioning request integration with cluster autoscalers.

In the RHOAI distribution, Kueue is deployed to the `opendatahub` namespace with RHOAI-specific feature gate configuration, OpenShift namespace exclusions for webhooks, Prometheus alerting rules, and network policies. It integrates with the broader RHOAI platform to provide workload scheduling for AI/ML training and inference pipelines.

## Architectural Analysis

Kueue follows a layered controller architecture built on controller-runtime with three distinct tiers of controllers. The **core controllers** (WorkloadReconciler, ClusterQueueReconciler, LocalQueueReconciler, ResourceFlavorReconciler, AdmissionCheckReconciler, CohortReconciler) manage the lifecycle of Kueue's own CRDs, maintaining an in-memory cache (`pkg/cache`) and queue manager (`pkg/queue`) that the scheduler queries for admission decisions. The **feature controllers** (Provisioning, MultiKueue, TAS) handle optional capabilities behind feature gates, each watching additional external CRDs. The **job framework controllers** provide a generic reconciler (`pkg/controller/jobframework/reconciler.go`) that integrates with 14+ workload types through a callback interface (`IntegrationCallbacks`), where each integration registers itself via Go `init()` functions.

The scheduler (`pkg/scheduler`) operates as a separate goroutine added to the controller manager. It pulls pending workloads from the queue manager, assigns them to ClusterQueues based on resource availability and flavor matching, and handles preemption logic including within-queue, within-cohort, and fair sharing preemption strategies. The separation of Workload objects from the actual job objects is a key design decision — it allows Kueue to manage quota without tightly coupling to each job type's lifecycle. The `LosslessDefaulter` wrapper in the webhook framework (`pkg/controller/jobframework/webhook/defaulter.go`) is a notable pattern that prevents accidentally removing fields from job CRDs that Kueue doesn't know about, ensuring forward compatibility with newer CRD versions.

The RHOAI overlay introduces several production-hardening measures not present in the upstream configuration. A NetworkPolicy restricts webhook ingress to TCP port 9443, preventing unauthorized access to the admission webhook endpoint. PrometheusRule alerts monitor four conditions: pod liveness (KueuePodDown, critical), low resource utilization (LowClusterQueueResourceUsage, info), over-reservation (ResourceReservationExceedsQuota, info), and long-pending workloads (PendingWorkloadPods, info). The RHOAI overlay also customizes webhook namespace selectors to exclude OpenShift system namespaces (kube-system, openshift-*, etc.) and explicitly enables `LocalQueueMetrics` while disabling `VisibilityOnDemand` and `MultiKueue`.

Certificate management follows a dual-path design: internal cert management (default) uses `pkg/util/cert` to self-sign and rotate certificates without external dependencies, while external cert-manager integration is available as an alternative. The readiness probe explicitly gates on certificate availability — the controller reports as not-ready until webhook certificates are in place, ensuring no admission requests are dropped during startup. The metrics endpoint uses a separate TLS certificate path (`/etc/kueue/metrics/certs`) from the webhook endpoint (`/tmp/k8s-webhook-server/serving-certs`), and both support dynamic certificate rotation via certwatcher.

The controller concurrency configuration reveals intentional design choices: high-throughput resources (Job, Pod, Workload) use concurrency of 5, while control-plane resources (ClusterQueue, LocalQueue, ResourceFlavor, Cohort) use concurrency of 1, ensuring serialized reconciliation of quota-affecting changes. Client-side rate limiting is configured at 50 QPS with burst of 100, balancing responsiveness with API server load.
