AUTHENTICATION & AUTHORIZATION DETAILS:

Notebook Access (OAuth Enabled):
  1. User → OpenShift Router: HTTPS (TLS 1.3), no auth
  2. Router → OAuth Proxy: HTTPS (TLS 1.2+), no auth
  3. OAuth Proxy redirects unauthenticated users to OpenShift OAuth Server
  4. After OAuth login, proxy performs SubjectAccessReview (SAR):
     - Verb: get
     - Resource: notebooks (kubeflow.org API group)
     - ResourceName: {notebook-name}
     - Namespace: {namespace}
  5. If SAR passes, proxy forwards requests to Jupyter on port 8888 (HTTP)
  6. OAuth proxy maintains session via encrypted cookie

Notebook Access (OAuth Disabled):
  1. User → Route → Service (port 8888) directly
  2. No authentication enforced by infrastructure
  3. Application-level authentication recommended (e.g., Jupyter token)

Webhook Admission:
  - Only Kubernetes API server can call webhook
  - mTLS client certificate authentication
  - Webhook validates requests using AdmissionReview structure

Controller API Access:
  - ServiceAccount token authentication
  - RBAC enforced by Kubernetes API server
  - ClusterRole permissions defined above
