NETWORK FLOWS
═══════════════════════════════════════════════════════════════════════════

Flow 1: RayCluster Creation
  User → Kubernetes API (443/TCP HTTPS TLS1.2+ Bearer Token)
  Kubernetes API → kuberay-operator (Controller Watch, ServiceAccount Token)
  kuberay-operator → Kubernetes API (443/TCP HTTPS TLS1.2+ ServiceAccount Token)
  Kubernetes Scheduler → Container Registry (443/TCP HTTPS TLS1.2+ Pull Secrets)
  Ray Worker Pods → Ray Head Pod GCS (6379/TCP Ray Protocol, Optional TLS, Optional mTLS)

Flow 2: RayJob Submission
  User → Kubernetes API (443/TCP HTTPS TLS1.2+ Bearer Token)
  kuberay-operator → Kubernetes API (443/TCP HTTPS ServiceAccount Token)
  Job Submitter Pod → Ray Head Client (10001/TCP gRPC, Optional TLS, Optional Token)
  Job Submitter Pod → Ray Head GCS (6379/TCP Ray Protocol, Optional TLS, Optional mTLS)

Flow 3: Ray Serve Inference
  Client → Ingress/Route (443/TCP HTTPS TLS1.2+ Optional Bearer)
  Ingress/Route → Ray Serve Service (8000/TCP HTTP/HTTPS, Optional TLS, Optional Bearer)
  Ray Serve Service → Ray Serve Replica Pods (8000/TCP HTTP/HTTPS, Optional TLS)
  Ray Serve Replica → Ray Head GCS (6379/TCP Ray Protocol, Optional TLS, Optional mTLS)

Flow 4: Metrics Collection
  Prometheus → kuberay-operator (8080/TCP HTTP, No encryption, Network Policy)
  Prometheus → Ray Dashboard (8265/TCP HTTP/HTTPS, Optional TLS, Optional Token)

═══════════════════════════════════════════════════════════════════════════
HIGH AVAILABILITY
═══════════════════════════════════════════════════════════════════════════

Operator:
  - Leader election enabled (single active replica)
  - coordination.k8s.io/leases for leader election
  - Single replica with automatic failover

Ray Clusters:
  - Ray Head is single point of failure by default
  - GCS fault tolerance available with external Redis (optional)
  - RayService: Zero-downtime upgrades via dual-cluster promotion

Autoscaling:
  - Ray Autoscaler: In-tree autoscaler sidecar on Ray head pod
  - Scales worker groups based on resource demands
  - Eventually consistent autoscaling

═══════════════════════════════════════════════════════════════════════════
KNOWN SECURITY LIMITATIONS
═══════════════════════════════════════════════════════════════════════════

1. Ray Head Single Point of Failure:
   - Without external Redis GCS fault tolerance, Ray head pod failure requires
     cluster recreation

2. No Multi-Tenancy Isolation:
   - Ray clusters in same namespace can potentially interfere if resource
     quotas not set

3. TLS Configuration Complexity:
   - Requires manual certificate provisioning and configuration via init
     containers

4. Limited Network Policy Support:
   - No default NetworkPolicy manifests provided
   - Users must create custom network policies for isolation

5. Autoscaler Over-Provisioning:
   - Ray autoscaler is eventually consistent and may over-provision during
     rapid scaling events
