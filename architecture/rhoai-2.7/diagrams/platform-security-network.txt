AUTHENTICATION MECHANISMS
════════════════════════════════════════════════════════════════════════════════════════════════════════

1. Bearer Tokens (JWT) from OpenShift OAuth
   - Components: ODH Dashboard, Notebooks, DSP UI, TrustyAI
   - Enforcement: OAuth Proxy sidecar (validates via Kubernetes API)
   - Flow: User → OAuth → Bearer Token → OAuth Proxy → SubjectAccessReview (SAR)

2. mTLS Client Certificates (Istio)
   - Components: KServe InferenceServices, ModelMesh (optional)
   - Enforcement: Istio service mesh (Envoy sidecar)
   - Mode: STRICT for predictor pods, PERMISSIVE for metrics ports

3. ServiceAccount Tokens
   - All operators authenticate to Kubernetes API using ServiceAccount tokens
   - Enforcement: Kubernetes RBAC via API server
   - Mounted at: /var/run/secrets/kubernetes.io/serviceaccount/token

4. Kubernetes Webhook mTLS
   - Components: All validating/mutating webhooks (KServe, Notebook, ModelMesh, TrustyAI)
   - Enforcement: Kubernetes API server validates client certificate
   - Certs: Managed by cert-manager (90-day rotation)

5. AWS IAM / S3 Signature V4
   - Components: KServe, ModelMesh, Data Science Pipelines
   - Enforcement: storage-initializer, runtime adapter
   - Methods: IRSA (IAM Roles for Service Accounts), static credentials

6. Subject Access Review (SAR)
   - OAuth proxies delegate authorization to Kubernetes RBAC
   - API: authorization.k8s.io/v1/subjectaccessreviews
   - Validates: User can perform action on resource

7. Optional Token Auth
   - KServe/ModelMesh inference endpoints can require Bearer tokens
   - Application-level validation (not enforced by platform)

NETWORK POLICIES
════════════════════════════════════════════════════════════════════════════════════════════════════════

ODH Model Controller creates 3 NetworkPolicies per InferenceService namespace:

1. allow-from-istio-ingressgateway
   - Ingress: Allow traffic from istio-system namespace (istio-ingressgateway)
   - Pods: All pods in namespace
   - Ports: All

2. allow-from-knative-serving
   - Ingress: Allow traffic from knative-serving namespace
   - Pods: Predictor pods (label: serving.kserve.io/inferenceservice)
   - Ports: 8080/TCP, 8443/TCP

3. allow-egress-to-external-storage
   - Egress: Allow traffic to external S3/GCS/Azure endpoints
   - Pods: Predictor pods with storage-initializer
   - Ports: 443/TCP (HTTPS)

Data Science Pipelines Operator:
  - Creates NetworkPolicies for DSPA namespace isolation
  - Allows: API server, persistence agent, database, Minio internal communication
  - Denies: Cross-namespace traffic (except explicitly allowed)

Default OpenShift NetworkPolicies:
  - allow-from-openshift-ingress: Allows traffic from OpenShift Router
  - allow-from-openshift-monitoring: Allows Prometheus scraping

DATA PROTECTION
════════════════════════════════════════════════════════════════════════════════════════════════════════

Encryption in Transit:
  - External: TLS 1.2+ for all ingress (HTTPS via OpenShift Routes)
  - Internal (Service Mesh): mTLS STRICT for KServe predictor traffic
  - Internal (non-mesh): HTTP (Kubernetes network policies provide isolation)
  - Webhooks: TLS 1.2+ with cert-manager managed certificates
  - Storage: HTTPS to S3/GCS/Azure with TLS 1.2+

Encryption at Rest:
  - Secrets: Encrypted by etcd (OpenShift platform encryption)
  - PVCs: Depends on storage class configuration (CSI driver encryption)
  - Model artifacts: S3-side encryption (SSE-S3, SSE-KMS, SSE-C)
  - Database: MariaDB encryption depends on PVC encryption

Certificate Management:
  - cert-manager: Manages webhook TLS certificates (90-day rotation)
  - service-ca-operator: Generates service serving certificates
  - Istio: Manages mTLS certificates for service mesh (24-hour rotation)

Secret Management:
  - Kubernetes Secrets (Opaque): Passwords, API keys, S3 credentials
  - External Secrets Operator: Optional integration for external secret stores
  - Sealed Secrets: Optional for GitOps workflows
