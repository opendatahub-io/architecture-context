SECURITY NOTES
═══════════════════════════════════════════════════════════════════════════

1. Pod Security Context:
   - runAsUser: 65532 (non-root)
   - runAsGroup: 65532
   - allowPrivilegeEscalation: false
   - capabilities: ALL dropped
   - FIPS 140-2 compliance via strictfipsruntime build tag

2. Network Isolation:
   - Ray worker pods isolated to same-cluster traffic only (NetworkPolicy)
   - Ray head pods allow specific ingress from KubeRay, monitoring, and same-namespace clients
   - OAuth proxy provides secure external access on OpenShift

3. OpenShift vs Kubernetes:
   - OpenShift: OAuth proxy enforces authentication for dashboard access
   - Vanilla Kubernetes: NO authentication on dashboard by default (security gap)
   - Users on vanilla K8s MUST implement their own authentication/ingress controls

4. mTLS Configuration:
   - Configurable mTLS for Ray internal cluster communication
   - When enabled: CA cert auto-generated, client certs required for Ray client connections
   - When disabled: Ray communication is unencrypted (plaintext)

5. Certificate Management:
   - Webhook certs: Automatically rotated by cert-controller
   - Ray cluster certs: Static, generated at cluster creation (no rotation)
   - OAuth proxy certs: Static, generated at cluster creation (no rotation)

6. Known Security Limitations:
   - AppWrapper controller requires Kueue at startup; no dynamic detection
   - OAuth dashboard access only on OpenShift; vanilla K8s has no default auth
   - Single-replica operator (no HA by default)
   - Ray cluster certificates (CA, TLS) do not auto-rotate
