RBAC Summary:
=============
- vLLM CPU does NOT define its own Kubernetes RBAC resources
- Runs as non-root user (UID 2000, GID 0)
- Pod RBAC managed by KServe and RHOAI operator
- kube-rbac-proxy sidecar enforces authorization via SubjectAccessReview

Container Security:
===================
- User: 2000 (non-root)
- Group: 0 (root group, OpenShift arbitrary UID support)
- VLLM_NO_USAGE_STATS=1 (telemetry disabled)
- VLLM_USAGE_SOURCE=production-docker-image
- FIPS-compliant build

Secrets:
========
- (model-serving-secret) [Opaque] - HF token or S3 credentials, User/Admin provisioned, No auto-rotate
- (ssl-certs) [kubernetes.io/tls] - TLS certs for --ssl-certfile/--ssl-keyfile, cert-manager or manual
- (api-key) [Opaque] - Bearer token(s) for --api-key, User/Admin provisioned, No auto-rotate

Authentication Summary:
=======================
- /v1/* endpoints: Bearer Token (SHA-256 compared), enforced by AuthenticationMiddleware
  - Only active when --api-key is provided at startup
  - Returns 401 on token mismatch; skips for non-/v1 paths and OPTIONS
- /health, /metrics, /ping: Public (no auth required)
- Platform ingress: kube-rbac-proxy enforces RBAC-based authorization

Network Notes:
==============
- Pod-internal communication: HTTP plaintext on localhost (trusted boundary)
- Inter-process: ZeroMQ (PyZMQ) for V1 engine multi-process arch
- No gRPC services exposed
- No service mesh sidecar (Istio) by default in RHOAI 3.x
- TLS termination at Platform Gateway layer
- Disconnected operation supported (HF_HUB_OFFLINE=0, models via PVC)
