RBAC Summary (kube-rbac-proxy mode):
  Required API Access (delegated to deploying component's ServiceAccount):
  - API Group: authorization.k8s.io
    Resources: subjectaccessreviews
    Verbs: create
    Purpose: Perform RBAC authorization checks

  - API Group: authentication.k8s.io
    Resources: tokenreviews
    Verbs: create
    Purpose: Validate bearer tokens via delegated authentication

  Note: kube-auth-proxy does not define its own ClusterRoles.
  RBAC is managed by the deploying RHOAI component.

Secrets:
  - kube-auth-proxy-secret (Opaque)
    Contents: client-secret, cookie-secret (OIDC/OAuth)
    Provisioned By: Admin/Operator
    Auto-Rotate: No

  - TLS certificates (kubernetes.io/tls)
    Purpose: TLS termination for HTTPS listener
    Provisioned By: cert-manager/Admin
    Auto-Rotate: Configurable (kube-rbac-proxy watches for cert changes)

  - Service account token (kubernetes.io/service-account-token)
    Purpose: OpenShift OAuth auto-configuration (service account mode)
    Provisioned By: Kubernetes
    Auto-Rotate: Yes

FIPS Compliance:
  - Build: CGO_ENABLED=1 GOEXPERIMENT=strictfipsruntime -tags strictfipsruntime
  - Base Image: registry.access.redhat.com/ubi9/ubi-minimal
  - Build Image: registry.access.redhat.com/ubi9/go-toolset:1.25
  - Validation: Automated check-payload FIPS compliance verification
  - Crypto: FIPS-approved cryptographic libraries only

Authentication Mechanisms:
  Auth Mode:
  - OIDC Bearer Token
  - OAuth Cookie Session
  - Basic Auth (htpasswd)
  - Supports email domain restriction and group filtering

  RBAC Mode:
  - Bearer Token (TokenReview)
  - Client Certificate (X.509)
  - OIDC JWT
  - SubjectAccessReview against Kubernetes API

Deployment Model:
  - Sidecar proxy container (not standalone)
  - Deployed alongside RHOAI platform service pods
  - Single container image with dual binary (entrypoint multiplexer)
  - CPU Architectures: amd64, arm64, ppc64le, s390x
