Service Mesh Configuration
═════════════════════════════════════════════════════════════════════════

Service Mesh: Not enforced at pod level
  - Dashboard does not require Istio sidecars
  - Uses OpenShift Routes for ingress (not Istio Gateway)
  - Internal communication via ClusterIP services (plain HTTP on localhost)
  - OAuth and TLS handled by kube-rbac-proxy, not service mesh

Authentication Flow:
  1. User → OpenShift Router (TLS 1.2+)
  2. Router → kube-rbac-proxy (TLS Reencrypt)
  3. kube-rbac-proxy → OpenShift OAuth (OAuth validation)
  4. kube-rbac-proxy → Kubernetes API (TokenReview)
  5. kube-rbac-proxy → Backend (HTTP localhost with impersonation headers)
  6. Backend → Kubernetes API (ServiceAccount Token + Impersonation)

User Impersonation:
  - kube-rbac-proxy validates user OAuth token
  - Adds impersonation headers: X-Auth-Request-User, X-Auth-Request-Groups
  - Backend uses ServiceAccount + impersonation to act on behalf of user
  - Kubernetes RBAC enforced based on impersonated user identity
