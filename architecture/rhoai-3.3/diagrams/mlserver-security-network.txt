RBAC Summary:
================================================================================
- No RBAC roles defined by MLServer
- MLServer runs as a non-privileged container within KServe-managed pods
- RBAC is managed by KServe and the OpenShift AI operator
- Authentication enforced at ingress layer (kube-rbac-proxy / Gateway API)

Container Security:
================================================================================
- Run as non-root: Yes (UID 1000)
- User ID: 1000 (Dockerfile.konflux line 27-28)
- World-writable workdir: Yes (mode 776) for OpenShift random UID compatibility
- Base Image: quay.io/aipcc/base-images/cpu:3.2.0-*
- gRPC binding: add_insecure_port() - NO TLS at server level

Secrets:
================================================================================
- No secrets defined directly by MLServer
- Model storage credentials (S3, etc.) are injected by KServe via:
  - Environment variables
  - Mounted secrets

Authentication & Authorization:
================================================================================
- MLServer itself does NOT implement authentication or authorization
- All REST/gRPC endpoints are unauthenticated at the server level
- Authentication enforcement points:
  1. KServe Gateway / Gateway API HTTPRoute (TLS termination, token validation)
  2. kube-rbac-proxy sidecar (RHOAI 3.x - RBAC-based access control)
- gRPC server uses add_insecure_port() for binding (no TLS)
