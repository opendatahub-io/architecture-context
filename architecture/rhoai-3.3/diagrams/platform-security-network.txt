NOTES
═══════════════════════════════════════════════════════════════════════════════════════

1. Trust Boundaries:
   - EXTERNAL: Untrusted user browsers, internet
   - INGRESS: OpenShift Router, Gateway API (TLS termination)
   - SERVICE MESH: Optional Istio with mTLS PERMISSIVE/STRICT
   - USER WORKLOADS: Namespace-isolated with NetworkPolicies
   - EXTERNAL SERVICES: S3, databases, external APIs (TLS encrypted)

2. Authentication Layers:
   - Layer 1: OpenShift OAuth (Bearer tokens) for user-facing services
   - Layer 2: kube-rbac-proxy / oauth-proxy for Kubernetes RBAC validation
   - Layer 3: ServiceAccount tokens for operator-to-K8s-API communication
   - Layer 4: AWS IAM, DB credentials, API keys for external services
   - Layer 5 (optional): Istio mTLS for service-to-service communication

3. Encryption Standards:
   - External: TLS 1.2+ for all HTTPS endpoints
   - Internal: TLS 1.2+ for webhooks, metrics (kube-rbac-proxy)
   - Database: TLS 1.2+ with sslmode=require (PostgreSQL), TLS for MySQL
   - Service Mesh: mTLS 1.2+ (Istio)
   - Storage: TLS 1.2+ for S3 API, server-side encryption (SSE-S3/SSE-KMS)

4. FIPS Compliance:
   - All Go operators built with strictfipsruntime
   - TLS ciphers: FIPS 140-2 approved (AES-GCM, SHA-256)
   - Container images scanned via Konflux CI/CD

5. Multi-Tenancy:
   - Namespace isolation: Each user/team gets dedicated namespace(s)
   - RBAC: RoleBindings scoped to user namespaces
   - NetworkPolicies: Prevent cross-namespace traffic (except platform services)
   - ResourceQuotas: Limit compute/storage per namespace (optional)

6. Audit Logging:
   - Kubernetes Audit Logs: All API calls logged (create, update, delete CRs)
   - OpenShift OAuth Audit: User authentication/authorization events
   - Istio Access Logs: Service mesh traffic logs (optional)
   - Application Logs: Component-specific logs (operator reconciliations, etc.)
