## Purpose

**Short**: A Go CLI tool that extracts, enriches, and catalogs metadata from Red Hat AI container images, producing a static data container used by downstream RHOAI components.

**Detailed**: Model Metadata Collection is a build-time data pipeline that discovers AI model containers from OCI registries (primarily `registry.redhat.io`) and HuggingFace collections, extracts model card content from container image layers, enriches the metadata with information from the HuggingFace API (model details, tags, README frontmatter), and generates structured YAML catalogs (`models-catalog.yaml`, `validated-models-catalog.yaml`).

The tool processes models concurrently, supports multiple input formats (OCI registry references, HuggingFace model links, static YAML catalogs), and implements a multi-source metadata enrichment pipeline with strict priority ordering: modelcard.md YAML frontmatter > modelcard.md regex extraction > HuggingFace YAML frontmatter > HuggingFace API/tags > generated defaults. The output catalogs include per-model metadata (name, provider, description, license, tasks, language, artifacts with OCI URIs and timestamps) along with quality tracking through separate enrichment.yaml files. A companion `metadata-report` CLI generates completeness and data-source reports.

The final container image (built via Konflux) contains only the pre-generated catalog YAML files — no Go binary is shipped. The container serves as a static data volume that other RHOAI components (such as the dashboard) mount to access the validated model catalog at runtime. This is a "data-only" container pattern: `CMD ["sleep", "infinity"]` keeps the container alive while consumers read from `/app/data/`.

## Architectural Analysis

The model-metadata-collection component follows a "data-only container" pattern that is uncommon but well-suited to its purpose. The Go CLI tools (`model-extractor` and `metadata-report`) exist solely as build-time utilities — they are never shipped in the production container image. Instead, the CI/CD pipeline (via GitHub Actions workflow `build-and-push-static-model-catalog-data.yml`) runs these tools to generate catalog YAML files, which are committed back to the repository. The Konflux Dockerfile then simply copies these pre-generated files into a minimal UBI9 container. This design separates the complex extraction/enrichment logic from the runtime artifact, resulting in a tiny, secure production image with no attack surface beyond the base UBI9-minimal.

The metadata enrichment pipeline implements a sophisticated multi-source data fusion strategy with explicit priority ordering and source tracking. Each metadata field (name, provider, description, license, tasks, etc.) records its provenance through an `enrichment.yaml` sidecar file, enabling auditing of where each piece of data originated. The priority hierarchy — modelcard YAML > modelcard regex > HuggingFace YAML frontmatter > HuggingFace API > HuggingFace tags > generated defaults — ensures that data closest to the model author takes precedence. The system also implements intelligent model-family matching (`isCompatibleModelFamily`) to prevent cross-contamination between unrelated models (e.g., preventing a Llama container from matching a Granite HuggingFace entry). Model name quality validation (`isLowQualityModelName`) catches document titles and code artifacts that regex extraction sometimes misidentifies as model names.

The concurrent processing architecture uses a semaphore-based pattern (`maxConcurrent` flag, default 5) to limit parallel model processing. Each model goes through a pipeline of: fetch OCI manifest → scan layers for modelcard annotation → extract tar/gzip content → parse YAML frontmatter and regex patterns → fetch HuggingFace enrichment → generate structured metadata. The `containers/image/v5` library handles OCI registry interactions including manifest list parsing for multi-architecture support (the Tekton pipeline builds for `linux/x86_64`, `linux/ppc64le`, `linux/s390x`, and `linux-m2xlarge/arm64`).

One notable design decision is the "skeleton metadata" pattern: when modelcard extraction fails (no modelcard layer found in the container image), the tool creates a minimal `metadata.yaml` file and attempts to fetch the HuggingFace README as a fallback modelcard. This ensures the enrichment pipeline can still populate metadata from external sources, maintaining catalog completeness even for models that don't embed model cards in their container layers. The model type classification system (`generative`, `predictive`, `unknown`) defaults to `generative` for all models, reflecting the current RHOAI catalog's focus on LLMs.
