## Purpose
**Short**: A FIPS-compliant authentication and authorization proxy that handles OIDC, OpenShift OAuth authentication, and Kubernetes RBAC authorization for ODH/RHOAI platform services.

**Detailed**: kube-auth-proxy is a consolidated authentication and authorization proxy designed specifically for OpenShift Data Hub (ODH) and Red Hat OpenShift AI (RHOAI) environments. It is derived from oauth2-proxy and has been streamlined to support only OIDC and OpenShift OAuth authentication providers, with a strong emphasis on FIPS compliance. The proxy serves as a drop-in replacement for existing oauth-proxy sidecars in the RHOAI platform.

The container image ships two binaries in a single image, selectable at runtime via an entrypoint multiplexer: **kube-auth-proxy** (authentication-focused, derived from oauth2-proxy) handles OIDC and OpenShift OAuth authentication flows including login, callback, session management, and token validation. **kube-rbac-proxy** (authorization-focused, derived from brancz/kube-rbac-proxy v0.19.1) performs Kubernetes RBAC authorization via SubjectAccessReview against the Kubernetes API. The entrypoint binary selects between these two modes based on the `PROXY_MODE` environment variable or first command-line argument (`auth` or `rbac`, defaulting to `rbac`).

This dual-binary design allows the same container image to serve different authentication/authorization roles across the RHOAI platform: components requiring full OIDC/OAuth authentication flows use the `auth` mode, while components needing Kubernetes RBAC-based access control (e.g., metrics endpoints) use the `rbac` mode. Both binaries are built with FIPS-compliant compilation flags (`GOEXPERIMENT=strictfipsruntime`) for enterprise deployment.

### Ingress

kube-auth-proxy does not define its own Ingress or Route resources. It is deployed as a sidecar proxy container within pods managed by RHOAI platform controllers. Ingress to the auth proxy is managed by the deploying component (e.g., via HTTPRoute resources created by component controllers that reference the RHOAI Gateway API data-science-gateway, or via OpenShift Routes).

**Envoy ext_authz Integration**: The `/oauth2/auth` endpoint supports Envoy's external authorization framework, enabling integration with RHOAI 3.x Gateway API infrastructure where Envoy acts as the ingress gateway and delegates authentication decisions to kube-auth-proxy.
