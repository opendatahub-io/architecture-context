## Purpose

**Short**: FIPS-compliant authentication and authorization reverse proxy providing OIDC and OpenShift OAuth authentication (kube-auth-proxy) and Kubernetes RBAC-based authorization (kube-rbac-proxy) for RHOAI/ODH component pods.

**Detailed**: kube-auth-proxy is a multi-binary authentication and authorization proxy designed specifically for Red Hat OpenShift AI (RHOAI) and Open Data Hub (ODH) environments. It ships as a single container image containing three binaries: `kube-auth-proxy` (OAuth2/OIDC authentication proxy derived from oauth2-proxy), `kube-rbac-proxy` (Kubernetes RBAC authorization proxy derived from brancz/kube-rbac-proxy), and an `entrypoint` binary that selects between the two at runtime via the `PROXY_MODE` environment variable.

The kube-auth-proxy component handles authentication (authn) — it intercepts requests, validates user identity via OAuth2/OIDC or OpenShift OAuth flows, manages encrypted session cookies, and proxies authenticated requests to upstream services. It supports both browser-based OAuth2 flows and API-oriented bearer token authentication (including OpenShift `sha256~` tokens and JWT tokens). The kube-rbac-proxy component handles authorization (authz) — it validates requests using Kubernetes SubjectAccessReview (SAR) for RBAC enforcement, with support for OIDC JWT authentication, X.509 client certificates, and delegating authentication via TokenReview.

This component is deployed as a sidecar container in RHOAI 3.x component pods, replacing the previous oauth-proxy sidecars with a FIPS-compliant, feature-focused alternative. The dual-binary architecture allows the platform operator to select either authentication-only mode (for components behind Gateway API with Envoy ext_authz) or RBAC-authorization mode (for components requiring Kubernetes-native access control).

## Architectural Analysis

The kube-auth-proxy repository consolidates two distinct proxy components (OAuth authentication and Kubernetes RBAC authorization) into a single container image with a runtime binary selector. This dual-binary design is architecturally significant for RHOAI 3.x: the platform operator can deploy the same image in two different modes depending on the component's authentication needs. Components behind the Gateway API ingress stack use `PROXY_MODE=auth` for OAuth/OIDC flows compatible with Envoy ext_authz, while components requiring direct Kubernetes RBAC enforcement (e.g., metrics endpoints) use `PROXY_MODE=rbac` (the default).

The kube-auth-proxy (OAuth) component implements a middleware chain architecture using `justinas/alice` for composable request processing. The chain runs in three phases: pre-auth (scope injection, health checks, metrics), session loading (bearer tokens, basic auth, stored sessions), and headers (identity header injection). Session management is notably sophisticated — it supports distributed locking for concurrent session refresh, preventing token refresh stampedes in multi-replica deployments. The session store is pluggable, supporting both cookie-based (encrypted AES-GCM) and Redis-backed backends.

The kube-rbac-proxy component implements a union authorizer pattern with three stacked authorizers evaluated in order: a hardcoded metrics authorizer (allowing the `system:serviceaccount:openshift-monitoring:prometheus-k8s` service account to GET /metrics without explicit RBAC), a static authorizer (rules from a YAML config file), and a Kubernetes SubjectAccessReview delegator with caching (5-minute allow cache, 30-second deny cache). This layered approach means most metrics scraping requests are authorized locally without hitting the Kubernetes API, while application-specific authorization is delegated to native Kubernetes RBAC. The kube-rbac-proxy also supports dynamic SAR parameter injection via request URL query parameters or HTTP headers, enabling per-request namespace and resource attribute templating.

Both components emphasize FIPS compliance as a first-class requirement. All Go binaries are built with `CGO_ENABLED=1 GOEXPERIMENT=strictfipsruntime -tags strictfipsruntime` on UBI9 go-toolset, ensuring cryptographic operations use FIPS-validated OpenSSL rather than Go's pure-Go crypto. The CI pipeline includes an automated FIPS compliance check using OpenShift's check-payload tool. No non-FIPS crypto algorithms (MD5, RC4, DES) were detected in the codebase. The TLS configuration defaults to TLS 1.2 minimum with configurable cipher suites in both components.

The entrypoint binary is intentionally minimal (87 lines) — it reads `PROXY_MODE` from the environment or the first CLI argument, then exec's into the selected binary with remaining arguments forwarded. This design avoids process management complexity (no supervisor daemon) and ensures signal handling is handled directly by the chosen proxy binary. The default mode is `rbac` for backward compatibility with existing kube-rbac-proxy deployments.
