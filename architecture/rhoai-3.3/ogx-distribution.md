## Purpose

**Short**: Builds and ships a containerized Llama Stack server distribution with pre-configured providers for inference, vector I/O, safety, evaluation, and agentic workflows within the RHOAI platform.

**Detailed**: ogx-distribution produces a container image (`odh-llama-stack-core-rhel9`) that packages Meta's Llama Stack server framework with a curated set of providers for the Red Hat OpenShift AI platform. The image ships a `config.yaml` that wires up 11 Llama Stack API surfaces (inference, agents, vector_io, safety, eval, scoring, datasetio, files, batches, tool_runtime, and vector_stores) to over 30 provider implementations, many conditionally enabled via environment variables at runtime.

The repository itself contains no Kubernetes operator logic, no CRDs, and no deployment manifests. It is purely a container image build repository. The build pipeline uses a Python script (`distribution/build.py`) that invokes `llama stack list-deps` to auto-generate a `Containerfile` from a template (`Containerfile.in`), pinning all Python dependencies. The generated image runs `llama stack run` against the bundled `config.yaml` at container startup. For RHOAI, the production image is built by Konflux using `Dockerfile.konflux`, which layers onto a pre-built base image from the AIPCC pipeline with wheel releases for the target architecture.

The Llama Stack server exposes a single HTTP port (8321/TCP) with an OpenAI-compatible API surface. It connects outward to external inference backends (vLLM, AWS Bedrock, Azure OpenAI, Google Vertex AI, OpenAI, WatsonX), vector databases (Milvus, PostgreSQL/pgvector, Qdrant, FAISS), object storage (S3), a PostgreSQL database for internal state, and external evaluation/safety services (TrustyAI LM-Eval, TrustyAI RAGAS, TrustyAI FMS Orchestrator).

## Architectural Analysis

The ogx-distribution repository is architecturally unusual within the RHOAI platform because it produces a stateless application container image rather than a Kubernetes operator. The repository contains no Go code, no CRDs, no controllers, and no Kubernetes deployment manifests. Its sole purpose is to define, build, and test a Llama Stack server image with a carefully curated set of provider plugins and configuration. All Kubernetes-level deployment concerns (Services, HTTPRoutes, RBAC, sidecars) are handled externally by the platform operator.

The provider architecture is remarkably broad — the `config.yaml` wires up 30+ providers across 11 API surfaces. Critically, most providers are **conditionally enabled** via environment variables using Llama Stack's `${env.VAR:+provider_id}` syntax, which means the provider is only instantiated if the env var is set. This pattern means the image ships as a "batteries-included" distribution where operators can selectively enable cloud inference backends (vLLM, Bedrock, Azure, Vertex AI, OpenAI, WatsonX), vector databases (Milvus, pgvector, Qdrant, FAISS), and evaluation frameworks (LM-Eval, RAGAS) at deployment time without rebuilding the image.

The build pipeline has a notable dual-track design. The **upstream/CI path** generates a `Containerfile` from `Containerfile.in` using `distribution/build.py`, which calls `llama stack list-deps` to resolve all Python package dependencies and produces pip install commands with sorted, deduplicated, and version-constrained packages. The **Konflux/production path** uses `Dockerfile.konflux`, which layers onto a pre-built AIPCC base image (`quay.io/aipcc/base-images/cpu`) and installs pre-compiled wheel releases via `install-wheel-release.sh`, bypassing PyPI entirely. This two-tier approach provides fast CI iteration while maintaining supply-chain control for production builds. However, the Konflux build is explicitly non-hermetic (`hermetic: false` in the Tekton pipeline), and no lock files (rpms.lock.yaml, uv.lock, artifacts.lock.yaml) are present on this branch.

The Llama Stack server itself uses `llama-stack` from the ODH fork (`opendatahub-io/llama-stack@v0.4.1+rhai0`), not the upstream Meta version. The fork carries RHOAI-specific patches, as indicated by the `+rhai0` version suffix. PostgreSQL is a hard dependency — the `config.yaml` configures both `kv_postgres` and `sql_postgres` storage backends as the defaults for all persistent state (agent state, inference store, metadata registry, file metadata). An in-container Milvus-lite instance is the default vector database, using a SQLite-backed file at `/opt/app-root/src/.llama/distributions/rh/milvus.db`. The image also pre-downloads the `ibm-granite/granite-embedding-125m-english` model at build time for out-of-the-box embedding support.

Security-wise, the Llama Stack server exposes its HTTP API on port 8321 without any built-in authentication or TLS. This is by design — the server is intended to run behind a platform-managed authentication proxy (kube-rbac-proxy in RHOAI 3.x). The image runs as a non-root user (`${CNB_USER_ID}:${CNB_GROUP_ID}` in the Konflux build). The numerous outbound integrations (cloud inference, vector DBs, search APIs) each have their own credential mechanisms (API keys, IAM credentials, tokens) passed via environment variables, which means the deployed pod will need access to multiple secrets depending on which providers are enabled.
