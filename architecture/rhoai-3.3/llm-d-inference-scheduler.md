## Purpose

**Short**: Intelligent inference request scheduler that routes LLM inference traffic to optimal vLLM backend pods via the Gateway API Inference Extension (GIE) framework.

**Detailed**: The llm-d-inference-scheduler implements an Endpoint Picker (EPP) component for the llm-d inference framework. It extends the upstream Kubernetes Gateway API Inference Extension (GIE) project by adding llm-d-specific scheduling plugins for KV-cache-aware routing, Prefill/Decode (P/D) disaggregation, data parallel scaling, and session affinity. The EPP operates as an Envoy External Processing (ext-proc) gRPC service that receives inference requests from the Envoy gateway data plane and returns routing decisions indicating which backend vLLM pod should handle each request.

The repository produces two deployable container images: (1) the EPP service itself, which runs as a standalone deployment and communicates with Envoy via gRPC ext-proc on port 9002, and (2) a routing sidecar proxy that runs alongside vLLM decoder pods to implement disaggregated prefill/decode protocols (NIXL V2, LMCache, SGLang). The sidecar intercepts inference requests on port 8000, orchestrates prefill-then-decode workflows across multiple pods, and supports data parallel scaling with multiple vLLM ranks.

The scheduler integrates with Kubernetes via the InferencePool CRD from the Gateway API Inference Extension, watching pod endpoints to build a real-time view of available inference workers. Plugins score and filter pods based on KV-cache hit rates (via ZMQ event subscription), request queue depth, active request counts, pod labels (prefill/decode roles), and session affinity. The architecture is designed for high-throughput LLM serving where routing decisions directly impact GPU utilization and inference latency.

## Architectural Analysis

The llm-d-inference-scheduler exemplifies a clean extension architecture built atop the upstream Gateway API Inference Extension (GIE) framework. Rather than forking GIE, the EPP binary imports the GIE runner wholesale (`sigs.k8s.io/gateway-api-inference-extension/cmd/epp/runner`) and injects custom plugins via a registration pattern (`plugins.RegisterAllPlugins()`). This design allows the project to track GIE upstream closely while adding llm-d-specific capabilities. The `cmd/epp/main.go` is remarkably minimal at 47 lines -- all complexity lives in the plugin implementations.

The plugin architecture is well-stratified into four categories: filters (pod selection), scorers (pod ranking), profile handlers (multi-phase orchestration), and pre-request hooks (header injection). The P/D profile handler (`pd_profile_handler.go`) is the most sophisticated component, implementing a two-phase scheduling cycle where it first runs decode scoring, then conditionally runs prefill scoring based on KV-cache hit analysis. The threshold-based decision (`pdThreshold`) enables adaptive behavior: requests with high cache hit rates skip the prefill phase entirely, reducing latency. This adaptive P/D decision is a key differentiator from simple round-robin or random routing.

The sidecar proxy implements three distinct P/D disaggregation protocols (NIXL V2, LMCache, SGLang), each with fundamentally different communication patterns. NIXL V2 uses a two-phase sequential flow with KV-cache transfer parameters embedded in the request/response JSON. SGLang uses concurrent prefill/decode with a bootstrap room mechanism. LMCache is the simplest (deprecated) protocol with `max_tokens=1` prefill. The protocol abstraction is cleanly separated into individual `connector_*.go` files, though they share common infrastructure via the `Server` struct. The SSRF protection mechanism (`allowlist.go`) is notable -- it watches InferencePool resources to dynamically maintain an allowlist of valid prefill targets, preventing the sidecar from being used to probe arbitrary cluster services.

The KV-cache awareness is the deepest integration point. The EPP subscribes to ZMQ events from vLLM servers to track which KV-cache blocks are stored on which pods. The `PrecisePrefixCacheScorer` uses the `llm-d-kv-cache-manager` library to tokenize incoming requests (via embedded Python for Jinja template rendering and Rust FFI for HuggingFace tokenizers), hash token sequences into blocks, and look up which pods have those blocks cached. This enables true KV-cache locality-aware routing, where requests are routed to pods that already have relevant KV-cache blocks, avoiding redundant prefill computation. The multi-language build (Go + Rust + Python) adds significant build complexity but is architecturally motivated by the need for high-performance tokenization (Rust) and template rendering (Python Jinja2).

A security consideration worth noting is the TLS posture. The sidecar defaults to `secure-proxy=true` with self-signed certificates, but the `InsecureSkipVerify` flags for both prefiller and decoder connections are exposed as configurable options. In development manifests, the Istio DestinationRule uses `InsecureSkipVerify: true`. The absence of `GOEXPERIMENT=strictfipsruntime` in both Konflux Dockerfiles means the Go binaries rely on system-level FIPS enforcement rather than compile-time FIPS opt-in, which may be insufficient for strict FIPS certification requirements.
