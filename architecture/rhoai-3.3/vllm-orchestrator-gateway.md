## Purpose

**Short**: An OpenAI-compatible HTTP gateway that routes chat completion requests to the FMS Guardrails Orchestrator with configurable per-route detector-based content filtering.

**Detailed**: The vLLM Orchestrator Gateway is a lightweight Rust HTTP service built on the Axum web framework that provides a configurable routing layer between LLM clients and the FMS Guardrails Orchestrator. It exposes multiple OpenAI-compatible `/v1/chat/completions` endpoints, each mapped to a named route with its own set of content detectors (e.g., PII detection, toxicity). When a client sends a chat completion request to a route endpoint, the gateway injects the route's detector configuration into the request payload and forwards it to the orchestrator's `/api/v2/chat/completions-detection` endpoint.

The gateway handles both streaming (SSE) and non-streaming chat completions. When detectors identify problematic content in the orchestrator's response (input or output detections), the gateway replaces the generated content with a configurable fallback message. This enables drop-in replacement of an unguardrailed chat completion model endpoint with a guardrailed one, requiring no client-side changes beyond pointing at the gateway's route-specific URL (e.g., `/pii/v1/chat/completions` instead of `/v1/chat/completions`).

The service supports mTLS for secure communication with the orchestrator backend, loading client certificates and CA bundles from well-known paths (`/etc/tls/private/`, `/etc/tls/ca/`). It forwards authorization headers from clients to the orchestrator, enabling end-to-end authentication. Configuration is driven by a YAML file that defines the orchestrator connection, detector definitions with parameters, and named routes mapping detectors to endpoints.

## Architectural Analysis

The vLLM Orchestrator Gateway follows a classic API gateway / reverse proxy pattern, purpose-built for the guardrails use case. Its core design decision is to expose per-route endpoints (`/{route_name}/v1/chat/completions`) that are OpenAI-compatible, allowing LLM clients to switch from direct model access to guardrailed access by simply changing the base URL. This is a strong design choice for adoption, as it requires zero client-side code changes. The route-to-detector mapping is entirely configuration-driven via a YAML file, making it straightforward to add new safety pipelines without code changes.

The codebase is remarkably small (3 Rust source files, ~650 lines total), which reflects its narrow scope as a request transformation and routing layer. The gateway performs three key transformations: (1) it resolves route detector names to orchestrator detector configuration objects, (2) it injects these detectors into the request payload, and (3) it inspects the orchestrator response for detections and substitutes a fallback message when detections are found. The streaming implementation (`handle_streaming_generation`) processes SSE chunks from the orchestrator, deserializing each chunk to check for detections and applying the fallback message per-chunk. This is notably more complex than the non-streaming path due to SSE parsing and per-chunk detection handling.

The TLS implementation in `build_orchestrator_client` (lines 351-406 of `main.rs`) is well-structured: it conditionally enables mTLS by checking for certificate files at well-known paths, and falls back to plain HTTP when certificates are absent. The PEM-to-PKCS#12 conversion is necessary because `native-tls` (which wraps system OpenSSL) requires PKCS#12 identity bundles. One notable security observation is the `danger_accept_invalid_hostnames(true)` call when the orchestrator host is `localhost` — this is a development convenience that should never trigger in production deployments where the orchestrator runs behind a Kubernetes Service DNS name. The gateway does not implement its own authentication; it relies on the network environment (e.g., kube-rbac-proxy sidecar or network policies) for ingress protection and simply forwards `Authorization` headers to the orchestrator.

A notable architectural gap is the absence of health check endpoints (`/health`, `/ready`). Kubernetes liveness/readiness probes would need to be configured externally (e.g., TCP socket checks on port 8090) or the deploying manifests would need to add a health endpoint. The configuration validation (`validate_registered_detectors` in `config.rs`) is thorough — it checks that all detectors referenced in routes are defined and that no route has duplicate input/output detectors for the same server. However, configuration is loaded once at startup with no support for runtime reloading; route changes require a pod restart.
