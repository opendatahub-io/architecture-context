## Purpose

**Short**: A pluggable KV-Cache management service that enables cache-aware routing and cross-node cache coordination for distributed LLM inference on vLLM-based serving platforms.

**Detailed**: The llm-d-kv-cache project provides a comprehensive suite of KV-Cache management capabilities for the llm-d distributed LLM inference platform. Its primary component is the KV-Cache Indexer, a high-performance Go library that maintains a global, near-real-time view of KV-Cache block locality across a fleet of vLLM pods. The indexer consumes structured KV-Events streamed from vLLM instances via ZeroMQ, tracking which cache blocks reside on which nodes and storage tiers (GPU or CPU). This metadata enables intelligent, cache-aware routing by allowing schedulers to score pods based on prefix cache hit ratios.

Beyond the core indexer, the project includes several connectors and utilities: a filesystem storage backend (`llmd_fs_backend`) that offloads KV-cache blocks between GPU memory and shared storage (POSIX, NFS, or S3 via NIXL); a PVC evictor that automatically manages disk space for offloaded cache on Kubernetes PersistentVolumeClaims; and a UDS tokenizer gRPC service that provides model tokenization and chat template rendering as a sidecar. The project supports both in-memory and Redis/Valkey-backed index backends for different scale requirements.

## Architectural Analysis

The llm-d-kv-cache project follows a layered library architecture where each concern — indexing, event processing, tokenization, storage — is encapsulated as a pluggable Go or Python module. The Go core (`pkg/`) is designed to be consumed either as an in-process library by the `llm-d-inference-scheduler` or as a standalone gRPC/HTTP service. This dual-mode design is visible in the telemetry initialization: standalone mode creates its own OpenTelemetry TracerProvider, while library mode defers to the parent application's global provider (`telemetry/tracing.go:60-135`). The decorator pattern is used extensively — `TracedScorer`, `TracedIndex`, and `InstrumentedIndex` wrap core types to add tracing and metrics without modifying business logic.

The event processing system demonstrates careful attention to ordering and scalability. The `Pool` uses FNV-1a hash-based sharding to ensure per-pod message ordering while allowing concurrent processing across pods (`kvevents/pool.go:88-166`). The `SubscriberManager` dynamically manages per-pod ZMQ subscribers with endpoint change detection, supporting both static configuration and Kubernetes-based pod discovery. The engine adapter abstraction (`vllm_adapter.go`, `sglang_adapter.go`) decouples the wire format from the domain model, enabling support for multiple inference engines with different serialization formats (msgpack with struct-as-array for vLLM, msgpack with field count validation for SGLang).

The token-to-block-key computation (`kvblock/token_processor.go`) is a critical compatibility point — it must produce identical hash keys as vLLM's internal prefix caching algorithm. The implementation uses deterministic CBOR encoding with chained FNV-64a hashing, where each block's hash incorporates all preceding blocks to ensure prefix-dependent keys. The `PYTHONHASHSEED=0` requirement (visible in the Helm chart values) ensures hash determinism between Go and Python runtimes. Multimodal support (`extra_keys.go`) extends this by mapping image/video content hashes to block granularity, enabling cache-aware routing for multimodal models.

The storage offloading subsystem (`kv_connectors/llmd_fs_backend/`) bridges the gap between GPU-resident KV-cache and persistent storage. The C++ extension provides a NUMA-aware thread pool with per-worker CUDA streams, CPU affinity binding, and staged memory buffers (`csrc/storage/thread_pool.cpp`). Write backpressure is managed via EMA-based duration tracking and configurable queue depth limits (`csrc/storage/storage_offload.cpp:272-288`), preventing unbounded queue growth under burst conditions. The NIXL backend adds object storage support through an agent-based transfer model, but notably uses `hashlib.md5` for object key sharding — a non-cryptographic but technically non-FIPS-approved hash function.

A notable architectural risk is the complete absence of transport-level encryption and authentication across all internal services. The HTTP scoring endpoints, gRPC IndexerService, ZMQ event streams, and even the Redis connections are all plaintext without authentication. This design assumes trusted cluster networking (e.g., pod-to-pod within a namespace), but offers no defense-in-depth against lateral movement within the cluster. The UDS tokenizer's use of Unix Domain Sockets provides implicit pod-local isolation, which is the strongest security boundary in the system. For RHOAI deployment, the platform operator's gateway infrastructure (HTTPRoute + kube-rbac-proxy) would need to front any externally-exposed endpoints.
