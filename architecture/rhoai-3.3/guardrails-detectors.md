## Purpose

**Short**: A collection of detector algorithm microservices for the FMS Guardrails Orchestrator, providing text content safety analysis via regex-based PII detection, file type validation, HuggingFace model inference, and LLM-as-a-Judge evaluation.

**Detailed**: The guardrails-detectors repository provides multiple independently-deployable detector microservices that integrate with the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator), an IBM-led open source project for invoking detectors on LLM text generation input and output. Each detector is packaged as a separate container image and deployed as a KServe InferenceService on the RHOAI platform.

The repository produces three detector types: (1) a **built-in detector** providing lightweight heuristic detection functions (regex-based PII detection for emails, credit cards, SSNs, phone numbers, IP addresses, and UK postal codes; file type validation for JSON, XML, and YAML with optional schema validation; and a custom detector framework allowing user-defined Python detection logic), (2) a **HuggingFace detector** that loads and runs AutoModelForSequenceClassification or GraniteForCausalLM models for content safety classification, and (3) an **LLM Judge detector** that uses the vllm_judge library to perform LLM-as-a-Judge evaluation using built-in metrics (safety, toxicity, accuracy, helpfulness, etc.) against an external vLLM-compatible inference server.

All detectors implement the IBM Detector API specification, exposing a `POST /api/v1/text/contents` endpoint for content analysis. Each detector is instrumented with Prometheus metrics (detections, requests, errors, runtime counters) and designed to be deployed as a KServe InferenceService using the RawDeployment mode with Istio sidecar injection for service mesh integration. The TrustyAI team at Red Hat maintains these detectors.

## Architectural Analysis

The guardrails-detectors repository follows a **multi-service, shared-library architecture** where three independent detector services share a common Python library (`detectors/common/`). The common library provides the FastAPI application base class (`DetectorBaseAPI`), Pydantic request/response schemas, Prometheus instrumentation infrastructure, and a health endpoint. This shared-nothing approach means each detector is independently deployable and scalable, while the common library ensures API consistency across all detectors.

The **built-in detector** uses a registry pattern (`BaseDetectorRegistry`) that enables pluggable detection functions. Three registries ship by default: `RegexDetectorRegistry` (PII patterns), `FileTypeDetectorRegistry` (JSON/XML/YAML validation), and `CustomDetectorRegistry` (user-defined Python functions). The custom detector framework is particularly notable -- it performs static code analysis on user-provided Python files to block dangerous imports (`os`, `subprocess`, `sys`, `shutil`) and function calls (`eval`, `exec`, `open`, `compile`, `input`) before loading them, providing a security sandbox for arbitrary user code. Custom detectors can also be decorated with `@non_blocking` for asynchronous background execution and `@use_instruments` for custom Prometheus metrics.

The **HuggingFace detector** supports two model architectures: `AutoModelForSequenceClassification` (multi-class classification with configurable safe labels and thresholds) and `GraniteForCausalLM` (IBM Granite Guardian models that generate safe/unsafe tokens with log-probability-based risk scoring). The detector auto-detects the model architecture from the HuggingFace config and supports CUDA GPU acceleration with fallback to CPU. The Konflux Dockerfile for this component is notably complex, with multi-stage builds for cross-compilation on ppc64le and s390x architectures, including building PyTorch and OpenBLAS from source for these platforms.

A key architectural observation is the **asymmetry in Konflux maturity** across the three detectors. The built-in and HuggingFace detectors have full Konflux pipelines (`.tekton/` PipelineRuns, `Dockerfile.konflux.*` files, RHOAI image labels), while the LLM Judge detector only has a standard `Dockerfile.judge` with no Konflux variant. This suggests the LLM Judge is at an earlier maturity stage for RHOAI shipping. The Tekton pipelines also reveal that the HuggingFace detector builds for four architectures (x86_64, arm64, ppc64le, s390x) while the built-in detector builds for the same four, but the LLM Judge has no Tekton pipeline at all. Both shipped Konflux pipelines set `hermetic: false`, which is a supply chain gap worth tracking.

The detector API design follows the IBM FMS Guardrails specification, where the `ContentAnalysisHttpRequest` carries a `contents` list and optional `detector_params` dict, and the response returns a nested list of `ContentAnalysisResponse` objects (one list per input content, each containing zero or more detections). This design enables batch processing and multi-detector orchestration in a single request. The LLM Judge extends this with a `/api/v1/text/generation` endpoint for prompt+response pair evaluation, which is architecturally significant as it enables different evaluation semantics (input-only vs input+output analysis).
