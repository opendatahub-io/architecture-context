## Purpose
**Short**: A collection of detector algorithm microservices for the FMS Guardrails Orchestrator, providing text content analysis for AI safety guardrailing.

**Detailed**: The Guardrails Detectors repository provides a set of detector algorithms and microservices that integrate with the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator), an IBM-led open source project for invoking detectors on text generation input and output. Each detector is packaged as an independent container image exposing a FastAPI-based REST API.

Three detector types are supported: (1) **Built-in detectors** — lightweight heuristic/algorithmic detection functions (regex-based PII detection, file-type validation, custom Python code detectors) deployed alongside the orchestrator; (2) **HuggingFace detector** — a generic detector runtime compatible with `AutoModelForSequenceClassification` and `GraniteForCausalLM` models, designed for deployment as a KServe InferenceService; (3) **LLM Judge detector** — an LLM-as-a-judge evaluator using the [vLLM Judge](https://github.com/trustyai-explainability/vllm_judge) library to assess content safety, toxicity, accuracy, and other metrics via an external vLLM-compatible LLM server.

All detectors conform to the IBM Detector API specification and expose Prometheus metrics for observability. The HuggingFace and LLM Judge detectors are intended to be deployed via KServe ServingRuntime/InferenceService CRDs in RawDeployment mode with Istio sidecar injection.

### Security Controls in Custom Detectors

The `CustomDetectorRegistry` performs static code analysis on user-provided custom detector Python files before loading them. The analysis checks for:
- **Forbidden imports**: `os`, `subprocess`, `sys`, `shutil`
- **Forbidden function calls**: `eval`, `exec`, `open`, `compile`, `input`

If any forbidden patterns are detected, the module fails to load with an `ImportError`.
