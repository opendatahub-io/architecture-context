## Purpose

**Short**: Manages the lifecycle of Model Registry instances and the Model Catalog service on OpenShift/Kubernetes clusters.

**Detailed**: The Model Registry Operator is a Kubernetes operator built with controller-runtime and Kubebuilder that manages two primary workloads: per-tenant Model Registry instances (via the `ModelRegistry` CRD) and a singleton Model Catalog service. For each `ModelRegistry` custom resource, the operator provisions a complete stack including a REST API deployment (the model-registry service), an optional kube-rbac-proxy sidecar for authentication, OpenShift Routes for external access, RBAC resources for user access control, optional auto-provisioned PostgreSQL databases, and NetworkPolicies for traffic isolation. The Model Catalog controller manages a shared catalog service with its own PostgreSQL backend, kube-rbac-proxy authentication, and admin group configuration sourced from the platform's Auth CR.

The operator supports multiple authentication patterns with automatic migration between them: the legacy Istio/service-mesh pattern (v1alpha1) is automatically converted to OAuth proxy, and OAuth proxy is automatically migrated to kube-rbac-proxy (v1beta1 / RHOAI 3.x pattern). It detects cluster capabilities at startup (OpenShift vs plain Kubernetes, BYOIDC mode) and conditionally creates platform-specific resources. The operator also handles CRD storage version migration (v1alpha1 to v1beta1) using either the Kubernetes StorageVersionMigration API or a manual re-read/re-write fallback strategy.

## Architectural Analysis

The model-registry-operator follows a dual-controller architecture with a clear separation of concerns. The `ModelRegistryReconciler` manages per-tenant model registry instances — each `ModelRegistry` CR triggers creation of an independent stack of Kubernetes resources including a Deployment with the model-registry REST service, optional kube-rbac-proxy sidecar, Service, ServiceAccount, Role, RoleBinding, and optionally a Route and NetworkPolicy. The `ModelCatalogReconciler` manages a singleton catalog service that is not tied to any individual ModelRegistry CR but instead uses the platform's `default-modelregistry` component CR as its owner reference. This split ensures that tenant-scoped resources are properly garbage-collected via owner references while the shared catalog service has stable lifecycle management.

The authentication migration path is particularly well-designed. The operator supports three generations of auth patterns: Istio service mesh (v1alpha1, fully deprecated), OAuth proxy (transitional), and kube-rbac-proxy (v1beta1 current). Migration happens automatically at multiple levels — the v1alpha1-to-v1beta1 conversion webhook replaces Istio config with OAuth proxy, the defaulting webhook migrates OAuth proxy to kube-rbac-proxy, and the reconciler catches any remaining OAuth proxy configurations. This multi-layered migration ensures that model registries created under any previous version are automatically upgraded to the current auth pattern without manual intervention. The kube-rbac-proxy pattern uses Kubernetes-native SubjectAccessReview to authorize access, checking whether the user has `get` permission on the registry's Service resource — a clean integration with Kubernetes RBAC.

The cluster capability detection system (`ClusterCapabilities`) enables the operator to run on plain Kubernetes, traditional OpenShift, and BYOIDC-enabled OpenShift 4.20+ clusters. The detection logic uses the Kubernetes discovery API and handles partial failures gracefully — a critical design choice since BYOIDC clusters intentionally remove the `user.openshift.io` API group, which causes partial discovery failures. When BYOIDC mode is detected, the operator skips creating OpenShift Group resources but continues creating RoleBindings that reference external IdP group names, ensuring compatibility with externally managed identity providers.

The template-based resource generation approach using Go's `text/template` with embedded YAML templates (stored in `internal/controller/config/templates/`) provides flexibility for complex resource configurations while keeping the controller code focused on orchestration logic. The `k8s-objectmatcher` library is used for intelligent diff-based updates — only resources that actually changed are updated, reducing unnecessary API server load. However, this approach means that the actual resource specifications are split between Go code (controller logic, defaults) and YAML templates (resource structure), which requires reading both to understand the full picture of what gets deployed.

The storage version migration system is a notable operational concern. When the operator upgrades from v1alpha1 to v1beta1 storage, it first attempts to use the Kubernetes `StorageVersionMigration` API (migration.k8s.io), and if that's not available, falls back to a manual strategy that reads each ModelRegistry in the source version, gets it in the target version (triggering the conversion webhook), and writes it back with a migration annotation. This ensures all resources stored in etcd are re-written in the new storage version format.
