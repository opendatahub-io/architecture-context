## Purpose

**Short**: Platform operator that manages the full lifecycle of Red Hat OpenShift AI (RHOAI), including component deployment, ingress infrastructure, authentication, monitoring, and RBAC.

**Detailed**: The rhods-operator (also known as opendatahub-operator downstream) is the central platform operator for Red Hat OpenShift AI 3.x. It manages 14 data science components (Dashboard, Workbenches, KServe, Kueue, DataSciencePipelines, Ray, TrustyAI, ModelRegistry, TrainingOperator, Trainer, FeastOperator, LlamaStackOperator, MLflowOperator, and ModelsAsService) through a hierarchical custom resource model. Users create a DataScienceCluster (DSC) CR to specify which components to enable, and the operator creates per-component CRs that trigger component-specific controllers.

Beyond component lifecycle, the operator deploys the platform's entire ingress infrastructure via the Gateway service controller. This includes a Kubernetes Gateway API Gateway CR, Envoy proxy data plane, EnvoyFilter CRs for authentication, kube-auth-proxy (OAuth2-proxy) for OpenShift OAuth or external OIDC authentication, HTTPRoute CRs for per-component routing, DestinationRule CRs for TLS, NetworkPolicy resources, and OpenShift Routes for OCP route mode or legacy redirects. The operator also manages platform-wide services: Auth (RBAC group management), Monitoring (Prometheus, Tempo, AlertManager, Perses, OpenTelemetry), and CertConfigMapGenerator (trusted CA bundle distribution).

The operator uses a registry pattern where component and service controllers self-register via Go init() functions, enabling modular addition of new components. All component manifests are prefetched and embedded into the operator image at build time under `/opt/manifests/`, with platform-specific kustomize overlays applied at reconciliation time.

## Architectural Analysis

The rhods-operator employs a sophisticated hierarchical reconciliation architecture. At the top level, the DSCInitialization controller establishes platform prerequisites (namespaces, Auth, GatewayConfig, monitoring), while the DataScienceCluster controller manages the lifecycle of 14 optional data science components. This separation ensures that platform infrastructure is always present regardless of which components are enabled. The pattern of creating singleton service CRs (Auth, GatewayConfig, Monitoring) from the DSCI controller and watching for their deletion to automatically recreate them provides resilience against accidental resource removal.

The Gateway service controller is architecturally the most complex piece, deploying a full multi-layer ingress stack entirely through controller reconciliation logic (no static manifests). It creates a Gateway API Gateway CR, a GatewayClass, kube-auth-proxy Deployment/Service/Secret, EnvoyFilter for ext_authz authentication, DestinationRule for TLS, HTTPRoute for OAuth callbacks, and NetworkPolicy -- all in the openshift-ingress namespace. The authentication flow is layered: external requests hit the Gateway (Envoy), which invokes the EnvoyFilter ext_authz to consult kube-auth-proxy (an OAuth2-proxy instance), which validates against OpenShift OAuth or an external OIDC provider. A Lua filter in the EnvoyFilter then extracts access tokens and forwards them to upstream services. This design supports three auth modes (IntegratedOAuth, OIDC, None) and two ingress modes (OcpRoute with ClusterIP + OpenShift Route re-encryption, and LoadBalancer with direct DNS).

The component controller pattern is highly uniform: each component self-registers in Go init() with a registry, implements the ComponentHandler interface, and follows the same action chain (initialize -> render kustomize -> deploy -> check status -> GC). This makes adding new components straightforward. The manifest embedding strategy (prefetched-manifests/ directory baked into the image at build time with platform-specific overlays) avoids runtime git clones while supporting ODH/RHOAI differentiation. The monitoring integration is particularly thorough: each component has PrometheusRule templates embedded in the binary, and the Monitoring service controller dynamically assembles a complete observability stack including MonitoringStack (Prometheus), Tempo (tracing), OpenTelemetry collectors, AlertManager, and Perses dashboards.

A notable design decision is the FIPS compliance posture: while the Go binary is built with `GOEXPERIMENT=strictfipsruntime` and `CGO_ENABLED=1` (passing check-payload validation), the OLM CSV annotation explicitly declares `fips-compliant: "false"`. This suggests the operator team has identified application-level crypto gaps that prevent full FIPS certification. The DestinationRule template uses `insecureSkipVerify: true` for TLS to the auth proxy, and the InsecureSkipVerify flag is configurable for OIDC provider connections, both of which are relevant for FIPS evaluation. The operator manages extremely broad RBAC (250+ rules for the DSC controller alone), which is necessary given its role as a platform-level orchestrator managing resources across multiple namespaces and API groups.

The webhook system provides validation and mutation across multiple resource types: DSC/DSCI singleton enforcement, HardwareProfile injection into Notebooks/InferenceServices/LLMInferenceServices, Kueue label validation, connection secret injection for Notebooks and serving resources, and deprecation webhooks for legacy resource types. The HardwareProfile webhook is particularly notable as it mutates pod specs at admission time to inject GPU resource requirements and node scheduling configuration.
