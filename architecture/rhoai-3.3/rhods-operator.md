## Purpose
**Short**: Platform operator for Red Hat OpenShift AI (RHOAI) that manages the lifecycle of AI/ML platform components on OpenShift, including Gateway API-based ingress, authentication, monitoring, and component deployment.

**Detailed**: The rhods-operator (also known as opendatahub-operator v2) is the central control plane for the Red Hat OpenShift AI (RHOAI) distribution. It provides a declarative API for configuring and managing all platform components through two primary Custom Resources: `DSCInitialization` (platform-level initialization including auth, monitoring, and gateway) and `DataScienceCluster` (component-level configuration for dashboard, workbenches, model serving, pipelines, and more).

In RHOAI 3.x, the operator has migrated to a Gateway API-based ingress architecture. The Gateway controller deploys `Gateway`, `GatewayClass`, `HTTPRoute`, and `EnvoyFilter` resources in the `openshift-ingress` namespace, replacing the legacy Route+oauth-proxy pattern. Authentication is handled via a `kube-auth-proxy` (OAuth2 proxy) deployment that integrates with both OpenShift OAuth and external OIDC providers, enforced through Envoy's external authorization filter.

The operator manages 14+ platform components via dedicated sub-controllers, each deploying kustomize-based manifests from prefetched bundles. Components include Dashboard, Workbenches (Jupyter), KServe, Data Science Pipelines, Kueue, Ray, TrustyAI, Model Registry, Training Operator, Feast Operator, LlamaStack Operator, MLflow Operator, Trainer, and Models-as-a-Service. The operator uses a plugin architecture where each component registers itself and implements a standard reconciliation interface.

**Gateway API Architecture (RHOAI 3.x)**:
The Gateway Controller (`internal/controller/services/gateway/`) dynamically creates the following resources:

1. **GatewayClass** (`data-science-gateway-class`): Registered with controller `openshift.io/gateway-controller/v1`
2. **Gateway** (`data-science-gateway` in `openshift-ingress` namespace): HTTPS listener on port 443 with TLS termination. Supports two ingress modes:
   - **OcpRoute mode** (default): Gateway uses ClusterIP service, exposed via OpenShift Route with reencrypt TLS
   - **LoadBalancer mode**: Gateway uses LoadBalancer service type directly
3. **kube-auth-proxy Deployment**: OAuth2 proxy handling authentication (ports: 4180/HTTP, 8443/HTTPS, 9000/metrics)
4. **HTTPRoute** (`oauth-callback-route`): Routes `/oauth2` path to kube-auth-proxy for OAuth callbacks
5. **EnvoyFilter** (`data-science-authn-filter`): Envoy ext_authz filter that sends all Gateway traffic through kube-auth-proxy for authentication, plus Lua filter for cookie stripping and Bearer token injection
6. **DestinationRule** (`data-science-tls-rule`): Configures Istio mTLS for ports 443 and 8443
7. **NetworkPolicy** (`kube-auth-proxy`): Restricts ingress to kube-auth-proxy from Gateway pods and monitoring namespaces
8. **OpenShift Route** (OcpRoute mode only): Routes external traffic to the Gateway service

Component controllers create their own HTTPRoute CRs referencing `data-science-gateway` as the parent Gateway. All component traffic flows through the Gateway and is authenticated by the EnvoyFilter + kube-auth-proxy chain.
