## Purpose

**Short**: llm-d is a high-performance distributed inference serving stack that provides optimized container images (vLLM-based) and composable Kubernetes deployment guides for production LLM inference across multiple hardware accelerators.

**Detailed**: llm-d is a CNCF sandbox project founded by Red Hat, Google Cloud, IBM Research, CoreWeave, and NVIDIA. It does not contain Go operator code or Python service logic of its own — instead, it serves as the integration and packaging layer for the llm-d inference stack. The repository produces multi-accelerator container images that bundle vLLM (the model server engine) with high-performance networking libraries (NIXL, UCX, NVSHMEM, gdrcopy), KV cache connectors (LMCache, llm-d filesystem backend), and GPU kernel libraries (DeepEP, DeepGEMM, FlashInfer). These images are the runtime artifact deployed to Kubernetes clusters.

The repository also provides a comprehensive set of deployment guides ("well-lit paths") structured as composable Kustomize bases and Helm values files. These guides cover deployment patterns ranging from an optimized baseline (single-role vLLM with intelligent routing) to advanced topologies like prefill/decode disaggregation, wide expert parallelism for MoE models, tiered KV cache offloading, flow control, workload autoscaling, and asynchronous batch processing. Each guide is organized as a kustomize overlay tree with per-accelerator patches (NVIDIA GPU, AMD GPU, Intel XPU/Gaudi, Google TPU, CPU), ensuring tested configurations for each hardware platform.

Within the RHOAI (Red Hat OpenShift AI) distribution, llm-d container images are rebuilt by Konflux from a downstream branch and integrated into the platform's inference serving capabilities. The deployment guides and Helm values serve as the reference architecture that the RHOAI platform operator can consume when deploying llm-d-based inference pools.

### llm-d-rdma-tools

**Dockerfile**: `docker/Dockerfile.rdma-tools`
**Intent**: Diagnostic utility image for validating RDMA networking. Not deployed as a service — used by operators and SREs to troubleshoot inter-node GPU communication. Contains iperf3, tcpdump, perftest, gdrcopy, and NCCL tests.
**Image**: `ghcr.io/llm-d/llm-d-rdma-tools`
**Language**: C/C++ (compiled tools)
**Port**: None (diagnostic tool)

### llm-d-routing-sidecar (external)

**Intent**: Disaggregated serving routing proxy deployed as an init container (with `restartPolicy: Always`) alongside the decode model server. Routes requests between clients and the decode vLLM instance, coordinating prefill-decode KV cache transfer via NIXL. Not built in this repo — sourced from `ghcr.io/llm-d/llm-d-routing-sidecar:v0.7.1`.
**Image**: `ghcr.io/llm-d/llm-d-routing-sidecar:v0.7.1`
**Port**: 8000/TCP (sidecar ingress) → 8200/TCP (decode modelserver)

### llm-d-inference-scheduler (external)

**Intent**: Endpoint Picker (EPP) that implements intelligent request routing for the InferencePool. Deployed via Helm chart — not built in this repo but referenced in scheduler values files. Uses a plugin-based scoring architecture with filters and scorers for prefix-cache-aware, load-aware, and PD-aware routing.
**Image**: `ghcr.io/llm-d/llm-d-inference-scheduler:v0.8.0`
**Port**: Configured by Helm chart

## Architectural Analysis

llm-d is architecturally distinctive in that it is not a service or operator itself — it is a composition and packaging layer. The repository's primary outputs are container images and deployment recipes, not application code. This "meta-project" pattern means the architecture is defined by how components are assembled rather than by internal control flow. The deployment guides use a layered kustomize composition model where a base model server Deployment (with just a placeholder image and standard health probes) is progressively specialized through overlays: first by deployment topology (single-role vs. prefill/decode disaggregation), then by hardware accelerator (GPU, HPU, XPU, TPU, CPU), and finally by cloud provider or infrastructure variant (GKE, CoreWeave, OpenShift). This three-dimensional overlay structure allows ~60+ deployment configurations from a small number of reusable bases.

The networking architecture relies entirely on the Kubernetes Gateway API standard and its Inference Extension. Rather than implementing its own load balancer or ingress controller, llm-d deploys a standard Gateway CR and relies on pluggable gateway providers (Istio, AgentGateway, GKE load balancers) for the data plane. The intelligent routing logic lives in the EPP (Endpoint Picker Plugin), which runs as a separate deployment managed by the InferencePool Helm chart. This clean separation between data plane (gateway provider), control plane (InferencePool/EPP), and compute plane (model server pods) makes the architecture highly modular but creates a multi-component deployment surface that requires careful coordination.

The prefill/decode disaggregation pattern is particularly noteworthy. Instead of using a single model server that handles both phases of autoregressive generation, llm-d splits the workload into separate Deployments — one for the compute-intensive prefill phase (processing the full input prompt) and one for the latency-sensitive decode phase (generating tokens one at a time). A routing sidecar container (`llm-d-routing-sidecar`) is injected into the decode pods as a restartable init container, fronting the decode vLLM process on port 8000 while the actual vLLM process is shifted to port 8200. KV cache state is transferred from prefill to decode pods via NIXL over RDMA (port 5600/TCP), avoiding network copies through the CPU. This architecture reduces TTFT by up to 47% for long-prompt workloads.

The container images themselves are complex multi-stage builds with extensive library compilation. The CUDA Dockerfile alone compiles gdrcopy, UCX, NIXL, DeepEP, DeepGEMM, FlashInfer, and LMCache from source before installing vLLM. All dependency versions are pinned via explicit git commit SHAs in `docker/common-versions`, providing reproducibility without traditional lock files. The build supports three vLLM installation modes: fully editable from source (for development), precompiled binaries from vLLM's wheel index (faster dev builds), and production wheels. The GB200/Blackwell variant is handled via Docker multi-target builds (`--target runtime-gb200`) that swap in a specialized DeepEP wheel.

A notable security consideration is that model server pods run without TLS and without authentication on their HTTP endpoints. All security enforcement is delegated to the gateway layer — the model servers trust that only authorized traffic reaches them through the Gateway/InferencePool routing chain. This is a deliberate design choice that simplifies the model server deployment but requires the gateway provider to be properly configured with TLS and authentication for production use. The interactive-pod helper has notably broad RBAC permissions (including secrets read/write and wildcard access to inference CRDs), which is appropriate for a development/debugging tool but should not be deployed in production namespaces.
