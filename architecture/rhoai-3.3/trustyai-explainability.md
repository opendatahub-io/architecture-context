## Purpose

**Short**: TrustyAI Explainability is a Java library and Quarkus-based REST service that provides model fairness metrics, explainability algorithms, and data drift detection for ML models served by KServe and ModelMesh on the RHOAI platform.

**Detailed**: TrustyAI Explainability is a multi-module Maven project organized around a core Java library (`explainability-core`) that implements XAI (Explainable AI) algorithms including LIME, SHAP, counterfactual explanations, time-series saliency, fairness metrics (Statistical Parity Difference, Disparate Impact Ratio), and data drift detection (Fourier MMD, Meanshift). The `explainability-service` module wraps these algorithms as a Quarkus REST service that integrates with the RHOAI platform.

The service acts as a payload processor for inference data: it receives inference request/response payloads from KServe (via CloudEvents or direct HTTP POST) and ModelMesh (via HTTP POST), reconciles input-output pairs, stores them in a configurable storage backend (PVC file storage, MinIO S3-compatible storage, or MariaDB/MySQL via Hibernate ORM), and exposes REST endpoints for computing fairness metrics, drift detection, and model explanations. Scheduled metrics are published to Prometheus for integration with the platform's monitoring stack.

The `explainability-connectors` module provides client libraries for interacting with KServe v1/v2 HTTP and v2 gRPC inference services, enabling the explainability algorithms to query model predictions. The `explainability-arrow` module provides Apache Arrow IPC serialization for high-performance data exchange with TrustyAI's Python library.

## Architectural Analysis

TrustyAI Explainability follows a modular Maven architecture where the core algorithmic library (`explainability-core`) is cleanly separated from the deployment wrapper (`explainability-service`) and protocol adapters (`explainability-connectors`, `explainability-arrow`). This separation allows the core library to be consumed independently (e.g., by the TrustyAI Python library via Arrow IPC) while the service module adds Kubernetes-native deployment, REST APIs, and platform integration. The Quarkus framework provides the runtime with reactive REST endpoints, CDI dependency injection, health checks, and Prometheus metrics out of the box.

The payload reconciliation pattern is a distinctive architectural choice: inference inputs and outputs arrive as separate events (via CloudEvents from KServe or HTTP POST from ModelMesh) and are matched by inference ID in the `InferencePayloadReconciler`. The `KServeInferencePayloadReconciler` and `ModelMeshInferencePayloadReconciler` handle the protocol-specific parsing, while the common reconciler manages the ID-matching and storage. Payload storage can be in-memory or Hibernate-backed, providing flexibility for different scale requirements.

The RHOAI distribution profile (`%rhoai` in `application.properties`) disables all explainer endpoints (LIME, SHAP, counterfactual, time-series saliency, aggregated LIME, PDP) while keeping fairness metrics and drift detection enabled. This is a deliberate product scoping decision: fairness and drift are the GA-supported features in RHOAI, while explainability algorithms remain available only in the upstream ODH distribution. The feature flag system uses Quarkus profile-based configuration, making it straightforward to toggle capabilities per deployment profile.

The storage abstraction layer supports three backends with significantly different capabilities: PVCStorage provides full-featured batch reading with tag filtering and model discovery on the local filesystem; HibernateStorage provides transactional database access with schema migration support; and MinioStorage provides S3-compatible object storage but with limitations (no batch reading or data slicing). This asymmetry suggests that PVC and database backends are the primary deployment targets, with MinIO support being more experimental.

Security at the application level is minimal by design: the service exposes all endpoints without authentication or authorization, relying on the TrustyAI Service Operator to inject a kube-rbac-proxy sidecar (in RHOAI 3.x) or oauth-proxy sidecar (in RHOAI 2.x) for access control. The gRPC connections to KServe inference services use plaintext by default, which is acceptable for intra-cluster communication but should be noted as a potential concern in multi-tenant environments. TLS support exists for the HTTP server on port 4443 via external certificate files, but this requires the certificates to be provisioned externally and mounted at `/etc/tls/internal/`.
