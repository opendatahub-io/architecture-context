## Purpose

**Short**: Kubernetes-native operator for managing distributed machine learning training jobs with support for PyTorch, MPI, DeepSpeed, and other frameworks.

**Detailed**: Kubeflow Trainer v2 is a controller-runtime-based Kubernetes operator that manages the lifecycle of distributed ML training workloads through the TrainJob CRD. Users create TrainJob resources referencing a TrainingRuntime (namespace-scoped) or ClusterTrainingRuntime (cluster-scoped) that defines the training framework, pod topology, and scheduling policy. The controller translates these declarative specifications into JobSet resources via a plugin-based runtime framework, supporting PyTorch distributed training (torchrun/torchtune), MPI-based training (OpenMPI with DeepSpeed), and plain multi-node configurations.

The RHOAI distribution adds progression tracking (real-time training metrics via HTTP polling), automatic NetworkPolicy generation for pod isolation, Prometheus monitoring via PodMonitor, and pre-configured GPU training runtimes for NVIDIA CUDA and AMD ROCm hardware. A secondary sub-component — the Data Cache — provides an Apache Arrow Flight-based distributed caching layer (written in Rust) that pre-loads Iceberg/Parquet datasets into memory for fast training data access, though this component does not have a Konflux Dockerfile and is not shipped in the RHOAI 3.3 release.

## Architectural Analysis

The Kubeflow Trainer v2 represents a complete rewrite from the v1 Training Operator, migrating from individual CRDs per framework (TFJob, PyTorchJob, MPIJob) to a unified TrainJob CRD with a plugin-based runtime framework. The core architectural insight is the separation between the *what* (TrainJob spec) and the *how* (TrainingRuntime), allowing users to submit training jobs without understanding the underlying Kubernetes resource topology. The plugin system (`pkg/runtime/framework/`) follows a pipeline pattern: EnforceMLPolicy → EnforcePodGroupPolicy → PodNetwork → ComponentBuilder → Status, where each plugin contributes to the final JobSet configuration through the PodSet abstraction layer.

The controller's use of Server-Side Apply (SSA) for all resource mutations is notable — rather than building full resource objects, the controller constructs ApplyConfigurations that declaratively describe only the fields it owns. This reduces conflict with other controllers and enables clean multi-tenant resource management. The `pkg/apply/` package provides upsert semantics for environment variables, ports, volumes, and mounts that merge rather than replace.

The RHOAI-specific extensions (`pkg/rhai/`) are cleanly separated from upstream code, gated behind the `ENABLE_RHAI_FEATURES` environment variable. The progression tracking system is architecturally interesting: rather than requiring training frameworks to report progress through Kubernetes APIs, it polls an HTTP endpoint (`/metrics`) on the rank-0 training pod at configurable intervals (5-300 seconds). The controller stores progress as a JSON annotation on the pod, making it accessible without custom APIs. This design avoids requiring changes to training framework code — any framework exposing a `/metrics` endpoint with the expected JSON schema gets automatic progress tracking. The NetworkPolicy generation provides defense-in-depth by restricting metrics access to only the controller pod in the controller namespace, even within the cluster network.

The data cache sub-component uses a head-worker architecture with Apache Arrow Flight as the transport layer. The head node reads Iceberg table metadata from S3, partitions data files across workers, and provides flight endpoints to training clients. Workers load assigned Parquet files into DataFusion in-memory tables and serve row-range queries. Worker discovery uses LeaderWorkerSet DNS conventions (`LWS_LEADER_ADDRESS`). This component is architecturally independent from the controller — it runs as a separate binary and communicates only through environment variables set by the dataset-initializer job. However, it lacks a Konflux Dockerfile and is not part of the RHOAI 3.3 release.

The webhook validation strategy is conservative — all three webhooks are validating-only (no mutations) with `failurePolicy: Fail`. This means webhook downtime blocks TrainJob creation, but the readiness probe waits for the webhook server to be fully operational before advertising readiness, preventing premature traffic. Certificate management uses the OPA cert-controller library for automatic rotation without cert-manager dependency, generating a self-signed CA with organization "kubeflow-trainer".
