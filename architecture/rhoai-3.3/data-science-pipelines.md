## Purpose

**Short**: Data Science Pipelines (DSP) is the Kubeflow Pipelines backend for RHOAI, providing a platform for defining, running, scheduling, and managing ML pipelines as Argo Workflows on Kubernetes.

**Detailed**: Data Science Pipelines is a fork of Kubeflow Pipelines (KFP) v2, adapted for Red Hat OpenShift AI. It provides a complete ML pipeline orchestration platform consisting of multiple microservices: an API server that exposes gRPC and REST APIs for pipeline CRUD operations, a driver/launcher system that orchestrates individual pipeline step execution within Argo Workflows, a persistence agent that synchronizes workflow state back to the API server's database, a scheduled workflow controller that manages recurring pipeline runs, and a cache server that acts as a mutating webhook to enable step-level caching.

The system integrates deeply with Argo Workflows as its execution engine, using ML-Metadata (MLMD) for artifact and execution lineage tracking, and S3-compatible object storage (MinIO) for pipeline artifact persistence. In RHOAI, the Data Science Pipelines Operator (DSPO) deploys and manages all DSP sub-components per namespace, but this repository contains only the backend services themselves.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| API Server | Go Service | Central REST/gRPC API for pipeline, run, experiment, and artifact management with database persistence |
| Driver | Go CLI (Argo step) | Orchestrates pipeline step execution — initializes DAGs, resolves inputs, checks cache, patches pod specs |
| Launcher | Go CLI (Argo step) | Executes inside user containers — downloads/uploads artifacts, publishes execution results to MLMD |
| Persistence Agent | Go Controller | Watches Argo Workflow and ScheduledWorkflow resources, reports state to API server for database persistence |
| Scheduled Workflow Controller | Go Controller | Manages ScheduledWorkflow CRD, creates Argo Workflows or API runs on cron/periodic schedules |
| Cache Server | Go Webhook Server | Mutating admission webhook that intercepts Argo Workflow pod creation to enable step-level caching |

## Architectural Analysis

Data Science Pipelines follows a microservices architecture inherited from Kubeflow Pipelines v2, where each component has a distinct responsibility in the pipeline lifecycle. The API server is the central hub — all client interactions (SDK, UI, CLI) flow through its REST/gRPC endpoints, and all internal components (persistence agent, scheduled workflow controller) communicate back to it via gRPC. This hub-and-spoke pattern simplifies authorization (all access goes through one enforcement point) but creates a single point of failure: if the API server is unavailable, pipeline state synchronization and scheduling halt.

The driver/launcher split implements a two-phase execution model within each Argo Workflow step. The driver runs first as an init-style container to resolve inputs, check caches, and patch the pod spec. The launcher then runs inside the user's container to manage artifact I/O and publish execution metadata. This separation allows the orchestration logic to execute before the user's container starts, enabling cache hits to skip the user container entirely by replacing it with a lightweight busybox image. The launcher ships both a standard and FIPS-compliant binary, with the API server injecting the appropriate one based on cluster configuration — this dual-binary approach is notable because misconfiguration could result in a non-FIPS binary running on a FIPS-enabled cluster.

The cache server operates as a Kubernetes mutating admission webhook, intercepting Argo Workflow pod creation requests. When a step matches a cached execution (based on input parameters, container image, and command fingerprint), the webhook mutates the pod spec to replace the user container with a cached result. This is an elegant but potentially fragile pattern — webhook failures cause pod creation failures, and the cache server requires its own database connection and TLS certificates. The cache server listens on port 8443 behind a Kubernetes Service on port 443, using the `webhook-server-tls` Secret for its TLS certificate.

The persistence agent and scheduled workflow controller both follow the standard Kubernetes controller pattern with informers, work queues, and exponential backoff. They authenticate to the API server using projected service account tokens with the `pipelines.kubeflow.org` audience, refreshed hourly. This is a secure pattern — short-lived tokens with audience binding prevent token reuse across services. Both controllers operate in a namespace-scoped manner, watching resources only in their deployment namespace (configurable to all namespaces for multi-user setups).

The system supports two pipeline storage modes: database-backed (default) and Kubernetes-native (using Pipeline and PipelineVersion CRDs). The Kubernetes-native mode is significant for RHOAI because it enables GitOps-style pipeline management and admission webhook validation of pipeline specs. When enabled, the API server registers mutating and validating webhooks that enforce pipeline spec immutability and add ownership labels to PipelineVersion resources.
