## Purpose

**Short**: Manages the lifecycle of Data Science Pipelines (Kubeflow Pipelines v2) custom resources and all associated Kubernetes infrastructure on Red Hat OpenShift AI.

**Detailed**: The Data Science Pipelines Operator (DSPO) is a controller-runtime-based Kubernetes operator that reconciles `DataSciencePipelinesApplication` (DSPA) custom resources. Each DSPA instance represents a complete, namespace-scoped deployment of the Kubeflow Pipelines v2 platform, including the API server, persistence agent, scheduled workflow controller, Argo workflow controller, ML Metadata (MLMD) services, database (MariaDB), and object storage (MinIO). The operator uses the manifestival library to render Go templates into Kubernetes manifests and apply them declaratively.

DSPO supports both managed and external infrastructure backends. Users can deploy operator-managed MariaDB and MinIO instances for quick setup, or point to external databases and S3-compatible object storage for production environments. The operator performs active health checks against database and object storage endpoints during reconciliation, ensuring infrastructure is available before deploying dependent services. All components support Pod-to-Pod TLS encryption, custom CA bundles, and HTTP proxy configuration.

The operator deploys kube-rbac-proxy sidecars alongside the API server and MLMD envoy proxy to enforce Kubernetes RBAC-based authentication on external-facing endpoints. It manages OpenShift Routes for external access, NetworkPolicies for traffic isolation, and a Kubernetes admission webhook for validating and mutating PipelineVersion resources when the Kubernetes-native pipeline store is enabled.

## Architectural Analysis

The Data Science Pipelines Operator follows a template-driven declarative pattern using the manifestival library, which is somewhat unusual among controller-runtime operators. Rather than constructing Kubernetes objects programmatically in Go code, the operator maintains a directory of Go templates (`config/internal/`) that are rendered with parameters extracted from the DSPA CR and then applied as unstructured manifests. This approach provides flexibility — operators can add new resource templates without changing controller logic — but introduces indirection that makes it harder to trace exactly what resources are created from the controller code alone.

The operator's health-check-before-deploy pattern is architecturally significant. During reconciliation, the operator first validates database connectivity (via a MySQL `SELECT 1` query) and object storage accessibility (via a MinIO `StatObject` call) before deploying dependent services like the API server. This prevents the common failure mode where services start but immediately crash because infrastructure isn't ready, reducing unnecessary pod restarts and improving status reporting. The health check results drive both Kubernetes conditions on the DSPA status and Prometheus gauge metrics, giving operators two monitoring surfaces.

The security architecture uses a consistent kube-rbac-proxy sidecar pattern for external-facing endpoints. Both the API server and the MLMD Envoy proxy receive a kube-rbac-proxy container that performs SubjectAccessReview checks against the `datasciencepipelinesapplications/api` resource before proxying requests. This means that access to the pipeline API is governed by standard Kubernetes RBAC policies — users need explicit permission on the DSPA resource to access the API. NetworkPolicies further restrict internal traffic: the MariaDB instance only accepts connections from the operator, API server, and MLMD gRPC server; the MLMD gRPC server only accepts connections from pipeline step pods (labeled `pipelines.kubeflow.org/v2_component: 'true'`) and DSP component pods.

The webhook component has an interesting lifecycle — it is cluster-scoped (MutatingWebhookConfiguration and ValidatingWebhookConfiguration are cluster resources) but is only deployed when at least one DSPA instance has `pipelineStore: "kubernetes"` enabled. The controller tracks all DSPAs with this configuration and only cleans up the webhook when the last such DSPA is deleted. This avoids unnecessary admission overhead when all instances use the default database-backed pipeline store.

One notable architectural risk is the `InsecureSkipVerify: true` in the database health check TLS configuration when TLS verification is disabled. While this is configurable and defaults to secure behavior, the code path exists and could be triggered by user configuration, potentially weakening the security posture. Similarly, the MinIO deployment uses plaintext HTTP on port 9000 without TLS — this is acceptable for the managed in-cluster MinIO instance but means artifact traffic is unencrypted within the cluster when using the default storage backend.
