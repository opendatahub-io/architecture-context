## Purpose

**Short**: OpenVINO Model Server is a high-performance, C++-based AI model inference server that serves models via gRPC and REST APIs compatible with TensorFlow Serving, KServe, and OpenAI protocols.

**Detailed**: OpenVINO Model Server (OVMS) provides a production-grade inference serving solution optimized for Intel architectures. It hosts AI models and makes them accessible over standard network protocols — gRPC and REST — enabling remote inference with lightweight clients. The server supports multiple API standards: TensorFlow Serving v1 for classic prediction workflows, KServe v2 inference protocol for cloud-native inference, and OpenAI-compatible v3 API for large language model (LLM) serving including chat completions, text embeddings, document reranking, image generation, and speech processing.

Within the RHOAI platform, OVMS is deployed as a container image consumed by KServe `ServingRuntime` custom resources. When users create an `InferenceService` targeting the OpenVINO runtime, KServe provisions OVMS pods that load models from local storage, S3, Azure Blob Storage, or Google Cloud Storage. The server uses the OpenVINO inference engine for model execution, supporting formats including OpenVINO IR, ONNX, TensorFlow, TensorFlow Lite, and PaddlePaddle models. For LLM workloads, OVMS implements continuous batching pipelines using OpenVINO GenAI, supporting text generation with streaming via Server-Sent Events.

The server features a modular architecture with optional components: MediaPipe graph execution for composable inference pipelines, Python code execution for custom preprocessing/postprocessing, HuggingFace model pulling for direct model download and conversion, and Prometheus metrics for observability. It is compiled with comprehensive SDL security hardening flags and deployed on UBI 9 minimal base images.

## Architectural Analysis

OpenVINO Model Server follows a modular C++ server architecture with cleanly separated concerns. The Server singleton initializes a chain of modules — MetricModule, ServableManagerModule, GRPCServerModule, HTTPServerModule, and optional PythonInterpreterModule and HFModelPullModule — each implementing a common Module lifecycle interface (initialize/shutdown). This design allows conditional compilation: MediaPipe and Python support can be disabled at build time via Bazel configs (`mp_off_py_off`, `mp_on_py_off`, `mp_on_py_on`), producing smaller binaries for constrained deployments.

The multi-API design is a defining architectural feature. OVMS simultaneously serves three distinct API families on separate URL namespaces: TensorFlow Serving v1 (`/v1/...`) for backward compatibility with TFS clients, KServe v2 (`/v2/...`) for cloud-native inference protocol compliance, and OpenAI-compatible v3 (`/v3/...`) for LLM and generative AI workloads. The v1/v2 APIs route directly to model inference, while v3 endpoints flow through MediaPipe graph executors that compose LLM calculators, embedding calculators, rerank calculators, and image generation calculators. This layering means that generative AI features (chat, embeddings, reranking, image generation, audio) are MediaPipe graph nodes rather than hardcoded HTTP handlers — enabling flexible composition and pipeline customization.

The continuous batching pipeline for LLM workloads deserves attention. OVMS wraps OpenVINO GenAI's ContinuousBatchingPipeline to batch multiple inference requests at the token generation level rather than the request level. This is architecturally significant because it allows high throughput with concurrent users without requiring explicit batching logic from clients. The four pipeline types (LM, VLM, LM_CB, VLM_CB) cover text-only and multimodal (vision-language) models in both single-request and continuous-batching modes. Streaming responses use Server-Sent Events (SSE) for token-by-token delivery.

The storage backend abstraction is well-designed — a common FileSystem interface with implementations for local filesystem, S3, Azure Blob Storage, and GCS. However, authentication for these backends is handled through environment variables and implicit credentials (IAM roles, managed identities) rather than through Kubernetes secrets directly, which means operators deploying OVMS must configure credentials at the pod level. The HuggingFace pull module adds a fourth model source, supporting Git clone (via libgit2), Optimum CLI export (for format conversion), and GGUF download for quantized models.

A notable security consideration is that OVMS uses `grpc::InsecureServerCredentials()` by default for its gRPC server, and the REST server (Drogon) also defaults to plaintext HTTP. This is by design — in RHOAI deployments, TLS termination happens at the KServe gateway (Envoy) or via kube-rbac-proxy sidecars, so OVMS operates within a trusted pod network. The v3 API endpoints support API key authentication via Bearer tokens, but v1/v2 APIs rely entirely on infrastructure-level authentication. The `extras/nginx-mtls-auth/` directory provides a reference architecture for mutual TLS authentication using nginx as a sidecar proxy, though this is not the default deployment pattern in RHOAI.
