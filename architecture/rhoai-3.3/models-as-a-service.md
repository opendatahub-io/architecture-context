## Purpose

**Short**: MaaS API is a Go-based REST service that provides tier-based token management, API key lifecycle, and model catalog endpoints for the Models as a Service platform on Red Hat OpenShift AI.

**Detailed**: The MaaS API acts as the central management plane for the Models as a Service platform. It sits behind a Kuadrant-managed Gateway API entry point and provides three primary capabilities: (1) tier-based ephemeral token issuance, where users are mapped to subscription tiers (free, premium, enterprise) based on their Kubernetes group memberships and receive time-limited ServiceAccount tokens scoped to their tier; (2) named API key management with persistent metadata tracking via SQLite or PostgreSQL backends; and (3) an OpenAI-compatible model catalog endpoint that lists available LLMInferenceService and InferenceService resources, filtering by user authorization through a loopback auth check against the gateway's AuthPolicy.

The service integrates deeply with Kubernetes-native constructs — it dynamically creates Namespaces and ServiceAccounts per tier, issues tokens via the Kubernetes TokenRequest API, and reads tier configuration from a ConfigMap. Authentication is handled externally by Kuadrant/Authorino, which validates OpenShift tokens and injects user identity headers (`X-MaaS-Username`, `X-MaaS-Group`) before requests reach the MaaS API. Rate limiting is enforced at the gateway level by Limitador via TokenRateLimitPolicy CRs, providing per-user, per-tier token usage quotas.

## Architectural Analysis

The MaaS API adopts a **gateway-centric architecture** where authentication, authorization, and rate limiting are fully delegated to the Kuadrant/Authorino/Limitador stack. The API itself never validates tokens — it relies entirely on the gateway's AuthPolicy to inject trusted headers (`X-MaaS-Username`, `X-MaaS-Group`) after Kubernetes TokenReview. This design keeps the API stateless with respect to authentication and allows policy changes without redeploying the service. However, it means the API is completely dependent on correct header injection — missing or malformed headers result in 500 errors, not 401/403, which could confuse API consumers.

The **tier-based namespace isolation** pattern is noteworthy: when a user requests a token, the API creates a namespace named `{instance}-tier-{tier}` and a ServiceAccount named after a sanitized hash of the username within it. Tokens are then issued via the Kubernetes TokenRequest API scoped to a custom audience (`{instance}-sa`). This approach leverages Kubernetes RBAC for model access control — administrators grant the tier namespace's ServiceAccount group access to specific LLMInferenceService resources, and the gateway's AuthPolicy enforces SubjectAccessReview checks on inference requests. The tradeoff is that the MaaS API requires cluster-wide permissions to create namespaces and ServiceAccounts.

The **model listing flow** (`/v1/models`) is architecturally complex: it uses shared informers to watch LLMInferenceService and HTTPRoute resources across all namespaces, filters for models associated with the configured gateway, then performs per-model authorization checks via HTTP loopback requests to each model's endpoint. The loopback pattern — making a GET request to the model's `/v1/models` endpoint with the user's SA token and interpreting 2xx/405 as authorized, 401/403 as denied — reuses the gateway's existing AuthPolicy as the single source of truth for authorization rather than duplicating RBAC logic. The `InsecureSkipVerify: true` in the loopback HTTP client is a pragmatic choice for internal cluster traffic where custom/self-signed TLS certificates are common, though it constitutes a FIPS gap at the application level.

The **storage architecture** supports three modes — in-memory (default, ephemeral), disk (SQLite file, single replica), and external (PostgreSQL, multi-replica). The SQLite store uses WAL journal mode with a single connection to avoid locking. PostgreSQL connection pooling is configurable via environment variables. Both backends share the same `MetadataStore` interface, and the schema is intentionally simple (a single `tokens` table). Token revocation works by recreating the ServiceAccount (invalidating all Kubernetes-issued tokens) and marking API key metadata as expired in the database. This is a bulk operation — per-key revocation is not supported.

The **kustomize structure** is well-layered: the base provides HTTP mode with in-memory storage, the TLS overlay adds HTTPS with OpenShift service-ca certificates and an Istio DestinationRule for encrypted backend traffic, and distribution overlays (ODH, OpenShift) add auth policies, rate limiting, observability, and parameterized configuration. The OpenShift overlay is the most complete — it includes the Gateway CR, TokenRateLimitPolicy, ServiceMonitor, and TelemetryPolicy. The separation allows incremental deployment during development while providing full policy enforcement in production.
