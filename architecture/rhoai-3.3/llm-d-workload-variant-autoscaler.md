## Purpose

**Short**: GPU-aware Kubernetes autoscaler that determines optimal replica counts for LLM inference model server variants based on saturation metrics, queueing theory, and cost optimization.

**Detailed**: The Workload Variant Autoscaler (WVA) is a Kubernetes-native controller that monitors LLM inference workloads (vLLM servers) via Prometheus metrics and calculates optimal replica counts. It introduces the concept of a "variant" — a specific combination of model, hardware (GPU type), runtime configuration, and serving approach. When multiple variants serve the same model, WVA performs cost-aware cross-variant optimization, preferring to scale cheaper variants up and expensive ones down.

WVA operates alongside standard Kubernetes HPA and external autoscalers like KEDA. Rather than directly scaling deployments, it writes desired replica counts to VariantAutoscaling CR status fields and emits Prometheus metrics that HPA/KEDA consume to drive actual scaling. The system supports three scaling analyzer paths: a percentage-based V1 analyzer (KV cache % and queue length), a token-based V2 saturation analyzer (capacity modeling with k1 memory and k2 compute constraints), and a queueing-model analyzer (SLO-driven scaling using online Kalman filter parameter learning). It also provides scale-from-zero capability by monitoring the llm-d inference scheduler's flow-control queue for pending requests against idle variants.

The controller supports Deployment and LeaderWorkerSet (LWS) scale targets, multi-vendor GPU discovery (NVIDIA, AMD, Intel), namespace-level configuration hierarchy with per-model overrides, prefill/decode disaggregated pipeline awareness, and multi-controller isolation for shared clusters.

## Architectural Analysis

The WVA architecture follows a multi-engine, multi-analyzer pattern that separates concerns cleanly: the Kubernetes reconciler handles CR lifecycle and status updates, while independent optimization engines run on separate polling loops and communicate decisions through an in-memory decision cache and event trigger channel. This decoupling means the controller never blocks on expensive Prometheus queries — the saturation engine runs its 30-second cycles independently, and the reconciler picks up cached decisions on its next pass.

The three-analyzer design (V1 percentage-based, V2 token-capacity, Queueing Model) reflects an evolution path rather than competing alternatives. V1 is the simplest (KV cache % and queue length thresholds), V2 introduces a capacity model with k1 (memory-bound) and k2 (compute-bound) constraints for more accurate token-level scaling, and the Queueing Model analyzer uses online Kalman filter parameter learning to derive SLO-based scaling targets. Analyzer selection is dynamic per-model based on ConfigMap presence: the queueing model config ConfigMap acts as a feature gate. The V2 analyzer includes a sophisticated capacity store that caches knowledge about zero-replica variants by parsing vLLM command-line arguments from deployment specs — enabling capacity estimation even when no metrics are available.

The scale-from-zero engine operates on a fundamentally different timescale (100ms vs 30s) because latency sensitivity is critical when a user's first request arrives at an idle model. It bypasses Prometheus entirely, scraping EPP pod metrics directly for flow-control queue sizes. The combination of scale-from-zero (fast detection, coarse decision: 0→1) and saturation engine (slower detection, fine-grained decision: N→M) provides a two-tier response system.

The cost-aware optimization pipeline is a notable architectural choice. By representing each variant with an explicit cost (from CR spec) and capacity (from metrics analysis), the CostAwareOptimizer can make cross-variant decisions: scaling up the cheapest variant first and scaling down the most expensive. This enables multi-tier GPU deployments (e.g., H100 premium + A100 standard + T4 budget) to be cost-optimized automatically. The GPU limiter (optional DefaultLimiter) adds a physical constraint layer, using a GreedyBySaturation algorithm to allocate limited GPU inventory to the most saturated models first.

The configuration system is notably sophisticated for a controller. It provides namespace-level configuration hierarchy (global → namespace → model-specific) for scaling thresholds, immutable parameter detection that prevents unsafe runtime changes to connection endpoints and TLS paths, and a bootstrap sequence that must complete before the readiness probe passes. The controller watches its own ConfigMaps as first-class resources with a dedicated ConfigMapReconciler, enabling live threshold tuning without pod restarts. The prefill/decode disaggregation support (detecting P/D roles from pod labels) shows awareness of the emerging disaggregated inference architecture pattern where prefill and decode phases run on separate GPU pools with independent scaling needs.
