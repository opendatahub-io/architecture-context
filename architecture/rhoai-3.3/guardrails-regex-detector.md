## Purpose

**Short**: A lightweight Rust HTTP service that detects PII (personally identifiable information) and custom patterns in text using regex matching, serving as a detector backend for the FMS Guardrails Orchestrator.

**Detailed**: The Guardrails Regex Detector is a stateless REST API service built with Axum (Rust async web framework) that scans text content for sensitive patterns using regular expressions. It provides three built-in PII detectors — email addresses, Social Security numbers, and credit card numbers — and also accepts arbitrary custom regex patterns from callers.

The service is designed to integrate with the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator), which delegates content detection tasks to specialized detector services. The orchestrator sends text content along with a list of regex identifiers (built-in names like `"email"`, `"ssn"`, `"credit-card"`, or raw regex strings) and receives back a list of matches with character positions, detection types, and confidence scores. This architecture allows the guardrails system to enforce content policies on LLM inputs and outputs by identifying PII before it reaches or leaves a model.

The service is intentionally minimal — approximately 170 lines of Rust across two source files — with no database, no persistent state, and no authentication. It is designed to run as an internal sidecar or backend service within the guardrails subsystem, fronted by the orchestrator which handles external-facing concerns.

## Architectural Analysis

The Guardrails Regex Detector follows an extremely minimalist architecture — under 170 lines of Rust across two files (`main.rs` and `detectors.rs`). The design separates concerns cleanly: `main.rs` handles HTTP server setup, configuration, and routing, while `detectors.rs` contains all detection logic including built-in PII patterns and custom regex handling. The service is fully stateless — each request is self-contained with no database, cache, or session state.

The detector architecture uses a function-pointer dispatch pattern (lines 106-113 of `detectors.rs`) where built-in detector names (`"email"`, `"ssn"`, `"credit-card"`) map to dedicated functions that encode specific regex patterns and metadata. When a regex name doesn't match a built-in, it falls through to a generic `regex_match` function that treats the string as a raw regex pattern. This two-tier approach allows adding new built-in detectors without changing the API contract, while still supporting arbitrary user-defined patterns. All detections return a uniform response structure with character-level position tracking (`start`/`end`), detection type classification (`"pii"` vs `"custom"`), matched text, and a hardcoded confidence score of 1.0.

A notable security consideration is the complete absence of authentication or TLS at the service level — the service binds to `0.0.0.0:8080` with plaintext HTTP and no auth middleware. This is by design: the service is intended to run as an internal backend behind the FMS Guardrails Orchestrator, which handles external-facing security. However, since the service processes PII-containing text (email addresses, SSNs, credit card numbers), the lack of transport encryption means sensitive data traverses the network in plaintext. In a production deployment, this should be mitigated by network policies, service mesh mTLS (e.g., Istio sidecar injection), or deployment within a trusted pod network. The Dockerfile also has a security note: no `USER` directive is set (as flagged in `component-architecture.json`), meaning the container runs as root by default.

The build pipeline uses a multi-stage Dockerfile with separate stages for building, testing (`cargo test`), linting (`cargo clippy`), and formatting checks (`cargo fmt --check`). The final release image is based on UBI9 minimal (`registry.access.redhat.com/ubi9/ubi-minimal`), which is standard for RHOAI components. However, the absence of a `Dockerfile.konflux` variant is unusual for a RHOAI 3.3 component — most RHOAI components at this version level have transitioned to Konflux builds. This may indicate the component is newer or follows a different build pipeline arrangement.
