## Purpose

**Short**: An out-of-tree Llama Stack evaluation provider that integrates NVIDIA Garak for automated LLM red-teaming, vulnerability scanning, and guardrail (shield) effectiveness testing.

**Detailed**: This component implements the Llama Stack `Eval` API as a provider plugin, enabling automated security scanning of Large Language Models through the Garak vulnerability scanner framework. It supports two execution modes: **inline** (local process execution) and **remote** (Kubeflow Pipelines-based execution on OpenShift AI / Data Science Pipelines).

The provider exposes predefined benchmark profiles aligned with industry security frameworks — OWASP LLM Top 10, AVID Security/Ethics/Performance taxonomies — and supports custom benchmark registration with specific Garak probes. When shields (guardrails) are configured, the provider orchestrates a function-based scanning mode that tests prompt→shield→LLM→shield→response chains, measuring how effectively guardrails block adversarial inputs.

In remote mode, the provider creates Kubeflow Pipeline runs with three stages: input validation, Garak scan execution, and results parsing. Scan artifacts (reports, hit logs, AVID-format reports, HTML visualizations) are uploaded to the Llama Stack Files API and retrievable through the standard job status/result APIs. The container image is built on UBI9 Python 3.12 and ships as `odh-trustyai-garak-lls-provider-dsp-rhel9`.

## Architectural Analysis

This component follows a **dual-execution architecture** with inline and remote modes sharing a common base class (`GarakEvalBase`). The base class handles benchmark registration, probe validation, Garak command construction, and result parsing. The inline adapter (`GarakInlineEvalAdapter`) manages local subprocess execution with asyncio-based concurrency control (semaphore-limited to `max_concurrent_jobs`), while the remote adapter (`GarakRemoteEvalAdapter`) delegates execution to Kubeflow Pipelines, creating a three-stage DAG (validate → scan → parse) with GPU/CPU branching.

The **shield orchestration pattern** is architecturally notable. When benchmarks include `shield_ids` or `shield_config`, the provider switches from an OpenAI-compatible generator (`openai.OpenAICompatible`) to a function-based generator (`function.Single`). The `SimpleShieldOrchestrator` class implements the input-shield → LLM → output-shield chain as a callable that Garak invokes per probe attempt. This allows measuring how effectively guardrails block adversarial prompts without modifying Garak's core scanning logic. The orchestrator is a singleton (`simple_shield_orchestrator`) that maintains a persistent `LlamaStackClient` connection across invocations.

The **remote execution path** has an interesting results-retrieval pattern. Since KFP pods and the Llama Stack server are separate processes, the pipeline's `parse_results` component uploads a `{job_id}_mapping.json` file to the Files API with purpose `batch`. When the server polls for job status, it searches the Files API for this predictable filename to retrieve the mapping between scan artifact filenames and their file IDs. This polling-based approach avoids direct pod-to-server callbacks but introduces latency proportional to the files list scan.

A **compatibility layer** (`compat.py`) abstracts Llama Stack API imports across two package layouts — the newer `llama_stack_api` package and the legacy `llama_stack` namespace. This allows the provider to work with both old and new versions of the framework, which is important for the RHOAI downstream where framework versions may lag upstream.

The **Cargo.toml and `.konflux/` workarounds** are architecturally interesting from a build-system perspective. The repository contains a dummy Rust project (`Cargo.toml` referencing `.konflux/main.rs` — an empty `fn main() {}`) solely to work around Hermeto build tool bugs. This dummy project's `Cargo.lock` enables Hermeto to prefetch Rust dependencies (like `fancy-regex`, `regex`) that are transitive build dependencies of Python packages (likely `tokenizers` or similar). The `.konflux/prep-hermeto.sh` script copies x86_64 requirements to s390x and ppc64le directories as another workaround for Hermeto's architecture-specific sdist fetching bug.
