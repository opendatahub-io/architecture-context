## Purpose

**Short**: A Kubernetes operator that manages the lifecycle of MLflow tracking server instances on OpenShift AI, deploying and configuring MLflow as a platform-integrated experiment tracking and model registry service.

**Detailed**: The MLflow Operator is a controller-runtime based Go operator that watches `MLflow` custom resources (cluster-scoped) and reconciles the full deployment stack for each MLflow instance. It uses an embedded Helm chart to render Kubernetes resources (Deployment, Service, ServiceAccount, PVC, NetworkPolicy, ClusterRole, ClusterRoleBinding) and applies them via Server-Side Apply. The operator also manages Gateway API HTTPRoute resources for platform ingress integration and OpenShift ConsoleLink resources for console UI integration.

The operator is designed for the RHOAI 3.x ingress pattern: it creates HTTPRoute resources that reference the platform's `data-science-gateway` Gateway, enabling MLflow to be exposed through the shared platform ingress with URL path-based routing (e.g., `/mlflow/api`, `/mlflow/v1`). The MLflow server itself handles TLS termination using OpenShift service-ca provisioned certificates and implements Kubernetes-native authentication via `self_subject_access_review` authorization mode.

The operator supports flexible storage backends including SQLite (local PVC), PostgreSQL, MySQL, and cloud object stores (S3, GCS) for artifacts. It exposes MLflow's workspace feature using Kubernetes namespace-based workspace isolation. Configuration is driven by the MLflow CR spec and operator-level environment variables sourced from a ConfigMap generated by kustomize.

## Architectural Analysis

The MLflow Operator follows a Helm-driven reconciliation pattern that is somewhat unusual for controller-runtime operators. Rather than imperatively constructing Kubernetes resources in Go code, the operator embeds a full Helm chart (`charts/mlflow/`) and uses the Helm v3 rendering engine to template resources from the MLflow CR spec. This approach provides a clean separation between the operator's reconciliation logic and the resource definitions, making it easy to add or modify deployed resources without changing controller code. The rendered resources are applied via Server-Side Apply with `ForceOwnership`, which provides clean conflict resolution when resources are modified externally.

The operator implements a conditional feature detection pattern for optional cluster capabilities. At startup, it uses the Kubernetes discovery API to check whether `ConsoleLink` (OpenShift) and `HTTPRoute` (Gateway API) CRDs are available. If absent, the operator gracefully skips creating those resources and does not add them to the controller's watch list or informer cache. This makes the operator portable across OpenShift (with ConsoleLink) and vanilla Kubernetes (without it), and across clusters with and without Gateway API installed. The cache is further optimized with label selectors (`app: mlflow`) and namespace restrictions to minimize memory consumption.

The shared ClusterRole pattern for the `mlflow` ClusterRole is noteworthy. Since the CR is cluster-scoped, multiple MLflow instances could theoretically exist, and all share a single `mlflow` ClusterRole (used by the MLflow server for namespace listing in the workspaces feature). The controller uses `appendOwnerReference` instead of `SetControllerReference` for this resource, allowing multiple MLflow CRs to co-own the same ClusterRole without conflicting controller references. A corresponding `Watches` handler (instead of `Owns`) maps ClusterRole events back to all owning MLflow instances.

The ingress architecture follows the RHOAI 3.x pattern: the operator creates an HTTPRoute with three rules — one for `/mlflow/api` (rewritten to `/api`), one for `/mlflow/v1` (rewritten to `/v1`), and one catch-all for `/mlflow` — all routing to the MLflow Service on port 8443. The HTTPRoute references the platform's `data-science-gateway` Gateway in the `openshift-ingress` namespace. This enables path-based multiplexing alongside other RHOAI components (dashboard, model registry, etc.) on the same external hostname. The MLflow server runs TLS natively using uvicorn with service-ca provisioned certificates, so traffic from the Gateway to the backend is encrypted end-to-end.

The MLflow server's authorization model integrates deeply with Kubernetes RBAC. The operator sets `MLFLOW_K8S_AUTH_AUTHORIZATION_MODE=self_subject_access_review` and deploys aggregate ClusterRoles (`mlflow-view`, `mlflow-edit`) that are automatically aggregated into the standard `view`, `edit`, and `admin` ClusterRoles. This means cluster users with existing RBAC roles automatically receive appropriate MLflow permissions without additional configuration. The aggregate roles also reference `mlflow.kubeflow.org` API group resources (experiments, registeredmodels, jobs), suggesting forward compatibility with future Kubeflow-aligned MLflow CRDs.
