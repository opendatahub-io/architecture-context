## Purpose

**Short**: NeMo Guardrails is a Python-based toolkit for adding programmable safety guardrails to LLM-based conversational systems, deployed as a FastAPI server within RHOAI.

**Detailed**: NeMo Guardrails, originally developed by NVIDIA, provides a comprehensive framework for implementing safety and content moderation controls around Large Language Model (LLM) interactions. The toolkit supports multiple types of guardrails — input validation, output filtering, dialog management, retrieval augmentation control, and tool usage validation — that can be composed and configured through YAML configuration files and the Colang domain-specific language (supporting both v1.0 and v2.x versions).

In the RHOAI 3.3 distribution, NeMo Guardrails is packaged as a container image (`odh-trustyai-nemo-guardrails-server-rhel9`) that runs as a FastAPI HTTP server on port 8000. The server exposes a chat completions API (`/v1/chat/completions`) that accepts user messages, applies configured guardrails pipelines (input rails → dialog rails → LLM generation → output rails), and returns safe responses. The image pre-downloads required ML models (sentence transformers, spaCy NLP models, FastEmbed) at build time so it can operate without internet access at runtime. The server integrates with external LLM providers (OpenAI, NVIDIA NIM, Anthropic, Cohere, etc.) via LangChain and supports conversation thread persistence via Redis or in-memory storage.

The component is part of the TrustyAI ecosystem within RHOAI, providing guardrails capabilities for AI safety and responsible AI use. The build profile system (`opensource` vs `all`) controls which guardrail libraries are included, with closed-source integrations (ActiveFence, Cleanlab, Clavata, PrivateAI, Fiddler, PatronusAI, Prompt Security, GCP) excluded in the open-source profile used for RHOAI builds.

## Architectural Analysis

NeMo Guardrails follows a pipeline architecture where user messages flow through a configurable sequence of "rails" — input rails, dialog rails, retrieval rails, generation, and output rails — before producing a response. Each rail type can invoke LLM calls for self-checking, run local ML models (spaCy, sentence transformers), or call external moderation APIs. This pipeline design allows flexible composition: operators can enable or disable specific guardrails through YAML configuration without modifying code. The Colang DSL provides an additional layer of expressiveness, allowing conversational flow definitions that control dialog behavior at a higher abstraction level than raw code.

The RHOAI distribution applies a build-time profile filtering mechanism (`scripts/filter_guardrails.py`) that removes closed-source guardrail libraries (ActiveFence, Cleanlab, Clavata, PrivateAI, Fiddler, PatronusAI, Prompt Security, GCP) from the image. This means the RHOAI image ships only with open-source guardrail implementations: injection detection, jailbreak detection, hallucination detection, sensitive data detection (via Presidio), self-check, content safety, LLaMA Guard, AI defense, attention, autoalign, factchecking, topic safety, Trend Micro, guardrails_ai, and Pangea. The `scripts/pre_download_required_models.py` script discovers and pre-downloads all ML models needed by the active guardrails, making the runtime image self-contained.

The server architecture is intentionally simple — a single FastAPI process with no built-in authentication, TLS, or Kubernetes-native features. Security is expected to be provided by the deployment infrastructure: kube-rbac-proxy sidecars for authentication, network policies for isolation, and Gateway API / Service Mesh for TLS termination. The CORS middleware is disabled by default and must be explicitly enabled via environment variables. The server supports conversation threading with a pluggable DataStore abstraction (in-memory or Redis), but thread IDs require a minimum 16-character length and are stored with a `thread-` prefix. The streaming implementation uses Server-Sent Events (SSE) with `[DONE]` terminators, compatible with the OpenAI API streaming format.

A notable FIPS concern exists in `nemoguardrails/embeddings/cache.py:71` where `hashlib.md5()` is called directly without the SHA256 fallback that `nemoguardrails/utils.py:409-417` implements. In FIPS-enforced environments where MD5 is blocked, the embedding cache key generation would raise a `ValueError`. The `utils.py` implementation correctly handles this with a try/except fallback, but the embeddings cache does not follow the same pattern. Additionally, the Tekton PipelineRun sets `hermetic: false`, meaning the Konflux build has network access — Python packages are downloaded from PyPI and ML models from HuggingFace Hub during build, rather than being fully pre-fetched via Hermeto.

The component uses a multi-stage Docker build (build + runtime) based on UBI9 Python 3.12, running as non-root user 1001 with group 0 permissions for OpenShift compatibility. The entrypoint script validates that configuration files (`config.yaml` and `rails.co`) exist before starting the server, expecting these to be mounted from ConfigMaps or Persistent Volumes at `/app/config/<CONFIG_ID>/`. The auto-reload feature (via watchdog) supports development workflows where configuration changes are detected and applied without server restart.
