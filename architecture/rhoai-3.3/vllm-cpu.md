## Purpose

**Short**: CPU-optimized build of vLLM, a high-throughput inference and serving engine for Large Language Models, deployed as an OpenAI-compatible API server.

**Detailed**: vllm-cpu is a Red Hat downstream fork of the vLLM project, specifically built to run LLM inference on CPU hardware without requiring GPUs. It provides an OpenAI-compatible REST API server that supports chat completions, text completions, embeddings, reranking, scoring, classification, audio transcription/translation, and the Anthropic Messages API. The server uses FastAPI with Uvicorn and uvloop for high-performance async HTTP handling.

The component is deployed as a container image (`odh-vllm-cpu-rhel9`) built on Red Hat UBI 9 Minimal. It supports multiple CPU architectures (x86_64, ppc64le, s390x) with architecture-specific build scripts and optimizations (Intel OpenMP for x86_64, OpenBLAS for ppc64le/s390x). The vLLM engine uses PyTorch as the computation backend with CPU-specific tensor operations and attention mechanisms.

Within the RHOAI platform, vllm-cpu serves as one of the available model serving runtimes, typically deployed via KServe as an InferenceService. It loads transformer models from Hugging Face Hub or local storage and exposes them through standardized API endpoints, enabling applications to perform inference against LLMs without GPU infrastructure.

## Architectural Analysis

The vllm-cpu component is a direct fork of the upstream vLLM project, adapted for Red Hat's distribution requirements (UBI base image, Konflux build pipeline, multi-architecture support). The codebase is large — over 200,000 lines of Python — but the architecturally significant portion for RHOAI integration is concentrated in the `vllm/entrypoints/` directory, which implements the HTTP API surface and server lifecycle.

**API Design**: The server implements a comprehensive API surface spanning three protocol families: OpenAI (chat completions, completions, embeddings, responses, audio), Anthropic (messages), and SageMaker (ping, invocations). All APIs are mounted on a single FastAPI application with a modular router architecture. The API handlers are conditionally initialized based on model capabilities — a model that only supports embeddings won't expose chat completion endpoints. Authentication is handled by an ASGI middleware (`AuthenticationMiddleware`) that uses SHA-256 hashed API keys with timing-safe comparison, but only protects `/v1/*` paths, leaving health/metrics/admin endpoints open.

**Engine Architecture**: The inference engine uses a V1 architecture (`AsyncLLM`) that runs in-process with the FastAPI server by default (with an option for frontend multiprocessing via RPC). The engine manages model loading, KV-cache allocation, request scheduling, and token generation. For CPU inference specifically, PyTorch with CPU-specific tensor operations is used instead of CUDA kernels. The engine supports several advanced features: speculative decoding (via numba for n-gram), LoRA adapter hot-loading, structured output enforcement (via outlines/xgrammar/lm-format-enforcer), and disaggregated prefill/decode.

**Multi-Architecture Support**: The Konflux Dockerfile (`Dockerfile.konflux.cpu`) is a single multi-architecture file that detects the platform via `uname -m` and runs architecture-specific build scripts (`build_vllm_ppc64le.sh`, `build_vllm_s390x.sh`). Each architecture has its own dependency chain — x86_64 uses Intel OpenMP, ppc64le/s390x build OpenBLAS and numactl from source. This pattern is robust but adds build complexity. The s390x support notably uses a different PyTorch version (2.10.0+cpu vs 2.9.1+cpu) due to upstream compatibility constraints.

**Security Considerations**: The server runs as a non-root user (UID 2000, GID 0) following OpenShift security best practices. TLS is optional and handled at the application level with support for dynamic certificate refresh via file watchers. In RHOAI deployments, TLS is typically terminated at the platform Gateway or kube-rbac-proxy sidecar level. The `VLLM_NO_USAGE_STATS=1` and `VLLM_USE_V1=1` environment variables are hardcoded in the Dockerfile, disabling telemetry and enabling the V1 engine architecture respectively. The container image sets `HF_HUB_OFFLINE=0` (allowing online model downloads), which differs from the GPU variant that sets it to `1` — CPU deployments may need model pre-loading in air-gapped environments.

**Observability**: Prometheus metrics are exposed at `/metrics` via the `prometheus-fastapi-instrumentator` library, which automatically instruments all API routes (excluding health/metrics/load/ping/version). The ORCA load reporting format is supported via response headers for integration with load balancers. A watchdog task monitors engine health every 5 seconds and triggers server shutdown if the engine dies (unless `VLLM_KEEP_ALIVE_ON_ENGINE_DEATH` is set).
