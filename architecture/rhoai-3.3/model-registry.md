## Purpose

**Short**: Central metadata registry for ML models, providing a REST API for storing and managing models, versions, artifacts, and inference service metadata across the AI/ML platform.

**Detailed**: Model Registry is a Kubeflow-originated component that serves as the central repository for AI/ML model metadata in Red Hat OpenShift AI. It provides a REST API (OpenAPI v1alpha3) backed by an embedded SQL database (MySQL or PostgreSQL) for managing the complete model lifecycle: registered models, model versions, model artifacts, serving environments, inference services, experiments, and experiment runs.

The component ships multiple deployable artifacts: (1) the core REST API proxy server that fronts the SQL database, (2) a Kubernetes controller that synchronizes KServe InferenceService CRDs with Model Registry metadata, (3) a Container Storage Initializer (CSI) plugin that enables KServe to download model artifacts from Model Registry using the `model-registry://` URI scheme, and (4) an async-upload Kubernetes Job that copies models between storage backends (S3, OCI, HuggingFace) and registers them in the registry. The component also has a catalog sub-component and a UI client (BFF + React frontend) that are co-located in the repository.

Model Registry integrates closely with KServe for model serving, the RHOAI dashboard for the user interface, and Istio for service mesh networking. The data model is based on ML Metadata (MLMD) type conventions with Kubeflow-prefixed type names.

## Architectural Analysis

Model Registry follows a modular architecture with multiple independently deployable components sharing a single repository. The core proxy server implements a REST API generated from an OpenAPI 3.0.3 specification (v1alpha3), using the go-chi router with CORS middleware and auto-generated server stubs. The data model is rooted in ML Metadata (MLMD) conventions, with Kubeflow-prefixed type names (e.g., `kf.RegisteredModel`, `kf.ModelVersion`) stored in a relational database via GORM. The proxy uses an interesting asynchronous startup pattern: an initial handler returns HTTP 503 until the database connection and migrations are complete, at which point the dynamic router swaps in the actual API handler atomically. This ensures the readiness probe (`/readyz/isDirty`) reflects actual database state including migration version and dirty flag.

The InferenceService controller implements a bidirectional sync between KServe InferenceService CRDs and Model Registry metadata. When users annotate InferenceServices with Model Registry labels (`modelregistry.kubeflow.org/namespace`, `modelregistry.kubeflow.org/name`, etc.), the controller discovers the corresponding Model Registry service (either by annotation or by searching for services with label `component=model-registry`), then creates or updates ServingEnvironment, InferenceService, and associated metadata in the registry. On deletion, it sets the desired state to UNDEPLOYED rather than deleting the record, preserving audit history. The controller uses a finalizer pattern for cleanup. A notable security concern is that the controller's HTTP client for Model Registry API calls has `InsecureSkipVerify: true` hardcoded (pkg/inferenceservice-controller/controller.go:55), which skips TLS certificate verification regardless of the `SKIP_TLS_VERIFY` environment variable — the env var only affects whether InsecureSkipVerify is set on the transport's TLSClientConfig, but the hardcoded value overrides it.

The CSI storage initializer enables a clean integration with KServe's model serving pipeline. It registers as a `ClusterStorageContainer` supporting the `model-registry://` URI prefix. When KServe needs to serve a model, the CSI plugin parses the URI to extract the model name and optional version, queries the Model Registry API for the artifact URI (e.g., `s3://bucket/model.tar.gz`), then delegates the actual download to KServe's native protocol-specific providers. This two-stage resolution (registry lookup → protocol delegation) allows model URIs to be stable references that survive artifact relocation.

The async-upload job represents a newer addition to the ecosystem, designed for asynchronous model lifecycle operations. It supports three intents: creating a new model with its first version, adding a version to an existing model, or updating an existing artifact's URI. The job downloads from various sources (S3, OCI, HuggingFace Hub, HTTP/HTTPS with auto-archive extraction), uploads to OCI registries via skopeo, and registers the resulting URI in Model Registry. The job uses Kubernetes-native patterns: credentials via mounted Secrets, metadata via mounted ConfigMaps, and error reporting via termination messages. Notably, the HuggingFace Hub dependency is capped at <0.31 to avoid the hf-xet Rust dependency, which breaks hermetic Konflux builds with `pip --no-binary :all:`.

The kustomize manifest structure shows clear separation between the base deployment (proxy + service), database variants (MySQL vs PostgreSQL), and optional features (Istio, controller, CSI, catalog, UI). Database deployments explicitly disable Istio sidecar injection and exclude database ports from Istio traffic management, ensuring database connections are not disrupted by mesh proxy behavior. The Istio option configures ISTIO_MUTUAL mTLS on the DestinationRule and exposes the API through a VirtualService matching `/api/model_registry/` prefixes.
