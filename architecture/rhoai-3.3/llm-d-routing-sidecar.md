## Purpose

**Short**: A reverse proxy sidecar that routes inference requests between prefill and decode workers in a disaggregated prefill/decode (P/D) LLM serving architecture.

**Detailed**: The llm-d-routing-sidecar is a lightweight HTTP/HTTPS reverse proxy designed to run as a sidecar container alongside a vLLM decoder instance. It intercepts incoming inference requests (OpenAI-compatible `/v1/chat/completions` and `/v1/completions` endpoints) and orchestrates the disaggregated prefill/decode protocol. When a request includes an `x-prefiller-host-port` header, the sidecar first forwards the request to the specified remote prefill worker, collects the KV-cache transfer parameters from the prefill response, then forwards the augmented request to the local decoder for final completion generation.

The sidecar supports three P/D connector protocols: NIXLv2 (current, default), NIXLv1 (deprecated), and LMCache (deprecated). It provides TLS support for both inbound connections (secure proxy with configurable certificates or self-signed fallback) and outbound connections to prefill/decode workers. A Server-Side Request Forgery (SSRF) protection mechanism is available that watches Kubernetes InferencePool resources to maintain an allowlist of valid prefill targets based on pod IPs and hostnames.

**Note**: This repository is deprecated. The code has been moved to the [llm-d-inference-scheduler](https://github.com/llm-d/llm-d-inference-scheduler) repository under `cmd/pd_sidecar` and `pkg/sidecar` directories. All future development occurs there.

## Architectural Analysis

The llm-d-routing-sidecar implements a clean sidecar proxy pattern for disaggregated prefill/decode LLM serving. The architecture is intentionally minimal: a single Go binary with no external framework dependencies beyond the standard library's `net/http/httputil.ReverseProxy`. This simplicity is appropriate for a data-plane sidecar that must add minimal latency to the inference path.

The proxy uses an LRU cache (capacity 16) for prefiller reverse proxy handlers, which means it can efficiently handle requests routed to up to 16 distinct prefiller endpoints without recreating proxy objects. Given that prefiller selection is typically determined by the upstream inference scheduler based on KV-cache affinity, this cache size is reasonable for most deployment topologies. The `bufferedResponseWriter` pattern used for prefill responses is notable: the entire prefill response is buffered in memory before parsing, which could be a concern for very large prefill responses, though in practice the prefill response with `max_tokens=1` is small.

The SSRF protection mechanism is architecturally interesting. Rather than relying on network policies alone, the sidecar maintains an in-process allowlist by watching InferencePool CRDs and their matching pods via Kubernetes informers. The allowlist tracks only host IPs and pod names (not ports), which provides a reasonable balance between security and flexibility. The 30-second resync period ensures the allowlist stays current with pod churn. However, the allowlist is disabled by default for backward compatibility, which means deployments must opt in to SSRF protection.

TLS configuration is thorough but has a noteworthy pattern: the proxy supports three independent TLS configurations: (1) inbound TLS for the proxy listener (enabled by default with self-signed cert fallback), (2) outbound TLS to prefiller workers (opt-in via `prefiller-use-tls`), and (3) outbound TLS to the local decoder (opt-in via `decoder-use-tls`). All three use the same hardcoded cipher suite list (ECDHE with AES-GCM and ChaCha20-Poly1305) with TLS 1.2 minimum. The `InsecureSkipVerify` options for prefiller and decoder connections are available but disabled by default, which is security-appropriate for production but needed for testing scenarios. The HTTP server configuration intentionally omits `ReadTimeout` and `WriteTimeout` to support long-running LLM inference requests that can take hours for large contexts, while still setting a 30-second `ReadHeaderTimeout` and 5-minute `IdleTimeout`.

The repository is deprecated in favor of the `llm-d-inference-scheduler` monorepo, which consolidates the sidecar with the scheduler. For RHOAI 3.3, this repository is the source of the routing sidecar component. The lack of Konflux Dockerfiles and FIPS build flags on this upstream branch suggests the downstream RHOAI build process adds its own Dockerfile with appropriate FIPS compliance measures.
