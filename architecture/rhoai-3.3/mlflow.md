## Purpose

**Short**: MLflow is an open-source ML lifecycle platform providing experiment tracking, model versioning, model deployment, LLM observability/tracing, and evaluation — deployed as a Python web service with a React UI and Kubernetes-native workspace/auth integration for RHOAI.

**Detailed**: MLflow serves as the experiment tracking and model registry component within the Red Hat OpenShift AI (RHOAI) platform. It provides a hybrid Flask/FastAPI server that exposes REST, GraphQL, and OpenTelemetry (OTLP) APIs for recording experiment runs, logging metrics/parameters/artifacts, managing registered models with versioning/aliasing, and ingesting distributed traces.

For RHOAI deployment, MLflow includes a purpose-built **kubernetes-workspace-provider** plugin that maps Kubernetes namespaces 1:1 to MLflow workspaces, enabling multi-tenant isolation. A companion **kubernetes-auth** plugin intercepts every API request and enforces Kubernetes RBAC via SelfSubjectAccessReview or SubjectAccessReview against a custom `mlflow.kubeflow.org` API group, ensuring that MLflow access decisions are governed by the cluster's native RBAC policies.

The Konflux-built container image bundles the MLflow Python server, the React frontend (pre-built), and the kubernetes-workspace-provider plugin into a single UBI 9-based image exposing port 5000. The server is launched with `--app-name kubernetes-auth --enable-workspaces --workspace-store-uri kubernetes:// --serve-artifacts` to activate workspace-aware, RBAC-enforced operation in the cluster.

## Architectural Analysis

MLflow's architecture in RHOAI is a **single-process Python server** that serves a dual role: a REST/GraphQL API backend and a static file server for the React frontend. The Flask/FastAPI hybrid design reflects MLflow's evolution — Flask handles the legacy REST API surface (`/api/2.0/mlflow/*`) while FastAPI handles newer features (jobs, OTLP traces) via ASGI. Both frameworks are composed into a single ASGI application using Starlette's `WSGIMiddleware` bridge, with FastAPI-native routes taking priority.

The **Kubernetes integration layer** is the most architecturally significant aspect of the RHOAI deployment. Rather than implementing its own user/group management, MLflow delegates all identity and authorization decisions to the Kubernetes API server. The `kubernetes-auth` plugin intercepts every API request, extracts the caller's identity from a Bearer token (or trusted proxy headers), and submits a SelfSubjectAccessReview (or SubjectAccessReview) against a custom API group `mlflow.kubeflow.org`. This design means that cluster RBAC policies (Roles, ClusterRoles, RoleBindings) directly govern who can create experiments, register models, submit jobs, and access workspaces — with no MLflow-specific user database or permission model needed. The 300-second authorization cache is critical for performance, as every API call would otherwise require a Kubernetes API round-trip.

The **workspace model** maps Kubernetes namespaces 1:1 to MLflow workspaces via a background watch loop. This is read-only by design — MLflow never creates or deletes namespaces, treating workspace lifecycle as an external concern managed by the platform operator or cluster admin. The namespace cache uses a Kubernetes watch API stream with automatic resync on resource version expiration (HTTP 410), providing near-real-time updates without polling overhead. System namespaces (`kube-*`, `openshift-*`) are excluded by default, and optional label selectors allow fine-grained control over which namespaces appear as workspaces.

The **database layer** supports PostgreSQL, MySQL, SQLite, and MSSQL via SQLAlchemy, but the RHOAI Konflux build explicitly includes `psycopg2` (PostgreSQL driver) as a runtime dependency. The schema includes 27+ tracking tables and 7 model registry tables, all workspace-aware with `(workspace, name)` unique constraints for multi-tenant isolation. Schema evolution is managed by Alembic migrations. The artifact storage layer is pluggable, with the RHOAI build including `boto3` for S3-compatible storage — consistent with the platform's use of object storage for ML artifacts.

A notable **security consideration** is that MLflow does not perform its own TLS termination — the server listens on plaintext HTTP (port 5000) and relies on external TLS termination at the ingress layer (Gateway API / OpenShift Route / kube-rbac-proxy sidecar). The `MLFLOW_TRACKING_INSECURE_TLS` environment variable can disable TLS verification on outbound connections, which is a risk if inadvertently enabled in production. The `cryptography` library (v46.0.5) provides FIPS-capable crypto via the system OpenSSL, but there are no explicit FIPS build flags since this is a Python application, not a Go binary — FIPS compliance depends on the UBI 9 base image's OpenSSL configuration.
