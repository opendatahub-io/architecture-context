## Purpose

**Short**: Kubernetes operator that manages the lifecycle of Llama Stack distribution server deployments, providing a CRD-driven interface for deploying, configuring, and scaling Llama Stack AI inference servers.

**Detailed**: The Llama Stack K8s Operator (also referred to as `ogx-k8s-operator` in RHOAI downstream or `llama-stack-k8s-operator` upstream) is a controller-runtime-based Kubernetes operator that reconciles `LlamaStackDistribution` custom resources. When a user creates a `LlamaStackDistribution` CR, the operator provisions a complete deployment stack including a Deployment running a Llama Stack server container, a ClusterIP Service, a ServiceAccount, optional PersistentVolumeClaim for storage, optional NetworkPolicy for network isolation, optional Ingress for external access, optional HorizontalPodAutoscaler for auto-scaling, and optional PodDisruptionBudget for availability guarantees.

The operator supports multiple Llama Stack distributions (starter, remote-vllm, meta-reference-gpu, postgres-demo) that map to pre-built container images, as well as custom images specified directly in the CR. It uses a kustomize-based manifest rendering pipeline with custom Go plugins to transform base manifests into per-instance resources. The operator watches for changes to referenced ConfigMaps (user configuration and CA bundles) and automatically triggers rolling updates when configurations change by annotating pod templates with content-based hashes.

Key features include: distribution image management with operator-level overrides via ConfigMap, user-provided server configuration via ConfigMap injection, CA bundle management with automatic aggregation from explicit and ODH-platform-detected trusted CA bundles, feature flags (currently NetworkPolicy enable/disable) configurable via operator ConfigMap, upgrade cleanup logic for migrating from cluster-scoped to namespace-scoped RBAC, and comprehensive status reporting including health checks against the Llama Stack server's `/v1/health`, `/v1/providers`, and `/v1/version` endpoints.

## Architectural Analysis

The Llama Stack K8s Operator follows a clean single-CRD, single-controller architecture built on the standard kubebuilder/controller-runtime scaffolding. The reconciler owns a well-defined set of Kubernetes resources per CR instance: Deployment, Service, ServiceAccount, PVC, NetworkPolicy, Ingress, HPA, PDB, and RoleBinding. Resource lifecycle is managed through owner references, ensuring cascading deletion when the CR is removed. The operator carefully checks ownership before patching resources to prevent "stealing" resources created by other controllers — a thoughtful safety measure seen in `kustomizer.go`.

The manifest rendering pipeline is a notable architectural choice. Rather than constructing Kubernetes objects programmatically in Go, the operator embeds kustomize base manifests (copied into the container image from `controllers/manifests/`) and applies a chain of custom Go transformer plugins at runtime: `NamePrefixPlugin` for resource naming, `NamespacePlugin` for namespace injection, `FieldMutator` for per-instance field patching (replicas, ports, service account names, label selectors), and `NetworkPolicyTransformer` for dynamic ingress rule generation based on `spec.network.allowedFrom`. Complex objects like the full PodSpec (with containers, volumes, volume mounts, env vars, probes) are constructed in Go and then merged into the kustomize-rendered Deployment, giving the operator fine-grained control over container configuration while keeping the base manifests readable. This hybrid approach provides the declarative clarity of YAML templates with the dynamic flexibility of programmatic construction.

The CA bundle management system is sophisticated. The operator aggregates CA certificates from two sources: an explicit user-configured ConfigMap (referenced via `spec.server.tlsConfig.caBundle`) and an auto-detected `odh-trusted-ca-bundle` ConfigMap (the ODH/RHOAI platform's trusted CA ConfigMap). Certificates are validated through a strict pipeline: PEM block decoding, type filtering (only `CERTIFICATE` blocks accepted), X.509 parsing validation, and size/count limits (10MB max, 1000 certificates max) to prevent resource exhaustion. The validated certificates are concatenated into a single operator-managed ConfigMap that is mounted into the Llama Stack server pod and referenced via the `SSL_CERT_FILE` environment variable. This design means the operator handles the complexity of multi-source CA aggregation while presenting a single, validated trust store to the application.

The operator's ConfigMap watching mechanism uses a dual strategy: field indexers for efficient O(1) lookups on Kubernetes versions that support custom field labels for CRDs, with a graceful fallback to manual list-and-filter for older versions. The reconciler also watches the operator's own configuration ConfigMap (`llama-stack-operator-config`) and triggers reconciliation of all CRs when feature flags or image mappings change. Configuration changes propagate to pods through content-based hash annotations on the Deployment pod template, avoiding unnecessary restarts when the ConfigMap content hasn't actually changed.

One architectural observation is that the operator's startup includes an upgrade cleanup phase (`PerformUpgradeCleanup`) that deletes legacy ClusterRoleBindings from a previous version that used cluster-scoped RBAC. This indicates an intentional migration from cluster-scoped to namespace-scoped RBAC, improving multi-tenancy isolation. The operator also detects legacy CA bundle volumes (emptyDir + ConfigMap pattern from an older version) and falls back to full Deployment replacement instead of server-side apply when detected, preventing SSA field ownership conflicts during upgrades.
