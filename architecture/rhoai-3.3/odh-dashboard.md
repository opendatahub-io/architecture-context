## Purpose

**Short**: Web-based management console for Red Hat OpenShift AI, providing a unified interface for managing notebooks, model serving, pipelines, model registry, and platform configuration.

**Detailed**: The odh-dashboard is a monorepo producing multiple deployable artifacts that form the management UI layer of the RHOAI platform. The primary component is a React frontend (host application) served by a Node.js Fastify backend. The backend acts as an API gateway, proxying authenticated requests to the Kubernetes API server and various platform services (Data Science Pipelines, TrustyAI, Model Registry, ML Metadata, Prometheus/Thanos).

The dashboard uses a Module Federation architecture to load feature-specific UI modules as sidecar containers within the same pod. Three federated modules — Gen AI, MaaS (Model-as-a-Service), and Model Registry — each have their own Go BFF (Backend-for-Frontend) that handles domain-specific API proxying and business logic. The host application dynamically loads these modules' React components and proxies their API requests through the Node.js backend to the appropriate BFF sidecar.

Authentication is enforced at the pod boundary by a `kube-rbac-proxy` sidecar that validates user tokens and injects identity headers before traffic reaches the dashboard. The dashboard backend reads these headers to determine user identity, admin status, and group membership. An HTTPRoute (Gateway API) exposes the dashboard externally through the platform's shared Gateway.

## Architectural Analysis

The odh-dashboard implements a sophisticated module federation architecture where the host application (React frontend + Node.js backend) dynamically loads federated UI modules from sidecar containers within the same pod. Each sidecar runs a Go BFF (Backend-for-Frontend) that serves both the federated JavaScript bundle (`remoteEntry.js`) and domain-specific API endpoints. The host's Fastify backend proxies requests to sidecars via in-cluster HTTPS calls on distinct ports (8043 for model-registry, 8143 for gen-ai, 8243 for maas), with the federation topology configured via a ConfigMap (`federation-config`) that maps module names to service ports and proxy paths.

The authentication architecture uses a layered approach. The outermost layer is `kube-rbac-proxy`, which runs as a sidecar on port 8443, terminating TLS and validating that the user can list OpenShift projects — a baseline authorization check. It then injects `X-Auth-Request-User`, `X-Auth-Request-Groups`, and `x-forwarded-access-token` headers into requests forwarded to the dashboard backend on port 8080 (plaintext, localhost only). The backend applies additional authorization logic: admin enforcement, resource ownership validation (e.g., a user can only manage notebooks matching `jupyter-nb-{username}`), and SelfSubjectAccessReview checks. When proxying to BFF sidecars, the backend forwards the user's access token via the `x-forwarded-access-token` header, and each BFF independently validates access via Kubernetes SubjectAccessReview.

The RHOAI vs ODH divergence is handled entirely through kustomize overlays, not code branching. The same application binary is deployed in both distributions; the differences are: (1) resource naming (`odh-dashboard` vs `rhods-dashboard`), (2) branding (logos, product name, doc links — set as build-time env vars in the Dockerfile), (3) application catalog entries (NVIDIA GPU quickstarts, Starburst, RHOAM are RHOAI-only), and (4) the ConsoleLink section title. The `modular-architecture` overlay adds the three BFF sidecar containers, a NetworkPolicy opening their ports, and the `federation-config` ConfigMap. In RHOAI, the `rhoai/shared/base` overlay applies JSON6902 patches to rename all resources from `odh-*` to `rhods-*`, including the deployment selector, service, HTTPRoute, and NetworkPolicy.

The backend's service proxy architecture (`utils/proxy.ts`) implements a sophisticated pattern for discovering and proxying to platform services. Each service proxy is configured with: (1) an optional "gating resource" — a CRD instance whose existence and status validates the service is ready, (2) a service discovery mechanism using either a Kubernetes annotation (`routing.opendatahub.io/external-address-rest`) or a URL extracted from the gating resource's status field, and (3) a local development fallback using environment variables for `kubectl port-forward` scenarios. This pattern is used for Data Science Pipelines, Model Registry, TrustyAI, ML Metadata, and LlamaStack. The proxy validates the gating resource status before forwarding requests, returning meaningful errors if the service is not ready.

A notable architectural decision is the deployment topology: all dashboard components (main dashboard, three BFF sidecars, and kube-rbac-proxy) run in a single pod with 2 replicas and pod anti-affinity across availability zones. The BFF sidecars share the same TLS certificate (`dashboard-proxy-tls`) generated by OpenShift's service-serving-cert-signer. This co-location simplifies networking (all communication is via localhost or pod-internal HTTPS) but means a BFF failure could affect pod health. Each container has independent liveness/readiness probes to mitigate this risk.
