## Purpose

**Short**: Manages the lifecycle of model inference services, serving runtimes, and NVIDIA NIM accounts on Red Hat OpenShift AI, automating networking, monitoring, authentication, and autoscaling resources for each deployed model.

**Detailed**: The ODH Model Controller is a Kubernetes controller that watches KServe InferenceService, ServingRuntime, LLMInferenceService, InferenceGraph, and NVIDIA NIM Account custom resources. When users deploy inference models, the controller automatically provisions supporting infrastructure including OpenShift Routes for external access, Prometheus ServiceMonitors and metrics dashboards for observability, KEDA TriggerAuthentications for autoscaling, ClusterRoleBindings for auth delegation, and NetworkPolicies for network isolation.

For LLM serving via the LLMInferenceService CRD, the controller integrates with Kuadrant AuthPolicy for Gateway API-based authentication and Istio EnvoyFilter for TLS bootstrap, enabling a unified auth model across all served models. The controller also manages tier-based RBAC for multi-tenant MaaS (Model-as-a-Service) access control.

The NIM module handles NVIDIA GPU Inference Microservice lifecycle management: validating NGC API keys, fetching available model catalogs from NVIDIA's cloud API, generating container registry pull secrets for nvcr.io, and provisioning OpenShift Template resources containing pre-configured ServingRuntime definitions. The controller ships pre-built serving runtime templates for vLLM (CUDA, ROCm, Gaudi, CPU, Spyre variants), OpenVINO Model Server (OVMS), MLServer, and NVIDIA NIM.

## Architectural Analysis

The ODH Model Controller exemplifies a multi-concern controller pattern where a single binary manages diverse Kubernetes resources across multiple domains (networking, security, monitoring, autoscaling, and vendor integration). Rather than splitting into separate operators, it uses a sub-reconciler composition pattern: the `InferenceServiceReconciler` delegates to `KserveRawInferenceServiceReconciler`, which in turn orchestrates six sub-reconcilers (Route, ClusterRoleBinding, metrics Service, ServiceMonitor, dashboard ConfigMap, and KEDA). Similarly, `LLMInferenceServiceReconciler` delegates to four sub-reconcilers (Role, RoleBinding, AuthPolicy, EnvoyFilter). This keeps the codebase manageable while avoiding the operational overhead of multiple operator deployments.

A notable architectural decision is the delta-based reconciliation pattern used throughout. Rather than the typical `CreateOrUpdate` approach, every reconciler computes an explicit delta (added/updated/removed) by comparing the desired resource state against the current cluster state through type-specific comparators. This provides fine-grained control over when resources are created, updated, or deleted, and enables conditional skipping when no changes are needed. The pattern is implemented through the `DeltaProcessor` interface and per-resource-type `ResourceComparator` functions.

The LLM serving architecture demonstrates a sophisticated multi-layer auth model. For gateway-level authentication, AuthPolicies are created with the Gateway as their owner (enabling shared lifecycle across multiple LLMInferenceServices). For per-service anonymous access, separate AuthPolicies target HTTPRoutes and are owned by the individual LLMInferenceService. The EnvoyFilter for TLS bootstrap is only created when both conditions are met: the gateway is labeled as managed (or has the authorino-tls-bootstrap annotation) AND Authorino TLS is enabled in the cluster. This conditional creation avoids deploying unnecessary Istio resources when the auth stack isn't present.

The NIM module uses a handler chain pattern where four handlers (Validation → ConfigMap → Template → PullSecret) execute sequentially, each returning a `HandleResponse` that can short-circuit the chain. The `APIKeyManager` uses `sync.Once` for lazy-loading to avoid redundant secret reads within a single reconciliation cycle. Server-side apply (SSA) with `Force: true` is used for all NIM-managed resources, which is well-suited for the controller-owned resource pattern but could conflict with other controllers if resource ownership isn't properly scoped.

The webhook architecture provides defense-in-depth validation: InferenceGraph webhooks enforce namespace isolation and reject external service URLs (preventing authorization header leakage), InferenceService webhooks protect the application namespace from model deployments, NIM Account webhooks enforce a singleton-per-namespace constraint, and tier ConfigMap webhooks validate uniqueness of tier levels. All webhooks use `failurePolicy: fail`, meaning webhook unavailability blocks resource creation — a security-first design choice that trades availability for correctness.

The controller's selective cache configuration is notable: it only caches Secrets with `opendatahub.io/managed=true` and Pods with `component=predictor`. This dramatically reduces memory consumption in clusters with thousands of pods and secrets, but means the controller cannot react to changes on unlabeled resources. The Ray TLS certificate management (5-year CA, 3-year server certs, per-pod SANs with wildcard DNS) creates a hierarchical PKI within the controller namespace that is propagated to user namespaces containing multi-node ServingRuntimes.
