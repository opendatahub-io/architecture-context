## Purpose

**Short**: Hardware plugin that enables vLLM large language model inference on Intel Gaudi (Habana) HPU accelerators, packaged as an OpenAI-compatible API server container image for RHOAI serving runtimes.

**Detailed**: The vLLM Gaudi plugin (`vllm_gaudi`) is a Python package that integrates Intel Gaudi AI accelerators (Gaudi2 and Gaudi3) with the vLLM inference engine via vLLM's hardware plugin architecture. It registers itself through Python entry points (`vllm.platform_plugins` and `vllm.general_plugins`) to provide HPU-specific implementations of attention backends, model runners, workers, distributed communicators, and quantization methods.

The plugin replaces GPU-specific operations with Habana-optimized equivalents including FusedSDPA attention kernels, HPU graph compilation for reduced latency, dynamic input bucketing to minimize recompilation overhead, and hardware-aligned FP8/GPTQ/AWQ quantization. It supports a wide range of model architectures including text-only (DeepSeek, Qwen3, GPT-OSS, HunYuan, MiniMax) and multimodal vision-language models (Gemma3, Qwen2.5-VL, Pixtral, ERNIE4.5-VL, Ovis).

For RHOAI, the component is built as a RHEL 9 UBI-based container image (`odh-vllm-gaudi-rhel9`) via Konflux CI. The container bundles the Habana SynapseAI runtime (v1.24.0), PyTorch 2.10 for HPU, the upstream vLLM server (v0.19.0), and the Gaudi plugin. It serves as a KServe/ModelMesh-compatible inference runtime, exposing an OpenAI-compatible REST API for chat completions, text generation, embeddings, and tokenization.

## Architectural Analysis

The vLLM Gaudi plugin follows a clean out-of-tree (OOT) plugin architecture, registering itself via Python entry points (`vllm.platform_plugins` for platform detection, `vllm.general_plugins` for utilities, custom ops, and model overrides). This design allows the plugin to override vLLM's GPU-centric implementations without forking the upstream codebase, though the override pattern is aggressive — it monkey-patches several vLLM core modules at import time (e.g., `_async_sched_module.AsyncScheduler = HPUAsyncScheduler` in utils.py, `torch_utils.make_tensor_with_pad = make_tensor_with_pad_hpu`).

The HPU execution model fundamentally differs from GPU execution. Where CUDA uses kernels and CUDA graphs, Gaudi uses SynapseAI recipe compilation. The plugin manages this through a sophisticated bucketing system (`HPUBucketingManager`) that pre-compiles graphs for common input shapes (batch size x sequence length combinations) during warmup, then routes incoming requests to the nearest matching bucket. This avoids the cost of recompilation on every new input shape, which is critical because SynapseAI graph compilation is significantly more expensive than CUDA kernel launches. The bucketing system supports multiple strategies (linear, exponential, file-based) configurable via environment variables.

The quantization support is notably comprehensive, covering FP8 (per-tensor, per-channel, per-block with hardware-aligned scale management differentiating Gaudi2 and Gaudi3), AWQ (4-bit), GPTQ (2/3/4/8-bit), INC (Intel Neural Compressor), compressed-tensors (W8A8 FP8, W8A8 Int8, WNA16), MXFP4, and ModelOpt. The `scales.py` module implements hardware-specific exponent bias alignment (Gaudi2 supports only 4 bias values: {3, 7, 11, 15}; Gaudi3 supports the full range 0-62), which is critical for numerical correctness on each generation of hardware.

The FIPS compliance posture is intentionally non-compliant — the Dockerfile explicitly removes `openssl-fips-provider-so` and the Tekton pipeline sets `fips-check-blocking: false`. This is a pragmatic decision driven by the Habana driver stack's dependencies on non-FIPS OpenSSL behavior. The build is also non-hermetic (`hermetic: false` in both push and PR pipelines), with no lock files for Python dependencies or RPM packages. This represents a supply chain gap, though it's architecturally constrained by the complexity of the Habana software stack (SynapseAI drivers, PT modules from Habana's artifactory, multiple GitHub clones during build).

The multi-model switching capability (`MultiModelAsyncLLM`) is a Gaudi-specific extension not present in upstream vLLM. It enables a single serving pod to host multiple models and switch between them at runtime via the `/v1/models/switch` API endpoint, using a sleep/wake/stash mechanism that moves model weights between HPU and CPU memory. This is particularly valuable for Gaudi hardware where each accelerator has substantial memory (96GB on Gaudi2, 128GB on Gaudi3) and re-deploying containers is slower than in-process model swaps.
