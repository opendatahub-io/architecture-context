## Purpose

**Short**: KServe is a Kubernetes-native platform for deploying, scaling, and managing machine learning inference services with support for serverless, raw deployment, and LLM-optimized serving modes.

**Detailed**: KServe provides a standardized control plane for ML model serving on Kubernetes. It defines Custom Resource Definitions (CRDs) for inference services, serving runtimes, inference graphs, trained models, local model caching, and LLM inference services. The operator manages the full lifecycle of these resources — from model storage initialization and deployment creation to ingress routing and autoscaling.

The operator supports three primary deployment modes: **Serverless** (via Knative), **RawDeployment** (vanilla Kubernetes Deployments with optional Gateway API or Kubernetes Ingress), and **ModelMesh** (for multi-model serving at scale). For LLM workloads, the dedicated LLMInferenceService controller provides advanced capabilities including prefill/decode separation, multi-node deployment via LeaderWorkerSet, Gateway API inference extensions (InferencePool, InferenceModel), and intelligent request scheduling.

In the RHOAI distribution, KServe integrates with the OpenShift platform — using OpenShift service-ca for certificate signing, OpenShift Routes for legacy ingress, Istio/Gateway API for traffic management, KEDA for advanced autoscaling, OpenTelemetry for observability, and Kuadrant AuthPolicy for authentication enforcement. The operator injects sidecars (storage initializer, agent, logger, batcher, metrics aggregator) into inference pods via mutating admission webhooks.

## Architectural Analysis

KServe's architecture reflects its dual nature as both an upstream CNCF project and a critical RHOAI platform component. The Go operator manages 11 CRDs through a single controller-manager binary (plus two auxiliary binaries for local model management), which is an unusually broad scope. The controller uses a modular reconciler pattern — the InferenceService controller delegates to specialized sub-reconcilers for deployment, service, ingress, autoscaler, and model config management. This decomposition is well-structured: each reconciler owns a single resource type and the orchestrating controller (e.g., `RawKubeReconciler`) sequences them correctly.

The LLMInferenceService controller is architecturally distinct from the InferenceService controller, despite sharing the same binary. It introduces Gateway API Inference Extensions (InferencePool, InferenceModel) as first-class concepts — a pattern that is still evolving in the Kubernetes ecosystem. The controller manages a complex resource creation matrix: depending on whether prefill/decode separation is enabled, whether multi-node (LeaderWorkerSet) is used, and whether a scheduler is deployed, the controller creates between 5 and 25+ resources per LLMInferenceService instance. The `lifecycle_crud.go` generic CRUD helpers provide type-safe, event-emitting resource management that simplifies this complexity, but the combinatorial state space (single-node vs multi-node × P/D vs non-P/D × managed vs custom routing) makes the controller logic intricate.

The FIPS compliance story has a gap: while the controller, agent, and router are built with `GOEXPERIMENT=strictfipsruntime` and CGO_ENABLED=1 on UBI 9 (passing check-payload), the localmodel manager and localmodel-agent are built with `CGO_ENABLED=0` on distroless static images without OpenSSL. These two components would fail check-payload validation for FIPS. Additionally, the Istio DestinationRules for LLM workloads use `InsecureSkipVerify: true` for the scheduler and shadow services, which is documented as a temporary workaround but represents a TLS verification gap.

The kustomize overlay structure (`config/overlays/odh/`) is well-designed for the ODH/RHOAI distribution. It systematically replaces cert-manager with OpenShift service-ca, removes upstream ClusterServingRuntimes (RHOAI uses its own), removes LocalModel support (not GA in RHOAI 3.3), adds NetworkPolicies, and parameterizes all container images through a `params.env` + `configMapGenerator` + `replacements` pattern. The LLMInferenceServiceConfig resources are patched with well-known config annotations and image references, enabling the platform operator to control default LLM serving configurations.

The webhook architecture is comprehensive — 1 mutating webhook (pod injection) and 8 validating webhooks cover all CRDs. The pod mutating webhook's injection pipeline (storage initializer → Istio CNI security context → agent/logger/batcher → metrics aggregator) runs in a specific order, and the `reinvocationPolicy: IfNeeded` setting ensures correct behavior when other webhooks modify the pod. The webhook uses an `objectSelector` with `serving.kserve.io/inferenceservice: Exists` to avoid mutating non-KServe pods, which is operationally important.
