## Purpose

**Short**: Python SDK providing an intuitive interface for requesting, managing, and monitoring Ray clusters and Ray jobs on Kubernetes, integrated with Kueue for resource queuing and the RHOAI platform for authentication and networking.

**Detailed**: The CodeFlare SDK (`codeflare-sdk`) is a Python client library that abstracts Kubernetes and KubeRay operator complexity, enabling data scientists to create and manage Ray clusters and submit Ray jobs from Jupyter notebooks or Python scripts. It provides two primary workflows: (1) interactive `Cluster` management — creating `RayCluster` custom resources with configurable head/worker node specifications, GPU/accelerator support, Kueue LocalQueue integration, and mTLS certificate generation; and (2) batch `RayJob` management — submitting `RayJob` custom resources with automatic local file packaging, runtime environment configuration, and managed cluster lifecycle.

The SDK is tightly integrated with the RHOAI platform. It supports multiple authentication methods via the `kube-authkit` library (OIDC, OpenShift OAuth, token-based, kubeconfig, in-cluster service accounts). For cluster dashboard access, it supports both the RHOAI 3.x Gateway API pattern (HTTPRoute-based URLs like `https://{gateway-host}/ray/{namespace}/{cluster}`) and the legacy RHOAI 2.x OpenShift Route pattern. The SDK generates client-side mTLS certificates by fetching the CA secret created by the KubeRay operator, enabling encrypted Ray client connections. It also includes Jupyter notebook widgets (ipywidgets) for interactive cluster management, Kueue WorkloadPriorityClass validation, and Ray version compatibility checking.

## Architectural Analysis

The CodeFlare SDK follows a layered architecture with clear separation between user-facing APIs, Kubernetes interaction, and utility functions. The top-level `codeflare_sdk` package re-exports all public symbols, presenting a flat API surface (`from codeflare_sdk import Cluster, ClusterConfiguration, RayJob`). Internally, the code is organized into two primary subsystems: `ray.cluster` for interactive RayCluster management and `ray.rayjobs` for batch RayJob workflows. Both subsystems share the `common` layer for authentication, Kueue integration, error handling, and certificate generation.

A notable architectural decision is the dual dashboard URL resolution strategy. The `_get_dashboard_url_from_httproute()` function implements RHOAI 3.x Gateway API support by searching for HTTPRoutes labeled with `ray.io/cluster-name` and `ray.io/cluster-namespace`, then extracting the hostname from the parent Gateway's `spec.listeners[].hostname`. This falls back to OpenShift Route lookup (RHOAI 2.x pattern) and then Kubernetes Ingress lookup (vanilla K8s/Kind). The HTTPRoute search attempts cluster-wide listing first, then falls back to namespace-specific searches across `redhat-ods-applications`, `opendatahub`, and other platform namespaces. This multi-tier fallback ensures backward compatibility but adds complexity to the dashboard resolution path.

The mTLS certificate generation (`generate_cert.py`) is a security-critical subsystem. It fetches the CA certificate from a KubeRay-managed Kubernetes Secret (labeled `ray.openshift.ai/cluster-name`), then generates a client key pair (RSA 3072-bit) and X.509 certificate with proper RFC 5280 extensions including SAN entries for `localhost`, `127.0.0.1`, `::1`, and the Ray head service DNS names. Certificates are stored with restrictive permissions (0600) under `~/.local/share/codeflare/tls/` (XDG-compliant). The SDK includes certificate lifecycle management: automatic generation on `apply()`, cleanup on `down()`, refresh via `refresh_certificates()`, and expired certificate cleanup utilities. A design consideration is that `apply()` polls for the CA secret (up to 300 seconds) since the KubeRay operator creates it asynchronously when the head pod starts.

The RayJob subsystem (`ray.rayjobs`) introduces automatic local file packaging. When users specify a local `working_dir` in their runtime environment, the SDK zips the directory (excluding `.ipynb` and `.md` files), base64-encodes it, and stores it in a Kubernetes Secret with an owner reference to the RayJob CR. An init container in the submitter pod template unzips the files before the job runs. This pattern enables data scientists to submit jobs with local files without needing a shared filesystem or remote storage, though it's limited to 1MB (Kubernetes Secret size limit). The Secret is automatically cleaned up when the RayJob is deleted due to the owner reference.

The vendored KubeRay Python client (`vendored/python_client/`) provides low-level CRUD operations against `ray.io/v1` RayCluster and RayJob resources. This vendoring approach ensures the SDK doesn't depend on the upstream KubeRay Python client's release cycle, but it means the SDK must maintain these API wrappers independently. The vendored client handles its own kubeconfig/in-cluster config loading separately from the main SDK's `kube-authkit`-based auth, which is a potential source of configuration inconsistency.
