## Purpose

**Short**: ML Metadata (MLMD) is a gRPC server and library for recording, retrieving, and querying metadata associated with ML workflows, including artifacts, executions, contexts, and their lineage relationships.

**Detailed**: ML Metadata is a core infrastructure component in the Kubeflow/RHOAI ML pipeline ecosystem. It provides a strongly-typed metadata store that tracks the provenance of ML artifacts (datasets, models, metrics), execution runs (training, transformation, evaluation), and organizational contexts (pipelines, experiments). The server exposes a comprehensive gRPC API defined via Protocol Buffers for CRUD operations on these entities and their relationships (events, attributions, associations, parent contexts), as well as lineage graph traversal queries.

The MLMD server is written in C++ using Bazel as its build system, with Python client bindings via pybind11. It supports three backend database engines: MySQL, PostgreSQL, and SQLite. For RHOAI deployments, the server is built as a statically-linked C++ binary from a Konflux Dockerfile and shipped as the `odh-mlmd-grpc-server` container image. The server is consumed primarily by Kubeflow Pipelines (via Data Science Pipelines / DSP) to persist pipeline run metadata, artifact lineage, and execution history.

The upstream project is Google's `google/ml-metadata`, and this repository (`red-hat-data-services/ml-metadata`) is the Red Hat downstream fork with modifications for building on RHEL/UBI9 and integration with the RHOAI Konflux build pipeline.

## Architectural Analysis

ML Metadata follows a clean separation of concerns: the gRPC service layer (`MetadataStoreServiceImpl`) is a thin adapter that connects to the database on each RPC call, delegates to the `MetadataStore` business logic layer, and converts between absl::Status and grpc::Status. Notably, the service implementation creates a new `MetadataStore` instance per request (via `ConnectMetadataStore`), which means each gRPC call opens a fresh database connection. This is a deliberate design choice for simplicity and thread safety but may impact performance under high concurrency — the code contains a TODO (`b/123039345`) acknowledging the need for a connection pool.

The data model is graph-structured: Artifacts, Executions, and Contexts are the node types, while Events (artifact↔execution), Attributions (context→artifact), Associations (context→execution), and ParentContexts (context→context) form the edge types. This graph structure directly supports ML lineage tracking — tracing which artifacts were consumed and produced by which execution steps, organized into pipeline/experiment contexts. The `GetLineageSubgraph` API enables graph traversal with configurable direction (upstream/downstream/bidirectional), max hops, and ending node filters, making it suitable for provenance queries.

The server supports three database backends (MySQL, PostgreSQL, SQLite) through an abstract `MetadataSource` interface, with backend-specific SQL query templates defined in the `MetadataSourceQueryConfig` protobuf. Database schema migrations are handled automatically via a versioned migration scheme, with both upgrade and downgrade paths, controlled by the `MigrationOptions` configuration. The `enable_database_upgrade` flag enables automatic schema migration on server startup, while `downgrade_to_schema_version` supports rolling back schema changes.

For the RHOAI deployment, the Konflux Dockerfile builds the C++ server binary from source using Bazel on a UBI9 builder image, then copies just the binary into a UBI9-minimal runtime image. The container runs as non-root (UID 65534:65534) and exposes port 8080. The entrypoint accepts configuration via environment variables (`GRPC_PORT`, `METADATA_STORE_SERVER_CONFIG_FILE`) or command-line flags for direct database connection parameters. The multi-arch Tekton pipeline builds for amd64, arm64, and ppc64le. This component has no Kubernetes-native features (no CRDs, no controllers, no RBAC manifests) — it is purely a stateless gRPC server that relies on the DSP operator for its Kubernetes deployment lifecycle.
