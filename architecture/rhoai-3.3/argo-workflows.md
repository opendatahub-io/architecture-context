## Purpose

**Short**: Argo Workflows is a container-native workflow engine for orchestrating parallel jobs on Kubernetes, used as the execution backend for Data Science Pipelines in RHOAI.

**Detailed**: Argo Workflows provides a Kubernetes-native workflow orchestration engine that defines workflows as custom resources (CRDs). It enables users to define complex DAG and step-based pipelines where each step runs as a container. In the RHOAI context, Argo Workflows serves as the underlying execution engine for Data Science Pipelines (DSP), executing ML pipeline runs as Kubernetes pods.

The system consists of three primary binaries: the **workflow-controller** which watches Workflow CRDs and orchestrates pod creation/lifecycle; **argoexec** which runs as an init/sidecar container within workflow pods to handle artifact loading, container monitoring, and output collection; and the **argo CLI/server** which provides a REST/gRPC API and web UI for workflow management (though the server component is not built via Konflux for RHOAI — only workflow-controller and argoexec are shipped).

The controller uses a work-queue based reconciliation pattern (not controller-runtime) with Kubernetes informers, supports optional persistence to PostgreSQL/MySQL for workflow archiving and node status offloading, and integrates with multiple artifact storage backends (S3, GCS, Azure Blob, HDFS).

## Architectural Analysis

Argo Workflows in the RHOAI context operates as a lean execution engine — the Konflux builds only produce the workflow-controller and argoexec binaries, deliberately excluding the argo CLI/server component. This means the Argo Server's REST/gRPC API, web UI, SSO authentication, and artifact serving capabilities are not shipped as part of RHOAI. Instead, the Data Science Pipelines API Server provides the user-facing API, and DSPO manages the deployment lifecycle. The workflow-controller runs as a passive Kubernetes controller, watching for Workflow CRs created by the DSP API server.

The argoexec component ships with a dual-binary approach for FIPS compliance: `argoexec-fips` (dynamically linked with strictfipsruntime for FIPS-enabled clusters) and `argoexec` (statically linked, non-FIPS fallback). This is notable because the workflow-controller Konflux build only produces a single FIPS-compliant binary, while argoexec needs both variants since it runs inside user workflow pods that may or may not be on FIPS-enabled infrastructure. The entrypoint defaults to the non-FIPS binary (`/bin/argoexec`), suggesting FIPS selection happens at the DSPO configuration level.

The controller uses a custom work-queue reconciliation pattern built directly on `k8s.io/client-go` informers rather than the `controller-runtime` framework used by most RHOAI operators. It maintains 32 concurrent workflow workers by default, processes DAG and step-based execution graphs, and supports leader election via Kubernetes Leases for HA deployments. The architecture includes sophisticated features like workflow hydration/dehydration (offloading large node status to a database to avoid etcd size limits), artifact garbage collection via finalizers, and cron-based workflow scheduling.

The inter-process communication between argoexec and workflow containers is entirely filesystem-based via the shared `/var/run/argo/` volume mount. Exit codes, stdout/stderr, signals, output parameters, and compressed artifacts are all exchanged through well-defined file paths. This design avoids network dependencies within the pod and works with any container runtime, but requires the emissary executor pattern where argoexec replaces the original container entrypoint to intercept execution.

Security posture is strong at the container level — both deployments enforce readOnlyRootFilesystem, runAsNonRoot, no privilege escalation, and drop ALL Linux capabilities. However, the InsecureSkipVerify option for OIDC provider connections (in the Argo Server, which is not shipped in RHOAI) represents a potential concern if the server were ever reintroduced. RBAC permissions are extensive but well-scoped — the controller needs broad pod management capabilities and full access to all argoproj.io CRDs, while access to Secrets is restricted to a single named resource (`argo-workflows-agent-ca-certificates`).
