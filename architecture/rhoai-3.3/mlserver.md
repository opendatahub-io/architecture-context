## Purpose

**Short**: MLServer is a multi-model inference server that serves machine learning models via REST (HTTP) and gRPC interfaces, fully compliant with the KServe V2 Inference Protocol (dataplane).

**Detailed**: MLServer provides a production-ready inference server for machine learning models on Red Hat OpenShift AI. It implements the KServe V2 Dataplane specification, exposing standardized REST and gRPC APIs for model inference, health checks, metadata queries, and model repository management. The server supports multi-model serving (multiple models within a single process), adaptive request batching, parallel inference via worker pools, response caching, and streaming inference (SSE for REST, bidirectional streaming for gRPC).

MLServer uses a pluggable runtime architecture where each ML framework (scikit-learn, XGBoost, LightGBM, HuggingFace Transformers, MLflow, Alibi Detect/Explain, CatBoost, Spark MLlib) is implemented as a separate runtime module that extends the base `MLModel` class. In the RHOAI 3.3 distribution, the container image ships with three runtimes pre-installed: `sklearn`, `xgboost`, and `lightgbm`. Additional runtimes are available as optional packages.

The server also supports an optional Kafka interface for message-driven inference, Prometheus metrics collection (on a dedicated metrics port), OpenTelemetry distributed tracing, and CloudEvents middleware for event-driven architectures. MLServer is the core Python inference server used by KServe to serve machine learning models on Kubernetes.

## Architectural Analysis

MLServer follows a clean layered architecture that separates protocol handling from model serving logic. The core abstraction is the `MLModel` base class, which defines the lifecycle methods (`load`, `predict`, `predict_stream`, `unload`) that all runtime plugins must implement. Above this sits the `DataPlane` handler that routes requests to the appropriate model and applies cross-cutting concerns (caching, batching, middleware). At the top layer, three protocol-specific servers (REST via FastAPI, gRPC via grpc.aio, Kafka via aiokafka) each translate their wire format into the common `InferenceRequest`/`InferenceResponse` types. This design means adding a new serving protocol requires only a new server implementation without touching model runtime code.

The runtime plugin system is well-designed for extensibility. Each runtime is a separate Python package (e.g., `mlserver-sklearn`, `mlserver-xgboost`) that can be installed independently. Runtimes are discovered via the `implementation` field in `model-settings.json`, which specifies the Python import path to the model class. The RHOAI 3.3 container image pre-installs three runtimes (`sklearn`, `xgboost`, `lightgbm`), but the architecture supports loading any runtime that extends `MLModel`. Runtime-specific custom REST endpoints (like MLflow's `/invocations` or Alibi Explain's `/explain`) are registered dynamically at model load time via the `@custom_handler` decorator pattern.

The parallel inference subsystem is architecturally significant. When `parallel_workers > 1`, MLServer wraps each model in a `ParallelModel` that dispatches requests to a pool of worker processes via multiprocessing queues. Each worker runs its own event loop and `MultiModelRegistry`, with the `Dispatcher` handling round-robin request routing and async response futures. This design enables CPU-bound inference (e.g., sklearn, xgboost) to bypass Python's GIL, but adds complexity around IPC, worker health monitoring (SIGCHLD handling), and model lifecycle synchronization. The `InferencePoolRegistry` further supports environment isolation — each pool can operate in a custom Python environment extracted from a tarball.

A notable design choice is that MLServer provides no authentication or TLS termination. All three ports (8080, 8081, 8082) bind insecurely. This is intentional in the KServe/RHOAI deployment model where authentication and TLS are handled by the platform's ingress layer (kube-rbac-proxy sidecar or Gateway API with EnvoyFilter). This simplifies MLServer's implementation but means it must never be exposed directly to untrusted networks.

The adaptive batching system uses an elegant async queue pattern. The `AdaptiveBatcher` wraps the model's `predict()` method, accumulating incoming requests in an `asyncio.Queue` and flushing them when either the `max_batch_size` or `max_batch_time` threshold is reached. The `BatchedRequests` class handles merging individual inference requests into a single batched request and splitting the batched response back to individual futures. This transparent batching requires no changes to runtime implementations — it's applied as a hook during model loading.
