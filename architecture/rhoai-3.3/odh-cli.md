## Purpose

**Short**: CLI tool for Red Hat OpenShift AI (RHOAI) that validates cluster configuration, assesses upgrade readiness, backs up workloads, and performs automated migrations.

**Detailed**: The odh-cli (also known as `rhai-cli` or `kubectl-odh`) is a diagnostic and operational CLI tool for RHOAI deployments on OpenShift. It is designed as a kubectl plugin and provides three primary capabilities: (1) a comprehensive lint/diagnostic system with 37 pluggable checks that validate current cluster state and assess upgrade readiness between RHOAI versions (particularly 2.x → 3.x migrations), (2) a workload backup system with concurrent dependency resolution that exports Notebooks, InferenceServices, RayClusters, and their dependent ConfigMaps, Secrets, and PVCs to portable YAML, and (3) a migration action framework with prepare/run phases for automated cluster migrations (e.g., transitioning from OpenShift AI Kueue to Red Hat Build of Kueue).

The tool connects to live OpenShift clusters via kubeconfig or token authentication and operates in read-only mode for lint/diagnostic commands, with optional write operations for migration actions. It is distributed both as a standalone Go binary (via goreleaser) and as a container image (`quay.io/rhoai/rhoai-upgrade-helpers-rhel9`) that bundles kubectl, oc, yq, jq, and Python-based upgrade helper scripts. The container image is built by Konflux with FIPS-compliant Go compilation and hermetic builds across linux/amd64, linux/arm64, and linux/ppc64le architectures.

## Architectural Analysis

The odh-cli is architecturally notable for being a **pure client-side tool** — it deploys no server components, creates no CRDs, and runs no controllers. Its value comes entirely from its deep knowledge of the RHOAI platform's resource topology and version-to-version migration semantics. This positions it as an "expert system" that encodes platform-specific validation rules that would otherwise require manual cluster inspection by support engineers.

The **lint check framework** is the most sophisticated subsystem, employing a plugin architecture with explicit registration (no global state), fluent validation builders (ComponentBuilder, WorkloadBuilder, OperatorBuilder), and a Kubernetes-inspired DiagnosticResult type with conditions, impacts, and remediation guidance. The framework enforces sequential check execution to guarantee deterministic output ordering — a deliberate design decision that prioritizes diff-able output over parallelism. The check execution follows a canonical group ordering (Dependencies → Services → Platform → Components → Workloads) that reflects the RHOAI platform's dependency chain: you can't meaningfully validate workloads if platform CRDs aren't healthy.

The **check catalog** reveals the complexity of RHOAI 2.x → 3.x migrations: 37 checks cover everything from serverless removal and ServiceMesh deprecation to per-workload image compatibility analysis. The notebook impacted workloads check (`pkg/lint/checks/workloads/notebook/impacted.go`) is particularly complex at ~1500 lines, implementing a 4-strategy image lookup engine that resolves Docker image references, SHA digests, repository URLs, and ImageStream spec.from.name references to determine image compatibility. This single check demonstrates why the tool exists — this level of analysis cannot be done manually at scale.

The **backup subsystem** uses a pipeline architecture with channel-based concurrency: discovery (list resources) → dependency resolution (N parallel goroutines) → sequential YAML writing. The dependency resolver registry allows different resource types (Notebooks, DSPAs) to define custom dependency extraction logic, such as the notebook resolver that extracts ConfigMap, Secret, and PVC references from both volume mounts and environment variables while filtering out infrastructure sidecars (oauth-proxy containers). Field stripping removes cluster-specific metadata (managedFields, resourceVersion, uid) to make backups portable across clusters.

The **migration framework** introduces write operations with a careful prepare/run separation: prepare performs preflight checks and presents a dry-run preview, while run executes the actual changes with a hierarchical step recorder that tracks each action's outcome. The RHBOK (Red Hat Build of Kueue) migration is the first implementation, handling the complex task of transitioning kueue resources from the OpenShift AI operator to a standalone kueue operator — including OLM subscription creation, CSV readiness polling, and ClusterQueue/LocalQueue relabeling.

A key architectural risk is the **tight coupling to RHOAI version semantics**: functions like `IsUpgradeFrom2xTo3x()`, `IsVersion3x()`, and version-gated `CanApply()` predicates embed knowledge of specific RHOAI version transitions. Each new major version migration will require updating many check implementations. The explicit check registration pattern mitigates this somewhat — new checks can be added without modifying existing ones — but the version comparison logic is distributed across 30+ check files rather than centralized.
