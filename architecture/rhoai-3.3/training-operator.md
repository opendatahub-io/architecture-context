## Purpose

**Short**: Kubernetes operator for distributed AI/ML training across multiple frameworks (PyTorch, TensorFlow, MPI, XGBoost, PaddlePaddle, JAX).

**Detailed**: The Kubeflow Training Operator is a Kubernetes-native controller that manages the lifecycle of distributed machine learning training jobs. It provides Custom Resource Definitions (CRDs) for six training frameworks: PyTorchJob, TFJob, MPIJob, XGBoostJob, PaddleJob, and JAXJob. Each CRD abstracts the complexity of setting up distributed training by automatically creating and managing pods, headless services, network policies, and framework-specific configuration (environment variables, init containers, hostfiles).

The operator supports gang scheduling through Volcano and scheduler-plugins integrations, elastic training with auto-scaling for PyTorch and Paddle jobs, and HorizontalPodAutoscaler management. It creates per-job NetworkPolicy resources to isolate training pod communication while allowing Prometheus monitoring access from the openshift-monitoring namespace. For MPI jobs, it implements a sophisticated launcher/worker pattern with dynamically generated ConfigMaps containing hostfiles and kubexec scripts, per-job ServiceAccounts, Roles, and RoleBindings.

In RHOAI, the operator is deployed via kustomize overlays in the `opendatahub` namespace with the `kubeflow-` name prefix, configured for FIPS-compliant builds using `GOEXPERIMENT=strictfipsruntime` and dynamic linking via `CGO_ENABLED=1`. The RHOAI overlay adds PodMonitor for Prometheus Operator integration and adapts RBAC roles for standard Kubernetes aggregation.

### Dynamically Created Network Resources

The operator creates the following network-related resources per training job at reconciliation time:

**Headless Services** (all frameworks except MPI):
Each replica in a training job gets a headless ClusterIP Service (`ClusterIP: None`) for DNS-based peer discovery. Service name follows the pattern `{job-name}-{replica-type}-{index}`, with ports matching the framework default (PyTorch: 23456, TensorFlow: 2222, XGBoost: 9999, Paddle: 36543, JAX: 6666).

**NetworkPolicies** (PyTorch only):
Each PyTorchJob creates a NetworkPolicy that:
1. Allows ingress only from pods with matching `training.kubeflow.org/job-name` label (intra-job isolation)
2. Allows ingress from `openshift-monitoring` namespace on port 8080/TCP (Prometheus scraping)

## Architectural Analysis

The Training Operator follows a well-structured controller-per-framework pattern built on a shared `JobController` base. All six framework controllers (PyTorch, TensorFlow, MPI, XGBoost, Paddle, JAX) embed the common `JobController` struct which provides Pod and Service reconciliation, expectation tracking, gang scheduling setup, and event recording. Each framework controller then overrides specific methods (`SetClusterSpec`, `IsMasterRole`, `ReconcilePods`, `ReconcileServices`, `UpdateJobStatus`) to implement framework-specific behavior. This design achieves strong code reuse while accommodating the significantly different topologies across frameworks.

The MPI controller is architecturally distinct from the other five controllers. While PyTorch/TF/XGBoost/Paddle/JAX controllers use the common Pod and Service reconciliation, MPI completely overrides `ReconcilePods` and bypasses `ReconcileServices` entirely. MPI creates a launcher/worker pattern where workers are created as long-running sleep pods, and a launcher pod uses `kubectl exec` via a delivered kubectl binary to execute MPI commands on workers. This requires dynamically creating per-job RBAC (ServiceAccount, Role with pods/exec permission scoped to specific worker pod names, RoleBinding), ConfigMaps containing hostfiles and kubexec scripts, and a `discover_hosts.sh` script that updates with running worker pod hostnames. The launcher init container (`kubectl-delivery`) copies the kubectl binary into a shared EmptyDir volume.

PyTorch's controller has the most sophisticated features among the standard controllers: it creates per-job NetworkPolicies using the Kubernetes Apply API (server-side apply) to isolate training pods while explicitly allowing Prometheus monitoring ingress from the `openshift-monitoring` namespace on port 8080. It also manages HorizontalPodAutoscalers for elastic training when `ElasticPolicy` is configured, and injects an init container that polls headless service DNS to discover worker addresses before training begins. The elastic policy supports rendezvous backends (c10d, etcd, etcd-v2) for PyTorch's distributed training coordination.

All webhooks are validating-only with `failurePolicy: Fail` (strict), ensuring that invalid job configurations are rejected before creation. Certificate management is handled internally via the OPA cert-controller library, avoiding a dependency on external cert-manager. The cert rotation runs without requiring leader election, so it works correctly even if multiple replicas were deployed (though the base deployment uses a single replica). The readiness probe gates on certificate readiness, ensuring the operator doesn't advertise itself as ready until webhooks can serve requests.

A notable gap is that NetworkPolicy isolation is only implemented for PyTorchJob. TFJob, XGBoostJob, PaddleJob, JAXJob, and MPIJob do not create NetworkPolicies, meaning their training pods are accessible to any pod in the namespace. This may be intentional (simplicity for other frameworks) or a gap worth addressing for security-sensitive deployments. The PyTorch NetworkPolicy implementation uses server-side apply with force, which is idempotent but could overwrite user-applied network policies if they share the same name.
