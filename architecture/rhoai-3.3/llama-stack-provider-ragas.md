## Purpose

**Short**: Implements Ragas evaluation framework as an out-of-tree Llama Stack provider, enabling LLM evaluation through both inline (in-process) and remote (Kubeflow Pipelines) execution modes.

**Detailed**: llama-stack-provider-ragas bridges the Ragas evaluation library with Meta's Llama Stack evaluation API. It registers as an external provider for Llama Stack's `eval` API, enabling users to run Ragas evaluation metrics (answer relevancy, context precision, faithfulness, context recall) against LLM outputs through the standard Llama Stack interface.

The provider ships in two execution modes. The **inline** mode runs Ragas evaluation in the same process as the Llama Stack server, wrapping the Llama Stack inference API (for LLM completions and embeddings) to present them as Ragas-compatible LLM/embedding interfaces. The **remote** mode offloads evaluation to Kubeflow Pipelines (Data Science Pipelines), constructing a two-step KFP pipeline that first retrieves dataset rows from the Llama Stack server and then runs Ragas evaluation in an isolated container, storing results in S3. This remote mode is designed for production RHOAI environments where evaluation workloads should be scheduled and tracked through the platform's pipeline infrastructure.

The component does not define any CRDs, controllers, or Kubernetes operators. It is a Python library that is either installed directly into a Llama Stack server process (inline mode) or packaged as a container image consumed by Kubeflow Pipelines (remote mode). The container image for Konflux builds is based on UBI9 Python 3.12.

## Architectural Analysis

The llama-stack-provider-ragas repository follows a clean provider plugin architecture for Llama Stack. The codebase is structured around a **dual-mode execution pattern**: the same Ragas evaluation logic can run either inline (in-process with the Llama Stack server) or remotely (as Kubeflow Pipeline jobs). This is achieved through a shared configuration model (`RagasProviderBaseConfig`) with mode-specific extensions (`RagasProviderInlineConfig`, `RagasProviderRemoteConfig`), and separate wrapper classes for the LLM and embedding interfaces.

The **wrapper pattern** is the central design choice. Both inline and remote modes must bridge Ragas's LangChain-derived interface (`BaseRagasLLM`, `BaseRagasEmbeddings`) with Llama Stack's inference API. The inline wrappers (`wrappers_inline.py`) use the Llama Stack server's in-process inference API directly via `async` calls, while the remote wrappers (`wrappers_remote.py`) use the `llama_stack_client` HTTP client to call back to the Llama Stack server from KFP pods. This means remote evaluations create a callback loop: the Llama Stack server submits a KFP pipeline, and the KFP pods call back to the same Llama Stack server for LLM inference. This has important networking implications — the Llama Stack server must be reachable from KFP pods, which the `KUBEFLOW_LLAMA_STACK_URL` configuration addresses.

The **compatibility layer** (`compat.py`) is noteworthy — it provides backward compatibility between the legacy `llama_stack` package layout and the newer `llama_stack_api` module structure, allowing the provider to work across Llama Stack versions. This dual-import pattern adds resilience but also complexity, and means the provider's API surface depends on which Llama Stack version is installed.

The **base image resolution** for KFP components (`components.py:get_base_image()`) follows a three-tier fallback: environment variable → Kubernetes ConfigMap (`trustyai-service-operator-config` in `redhat-ods-applications` or `opendatahub` namespaces) → hardcoded default. This couples the provider to the TrustyAI Service Operator's configuration infrastructure and to specific RHOAI/ODH namespace conventions. The ConfigMap lookup happens at module import time (via `@dsl.component(base_image=get_base_image())`), which means the Kubernetes API must be reachable when the module is loaded.

A security consideration is that the Llama Stack server's eval API, as configured in the sample distribution, has **no authentication**. The KFP pods call back to the server without any auth headers. In a production RHOAI deployment, network policies or service mesh would need to restrict access. Additionally, the KFP bearer token is handled via `SecretStr` in Pydantic (ensuring log redaction) but falls back to the local kubeconfig token, which ties the provider's auth lifecycle to the operator's `oc login` session.
