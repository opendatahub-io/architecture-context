## Purpose

**Short**: Manages the lifecycle of Ray clusters, jobs, and services on Kubernetes/OpenShift via custom resources (RayCluster, RayJob, RayService).

**Detailed**: The KubeRay operator is a Kubernetes operator built with controller-runtime that provides declarative management of Ray distributed computing infrastructure. It defines three primary CRDs — RayCluster, RayJob, and RayService — and reconciles them into the underlying Kubernetes resources (Pods, Services, Ingress/Routes, RBAC, NetworkPolicies) needed to run Ray workloads. The operator supports advanced features including in-tree autoscaling, GCS fault tolerance with external Redis, batch scheduling integration (Volcano, YuniKorn, scheduler-plugins), and zero-downtime upgrades for RayService.

On RHOAI/OpenShift, the operator provides deep platform integration: it uses Gateway API (HTTPRoute) for ingress routing through the platform's data-science-gateway, injects kube-rbac-proxy sidecars for OIDC-based authentication on Ray dashboards, manages mTLS certificates via cert-manager for intra-cluster encryption, creates NetworkPolicies for network segmentation, and deploys a SecurityContextConstraint (SCC) for Ray workloads. A mutating webhook automatically configures security defaults on OpenShift clusters.

The operator is deployed as a single Deployment (`kuberay-operator`) in the platform namespace and watches RayCluster CRs across configured namespaces. It registers five controllers (RayCluster, RayJob, RayService, mTLS, NetworkPolicy, Authentication) and two webhooks (mutating and validating for RayCluster).

## Architectural Analysis

The KubeRay operator exemplifies a well-structured multi-controller architecture where each concern (cluster lifecycle, mTLS, network policy, authentication) is separated into its own controller watching the same RayCluster CRD. This separation allows each controller to evolve independently while coordinating through annotations (`odh.ray.io/secure-trusted-network`) and status conditions (`AuthenticationReady`). The RayCluster controller defers pod creation until the AuthenticationController has prepared the ServiceAccount and kube-rbac-proxy ConfigMap, demonstrating a clean dependency chain without tight coupling.

The RHOAI authentication model follows a gateway-sidecar pattern: the AuthenticationController creates an HTTPRoute in the platform namespace (not the user's namespace) that routes `/ray/{namespace}/{cluster}` paths through the platform's data-science-gateway, while simultaneously injecting a kube-rbac-proxy sidecar into the Ray head pod. This sidecar performs TokenReview and SubjectAccessReview against the Kubernetes API to validate OIDC tokens before proxying to the Ray dashboard. Cross-namespace access is enabled through ReferenceGrants, which are carefully managed — shared across all RayClusters in a namespace and only cleaned up when the last authenticated cluster is removed.

The mTLS implementation leverages cert-manager with a two-tier PKI: a self-signed issuer bootstraps a root CA, which then issues separate head and worker certificates with appropriate SANs (including all pod IPs, service FQDNs, and localhost). Certificate rotation is automatic via cert-manager's renewal mechanism (90-day duration, 15-day renewal window). The controller monitors certificate expiry and logs warnings for certificates expiring within 24 hours. When pods lack the expected TLS configuration (volume mount or env var), they are deleted and recreated with correct mTLS setup.

The NetworkPolicyController creates a layered security model: worker pods are fully isolated (only intra-cluster traffic allowed), while head pods have granular ingress rules — intra-cluster communication, same-namespace access on client (10001) and dashboard (8265) ports, operator access from the platform namespace, optional monitoring access on metrics port (8080), optional gateway access on dashboard port (8265), and an open rule for the kube-rbac-proxy secured port (8443). The gateway namespace defaults to `openshift-ingress` on OpenShift but is configurable via the `GATEWAY_NAMESPACE` environment variable.

The operator's OpenShift integration is primarily driven by the mutating webhook, which detects OpenShift by checking for the Route API in the cluster's REST mapper. On OpenShift, it sets `odh.ray.io/secure-trusted-network=true` by default and disables vanilla Kubernetes Ingress (forcing Gateway API mode). This means all RayClusters on RHOAI automatically get mTLS, NetworkPolicies, and authenticated dashboard access unless explicitly opted out. The webhook's platform detection approach is notable — it uses API discovery rather than environment variables, making it robust across different deployment scenarios.
