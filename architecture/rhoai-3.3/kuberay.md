## Purpose
**Short**: KubeRay Operator manages the lifecycle of Ray clusters, jobs, and services on Kubernetes/OpenShift.

**Detailed**: The KubeRay Operator is a controller-runtime based Kubernetes operator that manages three custom resources: RayCluster, RayJob, and RayService. It handles the complete lifecycle of Ray distributed computing clusters, including head and worker node pod management, service creation, autoscaling, and batch scheduling integration (Volcano, Yunikorn).

In the RHOAI distribution, the operator includes significant OpenShift-specific enhancements: an Authentication Controller that injects kube-rbac-proxy sidecars and creates Gateway API HTTPRoute resources for secure access to Ray dashboards, a NetworkPolicy Controller that creates per-cluster network policies for workload isolation, an mTLS Controller that provisions cert-manager certificates for inter-node encryption, and mutating/validating webhooks that enforce security defaults on OpenShift (disabling basic Route/Ingress and enabling secure trusted networking). The operator creates HTTPRoutes in the platform namespace referencing the `data-science-gateway` Gateway in `openshift-ingress`, with cross-namespace ReferenceGrants to allow traffic to flow to user-namespace Ray head services.

**Controller-Managed Ingress (RHOAI 3.x Pattern)**:
- The **Authentication Controller** creates `HTTPRoute` CRs in the platform namespace (default: `opendatahub`) referencing the `data-science-gateway` Gateway in `openshift-ingress`
- Each RayCluster with authentication gets an HTTPRoute at path `/ray/{namespace}/{cluster}` with URL rewrite to `/`
- Cross-namespace service references are enabled via `ReferenceGrant` CRs created in the user's namespace
- The controller injects `kube-rbac-proxy` sidecar containers (port 8443/TCP) into Ray head pods for RBAC-based authentication
- Basic OpenShift Route/Ingress creation is **disabled** by the mutating webhook (`enableIngress=false`) on OpenShift
- The `reconcileIngress` function in RayClusterReconciler cleans up any existing Routes/Ingresses when `enableIngress=false`

### Network Policies (Dynamically Created)

The NetworkPolicy Controller creates two NetworkPolicies per RayCluster when `ray.io/enable-secure-trusted-network: "true"` annotation is set:

**Head Pod NetworkPolicy** (`{cluster}-head`):
- Rule 1: Intra-cluster communication (all ports) from pods with matching `ray.io/cluster` label
- Rule 2: Same-namespace pods access to ports 10001/TCP (client) and 8265/TCP (dashboard)
- Rule 3: KubeRay operator namespace access to ports 8265/TCP and 10001/TCP
- Rule 4 (optional): Monitoring namespace access to port 8080/TCP (metrics)
- Rule 4b: Gateway namespace (`openshift-ingress`) access to port 8265/TCP (dashboard)
- Rule 5: Secured ports (8443/TCP kube-rbac-proxy) open from anywhere
- Rule 6 (conditional): RayJob submitter pod access (all ports) if cluster owned by RayJob

**Worker Pod NetworkPolicy** (`{cluster}-workers`):
- Only intra-cluster communication from pods with matching `ray.io/cluster` label

## Kustomize Overlay Structure (RHOAI)

The operator is deployed via the `ray-operator/config/openshift/` kustomization overlay, which:
1. Includes base resources from `../default` (CRDs, RBAC, Manager deployment)
2. Adds OpenShift-specific resources: SCC (`run-as-ray-user`), webhook Service + MutatingWebhookConfiguration
3. Patches the operator Deployment to:
   - Set the container image via `$(image)` from params.env
   - Enable webhooks (`ENABLE_WEBHOOKS=true`) and mount webhook cert volume
   - Inject `APPLICATION_NAMESPACE` and `POD_NAMESPACE` env vars for Gateway API HTTPRoute placement
4. Sets namespace to `opendatahub` and applies common labels

**Image injection by rhods-operator**:
- `odh-kuberay-operator-controller-image` -> `RELATED_IMAGE_ODH_KUBERAY_OPERATOR_CONTROLLER_IMAGE`
- Default: `quay.io/opendatahub/kuberay-operator:v1.4.4`
