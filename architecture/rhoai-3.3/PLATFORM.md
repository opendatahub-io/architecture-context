## 1. Platform Overview

Red Hat OpenShift AI (RHOAI) 3.3 is a comprehensive AI/ML platform deployed on OpenShift Container Platform. It provides an integrated environment for data scientists, ML engineers, and platform administrators to develop, train, serve, monitor, and govern AI/ML models at enterprise scale.

The platform is composed of 40 component repositories that together deliver:

- **Interactive Development**: Jupyter notebooks, Code Server, and RStudio workbenches
- **Model Training**: Distributed training via Kubeflow Training Operator, Trainer (v2), KubeRay, and Kueue job scheduling
- **Model Serving**: KServe-based inference with multiple runtimes (vLLM, OpenVINO, MLServer, Llama Stack)
- **AI Pipelines**: Kubeflow Pipelines with Argo Workflows engine
- **Model Registry**: Centralized model metadata, versioning, and lifecycle management
- **Experiment Tracking**: MLflow-based experiment tracking and artifact management
- **Feature Store**: Feast-based feature engineering and serving
- **AI Safety & Trust**: TrustyAI fairness/bias monitoring, guardrails orchestration, NeMo Guardrails, LM evaluation
- **Models-as-a-Service**: API gateway with token/key management and tier-based rate limiting via Kuadrant
- **Platform Management**: Unified dashboard, Gateway API-based ingress, OAuth/OIDC authentication, monitoring

### Architecture Principles

1. **Operator-Managed Lifecycle**: The central `rhods-operator` manages all component lifecycles via `DataScienceCluster` and `DSCInitialization` CRDs
2. **Gateway API-First Ingress**: RHOAI 3.x uses Kubernetes Gateway API (`Gateway`, `HTTPRoute`, `GatewayClass`) with Envoy-based routing, replacing legacy Route+oauth-proxy patterns
3. **Authentication via External Authorization**: All Gateway traffic passes through an EnvoyFilter ext_authz chain to `kube-auth-proxy` for OAuth2/OIDC authentication
4. **Disconnected/Air-Gapped Support**: All 82 container images are declared via `RELATED_IMAGE_*` environment variables in the OLM bundle
5. **FIPS Compliance**: Go components built with `GOEXPERIMENT=strictfipsruntime`; TLS configurations use FIPS 140-2 approved cipher suites
6. **Multi-Architecture**: Builds target amd64, arm64, ppc64le, and s390x

## 5. Network Architecture

### 5.1 Platform Gateway (Primary Ingress)

RHOAI 3.x uses a centralized Gateway API architecture for all platform ingress:

| Resource | Name | Namespace | Purpose |
|----------|------|-----------|---------|
| GatewayClass | data-science-gateway-class | N/A | Controller: `openshift.io/gateway-controller/v1` |
| Gateway | data-science-gateway | openshift-ingress | HTTPS listener on port 443, TLS termination |
| EnvoyFilter | data-science-authn-filter | openshift-ingress | ext_authz to kube-auth-proxy + Lua filter for token injection |
| DestinationRule | data-science-tls-rule | openshift-ingress | mTLS for ports 443 and 8443 |
| NetworkPolicy | kube-auth-proxy | openshift-ingress | Restricts ingress to auth proxy |

**Ingress Modes:**
- **OcpRoute mode** (default): Gateway uses ClusterIP, exposed via OpenShift Route with reencrypt TLS
- **LoadBalancer mode**: Gateway uses LoadBalancer service type directly

**Authentication Flow:**
1. Client -> OpenShift Router/LB (443/TCP, HTTPS)
2. Router/LB -> Gateway Envoy (443/TCP, HTTPS)
3. Envoy -> kube-auth-proxy ext_authz (8443/TCP, HTTPS with service-CA)
4. kube-auth-proxy -> OpenShift OAuth / External OIDC Provider (443/TCP, HTTPS)
5. Envoy -> Backend service via HTTPRoute (Bearer token injected by Lua filter)

## 7. Data Flows

### 7.1 Model Inference (End-to-End)

```
Client (HTTPS, port 443)
  --> OpenShift Router / LoadBalancer
    --> data-science-gateway (Envoy)
      --> EnvoyFilter ext_authz --> kube-auth-proxy (8443)
        --> OpenShift OAuth / OIDC Provider (token validation)
      --> HTTPRoute match
        --> kube-rbac-proxy sidecar (8443, optional)
          --> Serving Runtime Container (8000/8080)
            --> Model weights (PVC / S3 / HuggingFace)
          <-- Inference response (JSON / SSE stream)
        <-- Response
      <-- Response
    <-- Response (Bearer token stripped by Lua filter)
  <-- Response
```

### 7.2 Model Training (TrainJob)

```
User creates TrainJob CR
  --> Kubernetes API (webhook validation, port 9443)
    --> Trainer controller reconciles
      --> Creates JobSet CR
        --> JobSet controller creates Jobs/Pods
          --> Creates NetworkPolicy (pod isolation)
      --> Training Pods communicate:
        - NCCL/MPI/gRPC (port 29500+)
        - Progression tracking (HTTP port 28080)
      --> Controller polls training metrics
      --> Updates TrainJob status/annotations
```

### 7.3 AI Pipeline Execution

```
User creates PipelineRun
  --> ds-pipeline API server (port 8888)
    --> Persists to database (MySQL/PostgreSQL)
    --> Creates Argo Workflow CR
      --> Argo controller creates Pods
        --> Pipeline steps execute
          --> Log artifacts to ML Metadata (gRPC port 8080)
          --> Store artifacts to Object Storage (S3)
    --> Persistence agent reconciles status
    --> Cache server checks for cache hits (port 8443)
```

### 7.4 AI Safety (Guardrails)

```
Client chat completion request
  --> vllm-orchestrator-gateway (port 8090)
    --> Injects detector specs
    --> FMS Guardrails Orchestrator (port 8033)
      --> Input detectors scan request
        --> guardrails-detectors (port 8080)
        --> guardrails-regex-detector (port 8080)
      --> LLM Backend inference (vLLM)
      --> Output detectors scan response
    <-- Detection results
  --> Gateway applies fallback if detections found
  <-- Response (filtered or original)
```

### 7.5 TrustyAI Bias Monitoring

```
Model inference occurs:
  --> ModelMesh: POST /consumer/kserve/v2 (port 80)
  --> KServe: CloudEvent (inference.request/response)
    --> TrustyAI service stores payload (PVC/DB/MinIO)

Scheduled metric computation:
  --> Quarkus scheduler triggers
    --> Computes SPD, DIR, drift metrics
    --> Exports to Prometheus (/q/metrics)
      --> Prometheus scrapes
        --> Dashboard/AlertManager visualization
```

## 13. Disconnected / Air-Gapped Support

RHOAI 3.3 fully supports disconnected (air-gapped) installations:

1. **RELATED_IMAGE_* References**: All 82 container images are declared as `RELATED_IMAGE_*` environment variables in the OLM CSV, enabling `oc adm catalog mirror` to discover and mirror all required images.

2. **ImageDigestMirrorSet**: The `RHOAI-Build-Config` repository ships an `ImageDigestMirrorSet` manifest enabling transparent image mirroring from `registry.redhat.io/rhoai/*` to `quay.io/rhoai/*`.

3. **Pre-downloaded Models**: The NeMo-Guardrails container pre-downloads NLP models (spaCy, sentence-transformers, FastEmbed, NLTK) at build time for offline operation.

4. **Model Metadata Container**: The `model-metadata-collection` component produces a static data container with pre-generated model catalogs, eliminating runtime dependency on HuggingFace API.

5. **Offline Model Loading**: vLLM containers support loading models from PVC-mounted storage (`HF_HUB_OFFLINE=1`), and OVMS supports local filesystem model repositories.
