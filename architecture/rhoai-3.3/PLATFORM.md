## Platform Overview

Red Hat OpenShift AI (RHOAI) 3.3 is an enterprise machine learning platform deployed on OpenShift Container Platform via the Operator Lifecycle Manager (OLM). At its core, the `rhods-operator` acts as a meta-operator that manages the lifecycle of 14 optional data science components through a two-level CRD hierarchy: `DSCInitialization` establishes platform prerequisites (namespaces, authentication, gateway infrastructure, monitoring), while `DataScienceCluster` enables individual components such as model serving (KServe), pipelines (DSP), notebooks (Kubeflow), training (Training Operator, Trainer), model registry, feature stores (Feast), and AI safety tooling (TrustyAI, Guardrails). Each component self-registers with the operator and follows a uniform action chain: initialize → render kustomize manifests → deploy → check status → garbage collect.

The ingress architecture implements a multi-layer Gateway API stack deployed entirely through controller reconciliation. External requests traverse an Envoy-based Gateway, which invokes an `ext_authz` EnvoyFilter to consult `kube-auth-proxy` (an OAuth2/OIDC proxy). The proxy validates against OpenShift OAuth or an external OIDC provider, and a Lua filter extracts access tokens for forwarding to upstream services. This supports three authentication modes (IntegratedOAuth, OIDC, None) and two ingress modes (OpenShift Route with re-encryption, and LoadBalancer with direct DNS). Components expose services via HTTPRoute (Gateway API), OpenShift Routes, or Istio VirtualServices depending on their maturity level.

The security posture is defense-in-depth: container images are predominantly UBI 9-based with non-root execution (USER 1001 in 81 Dockerfiles), Go binaries are compiled with `GOEXPERIMENT=strictfipsruntime` for FIPS compliance, admission webhooks enforce validation across all major CRDs with `failurePolicy: Fail`, and NetworkPolicies restrict inter-component traffic. The platform ships 96 container images referenced via `RELATED_IMAGE_*` environment variables for disconnected/air-gapped deployment support through OLM's `relatedImages` mechanism.

### Central Components

**rhods-operator** is the platform root — it deploys all other components via DSC component CRs and manages platform-wide infrastructure (Gateway, Auth, Monitoring). It has 193 controller watches and manages 23 CRDs spanning component lifecycle, platform services, and infrastructure resources.

**KServe** is the most depended-on runtime component (8 reverse dependencies). It serves as the universal model serving layer, managing InferenceService and LLMInferenceService CRDs. Components including odh-dashboard, Model Registry, TrustyAI, ODH Model Controller, and the llm-d stack all integrate with KServe for model deployment, monitoring, and lifecycle management.

**odh-dashboard** is the primary integration hub (21 forward dependencies). It proxies to Data Science Pipelines, Model Registry, TrustyAI, ML Metadata, and Llama Stack services, and provides the user-facing interface for managing notebooks, model serving, and pipelines. Its module federation architecture dynamically loads UI extensions from BFF sidecars.

**ODH Model Controller** is the platform's networking and policy orchestrator for serving workloads (11 forward dependencies, 40 controller watches). It manages OpenShift Routes, AuthPolicies, EnvoyFilters, RBAC, metrics services, and KEDA autoscalers for both InferenceService and LLMInferenceService resources.

**Data Science Pipelines Operator (DSPO)** manages the full pipeline infrastructure stack — API server, persistence agent, scheduled workflow controller, ML Metadata, Argo Workflows, and backing databases — through a template-driven manifest deployment pattern using the manifestival library.

## Deployment Architecture

### Deployment Topology

The platform follows a hub-and-spoke topology. The **operator namespace** (`redhat-ods-operator`) contains only the rhods-operator pod, deployed via OLM. When the administrator creates a DSCInitialization CR, the operator provisions the **applications namespace** (`redhat-ods-applications`) and deploys enabled component controllers there. The operator then provisions **gateway infrastructure** in `openshift-ingress` — a Gateway API Gateway, kube-auth-proxy deployment, EnvoyFilter, and supporting resources. User workloads (notebooks, inference services, pipelines, training jobs) run in **user-created namespaces** (data science projects), with controllers in the applications namespace watching across all namespaces for their CRDs.

Components that require Istio service mesh (KServe serverless mode) operate within mesh-enrolled namespaces, while the majority of components use the platform Gateway for ingress without requiring mesh enrollment. The monitoring stack (Prometheus, Tempo, OpenTelemetry, Perses) is deployed by the rhods-operator Monitoring service controller within the applications namespace.

#### Manifest Consumption Pipeline

Component manifests flow through a multi-stage pipeline: component repositories maintain kustomize bases and overlays → the `get_all_manifests.sh` script in the rhods-operator repository fetches specific commits/branches from each component repository → manifests are stored in `prefetched-manifests/` and embedded in the operator binary at build time → at runtime, the operator renders the appropriate overlay (ODH or RHOAI) for the target distribution and applies manifests to the cluster. The Konflux CI system builds the operator image with these embedded manifests, ensuring reproducibility. Image references in params.env files are replaced with digest-pinned RELATED_IMAGE_* environment variables from the operator CSV, enabling disconnected deployment.

### Disconnected Support

RHOAI 3.3 supports air-gapped (disconnected) deployment through the OLM `relatedImages` mechanism. All 96 container images are declared in the operator CSV with digest-pinned references via `RELATED_IMAGE_*` environment variables. The rhods-operator substitutes these environment variable references into component deployment manifests at reconciliation time, ensuring all images are pulled from the mirrored registry configured via `ImageContentSourcePolicy` or `ImageDigestMirrorSet`. Image mirroring is performed using `oc-mirror` against the operator catalog. The operator features metadata declares `disconnected: "true"`, indicating full disconnected support.

## Platform Architectural Analysis

RHOAI 3.3 demonstrates a mature meta-operator pattern where `rhods-operator` acts as the single control plane entry point, managing a heterogeneous ecosystem of 44 components through a uniform component handler interface. This design provides excellent operational simplicity — a single OLM subscription manages the entire platform — but creates a concentration risk: the operator binary embeds manifests for all components, meaning any component manifest update requires a new operator build. The two-tier CRD hierarchy (DSCInitialization for infrastructure, DataScienceCluster for components) cleanly separates platform setup from workload capabilities, and the automatic recreation of accidentally-deleted singleton service CRs (Auth, GatewayConfig, Monitoring) provides self-healing behavior.

A significant architectural tension exists in the authentication layer. The platform is transitioning from the legacy `oauth-proxy` pattern to `kube-rbac-proxy` and the centralized Gateway ext_authz model, but components are at different stages of this migration. The odh-dashboard already uses kube-rbac-proxy, while DSPO still deploys it for service endpoints. The LLM serving stack introduces a third approach via Kuadrant AuthPolicy for API-level authentication. This means a single platform currently runs three different authentication enforcement mechanisms, each with different token validation flows, RBAC semantics, and failure modes. The `InsecureSkipVerify: true` workarounds in at least four components (KServe, rhods-operator, Model Registry, llm-d-inference-scheduler) represent a systemic TLS verification gap that is documented as temporary but has persisted across releases.

The LLM serving architecture (KServe LLMInferenceService + llm-d stack) represents the most architecturally ambitious subsystem, introducing Gateway API Inference Extensions, prefill/decode disaggregation, KV-cache-aware routing, and multi-node scheduling via LeaderWorkerSet. The LLMInferenceService controller creates between 5 and 25+ resources per CR depending on configuration (single-node vs multi-node × P/D vs non-P/D × managed vs custom routing), making it the most complex reconciliation path in the platform. The llm-d-inference-scheduler's multi-language build (Go + Rust FFI + Python Jinja2) for tokenization and KV-cache block matching is architecturally motivated but adds significant build and debugging complexity.

The FIPS compliance story reveals a platform-wide gap between intent and implementation. The operator CSV explicitly declares `fips-compliant: "false"` even though most Go binaries are compiled with `GOEXPERIMENT=strictfipsruntime`. The specific gaps include: KServe's localmodel components built with `CGO_ENABLED=0` on distroless (bypassing FIPS crypto), the llm-d-inference-scheduler lacking strictfipsruntime entirely, and multiple components using `InsecureSkipVerify: true` in TLS configurations. Python and Rust components (vLLM, FMS Guardrails Orchestrator, Guardrails Regex Detector) rely on system-level FIPS enforcement via UBI 9's OpenSSL rather than language-level FIPS controls, which may be insufficient for strict certification.

Cross-component observability follows a consistent pattern: operators expose Prometheus metrics endpoints (typically on port 8443 behind kube-rbac-proxy or port 8080 for direct scraping), the rhods-operator Monitoring controller dynamically assembles PrometheusRules, and ServiceMonitor/PodMonitor resources are created by individual component operators. The monitoring stack includes Tempo for distributed tracing, OpenTelemetry collectors for trace collection, and Perses dashboards — a notable investment in observability that extends beyond basic metrics. However, the OpenTelemetry integration is uneven: some components (Kubeflow Notebook Controllers, FMS Guardrails Orchestrator, llm-d-kv-cache) export traces natively, while most operators only expose Prometheus metrics.

The controller-per-framework pattern in Training Operator (PyTorch, TensorFlow, MPI, XGBoost, Paddle, JAX) alongside the newer Kubeflow Trainer (TrainJob with TrainingRuntime/ClusterTrainingRuntime) reveals a platform in the middle of a training API evolution. Both operators coexist, with Training Operator handling the established per-framework CRDs and Trainer introducing a unified TrainJob abstraction. The NetworkPolicy isolation gap in Training Operator (only PyTorchJob has NetworkPolicies; the other five frameworks do not) is a security inconsistency worth noting for multi-tenant deployments.

The odh-dashboard's module federation architecture (host React app + Go BFF sidecars in a single pod) is a distinctive micro-frontend approach for the Kubernetes operator ecosystem. It enables independent development and deployment cycles for UI modules (model-registry-ui, gen-ai-ui, maas-ui) while keeping operational complexity low through pod co-location. The trade-off is that a BFF sidecar failure can affect the entire dashboard pod, though per-container health probes mitigate this risk.
