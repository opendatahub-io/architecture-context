## Purpose

**Short**: Builds and publishes container images for interactive notebook workbenches (JupyterLab, Code Server, RStudio) and pipeline runtime environments used by RHOAI data scientists.

**Detailed**: The notebooks repository is a multi-image build factory that produces all interactive development environment (IDE) container images shipped with Red Hat OpenShift AI. It does not contain an operator or controller — instead, it defines Dockerfiles, Python dependency lock files, and Kustomize manifests that describe OpenShift ImageStreams for each notebook image variant.

The repository produces three categories of images: (1) **Workbench images** — full IDE environments based on JupyterLab, VS Code (code-server), or RStudio Server, pre-loaded with data science libraries and the OpenShift CLI; (2) **Runtime images** — lightweight Python environments designed for headless execution of Elyra pipeline nodes; and (3) **Base images** — foundational layers for CPU, CUDA, and ROCm that the workbench/runtime images build upon. Each category includes variants for different hardware accelerators (CPU-only, NVIDIA CUDA, AMD ROCm) and ML frameworks (PyTorch, TensorFlow, LLM Compressor, TrustyAI).

Images are built by Konflux via Tekton pipelines, with base images sourced from `quay.io/aipcc/base-images` (AIPCC-managed) and previous RHOAI release images from `registry.redhat.io/rhoai/`. The Kustomize manifests under `manifests/` define OpenShift ImageStream resources that the platform operator (`rhods-operator`) deploys to make these images available in the RHOAI dashboard's workbench selector.

## Architectural Analysis

The notebooks repository follows a **layered image build pattern** where images are composed through multi-stage Dockerfile builds and base image inheritance. At the foundation, AIPCC-managed base images (`quay.io/aipcc/base-images/cpu|cuda|rocm`) provide the OS layer (RHEL 9.6) with hardware-specific drivers. The repo's own `base-images/` directory defines intermediate base images that install common tools (via `aipcc.sh`) and CUDA/ROCm toolkits. Workbench and runtime Dockerfiles then extend these bases with application-specific packages.

The **Konflux build system** has driven significant architectural decisions. Each shippable image has a dedicated `Dockerfile.konflux.*` file and a corresponding `build-args/konflux.*.conf` file that pins the base image to a specific AIPCC version tag. This separation from regular Dockerfiles (which may use different base images for local development) ensures reproducible builds. The Tekton pipeline definitions in `.tekton/` cover all image variants with pull-request and push pipelines. The repository contains approximately 8,700 lines of Tekton YAML — a substantial CI/CD surface area.

**Multi-architecture support** is a notable complexity driver. CPU images support four architectures (amd64, arm64, ppc64le, s390x), which requires complex multi-stage Dockerfile builds. The data science images include dedicated builder stages for ppc64le (compiling OpenBLAS 0.3.30, ONNX 1.20.1, PyArrow 21.0.0 from source) and s390x (compiling PyArrow with custom patches). CUDA images support amd64 and arm64; ROCm images are amd64-only. This architecture matrix is managed by the `scripts/sandbox.py` wrapper which detects platform and adjusts build context.

**Python dependency management** uses `pylock.toml` files (the PEP 751 lock file format) with `uv pip install --verify-hashes` for cryptographic verification of all packages. Each image variant has its own `pylock.toml`, and the `scripts/pylocks_generator.sh` script regenerates them. The repository supports two index modes (`public-index` and `aipcc-index`) for different build environments, controlled by the `INDEX_MODE` variable. This lock file infrastructure provides strong supply chain integrity at the Python layer, though RPM-level packages remain unlocked.

The **Kustomize manifests** serve a dual purpose: they define OpenShift ImageStream resources (the Kubernetes-native way to represent container image metadata) and they encode rich metadata via annotations (`opendatahub.io/notebook-software`, `opendatahub.io/notebook-python-dependencies`). The RHOAI Dashboard consumes these annotations to present users with an image catalog showing software versions, recommended images, and accelerator requirements. The `replacements` mechanism in `kustomization.yaml` maps over 90 parameterized image references from `params.env` into the ImageStream tag specifications, enabling release-by-release image pinning.

The **code-server and RStudio images** share an architectural pattern distinct from Jupyter: they use NGINX 1.24 as a reverse proxy fronting the IDE server, with Apache httpd providing CGI endpoints (`access.cgi`, `probe.cgi`) for health checks. This pattern exists because code-server and RStudio Server don't natively expose the health endpoints that Kubernetes probes expect, so the NGINX+CGI layer adapts them. The code-server image also builds code-server from source RPM (rather than using a pre-built binary) to support ppc64le and s390x architectures.
