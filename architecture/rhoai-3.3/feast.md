## Purpose
**Short**: Feast Operator is a Kubernetes operator that manages the lifecycle of Feast (Feature Store) deployments, providing automated provisioning of online, offline, and registry feature store services.

**Detailed**: The Feast Operator manages `FeatureStore` custom resources to deploy and reconcile the Feast feature store infrastructure on Kubernetes and OpenShift clusters. It automates the deployment of multiple feast service types: online store (serving features for real-time inference), offline store (serving features for batch training), registry (metadata catalog of feature definitions), and an optional UI server for visualization.

The operator handles OpenShift-specific integrations including automatic TLS certificate provisioning via OpenShift service serving certificates, Route creation for UI exposure, and trusted CA bundle injection for RHOAI/ODH deployments. It supports multiple persistence backends (file-based with PVCs, and database backends including Snowflake, Cassandra, DuckDB, and SQL) and multiple authorization models (Kubernetes RBAC and OIDC). The operator also manages a namespace registry ConfigMap to enable cross-namespace feature store discovery, and integrates with Kubeflow Notebooks via a secondary controller that creates feast configuration ConfigMaps for notebooks with the feast integration label.

For RHOAI deployments, the operator is deployed by the rhods-operator using kustomize overlays. Container images are injected via `RELATED_IMAGE_*` environment variables to support disconnected/air-gapped installations. The operator binary is built with FIPS-compliant Go runtime (`GOEXPERIMENT=strictfipsruntime`).

| FeatureStoreReconciler | Controller | Watches FeatureStore CRs and owned resources (Deployments, Services, ConfigMaps, PVCs, ServiceAccounts, RoleBindings, Roles, CronJobs, Routes); reconciles desired state |
| NotebookConfigMapReconciler | Controller | Watches Kubeflow Notebooks with `opendatahub.io/feast-integration` label; creates per-notebook ConfigMaps with feast client configuration |
| FeastAuthorization | Authorization Module | Deploys Kubernetes RBAC resources (Roles, RoleBindings, ClusterRoles, ClusterRoleBindings) when Kubernetes auth mode is configured |
| Feature Server | Python Application | Serves feast features via HTTP/gRPC endpoints for online, offline, and registry services; deployed as containers within operator-managed Deployments |
| CronJob | Batch Job | Runs `feast apply` and `feast materialize-incremental` commands on a schedule using `kubectl exec` into the feast deployment |

### TLS Configuration

On OpenShift, if no TLS is explicitly configured in the FeatureStore spec, the operator automatically provisions TLS using OpenShift service serving certificates:
- Service annotation `service.beta.openshift.io/serving-cert-secret-name` triggers automatic certificate creation
- For registry services with both gRPC and REST enabled, SANs are added via `service.beta.openshift.io/serving-cert-sans` annotation
- A CA ConfigMap with annotation `service.beta.openshift.io/inject-cabundle: true` is created for client certificate verification
- Custom CA bundles from `odh-trusted-ca-bundle` ConfigMap (labeled `config.openshift.io/inject-trusted-cabundle: true`) are auto-mounted to all feast containers
- FIPS compliance is ensured via `GOEXPERIMENT=strictfipsruntime` build flag
