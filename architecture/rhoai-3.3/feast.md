## Purpose

**Short**: Feast is an open-source feature store for machine learning that manages the storage, retrieval, and serving of features for model training and inference.

**Detailed**: Feast (Feature Store) provides a complete feature management platform consisting of a Kubernetes operator and a feature serving layer. The feast-operator is a Go-based controller-runtime operator that manages `FeatureStore` custom resources, orchestrating the deployment of offline stores, online stores, registry servers, and web UIs. It dynamically creates Deployments, Services, ConfigMaps, PVCs, RBAC resources, CronJobs, and (on OpenShift) Routes and TLS certificates.

The feature server component is a Python-based service built on FastAPI/Uvicorn that exposes HTTP REST endpoints for online feature retrieval, materialization, and push operations. It supports multiple online store backends (Redis, SQLite, DynamoDB, PostgreSQL, etc.) and registry backends (file, GCS, S3, SQL). A complementary Go feature serving library provides high-performance gRPC and HTTP serving with the same backend support. The operator integrates with OpenDataHub/RHOAI through Kubeflow Notebook integration, automatically distributing feature store client configurations to Notebook pods via ConfigMaps.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| feast-operator | Go Operator (controller-runtime) | Manages FeatureStore CRDs, deploys and configures all feast services (offline store, online store, registry, UI), handles RBAC, TLS, auth, and CronJob scheduling |
| feature-server (Python) | Python Service (FastAPI/Uvicorn) | REST API for online feature retrieval, materialization, push, and document retrieval with support for OIDC/Kubernetes auth |
| feature-server (Go) | Go Service (gRPC/HTTP) | High-performance gRPC and HTTP serving for online features with Redis, SQLite, DynamoDB backends |
| feast-ui | React/TypeScript Frontend | Web UI for browsing feature store metadata |
| feast CLI | Python CLI (Click) | Command-line tool for feature store management (feast apply, feast materialize, etc.) |

## Architectural Analysis

The Feast component follows a multi-language architecture that spans Go (operator and high-performance serving), Python (feature server, SDK, CLI), and TypeScript (UI). The operator is cleanly separated in `infra/feast-operator/` with its own `go.mod`, while the Go serving layer shares the root `go.mod` under `go/`. This separation means the operator and serving layer have independent dependency trees, reducing blast radius from dependency updates.

The operator's reconciliation model is noteworthy for its use of a "handler" pattern (`internal/controller/handler/`) that wraps the underlying service deployment logic. Each service type (offline, online, registry, UI) is handled through a unified `FeastHandler` that manages the full lifecycle — Deployment creation, Service exposure, PVC provisioning, and RBAC setup. The handler pattern also manages cross-FeatureStore references through `mapFeastRefsToFeastRequests`, which watches for changes to referenced FeatureStore CRs and triggers reconciliation of dependents. This enables a hub-and-spoke registry topology where multiple FeatureStores can share a single remote registry.

The TLS configuration is particularly well-designed for OpenShift environments. When the operator detects it's running on OpenShift, it automatically configures TLS via service serving certificates — no manual cert-manager setup required. The `setOpenshiftTls()` method creates appropriately-named TLS secrets and annotates ConfigMaps for CA injection. The registry service handles the complexity of dual-protocol exposure (gRPC + REST), using the gRPC service name as the primary certificate subject with the REST service as a SAN.

The Notebook integration controller (`notebook_configmap_controller.go`) demonstrates a thoughtful integration pattern with OpenDataHub. It conditionally starts only if the Notebook CRD exists in the cluster, uses unstructured API access (avoiding a hard dependency on Kubeflow types), and distributes feast client configurations via ConfigMaps with owner references. The label `opendatahub.io/feast-integration=true` on Notebooks enables opt-in integration, and the `opendatahub.io/feast-config` annotation specifies which FeatureStore project to connect to. This design allows feast to integrate with ODH/RHOAI Notebooks without requiring any changes to the Notebook controller itself.

A potential concern is that the Go feature serving gRPC endpoints have no authentication mechanism — they rely entirely on network-level controls. In contrast, the Python feature server supports OIDC and Kubernetes RBAC auth via FastAPI middleware. Organizations deploying the Go serving path should ensure network policies restrict access appropriately. Similarly, the Go transformation service client uses `insecure.NewCredentials()` for its gRPC connection to the transformation service, meaning transformation traffic is unencrypted.
