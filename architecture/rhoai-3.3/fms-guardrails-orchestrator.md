## Purpose
**Short**: A REST API orchestrator that coordinates AI text generation with safety guardrails by managing detectors, chunkers, and LLM backends.

**Detailed**: The FMS Guardrails Orchestrator (fms-guardrails-orchestr8) is a Rust-based middleware service that sits between clients and language models to enforce content safety. It orchestrates multiple backend services including text generation servers (TGIS or caikit-nlp via gRPC), content safety detector services (via HTTP), and text chunking services (via gRPC) to provide comprehensive guardrails for AI text generation.

The orchestrator exposes a unified HTTP REST API that supports multiple workflows: classification with text generation, standalone content detection, streaming text generation with guardrails, chat completions with detection (OpenAI-compatible), and context-aware document detection. It handles request routing, response aggregation, and safety filtering, blocking harmful content while enabling safe AI interactions.

The service supports TLS and mTLS for both its own server endpoints and for connections to downstream services. It integrates with OpenTelemetry for distributed tracing and metrics export. The orchestrator is built using the Axum web framework with Tokio async runtime, and uses Tonic for gRPC client connections to generation and chunker backends.

### TLS Configuration

The orchestrator supports three TLS modes for its server:
1. **No TLS**: Default, plaintext HTTP on port 8033
2. **TLS**: Server-side TLS with cert/key pair (env: `TLS_CERT_PATH`, `TLS_KEY_PATH`)
3. **mTLS**: Mutual TLS with additional client CA cert (env: `TLS_CLIENT_CA_CERT_PATH`)

For downstream client connections, TLS is configured per-service in the YAML config file, supporting:
- Named TLS configurations for reuse across services
- Per-service cert, key, and CA cert paths
- Insecure mode option for development
