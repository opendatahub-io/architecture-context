## Purpose

**Short**: A REST API orchestrator that coordinates AI text generation with safety guardrails by managing content safety detection, filtering, and generation across multiple backend services.

**Detailed**: The FMS Guardrails Orchestrator is a Rust-based middleware service that sits between clients and language models to enforce content safety. It orchestrates multiple detector services, chunker services, and LLM backends through unified REST APIs. The orchestrator receives user requests, optionally applies input detection (checking prompts for harmful content), forwards safe prompts to generation backends (TGIS, Caikit NLP, or OpenAI-compatible APIs), runs output detection on generated text, and returns filtered results to clients.

The service supports both unary (request-response) and streaming (SSE, NDJSON) patterns, enabling real-time content safety enforcement during streaming text generation. It implements a sophisticated detection pipeline that includes text chunking (splitting text into analyzable segments), parallel detection across multiple detectors, threshold-based filtering, and result aggregation. The orchestrator supports TLS and mutual TLS for all connections, OpenTelemetry for distributed tracing and metrics, and configurable header passthrough for integration with authentication systems.

Within the RHOAI platform, this component is deployed as a standalone service that integrates with TrustyAI's guardrails ecosystem. It acts as the central orchestration layer for the TrustyAI Guardrails feature, coordinating between model serving runtimes and content safety detectors.

## Architectural Analysis

The FMS Guardrails Orchestrator implements a classic middleware orchestration pattern adapted for AI content safety. Its architecture separates concerns cleanly into four layers: HTTP server (axum-based routing with TLS), orchestration handlers (task-specific pipeline logic), clients (typed abstractions over backend protocols), and a detection pipeline (chunking, parallel detection, threshold filtering, batching). This layered design allows each handler to compose detection and generation operations as building blocks without duplicating client management or TLS configuration.

The streaming architecture is particularly sophisticated. For streaming endpoints (SSE and NDJSON), the orchestrator uses a broadcast/fan-out pattern to distribute incoming generation chunks to multiple detection pipelines simultaneously. A `DetectionBatchStream` with pluggable batchers (`CompletionBatcher` for multi-choice OpenAI responses, `MaxProcessedIndexBatcher` for single-stream content) aggregates results from multiple detectors across multiple chunks, ensuring results are emitted in correct order even when detections arrive out-of-sequence. The actor pattern (`DetectionBatcherManager`) isolates mutable batching state behind async message-passing, avoiding lock contention. This is a complex but well-structured approach that enables real-time detection during streaming generation without buffering entire responses.

The TLS configuration is notable for its flexibility. Every outbound connection (detector, chunker, generation, OpenAI) can be independently configured with its own TLS certificates, client auth, and CA bundles via the YAML config file. Named TLS configs allow reuse across multiple services. The `insecure: true` option with a custom `NoVerifier` implementation is available as an escape hatch but is not the default. The server side supports TLS and mTLS with Rustls (ring backend), with graceful shutdown and 10-second connection drain timeout.

A notable design decision is the use of `ring` (BoringSSL-derived) as the sole cryptographic provider rather than OpenSSL bindings. This is set explicitly in `main.rs` line 30 (`rustls::crypto::ring::default_provider().install_default()`). While Ring provides strong cryptographic primitives, this means the Rust binary itself does not use the system OpenSSL that is installed in the container image — OpenSSL is present to satisfy check-payload requirements for the container image validation, but the application's TLS operations go through Ring. FIPS compliance for Ring requires building with its optional FIPS module, which is not explicitly configured in this repository.

The health check system is well-designed for a multi-backend orchestrator. Each client type implements health checks using the appropriate protocol — gRPC HealthCheck RPC for chunker/NLP/TGIS, HTTP GET /health for detector/OpenAI — with intelligent status mapping. The system tolerates `InvalidArgument` and `NotFound` gRPC errors as "healthy" responses (indicating the service is up but the probe's empty request is invalid), and provides a unified `/info` endpoint aggregating all backend health statuses. Optional startup health checks (`start_up_health_check` flag) allow fail-fast behavior when backends are unavailable.
