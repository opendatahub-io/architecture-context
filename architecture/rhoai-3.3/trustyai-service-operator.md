## Purpose

**Short**: Kubernetes operator that manages AI trustworthiness, fairness monitoring, guardrails orchestration, and language model evaluation services on OpenShift AI.

**Detailed**: The TrustyAI Service Operator is a multi-controller Kubernetes operator built with controller-runtime that manages four distinct CRDs: TrustyAIService, LMEvalJob, GuardrailsOrchestrator, and NemoGuardrails. Each CRD addresses a different aspect of AI/ML trustworthiness within the Red Hat OpenShift AI platform.

The TrustyAIService controller deploys and manages TrustyAI fairness and explainability monitoring services that integrate with KServe InferenceServices. It automatically patches InferenceServices to route model predictions through TrustyAI for bias detection and model monitoring. The LMEvalJob controller manages language model evaluation workloads using the lm-evaluation-harness framework, with Kueue-based batch scheduling, S3 offline storage, and OCI artifact upload capabilities.

The GuardrailsOrchestrator controller deploys a guardrails detection pipeline that auto-discovers InferenceServices to build routing configurations for content safety detectors, while the NemoGuardrails controller deploys NVIDIA NeMo Guardrails rule engines. All controllers use kube-rbac-proxy sidecars for OAuth-based authentication and create OpenShift Routes for external access. The operator is selectively enabled via `--enable-services` flags, allowing deployment of any subset of its four controllers.

## Architectural Analysis

The TrustyAI Service Operator is architecturally distinctive as a multi-CRD operator that bundles four logically separate controllers within a single binary. This design is driven by the `--enable-services` flag pattern (TAS, LMES, GORCH, NEMO_GUARDRAILS), where each controller registers itself via an `init()` function in its package-level Go file (e.g., `controllers/tas.go`, `controllers/lmes.go`). The registration system in `controllers/controllers.go` maps service names to setup functions, allowing selective activation. This mono-operator approach simplifies deployment but creates a large RBAC surface — the generated `manager-role` ClusterRole contains permissions for all four controllers regardless of which are enabled.

The controller implementations follow a consistent template-based resource creation pattern. Each controller embeds YAML templates via Go's `embed.FS` and renders them with runtime data using a generic `ParseResource[T]()` function. Templates for Deployments, Services, Routes, ServiceMonitors, and kube-rbac-proxy configurations live under each controller's `templates/` directory. This pattern ensures consistency across controllers but means the actual resource specifications are split between Go code (for dynamic logic like image selection, container injection, and conditional resources) and YAML templates (for static structure). Reviewers must examine both to understand what resources are created.

The authentication architecture uses a conditional kube-rbac-proxy pattern gated by the `security.opendatahub.io/enable-auth: "true"` annotation on CRs. When auth is required, each controller injects kube-rbac-proxy sidecar containers into managed Deployments, creates corresponding ConfigMaps with proxy configuration, and sets Route TLS termination to Reencrypt. The kube-rbac-proxy proxies authenticate requests via Kubernetes TokenReview and SubjectAccessReview APIs. This is notably different from the older oauth-proxy pattern — the operator is already aligned with the RHOAI 3.x auth model. The GORCH controller is most complex here, deploying up to three kube-rbac-proxy sidecars per Deployment (orchestrator, gateway, built-in detector), each with different upstream/downstream port mappings.

The LMEvalJob controller implements a state machine (New → Scheduled → Running → Complete/Cancelled/Suspended) with a polling-based status mechanism. Rather than using Kubernetes informers for pod status, the controller polls running pods by exec-ing the driver binary's `--get-status` flag at configurable intervals (default 10 seconds). This design trades API server load (exec calls) for simplicity — the driver binary runs a localhost-only HTTP server at port 18080 that serves status JSON, avoiding the need for a sidecar or shared volume for status communication. The input validation layer in `controllers/lmes/validation.go` is particularly thorough, with regex pattern matching, shell metacharacter blocking, path traversal prevention, and environment variable overrides that strip user-supplied values for security-sensitive variables like `TRUST_REMOTE_CODE`.

The GuardrailsOrchestrator's auto-configuration feature (via the `autoConfig` spec field) implements a service discovery pattern: it watches InferenceServices with a configurable label (default: `trustyai/guardrails=true`), extracts their serving endpoints and TLS certificates from ServingRuntime specs, and generates orchestrator YAML configuration as ConfigMaps. A SHA256 hash of the discovered resources is stored as a deployment annotation, triggering pod restarts only when the configuration actually changes. This is an effective reconciliation optimization that avoids unnecessary pod churn.
