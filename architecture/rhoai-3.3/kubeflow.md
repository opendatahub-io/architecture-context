## Purpose

**Short**: Manages the lifecycle of Jupyter notebook instances as Kubernetes-native workloads on Red Hat OpenShift AI.

**Detailed**: The kubeflow repository contains two cooperating Go-based Kubernetes controllers that together manage Jupyter notebook workloads on OpenShift. The **notebook-controller** (upstream Kubeflow) handles core lifecycle operations — creating StatefulSets, Services, and optionally Istio VirtualServices for each Notebook CR, plus idle notebook culling via Jupyter API polling. The **odh-notebook-controller** (OpenShift-specific) extends this with enterprise features: mutating webhook for sidecar injection (kube-rbac-proxy authentication), Gateway API HTTPRoute creation for ingress, NetworkPolicy enforcement, cross-namespace ReferenceGrant management, OpenShift ImageStream integration for container image resolution, Data Science Pipelines (DSPA) integration for Elyra runtime config, Feast feature store config injection, and cluster-wide proxy environment variable propagation.

Both controllers watch the same `Notebook` CRD (`kubeflow.org`) but handle different aspects of the notebook lifecycle. They are deployed together via a composite kustomize overlay and run in the same namespace. The upstream controller manages the core compute (StatefulSet, Service), while the ODH controller manages security, networking, and platform integration through a mutating webhook and its own reconciliation loop.

## Architectural Analysis

The kubeflow repository implements a dual-controller architecture where the upstream Kubeflow notebook-controller provides base lifecycle management and the ODH notebook-controller extends it with OpenShift-specific features. This separation allows the RHOAI team to maintain compatibility with upstream Kubeflow while adding enterprise features. Both controllers watch the same `Notebook` CRD but partition responsibility cleanly: the KF controller manages StatefulSets, Services, and culling, while the ODH controller handles authentication, network isolation, and platform integration.

The RHOAI 3.x ingress pattern is fully implemented in this repository. The ODH controller creates HTTPRoutes in a **central namespace** (the controller's namespace) rather than the user's namespace, using cross-namespace backend references to reach notebook Services. This design requires Gateway API `ReferenceGrant` resources in each user namespace to allow the central HTTPRoutes to reference user-namespace Services. The ReferenceGrants are shared — one per namespace, managed with a finalizer pattern since cross-namespace OwnerReferences are not supported. When the last notebook in a namespace is deleted, the controller removes the ReferenceGrant. The Gateway API integration uses configurable Gateway name (`NOTEBOOK_GATEWAY_NAME`, default `data-science-gateway`) and namespace (`NOTEBOOK_GATEWAY_NAMESPACE`, default `openshift-ingress`).

Authentication is handled through kube-rbac-proxy sidecar injection, controlled by the `notebooks.opendatahub.io/inject-auth` annotation. When enabled, the mutating webhook injects a kube-rbac-proxy container that listens on port 8443, fronting the notebook's port 8888. The proxy performs TokenReview and SubjectAccessReview against the Kubernetes API to enforce that the requesting user has `get` permission on the specific `notebooks` resource. This creates a per-notebook authentication boundary. The controller also creates a dedicated ClusterRoleBinding for each notebook's ServiceAccount to `system:auth-delegator`, enabling the sidecar to delegate authentication. Legacy OAuth support (RHOAI 2.25 OAuthClient pattern) is retained only for cleanup — the controller removes OAuthClient resources through a finalizer during notebook deletion.

The webhook mutation pipeline is notably comprehensive, performing nine distinct mutations on notebook CREATE/UPDATE: reconciliation lock injection (preventing race conditions during initial setup), container image resolution from ImageStreams, CA certificate bundle mounting, pipeline runtime images ConfigMap syncing, Elyra DSPA secret syncing, Feast feature store config mounting, kube-rbac-proxy sidecar injection, cluster proxy environment variable injection, and running-notebook update protection. The update protection logic is particularly notable — it detects when webhook mutations would modify the pod template of a running notebook and instead sets an `update-pending` annotation with a diff reason, deferring the actual restart to the user. This prevents unexpected pod disruptions from webhook-driven changes.

Network isolation follows the principle of least privilege: each notebook gets two NetworkPolicies. The first (`{name}-ctrl-np`) allows ingress on port 8888 only from the controller's namespace. The second (`{name}-kube-rbac-proxy-np`) allows ingress on port 8443 from any source, since external traffic through the Gateway needs to reach the kube-rbac-proxy sidecar. This ensures that unauthenticated access to port 8888 is limited to the controller namespace (for culling health checks), while authenticated access through the proxy is open.
