NETWORK POLICIES:

1. {cluster}-head (Ray Head Pods)
   - Target: Pods with ray.io/cluster={cluster}, ray.io/node-type=head
   - Ingress Rules:
     * Allow from same cluster pods (all ports)
     * Allow from namespace pods on 10001/TCP, 8265/TCP
     * Allow from KubeRay operator (opendatahub/redhat-ods-applications) on 8265/TCP, 10001/TCP
     * Allow from openshift-monitoring on 8080/TCP
     * Allow all on 8443/TCP (OAuth), 10001/TCP (Ray client with mTLS)

2. {cluster}-workers (Ray Worker Pods)
   - Target: Pods with ray.io/cluster={cluster}, ray.io/node-type=worker
   - Ingress Rules:
     * Allow only from same cluster pods (all ports)

SECRETS:

1. codeflare-operator-webhook-server-cert (kubernetes.io/tls)
   - Purpose: TLS certificate for webhook server
   - Provisioned by: cert-controller (embedded)
   - Rotation: Automatic (cert-controller manages lifecycle)
   - Validity: Managed by cert-controller

2. {cluster}-oauth-config (Opaque)
   - Purpose: OAuth proxy cookie secret for Ray dashboard
   - Provisioned by: RayCluster controller
   - Rotation: Manual (no auto-rotation)
   - Contains: SESSION_SECRET for OAuth proxy

3. {cluster}-proxy-tls-secret (kubernetes.io/tls)
   - Purpose: Service serving certificate for OAuth proxy
   - Provisioned by: OpenShift service-ca (service.beta.openshift.io/serving-cert-secret-name annotation)
   - Rotation: Automatic (30 days before expiry)
   - Validity: 26 months

4. ca-secret-{cluster} (Opaque)
   - Purpose: CA certificate and private key for Ray client mTLS
   - Provisioned by: RayCluster controller
   - Rotation: Manual (no auto-rotation)
   - Validity: 1 year
   - Contains: ca.crt, ca.key

SERVICE MESH CONFIGURATION:

- Not integrated with Istio/Service Mesh by default
- OAuth proxy provides authentication layer for Ray dashboard
- mTLS for Ray client connections managed by CA certificates (not service mesh)
- Network isolation via NetworkPolicies

ENCRYPTION:

- External ingress: TLS 1.3 (Router/Ingress Controller)
- Re-encrypt to OAuth proxy: TLS 1.2+ (service serving cert)
- OAuth proxy to Ray dashboard: HTTP (localhost, no encryption needed)
- Ray client API: mTLS with cluster-specific CA (optional, configurable)
- Kubernetes API: TLS 1.2+ (all operator-API communication)
- Webhook server: TLS 1.2+ (API server to webhook)
- Metrics endpoint: HTTP (no encryption, internal only)
