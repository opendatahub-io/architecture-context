TRUST BOUNDARIES
═══════════════════════════════════════════════════════════════════════════════════════════

1. EXTERNAL → INGRESS
   - Encryption: TLS 1.2+ (HTTPS for API, optional for inference)
   - Auth: kubeconfig credentials (admin), optional for inference (Istio, external auth)
   - Trust: Zero trust - validate all inputs

2. INGRESS → CONTROL PLANE
   - Encryption: TLS 1.2+ (webhook), TLS (metrics via kube-rbac-proxy)
   - Auth: Webhook TLS cert, Bearer token (metrics)
   - Trust: Kubernetes API server is trusted; webhook validates before admission

3. CONTROL PLANE → DATA PLANE
   - Encryption: Service Account token over TLS (to K8s API)
   - Auth: Service Account RBAC
   - Trust: Controller is trusted to manage resources in user namespaces

4. DATA PLANE (Internal Pod Communication)
   - Encryption: None (localhost communication)
   - Auth: None (same pod)
   - Trust: Containers in same pod trust each other

5. DATA PLANE → STATE MANAGEMENT (etcd)
   - Encryption: None (HTTP)
   - Auth: None (network isolation via NetworkPolicy)
   - Trust: etcd is internal, isolated by network policies

6. DATA PLANE → EXTERNAL SERVICES (S3)
   - Encryption: TLS 1.2+ (HTTPS)
   - Auth: AWS Signature v4 / S3 API keys
   - Trust: S3 is external - use encrypted connections and IAM/bucket policies

7. OPTIONAL: SERVICE MESH (Istio)
   - Encryption: mTLS STRICT (all service-to-service)
   - Auth: Service Identity via mTLS certificates
   - Trust: Istio enforces zero-trust networking within mesh
