SERVICE MESH CONFIGURATION

Istio mTLS Configuration:
- Mode: ISTIO_MUTUAL (default for Model Registry), STRICT for KServe predictor pods
- PeerAuthentication:
  * kserve-predictor (namespace-scoped): STRICT mTLS for all KServe pods in service mesh
  * model-registry (namespace-scoped): STRICT mTLS for Model Registry services
- Certificate Authority: Istio CA (auto-rotation)
- Certificate TTL: 24 hours (default Istio rotation)

Istio Gateway Configuration:
- KServe InferenceServices:
  * Gateway: {namespace}/{inferenceservice-name}
  * VirtualService: Routes traffic to Knative Service
  * Port: 443/TCP (external) → 8012/TCP (Knative Activator)
  * TLS Mode: TLS 1.3
  * Hosts: {isvc}.{namespace}.{cluster-domain}
- Model Registry:
  * Gateway: {modelregistry-name}-rest, {modelregistry-name}-grpc
  * Ports: 443/TCP (REST), 443/TCP (gRPC)
  * TLS Certificates: Provided via Secrets (kubernetes.io/tls)

Istio AuthorizationPolicy:
- KServe InferenceServices (when auth enabled):
  * Action: CUSTOM (delegates to Authorino)
  * Provider: authorino-external-auth
  * Authorino validates:
    - Token via Kubernetes TokenReview API
    - User permission via SubjectAccessReview (GET on InferenceService resource)
- Model Registry:
  * Similar CUSTOM action pattern
  * SubjectAccessReview checks GET permission on Service resource

Ray mTLS (Custom, not Istio):
- CodeFlare operator generates per-cluster CA certificate
- Secrets: ca-secret-{cluster} (CA cert + key)
- Client certificates required for Ray client connections
- Ray dashboard protected by OpenShift OAuth proxy (separate from mTLS)

COMPLIANCE & SECURITY STANDARDS
═══════════════════════════════════════════════════════════════════════════════════════════

FIPS Compliance:
- All RHOAI components built with FIPS-compliant crypto libraries
- UBI8/UBI9 base images with strictfipsruntime tags
- CGO-enabled builds for Go components

Pod Security Standards:
- Non-root users for all containers (except certain init containers)
- No privilege escalation (allowPrivilegeEscalation: false)
- Capabilities dropped (except NET_BIND_SERVICE for specific cases)
- Read-only root filesystems where possible

Image Provenance:
- All images from registry.redhat.io (signed and scanned)
- Konflux CI/CD for build traceability
- Git commit-based tags for versioning

Encryption in Transit:
- TLS 1.2+ for all external HTTPS endpoints
- TLS 1.3 for Istio Gateway
- mTLS for service-to-service communication in mesh
- Optional TLS for database connections

Encryption at Rest:
- OpenShift encrypted etcd (cluster secrets)
- Optional encrypted PVCs (depends on storage class)
- Database encryption (configured per database)

Audit Logging:
- Kubernetes API audit logs (all API calls)
- OpenShift OAuth audit logs (user authentication)
- Application logs for component operations
