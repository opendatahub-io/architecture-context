═══════════════════════════════════════════════════════════════════════════
AUTHENTICATION & AUTHORIZATION FLOWS
═══════════════════════════════════════════════════════════════════════════

External Access (/* via Route):
1. User → Route (443/TCP HTTPS TLS 1.2+)
2. Route → Service (8443/TCP HTTPS reencrypt)
3. Service → oauth-proxy (8443/TCP HTTPS)
4. oauth-proxy validates Bearer Token OR redirects to OAuth login
5. OAuth server validates user credentials
6. User must have permission to list projects (enforced by oauth-proxy)
7. oauth-proxy forwards to backend with x-forwarded-access-token header
8. Backend validates token and uses ServiceAccount for K8s API calls

Internal Access (backend → K8s API):
1. Backend extracts user token from x-forwarded-access-token header
2. Backend uses odh-dashboard ServiceAccount token for API calls
3. K8s API validates ServiceAccount token against RBAC policies
4. RBAC policies (ClusterRole + Role) authorize resource access

Metrics Endpoint (/metrics):
- Bypasses authentication (skip-auth-regex in oauth-proxy)
- Public within cluster network
- Intended for Prometheus scraping

Health Endpoints:
- /api/health (backend): No auth, public within pod network
- /oauth/healthz (oauth-proxy): No auth, public within cluster

NETWORK POLICIES
═══════════════════════════════════════════════════════════════════════════

No NetworkPolicy resources defined in manifests.
Security relies on:
- OpenShift RBAC (ClusterRole + Role + RoleBindings)
- OAuth authentication via oauth-proxy sidecar
- TLS encryption for all external traffic
- Service mesh not configured (no Istio policies)
