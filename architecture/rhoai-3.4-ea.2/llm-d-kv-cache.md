## Purpose

**Short**: A pluggable KV-cache indexing and scoring library for KV-cache-aware routing in distributed vLLM inference platforms.

**Detailed**: The llm-d-kv-cache component provides the core KV-cache management capabilities for the llm-d distributed LLM inference platform. It maintains a global, near-real-time view of KV-cache block locality across a fleet of vLLM pods, enabling intelligent routing decisions that maximize cache reuse. This improves Time To First Token (TTFT) and overall throughput by avoiding redundant KV-cache recomputation.

The system operates through two primary data flows: a **Read Path** where schedulers/routers query the indexer to score pods for a given prompt based on cached block locality (using a longest-prefix-match scoring algorithm), and a **Write Path** where KV-cache events streamed from vLLM pods via ZeroMQ are ingested and processed to keep the block index up to date. The architecture separates concerns into modular components: a tokenization subsystem (supporting embedded HuggingFace tokenizers, local file tokenizers, and an external UDS gRPC tokenizer service), a token-to-block-key processor using FNV-64a hashing over CBOR encoding (compatible with vLLM's prefix caching), swappable index backends (in-memory LRU, cost-aware Ristretto, Redis, Valkey), and a configurable scoring engine.

Additionally, the repository contains supporting components: a **PVC Evictor** (Python multiprocess daemon for managing KV-cache storage on persistent volumes) and a **shared storage filesystem backend** (C++/CUDA extension for vLLM's KV offloading interface). The primary Konflux-shipped component is the UDS tokenizer gRPC service, deployed as a sidecar container alongside the inference scheduler.

## Architectural Analysis

The llm-d-kv-cache architecture is designed around a clean separation of concerns with pluggable backends at every layer. The core `kvcache.Indexer` orchestrates four sub-systems — tokenization, token-to-block-key processing, index lookup, and scoring — each with well-defined interfaces and swappable implementations. This is exemplified by the `kvblock.Index` interface which supports four backends (in-memory LRU, cost-aware Ristretto, Redis, Valkey) and the `tokenization.Tokenizer` interface with three implementations (embedded HuggingFace, local filesystem, UDS gRPC). The `CompositeTokenizer` provides ordered fallback across backends, enabling graceful degradation in air-gapped or restricted environments.

The event ingestion pipeline uses an interesting sharded worker pool pattern where pod identifiers are hashed via FNV-1a to select worker shards, guaranteeing that events from the same pod are always processed in order. This is critical for maintaining index consistency, as out-of-order event processing could cause stale block entries. The ZMQ PUB/SUB transport with topic filtering (`kv@<pod-id>@<model-name>`) provides efficient event routing without a message broker.

The scoring algorithm implements a "longest consecutive prefix match" strategy that mirrors vLLM's prefix caching semantics. The `Lookup` method uses early termination — breaking on the first cache miss — because prefix caching requires all preceding blocks to be present. The scoring weights can be configured per device tier (GPU=1.0, CPU=0.8 by default), enabling multi-tier storage awareness. This design makes sense for vLLM's block manager which can spill KV blocks from GPU to CPU memory.

A notable architecture pattern is the pervasive use of the Decorator pattern for cross-cutting concerns. Both the index and scorer are wrapped with tracing decorators (`TracedIndex`, `TracedScorer`) and the index additionally with an instrumented decorator (`InstrumentedIndex`) for Prometheus metrics. This layered wrapping is composed at initialization time in `NewKVCacheIndexer`, keeping the core implementations clean. The OpenTelemetry spans (`llm_d.kv_cache.get_scores`, `llm_d.kv_cache.index`, `llm_d.kv_cache.scorer.compute`) create a complete trace tree for debugging scoring latency.

The token-to-block-key conversion uses FNV-64a hashing over CBOR canonical encoding to ensure deterministic hash chains compatible with vLLM's Python implementation. The `HashSeed` must be aligned with vLLM's `PYTHONHASHSEED` environment variable — a subtle but critical configuration requirement that, if misconfigured, silently produces zero cache hits. The `extra` parameter in the hash tuple enables cache differentiation for LoRA adapters and multi-modal inputs.

The PVC Evictor component uses an unusual N+2 multiprocess architecture (crawlers + activator + deleter) with hysteresis-based activation (start deletion at 85% disk usage, stop at 70%). The hex-modulo-based file distribution across crawler processes is an elegant solution for parallelizing directory traversal without file-level locking. The use of `xargs -0 rm -f` for batch deletion is pragmatic but carries a subprocess timeout risk documented at 60 seconds per batch.

The Konflux build pipeline only ships one container image (the UDS tokenizer service) from this repository. The main Go library is consumed as a Go module dependency by other components (e.g., llm-d-inference-scheduler). This means the Go code in this repository does not go through Konflux's check-payload FIPS validation — it relies on the consuming component's build pipeline for FIPS compliance. The ZMQ event transport is entirely unencrypted, relying on network-level isolation (typically within a Kubernetes cluster) for security.
