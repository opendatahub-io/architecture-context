## Purpose

**Short**: Kubernetes operator that manages the lifecycle of LlamaStack server distributions, providing declarative deployment and management of LlamaStack AI inference servers on OpenShift/Kubernetes.

**Detailed**: The LlamaStack K8s Operator (also referred to as `llama-stack-k8s-operator` or `ogx-k8s-operator` in the downstream RHOAI distribution) is a controller-runtime based Kubernetes operator that reconciles `LlamaStackDistribution` custom resources. It automates the deployment, configuration, scaling, and lifecycle management of LlamaStack server instances — a Meta-developed framework for building and running AI inference and agent applications.

The operator watches `LlamaStackDistribution` CRs and creates a full set of Kubernetes resources for each instance: Deployments (running LlamaStack distribution container images), Services (ClusterIP for internal access), ServiceAccounts, RoleBindings (for OpenShift SCC `anyuid`), PersistentVolumeClaims (optional persistent storage), NetworkPolicies (namespace-scoped access controls with feature flag toggle), Ingresses (optional external access), PodDisruptionBudgets (voluntary disruption tolerance), and HorizontalPodAutoscalers (CPU/memory-based autoscaling). The operator uses embedded kustomize manifests at runtime, applying Go-based transformer plugins (name prefixing, namespace injection, field mutation, NetworkPolicy rule generation) to customize resources per CR instance.

The operator supports multiple LlamaStack distributions (starter, remote-vllm, meta-reference-gpu, postgres-demo) via an embedded `distributions.json` file. For RHOAI deployments, the distribution image is overridden via the `RELATED_IMAGE_RH_DISTRIBUTION` environment variable. It also provides sophisticated CA bundle management — automatically detecting the platform's `odh-trusted-ca-bundle` ConfigMap and merging it with user-provided CA certificates into a managed ConfigMap, with PEM validation and X.509 certificate parsing for security. The operator probes the deployed LlamaStack server's `/v1/providers` and `/v1/version` HTTP endpoints to populate status with provider health and version information.

## Architectural Analysis

The LlamaStack K8s Operator follows a clean kubebuilder/controller-runtime pattern with a single reconciler managing one CRD. The operator embeds its runtime manifests (in `controllers/manifests/base/`) into the binary via the `Dockerfile.konflux` `COPY --from=builder /workspace/controllers/manifests ./manifests/` instruction, then uses kustomize's Go API at runtime to render and transform them per-instance. This is a notably different pattern from most RHOAI operators — rather than maintaining manifests externally and consuming them via `get_all_manifests.sh`, this operator carries its deployment templates internally and applies Go-based kustomize transformer plugins (namespace setter, name prefixer, field mutator, NetworkPolicy transformer) to customize resources. This gives the operator fine-grained control over resource generation but couples manifest evolution to operator releases.

The operator's ConfigMap watching mechanism is sophisticated: it creates field indexers for efficient lookups on `spec.server.userConfig.configMapName` and `spec.server.tlsConfig.caBundle.configMapName`, with a manual search fallback for older Kubernetes versions that don't support custom field labels on CRDs. When any referenced ConfigMap changes, the operator recalculates content hashes and updates pod template annotations, triggering a rolling restart. The CA bundle management is particularly well-engineered — it gathers certificates from both user-specified ConfigMaps and the platform's `odh-trusted-ca-bundle`, validates each PEM block as a proper X.509 certificate (rejecting non-CERTIFICATE blocks like private keys), enforces size limits (10MB total, 1000 certificates max), and writes a single managed ConfigMap with owner references for proper garbage collection.

The NetworkPolicy implementation uses a feature flag system (configurable via the `llama-stack-operator-config` ConfigMap) and a custom kustomize transformer plugin. When enabled, the NetworkPolicy restricts ingress to the LlamaStack service port from: (1) all pods in the same namespace, (2) the operator namespace, (3) explicitly configured namespaces/labels from `spec.network.allowedFrom`, and (4) OpenShift ingress router namespaces (labeled `network.openshift.io/policy-group=ingress`). This provides a defense-in-depth approach to network isolation while supporting the Ingress-based external access path.

The distribution image resolution has a three-tier priority: (1) image mapping overrides from the operator config ConfigMap's `image-overrides` key, (2) the `RELATED_IMAGE_RH_DISTRIBUTION` environment variable (which overrides all embedded distributions with a single "rh-dev" entry for RHOAI), and (3) the embedded `distributions.json` file. This allows the RHOAI platform operator to pin distribution images without modifying the operator binary, while still supporting upstream community distributions for ODH. The operator validates image references using `go-containerregistry/pkg/name` to prevent injection of malformed image strings.

The operator performs upgrade cleanup at startup — specifically, it scans for and deletes legacy ClusterRoleBindings from previous versions that were created with cluster-scoped bindings but should have been namespace-scoped RoleBindings. This handles the migration from cluster-scoped to namespace-scoped RBAC that occurred in a previous release, ensuring seamless upgrades.
