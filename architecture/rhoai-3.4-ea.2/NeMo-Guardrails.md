## Purpose

**Short**: NeMo Guardrails is an NVIDIA-developed toolkit that adds programmable safety guardrails to LLM-based conversational systems, deployed as a FastAPI server in RHOAI.

**Detailed**: NeMo Guardrails provides a framework for controlling and securing interactions with Large Language Models. It acts as a middleware layer between client applications and LLM inference endpoints, applying configurable "rails" that enforce input validation, output filtering, topic control, jailbreak detection, sensitive data detection, hallucination checking, and content safety policies. The system uses a domain-specific language called Colang (versions 1.0 and 2.x) to define conversation flows and guardrail rules.

In the RHOAI deployment, NeMo Guardrails runs as a standalone FastAPI HTTP server (port 8000) that exposes an OpenAI-compatible chat completions API with guardrails extensions. It proxies LLM requests through configurable guardrail pipelines before forwarding to upstream model servers (OpenAI, NVIDIA NIM, vLLM, Azure OpenAI, Anthropic, Cohere, and others). The server pre-downloads embedding models (sentence-transformers, SpaCy, FastEmbed, NLTK) at build time for local inference used by guardrail evaluations. The RHOAI build uses a Konflux pipeline targeting three architectures (x86_64, arm64, ppc64le) with multi-stage Docker builds for platform-specific compilation of PyTorch, tiktoken, onnxruntime, and OpenBLAS.

The guardrails library includes both open-source and closed-source modules. The RHOAI build uses the "opensource" profile, which excludes proprietary guardrails (ActiveFence, Cleanlab, Clavata, PrivateAI, Fiddler, PatronusAI, Prompt Security, GCP Moderate Text) while retaining open-source modules for content safety, jailbreak detection, sensitive data detection, hallucination detection, injection detection, topic safety, self-check, and regex-based filtering.

## Architectural Analysis

NeMo Guardrails follows a pipeline architecture where each incoming LLM request passes through configurable "rails" — ordered stages that can inspect, modify, or block messages. The system defines four rail categories: input rails (validate user messages), output rails (validate LLM responses), dialog rails (control conversation flow), and retrieval rails (filter RAG context). Each rail type maps to a Colang flow that orchestrates one or more "actions" — Python callables that perform the actual guardrail evaluation. This architecture cleanly separates policy definition (Colang) from implementation (Python actions), allowing security teams to configure guardrail behavior without modifying code.

The server exposes two primary API patterns. The `/v1/chat/completions` endpoint is a full proxy: it receives an OpenAI-compatible request, applies input rails, forwards to the upstream LLM, applies output rails, and returns the (potentially modified or blocked) response. The `/v1/guardrail/checks` endpoint is a validation-only path: it evaluates messages against configured rails without making LLM calls, returning a structured report of which rails passed or blocked. This dual-mode design supports both transparent proxying (where guardrails are invisible to the client) and explicit validation (where clients check content before/after their own LLM calls).

The RHOAI build is notably complex due to multi-architecture support. The `Dockerfile.konflux` has 8 stages that conditionally build native dependencies for ppc64le — compiling PyTorch, tiktoken (Rust-based), ONNX Runtime, and OpenBLAS from source. For x86_64 and arm64, pre-built wheels from PyPI are used. The build also runs `filter_guardrails.py` with the "opensource" profile to strip closed-source guardrail modules at build time, ensuring the RHOAI image contains only Apache-2.0-licensed guardrail implementations. The SpaCy `en_core_web_lg` model is the only artifact locked via Hermeto (`artifacts.lock.yaml`); all other dependencies are fetched live from PyPI, creating a hermeticity gap.

A noteworthy design choice is the server's lack of built-in authentication or TLS. The server binds to `0.0.0.0:8000` with plaintext HTTP and no auth middleware. This is intentional — in RHOAI 3.x, authentication is handled by a `kube-rbac-proxy` sidecar injected by the platform operator, and TLS is terminated at the Gateway API layer. However, this means the application has no defense-in-depth for authentication: if the sidecar is misconfigured or bypassed, the guardrails server is fully exposed. The `hashlib.md5` usage in the embedding cache is a minor FIPS concern — the code includes a fallback to SHA-256 when MD5 is unavailable (FIPS mode), demonstrating awareness of the constraint.

The Colang language engine (33 Python files across v1.0 and v2.x implementations) is a significant architectural investment. It provides a declarative way to define conversation flows with `define flow`, `define subflow`, `when user said`, and `execute action` primitives. The v2.x implementation adds support for concurrent flows, event-driven programming, and a more expressive type system. This DSL approach is unusual in the guardrails space and provides a differentiated capability for defining complex multi-turn guardrail policies.
