## Purpose

**Short**: A batch-oriented container image for evaluating language models against standardized benchmarks, integrated with the EvalHub platform via the eval-hub-sdk adapter.

**Detailed**: lm-evaluation-harness is a fork of EleutherAI's [lm-evaluation-harness](https://github.com/EleutherAI/lm-evaluation-harness) (v0.4.8), customized for Red Hat OpenShift AI (RHOAI). It provides a containerized runtime (`odh-ta-lmes-job-rhel9`) that executes language model evaluations as one-shot Kubernetes batch jobs. The harness supports a wide variety of NLP benchmarks (MMLU, HellaSwag, ARC, TruthfulQA, BBH, GPQA, IFEval, and hundreds more) and model backends (HuggingFace Transformers, vLLM, OpenAI-compatible APIs, IBM watsonx.ai, Anthropic, etc.).

In the RHOAI context, the harness is orchestrated by the TrustyAI LMEval operator (part of the `trustyai-service-operator`). The operator creates Kubernetes Jobs using this container image. The `main.py` adapter (`LMEvalAdapter`) implements the `FrameworkAdapter` interface from `eval-hub-sdk`, which loads a `JobSpec` (mounted as a ConfigMap at `/meta/job.json`), connects to the target model via OpenAI-compatible completions API, runs the specified benchmark, reports status/results back to the EvalHub sidecar via callbacks, and optionally persists evaluation artifacts as OCI images to a registry using `olot` and `skopeo`. The adapter also supports MLflow integration for logging metrics and parameters.

The image is built for multi-architecture support (amd64, s390x, ppc64le) with custom build stages for PyTorch and Apache Arrow on non-x86 platforms. It runs as a non-root user (UID 65532) and includes the Unitxt framework for flexible, customizable evaluation tasks.

## Architectural Analysis

The lm-evaluation-harness follows a **plugin-based architecture** with three key extension points: model backends (`lm_eval/models/`), evaluation tasks (`lm_eval/tasks/`), and metrics (`lm_eval/api/metrics.py`). Each model backend registers itself via the `@register_model` decorator (e.g., `@register_model("local-completions")` for the OpenAI-compatible backend), and tasks are defined via YAML configuration files with a `ConfigurableTask` base class that handles dataset loading, prompt construction, few-shot example sampling, and result aggregation. This design allows the RHOAI adaptation to operate primarily as an integration layer (`main.py`) without modifying the core evaluation engine.

The **RHOAI adapter layer** (`main.py`) wraps the upstream `simple_evaluate()` function with the `FrameworkAdapter` pattern from `eval-hub-sdk`. This adapter translates between the EvalHub job specification model (`JobSpec` with model URL, benchmark ID, parameters) and the lm-eval native API (model backend name, model_args dict, task names). The adapter always uses the `local-completions` backend, meaning models are evaluated via HTTP API calls rather than local inference — this is architecturally significant because it decouples the evaluation runtime from GPU requirements, allowing the evaluation job to run on CPU-only nodes while the model runs on a separate GPU-equipped serving pod.

The **multi-architecture build** is notably complex. The Konflux Dockerfile (`Dockerfile.konflux.lmes-job`) has five build stages to handle PyTorch and Apache Arrow compilation from source on s390x and ppc64le architectures. On x86_64, pre-built wheels are used. The build uses pinned `requirements.txt` with exact versions generated from `poetry.lock`, but the Dockerfile itself installs system packages via `dnf` without version locks, creating a hermeticity gap. The `patches/s390x/parquet-support.patch` applies a custom fix for Arrow/Parquet support on s390x.

The **security model** relies on the orchestrating operator (TrustyAI) to provision all credentials. Model authentication tokens, CA certificates, OCI registry credentials, and EvalHub auth tokens are all mounted into the container at runtime. The adapter includes a security-conscious path validation for CA certificate files that explicitly rejects symlinks (`api_models.py:134-138`), resolves to real paths, and validates the file exists before use. However, TLS verification can be disabled at the operator level via `verify_certificate=False` or `evalhub_insecure` settings, which is appropriate for development but should be audited in production deployments.

The **evaluation output pipeline** produces three outputs: (1) status updates reported via callbacks to the EvalHub sidecar during execution, (2) a final results JSON with per-metric scores written to the local filesystem and reported to EvalHub, and (3) optionally an OCI artifact containing the full evaluation output pushed to a registry. The OCI artifact creation uses `olot` (OCI Layout Tool) to package results into an OCI-compliant layout, then `skopeo` to push to the target registry. This provides an immutable, content-addressed record of evaluation results with OCI metadata including benchmark name, model name, job ID, and creation timestamp.
