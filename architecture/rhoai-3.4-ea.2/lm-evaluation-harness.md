## Purpose
**Short**: A batch-oriented container image for executing language model evaluations using the EleutherAI lm-evaluation-harness framework, integrated with the EvalHub platform.

**Detailed**: The LM Evaluation Harness (LMES Job) is a containerized execution environment for running standardized benchmarks against language models. It is a Red Hat downstream fork of the EleutherAI lm-evaluation-harness project, customized to operate within the RHOAI TrustyAI evaluation pipeline.

The component is packaged as a container image (`odh-ta-lmes-job-rhel9`) that runs as a one-shot Kubernetes Job. It receives evaluation job specifications via a mounted ConfigMap (`/meta/job.json`), connects to model inference endpoints using the OpenAI-compatible API protocol, executes benchmarks (accuracy, BLEU, ROUGE, F1, etc.), and reports results back to the EvalHub service via callback URLs. Evaluation artifacts can optionally be persisted to OCI-compliant registries using skopeo.

The adapter architecture uses the `eval-hub-sdk` to integrate with the EvalHub platform, providing lifecycle callbacks (status reporting, result submission, OCI artifact creation) and MLflow experiment tracking. It supports 100+ benchmark tasks (MMLU, GSM8K, HellaSwag, TruthfulQA, ARC, etc.) and 50+ metrics, with the Unitxt framework for custom task definitions.

## Data Flows

### Flow 1: Evaluation Job Execution

| Step | Source | Destination | Port | Protocol | Encryption | Auth |
|------|--------|-------------|------|----------|------------|------|
| 1 | LMEval Operator | Kubernetes API | 6443/TCP | HTTPS | TLS 1.3 | ServiceAccount Token |
| 2 | LMES Job Pod | EvalHub Service | configurable | HTTP/HTTPS | TLS 1.2+ | Bearer Token |
| 3 | LMES Job Pod | HuggingFace Hub | 443/TCP | HTTPS | TLS 1.2+ | HF Token (optional) |
| 4 | LMES Job Pod | Model Inference Endpoint | configurable | HTTP/HTTPS | TLS 1.2+ | Bearer Token (OPENAI_API_KEY) |
| 5 | LMES Job Pod | EvalHub Service | configurable | HTTP/HTTPS | TLS 1.2+ | Bearer Token |
| 6 | LMES Job Pod | OCI Registry | 443/TCP | HTTPS | TLS 1.2+ | Registry credentials |
| 7 | LMES Job Pod | MLflow Tracking | configurable | HTTP/HTTPS | TLS 1.2+ | configurable |

**Flow description**:
1. TrustyAI operator creates a Kubernetes Job with the LMES container image, mounting the job spec ConfigMap at `/meta/job.json`
2. LMES adapter reports INITIALIZING status to EvalHub via callback URL
3. LMES downloads tokenizer and benchmark datasets from HuggingFace Hub
4. LMES sends evaluation prompts to the model inference endpoint (OpenAI-compatible completions API) and collects responses
5. LMES reports evaluation results (metrics, scores) to EvalHub via callback URL
6. LMES packages results as OCI artifacts and pushes to the configured registry (when OCI exports configured)
7. LMES logs metrics and parameters to MLflow for experiment tracking (when configured)

### Flow 2: OCI Artifact Persistence

| Step | Source | Destination | Port | Protocol | Encryption | Auth |
|------|--------|-------------|------|----------|------------|------|
| 1 | LMES Job Pod | Local filesystem | N/A | N/A | N/A | N/A |
| 2 | LMES Job Pod (olot) | Local /tmp/oci-layout | N/A | N/A | N/A | N/A |
| 3 | LMES Job Pod (skopeo) | OCI Registry | 443/TCP | HTTPS | TLS 1.2+ | Registry credentials |

**Flow description**:
1. Results are saved to `/opt/app-root/src/output/` as JSON files
2. The `olot` library packages output files into an OCI-compliant layout at `/tmp/oci-layout`
3. `skopeo` transfers the OCI layout to the destination registry
