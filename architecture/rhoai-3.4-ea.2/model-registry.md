## Purpose

**Short**: Central metadata repository for ML models, versions, artifacts, and serving environments with an OpenAPI REST interface, Kubernetes controller integration, and model catalog discovery.

**Detailed**: Model Registry is a multi-component system that provides a central metadata store for machine learning workflows on Red Hat OpenShift AI. The core REST API server (Go, chi router) exposes a v1alpha3 OpenAPI interface for managing RegisteredModels, ModelVersions, Artifacts, Experiments, ExperimentRuns, InferenceServices, and ServingEnvironments. It persists metadata in either MySQL or PostgreSQL via GORM, using an embedded metadata (EmbedMD) datastore architecture.

The system includes a Kubernetes controller that watches KServe InferenceService CRDs and automatically synchronizes deployment state back to the Model Registry, maintaining a bidirectional link between cluster inference endpoints and registered model metadata. A Backend-for-Frontend (BFF) Go service proxies UI requests with Kubernetes RBAC enforcement, service discovery, and model transfer job orchestration. A Catalog service provides curated model discovery from HuggingFace, YAML sources, and MCP (Model Context Protocol) server registries with PostgreSQL-backed persistence and distributed leader election. An async-upload Python Job handles model uploads between storage backends (S3, OCI, HTTP/HuggingFace) with optional Sigstore signing and full registry lifecycle management.

## Architectural Analysis

Model Registry uses a multi-process, single-repo architecture where the same Go binary (`model-registry`) serves three distinct roles depending on the subcommand: `proxy` (REST API), `controller` (K8s operator), and `catalog` (model discovery). This monorepo approach simplifies dependency management — all Go components share the same `go.mod` — but means each deployment only activates a subset of the binary's capabilities. The BFF lives in `clients/ui/bff/` with its own router and middleware stack, compiled as a separate binary.

The datastore abstraction is notably well-designed. The `internal/datastore` package defines a `Connector` interface and `RepoSet` pattern that uses Go reflection to compose 14+ repository implementations at startup. Each repository (RegisteredModel, ModelVersion, Artifact, Experiment, etc.) maps to MLMD-compatible database tables via GORM. The system supports lazy initialization with exponential backoff — the HTTP server starts immediately and serves 503 until the database becomes available, ensuring Kubernetes probes work correctly during cold starts. Database migrations are managed via GORM's auto-migrate with a `schema_migrations` tracking table.

The controller is lean by design: it delegates to `InferenceServiceController` (in `pkg/inferenceservice-controller/`) which uses the Model Registry OpenAPI Go client to call back into the REST API rather than directly accessing the database. This means the controller requires the proxy service to be running and reachable. Service discovery is automatic via Kubernetes label selectors (`component: model-registry`) or explicit URL annotation. The controller adds a finalizer to each ISVC it processes, ensuring it can set `DesiredState=UNDEPLOYED` in the registry when an InferenceService is deleted from the cluster.

The catalog service introduces distributed coordination via PostgreSQL-based leader election (pglock). Only the leader pod performs writes (catalog loading, HuggingFace sync, database updates), while standby pods serve cached in-memory data. This design allows scaling read replicas while maintaining data consistency. The MCP (Model Context Protocol) catalog is a notable addition — it provides a registry of AI tools and servers alongside traditional model catalogs, suggesting convergence between model and tool discovery.

Security is layered: Istio provides mTLS between pods within the mesh, the BFF enforces Kubernetes RBAC via SubjectAccessReview before proxying requests, and the REST API itself has input validation middleware that rejects null bytes in all string fields (a defense against injection attacks). The database connection supports full TLS configuration with custom cipher suites and explicit insecure cipher rejection. However, the core REST API has no application-level authentication — it relies entirely on network isolation via Istio. This is a deliberate design choice consistent with platform components that are never exposed directly outside the mesh.
