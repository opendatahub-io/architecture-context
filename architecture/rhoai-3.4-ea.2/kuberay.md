## Purpose

**Short**: KubeRay Operator manages the lifecycle of Ray clusters, jobs, and services on Kubernetes, providing infrastructure for distributed computing workloads in RHOAI.

**Detailed**: KubeRay is a Kubernetes operator built with controller-runtime that manages three primary custom resources: RayCluster, RayJob, and RayService. It automates the creation and scaling of Ray head and worker pods, integrates with batch schedulers (Volcano, Yunikorn, Scheduler-Plugins), and supports GCS fault tolerance via Redis for high availability.

The operator includes an Authentication controller that integrates with the RHOAI Gateway API infrastructure, creating HTTPRoutes and ReferenceGrants for secure dashboard access through the platform's centralized gateway. On OpenShift, a mutating webhook enforces secure-by-default behavior by enabling the secure trusted network annotation and disabling direct Route/Ingress creation, forcing all access through Gateway API.

Additional controllers handle mTLS certificate management via cert-manager (for inter-pod encryption), NetworkPolicy enforcement (restricting ingress to Ray head and worker pods based on annotations), and authentication sidecar injection (kube-rbac-proxy for OIDC or oauth-proxy for OpenShift OAuth). The operator supports namespace-scoped or cluster-scoped watching and exposes Prometheus metrics for cluster, job, and service lifecycle tracking.

## Architectural Analysis

The KubeRay operator follows a controller-per-concern pattern with six distinct controllers registered against the same manager: RayCluster (core infrastructure), RayJob (job lifecycle), RayService (serve deployment), mTLS (certificate management), NetworkPolicy (network isolation), and Authentication (Gateway API integration). Each controller watches RayCluster as its primary resource but manages different aspects of the cluster lifecycle. This separation of concerns keeps individual controller logic manageable but requires coordination through Kubernetes primitives — annotations gate feature activation (e.g., `odh.ray.io/secure-trusted-network`), conditions track cross-controller readiness (e.g., `AuthenticationReady`), and finalizers ensure cleanup ordering.

The OpenShift integration is enforced at the admission layer rather than at runtime. The mutating webhook unconditionally sets `odh.ray.io/secure-trusted-network=true` and forces `enableIngress=false` on every RayCluster created on OpenShift. This means the Authentication controller always activates on OpenShift and the RayCluster controller never creates Routes or Ingresses. The design pushes all dashboard access through Gateway API HTTPRoutes, ensuring centralized authentication via kube-rbac-proxy or oauth-proxy sidecars. This is a deliberate security enforcement — even if a user explicitly sets `enableIngress: true`, the webhook overrides it.

The HTTPRoute architecture uses cross-namespace routing: the HTTPRoute is created in the platform namespace (APPLICATION_NAMESPACE, typically `opendatahub`) but references a backend Service in the RayCluster's namespace. A ReferenceGrant in the cluster namespace authorizes this cross-namespace reference. The HTTPRoute defines two rules: an exact match on `/ray/{ns}/{cluster}` that redirects to `/#/` (dashboard SPA entrypoint), and a prefix match on `/ray/{ns}/{cluster}/` that rewrites to `/` and forwards to the head service on port 8265. This multi-namespace pattern is consistent with how other RHOAI components integrate with the platform gateway.

The mTLS implementation uses cert-manager's full certificate chain: a SelfSigned Issuer bootstraps a root CA Certificate, which provisions a CA Issuer that signs per-pod certificates. Worker certificates use wildcard DNS SANs to cover headless service discovery, and all certificates include dynamic pod IP SANs that update as pods scale. Certificate validity is 90 days for pod certs and ~10 years for the CA, with a 1-minute reconciliation loop and 24-hour expiry warnings. The mTLS controller watches pod changes to keep SANs current, which is critical for Ray's autoscaling behavior where pods appear and disappear frequently.

The NetworkPolicy controller demonstrates a defense-in-depth approach activated by the same `odh.ray.io/secure-trusted-network` annotation. Head pods get fine-grained ingress rules: intra-cluster communication on all ports, namespace pods on client (10001) and dashboard (8265) ports, the operator namespace on dashboard and client ports, monitoring namespace on metrics (8080), gateway namespace on dashboard (8265), and RayJob submitter pods on all ports. Worker pods are restricted to intra-cluster communication only. Gateway and monitoring namespace configuration comes from environment variables (GATEWAY_NAMESPACE defaulting to `openshift-ingress`, MONITORING_NAMESPACE), making it configurable per deployment.
