## Purpose
**Short**: KubeRay operator manages the lifecycle of Ray clusters, jobs, and services on Kubernetes/OpenShift, with RHOAI-specific authentication and network security extensions.

**Detailed**: The KubeRay operator is a Kubernetes controller-runtime based operator that reconciles three custom resources -- RayCluster, RayJob, and RayService -- to manage distributed Ray computing infrastructure. It handles the full lifecycle of Ray clusters including head and worker pod creation, autoscaling, service exposure, batch job scheduling integration (Volcano, Yunikorn), and GCS fault tolerance via Redis.

In the RHOAI distribution, the operator is significantly extended with OpenShift-specific controllers for authentication (kube-rbac-proxy sidecar injection + Gateway API HTTPRoute creation), mutual TLS certificate management (via cert-manager), and network policy enforcement. The authentication controller creates HTTPRoute CRs in the platform namespace (referencing the `data-science-gateway` Gateway) and injects kube-rbac-proxy sidecars into Ray head pods for RBAC-based access control. The mTLS controller manages per-cluster CA certificates and head/worker certificates via cert-manager for encrypted intra-cluster communication. The network policy controller creates per-cluster NetworkPolicies to restrict traffic to Ray head and worker pods.

The operator is built using FIPS-compliant Go (`GOEXPERIMENT=strictfipsruntime`) on UBI9 base images for the RHOAI distribution. It supports disconnected (air-gapped) installations and runs with a dedicated SecurityContextConstraints (`run-as-ray-user`) on OpenShift.

### Ingress

| Name | Type | Hosts | Port | Protocol | Encryption | TLS Mode | Exposure |
|------|------|-------|------|----------|------------|----------|----------|
| {namespace}-{cluster} HTTPRoute | HTTPRoute (Gateway API) | Via data-science-gateway | 8265/TCP | HTTPS | TLS (Gateway-terminated) | SIMPLE | External (via Gateway) |

**Gateway API Ingress Pattern (RHOAI 3.x)**:
- The **Authentication Controller** creates `HTTPRoute` CRs in the platform namespace (e.g., `redhat-ods-applications`)
- HTTPRoutes reference the `data-science-gateway` Gateway in `openshift-ingress` namespace as their parent
- Path prefix: `/ray/{namespace}/{cluster}` with URL rewrite to `/` on the backend
- Cross-namespace service references are enabled via `ReferenceGrant` CRs in the cluster's namespace
- `kube-rbac-proxy` sidecar (port 8443/TCP) proxies to Ray Dashboard (127.0.0.1:8265) for authentication
- The operator enforces `enableIngress: false` on OpenShift to prevent legacy Route/Ingress creation
- On cluster deletion, a finalizer (`ray.io/authentication-resources`) ensures cleanup of cross-namespace HTTPRoutes and ReferenceGrants

### Network Policies (Created dynamically per RayCluster)

**Head Node NetworkPolicy** (`{cluster}-head`):
- **Rule 1**: Intra-cluster - Pods with `ray.io/cluster={cluster}` label: all ports
- **Rule 2**: Same-namespace pods: ports 10001/TCP (client), 8265/TCP (dashboard)
- **Rule 3**: KubeRay operator namespace: ports 8265/TCP, 10001/TCP
- **Rule 4** (optional): Monitoring namespace: port 8080/TCP (metrics)
- **Rule 4b** (optional): Gateway namespace (`openshift-ingress`): port 8265/TCP (dashboard)
- **Rule 5**: Port 8443/TCP (kube-rbac-proxy) from anywhere
- **Rule 6** (conditional): RayJob submitter pods: all ports

**Worker Node NetworkPolicy** (`{cluster}-workers`):
- Only allows ingress from pods with `ray.io/cluster={cluster}` label (intra-cluster only)
