## Purpose

**Short**: Argo Workflows is a Kubernetes-native workflow execution engine that orchestrates DAG and step-based workflows, used as the execution backbone for Data Science Pipelines in RHOAI.

**Detailed**: Argo Workflows provides a complete workflow execution platform on Kubernetes. It defines Custom Resource Definitions (CRDs) for workflows, cron-scheduled workflows, workflow templates, and supporting resources. The system consists of three primary binaries: a workflow controller that reconciles Workflow CRs and creates execution pods, an executor (argoexec) sidecar injected into workflow pods to manage container execution and artifact handling, and a server (argo server) that provides gRPC/REST APIs and a web UI for workflow management.

In the RHOAI context, Argo Workflows serves as the execution engine for Data Science Pipelines. The workflow controller watches for Workflow CRs created by the DSP pipeline orchestration layer, creates and manages execution pods for each workflow step, handles artifact storage (S3/MinIO), supports DAG-based parallelism, cron scheduling, and workflow archiving to SQL databases. The RHOAI distribution builds FIPS-compliant binaries using UBI 9 base images with `GOEXPERIMENT=strictfipsruntime`.

The system supports both cluster-scoped and namespace-scoped installation modes, with separate RBAC for the controller and server components. It integrates with external artifact storage systems (S3, GCS, Azure Blob, MinIO), optional SQL databases (PostgreSQL, MySQL) for workflow archiving, and OIDC providers for SSO authentication.

## Architectural Analysis

Argo Workflows follows a classic Kubernetes operator pattern but with significantly more complexity than a typical single-CRD operator. The workflow controller manages eight CRDs and orchestrates a multi-stage execution pipeline that spans pod creation, sidecar injection, artifact management, and optional database persistence. The architecture is designed for high throughput — configurable worker pools (default 32 workflow workers) process reconciliation events in parallel, with per-workflow key locks preventing concurrent modification of the same workflow.

The executor architecture uses an "emissary" pattern where the argoexec binary is injected as a sidecar into every workflow pod. This sidecar communicates with the main container via a shared `/var/run/argo` volume mount, capturing exit codes and managing artifact upload/download. The WorkflowTaskResult CRD provides a decoupled communication channel between argoexec and the controller — the executor writes results to CRs rather than requiring direct API calls, which improves reliability in high-concurrency scenarios.

The FIPS story is nuanced: the Konflux Dockerfiles build both FIPS-compliant (`argoexec-fips` with `strictfipsruntime`) and non-FIPS (`argoexec` with static linking) variants of the argoexec binary. The entrypoint defaults to the non-FIPS binary (`/bin/argoexec`), though both are present in the image. The workflow-controller Konflux Dockerfile builds only a FIPS-compliant binary. This asymmetry means that on a FIPS-required cluster, the argoexec entrypoint may need to be overridden to use `/bin/argoexec-fips`. The TLS configuration supports configurable minimum version (default 1.2) and the SSO integration has an `InsecureSkipVerify` option that could bypass FIPS-compliant TLS validation if enabled.

The kustomize manifest structure supports two installation modes (cluster-scoped and namespace-scoped) with clean separation of RBAC concerns. The aggregate roles pattern (`argo-aggregate-to-view/edit/admin`) integrates with Kubernetes' built-in RBAC aggregation, automatically granting workflow permissions to users who already have view/edit/admin roles. This is a well-designed RBAC integration that avoids the common anti-pattern of requiring separate role bindings for workflow access.

A notable security gap is the absence of NetworkPolicy resources in the manifests. The controller has broad pod creation permissions (required for its function), and the server has general secret read access. The system relies entirely on RBAC and pod security contexts for isolation, with no network-level segmentation. In a multi-tenant RHOAI deployment, this means any compromised workflow pod could potentially communicate with any other pod in the namespace or cluster, depending on the cluster's default network policies.
