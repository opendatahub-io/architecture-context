## Purpose

**Short**: Kubernetes operator that manages distributed machine learning training jobs across multiple frameworks (PyTorch, TensorFlow, XGBoost, JAX, MPI, PaddlePaddle).

**Detailed**: The Kubeflow Training Operator (KFTO) is a Kubernetes-native controller that automates the lifecycle of distributed ML training workloads. It defines Custom Resource Definitions (CRDs) for six ML frameworks — PyTorchJob, TFJob, XGBoostJob, JAXJob, MPIJob, and PaddleJob — and reconciles these CRDs into the underlying Kubernetes primitives (Pods, Services, ConfigMaps, RBAC resources) needed to run distributed training at scale.

Each framework controller understands its framework's distributed training topology: PyTorch uses master/worker with optional elastic scaling, TensorFlow supports parameter servers and chief/worker patterns, MPI uses launcher/worker with remote execution via kubectl, XGBoost uses a Rabit tracker, JAX uses a coordinator-based flat topology, and PaddlePaddle supports both collective and parameter server modes. The operator injects framework-specific environment variables (cluster addresses, ranks, world sizes) so user containers can discover peers without manual configuration.

The operator integrates with gang scheduling systems (Volcano and scheduler-plugins) to ensure all training replicas are co-scheduled atomically, preventing resource deadlocks in multi-tenant clusters. For RHOAI, the operator is deployed via kustomize overlays with FIPS-compliant builds, Prometheus monitoring, and aggregated RBAC roles for OpenShift integration.

## Architectural Analysis

The Training Operator follows a well-structured plugin architecture where each ML framework is implemented as an independent controller module that shares a common base (`pkg/controller.v1/common`). The `JobController` base type provides generic Pod, Service, and PodGroup reconciliation, while each framework controller implements framework-specific behavior through interface methods like `SetClusterSpec()`, `IsMasterRole()`, and `UpdateJobStatus()`. This design allows adding new frameworks with minimal code — each needs only a controller file, an envvar file, and a webhook. The `SupportedSchemeReconciler` registry pattern in `register_controller.go` enables runtime selection of which frameworks to enable via the `--enable-scheme` flag.

The gang scheduling integration is particularly well-designed as an optional, pluggable subsystem. The `GangSchedulingSetupFunc` pattern allows the operator to detect at startup whether Volcano or scheduler-plugins CRDs are installed (via REST mapper checks), and configures the appropriate `PodGroupControl` implementation. For Volcano, the operator delays pod creation until the PodGroup reaches "Inqueue" phase, preventing partial scheduling. For scheduler-plugins, no delay is applied since that scheduler handles admission differently. The `CalcPGMinResources` function orders replicas by PriorityClass value to calculate the minimum resource footprint needed for co-scheduling, preventing deadlocks in resource-constrained clusters.

The MPI controller stands out as architecturally distinct from other framework controllers. While PyTorch, TensorFlow, XGBoost, JAX, and PaddlePaddle all use the standard Pod+Service pattern with environment variable injection, MPI creates a fundamentally different topology: a launcher pod with RBAC permissions to execute `kubectl exec` into worker pods, a ConfigMap containing hostfiles and the `kubexec.sh` remote execution script, and a dedicated ServiceAccount+Role+RoleBinding per job. The MPI controller also overrides `ReconcileServices()` to a no-op since MPI workers don't need service-based DNS discovery — they communicate through the launcher's kubectl exec mechanism.

The webhook certificate management uses the OPA cert-controller library for automatic generation and rotation. A notable design choice is the readiness probe integration: the operator's `/readyz` endpoint blocks until certificates are ready, and controller setup waits on a `certsReady` channel before registering controllers. This prevents the operator from accepting webhook calls before TLS is established. The namespace-scoped Role for webhook Secret access (rather than ClusterRole) follows least-privilege principles — the operator only needs Secret access in its own namespace for certificate storage.

For RHOAI, the kustomize overlay applies production hardening: the image is pinned to a specific version (`v1.9.0-odh-3`), the PyTorch init container image is set to the same operator image (avoiding external image pulls), aggregate RBAC roles (`training-edit`, `training-view`) are added for OpenShift RBAC integration, and a PodMonitor enables Prometheus metrics collection. The FIPS build uses `GOEXPERIMENT=strictfipsruntime` with `CGO_ENABLED=1` on a UBI 9 base image, ensuring the Go runtime uses OpenSSL for all crypto operations.
