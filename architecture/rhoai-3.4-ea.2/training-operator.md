## Purpose
**Short**: Kubernetes-native operator for managing distributed training jobs across multiple ML frameworks (PyTorch, TensorFlow, XGBoost, MPI, PaddlePaddle, JAX).

**Detailed**: The Kubeflow Training Operator is a Kubernetes-native controller that manages the lifecycle of distributed machine learning training jobs. It provides Custom Resource Definitions (CRDs) for popular ML frameworks including PyTorchJob, TFJob, XGBoostJob, MPIJob, PaddleJob, and JAXJob, enabling users to declaratively specify distributed training workloads on Kubernetes.

The operator handles pod creation, service discovery, replica management, gang scheduling integration (Volcano and scheduler-plugins), and job status tracking. Each framework-specific controller manages the full lifecycle of training jobs including creating worker/master pods, setting up inter-node communication via headless services, configuring environment variables for distributed training (e.g., TF_CONFIG, PyTorch rendezvous), and managing job completion/failure conditions. The PyTorch controller additionally creates NetworkPolicy resources to isolate training job network traffic and allow monitoring access from the openshift-monitoring namespace.

For RHOAI, the operator is deployed with FIPS-compliant builds (using `strictfipsruntime` Go build tag), runs in the `opendatahub` namespace with a `kubeflow-` name prefix, and includes validating webhooks with auto-rotating certificates managed by the cert-controller library.

## Data Flows

### Flow 1: Training Job Submission

| Step | Source | Destination | Port | Protocol | Encryption | Auth |
|------|--------|-------------|------|----------|------------|------|
| 1 | User/Client | Kubernetes API Server | 443/TCP | HTTPS | TLS 1.2+ | Bearer Token / kubeconfig |
| 2 | Kubernetes API Server | Training Operator Webhook | 9443/TCP | HTTPS | TLS (auto-rotated) | API Server TLS cert |
| 3 | Training Operator | Kubernetes API Server | 443/TCP | HTTPS | TLS 1.2+ | ServiceAccount Token |
| 4 | Training Operator | Kubernetes API Server | 443/TCP | HTTPS | TLS 1.2+ | ServiceAccount Token |

Step 1: User creates a training job CR (e.g., PyTorchJob).
Step 2: API server calls validating webhook to validate the job spec.
Step 3: Controller reconciles and creates Pods, Services, NetworkPolicies per the job spec.
Step 4: Controller updates job status (ReplicaStatuses, conditions).

### Flow 2: Distributed Training (PyTorch Example)

| Step | Source | Destination | Port | Protocol | Encryption | Auth |
|------|--------|-------------|------|----------|------------|------|
| 1 | Worker Pod 0 | Headless Service DNS | 29500/TCP | TCP | plaintext | None |
| 2 | Worker Pod N | Worker Pod 0 (master) | 29500/TCP | TCP | plaintext | None |
| 3 | Worker Pods | Worker Pods | various/TCP | TCP | plaintext | None |

Step 1: Worker 0 (master) listens on the rendezvous port (default 29500).
Step 2: Other workers connect to master via DNS-resolved headless service address.
Step 3: Workers communicate for gradient synchronization (NCCL/Gloo backends).
