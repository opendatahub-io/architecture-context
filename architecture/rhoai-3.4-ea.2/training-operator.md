## Purpose
**Short**: Kubernetes-native operator for managing distributed training jobs across multiple ML frameworks (PyTorch, TensorFlow, XGBoost, MPI, PaddlePaddle, JAX).

**Detailed**: The Kubeflow Training Operator (KFTO) is a Kubernetes operator that provides a unified API for defining and running distributed machine learning training jobs. It supports six ML frameworks through dedicated Custom Resource Definitions: PyTorchJob, TFJob, XGBoostJob, MPIJob, PaddleJob, and JAXJob. Each CRD allows users to specify replica types (master, worker, parameter server, etc.), resource requirements, and training configurations.

The operator manages the full lifecycle of training jobs including pod creation, service creation for inter-pod communication, status tracking, gang scheduling integration (via Volcano or scheduler-plugins), horizontal pod autoscaling for elastic training, and cleanup policies. For PyTorchJob workloads, the operator also creates NetworkPolicy resources to isolate inter-pod training traffic while allowing Prometheus metrics scraping from the openshift-monitoring namespace.

The operator runs as a single-replica Deployment in the `opendatahub` namespace and exposes a validating webhook server (port 9443/TCP with self-managed TLS certificates) for admission control of all supported job types. It also serves Prometheus metrics on port 8080/TCP and health/readiness probes on port 8081/TCP. The RHOAI build uses FIPS-compliant Go compilation with `strictfipsruntime` build tags.

### Dynamically Created Resources (per training job)

The operator dynamically creates the following Kubernetes resources for each training job:

| Resource | Created By | Purpose |

| Services (headless) | All controllers | Inter-pod communication for distributed training (DNS-based discovery) |
| NetworkPolicy | PyTorchJob controller only | Isolates inter-pod training traffic; allows same-job pod communication and openshift-monitoring metrics scraping on 8080/TCP |
| HorizontalPodAutoscaler | PyTorchJob controller (elastic) | Auto-scales worker replicas for elastic PyTorch training |
| PodGroup | All controllers (optional) | Gang scheduling via Volcano or scheduler-plugins |

### Network Policies (Dynamically Created)

For each PyTorchJob, the controller creates a NetworkPolicy with:
- **Pod Selector**: Matches pods with label `training.kubeflow.org/job-name: <job-name>`
- **Ingress Rule 1**: Allow all traffic from pods with same job-name label (inter-worker communication)
- **Ingress Rule 2**: Allow TCP/8080 from pods in `openshift-monitoring` namespace (Prometheus scraping)
