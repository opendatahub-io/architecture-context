## Purpose

**Short**: CPU-optimized build of vLLM, a high-throughput and memory-efficient inference and serving engine for Large Language Models (LLMs).

**Detailed**: vLLM is a high-performance inference engine that provides OpenAI-compatible API endpoints for serving large language models. This repository is the CPU-specific variant of vLLM, optimized for running inference without GPU hardware. It supports x86_64, s390x (IBM Z), and ppc64le (IBM Power) architectures.

The server exposes an OpenAI-compatible REST API via FastAPI/Uvicorn on port 8000, supporting chat completions, text completions, embeddings, scoring, classification, and speech-to-text endpoints. It also provides a gRPC interface on port 50051 for efficient binary communication with routing infrastructure. The engine implements PagedAttention for efficient memory management, continuous batching for high throughput, and supports features like LoRA adapters, structured outputs (JSON schema, regex, grammar), streaming responses, and multimodal inputs.

In the RHOAI platform, vllm-cpu is deployed as a model serving runtime within KServe InferenceService pods. It is consumed by the platform's model serving infrastructure and accessed through the platform's Gateway API ingress layer. The CPU variant enables model serving on nodes without GPU resources, making it suitable for smaller models or environments where GPU hardware is unavailable.

## Architectural Analysis

vLLM's architecture follows a clear separation between the API layer (FastAPI/gRPC entrypoints) and the inference engine (AsyncLLM). The OpenAI-compatible HTTP server is the primary interface, built on FastAPI with a modular router structure — each API domain (chat completions, completions, embeddings, responses, scoring, classification, speech-to-text, Anthropic compatibility, SageMaker compatibility) is registered as a separate router with its own serving class. This design allows conditional registration based on model capabilities (e.g., pooling endpoints only registered if the model supports pooling tasks).

The gRPC server (`vllm/entrypoints/grpc_server.py`) provides a separate binary-efficient interface designed for communication with the Rust-based vllm-router. It implements the `VllmEngine` service defined in `vllm/grpc/vllm_engine.proto`, supporting streaming generation, health checks, request abortion, and model/server info queries. Notably, the gRPC server uses `add_insecure_port` without TLS, relying on network-level isolation rather than transport-level encryption. The Embed RPC is defined in the proto but not yet implemented.

The CPU variant's build system is notable for its complexity. The `Dockerfile.konflux.cpu` uses a multi-stage build where the builder stage runs architecture-specific build scripts (`build_vllm_s390x.sh`, `build_vllm_ppc64le.sh`) that compile system libraries (numactl, OpenBLAS, LAPACK, LLVM) from source. The `setup.py` orchestrates CMake-based compilation of C/C++ CPU kernels, producing `vllm._C` and `vllm._C_AVX2` native extensions on x86_64. The build supports both AVX2 and AVX512 instruction sets for optimal performance across CPU generations.

Security in this component follows a "defense in depth" model where the application itself provides optional authentication (API key-based Bearer Token via `AuthenticationMiddleware`) and optional TLS (via ssl module with certificate hot-reload through `SSLCertRefresher`), but the primary security enforcement comes from the platform layer: kube-rbac-proxy sidecars handle OAuth/RBAC, the RHOAI Gateway provides TLS termination, and Kubernetes network policies restrict pod-to-pod traffic. The gRPC endpoint has no authentication at all, which is a design choice reflecting its intended use as an internal-only communication channel.

The `VLLM_USE_V1=1` environment variable set in the Konflux Dockerfile enables the V1 engine architecture, which uses the `AsyncLLM` class from `vllm.v1.engine.async_llm` with a dedicated frontend multiprocessing model. This is the modern engine path that the RHOAI build uses exclusively. Configuration is extensive — the `envs.py` file defines 80+ environment variables covering every aspect of engine behavior, from memory management (`VLLM_CPU_KVCACHE_SPACE`) to logging (`VLLM_LOGGING_LEVEL`) to cache directories (`VLLM_CACHE_ROOT`).
