## Purpose
**Short**: CPU-optimized build of vLLM, providing high-throughput LLM inference and serving without GPU hardware via an OpenAI-compatible API.

**Detailed**: vLLM CPU is the CPU-only variant of the vLLM inference engine, packaged as a Red Hat UBI-based container image for deployment in Red Hat OpenShift AI (RHOAI). It provides a high-throughput, memory-efficient inference and serving engine for Large Language Models (LLMs) that can run on CPU-only nodes, making it suitable for environments without GPU resources or for cost-effective deployments of smaller models.

The component exposes an OpenAI-compatible REST API server (FastAPI/Uvicorn) and a gRPC server for efficient binary communication with external routers. It supports a wide range of model architectures including transformer-based LLMs (e.g., Llama), Mixture-of-Expert models (e.g., Mixtral), embedding models, and multi-modal LLMs. Key features include PagedAttention for efficient memory management, continuous batching, speculative decoding, structured output generation, LoRA adapter support, and streaming responses.

This image is built by Konflux using `Dockerfile.konflux.cpu` on a UBI 9 minimal base. Platform-specific build scripts (`build_vllm_s390x.sh`, `build_vllm_ppc64le.sh`) handle cross-architecture compilation of native dependencies including PyTorch, OpenBLAS, and LLVM. The container runs as a non-root user (UID 2000) and is designed for OpenShift deployment, typically managed by KServe as an inference runtime.

### Ingress

Ingress is managed by the RHOAI platform, not by this component directly:
- **RHOAI 3.x pattern**: KServe creates HTTPRoute CRs (Gateway API) pointing to the data-science-gateway for external model access
- **Authentication**: kube-rbac-proxy sidecar injected by KServe/notebook controller, or Istio mTLS
- **TLS termination**: Handled by platform Gateway/Envoy proxy, not by vLLM directly

The vLLM server itself supports optional SSL via `--ssl-keyfile` and `--ssl-certfile` CLI arguments, but in RHOAI deployments TLS is typically terminated at the platform level.
