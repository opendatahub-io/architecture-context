## Purpose

**Short**: Kubernetes operator that manages the lifecycle of Apache Spark applications, scheduled Spark applications, and Spark Connect servers on OpenShift/Kubernetes clusters.

**Detailed**: The Spark Operator (originally from Kubeflow) automates the submission, monitoring, scaling, and cleanup of Apache Spark workloads on Kubernetes. It watches three Custom Resource Definitions — `SparkApplication`, `ScheduledSparkApplication`, and `SparkConnect` — and translates them into Kubernetes-native resources (driver pods, executor pods, services, ingresses, ConfigMaps) using the `spark-submit` CLI under the hood.

The operator is deployed as two separate components: a **controller** deployment that runs the reconciliation loops for all three CRDs, and a **webhook** deployment that handles admission control (mutating and validating webhooks). The webhook component is responsible for injecting Spark-specific configuration into driver and executor pods at creation time (volumes, environment variables, security contexts, sidecar containers, scheduling labels, etc.) and validating that SparkApplication specs meet requirements before submission.

For RHOAI, the operator is deployed into the `redhat-ods-applications` namespace via kustomize overlays managed by the platform operator (`rhods-operator`). It supports pluggable batch schedulers (Volcano, Yunikorn, kube-scheduler-plugins), dynamic executor allocation, Prometheus metrics, Spark Web UI exposure via Kubernetes Ingresses, and self-managed TLS certificates for webhook communication. The operator is built from the upstream Kubeflow spark-operator with Red Hat downstream patches for OpenShift compatibility, RBAC hardening, and network policy enforcement.

## Architectural Analysis

The Spark Operator follows a clean separation between reconciliation (controller) and admission control (webhook) by deploying the same binary with different CLI subcommands. This architectural split is important because the webhook has strict latency requirements (10-second timeout) and different scaling characteristics than the controller. The webhook mutates Spark pods with 25+ configuration injections (volumes, env vars, sidecars, GPU resources, scheduling labels, security contexts, etc.), making it one of the most complex mutating webhooks in the platform.

The operator's SparkApplication reconciliation implements a rich state machine with 14 states (New → Submitted → Running → Succeeding → Completed, with branches for failure, suspension, invalidation, and retry). The state machine handles retry logic for both submission failures and execution failures with configurable intervals. Notably, the operator invokes `spark-submit` as an external process (`$SPARK_HOME/bin/spark-submit`) from within the controller container rather than using the Kubernetes API directly to create driver pods — this is because Spark's native Kubernetes scheduler handles the driver pod creation with the correct Spark configuration. The controller container therefore includes the full Apache Spark runtime (based on `docker.io/library/spark:4.0.1`).

The ScheduledSparkApplication controller implements cron-based scheduling using the `robfig/cron` library with timezone support. It creates SparkApplication instances on schedule with configurable concurrency policies (Allow, Forbid, Replace) and automatic cleanup of historical runs based on success/failure history limits. This provides a Kubernetes-native alternative to external job schedulers for recurring Spark workloads.

The SparkConnect reconciler (v1alpha1, still alpha) manages Spark Connect server deployments — long-running Spark sessions with a gRPC endpoint (port 15002) for interactive client connections. It creates a server pod, a ConfigMap with executor pod templates, and a ClusterIP service with four ports (driver-rpc, blockmanager, web-ui, spark-connect-server). This is a newer API that enables thin-client Spark usage patterns.

The batch scheduler integration uses a registry pattern supporting three schedulers: Volcano (creates PodGroup CRDs with queue/priority), kube-scheduler-plugins (creates PodGroup for gang scheduling), and Yunikorn (annotations-only, no CRD creation). Each scheduler calculates aggregate resource requirements (driver + all executors) for gang scheduling guarantees. The scheduler choice is per-application via `spec.batchScheduler` with a configurable default.

The FIPS posture is a notable gap — the operator binary is built with `CGO_ENABLED=0` (static linking, pure-Go crypto), and the base image is a community Debian-based Spark image rather than UBI. This means the operator would fail the check-payload FIPS gate if strict FIPS compliance were required. The webhook TLS implementation uses RSA 2048-bit self-signed certificates with 10-year validity and a 180-day renewal lookahead, which is functional but not aligned with typical enterprise certificate rotation policies.
