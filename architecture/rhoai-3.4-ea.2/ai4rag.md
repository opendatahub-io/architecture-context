## Purpose

**Short**: ai4rag is a provider-agnostic optimization engine that automatically finds optimal hyperparameters for Retrieval Augmented Generation (RAG) templates.

**Detailed**: ai4rag is a Python library developed by IBM that automates the process of configuring RAG (Retrieval Augmented Generation) pipelines. Rather than manually tuning chunking strategies, embedding models, retrieval methods, and foundation models, ai4rag defines a search space of hyperparameters and uses optimization algorithms — specifically Generalized Additive Models (GAM) and random search — to find the combination that maximizes a chosen quality metric (faithfulness, answer correctness, or context correctness).

The library is designed to be LLM and vector database provider-agnostic. It ships with built-in integrations for OGX (a unified AI services server) for embeddings, vector stores, and text generation, and supports ChromaDB as an alternative in-memory vector store. Users can extend the framework by implementing abstract base classes for foundation models (`BaseFoundationModel`), embedding models (`BaseEmbeddingModel`), vector stores (`BaseVectorStore`), RAG templates (`BaseRAGTemplate`), evaluators (`BaseEvaluator`), and event handlers (`BaseEventHandler`).

ai4rag is a pure Python library — it has no Dockerfiles, no Kubernetes manifests, no container images, and no deployment infrastructure of its own. It is intended to be consumed as a dependency by other platform components (e.g., Kubeflow Pipeline components) or used directly in notebooks and scripts. Within the RHOAI platform context, it provides the RAG optimization engine that powers automated RAG pattern generation workflows.

## Architectural Analysis

ai4rag follows a clean layered architecture with well-defined abstractions at each level. The core experiment engine (`AI4RAGExperiment`) acts as an orchestrator that composes pluggable components: optimizers, vector stores, embedding models, foundation models, chunkers, retrievers, evaluators, and event handlers. All major extension points use abstract base classes (ABC pattern), making the framework genuinely provider-agnostic despite its default OGX integrations.

The optimization strategy is noteworthy. The primary optimizer (`GAMOptimizer`) implements a two-phase approach: first, it evaluates a configurable number of random nodes from the search space to build initial training data, then trains a Generalized Additive Model (via `pygam.LinearGAM`) to predict scores for the remaining search space nodes and selects the most promising ones for actual evaluation. Categorical parameters are encoded via `sklearn.LabelEncoder` before GAM fitting. The optimizer supports warm-starting via `known_observations`, enabling continuation of previously interrupted experiments. The search space applies constraint rules (chunk size > 2× chunk overlap, window size consistency with retrieval method, hybrid search parameter consistency) to prune invalid combinations before optimization begins.

The model pre-selection (MPS) subsystem is an important architectural decision. When the search space contains many foundation models or embedding models, MPS runs a quick evaluation on a small sample of benchmark data using an in-memory ChromaDB vector store, then selects the top-performing models to narrow the search space before the full optimization run. This avoids wasting expensive API calls on poorly-performing models. The MPS evaluates all foundation-model × embedding-model pairs and uses mean-based scoring to rank them.

The event handler abstraction (`BaseEventHandler`) enables different deployment contexts. `LocalEventHandler` writes pattern results to disk as JSON files. `KFPEventHandler` aggregates results in memory for Kubeflow Pipelines integration, where results need to be passed between pipeline steps rather than written to local files. This design makes ai4rag a flexible building block for both local experimentation and cloud-native ML pipeline workflows.

A notable design choice is the separation between the library's core (`ai4rag/`) and development utilities (`dev_utils/`). The `dev_utils/` module provides `FileStore` for loading documents from local directories, `run_experiment.py` for scripted execution, and mock implementations for testing — none of which are included in the published package (excluded via `setuptools.packages.find`). The `samples/` directory contains a Jupyter notebook demonstrating the full workflow.
