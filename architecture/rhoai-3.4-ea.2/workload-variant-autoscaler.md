## Purpose

**Short**: A Kubernetes-based global autoscaler for LLM inference model servers that determines optimal replica counts based on saturation metrics, GPU capacity, and cost optimization across multiple serving variants.

**Detailed**: The Workload Variant Autoscaler (WVA) is a Kubernetes controller that performs intelligent autoscaling for inference model servers serving Large Language Models (LLMs). It works alongside standard Kubernetes HPA and external autoscalers like KEDA to scale workloads that support the scale subresource (Deployments, StatefulSets, LeaderWorkerSets).

WVA introduces the concept of a "variant" - a specific way of serving a model with a particular combination of hardware (e.g., A100, H100, L4 GPUs), runtime, and serving approach. When multiple variants serve the same base model, WVA optimizes which variants to scale (e.g., adding capacity on the cheapest variant, removing it from the most expensive). The autoscaler continuously monitors vLLM server metrics via Prometheus, including KV cache utilization and queue depth, to determine optimal replica counts using a saturation-based capacity model.

The controller emits custom Prometheus metrics (`wva_desired_replicas`, `wva_desired_ratio`) that external autoscalers (HPA/KEDA) consume to perform the actual scaling. This decoupled architecture allows WVA to focus on the intelligent scaling decision-making while delegating the mechanics of scaling to proven Kubernetes-native tools. WVA also supports scale-to-zero scenarios and integrates with the Gateway API Inference Extension (InferencePool) for workload-aware scheduling.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| VariantAutoscaling Controller | Go controller-runtime Reconciler | Watches VariantAutoscaling CRs, resolves scale targets (Deployments), applies optimization engine decisions from shared cache, updates CR status |
| InferencePool Controller | Go controller-runtime Reconciler | Watches InferencePool CRs (v1 and v1alpha2 from gateway-api-inference-extension), tracks pool endpoint data in datastore |
| ConfigMap Controller | Go controller-runtime Reconciler | Watches ConfigMaps for saturation-scaling and scale-to-zero configuration, supports global and namespace-local overrides |
| Saturation Engine | Background optimization loop | Runs saturation-based analysis using Prometheus metrics (KV cache utilization, queue depth) to compute desired replica counts |
| Scale-from-Zero Engine | Background optimization loop | Handles scale-from-zero scenarios by directly scaling target objects via the scale subresource API |
| Collector / Prometheus Source | Metrics data layer | Queries Prometheus for vLLM metrics, caches results with configurable TTL, supports query templating |
| Actuator | Metrics emitter + scale executor | Emits WVA custom metrics to Prometheus (wva_desired_replicas, wva_current_replicas, wva_desired_ratio) and optionally scales targets directly |
| Direct Actuator | Scale subresource client | Directly scales Deployments/StatefulSets via the Kubernetes scale subresource API for scale-from-zero |
| Datastore | In-memory data store | Tracks namespace associations, InferencePool data, and configuration state across controllers |
