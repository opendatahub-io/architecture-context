## Purpose
**Short**: A Kubernetes-based global autoscaler that intelligently scales inference model server replicas for LLM workloads based on saturation metrics.

**Detailed**: The Workload Variant Autoscaler (WVA) is a Kubernetes operator that provides intelligent autoscaling for inference model servers serving Large Language Models (LLMs). It works alongside standard Kubernetes HPA and external autoscalers like KEDA, observing the current system state through Prometheus metrics (KV cache utilization, queue depth, request rates) to determine optimal replica counts for inference servers.

WVA introduces the concept of "variants" - different ways of serving the same model with different hardware (A100, H100, L4), runtimes, parallelism strategies, or performance requirements. When multiple variants serve the same model, WVA makes cost-optimized decisions about which variant to scale (e.g., adding capacity on the cheapest variant, removing it from the most expensive). The controller manages `VariantAutoscaling` custom resources and emits Prometheus metrics that external autoscalers (HPA/KEDA) consume to perform the actual scaling.

The operator also integrates with the Gateway API Inference Extension's `InferencePool` resources to track endpoint pools and supports scale-to-zero scenarios. Configuration is managed through both static ConfigMaps (saturation scaling config, scale-to-zero config) and runtime flags, with support for namespace-local configuration overrides.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| VariantAutoscaling Controller | Go Controller (controller-runtime) | Reconciles VariantAutoscaling CRs, resolves scale targets, applies engine decisions from shared cache, updates VA status |
| InferencePool Controller | Go Controller (controller-runtime) | Reconciles InferencePool CRs (v1 and v1alpha2), tracks endpoint pools in datastore |
| ConfigMap Controller | Go Controller (controller-runtime) | Watches ConfigMaps for saturation scaling and scale-to-zero configuration changes, supports global and namespace-local config |
| Saturation Engine | Background Engine Loop | Continuously collects Prometheus metrics (KV cache utilization, queue depth), runs saturation analysis, computes optimal replica counts |
| Scale-from-Zero Engine | Background Engine Loop | Handles scale-from-zero scenarios by monitoring request traffic and triggering scale-up for idle deployments |
| Collector | Metrics Collector | Gathers cluster state and vLLM server metrics from Prometheus sources |
| Actuator | Metrics Emitter | Emits optimization metrics (desired replicas, current replicas, scaling ratios) to Prometheus for consumption by HPA/KEDA |
| Saturation V2 Analyzer | Analysis Engine | Token-based saturation analyzer using capacity knowledge store for computing scaling decisions |
| Datastore | In-memory Store | Tracks namespaces, InferencePool endpoints, and provides shared state between controllers and engines |
| Decision Cache | Shared Cache | Enables saturation engine to trigger VA reconciliation without direct API server updates |
