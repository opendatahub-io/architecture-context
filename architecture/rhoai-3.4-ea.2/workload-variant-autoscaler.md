## Purpose

**Short**: A Kubernetes controller that performs intelligent saturation-based autoscaling for LLM inference model servers, determining optimal replica counts based on GPU utilization, KV cache saturation, and queue depth.

**Detailed**: The Workload Variant Autoscaler (WVA) is a Kubernetes operator built with controller-runtime that manages `VariantAutoscaling` custom resources. Each `VariantAutoscaling` CR represents a "variant" — a specific way of serving a given LLM model with a particular combination of hardware (accelerator type), runtime, and serving configuration. WVA continuously monitors inference server metrics (KV cache utilization, queue depth, request rates) via Prometheus, performs saturation analysis to determine when servers are approaching capacity limits, and emits optimized replica count metrics that external autoscalers (HPA or KEDA) consume to perform actual scaling operations.

WVA supports two analysis engines: a V1 percentage-based saturation analyzer and a V2 token-based saturation analyzer. It features cost-aware optimization across multiple variants serving the same model (scaling the cheapest variant first), scale-to-zero support for idle models via integration with the Gateway API Inference Extension (EPP flow control queue monitoring), and GPU-limited mode that constrains scaling decisions to available cluster GPU capacity. The controller integrates with the llm-d inference scheduling stack and watches `InferencePool` resources from the Gateway API Inference Extension to discover endpoint pickers for scale-from-zero metrics collection.

Configuration is managed through a unified system that supports global ConfigMaps, namespace-local ConfigMap overrides, environment variables, and command-line flags with Viper-based precedence resolution. The controller supports multi-controller isolation via instance labels, namespace exclusion via annotations, and single-namespace or cluster-wide deployment modes.

## Architectural Analysis

The WVA follows a clean separation between **decision-making** (engines) and **decision-application** (reconciler + actuator). The saturation engine runs as a polling loop (30-second interval) on a leader-elected goroutine, collects metrics from Prometheus, runs analysis, and writes decisions to an in-memory `DecisionCache`. It then pushes a `GenericEvent` through a Go channel (`DecisionTrigger`) to wake the reconciler, which reads from the cache and patches the VA status. This design avoids the engine directly writing to the Kubernetes API server, reducing conflict rates and enabling the engine to operate independently of the reconciler's rate limiting.

The scale-from-zero engine has a fundamentally different latency requirement — it polls every 100ms (vs 30s for saturation) because detecting pending requests for scaled-to-zero models must happen quickly. It uses a `DirectActuator` that calls the Kubernetes scale subresource directly rather than going through the Prometheus → HPA path, since the HPA cannot act on zero-replica deployments. This direct scaling path is unusual — most WVA scaling goes through HPA — but necessary for the 0→1 transition.

A notable design pattern is the dual-version InferencePool support. The `InferencePoolReconciler` handles both `inference.networking.k8s.io/v1` (GA) and `inference.networking.x-k8s.io/v1alpha2` (experimental) API groups, selected at startup via the `POOL_GROUP` environment variable. This allows WVA to work with both stable and pre-release versions of the Gateway API Inference Extension without code changes.

The configuration system is sophisticated — it uses Viper for precedence (flags > env > config file > defaults) and supports runtime updates via ConfigMap watching. The namespace-aware configuration allows per-namespace saturation thresholds and scale-to-zero policies, with a resolution order of namespace-local > global. The `ConfigMapReconciler` handles both global ConfigMaps (in the controller's namespace) and namespace-local overrides (in namespaces with VAs), with opt-in via labels and opt-out via annotations. This multi-tenant configuration model is important for RHOAI deployments where different teams may need different scaling parameters.

The FIPS compliance gap is architecturally significant — the Dockerfile.konflux explicitly comments out `GOEXPERIMENT=strictfipsruntime` and uses `CGO_ENABLED=0`, producing a statically linked binary with pure-Go crypto. This binary would fail `check-payload` FIPS validation. The TLS configuration itself is well-designed (enforces TLS 1.2 minimum, supports CA bundles and mTLS), but the underlying crypto implementation is not FIPS-validated. This is likely acceptable for the EA release but represents a known gap for GA.
