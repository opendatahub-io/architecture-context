## Purpose
**Short**: REST API orchestrator that coordinates AI text generation with content safety guardrails by managing detectors, chunkers, and LLM backends.

**Detailed**: The FMS Guardrails Orchestrator serves as middleware between clients and language models, providing content safety detection, filtering, and secure text generation. It acts as a centralized orchestration layer that intercepts requests destined for LLMs, routes text through configurable detector services for content safety analysis, and applies guardrails to both input prompts and generated output.

The orchestrator supports multiple generation backends including TGIS (via gRPC) and OpenAI-compatible APIs (via HTTP), along with pluggable detector services for content analysis (text contents, chat, context documents, and generation detection types). It manages text chunking services that split input into processable segments before detection. The service supports both unary and streaming request patterns, including SSE (Server-Sent Events) for streaming responses and NDJSON for streaming content detection.

Configuration is entirely YAML-driven, allowing dynamic composition of detector pipelines, generation backends, and chunker services with per-service TLS configuration. The orchestrator provides OpenTelemetry-based observability with configurable trace and metrics export via gRPC or HTTP OTLP protocols.

## Security

### TLS Configuration

The orchestrator supports flexible TLS configuration at multiple levels:

**Server-side TLS**:
- Configured via `TLS_CERT_PATH` and `TLS_KEY_PATH` environment variables
- mTLS enabled by additionally providing `TLS_CLIENT_CA_CERT_PATH`
- Uses rustls with ring crypto provider (FIPS-compatible)
- Supports TLS 1.2 and 1.3

**Client-side TLS**:
- Per-service TLS configuration in YAML config
- Named TLS configs for reuse across services
- Supports cert/key pairs, CA certificates, and insecure mode
- gRPC clients use tonic TLS with native and WebPKI roots
- HTTP clients use hyper-rustls

### Authentication & Authorization

| Endpoint | Methods | Auth Mechanism | Enforcement Point | Policy |
|----------|---------|----------------|-------------------|--------|
| /health, /info | GET | None | None | Open access |
| /api/v1/*, /api/v2/* | POST | Optional mTLS, header passthrough | rustls TLS acceptor | Server-level mTLS if configured |
| Outbound to detectors | POST | Header passthrough (X-Forwarded-Access-Token rewrite to Bearer) | Orchestrator | Config-driven |
| Outbound to generation | gRPC/HTTP | mTLS client certs or API token (Bearer) | Client TLS config | Per-service config |

### Header Passthrough

The orchestrator supports configurable header passthrough to downstream services:
- Headers listed in `passthrough_headers` config are forwarded to detectors and generation services
- `X-Forwarded-Access-Token` can be rewritten to `Authorization: Bearer` via `rewrite_forwarded_access_header` config
- `detector-id` and `x-model-name` headers are injected for detector routing

## Configuration

The orchestrator is configured via a YAML file (default: `config/config.yaml`, override with `ORCHESTRATOR_CONFIG` env var):

| Config Section | Purpose | Details |
|----------------|---------|---------|
| `generation` | TGIS/NLP generation backend | Provider type (tgis/nlp), hostname, port, TLS |
| `openai` | OpenAI-compatible backend | Hostname, port, TLS, health service |
| `chunkers` | Text chunking services | Map of chunker ID to type (sentence/all) and service config |
| `detectors` | Content safety detectors | Map of detector ID to type, service, chunker, threshold |
| `tls` | Named TLS configurations | Reusable TLS configs (cert, key, CA, insecure flag) |
| `passthrough_headers` | Header forwarding | List of header keys to pass to downstream services |
| `rewrite_forwarded_access_header` | Token rewrite | Rewrite X-Forwarded-Access-Token to Authorization: Bearer |
| `detector_concurrent_requests` | Concurrency control | Max concurrent detector requests per task (default: 5) |
| `chunker_concurrent_requests` | Concurrency control | Max concurrent chunker requests per task (default: 5) |
