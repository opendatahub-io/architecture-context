## Purpose

**Short**: A REST API orchestrator that coordinates AI text generation with content safety guardrails by routing requests through detector, chunker, and LLM services.

**Detailed**: The FMS Guardrails Orchestrator (`fms-guardrails-orchestr8`) is a high-performance Rust service that acts as middleware between AI application clients and language model backends. It intercepts text generation and chat completion requests, coordinates content safety detection by routing text through configurable detector services, and optionally chunks text via chunker services before detection. The orchestrator supports both unary and streaming request patterns, enabling real-time content safety filtering on generated text.

The orchestrator exposes a REST API with two API versions: a v1 API for classification-with-generation tasks (legacy) and a v2 API for fine-grained detection tasks including text content detection, chat detection, context-aware document detection, and OpenAI-compatible chat/text completions with integrated safety detection. Downstream, it communicates with LLM backends via gRPC (TGIS and caikit-nlp protocols) or HTTP (OpenAI-compatible API), with detector services via HTTP REST, and with chunker services via gRPC.

The service runs as a dual-server architecture: a main API server (default port 8033) with optional TLS/mTLS support, and a separate health server (default port 8034) for Kubernetes probes. It integrates OpenTelemetry for distributed tracing and metrics export, propagating trace context across all downstream service calls.

## Architectural Analysis

The FMS Guardrails Orchestrator follows a **dual-server, fan-out orchestration** pattern. The main API server (port 8033) handles all business traffic while a separate health server (port 8034) serves Kubernetes probes without TLS overhead. This separation ensures health checks remain fast and independent of API server TLS configuration or load. The health server's `/info` endpoint goes beyond simple liveness — it can actively probe all configured downstream services (detectors, chunkers, generation) and cache results, making it suitable for sophisticated readiness gating.

The orchestrator's core architecture is built around **pluggable downstream service clients** abstracted behind a common `Client` trait. Each client type (TGIS, NLP, Chunker, Detector, OpenAI) wraps either a tonic gRPC channel or a hyper HTTP client, with consistent timeout configuration (60s connect, 600s request default), health check interfaces, and OpenTelemetry instrumentation. The gRPC clients use `ginepro` for client-side load balancing with configurable DNS probe intervals, supporting both "lazy" (on-demand) and "eager" (pre-resolved with timeout) resolution strategies. This design allows the orchestrator to handle backend service discovery in environments where DNS entries change dynamically.

The **orchestration handler pattern** is particularly well-designed for the content safety use case. Each API endpoint maps to a handler implementing the `Handle` trait, with handlers composing reusable task functions from `orchestrator/common/tasks.rs`. For example, a chat completions detection request first runs input detection (chunk → detect per detector → aggregate), then forwards to the OpenAI-compatible backend, then runs output detection on the response. Streaming variants use broadcast channels and `DashMap`-backed `CompletionState` for thread-safe concurrent access, with pluggable batching strategies (`CompletionBatcher` for multi-choice completions, `MaxProcessedIndexBatcher` for single-stream content detection).

A notable design decision is the **header passthrough and token rewriting** mechanism (`passthrough_headers` and `rewrite_forwarded_access_header` config options). This allows the orchestrator to forward authentication context from incoming requests to downstream services — critical in multi-tenant environments where detector or generation services need to know the original caller's identity. The `X-Forwarded-Access-Token` to `Authorization: Bearer` rewriting specifically supports OpenShift auth proxy integration patterns.

The FIPS compliance posture presents an **architectural gap**: while the container image includes OpenSSL (satisfying the check-payload gate for RHOAI builds), all application-level TLS is handled by rustls with the ring crypto backend, which is not FIPS 140-2/3 validated. This means every network connection the orchestrator makes — to generation services, detectors, chunkers, and the OTLP collector — uses non-FIPS-validated cryptography. The `insecure: true` config option, which disables all server certificate verification via a `NoVerifier` implementation, adds further risk in production deployments. This is a known trade-off: rustls provides better performance and memory safety than OpenSSL bindings, but at the cost of FIPS certification.
