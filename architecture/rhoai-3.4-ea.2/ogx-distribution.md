## Purpose

**Short**: Builds and ships the Red Hat OpenShift AI Llama Stack distribution container image, providing a multi-provider AI inference, agents, RAG, eval, and safety server.

**Detailed**: The ogx-distribution repository produces a container image that packages Meta's [Llama Stack](https://github.com/llamastack/llama-stack) server with Red Hat's customized distribution configuration. Llama Stack is a unified API server that abstracts multiple AI/ML backend providers (vLLM, AWS Bedrock, Google Vertex AI, Azure OpenAI, OpenAI, IBM watsonx) behind a single OpenAI-compatible REST API, plus additional APIs for agents, vector I/O, evaluation, safety, scoring, datasets, files, batches, and tool runtimes.

The repository itself contains no significant application source code. Instead, it defines a distribution configuration (`distribution/config.yaml`) that declares which Llama Stack APIs and providers are enabled, a build system (`distribution/build.py`) that auto-generates a `Containerfile` with all required Python dependencies, and a Konflux Dockerfile (`Dockerfile.konflux`) for the downstream RHOAI release build. The distribution image ships with the OpenDH fork of Llama Stack (`v0.5.0+rhai0`) and includes multiple external provider plugins for TrustyAI (safety, LM-Eval, RAGAS, Garak) and built-in providers for Milvus, FAISS, pgvector, Qdrant, S3, and more. Providers are conditionally enabled at runtime via environment variables, allowing the same image to serve diverse deployment scenarios from single-model CPU inference to multi-cloud enterprise deployments with PostgreSQL persistence.

The image is built for both `linux/x86_64` and `linux/arm64` architectures. For RHOAI, the Konflux build pipeline uses a pre-built base image with pre-compiled wheel releases, while the upstream CI builds from the UBI9 Python 3.12 base image with online dependency resolution.

## Architectural Analysis

The ogx-distribution repository is architecturally unusual within the RHOAI platform because it ships no operator or controller logic. It is purely a container image definition that packages an upstream project (Llama Stack) with a specific distribution configuration. The repository's primary intellectual value lies in the `distribution/config.yaml` file, which defines the complete provider topology -- which AI backends, vector databases, storage systems, safety frameworks, and evaluation tools are available and how they are conditionally activated. This configuration-as-code approach means the same image supports deployments ranging from a minimal single-vLLM-backend setup to a fully-loaded enterprise configuration with multi-cloud inference, PostgreSQL persistence, remote vector databases, safety guardrails, and evaluation pipelines.

The provider activation pattern uses Llama Stack's conditional environment variable syntax (`${env.VAR:+provider_id}`) to enable providers only when specific environment variables are set. This is a declarative alternative to feature flags: providers like Bedrock, Vertex AI, OpenAI, FAISS, pgvector, Qdrant, and S3 are all compiled into the image but remain dormant unless their activation variable is provided. The always-on providers (Milvus local, PostgreSQL storage, meta-reference agents, TrustyAI safety, HuggingFace datasets, Tavily/Brave search) define the baseline capability set.

The build system has a notable two-layer design. The upstream CI uses `distribution/build.py` to dynamically generate a `Containerfile` by querying `llama stack list-deps` for the full dependency tree, then adds pinned overrides for version compatibility. The downstream Konflux build (`Dockerfile.konflux`) takes a completely different approach: it uses a pre-built RHEL AI base image (`quay.io/aipcc/base-images/cpu:...`) that already contains most dependencies as pre-compiled wheels, and the Dockerfile just installs the wheel release and adds the distribution configuration. This split ensures reproducible RHOAI builds while allowing upstream development flexibility. The `konflux/cpu-ubi9.conf` file pins the exact base image, wheel release versions, and Llama Stack version for the Konflux build.

Security-wise, the Llama Stack server itself runs with no built-in authentication or TLS. All API endpoints are exposed as plaintext HTTP on port 8321. This is by design -- when deployed on RHOAI, the platform operator adds `kube-rbac-proxy` sidecars and `HTTPRoute` rules to provide authentication and TLS termination. The container runs as a non-root user (`CNB_USER_ID:CNB_GROUP_ID` in the Konflux build). Secrets for provider credentials are passed as environment variables, which is standard for container deployments but means credential rotation requires pod restarts. The PostgreSQL connection defaults to plaintext (`localhost:5432` with username/password auth), which is acceptable for co-located deployments but would need TLS configuration for remote PostgreSQL instances.

The telemetry integration via OpenTelemetry is opt-in: when `OTEL_SERVICE_NAME` is set, the entrypoint wraps the Llama Stack server with `opentelemetry-instrument`, automatically exporting traces and metrics via OTLP. Logs export is intentionally disabled by default (`OTEL_LOGS_EXPORTER=otlp` must be set explicitly). This design minimizes overhead in production while enabling full observability when needed.
