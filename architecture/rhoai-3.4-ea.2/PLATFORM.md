## 1. Platform Overview

Red Hat OpenShift AI (RHOAI) is an enterprise AI/ML platform deployed on OpenShift Container Platform (OCP). It provides an integrated stack for the full AI/ML lifecycle: interactive development (workbenches), model training (distributed and single-node), experiment tracking, model serving (single-model and multi-model), model registry, feature stores, AI safety (guardrails, explainability, evaluation), and inference optimization.

RHOAI 3.x introduces a Gateway API-based ingress architecture replacing the legacy OpenShift Route + oauth-proxy pattern. A central `data-science-gateway` Gateway in the `openshift-ingress` namespace provides unified HTTPS ingress, with `kube-auth-proxy` enforcing OAuth2/OIDC authentication via an Istio EnvoyFilter ext_authz integration. Platform components create HTTPRoute CRs referencing this parent Gateway for external access.

The platform is deployed and managed by the `rhods-operator`, which orchestrates 16+ component controllers through `DataScienceCluster` (DSC) and `DSCInitialization` (DSCI) Custom Resources. All 92 container images are pinned by SHA256 digest in the OLM ClusterServiceVersion, enabling fully disconnected installations.

## 5. Network Architecture

### 5.1 Platform Ingress (Gateway API)

```
Internet / User Browser
    │
    ▼ (443/TCP, HTTPS)
OpenShift Router / LoadBalancer
    │
    ▼ (443/TCP, HTTPS TLS Terminate)
data-science-gateway (Gateway API, openshift-ingress namespace)
  GatewayClass: data-science-gateway-class
  Controller: openshift.io/gateway-controller/v1
  Hostname: rh-ai.{cluster-domain}
    │
    ├── EnvoyFilter: data-science-authn-filter (ext_authz)
    │     └── kube-auth-proxy (8443/TCP, HTTPS, OAuth2/OIDC)
    │
    ├── DestinationRule: data-science-tls-rule (upstream TLS)
    │
    └── HTTPRoute references (per-component):
          ├── rhods-dashboard → 8443/TCP (kube-rbac-proxy)
          ├── notebook-{name} → 8443/TCP (kube-rbac-proxy sidecar)
          ├── mlflow-{name} → 8443/TCP
          ├── model-registry-{name} → 8443/TCP (kube-rbac-proxy)
          ├── kuberay-{name} → 8443/TCP
          └── (additional per-component HTTPRoutes)
```

**Ingress Modes**:
- `OcpRoute` (default): ClusterIP Service + OpenShift Route (reencrypt TLS)
- `LoadBalancer`: Direct LoadBalancer Service

**Legacy Redirect**: nginx-based `dashboard-redirect` Deployment creates OpenShift Routes that 301-redirect legacy dashboard/gateway URLs to the new Gateway API hostname.

## 7. Data Flows

### 7.1 User Dashboard Access (RHOAI 3.x)

```
User Browser
  │ (1) HTTPS/443
  ▼
OpenShift Router
  │ (2) HTTPS/443
  ▼
data-science-gateway (Envoy)
  │ (3) ext_authz → kube-auth-proxy (HTTPS/8443)
  │                    │ (4) OAuth/OIDC provider (HTTPS/443)
  │                    ▼
  │              Auth decision (allow/deny)
  │ (5) HTTPS (via DestinationRule)
  ▼
kube-rbac-proxy sidecar (8443)
  │ (6) HTTP/8080 (localhost, X-Auth-Request-User header)
  ▼
odh-dashboard backend
  │ (7) HTTPS/443 → Kubernetes API (impersonated)
  │ (8) HTTPS/9092 → thanos-querier (metrics)
  │ (9) HTTPS/8043-8743 → plugin containers
  ▼
Response → User Browser
```

### 7.2 Model Inference (KServe InferenceService)

```
Client Application
  │ (1) HTTPS/443 (Bearer token)
  ▼
data-science-gateway (Gateway API)
  │ (2) ext_authz → kube-auth-proxy
  │ (3) HTTPRoute → backend service
  ▼
Istio sidecar (proxyv2, mTLS)
  │ (4) HTTP/8000 (plaintext within pod)
  ▼
vLLM / OVMS / MLServer
  │ (5) Inference computation
  ▼
Response → Client Application
```

### 7.3 Model Inference with Guardrails

```
Client Application
  │ (1) HTTP/8090
  ▼
vllm-orchestrator-gateway
  │ (2) Inject detector config, HTTP/8085
  ▼
fms-guardrails-orchestrator
  │ (3a) HTTP/8080 → guardrails-detectors (content safety check)
  │ (3b) HTTP/8080 → guardrails-regex-detector (PII check)
  │ (3c) HTTP/8000 → vLLM backend (LLM generation)
  │ (4) Combine results, apply detections
  ▼
Response (with fallback if detection triggered)
  │ (5)
  ▼
vllm-orchestrator-gateway → Client Application
```

### 7.4 Data Science Pipeline Execution

```
User / Dashboard
  │ (1) Create Pipeline Run via API
  ▼
DSP API Server (8888/TCP)
  │ (2) Store pipeline/run in MariaDB
  │ (3) Create Argo Workflow CR
  ▼
Argo Workflow Controller
  │ (4) Create pipeline step pods
  ▼
Pipeline Step Pod (DSP driver/launcher)
  │ (5) Execute pipeline component
  │ (6) Log artifacts → ml-metadata (gRPC/8080)
  │ (7) Store artifacts → S3/MinIO
  ▼
DSP Persistence Agent
  │ (8) Sync Workflow status → MariaDB
  ▼
DSP API Server → Dashboard (status updates)
```

### 7.5 Distributed Training (TrainJob)

```
User / Dashboard
  │ (1) Create TrainJob CR
  ▼
Trainer Controller
  │ (2) Resolve ClusterTrainingRuntime
  │ (3) Create JobSet CR
  │ (4) Create NetworkPolicy (per-job isolation)
  ▼
JobSet Controller
  │ (5) Create training pods
  ▼
Training Pods (PyTorch distributed)
  │ (6) NCCL/MPI inter-pod communication
  │     (allowed by NetworkPolicy)
  │ (7) Training pod /metrics (28080/TCP)
  ▼
Trainer Controller (progression tracking)
  │ (8) Poll primary pod metrics
  │ (9) Update TrainJob annotations with progress
```

### 7.6 LLM Inference with KV-Cache Scheduling (llm-d)

```
Client Request
  │ (1) HTTPS/443 via Gateway API
  ▼
Gateway (Envoy) + InferencePool HTTPRoute
  │ (2) Ext-proc filter → llm-d-inference-scheduler (EPP)
  │     EPP: hash prompt → select pod by KV-cache affinity
  │     Consider: prefix cache hits, queue depth, LoRA affinity
  ▼
Selected vLLM Pod (prefill or decode)
  │ (3) If P/D disaggregation:
  │     Prefill pod → KV-cache transfer → Decode pod
  │     (via NIXL/UCX or Redis connector)
  ▼
Response → Client
```
