## Platform Overview

Red Hat OpenShift AI (RHOAI) 3.4.0-ea.2 is a comprehensive AI/ML platform deployed on OpenShift Container Platform via OLM (Operator Lifecycle Manager). The platform provides an integrated environment for the complete machine learning lifecycle: interactive development (notebooks/workbenches), experiment tracking (MLflow), data pipelines (Kubeflow Pipelines), distributed training (Training Operator, Trainer, KubeRay), model serving (KServe, vLLM, OpenVINO, MLServer), model registry, feature stores (Feast), AI safety/guardrails (NeMo Guardrails, FMS Guardrails Orchestrator), model evaluation (LM-Eval, EvalHub, Garak, RAGAS), and platform observability (Prometheus, Tempo, Perses).

### Key Architectural Patterns (RHOAI 3.x)

1. **Gateway API Ingress**: Migration from Istio Service Mesh + OpenShift Routes to Gateway API (`gateway.networking.k8s.io`). A centralized `data-science-gateway` Gateway resource in `openshift-ingress` namespace handles all external traffic, with components creating HTTPRoute CRs for exposure.

2. **kube-rbac-proxy Authentication**: Centralized authentication via `kube-auth-proxy` deployment at the gateway level (OAuth2/OIDC + Kubernetes TokenReview), plus per-component kube-rbac-proxy sidecars for service-level RBAC enforcement via SubjectAccessReview.

3. **Modular Dashboard Architecture**: The ODH Dashboard uses Webpack Module Federation with feature-specific UI modules (model-registry, gen-ai, MaaS, MLflow, eval-hub, automl, autorag) running as sidecar containers within the dashboard pod.

4. **Disaggregated LLM Inference (llm-d)**: A new architecture for high-throughput LLM serving with separated prefill/decode phases, KV-cache transfer via NIXL, and intelligent request routing via Gateway API Inference Extension (InferencePool, InferenceModel).

5. **Kuadrant-based API Management**: Models-as-a-Service uses Kuadrant (Authorino + Limitador) for authentication and rate limiting on inference Gateway endpoints.

6. **FIPS Compliance**: All Go components built with `GOEXPERIMENT=strictfipsruntime -tags strictfipsruntime`. Java components use FIPS-compliant JDK. Python containers built on UBI9 with FIPS-enabled OpenSSL.

7. **Disconnected/Air-gapped Support**: All 92 container images pinned by SHA256 digest as RELATED_IMAGE entries in the operator bundle CSV, enabling deployment without internet connectivity.

## Data Flows

### Flow 1: User Authentication and Dashboard Access

```
User Browser
  -> OpenShift Route/LB (443/HTTPS)
    -> data-science-gateway (Envoy) (443/HTTPS, TLS terminate)
      -> EnvoyFilter (data-science-authn-filter)
        -> kube-auth-proxy (8443/HTTPS, mTLS)
          -> OpenShift OAuth Server (443/HTTPS) [token validation]
        <- Auth result (allow/deny with user headers)
      -> Backend Service via HTTPRoute
        -> rhods-dashboard kube-rbac-proxy (8443/HTTPS)
          -> K8s API (443/HTTPS) [TokenReview + SubjectAccessReview]
          -> odh-dashboard backend (8080/HTTP, localhost)
            -> K8s API (443/HTTPS) [user token forwarded]
```

### Flow 2: Model Inference (LLMInferenceService with Kuadrant)

```
Client Application
  -> data-science-gateway (443/HTTPS)
    -> Kuadrant AuthPolicy (Authorino)
      -> Authentication check (UserDefined or Anonymous)
    -> HTTPRoute -> KServe predictor Service
      -> kube-rbac-proxy sidecar (8443/HTTPS)
        -> vLLM/OVMS/MLServer (8000/HTTP, localhost)
          -> Model weights (PVC/S3)
```

### Flow 3: Guardrailed LLM Inference

```
Client
  -> vllm-orchestrator-gateway (8090/HTTP)
    -> FMS Guardrails Orchestrator (8033/HTTP)
      -> Input detectors (guardrails-detectors, regex-detector) (8080/HTTP)
      -> vLLM inference endpoint (8000/HTTP)
      -> Output detectors (8080/HTTP)
    <- Detection results + LLM output (or fallback message)
```

### Flow 4: Notebook Workbench Access

```
User Browser
  -> data-science-gateway (443/HTTPS)
    -> HTTPRoute (per-notebook)
      -> kube-rbac-proxy sidecar (8443/HTTPS)
        -> K8s API (SubjectAccessReview)
        -> JupyterLab/VS Code/RStudio (8888/HTTP, localhost)
```

### Flow 5: ML Pipeline Execution

```
User (Dashboard/SDK)
  -> DSP API Server (8888/HTTP, via kube-rbac-proxy 8443/HTTPS)
    -> Create pipeline run
    -> Argo Workflows (2746/HTTP)
      -> Create Workflow CR
      -> Pipeline step pods (using runtime container images)
        -> S3 storage (artifacts)
        -> ML Metadata (8080/gRPC, lineage tracking)
```

### Flow 6: Distributed Training (TrainJob)

```
User
  -> K8s API (create TrainJob CR)
    -> Trainer webhook (9443/HTTPS, validation)
    -> Trainer controller
      -> Resolve ClusterTrainingRuntime
      -> Create JobSet (via JobSet API)
      -> Create NetworkPolicy (per-TrainJob isolation)
      -> Create PodGroup (gang scheduling, optional)
    -> JobSet controller -> creates Jobs -> creates training Pods
    -> Training pods communicate via headless Service (NCCL/MPI/gRPC)
    -> Trainer progression tracking (polls 28080/HTTP from training pods)
```

### Flow 7: Model Registration and Serving

```
Data Scientist
  -> Model Registry REST API (8080/HTTP, via Istio mTLS)
    -> Register model + version + artifact location
  -> Dashboard / SDK
    -> Create InferenceService CR referencing model
    -> KServe controller deploys predictor pod
    -> odh-model-controller creates Route/metrics/RBAC
    -> Model Registry controller syncs ISVC status
  -> Client
    -> Inference via Gateway API HTTPRoute
```

### Flow 8: MLflow Experiment Tracking

```
ML Client (SDK)
  -> kube-rbac-proxy (8443/HTTPS)
    -> x-remote-user/x-remote-groups headers
  -> MLflow Server (5000/HTTP)
    -> K8s API (SubjectAccessReview for mlflow.kubeflow.org resources)
    -> PostgreSQL (5432, metadata storage)
    -> S3 storage (443/HTTPS, artifact upload/download)
```

## Build System

### Konflux CI/CD Pipeline

All 92 container images are built via Konflux (Tekton-based) with:
- **Hermetic builds**: cachi2 dependency prefetch for supply chain security
- **Multi-architecture**: amd64, arm64, ppc64le, s390x (component-dependent)
- **Base images**: UBI9 (Red Hat Universal Base Image 9)
- **Enterprise Contract**: Supply chain verification before release
- **Image signing**: Cosign-based signature verification
- **SBOM generation**: .syft.yaml for Software Bill of Materials

### Build Stages (Typical)

| Stage | Purpose |
|-------|---------|
| Builder | Compile from source (Go, Rust, C++, Python, TypeScript) |
| Dependencies | Install runtime dependencies |
| Runtime | Minimal UBI9 image with application binaries |

### FIPS Compliance Build Flags

| Language | Build Flags |
|----------|------------|
| Go | `CGO_ENABLED=1 GOEXPERIMENT=strictfipsruntime -tags strictfipsruntime` |
| Java | FIPS-compliant OpenJDK 17 on UBI9 |
| Python | UBI9 with FIPS-enabled OpenSSL (compat-openssl11) |
| Rust | OpenSSL with FIPS mode via native-tls |
| C++ | System OpenSSL on UBI9 |
