## Platform Overview

Red Hat OpenShift AI (RHOAI) 3.4-ea.2 is a comprehensive AI/ML platform deployed on OpenShift Container Platform via an OLM-managed operator (rhods-operator). The platform provides end-to-end capabilities spanning notebook-based development, distributed training, ML pipeline orchestration, model serving and inference, model evaluation and safety, experiment tracking, feature stores, and model registry. The platform is architected as a two-tier CRD system: DSCInitialization (DSCI) handles platform-level prerequisites (namespace creation, CA bundles, network policies, and service singletons for Auth, GatewayConfig, and Monitoring), while DataScienceCluster (DSC) manages individual component lifecycles through 16 component CRDs in the `components.platform.opendatahub.io` API group.

Ingress is managed through a multi-layer Gateway API stack deployed by the rhods-operator gateway controller: GatewayClass → Gateway → Envoy (via Istio/Sail Operator) → EnvoyFilter (ext_authz + Lua cookie stripping) → kube-auth-proxy (OIDC/OAuth2) → component HTTPRoutes. This architecture replaces the legacy OpenShift Route-based ingress from RHOAI 2.x, though several components still support Routes as a fallback. The platform supports three authentication modes (IntegratedOAuth, OIDC, None/External) detected at runtime via cluster configuration.

The security posture combines multiple enforcement layers: Istio service mesh provides pod-to-pod mTLS, kube-rbac-proxy sidecars enforce SubjectAccessReview-based authorization on 7+ component metrics and API endpoints, the kube-auth-proxy handles gateway-level OIDC/OAuth2 authentication, and OpenShift serving certificates provide automated TLS for internal services. Container images are built on UBI base images with non-root execution, shipped with digest-pinned RELATED_IMAGE references for air-gapped deployment, and the platform declares FIPS compliance at the operator level.

### Central Components

**rhods-operator** is the platform's root operator, deploying and managing all other components through 16 component CRDs and 3 service CRDs. It orchestrates the gateway infrastructure (Istio/Envoy/kube-auth-proxy), monitoring stack (MonitoringStack, Thanos, Tempo, Perses, OpenTelemetry), and CA bundle distribution. Every component depends on it for lifecycle management.

**KServe** is the central model serving platform with the most inbound dependencies. It hosts inference workloads from 5 runtime backends (vLLM, OVMS, MLServer, vllm-cpu, vllm-gaudi), provides the InferenceService and LLMInferenceService CRDs that Model Registry, odh-model-controller, MaaS, and TrustyAI integrate with, and deploys the scheduling infrastructure (EPP/Endpoint Picker) for intelligent LLM request routing.

**odh-dashboard** is the primary user-facing entry point with 16 outgoing dependencies. Its modular federation architecture loads UI modules (Gen AI, Model Registry, MaaS, MLflow, AutoML, EvalHub) as sidecar containers, each with a dedicated Go BFF that proxies to downstream platform services.

**TrustyAI Service Operator** functions as an umbrella operator managing 5 distinct CRDs (TrustyAIService, LMEvalJob, GuardrailsOrchestrator, NemoGuardrails, EvalHub), deploying the evaluation, safety, and explainability subsystems.

**Data Science Pipelines Operator (DSPO)** orchestrates the entire pipeline stack per namespace — API Server, Workflow Controller, MLMD, Persistence Agent, Scheduled Workflow controller, and database/storage backends — rendered from Go templates.

**odh-model-controller** bridges KServe with platform infrastructure by managing Routes, AuthPolicies, EnvoyFilters, ServiceMonitors, and NIM integrations for InferenceServices and LLMInferenceServices.

## Deployment Architecture

### Deployment Topology

The platform follows a three-tier namespace topology. The **operator tier** (redhat-ods-operator namespace) contains only the rhods-operator controller manager, deployed via OLM subscription. The **platform services tier** (redhat-ods-applications namespace) hosts all component controllers, operators, and shared infrastructure including the Envoy Gateway, kube-auth-proxy, monitoring stack, and component operators (KServe, DSPO, KubeRay, Trainer, etc.). The **user workload tier** spans user-created project namespaces where notebooks, inference services, pipelines, model registries, and evaluation jobs run — each namespace typically containing its own DSPA instance, InferenceServices, and TrustyAI services provisioned by the platform-tier operators. An optional fourth tier (istio-system or equivalent) hosts the Istio/Sail control plane when deployed separately from the platform namespace.

### Resource Summary

The platform ships 114 container images across 47 components. At the operator level, 16 component controllers and 3 service controllers run within the platform namespace. The gateway infrastructure includes an Envoy-based Gateway (via Istio/Sail), kube-auth-proxy for authentication, and per-component HTTPRoutes. Seven components deploy webhook servers for admission control (KServe, KubeRay, Trainer, Training Operator, Spark Operator, DSPO, Feast). The monitoring stack optionally deploys MonitoringStack, ThanosQuerier, TempoMonolithic/TempoStack, OpenTelemetryCollector, and Perses dashboard instances.

#### Manifest Consumption Pipeline

Component manifests flow from component source repositories through a multi-stage pipeline into the cluster. Each component repo contains kustomize manifests under `config/` or `manifests/` directories. The rhods-operator repository runs `get_all_manifests.sh` at build time, which clones component repos at pinned commits and copies their manifest directories into `prefetched-manifests/`. These are then embedded into the operator binary via Go's `embed.FS`. At runtime, the operator's component controllers select the appropriate overlay (ODH or RHOAI), apply image substitutions from `params.env` (populated with RELATED_IMAGE values from the operator's environment), render any Go templates with CR-specific values, and apply the final manifests to the cluster via server-side apply with field ownership tracking.

### Disconnected Support

The platform supports fully air-gapped (disconnected) deployment through the RELATED_IMAGE pattern. All 114 container images are declared as RELATED_IMAGE environment variables in the operator's ClusterServiceVersion (CSV), which OLM includes in the `relatedImages` section for oc-mirror to discover and mirror. Image references use SHA-256 digests exclusively (never tags), ensuring reproducible deployments and compatibility with Konflux's check-payload verification. The operator's component controllers substitute these digest-pinned references into manifests at deployment time via params.env. The platform also supports custom CA bundles via the `odh-trusted-ca-bundle` ConfigMap, which is injected into all component pods for enterprise proxy environments. TrustyAI's RHOAI overlay additionally disables `lmes-allow-online` to prevent evaluation workloads from downloading models at runtime in disconnected clusters.

## Platform Architectural Analysis

RHOAI 3.4-ea.2 represents a platform in active architectural transition, with several design patterns coexisting across maturity stages. The most significant platform-level observations emerge from reading all 47 component analyses together:

**Gateway API Migration In Progress.** The platform is migrating from OpenShift Route-based ingress to Kubernetes Gateway API, but the transition is incomplete. The rhods-operator deploys a full Gateway API stack (GatewayClass → Gateway → Envoy → kube-auth-proxy), and KServe's LLMInferenceService uses HTTPRoutes exclusively. However, standard InferenceServices still create OpenShift Routes via odh-model-controller, DSPO creates Routes for API Server and MLMD endpoints, and the Argo Workflows UI uses Routes. This dual-ingress architecture means two parallel authentication paths exist: kube-auth-proxy for Gateway API traffic and kube-rbac-proxy sidecars or oauth-proxy for Route-based traffic. The KServe codebase contains three parallel ingress implementations (Istio VirtualService, Kubernetes Ingress, Gateway API HTTPRoute), reflecting this evolution.

**Authentication Fragmentation.** Despite the centralized gateway auth infrastructure, authentication mechanisms vary significantly across the platform. Seven components use kube-rbac-proxy sidecars, gateway-routed services use kube-auth-proxy, EvalHub and MLflow use Kubernetes TokenReview directly, MaaS implements dual-path auth (OpenShift tokens + API keys via Authorino), and inference servers generally delegate auth to the mesh or use no auth. This diversity means there is no single authentication enforcement point — a security auditor must verify each component's auth chain independently. The kube-auth-proxy binary consolidates oauth2-proxy and kube-rbac-proxy into a single image (selected via PROXY_MODE), which is a pragmatic step toward unification but doesn't eliminate the architectural fragmentation.

**Umbrella Operator Sprawl.** The TrustyAI Service Operator has grown to manage five distinct CRDs (TrustyAIService, LMEvalJob, GuardrailsOrchestrator, NemoGuardrails, EvalHub) that serve different functional domains (explainability, evaluation, safety). While the `--enable-services` flag allows selective activation, the coupling means a bug in the GuardrailsOrchestrator controller could affect EvalHub availability if the operator process restarts. The rhods-operator itself manages 19 CRDs. This concentration of control creates operational risk — a single operator upgrade affects all managed components simultaneously.

**LLM Serving Stack Complexity.** The llm-d inference stack introduces sophisticated distributed systems patterns (disaggregated prefill/decode, KV-cache-aware routing, SLO-based autoscaling) that represent a significant step up in operational complexity. The EPP uses Envoy's ext-proc protocol for request routing, the routing sidecar supports three different P/D connector protocols (NIXLV2, SGLang, Shared Storage), and the WVA autoscaler computes replicas from GPU utilization metrics. A notable FIPS discrepancy exists: the routing sidecar is built with `-tags strictfipsruntime` while the EPP binary is not, despite both shipping in RHOAI images.

**Consistent Kustomize Embedding Pattern.** Across all operator-managed components, a consistent manifest flow has emerged: source repo → get_all_manifests.sh → prefetched-manifests/ → embed.FS → runtime overlay selection → params.env image substitution → server-side apply. This pattern ensures reproducible deployments and supports the RELATED_IMAGE disconnected strategy, but it means manifest changes require operator rebuilds. The AllowListedFields RemoverPlugin in rhods-operator preserves user customizations (resources, replicas) across upgrades — a thoughtful design that prevents operator reconciliation from overwriting operational tuning.

**Evaluation and Safety as First-Class Platform Concerns.** RHOAI 3.4 elevates model evaluation and safety from optional add-ons to integrated platform capabilities. The EvalHub → lm-evaluation-harness → MLflow pipeline provides standardized benchmarking, the FMS Guardrails Orchestrator → Detectors → Regex Detector stack provides content safety, NeMo Guardrails adds programmable LLM safety rails, and the LlamaStack providers (Ragas, Garak) enable evaluation via the Llama Stack API. The TrustyAI operator's autoconfig feature automatically discovers KServe InferenceServices and wires them into the guardrails orchestrator, reducing manual configuration. This integrated evaluation/safety stack is a platform differentiator but adds significant component count and dependency complexity.

**Database Backend Heterogeneity.** Different components choose different database backends: DSPO uses MariaDB, Model Registry uses PostgreSQL (GORM), EvalHub uses PostgreSQL or SQLite, MLflow uses PostgreSQL, Feast uses PostgreSQL or SQLite, and MLMD uses a separate gRPC service backed by its own database. There is no shared database infrastructure — each component manages its own persistence lifecycle. While this isolation improves blast radius, it increases the operational burden of managing multiple database instances per namespace.

**Observability Architecture Is Conditional.** The monitoring stack deployment is gated on CRD availability — all required CRDs (MonitoringStack, ThanosQuerier, TempoMonolithic, etc.) must be present or the entire tier is skipped. Per-component PrometheusRules are registered dynamically as components transition to Ready state. The dashboard's Perses integration adds a second dashboarding system alongside the existing Grafana-based approach. The WVA autoscaler consumes Prometheus metrics for scaling decisions, creating a runtime dependency on the monitoring stack for LLM serving autoscaling.
