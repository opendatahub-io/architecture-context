## Purpose

**Short**: Extracts, enriches, and packages AI model metadata from OCI container registries and HuggingFace into a static YAML catalog shipped as a data-only container image for the RHOAI dashboard.

**Detailed**: model-metadata-collection is a build-time pipeline that produces a data container for Red Hat OpenShift AI. It contains two Go CLI tools — `model-extractor` and `metadata-report` — that run during the build process but are NOT included in the shipped container image. The tools crawl OCI container registries (e.g., `registry.redhat.io`) to discover "modelcar" containers, extract modelcard metadata from their image layers, then enrich that metadata with information from HuggingFace (API details, README content, YAML frontmatter, tags). The enrichment pipeline applies a priority hierarchy: HuggingFace YAML frontmatter > modelcard.md regex extraction > HuggingFace API > registry metadata > generated defaults. Each metadata field tracks its data source for auditability.

The final output is a set of static YAML files (catalog, per-model metadata, manifests index) that are copied into a `ubi9-micro`-based container image. This container runs `sleep infinity` and serves as a data volume — the RHOAI dashboard mounts or reads from it to populate the model catalog UI. The pipeline also supports merging supplemental static catalog entries (pre-defined models that bypass the extraction pipeline) and generates comprehensive metadata quality reports.

## Architectural Analysis

### Data Container Pattern

This component follows a **data container** pattern that is unusual compared to typical Kubernetes services or operators. The Go tooling exists solely at build time — it is never shipped, never runs in production, and has no runtime footprint. The final container image (`ubi9-micro` + static YAML files + `sleep infinity`) acts purely as a data sidecar or init-container data source for the RHOAI dashboard. This pattern decouples the metadata pipeline from the dashboard's deployment lifecycle.

### Multi-Source Enrichment with Source Tracking

The enrichment pipeline implements a sophisticated priority hierarchy across five data sources:

1. **HuggingFace YAML frontmatter** (highest priority) — structured metadata from model card headers
2. **Modelcard.md regex extraction** — unstructured text parsing from model cards embedded in OCI layers
3. **HuggingFace API** — programmatic metadata (downloads, likes, tags, license)
4. **Registry metadata** — OCI manifest timestamps and annotations
5. **Generated defaults** (lowest priority) — algorithmic fallbacks (description from model name, license URL lookup)

Every metadata field in the output records which source it came from via `enrichment.yaml` files, enabling auditability and debugging of data quality issues.

### Model Family Matching with Cross-Family Prevention

Model matching between OCI registry entries and HuggingFace models uses token-based similarity scoring (`CalculateSimilarity` in `pkg/utils/text.go`). A critical safety mechanism prevents cross-family matches — for example, a Granite container will never be matched to a Llama HuggingFace entry, even if their names share tokens. The family definitions are centralized in `internal/config/model_families.go` (10 families: deepseek, gemma, granite, kimi, llama, minimax, mistral, mixtral, phi, qwen) and shared between the normalization and enrichment subsystems.

### Version Normalization for Accurate Matching

Model name normalization (`NormalizeModelName`) uses a two-pass strategy to prevent version number corruption:
1. Dotted versions (e.g., `3.1`) are converted to `3v1` before separator normalization
2. Hyphenated versions in model names (e.g., `granite-3-3`) are converted to `granite-3v3` using a regex derived from the centralized model family list

This prevents `granite-3.1-8b` from matching `granite-3.3-8b` — a subtle but critical correctness property.

### Metadata Format Migration

The codebase supports three metadata formats (`internal/metadata/migration.go`): the current structured format with `OCIArtifact` objects, a legacy format with string artifacts, and a mixed-type format with polymorphic timestamps. The `LoadExistingMetadata` function implements a fallback chain across all three formats, ensuring backward compatibility with previously generated metadata files.

### Supplemental Catalog and Deduplication

The catalog system merges dynamically-extracted models with static supplemental entries from `input/supplemental-catalog.yaml`. Deduplication is case-insensitive by model name. When duplicates are found, models are merged: artifacts are consolidated, and timestamps use the earliest creation date and latest update date. This allows pre-defined models to coexist with dynamically discovered ones without conflicts.

### Design Risks and Considerations

1. **Regex-based metadata extraction is fragile**: The modelcard parser (`internal/metadata/parser.go`) relies on compiled regex patterns to extract provider, description, license, dates, tasks, and languages from free-text model cards. Any change to model card formatting conventions can silently break extraction.

2. **HuggingFace collection discovery uses hardcoded fallbacks**: `internal/huggingface/collections.go` contains hardcoded collection slugs as fallback when API search fails. These become stale as new collections are created and must be manually maintained.

3. **Date parsing ambiguity**: `ParseDateToEpoch` in `pkg/utils/validation.go` tries both M/D/YYYY and D/M/YYYY formats, which means dates like `01/02/2006` could be interpreted as either January 2 or February 1. The first-match-wins strategy favors U.S. date formatting.

4. **No runtime health monitoring**: Since the shipped container runs `sleep infinity`, there are no health checks, liveness probes, or readiness probes. The container is assumed healthy if it starts. Data staleness can only be detected by examining timestamps in the catalog files.
