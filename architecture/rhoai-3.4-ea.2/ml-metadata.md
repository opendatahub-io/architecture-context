## Purpose

**Short**: ML Metadata (MLMD) is a gRPC server and client library for recording and retrieving metadata associated with ML workflows, providing lineage tracking for artifacts, executions, and contexts.

**Detailed**: ML Metadata is a Google-originated library that serves as the metadata backbone for ML pipeline orchestration systems like Kubeflow Pipelines and TFX. It provides a strongly-typed metadata store that tracks three core entity types — Artifacts (data/models), Executions (pipeline steps), and Contexts (grouping constructs like pipeline runs) — along with their relationships through Events (input/output links between artifacts and executions), Attributions (artifact-to-context), and Associations (execution-to-context).

The component ships as a C++ gRPC server binary (`metadata_store_server`) built with Bazel, backed by pluggable database backends (MySQL/MariaDB, PostgreSQL, SQLite). In the RHOAI platform, it is deployed as the `odh-mlmd-grpc-server` container, providing the metadata storage layer that Kubeflow Pipelines (Data Science Pipelines) uses to track pipeline runs, artifacts, and lineage graphs. The server exposes a comprehensive gRPC API with 40+ RPC methods for CRUD operations on types, artifacts, executions, contexts, events, and lineage graph traversal.

The Python client library (`ml-metadata` PyPI package) provides both a direct database connection mode (via pybind11 C++ bindings) and a gRPC client mode for remote server access, with built-in retry logic, SSL/TLS support, and pagination helpers.

## Architectural Analysis

ML Metadata follows a classic Google-style service architecture: a single C++ binary built with Bazel that implements a Protobuf-defined gRPC service backed by a pluggable relational database. The service is stateless itself — all state lives in the database — making it straightforward to deploy and scale. The server's main entry point (`metadata_store_server_main.cc`) supports three configuration modes: config file (protobuf text format), MySQL command-line flags, and PostgreSQL command-line flags, with a fallback to an in-memory SQLite database for development.

The data model is a typed property graph with three node types (Artifact, Execution, Context) and four edge types (Event, Attribution, Association, ParentContext). Types are schema-evolvable: new properties can be added to existing types without breaking backward compatibility, and the system enforces this through `can_add_fields` / `can_omit_fields` flags. The database schema includes migration support with versioned upgrade/downgrade paths, allowing the server to automatically migrate databases from older schema versions.

The query layer is notable for its ZetaSQL-based filter query parser, which allows SQL-like boolean expressions to filter metadata entities. This enables complex queries like filtering artifacts by type, state, custom properties, and related contexts in a single request. The ListOperationOptions provide pagination with cursor-based tokens and configurable ordering.

The Dockerfile.konflux uses a two-stage build: a UBI9 builder stage installs Bazel and compiles the C++ server, then copies the binary into a UBI9-minimal runtime image. The build also extracts the MariaDB Connector/C source from Bazel's external dependencies and includes it as third-party licensing material. The server runs as non-root user (65534:65534) in the production image. The Dockerfile.redhat variant includes a ZetaSQL commit patch for UBI9 compatibility, suggesting upstream ZetaSQL requires modifications to compile on RHEL 9.

A key architectural observation is that this repository does not include any Kubernetes deployment manifests. The MLMD server is deployed as part of the Data Science Pipelines stack, with the dspo (Data Science Pipelines Operator) managing its lifecycle. This means the MLMD server's service exposure, RBAC, network policies, and database connection configuration are all defined outside this repository, in the dspo controller logic and its CR specifications.
