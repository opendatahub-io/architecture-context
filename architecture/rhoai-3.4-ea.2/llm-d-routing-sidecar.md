## Purpose

**Short**: A reverse proxy sidecar that orchestrates disaggregated prefill/decode (P/D) inference by routing requests between prefill workers and a local vLLM decoder.

**Detailed**: The llm-d-routing-sidecar is a Go-based HTTP/HTTPS reverse proxy that runs alongside a vLLM decoder instance in a Kubernetes pod. It intercepts OpenAI-compatible completion requests and orchestrates the disaggregated prefill/decode (P/D) inference pattern used by llm-d. When a request arrives with an `x-prefiller-host-port` header, the sidecar first forwards the request to the specified remote prefill worker, then relays the prefill results (KV cache transfer parameters) to the local decoder for final token generation. When no prefill header is present, requests pass through directly to the local decoder.

The sidecar supports three P/D connector protocols: NIXLv2 (default, actively maintained), NIXLv1 (deprecated), and LMCache (deprecated). It provides TLS termination for inbound connections (with self-signed certificate generation or user-provided certs), optional TLS for outbound connections to prefillers and decoders, and SSRF protection via an InferencePool-based pod allowlist that watches Kubernetes API to maintain a dynamic set of permitted prefill targets.

**Note**: This repository is deprecated. The code has been moved to the [llm-d-inference-scheduler](https://github.com/llm-d/llm-d-inference-scheduler) repository under `cmd/pd_sidecar` and `pkg/sidecar` directories.

## Architectural Analysis

The llm-d-routing-sidecar implements a lightweight sidecar proxy pattern for disaggregated LLM inference. Its core design decision is to run as a co-located reverse proxy alongside each vLLM decoder pod, intercepting OpenAI-compatible API requests and orchestrating the two-phase prefill/decode flow. This avoids modifying vLLM itself and keeps the P/D protocol logic separate from the inference engine. The proxy uses Go's `net/http/httputil.NewSingleHostReverseProxy` for both decoder and prefiller connections, with an LRU cache (capacity 16) for prefiller proxy handlers keyed by host:port to avoid repeated proxy construction.

The P/D protocol implementation supports three connector variants, though only NIXLv2 is actively maintained. In NIXLv2, the sidecar modifies the request body to add `kv_transfer_params` fields, sets `max_completion_tokens=1` for the prefill stage (to trigger KV cache computation without full generation), then extracts the KV transfer parameters from the prefill response and injects them into the decode request. This body manipulation pattern means the sidecar must buffer and re-serialize JSON for every P/D request, which could become a bottleneck for very large request payloads. The server is configured with no ReadTimeout/WriteTimeout to support long-running LLM inference that can take hours for large contexts.

The SSRF protection system is architecturally notable. Rather than static configuration, it uses Kubernetes informers (via dynamic client) to watch InferencePool CRDs from the Gateway API Inference Extension (`inference.networking.x-k8s.io/v1alpha2`). It extracts pod selectors from the InferencePool spec, then watches pods matching those selectors to build a dynamic allowlist of pod IPs and hostnames. This means the allowlist automatically updates as pods scale up/down. The allowlist checks host only (not port), so any port on an allowed host is permitted - this is a deliberate design choice noted in the README but represents a reduced security boundary. The SSRF protection is disabled by default for backward compatibility.

The TLS configuration is thorough: the proxy can terminate TLS on its inbound listener (default enabled, with self-signed cert fallback for development), and independently configure TLS for outbound connections to prefillers and decoders. All TLS configurations enforce TLS 1.2 minimum with a curated set of 6 AEAD cipher suites. However, the `InsecureSkipVerify` flags for prefiller and decoder connections, while disabled by default, could weaken security if enabled in production.

This repository is deprecated and its code has been consolidated into `llm-d-inference-scheduler`. The deployment manifests use a StatefulSet (not a Deployment), which is unusual for a stateless proxy - this may be a carryover from early development when state was anticipated, or to ensure stable pod naming for the exec RBAC rules that reference specific pod names like `${PROJECT_NAME}-0-0`.
