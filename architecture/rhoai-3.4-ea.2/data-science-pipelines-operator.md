## Purpose
**Short**: Kubernetes operator that manages the lifecycle of Data Science Pipelines (DSP) application stacks, deploying and reconciling all sub-components needed to run Kubeflow Pipelines v2 on OpenShift.

**Detailed**: The Data Science Pipelines Operator (DSPO) watches `DataSciencePipelinesApplication` (DSPA) custom resources and, for each one, deploys a complete ML pipeline execution environment. This includes the KFP API server, persistence agent, scheduled workflow controller, Argo Workflow controller, ML Metadata (MLMD) gRPC server with Envoy proxy, a MariaDB database (or external DB connection), and object storage (Minio or external S3).

The operator performs health checks against both the database and object store before deploying pipeline components, and continuously monitors the readiness of each sub-component deployment. It manages TLS certificates for pod-to-pod communication via OpenShift service-ca, creates OpenShift Routes with kube-rbac-proxy sidecars for authenticated external access, deploys NetworkPolicies to restrict inter-component traffic, and exposes Prometheus metrics for monitoring DSPA readiness states.

DSPO is deployed into the `redhat-ods-applications` namespace by the rhods-operator via kustomize overlays. Each DSPA CR instance creates a namespaced pipeline stack, allowing multi-tenant pipeline deployments across the cluster. The operator also supports Kubernetes-native pipeline storage (via Pipeline/PipelineVersion CRDs with webhook validation) as an alternative to database-backed storage.

**Authentication Pattern**: DSPO uses kube-rbac-proxy sidecars for external authentication on both the API Server and MLMD Envoy deployments. The proxy performs `SubjectAccessReview` checks against the `datasciencepipelinesapplications.opendatahub.io` API group, requiring the caller to have access to the specific DSPA resource's `api` subresource. Routes use Reencrypt TLS termination, forwarding traffic to the kube-rbac-proxy's TLS endpoint (8443).
