## Purpose
**Short**: Kubernetes operator that manages the lifecycle of Data Science Pipelines (DSP) infrastructure, deploying and reconciling Kubeflow Pipelines v2 backend components on OpenShift.

**Detailed**: The Data Science Pipelines Operator (DSPO) is a Go-based Kubernetes operator built on controller-runtime that watches `DataSciencePipelinesApplication` (DSPA) custom resources and deploys the full Kubeflow Pipelines v2 backend stack. Each DSPA instance provisions a complete, namespace-scoped pipeline infrastructure including the API server, persistence agent, scheduled workflow controller, ML Metadata (MLMD) service, and optionally an Argo Workflow Controller, along with database (MariaDB) and object storage (Minio) backends.

The operator uses manifestival to apply Go templates from `config/internal/` during reconciliation, dynamically deploying Deployments, Services, Routes, NetworkPolicies, RBAC resources, ConfigMaps, Secrets, and PVCs. It performs health checks against the database (MySQL/MariaDB protocol) and object storage (S3 API) before deploying application components. The operator supports pod-to-pod TLS (enabled by default), FIPS-compliant builds, custom CA bundles, proxy configuration, and Kubernetes-native pipeline storage via validating/mutating webhooks for the `PipelineVersion` CRD. Authentication for external access is provided through kube-rbac-proxy sidecars with OpenShift Route reencrypt TLS termination.

| DS Pipelines Webhook | Deployment (conditional) | Validating/mutating webhook for PipelineVersion CRs when using Kubernetes pipeline store |

**Note**: This operator uses OpenShift Routes with kube-rbac-proxy sidecars for external access authentication (RHOAI pattern). The API server Route points to the kube-rbac-proxy sidecar port (8443), which authenticates via SubjectAccessReview before proxying to the API server (8888). The MLMD Envoy Route similarly uses kube-rbac-proxy for authentication.
