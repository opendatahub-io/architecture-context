## Purpose

**Short**: Kubernetes operator that manages the lifecycle of Data Science Pipelines (Kubeflow Pipelines v2) instances on Red Hat OpenShift AI.

**Detailed**: The Data Science Pipelines Operator (DSPO) is a controller-runtime-based Kubernetes operator that reconciles `DataSciencePipelinesApplication` (DSPA) custom resources. Each DSPA instance describes a complete Data Science Pipelines stack — including the API server, persistence agent, scheduled workflow controller, ML metadata (MLMD) services, workflow controller (Argo Workflows), and backing storage (database and object store).

DSPO uses a template-driven architecture: Go templates in `config/internal/` define Kubernetes manifests for each sub-component, and the operator's reconcile loop renders these templates with parameters extracted from the DSPA spec, then applies them via the Manifestival library. This design allows a single operator to manage multiple independent pipeline instances across namespaces, each with its own database, object store, and pipeline runtime.

The operator also manages health checks for database and object storage connectivity, pod-to-pod TLS certificate aggregation from multiple sources (user-provided, global ODH CA bundle, OpenShift service-CA), webhook infrastructure for Kubernetes-native pipeline storage, and Prometheus metrics for component readiness. It integrates with OpenShift Routes and kube-rbac-proxy for external access and authentication.

## Architectural Analysis

The Data Science Pipelines Operator follows a classic Kubernetes operator pattern with a notable twist: rather than managing a single infrastructure component, it orchestrates an entire multi-tier application stack per DSPA instance. Each DSPA creates 6-8 Deployments, multiple Services, Secrets, ConfigMaps, NetworkPolicies, and RBAC resources — all rendered from Go templates in `config/internal/`. The Manifestival library bridges template rendering and Kubernetes API application, with owner references ensuring garbage collection when a DSPA is deleted. Cluster-scoped resources (ClusterRoleBindings) require explicit finalizer cleanup since they cannot use owner references.

The reconciliation flow has a clear prerequisite gate: database and object storage connectivity must be verified before any other components are deployed. This prevents cascade failures where the API Server starts but cannot reach its database. Health checks use actual MySQL connections and S3 StatObject calls (not just TCP probes), providing meaningful readiness signals. The operator exposes 8 Prometheus gauges — one per sub-component plus an overall "Ready" — enabling fine-grained alerting on component-level failures.

TLS certificate management is particularly sophisticated. The operator aggregates certificates from three sources: user-provided CA bundles (via DSPA spec), the global `odh-trusted-ca-bundle` ConfigMap (injected by the ODH platform for enterprise proxy CAs), and OpenShift service-CA-generated certificates (for pod-to-pod TLS). These are concatenated into a single `dsp-trusted-ca-{name}` ConfigMap and mounted into all components, with `SSL_CERT_DIR` set to include both custom and system certificate paths. This design handles complex enterprise PKI scenarios where multiple CA chains must be trusted.

The operator has an unusually broad RBAC footprint — it requires full access to `kubeflow.org/*`, `argoproj.io/*`, `batch/jobs`, and several ML framework CRDs (KServe, Ray, CodeFlare, Seldon). This is because the managed API Server and Workflow Controller execute user-defined pipelines that may create arbitrary ML workloads. The operator's ServiceAccount permissions are inherited by the pipeline-runner ServiceAccount via delegated Role/RoleBindings created per DSPA instance. The kube-rbac-proxy sidecar pattern provides per-request RBAC enforcement for external access, checking `datasciencepipelinesapplications/api` subresource permissions via SubjectAccessReview.

The webhook infrastructure is conditionally deployed only when `pipelineStore` is set to `"kubernetes"` (as opposed to the default `"database"` mode). When active, it provides mutating and validating admission control for Pipeline and PipelineVersion CRs. The operator carefully manages webhook lifecycle: on DSPA deletion, it only removes webhook resources if no other DSPA instances in the cluster use Kubernetes-native pipeline storage. The webhook runs in the operator's namespace (not the DSPA's namespace) since webhook configurations are cluster-scoped.
