## Purpose

**Short**: Platform operator that deploys, configures, and manages the complete Red Hat OpenShift AI (RHOAI) platform, including all data science components, ingress infrastructure, authentication, monitoring, and lifecycle management.

**Detailed**: The rhods-operator is the central control plane for Red Hat OpenShift AI. It implements a two-tier CRD architecture: `DSCInitialization` (DSCI) handles cluster-wide platform initialization (namespace creation, CA bundle distribution, auth configuration, gateway infrastructure, monitoring), while `DataScienceCluster` (DSC) manages the lifecycle of 14 data science components (Dashboard, Workbenches, KServe, Kueue, Ray, Data Science Pipelines, Model Registry, TrustyAI, Training Operator, Trainer, Feast Operator, LlamaStack Operator, MLflow Operator, Spark Operator).

The operator deploys RHOAI's entire ingress stack using Gateway API, including a central Gateway CR, Envoy-based proxy, EnvoyFilter CRs for authentication enforcement, kube-auth-proxy for OAuth/OIDC authentication, and dashboard redirect services for legacy URL migration. Each data science component is deployed via kustomize manifests bundled into the operator image at build time, with per-component CRDs that track readiness and allow independent lifecycle management.

The operator supports three platform modes (Self-Managed RHOAI, Managed RHOAI/Cloud Service, and Open Data Hub), detecting the mode at startup and adjusting manifests, RBAC, default groups, and monitoring configuration accordingly. It includes a CloudManager subsystem for Helm-based infrastructure provisioning on cloud platforms (Azure, CoreWeave).

## Architectural Analysis

The rhods-operator implements a sophisticated two-tier CRD architecture that cleanly separates platform initialization (DSCI) from component lifecycle (DSC). The DSCI controller runs first, creating the application namespace, deploying network policies, configuring CA bundles, and initializing three service singletons (Auth, GatewayConfig, Monitoring). Only after DSCI reports Ready does the DSC controller begin reconciling components. This ordering is enforced by the DSC controller's `checkPreConditions` action, which blocks if no DSCI exists.

The gateway controller (`internal/controller/services/gateway/`) is the most architecturally significant piece, deploying a full multi-layer ingress stack: GatewayClass → Gateway → Envoy (via Istio/Sail Operator) → EnvoyFilter (ext_authz + Lua) → kube-auth-proxy (OAuth2/OIDC) → component HTTPRoutes. The EnvoyFilter is particularly noteworthy — it uses a Lua filter to strip OAuth2 proxy cookies from upstream requests and inject access tokens as headers, preventing double-authentication issues. The controller supports three auth modes (IntegratedOAuth, OIDC, None/External) detected at runtime via cluster configuration, and conditionally deploys different Deployment manifests and OAuthClient resources accordingly. The DestinationRule for auth proxy traffic uses `insecureSkipVerify: true`, which is acceptable within the Istio mesh but creates a dependency on mesh-level mTLS for security.

Component controllers follow a consistent pattern: each receives a component CRD from the DSC controller's `provisionComponents` action, loads kustomize manifests from the embedded filesystem (prefetched at build time into `/opt/manifests/`), applies platform-specific overlays and image parameter substitutions via `params.env`, and deploys via server-side apply. The `AllowListedFields` RemoverPlugin preserves user customizations to container resources and replica counts during operator upgrades. Seven components (Dashboard, DataSciencePipelines, KServe, ModelRegistry, TrustyAI, MLflow, Workbenches) inject `kube-rbac-proxy` sidecars for metrics endpoint authentication. Components with external dependencies (KServe→Istio/cert-manager, Trainer→JobSet, DSP→Argo, TrustyAI→KServe) implement precondition checks that set degraded conditions rather than failing silently.

The monitoring controller is the second most complex service, deploying an entire observability stack conditionally based on the Monitoring CR spec: MonitoringStack + ThanosQuerier (metrics), TempoMonolithic/TempoStack (traces), OpenTelemetryCollector (collection), and Perses (dashboards). It dynamically registers per-component PrometheusRules when components transition to Ready state, watching the DSC status for component readiness changes. The monitoring stack has atomic deployment semantics — all required CRDs must be present or the entire tier is skipped.

A notable design pattern is the operator's use of `embed.FS` for template files in service controllers (gateway, auth, monitoring). Controller actions render Go templates with runtime data (gateway domain, auth mode, cookie settings, TLS certs) and then apply the resulting resources. This allows complex multi-resource orchestration (the gateway controller manages 15+ resource types) without external manifest dependencies. The dashboard redirect subsystem creates nginx pods + OpenShift Routes to 301-redirect legacy dashboard and gateway URLs to the new Gateway API hostname, ensuring backward compatibility during RHOAI 2.x→3.x migration.

The CloudManager subsystem (`cmd/cloudmanager/`) is architecturally distinct — it uses Helm charts rather than kustomize for infrastructure provisioning on cloud platforms (Azure, CoreWeave). It implements pre-apply and post-apply hooks around Helm releases, enabling cloud-specific setup (e.g., storage class creation, node pool configuration) that must happen before or after the main chart deployment.
