## Purpose
**Short**: Platform operator that orchestrates the lifecycle of Red Hat OpenShift AI (RHOAI) components, platform services, and Gateway API ingress infrastructure on OpenShift clusters.

**Detailed**: The RHOAI operator (rhods-operator) is the central control plane for Red Hat OpenShift AI. It manages the full lifecycle of AI/ML platform components through two primary Custom Resources: `DSCInitialization` (DSCI) for platform-level initialization (namespace setup, monitoring, service mesh, trusted CA bundles) and `DataScienceCluster` (DSC) for enabling/disabling individual AI/ML components (Dashboard, Workbenches, KServe, Data Science Pipelines, Model Registry, TrustyAI, Ray, Kueue, Training Operator, Feast, LlamaStack, MLflow, Spark, and Models-as-a-Service).

In RHOAI 3.x, the operator deploys critical platform ingress infrastructure via a Gateway API controller. This creates a `data-science-gateway` Gateway resource in the `openshift-ingress` namespace, along with `kube-auth-proxy` for OAuth2/OIDC authentication, EnvoyFilter for ext_authz integration, and supporting infrastructure (DestinationRule, NetworkPolicy, HPA). Component controllers throughout the platform create HTTPRoute CRs that reference this parent Gateway for ingress. The operator supports two ingress modes: `OcpRoute` (ClusterIP + OpenShift Routes) and `LoadBalancer` (direct LoadBalancer service). Additionally, the operator creates dashboard redirect infrastructure (nginx-based) that 301-redirects legacy dashboard and gateway URLs to the new Gateway API hostname.

The operator also manages platform services including Auth (RBAC group management), Monitoring (MonitoringStack, Prometheus, Tempo, OpenTelemetry), and certificate/configmap generation. It is built with the `rhoai` build tag and deployed via Konflux CI to the `redhat-ods-operator` namespace.

**Gateway API Architecture (RHOAI 3.x)**:
- The Gateway Controller creates a `data-science-gateway` Gateway in `openshift-ingress` namespace
- GatewayClass: `data-science-gateway-class` using controller `openshift.io/gateway-controller/v1`
- Component controllers across the platform create HTTPRoute CRs referencing this parent Gateway
- Authentication is enforced at the Gateway level via an EnvoyFilter (`data-science-authn-filter`) that calls kube-auth-proxy via ext_authz
- The EnvoyFilter also includes a Lua filter for token forwarding (strips OAuth cookies, sets Bearer authorization header)
- A DestinationRule (`data-science-tls-rule`) configures TLS for upstream service communication
- Two ingress modes: `OcpRoute` (default, ClusterIP + OpenShift Route) and `LoadBalancer` (direct LB)
- Dashboard redirects: nginx Deployment + Service + OpenShift Routes that 301-redirect legacy URLs to the new Gateway hostname
