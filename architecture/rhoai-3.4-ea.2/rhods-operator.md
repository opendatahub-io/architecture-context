## Purpose
**Short**: The RHOAI Operator is the platform operator for Red Hat OpenShift AI, managing the lifecycle of all AI/ML platform components, platform services (gateway, auth, monitoring), and infrastructure resources on OpenShift clusters.

**Detailed**: The rhods-operator (also known as the opendatahub-operator for RHOAI distribution) is a Kubernetes operator built with controller-runtime that serves as the central orchestrator for the Red Hat OpenShift AI platform. It manages two primary cluster-scoped custom resources: `DSCInitialization` (DSCI) which handles one-time platform initialization including monitoring, auth, and trusted CA bundle configuration, and `DataScienceCluster` (DSC) which controls the lifecycle of all AI/ML component sub-operators and their configurations.

The operator implements a hierarchical controller architecture where the DSC controller creates and manages per-component CRs (e.g., Dashboard, Kserve, Workbenches, Ray, etc.) which are themselves reconciled by dedicated component controllers. Platform services such as Gateway (providing ingress via Gateway API), Auth (managing RBAC groups and permissions), and Monitoring (deploying MonitoringStack, Tempo, OpenTelemetry, and Perses for observability) are managed as always-on services independent of user configuration.

In RHOAI 3.x, the operator has migrated from Istio Service Mesh and OpenShift Routes to Gateway API (gateway.networking.k8s.io) for platform ingress. The Gateway controller deploys a `data-science-gateway` Gateway resource in the `openshift-ingress` namespace, along with a `kube-auth-proxy` deployment for centralized authentication (supporting both OpenShift OAuth and external OIDC providers), EnvoyFilter for traffic management, and NetworkPolicies for security. Component controllers create HTTPRoute CRs that reference this gateway for external access.

**Gateway API Architecture (RHOAI 3.x)**:
- The Gateway Controller deploys `data-science-gateway` (Gateway CR) in `openshift-ingress` namespace
- GatewayClass `data-science-gateway-class` uses controller `openshift.io/gateway-controller/v1`
- Gateway accepts HTTPRoutes from `openshift-ingress` and `redhat-ods-applications` namespaces
- `kube-auth-proxy` Deployment provides centralized OAuth2/OIDC authentication
- EnvoyFilter `data-science-authn-filter` intercepts all gateway traffic for auth enforcement
- DestinationRule `data-science-tls-rule` enforces mTLS between gateway Envoy and kube-auth-proxy
- NetworkPolicy restricts kube-auth-proxy access to gateway pods and monitoring namespaces
- Component controllers create HTTPRoute CRs referencing `data-science-gateway` as parent
- Supports two ingress modes: `OcpRoute` (ClusterIP + OpenShift Routes) and `LoadBalancer` (cloud LB)
- Certificate options: OpenShift default ingress cert (propagated), self-signed, or user-provided
