## Purpose

**Short**: KServe is a Kubernetes-native model serving platform that manages the lifecycle of inference services, providing model deployment, scaling, routing, and monitoring for machine learning workloads on OpenShift AI.

**Detailed**: KServe implements the core model serving infrastructure for Red Hat OpenShift AI (RHOAI). It provides a set of Kubernetes Custom Resource Definitions (CRDs) and controllers that automate the deployment and management of ML inference services. The platform supports multiple deployment modes including raw Kubernetes Deployments and Knative serverless, with sophisticated autoscaling via HPA, KEDA, and Workload Variant Autoscaling (WVA).

The repository contains multiple controllers: the primary InferenceService controller manages general-purpose model serving with predictor/transformer/explainer components; the LLMInferenceService controller provides specialized LLM serving infrastructure with scheduler-based routing via the Gateway Inference Extension (InferencePool/EPP pattern), disaggregated prefill/decode, and multi-node GPU deployments via LeaderWorkerSet; the LocalModel controller manages node-level model caching with PersistentVolumes; and the InferenceGraph controller enables DAG-based inference pipelines.

KServe integrates deeply with the RHOAI platform through Gateway API for ingress (HTTPRoute), Istio DestinationRules for mTLS, kube-rbac-proxy for authentication, Prometheus for monitoring, and OpenTelemetry for observability. The Python SDK provides the storage initializer for downloading models from S3, GCS, Azure, HuggingFace Hub, and other backends, along with serving frameworks for custom model servers.

## Architectural Analysis

### Multi-Controller Architecture
KServe uses a multi-binary controller architecture where separate controller processes handle different CRD groups. The main `kserve-controller-manager` handles InferenceService, InferenceGraph, TrainedModel, ServingRuntime, and LocalModel resources, while the `llmisvc-controller-manager` handles LLMInferenceService and LLMInferenceServiceConfig. Both share the same codebase but are built as separate binaries with feature flags (`ENABLE_ISVC_CONTROLLER`, `ENABLE_LLMISVC_CONTROLLER`). This separation allows independent scaling and reduces blast radius of controller failures.

### Ingress Pattern Evolution (2.x to 3.x)
The codebase contains three parallel ingress implementations reflecting the RHOAI platform evolution:
1. **Istio VirtualService** (legacy): Used for Knative/ModelMesh modes and upstream Istio deployments
2. **Kubernetes Ingress**: Generic ingress for non-OpenShift environments
3. **Gateway API HTTPRoute** (RHOAI 3.x primary): Used for both InferenceService and LLMInferenceService, with the platform Gateway deployed by rhods-operator

The `security.opendatahub.io/enable-auth` annotation controls whether kube-rbac-proxy (3.x) or oauth-proxy (2.x) is injected as the authentication sidecar. In the RHOAI 3.x overlay, kube-rbac-proxy is the default.

### LLMISvc Scheduler Architecture (EPP Pattern)
LLMInferenceService introduces a sophisticated scheduling layer for LLM workloads. The controller deploys a scheduler pod (EPP - Endpoint Picker) alongside workload pods. The scheduler watches InferencePool endpoints and uses the Gateway Inference Extension to route requests to optimal backends based on load, token counts, and other metrics. The tokenizer sidecar communicates with the scheduler via Unix Domain Socket (UDS) to provide token-aware request analysis. The architecture supports both InferencePool v1 (`inference.networking.k8s.io`) and v1alpha2 (`inference.networking.x-k8s.io`) with automatic migration.

### Autoscaling Strategy Stack
KServe provides a layered autoscaling strategy:
1. **HPA**: Basic CPU/memory autoscaling for standard workloads
2. **KEDA ScaledObject**: External metrics (Prometheus, custom) for advanced scaling
3. **Workload Variant Autoscaling (WVA)**: LLM-specific autoscaling from the llm-d project that computes desired replicas from workload-specific metrics, using HPA or KEDA as the actuator to enforce computed replica counts
4. **Knative Autoscaler**: Scale-to-zero capability in serverless deployment mode

### Pod Mutation Webhook Pipeline
The pod mutation webhook (`/mutate-pods`) implements a chainable injection pipeline that modifies inference pods at admission time. The pipeline injects: GKE accelerator node selectors, storage-initializer init containers, Istio CNI security context adjustments, agent sidecars (model pulling/batching/logging), metrics aggregator configuration, and optional modelcar sidecars. Each injector checks pod annotations to determine whether injection is needed, enabling selective sidecar composition.

### Configuration Management Complexity
Configuration flows through multiple layers: the `inferenceservice-config` ConfigMap provides global defaults, ServingRuntime/ClusterServingRuntime CRDs define model server templates, LLMInferenceServiceConfig CRDs provide LLM-specific templates with `baseRefs` for hierarchical merging, and the ODH overlay's configMapGenerator + replacements system (params.env) injects all image references at deployment time. Template variable substitution (`ReplaceVariables()`) in LLMISvc config allows dynamic values from the service spec to be injected into configuration templates.

### FIPS Compliance Gap
Four of the seven container images (controller, llmisvc-controller, agent, router) are FIPS-enabled with `GOEXPERIMENT=strictfipsruntime` and `CGO_ENABLED=1`. However, the localmodel-controller and localmodel-agent are built with `CGO_ENABLED=0` on distroless base images, meaning they use Go's pure-Go crypto which is not FIPS-validated. The storage-initializer (Python) inherits OpenSSL from the UBI9 base image but has no explicit FIPS mode enforcement. Additionally, the LLMISvc controller creates Istio DestinationRules with `InsecureSkipVerify=true` for scheduler and shadow service TLS, which bypasses certificate validation.

### Build Hermeticity Gaps
Only 1 of 7 Dockerfiles (`localmodel-agent.Dockerfile.konflux`) sources `cachi2.env` for hermetic Go dependency prefetch. The remaining 6 Dockerfiles download dependencies during build time, creating a supply chain risk in air-gapped environments. No `rpms.lock.yaml` or `artifacts.lock.yaml` files exist on this EA branch. Go dependencies are locked via `go.sum` and Python dependencies via `uv.lock`, but RPM-level hermeticity is absent.
