## Purpose

**Short**: A lightweight Rust HTTP service that detects PII (personally identifiable information) and custom patterns in text using regular expressions, serving as a detector backend for the FMS Guardrails Orchestrator.

**Detailed**: The guardrails-regex-detector is a stateless HTTP microservice built with Rust (Axum framework) that provides text content scanning via regular expression matching. It is designed to integrate with the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator) as one of potentially many detector backends in the RHOAI TrustyAI / Guardrails subsystem.

The service exposes a single detection endpoint (`POST /api/v1/text/contents`) that accepts a list of text strings and a set of regex patterns (either built-in named patterns or custom regex). Built-in patterns detect common PII types: email addresses, US Social Security numbers, and credit card numbers. When a custom regex is provided that doesn't match a built-in name, it is compiled and evaluated as a raw regular expression. All matches are returned with positional offsets, matched text, detection type, and a confidence score of 1.0 (deterministic regex matching).

The service runs as a single-binary container on a UBI 9 Minimal base image, listening on a configurable HTTP port (default 8080). It has no external dependencies — no database, no state, no upstream API calls — making it horizontally scalable and easy to deploy as a sidecar or standalone service within the guardrails pipeline.

## Architectural Analysis

The guardrails-regex-detector is one of the simplest components in the RHOAI platform — a single-purpose Rust microservice with approximately 170 lines of source code across two files. Its architectural simplicity is intentional: it serves as a stateless, high-performance regex evaluation engine that can be horizontally scaled without coordination.

The service uses the Axum web framework with Tokio's multi-threaded async runtime, which provides efficient request handling for I/O-bound workloads. The `regex` crate handles pattern compilation and matching, which is well-suited for the use case since Rust's regex engine guarantees linear-time matching (no catastrophic backtracking). This is a deliberate safety property for a service that accepts user-provided regex patterns via the `detector_params.regex` field — a service accepting arbitrary regexes in most other languages would be vulnerable to ReDoS (Regular Expression Denial of Service).

The service supports a hybrid detection model: three built-in named patterns (`email`, `ssn`, `credit-card`) map to pre-defined regex patterns with PII-specific detection types, while any other string in the regex list is treated as a raw custom regex and evaluated with a generic "custom" detection type. This design allows the orchestrator to use well-known pattern names for common PII categories while also supporting ad-hoc content policies. However, the built-in patterns are hardcoded in `src/detectors.rs` — there is no configuration file or environment variable to add new named patterns without code changes.

A notable gap is the absence of any authentication, authorization, or TLS at the application level. The service binds to plaintext HTTP on all interfaces (`0.0.0.0:8080`). This is consistent with the RHOAI 3.x architecture where security is enforced at the platform level (kube-rbac-proxy sidecars, NetworkPolicy, service mesh), but it means the service is fully open to any client that can reach its pod IP. The absence of a `Dockerfile.konflux` file and the lack of Hermeto/cachi2 prefetch integration suggest this component may be in an earlier stage of Konflux onboarding compared to other RHOAI components, or that its build pipeline is managed separately.
