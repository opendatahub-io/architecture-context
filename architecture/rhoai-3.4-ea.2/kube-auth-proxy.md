## Purpose
**Short**: A FIPS-compliant authentication and RBAC proxy designed as a drop-in replacement for oauth-proxy sidecars in OpenShift AI (RHOAI) and Open Data Hub (ODH) environments.

**Detailed**: kube-auth-proxy is a focused authentication proxy derived from oauth2-proxy that has been streamlined to support only OIDC and OpenShift OAuth authentication providers. It acts as a reverse proxy that intercepts requests to backend applications, authenticates users through external OIDC providers or OpenShift's built-in OAuth service, and forwards authenticated requests to upstream services with appropriate user identity headers. The project places a strong emphasis on FIPS compliance and maintains argument compatibility with the legacy oauth-proxy for seamless migration.

The repository ships a single container image containing three binaries: `kube-auth-proxy` (the authentication proxy, forked from oauth2-proxy), `kube-rbac-proxy` (a Kubernetes RBAC authorization proxy using SubjectAccessReview), and an `entrypoint` binary that dispatches to either mode based on the `PROXY_MODE` environment variable or a command-line argument. The default mode is `rbac`, running kube-rbac-proxy. This dual-mode design allows a single container image to serve both authentication-only and RBAC-based authorization use cases across the RHOAI platform, replacing the need for separate oauth-proxy and kube-rbac-proxy images.

kube-auth-proxy also supports Kubernetes service account token validation via the TokenReview API, enabling service accounts to authenticate alongside human users. It is compatible with Envoy's ext_authz external authorization framework, allowing integration with Gateway API-based ingress architectures.

**Sidecar Deployment Pattern (RHOAI 3.x)**:
- In RHOAI 3.x, component controllers inject kube-auth-proxy as a sidecar container
- The sidecar runs in `rbac` mode (kube-rbac-proxy) by default, or `auth` mode (kube-auth-proxy) when PROXY_MODE=auth
- In `rbac` mode: listens on a TLS-secured port (typically 8443/TCP), authenticates via Bearer Token or X.509 client certificates, authorizes via SubjectAccessReview, then proxies to the application container
- In `auth` mode: listens on 4180/TCP (HTTP) or 443/TCP (HTTPS), authenticates via OIDC/OpenShift OAuth, then proxies to the application container
- Component HTTPRoute CRs route traffic through the platform Gateway to the sidecar port
