## Purpose

**Short**: A FIPS-compliant authentication and authorization reverse proxy that provides OIDC, OpenShift OAuth, and Kubernetes RBAC enforcement for RHOAI platform components.

**Detailed**: kube-auth-proxy is a unified authentication proxy image that bundles two complementary proxies into a single container: **kube-auth-proxy** (derived from oauth2-proxy) for OAuth2/OIDC authentication flows, and **kube-rbac-proxy** (derived from brancz/kube-rbac-proxy) for Kubernetes-native RBAC authorization via SubjectAccessReview. An entrypoint binary selects which proxy to run at container startup based on the `PROXY_MODE` environment variable or a positional CLI argument (`auth` or `rbac`, defaulting to `rbac`).

The kube-auth-proxy component is deployed as a sidecar container in RHOAI 3.x to replace both the legacy `oauth-proxy` sidecar (for authentication) and the standalone `kube-rbac-proxy` sidecar (for RBAC enforcement). It serves as the authentication/authorization enforcement point in the RHOAI Gateway API ingress architecture, where it runs alongside application containers to intercept and validate requests before they reach the upstream application. The proxy supports Envoy ext_authz integration, making it compatible with the RHOAI 3.x EnvoyFilter-based ingress stack. It is built with FIPS-compliant compilation flags (`CGO_ENABLED=1`, `GOEXPERIMENT=strictfipsruntime`, `-tags strictfipsruntime`) and runs on UBI 9 base images.

## Architectural Analysis

The kube-auth-proxy repository represents a strategic consolidation of two distinct authentication/authorization proxies into a single container image, driven by RHOAI 3.x's migration from OpenShift Route-based ingress to Gateway API-based ingress. The dual-binary approach (kube-auth-proxy + kube-rbac-proxy) behind a mode-selecting entrypoint is pragmatic: it allows the same image to serve different roles across the platform without doubling the container image footprint. The `PROXY_MODE` environment variable or positional argument selects which binary runs, with `rbac` as the default to maintain backward compatibility with existing kube-rbac-proxy deployments.

The kube-auth-proxy binary (auth mode) is a substantial fork of oauth2-proxy, stripped down to support only OIDC and OpenShift OAuth providers. The removal of ~15 other providers (Google, GitHub, Azure, GitLab, etc.) significantly reduces the attack surface, which is an appropriate security tradeoff for an enterprise Kubernetes platform. The authentication middleware chain is carefully ordered: K8s token validation runs first (for service account tokens), then JWT/OAuth session loaders (for bearer tokens), then stored session loader (for cookie-based browser sessions). This ordering ensures that programmatic API access (service accounts, JWT bearers) is validated before the heavier cookie-based session management kicks in.

The Envoy ext_authz integration is architecturally significant. The `/oauth2/auth` endpoint returns HTTP 202 (Accepted) for authenticated requests, which is the expected success code for Envoy's ext_authz filter. On CSRF validation failure during OAuth callbacks, the proxy redirects (302) rather than returning 403, because ext_authz would forward the raw 403 HTML to the client instead of rendering it in the browser. This is a subtle but important design decision documented in `oauthproxy.go:596-615` that ensures graceful session expiration handling in the Gateway API architecture.

The kube-rbac-proxy component maintains its upstream architecture as a Kubernetes-native authorization proxy with a three-layer authorizer chain: (1) hardcoded metrics authorizer (allows `openshift-monitoring/prometheus-k8s` to scrape `/metrics`), (2) static YAML-based authorization rules, and (3) delegated SubjectAccessReview authorization. The hardcoded Prometheus allowlist is a pragmatic shortcut for OpenShift monitoring integration but couples the proxy to OpenShift's monitoring namespace conventions. The static authorizer supports templating with HTTP headers and query parameters (`{{ .Value }}` syntax), enabling dynamic resource-scoped authorization without hardcoding resource names.

The FIPS compliance story is thorough at the build level: `CGO_ENABLED=1`, `GOEXPERIMENT=strictfipsruntime`, and `-tags strictfipsruntime` are consistently applied across all three binaries (kube-auth-proxy, kube-rbac-proxy, entrypoint) in the Konflux Dockerfile. The UBI 9 minimal base image provides OpenSSL. However, the application-level crypto surface has a few notable points: the `InsecureSkipVerify` option is exposed as a configurable flag (`--ssl-insecure-skip-verify`), and the cipher suite parser includes `tls.InsecureCipherSuites()` in its lookup table (though this is for configuration flexibility, not default usage). The cookie encryption uses AES CFB mode with random IVs, which is FIPS-compatible when backed by the strictfipsruntime OpenSSL integration.

The Go workspace (`go.work`) manages the two modules (root and kube-rbac-proxy) as a unified development environment, while each module has its own `go.mod` and `go.sum`. The kube-rbac-proxy module still references `github.com/brancz/kube-rbac-proxy` as its module path, indicating it is tracked as a separate upstream project embedded within this repository rather than fully integrated into the kube-auth-proxy module namespace.
