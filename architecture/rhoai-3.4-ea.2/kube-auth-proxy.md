## Purpose
**Short**: A FIPS-compliant authentication reverse proxy for RHOAI/ODH that supports OIDC and OpenShift OAuth, designed as a drop-in replacement for oauth-proxy sidecars.

**Detailed**: kube-auth-proxy is a focused authentication proxy derived from oauth2-proxy, streamlined to support only OIDC and OpenShift OAuth authentication providers. It is designed specifically for OpenShift Data Hub (ODH) and Red Hat OpenShift AI (RHOAI) environments, serving as a drop-in replacement for existing oauth-proxy sidecars used for authentication.

The container image ships two binaries: `kube-auth-proxy` (authentication proxy) and `kube-rbac-proxy` (RBAC authorization proxy based on SubjectAccessReview). An entrypoint binary selects between these two modes via the `PROXY_MODE` environment variable or command-line argument (`auth` or `rbac`, defaulting to `rbac`). This dual-binary design allows the same container image to serve both authentication-only and RBAC-authorization use cases across the platform.

The proxy supports Kubernetes service account token validation via the TokenReview API (independent of the configured provider), Envoy ext_authz compatibility, cookie-based and Redis-based session storage, and is built with FIPS-compliant cryptographic libraries using `GOEXPERIMENT=strictfipsruntime`.

### Ingress

kube-auth-proxy does not define its own ingress resources. It is deployed as a sidecar that fronts upstream application containers. In the RHOAI 3.x architecture:

- Component controllers (e.g., notebook controller) create HTTPRoute CRs referencing the platform Gateway (e.g., "data-science-gateway")
- The HTTPRoute targets the parent Service which routes to the kube-auth-proxy sidecar
- kube-auth-proxy authenticates requests before forwarding to the upstream application container
