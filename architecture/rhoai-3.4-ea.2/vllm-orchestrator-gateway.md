## Purpose
**Short**: An HTTP gateway that provides OpenAI-compatible chat completion endpoints with configurable guardrail detector-based filtering by routing requests to the FMS Guardrails Orchestrator.

**Detailed**: The vLLM Orchestrator Gateway is a Rust-based HTTP service that sits between clients and the FMS Guardrails Orchestrator, providing controlled access to guardrailed LLM inference. It enables administrators to configure named detector pipelines (e.g., PII detection, content safety) and exposes unique `/v1/chat/completions` endpoints per configured pipeline. This allows drop-in replacement of unguardrailed chat completion models with guardrailed versions using OpenAI-compatible API semantics.

The gateway reads a YAML configuration file that defines the upstream orchestrator location, available detectors (with input/output filtering directions and parameters), and named routes that map to specific detector combinations. Each route creates a unique HTTP endpoint (e.g., `/{route_name}/v1/chat/completions`) that injects the configured detector specifications into requests before proxying them to the orchestrator. When detectors flag content, the gateway can substitute a configurable fallback message. The gateway supports both standard request-response and Server-Sent Events (SSE) streaming modes.

The service supports mTLS for communication with the orchestrator when TLS certificates are available at well-known paths (`/etc/tls/private/`), and forwards `Authorization` and `X-Forwarded-*` headers from clients to the upstream orchestrator.

## Security

### TLS / mTLS Configuration

The gateway supports optional mTLS for communication with the upstream FMS Guardrails Orchestrator:

| Certificate Path | Type | Purpose |
|------------------|------|---------|
| `/etc/tls/private/tls.crt` | Client certificate (PEM) | Client identity for mTLS to orchestrator |
| `/etc/tls/private/tls.key` | Private key (PEM) | Private key for client certificate |
| `/etc/tls/ca/service-ca.crt` | CA certificate (PEM) | Custom CA for verifying orchestrator's server certificate |

- When TLS cert + key are present: uses HTTPS with client certificate (mTLS)
- When only CA cert is present: adds custom CA to trust store but uses HTTP
- When no certs are present: uses plaintext HTTP
- When connecting to `localhost` with custom CA: accepts invalid hostnames (development convenience)

## Data Flows

### Flow 1: Non-Streaming Chat Completion with Detection

| Step | Source | Destination | Port | Protocol | Encryption | Auth |
|------|--------|-------------|------|----------|------------|------|
| 1 | Client | vllm-orchestrator-gateway | 8090/TCP | HTTP | None | Authorization header (pass-through) |
| 2 | vllm-orchestrator-gateway | FMS Guardrails Orchestrator | 8085/TCP (configurable) | HTTP/HTTPS | Optional mTLS | Client cert + forwarded Authorization |
| 3 | FMS Guardrails Orchestrator | vLLM / LLM Backend | Configured in orchestrator | HTTP/HTTPS | Configured in orchestrator | Configured in orchestrator |
| 4 | FMS Guardrails Orchestrator | Detector Services | Configured in orchestrator | HTTP/HTTPS | Configured in orchestrator | Configured in orchestrator |
| 5 | FMS Guardrails Orchestrator | vllm-orchestrator-gateway | - | HTTP/HTTPS | Optional mTLS | - |
| 6 | vllm-orchestrator-gateway | Client | - | HTTP | None | - |

**Detection Logic**: If detections are found in the orchestrator response and a `fallback_message` is configured for the route, the gateway replaces the LLM-generated content with the fallback message.

### Flow 2: Streaming Chat Completion with Detection (SSE)

| Step | Source | Destination | Port | Protocol | Encryption | Auth |
|------|--------|-------------|------|----------|------------|------|
| 1 | Client | vllm-orchestrator-gateway | 8090/TCP | HTTP | None | Authorization header (pass-through) |
| 2 | vllm-orchestrator-gateway | FMS Guardrails Orchestrator | 8085/TCP (configurable) | HTTP/HTTPS (SSE) | Optional mTLS | Client cert + forwarded Authorization |
| 3 | FMS Guardrails Orchestrator | vllm-orchestrator-gateway | - | HTTP/HTTPS (SSE stream) | Optional mTLS | - |
| 4 | vllm-orchestrator-gateway | Client | - | HTTP (SSE) | None | - |

**Streaming Detection Logic**: Each SSE chunk is inspected for detections. When a detection is found and a `fallback_message` is configured, the gateway modifies the chunk's delta content to the fallback message and sets `finish_reason: "stop"`.

## Configuration

### Gateway Configuration (YAML)

The gateway is configured via a YAML file (default: `config/config.yaml`, overridden by `GATEWAY_CONFIG` env var):

| Section | Field | Type | Required | Default | Purpose |
|---------|-------|------|----------|---------|---------|
| `orchestrator` | `host` | string | No | `localhost` | Orchestrator hostname |
| `orchestrator` | `port` | integer | No | `8032` | Orchestrator port |
| `detectors[]` | `name` | string | Yes | - | Detector identifier |
| `detectors[]` | `server` | string | No | Same as `name` | Server name for detector (allows multiple detectors on same server) |
| `detectors[]` | `input` | boolean | Yes | - | Apply detector to input messages |
| `detectors[]` | `output` | boolean | Yes | - | Apply detector to output messages |
| `detectors[]` | `detector_params` | object | No | - | Detector-specific parameters |
| `routes[]` | `name` | string | Yes | - | Route name (becomes URL path prefix) |
| `routes[]` | `detectors` | string[] | Yes | - | List of detector names to apply |
| `routes[]` | `fallback_message` | string | No | - | Message to return when detection is triggered |

### Environment Variables

| Variable | Default | Purpose |
|----------|---------|---------|
| `GATEWAY_CONFIG` | `config/config.yaml` | Path to gateway configuration file |
| `HTTP_PORT` | `8090` | HTTP listen port |
| `HOST` | `0.0.0.0` | HTTP listen address |

### Configuration Validation

At startup, the gateway validates:
1. All detectors referenced in routes exist in the detectors list
2. No route has multiple input detectors with the same server name
3. No route has multiple output detectors with the same server name

Validation failures cause a panic with detailed error messages.
