## Purpose
**Short**: An OpenAI-compatible HTTP gateway that routes chat completion requests through configurable detector-based content filtering pipelines via the FMS Guardrails Orchestrator.

**Detailed**: The vLLM Orchestrator Gateway is a lightweight Rust-based HTTP reverse proxy service that sits in front of the FMS Guardrails Orchestrator. It provides multiple OpenAI-compatible `/v1/chat/completions` endpoints, each mapped to a named route with a preconfigured set of content detectors (e.g., PII detection, regex-based filtering). This enables drop-in replacement of an unguarded chat completion model endpoint with a guardrailed one.

When a request arrives at a route-specific endpoint (e.g., `/{route_name}/v1/chat/completions`), the gateway injects the route's configured detector specifications into the request payload and forwards it to the FMS Guardrails Orchestrator's `/api/v2/chat/completions-detection` endpoint. If detections are found in the orchestrator's response and a fallback message is configured for that route, the gateway replaces the generated content with the fallback message. The gateway supports both non-streaming and Server-Sent Events (SSE) streaming responses, and can be configured with mTLS for secure communication with the orchestrator.

**mTLS Configuration**: The gateway automatically detects TLS certificates at fixed paths and enables HTTPS + mTLS when present:
- Client certificate: `/etc/tls/private/tls.crt`
- Client private key: `/etc/tls/private/tls.key`
- CA certificate: `/etc/tls/ca/service-ca.crt`

When TLS certificates are found, the scheme switches from `http://` to `https://` for orchestrator communication. The CA certificate is added as a trusted root, and the client cert/key pair is used for mTLS identity.
