## Purpose

**Short**: An HTTP gateway that provides multiple OpenAI-compatible chat completion endpoints with configurable detector-based content filtering, routing requests to the FMS Guardrails Orchestrator.

**Detailed**: The vLLM Orchestrator Gateway is a Rust-based HTTP service that acts as a routing layer between client applications and the FMS Guardrails Orchestrator. It enables controlled interaction with LLM inference services by enforcing configurable content safety policies through detector pipelines. Each configured route exposes a unique `/v1/chat/completions` endpoint that applies specific detector configurations (e.g., PII detection, content moderation) before forwarding requests to the orchestrator.

The gateway reads a YAML configuration file that defines the orchestrator backend, available detectors (with input/output filtering and detector-specific parameters), and named routes that bind detectors to endpoints. When a detection is triggered, the gateway can replace the LLM response with a configurable fallback message. The gateway supports both standard (non-streaming) and Server-Sent Events (SSE) streaming chat completions, maintaining OpenAI API compatibility throughout. It also supports mTLS for secure communication with the orchestrator backend when TLS certificates are mounted.

This component is part of the TrustyAI ecosystem within RHOAI, providing guardrail enforcement at the API gateway layer to ensure LLM outputs comply with organizational content safety policies.

## Architectural Analysis

The vllm-orchestrator-gateway follows a simple but effective reverse proxy pattern. Rather than implementing guardrail logic directly, it acts purely as a routing and configuration layer — translating route-specific detector policies into the orchestrator's API format. This separation of concerns keeps the gateway lightweight (three Rust source files totaling ~900 lines) while allowing complex detection pipelines to be managed by the orchestrator.

The configuration model is well-designed for multi-tenant or multi-policy deployments. Each route in the YAML config creates an independent `/v1/chat/completions` endpoint with its own detector pipeline and fallback behavior. Routes can reference different combinations of input and output detectors, and a `passthrough` route with no detectors provides an unfiltered endpoint. The validation logic in `config.rs` catches misconfigured routes at startup (referencing undefined detectors, duplicate server assignments), which is a good fail-fast practice.

The TLS implementation in `build_orchestrator_client` (`main.rs:351-406`) uses OpenSSL bindings for PKCS#12 conversion, which is an unusual but workable approach — the `native-tls` crate's `Identity` type requires PKCS#12 format, so the code loads PEM cert/key via the `openssl` crate, builds a PKCS#12 archive in memory, and passes it to `reqwest`'s TLS configuration. The `danger_accept_invalid_hostnames(true)` call when the orchestrator host is `localhost` is a development convenience that should not affect production deployments (where the orchestrator hostname would be a Kubernetes service DNS name).

One notable architectural limitation is the absence of health check or readiness endpoints. The gateway starts serving traffic immediately after binding to the socket, with no `/healthz` or `/readyz` endpoint for Kubernetes probes. This means the platform operator or deployment configuration must define appropriate probe strategies (e.g., TCP socket probes on port 8090) rather than HTTP probes. Additionally, the gateway has no built-in metrics endpoint (e.g., Prometheus `/metrics`), which limits observability in production deployments.

The Konflux build pipeline produces multi-architecture images (x86_64, arm64, ppc64le, s390x) from a UBI 9 minimal base image, which aligns with RHOAI's cross-platform support requirements. The build is explicitly non-hermetic (`hermetic: false`), which is atypical for production RHOAI components and may indicate this component is still maturing in the release pipeline.
