## Purpose

**Short**: A lightweight REST API service for orchestrating LLM evaluations across multiple backend frameworks on OpenShift.

**Detailed**: EvalHub is a Go-based REST API service that orchestrates Large Language Model (LLM) evaluations across multiple backend frameworks including lm-evaluation-harness, RAGAS, Garak, GuideLLM, LightEval, and MTEB. It provides a unified API for creating evaluation jobs, managing benchmark providers and collections, and tracking evaluation results through MLflow experiment tracking.

The service uses Go's standard `net/http` router with a layered middleware architecture for authentication (Kubernetes TokenReview), authorization (SubjectAccessReview), metrics (Prometheus), and distributed tracing (OpenTelemetry). It supports two runtime backends: a Kubernetes runtime that creates batch Jobs for benchmark execution, and a local runtime for development. Storage is pluggable, supporting SQLite for development and PostgreSQL for production, both using a JSON document store pattern (JSONB/TEXT columns).

EvalHub is deployed on OpenShift as a managed component of the TrustyAI stack. It is managed by the TrustyAI Service Operator via an `EvalHub` custom resource (`trustyai.opendatahub.io/v1alpha1`). The service also ships as a Python wheel (`eval-hub-server`) that bundles the Go binary, enabling Python developers to use it as a local development server without container infrastructure.

## Architectural Analysis

EvalHub follows a clean service architecture with well-defined layers: HTTP handlers, storage abstractions, runtime abstractions, and external integrations. The use of Go interfaces for both storage (`abstractions.Storage`) and runtime (`abstractions.Runtime`) enables clean testing and extensibility. The storage layer uses a JSON document store pattern where entities are serialized as JSONB (PostgreSQL) or TEXT (SQLite) columns, with SQL-level filtering via JSON path operators. This avoids ORM complexity while still supporting rich filtering with AND/OR operators on nested JSON fields.

The Kubernetes runtime is the most architecturally significant component. When a user creates an evaluation job, EvalHub resolves the benchmarks (either direct or via collection), then spawns a Kubernetes batch Job per benchmark with a ConfigMap containing the job specification as `job.json`. Each Job runs a provider-specific adapter container image (e.g., `quay.io/opendatahub/ta-lmes-job:odh-3.4-ea2` for lm-evaluation-harness) that reads the spec, executes the benchmark, and reports status back to EvalHub via the `/api/v1/evaluations/jobs/{jobId}/events` callback endpoint. This callback-based architecture decouples the server from the adapter lifecycle — the server only creates resources and receives status updates. Resource cleanup uses Kubernetes OwnerReferences (ConfigMap owned by Job) and TTL-based Job cleanup (3600 seconds).

The authentication and authorization system is noteworthy for its use of Kubernetes-native mechanisms. Rather than implementing custom auth, EvalHub delegates token validation to the Kubernetes API server via `TokenReview` (authenticatorfactory.DelegatingAuthenticator) and authorization to `SubjectAccessReview` (authorizerfactory.DelegatingAuthorizer). The auth configuration maps each API endpoint to Kubernetes resource attributes (API group, resource, verb), so standard Kubernetes RBAC policies control access to EvalHub's API. This means cluster administrators can use standard `Role` and `ClusterRole` resources to grant or restrict access to evaluation operations. The auth config supports Go template syntax for dynamic namespace resolution from request headers or query parameters, enabling multi-tenant deployments where different namespaces have different RBAC policies.

The MLflow integration uses a builder-pattern client with support for per-request token rotation from projected ServiceAccountToken volumes, which is critical for ROSA/STS clusters where static tokens are not available. The TLS configuration enforces a minimum of TLS 1.2 and supports custom CA certificates, which is necessary for clusters using self-signed certificates for internal services. The `InsecureSkipVerify` option exists as an escape hatch but defaults to `false`.

The Python wheel distribution (`eval-hub-server`) is an interesting packaging choice — it bundles the compiled Go binary inside a Python wheel, allowing Python ML developers to run the server locally via `pip install` without needing Go toolchain or container infrastructure. The wheel is built for multiple platforms (Linux amd64/arm64, macOS amd64/arm64, Windows amd64) using cross-compilation. This component is not deployed in production clusters.
