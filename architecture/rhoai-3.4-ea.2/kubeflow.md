## Purpose
**Short**: Extends the upstream Kubeflow Notebook Controller with OpenShift-specific features including Gateway API ingress, kube-rbac-proxy authentication sidecars, certificate management, network policies, and Elyra pipeline integration.

**Detailed**: The ODH Notebook Controller is a Kubernetes controller that watches `Notebook` custom resources (from the upstream Kubeflow `kubeflow.org` API group) and reconciles OpenShift/RHOAI-specific resources for each notebook workbench. It is deployed alongside the upstream Kubeflow Notebook Controller and provides additional functionality required for RHOAI.

In RHOAI 3.x, the controller implements the Gateway API ingress pattern. For each notebook, it creates `HTTPRoute` resources in a central namespace (the controller's own namespace) that reference the platform's `data-science-gateway` Gateway in the `openshift-ingress` namespace. Cross-namespace backend references are enabled through `ReferenceGrant` resources created in each user namespace. When authentication is enabled via the `notebooks.opendatahub.io/inject-auth` annotation, the controller creates kube-rbac-proxy sidecar infrastructure (Service, ConfigMap, ClusterRoleBinding) and the mutating webhook injects the kube-rbac-proxy sidecar container into the notebook pod.

The controller also manages network policies per notebook, trusted CA certificate bundles, pipeline runtime image ConfigMaps (from ImageStreams), Elyra pipeline configuration secrets (integrating with Data Science Pipelines), Feast feature store configuration mounting, and legacy OAuthClient cleanup for migrations from RHOAI 2.x to 3.x.

| nb-{namespace}-{notebook} (with auth) | HTTPRoute (Gateway API) | data-science-gateway (openshift-ingress) | 8443/TCP | HTTPS | TLS terminated at Gateway + mTLS to kube-rbac-proxy | /notebook/{namespace}/{notebook-name} | External via Gateway |

**Key architectural notes:**
- HTTPRoutes are created in the **controller's central namespace** (not the user namespace) for security isolation
- ReferenceGrants in user namespaces allow cross-namespace backend references from HTTPRoutes to notebook Services
- The data-science-gateway Gateway (in openshift-ingress) is deployed by rhods-operator's gateway controller
- Gateway name/namespace are configurable via NOTEBOOK_GATEWAY_NAME and NOTEBOOK_GATEWAY_NAMESPACE environment variables

### Sidecar Containers (injected by webhook)

| Container | Port | Protocol | Encryption | Auth | Purpose |
|-----------|------|----------|------------|------|---------|
| kube-rbac-proxy | 8443/TCP | HTTPS | TLS (service-ca signed cert) | TokenReview + SubjectAccessReview (RBAC on kubeflow.org/notebooks GET) | Authentication/authorization proxy fronting notebook (8888/TCP) |

The kube-rbac-proxy sidecar:
- Listens on 0.0.0.0:8443 with TLS
- Proxies authenticated requests to http://127.0.0.1:8888/ (notebook container)
- Health endpoint on port 8444/TCP (/healthz, HTTPS)
- Uses ConfigMap {notebook-name}-kube-rbac-proxy-config for authorization policy
- Uses Secret {notebook-name}-kube-rbac-proxy-tls for TLS certificates (provisioned by OpenShift service-ca)
- Forwards authenticated user info via X-Auth-Request-User and X-Auth-Request-Groups headers
