## Purpose

**Short**: Manages the lifecycle of Jupyter notebook workbenches on OpenShift, including pod creation, authentication, networking, idle culling, and integration with Data Science Pipelines and Feast feature stores.

**Detailed**: The Kubeflow Notebook Controller repository contains two cooperating Go controllers that together manage Jupyter notebook workbenches in the Red Hat OpenShift AI (RHOAI) platform. The upstream **Kubeflow Notebook Controller** (KF controller) handles the core Notebook CRD lifecycle: it creates a StatefulSet with a single replica per notebook, exposes it via a ClusterIP Service, optionally creates an Istio VirtualService, manages idle culling by querying the Jupyter kernel/terminal APIs, and updates the Notebook status based on pod state. The downstream **ODH Notebook Controller** extends this with OpenShift-specific concerns: it injects `kube-rbac-proxy` authentication sidecars, creates Gateway API HTTPRoutes for external access, manages per-notebook NetworkPolicies, creates ReferenceGrants for cross-namespace routing, resolves notebook images from OpenShift ImageStreams, injects CA certificate bundles and cluster proxy environment variables, and integrates with Data Science Pipelines Application (DSPA) and Feast feature stores.

The two controllers run as separate Deployments in the platform namespace. The KF controller watches `Notebook` CRs and reconciles the core StatefulSet, Service, and optional VirtualService. The ODH controller also watches `Notebook` CRs and reconciles the OpenShift-specific resources: authentication sidecars, HTTPRoutes, NetworkPolicies, RBAC bindings, and integration secrets. A mutating webhook registered by the ODH controller intercepts Notebook CREATE and UPDATE operations to inject sidecars, resolve images, mount certificates, and prevent disruptive updates to running notebooks. The RHOAI 3.x architecture uses Gateway API (HTTPRoute + `data-science-gateway` Gateway) with `kube-rbac-proxy` sidecars for authentication, replacing the RHOAI 2.x pattern of OpenShift Routes with `oauth-proxy` sidecars.

## Architectural Analysis

The kubeflow notebook controller repository implements a clean separation between upstream Kubeflow concerns and OpenShift-specific extensions. The KF controller manages the portable core (StatefulSet, Service, VirtualService, culling), while the ODH controller handles platform integration (authentication, Gateway API routing, RBAC, DSPA, ImageStreams). This dual-controller pattern allows the upstream CRD and lifecycle logic to remain stable while downstream extensions evolve independently. Both controllers watch the same `Notebook` CRD but reconcile different resource sets, communicating implicitly through annotations (e.g., `kubeflow-resource-stopped`, `kubeflow.org/stop`).

The RHOAI 3.x networking architecture is sophisticated. HTTPRoutes are created in the controller's namespace (a protected, central namespace) rather than in user namespaces, which prevents users from modifying their own routing rules. ReferenceGrants are created in user namespaces to authorize cross-namespace backend references from the central HTTPRoute to the user's notebook Service. This cross-namespace pattern requires careful lifecycle management: the controller uses three distinct finalizers (`HTTPRouteFinalizerName`, `ReferenceGrantFinalizerName`, `KubeRbacProxyFinalizerName`) to ensure cleanup of cluster-scoped and cross-namespace resources when notebooks are deleted. The ReferenceGrant (`notebook-httproute-access`) is shared per namespace and only deleted when the last notebook in that namespace is removed, which is tracked by counting remaining notebooks.

The authentication layer shows a complete migration from RHOAI 2.x's `oauth-proxy` pattern to RHOAI 3.x's `kube-rbac-proxy` pattern. The controller still retains OAuthClient deletion code (guarded by the `notebook-oauth-client-finalizer.opendatahub.io` finalizer) for migrating users from RHOAI 2.25 to 3.x, but no new OAuthClients are created. The kube-rbac-proxy sidecar on port 8443 performs Kubernetes-native authentication via TokenReview and authorization via SubjectAccessReview, checking if the user has `get` permission on the Notebook resource. Resource limits for the sidecar are configurable via annotations, with defaults of 100m CPU and 64Mi memory.

The mutating webhook is a critical component that performs 10 distinct mutations in sequence during Notebook create/update. Notably, it implements a "reconciliation lock" pattern: on CREATE, it adds a `kubeflow.org/stop` annotation to prevent the KF controller from starting the pod before the ODH controller has finished setting up networking and auth resources. The lock is removed by the ODH controller after reconciliation completes. The webhook also implements update-prevention logic: if a running notebook would require a pod restart due to webhook mutations (image resolution, CA cert changes, etc.), the webhook blocks the change and sets an `update-pending` annotation. This forces users to stop the notebook first, preventing disruptive restarts.

The idle culling system in the KF controller is elegant but limited. It queries the Jupyter `/api/kernels` and `/api/terminals` HTTP endpoints directly to determine activity, checking if all kernels are "idle" and no terminals exist. The culling threshold (`CULL_IDLE_TIME`, default 24 hours) and check period (`IDLENESS_CHECK_PERIOD`, default 1 minute) are configurable. When a notebook is idle, the controller sets the `kubeflow-resource-stopped` annotation, which the main reconciler interprets as "scale StatefulSet to 0 replicas". This annotation-based communication pattern is simple but means culling depends on the main controller's reconciliation loop running promptly.
