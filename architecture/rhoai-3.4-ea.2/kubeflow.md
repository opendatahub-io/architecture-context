## Purpose
**Short**: ODH Notebook Controller extends the upstream Kubeflow Notebook Controller to manage OpenShift-specific notebook workbench lifecycle including Gateway API ingress, kube-rbac-proxy authentication, network policies, CA certificates, pipeline integration, and Feast feature store configuration.

**Detailed**: The ODH Notebook Controller is an OpenShift-specific controller that watches Kubeflow `Notebook` custom resources and reconciles additional platform resources required for secure notebook (workbench) operation on Red Hat OpenShift AI (RHOAI). It operates alongside the upstream Kubeflow Notebook Controller (which manages the StatefulSet, Service, and idle culling lifecycle).

The controller's primary responsibilities include: (1) creating Gateway API `HTTPRoute` resources in a central namespace to expose notebooks via the platform's `data-science-gateway`, (2) injecting `kube-rbac-proxy` sidecar containers into notebook pods for Kubernetes-native RBAC-based authentication, (3) managing `NetworkPolicy` resources to control ingress traffic to notebook pods, (4) creating `ReferenceGrant` resources to enable cross-namespace HTTPRoute backend references, (5) managing CA certificate bundles for TLS trust chains, (6) syncing pipeline runtime image metadata from ImageStreams, (7) creating Elyra pipeline configuration secrets from DSPA (DataSciencePipelinesApplication) CRs, and (8) mounting Feast feature store configuration.

The controller also includes a mutating admission webhook that intercepts `Notebook` create/update operations to inject the kube-rbac-proxy sidecar, mount CA certificate bundles, resolve ImageStream references to container image digests, mount pipeline runtime images, handle Elyra/DSPA pipeline secrets, mount Feast configuration, inject proxy environment variables, and manage update-pending annotations to prevent unintended pod restarts. Legacy RHOAI 2.x OAuthClient CRs are cleaned up via finalizers during migration to 3.x.

### Ingress

| Name | Type | Path | Port | Protocol | Encryption | Auth | Exposure |
|------|------|------|------|----------|------------|------|----------|
| nb-{namespace}-{notebook} (auth mode, per notebook) | HTTPRoute (Gateway API) | /notebook/{namespace}/{notebook-name} (PathPrefix) | 8443/TCP | HTTPS | TLS termination at Gateway | kube-rbac-proxy (RBAC) | External via data-science-gateway |
| nb-{namespace}-{notebook} (non-auth mode, per notebook) | HTTPRoute (Gateway API) | /notebook/{namespace}/{notebook-name} (PathPrefix) | 8888/TCP | HTTP | TLS termination at Gateway | None (direct) | External via data-science-gateway |

**RHOAI 3.x Gateway API Ingress Pattern (Current)**:
- The controller creates `HTTPRoute` CRs in the **central controller namespace** (e.g., `opendatahub`), NOT in user namespaces, for security
- Each HTTPRoute references the parent Gateway `data-science-gateway` in `openshift-ingress` namespace (configurable via `NOTEBOOK_GATEWAY_NAME` and `NOTEBOOK_GATEWAY_NAMESPACE` env vars)
- Cross-namespace backend references (HTTPRoute in central namespace to Service in user namespace) are enabled via `ReferenceGrant` CRs created in each user namespace
- One `ReferenceGrant` named `notebook-httproute-access` per user namespace (shared by all notebooks in that namespace)
- **With auth** (`notebooks.opendatahub.io/inject-auth: "true"`): HTTPRoute backend points to `{notebook}-kube-rbac-proxy` Service (port 8443); kube-rbac-proxy enforces RBAC: users must have `get` permission on `notebooks` resource in `kubeflow.org` API group for the specific notebook name/namespace
- **Without auth**: HTTPRoute backend points directly to `{notebook}` Service (port 8888)
- The controller ensures only one type of HTTPRoute exists at a time (auth or non-auth), deleting conflicting routes when switching modes
- Auth headers `X-Auth-Request-User` and `X-Auth-Request-Groups` are forwarded to the upstream notebook container
- HTTPRoute names exceeding 63 characters use `generateName` with a truncated prefix

**Legacy RHOAI 2.x Pattern (Migration Support Only)**:
- OAuthClient CRs are cleaned up via finalizer during notebook deletion for users migrating from 2.x to 3.x
- No new Routes or OAuthClients are created in 3.x
