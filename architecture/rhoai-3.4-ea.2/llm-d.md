## Purpose

**Short**: llm-d is a Kubernetes-native distributed inference serving stack that provides optimized container images, deployment guides, and Kustomize recipes for serving large language models at scale using vLLM, Gateway API, and intelligent request routing.

**Detailed**: llm-d is a CNCF sandbox project founded by Red Hat, Google Cloud, IBM Research, CoreWeave, and NVIDIA. It serves as the integration and distribution point for a multi-component inference stack that orchestrates model servers (primarily vLLM) behind a Gateway API-based load balancer enhanced with an intelligent Endpoint Picker (EPP) for LLM-aware request routing.

The repository itself does not contain operator or controller source code. Instead, it provides: (1) Dockerfiles that build optimized vLLM container images for multiple accelerator platforms (CUDA/NVIDIA, ROCm/AMD, HPU/Intel Gaudi, CPU, XPU/Intel GPU), bundling critical dependencies like NIXL for RDMA-based KV cache transfer, LMCache for tiered caching, DeepEP/DeepGEMM for MoE kernels, and FlashInfer for attention acceleration; (2) Kustomize-based deployment guides ("well-lit paths") covering production scenarios including optimized baseline, prefill/decode disaggregation, wide expert parallelism, tiered prefix caching, and workload autoscaling; and (3) Gateway API recipes for configuring inference ingress with Istio, AgentGateway, and GKE Gateway providers.

Within the RHOAI distribution, llm-d provides the model serving runtime images and deployment patterns that integrate with the broader Red Hat OpenShift AI platform, leveraging the Kubernetes Inference Gateway (gateway-api-inference-extension) as its control plane API and Envoy's ext-proc protocol for intelligent request routing.

## Architectural Analysis

### Meta-Repository Architecture

llm-d follows a "meta-repository" pattern where the main repo (`llm-d/llm-d`) acts as an integration point rather than containing runtime source code. The actual inference intelligence lives in separate repositories — primarily `llm-d/llm-d-inference-scheduler` (the EPP/Router) and upstream `vllm-project/vllm` (the model server). This repo's primary artifacts are container images and deployment guides that compose these components into production-ready configurations.

This design reflects the project's principle of "composition over configurability" — each component connects at API boundaries (Gateway API CRDs, ext-proc gRPC, Prometheus metrics, RDMA transport), and this repo provides the "well-lit paths" that wire them together correctly. The trade-off is that understanding the full architecture requires cross-referencing multiple repositories, and the deployment guides encode significant operational knowledge in Kustomize overlays that may drift from the actual component capabilities.

### Container Image Build Complexity

The CUDA Dockerfile is the most complex artifact in the repository (593 lines), reflecting the challenge of building optimized inference runtimes. It uses a multi-stage build with a builder stage that compiles GDRCopy, UCX, NIXL, LMCache, DeepEP, DeepGEMM, and FlashInfer from source, then copies only the necessary artifacts into a runtime stage based on `nvcr.io/nvidia/cuda:13.0.2-runtime-ubi9`. The GB200 variant (`runtime-gb200` target) swaps in a platform-specific DeepEP wheel. Version pinning via `docker/common-versions` and sccache with S3 backend keep builds reproducible and fast.

A notable security consideration is that the CUDA image runs as UID 2000 (non-root, OpenShift compatible) while the CPU, HPU, and ROCm images default to root. The ROCm image also uses an unpinned `vllm/vllm-openai-rocm:v0.15.1` base image, creating a supply chain risk.

### Gateway API as Unified Control Plane

llm-d has fully embraced the Kubernetes Gateway API as its control plane, using `InferencePool` as the central abstraction that bridges Gateway routing (HTTPRoute → InferencePool → EPP → model server). The ext-proc protocol enables a clean separation between the data plane (Envoy handling connections, TLS, streaming) and the control plane (EPP making LLM-aware routing decisions). This architecture allows llm-d to leverage any conformant Gateway implementation — the recipes demonstrate Istio, AgentGateway, and GKE Gateway with minimal configuration differences, primarily the `gatewayClassName` and provider-specific parameters.

### Disaggregated Serving as First-Class Architecture

Prefill/decode disaggregation is deeply integrated into the architecture rather than being an afterthought. The EPP's `disagg-profile-handler` runs two independent scheduling profiles (prefill and decode) and uses a "decider" to dynamically choose between aggregated and disaggregated execution based on prefix cache hit rates. The routing sidecar in decode pods implements the multi-step protocol (prefill request → KV transfer metadata → decode request), with backend-specific adapters for vLLM (sequential `nixlv2` protocol) and SGLang (concurrent bootstrap protocol). NIXL provides the underlying RDMA transport with support for InfiniBand, RoCE, and TCP fallback.

### Security Posture: Intentionally Minimal

The llm-d stack intentionally pushes authentication and authorization to the Gateway/proxy layer rather than implementing it at the model server level. This means that by default, vLLM endpoints are completely unauthenticated — security depends entirely on the Gateway provider configuration (Istio AuthorizationPolicy, GKE IAP, etc.) and network isolation. The flow control headers (`x-gateway-inference-fairness-id`, `x-gateway-inference-objective`) represent a trust boundary risk, as noted in the docs: clients can self-assert priority levels unless an upstream trusted gateway strips and re-injects these headers. For RHOAI deployments, this security gap would need to be addressed by the platform operator's kube-rbac-proxy integration.

### Monitoring and Observability Maturity

The observability stack is well-developed with five pre-built Grafana dashboards, Prometheus metrics from both vLLM (KV cache utilization, queue depth, running requests) and the EPP (scheduling latency, plugin performance, prefix cache hit rates), and OpenTelemetry distributed tracing across the full request path. The EPP exposes 15+ custom metrics that enable both operational monitoring and autoscaling signals. This level of observability maturity is notable for a CNCF sandbox project and reflects production deployment experience.
