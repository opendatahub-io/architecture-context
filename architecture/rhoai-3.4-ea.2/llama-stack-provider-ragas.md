## Purpose

**Short**: Implements Ragas evaluation as an out-of-tree Llama Stack provider, enabling LLM evaluation via Llama Stack's eval API both inline (in-process) and remotely (via Kubeflow Pipelines).

**Detailed**: This component provides a plugin (provider) for the [Llama Stack](https://github.com/meta-llama/llama-stack) evaluation API that delegates actual evaluation to the [Ragas](https://github.com/explodinggradients/ragas) framework. Ragas is a popular open-source framework for evaluating Retrieval-Augmented Generation (RAG) pipelines. By bridging Llama Stack's eval API to Ragas, the provider enables users to evaluate LLM outputs using metrics such as answer relevancy, context precision, faithfulness, and context recall.

The provider ships in two execution modes. The **inline** provider runs Ragas evaluation in the same process as the Llama Stack server, wrapping Llama Stack's inference and embedding APIs so Ragas can use them as its LLM and embedding backends. The **remote** provider offloads evaluation to Kubeflow Pipelines (Data Science Pipelines in RHOAI), constructing a two-step KFP pipeline that retrieves data from Llama Stack and then executes Ragas evaluation in an isolated container. Results from the remote provider are persisted to S3 and fetched back via the Llama Stack job status API.

Within the RHOAI platform, this component is deployed as a container image (`odh-trustyai-ragas-lls-provider-dsp-rhel9`) used as the base image for Kubeflow Pipeline component steps. It is managed by the TrustyAI team and integrates with Data Science Pipelines, Llama Stack distributions, and S3-compatible object storage.

## Architectural Analysis

The llama-stack-provider-ragas component follows a clean plugin architecture, implementing the Llama Stack `Eval` and `BenchmarksProtocolPrivate` interfaces. The codebase is well-organized with a clear separation between the inline and remote execution paths. The `compat.py` module provides a backward-compatibility layer that supports both the legacy `llama_stack` package layout and the newer `llama_stack_api` module, ensuring the provider works across Llama Stack version transitions.

The inline provider uses an adapter pattern to bridge Ragas' LangChain-based LLM/embedding interfaces (`BaseRagasLLM`, `BaseRagasEmbeddings`) to Llama Stack's inference API. The `LlamaStackInlineLLM` and `LlamaStackInlineEmbeddings` wrappers translate Ragas' sync/async call conventions to Llama Stack's OpenAI-compatible completion and embedding endpoints. Notably, the inline provider hardcodes `max_workers=1` in the Ragas `RunConfig`, limiting parallelism — likely to avoid overwhelming the in-process inference API.

The remote provider implements a more complex orchestration pattern. It lazily initializes a KFP client with a pre-flight health check (using `requests` directly rather than the KFP client's built-in healthz handling, which is noted as "poorly handled"). Pipeline submission creates a two-step DAG: data retrieval from Llama Stack followed by Ragas evaluation. The evaluation step mounts AWS credentials from a Kubernetes Secret via `kfp-kubernetes`, writes results to S3 as JSONL, and the provider fetches them back when the user queries job results. This design means the KFP pods call back to the Llama Stack server URL, requiring network reachability from the KFP execution namespace to the Llama Stack endpoint — a non-trivial networking requirement in production clusters.

A notable design choice is the base image resolution chain in `components.py`: the KFP step image is resolved from (1) `KUBEFLOW_BASE_IMAGE` environment variable, (2) a `trustyai-service-operator-config` ConfigMap in candidate namespaces (`redhat-ods-applications`, `opendatahub`), or (3) a hardcoded default `quay.io/trustyai/llama-stack-provider-ragas:latest`. This ties the remote provider to the TrustyAI operator's configuration plane, creating an implicit dependency. The Dockerfile.konflux builds this image on UBI9, but the current `pip install` without Hermeto prefetch means the build is not fully hermetic — dependencies are fetched from PyPI at build time rather than pre-fetched by the Konflux pipeline.

The token handling for KFP authentication uses `pydantic.SecretStr` for the pipelines token (ensuring it's redacted in logs), with a fallback to reading the token from the active kubeconfig. This design supports both explicit token provisioning (via Secret) and developer-mode operation where `oc login` provides the token. The component has no direct TLS configuration — all HTTPS connections (to KFP, S3, Kubernetes API) rely on system-level TLS from the UBI9 base image's OpenSSL.
