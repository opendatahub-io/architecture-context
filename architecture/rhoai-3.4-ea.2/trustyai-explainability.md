## Purpose

**Short**: TrustyAI is a Java library and Quarkus-based REST service providing AI explainability, fairness metrics, and data drift detection for ML models served via KServe/ModelMesh on Red Hat OpenShift AI.

**Detailed**: TrustyAI Explainability is a multi-module Maven project that combines a core XAI (Explainable AI) Java library with a containerized Quarkus service. The core library implements explainability algorithms (LIME, SHAP, Counterfactual, Partial Dependency Plots, Time Series Saliency), fairness metrics (Statistical Parity Difference, Disparate Impact Ratio), and data drift detection methods (Kolmogorov-Smirnov Test, Fourier MMD, Meanshift). The connectors module provides integration with KServe v1 (HTTP) and KServe v2 (HTTP + gRPC) inference protocols, enabling the service to query model predictions for explanation generation.

The explainability-service module wraps these capabilities in a Quarkus REST API that ingests inference data from ModelMesh/KServe via direct REST calls or Knative CloudEvents, stores it in PVC, MariaDB/MySQL, or S3-compatible object storage (MinIO), and exposes endpoints for on-demand and scheduled metric computation. Computed fairness and drift metrics are published to Prometheus for monitoring. The service integrates into the RHOAI platform as a per-namespace sidecar deployment managed by the TrustyAI Service Operator, with each instance monitoring models within its namespace.

In the RHOAI distribution, the service runs with a restricted endpoint profile that disables local explainer endpoints (LIME, SHAP, Counterfactual, Time Series Saliency) and global explainer endpoints, exposing only fairness and drift metrics. The ODH profile enables all endpoints.

## Architectural Analysis

TrustyAI Explainability follows a modular monorepo pattern where the core algorithms, protocol adapters, and service layer are cleanly separated into Maven modules with well-defined dependency directions: `explainability-core` is the foundation (no upward dependencies), `explainability-connectors` depends on core for data model types and adds KServe protocol support, and `explainability-service` depends on both to compose the REST API. The `explainability-arrow` module is an optional bridge for Python interop via Apache Arrow columnar format. This separation allows the core library to be used independently (e.g., in the TrustyAI Python library via JNI/Arrow bridge) while the service module adds Quarkus-specific concerns like CDI, scheduling, and REST endpoints.

The service's data ingestion architecture supports two paths: synchronous REST consumption via `/consumer/kserve/v2` (the primary path when integrated with ModelMesh's payload processor webhook) and asynchronous CloudEvents consumption via Quarkus Funqy functions (the Knative-native path). Both paths feed into a reconciliation layer (`KServeInferencePayloadReconciler`) that matches partial inference payloads (separate request and response events) into complete inference records. This dual-ingestion design allows the service to work in both traditional Kubernetes deployments (REST webhook) and serverless/Knative environments.

A notable architectural decision is the RHOAI profile's endpoint restriction: the `%rhoai` Quarkus profile disables all local and global explainer endpoints (LIME, SHAP, Counterfactual, Time Series Saliency, Aggregated LIME, PDP), leaving only fairness metrics (SPD, DIR) and drift metrics (KSTest, ApproxKSTest, FourierMMD, Meanshift) enabled. This reflects a product maturity decision where explainer endpoints are considered tech preview in RHOAI. The feature flags are implemented via Quarkus configuration properties (`endpoints.explainers.local.*=disable`) checked at endpoint class initialization, allowing runtime profile selection.

The Konflux build pipeline uses an unusual hermeticity pattern for this Java project. Rather than compiling from source within the Dockerfile (as the regular `Dockerfile` does), the `Dockerfile.konflux` consumes a pre-built artifact produced by Red Hat's PNC (Project Newcastle) build system. The `artifacts.lock.yaml` file pins the exact PNC build artifact with its SHA-256 checksum, and Hermeto/cachi2 prefetches it into the build context. The Dockerfile then simply unzips the artifact and copies the Quarkus application layers into the UBI9 OpenJDK 17 runtime image. This provides supply chain integrity through PNC's controlled build environment rather than Dockerfile-level build hermeticity.

The service's gRPC client connections to KServe inference servers use plaintext channels without TLS (`ManagedChannelBuilder.forTarget(target).usePlaintext()`). This is acceptable for in-cluster communication where network policies provide isolation, but it means the service relies entirely on Kubernetes network-level security for inference traffic confidentiality. The service itself supports TLS on port 4443 for inbound connections when certificate files are mounted, but this is configured externally by the TrustyAI Service Operator rather than by the service itself.
