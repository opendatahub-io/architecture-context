RBAC Summary:
- OVMS is a container application, not an operator
- No ClusterRoles or ServiceAccount RBAC defined by OVMS itself
- RBAC is managed by the deploying platform:
  - KServe: Creates ServiceAccount for OVMS pods, manages RBAC via InferenceService
  - ModelMesh: Creates ServiceAccount for OVMS pods, manages RBAC via ServingRuntime
  - kube-rbac-proxy sidecar (RHOAI 3.x): Enforces Kubernetes RBAC on API access

Authentication Summary:
- API Key: Optional, set via API_KEY environment variable
  - Applies to all gRPC and REST endpoints
  - Passed as Bearer token in Authorization header
  - Disabled if API_KEY env var is not set
- kube-rbac-proxy: Enforced by platform when deployed in RHOAI 3.x
- Health endpoints (/v2/health/live, /v2/health/ready): Always unauthenticated
- Metrics endpoint (/metrics): Always unauthenticated

CORS Configuration:
- --allow_credentials: false (default)
- --allowed_origins: * (default)
- --allowed_methods: * (default)
- --allowed_headers: * (default)

Secrets:
- API_KEY (env var) - API key for client authentication, user-provisioned, no auto-rotation
- AWS credentials (env var / file) - S3 access, user-provisioned, no auto-rotation
- Azure Storage credentials (env var) - Azure Blob access, user-provisioned, no auto-rotation
- GCP credentials (service account JSON) - GCS access, user-provisioned, no auto-rotation
- HF_TOKEN (env var) - HuggingFace Hub access for private models, user-provisioned, no auto-rotation
