RBAC Summary:
================================================================================
No RBAC roles defined by MLServer. MLServer is a non-privileged application
container, not a Kubernetes operator. RBAC is managed at the KServe/RHOAI
operator level.

Authentication & Authorization:
================================================================================
- MLServer does NOT implement authentication or authorization at application level
- All auth is enforced by the platform:
  - RHOAI 3.x: kube-rbac-proxy sidecars (injected by KServe/RHOAI operator)
  - RHOAI 2.x: oauth-proxy sidecars
- REST endpoints (8080/TCP): No app-level auth
- gRPC endpoints (8081/TCP): No app-level auth
- Metrics endpoint (8082/TCP): No auth

Container Security:
================================================================================
- Runs as: non-root user (UID 1000)
- Workdir: /opt/mlserver
- /opt/mlserver: world-writable (chmod 776) for OpenShift arbitrary UID compat
- No secrets directly consumed or managed
- Model credentials handled by platform (KServe, service mesh)

Secrets:
================================================================================
No secrets are directly consumed or managed by MLServer.
- Model access credentials: Handled by KServe/platform
- TLS certificates: Handled by platform (kube-rbac-proxy, service mesh)

CORS Configuration:
================================================================================
- Disabled by default
- When enabled: origin allowlisting, credential forwarding,
  method/header restrictions, max-age caching (default 600s)

Network Binding:
================================================================================
- All servers bind to 0.0.0.0 (all interfaces)
- TLS termination handled externally by platform gateway
- Pod-local traffic between kube-rbac-proxy and MLServer is plaintext
