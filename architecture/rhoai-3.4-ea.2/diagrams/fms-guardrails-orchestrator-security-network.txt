TLS Configuration Summary:
================================================================================

Server-side TLS:
  - Env vars: TLS_CERT_PATH, TLS_KEY_PATH
  - mTLS: TLS_CLIENT_CA_CERT_PATH (optional)
  - Implementation: rustls with ring crypto provider
  - Protocols: TLS 1.2, TLS 1.3
  - FIPS: Compliant

Client-side TLS:
  - Per-service TLS configuration in YAML
  - Named TLS configs for reuse
  - gRPC clients: tonic TLS with native and WebPKI roots
  - HTTP clients: hyper-rustls
  - Supports: cert/key pairs, CA certificates, insecure mode

Secrets:
================================================================================
  - TLS server cert/key (kubernetes.io/tls) - Server TLS termination
    Provisioned by: Platform/cert-manager, No auto-rotation
  - TLS client CA cert (Opaque) - mTLS client verification
    Provisioned by: Platform/cert-manager, No auto-rotation
  - Per-service TLS certs (Opaque) - Client mTLS to backends
    Provisioned by: Platform/admin, No auto-rotation
  - API tokens (Opaque) - OpenAI-compatible API auth
    Provisioned by: Admin/Secret store, No auto-rotation

Authentication & Authorization:
================================================================================
  Endpoint                  Auth Mechanism           Enforcement Point
  ─────────────────────────────────────────────────────────────────────
  /health, /info            None                     None (open access)
  /api/v1/*, /api/v2/*      Optional mTLS,           rustls TLS acceptor
                            header passthrough
  Outbound to detectors     Header passthrough       Orchestrator
                            (X-Forwarded-Access-     (config-driven)
                            Token → Bearer)
  Outbound to generation    mTLS client certs        Client TLS config
                            or API token (Bearer)    (per-service)

Header Passthrough:
================================================================================
  - Configurable headers forwarded to downstream services
  - X-Forwarded-Access-Token rewrite to Authorization: Bearer
  - detector-id and x-model-name headers injected for detector routing

Operator Features:
================================================================================
  Enabled:  disconnected, fips-compliant
  Disabled: proxy-aware, tls-profiles, token-auth-aws, token-auth-azure,
            token-auth-gcp
