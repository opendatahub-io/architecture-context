## Purpose

**Short**: Build repository producing notebook workbench and pipeline runtime container images for the Red Hat OpenShift AI platform.

**Detailed**: The notebooks repository is a monorepo that defines and builds all workbench and pipeline runtime container images used by the OpenShift AI (RHOAI) platform. It does not contain a running service or operator — instead, it produces approximately 25+ container images spanning three IDE families (JupyterLab, Code Server, RStudio Server) and seven pipeline runtime variants. Each image targets a specific combination of Python version (3.12), accelerator (CPU, NVIDIA CUDA 12.8, AMD ROCm 6.3), OS base (UBI9, RHEL9, CentOS Stream 9), and ML framework (PyTorch 2.10, TensorFlow, LLMCompressor, TrustyAI).

The repository follows a layered Dockerfile inheritance model: base images (in `base-images/`) provide the Python runtime and AIPCC configuration, then workbench Dockerfiles (in `jupyter/`, `codeserver/`, `rstudio/`) and runtime Dockerfiles (in `runtimes/`) add IDE-specific tooling, ML frameworks, and Python dependency stacks. Kustomize manifests (in `manifests/`) define OpenShift ImageStream resources that register the built images with the RHOAI dashboard, allowing users to select workbenches when creating notebooks. The platform's notebook controller (odh-notebook-controller) consumes these ImageStreams to launch workbench pods.

RHOAI images are built by Konflux (via `Dockerfile.konflux.*` variants) and published to `registry.redhat.io/rhoai/`. Python dependencies are locked via `pylock.toml` files (uv-based PEP 665 lockfiles) with hash verification, enabling hermetic builds. The code-server image has the most advanced hermetic build pipeline, with full RPM, npm, and pip prefetch support via cachi2/hermeto.

_The code-server image (`Dockerfile.konflux.cpu`) has the most complete hermetic build pipeline with RPM locks, npm offline install, pip offline install via cachi2, and generic artifact prefetch. Other Konflux Dockerfiles (jupyter, rstudio, runtimes) use `pylock.toml` for Python dependency hash verification but still fetch RPMs from network repos and download tools (oc client, TeX Live, pandoc) at build time. These images rely on Konflux's build infrastructure for network isolation rather than full in-Dockerfile hermeticity._

## Architectural Analysis

The notebooks repository follows a layered image inheritance model that prioritizes consistency and reuse. Base images (`base-images/`) provide Python runtime configuration via AIPCC (AI Platform Container Configuration) — a shared setup script that configures pip and uv index URLs, sets environment variables for virtual environment paths, and installs the `fix-permissions` utility. Each workbench family then builds a stack of Dockerfile stages: `cpu-base` or `cuda-base` → `jupyter-minimal` → `jupyter-datascience` → specialized variants (pytorch, tensorflow, trustyai, llmcompressor). This approach means a fix to the base layer propagates to all derived images, but also means the deepest images (e.g., pytorch+llmcompressor) accumulate all packages from every layer.

The Python dependency management strategy is notable: the repository uses `pylock.toml` (PEP 665 format) files generated by `uv` with full hash verification (`--require-hashes`). Each image variant has its own lock file under `uv.lock.d/pylock.{flavor}.toml`, with flavor matching the Dockerfile suffix (cpu, cuda, rocm). The `uv pip install` command uses `--strict --no-deps` mode, meaning every transitive dependency must be explicitly listed in the lock file. This provides strong reproducibility guarantees. The project-level `uv.lock` (207KB) handles development dependencies (pytest, ruff, etc.) separately from image build dependencies.

The code-server workbench stands out as the most architecturally complex image. It compiles code-server from source in a multi-stage build (rpm-base → whl-cache → cpu-base → codeserver), with full hermetic build support via cachi2/hermeto. The build prefetches RPMs, npm packages, and Python wheels, rewrites npm registry URLs to local file paths, and supports ppc64le/s390x via a dedicated wheel compilation stage. The final image includes both nginx (reverse proxy for health checks) and httpd (CGI processing for API endpoints), creating a three-process architecture inside the container (code-server + nginx + httpd). This complexity reflects the need to build VS Code's web version from source while maintaining multi-architecture support and FIPS compliance.

The manifest system uses a sophisticated kustomize `replacements` mechanism to inject versioned image references. Each ImageStream carries up to 8 tags (current "N" version plus N-1 through N-7 historical versions), with image references parameterized through two ConfigMaps: `notebook-image-params` (image registry URLs pinned by digest) and `notebook-image-commithash` (git commit hashes). The "N" version uses placeholder values (`dummy`) that are populated at release time, while N-1 through N-7 are pinned with `@sha256:` digests. This enables version rollback — users can select older workbench versions through the RHOAI dashboard if the latest version has issues.

A significant architectural observation is that these images contain no authentication or authorization logic. Security is entirely delegated to the platform: odh-notebook-controller injects a kube-rbac-proxy sidecar container that fronts the workbench on port 8443 with TLS and SubjectAccessReview (SAR) checks. The workbench itself listens on plaintext HTTP (8888 for Jupyter, 8787 for code-server/RStudio) and is only reachable via the sidecar. This separation of concerns is clean but means the workbench images have no ability to enforce access control independently — if the sidecar is misconfigured or bypassed (e.g., via pod network), the workbench is open.
