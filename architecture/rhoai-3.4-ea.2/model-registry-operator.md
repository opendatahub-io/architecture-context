## Purpose

**Short**: Kubernetes operator that manages the lifecycle of Model Registry and Model Catalog instances on Red Hat OpenShift AI, deploying REST API services with PostgreSQL backends and kube-rbac-proxy authentication sidecars.

**Detailed**: The Model Registry Operator is a controller-runtime based Kubernetes operator that reconciles `ModelRegistry` custom resources (API group `modelregistry.opendatahub.io`) to deploy and manage model registry service instances. Each ModelRegistry CR results in the creation of a REST API deployment (backed by the `model-registry` container image), a ServiceAccount, a Service, RBAC resources, and optionally a kube-rbac-proxy sidecar for HTTPS authentication, an OpenShift Route for external access, and a PostgreSQL database deployment with persistent storage.

The operator also includes a dedicated `ModelCatalogReconciler` that manages a singleton Model Catalog service instance. The catalog provides a browsable interface for pre-configured model sources (Red Hat AI, validated models, and third-party models) with benchmark performance data. The catalog has its own PostgreSQL database, kube-rbac-proxy authentication, and admin RBAC configuration driven by the platform's Auth CR.

The operator supports CRD version conversion between v1alpha1 (deprecated) and v1beta1 (storage version) via a conversion webhook, automatic migration from OAuth proxy to kube-rbac-proxy authentication, and storage version migration using either the Kubernetes StorageVersionMigration API or a manual fallback strategy. It detects cluster capabilities at startup (OpenShift vs vanilla Kubernetes, BYOIDC mode) and conditionally creates platform-specific resources like OpenShift Routes and user Groups.

## Architectural Analysis

The Model Registry Operator follows a template-driven reconciliation pattern that is somewhat unusual among Kubernetes operators. Rather than constructing Kubernetes resources programmatically in Go, it uses embedded Go text templates (`.yaml.tmpl` files) that are rendered with parameters derived from the ModelRegistry CR spec. This approach has trade-offs: templates are easier to read and modify for Kubernetes-native engineers, but they obscure the resource creation logic from static analysis tools and make it harder to understand what resources the operator creates without reading both controller code and templates together. The `k8s-objectmatcher` library is used for idempotent resource updates, computing patches between current and desired state rather than blindly overwriting resources.

The operator manages two conceptually distinct workloads through separate controllers. The `ModelRegistryReconciler` is instance-scoped: each ModelRegistry CR creates a full stack of resources (Deployment, Service, ServiceAccount, RBAC, optional database, optional Route). The `ModelCatalogReconciler` is singleton-scoped: it manages exactly one catalog deployment regardless of how many ModelRegistry CRs exist, and uses the platform's `default-modelregistry` CR (from `components.platform.opendatahub.io`) as an owner reference. This dual-controller pattern enables the catalog to be independently enabled/disabled without affecting individual registry instances.

The authentication architecture reflects an active migration from OAuth proxy to kube-rbac-proxy. The v1alpha1 API supported both Istio-based service mesh authentication and OAuth proxy; the v1beta1 API replaces both with kube-rbac-proxy as the primary auth mechanism. The operator implements automatic migration: the defaulting webhook converts OAuthProxy config to KubeRBACProxy config, and the controller reconcile loop also performs this migration for existing resources that bypass the webhook. The kube-rbac-proxy sidecar enforces SubjectAccessReview-based authorization, checking whether the caller has `get` permission on the registry's Service resource. This ties authentication to Kubernetes RBAC rather than OpenShift-specific OAuth flows.

The cluster capability detection system (`DetectClusterCapabilities`) enables the operator to run on plain Kubernetes, traditional OpenShift, and BYOIDC-enabled OpenShift 4.20+ clusters. In BYOIDC mode, the `user.openshift.io` API is intentionally absent, and the operator skips creating `Group` resources while still creating `RoleBinding` resources that reference external IdP groups. This detection happens at startup via the Kubernetes discovery API, with careful handling of partial discovery failures to distinguish BYOIDC mode from transient API errors.

The storage migration system provides a pluggable strategy pattern with two implementations. The primary strategy uses the Kubernetes-native `StorageVersionMigration` API (`migration.k8s.io/v1alpha1`), which triggers the API server to re-read and re-write all resources in the new storage version. The fallback strategy manually lists all ModelRegistry resources and updates them to trigger conversion webhook processing. Both strategies monitor the CRD's `status.storedVersions` field to detect when migration is needed and verify completion. The migration monitor starts with a 30-second delay after operator startup to allow the controller to stabilize.
