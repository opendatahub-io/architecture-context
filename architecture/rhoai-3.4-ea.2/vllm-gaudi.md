## Purpose

**Short**: Hardware plugin and container image that enables vLLM large language model inference on Intel Gaudi (HPU) accelerators for Red Hat OpenShift AI.

**Detailed**: vllm-gaudi is the Intel Gaudi hardware plugin for the vLLM inference engine, packaged as a container image for RHOAI. It implements the vLLM hardware plugin interface to provide HPU-specific execution backends for attention computation, KV cache management, distributed communication, model loading, and speculative decoding. The plugin registers itself via Python entry points (`vllm.platform_plugins`, `vllm.general_plugins`) so that vLLM can transparently offload inference workloads to Gaudi 2 and Gaudi 3 accelerators.

The container image is built on a Red Hat UBI 9.6 base with Habana SynapseAI drivers (v1.23.0), PyTorch 2.9.0, and the vLLM v0.14.1 core engine pre-installed. At runtime, it exposes vLLM's OpenAI-compatible API server (`vllm.entrypoints.openai.api_server`) for serving LLM inference requests. The image is built and shipped via Konflux to `quay.io/rhoai/odh-vllm-gaudi-rhel9`. It supports advanced features including FP8 quantization (via Intel Neural Compressor), Multi-head Latent Attention (MLA), LoRA adapters, NIXL-based KV cache transfer for disaggregated prefill/decode, unified attention for mixed prefill/decode batching, and multi-HPU tensor parallel inference via HCCL.

## Architectural Analysis

The vllm-gaudi plugin demonstrates a well-designed hardware abstraction pattern. Rather than forking the entire vLLM codebase (as was done previously with `vllm-fork`), this component registers itself through vLLM's plugin architecture using three entry points: `vllm.platform_plugins` for device registration, `vllm.general_plugins` for custom ops and model overrides. This clean separation allows the plugin to track upstream vLLM releases independently — the `VLLM_STABLE_COMMIT` file pins the last verified compatible upstream commit hash, providing a reproducible integration point.

The attention subsystem is notably sophisticated, providing four distinct attention backends that can be selected at runtime based on model architecture and hardware capabilities. The standard `HPUAttentionImpl` supports paged attention with FusedSDPA, Flex Attention, and naive implementations. The `HPUUnifiedAttentionImpl` enables mixed prefill/decode batching in a single forward pass — a significant optimization for throughput. The MLA variants (`HPUMLAImpl`, `HPUUnifiedMLAImpl`) implement Multi-head Latent Attention, which compresses KV cache into a shared latent space to reduce memory usage (critical for models like DeepSeek). The unified MLA implementation is particularly complex, maintaining two computation paths simultaneously: a "compute-friendly" path for fresh causal tokens that expands latent KV to full attention, and a "data-movement-friendly" path for cached tokens that projects queries into the compressed latent space.

The runtime configuration system (`extension/config.py`, `extension/features.py`, `extension/environment.py`) implements a lazy-evaluated, dependency-aware configuration framework. Feature flags are expressed as composable predicates (`All`, `Any`, `Not`, `Hardware`, `VersionRange`, `Kernel`) that can depend on hardware detection, software version checks, and other feature state. This allows features to be automatically enabled or disabled based on the runtime environment — for example, `fused_block_softmax_adjustment` is only enabled on Gaudi 3 with SynapseAI ≥1.22.0 and when the required kernel is available.

A notable security concern is the explicit removal of the FIPS provider from OpenSSL in the Dockerfile (`rpm -e --nodeps openssl-fips-provider-so`). While the vllm-gaudi service itself performs no cryptographic operations (TLS termination is handled by the platform ingress), this removal means the container cannot be deployed in FIPS-enforcing environments without modification. The build is also not hermetic — `hermetic: false` is explicitly set in the Tekton pipeline configuration, and there are no pip lock files, RPM lock files, or Hermeto prefetch integration. Python packages are installed via `pip install --no-cache-dir` during the multi-stage Docker build, pulling from PyPI and Habana's private artifactory at build time.

The distributed execution architecture supports tensor parallelism via HCCL (Habana's collective communication library) and data parallelism via Ray. The `HpuCommunicator` wraps HCCL primitives with HPU-specific workarounds (e.g., marking steps before collectives for compatibility with the lazy execution bridge). The NIXL connector enables disaggregated prefill/decode, where KV cache blocks are transferred between workers — a key optimization for large-scale serving where prefill and decode have different compute profiles. The HPU-specific NIXL connector patches `torch.Tensor.data_ptr` globally to use Habana's experimental `_data_ptr` function, which maps host virtual addresses to device physical addresses.
