## Purpose

**Short**: A collection of detector algorithm microservices for the FMS Guardrails Orchestrator, providing text content analysis including PII detection, file type validation, HuggingFace model-based classification, and LLM-as-a-judge evaluation.

**Detailed**: Guardrails Detectors is a multi-component repository that produces independent detector microservices consumed by the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator). Each detector implements the IBM Detector API (`/api/v1/text/contents`) and specializes in a different detection paradigm.

The repository contains three detector types: (1) **Built-in detectors** — lightweight, heuristic detection functions including regex-based PII detection (email, credit card, SSN, phone numbers, IP addresses), file type validation (JSON, XML, YAML with optional schema enforcement), and user-defined custom Python detectors with static code analysis sandboxing; (2) **HuggingFace detector** — a generic detector that loads HuggingFace `AutoModelForSequenceClassification` or `GraniteForCausalLM` models to perform ML-based content classification for risks like harm, social bias, jailbreak, profanity, violence, and others; (3) **LLM Judge detector** — integrates the [vLLM Judge](https://github.com/trustyai-explainability/vllm_judge) library to perform LLM-as-a-judge evaluation against built-in metrics (safety, toxicity, accuracy, helpfulness, etc.) or custom natural-language criteria.

All detectors are designed to run as KServe InferenceService workloads within the RHOAI platform and expose FastAPI-based REST APIs with Prometheus metrics instrumentation. The built-in detector is deployed alongside the Guardrails Orchestrator as a sidecar/co-deployed service, while the HuggingFace and LLM Judge detectors are deployed as independent inference services requiring external model storage or vLLM endpoints respectively.

## Architectural Analysis

The guardrails-detectors repository follows a modular microservice architecture where each detector type is independently deployable as a container image. All detectors share a common FastAPI base (`DetectorBaseAPI`) that provides health checks, Prometheus instrumentation, and consistent error handling. This common layer defines four Prometheus counters — `trustyai_guardrails_detections`, `trustyai_guardrails_requests`, `trustyai_guardrails_errors`, and `trustyai_guardrails_runtime` — that are labeled by `detector_kind` and `detector_name`, enabling fine-grained observability across all detector types.

The built-in detector uses a registry pattern (`BaseDetectorRegistry` → `RegexDetectorRegistry`, `FileTypeDetectorRegistry`, `CustomDetectorRegistry`) that allows multiple detection paradigms to coexist in a single service. The custom detector system is noteworthy: it performs AST-based static code analysis on user-provided Python files, blocking imports of `os`, `subprocess`, `sys`, `shutil` and calls to `eval`, `exec`, `open`, `compile` before dynamically loading the module. This provides a sandbox-like safety layer, though it operates at the syntactic level rather than using true sandboxing (e.g., seccomp, namespaces). The custom detector framework also supports decorators (`@use_instruments` for custom Prometheus metrics, `@non_blocking` for async background execution), which are injected into the custom module's namespace at load time.

The HuggingFace detector supports two model architectures — `AutoModelForSequenceClassification` (binary/multi-class classification with softmax scoring) and `GraniteForCausalLM` (IBM Granite Guardian models with token-level safe/unsafe probability extraction). The causal LM path iterates over seven predefined risk categories (harm, social bias, jailbreak, profanity, unethical behavior, sexual content, violence) per input, which makes it significantly more compute-intensive than the sequence classifier path. The `SAFE_LABELS` environment variable allows operators to configure which label indices are considered "safe" and should be excluded from detection results, with support for both index-based and name-based exclusion.

The LLM Judge detector is architecturally distinct — it performs no local inference but instead delegates to an external vLLM server via the `vllm-judge` library. This means it has minimal resource requirements compared to the HuggingFace detector but introduces a runtime dependency on an available LLM server. Notably, this component has no Konflux Dockerfile (`Dockerfile.konflux.judge` does not exist), suggesting it is not yet part of the official RHOAI build pipeline and may be upstream-only or planned for future inclusion.

Multi-architecture support is a significant build complexity concern. The HuggingFace Dockerfile includes custom build stages for ppc64le and s390x that compile PyTorch from source (v2.6.0) and OpenBLAS (v0.3.30) using gcc-toolset-13, adding substantial build time. The Konflux HF Dockerfile pins OpenSSL to version `3.5.1-5.el9_7` to address mismatch issues (per commit `e75a15e`). Build platforms span x86_64, ppc64le, arm64, and s390x per the Tekton pipeline configuration.
