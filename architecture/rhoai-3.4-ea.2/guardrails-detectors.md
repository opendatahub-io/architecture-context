## Purpose
**Short**: A collection of detector algorithm microservices for the FMS Guardrails Orchestrator that analyze text content for safety, PII, file-type validation, and LLM-as-judge evaluation.

**Detailed**: The Guardrails Detectors repository provides three distinct detector microservices that integrate with the FMS Guardrails Orchestrator (led by IBM). Each detector is packaged as an independent container image exposing a REST API conforming to the IBM Detector API specification. The detectors are designed to be invoked by the Guardrails Orchestrator for input/output guardrailing of Large Language Model text generation.

The three detector types are: (1) **Built-in Detector** - lightweight heuristic/algorithmic detections including regex-based PII detection (email, credit card, SSN, phone numbers, IP addresses, UK postcodes), file-type validation (JSON, XML, YAML with optional schema validation), and custom Python-based detection functions with static code analysis for security; (2) **HuggingFace Detector** - a generic ML model-based detector supporting AutoModelForSequenceClassification and GraniteForCausalLM architectures with GPU/CUDA support, intended for deployment as a KServe InferenceService; (3) **LLM Judge Detector** - an LLM-as-a-judge detector using the vLLM Judge library to evaluate content against built-in metrics (safety, toxicity, accuracy, helpfulness, etc.) via an external vLLM-compatible inference server.

All detectors expose Prometheus metrics for monitoring detection counts, request counts, error rates, and runtime latency, prefixed with `trustyai_guardrails_`.

| FileTypeDetectorRegistry | Python Detector Registry | Registry of file-type validators (JSON, XML, YAML) with optional schema validation |
| CustomDetectorRegistry | Python Detector Registry | Dynamic loader for user-provided Python detection functions with static code analysis security sandbox |

### Security Controls

- **Custom Detector Static Code Analysis**: The CustomDetectorRegistry performs AST-based static code analysis on user-provided Python detector code, blocking forbidden imports (`os`, `subprocess`, `sys`, `shutil`) and forbidden calls (`eval`, `exec`, `open`, `compile`, `input`). Exception: `os.environ`/`os.getenv` are explicitly allowed.
- **SSL/TLS Support**: The DetectorBaseAPI `main()` function supports `ssl_ca_certs` configuration with `ssl.CERT_REQUIRED` via config.yaml, enabling mTLS when configured.

## Multi-Architecture Support

The HuggingFace detector (Dockerfile.konflux.hf) includes multi-architecture build support:

| Architecture | Special Build Steps |
|-------------|-------------------|
| amd64 | Standard pip install of PyTorch |
| arm64 | Standard pip install of PyTorch |
| ppc64le | Builds PyTorch from source (v2.6.0), builds OpenBLAS from source (v0.3.30), requires gcc-toolset-13 |
| s390x | Builds PyTorch from source (v2.6.0), requires gcc-toolset-13, gcc-gfortran, openblas-devel |
