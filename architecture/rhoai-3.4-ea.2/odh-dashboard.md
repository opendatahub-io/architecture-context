## Purpose

**Short**: Web-based management dashboard for Red Hat OpenShift AI (RHOAI), providing a unified UI for data science workflows including notebooks, model serving, model registry, pipelines, and Gen AI features.

**Detailed**: The ODH Dashboard is a full-stack web application consisting of a React/PatternFly frontend and a Node.js/Fastify backend-for-frontend (BFF). It serves as the primary user interface for OpenShift AI, enabling data scientists and administrators to manage workbenches (Jupyter notebooks), model serving runtimes, model registries, data connections, hardware profiles, and cluster settings.

The application employs a modular architecture using Webpack Module Federation, allowing feature-specific UI plugins (Model Registry, Gen AI Studio, MaaS, MLflow, Eval Hub, AutoML, AutoRAG) to be deployed as separate containers within the same pod. Each plugin runs its own HTTPS server on a dedicated port and is proxied through the main dashboard backend. The backend authenticates users via kube-rbac-proxy sidecar (RHOAI 3.x pattern) which enforces RBAC before proxying requests to the application. User identity is forwarded via `X-Auth-Request-User` and `X-Auth-Request-Groups` headers.

The dashboard is deployed by the rhods-operator using kustomize overlays. For RHOAI self-managed deployments, the `rhoai/onprem` overlay is applied, which patches the base resources to use RHOAI-specific naming (`rhods-dashboard`), configures the `data-science-gateway` Gateway API HTTPRoute for external access, and injects operator-managed container images.

### Ingress

| Name | Type | Parent Gateway | Port | Protocol | Encryption | TLS Mode | Exposure |
|------|------|----------------|------|----------|------------|----------|----------|
| rhods-dashboard | HTTPRoute (Gateway API v1) | data-science-gateway (openshift-ingress) | 8443/TCP | HTTPS | TLS | Passthrough to kube-rbac-proxy | External |

**Architecture Notes**:
- The HTTPRoute references the `data-science-gateway` Gateway in the `openshift-ingress` namespace
- All traffic is routed to the kube-rbac-proxy sidecar (port 8443) which terminates TLS and enforces RBAC
- kube-rbac-proxy authorizes by checking if the user can `list` `projects` in the `project.openshift.io` API group
- After authentication, requests are proxied to the dashboard backend on `localhost:8080`
- Headers `X-Auth-Request-User` and `X-Auth-Request-Groups` are injected by kube-rbac-proxy
- Federated plugin containers receive the forwarded access token via `x-forwarded-access-token` header
