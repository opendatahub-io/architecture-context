## Purpose
**Short**: The ODH Dashboard is the primary web UI for Red Hat OpenShift AI, providing a unified management interface for data science workloads, model serving, notebooks, pipelines, and platform administration.

**Detailed**: ODH Dashboard serves as the central management console for the Red Hat OpenShift AI (RHOAI) platform. It provides a React-based single-page application frontend paired with a Node.js/Fastify backend-for-frontend (BFF) that acts as a proxy to the Kubernetes API server and various platform services. The dashboard enables data scientists and platform administrators to manage Jupyter notebooks, model serving (KServe, ModelMesh), data science pipelines (Kubeflow Pipelines), model registries, hardware profiles, connection types, and cluster settings through an intuitive web interface.

In RHOAI 3.x, the dashboard adopts a modular architecture using Webpack Module Federation. Feature-specific UI modules (model-registry, gen-ai, MaaS, MLflow, eval-hub, automl, autorag) run as separate sidecar containers within the same pod, each serving their own federated frontend bundles over HTTPS. The main dashboard container orchestrates these modules, proxying API requests and loading remote UI components dynamically. Authentication is enforced via a kube-rbac-proxy sidecar that fronts the main dashboard, requiring users to have OpenShift project list permissions before accessing the UI. External access is provided via Gateway API HTTPRoute resources referencing the platform's data-science-gateway.

### kube-rbac-proxy Configuration

The kube-rbac-proxy sidecar is configured with:
- **Secure listen address**: `0.0.0.0:8443`
- **Upstream**: `http://localhost:8080` (dashboard backend)
- **Authorization**: ResourceAttributes check - user must be able to `list` `projects` in `project.openshift.io` API group
- **Auth header forwarding**: Enabled (`X-Auth-Request-User`, `X-Auth-Request-Groups`)
- **Health endpoint**: Port 8444 (`/healthz`)
- **TLS**: Uses OpenShift serving cert from `dashboard-proxy-tls` secret

## Deployment Configuration

### Pod Structure (RHOAI 3.x Modular Architecture)

The dashboard runs as a single Deployment (`rhods-dashboard`) with **2 replicas** and pod anti-affinity across availability zones. Each pod contains up to 9 containers:

1. **rhods-dashboard** (main): Dashboard backend on port 8080/TCP (HTTP, internal only)
2. **kube-rbac-proxy**: Auth proxy on port 8443/TCP (HTTPS, external-facing)
3. **model-registry-ui**: Federated module on port 8043/TCP (HTTPS)
4. **gen-ai-ui**: Federated module on port 8143/TCP (HTTPS)
5. **maas-ui**: Federated module on port 8243/TCP (HTTPS)
6. **mlflow-ui**: Federated module on port 8343/TCP (HTTPS)
7. **eval-hub-ui**: Federated module on port 8543/TCP (HTTPS)
8. **automl-ui**: Federated module on port 8643/TCP (HTTPS)
9. **autorag-ui**: Federated module on port 8743/TCP (HTTPS)

### Resource Requests/Limits

| Container | CPU Request | CPU Limit | Memory Request | Memory Limit |

| rhods-dashboard | 500m | 1000m | 1Gi | 2Gi |
