## Purpose

**Short**: The unified web dashboard for Red Hat OpenShift AI, providing a management UI for data science projects, model serving, notebooks, pipelines, and AI/ML platform administration.

**Detailed**: odh-dashboard is the primary user-facing component of the Red Hat OpenShift AI (RHOAI) platform. It is a monorepo that produces a host application (Node.js/Fastify backend + React/PatternFly frontend) and seven federated plugin modules, each with its own TypeScript frontend and Go Backend-for-Frontend (BFF) service. The host application provides the core platform UI — project management, notebook environments, model serving, pipeline management, cluster settings, and user administration. It communicates with the Kubernetes API both directly (via a service account) and on behalf of users (via token forwarding from kube-rbac-proxy).

The modular federation architecture allows feature-specific UI modules (Gen AI Studio, Model Registry, Model-as-a-Service, MLflow, AutoML, AutoRAG, Evaluation Hub) to be developed, built, and deployed independently as sidecar containers within the same pod. Each module exposes its UI via Webpack Module Federation and its API via a Go BFF service. The host application dynamically discovers and loads these remote modules at runtime based on a ConfigMap-driven configuration. All inter-container communication uses HTTPS with TLS certificates, and all user requests are authenticated via kube-rbac-proxy before reaching any container.

The dashboard is deployed as a Kubernetes Deployment with 2 replicas, fronted by a kube-rbac-proxy sidecar that terminates TLS and injects authentication headers. Ingress is provided via a Gateway API HTTPRoute referencing the platform's data-science-gateway. The component defines four CRDs for managing applications, documents, quickstarts, and dashboard configuration.

## Architectural Analysis

The odh-dashboard represents one of the most architecturally sophisticated components in the RHOAI platform. Its modular federation architecture is a significant departure from traditional monolithic dashboards. The host application (Node.js/Fastify backend + React frontend) acts as a shell that dynamically discovers and loads federated UI modules at runtime via Webpack Module Federation. Each module (Gen AI, Model Registry, MaaS, MLflow, AutoML, AutoRAG, Eval Hub) ships as a separate container image containing both a TypeScript frontend bundle and a Go BFF binary, deployed as sidecar containers in the same pod.

The authentication chain is well-layered: kube-rbac-proxy terminates TLS on port 8443, validates the user's OIDC token against OpenShift, and injects authentication headers (`x-forwarded-access-token`, `x-auth-request-user`, `x-auth-request-groups`) before forwarding to the Node.js backend on port 8080 (localhost, plaintext). The backend then either handles the request directly or proxies it to a BFF sidecar via HTTPS (each BFF listens on a unique port). BFF sidecars extract the user token from the `x-forwarded-access-token` header and use it for downstream service calls, maintaining end-to-end user identity propagation. The backend also supports a fallback authentication chain (kube-rbac-proxy headers → User API → SelfSubjectReview → JWT parsing → dev mode) for environments where not all headers are available.

A notable design decision is the use of the OpenShift Dynamic Plugin SDK as the extension point framework. Each module declares extensions (routes, navigation items, areas, model-serving platform registrations) that the host application resolves at runtime based on feature flags. The `SupportedArea` enum ties module availability to platform component state (e.g., Gen AI requires LlamaStack, AutoML requires DS Pipelines). This allows the same dashboard deployment to adapt its UI based on which platform components are installed without code changes.

The Go BFF services share a common middleware pattern: RecoverPanic → EnableTelemetry → EnableCORS → InjectRequestIdentity → AttachNamespace → RequireAccessToService → service-specific clients. This provides consistent error handling, auth propagation, and RBAC enforcement across all modules. However, there is inconsistency in FIPS compliance — the MaaS BFF has a hardcoded `InsecureSkipVerify: true` in its HTTP client (with a TODO to fix), which is a gap worth tracking.

The deployment manifest structure uses a layered kustomize composition: core-bases provide the fundamental Deployment, Service, and RBAC; the modular-architecture overlay adds sidecar containers and federation config via JSON6902 patches; and the RHOAI overlays rename resources from `odh-dashboard` to `rhods-dashboard` and add distribution-specific application definitions. The RHOAI shared base patches every resource (Deployment, Service, RBAC, HTTPRoute, NetworkPolicy) to use the `rhods-dashboard` name, while the addon and onprem overlays only differ in which ISV/partner applications are included. The NetworkPolicy is permissive — it allows ingress on all 8 ports from the OpenShift ingress namespace and any pod in the same namespace.
