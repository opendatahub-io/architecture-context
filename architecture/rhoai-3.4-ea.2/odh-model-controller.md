## Purpose

**Short**: A Kubernetes operator and API server that automates model serving infrastructure management for KServe InferenceServices, LLMInferenceServices, and NVIDIA NIM integrations on RHOAI.

**Detailed**: The odh-model-controller is a controller-runtime based Kubernetes operator that reconciles KServe InferenceService, LLMInferenceService, ServingRuntime, and NIM Account custom resources. It automates the creation and lifecycle management of supporting Kubernetes resources required for model serving: OpenShift Routes for external access, Kuadrant AuthPolicies and Istio EnvoyFilters for authentication and authorization, Prometheus ServiceMonitors and PodMonitors for observability, NetworkPolicies for network segmentation, RBAC ClusterRoleBindings for auth delegation, storage-config Secrets for S3 data connections, and Ray TLS certificates for multi-node distributed serving.

The controller also includes a dedicated LLMInferenceService reconciler that manages Gateway API-aware authentication via Kuadrant AuthPolicy (user-defined and anonymous access modes) and Istio EnvoyFilter for TLS/SSL handling. The NIM Account controller integrates with NVIDIA NGC APIs to validate API keys, fetch model catalogs, create ServingRuntime templates, and manage pull secrets for NVIDIA NIM container images.

A separate deployable component — the model-serving-api server — provides a REST API for gateway discovery, enabling the RHOAI Dashboard to list available Gateway API gateways that users can reference when deploying LLMInferenceServices. The server authenticates requests via Bearer tokens and performs SelfSubjectAccessReview checks to enforce namespace-level RBAC.

## Architectural Analysis

The odh-model-controller follows a multi-controller composition pattern where each controller is responsible for a specific resource type, and sub-reconcilers further decompose the InferenceService lifecycle into discrete concerns. The `KserveRawInferenceServiceReconciler` orchestrates six sub-reconcilers (Route, ClusterRoleBinding, MetricsService, ServiceMonitor, MetricsDashboard, KEDA) via a multierror aggregation pattern, ensuring that failures in one sub-reconciler don't block others. Similarly, the LLMInferenceService controller uses dedicated `AuthPolicyReconciler` and `EnvoyFilterReconciler` sub-reconcilers for the Gateway API auth stack. This separation enables independent evolution of concerns — for instance, the AuthPolicy reconciler can be updated to support new Kuadrant API versions without touching the EnvoyFilter logic.

The controller exhibits a dual-ingress pattern reflecting the RHOAI 2.x → 3.x migration. For standard InferenceServices, it creates OpenShift Routes with Edge or Reencrypt TLS termination. For LLMInferenceServices (the 3.x pattern), it manages Kuadrant AuthPolicies targeting either Gateways (UserDefined access) or HTTPRoutes (Anonymous access), plus Istio EnvoyFilters for SSL/TLS handling on the Gateway's Envoy proxy. The AuthPolicy templates are embedded as Go `//go:embed` resources and rendered with Go `text/template`, parameterized by gateway name/namespace and OAuth audience from the OpenShift Authentication CR. The EnvoyFilter is explicitly noted as a temporary workaround until upstream Kuadrant issue #1531 is resolved.

The NIM Account controller is architecturally distinct — it integrates with external NVIDIA APIs rather than just managing Kubernetes resources. It handles two API key formats (personal `nvapi-*` prefix using Bearer auth, and legacy keys using Basic Auth + token exchange), supports air-gapped mode (reading from user-provided ConfigMaps instead of calling NVIDIA APIs), and creates OpenShift Template resources that allow users to instantiate NIM-backed ServingRuntimes through the RHOAI Dashboard. The controller uses a handler chain pattern (ValidationHandler → ConfigMapHandler → TemplateHandler → PullSecretHandler) where each handler can short-circuit the reconciliation.

The model-serving-api server represents a deliberate architectural decision to separate the Gateway discovery API from the operator process. It runs as its own Deployment with minimal RBAC (only list Gateways and get Namespaces), uses SelfSubjectAccessReview to enforce that the caller (RHOAI Dashboard user) has permission to create LLMInferenceServices in the target namespace before returning gateway information. The server enforces FIPS-compliant TLS with explicit cipher suite restrictions (ECDHE+AES-GCM only, P-256/P-384 curves), certificate hot-reloading via fsnotify for Kubernetes Secret volume mount rotation, and comprehensive security headers (HSTS, X-Frame-Options, nosniff, no-store).

A notable security boundary is enforced by the InferenceGraph webhook: it validates that all `serviceUrl` references in inference graph nodes point to internal Kubernetes DNS names (service.namespace.svc or service.namespace.svc.cluster.local), rejecting external hostnames to prevent Authorization header leakage to services outside the cluster. The InferenceService webhook enforces a 57-character name limit (to allow the "-predictor" suffix to fit within Kubernetes' 63-character limit) and prevents creation in protected namespaces.
