## Purpose

**Short**: Feast is an open-source feature store for machine learning that manages the storage, retrieval, and serving of features for model training and inference.

**Detailed**: Feast (Feature Store) provides a unified platform for defining, storing, and serving machine learning features. It bridges the gap between data engineering and ML by allowing teams to register feature definitions, materialize features from offline to online stores, and serve features at low latency for real-time inference. The system supports a wide range of backend data stores (Redis, PostgreSQL, DynamoDB, BigQuery, Snowflake, etc.) for both online and offline feature storage.

In the RHOAI distribution, Feast is deployed via the Feast Operator — a Kubernetes operator that manages FeatureStore custom resources. The operator deploys and configures the feature server (a Python-based HTTP/gRPC service), an optional registry service (gRPC + REST), optional offline store service, and optional UI. The operator handles TLS configuration via OpenShift service serving certificates, manages RBAC for Kubernetes-based authorization or OIDC-based auth, creates CronJobs for materialization, and integrates with OpenDataHub Notebook resources for automatic feature store configuration injection.

The Python SDK provides the `feast` CLI tool and the feature server runtime (FastAPI-based HTTP server or gRPC server) that handles online feature retrieval, feature pushing, and materialization. A Go-based feature server is also available for gRPC serving of online features with higher performance.

## Architectural Analysis

The Feast repository is a polyglot monorepo that produces two distinct Konflux container images for RHOAI: the Go-based operator and the Python-based feature server. The operator follows the standard controller-runtime pattern with a single primary controller (FeatureStoreReconciler) that manages a rich set of Kubernetes resources — Deployments, Services, ConfigMaps, PVCs, ServiceAccounts, Roles, RoleBindings, CronJobs, and OpenShift Routes. Rather than deploying separate pods for each Feast service (online, offline, registry, UI), the operator consolidates them into a single Deployment with multiple containers, which simplifies lifecycle management but means all services share the same scaling characteristics.

A notable design decision is the CronJob-based materialization approach: instead of running materialization as a direct process, the operator creates Kubernetes CronJobs that use the OpenShift origin-cli image to `kubectl exec` into the running feature server pod and execute materialization commands. This avoids duplicating the feature store configuration and Python runtime in a separate pod but creates a dependency on the feature server pod being available when the CronJob fires. The CronJob's RBAC is scoped to a namespace Role allowing only `pods/exec` on the specific deployment.

The TLS architecture leverages OpenShift service serving certificates when running on OpenShift, with a clean fallback for non-OpenShift environments. Each Feast service gets a dedicated TLS secret auto-generated by the OpenShift service CA. The operator annotates services with `service.beta.openshift.io/serving-cert-secret-name` and mounts the resulting secrets into the deployment. For environments without OpenShift, services run without TLS by default (ports 80), creating a clear security posture difference between OpenShift and vanilla Kubernetes deployments.

The Notebook integration controller (NotebookConfigMapReconciler) is an innovative pattern that automatically injects feature store configuration into KubeFlow Notebook pods. When a Notebook CR carries the `opendatahub.io/feast-integration=true` label and `opendatahub.io/feast-config=<project-names>` annotation, the controller creates a ConfigMap containing the feature_store.yaml for the referenced projects. This ConfigMap uses non-controlling owner references to the Notebook, allowing cleanup tracking without blocking Notebook deletion. The controller guards itself by checking for the kubeflow.org Notebook CRD at startup and skipping registration if it's absent.

The authorization model supports two pluggable backends — Kubernetes RBAC (using TokenReview and SubjectAccessReview for in-cluster auth) and OIDC (for external identity providers). When Kubernetes auth is enabled, the operator creates dedicated Roles and ClusterRoles for the Feast service account with permissions to perform token reviews and subject access reviews. The auth configuration is woven into the generated feature_store.yaml ConfigMap, so the Python feature server's FastAPI middleware can enforce permissions on every data-access endpoint. The custom role support (user-defined roles in the FeatureStore CR) allows fine-grained access control at the feature view level.

One area of concern is the feature server's Python dependency chain for the RHOAI container image. The Dockerfile.feature-server.konflux copies a `requirements.txt` but no pinned lock file (like uv.lock or Pipfile.lock) was found in the Dockerfiles directory on this branch. This means Python dependencies may not be fully hermetic for reproducible builds, though the downstream release process may add lock files.
