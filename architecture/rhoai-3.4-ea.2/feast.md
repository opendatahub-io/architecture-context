## Purpose
**Short**: Feast Operator manages the lifecycle of Feast (Feature Store) deployments on Kubernetes/OpenShift for machine learning feature serving.

**Detailed**: Feast is an open-source feature store for machine learning that enables teams to define, store, and serve features for model training and online inference. The Feast Operator automates the deployment and management of Feast infrastructure on Kubernetes and OpenShift clusters.

The operator watches `FeatureStore` custom resources and reconciles the desired state by creating and managing Deployments, Services, ConfigMaps, PersistentVolumeClaims, ServiceAccounts, RBAC resources, CronJobs, and (on OpenShift) Routes. It supports multiple store backends including file-based (SQLite, DuckDB, Dask), database-backed (PostgreSQL, Snowflake, Cassandra), and object store persistence. The operator packages all Feast server components (online store, offline store, registry, UI) into a single Deployment with multiple containers.

On OpenShift, the operator automatically configures TLS using OpenShift's service serving certificate infrastructure, creates Routes for UI access, and integrates with the ODH trusted CA bundle. It also manages a namespace registry ConfigMap that tracks all deployed FeatureStore instances across namespaces, and provides Notebook integration by creating ConfigMaps with Feast client configurations for Kubeflow Notebooks that have the `opendatahub.io/feast-integration` label. The operator supports both Kubernetes RBAC-based and OIDC-based authorization for Feast services.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| feast-operator (controller-manager) | Go Operator (controller-runtime) | Reconciles FeatureStore CRs; creates Deployments, Services, ConfigMaps, PVCs, ServiceAccounts, RBAC, CronJobs, and Routes |
| FeatureStoreReconciler | Controller | Watches FeatureStore CRs and owned resources (Deployments, Services, ConfigMaps, PVCs, ServiceAccounts, RoleBindings, Roles, CronJobs, Routes); deploys Feast services |
| NotebookConfigMapReconciler | Controller | Watches Kubeflow Notebooks with `opendatahub.io/feast-integration` label; creates per-notebook ConfigMaps with Feast client configs from matching FeatureStore CRs |
| FeastServices | Service Logic | Manages deployment of online/offline/registry/UI server containers, TLS configuration, PVC creation, namespace registry, and CronJob deployment |
| FeastAuthorization | Auth Logic | Deploys Kubernetes RBAC or OIDC authorization for Feast services; creates Roles, RoleBindings, ClusterRoles, ClusterRoleBindings for token review and subject access review |
| Feature Server | Python Application | Feast feature server (online/offline/registry/UI) deployed as containers within a single Pod managed by the operator |
| CronJob | Batch Job | Runs `feast apply` and `feast materialize-incremental` commands via `kubectl exec` into the Feast deployment |

## Data Flows

### Flow 1: FeatureStore Deployment

| Step | Source | Destination | Port | Protocol | Encryption | Auth |
|------|--------|-------------|------|----------|------------|------|
| 1 | User/Admin | Kubernetes API | 443/TCP | HTTPS | TLS | Bearer Token |
| 2 | Kubernetes API | Feast Operator | N/A | Watch Event | N/A | ServiceAccount |
| 3 | Feast Operator | Kubernetes API | 443/TCP | HTTPS | TLS | ServiceAccount Token |
| 4 | Feast Operator | Kubernetes API | 443/TCP | HTTPS | TLS | ServiceAccount Token |

Steps: User creates FeatureStore CR -> API server notifies operator via watch -> Operator creates Deployment, Services, ConfigMaps, PVCs, RBAC, CronJob -> Operator creates Route (OpenShift) and updates status.

### Flow 2: Feature Retrieval (Online Store)

| Step | Source | Destination | Port | Protocol | Encryption | Auth |
|------|--------|-------------|------|----------|------------|------|
| 1 | ML Application | feast-{name}-online Service | 80/TCP or 443/TCP | gRPC | TLS (if configured) | Kubernetes RBAC / OIDC / None |
| 2 | Online Store Container | Database (if DB persistence) | 5432/TCP | TCP | TLS (optional) | Secret credentials |

### Flow 3: Notebook Feast Integration

| Step | Source | Destination | Port | Protocol | Encryption | Auth |
|------|--------|-------------|------|----------|------------|------|
| 1 | User | Kubernetes API | 443/TCP | HTTPS | TLS | Bearer Token |
| 2 | Kubernetes API | NotebookConfigMapReconciler | N/A | Watch Event | N/A | ServiceAccount |
| 3 | NotebookConfigMapReconciler | Kubernetes API | 443/TCP | HTTPS | TLS | ServiceAccount Token |
| 4 | Notebook Pod | feast-{name}-* Services | 80/TCP or 443/TCP | gRPC/HTTP | TLS (if configured) | Per auth config |

Steps: User labels Notebook with `opendatahub.io/feast-integration=true` and annotates with `opendatahub.io/feast-config={project-names}` -> Operator creates `{notebook-name}-feast-config` ConfigMap with feature_store.yaml content -> Notebook uses ConfigMap to connect to Feast services.

### Flow 4: CronJob Materialization

| Step | Source | Destination | Port | Protocol | Encryption | Auth |
|------|--------|-------------|------|----------|------------|------|
| 1 | Kubernetes CronJob Scheduler | CronJob Pod | N/A | N/A | N/A | ServiceAccount |
| 2 | CronJob Pod | Feast Deployment Pod | N/A | kubectl exec | N/A | ServiceAccount (RBAC) |
| 3 | Feast Container | Database/Store backend | Varies | Varies | TLS (optional) | Secret credentials |

Steps: CronJob runs on schedule -> `kubectl exec` into Feast deployment -> Executes `feast apply` then `feast materialize-incremental`.
