## Purpose

**Short**: Python SDK for programmatic creation, management, and monitoring of Ray clusters and Ray jobs on Kubernetes (OpenShift AI / Open Data Hub).

**Detailed**: The CodeFlare SDK is a Python client library that provides data scientists with an intuitive interface for requesting and managing distributed compute resources on Kubernetes. It abstracts the complexity of creating RayCluster and RayJob custom resources, handling Kubernetes authentication, Kueue queue integration, mTLS certificate management, and dashboard discovery.

The SDK is designed to run inside Jupyter notebooks on RHOAI/ODH workbenches or from any Python environment with Kubernetes access. It generates RayCluster CRs (via the KubeRay operator) with configurable head/worker node resources, GPU/accelerator requests, autoscaling, tolerations, and custom volumes. For job submission, it supports both direct Ray Job Submission Client connections to existing clusters and declarative RayJob CRs with managed cluster lifecycles including runtime environment packaging (working directory zip, pip dependencies) and Kueue-based scheduling with priority classes.

The SDK also provides Jupyter notebook widgets for interactive cluster management (apply/delete/view), TLS certificate generation for mTLS connections to Ray clusters, and integration with kube-authkit for flexible Kubernetes authentication (OIDC, OAuth, token, kubeconfig, in-cluster). It includes a vendored KubeRay Python client for direct CRD operations.

## Architectural Analysis

The CodeFlare SDK follows a layered architecture with clear separation between user-facing APIs, Kubernetes operations, and Ray cluster specifics. The top-level `Cluster` class provides the primary interactive interface (apply/down/status/wait_ready), while `RayJob` provides a declarative batch job interface. Both delegate to lower-level modules for Kubernetes API calls, certificate management, and queue integration. The vendored KubeRay Python client (`src/codeflare_sdk/vendored/`) provides a second path for CRD operations, used primarily by the RayJob class, while the Cluster class uses the standard `kubernetes` Python client directly.

A notable design decision is the dual dashboard discovery mechanism. The SDK first attempts to locate the Ray dashboard via Gateway API HTTPRoutes (RHOAI v3.0+ pattern), then falls back to OpenShift Routes (pre-v3.0), then to Kubernetes Ingresses (Kind/vanilla clusters). The HTTPRoute discovery searches cluster-wide first, then iterates through known platform namespaces (`redhat-ods-applications`, `opendatahub`, etc.), making it resilient to namespace permission restrictions. The dashboard URL construction follows the RHOAI v3.0 convention: `https://{gateway-hostname}/ray/{namespace}/{cluster-name}`.

The mTLS certificate lifecycle is tightly coupled with cluster creation. When `Cluster.apply()` is called, it polls for the KubeRay-generated CA secret (labeled `ray.openshift.ai/cluster-name`), generates client certificates signed by that CA (RSA 3072-bit, SHA-256, 30-day validity), and stores them in a user-private directory (`~/.local/share/codeflare/tls/`) with restricted 0600 permissions. The certificates include SAN entries for `localhost`, `127.0.0.1`, `::1`, and the cluster's head service DNS names for proper mTLS verification. This design means the SDK must be present and authenticated at cluster creation time to establish the TLS trust chain. Certificate cleanup is automatic on `Cluster.down()`, and `refresh_certificates()` handles CA rotation scenarios.

The Kueue integration demonstrates defensive programming against optional components. Queue labels are only applied when Kueue is actually installed (detected by successful API calls). The autoscaling feature explicitly rejects Kueue-managed clusters because upstream KEP-77 (autoscaling with Kueue) has not graduated, preventing data scientists from creating unsupported configurations. The `RayJob` class handles Kueue differently: it relies on Kueue's mutating webhook to set `suspend=true` rather than setting it explicitly, which means jobs work correctly both with and without Kueue installed.

The `RayJob` class includes a runtime environment packaging pipeline that deserializes local Python files and working directories into Kubernetes Secrets. Local working directories are zipped (excluding `.ipynb` and `.md` files), base64-encoded, and stored in a Secret with owner references to the RayJob CR for automatic cleanup. An init container on the submitter pod unzips the working directory before the job runs. This design avoids the need for shared storage (PVCs) for simple job submissions while keeping the Secret under the 1MB Kubernetes limit.
