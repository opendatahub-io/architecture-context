## Purpose

**Short**: Provides a multi-tenant Models as a Service platform with API key management, subscription-based rate limiting, and policy-driven access control for LLM inference endpoints.

**Detailed**: Models as a Service (MaaS) is a comprehensive platform that sits atop KServe to provide enterprise-grade model serving with multi-tenant access control, token-based rate limiting, and usage metering. It consists of two primary components: the MaaS API (a Go/Gin REST service) and the MaaS Controller (a Kubernetes operator using controller-runtime).

The MaaS API serves as the data plane management layer, providing an OpenAI-compatible model listing endpoint (`/v1/models`), API key lifecycle management (creation, validation, revocation, search), tier-based group lookups, and subscription selection. It uses PostgreSQL for API key storage with SHA-256 hash-only persistence, and Kubernetes informers for real-time CRD and ConfigMap watches. The API integrates with Kuadrant/Authorino as an external metadata callback: Authorino calls the MaaS API to validate API keys and select subscriptions during request authorization.

The MaaS Controller manages three custom resources — MaaSModelRef, MaaSAuthPolicy, and MaaSSubscription — which together define which models are available, who can access them, and at what rate. The controller uses a model-centric aggregation pattern: multiple MaaSAuthPolicy and MaaSSubscription CRs referencing the same model are aggregated into single Kuadrant AuthPolicy and TokenRateLimitPolicy resources per model, overcoming Kuadrant's one-policy-per-target limitation. The controller integrates with KServe (for model endpoint discovery via LLMInferenceService), Gateway API (for HTTPRoute management), and Kuadrant (for policy enforcement).

## Architectural Analysis

The Models as a Service platform implements a sophisticated policy-driven multi-tenant architecture built on the Kuadrant/Authorino stack. The most notable design decision is the **model-centric aggregation pattern**: rather than creating one Kuadrant AuthPolicy per MaaSAuthPolicy CR, the controller aggregates all MaaSAuthPolicies targeting the same model into a single generated AuthPolicy. This addresses Kuadrant's limitation of allowing only one policy per HTTPRoute target. The same pattern applies to MaaSSubscription → TokenRateLimitPolicy. This aggregation creates a many-to-one mapping where N parent CRs collapse into 1 generated policy, with the parent names tracked in annotations (`maas.opendatahub.io/auth-policies`, `maas.opendatahub.io/subscriptions`).

The authentication architecture implements a dual-path strategy: OpenShift identity tokens are validated via KubernetesTokenReview, while API keys (prefixed `sk-oai-`) are validated via an HTTP callback to the maas-api service. Both paths converge to inject `X-MaaS-Username` and `X-MaaS-Group` headers into the request, enabling downstream handlers to operate on a uniform identity model regardless of the authentication method. The API key validation endpoint (`/internal/v1/api-keys/validate`) is restricted by NetworkPolicy to only accept traffic from Authorino pods, providing defense-in-depth. API keys are stored as SHA-256 hashes only, with the plaintext shown once at creation time (show-once pattern), following industry best practices similar to OpenAI, Stripe, and GitHub.

The controller's provider pattern (`BackendHandler` / `RouteResolver` interfaces) is designed for extensibility. Currently, only the `LLMInferenceService` provider is fully implemented — it discovers HTTPRoutes created by KServe via label selectors and extracts endpoints from the LLMInferenceService status. The `ExternalModel` provider is stubbed (`ErrKindNotImplemented`), indicating planned support for external model endpoints not managed by KServe. The provider pattern cleanly separates model discovery from policy management, allowing new backends without modifying the core controller logic.

The observability stack is comprehensive: a TelemetryPolicy injects model, tier, user, subscription, organization, and cost center labels into gateway metrics, enabling per-tenant billing and usage tracking. Istio Telemetry adds per-tier latency tracking via the `x-maas-tier` request header. ServiceMonitors scrape Authorino auth evaluation metrics, Limitador rate limiting metrics, and Envoy proxy stats from the gateway. This layered observability enables platform administrators to monitor authentication decisions, rate limit enforcement, and per-model inference latency broken down by subscription tier.

A notable security consideration is the use of `InsecureSkipVerify` for model endpoint probing in `discovery.go`. This is documented as intentional for cluster-internal traffic where the gateway handles TLS termination, but it means the maas-api service does not verify the identity of model backends during access validation probes. The defense-in-depth here relies on the cluster network and NetworkPolicies to prevent MITM attacks on these internal connections.
