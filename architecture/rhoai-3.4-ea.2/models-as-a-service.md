## Purpose
**Short**: A platform for providing LLM inference models as managed services with API key authentication, subscription-based access control, and token-based rate limiting.

**Detailed**: Models as a Service (MaaS) provides a comprehensive platform for exposing LLM inference endpoints as managed services on OpenShift. It consists of two main components: the MaaS API server, a Go-based HTTP service built with the Gin framework that handles API key management (CRUD, validation, hashed storage), model discovery, subscription selection, and tier-based lookups; and the MaaS Controller, a controller-runtime based Kubernetes operator that manages three custom resources (MaaSModelRef, MaaSAuthPolicy, MaaSSubscription).

The controller translates high-level MaaS CRDs into Kuadrant policy resources (AuthPolicy and TokenRateLimitPolicy) that target Gateway API HTTPRoutes. MaaSModelRef references LLMInferenceService backends (managed by KServe) and tracks their HTTPRoute and endpoint status. MaaSAuthPolicy defines which users/groups can access specific models, and the controller aggregates all auth policies per model into a single Kuadrant AuthPolicy. MaaSSubscription defines token rate limits per user/group per model, and the controller creates aggregated TokenRateLimitPolicies with per-user counters and tier-based priority ordering.

The platform uses Gateway API for ingress (via an `openshift-default` GatewayClass), Kuadrant/Authorino for authentication and authorization enforcement, and Kuadrant/Limitador for token-based rate limiting. API keys (prefixed `sk-oai-`) are validated via HTTP callbacks from Authorino to the MaaS API. The system supports both API key and OpenShift token authentication for the management API, while model inference endpoints use API key authentication exclusively.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| maas-api | Go HTTP Service (Gin) | API key management, model discovery, subscription selection, tier lookup; serves as Authorino callback target for API key validation |
| maas-controller | Go Operator (controller-runtime) | Reconciles MaaSModelRef, MaaSAuthPolicy, MaaSSubscription CRDs; creates/manages Kuadrant AuthPolicy and TokenRateLimitPolicy resources targeting HTTPRoutes |
| MaaSModelRef Controller | Reconciler | Watches MaaSModelRef CRDs and LLMInferenceService/HTTPRoute resources; validates HTTPRoute existence and gateway association; tracks model endpoint and readiness status |
| MaaSAuthPolicy Controller | Reconciler | Aggregates all MaaSAuthPolicy CRDs per model into a single Kuadrant AuthPolicy; configures API key validation callbacks, group/user membership authorization, and subscription metadata injection |
| MaaSSubscription Controller | Reconciler | Aggregates all MaaSSubscription CRDs per model into a single Kuadrant TokenRateLimitPolicy; implements tier priority ordering, per-user token counters, and deny rules for unsubscribed users |
| maas-default-gateway | Gateway API Gateway | External ingress gateway (gatewayClassName: openshift-default) in openshift-ingress namespace; terminates TLS and routes traffic to model and API HTTPRoutes |
| gateway-default-auth | Kuadrant AuthPolicy | Gateway-level default-deny AuthPolicy for unconfigured model inference paths; overridden by per-route AuthPolicies created by MaaSAuthPolicy controller |
| gateway-default-deny | Kuadrant TokenRateLimitPolicy | Gateway-level zero-token rate limit for unconfigured model paths; defense-in-depth behind auth deny |

| Per-model HTTPRoutes | HTTPRoute (Gateway API) | Per-model hostnames | Model-specific | HTTPS | TLS via Gateway | Terminate at Gateway | Via Gateway |

**Ingress Architecture (RHOAI 3.x Pattern)**:
- The MaaS platform deploys a Gateway API `Gateway` resource (`maas-default-gateway`) in the `openshift-ingress` namespace using the `openshift-default` GatewayClass
- KServe creates HTTPRoute resources per LLMInferenceService, which reference the `maas-default-gateway` as their parent
- The MaaS API server has its own HTTPRoute (`maas-api-route`) exposing `/v1/models` and `/maas-api/*` paths
- MaaSAuthPolicy controller creates per-model Kuadrant AuthPolicy CRs targeting model HTTPRoutes
- MaaSSubscription controller creates per-model Kuadrant TokenRateLimitPolicy CRs targeting model HTTPRoutes
- Gateway-level default policies (`gateway-default-auth`, `gateway-default-deny`) deny traffic to models without explicit MaaSAuthPolicy/MaaSSubscription configuration
- Authorino enforces auth via API key validation callbacks to maas-api internal endpoint (`/internal/v1/api-keys/validate`)
- Istio DestinationRule configures TLS (SIMPLE mode) for gateway-to-maas-api backend traffic when TLS overlay is active
