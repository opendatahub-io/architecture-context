## Purpose
**Short**: High-performance C++ model inference server providing gRPC and REST APIs compatible with KServe, TensorFlow Serving, and OpenAI protocols.

**Detailed**: OpenVINO Model Server (OVMS) is a production-grade inference serving system developed by Intel and distributed as part of Red Hat OpenShift AI (RHOAI). It hosts machine learning models and exposes them over standard network protocols (gRPC and REST), enabling remote inference from lightweight client applications regardless of programming language or framework.

OVMS is optimized for deployment on Intel architectures and uses the OpenVINO runtime for inference execution. It supports a wide range of AI workloads including traditional model inference (image classification, object detection), large language model (LLM) text generation via OpenAI-compatible APIs, text embeddings, reranking (Cohere-compatible), image generation, speech-to-text, and text-to-speech. Models can be loaded from local storage, cloud object stores (S3, Azure Blob, GCS), or pulled directly from HuggingFace Hub.

The server features advanced capabilities including DAG (Directed Acyclic Graph) pipeline scheduling for chaining models, MediaPipe graph execution for complex processing pipelines, Python code execution support, model versioning and hot-reload, dynamic batching and shape handling, GPU acceleration support, and Prometheus-compatible metrics. In the RHOAI distribution, it is built with Konflux CI from the `Dockerfile.konflux` using UBI9-based images with LTO optimization enabled.

## Network Architecture

### Services

OVMS does not create Kubernetes Services itself. Services are created by the deploying platform (KServe, ModelMesh, or manual deployment). The server binary exposes:

| Service Name | Type | Port | Target Port | Protocol | Encryption | Auth | Exposure |
|--------------|------|------|-------------|----------|------------|------|----------|
| gRPC listener | Process port | configurable/TCP | configurable | gRPC/HTTP2 | plaintext | API Key (optional, via `API_KEY` env var) | Internal (container port) |
| REST/HTTP listener | Process port | configurable/TCP | configurable | HTTP/1.1 | plaintext | API Key (optional, via `API_KEY` env var) | Internal (container port) |

**Port Configuration**:
- gRPC port: Set via `--port` CLI flag (default: 0 = disabled)
- REST port: Set via `--rest_port` CLI flag (default: 0 = disabled)
- Bind addresses: `--grpc_bind_address` (default: 0.0.0.0), `--rest_bind_address` (default: 0.0.0.0)
- IPv6 supported

### Ingress

OVMS does not manage its own ingress. In RHOAI deployments:

| Name | Type | Hosts | Port | Protocol | Encryption | TLS Mode | Exposure |
|------|------|-------|------|----------|------------|----------|----------|
| KServe InferenceService | HTTPRoute (Gateway API) / Route (OpenShift) | Auto-generated | 443/TCP | HTTPS | TLS 1.2+ | SIMPLE (via platform gateway) | External |

- In RHOAI 3.x: KServe creates HTTPRoute CRs referencing the platform data-science-gateway
- In RHOAI 2.x (legacy): KServe creates OpenShift Route CRs
- Authentication is handled by the platform (kube-rbac-proxy sidecar in 3.x, oauth-proxy in 2.x)

## Build and Deployment

### Konflux Build Pipeline

- **Dockerfile**: `Dockerfile.konflux` (multi-stage build)
- **Build platform**: `linux-extra-fast/amd64`
- **Build time**: Up to 8 hours (heavy compilation from source)
