## Purpose

**Short**: OpenVINO Model Server is a high-performance C++ inference serving system that hosts AI models and exposes them via gRPC and REST APIs compatible with TensorFlow Serving, KServe v2, and OpenAI standards.

**Detailed**: OVMS is a standalone inference serving system implemented in C++ and optimized for deployment on Intel architectures. It uses the OpenVINO toolkit as its core inference engine, enabling efficient execution of AI models on CPUs, GPUs, and NPUs. The server supports multiple API protocols: TensorFlow Serving v1 (predict/classify/regress), KServe v2 (model inference, health, metadata), and OpenAI-compatible v3 APIs for generative AI workloads (chat completions, embeddings, reranking, image generation, and audio).

Within the RHOAI platform, OVMS serves as one of the model serving runtimes. It is deployed as a container image (`odh-openvino-model-server-rhel9`) and is typically instantiated by KServe's InferenceService controller or run standalone. The server supports single-model and multi-model configurations, DAG-based pipeline composition via MediaPipe graphs, Python code execution within graphs, and HuggingFace model pulling. It includes advanced LLM features such as continuous batching, speculative decoding, prefix KV caching, tool calling, and streaming output via Server-Sent Events.

The build system uses Bazel 6.1.1 with OpenVINO built from source for RHEL, compiled with Haswell CPU optimizations and LTO (Link-Time Optimization). The server binary is packaged into a UBI9-minimal-based container with GPU driver support. An optional nginx mTLS authentication sidecar is available in the `extras/` directory for environments requiring mutual TLS.

## Architectural Analysis

OpenVINO Model Server follows a modular architecture built around a singleton `Server` instance that orchestrates several modules: gRPC server, HTTP server (Drogon), C API, MediaPipe graphs, Python interpreter, model management, and metrics. The startup flow parses CLI arguments through `cxxopts`, validates configuration, and initializes modules sequentially. The server supports two primary deployment modes: single-model (direct `--model_name`/`--model_path` flags) and multi-model (JSON config file via `--config_path`), with a third GenAI mode that integrates HuggingFace model pulling.

The threading model is carefully stratified. gRPC uses a configurable thread pool (default CPU_CORES * 8 threads) with support for multiple server instances via the `GRPC_SERVERS` environment variable or `--grpc_workers` flag. The REST layer uses Drogon's async event-driven architecture with separate thread pools for unary and streaming requests. Each model inference request is queued via an NIREQ-based request queue, providing backpressure when inference capacity is saturated. A periodic filesystem poller (default 1-second interval) detects model config changes, enabling hot-reload of model configurations without server restart.

The GenAI subsystem is the most architecturally complex layer, built on MediaPipe's graph composition framework. Each GenAI workload type (LLM, embeddings, rerank, image generation, audio S2T/T2S) is implemented as a MediaPipe calculator with its own protobuf configuration schema. The LLM calculator alone exposes 136+ configuration fields including continuous batching parameters, speculative decoding with draft models, prefix KV caching with eviction strategies (SnapKV, KV-Crush), sparse attention patterns (TRISHAPE, XATTENTION), tool calling with schema-guided generation, and reasoning chain-of-thought parsing. The OpenAI-compatible v3 API layer handles Jinja2 template processing for chat prompts and streaming via Server-Sent Events.

A notable architectural decision is the deliberate absence of TLS in the core server binary. The gRPC server always uses `InsecureServerCredentials()` (grpcservermodule.cpp:138) and the Drogon HTTP server binds as plaintext. This is by design: in RHOAI deployments, TLS termination is handled by the platform's ingress infrastructure (Gateway API + Envoy proxy, or kube-rbac-proxy sidecar) rather than by the application. For standalone deployments requiring TLS, the optional nginx mTLS sidecar in `extras/nginx-mtls-auth/` provides a wrapper that runs nginx alongside OVMS, with nginx terminating TLS and proxying to OVMS over loopback. The nginx configuration enforces TLS 1.2 with restricted cipher suites (AES256+EECDH), mandatory client certificate verification, and HSTS headers.

The build system uses Bazel with extensive security hardening flags: position-independent code (-fPIC, --force_pic), stack protection (-fstack-protector, -fstack-clash-protection), format string protection (-Werror=format-security), control-flow protection (-fcf-protection=full), RELRO (-Wl,-z,relro,-z,now), and non-executable stack (-Wl,-z,noexecstack). The Konflux build specifically targets Haswell CPU architecture with LTO enabled (`-flto=auto -ffat-lto-objects -march=haswell`) and builds OpenVINO from source rather than using pre-built binaries.
