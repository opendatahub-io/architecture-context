## Purpose

**Short**: Kubeflow Pipelines (KFP) provides a platform for building, deploying, and managing multi-step ML workflows as reusable, reproducible pipelines on Kubernetes via Argo Workflows.

**Detailed**: Data Science Pipelines is the RHOAI distribution of Kubeflow Pipelines v2. It provides a complete ML pipeline orchestration platform consisting of multiple microservices: an API server exposing gRPC and REST APIs for pipeline CRUD operations; a v2 driver/launcher system that compiles KFP pipeline specs into Argo Workflows and manages per-step execution with caching, artifact tracking, and Kubernetes resource provisioning; a persistence agent that watches Argo Workflows and syncs execution state back to the API server; a scheduled workflow controller that manages recurring pipeline executions via CronJob-like scheduling; and a cache server that acts as a mutating admission webhook to enable step-level output caching.

The platform integrates deeply with Kubernetes and Argo Workflows for pipeline execution, ML Metadata (MLMD) for artifact lineage tracking, and S3-compatible object storage (MinIO/S3/GCS) for pipeline artifacts. Authentication is handled via Kubernetes TokenReview and configurable HTTP header extraction, with RBAC enforced through SubjectAccessReview. The RHOAI distribution builds all Go binaries with FIPS-compliant crypto (GOEXPERIMENT=strictfipsruntime, CGO_ENABLED=1) on UBI9 base images.

## Architectural Analysis

Data Science Pipelines follows a microservice architecture where the API server acts as the central control plane, with satellite controllers and agents handling specific lifecycle concerns. The design separates concerns cleanly: the API server handles user-facing CRUD and compilation; the driver/launcher pair handles per-step execution within Argo Workflow pods; the persistence agent handles async state synchronization; and the scheduled workflow controller handles recurring execution. This separation allows each component to scale and fail independently.

The v2 pipeline execution model uses a driver/launcher split pattern that resembles a sidecar but operates as sequential Argo Workflow steps. The driver runs first to resolve inputs, check cache, and generate a pod spec patch; then the launcher runs inside the user container to execute the actual component, publish results to MLMD, and manage artifacts. This indirection enables the caching, lineage tracking, and Kubernetes resource provisioning features without modifying user containers. The driver creates PersistentVolumeClaims for workspace storage, mounts secrets and configmaps specified in the KubernetesExecutorConfig, and injects init containers for OCI/modelcar artifact pre-fetching.

The cache server is architecturally notable — it operates as a Kubernetes mutating admission webhook that intercepts pipeline step pod creation. When a cache hit is detected (by matching SHA256 fingerprint of inputs, outputs, command, and image), the webhook replaces the step's containers with a lightweight busybox image that simply outputs the cached results. This approach is elegant but has security implications: the webhook needs broad pod mutation permissions and runs without RBAC-based auth (relying on Kubernetes API server mTLS for webhook calls). The KFP v2 driver has its own cache path that bypasses this webhook entirely, handling caching at the driver level via gRPC calls to the API server's TaskService.

Authentication uses a dual-mode approach: HTTP header extraction (for Google IAP or reverse proxy scenarios) and Kubernetes TokenReview (for service-to-service and direct API access). The persistence agent and scheduled workflow controller authenticate to the API server using projected ServiceAccount tokens with the `pipelines.kubeflow.org` audience and 1-hour expiration — a significant security improvement over long-lived tokens. Authorization is enforced via Kubernetes SubjectAccessReview, mapping pipeline operations to custom RBAC verbs on `kubeflow.org` and `pipelines.kubeflow.org` API groups.

The pipeline-runner service account has notably broad permissions (`*` verbs on kubeflow.org `*` resources, full pod/deployment/service/job CRUD) because user pipeline steps can create arbitrary Kubernetes resources. This is a known security trade-off — pipeline authors can effectively perform any Kubernetes operation their pipeline's service account allows. In multi-user mode, namespace isolation limits the blast radius, but the pipeline-runner SA remains highly privileged within its namespace.
