## Purpose
**Short**: OpenAI-compatible Batch API gateway for processing bulk LLM inference requests via the `/v1/files` and `/v1/batches` endpoints.

**Detailed**: The Batch Gateway is an llm-d component that implements the OpenAI Batch API specification, enabling users to submit large batches of inference requests (up to 50,000 per file, 200 MB max) for asynchronous processing. It consists of two microservices: an API Server that exposes OpenAI-compatible REST endpoints for file and batch management, and a Batch Processor that polls a priority queue, dequeues jobs, and forwards individual inference requests to an inference gateway (llm-d or GAIE).

The system uses Redis as its backend for metadata storage, priority queuing, event channels, and status tracking. File storage is pluggable, supporting both local filesystem and S3-compatible object storage (AWS S3, MinIO, etc.). The architecture supports multi-tenancy via configurable HTTP headers (default: `X-MaaS-User`) and provides Prometheus metrics for both the API server and the processor. TLS is supported on all connections including optional mTLS for inference gateway communication. The component is built using Konflux CI with UBI9 base images for Red Hat distribution.

| Prometheus Metrics | Go Library | Exposes HTTP request metrics (API server) and job processing metrics (processor) |

**Middleware Stack** (outermost to innermost):
1. Recovery Middleware - catches panics
2. Request Middleware - request ID generation, tenant ID extraction, logging, metrics
3. Security Headers Middleware - X-Content-Type-Options, X-Frame-Options, X-XSS-Protection, CORS preflight

**Multi-tenancy**: Tenant isolation is enforced at the application level via the `X-MaaS-User` HTTP header (configurable). All database queries filter by tenant ID. Cross-tenant access returns 404 Not Found.
