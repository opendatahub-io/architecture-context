## Purpose

**Short**: CLI tool for validating, backing up, and migrating Red Hat OpenShift AI (RHOAI) deployments on Kubernetes.

**Detailed**: The `odh-cli` (also known as `rhai-cli` or `kubectl-odh`) is a command-line interface designed as a kubectl plugin for interacting with RHOAI deployments. It provides three core capabilities: (1) a lint/validation engine that assesses cluster configuration health and upgrade readiness with 20+ diagnostic checks across components, dependencies, platform resources, and workloads; (2) a backup system that discovers, serializes, and exports RHOAI workloads (Notebooks, DataSciencePipelinesApplications) along with their dependencies (ConfigMaps, Secrets, PVCs); and (3) a migration framework for executing version-aware cluster migrations with prepare/run/list stages.

The CLI is distributed as a container image (`quay.io/rhoai/rhoai-upgrade-helpers-rhel9`) based on `ose-cli-rhel9`, bundling `kubectl`, `oc`, `jq`, `yq`, Python 3 with Kubernetes client libraries, and upgrade helper scripts from the `rhoai-upgrade-helpers` submodule. It uses Kubernetes CLI Runtime for authentication and supports standard kubectl flags (`--kubeconfig`, `--context`, `--token`, `--server`). The lint engine is the most architecturally significant subsystem — it implements a check framework with DiagnosticResult CR-like structures, version-gated applicability conditions, and fluent builder patterns for composable validation logic. Most checks target the RHOAI 2.x → 3.x upgrade path, validating component removal (ModelMesh, CodeFlare, serverless KServe), dependency requirements (cert-manager, OpenShift ≥ 4.19.9), and workload compatibility (DSPA API migration, AcceleratorProfile → HardwareProfile migration).

## Architectural Analysis

The odh-cli implements a layered architecture that cleanly separates concerns across three tiers: command definitions (Cobra wiring in `cmd/`), business logic (implementations in `pkg/`), and shared utilities (clients, version detection, resource types in `pkg/util/`). Every command follows a consistent four-phase lifecycle (`AddFlags` → `Complete` → `Validate` → `Run`) enforced by the `Command` interface in `pkg/cmd/command.go`. This pattern enables independent testing of each phase and ensures all commands handle setup, validation, and execution uniformly.

The lint check framework is the most sophisticated subsystem. It uses a composition-based approach where concrete checks embed `BaseCheck` and implement `CanApply`/`Validate` methods. The framework provides fluent builder patterns (`validate.Component()`, `validate.DSC()`, `validate.DSCI()`, `validate.Operator()`, `validate.Workloads()`) that handle common boilerplate — fetching DSC/DSCI resources, checking component management states, querying OLM subscriptions — so individual checks focus on domain-specific validation logic. The `DiagnosticResult` output structure follows Kubernetes CR conventions (metadata, spec, status with conditions), making results machine-parseable and consistent with the platform's resource model. This is a thoughtful design that enables downstream tooling to consume lint results as structured data.

Version detection follows a priority cascade (DSC → DSCI → OLM CSV), reflecting the evolving RHOAI metadata model. The `IsUpgradeFrom2xTo3x()` helper gates the majority of checks, as the 2.x → 3.x transition is a major architectural shift involving component removals (ModelMesh, CodeFlare, serverless KServe), dependency changes (ServiceMesh removal, Kueue standalone migration), and API migrations (DSPA v1alpha1 → v1, AcceleratorProfile → HardwareProfile). The version-gating approach means the check set will need expansion as future major transitions occur.

The backup pipeline uses a three-stage concurrent architecture (discovery → dependency resolution → writer) with Go errgroups and channels. Worker concurrency is auto-scaled to CPU count (capped at 20), and the Kubernetes client is configured with elevated QPS/burst limits (50/100 vs kubectl's default 5/10) to support parallel operations. Dependency resolution is pluggable via a registry pattern, with dedicated resolvers for Notebooks (ConfigMaps, PVCs, Secrets) and DSPA workloads. The field-stripping approach (removing `.status`, `.metadata.uid`, etc.) makes backup artifacts portable across clusters.

One notable architectural risk is the FIPS compliance gap: the Go binary is built with `CGO_ENABLED=0` (static linking), which means it uses pure-Go crypto rather than OpenSSL-backed FIPS-validated crypto. While this is common for CLI tools that run client-side, it means the binary would fail check-payload validation if it were deployed as a container running in a FIPS-mandated environment. The base image (`ose-cli-rhel9`) provides FIPS-compliant OpenSSL, but the Go binary doesn't link against it. For a tool that connects to Kubernetes API servers via TLS, this gap is low-risk but worth noting for security audits.

The resource type registry in `pkg/resources/types.go` centralizes all 45+ Kubernetes resource GVK/GVR definitions used throughout the codebase (Architecture Principle VIII). This prevents scattered hardcoded API group strings and enables type-safe resource references. The JQ query engine (`gojq`) is used pervasively for extracting fields from unstructured Kubernetes resources, providing flexible path-based queries without requiring typed Go structs for every CRD.
