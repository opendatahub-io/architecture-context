## Purpose

**Short**: Kubernetes-native operator for orchestrating distributed machine learning training jobs across multiple frameworks including PyTorch, DeepSpeed, MLX, and MPI.

**Detailed**: Kubeflow Trainer (v2) is a controller-runtime based operator that manages the lifecycle of distributed ML training workloads on Kubernetes. It introduces three Custom Resource Definitions — TrainJob, TrainingRuntime, and ClusterTrainingRuntime — that separate the concern of "what to train" (TrainJob) from "how to train" (runtimes). Users submit TrainJobs referencing a runtime template, and the operator reconciles them into JobSet resources with appropriate distributed training configuration.

The operator implements a plugin-based runtime framework supporting PyTorch distributed (torchrun), MPI-based workloads (DeepSpeed, MLX), and TorchTune fine-tuning recipes. It integrates with gang-scheduling systems (Volcano, Kubernetes CoScheduling) for coordinated pod placement, and manages ancillary resources like SSH key Secrets for MPI, hostfile ConfigMaps, and PodGroups for scheduling. For RHOAI, the operator adds progression tracking (HTTP-based metrics polling from training pods), per-TrainJob NetworkPolicy isolation, and ImageStream definitions for training hub workbench images supporting CPU, CUDA, and ROCm accelerators.

The data cache sub-component is a separate Rust-based service (head/worker architecture) for caching training datasets, deployed alongside the main Go operator. It is not built via Konflux for RHOAI and exists only as an upstream feature.

## Architectural Analysis

The Kubeflow Trainer v2 architecture represents a significant redesign from the v1 Training Operator. The most notable design decision is the separation of training intent (TrainJob) from training configuration (TrainingRuntime/ClusterTrainingRuntime). This two-layer abstraction enables platform administrators to define reusable runtime templates — with pre-configured images, ML policies, and scheduling constraints — while data scientists submit lightweight TrainJob CRs that reference those templates and override only trainer-specific fields (image, command, env, numNodes, resources). The controller merges these via JSON strategic merge patches, which is both powerful and complex — the `mergePodTemplateOverrides` function in `pkg/runtime/core/trainingruntime.go` performs multi-step JSON marshaling/unmarshaling to apply user overrides to the runtime template's JobSet spec.

The plugin framework (`pkg/runtime/framework/`) is the core architectural pattern. Seven plugin interface types (CustomValidation, EnforceMLPolicy, EnforcePodGroupPolicy, PodNetwork, ComponentBuilder, TrainJobStatus, WatchExtension) are composed in a chain-of-responsibility pattern. Six plugins (JobSet, CoScheduling, Volcano, MPI, Torch, PlainML) implement subsets of these interfaces. The framework executes plugins in a defined phase order: ML policy enforcement → pod group policy → pod network → component building → status. This ordering matters because later phases depend on mutations from earlier ones (e.g., Torch sets environment variables that the JobSet builder needs to include in the final ApplyConfiguration). The single-TrainJobStatusPlugin constraint (enforced at framework initialization) ensures status mapping is unambiguous.

The RHOAI-specific additions (in `pkg/rhai/`) are cleanly separated from upstream code. Progression tracking adds HTTP-based metrics polling from training pods, with configurable intervals and port numbers via annotations (`trainer.opendatahub.io/*`). The NetworkPolicy reconciliation creates per-TrainJob policies that allow pod-to-pod communication within the same JobSet and, when progression tracking is enabled, also allow the controller pod to reach the training pod's metrics port. The controller discovers its own namespace from the service account mount, falling back to "opendatahub". This design avoids hardcoding but does assume the controller runs with a mounted service account.

A security consideration worth noting: the progression tracking feature polls training pods over plaintext HTTP on configurable ports (default 28080). While this is isolated by NetworkPolicy to controller-to-pod traffic only, the metrics endpoint has no authentication. This is typical for in-cluster metrics but differs from the secure-by-default pattern used for the controller's own metrics endpoint (8443/TCP with TLS). The MPI SSH key generation uses ECDSA P521, which is FIPS-approved, and the keys are stored in Kubernetes Secrets marked as immutable. The webhook TLS certificates are auto-rotated by cert-controller, with the CA bundle automatically injected into the ValidatingWebhookConfiguration.

The RHOAI distribution adds 15 ClusterTrainingRuntime variants covering PyTorch distributed training across three accelerator types (CPU, NVIDIA CUDA 12.8/13.0, AMD ROCm 6.4) and two framework generations (torch 2.9/2.9.1 with training-hub v0.5/v0.6). Three ImageStreams (CPU, CUDA, ROCm) are published for integration with the RHOAI workbench/notebook ecosystem. These images include the full ML training stack: PyTorch, transformers, accelerate, PEFT, TRL, DeepSpeed, and flash-attention (GPU variants). The kustomize overlay uses 18+ replacement configurations to parameterize all image references from a central ConfigMap, enabling single-point image updates during releases.
