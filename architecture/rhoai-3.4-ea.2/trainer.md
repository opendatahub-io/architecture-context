## Purpose
**Short**: Kubeflow Trainer is a Kubernetes-native operator for orchestrating distributed machine learning training jobs, supporting PyTorch, JAX, TensorFlow, MPI, and other frameworks.

**Detailed**: Kubeflow Trainer (v2) provides a CRD-based API for defining and managing distributed training workloads on Kubernetes. Users create `TrainJob` custom resources that reference pre-defined `TrainingRuntime` or `ClusterTrainingRuntime` templates, which encapsulate the specifics of distributed training frameworks (PyTorch torchrun, MPI, DeepSpeed, etc.). The operator reconciles these CRDs into `JobSet` resources (from the Kubernetes JobSet API) that manage the actual training pods.

The RHOAI distribution (Red Hat OpenShift AI) adds midstream extensions in the `pkg/rhai/` package, including real-time progression tracking that polls training pod HTTP metrics endpoints for live progress updates, and per-TrainJob NetworkPolicy enforcement for pod isolation and metrics endpoint security. The RHOAI overlay also ships pre-configured `ClusterTrainingRuntime` resources for various accelerator configurations (CUDA, ROCm, CPU) with multiple PyTorch and Training Hub versions, along with OpenShift `ImageStream` resources for workbench image management.

The operator integrates with gang-scheduling plugins (Kubernetes scheduler-plugins coscheduling, Volcano) and supports elastic training via PyTorch elastic policies with HPA-based autoscaling. It uses self-managed TLS certificates for webhook serving (via open-policy-agent/cert-controller) and exposes Prometheus metrics over HTTPS on port 8443.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| trainer-controller-manager | Go Operator (controller-runtime) | Main controller binary managing TrainJob, TrainingRuntime, and ClusterTrainingRuntime CRDs; creates JobSet resources, NetworkPolicies, and PodGroups dynamically |
| TrainJob Controller | Reconciler | Watches TrainJob CRs, resolves runtime references, creates JobSet objects via SSA, reconciles NetworkPolicy for pod isolation, and polls training metrics for progression tracking |
| TrainingRuntime Controller | Reconciler | Manages namespace-scoped TrainingRuntime CRs with resource-in-use finalizer protection |
| ClusterTrainingRuntime Controller | Reconciler | Manages cluster-scoped ClusterTrainingRuntime CRs with resource-in-use finalizer protection |
| Validating Webhooks | Admission Webhooks | Validates TrainJob, TrainingRuntime, and ClusterTrainingRuntime resources on CREATE/UPDATE |
| RHAI Progression Tracking | Extension (pkg/rhai/progression) | Polls training pod HTTP metrics endpoints for real-time progress, stores status in TrainJob annotations |
| RHAI NetworkPolicy | Extension (pkg/rhai/networkpolicy) | Creates per-TrainJob NetworkPolicies for pod isolation and metrics endpoint security |
| ClusterTrainingRuntimes (RHOAI) | Pre-configured CRs | 15 pre-built torch-distributed and training-hub runtimes for CUDA 12.8/13.0, ROCm 6.4, and CPU |
| ImageStreams (RHOAI) | OpenShift Resources | 3 ImageStreams (CPU, CUDA, ROCm) for training-hub workbench images |
| Certificate Manager | Internal Component | Self-managed TLS certificate rotation for webhook server via cert-controller |

## Data Flows

### Flow 1: TrainJob Creation and Reconciliation

| Step | Source | Destination | Port | Protocol | Encryption | Auth |
|------|--------|-------------|------|----------|------------|------|
| 1 | User/Client | Kubernetes API Server | 443/TCP | HTTPS | TLS 1.2+ | Bearer Token |
| 2 | Kubernetes API Server | Trainer Webhook | 9443/TCP | HTTPS | TLS (self-signed) | Kubernetes webhook auth |
| 3 | Trainer Controller | Kubernetes API Server | 443/TCP | HTTPS | TLS 1.2+ | ServiceAccount Token |
| 4 | Trainer Controller | Kubernetes API Server | 443/TCP | HTTPS | TLS 1.2+ | ServiceAccount Token |

Steps: (1) User creates TrainJob CR. (2) API server calls validation webhook. (3) Controller resolves runtime reference, builds JobSet spec. (4) Controller applies JobSet, PodGroup, and NetworkPolicy via server-side apply.

### Flow 2: RHAI Progression Tracking

| Step | Source | Destination | Port | Protocol | Encryption | Auth |
|------|--------|-------------|------|----------|------------|------|
| 1 | Trainer Controller | Kubernetes API Server | 443/TCP | HTTPS | TLS 1.2+ | ServiceAccount Token |
| 2 | Trainer Controller | Training Pod IP | 28080/TCP | HTTP | plaintext | NetworkPolicy-restricted |
| 3 | Trainer Controller | Kubernetes API Server | 443/TCP | HTTPS | TLS 1.2+ | ServiceAccount Token |

Steps: (1) Controller lists pods for the TrainJob to get pod IPs. (2) Controller polls `/metrics` endpoint on each ready training pod. (3) Controller updates TrainJob annotation `trainer.opendatahub.io/trainerStatus` with aggregated progress JSON.

### Flow 3: Prometheus Metrics Scrape

| Step | Source | Destination | Port | Protocol | Encryption | Auth |
|------|--------|-------------|------|----------|------------|------|
| 1 | Prometheus (via PodMonitor) | Trainer Controller Manager | 8443/TCP | HTTPS | TLS (self-signed, insecureSkipVerify) | None |
