## Purpose

**Short**: Intelligent inference request scheduler that routes LLM inference requests to optimal model-serving pods via Kubernetes Gateway API, with support for disaggregated Prefill/Decode (P/D) execution.

**Detailed**: The llm-d-inference-scheduler provides the Endpoint Picker (EPP) component for the llm-d inference framework. It extends the upstream Gateway API Inference Extension (GIE) project to make optimized routing decisions for incoming inference requests arriving through a Kubernetes Gateway. The EPP uses a pluggable architecture of filters, scorers, and profile handlers to select the best target pod based on criteria including KV-cache locality, load awareness, session affinity, and prefix cache state.

The project produces two distinct container images: the EPP itself (which runs as a gRPC external processor for Envoy, listening for ext-proc callbacks) and a P/D routing sidecar (which runs alongside vLLM decode workers to orchestrate disaggregated prefill/decode execution). The sidecar intercepts OpenAI-compatible API requests and coordinates between prefill and decode workers using one of three connector protocols: NIXLV2 (sequential with KV transfer), SGLang (concurrent with room-based coordination), or Shared Storage (deprecated, cache-hit-aware).

The scheduler integrates deeply with the Kubernetes Gateway API ecosystem. An InferencePool custom resource defines the set of model-serving pods and references the EPP Service. An HTTPRoute routes traffic through a Gateway to the InferencePool, and Envoy's ext-proc filter consults the EPP for endpoint selection on each request. The EPP also subscribes to KV-cache events from vLLM instances via ZMQ for precise prefix cache scoring.

## Architectural Analysis

The llm-d-inference-scheduler exhibits a clean separation between scheduling intelligence (EPP) and execution orchestration (P/D sidecar). The EPP is a pure decision-making service — it observes the cluster state (pods, KV-cache events, model metadata) and returns routing decisions via Envoy's ext-proc protocol without ever touching inference traffic directly. The sidecar, conversely, sits in the data path and handles the complex multi-step orchestration of disaggregated inference. This separation means the EPP can be scaled independently of the model-serving pods, and the sidecar can be upgraded or configured per-deployment without affecting scheduling logic.

The plugin architecture is well-designed for extensibility. The `EndpointPickerConfig` YAML format allows operators to compose different combinations of filters (by-label, by-label-selector, decode-filter, prefill-filter), scorers (precise-prefix-cache, load-aware, active-request, session-affinity, no-hit-lru), and profile handlers (single-profile, pd-profile, data-parallel) without code changes. The deploy/config directory ships nine pre-built configuration profiles covering common scenarios from simple decode-only routing to full P/D disaggregation with precise KV-cache scoring. The scheduling profiles support weighted scoring, allowing fine-tuned balance between cache locality and load distribution.

The P/D sidecar's three connector protocols (NIXLV2, SGLang, Shared Storage) represent different tradeoffs in the disaggregated inference space. NIXLV2 is sequential and deterministic — it completes the prefill phase fully before starting decode, transferring KV-cache metadata between stages. SGLang is concurrent — it launches both prefill and decode simultaneously using room-based coordination, trading determinism for lower latency. Shared Storage is deprecated but interesting architecturally — it attempted a speculative approach where decode starts first and falls back to P/D only if cache hit rate is too low. The sidecar's LRU cache of prefiller proxy handlers (size 16) and the atomic flag-based buffer/direct streaming switch in `responseWriterWithBuffer` show attention to performance in the hot path.

A notable FIPS discrepancy exists between the two images: the sidecar is built with `-tags strictfipsruntime` ensuring FIPS-compliant crypto, while the EPP is built with only `CGO_ENABLED=1` (required for ZMQ linking) without the FIPS tag. Both images will pass Konflux's check-payload gate (dynamically linked Go on UBI 9), but the EPP may use non-FIPS crypto paths at runtime. Since the EPP handles only cluster-internal gRPC traffic (no TLS by default) and the sidecar handles potentially sensitive inference traffic with optional TLS, this may be an intentional prioritization, but it should be documented for security review.

The SSRF protection mechanism in the sidecar (`AllowlistValidator`) is architecturally sound — it uses Kubernetes informers to maintain a live set of allowed prefiller targets based on InferencePool pod selectors, preventing a compromised request from redirecting prefill traffic to arbitrary hosts. The watch-based approach means the allowlist updates in near-real-time as pods are added or removed. The dual API group support (`inference.networking.k8s.io/v1` and `inference.networking.x-k8s.io/v1alpha2`) ensures compatibility during the GIE graduation from experimental to GA APIs.
