## Purpose
**Short**: Optimized inference request scheduler that routes LLM inference requests to model-serving pods via Gateway API Inference Extension (GIE) and Envoy ext-proc.

**Detailed**: The llm-d Inference Scheduler provides an Endpoint Picker (EPP) component for the llm-d inference framework. It schedules incoming inference requests arriving through a Kubernetes Gateway API by evaluating candidate model-serving pods using a pluggable pipeline of filters and scorers. Routing decisions consider KV cache locality, session affinity, load balancing, and model metadata to optimize request placement.

The scheduler extends the upstream Gateway API Inference Extension (GIE) project with llm-d-specific features, most notably disaggregated Prefill/Decode (P/D) execution. In P/D mode, the prefill (prompt processing) and decode (token generation) stages are routed to separate specialized pods. A routing sidecar (pd-sidecar) deployed alongside decode workers orchestrates communication between prefill and decode pods, handling KV cache transfer coordination via multiple connector protocols (NIXL v2, SGLang, shared storage).

The EPP operates as a gRPC ext-proc server that Envoy calls for each inference request. It watches InferencePool CRs to discover available model-serving pods, applies configured filter/scorer plugins, and returns the selected endpoint back to Envoy for routing. A companion UDS tokenizer sidecar provides tokenization services via Unix Domain Socket for precise prefix cache scoring.

### SSRF Protection

The PD routing sidecar implements SSRF protection via an InferencePool-based allowlist:
- Watches InferencePool CRs and their matching pods via Kubernetes informers
- Only allows prefill requests to pod IPs/names that belong to the configured InferencePool
- Enabled via `--enable-ssrf-protection` flag with `--inference-pool-namespace` and `--inference-pool-name`
- When disabled (default), all prefill targets are allowed for backward compatibility
