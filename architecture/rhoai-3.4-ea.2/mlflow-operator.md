## Purpose
**Short**: Kubernetes operator that deploys and manages MLflow tracking server instances with workspace isolation, Gateway API ingress, and Kubernetes-native authentication.

**Detailed**: The MLflow Operator is a Go-based Kubernetes operator that manages the lifecycle of MLflow tracking server deployments. It uses a cluster-scoped `MLflow` Custom Resource to define the desired state and renders Helm chart templates at reconciliation time to produce Kubernetes resources (Deployments, Services, NetworkPolicies, RBAC, PVCs) via Server-Side Apply.

The operator integrates with the RHOAI 3.x platform ingress pattern by creating `HTTPRoute` CRs (Gateway API v1) that reference the platform's `data-science-gateway` in the `openshift-ingress` namespace. URL path rewriting is configured so that MLflow API endpoints (`/mlflow/api/*`, `/mlflow/v1/*`) are served behind the gateway with proper prefix stripping. On OpenShift clusters, the operator also creates `ConsoleLink` resources for integration with the OpenShift web console application menu.

MLflow's built-in Kubernetes authorization plugin (`kubernetes-auth`) enforces RBAC via `SelfSubjectAccessReview` checks against pseudo-resources in the `mlflow.kubeflow.org` API group. A namespace-scoped `MLflowConfig` CRD allows workspace owners to configure per-namespace artifact storage credentials, enabling multi-tenant deployments with isolated S3-compatible object storage. The operator supports custom and platform CA bundles for TLS verification, configurable NetworkPolicy egress rules, and Prometheus metrics export via ServiceMonitor.

### Ingress

| Name | Type | Hosts | Port | Protocol | Encryption | TLS Mode | Exposure |
|------|------|-------|------|----------|------------|----------|----------|
| mlflow (HTTPRoute) | HTTPRoute (Gateway API v1) | Via data-science-gateway | 8443/TCP | HTTPS | TLS (gateway-terminated + backend TLS) | SIMPLE | External (via Gateway) |

The operator dynamically creates `HTTPRoute` CRs in the target namespace during reconciliation. Each HTTPRoute references the platform `data-science-gateway` Gateway in the `openshift-ingress` namespace. Three route rules are configured:
1. `/mlflow/api/*` -> rewrites to `/api/*` on backend service port 8443
2. `/mlflow/v1/*` -> rewrites to `/v1/*` on backend service port 8443
3. `/mlflow/*` -> passes through to backend service port 8443 (UI and static assets)

On OpenShift clusters, the operator also creates a `ConsoleLink` CR to add MLflow to the OpenShift web console application menu with an embedded SVG icon.

### Authentication & Authorization

| Endpoint | Methods | Auth Mechanism | Enforcement Point | Policy |
|----------|---------|----------------|-------------------|--------|
| /mlflow/api/* | All | Kubernetes ServiceAccount Token (SelfSubjectAccessReview) | MLflow kubernetes-auth middleware | RBAC check against mlflow.kubeflow.org pseudo-resources (experiments, datasets, registeredmodels, gatewayendpoints) |
| /mlflow/v1/* | All | Kubernetes ServiceAccount Token (SelfSubjectAccessReview) | MLflow kubernetes-auth middleware | Same as above |
| /metrics (operator) | GET | Bearer Token (TokenReview + SubjectAccessReview) | controller-runtime metrics server | metrics-reader ClusterRole required |
| /metrics (mlflow) | GET | None (scraped by Prometheus via ServiceMonitor) | ServiceMonitor TLS config | service-ca TLS verification on OpenShift |

The MLflow server uses `--app-name=kubernetes-auth` which enables the built-in Kubernetes authorization plugin. This plugin performs `SelfSubjectAccessReview` checks for each API request, mapping MLflow operations to RBAC verbs on pseudo-resources in the `mlflow.kubeflow.org` API group. The `MLFLOW_SERVER_DISABLE_SECURITY_MIDDLEWARE` env var is hardcoded to `"false"` and the CRD validation prevents users from overriding it.
