## Purpose

**Short**: A Kubernetes operator that deploys and manages MLflow tracking server instances on OpenShift/Kubernetes clusters, providing experiment tracking, model registry, and artifact management capabilities.

**Detailed**: The MLflow Operator is a controller-runtime based Go operator that reconciles cluster-scoped `MLflow` custom resources to deploy fully configured MLflow tracking server instances. It uses an embedded Helm chart to render Kubernetes resources (Deployments, Services, NetworkPolicies, PVCs, RBAC) and applies them via Server-Side Apply. The operator supports multiple concurrent MLflow instances differentiated by resource name suffixes.

The operator integrates deeply with the RHOAI platform: it creates Gateway API `HTTPRoute` resources to expose MLflow through the shared `data-science-gateway`, creates OpenShift `ConsoleLink` resources for the application menu, provisions TLS certificates via OpenShift's service-ca operator, and manages comprehensive CA bundle composition for secure connections to backend databases and artifact stores. MLflow server instances use Kubernetes-native RBAC authorization (via `SelfSubjectAccessReview`) and support multi-tenant workspaces backed by `MLflowConfig` namespace-scoped CRs for per-namespace artifact storage configuration.

The operator also manages a second CRD, `MLflowConfig` (group: `mlflow.kubeflow.org`), which enables namespace owners to configure per-namespace artifact storage credentials and paths, supporting a multi-tenant deployment model where a central MLflow server serves multiple data science teams.

|------|------|-------|------|----------|------------|----------|----------|

**Gateway API Integration (RHOAI 3.x pattern)**:
The operator dynamically creates `HTTPRoute` CRs (gateway.networking.k8s.io/v1) in the reconcile loop (`routing.go`). Each HTTPRoute:
- References parent Gateway `data-science-gateway` in `openshift-ingress` namespace
- Defines 3 rules with path-prefix matching:
  1. `/mlflow/api` -> URL-rewritten to `/api` on backend service port 8443
  2. `/mlflow/v1` -> URL-rewritten to `/v1` on backend service port 8443
  3. `/mlflow` -> Forwarded directly to backend service port 8443
- Supports multi-instance deployments with suffixed paths (e.g., `/mlflow-dev`)

### NetworkPolicy

The operator deploys a `NetworkPolicy` per MLflow instance:
- **Ingress**: Allows TCP on port 8443 from any pod in any namespace (cluster-wide service accessed by gateway, dashboard, Prometheus)
- **Egress**: Permits DNS (53), HTTPS (443), K8s API (6443), PostgreSQL (5432), MySQL (3306), MinIO (9000), SeaweedFS (8333)
- **Extensible**: Users can add custom egress rules via `spec.networkPolicyAdditionalEgressRules` (max 32 rules)

### Security Context (FIPS Compliance)

The operator binary is built with `GOEXPERIMENT=strictfipsruntime` and `-tags strictfipsruntime` for FIPS-compliant cryptographic operations. HTTP/2 is explicitly disabled in the operator's TLS configuration (`NextProtos: ["http/1.1"]`). On OpenShift, the operator sets `UVICORN_SSL_CIPHERS=PROFILE=SYSTEM` to use the system TLS cipher profile for the MLflow server.

All containers enforce:
- `runAsNonRoot: true`
- `readOnlyRootFilesystem: true`
- `allowPrivilegeEscalation: false`
- `capabilities.drop: ["ALL"]`
- `seccompProfile.type: RuntimeDefault`
