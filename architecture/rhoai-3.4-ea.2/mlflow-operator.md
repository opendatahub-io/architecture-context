## Purpose

**Short**: Kubernetes operator that manages the lifecycle of MLflow tracking server deployments on OpenShift AI and Open Data Hub platforms.

**Detailed**: The MLflow Operator is a controller-runtime based Kubernetes operator that watches cluster-scoped `MLflow` custom resources and reconciles them into fully operational MLflow tracking server deployments. It uses an embedded Helm chart as a templating engine to render Kubernetes resources (Deployments, Services, NetworkPolicies, PVCs, RBAC, ServiceMonitors) and applies them via Server-Side Apply.

The operator integrates with the RHOAI 3.x platform ingress architecture by conditionally creating Gateway API `HTTPRoute` resources that reference the platform's `data-science-gateway` Gateway. It also integrates with the OpenShift console by creating `ConsoleLink` resources for the application menu. The operator supports flexible storage backends including SQLite (local PVC), PostgreSQL, MySQL, and S3-compatible object storage for artifacts. It manages CA bundle aggregation through init containers and sidecar watchers, and enforces network security through automatically-generated NetworkPolicies.

A second CRD, `MLflowConfig` (group `mlflow.kubeflow.org`), provides namespace-scoped configuration allowing workspace owners to specify per-namespace artifact storage credentials and paths. The operator's MLflow server uses Kubernetes-native authentication (`--app-name=kubernetes-auth`) with SelfSubjectAccessReview-based authorization, enforcing RBAC through pseudo-resources defined in aggregate ClusterRoles.

## Architectural Analysis

The MLflow Operator employs an unusual but effective pattern of using Helm as a pure template engine rather than a deployment tool. The operator embeds a complete Helm chart (`charts/mlflow/`) in its container image and renders it in-memory during each reconciliation cycle using `helm.sh/helm/v3/pkg/engine`. The rendered unstructured objects are then applied to the cluster via Server-Side Apply with `mlflow-operator` as the field owner. This approach gives the operator the expressiveness of Helm templates (conditionals, loops, helper functions) while retaining the operational semantics of a reconciling controller (drift correction, status management, owner references). The tradeoff is that the chart must be kept in sync with the controller logic — the controller adds resources beyond what the chart produces (ConsoleLink, HTTPRoute) through dedicated reconcile methods.

The operator's CRD design splits configuration across two API groups: `mlflow.opendatahub.io` for the cluster-scoped operator CR (`MLflow`) and `mlflow.kubeflow.org` for the namespace-scoped workspace configuration (`MLflowConfig`). The MLflow CR is cluster-scoped but deploys resources into a single target namespace configured via the `--namespace` flag. This is a deliberate design choice — it allows a single MLflow instance to serve all namespaces via the workspace feature while keeping the deployment footprint contained. The MLflowConfig CRD is read by the MLflow server itself (not the operator) through its `--workspace-store-uri=kubernetes://` mechanism, which uses the server's ServiceAccount to list namespaces and read MLflowConfig resources.

Security is enforced at multiple layers. The MLflow server uses Kubernetes-native authentication (`--app-name=kubernetes-auth`) which validates Bearer tokens against the Kubernetes API via SelfSubjectAccessReview. Authorization is implemented through pseudo-resources (experiments, datasets, registeredmodels, gatewayendpoints, gatewaymodeldefinitions) in the `mlflow.kubeflow.org` API group — these aren't actual CRDs but permission targets checked via RBAC. The aggregate ClusterRoles (`mlflow-view`, `mlflow-edit`, `mlflow-integration`) automatically compose into the standard Kubernetes admin/edit/view roles, giving cluster admins a familiar permission model. The MLFLOW_SERVER_DISABLE_SECURITY_MIDDLEWARE env var is explicitly blocked via CEL validation on the CRD spec. Network isolation is enforced via a NetworkPolicy that allows ingress only on port 8443 from any namespace (necessary for the cross-namespace gateway) and restricts egress to specific protocols and ports.

The CA bundle management system is notably sophisticated for a component operator. When platform or custom CA ConfigMaps are detected, the Helm chart renders an init container that concatenates system, platform, and custom CA certificates into a single PEM file, plus a sidecar container that polls for changes every 30 seconds and regenerates the combined bundle. This combined bundle is then injected into the MLflow container via environment variables (SSL_CERT_FILE, REQUESTS_CA_BUNDLE, CURL_CA_BUNDLE, AWS_CA_BUNDLE, PGSSLROOTCERT) covering Python requests, curl, boto3, and psycopg2 libraries. The operator controller also watches the `odh-trusted-ca-bundle` ConfigMap and triggers reconciliation when it appears or disappears, ensuring the Deployment spec is updated accordingly.

The operator dynamically detects cluster capabilities at startup using the Kubernetes discovery API. It checks for ConsoleLink, HTTPRoute, and ServiceMonitor CRD availability and configures both the controller watches and the cache accordingly. This allows the same operator binary to run on vanilla Kubernetes (no ConsoleLinks, no HTTPRoutes, no ServiceMonitors), OpenShift without Gateway API (ConsoleLinks only), or full RHOAI (all features enabled). The conditional cache configuration (`cache.ByObject`) with label selectors (`app: mlflow`) ensures the operator only watches resources it owns, reducing API server load.
