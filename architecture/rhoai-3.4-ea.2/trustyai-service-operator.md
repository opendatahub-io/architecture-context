## Purpose

**Short**: Kubernetes operator that manages the lifecycle of TrustyAI AI explainability services, LM evaluation jobs, guardrails orchestration, NeMo Guardrails, and the EvalHub evaluation platform within RHOAI.

**Detailed**: The TrustyAI Service Operator is a multi-controller Kubernetes operator built with controller-runtime that manages five distinct CRDs for AI trustworthiness and evaluation workloads in Red Hat OpenShift AI. It deploys and configures: (1) TrustyAI explainability services that integrate with KServe InferenceServices and ModelMesh to monitor model predictions for bias and fairness metrics, (2) LMEvalJob resources that orchestrate language model evaluation tasks using the lm-evaluation-harness framework with Kueue-based job scheduling, (3) GuardrailsOrchestrator resources that deploy a multi-container guardrails stack with optional gateway and built-in detectors for LLM safety, (4) NemoGuardrails resources for deploying NVIDIA NeMo Guardrails servers, and (5) EvalHub resources providing a centralized evaluation platform with multi-tenant support and pluggable evaluation providers.

The operator runs as a single binary with selectable controller services (TAS, LMES, GORCH, NEMO_GUARDRAILS, EVALHUB). Each controller manages its own set of Kubernetes resources (Deployments, Services, Routes, ConfigMaps, Secrets, ServiceAccounts, RBAC) and integrates with the broader RHOAI platform through KServe InferenceServices, Istio service mesh, Kueue job scheduling, Prometheus monitoring, and OpenShift Routes. The operator injects kube-rbac-proxy sidecars for authentication enforcement on managed service endpoints.

## Architectural Analysis

The TrustyAI Service Operator is a single-binary, multi-controller operator that manages five architecturally distinct workloads under one umbrella. This design follows the "umbrella operator" pattern where related-but-separate CRDs share a single operator deployment to reduce cluster footprint and simplify lifecycle management. The `--enable-services` flag allows deploying subsets of controllers, enabling the same codebase to serve both full RHOAI deployments and standalone LMES-only configurations.

The operator's authentication pattern is consistently kube-rbac-proxy-based across all managed services (TAS, GORCH, and NemoGuardrails). Each controller injects a kube-rbac-proxy sidecar container that terminates TLS and performs SubjectAccessReview (SAR) authorization before proxying requests to the application container on localhost. The kube-rbac-proxy config templates explicitly restrict cipher suites to FIPS 140-2 approved algorithms with TLS 1.2 minimum. EvalHub is a notable exception — it serves TLS directly without a kube-rbac-proxy sidecar, instead relying on OpenShift serving certificates for TLS termination.

The LMEvalJob controller implements a sophisticated job execution model using a driver sidecar pattern. Rather than running evaluations directly, the controller creates Pods with an init container (ta-lmes-driver) that sets up the evaluation environment and an HTTP status API. The controller then polls the driver via `kubectl exec` to monitor progress — an unusual but effective approach that avoids exposing network endpoints from evaluation pods. The Kueue integration via the JobManager controller enables resource-aware scheduling without modifying the core LMES controller logic.

The GuardrailsOrchestrator controller is the most complex sub-controller, managing a multi-container pod with up to four containers (orchestrator, gateway, built-in detectors, kube-rbac-proxy) plus dynamic ConfigMap generation via autoconfig. The autoconfig feature watches KServe InferenceServices to automatically discover generation and detector services, generating orchestrator YAML configuration that wires services together. This represents a significant amount of dynamic infrastructure provisioning that goes beyond typical operator patterns. The GORCH controller creates up to four separate OpenShift Routes (main, health, gateway, built-in) with different TLS termination modes.

A notable security design decision is the RHOAI overlay's restriction of LMEvalJob capabilities. The RHOAI params.env disables both `lmes-allow-online` (preventing runtime downloads) and `lmes-allow-code-execution` (preventing arbitrary code execution), while the base/ODH overlay permits both. This defense-in-depth approach protects RHOAI clusters from supply chain risks in evaluation workloads. The operator also implements extensive input validation for LMEvalJob specs, including regex-based model name validation, shell metacharacter detection, HTTPS-only Git URLs, and protected environment variable stripping.
