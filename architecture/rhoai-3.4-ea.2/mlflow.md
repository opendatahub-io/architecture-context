## Purpose

**Short**: MLflow is an ML experiment tracking, model registry, and artifact management platform customized for multi-tenant Kubernetes-native deployment within Red Hat OpenShift AI.

**Detailed**: MLflow provides a comprehensive platform for managing the machine learning lifecycle, including experiment tracking, model versioning, artifact storage, and model deployment. The Red Hat fork (version 3.10.1+rhaiv.1) extends upstream MLflow with Kubernetes-native multi-tenancy: workspaces map 1:1 to Kubernetes namespaces, and authorization is enforced via Kubernetes RBAC through SelfSubjectAccessReview and SubjectAccessReview API calls.

The server exposes a REST API and GraphQL endpoint backed by SQLAlchemy (supporting PostgreSQL and SQLite), serves a React-based web UI with Module Federation support for embedding in the OpenShift/ODH console, and provides artifact proxying to S3-compatible object storage. The Kubernetes workspace provider plugin discovers namespaces via a background watch loop, supports per-namespace artifact storage overrides through an optional MLflowConfig CRD, and integrates with kube-rbac-proxy for header-based authentication in the RHOAI 3.x ingress pattern.

The RHOAI build uses Konflux CI/CD with Cachi2 integration, UBI9 base images, multi-architecture support (amd64, arm64, ppc64le), and dependencies sourced from Red Hat's AIPCC (AI Platform Community Content) Python package index.

## Architectural Analysis

MLflow in RHOAI represents a substantial extension of the upstream open-source project. While upstream MLflow is designed primarily as a single-user or team-level tool with optional basic authentication, the RHOAI fork introduces a full Kubernetes-native multi-tenancy model. The key architectural innovation is the `kubernetes-workspace-provider` plugin, which maps MLflow's workspace abstraction 1:1 to Kubernetes namespaces. This design elegantly reuses existing Kubernetes RBAC infrastructure — no separate user database or permission system is needed. Authorization decisions are delegated to the Kubernetes API server via SubjectAccessReview or SelfSubjectAccessReview calls, with a TTL-based cache (300s default) to avoid excessive API server load. The authorization model covers both REST API endpoints (mapped via protobuf message types) and GraphQL queries (mapped via operation name analysis with AST traversal), and workspace mutations (create/update/delete) are explicitly denied since namespace lifecycle is owned by cluster operators.

The server architecture uses a dual-framework approach: Flask handles the traditional REST API, static file serving, and the basic auth plugin system, while FastAPI handles async workloads including OpenTelemetry trace ingestion, job execution, and the AI gateway router. A WSGIMiddleware bridge connects the two frameworks. The production container uses Uvicorn as the ASGI server by default, with Gunicorn available as an alternative for WSGI-only deployments. This hybrid approach allows the codebase to gradually migrate from Flask to FastAPI while maintaining backward compatibility.

The frontend build produces two artifacts: a standard React SPA (`yarn build`) and a Module Federation bundle (`yarn build:federated`) that exposes `MlflowExperimentWrapper` and `MlflowPromptsWrapper` as remote modules. The federated build deliberately does NOT share react-router-dom because the host (OpenShift console) uses v7 while MLflow uses v6 — this isolation prevents routing conflicts. The `MLFLOW_API_BASE_URL=/mlflow` build variable ensures API calls route correctly when embedded behind the platform's reverse proxy.

The encryption system for secrets uses envelope encryption with AES-256-GCM and PBKDF2-HMAC-SHA256 key derivation (600,000 iterations). While the implementation is cryptographically sound and OWASP-compliant, the development-mode default KEK passphrase is hardcoded — production deployments MUST override `MLFLOW_CRYPTO_KEK_PASSPHRASE`. MD5 usage throughout the codebase (for content hashing in digest_utils, file_store, etc.) consistently uses `usedforsecurity=False`, which is FIPS-safe on Python 3.9+.

Build hermeticity is partially achieved: Python dependencies are hash-locked via AIPCC and cachi2 prefetch, RPMs are checksummed via rpms.lock.yaml (covering aarch64, x86_64, ppc64le), and the uv.lock file pins all Python development dependencies. However, the JavaScript/frontend build is not yet hermetic — the Tekton pipeline sets `hermetic: false` because Konflux doesn't yet support yarn/npm prefetching. This is a known supply chain gap that will be addressed when Konflux adds npm support.
