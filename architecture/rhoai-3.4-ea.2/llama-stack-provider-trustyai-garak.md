## Purpose

**Short**: An out-of-tree Llama Stack evaluation provider that integrates NVIDIA Garak red-teaming framework for automated LLM vulnerability scanning and security assessment.

**Detailed**: This component implements a Llama Stack `Eval` API provider that wraps the NVIDIA Garak red-teaming framework to perform automated vulnerability scanning of Large Language Models. It enables users to run predefined compliance benchmarks (OWASP LLM Top 10, AVID taxonomy, CWE) and custom red-teaming probes against LLMs deployed on OpenShift AI, producing detailed security assessment reports including attack success rates, TBSA (Tier-Based Security Aggregate) scores, and HTML/JSONL reports.

The provider supports three execution modes: **inline** (local subprocess execution within the Llama Stack server), **remote** (delegated to Kubeflow Pipelines for production-scale scanning on OpenShift AI), and **eval-hub** (integration with the Red Hat evaluation platform). The remote/KFP mode implements a 6-step DAG pipeline (validate, resolve taxonomy, SDG generate, prepare prompts, garak scan, parse results) that can run custom intent-based risk assessments using Synthetic Data Generation (SDG) to produce adversarial prompts from harm taxonomies.

The component ships as a single container image (`odh-trustyai-garak-lls-provider-dsp-rhel9`) that serves as both the Llama Stack provider plugin and the KFP pipeline base image. It integrates with the broader RHOAI platform via the `trustyai-service-operator-config` ConfigMap for image resolution and the Llama Stack Distribution operator for deployment.

## Architectural Analysis

This component demonstrates a sophisticated multi-execution-mode architecture designed to bridge the gap between local development (inline mode), production Kubernetes deployments (KFP remote mode), and platform-level evaluation orchestration (eval-hub mode). The core red-teaming logic is intentionally factored into framework-agnostic modules (`core/command_builder.py`, `core/config_resolution.py`, `core/garak_runner.py`, `core/pipeline_steps.py`) that are then consumed by three independent adapters. This design ensures that benchmark resolution, configuration merging, and result parsing are consistent regardless of execution path.

The deep-merge configuration system is a key architectural pattern. Predefined scan profiles (OWASP LLM Top 10, AVID, CWE, etc.) are defined in `GarakScanConfig.FRAMEWORK_PROFILES` with sensible defaults, but every aspect can be overridden via the `metadata.garak_config` field when registering benchmarks. The `deep_merge_dicts` utility ensures that users can surgically override specific nested values (e.g., changing only `run.soft_probe_prompt_cap`) without losing the rest of the profile configuration. This is elegant but creates a complex precedence chain: user overrides > existing benchmark overrides > profile defaults > `GarakCommandConfig` defaults.

The intent-based risk assessment (ART) pipeline is the most architecturally complex feature. It introduces a full SDG (Synthetic Data Generation) stage that generates adversarial prompts from a taxonomy of harm categories, feeds them through Garak's Context Aware Scanning (CAS) system with an EarlyStop harness, and then computes per-intent attack success rates. This requires orchestrating multiple LLM roles (target, judge, attacker, evaluator) with separate model endpoints and API keys, resolved via a layered secret system (`model-auth` K8s Secret mounted at `/mnt/model-auth` in KFP pods, with per-role fallback chain). The `_resolve_config_api_keys` function handles placeholder substitution at pod level, while `redact_api_keys` sanitizes configs before persistence -- a defense-in-depth approach to API key management.

The shield scan orchestrator (`shield_scan.py`) implements a module-level singleton pattern (`simple_shield_orchestrator`) that Garak calls as a `function.Single` generator. This allows transparent interception of all prompts and responses with configurable input/output shields (e.g., Prompt-Guard, Llama-Guard), measuring guardrail effectiveness against adversarial probes. The orchestrator lazily initializes its `LlamaStackClient` with configurable TLS verification, and the singleton ensures connection reuse across the scan lifecycle.

A notable risk area is the Rust/Cargo dependency structure. The `Cargo.toml`, `Cargo.lock`, and `.cargo/config.toml` at the repository root exist solely as workarounds for Hermeto build system bugs (documented in `.konflux/README.md`). The `main.rs` is an empty stub (`fn main() {}`). This is fragile -- it couples the build to Hermeto implementation details and could break if Hermeto fixes the underlying issues. However, it is currently necessary for hermetic builds of Python packages that transitively depend on Rust crates (e.g., `tiktoken`, `tokenizers`).
