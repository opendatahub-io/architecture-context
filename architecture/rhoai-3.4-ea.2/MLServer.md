## Purpose

**Short**: MLServer is a multi-model ML inference server implementing the KFServing V2 / Open Inference Protocol, providing REST and gRPC endpoints for model serving with pluggable ML framework runtimes.

**Detailed**: MLServer is an open-source inference server originally developed by Seldon Technologies, forked and maintained by Red Hat for inclusion in Red Hat OpenShift AI (RHOAI). It serves as one of the model serving runtimes available in the RHOAI platform, enabling users to deploy machine learning models behind standardized V2 Dataplane inference APIs.

The server supports multiple ML framework runtimes (scikit-learn, XGBoost, LightGBM, ONNX Runtime) as pluggable backends, with each runtime packaged as a separate Python package (`mlserver-sklearn`, `mlserver-xgboost`, `mlserver-lightgbm`, `mlserver-onnx`). The architecture provides dual-protocol access (REST via FastAPI/Uvicorn on port 8080 and gRPC on port 8081), parallel multi-process inference to bypass the Python GIL, adaptive request batching, response caching, Prometheus metrics (port 8082), OpenTelemetry distributed tracing, and optional Kafka-based event-driven inference. MLServer is deployed as a container image (`odh-mlserver-rhel9`) within KServe InferenceService pods, where it loads models from a mounted volume at `/mnt/models`.

## Architectural Analysis

MLServer follows a clean layered architecture with protocol abstraction at its core. The `DataPlane` class serves as the central business logic layer, decoupling inference operations from transport protocols. Both the REST (FastAPI) and gRPC servicers delegate to the same DataPlane, ensuring consistent behavior regardless of access protocol. This pattern also enables the Kafka server integration to reuse the same inference path. The middleware chain (currently CloudEvents only, but extensible) wraps all inference operations uniformly across protocols.

The parallel worker architecture is particularly well-designed for Python's limitations. By using `multiprocessing.Process` with queue-based communication, MLServer achieves true parallelism for CPU-bound inference workloads that would otherwise be serialized by the GIL. The round-robin dispatcher, worker auto-restart on failure, and per-worker model registry isolation demonstrate production-grade reliability. Each worker maintains its own event loop with uvloop, and the main process handles coordination through a shared response queue with future-based tracking. The architecture also supports custom Python environments per model — a critical feature when different models require conflicting dependency versions.

The adaptive batching system is transparent to runtime implementations. It intercepts `predict()` calls via the hook/callback pattern during model loading, accumulates requests into batches based on configurable size and time thresholds, and splits responses back to individual callers. This design means runtime authors need not be aware of batching — they simply implement `predict()` for single requests, and the framework handles batching automatically. However, streaming inference (`predict_stream`) is explicitly excluded from batching, and response caching is local-memory-only (FIFO eviction with no distributed cache support).

Security is entirely delegated to the platform. MLServer provides no built-in authentication, authorization, or TLS — all three ports (8080, 8081, 8082) bind plaintext on `0.0.0.0`. This is by design for Kubernetes deployment where TLS termination occurs at the ingress/gateway layer, and authentication is handled by kube-rbac-proxy sidecars or Istio mTLS. However, this means MLServer should never be exposed directly outside the cluster without an authenticating proxy in front of it.

The RHOAI-specific build pipeline demonstrates strong supply chain security practices. Dependencies are SHA256-pinned in `requirements/requirements-cpu.txt` with multi-arch support (x86_64 and aarch64). The Konflux/Tekton pipeline enforces hermetic builds via Cachi2 prefetch, meaning no external package downloads occur during the container build. The base image (`quay.io/aipcc/base-images/cpu:3.4.0-ea.2`) is a curated RHEL9-based image maintained by the AIPCC team. The dependency pinning script (`hack/generate-pinned-requirements.py`) implements a two-phase resolution and artifact collection process, ensuring reproducibility across build environments. One gap is the absence of `rpms.lock.yaml` — OS-level packages are inherited from the base image without independent locking at this layer.
