## Purpose
**Short**: MLServer is a Python-based inference server implementing the V2 Inference Protocol (KServe Data Plane) for serving machine learning models over REST and gRPC interfaces.

**Detailed**: MLServer is a multi-runtime inference server originally developed by Seldon Technologies, forked and maintained by Red Hat for integration into Red Hat OpenShift AI (RHOAI). It provides a standardized way to serve ML models using the V2 Inference Protocol (also known as the KServe Data Plane protocol), which defines a common REST and gRPC API for model inference across different ML frameworks.

The server supports multiple inference runtimes through a plugin architecture, with the RHOAI distribution shipping support for scikit-learn, XGBoost, LightGBM, and ONNX runtimes. Key features include adaptive request batching, parallel inference workers via multiprocessing, an optional Kafka interface for event-driven inference, response caching, OpenTelemetry distributed tracing, and Prometheus metrics exposition. MLServer is typically deployed as a container within KServe's InferenceService pods, serving as the model-serving backend for the RHOAI model serving stack.

The server runs three concurrent network servers: a FastAPI-based REST server (default port 8080), a gRPC inference server (default port 8081), and a Prometheus metrics server (default port 8082). Models are loaded from a configurable model repository directory (`/mnt/models` by default in the container image).

### Ingress

MLServer does not define its own ingress resources. Ingress is managed by KServe, which creates:
- In RHOAI 3.x: HTTPRoute CRs (Gateway API) referencing the platform gateway, with kube-rbac-proxy for authentication
- In RHOAI 2.x: OpenShift Routes with oauth-proxy for authentication

MLServer listens on `0.0.0.0` (all interfaces) and relies on the platform (KServe + RHOAI operator) to handle TLS termination, authentication, and external exposure.

No RBAC roles defined. MLServer is a non-privileged application container, not a Kubernetes operator. RBAC is managed at the KServe/RHOAI operator level.

### Secrets

No secrets are directly consumed or managed by MLServer. Model access credentials and TLS certificates are handled by the platform (KServe, service mesh).

### Authentication & Authorization

| Endpoint | Methods | Auth Mechanism | Enforcement Point | Policy |
|----------|---------|----------------|-------------------|--------|
| All REST endpoints | GET, POST | None (application level) | KServe / Platform | Platform handles auth via kube-rbac-proxy (3.x) or oauth-proxy (2.x) |
| All gRPC endpoints | gRPC | None (application level) | KServe / Platform | Platform handles auth |

MLServer itself does not implement authentication or authorization. These are enforced at the platform level through:
- **RHOAI 3.x**: kube-rbac-proxy sidecars injected by KServe/RHOAI operator
- **RHOAI 2.x**: oauth-proxy sidecars

### Container Security

- Runs as non-root user (UID 1000)
- Workdir: `/opt/mlserver`
- `/opt/mlserver` directory is world-writable (chmod 776) for compatibility with arbitrary UIDs (OpenShift security model)

### CORS

CORS is configurable but disabled by default. When enabled, supports origin allowlisting, credential forwarding, method/header restrictions, and max-age caching (default 600 seconds).
