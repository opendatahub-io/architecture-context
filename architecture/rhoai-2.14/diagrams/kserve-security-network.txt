SECURITY NOTES
═══════════════════════════════════════════════════════════════════════════════════

1. **Webhook Security**: kserve-webhook-server uses mTLS (mutual TLS) for communication
   with kube-apiserver. Certificates are auto-provisioned by cert-manager.

2. **Model Inference Authentication**: Optional. Users can configure Istio
   AuthorizationPolicy and RequestAuthentication for Bearer token validation.

3. **Cloud Storage Authentication**: Prefer IRSA (AWS) or Workload Identity (GCP) over
   static access keys. Static keys in secrets require manual rotation.

4. **Service Mesh mTLS**: Optional but recommended. Configure PeerAuthentication to
   STRICT mode for encrypted service-to-service communication.

5. **Scale-to-Zero Cold Start**: When a pod scales from zero, there is latency for
   model download (storage-initializer) and model loading. This may expose timing
   side-channels. Consider disabling scale-to-zero for sensitive models.

6. **Secrets Management**: storage-config secrets contain plaintext cloud credentials.
   Consider using external secret management (e.g., Vault, AWS Secrets Manager) with
   External Secrets Operator.

7. **Network Isolation**: Use NetworkPolicies to restrict egress to only required
   cloud storage endpoints. Monitor for data exfiltration.

8. **Model Artifact Integrity**: No built-in signature verification for model artifacts.
   Consider implementing checksum validation in storage-initializer or using
   signed containers.
