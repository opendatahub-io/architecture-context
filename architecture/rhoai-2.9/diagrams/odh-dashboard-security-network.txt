Authentication Flow:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. User → OpenShift Route (HTTPS/443, TLS 1.2+)
2. Route → OAuth Proxy (HTTPS/8443, TLS reencrypt, service cert)
3. OAuth Proxy → OpenShift OAuth (validates session/token)
4. OAuth Proxy → RBAC Check (user must have 'get' on odh-dashboard service)
5. OAuth Proxy → Backend (HTTP/8080 localhost, x-forwarded-access-token header)
6. Backend → Kubernetes API (HTTPS/6443, Service Account Token for dashboard ops)
7. Backend → Thanos/Other Services (HTTPS, User Bearer Token forwarded for user-scoped queries)

Security Considerations:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. OAuth Token Forwarding: User access tokens forwarded to backend via
   x-forwarded-access-token header for Kubernetes API calls with user context

2. Service Account Permissions: Dashboard SA has broad cluster-level read
   permissions and namespace-level write permissions for managing resources

3. Secret Management: OAuth secrets (cookie_secret, client_secret) are NOT
   auto-rotated - manual rotation required

4. TLS Termination: Route uses TLS reencrypt mode - backend must present
   valid service-ca TLS cert

5. RBAC Delegation: OAuth proxy validates user has 'get' permission on
   odh-dashboard service in dashboard namespace before allowing access
