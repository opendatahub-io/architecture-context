NETWORK POLICIES
═══════════════════════════════════════════════════════════════════════════════════

RHODS Operator operates in a cluster-scoped context and does not define specific
NetworkPolicies for itself. However, it creates and manages NetworkPolicies for
deployed components as part of the DSCInitialization infrastructure setup.

Operator egress requirements:
- Kubernetes API Server: 6443/TCP (HTTPS) - Required for all operations
- Component Git Repositories: 443/TCP (HTTPS) - Build-time only (devFlags)
- Container Registries: 443/TCP (HTTPS) - Runtime image pulls

Operator ingress requirements:
- Prometheus metrics: 8443/TCP (via Service) - Cluster monitoring
- Health probes: 8081/TCP - Kubelet health checks (pod-internal)

═══════════════════════════════════════════════════════════════════════════════════
SERVICE MESH INTEGRATION
═══════════════════════════════════════════════════════════════════════════════════

The RHODS Operator itself does NOT run within the service mesh. However, it is
responsible for:

1. Creating ServiceMeshMember resources for component namespaces
2. Configuring service mesh settings via DSCInitialization CR
3. Managing Istio resources (VirtualServices, Gateways, AuthorizationPolicies)
   for deployed components

Component namespaces managed by operator (added to service mesh):
- redhat-ods-applications (dashboard, workbenches)
- User namespaces (for KServe inference services)

Service mesh configuration:
- Operator has full RBAC permissions for maistra.io, networking.istio.io, security.istio.io API groups
- Components like KServe require service mesh for production deployments
- Service mesh provides mTLS encryption between component services

HIGH AVAILABILITY & LEADER ELECTION
═══════════════════════════════════════════════════════════════════════════════════

Leader Election: Enabled (lease-based)
├─ Mechanism: Kubernetes Lease API
├─ Replicas: 1 (single active instance)
├─ Purpose: Prevent split-brain, ensure single reconciliation instance
└─ Recovery: Automatic pod restart on failure, new leader elected

Note: Only one operator pod is active at a time. If multiple replicas are deployed,
only the leader performs reconciliation. This prevents conflicting updates to
cluster resources.
