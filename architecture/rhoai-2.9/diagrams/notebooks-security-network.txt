RBAC Summary:
- Notebook pods run with user-provided ServiceAccounts
- ServiceAccount permissions scoped to user namespace
- Inherited RBAC from Notebook Controller component
- Namespace viewer role required for /api endpoint access
- OpenShift RBAC enforced by OAuth Proxy for web interface

Service Mesh Configuration:
- Model Registry and KFP API protected by mTLS via Service Mesh sidecars
- OAuth Proxy handles authentication for external notebook access
- Internal pod-to-pod traffic uses HTTP over localhost (sidecar pattern)

Secrets:
- {notebook-name}-oauth-config (Opaque) - OAuth proxy config, Notebook Controller, No rotation
- {notebook-name}-tls (kubernetes.io/tls) - Route TLS cert, cert-manager/OpenShift, Auto-rotate
- git-credentials (kubernetes.io/ssh-auth) - Git auth, User provisioned, No rotation
- aws-credentials (Opaque) - S3 credentials (AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY), User provisioned, No rotation
- image-pull-secret (kubernetes.io/dockerconfigjson) - Private registry access, Cluster admin, No rotation

Network Policies:
- Allow ingress from OpenShift Router to notebook pods (port 8888/8787/8443)
- Allow egress to Kubernetes API (6443)
- Allow egress to S3, PyPI, GitHub (443)
- Allow egress to Model Registry and KFP within cluster (8080, 8888)

Image Versioning:
- ImageStreams manage N, N-1, N-2, N-3 releases
- Images referenced by SHA256 digest for immutability
- ConfigMaps track image commits and versions
- Pull from quay.io/modh/* with image-pull-secret

Deployment Model:
- Notebook Controller creates StatefulSets from ImageStream references
- Users select workbench images via ODH Dashboard
- Each notebook instance runs in user namespace with dedicated ServiceAccount
- PersistentVolumeClaims provide storage persistence across pod restarts
