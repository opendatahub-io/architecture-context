RBAC Summary:

Cluster Roles:
- manager-role (DSPO Controller):
  - datasciencepipelinesapplications.opendatahub.io: * (all verbs)
  - pipelines.kubeflow.org: * (all verbs)
  - argoproj.io: workflows, workflowartifactgctasks, workflowtaskresults (all verbs)
  - core: configmaps, secrets, serviceaccounts, persistentvolumeclaims, services (all verbs)
  - core: pods, pods/exec, pods/log (get, list, delete)
  - apps: deployments, replicasets (all verbs)
  - rbac.authorization.k8s.io: roles, rolebindings, clusterroles, clusterrolebindings (all verbs)
  - route.openshift.io: routes (all verbs)
  - networking.k8s.io: networkpolicies (all verbs)
  - monitoring.coreos.com: servicemonitors (all verbs)
  - serving.kserve.io: inferenceservices (create, delete, get, list, patch)
  - batch: jobs (all verbs)

Namespace-scoped Roles (Per DSPA instance):
- ds-pipeline-{name}: API server permissions
- ds-pipeline-persistenceagent-{name}: Persistence agent permissions
- ds-pipeline-scheduledworkflow-{name}: Scheduled workflow controller permissions
- ds-pipeline-argo-role-{name}: Argo workflow controller permissions
- ds-pipeline-pipeline-runner-role-{name}: Workflow pod execution permissions

Service Mesh Configuration:
- Network Policies enforced for namespace isolation
- Optional mTLS between components (via service-cert annotations)
- Pod-to-pod encryption supported via TLS certificates

Secrets (Auto-rotation via OpenShift service-cert):
- ds-pipelines-proxy-tls-{name} (kubernetes.io/tls) - OAuth proxy TLS cert
- ds-pipelines-mariadb-tls-{name} (kubernetes.io/tls) - MariaDB TLS cert
- ds-pipelines-envoy-proxy-tls-{name} (kubernetes.io/tls) - MLMD Envoy TLS cert
- ds-pipeline-metadata-grpc-tls-certs-{name} (kubernetes.io/tls) - MLMD gRPC TLS cert
- {custom-db-secret} (Opaque) - Database credentials (user-provided or auto-generated)
- {custom-storage-secret} (Opaque) - Object storage credentials (S3 access/secret keys)

Authentication Mechanisms:
- External API access: OpenShift OAuth → Bearer Token → Kubernetes RBAC
- Internal API access: Bearer Token (JWT) → TokenReview/SubjectAccessReview
- Database: Username/Password from secret
- Object Storage: Access Key/Secret Key from secret or AWS IAM
- MLMD gRPC: NetworkPolicy-based access control (no explicit auth)
- Kubernetes API: ServiceAccount Token

Network Policies:
- ds-pipelines-{name}: Restricts API server access to authorized sources
- mariadb-{name}: Restricts DB access to DSP components only
- ds-pipeline-metadata-grpc-{name}: Restricts MLMD gRPC to pipeline tasks
- ds-pipelines-envoy-{name}: Restricts MLMD Envoy access

Metrics Collection:
- Prometheus (OpenShift Monitoring) → DSPO /metrics (8080/TCP)
- Prometheus → API Server /metrics (8888/TCP, if ServiceMonitor deployed)
- Auth: ServiceAccount Token via Kube RBAC Proxy

Key Security Features:
1. Namespace Isolation: Each DSPA instance fully isolated within namespace
2. Optional mTLS: Pod-to-pod encryption via service-cert annotations
3. OAuth Proxy: OpenShift OAuth for external route access
4. RBAC: Kubernetes RBAC for API access control
5. Network Policies: Pod label-based network segmentation
6. Secret Management: Auto-rotating TLS certs via service-cert controller
7. Artifact URL Signing: Temporary signed URLs for artifact download (60s expiry)
8. FIPS Compliance: FIPS-enabled builds available
