═══════════════════════════════════════════════════════════════════════════
AUTHENTICATION FLOWS
═══════════════════════════════════════════════════════════════════════════

1. User Authentication (OAuth):
   Browser → Router (HTTPS/443) → kube-rbac-proxy (HTTPS/8443)
   → OAuth Server (redirect) → User login → Token issuance
   → kube-rbac-proxy validates token via TokenReview

2. API Authorization:
   kube-rbac-proxy → Kubernetes API (SubjectAccessReview)
   → Backend receives impersonation headers (X-Auth-Request-User)
   → Backend uses ServiceAccount + impersonation for K8s API calls

3. WebSocket Authentication:
   Same as API, but with WebSocket Upgrade protocol
   wss:// at edge → ws:// internally → wss:// to K8s API

4. Micro-frontend Authentication (Modular Architecture):
   User token forwarded via x-forwarded-access-token header
   from main dashboard to micro-frontend services

COMPLIANCE
═══════════════════════════════════════════════════════════════════════════

- Privilege Escalation: Disabled (allowPrivilegeEscalation: false)

- CSRF: Mitigated by OAuth token requirements
