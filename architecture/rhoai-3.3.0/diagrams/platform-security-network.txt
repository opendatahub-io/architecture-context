2. InferenceService Isolation:
   - Allow Ingress: From Istio Gateway, TrustyAI Service (for payload logging)
   - Allow Egress: To S3 (443/TCP), Model Registry (8443/TCP), Feast (6566/TCP),
                   Kubernetes API (443/TCP)
   - Deny: Cross-namespace InferenceService communication (unless explicit)

3. Training Job Isolation (Trainer):
   - Allow Ingress: From trainer controller (for metrics polling on 28080/TCP)
   - Allow Egress: To S3 (443/TCP), MLflow (8443/TCP), HuggingFace (443/TCP),
                   Kubernetes API (443/TCP)
   - Allow Pod-to-Pod: Within same TrainJob (for distributed training)
   - Deny: Access to other training jobs

4. Model Registry Isolation:
   - Allow Ingress: From kube-rbac-proxy (authenticated users), InferenceServices,
                   Data Science Pipelines, Dashboard
   - Allow Egress: To PostgreSQL (5432/TCP), Kubernetes API (443/TCP)
   - Deny: Direct access bypassing kube-rbac-proxy

5. MLflow Isolation:
   - Allow Ingress: From data-science-gateway (via HTTPRoute), Notebooks, Training Jobs
   - Allow Egress: To PostgreSQL (5432/TCP), S3 (443/TCP), Kubernetes API (443/TCP)
   - Deny: Cross-namespace access without authorization

6. Llama Stack Isolation (Default in RHOAI 3.3):
   - Allow Ingress: From specific namespaces/pods (user-defined)
   - Allow Egress: To Ollama/vLLM providers (custom ports), Kubernetes API (443/TCP)
   - Deny: All other traffic (default deny)

Default Namespace Policy (Recommended):
- Default Deny All Ingress (unless explicitly allowed)
- Default Deny All Egress except Kubernetes API, DNS (unless explicitly allowed)
- Explicit allow rules per component

═════════════════════════════════════════════════════════════════════════════════════════════
AUTHENTICATION & AUTHORIZATION MECHANISMS
═════════════════════════════════════════════════════════════════════════════════════════════

1. OpenShift OAuth (User Authentication):
   - Flow: User → OpenShift OAuth Server → Dashboard/Notebooks (Bearer Token)
   - Token: JWT with user identity, groups, expiry
   - Enforcement: kube-rbac-proxy (validates token via SubjectAccessReview)
   - Components: Dashboard, Notebooks (OAuth proxy sidecar), MLflow (via HTTPRoute)

2. Kubernetes RBAC (ServiceAccount Authentication):
   - Flow: Operator → Kubernetes API Server (ServiceAccount JWT)
   - Token: Mounted at /var/run/secrets/kubernetes.io/serviceaccount/token
   - Enforcement: Kubernetes API Server (validates token signature, RBAC rules)
   - Components: All operators, user workload pods

3. mTLS (Service-to-Service Authentication - Optional):
   - Flow: Service A → Istio Sidecar (mTLS) → Service B Sidecar
   - Certificate: Istio-issued workload certificates (short-lived, auto-rotated)
   - Enforcement: Istio sidecars (PERMISSIVE or STRICT mode)
   - Components: KServe InferenceServices, Data Science Pipelines, TrustyAI

4. Database Username/Password:
   - Flow: Application → Database (username/password from Secret)
   - Credentials: Stored in Opaque Secrets (user-provisioned)
   - Enforcement: Database server (PostgreSQL, MySQL, Redis)
   - Optional TLS: Recommended but not enforced by default
   - Components: Model Registry, MLflow, Feast, Data Science Pipelines

5. S3 Access Keys / AWS IAM:
   - Flow: Pod → S3 (AWS Access Key/Secret Key or IRSA token)
   - Credentials: Stored in Opaque Secrets or IRSA (ServiceAccount annotation)
   - Enforcement: S3 API (HMAC signature validation, IAM policy evaluation)
   - Components: KServe, Training Operator, Notebooks, Data Science Pipelines, MLflow

6. API Tokens (External Services):
   - HuggingFace: Bearer token (from hf-token Secret)
   - NVIDIA NGC: API key (from {nim}-ngc-api-key Secret)
   - Git: SSH keys or Personal Access Tokens
   - Enforcement: External API servers
   - Components: Notebooks, Training Jobs, Llama Stack, odh-model-controller

7. Kuadrant/Authorino (Optional Advanced Auth):
   - Flow: Client → Istio Gateway → Authorino (validates JWT/API key) → InferenceService
   - Policies: AuthPolicy CRs (declarative authorization rules)
   - Enforcement: Authorino service (external authorization server)
   - Components: odh-model-controller (KServe integration)

TRUST BOUNDARIES
═════════════════════════════════════════════════════════════════════════════════════════════

Boundary 1: External → Ingress
- Crossing: TLS 1.2+ (HTTPS)
- Authentication: OAuth (Dashboard, Notebooks), Bearer Token (InferenceServices - optional)
- Validation: OpenShift Router/Gateway (TLS termination/reencrypt)

Boundary 2: Ingress → Application Layer
- Crossing: Internal HTTPS (kube-rbac-proxy), HTTP (OAuth proxy)
- Authentication: OAuth token validation (SubjectAccessReview)
- Validation: kube-rbac-proxy, OAuth proxy sidecar

Boundary 3: Application Layer → User Workloads
- Crossing: Kubernetes API (HTTPS/443)
- Authentication: ServiceAccount JWT tokens
- Validation: Kubernetes API Server (RBAC evaluation)

Boundary 4: User Workloads → External Services
- Crossing: HTTPS/443 (S3, HuggingFace), PostgreSQL/5432, Redis/6379
- Authentication: S3 keys/IAM, DB credentials, API tokens
- Validation: External service authentication systems

Boundary 5: Service Mesh Internal (Optional)
- Crossing: mTLS (Istio sidecars)
- Authentication: Workload certificates (Istio-issued)
- Validation: Istio sidecars (PeerAuthentication policies)

✅ TLS Encryption: All external traffic (HTTPS), optional internal (mTLS)
✅ Authentication: OAuth (users), ServiceAccount JWT (operators), mTLS (service mesh)
✅ Authorization: Kubernetes RBAC, SubjectAccessReview, NetworkPolicies, AuthorizationPolicies
✅ Secrets Management: Auto-rotation (cert-manager, Service CA), encrypted at rest (etcd)
✅ Network Isolation: NetworkPolicies per component, namespace isolation, Istio policies
✅ Multi-Tenancy: Namespace-scoped resources, RBAC enforcement, separate workload namespaces
✅ Audit Logging: Kubernetes audit logs, OpenShift OAuth logs, operator logs
✅ Image Security: Signed images (Konflux), vulnerability scanning, pull secrets
✅ FIPS Compliance: FIPS-enabled builds for applicable components
✅ Air-Gap Support: Prefetched manifests, private registry configuration

⚠️  Manual Secret Rotation: S3 keys, DB passwords, API tokens (user responsibility)
⚠️  Optional TLS for Databases: Not enforced by default (recommended to enable)
⚠️  Service Mesh Optional: mTLS not required (PERMISSIVE mode), can enable STRICT
⚠️  NetworkPolicies Optional: Not enforced by default (recommended for production)
