NETWORK SERVICES:

ClusterIP Service (per workbench):
- Name: {workbench-name}-notebook
- Port: 8888/TCP → 8888/TCP
- Selector: notebook-name={workbench-name}
- Exposure: Internal only

OpenShift Route (per workbench):
- Name: {workbench-name}
- Host: {workbench-name}.apps.{cluster-domain}
- Port: 443/TCP (external) → 8443/TCP (to oauth-proxy)
- TLS: edge termination (TLS 1.2+)
- Backend: Service {workbench-name}-notebook:8888

RBAC SUMMARY:

Workbench Pod ServiceAccount:
- Name: {workbench-name}-sa (created by odh-notebook-controller)
- Default Permissions (namespace-scoped):
  - Resources: ConfigMaps, Secrets, Pods, Services
  - Verbs: get, list (read-only by default)
  - Can be extended by user with additional RoleBindings

Runtime Pod ServiceAccount:
- Name: pipeline-runner-sa (created by Kubeflow Pipelines)
- Default Permissions (namespace-scoped):
  - Resources: Pods, ConfigMaps, Secrets
  - Verbs: get, list, create (for pipeline execution)

No cluster-level RBAC defined by this component.

SECURITY CONTROLS:

Authentication:
- External access: OpenShift OAuth (enforced by oauth-proxy sidecar)
- Jupyter Server: Authentication DISABLED (token='', password='')
  - Security relies entirely on oauth-proxy
  - Users cannot bypass OAuth to access Jupyter directly
- Internal pod-to-pod: No mTLS (plain HTTP within pod network)

Authorization:
- Namespace-scoped: Users can only launch workbenches in namespaces they have access to
- RBAC enforcement: Kubernetes API Server enforces service account permissions
- No cross-namespace access by default

Encryption:
- External (browser to router): TLS 1.2+ (HTTPS)
- Router to oauth-proxy: TLS 1.2+ (HTTPS, 8443/TCP)
- oauth-proxy to Jupyter: None (HTTP, internal pod network)
- Egress to pypi.org: TLS 1.2+ (HTTPS)
- Egress to S3: TLS 1.2+ (HTTPS, user-configured)
- Egress to K8s API: TLS 1.2+ (HTTPS)

Container Security:
- Non-root user: UID 1001 (all workbench and runtime images)
- Read-only root filesystem: No (Jupyter needs writable /home for notebooks)
- Security Context Constraints (SCC): restricted (OpenShift default)
- Capabilities: Limited by SCC (no privileged, no host network)
- Image scanning: Konflux pipelines scan for CVEs before publishing

Secrets:
- {workbench}-oauth-config (Opaque):
  - Purpose: OAuth proxy configuration
  - Provisioned by: odh-notebook-controller
  - Auto-rotate: No
  - Mounted to: oauth-proxy container

- {workbench}-tls (kubernetes.io/tls):
  - Purpose: TLS certificate for HTTPS route (if reencrypt mode)
  - Provisioned by: OpenShift cert manager
  - Auto-rotate: Yes (90 days)
  - Used by: OpenShift Router

- User-created secrets (optional):
  - Examples: Git credentials, S3 access keys, API tokens
  - Mounted to: Jupyter container at user-specified paths
  - Management: User responsibility

EGRESS TRAFFIC:

Allowed egress (default):
- pypi.org:443 (HTTPS) - Python package installation
- quay.io:443 (HTTPS) - Container image pulls (build-time)
- registry.redhat.io:443 (HTTPS) - RHEL base images (build-time)
- mirror.openshift.com:443 (HTTPS) - OpenShift CLI downloads
- Kubernetes API:6443 (HTTPS) - kubectl/oc commands
- S3-compatible storage:443 (HTTPS) - User-configured data access
- Git repositories:443 (HTTPS/SSH) - User-configured source control

Network policies (if enforced):
- May restrict egress to specific destinations
- Cluster-specific configuration (not defined by this component)

BUILD INFRASTRUCTURE (Konflux CI/CD):

Build process (not part of runtime):
- Trigger: Git commit to notebooks repository
- Pipeline: Konflux Tekton PipelineRun
- Steps:
  1. Clone source repository (HTTPS/443)
  2. Pull base images from registry.redhat.io (HTTPS/443)
  3. Build multi-arch images (x86_64, aarch64, ppc64le, s390x)
  4. Scan for CVEs and security issues
  5. Sign images with Sigstore
  6. Push to quay.io/opendatahub (HTTPS/443)
  7. Push to registry.redhat.io (HTTPS/443)
  8. Update ImageStream manifests via K8s API (HTTPS/6443)

Build security:
- Buildah rootless builds (no privileged containers)
- Isolated build namespaces
- Image signing for supply chain security
- SBOM generation for vulnerability tracking
