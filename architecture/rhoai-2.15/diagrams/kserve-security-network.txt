TRUST BOUNDARIES
═══════════════════════════════════════════════════════════════════════════════════════════════

1. EXTERNAL → INGRESS
   - TLS 1.2+ termination at Istio Gateway
   - Optional Bearer Token authentication (OpenShift OAuth)
   - No client certificate required for external users

2. INGRESS → SERVICE MESH
   - Automatic mTLS upgrade via Istio
   - Service identity verification
   - STRICT mode enforcement

3. SERVICE MESH (Internal)
   - All pod-to-pod communication uses mTLS
   - Automatic certificate management via Istio
   - Service identity-based authorization

4. SERVICE MESH → EXTERNAL SERVICES (S3/GCS/Azure)
   - TLS 1.2+ required
   - Cloud provider authentication (AWS IAM, GCP SA, Azure creds)
   - Egress traffic allowed via Istio egress gateway (configurable)

5. CONTROL PLANE → KUBERNETES API
   - TLS 1.2+ with Service Account token
   - RBAC enforcement via ClusterRole/ClusterRoleBinding
   - Limited to necessary permissions (least privilege)
