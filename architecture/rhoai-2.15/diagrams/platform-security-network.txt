  - Contents: AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_S3_ENDPOINT (optional)

  - Components: Data Science Pipelines, Model Registry, TrustyAI
  - Contents: username, password, host, port, database
  - Rotation: Manual
  - Purpose: Metadata persistence (pipeline runs, model versions, inference logs)

Type: Opaque (OAuth Configuration)
  - Components: Dashboard, Notebooks, Ray Dashboard, TrustyAI
  - Contents: cookie_secret, client_id, client_secret
  - Rotation: Manual (regenerate cookie_secret periodically)
  - Purpose: OAuth proxy sidecar configuration for user authentication

Type: kubernetes.io/dockerconfigjson (Image Pull Secrets)
  - Components: All components
  - Contents: .dockerconfigjson (registry credentials)
  - Rotation: Per registry policy
  - Purpose: Pull container images from private registries (quay.io, registry.redhat.io)

Type: kubernetes.io/service-account-token (SA Tokens)
  - Components: All components (auto-generated per ServiceAccount)
  - Contents: token, ca.crt, namespace
  - Rotation: Automatic (Kubernetes TokenRequest API)
  - Purpose: Kubernetes API authentication for controllers and operators

Security-Critical Secrets (requires audit logging):
  - Webhook server certificates: Invalid certs prevent admission control (cluster impact)
  - Storage credentials: Compromise allows model theft or data exfiltration
  - OAuth cookie secrets: Session hijacking if leaked
  - Database passwords: Access to sensitive ML metadata
  - CA certificates: Trust chain compromise if custom CA leaked

CERTIFICATE MANAGEMENT
════════════════════════════════════════════════════════════════════════════════════════════════════

OpenShift Service CA Operator:
  - Automatic TLS certificate provisioning for services
  - Annotation: service.beta.openshift.io/serving-cert-secret-name=<secret-name>
  - Certificate Duration: 90 days
  - Automatic Rotation: Yes (triggers pod restart on renewal)
  - CA Bundle Injection: ConfigMap annotation (service.beta.openshift.io/inject-cabundle=true)

Istio Certificate Management:
  - Citadel CA (Istio's built-in CA) issues workload certificates
  - Certificate Duration: 24 hours (default)
  - Automatic Rotation: Yes (transparent to workloads)
  - mTLS Certificate: Includes SPIFFE ID (spiffe://cluster.local/ns/{ns}/sa/{sa})

Webhook Certificates:
  - All operators require TLS certs for admission webhooks
  - Provisioned by OpenShift service-ca-operator
  - ValidatingWebhookConfiguration / MutatingWebhookConfiguration includes CABundle
  - Failure Mode: Webhook unreachable if cert expired (fail-open or fail-closed per config)

Custom CA Bundles (User-Provided):
  - Purpose: Trust enterprise/self-signed certificates for S3, databases, registries
  - Mechanism: ConfigMap with ca-bundle.crt, mounted in notebooks and pipelines
  - Example: Model Registry RHOAIENG-14601 (ConfigMap CA support)
