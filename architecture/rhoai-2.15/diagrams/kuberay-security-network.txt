Network Security Considerations
═════════════════════════════════════════════════════════════════════════

⚠️  WARNING: Ray uses plaintext protocols internally:
   - Ray GCS: 6379/TCP (Redis Protocol) - NO encryption
   - Ray Dashboard: 8265/TCP (HTTP) - NO encryption
   - Ray Client: 10001/TCP (gRPC) - NO encryption
   - Ray Serve: 8000/TCP (HTTP) - NO encryption

   Mitigation:
   - Deploy NetworkPolicies to isolate Ray pods
   - Use namespace-scoped isolation
   - Consider external TLS termination at Ingress/Route
   - User-configured authentication for Dashboard and Serve

⚠️  Authentication is OPTIONAL and user-configured:
   - Ray Dashboard has no built-in auth by default
   - Ray Serve has no built-in auth by default
   - Users must configure authentication separately

✅  Kubernetes-level security:
   - RBAC enforced at K8s API level
   - Pod security via SCC (OpenShift) or PSP/PSA (K8s)
   - ServiceAccount token authentication
   - Network isolation via NetworkPolicies (recommended)
