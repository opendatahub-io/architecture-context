## Purpose

**Short**: Kubernetes-native operator for managing distributed AI/ML training jobs across multiple frameworks (PyTorch, TensorFlow, XGBoost, JAX, MPI, PaddlePaddle).

**Detailed**: The Kubeflow Training Operator (KFTO) is a controller-runtime-based Kubernetes operator that provides custom resource definitions and controllers for orchestrating distributed training workloads. It manages the full lifecycle of training jobs — from pod creation and headless service setup for inter-node communication, through status tracking and cleanup, to gang-scheduling integration with Volcano or scheduler-plugins for coordinated resource allocation.

Each supported ML framework (PyTorch, TensorFlow, XGBoost, JAX, MPI, PaddlePaddle) gets a dedicated CRD (e.g., `PyTorchJob`, `TFJob`) and controller that understands framework-specific topology requirements such as master/worker roles, parameter servers, rendezvous backends, and elastic scaling. The operator creates per-replica pods with framework-specific environment variables (e.g., `TF_CONFIG`, `WORLD_SIZE`, `MASTER_ADDR`) and headless services for peer discovery, enabling distributed training without manual infrastructure setup.

In the RHOAI distribution, the operator is deployed into the `opendatahub` namespace with the `kubeflow-` name prefix. It uses FIPS-compliant builds via `strictfipsruntime`, integrates with Prometheus via PodMonitor for metrics collection, and provides validating admission webhooks with auto-rotating TLS certificates managed by the OPA cert-controller.

## Architectural Analysis

The Training Operator follows a classic Kubeflow controller pattern with a shared reconciliation framework and framework-specific pluggable logic. The `pkg/controller.v1/common/` package provides the core `JobController` struct implementing pod lifecycle management, service creation, status tracking, and gang scheduling integration. Each ML framework (PyTorch, TensorFlow, XGBoost, JAX, MPI, PaddlePaddle) implements the `ControllerInterface` to provide framework-specific behavior — particularly the `SetClusterSpec()` method that injects environment variables like `TF_CONFIG` for TensorFlow or `WORLD_SIZE`/`RANK`/`MASTER_ADDR` for PyTorch. This separation allows adding new frameworks without modifying core reconciliation logic.

The operator's memory optimization strategy is notable. The `newCacheOptions()` function in `main.go` (lines 81-115) configures label-filtered informer caches that only watch resources with the `training.kubeflow.org/operator-name` label. Without this, the operator would cache all cluster-wide Pods, Services, and ConfigMaps, causing unbounded memory growth. Additionally, ConfigMaps and Secrets are excluded from cache reads entirely (`DisableFor`) to avoid stale configuration data. The `GOMEMLIMIT=460MiB` environment variable further constrains Go's GC behavior within the 512Mi memory limit.

The gang scheduling integration demonstrates a clean abstraction pattern. The `GangSchedulingSetupFunc` type allows swapping between Volcano (`scheduling.volcano.sh/v1beta1`) and scheduler-plugins (`scheduling.x-k8s.io/v1alpha1`) at startup time, with each implementation providing PodGroup creation, pod template decoration, and scheduling readiness checks. The `DelayPodCreationDueToPodGroup` mechanism prevents pod creation until the PodGroup is in `Inqueue` status, ensuring coordinated scheduling of all replicas.

PyTorch's elastic training support is the most sophisticated feature. The `ElasticPolicy` spec enables min/max replica bounds, rendezvous backend configuration (c10d, etcd, etcd-v2), and HPA integration via `autoscalingv2.MetricSpec`. The controller creates an init container (configurable via `--pytorch-init-container-image`) for distributed setup, and injects environment variables like `PET_RDZV_BACKEND`, `PET_RDZV_ENDPOINT`, `PET_NNODES`, and `PET_NPROC_PER_NODE` for PyTorch Elastic (torchrun) coordination. This allows dynamic scaling of training workers during a job run.

The webhook architecture uses the OPA cert-controller for fully automated certificate management. On startup, the operator generates a CA and server certificate, stores them in a Secret, and patches the `ValidatingWebhookConfiguration` with the CA bundle. The readiness probe blocks (`/readyz` returns "certificates are not ready") until certs are generated, preventing the API server from routing webhook requests before TLS is ready. Controllers are also started in a goroutine that blocks on the `certsReady` channel, ensuring correct startup ordering. The `RequireLeaderElection: false` setting means cert rotation runs on all instances, not just the leader, which is important for webhook availability during leader transitions.
