## Purpose
**Short**: A Kubernetes-native operator for orchestrating distributed training jobs across multiple ML/AI frameworks including PyTorch, TensorFlow, XGBoost, MPI, PaddlePaddle, and JAX.

**Detailed**: The Kubeflow Training Operator (KFTO) manages the full lifecycle of distributed machine learning training jobs on Kubernetes. It provides Custom Resource Definitions (CRDs) for six different training frameworks: PyTorchJob, TFJob, XGBoostJob, MPIJob, PaddleJob, and JAXJob. Each CRD allows users to define multi-replica training topologies with master and worker roles, and the operator handles pod creation, headless service creation for inter-pod communication, status tracking, and cleanup.

The operator includes a validating webhook server that validates job specifications on CREATE and UPDATE operations, ensuring correct configuration before jobs are submitted. It supports optional integration with gang scheduling systems (Volcano and scheduler-plugins) for coordinated pod scheduling, elastic training with HorizontalPodAutoscaler support for PyTorch jobs, and automatic NetworkPolicy creation for PyTorch jobs to restrict inter-pod communication. Certificate management is handled internally using the OPA cert-controller library to rotate webhook TLS certificates.

In the RHOAI distribution, the operator is deployed into the `opendatahub` namespace with the `kubeflow-` name prefix. The operator image is injected at deploy time by the rhods-operator via kustomize parameter substitution, and the same image is reused as the PyTorch init container for DNS readiness checks.

### Dynamically Created Resources (per training job)

The controller creates the following resources dynamically for each training job:

| Resource Type | Created By | Purpose |
|---------------|------------|---------|
| Pods | All job controllers | Worker and master pods for distributed training |
| Headless Services | All job controllers | DNS-based pod discovery for distributed communication |
| NetworkPolicy | PyTorchJob controller only | Restricts ingress to same-job pods + openshift-monitoring (port 8080/TCP) |
| HorizontalPodAutoscaler | PyTorchJob controller (elastic) | Auto-scales workers based on metrics for elastic PyTorch training |
| PodGroup | Gang scheduling (optional) | Coordinates scheduling of all job pods together (Volcano or scheduler-plugins) |

### Network Policies (Dynamically Created)

For each PyTorchJob, the controller creates a NetworkPolicy with:

| Rule | Source | Destination | Port | Protocol | Purpose |
|------|--------|-------------|------|----------|---------|

| Ingress Allow | Pods from `openshift-monitoring` namespace | Job pods | 8080/TCP | TCP | Allow Prometheus metrics scraping |
