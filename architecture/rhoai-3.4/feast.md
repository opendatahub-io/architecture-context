## Purpose

**Short**: Feast is a feature store for machine learning that manages, stores, and serves ML features for both training and online inference.

**Detailed**: Feast (Feature Store) provides a complete ML feature management platform. The component consists of a Kubernetes operator (`feast-operator`) that manages `FeatureStore` custom resources, deploying and configuring feature serving infrastructure including online stores, offline stores, a feature registry, and a web UI. The operator handles the full lifecycle of feature store deployments including TLS, OIDC authentication, RBAC authorization, autoscaling, and scheduled materialization jobs.

The Python SDK provides the primary feature server (FastAPI-based REST API), CLI tooling, and client libraries for interacting with the feature store. It supports 18+ online store backends (Redis, DynamoDB, PostgreSQL, etc.) and 15+ offline store backends (BigQuery, Snowflake, Spark, etc.). The Go feature server provides a high-performance alternative for online feature retrieval with gRPC and HTTP interfaces.

In the RHOAI platform, the feast-operator is deployed by the platform operator (`rhods-operator`) into the `redhat-ods-applications` namespace. It integrates with OpenDataHub by injecting feature store configuration into Kubeflow Notebooks and consuming OIDC issuer configuration from the platform's GatewayConfig.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| feast-operator | Go Operator (controller-runtime) | Manages FeatureStore CRs, deploys/configures all feast services (online, offline, registry, UI), handles RBAC, TLS, OIDC, autoscaling, CronJobs, and ServiceMonitors |
| feature-server (Python) | Python Service (FastAPI/Gunicorn) | REST API for online feature serving, push ingestion, materialization, and vector search; supports OIDC and Kubernetes auth |
| feature-server (Go) | Go Service (net/http + gRPC) | High-performance online feature retrieval via HTTP and gRPC with Prometheus metrics and OpenTelemetry tracing |
| feast-ui | React/TypeScript Frontend | Web dashboard for exploring feature definitions, data sources, entities, and feature services |
| feast Python SDK | Python Library | Client library and CLI for defining, applying, and querying features; includes registry, offline/online store abstractions |

## Architectural Analysis

Feast is a multi-language, multi-component feature store that follows the operator pattern for Kubernetes-native deployment. The feast-operator is the central orchestrator: it watches a single CRD (`FeatureStore`) and reconciles a complete stack of Kubernetes resources including Deployments, Services, ConfigMaps, Routes, CronJobs, HPA, PDB, RBAC, and ServiceMonitors. This "one CRD to rule them all" design simplifies the user experience but creates a large blast radius — a misconfigured FeatureStore CR can affect 10+ child resources simultaneously. The operator requeues every 30 seconds for periodic reconciliation and 5 seconds on errors, providing reasonable convergence times.

The authentication architecture is notably flexible, supporting three modes: no auth, OIDC, and Kubernetes RBAC. In OIDC mode, the Python feature server validates JWTs using JWKS keys fetched from the OIDC discovery endpoint, with special handling for Kubernetes ServiceAccount tokens (detected via the `kubernetes.io` claim). The RBAC framework is implemented at the application level within the Python SDK — it uses a `SecurityManager` with `ContextVar`-based user isolation for concurrent request handling. Permissions are defined as Feast objects in the registry and support resource-type, name-pattern (regex), tag, and action-level matching. The operator's "auto-access RBAC" feature bridges Feast-level permissions to Kubernetes RBAC by polling the registry REST API and creating corresponding ClusterRoles/RoleBindings.

The Go feature server and Python feature server serve different deployment scenarios. The Go server provides a lean, high-performance serving path for online feature retrieval only (HTTP + gRPC), while the Python server offers a full-featured REST API including materialization, push ingestion, vector search, and the complete OIDC/RBAC security stack. The operator deploys the feature server image specified by `RELATED_IMAGE_FEATURE_SERVER`, which defaults to the Python-based image. The Go embedded library is also available for Python integration via CGO/cffi, allowing the Go feature retrieval engine to be called from within the Python server process.

The TLS story is well-designed for OpenShift environments: the operator detects OpenShift at startup and automatically leverages service serving certificates for inter-service TLS. For non-OpenShift deployments, users can provide their own TLS secrets. The intra-service authentication uses a simple HMAC-SHA256 JWT token stored in a ConfigMap — this is adequate for namespace-level isolation but the token is not encrypted at rest. The `InsecureSkipVerify: true` used for the registry HTTP client is intentional for cluster-internal service certificates and documented in the code, but it does weaken the TLS security posture for that specific connection.

The integration with OpenDataHub is implemented via the `NotebookConfigMapReconciler`, which watches Kubeflow Notebook CRs and injects feast client configuration when the `opendatahub.io/feast-integration=true` label is present. This controller polls for Notebook CRD availability at startup (5-second intervals), gracefully handling the case where the Kubeflow operator is not installed. The namespace registry ConfigMap (`feast-configs-registry`) enables discovery of all FeatureStores within a namespace, supporting multi-tenant scenarios.
