## Purpose

**Short**: Feast is a feature store for machine learning that manages the storage, retrieval, and serving of ML features, deployed via a Kubernetes operator.

**Detailed**: Feast (Feature Store) is an open-source platform for managing and serving machine learning features. It provides a unified interface for storing feature definitions in a registry, persisting feature data in online and offline stores, and serving features at low latency for model inference. The Feast operator (`feast-operator`) automates the deployment and lifecycle management of Feast services on Kubernetes/OpenShift clusters.

In the RHOAI distribution, Feast is deployed as two primary container images: the Feast operator (a Go-based controller-runtime operator that manages `FeatureStore` custom resources) and the feature server (a Python-based HTTP/gRPC server built on FastAPI/Gunicorn that serves online features). The operator watches `FeatureStore` CRs and reconciles Deployments, Services, ConfigMaps, PVCs, CronJobs, RBAC resources, and OpenShift Routes. It supports multiple persistence backends (SQLite, Redis, PostgreSQL, DuckDB, Snowflake, and many others), Kubernetes and OIDC authorization, TLS with auto-provisioned OpenShift service serving certificates, horizontal pod autoscaling, and cross-namespace feature store discovery.

The Python SDK provides the `feast` CLI and client library for defining feature views, entities, and data sources, materializing features from offline to online stores, and querying features at inference time. The feature server image packages this SDK with production-grade Gunicorn worker configuration and Prometheus metrics instrumentation.

## Architectural Analysis

The Feast component follows a split-plane architecture: a Go-based operator (control plane) manages the lifecycle of Python-based feature servers (data plane). This is a deliberate design choice that leverages Go's strengths in Kubernetes operator development (type safety, controller-runtime framework, efficient resource watching) while using Python for ML-adjacent workloads where the rich ecosystem of data science libraries (PyArrow, Pandas, NumPy) is essential.

The operator deploys all Feast services into a single Kubernetes Deployment with multiple containers rather than separate Deployments per service. The online store, offline store, registry (gRPC and REST), and UI all run as containers in the same pod, exposed via separate ClusterIP Services. This co-location simplifies configuration sharing (all containers mount the same `feature_store.yaml` ConfigMap) and reduces inter-service latency, but it means all services scale together. The scaling subsystem (HPA + PDB) introduced in the `ScalingConfig` API addresses this by allowing horizontal pod autoscaling, though it requires DB-backed persistence (not file-based SQLite) for all stores to avoid data inconsistency across replicas.

Authorization is a sophisticated two-layer system. At the Feast application level, permissions are defined as Feast entities in the registry (not Kubernetes RBAC resources). The operator periodically fetches these permissions from the registry's REST API using a dedicated intra-communication JWT token, then translates them into Kubernetes ClusterRoleBindings and RoleBindings for namespace discovery and ConfigMap access. This "auto-access RBAC" pattern allows Feast administrators to define access policies within Feast's permission model while the operator automatically ensures that the corresponding Kubernetes-level access is provisioned. The OIDC authorization path supports external identity providers with automatic CA certificate detection for RHOAI/ODH environments.

The notebook integration controller (`notebook_configmap_controller.go`) is a secondary controller in the same operator binary that watches Kubeflow Notebook CRs. When a notebook is annotated with `opendatahub.io/feast-integration=true` and `opendatahub.io/feast-config={project-names}`, the controller creates a ConfigMap containing the Feast client configuration for the specified projects. This enables data scientists to use Feast from notebooks without manual configuration. The controller uses a three-way watch (Notebooks, ConfigMaps, FeatureStores) to keep configurations synchronized.

The cross-namespace registry feature (`namespace_registry.go`) allows Feast installations in different namespaces to discover each other. A central ConfigMap (`feast-configs-registry`) in the operator namespace maps namespaces to their available Feast client configurations. This supports multi-tenant scenarios where different teams deploy separate FeatureStore instances but need to discover and consume features from other teams' stores. The `feast-discover-namespaces` ClusterRole enables authenticated users to list namespaces with Feast installations.
