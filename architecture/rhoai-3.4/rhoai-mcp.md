## Purpose

**Short**: An MCP (Model Context Protocol) server that enables AI agents to interact with Red Hat OpenShift AI environments for managing workbenches, model serving, pipelines, training jobs, and data connections.

**Detailed**: The RHOAI MCP Server implements the Model Context Protocol specification to expose Red Hat OpenShift AI platform capabilities as structured tools, resources, and prompts that AI agents (such as Claude, ChatGPT, or other LLM-based assistants) can invoke. It acts as a bridge between AI agents and the Kubernetes/OpenShift API, translating high-level AI agent requests into Kubernetes API operations against RHOAI custom resources.

The server is built on the FastMCP framework with a pluggy-based plugin architecture that organizes functionality into domain plugins (notebooks, inference, pipelines, connections, storage, training, model registry, quickstarts, prompts) and composite plugins (cluster exploration, training workflows, neural navigator recommendations, tool discovery). Each plugin registers MCP tools, resources, and/or prompts and includes health checks against the cluster to verify CRD availability. The server supports three transport modes (stdio for Claude Desktop, SSE, and streamable-http for networked deployments), three Kubernetes authentication modes (in-cluster service account, kubeconfig, and explicit token), and configurable safety controls (read-only mode, dangerous operations gating).

The Neural Navigator integration provides AI-guided model selection and deployment configuration generation by calling an external recommendation service. The Model Registry integration supports both Kubeflow Model Registry and Red Hat AI Model Catalog APIs with automatic API type detection, port forwarding for out-of-cluster access, and benchmark extraction for performance analysis.

_The RHOAI MCP server is a Python application running on UBI9 which provides FIPS-validated OpenSSL. Python's ssl module uses the system OpenSSL, so TLS connections inherit FIPS compliance from the base image. The TLS skip for in-cluster .svc URLs is intentional — Kubernetes service serving CA is not in the service account CA bundle, so verification is skipped for cluster-internal connections where network isolation provides transport security._

## Architectural Analysis

The RHOAI MCP Server represents an innovative approach to AI platform management, using the Model Context Protocol to make Red Hat OpenShift AI accessible to AI agents. The architecture is well-structured around a pluggy-based plugin system that cleanly separates concerns into domain plugins (each managing a specific RHOAI resource type) and composite plugins (orchestrating cross-domain workflows). This separation enables independent development and testing of each domain while composite plugins provide the higher-level abstractions that AI agents need. The plugin health check system is particularly well-designed: each plugin declares its required CRDs, and the PluginManager automatically disables plugins whose CRDs are not available on the target cluster, allowing the server to gracefully degrade on clusters with partial RHOAI installations.

The composite layer is the most architecturally significant innovation. The `cluster_summary` and `explore_cluster` tools achieve 70-90% token savings compared to calling individual domain tools, which is critical for AI agent context window management. The unified `training()` tool collapses 14 training operations into a single action-based interface, reducing the cognitive load on AI agents. The Meta composite's `suggest_tools` implements intent-based tool discovery using pattern matching against natural language descriptions, effectively creating a tool search engine within the MCP server itself. These patterns demonstrate a deep understanding of AI agent interaction patterns and context window constraints.

The Neural Navigator integration follows a multi-step recommendation pipeline: intent extraction from natural language, SLO default retrieval, workload profiling, RPS estimation, ranked recommendation generation, and finally Kubernetes deployment YAML generation. This pipeline is designed to guide non-expert users from a vague request ("I need a chatbot") to a production-ready deployment configuration. The Model Registry integration is equally sophisticated, with automatic API type detection (probing both Kubeflow Model Registry and Red Hat AI Model Catalog endpoints), port-forward management for out-of-cluster access, and benchmark extraction from both structured custom properties and unstructured README markdown.

From a security perspective, the server implements a layered safety model: a global read-only mode disables all write operations, a dangerous operations flag specifically gates delete operations, and individual tools check operation permissions before executing. The server runs as non-root (UID 1001 in container, UID 1000 in Kubernetes), uses a read-only root filesystem with an emptyDir for `/tmp`, and drops all Linux capabilities. However, the MCP protocol endpoints themselves have no authentication, relying on network-level controls (ClusterIP service, Route with edge TLS) for access control. This is a notable gap for production deployments where the server is exposed outside the cluster. The Model Registry authentication logic handles the complexity of RHOAI's dual-port pattern (8080 for unauthenticated in-cluster access, 8443 for kube-rbac-proxy-protected access) with automatic detection and OAuth token sourcing via `oc whoami -t`.

The quickstarts domain introduces filesystem and subprocess operations (git clone, helm/oc commands) that differ from the pure-API pattern of other domains. The 60-second git clone timeout and 300-second deployment timeout provide safety bounds, and the `GIT_TERMINAL_PROMPT=0` environment variable prevents credential prompt hangs. However, executing arbitrary Helm charts and Kustomize manifests from external repositories carries supply chain risk that should be carefully considered for production use.
