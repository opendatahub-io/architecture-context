## Purpose

**Short**: Kubernetes operator for managing Apache Spark applications, scheduled Spark jobs, and Spark Connect sessions on OpenShift clusters.

**Detailed**: The Spark Operator extends Kubernetes with three Custom Resource Definitions (SparkApplication, ScheduledSparkApplication, and SparkConnect) that enable declarative management of Apache Spark workloads. It automates the full lifecycle of Spark applications including submission via `spark-submit`, driver and executor pod management, monitoring, UI exposure, and cleanup.

The operator is deployed as two separate processes: a controller that reconciles Spark CRDs and manages the Spark application lifecycle (creating pods, services, ingresses, and ConfigMaps), and a webhook server that mutates and validates Spark resources at admission time. The webhook injects pod configurations such as environment variables, volumes, sidecar containers, scheduler settings, and Prometheus monitoring into Spark driver and executor pods.

In the RHOAI distribution, the operator is deployed into the `redhat-ods-applications` namespace via Kustomize overlays managed by the platform operator (rhods-operator). It supports batch scheduling integration with Volcano, Apache YuniKorn, and Kubernetes scheduler plugins for gang scheduling. The operator produces Prometheus metrics for application and executor lifecycle tracking and exposes Spark Web UI via Kubernetes Services and optional Ingress resources.

## Architectural Analysis

The Spark Operator follows a clean separation between its controller and webhook processes, each deployed as independent pods with their own RBAC, health checks, and metrics endpoints. This split allows independent scaling and failure isolation: the webhook server can handle admission requests even if the controller is temporarily unavailable, and vice versa. Both processes share the same binary (`spark-operator`) with different subcommands (`controller start` and `webhook start`), which simplifies image management.

The SparkApplication controller implements a sophisticated 16-state state machine (New, Submitted, Running, Succeeding, Completed, Failing, Failed, FailedSubmission, Suspending, Suspended, Resuming, PendingRerun, Invalidating, etc.) that tracks applications through their complete lifecycle. The controller creates driver pods via `spark-submit` rather than directly, which preserves Spark's native submission semantics including configuration resolution, dependency handling, and pod template generation. This architectural choice means the operator image must include the full Spark distribution (via PySpark), which significantly increases image size but ensures compatibility with upstream Spark behavior.

The webhook system implements a 26-step ordered mutation pipeline for Spark pods (`sparkpod_defaulter.go`), which injects everything from environment variables and volumes to GPU resources and Prometheus monitoring configuration. This mutation-heavy approach means that Spark pods submitted by the driver are extensively modified before scheduling, enabling the operator to enforce organizational policies (scheduling, security contexts, resource limits) without requiring users to configure them in their SparkApplication specs. The webhook also performs resource quota validation, calculating total driver + executor resource requirements including memory overhead factors (0.4x for JVM, 0.1x for non-JVM applications) and checking against namespace ResourceQuota objects.

A notable FIPS compliance gap exists: the Go binary is built with `CGO_ENABLED=0` (static linking), which means it uses pure-Go crypto rather than OpenSSL. This will fail Konflux's check-payload validation for FIPS-enabled clusters. The TLS configuration does disable HTTP/2 (setting NextProtos to `["http/1.1"]`) as a security hardening measure, but the underlying crypto is not FIPS-compliant. The operator image does include Java 17 OpenJDK (for Spark) which uses its own TLS implementation, but the operator binary itself is the concern.

The SparkConnect controller (`v1alpha1`) represents a newer addition that deploys a persistent Spark Connect server pod with an associated Kubernetes Service exposing four ports: driver RPC (7078), BlockManager (7079), Web UI (4040), and the SparkConnect protocol (15002). Unlike SparkApplication which is fire-and-forget, SparkConnect maintains a long-running server that clients connect to for interactive query execution with dynamic executor allocation.
