## Purpose

**Short**: Kubernetes operator that automates Apache Spark application lifecycle management — submission, monitoring, scheduling, and cleanup of driver and executor pods.

**Detailed**: The Spark Operator extends Kubernetes with three Custom Resource Definitions (SparkApplication, ScheduledSparkApplication, SparkConnect) to provide a declarative interface for running Apache Spark workloads. Users create SparkApplication CRs specifying their Spark jobs (JAR/Python/R/Scala), and the operator handles spark-submit invocation, driver/executor pod creation, web UI service exposure, Prometheus monitoring configuration, and automatic retry/cleanup logic via a state-machine reconciliation loop.

The operator ships as a single container image that bundles both the Go operator binary and a PySpark 4.0.1 runtime with Java 17. It is deployed as two separate Deployments — a controller (reconciling CRDs) and a webhook server (validating and mutating SparkApplication and Pod resources). The webhook server injects Spark configuration, volumes, environment variables, security contexts, sidecar containers, and monitoring agents into driver/executor pods via a 24-step mutation pipeline. The operator integrates with batch schedulers (Volcano, Yunikorn, Kube-scheduler) for gang scheduling and with Prometheus for metrics collection.

The RHOAI distribution deploys the operator into the `redhat-ods-applications` namespace with NetworkPolicies restricting Spark pod communication to intra-namespace traffic and OpenShift ingress. The operator was forked from the upstream Kubeflow Spark Operator and adapted for Red Hat's Konflux build system with hermetic builds and multi-architecture support (x86_64, aarch64, ppc64le, s390x).

## Architectural Analysis

The Spark Operator implements a clean separation between its two runtime components — the controller and the webhook server — each running as independent Deployments with separate ServiceAccounts and RBAC policies. This separation allows independent scaling and failure isolation: the controller manages CRD reconciliation with a 10-thread worker pool and rate-limited work queue (50 QPS, 500 bucket, 6-hour max delay), while the webhook server handles admission requests on port 9443 with its own leader election and certificate lifecycle.

The SparkApplication controller uses a comprehensive state machine with 13+ states (New → Submitted → Running → Succeeding/Failing → Completed/Failed, plus Suspended, PendingRerun, Invalidating, Resuming states). This state machine drives the reconciliation logic and handles edge cases like spec updates during execution (Invalidating → PendingRerun), suspension/resumption, and retry with linear backoff. The controller also implements TTL-based garbage collection for terminated applications and tracks up to 1000 executors per application.

The webhook server's SparkPod defaulter implements a 24-step sequential mutation pipeline that transforms bare Spark pods into fully-configured workloads. This pipeline injects owner references, environment variables, ConfigMap mounts (Hadoop, Spark, application-specific), volumes, container ports, security contexts, affinity rules, GPU resources, Prometheus monitoring configuration, sidecar containers, init containers, and lifecycle hooks. The mutation is triggered by the label `sparkoperator.k8s.io/launched-by-spark-operator: "true"` on pods created by spark-submit. Container selection logic handles both Spark 2.x ("executor") and Spark 3.x ("spark-executor") naming conventions.

The batch scheduler integration uses a registry-based plugin architecture where Volcano (via PodGroup CRDs), Yunikorn (via pod annotations with task group definitions), and custom Kubernetes schedulers can be registered at startup. This allows gang scheduling of driver and executor pods — ensuring all pods in a Spark application are scheduled together or not at all, preventing resource deadlocks in multi-tenant clusters. The Volcano integration checks for both v1 and v1beta1 API versions for backward compatibility.

A notable FIPS concern exists: the Go operator binary is built with `CGO_ENABLED=0` (static linking), which means it uses pure-Go crypto rather than the system OpenSSL FIPS module. This would fail Konflux's check-payload validation if FIPS enforcement is enabled. The bundled Java 17 runtime uses the host OpenSSL for TLS, which is FIPS-compliant on RHEL/UBI. The operator's self-signed certificate generation uses RSA 2048-bit keys with a 10-year validity period and 180-day pre-expiry renewal check. HTTP/2 is disabled by default in both controller and webhook TLS configurations as a CVE mitigation measure.
