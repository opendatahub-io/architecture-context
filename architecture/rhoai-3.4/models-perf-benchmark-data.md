## Purpose

**Short**: Provides pre-built benchmark performance data for Red Hat AI validated models, packaged as a container image used as an init container by the model catalog service.

**Detailed**: The models-perf-benchmark-data component serves as the data supply pipeline and packaging layer for the RHOAI model catalog's performance benchmark data. It consists of two distinct parts: (1) a data container image that packages pre-processed model benchmark data (performance metrics, evaluation scores, validation results, and provider metadata) into a minimal UBI9-based container that runs as an init container alongside the model catalog deployment, and (2) a Kafka-based data processor that consumes model benchmark data from Kafka topics, transforms and organizes it into a hierarchical provider/model directory structure with JSON and NDJSON files, and produces the static data payload that gets baked into the container image.

The data container image does not run any services — it exists solely to provide a volume mount at `/app/benchmarks` containing the structured benchmark data files. It is injected as an init container into the `model-catalog` deployment in the `odh-model-registries` namespace, where the main model catalog service reads the benchmark data at startup. The Kafka processor is a development/build-time tool that is not deployed to clusters; it runs offline to pull data from Kafka topics (via SASL_SSL authenticated connections), apply transforms (accuracy aggregation, hardware configuration enrichment), and write the `data/` directory that is then COPYed into the container image at build time.

## Architectural Analysis

The models-perf-benchmark-data component follows a data packaging pattern common in Kubernetes deployments where static data needs to be injected into service pods. Rather than having the model catalog service fetch benchmark data at startup (which would introduce runtime dependencies and latency), the data is pre-processed and baked into a container image that runs as an init container. This init container writes data to a shared volume, making it immediately available to the main container without network calls.

The two-phase architecture (offline Kafka processor + static data image) provides a clean separation of concerns. The Kafka processor handles the complexity of consuming CDC (Change Data Capture) messages from multiple Kafka topics (`dev.aihub-public-Model`, `dev.aihub-public-ModelProvider`, etc.), detecting entity types via topic name analysis or field-based heuristics, and organizing data into a hierarchical `provider/model` directory structure. It supports both simple message formats and legacy Debezium CDC envelopes, with the Debezium support marked as temporary. The processor also applies post-processing transforms like `OverallAccuracyTransform` (averaging evaluation scores) and `HardwareConfigurationTransform` (combining hardware type and count fields).

The data image itself is intentionally minimal — it uses `registry.access.redhat.com/ubi9-minimal` as a base (for Konflux/FIPS compliance) and contains no runtime code beyond `sleep infinity` as a default CMD, which is overridden by the Kubernetes init container configuration. The image runs as a non-root user (UID 1001) with read-only data permissions. The output data files follow a well-defined schema: `provider.json` and `metadata.json` for entity metadata (overwritten on updates), and `performance.ndjson`, `evaluations.ndjson`, `validation.ndjson` for time-series data (append-only NDJSON format). A `manifest.json` index file at the root catalogs all files and provides statistics.

One architectural consideration is that this is marked as a confidential repository containing private benchmark data. The Dockerfile.konflux uses `ubi9-minimal` while the upstream Dockerfile uses `ubi9-micro`, suggesting the Konflux build prioritizes FIPS compliance (ubi9-minimal includes more system libraries) over minimal image size.
