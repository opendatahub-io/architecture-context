## Purpose

**Short**: Kubernetes operator that automates model serving infrastructure lifecycle for KServe InferenceServices and LLMInferenceServices on Red Hat OpenShift AI.

**Detailed**: The odh-model-controller is a controller-runtime based Kubernetes operator that manages the auxiliary infrastructure required when users deploy machine learning models via KServe. It watches InferenceService, LLMInferenceService, ServingRuntime, and NIM Account custom resources and reconciles the networking, authentication, observability, and autoscaling resources needed to expose and operate model endpoints.

The controller creates OpenShift Routes for traditional InferenceServices and manages Kuadrant AuthPolicy and Istio EnvoyFilter resources for LLMInferenceServices that use Gateway API. It provisions Prometheus ServiceMonitors and metrics dashboards per model, configures KEDA TriggerAuthentication for autoscaling, aggregates CA certificates and data connection secrets into unified storage configs, and manages Ray TLS certificates for multi-node distributed inference workloads.

A companion API server (`model-serving-api`) provides an HTTPS endpoint for gateway discovery, enabling the RHOAI dashboard to enumerate available Gateway API gateways and their listeners for model deployment. The operator also integrates with NVIDIA NIM for GPU-accelerated inference runtime provisioning, managing API key validation, pull secrets, and OpenShift Templates for NIM-based ServingRuntimes.

## Architectural Analysis

The odh-model-controller employs a sophisticated sub-reconciler composition pattern that separates concerns while maintaining a unified reconciliation lifecycle. The primary `InferenceServiceReconciler` orchestrates six sub-reconcilers (route, metrics service, ServiceMonitor, dashboard, ClusterRoleBinding, KEDA) through a shared interface, each owning specific resource types. This design allows independent evolution of observability, autoscaling, and networking concerns while ensuring coordinated cleanup through finalizer-based deletion and non-controlling owner references that support shared namespace resources.

The LLM subsystem introduces a parallel reconciliation architecture with two independent controllers: the `LLMInferenceServiceReconciler` manages per-model AuthPolicies targeting HTTPRoutes, while the `GatewayReconciler` bootstraps gateway-level infrastructure (EnvoyFilter and AuthPolicy) independently of any specific model deployment. This separation ensures that the TLS/auth infrastructure is ready before models are deployed, which is critical for Authorino integration. The Gateway controller uses a multi-source event model where changes to LLMInferenceServices, LLMInferenceServiceConfigs, and Gateways all trigger reconciliation through different event handlers with deduplication.

The authentication architecture has two distinct layers: traditional InferenceServices use OpenShift Routes with optional `enable.auth.odh` annotation-based auth delegation via ClusterRoleBindings to `system:auth-delegator`, while LLMInferenceServices use Kuadrant AuthPolicies with Authorino performing KubernetesTokenReview and SubjectAccessReview checks. The AuthPolicy templates embed CEL expressions for path-based authorization decisions and support fairness/objective header injection for the Gateway API inference extension. The EnvoyFilter configures an upstream TLS context to the Authorino gRPC service at port 50051, using the service-ca.crt trust anchor. A significant design decision is the annotation-driven auth type detection: `odh-auth.enabled=false` switches from UserDefined (full RBAC enforcement) to Anonymous (public access), with the controller managing the lifecycle of the corresponding AuthPolicy variant.

The NIM integration is a self-contained subsystem with its own CRD (`nim.opendatahub.io/Account`), controller, webhook, and handler pipeline. It manages the complete lifecycle of NVIDIA GPU inference runtimes: API key validation against NGC, model catalog fetching, Docker pull secret generation for nvcr.io, and OpenShift Template creation embedding ServingRuntime configurations. The air-gapped mode skips external validation entirely, making NIM deployable in disconnected environments. The singleton-per-namespace constraint enforced by the validating webhook prevents conflicting NIM configurations.

The model-serving-api server is a minimal, security-hardened HTTPS service with a notable design choice: it passes user bearer tokens through to the Kubernetes API for SelfSubjectAccessReview checks, ensuring that gateway discovery respects the calling user's RBAC permissions rather than using the server's own elevated service account. The server implements FIPS-compliant TLS with restricted cipher suites (ECDHE+AES-GCM only), hot-reloadable certificates via fsnotify, and comprehensive security headers. The topology spread constraints in its Deployment manifest ensure high availability across zones and nodes.
