## Purpose

**Short**: Kubernetes controller that manages the lifecycle of model serving infrastructure for KServe InferenceServices, LLMInferenceServices, and NVIDIA NIM integrations on Red Hat OpenShift AI.

**Detailed**: odh-model-controller is a controller-runtime based operator that extends and customizes KServe model serving on the RHOAI platform. It watches KServe CRDs (InferenceService, ServingRuntime, InferenceGraph, LLMInferenceService) and creates supplementary Kubernetes resources required for production model serving — including OpenShift Routes for traffic ingress, Prometheus ServiceMonitors for metrics collection, KEDA TriggerAuthentication for autoscaling, Kuadrant AuthPolicies for gateway-level authentication, Istio EnvoyFilters for TLS bootstrapping, and network policies for isolation.

The controller also provides NVIDIA NIM integration through a custom Account CRD that manages NGC API key validation, model catalog synchronization, and OpenShift Template generation for NIM ServingRuntimes. A companion model-serving-api server provides a REST API for gateway discovery, enabling the RHOAI dashboard to discover available Gateway API gateways for LLMInferenceService deployments. The controller handles complex operational concerns like multi-node Ray TLS certificate management, S3 data connection secret aggregation, CA certificate bundle aggregation, and migration cleanup from RHOAI 3.3 to 3.4+.

## Architectural Analysis

The odh-model-controller follows a composition-over-inheritance architecture for reconciliation. Each major controller (InferenceService, LLMInferenceService) delegates to multiple sub-resource reconcilers that each manage a single Kubernetes resource type. The `KserveRawInferenceServiceReconciler` is an orchestrator that composes six sub-reconcilers (Route, ClusterRoleBinding, MetricsService, ServiceMonitor, MetricsDashboard, KEDA), calling each in sequence and collecting errors via `multierror`. This pattern provides clean separation of concerns and allows enabling/disabling individual sub-reconcilers.

The reconcilers use a consistent delta-processing pattern: compute desired state, fetch existing state, compare using resource-specific comparators (e.g., `RouteComparator`, `ServiceComparator`, `EnvoyFilterComparator`), and apply the delta. The `DeltaProcessor` handles the Added/Updated/Removed state machine. Shared resources (KEDA TriggerAuthentication, ServiceAccount) use a `SingleResourcePerNamespace` mixin that prevents deletion during individual ISVC reconciliation and only cleans up during explicit namespace cleanup when all ISVCs are removed. KEDA resources use non-controlling OwnerReferences allowing multiple ISVCs to co-own the same resources.

The model-serving-api is architecturally separate from the controller — it's a standalone Go HTTP server with its own Deployment, ServiceAccount, and ClusterRole. It uses a minimal RBAC footprint (only `list gateways` and `get namespaces`) and enforces authorization by performing a SelfSubjectAccessReview with the user's forwarded Bearer token before returning gateway data. The TLS configuration is explicitly FIPS-compliant, restricting cipher suites to ECDHE+AES-GCM with NIST P-256/P-384 curves. Certificate reloading is handled via fsnotify filesystem watching with debouncing, supporting Kubernetes secret volume mount rotation patterns.

The controller manages a complete CA certificate trust chain for model serving. The ConfigMap controller aggregates CA certificates from three sources (`odh-kserve-custom-ca-bundle`, `kserve-custom-ca-cert`, and OpenShift's `service-ca.crt`) into a unified bundle. The Secret controller then constructs `storage-config` secrets that include CA bundle paths when HTTPS S3 endpoints are detected. For multi-node Ray serving, the controller implements its own PKI — generating a self-signed RSA 4096-bit CA, issuing per-pod certificates with Pod IP and wildcard DNS SANs, and injecting them via a mutating webhook that adds init containers and volumes. The MaaS RBAC Cleanup runner implements a one-shot migration task that removes legacy Role/RoleBinding resources from the RHOAI 3.3 tier-based access control model, executing once on leader election startup.

The NIM integration is a self-contained subsystem within the controller. The Account CRD provides a declarative interface for NGC API key management. Four handlers (ValidationHandler, ConfigMapHandler, TemplateHandler, PullSecretHandler) execute sequentially during reconciliation, each updating Account status conditions. The NIM utilities include a hardened HTTP client with TLS 1.2+ minimum version and 30-second timeout for NGC API calls. The controller supports air-gapped deployments where the model list ConfigMap is pre-populated rather than fetched from NGC. A singleton webhook ensures only one Account per namespace, preventing configuration conflicts.
