## Purpose

**Short**: Python SDK for creating, managing, and interacting with Ray clusters and Ray jobs on Kubernetes (OpenShift AI / Open Data Hub).

**Detailed**: The CodeFlare SDK is a Python client library that provides a high-level interface for data scientists and ML engineers to manage distributed computing resources on Kubernetes. It abstracts the complexity of creating and managing KubeRay RayCluster and RayJob custom resources, handling Kubernetes authentication, Kueue queue integration, mTLS certificate generation, and cluster dashboard access.

The SDK operates as a client-side library typically running inside Jupyter notebooks on the RHOAI/ODH platform. It communicates with the Kubernetes API server to create RayCluster CRs (managed by the KubeRay operator), submit RayJob CRs, manage Kueue local queue assignments, generate client-side TLS certificates for mTLS connections to Ray head nodes, and discover dashboard URLs via HTTPRoute (RHOAI 3.x Gateway API) or OpenShift Routes (legacy). The SDK also includes Jupyter widget integrations (ipywidgets) for interactive cluster management directly within notebooks.

The SDK vendors a subset of the KubeRay Python client (`kuberay_cluster_api` and `kuberay_job_api`) for direct interaction with the KubeRay operator API, and depends on `kube-authkit` for Kubernetes authentication abstraction.

## Architectural Analysis

The CodeFlare SDK follows a **client-side abstraction** pattern where all Kubernetes interactions happen through the user's credentials. Unlike operators that run in-cluster with service accounts, the SDK runs in the user's notebook and relies entirely on the user's RBAC permissions. This design means the SDK's security posture is directly tied to the namespace-scoped RBAC of the data scientist — it cannot escalate privileges beyond what the user already has.

The SDK implements a **dual-mode ingress discovery** pattern to handle the RHOAI 2.x → 3.x migration. The `cluster_dashboard_uri()` method first attempts to find an HTTPRoute via Gateway API labels (`ray.io/cluster-name`, `ray.io/cluster-namespace`), searching cluster-wide first and then falling back to namespace-specific searches across known platform namespaces (`redhat-ods-applications`, `opendatahub`). If no HTTPRoute is found, it falls back to OpenShift Routes and then Kubernetes Ingresses. This tri-layer fallback ensures compatibility across RHOAI 3.x (Gateway API), RHOAI 2.x (OpenShift Routes), and vanilla Kubernetes (Ingress). The dashboard URL construction for Gateway API follows the pattern `https://{gateway-hostname}/ray/{namespace}/{cluster-name}`.

The **mTLS certificate lifecycle** is tightly integrated with the cluster lifecycle. When `cluster.apply()` is called, the SDK polls for the KubeRay-created CA secret (labeled `ray.openshift.ai/cluster-name`), then generates client certificates using RSA 3072-bit keys and SHA-256 signatures. Certificates include SANs for `localhost`, `127.0.0.1`, `::1`, and the cluster's service DNS names (`{cluster}-head-svc.{ns}.svc.cluster.local`). Files are stored in `~/.local/share/codeflare/tls/` with 0600 permissions and are automatically cleaned up when `cluster.down()` is called. The SDK also provides certificate refresh (`refresh_certificates()`), expiry-based cleanup, and age-based cleanup utilities.

The **RayJob module** introduces a `ManagedClusterConfig` dataclass that builds embedded RayCluster specs for RayJob CRs. This allows submitting batch jobs that create their own ephemeral clusters. The module handles local file upload by creating Kubernetes Secrets with owner references to the RayJob CR (ensuring cleanup), and supports working directory zipping with base64 encoding. An init container on the submitter pod template unzips working directories before the job runs. Kueue integration is automatic: for managed clusters, the SDK auto-detects the default local queue and adds `kueue.x-k8s.io/queue-name` labels; for existing clusters, Kueue labels are intentionally skipped since Kueue only manages RayJobs that create new clusters.

The codebase shows clear **deprecation patterns**: `TokenAuthentication` and `KubeConfigFileAuthentication` are deprecated in favor of `kube-authkit`'s `AuthConfig`, and `cluster.up()` is deprecated in favor of `cluster.apply()` (which uses server-side apply). The ODH trusted CA bundle is mounted as optional ConfigMap volumes in all Ray pods, ensuring custom CA certificates are available without failing if the ConfigMap doesn't exist.
