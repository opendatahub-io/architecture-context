## Purpose

**Short**: Python SDK that provides an intuitive interface for creating, managing, and submitting jobs to Ray clusters on Kubernetes/OpenShift.

**Detailed**: The CodeFlare SDK is a Python client library that simplifies the lifecycle management of Ray clusters and Ray jobs on Kubernetes and OpenShift platforms. It provides a high-level, data-scientist-friendly API for requesting compute resources, creating and configuring RayCluster custom resources, submitting RayJob custom resources, and monitoring cluster and job status. The SDK is designed to be used from Jupyter notebooks running within RHOAI/ODH workbenches, and includes interactive Jupyter widgets for cluster management.

The SDK interacts with the Kubernetes API server to create and manage RayCluster and RayJob custom resources (managed by the KubeRay operator). It handles authentication through multiple mechanisms via the kube-authkit library, including auto-detection, token-based, OIDC, OpenShift OAuth, kubeconfig, and in-cluster service account authentication. The SDK also generates mTLS certificates for secure Ray client connections by fetching CA secrets from the cluster and creating signed client certificates locally.

For RHOAI 3.x, the SDK supports Gateway API HTTPRoute-based dashboard access (alongside legacy OpenShift Route and Kubernetes Ingress fallbacks), Kueue integration for workload queuing and prioritization, Ray in-tree autoscaling, GCS fault tolerance with Redis, and runtime environment packaging for RayJob submissions.

## Architectural Analysis

The CodeFlare SDK follows a clean layered architecture. At the top layer, `Cluster` and `RayJob` classes provide the user-facing API. These delegate to a middle layer of builder functions (`build_ray_cluster`, `ManagedClusterConfig.build_ray_cluster_spec`) that construct Kubernetes resource dicts using the `kubernetes` client library's typed models (V1Container, V1PodSpec, etc.). The bottom layer consists of the vendored KubeRay Python client (`kuberay_cluster_api.py`, `kuberay_job_api.py`) and the Kubernetes API client wrapper (`auth.py`) that handle actual API calls. This separation means resource construction is testable independently of API calls.

Authentication is notable for its dual-path design. The SDK is migrating from legacy `TokenAuthentication` / `KubeConfigFileAuthentication` classes to the `kube-authkit` library, with deprecation warnings on the old classes. The `config_check()` function implements a priority chain: (1) existing api_client, (2) kube-authkit auto-detection, (3) legacy kubeconfig/in-cluster fallback. The global `api_client` variable pattern is a pragmatic choice for notebook environments where auth state persists across cells, though it introduces global mutable state that complicates testing.

The mTLS certificate management in `generate_cert.py` is architecturally significant. The SDK fetches the CA certificate and private key from a KubeRay-managed Kubernetes Secret (labeled `ray.openshift.ai/cluster-name`), then generates RSA-3072 client certificates locally with proper x509v3 extensions (BasicConstraints, KeyUsage for digitalSignature+keyEncipherment, ExtendedKeyUsage for serverAuth+clientAuth, and SANs including the head service DNS names). Certificates are stored in `~/.local/share/codeflare/tls/` with 0600 permissions and cleaned up automatically on `cluster.down()`. This approach avoids requiring users to manage certificates manually, but means the CA private key is temporarily in SDK process memory during generation.

The RHOAI 3.x Gateway API integration in `cluster.py` is a noteworthy evolution. The `_get_dashboard_url_from_httproute()` function implements a multi-step resolution: list HTTPRoutes by label selector, extract the parent Gateway reference, fetch the Gateway to get the hostname (from spec.listeners, then Route fallback, then status.addresses), and construct the URL as `https://{hostname}/ray/{namespace}/{cluster-name}`. This three-layer fallback (HTTPRoute -> Route -> Ingress) ensures backward compatibility with pre-3.x deployments while supporting the new Gateway API pattern.

The RayJob subsystem (`ray/rayjobs/`) introduced a new `ManagedClusterConfig` dataclass alongside the existing `ClusterConfiguration`, with a cleaner API (e.g., `head_accelerators` instead of `head_extended_resource_requests`). It handles local file packaging by zipping working directories into base64-encoded Secrets with RayJob owner references for automatic cleanup. The validation layer checks Ray version compatibility between the SDK (2.54.1) and the specified runtime image, preventing subtle version mismatch issues in distributed Ray environments.
