## Purpose

**Short**: Data Science Pipelines provides a Kubernetes-native ML pipeline orchestration platform for defining, scheduling, running, and tracking machine learning workflows on Red Hat OpenShift AI.

**Detailed**: Data Science Pipelines (DSP) is a downstream distribution of Kubeflow Pipelines v2, providing a complete ML pipeline management system. It enables data scientists to define reproducible ML workflows as directed acyclic graphs (DAGs), store pipeline definitions, schedule recurring executions, cache intermediate results, and track execution metadata and artifacts.

The platform consists of five deployable components built from this repository: an API server that exposes REST/gRPC endpoints for pipeline CRUD operations, a driver that prepares execution context and generates Kubernetes pod specifications, a launcher that executes user containers and manages artifact I/O, a persistence agent that synchronizes workflow state to the database, and a scheduled workflow controller that manages recurring pipeline executions. These components work together with external dependencies including Argo Workflows (for workflow orchestration), ML Metadata (MLMD) for lineage tracking, and object storage (S3/GCS/MinIO) for artifact persistence.

In the RHOAI distribution, all components are built with FIPS-compliant cryptography using `GOEXPERIMENT=strictfipsruntime` and deployed on UBI 9 base images. The platform is managed by the Data Science Pipelines Application (DSPA) operator (separate repository), which handles deployment, configuration, and lifecycle management of these components within each user namespace.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| API Server | Go Service | REST/gRPC API for pipeline definitions, runs, experiments, artifacts, and recurring jobs; includes Kubernetes webhook for PipelineVersion validation/mutation |
| Driver | Go CLI (Argo step) | Runs as init container in Argo Workflow steps; prepares execution context, resolves inputs, generates pod spec patches, manages caching |
| Launcher | Go CLI (Argo step) | Runs alongside user containers; manages artifact download/upload, publishes execution metadata to MLMD |
| Persistence Agent | Go Service | Watches Argo Workflows and ScheduledWorkflows; syncs execution state to API server database via gRPC |
| Scheduled Workflow Controller | Go Controller | Watches ScheduledWorkflow CRDs; evaluates cron/periodic schedules; submits new pipeline runs when due |

## Architectural Analysis

Data Science Pipelines follows a microservices architecture derived from Kubeflow Pipelines v2, with five distinct Go binaries sharing a common codebase. The API server serves as the central hub, exposing a dual-protocol interface (REST via gRPC Gateway on port 8888, native gRPC on port 8887) and an optional webhook server on port 8443 for Kubernetes-native pipeline storage. This multi-server design within a single binary is pragmatic — it reduces deployment complexity while maintaining protocol flexibility for different clients (SDK uses REST, internal components use gRPC).

The pipeline execution model uses a clever three-phase architecture built on top of Argo Workflows. The Compiler transforms KFP pipeline specifications into Argo Workflow manifests, injecting driver and launcher images as task containers. The Driver runs as init/step containers within Argo Workflow pods and performs the "control plane" work — resolving input parameters from upstream tasks, computing cache fingerprints, generating pod spec patches via Kubernetes strategic merge patches, and managing PVC lifecycle. The Launcher then runs alongside the user's container to handle the "data plane" — downloading input artifacts from object storage, executing user code, uploading outputs, and publishing execution metadata to MLMD. This separation of concerns between Driver (scheduling-time logic) and Launcher (execution-time I/O) is essential for supporting the caching optimization: when a cache hit is detected, the Driver can skip the Launcher entirely, returning cached outputs without executing the user container.

The caching system uses content-addressed fingerprints generated from a composite key of input parameters, output parameters, container image, user command, and mounted PVC names. The cache is queried during the Driver phase; on a hit, the execution is marked as CACHED in MLMD and no Launcher pod is spawned. Cache staleness is configurable per-task and globally, with a maximum staleness cap that prevents unbounded reuse.

A notable FIPS design decision in this codebase is the Launcher's dual-binary approach: `Dockerfile.konflux.launcher` builds both `launcher-v2` (statically linked, non-FIPS, CGO_ENABLED=0) and `launcher-v2-fips` (dynamically linked, FIPS-compliant, CGO_ENABLED=1). The non-FIPS binary exists for environments where static linking is preferred for compatibility. The DSPA operator is responsible for selecting the appropriate binary at deployment time based on the cluster's FIPS mode. All other components (API server, Driver, Persistence Agent, Scheduled Workflow Controller) are built exclusively with FIPS flags.

The multi-user model relies on Kubernetes-native RBAC rather than application-level authorization. The API server performs SubjectAccessReview checks against the Kubernetes API before allowing resource operations, and user identity is extracted either from HTTP headers (when behind an auth proxy like kube-rbac-proxy) or via TokenReview validation of bearer tokens. Namespace isolation is the primary tenancy boundary. Projected service account tokens with 1-hour expiration and the `pipelines.kubeflow.org` audience are used for inter-service authentication between the Persistence Agent/Scheduled Workflow Controller and the API server, providing time-bounded credentials without static secrets.
