## Purpose

**Short**: Data Science Pipelines provides an ML pipeline orchestration platform that enables users to define, schedule, execute, and monitor machine learning workflows on Kubernetes using Argo Workflows as the execution engine.

**Detailed**: Data Science Pipelines (DSP) is the RHOAI distribution of Kubeflow Pipelines v2, a Kubernetes-native platform for building and deploying portable, scalable ML workflows. The platform decomposes pipeline execution into multiple cooperating services: an API server that manages pipeline definitions, runs, and experiments via gRPC/REST APIs with database persistence; a driver/launcher pair that orchestrates individual pipeline steps within Argo Workflow pods; a scheduled workflow controller that implements cron-based and periodic pipeline scheduling; a persistence agent that watches Argo Workflow execution state and syncs it back to the API server; and a cache server that implements a Kubernetes mutating admission webhook to skip redundant pipeline steps by injecting cached results.

The platform stores pipeline metadata in MySQL or PostgreSQL, pipeline artifacts in S3-compatible object storage (MinIO, AWS S3, or GCS), and execution lineage in ML Metadata (MLMD). In RHOAI, the platform is deployed by the data-science-pipelines-operator (DSPO), which provisions per-namespace DSP instances based on DataSciencePipelinesApplication (DSPA) custom resources. All Go binaries are built with FIPS-compliant cryptography (`GOEXPERIMENT=strictfipsruntime`).

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| API Server | Go Service | Central REST/gRPC API for managing pipelines, runs, experiments, and artifacts with database and object store backends |
| Driver | Go CLI (Argo sidecar) | Orchestrates pipeline step execution within Argo Workflow pods; resolves inputs, evaluates conditions, manages caching, generates pod spec patches |
| Launcher | Go CLI (container wrapper) | Wraps user container execution; downloads input artifacts, runs user code, uploads output artifacts, publishes to MLMD |
| Persistence Agent | Go Controller | Watches Argo Workflows and ScheduledWorkflows, persists execution state to API server database |
| Scheduled Workflow Controller | Go Controller | Manages ScheduledWorkflow CRD; creates Argo Workflows or API Runs on cron/periodic schedules |
| Cache Server | Go Webhook | Kubernetes mutating admission webhook that intercepts pod creation and injects cached pipeline outputs |
| Viewer Controller | Go Controller | Manages Viewer CRD instances; creates Tensorboard deployments and services |
| Frontend UI | TypeScript/React | Web interface for browsing pipelines, runs, experiments, and visualizations |
| Visualization Server | Python Service | Generates pipeline run visualizations |
| Metadata Writer | Python Service | Writes pipeline metadata to MLMD |

## Architectural Analysis

Data Science Pipelines implements a sophisticated distributed architecture for ML pipeline orchestration that splits concerns across five cooperating Go services, all coordinated through Kubernetes as the shared control plane. The most distinctive architectural pattern is the **driver/launcher separation**: the driver runs as an Argo Workflow init container to resolve pipeline DAG semantics (input resolution, condition evaluation, caching, pod spec patching) while the launcher runs inside the user container to manage artifact I/O and MLMD publishing. This separation means the user container never needs direct knowledge of the pipeline framework—it simply executes with pre-staged inputs and the launcher handles metadata bookkeeping.

The **caching system** spans three components (driver, cache server webhook, API server) in an interesting design. The driver generates SHA256 fingerprints from task inputs, outputs, image, and command, then queries the API server's TaskService for matching cached executions. Independently, the cache server operates as a Kubernetes mutating admission webhook that intercepts pod creation and can inject cached results via JSON patches, providing a second caching checkpoint at the pod scheduling level. This dual-layer caching (driver-level and webhook-level) provides defense in depth but introduces complexity—the two caching paths must be kept consistent. PVC names are included in cache fingerprints to detect side-effect changes from persistent storage, which is a thoughtful design choice for ML workloads that often depend on data stored on volumes.

The **database abstraction** deserves note: the API server uses GORM as an ORM layer and supports both MySQL and PostgreSQL as backends, with a custom SQL dialect abstraction (storage/db.go) that handles differences in GROUP_CONCAT, upsert syntax, and locking semantics. SQLite is additionally supported for testing. The cache server maintains its own separate database connection for cache metadata, which means a DSP deployment has two independent database clients hitting the same (or different) database instances.

The **launcher dual-binary pattern** (in `Dockerfile.konflux.launcher`) is architecturally significant for FIPS compliance: the image ships both a statically-linked non-FIPS binary (`launcher-v2`, CGO_ENABLED=0) and a dynamically-linked FIPS binary (`launcher-v2-fips`, CGO_ENABLED=1 + strictfipsruntime). The API server selects which binary to inject into user pods based on cluster FIPS mode. This is necessary because the launcher runs inside arbitrary user containers where OpenSSL may not be installed—the static binary provides a fallback for non-FIPS clusters where dynamic linking to OpenSSL would fail.

Security is implemented at multiple layers: Kubernetes RBAC via SubjectAccessReview/TokenReview for API authentication, projected ServiceAccount tokens with custom audience (`pipelines.kubeflow.org`) for inter-service auth with hourly rotation, TLS for gRPC connections, and Pod Security Standards baseline enforcement for pipeline step pods (capabilities dropped, seccomp RuntimeDefault, non-root execution). The multi-user mode adds Istio AuthorizationPolicies and VirtualServices for namespace-level isolation. The pipeline-runner ServiceAccount has notably broad permissions (wildcard access to kubeflow.org resources, deployments, pods, services) which is inherent to the pipeline execution model—pipelines can create arbitrary Kubernetes resources.
