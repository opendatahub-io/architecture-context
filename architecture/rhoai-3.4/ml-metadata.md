## Purpose

**Short**: ML Metadata (MLMD) is a gRPC server that records and retrieves metadata associated with ML workflows, tracking artifacts, executions, contexts, and their lineage relationships.

**Detailed**: ML Metadata is a library and server originally developed by Google as part of the TFX (TensorFlow Extended) ecosystem. It provides a centralized metadata store for recording and retrieving metadata about ML pipelines, including artifacts (datasets, models, metrics), executions (training runs, transformations), and contexts (pipelines, experiments). The metadata store tracks provenance and lineage through events that link artifacts to executions, and through attributions and associations that link artifacts and executions to contexts.

In the RHOAI platform, MLMD is deployed as a gRPC server (`metadata_store_server`) that provides a persistent metadata backend for Kubeflow Pipelines. The server is built from C++ source using Bazel, supports multiple database backends (MySQL/MariaDB, PostgreSQL, SQLite), and exposes a comprehensive gRPC API defined in Protocol Buffers for CRUD operations on metadata entities and lineage graph traversal. The Python client library provides a high-level API for interacting with the gRPC server or directly with a local database.

This is a Red Hat downstream fork of the upstream `google/ml-metadata` project, customized with Red Hat UBI-based container images for deployment on OpenShift. The Konflux pipeline builds multi-architecture images (amd64, arm64, ppc64le) for the RHOAI platform.

## Architectural Analysis

ML Metadata follows a classic three-tier architecture: gRPC API layer → metadata store logic → database backend. The `MetadataStoreServiceImpl` class implements the gRPC service interface and delegates to the `MetadataStore` class, which abstracts database operations through `MetadataAccessObject` and database-specific query executors (MySQL, PostgreSQL, SQLite). This design cleanly separates the API contract from storage implementation.

The data model is centered around three core entity types (Artifact, Execution, Context) with a rich type system that supports schema evolution. Types define property schemas (int, double, string, struct, proto, boolean), and the system enforces backward compatibility during type updates — properties can be added but not removed or changed in type. The lineage model uses Events to link Artifacts and Executions (with typed relationships: INPUT, OUTPUT, DECLARED_INPUT, DECLARED_OUTPUT, etc.), and Attributions/Associations to link entities to Contexts. Parent-child relationships between Contexts enable hierarchical organization (e.g., pipeline → run → task).

The server's `metadata_store_server_main.cc` reveals a flexible configuration system: the server accepts database configuration via either a protobuf text config file (`--metadata_store_server_config_file`) or individual command-line flags for MySQL/PostgreSQL connection parameters. The `--metadata_source_config_type` flag controls which configuration approach to use. Database schema migration is built in, with upgrade/downgrade support and retry logic for transient connection failures (up to 5 retries by default for Aborted errors).

From a security perspective, the server supports TLS for the gRPC endpoint (configurable via `SSLConfig` in the server config protobuf) and optional mutual TLS with client certificate verification. Database connections also support TLS with per-backend SSL options. However, the crypto implementation uses BoringSSL compiled from source via Bazel rather than system OpenSSL, which creates a FIPS compliance question — BoringSSL has its own FIPS validation but is not Red Hat-certified. The container runs as non-root user 65534:65534, which is good security practice. The non-hermetic build (`hermetic: false` in the Tekton pipeline) is another security concern, as build-time dependencies are fetched from external sources without comprehensive lock files.

The Konflux build pipeline produces multi-architecture images (amd64, arm64, ppc64le) from `Dockerfile.konflux`, which uses a two-stage build: a UBI9 builder stage with Bazel 5 from COPR, and a minimal UBI9-minimal runtime stage containing only the compiled binary, third-party source attributions (mariadb-connector-c), and tzdata. The image is published to `quay.io/rhoai/odh-mlmd-grpc-server-rhel9`.
