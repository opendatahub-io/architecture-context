## Purpose

**Short**: ML Metadata (MLMD) is a library and gRPC server for recording and retrieving metadata associated with ML workflows, tracking artifacts, executions, and their lineage.

**Detailed**: ML Metadata is a Google-originated open source project (from the TFX ecosystem) that provides a metadata store for machine learning pipelines. It records artifacts (datasets, models, metrics), executions (training runs, transformations), and contexts (pipelines, experiments) along with the events that connect them, enabling full provenance tracking and lineage graph traversal across ML workflows.

The Red Hat downstream fork (`red-hat-data-services/ml-metadata`) packages the C++ gRPC server binary into a UBI9-based container image (`odh-mlmd-grpc-server`) for deployment on OpenShift. The server supports multiple database backends including MySQL, PostgreSQL, and SQLite, with PostgreSQL being the primary backend in RHOAI deployments. The server exposes a gRPC API on port 8080 that Kubeflow Pipelines and other ML platform components use to store and query pipeline run metadata, artifact lineage, and execution history.

The Python client library (`ml-metadata` PyPI package) provides both direct database connectivity and gRPC client access, allowing pipeline components to record metadata during execution. The proto definitions establish a type system for artifacts, executions, and contexts with support for custom properties, versioned type evolution, and SQL-like filtering queries.

## Architectural Analysis

ML Metadata follows a clean separation between the data model (defined in proto files), the storage layer (C++ metadata access objects for MySQL/PostgreSQL/SQLite), and the service layer (gRPC server). The proto files define a graph-based data model centered on three node types (Artifact, Execution, Context) connected by three edge types (Event, Attribution, Association), with ParentContext edges for hierarchical context organization. This design directly maps to the ML pipeline lineage tracking use case: pipeline steps are Executions, their inputs and outputs are Artifacts, and pipeline runs are Contexts.

The server architecture is stateless by design - all state lives in the backing database. Each gRPC request creates a fresh MetadataStore instance connected to the database, and the server can be horizontally scaled behind a load balancer (though the current deployment typically uses a single replica). The server performs schema migrations on startup (`enable_database_upgrade` flag), supporting version upgrades and downgrades. The schema migration system tracks database schema versions and applies incremental migrations, with verification queries to ensure correctness.

The Python client library has a dual-mode design: it can either connect directly to a database (using the C++ MetadataStore via pybind11) or connect to the gRPC server. The direct database mode is used in testing and single-process scenarios, while the gRPC mode is used in production deployments. The client implements retry logic with exponential backoff for transient Aborted errors, which occur during database transaction conflicts.

A notable security consideration is that the gRPC server has minimal built-in authentication. TLS/mTLS is optional and configured via protobuf config. In RHOAI deployments, network-level isolation (Kubernetes NetworkPolicies, namespace boundaries) provides the primary access control rather than application-level authentication. The server runs as non-root user (UID 65534) in the container. The BoringSSL dependency compiled from source (rather than using system OpenSSL for gRPC) represents a FIPS compliance gap - while database connections use the system's FIPS-validated OpenSSL via libpq, the gRPC TLS termination uses non-validated BoringSSL.

The Konflux build pipeline builds for three architectures (x86_64, arm64, ppc64le), using a multi-stage Docker build with a UBI9 builder stage (with Bazel 5 from COPR repository) and a UBI9-minimal runtime stage. The build is currently non-hermetic (`hermetic: false` in Tekton pipeline), meaning external dependencies are fetched during build time rather than being pre-fetched.
