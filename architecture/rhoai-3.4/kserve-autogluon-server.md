## Purpose

**Short**: KServe model server for AutoGluon TabularPredictor and TimeSeriesPredictor models, serving predictions via KServe inference protocol v1 and v2.

**Detailed**: The kserve-autogluon-server is a Python-based inference server that integrates the AutoGluon machine learning framework with the KServe serving infrastructure. It enables deployment of AutoGluon tabular and time series models as KServe `InferenceService` resources on Red Hat OpenShift AI (RHOAI). The server auto-detects whether a model artifact is a `TabularPredictor` or `TimeSeriesPredictor` by attempting to load each in sequence, then delegates inference requests to the appropriate handler.

For tabular models, the server supports both KServe inference protocol v1 (JSON with `instances` array) and v2 (tensor-based `InferRequest`), including optional `predict_proba` for classification tasks. For time series models, only REST v1 JSON is supported, accepting historical time series data with optional known covariates for the forecast horizon. The server runs on the KServe Python SDK (`kserve` package) which provides a FastAPI/Uvicorn HTTP server with gRPC support, Prometheus metrics, and storage integration for downloading models from cloud storage (S3, GCS, Azure Blob).

This repository is a downstream fork of the upstream `kserve/kserve` monorepo, containing only the autogluon server component and its dependencies. The RHOAI build uses Red Hat's internal PyPI mirror (`console.redhat.com/api/pypi/public-rhai/rhoai/3.4/cpu-ubi9/simple/`) for hermetic dependency resolution, and produces a multi-architecture container image (x86_64, ppc64le, s390x, arm64) via Konflux.

## Architectural Analysis

The kserve-autogluon-server follows the standard KServe serving runtime pattern: a Python server that implements the KServe `Model` interface, wrapping a specific ML framework (AutoGluon) to serve predictions via the KServe inference protocol. The architecture is straightforward — a single container with no controller logic, no CRD definitions, and no direct Kubernetes API interaction. The server's complexity lies in bridging AutoGluon's DataFrame-oriented API with KServe's tensor-based v2 protocol.

A notable design choice is the auto-detection pattern in `predictor_factory.py` and `predictor_detect.py`: rather than requiring users to specify whether their model is tabular or time series, the server tries `TimeSeriesPredictor.load()` first (since tabular models sometimes partially load as time series), then falls back to `TabularPredictor.load()`. This simplifies the user experience at the cost of a failed load attempt on every tabular model. The `AutoGluonModelRepository` class provides a fallback path — if single-model loading fails, the server treats the model directory as a repository containing multiple named model subdirectories.

The v2 protocol support for tabular models (`tabular_model.py`) is comprehensive, handling the impedance mismatch between tensor-based v2 payloads and AutoGluon's feature-name-based DataFrames. The server maps each v2 input tensor name to a feature column, validates shapes (requiring `[batch]` or `[batch,1]`), and maps AutoGluon feature types to v2 datatypes (INT64, FP64, BOOL, BYTES). Classification output includes a dynamic datatype determination (`_determine_prediction_datatype`) that inspects the predictor's class labels to select the most appropriate v2 output type. Time series models explicitly reject v2 tensor payloads (`InferRequest`), supporting only v1 JSON — this is documented as a phase 1 limitation.

The `runtime_paths.py` module addresses a practical container security concern: AutoGluon's `SeasonalNaive` fallback attempts to create directories under the current working directory, which fails in non-root OpenShift containers where cwd is typically `/`. The `ensure_autogluon_runtime_paths()` function probes writable directories (`/tmp`, `$HOME`, `/home/kserve`) and changes cwd to the first writable candidate. This is called both at startup and before each time series prediction to handle edge cases where cwd might be reset.

The RHOAI build pipeline (`Dockerfile.konflux.autogluon`) uses a notable dependency strategy: all 130+ transitive Python dependencies are pinned in `autogluon-all-requirements.txt` and sourced exclusively from Red Hat's internal PyPI mirror. The `autogluon` packages themselves carry a `+rhaiv.2` version suffix, indicating Red Hat-specific patches (likely for build compatibility or security fixes). The Konflux pipeline enables hermetic builds with Hermeto pip prefetch, though no `rpms.lock.yaml` is present for OS-level package hermeticity — the UBI9 base image is trusted as-is. The build targets four architectures (x86_64, ppc64le, s390x, arm64), with `hf-transfer` and `triton` conditionally excluded on s390x.
