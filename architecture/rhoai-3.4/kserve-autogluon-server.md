## Purpose

**Short**: A KServe model serving runtime that serves AutoGluon TabularPredictor and TimeSeriesPredictor models for inference via REST APIs.

**Detailed**: The kserve-autogluon-server is a Python-based inference server that implements the KServe model server interface for AutoGluon machine learning models. It is deployed as a container within KServe `InferenceService` pods when users specify `autogluon` as the model format. The server auto-detects whether a model artifact is a TabularPredictor or TimeSeriesPredictor by attempting to load each type in sequence.

For tabular models, the server supports both KServe inference protocol v1 (JSON with `instances`) and v2 (tensor-based `InferRequest`), including optional `predict_proba` for classification tasks via the `PREDICT_PROBA` environment variable. For time series models, only REST v1 JSON is supported, accepting historical data via `instances` and optional `known_covariates` for the forecast horizon.

This repository is a downstream fork of the upstream kserve/kserve repository, carrying only the AutoGluon-specific serving code and its Konflux build configuration for RHOAI. The AutoGluon library versions are pinned to Red Hat-rebuilt variants (e.g., `autogluon.tabular==1.5.0+rhaiv.2`) sourced from a private Red Hat PyPI index. The server is built as a multi-architecture container image (x86_64, ppc64le, s390x, arm64) using hermetic builds via Konflux.

## Architectural Analysis

The kserve-autogluon-server follows a clean delegation pattern built on the KServe Python SDK. The `AutoGluonDetectedModel` class acts as a facade that auto-detects whether the saved model is a `TabularPredictor` or `TimeSeriesPredictor` by attempting to load each in sequence (timeseries first, then tabular). Once detected, it delegates all inference calls to the appropriate implementation class (`AutoGluonTabularModel` or `AutoGluonTimeSeriesModel`). This pattern avoids requiring users to specify the model type in the InferenceService manifest — they simply point `storageUri` at the AutoGluon save directory.

The tabular model implementation (`tabular_model.py`) is notably more sophisticated than the time series one. It supports both v1 and v2 KServe inference protocols, with extensive v2 tensor handling: it maps feature tensors by name to DataFrame columns, validates feature counts against the predictor's expected features, and converts prediction outputs into properly typed v2 InferOutput tensors. The code includes careful handling of prediction datatypes — it inspects `class_labels` to determine whether outputs should be INT64, FP64, or BYTES. The `predict_proba` mode (controlled by the `PREDICT_PROBA` environment variable) produces per-class probability outputs with sanitized label names. In contrast, the time series model (`timeseries_model.py`) only supports v1 JSON and explicitly rejects v2 InferRequest payloads.

The `runtime_paths.py` module addresses a container-specific challenge: AutoGluon's SeasonalNaive fallback attempts to create directories under the current working directory, which fails in non-root OpenShift containers where cwd defaults to `/`. The module intelligently searches for writable directories (`/tmp`, `$HOME`, `/home/kserve`) and changes the process working directory to the first writable candidate. This is a practical adaptation for running AutoGluon in restricted container environments.

The build system uses a hermetic Konflux pipeline with all Python dependencies pinned in `autogluon-all-requirements.txt` and sourced from Red Hat's private PyPI index (`console.redhat.com/api/pypi/public-rhai/rhoai/3.4/cpu-ubi9/simple/`). The AutoGluon packages themselves carry Red Hat version suffixes (`+rhaiv.2`), indicating they are rebuilt from source with patches or configuration specific to the RHOAI platform. The image builds for four architectures (x86_64, ppc64le, s390x, arm64). Notable is the comment in the Dockerfile where `libgomp` installation is commented out — this runtime dependency for OpenMP-based backends (LightGBM, XGBoost) appears to be provided by the UBI9 base image.

The `AutoGluonModelRepository` class provides a fallback loading path: if direct model loading fails (e.g., multi-model serving where the model is in a subdirectory), the server initializes with a model repository that scans the model directory for subdirectories and loads each as a separate model. This enables multi-model serving scenarios without requiring separate InferenceService instances per model.
