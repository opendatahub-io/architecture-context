## Purpose

**Short**: Web dashboard providing the primary user interface for Red Hat OpenShift AI, integrating project management, model serving, pipelines, model registry, and Gen AI capabilities through a modular plugin architecture.

**Detailed**: The odh-dashboard is the central UI component of the Red Hat OpenShift AI (RHOAI) platform. It consists of a Node.js/Fastify backend that serves a React/PatternFly single-page application and proxies Kubernetes API calls on behalf of authenticated users. The dashboard implements a modular architecture using Webpack Module Federation, enabling federated UI plugins to be loaded dynamically from Backend-for-Frontend (BFF) sidecar containers running in the same pod.

The deployment topology places 9 containers in a single pod: the main dashboard (Node.js on port 8080), a kube-rbac-proxy sidecar (ports 8443/8444) for authentication enforcement, and 7 Go BFF sidecars (ports 8043-8743) each serving a React micro-frontend and proxying domain-specific API calls to upstream services. The BFF modules cover model registry, Gen AI studio, Models-as-a-Service (MaaS), MLflow experiment tracking, evaluation hub, AutoML, and AutoRAG.

The frontend uses the OpenShift Dynamic Plugin SDK for extensibility, a Redux store for user state, and React Context providers for dashboard configuration, area-based feature flags, and RBAC enforcement. All external traffic enters through an HTTPRoute (Gateway API) referencing the `data-science-gateway`, terminating TLS at the kube-rbac-proxy which enforces SubjectAccessReview authorization before forwarding to the dashboard backend.

## Architectural Analysis

### Module Federation Architecture

The dashboard employs a sophisticated module federation architecture to achieve UI modularity without requiring multiple separate web applications. The main dashboard acts as the federation host, loading `remoteEntry.js` from 7 BFF sidecars. Each BFF serves both static React assets and domain-specific API endpoints, creating a self-contained micro-frontend unit. This design means a BFF module (e.g., gen-ai) can be developed, built, and tested independently, and its federated bundle is loaded on-demand when the user navigates to its section.

The federation configuration is injected into the SPA at render time via EJS templating in `backend/src/routes/root.ts`, which reads the `MODULE_FEDERATION_CONFIG` environment variable (derived from the `federation-configmap` ConfigMap). The backend registers HTTP proxies for each module prefix (`/_mf/{name}`) that rewrite paths and forward to the appropriate BFF sidecar port. This proxy layer is necessary because browsers cannot directly reach the BFF sidecar ports from outside the pod -- all traffic enters through the Gateway/kube-rbac-proxy on port 8443.

### Authentication Chain

The authentication model is layered:
1. **Gateway level**: TLS termination at the data-science-gateway (Envoy)
2. **kube-rbac-proxy**: Performs SubjectAccessReview against `project.openshift.io/v1/projects/list` -- this gates access to the entire dashboard
3. **Dashboard backend**: Receives `x-forwarded-access-token` from kube-rbac-proxy and forwards it on K8s API calls
4. **BFF sidecars**: Each BFF uses `--auth-method=user_token` and `--auth-token-header=x-forwarded-access-token` to extract the user's token and forward it to upstream services

This chain ensures that every API call carries the original user's identity, maintaining proper K8s RBAC enforcement throughout.

### Plugin Extension System

The frontend uses the OpenShift Dynamic Plugin SDK to implement a plugin architecture. Routes, navigation items, and project detail sections are registered as plugin extensions. Feature visibility is controlled by "area flags" derived from the DataScienceCluster status and the OdhDashboardConfig CR's 80+ feature flags. This allows Red Hat to enable/disable entire feature sections (e.g., pipelines, model serving, Gen AI) without code changes.

### Handler Override Pattern (BFF)

The model-registry BFF implements a handler override mechanism (`extensions.go`) that allows downstream (Red Hat) implementations to replace upstream handler implementations. This enables the same BFF codebase to serve both the upstream open-source community (opendatahub-io) and the downstream RHOAI product, with Red Hat-specific handlers registered in `internal/redhat/handlers/`. This pattern avoids code forks while allowing product-specific behavior.

### Multi-Container Pod Topology

The 9-container pod (1 dashboard + 1 auth proxy + 7 BFFs) is an unusual deployment pattern that trades pod complexity for simplified networking. All containers share the pod's network namespace, so BFF-to-BFF or dashboard-to-BFF communication uses localhost. The NetworkPolicy allows ingress on all 8 service ports (8443 + 7 BFF ports) from the ingress namespace and same-namespace pods.

### Risk: Pod Resource Contention

With 9 containers in a single pod, resource limits and requests must be carefully tuned. Each Go BFF is relatively lightweight, but the aggregate memory and CPU across all containers could be significant. The deployment specifies 2 replicas, providing redundancy but meaning each replica runs all 9 containers.
