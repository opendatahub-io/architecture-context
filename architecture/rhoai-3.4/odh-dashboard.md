## Purpose

**Short**: The ODH Dashboard is the web-based management console for Red Hat OpenShift AI (RHOAI), providing a unified interface for managing data science projects, notebooks, model serving, pipelines, and AI/ML workflows.

**Detailed**: The odh-dashboard is a multi-tier web application consisting of a React/PatternFly 6 single-page application (frontend), a Node.js/Fastify backend-for-frontend (BFF) server, and multiple Go-based BFF microservices that are deployed as sidecar containers within the same pod. The frontend serves as the primary user interface for the RHOAI platform, enabling data scientists and ML engineers to manage Jupyter notebooks, deploy and monitor ML models via KServe, configure data science pipelines, interact with model registries, and orchestrate generative AI workflows.

The application uses a Module Federation architecture (via Webpack Module Federation / `@module-federation/runtime`) to dynamically load feature-specific UI modules (Gen AI Studio, Model Registry, MaaS, MLflow, Eval Hub, AutoML, AutoRAG) from dedicated Go-based BFF containers running alongside the main dashboard. Each BFF module serves its own frontend assets and provides a domain-specific API layer that proxies to upstream platform services. The main backend proxies Kubernetes API calls, handles authentication via kube-rbac-proxy headers, manages WebSocket connections for real-time resource watching, and integrates with platform services such as Prometheus/Thanos, Data Science Pipelines, TrustyAI, and Model Registry.

The dashboard is deployed behind kube-rbac-proxy for authentication and exposed via Gateway API (HTTPRoute) in RHOAI 3.x. It defines four custom CRDs for managing dashboard configuration, application tiles, documentation resources, and quick starts. The deployment uses kustomize overlays to support ODH, RHOAI addon, and RHOAI on-prem distribution variants.

## Architectural Analysis

The odh-dashboard represents a sophisticated Module Federation architecture that enables independent feature development and deployment while maintaining a unified user experience. The core insight is that the dashboard serves as both a monolithic SPA for base platform features (notebooks, projects, pipelines) and a federation host that dynamically loads feature-specific micro-frontends from Go-based BFF sidecars.

**Module Federation Pattern**: The architecture uses Webpack Module Federation to load remote UI modules (Gen AI, MaaS, MLflow, etc.) at runtime. Each module is built as an independent container with a Go BFF serving both static frontend assets and a domain-specific API. The main dashboard backend acts as a reverse proxy, routing requests prefixed with module paths (e.g., `/gen-ai/api/*`) to the corresponding BFF container on its dedicated port. This is configured via a `federation-config` ConfigMap that maps module names to service ports and proxy rewrite rules. The pattern enables each module team to develop, test, and release independently, while the platform operator controls which modules are deployed via image parameters.

**Authentication Layering**: The security model has three distinct layers. First, the Gateway API + Envoy terminates external TLS and routes to the pod. Second, kube-rbac-proxy on port 8443 validates the user's OAuth token against a permissive policy (can the user `list` projects?), then injects identity headers (`x-forwarded-access-token`, `x-auth-request-user`, `x-auth-request-groups`). Third, the backend applies application-level authorization: `secureRoute()` enforces namespace isolation (users can only modify resources in their own workbench namespace), `secureAdminRoute()` restricts admin-only endpoints (cluster settings, model registry management), and individual routes perform fine-grained checks. The BFF modules receive the already-validated token via the `x-forwarded-access-token` header and use it for upstream K8s API calls.

**Proxy Architecture**: The backend is fundamentally a proxy server. It does not maintain its own data store; instead, it proxies all state to the Kubernetes API or to specialized platform services. The `passThroughResource()` function in `pass-through.ts` handles the core K8s API proxy pattern, while `proxyService()` in `proxy.ts` sets up dynamic HTTP proxies to backend services (DSP, Model Registry, TrustyAI, etc.) with optional CR-gating (verifying a CR exists and has the expected status before allowing proxy traffic). This design means the dashboard is stateless and can be horizontally scaled (the deployment runs 2 replicas with pod anti-affinity across zones).

**Resource Watching and Polling**: Rather than maintaining long-lived K8s watch connections server-side, the backend uses a `ResourceWatcher` class with adaptive polling. When a client actively requests a resource, it polls every 2 minutes; when idle, it backs off to 30-minute intervals. Real-time updates are provided via WebSocket proxy to the K8s watch API, with robust connection management including heartbeats (15s pings), stale connection cleanup (5-minute timeout), and resource version tracking for debugging.

**FIPS Considerations**: The architecture handles FIPS compliance asymmetrically. The main dashboard container is Node.js-based and inherits OpenSSL from the UBI 9 base image without explicit FIPS flags. The Go-based BFF modules are compiled with `CGO_ENABLED=1 -tags strictfipsruntime`, ensuring FIPS-compliant crypto. Notably, the Dockerfile explicitly removes esbuild Go binaries (`rm -rf node_modules/esbuild`) that would fail check-payload validation, documenting this as a known transitive dependency from the `@module-federation/enhanced` build chain. The JWT token in the backend is decoded without cryptographic verification (`jwtUtils.ts`) because kube-rbac-proxy has already validated it - this is an intentional trust boundary, not a security gap.
