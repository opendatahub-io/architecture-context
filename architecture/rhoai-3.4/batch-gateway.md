## Purpose

**Short**: A high-performance system for processing large-scale batch inference jobs in Kubernetes environments, providing an OpenAI-compatible API for submitting, tracking, and managing batch inference workloads.

**Detailed**: Batch Gateway is a multi-component service that enables efficient processing of batch inference workloads alongside interactive workloads. It exposes an OpenAI-compatible REST API (`/v1/batches` and `/v1/files` endpoints) for submitting batch jobs of up to 50,000 requests per job, tracking their progress in real-time, and retrieving results. The system is designed for large-scale LLM inference workloads including dataset inferencing, embedding generation, model evaluations, and cost-optimized batch processing.

The architecture follows a producer-consumer pattern with three independently scalable components: an API Server that handles REST requests and enqueues jobs, a Batch Processor that pulls jobs from a priority queue and dispatches inference requests to downstream gateways with model-aware scheduling and multi-level concurrency control, and a Garbage Collector that periodically removes expired jobs and files. The data layer uses PostgreSQL for persistent metadata, Redis for priority queuing and real-time status updates, and S3-compatible object storage (or local filesystem) for input/output files. The processor implements sophisticated flow control including SLO-driven expiry, prefix-cache-optimized request ordering, per-model concurrency limits, and crash recovery with partial result preservation.

## Architectural Analysis

The batch-gateway follows a classic producer-consumer architecture adapted for LLM batch inference. The API server acts as a thin frontend — it validates requests, stores metadata, and enqueues jobs to Redis, but performs no inference itself. The processor is the compute-intensive component, managing the full lifecycle of batch jobs from ingestion through execution to finalization. This separation enables independent scaling: the API server scales with client request volume while processors scale with inference throughput requirements.

The processor's concurrency model is particularly well-designed. Three levels of semaphores control resource usage: worker tokens (limits concurrent jobs), a global semaphore (limits total in-flight inference requests across all jobs), and per-model semaphores (prevents any single model from monopolizing capacity). The acquisition order (local → global) prevents deadlocks. A "double-release guard" detects concurrency bugs at runtime and triggers graceful shutdown rather than risking data corruption — a production-hardened pattern rarely seen in open-source projects.

The prefix-cache optimization is architecturally significant: during ingestion, the processor computes FNV-32a hashes of system prompts and sorts plan entries by hash. This groups requests with identical system prompts together, maximizing KV-cache hit rates at downstream inference gateways. Combined with the `x-slo-ttft-ms` and `x-gateway-inference-objective` headers injected per request, the processor actively participates in flow control — not just dispatching requests but signaling priority and deadline information to the inference layer.

The crash recovery mechanism deserves attention. On startup, the processor scans its work directory for leftover job artifacts from a previous instance. It inspects each job's database status (validating, in_progress, finalizing, cancelling) and takes phase-appropriate action: finalizing jobs get their partial results uploaded, in-progress jobs with partial results are marked failed (preserving what was done), and jobs still in the validating phase are re-enqueued with reconstructed SLO scores. This ensures no job gets permanently stuck in an intermediate state after a pod crash.

From a security perspective, the system relies on tenant isolation via HTTP headers rather than Kubernetes-level multi-tenancy. The API server extracts a tenant ID from a configurable header (defaulting to `X-MaaS-Username`) and scopes all database queries by tenant — cross-tenant access returns 404 rather than 403 to avoid information leakage. Secrets are mounted at `/etc/.secrets` and read at runtime, never stored in configuration files. The S3 secret access key is explicitly excluded from YAML serialization via `yaml:"-"` tags. However, the current FIPS posture (CGO_ENABLED=0, static linking) is not compatible with OpenShift check-payload validation and would need to be addressed for FIPS-certified deployments.
