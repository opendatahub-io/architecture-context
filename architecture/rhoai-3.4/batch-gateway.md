## Purpose

**Short**: High-performance system for processing large-scale batch inference jobs in Kubernetes, exposing an OpenAI-compatible API for submitting, tracking, and managing batch workloads.

**Detailed**: Batch Gateway provides an OpenAI-compatible REST API (`/v1/batches` and `/v1/files`) for submitting and managing batch inference jobs at scale. It is designed to coexist with interactive (real-time) inference workloads, minimizing interference while meeting batch job SLOs. The system supports up to 50,000 requests per job, model-aware scheduling with per-model execution plans, and intelligent flow control via downstream inference gateway metrics.

The architecture follows a producer-consumer pattern with three independently scalable components: an API server that handles REST requests and stores metadata, a batch processor that dequeues jobs from a priority queue and dispatches inference requests to downstream gateways, and a garbage collector that cleans up expired files and batch metadata. The data layer is pluggable, supporting PostgreSQL or Redis for metadata, Redis for priority queuing and event channels, and S3 or local filesystem for file storage. The system is built for Kubernetes with Helm chart deployment, OpenShift SCC compatibility, Prometheus metrics, OpenTelemetry tracing, and cert-manager TLS integration.

Batch Gateway integrates with the llm-d inference ecosystem, referencing Gateway API InferenceObjective CRDs for priority-based flow control, and supports pass-through headers for external authentication (e.g., Envoy ext_authz) and tenant isolation via configurable HTTP headers.

## Architectural Analysis

Batch Gateway follows a clean producer-consumer architecture with the API server acting as producer (accepting jobs, storing metadata, enqueuing to priority queue) and processors acting as consumers (dequeuing, planning, dispatching, finalizing). This separation enables independent scaling: API servers scale for request throughput while processors scale for inference dispatch throughput. The GC runs as a singleton to avoid concurrent delete races.

The most notable design decision is the **binary plan file architecture** for batch processing. During the ingestion phase, the processor parses each JSONL input file, groups requests by model, and within each model sorts by FNV-32a hash of the system prompt. This creates execution plans where requests with identical system prompts are adjacent, enabling downstream inference gateways to maximize prefix cache hits. The plan files use a compact 16-byte binary format (offset, length, prefix hash) that enables efficient random-access reads from the original input file during execution. This approach keeps memory usage constant regardless of batch size, as only the plan entries (not full request bodies) are held in memory during dispatch.

The **concurrency control** uses a two-level semaphore pattern: a global semaphore limits total in-flight inference requests (default: 100) while per-model semaphores prevent any single model from monopolizing all slots (default: 10 per model). Semaphores are acquired in local-then-global order to prevent starvation. A novel safety feature is the double-release guard: if a semaphore is released twice (programming error), it triggers graceful shutdown rather than silently corrupting state.

The **crash recovery** mechanism is comprehensive. On startup, the processor scans its work directory for stale job artifacts from previous container lifetimes. Recovery is phase-aware: finalizing jobs get their remaining files uploaded, in-progress jobs with partial output get failed-with-partial treatment, and validating jobs get re-enqueued if their SLO hasn't expired. This design ensures that processor pod restarts (during rolling upgrades or node failures) don't permanently lose jobs.

The **tenant isolation model** relies on external authentication. The API server extracts a tenant ID from a configurable HTTP header (default: X-MaaS-Username), and all database queries are scoped to that tenant. The system is designed to sit behind an Envoy proxy with ext_authz, which authenticates requests and injects the tenant header. Pass-through headers (captured at batch creation and stored as tags) are forwarded to the inference gateway, enabling end-to-end auth propagation without the batch gateway needing to understand the auth scheme. This is a clean separation-of-concerns pattern common in MaaS (Model-as-a-Service) architectures.

A potential risk area is the **FIPS compliance gap**: all three Konflux Dockerfiles default to CGO_ENABLED=0, producing statically linked Go binaries with pure-Go crypto. This will fail Red Hat's check-payload validation for FIPS. The CGO_ENABLED is defined as a build ARG (allowing override), but no GOEXPERIMENT=strictfipsruntime flag is configured, suggesting FIPS compliance is not yet prioritized. The codebase does use proper TLS practices (TLS 1.2+ minimum, configurable cipher suites, no MD5/RC4), so the application-level crypto is sound — only the build-time linking needs attention.
