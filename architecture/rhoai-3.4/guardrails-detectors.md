## Purpose

**Short**: A collection of text content detector microservices for the FMS Guardrails Orchestrator, providing PII detection, file-type validation, custom detectors, HuggingFace model-based classification, and LLM-as-a-judge evaluation.

**Detailed**: The Guardrails Detectors repository provides a set of detector algorithm microservices that integrate with the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator), an IBM-led open source project for invoking detectors on text generation input and output. The TrustyAI team maintains these detectors as part of the RHOAI platform's guardrails capability.

The repository produces three distinct detector services, each packaged as a separate container image: (1) the **built-in detector**, which bundles lightweight heuristic detection functions including regex-based PII detection (email, credit card, SSN, phone numbers, IP addresses), file-type validation (JSON, XML, YAML with optional schema), and user-defined custom detectors with static code analysis sandboxing; (2) the **HuggingFace detector**, which loads HuggingFace `AutoModelForSequenceClassification` or `GraniteForCausalLM` models to perform ML-based content classification for risks such as toxicity, jailbreak, and social bias; and (3) the **LLM Judge detector**, which uses the `vllm_judge` library to evaluate content against arbitrary metrics via an external LLM server using an LLM-as-a-judge architecture.

All three detectors share a common FastAPI-based framework (`detectors/common/`) that provides the base application class, health endpoints, Prometheus instrumentation (request counts, detection counts, error counts, runtime latency), and a standardized API schema conforming to the IBM Detector API specification. Each detector is deployed as a KServe `InferenceService` via a `ServingRuntime` definition and exposes a REST API on the `/api/v1/text/contents` endpoint.

## Architectural Analysis

The guardrails-detectors repository follows a clean plugin architecture where each detector type is a self-contained microservice sharing a common framework. The `detectors/common/` package provides `DetectorBaseAPI`, a FastAPI subclass that handles health checks, request validation, error handling, and Prometheus instrumentation. Each detector (built-in, huggingface, llm_judge) extends this base to register its specific detection logic. This pattern means adding a new detector type requires only implementing the detection logic and creating a new `app.py` entry point — the HTTP serving, metrics, and error handling are inherited.

The built-in detector is particularly interesting architecturally because it implements a registry-based plugin system with three sub-registries: `RegexDetectorRegistry` for PII detection patterns, `FileTypeDetectorRegistry` for format validation, and `CustomDetectorRegistry` for user-defined Python functions. The custom detector subsystem includes a static code analysis sandboxing mechanism (`static_code_analysis()` in `custom_detectors_wrapper.py`) that scans user-provided Python code for forbidden imports (`os`, `subprocess`, `sys`, `shutil`) and forbidden function calls (`eval`, `exec`, `open`, `compile`, `input`) before loading it. While this provides some defense-in-depth, it is AST-based static analysis and can be bypassed through dynamic import patterns or obfuscation — it is not a security boundary. The custom detector system also supports Prometheus metric injection via the `@use_instruments` decorator and non-blocking background execution via `@non_blocking`.

The HuggingFace detector supports two distinct model architectures: `AutoModelForSequenceClassification` (multi-label classification with configurable safe labels and threshold) and `GraniteForCausalLM` (generative model that produces "safe"/"unsafe" labels with log-probability scoring). The `SAFE_LABELS` environment variable allows operators to configure which classification labels should be treated as benign, providing per-deployment customization without code changes. The detector uses a single-worker uvicorn configuration (as opposed to the built-in detector's 4 workers) because ML model inference is memory-intensive and GPU-bound — running multiple workers would multiply GPU memory consumption.

A notable architectural gap is that the LLM Judge detector does not have a Konflux Dockerfile variant (`Dockerfile.konflux.judge` does not exist), meaning it is not currently built through the RHOAI Konflux pipeline. Only the built-in detector and HuggingFace detector have Konflux build definitions and Tekton PipelineRuns. This suggests the LLM Judge detector may be in a pre-GA state for the RHOAI 3.4 release, or it may be offered as a community/upstream-only feature. The Tekton PipelineRuns specify multi-arch builds across `linux/x86_64`, `linux/ppc64le`, `linux-m2xlarge/arm64`, and `linux/s390x`, and the HuggingFace Dockerfile.konflux.hf includes extensive architecture-specific build logic for compiling PyTorch and OpenBLAS from source on ppc64le and s390x.

All detectors conform to the [IBM Detector API specification](https://foundation-model-stack.github.io/fms-guardrails-orchestrator/?urls.primaryName=Detector+API) on the `/api/v1/text/contents` endpoint. The request schema (`ContentAnalysisHttpRequest`) accepts a list of text strings and optional detector parameters, while the response (`ContentsAnalysisResponse`) returns per-text detection results with start/end positions, detection type, and confidence scores. The LLM Judge detector additionally exposes `/api/v1/text/generation` for prompt+response evaluation and `/api/v1/metrics` for listing available evaluation metrics from the vllm_judge built-in metric catalog.
