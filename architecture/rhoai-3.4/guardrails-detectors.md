## Purpose

**Short**: A collection of detector algorithm microservices for the FMS Guardrails Orchestrator that perform text content analysis including PII detection, file-type validation, HuggingFace model-based classification, and LLM-as-a-judge evaluation.

**Detailed**: The Guardrails Detectors repository provides multiple independent detector microservices that integrate with the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator), an IBM-led open source project for invoking detectors on text generation input and output. Each detector is a standalone FastAPI-based Python microservice exposing a REST API conforming to the IBM Detector API specification (`/api/v1/text/contents`).

The repository produces three detector types: (1) a **built-in detector** that bundles lightweight heuristic detections (regex-based PII detection for emails, credit cards, SSNs, phone numbers, IP addresses; file-type validation for JSON, XML, YAML with optional schema enforcement; and user-defined custom Python detection functions), (2) a **HuggingFace detector** that loads any `AutoModelForSequenceClassification` or `GraniteForCausalLM` model for deep-learning-based content classification (toxicity, prompt injection, etc.), and (3) an **LLM Judge detector** that uses the `vllm_judge` library to perform LLM-as-a-judge evaluation via an external vLLM-compatible inference server.

Only the built-in and HuggingFace detectors have Konflux Dockerfiles and are shipped as part of RHOAI. The LLM Judge detector has a standard Dockerfile but no Konflux variant on this branch, indicating it may be at a different maturity level for RHOAI shipping. All detectors are designed for deployment as KServe InferenceServices with optional GPU acceleration for the HuggingFace detector.

## Architectural Analysis

The Guardrails Detectors repository follows a **plugin architecture** where each detector type is an independent microservice sharing a common framework. The `detectors/common/` package provides the FastAPI application base (`DetectorBaseAPI`), Pydantic request/response schemas conforming to the IBM Detector API, and instrumentation infrastructure (`InstrumentedDetector`). This shared framework ensures all detectors expose consistent API contracts (`/api/v1/text/contents`, `/health`, `/metrics`) while allowing completely different detection strategies internally.

The **built-in detector** uses a registry pattern (`BaseDetectorRegistry`) where detection functions are organized into three registries: `RegexDetectorRegistry` (PII patterns), `FileTypeDetectorRegistry` (format validation), and `CustomDetectorRegistry` (user-provided Python code). The custom detector subsystem is notable for its security-conscious design — it performs **static code analysis** (AST parsing) on user-provided Python modules before loading them, blocking imports of `os`, `subprocess`, `sys`, `shutil` and calls to `eval`, `exec`, `open`, `compile`, `input`. It also supports decorators for non-blocking execution (`@non_blocking`) and custom Prometheus metrics (`@use_instruments`), enabling background guardrail execution that doesn't block the detection response. This pattern trades strict sandboxing for developer ergonomics — it's a best-effort code inspection, not a process isolation boundary.

The **HuggingFace detector** supports two model architectures: sequence classification (`AutoModelForSequenceClassification`) for standard toxicity/content classifiers, and causal language models (`GraniteForCausalLM`) specifically for IBM Granite Guardian models. The causal LM path iterates over predefined risk categories (harm, social_bias, jailbreak, profanity, unethical_behavior, sexual_content, violence) and uses token-level log probabilities to compute risk scores — a more computationally expensive but nuanced approach than simple classification. GPU acceleration via CUDA is supported with graceful fallback to CPU.

The **LLM Judge detector** takes a fundamentally different architectural approach — it delegates all evaluation logic to an external vLLM server via the `vllm_judge` library, acting as a thin adapter layer between the IBM Detector API format and the OpenAI-compatible vLLM API. It supports both content analysis and generation analysis (evaluating prompt + response pairs). This design enables evaluation using arbitrary large language models without requiring GPU resources on the detector pod itself, but introduces a network dependency on the vLLM server's availability and latency.

A notable architectural observation is the **asymmetry in Konflux shipping readiness**: only `builtIn` and `hf` have Konflux Dockerfiles, while `llm_judge` has only a standard Dockerfile with no `Dockerfile.konflux.*` variant. This suggests the LLM Judge detector is at a different maturity level for RHOAI inclusion. The HuggingFace Konflux Dockerfile is significantly more complex than the built-in one, with multi-arch support (amd64, ppc64le, s390x) including building PyTorch and OpenBLAS from source for non-x86 architectures, and explicit OpenSSL version pinning.

All three detectors produce **Prometheus metrics** using a consistent naming scheme (`trustyai_guardrails_*`) with counters for detections, requests, errors, and runtime. The metric labels include `detector_kind` and `detector_name`, and the code is careful to avoid unbounded label cardinality (e.g., custom regex patterns are bucketed under `custom_regex` rather than publishing the full regex as a label). The built-in detector uses `prometheus_client.multiprocess.MultiProcessCollector` for compatibility with uvicorn's multi-worker mode (4 workers for built-in, 1 worker for HF and judge due to model memory constraints).
