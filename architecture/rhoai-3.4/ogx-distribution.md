## Purpose

**Short**: Builds and configures the Llama Stack AI inference server container image for Red Hat OpenShift AI, bundling all providers, dependencies, and configuration needed to run a Llama Stack server in production.

**Detailed**: The ogx-distribution repository is a container image build project that produces the `odh-llama-stack-core-rhel9` container image for RHOAI. It does not contain application source code itself; instead, it assembles a pre-configured Llama Stack server by installing the upstream `llama-stack` Python package (from the opendatahub-io fork at version v0.7.1+rhaiv.1) along with numerous provider plugins, ML libraries, and runtime dependencies into a UBI 9 Python 3.12 base image.

The distribution is configured via `distribution/config.yaml`, which defines the complete server configuration including API endpoints (inference, vector_io, safety, eval, scoring, files, responses, batches, datasetio, tool_runtime), provider registrations (vLLM, AWS Bedrock, Azure, Vertex AI, OpenAI, WatsonX, Milvus, pgvector, Qdrant, TrustyAI, and more), storage backends (PostgreSQL for both KV and SQL stores), authentication (OAuth2 token validation), and telemetry (OpenTelemetry). Providers are conditionally activated via environment variables, allowing the same image to serve diverse deployment scenarios.

The RHOAI build (`Dockerfile.konflux`) uses pre-built wheel releases from a private GitLab registry (`quay.io/aipcc/base-images/cpu` base image), while the community ODH build (`distribution/Containerfile`) installs dependencies directly from PyPI and GitHub. Both produce a container that runs `llama stack run` via an entrypoint script, optionally wrapped with OpenTelemetry instrumentation. The image pre-caches an embedding model (`ibm-granite/granite-embedding-125m-english`) and tiktoken encodings to avoid runtime downloads.

## Architectural Analysis

The ogx-distribution repository follows a "distribution-as-configuration" pattern rather than containing application source code. The entire value of this repository lies in the curated selection of provider plugins, dependency versions, and the `config.yaml` that wires them together. The upstream `llama-stack` framework (from the opendatahub-io fork) provides all runtime behavior; this repo's job is to assemble a production-ready container image with the right set of dependencies pre-installed and the correct default configuration.

A notable design pattern is the conditional provider activation system. Nearly every provider in `config.yaml` uses the `${env.VAR:+provider_id}` syntax, which means the provider is only registered when the corresponding environment variable is set. This allows a single container image to serve diverse deployment scenarios — from a minimal setup with just vLLM inference to a full-featured deployment with multiple cloud providers, vector databases, safety guardrails, and evaluation pipelines. The base configuration enables only the essential providers (TrustyAI FMS safety, HuggingFace datasets, basic scoring, file search), while everything else requires explicit opt-in.

The build system has two distinct paths: the RHOAI Konflux build (`Dockerfile.konflux`) and the community ODH build (`distribution/Containerfile`). The Konflux build uses a specialized base image (`quay.io/aipcc/base-images/cpu`) with pre-compiled wheel releases fetched from a private GitLab registry, enabling multi-architecture support (x86_64, aarch64, ppc64le) without compiling native extensions during the Docker build. The ODH community build installs directly from PyPI and GitHub, making it simpler but single-architecture. The `distribution/build.py` script automates the ODH Containerfile generation by running `llama stack list-deps` to resolve the dependency tree, then templating the results into `distribution/Containerfile.in`. This is invoked via pre-commit hooks to ensure the generated Containerfile stays synchronized with `config.yaml`.

The security model relies on OAuth2 JWT token validation built into the Llama Stack server itself (not a sidecar proxy), with configurable issuer, audience, and JWKS endpoint. Access policies implement a simple owner-based authorization model: all authenticated users can read system resources and create new resources, but only owners can modify or delete their own resources. The container runs as a non-root user (`CNB_USER_ID`/`CNB_GROUP_ID`), and TLS is not terminated by the application — it relies on the platform's ingress infrastructure (Gateway API / kube-rbac-proxy in RHOAI 3.x) for encryption. The build hermeticity is notably weak: no lock files exist, `hermetic: false` is explicitly set in the Tekton pipeline, and Python dependencies are pinned by version specifier but not by hash.

The storage architecture uses PostgreSQL as the universal backend, configured with two storage types: `kv_postgres` for key-value storage (metadata registry, provider state) and `sql_postgres` for structured data (inference logs, agent responses, conversations, file metadata). This dual-backend pattern allows the Llama Stack server to use a single PostgreSQL instance for all persistence needs while maintaining schema separation between different data types.
