## Purpose

**Short**: Builds and packages the Llama Stack AI framework as a containerized distribution image for deployment within Red Hat OpenShift AI.

**Detailed**: ogx-distribution is a packaging and distribution repository that produces the container image for running a Llama Stack server within the RHOAI platform. Llama Stack is Meta's open framework for building AI applications, providing standardized APIs for inference, safety, evaluation, tool runtime, vector I/O, and more. This repository customizes the upstream Llama Stack framework with an Open Data Hub/RHOAI-specific distribution configuration, pre-installed provider plugins, and a curated set of Python dependencies.

The repository does not contain application source code itself — it defines how the Llama Stack framework and its provider ecosystem are assembled into a deployable container image. The distribution configuration (`distribution/config.yaml`) is the core artifact, defining which API providers are active, how they are configured via environment variables, and what storage backends are used. The build system uses a Python script (`distribution/build.py`) to auto-generate a `Containerfile` from a template, resolving Llama Stack's own dependency tree via the `llama stack list-deps` CLI command.

Two container images are produced: the primary Llama Stack distribution image (built via `Dockerfile.konflux` for RHOAI, or `distribution/Containerfile` for upstream ODH) and an auxiliary vLLM CPU image (`vllm/Containerfile`) used for CI testing and local development. The primary image ships with pre-cached models (e.g., `ibm-granite/granite-embedding-125m-english`), OpenTelemetry instrumentation, and configurable OAuth2 token-based authentication.

## Architectural Analysis

The ogx-distribution repository follows a **packaging-centric architecture** rather than a traditional service architecture. It contains no application source code — the Llama Stack framework (`llama-stack==0.7.1+rhaiv.1`) is installed as a Python package from a Red Hat fork at `github.com/opendatahub-io/llama-stack`. The repository's primary value is the distribution configuration (`distribution/config.yaml`) and the build infrastructure that assembles a validated, production-ready container image.

The configuration uses Llama Stack's **conditional provider activation pattern**, where providers are enabled or disabled based on the presence of environment variables. This is implemented through the `${env.VAR:+value}` syntax in YAML — if the env var is set, the provider is activated; otherwise it is silently skipped. This design allows a single image to serve vastly different deployment topologies: from a minimal vLLM-only setup to a full-featured deployment with multiple cloud inference providers, vector databases, safety services, and evaluation pipelines. The 45+ environment variables create a large configuration surface area, but the defaults are sensible (PostgreSQL-backed storage, TrustyAI safety, vLLM inference).

The **dual-Dockerfile pattern** is notable: `Dockerfile.konflux` is the production build path for RHOAI, consuming pre-built wheel releases from a private AIPCC (AI Platform Container Content) index, while `distribution/Containerfile` is the upstream community path, installing directly from PyPI and GitHub. The upstream Containerfile is auto-generated by `distribution/build.py` using `llama stack list-deps` to resolve the dependency tree — this ensures dependencies stay synchronized with the upstream Llama Stack version. The Konflux build uses the `cpu-ubi9.conf` configuration file to specify the AIPCC base image, wheel versions, and multi-arch support (x86_64, aarch64, ppc64le).

The **storage architecture** is PostgreSQL-centric, with two backend types: `kv_postgres` for key-value storage (used by metadata, vector IO state, batches, etc.) and `sql_postgres` for relational storage (used by inference store, files metadata, agent responses). Both point to the same PostgreSQL instance by default but use separate tables. This design means PostgreSQL is a hard runtime dependency — the server will fail to start without it (as verified by the smoke tests checking for `llamastack_kvstore` and `inference_store` tables).

A **security concern** worth noting is that the server runs on plaintext HTTP (port 8321) and relies on the platform's ingress infrastructure (Gateway API / HTTPRoute managed by rhods-operator) for TLS termination. The OAuth2 authentication is optional and only enabled when `AUTH_ISSUER` is set. The access control policy (read unowned resources, create as authenticated, CRUD own resources) provides resource-level authorization but requires the OAuth2 middleware to be active. Without `AUTH_ISSUER` set, the server is completely open — appropriate for development but a risk if exposed without platform-level auth enforcement.
