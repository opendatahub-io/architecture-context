## Purpose

**Short**: Kubernetes operator that manages LlamaStack distribution server deployments, providing lifecycle management for AI inference serving via the Llama Stack framework.

**Detailed**: The llama-stack-k8s-operator (also known as ogx-k8s-operator in the RHOAI distribution) is a controller-runtime based Kubernetes operator built with kubebuilder. It introduces the `LlamaStackDistribution` (shortName: `llsd`) custom resource, which allows users to declaratively deploy and manage Llama Stack inference server instances.

The operator handles the full lifecycle of a LlamaStack deployment including: resolving container images from a distribution name or custom image reference, creating Deployments with appropriate container specs (including uvicorn workers, health probes, and resource limits), managing persistent storage via PVCs, creating Services for internal access, creating Ingress resources for external access, applying NetworkPolicies for network isolation, managing HorizontalPodAutoscalers for autoscaling, and PodDisruptionBudgets for availability. It also handles CA bundle management by gathering certificates from user-specified ConfigMaps and the ODH trusted CA bundle, concatenating and validating them, and mounting the result into server pods.

The operator uses a kustomize-based manifest rendering pipeline with custom Go plugins (name prefix, namespace, field mutator, NetworkPolicy transformer) to generate Kubernetes resources from base manifests, then applies them to the cluster via server-side apply. It supports both ODH (Open Data Hub) and RHOAI deployments through kustomize overlays with distribution-specific image references and namespaces.

## Architectural Analysis

The llama-stack-k8s-operator follows a clean kubebuilder operator pattern with a single controller managing a single CRD (`LlamaStackDistribution`). The reconciliation loop is well-structured: it first refreshes operator configuration from a shared ConfigMap, then reconciles sub-resources (ConfigMaps for CA bundles, manifest-based resources, and Ingress), and finally updates the CR status with deployment health, provider info, and version information gathered by making HTTP calls to the running LlamaStack pods.

The manifest rendering pipeline is particularly noteworthy. Rather than building Kubernetes objects entirely in Go code, the operator embeds kustomize base manifests (copied into the container image at `/manifests/base/`) and transforms them at runtime using a chain of custom Go kustomize plugins: `NamePrefixPlugin` (adds instance name prefix to resource names, excluding Deployment for backward compatibility), `NamespacePlugin` (sets target namespace), `FieldMutator` (uses JSON Pointer paths to inject dynamic values like replica count, service port, storage size, and instance labels), and `NetworkPolicyTransformer` (dynamically builds NetworkPolicy ingress rules based on `spec.network.allowedFrom` configuration). This approach keeps the resource templates declarative while allowing complex runtime customization.

The CA bundle management system is sophisticated and security-conscious. It supports three sources of CA certificates: explicit user-provided ConfigMaps (via `spec.server.tlsConfig.caBundle`), the platform-level `odh-trusted-ca-bundle` ConfigMap (auto-detected), and combines them into a single operator-managed ConfigMap per instance. All PEM data is validated through X.509 parsing, with size limits (10MB, 1000 certificates) to prevent resource exhaustion. The managed ConfigMap is mounted into pods and referenced via `SSL_CERT_FILE`, ensuring the LlamaStack server trusts both user and platform CAs.

The operator supports a distribution resolution mechanism via an embedded `distributions.json` file that maps distribution names (e.g., "starter", "remote-vllm") to container image references. This can be overridden at runtime via the `RELATED_IMAGE_RH_DISTRIBUTION` environment variable (which replaces the entire map with a single `rh-dev` distribution) or via `image-overrides` in the operator config ConfigMap (which overrides individual distribution name mappings). This layered override system allows RHOAI to ship hardened images while preserving flexibility for development.

The NetworkPolicy implementation follows a defense-in-depth model. When enabled via the `enableNetworkPolicy` feature flag (default: true), the operator creates a NetworkPolicy per LlamaStack instance that allows ingress only from: pods in the same namespace, pods in the operator namespace, explicitly listed namespaces, namespaces matching label selectors, and OpenShift router namespaces (for Ingress support). The wildcard `"*"` in `allowedFrom.namespaces` opens access to all namespaces. This provides sensible defaults while remaining configurable for multi-tenant environments.
