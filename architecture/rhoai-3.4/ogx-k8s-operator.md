## Purpose

**Short**: Kubernetes operator that creates and manages LlamaStack server deployments for running Meta Llama AI model distributions.

**Detailed**: The LlamaStack K8s Operator (ogx-k8s-operator) is a controller-runtime based Kubernetes operator that manages the lifecycle of LlamaStack distribution servers on OpenShift and Kubernetes clusters. It watches a custom resource called `LlamaStackDistribution` (short name: `llsd`) and reconciles the desired state by creating and managing a complete set of Kubernetes resources: Deployments, Services, ServiceAccounts, RoleBindings, NetworkPolicies, Ingresses, PersistentVolumeClaims, PodDisruptionBudgets, and HorizontalPodAutoscalers.

The operator supports multiple LlamaStack distributions (starter, remote-vllm, meta-reference-gpu, postgres-demo) as well as custom container images. It implements CA bundle management by aggregating certificates from user-specified ConfigMaps and the platform's `odh-trusted-ca-bundle` ConfigMap into a single managed ConfigMap mounted into the server pods. The operator uses kustomize internally to render base manifests and applies Go-based transformer plugins for name prefixing, namespace setting, field mutation, and NetworkPolicy configuration.

The operator is designed for the Red Hat OpenShift AI (RHOAI) platform, integrating with OpenShift-specific features such as Security Context Constraints (anyuid SCC), and is deployed into the `redhat-ods-applications` namespace. It supports feature flags (e.g., NetworkPolicy enablement) via an operator-managed ConfigMap, and image mapping overrides for distribution image resolution.

## Architectural Analysis

The LlamaStack K8s Operator follows a clean single-controller architecture built on controller-runtime. The `LlamaStackDistributionReconciler` is the sole controller, watching the `LlamaStackDistribution` CRD and owning all downstream resources. The operator uses kustomize internally (not externally applied) to render base manifests from embedded templates in `controllers/manifests/base/`, then applies a pipeline of Go-based transformer plugins (name prefix, namespace setter, field mutator, NetworkPolicy transformer) before applying resources to the cluster. This approach keeps manifest templates simple while allowing programmatic customization.

A notable design pattern is the dual-client architecture: the operator maintains both a cached client (via the controller-runtime manager cache with label-filtered informers) and a direct (non-cached) API client. The cached client uses label selectors (`app.kubernetes.io/managed-by: llama-stack-operator` and `llamastack.io/watch: true`) to efficiently watch only operator-managed resources and opted-in ConfigMaps. The direct client is used for reading ConfigMaps that lack operator labels, such as user-referenced configuration ConfigMaps, the `odh-trusted-ca-bundle` platform ConfigMap, and the operator's own config ConfigMap. This prevents stale cache reads for resources outside the operator's label scope.

The CA bundle management system is particularly well-designed from a security perspective. Rather than mounting multiple ConfigMap volumes, the operator gathers certificates from both user-specified CA ConfigMaps and the auto-detected `odh-trusted-ca-bundle` ConfigMap, validates every PEM block as a proper X.509 certificate (rejecting non-CERTIFICATE PEM types like private keys), enforces size limits (10MB max, 1000 certificates max), and writes the aggregated result into a single managed ConfigMap. This managed ConfigMap is then mounted into pods via the `SSL_CERT_FILE` environment variable. The approach prevents injection of non-certificate content and provides a single point of CA trust configuration.

The operator supports a rich feature set for production deployments: HorizontalPodAutoscaler integration (with automatic removal of `spec.replicas` from Deployments when HPA is enabled), PodDisruptionBudgets (with a fix for single-replica PDB deadlocks using maxUnavailable instead of minAvailable), topology spread constraints with sensible defaults for multi-replica deployments, and pod anti-affinity rules. The startup script embedded in the reconciler handles version detection across LlamaStack API versions (pre-0.2.17, 0.2.17-0.2.x, 0.3.0+) to select the correct server launch mechanism (legacy module path, core module path, or uvicorn CLI).

One area worth noting is that the operator communicates with managed LlamaStack pods via plaintext HTTP for health checks and provider/version introspection. While this is acceptable for in-cluster communication protected by NetworkPolicies, it means the operator-to-pod communication path has no transport encryption. The NetworkPolicy implementation provides namespace-level isolation with configurable `allowedFrom` rules supporting explicit namespace lists, label-based selection, wildcard access, and OpenShift router ingress traffic.
