## Purpose

**Short**: KServe is a Kubernetes-native model serving platform that provides CRDs and controllers for deploying, scaling, and managing ML inference workloads on OpenShift AI.

**Detailed**: KServe manages the complete lifecycle of ML model serving through two primary CRDs: `InferenceService` (v1beta1) for general-purpose model serving across frameworks like TensorFlow, PyTorch, Triton, HuggingFace, and others; and `LLMInferenceService` (v1alpha2) for LLM-optimized serving using the llm-d stack with disaggregated prefill/decode, Gateway API routing, Inference Gateway scheduler (EPP), and WVA-based autoscaling.

The component produces five container images via Konflux: the main kserve controller, the llmisvc controller, the inference agent (sidecar for logging/batching/model pulling), the inference router (InferenceGraph DAG execution), and the storage initializer (Python-based model downloader supporting S3, GCS, Azure, HuggingFace, OCI, PVC, and HDFS). On RHOAI, the controllers are deployed in the `opendatahub` namespace by the platform operator (rhods-operator) using kustomize manifests from the `config/overlays/odh/` overlay. The ODH overlay replaces cert-manager with OpenShift service-serving-cert-signer, removes ClusterServingRuntime (using namespace-scoped ServingRuntime only), parameterizes all image references via a ConfigMap, and adds RBAC, SCC, monitoring, and accelerator-specific LLMInferenceServiceConfig presets for NVIDIA CUDA, AMD ROCm, Intel Gaudi, and IBM Spyre.

## Architectural Analysis

KServe on RHOAI 3.4 has evolved into a dual-controller architecture, splitting the original monolithic kserve-controller-manager into two independent controllers: the original `kserve-controller-manager` for general-purpose InferenceService workloads, and a new `llmisvc-controller-manager` dedicated to LLMInferenceService workloads. This split is driven by the vastly different lifecycle and dependency requirements of LLM serving — the llmisvc controller depends on Gateway API, Inference Gateway Extensions (InferencePool), LeaderWorkerSet, WVA, and KEDA, none of which are required by the base InferenceService controller. The controllers share the same Go module but are built from different `cmd/` entry points (`cmd/manager` vs `cmd/llmisvc`), and the llmisvc controller uses the `distro` build tag to compile OpenShift-specific features (Istio DestinationRules, SCC management, service-ca certificate signing, Kuadrant AuthPolicy checks).

The LLMInferenceService controller implements a sophisticated configuration inheritance system via LLMInferenceServiceConfig CRDs. Well-known config templates (e.g., `kserve-config-llm-template`, `kserve-config-llm-decode-template`, `kserve-config-llm-scheduler`) are automatically injected as base references depending on the deployment topology (single-node, multi-node, disaggregated P/D). Users can override any field in their LLMInferenceService spec, and the controller merges configs using a deterministic priority order. Config names are pinned in status annotations after first resolution to survive config renames across upgrades. The RHOAI overlay provides accelerator-specific templates for NVIDIA CUDA, AMD ROCm, Intel Gaudi, and IBM Spyre, enabling hardware-aware scheduling without users needing to specify runtime container images.

The networking architecture uses Gateway API as the primary ingress mechanism for LLMInferenceService, with HTTPRoutes pointing to InferencePool backends managed by the Inference Gateway scheduler (EPP). On OpenShift, Istio DestinationRules are created to configure TLS origination from the Istio-based gateway to self-signed or service-ca-signed workload/scheduler services. The controller manages a complex InferencePool API migration from v1alpha2 to v1, creating both versions simultaneously and using annotation-based locking to track which API group the gateway has accepted. For InferenceService, the controller supports multiple ingress strategies: Istio VirtualService, Gateway API HTTPRoute, Kubernetes Ingress, or Knative Service, with priority-based selection.

The autoscaling architecture for LLMInferenceService is built around the Workload Variant Autoscaler (WVA, llmd.ai/v1alpha1). The controller creates a VariantAutoscaling CR that instructs WVA to compute desired replicas based on model characteristics and publish a `wva_desired_replicas` Prometheus metric. An HPA or KEDA ScaledObject then reads this metric to enforce the scaling decision. HPA requires a Prometheus Adapter in the cluster, while KEDA queries Prometheus directly. This two-layer design separates the scaling decision (WVA) from enforcement (HPA/KEDA), allowing the same autoscaling logic to work with different Kubernetes autoscaling backends.

The webhook architecture includes a pod mutating webhook that injects sidecar containers (agent for logging/batching, storage-initializer for model downloads, metrics aggregator, accelerator resources) into InferenceService pods based on annotations set by the controller during reconciliation. This annotation-based injection pattern decouples the webhook from the controller — the webhook reads annotations like `internal.serving.kserve.io/storage-initializer-sourceuri` rather than querying the InferenceService CR directly. The kserve-agent sidecar acts as a reverse proxy on port 9081, intercepting traffic between the queue-proxy (Knative) or external clients and the model server container, providing request/response logging with configurable backends (HTTP, S3-compatible blob storage with JSON/CSV/Parquet marshalling) and request batching.
