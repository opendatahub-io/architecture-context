## Purpose

**Short**: KServe is a Kubernetes-native model serving platform that provides CRDs and controllers for deploying, scaling, and managing machine learning inference services with support for standard (raw), Knative serverless, and LLM-optimized deployment modes.

**Detailed**: KServe is a component operator within the Red Hat OpenShift AI (RHOAI) platform that manages the complete lifecycle of machine learning model inference services. It defines and reconciles multiple Custom Resource Definitions (CRDs) including InferenceService (v1beta1), LLMInferenceService (v1alpha2), InferenceGraph (v1alpha1), TrainedModel, LocalModelCache, and ServingRuntime.

The operator consists of two separate controller managers: the main KServe controller (managing InferenceService, InferenceGraph, TrainedModel, and LocalModel resources) and the LLMInferenceService controller (dedicated to LLM workloads with Gateway API, LeaderWorkerSet, disaggregated prefill/decode, and WVA autoscaling integration). Both controllers use distro-aware build tags to provide OpenShift-specific behavior including Istio DestinationRule management, SecurityContextConstraint bindings, OpenShift Route creation, and CA-signed TLS certificate generation.

The repository also produces multiple companion images: a storage initializer (Python) for downloading model artifacts from cloud storage (S3, GCS, Azure, HuggingFace), an inference agent sidecar (Go) for model pulling/logging/batching, and an inference graph router (Go) for orchestrating multi-step inference pipelines. The Python SDK provides client libraries and a model serving framework used by runtime servers (HuggingFace, sklearn, etc.).

## Architectural Analysis

The KServe repository is a multi-component operator with a complex but well-organized architecture. The codebase is split between two independent controller managers (kserve-controller-manager and llmisvc-controller-manager), reflecting the architectural decision to separate the mature InferenceService API (v1beta1) from the rapidly evolving LLMInferenceService API (v1alpha2). This separation allows the LLM controller to iterate independently with its own webhook, leader election, and deployment lifecycle, which is critical for the 3.3-to-3.4 upgrade path where the single uber-controller was split.

The distro build tag system (`//go:build distro` vs `//go:build !distro`) is used pervasively throughout the controller code to provide OpenShift-specific behavior without runtime feature detection. On OCP, controllers create Istio DestinationRules for TLS origination, manage SecurityContextConstraint RoleBindings for multi-node workloads, check Kuadrant AuthPolicy CRD availability as a gateway precondition, and generate CA-signed TLS certificates using the OpenShift Service CA signing key. This compile-time branching means the Konflux Dockerfiles include `GOTAGS="distro"` (controller) or implicit distro tags to produce OCP-specific binaries.

The LLMInferenceService controller implements a sophisticated configuration composition system: well-known LLMInferenceServiceConfig templates are automatically applied based on the deployment pattern (single-node, multi-node data-parallel, multi-node pipeline-parallel, disaggregated prefill/decode), and user-specified BaseRefs can override these defaults. Template variable substitution (`{{ .Name }}`, `{{ .Namespace }}`) allows config templates to be namespace-aware. The controller also manages complex lifecycle concerns including InferencePool API version migration (v1alpha2 to v1), scheduler configuration migration (v0.6 to v0.7 plugin renames), and workload TLS certificate rotation with DNS SAN and IP SAN tracking.

The pod mutating webhook is a critical integration point: it intercepts pod creation for all InferenceService pods (identified by the `kserve.io/inference-service-pod=true` label) and injects the storage-initializer init container, model agent sidecar (for logging/batching/pulling), metrics aggregation configuration, and credential volumes. This webhook-driven injection pattern means the controller never directly modifies pod specs; instead, it sets annotations on the InferenceService that the webhook interprets during pod creation. This separation of concerns allows the same webhook to work across different deployment modes (raw Deployment, Knative Service) without code duplication.

The kustomize overlay structure for ODH/RHOAI is notably complex, using a `kserve-parameters` ConfigMap with `configMapGenerator` and `replacements` to inject distribution-specific container images. The ODH overlay replaces 10+ image references (controller, storage-initializer, agent, router, scheduler, routing sidecar, tokenizer, and multiple GPU-vendor-specific LLM-D images) and applies OpenShift-specific patches (service CA injection replacing cert-manager, removal of ClusterServingRuntime, network policies). The LLMInferenceServiceConfig resources are intentionally ordered last in the overlay to avoid webhook validation failures during upgrades.
