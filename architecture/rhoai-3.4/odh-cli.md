## Purpose

**Short**: CLI tool for validating, backing up, and migrating Red Hat OpenShift AI (RHOAI) deployments on Kubernetes clusters.

**Detailed**: odh-cli (also known as `kubectl-odh` or `rhai-cli`) is a comprehensive CLI tool that provides upgrade readiness assessment, workload backup, and migration capabilities for RHOAI deployments. It operates as a kubectl plugin that connects to a Kubernetes cluster and performs read-only or write operations depending on the command.

The tool's primary function is the `lint` command, which runs 38 validation checks across platform resources, component configurations, dependency availability, and user workloads to assess whether a cluster is ready for upgrade — particularly from RHOAI 2.x to 3.x. Checks are organized into five groups: platform (DSC/DSCI readiness), components (Dashboard, KServe, Kueue, ModelMesh, Ray, TrainingOperator), dependencies (cert-manager, OpenShift version, Service Mesh v3), workloads (Notebooks, InferenceServices, Guardrails, Kueue queues, LlamaStack, Ray clusters, PyTorchJobs), and services. Each check produces a structured diagnostic result following a Kubernetes-CR-like pattern with severity levels (prohibited, blocking, advisory) and actionable remediation guidance.

The `backup` command discovers and exports workload resources (Notebooks, DSPAs) and their dependencies (ConfigMaps, Secrets, PVCs) to portable YAML files. The `migrate` command provides a framework for version-aware cluster migrations, currently implementing the RHBOK (Red Hat Build of Kueue) migration that transitions Kueue from the embedded RHOAI operator to the standalone Red Hat Kueue Operator via OLM. The container image bundles the CLI alongside `oc`, `kubectl`, `jq`, `yq`, and Python-based upgrade helper scripts for comprehensive upgrade workflows.

## Architectural Analysis

The odh-cli architecture follows a clean layered design with strong separation of concerns. At the foundation, the `pkg/util/client` package provides a unified Kubernetes client abstraction that aggregates dynamic, discovery, API extensions, OLM, and metadata clients into a single `Client` interface. This interface is split into `Reader` (read-only) and `Writer` (write, currently sparse) sub-interfaces, which enforces the principle that most operations are read-only. The client layer is configured with aggressive throttling defaults (QPS=50, Burst=100 — 10x higher than kubectl defaults) to handle the large number of API calls the lint system makes when inspecting dozens of resource types across all namespaces.

The lint framework is the most architecturally sophisticated subsystem. It implements a registry-based check system where each of the 38 checks implements a `Check` interface with `CanApply()` (version-gated applicability) and `Validate()` (execution) methods. Checks use fluent builder helpers (`validate.Component()`, `validate.Workloads()`, `validate.Operator()`, `validate.DSC()`, `validate.DSCI()`) that abstract common patterns like fetching the DSC singleton, filtering by management state, and auto-populating impacted objects. Results follow a Kubernetes-CR-like structure with `DiagnosticResult` containing `DiagnosticSpec` and `DiagnosticStatus` with `Condition` arrays. This design makes results machine-parseable for automation while providing rich human-readable output. The explicit registration pattern in `NewCommand()` (no global state, no `init()` functions) enables full test isolation.

The backup subsystem uses a three-stage pipeline architecture: Discovery (list resources by GVR) → Dependency Resolution (resolve ConfigMaps/Secrets/PVCs) → Writer (strip metadata and output). The pipeline processes workload types sequentially but uses concurrent workers within each stage via `errgroup`. Dependency resolution is pluggable via a `Registry` of `Resolver` interfaces, with current implementations for Notebooks and DSPAs. The field-stripping mechanism uses JQ expressions to remove cluster-specific metadata (`status`, `resourceVersion`, `uid`, `managedFields`, etc.), producing portable YAML.

The migrate subsystem introduces a two-phase `Action` pattern: `Prepare()` (backup and pre-validation) and `Run()` (mutation). The currently registered RHBOK migration demonstrates the pattern well — it checks Kueue management state, preserves the kueue-manager-config ConfigMap by annotating it `opendatahub.io/managed=false`, creates an OLM Subscription for the Red Hat Build of Kueue, updates the DataScienceCluster to set Kueue to `Unmanaged`, and verifies all ClusterQueue/LocalQueue resources survived the transition. Write operations use exponential backoff with conflict retry (`wait.ExponentialBackoff`), and all mutations support dry-run mode and interactive confirmation prompts.

The container image design packages the CLI as an "upgrade toolkit" rather than just a binary. Based on `ose-cli-rhel9` (which provides `oc`), it layers in `kubectl`, `jq`, `python3`, a FIPS-compliant `yq` build, and the `rhoai-upgrade-helpers` scripts. This design supports two usage modes: running `rhai-cli` commands directly via `ENTRYPOINT`, or entering an interactive bash session (`--entrypoint /bin/bash`) for manual upgrade workflows that combine CLI commands with shell scripts. The Konflux Dockerfile differs from the development Dockerfile in several important ways: it uses pinned image digests (not tags), builds yq from vendored source with FIPS flags, copies upgrade helpers from a git submodule (not cloned at build time), and uses `ose-cli-rhel9` as the runtime base (not generic UBI9).
