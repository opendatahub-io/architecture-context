## Purpose

**Short**: CLI tool for validating, diagnosing, and managing RHOAI (Red Hat OpenShift AI) installations and upgrades on Kubernetes/OpenShift clusters.

**Detailed**: The odh-cli (also called `kubectl-odh` or `rhai-cli`) is a standalone Go application designed as a kubectl plugin for interacting with RHOAI deployments. It provides three primary capabilities: (1) a lint/upgrade-readiness engine that validates cluster state and assesses upgrade compatibility between RHOAI versions (particularly 2.x to 3.x migrations), (2) a backup framework for exporting workloads (Notebooks, InferenceServices, etc.) and their dependencies (ConfigMaps, Secrets, PVCs) to portable YAML, and (3) a migration framework for executing version-specific migration actions such as transitioning from built-in Kueue to the Red Hat Build of Kueue Operator.

The CLI connects to a live Kubernetes/OpenShift cluster via kubeconfig and performs read-only inspection of custom resources (DataScienceCluster, DSCInitialization, Notebooks, InferenceServices, etc.) to produce diagnostic reports. It supports table, JSON, and YAML output formats, making it suitable for both interactive use and CI/CD integration. The tool is distributed as a container image (`quay.io/rhoai/rhoai-upgrade-helpers-rhel9`) that bundles the CLI alongside `kubectl`, `oc`, `jq`, `yq`, and Python-based upgrade helper scripts.

## Architectural Analysis

The odh-cli follows a clean, extensible architecture organized around three independent command subsystems (lint, backup, migrate) that share a common Kubernetes client layer and output formatting infrastructure. The client layer wraps `controller-runtime` and `client-go` with a custom `Reader` interface that provides both dynamic (unstructured) and metadata-only resource access, OLM operator discovery, and configurable API throttling (default QPS=50, Burst=100). This is notably higher than kubectl defaults and allows the CLI's parallel backup workers to operate efficiently.

The lint subsystem is the most sophisticated component, implementing a full diagnostic framework modeled after Kubernetes condition semantics. Each check implements a `Check` interface with `CanApply()` (version-gated applicability) and `Validate()` (diagnostic execution) methods, and returns `DiagnosticResult` objects containing conditions with four impact levels (Prohibited > Blocking > Advisory > None). The framework includes a `CheckRegistry` with glob-pattern-based selector matching, allowing users to filter checks by category (components, workloads, dependencies, platform) or individual check ID. Checks are composed using builder patterns (`validate.DSC()`, `validate.Component()`, `validate.Workloads()`, `validate.Operator()`) that handle common operations like fetching singleton resources, verifying component management states, and auto-populating result annotations. Currently there are 37 registered checks (2 platform, 13 component, 3 dependency, 19 workload) with the majority targeting the RHOAI 2.x-to-3.x upgrade path.

The backup subsystem implements a three-stage pipeline pattern (Discovery -> Resolution -> Writing) with configurable worker parallelism. The resolver stage uses a dependency registry to chase references from workloads to their ConfigMaps, Secrets, and PVCs, producing self-contained backup bundles with cluster-specific metadata stripped. The migration subsystem follows a similar registry pattern but with an action lifecycle (CanApply -> Prepare -> Run) and interactive confirmation prompts. The sole migration currently implemented is the Kueue-to-RHBOK migration, which installs the Red Hat Build of Kueue Operator via OLM and transitions the DSC management state from Managed to Unmanaged.

The container image architecture is noteworthy: the Konflux Dockerfile uses a multi-stage build with two builder stages (one for the main Go binary, one for a FIPS-compliant yq binary from vendored source), producing a runtime image based on `ose-cli-rhel9` that bundles the CLI alongside system tools. Both Go binaries are built with `CGO_ENABLED=1 GOEXPERIMENT=strictfipsruntime -tags strictfipsruntime` for FIPS compliance. The image also includes Python 3 with the `kubernetes` client library to support legacy upgrade helper scripts from the `rhoai-upgrade-helpers` submodule. Build hermeticity is enforced through `rpms.lock.yaml` for OS packages, `go.sum` for Go dependencies, and `pip-compile`-generated `requirements.txt` for Python dependencies.
