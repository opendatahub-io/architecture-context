## Purpose

**Short**: A unified Python library providing a single-API interface for multiple LLM fine-tuning algorithms with pluggable backends.

**Detailed**: training-hub is a Python library that abstracts multiple LLM fine-tuning approaches (SFT, OSFT, LoRA+SFT, LoRA+GRPO, GRPO) behind a consistent function-call API. Each algorithm is a thin orchestration layer that delegates actual training to a pluggable backend -- InstructLab Training for SFT, Mini-Trainer (RHAI Innovation) for OSFT, Unsloth for LoRA, ART/OpenPipe for single-GPU GRPO, and verl for multi-GPU distributed GRPO. Users call convenience functions like `sft()`, `osft()`, `lora_sft()`, `lora_grpo()`, and `grpo()` with keyword arguments; the library handles parameter validation, data processing, distributed launch configuration (torchrun), and experiment tracking integration (W&B, MLflow, TensorBoard).

The library also includes a profiling suite with GPU memory estimators for all four algorithm families (SFT, OSFT, LoRA, QLoRA) and an experimental timing estimator that runs a data subset to extrapolate full-run duration. A visualization module produces training loss plots from checkpoint logs. The package is published to PyPI via GitHub Actions with Sigstore signing and is designed to be installed into GPU-equipped training environments as a pip-installable dependency rather than deployed as a service.

## Architectural Analysis

### Algorithm-Backend Abstraction Pattern

The library employs a clean separation between algorithm definition and backend execution. The `Algorithm` ABC defines the parameter contract (required/optional params with types and defaults), while the `Backend` ABC handles actual training orchestration. This allows multiple backends per algorithm (e.g., GRPO can use either ART or verl) and makes adding new training frameworks a matter of implementing a new `Backend` subclass.

The `AlgorithmRegistry` provides static registration and factory lookup, enabling programmatic discovery of available algorithms. Each algorithm self-registers at import time via `AlgorithmRegistry.register()`.

### Parameter Validation Architecture

Each algorithm implements `get_required_params()` and `get_optional_params()` as class methods returning dictionaries mapping parameter names to `(type, default)` tuples. The OSFT algorithm adds particularly thorough runtime type validation, checking each parameter against its declared type and raising descriptive `TypeError` messages. This pattern is inconsistently applied -- SFT and LoRA algorithms rely more on duck typing.

### Distributed Training Launch

The `get_torchrun_params()` utility in `utils.py` implements a sophisticated parameter resolution system with three-level precedence: explicit args > environment variables > defaults. It handles the mutual exclusivity of `master_addr` vs `rdzv_endpoint` with precedence-aware conflict resolution, and supports both `PET_` and non-`PET_` environment variable prefixes for PyTorch Elastic compatibility.

### Backend Execution Strategies

Backends use three distinct execution patterns:

1. **In-process Python call**: SFT (`instructlab.training.run_training()`) and OSFT (mini-trainer API) run within the same Python process.
2. **Subprocess with CLI**: ART GRPO and verl GRPO launch training as subprocesses with command-line arguments, providing process isolation. The verl backend generates extensive Hydra config overrides as CLI flags.
3. **Subprocess with generated files**: The verl backend writes temporary Python modules (reward functions, agent loops) to the checkpoint directory, then references them via CLI config.

### vLLM V1 Compatibility Workaround

The ART GRPO backend includes a notable workaround (`_create_seed_lora_checkpoint()`) for vLLM V1's eager LoRA validation. vLLM V1 validates LoRA adapter dimensions at startup (unlike V0's lazy loading), so the backend creates a zero-effect LoRA checkpoint with correct dimensions before training begins.

### Memory Estimation Architecture

The profiling module implements four estimator classes (`BasicEstimator`, `LoRAEstimator`, `QLoRAEstimator`, `OSFTEstimator`) that calculate theoretical GPU memory requirements without loading models. Each estimator accounts for: model parameters (FP16/BF16/4-bit), gradients, optimizer states (AdamW: 2 copies for momentum/variance), activations (with sequence length and batch size), and algorithm-specific overhead (e.g., SVD decomposition buffers for OSFT, quantization metadata for QLoRA). Model metadata is cached in a local CSV file to avoid repeated HuggingFace Hub queries.

### Supply Chain Security Concern

The `litellm` dependency is pinned to `<1.82.7` with a code comment referencing the TeamPCP supply chain attack on 2026-03-24. This is a runtime-optional dependency (in the `grpo` extras group) used for LLM API calls in GRPO rollouts.

### Missing Test Infrastructure

The repository has no test suite (`tests/` directory, pytest configuration, or test dependencies). The `model_validation.py` script in `scripts/` provides integration-level validation across model architectures but requires GPU hardware. The PyPI CI workflow does not run tests before publishing.
