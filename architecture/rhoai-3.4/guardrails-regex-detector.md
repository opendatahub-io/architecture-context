## Purpose

**Short**: A lightweight Rust HTTP service that detects sensitive content (PII) in text using built-in and custom regular expressions, serving as a detector plugin for the FMS Guardrails Orchestrator.

**Detailed**: The guardrails-regex-detector is a stateless HTTP microservice built with the Axum web framework in Rust. It exposes a single POST endpoint (`/api/v1/text/contents`) that accepts text content and a list of regex patterns, returning detected matches with their positions, types, and confidence scores.

The service ships with three built-in PII detection patterns — email addresses, Social Security numbers, and credit card numbers — and also supports arbitrary custom regular expressions passed at request time via the `detector_params.regex` field. Built-in patterns return typed detections (e.g., `detection_type: "pii"`, `detection: "EmailAddress"`), while custom patterns return `detection_type: "custom"`.

This component is designed to be deployed as a sidecar or auxiliary service alongside the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator), which routes content through multiple detector services to enforce guardrail policies on LLM inputs and outputs within the RHOAI TrustyAI / Guardrails subsystem.

## Architectural Analysis

The guardrails-regex-detector is an intentionally minimal service — under 170 lines of Rust across two source files. This simplicity is a design strength: the component has no state, no database, no outbound network calls, no authentication logic, and no Kubernetes API interaction. Its entire attack surface is a single HTTP POST endpoint and a health check.

The service follows a **plugin detector pattern** for the FMS Guardrails framework. The orchestrator treats each detector as a black box behind a standardized API contract (`POST /api/v1/text/contents` with `contents` and `detector_params`). This allows the regex detector to be swapped, scaled, or replaced independently. The API contract supports both built-in named patterns (`email`, `ssn`, `credit-card`) and arbitrary custom regex strings passed at request time, providing flexibility without configuration files or restarts.

The built-in detector functions (`email_address_detector`, `ssn_detector`, `credit_card_detector`) are hardcoded in `detectors.rs` (lines 38-72) using a function pointer HashMap dispatch pattern (lines 106-113). Adding new built-in detectors requires code changes — there is no dynamic plugin loading or configuration file for patterns. All detections return a confidence score of `1.0` (line 87), which is appropriate for exact regex matches but means the service cannot express partial confidence.

One architectural concern is the **absence of TLS and authentication** on the service port. The service binds to `0.0.0.0:8080` (configurable via `HOST` and `HTTP_PORT` env vars) with no encryption. This is common for cluster-internal services in RHOAI where the platform operator or service mesh provides transport security, but it means the component is entirely dependent on external security controls. If deployed outside a properly configured network namespace, the unauthenticated endpoint could accept arbitrary content for scanning.

The Dockerfile uses a multi-stage build with a Rust 1.84.0 builder and a UBI 9 minimal runtime image. Notably, there is no Konflux-specific Dockerfile (`Dockerfile.konflux`), which is unusual for RHOAI 3.4 components. The `compat-openssl11` package is installed in the runtime image, suggesting some OpenSSL dependency (possibly for check-payload compliance or transitive Rust crate needs), even though the application code itself performs no cryptographic operations. The absence of a `USER` directive means the container runs as root by default — a security finding noted in the component-architecture.json.
