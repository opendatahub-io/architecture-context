## Purpose

**Short**: A lightweight HTTP server that detects PII and custom patterns in text using regular expressions, designed as a detection backend for the FMS Guardrails Orchestrator.

**Detailed**: The Guardrails Regex Detector is a Rust-based microservice that provides regex-based content detection as part of the Red Hat OpenShift AI guardrails stack. It exposes a single POST endpoint that accepts text content and a list of regex patterns (either built-in named patterns or custom regex strings), returning structured detection results with match positions, scores, and classification metadata.

The service ships with three built-in PII detectors — email addresses, Social Security numbers, and credit card numbers — and also supports arbitrary custom regex patterns provided at request time. It is designed to be called by the FMS Guardrails Orchestrator as one of potentially many detection backends, enabling the orchestrator to evaluate LLM inputs and outputs against content safety policies.

The component is intentionally minimal — a stateless HTTP service with no persistent storage, no authentication, no TLS, and no Kubernetes resource management. It relies on the platform infrastructure (network policies, service mesh, kube-rbac-proxy sidecars) for security hardening in production deployments.

## Architectural Analysis

The Guardrails Regex Detector is architecturally one of the simplest components in the RHOAI platform — a stateless, single-endpoint HTTP service with approximately 170 lines of Rust source code (excluding tests). This simplicity is intentional: the component is designed as a pluggable detection backend within the FMS Guardrails framework, where the orchestrator handles routing, aggregation, and policy enforcement while individual detectors focus on a single detection modality.

The detection architecture uses a registry pattern (`HashMap<&str, fn>`) mapping named patterns to built-in detector functions. Three PII detectors are hardcoded: email addresses, Social Security numbers, and credit card numbers. Any regex string not matching a built-in name is treated as a custom pattern and compiled at request time via `Regex::new()`. This means custom patterns incur regex compilation cost on every request — there is no caching of compiled regex patterns across requests, which could be a performance concern for high-throughput workloads with complex custom patterns.

The service has zero security hardening at the application level — no TLS, no authentication, no authorization, no rate limiting, no input validation beyond checking for non-empty regex lists. This is consistent with its role as an internal microservice where the platform infrastructure (Kubernetes network policies, service mesh mTLS, kube-rbac-proxy sidecars) provides the security boundary. However, the absence of any input size limits on the `contents` array or individual content strings means a malicious or misconfigured caller could send arbitrarily large payloads. The regex engine itself is safe against ReDoS (Rust's `regex` crate guarantees linear-time matching), which is an important property for a service that accepts user-supplied patterns.

The Dockerfile uses a multi-stage build with a `rust:1.84.0` builder image and a `ubi9/ubi-minimal` release image, which is the standard RHOAI base image pattern. Notably, there is no Konflux-specific Dockerfile (`Dockerfile.konflux`), which is unusual for a RHOAI 3.4 component. The `compat-openssl11` package is installed in the release image for shared library compatibility but the application itself performs no cryptographic operations. The container runs as root (no `USER` directive), which is a potential security finding for hardened deployments.
