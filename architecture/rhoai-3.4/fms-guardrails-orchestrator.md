## Purpose

**Short**: REST API orchestrator that coordinates AI text generation with content safety guardrails, acting as middleware between clients and language models.

**Detailed**: The FMS Guardrails Orchestrator is a Rust-based microservice that provides a unified API layer for integrating large language model (LLM) text generation with content safety detection. It sits between client applications and multiple backend services — generation services (TGIS or Caikit NLP via gRPC), chunker services (for text segmentation via gRPC), and detector services (for content safety analysis via HTTP REST) — and orchestrates multi-step workflows that combine text generation with input and output guardrails.

The orchestrator supports both unary and streaming operation modes. In the unary path, it validates input, runs input detection, calls the generation service, runs output detection, and returns a combined result. In the streaming path, it uses a sophisticated pipeline architecture with broadcast channels, concurrent detection streams, and detection batch ordering to deliver real-time guardrailed generation results. The service exposes both a legacy v1 classification API and a modern v2 API surface covering content detection, chat detection, context-document detection, generation-with-detection, and OpenAI-compatible chat/text completions with guardrails.

The orchestrator is designed for deployment in Red Hat OpenShift AI (RHOAI) as part of the TrustyAI guardrails stack, providing content safety for inference services. It supports TLS and mTLS for both its own server endpoints and all upstream connections, and includes comprehensive observability through OpenTelemetry traces and metrics exported via OTLP.

## Architectural Analysis

The FMS Guardrails Orchestrator implements a task-based architecture where each API endpoint maps to a specific task type that is dispatched through a generic `Handle<Task>` trait on the `Orchestrator` struct. This design cleanly separates request routing (in `server/routes.rs`) from orchestration logic (in `orchestrator/handlers/`), making it straightforward to add new guardrails workflows without modifying the server layer. The ten handler implementations cover four distinct orchestration patterns: direct detection (content, chat, context), sequential generation-then-detection (generation-with-detection, classification-with-gen), streaming generation with concurrent chunked detection (streaming-classification, streaming-content-detection), and OpenAI-compatible chat/completions with branched streaming/unary execution.

The streaming architecture is notably sophisticated. When a streaming request arrives, the handler spawns a tokio task that manages a pipeline of concurrent async operations: consuming the generation stream, broadcasting generated text chunks to multiple detector streams via `tokio::sync::broadcast`, running detection concurrently across multiple detectors with configurable parallelism (`detector_concurrent_requests`, `chunker_concurrent_requests`), and ordering results through a `DetectionBatcher` trait with two implementations — `CompletionBatcher` for chat/completions (which must handle multiple choices) and `MaxProcessedIndexBatcher` for simpler generation streams. The `CompletionState<T>` struct uses `DashMap` and `OnceLock` for lock-free concurrent access from multiple tokio tasks, with a 30-second timeout for synchronization.

The client architecture abstracts over multiple backend protocols: gRPC for generation (TGIS and Caikit NLP) and chunking, HTTP REST for detectors and OpenAI-compatible services. Each client type provides health checks, TLS/mTLS configuration, and OpenTelemetry instrumentation. The generation client includes retry logic with exponential backoff for retryable status codes (502-506). A notable design decision is the use of the `ginepro` crate (a custom fork) for gRPC client-side load balancing with DNS-based service discovery, supporting both eager and lazy resolution strategies. The `OtelGrpcLayer` provides tower middleware for gRPC request instrumentation with counters for success, 4xx, and 5xx responses plus duration histograms.

From a security perspective, the orchestrator uses rustls with the `ring` crypto backend throughout (not OpenSSL), which has FIPS implications discussed in the FIPS section. TLS is optional for the server endpoints but configurable with mTLS client verification via `WebPkiClientVerifier`. Upstream connections support per-service TLS configuration including an insecure mode (via `NoVerifier`) for development environments. The header passthrough mechanism (`passthrough_headers` in config) allows controlled forwarding of authentication headers to backend services, with a specific `rewrite_forwarded_access_header` option that converts `X-Forwarded-Access-Token` headers into `Authorization: Bearer` headers — this is relevant for KServe/InferenceService integration where auth tokens are forwarded through the inference gateway.

The service runs as a non-root user (`orchestr8`, UID 1001, GID 0) in the container, with no capabilities or privilege escalation. The Dockerfile.konflux uses a multi-stage build with UBI 9 minimal as the base image, pinning OpenSSL to a specific version (3.5.1-7.el9_7) for reproducibility. The build includes running `cargo test` in the builder stage, ensuring that the released binary passes all unit tests.
