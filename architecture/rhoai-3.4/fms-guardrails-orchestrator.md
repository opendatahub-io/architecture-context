## Purpose

**Short**: REST API orchestrator that coordinates AI text generation with content safety guardrails, managing detectors, chunkers, and LLM backends to enforce content safety.

**Detailed**: The FMS Guardrails Orchestrator is a Rust-based middleware service that sits between AI application clients and language model backends, providing content safety detection, filtering, and secure text generation. It receives user requests containing prompts and guardrail configurations, routes text through configurable detector services for safety analysis, chunks text via chunker services for efficient processing, and coordinates with text generation backends (TGIS, Caikit NLP, or OpenAI-compatible servers) to produce guarded outputs.

The orchestrator exposes both v1 (classification-with-generation) and v2 (detection-oriented) REST API surfaces, supporting unary and streaming request patterns. It implements a pluggable architecture where detectors, chunkers, and generation backends are configured via YAML and connected over gRPC (chunkers, TGIS, NLP) or HTTP (detectors, OpenAI). The service supports TLS/mTLS for both inbound connections and outbound client calls, OpenTelemetry-based distributed tracing and metrics, and configurable header passthrough for multi-tenant environments.

Within the RHOAI platform, this component is deployed as part of the TrustyAI stack to provide AI safety guardrails for inference services. It acts as a transparent proxy that can intercept and analyze both input prompts and generated outputs, blocking or flagging content that violates configured safety policies.

## Architectural Analysis

The FMS Guardrails Orchestrator implements a middleware pattern that cleanly separates content safety concerns from text generation logic. By acting as a transparent proxy between clients and LLM backends, it allows safety guardrails to be applied without modifying either the client application or the generation server. The architecture is built around three pluggable backend types — generation servers (TGIS/NLP/OpenAI), chunker services, and detector services — each configured independently via YAML and connected through type-safe Rust clients.

The concurrency model is well-designed for streaming workloads. The orchestrator uses `tokio::broadcast` channels for fan-out patterns (distributing input streams to multiple chunkers), `buffer_unordered` for concurrent detector/chunker requests with configurable parallelism limits (`detector_concurrent_requests` and `chunker_concurrent_requests`), and `mpsc` channels for collecting results from parallel processing pipelines. This allows multiple detectors to analyze the same content concurrently while respecting backpressure. The generation client adds exponential backoff retry logic specifically for transient HTTP errors (502-506), which is appropriate for LLM inference workloads that may experience intermittent overload.

The TLS architecture is comprehensive but has a notable FIPS gap. Server-side TLS supports both one-way TLS and mutual TLS (mTLS) with configurable client CA verification via `WebPkiClientVerifier`. Client-side TLS is configurable per-service through named TLS configs that can be shared across backends. However, the entire TLS stack uses `rustls` with the `ring` crypto backend, which is not FIPS-validated. While OpenSSL is installed in the container image (required for UBI compliance), the application binary does not use it — ring provides its own BoringSSL-derived cryptographic primitives. The `NoVerifier` struct in `src/utils/tls.rs` allows disabling certificate verification entirely when `insecure: true` is set, which is useful for development but represents a security risk if enabled in production.

The service exposes two distinct HTTP servers on separate ports: the guardrails API server on port 8033 (with optional TLS) and a health server on port 8034 (always plaintext). This separation is a good practice — health probes from Kubernetes can reach the health endpoint without TLS configuration, while the API server can enforce mTLS. The health server aggregates health status from all configured backends by performing health checks against each (gRPC Health.check() for TGIS/NLP/Chunker, HTTP GET /health for Detector/OpenAI), providing a unified view of system readiness.

OpenTelemetry integration is thorough, with both trace and metric export via OTLP (gRPC or HTTP). Trace context propagation follows W3C Trace Context standard — incoming trace IDs are extracted and propagated to all downstream gRPC and HTTP calls via traceparent headers. Custom metrics include request counts, response durations, error rates, and streaming response tracking. The `OtelGrpcLayer` tower middleware automatically instruments all gRPC client calls.
