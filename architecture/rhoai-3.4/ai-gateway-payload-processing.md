## Purpose

**Short**: Payload processing plugins for the AI Gateway that mutate inference request/response bodies and headers, enabling multi-provider API translation, API key injection, model-provider resolution, and NeMo guardrail enforcement.

**Detailed**: AI Gateway Payload Processing is a set of plugins built on top of the Kubernetes Gateway API Inference Extension's Body Based Routing (BBR) framework. It runs as an Envoy External Processing (ext_proc) filter attached to a Gateway, intercepting inference API traffic in the data plane. The service provides four core plugins: (1) a **model-provider-resolver** that watches `ExternalModel` CRDs to resolve model names to cloud provider metadata, (2) an **api-translation** plugin that translates between OpenAI Chat Completions format and provider-native formats (Anthropic Messages API, Google Vertex AI GenerateContent, Azure OpenAI, AWS Bedrock), (3) an **apikey-injection** plugin that reads Kubernetes Secrets and injects provider-appropriate authentication headers, and (4) a **nemo-request-guard** plugin that calls NVIDIA NeMo Guardrails to enforce content safety policies on incoming requests.

The component is deployed via a Helm chart that wraps the upstream `body-based-routing` chart from the Gateway API Inference Extension project. It is designed for the RHOAI Model-as-a-Service (MaaS) flow, where `ExternalModel` CRDs define external AI provider endpoints and the payload processing service handles the protocol translation and credential management transparently for consumers using the standardized OpenAI Chat Completions API.

## Architectural Analysis

The AI Gateway Payload Processing component follows a clean plugin architecture built on the upstream Gateway API Inference Extension's BBR (Body Based Routing) framework. Each plugin implements the `framework.RequestProcessor` and/or `framework.ResponseProcessor` interfaces, enabling composable request/response mutation in a well-defined pipeline order. The plugin chain is configured declaratively in the Helm chart's `values.yaml` and executed by the BBR runtime as an Envoy ext_proc filter. This design allows the service to run entirely in the data plane without requiring a control plane component of its own — it leverages controller-runtime only for in-memory caching of ExternalModel CRDs and Secrets.

The inter-plugin communication pattern via `CycleState` is particularly well-designed. The `model-provider-resolver` plugin writes provider metadata (provider name, target model, credential reference) into CycleState during request processing, and downstream plugins (`api-translation`, `apikey-injection`) read this state to make their decisions. This avoids polluting HTTP headers with internal state and keeps plugins loosely coupled — each plugin only depends on well-defined state keys (`provider`, `model`, `credential-ref-name`, `credential-ref-namespace`) rather than on other plugins directly.

The API translation layer supports six provider backends: OpenAI (pass-through with path rewrite), Anthropic (full body translation between OpenAI Chat Completions and Anthropic Messages API including tool calling), Azure OpenAI (path rewrite + response field stripping), Google Vertex AI native GenerateContent (full body translation, currently commented out for 3.4), Google Vertex AI OpenAI-compatible (path rewrite with project/location/endpoint), and AWS Bedrock OpenAI-compatible (path rewrite). The Anthropic translator is the most complex, handling message role mapping (system→system prompt, developer→system, tool→tool_result), tool call translation between OpenAI function calling and Anthropic tool_use formats, and response normalization including usage metrics. The `ResponseFieldStripper` utility provides a reusable mechanism for removing provider-specific response fields via configurable dot-path expressions (e.g., `choices[].content_filter_results`).

A notable security consideration is the NeMo guardrail integration. The `nemo-request-guard` plugin makes an HTTP POST to a configurable NeMo Guardrails service URL, which is expected to be a cluster-internal service. The HTTP client has a configurable timeout (default 360 seconds) and response size limit (1 MiB). NeMo always returns HTTP 200 for both allowed and blocked requests; the block decision is conveyed through the response body's `status` field. If NeMo blocks a request, the plugin returns a 403 Forbidden error that short-circuits the ext_proc pipeline. The NeMo URL and timeout are configured per-plugin instance via JSON config in the BBR plugin chain, allowing different NeMo configurations for different deployment scenarios.

The Secret reconciler in the `apikey-injection` plugin uses a label-based predicate (`inference.networking.k8s.io/bbr-managed=true`) to limit the scope of watched Secrets, which is a good security practice — only explicitly labeled Secrets are cached in memory. The in-memory store uses `sync.RWMutex` for thread safety and maps `namespace/name` keys to the full credential data map from the Secret's `Data` field. All Secret fields must be non-empty or the Secret is rejected from the store.
