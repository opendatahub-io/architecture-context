## Purpose

**Short**: Payload processing service that mutates inference request and response bodies/headers as they flow through the AI Gateway, enabling multi-provider API translation, credential injection, and content guardrails.

**Detailed**: AI Gateway Payload Processing is a set of plugins built on top of the Kubernetes Gateway API Inference Extension's BBR (Body Based Routing) framework. It runs as an Envoy external processor (ext_proc) attached to an Istio Gateway via an EnvoyFilter, intercepting inference requests before they reach backend model servers and responses before they return to clients.

The service implements four production plugins that form a request processing pipeline: (1) **model-provider-resolver** watches `ExternalModel` CRDs to resolve incoming model names to their cloud provider, target model name, and credential reference; (2) **api-translation** translates request bodies from OpenAI Chat Completions format to provider-native formats (Anthropic Messages, Vertex AI GenerateContent, Azure OpenAI, Bedrock OpenAI) and translates responses back; (3) **apikey-injection** watches labeled Kubernetes Secrets and injects provider-appropriate authentication headers (Bearer tokens, API keys); (4) **nemo-request-guard** calls an external NeMo Guardrails service to evaluate input safety rails before forwarding requests.

This architecture enables RHOAI's Models-as-a-Service (MaaS) capability: users send requests in a unified OpenAI-compatible format, and the payload processing pipeline transparently handles provider differences, credential management, and content safety enforcement.

## Architectural Analysis

The AI Gateway Payload Processing component follows a clean plugin architecture built on the upstream Kubernetes Gateway API Inference Extension's BBR framework. Each plugin implements one of two interfaces — `RequestProcessor` or `ResponseProcessor` — and communicates with other plugins exclusively through the `CycleState` shared per-request state object. This design achieves strong separation of concerns: the model-provider-resolver plugin writes provider/credential metadata to CycleState, which downstream plugins (api-translation, apikey-injection) read without direct coupling. The plugin chain ordering is defined in Helm values, not in code, making it reconfigurable without rebuilds.

The reconciler pattern used by both model-provider-resolver and apikey-injection is architecturally significant. Rather than making Kubernetes API calls on the hot path of each inference request, both plugins maintain in-memory stores (`modelInfoStore` and `secretStore`) that are populated asynchronously by controller-runtime reconcilers watching ExternalModel CRDs and labeled Secrets respectively. This means request processing latency is bounded by in-memory map lookups rather than API server round-trips. The `managedLabel` predicate on the Secret reconciler (`inference.networking.k8s.io/bbr-managed=true`) narrows the watch scope to only relevant Secrets, avoiding unnecessary reconciliation of unrelated cluster Secrets.

The API translation layer supports six provider formats: OpenAI (passthrough with path rewrite), Anthropic (full body translation including tool calling), Azure OpenAI (path rewrite + response field stripping), Vertex AI native (full GenerateContent format translation), Vertex AI OpenAI-compatible (path rewrite with project/location/endpoint URL construction), and Bedrock OpenAI-compatible (path rewrite). The Anthropic translator is the most complex, handling system message extraction, tool_call/tool_use format conversion, and response content block merging. All translators strip the `authorization` header from translated requests since the apikey-injection plugin handles credential injection independently — this prevents the user's original auth token from leaking to the upstream provider.

The NeMo guardrails integration uses a synchronous HTTP callout pattern that blocks request processing until NeMo responds. The 360-second default timeout is high, reflecting that NeMo may need to run its own model inference. The plugin extracts only the last user message for the input rail check rather than sending the full conversation, which is a latency optimization. However, the HTTP client uses no TLS configuration, relying on the cluster network (or Istio mTLS) for transport security — this is acceptable for in-cluster NeMo deployments but would need enhancement for cross-cluster scenarios.

The Helm chart structure wraps the upstream `body-based-routing` chart as a dependency, adding only RBAC resources (ClusterRole/ClusterRoleBinding for cross-namespace Secret and ExternalModel access) and plugin configuration. The `multiNamespace: true` default is critical for the MaaS use case where ExternalModel CRs and their credential Secrets can exist in any user namespace. The EnvoyFilter is created by the upstream chart and uses `INSERT_AFTER` positioning relative to a Kuadrant WASM filter, indicating integration with Kuadrant for rate limiting in the inference gateway stack.
