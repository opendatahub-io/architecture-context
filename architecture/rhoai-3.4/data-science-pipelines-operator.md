## Purpose

**Short**: Manages the lifecycle of Data Science Pipelines Application (DSPA) custom resources and deploys the complete Kubeflow Pipelines v2 stack on OpenShift.

**Detailed**: The Data Science Pipelines Operator (DSPO) is a Kubernetes operator built with controller-runtime that watches `DataSciencePipelinesApplication` (DSPA) custom resources and reconciles a full ML pipeline execution environment per namespace. Each DSPA instance triggers the deployment of an API server, persistence agent, scheduled workflow controller, Argo Workflow controller, ML Metadata (MLMD) service with Envoy proxy, and optionally a MariaDB database and MinIO object store.

The operator uses the Manifestival library to render Go templates from `config/internal/` into Kubernetes manifests, applying them declaratively. This template-driven approach allows per-instance customization of images, resource limits, TLS settings, proxy configuration, and storage backends. The operator supports both embedded (MariaDB/MinIO) and external (any MySQL-compatible DB / S3-compatible storage) backends, with health checks that validate connectivity before marking the DSPA as ready.

DSPO also manages a global webhook deployment (in the operator namespace) that validates and mutates `PipelineVersion` CRs when the Kubernetes-native pipeline store is enabled. The operator exposes Prometheus metrics for each DSPA instance's component health and supports managed pipelines — pre-built pipeline definitions fetched from OCI container images and automatically loaded into the API server.

The Data Science Pipelines Operator follows a well-established pattern for Kubernetes operators — a single controller manager binary that watches a primary CRD and reconciles a constellation of sub-resources. What distinguishes DSPO is the breadth of infrastructure it deploys: each DSPA instance creates 7+ Deployments, multiple Services, NetworkPolicies, RBAC resources, Secrets, ConfigMaps, and optionally Routes. The Manifestival library is the architectural cornerstone — all child resources are rendered from Go templates in `config/internal/`, allowing the same operator binary to produce different manifests based on DSPA spec fields like `PodToPodTLS`, `EnableRoute`, and `ManagedPipelines`.

The operator implements a "batteries-included-but-swappable" storage model. Users can deploy with embedded MariaDB and MinIO for development, then switch to external databases and S3 storage for production — the transition requires only changing DSPA spec fields. Health checks validate connectivity to both storage backends before marking the DSPA as ready. The health check logic is notable for its TLS sophistication: it constructs custom `tls.Config` with system cert pools augmented by user-provided CA bundles, and supports MySQL TLS modes including `skip-verify` and `preferred` for gradual TLS adoption. The `InsecureSkipVerify` option in database.go is a controlled risk — it's only activated when users explicitly configure non-strict TLS modes via the DSPA CR.

The managed pipeline system is architecturally interesting: the operator fetches pipeline definitions from OCI container images, parses them from tar layers (with full OCI whiteout support), validates pipeline names against a manifest, and injects them into the API server via init containers. This creates a supply chain from pipeline authoring → OCI image build → operator deployment → API server availability. The `MANAGED_PIPELINES_ALLOWED_REGISTRIES` environment variable provides a security boundary, restricting which registries the operator will pull from.

Network security follows a defense-in-depth approach. The `kube-rbac-proxy` sidecar authenticates external traffic via SubjectAccessReview before proxying to the API server, while NetworkPolicies restrict internal traffic to only the expected pod selectors. Pod-to-pod TLS (enabled by default) ensures that even internal service-to-service communication is encrypted, using OpenShift's service-ca operator for automatic certificate provisioning. The Envoy proxy in front of MLMD adds gRPC-Web support and CORS headers for browser-based access, enabling the dashboard to query metadata directly.

The webhook subsystem has a unique lifecycle: it's deployed in the operator's namespace (not the DSPA namespace) and shared across all DSPAs that use the Kubernetes-native pipeline store. The operator tracks whether any DSPA across the cluster uses `PipelineStore: kubernetes` before deciding to clean up webhook resources. This cluster-wide coordination is handled through a simple list-and-filter pattern rather than a separate controller.
