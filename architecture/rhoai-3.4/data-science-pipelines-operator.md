## Purpose

**Short**: Manages the lifecycle of Data Science Pipelines (DSP) infrastructure by reconciling DataSciencePipelinesApplication (DSPA) custom resources into a full pipeline execution stack including API server, persistence agent, scheduled workflow controller, workflow controller, MLMD gRPC/Envoy, database, and object storage.

**Detailed**: The Data Science Pipelines Operator (DSPO) is a Kubernetes operator built with controller-runtime that watches DataSciencePipelinesApplication (DSPA) custom resources and deploys the complete Kubeflow Pipelines v2 stack per namespace. Each DSPA instance provisions up to eight sub-components: the KFP API server (with kube-rbac-proxy sidecar for authentication), a persistence agent, a scheduled workflow controller, an Argo Workflows controller, an MLMD gRPC metadata server, an MLMD Envoy proxy (with kube-rbac-proxy sidecar), a MariaDB database (or external DB connection), and Minio object storage (or external S3-compatible storage).

The operator uses the manifestival library to render Go templates (`config/internal/`) into Kubernetes resources at reconcile time, injecting owner references and dsp-version labels for garbage collection and cache scoping. It performs health checks against both the database (MySQL wire protocol) and object storage (S3 StatObject) before deploying downstream components. Network isolation is enforced through NetworkPolicy resources that restrict pod-to-pod communication to only the DSP component mesh. Pod-to-pod TLS is enabled by default using OpenShift service-ca signed certificates. The operator also manages a managed-pipelines feature that fetches pre-compiled pipeline definitions from OCI container images and injects them into the API server via an init container.

DSPO exposes Prometheus metrics for each sub-component's readiness state and publishes detailed status conditions on the DSPA CR. It supports FIPS-compliant builds via `GOEXPERIMENT=strictfipsruntime` and delegates authentication to kube-rbac-proxy sidecars that enforce Kubernetes RBAC SubjectAccessReviews against the DSPA resource.

## Architectural Analysis

The DSPO follows a single-controller architecture where one reconciler (`DSPAReconciler`) manages the entire DSP stack. Rather than using separate controllers per sub-component, it sequentially reconciles database, storage, common resources, webhook, managed pipelines validation, API server, persistence agent, scheduled workflow, workflow controller, and MLMD — with early exits and condition-specific status reporting at each stage. This design simplifies the deployment model (one operator pod per cluster) but creates long reconciliation cycles where a single failure blocks all downstream components.

The template-based deployment model using manifestival is distinctive. Instead of creating Kubernetes resources programmatically via Go structs, DSPO maintains 70+ YAML template files in `config/internal/` that are rendered with Go's `text/template` engine using the `DSPAParams` struct as context. This enables rich parameterization (conditional blocks for TLS, proxy config, managed pipelines) without code changes, but the templates are not validated at compile time and errors only surface at reconcile time.

The managed-pipelines feature demonstrates careful supply chain design. The operator validates pipeline names against a `managed-pipelines.json` manifest extracted from an OCI container image, with a registry allowlist, digest-based caching (10-minute TTL), OCI whiteout awareness, and distinction between permanent errors (invalid image ref, file not found) and transient errors (network issues). Permanent validation failures block API server deployment entirely, while transient errors allow deployment to proceed with a timed requeue for status self-healing.

Network security is enforced at multiple layers. NetworkPolicies restrict each component to only its required communication peers — the MariaDB NetworkPolicy only allows connections from the API server, MLMD gRPC, and the operator; the API server NetworkPolicy allows component pods, monitoring namespaces, pipeline task pods (`pipelines.kubeflow.org/v2_component: true`), and workbench pods (`opendatahub.io/workbenches: true`). Pod-to-pod TLS (enabled by default) uses OpenShift service-ca signed certificates, with the operator watching for service-ca-signed Secret changes to trigger reconciliation. External access is mediated through kube-rbac-proxy sidecars that enforce Kubernetes SubjectAccessReviews against the DSPA resource's `api` subresource, meaning users need explicit RBAC permissions on the `datasciencepipelinesapplications/api` resource to access the pipeline API.

The operator's cache configuration is carefully optimized to minimize memory usage. It strips managed fields from all cached objects, strips data payloads from ConfigMaps and Secrets (reading them via direct API calls instead), and scopes watches using label selectors (`dsp-version` label) and field selectors (webhook name). This is essential for operators managing many resources across multiple namespaces.
