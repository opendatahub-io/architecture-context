## Purpose

**Short**: ai4rag is an optimization engine for Retrieval-Augmented Generation (RAG) templates that automatically discovers optimal hyperparameter configurations for RAG pipelines.

**Detailed**: ai4rag (AI for RAG) is a Python library developed by IBM that provides automated hyperparameter optimization (HPO) for RAG pipelines. It accepts a variety of RAG template definitions and a search space of parameters, then uses Generalized Additive Models (GAM) or random search to find the optimal parameter values (called a "RAG Pattern") that maximize evaluation metrics such as answer correctness, faithfulness, and context correctness.

The library is provider-agnostic by design, with abstract base classes for foundation models, embedding models, vector stores, chunkers, and evaluators. Out of the box, it integrates with OGX (formerly Llama Stack) for foundation models, embeddings, and vector store operations, and with ChromaDB for in-memory vector storage. Evaluation is performed using the Unitxt framework, which provides standardized RAG evaluation metrics with confidence intervals.

ai4rag is used within the Red Hat OpenShift AI (RHOAI) platform as a library component — it does not run as a standalone service or operator. It is consumed programmatically by other components (e.g., Kubeflow Pipelines components) to run RAG optimization experiments. The library manages the full optimization lifecycle: model pre-selection, search space construction, chunking, embedding, retrieval, generation, evaluation, and result aggregation.

## Architectural Analysis

ai4rag follows a clean layered architecture with well-defined abstractions at each layer of the RAG pipeline. The library is organized into four major subsystems: the **experiment orchestrator** (`core/experiment/`), the **HPO engine** (`core/hpo/`), the **RAG pipeline** (`rag/`), and the **search space** management (`search_space/`). Each layer communicates through abstract base classes (`BaseFoundationModel`, `BaseEmbeddingModel`, `BaseVectorStore`, `BaseChunker`, `BaseEvaluator`, `BaseRAGTemplate`, `BaseOptimizer`, `BaseEventHandler`), making the system extensible and provider-agnostic.

The optimization strategy is notable: ai4rag uses a two-phase approach. First, the **Models Pre-Selector (MPS)** performs a fast screening of foundation model × embedding model pairs using a small sample of benchmark data and ChromaDB in-memory storage. This eliminates poorly performing model combinations before the expensive full optimization. Second, the **GAM Optimizer** uses Generalized Additive Models from the `pygam` library to predict which untested parameter combinations are likely to perform well, guided by the results of an initial random exploration phase (`n_random_nodes`). This is a Bayesian-style approach that reduces the number of expensive evaluations needed compared to pure random or grid search.

The library supports warm-starting via `known_observations`, allowing experiments to resume from previously computed results. This is particularly useful in the Kubeflow Pipelines context where experiments may be distributed across multiple pipeline steps. The `KFPEventHandler` aggregates all status changes and pattern results in memory for post-experiment consumption by KFP components, while `LocalEventHandler` writes results to disk as JSON files.

A design constraint worth noting is that all RAG queries during an experiment are executed concurrently using Python's `ThreadPoolExecutor` (default 10 threads), which is appropriate for I/O-bound OGX API calls but means the library relies on the OGX server's ability to handle concurrent requests. The search space constraint system is well-designed — it uses composable rule functions that prune invalid parameter combinations (e.g., chunk overlap must be less than half the chunk size, window retrieval requires non-zero window size, hybrid search requires a ranker strategy). The search space can contain both the default and OGX-specific hybrid search parameters (ranker_strategy, ranker_k, ranker_alpha), which are automatically excluded when using ChromaDB.

The library includes model-specific prompt templates for several foundation model families (Granite, Llama, Mistral, OpenAI-compatible) with appropriate system messages, user messages, and context templates. This attention to model-specific prompting is important for fair cross-model evaluation during optimization. The multilingual support instruction is always injected into user prompts, instructing models to respond in the question's language by default.
