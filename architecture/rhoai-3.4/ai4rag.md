## Purpose

**Short**: ai4rag is an optimization engine that automatically finds optimal hyperparameters for Retrieval-Augmented Generation (RAG) templates using Generalized Additive Models (GAM) and Bayesian optimization.

**Detailed**: ai4rag (AI for RAG) is a Python library developed by IBM that automates the hyperparameter optimization (HPO) of RAG pipelines. Given a set of documents, benchmark questions/answers, and a configurable search space of RAG parameters (chunking strategy, embedding model, foundation model, retrieval method, number of chunks, search mode, ranker strategy), ai4rag explores the parameter space to find the combination that maximizes a chosen evaluation metric (faithfulness, answer correctness, or context correctness).

The library is designed to be LLM and vector database provider-agnostic. Out of the box, it integrates with Llama Stack for foundation models, embedding models, and vector stores (Milvus, Qdrant), and also supports OpenAI-compatible APIs. For evaluation, it uses IBM's Unitxt framework to compute RAG quality metrics with confidence intervals. The optimization engine uses a GAM-based surrogate model (via `pygam`) to predict the performance of unexplored parameter combinations, significantly reducing the number of expensive LLM evaluations needed.

ai4rag is a pure Python library distributed as a pip package — it has no Dockerfiles, no Kubernetes manifests, and no deployment infrastructure of its own. It is intended to be consumed as a dependency by other RHOAI components or used directly in Kubeflow Pipelines (KFP) via its `KFPEventHandler`. The library communicates with external services (Llama Stack servers, OpenAI-compatible endpoints, vector databases) over HTTP/HTTPS at runtime.

## Architectural Analysis

ai4rag follows a clean, layered architecture with well-defined abstractions. At the core is the `AI4RAGExperiment` class, which orchestrates the entire optimization lifecycle: model pre-selection, hyperparameter search, RAG pattern evaluation, and results reporting. The design uses the Strategy pattern extensively — `BaseOptimizer`, `BaseRAGTemplate`, `BaseVectorStore`, `BaseEmbeddingModel`, `BaseFoundationModel`, and `BaseEvaluator` are all abstract base classes with concrete implementations, making the system genuinely provider-agnostic.

The optimization approach is notable for its efficiency. Rather than exhaustively evaluating every parameter combination (which would require expensive LLM calls), ai4rag uses a surrogate model approach: it evaluates a small random sample, trains a Generalized Additive Model (GAM) via `pygam`, then uses the GAM to predict which unexplored combinations are most promising. The `LabelEncoder` from scikit-learn encodes categorical parameters (model IDs, chunking methods) into numeric features for the GAM. This is a well-established Bayesian optimization pattern adapted for RAG parameter tuning. The `ModelsPreSelector` further reduces cost by filtering out poor-performing models on a small benchmark sample before the main HPO loop begins.

The search space management is sophisticated, with a constraint rule system that prunes invalid parameter combinations before they are evaluated. Rules enforce domain-specific constraints like "chunk_size must be > 2× chunk_overlap", "window_size must be 0 when retrieval_method is 'simple'", and "ranker parameters are only valid for hybrid search mode". This prevents wasted evaluations on configurations that would fail at runtime. The `AI4RAGSearchSpace` class also supports hybrid search parameters (ranker strategy, ranker_k, ranker_alpha) that are conditionally included based on the vector store type — ChromaDB does not support hybrid search, so those parameters are excluded.

A design risk worth noting is that the library uses `ThreadPoolExecutor` for parallel RAG query execution (in `query_rag`), which is appropriate for I/O-bound HTTP calls but could cause issues if the external services have rate limits. The default `max_threads=10` is hardcoded based on "client's inference endpoint settings" per the docstring, but there is no explicit rate limiting or backoff logic. The exception handling in `ExperimentExceptionHandler` is well-structured, aggregating errors across iterations and providing summary messages, but individual failed iterations silently return `None` as the score without any retry mechanism.

The event handler system (`BaseEventHandler` → `LocalEventHandler` / `KFPEventHandler`) provides a clean separation between the optimization engine and output handling. The `PatternPayload` TypedDict defines a versioned schema (`schema_version: "1.0"`) for pattern evaluation results, suggesting this interface is designed for consumption by downstream systems. The `KFPEventHandler` is particularly interesting — it accumulates results in memory for post-experiment processing, enabling integration with Kubeflow Pipeline components where results need to be passed as component outputs rather than written to files.
