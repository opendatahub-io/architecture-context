## Purpose

**Short**: A central metadata repository for ML model developers to store, manage, and track models, versions, artifacts, and serving infrastructure across the AI/ML lifecycle.

**Detailed**: Model Registry provides a REST API service that acts as the system of record for ML model metadata within the Red Hat OpenShift AI (RHOAI) platform. It stores and manages RegisteredModels, ModelVersions, ModelArtifacts, ServingEnvironments, InferenceServices, Experiments, and ExperimentRuns — enabling teams to track which models exist, what versions are available, where artifacts are stored, and how models are being served.

The component ships as multiple deployable artifacts: (1) the core REST API server backed by MySQL or PostgreSQL via an embedded metadata datastore (embedmd), (2) a Kubernetes controller that watches KServe InferenceService CRs and synchronizes their lifecycle with the Model Registry API, (3) a CSI storage initializer that resolves `model-registry://` URIs for KServe model loading, and (4) a Python-based async-upload Kubernetes Job that copies model artifacts between storage backends (S3, OCI, HuggingFace) and registers them in the Model Registry. The architecture follows a microservices pattern where the REST API is the central integration point, and the controller/CSI/job components interact with it over HTTP.

## Architectural Analysis

The Model Registry follows a hub-and-spoke architecture where the REST API server is the central hub and multiple satellite components (controller, CSI initializer, async-upload job) interact with it over HTTP. This design decouples the metadata management from the integration logic — the controller knows about Kubernetes but not about databases, while the API server knows about databases but not about Kubernetes resources. This separation of concerns enables independent scaling and deployment of each component.

The database layer uses an interesting "embedmd" abstraction that wraps the MLMD (ML Metadata) schema concepts — types, properties, contexts, artifacts, executions — into a GORM-based ORM layer with support for both MySQL and PostgreSQL. The type synchronization mechanism (`syncTypes` in `internal/datastore/embedmd/service.go`) ensures that all 13 MLMD type definitions (RegisteredModel, ModelVersion, ModelArtifact, etc.) exist in the database before serving traffic. The database connection includes a retry loop with 25 attempts over ~5 minutes, making the service resilient to database startup ordering in Kubernetes deployments.

The CSI storage initializer (`cmd/csi/`) implements the KServe `storage.Provider` interface, enabling users to reference models via `model-registry://` URIs in their InferenceService specs. The initializer resolves the URI by querying the Model Registry REST API for the registered model and version, extracting the actual storage URI from the model artifact, and then delegating to the appropriate KServe built-in provider (S3, GCS, etc.) for the actual download. This provides an abstraction layer where model deployment references are decoupled from storage locations — a model can be moved between storage backends without updating InferenceService specs.

The async-upload job is a notable addition to the architecture, implementing a multi-intent pattern where a single job image can perform three distinct operations: `update_artifact` (update an existing artifact's URI after upload), `create_model` (create a new registered model with version and artifact), and `create_version` (add a version to an existing model). Each intent follows a download→sign→upload→register pipeline, with fast-fail validation before expensive operations. The job supports Sigstore integration for model and OCI image signing, with auto-detection of signing enablement from environment variables.

A security concern worth noting is that the controller component (`pkg/inferenceservice-controller/controller.go`, line 245) uses `InsecureSkipVerify: true` when connecting to the Model Registry API. While the controller typically communicates within the cluster network, this bypasses TLS certificate validation entirely.
