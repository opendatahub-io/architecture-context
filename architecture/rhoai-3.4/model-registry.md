## Purpose

**Short**: A central REST API service for recording, retrieving, and managing ML model metadata including registered models, model versions, artifacts, serving environments, inference services, and experiments.

**Detailed**: Model Registry provides a metadata management layer for ML workflows in the Red Hat OpenShift AI (RHOAI) platform. It exposes an OpenAPI v3-based REST API (v1alpha3) allowing users and platform components to register ML models, track model versions and their associated artifacts, manage serving environments and inference services, and record experiment runs with metrics. The service persists metadata in either MySQL or PostgreSQL databases using an embedded metadata store (EmbedMD) backed by GORM.

The repository produces multiple deployable components: a Go-based REST API server (the primary service), a Kubernetes controller that synchronizes KServe InferenceService resources with Model Registry metadata, a CSI storage initializer for KServe that resolves `model-registry://` URIs to actual storage backends, a Python-based async upload job that copies models between storage backends (S3/OCI) and registers them, and a web UI with a Go BFF (Backend-for-Frontend) layer. These components integrate tightly with the broader RHOAI platform — the controller watches KServe CRDs, the CSI initializer plugs into KServe's storage initialization framework, and the UI discovers Model Registry instances via Kubernetes service discovery.

## Architectural Analysis

The Model Registry follows a multi-component architecture that separates concerns across five distinct deployable artifacts, each serving a different role in the ML model lifecycle. The primary service is a Go REST API server generated from an OpenAPI 3.0.3 specification (v1alpha3, with 91 operations). Code generation with openapi-generator produces both the server router stubs (`internal/server/openapi/`) and the Go client library (`pkg/openapi/`), ensuring consistency between the API contract and implementation. The service layer (`internal/core/`) implements business logic over a repository pattern (`internal/db/service/`), with GORM as the ORM abstracting MySQL and PostgreSQL backends. The database schema derives from MLMD (ML Metadata) concepts — registered models and serving environments map to MLMD Contexts, model versions and experiment runs to Executions, and artifacts to MLMD Artifacts. This heritage is visible in the generated schema files (`internal/db/schema/*.gen.go`) and the type naming conventions (`kf.RegisteredModel`, `kf.ModelVersion`, etc.).

The proxy server uses a notable dynamic router pattern (`internal/proxy/dynamic_router.go`) where the HTTP handler is swapped at runtime — initially returning HTTP 503 while the database connects, then atomically replacing the handler with the full API router once initialization completes. This provides graceful degradation during startup without requiring an external readiness gate. Health checks are comprehensive: `/readyz/isDirty` checks database schema migration state (detecting partial migrations that left the schema dirty), while `/readyz/health` performs end-to-end validation by querying every entity type with page-size-1 requests.

The TLS configuration layer (`internal/tls/config.go`) enforces a minimum of TLS 1.2 and validates cipher suites against a secure allowlist, rejecting deprecated ciphers. However, `InsecureSkipVerify` defaults to `true` (since `VerifyServerCert` defaults to `false`), and the InferenceService controller unconditionally sets `InsecureSkipVerify: true` on its HTTP client (`pkg/inferenceservice-controller/controller.go:86`). These defaults are typical for in-cluster communication where Istio mTLS provides transport security, but they represent a risk if the components are deployed without a service mesh.

The async-upload job is architecturally interesting as a Python batch component in an otherwise Go-dominated repository. It uses the model-registry Python SDK (the same `model_registry` package published for external users) to interact with the API, and integrates Sigstore for model and image signing. The job supports three intent types — `update_artifact`, `create_model`, and `create_version` — each with distinct validation and execution flows. Its Dockerfile.konflux is the only one that uses Hermeto (cachi2) prefetch for fully hermetic builds and includes an RPM lock file.

The kustomize manifest structure uses a well-organized base/overlay/options pattern. The base contains only the core API server resources, while database backends are isolated in overlays (MySQL vs PostgreSQL), and optional components (Istio, controller, CSI, UI, catalog) are separate kustomize components that can be composed independently. The controller's parameterization system uses `configMapGenerator` with `replacements` to inject label/annotation values into environment variables — a more maintainable pattern than direct environment variable patching. The Istio integration uses a DestinationRule for ISTIO_MUTUAL mTLS and VirtualService for gateway routing, but notably the AuthorizationPolicy has empty rules (allows all authenticated traffic), deferring fine-grained access control to the application or external policy.
