## Purpose

**Short**: A secure, batch-oriented runtime for executing language model evaluations using the lm-evaluation-harness framework, integrated with EvalHub for orchestrated benchmark execution.

**Detailed**: The lm-evaluation-harness component provides the container image (`odh-ta-lmes-job-rhel9`) used as the execution environment for language model evaluation jobs in Red Hat OpenShift AI. It is a downstream fork of the EleutherAI lm-evaluation-harness (v0.4.8) with a Red Hat-specific adapter layer (`main.py`) that integrates with EvalHub — the platform's evaluation orchestration service.

When a user submits a benchmark evaluation request through the RHOAI TrustyAI / EvalHub service, the platform spawns a Kubernetes Job using this container image. The job receives its configuration via a mounted JSON job specification (`/meta/job.json`), executes the specified benchmark against a model endpoint using the OpenAI-compatible completions API, and reports results back to EvalHub via callback URLs. Results can optionally be persisted as OCI artifacts to an OCI-compliant registry and logged to MLflow.

The component supports a wide range of evaluation benchmarks (200+ built-in tasks including MMLU, GSM8K, HellaSwag, ARC, GPQA, IFEval, and leaderboard suites), multiple model backends (HuggingFace Transformers, vLLM, OpenAI-compatible APIs, IBM watsonx.ai, Anthropic, SGLang), the Unitxt evaluation framework for custom tasks, concurrent inference requests, offline/disconnected evaluation mode, and multi-architecture builds (x86_64, s390x, ppc64le).

## Architectural Analysis

The lm-evaluation-harness in RHOAI follows a **batch job pattern** rather than a service pattern. The component is packaged as a container image that runs as a one-shot Kubernetes Job, not a long-running service. This design is well-suited for model evaluation workloads that are compute-intensive, potentially long-running, and inherently stateless between runs. The Job lifecycle is managed by the TrustyAI / LMEval operator, which creates Jobs based on `LMEvalJob` CRDs.

The RHOAI adapter layer (`main.py`) cleanly separates the upstream EleutherAI evaluation framework from the platform integration concerns. The adapter implements the `FrameworkAdapter` pattern from the `eval-hub-sdk`, handling job specification parsing, credential resolution, status reporting, result persistence, and error handling. The upstream `lm_eval` library is used as a library — its `simple_evaluate()` function is called with parameters derived from the EvalHub job spec. This separation means upstream updates can be merged with minimal conflict.

The **model backend architecture** is particularly notable. Rather than loading models locally (which would require GPU allocation for the job), the RHOAI adapter exclusively uses the `local-completions` backend, which sends HTTP requests to an external OpenAI-compatible API endpoint. This delegates GPU management to the model serving layer (KServe, vLLM, etc.) while keeping the evaluation job itself CPU-only. The adapter supports concurrent inference requests via the `num_concurrent` parameter (capped at 128) using `aiohttp` async HTTP, which significantly improves throughput for large benchmark suites.

The **offline mode** implementation (`config.parameters.offline`) is designed for air-gapped / disconnected OpenShift clusters. When enabled, the adapter sets `HF_DATASETS_OFFLINE=1`, `HF_HUB_OFFLINE=1`, `HF_EVALUATE_OFFLINE=1`, and `TRANSFORMERS_OFFLINE=1`, and redirects `HF_HOME` to `/test_data` — a directory populated by an init container that pre-downloads datasets and tokenizers from S3. The `__main__.py` file also patches the `evaluate.load()` function to resolve metrics from a local `metrics/` directory when `HF_EVALUATE_OFFLINE=1` is set, ensuring fully offline operation.

The **multi-architecture support** (x86_64, s390x, ppc64le) adds significant build complexity. The Konflux Dockerfile has a 5-stage build process with conditional compilation of PyArrow (from source on ppc64le and s390x) and PyTorch (from source on both non-x86 architectures). OpenBLAS is also compiled from source for ppc64le. This is necessary because pre-built wheels for these ML libraries are not available for these architectures. The build includes architecture-specific patches (e.g., `patches/s390x/parquet-support.patch` for Arrow on s390x).

The component ships 200+ evaluation tasks spanning major NLP benchmarks (MMLU, GSM8K, HellaSwag, ARC, GPQA, BBH, IFEval, HumanEval, TruthfulQA, Winogrande, and many more) organized into groups, subtasks, and tags via YAML configuration files. The task system supports custom task definitions via YAML, the Unitxt framework for structured data preparation, and external task paths. This comprehensive benchmark coverage is a key differentiator for the TrustyAI evaluation workflow.
