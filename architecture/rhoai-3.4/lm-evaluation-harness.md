## Purpose

**Short**: A batch-oriented execution environment for evaluating language models using the EleutherAI lm-evaluation-harness framework, integrated with the EvalHub platform.

**Detailed**: LM Evaluation Harness (LMES) provides a secure, containerized runtime for one-shot language model evaluations within Red Hat OpenShift AI. It is a downstream fork of the EleutherAI [lm-evaluation-harness](https://github.com/EleutherAI/lm-evaluation-harness) (v0.4.8), extended with an EvalHub adapter (`main.py`) that integrates the framework into RHOAI's evaluation pipeline.

The component is deployed as a Kubernetes Job (not a long-running service). When an evaluation is requested through EvalHub, the LMES job container is launched with a job specification (`/meta/job.json`) that defines the model endpoint, benchmark tasks, and evaluation parameters. The adapter configures the lm-evaluation-harness to use the `local-completions` backend (OpenAI-compatible API), runs the specified benchmarks against the model endpoint, collects metrics, and reports results back to EvalHub via callbacks. It also supports persisting evaluation artifacts as OCI images to a registry via the `olot` library and `skopeo`, and logging results to MLflow.

The harness ships with approximately 6,586 task definition YAML files covering a broad range of NLP benchmarks (MMLU, HellaSwag, ARC, GPQA, IFEval, etc.) plus integration with the Unitxt evaluation framework. It supports multiple model backends (HuggingFace, vLLM, OpenAI, Anthropic, IBM watsonx.ai, etc.) though the RHOAI adapter specifically uses the `local-completions` backend for inference against deployed model endpoints. The container image supports multi-architecture builds (x86_64, ARM64, ppc64le, s390x) with custom PyTorch and Apache Arrow builds for non-x86 platforms.

## Architectural Analysis

The LM Evaluation Harness is architecturally a batch job adapter that bridges the upstream EleutherAI evaluation framework with RHOAI's EvalHub platform. The `main.py` adapter implements the `FrameworkAdapter` interface from the `eval-hub-sdk`, receiving job specifications as JSON and translating them into `simple_evaluate()` calls on the upstream library. This design cleanly separates platform integration concerns (job lifecycle, callbacks, artifact persistence) from the evaluation logic itself.

The adapter exclusively uses the `local-completions` model backend, which speaks the OpenAI-compatible completions API. This is significant because it means the LMES job never loads model weights itself — it always delegates inference to an external serving endpoint (vLLM, TGI, etc.). The job container runs with `device=cpu` forced, as it only needs compute for tokenization, prompt construction, and metric calculation, not for model inference. The `num_concurrent` parameter (capped at 128) enables concurrent HTTP requests to the model endpoint, significantly improving throughput for large benchmark suites.

The offline mode (`parameters.offline=true`) is a notable architecture pattern for disconnected/air-gapped environments. When enabled, the adapter sets `HF_HUB_OFFLINE=1`, `HF_DATASETS_OFFLINE=1`, `HF_EVALUATE_OFFLINE=1`, and `TRANSFORMERS_OFFLINE=1`, then redirects `HF_HOME` to `/test_data` — a directory expected to be pre-populated by an init container that pulls data from S3 via `test_data_ref`. The offline metrics in `metrics/` directory (55+ pre-packaged metrics like accuracy, BLEU, F1) are also loaded from local paths when `HF_EVALUATE_OFFLINE=1`, with the `__main__.py` monkey-patching `evaluate.load` to resolve metrics from a local prefix directory.

The multi-architecture support is particularly complex. The Konflux Dockerfile uses a 5-stage build: (1) base builder with system dependencies, (2) Apache Arrow build from source for ppc64le/s390x (PyArrow wheels are unavailable for these architectures), (3) PyTorch build from source for ppc64le/s390x, (4) OpenBLAS build from source for ppc64le, and (5) final image assembly. The s390x Arrow build includes a custom patch (`patches/s390x/parquet-support.patch`) for parquet support. This multi-stage approach adds significant build complexity but enables RHOAI to support evaluations on IBM Power and Z platforms.

The TLS/certificate handling in `api_models.py` is security-conscious: custom CA bundles are validated to not be symlinks (preventing symlink attacks from Kubernetes secret volume mounts), resolved to absolute paths, and verified to be regular files before being passed to `ssl.create_default_context()`. This is a defense-in-depth measure specific to the Kubernetes deployment context where secrets are mounted as symlink trees.
