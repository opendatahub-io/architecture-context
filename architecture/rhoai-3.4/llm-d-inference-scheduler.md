## Purpose

**Short**: Optimized inference request scheduling and routing service for the llm-d inference framework, implementing KV-cache-aware endpoint selection and disaggregated prefill/decode orchestration via the Kubernetes Gateway API Inference Extension (GIE).

**Detailed**: The llm-d Inference Scheduler provides two core components: an Endpoint Picker Processor (EPP) and a P/D Routing Sidecar. The EPP extends the Gateway API Inference Extension project to make intelligent routing decisions for LLM inference requests arriving through a Kubernetes Gateway (Envoy). It uses a pluggable architecture of filters, scorers, and profile handlers to select optimal model-serving pods based on KV cache locality, load distribution, session affinity, context length ranges, and disaggregation requirements.

The P/D Routing Sidecar runs alongside each model-serving pod (vLLM) and orchestrates the physical data transfer for disaggregated inference. When the EPP decides to split a request across prefill and decode workers, the sidecar handles the multi-stage protocol: sending the prefill request, extracting KV transfer parameters, and forwarding the decode request with the cached context. It supports multiple KV cache transfer mechanisms including NIXL v2 (the default sequential protocol), shared storage, SGLang (concurrent prefill/decode), and an experimental Encoder-Prefiller-Decoder (EPD) protocol for multimodal workloads.

The scheduler integrates with the Kubernetes Gateway API and InferencePool CRD to discover available model-serving pods and uses Envoy's ext-proc (External Processing) filter as its callback mechanism for routing decisions. It is designed for the llm-d inference framework within RHOAI, building on the upstream GIE project with llm-d-specific extensions such as precise KV cache tracking via ZMQ events, prefix-based disaggregation decisions, and multi-stage encode/prefill/decode orchestration.

## Architectural Analysis

The llm-d Inference Scheduler demonstrates a sophisticated multi-layered scheduling architecture that extends the upstream Gateway API Inference Extension (GIE) project with llm-d-specific optimizations. The most notable design decision is the clean separation between the control plane (EPP) and the data plane orchestration (P/D sidecar). The EPP makes routing decisions purely through Envoy's ext-proc callback mechanism — it never touches inference request bodies. The sidecar handles the actual multi-stage protocol coordination. This separation allows the EPP to be stateless with respect to inference payloads while maintaining rich scheduling state through its plugin system.

The plugin architecture follows a Kubernetes scheduler-inspired design with scheduling profiles, filters, and scorers organized in weighted pipelines. Each InferencePool has a configurable set of plugins via EndpointPickerConfig YAML. The most architecturally interesting plugin is the `precise-prefix-cache-scorer`, which subscribes to real-time KV cache events from vLLM engines via ZMQ and maintains an in-memory index of cache block hashes across all pods. This enables cache-aware routing that significantly improves prompt reuse efficiency compared to the upstream GIE's estimation-based approach. The `speculativeIndexing` feature closes the blind spot between routing decisions and actual KV cache population by optimistically writing predicted cache entries.

The disaggregation subsystem supports four distinct KV transfer protocols, each with different latency/throughput tradeoffs. NIXL v2 (default) uses a sequential prefill-then-decode flow where KV transfer parameters are extracted from the prefiller response and injected into the decode request. Shared storage uses a decode-first optimization that checks cache hit thresholds before deciding whether prefill is needed, reducing latency for cache-hit cases. SGLang enables concurrent prefill and decode by sharing bootstrap coordination parameters, potentially reducing time-to-first-token. The EPD protocol adds an encode stage for multimodal workloads with fan-out distribution across encoder pods. All protocols share the same header-based routing interface (x-prefiller-host-port, x-encoder-hosts-ports), keeping the EPP agnostic to the specific transfer mechanism.

The SSRF protection mechanism in the sidecar is noteworthy — it watches InferencePool CRDs to dynamically build an allowlist of pod IPs and names, then validates that prefill/encoder targets specified in request headers are members of the pool. This prevents arbitrary internal network access through crafted headers. The implementation supports both the stable (`inference.networking.k8s.io`) and legacy (`inference.networking.x-k8s.io`) API groups, demonstrating the ongoing GIE API migration.

The downstream RHOAI fork replaces the upstream GIE dependency with `red-hat-data-services/gateway-api-inference-extension` (go.mod line 37), enabling RHOAI-specific patches. Both Konflux Dockerfiles use FIPS-compliant build flags (CGO_ENABLED=1 with strictfipsruntime), while the non-Konflux Dockerfiles use CGO_ENABLED=0 — confirming that the Konflux variants are the production RHOAI build path. The sidecar's TLS configuration explicitly selects FIPS-compatible cipher suites (ECDHE with AES-GCM and CHACHA20-POLY1305), though the InsecureSkipVerify option for upstream connections introduces a potential FIPS compliance gap when enabled.
