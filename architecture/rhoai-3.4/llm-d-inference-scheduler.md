## Purpose

**Short**: Optimized inference request scheduler and routing sidecar for the llm-d disaggregated LLM serving framework, built on the Kubernetes Gateway API Inference Extension.

**Detailed**: The llm-d inference scheduler provides two deployable components for routing LLM inference requests within the llm-d framework. The **Endpoint Picker (EPP)** is the scheduling control plane: it integrates with Envoy via the ext-proc gRPC protocol to make per-request routing decisions using a pluggable system of filters, scorers, and profile handlers. It selects optimal model-serving pods based on KV-cache locality, queue depth, session affinity, active request counts, and context length. The **Routing Sidecar (pd-sidecar)** runs alongside vLLM model-serving pods and orchestrates disaggregated inference execution, supporting Prefill/Decode (P/D), Encoder/Prefill/Decode (E/P/D), and Data Parallel modes with multiple KV transfer protocols (NIXL v2, Shared Storage, SGLang).

The project extends the upstream Kubernetes Gateway API Inference Extension (GIE) project with llm-d-specific features such as P/D disaggregation, precise KV-cache scoring with speculative indexing, multimodal encoder fanout, and context-length-aware routing. The EPP watches `InferencePool` custom resources to discover model-serving pods and receives scheduling configuration via YAML-based `EndpointPickerConfig`. The sidecar handles the runtime mechanics of multi-stage inference: cloning requests, injecting KV transfer parameters, managing concurrent prefill/decode stages, and providing SSRF protection against unauthorized endpoint targeting.

## Architectural Analysis

The llm-d inference scheduler exhibits a clean separation between the scheduling control plane (EPP) and the runtime data plane (sidecar). The EPP makes routing decisions without touching inference payloads — it only sees request metadata via Envoy's ext-proc protocol. The sidecar handles the actual request proxying and multi-stage orchestration. This separation allows the EPP to be deployed as a single shared service per InferencePool while sidecars scale per model-serving pod.

The plugin architecture is a key design strength. The EPP implements the Gateway API Inference Extension plugin framework with 43 registered plugins across 8 categories (filters, scorers, profile handlers, deciders, prepare data, pre-request, data layer, multi). Plugins communicate via three state-sharing mechanisms: CycleState (per-scheduling-cycle data like tokenized prompts), PluginState (per-request data persisting across plugin lifecycle phases), and endpoint attributes (persistent metadata on model-serving endpoints). The two-tier scoring system (Affinity vs Distribution) enables clear separation of concerns — affinity scorers (prefix cache, session) favor specific pods while distribution scorers (load-aware, active-request, no-hit-LRU) spread load evenly. The scoring tiers compose naturally through weighted summation.

The disaggregation architecture supports three modes with distinct KV transfer protocols. NIXL v2 (default for RHOAI) uses a sequential prefill-then-decode pattern where the prefiller returns `kv_transfer_params` in its response body, which the sidecar injects into the decode request. Shared Storage uses an optimistic decode-first approach with `cache_hit_threshold` to avoid unnecessary prefill when cache hits are sufficient. SGLang uses concurrent prefill and decode with a bootstrap coordination port. The E/P/D path extends P/D with a concurrent encoder fanout stage for multimodal content (images, audio, video). All protocols include graceful fallback to decoder-only mode on prefill failure — the sidecar returns to the decoder path on 4xx errors, ensuring request completion even when disaggregation fails.

The SSRF protection mechanism in the sidecar is notable. Since the sidecar receives prefill/encode target addresses from request headers (injected by the EPP's disagg-headers-handler), it validates these targets against an InferencePool-based allowlist. The allowlist watches InferencePool and Pod resources via Kubernetes informers, building a set of allowed pod IPs and names. This prevents a malicious or misconfigured EPP from directing the sidecar to arbitrary network endpoints. The allowlist rebuilds dynamically on pod add/update/delete events.

The speculative indexing feature in the precise-prefix-cache-scorer closes a timing gap between routing decisions and KV-cache event arrival. When a request is routed to a pod, the scorer immediately writes predicted cache entries to the KV-cache index with a configurable TTL (default 2s). This allows the next request with the same prefix to benefit from cache locality scoring before the engine's actual KV event confirms the cache population. The TTL ensures speculative entries are automatically cleaned up if the engine doesn't confirm them.
