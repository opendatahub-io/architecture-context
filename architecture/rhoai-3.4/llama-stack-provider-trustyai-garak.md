## Purpose

**Short**: Out-of-tree Llama Stack evaluation provider that wraps NVIDIA's Garak LLM vulnerability scanner for automated red-teaming of language models.

**Detailed**: This component implements the Llama Stack Eval API (`Api.eval`) as a pluggable provider, enabling automated LLM security assessment through Garak vulnerability scanning. It supports two execution modes: **inline** (running Garak as a local subprocess within the Llama Stack server process) and **remote** (delegating scan execution to Kubeflow Pipelines as a 6-step DAG). A third integration path provides an **eval-hub adapter** (`FrameworkAdapter`) for the EvalHub platform, supporting both simple (subprocess) and KFP execution within that ecosystem.

The provider manages the full scan lifecycle: registering benchmarks from predefined security profiles (OWASP LLM Top 10, AVID, CWE, quality, intents-based risk), building Garak configuration, executing scans, parsing multi-format results (JSONL reports, AVID vulnerability reports, hit logs), computing a Tier-Based Security Aggregate (TBSA) score, and returning structured `EvaluateResponse` objects via the Llama Stack API. It also integrates with Llama Stack's Safety API for shield-based scanning, where a `SimpleShieldOrchestrator` wraps LLM calls with input/output shield checks.

The component includes a Synthetic Data Generation (SDG) subsystem that uses the `sdg-hub` library to generate adversarial prompts from harm category taxonomies. Users can supply custom policy taxonomies, pre-generated intent prompts, or use the built-in 8-category base taxonomy. For Context Aware Scanning (CAS), the provider generates Garak topology files (trait typologies and intent stubs) that enable intent-based vulnerability probing.

## Architectural Analysis

### Dual Pipeline Architecture

The component maintains two parallel KFP pipeline implementations that share core logic but differ in their I/O layer:

1. **Llama Stack Pipeline** (`remote/kfp_utils/pipeline.py`): Uses Llama Stack Files API for all artifact transfer (taxonomy datasets, SDG results, prompts, scan outputs). The 6-step DAG passes KFP `Output[Dataset]`/`Input[Dataset]` artifacts between steps. The `garak_scan_pipeline` function defines the DAG with explicit `.after()` ordering and selective caching (taxonomy and SDG steps cached; validate, prepare, scan, parse steps not cached).

2. **EvalHub Pipeline** (`evalhub/kfp_pipeline.py`): Uses S3 Data Connections for artifact transfer, with `kubernetes.use_secret_as_env()` injecting S3 credentials and `use_secret_as_volume()` mounting model auth secrets. The pipeline structure mirrors the Llama Stack version but substitutes S3 upload/download for Files API calls.

Both pipelines share the core scan logic in `core/pipeline_steps.py` and `core/garak_runner.py`, which handle configuration building, Garak subprocess execution, timeout management, and result parsing.

### Deep-Merge Configuration Resolution

Benchmark registration uses a sophisticated configuration resolution strategy. The `GarakScanConfig` class defines `FRAMEWORK_PROFILES` — predefined scan profiles (OWASP LLM Top 10, AVID, CWE, quality, intents) that map to specific Garak probe/detector configurations. When a user registers a benchmark, the system deep-merges the predefined profile with user-supplied overrides via `base_eval.py:register_benchmark()`. This allows users to customize individual fields (e.g., override the detector for a specific probe) while inheriting all other profile defaults.

### Multi-Level API Key Resolution

The `resolve_api_key(role)` function in `core/pipeline_steps.py` implements a three-tier key resolution strategy for KFP pipeline components:
1. **Environment variable**: Check for role-specific env vars (e.g., `MODEL_API_KEY`, `SDG_API_KEY`)
2. **Mounted Secret**: Read from a well-known file path where K8s Secret volumes are mounted
3. **Fallback**: Return `"DUMMY"` to allow execution to proceed (Llama Stack models may handle their own auth at the server level)

This design accommodates diverse deployment scenarios — from fully authenticated production clusters to development environments without Secret infrastructure.

### Shield Orchestration Pattern

The `SimpleShieldOrchestrator` in `shield_scan.py` implements a middleware pattern for Garak's function-based generator mode. Instead of calling the LLM directly, Garak calls the orchestrator, which wraps each LLM interaction with Safety API shield checks:
- **Input shield**: Validates the adversarial prompt before sending to the LLM
- **LLM call**: Sends the prompt to the Llama Stack Inference API
- **Output shield**: Validates the LLM response for harmful content

This enables testing whether the platform's safety guardrails properly block adversarial inputs and filter harmful outputs. The orchestrator is instantiated as a module-level singleton (`simple_shield_orchestrator`) that Garak's generator calls as a Python function.

### Synthetic Data Generation Pipeline

The SDG subsystem provides three modes for generating adversarial test prompts:
1. **Default taxonomy + SDG**: Uses the built-in `BASE_TAXONOMY` (8 harm categories: Illegal Activity, Hate Speech, Security & Malware, Violence, Fraud, Sexually Explicit, Misinformation, Self Harm) with `sdg-hub` to generate synthetic adversarial prompts
2. **Custom taxonomy + SDG**: User uploads a policy CSV/JSON via Files API; SDG generates prompts from that custom taxonomy
3. **Bypass SDG**: User uploads pre-generated intent prompts directly via `intents_file_id`, skipping the SDG step entirely

The `generate_sdg_dataset()` function in `sdg.py` uses `sdg_hub.FlowRegistry` to load flow configurations and applies runtime overrides for `num_samples` and `max_tokens` via flow block patching.

### Context Aware Scanning (CAS) Integration

For intent-based scanning, the provider generates Garak topology files that enable Context Aware Scanning. The `generate_intents_from_dataset()` function in `intents.py` creates:
- `trait_typology.json`: Maps harm categories to trait hierarchies
- Individual intent stub JSON files: One per intent category, containing the adversarial prompts

These files configure Garak's `intents` probe to test model responses against specific harm categories with context-appropriate prompts.

### Compatibility Layer

The `compat.py` module provides a compatibility shim between `llama_stack_api` (newer package) and `llama_stack` (legacy package layout). It attempts to import from `llama_stack_api` first, falling back to `llama_stack` for older installations. It also patches the `Job` model to accept extra fields (specifically `metadata`) that aren't in the upstream schema, enabling the provider to attach scan metadata to job responses.

### Base Image Resolution for KFP

KFP pipeline components need a container base image with the provider pre-installed. The `get_base_image()` function in `remote/kfp_utils/components.py` implements a three-tier resolution:
1. **Environment variable**: `GARAK_PROVIDER_IMAGE`
2. **K8s ConfigMap**: Reads `garak-provider-image` key from `trustyai-service-operator-config` ConfigMap (searches `redhat-ods-applications` and `opendatahub` namespaces)
3. **Default**: `quay.io/trustyai/trustyai-garak-lls-provider-dsp:latest`
