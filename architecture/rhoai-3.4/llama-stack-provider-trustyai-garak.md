## Purpose

**Short**: An out-of-tree Llama Stack evaluation provider and Eval-Hub framework adapter that performs automated LLM red-teaming and vulnerability scanning using Garak.

**Detailed**: This component implements two completely independent integration surfaces for running Garak-based LLM security assessments within the RHOAI platform. The first surface is a Llama Stack provider that registers with the Llama Stack framework's evaluation API, supporting both inline (local subprocess) and remote (Kubeflow Pipelines) execution modes. The second surface is an Eval-Hub adapter (`FrameworkAdapter`) that integrates with the RHOAI evaluation platform, also supporting simple (in-pod) and KFP execution modes.

The provider runs predefined and custom vulnerability benchmarks (OWASP LLM Top 10, AVID taxonomy, CWE, quality, intents-based risk assessment) against target LLMs. It orchestrates Garak scans, parses results into structured metrics (including a Tier-Based Security Aggregate score), generates HTML reports with Vega visualizations, and uploads artifacts to the Llama Stack Files API or S3 storage. The "intents" benchmark is a key differentiator — it loads custom harm taxonomies, runs synthetic data generation (SDG) to produce adversarial prompts, and evaluates models with TAPIntent probes and MulticlassJudge detectors through a six-step KFP pipeline.

The component is packaged as a Python library published to PyPI and as a Konflux-built container image (`odh-trustyai-garak-lls-provider-dsp-rhel9`) used as the base image for KFP pipeline steps. It does not run as a standalone service — it is loaded into a Llama Stack distribution server or executed within Eval-Hub K8s jobs and KFP workflow pods.

## Architectural Analysis

This component has a notably sophisticated multi-surface architecture that serves two entirely independent orchestration platforms (Llama Stack and Eval-Hub) while sharing core scanning logic. The `core/` module provides framework-agnostic utilities — config resolution, command building, scan execution, pipeline steps, and result parsing — which both the Llama Stack providers (`inline/`, `remote/`) and the Eval-Hub adapters (`evalhub/`) consume. The Eval-Hub surface intentionally has zero imports from Llama Stack types, enforcing a clean dependency boundary. This design allows the same garak scanning capability to be deployed in either the Llama Stack ecosystem or the RHOAI evaluation platform without coupling the two frameworks.

The configuration system uses a multi-layered cascade: predefined scan profiles (OWASP LLM Top 10, AVID, CWE, quality, intents) provide baseline garak configurations, which are deep-merged with user-supplied overrides, then further enriched by legacy flat-key compatibility mappings. The intents benchmark adds a model configuration overlay that injects per-role API keys (judge, attacker, evaluator, SDG) using `__FROM_ENV__` placeholders resolved from Kubernetes Secrets at runtime. This placeholder-based secret injection is a strong security pattern — API keys never appear in pipeline parameters or config JSON, and the `redact_api_keys()` function sanitizes configs before upload. The pipeline steps also validate configs for dangerous flags (`--rm`, `--force`, `--no-limit`) before execution.

The KFP pipeline architecture differs significantly between Llama Stack and Eval-Hub modes despite executing the same six-step workflow (validate → taxonomy → SDG → prompts → scan → parse). The Llama Stack remote pipeline uses the Llama Stack Files API for inter-step artifact transfer, while the Eval-Hub KFP pipeline uses S3 Data Connections with explicit secret injection via `kfp-kubernetes` `use_secret_as_env` and `use_secret_as_volume`. Each KFP step resolves its container image dynamically: first checking a `GARAK_PROVIDER_IMAGE` environment variable, then a Kubernetes ConfigMap in candidate namespaces (`redhat-ods-applications`, `opendatahub`), finally falling back to `quay.io/trustyai/trustyai-garak-lls-provider-dsp:latest`. This allows the platform operator to control the image version centrally.

The Hermeto/Konflux build configuration is noteworthy for its workarounds. The repository contains a `Cargo.toml` and `Cargo.lock` at the root — not because the component has any Rust code — but because Hermeto fails to prefetch Rust dependencies of pip packages (e.g., `fancy-regex` used by `garak`) without a root Cargo project. The `.konflux/main.rs` is an empty `fn main() {}` placeholder. Additionally, `.konflux/prep-hermeto.sh` copies x86_64 requirements to ppc64le and s390x directories to work around a Hermeto bug where sdists aren't fetched when binary wheels aren't available for a given architecture. These workarounds are well-documented in `.konflux/README.md` and demonstrate the practical complexity of achieving hermetic multi-architecture builds for Python+Rust dependency chains.

The process management in `core/garak_runner.py` demonstrates careful subprocess handling: garak scans run in new process groups (via `os.setsid`), timeout enforcement uses a SIGTERM → SIGKILL escalation pattern, stdout/stderr are streamed via threads with a configurable tail buffer (default 500 lines) for memory management, and the environment is sanitized (including `PYTHONUNBUFFERED=1`). The inline provider adds concurrency control via `asyncio.Semaphore(max_concurrent_jobs)` to limit parallel scans. This is important because garak scans are CPU and memory intensive — each scan spawns probe/detector combinations that generate hundreds to thousands of prompts against the target model.
