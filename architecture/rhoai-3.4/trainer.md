## Purpose

**Short**: Kubeflow Trainer is a Kubernetes-native operator for orchestrating distributed machine learning training jobs across frameworks like PyTorch, JAX, TensorFlow, and MPI on OpenShift.

**Detailed**: Kubeflow Trainer v2 provides a declarative API for submitting and managing distributed ML training workloads on Kubernetes. Users create `TrainJob` custom resources that reference reusable `TrainingRuntime` or `ClusterTrainingRuntime` templates defining the training infrastructure (framework, node count, GPU configuration, container images). The operator translates these into `JobSet` resources (from the Kubernetes SIG `sigs.k8s.io/jobset`) which manage the underlying batch Jobs and Pods.

The RHOAI distribution adds several enhancements over upstream Kubeflow Trainer: RHAI (Red Hat AI) progression tracking that polls training pods for real-time metrics via HTTP, NetworkPolicy-based pod isolation for training workloads, FIPS-compliant builds via `GOEXPERIMENT=strictfipsruntime`, and a curated set of `ClusterTrainingRuntime` resources with pre-configured PyTorch images for CUDA, ROCm, and CPU accelerators. The operator integrates with JobSet for workload management, and optionally with Kueue (via `managedBy` field) for multi-cluster job scheduling, and with Volcano or Kubernetes scheduler-plugins for gang-scheduling via PodGroup resources.

## Architectural Analysis

The Kubeflow Trainer v2 architecture represents a significant redesign from the v1 Training Operator. Instead of having separate controllers for each ML framework (PyTorchJob, MPIJob, TFJob, etc.), v2 consolidates into a single `TrainJob` CRD with a pluggable runtime framework. The `RuntimeRef` field in TrainJob points to either a namespace-scoped `TrainingRuntime` or a cluster-scoped `ClusterTrainingRuntime`, which defines the JobSet template, ML policy (Torch, MPI), and gang-scheduling configuration. This plugin architecture is implemented through the `pkg/runtime/framework` package, where plugins like `JobSet`, `Torch`, `MPI`, `Coscheduling`, `Volcano`, and `PlainML` each implement specific interfaces (`EnforceMLPolicyPlugin`, `ComponentBuilderPlugin`, `WatchExtensionPlugin`, etc.) and are composed at runtime.

The RHOAI distribution adds a distinctive RHAI (Red Hat AI) layer that is not present in upstream Kubeflow Trainer. The progression tracking system (`pkg/rhai/`) enables real-time training progress monitoring by polling an HTTP `/metrics` endpoint on the primary training pod (rank-0). This endpoint is expected to return JSON with fields like `progressPercentage`, `currentStep`, `totalSteps`, `currentEpoch`, and training/eval metrics. The controller writes this data back to the TrainJob as annotations (`trainer.opendatahub.io/trainerStatus`), making training progress visible to the platform UI without requiring users to inspect pod logs. The system handles edge cases like NaN/Infinity values in metrics JSON (common from Python's json.dumps), pod termination message reading for final metrics capture, and configurable polling intervals (5s-300s, default 30s).

A notable security feature in the RHOAI distribution is per-TrainJob NetworkPolicy creation (`pkg/rhai/networkpolicy.go`). Each TrainJob gets a NetworkPolicy that restricts ingress to: (1) pods belonging to the same JobSet (for NCCL/MPI/gRPC inter-node communication), and (2) the controller pod in the opendatahub namespace (for metrics polling when progression tracking is enabled). This provides pod-level isolation between training workloads running in the same namespace.

The controller uses a finalizer pattern (`trainer.kubeflow.org/resource-in-use`) on TrainingRuntime and ClusterTrainingRuntime resources to prevent deletion while TrainJobs reference them. The TrainingRuntime and ClusterTrainingRuntime controllers watch for TrainJob create/update/delete events via channel-based notifications from the TrainJob controller, enabling immediate finalizer management without relying on label selectors or field indexers across different resource types.

The operator also supports sophisticated workload management features: suspend/resume via the `spec.suspend` field (which propagates to the underlying JobSet), immutable PodTemplateOverrides that can only be modified when a TrainJob is suspended and its JobSet has no active replicated jobs, and elastic PyTorch training via HPA-based auto-scaling of node count. The MPI runtime plugin generates SSH key pairs (ECDSA P-256) per TrainJob for passwordless SSH between launcher and worker pods, storing them in a generated Secret.
