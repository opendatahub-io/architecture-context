## Purpose

**Short**: Gateway API Inference Extension extends any Gateway API-compatible proxy into an inference gateway that optimizes routing and load balancing for self-hosted generative AI workloads on Kubernetes.

**Detailed**: The Gateway API Inference Extension (also known as Inference Gateway or IGW) leverages Envoy's External Processing (ext-proc) filter to intercept inference requests and make intelligent routing decisions. It consists of two primary gRPC servers — the Endpoint Picker Proxy (EPP) and the Body-Based Router (BBR) — that together transform a standard gateway into an inference-aware load balancer.

The EPP watches Kubernetes CRDs (InferencePool, InferenceObjective, InferenceModelRewrite) and Pod resources to maintain an in-memory view of available model-serving endpoints. When an inference request arrives via Envoy's ext-proc protocol, the EPP applies a configurable scheduling pipeline (filters, scorers, pickers) that considers KV-cache utilization, queue depth, LoRA adapter affinity, and optionally ML-predicted latency to select the optimal backend pod. It supports flow control with priority-based queuing, fairness policies, and saturation detection to maintain SLO compliance under load.

The BBR is an optional companion ext-proc server that parses HTTP request bodies to extract the model name from OpenAI-compatible payloads and sets routing headers, enabling the gateway to route requests before they reach the EPP. Together these components enable Kubernetes-native declarative management of inference workloads with A/B traffic splitting, LoRA adapter rollout, and multi-workload isolation on shared GPU pools.

## Architectural Analysis

The Gateway API Inference Extension follows a **plugin-oriented architecture** where virtually every decision point in the request processing pipeline is extensible. The scheduling system uses a profile-based approach: each `SchedulingProfile` chains together Filter, Scorer, and Picker plugins, and the EPP can evaluate multiple profiles iteratively before selecting a final endpoint. This design mirrors the Kubernetes scheduler's extension point model and enables use-case-specific routing strategies without modifying core code.

The **flow control subsystem** implements a sophisticated sharded queuing architecture. The `FlowController` distributes incoming requests across shards using a Join-Shortest-Queue-by-Bytes (JSQ-Bytes) algorithm, with each shard running an independent worker goroutine. Within each shard, requests are organized into flows (identified by `x-gateway-fairness-id` header) and priority bands. The system supports configurable fairness policies (round-robin across flows or global-strict best-head) and ordering policies (FCFS, Earliest Deadline First, SLO-deadline-based). The queue implementation offers both a simple `ListQueue` (FIFO, O(1) enqueue) and a `MaxMinHeap` (priority-configurable, O(log n)) depending on whether the ordering policy requires priority access.

The **dual ext-proc architecture** (BBR + EPP) is a deliberate design choice that separates concerns: the BBR handles body parsing early in the Envoy filter chain (before routing decisions), while the EPP handles endpoint selection later. This separation allows the gateway to make coarse routing decisions (which InferencePool to target) based on model name before the EPP performs fine-grained endpoint selection within the pool. The BBR maintains an in-memory LoRA-to-base-model mapping by reconciling ConfigMaps, enabling model name resolution without querying external services.

The **saturation detection** uses two complementary strategies: a concurrency-based detector that tracks in-flight requests per endpoint (incremented on PreRequest, decremented on ResponseComplete), and a utilization-based detector that applies a roofline model combining KV-cache utilization and queue depth metrics scraped from model servers. The concurrency detector provides real-time signal while the utilization detector provides infrastructure-level signal — together they enable both reactive (reject requests immediately) and proactive (avoid scheduling to near-capacity endpoints) load management.

A notable design tension exists around **TLS and FIPS compliance**. The upstream Dockerfile uses `CGO_ENABLED=0` with a distroless base image, which produces a statically-linked binary with Go's pure-Go crypto — not FIPS-compliant. The TLS configuration uses default Go cipher suites with no explicit restrictions, and the `InsecureSkipVerify` option for model server metrics scraping is configurable but defaults to false. For RHOAI deployment, the Konflux build pipeline is expected to replace the Dockerfile with a FIPS-compliant variant using UBI base images and `GOEXPERIMENT=strictfipsruntime`, but no such Dockerfile was found on this branch.
