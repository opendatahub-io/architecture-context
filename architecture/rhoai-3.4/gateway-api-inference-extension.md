## Purpose

**Short**: Extends any Gateway API-compatible proxy (Envoy) into an inference gateway that provides intelligent, KV-cache-aware routing and load balancing for self-hosted generative AI model servers on Kubernetes.

**Detailed**: Gateway API Inference Extension (also known as Inference Gateway or IGW) is a Kubernetes SIG project that optimizes serving of generative AI models on Kubernetes clusters. It leverages Envoy's External Processing (ext-proc) filter to intercept inference requests at the gateway layer and make intelligent routing decisions based on real-time model server metrics such as KV-cache utilization, request queue depth, LoRA adapter availability, and prefix cache state.

The project consists of two primary gRPC ext-proc servers: the Endpoint Picker Provider (EPP), which implements the core scheduling and routing logic through a sophisticated plugin framework with filter/score/pick phases, and the Body Based Router (BBR), which parses HTTP request bodies to extract model names and inject routing headers. Together they enable features like weighted model rewriting (A/B testing, canary rollouts), workload priority and fairness via flow control, latency-aware scheduling with ML-based TTFT/TPOT prediction, and prefix-cache-aware routing.

The extension defines several Custom Resource Definitions (InferencePool, InferenceObjective, InferenceModelRewrite, InferencePoolImport) that provide Kubernetes-native declarative APIs for configuring inference workload routing, SLO objectives, and multi-cluster federation. Within RHOAI, this component provides the intelligent inference routing layer that sits between the platform's Gateway API ingress and backend model serving infrastructure (vLLM, SGLang).

## Architectural Analysis

The Gateway API Inference Extension employs a sophisticated multi-layered plugin architecture that mirrors the design philosophy of the Kubernetes scheduler. The core EPP server implements a three-phase scheduling pipeline (Filter -> Score -> Pick) where each phase is composed of pluggable components. Scorers produce normalized [0,1] scores with configurable weights, and the system supports multiple scheduling profiles that can be selected based on request characteristics. Default plugins include KV-cache utilization scoring (weight 2.0), queue depth scoring (weight 2.0), and prefix cache affinity scoring (weight 3.0), with the prefix cache scorer using an xxhash-based LRU indexer that tracks per-pod block hashes for longest-prefix matching.

The flow control system implements a sharded, lock-free request queuing architecture with Join-Shortest-Queue-by-Bytes (JSQ-Bytes) distribution strategy. It provides per-priority-band fairness enforcement with pluggable ordering policies (FCFS, EDF/earliest-deadline-first, SLO-deadline-based) and fairness policies (best-head, round-robin). The sharded design enables horizontal scalability with JIT flow provisioning and lease-based lifecycle management, including automatic garbage collection of idle flows (5-minute timeout). This represents a significant engineering investment in supporting multi-tenant inference workloads with differentiated SLOs.

The data layer architecture introduces a pluggable data source and extractor framework that separates data collection (polling HTTP endpoints, watching Kubernetes events) from data transformation (extracting structured metrics from raw Prometheus output). A dependency graph validates that producer plugins run before consumer plugins across three execution layers (FlowControl -> RequestControl -> Scheduling), using topological sort via Kahn's algorithm to detect cycles. This enables third-party plugins to declare data dependencies and have them automatically ordered.

The separation between EPP and BBR reflects an important architectural boundary: BBR handles protocol-level concerns (parsing OpenAI API request bodies to extract model names) while EPP handles scheduling-level concerns (choosing optimal endpoints based on real-time metrics). This separation allows BBR to be deployed independently or omitted entirely when model names are already available in request headers. The BBR plugin framework is intentionally simpler than EPP's, focused on payload processing and guardrail enforcement rather than complex scheduling logic.

A notable security consideration is that the upstream Dockerfiles use `CGO_ENABLED=0` with distroless base images, which provides a minimal attack surface but is incompatible with FIPS requirements. The model server metrics scraping defaults to `InsecureSkipVerify=true` for HTTPS connections, prioritizing operational simplicity over strict certificate validation. For RHOAI deployment, downstream Konflux Dockerfiles would need to address both concerns with UBI bases, dynamic linking, and proper TLS configuration.
