## Purpose

**Short**: Implements Ragas evaluation framework as an out-of-tree provider plugin for Meta's Llama Stack eval API, enabling LLM evaluation scoring (faithfulness, context precision, answer relevancy, context recall) either inline or remotely via Kubeflow Pipelines.

**Detailed**: This component bridges the Ragas evaluation framework into the Llama Stack ecosystem by implementing the Llama Stack `Eval` and `BenchmarksProtocolPrivate` APIs. It provides two execution modes: an **inline provider** that runs Ragas evaluations in the same process as the Llama Stack server, and a **remote provider** that submits evaluation jobs to Kubeflow Pipelines (Data Science Pipelines) for distributed execution on OpenShift.

The provider wraps Llama Stack's inference and embedding APIs to make them compatible with Ragas' expected LangChain-style `BaseRagasLLM` and `BaseRagasEmbeddings` interfaces. For remote execution, evaluation datasets are retrieved from Llama Stack, serialized as JSONL, and passed through a two-step KFP pipeline: data retrieval followed by Ragas evaluation. Results are persisted to S3 and fetched back by the provider when job status is queried.

In the RHOAI context, this component is deployed as part of the TrustyAI stack. The container image (`odh-trustyai-ragas-lls-provider-dsp-rhel9`) is used as the base image for Kubeflow Pipeline component steps, while the Python package itself is installed into a Llama Stack distribution server. The component integrates with the Llama Stack Operator for deployment on OpenShift and reads its base image configuration from the `trustyai-service-operator-config` ConfigMap.

## Architectural Analysis

This component follows a clean **provider plugin architecture** that separates the evaluation interface (Llama Stack's `Eval` API) from the execution engine (Ragas). The inline/remote split is a classic strategy pattern: both `RagasEvaluatorInline` and `RagasEvaluatorRemote` implement the same `Eval` + `BenchmarksProtocolPrivate` interfaces, and the `provider.py` module dynamically selects which providers to expose based on installed dependencies (`_has_remote_dependencies()`). This design means the base package is lightweight and the heavier KFP/Kubernetes/S3 dependencies are only pulled in when needed.

The **wrapper pattern** for bridging Llama Stack APIs to Ragas is notably duplicated between inline and remote modes. `wrappers_inline.py` wraps Llama Stack's in-process inference API using `OpenAICompletionRequestWithExtraBody`, while `wrappers_remote.py` wraps the `LlamaStackClient`/`AsyncLlamaStackClient` HTTP clients. Both produce LangChain-compatible `BaseRagasLLM` and `BaseRagasEmbeddings` implementations. The duplication exists because the inline provider runs within the Llama Stack server process (using internal API objects), while the remote provider's KFP pipeline steps run in separate pods that must use the HTTP client to reach the Llama Stack server.

The **compatibility layer** (`compat.py`) is architecturally significant — it maintains backward compatibility across Llama Stack API reorganization (legacy `llama_stack.apis.*` layout vs newer `llama_stack_api` flat package). Every import is wrapped in try/except, making the provider resilient to upstream API package restructuring. This is a smart defensive pattern for an out-of-tree provider that must track a rapidly evolving upstream.

The **Kubeflow Pipelines integration** uses a two-step pipeline: data retrieval from Llama Stack followed by Ragas evaluation. The `get_base_image()` function in `components.py` implements a three-tier resolution strategy (environment variable → Kubernetes ConfigMap → hardcoded default) for determining which container image runs the pipeline steps. This ConfigMap-based image resolution creates a coupling with the TrustyAI Service Operator, which is expected to manage the `trustyai-service-operator-config` ConfigMap. The candidate namespace list (`redhat-ods-applications`, `opendatahub`) ties the component to RHOAI/ODH platform deployments.

A notable **security consideration**: the KFP pipeline steps communicate with the Llama Stack server over plaintext HTTP (no TLS), and the demo deployment template includes `VLLM_TLS_VERIFY="false"`. In production, the Llama Stack server URL used by KFP pods (`KUBEFLOW_LLAMA_STACK_URL`) must be an external route (as noted in the demo README), but TLS verification and authentication between KFP pods and the Llama Stack server are not enforced by this component. The `pipelines_api_token` is properly handled as a Pydantic `SecretStr` to prevent log leakage, which is a good security practice.
