## Purpose

**Short**: Implements Ragas as an out-of-tree evaluation provider for the Llama Stack platform, enabling LLM evaluation via both inline (in-process) and remote (Kubeflow Pipelines) execution modes.

**Detailed**: The llama-stack-provider-ragas component bridges the [Ragas](https://github.com/explodinggradients/ragas) evaluation framework with Meta's [Llama Stack](https://github.com/meta-llama/llama-stack) platform. It implements the Llama Stack `Eval` API to provide standardized LLM evaluation capabilities using Ragas metrics such as answer relevancy, context precision, faithfulness, and context recall.

The provider ships as a Python package (`llama-stack-provider-ragas`) that plugs into a Llama Stack distribution server. It operates in two modes: **inline** mode runs Ragas evaluation in the same process as the Llama Stack server, wrapping the Llama Stack inference API (for LLM completions and embeddings) into Ragas-compatible adapters. **Remote** mode offloads evaluation workloads to Kubeflow Pipelines, submitting evaluation jobs that run as pipeline steps on a Kubernetes cluster. The remote mode stores results in S3-compatible storage and retrieves them asynchronously.

In the RHOAI platform context, this component is deployed as part of a Llama Stack distribution managed by the Llama Stack Operator. The container image is built from a UBI9 Python 3.12 base via Konflux. The provider is configured through the distribution's `run.yaml` and Kubernetes ConfigMaps/Secrets, integrating with Kubeflow Pipelines (Data Science Pipelines), vLLM inference servers, and S3 storage.

## Architectural Analysis

The llama-stack-provider-ragas component follows a **plugin architecture pattern**, implementing Meta's Llama Stack provider interface to bridge the Ragas evaluation framework with the Llama Stack Eval API. The dual-mode design (inline vs. remote) is a key architectural decision that enables both development convenience and production scalability. The inline provider runs evaluation synchronously in-process, while the remote provider delegates to Kubeflow Pipelines for distributed execution. This split is cleanly encapsulated: both modes implement the same `Eval` and `BenchmarksProtocolPrivate` interfaces but with fundamentally different execution strategies.

The **compatibility layer** (`compat.py`) is architecturally notable — it provides import-time abstraction over two generations of the Llama Stack package structure (`llama_stack` legacy vs `llama_stack_api` newer). This allows the provider to work with multiple Llama Stack versions without code duplication, using a try/except import pattern. This is pragmatic but creates a maintenance surface: every new type imported from Llama Stack requires a dual-path import block.

The **wrapper pattern** for LLM and embeddings adapters is central to the integration strategy. Both `LlamaStackInlineLLM`/`LlamaStackInlineEmbeddings` (inline) and `LlamaStackRemoteLLM`/`LlamaStackRemoteEmbeddings` (remote) extend Ragas base classes (`BaseRagasLLM`, `BaseRagasEmbeddings`) and translate between Llama Stack's API conventions and Ragas' LangChain-derived interfaces. The inline wrappers use the in-process inference API directly, while the remote wrappers use the `llama_stack_client` HTTP SDK. Both enforce `max_workers=1` in Ragas `RunConfig`, which serializes metric computation — this is a deliberate concurrency constraint, likely to avoid overwhelming inference backends.

The **Kubeflow Pipelines integration** uses a two-step pipeline: `retrieve_data_from_llama_stack` fetches the evaluation dataset from the Llama Stack server, and `run_ragas_evaluation` executes the Ragas evaluation and writes results to S3. The base image for these pipeline components is resolved through a three-tier fallback: environment variable → Kubernetes ConfigMap (`trustyai-service-operator-config`) → hardcoded default. This design couples the provider to the TrustyAI operator's configuration infrastructure. The KFP client initialization includes a manual health check before constructing the `kfp.Client`, working around a known issue where the KFP SDK handles unhealthy endpoints poorly.

A notable **security consideration** is that all Secrets referenced by the component (KFP tokens, AWS credentials, inference model tokens) are user-provisioned with no auto-rotation. The `KUBEFLOW_PIPELINES_TOKEN` is particularly fragile — it's an OpenShift user token (from `oc whoami -t`) with a limited TTL, which means remote evaluation jobs will fail after token expiration. The `VLLM_TLS_VERIFY=false` option in the demo deployment disables TLS verification for inference endpoints, which is documented as non-production but presents a risk if carried into production configurations.
