## Purpose

**Short**: An OpenAI-compatible HTTP gateway that routes chat completion requests through configurable detector-based content filtering pipelines via the FMS Guardrails Orchestrator.

**Detailed**: The vLLM Orchestrator Gateway serves as a routing layer between clients and the FMS Guardrails Orchestrator, providing multiple OpenAI-compatible `/v1/chat/completions` endpoints — one per configured detector pipeline. Each route enforces a specific set of content detectors (e.g., PII detection via regex patterns for email addresses, SSNs) on input and/or output, enabling drop-in replacement of unguarded chat completion endpoints with guardrailed ones.

The gateway reads a YAML configuration file at startup that defines the upstream orchestrator address, available detector definitions with their parameters, and named routes that map to specific detector combinations. When a request arrives at a route endpoint (e.g., `/{route_name}/v1/chat/completions`), the gateway injects the route's detector specifications into the payload and forwards it to the orchestrator's `/api/v2/chat/completions-detection` endpoint. If any detection is found in the response, a configurable fallback message replaces the generation output. The gateway supports both streaming (SSE) and non-streaming chat completions.

The service is built in Rust using Axum for HTTP handling, Reqwest for upstream HTTP calls, and Tokio as the async runtime. It supports mTLS communication with the orchestrator using OpenSSL-backed TLS with certificate/key pairs mounted at well-known paths (`/etc/tls/private/`). It also supports custom CA certificates for trust chain validation.

## Architectural Analysis

The vLLM Orchestrator Gateway is a lightweight, single-binary Rust service with a focused architectural role: it translates the OpenAI chat completions API into the FMS Guardrails Orchestrator's detection-enriched API. The architecture is intentionally simple — three source files (`main.rs`, `api.rs`, `config.rs`) totaling approximately 750 lines of Rust. This simplicity reflects its narrow responsibility: route multiplexing, detector injection, and fallback message substitution.

The gateway's route generation pattern is notable: routes are created dynamically at startup from the YAML config, with each route binding a unique URL path to a fixed set of detectors. This means the number of exposed HTTP endpoints is determined entirely by configuration, not code. A route like `pii` with detectors `[regex-language]` produces `POST /pii/v1/chat/completions`. A route with no detectors (`passthrough`) acts as a transparent proxy to the orchestrator. This pattern enables operators to define multiple content filtering policies (strict PII filtering, permissive passthrough, profanity filtering) as distinct endpoints that can be individually referenced by inference services.

The mTLS implementation uses an unconventional approach: it constructs a PKCS#12 identity at runtime from PEM files using the `openssl` crate, then passes it to `reqwest`'s `native-tls` backend. This is necessary because `reqwest`'s `Identity::from_pem` requires the `rustls-tls` feature, while this service uses `native-tls` (which only accepts PKCS#12/DER). The code auto-detects whether to use HTTP or HTTPS based on the presence of TLS certificate files at well-known paths, falling back gracefully to plaintext when no certs are mounted. The `danger_accept_invalid_hostnames(true)` workaround for localhost is a development convenience that does not apply in production where the orchestrator hostname is a Kubernetes service DNS name.

One architectural consideration is the absence of health/readiness endpoints. The service has no `/healthz` or `/readyz` endpoint, which means Kubernetes liveness/readiness probes must use TCP socket checks on port 8090 rather than HTTP probes. Additionally, there is no metrics endpoint (no Prometheus exposition), which limits observability in production deployments. The tracing layer provides structured DEBUG-level logging but not metrics export.

The streaming implementation deserves attention: the gateway parses SSE chunks from the orchestrator, deserializes each `data:` line into a `StreamingResponse`, checks for detections, and re-serializes with potential fallback substitution. This per-chunk JSON parsing adds latency but is necessary to intercept detection results within the stream. Empty chunks (non-`data:` lines like `[DONE]`) are filtered out during parsing.
