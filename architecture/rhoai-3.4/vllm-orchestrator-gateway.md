## Purpose

**Short**: An HTTP gateway that provides multiple OpenAI-compatible chat completion endpoints with configurable detector-based content filtering, routing requests to the FMS Guardrails Orchestrator.

**Detailed**: The vllm-orchestrator-gateway is a Rust-based HTTP service that acts as a routing layer between API clients and the FMS Guardrails Orchestrator. It enables controlled interaction with the orchestrator by enforcing stricter access to its exposed endpoints. The gateway reads a YAML configuration file that defines detector pipelines and routes, then dynamically registers HTTP endpoints of the form `/{route-name}/v1/chat/completions` for each configured route.

Each route maps to a specific set of content detectors (e.g., PII detection via regex patterns for email addresses and SSNs). When a client sends a standard OpenAI-compatible chat completion request to a route endpoint, the gateway injects the route's detector configuration into the request payload and forwards it to the orchestrator's `/api/v2/chat/completions-detection` endpoint. If the orchestrator returns detections, the gateway can replace the response with a configurable fallback message, providing route-specific content filtering policies.

The gateway supports both streaming (SSE) and non-streaming chat completion responses, and can communicate with the orchestrator over mTLS when TLS certificates are mounted. It is part of the TrustyAI ecosystem within RHOAI, enabling guardrailed LLM inference as a drop-in replacement for unguardrailed chat completion endpoints.

## Architectural Analysis

The vllm-orchestrator-gateway follows a straightforward reverse proxy pattern specialized for LLM guardrailing. It sits between API clients and the FMS Guardrails Orchestrator, providing a simplified OpenAI-compatible API surface while injecting per-route detector configurations. The design is intentionally minimal — the gateway handles only routing and detection response policy, delegating all LLM inference and content detection logic to downstream services.

The configuration-driven route registration model (`config.yaml` → dynamic axum routes) is a key architectural choice. Each route gets its own `/{name}/v1/chat/completions` endpoint, allowing operators to expose multiple guardrail profiles (e.g., PII filtering, toxicity detection, passthrough) as separate endpoints. This makes the gateway a drop-in replacement for unguardrailed chat completion endpoints — clients only need to change the base URL to enable guardrailing. The validation logic in `config.rs` (lines 64-108) ensures that routes only reference defined detectors and that no route has duplicate input/output detectors for the same server, catching misconfigurations at startup.

The mTLS implementation (src/main.rs:351-406) is notable for using the `openssl` crate to create PKCS12 archives in memory from PEM-encoded certificates and keys, then loading them as `native-tls` identities. This approach bridges the OpenSSL and native-tls ecosystems required by the `reqwest` HTTP client. The TLS configuration is entirely optional — the gateway dynamically detects whether TLS certificates are mounted and falls back to plain HTTP if they are absent. The `danger_accept_invalid_hostnames(true)` call on localhost connections is a development convenience that should not affect production deployments where the orchestrator hostname matches its certificate.

The streaming implementation (src/main.rs:259-349) parses SSE chunks from the orchestrator, deserializes each as JSON to check for detections, and applies fallback messages on the first detection-bearing chunk. This means detection enforcement in streaming mode is chunk-granular — a partial response may be streamed before a detection is found in a later chunk. This is an inherent tradeoff of streaming detection. The gateway uses `KeepAlive::default()` for SSE connections.

The gateway has no authentication enforcement of its own — it relies entirely on header forwarding (`Authorization` and `x-forwarded-*` headers are passed through to the orchestrator). This means the gateway itself is unauthenticated and must be protected by upstream infrastructure (e.g., kube-rbac-proxy sidecar, network policies, or service mesh) in production deployments.
