## Purpose

**Short**: Kubernetes operator that manages TrustyAI AI explainability services, language model evaluation jobs, guardrails orchestration, and NVIDIA NeMo Guardrails deployments on OpenShift.

**Detailed**: The trustyai-service-operator is a multi-controller Kubernetes operator that provides the AI trustworthiness and safety layer for Red Hat OpenShift AI. It manages five distinct service controllers within a single operator binary, each handling a different aspect of AI/ML trustworthiness:

1. **TrustyAI Service (TAS)** - Deploys and manages the TrustyAI explainability service, which monitors ML model predictions via KServe/ModelMesh integration, providing bias detection, fairness metrics, and explainability. It integrates deeply with the KServe InferenceService ecosystem by patching InferenceServices to route prediction payloads to TrustyAI for analysis.

2. **LMEval (LMES)** - Manages language model evaluation jobs using the EleutherAI lm-evaluation-harness framework. It creates ephemeral pods with a driver sidecar architecture for running evaluation benchmarks against LLM endpoints, with support for offline execution, S3 storage, OCI artifact upload, and Kueue-based job scheduling.

3. **GuardrailsOrchestrator (GORCH)** - Deploys the FMS Guardrails Orchestrator, which routes LLM requests through configurable content safety detectors. Features automatic service discovery of KServe InferenceServices to dynamically configure generation endpoints and detector services, with an optional sidecar gateway.

4. **NemoGuardrails** - Manages NVIDIA NeMo Guardrails server deployments for programmable guardrail configurations, with comprehensive CA bundle aggregation from multiple trust sources (ODH, OpenShift, user-provided).

5. **EvalHub** - Deploys a centralized evaluation hub API that orchestrates evaluation jobs across multiple tenant namespaces, integrating with multiple evaluation providers (garak, lm-evaluation-harness, guidellm, lighteval, IBM CLEAR) and collections of benchmarks.

All controllers share a common pattern of deploying kube-rbac-proxy sidecars for authentication, OpenShift Route creation for external access, and comprehensive status management. The operator uses a service-registration architecture where controllers are selectively enabled via `--enable-services` flags.

## Architectural Analysis

The trustyai-service-operator follows a **multi-controller single-binary** architecture where five independent controllers share one operator process. Controllers are registered via a service-registration pattern (`controllers/controllers.go`) and selectively enabled via `--enable-services` flags injected by kustomize overlays. This design allows the same codebase to serve both full platform deployments (RHOAI/ODH with all services) and focused deployments (e.g., LMES-only or guardrails-only).

The operator exhibits a **template-driven resource creation** pattern across all controllers. Go templates embedded via `//go:embed` generate Kubernetes manifests at runtime, with dynamic values injected from CR specs and operator ConfigMaps. This is most complex in the GORCH controller, where the deployment template conditionally includes 1-6 containers (orchestrator, gateway, built-in detector, and up to 3 kube-rbac-proxy sidecars) based on feature flags. The template approach enables flexible deployment topologies but makes the actual resource specifications harder to audit statically.

The **kube-rbac-proxy sidecar pattern** is used universally across TAS, GORCH, NemoGuardrails, and EvalHub controllers. Each creates a ServiceAccount, ClusterRoleBinding to `system:auth-delegator`, and a ConfigMap containing RBAC authorization rules. The proxy TLS configuration enforces TLS 1.2 minimum with FIPS-approved cipher suites only. EvalHub extends this pattern with **multi-tenant authorization** via X-Tenant header mapping to Kubernetes SubjectAccessReview checks, enabling per-namespace access control to the centralized evaluation API.

The **LMES driver architecture** is particularly noteworthy. Rather than running the lm-eval process directly, the controller creates a pod with an init container that copies a Go driver binary to a shared EmptyDir, then the main container runs the driver which wraps the Python lm-eval subprocess. The driver provides an HTTP status API on localhost:18080 that the controller polls via `kubectl exec` (SPDY) at configurable intervals. This design enables progress tracking (tqdm parsing), graceful shutdown, and decouples the controller's status polling from the evaluation framework's internal state. The driver also handles S3 asset download, OCI result upload, and custom task git cloning — responsibilities that would otherwise require complex init container chains.

The **GORCH auto-configuration** mechanism demonstrates a sophisticated service discovery pattern. The controller watches KServe InferenceServices and ServingRuntimes, computes a SHA256 hash of all discovered resource versions, and only regenerates configuration when the hash changes. This prevents unnecessary deployment rollouts while ensuring configuration stays current as InferenceServices are added or removed. The generated ConfigMap is further hash-tracked via pod template annotations to trigger rolling updates only when actual configuration content changes.

The **EvalHub multi-tenancy** model stands out as the most complex RBAC setup. The controller watches all namespaces for a tenant label (`evalhub.trustyai.opendatahub.io/tenant`), and for each labeled namespace provisions a job ServiceAccount, access Roles, RoleBindings, and service CA ConfigMaps. The Evaluation Job Failure Reconciler watches batch Jobs and Pods across all tenant namespaces, automatically detecting evaluation failures and POST-ing failure events back to the EvalHub API with proper authentication. Cleanup uses label selectors rather than owner references due to the cross-namespace nature of the resources.

One architectural concern is the **cache disabling** for core resources (ConfigMaps, Secrets, Pods, Services) in the operator's controller-manager configuration (`cmd/main.go:117-125`). This was explicitly added to prevent OOM from cluster-wide informer cache flooding, suggesting the operator operates in environments with large numbers of these resources. The trade-off is increased API server load (~50ms per read vs ~1ms cached) for infrequent operations.
