## Purpose

**Short**: Multi-CRD Kubernetes operator that manages AI trustworthiness, model evaluation, guardrails orchestration, and evaluation hub services on the RHOAI platform.

**Detailed**: The TrustyAI Service Operator is a controller-runtime based Go operator that manages five distinct Custom Resource Definitions across four functional domains: model explainability and fairness monitoring (TrustyAIService), language model evaluation (LMEvalJob), guardrails orchestration for content safety (GuardrailsOrchestrator), standalone NVIDIA NeMo guardrails provisioning (NemoGuardrails), and centralized evaluation management (EvalHub). Each CRD has a dedicated controller that dynamically creates and manages Kubernetes resources — Deployments, Services, Routes, ConfigMaps, Secrets, PVCs, and RBAC objects — to deploy the appropriate service infrastructure.

The operator uses a pluggable controller registration system where individual controllers are selectively enabled via the `--enable-services` flag, allowing the platform operator (rhods-operator) to compose deployments with only the needed functionality. Controllers integrate deeply with the RHOAI ecosystem: TrustyAIService integrates with KServe InferenceServices and ModelMesh for model monitoring; LMEvalJob creates ephemeral evaluation pods with a driver/worker architecture and optional Kueue job scheduling; GuardrailsOrchestrator auto-discovers InferenceServices to configure guardrails detection pipelines; EvalHub provides multi-tenant evaluation management with provider/collection frameworks. All external-facing endpoints are protected by kube-rbac-proxy sidecars with SubjectAccessReview authentication when the `security.opendatahub.io/enable-auth` annotation is set.

### ta-lmes-driver

**Dockerfile**: `Dockerfile.konflux.driver`
**Intent**: Kubernetes-native execution engine injected as an init container into LMEvalJob pods. Copies the driver binary to a shared volume, then runs alongside the main lm-eval container to coordinate evaluation, report progress via HTTP, and upload results to S3/OCI registries.
**Image**: `quay.io/trustyai/ta-lmes-driver:latest`
**Language**: Go 1.23
**Port**: 18080/TCP (pod-internal HTTP status API)

#### API Routes

## Architectural Analysis

The TrustyAI Service Operator is architecturally notable for packing five distinct controller domains into a single operator binary. This design choice — enabled by the `--enable-services` pluggable registration system in `controllers/controllers.go` — allows the platform operator to compose exactly the controller set needed for a given deployment. Each controller is fully self-contained with its own API types, templates, constants, and reconciliation logic, connected only through the shared operator namespace and ConfigMap for image references. The trade-off is binary size and memory footprint: even with `GOMEMLIMIT=630MiB` and disabled informer caching for high-volume types (ConfigMaps, Secrets, Pods, Services), the operator carries the full dependency tree of all five controllers regardless of which are enabled.

The authentication pattern across all controllers is consistent and well-architected. When the `security.opendatahub.io/enable-auth: "true"` annotation is set on a CR, the controller injects a kube-rbac-proxy sidecar container into the managed Deployment. The proxy performs TokenReview and SubjectAccessReview against the Kubernetes API before forwarding requests to the upstream application container. The kube-rbac-proxy configuration templates (found in each controller's `templates/kube-rbac-proxy/` directory) restrict cipher suites to FIPS 140-2 approved algorithms. EvalHub extends this pattern with multi-tenant SubjectAccessReview where the `X-Tenant` header is mapped to a namespace for authorization checks, enabling cross-namespace evaluation job management without cluster-admin privileges.

The LMEvalJob controller implements a sophisticated pod-based execution model rather than using Kubernetes Jobs. Evaluation pods consist of an init container (driver binary copy), a main container (lm-eval framework), and optional sidecar containers. The driver process provides an HTTP API on port 18080 for progress reporting, and the controller polls this API via pod exec to track evaluation state. This architecture enables real-time progress bars in the CRD status, graceful cancellation, and integration with Kueue for queue-based job scheduling via the job manager controller. Input validation is thorough — the controller blocks shell metacharacters (`;`, `&`, `|`, `$`, backtick, `(`, `)`, etc.) in all user-provided arguments to prevent command injection, since evaluation commands are constructed as shell strings.

The GuardrailsOrchestrator controller's auto-configuration system is a standout design pattern. Rather than requiring manual configuration of detector endpoints, the controller watches InferenceServices with the `trustyai/guardrails=true` label and automatically generates orchestrator and gateway configuration YAML. It uses SHA256 hashing of the generated configuration to detect changes and trigger deployment rollouts only when the configuration actually differs. The controller also introspects ServingRuntimes to resolve ports and auto-detects HTTPS services by parsing their status URLs, mounting the corresponding TLS serving certificates into the orchestrator pod.

The EvalHub controller is the most complex in terms of RBAC management. It implements a full multi-tenant model where tenant namespaces are marked with the `evalhub.trustyai.opendatahub.io/tenant` label. For each tenant namespace, the controller provisions ServiceAccounts, Roles, RoleBindings, and CA ConfigMaps — using DNS-1123 normalized names with truncation and hash suffixes to handle long namespace names within the 63-character label value limit. The evaluation job failure reconciler is a separate controller that watches batch Jobs for failures in container states (OOMKilled, CrashLoopBackOff, ImagePullBackOff) and POSTs failure events back to the EvalHub API, enabling the UI to surface infrastructure failures that the evaluation framework itself cannot detect.
