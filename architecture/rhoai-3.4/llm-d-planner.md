## Purpose

**Short**: llm-d Planner is a capacity planning and deployment recommendation service that guides users from business requirements to production LLM deployments with GPU sizing, performance estimation, and Kubernetes YAML generation.

**Detailed**: llm-d Planner unifies LLM deployment planning into a single platform with three capabilities: (1) the main **Planner** workflow that uses conversational intent extraction via a local Ollama LLM to convert natural-language business requirements into structured deployment specifications, queries a PostgreSQL benchmark database to find GPU configurations meeting SLO targets, scores and ranks solutions across accuracy, price, latency, and complexity criteria, and generates production-ready KServe/vLLM Kubernetes YAML; (2) a **Capacity Planner** that estimates GPU memory requirements (model weights, KV cache for MHA/GQA/MQA/MLA attention types, activation memory, system overhead) for any HuggingFace model; and (3) a **GPU Recommender** that estimates inference performance (TTFT, ITL, throughput) across GPU types using the BentoML llm-optimizer roofline model.

The system consists of a FastAPI backend API (28 endpoints across 9 route modules), a Streamlit web UI, a PostgreSQL benchmark database, an Ollama LLM inference sidecar, and an optional vLLM simulator for GPU-free development. It integrates with the RHOAI Model Catalog API for external model and benchmark data, and with the Kubernetes API for cluster GPU detection and InferenceService deployment. The backend deploys on OpenShift with security hardening (non-root, dropped capabilities, seccomp profile) and uses OpenShift service-serving CA injection for TLS communication with the Model Catalog.

## Architectural Analysis

The llm-d Planner follows a classic three-tier architecture (UI → API → Database) augmented with two ML-adjacent sidecars: Ollama for local LLM inference and an optional vLLM simulator for GPU-free development. The FastAPI backend is the central orchestration point — it houses all business logic in a modular Python package structure with clear separation between API routing (`api/routes/`), business orchestration (`orchestration/workflow.py`), domain modules (`recommendation/`, `knowledge_base/`, `intent_extraction/`, `specification/`), and infrastructure (`cluster/`, `configuration/`).

The recommendation engine implements a sophisticated multi-stage pipeline: (1) LLM-powered intent extraction converts free-text into a structured `DeploymentIntent` schema with extensive hallucination cleanup logic (`_clean_llm_output` handles LLM output normalization across 9 use case types, 5 experience classes, and 3 priority levels); (2) the intent maps to traffic profiles from GuideLLM benchmark configurations; (3) PostgreSQL queries find benchmark configurations meeting SLO targets using window functions that rank by throughput; (4) when benchmarks are missing, the BentoML llm-optimizer roofline model generates estimated performance data that gets persisted back to PostgreSQL; (5) a multi-criteria scorer evaluates each configuration on accuracy (from Artificial Analysis CSV data + task-specific bonuses), price (non-linear cost^0.7 formula), latency (capped range scoring), and complexity (GPU count penalty); (6) an analyzer produces five ranked views (best accuracy, lowest cost, lowest latency, simplest, balanced). The scalability penalty system (lines 514-530 of `config_finder.py`) progressively penalizes configurations requiring many replicas, from 2% for 1-3 replicas to 35% for 11-20 replicas.

The YAML generation system uses Jinja2 templates to produce KServe InferenceService, HPA, and ServiceMonitor resources.
