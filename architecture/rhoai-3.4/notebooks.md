## Purpose

**Short**: Builds container images for interactive data science workbenches (Jupyter, RStudio, Code-Server) used in Red Hat OpenShift AI.

**Detailed**: The notebooks repository is the image factory for RHOAI's workbench ecosystem. It produces ~25 distinct container images spanning three IDE families (JupyterLab, RStudio Server, Code-Server/VS Code) across multiple accelerator variants (CPU, CUDA, ROCm) and Python versions. These images are not deployed directly — they are published as OpenShift ImageStreams that the platform operator (`rhods-operator`) deploys to clusters, and the ODH Dashboard presents to users for workbench creation.

The repository also produces pipeline runtime images consumed by Elyra for executing notebook pipeline nodes on Data Science Pipelines. Each image directory contains Dockerfiles (standard and Konflux hermetic variants), Python dependency lockfiles (`pylock.toml`), and build argument configuration files. The images form a layered hierarchy where `jupyter/minimal` serves as the foundation, `jupyter/datascience` adds scientific Python packages, and specialized images (PyTorch, TensorFlow, TrustyAI, LLM Compressor) extend from there with ML framework-specific dependencies.

The build system is entirely Makefile-driven and supports two modes: ODH mode (standard Dockerfiles with network access) and RHOAI/Konflux mode (hermetic `Dockerfile.konflux.*` variants with all dependencies prefetched via cachi2/Hermeto). Kubernetes manifests in `manifests/rhoai/` define ImageStream resources with multi-version tags (N through N-7), kustomize parameterization via `configMapGenerator` and `replacements`, and annotations consumed by the ODH Dashboard for image discovery and display.

## Architectural Analysis

The notebooks repository is architecturally unusual within the RHOAI ecosystem because it produces container images rather than running services. Its primary artifact is ~25 container images organized in a layered hierarchy, where each layer adds progressively more ML-specific packages. This hierarchy is conceptual rather than build-time — each image has a standalone multi-stage Dockerfile that references its parent via `FROM`, not through a build dependency within the repository. This design allows independent rebuilds of any image without cascading rebuilds of the full tree.

The hermetic build system is a distinguishing architectural feature. Konflux Dockerfiles (`Dockerfile.konflux.*`) are completely isolated from the network during build. All dependencies — RPMs, Python wheels, npm packages, even GPG keys — are prefetched by cachi2/Hermeto and mounted at `/cachi2/output/deps/`. The repository maintains lock files at three layers: `rpms.lock.yaml` for OS packages, `pylock.*.toml` (PEP 751 format) for Python dependencies per image variant, and `artifacts.lock.yaml` for generic artifacts. This multi-layer hermeticity is more thorough than most RHOAI components, reflecting the supply chain sensitivity of images that data scientists use directly.

The Code-Server and RStudio images contain an interesting architectural hack: a three-process stack (nginx → httpd → bash CGI) that emulates the Jupyter `/api/kernels/` endpoint. This exists because the notebook controller's idle culler expects a Jupyter-compatible API, but neither VS Code nor RStudio provides one natively. The CGI scripts detect activity by polling code-server's healthz endpoint or parsing nginx access logs. The ARCHITECTURE.md notes this is "fragile and is planned for replacement with a single Go reverse proxy." This is a significant maintenance burden — the nginx/httpd/CGI chain adds failure modes and complexity to images that are otherwise straightforward IDE containers.

The manifest system uses kustomize `replacements` extensively (80+ replacement entries in the RHOAI kustomization.yaml) to inject image digests from `params.env` into ImageStream tags. Each ImageStream maintains up to 8 version tags (N through N-7), enabling rolling upgrades where users can continue using older image versions. The `params.env` file stores image references as `registry.redhat.io/rhoai/...@sha256:...` digests, not tags, ensuring immutable image references. The `params-latest.env` file uses `dummy` placeholders for the current (N) version, which are replaced by the release pipeline.

The multi-architecture support (x86_64, aarch64, ppc64le, s390x) introduces complexity in the build system, particularly for Python packages. The `whl-cache` stage in the Code-Server Dockerfile demonstrates this: on x86_64/aarch64 pre-built wheels exist, but on ppc64le/s390x, packages like numpy and scipy must be compiled from source using RHOAI-provided wheels with OpenBLAS runtime libraries. This cross-architecture support is a significant engineering investment that most open-source notebook projects do not undertake.
