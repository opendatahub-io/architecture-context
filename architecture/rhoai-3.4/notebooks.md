## Purpose

**Short**: Builds and maintains container images for interactive data science workbenches (JupyterLab, RStudio, Code Server) and pipeline runtime environments used by Red Hat OpenShift AI.

**Detailed**: The notebooks repository is a container image factory that produces over 20 distinct container images spanning three workbench types (JupyterLab, RStudio Server, Code Server) and seven pipeline runtime variants. Each workbench type ships in multiple hardware acceleration flavors — CPU, NVIDIA CUDA 12.9, and AMD ROCm 6.4 — with specialized ML framework stacks (PyTorch 2.10, TensorFlow 2.21, LLMCompressor 0.10, TrustAI). All images are based on Red Hat UBI9 with Python 3.12 and follow a layered build chain where specialized images inherit from more minimal base images.

The repository implements a fully hermetic build system integrated with Konflux CI/CD and Cachi2 dependency prefetching. Every dependency layer — OS RPMs, Python wheels, Go modules, and npm packages — is locked, hash-verified, and prefetched for offline builds. The resulting images are deployed to OpenShift clusters as workbench pods managed by the odh-notebook-controller, with image references distributed via OpenShift ImageStream manifests that the RHOAI platform operator (rhods-operator) applies to clusters. The manifests use kustomize replacements to inject pinned image digests from ConfigMaps, enabling version management across 8 historical release tags per workbench type.

## Architectural Analysis

The notebooks repository follows a **layered image inheritance pattern** where each workbench variant builds upon a chain of progressively more specialized base images. The CPU chain is: UBI9 Python 3.12 → base-images/cpu → jupyter-minimal → jupyter-datascience → (pytorch | tensorflow | trustyai). The CUDA chain substitutes a CUDA base layer that adds the NVIDIA toolkit (12.9.1), cuDNN (9.14.0), and NCCL (2.27.3). The ROCm chain provides AMD GPU support via ROCm 6.4. This inheritance model ensures that security patches and base-layer updates automatically propagate to all downstream images, but it also means a single base-layer CVE triggers rebuilds of 20+ images.

The **hermetic build system** is notably sophisticated for a container image repository. Rather than relying on network access during builds, every dependency — RPMs, Python wheels, Go modules, npm packages — is pre-resolved into lock files and prefetched by Cachi2 before the build starts. The `Dockerfile.konflux.*` variants use `--no-index --find-links /cachi2/output/deps/pip` to install Python packages entirely offline. RPM lock files (`rpms.lock.yaml`) pin every OS package with architecture-specific sha256 checksums across x86_64, aarch64, ppc64le, and s390x. This is a defense-in-depth approach to supply chain security: even if a PyPI package is compromised between lock resolution and image build, the hash verification will catch the mismatch.

The **multi-architecture support** is handled through careful per-architecture conditional logic in Dockerfiles and lock files. The x86_64 and aarch64 architectures use pre-built wheels from PyPI and Red Hat's RHOAI Python index, while ppc64le and s390x require special handling: a `whl-cache` build stage compiles C-extension packages from source using RHOAI-provided wheels (numpy, scipy, pyarrow) that are not available as pre-built wheels on PyPI for these architectures. The `openblas-threads` RPM provides the specific soname (`libopenblasp.so.0`) these wheels link against at runtime. PDF export dependencies (texlive, pandoc) are skipped entirely on s390x and ppc64le due to unavailable packages.

The **workbench proxy architecture** is consistent across all workbench types: nginx serves as a reverse proxy in front of the application server (JupyterLab/code-server/RStudio), handling path prefix rewriting via `NB_PREFIX` environment variable injection, activity logging for idle culling (writing JSON session activity to access logs), and health probe endpoints. Apache httpd runs alongside nginx to serve CGI scripts that provide the `/api/kernels` activity endpoint consumed by the notebook controller's culling logic. This dual-proxy pattern (nginx + httpd) exists because nginx handles the main traffic proxying while httpd's CGI support is needed for the activity API without adding application-level HTTP handlers. Code-server and RStudio use the same nginx+httpd pattern while JupyterLab uses its native `ServerApp` API for activity reporting.

The **manifest system** uses a ConfigMap-driven parameterization pattern. Image digests live in `params.env` / `params-latest.env` files, which kustomize turns into ConfigMaps. Over 40 kustomize replacement blocks then inject these ConfigMap values into ImageStream tag `from.name` fields and commit hash annotations. At deploy time, the platform operator can override `params-latest.env` entries via `RELATED_IMAGE_*` environment variables (standard OLM disconnected-cluster pattern), enabling air-gapped deployments where images come from a mirrored registry rather than registry.redhat.io.
