## Purpose

**Short**: A reverse proxy sidecar that routes LLM inference requests between prefill and decode workers in a disaggregated prefill/decode (P/D) serving architecture.

**Detailed**: The llm-d-routing-sidecar is a lightweight HTTP/HTTPS reverse proxy that sits alongside a vLLM decoder instance in a Kubernetes pod. It intercepts OpenAI-compatible completion requests and orchestrates the disaggregated prefill/decode protocol used by the llm-d inference serving stack.

When a request arrives with an `x-prefiller-host-port` header (set by the upstream routing layer, typically the llm-d-inference-scheduler), the sidecar forwards the request to the specified remote prefill worker first, captures the KV-cache transfer parameters from the prefill response, then forwards a modified request to the local decoder with those parameters so the decoder can load the pre-computed KV cache via NIXL (or LMCache) and generate tokens without repeating the prefill computation.

The sidecar supports three connector protocols (nixlv2, nixl v1 deprecated, lmcache deprecated), TLS for both prefiller and decoder connections, and an optional SSRF protection feature that watches Kubernetes InferencePool resources to maintain a dynamic allowlist of valid prefill targets. Note: this repository is deprecated — the code has been moved to the llm-d-inference-scheduler repository under `cmd/pd_sidecar` and `pkg/sidecar`.

## Architectural Analysis

The llm-d-routing-sidecar implements a lightweight but architecturally significant pattern in the llm-d disaggregated serving stack. It runs as a sidecar container alongside each vLLM decoder pod, intercepting all OpenAI-compatible API traffic on port 8000 before it reaches the decoder on port 8001. This sidecar pattern means every decoder pod in the cluster runs an instance of this proxy, making it a critical data-plane component for inference latency.

The protocol connector abstraction (`protocolRunner` function type in `proxy.go:94`) cleanly separates the P/D protocol logic from the HTTP proxy infrastructure. The NIXL V2 connector (`connector_nixlv2.go`) implements a two-phase request flow: it first sends a modified request to the remote prefiller with `kv_transfer_params` containing `do_remote_decode: true`, captures the `kv_transfer_params` from the prefiller's response (which now contains NIXL block IDs and engine metadata), then forwards the original request to the local decoder with those transfer parameters attached. This enables the decoder to load pre-computed KV cache blocks over NIXL without re-running the prefill computation. The V1 connector handles individual fields (`remote_block_ids`, `remote_engine_id`, `remote_host`, `remote_port`) rather than the consolidated `kv_transfer_params` object.

The SSRF protection mechanism (`allowlist.go`) is noteworthy for its use of Kubernetes dynamic client and shared informers to watch InferencePool resources at the Gateway API `inference.networking.x-k8s.io/v1alpha2` API group. When enabled, it extracts the label selector from the InferencePool spec, creates a pod informer filtered by that selector, and maintains an in-memory set of allowed pod IPs and hostnames. The allowlist is rebuilt on every pod add/update/delete event, with a 30-second resync period. The validation only checks hosts (not ports), so any port on an allowed host is permitted — this is a deliberate design tradeoff noted in the README. The SSRF protection is disabled by default for backward compatibility.

The TLS implementation shows careful security consideration: the proxy serves TLS by default (secure-proxy flag defaults to true) with a curated cipher suite list excluding weak algorithms. It supports separate TLS configuration for prefiller and decoder connections, including optional `InsecureSkipVerify` flags for development environments. The self-signed certificate generation (`tls.go`) uses 4096-bit RSA keys and is intended only for testing — production deployments should provide certificates via `--cert-path`. However, the upstream Dockerfile builds with `CGO_ENABLED=0` (static linking), which means the binary uses Go's pure-Go crypto implementation rather than OpenSSL, and would not pass the FIPS check-payload gate. The downstream RHOAI build presumably addresses this.

This repository is now deprecated — the codebase has been merged into the llm-d-inference-scheduler repository. This architectural snapshot captures the standalone sidecar as it exists on the main branch, which is the version consumed by the RHOAI 3.4 release.
