## Purpose

**Short**: A reverse proxy sidecar that implements disaggregated prefill/decode (P/D) routing for LLM inference, redirecting incoming requests to remote prefill workers before forwarding to the local decoder (vLLM).

**Detailed**: The llm-d-routing-sidecar is a lightweight HTTP/HTTPS reverse proxy designed to run as a sidecar container alongside a vLLM decoder instance in a disaggregated serving (prefill/decode) architecture. It intercepts OpenAI-compatible chat completion and legacy completion requests, inspects the `x-prefiller-host-port` HTTP header, and orchestrates a two-phase inference flow: first sending the request to a remote prefiller pod, then forwarding the KV-transfer parameters from the prefiller response to the local decoder for final token generation.

The sidecar supports three P/D connector protocols — NIXL v2 (default, active), NIXL v1 (deprecated), and LMCache (deprecated). It includes an optional SSRF protection feature that watches Kubernetes `InferencePool` resources (from the `inference.networking.x-k8s.io` API group) to maintain a dynamic allowlist of valid prefill target pod IPs. The proxy serves TLS by default (with configurable certificate paths or self-signed fallback) and enforces TLS 1.2+ with a restricted set of AEAD cipher suites on all connections — both inbound (proxy listener) and outbound (prefiller and decoder).

**Note**: This repository is **deprecated**. The code has been moved to the [llm-d-inference-scheduler](https://github.com/llm-d/llm-d-inference-scheduler) repository under `cmd/pd_sidecar` and `pkg/sidecar`. No new development will occur in this repository.

## Architectural Analysis

The llm-d-routing-sidecar implements a clean sidecar proxy pattern for disaggregated LLM inference. Its architecture is intentionally minimal — a single Go binary with no CRDs, no controller-runtime dependency, and no operator framework. The proxy intercepts only two API paths (`/v1/chat/completions` and `/v1/completions`) and passes everything else through to the local vLLM decoder via a catch-all reverse proxy handler. This design ensures the sidecar is transparent to non-P/D traffic and adds negligible overhead for direct decode operations.

The NIXL v2 protocol implementation (`connector_nixlv2.go`) is the core of the disaggregated serving logic. It performs a two-phase request transformation: (1) modifies the original completion request by injecting `kv_transfer_params` with `do_remote_decode: true`, sets `max_tokens: 1` and disables streaming, then sends this to the remote prefiller; (2) captures the prefiller's response (which contains filled-in `kv_transfer_params` with block IDs, engine ID, host, and port), restores the original streaming/token settings, injects the prefiller's `kv_transfer_params`, and sends the modified request to the local decoder. This stateless two-phase orchestration avoids any persistent state between requests, relying entirely on the request/response cycle for KV transfer coordination. The LRU cache for prefiller proxy handlers (capacity: 16) is an optimization to avoid repeatedly constructing reverse proxy instances for frequently-used prefiller endpoints.

The SSRF protection system is architecturally notable for using raw Kubernetes dynamic client informers rather than controller-runtime, keeping the binary lightweight but requiring manual resource lifecycle management. The `AllowlistValidator` watches a specific named `InferencePool` resource (from the Gateway API inference extension project, API group `inference.networking.x-k8s.io/v1alpha2`), extracts its pod label selector, then creates a secondary pod informer filtered by those labels. The allowlist tracks pod IPs and pod names (host-only, not host:port), meaning any port on an allowed pod IP is permitted. The informer uses a 30-second resync period. This approach has a security trade-off: it allows connections to any port on allowed hosts, which is sufficient for the P/D use case but broader than strictly necessary.

The TLS configuration is security-conscious: the proxy defaults to serving TLS (`--secure-proxy=true`), enforces TLS 1.2 as the minimum version, and restricts cipher suites to six AEAD-only ciphers (AES-128-GCM-SHA256, AES-256-GCM-SHA384, and CHACHA20-POLY1305 for both RSA and ECDSA key exchange). This same cipher suite configuration is applied consistently across the proxy listener, decoder client, and prefiller client TLS contexts. The `InsecureSkipVerify` flags for prefiller and decoder are disabled by default but available for testing environments, which is an appropriate compromise for development flexibility vs. production security.

This repository is deprecated — all development has moved to the `llm-d-inference-scheduler` repository. The RHOAI 3.4 distribution uses this codebase (or its migrated equivalent in the inference scheduler) as a sidecar in P/D-enabled inference pods. The absence of a Konflux Dockerfile on this branch suggests the downstream build system either maintains its own Dockerfile or builds from the inference-scheduler repository instead.
