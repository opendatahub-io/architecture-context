## Purpose

**Short**: Provides a multi-tenant API gateway platform for serving LLM models (KServe-hosted and external providers) with token-based rate limiting, API key management, and fine-grained access control via Kuadrant/Authorino.

**Detailed**: Models as a Service (MaaS) is a platform component for Red Hat OpenShift AI (RHOAI) that creates a unified inference gateway for large language models. It consists of two main sub-components: a Go-based Kubernetes controller (`maas-controller`) that manages the platform lifecycle, model routing, authentication policies, and subscription-based rate limiting; and a Go-based REST API service (`maas-api`) that provides API key management, subscription selection, and model discovery.

The controller defines five CRDs in the `maas.opendatahub.io/v1alpha1` API group: `Tenant` (platform singleton for global configuration), `MaaSModelRef` (model endpoint references), `ExternalModel` (external LLM provider definitions), `MaaSAuthPolicy` (authentication and authorization rules), and `MaaSSubscription` (token rate limits and billing). The controller reconciles these CRDs into Gateway API HTTPRoutes, Kuadrant AuthPolicies, Kuadrant TokenRateLimitPolicies, and Istio resources (ServiceEntry, DestinationRule, EnvoyFilter) to expose models through a shared Gateway with per-user/per-subscription rate limiting.

The API service manages a PostgreSQL-backed API key store with hash-only storage (SHA-256 with per-key salt), integrates with Authorino for API key validation callbacks, and provides subscription selection logic that resolves which subscription to apply for each inference request. A payload-processing sidecar (EnvoyFilter ext_proc) handles model name extraction, API translation, and credential injection at the gateway level.

| Component | Type | Purpose |
|-----------|------|---------|

## Architectural Analysis

The Models-as-a-Service architecture follows a **two-tier reconciliation pattern**: the platform operator (rhods-operator/opendatahub-operator) deploys maas-controller via the DataScienceCluster CR, and maas-controller in turn deploys maas-api and payload-processing via its Tenant reconciler using embedded kustomize overlays. The Tenant CR acts as a platform-level configuration singleton (enforced by CEL validation to name "default-tenant") that drives the entire platform lifecycle. This pattern allows maas-controller to own the deployment lifecycle of its dependent services while the platform operator only needs to manage a single controller deployment.

The **authentication architecture** is notably layered: Authorino evaluates three credential types in priority order (API keys → OpenShift tokens → external OIDC JWTs), then calls back to maas-api internal endpoints for API key validation and subscription selection. The controller generates **aggregated AuthPolicies** — one per model rather than one per MaaSAuthPolicy — by merging all MaaSAuthPolicy subjects that reference the same model into a single Kuadrant AuthPolicy with OPA Rego rules. This aggregation avoids policy conflicts on HTTPRoutes (Kuadrant enforces one AuthPolicy per target) while supporting multi-tenant access patterns. The Authorization header is explicitly stripped after validation to prevent credential exfiltration to model backends — this is the critical security property that requires Kuadrant 1.4.2+ (Authorino v0.23.1+).

The **ExternalModel reconciler** creates a complete Istio networking stack per external provider: an ExternalName Service for DNS resolution, a ServiceEntry for mesh routing (MESH_EXTERNAL location), a DestinationRule for TLS origination, and an HTTPRoute with both path-based and header-based matching rules. The dual-rule HTTPRoute pattern supports two routing modes: path-based matching for the Kuadrant Wasm plugin (which needs path visibility) and header-based matching for the payload-processing BBR (Body-Based Router) that sets X-Gateway-Model-Name after parsing the request body. The payload-processing ext_proc runs as a full-duplex streamed external processor in the Envoy data plane, enabling real-time request/response body manipulation without buffering entire payloads.

The **subscription and rate limiting model** uses a priority-based system where each MaaSSubscription has a numeric priority and per-model token rate limits. The controller generates aggregated TokenRateLimitPolicies per model (similar to AuthPolicy aggregation), with CEL predicates that key on `auth.identity.selected_subscription_key` to apply the correct rate limit per user/subscription combination. The subscription selection logic in maas-api handles multiple subscriptions for the same user by requiring explicit selection via the X-Maas-Subscription header, with auto-selection only when a single subscription matches. Duplicate priority detection across subscriptions triggers warning conditions.

The **API key security design** implements defense-in-depth: keys use a per-key salt derived from the key_id (SHA-256(key_id + "\x00" + secret)), plaintext keys are never stored (hash-only with show-once pattern), validation uses constant-time comparison, and IDOR protection returns 404 instead of 403 for unauthorized access attempts. Ephemeral keys (max 1 hour lifetime) support programmatic use cases with automatic cleanup via a CronJob running every 15 minutes with a 30-minute grace period. The fail-closed design rejects API keys with no bound subscription (CWE-284 mitigation).

A potential security consideration is the use of `InsecureSkipVerify: true` in the model discovery HTTP client (maas-api/internal/models/discovery.go:78) for cluster-internal model endpoint probing. While this is documented as intentional for cluster-internal traffic, it does bypass certificate validation for the access probes. This is mitigated by the fact that these probes only reach KServe endpoints within the same cluster's service mesh.
