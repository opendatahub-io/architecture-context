## Purpose

**Short**: Multi-tenant platform for exposing LLM models (internal KServe and external providers) as OpenAI-compatible API endpoints with API key management, fine-grained authorization, and token-based rate limiting.

**Detailed**: Models as a Service (MaaS) provides a comprehensive platform for serving large language models through a unified Gateway API-based ingress layer. It consists of two primary components: `maas-controller`, a Kubernetes operator that manages the lifecycle of model exposure, authorization policies, subscription-based rate limiting, and platform deployment; and `maas-api`, a Go REST service that provides API key management, model discovery, and internal callback endpoints for Authorino-based authentication.

The platform integrates deeply with the Kuadrant policy engine (AuthPolicy for authentication/authorization, TokenRateLimitPolicy for rate limiting) and KServe (LLMInferenceService for model serving). It supports both internal models served by KServe and external LLM providers (OpenAI, Anthropic, etc.) through ExternalModel CRs, which route traffic via Istio ServiceEntry and DestinationRule resources. A payload processing component (Envoy ext_proc filter) handles API translation, model name extraction, and credential injection for external provider requests.

MaaS enforces a defense-in-depth security model: API keys are validated by Authorino via HTTP callback to maas-api, user identity is propagated via injected headers (X-MaaS-Username, X-MaaS-Group, X-MaaS-Subscription), authorization is enforced via OPA rules in Authorino, rate limiting is enforced by Limitador, and credentials are stripped before forwarding to backends. The Tenant CR drives platform lifecycle via Kustomize-based rendering and Server-Side Apply.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| maas-controller | Go Operator (controller-runtime) | Manages CRDs (Tenant, MaaSModelRef, MaaSAuthPolicy, MaaSSubscription, ExternalModel), creates Kuadrant AuthPolicies/TokenRateLimitPolicies, deploys platform via Kustomize/SSA |
| maas-api | Go REST Service (Gin) | API key management, model discovery (/v1/models), internal Authorino callbacks (validate, subscription select), PostgreSQL-backed |
| payload-processing | Go gRPC Service (Envoy ext_proc) | Request/response transformation: model name extraction, provider resolution, API translation, credential injection |

## Architectural Analysis

Models as a Service represents a sophisticated multi-layer architecture that sits atop the Kuadrant policy engine and KServe model serving platform. The most notable design decision is the **aggregated policy pattern**: rather than creating one Kuadrant AuthPolicy per MaaSAuthPolicy or one TokenRateLimitPolicy per MaaSSubscription, the controllers aggregate all policies for a given model into a single Kuadrant resource. This avoids Kuadrant's conflict resolution limitations (only one AuthPolicy/TRLP per HTTPRoute is accepted) while supporting multi-tenant access control. The aggregation logic in `maasauthpolicy_controller.go` is the most complex code in the repository, building CEL-expression-based cache keys, OPA authorization rules, and response header injection rules that encode user identity, group membership, and subscription metadata.

The **Tenant reconciler** implements a miniature platform operator within MaaS. It renders Kustomize manifests from embedded overlays, applies post-render transformations (injecting gateway-aware AuthPolicies, TelemetryPolicies, and DestinationRules), and uses Server-Side Apply to deploy the entire maas-api stack including its RBAC, networking, and observability resources. This design allows the platform operator (rhods-operator/opendatahub-operator) to delegate MaaS deployment to a single Tenant CR, while the Tenant reconciler handles all internal complexity. The `prerequisites.go` and `pipeline.go` files implement a staged deployment pipeline with dependency checking (AuthConfig CRD from Kuadrant), prerequisite validation (PostgreSQL secret), and deployment readiness polling.

The **security model** is defense-in-depth with five enforcement layers: (1) Authorino validates credentials and injects identity headers, (2) OPA rules in AuthPolicy enforce group-based access, (3) Limitador enforces token rate limits, (4) NetworkPolicy restricts internal endpoint access, and (5) the Authorization header is stripped before forwarding to backends (preventing credential exfiltration to model services). API keys use a per-key salt design (the `key_id` portion of `sk-oai-{key_id}_{secret}` serves as salt) with SHA-256 hashing and constant-time comparison. The subscription binding at key creation time ensures API keys cannot be used with unauthorized subscriptions.

The **ExternalModel** flow is architecturally distinct from the LLMInferenceService path. External models require Istio mesh integration: the ExternalModel reconciler creates a Service (ExternalName), ServiceEntry (to register the external host in the mesh), DestinationRule (for TLS SNI), and HTTPRoute (for gateway routing). The payload-processing ext_proc filter then handles the critical transformation: extracting the model name from the request body, resolving the provider from the ExternalModel CR, translating the OpenAI API format to the provider's format, and injecting the provider's API key from a Kubernetes Secret. This pipeline ensures external model access is transparent to clients using the OpenAI-compatible API.

The **observability architecture** is comprehensive: Kuadrant TelemetryPolicy extracts model, user, subscription, organization, and cost center labels from every request; Istio Telemetry adds per-subscription latency metrics; ServiceMonitors scrape Authorino, Limitador, Envoy, and controller metrics; and optional Perses dashboards provide pre-built visualizations. The telemetry data enables both operational monitoring and billing/chargeback use cases.
