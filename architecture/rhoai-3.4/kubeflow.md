## Purpose
**Short**: ODH Notebook Controller extends the upstream Kubeflow Notebook Controller with OpenShift-specific integrations for authentication, certificate management, networking, and pipeline connectivity for RHOAI workbenches.

**Detailed**: The ODH Notebook Controller is a Kubernetes operator that watches Kubeflow `Notebook` custom resources and augments them with Red Hat OpenShift AI (RHOAI) platform capabilities. For each Notebook (workbench), the controller manages Gateway API `HTTPRoute` resources in a central namespace (the controller's own namespace) to expose workbenches externally via the `data-science-gateway` Gateway, creates `ReferenceGrant` resources in user namespaces to enable cross-namespace backend references, and reconciles `NetworkPolicy` resources to secure pod-level traffic.

When authentication is enabled via the `notebooks.opendatahub.io/inject-auth` annotation, the controller creates a kube-rbac-proxy sidecar infrastructure: a dedicated `Service` on port 8443/TCP, a `ServiceAccount`, a `ConfigMap` for RBAC authorization configuration, and a `ClusterRoleBinding` to `system:auth-delegator`. The mutating webhook injects the kube-rbac-proxy sidecar container into the notebook pod, which terminates TLS and performs Kubernetes RBAC-based authorization before proxying traffic to the notebook container on port 8888/TCP. The controller also manages trusted CA certificate bundles, Elyra pipeline runtime configuration secrets (using DSPA and Gateway API hostname), Feast feature store configuration, and OpenShift ImageStream-based image resolution.

**HTTPRoute Architecture (RHOAI 3.x)**:
- The controller creates HTTPRoute CRs in the **central namespace** (controller's own namespace, e.g., `opendatahub`) -- not in user namespaces
- Each HTTPRoute references `data-science-gateway` in `openshift-ingress` as its parent Gateway
- HTTPRoutes use path-based routing: `/notebook/{namespace}/{notebook-name}`
- Cross-namespace backend references are enabled via ReferenceGrant CRs created in each user namespace
- **Without auth**: HTTPRoute backend points directly to notebook Service on port 8888/TCP
- **With auth**: HTTPRoute backend points to kube-rbac-proxy Service (`{name}-kube-rbac-proxy`) on port 8443/TCP, which proxies to notebook on port 8888/TCP after RBAC authorization
- Finalizers ensure HTTPRoutes and ReferenceGrants are cleaned up when notebooks are deleted
- Gateway name/namespace configurable via `NOTEBOOK_GATEWAY_NAME` and `NOTEBOOK_GATEWAY_NAMESPACE` env vars (defaults: `data-science-gateway` / `openshift-ingress`)

**Legacy OAuthClient Cleanup (Migration from RHOAI 2.x)**:
- The controller retains finalizer-based cleanup logic for OAuthClient CRs from RHOAI 2.x
- No new OAuthClient CRs are created in RHOAI 3.x; this is migration-only code

### Kustomize Overlay Structure

The component is deployed by rhods-operator using kustomize. The overlay chain is:

```
config/rhoai/overlays/additional/kustomization.yaml (operator entry point)
  -> config/base/kustomization.yaml
    -> config/default/kustomization.yaml
      -> config/rbac/ (ClusterRole, ClusterRoleBinding, ServiceAccount, user ClusterRoles)
      -> config/manager/ (Deployment, Service, params)
      -> config/webhook/ (MutatingWebhookConfiguration, Service)
```

The Deployment is patched by `config/default/webhook_manager_patch.yaml` to add the webhook port (8443/TCP) and mount the webhook TLS certificate secret.
