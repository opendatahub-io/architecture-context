## Purpose

**Short**: Manages the lifecycle of Jupyter Notebook workbenches on OpenShift, including pod scheduling, network exposure via Gateway API, authentication via kube-rbac-proxy sidecars, and integration with Data Science Pipelines, MLflow, and Feast.

**Detailed**: This repository contains two Kubernetes controllers that work together to manage Jupyter Notebook workbenches for Red Hat OpenShift AI (RHOAI). The upstream **notebook-controller** (forked from Kubeflow) handles core notebook lifecycle — creating StatefulSets, Services, and optionally Istio VirtualServices for each Notebook CR, plus an idle-culling system that monitors kernel/terminal activity and stops notebooks after configurable inactivity periods.

The **odh-notebook-controller** extends the upstream controller with OpenShift-specific capabilities: Gateway API integration (HTTPRoutes in a central namespace with cross-namespace ReferenceGrants), kube-rbac-proxy sidecar injection for authentication/authorization, certificate bundle management, and integration with platform services like Data Science Pipelines (DSPA), MLflow tracking, and Feast feature stores. It implements both mutating and validating admission webhooks to inject configuration at notebook creation time and protect running notebooks from breaking changes.

Both controllers are deployed as separate Deployments in the `opendatahub` namespace and operate on the same `Notebook` CRD (`kubeflow.org`). The upstream controller manages the compute infrastructure (StatefulSet, Service) while the ODH controller manages the platform integration layer (networking, auth, secrets, RBAC).

## Admission Webhooks

This component defines 3 webhook(s) (1 mutating, 1 validating, 1 conversion).

| Name | Type | Target Resources | Purpose |
|------|------|-----------------|---------|
| notebooks-validation.opendatahub.io | validating |  | Validates updates to Notebook resources, specifically preventing removal of the MLflow instance annotation while a notebook is running. ## Architectural Analysis

The kubeflow repository implements a dual-controller architecture where two independent controllers operate on the same Notebook CRD but manage different aspects of the notebook lifecycle. The upstream `notebook-controller` handles "compute plane" concerns (StatefulSet, Service, idle culling) while the `odh-notebook-controller` handles "platform plane" concerns (networking, auth, integrations). This separation preserves upstream compatibility — the upstream controller's code is a near-direct fork from Kubeflow with minimal modifications — while the ODH controller adds OpenShift-specific value without patching upstream logic. The two controllers coordinate implicitly through annotations on the Notebook CR, particularly the `kubeflow-resource-stopped` annotation (set by the culler, read by both controllers) and the reconciliation lock annotation `odh-notebook-controller-lock` (set by the ODH mutating webhook, cleared by the ODH controller after first reconciliation).

The RHOAI 3.x networking pattern is fully implemented here. Each notebook gets an HTTPRoute in the controller's central namespace (not the notebook's namespace) that references the platform's `data-science-gateway` Gateway CR. Cross-namespace service references are enabled via a shared ReferenceGrant (`notebook-httproute-access`) created per user namespace. When authentication is enabled (`notebooks.opendatahub.io/inject-auth=true`), the HTTPRoute targets the kube-rbac-proxy service (port 8443) instead of the notebook service (port 80), and the mutating webhook injects the kube-rbac-proxy sidecar container. The kube-rbac-proxy performs SubjectAccessReview checks against the Kubernetes API to verify that the requesting user has `get` permission on the specific Notebook CR, then proxies to the notebook container on localhost:8888 with X-Auth-Request-User and X-Auth-Request-Groups headers. This is a clean per-notebook authorization model tied to Kubernetes RBAC, replacing the RHOAI 2.x pattern of OAuthClient CRs per notebook. Legacy OAuthClient cleanup code remains in `notebook_oauth.go` for migration.

The mutating webhook is architecturally critical — it performs 8+ distinct mutations on notebook creation including image resolution from ImageStreams, CA certificate bundle mounting, pipeline runtime image ConfigMap syncing, Elyra secret injection, Feast config mounting, MLflow environment variable injection, kube-rbac-proxy sidecar injection, and cluster proxy environment variable injection. The webhook also implements a "reconciliation lock" — injecting a `kubeflow-resource-stopped=odh-notebook-controller-lock` annotation on CREATE to prevent the pod from starting before the ODH controller completes its first reconciliation (which creates the HTTPRoute, NetworkPolicies, and other resources the notebook needs). This prevents a race condition where the notebook pod could start before its networking is ready.

The idle culling system is noteworthy for directly querying Jupyter's HTTP API (/api/kernels and /api/terminals) to assess notebook activity, rather than relying on proxy-level metrics. It constructs cluster-internal HTTP URLs to reach each notebook's Jupyter server and checks kernel execution states and terminal last-activity timestamps. This design is effective but creates a dependency on the Jupyter API contract — non-Jupyter notebook images (e.g., RStudio, VS Code) would need compatible API endpoints for culling to work. The culling configuration is externalized to a ConfigMap (`notebook-controller-culler-config`) that the platform operator manages, keeping the culling policy separate from the controller deployment.

Resource cleanup uses a mixed strategy: resources in the notebook's namespace use owner references (automatic garbage collection), while cross-namespace resources (HTTPRoutes in the central namespace) and cluster-scoped resources (ClusterRoleBindings for kube-rbac-proxy auth delegation) use finalizers on the Notebook CR. Three finalizers are managed: `notebook.opendatahub.io/httproute-cleanup`, `notebook.opendatahub.io/referencegrant-cleanup`, and `notebook.opendatahub.io/kube-rbac-proxy-cleanup`. The ReferenceGrant finalizer has additional logic — the ReferenceGrant is only deleted when the last notebook in a namespace is deleted, since it is shared across all notebooks in that namespace.
