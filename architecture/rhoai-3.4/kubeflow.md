## Purpose

**Short**: Manages the lifecycle of Jupyter Notebook workbenches on OpenShift, including pod creation, idle culling, authentication sidecar injection, and Gateway API routing.

**Detailed**: The kubeflow repository contains two cooperating Go controllers that together form the notebook workbench platform for Red Hat OpenShift AI (RHOAI). The **notebook-controller** (upstream Kubeflow) is responsible for core notebook lifecycle management: creating StatefulSets and Services for each Notebook CR, monitoring pod status, and implementing idle notebook culling by querying Jupyter kernel/terminal APIs. The **odh-notebook-controller** (downstream ODH/RHOAI extension) adds OpenShift-specific capabilities including Gateway API HTTPRoute creation for external access, kube-rbac-proxy sidecar injection for authentication, CA certificate bundle management, mutating/validating webhooks, and integrations with Data Science Pipelines (Elyra), MLflow, Feast, and OpenShift ImageStreams.

Both controllers are deployed as separate manager binaries in the `opendatahub` namespace. The notebook-controller watches Notebook CRs and manages StatefulSets, while the odh-notebook-controller intercepts Notebook create/update operations via admission webhooks to inject sidecars, mount configurations, and resolve container images from ImageStreams. Together they implement the RHOAI 3.x ingress pattern: HTTPRoutes referencing a central Gateway with kube-rbac-proxy sidecars for per-notebook RBAC enforcement.

## Architectural Analysis

The kubeflow repository implements a **dual-controller architecture** where the upstream Kubeflow notebook-controller provides core lifecycle management and the downstream odh-notebook-controller adds RHOAI-specific features through a combination of admission webhooks and supplementary reconciliation. This split allows the RHOAI team to rebase against upstream Kubeflow while maintaining downstream extensions cleanly. The two controllers share the `kubeflow.org/v1.Notebook` CRD but have distinct responsibilities: the notebook-controller creates StatefulSets and Services while the odh-notebook-controller creates HTTPRoutes, NetworkPolicies, and injects sidecars.

The **RHOAI 3.x ingress pattern** is well-implemented here: each notebook gets an HTTPRoute in the controller's namespace (e.g., `opendatahub`) that references the `data-science-gateway` as its parent and uses a cross-namespace backend reference to the notebook's Service in the user's namespace. A ReferenceGrant in the user's namespace enables this cross-namespace reference. This is a clean implementation of Gateway API's cross-namespace routing model, but it creates an interesting operational characteristic: all notebook HTTPRoutes live in the controller namespace, not the user's namespace. The `EnsureConflictingHTTPRouteAbsent()` function (notebook_route.go:267-324) handles switching between auth and non-auth modes by deleting conflicting routes, which prevents misconfigured routing when users toggle the auth annotation.

The **kube-rbac-proxy sidecar injection** follows a comprehensive pattern: the mutating webhook injects the container, volumes, and service account reference, while the reconciliation loop creates the supporting Service, ConfigMap (authorization config), ServiceAccount, and ClusterRoleBinding. The RBAC check enforces that users must have GET permission on `kubeflow.org/v1.notebooks` in the notebook's namespace. One architectural risk is that the per-notebook ClusterRoleBinding (`{notebook-name}-rbac-{namespace}-auth-delegator`) is cluster-scoped and cannot use cross-namespace owner references, requiring explicit cleanup during notebook deletion (notebook_kube_rbac_auth.go:344-368). If cleanup fails, orphaned ClusterRoleBindings accumulate.

The **mutating webhook** is notably feature-rich (974 lines in notebook_mutating_webhook.go), handling: reconciliation lock injection, ImageStream resolution, CA bundle mounting, pipeline runtime images, Elyra runtime config, Feast configuration, MLflow environment variables, kube-rbac-proxy injection, cluster proxy env vars, and restart blocking. The restart-blocking logic (lines 518-581) is particularly interesting: it detects pod-spec-changing updates to running notebooks and blocks them by setting a `notebooks.opendatahub.io/update-pending` annotation with the diff reason, requiring users to explicitly restart. This prevents accidental disruption of running workloads.

The **idle culling mechanism** in the notebook-controller is straightforward but relies on HTTP access to the Jupyter server's `/api/kernels` and `/api/terminals` endpoints. The culling controller directly queries the notebook pod's service (port 8888) to check kernel and terminal activity. A NetworkPolicy (`{notebook-name}-ctrl-np`) explicitly allows this ingress from the controller namespace. If the notebook has been idle longer than `CULL_IDLE_TIME` (default 24 hours), the controller sets the `kubeflow-resource-stopped` annotation which triggers the notebook-controller to scale the StatefulSet replicas to 0.
