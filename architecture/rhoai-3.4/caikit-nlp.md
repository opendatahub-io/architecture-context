## Purpose

**Short**: Caikit-NLP is a Python library providing Natural Language Processing capabilities (text generation, embeddings, classification, reranking) built on the caikit framework, serving as a pluggable runtime library for RHOAI model serving.

**Detailed**: Caikit-NLP implements the caikit framework's "task" and "module" abstractions to provide a comprehensive set of NLP capabilities. It supports text generation (via local HuggingFace models or remote TGIS backends), text embeddings and sentence similarity (via sentence-transformers), document reranking (via both embedding-based and cross-encoder approaches), sequence classification, token classification, and tokenization. Each capability is implemented as a caikit "module" that can be loaded, trained, saved, and served through caikit's gRPC and HTTP runtime servers.

The library is designed with a dual-backend architecture: modules exist in both "local" variants (running inference directly on the serving pod via PyTorch/HuggingFace Transformers) and "remote" variants (delegating inference to a Text Generation Inference Server via gRPC). This allows the same API surface to be served regardless of whether models are co-located or running on dedicated GPU-accelerated TGIS instances. Caikit-NLP also provides training capabilities, including PEFT prompt tuning, multitask prompt tuning, and full model fine-tuning with distributed training support via PyTorch elastic launch and FSDP.

Within the RHOAI platform, caikit-nlp is packaged as a container image (`odh-caikit-nlp-rhel9`) and used as a serving runtime for the KServe/ModelMesh model serving stack. It is not deployed as a standalone service but rather loaded as a library within caikit runtime containers that expose gRPC (port 8085) and HTTP (port 8080) endpoints for model inference.

## Architectural Analysis

Caikit-NLP follows a clean modular architecture built on the caikit framework's module system. Each NLP capability is implemented as a caikit `@module`-decorated class that registers itself with the framework's module registry. This design allows the caikit runtime to automatically discover and expose all registered modules via both gRPC and HTTP servers without any per-module wiring code. The module IDs are UUIDs (e.g., `6655831b-960a-4dc5-8df4-867026e2cd41` for PeftPromptTuning), enabling stable references across versions.

The dual-backend architecture is the most architecturally significant pattern. Text generation modules come in paired implementations: `TextGeneration` / `TextGenerationTGIS` and `PeftPromptTuning` / `PeftPromptTuningTGIS`. The TGIS variants inherit from their local counterparts and override inference methods to delegate to a remote TGIS server via gRPC. The `TGISAutoFinder` model finder enables automatic model discovery — when a model path corresponds to a TGIS-accessible model, the finder returns a `ModuleConfig` pointing to the TGIS module variant. This means model routing (local vs. remote) is determined at load time, not at inference time, which simplifies the serving pipeline. The `TGISGenerationClient` handles all gRPC communication including protobuf construction, error code translation (gRPC status codes to caikit exceptions), and streaming response handling.

The resource abstraction layer (`PretrainedModelBase` and its subclasses `HFAutoCausalLM`, `HFAutoSeq2SeqLM`, `HFAutoSequenceClassifier`) provides a clean separation between model type handling (causal LM vs. seq2seq vs. classifier) and the module logic. Each resource wraps a specific HuggingFace `AutoModel*` class and implements type-specific tokenization, data collation, and trainer creation. This abstraction allows modules like `TextGeneration` to work with any supported model architecture through a uniform interface. The resource classes also handle model type-specific concerns like left-padding for GPT-family models and encoder/decoder transformer counting for seq2seq models.

The embedding module (`EmbeddingModule`) is notably more feature-rich than the text generation modules for inference. It implements seven caikit tasks (Embedding, EmbeddingTasks, SentenceSimilarity, SentenceSimilarityTasks, Rerank, RerankTasks, Tokenization) with sophisticated truncation handling, per-thread tokenizer caching (via `SentenceTransformerWithTruncate`), and optional hardware optimizations (IPEX for Intel, PyTorch 2.0 `torch.compile()`, autocast for mixed-precision inference). The `CrossEncoderModule` provides an alternative reranking approach using cross-encoder models, offering better accuracy at the cost of performance since it processes query-document pairs jointly rather than encoding them independently.

Build hermeticity is notably absent — the Konflux pipeline sets `hermetic: false`, there are no lock files (no `pip.lock`, `uv.lock`, or `rpms.lock.yaml`), and no Hermeto/cachi2 prefetch is configured. Python dependencies are resolved at build time via pip from version ranges in `pyproject.toml`. This is a supply chain risk worth noting, though it is consistent with the non-hermetic build pattern seen in some Python-based RHOAI components. The Dockerfile.konflux and regular Dockerfile are nearly identical, suggesting the Konflux build process for this component has not been heavily customized beyond the standard multi-arch pipeline.
