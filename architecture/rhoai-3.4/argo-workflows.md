## Purpose

**Short**: Argo Workflows is the workflow execution engine that powers Data Science Pipelines in RHOAI, providing a Kubernetes-native workflow controller and executor for running ML pipeline steps as pods.

**Detailed**: Argo Workflows is a container-native workflow engine for orchestrating parallel jobs on Kubernetes. In the RHOAI context, it serves as the execution backend for Data Science Pipelines (DSP), where Kubeflow Pipelines compiles ML pipeline definitions into Argo Workflow CRs that the workflow controller then schedules and executes.

The system consists of two primary components shipped for RHOAI: the **workflow-controller** (a Kubernetes controller that watches Workflow CRs and orchestrates pod execution) and **argoexec** (an executor binary injected into workflow step pods as an emissary that manages artifact I/O, captures outputs, and coordinates container lifecycle within a pod). The upstream Argo Server (CLI/UI/API) is NOT shipped as a separate Konflux image for RHOAI — the Data Science Pipelines operator manages API access through its own server components.

The controller uses Kubernetes informers (not controller-runtime) to watch Workflow CRs, create pods for each workflow step, manage DAG/step dependencies, handle artifact garbage collection, and optionally archive completed workflows to a SQL database. It supports leader election for HA deployments and exposes Prometheus metrics on port 9090 and a health endpoint on port 6060.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|

## Architectural Analysis

Argo Workflows in the RHOAI context operates as a specialized execution engine embedded within the Data Science Pipelines stack. Unlike a standalone Argo Workflows deployment, the RHOAI variant ships only two components (workflow-controller and argoexec) without the Argo Server, CLI, or UI. This is because the DSP operator provides its own API layer and the RHOAI Dashboard provides the user interface. This decomposition is visible in the Konflux Dockerfiles: only `argo-workflowcontroller/Dockerfile.konflux` and `argo-argoexec/Dockerfile.konflux` exist, deliberately omitting the `argocli` build that includes the embedded UI.

The controller is built on raw Kubernetes client-go informers rather than the controller-runtime framework used by most RHOAI operators. It maintains eight informers with a 20-minute resync period and processes work through three rate-limited queues: the main workflow queue (with a custom fixed-interval limiter), a workflow archive queue, and a pod cleanup queue. The controller supports leader election via Kubernetes Leases, with configurable lease duration/renew deadline/retry period via environment variables. Worker counts are tunable via CLI flags, defaulting to 32 workflow workers, 8 cron and archive workers each, and 4 each for TTL and pod cleanup.

A notable architectural detail is the dual-binary approach for argoexec in the Konflux build. The Dockerfile builds both a standard binary (`argoexec`, CGO_ENABLED=0, statically linked) and a FIPS-compliant binary (`argoexec-fips`, CGO_ENABLED=1, GOEXPERIMENT=strictfipsruntime). Both are packaged in the same container image. This allows the runtime to select the appropriate binary based on whether the cluster requires FIPS compliance, without maintaining separate images. The workflow-controller, by contrast, is built only with FIPS flags since it always runs as a platform component.

The TLS configuration deserves security attention. The metrics server uses self-signed ECDSA P-256 certificates generated at startup (`util/tls/tls.go`) with `InsecureSkipVerify: true` — acceptable for internal metrics but worth noting. The SSO configuration also supports `InsecureSkipVerify` for OIDC providers, which is a development convenience that should be disabled in production. The TLS minimum version defaults to TLS 1.2 and is configurable via the `TLS_MIN_VERSION` environment variable, supporting proper FIPS-compliant cipher negotiation when the strictfipsruntime build tag is active.

The CRD design follows Argo's v1alpha1 API group (`argoproj.io/v1alpha1`) with eight CRD kinds. The core Workflow CRD is the primary interface — it defines DAGs and step sequences that the controller decomposes into individual pods. Supporting CRDs like WorkflowTaskResult and WorkflowTaskSet enable the emissary executor pattern where argoexec writes results back to the API server rather than relying on annotations (which have size limits). The WorkflowArtifactGCTask CRD supports automated cleanup of stored artifacts when workflows complete or are deleted.
