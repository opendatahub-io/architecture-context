## Purpose

**Short**: Argo Workflows is the workflow execution engine that powers Data Science Pipelines in Red Hat OpenShift AI, providing container-native workflow orchestration on Kubernetes.

**Detailed**: Argo Workflows implements a Kubernetes-native workflow engine that executes directed acyclic graphs (DAGs) and step-based workflows as sequences of containers. In the RHOAI platform, it serves as the backend execution engine for Data Science Pipelines (DSP), enabling users to define, schedule, and monitor ML training pipelines and data processing workflows.

The component consists of two primary binaries shipped for RHOAI: the **workflow-controller** (which watches Workflow CRDs and orchestrates pod execution) and **argoexec** (a sidecar executor injected into workflow pods to manage artifact I/O, container lifecycle, and result reporting). The upstream project also includes an API server (argo-server) and CLI (argo), but these are not built as separate Konflux images for RHOAI — the DSP API server handles user-facing API interactions instead.

The workflow-controller operates as a leader-elected singleton, watching 8 CRD types from the `argoproj.io` API group. It creates pods for each workflow step, manages artifact garbage collection, supports workflow archival to PostgreSQL/MySQL databases, and exposes Prometheus metrics. The argoexec binary uses the "emissary" execution pattern where it wraps user container commands to capture outputs, manage artifacts, and report results back via WorkflowTaskResult CRDs.

## Architectural Analysis

Argo Workflows in RHOAI follows a **controller + sidecar** architecture pattern. The workflow-controller is a centralized reconciliation loop that watches Workflow CRDs and translates them into pod specifications, while the argoexec binary runs inside every workflow pod as a process wrapper (the "emissary" pattern). This separation means the controller handles orchestration and state management, while argoexec handles the execution boundary — artifact I/O, output capture, and result reporting.

The **emissary executor pattern** is architecturally notable. Rather than using traditional sidecar containers that run alongside the main container, argoexec replaces the user's container command. The init container copies the argoexec binary into a shared emptyDir volume (`/var/run/argo/`), and the main container's entrypoint is overridden to `argoexec emissary <original-command>`. This allows argoexec to wrap the user process, capture its exit code, intercept stdout/stderr, and manage artifact serialization — all without requiring a separate container or a shared PID namespace. Communication between the emissary and the wait container happens via filesystem (signal files, exit code files), not network sockets.

The **FIPS strategy** in this repository is interesting: the argoexec Dockerfile builds two binaries — `argoexec` (statically linked, non-FIPS) and `argoexec-fips` (dynamically linked with `GOEXPERIMENT=strictfipsruntime`). Both are shipped in the same image. The workflow-controller only builds the FIPS variant. This dual-binary approach for argoexec may be related to environments where the FIPS binary's dynamic linking requirements cannot be met (e.g., distroless sidecars), though the standard RHOAI deployment on UBI9 should use the FIPS variant.

The **TLS posture** has known gaps: several internal components use `InsecureSkipVerify: true` — most notably the self-signed certificate path in `util/tls/tls.go` and the agent executor's HTTP client in `workflow/executor/agent.go`. These are mitigated by the fact that in RHOAI, the workflow-controller and argoexec communicate exclusively via Kubernetes API (not direct network connections), and the self-signed certs are only used for the argo-server which is not directly exposed in RHOAI deployments.

The **result reporting mechanism** uses WorkflowTaskResult CRDs as a reliable communication channel from argoexec back to the workflow-controller. This is more robust than the older annotation-based approach (which is still available as a fallback) because CRDs are not subject to the annotation size limits of Kubernetes objects. The controller watches WorkflowTaskResults with a label selector matching the parent workflow, enabling efficient fan-in of results from parallel workflow steps.
