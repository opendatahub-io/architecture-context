## Purpose

**Short**: NeMo Guardrails is a programmable guardrails toolkit for adding safety, security, and topic controls to LLM-based conversational systems, deployed as a FastAPI server in RHOAI.

**Detailed**: NeMo Guardrails, originally developed by NVIDIA, provides a framework for defining and enforcing guardrails on Large Language Model (LLM) interactions. It supports input/output content filtering, jailbreak detection, injection detection, sensitive data detection (PII), hallucination checking, fact-checking, and topic safety enforcement. The toolkit uses a custom domain-specific language called Colang (versions 1.0 and 2.x) to define conversational flows and rail logic.

In the RHOAI distribution, NeMo Guardrails is packaged as a container image (`odh-trustyai-nemo-guardrails-server-rhel9`) running a FastAPI server that exposes an OpenAI-compatible chat completions API with guardrails enforcement. The RHOAI build uses the "opensource" profile, which strips out closed-source guardrail modules (ActiveFence, AI Defense, AutoAlign, Clavata, CrowdStrike, etc.) and retains only open-source modules such as self-check, content safety, regex, injection detection, sensitive data detection, and topic safety. The server supports OpenTelemetry auto-instrumentation for observability when deployed in the platform.

The component pre-downloads ML models (spaCy, Sentence Transformers, FastEmbed, NLTK) at build time to enable air-gapped/disconnected deployment. It supports configurable LLM backends via LangChain (OpenAI, NVIDIA NIM, HuggingFace, etc.) and can persist conversation threads via Redis or in-memory stores.

## Architectural Analysis

NeMo Guardrails follows a **dual-engine architecture** with two distinct execution paths optimized for different use cases. The `IORails` engine provides a fast path for pure content safety checks that don't require LLM generation — it directly evaluates content against safety models without invoking the main LLM. The `LLMRails` engine is the full-featured path that supports dialog flows, retrieval-augmented generation, and complex multi-step rail evaluation. The `Guardrails` facade class automatically selects the appropriate engine based on the configured rails, providing a transparent optimization for simpler safety-only configurations.

The **Colang DSL** (both 1.0 and 2.x versions) is a key architectural differentiator. It provides a declarative language for defining conversational guardrail flows, enabling non-developer personas to configure safety behaviors without Python code changes. The Colang 2.x runtime implements a full state machine with event-driven flow processing, supporting complex multi-turn conversation management. This is architecturally significant because it means the guardrail logic is configuration-driven rather than code-driven, making the system highly extensible.

The **RHOAI-specific build profile** (`opensource`) is enforced at container build time via `scripts/filter_guardrails.py`, which physically removes closed-source guardrail modules from the image. This is a deliberate supply chain decision — rather than runtime feature flags, the closed-source modules (15 of 26 total, including ActiveFence, Pangea, CrowdStrike, Trend Micro, Patronus AI, etc.) are deleted during the Docker build. The remaining 11 open-source modules are self-contained and don't require external API keys. Notably, `jailbreak_detection` is also excluded from the opensource profile due to upstream bugs (tracked in NVIDIA/NeMo-Guardrails#1739).

The **OpenTelemetry integration** follows library best practices — it uses only the OTel API (not SDK), does not modify global tracer state, and relies on the application/platform to configure the SDK. When `OTEL_EXPORTER_OTLP_ENDPOINT` is set, the entrypoint script wraps the server with `opentelemetry-instrument` for automatic instrumentation of FastAPI and httpx. This approach integrates cleanly with the RHOAI observability stack without coupling to any specific collector configuration.

The **build pipeline** supports multi-architecture images (x86_64, arm64, ppc64le) with architecture-specific build stages in the Konflux Dockerfile. The ppc64le build is particularly complex — it compiles PyTorch, tiktoken, onnxruntime, and OpenBLAS from source due to the lack of pre-built wheels for that architecture. This adds significant build complexity and time but enables IBM Power Systems support. The build also pre-downloads ML models (spaCy en_core_web_lg, Sentence Transformers, FastEmbed, NLTK punkt) into the image cache to enable disconnected/air-gapped deployment without runtime network access to model repositories.

A notable **crypto finding**: the codebase uses `hashlib.md5` for embedding cache key generation, but this is a non-cryptographic use (dictionary key hashing for cache lookups) and the code in `nemoguardrails/utils.py` includes a graceful fallback to SHA256 when MD5 is unavailable in FIPS-restricted environments. The actual cryptographic operations are handled by the `cryptography` library (transitively via Presidio) which links against system OpenSSL on UBI 9, inheriting FIPS compliance from the base image.
