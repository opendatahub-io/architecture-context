## Purpose

**Short**: NeMo Guardrails is an open-source toolkit for adding programmable safety guardrails to LLM-based conversational systems, providing input/output validation, content safety, jailbreak detection, and hallucination prevention.

**Detailed**: NeMo Guardrails, originally developed by NVIDIA, is shipped as part of Red Hat OpenShift AI (RHOAI) via a downstream fork maintained by the TrustyAI team. The component provides a FastAPI-based server that acts as a programmable proxy between clients and LLM providers, intercepting requests and responses to apply configurable safety rails. These rails include content safety checks, jailbreak detection (both heuristic and model-based), sensitive data detection/masking via Presidio, hallucination detection through self-consistency checking, and self-check verification of inputs, outputs, and factual claims.

The RHOAI distribution applies several customizations to the upstream NVIDIA project: a `/v1/guardrail/checks` endpoint for standalone rail evaluation without LLM generation, header forwarding for authentication token passthrough to LLM providers, build-time filtering of closed-source guardrail libraries (via `filter_guardrails.py`), a UBI9-based container image with pre-downloaded ML models for air-gapped deployments, and OpenTelemetry auto-instrumentation support. The server exposes an OpenAI-compatible API at `/v1/chat/completions` and supports multiple LLM backends including OpenAI, Anthropic, Azure OpenAI, Cohere, vLLM, NVIDIA NIM, and TRT-LLM.

The component uses NVIDIA's Colang domain-specific language (versions 1.0 and 2.x) to define guardrail flows and policies. Two execution engines are available: IORails (an optimized fast path for standard content/topic/jailbreak flows) and LLMRails (the full Colang runtime supporting arbitrary flow definitions). Configuration is managed through YAML config files and Colang flow definitions, with support for both single-config and multi-config server modes.

## Architectural Analysis

NeMo Guardrails employs a **dual-engine architecture** that balances performance with flexibility. The IORails engine provides an optimized fast path for the most common guardrail patterns (content safety, topic safety, jailbreak detection), executing a simple three-phase pipeline: input rails → LLM generation → output rails. The LLMRails engine, by contrast, supports the full Colang DSL with arbitrary flow definitions, state machines, and multi-turn conversation tracking. The engine selection is controlled by the `NEMO_GUARDRAILS_IORAILS_ENGINE` environment variable and transparently wrapped by the `Guardrails` facade class. This dual-engine approach allows RHOAI deployments to favor the IORails path for production workloads where latency matters, while retaining full customizability for complex guardrail configurations.

The **RHOAI-specific customizations** are architecturally significant. The `/v1/guardrail/checks` endpoint (server/api.py lines 1076-1222) enables standalone rail evaluation — clients can check whether messages would be blocked without invoking the LLM, enabling pre-flight validation patterns. Header forwarding propagates `X-*` headers and Authorization tokens from client requests to LLM provider calls, allowing RHOAI's authentication chain to flow through the guardrails layer. The `filter_guardrails.py` script strips 15 closed-source guardrail libraries (ActiveFence, CrowdStrike, GCP, etc.) at build time via the "opensource" profile, ensuring the RHOAI image contains only open-source dependencies. Notably, jailbreak_detection is also filtered out due to upstream bugs (provider-list.yaml line 21).

The **caching architecture** operates at multiple levels. LLM responses are cached via an LFU (Least Frequently Used) cache with SHA-256 normalized keys, reducing redundant LLM calls for repeated guardrail checks. Embedding vectors are cached via a pluggable store (in-memory, filesystem, or Redis) with configurable key generators. Content safety and jailbreak detection results are cached per-model to avoid re-evaluating identical inputs. Cache statistics (hit rates, evictions) are tracked and can be logged periodically for monitoring. This multi-level caching is essential for production deployments where the same guardrail checks may be evaluated thousands of times.

The **container image** (Dockerfile.konflux) is a complex multi-stage build with 8 stages, primarily to support ppc64le architecture alongside x86_64 and aarch64. The ppc64le path requires building PyTorch, tiktoken, ONNX Runtime, and OpenBLAS from source since pre-built wheels are unavailable. ML models (spacy en_core_web_lg, FastEmbed all-MiniLM-L6-v2, sentence-transformers) are pre-downloaded during build for air-gapped/disconnected deployment — a critical requirement for RHOAI environments. The final image runs as non-root user 1001, uses a virtual environment at `/app/.venv`, and exposes port 8000 via the `entrypoint.sh` script which validates configuration and optionally wraps the server with OpenTelemetry auto-instrumentation.

The **OpenTelemetry integration** follows a thoughtful pattern: the tracing adapter uses the OpenTelemetry API (not SDK), meaning it respects whatever SDK the application's deployment environment configures. The entrypoint script checks for `OTEL_EXPORTER_OTLP_ENDPOINT` and, if set, wraps the server with `opentelemetry-instrument` for automatic FastAPI and httpx instrumentation. This allows RHOAI platform operators to enable distributed tracing without modifying the application code.
