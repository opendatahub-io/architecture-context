## Purpose

**Short**: llm-d is a high-performance distributed inference serving stack that provides optimized container images (vLLM on CUDA/CPU/HPU/ROCm) and Kubernetes deployment recipes for production LLM serving with intelligent routing, prefill/decode disaggregation, and KV-cache management.

**Detailed**: llm-d is a CNCF sandbox project that accelerates distributed LLM inference by integrating industry-standard open technologies: vLLM as the default model server engine, Kubernetes Gateway API (via the Inference Gateway extension) as the control plane and load balancing orchestrator, and Kubernetes as the infrastructure workload control plane. The project produces optimized container images that bundle vLLM with high-performance networking libraries (NIXL, UCX, NVSHMEM, GDRCopy), GPU kernel optimizations (DeepEP, DeepGEMM, FlashInfer), and KV-cache management connectors (LMCache, llm-d filesystem backend). These images support multiple accelerator families: NVIDIA CUDA GPUs, Intel HPU (Gaudi), AMD ROCm GPUs, Intel XPU, and CPU-only deployments.

Beyond container images, llm-d provides a comprehensive set of Kustomize-based deployment recipes and Helm values files that implement "well-lit paths" for common production serving patterns: optimized baseline routing, prefill/decode disaggregation for reduced TTFT, wide expert-parallelism for large MoE models like DeepSeek-R1, tiered KV-cache prefix caching with CPU/storage offload, workload autoscaling, and flow control with multi-tenant fairness. Each guide includes model server manifests (Deployments or LeaderWorkerSets), scheduler configuration (the llm-d Inference Scheduler / EPP), Gateway API recipes for multiple providers (Istio, AgentGateway/kgateway, GKE), monitoring (PodMonitors for Prometheus), and distributed tracing (OpenTelemetry). The repository does not contain controller or operator source code — it is the integration and deployment layer that composes external components (vLLM, Gateway API Inference Extension, NIXL) into tested production configurations.

## Architectural Analysis

The llm-d repository represents an unusual but effective architectural pattern: it is an **integration-first project** rather than a traditional software component. Unlike most RHOAI components that contain operator or service source code, llm-d is primarily a curated collection of container image build recipes and Kubernetes deployment manifests. The actual inference engine (vLLM), routing intelligence (llm-d-inference-scheduler), and transfer protocol (NIXL) live in separate repositories — this repo composes them into tested, production-ready configurations. This separation follows the project's principle of "composition over configurability" and "respecting upstreams."

The container image architecture employs a sophisticated multi-stage build pattern, particularly in `Dockerfile.cuda`. The builder stage compiles native libraries (GDRCopy, UCX, NIXL, NVSHMEM, DeepEP, DeepGEMM, FlashInfer, LMCache) from source with CUDA support, then copies only runtime artifacts to a minimal CUDA runtime image. The use of `sccache` for build acceleration and UV for Python dependency management reflects production build engineering maturity. The GB200/Blackwell variant uses Docker multi-target builds (`--target runtime-gb200`) to swap in a different DeepEP wheel without duplicating the entire build, which is an elegant approach to hardware-specific optimization.

The deployment recipes use a **layered Kustomize composition model** that deserves attention. The `guides/recipes/` directory provides reusable bases (generic decode Deployment, prefill Deployment, ServiceAccount, Gateway, monitoring), and each well-lit path guide (optimized-baseline, pd-disaggregation, wide-ep-lws, etc.) overlays accelerator-specific patches (GPU vendor, count, model arguments) and platform-specific patches (GKE, CoreWeave, OpenShift). The scheduler configuration is entirely Helm-driven via layered values files that compose EPP plugins (scorers, filters, pickers) into scheduling profiles. This is a clean separation: Kustomize for infrastructure manifests, Helm for EPP behavioral configuration.

The networking architecture reveals the critical role of NIXL in the disaggregated serving pattern. In P/D mode, the routing sidecar (`llm-d-routing-sidecar:v0.7.1`) intercepts requests on port 8000, the vLLM backend listens on 8200, and NIXL transfers KV-cache blocks between prefill and decode pods over port 5600. For wide expert-parallelism with LeaderWorkerSets, additional coordination happens via gRPC on port 5555 (data-parallel RPC) and NVSHMEM over InfiniBand for GPU-to-GPU all-to-all communication. The ZMQ event port 5556 is used exclusively by the precise-prefix-cache-aware scheduler to index vLLM's internal KV-cache state in real time. This creates a multi-layered network topology: HTTP for API traffic, RDMA for data transfer, and ZMQ for metadata — all without encryption, relying entirely on Kubernetes network isolation.

A notable security observation is that the entire inference data path is unencrypted HTTP within the cluster. The model server, routing sidecar, EPP, and Gateway all communicate in plaintext. This is consistent with high-performance inference serving where TLS overhead is unacceptable on the data path, but it means network segmentation (NetworkPolicies, namespaces, or service mesh mTLS) is the only security boundary. The ServiceAccount configuration (`automountServiceAccountToken: false`) and OpenShift-specific security contexts (dropped capabilities, read-only root filesystem, non-root user) demonstrate defense-in-depth at the pod level. For RHOAI integration, the platform operator would typically add kube-rbac-proxy sidecars or Gateway-level authentication, but this upstream repository does not implement either.
