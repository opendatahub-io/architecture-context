## Purpose

**Short**: llm-d is a Kubernetes-native distributed inference serving stack that provides optimized container images, deployment manifests, and well-lit path guides for serving large language models at production scale across multiple hardware accelerators.

**Detailed**: llm-d is the integration repository for a CNCF sandbox project founded by Red Hat, Google Cloud, IBM Research, CoreWeave, and NVIDIA. Rather than containing application source code, this repository serves as the central orchestration point that defines how vLLM-based model server container images are built for multiple hardware platforms (NVIDIA CUDA, AMD ROCm, Intel Gaudi HPU, CPU) and provides comprehensive Kustomize-based deployment manifests for Kubernetes.

The repository produces five container image variants (`llm-d-cuda`, `llm-d-cpu`, `llm-d-hpu`, `llm-d-rocm`, `llm-d-rdma-tools`) and defines the deployment topology through a layered kustomize structure organized by deployment pattern (optimized baseline, prefill/decode disaggregation, wide expert-parallelism, tiered prefix caching) and hardware backend (NVIDIA GPU, AMD GPU, Intel XPU/HPU, Google TPU, CPU). The deployment architecture centers on three core components: the **Model Server** (vLLM/SGLang running on accelerators), the **Router** (an intelligent Envoy-based proxy with an Endpoint Picker Plugin for KV-cache-aware and load-aware routing), and the **Gateway** (Kubernetes Gateway API integration via AgentGateway, Istio, or GKE). These components are deployed via Helm charts and Kustomize overlays and communicate through well-defined ports and protocols documented in the guides.

llm-d is consumed by Red Hat OpenShift AI (RHOAI) as the model serving runtime, where the CUDA container image is rebuilt via Konflux pipelines and deployed as part of the RHOAI inference stack. The project also supports standalone Kubernetes, GKE, and CoreWeave deployments.

## Architectural Analysis

**Integration-First Architecture**: llm-d is architecturally distinctive in that it contains no application source code itself. Instead, it serves as a sophisticated integration layer that composes numerous open-source components (vLLM, PyTorch, CUDA, NVSHMEM, UCX, NIXL, FlashInfer, DeepEP, LMCache) into production-ready container images and provides structured deployment patterns via Kustomize and Helm. This "composition over code" approach aligns with the project's principle that "components connect at API boundaries" and allows rapid adoption of upstream improvements without maintaining forks. The `docker/common-versions` file centralizes all dependency version pins, making version bumps atomic and auditable.

**Layered Kustomize Design**: The deployment manifests follow a well-structured layering pattern: `guides/recipes/` provides hardware-agnostic base resources (Deployments, ServiceAccounts, Gateways), while `guides/{guide-name}/modelserver/{hardware}/` applies guide-specific and hardware-specific overlays. This allows the same base Deployment template to be customized for NVIDIA GPUs, AMD GPUs, Intel Gaudi HPUs, Google TPUs, and CPUs with minimal duplication. The kustomize component mechanism is used for optional features like monitoring (PodMonitor injection). However, the sheer number of permutations (7 guides x 8+ hardware backends x 3+ cloud providers) creates a combinatorial explosion of kustomization files that may be challenging to maintain as the project scales.

**Security Model — Plaintext by Design**: The model server containers expose plaintext HTTP on port 8000 with no authentication or TLS. This is an intentional architectural decision: security (TLS termination, authentication, authorization) is delegated entirely to the Gateway provider and cluster network policies. The `automountServiceAccountToken: false` setting on the model server ServiceAccount demonstrates deliberate least-privilege design. However, the KV-cache transfer protocol (NIXL on port 5600) and RDMA communication (NVSHMEM/UCX) between pods are completely unauthenticated and unencrypted, relying solely on network-level isolation. In multi-tenant environments, this could be a security concern if pod-to-pod network policies are not properly configured.

**Routing Intelligence**: The EPP (Endpoint Picker Plugin) is the most architecturally significant component in the llm-d stack, despite being deployed from a separate repository (llm-d-inference-scheduler). It implements a Filter-Score-Pick pipeline with composable plugins (kv-cache-utilization-scorer, queue-depth-scorer, prefix-cache-scorer, latency-scorer) that transform Kubernetes pod selection from naive round-robin into an LLM-aware scheduling decision. The EPP communicates with Envoy via the ext-proc gRPC protocol, which means all routing intelligence is decoupled from the data plane. This is a clean separation that allows the EPP to be upgraded independently of the proxy.

**Hardware Abstraction Gap**: While the Kustomize overlays provide excellent hardware abstraction for deployment configuration, the Dockerfiles are entirely hardware-specific with no shared build logic. The CUDA Dockerfile (617 lines) shares nothing with the ROCm Dockerfile (158 lines) or HPU Dockerfile (84 lines). Each hardware backend has its own dependency tree, version pins, and build procedures. The `docker/common-versions` file only covers CUDA/CPU builds. This means a vLLM version bump requires coordinated changes across 4+ Dockerfiles with different build systems — a maintenance risk as the project adds more hardware backends.
