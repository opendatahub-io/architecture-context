## Purpose

**Short**: Kubernetes operator that manages the lifecycle of Model Registry instances and the Model Catalog service for the RHOAI/ODH platform.

**Detailed**: The Model Registry Operator manages `ModelRegistry` custom resources (CRD group `modelregistry.opendatahub.io`) across two API versions: v1alpha1 (deprecated) and v1beta1 (storage version). For each ModelRegistry CR, the operator provisions a complete stack of Kubernetes resources including a REST API deployment backed by a configurable database (PostgreSQL or MySQL), authentication via kube-rbac-proxy sidecars, OpenShift Routes for external access, RBAC roles and bindings, and NetworkPolicies for isolation.

The operator also manages a Model Catalog subsystem — a centralized catalog service that aggregates model metadata from multiple sources (default, user-managed, and labeled ConfigMaps). The catalog runs its own PostgreSQL instance, kube-rbac-proxy authentication, and admin RBAC tied to the platform's Auth CR. The operator handles automatic migration from OAuth proxy to kube-rbac-proxy authentication, CRD storage version migration (v1alpha1 to v1beta1 via Kubernetes StorageVersionMigration API or manual re-read/re-write fallback), and supports async model artifact upload jobs (S3-to-OCI and URI-to-OCI).

The operator detects cluster capabilities at startup (OpenShift, BYOIDC mode, ODH Auth API) and adapts its behavior accordingly — creating OpenShift-specific resources like Routes, Groups, and serving certificates only when running on OpenShift, and using different RBAC strategies for BYOIDC vs. traditional OpenShift authentication.

## Architectural Analysis

The Model Registry Operator follows a well-structured controller-runtime pattern with two distinct reconcilers — one for per-instance ModelRegistry CRs and one for the singleton ModelCatalog subsystem. The ModelRegistry controller creates a complete resource stack per instance using Go templates (`*.yaml.tmpl` files), applying them via a `TemplateApplier` that supports OpenShift-conditional logic. The ModelCatalog controller uses a fixed-request deduplication pattern where all watched resources map to a single reconcile key (`model-catalog` in the target namespace), preventing reconciliation storms from cascading resource changes.

A notable design decision is the automatic migration from OAuth proxy to kube-rbac-proxy authentication. When a reconcile detects `Spec.OAuthProxy != nil`, it calls `MigrateOAuthProxyToKubeRBACProxy()` on the CR, updates it, and requeues. This in-place migration approach means existing ModelRegistry instances seamlessly transition to kube-rbac-proxy without user intervention. The shared resource naming convention (`{name}-auth-delegator`, `{name}-https`, `{name}-https-route`) between OAuth proxy and kube-rbac-proxy simplifies this transition — the same ClusterRoleBinding, Route, and NetworkPolicy names are reused with new configuration. The v1beta1 webhook also performs this migration during defaulting, ensuring CRs are migrated even before the controller processes them.

The cluster capability detection system (`ClusterCapabilities` in `util.go`) adapts the operator's behavior at startup by probing for OpenShift-specific APIs. The `IsBYOIDC()` method (returns true when `IsOpenShift && !HasUserAPI`) handles the emerging BYOIDC authentication mode where OpenShift Routes are available but the internal User API is not. In BYOIDC mode, the operator skips Group creation and uses external IdP groups in RoleBindings instead. The `HasAuthAPI` flag enables integration with the ODH Auth CR for admin group discovery, gracefully degrading when the platform API is unavailable.

The storage migration subsystem uses a two-strategy approach: it first attempts Kubernetes-native `StorageVersionMigration` (SVM) API, which delegates the migration to the API server. If SVM is not available (older clusters), it falls back to a manual strategy that re-reads each ModelRegistry in the target version (triggering the conversion webhook), annotates it with migration markers, and writes it back to force etcd rewrite. Both strategies handle partial failures — the manual strategy continues with remaining resources if one fails. The migration manager starts with a 30-second delay to allow the webhook server to become ready.

The ModelCatalog subsystem introduces PostgreSQL auto-provisioning with idempotent secret management — the `createOrUpdatePostgresSecret()` method generates a random 16-byte password on first creation but preserves existing passwords on subsequent reconciles. It also handles migration from pre-3.0 catalogs by removing deprecated `default_catalog` entries from user-managed source ConfigMaps. The catalog supports dynamic source discovery via labeled ConfigMaps (`opendatahub.io/catalog-source=true`) with strict name validation (max 55 characters for volume naming, DNS label compliance). Admin access is dynamically toggled based on the Auth CR — if admin groups are configured, admin Role/RoleBinding are created; if removed, they are cleaned up.
