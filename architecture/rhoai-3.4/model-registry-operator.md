## Purpose

**Short**: Manages the lifecycle of Model Registry and Model Catalog custom resources, deploying and configuring the model-registry REST service, backing database, authentication proxy, and catalog infrastructure on Kubernetes/OpenShift.

**Detailed**: The Model Registry Operator is a Kubernetes operator built with controller-runtime that manages `ModelRegistry` custom resources (CRD group `modelregistry.opendatahub.io`). For each ModelRegistry CR, the operator deploys a model-registry REST API server (backed by either PostgreSQL or MySQL), an optional kube-rbac-proxy sidecar for HTTPS/authentication, OpenShift Routes for external access, RBAC resources for user/group authorization, NetworkPolicies for traffic isolation, and optionally auto-provisions a PostgreSQL database with persistent storage.

The operator also includes a ModelCatalog controller that manages a separate catalog service and its own PostgreSQL database, providing curated model metadata sourced from multiple ConfigMaps (user-provided, MCP servers, and built-in Red Hat AI catalogs). The catalog supports admin RBAC via groups configured through the platform Auth CR.

The operator supports two CRD versions: v1beta1 (storage version, conversion hub) and v1alpha1 (deprecated spoke). It includes admission webhooks for defaulting and validation, automatic migration from Istio-based networking to kube-rbac-proxy, and storage version migration between CRD versions using either Kubernetes StorageVersionMigration API or manual re-read/re-write fallback. The operator detects cluster capabilities at startup (OpenShift, BYOIDC, config/user/auth APIs) and conditionally creates platform-specific resources.

## Architectural Analysis

The Model Registry Operator follows a clean controller-runtime architecture with two independent reconciliation loops: one for ModelRegistry CRs and one for ModelCatalog infrastructure. The separation is meaningful — the catalog controller watches a fixed "default-modelregistry" resource and manages catalog-specific infrastructure (separate PostgreSQL, separate RBAC, multiple ConfigMap sources), while the registry controller manages per-instance infrastructure. Both controllers share the same template rendering engine (`config.ParseTemplates()`) and resource management utilities (`ResourceManager.CreateOrUpdate()` with 3-way diff via banzaicloud/k8s-objectmatcher).

A notable architectural pattern is the operator's progressive auth migration strategy. The codebase tracks three generations of authentication: Istio service mesh (v1alpha1, fully deprecated), OAuth proxy (v1alpha1→v1beta1 transition), and kube-rbac-proxy (v1beta1 current). Each generation migrates forward automatically through the defaulting webhook — `v1alpha1.ReplaceIstioWithOAuthProxy()` converts Istio config to OAuth, and `v1beta1.MigrateOAuthProxyToKubeRBACProxy()` converts OAuth to kube-rbac-proxy. The CRD validation rule `rule: '!(has(self.oauthProxy) && has(self.kubeRBACProxy))'` prevents both proxies from being configured simultaneously, enforcing the migration path. This is a well-designed backward-compatibility story that allows existing CRs to transparently upgrade through admission webhooks.

The storage version migration system is thoughtfully designed with a strategy pattern: the primary strategy uses the Kubernetes StorageVersionMigration API (SVM), which is a native Kubernetes mechanism for re-writing stored objects. The fallback is a manual re-read/re-write approach that reads each resource in the target version (triggering the conversion webhook) and writes it back with a migration annotation marker. The migration monitor starts 30 seconds after the operator boots and runs once, checking the CRD's `status.storedVersions` to detect whether migration is needed. The idempotent annotation markers (`storage-migration.opendatahub.io/migrated-v1alpha1-to-v1beta1`) prevent duplicate migrations.

The operator's cluster capability detection (`DetectClusterCapabilities`) gracefully handles BYOIDC (Bring Your Own Identity Provider) clusters where `user.openshift.io` APIs are absent despite being on OpenShift. The `IsBYOIDC()` function returns true when `IsOpenShift && !HasUserAPI`, and the operator adjusts behavior accordingly — for example, skipping OpenShift Group creation. This is a practical design for hybrid environments. The `HasAuthAPI` capability checks for `services.platform.opendatahub.io` CRDs, which indicates the full RHOAI platform is installed (as opposed to standalone ODH).

The template-based resource generation is thorough, with 33 embedded YAML templates covering every resource the operator creates. The conditional kube-rbac-proxy sidecar injection in the deployment template is particularly well-structured — it adds a second container with TLS certificates, ConfigMap-based authorization config, and appropriate security context. The NetworkPolicy templates correctly scope traffic: the postgres network policy allows ingress only from pods with the `app: {name}` label on port 5432, while the kube-rbac-proxy network policy allows ingress from the OpenShift ingress namespace on the proxy port. The operator also manages cache filtering via `cache.ByObject` with label selectors (`app.kubernetes.io/created-by: model-registry-operator`) to minimize API server load, with a special exception for Secrets which are read directly (not cached) since they are user-provided, not operator-created.
