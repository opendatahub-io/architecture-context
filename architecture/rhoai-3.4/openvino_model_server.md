## Purpose

**Short**: OpenVINO Model Server is a high-performance AI model inference server that exposes models via gRPC and REST APIs with support for LLM text generation, embeddings, reranking, image generation, and audio processing.

**Detailed**: OpenVINO Model Server (OVMS) is a C++-based inference serving system optimized for Intel architectures. It serves AI/ML models through multiple API protocols: TensorFlow Serving v1, KServe v2, OpenAI-compatible v1 (chat completions, completions, embeddings), Cohere-compatible reranking, and custom v3 endpoints for image generation. The server uses the OpenVINO runtime for model execution and supports models stored locally, on Azure Blob Storage, AWS S3, Google Cloud Storage, or pulled directly from HuggingFace Hub.

The architecture is built around a modular pipeline system using MediaPipe graphs for composable inference workflows. This enables complex multi-model pipelines through a Directed Acyclic Graph (DAG) scheduler, custom node plugins, and Python-scriptable processing steps. The server supports both traditional vision/NLP model serving and modern generative AI workloads including LLM continuous batching, visual language models, text-to-image generation, speech-to-text, and text-to-speech.

For RHOAI, OVMS is built from source on UBI 9 with LTO optimization and GPU support via Intel compute drivers. It is compiled using Bazel with OpenVINO, OpenVINO GenAI, and OpenVINO Tokenizers built from the `releases/2026/0` branch of the `opendatahub-io` forks. The final container image runs as a non-root user (`ovms`, UID 5000) on `ubi9-minimal`.

## Architectural Analysis

OpenVINO Model Server follows a modular monolith architecture where all inference capabilities are compiled into a single binary. The server's lifecycle is managed through a module system (defined in `src/module.hpp`) with nine core modules started in sequence: Profiler, Metrics, gRPC Server, HTTP Server, ServableManager, CAPI, HuggingFace Pull, Config Export, and Servables Config Manager. Each module can be independently configured and disabled.

The most architecturally significant design choice is the use of MediaPipe graphs as the execution framework for generative AI workloads. Rather than implementing direct handler code for LLM/embedding/reranking endpoints, OVMS routes these requests through MediaPipe calculator graphs. Each GenAI capability (LLM, embeddings, reranking, image generation, audio) is implemented as a MediaPipe calculator that can be composed into pipelines. This enables flexible multi-model workflows (e.g., RAG pipelines combining embeddings + reranking + LLM generation) through graph configuration rather than code changes. The `HttpLLMCalculator` (src/llm/http_llm_calculator.cc) demonstrates this pattern: it wraps the OpenVINO GenAI continuous batching engine in a MediaPipe calculator with streaming support via a loopback control signal.

The inference serving layer supports three distinct API protocols -- TensorFlow Serving v1, KServe v2, and OpenAI-compatible -- through separate frontend modules (tfs_frontend, kfs_frontend, http_frontend). Each frontend has its own validation, serialization, and deserialization implementations, leading to significant code duplication (visible in the BUILD file where `tfs_validation`, `kfs_validation`, and `capi_validation` are parallel implementations). The REST API is served by the Drogon HTTP framework with an alternative net_http (TensorFlow Serving) backend available via build configuration.

A notable security consideration is that OVMS provides no built-in TLS support. The gRPC server uses `InsecureServerCredentials()` and the HTTP server runs plaintext. This is by design -- TLS termination is delegated to the orchestration layer (Istio sidecar proxy in RHOAI, or the optional NGINX mTLS proxy in `extras/nginx-mtls-auth/`). API authentication is limited to a simple API key mechanism (`--api_key_file` CLI flag). For RHOAI deployments, authentication and authorization are handled by KServe and the platform's auth infrastructure (kube-rbac-proxy, OAuth).

The build system is notable for its complexity. OVMS uses Bazel 6.1.1 with OpenVINO, GenAI, and Tokenizers built from source on RHEL (ov_use_binary=0). The Konflux build (`Dockerfile.konflux`) is explicitly non-hermetic (`hermetic: false` in the Tekton pipeline), downloading and compiling approximately 50+ external dependencies including Boost, Azure SDK, OpenCV, pugixml, and the entire OpenVINO stack. The build takes up to 8 hours (Tekton timeout) and uses LTO with `-march=haswell` for performance optimization. SDL (Secure Development Lifecycle) compiler flags are applied including stack protector, FORTIFY_SOURCE, and format security warnings.
