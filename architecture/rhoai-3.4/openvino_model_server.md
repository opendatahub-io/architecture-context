## Purpose

**Short**: OpenVINO Model Server is a high-performance C++ inference serving system that hosts AI models and exposes them via gRPC, REST, and C APIs using the OpenVINO runtime for optimized execution on Intel hardware.

**Detailed**: OpenVINO Model Server (OVMS) provides a production-grade model serving infrastructure for Red Hat OpenShift AI. It implements multiple API protocols — TensorFlow Serving v1, KServe v2, OpenAI-compatible (chat completions, embeddings, reranking, image generation, audio), and a native C API — to serve a broad range of AI workloads. The server is implemented in C++ for maximum throughput and uses the OpenVINO runtime for inference, with hardware acceleration support for Intel CPUs, GPUs (via OpenCL), and NPUs.

OVMS supports sophisticated LLM serving through a continuous batching pipeline built on the OpenVINO GenAI SDK, enabling efficient text generation with streaming, speculative decoding, tool calling, and vision-language model support. Beyond LLMs, it serves traditional ML models (classification, detection, segmentation), embeddings, reranking, image generation (diffusion pipelines), audio (speech-to-text and text-to-speech), and custom DAG pipelines via MediaPipe graphs. Models can be loaded from local storage, Amazon S3, Google Cloud Storage, Azure Blob Storage, or pulled directly from HuggingFace Hub.

Within the RHOAI platform, OVMS serves as the default runtime for KServe InferenceService workloads alongside other serving runtimes like vLLM and Caikit. It is deployed as a container within model serving pods orchestrated by KServe and the RHOAI operator.

## Architectural Analysis

OpenVINO Model Server follows a modular architecture with carefully orchestrated initialization. The server creates a singleton `Server` instance that bootstraps nine modules in dependency order: Python Interpreter → Profiler → Metrics → ServableManager → CAPI → gRPC Server → HTTP Server, with the ServableManager started last to ensure all infrastructure is ready before models begin loading. This module ordering is critical — the HTTP module depends on gRPC, Servable, and Metrics modules; the Servable module depends on Metrics and Python; and the CAPI module requires MediaPipe to function.

The LLM serving architecture is particularly sophisticated. The continuous batching engine (`LLMExecutor`) runs a dedicated worker thread that calls `pipeline.step()` in a tight loop, processing multiple requests concurrently through the OpenVINO GenAI `ContinuousBatchingPipeline`. Requests are added via `addRequestToPipeline()`, which returns a `GenerationHandle` immediately — enabling async and streaming response patterns. The worker thread uses condition variables for efficient wake-up when new requests arrive. This design mirrors production LLM serving systems like vLLM's continuous batching approach but runs entirely in C++ with the OpenVINO backend. Model-specific tool calling parsers (Llama 3, Mistral, Phi 4, Hermes 3, Qwen 3, Devstral) enable function calling across different model architectures, and a reasoning parser supports chain-of-thought extraction for reasoning models.

The storage subsystem uses a clean factory pattern: `FileSystemFactory` inspects URL prefixes (`s3://`, `gs://`, `azureblob://`, `azurefs://`) to instantiate the correct backend. The `ModelManager` orchestrates a model lifecycle with three threads: a watcher thread polling for config changes every second (with MD5-based change detection), a cleaner thread for resource cleanup, and a sequence cleaner for stateful model idle sequence management. Hot-reload is supported — the manager computes version diffs (add, reload, retire) and applies them without server restart.

A notable security design choice is that OVMS provides **no built-in TLS**. The gRPC server explicitly uses `InsecureServerCredentials()`, and the Drogon HTTP server binds without encryption. This is intentional for the RHOAI deployment model: TLS termination is handled by the platform's ingress layer (Gateway API with Envoy, or OpenShift Routes) and authentication by `kube-rbac-proxy` sidecars injected into the pod. The optional `--api_key_file` flag provides application-level Bearer token auth for the V3 (OpenAI-compatible) endpoints, but this is a convenience feature, not a security boundary. The Tekton PipelineRun explicitly sets `hermetic: false`, meaning the Konflux build fetches dependencies from the network — expected given the complexity of building OpenVINO from source with Bazel, but a supply chain consideration.

The build system is complex: Bazel manages the C++ compilation with over 40 build flags controlling features (MediaPipe, Python, GPU support), optimizations (LTO, march=haswell), and distribution (Red Hat vs Ubuntu). The Konflux build args file (`.konflux/build-args.conf`) enables LTO with haswell architecture targeting, GPU support, and both MediaPipe and Python subsystems. OpenVINO, OpenVINO Tokenizers, and OpenVINO GenAI are all built from source from the `opendatahub-io` organization's `releases/2026/0` branches, creating a tightly integrated binary with specific performance optimizations for the RHOAI platform.
