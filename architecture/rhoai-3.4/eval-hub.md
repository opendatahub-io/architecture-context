## Purpose

**Short**: REST API service for orchestrating LLM evaluations across multiple evaluation framework backends, tracking experiments via MLflow, and running evaluation jobs as Kubernetes batch Jobs.

**Detailed**: Eval-hub is the central evaluation orchestration service in the RHOAI platform. It provides a RESTful HTTP API (OpenAPI 3.1) that enables users to create, manage, and monitor LLM evaluation jobs against multiple backend frameworks including lm-evaluation-harness, RAGAS, Garak, GuideLLM, LightEval, and MTEB. The service supports a Bring Your Own Framework (BYOF) model via the companion eval-hub-sdk, where adapter authors implement a single Python class to integrate new evaluation backends.

The service manages evaluation lifecycle through a pluggable runtime abstraction — Kubernetes (production) and local process (development). In Kubernetes mode, it creates batch/v1 Jobs with a multi-container pod pattern: a framework adapter container that runs the evaluation, a sidecar container (KEP-753 native sidecar) that reverse-proxies to eval-hub and MLflow with auth token injection, and an optional init container for S3 test-data download. The eval-hub API server itself is deployed as a single-replica Deployment managed by the TrustyAI Service Operator via an EvalHub custom resource.

Persistence uses a SQL storage layer with SQLite for development and PostgreSQL for production. Multi-tenancy is implemented through X-Tenant header propagation, Kubernetes namespace isolation for job pods, and tenant-scoped SQL queries. Authentication and authorization leverage Kubernetes TokenReview and SubjectAccessReview APIs mapped to custom API groups (trustyai.opendatahub.io, mlflow.kubeflow.org).

## Architectural Analysis

Eval-hub follows a clean Go service architecture with strict separation between entry points (`cmd/`), internal application logic (`internal/`), and shared API types (`pkg/api`). The three binaries (eval-hub, eval-runtime-sidecar, eval-runtime-init) are built from a single repository and shipped in a single container image, with the pod command determining which binary executes. This design minimizes image sprawl while keeping each binary focused on a single responsibility.

The multi-container pod pattern for evaluation jobs is well-designed. The sidecar runs as a KEP-753 native sidecar (init container with `RestartPolicy=Always`), which ensures it starts before the adapter and survives adapter completion. The sidecar handles three distinct proxy targets — eval-hub API, MLflow tracking, and OCI registry — each with independent TLS configuration, auth token resolution, and HTTP client settings. Auth tokens are cached with configurable TTL (default 5 minutes) using atomic pointers for thread safety, and token files are re-read on cache miss to support Kubernetes projected token rotation.

The authorization model is noteworthy: eval-hub maps HTTP endpoints to Kubernetes resource attributes via a YAML config file (config/auth.yaml), using template-based rewrites where the X-Tenant header determines the namespace for SubjectAccessReview checks. This delegates authorization decisions to the cluster's RBAC system while maintaining custom API groups (trustyai.opendatahub.io, mlflow.kubeflow.org). The design means that RBAC policies for evaluation access are standard Kubernetes Role/RoleBinding resources, making them auditable and manageable with standard K8s tooling.

The storage layer uses an interface-based design with SQL implementations for SQLite and PostgreSQL. Entities are stored as JSON documents within SQL tables (TEXT for SQLite, JSONB for PostgreSQL), providing schema flexibility while leveraging SQL for indexing, filtering, and transactions. The builder pattern (`WithTenant`, `WithOwner`, `WithContext`, `WithLogger`) creates immutable copies of the storage instance per request, ensuring tenant isolation without global state. System resources (built-in providers and collections shipped with the image) are loaded at startup and reloaded on config file changes via fsnotify.

A potential concern is that the sidecar's OCI registry proxy implements a full OCI auth challenge flow (initiate challenge, parse Bearer realm, exchange credentials for token), which is complex and security-sensitive. The token caching and response header consumption (stripping WWW-Authenticate, rewriting Location headers) suggest this was designed for production registry interaction. The OCI credentials come from mounted Kubernetes secrets in Docker config.json format, supporting both `username:password` and base64-encoded `auth` fields.
