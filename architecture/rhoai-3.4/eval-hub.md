## Purpose

**Short**: A lightweight REST API service for orchestrating LLM evaluations across multiple evaluation backends, tracking experiments via MLflow, and managing benchmark collections and providers.

**Detailed**: EvalHub is the central evaluation orchestration service for Red Hat OpenShift AI. It provides a REST API that manages evaluation jobs, benchmark collections, and evaluation providers. When a user submits an evaluation job, EvalHub validates the request, creates Kubernetes batch Jobs with sidecar containers to execute the evaluations, and tracks results through MLflow experiment tracking.

The service supports multiple evaluation frameworks including lm-evaluation-harness (180+ benchmarks), LightEval, Garak (security testing), GuideLLM (performance), and IBM CLEAR via a pluggable provider architecture. Each provider defines its own container image and runtime configuration. EvalHub creates Kubernetes Jobs with a three-container pod pattern: an init container for S3 test data download, the evaluation adapter container running the framework, and a sidecar proxy that handles authentication and routing to upstream services (eval-hub API callbacks, MLflow tracking, OCI registries).

EvalHub is managed by the TrustyAI Service Operator via an `EvalHub` custom resource (`trustyai.opendatahub.io/v1alpha1`). Authentication uses Kubernetes TokenReview for identity verification and SubjectAccessReview for authorization, mapping REST API endpoints to Kubernetes RBAC resources in the `trustyai.opendatahub.io` and `mlflow.kubeflow.org` API groups. Multi-tenancy is enforced via the `X-Tenant` header, which maps to Kubernetes namespaces for resource isolation.

## Architectural Analysis

EvalHub follows a clean service architecture with clear separation of concerns. The codebase uses Go's standard `net/http` router (no framework dependencies) with a middleware chain for authentication, authorization, metrics, CORS, and request context. The storage layer is abstracted behind interfaces, enabling SQLite for development and PostgreSQL for production without code changes. The runtime layer similarly abstracts job execution, with Kubernetes and local process implementations sharing a common interface.

The three-container pod pattern (init + adapter + sidecar) is a sophisticated deployment design. The sidecar acts as a transparent reverse proxy that eliminates the need for evaluation framework adapters to handle authentication, TLS, or service discovery. Adapters simply call `localhost:8080` and the sidecar routes to the correct upstream service based on URL path patterns. The sidecar uses atomic token caching with 5-minute TTL for eval-hub and MLflow tokens, and OCI registry tokens use a challenge/response flow with automatic refresh. This design means new evaluation frameworks can be added by simply providing a container image and YAML provider configuration — no auth integration required.

The authorization model maps REST API endpoints to Kubernetes RBAC resources using a template-based rules engine. The `X-Tenant` header is extracted and used as the Kubernetes namespace for SubjectAccessReview checks, enabling multi-tenant isolation through standard Kubernetes RBAC. This approach means existing Kubernetes RBAC tools (oc adm policy, kubectl create rolebinding) work natively for managing eval-hub access. The SAR authorization cache (5-minute allow, 30-second deny) balances performance with security responsiveness.

A notable design decision is the Kueue integration via the `kueue.x-k8s.io/queue-name` label on Jobs. This allows evaluation workloads to participate in cluster-wide fair-scheduling and resource quotas without any code changes to EvalHub — Kueue's webhook intercepts the Job creation and gates it through the queue. The `queueKind` and `queueName` fields in the job configuration enable opt-in scheduling per evaluation request. This is critical for GPU-bound evaluation workloads that compete for scarce accelerator resources.

The FIPS compliance story has two faces: the Konflux build (`Dockerfile.konflux`) correctly enables `GOEXPERIMENT=strictfipsruntime` with `CGO_ENABLED=1` for FIPS-validated cryptography, while the upstream `Containerfile` uses `CGO_ENABLED=0` (static linking, no FIPS). The application code uses `tls.VersionTLS12` as the minimum TLS version and supports custom CA certificates per upstream service. The `InsecureSkipVerify` option exists for development but defaults to `false`. No non-FIPS cryptographic primitives (MD5, RC4, DES) were detected in the codebase.
