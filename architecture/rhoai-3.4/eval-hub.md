## Purpose

**Short**: A lightweight REST API service that orchestrates LLM evaluations across multiple backends, tracks experiments via MLflow, and runs natively on OpenShift.

**Detailed**: EvalHub is a Kubernetes-native evaluation orchestration control plane that routes evaluation requests to frameworks such as lm-evaluation-harness, RAGAS, Garak, GuideLLM, LightEval, and MTEB. It provides a unified REST API under `/api/v1` for creating, listing, and managing evaluation jobs, benchmark collections, and evaluation providers. The service uses Go's standard `net/http` router with structured zap logging, Prometheus metrics, and OpenTelemetry tracing support.

On OpenShift/Kubernetes, EvalHub creates Kubernetes batch Jobs for each benchmark evaluation. Each Job runs an adapter container image specific to the evaluation framework, with a ConfigMap holding the job specification. The service supports multi-tenant operation through X-Tenant header-based namespace isolation, with authentication via Kubernetes TokenReview and authorization via SubjectAccessReview (SAR). Experiment tracking is integrated with MLflow, using projected ServiceAccount tokens for authentication. The service is deployed and managed by the TrustyAI Service Operator through a custom `EvalHub` CRD (`trustyai.opendatahub.io/v1alpha1`).

The container image is built using FIPS-compliant UBI9 base images with `GOEXPERIMENT=strictfipsruntime` for FIPS compliance. The service supports both SQLite (in-memory, for development) and PostgreSQL (production) as storage backends, with a pluggable storage layer abstraction.

### Dynamically Created Kubernetes Resources

The K8s runtime creates the following resources per evaluation benchmark:

| Resource | API Group | Purpose |
|----------|-----------|---------|
| Job | batch/v1 | Runs adapter container for a specific benchmark evaluation |
| ConfigMap | v1 | Holds job specification JSON (job.json) mounted into adapter container |

Job pods include:
- **Adapter container**: Provider-specific evaluation framework container (e.g., `quay.io/opendatahub/ta-lmes-job:odh-3.4-ea2`)
- **Init container** (optional): Downloads test data from S3 when `testDataRef.s3` is configured
- **Security context**: RunAsNonRoot, drop ALL capabilities, SeccompProfile RuntimeDefault
- **ServiceAccount**: `<instance>-<namespace>-job` (created by TrustyAI operator)
- **Volume mounts**: job-spec ConfigMap, data EmptyDir, service-ca ConfigMap, MLflow projected SA token, OCI credentials secret, model auth secret, test data volumes

## Security

### RBAC - Authorization Model

EvalHub uses Kubernetes SubjectAccessReview (SAR) for fine-grained authorization. Authorization rules are defined in `config/auth.yaml` and map HTTP endpoints/methods to Kubernetes resource attributes:

| Endpoint | HTTP Method | API Group | Resource | Verb |
|----------|-------------|-----------|----------|------|
| /api/v1/evaluations/jobs | POST | trustyai.opendatahub.io | evaluations | create |
| /api/v1/evaluations/jobs | POST | mlflow.kubeflow.org | experiments | create, get |
| /api/v1/evaluations/jobs | GET | trustyai.opendatahub.io | evaluations | get |
| /api/v1/evaluations/jobs/*/events | POST | trustyai.opendatahub.io | status-events | create |
| /api/v1/evaluations/collections | * | trustyai.opendatahub.io | collections | (from HTTP method) |
| /api/v1/evaluations/providers | * | trustyai.opendatahub.io | providers | (from HTTP method) |

Namespace for authorization is resolved from the `X-Tenant` HTTP header, enabling multi-tenant isolation.
