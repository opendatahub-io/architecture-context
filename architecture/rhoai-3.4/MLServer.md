## Purpose

**Short**: MLServer is a Python-based inference server implementing the V2 Inference Protocol (KFServing/Open Inference Protocol) for serving machine learning models over REST (HTTP) and gRPC.

**Detailed**: MLServer provides a standards-compliant inference serving runtime for the Red Hat OpenShift AI platform. It implements the V2 Inference Protocol (also known as KFServing dataplane protocol or Open Inference Protocol) which defines a standardized API for model inference, health checks, model metadata, and model repository management. MLServer supports multiple ML frameworks through a pluggable runtime architecture — the RHOAI build ships with runtimes for scikit-learn, XGBoost, LightGBM, and ONNX.

The server exposes three network interfaces simultaneously: an HTTP/REST server (powered by FastAPI/Uvicorn on port 8080), a gRPC server (on port 8081), and a dedicated Prometheus metrics server (on port 8082). It also supports an optional Kafka consumer/producer interface for event-driven inference. MLServer includes support for adaptive batching, parallel inference via multiprocessing workers (to bypass the Python GIL), response caching, OpenTelemetry distributed tracing, and CloudEvents middleware.

In RHOAI, MLServer is deployed as a container image (`odh-mlserver-rhel9`) that serves as an alternative inference runtime to Triton or custom serving runtimes. It is typically deployed as a sidecar or primary container within a KServe InferenceService pod. The RHOAI build includes a security-hardened "trusted runtimes allowlist" mechanism that restricts which model implementations can be loaded in production mode, preventing arbitrary code execution through model configuration.

## Architectural Analysis

MLServer follows a clean layered architecture with clear separation between protocol handling and business logic. The `DataPlane` class serves as the central handler layer, implementing all inference and metadata operations independent of the transport protocol. Both the `RESTServer` (FastAPI) and `GRPCServer` (grpc.aio) delegate to the same `DataPlane` instance, ensuring consistent behavior across protocols. The `KafkaServer` provides a third transport option for asynchronous, event-driven inference. This design makes it straightforward to add new transport protocols without modifying core inference logic.

The runtime plugin system is based on Python's import mechanism — model implementations are specified as dotted Python import paths (e.g., `mlserver_sklearn.SKLearnModel`) in `model-settings.json`. The RHOAI build adds a significant security layer on top of this: a "trusted runtimes allowlist" mechanism implemented in `settings.py`. In production mode, a JSON file at `/etc/mlserver/trusted-runtimes.json` (baked into the container image at build time) restricts which import paths can be loaded. This prevents a user from specifying an arbitrary Python class as a model implementation, which would otherwise allow arbitrary code execution. The allowlist is enforced at two points: during model settings validation (early rejection) and at implementation access time (defense in depth). Additionally, production mode blocks custom environment tarballs and environment paths, preventing sys.path injection attacks. Wildcard CORS origins are also blocked in production mode.

The parallel inference subsystem is noteworthy — it uses Python's `multiprocessing` module to spawn worker processes, bypassing the GIL for CPU-bound inference workloads. The `InferencePool` manages a pool of workers, with a `Dispatcher` coordinating request routing and a `WorkerRegistry` tracking loaded models for automatic recovery when workers crash. The `ParallelModel` wrapper transparently redirects `predict()` calls to the worker pool. This is critical for ML frameworks like scikit-learn that release the GIL poorly during inference.

The RHOAI Konflux build pipeline (`Dockerfile.konflux`) takes a notably different approach from the upstream `Dockerfile`. Instead of building wheels from source, it installs pre-built packages from a Red Hat-curated private PyPI index (`console.redhat.com/api/pypi/public-rhai/rhoai/3.4/cpu-ubi9/simple/`) using a pinned `requirements-cpu.txt` file with SHA-256 hashes for every dependency. The Tekton pipeline enables hermetic builds with pip prefetch, ensuring build reproducibility and supply chain integrity. The base image (`quay.io/aipcc/base-images/cpu:3.4.0`) is a UBI-based image that provides FIPS-compliant OpenSSL and necessary system libraries (e.g., `libgomp` for LightGBM).

MLServer does not implement TLS at the application level — all three servers bind to insecure ports. This is by design: in the RHOAI deployment model, TLS termination and authentication are handled by external infrastructure (KServe Gateway API with Envoy, kube-rbac-proxy sidecars, or Istio). The OpenTelemetry exporter also uses insecure gRPC connections to the collector, which is acceptable for cluster-internal traffic but could be a concern if the collector is external.
