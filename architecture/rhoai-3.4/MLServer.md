## Purpose

**Short**: MLServer is a Python-based inference server implementing the KServe V2 Inference Protocol, serving ML models over REST, gRPC, and Kafka interfaces with a pluggable runtime system.

**Detailed**: MLServer provides a multi-protocol inference server that implements the Open Inference Protocol (KServe V2 dataplane) for serving machine learning models. It exposes three server interfaces — a FastAPI-based REST server (port 8080), a gRPC server (port 8081), and an optional Kafka consumer/producer — all backed by a shared data plane that handles model lifecycle, inference routing, and response caching.

The server supports a pluggable runtime system where each ML framework (scikit-learn, XGBoost, LightGBM, ONNX, HuggingFace Transformers, MLflow, CatBoost, Alibi Detect/Explain, Spark MLlib) is implemented as a separate runtime plugin extending the `MLModel` base class. Models are discovered from a configurable repository directory via `model-settings.json` files and loaded at startup or on-demand via the model repository API.

MLServer includes advanced features such as adaptive request batching (merging concurrent requests into a single batch for efficiency), parallel inference workers (multiprocessing-based pool for CPU-bound models), inference response caching, CloudEvents header injection for event-driven architectures, and comprehensive observability via OpenTelemetry tracing and Prometheus metrics. In RHOAI, MLServer is deployed as a KServe InferenceService container image with a curated set of runtimes (LightGBM, ONNX, scikit-learn, XGBoost) and a trusted-runtimes security model that restricts which model implementations can be loaded.

### Runtime Security (Trusted Runtimes)

MLServer implements a production/development security mode for controlling which model implementations can be loaded:

- **Development Mode**: No `/etc/mlserver/trusted-runtimes.json` file exists; all runtime implementations are allowed
- **Production Mode**: The trusted-runtimes.json file (baked into the container image at build time) contains an allowlist of permitted model implementation import paths; loading any implementation not on the list is rejected
- The Dockerfile.konflux generates this file at build time with the specific runtimes installed in the image
- CORS wildcard origins are rejected in production mode
- Custom Python environments are rejected in production mode

## Architectural Analysis

MLServer's architecture is centered on the KServe V2 Open Inference Protocol, implementing a clean separation between protocol handling (REST/gRPC/Kafka), the data plane (inference routing, caching, batching), and the model runtime plugins. This layered design allows the same model to be served simultaneously over all three protocols without any runtime-specific code changes. The `DataPlane` class acts as the single entry point for all inference operations regardless of protocol, applying middleware (CloudEvents header injection), caching, and metrics uniformly.

The runtime plugin system uses a simple inheritance pattern where each ML framework implements `MLModel.load()` and `MLModel.predict()`. The codec system provides the content-type negotiation layer — it automatically maps between V2 protocol tensor formats and framework-native types (numpy arrays, pandas DataFrames, PIL images, Python dicts). This design is notably flexible: the `@decode_args` decorator on `predict()` can inspect Python type hints to automatically select the correct codec, reducing boilerplate in runtime implementations. The model repository uses a schemaless file-based discovery pattern, recursively scanning for `model-settings.json` files, which aligns well with KServe's model storage conventions where models are mounted at `/mnt/models`.

The parallel inference pool is an interesting architectural choice — rather than using thread-based parallelism (which would be limited by the GIL for CPU-bound ML inference), MLServer spawns subprocess workers that each run their own asyncio event loop with a full `MultiModelRegistry`. Requests are dispatched via `multiprocessing.Queue` using round-robin scheduling, and responses are correlated back to the original async future. Each worker can optionally run in an isolated Python environment (extracted from a tarball), enabling per-model dependency isolation. This is particularly important for ML workloads where different models may require incompatible versions of the same library.

The RHOAI-specific security hardening deserves attention. The trusted-runtimes mechanism bakes an allowlist of permitted model implementation import paths into `/etc/mlserver/trusted-runtimes.json` at container build time. When this file exists, MLServer enters "production mode" — only allowlisted runtimes can be loaded, custom Python environments are rejected, and wildcard CORS origins are blocked. This prevents arbitrary code execution through model-settings.json injection, which would otherwise be a significant attack vector since model implementations are loaded via Python's import machinery. The Dockerfile.konflux validates each runtime import path against `is_valid_runtime_import_path()` before writing the allowlist, adding a build-time safety check.

A notable absence in MLServer's architecture is any application-level TLS or authentication. The gRPC server explicitly uses `add_insecure_port()`, the REST server binds plaintext HTTP via uvicorn, and the OpenTelemetry exporter connects with `insecure=True`. This is by design — in the RHOAI deployment model, TLS termination and authentication are handled by the platform's ingress infrastructure (Gateway API + kube-rbac-proxy sidecar), so the inference container only needs to handle plaintext traffic on the pod network. However, this means MLServer must never be exposed directly outside the cluster without an authenticating proxy in front of it.
