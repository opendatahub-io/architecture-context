## Purpose

**Short**: A centralized library of reusable Kubeflow Pipelines components and pre-built AI/ML pipelines that are compiled, packaged into a container image, and initialized onto the KFP API server as managed pipelines for Red Hat OpenShift AI.

**Detailed**: The pipelines-components repository provides a structured collection of KFP (Kubeflow Pipelines) components and pipelines organized by functional category: data processing, training, evaluation, and deployment. Components are individual pipeline steps (decorated with `@dsl.component`) that perform specific ML tasks such as dataset downloading, model fine-tuning (SFT, LoRA, OSFT), AutoML training (AutoGluon tabular/timeseries), RAG optimization (AutoRAG with ai4rag), LLM evaluation (lm-evaluation-harness), and model registry integration.

The repository produces three container images via Konflux: the primary `odh-pipelines-components` image that packages all components and managed pipelines, the `odh-automl` runtime image for AutoGluon workloads, and the `odh-autorag` runtime image for RAG optimization with docling and ai4rag. At build time, managed pipelines (those with `managed: true` in metadata.yaml) are compiled to KFP YAML specs. At runtime, the init container copies these compiled pipeline definitions to a shared volume (`/config/managed-pipelines/`) where the KFP API server can discover and expose them to users.

The component library is distributed as the Python package `kfp-components` (version 1.11.0), importable as `kfp_components.components` and `kfp_components.pipelines`. Each component and pipeline carries standardized metadata (stability level, dependencies, tags, last verification date) and follows a consistent structure enforced by scaffolding scripts, CI validation, and automated README generation.

## Architectural Analysis

The pipelines-components repository represents a clean separation of concerns in the RHOAI ML platform: it provides the "what" (pipeline definitions and component implementations) while delegating the "how" (execution orchestration, scheduling, resource management) to the Kubeflow Pipelines engine and Argo Workflows. This is fundamentally a library + init container architecture — there are no persistent services, no controllers, and no CRDs. The entire runtime footprint is ephemeral pipeline step pods.

**Component Architecture Pattern**: Every component follows the KFP Python function-based pattern: a single Python function decorated with `@dsl.component`, specifying a `base_image` (either UBI9 Python or a custom runtime image like `odh-automl`/`odh-autorag`) and `packages_to_install` for additional pip dependencies. Components communicate through KFP artifacts (Dataset, Model, Metrics, HTML, Artifact) and typed parameters. Inter-step data sharing uses PVC workspaces (50Gi for fine-tuning with ReadWriteMany NFS, 12Gi for AutoML with ReadWriteOnce) rather than artifact stores, which is efficient for large ML model checkpoints but creates a coupling to storage class availability.

**Pipeline Composition Patterns**: Three distinct pipeline architectures emerge: (1) **Linear fine-tuning pipelines** (SFT, LoRA, OSFT) follow a consistent 4-stage pattern — download → train → evaluate → register — with minimal and full variants offering different parameter granularity. (2) **AutoML pipelines** use a 2-stage selection-then-refit pattern with `dsl.ParallelFor` for parallel model refitting, though parallelism is constrained to 1 by ReadWriteOnce PVC limitations. (3) **AutoRAG pipelines** are the most complex, with 7 stages including document discovery, text extraction, search space preparation, RAG template optimization, and Responses API request generation.

**Managed Pipeline Lifecycle**: The managed pipeline mechanism is particularly well-designed. Pipelines opt in via `managed: true` in metadata.yaml. At build time, `generate_managed_pipelines.py` discovers these, validates metadata (rejecting `experimental` stability), compiles them to KFP YAML via the KFP compiler, and writes a `managed-pipelines.json` manifest. The init container then copies these compiled YAMLs to the KFP API server's discovery volume. This provides a clean contract between the pipeline library and the platform operator — the operator only needs to know the container image, not individual pipeline definitions. Currently, three pipelines are managed: `autogluon_tabular_training_pipeline`, `autogluon_timeseries_training_pipeline`, and `documents_rag_optimization_pipeline`.

**Runtime Image Strategy**: The repository uses a split image strategy. The primary `odh-pipelines-components` image is lightweight (UBI9 Python + KFP SDK) and serves as the init container. The `odh-automl` and `odh-autorag` images are heavyweight runtime environments with ML frameworks pre-installed. Components reference these via the `RELATED_IMAGE_MPI_AUTOML_RUNTIME` and `RELATED_IMAGE_MPI_AUTORAG_RUNTIME` environment variables (defined in `utils/consts.py`), allowing the platform operator to pin exact image digests without modifying component source code. The AutoRAG image is notably complex: it builds SQLite from source for ChromaDB compatibility and pre-seeds docling ML models using Hermeto generic prefetch for offline text extraction.

**Development Tooling**: The repository includes a comprehensive development automation layer: skeleton scaffolding (Jinja2 templates), automated README generation, AST-based base image validation, compile-time checks, import validation, and metadata schema enforcement. The `scripts/lib/` shared library provides reusable utilities for module loading, KFP compilation, and base image management that underpin all CI/CD validation. This tooling ensures consistency across 25+ components and 11 pipelines while enabling a contributor-friendly "make component" workflow.
