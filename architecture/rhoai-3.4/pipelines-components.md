## Purpose

**Short**: A centralized library of reusable Kubeflow Pipelines (KFP) components and pre-built pipelines for AI/ML workflows on Red Hat OpenShift AI, with an init container that stages compiled pipeline YAMLs for the KFP API server.

**Detailed**: The pipelines-components repository provides a structured collection of KFP v2 components and pipelines spanning data processing, model training, evaluation, and deployment. Components are Python functions decorated with `@dsl.component` or `@dsl.pipeline` that execute as containerized steps within Kubeflow Pipelines on Kubernetes. The repository covers four major AI workflow categories: AutoML (tabular and time-series training via AutoGluon), AutoRAG (automated RAG optimization using ai4rag and LlamaStack), fine-tuning (LoRA, OSFT, and SFT using Kubeflow Trainer with HuggingFace Transformers), and supporting utilities (dataset download, synthetic data generation, LLM evaluation, model registry).

The repository also ships a build-time pipeline compilation system (`generate_managed_pipelines`) and a runtime init container (`init_managed_pipelines`) that together form a pipeline delivery mechanism: at image build time, managed pipelines are discovered via `metadata.yaml` files, compiled to KFP IR YAML using the KFP compiler, and bundled into the container image. At runtime, the init container copies these compiled YAMLs to a shared Kubernetes volume (`/config/managed-pipelines`) where the KFP API server picks them up for registration. Three Konflux-built container images are produced: the main pipelines-components init container, an AutoML runtime image, and an AutoRAG runtime image.

## Architectural Analysis

The pipelines-components repository implements a layered architecture with clear separation between build-time pipeline compilation and runtime pipeline staging. The build-time layer (`generate_managed_pipelines.py`) uses a metadata-driven discovery pattern: it scans the `pipelines/` directory tree for `metadata.yaml` files with `managed: true`, dynamically loads the corresponding `pipeline.py` modules, and compiles them to KFP IR YAML using the KFP Compiler v2.16.0. This approach avoids hardcoding pipeline paths and supports easy addition of new managed pipelines. The runtime layer (`init_managed_pipelines.py`) is deliberately minimal — a 25-line script that copies pre-compiled YAMLs to a shared volume with path-escape validation, reducing the attack surface of the init container.

The component library follows a consistent pattern: each component is a Python function decorated with `@dsl.component` that declares its base_image and packages_to_install. Components interact with external services (S3, LlamaStack, HuggingFace, Model Registry) through environment variables injected from Kubernetes Secrets via KFP's `use_secret_as_env` mechanism. This avoids hardcoding credentials and supports the RHOAI platform's secret management patterns. However, the Kubeflow Model Registry component uses HTTP without TLS (`insecure=False` parameter in the SDK, which confusingly means "insecure mode enabled"), which is a security gap worth noting — in production, mTLS should be enforced between components.

The fine-tuning pipeline architecture uses the Kubeflow Trainer (formerly Training Operator) for distributed training, with a notable design decision: LoRA training is limited to single-node execution due to Unsloth backend constraints (`nnodes=1` hardcoded in `lora/component.py`), while SFT and OSFT support multi-node FSDP sharding. All three fine-tuning approaches share a common utility layer (`shared/data.py`, `shared/output.py`, `shared/setup.py`, `shared/training.py`) that handles dataset resolution, model persistence, K8s initialization, and training job monitoring. The LoRA component includes sophisticated post-training processing — merging LoRA adapter weights into the base model, stripping `base_model.` key prefixes from safetensors, and removing quantization config from `config.json` — to ensure output models are compatible with downstream evaluation (lm-eval) and serving (vLLM).

The AutoRAG subsystem is the most complex workflow, implementing a multi-stage RAG optimization pipeline: document discovery → text extraction → search space preparation → template optimization → leaderboard evaluation. The `rag_templates_optimization` component uses the ai4rag library's `AI4RAGExperiment` with GAM (Generalized Additive Model) optimization to search over chunking strategies, embedding models, retrieval methods, and generation configurations. It generates both optimized RAG pattern configurations and Jupyter notebooks for reproducibility. The component supports dual execution modes: LlamaStack-based (using Milvus/Weaviate/Pinecone vector stores) and in-memory (using ChromaDB with OpenAI-compatible endpoints), with ChromaDB requiring a SQLite3 compatibility patch at runtime (lines 66-73 of `rag_templates_optimization/component.py`).

A noteworthy supply chain aspect: the AutoRAG Dockerfile uses Hermeto generic prefetch to download docling ML models (layout detection and table extraction) at build time, pinned by SHA-256 checksums in `artifacts.lock.yaml`. This enables fully hermetic Konflux builds that don't require network access during the build phase, but means model updates require lockfile regeneration. The `seed_docling_models.py` script supports both network-fetched (HuggingFace Hub) and Hermeto-prefetched paths, providing a graceful development vs. production build split.
