## Purpose

**Short**: MLflow is an open-source ML experiment tracking and model registry server, adapted for Red Hat OpenShift AI to provide workspace-aware experiment management with Kubernetes-native authentication.

**Detailed**: MLflow provides a comprehensive platform for the machine learning lifecycle, including experiment tracking, model registry, artifact storage, and model serving. The RHOAI adaptation (version 3.10.1+rhaiv.3) extends upstream MLflow with Kubernetes-native authentication via ServiceAccount tokens, multi-workspace isolation for multi-tenant deployments, and envelope encryption for secrets management.

The server is a hybrid Flask/FastAPI application that serves both a REST API and a React-based web UI. It supports multiple backend database engines (PostgreSQL, MySQL, SQLite) for tracking metadata and model registry, and multiple artifact storage backends (S3, Azure Blob, GCS, local filesystem). The RHOAI build is configured with `--app-name kubernetes-auth`, `--enable-workspaces`, and `--workspace-store-uri kubernetes://`, enabling Kubernetes namespace-based multi-tenancy.

The Konflux build produces a single container image (`odh-mlflow-rhel9`) based on UBI 9 minimal with Python 3.12. The image includes a pre-built React UI (built via Node.js/Yarn) and is configured with telemetry disabled, assistant features disabled, and AI gateway disabled for the RHOAI distribution.

## Architectural Analysis

MLflow's RHOAI adaptation represents a sophisticated integration of an upstream open-source ML platform into a Kubernetes-native, multi-tenant environment. The most architecturally significant decisions are the Kubernetes auth provider and workspace isolation layer.

**Hybrid Flask/FastAPI Architecture**: The server uses a progressive migration pattern where the legacy Flask app is wrapped via WSGIMiddleware and mounted inside a FastAPI application. New features (jobs, gateway, OTel) use native FastAPI routers, while the core tracking API still runs through Flask handlers generated from protobuf service definitions. This dual-stack approach is visible in `mlflow/server/fastapi_app.py` where routers are included in a specific order before mounting the Flask WSGI app at `/`. The security middleware is duplicated across both frameworks — Flask uses `before_request`/`after_request` hooks while FastAPI uses ASGI middleware classes — which adds maintenance burden but ensures consistent security posture regardless of which framework handles a given request.

**Kubernetes-Native Authentication**: The `kubernetes-request-auth-provider` module (`mlflow/tracking/request_auth/kubernetes_request_auth_provider.py`) implements two auth strategies: `kubernetes` (token-only) and `kubernetes-namespaced` (token + workspace header). The namespaced variant reads the pod's namespace from `/var/run/secrets/kubernetes.io/serviceaccount/namespace` and maps it to an `X-MLFLOW-WORKSPACE` header, enabling automatic workspace isolation per Kubernetes namespace. Token resolution uses a 60-second TTL cache to avoid repeated file reads. The Dockerfile CMD (`--app-name kubernetes-auth`) activates a custom auth plugin registered via the `mlflow.app` entry point, which replaces the default basic-auth with Kubernetes SA token validation.

**Envelope Encryption for Secrets**: The crypto module (`mlflow/utils/crypto.py`) implements a full envelope encryption scheme using AES-256-GCM for data encryption and PBKDF2-HMAC-SHA256 (600,000 iterations) for key derivation. Each secret gets a unique DEK (Data Encryption Key), which is wrapped with a KEK (Key Encryption Key) derived from an admin passphrase. The AAD (Additional Authenticated Data) binds encryption to secret identity, preventing ciphertext substitution attacks. KEK rotation re-wraps DEKs without re-encrypting data — an efficient design that limits exposure during rotation. The default passphrase is intentionally weak for development convenience, with prominent warnings in logs and documentation.

**Multi-Workspace Isolation**: Workspace awareness is implemented at the ORM query layer via `WorkspaceAwareSqlAlchemyStore` mixins that inject workspace filters into every database query. This provides strong data isolation without requiring separate databases per tenant. The workspace context flows through HTTP headers (`X-MLFLOW-WORKSPACE`), request middleware, and thread-local storage. Permission evaluation follows a three-tier hierarchy: resource-specific permissions, workspace-level permissions, then default permissions. The caching strategy (10,000-entry TTL cache for workspace permissions) trades consistency for performance.

**Build and Supply Chain**: The Konflux build is well-structured with three stages (Node.js UI builder, Python wheel builder, UBI minimal runtime). Python dependencies use `--require-hashes --no-deps` for reproducible, tamper-resistant builds. The cachi2 integration supports fully air-gapped builds. The `rpms.lock.yaml` locks OS-level packages across four architectures (aarch64, ppc64le, s390x, x86_64). The UI build produces both a standard build and a Module Federation build (`yarn build:federated` with `MLFLOW_API_BASE_URL=/mlflow`), suggesting the UI can be embedded as a micro-frontend in the RHOAI dashboard.
