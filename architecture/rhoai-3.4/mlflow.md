## Purpose

**Short**: MLflow is the experiment tracking, model registry, and ML lifecycle management service for Red Hat OpenShift AI.

**Detailed**: MLflow provides a shared tracking server that records ML experiment runs, parameters, metrics, and artifacts across data science projects. In the RHOAI deployment, it extends upstream MLflow with Kubernetes-native workspace isolation — each Kubernetes namespace becomes an MLflow workspace, and Kubernetes RBAC governs access to experiments and models within each workspace. The server exposes a REST API for SDK clients and an embedded React UI for interactive use.

The RHOAI build produces a single container image that runs the MLflow tracking server with the `kubernetes-auth` application plugin and `kubernetes://` workspace store URI. This configuration maps data science project namespaces to MLflow workspaces, enforces authorization via `SelfSubjectAccessReview`, and supports per-namespace artifact storage overrides via `MLflowConfig` custom resources. The server is accessed through the RHOAI data science gateway's `/mlflow` HTTPRoute endpoint.

MLflow also includes an envelope encryption system for secrets management (AES-256-GCM with PBKDF2-derived KEKs), OpenTelemetry trace ingestion, Prometheus metrics export, and a pluggable authentication/authorization framework. The upstream codebase supports numerous ML framework integrations (scikit-learn, PyTorch, TensorFlow, etc.), but the RHOAI container disables optional features like the AI Gateway, Assistant, and server-side job execution.

## Architectural Analysis

MLflow's RHOAI integration follows a "platform-native multi-tenancy" pattern where Kubernetes primitives — namespaces, service account tokens, and RBAC — replace MLflow's built-in user/permission system. The `kubernetes://` workspace store is a read-only projection: it treats namespaces as the source of truth for workspaces rather than maintaining its own workspace objects. This design eliminates workspace lifecycle management from MLflow entirely, delegating it to the platform's project management. The `MLflowConfig` CRD allows per-namespace artifact storage overrides while the shared deployment handles everything else centrally.

The server architecture uses a dual-stack approach: a Flask application handles the legacy REST API (100+ endpoints registered dynamically from protobuf definitions), while a FastAPI wrapper provides the ASGI entry point for newer features like OTLP trace ingestion and the jobs API. The FastAPI app mounts the Flask app via WSGIMiddleware, ensuring 100% backward compatibility. Security middleware (host validation, CORS blocking, security headers) is implemented at both layers. This is a transitional architecture — the codebase is migrating from Flask to native FastAPI endpoints.

The envelope encryption system in `mlflow/utils/crypto.py` is notably well-engineered for a tracking server. It uses AES-256-GCM with per-secret random DEKs wrapped by a passphrase-derived KEK (PBKDF2-HMAC-SHA256, 600k iterations). AAD (Additional Authenticated Data) binding prevents ciphertext substitution attacks. KEK rotation is atomic and idempotent via a dedicated CLI command. This secures gateway secrets (API keys for LLM providers) at rest in the database. However, the default development passphrase is hardcoded — production deployments must override `MLFLOW_CRYPTO_KEK_PASSPHRASE`.

The Kubernetes auth flow is efficient: the `KubernetesAuth` callable injects `Authorization` (bearer token) and `X-MLFLOW-WORKSPACE` (namespace) headers into every outbound SDK request. Token and namespace resolution use a 60-second TTL cache with thread-safe locking, supporting both in-cluster service account paths and out-of-cluster kubeconfig credentials. The server-side `kubernetes-auth` plugin then validates each request via `SelfSubjectAccessReview` against pseudo-resources in the `mlflow.kubeflow.org` API group. This means workspace authorization is consistent with the rest of the OpenShift RBAC model — no separate permission database.

The Konflux build pipeline produces a multi-architecture image (amd64, arm64, ppc64le, s390x) with full hermeticity on push builds. The three-stage Dockerfile separates UI compilation (Node.js 20), Python wheel building, and minimal runtime (UBI9 ubi-minimal). All Python dependencies are hash-verified, and the push pipeline enforces hermetic mode with cachi2 offline dependency prefetch. The RHOAI-specific feature flags (`MLFLOW_ENABLE_AI_GATEWAY=false`, `MLFLOW_ENABLE_ASSISTANT=false`, `MLFLOW_SERVER_ENABLE_JOB_EXECUTION=false`) reduce the attack surface by disabling non-essential features in the production image.
