## Purpose

**Short**: TrustyAI is a Java library and Quarkus microservice providing AI model explainability, fairness metrics, and data drift detection for models served via KServe/ModelMesh on Red Hat OpenShift AI.

**Detailed**: TrustyAI Explainability is a multi-module Maven project that provides a comprehensive suite of Responsible AI capabilities. The core library (`explainability-core`) implements model-agnostic explainability algorithms including LIME, SHAP, Counterfactual explanations, and Partial Dependence Plots, along with fairness metrics (Statistical Parity Difference, Disparate Impact Ratio), drift detection (Kolmogorov-Smirnov test, Fourier MMD, Meanshift), and NLP evaluation metrics (BLEU, ROUGE).

The service layer (`explainability-service`) is a Quarkus-based REST microservice that wraps the core library and provides HTTP and Knative CloudEvent interfaces for consuming inference data from KServe v1/v2 model servers. It computes fairness and drift metrics on a configurable schedule, publishes results to Prometheus, and supports multiple storage backends (PVC, in-memory, MariaDB/MySQL database). The service is deployed per-namespace alongside model serving infrastructure and integrates with ModelMesh via a dynamically-generated ConfigMap that registers the TrustyAI consumer endpoint.

In the RHOAI distribution, the service runs with a restricted feature set: fairness metrics and drift detection are enabled, but local explainer endpoints (LIME, SHAP, Counterfactual, TSSaliency) and global explainers are disabled. The ODH distribution enables the full feature set. The service is deployed by the TrustyAI operator (separate repository) which manages the TrustyAIService CR lifecycle.

## Architectural Analysis

TrustyAI Explainability follows a clean modular architecture with four Maven modules that separate concerns effectively. The `explainability-core` library is model-agnostic and framework-independent — it defines the core `PredictionProvider` interface and implements all AI explainability algorithms without coupling to any serving infrastructure. The `explainability-connectors` module bridges the gap between TrustyAI's internal data model and the KServe inference protocol, supporting three transport modes (v1 HTTP, v2 HTTP, v2 gRPC). The `explainability-arrow` module provides efficient columnar data handling via Apache Arrow. The `explainability-service` module wraps everything in a Quarkus REST service suitable for Kubernetes deployment.

The service's integration with ModelMesh is notable for its use of an init container that dynamically generates a `model-serving-config` ConfigMap at deployment time. This ConfigMap registers TrustyAI's `/consumer/kserve/v2` endpoint as a payload processor, enabling ModelMesh to transparently forward inference data to TrustyAI without requiring changes to model serving configuration. This is a pull-based architecture where TrustyAI passively receives data rather than actively intercepting inference traffic. The init container uses `oc apply` with a ClusterRole-scoped ServiceAccount, which grants ConfigMap CRUD access cluster-wide — this is a broader permission than strictly necessary (a namespace-scoped Role would suffice for the init container's needs).

The RHOAI profile significantly restricts the feature surface compared to ODH. All local explainer endpoints (LIME, SHAP, Counterfactual, TSSaliency) and global explainers (Aggregated LIME, PDP) are disabled in the `%rhoai` profile, leaving only fairness metrics (SPD, DIR) and drift detection (KS test, Approximate KS test, Fourier MMD, Meanshift) active. This suggests that explainability features are not yet productized for RHOAI, while monitoring/compliance features (fairness and drift) are. The feature flags are managed via Quarkus profile-based configuration (`%rhoai.*` properties), making it straightforward to enable features in future releases.

The Konflux build pipeline is unconventional for this repository. Rather than building from source in the Dockerfile, the `Dockerfile.konflux` consumes a pre-built ZIP artifact from Red Hat's PNC (Project Newcastle) build system, locked via `artifacts.lock.yaml`. The Hermeto (cachi2) prefetch mechanism downloads this artifact and the Dockerfile simply unzips and copies the Quarkus application layers into the runtime image. This means the actual Java compilation happens in PNC, not in the container build — the Konflux pipeline is essentially a packaging step. This provides strong build provenance (PNC builds are auditable) but means the `Dockerfile.konflux` alone does not reveal the full build process.

The service's network security posture relies primarily on Kubernetes namespace isolation rather than application-level authentication. None of the REST endpoints enforce authentication — the service trusts that network-level controls (NetworkPolicies, namespace boundaries) and the kube-rbac-proxy sidecar injected by the TrustyAI operator restrict access. TLS is available on port 4443 with certificates mounted from `/etc/tls/internal/`, but the manifest Route has `tls: null`, indicating plaintext HTTP exposure. The ServiceMonitor authenticates to the metrics endpoint using a ServiceAccount bearer token, but this is Prometheus-specific, not application-enforced.
