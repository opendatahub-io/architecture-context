## Purpose

**Short**: TrustyAI is a Quarkus-based Java service providing AI fairness metrics, explainability algorithms, and data drift detection for ML models deployed on Red Hat OpenShift AI.

**Detailed**: TrustyAI Explainability is a multi-module Java library and REST service that integrates with KServe and ModelMesh inference platforms to provide responsible AI capabilities. The service intercepts inference payloads (via REST consumer endpoints and CloudEvents), stores them in configurable backends (PVC, MinIO/S3, MariaDB/MySQL, or in-memory), and exposes REST endpoints for computing fairness metrics (Statistical Parity Difference, Disparate Impact Ratio), data drift metrics (KS Test, Approximate KS Test, Fourier MMD, Meanshift), and local/global AI explainability algorithms (LIME, SHAP, TSSaliency, Counterfactual, Partial Dependence Plots).

The service publishes computed metrics to Prometheus via Micrometer, enabling continuous fairness and drift monitoring through Grafana dashboards. It supports scheduled metric computation on configurable intervals, with results exposed as Prometheus gauges prefixed with `trustyai_`. The RHOAI distribution profile disables explainer endpoints (LIME, SHAP, Counterfactual, TSSaliency, PDP) while enabling fairness and drift endpoints, reflecting the current supported feature set.

The repository is structured as a Maven multi-module project: `explainability-core` (XAI algorithms and data structures), `explainability-connectors` (KServe/ModelMesh gRPC and HTTP adapters), `explainability-service` (Quarkus REST service), and `explainability-arrow` (Arrow IPC for Python interop). The service is deployed per-namespace alongside model serving infrastructure, with each instance managing data for models in its namespace.

## Architectural Analysis

TrustyAI Explainability follows a modular architecture with clear separation between core algorithms, protocol adapters, and the service layer. The `explainability-core` module is a pure Java library containing fairness metrics (SPD, DIR), drift detectors (KS Test, Fourier MMD, Meanshift), and XAI algorithms (LIME, SHAP, Counterfactual via OptaPlanner). The `explainability-connectors` module provides KServe v1/v2 protocol adapters with both HTTP and gRPC clients, decoupling the service from specific inference protocols. This modular design allows the core library to be used independently (e.g., in the TrustyAI Python library via Arrow IPC).

The storage architecture uses a strategy pattern with four interchangeable backends selected at startup via the `service.storage-format` configuration. The abstract `DataSource` class defines a uniform API for dataframe reads/writes, metadata management, ground truth handling, and tag-based filtering. `CSVDataSource` handles PVC/MinIO/Memory backends using `FlatFileStorage` (ByteBuffer-based serialization), while `HibernateDataSource` uses JPA entities directly. The PVC-to-Database migration path (`HibernateMigrator`) enables seamless transition between storage modes using async CDI events. This flexibility is important because the TrustyAI Operator may choose different storage backends based on cluster capabilities and user preferences.

The feature flag system using Quarkus `@EndpointDisabled` annotations enables profile-based endpoint activation. The RHOAI profile (`%rhoai.*`) explicitly disables all explainer endpoints (LIME, SHAP, TSSaliency, Counterfactual, PDP) while enabling fairness and drift endpoints. This reflects the current RHOAI support matrix where explainability features are not yet generally available. The ODH profile enables all endpoints. This design allows gradual feature rollout without code changes.

The inference data pipeline has two ingestion paths: synchronous REST (`/consumer/kserve/v2`) for ModelMesh and asynchronous CloudEvents (Knative Funqy) for KServe. Both paths use payload reconcilers that match partial request/response payloads by inference ID, convert them to dataframes, and persist to the configured storage backend. The reconciler architecture supports both in-memory and Hibernate-backed payload storage, with the latter providing crash-resilient reconciliation for database-mode deployments.

A notable architectural concern is the OpenShift Route in the base manifests which has `tls: null`, exposing the service over unencrypted HTTP externally. In production RHOAI deployments, this is mitigated by the TrustyAI Operator injecting a `kube-rbac-proxy` sidecar and configuring TLS certificates, but the base manifests themselves do not enforce encryption. The service supports TLS on port 4443 when certificate files are mounted at `/etc/tls/internal/`, but this is optional and not configured in the default kustomize base.
