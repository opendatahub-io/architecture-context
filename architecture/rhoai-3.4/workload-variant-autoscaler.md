## Purpose

**Short**: A Kubernetes controller that performs intelligent autoscaling for LLM inference model servers based on saturation metrics (KV cache utilization, queue depth, and token-level demand analysis).

**Detailed**: The Workload Variant Autoscaler (WVA) is a Kubernetes operator that manages autoscaling decisions for inference model servers serving Large Language Models (LLMs). It defines and reconciles a custom resource called `VariantAutoscaling` (VA), which represents an autoscaling configuration for a specific model variant — a combination of hardware (GPU type), runtime, and serving approach for a given base model.

WVA continuously monitors inference server metrics (from vLLM servers) via Prometheus, including KV cache utilization, request queue depth, and token generation rates. It uses saturation-based scaling algorithms to determine optimal replica counts, then emits those decisions as Prometheus metrics that external autoscalers (HPA or KEDA) consume to actually scale the target workloads (Deployments, StatefulSets, or LeaderWorkerSets). This architecture separates the scaling intelligence (WVA) from the scaling execution (HPA/KEDA), allowing WVA to focus on domain-specific LLM serving analysis while leveraging battle-tested Kubernetes scaling primitives.

WVA also integrates with the Gateway API Inference Extension's `InferencePool` resource to discover and track inference serving endpoints, and supports scale-to-zero functionality with configurable retention periods for idle models. The controller supports multi-instance isolation via labels, namespace-scoped configuration overrides, and namespace exclusion annotations, enabling shared clusters with independent scaling policies per team or model.

## Architectural Analysis

**Engine-Reconciler Separation Pattern**: WVA employs a distinctive architectural pattern where the scaling intelligence runs in separate engine loops (saturation engine, scale-from-zero engine) rather than in the reconcile loop itself. The engines run on a configurable interval (default 60s), query Prometheus for metrics, compute scaling decisions, and publish them to an in-memory `DecisionCache`. The reconciler then reads from this cache and patches the VariantAutoscaling CR status. This separation has several advantages: the engine can batch-process all VAs efficiently in a single Prometheus query round-trip, the reconciler remains fast and idempotent, and engine failures don't block VA status updates for already-computed decisions. The engine triggers reconciliation via a `DecisionTrigger` channel (a `source.Channel` from controller-runtime), avoiding API server polling.

**Multi-Tenancy and Namespace Isolation**: WVA has a sophisticated multi-tenancy model. The `CONTROLLER_INSTANCE` environment variable enables running multiple WVA instances in the same cluster, each managing only VAs with a matching `wva.llmd.ai/controller-instance` label. Namespace exclusion via the `wva.llmd.ai/exclude` annotation prevents WVA from managing resources in specific namespaces. Namespace-local ConfigMap overrides (saturation scaling, scale-to-zero, queueing model) allow per-namespace tuning without affecting the global configuration. The namespace tracking system in the Datastore ensures ConfigMap predicates don't trigger watches for unrelated namespaces, avoiding performance problems in large clusters.

**Pluggable Metrics Collection**: The collector subsystem uses a source registry pattern that supports multiple backends. The primary source is Prometheus (querying vLLM metrics via PromQL), but the system also supports direct pod scraping for EPP (Endpoint Picker) flow control metrics. Each InferencePool gets its own `PodScrapingSource` registered in the source registry, using a bearer token from a mounted secret. This design allows WVA to collect metrics from components that don't export to the cluster's Prometheus instance. The metrics cache has configurable freshness thresholds (fresh < 1m, stale < 5m, unavailable ≥ 5m) that influence scaling decisions — stale metrics produce more conservative scaling.

**FIPS-Compliant Build Pipeline**: The Konflux Dockerfile uses `GOEXPERIMENT=strictfipsruntime` with `CGO_ENABLED=1`, ensuring the Go binary uses OpenSSL for cryptographic operations rather than Go's pure-Go crypto. The application-level TLS configuration enforces a minimum of TLS 1.2 (per Red Hat's security guidelines) and disables HTTP/2 by default to mitigate stream cancellation CVEs. The `InsecureSkipVerify` option exists for development but is validated at startup when used with production Prometheus endpoints.

**Scaling Algorithm Versioning**: The saturation scaling configuration supports two analyzer generations. V1 uses percentage-based thresholds (KV cache threshold, queue length threshold) with spare capacity triggers. V2 uses token-level demand analysis with configurable scale-up thresholds and scale-down boundaries, plus a composite analyzer scoring system where multiple analyzers can contribute weighted scores. The V2 path is selected when `analyzerName: "saturation"` is set or when the `analyzers` list is populated. Both versions coexist — the config's `IsV2()` method determines which path to use.
