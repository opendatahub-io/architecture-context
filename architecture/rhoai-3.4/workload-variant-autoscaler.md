## Purpose

**Short**: A Kubernetes controller that performs intelligent autoscaling for LLM inference model servers based on saturation metrics, cost optimization, and queueing theory.

**Detailed**: The Workload Variant Autoscaler (WVA) is a global autoscaler for inference model servers serving Large Language Models (LLMs). It works alongside standard Kubernetes HPA and external autoscalers like KEDA to scale objects that support the scale subresource (Deployments, StatefulSets, LeaderWorkerSets).

WVA introduces the concept of a "variant" — a distinct way of serving a given model with a particular combination of hardware (e.g., A100, H100, L4), runtimes, and serving approaches. Each variant is represented by a `VariantAutoscaling` custom resource. When multiple variants serve the same model, WVA performs cost-aware optimization to add capacity on the cheapest variant and remove it from the most expensive.

The controller continuously monitors request rates and server performance via Prometheus metrics (vLLM KV cache utilization, queue depth, request counts), runs saturation analysis through multiple analyzer engines (V1 percentage-based, V2 token-based, and queueing model-based), and emits optimization metrics that HPA/KEDA consume to perform the actual scaling. It also supports scale-to-zero and scale-from-zero capabilities by monitoring Gateway API Inference Extension (EPP) flow control queue metrics.

## Architectural Analysis

The Workload Variant Autoscaler follows a clean separation between decision-making and actuation. The controller itself does not scale workloads directly in the steady-state path. Instead, the Saturation Engine computes optimal replica counts and writes them to an in-memory DecisionCache. The controller reconciler reads from this cache on each reconcile cycle and updates the VariantAutoscaling status with `desiredOptimizedAlloc`. The Actuator then emits Prometheus gauge metrics (`wva_desired_replicas`, `wva_desired_ratio`) that external autoscalers (HPA/KEDA) consume to perform the actual scaling. This indirect actuation pattern provides a clean separation of concerns and allows WVA to work with any Prometheus-based scaler.

The exception to this pattern is scale-from-zero, which must bypass HPA entirely because HPA cannot scale from zero replicas based on custom metrics alone. The Scale-from-Zero Engine uses a DirectActuator that patches the scale subresource directly via the Kubernetes API. It polls EPP flow control queue metrics at 100ms intervals — an aggressive frequency justified by the latency-critical nature of scale-from-zero (users are waiting for their first request to be served).

WVA supports three analyzer engines with a priority-based selection: (1) the Queueing Model Analyzer (activated by presence of `wva-queueing-model-config` ConfigMap), (2) the V2 token-based saturation analyzer (selected via `analyzerName: "saturation"` in config), and (3) the V1 percentage-based saturation analyzer (default fallback). The V1 path is being deprecated in favor of V2, which uses absolute token counts from vLLM's `cache_config_info` metric rather than percentage-based KV cache utilization. The queueing model path uses Kalman filters for state estimation and M/M/1/k queueing theory models.

Multi-controller isolation is a notable architectural feature. Multiple WVA instances can run simultaneously in the same cluster by setting the `CONTROLLER_INSTANCE` environment variable and labeling VariantAutoscaling resources with `wva.llmd.ai/controller-instance`. Each controller filters events using a predicate that matches only its assigned VAs. Namespace-level isolation is also supported via the `wva.llmd.ai/exclude` annotation and `wva.llmd.ai/config-enabled` label, allowing fine-grained control over which namespaces WVA manages. The ConfigMap reconciler supports hierarchical configuration: global ConfigMaps in the system namespace provide defaults, while namespace-local ConfigMaps in VA namespaces provide per-namespace overrides.

The InferencePool integration with Gateway API Inference Extension is architecturally significant. WVA watches InferencePool CRs to discover EPP (Endpoint Picker) pods, then scrapes their metrics directly using a PodScrapingSource. This enables WVA to detect pending requests in the inference scheduler's flow control queue — a signal that only the EPP has, not Prometheus. The EPP metrics token is mounted from a Kubernetes Secret, providing authenticated access to EPP metrics endpoints. This tight coupling with the Gateway API Inference Extension is a deliberate design choice for the llm-d inference serving stack.
