## Purpose

**Short**: KV-Cache indexing library and tokenization sidecar for KV-cache-aware routing in the llm-d distributed LLM inference platform.

**Detailed**: The llm-d-kv-cache component provides the KV-Cache Indexer, a high-performance Go library that maintains a global, near-real-time view of KV-cache block locality across a fleet of vLLM pods. It tracks which KV-cache blocks reside on which pods and on which device tier (GPU, CPU) by ingesting `KVEvents` streamed from vLLM engines over ZeroMQ. This metadata enables inference schedulers (such as llm-d-inference-scheduler and the Envoy Gateway/EPP) to make KV-cache-aware routing decisions, maximizing cache reuse and minimizing Time To First Token (TTFT).

The repository also ships a standalone Python gRPC service (UDS Tokenizer) that provides tokenization and chat template rendering over Unix Domain Sockets, enabling the Go indexer to tokenize prompts without embedding a Python runtime. Additionally, the repository contains utility components: a PVC Evictor for managing disk-based KV-cache storage, and a filesystem backend connector (llmd_fs_backend) with CUDA-accelerated tensor operations for vLLM's KV-cache offloading to shared storage.

The primary Go library is consumed as a dependency by llm-d-inference-scheduler (and the Inference Gateway EPP plugin), while the UDS Tokenizer runs as a sidecar container alongside the scheduler. The PVC Evictor and filesystem backend connector are deployed independently alongside vLLM pods when filesystem-based KV-cache storage is enabled.

## Architectural Analysis

The llm-d-kv-cache repository follows a **library-first architecture** — the core Go packages (`pkg/kvcache`, `pkg/kvevents`, `pkg/tokenization`) are designed to be consumed as dependencies by other services (primarily `llm-d-inference-scheduler`) rather than deployed as standalone binaries. The `examples/` directory contains reference implementations showing how to wire the components together, but these are not production-deployed artifacts. The only production-deployed artifact from this repository is the UDS Tokenizer sidecar container, built via the Konflux pipeline.

The **event ingestion pipeline** uses a sharded worker pool pattern for ordered, concurrent processing of KVEvents. Messages are routed to workers using FNV-1a hashing of the pod identifier, ensuring all events for a given pod are processed sequentially (maintaining cache consistency) while allowing parallel processing across pods. The ZMQ transport layer is deliberately chosen for its low-latency pub/sub semantics, avoiding the overhead of a message broker. The `EngineAdapter` abstraction allows supporting different inference engines (vLLM and SGLang adapters exist), each with their own serialization format.

The **tokenization architecture** has a notable design split: the Go code can either embed Python tokenizers via CGO (using build tag `embedded_tokenizers`) or communicate with an external UDS tokenizer service. The UDS approach is preferred for production because it avoids Python GIL contention in the Go process and enables independent scaling. The UDS Tokenizer wraps vLLM's `OpenAIServingRender` for full compatibility with vLLM's chat template rendering pipeline, including multimodal content (image_url blocks) and tool definitions. The gRPC interface over Unix Domain Socket is a deliberate zero-network-hop choice — the tokenizer runs in the same pod as the scheduler, communicating through an emptyDir volume mount.

The **block index abstraction** (`kvblock.Index`) supports four backends: in-memory LRU (default), Redis, Valkey, and cost-aware memory. The in-memory backend uses a two-level LRU structure (block hash → pod cache → pod entries) with fine-grained mutex locking for concurrent access. The Redis/Valkey backends use pipelined operations for batch efficiency and Lua scripts for atomic cleanup operations. The cost-aware memory backend extends the in-memory backend with eviction cost tracking. This pluggable design allows deployments to choose between latency (in-memory) and durability/scale (Redis/Valkey) based on fleet size.

The **token-to-block-key hashing** uses CBOR canonical encoding followed by FNV-64a, matching vLLM's prefix caching algorithm for interoperability. The hash chain (parent → child blocks) enables prefix tree matching, and the `PYTHONHASHSEED` alignment between the indexer and vLLM deployments is critical for correctness. The `BlockExtraFeatures` mechanism extends this to multimodal content, where image hashes taint the block keys to differentiate cached blocks with different visual content.

A notable security consideration is that all internal communication channels (ZMQ, gRPC over UDS, and the default gRPC example server) use **plaintext transport** without authentication. This is architecturally intentional — the components are designed to run within a trusted pod or cluster network boundary — but it means the component relies entirely on Kubernetes network policies and namespace isolation for security. Redis/Valkey connections support optional TLS via `rediss://` URLs.
