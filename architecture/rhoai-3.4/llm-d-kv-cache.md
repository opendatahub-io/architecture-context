## Purpose

**Short**: A pluggable KV-cache indexing and management service that enables KV-cache-aware routing for distributed LLM inference on vLLM and SGLang pods.

**Detailed**: The llm-d-kv-cache project is a core component of the llm-d distributed LLM inference platform. It maintains a global, near-real-time view of KV-cache block locality across a fleet of vLLM (or SGLang) pods, enabling intelligent routing decisions that maximize cache reuse and minimize redundant computation.

The system operates via two primary data paths: a **Write Path** that ingests KV-cache events (block stored, removed, cleared) streamed from inference engine pods over ZMQ PUB/SUB using MessagePack serialization, and a **Read Path** that scores pods for a given prompt by computing KV-block keys from tokenized input and looking up which pods hold matching blocks in its index. The index supports multiple backends including in-memory LRU, cost-aware memory (Ristretto), and distributed Redis/Valkey stores.

The repository also ships several supporting components: a **UDS Tokenizer Service** (Python gRPC service for tokenization and chat template rendering via vLLM), an **llmd_fs_backend** (a vLLM KV-cache offloading connector for shared filesystem storage with optional GPU Direct Storage), and a **PVC Evictor** (an automated disk space management service for KV-cache PVCs). Only the UDS Tokenizer Service has a Konflux Dockerfile for RHOAI builds; the other components are upstream-only or consumed as libraries.

## Architectural Analysis

**Multi-language design with clear boundaries.** The repository spans Go, Python, and C++ but maintains clean separation: Go provides the core indexing engine consumed as a library by the inference scheduler, Python provides the tokenization sidecar service and storage connectors, and C++ provides the CUDA GPU Direct Storage acceleration. The Go library uses CGO to embed Python for chat template rendering (via build tag `embedded_tokenizers`), but the default production path uses the external UDS Tokenizer Service instead, keeping the Go binary free of Python dependencies in the Konflux build.

**Sharded event processing for scalability.** The KV Events Pool uses FNV-1a hash-based sharding to distribute event processing across worker goroutines. Messages from the same pod are always processed by the same worker (same sharding key), ensuring per-pod ordering guarantees without global locks. This design scales linearly with `POOL_CONCURRENCY` (default: 4 workers) while maintaining consistency for per-pod block state. The ZMQ subscriber manager creates one subscriber per vLLM pod, either via explicit endpoint configuration or Kubernetes pod auto-discovery using controller-runtime watches.

**Multiple index backends with a common interface.** The `kvblock.Index` interface abstracts three distinct storage backends: in-memory LRU (hashicorp/golang-lru), cost-aware memory (dgraph-io/ristretto with byte-level cost tracking), and distributed Redis/Valkey (with Lua script-based atomic operations and optional RDMA). The in-memory backend uses a two-level cache (block hash → pod set, engine key → request key), while the Redis backend uses hash sets (HSET/HKEYS) for O(1) pod lookups and a Lua pruning script for atomic eviction with empty-key cleanup. This pluggable architecture allows deployment ranging from single-node in-memory to cluster-wide distributed caching.

**Tokenization pool with composite fallback.** The tokenization subsystem supports three strategies: local file-based tokenizers (auto-discovered from `/mnt/models`), external UDS tokenizers (gRPC over Unix socket), and HuggingFace Hub tokenizers (downloaded on first use). The `CompositeTokenizer` wraps all configured strategies and tries them in order, providing resilience against individual tokenizer failures. The UDS tokenizer client includes retry logic with exponential backoff (5 attempts) and warmup requests to verify model loading before accepting traffic.

**RHOAI ships only the UDS Tokenizer.** Despite the repository containing multiple components, only `services/uds_tokenizer/Dockerfile.konflux` is built by the Konflux pipeline. The Tekton PipelineRun triggers on changes to `services/uds_tokenizer/**` and builds against `quay.io/aipcc/base-images/cpu:3.4` (a RHOAI-specific UBI base image). The Go indexer library is consumed as a Go module dependency by the inference scheduler, not as a standalone container. The llmd_fs_backend and PVC Evictor are upstream-only components that are not part of the RHOAI build pipeline.

**Unencrypted internal transport is a design trade-off.** ZMQ event streams, gRPC tokenizer calls, and Redis index operations all use plaintext by default. This is acceptable for pod-local (UDS) and cluster-internal (ClusterIP service) communication, but the Redis backend supports TLS via the `rediss://` URL scheme for environments requiring encryption at rest or in transit. The OpenTelemetry exporter also uses insecure gRPC by default. Production deployments should evaluate whether NetworkPolicies or service mesh mTLS are sufficient, or whether per-protocol TLS configuration is needed.
