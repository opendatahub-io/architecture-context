## Purpose

**Short**: CPU-only build of the vLLM high-throughput LLM inference and serving engine, providing an OpenAI-compatible API for running large language model inference without GPU hardware.

**Detailed**: vLLM CPU is a downstream fork of the upstream vLLM project, packaged for Red Hat OpenShift AI (RHOAI) as a CPU-optimized inference runtime. It provides an OpenAI-compatible REST API server built on FastAPI/Uvicorn, supporting chat completions, text completions, embeddings, and model management endpoints. The server is designed for high-throughput, memory-efficient LLM inference using techniques such as PagedAttention, continuous batching, and various quantization methods.

This component is deployed as a container image (`odh-vllm-cpu-rhel9`) built on UBI 9 minimal, targeting the `s390x` and `ppc64le` architectures via Konflux multi-arch builds. It serves as the CPU inference backend in the RHOAI model serving stack, typically fronted by KServe's InferenceService CRD which manages routing, scaling, and lifecycle. The vLLM server itself is stateless — model weights are loaded at startup from Hugging Face Hub or local storage, and all inference state is held in memory.

The server also supports a gRPC interface (via optional `smg-grpc-servicer` package), OpenTelemetry tracing, Prometheus metrics, SSL/TLS with hot-reloading certificates, API key authentication, and CORS middleware. The CPU build excludes GPU-specific kernels and uses PyTorch CPU backend with optional Intel OpenMP optimizations for x86_64 or OpenBLAS for ppc64le/s390x.

## Architectural Analysis

vLLM CPU is architecturally a monolithic Python inference server that packs an enormous amount of functionality into a single process. The codebase contains over 105,000 lines of Python across the `vllm/` package, covering everything from low-level memory management and attention kernels to high-level OpenAI-compatible API routing. For the RHOAI CPU build, the GPU-specific code paths (CUDA, ROCm, TPU, XPU) are present in the source tree but inactive at runtime because PyTorch is built with CPU-only support and `VLLM_TARGET_DEVICE=cpu`.

The server uses a two-tier architecture: the FastAPI frontend handles request parsing, validation, and streaming, while the `AsyncLLM` engine (v1) manages model execution, KV-cache scheduling, and batch formation. In the default deployment mode, both tiers run in the same process with frontend multiprocessing disabled. The engine supports speculative decoding, LoRA adapter hot-loading, prefix caching, and tensor/pipeline/expert parallelism, though the CPU build primarily benefits from continuous batching and prefix caching.

The API surface is notably broad — beyond the standard OpenAI endpoints (chat completions, completions, embeddings, models), vLLM exposes Anthropic-compatible messages, SageMaker-compatible ping/invocations, RLHF weight update endpoints, elastic expert parallelism scaling, sleep/wake lifecycle management, profiling controls, and cache management. This breadth reflects vLLM's ambition as a universal inference server, but for the RHOAI deployment the primary consumption path is the OpenAI-compatible chat completion API fronted by KServe.

A notable security consideration is that the Konflux build is explicitly non-hermetic (`hermetic: false` in the Tekton pipeline), and no dependency lock files are present despite this being a downstream release branch. Python packages are fetched from PyPI and IBM's DevPI mirror at build time, creating a supply chain risk surface. The multi-architecture build (s390x, ppc64le) uses architecture-specific build scripts (`build_vllm_s390x.sh`, `build_vllm_ppc64le.sh`) that compile system libraries (numactl, OpenBLAS, LAPACK) from source within the container build, adding further build reproducibility concerns.

The SSL/TLS implementation is well-designed, using Python's stdlib `ssl` module (which delegates to system OpenSSL) and including a `SSLCertRefresher` that uses `watchfiles` to hot-reload certificates without server restart. This is important for Kubernetes environments where cert-manager rotates TLS certificates. However, TLS is entirely optional — by default the server listens on plaintext HTTP, relying on KServe's gateway and kube-rbac-proxy sidecar for encryption and authentication at the pod boundary.
