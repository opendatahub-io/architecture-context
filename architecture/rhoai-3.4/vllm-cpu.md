## Purpose

**Short**: CPU-optimized build of the vLLM inference engine that serves large language models via an OpenAI-compatible HTTP API and optional gRPC interface, without requiring GPU hardware.

**Detailed**: vLLM CPU is the CPU-only variant of the vLLM high-throughput inference and serving engine for large language models (LLMs). This downstream fork maintained by Red Hat Data Services provides a hardened, UBI-based container image (`odh-vllm-cpu-rhel9`) for deployment within Red Hat OpenShift AI (RHOAI) as a model serving runtime.

The server exposes an OpenAI-compatible REST API (chat completions, text completions, responses, embeddings, scoring, reranking, classification, tokenization, and audio transcription), an Anthropic Messages API, and an optional gRPC interface. It supports PagedAttention for efficient memory management, continuous batching of incoming requests, multiple quantization formats (GPTQ, AWQ, INT4/INT8/FP8), LoRA adapters, speculative decoding, structured output generation, and multi-modal inputs (vision, audio). The CPU variant is optimized for x86_64, ppc64le, and s390x architectures with platform-specific build scripts and libraries (OpenBLAS, libomp, numactl).

Within RHOAI, vLLM CPU is deployed as a ServingRuntime container managed by KServe, typically fronted by an HTTPRoute (Gateway API) and kube-rbac-proxy sidecar for authentication. It connects to Hugging Face Hub or S3-compatible storage for model downloads, and exposes Prometheus metrics and OpenTelemetry traces for observability.

## Architectural Analysis

The vLLM CPU repository is a downstream fork of the upstream vllm-project/vllm, customized for Red Hat's multi-architecture CPU deployment needs. The primary Konflux Dockerfile (`Dockerfile.konflux.cpu`) is a sophisticated multi-stage build that handles three CPU architectures (x86_64, ppc64le, s390x) using architecture-detection conditionals and platform-specific build scripts (`build_vllm_ppc64le.sh`, `build_vllm_s390x.sh`). This approach is necessitated by the vastly different library ecosystems on each architecture — ppc64le requires source-built OpenBLAS and LAPACK, s390x needs source-built LLVM and numactl, while x86_64 leverages pre-built Intel OpenMP packages.

The FastAPI-based API server is the primary entrypoint (port 8000), implementing a remarkably comprehensive API surface that covers OpenAI-compatible endpoints (chat/completions, responses, embeddings, scoring, reranking), Anthropic Messages compatibility, SageMaker container standards (ping/invocations), and operational endpoints (health, metrics, profiling, LoRA management). Authentication is handled by a custom ASGI middleware that validates Bearer tokens via SHA-256 comparison, but this is typically bypassed in RHOAI deployments where kube-rbac-proxy handles auth at the sidecar level. The server supports TLS with hot-reloading of certificate files via the `SSLCertRefresher` class using the `watchfiles` library.

The engine architecture uses a v1 `AsyncLLM` design with optional frontend multiprocessing, where the API server and engine can run in separate processes communicating via ZeroMQ IPC. This separation allows the API server to remain responsive while the engine performs synchronous GPU/CPU operations. The gRPC server (`grpc_server.py`) is an alternative entrypoint that wraps the same `AsyncLLM` engine with a protobuf interface on port 50051, though it currently only supports insecure (plaintext) connections.

A significant architectural consideration is the non-hermetic build: the Tekton pipeline explicitly sets `hermetic: false`, and no dependency lock files (rpms.lock.yaml, uv.lock, pip freeze) are present. This means builds pull dependencies from PyPI and IBM's DevPI index at build time, creating a supply chain reproducibility gap. The `LD_PRELOAD=/usr/lib64/libtcmalloc.so.4` in the Konflux Dockerfile indicates Google's TCMalloc is used for memory allocation in the final image, while the non-Konflux variant uses `libomp.so` for OpenMP runtime preloading — these are important runtime behavioral differences between Dockerfile variants.

The container runs as a non-root user (UID 2000, GID 0) following OpenShift security best practices, with `VLLM_NO_USAGE_STATS=1` to disable telemetry and `HF_HUB_OFFLINE=0` to allow model downloads. The `VLLM_USE_V1=1` flag enables the newer v1 engine architecture which is the default for this release.
