## Purpose

**Short**: Platform operator that orchestrates the deployment, configuration, and lifecycle management of all Red Hat OpenShift AI (RHOAI) components, services, and ingress infrastructure.

**Detailed**: The rhods-operator is the central control plane for Red Hat OpenShift AI (RHOAI). It implements a hierarchical controller architecture with two primary singleton CRDs — `DSCInitialization` (DSCI) for cluster-level setup (namespaces, authentication, monitoring, gateway) and `DataScienceCluster` (DSC) for component lifecycle management (Dashboard, KServe, Workbenches, Kueue, etc.). The operator manages 16 component handlers and 5 service handlers through an action-based reconciliation pattern using Server-Side Apply (SSA).

The operator deploys the platform's entire ingress infrastructure via the Gateway controller, which implements Kubernetes Gateway API with Envoy-based authentication (EnvoyFilter + kube-auth-proxy), supporting both OpenShift OAuth and external OIDC providers. It also manages the observability stack (MonitoringStack, Tempo, OpenTelemetry, Perses), RBAC authorization, trusted CA bundle distribution, and cloud manager controllers for non-OpenShift Kubernetes platforms (Azure AKS, CoreWeave). The operator produces two binaries: the main `manager` and a separate `cloudmanager` for XKS (external Kubernetes) deployments.

## Architectural Analysis

The rhods-operator implements a sophisticated hierarchical control plane architecture. At the top level, the DSCI controller bootstraps the platform — creating namespaces, deploying network policies, establishing the Auth and GatewayConfig service singletons, and configuring the monitoring stack. The DSC controller sits one level below, iterating over a component registry to create/update/delete component CRs based on the user's desired state. Each component CR triggers its own dedicated controller, which deploys kustomize manifests from the `prefetched-manifests/` directory. This layered design provides clear separation of concerns: DSCI handles "where" (namespaces, networking, auth), DSC handles "what" (which components are enabled), and individual component controllers handle "how" (specific deployment details).

The Gateway controller is the most architecturally complex piece, implementing a full ingress stack with multiple modes. In RHOAI 3.x, the primary pattern is Gateway API + EnvoyFilter + kube-auth-proxy, replacing the older Route + oauth-proxy pattern from 2.x. The controller supports dual ingress modes — LoadBalancer (for cloud environments) and OcpRoute (for single-node OpenShift) — with auto-detection and mode change handling that deletes and recreates the Gateway when switching. Authentication supports three modes: integrated OpenShift OAuth (creates OAuthClient CR), external OIDC (reads client credentials from user-supplied secret), and none (external auth). The EnvoyFilter applies ext_authz for authentication enforcement and a Lua filter for token forwarding and cookie stripping, ensuring downstream applications receive clean authorization headers. Legacy dashboard redirects use a separate nginx deployment with OpenShift Routes to maintain backward compatibility with old URLs.

The monitoring controller demonstrates advanced CRD-conditional ownership — it dynamically watches MonitoringStack, TempoMonolithic, TempoStack, Instrumentation, OpenTelemetryCollector, Perses, and PrometheusRule CRDs only if they exist in the cluster. This prevents errors on clusters without the required operators while allowing seamless integration when they're installed. The monitoring stack includes exporter validation with strict security controls (max config size, prohibited names, nesting depth limits) to prevent abuse.

A notable design pattern is the use of Server-Side Apply (SSA) throughout the operator for conflict-free updates. All resource creation goes through a deploy action that uses SSA with field ownership, eliminating the need for three-way merges and enabling multiple controllers to manage different fields of the same resource. The garbage collection action complements this by cleaning up resources that should no longer exist based on the current desired state.

The cloud manager controllers (Azure AKS, CoreWeave) represent an extension of the operator for non-OpenShift Kubernetes environments. They use Helm charts (via k8s-manifest-kit) to deploy operator dependencies (cert-manager, LWS, Sail, Gateway API) that OpenShift provides natively. Each dependency has a ManagementPolicy (Managed/Unmanaged) allowing administrators to bring their own operators. The cloud managers include workarounds for real-world issues like the OSSM-12397 Istio webhook annotation bug.
