## Purpose
**Short**: The RHOAI Operator is the central control-plane operator for Red Hat OpenShift AI, managing the lifecycle of all AI/ML platform components, platform services, and gateway ingress infrastructure on OpenShift clusters.

**Detailed**: The RHOAI Operator (rhods-operator) serves as the meta-operator for the Red Hat OpenShift AI (RHOAI) platform. It uses two primary cluster-scoped Custom Resources — `DSCInitialization` (DSCI) for platform initialization and `DataScienceCluster` (DSC) for component lifecycle management — to declaratively manage the deployment and configuration of all RHOAI components including Dashboard, Workbenches (Jupyter notebooks), KServe (model serving), Data Science Pipelines, Model Registry, Kueue (job queueing), Ray, TrustyAI, Training Operator, Feast Operator, LlamaStack Operator, MLflow Operator, Spark Operator, Trainer, and Models-as-a-Service.

In RHOAI 3.x, the operator has transitioned to a Gateway API-based ingress architecture. The Gateway Controller deploys a `data-science-gateway` Gateway resource in the `openshift-ingress` namespace along with `kube-auth-proxy` for OAuth2/OIDC-based authentication, an EnvoyFilter for ext_authz integration, and supporting infrastructure. Components create HTTPRoute CRs referencing this parent gateway. The operator also supports two ingress modes: `OcpRoute` (ClusterIP + OpenShift Routes) and `LoadBalancer` (external LB with DNS hostname). Dashboard redirect routes are created to transition users from legacy dashboard/gateway URLs to the new gateway hostname.

The operator manages component deployment through a reconciliation framework that renders kustomize manifests from embedded (prefetched) manifests at `/opt/manifests/` and applies them via server-side apply. Each component has a dedicated controller with management states (Managed, Unmanaged, Removed). Platform services (Gateway, Auth, Monitoring, CertConfigMapGenerator, Setup) are always-on infrastructure that supports the component ecosystem.

**Gateway API Architecture (RHOAI 3.x)**:
- The operator creates a `GatewayClass` named `data-science-gateway-class` with controller `openshift.io/gateway-controller/v1`
- A `Gateway` named `data-science-gateway` is deployed in `openshift-ingress` namespace with HTTPS listener on port 443
- The `kube-auth-proxy` (OAuth2 Proxy) deployment handles authentication via OpenShift OAuth or external OIDC
- An `EnvoyFilter` (`data-science-authn-filter`) injects ext_authz + Lua filters into the gateway's Envoy for authentication enforcement
- A `DestinationRule` (`data-science-tls-rule`) configures SIMPLE TLS for upstream connections on ports 443 and 8443
- Component controllers create `HTTPRoute` CRs referencing `data-science-gateway` as parent
- Two ingress modes are supported: `OcpRoute` (default, ClusterIP + OCP Route) and `LoadBalancer` (external LB)
- Legacy hostname redirect: When subdomain changes from `data-science-gateway` to `rh-ai`, a second listener handles the old hostname and redirects

**Dashboard Redirect Architecture**:
- An nginx-based redirect Deployment + Service + OCP Routes redirects legacy dashboard URLs to the new gateway hostname
- Creates OCP Route CRs for the legacy dashboard route name (`rhods-dashboard`) pointing to the nginx redirect service
- Creates OCP Route CR for legacy gateway subdomain (`data-science-gateway.{domain}`) redirecting to new subdomain (`rh-ai.{domain}`)
