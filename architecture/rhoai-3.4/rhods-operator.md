## Purpose

**Short**: Platform operator that orchestrates the entire Red Hat OpenShift AI (RHOAI) stack, managing component lifecycle, ingress infrastructure, authentication, monitoring, and cluster-wide configuration.

**Detailed**: The rhods-operator (also known as opendatahub-operator) is the central platform operator for RHOAI 3.x. It defines and reconciles two primary cluster-scoped CRDs -- DataScienceCluster (DSC) and DSCInitialization (DSCI) -- which together control the deployment and configuration of all AI/ML platform components. The DSC CR enables/disables individual components (Dashboard, KServe, Kueue, Workbenches, Model Registry, etc.), while DSCI handles platform-wide initialization including namespace creation, networking, monitoring, and authentication setup.

The operator deploys a complete multi-layer ingress stack using Kubernetes Gateway API with Envoy data plane, kube-auth-proxy for OAuth2/OIDC authentication, EnvoyFilter CRs for traffic shaping, and OpenShift Routes for legacy URL redirects. Each managed component gets its own component CR (e.g., `Dashboard`, `Kserve`, `Kueue`) with an independent reconciler that renders kustomize manifests into the cluster.

Beyond component lifecycle, the operator manages platform services: a full observability stack (MonitoringStack, OpenTelemetry Collector, Tempo, Perses, ThanosQuerier), RBAC for admin/allowed user groups, CA certificate distribution, hardware profile management, and cloud-specific controllers for Azure and CoreWeave Kubernetes environments. The operator also provides admission webhooks for singleton enforcement, hardware profile injection, connection secret management, and deprecation guardrails.

## Architectural Analysis

The rhods-operator is architecturally the most complex component in the RHOAI platform, functioning as both a meta-operator (managing other operators' lifecycles) and an infrastructure operator (deploying networking, auth, and observability stacks directly). This dual role creates a wide blast radius -- a bug in the operator can affect any component on the cluster.

**Component Registry Pattern**: The operator uses a registry-based architecture where each of 16+ components implements a common handler interface (Init, NewCRObject, NewComponentReconciler, UpdateDSCStatus, IsEnabled). The DSC controller iterates this registry to provision components, while each component gets its own independent reconciler. This pattern enables clean separation of concerns but creates a deep ownership hierarchy: DSC owns component CRs, which own kustomize-rendered resources. The registry also supports build-tag-based conditional compilation (e.g., `modelregistry_support.odh.go` vs `modelregistry_support.rhoai.go`) for platform-specific defaults.

**Multi-Layer Ingress Stack**: The RHOAI 3.x ingress architecture is significantly more complex than RHOAI 2.x. The gateway controller deploys a full chain: GatewayClass -> Gateway (with HTTPS listeners) -> Envoy proxy -> EnvoyFilter (ext_authz + Lua) -> kube-auth-proxy (OAuth2/OIDC) -> backend HTTPRoutes. This replaces the 2.x pattern of direct OpenShift Routes with oauth-proxy sidecars. The operator supports two ingress modes (LoadBalancer and OcpRoute/ClusterIP) with automatic detection and mode-change reconciliation. Legacy URL support is handled through nginx-based redirect Deployments with OpenShift Routes that 301-redirect old dashboard/gateway hostnames to the new Gateway API hostname.

**Observability Stack Depth**: The monitoring controller deploys a surprisingly deep observability stack: MonitoringStack (Prometheus + Alertmanager), OpenTelemetry Collector (metrics + traces), Tempo (TempoMonolithic or TempoStack), Perses (dashboards + datasources), ThanosQuerier, and multiple kube-rbac-proxy instances for secure access. The stack uses dynamic CRD detection (OwnsGVK with CrdExists predicates) to gracefully handle missing operator dependencies. Network policies restrict Prometheus access to only proxy pods, and the monitoring admission policy validates monitoring labels on resources.

**Singleton Resource Management**: Several critical resources (Auth, GatewayConfig, Monitoring, Tenant) are enforced as cluster-scoped singletons. The DSCI controller creates these during initialization and re-creates them if accidentally deleted (via deletion event watches). The webhooks additionally enforce that only one DSC and one DSCI can exist. This pattern prevents configuration drift but requires careful upgrade handling.

**Cloud Manager for XKS**: The cloudmanager binary enables RHOAI on non-OpenShift Kubernetes (Azure AKS, CoreWeave) by deploying prerequisite infrastructure via Helm charts: gateway-api CRDs, cert-manager, LeaderWorkerSet operator, and sail-operator (Istio). It includes workarounds for upstream bugs (e.g., annotating Istio webhooks to avoid sail-operator race conditions). This binary shares the same container image as the main operator but runs as a separate process.

**Webhook Complexity**: The operator registers 12 webhook endpoints covering singleton enforcement, hardware profile injection (CPU/memory/GPU resources, nodeSelector, tolerations, Kueue labels), connection secret management (notebooks, InferenceServices, LLMInferenceServices), monitoring label injection, and deprecation guards. The hardware profile webhook is particularly complex -- it handles profile changes, container name validation, cleanup on removal, and must work across Notebooks, InferenceServices, and LLMInferenceServices with different spec paths.
