RBAC Summary:
- No Kubernetes RBAC roles or bindings required
- Runs as non-root user (UID: 1001, GID: 0)
- No Kubernetes API access needed (standalone service)

TLS Configuration:
- Server TLS: rustls with ring crypto backend (NOT FIPS-validated)
- mTLS support: WebPkiClientVerifier with configurable client CA
- Client TLS: Per-service TLS configs (cert/key/CA per backend)
- Insecure mode: NoVerifier struct disables cert verification when insecure: true
  ⚠ WARNING: Disabling verification is a security risk in production

FIPS Compliance Notes:
- ⚠ rustls + ring backend is NOT FIPS-certified
- OpenSSL 3.5.1-7.el9_7 is installed in container but NOT used by the application
- Application TLS uses ring's BoringSSL-derived crypto (not FIPS-validated)
- For FIPS compliance: would need aws-lc-rs in FIPS mode or openssl crate

Secrets:
- TLS server keypair (kubernetes.io/tls) - Server cert/key via TLS_CERT_PATH, TLS_KEY_PATH
  Provisioned by: Platform operator / cert-manager
  Auto-Rotate: Depends on provisioner
- TLS client CA cert (Opaque) - mTLS client verification via TLS_CLIENT_CA_CERT_PATH
  Provisioned by: Platform operator
  Auto-Rotate: Depends on provisioner
- Client TLS configs (Opaque) - Per-service client TLS certs/keys (config.yaml tls section)
  Provisioned by: Administrator
  Auto-Rotate: No
- API tokens (Opaque) - Bearer tokens for HTTP backends (api_token env var)
  Provisioned by: Administrator
  Auto-Rotate: No

Authentication & Authorization:
- /api/v2/* endpoints: Optional mTLS (WebPkiClientVerifier enforcement)
- /api/v2/* endpoints: Header passthrough (X-Forwarded-Access-Token → Bearer rewrite)
- /health, /info: No authentication (open access)
