RBAC Summary:
- No RBAC defined — MLServer is a user-space application, not an operator
- RBAC is managed by KServe and the platform operator
- kube-rbac-proxy sidecar enforces Kubernetes RBAC for external access

Secrets:
- No Kubernetes Secrets directly referenced by MLServer
- Model storage credentials managed by KServe infrastructure
- TLS certificates managed by KServe gateway / Istio

Authentication & Authorization:
- Application Level: No authentication implemented
- All auth delegated to external infrastructure:
  - kube-rbac-proxy sidecar (RHOAI 3.x): Bearer token validation, RBAC enforcement
  - KServe Gateway (Gateway API / Envoy): TLS termination, auth policies
- Runtime Security: Trusted runtimes allowlist prevents arbitrary code execution
  - /etc/mlserver/trusted-runtimes.json (read-only, baked into image at build)
  - Blocks custom environment_tarball and environment_path in production mode
  - Wildcard CORS origins blocked in production mode

FIPS Compliance:
- No TLS at application level (all three servers use insecure ports)
- TLS termination delegated to external infrastructure
- Base image: UBI-based (quay.io/aipcc/base-images/cpu:3.4.0) provides FIPS-compliant OpenSSL
- Risk: OpenTelemetry exporter uses insecure=True for OTLP gRPC connections

Build Security:
- Hermetic builds (hermetic: true) via Konflux/Tekton pipeline
- pip prefetch with SHA-256 hashed requirements (requirements-cpu.txt)
- Red Hat-curated private PyPI index for Python packages
- Base image inherits FIPS-compliant OpenSSL from RHEL 9 UBI
