## Diagram Descriptions

### Component Dependency Graph
Shows how RHOAI 3.4's 53 components depend on each other. The rhods-operator (meta-operator) is the root node deploying 16+ sub-operators. Central components (odh-dashboard with 17 outgoing deps, KServe with dual controllers, odh-model-controller as the gateway bridge) are highlighted. Color-coded by category: red (meta-operator), blue (core platform), green (MLOps), purple (training), orange (evaluation), teal (MaaS), gray (runtimes/infra).

### Platform Network Topology
Complete network architecture showing RHOAI 3.4's Gateway API-centric ingress pattern. Covers:
- **Gateway API chain**: GatewayClass → Gateway → Envoy → kube-auth-proxy → HTTPRoutes
- **7 namespace types**: operator, platform, user, DSP, model registry, MaaS, legacy
- **Cross-namespace routing**: ReferenceGrant pattern for notebook, model, and Ray Dashboard access
- **Legacy paths**: Istio/Knative for Serverless KServe, nginx redirects for old URLs

**Two formats provided**:
- **Mermaid (.mmd)**: Visual diagram with color-coded trust zones. Great for presentations.
- **ASCII (.txt)**: Precise text format with complete TLS, auth, and RBAC summaries. Required for SAR submissions.

### Cross-Component Workflows
5 sequence diagrams showing end-to-end user workflows:
1. **Notebook → Pipeline**: Project creation, notebook launch, pipeline submission via kfp SDK
2. **LLM Serving**: LLMInferenceService → llmisvc-controller → vLLM + EPP routing → inference
3. **Models as a Service**: Multi-provider gateway with Authorino auth and API translation (OpenAI ↔ Anthropic/Azure/Vertex/Bedrock)
4. **Distributed Training**: RayCluster creation, head/worker coordination, Gateway HTTPRoute access
5. **Evaluation & Guardrails**: LMEvalJob execution, result logging to MLflow, runtime guardrails interception

### Platform Security Overview
Visual representation of RHOAI 3.4's defense-in-depth security architecture:
- **6 authentication mechanisms** (Gateway OAuth, kube-rbac-proxy, Kuadrant/Authorino, K8s API auth, Bearer forwarding, application RBAC)
- **8 key ServiceAccounts** with their ClusterRole bindings
- **5 secret categories** (webhook TLS, storage creds, gateway TLS, MaaS API keys, DB passwords)
- **6 TLS tiers** (Gateway, webhooks, metrics, internal service-CA, pod-to-pod optional, gRPC plaintext gap)
- **6 container security patterns** (UBI 81%, non-root 85%, distroless, multi-stage 69%, hermetic, digest-pinned)
- **Gateway API security chain** and legacy Istio mesh policies

### Platform Observability
Maps all monitoring and observability infrastructure:
- **Prometheus scrape targets**: 17+ endpoints with auth details (kube-rbac-proxy for controllers, None for runtimes, Bearer for pipelines)
- **Health probes**: Liveness + readiness for all 15 controllers (:8081) plus runtime-specific probes
- **Distributed tracing**: OpenTelemetry Collector → Tempo pipeline
- **KEDA autoscaling**: Prometheus-driven scaling for InferenceService pods (REQUEST_COUNT metric)
- **Perses dashboards**: Visualization layer querying Prometheus

### Platform Maturity Dashboard
High-level metrics for executive reviews:
- 53 components, 17 operators, 122 images, 104 CRDs, 46 gRPC services
- Security: 81% UBI, 85% non-root, FIPS enforced, 6 auth mechanisms
- API maturity: mix of v1 (GA), v1beta1, v1alpha1/v1alpha2
- Deployment: HA via leader election, full disconnected support, 4-architecture multi-arch
- Key gap: gRPC plaintext (46 services), optional pod-to-pod TLS
