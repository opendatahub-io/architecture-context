## Diagram Descriptions

### Component Dependency Graph
Shows how RHOAI 3.4's 57 components depend on each other, organized in a 3-tier namespace model: operator tier (rhods-operator), platform tier (redhat-ods-applications), and external dependencies. Central components (rhods-operator with 194 watches, KServe with 9 reverse dependencies, odh-dashboard with 12 outgoing dependencies) are highlighted. The LLM serving stack (llm-d, EPP, WVA, MaaS, Batch Gateway) is shown as a distinct subgraph reflecting its "platform-within-a-platform" architecture. Useful for understanding blast radius of changes and planning upgrades.

### Platform Network Topology
Complete network architecture showing the centralized data-science-gateway (Envoy) ingress, dual-mode ingress pattern (20+ HTTPRoutes + 26 legacy OpenShift Routes), service mesh communication (119 ClusterIP services, 77 HTTP, 46 HTTPS, 18 gRPC), and egress destinations. Includes exact port numbers, protocols, encryption, and authentication mechanisms for each path.

**Two formats provided**:
- **Mermaid (.mmd)**: Visual diagram with color-coded trust zones (External, Ingress, Platform, User Workloads, Monitoring, Egress). Great for presentations and architecture discussions.
- **ASCII (.txt)**: Precise text format with no ambiguity. Shows every ingress point with port mappings, auth mechanisms, and the 7-pattern authentication architecture. Required for SAR (Security Architecture Review) submissions and compliance documentation.

Both formats contain the same information, just different representations.

### Cross-Component Workflows
Four sequence diagrams showing end-to-end user workflows:

1. **LLM Inference via LLMInferenceService** - The new Gateway API path: user creates LLMInferenceService CR, llmisvc-controller creates InferencePool + WVA ScaledObject, odh-model-controller creates HTTPRoute on data-science-gateway, EPP performs KV-cache-aware routing via ext-proc, vLLM worker executes inference.

2. **ML Pipeline Execution** - Pipeline submission through DSP API Server, Argo Workflows scheduling, step pod execution with S3 artifact management, ML Metadata lineage tracking, Model Registry version storage.

3. **Interactive Development (Notebook Workbench)** - Dashboard creates Notebook CR, dual notebook controllers (ODH + upstream) manage pod lifecycle with kube-rbac-proxy sidecar, Gateway creates HTTPRoute, kube-auth-proxy authenticates access.

4. **Model Evaluation and Guardrails** - TrustyAI evaluation jobs, MLflow experiment tracking, FMS Guardrails Orchestrator detector pipeline, vllm-orchestrator-gateway content filtering.

### Platform Security Overview
Visual representation of RHOAI 3.4's defense-in-depth security architecture:
- **7 authentication mechanisms**: kube-rbac-proxy, OIDC JWT, OpenShift OAuth, SA tokens, Kuadrant AuthPolicy, API keys, Istio mTLS
- **6 most privileged ClusterRoles** with resource access details
- **6 categories of critical secrets** (TLS certs, S3 credentials, DB passwords, API keys)
- **6 TLS configuration patterns** (webhooks, metrics, internal services, gateway, Istio, pod-to-pod)
- **Container security**: UBI 9 (133/384), non-root (248/384), distroless (32), FIPS crypto, digest pinning
- **Network security**: 30 NetworkPolicies, namespace isolation, centralized gateway

### Platform Observability
Maps all monitoring and observability endpoints for RHOAI 3.4:

- **ServiceMonitor creation patterns** (operators, odh-model-controller, trustyai, DSPO)
- **Logging format matrix**: Go (zap JSON), Python (mixed), Java (Quarkus), Rust (tracing crate)
