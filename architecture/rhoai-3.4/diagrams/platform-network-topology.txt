CROSS-NAMESPACE NETWORKING
================================================================================

The platform uses ReferenceGrant resources to enable cross-namespace routing:

  Platform NS (opendatahub)          User NS (project-*)
  ┌────────────────────┐             ┌────────────────────┐
  │  HTTPRoute         │             │  ReferenceGrant    │
  │  spec:             │ ────────→   │  from: opendatahub │
  │    backendRefs:    │  permitted  │  to: Services in   │
  │    - name: svc     │  by grant   │    this namespace  │
  │      namespace:    │             │                    │
  │      project-*     │             │  (created by       │
  │                    │             │   Kubeflow/KubeRay  │
  └────────────────────┘             │   controllers)     │
                                     └────────────────────┘

Components creating cross-namespace HTTPRoutes:
  - Kubeflow Notebook Controller (notebook access)
  - KubeRay AuthenticationController (Ray Dashboard access)
  - odh-model-controller (InferenceService endpoints)
  - MLflow Operator (MLflow UI access)
  - Feast Operator (Feast UI access)

================================================================================
TLS CONFIGURATION SUMMARY
================================================================================

External TLS:
  - Gateway HTTPS listeners: TLS 1.3, OpenShift service-CA certificates
  - Legacy OpenShift Routes: TLS 1.2+ (edge termination)

Internal TLS (service-CA signed):
  - Webhook servers (9443/TCP): All operators with admission/conversion webhooks
  - Metrics endpoints (8443/TCP): kube-rbac-proxy sidecars (FIPS cipher suites)
  - odh-dashboard: service-CA TLS for backend connections

Pod-to-Pod TLS (optional):
  - KServe InferenceService: PodToPodTLS flag (opt-in)
  - llm-d components: TLS between scheduler sidecar and vLLM

gRPC (plaintext within cluster):
  - ML Metadata, MLServer V2 protocol, Caikit, guardrails orchestrator
  - gateway-api-inference-extension, Model Registry (46 gRPC services total)
  - Risk: inference payloads traverse cluster network unencrypted by default

Istio mTLS (legacy, Serverless mode only):
  - Mode: STRICT for KServe Serverless InferenceServices
  - PeerAuthentication: Namespace-scoped
  - AuthorizationPolicy: Model Registry (optional)

FIPS Cipher Suites:
  - ECDHE+AES-GCM with NIST P-256/P-384 curves
  - Enforced in: kube-rbac-proxy sidecars, odh-model-controller,
    TrustyAI Service Operator HTTPS servers
  - UBI base images include FIPS-validated OpenSSL

RBAC SUMMARY (Top Privileged Roles)
================================================================================

  rhods-operator:
    - Full cluster-admin equivalent for platform CRDs
    - manages 16+ component lifecycles
    - Creates/patches Deployments, Services, ConfigMaps, RBAC across namespaces

  kserve-controller-manager:
    - serving.kserve.io/*, serving.knative.dev/*
    - core/services, core/secrets, apps/deployments
    - networking.k8s.io/ingresses, gateway.networking.k8s.io/*

  odh-model-controller:
    - Creates HTTPRoutes, ServiceMonitors, KEDA resources
    - Patches InferenceService status
    - Cross-namespace resource creation

  kubeflow-notebook-controller:
    - Creates StatefulSets, Services in user namespaces
    - Creates HTTPRoutes in platform namespace
    - Creates ReferenceGrants in user namespaces
    - Mutating webhook for pod injection

  kuberay-operator:
    - ray.io/rayclusters, ray.io/rayjobs, ray.io/rayservices
    - Creates pods, services across user namespaces
    - Mutating webhook for pod defaults
