NAMESPACES:

- redhat-ods-applications:   Platform services, Gateway, controllers (14 operators)
- redhat-ods-monitoring:     Prometheus, monitoring stack, ServiceMonitors
- istio-system:              Istio control plane (KServe serverless, MaaS)
- knative-serving:           Knative Serving runtime (KServe serverless mode)
- User namespaces:           Per-project workloads (Notebooks, InferenceServices,
                             Pipelines, Ray clusters, Training jobs, etc.)

INTERNAL SERVICE MESH CONFIGURATION:
- Service Types: 119 ClusterIP, 6 Headless (of 177 total services)
- Protocols: HTTP (77 services), HTTPS (46 services), gRPC (18 services)
- mTLS: MIXED — Istio STRICT for KServe serverless; service-ca TLS for
  platform services; configurable pod-to-pod TLS for pipelines
- Certificate Management: OpenShift service-ca (serving-cert-secret-name)
- NetworkPolicies: 30 policies across 10 components

INGRESS ARCHITECTURE (Dual-Mode — Gateway API + Legacy Routes):
- Primary: data-science-gateway (Gateway CR) with Envoy data plane
  - 20+ HTTPRoute resources attached
  - TLS termination at Envoy
  - Auth: kube-auth-proxy sidecar or Kuadrant AuthPolicy
- Legacy: 26 OpenShift Routes with OAuth proxy sidecars
  - Being migrated to Gateway API pattern
  - Retained for backward compatibility

AUTHENTICATION MECHANISMS:
1. kube-rbac-proxy (SubjectAccessReview + TokenReview)
   - Components: odh-dashboard, trustyai, Notebooks, Model Registry, Feast
2. OIDC JWT validation
   - Components: kube-auth-proxy, Feast (OIDC mode), MaaS (Kuadrant)
3. OpenShift OAuth bearer tokens (sha256~ format)
   - Components: kube-auth-proxy, odh-dashboard (legacy)
4. Kubernetes ServiceAccount tokens
   - Components: All operators, KServe, DSP
5. Kuadrant AuthPolicy (CEL + OPA)
   - Components: MaaS, odh-model-controller (LLMInferenceService)
6. API key management (MaaSAPIKey CRD)
   - Components: MaaS, AI Gateway Payload Processing
7. mTLS (Istio sidecar)
   - Components: KServe serverless, MaaS
