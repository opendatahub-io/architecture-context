TRUST BOUNDARY ANALYSIS:
━━━━━━━━━━━━━━━━━━━━━━━━
- API Client → Gateway: UNTRUSTED. No authentication enforced at gateway.
  Gateway MUST be protected by upstream infrastructure (network policies,
  kube-rbac-proxy sidecar, or service mesh) in production.
- Gateway → Orchestrator: SEMI-TRUSTED. Optional mTLS when TLS certs are
  present (dynamically detected). Auth header forwarded for downstream
  enforcement.
- Orchestrator → Detectors/vLLM: Internal to orchestrator (not traversing
  gateway). Trust depends on orchestrator's own configuration.

CRYPTO CONFIGURATION:
━━━━━━━━━━━━━━━━━━━━━━
- TLS Library: openssl 0.10.73 (Rust bindings) + native-tls 0.2.12 (system backend)
- Certificate Handling: PKCS12 in-memory from PEM certs (openssl → native-tls bridge)
- Custom CA: Loaded from /etc/tls/ca/service-ca.crt
- Non-FIPS Risks: danger_accept_invalid_hostnames(true) on localhost only
- Cipher Suites: System OpenSSL defaults (no explicit configuration)
- FIPS: Rust binary (not Go). Dynamic linking against system OpenSSL.
  compat-openssl11 installed in release image.
