RBAC Summary:
- No Kubernetes RBAC defined — standalone service deployed as a Pod
- RBAC is managed entirely by the RHOAI platform operator (rhods-operator)
- Platform-level auth via kube-rbac-proxy or HTTPRoute policies

Secrets:
- LLM API Key (Opaque) - OPENAI_API_KEY, ANTHROPIC_API_KEY, etc. - Manual/Platform, no auto-rotate
- Azure OpenAI Config (Opaque) - Azure endpoint, API version, API key - Manual/Platform, no auto-rotate
- Cohere API Key (Opaque) - COHERE_API_KEY - Manual/Platform, no auto-rotate
- Redis Credentials (Opaque) - Redis username/password - Manual/Platform, no auto-rotate

Authentication Flow:
- External auth: Managed by RHOAI platform ingress (kube-rbac-proxy / HTTPRoute)
- LLM auth: Authorization header forwarded from client → guardrails server → LLM provider
- X-* headers: Forwarded from client to LLM provider (auth tokens redacted in logs)
- Internal services (8000, 8001): No auth (protected by cluster network)

FIPS Compliance:
- Python runtime inherits FIPS from UBI9 base image OpenSSL
- MD5 used for cache keys with FIPS-aware fallback to SHA-256 (utils.py:409-417)
- Embedding cache MD5KeyGenerator lacks FIPS fallback (SHA256KeyGenerator available as alternative)
- HTTP clients (aiohttp, httpx) use system OpenSSL respecting OS-level FIPS mode

Container Security:
- Base image: UBI9/python-312
- Runs as non-root user 1001
- Virtual environment at /app/.venv
- Pre-downloaded ML models for air-gapped deployments
- Build hermeticity: RPM lockfile + poetry.lock + artifacts.lock.yaml
