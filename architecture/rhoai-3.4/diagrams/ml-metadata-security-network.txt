INGRESS:
  None. The MLMD gRPC server has no external ingress. It is accessed
  only by cluster-internal clients (DSP backend, pipeline steps).

EGRESS:
  - PostgreSQL: 5432/TCP | Optional SSL | Password + client certs
  - MySQL:      3306/TCP | Optional SSL | Password + client certs
  (No external internet egress required at runtime)

RBAC Summary:
  No Kubernetes RBAC definitions in this repository.
  The MLMD server is a stateless gRPC service without Kubernetes API access.
  RBAC for the MLMD pod is managed by the deploying Data Science Pipelines operator.

Secrets:
  - DB connection credentials (Opaque)
    - Purpose: PostgreSQL/MySQL username and password
    - Provisioned by: Data Science Pipelines operator
    - Auto-Rotate: No
  - SSL certificates (kubernetes.io/tls) [Optional]
    - Purpose: TLS certs for gRPC server and database connections
    - Provisioned by: Deploying operator / manual
    - Auto-Rotate: No

Authentication & Authorization:
  - Endpoint: MetadataStoreService (all RPCs)
  - Auth: Optional mTLS (SSLConfig.client_verify)
  - Enforcement: C++ grpc::SslServerCredentials
  - Policy: If SSLConfig with client_verify=true, clients must present
    valid TLS certificates. Otherwise, no application-level authentication.
  - Network-level isolation (NetworkPolicies, namespace boundaries)
    provides primary access control in RHOAI deployments.

FIPS Compliance Concerns:
  - gRPC TLS: Uses BoringSSL compiled from source (NOT FIPS-validated)
    Risk: gRPC connections may not use FIPS-validated crypto modules
  - Database SSL: Uses system OpenSSL via libpq (FIPS-validated on UBI9)
  - Recommendation: Evaluate replacing BoringSSL with system OpenSSL for
    gRPC TLS, or use FIPS-validated BoringCrypto module

Container Security:
  - Base image: UBI9-minimal (Red Hat Universal Base Image)
  - Runtime user: UID 65534 (non-root)
  - Build: Multi-arch (x86_64, arm64, ppc64le)
  - Hermeticity: Non-hermetic (hermetic: false in Tekton pipeline)
