## Purpose

**Short**: A CLI-driven build-time pipeline that collects, enriches, and packages AI model metadata and MCP server catalogs into a static data container image consumed by the RHOAI dashboard.

**Detailed**: The model-metadata-collection component is a data pipeline — not a runtime service — that runs during CI/CD to produce curated YAML catalog files. It discovers Red Hat AI validated models from HuggingFace collections, extracts modelcard metadata from OCI container registry images (modelcar format), enriches that metadata with HuggingFace API data (license, provider, description, language, tasks, tool-calling config, vLLM recommended configurations), and generates structured catalog YAML files. It also produces MCP (Model Context Protocol) server catalogs by reading input YAML definitions and enriching them with OCI registry metadata (architectures, timestamps).

The final output is a minimal container image (`ubi9-minimal` base) that contains only the generated YAML catalog and index files under `/app/data/`. This image is deployed as a data volume source — it runs `sleep infinity` and other platform components (such as the RHOAI dashboard) mount its `/app/data/` volume to read catalog data. The Go CLI tools (`model-extractor` and `metadata-report`) run only during CI/CD and are never included in the shipped container image.

## Architectural Analysis

The model-metadata-collection component follows a **build-time data pipeline** pattern rather than a traditional runtime service architecture. The Go source code — two CLI tools totaling ~4,000 lines across 20+ packages — exists solely to produce static YAML files during CI/CD. The shipped artifact is a ~15MB container image containing nothing but those YAML files and a `sleep infinity` entrypoint. This separation between the build-time pipeline (complex Go code with external API integrations) and the runtime artifact (trivially simple data container) is a deliberate architectural choice that minimizes attack surface and runtime complexity.

The metadata enrichment pipeline implements a sophisticated **multi-source data fusion** strategy with explicit provenance tracking. Each metadata field (name, provider, description, license, tasks, language, etc.) can originate from multiple sources ranked by priority: HuggingFace YAML frontmatter (highest) → HuggingFace API → HuggingFace tags → HuggingFace README regex → container modelcard YAML frontmatter → container modelcard regex (lowest). The `MetadataSource` type tracks both value and source for every field, and a separate `enrichment.yaml` file records which source won for each field per model. This design enables the `metadata-report` tool to audit data quality across the entire catalog.

The **concurrency model** in the model extractor uses a bounded semaphore pattern (`maxConcurrent=5` by default) with a WaitGroup for parallel model processing. Each model's OCI registry access, modelcard extraction, and metadata parsing runs in a separate goroutine. The HuggingFace API calls include retry logic with exponential backoff (max 3 retries, 2-minute overall timeout). The MCP server enrichment pipeline similarly enriches artifacts in sequence but with per-artifact retry logic for registry operations.

A notable architectural decision is the **model family guardrail** in enrichment matching. The `isCompatibleModelFamily()` function (using a centralized family list in `config/model_families.go`) prevents cross-family matching during HuggingFace enrichment — e.g., preventing a Llama container from being enriched with Granite HuggingFace metadata, even if the similarity score is high. The family list also drives version normalization in `NormalizeModelName()`, ensuring that version-embedded model names like "granite-3-1" normalize consistently for matching.

The component also manages **MCP (Model Context Protocol) server catalogs** as a secondary pipeline. MCP server metadata is manually curated in `input/mcp_servers/` YAML files, enriched with OCI registry data (architecture lists, timestamps), and aggregated into catalog files. This dual-catalog design (models + MCP servers) reflects the RHOAI dashboard's need to present both model serving and agent-tool-serving catalogs.
