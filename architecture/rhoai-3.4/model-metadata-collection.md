## Purpose

**Short**: A Go-based build-time tool that extracts, enriches, and catalogs AI model metadata and MCP server metadata from container registries and HuggingFace, producing YAML catalog files shipped in a minimal data-only container image.

**Detailed**: Model Metadata Collection is a pipeline tool, not a runtime service. It has two CLI binaries: `model-extractor` (the primary pipeline) and `metadata-report` (a reporting utility). The `model-extractor` binary discovers Red Hat AI validated model collections from HuggingFace, fetches OCI container images from registries (registry.redhat.io), extracts modelcard.md files from annotated container layers, enriches metadata from HuggingFace YAML frontmatter and APIs, and generates structured YAML catalogs. It also generates MCP (Model Context Protocol) server catalogs from curated input YAML files, enriching them with OCI registry architecture and timestamp data.

The final output is a set of pre-generated YAML catalog files (`models-catalog.yaml`, `validated-models-catalog.yaml`, `other-models-catalog.yaml`, and MCP server catalogs for Red Hat, partner, and community sources). These are checked into the `data/` directory by CI/CD, then the Konflux Dockerfile copies them into a minimal `ubi9-micro` container image. This image runs `sleep infinity` and serves as a data volume source — other platform components (e.g., the RHOAI dashboard) mount or copy catalog files from this container to display model and MCP server information to users. The tool itself never runs in production; it runs in CI pipelines and developer workstations.

## Architectural Analysis

The model-metadata-collection repository follows an unusual pattern for the RHOAI platform: it is a **build-time data pipeline**, not a runtime service or operator. The Go binaries (`model-extractor` and `metadata-report`) never execute in production. Instead, they run during CI/CD to generate YAML catalog files, which are committed to the `data/` directory and then packaged into a minimal container image. This container image contains no executable code — it runs `sleep infinity` and serves purely as a data volume source. This design decouples catalog generation (which requires network access to HuggingFace and container registries) from runtime deployment (which needs only static YAML files).

The metadata enrichment pipeline implements a sophisticated priority hierarchy for data sources. HuggingFace YAML frontmatter takes highest priority, followed by container modelcard.md content (parsed via YAML frontmatter first, then regex extraction), HuggingFace API data, HuggingFace repository tags, and finally generated defaults. Each field in the output metadata tracks its provenance (e.g., `huggingface.yaml`, `modelcard.regex`, `generated`), and a separate `enrichment.yaml` file records the data source for every field. This provenance tracking enables the `metadata-report` tool to generate completeness and quality reports, giving visibility into how much metadata comes from structured YAML versus regex extraction.

The codebase includes several domain-specific intelligence features: model family detection prevents cross-family matching (e.g., a Llama container matching a Granite HuggingFace entry), version-aware name normalization preserves version numbers as complete tokens to prevent mismatches (3.1 vs 3.3), and a similarity scoring system with configurable thresholds determines match confidence. The tool-calling configuration extraction from HuggingFace README YAML frontmatter and the vLLM recommended configurations from static input files are rendered into the model README via Go `text/template` with embedded template files. The serving runtime override feature handles models that require a preview runtime image, injecting deployment instructions into the model README.

The MCP server catalog generation is a parallel pipeline that reads curated input YAML files (organized by source: Red Hat, partner, community), enriches them with OCI registry data (container architectures and timestamps), and produces aggregated catalog YAML files. This pipeline uses the same retry-with-exponential-backoff infrastructure as the model pipeline and shares the registry interaction layer. The Konflux Dockerfile copies all generated catalogs — both model and MCP server — into the final data container image.

A notable architectural decision is the use of `ubi9-micro` as the base image for the Konflux Dockerfile, which is the absolute minimum Red Hat base image with essentially no packages. This is appropriate since the image contains only YAML files and needs no runtime dependencies. The development Dockerfile uses `ubi9-minimal` which has slightly more packages but follows the same data-only pattern.
