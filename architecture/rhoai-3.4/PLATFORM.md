## Platform Overview

Red Hat OpenShift AI (RHOAI) 3.4 is an enterprise machine learning platform deployed on OpenShift Container Platform via OLM. A single meta-operator (rhods-operator) manages 16+ component lifecycles through a registry-based architecture where each component implements a common handler interface and receives its own independent reconciler. The platform spans the full ML lifecycle: notebook-based development (Kubeflow Notebook Controller), pipeline orchestration (Data Science Pipelines backed by Argo Workflows), model training (Kubeflow Training Operator, Kubeflow Trainer v2, KubeRay), model serving (KServe, vLLM, OpenVINO, MLServer, Caikit), model registry (Model Registry), feature management (Feast), experiment tracking (MLflow), model evaluation (TrustyAI, LMEval, EvalHub), guardrails (NeMo Guardrails, FMS Guardrails Orchestrator), and multi-provider model access (Models as a Service).

RHOAI 3.4 introduces a Gateway API-centric ingress architecture that replaces the RHOAI 2.x pattern of direct OpenShift Routes with oauth-proxy sidecars. The rhods-operator deploys a full chain: GatewayClass, Gateway (with HTTPS listeners), Envoy proxy, EnvoyFilter (ext_authz + Lua), kube-auth-proxy (OAuth2/OIDC), and backend HTTPRoutes. Two ingress modes are supported (LoadBalancer and OcpRoute/ClusterIP) with automatic detection and mode-change reconciliation. Legacy URL support is maintained through nginx-based redirect Deployments with OpenShift Routes that 301-redirect old hostnames to the new Gateway API hostname. The platform also deploys a deep observability stack: MonitoringStack (Prometheus + Alertmanager), OpenTelemetry Collector (metrics + traces), Tempo (TempoMonolithic or TempoStack), and Perses dashboards.

The security posture enforces defense-in-depth: UBI-based container images across 43 components, non-root execution in 45 components, FIPS-compliant TLS cipher suites in controller-to-controller communication, kube-rbac-proxy sidecars for metrics and API endpoints, Kuadrant/Authorino for multi-tenant API gateway authentication, OpenShift service-CA for internal TLS certificates, and NetworkPolicies restricting component-to-component traffic. The platform supports disconnected (air-gapped) deployments via the RELATED_IMAGE pattern and OLM relatedImages, with all 122 images pinned by digest.

### Central Components

**rhods-operator** is the platform's root orchestrator with the highest fan-out: it deploys and manages all other operators and controllers. It receives 13 incoming dependency references (components declaring it as a deployment source or configuration target) and emits 16 outgoing dependencies.

**odh-dashboard** is the primary user-facing integration hub with 17 outgoing dependencies, connecting to virtually every platform service (pipelines, serving, registry, training, notebooks, MLflow, MaaS, TrustyAI, Feast). It acts as both a Module Federation host for micro-frontends and a reverse proxy for backend-for-frontend services.

**KServe** is the serving control plane with the widest blast radius in the data plane. It manages InferenceService and LLMInferenceService lifecycles, receives model serving requests from dashboard, model registry, and odh-model-controller, and orchestrates runtime containers (vLLM, OpenVINO, MLServer, Caikit). The RHOAI 3.4 split into two independent controller managers (kserve-controller-manager and llmisvc-controller-manager) allows the LLM serving path to iterate independently.

**odh-model-controller** bridges KServe and the platform gateway, creating Routes/HTTPRoutes, ServiceMonitors, KEDA autoscaler resources, and Envoy filters for every InferenceService. It composes six sub-reconcilers per InferenceService using a delta-processing pattern.

**Kubeflow Notebook Controller** manages notebook pod lifecycles and injects the RHOAI ingress pattern (cross-namespace HTTPRoutes, ReferenceGrants, kube-rbac-proxy sidecars) into user workspaces.

## Deployment Architecture

### Deployment Topology

The platform follows a three-tier namespace topology. The **operator tier** runs in `redhat-ods-operator` (or `openshift-operators` for global installs) and contains only the rhods-operator pod managed by OLM. The **platform tier** runs in `opendatahub` (or `redhat-ods-applications`) and contains all platform controllers, the dashboard, the Gateway (Envoy proxy), the observability stack (Prometheus, Alertmanager, OTel Collector, Tempo, Perses), and platform-level HTTPRoutes for notebooks and other user-facing services. The **user tier** consists of dynamically-created user namespaces containing notebook StatefulSets, pipeline runs, RayCluster pods, training jobs, InferenceService model servers, and FeatureStore instances. Cross-namespace networking between the platform and user tiers is enabled by ReferenceGrant resources that permit HTTPRoutes in the platform namespace to reference Services in user namespaces.

#### Manifest Consumption Pipeline

Component repositories maintain their own kustomize bases and overlays. The `get_all_manifests.sh` script in the rhods-operator repository clones each component repository at a pinned commit, copies the relevant manifests into the operator's `opt/manifests/` directory, and applies any necessary patches. At reconcile time, the rhods-operator renders these manifests using kustomize with distribution-specific overlays and applies them to the cluster via Server-Side Apply. Image references are injected from RELATED_IMAGE environment variables (populated from the OLM CSV relatedImages), ensuring that the operator never references floating tags and all images are digest-pinned for both security and disconnected deployment support.

### Disconnected Support

RHOAI 3.4 fully supports air-gapped (disconnected) deployments. All 122 shipped container images are pinned by `@sha256:` digest in the OLM CSV `relatedImages` section and exposed to the operator via `RELATED_IMAGE_*` environment variables. The rhods-operator propagates these environment variables to sub-operators (e.g., DSPO, MaaS Controller) so that child components also resolve images from the local mirror rather than upstream registries. Image mirroring is performed using `oc-mirror` or `oc adm catalog mirror`, which reads the relatedImages list from the CSV and mirrors all referenced digests to the disconnected registry. The platform also supports custom CA bundles for registry TLS via the `odh-trusted-ca-bundle` ConfigMap, which is injected into operator and workload pods. The Notebooks component uses hermetic Konflux builds with `rpms.lock.yaml` and pip lock files to ensure build reproducibility without network access.

## Platform Architectural Analysis

RHOAI 3.4 exhibits several cross-cutting architectural patterns that emerge only when examining the platform as a whole.

**The meta-operator anti-pattern and its mitigations.** rhods-operator functions as both a meta-operator (managing other operators' lifecycles) and an infrastructure operator (deploying networking, auth, and observability stacks directly). This dual role creates a wide blast radius: a bug in rhods-operator can affect any component on the cluster. The registry-based architecture mitigates this by giving each component an independent reconciler with its own error handling and status reporting, but the sequential reconciliation of 16+ components still creates long reconciliation cycles where a single failure can block downstream components. The build-tag-based conditional compilation (`_support.odh.go` / `_support.rhoai.go`) adds compile-time platform specialization but makes it impossible to test ODH and RHOAI code paths from the same binary.

**Authentication fragmentation.** The platform uses at least six distinct authentication mechanisms, each with different trust boundaries and failure modes. Gateway-level OAuth (kube-auth-proxy) protects user-facing UIs, kube-rbac-proxy sidecars protect metrics and API endpoints, Kuadrant/Authorino protects the MaaS multi-tenant gateway, Feast implements its own OIDC-based permission system stored in its registry, Model Registry uses SelfSubjectAccessReview with forwarded tokens, and EvalHub maps tenant headers to namespace-scoped SAR checks. While each mechanism is individually well-implemented, the diversity means there is no single authentication policy language or audit point for the platform. A security administrator must understand six different systems to audit access control across the platform.

**The Gateway API migration is incomplete but well-managed.** RHOAI 3.4 represents a transitional state where the primary ingress mechanism has moved to Gateway API (GatewayClass + Gateway + HTTPRoutes + Envoy + kube-auth-proxy), but Istio VirtualServices and OpenShift Routes remain for specific use cases. KServe Serverless mode still uses Istio DestinationRules and Knative Routes; legacy URL redirects use nginx Deployments with OpenShift Routes; and Model Registry can optionally use Istio AuthorizationPolicies. The coexistence is managed through operator-level ingress mode detection and reconciliation, but it means the platform must maintain expertise and testing for three ingress technologies simultaneously.

**Consistent controller patterns with divergent templating.** All 17 operators use controller-runtime with leader election, but their manifest rendering strategies diverge significantly. rhods-operator uses kustomize with distribution overlays; DSPO uses Go `text/template` with 70+ YAML templates; MLflow Operator and Spark Operator embed Helm charts; ogx-k8s-operator uses runtime kustomize plugins with custom transformers; and gateway-api-inference-extension uses standalone Helm charts. This diversity means operational teams need familiarity with four different templating systems to debug deployment issues across the platform. A unified rendering approach would reduce operational complexity.

**The gRPC plaintext gap.** While the platform enforces TLS for external-facing HTTP endpoints and supports FIPS-compliant cipher suites in controller-to-controller communication, the 46 gRPC services (MLServer, Caikit, ML Metadata, guardrails orchestrator, model registry) predominantly use plaintext within the cluster network. The assumption is that cluster networking provides sufficient isolation, but this creates a gap in defense-in-depth for the inference data path. The `PodToPodTLS` flag in KServe addresses this for model serving specifically, but it is optional and not enabled by default. Inference payloads, which may contain sensitive data, traverse the cluster network unencrypted in the default configuration.

**Training operator bifurcation.** The platform ships both the Kubeflow Training Operator (v1, with PyTorchJob, TFJob, MPIJob, JAXJob, PaddleJob, XGBoostJob CRDs) and Kubeflow Trainer (v2, with the unified TrainJob CRD and pluggable runtimes). This reflects the upstream Kubeflow migration from per-framework CRDs to a single TrainJob abstraction with RuntimeRef composition. Both operators are active in RHOAI 3.4, creating a dual-path training experience. Trainer v2 adds RHAI-specific features (progression tracking, metrics polling, annotation-based status reporting) not available in v1, incentivizing migration, but the v1 CRDs remain for backward compatibility.

**LLM serving stack complexity.** The LLM serving path through KServe + llm-d + gateway-api-inference-extension involves a deep component chain: LLMInferenceService CR triggers llmisvc-controller, which creates an InferencePool, Deployment (vLLM + scheduler sidecar + optional KV cache sidecar), ConfigMaps, TLS certificates, and Services; odh-model-controller then creates HTTPRoutes, ServiceMonitors, and KEDA resources; Envoy routes to EPP (ext_proc) for scheduling decisions; EPP scrapes model server metrics and selects the optimal pod; the request finally reaches vLLM. This depth provides sophisticated features (prefix-cache-aware routing, SLO-based scheduling, disaggregated prefill/decode) but makes debugging inference failures challenging since the request traverses 5+ components before reaching the model.

**Supply chain maturity varies across components.** Core platform components (rhods-operator, notebooks, pipelines-components, model-registry) use full hermetic builds with Konflux/Cachi2, architecture-specific lock files, and reproducible builds. However, many components still use non-hermetic builds with standard Dockerfiles that pull dependencies at build time. The inconsistency means supply chain security guarantees are not uniform across the platform's 122 shipped images.
