## Platform Overview

Red Hat OpenShift AI (RHOAI) 3.4 is an enterprise AI/ML platform that runs on OpenShift Container Platform v4.19-v4.21. It provides a comprehensive, integrated set of tools for the full machine learning lifecycle: interactive development (Jupyter workbenches), distributed training (Kubeflow Training Operator, KubeRay, Kueue), ML pipelines (Data Science Pipelines/Kubeflow Pipelines), model serving (KServe, vLLM, OpenVINO, MLServer), model registry and metadata tracking (Model Registry, MLflow, ML Metadata), LLM inference optimization (llm-d scheduler, KV-cache, batch gateway, workload-variant autoscaler), AI safety and trustworthiness (TrustyAI, guardrails orchestrator, NeMo Guardrails, EvalHub), feature stores (Feast), and LLM application frameworks (Llama Stack). The platform ships 86 container images and is FIPS-compliant for all Go-based components.

The platform architecture follows a layered operator model. The central **rhods-operator** (deployed via OLM) manages the lifecycle of all platform components through two primary cluster-scoped CRDs: `DataScienceCluster` (DSC) for component enablement and `DSCInitialization` (DSCI) for platform initialization. Each AI/ML capability is managed by a dedicated sub-operator or controller, deployed via kustomize manifests rendered at reconciliation time. In RHOAI 3.x, the platform has transitioned to a Gateway API-based ingress architecture, deploying a `data-science-gateway` Gateway resource in the `openshift-ingress` namespace with kube-auth-proxy providing OAuth2/OIDC authentication via ext_authz integration. Components create HTTPRoute CRs referencing this parent gateway for external access.

The platform emphasizes security-by-default: all operator images are built with FIPS-compliant Go (`GOEXPERIMENT=strictfipsruntime`), pod-to-pod TLS is supported via OpenShift service-CA certificates, authentication is enforced through kube-rbac-proxy sidecars with SubjectAccessReview checks, and network policies isolate sensitive data plane components. Multi-tenancy is supported through Kubernetes RBAC, namespace isolation, and service mesh integration (Istio mTLS for legacy deployments).

### Central Components

1. **rhods-operator**: The meta-operator orchestrating all 20+ component deployments. Every managed component depends on it.
2. **KServe**: Core model serving infrastructure consumed by vLLM, OpenVINO, MLServer, Model Registry, odh-model-controller, and the llm-d stack.
3. **data-science-gateway (Gateway API)**: Central ingress point used by Dashboard, KServe, MLflow, Model Registry, and other components via HTTPRoute CRs.
4. **kube-auth-proxy**: Authentication/authorization proxy injected as sidecar into Dashboard, KServe InferenceServices, KubeRay, and other components.
5. **Data Science Pipelines**: ML pipeline platform integrating with Argo Workflows, KServe, Model Registry, and training operators.

## Data Flows

### Key Platform Workflows

#### Workflow 1: Model Training to Deployment

| Step | Component | Action | Next Component |
|------|-----------|--------|----------------|
| 1 | Notebook (Workbench) | User develops and trains model | Uploads to S3/PVC |
| 2 | Data Science Pipelines | Orchestrates training pipeline | Creates Workflow CRs |
| 3 | Argo Workflows | Executes training steps as pods | Reports status to Persistence Agent |
| 4 | Model Registry | Registers trained model metadata | Stores version/artifact info |
| 5 | KServe | Creates InferenceService from registry | Deploys model serving pod |
| 6 | odh-model-controller | Configures route, monitoring, auth | Creates HTTPRoute, PodMonitor |

#### Workflow 2: LLM Inference with Guardrails

| Step | Component | Action | Next Component |
|------|-----------|--------|----------------|
| 1 | Client | Sends chat completion request | data-science-gateway |
| 2 | data-science-gateway | Authenticates via kube-auth-proxy | Routes to InferenceService |
| 3 | vllm-orchestrator-gateway | Intercepts and routes to orchestrator | FMS Guardrails Orchestrator |
| 4 | FMS Guardrails Orchestrator | Runs input through detectors | Guardrails Detectors |
| 5 | Guardrails Detectors | Evaluates content safety | Returns detection results |
| 6 | FMS Guardrails Orchestrator | Forwards safe request to LLM | vLLM/inference server |
| 7 | vLLM | Generates response | Returns to orchestrator |
| 8 | FMS Guardrails Orchestrator | Runs output through detectors | Returns filtered response |

#### Workflow 3: Distributed Training with Kueue

## Deployment Architecture

### Deployment Topology

The RHOAI 3.4 platform follows a hub-and-spoke deployment model:

- **Hub (Operator Namespace - `redhat-ods-operator`)**: The rhods-operator runs here, deployed via OLM. It reconciles all component manifests and deploys sub-operators.
- **Platform Namespace (`redhat-ods-applications`)**: Core platform services (Dashboard, notebook controller, model controller, sub-operators) run here.
- **Ingress Namespace (`openshift-ingress`)**: The data-science-gateway Gateway, kube-auth-proxy, and EnvoyFilter are deployed here for centralized ingress.
- **Monitoring Namespace (`redhat-ods-monitoring`)**: Prometheus, alerting rules, and Tempo traces.
- **User Namespaces (Data Science Projects)**: User workloads (notebooks, InferenceServices, DSP instances, training jobs, feature stores) are deployed per-namespace with RBAC isolation.
