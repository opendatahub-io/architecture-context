## Platform Overview

Red Hat OpenShift AI (RHOAI) 3.4 is a comprehensive AI/ML platform deployed on OpenShift Container Platform that provides a unified environment for the full machine learning lifecycle — from interactive development through distributed training to production model serving. The platform is delivered as an OLM-managed operator (`rhods-operator`) that orchestrates the deployment of 57 components across model serving (KServe, vLLM, OpenVINO, MLServer), pipeline orchestration (Data Science Pipelines/Argo Workflows), distributed training (Kubeflow Training Operator, KubeRay), feature management (Feast), experiment tracking (MLflow), model governance (Model Registry, TrustyAI), and an emerging generative AI stack (llm-d, LlamaStack, Guardrails, Gateway API Inference Extension).

RHOAI 3.4 represents an architectural inflection point with a full ingress modernization from OpenShift Routes to Gateway API. The rhods-operator deploys a centralized `data-science-gateway` Gateway with Envoy-based data plane, replacing per-component OAuth proxy sidecars with a unified kube-auth-proxy authentication layer. This transition enables sophisticated traffic management including KV-cache-aware routing for LLM inference, multi-tenant API gateway capabilities (Models as a Service), and fine-grained authorization via Kuadrant AuthPolicy integration. The platform ships 122 container images built primarily on UBI 9 base images with digest pinning for disconnected/air-gapped deployment support, and supports FIPS-compliant cryptography across its Go and Rust components.

The security architecture follows a defense-in-depth model with multiple enforcement points: Gateway-level TLS termination, kube-rbac-proxy sidecars for Kubernetes RBAC enforcement, Kuadrant AuthPolicy for per-route authorization, OpenShift service-ca for internal certificate management, and NetworkPolicy-based microsegmentation. The platform supports disconnected installations via the RELATED_IMAGE pattern and OLM relatedImages, enabling air-gapped deployments with image mirroring through oc-mirror.

### Central Components

**rhods-operator** is the platform's root orchestrator with 20 outgoing dependencies and 12 reverse dependencies. It implements a hierarchical control plane: DSCI controller bootstraps namespaces and platform services, DSC controller manages which components are enabled, and per-component controllers deploy kustomize manifests from `prefetched-manifests/`. Every operator and controller in the platform is ultimately deployed and configured by rhods-operator.

**KServe** is the second most depended-upon component (9 reverse dependencies), serving as the universal model serving substrate. Both traditional InferenceService workloads and the new LLMInferenceService CRD flow through KServe controllers. Components including odh-model-controller, trustyai-service-operator, Model Registry, and the llm-d stack all integrate with KServe's CRDs.

**odh-dashboard** is the primary user-facing entry point (12 outgoing dependencies), integrating with Data Science Pipelines, MLflow, Model Registry, LlamaStack, and TrustyAI through a module federation architecture with 7 BFF (Backend-for-Frontend) sidecars.

**kube-rbac-proxy** is referenced by 6 components as the universal authentication sidecar, providing OIDC, OpenShift OAuth, and Kubernetes RBAC enforcement in a single container image that packages two proxy implementations (kube-auth-proxy and kube-rbac-proxy) selectable via `PROXY_MODE` environment variable.

**odh-model-controller** sits at the intersection of serving and networking, automating model serving infrastructure lifecycle (Routes, HTTPRoutes, AuthPolicies, ServiceMonitors, KEDA ScaledObjects) for both InferenceService and LLMInferenceService workloads.

## Deployment Architecture

### Deployment Topology

The platform follows a three-tier namespace model. The **operator tier** (`redhat-ods-operator`) contains only the rhods-operator deployment managed by OLM. The **platform tier** (`redhat-ods-applications`) hosts all platform controllers, the data-science-gateway Gateway and its Envoy data plane, the dashboard, and shared infrastructure services. The **user tier** spans dynamically created namespaces where Data Science Projects contain workbenches, pipeline runs, model serving endpoints, and training jobs. A separate monitoring namespace (`redhat-ods-monitoring`) isolates the observability stack.

The Gateway architecture centralizes ingress through a single `data-science-gateway` Gateway CR in the platform namespace. All user-facing services (dashboard, notebooks, model endpoints, pipelines UI) are exposed via HTTPRoute resources that attach to this gateway. The Envoy proxy handles TLS termination, and kube-auth-proxy sidecars or Kuadrant AuthPolicies enforce authentication at the gateway level. This replaces the RHOAI 2.x pattern of per-component OpenShift Routes with OAuth proxy sidecars.

### Resource Summary

The platform deploys 14 operator controllers (rhods-operator, KServe dual controllers, DSPO, KubeRay, Feast, Spark, Training Operator, MLflow Operator, Model Registry Operator, LlamaStack Operator, MaaS, trustyai-service-operator, odh-model-controller, WVA), 1 centralized Gateway with Envoy data plane, 3 webhook servers (KServe, Kubeflow, rhods-operator), and 122 container images shipped via the operator bundle. The platform defines 124 CRDs across 27 components, exposes 177 services (119 ClusterIP, 6 Headless), and configures 30 NetworkPolicies for microsegmentation.

#### Manifest Consumption Pipeline

Component repositories maintain their deployment manifests in standard kustomize directory structures. The `get_all_manifests.sh` script in the rhods-operator repository clones each component repository at a pinned commit and copies its manifests into the `prefetched-manifests/` directory. During operator build, these manifests are embedded into the operator binary. At runtime, per-component controllers select the appropriate overlay (ODH or RHOAI), render the kustomize output with environment-specific values (namespace names, image references, feature flags), and apply the resulting resources via server-side apply. Image references are resolved from RELATED_IMAGE environment variables injected by OLM from the ClusterServiceVersion's relatedImages section, ensuring disconnected deployment support.

### Disconnected Support

RHOAI 3.4 fully supports disconnected (air-gapped) deployments through a multi-layered image management strategy. All 122 shipped container images are declared as `RELATED_IMAGE_*` environment variables in the operator deployment, sourced from the ClusterServiceVersion's `relatedImages` section. Each image reference uses a sha256 digest rather than a mutable tag, ensuring exact reproducibility. The `oc-mirror` tool can mirror the complete image set from `registry.redhat.io` to an internal registry. At runtime, operators resolve image references from their `RELATED_IMAGE_*` environment variables rather than hardcoding registry URLs, enabling transparent registry substitution via ImageContentSourcePolicy or ImageDigestMirrorSet CRDs. The rhods-operator's component controllers inject these resolved image references into child deployments during kustomize rendering.

## Platform Architectural Analysis

RHOAI 3.4 exhibits several cross-cutting architectural patterns that reveal both deliberate design choices and emergent tensions across its 57-component ecosystem.

**Hierarchical operator orchestration as platform control plane.** The most distinctive architectural pattern is the three-level operator hierarchy: rhods-operator's DSCI controller creates the platform foundation (namespaces, networking, auth infrastructure), the DSC controller enables/disables component CRs, and per-component controllers deploy actual workloads. This creates a clean separation of concerns but introduces a deep reconciliation chain — a change to platform configuration must cascade through three controller levels before reaching a running pod. The 194 controller watches registered by rhods-operator alone reflect the complexity of maintaining coherent state across this hierarchy.

**Ingress modernization creating dual-mode complexity.** The migration from OpenShift Routes + OAuth proxy to Gateway API + kube-auth-proxy is the defining architectural transition of 3.x. However, 3.4 still carries both patterns: 26 Route-based ingresses coexist with 20+ HTTPRoute-based ingresses. Components like Data Science Pipelines and Kubeflow Notebooks have been migrated to the Gateway API pattern, but legacy compatibility paths remain. The kube-auth-proxy component itself packages two entirely separate proxy implementations (oauth2-proxy fork and kube-rbac-proxy) in a single image, reflecting the transitional state. This dual-mode ingress increases the surface area for security misconfigurations and makes network troubleshooting more complex.

**Authentication fragmentation across the stack.** Despite the unifying intent of kube-auth-proxy, the platform employs at least seven distinct authentication mechanisms. Kuadrant AuthPolicy (CEL + OPA) handles MaaS multi-tenant authorization, kube-rbac-proxy performs SubjectAccessReview for platform services, Feast implements application-level OIDC, KServe relies on Istio mTLS for serverless mode, and external model providers use API key injection via the AI Gateway. The Model Registry Operator's codebase documents three generations of auth migration (Istio → OAuth → kube-rbac-proxy), suggesting this fragmentation is a transitional state, but the timeline for full convergence is unclear. The risk is that each auth mechanism has its own failure modes, token formats, and audit characteristics.

**The LLM serving stack as a platform-within-a-platform.** The llm-d ecosystem (llm-d, llm-d-inference-scheduler, llm-d-kv-cache, llm-d-routing-sidecar, Gateway API Inference Extension, WVA, Batch Gateway, vllm-orchestrator-gateway) introduces a sophisticated inference-optimized architecture that parallels and extends the existing KServe serving path. The KServe dual-controller split (kserve-controller-manager for InferenceService, llmisvc-controller-manager for LLMInferenceService) is architecturally significant — the LLM controller depends on Gateway API, Inference Gateway Extensions, LeaderWorkerSet, WVA, and KEDA, none of which are required by the base controller. This creates an implicit "traditional ML serving" vs "LLM serving" tier split within the platform, with different dependency requirements, scaling strategies (replica-based vs KV-cache-aware), and routing algorithms (round-robin vs prefix-cache-affinity scoring).

**Template-driven resource generation as a double-edged pattern.** Multiple operators (DSPO, trustyai-service-operator, rhods-operator) generate Kubernetes resources from Go templates embedded via `//go:embed`. This enables flexible deployment topologies — the trustyai-service-operator's GORCH controller conditionally includes 1–6 containers based on feature flags — but makes the actual deployed resource specifications harder to audit statically. The tension between deployment flexibility and auditability is particularly acute for security-sensitive configurations like TLS settings and RBAC rules.

**OpenAI API as the de facto LLM interface standard.** Six components independently implement OpenAI-compatible APIs (`/v1/chat/completions`, `/v1/completions`): vLLM, Batch Gateway, MaaS, llm-d, FMS Guardrails Orchestrator, and vllm-orchestrator-gateway. This convergence on a single API standard enables composability — guardrails can wrap any model server, batch processing works with any backend — but the implementations vary in completeness (streaming support, tool calling, structured output). There is no platform-level conformance test for OpenAI API compatibility.

**Observability is consistently implemented but not unified.** Nearly every operator exposes Prometheus metrics endpoints (typically `:8080/metrics` or `:8443/metrics`), creates ServiceMonitor resources, and supports configurable metric scraping. However, tracing is inconsistently adopted (MLServer supports OpenTelemetry, most components do not), and logging formats vary across the Go (structured JSON via zap), Python (mixed), Java (Quarkus), and Rust components. The odh-dashboard's emerging Perses integration suggests a move toward unified observability dashboards.

**Container security shows strong UBI adoption with residual gaps.** 133 of 384 Dockerfiles use UBI base images, and 248 run as non-root users, reflecting strong security hygiene. However, 136 Dockerfiles still default to root execution, and 552 contain unpinned base image references (though production images use digest pinning via RELATED_IMAGE). The gap between development Dockerfiles (often unpinned, sometimes root) and production images (pinned, non-root) suggests the security posture is better than the source code audit implies, but the development-time Dockerfiles represent a supply chain risk if used in non-production contexts.
