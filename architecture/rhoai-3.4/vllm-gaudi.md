## Purpose

**Short**: vLLM hardware plugin for Intel Gaudi (HPU) accelerators, providing optimized large language model inference serving with an OpenAI-compatible API.

**Detailed**: vllm-gaudi is the Intel Gaudi hardware plugin for the vLLM inference engine. It implements the vLLM plugin architecture (`vllm.platform_plugins` and `vllm.general_plugins` entry points) to enable LLM inference on Intel Gaudi 2 and Gaudi 3 accelerators. The plugin replaces vLLM's default CUDA-based execution paths with HPU-optimized implementations for attention backends, model execution workers, distributed communication (HCCL), and quantization kernels.

The component is deployed as a container image (`quay.io/rhoai/odh-vllm-gaudi-rhel9`) running the vLLM OpenAI-compatible API server (`vllm.entrypoints.openai.api_server`). It supports a wide range of LLM architectures including text-only models and vision-language models (Gemma3, Qwen2.5-VL, Qwen3-VL, Qwen3-VL-MoE). Key capabilities include FP8/AWQ/GPTQ quantization, LoRA adapters, speculative decoding (EAGLE), disaggregated prefill via NIXL, tensor/pipeline/data/expert parallelism, and HPU-specific optimizations like bucketed compilation, unified attention, and contiguous paged attention.

Within RHOAI, this image is referenced by KServe `InferenceService` or `ServingRuntime` CRs as the serving container for Intel Gaudi-accelerated model serving workloads. It does not define CRDs or act as an operator — it is a runtime container image consumed by the platform's model serving infrastructure.

## Architectural Analysis

The vllm-gaudi plugin follows vLLM's hardware plugin architecture (RFC #11162 and RFC #19161), registering itself through Python entry points. Three entry points are defined in `setup.py`: `vllm.platform_plugins` registers the `HpuPlatform` class, `vllm.general_plugins` registers custom ops and model overrides. This design allows the Gaudi plugin to be installed alongside upstream vLLM without forking, replacing the earlier `vllm-fork` approach that reached end-of-life with Synapse AI 1.22.2.

The extension framework (`vllm_gaudi/extension/`) is the most architecturally significant subsystem. It implements a lazy-evaluated configuration system (`Config` class in `config.py`) with environment variable overrides, hardware detection, and version-gated feature flags. The `features.py` module defines dozens of runtime flags controlling bucketing strategies, attention implementations, compilation modes, and quantization behavior. This system enables the same container image to run optimally across Gaudi 2 (FP8 E4M3FNUZ, scale factor 4.0, exponent biases [3,7,11,15]) and Gaudi 3 (FP8 E4M3FN, scale factor 1.0, exponent biases [0-62]) without separate builds.

The bucketing system (`extension/bucketing/`) is critical to HPU performance. Unlike GPU inference where dynamic shapes are handled by CUDA kernels, HPU graph compilation is expensive and each unique input shape triggers a new graph compilation. The bucketing manager pre-defines exponential or linear bucket sizes for prompt lengths, decode batch sizes, and unified attention dimensions. Inputs are padded to the nearest bucket boundary, trading some wasted computation for dramatically reduced compilation overhead. The unified attention bucketing adds a fourth dimension for shared KV cache context, enabling efficient prefix caching.

Quantization support is comprehensive: FP8 (per-channel and per-block), AWQ (4-bit weights), GPTQ (2/3/4/8-bit weights), and compressed tensors (W8A8 FP8, WNA16). All quantization methods use HPU-native operators (`torch.ops.hpu.convert_from_uint4`, `torch.ops.hpu.cast_to_fp8_v2`, `torch.ops.hpu.fp8_gemm_v2`) rather than generic PyTorch implementations. The calibration subsystem provides a multi-step pipeline (device detection → dataset preparation → scale measurement → quantization) for FP8 model calibration on Gaudi hardware.

The NIXL connector (`distributed/kv_transfer/kv_connector/v1/hpu_nixl_connector.py`) patches vLLM's data pointer resolution to use Habana's physical device pointer API (`htexp._data_ptr()`), enabling RDMA-based KV cache transfer between disaggregated prefill and decode nodes. This is a key architectural enabler for large-scale serving where prefill and decode are split across separate Gaudi node pools.

A notable security consideration is the explicit removal of `openssl-fips-provider-so` in the Dockerfile. The non-hermetic build configuration (`hermetic: false` in Tekton PipelineRuns) combined with network fetches from Habana's proprietary artifact repository (`vault.habana.ai`) and PyPI during image builds creates a supply chain risk. This is architecturally constrained by the proprietary nature of the Habana software stack, which does not publish to standard hermetic-friendly repositories.
