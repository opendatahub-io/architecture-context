## Purpose

**Short**: vLLM hardware plugin that enables optimized large language model (LLM) inference on Intel Gaudi (HPU) accelerators, packaged as a container image for RHOAI serving runtimes.

**Detailed**: The vllm-gaudi component is the Intel Gaudi hardware plugin for the vLLM inference engine. It provides HPU-specific implementations of attention backends, model runners, distributed communication, quantization methods, and memory management that allow vLLM to run efficiently on Intel Gaudi 2 and Gaudi 3 accelerators. The plugin follows vLLM's hardware-pluggable architecture (RFC #11162 and #19161), registering as an out-of-tree platform extension via Python entry points.

In the RHOAI platform, vllm-gaudi is shipped as a container image (`odh-vllm-gaudi-rhel9`) that serves as a model serving runtime for KServe InferenceService CRs. The container runs vLLM's OpenAI-compatible API server (`vllm.entrypoints.openai.api_server`) with the Gaudi plugin automatically loaded, exposing an OpenAI-compatible REST API on port 8000. It supports multi-card inference via HCCL (Habana Collective Communication Library), multi-node KV cache transfer via NIXL, speculative decoding (EAGLE), LoRA adapters, and multiple quantization methods (FP8, AWQ, GPTQ).

The repository is a downstream fork of [vllm-project/vllm-gaudi](https://github.com/vllm-project/vllm-gaudi), overlaying RHOAI-specific Konflux build infrastructure (Dockerfiles, Tekton pipelines, build-args) onto the upstream plugin code. The Konflux build produces a multi-stage UBI9-based container image with Habana Synapse AI drivers (v1.23.0), PyTorch for HPU (v2.9.0), and the vLLM + vllm-gaudi plugin stack.

## Architectural Analysis

**Plugin Architecture and vLLM Integration**: The vllm-gaudi plugin uses vLLM's entry_points mechanism to register three plugin types: `vllm.platform_plugins` (registers `HpuPlatform`), and two `vllm.general_plugins` (registers custom ops and custom models). The ops plugin is prefixed `01.` and models `02.` to enforce loading order — ops must be registered before model classes that reference them. The `HpuPlatform` class extends vLLM's `Platform` base and overrides device detection, quantization method enumeration, attention backend selection, and memory management. This design allows the Gaudi-specific code to remain fully decoupled from vLLM core, enabling independent versioning — the `VLLM_STABLE_COMMIT` file pins the exact vLLM upstream commit that has been validated against this plugin version.

**Four-Way Attention Backend Selection**: The plugin implements four distinct attention backends, selected based on model architecture and runtime configuration: (1) `HPUAttentionV1` — standard paged attention for general models, (2) `HPUAttentionMLA` — Multi-head Latent Attention for models like DeepSeek that compress KV cache into a latent space, (3) `HPUUnifiedAttention` — a batched causal+cached attention path that processes prefill and decode in a single kernel call, and (4) `HPUUnifiedMLA` — combines unified batching with MLA latent compression. The unified attention backends (`VLLM_UNIFIED_ATTN=true`) are experimental and enable merged prefill, contiguous paged attention, and defragmentation. Each backend has its own metadata builder, KV cache shape, and attention implementation, making the attention layer the most complex part of the plugin at ~2200 lines for unified MLA alone.

**Bucketing for HPU Graph Compilation**: HPU devices require pre-compiled execution graphs, and dynamic input shapes trigger expensive recompilation. The bucketing system quantizes input dimensions (batch size, sequence length, query length, context length, block count) into discrete buckets, ensuring that all inputs within a bucket share the same tensor shape. Two strategies are available: exponential (default, providing logarithmic bucket spacing) and linear. A file-based strategy allows users to specify exact bucket configurations for production workloads. The bucketing system uses a cubic-root-based fallback algorithm for unknown inputs. Vision models (Gemma3, Qwen VL) have separate vision bucket managers that use greedy planning to batch image inputs of varying sizes.

**Gaudi 2 vs Gaudi 3 Hardware Differentiation**: The plugin detects hardware generation at runtime and adapts behavior accordingly. Key differences include: FP8 format (`float8_e4m3fnuz` on Gaudi2 vs `float8_e4m3fn` on Gaudi3), vector alignment size (1 on Gaudi2 vs 128 for bf16 on Gaudi3), scale adjustment behavior (different exponent bias sets), and kernel availability (softmax_fa2 and block_softmax_adjustment are Gaudi3-only). The scales module computes hardware-specific exponent biases for FP8 quantization, with Gaudi2 limited to a small set of fixed biases while Gaudi3 supports the full 0-62 range. This dual-hardware support adds complexity but allows a single container image to serve both hardware generations.

**Non-Hermetic Build and FIPS Gap**: The Konflux build for this component is explicitly non-hermetic (`hermetic: false`), with the Dockerfile cloning upstream repositories from GitHub and downloading packages from PyPI and Habana's artifactory at build time. No pip lock files, RPM lock files, or Hermeto prefetch integration are present. Additionally, the Dockerfile explicitly removes `openssl-fips-provider-so` from the UBI9 base image, disabling FIPS-validated cryptography. The HCCL and NIXL communication channels between HPU workers and nodes also run without encryption. These are significant supply chain and security observations — the component trades build reproducibility and FIPS compliance for compatibility with the Habana software stack.

**Container User and OpenShift Compatibility**: The container runs as a non-root user (UID 2000, GID 0) with group-writable permissions on key directories, following OpenShift's arbitrary UID security pattern. The `vllm` user is a member of group 0 (root group), allowing OpenShift's random UID assignment to work correctly while still providing write access to `/home/vllm`, `/usr/src`, `/workspace`, and `/var/log/habana_logs`.
