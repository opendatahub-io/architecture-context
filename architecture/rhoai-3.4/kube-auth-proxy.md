## Purpose

**Short**: A FIPS-compliant authentication and authorization reverse proxy that provides OIDC, OpenShift OAuth, and Kubernetes RBAC enforcement for RHOAI platform component pods.

**Detailed**: kube-auth-proxy is a unified authentication and authorization proxy image that combines two distinct proxy binaries — **kube-auth-proxy** (OAuth2/OIDC authentication proxy) and **kube-rbac-proxy** (Kubernetes SubjectAccessReview authorization proxy) — into a single container image with an entrypoint multiplexer. This design allows RHOAI platform components to use either authentication mode (OIDC/OAuth-based session management) or authorization mode (Kubernetes RBAC-based SubjectAccessReview) depending on the component's security requirements.

The kube-auth-proxy binary is a focused fork of oauth2-proxy, stripped down to support only OIDC and OpenShift OAuth providers, and enhanced with Envoy ext_authz compatibility, Kubernetes service account token validation via the TokenReview API, and drop-in compatibility with the legacy oauth-proxy sidecar. The kube-rbac-proxy binary is an embedded copy of the brancz/kube-rbac-proxy project that performs Kubernetes RBAC authorization on every request via SubjectAccessReview, with support for OIDC JWT validation, client certificate authentication, and delegating authentication via the Kubernetes API.

The image is deployed as a sidecar container within RHOAI component pods, sitting in front of application containers and enforcing authentication/authorization before proxying traffic to the upstream application. The `PROXY_MODE` environment variable or the entrypoint's first argument (`auth` or `rbac`) selects which binary runs. This architecture replaces the previous oauth-proxy sidecar with a FIPS-compliant, dual-mode proxy that supports both the RHOAI 3.x Gateway API ingress pattern (where kube-rbac-proxy validates service account tokens from the gateway) and traditional OAuth2 browser-based authentication flows.

## Architectural Analysis

**Dual-mode proxy in a single image.** The most distinctive architectural decision in this component is packaging two entirely separate proxy implementations — kube-auth-proxy (an oauth2-proxy fork) and kube-rbac-proxy (a brancz/kube-rbac-proxy embed) — into a single container image with a Go multiplexer entrypoint. This design trades image size for operational simplicity: platform operators only need to manage one image reference and can switch authentication strategies per-component via a single environment variable (`PROXY_MODE`). The two binaries share no code; they have separate `go.mod` files connected via a Go workspace (`go.work`). This means security patches in one proxy binary don't automatically apply to the other, and they may drift in dependency versions over time.

**Authentication chain architecture (auth mode).** The kube-auth-proxy binary implements a sophisticated middleware chain with a "first match wins" session loading pattern. The chain evaluates authentication methods in a fixed priority order: Kubernetes service account tokens (TokenReview API) → JWT bearer tokens (OIDC verification) → OpenShift OAuth bearer tokens (sha256~ format validation) → HTTP basic auth (htpasswd) → stored cookie sessions (with refresh). This ordering is significant: it means a request carrying both a K8s service account token and a cookie will always be authenticated as the service account, not the cookie session user. The session refresh mechanism includes distributed locking via Redis (with 5-second obtain timeout and 2-second lock duration) to prevent thundering-herd token refresh in scaled deployments.

**RBAC authorization model (rbac mode).** The kube-rbac-proxy binary implements a union authorizer pattern: hardcoded metrics exception → static rules from config file → Kubernetes SubjectAccessReview. Every non-exempted request results in a SubjectAccessReview API call to the Kubernetes API server, with a 5-minute allow cache TTL and 30-second deny cache TTL. The hardcoded metrics exemption for `system:serviceaccount:openshift-monitoring:prometheus-k8s` is OpenShift-specific and intended to be migrated to static configuration. The HTTP method to Kubernetes verb mapping (GET→get, POST→create, PUT→update, PATCH→patch, DELETE→delete) enables fine-grained RBAC policies that match the semantics of the proxied API.

**TLS and certificate management.** Both proxy modes handle TLS with notable differences. kube-rbac-proxy will auto-generate a self-signed certificate if none is provided (suitable for development but producing a warning), supports dynamic certificate reloading via file polling, and supports requesting (but not requiring) client certificates for mTLS. kube-auth-proxy delegates TLS to its server module with configurable MinVersion (TLS1.2 or TLS1.3), MaxVersion (TLS1.3), and cipher suites, but does not support auto-generated certificates. Both binaries configure TLS using Go's `crypto/tls`, which under FIPS mode (via `GOEXPERIMENT=strictfipsruntime`) delegates to OpenSSL for FIPS-approved algorithms. The auth-mode proxy's OpenShift client maintains a per-CA-configuration HTTP client cache to avoid repeatedly creating TLS configurations.

**FIPS compliance considerations.** The Konflux Dockerfile builds all three binaries (kube-auth-proxy, kube-rbac-proxy, entrypoint) with `CGO_ENABLED=1`, `GOEXPERIMENT=strictfipsruntime`, and `-tags strictfipsruntime`, ensuring FIPS-compliant cryptography via the Go runtime's OpenSSL integration. However, the htpasswd validator uses `crypto/sha1` (marked with `#nosec G401/G505` annotations) for legacy SHA1 password validation — while this is for local password checking rather than network cryptography, it represents a non-FIPS crypto usage that should be noted. The `InsecureSkipVerify` option is available for upstream TLS and Redis connections, which when enabled bypasses certificate validation entirely.
