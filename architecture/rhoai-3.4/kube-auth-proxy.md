## Purpose

**Short**: A FIPS-compliant authentication and authorization reverse proxy that ships two binaries — kube-auth-proxy (OAuth2/OIDC) and kube-rbac-proxy (Kubernetes RBAC/SAR) — in a single container image for securing RHOAI platform components.

**Detailed**: kube-auth-proxy is a unified container image that packages two complementary security proxies for Red Hat OpenShift AI. The first, **kube-auth-proxy**, is a streamlined fork of oauth2-proxy that supports only OIDC and OpenShift OAuth authentication, serving as a drop-in replacement for legacy oauth-proxy sidecars. It handles authentication flows, session management (via cookies or Redis), and upstream header injection. The second, **kube-rbac-proxy**, is an embedded fork of brancz/kube-rbac-proxy that enforces Kubernetes RBAC authorization via SubjectAccessReview (SAR) and TokenReview APIs.

An entrypoint binary selects which proxy to run based on a `PROXY_MODE` environment variable (defaulting to `rbac` mode). This design allows RHOAI platform operators to deploy a single image for both authentication-only sidecars (OIDC/OAuth flows via `auth` mode) and authorization-enforcement sidecars (SAR-based RBAC via `rbac` mode). Both binaries are built with FIPS-compliant flags (`GOEXPERIMENT=strictfipsruntime`, `CGO_ENABLED=1`) for enterprise deployment on OpenShift.

The component supports Envoy ext_authz integration through its `/oauth2/auth` endpoint, making it compatible with the RHOAI 3.x Gateway API ingress pattern where EnvoyFilter CRs delegate authentication decisions to kube-auth-proxy sidecars.

## Architectural Analysis

kube-auth-proxy represents a deliberate architectural decision to consolidate RHOAI's authentication and authorization proxy needs into a single container image. By embedding both oauth2-proxy (for OAuth2/OIDC authentication) and kube-rbac-proxy (for Kubernetes RBAC authorization) behind a mode-switching entrypoint, the RHOAI platform avoids shipping and managing two separate container images. The entrypoint defaults to `rbac` mode, which aligns with the most common RHOAI deployment pattern where kube-rbac-proxy sidecars secure component metrics endpoints and API servers.

The kube-auth-proxy (auth mode) component is a streamlined fork of oauth2-proxy with all non-essential providers removed — only OIDC and OpenShift OAuth remain. This reduction in provider surface area significantly decreases the attack surface compared to upstream oauth2-proxy, which supports ~20 providers. The OpenShift OAuth provider is particularly noteworthy: it auto-discovers OAuth endpoints from the Kubernetes API, supports OpenShift's `sha256~` token format, and can delete `OAuthAccessToken` resources on logout. The middleware chain is layered (pre-auth → session loading → headers → proxy), with session loading supporting five distinct token sources evaluated in priority order: K8s SA tokens, OAuth bearer tokens, JWT bearer tokens, HTTP basic auth, and stored cookie/Redis sessions.

The kube-rbac-proxy (rbac mode) component follows a different authorization model — instead of session-based authentication, it validates bearer tokens via Kubernetes TokenReview or OIDC JWT verification, then authorizes each request via SubjectAccessReview with configurable resource attributes. A notable OpenShift-specific customization is the hardcoded authorization rule allowing `system:serviceaccount:openshift-monitoring:prometheus-k8s` to access `/metrics` without a SAR check. The proxy supports dynamic attribute rewriting from query parameters or HTTP headers, enabling per-request namespace scoping. TLS certificates are hot-reloadable without restart, and HTTP/2 can be disabled to mitigate CVE-2023-44487 (rapid reset attack).

FIPS compliance is a first-class concern across both binaries. All Go binaries are compiled with `CGO_ENABLED=1` and `GOEXPERIMENT=strictfipsruntime`, ensuring cryptographic operations use OpenSSL from the UBI 9 base image rather than Go's native crypto. A dedicated CI workflow validates every PR against Red Hat's check-payload tool. At the application level, both components enforce TLS 1.2 minimum by default, support configurable cipher suites, and no non-FIPS cryptographic algorithms (MD5, RC4, DES) were detected in the codebase. The `InsecureSkipVerify` option exists but is opt-in and annotated with `nosec`.

The Go workspace configuration (`go.work`) is architecturally significant — it unifies the root module (`github.com/opendatahub-io/kube-auth-proxy/v1`) and the embedded kube-rbac-proxy module (`github.com/brancz/kube-rbac-proxy`) into a single workspace. This enables coordinated dependency management and a single `go mod download` step in Dockerfiles. The kube-rbac-proxy retains its original module path (`github.com/brancz/kube-rbac-proxy`) rather than being rewritten, which simplifies rebasing from upstream while accepting the trade-off of a slightly confusing module identity.
