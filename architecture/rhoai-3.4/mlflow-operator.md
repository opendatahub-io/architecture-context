## Purpose

**Short**: Kubernetes operator that manages MLflow tracking server deployments on OpenShift/Kubernetes, using a Helm chart-based reconciliation approach.

**Detailed**: The MLflow Operator is a controller-runtime based Go operator that watches cluster-scoped `MLflow` custom resources and reconciles them into complete MLflow tracking server deployments. It uses an embedded Helm chart (`charts/mlflow`) to render Kubernetes resources (Deployment, Service, NetworkPolicy, PVC, RBAC, ServiceMonitor) and applies them via Server-Side Apply. The operator dynamically configures TLS (via OpenShift service-ca), CA bundle aggregation (platform and custom CA certificates), CORS origins, and Gateway API HTTPRoutes for external access through the RHOAI data science gateway.

The operator also manages a second CRD — `MLflowConfig` (group `mlflow.kubeflow.org`) — which is a namespace-scoped resource allowing workspace owners to override artifact storage credentials per namespace. The operator creates an `HTTPRoute` referencing the platform's `data-science-gateway` Gateway for external ingress, and an OpenShift `ConsoleLink` for application menu integration. A db-migration init container is injected into every MLflow Deployment to handle Alembic schema migration gaps during RHOAI upgrades (3.3 → 3.4). The deployed MLflow server uses Kubernetes-native RBAC for authorization via `SelfSubjectAccessReview`, enforced by the `kubernetes-auth` app plugin.

## Architectural Analysis

The MLflow Operator follows a **Helm-chart-as-template** pattern: rather than constructing Kubernetes objects programmatically in Go, it embeds a Helm chart (`charts/mlflow`) and uses the Helm rendering engine to produce unstructured objects, which are then applied via Server-Side Apply. This design cleanly separates deployment topology (chart templates) from reconciliation logic (controller), and allows the Helm chart to also be used standalone for development environments. The operator enriches the rendered output with post-render mutations — specifically, the db-migration init container injection in `migration.go` — which is a pragmatic pattern for RHOAI-specific concerns that shouldn't pollute the general-purpose chart.

The **CRD design** is notable: `MLflow` is cluster-scoped (with a singleton validation rule enforcing `name == "mlflow"`), while `MLflowConfig` is namespace-scoped for per-workspace configuration. This split reflects the MLflow workspaces model where a single server instance serves multiple namespaces, each of which may override artifact storage credentials. The operator reads `mlflow-artifact-connection` Secrets and `MLflowConfig` resources across namespaces to enable this multi-tenancy. The Kubernetes-native authorization model (SelfSubjectAccessReview against pseudo-resources like `experiments`, `datasets`, `registeredmodels` in the `mlflow.kubeflow.org` API group) provides fine-grained RBAC without introducing a separate auth layer.

The **networking architecture** is well-structured: MLflow serves directly over HTTPS via uvicorn with service-ca provisioned TLS certificates, eliminating the need for a TLS-terminating proxy. The NetworkPolicy is restrictive by default — ingress is limited to port 8443 from any namespace (since the gateway, dashboard, and Prometheus may be in different namespaces), while egress whitelists specific port ranges for DNS, HTTPS, Kubernetes API, PostgreSQL, MySQL, and S3-compatible storage. Users can extend egress via the `networkPolicyAdditionalEgressRules` field in the CR spec, which is a good extension point that avoids requiring users to manage raw NetworkPolicy resources. HTTP/2 is explicitly disabled on the operator's metrics endpoint (`NextProtos: ["http/1.1"]`), which mitigates HTTP/2-specific attack vectors.

The **CA bundle management** uses a three-layer approach: system CAs from the UBI base image, platform CAs from the `odh-trusted-ca-bundle` ConfigMap (managed by the RHOAI platform operator), and optional user-provided CA certificates. An init container concatenates these into a combined PEM file at startup, and a sidecar watcher polls for changes every 30 seconds to regenerate the bundle — this handles the case where ConfigMaps are updated post-deployment without requiring pod restarts. The environment variables (`SSL_CERT_FILE`, `REQUESTS_CA_BUNDLE`, `PGSSLROOTCERT`, `AWS_CA_BUNDLE`, etc.) ensure all Python libraries and database drivers honor the combined bundle.

The **aggregate RBAC roles** (`mlflow-view`, `mlflow-edit`, `mlflow-integration`) are carefully designed for role-based access. The `mlflow-view` role aggregates into the standard Kubernetes `view`, `edit`, and `admin` ClusterRoles, meaning any user with namespace `view` access automatically gets read access to MLflow pseudo-resources. The `mlflow-integration` role is purpose-built for service accounts of components that integrate with MLflow (e.g., pipeline runners), granting data-plane access without delete permissions and gateway endpoint invocation without editing.
