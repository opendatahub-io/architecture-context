## Purpose

**Short**: A Kubernetes operator that deploys and manages MLflow tracking server instances on OpenShift via a cluster-scoped `MLflow` CRD.

**Detailed**: The MLflow Operator watches a cluster-scoped `MLflow` custom resource (API group `mlflow.opendatahub.io/v1`) and reconciles a complete MLflow tracking server deployment within a configured target namespace (default: `redhat-ods-applications` for RHOAI). It uses an embedded Helm chart to render deployment manifests — including Deployment, Service, ServiceAccount, ClusterRole/ClusterRoleBinding, NetworkPolicy, PVC, and ServiceMonitor — then applies them via Server-Side Apply. The operator also manages an `HTTPRoute` (Gateway API) for ingress through the RHOAI data-science gateway and an OpenShift `ConsoleLink` for dashboard integration.

The operator supports configurable backend stores (PostgreSQL, MySQL, SQLite), artifact storage (S3-compatible, local), workspace-aware multi-tenancy via Kubernetes namespaces, custom CA bundles for TLS verification, and FIPS-compliant builds. It injects a `db-migration` init container for safe Alembic schema migration between RHOAI versions. The MLflow server itself runs with Kubernetes-native authentication (`--app-name=kubernetes-auth`) using `SelfSubjectAccessReview` checks, served over HTTPS via uvicorn with OpenShift service-CA-provisioned TLS certificates.

The operator also deploys a second CRD, `MLflowConfig` (API group `mlflow.kubeflow.org/v1`, namespaced), which allows namespace owners to override artifact storage configuration per workspace by pointing to namespace-scoped secrets. The MLflow server reads these configs at runtime — the operator does not reconcile them but provisions the CRD and grants RBAC for their use.

## Architectural Analysis

The MLflow Operator follows an interesting hybrid pattern: it is a controller-runtime based Kubernetes operator that uses an embedded Helm chart as its primary templating engine rather than Go-native manifest construction. The `HelmRenderer` in `internal/controller/helm.go` loads the chart from `charts/mlflow/`, translates the `MLflow` CR spec into Helm values, renders templates, and then applies the resulting unstructured objects via Server-Side Apply. This approach provides the best of both worlds — the chart can be used standalone for non-operator deployments, while the operator adds lifecycle management, status reporting, and Gateway API integration on top.

The operator's resource ownership model is noteworthy. Since the `MLflow` CRD is cluster-scoped but manages namespace-scoped resources, it uses `SetControllerReference` for most resources. However, the shared `ClusterRole` named `mlflow` uses a multi-owner pattern via `appendOwnerReference` (non-controller refs), allowing multiple MLflow instances to co-own the same ClusterRole. This is complemented by a custom watch handler (`clusterRoleToMLflowRequests`) that enqueues reconcile requests for all MLflow instances listed in the ClusterRole's owner references.

The security architecture is defense-in-depth. The MLflow server runs with TLS (uvicorn `--ssl-keyfile`/`--ssl-certfile` from OpenShift service-CA), Kubernetes-native authentication (`--app-name=kubernetes-auth` with `SelfSubjectAccessReview`), and a hardcoded `MLFLOW_SERVER_DISABLE_SECURITY_MIDDLEWARE=false` that cannot be overridden by users (enforced by CRD CEL validation). The NetworkPolicy is restrictive with explicit egress allow-lists for DNS, HTTPS, Kubernetes API, PostgreSQL, MySQL, and S3 ports, plus extensibility via `networkPolicyAdditionalEgressRules` in the CR spec.

The CA bundle management is particularly well-engineered. Rather than simply mounting a CA ConfigMap, the operator deploys a three-stage pipeline: (1) an init container (`combine-ca-bundles`) concatenates system CAs, platform CAs (`odh-trusted-ca-bundle`), and user-provided CAs into a single PEM file, (2) a sidecar (`ca-bundle-watcher`) polls for changes every 30 seconds and regenerates the combined bundle, and (3) the MLflow container reads the combined bundle from a shared `emptyDir` volume. This ensures CA changes propagate without pod restarts while maintaining a single trust store for all Python libraries (requests, psycopg2, boto3). The operator also sets `UVICORN_SSL_CIPHERS=PROFILE=SYSTEM` on OpenShift to ensure FIPS-compatible cipher suites.

The `db-migration` init container (`internal/controller/migration.go`) is injected as a post-render mutation after Helm chart rendering. It clones the main container's image, security context, and relevant environment variables (backend store URI, SSL/CA settings) to run `mlflow db fix-migration-gap` — an idempotent operation that handles Alembic schema migration gaps between RHOAI 3.3 and 3.4. This pattern of post-render mutation keeps the Helm chart generic while allowing RHOAI-specific concerns to be layered on by the operator.
