## Purpose

**Short**: KubeRay is a Kubernetes operator that manages the lifecycle of Ray distributed computing clusters, jobs, and serving deployments.

**Detailed**: KubeRay provides three custom resources — RayCluster, RayJob, and RayService — to declaratively manage Ray workloads on Kubernetes. The operator handles pod creation and deletion for head and worker nodes, service exposure (head, serve, headless), autoscaler injection, GCS fault tolerance with Redis, and integration with batch schedulers (Volcano, YuniKorn, scheduler-plugins).

On RHOAI, the operator includes additional controllers for authentication (OAuth/OIDC via kube-rbac-proxy sidecars), mTLS certificate management (via cert-manager), network policy enforcement, and Gateway API integration (HTTPRoute with cross-namespace ReferenceGrants). The mutating webhook on OpenShift forces `enableIngress=false` and sets the `ray.io/secure-trusted-network=true` annotation, directing traffic through the platform's Gateway API infrastructure rather than individual Routes or Ingresses.

The operator runs as a single binary (`/manager`) with six controllers coordinating via Kubernetes status conditions, finalizers, and owner references. It supports multi-namespace watching, configurable reconciliation concurrency, and emits Prometheus metrics for cluster provisioning duration and job execution time.

## Architectural Analysis

The KubeRay operator follows a multi-controller architecture where six controllers coordinate through Kubernetes-native primitives rather than direct inter-controller communication. The RayClusterReconciler serves as the central coordination point — it gates pod creation on the `AuthenticationReady` status condition set by the AuthenticationController and on mTLS secret availability managed by the RayClusterMTLSController. This condition-based gating prevents race conditions where pods could start before authentication infrastructure is ready.

The RHOAI 3.x integration introduces a cross-namespace networking pattern that is architecturally significant. The AuthenticationController creates HTTPRoute resources in the platform namespace (e.g., `opendatahub`) while the actual RayCluster services live in user namespaces. This cross-namespace reference is made possible by ReferenceGrant resources, which the controller manages with reference counting — a single ReferenceGrant per namespace is shared across all auth-enabled RayClusters in that namespace, and is only deleted when the last such cluster is removed. The mutating webhook enforces this pattern on OpenShift by disabling Ingress/Route creation and enabling the secure trusted network annotation.

The operator's pod construction pipeline in `common/pod.go` is notably complex. It transforms user-provided PodTemplateSpecs through a mutation pipeline that injects environment variables (18+ Ray-specific env vars), configures health probes (raylet + GCS + serve proxy checks), mounts shared memory volumes, injects autoscaler sidecar containers, and generates Ray start commands with calculated resource parameters (CPU, memory, GPU, TPU, Neuron cores). The init container injection for GCS readiness polling uses a retry loop with graduated warning messages, ensuring workers don't start until the head node's GCS is available.

The batch scheduler integration uses a factory pattern (`SchedulerManager`) that supports Volcano, YuniKorn, and scheduler-plugins backends. Each scheduler factory implements `DoBatchSchedulingOnSubmission()` to create PodGroup-like resources and `AddMetadataToPod()` to annotate pods with scheduling metadata. The scheduler is selected at operator startup time and cannot be changed dynamically. This architecture cleanly separates scheduling concerns from core cluster management.

The GCS fault tolerance implementation demonstrates careful lifecycle management. When enabled, the operator adds a finalizer (`ray.io/gcs-ft-redis-cleanup-finalizer`) to RayClusters and creates a Redis cleanup K8s Job during cluster deletion to clean up the Redis storage namespace. The reconciler monitors the cleanup job's completion before removing the finalizer, preventing premature cluster deletion that would leave orphaned Redis data. Worker pods are configured with a 600-second GCS reconnect timeout (vs. 60s default) to survive extended head pod restarts.
