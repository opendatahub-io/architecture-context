## Purpose

**Short**: KubeRay manages the lifecycle of Ray clusters, jobs, and services on Kubernetes/OpenShift, providing distributed computing infrastructure for AI/ML workloads.

**Detailed**: The KubeRay operator is a controller-runtime based Kubernetes operator that manages three Custom Resources — RayCluster, RayJob, and RayService — to deploy and operate Ray distributed computing clusters. It handles head and worker pod lifecycle, autoscaler RBAC, batch scheduler integration (Volcano, Yunikorn), service exposure, and zero-downtime upgrades for serving workloads.

On RHOAI (Red Hat OpenShift AI), the operator is significantly extended with OpenShift-specific controllers: an **AuthenticationController** that integrates with Gateway API via HTTPRoutes, kube-rbac-proxy sidecars, and cross-namespace ReferenceGrants; a **NetworkPolicyController** that creates per-cluster ingress policies to isolate Ray head and worker pods; and an **mTLS Controller** that provisions cert-manager certificates for intra-cluster encrypted communication. The operator includes mutating and validating webhooks that enforce secure defaults on OpenShift, including mandatory `odh.ray.io/secure-trusted-network` annotation and disabled basic Route/Ingress creation in favor of Gateway API access.

The operator is deployed as a single binary (`/manager`) in the platform namespace (typically `redhat-ods-applications` or `opendatahub`) and watches RayCluster, RayJob, and RayService resources across configured namespaces.

## Architectural Analysis

The KubeRay operator on RHOAI represents a substantially customized fork of the upstream KubeRay project. While the core controllers (RayCluster, RayJob, RayService) remain largely aligned with upstream, three entirely new controllers — AuthenticationController, NetworkPolicyController, and RayClusterMTLSController — have been added to address OpenShift-specific security and networking requirements. These controllers are **annotation-gated** via `odh.ray.io/secure-trusted-network`, which the mutating webhook automatically sets to `true` on all RayClusters created on OpenShift. This means security features are enforced by default without requiring user opt-in.

The **authentication architecture** follows the RHOAI 3.x pattern: Gateway API + kube-rbac-proxy. The AuthenticationController creates HTTPRoutes in the **platform namespace** (not the user namespace) that reference the platform's `data-science-gateway` Gateway via cross-namespace backend references. A ReferenceGrant in the user's namespace authorizes this cross-namespace access. The kube-rbac-proxy sidecar injected into head pods performs SubjectAccessReview checks — the user must have `get` permission on the head-svc Service resource to access the Ray dashboard. This provides namespace-scoped RBAC without requiring per-cluster Route/Ingress creation. The controller also includes migration logic to clean up old HTTPRoutes from cluster namespaces (from a previous implementation) and enforces `enableIngress=false` on existing clusters that predate the webhook enforcement.

The **network isolation** model creates two NetworkPolicies per RayCluster: one for head pods (allowing intra-cluster traffic on all ports, same-namespace access to client/dashboard ports, operator access, monitoring access, gateway access, and 8443 from anywhere for the auth proxy) and one for worker pods (allowing only intra-cluster traffic). The head NetworkPolicy is carefully constructed to allow the Gateway data plane to reach the kube-rbac-proxy sidecar on port 8443 from the gateway namespace (`openshift-ingress` by default). The monitoring access rule is gated on the `MONITORING_NAMESPACE` environment variable set by the platform operator via DSCI configuration.

The **mTLS subsystem** uses cert-manager to provision a complete PKI per RayCluster: a self-signed bootstrap Issuer → CA Certificate (10-year, RSA 4096) → CA Issuer → head and worker Certificates (90-day, 15-day renewal). Certificates are dynamically updated with pod IPs as the cluster scales. This is independent of the platform's mTLS (Istio service mesh) and provides Ray-internal encrypted communication between head and worker nodes. The mTLS controller monitors certificate expiry and triggers requeue for renewal.

A notable design pattern is the **separation of concerns** between controllers: the RayCluster controller handles pod/service lifecycle but delegates authentication resource creation to the AuthenticationController. The RayCluster controller's `createHeadPod` method waits for the `AuthenticationReady` condition (set by the AuthenticationController) before creating the head pod, ensuring ServiceAccount and auth resources exist before the pod starts. This prevents race conditions where pods might start without authentication configured.
