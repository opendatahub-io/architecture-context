Service Mesh / Network Policy Considerations
═══════════════════════════════════════════════════════════════════════════════

Default Network Policies: None defined by operator
Recommendation: Implement namespace-level NetworkPolicies for multi-tenant deployments

Service Mesh Integration:
  - Compatible with Istio via annotations
  - OAuth Proxy handles authentication before service mesh
  - mTLS can be configured for service-to-service communication
  - Model serving payloads use HTTPS with service certificates

Trust Boundaries:
  1. External → OpenShift Router (TLS 1.3, unauthenticated)
  2. Router → OAuth Proxy (TLS 1.2+ re-encrypt, OAuth required)
  3. OAuth Proxy → TrustyAI Service (HTTP localhost, trusted)
  4. Prometheus → TrustyAI Service (HTTP ClusterIP, cluster-internal)
  5. Operator → Kubernetes API (HTTPS TLS 1.2+, SA token)
  6. Model Serving → TrustyAI Service (HTTPS TLS 1.2+, service cert)
  7. TrustyAI Service → Database (JDBC TLS optional, credentials)
