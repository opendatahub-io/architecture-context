RBAC Summary:
- ServiceAccount: notebook-{name}
  - Created per workbench instance by notebook-controller
  - Minimal permissions: read ConfigMaps/Secrets in workbench namespace
  - No cluster-level permissions required
  - Auto-mounted token used for K8s API and KFP access

- RoleBinding: notebook-{name}-view (namespace-scoped)
  - ServiceAccount: notebook-{name}
  - Role: view (read-only access to namespace resources)

Security Configuration:
- Pod Security: Runs as non-root (UID 1001)
- No privileged mode required (even for GPU workloads)
- Security Context Constraint: restricted (OpenShift)
- Capabilities: None required

Secrets (User-provided, mounted at runtime):
- git-credentials (kubernetes.io/ssh-auth)
  - Purpose: Git repository SSH access for nbgitpuller
  - Rotation: Manual by user

- aws-credentials (Opaque)
  - Keys: AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY
  - Purpose: S3 access via boto3
  - Rotation: Manual by user

- data-connection (Opaque)
  - Purpose: Object storage credentials for data science work
  - Provisioned: User or ODH Dashboard
  - Rotation: Manual

Network Policies (Cluster-level, not defined by notebooks repo):
- Allow ingress from OpenShift Router to workbench pods
- Allow egress to internet (PyPI, CRAN, Git, S3)
- Allow egress to OpenShift API server (6443/TCP)
- Allow egress to Kubeflow Pipelines (8888/TCP)
- Deny all other ingress by default

Health Checks:
- Jupyter images:
  - Liveness: HTTP GET /api (8888/TCP)
  - Readiness: HTTP GET /api (8888/TCP)

- CodeServer/RStudio images:
  - Liveness: HTTP GET /codeserver/healthz or /rstudio/events/get_events (8888/TCP)
  - Readiness: HTTP GET /codeserver/healthz or /rstudio/events/get_events (8888/TCP)
  - NGINX proxy layer handles health check translation

Authentication Flow:
1. User accesses https://notebook-{name}-{namespace}.apps.{cluster}
2. OpenShift Router terminates TLS (443 → 8443)
3. oauth-proxy sidecar validates OAuth2 token
4. If valid, oauth-proxy forwards to Jupyter container (localhost:8888)
5. Jupyter serves content without additional auth (trusts oauth-proxy)

Known Security Considerations:
1. No mTLS between oauth-proxy and workbench container (localhost trust)
2. Workbench has full internet egress by default (data exfiltration risk)
3. Users have full Python/R code execution (arbitrary command execution)
4. Weekly image rebuilds for CVE patching (users must upgrade)
5. Secrets mounted as env vars visible via /proc (use file mounts preferred)
6. GPU access via device plugin (no privileged mode required)
