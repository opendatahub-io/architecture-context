Trust Boundaries:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

1. External → Kubernetes API: TLS 1.2+, Bearer Token authentication
2. Kubernetes API → Webhook: TLS 1.2+, Client certificate verification
3. Controller → etcd: HTTP, no encryption (internal only, should be secured in production)
4. Runtime Pods → S3: TLS 1.2+ (HTTPS) or None (HTTP), Access Key/IAM authentication
5. Client → Inference Services: Application-level (delegated to service mesh or custom auth)
6. Pod-internal communication: No encryption (containers within same pod)

Security Notes:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

⚠️  etcd communication is unencrypted HTTP - should use TLS in production
⚠️  Inference endpoints (8033, 8008) have no built-in authentication - rely on service mesh
⚠️  Storage credentials stored in plaintext secrets - consider using external secret management
✓  Controller metrics protected by kube-rbac-proxy
✓  Webhook server uses TLS 1.2+ with cert-manager integration
✓  Runtime pods support optional mTLS via service mesh integration
