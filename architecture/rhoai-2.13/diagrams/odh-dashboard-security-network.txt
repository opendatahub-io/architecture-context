│  AUTHENTICATION & AUTHORIZATION                                          │
└─────────────────────────────────────────────────────────────────────────┘

Endpoint: / (all frontend routes)
  Methods: GET, POST
  Auth Mechanism: Bearer Token (OpenShift OAuth JWT)
  Enforcement Point: OAuth Proxy (port 8443)
  Policy: OpenShift SAR - user can list projects
  Flow: User → OAuth Server → OAuth Proxy (validates token) → Backend

Endpoint: /api/* (all backend APIs)
  Methods: GET, POST, PUT, DELETE
  Auth Mechanism: Bearer Token (forwarded from OAuth proxy)
  Enforcement Point: Backend (Kubernetes client with user token)
  Policy: Per-resource RBAC via user impersonation
  Flow: Frontend → OAuth Proxy → Backend → K8s API (with user token)

Endpoint: /metrics
  Methods: GET
  Auth Mechanism: None (bypass authentication)
  Enforcement Point: OAuth Proxy
  Policy: Public metrics endpoint for Prometheus scraping

Endpoint: /oauth/healthz
  Methods: GET
  Auth Mechanism: None
  Enforcement Point: OAuth Proxy
  Policy: Health check bypass for probes

┌─────────────────────────────────────────────────────────────────────────┐
│  NETWORK POLICIES & SECURITY CONSIDERATIONS                              │
└─────────────────────────────────────────────────────────────────────────┘

1. Backend is stateless - No persistent storage; all state via K8s API
2. OAuth dependency - Requires OpenShift OAuth; not vanilla K8s compatible
3. Single namespace deployment - Typically in redhat-ods-applications
4. User impersonation - Backend uses user tokens for K8s API calls (RBAC respected)
5. No service mesh integration - No Istio/mTLS for backend-to-proxy communication
6. Internal HTTP - Backend container uses HTTP (localhost only, not exposed)
7. Metrics endpoint exposed - /metrics bypasses auth for Prometheus
