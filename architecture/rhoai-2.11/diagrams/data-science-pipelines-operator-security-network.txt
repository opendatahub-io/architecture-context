════════════════════════════════════════════════════════════════════════════════════════
THREAT MODEL & SECURITY CONTROLS
════════════════════════════════════════════════════════════════════════════════════════

Trust Boundaries:
1. External → Ingress: TLS 1.3, OAuth Bearer Token validation
2. Ingress → Application: TLS 1.2+ reencryption, OAuth Proxy enforcement
3. Application → Data: Password-based auth (MariaDB), Key-based auth (S3)
4. Application → Control Plane: ServiceAccount Token, RBAC enforcement

Attack Surface:
- External: OpenShift Routes (443/TCP) - Mitigated by OAuth and RBAC
- Internal: Cluster Services (8888, 3306, 9000) - Mitigated by namespace isolation
- Secrets: Database passwords, S3 credentials - Mitigated by Kubernetes Secrets (encrypted at rest if enabled)

Security Controls:
✓ TLS encryption for all external traffic (Routes)
✓ OAuth authentication and OpenShift RBAC authorization
✓ Namespace-scoped RBAC (principle of least privilege)
✓ ServiceAccount tokens for pod-to-API communication
✓ Secrets encryption at rest (if etcd encryption enabled)
✓ Auto-rotating TLS certificates via OpenShift Service CA
✓ Optional mTLS for gRPC services (API Server, MLMD)
✗ Network Policies (not enforced by default - should be added)
✗ Secret rotation for DB/S3 credentials (manual process)
✗ Pod Security Standards enforcement (should verify PSS/PSA policies)

Compliance Considerations:
- Audit Logging: Enable Kubernetes audit logs for DSPA resource changes
- Secret Management: Consider using external secret managers (Vault, AWS Secrets Manager)
- Network Segmentation: Implement NetworkPolicies to limit lateral movement
- Encryption at Rest: Enable etcd encryption for sensitive data
- Regular Secret Rotation: Establish process for rotating DB and S3 credentials
