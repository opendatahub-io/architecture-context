═══════════════════════════════════════════════════════════════════════════
KNOWN SECURITY LIMITATIONS
═══════════════════════════════════════════════════════════════════════════

1. Ray GCS (6379/TCP) uses unencrypted Redis protocol
   ⚠️  Not suitable for multi-tenant environments without NetworkPolicy
   ⚠️  All worker pods can access GCS without authentication

2. Ray Dashboard and Client do not enforce authentication by default
   ⚠️  Authentication must be configured at Ingress/Route level
   ⚠️  Recommended: Use OAuth/OIDC via Route annotations

3. Ray Serve authentication is application-defined
   ⚠️  User applications must implement their own auth logic
   ⚠️  No built-in authentication enforcement

4. Single head node without HA by default
   ⚠️  GCS fault tolerance requires external Redis configuration
   ⚠️  Head pod failure causes cluster downtime

5. Validation webhooks not enabled by default
   ⚠️  No automatic validation of RayCluster/RayJob/RayService specs
   ⚠️  Malformed resources may cause reconciliation loops
