Trust Boundaries:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
1. EXTERNAL → INGRESS
   - Trust level: Untrusted
   - Entry point: OpenShift Route (TLS edge termination)
   - Security controls: TLS 1.2+, OAuth redirect

2. INGRESS → SERVICE MESH
   - Trust level: Semi-trusted (authenticated users)
   - Entry point: OAuth Proxy service
   - Security controls: TLS 1.2+ reencrypt, OAuth authentication

3. SERVICE MESH (Internal)
   - Trust level: Trusted (service-to-service)
   - Security controls: ServiceAccount tokens, mTLS for Ray clusters
   - Network isolation: NetworkPolicies per Ray cluster

4. OPERATOR → KUBERNETES API
   - Trust level: Highly trusted (control plane)
   - Security controls: ServiceAccount token, TLS 1.2+
   - Permissions: ClusterRole with fine-grained RBAC

5. WEBHOOK ← KUBERNETES API
   - Trust level: Highly trusted (mutual authentication)
   - Security controls: mTLS (K8s API server presents client cert)
   - Certificate: Managed by cert-controller

Multi-Tenancy Considerations:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
- Each RayCluster gets isolated NetworkPolicies (namespace-scoped)
- OAuth proxy enforces user-level authentication (OpenShift)
- Unique ServiceAccount per RayCluster for pod identity
- CA certificates are cluster-specific (not shared across RayClusters)
- AppWrappers are namespace-scoped resources

OpenShift vs Kubernetes Differences:
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
OpenShift:
- OAuth proxy sidecar injected into Ray head pods
- OpenShift Routes used for external access
- service-ca-operator provides TLS certificates for services
- OAuth authentication integrated

Vanilla Kubernetes:
- No OAuth proxy (direct mTLS access only)
- Kubernetes Ingress used for external access
- Manual TLS certificate management for services
- No integrated authentication (rely on mTLS certs)
