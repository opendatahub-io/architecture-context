═══════════════════════════════════════════════════════════════════════════════════════
OAuth Authorization Flow (OAuth Mode)
═══════════════════════════════════════════════════════════════════════════════════════

1. User accesses notebook route: https://{notebook-name}-{namespace}.apps.{cluster}
2. OpenShift Router (Reencrypt TLS) → OAuth Proxy Sidecar (8443/TCP HTTPS)
3. OAuth Proxy checks session cookie
   - If valid: proxy to localhost:8888 → Notebook Container
   - If invalid/missing: redirect to OpenShift OAuth Server
4. User redirected to OAuth login page
5. User authenticates with OpenShift credentials (LDAP/htpasswd/etc)
6. OAuth Server performs SubjectAccessReview (SAR):
   - Resource: notebooks.kubeflow.org
   - Name: {notebook-name}
   - Namespace: {user-namespace}
   - Verb: get
   - User: {authenticated-user}
7. If SAR passes: OAuth Server issues callback with authorization code
8. OAuth Proxy exchanges code for token, creates session cookie (24h expiry)
9. OAuth Proxy validates token and proxies request to localhost:8888 (HTTP)
10. Notebook Container serves Jupyter UI

Authorization enforcement: OpenShift RBAC (user must have GET permission on Notebook CR)

═══════════════════════════════════════════════════════════════════════════════════════
TLS Cipher Suites and Versions
═══════════════════════════════════════════════════════════════════════════════════════

Webhook Server (port 8443):
- TLS Version: 1.2+
- Authentication: mTLS (Kubernetes API client certificate)
- Certificate: Managed by Service CA Operator

OAuth Proxy Sidecar (port 8443):
- TLS Version: 1.2+
- Authentication: None (initial request), OpenShift OAuth (after authentication)
- Certificate: Managed by Service CA Operator
- Session: Cookie-based (32-byte random secret, 24h expiry)

OpenShift Routes:
- Edge TLS (non-OAuth): Router terminates TLS 1.2+, forwards HTTP to pod (8888/TCP)
- Reencrypt TLS (OAuth): Router terminates TLS 1.2+, re-encrypts TLS 1.2+ to pod (8443/TCP)

═══════════════════════════════════════════════════════════════════════════════════════
Security Considerations
═══════════════════════════════════════════════════════════════════════════════════════

1. Non-OAuth Mode: No authentication - notebooks are publicly accessible via routes
   - Use only for demo/development environments
   - Recommendation: Always enable OAuth in production

2. OAuth Mode: Multi-layered security
   - TLS encryption end-to-end (Reencrypt mode)
   - OpenShift OAuth authentication (integrates with enterprise IdP)
   - OpenShift RBAC authorization (namespace-scoped, least privilege)
   - Network policies restrict traffic to authorized sources
   - Session cookies with 24-hour expiry

3. Certificate Management:
   - All certificates auto-rotate every 30 days (Service CA Operator)
   - mTLS for webhook communication (Kubernetes native)
   - TLS 1.2+ enforced across all encrypted endpoints

4. Least Privilege:
   - Controller runs as non-root (UID 1001)
   - ServiceAccounts with minimal RBAC permissions
   - NetworkPolicies restrict pod-to-pod communication
   - OAuth proxy only allows authenticated users with GET permission on Notebook CR

5. Multi-Tenancy:
   - Notebooks run in user namespaces, isolated by Kubernetes RBAC
   - OAuth integration enforces namespace-scoped access control
   - Users cannot access notebooks they don't have permissions for
