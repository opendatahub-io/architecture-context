## Purpose

**Short**: A Kubernetes module operator that manages AI Gateway sub-components (currently batch-gateway) for the Red Hat OpenShift AI platform.

**Detailed**: The ai-gateway-operator is a controller-runtime based Kubernetes operator that runs as a module operator under the opendatahub-operator platform operator. It reconciles a cluster-scoped `AIGateway` custom resource to deploy and manage AI gateway sub-components. The operator follows the ODH modular architecture pattern where the platform operator (opendatahub-operator/rhods-operator) deploys module operators, which in turn manage their own sub-components independently.

Currently, the operator manages a single sub-component: the **batch-gateway operator** (llm-d-batch-gateway-operator), which provides batch inference capabilities for LLM workloads via the LLM-D project. The batch-gateway operator is deployed conditionally based on the `spec.batchGateway.managementState` field (Managed/Removed). The architecture is designed for extensibility -- a MaaS (Models-as-a-Service) sub-component is planned but not yet implemented (TODO markers in the code).

The operator uses kustomize-based manifest rendering from the opendatahub-operator framework. Manifests for sub-components are vendored into the repository at build time via `hack/scripts/get-manifests.sh`, embedded into the container image, and rendered at reconciliation time using the platform's kustomize engine. The operator reports its version, build source, platform identity, and manifest sources back to the `AIGateway` CR status.

## Architectural Analysis

The ai-gateway-operator exemplifies the ODH modular operator pattern introduced for RHOAI 3.x. Rather than embedding all component logic in the monolithic rhods-operator, each component domain gets its own module operator that is independently versioned and deployed. The platform operator creates a well-known CR (AIGateway), and the module operator reconciles it to deploy its sub-components. This separation enables independent release cadences, smaller blast radii, and clearer ownership boundaries.

The operator's reconciliation pipeline is a linear chain of actions provided by the opendatahub-operator framework: initialize (select manifests based on spec) -> upgrade check -> release tracking -> kustomize render -> deploy -> deployment status check -> condition overwrite -> status report -> garbage collection. This pipeline is declarative -- the operator does not directly create Kubernetes resources via Go code. Instead, it configures `ManifestInfo` entries that the framework's kustomize renderer processes, and the `deploy` action applies the rendered resources. This design minimizes controller code and leverages the framework's battle-tested rendering and apply logic.

A notable design choice is the `overWriteCondition` action: when no sub-module is set to Managed, the operator explicitly marks the `DeploymentsAvailable` condition as `False` with severity `Info` rather than `Error`. This prevents the platform operator from flagging the AIGateway as unhealthy when there's legitimately nothing to deploy. This is a defensive pattern addressing the fact that an operator with zero workloads would otherwise appear broken.

The manifest vendoring strategy (via `hack/scripts/get-manifests.sh`) pins sub-component manifests to specific git commits from the upstream repositories. This creates a tight, auditable supply chain: the ai-gateway-operator image ships with exact manifest snapshots that are immutable once built. The init container pattern (`copy-manifests`) copies these embedded manifests from the image to an emptyDir volume, making them available to the main operator process at `/opt/manifests`. This avoids mounting manifests from external sources at runtime.

The batch-gateway sub-component (LLMBatchGateway CRD) reveals a sophisticated batch inference architecture with three workload components (API server, processor, garbage collector), configurable storage backends (S3 and PVC-backed filesystem), database backends (PostgreSQL, Redis, Valkey), and built-in support for TLS via cert-manager, Gateway API HTTPRoutes for ingress, OpenTelemetry tracing, and Prometheus monitoring. The CRD's validation rules enforce that `tlsInsecureSkipVerify` is prohibited, showing a security-first design for production inference workloads.
