## Purpose

**Short**: Manages the lifecycle of Data Science Pipelines (Kubeflow Pipelines v2) custom resources and their associated Kubernetes infrastructure on OpenShift AI.

**Detailed**: The Data Science Pipelines Operator (DSPO) is a Kubernetes operator built with controller-runtime and kubebuilder v3 that reconciles `DataSciencePipelinesApplication` (DSPA) custom resources. When a user creates a DSPA CR in a namespace, the operator deploys a complete Kubeflow Pipelines v2 stack including the API server, persistence agent, scheduled workflow controller, Argo Workflow controller, ML Metadata (MLMD) gRPC server with Envoy proxy, and optionally MariaDB and MinIO for database and object storage.

The operator uses the manifestival library to render Go templates into Kubernetes resources, applying them declaratively. Each sub-component (API server, database, object storage, MLMD, persistence agent, scheduled workflow, workflow controller, webhook) has its own set of templates under `config/internal/` and a corresponding controller method that conditionally deploys resources based on the DSPA spec. The operator supports both managed (operator-deployed) and external database/storage backends, pod-to-pod TLS, kube-rbac-proxy sidecars for API authentication, network policies for pod isolation, and FIPS-compliant builds. It also manages Kubeflow Pipeline and PipelineVersion CRDs, including validating and mutating webhooks for PipelineVersion resources, and supports managed pipeline image injection for pre-configured pipeline definitions.

## Architectural Analysis

The Data Science Pipelines Operator is a well-structured kubebuilder v3 operator that uses the manifestival library for declarative resource management. Rather than imperatively creating Kubernetes resources in Go, DSPO renders Go templates from `config/internal/` and applies them via manifestival's `Apply` method. This design separates infrastructure definitions (templates) from reconciliation logic (controller code), making it easier to modify deployed resources without changing Go code. The templates use a rich `DSPAParams` struct populated from the DSPA CR spec, operator configuration, and runtime state (e.g., resolved Route hostnames, decoded secrets).

The operator implements a comprehensive network security model using NetworkPolicy resources. The API server NetworkPolicy allows proxy traffic (8443) from any source but restricts direct backend ports (8888, 8887) to only internal DSP components, monitoring namespaces, and whitelisted pod labels (v2 pipeline components, workbench pods). MariaDB is locked down to only accept connections from the operator, API server, and MLMD gRPC pods. The MLMD gRPC server similarly restricts access to pipeline component pods and DSP-labeled pods. This defense-in-depth approach means that even if kube-rbac-proxy is bypassed, network-level controls prevent unauthorized access to backend services.

A notable architectural feature is the kube-rbac-proxy sidecar pattern used on both the API server and MLMD Envoy deployments. These sidecars terminate external TLS on port 8443, perform SubjectAccessReview-based authorization against the `datasciencepipelinesapplications` RBAC resource, and proxy authenticated requests to the backend. The `/healthz` and `/metrics` paths are configured as `--ignore-paths` to bypass auth for Kubernetes probes and Prometheus scraping. This pattern aligns with the RHOAI 3.x ingress model, though DSPO uses OpenShift Routes (reencrypt) rather than Gateway API HTTPRoutes for external exposure — the platform operator (rhods-operator) may still place an HTTPRoute in front of these Routes.

The operator demonstrates careful memory optimization in its controller-runtime cache configuration. It strips `managedFields` from all cached objects, strips `data` from cached ConfigMaps and Secrets (reading data via direct API calls instead), and scopes informer watches using label selectors (`dsp-version` label) and field selectors (webhook configuration name). The ConfigMap and Secret caches serve as change-notification mechanisms only — actual data reads bypass the cache entirely via `client.CacheOptions.DisableFor`. This design significantly reduces the operator's memory footprint in namespaces with many ConfigMaps and Secrets.

The managed pipelines feature introduces an init-container pattern where an OCI image containing pre-built pipeline definitions is pulled, validated against an optional registry allowlist (`MANAGED_PIPELINES_ALLOWED_REGISTRIES`), and its contents are extracted into a shared emptyDir volume that the API server mounts. The operator validates managed pipeline image references by inspecting OCI manifests via the `go-containerregistry` library, checking that images come from approved registries before deployment. This provides a supply-chain security control for pipeline definitions distributed as container images.
