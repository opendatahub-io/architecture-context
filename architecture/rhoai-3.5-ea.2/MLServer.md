## Purpose

**Short**: MLServer is a Python-based machine learning inference server that implements the Open Inference Protocol (V2 Dataplane API) over both REST (HTTP) and gRPC transports, serving as the multi-model serving runtime for traditional ML models in Red Hat OpenShift AI.

**Detailed**: MLServer provides a standards-compliant inference serving platform originally developed by Seldon Technologies. It implements the KServe V2 Inference Protocol (also known as the Open Inference Protocol) which defines a standardized API for model inference, health checking, model metadata, and model repository management. The server exposes three network interfaces: an HTTP/REST API (default port 8080) built on FastAPI/Uvicorn, a gRPC API (default port 8081) built on grpcio, and a Prometheus metrics endpoint (default port 8082).

The RHOAI distribution ships MLServer with four bundled inference runtimes: scikit-learn (SKLearnModel), XGBoost (XGBoostModel), LightGBM (LightGBMModel), and ONNX (OnnxModel). These runtimes are baked into the container image along with a trusted-runtimes allowlist (`/etc/mlserver/trusted-runtimes.json`) that restricts which model implementations can be loaded at runtime, providing a defense-in-depth security mechanism against arbitrary code execution via untrusted model definitions. In production mode (when the allowlist file exists), custom environments, environment tarballs, sys.path injection, and wildcard CORS are all blocked.

MLServer supports advanced serving features including adaptive batching (configurable per-model batch size and time windows), response caching (in-memory LRU), parallel inference via multiprocessing worker pools, OpenTelemetry distributed tracing, Kafka-based async inference, CloudEvents middleware, and streaming inference via Server-Sent Events (REST) and bidirectional streaming (gRPC).

## Architectural Analysis

MLServer follows a clean separation-of-concerns architecture with protocol handlers (REST, gRPC, Kafka) decoupled from the core inference logic (DataPlane). The DataPlane acts as a facade that orchestrates model registry lookups, inference middleware chains (including CloudEvents), response caching, and metrics collection. This design allows the same inference logic to be served over multiple transport protocols without duplication.

The runtime security model is a notable architectural feature. MLServer implements a two-mode system: **development mode** (no allowlist file) permits loading any Python class as a model runtime, including dynamic imports from model directories. **Production mode** (when `/etc/mlserver/trusted-runtimes.json` exists) enforces a strict allowlist of permitted model implementation import paths, blocks custom environment tarballs and environment_path settings (which could be used for sys.path injection), and prevents wildcard CORS configurations. The allowlist is baked into the container image at build time via the Dockerfile's `TRUSTED_RUNTIMES` build arg and validated at multiple points: settings parse time (early rejection), property access time (defense-in-depth against programmatic mutation), and server startup (logging the security mode). The import path validation uses a regex pattern that requires explicit dotted paths with uppercase class names, preventing path traversal attacks. This dual-validation approach (at parse time and at access time) with an immutable allowlist file represents a defense-in-depth strategy against model-definition-based code injection.

The parallel inference architecture uses Python multiprocessing to spawn configurable worker processes, each running their own async event loop with uvloop. A dispatcher routes requests round-robin across workers using IPC queues, and handles worker lifecycle (restart on crash, model state reconstruction). This architecture enables CPU-bound ML inference to escape the GIL limitation while maintaining an async-first design for I/O operations. The metrics subsystem uses `PROMETHEUS_MULTIPROC_DIR` for cross-process metric aggregation, and includes logic to clean up stale worker metric files when workers die unexpectedly.

The Konflux build pipeline (`odh-mlserver-v3-5-ea-2-push.yaml`) demonstrates strong supply chain security: hermetic builds are enabled (`hermetic: true`), pip dependencies are prefetched from the RHEL AI Python Package Index with SHA256 hash verification via `requirements/requirements-cpu.txt`, and multi-architecture builds (x86_64 + arm64) are produced. The `requirements-cpu.txt` file is generated by `hack/generate-pinned-requirements.py`, a sophisticated tool that resolves dependency trees from the RHEL AI PyPI, downloads platform-specific wheels for x86_64 and aarch64, and computes SHA256 hashes. This ensures bit-for-bit reproducible builds across architectures. However, no `rpms.lock.yaml` is present on this branch, meaning OS-level package hermeticity relies entirely on the AIPCC base image's own package locking.

The adaptive batching system deserves mention: it implements a queue-based batching strategy where incoming inference requests are accumulated until either the max batch size or max batch time threshold is reached. The batcher then merges multiple requests into a single batch, runs inference once, and splits the response back into individual responses. This is critical for throughput optimization with ML models where batch inference is significantly more efficient than sequential requests. The batching is configured per-model via `max_batch_size` and `max_batch_time` in ModelSettings.
