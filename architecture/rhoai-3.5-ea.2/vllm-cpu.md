## Purpose

**Short**: CPU-only build of vLLM, a high-throughput inference and serving engine for large language models.

**Detailed**: vllm-cpu is the CPU-optimized variant of the vLLM inference engine, packaged as a container image for Red Hat OpenShift AI (RHOAI). It provides an OpenAI-compatible REST API server for serving large language models on CPU hardware, without requiring GPU accelerators. The server supports chat completions, text completions, embeddings, speech-to-text, and other inference tasks through a FastAPI-based HTTP server, with an optional gRPC interface.

The CPU variant specifically targets multi-architecture deployments across x86_64, ppc64le (IBM Power), and s390x (IBM Z) platforms. Architecture-specific build scripts (`build_vllm_ppc64le.sh`, `build_vllm_s390x.sh`) compile optimized native libraries (OpenBLAS, Apache Arrow, LLVM, numactl) from source for each platform. The build process uses `uv` for Python dependency management with architecture-specific package indices (IBM DevPI for ppc64le, PyTorch CPU wheels for x86_64).

Within the RHOAI platform, this component is deployed as a ServingRuntime for KServe/ModelMesh, enabling users to serve models on clusters without GPU hardware. It is consumed by the model serving infrastructure and does not manage any Kubernetes resources directly—it is a pure inference workload container.

## Architectural Analysis

The vllm-cpu repository is a downstream fork of the upstream [vllm-project/vllm](https://github.com/vllm-project/vllm) project, rebased onto the `rhoai-3.5-ea.2` branch. The fork adds Red Hat-specific packaging (Konflux Dockerfiles, Tekton pipelines, UBI-based images), multi-architecture build scripts for ppc64le and s390x, and RHOAI-specific configuration (usage stats endpoint, pre-downloaded tokenizers for disconnected environments).

**Build Architecture**: The Konflux Dockerfile (`Dockerfile.konflux.cpu`) uses a two-stage build. The builder stage installs system dependencies via `microdnf`, creates a Python virtual environment at `/opt/vllm`, and invokes architecture-specific build scripts (`build_vllm_ppc64le.sh` or `build_vllm_s390x.sh`) to compile native libraries from source (OpenBLAS, Apache Arrow, LLVM for s390x, numactl). The vLLM wheel is then built with `uv build --wheel` and installed with tensorizer support. The final stage copies the entire virtual environment from the builder, avoiding wheel reinstallation. This is a pragmatic but non-hermetic build strategy — native library versions depend on what the build scripts clone from upstream git repos at build time.

**API Surface**: The server exposes a comprehensive OpenAI-compatible API through FastAPI, including chat completions, completions, embeddings, responses (with reasoning content), speech-to-text, rendering, tokenization, and scoring endpoints. Beyond the OpenAI API, vllm-cpu exposes operational endpoints: LoRA adapter hot-loading, RLHF weight transfer, disaggregated prefill/decode, elastic expert parallelism, profile capture, sleep/wake lifecycle, and cache management. The Anthropic Messages API is also supported. An optional gRPC interface using `smg-grpc-servicer` provides an alternative protocol for inference requests.

**CPU Optimization Strategy**: The CPU build uses Intel Extension for PyTorch (IPEX) on x86_64 for optimized inference kernels, and preloads `libomp.so` (LLVM OpenMP) to avoid soft hangs with >16 threads (per upstream vllm#32651). Memory allocation is optimized via `tcmalloc` preloading (`LD_PRELOAD=/usr/lib64/libtcmalloc.so.4` in the Konflux image). The `VLLM_USE_V1=1` environment variable enables the v1 engine architecture which is the default for the CPU target.

**AIPCC Migration Gap**: This component does not currently use AIPCC base images, which is the most significant architectural gap identified. The Konflux Dockerfile builds directly on `ubi9/ubi-minimal` and sources packages from upstream PyPI, PyTorch CPU index, and IBM DevPI. Migration to the AIPCC CPU base image (`quay.io/aipcc/base-images/cpu`) would provide the RHEL AI Python Package Index with Red Hat-rebuilt wheels, align with the platform's secure build pipeline, and standardize the base image layer across all RHOAI inference components.

**Disconnected Environment Support**: The image pre-downloads tiktoken tokenizer data at build time (`tools/predownload_tiktoken_tokenizers.py`) and sets `HF_HUB_OFFLINE=0` in the Konflux image (allowing online model downloads by default) vs `HF_HUB_OFFLINE=1` in the non-Konflux `Dockerfile.cpu.ubi`. Usage statistics are disabled (`VLLM_NO_USAGE_STATS=1`) in the Konflux image, while the non-Konflux UBI Dockerfiles point stats to `https://console.redhat.com/api/rhaiis-stats`.
