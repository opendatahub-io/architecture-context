## Purpose

**Short**: Kubernetes operator that manages the lifecycle of MLflow tracking server deployments on OpenShift AI, including database migration orchestration, Gateway API routing, and garbage collection.

**Detailed**: The MLflow operator watches a cluster-scoped `MLflow` custom resource (API group `mlflow.opendatahub.io/v1`) and reconciles a complete MLflow tracking server deployment within a target namespace. Rather than using static manifests, the operator loads an embedded Helm chart (`charts/mlflow/`) at runtime, converts the CR spec into Helm values, renders the templates via the Helm SDK, and applies the resulting Kubernetes objects using Server-Side Apply. This approach allows the operator to dynamically compose deployment resources based on cluster capabilities (OpenShift vs vanilla Kubernetes, Gateway API availability, ServiceMonitor CRD presence) and user configuration (storage backends, artifact destinations, CA bundles, network policies).

The operator manages the full MLflow infrastructure stack: a Deployment running the MLflow server with TLS (via OpenShift service-ca or user-provided certs), a ClusterIP Service on port 8443, an HTTPRoute for Gateway API integration with the platform's data-science-gateway, a NetworkPolicy with configurable egress rules, optional PersistentVolumeClaim for SQLite/file-based storage, optional ServiceMonitor for Prometheus metrics scraping, an OpenShift ConsoleLink for dashboard integration, and optional CronJob-based garbage collection. The operator also orchestrates database migrations when the MLflow version changes — scaling down the deployment, running a migration Job derived from the deployment's pod spec, and scaling back up only after successful completion.

A second CRD, `MLflowConfig` (API group `mlflow.kubeflow.org/v1`, namespaced), allows namespace owners to override artifact storage credentials per workspace, supporting multi-tenant artifact isolation. The operator also defines aggregate ClusterRoles (`mlflow-view`, `mlflow-edit`, `mlflow-integration`) that plug into Kubernetes RBAC aggregation for user-facing access control to MLflow pseudo-resources used by the Kubernetes authorization plugin.

## Architectural Analysis

The MLflow operator follows a **Helm-in-operator** pattern, where the controller embeds a Helm chart and renders it programmatically rather than relying on static kustomize manifests or direct Go resource construction. This design choice provides significant flexibility: the same chart can be used standalone via `helm install` for development, while the operator adds lifecycle management (migration orchestration, status tracking, conditional resource creation) on top. The `RenderChart` function in `helm.go` converts the MLflow CR spec into a `map[string]interface{}` values tree, renders templates via the Helm SDK engine, and returns `*unstructured.Unstructured` objects that are applied via Server-Side Apply with `mlflow-operator` field ownership. This avoids the typical operator challenge of drift detection — SSA handles merge semantics.

The **database migration orchestration** is notably sophisticated. When the operator detects a version change (comparing `status.version` against the build-time-injected `SupportedMLflowVersion`), it enters a multi-phase migration flow: (1) apply all rendered objects with the Deployment scaled to zero replicas, (2) wait for existing pods to quiesce, (3) create a migration Job derived from the rendered Deployment's pod spec (same image, volumes, env vars, but with the `mlflow db upgrade` Python script injected via embedded Go asset), (4) monitor Job completion with retry logic (operator-level retries with 2-minute delays on top of Kubernetes Job backoff), and (5) only restore Deployment replicas after successful migration. The migration Job uses specific exit codes (10-15) for different failure modes, allowing the operator to distinguish terminal failures (version mismatch, unsupported backend) from retryable ones (transient database errors). A force-migrate annotation (`mlflow.opendatahub.io/force-migrate`) allows admins to manually trigger reruns.

The operator uses **conditional feature detection** via the Kubernetes discovery API at startup to determine which CRDs are available (HTTPRoute, ConsoleLink, ServiceMonitor) and configures both the controller watches and cache accordingly. This makes the operator portable across OpenShift (which has ConsoleLink and service-ca) and vanilla Kubernetes (where only core APIs may be available). The cache is tightly scoped: owned resources use label selectors (`app: mlflow`), shared ClusterRole/ClusterRoleBinding objects use `metadata.name` field selectors to stay within `resourceNames`-scoped RBAC, and a dedicated secondary cache watches the `mlflow-gc` ClusterRole/ClusterRoleBinding independently.

The **CA bundle management** uses a three-layer approach: system CAs from the UBI base image, platform CAs from the `odh-trusted-ca-bundle` ConfigMap (injected by the platform operator), and user-provided CAs via the `caBundleConfigMap` CR field. An init container combines all sources into a single PEM file, and a sidecar watches for changes and regenerates the combined bundle. The combined bundle is then referenced by multiple environment variables (`SSL_CERT_FILE`, `REQUESTS_CA_BUNDLE`, `CURL_CA_BUNDLE`, `AWS_CA_BUNDLE`, `PGSSLROOTCERT`, `MLFLOW_MYSQL_CA`) to ensure all Python libraries use it consistently.

The **network security model** is defense-in-depth: NetworkPolicy restricts both ingress (only port 8443, from any namespace — necessary because the gateway, dashboard, and Prometheus access MLflow from different namespaces) and egress (DNS, HTTPS, database ports, S3 ports). A separate migration NetworkPolicy restricts migration Job pods to only DNS and database egress. Users can fully override egress rules via `networkPolicyEgressRules` or extend them via `networkPolicyAdditionalEgressRules`. The MLflow server itself enforces Kubernetes-native authentication via `SelfSubjectAccessReview` — every API request must carry a valid Kubernetes token, and the server checks whether the caller has the appropriate RBAC permissions on MLflow pseudo-resources (`mlflow.kubeflow.org` experiments, datasets, registeredmodels, etc.).
