## Platform Overview

Red Hat OpenShift AI (RHOAI) 3.5-ea.2 is a comprehensive AI/ML platform deployed on OpenShift Container Platform that provides end-to-end lifecycle management for machine learning workloads — from interactive development through model training, serving, evaluation, and governance. The platform is orchestrated by the rhods-operator, which implements a hierarchical operator pattern: a single DataScienceCluster CR drives 16+ component controllers, each managing its own reconciliation lifecycle while inheriting platform-wide configuration (gateway domain, OIDC settings, admin groups) from parent DSCInitialization and DataScienceCluster controllers. Components are delivered as OLM-managed operators and services, with a module system enabling out-of-tree extensions without modifying the core operator.

The platform's ingress architecture is undergoing a generational transition from Istio VirtualService and OpenShift Routes (RHOAI 2.x era) to Gateway API HTTPRoutes with Kuadrant/Authorino for authentication enforcement. The rhods-operator's Gateway controller dynamically constructs a full ingress stack at reconciliation time — a Gateway CR for the Envoy data plane, EnvoyFilters for ext_authz authentication, and kube-auth-proxy for OAuth2/OIDC session management. A Lua filter in the EnvoyFilter converts cookie-based OAuth sessions into Bearer tokens before traffic reaches component backends, ensuring components see a clean token regardless of authentication method. The newer LLMInferenceService pathway uses Gateway API natively with InferencePool CRs for LLM-optimized request routing.

Security is enforced through defense-in-depth: kube-rbac-proxy sidecars perform SubjectAccessReview-based authorization at the pod level, NetworkPolicies restrict inter-component communication, and container images are built on UBI9 base images with FIPS-compliant crypto (Go `strictfipsruntime` build tags or system OpenSSL). The platform supports disconnected/air-gapped deployments via the RELATED_IMAGE pattern for OLM relatedImages and digest-pinned container references. Multi-architecture support spans amd64, arm64, ppc64le, and s390x, though some components (notably llm-d inference scheduler) have reverted multi-arch builds due to build issues.

### Central Components

**rhods-operator** is the platform orchestrator with the highest reverse dependency count (20+ components depend on it). It provisions the DataScienceCluster CR and manages the lifecycle of all platform components through a hierarchical controller pattern with DSCInitialization → DataScienceCluster → per-component controllers.

**KServe** is the second most central component (8+ reverse dependencies), serving as the model serving control plane that both the traditional InferenceService (v1beta1) and new LLMInferenceService (v1alpha2) pathways rely on. It is depended upon by odh-model-controller, Model Registry, TrustyAI, RHOAI MCP Server, and the llm-d ecosystem.

**kube-rbac-proxy** is the platform-wide authentication sidecar (5+ components), providing SubjectAccessReview-based authorization for odh-dashboard, KubeRay, Model Registry, eval-hub, MLflow, and TrustyAI.

**odh-dashboard** has the most outgoing dependencies (22), integrating with nearly every user-facing component to provide a unified web console. It connects to Data Science Pipelines, Notebooks, Model Registry, MLflow, eval-hub, TrustyAI, MaaS, and the RHOAI MCP Server.

**odh-model-controller** bridges KServe and the platform's networking layer (10 outgoing dependencies), managing OpenShift Routes for v1beta1 InferenceServices and Gateway API + Kuadrant AuthPolicies for LLMInferenceService.

## Deployment Architecture

### Deployment Topology

The platform deploys across four namespace tiers. The **operator namespace** (`redhat-ods-operator`) hosts the rhods-operator and is managed by OLM. The **platform namespace** (`redhat-ods-applications`) hosts shared services: odh-dashboard, odh-model-controller, RHOAI MCP Server, and platform-wide configuration resources. The **ingress namespace** (`opendatahub-gateway-system` or `istio-system`) hosts the platform Gateway (Envoy), kube-auth-proxy, and associated networking resources (EnvoyFilters, DestinationRules). **User namespaces** host tenant workloads: notebook servers, InferenceServices, pipeline runs, Ray clusters, and Model Registry instances. Component operators (DSPO, KubeRay, Spark, Training, KServe, etc.) are deployed in the operator namespace but reconcile resources across user namespaces via cluster-scoped watches.

The platform ships 129 container images managed via RELATED_IMAGE environment variables in the OLM ClusterServiceVersion. It comprises 26 operator/controller-based components, 138 CRDs across 31 components, 684 RBAC role entries, and 21 NetworkPolicy definitions. The platform includes 63 gRPC service endpoints and 199 internal Kubernetes Services (137 ClusterIP, plus headless, NodePort, and KServe-managed variants). 349 architectural sub-components are documented across 65 analyzed component repositories.

#### Manifest Consumption Pipeline

Component manifests flow through a multi-stage pipeline: component repositories maintain kustomize bases and overlays, which are pulled into the rhods-operator repository via `get_all_manifests.sh` scripts that clone specific branches and copy manifests. The rhods-operator's per-component controllers then render these manifests at reconciliation time, applying image overrides from RELATED_IMAGE environment variables injected by the OLM CSV. For module-based components, the module system renders operator manifests separately, deploys the module operator, and then the module operator manages its own downstream resources. This pipeline enables each component to maintain independent release cadences while the platform operator pins specific versions per release.

### Disconnected Support

The platform provides comprehensive disconnected/air-gapped deployment support through the RELATED_IMAGE pattern. All 129 container images are referenced via `RELATED_IMAGE_*` environment variables in the OLM ClusterServiceVersion, enabling OLM's `relatedImages` mechanism to catalog and mirror all required images. The `oc-mirror` tool can mirror these images to a disconnected registry. Operators resolve container image references at runtime from these environment variables rather than hardcoding registry URLs. The DSPO additionally validates managed pipeline image references against an allowlist (`MANAGED_PIPELINES_ALLOWED_REGISTRIES`) to enforce supply-chain controls in disconnected environments. The platform's `operator_features` metadata confirms disconnected support is enabled.

## Platform Architectural Analysis

RHOAI 3.5-ea.2 reveals a platform in active architectural transition across multiple dimensions, with clear evolutionary patterns emerging from the analysis of all 65 components.

**The Ingress Generation Gap**: The platform simultaneously supports three generations of ingress: (1) Istio VirtualService (legacy Knative/KServe mode), (2) OpenShift Routes with kube-rbac-proxy (RHOAI 2.x era, still used by DSPO and traditional InferenceService), and (3) Gateway API HTTPRoutes with Kuadrant/Authorino (RHOAI 3.x forward, used by LLMInferenceService, MaaS, and Model Registry). The Model Registry Operator's three-stage migration path (Istio → OAuthProxy → KubeRBACProxy) is the clearest example of this transition's complexity. Components at different maturity levels use different ingress patterns, creating operational complexity for platform administrators who must understand all three models. The TrustyAI NemoGuardrails controller sits at the intersection of all three networking APIs (OpenShift Routes, Gateway API, and Istio), highlighting the integration burden of supporting multiple generations.

**Authentication Fragmentation**: While kube-rbac-proxy provides a consistent SubjectAccessReview pattern across 7+ components, newer components have introduced divergent authentication mechanisms. The agents-operator (Kagenti) injects AuthBridge sidecars with SPIFFE/SPIRE identity, OPA policy bundles, and JWT validation — architecturally distinct from the platform's Gateway API + kube-rbac-proxy pattern. MaaS uses Kuadrant AuthPolicy with dual API-key/TokenReview paths. The rhods-operator's Lua EnvoyFilter performs cookie-to-Bearer token conversion. These represent different design philosophies (sidecar injection vs. gateway-level enforcement vs. filter-chain transformation) that may converge as the platform matures but currently require distinct operational expertise.

**The Operator Density Challenge**: With 26 operator-based components and 138 CRDs, the platform has significant control-plane overhead. The TrustyAI Service Operator addresses this by consolidating 5 controller loops (TAS, LMES, EvalHub, GORCH, NemoGuardrails) into a single binary with selective activation — a pattern that trades blast radius for resource efficiency. The rhods-operator's module system provides an extensibility mechanism for out-of-tree operators, but the sheer number of CRD watches (350 controller watches across the platform, with Deployments, Services, and ConfigMaps each watched by 20+ controllers) creates reconciliation overhead and increases the Kubernetes API server load.

**LLM-Specialized Infrastructure**: The llm-d ecosystem (llm-d, llm-d-inference-scheduler, llm-d-router, llm-d-routing-sidecar, llm-d-kv-cache, llm-d-planner, llm-d-latency-predictor, llm-d-async, llm-d-batch-gateway-operator, batch-gateway, vllm-orchestrator-gateway) represents a significant investment in LLM-optimized serving infrastructure that goes far beyond traditional ML serving. The inference scheduler's plugin framework mirrors kube-scheduler's design, with 20+ plugins for KV cache-aware routing, prefix cache scoring, session affinity, and disaggregated prefill/decode. This level of specialization creates a deep stack (KServe → LLMInferenceService → InferencePool → EPP → llm-d Router → vLLM) that requires expertise across multiple abstraction layers.

**Security Posture Asymmetry**: FIPS compliance is inconsistently applied — Go components generally use `strictfipsruntime` build tags, but exceptions exist (KServe module controller with CGO_ENABLED=0, EPP without strictfipsruntime). The llm-d inference scheduler sidecar enforces TLS 1.2+ with curated cipher suites, while the EPP defaults to TLS 1.0+ and the model server metrics scraping uses `InsecureSkipVerify: true`. The rhods-operator's DestinationRule for kube-auth-proxy also uses `insecureSkipVerify: true` as a documented trade-off. These inconsistencies are individually justified but collectively create a non-uniform security surface.

**Emerging Agentic Architecture**: The agents-operator (Kagenti) introduces a fundamentally different deployment model — sidecar-based identity, A2A protocol with JWS-signed agent cards, SPIFFE/SPIRE workload identity, OPA policy bundles, and network policy enforcement based on agent verification status. This is architecturally distinct from the rest of the platform and represents the leading edge of the platform's expansion from ML serving into autonomous agent orchestration. The 4-layer configuration precedence chain and feature gate system suggest this component is being designed for rapid iteration.

**Data Layer Consolidation**: PostgreSQL is the dominant state store across the platform (used by Model Registry, MaaS, Data Science Pipelines via MariaDB, MLflow, agents-operator via Keycloak) while ML Metadata provides a specialized gRPC-based artifact store for pipeline lineage tracking. The model-metadata-collection and models-perf-benchmark-data components introduce a container-as-data pattern where model catalogs and benchmark datasets are shipped as OCI images. This pattern enables version-controlled data delivery through the same supply chain as code, but the proliferation of init containers adds pod startup latency.

**Multi-Language Reality**: The platform spans Go (primary for operators and controllers), Python (ML runtimes, SDKs, evaluation), Rust (guardrails orchestrator, vllm-orchestrator-gateway, trainer data cache), Java (TrustyAI Explainability via Quarkus), TypeScript (dashboard frontend and BFFs), C++ (ML Metadata, OpenVINO, vLLM CPU kernels), and Protocol Buffers (gRPC service definitions). This polyglot architecture is appropriate for the diverse workload types but complicates FIPS compliance (each language ecosystem requires different mechanisms) and increases the build infrastructure complexity.
