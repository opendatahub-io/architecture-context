## Purpose

**Short**: Automated LLM vulnerability scanning and red-teaming adapter for the eval-hub evaluation platform, using NVIDIA's Garak framework.

**Detailed**: This component provides a `FrameworkAdapter` implementation that integrates the Garak red-teaming framework with the eval-hub evaluation platform on RHOAI. It enables automated security assessments of LLMs deployed on OpenShift AI, scanning for vulnerabilities such as prompt injection, jailbreaks, toxicity, bias, and other safety concerns against established benchmarks including OWASP LLM Top 10, AVID taxonomy, and CWE.

The adapter supports two execution modes: **simple mode**, where Garak runs as a subprocess inside the eval-hub Kubernetes Job pod, and **KFP mode**, where it submits a six-step Kubeflow Pipeline that runs Garak in a dedicated pod with S3-based artifact transfer. Both modes produce structured JSONL reports, AVID-format vulnerability reports, and HTML visualizations. Results are reported back to eval-hub and optionally persisted to MLflow and OCI registries.

For advanced use cases, the adapter supports **intents-based risk assessment** which uses Synthetic Data Generation (SDG) via the `sdg-hub` library to produce targeted adversarial prompts from a policy taxonomy (e.g., "Illegal Activity", "Hate Speech", "Fraud"). These prompts are then run through a multi-probe attack pipeline (baseline, SPO augmentation, multilingual translation, TAP jailbreak) with early-stop harness optimization and results visualized in a custom ART (Automated Red Teaming) HTML report.

## Architectural Analysis

The TrustyAI Garak adapter follows a layered architecture with clear separation between framework-agnostic core logic and eval-hub-specific integration. The `core/` module contains pure functions for command building, configuration resolution, subprocess management, and pipeline step logic — all usable outside the eval-hub context. The `evalhub/` module wraps these with the FrameworkAdapter interface, status callbacks, and S3/OCI artifact management. This design allows the same scan logic to be invoked from a K8s Job (simple mode), a Kubeflow Pipeline (KFP mode), or potentially other orchestrators.

The dual execution model (simple vs. KFP) reflects a practical trade-off. Simple mode is lower-latency and simpler to debug — garak runs as a subprocess in the same pod. KFP mode enables caching of expensive SDG runs, visibility in the KFP dashboard, and isolation of the garak process. The six-step KFP pipeline (validate → resolve_taxonomy → sdg_generate → prepare_prompts → garak_scan → write_kfp_outputs) is designed so the SDG step is cacheable independently of the scan, which is valuable when iterating on scan configurations with the same taxonomy. API key management is particularly careful: keys are never embedded in the garak config dict sent to KFP. Instead, `__FROM_ENV__` placeholders are written and resolved at pod startup via `_resolve_config_api_keys`, which reads from volume-mounted Kubernetes Secrets. The config.json is always redacted before upload to S3 or OCI.

The intents-based risk assessment pipeline is the most complex flow. It chains SDG → prompt normalization → intent stub generation → multi-probe scan with early-stop optimization. The multi-probe strategy (baseline IntentProbe → SPO augmentation → multilingual translation → TAP jailbreak) implements an escalating attack difficulty: if a model refuses a baseline prompt, progressively more sophisticated attack methods are tried. The EarlyStopHarness short-circuits when a stub is already jailbroken, saving compute. Results are aggregated at the stub level (not attempt level) to match the ART HTML report semantics. The adapter generates both garak's native HTML report and a custom ART HTML report (via Jinja2 templates with embedded Vega chart specs) for intents scans.

A notable security consideration: the `utils.get_http_client_with_tls()` function supports disabling TLS verification via string configuration (`"false"`, `"0"`, `"no"`, `"off"`). While TLS defaults to enabled, this flexibility exists for testing against self-signed certificate endpoints in development environments. In production, TLS verification should always remain enabled. The KFP client similarly supports `verify_ssl=False` and custom `ssl_ca_cert` paths for non-standard PKI configurations common in air-gapped OpenShift deployments.
