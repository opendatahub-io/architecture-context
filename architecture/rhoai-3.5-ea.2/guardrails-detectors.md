## Purpose

**Short**: A collection of text detection microservices that provide content analysis capabilities (PII detection, file-type validation, HuggingFace model inference, and LLM-as-judge evaluation) for the FMS Guardrails Orchestrator.

**Detailed**: Guardrails Detectors is a multi-service Python repository that provides detector algorithms for the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator), an IBM-led open source project for text generation input/output inspection. The repository produces two Konflux-built container images (built-in detector and HuggingFace runtime) and includes code for a third detector type (LLM Judge) that does not yet have a Konflux Dockerfile.

Each detector is packaged as an independent FastAPI microservice exposing the same `/api/v1/text/contents` REST API endpoint. The built-in detector runs lightweight heuristic algorithms (regex-based PII detection, file-type validation, custom Python detectors) on port 8080. The HuggingFace detector loads transformer models (AutoModelForSequenceClassification, AutoModelForTokenClassification, or GraniteForCausalLM) for ML-based content classification on port 8000, and is designed to be deployed as a KServe inference service. The LLM Judge detector delegates evaluation to an external OpenAI-compatible LLM server via the `vllm_judge` library.

The detectors share a common FastAPI base class (`DetectorBaseAPI`) that provides health endpoints, Prometheus metrics instrumentation (detection counts, request counts, errors, runtime latency), and error handling. All detectors conform to the IBM Detector API specification and integrate with the Guardrails Orchestrator as content-type detectors.

## Architectural Analysis

The guardrails-detectors repository follows a plugin-based architecture where each detector type (regex, file-type, custom, HuggingFace, LLM Judge) implements a common interface but is packaged as a separate microservice. The common framework (`detectors/common/`) provides a `DetectorBaseAPI` FastAPI subclass and `InstrumentedDetector` base that standardize health checks, error handling, and Prometheus instrumentation across all detector types. This design allows each detector to be independently scaled and deployed while maintaining API consistency.

The built-in detector uses a registry pattern (`BaseDetectorRegistry`) where each detector kind (regex, file_type, custom) registers a dictionary of named detection functions. Requests specify which detector kinds and functions to invoke via the `detector_params` field, allowing flexible composition. The custom detector subsystem is particularly notable — it dynamically loads user-provided Python code from `custom_detectors/custom_detectors.py`, performing static code analysis (AST-based) to block dangerous imports (`os`, `subprocess`, `sys`, `shutil`) and function calls (`eval`, `exec`, `open`, `compile`, `input`) before execution. This provides a sandboxed extension mechanism with security controls, though the sandbox relies on static analysis rather than runtime isolation (e.g., no seccomp or capability restrictions at the Python level). The custom detector framework also supports `@non_blocking` and `@use_instruments` decorators, allowing user-defined detectors to run asynchronously and register custom Prometheus metrics.

The HuggingFace detector supports three model architectures: `AutoModelForSequenceClassification` (binary/multi-label text classification), `AutoModelForTokenClassification` (NER-style per-token detection with span grouping), and `GraniteForCausalLM` (IBM Granite Guardian models using generative risk assessment). The model type is auto-detected from the model config's `architectures` field. Token classification is a recent addition (commit `70ba220`) that produces character-offset spans by grouping adjacent tokens with the same predicted label. The detector supports per-request parameter overrides for threshold, label thresholds, safe labels, and max sequence length, with environment variable defaults providing cluster-level configuration.

A significant architectural concern is the non-hermetic build process. Both Konflux PipelineRuns set `hermetic: false`, and no dependency lock files exist. The HuggingFace Dockerfile builds PyTorch from source for ppc64le/s390x architectures by cloning from GitHub during the build, which is both non-reproducible and a supply chain risk. The component also does not use AIPCC base images, which means it does not benefit from the RHEL AI Python Package Index or the standardized accelerator runtime libraries. Given that the HuggingFace detector uses PyTorch and can leverage GPU acceleration, migrating to AIPCC base images (particularly the CPU and CUDA variants) would improve build security, reduce Dockerfile complexity, and ensure access to validated Python package versions.
