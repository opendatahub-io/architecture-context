## Purpose

**Short**: Kubernetes operator that manages the lifecycle of the Kubeflow Trainer v2 component for distributed ML training workloads on RHOAI.

**Detailed**: The trainer-operator is a modular component operator for Red Hat OpenShift AI that deploys and manages the upstream Kubeflow Trainer v2 training infrastructure. It watches a cluster-scoped `Trainer` custom resource (singleton named `default-trainer`) and reconciles the upstream trainer controller deployment, pre-configured `ClusterTrainingRuntime` resources for PyTorch distributed training, and OpenShift `ImageStream` resources for Training Hub workbench images.

The operator implements a modular architecture pattern used by RHOAI 3.x, where the platform operator (`rhods-operator`) creates a `Trainer` CR, and this component operator handles the actual deployment. The operator fetches upstream Kubeflow Trainer manifests at build time (from the `trainer` repository), renders them through kustomize with RHOAI-specific image overrides at runtime, and applies them using server-side apply. It manages dependency checking (JobSet Operator and CRD), namespace provisioning, deployment health monitoring, drift correction via downstream resource watches, and garbage collection on removal/deletion.

The operator deploys ClusterTrainingRuntime resources across three accelerator variants (CUDA, ROCm, CPU) with multiple pinned version combinations (PyTorch 2.9/2.10, CUDA 12.8/13.0, ROCm 6.4), providing users with ready-to-use distributed training templates. On OpenShift clusters, it additionally creates ImageStream resources so training images appear in the RHOAI dashboard's workbench image selector.

## Architectural Analysis

The trainer-operator follows the RHOAI 3.x modular component operator pattern, where each platform component has its own dedicated operator that watches a singleton CR. This decouples component lifecycle management from the platform operator, allowing independent release cycles and simpler testing. The `Trainer` CR uses a standard `ManagementState` field (`Managed`/`Removed`) following the ODH platform-utilities convention, and exposes conditions (`Ready`, `ProvisioningSucceeded`, `Degraded`) for the platform operator to aggregate.

The manifest management strategy is notable: the operator bundles upstream Kubeflow Trainer manifests at Docker build time by running `hack/get_trainer_manifests.sh`, which clones the `red-hat-data-services/trainer` repo at a pinned ref (`rhoai-3.5-ea.1`). At runtime, the operator copies these templates to a writable work directory (`/opt/manifests-work`), patches `params.env` with `RELATED_IMAGE_*` environment variables for air-gapped/disconnected image overrides, renders through kustomize using `odh-platform-utilities/pkg/render/kustomize`, and applies via server-side apply with the field owner `trainer-module-controller`. This approach ensures image references can be overridden by the Operator Lifecycle Manager without code changes, supporting disconnected deployments.

The operator manages three distinct resource sets: (1) upstream trainer manifests (the core Kubeflow Trainer controller deployment, CRDs, RBAC, webhooks), (2) ClusterTrainingRuntime resources (15 variants covering CUDA 12.8/13.0, ROCm 6.4, and CPU with PyTorch 2.9/2.10), and (3) OpenShift ImageStream resources (3 accelerator variants with tags for RHOAI 3.4 and 3.5). The runtimes and imagestreams are managed separately from the upstream manifests, with ConfigMap-based kustomize replacements for image parameterization. The `filterConfigMaps` function strips ConfigMap resources from rendered output for runtimes and imagestreams, since those ConfigMaps are only used as kustomize parameter sources and shouldn't be deployed to the cluster.

Drift correction is implemented by watching downstream resources (Deployments, Services, ConfigMaps, ValidatingWebhookConfigurations) with a label selector for `platform.opendatahub.io/part-of: trainer`. When any managed resource is modified externally, the controller re-reconciles and re-applies the desired state. Garbage collection uses the `odh-platform-utilities/pkg/controller/gc` package, which cleans up all labeled resources when the Trainer CR is deleted or set to `Removed` state. The GC explicitly marks `ClusterTrainingRuntime` and `TrainingRuntime` as "unremovable" by the generic GC, instead handling their cleanup separately via `cleanupTrainerResources()` with best-effort deletion to avoid deletion deadlocks caused by finalizers on training runtimes.

The dependency checking logic is platform-aware: on OpenShift, it verifies three conditions (JobSet Operator installed via OLM, JobSetOperator CR exists, JobSet CRD established), while on vanilla Kubernetes it only checks for the JobSet CRD. When any dependency is missing, the operator sets the `Degraded` condition with a descriptive error message and requeues after 60 seconds, rather than failing hard. This allows the operator to start up even if dependencies aren't yet available.
