## Purpose

**Short**: Kubernetes operator that automates the submission, scheduling, monitoring, and lifecycle management of Apache Spark applications on OpenShift/Kubernetes clusters.

**Detailed**: The Spark Operator is a Kubeflow-originated Kubernetes operator that manages Apache Spark workloads through Custom Resource Definitions. It provides three CRDs: SparkApplication (v1beta2) for one-shot Spark job submission and lifecycle management, ScheduledSparkApplication (v1beta2) for cron-based recurring Spark jobs, and SparkConnect (v1alpha1) for provisioning persistent Spark Connect servers that enable remote interactive Spark sessions.

The operator handles the complete Spark application lifecycle including submission via spark-submit, driver/executor pod monitoring, automatic retry on failure, Spark Web UI exposure via Services and Ingress resources, Prometheus metrics integration via JMX exporter, and TTL-based garbage collection. It supports pluggable batch schedulers (Volcano, YuniKorn, kube-scheduler) for gang scheduling and resource queue management.

In the RHOAI platform, the operator is deployed in the `redhat-ods-applications` namespace by the rhods-operator. It uses a split architecture with separate controller and webhook Deployments, self-signed TLS certificates for webhook communication, and integrates with the OpenShift TLS security profile for cipher suite configuration. The single Konflux-built container image serves both controller and webhook roles, with the entry point determined by CLI subcommands.

## Architectural Analysis

The Spark Operator follows a **split-binary architecture** where a single container image serves two distinct roles: the controller (reconciliation loops) and the webhook (admission control), selected via CLI subcommands (`spark-operator controller start` vs `spark-operator webhook start`). This is deployed as two separate Kubernetes Deployments with independent ServiceAccounts, RBAC, and leader election locks. The split allows independent scaling and failure isolation -- a webhook outage does not block running applications, only new submissions.

The operator implements a **complex state machine** for SparkApplication lifecycle with 14 states (New, Submitted, Running, Succeeding, Failing, Completed, Failed, FailedSubmission, PendingRerun, InvalidatingState, Suspending, Suspended, Resuming, Unknown). State transitions are driven by driver pod phase changes. The controller uses spark-submit for job submission rather than direct pod creation -- this preserves Spark's native scheduling and configuration logic but couples the operator to having the Spark runtime available in the operator image. The Konflux build installs PySpark 4.0.1 into the operator image specifically for this purpose.

The **webhook system is architecturally significant** as the primary configuration injection mechanism. Rather than the controller modifying pod specs, the SparkPodDefaulter webhook intercepts pod creation and applies 26 distinct mutation options (env vars, volumes, configmaps, security context, scheduling, etc.) from the SparkApplication spec onto driver/executor pods. This webhook-driven design means pods are created by spark-submit with minimal config, then enriched by the webhook. The PartialRestart feature gate (alpha) optimizes this by skipping reconciliation when only webhook-patched executor fields change.

A **crypto/md5 import** exists in `pkg/util/sparkapplication.go` for generating truncated DNS-compliant resource names (when `{app-name}-{suffix}` exceeds 63 characters). This is purely a hashing function for name uniqueness, not a security function, so it does not constitute a FIPS compliance risk. The webhook TLS configuration is security-conscious: it reads the OpenShift APIServer's TLS security profile and applies the cluster-configured cipher suites and minimum TLS version, falling back to TLS 1.2 with a hardened cipher list on non-OpenShift clusters.

The **NetworkPolicy** in the RHOAI overlay (`config/overlays/rhoai/networkpolicy.yaml`) is notable for its SparkConnect awareness -- it explicitly allows ingress on port 15002 (SparkConnect gRPC) from pods in namespaces labeled `opendatahub.io/dashboard: "true"` with workbench pods labeled `opendatahub.io/workbenches: "true"`. This enables the pattern of data scientists connecting to persistent Spark Connect servers from their workbench notebooks.

The operator uses the **AIPCC CPU base image** (`quay.io/aipcc/base-images/cpu`) pinned by SHA256 digest, inheriting the RHEL AI Python Package Index configuration. PySpark 4.0.1 is installed on top via pip. The operator binary itself is built with FIPS-compliant Go flags (`CGO_ENABLED=1`, `GOEXPERIMENT=strictfipsruntime`, `-tags strictfipsruntime`) ensuring check-payload compliance. The RPM layer is locked via `rpms.lock.yaml` and Go dependencies via `go.sum`, providing build hermeticity at both layers.
