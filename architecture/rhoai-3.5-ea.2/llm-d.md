## Purpose

**Short**: llm-d is a Kubernetes-native distributed inference serving stack that provides container image builds, deployment guides, and kustomize recipes for achieving state-of-the-art LLM inference performance across multiple hardware accelerators.

**Detailed**: llm-d is a CNCF sandbox project that serves as the umbrella repository for the llm-d distributed inference stack. Unlike a typical operator or service, this repository does not contain application source code. Instead, it provides three key artifacts: (1) Dockerfiles that build optimized vLLM-based inference serving images for NVIDIA CUDA, AMD ROCm, Intel XPU, and CPU platforms, compiling critical high-performance libraries (UCX, UCCL, NVSHMEM, DeepEP, DeepGEMM, FlashInfer, LMCache, nixl/RIXL) from source; (2) a comprehensive set of "well-lit path" deployment guides with kustomize manifests and Helm chart values that compose the full inference stack (model servers, intelligent router, gateway, monitoring, autoscaling); and (3) helper utilities for benchmarking, smoke testing, and interactive debugging on Kubernetes clusters.

The llm-d stack accelerates distributed inference by integrating vLLM/SGLang model servers with the llm-d Router (an Envoy proxy + Endpoint Picker that performs prefix-cache-aware, load-aware, and predicted-latency-based request routing), Kubernetes Gateway API for traffic ingress, and advanced features like prefill/decode disaggregation, tiered KV-cache offloading, wide expert-parallelism, and workload-variant autoscaling. This repository is the central integration point where all these components are composed into tested, reproducible deployment topologies.

## Architectural Analysis

The llm-d repository represents a unique architectural pattern — it is not a service, operator, or library, but rather an **integration and composition layer** for a distributed inference stack. The primary value of this repository lies in three areas: (1) building highly optimized container images that package vLLM with a curated set of HPC communication libraries, (2) providing composable kustomize recipes that encode tested deployment topologies, and (3) serving as the canonical reference for how the llm-d stack's independent components (Router, model servers, Gateway API, observability) fit together.

The container image build system is notably complex. The CUDA Dockerfile (`docker/Dockerfile.cuda`) is over 630 lines and compiles nine libraries from source (gdrcopy, UCX, UCCL, NVSHMEM, nixl, LMCache, InfiniStore, DeepEP, DeepGEMM, FlashInfer) in a multi-stage build. This is necessary because these libraries require CUDA-specific compilation with precise version alignment, and prebuilt packages are not available for all configurations. The build supports three vLLM installation modes — prebuilt wheel (production), precompiled binaries with editable source (development), and full source compilation — controlled by `VLLM_PREBUILT` and `VLLM_USE_PRECOMPILED` build args. A separate GB200 build target (`runtime-gb200`) swaps the DeepEP wheel for a GB200-specific variant.

The kustomize recipe system uses a **layered composition pattern** where base deployments (single-host decode-only, or prefill/decode disaggregated) are combined with image components (GPU, CPU, TPU, ROCm, XPU, SGLang), feature patches (monitoring, sidecar injection, platform-specific tunables), and guide-specific overlays. This enables rapid composition of complex topologies while maintaining DRY principles. Each guide directory (e.g., `guides/optimized-baseline/`, `guides/pd-disaggregation/`) provides a complete deployment recipe by referencing these shared bases and components. The Router is deployed separately via Helm charts (standalone or gateway mode) with values layered similarly (`base.values.yaml` + feature values + guide values).

A significant architectural observation is the **separation of data plane from control plane**: the actual routing intelligence lives in the llm-d-router repository (EPP), while this repository only provides the Envoy configuration and Helm values that wire it together. The `ext_proc` integration pattern — where Envoy forwards all request/response headers and bodies to the EPP via gRPC for routing decisions — is central to the architecture and enables sophisticated routing strategies (prefix-cache-aware, load-aware, predicted-latency) without modifying Envoy itself.

For RHOAI productization, several gaps are evident: no Konflux Dockerfiles exist (the build system uses upstream Dockerfiles without Hermeto/cachi2 prefetch), no AIPCC base images are used (CUDA images use NVIDIA's official CUDA UBI images directly), the CPU image uses Ubuntu 22.04 (not UBI), and there are no lock files for hermetic builds. The `main` branch analyzed here represents the upstream CNCF project; the RHOAI-specific productization (FIPS hardening, Konflux integration, AIPCC migration) would need to be done on a downstream release branch.
