## Purpose

**Short**: Feast is a feature store for machine learning that provides a Kubernetes operator for deploying and managing feature store services (online/offline stores, registry, and UI) on OpenShift AI.

**Detailed**: Feast (Feature Store) is an open-source platform for managing and serving machine learning features. This repository contains the upstream Feast project plus Red Hat downstream adaptations for deployment on Red Hat OpenShift AI (RHOAI). The component consists of two primary shippable artifacts: the Feast operator (a Go-based Kubernetes operator built with kubebuilder) and the feature server (a Python-based multi-protocol service).

The Feast operator manages the lifecycle of FeatureStore custom resources, deploying and configuring Feast services including online stores (for low-latency feature serving), offline stores (for batch feature retrieval), a metadata registry (tracking feature definitions and permissions), and an optional web UI. The operator handles TLS provisioning, OIDC/Kubernetes RBAC authorization, horizontal pod autoscaling, CronJob-based materialization, Prometheus monitoring, and integration with OpenDataHub Notebooks.

The feature server is the Python runtime that executes Feast operations. It exposes REST APIs via FastAPI and gRPC services for registry operations, supporting multiple online/offline store backends (Redis, PostgreSQL, DynamoDB, Snowflake, BigQuery, DuckDB, and many more). The Go feature server provides an alternative high-performance serving layer for online feature retrieval with HTTP and gRPC endpoints.

## Architectural Analysis

The Feast repository is a large multi-language monorepo (Go operator, Python SDK/server, Go feature server, TypeScript UI) with two Konflux-built container images that ship to RHOAI. The operator and feature server have a clear separation of concerns: the operator handles Kubernetes lifecycle management while the feature server handles ML feature serving. The operator deploys the feature server as containers within a single Deployment, using different CLI commands (`feast serve`, `feast serve_offline`, `feast serve_registry`, `feast ui`) to run each service type in its own container.

A notable design decision is the "single Deployment, multi-container" pattern. Rather than creating separate Deployments for each Feast service (online, offline, registry, UI), the operator creates a single Deployment with multiple containers. This simplifies networking (services in the same pod can communicate via localhost) and reduces resource overhead, but means all services scale together. The operator mitigates this by requiring database-backed stores when scaling beyond one replica, since file-based stores cannot be shared across pods. HPA support is built in with configurable metrics, and PodDisruptionBudgets are automatically created when replicas exceed one.

The TLS architecture leverages OpenShift's serving certificate controller for automatic cert provisioning. When running on OpenShift, the operator annotates Services with `service.beta.openshift.io/serving-cert-secret-name` to trigger automatic TLS certificate creation. It also creates custom CA bundle ConfigMaps annotated with `config.openshift.io/inject-trusted-cabundle: "true"` for trust chain management. For non-OpenShift deployments, users can provide their own TLS secrets. The Go feature server explicitly sets `MinVersion: tls.VersionTLS12` with modern curve preferences (P256 and X25519MLKEM768), demonstrating post-quantum readiness.

The inter-service authentication mechanism is noteworthy. The operator generates a random 32-byte token stored in a ConfigMap and passed to all containers via the `INTRA_COMMUNICATION_BASE64` environment variable. This token is converted to a JWT (HMAC-SHA256 signed with an empty key, subject `:::{token}`) for service-to-service calls, allowing the operator to query the registry API without requiring per-user OIDC credentials. The use of `InsecureSkipVerify: true` for these internal calls is a pragmatic choice for cluster-internal traffic but worth noting for security audits.

The Notebook integration controller (`NotebookConfigMapReconciler`) demonstrates deep platform integration with OpenDataHub. It watches for Notebooks labeled with `opendatahub.io/feast-integration: "true"` and automatically injects Feast client configuration, enabling data scientists to use Feast from Jupyter notebooks without manual configuration. The controller also maintains a cluster-wide namespace registry (`feast-configs-registry` ConfigMap) for service discovery across namespaces.

The FIPS story is split: the Go operator binary is properly compiled with `GOEXPERIMENT=strictfipsruntime` and `CGO_ENABLED=1` for check-payload compliance, while the Python feature server inherits FIPS compliance from the UBI9 base image's system OpenSSL. The Python `cryptography` library delegates to OpenSSL, so it should respect the system FIPS mode. However, the feature server image uses `registry.access.redhat.com/ubi9/python-312-minimal:1` rather than an AIPCC base image, which means Python packages are installed from PyPI rather than the RHEL AI Python Package Index. This is a gap that should be evaluated for migration to AIPCC base images.
