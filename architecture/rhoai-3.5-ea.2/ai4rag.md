## Purpose

**Short**: ai4rag is an optimization engine that automatically discovers optimal hyperparameter configurations for Retrieval-Augmented Generation (RAG) pipelines.

**Detailed**: ai4rag (AI for RAG) is a Python library that automates the process of finding optimal hyperparameters for RAG pipelines. It accepts RAG templates, a search space definition of tunable parameters (chunking strategy, chunk size, overlap, embedding model, foundation model, retrieval method, number of chunks, search mode, and ranker configuration), and a benchmark dataset, then uses hyperparameter optimization (HPO) algorithms to evaluate different RAG pattern configurations and identify the best-performing one.

The library is provider-agnostic by design, with abstract base classes for foundation models, embedding models, vector stores, chunkers, and evaluators. Out of the box, it integrates with OGX (formerly LLama Stack) for foundation models, embeddings, and vector storage (Milvus, Qdrant), and with ChromaDB for in-memory vector storage. Document chunking uses either LangChain's `RecursiveCharacterTextSplitter` (token-based via tiktoken) or Docling's `HybridChunker` (structure-aware chunking). Evaluation is performed using the Unitxt framework with metrics for answer correctness, faithfulness, and context correctness.

The optimization engine supports two strategies: a GAM (Generalized Additive Model) optimizer that uses a statistical model to predict promising parameter combinations after an initial random exploration phase, and a simpler random search optimizer. The library also includes a Models Pre-Selector (MPS) stage that narrows down the set of foundation and embedding models before the main HPO loop, reducing the search space when many models are available. ai4rag is designed to be consumed as a library by higher-level services (e.g., Kubeflow Pipeline components via the `KFPEventHandler`) and does not expose any network services itself.

## Architectural Analysis

ai4rag follows a clean layered architecture with well-defined abstractions at each boundary. The core experiment orchestrator (`AI4RAGExperiment`) coordinates the full optimization loop: optional model pre-selection, search space construction, optimizer initialization, and iterative RAG pattern evaluation. Each evaluation involves five stages -- chunking, embedding, indexing, retrieval, generation -- followed by metric evaluation via Unitxt. The optimizer treats each evaluation as a black-box objective function, receiving a score (float) and using it to guide the next parameter combination.

The HPO layer implements the Strategy pattern through `BaseOptimizer`, with two concrete strategies: `GAMOptimizer` (the default) uses a Generalized Additive Model to learn the score surface from evaluated points and predict which unexplored combinations are most promising, while `RandomOptimizer` provides a simpler baseline. The GAM optimizer starts with a configurable number of random evaluations (`n_random_nodes`, default 4), then iteratively fits a `LinearGAM` model to predict scores for remaining combinations, selecting the highest-predicted candidates for actual evaluation. This Bayesian-optimization-like approach is efficient for the typically small search spaces in RAG tuning (hundreds to low thousands of combinations). The optimizer supports warm-starting via `known_observations`, allowing experiments to resume from prior results.

The search space design uses constraint rules to prune invalid parameter combinations before optimization begins. Seven built-in rules enforce consistency: chunk size must exceed 2x chunk overlap, hybrid chunking requires zero overlap (DoclingChunker limitation), recursive chunking requires non-zero overlap, window retrieval requires non-zero window size, chunk size must fit within embedding model context length (with 10% safety margin), and hybrid search parameters (ranker strategy, ranker_k, ranker_alpha) must be internally consistent. This rule-based pruning prevents wasted evaluations on invalid configurations. The search space also supports automatic model discovery from OGX, where `prepare_search_space_with_ogx()` queries the OGX server for registered models and validates their availability before including them in the search space.

The Models Pre-Selector (MPS) is a noteworthy optimization: when the search space contains more than 3 foundation models or 2 embedding models, MPS performs a lightweight evaluation of all model pairs using a small benchmark sample (default 5 questions) and an in-memory ChromaDB store with fixed chunking/retrieval settings. It ranks models by mean score and selects the top performers for the full HPO loop. This dramatically reduces the search space dimensionality when many models are available on the OGX server.

The event handler system (`BaseEventHandler` / `LocalEventHandler` / `KFPEventHandler`) provides a pluggable integration point for consuming services. `KFPEventHandler` aggregates all status changes and pattern results in memory for Kubeflow Pipeline components to read after experiment completion. `LocalEventHandler` writes pattern results and evaluation data to disk as JSON files. This design cleanly separates the optimization engine from its deployment context, allowing ai4rag to be embedded in CLI tools, Jupyter notebooks, or pipeline components without code changes.

A notable design choice is the collection reuse optimization: when the same indexing parameters (chunking method, chunk size, chunk overlap, embedding model, distance metric) are used across different RAG pattern evaluations, the experiment reuses the existing vector store collection rather than re-indexing documents. This avoids redundant embedding API calls and significantly reduces experiment runtime when only retrieval/generation parameters differ between evaluations. The deduplication logic also applies at the chunk level within a single indexing operation, using content hashing (SHA-256 for ChromaDB, text hash for OGX) to skip duplicate chunks.
