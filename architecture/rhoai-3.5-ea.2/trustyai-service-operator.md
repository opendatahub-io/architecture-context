## Purpose

**Short**: Multi-controller Kubernetes operator that manages AI trustworthiness, model evaluation, and guardrails services for the RHOAI platform.

**Detailed**: The TrustyAI Service Operator is a controller-runtime based operator that manages five distinct service controllers within a single binary, selectively enabled via the `--enable-services` flag. The controllers are: (1) **TAS** — deploys TrustyAI fairness and explainability services that integrate with KServe InferenceServices and ModelMesh to provide bias detection and model monitoring; (2) **LMES** (LM Evaluation Service) — orchestrates language model evaluation jobs using the lm-evaluation-harness framework, creating ephemeral Pods with an init-container driver pattern for evaluation execution; (3) **EvalHub** — manages a centralized evaluation hub that aggregates multiple evaluation providers (garak, lm-evaluation-harness, guidellm, lighteval, ibm-clear) with MCP (Model Context Protocol) server support and multi-tenant namespace isolation; (4) **GORCH** (Guardrails Orchestrator) — deploys the FMS Guardrails Orchestrator (a Rust service) with auto-configuration from KServe InferenceServices, optional built-in detectors, and a sidecar gateway; (5) **NemoGuardrails** — deploys NVIDIA NeMo Guardrails servers with CA bundle management, EnvoyFilter integration, and MCP Gateway support.

The operator is deployed by the platform operator (rhods-operator/opendatahub-operator) as the `trustyai` component. It reads configuration from a ConfigMap (`trustyai-service-operator-config`) for image references and operational settings. Each controller manages its own set of Kubernetes resources (Deployments, Services, Routes, ServiceMonitors) and uses kube-rbac-proxy sidecars for authentication on service endpoints.

## Architectural Analysis

The TrustyAI Service Operator is one of the more architecturally complex operators in the RHOAI platform due to its multi-controller design. Rather than deploying a separate operator per service, it consolidates five distinct controller loops (TAS, LMES, EvalHub, GORCH, NemoGuardrails) into a single binary, selectively activated via the `--enable-services` flag. This design reduces resource overhead at the cost of a larger blast radius — a bug in one controller could affect the entire operator process. The operator mitigates this by disabling the informer cache for high-volume core resources (ConfigMaps, Secrets, Pods, Services) to prevent OOM from cluster-wide cache flooding, a pattern visible in `cmd/main.go` lines 122-130.

The LMES controller employs a unique "driver injection" pattern: it creates evaluation Pods with an init container (`ta-lmes-driver`) that copies itself to a shared emptyDir volume, then the main container (`ta-lmes-job`) runs this binary to coordinate evaluation execution. The operator polls the driver via `pods/exec` (SPDY protocol) to retrieve status updates rather than using a sidecar or webhook callback pattern. This design avoids the need for a long-running sidecar but requires `pods/exec` RBAC permissions and creates polling overhead via the `syncedMap4Reconciler` mechanism.

The EvalHub controller introduces multi-tenant namespace isolation via a label-based watch on Namespace resources (`evalhub.trustyai.opendatahub.io/tenant`). When a namespace gains this label, the controller creates ServiceAccounts, Roles, and RoleBindings in that namespace to allow evaluation jobs to run there. This is a cross-namespace resource management pattern that cannot use standard ownerReferences for garbage collection, so the controller implements explicit cleanup via label selectors during deletion. The EvalHub also supports an optional MCP (Model Context Protocol) server deployment, adding a separate Deployment, Service, and Route per EvalHub instance when `spec.mcp.enabled=true`.

The GORCH controller implements a sophisticated auto-configuration mechanism: it watches KServe InferenceServices, detects generator and detector services via label matching (`trustyai/guardrails: true` or user-specified labels), discovers their serving protocols (gRPC vs REST) from ServingRuntimes, and generates orchestrator configuration ConfigMaps automatically. This eliminates manual configuration but creates a tight coupling with the KServe CRD schema. The deployment template supports up to three containers: the orchestrator itself, a sidecar gateway, and built-in detectors — each optionally fronted by their own kube-rbac-proxy sidecar when OAuth is required.

The NemoGuardrails controller is the newest addition and shows the most modern patterns: it integrates with both the Kuadrant MCP Gateway API (`mcp.kuadrant.io/v1alpha1 MCPGatewayExtension`) and Istio EnvoyFilters for traffic management, and it reads Gateway API resources (`gateway.networking.k8s.io/v1 Gateway`). This positions it at the intersection of three networking APIs (OpenShift Routes, Gateway API, and Istio). Its CA bundle management (`controllers/nemo_guardrails/ca.go`) aggregates certificates from multiple sources (ODH trusted CA, OpenShift serving CA, user-provided CA) into a single mounted bundle, a pattern shared with the LMES controller.
