## Purpose

**Short**: An OpenAI-compatible HTTP gateway that provides per-route chat completion endpoints with configurable detector-based content filtering, routing requests to the FMS Guardrails Orchestrator.

**Detailed**: The vLLM Orchestrator Gateway is a Rust-based HTTP reverse proxy that sits in front of the FMS Guardrails Orchestrator. It enables controlled interaction with the orchestrator by exposing multiple OpenAI-compatible `/v1/chat/completions` endpoints, each with a distinct set of detector configurations. This allows operators to define named routes (e.g., "pii", "passthrough") that enforce specific content filtering policies, providing drop-in replacement of an unguardrailed chat completion model endpoint with a guardrailed one.

The gateway reads a YAML configuration file at startup that defines the upstream orchestrator address, a set of detector definitions (with input/output directives and detector-specific parameters), and a set of named routes that map detector combinations to HTTP paths. For each route, the gateway dynamically registers a POST endpoint at `/{route_name}/v1/chat/completions`. When a request arrives, the gateway injects the route's detector specifications into the payload and forwards it to the orchestrator's `/api/v2/chat/completions-detection` endpoint. If the orchestrator detects content violations, the gateway can replace the response with a configurable fallback message. The gateway supports both non-streaming and streaming (SSE) chat completions.

The gateway communicates with the orchestrator using optional mTLS, loading TLS certificates from well-known Kubernetes-provisioned paths (`/etc/tls/private/` for client certs and `/etc/tls/ca/` for CA certificates). It is built as a single stateless binary using the Axum web framework and Tokio async runtime.

## Architectural Analysis

The vllm-orchestrator-gateway is a thin, stateless routing layer built with a focused purpose: to present multiple OpenAI-compatible endpoints, each enforcing a different detector policy, while keeping the actual detection and inference logic in the FMS Guardrails Orchestrator. The architecture is deliberately simple — three source files totaling ~29KB — with no database, no state persistence, and no CRDs. The gateway reads its configuration once at startup from a YAML file and registers routes dynamically based on the `routes` array, making it easy to add or remove guardrailed endpoints by editing a ConfigMap.

The mTLS implementation in `build_orchestrator_client()` (main.rs:351-406) is well-structured: it probes for TLS certificate files at Kubernetes-standard paths and gracefully degrades to plaintext HTTP if certificates are absent. The PKCS#12 conversion dance (loading PEM via `openssl` crate, converting to PKCS#12 in-memory, then loading as a `native-tls` Identity) is required because the `reqwest` client's Identity API expects PKCS#12 format. One notable risk: when the orchestrator host is `localhost`, the gateway calls `danger_accept_invalid_hostnames(true)`, which disables hostname verification. This is clearly a development convenience and should not be reachable in production deployments where the orchestrator is addressed by its Kubernetes Service DNS name.

The header forwarding logic (main.rs:421-432, 493-502) selectively passes `Authorization` and `x-forwarded-*` headers to the orchestrator, which means the gateway relies on external infrastructure (kube-rbac-proxy sidecar, ingress controller, or network policy) for its own access control. The gateway itself performs no authentication — it is an open HTTP listener on port 8090. This is consistent with the RHOAI 3.x pattern where kube-rbac-proxy sidecars handle auth enforcement.

The config validation logic (config.rs:64-109) validates that all detectors referenced by routes exist in the detector definitions, and that no route has duplicate input/output detectors with the same server name. This prevents misconfiguration at startup rather than at request time, which is a good fail-fast pattern. However, the config is only validated at startup — there is no hot-reload mechanism, so configuration changes require a pod restart.

The streaming implementation (main.rs:259-349) processes SSE chunks individually, parsing each for detection metadata and optionally replacing content with the fallback message. This means detection-triggered fallback works per-chunk in streaming mode, though the current implementation replaces content on the first chunk that carries detection metadata rather than buffering all chunks. This is a pragmatic design choice that trades perfect detection coverage for low latency.
