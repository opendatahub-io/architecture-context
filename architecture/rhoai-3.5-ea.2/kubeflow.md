## Purpose

**Short**: Manages the lifecycle of Jupyter Notebook workbenches on OpenShift, reconciling Notebook custom resources into StatefulSets, Services, HTTPRoutes, and authentication sidecars.

**Detailed**: The kubeflow repository contains two Go controllers that work together to provide Jupyter Notebook workbench management for Red Hat OpenShift AI (RHOAI). The upstream **notebook-controller** (from the Kubeflow project) reconciles `Notebook` custom resources into Kubernetes StatefulSets and Services, and includes an idle notebook culling subsystem that monitors Jupyter kernel/terminal activity via HTTP and scales down idle notebooks. The downstream **odh-notebook-controller** extends this with OpenShift-specific capabilities: Gateway API HTTPRoute creation for external access, kube-rbac-proxy sidecar injection for authentication, integration with Data Science Pipelines (DSPA) for Elyra pipeline runtime configuration, MLflow tracking URI injection, Feast feature store config mounting, OpenShift ImageStream-based image resolution, network policy enforcement, and CA certificate bundle aggregation.

Both controllers are built as separate container images via Konflux and deployed as Deployments in the `opendatahub` namespace. The odh-notebook-controller also registers mutating and validating admission webhooks that intercept Notebook create/update operations to inject sidecars, volumes, and environment variables at admission time. The RHOAI 3.x architecture uses Gateway API (HTTPRoute + ReferenceGrant) for notebook ingress and kube-rbac-proxy for per-notebook authentication, replacing the RHOAI 2.x pattern of OpenShift Routes + oauth-proxy.

### odh-kf-notebook-controller

**Dockerfile**: `components/notebook-controller/Dockerfile.konflux`
**Intent**: Primary upstream notebook lifecycle manager. Deployed as a Deployment in the platform namespace. Reconciles every Notebook CR into a StatefulSet (for the Jupyter server pod) and a ClusterIP Service. ## Architectural Analysis

The kubeflow repository implements a layered controller architecture where the upstream Kubeflow notebook-controller provides core lifecycle management (StatefulSet + Service reconciliation, idle culling) and the ODH extension controller adds platform-specific capabilities on top. Both controllers watch the same `Notebook` CRD but handle different aspects of the reconciliation: the upstream controller owns the StatefulSet and Service, while the ODH controller owns HTTPRoutes, NetworkPolicies, ConfigMaps, Secrets, ServiceAccounts, and RoleBindings. This separation preserves upgrade compatibility with the upstream Kubeflow project while allowing significant platform-specific extensions.

The RHOAI 3.x ingress pattern is fully implemented here: each notebook gets an HTTPRoute in the central (`openshift-ingress`) namespace that references the platform's `data-science-gateway` Gateway, with a ReferenceGrant in the notebook's namespace allowing the cross-namespace Service reference. When auth injection is enabled (via `notebooks.opendatahub.io/inject-auth: "true"` annotation), the mutating webhook injects a kube-rbac-proxy sidecar container that fronts the Jupyter server on port 8443 with TLS and Kubernetes RBAC-based authentication. This replaces the RHOAI 2.x pattern of per-notebook OAuth proxy sidecars and OpenShift Routes. Legacy OAuthClient cleanup is preserved via a finalizer for migration from 2.x.

The webhook architecture is notably heavy — the mutating webhook handles image resolution (ImageStream tag lookup), CA bundle injection, pipeline runtime images, Elyra DSPA secret creation, Feast config mounting, MLflow env var injection, kube-rbac-proxy sidecar injection, proxy env var injection, and update-blocking logic for running notebooks. This "do everything at admission" pattern ensures notebooks are fully configured before the pod starts, but means the webhook is a critical path component — if it fails, no notebooks can be created or updated. The `failurePolicy: Fail` setting on both webhooks confirms this is intentional.

The controller uses a custom kube-rbac-proxy wrapper image (`odh-kube-auth-proxy`) rather than the standard OpenShift kube-rbac-proxy, pinned by digest in `params.env`. This is noted as temporary (per RHOAIENG-35698). The kube-rbac-proxy sidecar performs TokenReview and SubjectAccessReview against the Kubernetes API server, with the authorization config stored in a per-notebook ConfigMap. Network isolation is enforced via two NetworkPolicies per notebook: one allowing the controller to reach port 8888, and one allowing external traffic to the kube-rbac-proxy port 8443.

The controller integrates with the OpenShift TLS security profile system via `openshift/controller-runtime-common`, which configures the webhook server's TLS cipher suites based on the cluster's `APIServer` CR (`config.openshift.io/v1`). This ensures the webhook server's TLS configuration matches the cluster-wide security policy (Intermediate by default, with MinVersion TLS 1.2). The build-time FIPS configuration is complete — both Dockerfiles use `GOEXPERIMENT=strictfipsruntime` with `CGO_ENABLED=1`, and the runtime images are based on UBI9 which includes OpenSSL.
