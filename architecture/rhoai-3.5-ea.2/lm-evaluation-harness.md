## Purpose

**Short**: LMEval is a secure, batch-oriented execution environment for running language model evaluations using the lm-evaluation-harness framework, integrated with the EvalHub platform.

**Detailed**: lm-evaluation-harness is a downstream fork of EleutherAI's [lm-evaluation-harness](https://github.com/EleutherAI/lm-evaluation-harness) (v0.4.8), extended with a Red Hat-specific adapter layer (`main.py`) that integrates it into the EvalHub evaluation orchestration platform. The component runs as a one-shot Kubernetes Job — it receives a job specification (mounted as `/meta/job.json` via ConfigMap), connects to an OpenAI-compatible model inference endpoint, executes a configured benchmark evaluation, and reports results back to EvalHub via callback URLs. Results are optionally persisted as OCI artifacts to a container registry and logged to MLflow.

The adapter layer handles model credential resolution (API keys, HF tokens, custom CA certificates), air-gapped/offline operation (auto-detecting local tokenizer and dataset files under `/test_data`), and integration with the `eval-hub-sdk` for lifecycle callbacks (status updates, error reporting, result submission). The harness supports hundreds of benchmark tasks (MMLU, HellaSwag, ARC, GPQA, BBH, IFEval, etc.) configured via YAML, plus custom tasks through the Unitxt framework. It uses the `local-completions` model backend to send text prompts to remote inference endpoints, supporting concurrent requests for throughput optimization.

The container image is built for four architectures (x86_64, arm64, ppc64le, s390x) and includes platform-specific build stages for PyTorch and Apache Arrow on s390x and ppc64le where pre-built wheels are unavailable.

## Architectural Analysis

The lm-evaluation-harness is architecturally distinctive within the RHOAI platform because it is not a long-running service or operator — it is a one-shot batch execution container. The TrustyAI lm-eval-controller operator creates Kubernetes Jobs using this image, with each Job executing a single benchmark evaluation against a specified model. This fire-and-forget pattern means the component has no ingress, no persistent state, and no RBAC definitions of its own; all infrastructure concerns are delegated to the deploying operator.

The adapter pattern (`main.py`) cleanly separates Red Hat-specific integration from the upstream evaluation engine. The `LMEvalAdapter` class implements the `FrameworkAdapter` interface from `eval-hub-sdk`, translating EvalHub's `JobSpec` into lm-evaluation-harness parameters and bridging lifecycle events back through callbacks. This design allows the upstream evaluation library (`lm_eval/`) to remain largely unmodified, simplifying rebase operations. Key Red Hat extensions include: credential resolution via the eval-hub-sdk auth module, automatic offline mode detection from `/test_data` filesystem layout, error message sanitization (redacting Bearer tokens, API keys, and query parameters before sending to external callbacks), and HTTP error code mapping to EvalHub-specific message codes (e.g., `gated_dataset_auth_required`, `model_rate_limited`).

The multi-architecture build is notably complex. The Konflux Dockerfile has five stages: a base builder, an Apache Arrow builder (compiles Arrow from source on ppc64le/s390x), a PyTorch builder (compiles PyTorch from source on ppc64le/s390x), an OpenBLAS builder (compiles OpenBLAS on ppc64le), and the final application image. On x86_64 and arm64, pre-built wheels from PyPI are used directly. The s390x build includes a custom patch (`patches/s390x/parquet-support.patch`) for Arrow parquet support. This per-architecture build complexity is the primary reason for the 8-hour pipeline timeout configured in the Tekton PipelineRun.

The offline/air-gapped support (`_infer_auto_offline_from_local_test_data`) is carefully designed: it detects disconnected operation by checking whether `parameters.tokenizer` points to a path under `/test_data` and whether a sibling directory contains `dataset_dict.json`. When detected, it sets all HuggingFace environment variables (`HF_HUB_OFFLINE`, `HF_DATASETS_OFFLINE`, `TRANSFORMERS_OFFLINE`) before any HuggingFace library imports occur, and patches `evaluate.load()` to load metrics from a local `metrics/` directory. The task download path in `lm_eval/api/task.py` similarly checks for offline bundles under `HF_HOME` and falls back to `load_from_disk()`. This two-layer offline detection (adapter-level and library-level) ensures reliable air-gapped operation.

Security-wise, the adapter takes care to sanitize error messages before they reach external systems. The `_sanitize_error_message()` function strips Authorization headers, Bearer tokens, key=value secrets, JSON-style credential fields, URL userinfo, query parameters, and fragments. This defense-in-depth approach prevents credential leakage through error callbacks. The CA certificate handling in `api_models.py` explicitly rejects symlinks for `verify_certificate` paths, mitigating symlink-based path traversal attacks against the TLS trust store. The job spec path is confined to `/meta/` via `_resolve_job_spec_path_for_read()`, preventing path traversal (CWE-22) when reading the job specification.
