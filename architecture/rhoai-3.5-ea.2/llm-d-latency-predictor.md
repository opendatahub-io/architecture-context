## Purpose

**Short**: Online ML-based latency prediction service for the llm-d inference gateway ecosystem, predicting Time To First Token (TTFT) and Time Per Output Token (TPOT) for LLM inference requests.

**Detailed**: The llm-d-latency-predictor is a two-server ML system that provides real-time latency predictions for LLM inference workloads. It consists of a training server that continuously learns from observed request traces (actual TTFT and TPOT measurements along with system state features like KV cache utilization, queue depth, and prefix cache scores), and a prediction server that serves low-latency inference requests using the trained models.

The system supports three model backends — XGBoost (default), LightGBM, and Bayesian Ridge — with both quantile regression (default p90) and mean prediction objectives. A key architectural feature is the queue-gated ensemble mode, which trains separate sub-models for "noqueue" (idle) and "queued" (congested) regimes, improving prediction accuracy across different load conditions. Models are serialized as joblib files and synchronized from the training server to prediction server replicas via HTTP polling.

Within the llm-d ecosystem, this component enables the Gateway API Inference Extension to make latency-aware routing decisions, directing incoming LLM requests to backend pods that are predicted to deliver the best latency characteristics based on current system state.

## Architectural Analysis

The llm-d-latency-predictor follows a clean two-tier architecture that decouples model training from model serving, connected by an HTTP-based model synchronization mechanism. This design avoids the need for shared persistent volumes (ReadWriteMany) across prediction server replicas — each replica maintains its own local model cache in an emptyDir volume and independently polls the training server for updates. The `ModelSyncer` class implements conditional downloads using server-side timestamps, atomic file replacement via `os.replace()`, and joblib integrity validation before swapping in new models.

The prediction server is optimized for high-throughput inference. The `predict_batch_fast()` method constructs DataFrames from dict-of-numpy-arrays rather than list-of-dicts, avoiding Pandas per-row type inference overhead. Bulk responses use ORJSONResponse with plain dicts instead of Pydantic model construction per prediction. Model pointer swaps under the lock are microsecond operations — all heavy computation (DataFrame construction, model inference) runs without holding the lock, enabling concurrent prediction across uvicorn workers.

The queue-gated ensemble mode is a notable design decision. When enabled (default), the system trains separate sub-models for idle (num_request_waiting=0) and congested (num_request_waiting>0) regimes, packaged into `QueueGatedModel` wrappers. At prediction time, the system routes to the appropriate sub-model based on queue depth. This addresses the fundamental non-linearity in LLM latency behavior: idle pods exhibit latency dominated by compute (input token processing), while queued pods exhibit latency dominated by scheduling and resource contention. The ensemble avoids a single model trying to fit both regimes with the same parameters.

A security concern is the complete absence of authentication on all API endpoints, including the training data ingestion endpoint (`/add_training_data_bulk`). An attacker with network access to the training server could inject adversarial training data to manipulate model predictions, potentially causing the gateway router to make suboptimal routing decisions. The model download path is also unauthenticated plaintext HTTP. In production RHOAI deployments, these services should be isolated by NetworkPolicy and/or service mesh mTLS to prevent unauthorized access.

The model file transfer uses MD5 checksumming (via `hashlib.md5`) purely for change detection between sync cycles, not for cryptographic security. While MD5 is not FIPS-approved for cryptographic purposes, its use here is non-security-critical — it only determines whether a local model file differs from the previously loaded version to avoid redundant deserialization. The joblib deserialization itself serves as the real integrity check, since corrupt or malformed files will fail to deserialize.
