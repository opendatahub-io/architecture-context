## Purpose

**Short**: NeMo Guardrails is a programmable guardrails server that intercepts LLM-based conversational systems to enforce safety, security, and topic-adherence policies through configurable input/output rails.

**Detailed**: NeMo Guardrails, originally developed by NVIDIA, provides a FastAPI-based server with an OpenAI-compatible chat completion API that wraps any LLM provider with configurable safety guardrails. The server evaluates user inputs and model outputs against a library of safety rails including content safety checks, jailbreak detection, injection detection, sensitive data detection (PII), hallucination detection, fact-checking, and topic safety enforcement. Rails are defined using NVIDIA's Colang domain-specific language (supporting both v1.0 and v2.x syntax) and can leverage multiple LLM backends for evaluation.

In the RHOAI platform context, NeMo Guardrails is deployed as a standalone service container that can be placed in front of any LLM endpoint. It receives chat completion requests, applies configured input rails to validate user messages, forwards approved messages to the upstream LLM, then applies output rails to validate the LLM's response before returning it to the caller. The server supports streaming responses via SSE, thread-based conversation persistence (in-memory or Redis), and OpenTelemetry-based observability. The RHOAI distribution uses a build-time profile filter (`opensource`) that removes closed-source guardrail modules, retaining only open-source safety rails for the product build.

## Architectural Analysis

NeMo Guardrails follows a middleware/proxy architecture pattern: it sits between the client and the upstream LLM, intercepting requests to apply configurable safety policies. The server is stateless by default (no persistent storage required for guardrail evaluation), though it supports optional state management via thread IDs stored in either an in-memory dictionary or Redis. This design allows horizontal scaling of guardrail instances behind a load balancer.

The guardrail evaluation pipeline is organized as a sequence of "rails" executed in configurable order. Input rails validate user messages before they reach the LLM, output rails validate LLM responses before they're returned, and dialog/retrieval rails manage conversation flow. Each rail type is implemented as an "action" — a Python function decorated with `@action` that receives the conversation context and returns a pass/block decision. The rail library is extensible: operators can add custom rails by placing Colang flow files and Python action modules in the configuration directory. The RHOAI build uses a profile-based filter (`scripts/filter_guardrails.py`) to strip closed-source guardrail modules (ActiveFence, CrowdStrike, Fiddler, etc.) at build time, retaining only open-source rails like self-check, content safety, regex detection, sensitive data detection, and injection detection.

The server's OpenAI API compatibility is a key architectural decision — it implements `/v1/chat/completions` with the same request/response schema as the OpenAI API, extended with a `guardrails` field for configuration. This means NeMo Guardrails can be used as a drop-in replacement for any OpenAI-compatible endpoint, adding safety guardrails transparently. The `/v1/guardrail/checks` endpoint is a NeMo-specific extension that allows checking messages against guardrails without triggering an actual LLM generation — useful for pre-validation or content moderation pipelines. The server supports streaming via SSE (Server-Sent Events) for real-time token delivery with output rail evaluation.

The embedding and knowledge base subsystem uses Annoy (Approximate Nearest Neighbors) for semantic search with a multi-level caching strategy: content-hash-based index caching avoids rebuilding the nearest-neighbor index, while a pluggable embedding cache (in-memory, filesystem, or Redis) avoids recomputing embeddings. The embedding provider is configurable — FastEmbed is the default for local inference, with OpenAI, SentenceTransformers, and NVIDIA NIM as alternatives. The knowledge base supports markdown document ingestion with automatic chunking based on heading structure.

A notable security consideration is that NeMo Guardrails itself provides no authentication or authorization on its API endpoints. The server exposes HTTP (not HTTPS) on port 8000 and expects TLS termination and auth enforcement to be handled by the platform ingress layer (e.g., kube-rbac-proxy sidecar in RHOAI 3.x, or Istio service mesh). API keys for upstream LLM providers are managed via environment variables, and the server implements request header forwarding to propagate client auth tokens to the upstream LLM. The MD5 usage for embedding cache keys includes a FIPS-aware fallback that detects `hashlib.md5` availability and switches to SHA256 when running in FIPS mode.
