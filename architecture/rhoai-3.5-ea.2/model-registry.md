## Purpose

**Short**: Central metadata repository for ML models, versions, artifacts, and serving environments, providing REST API, web UI, model catalog, and KServe integration.

**Detailed**: Model Registry is a multi-component platform service that provides a centralized repository for model developers and MLOps teams to store and manage models, model versions, and artifact metadata. It serves as the metadata backbone for the AI/ML lifecycle on RHOAI, connecting model training outputs to serving infrastructure.

The system exposes a REST API (OpenAPI v1alpha3) backed by MySQL or PostgreSQL for CRUD operations on registered models, model versions, artifacts, inference services, serving environments, experiments, and experiment runs. A Kubernetes controller watches KServe InferenceService CRs and synchronizes them with Model Registry resources, enabling automatic tracking of deployed models. A CSI storage initializer allows KServe to pull model artifacts directly from the registry using `model-registry://` URIs.

The web UI, built as a React frontend with a Go Backend-for-Frontend (BFF), provides a user-facing interface for browsing, registering, and managing models across namespaces. A separate Model Catalog service offers a plugin-based discovery layer for curated model collections (from YAML, Hugging Face, and database sources) and MCP server catalogs. An asynchronous upload job handles model transfer between storage backends (S3, OCI, HTTP) with optional Sigstore-based signing.

## Architectural Analysis

Model Registry follows a hub-and-spoke architecture where the core REST API server acts as the central metadata store, and multiple satellite components (controller, CSI, UI, catalog, async-upload) interact with it through its OpenAPI v1alpha3 interface. This design enables independent deployment and scaling of each component while maintaining a single source of truth for model metadata.

The database abstraction layer is particularly well-designed, using a factory/registry pattern (`internal/platform/db/connector.go`) to support both MySQL and PostgreSQL through GORM. The connector includes robust retry logic (25 attempts with incremental backoff) and the TLS configuration module (`internal/platform/tls/config.go`) provides comprehensive cipher suite management with an explicit allowlist of 13 secure cipher suites and rejection of deprecated ones. However, the default `InsecureSkipVerify` setting being controlled by `!VerifyServerCert` (which defaults to `false`, meaning skip-verify is `true` by default) is a security concern for production deployments — the default behavior is to NOT verify server certificates.

The UI BFF implements a sophisticated authentication and authorization stack with two modes: `internal` (kubeflow-userid header from service mesh) and `user_token` (Bearer token forwarding). The middleware chain is well-layered: panic recovery → telemetry → identity injection → CORS → namespace attachment → SAR/SSAR checks → client creation → handler. The SAR-based authorization model scales well for multi-tenant deployments, with worker pool parallelism (10 workers) for namespace access checks.

The catalog service introduces a plugin architecture (`internal/plugin/`) with global registration, lifecycle management, and file watching for configuration hot-reload. The leader election via PostgreSQL pglock is pragmatic — it avoids requiring a separate distributed coordination service. The dual-plugin design (model catalog + MCP catalog) sharing a common base framework enables extensibility. However, the catalog currently has no authentication on its API endpoints, relying entirely on the upstream Istio AuthorizationPolicy or BFF token forwarding for access control.

The async-upload job represents a newer architectural pattern in the repository — a Python-based Kubernetes Job that bridges multiple storage backends. Its intent-driven workflow routing (create_model, create_version, update_artifact) with Pydantic-validated configuration is well-structured. The integration with Sigstore for model signing adds a supply chain security dimension. The component's use of UBI9/python-312-minimal rather than AIPCC base images is notable as a gap in the secure build pipeline.
