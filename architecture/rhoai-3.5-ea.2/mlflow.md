## Purpose

**Short**: MLflow is the experiment tracking, model registry, and artifact management service for Red Hat OpenShift AI, providing a web UI and REST/GraphQL APIs for managing the full machine learning lifecycle.

**Detailed**: MLflow provides a centralized tracking server for recording and querying ML experiments, runs, metrics, parameters, and artifacts. It exposes a hybrid FastAPI/Flask REST API and a GraphQL interface for the React-based web UI. The server supports multi-tenant workspaces (backed by Kubernetes-native workspace stores), role-based access control (RBAC) with workspace-scoped permissions, and envelope encryption for secrets management using AES-256-GCM.

In the RHOAI distribution, MLflow is deployed as a containerized Python service on OpenShift. The Konflux-built container image bundles the Python backend, the pre-built React frontend (including module federation support for embedding in other UIs such as EvalHub), and all runtime dependencies installed from the AIPCC Python Package Index. The server is designed to run behind the platform's Gateway API ingress and supports TLS termination, Prometheus metrics export, and database backends including PostgreSQL and SQLite.

MLflow also includes an AI Gateway for proxying LLM invocations, an OpenTelemetry OTLP HTTP endpoint for trace ingestion, and a job execution subsystem using a Huey task queue. A Helm chart is provided for standalone Kubernetes deployments outside the RHOAI operator-managed flow.

## Architectural Analysis

MLflow's server architecture is a deliberate hybrid: a modern FastAPI ASGI application that wraps the legacy Flask app via WSGI middleware, preserving full backward compatibility while enabling new endpoints to be developed natively in FastAPI. The `_EfficientWSGIMiddleware` class (in `fastapi_app.py`) patches Starlette's deprecated WSGI middleware to avoid O(n²) body buffering, which is a pragmatic fix since Starlette has declined to fix this upstream. New subsystems (OTLP traces, AI Gateway, job execution, assistant) are implemented as FastAPI routers, while the extensive existing tracking API remains in Flask handlers.

The RHOAI fork introduces several significant downstream modifications visible in the commit history: Kubernetes-native workspace stores (`--workspace-store-uri kubernetes://`), the `--app-name kubernetes-auth` authentication mode (via `mlflow-kubernetes-plugins`), and module federation support for embedding MLflow UI components in the RHOAI EvalHub dashboard. The `pyproject.release.toml` splits the full MLflow package into three installable components (`mlflow`, `mlflow-skinny`, `mlflow-tracing`), all of which are built as separate wheels in the Konflux Dockerfile and installed in the final image.

The security architecture is multi-layered: MLflow provides its own RBAC system with workspace-scoped roles and permissions (stored in a separate auth database), envelope encryption for secrets using AES-256-GCM with KEK-derived keys (PBKDF2-HMAC-SHA256, 600,000 iterations), host header validation against DNS rebinding, CORS protection, and security headers. In RHOAI, an additional layer is provided by the kube-rbac-proxy sidecar which enforces Kubernetes RBAC before requests reach MLflow. The Helm chart includes strong pod security defaults (non-root UID 1000, read-only root filesystem, dropped capabilities, seccomp RuntimeDefault).

The build pipeline demonstrates mature supply chain security: all Python dependencies are locked with hash verification across four architecture targets (amd64, arm64, ppc64le, s390x), RPM dependencies are locked via `rpms.lock.yaml`, JavaScript dependencies are locked via Yarn Berry, and Hermeto (cachi2) prefetch is configured for fully hermetic builds. The AIPCC Python Package Index is used for runtime dependencies but the container base is standard UBI9 ubi-minimal rather than an AIPCC base image, which is appropriate since MLflow is a control-plane service without accelerator requirements.

The network policy configuration in the Helm chart is notable for its awareness of OpenShift OVN-Kubernetes networking: DNS egress rules include both standard port 53 and OVN-K's post-DNAT port 5353, and S3/MinIO egress includes ports 9000, 8333, and 8334 to cover various object storage implementations. The garbage collection CronJob is a thoughtful operational feature that runs `mlflow gc` to permanently remove soft-deleted resources, with support for age-based filtering and multi-workspace scoping.
