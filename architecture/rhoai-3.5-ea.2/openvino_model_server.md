## Purpose

**Short**: OpenVINO Model Server is a high-performance C++ inference serving system that hosts AI models and exposes them via gRPC and REST APIs using OpenVINO as the inference backend.

**Detailed**: OpenVINO Model Server (OVMS) provides a production-ready serving infrastructure for machine learning models optimized for Intel architectures. It implements multiple serving protocols -- TensorFlow Serving API, KServe Inference Protocol v2, and OpenAI-compatible API -- allowing clients to perform remote inference via gRPC or REST without framework dependencies. The server supports traditional vision and NLP models, large language models (LLMs) with continuous batching, visual language models (VLMs), text embeddings, document reranking, speech-to-text, text-to-speech, and image generation.

Within the RHOAI platform, OVMS serves as the default model serving runtime for the KServe/ModelMesh ecosystem. It is deployed as a container within InferenceService pods, receiving inference requests routed through the platform's ingress infrastructure. The server supports loading models from local storage, S3, Azure Blob Storage, Google Cloud Storage, and HuggingFace Hub. It uses Bazel as its build system and compiles OpenVINO from source (via the opendatahub-io fork) for the RHOAI distribution, enabling LTO and GPU driver support.

## Architectural Analysis

OpenVINO Model Server follows a modular singleton-based architecture where the Server class initializes nine modules in a defined order: Profiler, Servable Manager, HF Model Pull, Config Manager, Python Interpreter, Metrics, gRPC Server, HTTP Server, and C API. Each module implements a `Module` base class with `start()` and `shutdown()` lifecycle methods. The shutdown order reverses initialization. This design allows clean startup sequencing -- the gRPC and HTTP servers only start after models are loaded and ready.

The inference architecture supports two distinct pipeline paradigms. The first is the traditional model serving path where individual model instances are loaded into OpenVINO and exposed directly via TFS or KServe APIs. The second is the MediaPipe graph path where models are composed into calculator graphs for complex multi-step pipelines. The LLM, embedding, rerank, STT, TTS, and image generation capabilities all run as MediaPipe calculators, receiving input as MediaPipe packets and emitting output through the graph framework. This means the newer AI capabilities (LLM chat, embeddings, etc.) are architecturally layered on top of the MediaPipe graph engine rather than the direct model serving path.

For LLM serving specifically, OVMS provides two execution strategies: continuous batching (via OpenVINO GenAI's ContinuousBatchingPipeline) and legacy sequential processing (via LLMPipeline). The continuous batching path is the primary production path, supporting dynamic request batching, KV cache management with configurable eviction strategies (SnapKV, KVCrush), speculative decoding (EAGLE3, prompt lookup), and multiple model-specific IO processing implementations (Hermes3, Qwen3, Llama3, Mistral, GPT-OSS, Phi4, Devstral). The architecture handles both text-only LLMs and visual language models (VLMs) through a class hierarchy rooted in `GenAiServable`.

A notable architectural characteristic is that OVMS does not implement TLS termination. Both the gRPC server (using `InsecureServerCredentials`) and the Drogon HTTP server operate over plaintext. This is by design -- in RHOAI deployments, TLS is provided by the platform's ingress stack (Envoy proxy + kube-rbac-proxy sidecar). However, this means standalone deployments outside of RHOAI/Kubernetes require an external reverse proxy for encryption. The only authentication mechanism is an optional API key (passed via `Authorization: Bearer` header), which provides minimal security without TLS.

The build process for the RHOAI distribution is complex and resource-intensive. It compiles OpenVINO, OpenVINO Tokenizers, and OpenVINO GenAI from source using the opendatahub-io fork branches (releases/2026/1), builds Boost, Azure SDK, OpenCV, and pugixml from source, then compiles OVMS itself using Bazel with both the server binary and the C API shared library. The Konflux Dockerfile is a multi-stage build with five stages (base_build, build, capi-build, pkg, release). LTO (Link-Time Optimization) is enabled in the RHOAI build configuration, and GPU driver support (Intel compute runtime 25.05.32567) is included. The final release image is based on ubi9-minimal and runs as non-root user `ovms` (UID 5000).
