RBAC Summary:
- No RBAC resources defined in the repository
- RBAC is managed by KServe and the platform operator
- Example binding only: user-one-view → view ClusterRole (demo purposes)

Service Mesh Configuration:
- Istio sidecar injection: sidecar.istio.io/inject: 'true' (on detector pods)
- mTLS: Provided by Istio sidecar (pod-to-pod encryption)
- Auth annotation: security.opendatahub.io/enable-auth: 'true' (KServe InferenceService)
- AuthZ: kube-rbac-proxy (when auth enabled via platform annotation)

Secrets:
- aws-connection-minio-data-connection-guardrails-guardian (Opaque/S3)
  Purpose: S3 credentials for model storage access
  Provisioned: Manual (example manifest)
  Auto-Rotate: No

- git_auth_secret (Opaque)
  Purpose: Git authentication for Tekton pipeline
  Provisioned: Konflux/PaC
  Auto-Rotate: No

TLS Configuration:
- Application supports optional TLS via ssl_ca_certs, ssl_certfile, ssl_keyfile in config YAML
- Defaults to plaintext HTTP (TLS handled by Istio sidecar in production)
- Python ssl module delegates to UBI9 system OpenSSL (FIPS-compliant)
- No non-FIPS crypto algorithms detected

Security Concerns:
- Custom detector sandbox relies on AST-based static analysis (no runtime isolation)
- Blocked imports: os, subprocess, sys, shutil
- Blocked calls: eval, exec, open, compile, input
- No seccomp or capability restrictions at Python level
- Non-hermetic builds (hermetic: false in Tekton PipelineRuns)
- PyTorch source-built from GitHub for ppc64le/s390x during build (supply chain risk)
- S3 connection uses HTTP (port 9000) without encryption in example manifests
