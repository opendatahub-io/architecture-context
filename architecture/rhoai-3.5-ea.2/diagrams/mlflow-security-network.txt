RBAC Summary:
- MLflow Helm chart does not create RBAC by default
- Roles created only when namespace_rbac.extraRules or cluster_rbac.extraRules configured
- In RHOAI, RBAC is managed by the platform operator (rhods-operator)
- Application-level RBAC: workspace-scoped roles and permissions
  - Per-resource permissions: experiments, runs, registered-models
  - Admin-only endpoints: user management, role management, permission grant/revoke

Authentication Layers:
1. Platform Gateway (TLS termination, HTTPRoute routing)
2. kube-rbac-proxy sidecar (JWT validation via Kubernetes TokenReview + SubjectAccessReview)
3. MLflow app-level RBAC (workspace-scoped roles and permissions per resource)
4. OTLP /v1/traces endpoint: No auth (OTLP standard - open for trace ingestion)

Secrets:
- mlflow-tls (kubernetes.io/tls) - TLS cert/key for HTTPS, cert-manager/manual, No auto-rotate
- mlflow-db-secret (Opaque) - PostgreSQL connection URI, platform operator/manual, No auto-rotate
- MLFLOW_CRYPTO_KEK_PASSPHRASE (env var) - KEK for AES-256-GCM envelope encryption, platform operator
- MLFLOW_FLASK_SERVER_SECRET_KEY (env var) - Flask session cookie signing key, platform operator

Encryption Summary:
- External ingress: TLS 1.2+ (Platform Gateway)
- Auth proxy: TLS (kube-rbac-proxy on 8443)
- Internal app: HTTP (plaintext on 5000, localhost only)
- Database: Plaintext PostgreSQL wire protocol (in-cluster)
- Artifact storage: TLS 1.2+ (S3/MinIO HTTPS)
- Kubernetes API: TLS 1.2+ (6443 HTTPS)
- LLM Providers: TLS 1.2+ (443 HTTPS)
- Secrets at rest: AES-256-GCM with PBKDF2-HMAC-SHA256 (600,000 iterations)

FIPS Compliance:
- Python cryptography library (v43+) uses OpenSSL backend on UBI9
- FIPS compliance inherited from host OS FIPS mode
- No non-FIPS crypto detected (no MD5/RC4/DES in production code)
- AES-256-GCM with PBKDF2-HMAC-SHA256 used for envelope encryption
