RBAC Summary:
- Not applicable — standalone service binary, not a Kubernetes operator
- No ClusterRoles, Roles, or RoleBindings defined

Secrets:
- TLS server cert/key (PEM files) - Server-side TLS for API server, external provisioning, no auto-rotate
- TLS client CA cert (PEM file) - mTLS client verification, external provisioning, no auto-rotate
- Per-service TLS certs (PEM files) - Client TLS for outbound connections, per-service config in config.yaml
- API tokens (env var) - Bearer token for OpenAI-compatible servers, injected via environment variable

Authentication & Authorization:
- Header passthrough: Configurable passthrough_headers list in config.yaml
- X-Forwarded-Access-Token rewrite: Orchestrator middleware rewrites to Authorization: Bearer
- mTLS: Optional server-level mTLS when tls_client_ca_cert_path configured
- Downstream auth: Per-service api_token or mTLS client certificates

FIPS Compliance Notes:
- ⚠️ CRITICAL: rustls with ring crypto backend is NOT FIPS-validated
- OpenSSL installed in container image (passes check-payload) but NOT used by Rust binary
- NoVerifier mode (insecure: true) bypasses ALL certificate verification — dev only
- Known architectural gap: passes image-level FIPS gate but uses non-FIPS crypto at runtime

TLS Configuration:
- Server TLS: rustls ServerConfig with ring backend
- Client TLS (gRPC): tonic TLS with per-service cert/key/CA
- Client TLS (HTTP): hyper-rustls with per-service configuration
- Named TLS configs: config.yaml allows different TLS settings per downstream service
