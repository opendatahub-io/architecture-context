RBAC Summary:
- None. OVMS is a service container and does not interact with the Kubernetes API.
- RBAC is managed by KServe/ModelMesh at the platform level.

Service Mesh Configuration:
- OVMS does not configure service mesh directly.
- When deployed in RHOAI, pod-level network policies and service mesh
  configuration are managed by the platform operator.
- gRPC server uses InsecureServerCredentials (no TLS).
- HTTP server uses plaintext Drogon (no TLS).
- TLS termination is delegated to platform ingress (Envoy + kube-rbac-proxy).

Secrets:
- S3 credentials (Opaque) - AWS access key/secret for S3 model storage
  - Provisioned by: User/platform config
  - Auto-rotate: No
- Azure storage credentials (Opaque) - Azure storage account key
  - Provisioned by: User/platform config
  - Auto-rotate: No
- GCS credentials (Opaque) - Google Cloud service account JSON
  - Provisioned by: User/platform config
  - Auto-rotate: No
- HuggingFace token (Opaque) - HF Hub authentication token
  - Provisioned by: User/platform config
  - Auto-rotate: No
- API key (Opaque) - Optional API key for client authentication
  - Provisioned by: User/platform config
  - Auto-rotate: No

Authentication & Authorization:
- REST endpoints: Optional API Key (Authorization: Bearer header)
  - Enforced by: OVMS HTTP handler (if --api_key flag set)
- gRPC endpoints: No authentication at gRPC level
  - Relies on platform ingress for auth
- /metrics: Open (no auth)
- /v2/health/*: Open (no auth)

FIPS Compliance:
- C++ binary (not Go), check-payload does not apply
- Dynamic linking against system OpenSSL via UBI 9 base image
- No application-level TLS termination
- Cloud SDK TLS delegates to system OpenSSL (FIPS-validated in RHEL 9)

Certificate Trust:
- System trust store: /etc/ssl/certs/ca-bundle.crt → ca-certificates.crt
- Used by cloud storage SDKs for TLS verification

Container Security:
- Base image: registry.access.redhat.com/ubi9/ubi-minimal:9.7
- Runs as non-root user: ovms (UID 5000)
