## Diagram Descriptions

### Component Dependency Graph
Shows how platform components depend on each other. The rhods-operator sits at the top as the platform orchestrator deploying all 65 components. KServe is the second most central component with 8+ reverse dependencies. Color-coded by function: blue (core/serving), green (UI), purple (ML lifecycle), gold (pipelines/training), gray (security), orange (external). Useful for understanding blast radius of changes and planning upgrades.

### Platform Network Topology
Complete network architecture showing all ingress points, namespace boundaries, and egress destinations. Trust zones: External → Ingress (opendatahub-gateway-system) → Platform Services (redhat-ods-applications) → Operators (redhat-ods-operator) → User Namespaces → Egress.

**Two formats provided**:
- **Mermaid (.mmd)**: Visual diagram with color-coded trust zones. Great for presentations and architecture discussions.
- **ASCII (.txt)**: Precise text format with exact port numbers, protocols, TLS versions, and authentication mechanisms. Required for SAR (Security Architecture Review) submissions and compliance documentation. Includes authentication flow summaries and security posture notes.

Both formats contain the same information — use Mermaid for visual clarity, ASCII for security reviews.

### Cross-Component Workflows
Three sequence diagrams showing end-to-end flows:

1. **LLM Model Serving** (`platform-workflows.mmd`) — The most architecturally significant flow in RHOAI 3.5. Shows the complete path from user creating an LLMInferenceService through KServe reconciliation, llm-d EPP scheduling (20+ plugins), Gateway API + Kuadrant AuthPolicy setup, and external inference request processing.

2. **Pipeline Execution** (`platform-workflow-pipeline.mmd`) — Shows DSP API → MariaDB → ML Metadata → Argo Workflows → pipeline step pods with driver/launcher pattern → S3 artifact staging → persistence agent status sync.

3. **MaaS External Provider Routing** (`platform-workflow-maas.mmd`) — Shows ExternalProvider/ExternalModel CR creation, controller reconciliation (ServiceEntry + HTTPRoute + AuthPolicy), and the ext_proc pre-auth/post-auth pipeline for API format translation and credential injection.

### Platform Security Overview
Visual representation of the platform's defense-in-depth security architecture:
- **6 authentication mechanisms**: kube-rbac-proxy (SubjectAccessReview), Gateway API + Kuadrant, OAuth2/OIDC, AuthBridge mTLS (SPIFFE), API Key SHA-256, ServiceAccount tokens
- **Container security**: UBI9 base images (96+), Go FIPS runtime, distroless images (28), non-root execution, capability drops
- **TLS configuration**: TLS 1.3 at ingress, kube-rbac-proxy TLS on :8443, webhook TLS on :9443, noted FIPS gaps
- **RBAC**: 684 role entries across 5 key roles (rhods-operator, KServe, odh-model-controller, DSPO, agents-operator)
- **NetworkPolicies**: 21 policies with per-component isolation (DSPO MariaDB, MaaS gateway, Model Registry PostgreSQL)

### Platform Observability
Maps all monitoring and observability endpoints:
- **Prometheus scrape targets**: All operators expose metrics on :8443 behind kube-rbac-proxy; platform services on dedicated ports; model serving workload metrics
- **Health probes**: controller-runtime standard pattern (liveness :8081 healthz, readiness :8081 readyz)
- **Webhook endpoints**: :9443 TLS for admission and conversion webhooks
- **gRPC services**: 63 endpoints including DSP :8887, MLMD :8080, Argo :2746, MLServer :8081, ext_proc :9004

### Platform Maturity Dashboard
High-level metrics about platform health and readiness:
- Component statistics (65 total, 26 operators, 138 CRDs)
- API maturity (mixed CRD versions from v1alpha1 through v2)
- Security posture (MIXED mTLS, PARTIAL FIPS, STRONG container security)
- Deployment readiness (disconnected ENABLED, multi-arch PARTIAL)
- Ingress architecture transition (3 generations coexisting)
- Language distribution (Go, Python, Rust, TypeScript, Java, C++)
