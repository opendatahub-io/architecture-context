RBAC Summary:
- No Kubernetes RBAC resources defined in this repository
- RBAC managed by platform operator (rhods-operator)

Authentication & Authorization:
- Mechanism: OAuth2 Bearer Token (JWT) validated via JWKS endpoint
- Enforcement: OGX server built-in auth middleware
- Access Policy:
  - /v1/health: unauthenticated
  - Unowned resources: read allowed for all authenticated/unauthenticated
  - Create: requires authenticated user
  - Read/Update/Delete owned resources: resource owner only

Secrets (all delivered as environment variables):
- VLLM_API_TOKEN (inference auth) - no auto-rotation
- POSTGRES_PASSWORD (database auth) - no auto-rotation
- MILVUS_TOKEN, QDRANT_API_KEY (vector DB auth) - no auto-rotation
- AWS credentials (Bedrock/S3 auth) - STS rotation possible
- OPENAI_API_KEY, ANTHROPIC_API_KEY, AZURE_API_KEY,
  WATSONX_API_KEY (cloud provider auth) - no auto-rotation
- BRAVE_SEARCH_API_KEY, TAVILY_SEARCH_API_KEY (tool auth) - no auto-rotation
- rhel-ai-private-index-auth (build-time Konflux secret)
- git_auth_secret (build-time Pipelines-as-Code secret)

FIPS Compliance:
- Python application (no Go binaries, no custom crypto)
- FIPS via system OpenSSL from AIPCC/UBI base image
- TLS delegated to Python ssl module → system OpenSSL
- Provider SDKs (boto3, google-genai, anthropic) use system OpenSSL
- No non-FIPS crypto risks detected in repository code

Build Hermeticity:
- Build is NON-HERMETIC (hermetic: false in Tekton pipeline)
- requirements.txt has no version pins or hashes for most packages
- No rpms.lock.yaml or artifacts.lock.yaml present
- Supply chain risk: each build may resolve different package versions
- Partially mitigated by curated AIPCC Python Package Index

Network Security Observations:
- OGX server listens on plaintext HTTP (8321/TCP) - TLS termination by platform
- PostgreSQL connection is plaintext (no SSL/TLS) - relies on network isolation
- pgvector connection is plaintext - same risk as PostgreSQL
- VLLM_TLS_VERIFY is configurable (default: true) - can be disabled
- AUTH_VERIFY_TLS is configurable (default: true) - can be disabled
- Milvus supports mTLS with CA/client certificates
- All cloud provider connections use TLS 1.2+
