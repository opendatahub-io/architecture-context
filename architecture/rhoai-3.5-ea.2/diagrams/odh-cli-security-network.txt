RBAC Summary:
- odh-cli does NOT install any ClusterRoles or RoleBindings
- It uses the RBAC permissions of the authenticated user's kubeconfig
- SelfSubjectAccessReview-based permission pre-check gracefully detects
  and reports missing permissions before operations
- Required permissions depend on the command being run:
  - lint: read-only access to DSC, DSCI, ISVCs, Notebooks, DSPAs, etc.
  - migrate: read + patch/create/delete on target workload CRs
  - deps install: create Subscriptions, OperatorGroups
  - status/components/events/logs/get: read-only access

Authentication & Authorization:
- Kubernetes API: Bearer Token / Client Certificate / OIDC (via kubeconfig)
- TrustyAI REST API: Bearer Token (inherited from OpenShift Route OAuth)
- No service accounts created or managed by odh-cli

FIPS Compliance:
- Build flags: GOEXPERIMENT=strictfipsruntime, CGO_ENABLED=1
- Dynamic linking with OpenSSL backend (UBI9 via ose-cli-rhel9)
- TLS delegated to k8s.io/client-go (Go stdlib crypto/tls with OpenSSL)
- yq also built with FIPS-compliant flags
- No direct crypto imports; all TLS via client-go and Go runtime

Secrets Handling:
- odh-cli does NOT create or manage Secrets
- Reads Secrets referenced by Notebooks during backup operations (read-only)
- Writes backup data to local filesystem only

Build Hermeticity:
- Konflux hermetic build (hermetic: true in Tekton pipeline)
- Prefetch: RPM (rpms.lock.yaml), Go modules (go.sum), Go modules/yq,
  pip packages (requirements.txt)
- Multi-arch: x86_64, arm64, ppc64le
