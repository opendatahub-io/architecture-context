- mTLS (STRICT mode) should be enforced at the namespace level by the
  platform to provide transport encryption for the plaintext HTTP listener.
- Recommendation: Inject Istio/Envoy sidecar for automatic mTLS.

Secrets:
- No secrets defined or consumed by this service.
- No TLS certificates (service is plaintext HTTP).
- No API keys or credentials required.

Security Observations:
- Container runs as root (no USER directive in Dockerfile) — should be remediated.
- No authentication at the application level — relies on network-level controls.
- No TLS at the application level — relies on service mesh or ingress TLS termination.
- The regex crate uses finite automaton matching (not backtracking), making it
  immune to ReDoS attacks even with user-supplied patterns.
- No persistent state, no database connections, no secrets — minimal attack surface.

FIPS Compliance:
- Build: compat-openssl11 installed in image; Rust binary dynamically linked.
- Application: No cryptographic operations performed (vacuously FIPS-compliant).
- Network encryption must be provided by the platform (service mesh mTLS).
