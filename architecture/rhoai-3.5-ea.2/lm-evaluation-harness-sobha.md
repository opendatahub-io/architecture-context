## Purpose

**Short**: LMEval is a secure, batch-oriented runtime for one-shot language model evaluations using the lm-evaluation-harness framework, integrated with EvalHub for job orchestration and results reporting.

**Detailed**: This component is a downstream fork of EleutherAI's [lm-evaluation-harness](https://github.com/EleutherAI/lm-evaluation-harness) (version 0.4.8) adapted for Red Hat OpenShift AI's TrustyAI evaluation pipeline. It provides a containerized execution environment (`odh-ta-lmes-job`) that runs language model benchmarks as Kubernetes batch jobs.

The primary entry point is `main.py`, which implements a `LMEvalAdapter` class conforming to the EvalHub `FrameworkAdapter` interface. When launched as a Kubernetes Job, the adapter reads a job specification from a mounted ConfigMap (`/meta/job.json`), resolves model credentials from Kubernetes secrets, configures an OpenAI-compatible completions backend (`local-completions`), executes the requested benchmark via `lm_eval.simple_evaluate()`, and reports results back to EvalHub through status callbacks and optional OCI artifact persistence. The adapter supports offline/air-gapped operation via automatic Hugging Face offline mode detection when a local tokenizer and datasets are present under `/test_data`.

The container image is built for multiple architectures (x86_64, arm64, ppc64le, s390x) with architecture-specific build stages for PyTorch and Apache Arrow compilation. The image ships with 150+ built-in benchmark tasks (MMLU, HellaSwag, ARC, TruthfulQA, GPQA, etc.) and the Unitxt framework for custom evaluation tasks.

## Architectural Analysis

The lm-evaluation-harness-sobha repository is a downstream fork of EleutherAI's lm-evaluation-harness, augmented with a custom EvalHub adapter layer (`main.py`) that transforms it from a standalone CLI tool into a managed batch evaluation service. The adapter pattern is clean: `LMEvalAdapter` extends `FrameworkAdapter` from `eval-hub-sdk`, implementing `run_benchmark_job()` to bridge the framework's `simple_evaluate()` function with EvalHub's job lifecycle protocol. The adapter handles concerns the upstream library was not designed for: credential resolution from Kubernetes secrets, phased status reporting, OCI artifact persistence, MLflow integration, and air-gapped operation.

The offline/air-gapped support deserves attention as a RHOAI-specific addition. The `_seed_hf_offline_before_lm_eval_import()` function runs before any Hugging Face library imports, detecting whether a local tokenizer path under `/test_data` co-exists with dataset files (identified by `dataset_dict.json`). When detected, it sets six environment variables (`HF_HOME`, `HF_HUB_CACHE`, `HF_DATASETS_CACHE`, `HF_HUB_OFFLINE`, `HF_DATASETS_OFFLINE`, `HF_EVALUATE_OFFLINE`, `TRANSFORMERS_OFFLINE`) to prevent any outbound calls to huggingface.co. The `__main__.py` also patches `evaluate.load` when in offline mode to redirect metric loading to a local `metrics/` directory containing 54 pre-packaged metric implementations. This is critical for disconnected OpenShift deployments.

The multi-architecture build strategy is complex. The `Dockerfile.konflux.lmes-job` uses a 5-stage build: (1) base builder with common tooling, (2) Apache Arrow compilation from source for ppc64le/s390x (with an s390x-specific parquet patch), (3) PyTorch compilation from source for ppc64le/s390x using gcc-toolset-13, (4) OpenBLAS compilation for ppc64le, and (5) final image assembly. For x86_64 and arm64, pre-built wheels from PyPI are used directly. This complexity exists because PyTorch and Arrow do not publish official wheels for ppc64le and s390x. The `TARGETARCH` build arg drives conditional logic throughout. The Tekton pipeline builds for all four architectures (`linux/x86_64`, `linux-m2xlarge/arm64`, `linux/ppc64le`, `linux/s390x`) and creates a multi-arch image index.

A significant security consideration is the error message sanitization in `_sanitize_error_message()` (lines 255-299 of `main.py`). Since evaluation failures may include raw exception text containing credentials (e.g., Bearer tokens in HTTP error responses), the adapter applies aggressive regex-based redaction of authorization headers, key=value pairs, JSON credential fields, and URL userinfo before passing error messages to EvalHub callbacks. This prevents credential leakage through the status reporting channel. The job spec path validation (`_resolve_job_spec_path_for_read`) also includes CWE-22 (path traversal) protection by resolving paths and checking they remain under `/meta`.

The component does not define its own RBAC, Services, or CRDs — it is a "dumb" container image that the TrustyAI LMEval operator instantiates as Kubernetes Jobs. All deployment configuration (service accounts, resource limits, volume mounts for secrets and configmaps) is managed by the operator. The `num_concurrent` parameter (capped at 128) controls how many inference requests are sent in parallel to the model endpoint, which is important for throughput when evaluating large benchmark suites. The adapter forces `device="cpu"` regardless of cluster GPU availability, since the actual model inference happens on the remote model serving endpoint — the job container only needs CPU for tokenization, dataset processing, and metrics computation.
