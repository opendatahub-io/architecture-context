## Purpose

**Short**: Python SDK providing a high-level interface for creating, managing, and submitting jobs to Ray clusters on Kubernetes via the KubeRay operator.

**Detailed**: The CodeFlare SDK is a Python client library that abstracts the complexity of interacting with KubeRay-managed Ray clusters on Kubernetes/OpenShift. It provides two primary workflows: (1) interactive cluster lifecycle management via the `Cluster` class, which generates RayCluster CRs with configurable resources, GPU/accelerator support, and Kueue queue integration; and (2) batch job submission via the `RayJob` class, which creates RayJob CRs with support for managed clusters, runtime environments, local file upload via Kubernetes Secrets, and Kueue workload priority.

The SDK is designed primarily for use within Jupyter notebooks on Red Hat OpenShift AI (RHOAI) and Open Data Hub (ODH) platforms. It includes IPython widgets for interactive cluster management, mTLS certificate generation for secure Ray client connections, and support for both RHOAI 3.x (Gateway API / HTTPRoute) and legacy 2.x (OpenShift Route) dashboard URL discovery patterns. Authentication is handled through the `kube-authkit` library with support for auto-detection, OIDC, OpenShift OAuth, and token-based authentication.

The SDK does not deploy any services itself — it is a pure client library distributed via PyPI (`pip install codeflare-sdk`). It interacts with the Kubernetes API to create and manage RayCluster and RayJob custom resources, which are then reconciled by the KubeRay operator running on the cluster.

## Architectural Analysis

The CodeFlare SDK follows a two-tier client architecture: a high-level Python API (`Cluster`, `RayJob`) that maps user intent into Kubernetes custom resources, and a low-level vendored KubeRay Python client that handles the actual Kubernetes API calls. This layering allows the SDK to provide a simplified developer experience while maintaining compatibility with KubeRay's evolving CR schemas. The vendored client (`src/codeflare_sdk/vendored/python_client/`) is a fork of the upstream KubeRay Python SDK, enabling the team to patch and extend it independently of upstream releases.

A notable design pattern is the progressive dashboard URL discovery chain in `cluster.py` (lines 612-669 and 1262-1419). The SDK implements a four-level fallback: (1) HTTPRoute with Gateway API labels (`ray.io/cluster-name`, `ray.io/cluster-namespace`) for RHOAI 3.x, (2) OpenShift Routes for RHOAI 2.x, (3) GatewayConfig CR for ODH/RHOAI domain resolution, and (4) Gateway status addresses as a last resort. This progression reflects the platform's migration from OpenShift-native Routes to Gateway API, and the SDK must support clusters at different upgrade stages simultaneously.

The mTLS certificate management (`generate_cert.py`) is security-conscious: certificates use 3072-bit RSA keys, include proper X.509 v3 extensions (BasicConstraints, KeyUsage, ExtendedKeyUsage for both serverAuth and clientAuth, SubjectAlternativeName with service DNS names), and are stored with 0600 permissions in a user-private directory (`~/.local/share/codeflare/tls/`). The `apply()` method now integrates TLS certificate generation, polling for the CA secret created by KubeRay before generating client certificates. This eliminates the previous manual step where users had to call `generate_cert` separately.

The RayJob subsystem (`ray/rayjobs/`) introduces a clean separation between managed and existing cluster workflows. When `cluster_config` is provided, the SDK embeds a complete `rayClusterSpec` in the RayJob CR and sets `shutdownAfterJobFinishes: true` for automatic cleanup. When `cluster_name` is provided instead, it uses `clusterSelector` to target an existing cluster. Local file handling is sophisticated: single entrypoint files are stored in Kubernetes Secrets, while working directories are zipped, base64-encoded, stored in Secrets, and unzipped via init containers on the submitter pod. The Kueue integration is deliberately non-blocking — if no Kueue is installed, labels are simply omitted and the job runs directly, matching the interactive Cluster behavior.

The authentication subsystem is undergoing migration from legacy `TokenAuthentication`/`KubeConfigFileAuthentication` classes to `kube-authkit`, a shared library supporting auto-detection, OIDC device flow, OpenShift OAuth, and kubeconfig. The legacy classes are deprecated but still functional, with clear migration paths documented. The `config_check()` function implements a priority chain: existing API client → kube-authkit auto-detection → legacy kubeconfig → in-cluster config → error.
