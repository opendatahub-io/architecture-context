## Purpose

**Short**: ML Metadata (MLMD) is a gRPC server that provides a metadata store for tracking ML artifacts, executions, contexts, and their lineage relationships in ML pipelines.

**Detailed**: ML Metadata is a Google-originated library and server for recording and retrieving metadata associated with ML developer and data scientist workflows. It provides a schema-driven metadata store that manages three core entity types — Artifacts (datasets, models, metrics), Executions (training runs, transformations), and Contexts (pipelines, experiments) — along with typed relationships between them (Events, Attributions, Associations, and ParentContexts). This forms a provenance graph that enables lineage tracking across ML workflows.

The server component is a C++ gRPC service built with Bazel that supports multiple database backends: MySQL/MariaDB, PostgreSQL, and SQLite (in-memory). It provides schema migration capabilities for database upgrades and downgrades. In the RHOAI platform, MLMD serves as the metadata backbone for ML Pipelines (Kubeflow Pipelines), storing pipeline run metadata, artifact lineage, and execution provenance. A Python client library (published as the `ml-metadata` PyPI package) provides a higher-level API for interacting with the gRPC server or directly with the database.

The Red Hat downstream fork (`red-hat-data-services/ml-metadata`) packages the upstream Google `ml-metadata` project into a UBI9-based container image (`odh-mlmd-grpc-server`) for deployment within RHOAI clusters. The downstream primarily maintains Dockerfile variants, Tekton/Konflux pipeline configurations, and UBI9 compatibility patches (e.g., ZetaSQL commit adjustments).

## Architectural Analysis

ML Metadata follows a clean separation between its storage abstraction and transport layer. The core metadata store logic is implemented in C++ through `MetadataStore` and `RDBMSMetadataAccessObject` classes that abstract over three SQL backends (MySQL, PostgreSQL, SQLite). The `MetadataSourceQueryConfig` proto (in `metadata_source.proto`) provides a template-based SQL query system where each backend can define its own SQL dialect for the same logical operations, with parameterized query templates supporting up to 10 variables. This design means adding a new SQL backend requires only implementing a new query config rather than modifying core logic.

The gRPC service layer (`MetadataStoreServiceImpl`) is a thin wrapper that delegates directly to the `MetadataStore` C++ class, with thread safety noted but sequential execution in the current implementation (as flagged by the TODO at line 29 of `metadata_store_service_impl.h` regarding connection pooling). The server supports schema migrations bidirectionally — both upgrade and downgrade — which is critical for the platform's ability to roll back versions. The migration system uses a versioned scheme stored in an `MLMDEnv` table, with each version having explicit upgrade/downgrade SQL queries and verification steps.

The Python client library is dual-mode: it can either connect directly to a database (using a C++ extension module via pybind11) or connect to the gRPC server as a remote client. The direct database mode uses `metadata_store_serialized` (pybind11 wrapper) while the gRPC mode uses the standard `grpc` Python library. Both modes present the same `MetadataStore` Python API, making the choice transparent to consumers. The client includes retry logic with exponential backoff for transactional `AbortedError` scenarios, which is important for concurrent pipeline execution environments.

The ZetaSQL dependency is notable — it provides the SQL parsing engine for the `filter_query` functionality in `ListOperationOptions`, enabling SQL-like boolean expressions for filtering artifacts, executions, and contexts. This is a substantial dependency (requiring C++17 and specific commit pinning) that enables powerful query capabilities but adds build complexity. The downstream fork maintains a patched ZetaSQL commit for UBI9 compatibility, visible in `Dockerfile.redhat` where the WORKSPACE file is patched to swap ZetaSQL commits.

From a security architecture perspective, the MLMD server itself does not implement authentication or authorization beyond optional TLS/mTLS at the gRPC transport layer. In the RHOAI deployment context, network-level security (NetworkPolicies, service mesh) and the platform operator's deployment configuration provide the actual security boundary. The server runs as a non-root user (UID 65534) in the container. The Konflux build is explicitly non-hermetic (`hermetic: false`), and there are no RPM lock files, which represents a gap compared to other RHOAI components that have fully hermetic builds.
