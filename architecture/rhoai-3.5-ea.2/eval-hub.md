## Purpose

**Short**: Evaluation Hub is a REST API service that orchestrates LLM evaluation jobs by managing evaluation providers, benchmark collections, and job execution on Kubernetes.

**Detailed**: Eval Hub serves as the backend orchestration layer for AI model evaluation within the Red Hat OpenShift AI platform. It provides a RESTful API for creating, managing, and monitoring evaluation jobs that run benchmarks against large language models. The service manages three core resource types: evaluation providers (which define benchmark runtimes like LM Evaluation Harness and Lighteval), benchmark collections (curated groups of benchmarks for specific evaluation goals), and evaluation jobs (execution units that run selected benchmarks against target models).

The system follows a multi-component architecture where a single container image ships four Go binaries: the main API server (`eval-hub`), a reverse proxy sidecar (`eval-runtime-sidecar`) that handles authentication and traffic routing to upstream services (eval-hub API, MLflow, OCI registries), an init container (`eval-runtime-init`) for downloading test data from S3, and a Model Context Protocol (MCP) server (`evalhub-mcp`) that enables AI-native integration for IDE-based evaluation workflows. Evaluation jobs are executed as Kubernetes Jobs, with the sidecar injected alongside provider-specific runtime containers. Results can optionally be tracked via MLflow integration. The service supports both PostgreSQL and SQLite backends, multi-tenant data isolation, OpenTelemetry tracing, and Prometheus metrics.

## Architectural Analysis

Eval Hub follows a hub-and-spoke architecture where the main API server acts as an orchestration hub, dispatching evaluation work to provider-specific runtime containers via Kubernetes Jobs. This design decouples the evaluation framework (LM Evaluation Harness, Lighteval, Garak, etc.) from the orchestration layer, allowing new evaluation providers to be added by simply registering a YAML configuration file with the runtime image reference, entrypoint, resource requirements, and benchmark definitions. The provider configuration system supports 180+ benchmarks across the LM Evaluation Harness provider alone, covering accuracy, reasoning, safety, multilingual, code, and long-context evaluation categories.

The sidecar proxy pattern is a key architectural decision. Rather than embedding auth logic and upstream client code in each evaluation runtime container, eval-hub injects a Go reverse proxy sidecar (`eval-runtime-sidecar`) into every evaluation Job pod. This sidecar handles three distinct upstream connections — eval-hub API (for status reporting), MLflow (for metrics logging), and OCI registries (for artifact access) — each with its own TLS configuration, bearer token source, and retry behavior. The auth token caching mechanism uses atomic pointers with a 5-minute TTL to avoid repeated file I/O for Kubernetes ServiceAccount tokens and MLflow tokens. This pattern keeps runtime containers lightweight and provider-agnostic while centralizing security-sensitive credential handling.

The MCP (Model Context Protocol) server is architecturally notable as one of the first RHOAI components to provide an AI-native integration surface. It exposes eval-hub's capabilities through the MCP standard (tools for job submission, resources for discovery, prompts for guided workflows), enabling IDE-based AI assistants to orchestrate evaluations conversationally. The MCP server supports three transport modes (stdio for local IDE integration, HTTP for cluster deployment, and HTTP+SSE for legacy compatibility) and includes a 30-second completion cache for responsive auto-complete. The EDD (Explore-Define-Develop) workflow prompt is particularly interesting — it provides structured multi-phase evaluation guidance tailored to application types (RAG, agents, safety, classifiers).

Multi-tenancy is implemented at the data layer rather than the API gateway layer. The eval-hub API extracts tenant identity from X-Tenant and X-User headers (injected by kube-rbac-proxy) and scopes all database queries to the tenant's namespace. System-managed resources (bundled providers and collections) are visible to all tenants but not modifiable; user-created resources are tenant-isolated. This approach relies entirely on the kube-rbac-proxy sidecar for authentication — the eval-hub service itself performs no token validation, making the kube-rbac-proxy a critical security dependency. In local mode (the `-local` flag), identity headers are not required, which is convenient for development but must never be used in production.

The single-image, multi-binary deployment model deserves attention. All four Go binaries (eval-hub, eval-runtime-sidecar, eval-runtime-init, evalhub-mcp) are compiled into a single container image, with the specific binary selected via the container's command override. This simplifies image management and ensures all components share the same version, but it also means the image contains binaries that are never used in a given pod (e.g., the main eval-hub pod ships the sidecar and init binaries it doesn't need). The Python wheel distributions (`eval-hub-server`, `eval-hub-mcp`) are thin wrappers that bundle the precompiled Go binary for cross-platform pip installation, and the Homebrew formula provides macOS/Linux distribution for the MCP server specifically.
