## Purpose

**Short**: KubeRay is a Kubernetes operator that manages the lifecycle of Ray clusters, jobs, and services on Kubernetes and OpenShift.

**Detailed**: KubeRay manages three Custom Resources — RayCluster, RayJob, and RayService — to automate the provisioning, scaling, and lifecycle management of Ray distributed computing clusters. The operator creates and manages head and worker pods, services, ingress resources, and autoscaler configurations. It communicates with the Ray Dashboard API to submit jobs, deploy serve applications, and monitor cluster health.

On RHOAI (Red Hat OpenShift AI), the operator is significantly enhanced with security-hardened features: a mutating webhook enforces secure-trusted-network annotations on all RayClusters, an authentication controller creates HTTPRoutes (Gateway API) and kube-rbac-proxy sidecars for OIDC-based access control, an mTLS controller provisions cert-manager certificates for inter-pod encryption, and a NetworkPolicy controller creates restrictive network policies that limit traffic to intra-cluster communication and authorized namespaces. These controllers work together to create a defense-in-depth security posture for Ray workloads.

The operator integrates with batch scheduling systems (Volcano, YuniKorn, scheduler-plugins) for gang scheduling of distributed workloads, and supports GCS fault tolerance via external Redis for head node recovery. For RHOAI 3.x, the operator follows the Gateway API ingress pattern, creating HTTPRoutes in the platform namespace with ReferenceGrants for cross-namespace service access, replacing the legacy OAuth-proxy + Route pattern.

## Architectural Analysis

The KubeRay operator follows a multi-controller architecture where specialized controllers handle orthogonal concerns for each RayCluster. The main RayCluster controller handles pod and service lifecycle, while the Authentication, mTLS, and NetworkPolicy controllers run independently and communicate via status conditions (e.g., `AuthenticationReady`) and annotations (e.g., `odh.ray.io/secure-trusted-network`). This separation of concerns allows security features to be toggled independently — the `secure-trusted-network` annotation acts as the master switch for all three security controllers.

The RHOAI-specific security architecture implements a defense-in-depth pattern. The mutating webhook guarantees that every RayCluster on OpenShift gets the secure-trusted-network annotation, which in turn activates: (1) the Authentication controller to create HTTPRoutes and kube-rbac-proxy sidecars for OIDC authentication, (2) the mTLS controller to provision cert-manager certificates for inter-pod encryption, and (3) the NetworkPolicy controller to restrict network access to authorized namespaces only. The webhook also forcibly disables traditional Ingress/Route creation (`enableIngress=false`), ensuring all dashboard access flows through the Gateway API path with authentication.

The HTTPRoute placement strategy in RHOAI 3.x is noteworthy: HTTPRoutes are created in the platform namespace (e.g., `redhat-ods-applications`) rather than the cluster's namespace, with ReferenceGrants enabling cross-namespace service references. This centralizes ingress configuration in a single namespace and works with the platform's shared Gateway (data-science-gateway). The routing pattern uses path-based matching (`/ray/{namespace}/{cluster-name}/...`) with URL rewriting to strip the prefix before forwarding to the Ray Dashboard. The authentication controller includes migration logic to clean up old HTTPRoutes from previous implementations where they were placed in the cluster namespace.

The RayService controller implements a sophisticated zero-downtime upgrade strategy using an active/pending cluster pattern. When the RayService spec changes, the controller creates a pending RayCluster alongside the active one, deploys the serve configuration to it, verifies health via proxy actor health checks, then atomically switches the Kubernetes Service selectors to point to the pending cluster. An LRU cache prevents redundant serve configuration deployments, and a 60-second deletion delay provides a grace window before removing old clusters.

A significant architectural decision is the pluggable HTTP client pattern for Ray Dashboard communication. The `RayDashboardClientInterface` supports two modes: direct HTTP connections to the head service (default) and Kubernetes API proxy mode for environments where network policies block pod-to-pod traffic. The factory pattern enables test mocking and runtime mode selection via the `useKubernetesProxy` configuration flag. HTTP response bodies are limited to 4 MiB to prevent denial-of-service from large responses.
