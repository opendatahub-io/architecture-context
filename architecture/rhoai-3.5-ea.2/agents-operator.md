## Purpose

**Short**: Kubernetes operator that automates the deployment, discovery, security, and observability of AI agents in RHOAI clusters.

**Detailed**: The Kagenti Operator manages the lifecycle of AI agents running on Kubernetes and OpenShift. It provides three custom resource definitions (AgentRuntime, AgentCard, AuthorizationPolicy) that declaratively enroll workloads into the Kagenti platform. When a user creates an AgentRuntime CR pointing to a Deployment or StatefulSet, the operator applies labels, computes a configuration hash, and triggers sidecar injection via a mutating admission webhook. The injected AuthBridge sidecar provides workload identity (SPIFFE/SPIRE), mutual authentication (mTLS), token exchange (OAuth2/Keycloak), and protocol-aware request inspection (A2A, MCP, inference parsing).

The operator also implements an agent discovery system through AgentCard CRs that fetch, verify (JWS and Sigstore), and cache A2A protocol agent cards from running workloads. Signature verification results drive NetworkPolicy enforcement (permissive for verified agents, restrictive for unverified). Additional controllers manage Keycloak client registration, MLflow experiment tracking, cert-manager TLS certificates for the TLS bridge feature, Istio ambient mesh enrollment, shared trust distribution across Istio namespaces, Kuadrant rate limiting operand reconciliation, and Tekton pipeline configuration. A dedicated bundle-service deployment serves OPA/Rego authorization policy bundles to AuthBridge sidecars.

The repository also contains the AuthBridge proxy sidecar binary (`authbridge-proxy`), which operates as an HTTP forward+reverse proxy with a plugin pipeline for JWT validation, token exchange, OPA policy enforcement, and protocol parsing. It supports three deployment modes: proxy-sidecar (default, HTTP_PROXY-based), envoy-sidecar (Envoy + ext_proc), and lite (minimal JWT-only).

## Architectural Analysis

The Kagenti Operator implements a sophisticated multi-layered agent lifecycle management system designed for zero-trust Kubernetes environments. The architecture follows a controller-runtime operator pattern with 11+ reconcilers, 3 webhooks, and 3 bootstrap runnables, making it one of the more complex operators in the RHOAI platform despite being a relatively new component (v0.2.0-alpha).

**Sidecar injection model**: The operator uses a mutating admission webhook to inject AuthBridge sidecar containers into agent pods. This is architecturally distinct from the RHOAI platform's Gateway API + kube-rbac-proxy pattern used by other components. Instead of per-component HTTPRoutes and auth proxies, Kagenti injects a full authentication/authorization proxy directly into the agent pod. The sidecar supports four deployment modes (proxy-sidecar, envoy-sidecar, lite, waypoint), each with different tradeoffs between capability and resource consumption. The proxy-sidecar mode (default) uses HTTP_PROXY environment variable injection, which means agent applications transparently route outbound traffic through the AuthBridge pipeline without code changes.

**Identity and trust chain**: The identity architecture is layered: SPIFFE/SPIRE provides workload identity at the infrastructure level (X.509 SVIDs for mTLS, JWT SVIDs for token exchange), while Keycloak provides application-level OAuth2/OIDC identity. The AgentCard verification system adds a third layer: A2A protocol agent cards are JWS-signed using SPIRE SVIDs, and the operator verifies these signatures against the trust bundle. A Sigstore layer provides supply-chain verification of SignedAgentCard bundles. Identity binding enforces that the SPIFFE ID in a card's signature matches the expected trust domain. This multi-layer identity model drives NetworkPolicy enforcement: verified agents get permissive policies (access to other verified agents + external APIs), unverified agents get restrictive policies (Kubernetes API only).

**Configuration precedence**: The operator implements a 4-layer configuration precedence chain: platform defaults (Helm values) < cluster-level ConfigMaps < namespace-level ConfigMaps/labels < per-workload AgentRuntime CRs. Configuration changes at any layer trigger a deterministic SHA256 hash recomputation, and hash changes trigger workload rolling updates to pick up new sidecar configuration. Feature gates provide a global kill switch and per-feature toggles.

**Bootstrap complexity**: The operator bootstraps significant infrastructure at startup: PostgreSQL StatefulSet + Service, Keycloak CR + Route + realm import with test users, OTel collector ConfigMap assembly with component auto-discovery (MLflow, Phoenix). This bootstrap is leader-election-gated and runs as manager Runnables. The Keycloak bootstrap includes polling for Route host assignment with retry logic, reflecting the real-world timing challenges of OpenShift router host propagation.

**CRD interaction breadth**: The operator watches or manages CRDs from 9 different API groups beyond its own (cert-manager.io, k8s.keycloak.org, datasciencecluster.opendatahub.io, mlflow.opendatahub.io, mlflow.kubeflow.org, kuadrant.io, operator.tekton.dev, operator.openshift.io, agents.x-k8s.io), with conditional enablement via CRD existence checks at startup. This makes the operator resilient to missing optional components but also creates a large RBAC surface area.
