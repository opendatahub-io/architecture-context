## Purpose

**Short**: A FIPS-compliant reverse proxy that provides OIDC and OpenShift OAuth authentication for upstream services, designed as a drop-in replacement for oauth-proxy in ODH/RHOAI environments.

**Detailed**: kube-auth-proxy is a focused authentication proxy derived from the oauth2-proxy project, streamlined to support only OIDC and OpenShift OAuth providers. It is deployed as a sidecar container alongside RHOAI platform components to enforce authentication before requests reach upstream application services. The proxy handles the complete OAuth2/OIDC authentication flow — login redirects, token exchange, session management, and token validation — while forwarding authenticated requests with identity headers to the upstream service.

The proxy supports three authentication methods that can operate simultaneously: (1) browser-based OAuth2/OIDC flows via cookies, (2) bearer token authentication for CLI/API access (including OpenShift `sha256~` tokens and JWT ID tokens), and (3) Kubernetes service account token validation via the TokenReview API. This multi-method approach enables both human users (via browser) and service accounts (via bearer tokens) to authenticate through the same proxy instance.

kube-auth-proxy is specifically built for FIPS compliance using `GOEXPERIMENT=strictfipsruntime` and `CGO_ENABLED=1`, producing dynamically-linked Go binaries that use OpenSSL for cryptographic operations. It is designed to pass the `check-payload` FIPS validation required by Konflux builds. The proxy also integrates with Envoy's `ext_authz` external authorization framework, returning 200/OK for authenticated requests in auth-only mode, which is critical for the RHOAI 3.x Gateway API ingress pattern.

## Architectural Analysis

kube-auth-proxy is a strategically important component in the RHOAI 3.x authentication architecture. It serves as the successor to the upstream `oauth-proxy` sidecar, with a narrowed scope (OIDC and OpenShift OAuth only) and explicit FIPS compliance. The project was created as a disconnected fork of `oauth2-proxy/oauth2-proxy`, removing all non-essential providers (Google, GitHub, Azure, Facebook, etc.) to reduce the attack surface and simplify maintenance. This design decision trades breadth of provider support for depth of compliance and supportability.

The authentication middleware chain demonstrates a well-structured layered architecture. The `buildSessionChain` function in `oauthproxy.go` (lines 425-468) constructs a chain where K8s TokenReview runs first, followed by provider-specific token validation (JWT for OIDC, OAuth for OpenShift), basic auth, and finally stored session loading from cookies. Each layer passes to the next if it doesn't handle the request, enabling graceful fallback across authentication methods. This design is critical for RHOAI, where the same proxy instance must handle both human browser sessions and programmatic service account access.

The Envoy `ext_authz` integration (the `/oauth2/auth` endpoint) is central to RHOAI 3.x's Gateway API ingress pattern. When used with Envoy, the proxy operates in auth-only mode — it validates the session and returns 200 with forwarded identity headers (allowing Envoy to proceed) or 401/403 (blocking the request). The `redirectOnCSRFError` mechanism (enabled by default, lines 620-639) addresses a specific ext_authz edge case: when CSRF cookies expire during the OAuth callback, Envoy would forward the 403 error page body directly to the client instead of rendering it. The redirect to sign-in with a session-expired message provides a better user experience in this scenario. The custom MLflow handler (`pkg/authdeny/mlflow.go`) further demonstrates protocol-aware error handling — MLflow SDK clients receive structured JSON error responses with actionable guidance for `oc login`, instead of HTML error pages.

The OpenShift provider implementation (`providers/openshift.go`) shows careful handling of the OpenShift-specific authentication lifecycle. It supports on-demand OAuth endpoint discovery from the Kubernetes API server's well-known endpoint (with mutex-protected caching), custom CA certificate chains for secure API communication, and sign-out behavior that mirrors the OpenShift console — including background deletion of `OAuthAccessToken` resources and RP-Initiated Logout via the IdP's `end_session_endpoint`. The `sha256~` token format handling in `tokenToObjectName` (line 909-913) correctly computes the double-SHA256 hash needed to derive the OAuthAccessToken resource name from a bearer token, matching the OpenShift console implementation.

A notable architectural risk is that the TLS configuration allows `InsecureSkipVerify` via the `--ssl-insecure-skip-verify` flag, which, if enabled, would bypass certificate validation for all provider connections. While this is marked with `#nosec G402` annotations and is disabled by default, it represents a potential FIPS compliance gap if misconfigured in production. The codebase correctly enforces `MinVersion: tls.VersionTLS12` on both the server-side TLS listener (`pkg/http/server.go:138`) and the OpenShift client TLS configuration (`providers/openshift.go:457`), ensuring no connections below TLS 1.2 are accepted.
