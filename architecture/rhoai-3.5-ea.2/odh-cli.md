## Purpose

**Short**: CLI tool for inspecting, diagnosing, linting, and migrating Red Hat OpenShift AI (RHOAI) clusters across version upgrades.

**Detailed**: odh-cli (also packaged as `rhai-cli` and `kubectl-odh`) is a Kubernetes-native command-line tool designed to support RHOAI cluster operators through the full lifecycle of platform management. The tool connects to live Kubernetes clusters to inspect platform health, validate component configurations, assess upgrade readiness, and execute data migrations between RHOAI versions.

The CLI operates in two primary modes: **diagnostic mode** (lint, status, components, deps, events, logs, get) for understanding current cluster state, and **migration mode** (migrate prepare/run) for executing versioned migration actions that transform cluster resources during upgrades. Migration actions cover all major RHOAI components including AI Pipelines, Model Serving (KServe), Kueue, TrustyAI, Workbenches (Notebooks), RayCluster, LlamaStack, and Training Operator workloads.

The tool is distributed as a standalone Go binary, a kubectl plugin (`kubectl odh`), and a container image (`quay.io/rhoai/odh-cli-rhel9`) that bundles additional utilities (oc, jq, yq, Python with Kubernetes SDK) for use in CI/CD pipelines and automated upgrade workflows. It integrates with the Operator Lifecycle Manager (OLM) for dependency management and uses the `clusterhealth` library from the opendatahub-operator for platform health assessment.

## Architectural Analysis

The odh-cli follows a **command-pattern architecture** with a consistent lifecycle across all commands: `AddFlags() → Complete() → Validate() → Run(ctx)`. Each command is a self-contained unit with its own cobra subcommand, flag registration, validation logic, and execution function. This pattern enables clean separation between CLI argument parsing (in `cmd/`) and business logic (in `pkg/`), making the codebase highly testable and extensible.

The **lint subsystem** is the most architecturally sophisticated part of the CLI. It implements a registry-based check system with 38+ checks organized into five groups (dependencies, services, platform, components, workloads). Each check uses a builder pattern (e.g., `WorkloadBuilder`, `ComponentBuilder`, `DSCBuilder`) that provides fluent APIs for common validation scenarios. The check results follow a Kubernetes CR-like pattern with `DiagnosticResult` types carrying conditions, severity levels (critical/warning/info), and structured remediation guidance. Checks are version-aware — they self-select based on the `--target-version` flag, enabling the same CLI binary to validate upgrades across multiple RHOAI releases.

The **migration framework** uses a two-phase execution model: `prepare` (backup resources, validate prerequisites) and `run` (execute transformations). Each migration action registers itself with an `ActionRegistry` and declares its version applicability, phase (pre-upgrade, post-upgrade, pre-enablement), and group (migration, backup, validation). The framework supports sequential execution with failure-stops, dry-run mode, and a hierarchical step recorder that streams progress to the terminal in real-time. This design allows the CLI to orchestrate complex multi-step migrations (e.g., ModelMesh-to-RawDeployment conversion requires listing ISVCs, updating ServingRuntimes, patching ISVCs, and creating auth resources) while giving operators clear visibility into what's happening.

The CLI's **Kubernetes client abstraction** (`pkg/util/client/`) wraps multiple client types (dynamic, metadata, discovery, API extensions, OLM, core, authorization) behind a unified interface. This design supports both typed operations (OLM Subscriptions, CSVs) and unstructured operations (arbitrary CRDs discovered at runtime). The client includes a SelfSubjectAccessReview-based permission pre-check that allows the CLI to gracefully degrade when the authenticated user lacks permissions for certain operations, rather than failing with opaque RBAC errors. This is particularly important since the CLI is designed to be run by cluster operators who may have varying permission levels.

A notable architectural decision is the **embedded dependency manifest**: the CLI fetches the Helm chart `values.yaml` from `odh-gitops` at build time (pinned to a specific commit SHA) and embeds it via Go's `embed` package. This allows the `deps` command to know the expected dependency graph for each RHOAI version without requiring network access at runtime. The pinned commit is tracked in the Makefile as `ODH_GITOPS_COMMIT`, ensuring reproducible builds and making dependency data changes auditable via git history.

The container image architecture bundles multiple tools beyond the primary Go binary: `oc` (OpenShift CLI, from the base image `ose-cli-rhel9`), `yq` (YAML processor, built from vendored source with FIPS flags), `jq` (installed via RPM), and Python with the Kubernetes SDK (for legacy migration scripts in `rhoai-upgrade-helpers/`). The legacy scripts are being progressively replaced by native Go migration actions — as evidenced by the recent commits converting AI Pipelines and RayCluster migration scripts to Go. The container image supports three architectures (x86_64, arm64, ppc64le) via Konflux multi-arch builds.
