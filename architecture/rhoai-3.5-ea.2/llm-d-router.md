## Purpose

**Short**: Intelligent inference request router that performs LLM-aware endpoint selection via the Envoy ext-proc protocol, with a companion sidecar for prefill/decode disaggregation.

**Detailed**: The llm-d Router is the intelligent entry point for inference traffic in the RHOAI platform. It delivers LLM load-aware and prefix-cache-aware routing, request prioritization, and advanced flow control across diverse request formats (OpenAI, Anthropic, Vertex AI, vLLM). The core component is the Endpoint Picker (EPP), a gRPC server that integrates with Envoy via the ext-proc protocol to inject real-time routing decisions into the data plane. The EPP uses a pluggable scheduling framework with filters, scorers, and pickers to evaluate endpoints based on KV-cache locality, queue depth, running requests, LoRA adapter affinity, and other signals.

The repository also ships a Disaggregation Sidecar (pd-sidecar) deployed alongside decode model servers. The sidecar orchestrates multi-stage inference lifecycles -- Prefill/Decode (P/D), Encode/Prefill/Decode (E/P/D), and data-parallel configurations -- by coordinating with specialized prefill and encode workers via multiple KV transfer protocols (NIXLv2, shared storage, SGLang, Mooncake). The router supports two deployment modes: Standalone Mode (self-managed Envoy sidecar in the same pod) and Gateway Mode (leveraging Kubernetes Gateway API with an external gateway).

## Sub-Component Details

### Endpoint Picker (EPP)

**Dockerfile**: `Dockerfile.konflux.epp`
**Intent**: The primary service of this repository. The EPP is the "brain" of the router -- it runs as a gRPC ext-proc server that Envoy consults for every inference request. It watches Kubernetes resources (InferencePool, Pods, InferenceObjective, InferenceModelRewrite) to maintain a live view of available endpoints and scheduling objectives, then uses a pluggable filter/scorer/picker pipeline to select the optimal endpoint for each request. Deployed as a Deployment or StatefulSet alongside an Envoy proxy (sidecar or service mode).
**Image**: odh-llm-d-inference-scheduler
**Language**: Go
**Ports**: 9002/TCP (gRPC ext-proc), 9003/TCP (gRPC health), 9090/TCP (Prometheus metrics), 5557/TCP (KV-Events ZMQ SUB)

### Disaggregation Sidecar (pd-sidecar)

**Dockerfile**: `Dockerfile.konflux.sidecar`
**Intent**: A sidecar module that runs alongside the decode model server pod. It intercepts OpenAI-compatible inference requests and orchestrates multi-stage inference by routing between prefill, encode, and decode workers. It transforms request bodies (injecting kv_transfer_params, manipulating token limits, managing cache hit thresholds) and coordinates KV cache transfers via pluggable connector protocols. Deployed only when disaggregated inference (P/D or E/P/D) is enabled.
**Image**: odh-llm-d-routing-sidecar
**Language**: Go
**Ports**: configurable (default 8000/TCP)

The EPP's default `InsecureSkipVerify=true` for model server connections is a security concern in production environments but does not affect FIPS compliance per se._

## Architectural Analysis

The llm-d Router implements a sophisticated, modular plugin framework for LLM inference routing. The architecture follows a clear separation between the control plane (K8s reconcilers that maintain endpoint state) and the data plane (ext-proc gRPC server that makes real-time routing decisions). The EPP's plugin system provides five major extension points: data layer (endpoint metrics collection), request control (admission, data production, pre/post hooks), request handling (protocol parsing), scheduling (filter/scorer/picker), and flow control (priority-based queuing with fairness policies). This design allows operators to compose custom routing strategies from 50+ built-in plugins without modifying core code.

The scheduling engine uses a multi-profile architecture where a ProfileHandler selects which scheduling profiles to execute (e.g., separate prefill vs. decode profiles for disaggregated inference). Each profile runs a pipeline of filters (ByLabel, PrefixCacheAffinity, SessionAffinity, SLOHeadroomTier), weighted scorers (LoadAware, KVCacheUtilization, PrecisePrefixCache, LoRAffinity, QueueDepth, etc.), and a picker (MaxScore, WeightedRandom, Random). Scores are categorized as Affinity, Distribution, or Balance, enabling different weighting strategies. The flow control layer adds priority-based queuing with configurable fairness policies (global strict, program-aware, round-robin), ordering policies (FCFS, EDF, SLO deadline), and saturation detection (concurrency-based or utilization-based).

The disaggregation sidecar is architecturally independent from the EPP -- it shares only common utility packages and the Go module. The sidecar implements four KV transfer connector protocols (NIXLv2, shared storage, SGLang, Mooncake) and two encoder connector protocols (ec-nixl, ec-example), each with distinct coordination patterns. NIXLv2 supports both serial (prefill-then-decode) and parallel (MoRI-IO WRITE) modes. The sidecar's "chunked decode" feature splits large context windows into multiple decode passes, enabling processing of prompts that exceed single-pass limits. This is controlled by request body transformations (injecting `kv_transfer_params`, manipulating token limits, managing `cache_hit_threshold`) rather than protocol-level changes, maintaining transparent API compatibility.

A notable security consideration is that the EPP's model server HTTP clients default to `InsecureSkipVerify=true` for both metrics and models polling. This is configurable but the default leaves model server connections without certificate verification. Additionally, the EPP's Konflux Dockerfile does not include the `-tags strictfipsruntime` build tag that the sidecar uses, creating an asymmetry in FIPS compliance between the two components. The sidecar's TLS configuration is more hardened, with explicit TLS 1.2 minimum version and FIPS-compatible cipher suites (ECDHE with AES-128/256-GCM and CHACHA20-POLY1305).

The repository ships two Helm charts (gateway and standalone) built on a shared library chart (routerlib). The gateway chart integrates with external Gateway API implementations (Istio, GKE), while the standalone chart embeds an Envoy proxy as either a sidecar or a separate service. Both charts support the full EPP configuration via ConfigMap with the EndpointPickerConfig API type. The deployment manifests in `deploy/` use shell-variable-based templating (`${EPP_NAME}`, `${NAMESPACE}`) rather than kustomize, suggesting they are consumed by deployment scripts rather than directly by kustomize.
