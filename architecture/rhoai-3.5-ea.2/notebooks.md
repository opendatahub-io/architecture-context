## Purpose

**Short**: Builds and publishes container images for Jupyter notebook workbenches, code-server IDE workbenches, and Elyra pipeline runtime environments used by data scientists on the Red Hat OpenShift AI platform.

**Detailed**: The notebooks repository is the central build system for all interactive workbench and pipeline runtime container images in RHOAI. It produces 18 distinct container images spanning three accelerator variants (CPU, NVIDIA CUDA, AMD ROCm) across two categories: workbench images (JupyterLab and VS Code Server environments where data scientists write and execute code interactively) and runtime images (lightweight execution environments used by Elyra to run notebook-based pipelines on Kubeflow Pipelines).

The repository does not contain application source code in the traditional sense — its primary artifacts are Dockerfiles, Python dependency lock files, kustomize manifests (OpenShift ImageStream definitions), and build/CI tooling. Each Dockerfile assembles a complete data science environment by layering Python packages (via `uv pip install` from pre-fetched Cachi2 caches), system libraries (via `dnf`), and optional GPU runtime libraries onto UBI9 or CentOS Stream 9 base images. The build system enforces hermetic, reproducible builds: all dependencies are pre-resolved and locked, and no network access occurs during container image builds.

The manifests directory contains OpenShift ImageStream definitions that the platform operator (`rhods-operator`) consumes to present available notebook images to users through the RHOAI dashboard. Each ImageStream carries four version tags (N, N-1, N-2, N-3) with annotations describing included software versions, enabling the dashboard to display human-readable descriptions and version history.

## Architectural Analysis

The notebooks repository exemplifies a "Dockerfile-as-product" pattern where the primary engineering output is not application code but carefully crafted container build definitions. Each of the 16 Konflux Dockerfiles encodes complex dependency chains spanning four layers: OS packages (RPMs from UBI9/CentOS Stream 9), accelerator libraries (CUDA toolkit, ROCm runtime), Python packages (200+ scientific computing and ML libraries), and application configuration (JupyterLab settings, code-server extensions). The challenge is maintaining reproducibility, security compliance, and accelerator compatibility across all permutations.

The hermetic build strategy is particularly sophisticated. Every Dockerfile uses `uv pip install --no-index --find-links /cachi2/output/deps/pip` to install Python packages exclusively from pre-fetched wheels — no network access during builds. RPM installations also use pre-fetched repositories. This is enforced by Konflux/Cachi2 at the build pipeline level, making builds reproducible regardless of upstream package index availability. The `pylock.*.toml` files (uv's newer lock format with `UV_PREVIEW_FEATURES=pylock`) provide a deterministic resolution record that supersedes the older `requirements.txt` files.

The accelerator matrix creates a combinatorial explosion that the repository manages through shared Dockerfile patterns and a `versions_config.yml` configuration file. CPU, CUDA 12.9, CUDA 13.0, and ROCm 7.1 variants share the same application-layer Dockerfile structure but diverge at the base image and GPU library levels. The ROCm variants require additional post-processing — `de-vendor-torch.py` removes PyTorch-bundled AMD libraries (to avoid conflicts with system ROCm), and `link-solibs.py` creates symlinks for ROCm shared objects. These ROCm-specific workarounds indicate that ROCm integration is less mature than CUDA.

The code-server image is architecturally distinct from the Jupyter images. Its 4-stage build compiles code-server from source (Node.js 22 + gcc-toolset-14), installs nginx as a reverse proxy, httpd for CGI-based kernel access tracking, and bundles VS Code extensions. The code-server process binds to port 8787 with `--auth none` — authentication is deliberately delegated to the platform's kube-rbac-proxy sidecar, which the odh-notebook-controller injects into every workbench pod. This separation of concerns (application serves plain HTTP, platform handles TLS and auth) is consistent across all workbench types.

The CVE constraint system (`dependencies/cve-constraints.txt`) is a notable supply chain security pattern. Rather than patching individual lock files when a CVE is disclosed, the team adds a minimum version constraint to this centralized file, then regenerates all lock files with `make refresh-lock-files`. This ensures CVE fixes propagate to all 18 images simultaneously. The current constraints cover 6 CVEs across urllib3, keras, pillow, jupyterlab, nltk, and starlette — indicating active security monitoring.

The manifest system uses a ConfigMap-based image reference injection pattern rather than hardcoding container image digests into ImageStream definitions. The kustomize `replacements` feature (40+ blocks) substitutes placeholder strings like `odh-workbench-jupyter-minimal-cpu-py312-ubi9-n_PLACEHOLDER` with actual image references from `params.env` files. The platform operator (rhods-operator) further patches these with `RELATED_IMAGE_*` environment variables from the operator CSV, providing a final layer of indirection that enables disconnected/air-gapped deployments.
