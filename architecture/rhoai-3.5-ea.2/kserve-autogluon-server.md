## Purpose

**Short**: A KServe model server that serves AutoGluon TabularPredictor and TimeSeriesPredictor models via REST v1/v2 inference protocols.

**Detailed**: The KServe AutoGluon Server is a Python-based model serving runtime that enables deployment of AutoGluon machine learning models on KServe. It supports two predictor types: TabularPredictor (for classification and regression on structured/tabular data) and TimeSeriesPredictor (for time series forecasting). The server auto-detects the predictor type at startup by attempting to load the model artifact as each type, requiring no explicit configuration from users.

For tabular models, the server supports both KServe REST v1 (JSON) and v2 (tensor) inference protocols, with optional class probability output via `predict_proba()`. For time series models, only REST v1 JSON is supported, with forecasts returned including mean and quantile predictions. The server handles model storage download (S3, GCS, Azure Blob, local filesystem) via the KServe storage module.

This component is built from the broader kserve repository but ships as a standalone container image (`odh-kserve-autogluon-server-rhel9`) specifically for the AutoGluon serving runtime. It is deployed as a KServe `ClusterServingRuntime` or `ServingRuntime` with `modelFormat.name: autogluon` in `InferenceService` specs.

## Architectural Analysis

The AutoGluon server follows the standard KServe serving runtime pattern: it implements the `kserve.Model` interface and delegates to the KServe SDK's `ModelServer` for HTTP/gRPC protocol handling, health checks, and metrics. The architecture is notable for its auto-detection mechanism -- at startup, the server attempts to load the model artifact as a `TimeSeriesPredictor` first, falling back to `TabularPredictor` if that fails. This eliminates the need for users to specify the predictor type in their `InferenceService` YAML, simplifying deployment.

The codebase demonstrates a clean separation between protocol handling and model execution. The `AutoGluonDetectedModel` class acts as a facade that delegates to either `AutoGluonTabularModel` or `AutoGluonTimeSeriesModel` after detection. The tabular model supports both v1 (JSON dict) and v2 (tensor-based `InferRequest`) protocols, with careful handling of feature name mapping from tensor inputs to pandas DataFrames. The time series model explicitly rejects v2 tensor requests, supporting only v1 JSON -- this is a deliberate design choice given that time series data (variable-length history, known covariates, id/timestamp columns) does not map cleanly to fixed-shape tensors.

A key design element is version compatibility handling via `version_compat.py`. AutoGluon saves the training-time library version alongside model artifacts; the server allows patch-level mismatches (e.g., 1.5.0 vs 1.5.0+rhaiv.3) but rejects major/minor version differences. This is critical because the RHOAI build uses Red Hat-patched AutoGluon versions (`+rhaiv.3` local version label) that would otherwise fail strict version checks. The `runtime_paths.py` module addresses container runtime constraints -- AutoGluon's `SeasonalNaive` fallback attempts to create directories under the current working directory, which fails in read-only or root-owned container filesystems. The module ensures writable paths exist before model loading begins.

The build pipeline uses the AIPCC CPU base image with Python packages sourced exclusively from the RHEL AI Python Package Index (`console.redhat.com/api/pypi/public-rhai/rhoai/3.5-EA2/cpu-ubi9/simple/`). All transitive dependencies are locked with SHA256 hashes in `autogluon-all-requirements.txt`, providing hermetic Python dependency resolution. The image carries a substantial dependency footprint (~100 packages) due to AutoGluon's ML backend requirements (PyTorch, LightGBM, XGBoost, scikit-learn, spaCy, transformers, etc.), which impacts image size and attack surface.
