## Purpose

**Short**: A containerized distribution of OGX (the opendatahub-io fork of Llama Stack) providing a multi-provider AI gateway for inference, vector storage, file processing, and agentic APIs.

**Detailed**: The OGX Distribution packages the Open Data Hub fork of OGX (originally Llama Stack) into a production-ready container image that serves as a unified API gateway for AI/ML workloads. The image ships pre-configured providers for multiple inference backends (vLLM, OpenAI, Anthropic, AWS Bedrock, Azure, Vertex AI, WatsonX, Gemini), vector databases (Milvus, pgvector, Qdrant, FAISS), file processing (Docling, MarkItDown, pypdf), and tool runtimes (Brave Search, Tavily Search, MCP, file search). Providers activate dynamically at runtime based on environment variables, allowing a single image to serve diverse deployment scenarios.

The repository does not contain application source code — it is a distribution and packaging project. The upstream OGX framework (from `opendatahub-io/ogx`) provides the server runtime, API implementations, and provider integrations. This repository's role is to define the provider configuration (`build/build.yaml`), generate deployment artifacts (`distribution/config.yaml`, `distribution/requirements.txt`, `Containerfile`), build the container image via Konflux, and maintain CI/CD pipelines for testing across multiple inference providers and architectures (amd64, arm64, ppc64le). The server exposes OpenAI-compatible, Anthropic Messages-compatible, and Llama Stack-native REST APIs on port 8321, with PostgreSQL as the backing store for state persistence, inference logging, and batch processing. Optional OAuth2/OIDC authentication is supported via JWKS token validation with configurable access policies.

## Architectural Analysis

The OGX Distribution follows a "distribution packaging" pattern — the repository contains no application source code, only configuration, build tooling, and CI/CD pipelines. The upstream OGX framework (forked at `opendatahub-io/ogx`) provides the entire runtime. This separation means the distribution repository's primary value is in its `build/build.yaml` provider configuration, which defines the universe of available AI providers and their environment-variable-driven activation pattern.

The provider activation pattern (`${env.VAR:+provider-name}`) is particularly elegant: a single `config.yaml` works for all deployment scenarios because providers are conditionally activated based on whether their trigger environment variable is set. This avoids the need for separate images or config files per deployment topology. However, this also means the container image ships with all provider dependencies (including heavy libraries like torch, sentence-transformers, milvus-lite, faiss-cpu) regardless of which providers will actually be used at runtime, resulting in a large image size.

The build pipeline has a notable code generation chain: `build/build.yaml` (source of truth) → `build/build.py` generates `distribution/config.yaml` + `distribution/requirements.txt` + `Containerfile` → `scripts/sync_labels.py` generates `Dockerfile.konflux` from `Dockerfile.konflux.in`. The `config.yaml` is embedded as a base64-encoded OCI label in the container image, allowing runtime tools to inspect the distribution's configuration without starting the container. Some provider types (`inline::sentence-transformers`, `inline::milvus`, `inline::faiss`, `inline::docling`, `inline::markitdown`, `inline::pypdf`, `remote::docling-serve`) are "stripped" from the runtime config but their Python dependencies remain installed, enabling users to activate them via custom config files.

The non-hermetic build (`hermetic: false` in the Tekton pipeline) combined with the lack of version-pinned requirements represents a supply chain risk. Most packages in `requirements.txt` are unpinned (e.g., just `anthropic`, `boto3`, `httpx` without version constraints), meaning each build may resolve different versions depending on the RHEL AI Python Package Index state. The `distribution/constraints.txt` file exists but is currently empty, providing no guardrails. This is partially mitigated by the AIPCC base image providing a curated package index, but the risk of non-reproducible builds remains.

The OGX server listens on plaintext HTTP (port 8321) with no built-in TLS. In RHOAI deployments, TLS termination is handled by the platform's ingress infrastructure (Gateway API + Envoy proxy). The optional OAuth2/OIDC authentication layer validates JWT tokens via JWKS but delegates TLS verification to the upstream connection. The access policy model (unowned resources readable by all, creation requires authentication, CRUD restricted to resource owners) provides basic multi-tenancy without namespace-level isolation.
