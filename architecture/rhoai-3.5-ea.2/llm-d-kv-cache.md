## Purpose

**Short**: A multi-component KV cache indexing and management system that enables prefix-cache-aware scheduling for distributed LLM inference engines (vLLM, SGLang).

**Detailed**: llm-d-kv-cache provides the infrastructure for tracking, scoring, and managing KV cache blocks across distributed LLM serving pods. The system consists of four major sub-components: (1) a Go library that implements a KV block index (backed by in-memory LRU, cost-aware Ristretto, or Redis/Valkey), a longest-prefix-match scorer, and a ZMQ-based event processing pipeline for real-time cache state synchronization; (2) a Python gRPC UDS Tokenizer sidecar service that performs tokenization and chat template rendering using HuggingFace/vLLM toolchains; (3) a Python vLLM filesystem connector (llmd_fs_backend) that offloads KV cache blocks between GPU memory and shared storage via a C++ CUDA extension; and (4) a PVC Evictor daemon that manages cache disk pressure using an N+2 multiprocessing architecture.

The Go library is consumed as a dependency by the llm-d routing gateway (llm-d-inference-scheduler), where it scores incoming requests against cached KV blocks to route them to pods with the highest prefix cache hit ratio. The tokenizer sidecar runs alongside the indexer as a sidecar container communicating over a Unix Domain Socket. The filesystem connector runs as a vLLM plugin inside vLLM inference pods. The PVC evictor runs as a standalone Kubernetes deployment alongside the shared storage PVC.

## Architectural Analysis

The llm-d-kv-cache system follows a **distributed event-sourced indexing** pattern. Rather than maintaining a centralized cache directory, each vLLM inference pod publishes KV cache block lifecycle events (stored, removed, cleared) over ZMQ PUB/SUB. The indexer subscribes to these event streams and maintains a local or distributed index of which pods hold which KV blocks. This design decouples the inference engine from the scheduling layer — vLLM pods are unaware that their cache events are being consumed for routing decisions.

The **token-to-block-key hashing** algorithm is architecturally critical. It uses FNV-64a over CBOR-serialized token chunks (default 16 tokens per block), chaining each block's hash as the parent of the next. This creates a deterministic prefix tree where any prompt sharing a common prefix will produce identical block keys up to the divergence point. The scoring algorithm exploits this by computing a "longest prefix match" — pods that have the most consecutive matching blocks from the start of the prompt receive higher scores. The hash seed must match vLLM's `PYTHONHASHSEED` for cross-process consistency, which is why the Helm chart sets `PYTHONHASHSEED: "0"` globally.

The **three-tier index backend** design (in-memory LRU → cost-aware Ristretto → Redis/Valkey) provides flexibility for different deployment scales. Single-pod deployments use the in-memory LRU with 100M key capacity. Memory-constrained environments use the Ristretto backend with byte-level cost tracking and automatic eviction. Multi-pod deployments use Redis/Valkey with pipelined operations and Lua scripts for atomic pruning. The backend is wrapped with decorator layers for metrics (Prometheus) and tracing (OpenTelemetry), following a clean composition pattern.

The **fs_backend connector** is notable for its C++/CUDA extension (`storage_offload`) that handles GPU↔Storage block transfers with fine-grained control over CUDA streams, pinned memory pools, and optional GPU Direct Storage (GDS). The Python layer manages the scheduling-side logic (lookup, prepare_load/store, event publishing) while the C++ layer handles the data plane. The NIXL object storage backend adds a staging buffer pool pattern — GPU blocks are first copied to pinned CPU memory, then NIXL transfers them to S3, avoiding GPU memory pressure during network I/O.

The **PVC evictor** is an interesting design using Python's multiprocessing module with a fixed N+2 process architecture. The activator monitors disk usage via `statvfs` (O(1) operation critical for multi-TB volumes), the crawlers discover eligible files using a hex-modulo load balancing scheme across the FileMapper directory structure, and the deleter performs batch deletions via `xargs rm -f`. The hysteresis pattern (85% trigger → 70% target) prevents deletion thrashing. A notable gap is that the evictor publishes block removal events to ZMQ (for index consistency) but has no mechanism to coordinate with active writers, relying on filesystem atomicity.

A **security concern** is that all inter-component communication (ZMQ, gRPC, Redis, HTTP) is unauthenticated and unencrypted. The UDS tokenizer mitigates this via Unix socket permissions (0o700), but ZMQ subscribers accept connections from any pod that can reach port 5557. In a production RHOAI deployment, this is partially mitigated by namespace-level network policies, but the components themselves have no authentication enforcement.
