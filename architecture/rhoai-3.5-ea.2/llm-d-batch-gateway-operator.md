## Purpose

**Short**: A Kubernetes operator that manages the full lifecycle of LLM-D batch gateway deployments by reconciling a single `LLMBatchGateway` custom resource into a set of Kubernetes resources rendered from an embedded Helm chart.

**Detailed**: The LLM-D Batch Gateway Operator automates the deployment and management of the [batch-gateway](https://github.com/opendatahub-io/batch-gateway) system, which provides an OpenAI-compatible batch inference API for LLM workloads. The operator watches `LLMBatchGateway` custom resources and renders the upstream batch-gateway Helm chart at runtime to produce Kubernetes Deployments, Services, ConfigMaps, ServiceAccounts, and optionally HTTPRoutes, Certificates, ServiceMonitors, PodMonitors, PrometheusRules, and Grafana dashboard ConfigMaps. All rendered resources are applied via Server-Side Apply with owner references for automatic garbage collection.

The operator manages three workload components: an **API server** (HTTP endpoint for batch job submission and file management), a **processor** (worker that dequeues jobs and forwards individual inference requests to an inference gateway), and a **garbage collector** (periodic cleanup of expired jobs and files). Component images are not user-configurable — they are pinned by the platform operator via environment variables sourced from `params.env`, ensuring the platform controls which images run. The operator supports cross-namespace secret references via Gateway API ReferenceGrants, cert-manager integration for automatic TLS, and Gateway API HTTPRoutes for ingress.

## Architectural Analysis

The LLM-D Batch Gateway Operator follows a Helm-rendering operator pattern that is unusual compared to most Kubernetes operators. Rather than creating resources directly through Go structs and the controller-runtime client, it loads the upstream batch-gateway Helm chart from the filesystem at startup (`/charts/batch-gateway`), converts the LLMBatchGateway CR spec into Helm values via `specToHelmValues()`, renders the chart templates in-process using the Helm SDK, and applies the resulting unstructured objects via Server-Side Apply. This design enables the operator to leverage the upstream community's Helm chart without maintaining a parallel set of Go resource constructors, dramatically reducing the surface area for drift between operator-managed and standalone Helm deployments.

A key architectural decision is that **component images are not user-configurable through the CRD**. The three workload images (apiserver, processor, gc) are injected as environment variables on the operator's own Deployment by the platform operator (opendatahub-operator/rhods-operator) via `params.env`. This separation ensures the platform — not individual users — controls which image versions are deployed, which is critical for supply chain security and version consistency across the fleet. The operator reads these environment variables at startup and fails immediately if any are missing (`componentImagesFromEnv()` in `cmd/main.go`).

The cross-namespace secret handling is well-designed. The operator supports a `secretRef` that can point to a Secret in a different namespace, protected by Gateway API ReferenceGrant semantics. When a cross-namespace reference is used, the controller verifies a matching ReferenceGrant exists, copies the Secret data into the CR's namespace under a managed name (`{name}-credentials`), and sets controller owner references for automatic cleanup. The `secretRef` field is immutable after creation — once a managed copy exists, the controller refuses to change the source reference, requiring CR deletion and recreation. This prevents silent credential rotation to a different secret.

The operator runs two controllers: the primary `LLMBatchGatewayReconciler` for CR reconciliation, and a `MetricsController` that self-heals the operator's own monitoring resources (Service, ServiceMonitor, PrometheusRule). The MetricsController watches its own monitoring resources and re-applies them via SSA on any change or deletion, ensuring observability infrastructure is self-healing. It includes two built-in alerts: high reconcile error rate and leader election loss.

The orphan cleanup mechanism (`deleteOrphanedResources`) deserves attention. After rendering and applying all chart resources, the controller lists all resources of managed GVKs in the CR namespace, checks for controller ownership, and deletes any owned resources that are no longer in the rendered output. This ensures that when optional features are disabled (e.g., removing an HTTPRoute or ServiceMonitor), the previously-created resources are cleaned up. The `managedGVKs` list must be kept in sync with the Helm chart's possible resource types — a new resource type in the chart requires adding its GVK to this list.
