## Purpose

**Short**: A centralized library of reusable Kubeflow Pipelines (KFP) components and managed pipeline definitions for AI/ML workflows in Red Hat OpenShift AI.

**Detailed**: The pipelines-components repository is the canonical source for KFP pipeline components and end-to-end managed pipelines used within the RHOAI platform. It provides a structured collection of reusable building blocks organized into four categories — data processing, training, evaluation, and deployment — that users compose into ML workflows via the Kubeflow Pipelines SDK.

The repository produces three container images: (1) a primary `pipelines-components` init container that pre-compiles managed pipeline definitions and stages them for consumption by the KFP API server at pod startup; (2) an `automl` runtime image containing AutoGluon dependencies for tabular and time-series AutoML training; and (3) an `autorag` runtime image bundling Docling models and RAG optimization dependencies for automated retrieval-augmented generation pipelines. The component library itself (`kfp-components` Python package) is installed in these images and provides the `@dsl.component` and `@dsl.pipeline` decorated functions that define the actual ML workflow steps.

The repository includes an extensive build and validation toolchain — compile checks, metadata validation, base image allowlisting, AST-based code analysis, and skeleton generators — that enforces quality standards across all contributed components and pipelines. Managed pipelines (those marked `managed: true` in metadata) are compiled at image build time and staged to a shared volume during KFP API server initialization, enabling the platform to offer pre-built AI workflows out of the box.

## Sub-Component Details

### pipelines-components (Init Container)

**Dockerfile**: `Dockerfile.konflux.pipelines-components`
**Intent**: Primary init container for the KFP API server. At image build time, it compiles all `managed: true` pipelines into KFP YAML specs and bundles them into the image. At pod startup, it copies these pre-compiled YAMLs to a shared volume (`/config/managed-pipelines`) where the KFP API server reads them. If `RELATED_IMAGE_*` environment variables are present (set by the Data Science Pipelines Operator for CI image overrides), it recompiles from source instead of copying pre-built artifacts.
**Image**: odh-pipelines-components
**Language**: Python 3.12
**Port**: None (init container, no network services)

## Architectural Analysis

The pipelines-components repository follows a **component-library architecture** rather than a service-oriented pattern. It produces no long-running processes and exposes no network services of its own. Instead, it defines KFP pipeline components as decorated Python functions that are compiled into Argo Workflow step definitions at build time. The init container pattern — compiling at build, staging at pod startup — decouples the pipeline definitions from the KFP API server lifecycle and enables atomic updates when the operator deploys a new version.

The **metadata-driven discovery system** is a key architectural pattern. Every component and pipeline directory must contain a `metadata.yaml` file with a strict schema (name, stability, dependencies, lastVerified). The build toolchain uses this metadata for discovery, filtering (`managed: true`), stability classification, and validation. The AST-based parsing approach (used in `metadata_parser.py`, `parsing.py`, and `pipeline_description.py`) is a deliberate security decision — the system extracts function signatures, decorators, and descriptions from source files without executing any user-contributed code. This enables safe validation in CI environments.

The **three-container architecture** reflects distinct execution profiles: the init container (`pipelines-components`) is small, fast, and runs once at startup; the AutoML container (`odh-automl`) bundles a heavy ML stack (AutoGluon, PyTorch, scikit-learn) for CPU-intensive ensemble training; and the AutoRAG container (`odh-autorag`) embeds ~1GB of Docling models for offline document processing. This separation avoids bloating the init container with ML dependencies and allows independent versioning of the runtime environments.

The **fine-tuning pipeline matrix** (SFT × LoRA × OSFT) × (standard × minimal × evalhub) = 18 pipeline variants is notable for its combinatorial structure. The `_minimal` variants reduce parameter count from ~53 to ~12, hardcoding sensible defaults for users who don't need fine-grained control. The `_evalhub` variants swap local `lm-eval` evaluation for remote KServe-based evaluation via the Eval Hub API, which requires additional RBAC (InferenceService create/delete) and a ReadWriteMany PVC for model sharing between the training pod and the KServe predictor.

The **RELATED_IMAGE_* recompilation mechanism** in the init container is architecturally significant for the platform operator (DSPO). When DSPO sets `RELATED_IMAGE_*` environment variables to override component base images (e.g., for testing or air-gapped deployments), the init container detects this and recompiles all managed pipelines from source, injecting the overridden images into the compiled YAML. This dual-path (copy-prebuilt vs. recompile) design balances startup speed (pre-built path) with flexibility (recompile path).
