## Purpose

**Short**: A reverse proxy sidecar that orchestrates disaggregated prefill/decode inference by routing requests between prefiller and decoder vLLM instances using the NIXL or LMCache KV-transfer protocols.

**Detailed**: The llm-d-routing-sidecar is a Go-based HTTP reverse proxy designed to run as a sidecar container alongside a vLLM decoder instance in an llm-d disaggregated serving deployment. Its primary role is to intercept OpenAI-compatible completion requests, identify a prefiller target from the `x-prefiller-host-port` HTTP header, orchestrate a two-phase prefill-then-decode protocol, and return the final completion response to the client.

The sidecar supports three P/D (Prefill/Decode) connector protocols: NIXL v2 (default, current), NIXL v1 (deprecated), and LMCache (deprecated). In the NIXL v2 protocol, the sidecar first sends a modified request to the prefiller with `kv_transfer_params` indicating remote decode, captures the KV-transfer metadata from the prefiller's response, then forwards a decode request to the local vLLM instance with the transfer parameters attached. This enables KV-cache transfer between GPU workers via the NIXL library without client-side coordination.

The sidecar also implements SSRF (Server-Side Request Forgery) protection by watching Kubernetes `InferencePool` custom resources (from the Gateway API Inference Extension) and maintaining a dynamic allowlist of valid prefill target pod IPs. When enabled, only requests targeting pods that match the InferencePool's label selector are permitted, preventing arbitrary request forwarding. The proxy supports TLS termination with configurable cipher suites and can operate in both secure (HTTPS) and insecure (HTTP) modes.

## Architectural Analysis

The llm-d-routing-sidecar implements a classic sidecar proxy pattern for disaggregated LLM inference. It sits alongside a decoder vLLM instance and intercepts OpenAI-compatible completion requests on port 8000. The core routing decision is simple: if the `x-prefiller-host-port` header is present, the sidecar orchestrates a two-phase prefill-then-decode protocol; otherwise, it passes the request directly to the local decoder. This header-driven routing delegates scheduling intelligence entirely to the upstream component (the llm-d-inference-scheduler/EPP), keeping the sidecar stateless with respect to routing policy.

The NIXL v2 protocol implementation (`connector_nixlv2.go`) is the most architecturally significant code path. It modifies the original request by injecting `kv_transfer_params` with `do_remote_decode: true` and `max_completion_tokens: 1` (to trigger only KV-cache population on the prefiller), sends this to the remote prefiller, extracts the KV-transfer metadata from the response, then reconstructs the original request with the transfer parameters attached and forwards it to the local decoder. The UUID-based request correlation ensures the decoder can match the KV-cache blocks from the prefiller. The older NIXL v1 protocol uses flat fields (`do_remote_prefill`, `remote_block_ids`, `remote_engine_id`) instead of the nested `kv_transfer_params` envelope. The LMCache protocol is simpler — it sends max_tokens=1 to the prefiller (relying on shared cache) and then the unmodified original request to the decoder.

The SSRF protection system (`allowlist.go`) is a notable security feature that uses Kubernetes dynamic client informers to watch a specific `InferencePool` custom resource (from the `inference.networking.x-k8s.io/v1alpha2` API group). It extracts the pool's label selector, watches pods matching those labels, and maintains an in-memory set of allowed pod IPs and hostnames. The allowlist is rebuilt on every pod add/update/delete event with a 30-second resync period. This prevents the sidecar from being used as an open proxy to arbitrary internal services — only pods in the InferencePool are valid prefill targets. The implementation normalizes `host:port` inputs to extract only the host component for validation, meaning any port on an allowed host is permitted.

The TLS configuration is well-structured with hardcoded FIPS-compatible cipher suites (ECDHE-RSA/ECDSA with AES-GCM-SHA256/384 and ChaCha20-Poly1305) and a minimum TLS version of 1.2. However, the upstream Dockerfile uses `CGO_ENABLED=0` which produces statically linked binaries that would fail Konflux's check-payload FIPS validation. The RHOAI downstream build must use a different Dockerfile with `CGO_ENABLED=1` and `GOEXPERIMENT=strictfipsruntime`. The `InsecureSkipVerify` flags for both prefiller and decoder connections are available but disabled by default, which is appropriate for development/testing but should be verified as disabled in production deployments. The LRU cache for prefiller proxy handlers (capacity 16) prevents creating new reverse proxy instances on every request, which is important for performance in disaggregated serving scenarios with many prefillers.
