## Purpose

**Short**: An Envoy external processing (ext_proc) service that intercepts and transforms inference requests to route them to external LLM providers (OpenAI, Anthropic, Azure, Bedrock, Vertex AI) via the AI Gateway.

**Detailed**: The AI Gateway Payload Processing component is an Inference Payload Processor (IPP) built on the llm-d pluggable framework. It runs as a sidecar or standalone service attached to an Istio Gateway via EnvoyFilter, processing incoming inference requests through a plugin pipeline. The pipeline resolves model names to external provider endpoints, translates between API formats (OpenAI Chat Completions, Anthropic Messages, Vertex AI GenerateContent), injects provider-specific authentication credentials, and optionally enforces content safety via NeMo Guardrails.

The component also includes three controller-runtime controllers that manage Kubernetes resources declaratively: the ExternalProvider controller creates ExternalName Services, Istio ServiceEntries, and DestinationRules for each configured provider endpoint; the ExternalModel controller creates Gateway API HTTPRoutes that route client traffic to the correct provider's Service; and a legacy migration controller translates old `maas.opendatahub.io` CRs into the new `inference.opendatahub.io` API. Together, these controllers automate the networking configuration required to proxy inference traffic from the platform's Gateway to external LLM provider APIs over TLS.

## Architectural Analysis

The AI Gateway Payload Processing component follows a plugin-based architecture inherited from the llm-d Inference Payload Processor (IPP) framework. The framework provides the Envoy ext_proc gRPC server, plugin registry, controller runner, and health check endpoints, while this repository contributes the domain-specific plugins and CRD controllers. This separation is reflected in `go.mod` where `github.com/llm-d/llm-d-inference-payload-processor` is the primary dependency, and in `cmd/main.go` where the llm-d runner is instantiated with custom plugins and controllers. The Helm chart similarly wraps the upstream `payload-processor` chart as a dependency.

The plugin pipeline is designed as a chain with explicit state passing via `CycleState` — a request-scoped key-value store. The model-provider-resolver plugin runs first and writes provider, model, auth type, endpoint, and credentials reference into CycleState. Downstream plugins (api-translation, apikey-injection) read these keys to determine their behavior. This design decouples plugin logic — the api-translation plugin doesn't know how models are resolved, and the apikey-injection plugin doesn't know about API formats. The passthrough optimization in api-translation is notable: when the client's input format matches the upstream provider's format (and is not OpenAI-chat, which needs path rewriting), translation is skipped entirely.

The controller architecture creates a clean separation between the data plane (plugin pipeline) and control plane (CRD reconciliation). The ExternalProvider controller creates Istio networking primitives (ServiceEntry for DNS resolution, DestinationRule for TLS origination, ExternalName Service for in-mesh addressing), while the ExternalModel controller creates Gateway API HTTPRoutes for traffic routing. This means the controllers configure Istio and the Gateway API, but the actual request processing (body mutation, auth injection) happens in the ext_proc plugin pipeline. The model-provider-resolver plugin also watches the same CRDs via its own in-process reconcilers, maintaining an in-memory store (`infoStore`) for fast request-time lookups — avoiding API server calls on the hot path.

The weighted random selection in the model-provider-resolver plugin enables multi-provider load distribution for a single model. An ExternalModel can reference up to 64 ExternalProviderRefs, each with a weight (1-100), enabling scenarios like A/B testing between providers or cost-optimized routing. The legacy migration controller provides backward compatibility with the older `maas.opendatahub.io` API group by watching old CRs and creating corresponding new-API CRs, which then trigger the standard controller flow.

The NeMo Guardrails integration supports both OpenAI chat format and MCP JSON-RPC format for content safety checks. The request guard extracts user messages (or MCP tool arguments) and posts them to NeMo's `/v1/guardrail/checks` endpoint, while the response guard extracts assistant messages from OpenAI choices or MCP result content. NeMo always returns HTTP 200 with a status field (`passed`, `modified`, `blocked`), and the plugins translate `blocked` to a 403 Forbidden error. The `modified` status (content redaction) is acknowledged but not yet applied — a noted TODO.
