## Purpose

**Short**: An HTTP reverse proxy that enforces Kubernetes RBAC authorization via SubjectAccessReview before forwarding requests to an upstream service.

**Detailed**: kube-rbac-proxy is a small HTTP/HTTPS reverse proxy designed to run as a sidecar container alongside application containers in Kubernetes pods. It intercepts incoming requests, authenticates them (via Kubernetes TokenReview delegation or OIDC JWT verification), and authorizes them by issuing SubjectAccessReview requests against the Kubernetes API server. Only authenticated and authorized requests are forwarded to the upstream application container, which typically listens on localhost.

In RHOAI 3.x, kube-rbac-proxy is the primary authentication/authorization sidecar injected into component pods by the platform operator (rhods-operator). It replaces the oauth-proxy pattern used in RHOAI 2.x. Each RHOAI component that exposes an HTTP endpoint (dashboard, model registry, TrustyAI, notebooks, etc.) runs a kube-rbac-proxy sidecar on port 8443/TCP that fronts the application container. The proxy enforces fine-grained Kubernetes RBAC policies on every request, enabling per-resource authorization (e.g., "can this user GET metrics for this namespace?") rather than simple all-or-nothing authentication.

The proxy supports two authorization configuration formats: Format1 (top-level `resourceAttributes` with optional `rewrites` for query parameter or header-based SAR field templating) and Format2 (path-scoped `endpoints` with per-HTTP-method resource rules). It also supports a hardcoded authorizer that always allows the OpenShift monitoring Prometheus service account to scrape `/metrics`, static authorization rules, and X.509 client certificate authentication.

## Architectural Analysis

kube-rbac-proxy follows the sidecar proxy pattern, a design choice that decouples authentication and authorization logic from the application it protects. The application container listens only on the loopback interface (127.0.0.1), making it inaccessible directly from the pod network. All external traffic must pass through the kube-rbac-proxy container, which terminates TLS, authenticates the caller, authorizes the request against Kubernetes RBAC, and only then forwards the request to the application. This separation of concerns means application developers do not need to implement RBAC logic, and the security posture is consistent across all RHOAI components.

The authorization model is notably flexible. The proxy supports two configuration formats: Format1, which applies a single `resourceAttributes` template (with optional query parameter or HTTP header rewrites via Go's `text/template`) to all requests; and Format2, which provides path-scoped endpoint rules with per-HTTP-method resource mappings. Format2 enables fine-grained per-API-path authorization, such as allowing POST to `/api/v1/evaluations` to require `create` on `trustyai.opendatahub.io/status-events` while GET on the same path requires `get`. This is architecturally significant because it enables RHOAI components to define precise RBAC policies for each API endpoint without modifying the proxy code -- only the config file changes.

The hardcoded metrics authorizer (`pkg/hardcodedauthorizer/metrics.go`) is an OpenShift-specific addition that unconditionally allows the `system:serviceaccount:openshift-monitoring:prometheus-k8s` service account to GET `/metrics`. This bypasses the normal SAR flow for metrics scraping, ensuring that Prometheus can always collect metrics regardless of RBAC configuration. The authorizer is part of a union chain (hardcoded metrics -> static rules -> SAR), meaning the first authorizer to return `Allow` wins. This chain ordering means the hardcoded metrics check runs first on every request, a conscious tradeoff of a small per-request cost for reliable observability.

The TLS implementation is thorough. The server supports configurable minimum TLS version (default 1.2), configurable cipher suites, HTTP/2 with configurable max concurrent streams and frame sizes, and optional HTTP/2 disable (motivated by CVE-2023-44487, the HTTP/2 rapid reset attack). TLS certificates are hot-reloaded via a polling-based `CertReloader` that watches for file changes at a configurable interval. The upstream transport also enforces TLS 1.2+ minimum and sets `NextProtos` to `["h2", "http/1.1"]` for ALPN negotiation. A log sanitization filter (`SanitizingFilter`) masks tokens in `TokenReview` log messages to prevent credential leakage.

The FIPS compliance posture is strong at both layers. Build-time, the Konflux Dockerfile uses `GOEXPERIMENT=strictfipsruntime` and `CGO_ENABLED=1` to produce a dynamically-linked binary that delegates cryptographic operations to OpenSSL via the Go FIPS runtime. Application-level, the code uses only `crypto/tls` and `crypto/x509` from the Go standard library (no third-party crypto), enforces TLS 1.2+ on all connections (both client-facing and upstream), and does not use any non-FIPS algorithms like MD5 or RC4.
