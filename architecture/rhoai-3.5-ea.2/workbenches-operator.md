## Purpose

**Short**: A Kubernetes operator that manages the lifecycle of the Workbenches component (Kubeflow Notebook infrastructure) for the Red Hat OpenShift AI platform.

**Detailed**: The workbenches-operator is a controller-runtime-based Kubernetes operator that reconciles a cluster-scoped `Workbenches` custom resource. It serves as the component-level operator for workbenches (Kubeflow Notebooks) within the RHOAI/ODH decomposed operator architecture. The platform orchestrator (rhods-operator or opendatahub-operator) creates and projects configuration into the `Workbenches` CR, and this operator handles component-specific lifecycle management.

The operator's current responsibilities include: creating and labeling the workbench namespace (defaulting to `rhods-notebooks` for RHOAI or `opendatahub` for ODH), computing kustomize parameters for manifest rendering (gateway domain, MLflow enablement, section titles), and monitoring the health of workbench-related deployments. It reports readiness through a standard set of status conditions (`Ready`, `ProvisioningSucceeded`, `Degraded`, `DeploymentsAvailable`) and supports two management states: `Managed` (active reconciliation) and `Removed` (teardown).

This operator is in its early development phase. It currently performs namespace management and deployment health checking but does not yet apply kustomize manifests directly — parameter computation is implemented but manifest application is pending. The operator is part of the RHOAI 3.5+ decomposed architecture where each platform component has its own dedicated operator, replacing the monolithic approach of earlier versions.

## Architectural Analysis

The workbenches-operator is a newly scaffolded component operator that is part of RHOAI's transition from a monolithic operator architecture to a decomposed model where each platform component has its own dedicated operator. This is evident from the repository's young commit history (only 6 commits), the v1alpha1 API version, and several forward-looking code patterns that establish the foundation for future functionality.

**Decomposed Operator Pattern**: The operator follows a "projected configuration" pattern where the platform orchestrator creates the singleton `Workbenches` CR (enforced by the CEL validation rule `self.metadata.name == 'default'`) and projects platform-wide configuration into its spec. Fields like `gatewayDomain`, `platform`, and `mlflowEnabled` are explicitly documented as "projected by the orchestrator," meaning the orchestrator writes these values and the component operator consumes them. This creates a clean contract boundary — the orchestrator owns the spec, the component operator owns the status. The contract schema in `contracts/schemas/workbenches.v1alpha1.json` formalizes this API.

**Incomplete Reconciliation Loop**: The `computeKustomizeParams()` method (workbenches_controller.go:279-290) computes parameters like `section-title`, `mlflow-enabled`, and `gateway-url`, but the reconciler does not yet apply these parameters to any kustomize manifests. The `reconcileManaged()` flow calls `configureDependencies()` (namespace management) and `checkDeployments()` (health monitoring), but there is no manifest rendering or application step. This suggests the operator is being developed incrementally — namespace management first, then manifest application.

**Forward-Looking GVK Definitions**: The `internal/gvk/gvk.go` file defines GroupVersionKind constants for `Notebook` (kubeflow.org/v1), `ImageStream` (image.openshift.io/v1), and `HardwareProfile` (infrastructure.platform.opendatahub.io/v1). None of these are currently watched or reconciled, but they signal the operator's future scope: managing Kubeflow Notebook resources, OpenShift ImageStreams (for notebook images), and HardwareProfiles (for GPU/accelerator allocation). Similarly, the annotations defined in `internal/metadata/annotations.go` (hardware profile name/namespace, accelerator name/namespace, connection annotations, last size selection) indicate planned integration with the dashboard and notebook infrastructure.

**Security Posture**: The operator has a reasonable security baseline for an early-stage component. The container runs as non-root (UID 65532), drops all capabilities, uses a read-only root filesystem, and the Go binary is built with `strictfipsruntime` and `CGO_ENABLED=1` for FIPS compliance. The metrics endpoint uses HTTPS with SubjectAccessReview-based authentication. However, the operator has no Konflux Dockerfiles yet, no `rpms.lock.yaml`, and no NetworkPolicy — the kustomization comment in `config/manager/kustomization.yaml` explicitly notes the NetworkPolicy TODO with a CWE-284 reference. The absence of Konflux build infrastructure means the component is not yet going through the hermetic build pipeline required for GA.

**Operational Resilience**: The controller uses a custom exponential backoff rate limiter (5s base, 5m max) to avoid tight retry loops on persistent failures. The requeue delay for deployment health checks is 30 seconds. These are sensible defaults for an operator that primarily monitors deployment readiness. No PodDisruptionBudget or HorizontalPodAutoscaler is configured, which is appropriate for a single-replica controller manager but should be addressed before production deployment.
