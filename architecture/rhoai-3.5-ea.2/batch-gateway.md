## Purpose

**Short**: OpenAI-compatible batch inference API gateway for llm-d, providing job submission, processing, and lifecycle management in Kubernetes environments.

**Detailed**: Batch Gateway is a high-performance distributed system for processing large-scale batch inference jobs. It provides full OpenAI API compatibility via `/v1/batches` and `/v1/files` endpoints, enabling users to submit, track, and manage batch inference workloads of up to 50,000 requests per job.

The system is composed of three independently deployable Go binaries: an API server that handles REST requests and manages job/file metadata, a batch processor that dequeues jobs from a priority queue and dispatches inference requests to downstream llm-d routers with adaptive concurrency control (AIMD), and a garbage collector that performs TTL-based cleanup and orphan reconciliation. The data layer is pluggable, supporting PostgreSQL for persistent metadata storage, Redis/Valkey for priority queues and event channels, and S3 or filesystem backends for file storage.

Batch Gateway is designed to coexist with interactive inference workloads. The processor monitors downstream system metrics and dynamically adjusts dispatch flow to minimize interference with interactive requests while meeting batch job SLO deadlines. Authentication and authorization are delegated to the gateway layer (Kuadrant/Authorino), while multi-tenancy isolation is enforced at the data layer through tenant-scoped queries and storage partitioning.

## Architectural Analysis

Batch Gateway follows a clean three-tier architecture with well-separated concerns. The API server is stateless and horizontally scalable, handling only request validation and metadata management. The processor is the computational core, implementing a sophisticated job lifecycle with four distinct phases (polling, preprocessing, execution, finalization) and comprehensive error recovery. The garbage collector runs independently as a single replica, performing periodic cleanup and orphan reconciliation. This separation allows each component to be scaled, deployed, and configured independently.

The concurrency control system is particularly well-designed. The processor implements a three-level semaphore hierarchy: worker tokens limit concurrent job goroutines, global semaphores limit total in-flight requests, and per-endpoint semaphores implement AIMD (Additive Increase / Multiplicative Decrease) adaptive concurrency. When downstream endpoints return 429 or 5xx errors, the AIMD controller multiplicatively decreases the per-endpoint limit (by the backoff factor of 0.5), and additively increases it on success windows. This ensures batch workloads adapt to real-time downstream capacity without overwhelming interactive traffic. The context hierarchy (pollingCtx → jobBaseCtx → sloCtx → requestAbortCtx) enables graceful degradation: SIGTERM stops new job intake while allowing in-flight jobs to complete, and SLO deadline expiration aborts only dispatch while preserving partial results for upload.

The data layer abstraction is noteworthy for its pluggability and consistency guarantees. All database operations use a common `DBClient[T, Q]` interface that supports CAS (Compare-and-Swap) updates via an `expectedStatus` parameter, preventing lost updates when multiple actors (processor, GC reconciler) attempt concurrent status transitions. The Redis backend uses embedded Lua scripts for atomic store/update/delete operations, while PostgreSQL uses conditional `WHERE status=$expected` clauses. The priority queue uses Redis sorted sets with SLO deadline microseconds as scores, ensuring jobs with tighter deadlines are processed first. The GC reconciler detects orphaned jobs (non-terminal, not in queue, not in-flight or stale) and triages them based on status and SLO validity — re-enqueueing if still valid, or transitioning to terminal status if expired.

The security model delegates authentication to the external gateway layer (Kuadrant/Authorino) while enforcing multi-tenancy isolation internally. Every database query is scoped by tenant ID (extracted from the `X-MaaS-Username` header), and file storage uses deterministic SHA256-based folder names derived from tenant IDs. The API server applies strict input validation: JSON decoding with `DisallowUnknownFields`, file size limits (200 MB), line count limits (50,000), and parameter range validation. Secrets are mounted read-only at `/etc/.secrets/` and accessed via `os.OpenInRoot()` to prevent path traversal. All three Konflux Dockerfiles build with `GOEXPERIMENT=strictfipsruntime` and `CGO_ENABLED=1` for FIPS compliance, and every TLS configuration in the codebase enforces a minimum of TLS 1.2. Pod security contexts enforce non-root execution, read-only root filesystem, all capabilities dropped, no privilege escalation, and seccomp RuntimeDefault.

One design risk worth noting: the processor uses an emptyDir volume for local job artifacts (execution plans, intermediate files), which means data survives container restarts within the same pod but is lost on pod rescheduling. The startup recovery mechanism handles container restarts by scanning the work directory and recovering jobs based on their last known status, but pod-level failures require the GC reconciler to detect and re-enqueue orphaned jobs (with up to a 60-minute detection interval by default). For production deployments with strict SLOs, reducing the reconciler interval and ensuring adequate processor replicas is important.
