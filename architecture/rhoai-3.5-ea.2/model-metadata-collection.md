## Purpose

**Short**: A Go CLI tool that extracts, enriches, and catalogs metadata from Red Hat AI container model images and MCP servers, producing YAML catalog files shipped as a data-only container image.

**Detailed**: Model Metadata Collection is a build-time data pipeline, not a runtime service. It discovers AI models from HuggingFace collections, processes OCI container image layers to extract model cards, enriches metadata from multiple sources (HuggingFace API, YAML frontmatter, container registry), and generates structured YAML catalogs. It also processes MCP (Model Context Protocol) server metadata from index files, enriching them with OCI registry data (architectures, timestamps).

The primary output is a set of pre-generated YAML catalog and index files (`data/*.yaml`) that are baked into a minimal container image (`registry.access.redhat.com/ubi9-minimal`). This container runs `sleep infinity` and serves as a data volume source — other platform components (such as the RHOAI dashboard) mount its `/app/data` directory to access the model and MCP server catalogs at runtime. The tool itself (the Go binary) is never shipped in the production image; only its output artifacts are.

The pipeline supports three catalog tiers: Red Hat validated models, other/community models, and MCP servers (Red Hat, partner, and community). Each tier has its own index file and generated catalog. The tool includes concurrent processing, retry logic with exponential backoff for registry operations, model family matching to prevent cross-family enrichment errors, and comprehensive metadata completeness reporting.

## Architectural Analysis

The model-metadata-collection component follows a **build-time data pipeline** architecture rather than the typical runtime service pattern seen in most RHOAI components. The Go binary (`model-extractor`) is a sophisticated ETL tool that runs during CI/CD, and its output (YAML files) is what gets shipped. This is a deliberate design choice that eliminates runtime dependencies on external APIs — the production container needs no network access, no secrets, and no elevated permissions.

The enrichment pipeline implements a **multi-source priority hierarchy** for metadata: HuggingFace YAML frontmatter (highest priority) > modelcard.md regex extraction > HuggingFace API data > registry metadata > generated defaults. This priority chain is implemented across `internal/enrichment/enrichment.go` and `internal/enrichment/update.go`, with source tracking via the `EnrichmentInfo` struct that records the provenance of every metadata field. The `enrichment.yaml` sidecar file created alongside each `metadata.yaml` enables the `metadata-report` tool to produce accurate data source analysis.

The **model family matching** system (`internal/config/model_families.go`) is a critical correctness feature that prevents cross-family enrichment errors. Without it, a Llama container image could incorrectly match with a Granite HuggingFace entry due to string similarity. The centralized `SupportedModelFamilies` slice serves as the single source of truth, with a dynamically-generated regex pattern used by both `NormalizeModelName()` and `extractModelFamily()` — ensuring consistent behavior across name normalization and family-based match filtering.

The container image architecture is notable for its minimalism. `Dockerfile.konflux` uses `ubi9-minimal` as the base and contains only `COPY` and permission-setting instructions — no package installation, no build stages, no binary. The `VOLUME ["/app/data"]` directive and `CMD ["sleep", "infinity"]` clearly signal the init-container/sidecar usage pattern. This produces an extremely small, auditable image with a minimal attack surface.

The MCP server catalog generation was added as a parallel pipeline to the model catalog, sharing the same enrichment infrastructure (OCI registry inspection for architectures and timestamps) but with its own index/input/output structure. The three-tier MCP catalog (Red Hat, partner, community) mirrors the model catalog tiers and supports independent processing via the `--skip-huggingface --skip-enrichment --skip-catalog` flags that enable MCP-only mode.
