## Purpose

**Short**: Manages the lifecycle of Model Registry and Model Catalog instances on OpenShift/Kubernetes, deploying REST API services, database backends, authentication proxies, and Gateway API ingress.

**Detailed**: The model-registry-operator is a Kubernetes operator built with controller-runtime (kubebuilder v4 scaffold) that reconciles `ModelRegistry` custom resources. For each CR, the operator deploys a complete Model Registry stack: a REST API service backed by PostgreSQL or MySQL, a kube-rbac-proxy authentication sidecar, OpenShift Routes for backward compatibility, and Gateway API HTTPRoutes for the RHOAI 3.x ingress pattern. The operator supports three platform modes: plain Kubernetes, traditional OpenShift (with user.openshift.io API), and BYOIDC OpenShift 4.20+ (where user.openshift.io is absent).

In addition to per-instance Model Registry management, the operator runs a separate ModelCatalog controller that manages a single shared Model Catalog deployment. The catalog aggregates model metadata from multiple configurable sources (ConfigMaps with label `opendatahub.io/catalog-source=true`), supports benchmark data loading via init containers, and shares the same kube-rbac-proxy authentication and Gateway API ingress patterns as the registry instances.

The operator handles CRD version migration (v1alpha1 to v1beta1) using either Kubernetes StorageVersionMigration API or a manual re-read/re-write fallback strategy. It also performs in-place migration from OAuth proxy to kube-rbac-proxy authentication during reconciliation for legacy registries.

## Architectural Analysis

The model-registry-operator exemplifies the RHOAI 3.x migration pattern from Istio-based networking to Gateway API with kube-rbac-proxy authentication. The v1alpha1 API version included Istio configuration (authProvider, authConfigLabels, tlsMode, gateway) which the operator now automatically migrates: the defaulting webhook converts Istio config to OAuthProxy config, and the controller then migrates OAuthProxy to KubeRBACProxy in-place during reconciliation. This three-stage migration path (Istio → OAuthProxy → KubeRBACProxy) enables seamless upgrades from RHOAI 2.x era registries without user intervention.

The cross-namespace networking design is architecturally significant. HTTPRoutes are created in a centralized namespace (default: `redhat-ods-applications`) referencing a Gateway in the ingress namespace, while the actual Service backends live in user namespaces. This requires ReferenceGrants in each user namespace to authorize cross-namespace references. The operator creates one ReferenceGrant per namespace (shared across all registries in that namespace) using create-only semantics — it never updates or deletes the ReferenceGrant during individual registry deletion, preventing one registry's cleanup from breaking another's ingress. This is a deliberate design choice documented in comments.

The BYOIDC (Bring Your Own Identity Provider) support reveals careful platform adaptation. On traditional OpenShift clusters, the operator creates `user.openshift.io/v1/Group` resources for registry access control. On BYOIDC OpenShift 4.20+ clusters, the User API is absent (returns "not found" during discovery), so the operator skips Group creation and instead references external IdP groups directly in RoleBindings. The `DetectClusterCapabilities()` function handles partial API discovery failures gracefully, distinguishing between expected BYOIDC "not found" errors and unexpected failures that should block the controller.

The Model Catalog controller (added in 3.0) uses a single-instance deduplication pattern: all watched resources map to a fixed reconcile request `{Name: "model-catalog", Namespace: TargetNamespace}`, ensuring only one reconcile loop runs regardless of how many resources change. The catalog supports dynamically discovered sources via labeled ConfigMaps, with alphabetical sorting for deterministic ordering. The catalog deployment uses init containers to load metadata and benchmark data from separate container images, keeping the catalog service image focused on the API layer.

Security is enforced at multiple layers. Network policies restrict PostgreSQL access to only the registry pods that own the database. kube-rbac-proxy performs SubjectAccessReview checks against Kubernetes RBAC, requiring users to have `get` permission on the specific Service resource before proxying requests. The operator integrates with OpenShift's TLS security profile system, watching for profile changes on `config.openshift.io/APIServer` and performing a graceful restart to apply updated cipher suites and TLS versions. All containers run as non-root with `RuntimeDefault` seccomp profiles and dropped capabilities.
