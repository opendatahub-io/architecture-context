## Purpose

**Short**: A Kubernetes controller that performs intelligent autoscaling for LLM inference model servers based on saturation, queueing theory, and throughput analysis across heterogeneous GPU variants.

**Detailed**: The Workload Variant Autoscaler (WVA) is a global autoscaler for inference model servers serving LLMs on Kubernetes. It introduces the concept of "variants" — multiple model servers in an InferencePool that serve the same base model but differ in hardware configuration (GPU type), serving configuration (tensor parallelism, max batch size, quantization), or both. Use cases include prefill/decode disaggregation (P/D), batch vs. interactive workload separation, and cost-optimized accelerator selection.

WVA operates alongside standard Kubernetes HPA and KEDA external autoscalers. Rather than directly scaling deployments, WVA computes optimal replica counts using multiple analyzer engines (saturation-based, queueing model, throughput), emits custom Prometheus metrics (`wva_desired_replicas`), and lets HPA/KEDA read those metrics to drive the actual scale subresource updates. This architecture decouples the scaling intelligence from the scaling actuation, enabling WVA to work with any external autoscaler that supports Prometheus metrics.

The controller continuously monitors request rates and server performance via Prometheus, computes capacity using KV cache utilization and queue depth, supports GPU-aware resource limiting with per-accelerator-type budgets, and provides scale-to-zero capability for idle models. Configuration is managed via a multi-tiered ConfigMap system supporting global defaults with per-namespace and per-model overrides.

## Architectural Analysis

The Workload Variant Autoscaler implements a sophisticated multi-engine scaling architecture that separates scaling intelligence from scaling actuation. The core design insight is that WVA never directly modifies the `replicas` field on Deployments or LeaderWorkerSets. Instead, it computes optimal replica counts and publishes them as custom Prometheus metrics (`wva_desired_replicas`), which HPA or KEDA reads as external metrics to drive actual scaling. This decoupling enables WVA to work with any Kubernetes autoscaler that supports Prometheus-based external metrics, and prevents conflicts between multiple scaling controllers.

The analyzer pipeline is the most architecturally significant subsystem. Three independent analyzers — Saturation V2 (token-based capacity model), Queueing Model (SLO-driven with Kalman filter tuning), and Throughput (ITL linear model) — can be composed via the `Analyzers[]` configuration. Each analyzer produces a unified `AnalyzerResult` with per-variant supply/demand/utilization breakdowns. The engine post-step applies universal thresholds to compute required and spare capacity, then feeds results to one of two optimizers: CostAwareOptimizer (unlimited GPU mode, sorts by cost-efficiency) or GreedyByScoreOptimizer (GPU-limited mode, fair-share allocation across models). The pipeline then applies a scale-to-zero enforcer and GPU limiter before caching final decisions.

The VariantAutoscaling CRD is explicitly deprecated (kubebuilder deprecation marker and reconciler warning), with annotation-based discovery (`llm-d.ai/managed=true` on HPA/ScaledObject) as the preferred path. This migration reflects a design philosophy shift: rather than requiring users to create a new CRD alongside their HPA, WVA annotates existing HPAs. The HPA and ScaledObject reconcilers are lightweight — they only track namespace membership in the in-memory datastore, enabling the engine's polling loop to scope its List queries to namespaces that actually have managed scalers.

The multi-tiered ConfigMap architecture (global + namespace-local with opt-in labels and exclusion annotations) enables sophisticated multi-tenant configuration without requiring per-namespace operator instances. ConfigMaps are bootstrapped before the engine starts via `BootstrapInitialConfigMaps()` using a temporary uncached client, ensuring configuration is available before any scaling decision is made. Dynamic updates flow through the ConfigMap reconciler with predicate-based filtering to minimize unnecessary reconciliation.

Prefill/Decode disaggregation support is deeply integrated throughout the stack. Pod template labels (`llm-d.ai/role`) propagate through the analyzer pipeline as `RoleCapacities` maps, enabling per-role supply/demand tracking. The GreedyByScoreOptimizer implements paired variant picking with joint-commit constraints to ensure balanced allocation across disaggregated roles. Scale-to-zero is applied at the model level (all variants together) rather than per-role, preventing partial shutdown of disaggregated pipelines.

The experimental Coordinator subsystem (disabled by default, opt-in via `EXPERIMENTAL_COORDINATOR_ENABLED`) introduces a leader-elected periodic loop (15-second default interval) for cluster-wide cross-cutting optimizations. Currently, only the GPU Rebalance plugin is implemented, which proportionally allocates GPU quota across managed HPAs based on inference pool queue depth queried from Prometheus. The plugin architecture is extensible — new plugins implement a simple `Tick(ctx, selected)` interface and are dispatched by the coordinator each cycle.
