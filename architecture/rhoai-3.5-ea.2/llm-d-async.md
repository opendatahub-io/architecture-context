## Purpose

**Short**: Asynchronous inference processor that dequeues LLM inference requests from a message queue, dispatches them to an inference gateway, and writes results back to the queue.

**Detailed**: llm-d-async (the "Async Processor" or AP) is a Go service that provides asynchronous inference processing for LLM workloads within the llm-d ecosystem. It sits between a message queue (Redis or GCP Pub/Sub) and an inference gateway (IGW), consuming requests from one or more named queues, forwarding them as HTTP POST requests to the gateway, and publishing results back to the queue system. The processor supports concurrent workers, configurable flow control gates (dispatch budgets, rate limiting, concurrency quotas), and exponential backoff retry logic with deadline-based request expiration.

The component is structured around a pluggable `Flow` pipeline abstraction with four concrete implementations: Redis pub/sub (`RedisMQFlow`), Redis sorted set (`RedisSortedSetFlow`), GCP Pub/Sub (`PubSubMQFlow`), and GCP Pub/Sub with gating (`PubSubMQFlow` with gate factory). Each flow manages its own request subscription, retry queue, and result publishing. A `RequestMergePolicy` (currently "random-robin") merges multiple request channels into a single dispatch stream. Workers process requests with per-request deadlines, exponential backoff with jitter, and categorized error handling (rate-limit, server, invalid, auth errors). The service also includes a producer SDK module (`producer/`) that provides a Go client library for submitting requests and retrieving results from the sorted-set queue.

Flow control is a key architectural feature. The processor supports multiple gate types — Redis-backed dispatch gates, Prometheus-based saturation and budget gates, composite gates, and Redis-backed quota gates (rate-limit and concurrency modes). These gates dynamically throttle request dispatch based on real-time metrics from the inference pool, enabling back-pressure propagation from model servers to the async queue.

## Architectural Analysis

The async processor implements a classic producer-consumer pattern with a pluggable message queue abstraction. The `pipeline.Flow` interface cleanly separates the message transport (Redis pub/sub, Redis sorted set, GCP Pub/Sub) from the worker dispatch logic. This design allows the same worker pool to operate against different queue backends without modification. The sorted-set flow is the most feature-complete, supporting deadline-priority ordering (earliest deadline first), per-queue dispatch gates, attribute-based quota gates, and configurable result queue routing. The pub/sub flow is simpler and relies on Redis's built-in subscription semantics.

The flow control system is notably sophisticated for a queue processor. The `GateFactory` creates dispatch gates from declarative configuration (JSON or CLI flags), supporting a layered gating strategy: Prometheus-based pool saturation gates use PromQL queries to compute dispatch budgets dynamically from inference pool metrics, while Redis-backed quota gates enforce per-attribute rate limits or concurrency caps. The `CompositeGate` allows combining multiple gate types (e.g., a Prometheus saturation gate AND a Redis quota gate), taking the minimum budget across all inner gates and performing all-or-nothing quota acquisition for attribute gates. The `CascadeMetricSource` provides automatic fallback between metric sources — if the primary EPP flow-control queue-size metric is unavailable, it falls back to computing saturation from raw vLLM `num_requests_running` metrics.

The retry system uses Redis sorted sets with score-based scheduling. Failed requests are enqueued with a future timestamp as the score (current time + backoff duration), and a polling worker atomically pops due messages using a Lua script. The backoff follows exponential growth with equal jitter (base 2 seconds, cap 60 seconds), respecting server-specified `Retry-After` headers. The retry worker itself uses adaptive polling with exponential backoff (1s to 30s) when idle, resetting to 1s on activity. Graceful shutdown is carefully implemented: workers re-enqueue in-flight requests on SIGTERM, and the drain context ensures retry and result workers flush buffered data before exit.

The producer SDK (`producer/` module) is a separate Go module designed for import by external applications. It provides a `RedisSortedSetProducer` that submits requests to the sorted set with deadline-based scoring and retrieves results via blocking list pop (`BRPOP`). The `WithRedisClient` option allows callers to inject pre-instrumented Redis clients (e.g., with OpenTelemetry hooks). The producer module has no dependency on the main async-processor code — it depends only on the `api/` module for shared types, making it lightweight for external consumers.

The codebase uses `controller-runtime` primarily for its metrics server infrastructure and logging framework, not for Kubernetes controller/reconciler patterns. The service does not watch any CRDs or manage Kubernetes resources. It uses `k8s.io/client-go` for kubeconfig-based authentication of the metrics endpoint via SubjectAccessReview, which is the only Kubernetes API interaction.
