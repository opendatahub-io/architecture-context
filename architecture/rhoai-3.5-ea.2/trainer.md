## Purpose

**Short**: Kubernetes operator that manages distributed ML training jobs by orchestrating JobSet-based workloads with PyTorch, MPI, DeepSpeed, MLX, and TorchTune runtimes.

**Detailed**: The Kubeflow Trainer (v2) is a Kubernetes operator that provides a declarative API for running distributed machine learning training jobs on OpenShift. Users create TrainJob custom resources that reference TrainingRuntime or ClusterTrainingRuntime templates, which define the training framework configuration (PyTorch distributed, MPI, TorchTune, etc.), node topology, and container images. The controller reconciles these CRs into JobSet resources that manage the actual training pods.

The operator implements a plugin-based runtime framework where ML framework-specific logic (PyTorch torchrun configuration, MPI hostfile generation, TorchTune recipe selection) is encapsulated in separate plugins. Gang-scheduling is supported through both the Kubernetes scheduler-plugins (coscheduling) and Volcano PodGroup integrations.

For RHOAI 3.5, the operator includes Red Hat-specific enhancements (`pkg/rhai/`): progression tracking that polls training pod HTTP metrics endpoints and writes real-time progress (steps, epochs, loss) into TrainJob annotations, and NetworkPolicy creation for pod isolation and metrics endpoint security. RHOAI ships pre-configured ClusterTrainingRuntimes for PyTorch distributed training with CUDA, ROCm, and CPU variants, as well as Training Hub runtimes that provide notebook-style interactive training environments.

## Architectural Analysis

The Kubeflow Trainer v2 represents a significant redesign from the v1 training-operator. The v1 operator had separate CRDs per framework (PyTorchJob, MPIJob, TFJob, etc.), while v2 unifies everything under a single TrainJob CRD with a pluggable runtime framework. The runtime is selected via `runtimeRef` which points to a TrainingRuntime or ClusterTrainingRuntime, and the framework plugins (Torch, MPI, PlainML, TorchTune) dynamically configure the JobSet spec based on the ML policy. This design is cleaner architecturally but introduces indirection -- understanding what a TrainJob will actually do requires resolving the runtime reference and understanding which plugins will be activated.

The runtime framework's plugin architecture (`pkg/runtime/framework/`) follows a clear interface pattern: `EnforceMLPolicyPlugin` for ML-specific configuration (torchrun env vars, MPI hostfiles), `EnforcePodGroupPolicyPlugin` for scheduling (coscheduling/Volcano PodGroups), `ComponentBuilderPlugin` for resource construction (JobSet building), `PodNetworkPlugin` for endpoint discovery, and `CustomValidationPlugin` for admission validation. Each plugin is registered in `pkg/runtime/framework/plugins/registry.go` and composed at runtime. The JobSet plugin is the central builder that constructs the actual Kubernetes resources, while the Torch and MPI plugins inject framework-specific environment variables, containers, volumes, and network configuration.

The RHOAI-specific extensions in `pkg/rhai/` add two key features not present upstream: progression tracking and NetworkPolicy management. Progression tracking works by polling training pods' HTTP `/metrics` endpoints during reconciliation, parsing JSON training status (steps, epochs, loss metrics), and writing the results as annotations on the TrainJob CR. This is a pull-based model -- the controller uses `RequeueAfter` with the configurable poll interval (default 30s) rather than watching pod events. The NetworkPolicy manager creates per-TrainJob ingress rules that allow inter-pod communication within the same job (essential for NCCL/MPI/gRPC) and controller-to-metrics access on the metrics port. The controller identifies itself via namespace selector and pod labels (`app.kubernetes.io/name: trainer`, `app.kubernetes.io/component: controller`), which differ between upstream (component=manager) and RHOAI (component=controller).

A notable design choice is the use of the cert-controller (from OPA) for webhook certificate management instead of cert-manager. The operator generates its own CA and server certificates, stores them in a Secret (`kubeflow-trainer-webhook-cert`), and patches the ValidatingWebhookConfiguration's `caBundle` field. This avoids a hard dependency on cert-manager but means the operator manages its own PKI. The certificate readiness is gated via a channel (`certsReady`) that blocks controller and webhook setup until certs are available, with the readiness probe also waiting on cert readiness to prevent premature traffic.

The RHOAI runtime landscape ships 15 ClusterTrainingRuntimes covering three accelerator types (CUDA, ROCm, CPU) across two runtime families: `torch-distributed` (lightweight PyTorch distributed) and `training-hub` (interactive training environments). Images are parameterized through kustomize replacements from `params.env`, allowing platform operators to pin image digests per release. The ImageStream resources (`manifests/rhoai/imagestreams/`) provide OpenShift-native image management for the training-hub workbench images with multi-version tag support (2026.1 and 2026.2 tags).
