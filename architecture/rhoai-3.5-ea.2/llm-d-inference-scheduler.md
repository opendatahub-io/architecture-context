## Purpose

**Short**: An inference request scheduler that makes optimized routing decisions for LLM inference workloads via the Kubernetes Gateway API Inference Extension (GIE) ext-proc protocol.

**Detailed**: The llm-d Inference Scheduler provides two deployable components: the **Endpoint Picker (EPP)** and the **Routing Sidecar**. The EPP extends the Gateway API Inference Extension project to serve as the control plane for inference request routing. It integrates with an Envoy-based gateway via the ext-proc (External Processing) protocol, receiving incoming inference requests and making real-time scheduling decisions based on pluggable filters, scorers, and profile handlers. Scheduling considers KV cache locality, session affinity, load balancing, model compatibility, and context length to select optimal model-serving pods.

The **Routing Sidecar** (pd-sidecar) runs alongside vLLM model-serving pods and implements disaggregated inference protocols. It supports Prefill/Decode (P/D), Encoder/Prefill/Decode (E/P/D), and Data Parallel execution patterns through pluggable KV transfer connectors (NIXL v2, Shared Storage, SGLang). The sidecar intercepts OpenAI-compatible API requests and orchestrates multi-stage inference across specialized pods (prefill, decode, encode), enabling efficient KV cache transfer and reduced time-to-first-token.

Together, these components enable scalable, multi-model inference serving on Kubernetes with advanced scheduling algorithms, disaggregated execution, and deep integration with the Gateway API ecosystem.

## Architectural Analysis

The llm-d Inference Scheduler is a sophisticated two-component system that bridges the Kubernetes Gateway API ecosystem with advanced LLM inference optimization. Its architecture reflects several notable design patterns and trade-offs.

**Plugin-Based Extensibility**: The EPP implements a Kubernetes-style plugin framework where filters, scorers, pickers, and profile handlers are registered via factory functions and instantiated from YAML configuration (`EndpointPickerConfig`). This mirrors the scheduler framework pattern from `kube-scheduler`, enabling new scheduling algorithms to be added without modifying core code. The system supports 20+ plugin types including precise prefix cache scoring, context-length-aware routing, session affinity, load balancing, and disaggregation profile handlers. The `register.go` file in `pkg/epp/framework/plugins/` serves as the central plugin registry. Feature gates (`prepareDataPlugins`, `experimentalFlowControlLayer`, `dataLayer`) enable progressive rollout of experimental capabilities.

**Disaggregation Architecture**: The sidecar component implements a protocol-pluggable disaggregation layer supporting three distinct KV transfer mechanisms: NIXL v2 (explicit two-stage with KV cache parameter passing), Shared Storage (cache-first with decode-early optimization), and SGLang (concurrent prefill-decode with bootstrap-based KV sharing). The EPD (Encoder/Prefill/Decode) extension adds multimodal content processing with fan-out to encoder pods. Each protocol connector is isolated in its own file (`connector_*.go`) with a common interface, allowing new transfer protocols to be added. The sidecar carefully handles streaming SSE responses, buffering initial chunks to detect cache threshold finish reasons before switching to passthrough mode.

**Data Layer Architecture**: The EPP's data collection system uses a source-extractor pattern where `PollingDataSource` plugins periodically poll endpoints (e.g., scraping Prometheus metrics from vLLM) and `Extractor` plugins transform raw data into endpoint attributes (concurrency, latency, prefix cache state). The `Collector` manages per-endpoint goroutines with ticker-based polling (default 50ms intervals). For KV cache tracking, the `precise-prefix-cache-scorer` can either opportunistically subscribe to ZMQ events during scoring (legacy path) or receive endpoint lifecycle events from the data layer (recommended path), with automatic switchover on first `ExtractEndpoint` call.

**SSRF Protection**: The sidecar implements an allowlist-based SSRF protection mechanism that watches InferencePool resources and maintains a dynamic set of approved pod IPs and names. Prefill and encoder targets specified in request headers (`x-prefill-endpoints`, `x-encoder-endpoints`) are validated against this allowlist before requests are forwarded. This is critical because the EPP sets these headers based on scheduling decisions, and the sidecar must verify that the targets are legitimate pool members rather than arbitrary endpoints an attacker might inject.

**FIPS Asymmetry**: The two Konflux Dockerfiles show different FIPS treatment. The sidecar builds with `-tags strictfipsruntime` and `CGO_ENABLED=1`, indicating full FIPS crypto compliance. The EPP builds with `CGO_ENABLED=1` (required for check-payload) but does not include the `strictfipsruntime` build tag, meaning it uses OpenSSL-backed crypto but may not enforce FIPS-only cipher suites at the Go level. The EPP's self-signed certificate generation (`internal/tls/tls.go`) uses RSA 4096-bit keys but does not set `tls.MinVersion`, defaulting to TLS 1.0+. The sidecar explicitly sets `MinVersion: tls.VersionTLS12` with a curated list of ECDHE cipher suites. The model server metrics scraping defaults to `InsecureSkipVerify: true`, which is acceptable for pod-to-pod communication but should be documented as a known relaxation.
