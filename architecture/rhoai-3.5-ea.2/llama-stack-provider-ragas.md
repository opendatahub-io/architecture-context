## Purpose

**Short**: An out-of-tree Llama Stack evaluation provider that implements Ragas (Retrieval-Augmented Generation Assessment) metrics for RAG quality evaluation, with both inline (in-process) and remote (Kubeflow Pipelines) execution modes.

**Detailed**: The llama-stack-provider-ragas component implements the Ragas evaluation framework as a pluggable provider for Meta's Llama Stack platform. It exposes Ragas metrics (answer relevancy, context precision, faithfulness, context recall) through Llama Stack's standard Eval API, allowing users to evaluate RAG pipeline quality using Llama Stack's built-in inference, dataset, and benchmark APIs.

The provider ships in two execution modes. The **inline provider** runs Ragas evaluations in the same process as the Llama Stack server, wrapping the server's inference API (for LLM completions and embeddings) into Ragas-compatible LLM and embedding interfaces via LangChain adapters. The **remote provider** offloads evaluation to Kubeflow Pipelines (Data Science Pipelines), constructing a two-step KFP pipeline that retrieves data from Llama Stack and runs Ragas evaluation in a separate container, with results stored in S3 and fetched back on job completion.

The component is built by Konflux using a UBI 9 Python 3.12 base image and deployed as part of the TrustyAI evaluation stack within RHOAI. It integrates with the TrustyAI Service Operator, which provides the container image reference via a ConfigMap (`trustyai-service-operator-config`), and with Kubeflow Pipelines for remote execution orchestration.

## Architectural Analysis

The llama-stack-provider-ragas component follows a clean plugin architecture that separates evaluation logic from the Llama Stack server framework. The provider implements two protocol interfaces (`Eval` and `BenchmarksProtocolPrivate`) and registers itself via `ProviderSpec` objects, allowing the Llama Stack framework to discover and load it at startup. The `compat.py` module provides a notable backward-compatibility layer, using try/except import chains to support both the legacy `llama_stack` package layout and the newer `llama_stack_api` module structure. This dual-import pattern suggests the project is tracking rapid upstream API evolution.

The inline and remote providers share a common configuration base (`RagasProviderBaseConfig`) but diverge significantly in execution model. The inline provider wraps the Llama Stack inference API using LangChain-compatible adapters (`LlamaStackInlineLLM` extends `BaseRagasLLM`, `LlamaStackInlineEmbeddings` extends `BaseRagasEmbeddings`), bridging the async Llama Stack API with Ragas's synchronous-first design via `asyncio.to_thread()` for the main `ragas.evaluate()` call. The remote provider takes a fundamentally different approach: it constructs a Kubeflow Pipelines DAG with two steps (data retrieval and evaluation), submits it to the DSP API, and provides asynchronous job tracking with S3-based result persistence. The remote wrappers (`LlamaStackRemoteLLM`, `LlamaStackRemoteEmbeddings`) use the `llama_stack_client` HTTP client instead of in-process API calls, enabling execution in isolated KFP containers.

The base image resolution strategy for KFP pipeline components (`components.py:get_base_image()`) reveals an important integration pattern: it first checks the `KUBEFLOW_BASE_IMAGE` environment variable, then falls back to reading a `trustyai-service-operator-config` ConfigMap from either `redhat-ods-applications` or `opendatahub` namespaces, and finally defaults to `quay.io/trustyai/llama-stack-provider-ragas:latest`. This three-tier resolution chain ties the component to the TrustyAI operator's lifecycle management. The candidate namespace list (`KUBEFLOW_CANDIDATE_NAMESPACES` in `constants.py`) hardcodes both RHOAI and ODH namespace conventions.

A security consideration is the KFP authentication flow: when no `KUBEFLOW_PIPELINES_TOKEN` is provided, the remote provider loads the kubeconfig and extracts the bearer token from `api_key["authorization"]`. This means the KFP client inherits whatever OpenShift token the pod (or user) has, with no token refresh mechanism. The code explicitly notes this gap with `TODO: handle expired token issues`. Additionally, the KFP healthz pre-flight check (`ragas_remote_eval.py:86-93`) uses `requests.get()` with default TLS settings, which is correct for FIPS compliance on UBI but does not validate server certificates beyond the system trust store.

The current metric set is limited to four Ragas metrics (answer_relevancy, context_precision, faithfulness, context_recall), with commented-out code in `constants.py` indicating planned expansion to answer_correctness, factual_correctness, summarization_score, BLEU, and ROUGE. The `Dockerfile.konflux` uses a digest-pinned UBI 9 Python 3.12 base image but does not use AIPCC base images, which is a gap for product security compliance even though this component has no GPU/accelerator dependencies.
