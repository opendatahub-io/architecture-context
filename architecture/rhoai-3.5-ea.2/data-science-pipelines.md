## Purpose

**Short**: Data Science Pipelines provides a platform for defining, deploying, and managing reproducible machine learning workflows on Kubernetes using Argo Workflows as the execution engine.

**Detailed**: Data Science Pipelines (DSP) is the Red Hat downstream distribution of Kubeflow Pipelines (KFP) v2, providing a complete ML pipeline orchestration platform. It enables data scientists to define multi-step ML workflows as directed acyclic graphs (DAGs), compile them into Argo Workflow specifications, and execute them on Kubernetes clusters with full metadata tracking, artifact management, and execution caching.

The platform consists of several cooperating services: an API server that exposes gRPC and REST endpoints for pipeline management, a v2 execution engine (driver/launcher) that orchestrates individual pipeline steps within Argo Workflow pods, controllers for scheduled workflows and viewer instances, a persistence agent that synchronizes workflow state back to the API server, a cache server that deduplicates identical executions, and a React frontend with a Node.js BFF (backend-for-frontend) server. All execution metadata is recorded in ML Metadata (MLMD), and pipeline artifacts are stored in S3-compatible object storage (MinIO/SeaweedFS).

In the RHOAI platform, DSP is deployed and managed by the DataSciencePipelinesApplication (DSPA) custom resource through the data-science-pipelines-operator (DSPO). The operator handles provisioning of all DSP sub-components, database connections, object storage configuration, and integration with the platform's authentication and networking infrastructure.

### ml-pipeline-launcher

**Dockerfile**: `backend/Dockerfile.konflux.launcher`

## Architectural Analysis

Data Science Pipelines implements a distributed pipeline orchestration architecture with a clear separation between control plane (API server, controllers, persistence agent) and data plane (driver, launcher running inside Argo Workflow pods). The control plane manages pipeline lifecycle and metadata, while the data plane handles actual step execution. This separation means the control plane services are long-running deployments, while data plane components are ephemeral containers created per pipeline step by Argo.

The v2 execution engine uses a three-layer orchestration model (compiler → driver → launcher) that is architecturally significant. The compiler translates KFP pipeline specs to Argo Workflow YAML at submission time. The driver runs as an Argo task to orchestrate each step: it resolves inputs from MLMD (not from Argo parameters), checks the execution cache, provisions artifact URIs, and generates a pod spec patch. The launcher then executes user code, manages artifact I/O, and records results. This metadata-driven execution model (inputs resolved from MLMD rather than passed as workflow parameters) enables sophisticated features like caching, lineage tracking, and loop fan-out, but creates a tight dependency on the MLMD service.

The cache system operates at two levels. The v1 cache (cache-server webhook) intercepts pod creation and replaces expensive containers with lightweight busybox when a matching cache entry exists in MySQL. The v2 cache operates within the driver, querying the API server's cache service by execution fingerprint before generating a pod spec patch. This dual-cache approach supports both legacy v1 pipelines and the newer v2 execution model.

The FIPS compliance story is noteworthy: four of the five Konflux-built Go binaries use `GOEXPERIMENT=strictfipsruntime` with `CGO_ENABLED=1` for dynamic linking against OpenSSL, and `go.mod` declares `godebug fips140=on`. However, the launcher ships dual binaries — a static non-FIPS build (`launcher-v2`) and a dynamic FIPS build (`launcher-v2-fips`) — because the launcher runs inside user pipeline pods where FIPS may not be required. The DSPO operator selects which binary to use based on cluster FIPS configuration. This dual-binary pattern is an intentional design choice to support both FIPS-enforced clusters and non-FIPS development environments.

The multi-user security model relies heavily on Istio service mesh. Istio AuthorizationPolicies restrict inter-service communication to specific service account principals, and DestinationRules enforce mTLS between services. The API server performs its own authentication via HTTP header extraction or Kubernetes TokenReview, followed by SubjectAccessReview for RBAC authorization. This creates a defense-in-depth model where Istio handles transport-level auth and the API server handles application-level auth. Without Istio (single-user mode), the API server operates without enforced authentication.
