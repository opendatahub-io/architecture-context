## Purpose

**Short**: The RHOAI/ODH web dashboard providing a unified management UI for data science workloads, model serving, notebooks, pipelines, and AI/ML platform features.

**Detailed**: odh-dashboard is a multi-component monorepo that produces the web-based management console for Red Hat OpenShift AI (RHOAI) and Open Data Hub (ODH). The architecture consists of three tiers: (1) a React/PatternFly 6 frontend host application using Webpack Module Federation to dynamically load feature-specific plugin modules at runtime, (2) a Node.js/Fastify backend-for-frontend (BFF) that serves the static frontend, proxies Kubernetes API calls, and routes traffic to per-module Go BFF sidecars, and (3) a Go controller-runtime operator (`dashboard-operator`) that reconciles a `Dashboard` custom resource to deploy and manage the dashboard and its modules via kustomize manifests.

The dashboard supports a modular architecture where feature domains (GenAI, Model Registry, MaaS, MLflow, AutoML, AutoRAG, EvalHub, AgentOps) are implemented as independently deployable Module Federation plugins. Each plugin consists of a TypeScript frontend (exposing `./extensions` and `./extension-points` via Module Federation) and a Go BFF server that proxies domain-specific API calls to upstream platform services. In the sidecar deployment mode, all BFF containers run as sidecars in the main dashboard pod behind kube-rbac-proxy authentication. In standalone mode, each module runs as its own Deployment with dedicated Service, NetworkPolicy, and RBAC resources. The dashboard-operator manages module lifecycle, resolving inter-module dependencies and DSC component availability to determine which modules should be deployed.

## Architectural Analysis

The odh-dashboard repository represents one of the most architecturally complex components in the RHOAI platform, implementing a full micro-frontend architecture with a novel sidecar-based BFF pattern. The design reflects a deliberate evolution from a monolithic dashboard toward a modular, independently deployable plugin system.

**Module Federation as an Extensibility Mechanism**: The frontend uses Webpack Module Federation with a host/remote pattern where the main dashboard (`host`) dynamically loads plugin modules at runtime via `@module-federation/runtime.loadRemote()`. Each plugin exports `./extensions` and `./extension-points` that register into a centralized `PluginStore` (from `@openshift/dynamic-plugin-sdk`). This allows new features to be added without rebuilding the host application. Extension points include routes (`app.route`), navigation items (`app.navigation/href`), status providers (`app.status-provider`), and feature area flags (`app.area`). The extension filtering respects feature flags, enabling progressive feature rollout tied to DSC component availability.

**Dual Deployment Modes**: The modular architecture supports two deployment topologies. In **sidecar mode** (the default for RHOAI), all BFF containers run as sidecars within the main `odh-dashboard` pod. The Node.js backend proxies `/_mf/{name}/*` requests to the appropriate sidecar via HTTPS on distinct ports (8043-8843). Service account isolation is achieved by mounting separate projected volumes — the main dashboard containers (odh-dashboard + kube-rbac-proxy) use the `odh-dashboard` SA token, while BFF sidecars use the `odh-dashboard-modules` SA token from a pre-created Secret. In **standalone mode**, each module deploys as its own Deployment+Service+NetworkPolicy+RBAC combination under `manifests/modular-architecture/modules/`, with its own service account and dedicated TLS certificate. The standalone mode has tighter NetworkPolicy — for example, the gen-ai standalone module's egress is restricted to DNS (port 5353), the MaaS pod (port 8243), the K8s API (6443), and external HTTPS (443/80), while the sidecar mode's NetworkPolicy allows all in-cluster egress.

**Inter-BFF Communication**: A notable pattern is the gen-ai BFF communicating with the maas BFF for model deployment status (RHOAIENG-49012). This is configured via environment variables (`BFF_MAAS_SERVICE_NAME`, `BFF_MAAS_SERVICE_PORT`) that differ between sidecar mode (targeting `odh-dashboard:8243`) and standalone mode (targeting `odh-dashboard-maas-ui:8243`). The gen-ai module calls the MaaS BFF using the user's forwarded access token, creating a trust chain: user → kube-rbac-proxy → dashboard backend → gen-ai BFF → maas BFF → upstream MaaS controller.

**Dashboard Operator Module Lifecycle**: The `dashboard-operator` implements a dependency-resolution algorithm in `resolveModuleStatuses()` that determines which modules should be deployed based on three inputs: (1) explicit overrides in `spec.modules` (Enabled/Disabled), (2) DSC component availability in `spec.components` (projected by the platform operator), and (3) inter-module dependencies (e.g., `mlflowEmbedded` requires `mlflow`). The operator renders kustomize manifests using the `odh-platform-utilities` library, resolving image references from `RELATED_IMAGE_*` environment variables. This enables the platform operator to pin images per-release via the CSV. The operator also deploys a singleton validation webhook that ensures only one Dashboard CR named `default-dashboard` can exist.

**Security Architecture**: Authentication flows through kube-rbac-proxy, which validates user tokens via Kubernetes TokenReview and checks authorization via SubjectAccessReview (specifically, the user must be able to list `projects` in `project.openshift.io`). Once authenticated, kube-rbac-proxy injects `X-Auth-Request-User` and `X-Auth-Request-Groups` headers and proxies to the backend. The backend's `route-security.ts` implements namespace-scoped resource access guards — ensuring users can only access notebooks and ConfigMaps named after their username (e.g., `jupyter-nb-{username}`). BFF modules use their own auth middleware (`InjectRequestIdentity`) that extracts user identity from the `x-forwarded-access-token` header and performs SubjectAccessReview checks for namespace access. The Perses observability proxy module is notable for supporting cross-namespace resource cleanup in its finalizer logic, managing resources in a separate observability namespace.
