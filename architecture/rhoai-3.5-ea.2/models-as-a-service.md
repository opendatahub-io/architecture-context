## Purpose

**Short**: Models as a Service (MaaS) provides a comprehensive platform for exposing AI/ML models as managed API endpoints with real-time policy management, subscription-based access control, token rate limiting, and API key management.

**Detailed**: MaaS consists of two primary components: a Kubernetes controller (`maas-controller`) and an API server (`maas-api`). The controller manages the lifecycle of model references, subscriptions, authentication policies, external model integrations, and tenant bootstrapping through custom resources. It reconciles these into lower-level Kuadrant AuthPolicies, TokenRateLimitPolicies, and Gateway API HTTPRoutes to enforce access control and rate limiting at the gateway layer. The API server provides an OpenAI-compatible REST interface for model discovery (`/v1/models`), API key management (create, revoke, search, validate), and subscription queries. API keys use a hash-based security model (SHA-256 with embedded salt) and are stored in PostgreSQL.

The platform integrates deeply with the RHOAI/ODH ecosystem. It relies on Gateway API for traffic routing, Kuadrant/Authorino for authentication and authorization enforcement at the gateway, Limitador for token-based rate limiting, and KServe LLMInferenceService for model serving backends. A payload-processing pipeline (Envoy ext_proc) handles request transformation, API format translation, and credential injection for external model providers (OpenAI, Anthropic, etc.). Multi-tenancy support is provided through the AITenant CRD, which bootstraps per-tenant namespaces, RBAC, and gateway references.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| maas-controller | Go Operator (controller-runtime) | Manages MaaS CRDs, reconciles Kuadrant policies, deploys maas-api via Tenant reconciler, handles multi-tenancy bootstrapping |
| maas-api | Go REST Service (Gin framework) | OpenAI-compatible API for model discovery, API key management, subscription selection, and Authorino validation callbacks |
| payload-processing | Envoy External Processor (ext_proc gRPC) | Request/response transformation pipeline: model header extraction, provider resolution, API format translation, credential injection |

## Architectural Analysis

The Models as a Service platform follows a layered architecture with clear separation between user-facing abstractions (MaaSModelRef, MaaSSubscription, MaaSAuthPolicy) and the underlying enforcement mechanisms (Kuadrant AuthPolicy, TokenRateLimitPolicy, Gateway API HTTPRoute). This abstraction layer is a deliberate design choice: platform administrators work with domain-specific MaaS CRDs while the controller translates them into the correct Kuadrant and Gateway API resources. This shields administrators from the complexity of Kuadrant's policy model while enabling the MaaS controller to implement cross-cutting concerns like governance pairing (requiring both a subscription AND an auth policy before a model is considered "Ready").

The authentication architecture is notably sophisticated. Rather than a simple token validation model, MaaS implements a dual-authentication path through Kuadrant's AuthPolicy: API keys (sk-oai-* format) are validated via HTTP callbacks to the maas-api internal endpoint, while OpenShift tokens use Kubernetes TokenReview. Both paths extract user identity metadata (username, groups, tenant) into X-MaaS-* headers that flow through the entire request pipeline. The API key security model is well-designed: keys use SHA-256 hashing with an embedded per-key salt (the key_id), constant-time comparison, fail-closed validation (empty subscription rejects), and IDOR protection (404 instead of 403). The CWE-116 mitigation for group name validation (preventing CEL injection into Authorino policies) demonstrates security-aware design.

The payload processing pipeline introduces an innovative pattern using Envoy's ext_proc protocol. Two separate ext_proc stages (pre-auth and post-auth) handle different responsibilities: the pre-auth stage extracts the model name from the request body into an HTTP header (enabling gateway-level routing decisions before authentication), while the post-auth stage handles API format translation and credential injection for external providers. This allows MaaS to proxy requests to external LLM providers (OpenAI, Anthropic) through the same gateway endpoint, transparently injecting provider-specific API keys from Kubernetes Secrets.

The multi-tenancy model is evolving through two mechanisms. The original Tenant CR is a namespace-scoped singleton ("default-tenant") that drives the platform reconcile via kustomize. The newer AITenant CR provides a higher-level bootstrap abstraction: creating an AITenant automatically provisions a tenant namespace, RBAC, gateway references, and deploys a tenant-scoped maas-api instance. The controller supports cluster-wide Tenant watching when tenant namespace discovery is enabled, allowing AITenant-created tenants in arbitrary namespaces. The TenantReconciler's kustomize pipeline is notable: it builds deployment manifests from embedded kustomize bases at runtime, applying image overrides, namespace transformations, and label injection before server-side applying the rendered resources.

A potential architectural risk is the model access validation probe pattern. When listing models, maas-api performs per-model HTTP probes through the gateway to validate whether the user can actually access each model's /v1/models endpoint. While this provides accurate authorization filtering, it introduces latency proportional to the number of models and creates a dependency on gateway availability for the API response. The 15-second timeout and fail-closed behavior are appropriate safety measures, but large deployments with many models could see degraded /v1/models response times.
