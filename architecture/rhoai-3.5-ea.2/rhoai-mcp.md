## Purpose

**Short**: MCP (Model Context Protocol) server that enables AI agents to interact with Red Hat OpenShift AI environments for managing workbenches, model serving, pipelines, training, and data connections.

**Detailed**: The RHOAI MCP server is a Python service implementing the Model Context Protocol (MCP) specification. It exposes a comprehensive set of tools, resources, and prompts that allow AI agents (such as Claude, GPT, or Llama-based agents) to manage the full ML lifecycle on OpenShift AI clusters. The server uses a pluggy-based plugin architecture where each RHOAI domain (notebooks, inference, pipelines, training, connections, storage, model registry, quickstarts) is a self-contained plugin that registers its own MCP tools and resources.

The server connects to the Kubernetes API using the official Python kubernetes client and provides both stdio and HTTP-based transports (SSE, streamable-http) for MCP communication. It supports multiple authentication modes: kubeconfig-based, bearer token, and in-cluster service account. When deployed in multi-user scenarios, it supports OIDC authentication with JWT validation or OpenShift TokenReview, and enforces per-user RBAC filtering on tool visibility using Kubernetes SubjectAccessReview.

The MCP server acts as a bridge between AI agent conversations and the OpenShift AI platform, translating natural-language-driven tool calls into Kubernetes API operations. It includes composite tools that optimize for AI agent context windows by providing compact cluster and project summaries, and workflow guidance prompts that help agents navigate multi-step operations like model training and deployment.

## Architectural Analysis

The RHOAI MCP server demonstrates a well-structured plugin architecture built on pluggy hooks. Each RHOAI domain (notebooks, inference, pipelines, training, connections, storage, model registry, quickstarts) is encapsulated as an independent plugin that declares its metadata, CRD requirements, tool registrations, and RBAC permission mappings. This design enables graceful degradation: if a domain's required CRDs are not installed on the cluster (e.g., no KServe means no inference tools), the plugin is marked unhealthy but the server continues operating with remaining plugins. The plugin system also supports external extensions via Python entry points (`rhoai_mcp.plugins` group), allowing third-party packages to register additional MCP tools.

The authentication architecture is notably sophisticated for a service at this stage (v0.1). It supports three layers: (1) single-user mode where the server's service account identity is used for all K8s operations, (2) OIDC JWT mode where user tokens are validated against a JWKS endpoint and the server impersonates the user for K8s API calls, and (3) OpenShift TokenReview mode where opaque OAuth tokens are validated via the K8s TokenReview API. In multi-user modes, per-tool RBAC filtering is implemented by mapping each MCP tool to the Kubernetes permissions it requires (e.g., `list_workbenches` requires `get` on `kubeflow.org/notebooks`), then checking those permissions via SubjectAccessReview for the authenticated user. This ensures that users only see tools they have permission to use. The implementation handles the ASGI middleware challenge carefully — it uses pure ASGI wrapping instead of Starlette's `BaseHTTPMiddleware` because the latter is incompatible with SSE streaming.

The composite tools layer is an interesting design choice for AI agent ergonomics. Rather than requiring agents to make many individual MCP tool calls (which consume context window tokens), composite tools like `cluster_summary` and `project_summary` aggregate multiple domain queries into compact responses. The meta tools (`suggest_tools`, `list_tool_categories`) provide intent-based tool discovery, helping agents find the right tool without scanning all available tools. The training domain has a unified action-based tool (`training`) that routes to specific operations via an `action` parameter, reducing the cognitive load on agents. Workflow tokens (HMAC-SHA256 signed) implement multi-step confirmation flows for dangerous operations, ensuring agents must explicitly confirm before destructive actions proceed.

The deployment architecture uses Kustomize with a clear base/overlay pattern. The RHOAI overlay notably enables read-only mode by default, reflecting that in production RHOAI deployments the MCP server should observe rather than modify the cluster. The image parameterization via `params.env` and `configMapGenerator` replacements is designed to integrate with the RHOAI operator's `ApplyParams` mechanism, which substitutes `RELATED_IMAGE` values from the CSV for disconnected/air-gapped deployments. The NetworkPolicy in the OpenShift overlay specifically allows the MCP server to reach the model-catalog service on ports 8080/8443/443, indicating that cross-namespace communication to the model registries namespace is expected. A notable gap is the absence of a Konflux Dockerfile — the current `Containerfile` uses standard UBI9 but lacks the Konflux build pipeline integration needed for RHOAI product shipment.
