## Purpose

**Short**: Kubernetes operator that deploys and manages OGX (Open GenAI Stack) server workloads as first-class Kubernetes resources via the OGXServer CRD.

**Detailed**: The OGX Kubernetes Operator manages the full lifecycle of OGX server instances on Kubernetes and OpenShift clusters. It provides a declarative API (the `OGXServer` CRD) that abstracts the complexity of deploying, configuring, and operating OGX-based GenAI stack servers. Users specify their desired distribution, AI provider integrations (inference, vector I/O, tool runtimes, file processing, etc.), storage backends, and network configuration in a single custom resource, and the operator reconciles the underlying Kubernetes infrastructure — Deployments, Services, PVCs, NetworkPolicies, HPAs, PDBs, and Ingresses — to match.

The operator includes a sophisticated configuration generation pipeline that expands typed CRD provider specifications into OGX server configuration YAML. It resolves base configurations from OCI image labels or user-provided ConfigMaps, merges user-specified providers and storage backends, and produces content-hashed immutable ConfigMaps for zero-downtime config rollouts. The operator also manages CA bundle aggregation with X.509 validation, supports legacy resource adoption from the predecessor LlamaStackDistribution CRD, and integrates with OpenShift-specific features like security context constraints and service-serving certificates.

The repository additionally contains an `ogx-module` sub-project that defines the ODH/RHOAI platform component CRD (`OGX` at `components.platform.opendatahub.io`), enabling the platform operator (`rhods-operator` or `opendatahub-operator`) to manage the OGX operator as a first-class platform component. This module is currently scaffolded with the reconciler to be added in a later task.

## Architectural Analysis

The OGX operator follows a clean single-CRD controller pattern, where the `OGXServer` custom resource serves as the sole declarative input and the operator reconciles a predictable set of Kubernetes resources. This architectural simplicity — one CRD, one controller, one reconcile loop — makes the operator straightforward to understand and debug, while the sophistication lies in the configuration generation pipeline and the kustomize-based manifest rendering system.

The configuration generation pipeline is the operator's most distinctive architectural feature. Rather than mounting a static ConfigMap, the operator implements a multi-stage pipeline: it resolves a base configuration from OCI image labels (or a user-provided ConfigMap), expands typed provider specifications into OGX configuration YAML, merges user overrides, and produces content-hashed immutable ConfigMaps. This content-hashing strategy enables zero-downtime config rollouts — each unique configuration gets its own ConfigMap, the Deployment template annotation changes trigger a rolling update, and old ConfigMaps are retained during rollout for in-flight ReplicaSets. The retention logic (keep 2 latest + any referenced by current Deployment or in-flight ReplicaSets) is carefully designed to prevent premature cleanup during rollouts.

The kustomize plugin architecture deserves attention. Instead of generating complete manifests programmatically, the operator maintains base YAML templates in `controllers/manifests/base/` and applies a 5-stage Go plugin chain (Name Prefix → Namespace → Field Mutator → NetworkPolicy Transformer → Autoscaling Handler) to customize them per instance. This approach provides a clear separation between resource structure (YAML templates) and instance-specific values (plugin transformations), making it easier to add new resources without modifying Go code. The Field Mutator plugin uses RFC 6901 JSON Pointer syntax, which is declarative and auditable.

The legacy adoption system for migrating from `LlamaStackDistribution` to `OGXServer` is well-designed for safety. It uses annotations (`ogx.io/adopt-storage`, `ogx.io/adopt-networking`) to explicitly opt in to migration, scales down the legacy Deployment to zero before detaching PVCs, transfers Service selectors to point at the new pods, and leaves an audit trail via labels and annotations. The adoption is idempotent at every stage and can be cleanly reversed by removing the annotation, which triggers cleanup of adopted resources.

The dual-Dockerfile setup reveals an interesting FIPS compliance gap. The upstream `Dockerfile` has comprehensive FIPS support with `GOEXPERIMENT=strictfipsruntime`, `-tags=strictfipsruntime,openssl`, and explicit OpenSSL installation. However, the `Dockerfile.konflux` used for RHOAI builds sets only `CGO_ENABLED=1` without the FIPS build tags, relying on the UBI9 Go toolset's implicit FIPS behavior. While this likely works in practice (UBI9 Go toolsets are built with FIPS support), the lack of explicit FIPS flags in the Konflux build is a notable gap compared to the upstream Dockerfile's explicit approach. The OLM FIPS annotation (`features.operators.openshift.io/fips-compliant: "true"`) is present, asserting FIPS compliance to the platform.

The `ogx-module` sub-project represents a forward-looking architectural decision. By defining a cluster-scoped `OGX` CRD at `components.platform.opendatahub.io`, the operator follows the ODH/RHOAI component onboarding pattern that allows the platform operator to manage OGX as a first-class component with standardized lifecycle management (install, upgrade, health reporting). The module is currently scaffolded — the reconciler stub exists but delegates no work — indicating this is an in-progress integration.
