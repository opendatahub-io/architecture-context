## Purpose

**Short**: Platform operator that orchestrates the entire Red Hat OpenShift AI (RHOAI) platform — managing component lifecycle, ingress infrastructure, authentication, monitoring, and cloud provider dependencies.

**Detailed**: The rhods-operator (also known as opendatahub-operator in its upstream incarnation) is the central control plane for the RHOAI platform. It implements a multi-tier CRD hierarchy where a `DataScienceCluster` (DSC) resource orchestrates 16+ component operators (Dashboard, KServe, Kueue, Ray, TrustyAI, ModelRegistry, etc.) and a `DSCInitialization` (DSCI) resource configures platform-wide services including authentication, monitoring, and trusted CA bundle propagation.

The operator deploys and manages the platform's full ingress stack via the Gateway controller, which creates Gateway API resources (Gateway CR, HTTPRoute CRs), an Envoy proxy data plane, EnvoyFilter CRs for authentication integration, kube-auth-proxy deployments for OAuth/OIDC authentication, and OpenShift Routes for legacy URL redirects. This ingress architecture replaced the RHOAI 2.x Route-based pattern.

A separate `cloudmanager` binary handles multi-cloud deployments (AWS, Azure, CoreWeave) by managing infrastructure dependencies — cert-manager, LeaderWorkerSet operator, Sail/Istio operator, and Gateway API — via Helm charts. The operator also includes a module system for out-of-tree component operators, enabling extensibility without modifying the core codebase.

### Ingress Traffic Flow

The RHOAI 3.x ingress stack operates as a multi-layer chain:

1. **External client** → **Gateway CR** (port 443, TLS termination) — Gateway API listener with TLS certificate
2. **Gateway (Envoy data plane)** → **EnvoyFilter** (data-science-authn-filter) — ext_authz check against kube-auth-proxy
3. **EnvoyFilter ext_authz** → **kube-auth-proxy** (port 8443, HTTPS) — OAuth2/OIDC token validation
4. **EnvoyFilter Lua filter** — strips OAuth cookies, sets Authorization: Bearer header, forwards x-forwarded-access-token
5. **Gateway (Envoy)** → **Component HTTPRoutes** → **Component Services** — routed by path/host to component backends

**Two ingress modes supported:**
- **LoadBalancer** (default): Gateway exposes external LoadBalancer service, hostname listeners route traffic
- **OcpRoute**: Gateway uses ClusterIP service, OpenShift Route with re-encrypt TLS handles external exposure

**Authentication modes:**
- **IntegratedOAuth** (default on OpenShift): Creates OAuthClient `data-science`, uses `oauth2-proxy --provider=openshift`
- **OIDC** (external): Uses external OIDC provider via `oauth2-proxy --provider=oidc --oidc-issuer-url=...`
- **None**: No auth proxy deployed (external auth assumed)

**Dashboard redirect infrastructure:**
- Nginx-based Deployment+Service+Route serves 301 redirects from legacy dashboard hostname to new Gateway hostname
- Legacy gateway Route redirects old `data-science-gateway.{domain}` to new `rh-ai.{domain}`

## Architectural Analysis

The rhods-operator exemplifies a **hierarchical operator pattern** with three distinct tiers of control. At the top, the DSCInitialization controller establishes platform-wide infrastructure — namespaces, networking, authentication, and monitoring. The DataScienceCluster controller sits one level below, translating a single user-facing CR into 16+ component CRs that each have their own dedicated controllers. Each component controller manages its own reconciliation lifecycle (kustomize rendering, deployment, status aggregation) while inheriting platform context (gateway domain, OIDC configuration, admin groups) from the parent controllers. This design allows components to be developed and tested independently while maintaining consistent platform behavior.

The **Gateway controller** is the architectural centerpiece for networking. Rather than relying on static Route manifests, it dynamically constructs a full ingress stack at reconciliation time: a Gateway CR for the Envoy data plane, an EnvoyFilter for ext_authz authentication enforcement, a kube-auth-proxy Deployment for OAuth2/OIDC session management, and supporting resources (HPA, NetworkPolicy, DestinationRule, ServiceAccount, ClusterRoleBinding). The Lua filter in the EnvoyFilter deserves special attention — it sits in the response path after ext_authz, stripping OAuth2 proxy cookies and injecting a Bearer token into the Authorization header, effectively converting a cookie-based auth session into a token-based one before traffic reaches component backends. This ensures components see a clean Bearer token regardless of whether the user authenticated via OAuth or OIDC.

The **cloud manager** binary demonstrates a clean provider-abstraction pattern for multi-cloud support. All three providers (AWS, Azure, CoreWeave) share identical reconciliation pipelines — BuildHelmCharts → certmanager.Bootstrap → ReconcileAction → GCAction — with provider-specific behavior isolated to the CRD types and Helm chart values. The hook system (PreApply/PostApply) handles operational edge cases: SkipCRDIfPresent prevents Server-Side Apply conflicts when Prometheus Operator already owns the ServiceMonitor CRD, and AnnotateIstioWebhooksHook works around a sail-operator infinite reconciliation loop (OSSM-12397). The label-based cache filtering strategy (infrastructure.opendatahub.io/part-of) is notable — since dependency namespaces are configurable per CR, the cache watches all namespaces but filters by label rather than pre-selecting namespaces.

The **module system** introduces a fourth tier for out-of-tree component operators that are deployed as separate Helm charts or Kustomize manifests. Unlike the built-in component controllers, modules are provisioned through a generic handler interface that builds a module CR, renders the operator manifests, and aggregates status back into the DSC. The two-phase cleanup (delete CR first to trigger finalizers, then delete operator resources) prevents resource leaks when modules are disabled. This extensibility mechanism allows new components to be added to the platform without modifying the core operator code.

A key **FIPS compliance** observation: the Gateway controller reads the OpenShift APIServer's TLS security profile and floors it to Intermediate if the configured profile would result in unsupported TLS versions or ciphers. The OpenSSL-to-IANA cipher suite conversion in `gateway_tls.go` explicitly logs dropped ciphers (DHE variants not supported by Go's crypto/tls), ensuring administrators can trace any cipher suite gaps. However, the DestinationRule for kube-auth-proxy uses `insecureSkipVerify: true` because the service-CA generated certificate is not in the Istio trust store — this is a known trade-off for in-mesh communication.
