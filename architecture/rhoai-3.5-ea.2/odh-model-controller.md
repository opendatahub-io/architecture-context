## Purpose

**Short**: Kubernetes operator and API server that manages the lifecycle of KServe model serving workloads, NVIDIA NIM accounts, LLM inference services, and Gateway API authentication on Red Hat OpenShift AI.

**Detailed**: The odh-model-controller is a controller-runtime based Kubernetes operator that orchestrates the auxiliary infrastructure required for KServe model serving on the RHOAI platform. It watches InferenceService, ServingRuntime, LLMInferenceService, InferenceGraph, and NIM Account custom resources and dynamically creates supporting Kubernetes resources including OpenShift Routes, NetworkPolicies, RBAC bindings, Prometheus ServiceMonitors/PodMonitors, KEDA TriggerAuthentications, Kuadrant AuthPolicies, and Istio EnvoyFilters.

The controller provides three distinct functional domains: (1) traditional KServe inference service lifecycle management with OpenShift Route exposure, metrics collection, and autoscaling; (2) LLM-specific inference service management with Gateway API integration, Kuadrant/Authorino-based authentication, and EnvoyFilter TLS bootstrap; and (3) NVIDIA NIM account management with API key validation, runtime template generation, and model catalog synchronization. A companion API server (`model-serving-api`) exposes a Gateway discovery endpoint that allows the dashboard to list available Gateway API gateways for LLM inference service routing.

The operator also manages infrastructure-level concerns including CA certificate bundle aggregation for model storage, storage credential secret composition, and Ray TLS certificate lifecycle for multi-node distributed inference workloads.

## Architectural Analysis

The odh-model-controller exhibits a well-structured multi-layer reconciliation architecture. The top-level controllers (InferenceServiceReconciler, LLMInferenceServiceReconciler, GatewayReconciler) delegate to specialized sub-reconcilers via the `KserveRawInferenceServiceReconciler` orchestrator, which manages six independent sub-reconcilers for Routes, RBAC, metrics Services, ServiceMonitors, dashboards, and KEDA. This delegation pattern keeps individual reconcilers focused and testable, though it creates a deep call chain where the InferenceService reconciler's success depends on all six sub-reconcilers completing without error. The delta processor pattern (comparing desired vs. deployed state via typed comparators) prevents unnecessary API calls and reduces reconciliation chatter.

The controller manages two distinct ingress patterns reflecting the RHOAI 2.x-to-3.x migration: (1) OpenShift Routes for traditional KServe raw deployment InferenceServices, and (2) Gateway API with Kuadrant/Authorino for LLMInferenceService. These coexist in the same binary but are cleanly separated -- the `serving/` package handles Routes while the `serving/llm/` package handles Gateway API resources. The GatewayReconciler creates EnvoyFilters for Authorino TLS bootstrap and AuthPolicies for Gateway-level authentication, with a "last one wins" merge semantic for gateway references from LLMInferenceService and LLMInferenceServiceConfig specs. The controller uses conditional CRD availability checking (`IsCRDAvailable`) so it degrades gracefully when optional dependencies (KEDA, Kuadrant, Authorino) are not installed.

The NIM integration is architecturally isolated in the `controller/nim/` and `webhook/nim/` packages with its own CRD (Account), finalizer-based cleanup, and external HTTP clients for NVIDIA NGC API communication. The singleton-per-namespace webhook constraint and 24-hour validation/refresh cycles prevent API key abuse and stale configuration. The template-based ServingRuntime generation includes GPU resource specifications (2x GPU, 4-8Gi memory), volume mounts for model cache and shared memory, and Prometheus metrics configuration -- all computed dynamically from the NGC catalog.

The model-serving-api server is a lightweight, security-hardened companion service with FIPS-compliant TLS (explicit cipher suites, TLS 1.2+ minimum, P-256/P-384 curves), zero-downtime certificate rotation via fsnotify, and SelfSubjectAccessReview-based authorization. Its single `/api/v1/gateways` endpoint performs impersonation-style access checks: it uses the caller's bearer token to verify `create llminferenceservices` permission before listing gateways, ensuring namespace-scoped access control. The Gateway listener filtering logic correctly implements Gateway API's `AllowedRoutes.Namespaces` semantics (NamespacesFromAll, NamespacesFromSame, NamespacesFromSelector).

A notable security consideration is the Ray TLS certificate management. The controller generates a self-signed CA (4096-bit RSA, 5-year validity) and per-pod certificates (3-year validity) with SAN entries for pod IPs and `*.{namespace}.svc.cluster.local`. Certificate rotation is triggered by pod IP changes, and the PodReconciler uses optimistic locking with retry to handle concurrent updates to the shared ray-tls Secret. The webhook injects an init container that waits up to 4 minutes (24 retries at 10-second intervals) for certificates to appear, which could delay pod startup for multi-node workloads. The init container image is sourced from a ConfigMap (`ray-tls-generator-image`), which is a mutable reference -- a supply chain consideration worth noting.
