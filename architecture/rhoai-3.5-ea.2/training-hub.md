## Purpose

**Short**: A unified Python SDK providing a common interface for multiple fine-tuning algorithms (SFT, OSFT, LoRA, GRPO, GEPA) used in Red Hat OpenShift AI training workflows.

**Detailed**: Training Hub is a Python library that abstracts multiple model fine-tuning strategies behind a consistent API. It implements a strategy pattern where each training algorithm (SFT, OSFT, LoRA SFT, LoRA GRPO/GRPO, GEPA) can have one or more pluggable backend implementations. Users call simple top-level functions like `sft()`, `osft()`, `lora_sft()`, `lora_grpo()`, `grpo()`, or `gepa()` and the library handles parameter validation, backend selection, distributed training configuration (torchrun), and execution.

The library serves as the training layer for RHOAI, integrating several external training frameworks: `instructlab-training` for SFT, `rhai-innovation-mini-trainer` for OSFT, Unsloth/TRL for LoRA, OpenPipe ART and verl for GRPO reinforcement learning, and the `gepa` library for genetic-Pareto prompt optimization. It also provides profiling utilities (GPU memory estimation, training time estimation), loss curve visualization, experiment tracking integration (W&B, MLflow, TensorBoard), and a coding agent plugin for Claude Code and Codex CLI.

Training Hub is published on PyPI as `training-hub` and is designed to be used both as a library in Python scripts and as a standalone tool via its convenience functions. It does not deploy any services or containers itself -- it is invoked by users or by higher-level RHOAI components that orchestrate training jobs.

## Architectural Analysis

### Strategy Pattern with Algorithm Registry

Training Hub's core architectural pattern is a two-level strategy pattern. The `AlgorithmRegistry` class maintains class-level dictionaries `_algorithms` (mapping algorithm names like `'sft'` to `Algorithm` subclasses) and `_backends` (mapping `(algorithm_name, backend_name)` tuples to `Backend` subclasses). Each algorithm class declares its required and optional parameters with types, enabling runtime validation. The `create_algorithm()` factory function instantiates the appropriate algorithm with its selected backend.

This design enables several key capabilities:
1. **Backend pluggability**: The same algorithm can have multiple backend implementations (e.g., GRPO with ART or verl backends, GEPA with gepa or MLflow backends)
2. **Parameter isolation**: Each algorithm declares its own parameter schema; the registry validates inputs before passing them to backends
3. **Consistent public API**: All algorithms are invoked through simple top-level functions that delegate to the registry

### PEFT Composition via Extender Pattern

The `LoRAPEFTExtender` class provides a reusable set of LoRA/QLoRA parameters (rank, alpha, dropout, quantization config) that can be composed with any base algorithm. `LoRASFTAlgorithm` and `LoRAGRPOAlgorithm` both use this extender to augment their parameter schemas. This avoids duplicating LoRA parameter definitions across algorithms while keeping the parameter sets type-safe.

### Subprocess Isolation for Training Backends

Several backends execute training in isolated subprocesses rather than in-process:
- **ART backend** (`lora_grpo.py`): Spawns ART training as a subprocess for GPU isolation, creating a seed LoRA checkpoint for vLLM V1 compatibility
- **verl backend** (`lora_grpo_verl.py`): Invokes `verl.trainer.main_ppo` via subprocess with Hydra configuration, serializing reward functions to files for the subprocess to load
- **SFT/OSFT backends**: Use torchrun for distributed training, which itself manages worker processes

This subprocess pattern provides GPU memory isolation and process-level fault containment, but introduces complexity around parameter serialization (pickling reward functions, writing Parquet data files, Hydra config generation).

### Distributed Training Configuration

The `get_torchrun_params()` utility in `utils.py` implements a three-level parameter precedence hierarchy (function args > environment variables > defaults) for PyTorch distributed training. It handles mutual exclusivity between `master_addr` and `rdzv_endpoint`, validates `nproc_per_node` values, and checks for conflicting environment variables. This centralizes distributed training configuration across all algorithms that use torchrun.

### GRPO Design: ART vs verl Tradeoffs

The GRPO implementation offers two backends with distinct scaling characteristics:
- **ART** (OpenPipe): Single-GPU, time-shared between training and inference. Simpler setup, supports multi-turn trace decomposition, but limited by single-GPU memory.
- **verl**: Multi-GPU with FSDP. Supports Dr. GRPO (no reference model, token-level loss normalization), handles large models, but requires more complex distributed setup.

The `grpo()` convenience function defaults to verl with `lora_r=0` (full fine-tuning), while `lora_grpo()` defaults to verl with LoRA enabled. The ART backend can be explicitly selected for single-GPU scenarios.

### GEPA: Prompt Optimization Without Weight Changes

GEPA (Genetic-Pareto) is architecturally distinct from other algorithms -- it optimizes prompts, not model weights. It uses evolutionary search to find optimal system prompts given evaluation criteria. The MLflow backend variant integrates with MLflow's prompt registry for versioning optimized prompts. This makes GEPA the only algorithm that doesn't require GPU access for the optimization itself (only for the evaluation LLM endpoint).

### Supply Chain Security: litellm Version Exclusions

The dependency specification for litellm explicitly excludes versions 1.82.7 and 1.82.8 (`!=1.82.7,!=1.82.8`), which were compromised in the TeamPCP supply chain attack on 2026-03-24. This is documented as a conscious security decision in the pyproject.toml.

### VLM (Vision Language Model) Detection in LoRA

The Unsloth LoRA backend includes architecture-based VLM detection by checking if the model architecture class name contains `ForConditionalGeneration` (VLM) vs `ForCausalLM` (text-only). VLM models use `FastModel` instead of `FastLanguageModel` from Unsloth and require different dataset processing (passthrough format instead of chat template formatting).
