## Purpose

**Short**: TrustyAI is a Quarkus-based REST service providing AI fairness metrics, bias tracking, model drift detection, and explainability algorithms for models served via KServe and ModelMesh on OpenShift AI.

**Detailed**: TrustyAI Explainability is a Java library and containerized service that provides Explainable AI (XAI) capabilities for the Red Hat OpenShift AI platform. The service integrates with KServe (v1 and v2 protocols, both HTTP and gRPC) and ModelMesh inference servers to intercept inference payloads, reconcile input/output pairs, and compute fairness and drift metrics over the collected data.

The service exposes a comprehensive REST API for computing fairness metrics (Statistical Parity Difference, Disparate Impact Ratio), drift detection (Meanshift, FourierMMD), and local/global explainability algorithms (LIME, SHAP, Counterfactual, TSSaliency, Partial Dependence Plots). Metrics can be computed on-demand or scheduled for periodic recalculation, with results published as Prometheus gauges for monitoring integration.

The codebase is organized as a Maven multi-module project: `explainability-core` provides the XAI algorithm library, `explainability-connectors` implements KServe v1/v2 protocol support (including gRPC via protobuf), `explainability-arrow` provides Apache Arrow interop for TrustyAI-Python communication, and `explainability-service` is the Quarkus application that ties everything together. Data can be persisted to PVC (CSV files), MinIO (S3-compatible object storage), or a relational database (MariaDB/MySQL via Hibernate ORM). In RHOAI deployments, the explainer endpoints (LIME, SHAP, Counterfactual, TSSaliency, PDP) are disabled by profile configuration — only fairness and drift metrics are enabled.

## Architectural Analysis

The TrustyAI Explainability service follows a layered architecture with clean separation between the XAI algorithm library (`explainability-core`), protocol adapters (`explainability-connectors`), data interchange (`explainability-arrow`), and the service layer (`explainability-service`). This design enables the core algorithms to be used as a standalone Java library (published to Maven Central) independently of the Quarkus service wrapper. The `explainability-connectors` module encapsulates all KServe protocol complexity — supporting both v1 (simple JSON with `instances`) and v2 (tensor-based with named inputs/outputs, both HTTP and gRPC) — behind a `PredictionProvider` abstraction that the core algorithms consume.

The inference payload reconciliation system is a critical architectural element. Both ModelMesh and KServe send inference inputs and outputs as separate, asynchronous events. The `InferencePayloadReconciler` pattern uses synchronized ID-based matching to pair inputs with outputs, then converts the matched pair into a `Dataframe` row for persistence. The ModelMesh reconciler processes Base64-encoded protobuf payloads and supports synthetic data detection (via `BIAS_IGNORE_PARAM`), while the KServe reconciler handles JSON payloads via CloudEvents. This reconciliation happens in memory (or optionally in Hibernate) before final persistence, which provides resilience against out-of-order delivery.

The storage abstraction is notably flexible — four backends (MEMORY, PVC, MINIO, DATABASE) selected via a single configuration property with Quarkus conditional bean injection (`@LookupIfProperty` / `@LookupUnlessProperty`). However, the MinIO backend is significantly less capable than PVC or DATABASE — it lacks batch reading, data slicing, and tag/ID-based filtering. The DATABASE mode (MariaDB/MySQL via Hibernate) is the most feature-complete, supporting fine-grained queries, tag-based filtering, and transactional writes with 50-row statement batching. The PVC mode uses direct `FileChannel` I/O for CSV files, which is efficient but limits query flexibility.

The RHOAI profile configuration is architecturally significant: it disables all explainer endpoints (LIME, SHAP, Counterfactual, TSSaliency, PDP) while keeping fairness metrics (SPD, DIR) and drift detection (Meanshift, FourierMMD) enabled. This means the gRPC client to model servers is effectively unused in RHOAI deployments since it's only needed for explainer computations. The feature gating uses a custom `@EndpointDisabled` annotation checked via `@IfBuildProperty` style conditions, allowing endpoints to be toggled at build/deploy time without code changes.

The service's security posture relies entirely on platform-level enforcement rather than application-level authentication. No endpoint implements authentication — the expectation is that the TrustyAI Service Operator deploys kube-rbac-proxy sidecars (RHOAI 3.x pattern) or oauth-proxy sidecars (RHOAI 2.x pattern) in front of the service. The ClusterRole grants broad ConfigMap permissions (create, delete, get, list, patch, update, watch) cluster-wide via a ClusterRoleBinding, which is used by the init container to write the `model-serving-config` ConfigMap. This is a wide permission scope that could be tightened to namespace-level RoleBindings in future iterations.
