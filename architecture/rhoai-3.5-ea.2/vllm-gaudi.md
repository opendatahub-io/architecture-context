## Purpose

**Short**: Hardware plugin that enables vLLM large language model inference on Intel Gaudi (Habana) accelerators.

**Detailed**: vllm-gaudi is the Intel Gaudi hardware plugin for vLLM, the high-throughput LLM inference engine. It implements the vLLM hardware plugin interface to provide HPU-specific execution, attention, memory management, and model optimizations for Intel Gaudi 2 and Gaudi 3 accelerators. The plugin registers itself via Python entry points (`vllm.platform_plugins`, `vllm.general_plugins`) and overrides vLLM's default GPU implementations with Habana-optimized equivalents.

The component ships as a container image (`odh-vllm-gaudi-rhel9`) based on UBI 9.6 with Intel Synapse AI runtime 1.23.0 and PyTorch 2.9.0. At runtime, it provides the OpenAI-compatible API server (`vllm.entrypoints.openai.api_server`) for model inference. The plugin supports advanced inference features including speculative decoding (Eagle3/MTP), unified attention, FP8 quantization (INC, compressed-tensors, ModelOpt), LoRA adapters, KV cache offloading to CPU, NIXL-based KV transfer for disaggregated inference, and distributed inference via tensor parallelism, data parallelism, and expert parallelism using HCCL.

Within RHOAI, vllm-gaudi is deployed as a ServingRuntime container by KServe when users create InferenceService resources targeting Intel Gaudi hardware. It is one of several vLLM accelerator variants (alongside CUDA and ROCm builds) that enable model serving across different hardware backends.

## Architectural Analysis

The vllm-gaudi plugin follows vLLM's hardware plugin architecture (RFC #11162 and #19161), cleanly separating hardware-specific code from the core inference engine. The plugin registers four types of extensions: a platform plugin (`HpuPlatform`), utility plugins, custom operation plugins, and model plugins. This modular design allows the Gaudi-specific code to evolve independently from upstream vLLM while maintaining API compatibility through the plugin interface.

The attention subsystem is the most architecturally complex part of the plugin. It implements four attention backends — HPUAttentionV1 (standard), HPUUnifiedAttention (merged prefill/decode), HPUMLAAttention (Multi-Head Latent Attention for DeepSeek-style models), and HPUUnifiedMLA. The unified attention backend is a significant optimization that merges prefill and decode phases into a single forward pass with shared and unique KV cache blocks, reducing graph recompilation overhead. The selection between backends is driven by runtime configuration (`get_config().unified_attn`) and model capabilities (`use_mla`), with the platform's `get_attn_backend_cls` method serving as the dispatch point (platform.py:44-67).

The bucketing system (`vllm_gaudi/extension/bucketing/`) is architecturally critical for Gaudi performance. Unlike GPUs that use CUDA graphs with fixed shapes, Gaudi's lazy compilation model requires pre-warming graphs for specific input shapes. The bucketing manager generates prompt, decode, and unified buckets using configurable strategies (exponential, linear, or file-based), and dynamically creates fallback buckets for unseen sizes using cubic-root scaling. This system directly impacts both startup time (warmup phase) and runtime latency (graph cache hit rate). The choice between exponential and linear strategies is hardware-dependent, with exponential being the default for Gaudi 3.

The build architecture has a notable FIPS compliance gap: the Dockerfile explicitly removes the OpenSSL FIPS provider (`rpm -e --nodeps openssl-fips-provider-so`) to work around repository compatibility issues with CentOS/EPEL packages needed for system dependencies like `numactl-devel` and `libfdt-devel`. This means the image cannot operate in FIPS mode on FIPS-enabled clusters. Additionally, the build is not hermetic (`hermetic: false` in the Tekton pipeline), performing live downloads from Habana Artifactory, PyPI, and GitHub during image construction. Both issues are expected to be addressed as the component matures — the AIPCC Gaudi base image (currently disabled per AIPCC-3471) would resolve both by providing a pre-built, FIPS-compliant base with secure package indexes.

The distributed inference architecture supports three parallelism modes: tensor parallelism (TP) for model sharding across HPUs, data parallelism (DP) for batch replication, and expert parallelism (EP) for MoE model expert distribution. Communication uses HCCL (Habana's collective library) via PyTorch distributed, with a custom `HpuCommunicator` that inserts `htcore.mark_step()` calls to work around a Habana bridge bug with lazy collectives (hpu_communicator.py:42). The NIXL connector (`hpu_nixl_connector.py`) enables disaggregated inference by transferring KV cache blocks between prefill and decode nodes, with a custom `_hpu_data_ptr` function that bridges HPU virtual addresses for the NIXL memory registration protocol.

The model override system registers 11 HPU-optimized model implementations that replace upstream vLLM models at runtime. Key optimizations include: replacing complex-number rotary embeddings with real-number equivalents (pixtral), using `htcore.mark_step()` synchronization points for vision encoders (deepseek_ocr), vision bucket managers to reduce graph recompilation (gemma3_mm), and in-place class patching to upgrade MoE blocks for 3D tensor handling (qwen3_moe). The pattern of patching upstream model classes rather than fully re-implementing them keeps the codebase compact but creates tight coupling to specific upstream vLLM versions.
