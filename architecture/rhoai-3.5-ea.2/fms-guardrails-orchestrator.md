## Purpose

**Short**: A REST API orchestrator that coordinates AI text generation with configurable content safety guardrails, acting as middleware between clients and language model backends.

**Detailed**: The FMS Guardrails Orchestrator (internally named `fms-guardrails-orchestr8`) is a high-performance Rust service that sits between AI application clients and downstream LLM generation servers, content detector services, and text chunker services. It receives text generation or detection requests via REST APIs, orchestrates parallel calls to configured detector and chunker services to evaluate content safety, and conditionally forwards generation requests to LLM backends (TGIS, caikit-nlp, or OpenAI-compatible servers like vLLM).

The orchestrator supports multiple API versions: v1 endpoints for legacy classification-with-generation workflows, and v2 endpoints for content detection, chat detection, context document detection, generation-with-detection, and streaming content detection. It also provides OpenAI-compatible `/v1/chat/completions` and `/v1/completions` proxy endpoints with integrated guardrails detection. The service runs a dual-server architecture with a main guardrails API server (default port 8033) and a separate health check server (default port 8034), supports TLS and mTLS for both inbound and outbound connections, and exports OpenTelemetry traces and metrics via OTLP (gRPC or HTTP).

## Architectural Analysis

The FMS Guardrails Orchestrator follows a well-defined orchestration pattern where a single stateless service coordinates multiple downstream AI services. The architecture is built around a `ClientMap` abstraction (`src/clients.rs:136`) that provides a type-safe heterogeneous map of client implementations (DetectorClient, GenerationClient, ChunkerClient, OpenAiClient). Each client type implements the `Client` trait with a `health()` method, enabling the orchestrator to report health status of all downstream services through the `/info` endpoint. The `GenerationClient` uses a strategy pattern internally, wrapping either a `TgisClient` or `NlpClient` based on configuration.

The dual-server architecture is a notable design choice. The health server runs on a separate port (8034) without TLS, ensuring health probes from Kubernetes can always reach the service regardless of TLS configuration on the main API server (8033). The main server supports optional TLS and mTLS using `rustls` with the `ring` crypto backend, implemented via a custom low-level `serve_with_tls` function (`src/server/tls.rs:69-142`) that wraps `hyper`'s connection handling with `TlsAcceptor`. This allows the server to handle both TLS and non-TLS configurations from the same binary.

The handler architecture uses a trait-based dispatch pattern (`Handle<Task>` in `src/orchestrator/handlers.rs:42-46`) where each API endpoint creates a typed task struct and submits it to the `Orchestrator::handle()` method. This allows the orchestrator to remain agnostic about specific handler implementations while maintaining type safety. Streaming responses use SSE (Server-Sent Events) for v1/v2 endpoints and ND-JSON for the streaming content detection endpoint, both built on `axum`'s streaming response primitives.

A significant architectural concern is the TLS implementation. While OpenSSL is installed in the container image to pass check-payload validation, all actual TLS operations use `rustls` with the `ring` crypto backend, which is not FIPS-validated. The `NoVerifier` implementation in `src/utils/tls.rs:44-91` provides an insecure mode that bypasses all certificate verification, which should only be used in development environments. The service supports per-downstream-service TLS configuration via named TLS configs in `config.yaml`, allowing different cert/key/CA combinations for each detector, chunker, and generation service.

The configuration system (`src/config.rs`) demonstrates a flexible service discovery model. Each downstream service is configured with hostname, port, optional path prefix (for services behind reverse proxies or routed through llm-d deployments — see commit 95b3b4d), and optional TLS configuration. The `api_token` field supports environment variable indirection via `from_env` deserialization, allowing secrets to be injected at runtime. Header passthrough and the `rewrite_forwarded_access_header` feature enable integration with authentication proxies like kube-rbac-proxy, where `X-Forwarded-Access-Token` headers can be rewritten to standard `Authorization: Bearer` headers before being forwarded to downstream services.
