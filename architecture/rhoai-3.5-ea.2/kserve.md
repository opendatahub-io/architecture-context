## Purpose

**Short**: KServe is a Kubernetes-native model serving platform that provides CRDs and controllers for deploying, scaling, and managing machine learning inference services on OpenShift.

**Detailed**: KServe manages the full lifecycle of ML model inference services through multiple controllers. The primary InferenceService (v1beta1) controller handles traditional model serving with support for predictor, transformer, and explainer components across multiple deployment modes (raw Kubernetes, Knative serverless, and ModelMesh). The LLMInferenceService (v1alpha2) controller is purpose-built for large language model deployments, supporting disaggregated serving (separate prefill/decode), multi-node tensor/pipeline parallelism, LoRA adapter management, Gateway API-based routing with Inference Gateway Extension (EPP scheduler), and Workload Variant Autoscaler (WVA) integration.

Additional controllers manage InferenceGraphs for multi-model DAG-based inference pipelines, LocalModel caching for node-local model storage with deduplication, and TrainedModel resources for multi-model server configuration. The kserve-module sub-project provides platform-level integration with OpenDataHub/RHOAI, reconciling a Kserve CR that deploys all KServe components via kustomize manifests with dependency checking for Istio, cert-manager, LeaderWorkerSet, and Custom Metrics Autoscaler operators.

## Architectural Analysis

KServe's architecture exhibits a clear evolutionary pattern: the original InferenceService (v1beta1) controller was designed for traditional ML model serving with a component-based abstraction (predictor/transformer/explainer), while the newer LLMInferenceService (v1alpha2) controller was purpose-built for large language model workloads with fundamentally different operational requirements. The two controllers share the same repository and Go module but run as separate binaries (`cmd/manager` and `cmd/llmisvc`), each with its own webhook server, leader election, and cache configuration. This separation allows independent scaling and failure isolation.

The distribution-aware build system is a notable architectural decision. Go build tags (`distro` vs `!distro`) create compile-time divergence between upstream KServe and RHOAI builds. ODH-specific files (`*_odh.go`) add OpenShift Route support, OAuth proxy integration, ODH-specific RBAC markers, and platform-specific behaviors like SELinux permission-fix Jobs for local model caching. This pattern avoids runtime feature flags but increases the number of source files that must be maintained in parallel. The `kserve-module` sub-project further extends this by providing an ODH platform integration layer that reconciles a Kserve CR and deploys all components via kustomize, with dependency checking for Istio, cert-manager, and other operators.

The ingress strategy supports three generations of routing: Istio VirtualService (Knative mode), Kubernetes Ingress (legacy raw mode), and Gateway API HTTPRoute (modern raw mode and LLMInferenceService). The LLMInferenceService controller adds Inference Gateway Extension support with EPP (Endpoint Picker) scheduler integration, creating InferencePool CRs that enable LLM-optimized request routing based on KV cache utilization and queue depth rather than simple round-robin. This represents a significant architectural advancement over traditional Kubernetes load balancing for inference workloads.

The local model caching subsystem (LocalModelCache, LocalModelNamespaceCache, LocalModelNode, LocalModelNodeGroup) introduces a distributed storage layer that pre-downloads models to node-local storage. This uses a deduplication strategy based on SHA-256 hashing of storage URIs, ensuring that multiple CRs referencing the same model share a single download. The cluster-scoped vs namespace-scoped split allows both platform-wide model sharing and tenant-isolated caching. The system creates download Jobs using the storage-initializer image, maintaining consistency between init-container and pre-cache download paths.

FIPS compliance is enforced at build time for all Go binaries except the kserve-module-controller, which is built with `CGO_ENABLED=0` (statically linked without FIPS crypto). This is architecturally acceptable since the module controller only manages kustomize manifests and does not handle cryptographic operations or sensitive data. The storage-initializer Python component relies on the UBI9 system OpenSSL for FIPS compliance, which is inherited from the base image rather than explicitly configured. The `cryptography>=46.0.5` pinning in the Python dependencies addresses CVE-2026-26007 but the component would benefit from migration to AIPCC base images for consistent FIPS enforcement.
