## Purpose

**Short**: A lightweight HTTP server that detects PII and custom patterns in text using regex matching, serving as a detector backend for the FMS Guardrails Orchestrator.

**Detailed**: The Guardrails Regex Detector is a Rust-based microservice within the RHOAI TrustyAI / Guardrails stack. It receives text content via a REST API and applies regex-based pattern matching to detect personally identifiable information (PII) such as email addresses, Social Security Numbers (SSNs), and credit card numbers. It also supports arbitrary custom regex patterns supplied at request time.

The service is designed to be called by the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator), which routes text content to various detector backends (including this one) for content analysis and guardrailing. The orchestrator invokes this service's `/api/v1/text/contents` endpoint with a list of text strings and a list of regex identifiers (either built-in names like `"email"`, `"ssn"`, `"credit-card"`, or arbitrary regex patterns). The detector returns all matches with character offsets, detection type, and a confidence score (always 1.0 for regex matches).

The service is intentionally minimal — a single-binary Rust HTTP server using the Axum framework with no persistent state, no database connections, and no external service dependencies beyond the orchestrator that calls it. It is deployed as a standalone container on UBI 9 Minimal.

## Architectural Analysis

The Guardrails Regex Detector is an intentionally minimal service — roughly 200 lines of Rust across two source files. Its architecture reflects a design philosophy where each detector in the Guardrails stack is a standalone microservice with a single responsibility. This service handles regex-based pattern detection, while other detectors (HAP, toxicity, etc.) handle different detection modalities. The FMS Guardrails Orchestrator coordinates among them.

The service uses the Axum web framework with Tokio's multi-threaded async runtime, which provides excellent performance characteristics for I/O-bound HTTP workloads. The regex crate (which uses finite automaton-based matching rather than backtracking) ensures that pattern matching is guaranteed to complete in linear time, avoiding ReDoS (Regular Expression Denial of Service) vulnerabilities that plague backtracking regex engines. This is a significant security property for a service that accepts user-supplied regex patterns.

The built-in detector registry (`email`, `ssn`, `credit-card`) is hardcoded as a `HashMap` of function pointers in `handle_text_contents`. There is no plugin system, configuration file, or dynamic detector loading — adding new built-in patterns requires code changes and redeployment. This is a reasonable trade-off for a service of this scope, but it means the set of "named" detectors is fixed at build time. Custom regex patterns can still be supplied at request time as arbitrary strings, providing extensibility without code changes.

A notable architectural gap is the complete absence of TLS or authentication. The service listens on plaintext HTTP with no auth mechanism. This is typical for microservices in a Kubernetes environment where network-level isolation (NetworkPolicies) and service mesh (mTLS via Istio/Envoy) provide transport security. However, the repository contains no Kubernetes manifests, Helm charts, or kustomize overlays — it is purely the application binary. The deployment topology (Service, NetworkPolicy, sidecar injection) is likely defined by the Guardrails Orchestrator's deployment manifests or the platform operator.

The absence of a `Dockerfile.konflux` is notable for a RHOAI 3.5 component. The standard `Dockerfile` uses `rust:1.84.0` as the builder (a Debian-based image from Docker Hub) and `registry.access.redhat.com/ubi9/ubi-minimal` as the runtime base. The runtime image installs `compat-openssl11` and `shadow-utils`, but the Rust binary itself does not link against OpenSSL. The `compat-openssl11` package may be present as a system dependency for other runtime needs or as a forward-looking compliance measure. The Dockerfile also lacks a `USER` directive, meaning the container runs as root by default — a security concern that should be addressed.
