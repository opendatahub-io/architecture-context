## Purpose

**Short**: Provides a container image containing pre-built model performance benchmark data (metadata, evaluations, performance metrics, validations, and variant groups) that is mounted as an init container in the model-catalog deployment.

**Detailed**: The models-perf-benchmark-data component is a data-only container image that packages AI/ML model benchmark data for consumption by the RHOAI Model Catalog. The container holds structured JSON and NDJSON files organized by model provider and model ID, covering model metadata, performance benchmarks (latency, throughput, tokens-per-second), evaluation scores, Red Hat validation status, and variant group mappings.

The repository contains three functional layers: (1) a **Konflux-built data container** (`Dockerfile.konflux`) based on UBI 9 minimal that simply copies a `data/` directory into the image and runs `sleep infinity` — it serves purely as a volume source when deployed as an init container; (2) an **api-client** that fetches benchmark data from a REST API (the model-validation v1 API) and writes it to the `data/` directory in the format expected by the container; and (3) a **kafka-processor** that consumes CDC (Change Data Capture) events from Kafka topics and produces the same `data/` directory structure. Both the api-client and kafka-processor are developer/CI tools used to populate the data directory before the container image is built — they are not shipped in the Konflux image.

The component integrates with the Model Registry ecosystem: the built container image is referenced as an init container in the `model-catalog` Deployment (in the `odh-model-registries` namespace), where it copies benchmark data into a shared volume that the model-catalog service reads at startup.

## Architectural Analysis

The models-perf-benchmark-data component follows an unusual but effective pattern for Kubernetes data delivery: it packages static data files into a container image and deploys them as an init container that populates a shared volume. This pattern avoids the need for a persistent data service or ConfigMap (which would hit size limits with the volume of benchmark data involved). The data is essentially "baked" into the container image at build time, making deployments deterministic and rollbacks trivial — reverting to a previous image tag restores the exact data set.

The repository implements a dual-ingestion architecture with two independent tools (api-client and kafka-processor) that produce identical output directory structures. The `validate-parity.sh` and `find-missing-records.sh` scripts exist specifically to verify that both ingestion paths produce equivalent data, accounting for known differences (e.g., the REST API returns `[]` where Kafka sometimes sends `null` for empty arrays, and the REST API includes fields like `modelcar_size_gb` that have no Kafka equivalent). This dual-path design provides resilience — if the Kafka pipeline has issues, the REST API can serve as a fallback source, and vice versa.

The data schema covers six entity types: providers, models, evaluations (benchmark scores), performances (latency/throughput metrics), validations (Red Hat certification status including RHOAI version, vLLM version, and RHAIIS version), and variant groups. Post-processing transforms are applied after ingestion: `OverallAccuracyTransform` computes an average accuracy score from all evaluation records, and `HardwareConfigurationTransform` combines hardware type and count into a human-readable string (e.g., "NVIDIA A100 x 4"). The transform framework is extensible via the `Transform` abstract base class.

The kafka-processor supports both a modern CDC envelope format (with `op`, `data`, `submitted_at` fields) and a legacy Debezium format (with `schema`, `payload`, and `payload.after` fields), handling the transition between data pipeline generations. Entity type detection is performed through a priority cascade: Kafka topic name mapping, explicit `entity_type` field, and finally heuristic field detection based on the presence of characteristic fields (e.g., `ttft_mean` indicates a performance record).

A notable security consideration is that the README explicitly warns this repository contains **confidential data** and must remain private. The `data/` directory (which is `.gitignore`d) contains proprietary model performance benchmark results. The cold-start-vram-data enrichment script adds VRAM requirements and cold-start timing matrices, including vLLM runtime commands, to model metadata — this enrichment data is checked into the repository as JSON files.
