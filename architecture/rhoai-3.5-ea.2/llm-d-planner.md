## Purpose

**Short**: LLM deployment planning service that translates business requirements into optimized GPU configurations and Kubernetes deployment manifests.

**Detailed**: llm-d-planner is a FastAPI-based Python service that guides users from natural language descriptions of their LLM deployment needs to production-ready Kubernetes configurations. The system uses a four-stage conversational flow: (1) extract deployment intent from natural language using an LLM provider (Ollama, OpenAI, or Vertex AI), (2) map intent to traffic profiles and SLO targets using curated templates, (3) query a PostgreSQL benchmark database to find GPU configurations meeting SLO constraints and rank them across five criteria (accuracy, cost, latency, simplicity, balanced), and (4) generate KServe InferenceService YAML with HPA and ServiceMonitor manifests for one-click deployment.

The service integrates with the RHOAI Model Catalog API to synchronize validated model metadata and performance benchmarks. It also provides a Streamlit-based UI for interactive recommendation exploration, specification editing, and deployment management. A vLLM simulator component enables GPU-free development and testing by replaying benchmark data. The backend connects to Kubernetes clusters to detect available GPU types, deploy InferenceServices, and monitor deployment status, making it an end-to-end planning and deployment tool for LLM workloads on OpenShift.

## Architectural Analysis

The llm-d-planner follows a clean layered architecture with well-separated concerns. The API layer (FastAPI routes) delegates to an orchestration workflow (`RecommendationWorkflow`) that coordinates five independent engines: intent extraction, specification generation, configuration finding, scoring, and deployment generation. This pipeline architecture makes it straightforward to swap individual components — for example, replacing the Ollama LLM provider with OpenAI or Vertex AI requires only changing an environment variable, not modifying the pipeline logic. The factory pattern for LLM clients (`LLMClient` protocol + `create_llm_client()` factory) is a good design that cleanly separates provider-specific logic.

The scoring system is the architectural centerpiece. Four scoring dimensions (accuracy, price, latency, complexity) each produce a 0-100 score using different algorithms: accuracy uses use-case-specific benchmark data from Artificial Analysis, price uses a non-linear cost formula, latency uses SLO-range-capped scoring from per-use-case workload profiles, and complexity is a simple inverse of GPU count. The configurable weights (default: 40% accuracy, 40% price, 10% latency, 10% complexity) and five independent ranking views (best accuracy, lowest cost, lowest latency, simplest, balanced) give users multiple perspectives on the same configuration space. The quality scoring module (`UseCaseQualityScorer`) has a sophisticated multi-strategy model name matching system (exact → token-set → mapping table → partial → catalog fallback) that handles the messy reality of model naming conventions.

A notable risk is the deployment approach: the project uses sed-based string substitution in `deploy-all.sh` for secret injection and configuration templating. This is fragile (the script itself acknowledges it's "broken by pipes, ampersands, newlines") and should be migrated to kustomize overlays for production use. The Dockerfiles use `python:3.14-slim` (Debian-based) rather than UBI, which is a blocker for RHOAI productization — the images need migration to UBI-based or AIPCC base images for FIPS compliance and Red Hat support. The `uv.lock` file is referenced in the Dockerfile but missing from the repository, which means container builds are not reproducible from the main branch alone.

The RHOAI Model Catalog integration (`model_catalog_sync.py`) runs as a background daemon thread during application startup when `PLANNER_BENCHMARK_SOURCE=model_catalog`. This ETL pipeline fetches models and performance artifacts from the v1alpha1 REST API, transforms them into benchmark rows, and inserts them into the same PostgreSQL table used by local benchmarks. The sync uses a deterministic `config_id` hash for deduplication, which means re-syncing is idempotent. The TLS configuration for the Model Catalog client properly supports OpenShift service-serving CA certificates via a configurable CA bundle path, demonstrating awareness of the RHOAI deployment environment.
