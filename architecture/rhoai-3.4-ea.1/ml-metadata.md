## Purpose

**Short**: ML Metadata (MLMD) is a gRPC server and Python client library for recording and retrieving metadata associated with ML workflows, providing lineage tracking for artifacts, executions, and contexts.

**Detailed**: ML Metadata is a Google-originated library (upstream: `google/ml-metadata`) that provides a metadata store for tracking the provenance and lineage of ML artifacts, executions, and contexts. In the RHOAI platform, it is deployed as a standalone gRPC server (`metadata_store_server`) that acts as the metadata backbone for ML pipeline orchestration, enabling Kubeflow Pipelines and Data Science Pipelines to record which artifacts were produced by which executions, what types they conform to, and how they relate through events and lineage graphs.

The component consists of two primary layers: a C++ gRPC server binary compiled with Bazel that implements the `MetadataStoreService` protobuf service definition, and a Python client library (`ml-metadata` PyPI package) that provides both direct database access (via pybind11 C++ bindings to SQLite/MySQL/PostgreSQL) and remote gRPC client connectivity. The server supports three relational database backends — SQLite (development/testing), MySQL (production), and PostgreSQL (production) — with built-in schema migration support for upgrades and downgrades.

In the RHOAI deployment, the server is containerized on a UBI9-minimal base image and exposed on port 8080/TCP as a gRPC service. It is typically deployed alongside Data Science Pipelines components and configured via a `MetadataStoreServerConfig` protobuf file that specifies the database connection and optional TLS settings.

## Architectural Analysis

ML Metadata follows a clean layered architecture where the protobuf definitions serve as the single source of truth for both the data model and the service API. The `metadata_store.proto` file defines the core domain entities — Artifact, Execution, Context, Event, and their type hierarchies — while `metadata_store_service.proto` defines the complete RPC surface. The `metadata_source.proto` file provides the database abstraction layer with templated SQL queries, enabling the same codebase to target SQLite, MySQL, and PostgreSQL with minimal code duplication. This design allows the database backend to be swapped via configuration without any code changes.

The server's approach to type evolution is notable: it enforces backward-compatible schema changes through the `can_add_fields` and `can_omit_fields` parameters, preventing breaking changes to stored instances while allowing controlled evolution. The `ListOperationOptions` provides a sophisticated SQL-like filter query language that supports filtering on node attributes, properties, contexts, parent/child relationships, and events — making the metadata store queryable far beyond simple CRUD operations.

The Python client library has a dual-mode architecture: it can either connect directly to a local database via C++ pybind11 bindings (useful for testing and single-process usage) or connect to a remote gRPC server. Both paths implement identical semantics with retry logic for Aborted errors using exponential backoff. This dual-mode design means the same `MetadataStore` Python class works transparently in both scenarios, differing only in the `config` parameter type (`ConnectionConfig` for direct DB vs `MetadataStoreClientConfig` for gRPC).

From a security perspective, TLS is entirely optional and configuration-driven. The server does not enforce any encryption or authentication by default — security is expected to be provided by the deployment environment (e.g., network policies, service mesh, or the deploying operator). The `SSLConfig` in `MetadataStoreServerConfig` supports server-side TLS with optional mutual TLS (`client_verify`), and the database connections support SSL for MySQL (with cipher control and server certificate verification) and PostgreSQL (with sslmode, sslcert, sslkey, sslrootcert). The use of BoringSSL (bundled via Bazel) rather than system OpenSSL for the C++ server is worth noting for FIPS compliance assessments, as BoringSSL requires explicit FIPS module configuration that is not present in this build.

The Red Hat downstream fork (`red-hat-data-services/ml-metadata`) is relatively thin compared to the Google upstream — the main downstream changes are the addition of Konflux Dockerfiles (`Dockerfile.konflux`, `Dockerfile.redhat`) that build on UBI9 base images, Tekton pipeline definitions for CI, and pinned base image digests for reproducibility. The core C++ and Python source code remains upstream-aligned, with the only substantive patch being the ZetaSQL commit adjustment in `Dockerfile.redhat` for UBI9 build compatibility.
