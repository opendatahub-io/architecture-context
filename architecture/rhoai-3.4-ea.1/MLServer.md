## Purpose

**Short**: MLServer is a Python-based ML inference server implementing the KServe V2 Inference Protocol, providing multi-model serving over REST, gRPC, and Kafka with pluggable runtime backends for multiple ML frameworks.

**Detailed**: MLServer serves as the runtime inference engine for machine learning models deployed on Red Hat OpenShift AI (RHOAI). It implements the KServe V2 Dataplane specification, enabling standardized model serving through both REST (HTTP/1.1 via FastAPI/Uvicorn) and gRPC interfaces, with an optional Kafka consumer/producer for event-driven inference. The server supports multi-model serving, where multiple models with independent version management can be loaded simultaneously in a single process.

The server provides a pluggable runtime architecture supporting 10 ML frameworks — Scikit-Learn, XGBoost, LightGBM, ONNX Runtime, CatBoost, MLflow, HuggingFace Transformers, Alibi-Detect, Alibi-Explain, and Apache Spark MLlib — each implemented as a separate Python package extending a common `MLModel` base class. Advanced inference features include adaptive batching (grouping concurrent requests for throughput), response caching (FIFO eviction for repeated queries), parallel multi-process inference (bypassing the Python GIL), and streaming inference via Server-Sent Events and bidirectional gRPC streams.

Within the RHOAI platform, MLServer is deployed as a container image (`odh-mlserver-rhel9`) that serves as an alternative inference runtime to Triton Inference Server. It is typically deployed as a KServe InferenceService pod, where models are mounted at `/mnt/models` and the server auto-discovers and loads them on startup. The RHOAI distribution ships with a curated subset of runtimes (SKLearn, XGBoost, LightGBM) pre-installed in the container image, with ONNX Runtime added in the rhai6 release.

## Architectural Analysis

MLServer follows a clean **multi-protocol gateway** pattern where a single DataPlane handler implements all inference logic, and separate REST, gRPC, and Kafka servers act as protocol-specific frontends translating between their native formats and the internal `InferenceRequest`/`InferenceResponse` types. This design ensures consistent behavior across all protocols and centralizes concerns like caching, batching, and middleware into one code path. The DataPlane is the single point where inference middleware (currently CloudEvents header injection) is applied, and where the response cache is consulted before delegating to the model.

The **parallel inference architecture** is noteworthy for its approach to the Python GIL limitation. Rather than using threading (which cannot achieve true parallelism for CPU-bound inference in CPython), MLServer spawns separate worker processes via `multiprocessing.Process`, each with its own Python interpreter, model registry, and event loop. A `Dispatcher` in the main process routes requests to workers using round-robin load balancing via `itertools.cycle`, and responses flow back through a shared `multiprocessing.Queue`. Worker failures trigger automatic restart with model reloading, and the Prometheus multiprocess collector aggregates metrics across worker PIDs by using file-backed `.db` storage. This design trades memory (each worker loads a copy of the model) for true parallel CPU inference.

The **adaptive batching** system demonstrates a well-designed producer-consumer pattern. Incoming requests are queued with individual `asyncio.Future` objects, and a batcher task accumulates requests up to `max_batch_size` or `max_batch_time` (whichever triggers first). The `BatchedRequests` class handles the complex task of merging multiple `InferenceRequest` objects into a single batch request (concatenating input tensors, tracking minibatch boundaries) and then splitting the combined response back into individual responses mapped to the correct futures. This is implemented as a hook on `model.predict()` via the model load callback system, keeping batching logic decoupled from model implementations.

The **runtime plugin architecture** uses Python class inheritance where each ML framework runtime extends the `MLModel` base class and overrides `load()`, `predict()`, and optionally `predict_stream()` and `unload()`. Runtimes are distributed as separate Python packages (`mlserver-sklearn`, `mlserver-xgboost`, etc.) that register their implementation class. The codec system provides pluggable input/output encoding — `NumpyCodec` for numerical arrays, `PandasCodec` for DataFrames, `StringCodec` for text, and framework-specific codecs like `HuggingfaceRequestCodec`. Some runtimes add custom REST endpoints via the `@custom_handler` decorator (MLflow adds `/invocations`, `/ping`, `/version`; Alibi-Explain adds `/explain`), demonstrating the extensibility of the REST layer.

A notable architectural limitation is that MLServer has **no built-in authentication or TLS**. The REST server uses insecure Uvicorn, and the gRPC server binds insecure ports. This is by design for the KServe deployment model, where the platform's ingress infrastructure (Istio mTLS, kube-rbac-proxy sidecars, or Gateway API with TLS termination) handles all security concerns external to the inference container. However, this means MLServer should never be exposed directly without platform-level security.
