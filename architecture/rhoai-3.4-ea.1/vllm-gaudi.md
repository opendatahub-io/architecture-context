## Purpose

**Short**: vLLM hardware plugin that enables large language model inference on Intel Gaudi (HPU) accelerators with an OpenAI-compatible API server.

**Detailed**: The vLLM Gaudi plugin (`vllm_gaudi`) integrates Intel Gaudi AI accelerators (Gaudi2 and Gaudi3) with the vLLM inference engine. It implements the vLLM hardware plugin interface, providing HPU-specific implementations for attention backends, model execution workers, distributed communication (via HCCL), KV cache management, and custom operation kernels. The plugin supports FP8 quantization (via Intel Neural Compressor), LoRA adapters, multi-modal models (Gemma3, Qwen2.5-VL, Qwen3-VL), speculative decoding (EAGLE), and advanced memory management including online defragmentation.

The component ships as a container image (`odh-vllm-gaudi-rhel9`) built on RHEL 9.6 UBI with Habana SynapseAI 1.23.0 runtime, PyTorch 2.9.0 (Habana-optimized), and upstream vLLM v0.14.1. The container runs as a non-root user (UID 2000) for OpenShift compatibility and exposes an OpenAI-compatible API server as its entrypoint. It supports both single-HPU and multi-HPU inference with tensor parallelism, data parallelism, and expert parallelism for Mixture-of-Experts models. The plugin also includes calibration tooling for FP8 quantization workflows using Intel Neural Compressor.

## Architectural Analysis

The vLLM-Gaudi plugin demonstrates a well-structured hardware abstraction design that cleanly separates device-specific concerns from the core vLLM inference engine. The plugin uses Python's entry_points mechanism (`vllm.platform_plugins`, `vllm.general_plugins`) to register itself without requiring modifications to upstream vLLM, allowing the same vLLM codebase to run on both NVIDIA GPUs and Intel Gaudi HPUs by simply installing the appropriate plugin package.

The extension framework (`vllm_gaudi/extension/`) is architecturally noteworthy. It implements a lazy-loading configuration system with dependency resolution — configuration values can reference other values and are only computed when accessed. The system uses predicate combinators (`All()`, `Any()`, `Not()`, `Hardware()`, `Kernel()`) to express feature availability as composable boolean logic, enabling hardware-aware feature flags that automatically adapt to Gaudi2 vs Gaudi3 capabilities. For example, fused block softmax adjustment is only enabled on Gaudi3 with SynapseAI >= 1.22.0 and the appropriate kernel available. This pattern avoids scattered `if/else` checks and centralizes hardware-dependent behavior.

The attention subsystem supports multiple implementations (flex attention, fused SDPA, naive, unified attention, MLA) selected at runtime based on hardware capabilities and model requirements. The "unified attention" mode on V1 engine merges prefill and decode operations with online defragmentation, representing a performance optimization specific to HPU's memory architecture. The bucketing system (exponential or linear) pads input sequences to fixed sizes to enable HPU graph reuse, trading memory for compilation overhead reduction — a pattern necessitated by Gaudi's graph-based execution model where each unique input shape requires a new compiled graph.

The explicit removal of the OpenSSL FIPS provider (`rpm -e --nodeps openssl-fips-provider-so`) in all Dockerfiles is a significant security architecture decision. This indicates that either the Habana SDK or its dependencies are incompatible with FIPS-enforced OpenSSL. Since the container itself does not handle TLS (plaintext HTTP on port 8000), the FIPS impact is limited to outbound connections (Hugging Face Hub model downloads), which use the system's non-FIPS OpenSSL. However, this means vLLM-Gaudi cannot be deployed on clusters requiring FIPS-enforced containers at the OS level.

The distributed communication layer wraps HCCL (Habana Collective Communication Library) through vLLM's `DeviceCommunicatorBase` interface. The `mark_step()` calls before collective operations work around a Habana PyTorch bridge bug with lazy collectives — a pragmatic workaround that reveals the tight coupling between the lazy execution model and distributed communication timing. The NIXL connector for cross-node KV transfer includes HPU-specific address translation (`_hpu_data_ptr()`) to map virtual to physical addresses, which is necessary because HPU memory management differs from CUDA's unified virtual addressing.
