## Purpose

**Short**: OpenAI-compatible batch inference gateway that implements the `/v1/file` and `/v1/batch` APIs for asynchronous LLM inference processing.

**Detailed**: The batch-gateway is an llm-d component that provides the OpenAI Batch API specification for Red Hat OpenShift AI. It enables users to submit large batches of inference requests (up to 50,000 per file, 200 MB max) as JSONL files and have them processed asynchronously against an inference backend (llm-d or GAIE).

The system consists of two independently deployable sub-components: an **API server** that handles incoming HTTP requests for file management and batch job lifecycle, and a **batch processor** that polls a Redis priority queue for pending jobs and forwards individual inference requests to the inference gateway. Both components use Redis as their primary data store for batch metadata, priority queuing, event channels, and temporary status tracking. File storage is pluggable with local filesystem and S3-compatible object store backends.

The component is in active early development (RHOAI 3.4 EA). Several features remain stubbed (authentication middleware, rate limiting, file download/streaming in the processor, event-driven cancellation), and the processor currently uses mock database clients for some operations.

## Architectural Analysis

The batch-gateway follows a clean two-tier architecture separating API handling from batch processing. The API server provides a standard OpenAI-compatible REST surface for file and batch management, while the processor is a standalone worker service that polls a Redis-backed priority queue. This separation allows independent scaling: multiple API servers can accept requests while processor instances scale to handle batch throughput. The priority queue uses Redis sorted sets with SLO timestamps as scores, enabling deadline-aware scheduling.

Multi-tenancy support is partially implemented through a configurable HTTP header (`X-MaaS-User`, default) that is extracted by middleware and propagated through the context. All database queries include tenant ID filtering, and file storage uses the tenant ID as a folder namespace. However, there is **no authentication layer enforcing tenant identity** — the middleware stubs for authentication, authorization, and rate limiting are present in the codebase but commented out (`server.go` lines 151-154). This means any client can impersonate any tenant by setting the header, which is a significant security gap for production use.

The file storage layer is well-abstracted with a pluggable `BatchFilesClient` interface supporting local filesystem (using Go 1.25's `os.Root` for path traversal prevention) and S3-compatible object storage. The local filesystem implementation uses atomic writes (temp file + rename) for crash safety. The Redis data layer uses Lua scripts for atomic operations (store, get-by-tags, get-by-expiry) and supports TLS with client certificates for production deployments.

The inference client is production-ready with exponential backoff and jitter via the resty library, configurable TLS (custom CA, mTLS, min/max TLS versions), and structured error categorization (retryable server/rate-limit errors vs. non-retryable client/auth errors). The processor uses a worker pool pattern with bounded concurrency — both at the job level (configurable `num_workers`) and within each job (configurable `max_job_concurrency` using a semaphore channel). However, the actual per-line processing pipeline is largely stubbed: file reading, line parsing, request validation, response writing, and output file assembly are all marked as TODO. The processor currently creates mock request objects rather than parsing actual JSONL input.

A notable FIPS compliance risk exists: both Konflux Dockerfiles default to `CGO_ENABLED=0`, producing statically linked binaries using Go's pure-Go crypto. This will fail OpenShift's check-payload validation for FIPS-enabled clusters. The `ARG CGO_ENABLED=0` pattern allows build-time override, but the default is non-compliant. The TLS configuration in the API server explicitly sets cipher suites (ECDHE+AES-GCM, ECDHE+ChaCha20) and MinVersion TLS 1.2, which are FIPS-appropriate ciphers, but they would use the wrong crypto backend (pure-Go vs. OpenSSL) with the current build flags.
