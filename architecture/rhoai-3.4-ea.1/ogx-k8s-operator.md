## Purpose

**Short**: Kubernetes operator that creates and manages Llama Stack server deployments on OpenShift/Kubernetes clusters.

**Detailed**: The Llama Stack K8s Operator (originally `llama-stack-k8s-operator`, downstream as `ogx-k8s-operator`) is a controller-runtime based Go operator built with Kubebuilder v4. It manages the full lifecycle of Llama Stack server instances through a custom resource called `LlamaStackDistribution` (shortname: `llsd`). When a user creates a `LlamaStackDistribution` CR, the operator reconciles a complete set of Kubernetes resources including a Deployment running a Llama Stack distribution image, a ClusterIP Service, a ServiceAccount, optional persistent storage (PVC), optional NetworkPolicy for network isolation, optional Ingress for external access, optional PodDisruptionBudget for high availability, and optional HorizontalPodAutoscaler for autoscaling.

The operator supports multiple Llama Stack distribution images (starter, remote-vllm, meta-reference-gpu, postgres-demo) which are embedded in the operator binary via `distributions.json`. For RHOAI deployments, a `RELATED_IMAGE_RH_DISTRIBUTION` environment variable overrides the embedded distribution map with a single Red Hat-certified image. The operator also provides advanced features including ConfigMap-driven user configuration with automatic rollout on change, CA bundle management with certificate validation (supporting both explicit CA bundles and auto-detected ODH trusted CA bundles), feature flags via an operator-scoped ConfigMap (`llama-stack-operator-config`), and image mapping overrides for distribution image replacement without operator redeployment.

The operator uses kustomize internally to render base manifests from embedded templates (`controllers/manifests/base/`) and applies Go-based transformer plugins to customize resource names, namespaces, field values, and NetworkPolicy ingress rules per CR instance. Status conditions (DeploymentReady, HealthCheck, StorageReady, ServiceReady) provide detailed observability, and the operator actively probes the deployed Llama Stack server's `/v1/health`, `/v1/providers`, and `/v1/version` endpoints to populate distribution config and version information in the CR status.

## Architectural Analysis

The Llama Stack K8s Operator follows a clean single-controller architecture with one reconciler managing all aspects of the `LlamaStackDistribution` lifecycle. Rather than using raw Go templates, it employs an unusual but effective pattern of rendering resources through kustomize at runtime: base manifests in `controllers/manifests/base/` are processed through kustomize's `krusty` API, then transformed by a chain of custom Go plugins (NamePrefix, Namespace, FieldMutator, NetworkPolicyTransformer). This approach decouples manifest structure from reconciliation logic and allows the base YAML to be validated and tested independently.

The ConfigMap watching system is particularly well-designed. The controller uses field indexers for efficient lookup of which `LlamaStackDistribution` CRs reference a given ConfigMap, with a graceful fallback to manual list-and-filter on older Kubernetes versions that don't support custom field labels. Content-based hashing of ConfigMap data drives pod template annotation updates, which in turn trigger rolling restarts — a standard pattern but cleanly implemented here. The operator watches both user-provided ConfigMaps (via `spec.server.userConfig`) and CA bundle ConfigMaps (via `spec.server.tlsConfig.caBundle`), plus the well-known `odh-trusted-ca-bundle` ConfigMap for automatic CA injection.

The CA bundle management is the most security-critical code path. The operator validates all PEM certificates via `x509.ParseCertificate()`, enforces size limits (10MB max, 1000 certificates max), rejects non-CERTIFICATE PEM blocks (preventing private key injection), and re-encodes validated certificates to produce a clean managed ConfigMap. Multiple CA sources (explicit user bundle + auto-detected ODH bundle) are concatenated into a single managed ConfigMap mounted at `/etc/ssl/certs/ca-bundle/ca-bundle.crt` with `SSL_CERT_FILE` set accordingly.

The operator supports upgrade path management through a `PerformUpgradeCleanup` function that removes legacy ClusterRoleBindings from previous versions (cluster-scoped to namespace-scoped migration), and a `hasLegacyCABundleVolumes` detection that switches from SSA patching to full replacement when legacy emptyDir-based CA bundle volumes are detected. The startup script embedded in `resource_helper.go` handles three generations of Llama Stack CLI interfaces (pre-0.2.17, 0.2.17-0.3.0, and 0.3.0+), using Python version detection to select the correct module path or uvicorn command.

One notable design decision is that the managed Llama Stack servers run without TLS — the `http.Client` used for health checks has no TLS configuration, and the Service is plaintext HTTP on port 8321. Network security is delegated to the optional NetworkPolicy, which restricts ingress to same-namespace pods with `app.kubernetes.io/part-of: llama-stack`, the operator namespace, explicitly allowed namespaces/labels, and OpenShift router namespaces. For external access, the Ingress resource also does not configure TLS — this is expected to be handled by the Ingress controller (e.g., OpenShift Router with edge TLS termination).
