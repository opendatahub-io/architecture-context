## Purpose

**Short**: OpenVINO Model Server is a high-performance C++ inference serving system that hosts AI models and exposes them via REST, gRPC, and OpenAI-compatible APIs.

**Detailed**: OpenVINO Model Server (OVMS) is a production-grade inference server built in C++ on top of Intel's OpenVINO toolkit. It provides model serving for a wide range of AI workloads including traditional deep learning inference, large language model (LLM) text generation with continuous batching, text embeddings, reranking, speech-to-text/text-to-speech, and image generation. OVMS implements multiple API standards: TensorFlow Serving v1, KServe v2 inference protocol, and an OpenAI-compatible v3 API for generative AI workloads.

In the RHOAI platform, OVMS functions as a KServe `ClusterServingRuntime`, enabling users to deploy models via KServe's `InferenceService` CRDs. It supports models stored locally, on S3-compatible object storage, Azure Blob Storage, Google Cloud Storage, or pulled from HuggingFace Hub. The server features a modular architecture with pluggable frontends (gRPC, HTTP/REST), a DAG pipeline scheduler for multi-model workflows, MediaPipe graph integration for generative AI calculators, and optional Python node execution. It is optimized for Intel architectures with features like LTO builds, Haswell ISA targeting, and GPU driver support.

## Architectural Analysis

OpenVINO Model Server follows a modular plugin architecture where the main `Server` singleton orchestrates a set of independently-startable modules. The startup order is carefully sequenced: Python interpreter first (if enabled), then metrics, then the servable manager and C API modules, followed by gRPC and HTTP server modules. This ensures models are loaded before network frontends begin accepting traffic. The shutdown path reverses this order, draining network connections before unloading models. Signal handlers (SIGINT, SIGTERM, SIGILL) trigger graceful shutdown through a flag-based mechanism.

The generative AI capabilities (LLM, embeddings, rerank, audio, image generation) are implemented as MediaPipe calculators rather than being hardcoded into the HTTP/gRPC frontends. This is a significant architectural choice: the HTTP frontend dispatches v3 API requests to a MediaPipe graph executor, which runs the appropriate calculator chain. Each calculator type (HttpLLMCalculator, EmbeddingsCalculatorOV, RerankCalculator, S2tCalculator, T2sCalculator, HttpImageGenCalculator) encapsulates its own request parsing, model interaction, and response serialization. This design allows new AI capabilities to be added as calculators without modifying the server core, and enables graph-based composition of processing steps.

The continuous batching implementation for LLM workloads uses a dedicated executor thread (`LLMExecutorWrapper`) that runs a tight loop calling `pipeline->step()` on the OpenVINO GenAI `ContinuousBatchingPipeline`. Requests are added to the pipeline via `GenerationHandle` objects that allow per-request result retrieval and disconnection callbacks. The scheduler supports configurable cache sizes, maximum concurrent sequences (default 256), and dynamic split-fuse batching. Speculative decoding and prompt lookup are supported as alternative decoding strategies. Tool-guided generation (function calling) is supported with model-specific parsers for Llama3, Hermes3, Mistral, Phi4, GPT-O series, and Qwen3 architectures.

A notable security consideration is that OVMS does not implement TLS natively on either its gRPC or HTTP frontends. The gRPC server explicitly uses `grpc::InsecureServerCredentials()`, and the Drogon HTTP server has no SSL configuration. This is by design for the RHOAI deployment model where TLS termination is handled by the platform's ingress stack (KServe/Istio/Envoy). However, the API key authentication mechanism only protects v3 endpoints - the v1 (TFS) and v2 (KServe) endpoints and all gRPC services have no authentication, relying entirely on network-level access control. The CORS configuration defaults to allowing all origins (`*`), all methods, and all headers, which is permissive for production deployments.

The build pipeline is non-hermetic, which is unusual for a downstream RHOAI release branch. The Tekton PipelineRun explicitly sets `hermetic: false`, and the Dockerfile clones multiple git repositories at build time using pinned commit SHAs. While SHA pinning provides reproducibility, it does not prevent supply chain attacks if upstream repositories are compromised. There are no RPM lock files, no Hermeto/cachi2 integration, and no artifact lock files. The Bazel WORKSPACE file provides some dependency pinning for C++ libraries, but external downloads still occur during the build.
