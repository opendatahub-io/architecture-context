## Purpose

**Short**: Kubernetes-native operator for managing distributed machine learning training jobs using the Kubeflow Trainer v2 API.

**Detailed**: The Kubeflow Trainer v2 operator manages the lifecycle of distributed ML training workloads on Kubernetes. It introduces a simplified two-resource model: `TrainJob` (the user's training request) and `TrainingRuntime`/`ClusterTrainingRuntime` (reusable templates that define how training is executed). The operator reconciles TrainJob resources by resolving the referenced runtime, composing a set of Kubernetes objects (primarily JobSets) through a plugin-based framework, and applying them to the cluster.

The runtime framework supports pluggable ML policies (PyTorch distributed, MPI, plain multi-node), co-scheduling integrations (Kubernetes scheduler-plugins and Volcano), and pod network configuration. Each plugin implements specific interfaces (EnforceMLPolicy, ComponentBuilder, PodNetwork, etc.) and composes sequentially to produce the final set of Kubernetes resources. The operator creates JobSet resources (from the jobset.x-k8s.io API) as the primary workload primitive, along with supporting resources like PodGroups for gang-scheduling, Secrets and ConfigMaps for MPI SSH authentication, and NetworkPolicies for pod isolation.

For RHOAI, the operator includes Red Hat AI-specific enhancements: progression tracking (polling training metrics from pods and storing progress in TrainJob annotations), NetworkPolicy generation to restrict metrics endpoint access, and pre-configured ClusterTrainingRuntime definitions for CUDA and ROCm GPU environments. The operator ships as a single controller-manager binary with built-in webhook certificate management via the OPA cert-controller.

## Architectural Analysis

The Kubeflow Trainer v2 represents a significant architectural simplification over v1. The original training-operator supported multiple framework-specific CRDs (TFJob, PyTorchJob, MPIJob, etc.), each with its own controller. Trainer v2 collapses these into a single `TrainJob` CRD with pluggable runtime definitions, achieving framework-agnostic job submission. The `TrainingRuntime` / `ClusterTrainingRuntime` resources serve as templates that encode framework-specific knowledge (PyTorch torchrun configuration, MPI hostfile generation, etc.) while the `TrainJob` remains framework-unaware. This separation of concerns lets platform operators pre-configure runtimes and users simply reference them.

The plugin framework (`pkg/runtime/framework/`) implements a Kubernetes scheduler-inspired extensibility pattern. Plugins register for specific extension points (EnforceMLPolicy, EnforcePodGroupPolicy, PodNetwork, ComponentBuilder, TrainJobStatus, CustomValidation), and the framework executes them sequentially during reconciliation. This design enables clean separation: the Torch plugin handles PyTorch-specific environment variables (PET_NNODES, PET_MASTER_ADDR, etc.) and torchrun configuration, the MPI plugin generates SSH keys and hostfiles, and the CoScheduling/Volcano plugins create PodGroup resources for gang scheduling. The single-TrainJobStatusPlugin constraint (only one plugin can report status) is noteworthy - it means JobSet is the authoritative workload orchestrator and status source.

The RHAI progression tracking module (`pkg/rhai/`) is a downstream-only enhancement that adds training observability without modifying the upstream API surface. It stores progress metadata in TrainJob annotations rather than in the CRD status, avoiding upstream API changes. The progression system uses a polling model (HTTP GET to pod IP:28080/metrics at configurable intervals) rather than event-driven updates, which is pragmatic for ML workloads that report metrics at their own cadence. The NetworkPolicy generation is a thoughtful security addition - it restricts the metrics endpoint to only be accessible from the controller pod, preventing lateral movement from compromised training pods. The controller namespace detection (`getControllerNamespace()`) falls back to "opendatahub" when not running in a pod, which hard-codes a platform assumption.

The operator's finalizer pattern on TrainingRuntime/ClusterTrainingRuntime is designed to prevent orphaned workloads. When a runtime has TrainJobs referencing it, the finalizer prevents deletion until all TrainJobs are removed. The TrainJobWatcher interface creates a bidirectional notification channel between runtime reconcilers and the TrainJob reconciler, ensuring finalizer state stays consistent.

A potential area of concern is the webhook certificate management. The operator manages its own TLS certificates via the OPA cert-controller library rather than delegating to cert-manager or the platform's certificate infrastructure. This means the webhook certificate lifecycle is independent of the platform's PKI, which simplifies deployment but creates a separate trust chain. The readiness probe (`/readyz`) correctly gates on certificate generation, preventing the webhook from receiving traffic before TLS is configured. HTTP/2 is disabled by default as a mitigation for CVE-2023-44487 (HTTP/2 rapid reset attack), which is a sensible security posture for a webhook server.
