## Purpose

**Short**: The ODH Dashboard is the web-based management console for Red Hat OpenShift AI, providing a unified interface for data scientists and administrators to manage notebooks, model serving, pipelines, model registry, and platform configuration.

**Detailed**: The ODH Dashboard implements a modular micro-frontend architecture using Webpack Module Federation, allowing feature domains (Gen AI Studio, Model-as-a-Service, Model Registry) to be developed, built, and deployed as independent federated modules while sharing a common host shell. The host shell is a React/PatternFly application backed by a Node.js/Fastify BFF server that proxies Kubernetes API calls and upstream platform services (Prometheus, Data Science Pipelines, TrustyAI, Llama Stack, MLflow). Each federated module ships its own Go-based BFF sidecar that runs alongside the main dashboard container in the same pod.

The dashboard integrates deeply with the RHOAI platform operator ecosystem. It reads configuration from the `OdhDashboardConfig` CRD, manages notebooks via the `kubeflow.org/v1/Notebook` CRD, orchestrates model serving through KServe `InferenceService` and `ServingRuntime` CRDs, and connects to Model Registry, Data Science Pipelines, TrustyAI, and Perses for observability. Authentication is handled by a kube-rbac-proxy sidecar that enforces RBAC at the ingress layer, forwarding user identity headers to the backend. In RHOAI 3.x, the dashboard is exposed via an HTTPRoute (Gateway API) rather than an OpenShift Route, with the Route retained only for legacy URL redirects.

## Architectural Analysis

The ODH Dashboard is one of the most architecturally complex components in the RHOAI platform, implementing a micro-frontend pattern through Webpack Module Federation. The host shell (React/PatternFly frontend + Node.js/Fastify BFF) provides the core dashboard experience while three federated modules (Gen AI, MaaS, Model Registry) are built and deployed as independent Go+TypeScript sidecars. This architecture enables feature teams to develop and ship independently while sharing authentication, routing, and the PatternFly design system through singleton shared dependencies. The federation configuration is managed via a Kubernetes ConfigMap (`federation-config`) that maps module names to in-pod sidecar services, with the Node.js backend acting as a reverse proxy to route `/_mf/{module}` requests to the appropriate sidecar port.

The authentication architecture follows a layered proxy pattern. The outermost layer is kube-rbac-proxy, which listens on port 8443 with TLS and enforces that the user can list OpenShift projects (a permissive but effective gate). Upon successful authentication, kube-rbac-proxy forwards requests to the Node.js backend on localhost:8080, injecting `x-auth-request-user`, `x-auth-request-groups`, and `x-forwarded-access-token` headers. The backend then uses the forwarded token for all Kubernetes API calls, ensuring that every operation respects the user's RBAC permissions. Admin routes perform additional checks via SubjectAccessReview. The BFF sidecars receive the same token via a configurable header and forward it to upstream services.

The deployment topology is notable: in the modular architecture overlay (which RHOAI 3.x uses), a single pod contains five containers: (1) the main dashboard (Node.js, port 8080), (2) kube-rbac-proxy (port 8443/8444), (3) model-registry-ui BFF (Go, port 8043), (4) gen-ai-ui BFF (Go, port 8143), and (5) maas-ui BFF (Go, port 8243). All BFF sidecars share the same TLS certificate from the `dashboard-proxy-tls` secret and mount the same CA bundles. The NetworkPolicy allows ingress only from the OpenShift ingress controller namespace and same-namespace pods, restricting access to ports 8443, 8043, 8143, and 8243.

The dashboard's RBAC footprint is broad because it acts as a proxy for user actions across the entire platform. The ClusterRole includes permissions to manage notebooks, model registries, inference services, secrets, configmaps, PVCs, and RBAC bindings. It also watches CRDs from multiple components (DataScienceCluster, DSCInitialization, LlamaStackDistribution, FeatureStore, GuardrailsOrchestrator). This breadth is intentional: the dashboard is the user's single pane of glass, and it needs to read and manage resources across the platform on behalf of users, always forwarding the user's own token to ensure RBAC is respected at the Kubernetes API level.

FIPS compliance is carefully managed despite the multi-language architecture. The Go BFF sidecars are compiled with `CGO_ENABLED=1 -tags strictfipsruntime`, ensuring they use OpenSSL for cryptography. The Node.js primary service relies on the UBI9 base image's OpenSSL. A notable defensive measure is the explicit removal of esbuild Go binaries from the Node.js image (`rm -rf node_modules/esbuild`) -- these are transitive build dependencies that are not needed at runtime and would fail check-payload validation because they are statically linked Go binaries.
