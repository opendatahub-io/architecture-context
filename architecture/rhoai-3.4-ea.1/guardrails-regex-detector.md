## Purpose

**Short**: A lightweight Rust HTTP service that detects PII and custom patterns in text using regex matching, integrated with the FMS Guardrails Orchestrator.

**Detailed**: The Guardrails Regex Detector is a stateless HTTP microservice built in Rust using the Axum web framework. It provides a REST API endpoint that accepts text content and applies regex-based pattern matching to detect personally identifiable information (PII) and custom patterns. The service ships with three built-in PII detectors — email address, Social Security number (SSN), and credit card number — and also supports arbitrary user-supplied regex patterns passed at request time.

This component is part of the RHOAI TrustyAI Guardrails stack, where it serves as a downstream detector invoked by the FMS Guardrails Orchestrator. When the orchestrator receives a guardrails evaluation request, it dispatches text content to registered detectors — including this one — which return structured detection results with match positions, types, and confidence scores. The regex detector focuses specifically on deterministic pattern-based detection (score is always 1.0), complementing ML-based detectors in the guardrails pipeline.

The service is intentionally minimal — no database, no persistent state, no authentication, no TLS termination. It is designed to run as an internal sidecar or pod-level service behind the orchestrator, relying on the platform's network policies and service mesh for security.

## Architectural Analysis

The Guardrails Regex Detector is among the simplest services in the RHOAI platform — by design. It implements a single-purpose microservice pattern: one HTTP endpoint, no state, no external dependencies, no auth, no TLS. This simplicity is an architectural strength: the service has a minimal attack surface, fast startup time, and trivial horizontal scaling characteristics.

The detector uses a two-tier regex resolution strategy (lines 106-133 in `detectors.rs`): a `HashMap` of built-in detector functions mapped to string keys ("email", "ssn", "credit-card"), with fallthrough to arbitrary user-supplied regex compilation via `Regex::new()`. The built-in detectors hardcode both the regex pattern and the detection metadata (type: "pii", detection: "EmailAddress"), while custom patterns receive generic metadata (type: "custom", detection: "custom"). This design allows the orchestrator to reference well-known PII types by name while retaining full flexibility for arbitrary pattern matching.

One notable architectural gap is the absence of a Konflux Dockerfile (`Dockerfile.konflux`). Since RHOAI is entirely built by Konflux, the standard `Dockerfile` present in this repo may not be the actual build artifact used in production. The Dockerfile uses `rust:1.84.0` as the build image (a Debian-based image from Docker Hub) and `ubi9/ubi-minimal` for the release stage, which is appropriate for RHOAI, but lacks Hermeto prefetch integration (`cachi2.env`) and does not pin RPMs. The `compat-openssl11` package is installed at runtime, though the application code itself never calls into OpenSSL — this dependency appears to be present for platform compliance (ensuring OpenSSL is available in the image for FIPS validation tooling) rather than functional necessity.

The service runs as root by default — the Dockerfile contains no `USER` directive. This is flagged in the `component-architecture.json` issues list and represents a security hardening gap. The `shadow-utils` package is installed (providing `useradd`/`groupadd`), suggesting the intent to add a non-root user, but the step was never completed.

The Rust toolchain is pinned to version 1.84.0 via both `rust-toolchain.toml` and the Dockerfile `FROM` directive, ensuring build reproducibility across environments.
