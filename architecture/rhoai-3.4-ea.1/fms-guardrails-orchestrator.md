## Purpose
**Short**: A REST API orchestrator that coordinates AI text generation with content safety guardrails by managing detector, chunker, and LLM backend services.

**Detailed**: The FMS Guardrails Orchestrator (fms-guardrails-orchestr8) is a Rust-based middleware service that sits between clients and language models to enforce content safety. It receives text generation and detection requests through HTTP REST APIs, orchestrates calls to multiple downstream services (LLM generation backends, content safety detectors, text chunkers), and aggregates results to ensure harmful content is detected and filtered before being returned to clients.

The orchestrator supports multiple generation backends including TGIS (Text Generation Inference Server) via gRPC and caikit-nlp via gRPC, as well as OpenAI-compatible APIs via HTTP. It manages detector services for content analysis across multiple detection types (text contents, text generation, text chat, and context documents). Text chunking services split input text into manageable segments for parallel detection processing.

The service supports TLS and mTLS for both inbound connections (server-side) and outbound connections to downstream services (client-side). It provides OpenTelemetry-based observability with configurable trace and metrics export via gRPC or HTTP OTLP protocols. The orchestrator is built for multi-architecture deployment (amd64, arm64, ppc64le, s390x) using Konflux CI pipelines on UBI 9 minimal base images.

## Security

### TLS Configuration

**Server-side TLS** (guardrails server on port 8033):
- Configured via `--tls-cert-path` and `--tls-key-path` CLI args
- Optional mTLS with `--tls-client-ca-cert-path` for client certificate verification
- Uses rustls with ring crypto provider
- WebPKI client certificate verification when mTLS is enabled
- Graceful shutdown with 10-second timeout

**Client-side TLS** (outbound connections to downstream services):
- Per-service TLS configuration via YAML config file
- Named TLS configs allow reuse across multiple services
- Supports: cert_path, key_path, client_ca_cert_path, insecure mode
- gRPC clients use tonic TLS with native roots + WebPKI roots
- HTTP clients use hyper-rustls with HTTP/1.1 and HTTP/2 support
