## Purpose

**Short**: REST API orchestrator that coordinates AI text generation with content safety guardrails by managing detectors, chunkers, and LLM backends.

**Detailed**: The FMS Guardrails Orchestrator acts as middleware between clients and language models, providing content safety detection, filtering, and secure text generation. It receives user requests containing prompts and detector configurations, then orchestrates calls to multiple downstream services: text generation backends (TGIS via gRPC or caikit-nlp via gRPC), content safety detectors (via HTTP), and text chunker services (via gRPC). The orchestrator runs detections on both input (user prompts) and output (generated text) to identify and block harmful content before returning results to the client.

The service exposes a REST API with multiple endpoint families: v1 classification-with-generation endpoints for legacy compatibility, v2 detection endpoints for standalone content detection (text, chat, context documents, generated text), and OpenAI-compatible completions/chat-completions endpoints with integrated guardrail detection. Both unary and streaming response modes are supported across all endpoint families. The orchestrator supports TLS and mTLS for both its server-side connections and client-side connections to downstream services, with configurable per-service TLS profiles. OpenTelemetry integration provides traces and metrics export via OTLP (gRPC or HTTP protocols).

## Architectural Analysis

The FMS Guardrails Orchestrator follows a clean middleware/proxy architecture written in Rust for high performance and safety. The codebase is organized into clearly separated modules: `server` (HTTP endpoint routing and TLS), `orchestrator` (business logic and task orchestration), `clients` (downstream service communication), `config` (YAML-based configuration), and `models` (request/response data structures). The orchestrator pattern uses a `Handle<Task>` trait that dispatches each request type to a dedicated handler, enabling independent implementation and testing of each workflow.

A notable design decision is the dual-protocol backend support: the orchestrator can connect to text generation services via either the TGIS gRPC protocol (IBM's text-generation-inference fork) or the caikit-nlp gRPC protocol, selected via configuration. For the OpenAI-compatible chat completions path, it uses HTTP with server-sent events (SSE) for streaming. This flexibility allows the orchestrator to work with different inference backends deployed in the RHOAI platform (vLLM via OpenAI API, or TGIS/caikit-nlp via gRPC).

The TLS implementation deserves attention from a security perspective. Server-side TLS uses `rustls` with the `ring` crypto backend and supports both TLS and mTLS (with `WebPkiClientVerifier`). Client-side TLS is configurable per service via named TLS profiles in the config YAML, allowing different cert/key pairs for detectors, chunkers, and generation services. However, the `insecure: true` option (backed by a `NoVerifier` struct that accepts any certificate) represents a significant security risk if used in production. The `ring` crypto backend, while high-quality, is not FIPS-certified, creating a gap between container-level FIPS compliance (OpenSSL is installed in the image) and application-level crypto operations.

The concurrency model uses Tokio's multi-threaded runtime with configurable concurrent request limits for detectors (default 5) and chunkers (default 5). The retry mechanism applies exponential backoff for generation client calls with configurable max retries (default 3), but only retries on specific HTTP status codes (502, 503, 504, 505, 506). Header passthrough is configurable, allowing specific HTTP headers from client requests to be forwarded to downstream services, with an optional rewrite of `X-Forwarded-Access-Token` to `Authorization: Bearer` for environments using token-forwarding authentication patterns.

The streaming architecture supports three distinct patterns: SSE (Server-Sent Events) for v1 classification and v2 chat completions, ND-JSON (newline-delimited JSON) for v2 streaming content detection, and the standard OpenAI SSE format with `[DONE]` termination for completions endpoints. The bidirectional streaming support for chunkers (via `BidiStreamingChunkerTokenizationTaskPredict`) enables real-time text chunking in streaming pipelines.
