## Purpose

**Short**: KServe is a Kubernetes-native model serving platform that provides CRD-based lifecycle management for deploying, scaling, and routing inference services across multiple ML frameworks, protocols, and deployment modes.

**Detailed**: KServe is the core model inference component of Red Hat OpenShift AI (RHOAI). It defines a set of Custom Resource Definitions (CRDs) that enable declarative deployment of machine learning models as inference services on Kubernetes. The primary CRD, InferenceService (v1beta1), supports predictor, transformer, and explainer components with multiple deployment modes: raw Kubernetes Deployments, Knative serverless, and ModelMesh for multi-model serving.

KServe also provides LLMInferenceService (v1alpha2), a purpose-built CRD for large language model serving with disaggregated prefill/decode architecture, multi-node/multi-GPU support via LeaderWorkerSet, Gateway API-based routing with InferencePool scheduling, and integration with the Inference Gateway (llm-d) ecosystem. The operator manages the full inference stack: workload deployments, autoscaling (HPA/KEDA/VariantAutoscaling), ingress routing (Istio VirtualService, Gateway API HTTPRoute, Kubernetes Ingress, OpenShift Routes), monitoring (Prometheus PodMonitor/ServiceMonitor), and TLS certificate management.

The Python SDK provides a model server framework implementing the KServe V2 inference protocol over both REST (FastAPI/Uvicorn) and gRPC, with pluggable model handlers for frameworks including TensorFlow, PyTorch, Scikit-learn, XGBoost, HuggingFace, and more. A storage initializer system supports downloading models from S3, GCS, Azure Blob, HDFS, HuggingFace Hub, OCI registries, and PVCs. Admission webhooks inject sidecar containers (agent, batcher, storage-initializer, metrics aggregator) into inference pods at creation time.

## Architectural Analysis

### Multi-Deployment Mode Architecture

KServe implements a sophisticated multi-deployment mode architecture that supports three fundamentally different deployment patterns through a shared controller: Knative serverless (scale-to-zero), raw Kubernetes Deployments (static or HPA-scaled), and ModelMesh (multi-model aggregation). The deployment mode is selected per InferenceService via annotation override or namespace/cluster defaults from the inferenceservice-config ConfigMap. Each mode has its own reconciler chain (e.g., KsvcReconciler vs RawKubeReconciler) and ingress strategy (VirtualService vs HTTPRoute vs Kubernetes Ingress). This flexibility enables the same CRD to serve very different operational models -- from cost-optimized serverless for bursty workloads to always-on raw deployments for latency-sensitive inference.

### LLM-Native Serving (LLMInferenceService)

The LLMInferenceService controller (v1alpha2) represents a significant architectural evolution designed specifically for large language model workloads. Unlike the general-purpose InferenceService, it deeply integrates with the Inference Gateway ecosystem (llm-d): deploying an EPP scheduler as a sidecar Deployment, creating InferencePool CRDs for endpoint management, and supporting disaggregated prefill/decode architecture where prefill and decode workloads scale independently. The controller manages a complex resource tree including LeaderWorkerSets for multi-node deployments, DestinationRules for Istio mTLS origination to schedulers, PodMonitors/ServiceMonitors for Prometheus metrics, and self-signed TLS certificates with automatic rotation via pod template hash annotations. Configuration inheritance through LLMInferenceServiceConfig BaseRefs enables reusable templates with Go template variable substitution.

### Webhook-Based Sidecar Injection

KServe uses a pod mutating webhook to inject sidecar containers at pod creation time rather than modifying Deployment specs directly. This is architecturally significant because it allows the controller to create clean Deployments while the webhook transparently adds the storage-initializer init container, agent sidecar (for model pulling/batching/logging), metrics aggregator configuration, and GKE accelerator node selectors. The injection pipeline chains multiple injectors (`storage_initializer_injector.go` -> `agent_injector.go` -> `metrics_aggregate_injector.go`) with each injector checking pod annotations to determine if injection is needed. This design cleanly separates concerns: controllers manage the workload lifecycle, while webhooks handle cross-cutting operational concerns.

### Gateway API Migration (RHOAI 3.x)

The codebase shows an active migration from Istio VirtualService-based routing to Gateway API HTTPRoutes. The LLMInferenceService controller exclusively uses Gateway API (HTTPRoute + InferencePool), while the InferenceService controller supports both paths based on the `enableGatewayApi` configuration flag. On OpenShift (RHOAI), the controller creates Istio DestinationRules alongside HTTPRoutes for TLS origination between the scheduler and workload services, reflecting the hybrid state of the ingress stack where Gateway API handles L7 routing but Istio still manages mTLS and traffic policies. The router_discovery.go module implements sophisticated gateway discovery, checking for gateway readiness and building both internal and external URLs from gateway listener addresses.

### Local Model Caching System

The LocalModelCache, LocalModelNodeGroup, and LocalModelNode CRDs form a distributed caching system for ML models. The architecture uses three separate controllers: the localmodel-manager creates PersistentVolumes and PersistentVolumeClaims per node group, while the localmodelnode-agent runs per-node and creates batch Jobs to download models. The validation webhook ensures unique StorageURIs across caches and prevents deletion of caches still referenced by InferenceServices. This system avoids repeated downloads from cloud storage and is critical for large model deployment where download times significantly impact cold-start latency.

### Multi-Language Repository

KServe is notable as a multi-language repository with Go operators and Python SDK sharing the same repo. The Go components (controller, agent, router, localmodel managers) handle the Kubernetes control plane, while the Python SDK provides the model server framework and storage utilities. The storage-initializer is a Python init container that bridges these worlds -- it's configured by Go controller annotations but runs Python code to download models. The qpext queue-proxy extension is a separate Go module (`qpext/go.mod`) that integrates with Knative's queue-proxy for metrics aggregation.
