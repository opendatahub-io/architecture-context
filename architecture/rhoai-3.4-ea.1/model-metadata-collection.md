## Purpose

**Short**: A CI/CD pipeline tool that collects, enriches, and catalogs AI/ML model metadata from OCI registries and HuggingFace, packaging the results into a static data container consumed by the RHOAI dashboard.

**Detailed**: The model-metadata-collection component serves two distinct roles in the RHOAI platform. First, it provides two Go CLI tools (`model-extractor` and `metadata-report`) that run during CI/CD pipelines to discover models from OCI container registries and HuggingFace collections, extract metadata from model cards and registry manifests, enrich that metadata through a multi-source priority hierarchy, and generate standardized YAML catalog files.

Second, the component produces a minimal container image that contains only the pre-generated catalog YAML files and runs `sleep infinity`. This container is deployed as a static data sidecar — other RHOAI components (primarily the dashboard) mount its filesystem to read model catalog data at runtime. The Go binaries themselves are not included in the production container; they exist solely as build-time tools.

The metadata enrichment pipeline implements a sophisticated multi-source strategy with source tracking. Each metadata field records where its value came from (e.g., `huggingface.yaml`, `huggingface.api`, `modelcard.regex`, `generated`), enabling auditability and debugging of catalog content. The system uses token-based similarity scoring with model family compatibility checks to accurately match registry models to their HuggingFace counterparts.

## Architectural Analysis

### Static Data Container Pattern

This component uses an unusual but effective architectural pattern: the production deliverable is a container that holds only static data files and runs `sleep infinity`. The actual application logic (Go CLI tools) runs exclusively during CI/CD and is never shipped to customers. This pattern provides several advantages:

1. **Minimal attack surface**: The production container contains no executable code, no network listeners, and no writable state. It's a read-only data volume.
2. **Deterministic content**: The catalog YAML is generated once during CI/CD and immutably packaged. There's no runtime data fetching that could fail or produce inconsistent results.
3. **Simple consumption**: Other components (the dashboard) just mount the filesystem and read YAML files — no API calls, no service discovery, no authentication required.

### Multi-Source Metadata Enrichment

The metadata pipeline implements a sophisticated priority hierarchy for resolving conflicts between data sources:

1. **huggingface.yaml** (highest priority) — Structured YAML frontmatter from HuggingFace READMEs
2. **huggingface.api** — HuggingFace model API responses
3. **huggingface.tags** — Parsed HuggingFace repository tags
4. **modelcard.yaml** — YAML frontmatter from model cards within OCI image layers
5. **modelcard.regex** — Regex-extracted values from model card markdown content
6. **generated** (lowest priority) — Algorithmically generated values (e.g., description from model name)

Every field in the output records its provenance source, enabling operators to audit where each piece of catalog data originated.

### Model Family Matching

The enrichment system uses token-based similarity scoring with a critical safety mechanism: model family compatibility checking. Before accepting a HuggingFace match for a registry model, the system verifies they belong to the same model family (e.g., Llama, Granite, Mistral). This prevents incorrect cross-family matches where similar naming patterns might otherwise produce high similarity scores. The similarity threshold is 0.5 for accepting matches, with 0.8+ considered high confidence.

### Metadata Format Migration

The codebase supports three metadata formats: current (structured `OCIArtifact` with timestamps and custom properties), mixed-type (string timestamps), and legacy (string artifact URIs). The `internal/metadata/migration.go` module handles transparent conversion between formats, ensuring backward compatibility as the catalog format evolves.

### Concurrency Model

Model processing uses a semaphore-based concurrency pattern with configurable parallelism (default: 5 concurrent goroutines). Each model's OCI manifest inspection and HuggingFace enrichment runs independently, with results collected and merged sequentially during catalog generation.
