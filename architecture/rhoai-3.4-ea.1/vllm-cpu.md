## Purpose

**Short**: CPU-optimized build of vLLM, a high-throughput inference and serving engine for large language models.

**Detailed**: vllm-cpu is the CPU-only variant of the vLLM inference engine, packaged for Red Hat OpenShift AI (RHOAI). It provides an OpenAI-compatible REST API server and a gRPC server for serving large language models on CPU hardware without GPU requirements. The component supports text generation (chat completions, completions, responses), embeddings, classification, scoring/reranking, audio transcription/translation, and tokenization, all exposed through a FastAPI-based HTTP server running on Uvicorn.

The repository is a downstream fork of the upstream vLLM project (vllm-project/vllm), adapted for RHOAI with Red Hat UBI base images, Konflux CI/CD pipelines, multi-architecture builds (x86_64, s390x, ppc64le), and RHOAI-specific usage telemetry. The inference engine uses PyTorch for CPU-based model execution and supports advanced features like LoRA adapters, structured output (JSON schema, regex, grammar constraints), tool calling, streaming responses, and ORCA load-balancing metrics. In the RHOAI platform, vllm-cpu serves as the inference runtime for KServe InferenceService custom resources when GPU hardware is not available.

## Architectural Analysis

The vllm-cpu component is architecturally a monolithic Python inference server that bundles multiple API protocol implementations (OpenAI, Anthropic, SageMaker, gRPC) into a single deployable container. The core design pattern is a FastAPI application with modular route registration: the main `api_server.py` registers core OpenAI routes, while `vllm/entrypoints/serve/__init__.py` dynamically attaches sub-routers for health, metrics, LoRA, caching, disaggregation, RLHF, elastic expert parallelism, profiling, sleep, RPC, and tokenization. This modular router architecture allows conditional feature enablement — for example, the `/server_info` endpoint only activates in dev mode (`VLLM_SERVER_DEV_MODE=1`).

The authentication model is notable for its simplicity and limitations. The `AuthenticationMiddleware` class implements Bearer token authentication using SHA-256 hash comparison with constant-time `secrets.compare_digest()`, but it only protects `/v1/*`-prefixed endpoints. Non-prefixed endpoints like `/score`, `/rerank`, `/classify`, `/pooling`, `/tokenize`, and `/detokenize` are completely unauthenticated. The gRPC server (`vllm/entrypoints/grpc_server.py` line 455) binds with `add_insecure_port()`, meaning it has no TLS or authentication. In a RHOAI deployment, external authentication is expected to be handled by KServe's ingress layer (kube-rbac-proxy sidecars or Gateway API + EnvoyFilter).

The CPU-specific build is the key differentiator from the GPU variants. The `Dockerfile.konflux.cpu` builds for multiple architectures (x86_64 via the default flow, s390x and ppc64le via architecture-specific build scripts `build_vllm_s390x.sh` and `build_vllm_ppc64le.sh`). The Tekton pipeline configuration explicitly builds for `linux/s390x` and `linux/ppc64le` platforms and uses the `Dockerfile.konflux.cpu` file. The build is non-hermetic (`hermetic: false` in the Tekton PipelineRun), which is expected for an Early Access (EA) release — GA releases typically add rpms.lock.yaml and hermetic build configuration.

The V1 engine architecture (`VLLM_USE_V1=1`, set as default in `Dockerfile.cpu.ubi`) uses the `AsyncLLM` class for asynchronous inference with a single-process or multi-process deployment model. The `VLLM_WORKER_MULTIPROC_METHOD=fork` environment variable in `Dockerfile.konflux.cpu` configures the multi-process spawning behavior. Usage telemetry is configured to report to RHOAI-specific endpoints (`VLLM_USAGE_STATS_SERVER=https://console.redhat.com/api/rhaiis-stats` in `Dockerfile.cpu.ubi`) while upstream vLLM stats are disabled (`VLLM_NO_USAGE_STATS=1`).
