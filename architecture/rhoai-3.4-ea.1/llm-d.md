## Purpose

**Short**: llm-d is a Kubernetes-native distributed inference serving stack that builds optimized vLLM container images and provides deployment guides for production-grade LLM serving across multiple hardware accelerators.

**Detailed**: llm-d accelerates distributed inference by integrating industry-standard open technologies: vLLM as the default model server engine, Kubernetes Inference Gateway (Gateway API) as the control plane API and load balancing orchestrator, and Kubernetes as infrastructure orchestrator and workload control plane. The project is a CNCF sandbox project founded by Red Hat, Google Cloud, IBM Research, CoreWeave, and NVIDIA.

The repository itself does not contain an operator or a compiled service — it is an integration and orchestration repository that: (1) builds optimized container images for vLLM across CUDA, CPU, ROCm, HPU, and XPU accelerators; (2) provides kustomize-based deployment recipes for model servers, gateways, and the llm-d Router (EPP); and (3) offers well-tested guides for production deployment patterns including optimized baseline, prefill/decode disaggregation, wide expert-parallelism, tiered prefix caching, workload autoscaling, and flow control.

llm-d adds intelligent request routing via the llm-d Router (Endpoint Picker/EPP), disaggregated serving coordination via a routing sidecar, KV cache management (prefix-cache-aware routing, offloading), and workload variant autoscaling. The system targets online serving of generative models with 1B+ parameters on recent datacenter-class accelerators (NVIDIA A100+, AMD MI250+, Google TPU v5e+, Intel GPU Max).

## Architectural Analysis

llm-d follows a composition-over-configuration philosophy, assembling production inference stacks from standard open-source components (vLLM, Kubernetes Gateway API, Envoy) rather than building a monolithic serving platform. The repository itself is an integration layer — it builds container images and provides deployment recipes, while the actual routing intelligence, autoscaling logic, and serving engine live in separate repositories (llm-d-inference-scheduler, workload-variant-autoscaler, vLLM upstream). This separation enables each component to evolve independently with its own release cycle, but creates a version coordination challenge that the `docker/common-versions` file and Helm chart version pinning address.

The Dockerfile.cuda is the most architecturally significant artifact in the repository. Its multi-stage build produces a comprehensive inference image that includes not just vLLM but an entire distributed computing stack: UCX for communication, NIXL for KV cache transfer, NVSHMEM for symmetric memory, FlashInfer for attention kernels, DeepEP for expert parallelism, DeepGEMM for GEMM operations, and LMCache for tiered caching. This "batteries-included" approach means a single image supports all deployment patterns (optimized baseline, P/D disaggregation, wide EP, tiered prefix cache) without requiring different images per pattern. The trade-off is image size — the CUDA image includes components that many deployments won't use. The vLLM installation supports three modes (prebuilt wheel, precompiled binaries, full source build) to balance flexibility and build speed.

The kustomize-based deployment system is organized as a recipe/overlay pattern where `guides/recipes/` defines reusable base resources (Deployments, ServiceAccounts, Gateways) and each guide (optimized-baseline, pd-disaggregation, etc.) layers accelerator-specific and model-specific patches on top. This enables consistent deployment patterns across seven hardware backends (NVIDIA, AMD, Intel XPU, Intel HPU, Google TPU v6/v7, CPU) while keeping guide-specific configuration in focused overlay files. The labeling scheme (`llm-d.ai/model`, `llm-d.ai/guide`, `llm-d.ai/accelerator-variant`, `llm-d.ai/accelerator-vendor`) serves dual purposes: Kubernetes resource management and InferencePool label selection by the EPP.

The security model delegates authentication to the gateway layer — vLLM model servers expose plaintext HTTP with no authentication. This is appropriate for a Kubernetes-internal service where the gateway enforces auth before forwarding requests, but the documentation explicitly warns that flow control headers (`x-gateway-inference-fairness-id`, `x-gateway-inference-objective`) are self-asserted by clients and must be stripped/injected by a trusted upstream gateway in production. The lack of TLS between gateway and model server pods, and between NIXL peers, means network-level security relies on Kubernetes NetworkPolicies or service mesh (Istio) rather than application-level encryption — a common pattern for GPU workloads where encryption overhead on high-bandwidth RDMA links is prohibitive.

The observability stack is well-integrated: Prometheus scraping via PodMonitor CRDs, Grafana dashboards, and OpenTelemetry distributed tracing across all three layers (vLLM, EPP, routing sidecar). The EPP's plugin architecture (prefix-cache-scorer, queue-scorer, kv-cache-utilization-scorer, active-request-scorer) provides composable routing intelligence that can be configured per deployment pattern via Helm values — the optimized baseline uses one scoring profile while P/D disaggregation uses separate prefill and decode scheduling profiles with different plugin weights.
