## Purpose
**Short**: Kubernetes controller and mutating webhook that extends Kubeflow Notebook CRDs with OpenShift/RHOAI-specific networking, authentication, certificate management, and pipeline integration features.

**Detailed**: The ODH Notebook Controller is a secondary controller that watches `Notebook` custom resources (defined by the upstream Kubeflow notebook-controller) and augments them with RHOAI platform capabilities. It operates alongside the upstream Kubeflow notebook controller, adding OpenShift-specific functionality.

The controller's primary responsibilities include: (1) Creating Gateway API `HTTPRoute` resources in a centralized namespace to expose notebooks through the platform's `data-science-gateway` Gateway, with cross-namespace `ReferenceGrant` resources to authorize backend service references; (2) Injecting `kube-rbac-proxy` sidecar containers via a mutating webhook for RBAC-based authentication on notebook endpoints; (3) Managing `NetworkPolicy` resources per notebook to control ingress traffic; (4) Aggregating and distributing trusted CA certificate bundles to notebook pods; (5) Syncing Elyra pipeline runtime configuration from `DataSciencePipelinesApplication` CRs and mounting them as secrets; (6) Managing pipeline runtime image ConfigMaps from ImageStreams; (7) Supporting Feast feature store config mounting.

The controller implements the RHOAI 3.x ingress pattern using Gateway API (`HTTPRoute` + `ReferenceGrant`) and `kube-rbac-proxy` for authentication, replacing the legacy RHOAI 2.x pattern of OpenShift `Route` + `oauth-proxy`. Legacy OAuthClient cleanup finalizers are retained for users migrating from 2.x to 3.x.

**RHOAI 3.x Ingress Pattern (current)**:
- The controller creates `HTTPRoute` CRs in the **centralized controller namespace** (not user namespaces) for security
- Each HTTPRoute references the parent `data-science-gateway` Gateway in `openshift-ingress` namespace
- Path matching: `/notebook/{user-namespace}/{notebook-name}` (PathPrefix)
- Gateway and namespace are configurable via `NOTEBOOK_GATEWAY_NAME` and `NOTEBOOK_GATEWAY_NAMESPACE` env vars
- **With auth** (`notebooks.opendatahub.io/inject-auth: "true"`): HTTPRoute backends point to `{notebook}-kube-rbac-proxy` service on port 8443/TCP
- **Without auth**: HTTPRoute backends point to notebook service on port 8888/TCP in user namespace
- Cross-namespace backend references authorized by `ReferenceGrant` CRs in user namespaces (one per namespace, shared by all notebooks)
- Cleanup uses finalizers since cross-namespace ownerReferences are not supported:
  - `notebook.opendatahub.io/httproute-cleanup`
  - `notebook.opendatahub.io/referencegrant-cleanup`
  - `notebook.opendatahub.io/kube-rbac-proxy-cleanup`
- Switching between auth/non-auth modes automatically cleans up conflicting HTTPRoutes

**kube-rbac-proxy sidecar (RHOAI 3.x auth pattern)**:
- Injected via mutating webhook when annotation `notebooks.opendatahub.io/inject-auth: "true"` is present
- Listens on port 8443/TCP (HTTPS with TLS from OpenShift service-ca via `{notebook}-kube-rbac-proxy-tls` secret)
- Proxies to upstream notebook container on `127.0.0.1:8888/TCP` (HTTP, localhost)
- Health endpoint on port 8444/TCP (HTTPS)
- Enforces Kubernetes RBAC via `SubjectAccessReview`: verb=`get`, resource=`notebooks`, apiGroup=`kubeflow.org`, name=`{notebook}`, namespace=`{namespace}`
- Auth delegation via `ClusterRoleBinding` to `system:auth-delegator` per notebook ServiceAccount
- Forwards authenticated identity via headers: `X-Auth-Request-User`, `X-Auth-Request-Groups`
- Resource configuration via annotations: `notebooks.opendatahub.io/auth-sidecar-cpu-request`, `-memory-request`, `-cpu-limit`, `-memory-limit`
- Default resources: 100m CPU / 64Mi memory (requests and limits)

**RHOAI 2.x Ingress Pattern (legacy - cleanup code only)**:
- Previously used OpenShift Routes with oauth-proxy sidecars and OAuthClient CRs
- OAuthClient cleanup code retained for 2.x to 3.x migration (finalizer `notebook-oauth-client-finalizer.opendatahub.io`)
- No new Route or oauth-proxy resources are created
