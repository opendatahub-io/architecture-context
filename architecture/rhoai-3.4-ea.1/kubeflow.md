## Purpose

**Short**: Manages the lifecycle of Jupyter Notebook servers on OpenShift, including pod creation, authentication sidecar injection, Gateway API routing, and idle notebook culling.

**Detailed**: This repository contains two complementary Kubernetes controllers that together manage Jupyter Notebook workloads on Red Hat OpenShift AI. The **Kubeflow Notebook Controller** (upstream) reconciles `Notebook` custom resources into StatefulSets and Services, monitors pod status, re-emits events, and culls idle notebooks by querying the Jupyter API. The **ODH Notebook Controller** (downstream, OpenShift-specific) extends this with a mutating webhook that injects `kube-rbac-proxy` authentication sidecars, CA certificate bundles, cluster proxy environment variables, pipeline runtime configurations, and Feast feature store mounts. It also manages per-notebook Gateway API HTTPRoutes in a central namespace, cross-namespace ReferenceGrants, NetworkPolicies, and integrates with Data Science Pipelines for Elyra notebook pipeline execution.

The two controllers run as separate Deployments in the platform namespace (`opendatahub` / `redhat-ods-applications`). The upstream controller owns the core notebook lifecycle (StatefulSet, Service, culling), while the ODH controller handles OpenShift-specific concerns (authentication, networking, certificate management, pipeline integration). Both are built as FIPS-compliant Go binaries using `GOEXPERIMENT=strictfipsruntime` and shipped as UBI9-based container images via Konflux.

## Architectural Analysis

The kubeflow repository implements a **two-controller architecture** where the upstream Kubeflow controller handles the core Notebook lifecycle (StatefulSet, Service, culling) while the ODH controller extends it with OpenShift-specific capabilities. This separation preserves upstream rebase compatibility -- the ODH controller watches the same Notebook CRD but adds behavior orthogonally through a mutating webhook and a separate reconcile loop. The webhook intercepts Notebook CREATE and UPDATE operations to inject sidecars, volumes, and environment variables before the upstream controller creates the StatefulSet. The ODH reconciler then creates the networking and security resources (HTTPRoutes, NetworkPolicies, ReferenceGrants) that the upstream controller knows nothing about.

The **kube-rbac-proxy sidecar injection pattern** is central to the RHOAI 3.x authentication model. When a Notebook has the `notebooks.opendatahub.io/inject-auth=true` annotation, the webhook injects a kube-rbac-proxy container that listens on port 8443 with TLS (using OpenShift service-ca certificates) and proxies authenticated requests to the notebook's port 8888 on localhost. The reconciler then creates a separate Service for the kube-rbac-proxy, a ConfigMap with authorization rules (requiring GET permission on the Notebook resource), and a ClusterRoleBinding granting `system:auth-delegator` to the notebook's ServiceAccount. The HTTPRoute is updated to point to the kube-rbac-proxy Service instead of the notebook Service directly. This model replaces the RHOAI 2.x OAuth proxy pattern, and the controller retains cleanup code for migrating OAuthClient CRs via finalizers.

The **cross-namespace routing architecture** is noteworthy: HTTPRoutes are created in the controller's namespace (the central/platform namespace), not in the user's namespace. This protects routing rules from user modification. To make this work, the controller creates ReferenceGrant CRs in each user namespace that authorize HTTPRoute backends from the controller namespace to reference Services in the user namespace. ReferenceGrants are shared per user namespace (one grant covers all notebooks) and are cleaned up when the last notebook in a namespace is deleted. The HTTPRoute hostname is generated per-notebook and constrained to 63 characters.

The **running notebook protection mechanism** prevents unexpected pod restarts. When the webhook detects that mutations (e.g., image update, sidecar config change) would alter the pod template spec of a running notebook, it blocks the update by preserving the old pod spec and sets an `update-pending` annotation with a human-readable diff. The user must explicitly stop and restart the notebook to apply pending changes. This prevents data loss in active Jupyter sessions.

The **Elyra pipeline integration** demonstrates the controller's role as a configuration aggregator: it watches DataSciencePipelinesApplication CRs, extracts the pipeline API endpoint and S3 storage credentials (from a referenced Secret), constructs a JSON configuration file (`odh_dsp.json`), and mounts it as a Secret into the notebook pod. The public API endpoint is resolved from the Gateway CR listener hostname, with a fallback to OpenShift Routes when the Gateway uses OcpRoute ingressMode. This dual-path hostname resolution shows the controller adapting to multiple ingress configurations.
