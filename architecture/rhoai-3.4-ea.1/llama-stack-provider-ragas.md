## Purpose

**Short**: Implements Ragas as an out-of-tree evaluation provider for the Llama Stack eval API, supporting both in-process (inline) and Kubeflow Pipelines-based (remote) execution modes.

**Detailed**: The llama-stack-provider-ragas component bridges the Ragas evaluation framework with Meta's Llama Stack platform. It registers as an external provider implementing the Llama Stack `Eval` API, enabling users to run LLM evaluation benchmarks (faithfulness, answer relevancy, context precision, context recall) through Llama Stack's standardized eval interface.

The provider ships in two execution modes. The **inline** mode runs Ragas evaluations in the same process as the Llama Stack server, wrapping the Llama Stack inference API (completions and embeddings) into LangChain-compatible LLM and embedding interfaces that Ragas expects. The **remote** mode offloads evaluation to Kubeflow Pipelines (Data Science Pipelines in RHOAI), constructing a two-step KFP pipeline: one component retrieves datasets from Llama Stack, and the other runs the Ragas evaluation using a dedicated container image. Results are persisted to S3 and fetched back through the Llama Stack job status API.

This component is not deployed as a standalone service. It is packaged as a Python library (`pip install llama-stack-provider-ragas`) and loaded by a Llama Stack server at runtime. The Konflux-built container image (`odh-trustyai-ragas-lls-provider-dsp-rhel9`) serves as the base image for Kubeflow pipeline steps in the remote execution mode.

## Architectural Analysis

The llama-stack-provider-ragas follows a plugin architecture pattern where the component implements Llama Stack's `Eval` and `BenchmarksProtocolPrivate` interfaces. The provider factory (`get_provider_spec()` in `provider.py`) dynamically determines at import time whether remote dependencies (kfp, kubernetes, s3fs) are available, returning either one or two provider specs. This graceful degradation means the base installation always provides inline evaluation, while remote capabilities are opt-in via extras.

The inline execution mode uses an adapter pattern to bridge Llama Stack's inference API with Ragas' LangChain-based LLM/Embeddings interfaces. `LlamaStackInlineLLM` extends `BaseRagasLLM` and `LlamaStackInlineEmbeddings` extends `BaseRagasEmbeddings`, converting between OpenAI-compatible completion/embedding requests and Ragas' expected `LLMResult`/embedding vector formats. The inline evaluator runs Ragas in a background thread (`asyncio.to_thread`) with `max_workers=1` to avoid event loop conflicts, managing job state in an in-memory dictionary keyed by sequential job IDs.

The remote execution mode is architecturally more complex. It constructs a Kubeflow Pipeline with two `@dsl.component`-decorated functions: `retrieve_data_from_llama_stack` (fetches the dataset) and `run_ragas_evaluation` (runs the actual evaluation). Both components run in a container built from the Konflux-produced image. The remote wrappers (`LlamaStackRemoteLLM`, `LlamaStackRemoteEmbeddings`) differ from inline wrappers in that they use `llama_stack_client` HTTP clients rather than in-process API calls, since KFP pods need to reach the Llama Stack server over the network. AWS credentials for S3 result storage are injected into the evaluation pod via `kubernetes.use_secret_as_env()`.

A noteworthy design decision is the base image resolution strategy in `components.py`. The KFP component base image is resolved at **module import time** (not at runtime) through a three-tier fallback: environment variable `KUBEFLOW_BASE_IMAGE` → Kubernetes ConfigMap `trustyai-service-operator-config` in known namespaces (`redhat-ods-applications`, `opendatahub`) → hardcoded default `quay.io/trustyai/llama-stack-provider-ragas:latest`. This means the ConfigMap read and Kubernetes API call happen when the module is first imported by the Llama Stack server, which has implications for startup latency and error handling.

The compatibility layer (`compat.py`) deserves attention — it implements a dual-import strategy supporting both the legacy `llama_stack` package layout and the newer `llama_stack_api` module structure. This suggests the component is tracking an actively evolving upstream API. The version pins (`ragas==0.3.0`, `llama-stack>0.2.23`) and the recent commit history (bumps to 0.5.0, moves to new `llama-stack-api` module) confirm this is a rapidly iterating component.
