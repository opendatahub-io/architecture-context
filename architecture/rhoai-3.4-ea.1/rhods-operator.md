## Purpose

**Short**: The RHOAI platform operator that orchestrates the entire Red Hat OpenShift AI platform — provisioning, configuring, and managing all AI/ML component operators, the platform ingress stack, authentication, monitoring, and RBAC infrastructure.

**Detailed**: The rhods-operator (also known as the opendatahub-operator in upstream) is the central control plane for Red Hat OpenShift AI (RHOAI) 3.x. It uses the Operator Lifecycle Manager (OLM) pattern with two primary Custom Resources — `DataScienceCluster` (DSC) and `DSCInitialization` (DSCI) — to provide a single declarative API for the entire platform.

The operator manages 16+ component sub-operators (Dashboard, KServe, Kueue, Ray, Training Operator, Trainer, TrustyAI, Model Registry, Data Science Pipelines, Feast, LlamaStack, MLflow, Spark, Workbenches, Model Controller, Models-as-a-Service) through a registry pattern. Each component has its own controller that deploys manifests from the `prefetched-manifests/` directory using kustomize rendering. The operator watches each component's CRD and aggregates status back into the DataScienceCluster.

Critically, the rhods-operator also deploys the platform's entire ingress infrastructure for RHOAI 3.x: a Gateway API-based ingress stack with Envoy proxy, EnvoyFilter CRs for authentication, kube-auth-proxy (OAuth2/OIDC), OpenShift Routes for legacy URL redirects, and per-component HTTPRoutes. It also manages the platform monitoring stack (Prometheus, Thanos, Tempo, Perses, OpenTelemetry), authentication RBAC (admin groups, allowed groups), CA certificate distribution, and hardware profile injection via admission webhooks.

## Architectural Analysis

The rhods-operator is architecturally the most complex component in the RHOAI platform. It functions as a meta-operator — an operator that manages operators — with a two-tier CR hierarchy: the top-level `DataScienceCluster` and `DSCInitialization` CRs drive creation of second-tier component CRs (Dashboard, Kserve, Kueue, etc.), each managed by its own controller. This registry-based pattern (`internal/controller/components/registry/registry.go`) provides a clean extensibility model: new components are added by implementing the `ComponentHandler` interface and registering in the component registry.

The RHOAI 3.x ingress architecture represents a significant evolution from 2.x. The gateway controller (`internal/controller/services/gateway/`) deploys a multi-layer ingress stack: a Gateway API Gateway CR (`data-science-gateway`) in the `openshift-ingress` namespace, with an Envoy proxy as the data plane. Authentication is enforced at the Envoy layer through an `EnvoyFilter` CR that configures `ext_authz` to call the `kube-auth-proxy` service. The auth proxy supports two modes — `IntegratedOAuth` (OpenShift-native) and `OIDC` (external provider) — selected at runtime by querying cluster authentication configuration. Legacy URL compatibility is maintained through nginx-based redirect services and OpenShift Route CRs that 301-redirect old dashboard and gateway hostnames to the new Gateway API hostname. The entire stack is parameterized through the `GatewayConfig` singleton CR, which controls TLS certificates, domain/subdomain, OIDC settings, cookie configuration, and ingress mode (LoadBalancer vs OcpRoute).

The admission webhook infrastructure is notably sophisticated. The hardware profile webhook (`internal/webhook/hardwareprofile/mutating.go`) intercepts Notebooks, InferenceServices, and LLMInferenceServices to inject GPU/CPU resource requirements, scheduling constraints, and Kueue queue labels. It uses a non-blocking design — validation failures emit warning events but don't deny admission, ensuring upgrade compatibility from 2.x to 3.x. The connection webhooks (`internal/webhook/serving/`, `internal/webhook/notebook/`) inject S3, OCI, and URI connection secrets into workloads, with SubjectAccessReview checks ensuring users can only mount secrets they have permission to read. The Kueue webhook validates that workloads in Kueue-managed namespaces carry the required queue-name label.

The monitoring controller deploys a comprehensive observability stack that conditionally adapts to available CRDs. It creates a MonitoringStack (via cluster-observability-operator), OpenTelemetryCollector, Tempo for distributed tracing, Perses for dashboards, and Thanos for metrics aggregation. Each component controller also contributes its own PrometheusRule templates for component-specific alerting. The monitoring stack uses kube-rbac-proxy sidecars for secure Prometheus access, with separate proxies for namespace-scoped and cluster-scoped queries.

A notable design decision is the use of build tags (`-tags rhoai` in the Konflux Dockerfile) to compile platform-specific behavior directly into the binary. Several files have `.odh.go` and `.rhoai.go` variants (e.g., `modelregistry_support.odh.go` / `modelregistry_support.rhoai.go`) that provide different default values for namespaces, route names, and admin groups. This compile-time platform awareness eliminates runtime platform detection for these values, reducing configuration complexity.
