## Purpose
**Short**: The RHOAI platform operator manages the lifecycle of all Red Hat OpenShift AI components, services, and platform infrastructure including Gateway API ingress, authentication, monitoring, and RBAC.

**Detailed**: The rhods-operator (also known as opendatahub-operator) is the central control plane for Red Hat OpenShift AI (RHOAI). It is built on controller-runtime and implements an OLM-managed operator that reconciles two primary cluster-scoped CRDs: `DSCInitialization` (platform initialization and service configuration) and `DataScienceCluster` (component lifecycle management). The operator uses a modular architecture where each AI/ML component (Dashboard, KServe, Kueue, Data Science Pipelines, Model Registry, TrustyAI, Workbenches, Ray, Training Operator, Feast Operator, LlamaStack Operator, MLflow Operator, Trainer, Spark Operator, Models-as-a-Service) has its own sub-controller that deploys and manages component-specific manifests from prefetched kustomize bundles.

In RHOAI 3.x, the operator deploys critical platform ingress infrastructure through its Gateway service controller. This controller creates a full Gateway API stack in the `openshift-ingress` namespace: a `GatewayClass`, a `Gateway` CR (`data-science-gateway`), an `EnvoyFilter` for authentication, and `kube-auth-proxy` deployment for OAuth2/OIDC authentication. Component controllers (in other repositories) then create `HTTPRoute` CRs that reference this parent Gateway. The operator also manages an Auth service controller (RBAC group management), a CertConfigmapGenerator (CA bundle distribution), a Monitoring service (Prometheus, OpenTelemetry, Tempo, Perses), and admission webhooks for all key CRDs.

The operator is built with RHOAI-specific build tags (`-tags rhoai`) and bundles all component manifests into `/opt/manifests/` within the container image. It supports both OpenShift integrated OAuth and external OIDC authentication, with two ingress modes: `OcpRoute` (ClusterIP + OpenShift Routes) and `LoadBalancer` (cloud/MetalLB).

**Gateway API Architecture (RHOAI 3.x)**:
- The Gateway controller deploys a `GatewayClass` (`data-science-gateway-class`) referencing the OpenShift Gateway controller (`openshift.io/gateway-controller/v1`)
- A `Gateway` CR (`data-science-gateway`) is created in `openshift-ingress` namespace with HTTPS listeners on port 443
- The `EnvoyFilter` (`data-science-authn-filter`) injects ext_authz + Lua filters into the Envoy proxy to enforce authentication via kube-auth-proxy
- The `kube-auth-proxy` deployment handles OAuth2/OIDC authentication, supporting both OpenShift integrated OAuth and external OIDC providers
- Component controllers (e.g., notebook controller, dashboard) create `HTTPRoute` CRs in their namespaces referencing `data-science-gateway` as parent
- Routes from two namespaces are allowed: `openshift-ingress` and the application namespace (e.g., `redhat-ods-applications`)
- Legacy hostname redirect: If subdomain changed from `data-science-gateway`, a second Gateway listener accepts traffic on the old hostname and the EnvoyFilter redirects to the new one
- Dashboard redirect resources (nginx Deployment + ConfigMap + Service + OpenShift Routes) redirect old dashboard/gateway URLs to the new Gateway API URL
