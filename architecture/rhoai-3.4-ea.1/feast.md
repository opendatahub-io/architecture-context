## Purpose
**Short**: Kubernetes operator that manages the lifecycle of Feast feature store deployments, including online/offline store servers, registry, UI, and CronJob-based materialization.

**Detailed**: The Feast Operator is a controller-runtime based Kubernetes operator that reconciles `FeatureStore` custom resources to deploy and manage Feast feature store infrastructure. It manages the deployment of multiple feast services (online store, offline store, registry, UI) as containers within a single Deployment, along with supporting resources like Services, ConfigMaps, PersistentVolumeClaims, CronJobs, and RBAC resources.

The operator supports multiple persistence backends (file-based with PVC, and database-backed for stores like PostgreSQL, Snowflake, Redis, etc.), automatic TLS configuration on OpenShift (via service serving certificates), Kubernetes and OIDC-based authorization, and remote registry federation across namespaces. It also includes a secondary controller (`NotebookConfigMapReconciler`) that watches Kubeflow Notebook CRs to automatically provision feast client configuration ConfigMaps for notebooks with feast integration labels.

On OpenShift/RHOAI, the operator automatically configures TLS for all services using OpenShift service serving certificates, creates OpenShift Routes for the UI service, manages a namespace registry ConfigMap for cross-namespace feast discovery, and supports custom CA bundle injection from the `odh-trusted-ca-bundle` ConfigMap. The operator is built with FIPS-compliant Go runtime (`GOEXPERIMENT=strictfipsruntime`).

## Kustomize Deployment

The operator is deployed by rhods-operator using the RHOAI overlay at `infra/feast-operator/config/overlays/rhoai/kustomization.yaml`. This overlay:
- Sets namespace to `redhat-ods-applications`
- References `../../default` which includes CRDs, RBAC, manager, and metrics service
- Uses `configMapGenerator` with `params.env` to inject image references
- Replaces operator image and RELATED_IMAGE environment variables in the controller-manager Deployment via kustomize `replacements`

Image parameters injected by rhods-operator at deploy time:
- `RELATED_IMAGE_FEAST_OPERATOR` -> `RELATED_IMAGE_ODH_FEAST_OPERATOR_IMAGE`
- `RELATED_IMAGE_FEATURE_SERVER` -> `RELATED_IMAGE_ODH_FEATURE_SERVER_IMAGE`
