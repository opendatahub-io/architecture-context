## Purpose

**Short**: Feast is a feature store for machine learning that provides an operator-managed deployment of offline stores, online stores, registry servers, and a web UI for managing and serving ML features on Kubernetes.

**Detailed**: Feast (Feature Store) is an open-source platform that bridges the gap between data engineering and machine learning by providing a consistent interface for managing, storing, and serving features. The system consists of a Python SDK for feature definition and transformation, a Go-based high-performance feature server for online serving, and a Kubernetes operator that orchestrates the deployment of the full feature store stack.

The Feast operator (`feast-operator`) watches `FeatureStore` custom resources and deploys the appropriate combination of services: an offline store server (for batch/historical feature retrieval via Arrow Flight), an online store server (for low-latency feature serving via HTTP/REST), a registry server (for metadata management via gRPC and/or REST), and optionally a web UI. Each FeatureStore CR is reconciled into a single Deployment with multiple containers (one per enabled service), along with associated Services, ConfigMaps, PersistentVolumeClaims, ServiceAccounts, and RBAC resources. The operator supports both Kubernetes RBAC and OIDC-based authorization models.

On OpenShift (including RHOAI), the operator automatically provisions TLS certificates using OpenShift service serving certificates, creates OpenShift Routes for the UI, and integrates with the OpenDataHub/RHOAI trusted CA bundle. The operator also manages a namespace registry ConfigMap and a Notebook ConfigMap controller that integrates with Kubeflow Notebooks for seamless feature store discovery.

## Architectural Analysis

The Feast operator employs a **single-Deployment multi-container** pattern: rather than creating separate Deployments for each service type (online, offline, registry, UI), all enabled services run as separate containers within a single Deployment pod. This simplifies lifecycle management and ensures all services share the same feature_store.yaml configuration (injected via a base64-encoded environment variable), but creates a coupling where restarting one service restarts all services in the pod. The operator uses a `Recreate` deployment strategy by default, which means brief downtime during updates.

**TLS automation on OpenShift** is a notable design strength. When running on OpenShift, the operator detects the platform at startup via API server group discovery (`route.openshift.io`) and automatically provisions TLS for all feast services using OpenShift service serving certificates. Each service gets a `service.beta.openshift.io/serving-cert-secret-name` annotation, and the operator mounts the generated TLS secrets into the pod. For registry services with both gRPC and REST endpoints enabled, the operator creates a single TLS certificate with Subject Alternative Names (SANs) covering both service hostnames. This eliminates the need for users to manually manage certificates, though they can disable auto-TLS and provide their own secrets.

The **Notebook ConfigMap controller** is a sophisticated integration point: it watches Kubeflow Notebooks across all namespaces for a specific label (`opendatahub.io/feast-integration: "true"`) and annotation (`opendatahub.io/feast-config`), then automatically creates per-notebook ConfigMaps containing the `feature_store.yaml` client configuration for each referenced feast project. This controller also watches FeatureStore CRDs to propagate configuration changes to affected notebook ConfigMaps, and watches the ConfigMaps themselves to restore them if modified or deleted. The CRD existence check at startup (`validateNotebookCRD`) makes this integration gracefully optional — the controller is skipped if the Notebook CRD is not installed.

The **namespace registry** pattern provides cluster-wide feast service discovery. The operator maintains a `feast-configs-registry` ConfigMap in the operator namespace that maps namespaces to their FeatureStore client ConfigMap names. A `system:authenticated` RoleBinding ensures any authenticated user can read this registry, enabling clients to discover available feature stores without listing CRDs across namespaces.

One architectural risk is the **CronJob execution model**: CronJobs use `kubectl exec` to run feast commands inside the running Deployment pod rather than running feast commands in their own containers. This requires `pods/exec` RBAC permissions and creates a dependency on the running Deployment being available. If the Deployment is scaled to zero or unavailable, CronJobs will fail. The default CronJob configuration ships suspended (`@yearly` with `suspend: true`) to avoid unexpected execution.
