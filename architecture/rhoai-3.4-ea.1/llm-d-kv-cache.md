## Purpose

**Short**: A pluggable KV-Cache indexing and scoring library that enables cache-aware routing for distributed vLLM inference fleets.

**Detailed**: The llm-d-kv-cache project is the KV-Cache management component of the llm-d distributed LLM inference platform. It maintains a global, near-real-time view of KV-Cache block locality across a fleet of vLLM pods by ingesting structured events (BlockStored, BlockRemoved) over ZMQ pub/sub. This metadata enables intelligent routing — schedulers query the indexer to score pods based on how much of a prompt's KV-cache is already resident on each pod, minimizing redundant computation and improving Time To First Token (TTFT).

The project is primarily a Go library consumed by the `llm-d-inference-scheduler`, but it also ships a standalone Python gRPC tokenizer sidecar (`uds_tokenizer`) and a Python/CUDA KV connector backend (`llmd_fs_backend`) for shared-storage offloading. The Go library uses CGO to embed Python's vLLM chat template rendering, and communicates with the tokenizer sidecar over Unix Domain Sockets. The index backend is pluggable: in-memory LRU (default), cost-aware Ristretto, or distributed Redis/Valkey.

The project plays a critical infrastructure role in RHOAI's LLM serving stack, sitting between the vLLM inference fleet (which publishes cache events) and the inference scheduler (which queries for cache-aware pod scores).

## Architectural Analysis

**Modular library design with pluggable backends.** The llm-d-kv-cache is architected as a composable Go library rather than a standalone service. Each major concern — indexing, event processing, tokenization, scoring — is encapsulated behind Go interfaces with multiple implementations. The `kvblock.Index` interface has three backends (in-memory LRU, cost-aware Ristretto, Redis/Valkey), the tokenization system has a composite fallback chain (local → UDS → HuggingFace), and the scoring strategy is pluggable (currently only LongestPrefixMatch). This design allows the consuming scheduler to embed the library with configuration tuned to its deployment constraints, avoiding unnecessary network hops for what is fundamentally a latency-sensitive scoring operation.

**ZMQ-based event ingestion with sharded ordering guarantees.** The write path uses a carefully designed sharded worker pool where each vLLM pod's events are consistently routed to the same worker shard via FNV-32a hashing of the pod identifier. This ensures per-pod event ordering without requiring a global lock on the index. The ZMQ subscriber uses a 3-part message protocol (topic, sequence number, MessagePack payload) that is directly compatible with vLLM's KV-event publisher format. The topic format `kv@{pod-id}@{model-name}` allows both pod-level filtering and model-aware indexing.

**Cross-language embedding for vLLM compatibility.** A notable architectural decision is embedding Python via CGO for chat template rendering. The `preprocessing` package calls into vLLM's Python chat template logic through a C FFI bridge, ensuring bit-perfect compatibility with vLLM's template rendering. This avoids reimplementing vLLM's Jinja2-based template engine in Go. The trade-off is build complexity (requires Python development headers, specific vLLM version) and the `embedded_tokenizers` build tag that gates this functionality. The UDS tokenizer sidecar provides an alternative path that avoids CGO at the cost of a gRPC hop over Unix Domain Socket.

**Hash compatibility is critical.** The token-to-block-key pipeline uses FNV-64a hashing with CBOR canonical encoding, precisely matching vLLM's prefix caching algorithm. The `PYTHONHASHSEED` environment variable must be synchronized between vLLM pods and the kv-cache-manager to ensure deterministic hashing. This is a fragile coupling — a mismatch would silently produce zero cache hits. The Helm chart addresses this by setting `PYTHONHASHSEED=0` on both containers.

**Security posture is minimal by design.** All network protocols (ZMQ, gRPC over UDS, Redis) use plaintext within the cluster. There is no authentication on the ZMQ event stream or gRPC tokenizer service. This is acceptable for cluster-internal communication but relies entirely on Kubernetes network policies and namespace isolation for security. The Redis/Valkey backend optionally supports TLS (`rediss://`) and URL-embedded credentials, but these are not enabled by default. The tokenizer sidecar runs as non-root (UID 1001 in Konflux, UID 65532 in upstream Dockerfile) with restricted filesystem permissions on the UDS socket directory.
