## Platform Overview

Red Hat OpenShift AI (RHOAI) 3.4.0-ea.1 is a comprehensive AI/ML platform built on OpenShift Container Platform that provides an integrated environment for the complete machine learning lifecycle. The platform enables data scientists and ML engineers to develop models in Jupyter notebook workbenches, orchestrate training and inference pipelines, serve models at scale, track experiments and model metadata, and enforce AI safety guardrails -- all within a Kubernetes-native, enterprise-grade environment with FIPS-compliant cryptography and disconnected deployment support.

The platform is structured as a layered operator architecture. At the top sits the **rhods-operator**, an OLM-managed operator that orchestrates the deployment and lifecycle of 17+ sub-component operators and services via kustomize overlays. These components span six major functional domains: **Workbenches** (Jupyter notebooks, code-server, RStudio), **Model Serving** (KServe, vLLM, OpenVINO, MLServer with Gateway API ingress), **ML Pipelines** (Data Science Pipelines built on Argo Workflows with MLMD lineage tracking), **Distributed Training** (Kubeflow Training Operator, Trainer v2, KubeRay, Kueue job queuing), **Model Management** (Model Registry, MLflow experiment tracking, model metadata catalog), and **AI Safety** (TrustyAI explainability, FMS Guardrails Orchestrator, NeMo Guardrails, evaluation hub).

RHOAI 3.4 introduces several new capabilities including llm-d inference scheduling with KV-cache-aware routing and Prefill/Decode disaggregation, Llama Stack distribution for unified AI APIs, the Feast feature store operator, batch inference gateway, Models-as-a-Service with tiered access control via Kuadrant, and the Workload Variant Autoscaler for LLM-aware scaling. The platform ships 84 container images across the 43 source repositories, all built via Konflux CI/CD pipelines on UBI9 base images. Networking in RHOAI 3.x has migrated from Istio service mesh to Gateway API (via OpenShift's `data-science-gateway`) for ingress, with kube-rbac-proxy sidecars providing authentication.

### Central Components

Components with the most dependents -- these are core platform services:

1. **rhods-operator** - Platform orchestrator; every component depends on it for deployment
2. **kserve** - Central model serving infrastructure; used by dashboard, model-controller, model-registry, DSP, vLLM runtimes, llm-d scheduler, MaaS
3. **Kubernetes API Server** - All operators and controllers interact via the K8s API
4. **Gateway API (data-science-gateway)** - Central ingress for notebooks, model serving, dashboard, MLflow, MaaS
5. **kube-auth-proxy** - Authentication sidecar used by dashboard, notebooks, model registry, kuberay, MLflow
6. **odh-dashboard** - Primary UI aggregating data from nearly all components
7. **trustyai-service-operator** - Multi-controller operator managing 5 distinct services
8. **data-science-pipelines-operator** - Manages the entire DSP stack (6+ sub-components)

## Deployment Architecture

### Deployment Topology

The platform follows a hub-and-spoke deployment model:

- **Hub namespace** (`redhat-ods-applications`): Contains all operator controllers, the dashboard, and shared services. The rhods-operator runs here and deploys sub-operator controllers.
- **User namespaces**: Individual data science projects where users create workloads. Each namespace can contain Notebook pods, DSPA instances, InferenceService instances, FeatureStore instances, and Model Registry instances.
- **Gateway namespace** (`openshift-ingress`): Houses the `data-science-gateway` Gateway resource that serves as the parent for all HTTPRoute-based ingress.
- **Monitoring namespace** (`redhat-ods-monitoring`): Prometheus stack for platform observability.

### OLM Deployment

The platform is installed via OLM (Operator Lifecycle Manager):
1. The `rhods-operator` CSV (ClusterServiceVersion) is assembled by RHOAI-Build-Config from 43 source repositories
2. The CSV declares 84 `relatedImages` for disconnected environment support
3. OLM deploys the rhods-operator, which then deploys all sub-components
4. Users create a `DataScienceCluster` CR to enable/disable individual components

### Resource Requirements

Components are deployed with the following typical resource profiles:

| Tier | Components | CPU Request | Memory Request |
|------|-----------|-------------|----------------|
| Operators | rhods-operator, kserve, DSPO, model-registry-operator, etc. | 100-500m | 128-512Mi |
| Services | odh-dashboard, DSP API server, model-registry, MLflow | 250m-1 | 256Mi-1Gi |
| Inference Runtimes | vLLM, OpenVINO, MLServer | 1-8+ (GPU-dependent) | 2Gi-64Gi+ |
| Training Runtimes | CUDA/ROCm training images | GPU-dependent | GPU-dependent |
| Data Stores | MariaDB, MinIO, Redis, PostgreSQL | 250m-1 | 256Mi-2Gi |
