## Platform Overview

Red Hat OpenShift AI (RHOAI) 3.4-ea.1 is a comprehensive AI/ML platform built on OpenShift Container Platform that provides end-to-end capabilities for the machine learning lifecycle. The platform enables data scientists and ML engineers to develop models in interactive notebook workbenches, orchestrate training and data processing pipelines, serve models at scale with autoscaling inference endpoints, track model metadata and artifacts in registries, evaluate LLM performance, enforce AI safety guardrails, and manage distributed computing workloads across GPU-accelerated clusters.

The platform is architected as a collection of Kubernetes operators orchestrated by a central platform operator (RHODS Operator). In RHOAI 3.x, the platform has adopted a Gateway API-based ingress architecture with a centralized `data-science-gateway` that replaces the legacy per-component OpenShift Route + oauth-proxy pattern from RHOAI 2.x. Authentication flows through an EnvoyFilter ext_authz integration with a `kube-auth-proxy` deployment supporting both OpenShift OAuth and external OIDC providers. Individual components create HTTPRoute CRs referencing this central gateway, while `kube-rbac-proxy` sidecars provide per-service RBAC authorization via Kubernetes SubjectAccessReview.

The platform spans model development (Notebooks, VS Code, RStudio), model training (Training Operator, Trainer, KubeRay), model serving (KServe with serverless and raw deployment modes, LLMInferenceService for disaggregated LLM serving), model management (Model Registry, Model Catalog), data pipelines (Data Science Pipelines with Argo Workflows), feature engineering (Feast), AI safety (TrustyAI guardrails, NeMo guardrails, LLM evaluation), distributed computing (Ray, Spark), and emerging AI infrastructure (LlamaStack). All components are built with FIPS-compliant Go tooling on UBI9 base images and deployed via Konflux CI/CD pipelines.

### Central Components

1. **RHODS Operator** - The platform orchestrator; all components depend on it for deployment and lifecycle management
2. **data-science-gateway** - Central ingress gateway; Dashboard, Notebooks, KubeRay, and other components create HTTPRoutes referencing it
3. **KServe** - Core model serving infrastructure; Dashboard, Model Controller, Pipelines, and TrustyAI interact with InferenceService CRDs
4. **ODH Dashboard** - Central UI hub; consumes APIs from nearly every other component
5. **kube-auth-proxy / kube-rbac-proxy** - Authentication and authorization backbone used across all components

### Integration Patterns

| Pattern | Description | Components Using |
|---------|-------------|------------------|
| CRD Watch/Reconcile | Controller watches CRDs and reconciles dependent resources | All operators |
| Gateway API HTTPRoute | Components create HTTPRoute CRs referencing data-science-gateway | Dashboard, Notebooks, KubeRay |
| kube-rbac-proxy Sidecar | RBAC auth proxy injected into pods for SubjectAccessReview | Dashboard, Notebooks, Model Registry, TrustyAI, KubeRay, DSPO |

## Deployment Architecture

### Deployment Topology

The platform follows a hub-and-spoke deployment model:

- **Hub (redhat-ods-applications)**: All platform operators and central services deploy here. This includes the RHODS operator, dashboard, notebook controller, model registry operator, DSPO, odh-model-controller, feast operator, llama-stack operator, and trustyai operator.
- **Hub (openshift-ingress)**: The data-science-gateway and kube-auth-proxy deploy here for centralized ingress and authentication.
- **Hub (kserve)**: KServe controller and webhook deploy in a dedicated namespace.
- **Hub (opendatahub)**: Training-related operators (Training Operator, Trainer, Spark) deploy here.
- **Spoke (user namespaces)**: Each data science project gets its own namespace where user workloads run: notebooks, DSPA instances, InferenceServices, Ray clusters, training jobs, feature stores, etc.

### Resource Requirements

Component operators are lightweight Go binaries with typical resource footprints:
- Operator pods: 100-500m CPU, 128-512Mi memory
- kube-rbac-proxy sidecars: 100m CPU, 64Mi memory (default)
- User workloads (notebooks, training, serving): User-configurable via HardwareProfiles
- GPU workloads: Require NVIDIA CUDA 12.6-13.0 or AMD ROCm 6.2-6.4 capable nodes

## Platform Maturity

- **Total Components**: 15
- **Operator-based Components**: 13 (all except Notebooks images and Dashboard web app)
- **Gateway API Coverage**: ~40% (Dashboard, Notebooks, KubeRay migrated; Model Registry, DSPO, TrustyAI still on Routes)
- **mTLS Enforcement**: MIXED (KubeRay via cert-manager, KServe via Istio, DSPO via OpenShift service-ca, most services PERMISSIVE)
- **Authentication Pattern**: Transitioning from oauth-proxy (2.x) to kube-rbac-proxy (3.x)
- **CRD API Versions**: v1, v1beta1, v1alpha1 (most new APIs at v1alpha1)
- **Base Images**: UBI9 (Red Hat Universal Base Image 9)
- **Go Versions**: 1.23-1.25 across components
- **Build System**: Konflux (Tekton-based CI/CD) with hermetic builds
- **FIPS Compliance**: Built with `GOEXPERIMENT=strictfipsruntime`
- **Total CRDs Managed**: ~55+ distinct CRD kinds across all components
