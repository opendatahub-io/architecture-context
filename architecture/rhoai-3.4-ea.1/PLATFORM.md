## Platform Overview

Red Hat OpenShift AI (RHOAI) 3.4-ea.1 is a Kubernetes-native AI/ML platform deployed on OpenShift Container Platform via Operator Lifecycle Manager (OLM). The platform is orchestrated by the `rhods-operator`, a meta-operator that manages 16+ component operators and services through a two-tier Custom Resource hierarchy: a top-level `DataScienceCluster` CR drives creation of per-component CRs (Dashboard, KServe, Kueue, ModelRegistry, etc.), each reconciled by its own controller. This registry-based extensibility model allows new platform capabilities to be added by implementing a `ComponentHandler` interface and registering in the component registry.

The RHOAI 3.x ingress architecture represents a major evolution from 2.x, centered on Kubernetes Gateway API. The rhods-operator deploys a multi-layer ingress stack: a `data-science-gateway` Gateway CR in the `openshift-ingress` namespace backed by an Envoy proxy data plane. Authentication is enforced at the Envoy layer via an `ext_authz` EnvoyFilter that delegates to the `kube-auth-proxy` service, which supports both OpenShift-native IntegratedOAuth and external OIDC providers. Legacy URL compatibility is maintained through nginx-based redirect services and OpenShift Routes that 301-redirect old hostnames to the new Gateway API hostname. Cross-namespace HTTPRoute routing enables centralized routing rules that are protected from user modification through ReferenceGrant CRs.

The platform's security posture is defense-in-depth: kube-rbac-proxy sidecars enforce SubjectAccessReview authorization on user-facing services, NetworkPolicies restrict pod-to-pod traffic, OpenShift service-CA provides automated TLS certificate provisioning, and hardware profile webhooks inject GPU/CPU resource constraints and Kueue queue labels into workloads. Container images are predominantly built on UBI9 base images with non-root execution (USER 1001 or 65532), and the platform supports FIPS 140-2 compliant cryptography and disconnected (air-gapped) deployments via the RELATED_IMAGE pattern and OLM relatedImages.

### Central Components

**KServe** is the most depended-upon component (11 reverse dependencies), serving as the universal model serving layer. Every inference runtime (vLLM, OpenVINO, MLServer, vLLM-CPU, vLLM-Gaudi) deploys through KServe's InferenceService CRD. The new LLMInferenceService CRD extends this for LLM-specific workloads with disaggregated prefill/decode and llm-d integration.

**rhods-operator** is the platform orchestrator (6 direct deployment relationships, 205 controller watches). It deploys and configures all other operators and services, manages the Gateway API ingress stack, and provides the DataScienceCluster/DSCInitialization CRD hierarchy that defines the platform topology.

**odh-dashboard** has the most outgoing dependencies (15), acting as the unified user interface connecting to pipelines, model serving, model registry, notebooks, feature stores, MLflow, and platform configuration.

**Data Science Pipelines Operator (DSPO)** is a key integration hub (10 outgoing dependencies), orchestrating the pipeline execution stack including Argo Workflows, ML Metadata, MariaDB, and MinIO, with connections to KServe, notebooks, and KubeRay for pipeline steps.

**TrustyAI Service Operator** manages five distinct controllers (TrustyAI, LMES, GORCH, NEMO, EvalHub) from a single binary, making it the most functionally diverse operator with 9 outgoing dependencies spanning model explainability, LLM evaluation, and guardrails orchestration.

## Deployment Architecture

### Deployment Topology

The platform follows a three-tier namespace model. The **operator tier** (`redhat-ods-operator`) contains the rhods-operator and admission webhooks that intercept cluster-wide resource creation. The **platform tier** (`redhat-ods-applications`) hosts all component controllers, the dashboard, and shared platform services. The **workload tier** consists of user-created namespaces (data science projects) where notebooks, inference services, pipeline runs, Ray clusters, and training jobs execute. A separate **ingress tier** (`openshift-ingress`) hosts the Gateway API infrastructure, keeping the data plane isolated from both operator and application namespaces.

Cross-tier communication flows through well-defined interfaces: the Gateway API for external traffic, Kubernetes API for control plane operations, and direct Service-to-Service for internal data plane traffic. The rhods-operator's component controllers create resources across all tiers, but workload-tier resources are created with least-privilege RBAC scoped to individual namespaces.

### Disconnected Support

RHOAI supports air-gapped (disconnected) deployment through several mechanisms. All operator images use the `RELATED_IMAGE_*` environment variable pattern, which OLM resolves through the ClusterServiceVersion's `relatedImages` section to support image mirroring via `oc-mirror`. Container image references use SHA256 digests rather than tags for reproducibility and mirroring compatibility. The rhods-operator distributes the `odh-trusted-ca-bundle` ConfigMap to all managed namespaces, enabling components to trust custom CA certificates for internal registries. The Kubeflow Notebook Controller and other components inject this CA bundle into workload pods automatically. The platform's `operator_features` metadata confirms `disconnected: true` support.

## Platform Architectural Analysis

### The Meta-Operator Pattern: Power and Complexity

RHOAI's defining architectural decision is the meta-operator model, where rhods-operator orchestrates 13+ component operators through a two-tier CR hierarchy. This provides a clean user experience (a single DataScienceCluster CR to configure the entire platform) and a clean extensibility model (new components implement ComponentHandler and register). However, it creates deep controller nesting — rhods-operator creates component CRs, which trigger component controllers, which create workload resources, which may trigger further controllers (e.g., KServe creating InferenceServices that trigger odh-model-controller). Debugging failures requires tracing through 3-4 levels of controller reconciliation, and the 205 controller watches in rhods-operator alone represent a significant cognitive and operational complexity surface.

### Authentication Architecture: Convergence on kube-rbac-proxy

The platform's most consistent cross-cutting pattern is the kube-rbac-proxy sidecar model, used by at least 7 major components. This creates a uniform security layer: every user-facing service is fronted by a sidecar that performs TokenReview + SubjectAccessReview before proxying to a localhost-bound application port. The pattern's strength is its transparency to application code — services don't implement auth logic — but it introduces operational overhead (double the containers, certificates to manage) and a potential latency penalty on every request. Notably, the Gateway API ext_authz pattern (kube-auth-proxy at the Envoy level) provides a *second* authentication layer for Gateway-routed services, meaning some requests pass through two independent auth checks. This defense-in-depth is intentional but adds complexity to troubleshooting auth failures.

### The LLM Serving Bifurcation

RHOAI 3.4 exhibits a clear architectural bifurcation between traditional ML model serving and LLM-optimized serving. The traditional path (InferenceService → KServe → MLServer/OVMS) focuses on multi-model serving, scale-to-zero, and framework-agnostic deployment. The LLM path introduces an entirely new stack: LLMInferenceService → llm-d → vLLM with disaggregated prefill/decode, KV cache-aware routing (llm-d-inference-scheduler), distributed KV cache transfer (llm-d-kv-cache via NIXL/LMCache), and saturation-based autoscaling (WVA). These two paths share the KServe controller but diverge significantly in networking (Gateway API Inference Extension vs standard HTTPRoutes), scaling (WVA vs HPA/Knative), and operational complexity. The guardrails subsystem (FMS Guardrails Orchestrator + vllm-orchestrator-gateway + detectors) adds another layer specific to the LLM path.

### Observability: Comprehensive but Heterogeneous

Prometheus metrics exposure is nearly universal (22+ components expose `/metrics`), and ServiceMonitor/PodMonitor CRDs automate scrape configuration. However, distributed tracing support is inconsistent: MLServer exports traces via OpenTelemetry OTLP, KServe deploys OpenTelemetry collectors per InferenceService, and the FMS Guardrails Orchestrator uses OpenTelemetry — but most operators and controllers lack tracing instrumentation. This makes end-to-end request tracing across the platform difficult, particularly for multi-hop workflows like guardrailed inference (client → gateway → orchestrator → detectors + vLLM → response).

### Configuration Drift Risk Between Distributions

The ODH/RHOAI distribution variant pattern (separate kustomize overlays) creates a maintenance burden and drift risk. Each component maintains parallel overlay directories where RHOAI adds stricter security defaults (FIPS enforcement, restricted evaluation capabilities, Red Hat registry images). As components evolve, these overlays must be kept in sync — a divergence in an RHOAI overlay that isn't caught during upstream rebasing could silently weaken security posture or break functionality. The rhods-operator's `get_all_manifests.sh` script pins component manifests to specific commit SHAs, which provides reproducibility but means manifest updates require explicit version bumps.

### Emerging Llama Stack Integration

The Llama Stack ecosystem (ogx-distribution, ogx-k8s-operator, llama-stack-provider-ragas, llama-stack-provider-trustyai-garak) represents an emerging architectural layer that provides a unified API gateway for inference, agents, evaluation, RAG, and safety. This overlaps with existing capabilities (KServe for inference, EvalHub/lm-evaluation-harness for evaluation, FMS Guardrails for safety) but targets a different user persona — developers who want a single API surface rather than composing individual platform components. The coexistence of these parallel approaches suggests the platform is in a transitional period, potentially consolidating around Llama Stack for developer-facing use cases while retaining the lower-level components for platform operators.

### Container Security: Strong Baseline with Gaps

The container security posture is strong overall: 76% of Konflux-built images run as non-root (UID 1001 or 65532), UBI9 base images provide certified foundations, and multi-stage builds minimize attack surface. However, 7 production Dockerfiles still run as root, 4 lack USER directives entirely, and 107 Dockerfiles across the codebase (including non-Konflux) have no USER directive. Unpinned base images remain common (65 "base" references, 53 "${BASE_IMAGE}" references), which could introduce supply chain risks if base image tags are reassigned. The FIPS compliance story is credible but not comprehensive — kube-auth-proxy and kube-rbac-proxy enforce FIPS cipher suites, but not all TLS-terminating services explicitly configure FIPS-only cipher lists.
