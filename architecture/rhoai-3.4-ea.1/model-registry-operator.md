## Purpose

**Short**: Manages the lifecycle of Model Registry custom resources and their associated Kubernetes resources for storing and serving ML model metadata in Red Hat OpenShift AI.

**Detailed**: The Model Registry Operator is a Kubernetes operator built with controller-runtime that manages `ModelRegistry` custom resources. Each ModelRegistry CR declaratively defines a model registry instance, including its database backend (PostgreSQL or MySQL), authentication proxy configuration, and network exposure settings. The operator reconciles these CRs by creating and managing a set of Kubernetes resources — Deployments running the model-registry REST service, Services, ServiceAccounts, RBAC Roles/RoleBindings, OpenShift Routes, NetworkPolicies, and authentication sidecars — to provide a fully operational model metadata service.

The operator supports two API versions: v1alpha1 (deprecated) and v1beta1 (current storage version), with automatic webhook-driven conversion between them. A key architectural evolution is the migration from Istio-based service mesh integration and OAuth proxy authentication (v1alpha1 patterns) to kube-rbac-proxy sidecar authentication (v1beta1), which provides Kubernetes-native RBAC authorization without requiring a service mesh. The operator also includes a Model Catalog controller that manages a shared catalog deployment with its own PostgreSQL database, RBAC, and kube-rbac-proxy authentication. Storage version migration between CRD versions is handled transparently via the Kubernetes StorageVersionMigration API with a manual fallback strategy.

## Architectural Analysis

The Model Registry Operator follows a mature controller-runtime pattern with several notable architectural decisions. The operator manages two distinct but related controllers — the ModelRegistry controller (per-instance) and the ModelCatalog controller (singleton) — sharing a common template rendering engine and resource management utilities. Both controllers use Go templates embedded via `//go:embed` to define the Kubernetes resources they create, which provides a clean separation between resource structure and reconciliation logic.

The authentication architecture has undergone a significant evolution across API versions. The v1alpha1 API supported Istio service mesh integration with Authorino-based AuthConfig, which was first deprecated in favor of OAuth proxy sidecars, and then further migrated to kube-rbac-proxy in v1beta1. This three-stage migration is handled transparently: the v1alpha1 webhook automatically converts Istio config to OAuth proxy config, while the v1beta1 webhook automatically converts OAuth proxy config to kube-rbac-proxy config. The controller also performs runtime migration, ensuring existing CRs are updated. The kube-rbac-proxy sidecar uses Kubernetes SubjectAccessReview to authorize requests — users must have `get` permission on the registry's Service resource — which integrates naturally with Kubernetes RBAC without requiring external auth infrastructure.

A noteworthy design pattern is the operator's BYOIDC (Bring Your Own Identity Provider) detection. On OpenShift clusters where `user.openshift.io` API is not available (external IdP mode), the operator gracefully skips Group creation and falls back to external IdP group references in RoleBindings. This is detected at startup via the discovery API and propagated through the `IsBYOIDC()` method, allowing a single operator binary to handle both traditional OpenShift and externally-managed identity scenarios.

The ModelCatalog controller uses a deduplication strategy where all watched resource changes map to a single fixed reconcile request (`model-catalog` in the target namespace), preventing reconcile storms when multiple catalog resources change simultaneously. The catalog integrates with the platform's Auth CR (`services.platform.opendatahub.io/v1alpha1/Auth`) to discover admin groups, creating an admin RoleBinding dynamically when groups are configured. The catalog also manages a "default sources" migration that removes deprecated default catalog entries from user-managed ConfigMaps on first reconciliation.

The storage version migration system demonstrates defense-in-depth: it first attempts migration via the Kubernetes StorageVersionMigration API (a cluster-level resource that triggers server-side migration), and falls back to a manual strategy that re-reads and re-writes every ModelRegistry CR to force conversion through the webhook. Migration state is tracked via annotations on each CR to prevent duplicate processing. The 30-second startup delay before migration checks ensures the operator is fully initialized before attempting CRD modifications.
