## Purpose
**Short**: Kubernetes operator that manages the lifecycle of Model Registry instances and a Model Catalog service on OpenShift/Kubernetes clusters.

**Detailed**: The Model Registry Operator provides automated deployment and lifecycle management of Model Registry custom resources. Each ModelRegistry CR represents a metadata registry service backed by PostgreSQL or MySQL that stores ML model metadata using the ml-metadata (MLMD) protocol. The operator creates and manages Deployments, Services, ServiceAccounts, RBAC resources, OpenShift Routes, NetworkPolicies, and kube-rbac-proxy sidecars for each registry instance.

The operator includes two controllers: the ModelRegistryReconciler which manages individual ModelRegistry CR instances (creating per-instance deployments, services, routes, and auth proxies), and the ModelCatalogReconciler which manages a singleton model catalog service with its own PostgreSQL database, RBAC, and routing infrastructure. The operator supports platform-aware behavior, detecting OpenShift capabilities at runtime (including BYOIDC mode on OpenShift 4.20+) and conditionally creating platform-specific resources like Routes and Groups. It also handles CRD storage version migration from v1alpha1 to v1beta1.

**Note on ingress pattern**: This operator uses OpenShift Routes with kube-rbac-proxy sidecars (RHOAI 3.x pattern). The kube-rbac-proxy container on port 8443/TCP fronts the model-registry REST container on port 8080/TCP, enforcing Kubernetes RBAC (SubjectAccessReview on `services/get` in the registry namespace) before proxying requests. Routes are created with `reencrypt` TLS termination pointing at the kube-rbac-proxy HTTPS port.

**Legacy OAuth proxy migration**: The controller automatically migrates ModelRegistry CRs with `oauthProxy` configuration to `kubeRBACProxy` configuration (see `modelregistry_controller.go` line 183). OAuth proxy is no longer used.
