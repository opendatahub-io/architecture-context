## Purpose

**Short**: A FIPS-compliant authentication and authorization reverse proxy that provides OIDC, OpenShift OAuth, and Kubernetes RBAC enforcement for RHOAI platform components.

**Detailed**: kube-auth-proxy is a focused authentication proxy derived from oauth2-proxy, designed specifically for Red Hat OpenShift AI (RHOAI) and Open Data Hub (ODH) environments. It acts as a sidecar container that intercepts HTTP requests to protected upstream applications, performing authentication via external OIDC providers or OpenShift's internal OAuth service, and optionally enforcing Kubernetes RBAC authorization via SubjectAccessReview (SAR).

The single container image ships three binaries selected at runtime by a Go entrypoint: `kube-auth-proxy` (OIDC/OAuth authentication proxy), `kube-rbac-proxy` (Kubernetes RBAC authorization proxy), and `entrypoint` (mode selector). The entrypoint defaults to `rbac` mode (running kube-rbac-proxy) and can be switched to `auth` mode (running kube-auth-proxy) via the `PROXY_MODE` environment variable or a positional CLI argument. This dual-binary design allows a single image to serve both the authentication-only use case (replacing legacy oauth-proxy sidecars) and the RBAC authorization use case (replacing legacy kube-rbac-proxy sidecars) depending on deployment configuration.

The project is a drop-in replacement for existing oauth-proxy and kube-rbac-proxy sidecars used throughout the RHOAI platform. It maintains argument compatibility with both upstream projects while consolidating them into a single FIPS-compliant image. All non-OIDC authentication providers (Google, GitHub, Azure, etc.) have been removed to minimize attack surface.

## Architectural Analysis

**Dual-Binary Architecture**: The most distinctive architectural decision is packaging two independent proxy binaries (kube-auth-proxy and kube-rbac-proxy) into a single container image, selected at runtime by a thin Go entrypoint. This consolidation serves RHOAI's need to replace two separate sidecar images (oauth-proxy and kube-rbac-proxy) with a single FIPS-compliant image. The entrypoint uses `os/exec` to exec into the selected binary, passing all remaining CLI arguments through. The default mode is `rbac`, reflecting that most RHOAI component sidecars historically used kube-rbac-proxy. This design minimizes image pull overhead and simplifies CVE remediation across the platform.

**Authentication Priority Chain**: The kube-auth-proxy session loading follows a strict priority order: K8s service account tokens (TokenReview) are validated first, then JWT bearer tokens (OIDC), then OpenShift OAuth bearer tokens (sha256~ format), then HTTP Basic Auth (htpasswd), and finally cookie-based stored sessions. This ordering ensures machine-to-machine authentication (service accounts) takes precedence over human user flows, which is critical for RHOAI where controllers and operators need to call authenticated endpoints programmatically. Each session loader checks if a session was already loaded by a prior loader and short-circuits, implementing a first-match-wins pattern.

**Session Security Model**: Session cookies use a two-layer protection scheme: AES-GCM authenticated encryption for the session payload (with random nonces per encryption), and HMAC-SHA256 signatures with embedded timestamps for cookie integrity and expiration validation. CSRF protection uses MsgPack-serialized state containing separate OAuth state nonces, OIDC nonces, and PKCE code verifiers, all encrypted before being stored as cookies. The per-request CSRF cookie feature (CSRFPerRequest) generates unique cookie names incorporating a state substring, preventing cookie collision during parallel OAuth flows. Session refresh uses distributed locking (via Redis or cookie store) with a 5-second timeout and 10ms retry interval to prevent thundering herd during token refresh.

**kube-rbac-proxy Authorization Chain**: The kube-rbac-proxy uses a union authorizer with a fixed evaluation order: (1) a hardcoded metrics authorizer that allows `system:serviceaccount:openshift-monitoring:prometheus-k8s` to GET /metrics without SAR, (2) static authorization rules from a config file that match on user/verb/resource/namespace, and (3) the Kubernetes SubjectAccessReview API as the final fallback. This three-tier design optimizes for the common case (Prometheus scraping) while allowing fine-grained static rules before incurring the latency of an API server round-trip for SAR. Authentication results are cached for 2 minutes, SAR allow decisions for 5 minutes, and SAR deny decisions for 30 seconds.

**FIPS Compliance Approach**: Both binaries are compiled with `CGO_ENABLED=1`, `GOEXPERIMENT=strictfipsruntime`, and `-tags strictfipsruntime`, ensuring Go's crypto operations delegate to OpenSSL rather than using pure-Go crypto implementations. The base image is UBI9-minimal, which includes FIPS-validated OpenSSL. One notable crypto risk is the continued support for SHA1-hashed htpasswd entries (`crypto/sha1`) alongside bcrypt, inherited from the oauth2-proxy codebase. While SHA1 is used only for password hash comparison (not for TLS or token signing), it is not FIPS-approved for new deployments. The `InsecureSkipVerify` TLS option in the OpenShift provider is configurable but disabled by default.
