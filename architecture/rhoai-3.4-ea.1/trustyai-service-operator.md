## Purpose

**Short**: Kubernetes operator that manages the lifecycle of TrustyAI explainability services, LM evaluation jobs, guardrails orchestration, and NeMo guardrails deployments within OpenShift AI.

**Detailed**: The TrustyAI Service Operator is a multi-controller Kubernetes operator built with controller-runtime that manages five distinct CRD types under the `trustyai.opendatahub.io` API group. It provides the operational backbone for AI model trustworthiness capabilities within RHOAI, including model explainability (TrustyAIService), LLM evaluation benchmarking (LMEvalJob and EvalHub), LLM output guardrails (GuardrailsOrchestrator), and NeMo-based guardrails inference (NemoGuardrails).

The operator deploys and configures complex multi-container pods with kube-rbac-proxy sidecars for authentication, integrates with KServe InferenceServices for model payload interception, supports Istio service mesh with DestinationRules and VirtualServices, and manages OpenShift Routes for external access. It also integrates with Kueue for workload scheduling of evaluation jobs. The operator is FIPS-compliant with `GOEXPERIMENT=strictfipsruntime` build flags and all kube-rbac-proxy configurations enforce FIPS 140-2 approved cipher suites.

## Architectural Analysis

The TrustyAI Service Operator exemplifies a "one operator, many controllers" pattern where a single binary manages five distinct but thematically related CRDs. Each controller (TAS, LMES, GORCH, NEMO, EvalHub) operates independently with its own reconcile loop, templates, and resource lifecycle. Controllers can be selectively enabled via the `--enable-services` flag, and distribution-specific overlays (RHOAI vs ODH) provide different security postures — notably, RHOAI restricts `lmes-allow-online` and `lmes-allow-code-execution` to `false` by default, preventing evaluation jobs from downloading remote datasets or executing arbitrary code.

A consistent architectural pattern across all controllers is the kube-rbac-proxy sidecar injection. Every user-facing service (TrustyAI, orchestrator, gateway, built-in detector, NeMo, EvalHub) is fronted by a kube-rbac-proxy container that performs SubjectAccessReview authorization before proxying requests to the application container listening on localhost. The proxy templates consistently enforce TLS 1.2 minimum and FIPS 140-2 approved cipher suites. This pattern provides a uniform authentication/authorization layer without requiring each service to implement its own auth logic. The application containers bind to 127.0.0.1 (e.g., EvalHub on 127.0.0.1:8080) ensuring they are only accessible through the proxy.

The GuardrailsOrchestrator controller demonstrates sophisticated auto-configuration: it watches KServe InferenceServices and ServingRuntimes to automatically discover generation and detector services, extracts their hostnames, ports, TLS certificates, and protocols, then generates orchestrator and gateway configuration ConfigMaps. Hash-based change detection triggers redeployment only when the configuration actually changes. The orchestrator supports multi-container pods with up to 4 containers (orchestrator, gateway sidecar, built-in regex detector, and kube-rbac-proxy sidecars per exposed endpoint), each with its own port and auth configuration.

The LMES controller has a unique execution model: rather than creating Kubernetes Jobs, it directly creates Pods with a two-container architecture — the main lm-evaluation-harness container runs the evaluation while the ta-lmes-driver sidecar monitors progress via tqdm output parsing and exposes a simple HTTP status API. The controller polls this sidecar at configurable intervals (default 10s) to update the LMEvalJob CR status with real-time progress bars. Input validation is notably thorough, with regex patterns enforcing safe characters for model arguments, task names, S3 paths, git URLs, and custom artifact JSON — specifically blocking shell injection patterns like `$(`, backticks, `&&`, and `||`.

The NeMo Guardrails controller takes a different approach to CA bundle management compared to the TAS controller. While TAS simply detects an existing `odh-trusted-ca-bundle` ConfigMap via label selector, NEMO actively collates CA certificates from three sources (ODH trusted CA, OpenShift serving CA, and user-specified CAs) using an init container that merges them into a single `ca-certificates.crt` file. This design accommodates NEMO's Python-based NeMo Guardrails server, which relies on the system trust store rather than Go's native cert pool.
