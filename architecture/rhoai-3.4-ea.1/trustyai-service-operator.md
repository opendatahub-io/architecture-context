## Purpose
**Short**: Kubernetes operator managing TrustyAI AI explainability services, LM evaluation jobs, guardrails orchestration, NeMo Guardrails, and evaluation hub deployments.

**Detailed**: The TrustyAI Service Operator is a multi-controller Kubernetes operator that manages several AI trustworthiness and safety services within the Red Hat OpenShift AI (RHOAI) platform. It provides five distinct controllers registered as services: TAS (TrustyAI Service), LMES (LM Evaluation Service), GORCH (Guardrails Orchestrator), NEMO_GUARDRAILS (NeMo Guardrails), and EVALHUB (Evaluation Hub).

The TAS controller deploys the TrustyAI explainability service that monitors ML model inference payloads, provides bias metrics, and integrates with ModelMesh/KServe inference services. It manages PVC or database-backed storage, creates OpenShift Routes with kube-rbac-proxy sidecars for authenticated access, and optionally manages Istio DestinationRules and VirtualServices for service mesh integration.

The LMES controller manages LMEvalJob custom resources that orchestrate language model evaluation jobs using the lm-evaluation-harness framework. It creates Pods with a driver sidecar pattern, supports Kueue integration for job queuing, and handles offline evaluation with S3/PVC storage. The GORCH controller manages GuardrailsOrchestrator CRs that deploy the FMS Guardrails Orchestrator with optional built-in detectors and sidecar gateway, auto-configuring from KServe InferenceServices. The NeMo Guardrails controller manages NeMo Guardrails server deployments with CA bundle handling. The EvalHub controller manages a centralized evaluation hub service with PostgreSQL support and MLFlow integration.

| Job Manager Controller | Go Controller | Integrates LMEvalJob with Kueue workload framework for job scheduling and resource management |
| LMES Driver | Go Binary | Sidecar driver binary injected into LMEvalJob pods; manages job lifecycle, collects results via gRPC |
| Operator Manager | Go Binary | Main operator binary; runs all controllers with leader election, health/readiness probes |

## Kustomize Deployment

### Overlay Structure

The operator is deployed by rhods-operator using the kustomize overlay at `config/overlays/rhoai/kustomization.yaml`, which layers on top of `config/base/kustomization.yaml`.

**Base resources include**:
- CRDs: TrustyAIService, LMEvalJob, GuardrailsOrchestrator, EvalHub, NemoGuardrails
- RBAC: ClusterRole, ClusterRoleBinding, ServiceAccount, leader election Role
- Manager: Deployment with all 5 controllers enabled (`--enable-services TAS,LMES,GORCH,NEMO_GUARDRAILS,EVALHUB`)
- Prometheus: ServiceMonitor and metrics Service

**Image parameters** injected by the rhods-operator at deploy time override default images in `params.env`.

### Enabled Services

The operator deployment enables all controllers via the `--enable-services` flag:
- **TAS**: TrustyAI Service (explainability)
- **LMES**: LM Evaluation Service (model evaluation)
- **GORCH**: Guardrails Orchestrator (content safety)
- **NEMO_GUARDRAILS**: NeMo Guardrails (programmable guardrails)
- **EVALHUB**: Evaluation Hub (evaluation management)
