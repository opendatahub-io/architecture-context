## Purpose
**Short**: Manages the lifecycle of Data Science Pipelines (DSP) infrastructure by reconciling `DataSciencePipelinesApplication` (DSPA) custom resources, deploying and configuring all required pipeline execution components per namespace.

**Detailed**: The Data Science Pipelines Operator (DSPO) is a Kubernetes operator built with controller-runtime that watches `DataSciencePipelinesApplication` (DSPA) custom resources. When a DSPA CR is created in a namespace, the operator deploys a complete Kubeflow Pipelines V2 stack consisting of: an API server, persistence agent, scheduled workflow controller, Argo workflow controller, ML Metadata (MLMD) gRPC server with Envoy proxy, and backing storage (MariaDB for metadata, MinIO or external S3 for artifacts).

The operator uses Go templates (located in `config/internal/`) to dynamically generate and apply Kubernetes manifests for each component. It handles database and object storage health checks, TLS configuration (pod-to-pod TLS via OpenShift service-ca), CA bundle management, kube-rbac-proxy sidecar injection for API authentication, OpenShift Route creation for external access, NetworkPolicy creation for security isolation, and webhook management for Kubernetes-based pipeline storage. Each DSPA instance is fully isolated within its namespace and the operator manages the complete lifecycle including cleanup via finalizers.

**Note**: This operator uses the RHOAI 2.x ingress pattern with OpenShift Routes and kube-rbac-proxy sidecars. Routes are created conditionally:
- API Server route: Created when `spec.apiServer.enableOauth=true` (default: true)
- MLMD Envoy route: Created when `spec.mlmd.envoy.deployRoute=true` (default: true)
- MinIO route: Created when `spec.objectStorage.enableExternalRoute=true` (default: false)
- All routes use kube-rbac-proxy sidecars (port 8443) for authentication via SubjectAccessReview
