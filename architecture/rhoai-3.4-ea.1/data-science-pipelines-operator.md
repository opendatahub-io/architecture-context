## Purpose

**Short**: Manages the lifecycle of Data Science Pipelines (Kubeflow Pipelines v2) Custom Resources and all associated Kubernetes resources on OpenShift.

**Detailed**: The Data Science Pipelines Operator (DSPO) is a Kubernetes operator built with controller-runtime that watches `DataSciencePipelinesApplication` (DSPA) custom resources and deploys the complete Kubeflow Pipelines v2 stack within the target namespace. Each DSPA CR triggers reconciliation of up to eight sub-components: the KFP API Server, Persistence Agent, Scheduled Workflow controller, Argo Workflow Controller, ML Metadata (MLMD) gRPC server, MLMD Envoy proxy, MariaDB database, and MinIO object storage.

The operator uses a template-based deployment model via the Manifestival library, rendering Go text/template YAML files from `config/internal/` into Kubernetes resources. It supports both internal (MariaDB + MinIO) and external (any MySQL-compatible DB + S3-compatible storage) backends. The operator manages pod-to-pod TLS using OpenShift service-CA annotations, network isolation via NetworkPolicies, RBAC for each sub-component, and health monitoring through Prometheus metrics and status conditions on the DSPA CR. It also deploys cluster-scoped admission webhooks (mutating and validating) for PipelineVersion resources when Kubernetes-based pipeline storage is enabled.

The operator integrates with the broader RHOAI/ODH platform by reading the platform's trusted CA bundle, supporting proxy configuration for air-gapped environments, and providing aggregate RBAC ClusterRoles that allow OpenShift admin/edit/view roles to manage DSPA resources. It runs in a dedicated controller namespace (`redhat-ods-applications` for RHOAI, `opendatahub` for ODH) while deploying workloads into user-specified namespaces.

## Architectural Analysis

The Data Science Pipelines Operator follows a single-CRD, multi-component reconciliation pattern. A single `DataSciencePipelinesApplication` CR drives the deployment of up to eight distinct workloads, each with its own Deployment, Service, ServiceAccount, Role, and RoleBinding. This design centralizes configuration (users manage one CR) while maintaining operational separation (each component has least-privilege RBAC). The operator uses the Manifestival library to render Go text/template YAML files at reconciliation time, which provides a clean separation between deployment templates (`config/internal/`) and the controller logic that parameterizes them.

The networking architecture layers multiple security boundaries. Internal pod-to-pod communication can optionally use TLS via OpenShift's service-CA annotation pattern (controlled by `spec.podToPodTLS`). External access is gated through kube-rbac-proxy sidecars that enforce SubjectAccessReview checks against the DSPA CR's `/api` subresource, meaning only users with RBAC permission to access a specific DSPA can reach its API Server or MLMD Envoy endpoint. NetworkPolicies further restrict internal traffic: MariaDB only accepts connections from the API Server, MLMD gRPC server, and the operator itself; the API Server only accepts connections from monitored namespaces, DSP components, KFP v2 pipeline components, and OpenDataHub workbenches.

The ML Metadata subsystem uses a two-tier proxy architecture: an Envoy proxy provides gRPC-web translation (enabling browser-based access from the dashboard), and an optional kube-rbac-proxy sidecar fronts the Envoy proxy for authenticated external access. The Envoy configuration supports CORS with permissive rules and HTTP/2 for gRPC transport. This dual-proxy approach separates protocol translation concerns (Envoy) from authentication concerns (kube-rbac-proxy).

The operator has broad RBAC reach because it manages RBAC for the `pipeline-runner` service account, which must be able to create resources across multiple ML framework CRDs (KServe, Ray, Seldon, Kubeflow Training, CodeFlare). This is an intentional design choice: pipeline authors use the KFP SDK to compose steps that create these resources, so the runner SA needs pre-provisioned permissions. However, it means the DSPO's manager ClusterRole is one of the widest in the platform. The aggregate ClusterRoles (`aggregate-dspa-admin-edit`, `aggregate-dspa-view`) integrate cleanly with OpenShift's built-in RBAC aggregation, automatically extending admin/edit/view cluster roles to cover DSPA resources.

The webhook subsystem is architecturally distinct from per-DSPA resources: it deploys once in the operator's namespace (not the DSPA namespace) and handles PipelineVersion validation/mutation cluster-wide. The operator tracks whether any DSPA uses Kubernetes-based pipeline storage (`pipelineStore: kubernetes`) and only deploys the webhook when at least one such DSPA exists, cleaning it up when the last one is deleted. This avoids unnecessary cluster-scoped webhook overhead for the common database-backed pipeline storage mode.
