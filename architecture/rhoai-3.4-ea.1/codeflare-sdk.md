## Purpose

**Short**: Python SDK for creating, managing, and interacting with Ray clusters on Kubernetes (OpenShift AI / Open Data Hub), with Kueue integration for resource queuing.

**Detailed**: The CodeFlare SDK is a Python client library that provides a high-level API for data scientists and ML engineers to create, configure, and manage Ray clusters running on Kubernetes. It abstracts the complexity of Kubernetes custom resource management (RayCluster and RayJob CRDs from the KubeRay operator) behind a Pythonic interface designed for use in Jupyter notebooks and scripts.

The SDK supports two primary workflows: (1) interactive cluster management via the `Cluster` class, where users define a `ClusterConfiguration`, apply it to create a RayCluster CR, and then submit jobs interactively using the Ray Job Submission Client; and (2) declarative job submission via the `RayJob` class, where users define a `ManagedClusterConfig` and entrypoint, and the SDK creates a RayJob CR that includes an embedded RayCluster spec for lifecycle-managed clusters.

Key capabilities include automatic mTLS certificate generation for secure Ray client connections, Kueue integration for workload queuing and admission control, Gateway API (HTTPRoute) support for dashboard access on RHOAI 3.x+ with fallback to OpenShift Routes, Jupyter notebook widget integration for cluster management UI, and support for GPU/accelerator scheduling across multiple vendors (NVIDIA, Intel, AMD, Habana, Huawei). The SDK also vendors a KubeRay Python client for direct CRD manipulation.

## Architectural Analysis

The CodeFlare SDK follows a **client-side abstraction** pattern: it runs entirely within the user's environment (notebook, script, or CI pipeline) and communicates exclusively with the Kubernetes API server. No server-side component is deployed by the SDK itself — all cluster infrastructure is delegated to the KubeRay operator. This design keeps the SDK lightweight and avoids introducing additional attack surface, but it means the SDK must handle authentication, certificate management, and platform-specific ingress discovery on the client side.

**Dual ingress discovery** is a significant architectural feature. The SDK implements a priority-based fallback for finding Ray dashboard URLs: it first queries for HTTPRoutes (RHOAI 3.x Gateway API pattern, searching cluster-wide then across known platform namespaces), then falls back to OpenShift Routes (RHOAI 2.x pattern), and finally to Kubernetes Ingresses (vanilla K8s/Kind). This tri-level fallback (`_get_dashboard_url_from_httproute` → Routes → Ingresses) ensures backward compatibility across platform versions but adds complexity to the dashboard discovery code path.

**mTLS certificate management** is tightly coupled to the cluster lifecycle. When `Cluster.apply()` is called, the SDK polls for the KubeRay-generated CA secret, then generates client certificates locally using RSA 3072-bit keys with RFC 5280-compliant extensions (BasicConstraints, KeyUsage, ExtendedKeyUsage for both serverAuth and clientAuth, SAN including service DNS names). Certificates are stored in a user-private directory (`~/.local/share/codeflare/tls/`) with 0600 permissions and cleaned up on `Cluster.down()`. The `refresh_certificates()` method handles CA rotation scenarios. This approach avoids requiring cert-manager but means certificate lifecycle is managed by the SDK rather than the platform.

**The RayJob subsystem** provides a distinct code path from interactive cluster management. `RayJob` uses a `ManagedClusterConfig` dataclass (separate from `ClusterConfiguration`) and builds RayJob CRs with embedded RayCluster specs. It handles local file shipping via Kubernetes Secrets with ownerReferences for automatic cleanup, runtime environment processing (zipping working directories, parsing pip requirements), and Kueue integration (queue labels, priority classes). The architectural split between `Cluster` (interactive, long-lived) and `RayJob` (declarative, batch) reflects two distinct usage patterns in ML workflows.

**Kueue integration** spans both code paths. For interactive clusters, the SDK discovers the default LocalQueue and adds `kueue.x-k8s.io/queue-name` labels to RayCluster metadata. For RayJobs, it additionally supports `kueue.x-k8s.io/priority-class` labels and validates queue/priority existence before submission. Notably, the SDK intentionally does NOT set `suspend: true` on RayJobs — it relies on Kueue's mutating webhook to do so, making the queue label harmless metadata if Kueue isn't installed. Autoscaling (`enableInTreeAutoscaling`) is explicitly incompatible with Kueue and raises `ValueError` if both are configured.

**The vendored KubeRay Python client** (`src/codeflare_sdk/vendored/python_client/`) provides low-level CRUD operations for RayCluster and RayJob CRDs. This vendored copy decouples the SDK from upstream KubeRay client releases and allows the SDK team to maintain stability guarantees. The vendored client uses its own kubeconfig loading (independent of the main SDK's `config_check()` flow), which could lead to authentication inconsistencies in edge cases.
