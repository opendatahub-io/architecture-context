## Purpose

**Short**: Kubeflow Pipelines backend providing ML workflow orchestration, scheduling, artifact management, and execution caching for the RHOAI platform.

**Detailed**: Data Science Pipelines (DSP) is the RHOAI distribution of Kubeflow Pipelines v2, providing a complete ML pipeline orchestration platform. It enables users to define, compile, schedule, and execute multi-step machine learning workflows as Argo Workflows on Kubernetes. The system consists of five deployable Go services: an API server exposing both REST and gRPC endpoints for pipeline CRUD and execution management; a persistence agent that synchronizes workflow state from Kubernetes into a relational database; a scheduled workflow controller that manages recurring pipeline executions via cron/periodic triggers; a driver that orchestrates pipeline DAG execution and prepares container execution contexts; and a launcher that executes user containers and publishes results to ML Metadata.

The platform integrates with Argo Workflows for container orchestration, ML Metadata (MLMD) for lineage tracking, and S3-compatible object storage (MinIO) for artifact persistence. It supports both single-user and multi-user modes with Kubernetes RBAC-based authorization, and provides a webhook-based execution caching system to avoid redundant computation. In RHOAI, it is deployed and managed by the DSP Operator (data-science-pipelines-operator), which handles the lifecycle of all sub-components.

| Visualization Server | Python Service | Generates pipeline visualizations from artifacts (not built by this repo's Konflux files) |

## Architectural Analysis

### Two-Phase Pipeline Execution Model

The DSP architecture implements a sophisticated two-phase execution model for pipeline steps within Argo Workflows. Each pipeline task runs as an Argo step containing two sequential containers: the **driver** (phase 1) and the **launcher** (phase 2). The driver resolves inputs from parent DAG executions stored in MLMD, generates cache fingerprints, checks for cached results, and produces a pod spec patch that configures the launcher. If a cache hit is found, the launcher phase is skipped entirely, and cached outputs are reused. This separation of concerns allows the driver to handle all orchestration logic (input resolution, caching, pod configuration) while the launcher focuses purely on container execution and artifact management.

The driver operates in three modes — ROOT_DAG (pipeline initialization), DAG (sub-graph orchestration with iteration support), and CONTAINER (task preparation) — enabling support for complex pipeline topologies including nested DAGs, parallel fan-out via parameter iteration, and conditional task execution. This is architecturally notable because it pushes DAG orchestration logic into the individual workflow steps rather than a central controller, enabling horizontal scalability without a bottleneck.

### Dual Storage Backend Strategy

The API server supports two fundamentally different storage backends for pipeline definitions: a traditional relational database (MySQL/PostgreSQL via GORM) and Kubernetes-native CRD storage (Pipeline and PipelineVersion CRDs via controller-runtime). The CRD storage mode (`--pipelinesStoreKubernetes`) was introduced to support multi-cluster and GitOps workflows where pipeline definitions should be first-class Kubernetes resources rather than database rows. When Kubernetes storage is enabled, the API server also runs admission webhooks (mutating + validating) to enforce pipeline spec validity and set ownership metadata. This dual-backend design adds complexity but provides flexibility for different deployment topologies.

### Caching Architecture Split

Execution caching is implemented via two separate mechanisms depending on the pipeline version. For v1 (Argo-native) pipelines, caching is handled by the **cache server**, a MutatingAdmissionWebhook that intercepts pod creation, generates a cache key from the Argo template, and patches cache metadata as pod annotations. A pod watcher detects successful completions and records outputs in a MySQL cache table for future reuse. For v2 (KFP IR) pipelines, caching is handled entirely within the **driver**, which generates a SHA256 fingerprint from inputs, container image, command, and PVC names, then queries the API server's cache gRPC endpoint. This split exists because v2 pipelines have richer metadata (MLMD execution context) that enables more precise cache key generation than the v1 template-hashing approach. The v1 cache server explicitly skips v2 pods via label detection.

### Security Posture

All containers run as non-root (UID 1000) with `seccompProfile: RuntimeDefault`, dropped capabilities, and privilege escalation disabled. The multi-user mode integrates with Kubernetes RBAC via SubjectAccessReview and TokenReview APIs, mapping pipeline resources to `kubeflow.org` API groups. TLS is optional but supported across all gRPC connections (API server, MLMD, cache service) via custom CA certificate paths. The `InsecureSkipVerify: true` usage in the HTTP-to-gRPC proxy is limited to localhost loopback connections and does not affect inter-service communication. The `pipeline-runner` service account has broad permissions (including `*` on kubeflow.org resources) because it represents the execution context for user pipeline code — this is an intentional design trade-off for pipeline flexibility.

### RHOAI Integration Model

In RHOAI, the DSP components are not deployed directly from these kustomize manifests. Instead, the **DSP Operator** (`data-science-pipelines-operator`) watches DataSciencePipelinesApplication CRDs and deploys all sub-components (API server, persistence agent, scheduled workflow controller, etc.) with RHOAI-specific configuration including image references, database credentials, and object storage settings. The kustomize manifests in this repo serve as the upstream reference implementation. The Konflux Dockerfiles build the RHOAI-specific container images with FIPS compliance (strictfipsruntime build tags, UBI9 base images, OpenSSL).
