## Purpose

**Short**: Manages the lifecycle of model serving infrastructure — including routing, monitoring, authentication, certificate management, and NVIDIA NIM integration — for InferenceServices, LLMInferenceServices, and InferenceGraphs on the RHOAI platform.

**Detailed**: The odh-model-controller is a Kubernetes operator built with controller-runtime that extends KServe's model serving capabilities for Red Hat OpenShift AI. It reconciles several KServe CRDs (InferenceService, ServingRuntime, InferenceGraph, LLMInferenceService) and its own NIM Account CRD to provide platform-specific features that KServe alone does not offer.

The controller manages six primary domains: (1) **Network exposure** — creating OpenShift Routes for InferenceService endpoints with configurable TLS termination; (2) **Monitoring** — generating Prometheus ServiceMonitors, metrics Services, and runtime-specific dashboard ConfigMaps for model serving workloads; (3) **Authentication and authorization** — creating Kuadrant AuthPolicies and Istio EnvoyFilters that integrate with Authorino for LLMInferenceService tier-based access control via Gateway API; (4) **Certificate management** — propagating CA certificate bundles to KServe storage configurations and generating self-signed Ray TLS certificates for multi-node model serving; (5) **Data connection management** — aggregating S3 data connection Secrets into KServe-compatible storage-config Secrets; (6) **NVIDIA NIM integration** — managing NIM Account resources that validate NGC API keys, fetch model catalogs, and create ServingRuntime templates and pull secrets for NVIDIA inference microservices.

The operator also provides admission webhooks that enforce namespace isolation for InferenceServices, validate InferenceGraph service URL locality, enforce singleton constraints for NIM Accounts, validate tier configuration for LLMInferenceServices, and inject Ray TLS init containers into multi-node model serving pods.

## Architectural Analysis

The odh-model-controller demonstrates a **sub-reconciler composition pattern** where the main InferenceService and LLMInferenceService controllers delegate to specialized reconcilers for each resource type they manage. The InferenceServiceReconciler orchestrates sub-reconcilers for Routes, metrics Services, ServiceMonitors, KEDA TriggerAuthentication, ClusterRoleBindings, and optionally Model Registry sync. The LLMInferenceServiceReconciler similarly delegates to AuthPolicy, EnvoyFilter, Role, and RoleBinding reconcilers. Each sub-reconciler implements a common interface (LLMSubResourceReconciler with Reconcile, Delete, Cleanup methods), making the system extensible and testable. This pattern avoids monolithic reconciliation loops and isolates failure domains.

The NIM Account controller uses a distinct **handler chain pattern** — sequential handlers (ValidationHandler → ConfigMapHandler → TemplateHandler → PullSecretHandler) that each return a structured HandleResponse indicating whether to continue, requeue, or error. This pattern supports both air-gapped and connected modes: when the NIM account lacks external API connectivity, handlers short-circuit without error. The controller also implements time-based refresh gates (configurable validation and config refresh rates with 24h defaults), which prevent unnecessary external API calls while ensuring credential validity.

A notable design decision is the **conditional CRD registration pattern**: the controller checks CRD availability at startup for Kuadrant AuthPolicy, Authorino, KEDA TriggerAuthentication, EnvoyFilter, and Gateway API resources. If a CRD is not installed on the cluster, the corresponding watch is silently skipped. This makes the controller resilient to partial platform installations — it can operate without Kuadrant, KEDA, or Istio, managing only the resources whose CRDs are available.

The authentication architecture for LLMInferenceService follows a **multi-layer Gateway API pattern**: the controller creates a Kuadrant AuthPolicy targeting the platform Gateway for user-defined authentication (Kubernetes TokenReview + SubjectAccessReview), and separately creates an Istio EnvoyFilter that configures the Envoy proxy's upstream TLS connection to the Authorino authorization service on port 50051. The EnvoyFilter is a temporary workaround (tracked in Kuadrant issue #1531) that will be removed once Kuadrant natively supports TLS configuration for its authorization backend. The tier-based access control system uses a ConfigMap-driven mapping from tier names to Kubernetes groups (e.g., `system:serviceaccounts:{tenant}-tier-{tierName}`), enabling multi-tenant LLM serving with RBAC-enforced access levels.

The controller's certificate management spans two independent subsystems: (1) **CA bundle propagation** — aggregating certificates from `odh-trusted-ca-bundle` and `openshift-service-ca.crt` ConfigMaps into `odh-kserve-custom-ca-bundle` for KServe storage access, and (2) **Ray TLS** — generating self-signed CA and per-pod server certificates for multi-node model serving with Ray. The Pod webhook injects init containers that copy certificates from Secrets to emptyDir volumes, enabling Ray workers to establish mTLS connections. Certificate rotation uses a 2-year pre-expiration threshold on 5-year CA certificates and 3-year server certificates.
