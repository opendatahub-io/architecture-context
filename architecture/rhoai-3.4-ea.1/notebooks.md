## Purpose

**Short**: Builds and ships container images for interactive data science workbenches (JupyterLab, code-server, RStudio) and pipeline runtime environments used in RHOAI.

**Detailed**: The notebooks repository produces approximately 22 container images that serve as the interactive development environments and pipeline execution runtimes for Red Hat OpenShift AI. These images are not deployed as standalone services — they are consumed by the ODH Notebook Controller, which creates StatefulSet pods from them when users launch workbenches through the RHOAI Dashboard.

The repository is organized into four major image families: **Jupyter workbenches** (minimal, data science, PyTorch, TensorFlow, TrustyAI, PyTorch+LLMCompressor variants across CPU/CUDA/ROCm accelerators), **code-server workbenches** (VS Code in the browser), **RStudio workbenches** (RStudio Server), and **pipeline runtimes** (minimal environments for Elyra pipeline node execution). All images are built on UBI9/RHEL9 base images with Python 3.12, and many support multi-architecture builds (x86_64, aarch64, ppc64le, s390x).

The deployment manifests define OpenShift `ImageStream` resources that are deployed by the platform operator (`rhods-operator`/`opendatahub-operator`) into the cluster. These ImageStreams reference container images from `registry.redhat.io` (for RHOAI) and provide versioned tags (N through N-7) so users can select specific image versions when launching workbenches from the Dashboard.

## Architectural Analysis

The notebooks repository is fundamentally a container image build system, not a service or operator. Its architectural complexity lies in the multi-layered Dockerfile inheritance chain and the parameterized kustomize manifests that register images with the platform.

**Image Inheritance Hierarchy**: The Jupyter workbenches follow a strict layering: `UBI9 base → cpu-base/cuda-base/rocm-base → jupyter-minimal → jupyter-datascience → jupyter-pytorch/tensorflow/trustyai/llmcompressor`. Each layer adds specific dependencies while inheriting the complete environment from its parent. This design minimizes image size by sharing layers and ensures consistent base configurations across all variants. The CUDA and ROCm variants diverge at the base layer, pulling in GPU-specific base images that include the respective toolkits.

**Hermetic Build Transition**: The codeserver workbench is the flagship hermetic build implementation, with a sophisticated 4-stage Dockerfile (`rpm-base → whl-cache → cpu-base → codeserver`) that installs all dependencies from prefetched cachi2/hermeto caches. This includes npm packages for the code-server build, Python wheels (with special handling for ppc64le/s390x architectures that need source compilation), and RPMs. The remaining Jupyter, RStudio, and runtime images are still in transition — they download from PyPI and system repositories during build. The `prefetch-input/` directory and `rpms.lock.yaml` files are expected to appear on release branches as hermeticity is expanded across all images. This is tracked by RHAIENG-2846 and RHAIENG-2860.

**Multi-Architecture Support**: The repository supports x86_64, aarch64, ppc64le, and s390x for CPU images. The hermetic codeserver build demonstrates sophisticated arch-aware logic: on ppc64le/s390x, pre-built RHOAI wheels are used for native extensions (numpy, scipy), and OpenBLAS is installed via RPM for runtime linking. CUDA images support x86_64 and aarch64 only, while ROCm images are x86_64-only — reflecting vendor GPU support limitations.

**Kustomize Manifest Design**: The manifests use a notable pattern where image references in ImageStream YAMLs are placeholders (`_PLACEHOLDER` suffix) that are substituted via kustomize `replacements` from ConfigMaps generated by `configMapGenerator` with `params.env` values. This decouples the image reference lifecycle from the manifest structure — new image builds only require updating `params.env` entries without touching the ImageStream YAML. The manifests maintain N through N-7 version tags (spanning 3.4 down to 1.2), providing long-tail support for users who need older image versions.

**Notebook Culling Integration**: The code-server and RStudio images initialize JSON-formatted access log files (`/var/log/nginx/codeserver.access.log`, `/var/log/nginx/rstudio.access.log`) at startup with a kernel activity record. This is consumed by the odh-notebook-controller's notebook culling logic, which monitors these logs to determine when a workbench is idle and can be scaled down. The nginx proxy is essential for this — it sits between the user and the IDE, logging activity in the format the controller expects.
