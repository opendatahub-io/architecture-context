## Purpose
**Short**: Kubernetes operator for managing distributed machine learning training jobs across multiple frameworks (PyTorch, TensorFlow, MPI, XGBoost, PaddlePaddle, JAX).

**Detailed**: The Kubeflow Training Operator (KFTO) is a Kubernetes-native project for fine-tuning and scalable distributed training of AI/ML models. It provides Custom Resource Definitions (CRDs) for six different ML frameworks: PyTorchJob, TFJob, MPIJob, XGBoostJob, PaddleJob, and JAXJob. Each CRD allows users to define distributed training topologies with multiple replica types (master, worker, PS, etc.).

The operator manages the full lifecycle of training jobs including pod creation/deletion, service creation for inter-pod communication, status tracking, gang scheduling integration (via Volcano or scheduler-plugins), elastic training support (for PyTorch), and automatic NetworkPolicy creation for job isolation. It uses a validating webhook for CRD validation and self-managed certificate rotation via the OPA cert-controller. In the RHOAI distribution, it is deployed to the `opendatahub` namespace with the `kubeflow-` name prefix and includes Prometheus metrics monitoring via PodMonitor.

### Dynamically Created Resources

The operator dynamically creates the following resources per training job:

| Resource Type | Created By | Purpose |
|---------------|-----------|---------|
| Pods | All job controllers | Worker/master/PS/launcher pods for distributed training |
| Services (headless) | All job controllers | Inter-pod communication within training jobs |
| NetworkPolicy | PyTorchJob controller | Job-level network isolation allowing intra-job communication and OpenShift monitoring access on port 8080/TCP |
| HorizontalPodAutoscaler | PyTorchJob controller | Elastic scaling for PyTorch elastic training |
| PodGroup (Volcano) | Gang scheduler integration | Gang scheduling coordination (optional) |
| PodGroup (scheduler-plugins) | Gang scheduler integration | Gang scheduling coordination (optional) |

### Ingress

No external ingress is configured. The operator runs as an internal cluster service with no external route, HTTPRoute, or Gateway API resources. The only inbound traffic is:
- Kubernetes API server to webhook endpoint (443/TCP -> 9443/TCP)
- Prometheus scraping metrics from pods (8080/TCP)

### Egress

| Destination | Port | Protocol | Encryption | Auth | Purpose |
|-------------|------|----------|------------|------|---------|
| Kubernetes API server | 443/TCP | HTTPS | TLS 1.2+ | ServiceAccount token (Bearer) | CRUD operations on Pods, Services, CRDs, NetworkPolicies, etc. |

### Network Policies (Controller-Created)

The PyTorchJob controller dynamically creates a NetworkPolicy per job that:
1. Allows **intra-job pod communication**: Pods with matching `training.kubeflow.org/job-name` label can communicate freely
2. Allows **OpenShift monitoring access**: Pods from `openshift-monitoring` namespace can access port 8080/TCP
