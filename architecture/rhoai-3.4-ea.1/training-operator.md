## Purpose

**Short**: Kubernetes-native operator for orchestrating distributed training jobs across multiple ML/AI frameworks including PyTorch, TensorFlow, JAX, MPI, XGBoost, and PaddlePaddle.

**Detailed**: The Kubeflow Training Operator (KFTO) is a Kubernetes operator built on controller-runtime that manages the complete lifecycle of distributed machine learning training jobs. It provides six Custom Resource Definitions (CRDs) — PyTorchJob, TFJob, JAXJob, MPIJob, XGBoostJob, and PaddleJob — each tailored to the communication patterns and topology requirements of their respective ML frameworks.

The operator creates and manages pods, headless services for inter-pod communication, and optional gang scheduling resources (via Volcano or scheduler-plugins) to coordinate distributed training. Each framework controller handles framework-specific concerns: PyTorch manages elastic training with rendezvous and HPA integration, TensorFlow manages parameter server and multi-worker topologies, MPI creates launcher/worker pod architectures with dedicated RBAC, and JAX/XGBoost/PaddlePaddle manage their respective distributed paradigms.

In the RHOAI distribution, the operator is deployed into the `opendatahub` namespace with the `kubeflow-` name prefix, uses FIPS-compliant builds (CGO_ENABLED=1, GOEXPERIMENT=strictfipsruntime), and integrates with OpenShift monitoring via PodMonitor. The PyTorch controller also creates NetworkPolicies to allow monitoring traffic from the `openshift-monitoring` namespace. The operator exposes validating webhooks (with auto-rotated TLS certificates via OPA cert-controller) for all job types except MPI.

## Architectural Analysis

The Training Operator follows a well-structured pattern of a shared controller framework (`pkg/controller.v1/common/`) with framework-specific controllers inheriting common behavior. The `JobController` base provides pod reconciliation, service reconciliation, expectation tracking (to prevent premature re-reconciliation), and gang scheduling integration. Each framework controller (PyTorch, TensorFlow, JAX, MPI, XGBoost, PaddlePaddle) implements the `ControllerInterface` to customize environment variable injection, pod topology, and framework-specific lifecycle management.

The MPI controller stands out architecturally as the most complex: unlike other frameworks that use headless services for inter-pod communication, MPI creates a dedicated launcher pod with its own ServiceAccount, Role, and RoleBinding to enable `kubectl exec` into worker pods. This is because MPI's communication model (SSH/RSH-based process spawning) fundamentally differs from the parameter-server or collective-communication models used by other frameworks. The MPI controller creates a ConfigMap containing a hostfile and kubexec.sh script that replaces SSH with `kubectl exec` for launching MPI processes on worker pods.

PyTorch is the only framework supporting elastic training with HPA integration. When `ElasticPolicy` is configured, the controller creates a `HorizontalPodAutoscaler` targeting worker pods and configures PyTorch's elastic rendezvous (c10d, etcd, or etcd-v2 backends). The PyTorch controller also uniquely creates `NetworkPolicy` resources to allow monitoring traffic from the `openshift-monitoring` namespace to job pods — a RHOAI-specific enhancement not found in upstream Kubeflow. An init container (`init-pytorch`) is injected into worker pods to wait for the master pod's DNS to resolve before starting training, ensuring proper distributed initialization order.

The webhook architecture uses OPA's cert-controller library for automated TLS certificate management, which generates a self-signed CA, creates TLS certificates stored in a Secret, and patches the `ValidatingWebhookConfiguration` with the CA bundle. This eliminates the need for cert-manager or manual certificate provisioning. All webhooks are validating-only (no mutation) with `failurePolicy: Fail`, meaning job creation/update is blocked if the webhook is unavailable — a deliberate choice for data integrity over availability.

The RHOAI overlay makes several significant modifications: it prefixes all resource names with `kubeflow-`, deploys into the `opendatahub` namespace, adds a `PodMonitor` for Prometheus scraping, removes the monitoring port from the Service (relying on direct pod scraping instead), and reuses the operator image as the PyTorch init container image (avoiding a separate image dependency). The overlay also adds `training-edit` and `training-view` ClusterRoles that aggregate into the standard Kubernetes `edit` and `admin` roles, enabling RBAC inheritance for OpenShift users.
