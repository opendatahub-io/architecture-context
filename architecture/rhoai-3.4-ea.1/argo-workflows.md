## Purpose

**Short**: Argo Workflows is a Kubernetes-native workflow orchestration engine that executes multi-step containerized workflows as DAGs or sequential steps, used as the pipeline execution backend for Data Science Pipelines in RHOAI.

**Detailed**: Argo Workflows provides workflow orchestration for Kubernetes by defining workflows as Custom Resources (CRDs). The system has three main runtime components: the **workflow controller** watches Workflow CRs and dynamically creates Pods to execute workflow steps; the **argoexec executor** runs as an init/sidecar container within workflow pods to manage artifact I/O, container execution, and result reporting; and the **argo server** (not shipped in RHOAI — included in upstream only) provides a gRPC+REST API gateway with a web UI.

In the RHOAI context, this repository produces two container images — the workflow controller and the argoexec executor — that are consumed by the Data Science Pipelines operator (dspo). The DSP operator deploys both components when a user creates a `DataSciencePipelinesApplication` CR. The argo server and CLI are not shipped; pipeline orchestration is accessed through the DSP API server instead. The workflow controller manages the lifecycle of Workflow CRs by creating Pods, ConfigMaps, PVCs, and PodDisruptionBudgets to execute pipeline steps, with optional database integration for workflow archiving and node status offloading.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| workflow-controller | Go Controller | Watches Workflow CRDs and orchestrates pod-based execution of workflow steps, manages CronWorkflows, TTL cleanup, artifact GC, and optional DB archiving |

## Architectural Analysis

The Argo Workflows codebase is a mature upstream project (argoproj/argo-workflows) that Red Hat forks and adapts for RHOAI. The RHOAI fork's primary modifications are limited to build infrastructure: Konflux Dockerfiles with FIPS-enabled Go builds, UBI9 base images, RPM lock files for hermetic builds, and Tekton pipeline definitions for CI/CD. The upstream Go source code is used largely unmodified, which means the controller and executor inherit all upstream capabilities (database archiving, multiple artifact backends, CronWorkflows) even though RHOAI uses only a subset.

The controller architecture uses a work-queue pattern with configurable worker pools rather than controller-runtime's reconciler pattern. The main controller runs 32 workflow workers, 8 archive workers, 4 pod cleanup workers, 4 TTL workers, and 8 cron workers by default. This high degree of concurrency is necessary because each workflow can have hundreds of nodes, and each node transition requires a reconciliation pass. The controller also maintains indexed caches for workflows by phase, by semaphore reference, by template reference, and by UID — enabling efficient lookups without full list operations.

The executor (argoexec) uses an "emissary" execution model where it directly executes the user's container command via `exec.Command()`, managing signal forwarding and exit code capture. This is distinct from the older "docker" and "pbs" executors. The emissary pattern avoids requiring Docker socket access or privileged containers, which is important for OpenShift's security constraints. The argoexec Konflux Dockerfile produces both a FIPS-enabled binary (`argoexec-fips`) and a standard binary (`argoexec`), shipping both in the container image. The selection of which binary to use at runtime is controlled externally.

A noteworthy security observation is the `InsecureSkipVerify=true` setting in the argo server's internal gRPC loopback connection (`server/apiserver/argoserver.go:351`). This is a localhost-only connection where the server's gRPC gateway connects back to its own gRPC endpoint, so the practical risk is low. However, the `InsecureSkipVerify` flag in the HTTP template executor (`workflow/executor/agent.go:301-310`) is user-controllable via the Workflow CRD spec, which could allow workflows to bypass TLS verification when calling external HTTP endpoints — this is by design but worth noting in security reviews.

The kustomize manifests provide two deployment modes: cluster-install (with ClusterRoles for cross-namespace workflow management) and namespace-install (with namespace-scoped Roles and `--namespaced` flag). In RHOAI, the DSP operator manages deployment directly and does not use these upstream kustomize manifests; instead, it templates the Deployment and RBAC resources based on the DSPA CR configuration.
