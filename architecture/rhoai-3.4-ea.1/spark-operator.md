## Purpose

**Short**: Kubernetes operator that automates the deployment, scheduling, and lifecycle management of Apache Spark applications on Kubernetes clusters.

**Detailed**: The Spark Operator (Kubeflow Spark Operator v2) is a controller-runtime based Kubernetes operator that manages three custom resources: `SparkApplication`, `ScheduledSparkApplication`, and `SparkConnect`. It automates the full lifecycle of Spark workloads including submission via spark-submit, driver and executor pod management, monitoring, web UI exposure, and automatic retry/restart handling.

The operator runs as two separate deployments: a controller that watches CRDs and manages Spark workloads, and a webhook server that validates and mutates SparkApplication resources and Spark driver/executor pods. It supports pluggable batch schedulers (Volcano, Yunikorn, kube-scheduler plugins), Prometheus metrics integration, dynamic executor allocation, and resource quota enforcement. The operator integrates with cert-manager for TLS certificate management or can generate self-signed certificates internally.

For RHOAI deployment, the operator is deployed in the `redhat-ods-applications` namespace via kustomize overlays managed by the platform operator. The same container image serves both the controller and webhook roles, differentiated by the subcommand (`controller start` vs `webhook start`).

## Architectural Analysis

The Spark Operator follows a clean separation of concerns between its controller and webhook deployments. Both run from the same container image (`spark-operator`) but are invoked with different subcommands (`controller start` vs `webhook start`). This allows independent scaling and failure isolation — the webhook must remain available for pod admission while the controller handles reconciliation at its own pace. The two deployments share no state beyond the Kubernetes API and use separate leader election locks (`spark-operator-controller-lock` and `spark-operator-webhook-lock`).

The SparkApplication controller implements a comprehensive state machine with 14 states (New → Submitted → Running → Completed/Failed, plus Suspending, Suspended, Resuming, PendingRerun, Invalidating, etc.). This state machine is the core of the operator's value — it handles retry logic (with configurable backoff intervals), spec invalidation during execution (triggering resubmission), suspend/resume workflows, and TTL-based cleanup. The controller creates Spark submissions by constructing spark-submit CLI arguments from the CRD spec, then creates driver pods directly. Executor pods are created by the Spark driver itself, not by the operator — the operator only monitors their state via label-based watches (`sparkoperator.k8s.io/launched-by-spark-operator=true`).

The webhook architecture is notable for its pod mutation webhook that intercepts Spark driver and executor pod creation. This webhook applies 26 sequential mutations to pods — injecting environment variables, volumes, sidecar containers, security contexts, scheduling constraints, GPU resources, and Prometheus monitoring configuration. This design delegates pod customization to admission time rather than embedding it in the Spark submission, allowing the operator to enrich pods with Kubernetes-native features that Spark's own configuration system doesn't support natively.

The batch scheduler integration is designed as a pluggable registry pattern supporting three scheduler backends. Volcano and kube-scheduler plugins create PodGroup CRDs for gang scheduling, while Yunikorn uses pod annotations to avoid CRD dependencies. This flexibility allows RHOAI deployments to choose the scheduling backend that best fits their cluster configuration. The scheduler integration is particularly important for large Spark jobs where driver and executor pods must be co-scheduled to avoid resource deadlocks.

The FIPS compliance posture is a concern: the operator is built with `CGO_ENABLED=0` (static linking with pure-Go crypto), which would fail OpenShift's check-payload validation for FIPS-enabled clusters. This may need to be addressed in downstream builds by switching to `CGO_ENABLED=1` with `GOEXPERIMENT=strictfipsruntime` to use OpenSSL-backed crypto. The use of `crypto/md5` for submission ID hashing is non-cryptographic and not a security concern, but the static linking is a real FIPS gap.
