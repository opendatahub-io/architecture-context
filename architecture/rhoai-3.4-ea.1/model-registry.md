## Purpose

**Short**: A central repository for ML developers to store, manage, and retrieve metadata about models, model versions, artifacts, experiments, and inference services.

**Detailed**: Model Registry provides a REST API for recording and retrieving metadata associated with ML workflows. It stores information about registered models, model versions, model artifacts, serving environments, inference services, experiments, and experiment runs. The service uses an embedded metadata (EmbedMD) architecture backed by either MySQL or PostgreSQL databases, replacing the original C++ gRPC ML Metadata (MLMD) service with a native Go implementation that adds higher-level API abstractions, RBAC support, and OpenAPI-defined REST interfaces.

The component ships as multiple deployable artifacts: (1) the core model-registry REST proxy server that exposes the OpenAPI v1alpha3 API, (2) a Kubernetes controller that reconciles KServe InferenceService custom resources with Model Registry entities, (3) a CSI storage initializer plugin that resolves `model-registry://` URIs to download model artifacts from the registry via KServe's storage initialization framework, (4) a model catalog server that aggregates models from multiple sources (Hugging Face, YAML, database), and (5) an async-upload job (Python) that copies models between storage backends (S3, OCI, HTTP/HuggingFace) and registers them in the registry. The platform operator deploys the model registry as a per-namespace service, optionally with Istio sidecar injection for mTLS.

## Architectural Analysis

The model-registry repository is a multi-component system that has evolved from a thin proxy over Google's ML Metadata (MLMD) C++ gRPC service into a standalone Go service with an embedded metadata engine (EmbedMD). This transition is architecturally significant: the EmbedMD approach eliminates the external MLMD dependency by embedding the metadata schema and migration logic directly into the Go binary, using GORM as the ORM layer over MySQL or PostgreSQL. The repository pattern (`internal/datastore/`, `internal/db/`) provides a clean abstraction with type-safe generics (`getRepo[T any]`) and a plugin-based connector system, making database backends swappable.

The REST API layer is generated from OpenAPI specifications (`api/openapi/model-registry.yaml` at v1alpha3) using `openapi-generator`, producing both the router definitions and data models. This is then wrapped with hand-written business logic in the converter, reconciler, and middleware layers. The converter pattern (`internal/converter/`) handles bidirectional translation between OpenAPI models and domain models, while the reconciler manages merge semantics for PATCH operations. A notable security measure is the null-byte validation middleware (`internal/server/middleware/validation.go`) that screens all query parameters and request bodies before they reach the handler layer, preventing database errors and potential injection attacks.

The controller component (`cmd/controller/`) is a standard controller-runtime operator that watches KServe InferenceService CRs. It performs service discovery to find model registry instances -- either by explicit annotation, named label lookup, or by searching for Services with `component=model-registry` labels. The controller syncs ISVC state (URL, desired state) with Model Registry entities and uses finalizers for clean deletion handling. It communicates with the registry over plain HTTP using bearer token authentication, which works within the mesh but means the controller-to-registry communication is not encrypted without Istio sidecar injection.

The CSI storage initializer (`cmd/csi/`) is an elegant integration with KServe's model download pipeline. It registers as a `ClusterStorageContainer` to handle `model-registry://` URIs, resolving model names and versions through the registry's API, then delegating actual artifact downloads to KServe's native storage providers (S3, GCS, etc.). This provides a layer of indirection: users specify model names instead of storage URIs, and the registry resolves them to the actual artifact locations. The async-upload job (`jobs/async-upload/`) completes the upload side, providing S3-to-OCI and HTTP/HuggingFace-to-OCI transfer capabilities with automatic model registration.

The catalog subsystem is particularly sophisticated. It implements leader election via PostgreSQL advisory locks (`pglock`), supporting a standby/leader model where all pods serve read traffic but only the leader writes to the database and syncs with external sources. The Hugging Face integration discovers models by pattern matching with configurable organization filters and sync intervals. The RHOAI overlay adds performance benchmark data and Red Hat registry credentials, indicating that the catalog serves as a curated model marketplace in the RHOAI distribution.
