## Purpose

**Short**: An out-of-tree Llama Stack evaluation provider that enables automated LLM red-teaming and vulnerability scanning using NVIDIA's Garak framework.

**Detailed**: This component implements a Llama Stack provider for the `Eval` API that integrates NVIDIA's Garak vulnerability scanner to perform automated red-teaming of Large Language Models. It enables security assessment of LLMs for prompt injection, jailbreaks, toxicity, bias, and other vulnerabilities, supporting compliance frameworks such as OWASP LLM Top 10, AVID taxonomy, and CWE.

The provider operates in two distinct modes: an **inline mode** where Garak scans run locally as subprocesses within the provider process, and a **remote mode** where scans are dispatched as Kubeflow Pipelines (KFP) jobs on OpenShift AI / Data Science Pipelines. In remote mode, the provider constructs KFP pipeline definitions with three stages (validation, scan, result parsing), submits them to the KFP API, and monitors their progress. Both modes produce structured results compatible with the Llama Stack `EvaluateResponse` schema, including per-probe vulnerability scores, AVID taxonomy mappings, and HTML reports.

The provider also supports **shield testing** — the ability to measure guardrail (shield) effectiveness by orchestrating prompts through input/output shields before and after LLM inference. This uses a function-based Garak generator that wraps the Llama Stack Safety API to run shield checks inline with red-teaming probes, enabling comparative analysis of protected vs. unprotected model behavior.

## Architectural Analysis

This provider follows a clean dual-execution-mode architecture where the same scan configuration model (`GarakCommandConfig`) drives both inline and remote execution paths. The `GarakEvalBase` class provides shared logic for benchmark registration, command construction, and result parsing, while `GarakInlineEvalAdapter` and `GarakRemoteEvalAdapter` specialize execution and lifecycle management. This design avoids code duplication while allowing each mode to handle its unique concerns (subprocess management vs. KFP API interaction).

The **shield scanning architecture** is particularly noteworthy. Rather than extending Garak itself, the provider implements a custom Garak "generator" (`function.Single`) that wraps the `SimpleShieldOrchestrator`. This orchestrator intercepts each probe prompt, runs it through configured input shields via the Llama Stack Safety API, forwards non-violating prompts to the LLM, and then runs the LLM's response through output shields. This approach enables measuring shield effectiveness using Garak's existing probe/detector framework without modifying Garak's codebase — a clean integration pattern that keeps the provider loosely coupled.

The **remote execution pipeline** uses a three-stage KFP design (validate → scan → parse) with conditional branching for GPU/CPU execution and retry logic with exponential backoff. The pipeline communicates results back to the provider through the Llama Stack Files API — the KFP pod uploads scan artifacts with predictable filenames (`{job_id}_mapping.json`), and the provider retrieves them by listing files and matching on filename. This is an eventually-consistent pattern that requires the provider to poll both KFP run status and file availability.

The **Cargo.toml and Cargo.lock at the repository root are not for a real Rust project** — they exist solely as a workaround for a Hermeto (cachi2) limitation where Rust dependencies of Python packages (via pyo3/maturin) were not being prefetched unless a top-level Cargo project exists. The `.konflux/main.rs` is a dummy `fn main() {}` file. This is documented in `.konflux/README.md` and is an important architectural detail for understanding the build system, as it could be mistaken for a legitimate Rust component.

The provider demonstrates good **version compatibility hygiene** through the `compat.py` module, which provides a try/except import pattern supporting both the newer `llama_stack_api` package layout and the legacy `llama_stack` layout. This enables the provider to work with multiple Llama Stack versions simultaneously. The `version_utils.py` module similarly extracts the garak version from package metadata with multiple fallback strategies.
