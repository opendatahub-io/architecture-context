## Purpose

**Short**: Intelligent inference request scheduler that routes LLM inference requests to optimal model-serving pods via Kubernetes Gateway API, with support for Prefill/Decode disaggregation.

**Detailed**: The llm-d-inference-scheduler provides an Endpoint Picker (EPP) component for the llm-d inference framework. It extends the upstream Gateway API Inference Extension (GIE) project to make optimized routing decisions for inference requests arriving through a Kubernetes Gateway. The EPP receives routing callbacks from Envoy via the ext-proc (External Processing) filter and selects the best backend pod based on a configurable pipeline of filter and scorer plugins.

The scheduler supports multiple scheduling strategies including KV-cache locality scoring, load-aware balancing, session affinity, and active request tracking. A key differentiator is its support for disaggregated Prefill/Decode (P/D) execution, where prefill (prompt processing) and decode (token generation) stages can be routed to different specialized pods. The P/D decision is informed by real-time KV-cache state tracking via integration with the llm-d-kv-cache-manager.

The repository produces two container images: the EPP server itself (primary scheduling service) and a PD routing sidecar that runs alongside vLLM model-serving pods to orchestrate the two-stage P/D protocol. The sidecar acts as a reverse proxy, intercepting inference requests and coordinating prefill-then-decode flows using pluggable connector protocols (NIXL v2, LMCache, SGLang).

## Architectural Analysis

The llm-d-inference-scheduler demonstrates a well-designed plugin architecture that mirrors the upstream Gateway API Inference Extension (GIE) framework while adding llm-d-specific capabilities. The plugin system uses a factory pattern with 12 registered plugin types across four categories: filters (4), scorers (5), profile handlers (2), and pre-request handlers (1). Each plugin type implements framework interfaces from the GIE project, enabling a clean separation between the upstream scheduling framework and llm-d-specific scheduling logic. The `RegisterAllPlugins()` function in `pkg/plugins/register.go` serves as the single registration point, making the plugin inventory explicit and auditable.

The Prefill/Decode disaggregation architecture is the most complex aspect of this component. The EPP runs scheduling in two phases: first the decode profile selects a decode pod, then (if the P/D threshold is met) the prefill profile selects a prefill pod. The `PdProfileHandler` makes an intelligent decision based on KV-cache hit percentage - if enough of the prompt is already cached on the decode pod, it skips the prefill stage entirely. This threshold-based optimization avoids unnecessary network hops and reduces latency for cache-warm requests. The decision metrics are tracked via the `llm_d_inference_scheduler_pd_decision_total` Prometheus counter.

The PD sidecar proxy supports three connector protocols (NIXL v2, LMCache, SGLang), each implementing a different coordination strategy between prefill and decode stages. NIXL v2 is the primary protocol, using a two-stage sequential flow with KV transfer parameters passed from prefill response to decode request. SGLang uses a concurrent approach with bootstrap metadata injection for room-based coordination. LMCache is deprecated and relies on external shared storage. The sidecar includes SSRF protection that dynamically watches InferencePool and Pod resources to build an allowlist of valid prefiller targets, preventing unauthorized request forwarding.

A notable design decision is the use of embedded Python 3.12 in the EPP binary for Jinja template rendering of chat completions. This is required by the llm-d-kv-cache-manager dependency for accurate tokenization of chat messages. The multi-language build (Go + Rust + Python) significantly increases build complexity, as evidenced by the Konflux Dockerfile which installs Rust/Cargo from Red Hat repos, builds libtokenizers from source, and embeds Python with CGO. The sidecar image, by contrast, is pure Go with CGO_ENABLED=1 and has a much simpler build. The data-parallel mode in the sidecar spawns additional HTTP servers on incremented ports (one per rank), which is a pragmatic workaround for vLLM's data-parallel architecture but could create port management complexity at scale.
