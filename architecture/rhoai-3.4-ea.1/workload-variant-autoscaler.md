## Purpose

**Short**: A Kubernetes controller that performs intelligent, saturation-based autoscaling for LLM inference model servers, determining optimal replica counts based on KV cache utilization, queue depth, and GPU availability.

**Detailed**: The Workload Variant Autoscaler (WVA) is a global autoscaler designed for inference model servers serving Large Language Models (LLMs) on Kubernetes and OpenShift. It works alongside standard Kubernetes HPA and external autoscalers like KEDA by emitting Prometheus metrics (`wva_desired_replicas`) that these autoscalers consume to perform the actual scaling.

WVA introduces the concept of "variants" — different deployments of the same model that may use different accelerator types (e.g., H100, A100, L40S) with different costs. The controller continuously monitors vLLM server metrics via Prometheus (KV cache utilization, request queue depth, token throughput), analyzes saturation across all replicas of a model, and makes cost-aware scaling decisions. When scaling up, it selects the cheapest variant; when scaling down, it removes the most expensive variant's replicas.

The system also integrates with the Gateway API Inference Extension (InferencePool) for scale-from-zero capabilities — detecting pending requests in the scheduler flow control queue and immediately scaling up inactive variants. GPU-aware limiting constrains scaling decisions to available cluster GPU capacity using a greedy-by-saturation allocation algorithm. Configuration is namespace-aware, supporting per-model and per-namespace overrides for saturation thresholds and scale-to-zero policies via ConfigMaps.

## Architectural Analysis

The WVA implements a **decision cache pattern** that cleanly separates optimization engines from the Kubernetes reconciliation loop. The saturation engine and scale-from-zero engine run as independent polling loops (30s and 100ms respectively), writing their scaling decisions to a shared in-memory cache. When a decision is ready, the engine sends a generic event through a Go channel (`common.DecisionTrigger`) that triggers the VariantAutoscaling reconciler, which reads the cached decision and patches the VA status. This design prevents the optimization engines from needing direct Kubernetes API write access for status updates, reduces API server load by batching status changes, and allows the reconciler to merge decisions from multiple engines cleanly.

The **cost-aware variant selection** algorithm is a distinctive feature. When scaling up, WVA selects the cheapest variant (by `spec.variantCost`) to add replicas to; when scaling down, it removes from the most expensive variant. This drives cost-optimal resource allocation across heterogeneous GPU fleets (e.g., preferring L40S over H100 when possible). The algorithm uses alphabetical variant name ordering as a deterministic tie-breaker, ensuring consistent decisions across controller restarts. Notably, WVA only adjusts one replica at a time per optimization cycle (scale-up adds 1 to cheapest, scale-down removes 1 from most expensive), creating a gradual, stabilized scaling behavior that pairs well with HPA's stabilization windows.

The **transition state protection** mechanism deserves attention: before making any scaling decision, the saturation analyzer checks whether ANY variant for a model has a mismatch between desired and current replicas. If any variant is in transition (scaling in progress), ALL scaling decisions for that model are blocked. This prevents cascading, oscillating decisions based on incomplete data — a common problem with inference workload autoscalers where startup time is measured in minutes.

The **namespace-aware configuration** system supports multi-tenancy. Saturation thresholds and scale-to-zero policies can be overridden per-namespace via ConfigMaps with well-known names. The resolution order is namespace-local ConfigMap > global ConfigMap > defaults. The ConfigMap reconciler uses namespace tracking (maintained by the VA reconciler) to efficiently filter events — only ConfigMaps in namespaces with active VariantAutoscaling resources trigger reconciliation, avoiding cluster-wide ConfigMap watching that would cause cache sync timeouts.

A notable security consideration is the FIPS compliance gap: the Dockerfile.konflux explicitly comments out `GOEXPERIMENT=strictfipsruntime` and uses `CGO_ENABLED=0` (static linking), noting that "strictfipsruntime does not work with disabling CGO, which is a best practice in this case." This means the binary will **not** pass check-payload FIPS validation. The TLS configuration is otherwise sound (TLS 1.2 minimum, configurable CA/client certs), but the static linking choice means RHOAI builds of this component may need to address FIPS compliance before GA.
