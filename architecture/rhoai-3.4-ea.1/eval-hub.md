## Purpose

**Short**: A lightweight REST API service for orchestrating LLM evaluations across multiple backends, tracking experiments via MLflow, and running natively on OpenShift.

**Detailed**: EvalHub is a Go-based REST API service that receives evaluation requests from clients and dispatches benchmark jobs to pluggable evaluation frameworks — including lm-evaluation-harness (167 benchmarks), RAGAS, Garak, GuideLLM, LightEval, and MTEB. Each evaluation job spawns Kubernetes batch Jobs that run adapter containers for the selected provider/benchmark. The service tracks job state in a relational database (SQLite for development, PostgreSQL for production) and optionally integrates with MLflow for experiment tracking with workspace-scoped token authentication.

On RHOAI clusters, EvalHub is deployed by the TrustyAI Service Operator via an `EvalHub` custom resource (apiVersion `trustyai.opendatahub.io/v1alpha1`). The operator provisions the service, its service account, and related ConfigMaps. EvalHub exposes a versioned REST API under `/api/v1` for creating, listing, updating, and cancelling evaluation jobs, along with endpoints for querying available providers and benchmarks. Evaluation adapter pods receive job specifications via mounted ConfigMaps and call back to EvalHub over HTTPS (port 8443 via kube-rbac-proxy) to report status updates and results.

The repository also produces a Python wheel (`eval-hub-server`) that bundles the compiled Go binary for distribution via PyPI, and includes community adapter container definitions (GuideLLM, LightEval) for running evaluations outside of the built-in lm-evaluation-harness.

## Architectural Analysis

EvalHub follows a clean separation-of-concerns architecture with strategy and factory patterns throughout. The runtime layer abstracts Kubernetes vs. local execution behind a `Runtime` interface, allowing the service to run locally for development (with a no-op `LocalRuntime`) or on-cluster (with `K8sRuntime` that creates batch Jobs). The storage layer similarly abstracts SQLite vs. PostgreSQL behind a `Storage` interface with SQL dialect-specific query generation. This dual-abstraction design makes the service genuinely portable between development and production environments without code changes.

The Kubernetes job execution model is well-designed for multi-benchmark evaluations. When an evaluation job contains multiple benchmarks, EvalHub creates separate Kubernetes batch Jobs (up to 5 concurrently, controlled by a worker pool with buffered channels) — each with its own ConfigMap containing the job specification. The ConfigMap has an owner reference to the Job for garbage collection. Job naming follows DNS-1123 conventions with a SHA1-derived suffix for uniqueness. Each adapter pod receives its configuration via a mounted ConfigMap at `/meta/job.json`, calls back to EvalHub's `/api/v1/evaluations/jobs/{id}/events` endpoint to report progress, and optionally logs to MLflow. The callback URL uses the internal service DNS (`https://<instance>.<namespace>.svc.cluster.local:8443`), ensuring traffic stays within the cluster and traverses kube-rbac-proxy for auth.

The MLflow integration is notable for its attention to OpenShift-specific patterns. It uses projected ServiceAccountToken volumes with 1-hour rotation (rather than static tokens), supports the ODH-specific `X-MLFLOW-WORKSPACE` header for workspace scoping, and properly handles service-serving CA certificates for internal TLS. The TLS configuration enforces a minimum of TLS 1.2 and supports custom CA bundles — though the `InsecureSkipVerify` toggle exists (with appropriate warnings logged) for development scenarios.

A significant architectural observation is that EvalHub has no application-level authentication or authorization. All auth is externalized to kube-rbac-proxy (deployed as a sidecar by the TrustyAI Service Operator), which intercepts traffic on port 8443 and performs Kubernetes TokenReview/SubjectAccessReview before proxying to the application on port 8080. This is consistent with the RHOAI 3.x ingress pattern but means the application cannot distinguish between users or enforce fine-grained access control beyond what Kubernetes RBAC provides. The `Tenant` field exists in the data model (`pkg/api/common.go`) but tenant isolation is not enforced at the application level.

The Collections API (`/api/v1/evaluations/collections`) is defined in the routing and data models but all six endpoints return 501 Not Implemented. The database schema includes a `collections` table, suggesting this feature is planned but not yet delivered. The provider configuration system is file-based (YAML files in `config/providers/`) and loaded at startup — adding new providers requires restarting the service or updating the ConfigMap and redeploying.
