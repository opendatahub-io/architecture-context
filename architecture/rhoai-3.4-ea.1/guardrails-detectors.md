## Purpose

**Short**: A collection of text detection algorithm microservices for the FMS Guardrails Orchestrator, providing PII detection, file-type validation, HuggingFace model-based content classification, and LLM-as-a-judge evaluation.

**Detailed**: The Guardrails Detectors repository provides a set of detector algorithm microservices designed to work with the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator). The orchestrator invokes these detectors to analyze text generation input and output for safety, compliance, and quality concerns.

The repository produces three distinct detector services: (1) a **Built-in Detector** containing lightweight, heuristic-based detection functions such as regex-based PII detection (email, SSN, credit card, phone numbers, IP addresses) and file-type validation (JSON, XML, YAML with optional schema validation), plus a custom detector framework that allows user-defined Python detection logic; (2) a **HuggingFace Runtime Detector** that loads and serves any `AutoModelForSequenceClassification` or `GraniteForCausalLM` model for ML-based content analysis such as toxicity, hate speech, and prompt injection detection; and (3) an **LLM Judge Detector** that integrates the [vLLM Judge](https://github.com/trustyai-explainability/vllm_judge) library to use any OpenAI-compatible LLM as a judge, supporting built-in metrics (safety, toxicity, bias, helpfulness, etc.) and custom natural-language criteria.

All three detectors expose a common REST API conforming to the IBM Detector API specification (`/api/v1/text/contents`) and are designed to be deployed as KServe InferenceServices on Red Hat OpenShift AI. The detectors are maintained by the TrustyAI team and are part of the RHOAI Guardrails feature.

## Architectural Analysis

The Guardrails Detectors repository follows a **shared-library microservice** architecture: all three detectors inherit from a common FastAPI base application (`DetectorBaseAPI`) that provides health checks, exception handling, detector lifecycle management, and Prometheus instrumentation. This pattern ensures consistent API behavior and metrics across detector types while allowing each detector to implement domain-specific logic.

The **built-in detector** uses a registry pattern (`BaseDetectorRegistry`) with three concrete implementations: `RegexDetectorRegistry` for PII detection, `FileTypeDetectorRegistry` for format validation, and `CustomDetectorRegistry` for user-defined Python functions. The custom detector framework is notable — it dynamically loads Python code from `custom_detectors.py`, performs static code analysis to block dangerous imports (`os`, `subprocess`, `sys`, `shutil`) and function calls (`eval`, `exec`, `open`), and provides utility decorators for Prometheus metrics and non-blocking execution. This creates a safe extensibility mechanism for operators to add custom detection logic without modifying the core service.

The **HuggingFace detector** supports two distinct model architectures: `AutoModelForSequenceClassification` for traditional classification models (e.g., toxicity, prompt injection) and `GraniteForCausalLM` for IBM's Granite Guardian causal language model. The sequence classifier path outputs per-label probabilities with configurable threshold and safe label filtering, while the causal LM path generates safe/unsafe classifications with probability scores derived from token log-probabilities. The detector includes CUDA support with automatic GPU detection and memory configuration. A notable design choice is the `SAFE_LABELS` environment variable, which allows per-deployment customization of which model output labels should be treated as non-detections — this is critical because different models may use different label indices for "safe" content.

The **LLM Judge detector** takes a fundamentally different approach by delegating all evaluation logic to an external LLM via the `vllm-judge` library. Rather than running a model locally, it sends structured evaluation prompts to any OpenAI-compatible server. This design is lightweight (no ML framework dependencies) but introduces a network dependency and additional latency. It supports 31 built-in evaluation metrics and allows custom criteria specified in natural language with optional Jinja2 templating, rubrics, and few-shot examples.

Multi-architecture support is a significant engineering investment visible in `Dockerfile.konflux.hf`: the HuggingFace detector builds PyTorch from source for ppc64le and s390x, compiles OpenBLAS from source for ppc64le, and installs cmake from source for both architectures. This adds substantial build complexity and time but enables the detector to run on IBM Power and IBM Z platforms. The OpenSSL version is explicitly pinned (`3.5.1-5.el9_7`) in the Konflux Dockerfile, likely for FIPS or CVE compliance. The built-in detector's Konflux Dockerfile is much simpler since it has no ML framework dependencies.
