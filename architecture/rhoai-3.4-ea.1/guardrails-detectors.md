## Purpose
**Short**: A collection of detector algorithm microservices for the FMS Guardrails Orchestrator, providing text content analysis for safety, PII detection, file-type validation, and LLM-as-a-judge evaluation.

**Detailed**: The guardrails-detectors repository provides a suite of detector microservices that integrate with the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator). Each detector is packaged as a standalone container image that exposes a REST API conforming to the IBM Detector API specification. These detectors analyze text content for various concerns including personally identifiable information (PII), content safety, toxicity, prompt injection, and file-type validation.

The repository produces two container images for RHOAI: the **built-in detector** (lightweight heuristic/algorithmic detections including regex-based PII, file-type validation, and custom Python-code detectors) and the **HuggingFace detector** (a generic detector compatible with HuggingFace AutoModelForSequenceClassification and GraniteForCausalLM models for ML-based content classification). A third detector type, the **LLM Judge**, uses the vllm-judge library to perform LLM-as-a-judge based evaluation but does not have a Konflux build pipeline in this branch and is not shipped as a standalone RHOAI image.

All detectors expose Prometheus metrics for observability (detection counts, request counts, error counts, and runtime latency) and implement the same base API pattern via the shared `DetectorBaseAPI` FastAPI class.

### Request/Response Schemas

**ContentAnalysisHttpRequest** (primary input for all detectors):
- `contents`: List[str] — texts to analyze
- `detector_params`: Optional[Dict] — detector-specific parameters (e.g., `{"regex": ["email", "us-social-security-number"]}`)

**ContentAnalysisResponse** (per-detection result):
- `start`/`end`: int — character positions of detection
- `text`: str — the detected text span
- `detection`: str — detection identifier (e.g., "email_address", "invalid_json")
- `detection_type`: str — category (e.g., "pii", "file_type", "llm_judge")
- `score`: float — confidence score (0.0-1.0)
- `evidences`: Optional[List[EvidenceObj]] — supporting evidence
- `metadata`: Optional[Dict] — additional evaluation metadata

**GenerationAnalysisHttpRequest** (LLM Judge only):
- `prompt`: str — user prompt to the LLM
- `generated_text`: str — LLM-generated response
- `detector_params`: Optional[Dict] — evaluation parameters (e.g., `{"metric": "safety"}`)

### Custom Detector Security

The built-in detector's `CustomDetectorRegistry` performs **static code analysis** on user-provided custom detector Python files before loading them. The following are forbidden:
- **Imports**: `os` (except `os.environ`/`os.getenv`), `subprocess`, `sys`, `shutil`
- **Function calls**: `eval`, `exec`, `open`, `compile`, `input`

If any forbidden patterns are detected, the custom detector module is rejected and the service raises an `ImportError`.

### SSL/TLS Configuration

The `DetectorBaseAPI` supports optional SSL configuration via `config.yaml`:
- `ssl_ca_certs`: Path to CA certificate bundle (triggers `ssl.CERT_REQUIRED` mode)
- Server-level SSL can be configured through the config file or environment variables (`SERVER_*` prefix)
