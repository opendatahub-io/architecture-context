## Purpose

**Short**: A batch-oriented execution environment for language model evaluations using the EleutherAI lm-evaluation-harness framework, integrated with the EvalHub evaluation platform.

**Detailed**: lm-evaluation-harness is a fork of the upstream EleutherAI lm-evaluation-harness (v0.4.8) adapted for Red Hat OpenShift AI. It provides a standardized runtime for executing language model benchmarks as one-shot Kubernetes Job workloads. The component evaluates LLM performance against 156+ benchmark task suites (e.g., ARC, MMLU, HellaSwag, TruthfulQA) using 54 bundled evaluation metrics (accuracy, BLEU, F1, etc.).

In the RHOAI platform, this component is deployed as a container image (`odh-ta-lmes-job-rhel9`) that runs inside Kubernetes Jobs created by the TrustyAI LMEval operator. Each job receives a specification (via mounted ConfigMap at `/meta/job.json`) that defines the model endpoint, benchmark suite, and evaluation parameters. The adapter connects to model inference endpoints using the OpenAI-compatible completions API (`local-completions` backend), runs the benchmark, and publishes results as OCI artifacts to a container registry via `skopeo`. Status updates and results are reported back to EvalHub via HTTP callbacks.

The image is built for multi-architecture support (amd64, s390x, ppc64le) with architecture-specific build stages for PyTorch, Apache Arrow, and OpenBLAS compilation on non-x86 platforms. The container runs as a non-root user (UID 65532) and integrates with the Unitxt evaluation framework for standardized NLP task definitions.

## Architectural Analysis

The lm-evaluation-harness is a compute-only batch workload with no persistent state, no inbound network exposure, and no cluster-level resources (no CRDs, no Services, no RBAC definitions). Its architecture is that of a **job container image** — it runs to completion and exits. All orchestration, scheduling, and lifecycle management is delegated to the TrustyAI LMEval operator.

The EvalHub adapter pattern (`main.py`) cleanly separates the evaluation framework from the platform integration. The `LMEvalAdapter` class extends the `FrameworkAdapter` base from `eval-hub-sdk`, implementing `run_benchmark_job()` to map EvalHub's `JobSpec` into lm-evaluation-harness's `simple_evaluate()` call. This adapter handles five phases (initialization, data loading, evaluation, post-processing, artifact persistence) with progress callbacks at each stage. The design allows the same evaluation engine to be used standalone (via `lm-eval` CLI in `__main__.py`) or through the EvalHub platform (via the adapter).

The model backend system uses a decorator-based registry (`@register_model`) that supports 18+ backends including `local-completions` (OpenAI-compatible API), `hf` (HuggingFace local inference), `vllm` (vLLM server), and `anthropic`. In the RHOAI context, the `local-completions` backend is the primary path — it sends string prompts (not token IDs) to an OpenAI-compatible endpoint, using a HuggingFace tokenizer for context length management. The `verify_certificate` flag on the API model base class (`TemplateAPI`) controls TLS verification, which is architecturally significant in environments with self-signed certificates.

The multi-architecture build is the most complex aspect of the Dockerfile. The Konflux Dockerfile uses 5 build stages: a shared builder base, then parallel stages for Arrow (s390x/ppc64le), PyTorch (s390x/ppc64le), and OpenBLAS (ppc64le only), followed by a final assembly stage. On x86_64/amd64, the PyTorch and Arrow stages are no-ops (pre-built wheels are available from PyPI). On s390x and ppc64le, these libraries are compiled from source, requiring gcc-toolset-13 and custom build flags. The OpenBLAS build targets POWER9 architecture with 120-thread parallelism. A s390x-specific patch (`patches/s390x/parquet-support.patch`) adds endian-compatibility headers to Apache Arrow. The final image runs as UID 65532:65532 (nonroot) with writable directories for HuggingFace cache, output, custom tasks, and Unitxt catalogs.

The OCI artifact pipeline (`scripts/oci.py`) uses `olot` (OCI Layout Tools) to package evaluation output files into an OCI image layout, then pushes via `skopeo`. It supports OCI subject/referrer linking (for artifact provenance chains) and authenticates via Docker config or username/password environment variables. The results JSON includes the full lm-evaluation-harness configuration, per-sample outputs, and computed metrics — all serialized with `model_dump(mode="json")` from pydantic models.
