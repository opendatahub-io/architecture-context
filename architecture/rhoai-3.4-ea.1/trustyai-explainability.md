## Purpose

**Short**: TrustyAI Explainability is a Quarkus-based REST service providing AI fairness metrics, model explainability algorithms, and data drift detection for ML models deployed on KServe/ModelMesh inference platforms.

**Detailed**: TrustyAI is a Java library and containerized service for Explainable AI (XAI). At its core, the `explainability-core` module provides fairness metrics (Statistical Parity Difference, Disparate Impact Ratio), local explainability methods (LIME, SHAP, Counterfactual, Time Series Saliency), global explainability methods (Aggregated LIME, Partial Dependence Plots), and data drift detection algorithms (Kolmogorov-Smirnov test, FourierMMD, Meanshift).

The `explainability-service` module wraps the core library into a Quarkus-based REST microservice that integrates with KServe/ModelMesh inference servers via both HTTP (v1/v2) and gRPC (v2) protocols. It receives inference payloads (model inputs and outputs) either via direct REST POST to a consumer endpoint or via Knative CloudEvents from KServe, reconciles partial input/output payloads into complete inference records, stores them persistently (PVC, MinIO, MariaDB/MySQL, or in-memory), and exposes metrics to Prometheus for monitoring fairness and drift over time. The service supports scheduled recurring metric computations published as Prometheus metrics and provides OpenAPI-documented REST endpoints for on-demand metric calculation and model explanation generation.

The `explainability-connectors` module provides the integration layer with KServe inference servers, implementing the KServe V2 Predict Protocol over both HTTP and gRPC, including Protobuf-based tensor serialization with support for both numpy (NP) and pandas (PD) codec formats. The `explainability-arrow` module enables high-performance columnar data exchange with the TrustyAI Python library using Apache Arrow IPC format.

## Architectural Analysis

TrustyAI Explainability follows a modular Maven multi-module architecture with clean separation between core algorithms, inference server connectors, and the service layer. The `explainability-core` module (195 source files) is a standalone Java library with no framework dependencies, containing implementations of LIME, SHAP, Counterfactual, and Time Series Saliency explainers, plus fairness metrics (SPD, DIR) and drift detectors (KS-test, FourierMMD, Meanshift). This module uses OptaPlanner for the Counterfactual explainer's constraint optimization. The core module is also published independently as a Maven artifact and consumed by the TrustyAI Python library through the Arrow bridge.

The service layer uses Quarkus 3.8.5 with RESTEasy Reactive and employs a hierarchical endpoint design: abstract base classes (`BaseEndpoint`, `GroupEndpoint`, `DriftEndpoint`, `ExplainerEndpoint`) provide common request handling, scheduling, validation, and caching, while concrete endpoint classes implement metric-specific or explainer-specific logic. Feature flags controlled via Quarkus profiles (`%odh`, `%rhoai`) and the custom `@EndpointDisabled` annotation allow the RHOAI distribution to disable explainer endpoints while keeping fairness and drift metrics active. This profile-based gating is notable: RHOAI ships with explainer endpoints disabled, meaning explanation generation is only available in the ODH distribution.

The storage architecture implements a Strategy pattern with four interchangeable backends selected via the `service.storage-format` configuration property and Quarkus's `@LookupIfProperty` conditional bean selection: in-memory (ConcurrentHashMap for development), PVC (flat files at `/inputs`), MinIO (S3-compatible objects), and DATABASE (Hibernate/JPA with MariaDB or MySQL). The PVC and MinIO backends use CSV-format flat files, while the database backend uses a transposed row model where each `DataframeRow` entity stores individual data points with model ID correlation. The MinIO backend has notable limitations: it does not support batch size filtering, data slicing, or tag-based filtering, making it less suitable for production workloads with large datasets.

The inference payload reconciliation system is architecturally interesting. ModelMesh sends input and output payloads separately (partial payloads) with correlation IDs. The `ModelMeshInferencePayloadReconciler` buffers these partial payloads and joins them when both input and output arrive for the same inference ID. KServe uses a different path: CloudEvents consumed via Quarkus Funqy/Knative, handled by a separate `KServeInferencePayloadReconciler`. This dual-reconciler design supports both ModelMesh (partial payload REST) and KServe (CloudEvent) inference logging patterns simultaneously.

The Dockerfile.konflux build is noteworthy because it does NOT compile the Java application at build time. Instead, the application is pre-built by Red Hat's PNC (Product Newcastle) build system, and the resulting artifact ZIP is fetched via Hermeto/cachi2 generic prefetch and simply extracted into the runtime image. This is the `artifacts.lock.yaml` mechanism: the SHA256-locked download URL points to the PNC-hosted build artifact. The runtime image uses `ubi9/openjdk-17-runtime`, keeping the final image minimal. Multi-architecture builds for x86_64, arm64, ppc64le, and s390x are handled by the Konflux/Tekton pipeline.

A security consideration: the base deployment manifests define an OpenShift Route with `tls: null`, meaning traffic to the service is unencrypted by default. The application does support HTTPS on port 4443 with certificate files, but this requires the TrustyAI Operator to provision and mount the certificates. In production RHOAI deployments, the operator injects a kube-rbac-proxy sidecar for authentication, adding a security layer not visible in this repository's manifests. The init container pattern for creating the `model-serving-config` ConfigMap (which configures ModelMesh to send payloads to TrustyAI) uses `oc apply` and requires the `trustyai-serviceaccount` to have ConfigMap write permissions, which is granted by the `trustyai-role` ClusterRole.
