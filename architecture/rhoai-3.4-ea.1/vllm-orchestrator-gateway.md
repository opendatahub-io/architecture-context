## Purpose

**Short**: An HTTP gateway that provides multiple OpenAI-compatible chat completion endpoints with configurable detector-based content filtering, routing client requests to a backend FMS Guardrails Orchestrator.

**Detailed**: The vllm-orchestrator-gateway is a lightweight Rust-based HTTP gateway service that sits between clients and the FMS Guardrails Orchestrator. It enables controlled interaction with the orchestrator by enforcing stricter access to its exposed endpoints through configurable detector pipelines. Each configured route exposes a unique `/{route}/v1/chat/completions` endpoint, allowing drop-in replacement of an unguardrailed chat completions model with a guardrailed one.

The gateway reads a YAML configuration file that defines three key elements: the orchestrator backend connection details, a set of named detectors with their parameters (supporting both input and output detection), and a set of routes that bind specific detectors to specific endpoint paths. When a request arrives on a route, the gateway injects the appropriate detector configuration into the payload and forwards it to the orchestrator's `/api/v2/chat/completions-detection` endpoint. If the orchestrator detects problematic content, the gateway can replace the response with a configurable fallback message. The service supports both standard and streaming (SSE) chat completions.

The gateway supports mTLS for secure communication with the orchestrator, loading TLS certificates and CA bundles from well-known paths (`/etc/tls/private/` and `/etc/tls/ca/`). It forwards `Authorization` and `X-Forwarded-*` headers from incoming requests to the orchestrator, preserving the authentication chain. The service is part of the TrustyAI component family within RHOAI and provides content safety guardrails for vLLM-based model serving.

## Architectural Analysis

The vllm-orchestrator-gateway follows a classic API gateway pattern — it acts as a thin routing and policy-injection layer between clients and a more complex backend (the FMS Guardrails Orchestrator). The gateway itself performs no content detection; its sole responsibility is mapping named routes to detector configurations and forwarding enriched requests to the orchestrator. This separation of concerns keeps the gateway simple and stateless, with all detection logic residing in the orchestrator and its registered detector services.

The configuration-driven routing model (`config/config.yaml`) is the architectural centerpiece. Each route entry creates a dynamic HTTP endpoint at `/{name}/v1/chat/completions`, making the gateway's API surface entirely configurable at startup. Routes bind to named detectors with explicit input/output directionality and parameters. This design enables a single gateway instance to expose multiple guardrail policies — for example, a "pii" route with aggressive PII detection alongside a "passthrough" route with no detectors — all served from the same process. Config validation at startup (`validate_registered_detectors`) ensures route-detector references are consistent, preventing runtime misconfigurations.

The TLS implementation in `build_orchestrator_client()` (src/main.rs:351-406) deserves attention. The gateway dynamically selects HTTP or HTTPS based on whether TLS certificates exist at well-known paths. When certificates are present, it constructs an mTLS client identity by loading PEM files, converting them to PKCS#12 via the `openssl` crate, and creating a `reqwest::Identity`. A notable development convenience is the `danger_accept_invalid_hostnames(true)` call when the orchestrator host is `localhost`, which allows local development with certificates issued for service DNS names. The CA certificate path (`/etc/tls/ca/service-ca.crt`) follows the OpenShift service-ca-operator convention, suggesting deployment in an OpenShift cluster with automatic CA injection.

The streaming implementation handles SSE (Server-Sent Events) pass-through with detection awareness. In streaming mode, the gateway reads SSE chunks from the orchestrator, parses each chunk for detection results, and if a detection is found, replaces the chunk content with the route's fallback message. This means detection-based content blocking works for both streaming and non-streaming flows. The `KeepAlive::default()` configuration on the SSE stream maintains the connection during potentially long LLM generation pauses.

One architectural limitation is the lack of health check or readiness endpoints — the gateway exposes only the dynamic route endpoints. There are no `/healthz`, `/readyz`, or `/metrics` endpoints, which means Kubernetes liveness/readiness probes would need to use TCP socket checks rather than HTTP probes. The `hermetic: false` setting in the Tekton pipeline and the absence of RPM lock files indicate this component is still in early maturity for supply chain hardening, though the Cargo.lock provides reproducible Rust dependency resolution.
