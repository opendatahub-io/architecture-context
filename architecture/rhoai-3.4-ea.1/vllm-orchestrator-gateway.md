## Purpose
**Short**: An OpenAI-compatible HTTP gateway that provides per-route configurable detector-based content filtering for chat completions via the FMS Guardrails Orchestrator.

**Detailed**: The vLLM Orchestrator Gateway is a Rust-based HTTP service that acts as a routing layer between clients and the FMS Guardrails Orchestrator. It exposes multiple OpenAI-compatible `/v1/chat/completions` endpoints, each mapped to a named route with a configurable set of content detectors (e.g., PII detection via regex). When a client sends a chat completion request to a specific route endpoint (e.g., `/{route_name}/v1/chat/completions`), the gateway injects the configured detector specifications into the payload and forwards it to the orchestrator's `/api/v2/chat/completions-detection` endpoint.

The gateway supports both streaming (SSE) and non-streaming chat completions. When detections are found (e.g., PII in output), the gateway can replace the response content with a configurable fallback message per route. It supports mTLS for secure communication with the orchestrator, using OpenSSL-based certificate handling with PKCS#12 identity construction. Configuration is YAML-driven, defining the orchestrator endpoint, detector definitions (with input/output directionality and parameters), and route-to-detector mappings.

**mTLS Configuration**: The gateway supports optional mTLS for orchestrator communication:
- **Client certificate**: `/etc/tls/private/tls.crt` (PEM)
- **Client key**: `/etc/tls/private/tls.key` (PEM)

- When TLS cert/key files are present, HTTPS with client certificate authentication is used automatically
- When only CA cert is present, the orchestrator's server certificate is verified against it
- When no TLS files are present, plaintext HTTP is used
