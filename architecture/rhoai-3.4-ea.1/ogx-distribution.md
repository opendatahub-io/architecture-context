## Purpose

**Short**: Builds and ships a containerized Llama Stack server distribution for Red Hat OpenShift AI, providing a unified API gateway for LLM inference, agents, evaluation, RAG, and safety services across multiple AI providers.

**Detailed**: The ogx-distribution repository produces a container image (`odh-llama-stack-core-rhel9`) that packages [Meta's Llama Stack](https://github.com/llamastack/llama-stack) with a Red Hat-specific configuration and curated set of providers. The image ships Llama Stack version `v0.5.0+rhai0` (a Red Hat-patched fork from `opendatahub-io/llama-stack`) and exposes a single HTTP API on port 8321/TCP that unifies access to inference (vLLM, AWS Bedrock, Vertex AI, Azure, WatsonX, OpenAI), vector storage (Milvus, FAISS, pgvector, Qdrant), safety (TrustyAI FMS), evaluation (TrustyAI LMEval, RAGAS, Garak), agents, tool runtimes (Brave Search, Tavily, MCP, RAG), file storage (local, S3), and batch processing.

The distribution uses a provider activation model driven by environment variables: providers are conditionally enabled at startup based on which environment variables are set (e.g., setting `VLLM_URL` activates the vLLM inference provider). PostgreSQL is used as the persistence backend for key-value and SQL storage. The image also bundles the `ibm-granite/granite-embedding-125m-english` embedding model and supports OpenTelemetry instrumentation when `OTEL_SERVICE_NAME` is set.

The repository contains no operator or controller code — it is purely a container image build project. The `distribution/build.py` script auto-generates a `Containerfile` from a template (`Containerfile.in`) by querying `llama stack list-deps` for the dependency tree. For RHOAI releases, a separate `Dockerfile.konflux` builds from a pre-built wheel release via the AIPCC base image pipeline, rather than installing from source.

## Architectural Analysis

The ogx-distribution follows a **configuration-driven provider activation** pattern that is central to its architecture. Rather than deploying separate containers for each AI provider integration, the distribution image bundles all provider clients into a single image and selectively activates them at startup via environment variable presence checks. This is implemented through Llama Stack's `${env.VAR:+provider_id}` syntax in `config.yaml` — when a variable is set, the provider ID evaluates to a non-empty string and the provider loads; when unset, the provider ID is empty and skipped. This means a single container image serves all deployment scenarios, from a minimal local vLLM-only setup to a full multi-cloud deployment with AWS Bedrock, Vertex AI, and Azure simultaneously.

The build pipeline has an interesting two-tier structure. The upstream community image (`distribution/Containerfile`) is auto-generated by `build.py`, which invokes `llama stack list-deps` to enumerate all Python packages needed by the configured providers, then templates them into Dockerfile `RUN` commands. This ensures the dependency list stays in sync with `config.yaml` changes. The downstream RHOAI image (`Dockerfile.konflux`) takes a completely different approach — it skips the dependency resolution entirely and instead installs a pre-built wheel release (`install-wheel-release.sh`) from a private GitLab registry managed by the AIPCC (AI Platform Container Components) team. The wheel release (`3.4-EA1.1042`) pins exact versions of every dependency for both x86_64 and aarch64 architectures, providing supply chain control at the wheel level even without traditional lock files.

The lack of build hermeticity (`hermetic: false` in the Tekton pipeline) is notable but intentional for this component. The `Dockerfile.konflux` build requires network access to the private GitLab wheel index (`rhel-ai-private-index-auth` secret), making fully hermetic builds impractical without first pre-fetching all wheels. The wheel release mechanism serves as an alternative hermetic control — the AIPCC pipeline produces a self-contained wheel bundle with pinned versions, providing reproducibility at the release level if not at the build level.

The distribution bundles one pre-downloaded model (`ibm-granite/granite-embedding-125m-english`, 125M parameters) directly into the container image via `hf download` in the Dockerfile. This design ensures that the default embedding provider works immediately without external network access at runtime. All other model weights are accessed remotely via the configured inference providers (vLLM, Bedrock, etc.). The embedded model is 768-dimensional and used by default for RAG vector operations.

A security consideration worth noting is that the Llama Stack server itself exposes a completely unauthenticated HTTP API on port 8321. There is no built-in authentication, authorization, or TLS. The expectation is that the RHOAI platform operator deploys a `kube-rbac-proxy` sidecar in front of the Llama Stack container and routes traffic through the platform's Gateway API ingress, which handles TLS termination and token-based authentication. If deployed outside the RHOAI platform (e.g., for local development using `podman run -p 8321:8321`), the API is accessible to anyone who can reach the port.
