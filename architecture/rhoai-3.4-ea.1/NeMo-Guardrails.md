## Purpose

**Short**: NeMo Guardrails is an open-source toolkit for adding programmable guardrails (input/output safety rails, content moderation, jailbreak detection, and retrieval validation) to LLM-based conversational systems.

**Detailed**: NeMo Guardrails, originally developed by NVIDIA, provides a comprehensive framework for controlling and securing LLM-based conversational applications. The toolkit implements a domain-specific language called Colang (versions 1.0 and 2.x) that allows developers to define conversation flows and safety policies declaratively. The server exposes an OpenAI-compatible REST API that intercepts LLM requests, applies configurable input rails (prompt validation, jailbreak detection, sensitive data detection), dialog rails (conversation flow control), and output rails (response filtering, fact checking, content safety), before returning sanitized responses.

In the RHOAI platform, NeMo Guardrails is deployed as a standalone server container that sits between client applications and LLM inference endpoints (such as vLLM or NVIDIA NIM). The server loads guardrail configurations at startup (including Colang flow definitions, guardrail library selections, and pre-downloaded embedding models), processes incoming chat completion requests through the configured rail pipeline, and returns filtered responses. The component supports streaming via Server-Sent Events, multi-configuration deployment (serving multiple guardrail profiles simultaneously), and optional Redis-backed thread state persistence for multi-turn conversations.

The RHOAI distribution uses Konflux for building the container image (`Dockerfile.konflux`) with a multi-stage build that compiles platform-specific dependencies (PyTorch, ONNX Runtime, tiktoken) for ppc64le architecture support, while using pre-built wheels for x86_64 and arm64. The build includes a guardrail profile filter (`scripts/filter_guardrails.py`) that removes closed-source guardrail integrations (ActiveFence, Cleanlab, Clavata, etc.) from the "opensource" profile, and pre-downloads required NLP models (spaCy, sentence-transformers, FastEmbed) into the container image for air-gapped deployment.

## Architectural Analysis

NeMo Guardrails implements a sophisticated multi-layered architecture that transforms a simple LLM chat completion call into a pipeline of programmable safety checks. The core abstraction is the `LLMRails` class, which orchestrates the entire flow: converting messages to an event stream, executing those events through the Colang runtime (either v1.0 or v2.x), applying input/output/dialog/retrieval rails, and producing filtered responses. This event-driven architecture is the key design pattern — every user message, bot response, and context update is represented as an event that flows through a state machine, allowing fine-grained interception and modification at any point.

The Colang DSL is a notable architectural choice. Version 2.x uses a Lark-based parser with a formal grammar (`colang.lark`) that compiles flow definitions into an AST, then executes them via a state machine runtime. This gives users Python-like syntax for defining conversation guards without writing actual Python code. The two-version support (1.0 and 2.x) adds complexity but enables backward compatibility. The runtime detects which version to use based on syntax heuristics (presence of `import` keywords vs. `define` keywords in `.co` files).

The guardrail library architecture follows a plugin pattern. Each of the 25 guardrail types lives in its own directory under `nemoguardrails/library/` with standardized `actions.py` and Colang flow definitions. The build-time profile system (`scripts/filter_guardrails.py` + `scripts/provider-list.yaml`) physically removes closed-source guardrail directories during container build for the "opensource" profile, ensuring proprietary integrations (ActiveFence, Cleanlab, Clavata, Private AI, Fiddler, Patronus AI, Prompt Security, GCP) never ship in the RHOAI image. This is a hard gate — the code is deleted, not just disabled.

A significant security observation is the absence of built-in authentication. The server exposes its API on port 8000 without any token validation, API key checking, or TLS configuration. Request headers are captured and stored in `contextvars.ContextVar` for pass-through to guardrail actions but are never validated for identity. This design assumes network-level security (Kubernetes network policies, service mesh, or RHOAI platform ingress with kube-rbac-proxy). In a RHOAI deployment, the platform's Gateway API infrastructure and kube-rbac-proxy sidecars would provide the authentication layer.

The embedding and caching subsystem is architecturally interesting. It supports 8 different embedding providers through a pluggable interface, with three cache backend options (in-memory, filesystem, Redis). The cache key generation uses MD5 by default but includes a FIPS-aware fallback to SHA-256 in `utils.py` (though not in the embeddings cache module's `MD5KeyGenerator`). The Knowledge Base uses a topic-based chunking strategy for markdown documents, splitting by heading hierarchy, and performs vector search via the configured embedding provider.

The Konflux build pipeline deserves attention for its multi-architecture support. The `Dockerfile.konflux` is an 8-stage build that conditionally compiles PyTorch, tiktoken, ONNX Runtime, OpenBLAS, and hf-xet from source on ppc64le (POWER) architecture while using pre-built wheels on x86_64 and arm64. This is a substantial engineering investment for ppc64le support. The build also pre-downloads NLP models (spaCy `en_core_web_lg`, sentence-transformers, FastEmbed models) into the container image at build time, enabling air-gapped deployment without runtime internet access. The `artifacts.lock.yaml` locks the spaCy model version for hermetic builds.
