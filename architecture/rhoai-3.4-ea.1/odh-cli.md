## Purpose

**Short**: CLI tool for validating, migrating, and backing up Red Hat OpenShift AI (RHOAI) installations and workloads.

**Detailed**: odh-cli (also known as `kubectl-odh` or `rhai-cli`) is a command-line diagnostic and migration tool for RHOAI deployments. It provides three core capabilities: **lint** (cluster validation and upgrade readiness assessment with 40+ diagnostic checks across components, dependencies, and workloads), **migrate** (pluggable migration framework for executing version-specific cluster migrations such as Kueue to Red Hat Build of Kueue), and **backup** (workload and dependency export for disaster recovery and migration preparation).

The tool operates as a read-only Kubernetes client for lint checks and a read-write client for migration operations. It connects to clusters via standard kubeconfig authentication and queries RHOAI-specific CRDs (DataScienceCluster, DSCInitialization), OLM resources (Subscriptions, ClusterServiceVersions), and ML/AI workload CRDs (Notebooks, InferenceServices, RayClusters, PyTorchJobs, etc.). The CLI is distributed as both a standalone binary and a container image (`odh-cli-rhel9`) that bundles `oc`, `kubectl`, `jq`, `yq`, and Python-based upgrade helper scripts from the `rhoai-upgrade-helpers` submodule.

## Architectural Analysis

The odh-cli follows a well-structured modular architecture with clear separation between the command layer (cobra), the domain logic (lint checks, migrate actions, backup pipeline), and the infrastructure layer (Kubernetes client, output rendering). The three-phase command execution pattern (`Complete() -> Validate() -> Run()`) is consistently applied across all commands and ensures clean initialization, input validation, and error handling.

The lint subsystem is the most architecturally significant component, employing a registry-based check framework with 40 diagnostic checks organized into 4 canonical groups (dependencies -> services -> components -> workloads). Each check follows the builder pattern (ComponentBuilder, WorkloadBuilder, OperatorBuilder, DSCIBuilder) that handles common boilerplate like fetching DataScienceCluster singletons, filtering by management state, and populating diagnostic result metadata. The check framework uses a Kubernetes Custom Resource-like pattern for diagnostic results (with conditions, impacts, and impacted objects), which enables structured output in table, JSON, and YAML formats. This design makes it straightforward to add new checks: embed `BaseCheck`, implement `CanApply()` for version filtering, and implement `Validate()` using the appropriate builder.

The backup subsystem implements a concurrent three-stage pipeline (Discovery -> Resolver -> Writer) with configurable worker parallelism. The dependency resolution system uses a registry pattern to map workload GVRs to type-specific resolvers (Notebook resolver extracts ConfigMap/Secret/PVC references; DSPA resolver extracts S3 credentials, database passwords, and CA bundles). The pipeline handles partial failures gracefully — missing dependencies are logged but do not abort the backup. The Kubernetes client is configured with elevated QPS (50) and burst (100) settings to support parallel operations, significantly higher than the kubectl default of 5/10.

The migrate subsystem currently has a single registered migration (Kueue RHBOK) but is designed for extensibility through the Action interface. The RHBOK migration demonstrates the full lifecycle: preflight checks (DSC state, existing subscription conflicts), preparation (backup ClusterQueues and ConfigMap), and execution (annotate ConfigMap, install operator, update DSC, verify resources). The step recording framework provides real-time verbose output with hierarchical step trees and status icons. The migration uses exponential backoff retries (5 attempts, 500ms initial, 2x factor) for conflict resolution during resource updates, which is critical when the operator may be reconciling simultaneously.

A notable architectural decision is the strict read-only/read-write client separation: the lint subsystem uses only the `Reader` interface, preventing diagnostic checks from accidentally modifying cluster state. The migrate subsystem uses the full `Client` interface with write capabilities. This separation is enforced at the type system level. The OLM integration is nil-safe — when OLM is unavailable, the tool degrades gracefully rather than failing, which is important for non-OpenShift environments.
