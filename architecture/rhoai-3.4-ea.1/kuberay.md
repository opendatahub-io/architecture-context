## Purpose
**Short**: KubeRay is a Kubernetes operator for managing Ray cluster lifecycles, including RayCluster, RayJob, and RayService custom resources.

**Detailed**: KubeRay Operator manages the full lifecycle of Ray distributed computing clusters on Kubernetes and OpenShift. It provides three primary custom resources: RayCluster for managing Ray cluster infrastructure (head and worker pods), RayJob for submitting and managing Ray jobs with automatic cluster provisioning, and RayService for deploying Ray Serve applications with zero-downtime upgrades.

The Red Hat fork extends the upstream KubeRay with RHOAI-specific features including: an Authentication Controller that manages OAuth/OIDC authentication via kube-rbac-proxy sidecars and HTTPRoute (Gateway API) ingress per RayCluster; a NetworkPolicy Controller that creates per-cluster network policies for head and worker pods; and an mTLS Controller that provisions cert-manager certificates for intra-cluster encrypted communication. The operator is built with FIPS-compliant Go tooling (`strictfipsruntime`) and runs on UBI9 base images for RHOAI production deployment.

### Ingress

| Name | Type | Hosts | Port | Protocol | Encryption | TLS Mode | Exposure |
|------|------|-------|------|----------|------------|----------|----------|
| {cluster}-ingress | Kubernetes Ingress | Configurable via annotations | 8265/TCP | HTTP | Depends on Ingress class | N/A | External (when enableIngress=true, non-OpenShift) |
| {cluster}-route | OpenShift Route | Auto-generated | 8265/TCP | HTTP | Edge (OpenShift default) | N/A | External (OpenShift legacy, RHOAI 2.x pattern) |
| {cluster} HTTPRoute | HTTPRoute (Gateway API) | Auto-generated | 8265/TCP | HTTPS | TLS (via gateway) | Passthrough to kube-rbac-proxy | External (RHOAI 3.x pattern, created by Authentication Controller) |

**RHOAI 3.x Ingress Pattern (Current)**:
- The Authentication Controller creates HTTPRoute CRs per RayCluster referencing a parent Gateway (e.g., `data-science-gateway`)
- kube-rbac-proxy sidecar (port 8443/TCP) is injected into head pods for RBAC-based authentication
- The controller also creates ReferenceGrant resources for cross-namespace gateway access
- On OpenShift, `enableIngress` is enforced to `false` to prevent duplicate Route/Ingress creation
- Auth flow: Client -> Gateway (HTTPRoute) -> kube-rbac-proxy (8443/TCP, RBAC auth) -> Ray Dashboard (8265/TCP)

**Legacy Pattern (RHOAI 2.x, being phased out)**:
- OpenShift Routes created directly by RayCluster controller for dashboard access
- oauth-proxy sidecar (port 8443/TCP) for OpenShift OAuth-based authentication

### Network Policies (Created Dynamically per RayCluster)

When `odh.ray.io/secure-trusted-network: "true"` annotation is set:

**Head Pod NetworkPolicy (`{cluster}-head`)**:
| Rule | Source | Ports | Purpose |
|------|--------|-------|---------|
| Intra-cluster | Pods with `ray.io/cluster={name}` | All ports | Head-worker communication |
| Same-namespace | Any pod in namespace | 10001/TCP, 8265/TCP | Client and dashboard access |
| Operator access | KubeRay operator namespace | 8265/TCP, 10001/TCP | Operator management |
| Monitoring | MONITORING_NAMESPACE (if set) | 8080/TCP | Prometheus scraping |
| Gateway | GATEWAY_NAMESPACE or `openshift-ingress` | 8265/TCP | Gateway API proxy access |
| Auth proxy | Any source | 8443/TCP | kube-rbac-proxy / oauth-proxy access |
| RayJob submitter | Pods matching batch job name | All ports | Job submitter access (if owned by RayJob) |

**Worker Pod NetworkPolicy (`{cluster}-workers`)**:
| Rule | Source | Ports | Purpose |
|------|--------|-------|---------|
| Intra-cluster only | Pods with `ray.io/cluster={name}` | All ports | Only cluster-internal communication allowed |
