## Purpose
**Short**: KubeRay Operator manages the lifecycle of Ray clusters, jobs, and services on Kubernetes/OpenShift, providing distributed computing infrastructure for AI/ML workloads.

**Detailed**: The KubeRay Operator is a Kubernetes operator built with controller-runtime that reconciles three custom resource types: RayCluster, RayJob, and RayService. It automates the provisioning, scaling, and management of Ray distributed computing clusters. The operator creates and manages head and worker pods, services, ingresses, and handles autoscaling configuration for Ray workloads.

In the RHOAI distribution, the operator includes significant OpenShift-specific enhancements: an Authentication Controller that injects kube-rbac-proxy sidecars into Ray head pods and creates Gateway API HTTPRoute resources for secure dashboard access; a NetworkPolicy Controller that creates fine-grained network policies for Ray cluster isolation based on annotations; and an mTLS Controller that provisions cert-manager Certificate and Issuer resources for mutual TLS between Ray cluster nodes. The operator is deployed via kustomize overlays managed by the rhods-operator, with the OpenShift overlay adding webhook support, SecurityContextConstraints, and Gateway API environment configuration.

**Authentication Controller Ingress Pattern (RHOAI 3.x)**:
- When `odh.ray.io/secure-trusted-network: "true"` annotation is set on a RayCluster:
  1. Creates an HTTPRoute CR in the platform namespace (e.g., `opendatahub`) referencing the `data-science-gateway` Gateway in `openshift-ingress`
  2. Creates a ReferenceGrant in the cluster's namespace allowing cross-namespace Service references
  3. Injects a `kube-rbac-proxy` sidecar container (port 8443/TCP) into the Ray head pod
  4. Creates a ServiceAccount and RBAC ConfigMap for kube-rbac-proxy authorization
  5. HTTPRoute path: `/ray/{namespace}/{cluster}` rewritten to `/` on backend port 8265
- The kube-rbac-proxy sidecar enforces Kubernetes RBAC: users must have `get` verb on `services` resource for the head service

### Network Policies (dynamically created)

When `odh.ray.io/secure-trusted-network: "true"` annotation is set:

**Head Pod NetworkPolicy** (`{cluster}-head`):
- Rule 1: Allow all ports from pods in same RayCluster (intra-cluster)
- Rule 2: Allow ports 10001, 8265 from any pod in same namespace
- Rule 3: Allow ports 8265, 10001 from KubeRay operator namespace
- Rule 4 (optional): Allow port 8080 from monitoring namespace (MONITORING_NAMESPACE env)
- Rule 4b (optional): Allow port 8265 from gateway namespace (GATEWAY_NAMESPACE env, defaults to `openshift-ingress`)
- Rule 5: Allow port 8443 from anywhere (kube-rbac-proxy secured port)

**Worker Pod NetworkPolicy** (`{cluster}-workers`):
- Allow all ports from pods in same RayCluster only (strict intra-cluster isolation)
