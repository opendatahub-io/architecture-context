## Purpose

**Short**: Manages the lifecycle of Ray distributed computing clusters, jobs, and serving applications on Kubernetes/OpenShift.

**Detailed**: KubeRay is a Kubernetes operator built with controller-runtime that manages three custom resources: RayCluster, RayJob, and RayService. It automates the provisioning of Ray head and worker pods, configures networking (Services, Ingresses, Routes, HTTPRoutes via Gateway API), manages authentication sidecars (kube-rbac-proxy for OIDC, oauth-proxy for OpenShift OAuth), and handles mTLS certificate lifecycle through cert-manager integration.

The operator runs six controllers: RayCluster (pod lifecycle and service management), RayJob (job submission and retry logic), RayService (zero-downtime serving with dual-cluster upgrades), NetworkPolicy (secure network isolation), mTLS (certificate generation and rotation), and Authentication (OIDC/OAuth sidecar injection and HTTPRoute management). On OpenShift/RHOAI, the operator integrates with Gateway API to expose Ray dashboards through the platform gateway, creating HTTPRoutes and ReferenceGrants for cross-namespace routing.

The RHOAI distribution adds OpenShift-specific features: SecurityContextConstraints for Ray pods, webhook-based mutation to enforce Gateway API usage (disabling direct Ingress), kube-rbac-proxy sidecar injection for authentication, and NetworkPolicy enforcement for secure cluster isolation. The operator communicates with the Ray Dashboard REST API for job submission, serve application deployment, and health monitoring.

## Architectural Analysis

KubeRay follows a multi-controller pattern where six controllers coordinate through Kubernetes status conditions and annotations rather than direct communication. The RayCluster controller is the central orchestrator, but it defers pod creation until the Authentication controller sets `AuthenticationReady=True` and the mTLS controller provisions TLS secrets. This loosely-coupled coordination avoids race conditions during cluster startup but introduces latency — a new cluster with both mTLS and OIDC must wait for cert-manager certificate issuance and kube-rbac-proxy ConfigMap creation before the first pod is scheduled.

The RHOAI distribution transforms the operator's ingress strategy from direct Ingress/Route creation to the Gateway API pattern. The mutating webhook enforces this by overriding `enableIngress=false` on OpenShift clusters, forcing all external dashboard access through HTTPRoutes referencing the platform's `data-science-gateway`. The Authentication controller creates HTTPRoutes in the platform namespace (not the user namespace), which requires ReferenceGrants to allow cross-namespace service references. ReferenceGrant cleanup uses reference counting — the grant is only deleted when the last authenticated RayCluster in a namespace is removed.

The NetworkPolicy controller implements a defense-in-depth model for secure clusters. Head pods get multi-rule policies: intra-cluster communication is unrestricted, same-namespace pods can reach client (10001) and dashboard (8265) ports, the operator namespace gets the same access, monitoring namespace (if configured) can scrape metrics (8080), and the gateway namespace can reach the dashboard (8265). Worker pods have a single rule: only pods in the same RayCluster can communicate with them. The kube-rbac-proxy sidecar adds application-level auth — it sits on port 8443 and performs SubjectAccessReview checks before proxying to the dashboard on 8265, effectively requiring RBAC `get` permission on the head Service resource.

The RayService controller implements a blue-green deployment pattern for zero-downtime upgrades. When the serve config or cluster spec changes, it creates a "pending" RayCluster alongside the "active" one, deploys the Serve application to the pending cluster, verifies health via the HTTP proxy actor, then atomically switches the Kubernetes Service selector from the active to the pending cluster. The old cluster is deleted after the transition. A config caching layer (keyed by `{namespace}/{service-name}`) avoids redundant Serve deployments when the config hasn't changed, and handles head pod crash recovery by detecting zero running applications as a signal to re-deploy.

The cert-manager integration for mTLS uses a three-tier certificate chain: self-signed issuer bootstraps a root CA certificate, which provisions a CA issuer that signs head and worker certificates. Certificate SANs are dynamically updated as pods scale — the controller polls pod IPs every 60 seconds and updates the Certificate spec, triggering automatic re-issuance. Head certificates include service FQDNs and pod IPs; worker certificates include wildcard DNS patterns to accommodate dynamic scaling. The CA secret name includes the first 8 characters of the RayCluster UID to prevent CA reuse when clusters with the same name are deleted and recreated.
