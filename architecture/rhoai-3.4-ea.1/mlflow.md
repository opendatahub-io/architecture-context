## Purpose
**Short**: MLflow is an open-source ML lifecycle tracking server that manages experiments, model registry, artifact storage, AI gateway endpoints, and LLM tracing, extended for RHOAI with Kubernetes-native workspace isolation and RBAC.

**Detailed**: MLflow provides a centralized tracking server for the machine learning lifecycle. It enables teams to log experiments, compare results, manage model versions, store and serve artifacts, and perform LLM observability via OpenTelemetry trace ingestion. The server exposes REST APIs (Flask + FastAPI) and a GraphQL endpoint, backed by a SQL database (PostgreSQL in production) for metadata and S3-compatible object storage for artifacts.

The Red Hat distribution extends upstream MLflow with a Kubernetes workspace provider that maps MLflow workspaces 1:1 to Kubernetes namespaces, enabling multi-tenant isolation. An authorization plugin (`kubernetes-auth`) enforces Kubernetes RBAC by issuing `SelfSubjectAccessReview` or `SubjectAccessReview` API calls against the `mlflow.kubeflow.org` API group for every request. The server also supports per-namespace artifact storage overrides via an optional `MLflowConfig` CRD (`mlflowconfigs.mlflow.kubeflow.org`), allowing namespace owners to configure workspace-specific S3 buckets. The container image is built via Konflux using UBI9 base images, includes psycopg2 for PostgreSQL connectivity, prometheus-flask-exporter for metrics, and disables upstream telemetry, AI assistant, and AI gateway features by default.

|----------|---------|----------------|-------------------|--------|

| /health, /version, /metrics | GET | None | Unprotected paths | No auth required |
| /static/*, /favicon.ico | GET | None | Unprotected paths | Static assets served without auth |

The authorization plugin supports two modes:
- **SelfSubjectAccessReview** (default): The user's Bearer token is forwarded to the Kubernetes API for self-evaluation.
- **SubjectAccessReview**: The MLflow server performs authorization on behalf of the user using `x-remote-user` and `x-remote-groups` headers (for proxy-based auth).

Authorization decisions are cached with a configurable TTL (default: 300 seconds).
