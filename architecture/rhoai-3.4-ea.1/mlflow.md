## Purpose

**Short**: MLflow is an open-source machine learning lifecycle platform providing experiment tracking, model registry, artifact storage, and AI gateway capabilities, deployed as a multi-tenant web service on Red Hat OpenShift AI.

**Detailed**: MLflow serves as the experiment tracking and model management backbone for RHOAI. It provides a hybrid Flask/FastAPI Python server that exposes REST, GraphQL, and OpenTelemetry-compatible APIs for logging experiments, runs, metrics, parameters, model versions, and distributed traces. The server includes a React/TypeScript frontend for visualization and management.

For RHOAI deployment, MLflow integrates a custom Kubernetes workspace provider (`mlflow-kubernetes-workspace-provider`) that maps Kubernetes namespaces to MLflow workspaces, enabling multi-tenant isolation. This provider enforces Kubernetes RBAC on every API request by submitting `SelfSubjectAccessReview` or `SubjectAccessReview` objects against a custom `mlflow.kubeflow.org` API group, ensuring users can only access data their service account has permission for.

The server supports pluggable backends for tracking data (PostgreSQL, MySQL, SQLite), artifact storage (S3, GCS, Azure Blob, local filesystem, and more), and model registry. In the RHOAI deployment, PostgreSQL is the primary database backend and S3-compatible object storage is used for artifacts. The Dockerfile.konflux builds a single container image that bundles the Python server, React frontend, and Kubernetes workspace provider as a unified deployable unit.

## Architectural Analysis

MLflow in RHOAI represents a significant architectural adaptation of the upstream open-source MLflow project. The core upstream codebase (v3.10.0) is extended with a custom `kubernetes-workspace-provider` plugin that transforms MLflow from a single-tenant application into a Kubernetes-native multi-tenant service. This plugin intercepts every API request at the middleware level and maps it to Kubernetes RBAC decisions, using the `mlflow.kubeflow.org` API group as a virtual resource namespace. This design is elegant because it leverages existing Kubernetes RBAC infrastructure rather than implementing a parallel authorization system — cluster admins manage MLflow permissions using standard Kubernetes `Role` and `RoleBinding` objects.

The server architecture uses a hybrid Flask/FastAPI pattern where Flask serves the traditional Protobuf-based REST API (experiments, runs, models, artifacts) and GraphQL endpoint, while FastAPI handles newer endpoints (jobs, OTel ingestion, AI gateway) as mounted routers within the same process. The React frontend is built at image build time and served as static assets. GraphQL authorization is particularly sophisticated — the workspace provider parses GraphQL queries before execution to determine which Kubernetes resources are being accessed, then submits multiple AccessReview checks for queries that span resource types (e.g., a query that fetches runs with nested model version data triggers both `experiments` and `registeredmodels` permission checks).

The Konflux build pipeline uses a two-phase dependency model: most Python dependencies come from Red Hat's AIPCC (AI Platform Content Catalog) index with cryptographic hash verification, while a small set of packages (prometheus-flask-exporter, psycopg2) are fetched from PyPI with `--require-hashes`. The Hermeto/cachi2 prefetch integration is conditional (`if [ -f /cachi2/cachi2.env ]`), allowing the same Dockerfile to work both in Konflux CI (hermetic) and local development. RPM-level hermeticity is provided by `rpms.lock.yaml` covering the two runtime-installed packages (openblas, openblas-threads).

A notable security consideration is the KEK (Key Encryption Key) management system for server-side secrets encryption. MLflow stores secrets (e.g., gateway API keys) encrypted with AES-GCM, derived from a passphrase via PBKDF2. The KEK rotation workflow requires server downtime and is manual, which is appropriate for the deployment model but worth noting for operational runbooks. The `cryptography` library used for this is FIPS-aware when running on UBI 9 with FIPS mode enabled. The `hashlib.md5` usage throughout the codebase is explicitly marked with `usedforsecurity=False`, which is the correct Python 3.9+ pattern for non-security checksums that should not trigger FIPS enforcement.

The workspace provider's authorization caching (300-second TTL with SHA256-based cache keys) is important for performance — without it, every API request would trigger a Kubernetes AccessReview round-trip. The namespace watching architecture uses background threads with event-driven updates, maintaining an in-memory cache that refreshes automatically on Kubernetes watch events. The MLflowConfig CRD watch gracefully handles CRD absence (retries every 5 minutes), allowing deployment before the CRD is installed.
