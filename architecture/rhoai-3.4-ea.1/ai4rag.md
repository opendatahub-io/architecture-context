## Purpose

**Short**: ai4rag is a provider-agnostic optimization engine that automatically finds optimal hyperparameter configurations for Retrieval-Augmented Generation (RAG) pipelines.

**Detailed**: ai4rag accepts RAG template definitions and a search space of hyperparameters (chunking strategy, embedding model, retrieval method, foundation model, etc.), then uses Generalized Additive Models (GAM) or random search to systematically evaluate parameter combinations and return the best-performing RAG configuration (called a "RAG Pattern") for a given metric (faithfulness, answer correctness, or context correctness).

The library is designed to be LLM and vector database provider-agnostic. Users supply abstract implementations of foundation models, embedding models, and vector stores. Out of the box, ai4rag integrates with [OGX](https://github.com/ogx-ai/ogx) for embeddings, vector storage, and text generation, and with ChromaDB for in-memory local vector search. Evaluation is powered by IBM's [unitxt](https://pypi.org/project/unitxt/) library for computing RAG metrics.

ai4rag is not a deployed service or Kubernetes operator. It is a pure Python library consumed as a dependency by other platform components (e.g., Kubeflow Pipelines components) or used directly in Jupyter notebooks and scripts. It has no container images, no CRDs, no deployment manifests, and no network-facing services. Its integration into the RHOAI platform is as a library dependency, not as an independently deployed workload.

## Architectural Analysis

ai4rag follows a clean provider-agnostic architecture with abstract base classes at every integration boundary. `BaseFoundationModel`, `BaseEmbeddingModel`, `BaseVectorStore`, `BaseChunker`, `BaseEvaluator`, and `BaseEventHandler` define the contracts, while concrete implementations (OGX-based and Chroma-based) are injected at experiment creation time. This inversion-of-control pattern makes it straightforward to swap providers without modifying the optimization core.

The optimization engine implements a two-phase approach. First, a **Models Pre-Selector (MPS)** evaluates all foundation model x embedding model pairs on a small benchmark sample using a local ChromaDB instance, ranking them by the optimization metric. This narrows the search space to the top N models before the expensive full HPO begins. Second, the **GAMOptimizer** uses a Generalized Additive Model (via pygam) to predict scores for unexplored parameter combinations and greedily selects the most promising ones for evaluation. The GAM is retrained after each batch of evaluations, implementing a form of Bayesian-lite optimization that is more sample-efficient than random search while being computationally cheaper than full Gaussian Process methods.

The search space management is notably sophisticated. `AI4RAGSearchSpace` enforces six built-in constraint rules that prune invalid combinations before they reach the optimizer: chunk size must exceed 2x chunk overlap, window retrieval requires non-zero window size, chunk token count must fit the embedding model's context length, and three rules governing hybrid search ranker parameter consistency. These rules prevent the optimizer from wasting evaluations on configurations that would fail at runtime.

A notable architectural pattern is the **evaluation caching** in `ExperimentResults.evaluation_explored_or_cached()` and the **collection reuse** via `_get_reusable_collection_name()`. When the same indexing parameters (chunking + embedding) appear in multiple HPO iterations, the already-built vector store index is reused rather than re-created, significantly reducing experiment wall time since embedding and indexing are often the most expensive steps.

The event handler abstraction (`BaseEventHandler`) decouples the experiment engine from result persistence. `LocalEventHandler` writes pattern results to JSON files on disk; `KFPEventHandler` aggregates them in memory for Kubeflow Pipelines components. This extensibility point is explicitly documented as the primary integration mechanism for platform services that consume ai4rag.

The library uses concurrent RAG inference via `ThreadPoolExecutor` (default 10 threads) in `query_rag()`, parallelizing the question-answering step across all benchmark questions for a given pattern. This is bounded to avoid overwhelming the downstream LLM inference endpoint.
