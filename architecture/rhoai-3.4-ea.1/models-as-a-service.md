## Purpose

**Short**: MaaS API is a Go service that provides multi-tenant API key management, tier-based access control, and OpenAI-compatible model discovery for LLM inference services on OpenShift AI.

**Detailed**: Models as a Service (MaaS) is a platform component that creates a managed gateway layer on top of KServe LLMInferenceService resources. It provides a centralized API for issuing and managing ephemeral and persistent API keys (backed by Kubernetes ServiceAccount tokens), discovering available LLM models through an OpenAI-compatible `/v1/models` endpoint, and resolving user-to-tier mappings for consumption-based rate limiting.

The MaaS API sits behind a Gateway API (HTTPRoute) ingress path and integrates with Kuadrant/Authorino for authentication and authorization. Users authenticate via Kubernetes TokenReview, and the Authorino auth policy enriches requests with user identity headers (`X-MaaS-Username`, `X-MaaS-Group`). The API then maps user groups to subscription tiers (free/premium/enterprise) using a ConfigMap-driven tier configuration, creates per-tier namespaces and per-user ServiceAccounts, and issues scoped tokens that are validated by downstream Istio AuthorizationPolicies at the model serving layer.

The component uses a pluggable storage backend for API key metadata persistence: in-memory (ephemeral default), on-disk SQLite (single-replica persistent), or external PostgreSQL (multi-replica HA). Model discovery works by listing all LLMInferenceService resources, filtering those attached to the configured Gateway, and concurrently probing their `/v1/models` endpoints using the requesting user's ServiceAccount token to enforce authorization at the model server level.

## Architectural Analysis

The MaaS API implements a multi-tenant model access layer using Kubernetes-native primitives. The core design insight is using Kubernetes ServiceAccounts as user-scoped access tokens: when a user requests a token, MaaS creates a namespace per tier (e.g., `maas-default-gateway-tier-free`) and a ServiceAccount per user within that namespace, then issues tokens via the Kubernetes TokenRequest API. These tokens carry the correct audience claim (`maas-default-gateway-sa`) and are validated by Authorino at the gateway level using Kubernetes TokenReview. This avoids building a separate identity/token system — it piggybacks on Kubernetes RBAC and the TokenReview mechanism already available in the cluster. Token revocation works by deleting and recreating the user's ServiceAccount, which invalidates all previously issued tokens.

The model discovery flow is particularly noteworthy. Rather than maintaining a static registry, MaaS dynamically discovers models by listing all `LLMInferenceService` CRs (via shared informer cache), filtering those attached to the configured Gateway (via spec.router gateway references or HTTPRoute parentRef matching), and then concurrently probing each model's `/v1/models` endpoint using the requesting user's ServiceAccount token. This means model visibility is implicitly authorization-filtered: if the user's SA token lacks permissions to access a model (enforced by Istio AuthorizationPolicies at the model server), that model is excluded from the response. The discovery uses exponential backoff with retry on transient failures and handles edge cases like 405 responses (method not allowed but auth succeeded, uses spec.model.name as fallback). A risk worth noting: `InsecureSkipVerify: true` is used on the model discovery HTTP client because cluster-internal traffic uses service-ca certificates that the MaaS pod doesn't have in its trust store. The code documents this as an intentional trade-off.

The tier system drives both authorization and rate limiting. The `tier-to-group-mapping` ConfigMap maps Kubernetes groups to tiers with levels (higher wins). Authorino evaluates the gateway-level AuthPolicy, which calls MaaS's `/v1/tiers/lookup` endpoint to resolve the user's tier, caches the result for 5 minutes, and injects the tier into response headers. The Kuadrant `TokenRateLimitPolicy` then uses the tier from the identity context to apply tier-specific token rate limits (free: 100/min, premium: 50,000/min, enterprise: 100,000/min). This separation of concerns — MaaS provides tier resolution, Kuadrant enforces limits — keeps the rate limiting logic declarative and out of application code.

The storage architecture is intentionally simple and pluggable. API key metadata (name, description, JTI, creation/expiration dates) is stored in a SQL database using a generic interface that supports both SQLite (via `mattn/go-sqlite3` with CGO) and PostgreSQL (via `jackc/pgx`). The SQLite backend uses WAL journal mode and single-connection pooling to avoid lock contention, while PostgreSQL uses configurable connection pooling. The separation between "token" (ephemeral, Kubernetes-native) and "API key" (persistent, database-backed) is clean: an API key embeds a Kubernetes SA token but adds a name and persisted metadata for management purposes.
