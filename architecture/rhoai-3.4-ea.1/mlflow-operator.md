## Purpose

**Short**: Kubernetes operator that deploys and manages MLflow tracking server instances on OpenShift/Kubernetes using a Helm-chart-based reconciliation pattern.

**Detailed**: The MLflow Operator watches for `MLflow` custom resources (cluster-scoped) and reconciles them into fully functional MLflow tracking server deployments. It uses an embedded Helm chart (`charts/mlflow/`) to render Kubernetes manifests (Deployment, Service, NetworkPolicy, ServiceAccount, PVC, RBAC) and applies them via Server-Side Apply. The operator supports multiple MLflow instances with unique resource naming via a suffix convention derived from the CR name.

The operator integrates with the RHOAI platform's Gateway API ingress pattern by conditionally creating `HTTPRoute` resources that reference the platform's `data-science-gateway` Gateway, enabling MLflow to be exposed through the centralized platform ingress with URL path rewriting (e.g., `/mlflow/api/*` → `/api/*`). It also conditionally creates OpenShift `ConsoleLink` resources to add MLflow to the OpenShift console's application menu. The operator handles CA bundle management through an init container and sidecar pattern that combines system, platform (`odh-trusted-ca-bundle`), and user-provided CA certificates into a unified trust store for the MLflow container.

A second CRD, `MLflowConfig` (namespace-scoped, under `mlflow.kubeflow.org`), allows namespace owners to override artifact storage credentials per namespace, enabling multi-tenant workspace configurations where each team uses its own S3-compatible storage. The MLflow server itself runs with Kubernetes-native authentication (`--app-name=kubernetes-auth`) using SubjectAccessReview for authorization.

## Architectural Analysis

The MLflow Operator employs a Helm-chart-as-library pattern: rather than deploying a Helm release via Tiller/Helm CLI, it embeds the chart (`charts/mlflow/`) within the operator binary (copied via `COPY --from=builder /workspace/charts/mlflow charts/mlflow` in the Dockerfile) and uses the Helm Go SDK (`helm.sh/helm/v3`) to render templates in-memory during reconciliation. This approach gives the operator the expressiveness of Helm templates (conditionals, loops, helpers) while maintaining the operator lifecycle model (watches, status conditions, owner references). All rendered objects are applied via Server-Side Apply with `mlflow-operator` as the field owner, which cleanly handles merge semantics and avoids the "last writer wins" problem of client-side apply.

The operator's ingress integration follows the RHOAI 3.x Gateway API pattern: it conditionally creates `HTTPRoute` resources that reference the platform's `data-science-gateway` Gateway in the `openshift-ingress` namespace. The HTTPRoute defines three path-based rules with URL rewriting (e.g., `/mlflow/api/*` → `/api/*`) to integrate cleanly with the MLflow server's expectations. Importantly, both HTTPRoute and ConsoleLink creation are gated on runtime CRD discovery — the operator checks `ServerResourcesForGroupVersion` at startup and only watches/creates these resources if the corresponding CRDs exist in the cluster. This allows the same operator binary to run on clusters with or without Gateway API and OpenShift Console installed.

The CA bundle management architecture is notable: when platform or custom CA ConfigMaps are detected, the operator renders an init container (`combine-ca-bundles`) that concatenates system CA (`/etc/pki/tls/certs/ca-bundle.crt`), platform CA (`odh-trusted-ca-bundle`), and user-provided CA certificates into a single PEM file on an emptyDir volume. A sidecar container (`ca-bundle-watcher`) polls every 30 seconds for ConfigMap changes (using SHA256 checksums) and atomically replaces the combined bundle. The MLflow container then configures multiple environment variables (`SSL_CERT_FILE`, `REQUESTS_CA_BUNDLE`, `CURL_CA_BUNDLE`, `AWS_CA_BUNDLE`, `PGSSLROOTCERT`, `MLFLOW_MYSQL_CA`) to point all Python libraries at this combined bundle. This is a thorough approach that covers Python requests, boto3, psycopg2, and MySQL connectors.

The multi-instance support through the `getResourceSuffix()` function is cleanly designed: a CR named "mlflow" produces resources named "mlflow" (no suffix), while any other name produces "mlflow-{name}" resources. The shared `mlflow` ClusterRole uses non-controller owner references (multiple MLflow instances can share it), with a custom `appendOwnerReference()` method that preserves existing owners and a dedicated `Watches()` handler that enqueues all owning MLflow instances when the ClusterRole changes. This avoids the controller-runtime limitation where `Owns()` only works with single controller owner references.

The MLflow server authentication uses Kubernetes-native SubjectAccessReview (`MLFLOW_K8S_AUTH_AUTHORIZATION_MODE=self_subject_access_review`), meaning every API request to the MLflow server triggers a check against the requesting user's Kubernetes RBAC permissions. The aggregate roles (`mlflow-view`, `mlflow-edit`) are designed to integrate with Kubernetes' built-in role aggregation — they automatically merge into the standard `view`, `edit`, and `admin` ClusterRoles, giving existing namespace role holders appropriate MLflow access without explicit role binding.
