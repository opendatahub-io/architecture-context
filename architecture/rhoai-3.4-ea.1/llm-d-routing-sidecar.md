## Purpose

**Short**: A reverse proxy sidecar that enables disaggregated prefill/decode inference by routing prefill requests to designated prefill worker pods and decode requests to the co-located vLLM decoder.

**Detailed**: The llm-d-routing-sidecar is a lightweight HTTP/HTTPS reverse proxy designed to run as a sidecar container alongside a vLLM decoder instance in an llm-d disaggregated inference deployment. It implements the Prefill/Decode (P/D) disaggregation pattern where LLM inference is split into two phases: a prefill phase (computing KV cache for the prompt) that can run on a remote GPU worker, and a decode phase (autoregressive token generation) that runs locally.

The sidecar intercepts incoming OpenAI-compatible completion requests at `/v1/chat/completions` and `/v1/completions`. When a request includes an `x-prefiller-host-port` header, the sidecar orchestrates a two-phase protocol: it first sends the request to the specified prefill pod, receives KV transfer metadata in the response, then forwards the original request (augmented with KV transfer parameters) to the local vLLM decoder for token generation using the remotely-computed KV cache. Three connector protocols are supported: NIXLv2 (current default), NIXLv1 (deprecated), and LMCache (deprecated).

The sidecar also provides optional SSRF (Server-Side Request Forgery) protection by watching Kubernetes InferencePool resources and maintaining a dynamic allowlist of valid prefill target pod IPs. When enabled, only prefill targets matching pods selected by the specified InferencePool are permitted, preventing request forgery to arbitrary endpoints. Note: this repository is deprecated and its code has been moved to the llm-d-inference-scheduler repository under `cmd/pd_sidecar` and `pkg/sidecar`.

## Architectural Analysis

The llm-d-routing-sidecar implements a focused reverse proxy pattern for disaggregated LLM inference. Its architecture is notably simple: a single Go binary with no framework dependencies beyond the standard library's `net/http/httputil.ReverseProxy`. The proxy maintains an LRU cache (capacity 16) of prefiller reverse proxy handlers to avoid re-creating them for repeated requests to the same host:port, which is a practical optimization for the pod-to-pod communication pattern in Kubernetes deployments.

The connector abstraction (`protocolRunner` function type) cleanly separates the three P/D protocols (NIXLv2, NIXLv1, LMCache) behind a common interface. NIXLv2 is the most sophisticated: it modifies the prefill request to set `max_tokens=1` (preventing the prefiller from generating tokens), wraps KV transfer parameters in a `kv_transfer_params` object, and preserves the original stream/token settings for the decode phase. NIXLv1 uses a flatter request structure with top-level `do_remote_prefill`/`do_remote_decode` fields. LMCache uses the simplest approach, sending the full request to the prefiller with `max_tokens=1` and then forwarding the original to the decoder without KV metadata. All three protocols buffer the entire prefill response in memory (`bufferedResponseWriter`), which could be a concern for very large responses.

The SSRF protection subsystem is architecturally interesting as a defense-in-depth measure. It uses Kubernetes dynamic client with shared informers to watch a specific InferencePool custom resource and then dynamically watches pods matching that pool's label selectors. The allowlist is host/IP-based (not port-specific), rebuilt on every pod add/update/delete event, and uses a read-write mutex for concurrent access. The informer-based approach means the allowlist stays current without polling, though there is a 30-second resync period as a safety net. This protection is opt-in (disabled by default for backward compatibility) and requires appropriate RBAC permissions for pods and inferencepools resources.

TLS configuration is thorough: the server, decoder proxy, and prefiller proxy all use identical cipher suite restrictions (6 AEAD ciphers, TLS 1.2 minimum). The `InsecureSkipVerify` flags for prefiller and decoder connections are available for development/testing but disabled by default. The self-signed certificate fallback (4096-bit RSA, 10-year validity) is explicitly noted as being for testing. The HTTP server configuration intentionally omits `ReadTimeout` and `WriteTimeout` because LLM inference can take hours for large contexts, using only `IdleTimeout` (5 minutes) and `ReadHeaderTimeout` (30 seconds).

This repository is deprecated, with code moved to `llm-d-inference-scheduler` under `cmd/pd_sidecar` and `pkg/sidecar`. The absence of a Konflux Dockerfile on this branch suggests the RHOAI build pipeline references the downstream fork or the new consolidated repository.
