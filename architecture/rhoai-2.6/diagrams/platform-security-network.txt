AUTHENTICATION MECHANISMS
═══════════════════════════════════════════════════════════════════════════════════════

1. Bearer Tokens (JWT) - OpenShift OAuth
   - Components: Dashboard, Notebooks, Pipelines UI, Prometheus, Alertmanager, TrustyAI
   - Enforcement: OAuth Proxy sidecar (validates JWT via OpenShift OAuth server)
   - Token Type: OpenShift user token (expires in 24 hours by default)
   - Authorization: SubjectAccessReview (Kubernetes RBAC check)

2. Bearer Tokens - Service Accounts
   - Components: All operators, internal service-to-service calls
   - Enforcement: Kubernetes API Server
   - Token Type: ServiceAccount JWT (projected volume, auto-rotated hourly)
   - Authorization: ClusterRole and Role bindings

3. mTLS Client Certificates (Service Mesh)
   - Components: KServe InferenceServices (optional), ModelMesh (optional)
   - Enforcement: Istio Envoy sidecar
   - Certificate Type: x.509 client certs issued by Service Mesh CA
   - Validation: Mutual TLS handshake, service identity verification

4. AWS IAM Credentials
   - Components: Storage initializers (KServe, ModelMesh, Pipelines)
   - Enforcement: S3-compatible storage API
   - Credentials: AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY (from storage-config secret)
   - Signature: AWS Signature Version 4 HMAC

5. Database Credentials
   - Components: Data Science Pipelines, MLMD, ModelMesh (etcd)
   - Enforcement: Database server (MySQL, PostgreSQL, etcd)
   - Credentials: Username/password from secrets
   - Encryption: TLS 1.2+ (optional, recommended for production)

6. SubjectAccessReview (SAR)
   - Components: Notebook OAuth proxy, TrustyAI OAuth proxy
   - Enforcement: Kubernetes API Server via OAuth proxy
   - Check: User must have GET permission on notebook/service resource
   - Example: "Can user 'alice' GET notebook 'alice-notebook' in namespace 'alice-project'?"

═══════════════════════════════════════════════════════════════════════════════════════
SECURITY CONTEXT CONSTRAINTS (SCC)
═══════════════════════════════════════════════════════════════════════════════════════

All RHOAI workloads run under OpenShift's Restricted SCC (default):
- Run as non-root user (UID 1000+ from namespace UID range)
- No privileged escalation
- No host namespaces (network, PID, IPC)
- No capabilities beyond default set
- Read-only root filesystem (where possible)
- SELinux enforced (MCS labels)

Exceptions (require custom SCC):
- None (RHOAI 2.6 fully supports restricted SCC)

COMPLIANCE NOTES
═══════════════════════════════════════════════════════════════════════════════════════

Encryption in Transit:
- External ingress: TLS 1.2+ (HTTPS/443) on all Routes
- Service Mesh internal: mTLS (optional, PERMISSIVE mode allows plaintext fallback)
- S3 storage: HTTPS/443 (TLS 1.2+)
- Database connections: TLS 1.2+ (optional, recommended)
- Kubernetes API: HTTPS/6443 (TLS 1.2+)

Encryption at Rest:
- Secrets: Encrypted by etcd (Kubernetes secret encryption at rest)
- Persistent Volumes: Block storage encryption (cluster-level configuration)
- S3 artifacts: Server-side encryption (SSE-S3, SSE-KMS) recommended

Audit Logging:
- Kubernetes API audit: All CRD operations logged
- OpenShift OAuth audit: Authentication events logged
- Application logs: Stdout/stderr captured by OpenShift logging

Secret Management:
- Auto-rotation: Service CA certificates (90 days)
- Manual rotation: OAuth client secrets, storage credentials, database passwords
- Least privilege: Secrets scoped to namespace, mounted as volumes (not env vars)

Network Segmentation:
- Namespace isolation: NetworkPolicies block cross-namespace traffic
- Service Mesh: Optional mTLS enforcement for InferenceServices
- External egress: Controlled via NetworkPolicies and firewall rules
