SERVICE MESH CONFIGURATION (Optional)
═══════════════════════════════════════════════════════════════════════════

When Service Mesh is enabled (serviceMesh.managementState=Managed):
- PeerAuthentication: STRICT (namespace-scoped)
  • Enforces mTLS for all pod-to-pod communication
  • Service identity based on ServiceAccount
- AuthorizationPolicy: Component-specific access control
  • Example: Allow KServe controller to manage InferenceService pods
  • Methods: GET, POST, PUT, DELETE, PATCH
- Gateway: Istio ingress gateway for external traffic routing
- VirtualService: Traffic routing rules per component

SECRETS
═══════════════════════════════════════════════════════════════════════════

webhook-server-cert (kubernetes.io/tls)
- Purpose: Webhook admission controller TLS certificate
- Provisioned by: cert-manager or OLM webhook certificate injection
- Rotation: No auto-rotation (managed by cert-manager/OLM)
- Mounted at: /tmp/k8s-webhook-server/serving-certs/

{component}-oauth-client (Opaque)
- Purpose: OAuth client ID/secret for component authentication
- Provisioned by: SecretGenerator controller
- Fields: clientId, clientSecret
- Rotation: Manual (no auto-rotation)

prometheus-proxy-tls (kubernetes.io/tls)
- Purpose: OAuth proxy TLS for Prometheus Route
- Provisioned by: OpenShift Service CA
- Rotation: Automatic (service CA managed)

alertmanager-proxy-tls (kubernetes.io/tls)
- Purpose: OAuth proxy TLS for Alertmanager Route
- Provisioned by: OpenShift Service CA
- Rotation: Automatic (service CA managed)
