## Purpose
**Short**: KServe is a Kubernetes-native platform for serving machine learning models with support for serverless (Knative), raw Kubernetes deployments, and ModelMesh integration.

**Detailed**: KServe provides a standardized model inference platform on Kubernetes that abstracts away the complexity of deploying, scaling, and managing ML model serving infrastructure. It defines Custom Resource Definitions (CRDs) such as InferenceService, ServingRuntime, ClusterServingRuntime, InferenceGraph, TrainedModel, and ClusterStorageContainer to declaratively manage the lifecycle of model serving workloads.

The KServe controller manager reconciles InferenceService resources by creating the appropriate underlying infrastructure depending on the configured deployment mode: Knative Services for serverless auto-scaling, raw Kubernetes Deployments for direct deployment, or delegating to ModelMesh for multi-model serving. The controller also manages ingress through Istio VirtualServices (serverless mode) or Kubernetes Ingresses (raw deployment mode), and provides webhook-based pod mutation for injecting storage initializers, model agents, loggers, and batchers as sidecar containers.

In the RHOAI 2.6 distribution (this branch), KServe is deployed via the ODH overlay which configures the controller within the opendatahub namespace, integrating with Knative Serving and Istio service mesh for serverless inference with auto-scaling and traffic management.

| Name | Type | Hosts | Port | Protocol | Encryption | TLS Mode | Exposure |
|------|------|-------|------|----------|------------|----------|----------|
| {isvc-name} (Serverless mode) | Istio VirtualService | {isvc-name}-{namespace}.{domain}, {isvc-name}.{namespace}.svc.cluster.local | 80/TCP | HTTP | Depends on gateway config | SIMPLE (via Istio gateway) | External via Istio IngressGateway |
| {isvc-name} (Raw mode) | Kubernetes Ingress | {isvc-name}-{namespace}.{domain} | 80/TCP | HTTP | Depends on ingress controller | Configurable | External via Ingress Controller |

**Ingress Details (Serverless/Istio mode)**:
- The controller creates an Istio VirtualService per InferenceService that routes traffic through both `knative-serving/knative-ingress-gateway` (external) and `knative-serving/knative-local-gateway` (internal)
- An ExternalName Service is created per InferenceService pointing to the local gateway service (`knative-local-gateway.istio-system.svc.cluster.local`)
- VirtualService routes set Host headers to direct traffic to the correct Knative Service (predictor, transformer, or explainer)
- Path-based routing is supported via `pathTemplate` configuration
- When `disableIstioVirtualHost` is true, VirtualService creation is skipped and Knative's native networking is used

**Ingress Details (Raw Deployment mode)**:
- The controller creates a Kubernetes Ingress per InferenceService with `ingressClassName` from configuration
- Ingress rules route to predictor/transformer/explainer Services on port 80

## Deployment Configuration

### Deployment Modes

| Mode | Value | Description | Creates |
|------|-------|-------------|---------|
| Serverless | `Serverless` (default) | Uses Knative Serving for auto-scaling and scale-to-zero | Knative Services, Istio VirtualServices |
| Raw Deployment | `RawDeployment` | Direct Kubernetes Deployments without Knative | Deployments, Services, K8s Ingresses |
| ModelMesh | `ModelMesh` | Multi-model serving via ModelMesh controller | Delegates to ModelMesh (only transformer reconciled by KServe) |

### ConfigMap Configuration

The `inferenceservice-config` ConfigMap in the kserve namespace controls all runtime behavior:
- **deploy**: Default deployment mode (Serverless/RawDeployment/ModelMesh)
- **ingress**: Gateway, domain, URL scheme, VirtualHost toggle
- **storageInitializer**: Image and resource limits for model download init container
- **logger/batcher/agent**: Sidecar images and resource limits
- **router**: InferenceGraph router image and resource limits
- **explainers**: Explainer runtime images (AliBI, ART)
- **credentials**: Cloud storage credential configuration (S3, GCS)
- **metricsAggregator**: Prometheus metrics aggregation settings

### Serving Runtimes

Pre-configured ClusterServingRuntimes for common ML frameworks:
- TensorFlow Serving, Scikit-learn Server, XGBoost Server
- Triton Inference Server, TorchServe, MLServer
- PMML Server, PaddlePaddle Server, LightGBM Server
