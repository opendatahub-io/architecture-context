## Purpose
**Short**: KubeRay is a Kubernetes operator that manages Ray cluster lifecycle, including provisioning, scaling, and serving Ray workloads on OpenShift/Kubernetes.

**Detailed**: KubeRay operator provides three Custom Resource Definitions (CRDs) -- RayCluster, RayJob, and RayService -- that enable users to declaratively manage Ray distributed computing clusters on Kubernetes and OpenShift. The operator handles the full lifecycle of Ray clusters, including creating head and worker pods, managing services for inter-node communication, configuring ingress (OpenShift Routes or Kubernetes Ingress), setting up autoscaler RBAC, and managing GCS fault tolerance with Redis cleanup.

The RayJob controller extends cluster management by creating Kubernetes batch Jobs that submit Ray jobs to the cluster's dashboard API, tracking job status, and optionally tearing down clusters after job completion. The RayService controller manages Ray Serve deployments with zero-downtime upgrades by maintaining active and pending clusters, health checking serve deployments, and switching traffic between clusters.

This is the Red Hat fork of the upstream KubeRay project, customized for deployment within the Red Hat OpenShift AI (RHOAI) platform. It includes OpenShift-specific features such as Route creation for dashboard access, SecurityContextConstraints for the Ray user (UID 1000), and integration with the RHOAI component deployment model via kustomize overlays in `ray-operator/config/openshift/`.

The controller auto-detects OpenShift by querying API server groups for `*.openshift.io`. On OpenShift, it creates Routes; on vanilla Kubernetes, it creates Ingresses. This behavior can be overridden with `USE_INGRESS_ON_OPENSHIFT=true` environment variable. Ingress/Route creation is opt-in via `spec.headGroupSpec.enableIngress: true` on the RayCluster CR.
