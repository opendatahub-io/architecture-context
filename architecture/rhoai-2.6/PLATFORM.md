## Platform Overview

Red Hat OpenShift AI (RHOAI) 2.6 is an enterprise AI/ML platform deployed on OpenShift Container Platform that provides a comprehensive suite of tools for data scientists and ML engineers. The platform enables end-to-end ML workflows including interactive notebook development (Jupyter, VS Code, RStudio), ML pipeline orchestration (Kubeflow Pipelines with Tekton), model serving (KServe for single-model serverless serving and ModelMesh for multi-model high-density serving), distributed computing (Ray via KubeRay, CodeFlare for MCAD scheduling), and AI model governance (TrustyAI for explainability, fairness, and drift detection).

The platform is orchestrated by two central operators: the `rhods-operator` (also known as `kf-poc-rhods-operator`) which manages platform-level initialization and component lifecycle via `DSCInitialization` and `DataScienceCluster` custom resources, and the `odh-manifests` collection which provides the Kustomize-based deployment manifests consumed by the operator. RHOAI 2.6 follows the "RHOAI 2.x" deployment pattern, using OpenShift Routes with oauth-proxy sidecars for external ingress and authentication, OpenShift Service Mesh (Istio/Maistra) for KServe serverless serving, and OpenShift Serverless (Knative) for auto-scaling inference workloads.

The architecture spans multiple namespaces (`redhat-ods-operator`, `redhat-ods-applications`, `redhat-ods-monitoring`, `istio-system`, and user-created data science project namespaces) with NetworkPolicies controlling inter-namespace communication. All operator-to-API-server communication uses TLS 1.2+ with ServiceAccount tokens, while user-facing services are protected by OpenShift OAuth proxy sidecars with SubjectAccessReview-based authorization.

## Deployment Architecture

### Deployment Topology

The platform deploys across dedicated operator, application, and monitoring namespaces with strict NetworkPolicy boundaries:

- **redhat-ods-operator**: Contains the rhods-operator deployment (single replica)
- **redhat-ods-applications**: Contains all core platform services including dashboard, notebook controller, KServe, ModelMesh, DSPO, CodeFlare, KubeRay, and TrustyAI operators
- **redhat-ods-monitoring**: Contains Prometheus, Alertmanager, and Blackbox Exporter (Managed RHOAI only)
- **istio-system**: Contains Istio/Maistra Service Mesh control plane and ingress gateway; KServe Routes are created here
- **knative-serving**: Contains Knative Serving components for serverless auto-scaling
- **User namespaces**: Each data science project namespace contains user workloads (notebooks, pipelines, inference services, TrustyAI instances)

### Key Deployment Characteristics

| Characteristic | Value |
|---------------|-------|
| Operator deployment pattern | OLM-managed via Subscription/CSV |
| Manifest management | Kustomize with base/overlay pattern |
| Image references | SHA256 digest-pinned in params.env files |
| Platform modes | Self-Managed RHOAI, Managed RHOAI (OSD/ROSA addon), OpenDataHub (upstream) |
| Multi-tenancy | Namespace-scoped isolation with NetworkPolicies |
| Leader election | Enabled for all multi-replica operators |
| High availability | odh-model-controller runs 3 replicas with pod anti-affinity |
