## Purpose

**Short**: A containerized distribution of the Llama Stack server that packages all APIs, providers, and dependencies needed to run a multi-provider AI inference, evaluation, and agent orchestration platform within RHOAI.

**Detailed**: The rhds-llama-stack-distribution repository builds a production-ready container image that bundles Meta's Llama Stack server (forked at opendatahub-io/llama-stack v0.5.0+rhai0) with a comprehensive set of inference, vector storage, evaluation, safety, and agent providers. The distribution is designed to run as a stateless service pod in OpenShift, connecting to external inference backends (vLLM, AWS Bedrock, Google Vertex AI, Azure, WatsonX, OpenAI), vector databases (Milvus, pgvector, Qdrant, FAISS), and evaluation frameworks (TrustyAI LMEval, RAGAS, Garak).

The repository does not contain Llama Stack source code itself — it contains only the build infrastructure (Containerfile templates, build scripts, configuration), the distribution configuration (config.yaml defining APIs and providers), and CI/CD pipelines (GitHub Actions for upstream builds, Tekton/Konflux for RHOAI productization). The actual Llama Stack server is installed from the opendatahub-io fork during image build. The container image exposes an OpenAI-compatible REST API on port 8321/TCP and uses PostgreSQL as its persistent storage backend for key-value stores, inference logs, agent state, and file metadata.

For RHOAI productization, a separate Konflux build pipeline uses `Dockerfile.konflux` which expects a pre-built base image from the AIPCC (AI Platform Container Collection) build system, with wheel releases for architecture-specific builds (x86_64 and aarch64). This two-stage approach separates the open-source upstream image (built by GitHub Actions from `distribution/Containerfile`) from the productized RHOAI image (built by Konflux from `Dockerfile.konflux`).

## Architectural Analysis

The rhds-llama-stack-distribution is architecturally a **build-and-configuration wrapper** rather than a codebase in its own right. The repository contains zero lines of Llama Stack server code; instead, it provides the configuration file (`distribution/config.yaml`) that defines how Meta's Llama Stack framework should be assembled, and the build tooling (`distribution/build.py`, `Containerfile.in`) that generates a reproducible container image. This is an important architectural pattern: the distribution is a **composition** of upstream Llama Stack plus external Red Hat providers (TrustyAI LMEval, RAGAS, Garak, FMS) into a single deployable artifact.

The provider configuration uses an environment-variable-driven conditional activation pattern (`${env.VAR:+provider_id}`) that allows a single container image to serve multiple deployment profiles. Only providers whose activation environment variable is set will register at startup. This means the same image can run as a minimal vLLM-only inference server or as a full-featured platform with vector search, evaluation, safety, and multi-cloud inference — controlled entirely by the deployment's environment variables. The default configuration enables Milvus (inline), the meta-reference agent provider, TrustyAI LMEval, FMS safety, scoring providers, and tool runtimes, while all cloud inference backends and remote vector databases require explicit opt-in.

The two-tier build system is notable: the upstream GitHub Actions workflow builds from `distribution/Containerfile` (auto-generated by `build.py` from `Containerfile.in`), while the RHOAI productization uses `Dockerfile.konflux` which assumes a pre-built AIPCC base image containing RPMs, libraries, and a fromager-based wheel release system. The Konflux build does not run `pip install` directly — it installs pre-compiled wheel packages from an internal GitLab registry (`rhel-ai-private-index-auth`), ensuring binary reproducibility and CVE traceability for all dependencies. The `konflux/cpu-ubi9.conf` file pins the specific AIPCC base image version and wheel release identifiers (e.g., `3.4-EA1.1042+llama-stack-cpu-ubi9-x86_64`).

The storage architecture is centralized around PostgreSQL with two backend types: a key-value store (`kv_postgres`) used for metadata registry, agent state, vector IO persistence, and batch tracking; and a SQL store (`sql_postgres`) used for inference logging, agent responses, conversation history, and file metadata. All backends share the same PostgreSQL instance by default, differentiated by table name and namespace. This is a deliberate design choice for operational simplicity — a single PostgreSQL instance serves all persistence needs — but creates a single point of failure for the entire Llama Stack instance.

The observability integration is lightweight but well-designed. The entrypoint script conditionally wraps the Llama Stack server with `opentelemetry-instrument` when `OTEL_SERVICE_NAME` is set, enabling automatic trace injection for all HTTP requests and database calls. The image includes `opentelemetry-distro` and runs `opentelemetry-bootstrap` during build to discover and install additional OTel instrumentation packages for all installed libraries. OTLP HTTP export for traces and metrics is enabled by default; log export is intentionally omitted unless `OTEL_LOGS_EXPORTER=otlp` is explicitly set.
