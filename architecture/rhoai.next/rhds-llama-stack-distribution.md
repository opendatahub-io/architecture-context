## Purpose

**Short**: Builds and distributes a Red Hat-supported Llama Stack server container image with pre-configured inference, vector I/O, safety, evaluation, and tool runtime providers.

**Detailed**: The rhds-llama-stack-distribution repository is the downstream (Red Hat Data Services) fork of the Open Data Hub Llama Stack distribution. It produces a single container image that packages Meta's Llama Stack framework -- an open-source platform for building generative AI applications -- with a curated set of providers for inference (vLLM, AWS Bedrock, IBM Watsonx, Azure, Google Vertex AI, OpenAI, sentence-transformers), vector databases (Milvus, FAISS, pgvector, Qdrant), safety (TrustyAI FMS), evaluation (lmeval, RAGAS, Garak), tool runtimes (Brave Search, Tavily, RAG, MCP), file storage (local filesystem, S3), agents, scoring, batches, and dataset I/O.

The image runs the `llama stack run` command on startup, serving an HTTP API on port 8321 that is compatible with the OpenAI chat completions API. Providers are conditionally activated via environment variables at runtime, allowing the same image to target different inference backends and storage systems without rebuilding. The server uses PostgreSQL as its persistent storage backend for key-value stores and SQL tables (inference logs, agent state, file metadata, conversations). OpenTelemetry instrumentation is optionally enabled via the `OTEL_SERVICE_NAME` environment variable to export traces and metrics to an OTLP collector.

The repository contains no operator code, no CRDs, and no Kubernetes manifests. It is purely a container image build repository with build scripts (`distribution/build.py`), a configuration template (`distribution/config.yaml`), an entrypoint script, CI/CD pipelines (GitHub Actions and Tekton/Konflux), and test scripts. The upstream community Containerfile is auto-generated from a template (`Containerfile.in`) by the build script, while the RHOAI Konflux build (`Dockerfile.konflux`) uses AIPCC base images with wheel releases from an internal registry.

## Architectural Analysis

The rhds-llama-stack-distribution is architecturally a **thin packaging layer** around Meta's Llama Stack framework. It does not contain application logic beyond build tooling and configuration; all runtime behavior is defined by the upstream `llama-stack` Python package (from the opendatahub-io fork at version v0.5.0+rhai0) and its provider ecosystem. The repository's primary contribution is the `distribution/config.yaml` file, which defines which providers are enabled and how they are configured, plus the Dockerfiles that package everything into a deployable container image.

The provider architecture uses an **environment-variable-driven activation pattern** where providers are conditionally enabled at runtime. The config.yaml uses the syntax `${env.VAR:+value}` for provider IDs, which activates a provider only when the corresponding environment variable is set. This allows a single container image to serve dozens of different deployment topologies -- from a minimal setup with just vLLM inference to a full-featured deployment with multiple cloud inference backends, remote vector databases, evaluation pipelines, and tool runtimes. The trade-off is that the container image is large (it bundles all provider dependencies including PyTorch CPU, sentence-transformers, milvus-lite, and numerous Python packages) regardless of which providers are actually used at runtime.

The **dual-build strategy** is notable: the upstream community build (`distribution/Containerfile`) is auto-generated from a template by `distribution/build.py` and uses a public UBI9/Python 3.12 base image with direct pip/uv installs from PyPI. The downstream RHOAI build (`Dockerfile.konflux`) uses the AIPCC CPU base image with pre-built wheel releases from an internal GitLab package registry, providing better supply chain security and RHEL AI Python Package Index integration. The Konflux build also runs AIPCC tooling (`fromager-rpm-check.py`, `selftest.py`) for image validation. This dual approach means the community and product images share the same application config but have fundamentally different dependency resolution paths.

A significant security consideration is that the Llama Stack server **exposes its HTTP API on port 8321 without any authentication or authorization**. The `distribution/config.yaml` has no auth configuration, and the server itself does not enforce any access control. When deployed on RHOAI, authentication is expected to be provided by the platform's ingress layer (kube-rbac-proxy sidecars, Gateway API with auth enforcement). However, when run standalone (e.g., via `podman run`), the API is completely open. The egress surface is also broad -- the server can connect to numerous external services (cloud inference APIs, vector databases, search APIs, Kubeflow, S3) depending on configuration, each with its own credential management pattern (API keys, OAuth tokens, IAM credentials, username/password). Credential management is entirely via environment variables with no secrets rotation mechanism.

The **PostgreSQL dependency is mandatory** -- the storage configuration defines both `kv_postgres` and `sql_postgres` backends with no fallback to an in-memory or SQLite alternative. This means a running PostgreSQL instance is required for the Llama Stack server to start. The database stores agent state, inference logs, file metadata, conversations, prompts, and various provider-specific key-value data. The connection is over plaintext PostgreSQL wire protocol with username/password authentication and no TLS configuration visible in the config.
