## Purpose

**Short**: Pre-built container image providing a Llama Stack server with Red Hat-curated providers for inference, evaluation, safety, vector I/O, and agentic workflows.

**Detailed**: The rhds-llama-stack-distribution repository produces a container image that packages the [Llama Stack](https://github.com/llamastack/llama-stack) framework — Meta's open-source AI application server — with a Red Hat-specific distribution configuration. The image ships all Python dependencies pre-installed so that a Llama Stack server can be started immediately with `llama stack run`, exposing an OpenAI-compatible HTTP API on port 8321.

The distribution configuration (`distribution/config.yaml`) defines a comprehensive set of providers that are activated via environment variables. Inference providers include remote backends for vLLM, AWS Bedrock, Google Vertex AI, Azure OpenAI, IBM watsonx, and OpenAI, plus an inline sentence-transformers provider for embeddings. Vector I/O providers support Milvus (inline and remote), Faiss, PgVector, and Qdrant. Evaluation providers integrate with TrustyAI's LM-Eval, RAGAS, and Garak frameworks. Safety is handled by TrustyAI FMS. Agents, datasets, scoring, files, batches, and tool runtimes (including Brave Search, Tavily Search, RAG, and MCP) are all configured. PostgreSQL serves as the persistence backend for key-value and SQL storage.

The repository contains two build paths: (1) the upstream ODH build using `distribution/Containerfile` with direct pip/uv installs from PyPI and Git, and (2) the downstream RHOAI Konflux build using `Dockerfile.konflux` with AIPCC base images and pre-built wheel releases. The build tooling (`distribution/build.py`) auto-generates the Containerfile from a template by introspecting `llama stack list-deps` output.

## Architectural Analysis

The Llama Stack Distribution follows a **provider-based plugin architecture** where every API capability (inference, safety, evaluation, vector I/O, etc.) is backed by one or more configurable provider implementations. Providers are categorized as either `inline` (running in-process within the container) or `remote` (calling external services). This design creates a highly configurable but wide-surface-area container: a single image can talk to 15+ external services depending on which environment variables are set. The conditional activation pattern (`${env.VAR:+provider_id}`) means that providers with empty environment variables are silently disabled, keeping the default footprint small.

The **dual-build architecture** is a significant design decision. The upstream ODH build (`distribution/Containerfile`) installs everything from public PyPI and Git with version pins, producing a functionally identical image that can be built without Red Hat infrastructure. The downstream RHOAI Konflux build (`Dockerfile.konflux`) uses AIPCC base images and pre-built wheel releases, providing supply chain provenance and FIPS-compliant Python packaging. The `build.py` script bridges these by auto-generating the Containerfile from `llama stack list-deps`, ensuring dependency lists stay synchronized with the Llama Stack framework's requirements. The wheel release pinning in `konflux/cpu-ubi9.conf` (e.g., `3.4-EA1.1042+llama-stack-cpu-ubi9-x86_64`) provides exact reproducibility for the Konflux path even without traditional lock files.

The **persistence architecture** uses PostgreSQL as a unified backend for both key-value storage (`kv_postgres`) and relational data (`sql_postgres`). The KV store handles agent state, vector I/O metadata, dataset metadata, and the registry. The SQL store handles inference logs, agent responses, file metadata, and conversation history. This consolidation onto a single PostgreSQL instance simplifies deployment but creates a single point of failure for all stateful operations. The inline Milvus provider uses `milvus-lite` with a local SQLite-like file (`milvus.db`), providing zero-configuration vector search that is ephemeral unless the container's filesystem is persisted.

The **OpenTelemetry integration** in the entrypoint script wraps the `llama stack run` command with `opentelemetry-instrument` when `OTEL_SERVICE_NAME` is set. The instrumentation is bootstrapped during image build (`opentelemetry-bootstrap -a requirements | uv pip install --requirement -`), ensuring all supported Python libraries are auto-instrumented. Traces and metrics are exported via OTLP; logs export is intentionally disabled by default to avoid noise but can be re-enabled with `OTEL_LOGS_EXPORTER=otlp`.

A notable architectural risk is the **absence of application-level authentication**. The Llama Stack server exposes all APIs on port 8321 without any built-in auth mechanism. When deployed on RHOAI, the platform operator injects a kube-rbac-proxy sidecar to enforce authentication, but standalone or development deployments have no protection. The pre-downloaded `granite-embedding-125m-english` model baked into the image (via `hf download` in the Konflux Dockerfile) provides offline embedding capability but increases image size and couples the image to a specific embedding model version.
