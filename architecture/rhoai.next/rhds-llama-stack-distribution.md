## Purpose

**Short**: Builds and ships a containerized Llama Stack server distribution with pre-configured AI providers for inference, evaluation, safety, vector I/O, and agentic workflows in the Red Hat OpenShift AI platform.

**Detailed**: The Llama Stack Distribution repository produces a single container image that packages the [Llama Stack](https://github.com/llamastack/llama-stack) server — a unified API gateway for large language model (LLM) operations. The image bundles a curated set of inference backends (vLLM, AWS Bedrock, Google Vertex AI, Azure, WatsonX, OpenAI, sentence-transformers), vector databases (Milvus, FAISS, pgvector, Qdrant), evaluation providers (TrustyAI LMEval, RAGAS, Garak), safety shields (TrustyAI FMS), tool runtimes (Brave/Tavily search, RAG, MCP), file storage (local and S3), and agentic workflow support. All providers are environment-variable-driven, enabling dynamic activation at container startup without image rebuilding.

The repository does not contain the Llama Stack server code itself — it consumes a pinned version (`v0.5.0+rhai0`) from the [opendatahub-io/llama-stack](https://github.com/opendatahub-io/llama-stack) fork via pip install at image build time. The build tooling (`distribution/build.py`) uses the `llama stack list-deps` CLI to auto-generate a `Containerfile` with precise dependency pins from a `config.yaml` manifest. A separate Konflux build path (`Dockerfile.konflux`) exists for downstream RHOAI release builds using pre-built wheel bundles. The image exposes a single HTTP API on port 8321 implementing both Llama Stack native and OpenAI-compatible APIs.

## Architectural Analysis

The Llama Stack Distribution follows a **provider-plugin architecture** where the core server framework (llama-stack) defines abstract API interfaces (inference, vector_io, safety, eval, agents, etc.) and this distribution wires them to concrete provider implementations via `config.yaml`. This design enables a single container image to support dozens of backend combinations without code changes — provider activation is controlled entirely through environment variables using the conditional `${env.VAR:+value}` syntax in YAML. This is architecturally elegant but introduces operational complexity: misconfigured environment variables silently disable providers rather than failing explicitly.

The **dual build path** is the most significant architectural feature of this repository. The upstream build (`distribution/Containerfile`) installs dependencies via `pip` from public PyPI and GitHub, generating the Containerfile dynamically via `build.py`. The downstream Konflux build (`Dockerfile.konflux`) uses pre-built wheel bundles from a private GitLab registry (`WHEEL_RELEASE_*` in `konflux/cpu-ubi9.conf`) and a proprietary base image (`quay.io/aipcc/base-images/cpu:3.0-*`). This dual path means the upstream image is not identical to the RHOAI-shipped image — the Konflux build includes additional supply chain hardening (wheel integrity, base image scanning, RPM checks via `fromager-rpm-check.py`, library scanning via `scanlibs.py`, and self-tests via `selftest.py`).

The **PostgreSQL dependency** is hard-wired as the sole storage backend (both `kv_postgres` and `sql_postgres` types), making it a critical runtime dependency. The storage configuration supports separate stores for metadata, inference logs, conversations, prompts, agent state, file metadata, and batch operations — all pointing at the same PostgreSQL instance but using different tables/namespaces. This centralized storage design simplifies deployment but creates a single point of failure. The server performs async writes with configurable queue sizes (`max_write_queue_size: 10000`) and writer pools (`num_writers: 4`) for inference and agent response logging, indicating awareness of write throughput requirements.

The **observability integration** is optional but well-structured: when `OTEL_SERVICE_NAME` is set, the entrypoint script wraps the server process with `opentelemetry-instrument`, exporting traces and metrics via OTLP to a configurable collector. The `opentelemetry-bootstrap` command auto-discovers instrumentable libraries at build time, ensuring comprehensive coverage of httpx, fastapi, sqlalchemy, and other frameworks. Logs export is intentionally omitted by default, requiring explicit `OTEL_LOGS_EXPORTER=otlp` to enable.

A notable security consideration is that the Llama Stack server itself has **no authentication or authorization** — it relies entirely on the platform's ingress layer (kube-rbac-proxy, Gateway API, or external auth proxy) for access control. All API endpoints are unauthenticated at the application level. The numerous API keys and tokens in the configuration are for *outbound* calls to inference backends and tool APIs, not for protecting the server itself. This is consistent with the RHOAI platform pattern where authentication is enforced at the ingress layer, but it means a misconfigured deployment could expose the server without auth.
