## Purpose

**Short**: A containerized Llama Stack distribution that serves as a unified AI inference gateway, aggregating multiple LLM inference backends, vector stores, evaluation frameworks, and tool runtimes behind a single REST API.

**Detailed**: The Llama Stack Distribution is a container image packaging of Meta's [Llama Stack](https://github.com/llamastack/llama-stack) framework, customized for Red Hat OpenShift AI (RHOAI). It provides a single API server (port 8321) that aggregates multiple AI provider backends — including vLLM, AWS Bedrock, Google Vertex AI, Azure, OpenAI, and IBM watsonx — behind a unified OpenAI-compatible REST API. The distribution ships with a rich plugin architecture supporting inference, vector I/O (Milvus, FAISS, pgvector, Qdrant), safety evaluation (TrustyAI FMS), model evaluation (LM Eval, RAGAS, Garak), agents, scoring, file storage, and tool runtimes (web search, RAG, MCP).

The repository itself is a "distribution packaging" repo — it does not contain the Llama Stack server code itself (that lives in [opendatahub-io/llama-stack](https://github.com/opendatahub-io/llama-stack)). Instead, it defines the distribution configuration (`distribution/config.yaml`), the container build logic (`distribution/build.py` + `Containerfile.in` template), and a Konflux Dockerfile (`Dockerfile.konflux`) for RHOAI production builds. Providers are activated or deactivated via environment variables using a conditional `${env.VAR:+id}` syntax, allowing the same image to adapt to many deployment scenarios. The distribution uses PostgreSQL for persistent state storage (KV store and SQL) and supports OpenTelemetry instrumentation for observability.

Two container build paths exist: the upstream community build uses `distribution/Containerfile` (auto-generated from `Containerfile.in` by `build.py`) and builds from a UBI9 Python 3.12 base image with `uv pip install`; the downstream RHOAI Konflux build uses `Dockerfile.konflux` with a pre-built AIPCC base image and pre-compiled wheel releases for supply chain hardening.

## Architectural Analysis

The Llama Stack Distribution follows a **provider plugin architecture** where every API capability (inference, vector I/O, safety, eval, etc.) is backed by one or more swappable providers. Providers are classified as either `inline::` (code runs in-process) or `remote::` (delegates to an external service). This is a powerful pattern for multi-cloud AI deployments: the same container image can serve vLLM on-cluster inference, AWS Bedrock, Google Vertex AI, Azure OpenAI, IBM watsonx, or vanilla OpenAI — all simultaneously — with provider selection controlled entirely by environment variables at deployment time.

The conditional provider activation mechanism (`${env.VAR:+provider_id}`) is a Llama Stack v2 config feature that enables or disables providers based on whether an environment variable is set. For example, setting `VLLM_URL` activates the vLLM inference provider, while leaving it unset silently disables it. This eliminates the need for multiple config files or Helm value overrides for different deployment scenarios. The same image handles everything from a minimal single-provider setup to a full multi-cloud aggregation.

The repository has a **dual build path** that reflects the ODH/RHOAI relationship. The upstream community build (`distribution/Containerfile`) is auto-generated by `build.py`, which calls `llama stack list-deps` to resolve the complete dependency tree from `config.yaml` and writes concrete `uv pip install` commands into the Containerfile. This build path installs llama-stack from the opendatahub-io GitHub fork with a `+rhai` version suffix (e.g., `v0.5.0+rhai0`). The downstream Konflux build (`Dockerfile.konflux`) takes a fundamentally different approach: it starts from a pre-built AIPCC base image that already contains common ML dependencies, then installs a pre-compiled wheel release from a private GitLab registry (`install-wheel-release.sh`). This provides supply chain hardening — dependencies are pre-resolved and pre-compiled rather than resolved at build time from PyPI.

A notable architectural decision is the embedding of a **default embedding model** (`ibm-granite/granite-embedding-125m-english`) directly into the container image at build time via `hf download`. This enables immediate RAG functionality without requiring a separate embedding model deployment, though the `sentence-transformers` inline provider must be explicitly enabled via the `ENABLE_SENTENCE_TRANSFORMERS` environment variable. The distribution also registers this model in the config with configurable provider and model ID, allowing users to switch to a vLLM-served embedding model for production workloads.

The storage architecture uses **PostgreSQL as the single persistence backend** for all stateful data — KV store (via `kv_postgres`), SQL tables for inference logs and agent responses (via `sql_postgres`), and file metadata. The KV and SQL backends share the same PostgreSQL instance but use separate access patterns: KV for configuration and registry metadata, SQL for write-heavy inference and conversation logs with configurable write queue sizes (`max_write_queue_size: 10000`, `num_writers: 4`). This design simplifies deployment (single database dependency) while supporting high-throughput inference logging via async write queues.
