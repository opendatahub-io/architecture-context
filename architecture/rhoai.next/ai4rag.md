## Purpose

**Short**: ai4rag is an optimization engine that automatically finds optimal hyperparameters for Retrieval-Augmented Generation (RAG) templates across LLM providers, embedding models, and vector stores.

**Detailed**: ai4rag is a provider-agnostic RAG optimization library developed by IBM. It accepts RAG template definitions along with a search space of tunable hyperparameters (chunking strategy, chunk size, overlap, embedding models, foundation models, retrieval methods, number of retrieved chunks, search modes) and uses Generalized Additive Model (GAM)-based hyperparameter optimization to find the configuration that maximizes evaluation metrics such as faithfulness, answer correctness, and context correctness.

The library is designed to work primarily with Llama Stack as its backend for embeddings, vector stores, and text generation, but also supports OpenAI-compatible APIs and ChromaDB as alternative providers. It includes a models pre-selection phase (MPS) that efficiently narrows down large model pools before running the full optimization, and uses the Unitxt framework for evaluation metrics with confidence intervals.

ai4rag is a pure Python library distributed via PyPI — it has no Dockerfiles, no Kubernetes manifests, and no deployment infrastructure. It is consumed as a dependency by other services and pipelines, including Kubeflow Pipelines components via the `KFPEventHandler`.

## Architectural Analysis

ai4rag follows a clean layered architecture with well-defined abstractions at each level. The `rag/` module provides base classes for every RAG component — `BaseFoundationModel`, `BaseEmbeddingModel`, `BaseVectorStore`, `BaseChunker`, and `BaseRAGTemplate` — allowing users to plug in custom implementations. The library ships with two backend families: Llama Stack (primary, via `llama-stack-client`) and OpenAI-compatible (alternative, via `openai` SDK). ChromaDB serves as the in-memory vector store for local development and the models pre-selection phase, while Llama Stack vector stores (Milvus, Qdrant, etc. via `ls_<provider_id>` convention) handle production workloads.

The optimization core uses a two-phase approach. First, the `ModelsPreSelector` (MPS) runs a lightweight evaluation on a sample of benchmark data to narrow down foundation and embedding model candidates — this avoids wasting expensive full-optimization iterations on poorly-performing models. Second, the `GAMOptimizer` trains a `LinearGAM` on evaluated parameter combinations to predict scores for unexplored nodes, then selects the highest-predicted configurations for evaluation. This surrogate-model approach is more sample-efficient than pure random search, though the library also provides `RandomOptimizer` as a baseline. The search space is fully combinatorial with constraint rules that prune invalid combinations (e.g., chunk size must exceed 2× overlap, chunk token count must fit within the embedding model's context length, hybrid search parameters must be consistent with search mode).

The event handler abstraction (`BaseEventHandler`) enables flexible integration patterns. The `LocalEventHandler` writes JSON artifacts to disk for local development. The `KFPEventHandler` accumulates results in memory for Kubeflow Pipelines integration, allowing pipeline components to access experiment results programmatically. This pattern makes ai4rag deployable as a library call within larger ML pipelines without requiring its own service infrastructure. The typed payloads (`PatternPayload`, `EvaluationRecord`) provide a stable schema (version "1.0") for downstream consumers.

A notable design decision is the support for hybrid search modes (vector + keyword) with configurable ranker strategies (RRF, weighted, normalized). This is only available with Llama Stack vector stores — ChromaDB is restricted to vector-only search. The search space constraint system (`_rule_search_mode_ranker_consistency`, `_rule_ranker_k_for_rrf_only`, `_rule_ranker_alpha_for_weighted_only`) ensures that incompatible parameter combinations are pruned before optimization begins. The library also includes auto-detection of embedding dimensions and context lengths via binary search probing, which reduces configuration burden but requires the embedding service to be reachable at model construction time.

The prompt template system is model-family-aware, providing optimized system messages, user message templates, and context wrapping templates for Granite, Llama, Mistral, and OpenAI model families. Templates are matched by model ID with fuzzy fallback to model family name, ensuring reasonable defaults even for new model variants. The `RAGPromptTemplateString` descriptor validates that templates contain exactly the required placeholders (`{question}`, `{reference_documents}`, `{document}`), preventing runtime formatting errors.
