## Purpose

**Short**: ai4rag is a provider-agnostic optimization engine that automatically discovers optimal hyperparameters for Retrieval-Augmented Generation (RAG) pipelines.

**Detailed**: ai4rag (AI for RAG) is a Python library that accepts RAG template definitions and a search space of hyperparameters, then uses optimization algorithms (primarily Generalized Additive Models) to find the best-performing RAG configuration. It orchestrates the full RAG lifecycle — document chunking, embedding, vector store indexing, retrieval, foundation model generation, and evaluation — across a combinatorial parameter space, returning an initialized RAG Pattern with optimal settings.

The library is designed to be LLM and vector database provider-agnostic. Out of the box it integrates with OGX (formerly Llama Stack) for embeddings, vector storage, and text generation, and also supports ChromaDB as an in-memory vector store. Users can extend the system by implementing abstract base classes for foundation models, embedding models, vector stores, chunkers, evaluators, and RAG templates. The evaluation pipeline uses IBM's Unitxt framework to measure answer correctness, faithfulness, and context correctness with confidence intervals.

ai4rag is not a deployed service or Kubernetes operator. It is a pure Python library (distributed via PyPI) consumed by other services or Kubeflow Pipeline components in the RHOAI platform. It provides a `KFPEventHandler` for integration with Kubeflow Pipelines and a `LocalEventHandler` for local development. The library does not expose network endpoints, CRDs, or Kubernetes resources — it is imported and invoked programmatically by platform components that need automated RAG optimization.

## Architectural Analysis

ai4rag follows a clean layered architecture with well-defined abstractions at each tier. The core abstractions — `BaseFoundationModel`, `BaseEmbeddingModel`, `BaseVectorStore`, `BaseChunker`, `BaseRAGTemplate`, `BaseEvaluator`, and `BaseEventHandler` — are all abstract base classes with concrete implementations provided for OGX and ChromaDB. This design enables provider-agnostic RAG optimization: users can swap in custom foundation models, embedding services, or vector databases by implementing the corresponding ABC.

The optimization layer uses a Bayesian-style approach via `GAMOptimizer`, which trains a Generalized Additive Model (from pygam) on the scores of already-evaluated configurations to predict which unexplored parameter combination will score highest. The search loop consists of an initial random exploration phase (`n_random_nodes` random configurations), followed by iterative GAM-guided exploration where the model predicts scores for remaining combinations and selects the most promising ones. This is a pragmatic alternative to Gaussian Process-based Bayesian optimization — GAMs handle categorical parameters natively through label encoding, and the `evals_per_trial=1` setting means each iteration evaluates a single configuration, making it tractable for expensive RAG evaluations. A `RandomOptimizer` fallback is also provided.

A notable design pattern is the Models Pre-Selector (MPS), which runs before the main optimization loop when the search space contains many models. MPS performs a quick evaluation of all foundation model / embedding model pairs using a small data sample and in-memory ChromaDB, then selects the top-N performers for the full experiment. This significantly reduces the combinatorial explosion that would otherwise occur when multiple models are crossed with chunking/retrieval parameters. The warm-start feature (`known_observations`) allows resuming experiments by pre-loading previous evaluation results into the optimizer, which is useful in Kubeflow Pipeline scenarios where experiments may be interrupted and restarted.

The event handler abstraction (`BaseEventHandler`) is the integration seam between ai4rag and the platform. `LocalEventHandler` writes pattern results to disk as JSON files, while `KFPEventHandler` aggregates results in memory for post-experiment consumption by Kubeflow Pipeline components. This pattern allows ai4rag to remain deployment-agnostic while supporting both local development and production pipeline orchestration. The typed payload structures (`PatternPayload`, `EvaluationRecord`, etc.) provide a well-defined contract for event consumers.

The search space system deserves mention for its constraint-based pruning. `AI4RAGSearchSpace` defines rules that automatically filter invalid parameter combinations — for example, ensuring chunk_size > 2 * chunk_overlap, aligning window_size with the retrieval method, and enforcing that hybrid search parameters (ranker_strategy, ranker_k, ranker_alpha) are only set when search_mode is "hybrid". This prevents wasted evaluations on configurations that would fail at runtime. The search space also adapts based on vector store type: ChromaDB configurations exclude hybrid search parameters since ChromaDB doesn't support them.
