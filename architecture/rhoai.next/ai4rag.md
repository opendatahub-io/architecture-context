## Purpose

**Short**: ai4rag is an optimization engine that automatically finds optimal hyperparameter configurations for Retrieval-Augmented Generation (RAG) pipelines.

**Detailed**: ai4rag accepts RAG templates and a search space definition, then uses hyperparameter optimization (HPO) to find the best parameter configuration — called a "RAG Pattern" — for a given set of documents and benchmark questions. The engine is provider-agnostic: users can supply their own foundation model, embedding model, and vector store implementations, though out of the box it integrates with OGX (formerly Llama Stack) for model inference, embeddings, and vector storage.

The optimization pipeline consists of several stages: (1) model pre-selection, which narrows down candidate foundation and embedding models using a small benchmark sample; (2) search space construction with constraint rules that prune invalid parameter combinations; (3) hyperparameter optimization using either a Generalized Additive Model (GAM)-based optimizer or random search; and (4) evaluation using the Unitxt framework with metrics including faithfulness, answer correctness, and context correctness. Each iteration builds a complete RAG pipeline — chunking documents, embedding them into a vector store, retrieving relevant chunks for benchmark questions, generating answers via an LLM, and scoring the results.

ai4rag is designed as a library consumed by other services (e.g., KFP pipeline components via `KFPEventHandler`) rather than as a standalone deployed service. It has no Dockerfiles, no Kubernetes manifests, and exposes no network endpoints directly. Its architectural significance in the RHOAI platform lies in being the core optimization engine behind automated RAG pattern generation workflows.

## Architectural Analysis

ai4rag follows a clean provider-agnostic architecture with abstract base classes defining the contract for every pluggable component: `BaseFoundationModel`, `BaseEmbeddingModel`, `BaseVectorStore`, `BaseChunker`, `BaseEvaluator`, and `BaseRAGTemplate`. The only concrete implementations shipped are OGX-backed (for production use) and ChromaDB-backed (for local development and the model pre-selection phase). This separation allows consumers to plug in alternative providers without modifying the core optimization logic.

The optimization engine uses a two-phase approach. First, `ModelsPreSelector` evaluates all candidate foundation-model × embedding-model pairs on a small benchmark sample using ChromaDB in-memory stores, ranking them by mean score on the chosen metric. This narrows the search space before the expensive full optimization. Second, the `GAMOptimizer` (or `RandomOptimizer`) explores the remaining parameter space — chunk size, chunk overlap, retrieval method, number of chunks, search mode (vector vs. hybrid), and ranker configuration — using a Generalized Additive Model trained on observed scores to predict and prioritize promising configurations. The search space is constrained by seven built-in rules that prune invalid combinations (e.g., chunk size must exceed 2× chunk overlap, window retrieval requires window_size > 0, embedding context length must accommodate chunk size, hybrid search parameters must be internally consistent).

A notable design decision is the `EventHandler` abstraction, which decouples the experiment engine from its execution context. `LocalEventHandler` writes pattern results as JSON files to disk, while `KFPEventHandler` aggregates results in memory for Kubeflow Pipeline components. This makes ai4rag usable both in local development and in production KFP-based workflows without code changes.

The library has no container image, no Kubernetes manifests, and no network endpoints — it is purely consumed as a Python package. This means FIPS compliance, container hardening, and deployment security are entirely the responsibility of the consuming service (e.g., a KFP component container). The `uv.lock` file (658 KB) provides comprehensive Python dependency pinning, but the absence of Konflux Dockerfiles means there is no RPM-level or artifact-level hermeticity in this repository.

The prompt engineering layer is model-family-aware: `model_props.py` maintains separate system messages, user message templates, and context templates for Granite, Llama, Mistral, and OpenAI model families, with fallback heuristics based on model name substrings. This ensures that RAG patterns generated by the optimizer use prompt formats optimized for each model's instruction-following capabilities.
