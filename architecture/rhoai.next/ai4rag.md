## Purpose

**Short**: ai4rag is a provider-agnostic RAG (Retrieval-Augmented Generation) optimization engine that accepts RAG templates and search space definitions, then returns initialized RAG templates with optimal hyperparameter values through automated experimentation.

**Detailed**: ai4rag is a Python library that automates the process of finding optimal configurations for RAG pipelines. Given a set of documents, benchmark questions with ground-truth answers, and a search space of hyperparameters (chunking methods, chunk sizes, embedding models, foundation models, retrieval strategies, etc.), ai4rag runs systematic experiments using hyperparameter optimization (HPO) to identify the best-performing RAG pattern for a given evaluation metric.

The library's core workflow operates as follows: (1) optionally pre-select the most promising foundation and embedding models from a larger pool using a Models Pre-Selector (MPS), (2) define a search space of RAG hyperparameters with constraint rules, (3) run a GAM-based (Generalized Additive Models) or random optimizer that iteratively evaluates RAG configurations by chunking documents, embedding them into a vector store, retrieving relevant chunks for benchmark questions, generating answers via a foundation model, and evaluating the quality using Unitxt metrics (faithfulness, answer correctness, context correctness).

ai4rag is designed to be provider-agnostic through abstract base classes for foundation models, embedding models, and vector stores. The primary integration is with OGX (formerly Llama Stack) for model inference, embeddings, and vector storage, with ChromaDB as an alternative local vector store. The library is consumed as a Python dependency by other components in the RHOAI platform, particularly for automated RAG pipeline optimization in Kubeflow Pipelines (KFP) workflows.

## Architectural Analysis

ai4rag follows a clean layered architecture with well-defined abstractions at each tier. The library separates concerns into four distinct layers: the experiment orchestration layer (`core/experiment/`), the hyperparameter optimization layer (`core/hpo/`), the RAG pipeline layer (`rag/`), and the evaluation layer (`evaluator/`). Each layer communicates through clearly defined interfaces using abstract base classes, enabling provider-agnostic design.

The provider abstraction pattern is one of the most architecturally significant decisions. `BaseFoundationModel`, `BaseEmbeddingModel`, `BaseVectorStore`, `BaseEvaluator`, `BaseChunker`, and `BaseRAGTemplate` all define abstract interfaces that concrete implementations extend. Currently, OGX is the sole remote provider implementation, with ChromaDB available as a local alternative for vector storage. This design explicitly supports future provider integrations — the README and code comments indicate this extensibility is intentional. The `BaseEventHandler` abstraction similarly decouples result reporting from experiment execution, with `LocalEventHandler` for local development and `KFPEventHandler` for Kubeflow Pipelines integration.

The search space and constraint system is notably sophisticated for a library of this scope. The `AI4RAGSearchSpace` class composes a combinatorial search space from `Parameter` objects (supporting Categorical, Integer, Real, and Boolean types) and applies a chain of constraint rules that prune invalid combinations before optimization begins. Rules like `_rule_chunk_size_bigger_than_chunk_overlap`, `_rule_chunk_size_within_embedding_context_length`, and `_rule_search_mode_ranker_consistency` encode domain knowledge about valid RAG configurations directly into the search space definition, preventing the optimizer from wasting evaluations on invalid combinations. The hybrid search support (with RRF, weighted, and normalized ranker strategies) adds considerable complexity to the search space but reflects real production requirements for advanced retrieval.

The Models Pre-Selector (MPS) is a cost-optimization mechanism that runs before the main HPO loop. When the search space contains more models than the configured thresholds (default: >3 foundation models or >2 embedding models), MPS evaluates all model pairs on a small sample of benchmark data using ChromaDB (in-memory) and selects the top performers. This narrows the search space before the computationally expensive GAM optimization begins. The GAM optimizer itself uses scikit-learn's LabelEncoder to transform categorical parameters into numerical features, trains a `LinearGAM` on already-evaluated configurations, and predicts scores for remaining unevaluated combinations to guide the search toward promising regions.

One architectural risk is the tight coupling between the experiment orchestrator and the OGX client for search space auto-discovery (`prepare_search_space_with_ogx`). While the core experiment engine accepts any `BaseFoundationModel` and `BaseEmbeddingModel` implementations, the auto-discovery pathway is OGX-specific and validates model availability by making actual API calls during initialization. The `OGXEmbeddingModel` constructor also makes live API calls to auto-detect embedding dimensions and context lengths via binary search, which creates a runtime dependency on server availability at object construction time — not just at experiment execution time. This could be problematic in environments where model metadata is available but the inference endpoint is not yet ready.

The concurrent RAG query execution in `query_rag()` uses `ThreadPoolExecutor` with a configurable thread limit (default: 10), which is appropriate for I/O-bound API calls to the OGX server. However, there is no retry logic, rate limiting, or backoff mechanism — a transient failure in any single API call during the parallel query phase raises a `GenerationError` that fails the entire iteration. The exception handling hierarchy (`AI4RAGError` → `IndexingError`/`GenerationError`/`EvaluationError`) is well-structured but the GAM optimizer treats all failures uniformly by recording `None` scores, which means transient errors are indistinguishable from genuine model failures in the optimization results.
