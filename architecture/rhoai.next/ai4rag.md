## Purpose

**Short**: An optimization engine that automatically finds optimal hyperparameter configurations for Retrieval-Augmented Generation (RAG) pipelines.

**Detailed**: ai4rag is a provider-agnostic Python library that accepts RAG Templates (combinations of chunking, embedding, retrieval, and generation strategies) along with a search space definition, then uses hyperparameter optimization (HPO) algorithms to discover optimal parameter values — producing a tuned RAG Pattern. The library is designed to work primarily with Llama Stack as its backend for foundation models, embedding models, and vector stores, but also supports OpenAI-compatible APIs and ChromaDB as alternative backends.

The optimization engine implements two HPO algorithms: a Generalized Additive Model (GAM)-based optimizer that uses surrogate modeling to guide exploration, and a random search baseline. A Models Pre-Selection (MPS) phase can optionally narrow the model search space before the main optimization loop. Evaluation is performed using the Unitxt framework, which provides metrics for answer correctness, faithfulness, and context correctness with confidence intervals.

ai4rag is not deployed as a standalone service or operator. It is consumed as a Python library by data scientists and ML engineers who import it into their own notebooks or pipelines. It includes a Kubeflow Pipelines (KFP) event handler for integration with ML pipeline orchestration systems, enabling experiment results to be streamed as pipeline metadata.

## Architectural Analysis

ai4rag follows a clean layered architecture with four major subsystems — experiment orchestration, HPO algorithms, RAG pipeline components, and search space management — each with well-defined interfaces. The design is heavily based on abstract base classes and generic types, creating a plugin-like architecture where new model providers, vector stores, chunking strategies, and evaluation metrics can be added by implementing the appropriate base class. This extensibility is the project's primary architectural strength.

The RAG pipeline itself follows a Template Method pattern: `BaseRAGTemplate` defines the three-stage workflow (index → retrieve → generate), while `SimpleRAG` provides the concrete implementation. Each stage is composed of independent components (chunker, embedding model, retriever, vector store, foundation model) that are assembled by the experiment orchestrator. The factory pattern in `get_vector_store()` and the strategy pattern in search mode selection (vector vs. hybrid) further enable runtime flexibility. Notably, the foundation model implementations abstract over both Llama Stack and OpenAI APIs by mapping both to a common `chat(messages) -> choices` interface, with Pydantic-validated parameter objects for type safety.

The HPO layer is particularly well-designed. The GAM optimizer implements a two-phase approach: random initialization followed by surrogate model-guided exploration using `pygam.LinearGAM`. The search space is defined as a Cartesian product of Parameter objects with configurable types (Boolean, Integer, Real, Categorical), and constraint rules prune invalid combinations before optimization begins. This rule-based filtering (e.g., chunk_size must exceed chunk_overlap, hybrid search parameters only apply when search_mode is "hybrid") prevents wasted evaluations. The warm-start capability allows the GAM optimizer to incorporate previously known observations, which is valuable for resuming interrupted experiments or leveraging prior knowledge.

One notable design consideration is the Models Pre-Selection (MPS) phase, which addresses the combinatorial explosion when many foundation and embedding models are available. MPS evaluates all model pairs on a benchmark sample using fixed retrieval/chunking parameters, then selects the top-performing models to include in the main search space. This reduces the search space exponentially at the cost of potentially missing model-specific parameter interactions. The MPS implementation uses ChromaDB (in-memory) for consistency regardless of the main experiment's vector store choice, and includes retry logic for embedding operations with graceful degradation on individual model failures.

The event handler abstraction (`BaseEventHandler`) enables deployment flexibility. The `LocalEventHandler` writes JSON results to the filesystem for development, while `KFPEventHandler` accumulates results in memory for Kubeflow Pipelines metadata integration. The typed payload structures (`PatternPayload`, `EvaluationRecord`) use TypedDicts to ensure consistent event schemas. This design enables future event handlers for other pipeline orchestrators without modifying the core experiment logic. The project lacks any deployment manifests, Dockerfiles, or Kubernetes resources — it is purely a pip-installable Python library consumed by other systems.
