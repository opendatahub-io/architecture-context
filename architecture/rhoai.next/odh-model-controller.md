## Purpose

**Short**: A Kubernetes operator that extends KServe model serving with OpenShift ingress integration, NVIDIA NIM account management, Gateway API authentication, and observability infrastructure for RHOAI.

**Detailed**: The odh-model-controller is a multi-faceted Kubernetes operator that sits alongside KServe to provide platform-specific capabilities for Red Hat OpenShift AI model serving. It watches KServe InferenceService, LLMInferenceService, ServingRuntime, and InferenceGraph custom resources, and dynamically creates the supporting infrastructure that KServe itself does not manage: OpenShift Routes for external access, Kuadrant AuthPolicies for authentication, Istio EnvoyFilters for TLS termination, KEDA TriggerAuthentications for autoscaling, Prometheus ServiceMonitors for observability, and NetworkPolicies for network isolation.

The controller also manages NVIDIA NIM (NVIDIA Inference Microservices) Account resources, which orchestrate API key validation against the NVIDIA NGC registry, runtime template provisioning via OpenShift Templates, and container image pull secret creation for the nvcr.io registry. This enables seamless deployment of NVIDIA-optimized inference runtimes within the RHOAI model serving stack.

Additionally, the repository includes a standalone Model Serving API server (`model-serving-api`) that provides a Gateway discovery REST API for the RHOAI dashboard, allowing users to discover and select Gateway API listeners available for their namespaces with proper RBAC enforcement.

## Architectural Analysis

The odh-model-controller follows a pattern common in the RHOAI ecosystem: it acts as a "sidecar operator" to KServe, extending the upstream project's capabilities with platform-specific infrastructure. Rather than forking KServe, this controller watches the same CRDs and creates supplementary resources — a clean separation that enables upstream KServe upgrades while maintaining platform features. This is architecturally similar to how the rhods-operator extends the opendatahub-operator.

The controller's reconciler design uses a composable sub-resource pattern via the `SubResourceReconciler` interface. The `KserveRawInferenceServiceReconciler` orchestrates six sub-reconcilers (Route, ClusterRoleBinding, MetricsService, MetricsServiceMonitor, MetricsDashboard, KEDA), each implementing `Reconcile()`, `Delete()`, and `Cleanup()`. This design enables per-resource lifecycle management and clean namespace cleanup when the last InferenceService is deleted. The delta processor pattern (`DeltaProcessor.ComputeDelta()`) provides consistent create/update/delete semantics across all resource types using pluggable comparators.

The LLM serving path (LLMInferenceServiceReconciler + GatewayReconciler) represents the RHOAI 3.x architecture: Gateway API with Kuadrant AuthPolicies for authentication, Istio EnvoyFilters for TLS termination at the Authorino layer, and spec merging from LLMInferenceServiceConfig base references. The controller conditionally checks for CRD availability (Gateway, AuthPolicy, EnvoyFilter) at startup and only registers watches for available APIs — a pattern that enables graceful degradation when optional components are not installed. The GatewayReconciler creates EnvoyFilter resources as a temporary workaround for Kuadrant issue #1531, noting this should eventually be handled by Kuadrant directly.

The NIM integration is notable for its air-gapped mode awareness. The controller reads `spec.components.kserve.nim.airGapped` from the DataScienceCluster and short-circuits all NVIDIA API calls when enabled, relying instead on pre-provisioned resources. The NIM Account CRD enforces a singleton-per-namespace pattern via validating webhook, and uses Server-Side Apply with a dedicated field manager (`nim-account-controller`) for conflict-free resource management. The reconciliation pipeline (Validation → ConfigMap → Template → PullSecret) includes configurable refresh rates and force-validation annotations, making it operationally manageable.

The model-serving-api server component demonstrates security-conscious design: FIPS-compliant cipher suites (ECDHE with AES-GCM only), TLS 1.2 minimum, certificate hot-reloading via fsnotify with atomic pointer swap, security headers (HSTS, X-Frame-Options), and user-impersonated RBAC checks via SelfSubjectAccessReview before returning Gateway resources. The server's Gateway discovery logic implements the full Gateway API namespace allowance model (All, Same, Selector) for filtering listeners per-namespace.

The Ray TLS subsystem spans three components: the ServingRuntimeReconciler creates a self-signed CA (RSA 4096, 5-year validity), the PodReconciler generates per-pod certificates with IP-based SANs when pods are assigned IPs, and the Pod mutating webhook injects an init container (`ray-tls-generator`) that waits for certificates to be available before starting the Ray worker. This multi-component coordination uses shared Secrets (`ray-ca-tls`, `ray-tls`) with optimistic locking and retry logic for concurrent pod updates.
