## Purpose

**Short**: Kubernetes operator that automates model serving infrastructure management for KServe InferenceServices, LLM inference, NVIDIA NIM integration, and Gateway API auth on Red Hat OpenShift AI.

**Detailed**: The odh-model-controller is a controller-runtime based Kubernetes operator that eliminates manual configuration steps when deploying ML models on the RHOAI platform. It watches KServe InferenceService, LLMInferenceService, ServingRuntime, and NIM Account custom resources and automatically provisions the supporting infrastructure: OpenShift Routes for model endpoints, CA certificate aggregation for storage backends, Prometheus ServiceMonitors for observability, KEDA TriggerAuthentications for autoscaling, Kuadrant AuthPolicies for Gateway API authentication, and Istio EnvoyFilters for Authorino TLS bootstrap.

The project produces two deployable components. The **operator** (odh-model-controller) runs as a Deployment with leader election and manages the full lifecycle of model serving resources across namespaces. The **model-serving-api** is a separate HTTPS API server that provides a gateway discovery endpoint, allowing the RHOAI dashboard to list available Gateway API gateways with per-user RBAC authorization via SelfSubjectAccessReview. Both components are FIPS-compliant (built with `GOEXPERIMENT=strictfipsruntime`) and deploy into the platform operator namespace.

The controller also manages NVIDIA NIM integration, including API key validation against NGC APIs, runtime template provisioning via OpenShift Templates, and Docker pull secret generation for nvcr.io — with full support for air-gapped deployments.

## Architectural Analysis

The odh-model-controller exhibits a **layered reconciliation architecture** with three distinct functional domains sharing a single operator binary: (1) traditional InferenceService lifecycle management (Routes, monitoring, KEDA), (2) LLM-specific Gateway API auth management (AuthPolicies, EnvoyFilters), and (3) NVIDIA NIM integration (NGC APIs, templates, pull secrets). Each domain uses the same delta-processing pattern: compute desired state, fetch existing state, compare with a resource-specific comparator, then create/update/delete as needed. This pattern is implemented via `DeltaProcessor` (internal/controller/processors/deltaProcessor.go) and applied consistently across all resource types.

The **GatewayReconciler** and **LLMInferenceServiceReconciler** form a complementary pair for RHOAI 3.x auth architecture. The GatewayReconciler watches Gateway resources and creates gateway-level AuthPolicies and EnvoyFilters (for Authorino TLS bootstrap). The LLMInferenceServiceReconciler watches LLMInferenceService resources and creates per-HTTPRoute AuthPolicies. This separation means auth configuration operates at two levels: platform-wide gateway auth (managed by GatewayReconciler) and per-model route auth (managed by LLMInferenceServiceReconciler). Both controllers use Kuadrant AuthPolicy CRDs, with the GatewayReconciler supporting custom CEL objective expressions via gateway annotations. The design carefully tracks resource ownership — only deleting resources it created (checked via `IsManagedByOdhController`), and supporting "unmanaged" mode where users can take control of auth resources.

The **model-serving-api** is architecturally notable as a separate binary within the same repository, deployed as its own Deployment with its own ServiceAccount and RBAC. It implements a security pattern where the user's Bearer token is forwarded to the Kubernetes API via SelfSubjectAccessReview, ensuring that gateway listing respects per-user RBAC. The TLS configuration in `server/common/certreloader.go` is explicitly FIPS-compliant with a restricted cipher suite (ECDHE+AES-GCM only, P-256/P-384 curves) and dynamic certificate reloading with file watch and debounce. This is one of the few components in the platform that implements explicit, auditable FIPS crypto configuration rather than relying solely on build-time flags.

The **NIM integration** demonstrates a well-designed air-gapped pattern. When `spec.components.kserve.nim.airGapped` is true in the DataScienceCluster, the controller skips NGC API calls, omits NGC_API_KEY injection, and removes image pull secrets from the generated ServingRuntime — allowing operators to pre-populate images in a disconnected registry. The AccountReconciler uses a handler chain pattern (validation → configmap → template → pull secret) with periodic re-validation at configurable intervals (default 24h).

A notable risk area is the **Ray TLS certificate management**: the PodReconciler generates self-signed certificates with 3-year validity and no auto-rotation, using RSA-4096 keys stored in Kubernetes Secrets. The CA certificate has 5-year validity. While functional, this creates a long-lived credential that could become a security concern in production. Certificate expiration checking exists (`isCertExpired` in cert.go) but only at reconciliation time, not proactively.
