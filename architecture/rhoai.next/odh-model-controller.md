## Purpose

**Short**: Kubernetes operator and API server that automates model serving infrastructure lifecycle for KServe InferenceServices, LLMInferenceServices, and NVIDIA NIM on Red Hat OpenShift AI.

**Detailed**: The odh-model-controller is a controller-runtime based Kubernetes operator that manages the ancillary resources required when deploying ML models via KServe on the OpenShift AI platform. It watches InferenceService, ServingRuntime, LLMInferenceService, InferenceGraph, and NIM Account custom resources, then reconciles the supporting infrastructure — OpenShift Routes for external access, Prometheus ServiceMonitors for observability, ClusterRoleBindings for auth delegation, KEDA TriggerAuthentications for autoscaling, and EnvoyFilters plus Kuadrant AuthPolicies for Gateway API authentication.

The controller also provides a standalone model-serving-api REST server that implements gateway discovery with per-user RBAC enforcement, enabling the dashboard to present only the Gateways a user is authorized to target when creating LLMInferenceServices. Additionally, it manages NVIDIA NIM integration through the Account CRD, which validates NGC API keys, fetches available model catalogs, and generates ServingRuntime templates and pull secrets for NIM container images.

The repository ships pre-built ServingRuntime templates for major inference runtimes (vLLM CUDA/ROCm/Gaudi/CPU/Spyre, OpenVINO Model Server, MLServer) that the platform operator injects into user namespaces, enabling one-click model deployment from the RHOAI dashboard.

## Architectural Analysis

The odh-model-controller occupies a critical position in the RHOAI model serving stack as the "glue" operator between KServe's core inference infrastructure and the RHOAI platform's operational requirements. While KServe manages the core model serving lifecycle (predictor pods, transformers, explainers), odh-model-controller layers on platform-specific concerns: OpenShift Routes for external access, Prometheus monitoring for observability, Kuadrant AuthPolicies for authentication, and KEDA TriggerAuthentications for autoscaling. This separation of concerns allows KServe to remain upstream-compatible while enabling RHOAI-specific integration.

The controller uses a multi-level reconciliation pattern where top-level controllers (InferenceServiceReconciler, LLMInferenceServiceReconciler) delegate to specialized sub-reconcilers (KserveRawRouteReconciler, KserveMetricsServiceReconciler, KserveKedaReconciler, etc.). Each sub-reconciler manages a single resource type using a consistent delta-processor pattern: fetch the desired state, fetch the existing state, compute the delta (added/updated/removed), and apply changes. This pattern is enforced through the `DeltaProcessor` interface with pluggable comparators per resource type, providing a clean separation between reconciliation logic and resource comparison semantics.

The LLM/Gateway controller layer represents the RHOAI 3.x authentication architecture. The GatewayReconciler watches Gateway resources and creates EnvoyFilters for Authorino TLS bootstrap plus AuthPolicies for gateway-level authentication. The LLMInferenceServiceReconciler creates per-service AuthPolicies targeting HTTPRoutes, supporting both anonymous mode (for fairness tracking) and user-defined mode (Kubernetes TokenReview + SubjectAccessReview). The EnvoyFilter template configures Envoy's external authorization filter to communicate with Authorino over gRPC/mTLS on port 50051, forming the complete auth chain: client → Gateway/Envoy → EnvoyFilter → Authorino → backend. The controller discovers the Kuadrant installation namespace dynamically and checks Authorino's TLS configuration before creating EnvoyFilters, making the auth integration self-configuring.

The model-serving-api server is architecturally distinct from the controller — it's a standalone HTTPS REST server deployed as a separate Deployment with its own ServiceAccount and RBAC. Its primary function is gateway discovery with per-user RBAC enforcement: when the RHOAI dashboard needs to show which Gateways a user can target for LLMInferenceService creation, it calls `/api/v1/gateways` with the user's bearer token. The server forwards this token to the Kubernetes API via SelfSubjectAccessReview to check whether the user can `create llminferenceservices` in the target namespace, then filters the gateway list accordingly. The server enforces FIPS-compliant TLS with explicit cipher suite selection (ECDHE+AES-GCM only), certificate hot-reloading via fsnotify, and security headers (HSTS, X-Content-Type-Options, X-Frame-Options).

NIM (NVIDIA NIM) integration is implemented through the Account CRD and a handler chain pattern. The AccountReconciler orchestrates four handlers in sequence: ValidationHandler (validates NGC API key against NVIDIA's API), ConfigMapHandler (fetches available NIM models and creates a ConfigMap with deployment data), TemplateHandler (generates a ServingRuntime template for NIM deployment), and PullSecretHandler (creates a Docker pull secret for nvcr.io). The integration supports air-gapped mode by checking the DataScienceCluster's air-gapped flag — when enabled, API validation is skipped and the controller works with cached data. The Account has a validating webhook enforcing a singleton constraint (one Account per namespace), preventing conflicting NIM configurations.
