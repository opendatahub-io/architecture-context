## Purpose

**Short**: Kubernetes operator that extends KServe with OpenShift/RHOAI-specific model serving capabilities including ingress management, authentication policy, NVIDIA NIM integration, and a Gateway discovery API server.

**Detailed**: The odh-model-controller is a controller-runtime based Kubernetes operator that augments the upstream KServe serving stack with Red Hat OpenShift AI (RHOAI) platform features. It manages the lifecycle of InferenceService, LLMInferenceService, InferenceGraph, and ServingRuntime resources by reconciling supporting infrastructure such as OpenShift Routes, NetworkPolicies, ServiceMonitors, ClusterRoleBindings, and Prometheus dashboards.

The controller implements the RHOAI 3.x authentication and authorization pattern using Kuadrant AuthPolicies targeting Gateway API HTTPRoutes, and creates Istio EnvoyFilters for Authorino TLS bootstrap on Gateway resources. It also provides a NIM Account controller that manages NVIDIA NGC API key validation, model catalog synchronization, Docker pull secret creation, and OpenShift Template generation for NIM ServingRuntimes.

A second deployable component, the model-serving-api server, provides a REST API for gateway discovery. This API allows clients to discover which Gateway API Gateways are available for a given namespace, enforcing RBAC by performing SelfSubjectAccessReview checks with the caller's Bearer token before returning results.

## Architectural Analysis

The odh-model-controller follows a well-structured controller-runtime reconciliation pattern with clear separation of concerns. The main operator binary hosts seven reconcilers organized into three controller packages: `core` (ConfigMap, Pod, Secret), `serving` (InferenceService, InferenceGraph, ServingRuntime), and `serving/llm` (LLMInferenceService, Gateway). The InferenceService reconciler is the most complex, delegating to nine pluggable sub-resource reconcilers that each manage one type of Kubernetes resource (Routes, NetworkPolicies, ServiceMonitors, ClusterRoleBindings, KEDA TriggerAuthentications, dashboards, service accounts, and Model Registry sync). This plugin pattern makes the reconciliation pipeline extensible without modifying the core controller loop.

The authentication architecture reflects the RHOAI 3.x migration from oauth-proxy/Routes to Kuadrant AuthPolicy/Gateway API. The GatewayReconciler watches Gateway resources and conditionally creates EnvoyFilters (for Authorino TLS bootstrap) and AuthPolicies (targeting the Gateway). The LLMInferenceServiceReconciler creates per-HTTPRoute AuthPolicies via the KserveAuthPolicyReconciler sub-reconciler. Both controllers perform conditional CRD discovery — they check at startup whether Kuadrant, Authorino, and AuthPolicy CRDs exist before registering watches, enabling graceful degradation when these optional dependencies are absent. The AuthPolicy templates support two authentication modes: `UserDefined` (delegates policy to the user by not creating an AuthPolicy) and a default mode with Kubernetes TokenReview and configurable audience/objective CEL expressions.

The NIM Account controller implements a handler chain pattern (ValidationHandler → ConfigMapHandler → TemplateHandler → PullSecretHandler) with each handler returning a `HandleResponse` that can halt, requeue, or continue the chain. Air-gapped mode support is thorough — when `spec.components.kserve.nim.airGapped` is set on the DataScienceCluster, the validation handler skips NGC API calls and the config handler uses pre-populated data. The controller also includes a one-time `MaaSRBACCleanupRunner` that removes legacy Roles and RoleBindings from the 3.3 release, demonstrating disciplined upgrade hygiene.

The model-serving-api server is a clean, security-hardened REST server with explicit FIPS-compliant TLS configuration (restricted cipher suites: ECDHE+AES-GCM only, curves P-256/P-384, MinVersion TLS 1.2). Its architecture follows a layered middleware pattern (Recovery → SecurityHeaders → Logging → route handler) with OpenTelemetry instrumentation. The gateway discovery flow performs user-impersonated SelfSubjectAccessReview checks before returning results, preventing information leakage about cluster resources the caller cannot access. The server uses fsnotify-based certificate reloading for zero-downtime TLS cert rotation, which is essential for OpenShift service-ca provisioned certificates.

A notable design choice is the cache filtering in the operator's controller manager: Secrets are filtered to only cache those labeled `opendatahub.io/managed: true`, and Pods are filtered to only those labeled `component: predictor`. This prevents the operator's informer cache from growing unbounded in clusters with many Secrets and Pods, which is critical for multi-tenant RHOAI deployments. The LLMInferenceService controller also implements a global resync mechanism — when Kuadrant or Authorino resources change (creation, deletion, configuration), it re-enqueues all LLMInferenceServices to ensure AuthPolicies are reconciled against the new state.
