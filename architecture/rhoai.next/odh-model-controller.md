## Purpose

**Short**: Manages the lifecycle of model serving infrastructure on OpenShift AI, including InferenceService networking, authentication, monitoring, autoscaling, and NVIDIA NIM integration.

**Detailed**: The odh-model-controller is a Kubernetes operator built with controller-runtime that extends KServe's InferenceService, InferenceGraph, and ServingRuntime CRDs with RHOAI-specific functionality. It watches these KServe resources and reconciles OpenShift Routes, Prometheus metrics services, ServiceMonitors, KEDA autoscaling resources, NetworkPolicies, and ClusterRoleBindings for each deployed model. It also manages CA certificate aggregation for S3 storage connections and Ray TLS certificates for multi-node serving runtimes.

For RHOAI 3.x, the controller adds LLM-specific functionality through the LLMInferenceService and Gateway controllers, which manage Kuadrant AuthPolicy resources and Istio EnvoyFilter resources for Gateway API-based ingress with Authorino authentication. The controller also provides NVIDIA NIM integration through the Account CRD, which automates API key validation, image pull secret creation, and ServingRuntime template generation for NVIDIA NIM models.

The repository produces two deployable binaries: the main controller (`/manager`) and a separate model-serving-api HTTP server (`/model-serving-api`) that provides a REST API for gateway discovery, enabling the RHOAI dashboard to discover available model serving endpoints.

## Architectural Analysis

The odh-model-controller follows a multi-layer reconciliation architecture where the top-level InferenceServiceReconciler delegates to a KserveRawInferenceServiceReconciler, which in turn orchestrates six independent sub-reconcilers (Route, MetricsService, MetricsServiceMonitor, MetricsDashboard, KEDA, ClusterRoleBinding). This modular design allows each concern to be independently tested and extended. The same pattern appears in the LLM serving path, where the LLMInferenceServiceReconciler delegates AuthPolicy management to a KserveAuthPolicyReconciler.

A key architectural decision is the separation of Gateway-level and HTTPRoute-level authentication. The GatewayReconciler manages EnvoyFilter and AuthPolicy resources at the Gateway scope (for TLS bootstrap with Authorino), while the LLMInferenceServiceReconciler manages AuthPolicy resources at the HTTPRoute scope (for per-service authentication). This separation allows the Gateway infrastructure to be bootstrapped independently of individual services, and supports both anonymous and user-defined authentication modes via the `security.opendatahub.io/enable-auth` annotation.

The controller uses conditional CRD availability checks at startup to gracefully degrade when optional dependencies (Kuadrant, Authorino, KEDA, EnvoyFilter) are not installed. This makes the controller deployable in environments without the full RHOAI ingress stack while still providing core InferenceService management. The delta processing pattern (DeltaProcessor with ResourceComparator) ensures idempotent reconciliation by computing create/update/delete operations based on desired vs. actual state.

The model-serving-api server represents a clean separation of concerns: the controller manages cluster state, while the API server provides a user-facing discovery endpoint. The API server implements per-request RBAC by injecting the user's bearer token into SelfSubjectAccessReview calls, ensuring that gateway discovery respects Kubernetes RBAC without requiring the server's ServiceAccount to have broad permissions. The server also implements information leakage prevention by returning empty results (not errors) when access is denied.

The NIM integration is notable for its air-gapped mode support: when the DataScienceCluster indicates air-gapped operation, the NIM handlers skip API calls to NVIDIA and use locally-provided configuration instead. The Account CRD provides a clean abstraction for NIM lifecycle management with conditions-based status tracking and server-side apply for idempotent resource creation.
