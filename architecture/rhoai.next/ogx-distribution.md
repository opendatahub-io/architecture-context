## Purpose

**Short**: OGX Distribution is the container image packaging of the OGX (OpenShift GenAI eXchange) server — the opendatahub-io fork of Llama Stack — providing a multi-API AI gateway for inference, agentic responses, vector storage, file processing, and tool runtime capabilities.

**Detailed**: The ogx-distribution repository produces a production-ready container image (`quay.io/opendatahub/odh-ogx-core`) that packages the OGX server with pre-installed Python dependencies for multiple inference providers (vLLM, AWS Bedrock, Azure, OpenAI, Anthropic, Gemini, Vertex AI, WatsonX), vector databases (Milvus, pgvector, Qdrant, FAISS), file processing pipelines (Docling, MarkItDown, PyPDF), and tool runtimes (Brave Search, Tavily Search, MCP, file search). The image is built for multi-architecture support (amd64, arm64, ppc64le via Konflux) and exposes the OGX server on port 8321.

The repository does not contain application source code — OGX itself lives in the upstream `opendatahub-io/ogx` repository. Instead, this repo focuses on distribution engineering: a build pipeline (`build/build.py`) generates distribution artifacts (`config.yaml`, `requirements.txt`, `Containerfile`) from a canonical provider manifest (`build/build.yaml`), and the Dockerfile layers install those dependencies into an AIPCC base image. Providers are activated at runtime via environment variables using a conditional `${env.VAR:+provider-name}` pattern, allowing a single image to serve all deployment scenarios. The image embeds the runtime configuration as a base64-encoded OCI label (`com.ogx.config.config.yaml`), enabling external tooling to inspect the distribution's capabilities without starting the container.

## Architectural Analysis

The ogx-distribution repository represents a "distribution engineering" pattern common in Red Hat's productization workflow. Rather than containing application source code, it focuses entirely on packaging, configuration, and CI/CD for the upstream OGX project. The canonical configuration lives in `build/build.yaml`, which defines the full provider manifest. A Python build script (`build/build.py`) processes this manifest to generate three artifacts: a stripped `distribution/config.yaml` (removing dependency-only providers like `inline::milvus` and `inline::sentence-transformers` that are pre-installed but not default-enabled), a `requirements.txt` with all Python dependencies including auto-discovered OpenTelemetry instrumentation packages, and a `Containerfile` with base64-encoded OCI labels embedding the config. This code generation approach ensures the config, requirements, and container labels stay consistent — a change to `build/build.yaml` propagates everywhere via `pre-commit run --all-files`.

The provider activation pattern is architecturally significant. All providers are defined in the same `config.yaml`, but each uses conditional `provider_id` syntax (`${env.VLLM_URL:+vllm-inference}`) that OGX evaluates at startup. When the referenced environment variable is unset, the provider is skipped entirely. This means a single container image serves all deployment scenarios — from a minimal setup with just vLLM and PostgreSQL to a fully-loaded configuration with multiple cloud providers, vector databases, and tool runtimes. The pattern avoids the need for variant-specific images or runtime config file injection in most cases, though custom `config.yaml` injection is supported via `RUN_CONFIG_PATH` for advanced scenarios.

The AIPCC base image integration (`quay.io/aipcc/base-images/cpu:3.5.0-ea.2`) anchors the supply chain. For the Konflux (RHOAI) build path, `Dockerfile.konflux` uses a `${BASE_IMAGE}` build arg resolved from `konflux/cpu-ubi9.conf`, providing RHEL 9.6, Python 3.12, and pre-configured access to the RHEL AI Python Package Index. The upstream (ODH) build path uses `Containerfile` with `quay.io/opendatahub/odh-midstream-python-base-3-12:latest`, a different base image with `UV_INDEX_STRATEGY=unsafe-best-match` set. Both build paths converge on the same entrypoint (`distribution/entrypoint.sh`) and configuration, but differ in base image provenance and build hardening. The Konflux path includes additional validation steps — `fromager-rpm-check.py --check-rpmdb` for RPM dependency validation and `selftest.py` for base image self-testing — and notably installs a `models_dev` wheel from the RHAI pulp content repository, which is not present in the upstream path.

The OCI label embedding strategy (`com.ogx.config.config.yaml`) is a notable design decision. The full `config.yaml` is base64-encoded and stored as an OCI image label, allowing external tooling (such as the platform operator) to inspect the distribution's API surface, provider configuration, and storage topology without running the container. The `tests/verify-config-label.sh` test validates that this label stays in sync with the actual `distribution/config.yaml` file. This label is also used to discover the distribution name and version at deployment time.

Build hermeticity on `main` is intentionally relaxed — the Tekton pull-request pipeline declares `hermetic: true` but passes an empty `prefetch-input: "[]"`, meaning Python packages are downloaded from the RHEL AI PyPI index during the build rather than pre-fetched. The `distribution/requirements.txt` uses loosely-pinned versions (e.g., `numpy`, `scipy`, `torch` without version pins), relying on the AIPCC base image's pre-configured index for version resolution. Full hermeticity with lock files is expected on downstream release branches where Konflux release hardening applies `rpms.lock.yaml` and potentially `uv.lock`.
