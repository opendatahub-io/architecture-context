## Purpose

**Short**: ML-based latency prediction service for the llm-d ecosystem that predicts TTFT (Time To First Token) and TPOT (Time Per Output Token) for LLM inference requests.

**Detailed**: The llm-d-latency-predictor is a two-server microservice system that provides online machine learning-based latency predictions for LLM inference workloads. It consists of a training server that collects request traces and trains regression models (XGBoost, LightGBM, or Bayesian Ridge), and a prediction server that serves low-latency TTFT/TPOT predictions using those models.

The training server receives actual latency observations via a bulk data ingestion API, maintains stratified data buckets for balanced training, and periodically retrains quantile or mean regression models. It also supports a gated ensemble mode that trains separate sub-models for queued vs. non-queued request regimes. Trained models are serialized via joblib and served over HTTP for the prediction server to download.

The prediction server periodically syncs models from the training server via HTTP, loads them into memory, and serves single and batch prediction requests through a FastAPI REST API. It supports multi-worker deployment with per-worker model syncing and uses checksum-based deduplication to avoid redundant model reloads. The service is designed to integrate with the llm-d Gateway API inference extension for intelligent request routing based on predicted latencies.

## Architectural Analysis

The llm-d-latency-predictor follows a clean two-server architecture separating model training concerns from inference serving. The training server acts as a singleton that maintains the ground truth model state on persistent storage, while prediction servers are stateless replicas that periodically sync models via HTTP. This design elegantly avoids the need for ReadWriteMany shared storage — each prediction pod uses an emptyDir volume and downloads models independently.

The model sync mechanism in `ModelSyncer` (prediction_server.py:149-311) is notably robust: it uses timestamp comparison to avoid unnecessary downloads, writes to temp files with atomic `os.replace()` for crash safety, validates downloaded files with `joblib.load()` before swapping, and uses MD5 checksums to avoid redundant in-memory reloads across sync cycles. In multi-worker uvicorn deployments, only one worker wins the download race per cycle, but all workers call `load_models()` unconditionally, relying on the checksum-based early return to keep the cost minimal.

The gated ensemble mode (enabled by default via `LATENCY_ENSEMBLE_MODE=true`) is architecturally interesting: it trains separate sub-models for queued (num_request_waiting > 0) and non-queued (num_request_waiting == 0) request regimes, wrapping them in a `QueueGatedModel` that the predictor dispatches based on the request's queue state. This is motivated by the observation that queuing fundamentally changes latency characteristics. The training data is stratified into 3D buckets (queue × KV cache × prefix cache) to ensure balanced representation across operating conditions.

A significant security gap is the absence of TLS and authentication on all endpoints. The training data ingestion endpoint (`/add_training_data_bulk`) accepts arbitrary data without validation beyond schema checks, which could be used to poison the model. The model download endpoint serves serialized Python objects (joblib/pickle) over plaintext HTTP, creating risks for both data integrity and potential deserialization attacks. In a production RHOAI deployment, these endpoints would need to be either protected by a service mesh (mTLS via Istio) or wrapped with kube-rbac-proxy sidecars per the RHOAI 3.x ingress pattern. The reference deployment manifests use GKE-specific constructs (Hyperdisk StorageClass, GKE node pool selectors, LoadBalancer services) and would need adaptation for OpenShift deployment with RHOAI-standard ingress and storage patterns.
