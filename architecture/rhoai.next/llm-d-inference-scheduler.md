## Purpose

**Short**: An inference request scheduler that makes optimized routing decisions for LLM inference workloads through the Kubernetes Gateway API Inference Extension (GIE) framework using Envoy ext_proc.

**Detailed**: The llm-d Inference Scheduler provides an Endpoint Picker (EPP) component that schedules incoming inference requests arriving through a Kubernetes Gateway to optimal model-serving pods. It extends the upstream Gateway API Inference Extension (GIE) project with llm-d-specific features including disaggregated Prefill/Decode (P/D) and Encode/Prefill/Decode (E/P/D) execution, KV cache-aware scheduling via real-time cache event tracking, flow control with priority-based admission, and a pluggable framework of filters, scorers, and pickers.

The repository produces two container images: the EPP (Endpoint Picker Proxy) which is the main scheduling service that receives requests from Envoy via the ext_proc gRPC protocol, and the P/D Routing Sidecar which runs alongside model-serving pods to orchestrate disaggregated inference stages (prefill, decode, encode). The sidecar intercepts inference API calls and coordinates KV cache transfer between prefill and decode workers using pluggable connectors (NIXL V2, shared storage, SGLang).

The scheduler integrates deeply with Kubernetes: it watches InferencePool, Pod, InferenceObjective, and InferenceModelRewrite custom resources via controller-runtime reconcilers, collects real-time metrics from model server pods, and uses the Gateway API (HTTPRoute, Gateway) for traffic routing. The scheduling pipeline is fully configurable through a YAML-based EndpointPickerConfig API that defines plugins, scheduling profiles, feature gates, flow control policies, and data layer configuration.

## Architectural Analysis

The llm-d Inference Scheduler implements a sophisticated multi-layer scheduling architecture built on top of the Kubernetes Gateway API Inference Extension (GIE). The core design pattern is a **plugin framework with scheduling profiles** - the `EndpointPickerConfig` API allows operators to compose arbitrary combinations of filters, scorers, and pickers into named profiles, which a profile handler then selects per-request. This is architecturally similar to the Kubernetes scheduler's scheduling framework, and the code explicitly models this with `SchedulerProfile`, `WeightedScorer`, and `ProfileHandler` abstractions. The GIE v1.5 integration means the scheduling pipeline is partially upstream (core ext_proc handling, controller setup) and partially downstream (llm-d-specific plugins like disaggregation handlers, precise prefix cache scorer, flow control).

The **disaggregated inference architecture** (P/D and E/P/D) is implemented as a two-tier system. The EPP tier makes the routing decision (which pod should handle prefill vs decode), encoding this into HTTP headers. The sidecar tier, running on each model-serving pod, acts on those headers to orchestrate multi-stage inference. This separation of concerns is clean: the EPP knows about cluster-wide scheduling state but doesn't handle KV cache transfer; the sidecar handles KV cache transfer but only sees its local pod. The three KV connector implementations (NIXL V2, shared storage, SGLang) represent fundamentally different KV transfer strategies - NIXL V2 uses direct memory transfer via request parameters, shared storage uses an intermediate storage layer, and SGLang uses concurrent execution with room-based coordination.

A notable architectural risk is the **InsecureSkipVerify defaulting to true** for model server metrics scraping and HTTP data sources. While this simplifies development deployments, it means the EPP doesn't validate model server identities by default in production. Combined with the observation that the EPP Konflux Dockerfile omits `-tags strictfipsruntime` while the sidecar includes it, there's an asymmetry in FIPS posture between the two components that should be reconciled.

The **flow control layer** is an experimental but architecturally significant addition. It introduces priority-based admission with configurable bands, fairness policies (global strict, round robin), ordering policies (FCFS, EDF, SLO-deadline), and saturation detection. This transforms the EPP from a simple request router into a full admission controller capable of queueing, throttling, and shedding load based on priority and capacity. The implementation uses a registry/shard/queue architecture with per-priority-band configuration, which is more sophisticated than the legacy saturation-only admission control.

The **data layer** architecture represents a significant evolution from direct metrics polling to a graph-based data collection system. Sources (metrics HTTP endpoints, Kubernetes notifications, ZMQ event streams) feed extractors that populate an attribute map per endpoint. This DAG-based plugin execution order (topological sort of producer/consumer dependencies) ensures data is available before plugins that consume it run. The migration from legacy polling to this new system is gated behind feature flags, showing an incremental adoption strategy.
