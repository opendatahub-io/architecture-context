## Purpose

**Short**: Intelligent inference request scheduler that routes LLM inference traffic to optimal model server endpoints based on load, prefix cache state, latency predictions, and disaggregated prefill/decode topology.

**Detailed**: The llm-d Inference Scheduler (formerly llm-d-router) is the control-plane brain of the llm-d inference serving stack. It implements the Kubernetes Gateway API Inference Extension (GIE) `ExtensionPolicy` pattern as an Envoy External Processing (ext-proc) gRPC service. When an inference gateway (Envoy, Istio, or any Gateway API-compatible proxy) receives an LLM request, it forwards the request headers and body to this scheduler via ext-proc. The scheduler then selects the optimal backend model server pod based on a pluggable scheduling framework that considers KV-cache utilization, queue depth, running requests, prefix cache affinity, LoRA adapter affinity, session affinity, context length, and predicted latency.

The repository produces two deployable container images: the **Endpoint Picker (EPP)**, which is the main ext-proc scheduler service, and the **PD-Sidecar**, a reverse proxy that runs alongside each model server pod to orchestrate disaggregated prefill/decode inference. The PD-Sidecar intercepts inference requests, dispatches the prefill phase to a remote prefiller pod, receives KV-cache transfer parameters, and then dispatches the decode phase to the local model server — enabling KV-cache disaggregation via NIXL, Mooncake, SGLang, or shared storage connectors. The sidecar also supports multi-stage encoder-prefill-decode (EPD) pipelines for multimodal workloads.

The component defines two custom CRDs (`InferenceObjective` and `InferenceModelRewrite`) in the `llm-d.ai` API group, and watches the upstream `InferencePool` CRD from Gateway API Inference Extension. It integrates with Kubernetes via controller-runtime reconcilers that track pool membership, pod readiness, priority objectives, and model name rewriting rules.

## Architectural Analysis

The llm-d Inference Scheduler implements a sophisticated plugin-based scheduling framework that mirrors the Kubernetes scheduler's filter/score/pick paradigm but adapted for LLM inference workloads. The `EndpointPickerConfig` CRD-like configuration drives plugin instantiation and scheduling profile assembly — multiple named profiles can coexist (e.g., separate "prefill" and "decode" profiles for disaggregated serving) with weighted scorer plugins. The plugin registry uses factory functions and type-safe interfaces (`Filter`, `Scorer`, `Picker`, `ProfileHandler`, `DataProducer`, `PreAdmitter`, `Admitter`) that are registered at startup and instantiated from configuration. This design allows operators to compose scheduling behavior without code changes — for example, enabling latency-aware scheduling by adding the `predicted-latency-producer`, `latency-scorer`, and `latency-slo-admitter` plugins.

The data layer architecture uses a DAG-based dependency resolution system where data producers declare what attributes they provide and consumers declare what they need. The runtime topologically sorts plugins so producers execute before consumers, and auto-creates default producers when a consumer references an unregistered data key. Data sources (HTTP metrics scrape, ZMQ notifications, Kubernetes events) feed extractors that populate per-endpoint attribute maps (concurrency, latency, KV-cache usage, prefix cache state, session affinity, LoRA adapters). This pluggable data pipeline is the foundation for all scheduling decisions.

The PD-Sidecar's connector abstraction (`kv-connector` and `ec-connector` flags) cleanly separates the KV-cache transfer mechanism from the request orchestration logic. Each connector type (NIXLv2, Mooncake, SGLang, shared-storage) injects different `kv_transfer_params` into the request body to coordinate KV-cache transfer between prefiller and decoder pods. The NIXLv2 connector supports MoRI-IO WRITE-mode for reverse KV transfer direction, and the sidecar handles data-parallel routing via blake2s hashing of request IDs to deterministically pin requests to DP ranks. The SSRF protection feature uses Kubernetes watch to maintain a dynamic allowlist of valid prefill/encoder target IPs based on InferencePool pod membership — this is critical in multi-tenant environments where malicious headers could redirect prefill traffic to arbitrary endpoints.

A notable FIPS compliance gap exists: the EPP Dockerfile (`Dockerfile.epp.konflux`) builds with `CGO_ENABLED=1` but omits the `-tags strictfipsruntime` flag, whereas the sidecar Dockerfile includes it. This means the EPP binary may use Go's pure-Go crypto implementation rather than the OpenSSL-backed FIPS module on UBI. While the binary will pass the `check-payload` gate (it's dynamically linked with OpenSSL present in UBI), runtime crypto operations may not use FIPS-validated algorithms. The sidecar is correctly configured for FIPS with explicit cipher suite selection (TLS 1.2 minimum, ECDHE-RSA/ECDSA-AES-GCM-SHA256/384, ChaCha20-Poly1305).

The flow control layer (gated behind the `flowControl` feature gate) implements a priority-based queuing system where requests are classified into priority bands based on InferenceObjective CRs. Each band has configurable capacity limits (max bytes, max requests), fairness policies (global-strict, round-robin, program-aware), and ordering policies (FCFS, EDF, SLO-deadline). Saturation detection uses either KV-cache utilization thresholds or concurrency-based detection to trigger admission control. This system enables fine-grained traffic management — for example, reserving capacity for high-priority production traffic while allowing batch/offline requests to fill remaining capacity.
