## Purpose

**Short**: Intelligent inference request scheduler that makes optimized routing decisions for LLM workloads via Envoy ext-proc, extending the Gateway API Inference Extension (GIE) framework.

**Detailed**: The llm-d Inference Scheduler provides an Endpoint Picker (EPP) component for the llm-d inference framework. It schedules incoming inference requests arriving through a Kubernetes Gateway API-based ingress by executing a configurable pipeline of filters, scorers, and pickers to select the optimal model-serving pod for each request. The EPP operates as a gRPC ext-proc (External Processing) server called by Envoy during request processing, allowing it to inspect request headers/bodies and inject routing decisions as response headers.

The scheduler supports advanced LLM-specific routing strategies including KV cache locality-aware scheduling, prefix cache affinity scoring, LoRA adapter affinity, load-aware distribution, session stickiness, latency SLO-based admission control, and disaggregated Prefill/Decode (P/D) execution where prefill and decode stages run on separate GPU pools. The architecture is highly extensible through a plugin framework that supports custom filters, scorers, pickers, profile handlers, data producers, parsers, and flow control policies — all configured via a YAML EndpointPickerConfig resource.

A companion PD (Prefill-Decode) sidecar proxy runs alongside each model server pod, handling the disaggregated inference protocol by routing requests between prefiller, encoder, and decoder instances using pluggable KV transfer connectors (NIXL v2, shared storage, SGLang). Both components integrate with the Kubernetes API to watch InferencePool, Pod, InferenceObjective, and InferenceModelRewrite custom resources for dynamic configuration.

## Architectural Analysis

The llm-d Inference Scheduler is architecturally notable for its deep integration with the Kubernetes Gateway API Inference Extension (GIE) project. Rather than building a standalone scheduler, it extends GIE's ext-proc framework with LLM-specific scheduling intelligence. The EPP binary is essentially a GIE runner augmented with llm-d-specific plugins registered at startup via `plugins.RegisterAllPlugins()`. This design allows the project to inherit upstream improvements while adding domain-specific capabilities like disaggregated P/D scheduling, KV cache-aware routing, and latency SLO-based admission control. The migration from `sigs.k8s.io/gateway-api-inference-extension` imports (visible in recent commits) reflects an active upstream synchronization effort.

The plugin architecture is sophisticated and resembles the Kubernetes scheduler framework. Plugins implement typed interfaces (Filter, Scorer, Picker, ProfileHandler, DataProducer, Admitter, Parser) and are registered in a global factory registry. Configuration is driven by an `EndpointPickerConfig` YAML resource that declares plugin instances with parameters, then composes them into named `SchedulingProfiles`. A dependency DAG (via the DataLayer `data_graph.go`) ensures data producers run before consumers, and plugins can declare `Consumes()`/`Produces()` contracts for compile-time validation. The framework supports 12 scorer plugins, 6 filters, 3 pickers, and multiple profile handlers — a rich scheduling vocabulary for LLM workloads.

The disaggregated Prefill/Decode (P/D) architecture is a key differentiator. The PD sidecar proxy supports three KV transfer protocols (NIXL v2, shared storage, SGLang) plus an experimental Encode/Prefill/Decode (E/P/D) pipeline for multimodal workloads. Each protocol implements a different coordination strategy: NIXL v2 uses a sequential prefill-then-decode with explicit `kv_transfer_params` exchange; shared storage supports a decode-first optimization with `cache_hit_threshold` fallback; SGLang runs concurrent async prefill + sync decode with bootstrap room coordination. The sidecar includes SSRF protection via a Kubernetes-native allowlist that watches InferencePool pods — a practical security measure since the sidecar receives prefiller/encoder addresses from request headers (set by Envoy/EPP) that could be tampered with.

The flow control subsystem (behind the `ENABLE_EXPERIMENTAL_FLOW_CONTROL_LAYER` feature gate) implements a multi-priority queuing system with pluggable fairness policies (global-strict, round-robin), ordering policies (FCFS, EDF, SLO-deadline), and saturation detection (utilization-based, concurrency-based). This addresses the challenge of managing heterogeneous inference workloads with different latency SLOs and priorities — production LLM platforms often need to prioritize real-time serving over batch processing. The flow control layer includes eviction support for shedding low-priority requests when the system is saturated, with per-priority-band byte and request limits.

The observability design is thorough: every scheduling decision records Prometheus metrics (TTFT, TPOT, token counts, SLO violations, queue depths, prefix cache hit rates), the PD sidecar records fine-grained timing spans (prefill_duration, decode_duration, coordinator_overhead, true_ttft), and both components support OTLP trace export. The health checking implementation distinguishes between liveness (always healthy to prevent restart loops) and readiness (checks pool sync, leader election status, and protocol support) — a subtle but important distinction for Kubernetes deployments where a non-leader EPP instance should be alive but not receive traffic.
