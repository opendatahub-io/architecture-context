## Purpose

**Short**: The llm-d Router is an intelligent inference request routing engine that integrates with Envoy proxy via the ext_proc protocol to provide LLM-aware load balancing, KV-cache-aware scheduling, request prioritization, and prefill/decode disaggregation coordination.

**Detailed**: The llm-d Router (formerly Inference Scheduler) serves as the brain of the inference gateway in RHOAI's LLM serving stack. It produces two deployable components: the **Endpoint Picker (EPP)** and the **Disaggregation Sidecar (PD-Sidecar)**. The EPP watches Kubernetes resources (InferencePool, InferenceObjective, InferenceModelRewrite) and model server pods to build a real-time view of the serving cluster's state. When Envoy forwards inference requests via the ext_proc gRPC protocol, the EPP evaluates candidate endpoints using a configurable plugin-based scheduling framework that considers KV-cache locality, queue depth, running request count, LoRA adapter affinity, session affinity, and priority to select the optimal target pod.

The PD-Sidecar runs alongside decode worker pods in prefill/decode disaggregated deployments. It coordinates multi-stage inference lifecycles (P/D, E/P/D) by routing prefill requests to prefiller pods via pluggable KV-transfer connectors (NIXL v2, shared storage, SGLang, Mooncake) and encoder requests to encoder pods, then forwarding the decode leg to the local model server. This disaggregation enables independent scaling of prefill and decode stages for improved GPU utilization and time-to-first-token (TTFT) latency.

The project is a downstream fork of the upstream [gateway-api-inference-extension](https://github.com/kubernetes-sigs/gateway-api-inference-extension) project, extended with llm-d-specific CRDs (InferenceObjective, InferenceModelRewrite), the PD-Sidecar, and advanced scheduling plugins for production LLM serving.

## Architectural Analysis

The llm-d Router demonstrates a sophisticated plugin-based architecture that cleanly separates scheduling concerns into extension points. The EPP's request processing pipeline flows through distinct phases: request parsing (supporting OpenAI, Anthropic, vLLM HTTP, vLLM gRPC, VertexAI, and passthrough formats), pre-admission hooks, admission control, data production (tokenization, prefix cache analysis, inflight load estimation), scheduling (filter → score → pick), and pre-request mutation. Each phase is backed by a plugin interface with configurable implementations, allowing operators to compose scheduling profiles via YAML configuration without code changes. The profile handler abstraction enables multi-profile execution — for example, the disaggregation profile handler runs separate prefill and decode profiles and selects between them based on the request type.

The data layer architecture is particularly noteworthy. Rather than hard-coding metrics collection, it uses a generic polling/notification/endpoint-driven data source framework. Each model server endpoint gets a dedicated `Collector` goroutine that polls data sources at a configurable interval (default 50ms). Notification sources use controller-runtime's dynamic reconciler binding to watch arbitrary Kubernetes GVKs. This design enables new data sources (e.g., KV-cache events via ZMQ) to be added without modifying the core scheduling path. The data dependency graph is validated at startup using topological sort (Kahn's algorithm) to ensure acyclic dependencies between data producers across layers (FlowControl → RequestControl → Scheduling).

The PD-Sidecar implements a connector-based architecture where each KV-transfer protocol (NIXL v2, shared storage, SGLang, Mooncake) is encapsulated as a pluggable connector. The NIXL v2 connector is the most sophisticated, supporting both serial (prefill → decode) and parallel dispatch (prefill + decode concurrently via MoRI-IO WRITE-mode where the prefill engine pushes KV-cache directly to the decode engine via RDMA). The shared storage connector implements a decode-first optimization: it attempts decode without prefill, falling back to P/D only when the cache hit threshold is not met. This diversity of connectors reflects the rapidly evolving landscape of KV-cache transfer mechanisms in disaggregated LLM serving.

A notable security gap is that InsecureSkipVerify defaults to `true` for model server metrics and models data sources, meaning the EPP does not verify TLS certificates when scraping model server endpoints. While this simplifies deployment (model servers often use self-signed certs), it creates a potential MITM vector in the data collection path. The sidecar's SSRF protection mechanism (InferencePool-based allowlisting) is a pragmatic mitigation for server-side request forgery risks inherent in the disaggregation architecture, where the sidecar accepts prefiller/encoder targets from request headers.

The FIPS compliance posture is inconsistent between the two components: the PD-Sidecar Dockerfile uses `-tags strictfipsruntime` for full FIPS compliance, but the EPP Dockerfile only sets `CGO_ENABLED=1` without the strictfipsruntime tag. Both use UBI 9 minimal base images with OpenSSL available, and both dynamically link (satisfying check-payload), but the EPP's runtime crypto behavior depends on whether the Go runtime selects FIPS-compliant algorithms without the explicit build tag.
