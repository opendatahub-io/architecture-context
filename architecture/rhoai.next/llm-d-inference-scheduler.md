## Purpose

**Short**: An inference request scheduler that makes optimized routing decisions for LLM inference requests via the Kubernetes Gateway API, implementing the Endpoint Picker (EPP) component of the llm-d inference framework.

**Detailed**: The llm-d-inference-scheduler extends the upstream Gateway API Inference Extension (GIE/IGW) project to provide intelligent request routing for LLM inference workloads. It operates as an Envoy External Processing (ext-proc) gRPC server that intercepts inference requests flowing through a Kubernetes Gateway and makes real-time scheduling decisions about which model-serving backend pod should handle each request.

The scheduler employs a sophisticated plugin-based architecture with filters, scorers, and pickers to evaluate backend endpoints based on metrics such as KV-cache utilization, queue depth, prefix cache affinity, LoRA adapter locality, predicted latency, and running request counts. It supports advanced inference patterns including Prefill/Decode (P/D) disaggregation, Encoder/Prefill/Decode (E/P/D) multimodal pipelines, and data-parallel inference across multiple GPU ranks.

The repository produces two deployable container images: the EPP server (primary gRPC service) and the PD sidecar proxy (HTTP reverse proxy for disaggregated inference routing). Both integrate with the Kubernetes Gateway API ecosystem, watching InferencePool, InferenceObjective, and InferenceModelRewrite CRDs to dynamically configure scheduling behavior. The system includes an experimental flow control layer with priority-based admission, fairness policies, and request eviction for managing inference workload under load.

## Architectural Analysis

The llm-d-inference-scheduler represents a sophisticated plugin-based scheduling system for LLM inference that extends the upstream Gateway API Inference Extension (GIE). The architecture follows a clear separation between the scheduling decision plane (EPP) and the data transfer plane (PD sidecar), connected through Envoy's ext-proc mechanism and HTTP reverse proxying respectively.

The scheduling framework's three-stage pipeline — Filter → Scorer → Picker — with weighted score aggregation and DAG-ordered data producers is particularly well-designed. The topological sort (`data_graph.go`) ensures that data-producing plugins execute before their consumers, while the three scorer categories (Affinity, Distribution, Balance) provide semantic structure to scoring weights. The default configuration assigns higher weight (3.0) to prefix cache affinity versus load distribution (2.0 each for queue depth and KV-cache utilization), reflecting the system's optimization for KV-cache locality. The precise prefix cache scorer is notable for integrating with the `llm-d-kv-cache` library via ZMQ subscriptions for real-time block-level cache state awareness.

The disaggregated inference support (P/D, E/P/D) in both the EPP scheduling profiles and the PD sidecar proxy is a distinguishing feature. The EPP's `disagg-profile-handler` orchestrates multi-stage scheduling by running separate filter/scorer profiles for decode, encode, and prefill endpoints, with pluggable deciders (e.g., `prefix-based-pd-decider`) that inspect request characteristics to determine whether disaggregation is beneficial. The PD sidecar then executes the actual disaggregated inference flow, supporting three distinct KV-transfer protocols (NIXL v2, shared storage, SGLang) with specific fallback behaviors — for example, the shared storage connector implements a "try decode first" optimization that buffers initial response chunks to detect cache misses before falling back to remote prefill.

A notable FIPS compliance asymmetry exists between the two images: `Dockerfile.sidecar.konflux` includes `-tags strictfipsruntime` for FIPS-compliant crypto, while `Dockerfile.epp.konflux` does not include this build tag despite both using `CGO_ENABLED=1`. The PD sidecar also implements hardened TLS with restricted cipher suites (`TLS 1.2+, ECDHE + ChaCha20/AES-GCM`), while the EPP's ext-proc server uses default `crypto/tls` configuration. Additionally, several `InsecureSkipVerify` options exist for model server metrics scraping (defaulting to `true`) and per-stage TLS in the sidecar, which could undermine FIPS guarantees in production if not carefully configured.

The experimental flow control layer adds production-grade admission control with priority-based queuing, multiple fairness policies (global-strict and round-robin), ordering policies (FCFS, EDF, SLO-deadline), and request eviction with sheddable/non-sheddable classification. The join-shortest-queue-by-bytes distribution strategy with two-phase fallback (non-blocking attempt → blocking submission) provides effective backpressure management. This system is gated behind a feature flag and integrates with the existing admission control path, demonstrating a clean migration path from the legacy saturation-based admission controller.
