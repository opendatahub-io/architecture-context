## Purpose

**Short**: A Kubernetes operator that manages the lifecycle of Model Context Protocol (MCP) servers by reconciling MCPServer custom resources into Deployments and Services.

**Detailed**: The MCP Lifecycle Operator implements a controller-runtime based Kubernetes operator that watches `MCPServer` custom resources and automatically creates and manages corresponding Deployment and Service resources. Each MCPServer CR defines a container image that implements the Model Context Protocol, along with configuration for ports, environment variables, storage mounts, health probes, replicas, and security settings.

The operator goes beyond simple resource creation by performing an MCP protocol handshake against deployed servers to verify they actually speak MCP. This handshake extracts server identity, protocol version, and capability information (tools, resources, prompts, logging, completions) and surfaces it in the MCPServer status. The operator also watches referenced ConfigMaps and Secrets to compute a config hash that triggers rolling updates when configuration data changes.

This is an upstream Kubernetes SIG project (`kubernetes-sigs/mcp-lifecycle-operator`) that Red Hat ships as part of RHOAI. The downstream fork adds FIPS-compliant builds via `GOEXPERIMENT=strictfipsruntime` and Konflux pipeline integration. The operator runs in the `mcp-lifecycle-operator-system` namespace and manages MCPServer resources across all namespaces (namespaced CRD scope).

## Architectural Analysis

The MCP Lifecycle Operator follows a clean kubebuilder v4 scaffold pattern with a single controller (`MCPServerReconciler`) managing one CRD (`MCPServer`). The operator creates exactly two child resources per MCPServer: a Deployment and a ClusterIP Service, both with the same name as the MCPServer and owned via ownerReferences. This simplicity is intentional — the operator is a lifecycle manager, not a platform operator, and delegates MCP protocol concerns to the server containers themselves.

The most architecturally interesting aspect is the MCP protocol handshake verification. After ensuring the Deployment is available with healthy replicas, the controller performs an actual MCP initialize handshake against the service URL (`http://{name}.{namespace}.svc.cluster.local:{port}{path}`) using the `modelcontextprotocol/go-sdk` client. This goes beyond readiness probes by verifying protocol-level conformance: the server must respond to the MCP initialize request with valid server info and capabilities. Failed handshakes trigger exponential backoff (10s → 20s → 40s → 80s → 120s, max 10 retries) before the controller stops requeuing. Auth errors (401/403) are intentionally treated as "reachable" — the server speaks HTTP but requires authentication, which is a valid deployment state.

The config hash mechanism provides GitOps-friendly rolling updates: when any ConfigMap or Secret referenced by storage mounts, envFrom, or env valueFrom changes, the controller recomputes a SHA-256 hash of all referenced data and sets it as a pod template annotation. This triggers Kubernetes to roll the Deployment without any spec change to the MCPServer CR. The controller uses field indexes (`spec.configMapRefs`, `spec.secretRefs`) for efficient O(1) lookup of which MCPServers reference a given ConfigMap or Secret.

The ownership validation logic (`validateOwnership`) implements a nuanced adoption model: it rejects resources with no controller owner (preventing silent takeover of manually-created resources) but allows adoption of orphaned resources from a deleted MCPServer with the same name/namespace/API-group. This enables seamless MCPServer recreation without manual cleanup of stale Deployments and Services.

The RHOAI downstream adds FIPS compliance via `GOEXPERIMENT=strictfipsruntime` and `CGO_ENABLED=1` in the Konflux Dockerfile, using a UBI9-minimal base image that provides OpenSSL. The operator explicitly disables HTTP/2 on its webhook and metrics servers as a CVE mitigation (HTTP/2 Stream Cancellation and Rapid Reset). The default container security context for managed MCP server pods follows the restricted Pod Security Standard: non-root, read-only root filesystem, all capabilities dropped, RuntimeDefault seccomp profile.
