## Purpose

**Short**: An OpenAI-compatible HTTP gateway that routes chat completion requests through the FMS Guardrails Orchestrator with configurable detector-based content filtering per route.

**Detailed**: The vLLM Orchestrator Gateway is a Rust-based HTTP service built on the axum web framework that provides controlled interaction with the FMS Guardrails Orchestrator. It sits between clients and the orchestrator, exposing multiple OpenAI-compatible `/v1/chat/completions` endpoints, each bound to a specific set of content safety detectors.

The gateway reads a YAML configuration file that defines three concerns: the orchestrator backend connection details, a registry of available detector configurations (with input/output classification and parameters), and named routes that bind subsets of detectors to unique HTTP paths. At runtime, each route creates a `POST /{route_name}/v1/chat/completions` endpoint. When a request arrives, the gateway injects the route's detector configuration into the payload and forwards it to the orchestrator's `/api/v2/chat/completions-detection` endpoint. If the orchestrator returns detection results (e.g., PII found), the gateway can replace the response with a configurable fallback message. Both streaming (SSE) and non-streaming response modes are supported.

The gateway supports mTLS for communication with the orchestrator backend, loading client certificates and CA bundles from well-known Kubernetes-mounted paths (`/etc/tls/private/` and `/etc/tls/ca/`). It forwards `Authorization` and `x-forwarded-*` headers from incoming requests to the orchestrator, enabling pass-through authentication.

## Architectural Analysis

The vllm-orchestrator-gateway follows a straightforward reverse proxy pattern, purpose-built for the TrustyAI guardrails use case. The core architectural decision is to use configuration-driven dynamic routing: rather than hardcoding detection policies, routes are defined declaratively in YAML, and each route generates a separate HTTP endpoint. This enables a single gateway instance to serve multiple independent safety policies (e.g., `/pii/v1/chat/completions` for PII filtering, `/passthrough/v1/chat/completions` for unguardrailed access). The configuration is validated at startup with `validate_registered_detectors()`, which panics on misconfiguration — fail-fast rather than serving traffic with broken detector bindings.

The TLS implementation (`build_orchestrator_client` in `src/main.rs:351-406`) is noteworthy for its conditional mTLS approach. The gateway probes for TLS certificate files at well-known Kubernetes mount paths and dynamically configures the HTTP client: if certs are present, it constructs a PKCS#12 identity from PEM files using the `openssl` crate and enables HTTPS; if absent, it falls back to plaintext HTTP. This design supports both development (localhost, no TLS) and production (Kubernetes with mounted secrets) without configuration changes. The `danger_accept_invalid_hostnames(true)` call is gated on `hostname == "localhost"`, preventing it from activating in production. However, the gateway itself does not terminate TLS on its listening port — it serves plaintext HTTP on 8090/TCP, relying on the platform's ingress stack (Gateway API / kube-rbac-proxy) for external TLS termination.

The streaming implementation (`handle_streaming_generation` and `orchestrator_streaming_request`) processes SSE chunks from the orchestrator's response stream, parsing each `data:` line. Detection-aware fallback is applied per-chunk: if a streaming response chunk contains detection results and a fallback message is configured, the delta content is replaced inline. This preserves the streaming contract (clients receive SSE events) while still enforcing content safety policies on streamed output.

The codebase is compact (~550 lines of Rust across 3 source files), which makes it easy to audit but also means there is no health check endpoint, no metrics exposure (Prometheus or otherwise), no graceful shutdown handling, and no readiness/liveness probes. These concerns are likely expected to be handled at the deployment level (Kubernetes probes hitting the HTTP port, sidecar metrics collection). The absence of a `/health` or `/ready` endpoint is a gap worth noting for production readiness.
