## Purpose

**Short**: An HTTP gateway that provides multiple OpenAI-compatible chat completion endpoints with configurable detector-based content filtering, routing requests to the FMS Guardrails Orchestrator.

**Detailed**: The vLLM Orchestrator Gateway sits between API clients and the FMS Guardrails Orchestrator, enforcing stricter access to the orchestrator's endpoints by providing a mechanism for configuring fixed detector pipelines. Each configured pipeline is exposed as a unique `/v1/chat/completions` endpoint, enabling drop-in replacement of an unguardrailed chat completions model with a guardrailed one.

The gateway reads a YAML configuration file that defines the upstream orchestrator connection details, available content detectors (with input/output classification and parameters), and named routes. Each route maps to a set of detectors and an optional fallback message. When a request arrives at a route's endpoint, the gateway injects the configured detector specifications into the payload and forwards it to the orchestrator's `/api/v2/chat/completions-detection` endpoint. If detections are found and a fallback message is configured, the gateway replaces the response content with the fallback message.

The gateway supports both streaming (SSE) and non-streaming chat completion requests, mTLS for secure communication with the orchestrator, and forwards authorization and `x-forwarded-*` headers from clients to the orchestrator. It is part of the TrustyAI guardrails stack within RHOAI, providing a route-level content safety enforcement layer.

## Architectural Analysis

The vLLM Orchestrator Gateway implements a lightweight reverse-proxy pattern specialized for AI content safety enforcement. Rather than exposing the FMS Guardrails Orchestrator's full API surface, it constrains access to a set of pre-configured detector pipelines, each exposed as a standard OpenAI-compatible `/v1/chat/completions` endpoint. This design enables drop-in replacement of raw model endpoints with guardrailed versions — clients need only change the base URL, with no changes to their request format.

The gateway's configuration model cleanly separates three concerns: orchestrator connectivity (`orchestrator` section), detector definitions (`detectors` section with input/output classification), and route-to-detector mappings (`routes` section with optional fallback messages). Detector parameters are injected into the `detectors` field of the orchestrator request payload at proxy time (lines 232-237 in `main.rs`), which means the gateway augments rather than replaces the client's request. The fallback message mechanism (`check_payload_detections`, lines 138-152) provides a simple content replacement policy — if any detection is found and a fallback is configured, the entire response is replaced.

The mTLS implementation (`build_orchestrator_client`, lines 351-406) is notable for its use of `openssl` for PKCS#12 identity construction rather than direct PEM-based identity in `reqwest`. This is because `reqwest`'s `native-tls` backend requires PKCS#12 format for client identity. The code loads PEM cert/key, converts via OpenSSL to PKCS#12, then passes the DER-encoded identity to `reqwest`. The `danger_accept_invalid_hostnames(true)` flag is correctly scoped to `localhost` connections only (line 372-374), which is acceptable for development scenarios where the orchestrator's certificate names its Kubernetes service DNS name.

The streaming implementation (lines 259-349 and 478-547) processes the orchestrator's SSE stream by parsing `data:` prefixed lines, deserializing each chunk, applying fallback message logic per-chunk, and re-emitting as SSE events. This preserves the streaming user experience while still enforcing content policies. However, there is an architectural limitation: fallback messages are applied per-chunk rather than aggregated, meaning a detection in one chunk triggers fallback for that chunk only — other chunks in the stream may still pass through unmodified.

The service has no Kubernetes deployment manifests in its repository — it depends on external deployment orchestration (likely from the TrustyAI operator or RHOAI platform operator) to create the Kubernetes Service, mount TLS secrets, and provide the config file. This is a clean separation of concerns: the gateway owns its application logic, while the operator owns its lifecycle management and network exposure.
