## Purpose

**Short**: An HTTP gateway that provides multiple OpenAI-compatible chat completion endpoints with configurable detector-based content filtering, routing requests to the FMS Guardrails Orchestrator.

**Detailed**: The vLLM Orchestrator Gateway acts as a routing and policy enforcement layer between clients and the FMS Guardrails Orchestrator. It exposes dynamically configured HTTP endpoints following the OpenAI `/v1/chat/completions` pattern, each bound to a specific set of content detectors (e.g., PII regex detectors). When a request arrives on a configured route, the gateway injects the route's detector configuration into the payload and forwards it to the orchestrator's `/api/v2/chat/completions-detection` endpoint. If the orchestrator returns a detection result, the gateway can substitute a configurable fallback message in the response.

The gateway supports both streaming (SSE) and non-streaming chat completion modes. It reads its configuration from a YAML file that defines three concerns: orchestrator connection details (host/port), detector definitions (name, input/output scope, parameters), and route definitions (name, detector list, fallback message). This allows platform operators to expose multiple guardrailed endpoints from a single gateway instance, each with different safety policies. The gateway supports mTLS for secure communication with the orchestrator using OpenSSL-backed TLS via the `native-tls` and `openssl` Rust crates.

## Architectural Analysis

The vLLM Orchestrator Gateway follows a straightforward reverse-proxy pattern specialized for AI guardrailing. Its primary architectural contribution is translating the OpenAI chat completions API surface into the FMS Guardrails Orchestrator's detector-augmented API. By injecting detector configurations at the proxy layer, the gateway allows consumers to interact with a standard `/v1/chat/completions` endpoint while transparently receiving content safety enforcement — a drop-in replacement pattern for unguardrailed model endpoints.

The configuration model is declarative and static: routes, detectors, and the orchestrator endpoint are defined in a YAML file loaded at startup. There is no hot-reload capability; configuration changes require a gateway restart. This simplicity is appropriate for the use case (routes map 1:1 to safety policies), but limits dynamic reconfiguration in multi-tenant scenarios. The route-to-detector binding uses a name-based lookup with startup validation (`validate_registered_detectors`) that panics on misconfiguration, providing fail-fast behavior.

The TLS implementation is notable for its dual-path design: the gateway determines HTTP vs HTTPS scheme based on the presence of certificate files at well-known paths (`/etc/tls/private/tls.crt`, `/etc/tls/private/tls.key`), and adds a custom CA certificate if present at `/etc/tls/ca/service-ca.crt`. This follows the OpenShift service-ca pattern where the platform provisions certificates and CA bundles via volume mounts. The `danger_accept_invalid_hostnames(true)` call when the orchestrator host is `localhost` is a development convenience that does not pose a production risk, since production deployments use the service DNS name. The PKCS#12 conversion path (PEM → X509/PKey → PKCS#12 → native-tls Identity) is a workaround for `reqwest`'s `native-tls` backend requiring PKCS#12 format, despite the platform providing PEM-encoded certificates.

The streaming implementation uses axum's SSE support with a `futures::Stream` adapter. Each SSE chunk from the orchestrator is parsed, checked for detections, and optionally modified before re-emission. The gateway strips the `data: ` prefix from upstream SSE lines and filters `[DONE]` sentinels. One architectural risk is that the streaming fallback message logic modifies only the first chunk where detections are found, potentially leaving subsequent chunks with the original (detected) content still flowing to the client.

The service has no Kubernetes manifests, RBAC definitions, or CRDs — it is a pure application-layer service expected to be deployed by an external operator or deployment mechanism. The Konflux build pipeline produces multi-architecture images (x86_64, arm64, ppc64le, s390x) from a UBI 9 minimal base, with `compat-openssl11` installed in the release image for OpenSSL compatibility.
