## Purpose

**Short**: An HTTP gateway that provides route-specific OpenAI-compatible chat completion endpoints with configurable detector-based content filtering, proxying requests to the FMS Guardrails Orchestrator.

**Detailed**: The vllm-orchestrator-gateway is a Rust-based HTTP reverse proxy that sits between clients and the FMS Guardrails Orchestrator. It reads a YAML configuration file that defines detector pipelines (for content safety, PII detection, etc.) and exposes dynamically-generated `/{route-name}/v1/chat/completions` endpoints, one per configured route. Each route maps to a specific combination of input and/or output detectors with route-specific fallback messages.

When a client sends a standard OpenAI-compatible chat completion request to a route endpoint, the gateway injects the configured detector specifications into the request payload and forwards it to the orchestrator's `/api/v2/chat/completions-detection` endpoint. If the orchestrator returns detections (e.g., PII found in input or output), the gateway replaces the response content with the route's configured fallback message. The gateway supports both non-streaming and streaming (SSE) chat completions, with detection-based filtering applied in both modes.

The gateway supports mTLS for secure communication with the orchestrator when TLS certificates are present at well-known paths (`/etc/tls/private/`), and forwards `Authorization` and `X-Forwarded-*` headers from clients to the orchestrator. It is part of the TrustyAI guardrails stack within RHOAI.

## Architectural Analysis

The vllm-orchestrator-gateway follows a clean reverse-proxy pattern with a narrow scope: it transforms OpenAI-compatible chat completion requests by injecting detector specifications, then proxies them to the FMS Guardrails Orchestrator. The entire application is three Rust source files (~575 lines total), making it one of the simplest components in the RHOAI platform. The architectural intent is to provide a "drop-in replacement" for an unguardrailed model endpoint — clients send standard OpenAI API requests and receive standard responses, with content safety enforcement happening transparently.

The dynamic routing design (`/{route-name}/v1/chat/completions`) is notable: routes are registered at startup from the YAML config, and each route binds a specific set of detectors. This allows a single gateway instance to serve multiple safety profiles — for example, one route that enforces PII detection and another that passes through unfiltered. The route configuration includes per-route `fallback_message` strings that replace model output when detections are found, providing a consistent user experience regardless of the detection type.

The mTLS implementation (lines 351-406 of `main.rs`) is well-structured: the gateway probes for certificate files at startup and dynamically configures the HTTP client with TLS identity if certificates are present, otherwise falls back to plain HTTP. This supports both development (no certs, HTTP) and production (certs mounted from Kubernetes secrets, HTTPS with mTLS) modes without configuration changes. The use of the `openssl` crate for PKCS#12 construction ensures compatibility with the system's OpenSSL installation, which is important for FIPS compliance on RHEL/UBI — the native-tls backend delegates all crypto operations to the system OpenSSL library.

One concern is the `danger_accept_invalid_hostnames(true)` call when the orchestrator hostname is `localhost` (line 373). While this is a valid development convenience, it means a misconfigured production deployment pointing at `localhost` would bypass hostname verification. The gateway also lacks health check endpoints (no `/healthz` or `/readyz`), which may require a Kubernetes readiness probe based on TCP socket checks rather than HTTP endpoint checks.

The streaming implementation processes SSE chunks from the orchestrator, parsing each chunk for detection results and applying fallback messages on the first detected chunk. The SSE keep-alive mechanism (`KeepAlive::default()`) ensures long-running streaming connections are maintained. Error handling in the streaming path is graceful — parse failures on individual chunks result in pass-through rather than stream termination.
