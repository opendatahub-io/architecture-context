## Purpose

**Short**: An HTTP gateway that provides multiple OpenAI-compatible chat completion endpoints with configurable detector-based content filtering, routing requests to the FMS Guardrails Orchestrator.

**Detailed**: The vllm-orchestrator-gateway acts as a routing and policy enforcement layer between clients and the FMS Guardrails Orchestrator. It reads a YAML configuration file that defines detector pipelines and named routes, then dynamically exposes `/{route_name}/v1/chat/completions` POST endpoints for each configured route. Each route can bind specific content detectors (e.g., PII regex detectors) and define a fallback message to return when detections are found, enabling drop-in replacement of an unguarded chat completions model with a guardrailed one.

The gateway supports both streaming (Server-Sent Events) and non-streaming OpenAI-compatible chat completion responses. It injects detector configurations into the request payload before forwarding to the orchestrator's `/api/v2/chat/completions-detection` endpoint, then inspects the orchestrator's response for detections and substitutes the configured fallback message when content policy violations are found. The gateway forwards `Authorization` and `X-Forwarded-*` headers from incoming requests, and supports mTLS for secure communication with the orchestrator using certificates mounted at well-known paths.

## Architectural Analysis

The vllm-orchestrator-gateway follows a clean gateway/proxy pattern implemented as a single-binary Rust service using the axum web framework. The architecture is deliberately simple: a YAML configuration file defines the complete routing topology at startup, and no dynamic reconfiguration is supported. Each configured route becomes a POST endpoint that injects detector specifications into the client's payload before forwarding to the FMS Guardrails Orchestrator. This design enables platform operators to offer multiple "flavors" of a guardrailed chat completion endpoint (e.g., PII-filtered, passthrough, safety-filtered) behind a single gateway deployment, all backed by the same orchestrator and model server.

The mTLS implementation (`build_orchestrator_client` in `main.rs`, lines 351-406) is noteworthy for its approach: it reads PEM-encoded certificates using the `openssl` crate, constructs a PKCS#12 identity in memory, and loads it via `native-tls`. The code includes a `danger_accept_invalid_hostnames(true)` escape hatch specifically for localhost development, which should be carefully reviewed for production deployments. TLS is activated opportunistically — if the cert and key files exist at the well-known paths (`/etc/tls/private/tls.crt` and `/etc/tls/private/tls.key`), HTTPS is used; otherwise, the client falls back to plaintext HTTP. The CA certificate at `/etc/tls/ca/service-ca.crt` is similarly optional and is added as a root certificate when present, which is the standard OpenShift service-ca pattern.

The streaming implementation (lines 259-349) uses axum's SSE support with `futures::StreamExt` to process chunked SSE responses from the orchestrator. Each chunk is parsed for the SSE `data:` prefix, and detection-bearing chunks have their content replaced with the route's fallback message. This is a simple but effective approach — however, the current implementation only replaces the first chunk that contains detections rather than buffering the entire stream, which means partial streaming content may be visible to the client before a detection triggers the fallback. This is a design tradeoff between latency and strictness.

The gateway performs no authentication itself — it forwards `Authorization` and `X-Forwarded-*` headers to the orchestrator and relies on the orchestrator (or upstream infrastructure) for auth enforcement. This makes the gateway a pure routing and policy layer, which is appropriate for its position in the TrustyAI content filtering stack. The configuration validation at startup (`validate_registered_detectors` in `config.rs`, lines 64-108) ensures that all detector references in routes are valid and that no route has duplicate input or output detectors for the same server, catching configuration errors early.

The Konflux build pipeline produces multi-architecture images (x86_64, arm64, ppc64le, s390x) based on UBI 9 minimal, consistent with RHOAI's platform support requirements. The image label identifies it as `odh-trustyai-vllm-orchestrator-gateway-rhel9`, placing it within the TrustyAI component family.
