## Purpose

**Short**: A Kubernetes module operator that manages the lifecycle of the MCP Lifecycle Operator operand within the Open Data Hub / RHOAI platform.

**Detailed**: The MCP Lifecycle Module Operator follows the ODH modular architecture pattern (v2 of the Onboarding Guide). It acts as a thin lifecycle management layer: the ODH platform operator creates a cluster-scoped `MCPLifecycleOperator` custom resource, and this module operator reconciles it by deploying and managing the MCP Lifecycle Operator operand via Server-Side Apply (SSA).

The module operator does not implement any business logic. Its sole responsibilities are: (1) rendering vendored operand manifests with runtime parameters (namespace, image), (2) applying those manifests via SSA using the `odh-platform-utilities` deploy library, (3) garbage-collecting stale resources when the desired state changes, and (4) tearing down all owned resources when `managementState` is set to `Removed`. The operand it deploys (mcp-lifecycle-operator) is the actual controller that manages `MCPServer` CRDs for running Model Context Protocol servers in Kubernetes.

The operator reads its configuration from environment variables: `RELATED_IMAGE_ODH_MCP_LIFECYCLE_OPERATOR_IMAGE` for the operand container image (injected by the platform operator), `SYSTEM_NAMESPACE` for the deployment namespace (via Downward API), and `OPERATOR_VERSION` for version tracking. When the operand image env var is unset, the default image baked into the vendored manifest is used.

## Architectural Analysis

The MCP Lifecycle Module Operator is a clean implementation of the ODH modular architecture pattern. It serves as a reference example for how new components are onboarded to the platform: a thin module operator that wraps an upstream operand with platform-standard lifecycle management. The separation of concerns is strict -- the module operator handles only deployment mechanics (SSA apply, garbage collection, readiness checking), while all business logic resides in the operand (mcp-lifecycle-operator).

The manifest management approach is notable. Rather than using kustomize at runtime, the operator embeds a pre-rendered YAML file (`internal/controller/resources/mcp-lifecycle-operator.yaml`) containing the complete operand manifest (2107 lines). This file is vendored from the upstream MCP Lifecycle Operator via `make update-operand-manifests`, which clones the upstream repo and runs its build-installer target. At reconcile time, the `KustomizeProvider` loads this embedded YAML, applies transformers (namespace injection, image replacement, label injection), and passes the result to the SSA deployer. This design avoids runtime dependency on kustomize binaries and makes the deployed resources fully deterministic.

The garbage collection mechanism is sophisticated. The controller builds a `desiredSet` of resource keys (GVK + namespace + name) from the rendered manifests, then uses the `odh-platform-utilities` garbage collector to find and delete any resources labeled `platform.opendatahub.io/part-of=mcplifecycleoperator` that are NOT in the desired set. This handles manifest drift cleanly -- if a resource is removed from the vendored manifest in a new version, the old resource is automatically cleaned up. The `gc.WithOnlyCollectOwned(false)` flag means it collects by label rather than by ownerReference, which is appropriate for cross-namespace resources (the module operator runs in one namespace but may manage resources in the operand namespace).

The operand that gets deployed includes a full RBAC stack for MCPServer management: admin, editor, and viewer ClusterRoles with RBAC aggregation labels (`aggregate-to-admin`, `aggregate-to-edit`, `aggregate-to-view`). This means Kubernetes cluster-admin, edit, and view roles automatically inherit MCPServer permissions. The operand also deploys its own metrics service on port 8443/TCP with TokenReview + SubjectAccessReview authentication, following the standard controller-runtime metrics protection pattern.

One architectural observation: the module operator's RBAC is broader than strictly necessary for a lifecycle manager. It has full CRUD on `mcpservers`, `mcpservers/status`, and `mcpservers/finalizers` in the `mcp.x-k8s.io` API group. This is because the same ClusterRole needs to grant the module operator permission to deploy the operand's RBAC resources (which reference MCPServer verbs), and the garbage collector needs to enumerate and delete MCPServer-related resources during teardown. The `RHOAIENG-70870` commit explicitly addressed RBAC least-privilege hardening.
