## Purpose

**Short**: ModelMesh is a distributed LRU cache and routing layer for high-scale, high-density model serving that runs as a sidecar alongside model runtime containers.

**Detailed**: ModelMesh is a mature, general-purpose model serving management and routing framework designed for high-scale, high-density, and frequently-changing model use cases. It acts as a distributed LRU cache for serving runtime models, managing the lifecycle of models across a cluster of instances. Each ModelMesh instance runs as a sidecar container alongside a model runtime container (such as Triton, MLServer, or custom runtimes), communicating with the runtime via gRPC over either TCP or Unix Domain Sockets.

The framework provides intelligent model placement and routing across a cluster of pods. It uses a distributed key-value store (etcd or ZooKeeper) for model registry persistence, instance coordination, leader election, and dynamic configuration. ModelMesh automatically handles model loading, unloading, cache eviction based on LRU policy, and request forwarding to instances where models are loaded. It supports virtual models (vmodels) for zero-downtime model updates, type constraints for heterogeneous hardware clusters (e.g., GPU vs CPU nodes), and per-model concurrency limiting with latency-based autoscaling.

In the RHOAI platform, ModelMesh is deployed by the modelmesh-serving controller as part of ServingRuntime pods. The modelmesh-serving controller manages the Kubernetes-level orchestration (ServingRuntime and InferenceService CRDs), while ModelMesh handles the runtime-level orchestration (model placement, caching, routing, and lifecycle management within the serving cluster).

## Architectural Analysis

ModelMesh implements a sophisticated distributed cache architecture that is fundamentally different from typical Kubernetes services. Rather than being a stateless request handler, each ModelMesh instance maintains significant local state: a ConcurrentLinkedHashMap-based LRU cache of loaded models, with model placement decisions coordinated across the cluster through etcd. The system operates as a two-tier architecture: the external gRPC API (ModelMeshApi on port 8033) handles client requests, while the internal Thrift-based Litelinks layer (port 8080) handles inter-instance model forwarding. This separation allows the external API to support flexible TLS and client auth configurations while the internal layer uses its own TLS with mutual authentication.

The sidecar pattern is central to the design. SidecarModelMesh communicates with a colocated model runtime container via gRPC, using either TCP (port 8085) or Unix Domain Sockets (/tmp/mmesh/grpc.sock). The UDS option reduces network overhead for the high-frequency loadModel/unloadModel/inference traffic between containers in the same pod. The runtime communicates its capabilities at startup via the runtimeStatus RPC, including memory capacity, loading concurrency limits, and method routing information. ModelMesh then uses this information to make intelligent placement decisions across the cluster, treating each instance's model memory as a distributed cache with global awareness.

The virtual model (vmodel) system provides a critical abstraction for zero-downtime model updates. When a vmodel's target changes, the VModelManager (running on the cluster leader) orchestrates a transition: it loads the new target model to the same scale as the current active model, then atomically swaps the active model pointer once the target is ready. This prevents inference interruptions during model updates. The owner-based access control on vmodels provides basic multi-tenancy within a single ModelMesh cluster.

Type constraints enable heterogeneous hardware support. Instances can be labeled (e.g., "gpu") and model types can require or prefer specific label sets. The TypeConstraintManager maintains a mapping of label sets to instance statistics, ensuring models are placed only on compatible hardware. This is particularly important for RHOAI deployments where GPU and CPU nodes coexist. The constraint system uses a ProhibitedTypeSet per instance to efficiently reject incompatible models during placement.

A notable design decision is the custom Prometheus client implementation. Rather than using the standard Prometheus Java client directly, ModelMesh includes optimized Counter, Gauge, and Histogram implementations with linear-probe hash tables for label management, reducing allocation overhead on the hot path. The Prometheus metrics are served via a custom Netty HTTP server (port 2112) with TLS support using self-signed certificates generated via BouncyCastle, rather than the standard Prometheus HTTP server. This choice reflects the performance sensitivity of the model serving workload.
