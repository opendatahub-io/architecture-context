## Purpose

**Short**: ModelMesh is a distributed LRU cache framework that manages routing and lifecycle of machine learning models across a cluster of model serving runtime pods.

**Detailed**: ModelMesh provides a mature, general-purpose model serving management and routing layer designed for high-scale, high-density, and frequently-changing model use cases. It runs as a sidecar container alongside model runtime containers (such as Triton, Seldon MLServer, or custom runtimes) and manages model loading, unloading, and inference request routing across all instances in a deployment.

The framework uses etcd or ZooKeeper as a distributed key-value store to coordinate model placement decisions across pods. It implements an LRU cache eviction strategy to maximize model density — loading frequently-used models on demand and evicting least-recently-used models when capacity is exceeded. ModelMesh communicates with colocated model runtimes via the internal `ModelRuntime` gRPC API (over TCP or Unix Domain Sockets) and exposes an external `ModelMesh` gRPC API for model lifecycle management (register, unregister, ensure loaded) and inference request proxying.

ModelMesh is built on IBM's Litelinks RPC framework for inter-pod communication and service discovery. In RHOAI, it is deployed by the `modelmesh-serving` controller as part of `ServingRuntime` pod templates. The framework supports TLS for both external and internal communication, Prometheus metrics, payload logging/processing, virtual model (vModel) aliases for blue-green model transitions, type constraints for heterogeneous hardware, and per-model concurrency limiting with latency-based autoscaling.

## Architectural Analysis

ModelMesh implements a sophisticated distributed caching pattern where each pod maintains an in-memory LRU cache of loaded models, coordinated through a shared KV store (etcd or ZooKeeper). The `ModelMesh.java` class (~6000+ lines) is the core — it extends Litelinks' `ThriftService` and implements a `KVTable.Listener` to react to model record changes in the KV store. The class manages model placement decisions through leader election: the elected leader instance performs global rebalancing, while all instances independently handle local cache operations and LRU evictions.

The sidecar architecture (`SidecarModelMesh`) is the primary deployment mode. It communicates with colocated model runtimes via gRPC, supporting both TCP (port 8085 by default) and Unix Domain Sockets (`/tmp/mmesh/grpc.sock`). The sidecar probes the runtime for capabilities at startup via the `runtimeStatus` RPC, discovering supported inference methods, model ID injection paths, and concurrency limits. This probe also supports backward compatibility with the deprecated `ModelServer` (TAS) gRPC interface, automatically falling back when the newer `ModelRuntime` interface is not implemented.

The inference proxying path is highly optimized. The `ModelMeshApi` class implements `ServerCallHandler<ByteBuf, ByteBuf>` directly, working with raw Netty ByteBuf objects instead of deserialized protobuf messages. This zero-copy approach avoids serialization overhead for inference requests — the model request payload is passed through to the model runtime unchanged. Model ID injection into request messages is handled by `ProtoSplicer`, which performs in-place protobuf field replacement at the byte level. The external gRPC server uses a `HandlerRegistry` that dynamically creates method definitions for any inference method name, allowing ModelMesh to proxy arbitrary gRPC services without pre-registration.

A notable architectural risk is the explicit disabling of Java FIPS mode (`-Dcom.redhat.fips=false` in start.sh). The comments indicate this breaks when TLS is enabled, but the root cause is the NSS/PKCS11 security provider reconfiguration performed in both the Dockerfile and start.sh. This means ModelMesh's TLS uses BoringSSL-static (via Netty tcnative) rather than FIPS-validated cryptographic modules. For RHOAI deployments on FIPS-enabled OpenShift clusters, this is a significant compliance gap that should be tracked.

The virtual model (vModel) system, managed by `VModelManager`, provides blue-green deployment semantics at the model level. A vModel is an alias that maps to a concrete model ID. When a vModel's target is updated, the new model is loaded to the same scale as the current active model before the alias is switched. This enables zero-downtime model updates. The vModel state is stored in a separate KVTable in etcd, and the manager handles race conditions, owner-based access control, and automatic cleanup of unreferenced concrete models.
