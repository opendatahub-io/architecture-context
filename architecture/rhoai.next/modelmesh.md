## Purpose

**Short**: ModelMesh is a distributed LRU cache for serving runtime models, acting as a sidecar that manages model lifecycle and request routing across a cluster of model serving pods.

**Detailed**: ModelMesh is a mature, general-purpose model serving management and routing layer designed for high-scale, high-density, and frequently-changing model use cases. It works with existing or custom-built model servers and acts as a distributed LRU cache — automatically loading, unloading, and routing inference requests to the right instance where a model is loaded.

ModelMesh runs as a sidecar container alongside model runtime containers (e.g., Triton, MLServer, custom runtimes). It communicates with the colocated model runtime via an internal gRPC `ModelRuntime` API (for load/unload/status operations) and exposes an external gRPC API for model management (register/unregister/status) and inference request proxying. Instances within a deployment coordinate via a KV store (etcd or ZooKeeper) to maintain a consistent view of model placement across the cluster, perform leader election for global operations, and propagate model records.

In the RHOAI/KServe ecosystem, ModelMesh is deployed by the `modelmesh-serving` controller as a sidecar in `ServingRuntime` pods. The controller manages the Kubernetes-level orchestration (CRDs, storage, scaling) while ModelMesh handles the internal model cache management, inter-instance communication, and inference routing.

## Architectural Analysis

ModelMesh employs a sophisticated distributed systems architecture centered around a **distributed LRU cache** pattern. Each ModelMesh instance maintains a local in-memory cache of loaded models (backed by a `ConcurrentLinkedHashMap` with eviction listeners) while coordinating with other instances through a shared KV store. The `ModelMesh` class — at over 5,000 lines — is the core orchestration engine, implementing cache eviction policies, model placement decisions, request routing, and cluster-wide coordination.

The **sidecar pattern** is fundamental: ModelMesh runs as a container alongside the model runtime container in the same pod. Communication between them uses the `ModelRuntime` gRPC service over either a Unix Domain Socket (`/tmp/mmesh/grpc.sock`, preferred when epoll is available) or a localhost TCP port (8085). This design decouples the model serving logic (handled by Triton, MLServer, etc.) from the distributed cache management (handled by ModelMesh). The `SidecarModelMesh` class manages this lifecycle — it waits for the model runtime to report `READY` status, then extracts its capabilities (capacity, loading concurrency, timeout, method info for model ID injection).

A key architectural feature is the **model ID injection mechanism**. When a model runtime reports its `methodInfos` map in the `runtimeStatus` response, ModelMesh learns which protobuf field paths to use for injecting the model ID into inference request messages before forwarding them to the runtime. This is implemented in `ProtoSplicer`, which performs binary protobuf splicing to insert the model ID at the specified field path — avoiding full protobuf deserialization/reserialization for performance. This enables ModelMesh to proxy arbitrary gRPC methods to the model runtime without understanding their schema.

The **VModel (Virtual Model)** subsystem (`VModelManager`) provides zero-downtime model updates by maintaining a mapping layer between logical model IDs and underlying concrete models. When a VModel's target is updated, ModelMesh loads the new target model to the same scale as the current active model before atomically switching traffic — ensuring no inference requests fail during transitions.

The **metrics subsystem** is notable for its custom Prometheus client extensions (in the `com.ibm.watson.prometheus` package). Rather than using the standard Prometheus Java client directly, ModelMesh provides optimized `Counter`, `Gauge`, and `Histogram` implementations. The Prometheus endpoint runs on a dedicated Netty server (port 2112) with self-signed TLS by default, separate from the main gRPC server. StatsD is supported as an alternative metrics backend with both legacy and Sysdig-compatible formats.

The **payload processing framework** (`PayloadProcessor` interface) allows extensible observation of inference payloads. Built-in processors include `LoggingPayloadProcessor` (logs to disk), `RemotePayloadProcessor` (HTTP POST to external endpoint), and `MatchingPayloadProcessor` (filters by model ID pattern). These are composed via `CompositePayloadProcessor` and wrapped in `AsyncPayloadProcessor` for non-blocking execution. This is architecturally important because it means inference request/response data can flow to external systems — a consideration for data governance.
