## Purpose

**Short**: ModelMesh is a distributed LRU cache and routing layer for high-scale, high-density model serving that manages model lifecycle across a cluster of runtime containers.

**Detailed**: ModelMesh is a mature, general-purpose model serving management and routing layer designed for high-scale, high-density, and frequently-changing model use cases. It works with existing or custom-built model servers (e.g., Triton, Seldon MLServer) and acts as a distributed LRU cache for serving runtime models. The framework manages model loading, unloading, scaling (within a deployment), and failover across multiple pod replicas.

ModelMesh runs as a sidecar container alongside a model runtime container within the same pod. It communicates with the runtime container via a local gRPC channel (either TCP on localhost or Unix domain socket) using the `ModelRuntime` gRPC service. Externally, it exposes the `ModelMesh` gRPC API for model registration, status queries, and inference request routing. The framework uses a distributed key-value store (etcd or ZooKeeper) for cluster coordination, model registry persistence, instance state tracking, and leader election.

Within the RHOAI platform, ModelMesh is deployed and managed by the `modelmesh-serving` controller (part of the KServe/ModelMesh Serving component). It provides the core model serving infrastructure for `InferenceService` resources using the ModelMesh backend, enabling multi-model serving where many models share the same set of pods.

## Architectural Analysis

ModelMesh implements a sophisticated distributed caching architecture for model serving. The core design pattern is a **sidecar mesh** where each pod contains a ModelMesh container (the "mesh") alongside a model runtime container. The mesh handles all distributed coordination — which models are loaded where, request routing, cache eviction, scaling — while the runtime container handles actual model inference. This separation of concerns allows ModelMesh to work with any gRPC-compatible model server without modification.

The **distributed LRU cache** is the central architectural concept. Each ModelMesh instance maintains a portion of the overall model cache, with model records persisted in etcd/ZooKeeper. When a request arrives for a model, ModelMesh checks if it's loaded locally; if not, it forwards the request to the instance that has it loaded (a "cache miss" flow). If the model isn't loaded anywhere, ModelMesh selects an instance to load it based on available capacity, load balancing, and type constraints. The LRU eviction policy ensures that infrequently-used models are unloaded to make room for new ones, enabling high-density serving where the total number of models exceeds what any single pod can hold in memory.

The **virtual model (vmodel)** system is a notable design feature that enables atomic model version transitions. A vmodel is an alias that maps to a concrete model ID. When updating a model version, the new model is loaded and validated before the vmodel pointer is atomically swapped, ensuring zero-downtime transitions. This is managed by the VModelManager running on the cluster leader instance. The vmodel system also supports ownership semantics and automatic cleanup of unreferenced concrete models.

Network architecture is multi-layered: the **external gRPC API** (port 8033) handles client-facing operations with configurable TLS and mTLS; the **internal Litelinks Thrift channel** (port 8080) handles inter-pod communication with shared TLS; and the **sidecar gRPC channel** (port 8085 or Unix domain socket) handles runtime communication with no encryption since it's localhost. The Unix domain socket option reduces latency and avoids TCP overhead for the high-throughput inference path. The framework also supports separate ports for management and inference traffic to the runtime container (`INTERNAL_GRPC_PORT` vs `INTERNAL_SERVING_GRPC_PORT`), allowing different max message sizes or connection parameters.

A significant FIPS compliance concern exists: ModelMesh **explicitly disables Java FIPS mode** (`-Dcom.redhat.fips=false`) because it conflicts with TLS functionality. The SunPKCS11 security provider is reconfigured in the Dockerfile, and `security.useSystemPropertiesFile` is set to false. While the container runs on a FIPS-enabled UBI base image, the JVM itself does not operate in FIPS mode. This is documented in the startup script as intentional ("disable java FIPS which breaks when TLS is enabled") but represents a gap for FIPS-mandated deployments. The use of BouncyCastle (not FIPS-validated) for self-signed certificate generation adds another non-FIPS element, though this is only used for the optional Prometheus metrics endpoint.

The **payload processing pipeline** is a well-designed extensibility point. Processors can be chained (CompositePayloadProcessor), run asynchronously (AsyncPayloadProcessor), filtered by model/method (MatchingPayloadProcessor), and can forward payloads to external HTTP services (RemotePayloadProcessor). This enables monitoring, auditing, and debugging without modifying the core inference path. The pipeline is configured via the `MM_PAYLOAD_PROCESSORS` environment variable using a URI-based syntax.
