## Purpose

**Short**: ModelMesh is a distributed LRU cache and routing layer that manages model lifecycle and inference routing for colocated model runtime containers in a Kubernetes-based model serving deployment.

**Detailed**: ModelMesh acts as a sidecar container that runs alongside model runtime containers (such as Triton Inference Server, Seldon MLServer, or custom model servers). It provides intelligent model placement, caching, and routing across a cluster of pods. When an inference request arrives, ModelMesh determines which instance has the requested model loaded, routes the request appropriately, and handles model loading/unloading transparently.

The framework uses a key-value store (etcd or ZooKeeper) to maintain a distributed registry of model locations and states across the cluster. Each ModelMesh instance manages a local LRU cache of loaded models, automatically evicting least-recently-used models when capacity is reached and loading models on demand. Inter-instance communication uses an internal Thrift-based protocol via the Litelinks framework, while external clients interact through gRPC.

ModelMesh also supports "vmodels" (virtual models) — aliases that map to concrete model IDs, enabling blue-green deployments and model version transitions without client changes. The system includes comprehensive Prometheus and StatsD metrics, TLS support for both external and internal communication, payload processing hooks, and graceful shutdown with model propagation to other instances.

## Architectural Analysis

ModelMesh implements a sophisticated distributed model serving cache that solves a fundamental challenge in ML serving infrastructure: how to efficiently serve thousands of models across a cluster of pods with limited GPU/CPU memory per pod. The core design is a distributed LRU cache where models are dynamically loaded and evicted based on request patterns, with intelligent routing to minimize cache misses and loading latency.

The architecture follows a sidecar pattern where ModelMesh runs alongside model runtime containers in the same pod. Communication between ModelMesh and the model runtime occurs over localhost gRPC (port 8085) or Unix Domain Sockets, keeping this high-frequency data path fast and avoiding network overhead. The separation of concerns is clean: ModelMesh handles routing, caching, and lifecycle management while the model runtime handles actual model loading and inference. The ModelRuntime gRPC interface (`loadModel`, `unloadModel`, `predictModelSize`, `modelSize`, `runtimeStatus`) is the contract between these two components.

Inter-instance communication uses IBM's Litelinks framework, a Thrift-based RPC layer that provides service discovery via the shared KV store. When a model is not loaded locally, ModelMesh can forward the inference request to another instance that has the model loaded, transparently handling the cache miss. The framework also supports latency-based autoscaling when `limitModelConcurrency` is enabled, where request queuing time drives scaling decisions. The system includes extensive handling of edge cases: unload retries with exponential backoff, reference-counted model states to handle concurrent load/unload races, and graceful shutdown with model propagation to surviving instances.

A notable design decision is the Java FIPS bypass (`-Dcom.redhat.fips=false`). The start.sh script explicitly disables Red Hat's Java FIPS mode because it conflicts with TLS functionality. The security provider chain is also modified — SunPKCS11 is reordered in java.security configuration. Additionally, the Prometheus metrics server uses BouncyCastle to generate self-signed certificates for HTTPS, and the gRPC transport uses Netty's tcnative-boringssl-static for native TLS offload. These choices prioritize performance and compatibility over FIPS compliance and represent a known gap for environments requiring strict FIPS validation.

The build pipeline uses Red Hat's PNC (Product Newcastle) system for Maven artifact builds rather than building from source in the Dockerfile. The Konflux Dockerfile consumes a pre-built ZIP artifact from PNC via Hermeto (cachi2) prefetch, pins it with `artifacts.lock.yaml`, and layers it onto a `ubi9/openjdk-21-runtime` base image. RPM dependencies are locked via `rpms.lock.yaml`. This two-stage approach (PNC for Java compilation, Konflux for container image) achieves full build hermeticity while keeping the Dockerfile simple.

The vmodel (virtual model) subsystem deserves attention as an architectural feature. VModels provide an indirection layer that enables blue-green model deployments: a vmodel alias can be atomically transitioned from one concrete model to another, with the system ensuring the new model is loaded to the same scale as the old one before completing the transition. This is managed through the `VModelManager` class and persisted in the KV store, making it consistent across the cluster.
