## Purpose

**Short**: A Kubernetes operator that manages the lifecycle of LLM-D batch gateway deployments by reconciling `LLMBatchGateway` custom resources into Kubernetes resources rendered from an embedded Helm chart.

**Detailed**: The LLM-D Batch Gateway Operator automates the deployment and lifecycle management of the [batch-gateway](https://github.com/opendatahub-io/batch-gateway) system, which provides OpenAI-compatible batch inference API endpoints for large language models. The operator watches `LLMBatchGateway` custom resources and uses Helm chart rendering at runtime to produce the full set of Kubernetes resources (Deployments, Services, ConfigMaps, ServiceAccounts, HTTPRoutes, Certificates, and monitoring resources).

The batch gateway system consists of three components managed by this operator: an HTTP API server that accepts batch inference requests and file uploads, a processor that dispatches inference requests to backend LLM inference gateways (such as vLLM or llm-d inference endpoints), and a garbage collector that cleans up expired jobs and files. The operator supports both S3-compatible object storage and PVC-backed filesystem storage, integrates with cert-manager for TLS certificate provisioning, supports Gateway API HTTPRoute for ingress exposure, and provides Prometheus monitoring via ServiceMonitor/PodMonitor/PrometheusRule resources.

A key design pattern is that component container images are NOT specified in the CR; they are injected into the operator's Deployment via environment variables sourced from `params.env` by the platform operator (opendatahub-operator/rhods-operator). This ensures the platform -- not the end user -- controls which images are deployed, enabling centralized image pinning and digest-based references for supply chain security.

## Architectural Analysis

The LLM-D Batch Gateway Operator follows a Helm-as-library pattern that is uncommon in the Kubernetes operator ecosystem. Rather than using kustomize or raw Go templates, it embeds a full Helm chart (`prefetched-charts/batch-gateway/`) inside the operator container image and renders it at runtime using the Helm v3 engine library. The `specToHelmValues()` function in `internal/controller/helm.go` acts as a translation layer between the typed CRD spec and the chart's `values.yaml` schema. This approach gives the operator the full expressiveness of Helm templating while maintaining a strongly-typed CRD API -- users never interact with Helm directly.

The operator implements a sophisticated cross-namespace secret management pattern using Gateway API ReferenceGrant. When a user specifies a secret in a different namespace via `spec.secretRef.namespace`, the operator checks for a ReferenceGrant in the secret's namespace that explicitly permits the LLMBatchGateway (from group `batch.llm-d.ai`) in the CR's namespace to reference that secret. If permitted, it creates a controller-owned copy of the secret in the CR's namespace, annotated with `batch.llm-d.ai/copied-from` for immutability tracking. This prevents secret sprawl while maintaining namespace isolation. The `secretRef` field is immutable after creation -- changing it requires deleting and recreating the CR. The `secretWatchFilter` ensures the cluster-wide Secret informer only triggers reconciles for secrets the controller actually tracks, preventing unnecessary work.

The image pinning architecture is deliberately decoupled from the CR. Component images (`apiserver`, `processor`, `gc`) are read from environment variables on the operator pod itself (lines 35-37 of `cmd/main.go`), which are injected by the platform operator via kustomize replacements from `params.env`. This means the CRD spec has no image fields -- the platform controls which images run, enabling centralized digest pinning for supply chain security across the fleet. The operator fails to start if any image environment variable is missing, treating it as a hard configuration error.

The orphan deletion mechanism in `deleteOrphanedResources` deserves attention. The controller maintains a static list of all GVK types the Helm chart can produce (`managedGVKs`), and after each reconcile, it lists all resources of those types in the CR's namespace, identifies those owned by the CR, and deletes any that are no longer in the rendered output. This is critical for spec changes that remove optional resources (e.g., disabling HTTPRoute or ServiceMonitor) -- without it, previously-created resources would be left orphaned. The list must be manually updated when new resource types are added to the chart. Currently, the orphan check does a full namespace list for each GVK (with a TODO to optimize via label selectors), which could become expensive in namespaces with many resources.

The operator registers two controllers: the main `LLMBatchGatewayReconciler` and a `MetricsController`. The MetricsController is responsible for the operator's own observability infrastructure -- it creates and self-heals a metrics Service (`llm-d-batch-gateway-operator-metrics`), ServiceMonitor, and PrometheusRule with alerts for high reconcile error rate and leader election loss. This controller bootstraps on startup via a raw source and watches its own resources for deletion/modification, ensuring the monitoring stack is always present. It only runs when `POD_NAMESPACE` is set.
