## Purpose

**Short**: A small HTTP reverse proxy that enforces Kubernetes RBAC authorization against the API server using SubjectAccessReview before forwarding requests to an upstream service.

**Detailed**: kube-rbac-proxy is a sidecar container that sits in front of application containers to enforce authentication and authorization on incoming HTTP requests. It authenticates callers via Kubernetes TokenReview (bearer tokens), client TLS certificates, or OpenID Connect (OIDC) JWT tokens. Once authenticated, it authorizes requests by performing SubjectAccessReview calls against the Kubernetes API server, verifying that the caller has the required RBAC permissions before proxying the request to the upstream application.

In the RHOAI platform (3.x), kube-rbac-proxy is deployed as a sidecar container alongside component pods to replace the older oauth-proxy pattern. The platform operator (rhods-operator) injects kube-rbac-proxy sidecars into component deployments, typically listening on port 8443/TCP with TLS, and proxying authenticated/authorized requests to the application container on its localhost port. This provides per-component RBAC enforcement without requiring each component to implement its own authentication logic.

The proxy also includes a hardcoded authorizer that unconditionally allows the OpenShift monitoring Prometheus service account (`system:serviceaccount:openshift-monitoring:prometheus-k8s`) to scrape `/metrics` endpoints, ensuring metrics collection works without additional RBAC configuration.

## Architectural Analysis

kube-rbac-proxy implements a clean separation between authentication and authorization through a middleware chain pattern. The HTTP handler pipeline in `cmd/kube-rbac-proxy/app/kube-rbac-proxy.go` (lines 292-330) stacks filters in order: path allow-list filtering, authentication, authorization, auth header injection, and finally reverse proxy forwarding. This middleware chain is the core architectural pattern of the entire codebase.

The authentication layer supports three mutually exclusive modes: delegating (TokenReview), OIDC, and x509 client certificates. The choice is made at startup based on whether `--oidc-issuer` is configured (lines 227-247). The delegating authenticator wraps the Kubernetes apiserver's `authenticatorfactory.DelegatingAuthenticatorConfig`, which handles TokenReview API calls with a 2-minute cache TTL and retry backoff. The OIDC authenticator uses the Kubernetes apiserver's OIDC plugin (`k8s.io/apiserver/plugin/pkg/authenticator/token/oidc`) with configurable claim mappings for username and groups.

The authorization layer uses a union authorizer that chains three authorizers in priority order (lines 260-267): (1) a hardcoded metrics authorizer that unconditionally allows the `system:serviceaccount:openshift-monitoring:prometheus-k8s` service account to GET `/metrics`, (2) a static authorizer driven by the `--config-file` configuration, and (3) the SubjectAccessReview authorizer that delegates to the Kubernetes API server. This design ensures metrics scraping always works regardless of RBAC configuration, which is critical for platform observability. The hardcoded authorizer in `pkg/hardcodedauthorizer/metrics.go` is copied from OpenShift library-go to avoid a hard dependency.

The `SanitizingFilter` in `cmd/kube-rbac-proxy/app/sanitazion.go` is a security-relevant component that masks bearer tokens in log output. It intercepts all klog output and redacts the `Spec.Token` field from serialized `TokenReview` objects, preventing credential leakage through logs. This is implemented as a klog `LogFilter` that detects `"kind":"TokenReview"` strings in log arguments.

The TLS configuration is production-grade: certificates can be hot-reloaded via the `CertReloader` in `pkg/tls/reloader.go`, which polls cert files at a configurable interval (default 1 minute). The default TLS minimum version is 1.2, and cipher suites are configurable through the Kubernetes component-base TLS parsing utilities, which enforce FIPS-compatible defaults when running with `strictfipsruntime`. Self-signed certificates are auto-generated as a fallback when no cert files are provided, though this triggers a deprecation warning as insecure operation is being phased out.
