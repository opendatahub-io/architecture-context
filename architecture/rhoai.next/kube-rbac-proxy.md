## Purpose

**Short**: An HTTP reverse proxy that enforces Kubernetes RBAC authorization via SubjectAccessReview before forwarding requests to an upstream service.

**Detailed**: kube-rbac-proxy is a small HTTP/HTTPS reverse proxy designed to sit in front of any single upstream service and enforce Kubernetes RBAC-based authentication and authorization on every incoming request. It authenticates requests using either delegating token authentication (via the Kubernetes TokenReview API) or OIDC JWT validation, then authorizes them by performing SubjectAccessReview calls against the Kubernetes API server. Only requests that pass both authentication and authorization checks are proxied to the upstream service.

In the RHOAI platform, kube-rbac-proxy is deployed as a sidecar container alongside component pods (e.g., notebooks, model servers, dashboard backends). It listens on a TLS-secured port (typically 8443/TCP), authenticates incoming bearer tokens or client certificates, checks authorization via Kubernetes RBAC, and proxies approved requests to the application container listening on localhost. This is the primary authentication/authorization enforcement mechanism in RHOAI 3.x, replacing the oauth-proxy sidecar used in RHOAI 2.x.

The proxy supports two authorization configuration formats: Format1 (top-level resource attributes with optional query/header rewrites for templated SubjectAccessReview fields) and Format2 (path-scoped endpoint rules with per-HTTP-method resource mappings). It also includes a hardcoded authorizer that always permits the `system:serviceaccount:openshift-monitoring:prometheus-k8s` service account to GET `/metrics`, ensuring Prometheus metrics scraping works without additional RBAC configuration.

## Architectural Analysis

kube-rbac-proxy implements a clean separation between authentication, authorization, and proxying via an HTTP handler middleware chain. The request flow is: `WithAllowPaths` -> `WithAuthentication` -> `WithAuthorization` -> `WithAuthHeaders` -> `proxy.ServeHTTP`. Each layer is a composable `http.HandlerFunc` wrapper, making the security enforcement pipeline easy to reason about. The `--ignore-paths` flag allows bypassing the entire authn/authz chain for specific paths, while `--allow-paths` restricts which paths the proxy will accept at all (returning 404 for non-matching paths).

The authorization system supports two complementary configuration formats. Format1 provides top-level `resourceAttributes` with optional `rewrites` that template SubjectAccessReview fields from query parameters or HTTP headers (e.g., `namespace: "{{ .Value }}"`). Format2 adds path-scoped `endpoints` with per-HTTP-method `mappings`, each containing its own resource rules and rewrites. When a request matches a Format2 endpoint path, Format1 rules are entirely bypassed for that request. This dual-format design allows both simple use cases (single resource check for all requests) and complex multi-path APIs (different RBAC rules per endpoint and method), as demonstrated by the `EndpointAttributesFromRequest` function in `pkg/authz/endpoints.go`.

The hardcoded metrics authorizer (`pkg/hardcodedauthorizer/metrics.go`) is a notable design choice specific to the OpenShift/RHOAI deployment context. It unconditionally allows the `system:serviceaccount:openshift-monitoring:prometheus-k8s` service account to perform non-resource GET requests on `/metrics`. This ensures monitoring works out of the box without requiring explicit RBAC configuration for every component that uses kube-rbac-proxy as a sidecar. The authorizer is positioned first in the union authorizer chain (before static and SAR authorizers), so the Prometheus service account gets fast-path authorization without API server round-trips.

TLS handling is robust with several noteworthy patterns. The server enforces a configurable minimum TLS version (default TLS 1.2), supports configurable cipher suites, and includes a `CertReloader` that watches certificate files on disk and hot-reloads them without restarts. The upstream transport also enforces TLS 1.2+ minimum and negotiates HTTP/2 via ALPN (`NextProtos: ["h2", "http/1.1"]`). The `--upstream-force-h2c` flag enables cleartext HTTP/2 for upstream services that speak h2c only (e.g., insecure gRPC servers). The log sanitizer (`SanitizingFilter`) intercepts klog output to mask bearer tokens in TokenReview log entries, preventing accidental credential leakage in logs.

The FIPS compliance posture is strong. The Konflux Dockerfile builds with `GOEXPERIMENT=strictfipsruntime`, `CGO_ENABLED=1`, and `-tags=strictfipsruntime`, ensuring the Go binary links dynamically against OpenSSL for all crypto operations. At the application level, all TLS configurations enforce MinVersion TLS 1.2+, no non-FIPS crypto algorithms (MD5, RC4, DES) are used anywhere in the codebase, and `InsecureSkipVerify` is never set.
