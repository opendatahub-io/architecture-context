## Purpose

**Short**: An HTTP reverse proxy sidecar that enforces Kubernetes RBAC authorization via SubjectAccessReview before forwarding requests to an upstream service.

**Detailed**: kube-rbac-proxy is a small HTTP reverse proxy designed to run as a sidecar container alongside application pods. It intercepts incoming HTTP/HTTPS requests, authenticates the caller using either Kubernetes TokenReview (bearer tokens), client TLS certificates, or OIDC JWT tokens, and then authorizes the request by performing a SubjectAccessReview against the Kubernetes API server. Only requests from authenticated and authorized entities are proxied to the upstream application.

In the RHOAI (Red Hat OpenShift AI) platform, kube-rbac-proxy serves as the primary authentication and authorization sidecar for component pods in the RHOAI 3.x architecture. It replaces the older oauth-proxy sidecar pattern from RHOAI 2.x. The platform operator (`rhods-operator`) injects kube-rbac-proxy containers into component deployments, typically listening on port 8443/TCP with TLS, and proxying authenticated requests to the application container on an internal port. This provides fine-grained Kubernetes RBAC-based access control for component endpoints (metrics, APIs, dashboards) without requiring each component to implement its own authentication logic.

The proxy also supports a hardcoded authorizer that automatically allows the `system:serviceaccount:openshift-monitoring:prometheus-k8s` service account to scrape `/metrics` endpoints, ensuring seamless integration with the OpenShift monitoring stack without additional RBAC configuration.

## Architectural Analysis

kube-rbac-proxy follows a straightforward reverse-proxy-with-middleware architecture. The request processing pipeline is composed as a chain of HTTP handler functions: `WithAllowPaths` → `WithAuthentication` → `WithAuthorization` → `WithAuthHeaders` → `httputil.ReverseProxy.ServeHTTP`. This functional composition pattern (defined in `pkg/filters/auth.go` lines 33-125) is clean and testable — each filter is a pure function transforming an `http.HandlerFunc` by wrapping it with additional logic. The `--ignore-paths` flag short-circuits the auth chain, allowing matching paths to bypass authentication and authorization entirely, which is useful for health endpoints or public content.

The authentication layer supports three mutually exclusive strategies selected at startup: delegating (Kubernetes TokenReview, the default), OIDC (JWT validation against an external issuer), or X.509 client certificates. The delegating authenticator (in `pkg/authn/delegating.go`) uses the Kubernetes apiserver's `authenticatorfactory.DelegatingAuthenticatorConfig`, which provides built-in caching (2-minute TTL), retry backoff, and audience validation. The OIDC authenticator (`pkg/authn/oidc.go`) uses the apiserver's OIDC plugin with support for configurable claims mapping (username, groups), signing algorithm selection, and dynamic CA certificate reloading. Both authenticators support hot-reloading of CA certificates via `dynamiccertificates.DynamicFileCAContent`, which watches files on a goroutine.

The authorization layer uses a union authorizer that chains three authorizers in priority order: (1) a hardcoded metrics authorizer that unconditionally allows `system:serviceaccount:openshift-monitoring:prometheus-k8s` to GET `/metrics` — this is OpenShift-specific and ensures Prometheus can always scrape component metrics without additional RBAC setup; (2) a static authorizer loaded from a config file that matches requests against a list of user/verb/resource rules without making API calls; and (3) the SAR (SubjectAccessReview) authorizer that delegates to the Kubernetes API server. The SAR authorizer caches allow decisions for 5 minutes and deny decisions for 30 seconds (`pkg/authz/auth.go` lines 93-99), reducing API server load for repeated requests from the same user.

The TLS implementation (`pkg/tls/reloader.go`) supports hot-reloading of server certificates by polling files at a configurable interval (default 1 minute, configurable via `--tls-reload-interval`). When cert files change on disk, the new certificate is atomically swapped via a read-write mutex, allowing zero-downtime certificate rotation. The proxy also supports configurable TLS minimum version (default TLS 1.2), cipher suite selection, and optional HTTP/2 disabling (added in v0.15.0 as a mitigation for CVE-2023-44487). The server runs as UID 65534 (nobody) in the container, following security best practices.

A notable security feature is the `SanitizingFilter` (in `cmd/kube-rbac-proxy/app/sanitazion.go`) that intercepts all klog output and masks bearer tokens in TokenReview log messages. This prevents credential leakage through verbose logging — any log line containing a TokenReview JSON payload has its `spec.token` field replaced with `<masked>`. This is particularly important in production environments where debug-level logging might be temporarily enabled.
