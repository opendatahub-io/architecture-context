## Purpose

**Short**: An HTTP reverse proxy that enforces Kubernetes RBAC authorization via SubjectAccessReview before forwarding requests to an upstream service.

**Detailed**: kube-rbac-proxy is a small HTTP reverse proxy designed to sit in front of any single upstream service and enforce Kubernetes RBAC authorization on every incoming request. It intercepts HTTP/HTTPS traffic, authenticates the caller (via delegated token review against the Kubernetes API, OIDC JWT validation, or x509 client certificates), then authorizes the request by issuing a SubjectAccessReview against the Kubernetes API server. Only requests from authenticated and authorized identities are forwarded to the upstream service.

In the RHOAI platform (3.x+), kube-rbac-proxy is deployed as a sidecar container alongside component pods (e.g., notebooks, model servers, dashboard backends). The platform operator (rhods-operator) injects kube-rbac-proxy containers into pod templates, configuring them to listen on port 8443/TCP with TLS and proxy authorized traffic to the application container on its localhost port. This replaces the earlier oauth-proxy sidecar pattern used in RHOAI 2.x, providing fine-grained Kubernetes RBAC-based access control rather than OpenShift OAuth-based authentication.

The proxy supports two authorization configuration formats: Format1 (top-level resourceAttributes with optional query/header rewrites) for simple use cases, and Format2 (path-scoped endpoint rules with per-method resource mappings) for fine-grained per-endpoint authorization. It also includes a hardcoded authorizer that always permits the OpenShift Monitoring Prometheus service account to scrape `/metrics`.

## Architectural Analysis

kube-rbac-proxy follows a clean layered architecture with distinct packages for authentication, authorization, and HTTP filtering. The authentication layer supports three pluggable strategies: delegated Kubernetes TokenReview (the default and most common in RHOAI), OIDC JWT validation, and x509 client certificate authentication. These are mutually exclusive at the top level -- if an OIDC issuer URL is configured, the OIDC authenticator is used; otherwise, delegating authentication via Kubernetes API is used. The x509 client CA can be layered on top of either mode.

The authorization layer uses a union authorizer pattern that chains three authorizers in priority order: a hardcoded metrics authorizer (always permits Prometheus to scrape `/metrics`), a static authorizer (matches against config-file rules without API calls), and the SAR authorizer (issues SubjectAccessReview to the Kubernetes API). This chain means metrics scraping never hits the API server, reducing latency and load. The authorization configuration supports two formats: Format1 for simple resource-attribute-based SAR with optional query/header rewrites, and Format2 for path-scoped endpoint rules where each URL pattern can map different HTTP methods to different SAR resource attributes. Format2 is particularly useful for RHOAI components like TrustyAI that need different RBAC rules per API endpoint.

The TLS implementation is carefully designed for production use. The `CertReloader` in `pkg/tls/reloader.go` hot-reloads server certificates at configurable intervals (default 1 minute) using byte-level comparison to avoid unnecessary reloads. The upstream transport in `cmd/kube-rbac-proxy/app/transport.go` enforces TLS 1.2 minimum and sets `NextProtos: ["h2", "http/1.1"]` for proper ALPN negotiation, preventing potential protocol downgrade attacks. The proxy also supports HTTP/2 disable mode (motivated by CVE-2023-44487, the HTTP/2 Rapid Reset attack) which sets `TLSNextProto` to an empty map and restricts ALPN to `http/1.1` only.

A notable security feature is the `SanitizingFilter` in `cmd/kube-rbac-proxy/app/sanitazion.go` which intercepts klog output, detects `TokenReview` JSON payloads, and masks the bearer token before it reaches log storage. This prevents credential leakage through log aggregation systems, which is a common security concern in Kubernetes environments. The filter is installed globally as a klog `LogFilter` in the `PersistentPreRunE` hook.

The proxy runs as non-root (UID 65534) and is built with FIPS-compliant crypto (`GOEXPERIMENT=strictfipsruntime`, `CGO_ENABLED=1`). The FIPS compliance is validated in CI via a dedicated GitHub Actions workflow that builds with `Dockerfile.redhat` and runs `check-payload scan` against the container filesystem. The FIPS build applies to both the Konflux production image (`Dockerfile.konflux`) and the Red Hat downstream image (`Dockerfile.redhat`), while the upstream community `Dockerfile.ocp` uses different base images without FIPS flags.
