## Purpose

**Short**: An HTTP reverse proxy sidecar that enforces Kubernetes RBAC authorization on incoming requests via SubjectAccessReview before forwarding them to an upstream service.

**Detailed**: kube-rbac-proxy is a small HTTP/HTTPS reverse proxy designed to run as a sidecar container alongside application pods. It intercepts incoming requests, authenticates callers using either Kubernetes TokenReview, client TLS certificates, or OIDC JWT tokens, and then authorizes the request by performing a Kubernetes SubjectAccessReview against the API server. Only authenticated and authorized requests are forwarded to the upstream application (typically a metrics endpoint bound to localhost).

In the RHOAI platform (3.x), kube-rbac-proxy serves as the primary authentication/authorization sidecar injected by the platform operator (rhods-operator) into component pods. It replaces the earlier oauth-proxy sidecar pattern from RHOAI 2.x. The proxy typically listens on port 8443/TCP with TLS, fronting application containers that listen on localhost-only ports (e.g., 8080, 8081). This architecture ensures that no unauthenticated network traffic can reach component services, even within the cluster, without presenting a valid bearer token or client certificate that passes Kubernetes RBAC policy.

The proxy also includes a hardcoded authorization rule that always allows the OpenShift monitoring Prometheus service account (`system:serviceaccount:openshift-monitoring:prometheus-k8s`) to scrape `/metrics`, ensuring monitoring infrastructure always has access. It supports two authorization configuration formats: Format1 (top-level resourceAttributes with optional rewrites from query parameters or HTTP headers) and Format2 (path-scoped endpoint rules with per-method resource mappings), enabling fine-grained authorization policies per URL path and HTTP method.

## Architectural Analysis

kube-rbac-proxy follows a clean layered architecture with distinct separation between authentication, authorization, and proxying concerns. The HTTP middleware chain in `pkg/filters` composes these layers: path filtering (allow/ignore paths) wraps the outermost layer, followed by authentication, authorization, auth header injection, and finally the reverse proxy handler. This composable design makes it straightforward to understand the request lifecycle and audit the security boundary.

The proxy supports two authorization configuration formats that serve different use cases. Format1 (top-level `resourceAttributes` with optional `rewrites`) is simpler and maps all requests to a single SAR resource pattern, making it ideal for protecting single-purpose endpoints like `/metrics`. Format2 (`endpoints` with path-scoped `mappings`) enables fine-grained per-path, per-method authorization rules with template expansion from HTTP headers and query parameters. This dual-format system was introduced to support components like TrustyAI that need different RBAC policies for different API paths. The Format2 implementation uses Go `text/template` for field expansion, which introduces a minor injection risk if rewrite values from untrusted headers are not carefully constrained — though the template scope is limited to ResourceAttributes fields.

The hardcoded metrics authorizer (`pkg/hardcodedauthorizer/metrics.go`) is a notable design choice: it unconditionally allows the OpenShift monitoring Prometheus service account to read `/metrics` regardless of RBAC configuration. This ensures monitoring always works but creates a non-configurable trust relationship. The comment in the source indicates this is intended to be temporary until Cluster Monitoring Operator (CMO) supports static authorizer configuration, but it has persisted across many versions.

TLS certificate hot-reloading (`pkg/tls/reloader.go`) uses a polling approach with configurable interval (default 1 minute) rather than filesystem event watching (inotify). This is simpler and more portable but introduces a window where the old certificate is served after renewal. The `--tls-reload-interval` flag is marked deprecated, suggesting a future move to event-based reloading. The log sanitization filter (`cmd/app/sanitazion.go`) that masks tokens in TokenReview log entries is a security-conscious addition from the Red Hat Data Services fork, preventing credential leakage through verbose logging.

The proxy runs as a non-root user (UID 65534 in the Dockerfile) and the example deployments set `allowPrivilegeEscalation: false`, following least-privilege container security practices. The Konflux build pipeline produces multi-arch images (x86_64, arm64, ppc64le, s390x) with FIPS-compliant crypto (strictfipsruntime), hermetic builds, and source image generation — aligning with Red Hat's supply chain security requirements.
