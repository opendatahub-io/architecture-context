## Purpose

**Short**: Feast is an open-source feature store for machine learning that manages the storage, retrieval, and serving of features for model training and online inference.

**Detailed**: Feast (Feature Store) provides a unified platform for managing ML features across offline (batch) and online (real-time) environments. The component consists of a Kubernetes operator (`feast-operator`) that automates deployment and lifecycle management of feature store instances, and a Python-based feature server that serves features via REST and gRPC APIs. The operator manages the full stack of Feast services including offline stores (for historical feature retrieval), online stores (for low-latency serving), a metadata registry (tracking feature definitions, data sources, and permissions), and an optional web UI.

The Feast operator is built with controller-runtime and watches a single CRD (`FeatureStore` in the `feast.dev` API group). When a FeatureStore CR is created, the operator reconciles a comprehensive set of Kubernetes resources: a multi-container Deployment running the feature server image, Services for each Feast component (offline, online, registry gRPC, registry REST, UI), PersistentVolumeClaims for file-based persistence, RBAC resources for authorization, CronJobs for scheduled materialization, and optional OpenShift Routes for external access. The operator supports two authorization models (Kubernetes RBAC and OIDC), automatic TLS via OpenShift serving certificates, HPA-based autoscaling, and integration with OpenDataHub Notebooks.

The Python feature server (deployed as the `feature-server` container image) runs the Feast SDK with FastAPI for REST APIs and gRPC for serving. It supports a wide range of backend stores (Redis, PostgreSQL, Snowflake, DuckDB, etc.) and includes Model Context Protocol (MCP) integration, data quality monitoring, and Prometheus metrics export.

## Architectural Analysis

The Feast component follows a two-tier architecture: a Go-based Kubernetes operator manages the control plane, while a Python-based feature server provides the data plane. This separation is clean — the operator handles all Kubernetes resource lifecycle and platform integration, while the feature server focuses purely on feature store functionality. The operator embeds significant intelligence: it auto-detects OpenShift, provisions TLS certificates via service annotations, creates comprehensive RBAC hierarchies, and implements a periodic auto-access RBAC reconciliation loop that fetches permission policies from the running registry and creates corresponding Kubernetes RBAC resources.

The single-Deployment, multi-container pattern is noteworthy. Rather than deploying separate Deployments for each Feast service (offline, online, registry, UI), the operator creates one Deployment with up to four containers sharing the same Pod. This simplifies networking (services can communicate via localhost), but it means scaling affects all services equally. The v1 API added HPA support with autoscaling, PodDisruptionBudgets, topology spread constraints, and pod anti-affinity — but these apply to the entire Deployment, not individual services. This design trades service-level scaling granularity for operational simplicity.

The authorization model is comprehensive but adds complexity. Two auth modes (Kubernetes RBAC and OIDC) are supported with mutual exclusivity enforced via CEL validation. The operator creates cluster-scoped RBAC resources (ClusterRoles, ClusterRoleBindings) for token review and namespace discovery, which means a single FeatureStore CR has cluster-wide RBAC impact. The auto-access RBAC feature is particularly interesting: the operator periodically calls the registry REST API using a self-generated HMAC-SHA256 JWT token, fetches permission policies, and translates them into Kubernetes RBAC resources. This creates a bidirectional dependency between the operator and the feature server — the operator deploys the feature server, but also depends on it for permission data.

The FIPS posture is asymmetric between components. The operator Konflux Dockerfile correctly specifies `CGO_ENABLED=1`, `GOEXPERIMENT=strictfipsruntime`, and `-tags strictfipsruntime` for FIPS-compliant Go crypto. However, the upstream operator Dockerfile uses `CGO_ENABLED=0` (static linking), which would fail check-payload. The feature server relies on UBI9's OpenSSL for FIPS compliance at the OS level, but the Python code itself uses standard crypto libraries (PyJWT, etc.) without explicit FIPS configuration. The `InsecureSkipVerify: true` in the operator's registry client is an intentional trade-off for cluster-internal communication but should be documented as a known FIPS gap.

The CRD has evolved significantly between v1alpha1 and v1. The v1 API added materialization configuration, OpenLineage data lineage tracking, batch engine configuration, data quality monitoring, MCP (Model Context Protocol) integration, scaling with HPA, PodDisruptionBudgets, topology constraints, and serving metrics configuration. The v1alpha1 API is explicitly deprecated but still supported. CEL validation rules enforce complex constraints like mutual exclusivity between static replicas and autoscaling, and requirement for DB-backed persistence when scaling is enabled (file-based SQLite/DuckDB doesn't support concurrent access).
