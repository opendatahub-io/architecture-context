## Purpose

**Short**: Feast is a feature store platform that provides an operator to deploy and manage feature store instances for machine learning workloads on OpenShift.

**Detailed**: Feast (Feature Store) is an open-source feature store that serves as the bridge between data engineering and machine learning. The Red Hat distribution packages Feast as two primary components: a Go-based Kubernetes operator (`feast-operator`) that manages the lifecycle of FeatureStore custom resources, and a Python-based feature server that provides REST and gRPC APIs for online feature retrieval, feature ingestion, and materialization.

The operator watches FeatureStore CRDs and reconciles the desired state by creating Deployments, Services, ConfigMaps, PVCs, RBAC resources, CronJobs, and (on OpenShift) Routes and ServiceMonitors. It supports multiple persistence backends (PostgreSQL, Redis, Snowflake, BigQuery, DuckDB, S3/GCS, and more), horizontal pod autoscaling, TLS via OpenShift service-serving certificates, and two authorization modes: Kubernetes RBAC and OIDC. The operator also integrates with OpenDataHub/RHOAI Notebooks by automatically injecting feature store client configuration into notebook pods.

The feature server runs as a FastAPI application with Gunicorn/Uvicorn workers, exposing REST endpoints for online feature retrieval, feature push, and materialization. It includes a gRPC registry server and a REST registry server for metadata management, with support for OIDC and Kubernetes-based authentication and per-request RBAC enforcement.

## Architectural Analysis

The Feast component follows a two-tier architecture: a cluster-scoped Go operator that manages instance lifecycle, and per-instance Python feature servers that handle data plane traffic. This separation allows the operator to be lightweight (256Mi memory limit) while the feature servers scale independently based on workload demands. The operator uses controller-runtime's label-filtered caching to minimize memory footprint in large clusters, only watching resources tagged with `app.kubernetes.io/managed-by: feast-operator`.

The authorization system is notably sophisticated for a feature store. Feast supports two mutually exclusive auth modes: Kubernetes RBAC (where the operator creates Roles and RoleBindings, and the feature server validates tokens via the Kubernetes TokenReview API) and OIDC (where the server validates JWTs against an external OIDC provider). The "auto-access RBAC" subsystem is particularly interesting -- it fetches permission policies from the Feast registry REST API and automatically creates ClusterRoleBindings for groups defined in those policies, enabling a self-service permission model where data scientists define access in Feast and the operator propagates it to Kubernetes RBAC.

The notebook integration pattern (`NotebookConfigMapReconciler`) demonstrates tight coupling with the OpenDataHub/RHOAI platform. When a Notebook CR has the `opendatahub.io/feast-integration: "true"` label and a comma-separated list of project names in the `opendatahub.io/feast-config` annotation, the operator automatically creates a ConfigMap containing `feature_store.yaml` entries for each referenced project. This ConfigMap is mounted into the notebook pod, allowing notebook users to access feature stores without manual configuration. The controller dynamically detects the kubeflow.org Notebook CRD at startup and polls until it becomes available, then restarts the process to register the controller -- a pragmatic approach to optional CRD dependencies.

The TLS architecture leverages OpenShift service-serving certificates extensively. On OpenShift clusters, the operator annotates Services with `service.beta.openshift.io/serving-cert-secret-name`, triggering automatic TLS certificate provisioning. A CA bundle ConfigMap (`feast-{name}-ca-bundle`) is also created with the `service.beta.openshift.io/inject-cabundle` annotation for client-side certificate validation. HTTP/2 is disabled by default in the operator to mitigate CVE-GHSA-qppj-fm5r-hxr3, and the feature server uses Gunicorn pre-fork workers (not threads) to avoid GIL contention during CPU-bound feature transformations. The `OfflineWriteBatcher` in the feature server provides an asynchronous write path for offline stores, returning 202 ACCEPTED and buffering writes by size and time thresholds.

A namespace-level registry (`feast-configs-registry` ConfigMap) maintained by the operator enables cross-namespace FeatureStore discovery, which supports the remote registry pattern where one FeatureStore references another's registry via `FeastRef`. The operator's remote registry cross-reconciliation logic ensures that changes to a local registry trigger re-reconciliation of all FeatureStores referencing it, maintaining consistency across the cluster.
