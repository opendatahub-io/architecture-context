## Purpose

**Short**: Feast is a feature store platform that manages the lifecycle of machine learning features, providing feature registration, storage, serving, and materialization through a Kubernetes operator and pluggable feature server.

**Detailed**: Feast (Feature Store) enables data scientists and ML engineers to define, store, and serve features for machine learning models. The platform consists of three main layers: a Kubernetes operator (`feast-operator`) that automates deployment and lifecycle management of Feast services; a Python SDK that provides the core feature store logic including feature definitions, registry management, offline/online store abstractions, and REST/gRPC/Arrow Flight serving APIs; and a Go feature server that provides high-performance online feature retrieval with support for multiple storage backends.

The Feast operator watches `FeatureStore` custom resources and reconciles the desired state by deploying feature servers (online stores), offline stores, registry servers, and optional UI components. It handles TLS provisioning (automatic on OpenShift via service-serving certificates), authorization (Kubernetes RBAC or OIDC), horizontal pod autoscaling, pod disruption budgets, CronJobs for materialization, and integration with OpenDataHub notebooks. The operator creates all necessary Kubernetes resources including Deployments, Services, ConfigMaps, RBAC resources, and PersistentVolumeClaims.

In the RHOAI context, Feast is deployed via the platform operator (`rhods-operator`/`opendatahub-operator`) which consumes the kustomize overlays in `config/overlays/rhoai/`. The operator image and feature server image are injected via `RELATED_IMAGE` environment variables from `params.env`, enabling disconnected/air-gapped deployments. The feature server image bundles the Python SDK with FastAPI-based REST serving, gRPC registry, and an embedded Go online feature server for high-throughput inference workloads.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| feast-operator | Go Operator (controller-runtime) | Manages FeatureStore CRDs, deploys and configures Feast services on Kubernetes |
| feature-server | Python Service (FastAPI/uvicorn + embedded Go) | Serves online features via REST/gRPC, manages feature registry, provides offline data via Arrow Flight |

## Architectural Analysis

The Feast component follows a layered architecture with clear separation between the control plane (operator) and data plane (feature servers). The operator is a standard controller-runtime based Kubernetes operator that watches a single CRD (`FeatureStore`) and reconciles a rich set of dependent resources. What makes this operator notable is the breadth of resources it manages: a single `FeatureStore` CR can result in up to 4 Deployments (offline, online, registry, UI), each with its own Service, plus ConfigMaps, RBAC resources, PVCs, CronJobs, HPAs, PDBs, Routes, and ServiceMonitors. The operator handles this complexity through a well-structured `deployFeast()` method that processes each service type sequentially with condition-based status reporting.

The authorization architecture is sophisticated, with three interacting layers. First, the operator creates Kubernetes RBAC resources (ClusterRoles, Roles, Bindings) for the FeatureStore service accounts. Second, the feature server implements application-level authorization through a Python permissions framework that supports both OIDC token validation and Kubernetes token review. Third, the operator implements an "auto-access" pattern where it queries the registry REST API for permission policies, extracts group and namespace information, and creates corresponding Kubernetes RBAC resources to allow those subjects to access the FeatureStore's client ConfigMap. This three-layer approach enables both cluster-native and external identity provider integration. The intra-communication token (HMAC-SHA256 JWT with empty key) used between operator and registry is a pragmatic but low-security choice — it primarily prevents per-user auth checks on internal calls rather than providing strong authentication.

The TLS architecture reflects strong OpenShift integration. On OpenShift clusters, the operator automatically provisions service-serving certificates by annotating Services, which triggers the platform's certificate signer. The operator also auto-detects ODH/RHOAI CA bundles and injects them into pod containers. Manual TLS is supported via user-provided Secrets for non-OpenShift deployments. One notable concern is the `InsecureSkipVerify: true` in the operator's registry REST client — while used only for cluster-internal calls, this bypasses certificate verification for the registry API. The Go feature server enforces TLS 1.2+ with modern curve preferences (P256 and the post-quantum X25519MLKEM768), showing forward-thinking cryptographic choices.

The FIPS compliance posture is split: the operator Konflux Dockerfile uses `CGO_ENABLED=1 GOEXPERIMENT=strictfipsruntime` and `-tags strictfipsruntime` for proper FIPS-compliant Go compilation, but the feature server (Python) relies solely on inheriting FIPS compliance from the UBI9 base image's OpenSSL. The non-Konflux operator Dockerfile uses `CGO_ENABLED=0` (static linking), which would fail check-payload — this confirms the Konflux Dockerfile is the production build. The absence of an OLM `fips-compliant` annotation in the CSV is notable and may need attention for RHOAI certification.

The notebook integration via `NotebookConfigMapReconciler` is an elegant ODH-specific feature. It watches Kubeflow Notebook CRDs for a feast integration label, then assembles a ConfigMap containing `feature_store.yaml` configurations for all referenced Feast projects. This enables data scientists to access Feast features from their notebooks without manual configuration. The controller handles CRD absence gracefully by polling every 5 seconds until the Notebook CRD appears, though it must restart the operator to register the controller dynamically (a controller-runtime limitation).
