## Purpose

**Short**: Feast is an open-source feature store for machine learning that manages the storage, retrieval, and serving of features for both training and online inference.

**Detailed**: Feast (Feature Store) provides a unified platform for managing ML features across the full lifecycle — from feature definition and ingestion, through offline historical retrieval for training, to low-latency online serving for real-time inference. The system consists of a Kubernetes operator that manages the lifecycle of feature store deployments, a Python-based feature server (also serving as a registry), a high-performance Go-based online feature server, and a React-based web UI for exploring feature definitions, data sources, and feature services.

The Feast operator (`feast.dev/v1 FeatureStore` CRD) deploys and manages the complete feature store stack: registry (gRPC + REST), online store, offline store, UI, and scheduled materialization CronJobs. It supports multiple persistence backends (file-based, PostgreSQL, Redis, Snowflake, BigQuery, DynamoDB, and many others), flexible authentication (none, Kubernetes RBAC, OIDC), TLS via OpenShift service-serving certificates, horizontal pod autoscaling, and integration with OpenDataHub notebooks. The operator is deployed by the RHOAI/ODH platform operators and participates in the platform's OIDC-based authentication when configured.

## Architectural Analysis

The Feast repository is a substantial multi-language monorepo that produces two Konflux container images — an operator and a feature server — plus a bundled React UI. The operator follows a standard controller-runtime pattern with a single primary controller (`FeatureStoreReconciler`) that manages the full lifecycle of a FeatureStore CR by creating and reconciling Deployments, Services, ConfigMaps, ServiceAccounts, RBAC resources, CronJobs, HPAs, PDBs, ServiceMonitors, and OpenShift Routes. A secondary controller (`NotebookConfigMapReconciler`) watches Kubeflow Notebook CRs and automatically injects Feast client configuration, enabling zero-configuration integration with the ODH/RHOAI notebook ecosystem.

The operator's authentication architecture is notably sophisticated for a component-level operator. It supports three authorization modes — none, Kubernetes RBAC, and OIDC — with the OIDC issuer URL flowing from the platform operator's GatewayConfig through a kustomize replacement chain into the operator's environment. An internal "intra-communication" mechanism uses randomly generated tokens stored in ConfigMaps and transmitted as HMAC-SHA256 JWTs to authenticate operator-to-registry REST API calls for permission discovery. The auto-access RBAC system dynamically creates ClusterRoles and namespace-scoped Roles based on permissions defined in the Feast registry, enabling fine-grained cross-namespace access control.

The TLS implementation deserves attention: the operator supports both OpenShift service-serving certificates (automatic via annotations) and user-provided TLS secrets. When TLS is enabled, service ports shift from 80 to 443, and the target container ports change from HTTP to HTTPS variants (e.g., 6570→6571 for registry gRPC). However, the operator's internal registry client uses `InsecureSkipVerify=true` when connecting to the registry REST API, which is documented as intentional for cluster-internal service certificates but represents a trust-on-first-use pattern that bypasses certificate validation.

The Python feature server (`feature_server.py` at ~1000 lines) is the most complex service, implementing a multi-role FastAPI application that can serve as an online feature server, registry server, offline store, and UI host simultaneously. It includes advanced features like offline write batching (queue-based with configurable batch size and flush intervals), WebSocket-based chat streaming, MCP (Model Context Protocol) support for AI agent integration, and Data Quality Monitoring. The Go feature server is a simpler alternative focused on high-performance online serving with Redis backend support, OpenTelemetry tracing, and Prometheus metrics — it does not support the full feature set of the Python server.

A noteworthy architectural pattern is the dual-track image configuration: the operator accepts `RELATED_IMAGE_FEATURE_SERVER` and `RELATED_IMAGE_CRON_JOB` as environment variables (set via kustomize params.env), enabling the platform operator to pin specific image versions. The CronJob image uses `ose-cli` (RHOAI) or `origin-cli` (ODH) — a general-purpose OpenShift CLI image — to execute `feast apply` and `feast materialize-incremental` commands, which means the CronJob containers actually run the feast Python SDK from the feature server image while using the CLI image as a base. This pattern warrants verification to ensure the correct feature server image is actually used for CronJob pods.
