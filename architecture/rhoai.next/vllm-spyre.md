## Purpose

**Short**: IBM Spyre-accelerated vLLM inference server container image for Red Hat OpenShift AI, providing high-performance LLM serving on IBM Spyre hardware.

**Detailed**: The vllm-spyre component is a thin container image wrapper that repackages the Red Hat AI Inference Server (RHAIIS) vLLM Spyre image (`registry.redhat.io/rhaiis/vllm-spyre-rhel9`) for distribution within the RHOAI platform. It contains no application source code — all vLLM runtime code, the TGIS (Text Generation Inference Server) gRPC adapter, and IBM Spyre accelerator support are inherited from the RHAIIS base image.

The image runs the `vllm_tgis_adapter` Python module as its entrypoint, which provides both an OpenAI-compatible HTTP API (port 8000) and a TGIS-compatible gRPC API (port 8033) for large language model inference. The TGIS adapter allows this vLLM-based server to serve as a drop-in replacement for the legacy TGIS inference backend while also exposing the standard OpenAI chat/completions API.

This repository adds RHOAI-specific container labels (`rhoai/odh-vllm-spyre-rhel9`), environment variable defaults (disabling logprob suppression during speculative decoding), and runs as a non-root user (UID 2000). The IBM Spyre accelerator provides hardware-accelerated inference on IBM Z and LinuxONE (s390x), IBM Power (ppc64le), and x86_64 platforms.

## Architectural Analysis

The vllm-spyre repository follows the "thin wrapper" pattern common across RHOAI inference serving images. The repository contains no application source code — it exists solely as a Konflux build entry point that takes the RHAIIS product image and applies RHOAI-specific branding (container labels) and runtime configuration (environment variables, user ID, entrypoint). This pattern means the actual vLLM application code, IBM Spyre accelerator integration, and TGIS gRPC adapter all live in upstream repositories managed by the RHAIIS team, and this downstream repo simply pins a version (`RHAIIS_VERSION=3.2.2`) and customizes the container metadata.

The dual-protocol design is architecturally significant: the container exposes both an OpenAI-compatible HTTP API on port 8000 and a TGIS-compatible gRPC API on port 8033. The TGIS protocol compatibility is critical for RHOAI because KServe ModelMesh historically used TGIS as its inference backend, and this vLLM image must serve as a drop-in replacement. The `DISABLE_LOGPROBS_DURING_SPEC_DECODING=false` environment variable override indicates that speculative decoding is expected in production use, and the default vLLM behavior of suppressing logprobs during spec decoding would break user expectations for hosted model endpoints.

The IBM Spyre accelerator targets a distinct hardware ecosystem — IBM Z (s390x), IBM Power (ppc64le), and x86_64 — which differentiates this image from the CUDA and ROCm variants that target NVIDIA and AMD GPUs respectively. Per AIPCC overlay 0017, Spyre is at version 1.2.1 with Active status, using per-architecture versioning to allow independent updates across hardware platforms. This multi-architecture support makes the Spyre variant particularly important for enterprises running on IBM infrastructure where GPU accelerators may not be available.

A key architectural risk is the complete dependency on the RHAIIS base image for all security, FIPS, and runtime characteristics. This repository cannot independently assess or enforce FIPS compliance, TLS configuration, or vulnerability remediation — all such concerns must be addressed in the upstream RHAIIS build pipeline. The `RHAIIS_VERSION` build arg is the single control point for the entire dependency chain.
