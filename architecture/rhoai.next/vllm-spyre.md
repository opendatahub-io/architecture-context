## Purpose

**Short**: Container image packaging vLLM with IBM Spyre hardware acceleration and gRPC/TGIS adapter support for high-performance LLM inference on RHEL 9.

**Detailed**: vllm-spyre is a downstream packaging repository that produces a container image for running vLLM inference workloads accelerated by IBM Spyre hardware. The repository contains no application source code — it consists solely of a single Konflux Dockerfile that layers configuration and entrypoint settings on top of the pre-built RHAIIS (Red Hat AI Inference Server) vllm-spyre base image.

The container runs the `vllm_tgis_adapter` Python module as its entrypoint, which provides both an OpenAI-compatible HTTP API (port 8000) and a gRPC interface (port 8033) for serving LLM inference requests. The TGIS (Text Generation Inference Server) adapter bridges vLLM's inference engine with the gRPC protocol used by IBM's text generation ecosystem, enabling KServe-based model serving in RHOAI.

The upstream vLLM codebase (from vllm-project/vllm) is included in the repository history as a fork base, but the Spyre-specific value is the Dockerfile that assembles the final image from the RHAIIS base, configures the dual-protocol serving (HTTP + gRPC), and sets deployment-appropriate defaults.

## Architectural Analysis

The vllm-spyre repository follows a minimal "image-layering" pattern — the entire repository consists of a single Dockerfile that layers deployment configuration on top of a pre-built RHAIIS base image. There is no application source code, no build logic, no dependency management, and no test infrastructure in this repository. The architectural complexity lies entirely in the base image (`registry.redhat.io/rhaiis/vllm-spyre-rhel9:3.2.2`) and the upstream vLLM project whose code is forked into the git history.

The Dockerfile's key architectural decision is the dual-protocol serving configuration: port 8000 for OpenAI-compatible HTTP API (provided by vLLM's built-in API server) and port 8033 for gRPC via the TGIS (Text Generation Inference Server) adapter. The TGIS adapter is significant because it enables integration with IBM's text generation ecosystem and KServe's gRPC-based model serving protocol, which is the primary protocol used for model serving in RHOAI. The `DISABLE_LOGPROBS_DURING_SPEC_DECODING=false` environment variable override (line 13) is a deliberate UX decision to ensure log probabilities remain available to users even when speculative decoding is active, overriding vLLM's default performance optimization.

The container runs as non-root user (UID 2000), which is a security best practice for OpenShift deployments where Security Context Constraints (SCCs) restrict privileged execution. Authentication is entirely delegated to the platform infrastructure — the container itself exposes unauthenticated endpoints, relying on KServe's inference routing and the RHOAI platform's ingress stack (kube-rbac-proxy, Gateway API, or OAuth proxy) to enforce access control. This is the standard pattern for model serving runtimes in RHOAI.

The git history reveals that this repository was created from a fork of the upstream vLLM project (v0.9.1.0 tag), with only 4 Spyre-specific commits added on top. The upstream vLLM commits remain in the history but are not architecturally relevant to the downstream image build — the Dockerfile references the pre-built RHAIIS base image rather than building vLLM from source.
