## Purpose

**Short**: IBM Spyre accelerator-optimized vLLM inference serving container image rebuilt through the Konflux pipeline for RHOAI supply chain compliance.

**Detailed**: The vllm-spyre repository is a container image rebuild repository that produces a Konflux-built variant of the IBM Spyre-accelerated vLLM inference server. It contains no application source code — all runtime components (vLLM, vllm-spyre plugin, vllm-tgis-adapter, FMS libraries) are pre-built into the upstream base image `registry.redhat.io/rhaiis/vllm-spyre-rhel9`. This repository exists solely to pass that image through the Konflux build pipeline, adding RHOAI-specific container labels, environment variable configuration, and supply chain provenance attestations.

The resulting container image serves as a KServe ServingRuntime for deploying large language models on IBM Spyre AI accelerator hardware. It exposes both an OpenAI-compatible HTTP API (port 8000) and a gRPC endpoint (port 8033) via the vllm-tgis-adapter, which bridges vLLM's native OpenAI API to the TGIS (Text Generation Inference Service) gRPC protocol used by RHOAI's model serving infrastructure. The image runs as non-root user 2000, compatible with OpenShift's restricted SCC.

The repository was created by syncing from the neural-magic midstream vLLM fork (`nm/midstream-cuda-v0.9.1.0`) and then replacing all upstream source code with a single Konflux Dockerfile that rebases on the pre-built RHAIIS image. The original Spyre build artifacts (multi-stage Dockerfile, entrypoint script, FMS library clones, IBM AIU base image) were historically present but were removed in favor of the thin-wrapper approach.

## Architectural Analysis

The vllm-spyre repository follows a "thin wrapper" or "container image rebuild" pattern that is characteristic of RHOAI's supply chain architecture. Rather than building the vLLM inference server from source, this repository takes a pre-built image from the RHAIIS (Red Hat AI Inference Service) product line and passes it through the Konflux build pipeline. This achieves supply chain provenance attestation and RHOAI-specific labeling without duplicating the complex multi-stage build that produces the Spyre-accelerated vLLM binary. The Konflux Dockerfile is minimal — a single FROM directive, environment variables, a non-root USER, an ENTRYPOINT, and container labels.

The repository's git history reveals an interesting evolution. It was forked from the neural-magic midstream vLLM fork at the `nm/midstream-cuda-v0.9.1.0` tag, which carried the full upstream vLLM source tree (thousands of Python, C++, and CUDA files). An initial PR added Spyre-specific build artifacts: a multi-stage `Dockerfile.spyre.ubi` that built from the IBM AIU base image (`quay.io/ibm-aiu/base`), cloned FMS (Foundation Model Stack) libraries, installed vLLM from pre-built wheels, and configured IBM Spyre environment variables (FLEX_COMPUTE=SENTIENT, FLEX_DEVICE=VFIO, VLLM_PLUGINS=spyre). A subsequent PR then deleted all source code and replaced the build with the current thin-wrapper `Dockerfile.konflux.spyre` that rebases on the already-built RHAIIS image. This shift indicates that the Spyre build was moved upstream to the RHAIIS product pipeline, and this repository now serves purely as a Konflux rebuild entry point.

The environment variables configured in the Konflux Dockerfile are architecturally significant: `GRPC_PORT=8033` enables the TGIS gRPC protocol alongside the default HTTP API on port 8000, and `DISABLE_LOGPROBS_DURING_SPEC_DECODING=false` ensures users of hosted models get consistent logprobs behavior even when speculative decoding is enabled. The entrypoint (`python3 -m vllm_tgis_adapter`) launches the TGIS adapter rather than vanilla vLLM, bridging vLLM's OpenAI-compatible API to the TGIS protocol expected by RHOAI's KServe/ModelMesh model serving infrastructure.

A key architectural constraint is `HF_HUB_OFFLINE=1` (set in the original build but expected to persist through the base image), which enforces air-gapped operation — model weights must be pre-downloaded and mounted into the container via PersistentVolumeClaims or init containers. This is a security and compliance requirement for disconnected/restricted environments. The IBM Spyre accelerator is accessed via VFIO device passthrough (FLEX_DEVICE=VFIO), requiring the host node to have Spyre hardware and appropriate device plugins configured.
