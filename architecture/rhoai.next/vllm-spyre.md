## Purpose

**Short**: IBM Spyre accelerator-optimized vLLM container image for high-performance LLM inference on RHEL 9, exposing both OpenAI-compatible HTTP and TGIS gRPC APIs.

**Detailed**: The vllm-spyre component is a Konflux-built container image that wraps the Red Hat AI Inference Server (RHAIIS) vllm-spyre base image to produce a deployable inference runtime for Red Hat OpenShift AI. It targets IBM Spyre accelerator cards on x86_64 architecture, providing hardware-accelerated LLM inference through the vLLM engine.

The image packages the `vllm_tgis_adapter` Python module as its entrypoint, which exposes two network interfaces: an OpenAI-compatible HTTP API on port 8000 and a TGIS (Text Generation Inference Server) gRPC adapter on port 8033. This dual-protocol approach allows the runtime to serve both standard OpenAI API clients and IBM's TGIS protocol clients used within the RHOAI platform by KServe's inference serving infrastructure.

The repository itself is minimal — consisting solely of a Konflux Dockerfile, README, and LICENSE — because all application logic resides in the upstream RHAIIS base image (`registry.redhat.io/rhaiis/vllm-spyre-rhel9`). This repo's role is to apply RHOAI-specific labels, set environment variable defaults (ports, speculative decoding behavior), configure the non-root user, and define the container entrypoint for the Konflux build pipeline.

## Architectural Analysis

The vllm-spyre repository represents the thinnest possible container image wrapper pattern: a single Dockerfile with no application source code. All functional logic — the vLLM inference engine, IBM Spyre hardware integration, the TGIS gRPC adapter, and the OpenAI-compatible HTTP server — resides in the upstream RHAIIS base image (`registry.redhat.io/rhaiis/vllm-spyre-rhel9:3.2.2`). This design means the repository's sole architectural responsibility is Konflux build pipeline integration, RHOAI labeling compliance, and environment variable defaults.

The dual-protocol exposure (OpenAI HTTP on port 8000 and TGIS gRPC on port 8033) is architecturally significant. The OpenAI-compatible API enables standard model serving clients, while the TGIS protocol provides backward compatibility with IBM's Watson-era inference stack and is the protocol KServe uses internally for predictor-transformer communication in the RHOAI platform. The `vllm_tgis_adapter` module bridges vLLM's native serving with the TGIS protocol, running both servers in a single process via the uvicorn ASGI framework.

The `DISABLE_LOGPROBS_DURING_SPEC_DECODING=false` environment variable override is a deliberate architectural decision documented in the Dockerfile. Upstream vLLM disables logprobs during speculative decoding as a performance optimization, but this would break user expectations in a hosted model serving environment where speculative decoding is an implementation detail invisible to the API consumer. This override ensures API contract consistency at a small performance cost.

The repository contains the full upstream vLLM README but none of the upstream source code — the README was carried over from the fork's initial commit. The git history shows the repo was created from a vLLM fork (commit `abab1afe1 Initial commit`) with subsequent RHOAI-specific commits adding the Spyre Dockerfile and modifying labels. The upstream vLLM commit history is preserved, but none of the source files are present in the working tree — only the Dockerfile, README, and LICENSE remain, indicating aggressive pruning to create a build-only repository.
