## Purpose

**Short**: IBM Spyre-accelerated vLLM inference server container image for RHOAI, wrapping the RHAIIS vllm-spyre-rhel9 product image with TGIS adapter configuration.

**Detailed**: vllm-spyre is a thin container image that extends the Red Hat AI Inference Server (RHAIIS) vllm-spyre-rhel9 base image to provide an IBM Spyre accelerator-optimized vLLM inference server for Red Hat OpenShift AI. The repository contains no application source code — the entire component is defined by a single Konflux Dockerfile that layers runtime configuration (environment variables, non-root user, entrypoint) on top of the RHAIIS product image.

The image runs vLLM through the `vllm_tgis_adapter` module, which exposes both an OpenAI-compatible HTTP API (port 8000) and a gRPC TGIS (Text Generation Inference Server) API (port 8033). This dual-protocol support allows the image to be consumed by KServe's ServingRuntime framework for model serving on OpenShift AI, where the gRPC interface provides backward compatibility with the IBM TGIS protocol used by earlier RHOAI releases.

The IBM Spyre accelerator is a purpose-built AI inference chip. This image variant targets x86_64, ppc64le, and s390x architectures, aligning with the AIPCC Ecosystems base image availability for IBM Spyre (overlay 0017). The component is part of the broader vLLM family in RHOAI, which includes separate image variants for NVIDIA CUDA, AMD ROCm, Intel Gaudi, and CPU-only inference.

## Architectural Analysis

vllm-spyre is architecturally the simplest possible RHOAI component: a single Dockerfile with no application source code, no deployment manifests, and no operator logic. The entire repository exists to add a thin configuration layer — environment variables, a non-root user, and an entrypoint — on top of the RHAIIS product image. This design means the repository is essentially a "last mile" customization point, while all meaningful application logic, security configuration, and dependency management live in the upstream RHAIIS image build pipeline.

The dual-protocol exposure (HTTP on port 8000 and gRPC on port 8033) reflects RHOAI's migration from the IBM TGIS protocol to the OpenAI-compatible API. The `vllm_tgis_adapter` Python module wraps vLLM to provide both interfaces simultaneously, allowing existing TGIS clients to continue working while new consumers use the standard OpenAI API. The `DISABLE_LOGPROBS_DURING_SPEC_DECODING=false` environment variable is a notable runtime default override — it ensures that logprobs remain available even when speculative decoding is enabled, which would otherwise be silently disabled by vLLM as an optimization. This is critical for hosted model serving where consumers expect consistent API behavior regardless of the underlying inference optimization strategy.

The non-root user (UID 2000) differs from the AIPCC base image convention (UID 1001). This suggests the RHAIIS product image or the downstream configuration has specific UID requirements for OpenShift's security context constraints (SCCs). The choice of UID 2000 is arbitrary but ensures the container runs unprivileged, which is mandatory for OpenShift restricted-v2 SCC compliance.

A significant architectural risk is the lack of digest pinning for the base image. The Dockerfile references `registry.redhat.io/rhaiis/vllm-spyre-rhel9:${RHAIIS_VERSION}` using a tag (`3.2.2`), which can be overwritten if the RHAIIS image is republished. For production Konflux builds, the base image should be pinned by SHA256 digest to ensure reproducible builds. This is likely handled by the Konflux pipeline's image resolution step, but is not guaranteed by the Dockerfile alone.
