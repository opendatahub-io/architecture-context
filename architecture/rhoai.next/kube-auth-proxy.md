## Purpose

**Short**: A FIPS-compliant authentication reverse proxy providing OIDC, OpenShift OAuth, and Kubernetes RBAC-based authorization for RHOAI platform components.

**Detailed**: kube-auth-proxy is a dual-mode authentication proxy that ships two binaries in a single container image: **kube-auth-proxy** (OAuth2/OIDC authentication proxy derived from oauth2-proxy) and **kube-rbac-proxy** (Kubernetes RBAC authorization proxy derived from brancz/kube-rbac-proxy). A Go entrypoint binary selects which proxy mode to run based on the `PROXY_MODE` environment variable or a CLI argument, defaulting to `rbac` mode.

In the RHOAI 3.x architecture, this image is deployed as a sidecar container alongside platform components. In `rbac` mode (the default), it performs SubjectAccessReview (SAR) authorization against the Kubernetes API, authenticating requests via delegated token review or OIDC before checking RBAC permissions. In `auth` mode, it acts as an OAuth2/OIDC-aware reverse proxy that handles the full authentication flow — login redirects, token exchange, session management, cookie-based sessions, and token refresh — supporting both external OIDC providers and OpenShift's native OAuth service. The proxy also supports Kubernetes service account token validation via the TokenReview API, enabling machine-to-machine authentication alongside human user flows.

The project was created as a focused fork of oauth2-proxy, streamlined to support only OIDC and OpenShift OAuth providers, with FIPS compliance as a primary requirement. All builds use `GOEXPERIMENT=strictfipsruntime` with CGO enabled for FIPS-approved cryptographic libraries.

## Architectural Analysis

The kube-auth-proxy repository implements a **dual-proxy architecture** that consolidates two distinct authentication/authorization patterns into a single container image. The entrypoint binary (`cmd/entrypoint/main.go`) acts as a simple mode dispatcher, using `os.Exec` to launch either the OAuth2-based kube-auth-proxy or the RBAC-based kube-rbac-proxy. This design enables a single image to serve both RHOAI 3.x authentication patterns: OAuth2/OIDC for browser-based human users and Kubernetes RBAC for machine-to-machine API access. The default mode is `rbac`, indicating that most sidecar deployments use the kube-rbac-proxy for metrics protection and API authorization.

The **authentication middleware chain** in the auth-mode proxy is sophisticated and layered. The session loading order is: (1) Kubernetes SA token validation via TokenReview API, (2) JWT bearer token verification or OpenShift OAuth token validation, (3) Basic auth via htpasswd, and (4) stored session (cookie) loading with optional refresh. This ordering means K8s service account tokens are checked first, allowing service-to-service authentication to bypass the full OAuth2 flow. The OpenShift provider includes intelligent auto-discovery — it reads the well-known OAuth metadata endpoint from the Kubernetes API to discover login and token URLs dynamically, and auto-detects client credentials from mounted service account secrets. The provider also handles OpenShift-specific logout, deleting `OAuthAccessToken` resources from the cluster (mirroring the OCP console's behavior) and supporting RP-initiated logout via OIDC `end_session_endpoint` discovery.

The **kube-rbac-proxy** sub-component follows a different architectural pattern — it uses the `k8s.io/apiserver` framework for delegated authentication and authorization, matching how the kubelet authenticates requests. It supports a union authorizer combining hardcoded metrics permissions (for OpenShift monitoring), static authorization rules from a config file, and dynamic SubjectAccessReview-based authorization. The `ResourceAttributes` configuration allows mapping HTTP requests to Kubernetes API resource attributes, with template-based rewrites from query parameters or HTTP headers — a powerful pattern for protecting custom endpoints with fine-grained RBAC without modifying the upstream application.

The repository's FIPS compliance strategy is notable: all builds in `Dockerfile.redhat` and `Dockerfile.konflux` use `CGO_ENABLED=1` with `GOEXPERIMENT=strictfipsruntime` and the `-tags strictfipsruntime` build tag. This ensures the Go runtime uses only FIPS-validated cryptographic implementations. The build produces binaries for amd64, arm64, ppc64le, and s390x architectures, supporting the full range of RHOAI-supported platforms. TLS configuration enforces a minimum of TLS 1.2 across both proxy modes, with configurable cipher suites.

A key design decision is the **ext_authz compatibility** pattern. The `/oauth2/auth` endpoint is specifically designed for use with Envoy's external authorization filter or nginx subrequest authentication. When CSRF cookie validation fails during OAuth callbacks, the proxy redirects to the sign-in page (302) instead of returning 403, because ext_authz passes the raw response body for non-200 responses — users would see raw HTML instead of the sign-in page. This is documented explicitly in the code and represents a deliberate UX trade-off for proxy-behind-proxy architectures that are common in RHOAI 3.x's Gateway API + Envoy ingress stack.
