## Purpose

**Short**: A FIPS-compliant authentication reverse proxy supporting OIDC and OpenShift OAuth, designed as a drop-in replacement for oauth-proxy sidecars in ODH/RHOAI environments.

**Detailed**: kube-auth-proxy is a focused authentication proxy derived from oauth2-proxy, streamlined to support only the authentication methods required for enterprise Kubernetes environments: external OIDC providers and OpenShift's internal OAuth service. It intercepts requests to upstream applications, authenticates users through configurable identity providers, and forwards authenticated requests with identity headers (X-Forwarded-User, X-Forwarded-Email, X-Forwarded-Groups, etc.).

The proxy is deployed as a sidecar container alongside RHOAI component pods (replacing the legacy oauth-proxy sidecar) or as a standalone reverse proxy service. It supports Envoy ext_authz compatibility (returning static 200/OK after auth validation), Kubernetes service account token validation via the TokenReview API, PKCE code challenge, Redis-backed session persistence, WebSocket proxying, and RP-initiated logout. A custom MLflow auth-denied handler provides user-friendly error messages for MLflow Python SDK clients.

The project emphasizes FIPS compliance through `GOEXPERIMENT=strictfipsruntime` build flags, automated check-payload validation in CI, and TLS 1.2+ enforcement throughout. It is built by Konflux for RHOAI with hermetic Go module prefetch and multi-architecture support (x86_64, ARM64, ppc64le).

## Architectural Analysis

kube-auth-proxy is architecturally a focused fork of oauth2-proxy that strips away all authentication providers except OIDC and OpenShift OAuth, adding RHOAI-specific capabilities. The codebase follows a middleware chain pattern using the `justinas/alice` library, where authentication processing flows through a well-defined pipeline: pre-auth middleware (logging, health checks, metrics, HTTPS redirect) -> session loading middleware (K8s token -> OAuth/JWT token -> basic auth -> stored session cookie) -> request handling (auth-only check or upstream proxy). This chain-of-responsibility design allows multiple authentication methods to coexist transparently: a single kube-auth-proxy instance can simultaneously accept browser-based OAuth sessions, service account bearer tokens (via K8s TokenReview), and htpasswd basic auth credentials.

The session management architecture is noteworthy for its distributed lock mechanism in the stored session loader. When multiple concurrent requests arrive with the same session cookie that needs refresh, the proxy uses a distributed lock (with Redis if configured) to prevent thundering herd token refresh: only one request acquires the lock, refreshes the token with the IdP, and saves the updated session, while other requests wait briefly and then use the already-refreshed session. The cookie-based session store automatically splits sessions exceeding the 4KB browser cookie limit across multiple numbered cookies (`_oauth2_proxy`, `_oauth2_proxy_1`, etc.), though Redis is recommended for production to avoid this complexity.

The OpenShift OAuth provider implementation (`providers/openshift.go`) demonstrates defensive architecture patterns appropriate for a sidecar running in diverse cluster environments. It performs lazy, thread-safe OAuth endpoint discovery (double-checked locking with `sync.Mutex`), caches HTTP clients per CA configuration (`sync.Map`), and handles both JSON and form-encoded token responses from the OAuth server. On sign-out, it performs background cleanup of OAuthAccessToken resources via the Kubernetes API (matching OCP console behavior from `openshift/console#6445`), using SHA256 hashing for `sha256~` prefixed token names.

The ext_authz compatibility mode (`/oauth2/auth` endpoint returning 202 on success) is critical for RHOAI 3.x integration where Envoy sits in front of component pods. The proxy also implements CSRF-error-to-redirect behavior specifically for ext_authz: when CSRF validation fails on the OAuth callback (e.g., expired cookie), it redirects to sign-in instead of returning 403, because ext_authz would pass the raw 403 HTML to the client. The MLflow auth-denied handler (`pkg/authdeny/mlflow.go`) detects the `mlflow-python-client/` User-Agent and returns structured JSON error responses with actionable guidance (oc login, MLFLOW_TRACKING_AUTH=kubernetes-namespaced), rather than HTML error pages that would confuse SDK clients.

From a security perspective, the FIPS compliance story is comprehensive: the Konflux Dockerfile uses `CGO_ENABLED=1 GOEXPERIMENT=strictfipsruntime -tags strictfipsruntime` for the Go build, the UBI 9 Minimal base image provides OpenSSL, and the GitHub Actions CI includes an automated check-payload validation step. The TLS server configuration enforces TLS 1.2 minimum with TLS 1.3 maximum and configurable cipher suites. No deprecated cryptographic algorithms (MD5, RC4, DES) were found in the codebase. The `InsecureSkipVerify` option exists for development/testing but is disabled by default and clearly documented with `#nosec G402` annotations.
