## Purpose

**Short**: A dual-mode authentication reverse proxy that provides OAuth2/OIDC and Kubernetes RBAC-based authentication for RHOAI platform components deployed as a sidecar container.

**Detailed**: kube-auth-proxy is a unified authentication proxy image that packages two distinct proxy binaries — an OAuth2/OIDC proxy (kube-auth-proxy, derived from oauth2-proxy) and a Kubernetes RBAC proxy (kube-rbac-proxy, derived from brancz/kube-rbac-proxy) — behind a single entrypoint that selects the appropriate mode at runtime. This design allows the RHOAI platform operator to deploy one container image that can serve either authentication pattern depending on the component's requirements.

The OAuth2/OIDC proxy mode (selected via `PROXY_MODE=auth`) handles browser-based authentication flows including OpenShift OAuth and generic OIDC providers, with session management via cookies or Redis, CSRF protection, and PKCE support. It is the RHOAI replacement for the legacy `openshift/oauth-proxy` sidecar. The RBAC proxy mode (selected via `PROXY_MODE=rbac`, the default) handles API-level authentication by delegating to the Kubernetes TokenReview and SubjectAccessReview APIs, enforcing RBAC policies on upstream service access. Both modes are built with FIPS-compliant cryptography for Red Hat environments.

The project originated as a disconnected fork of `oauth2-proxy/oauth2-proxy` with all non-OIDC providers removed, the OpenShift OAuth provider ported from `openshift/oauth-proxy`, and `brancz/kube-rbac-proxy` embedded as a Go workspace member. It serves as a drop-in replacement for the existing oauth-proxy and kube-rbac-proxy sidecars used across RHOAI components.

## Architectural Analysis

**Dual-Binary Design**: The most distinctive architectural decision in this project is packaging two independently-developed reverse proxies — an OAuth2/OIDC proxy and a Kubernetes RBAC proxy — into a single container image with a thin entrypoint dispatcher. This design eliminates the need for RHOAI to maintain two separate container images and simplifies the platform operator's sidecar injection logic. The `PROXY_MODE` environment variable approach means a single container spec template can be reused across components with only the mode and flags varying. The entrypoint uses `exec.Command` rather than `syscall.Exec` to preserve stdio piping and exit code forwarding.

**Authentication Chain Architecture**: The OAuth2/OIDC proxy implements a sophisticated multi-layer authentication chain built on the `justinas/alice` middleware composition library. The chain processes in strict order: (1) Kubernetes TokenReview, (2) OAuth bearer tokens, (3) JWT bearer tokens, (4) Basic auth via htpasswd, (5) Stored cookie sessions with auto-refresh. Each layer can short-circuit with a successful authentication or pass to the next layer, and the chain is configurable — layers are only added when their corresponding feature flags are enabled. This design supports the RHOAI requirement that a single sidecar must handle both browser users (cookie sessions) and API clients (bearer tokens, service account tokens) simultaneously.

**Session Management and Security**: The auth mode proxy supports two session storage backends — client-side encrypted cookies (AES-CFB with HMAC-SHA256 signing) and server-side Redis storage with distributed locking for concurrent session refresh. Cookie sessions are split across multiple cookies when they exceed the 4KB browser limit. Redis sessions use a ticket-based approach where only a ticket identifier is stored in the cookie while the full session state lives in Redis. The Redis backend supports standalone, Sentinel, and Cluster modes, making it suitable for production HA deployments. CSRF protection uses a combination of PKCE code verifiers (S256), OIDC nonce claims, and signed CSRF cookies — all serialized together via msgpack and encrypted.

**RBAC Proxy Authorization Model**: The kube-rbac-proxy implements a three-tier authorization chain: (1) a hardcoded metrics authorizer that auto-allows the OpenShift Prometheus service account to scrape `/metrics`, (2) a static authorizer for pre-configured rules in a YAML config file, and (3) a full SubjectAccessReview authorizer that delegates to the Kubernetes API server. Authorization decisions are cached — allow decisions for 5 minutes and deny decisions for 30 seconds — to reduce API server load. The proxy also supports request attribute rewrites where URL query parameters or HTTP headers can be mapped to Kubernetes RBAC resource attributes, enabling fine-grained resource-level authorization on HTTP endpoints. HTTP/2 is disabled by default as a mitigation for CVE-2023-44487 (HTTP/2 rapid reset attack).

**FIPS Compliance Strategy**: Both binaries are built with `CGO_ENABLED=1`, `GOEXPERIMENT=strictfipsruntime`, and `-tags strictfipsruntime`, which forces Go's crypto library to use OpenSSL via the FIPS-validated module in UBI9. The Dockerfile.konflux uses pinned digest references for both the builder image (`ubi9/go-toolset:1.25`) and runtime image (`ubi9/ubi-minimal`), ensuring reproducible FIPS-compliant builds. The runtime image inherits OpenSSL from UBI9. No non-FIPS crypto primitives (MD5, RC4, DES) were detected in the codebase, though `InsecureSkipVerify` is available as an opt-in flag for development/testing scenarios.
