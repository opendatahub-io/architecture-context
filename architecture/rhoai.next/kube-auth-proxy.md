## Purpose

**Short**: A FIPS-compliant authentication reverse proxy that provides OpenID Connect (OIDC) and OpenShift OAuth authentication for upstream services in RHOAI.

**Detailed**: kube-auth-proxy is a focused fork of oauth2-proxy, stripped down to support only two authentication providers: generic OIDC and OpenShift OAuth. It is designed as a drop-in replacement for the existing oauth-proxy sidecars used throughout Red Hat OpenShift AI (RHOAI) and Open Data Hub (ODH) platforms.

The proxy sits in front of upstream services and intercepts all HTTP requests, enforcing authentication before forwarding traffic. It supports multiple authentication methods simultaneously: browser-based OAuth2/OIDC flows (with session cookies), bearer token passthrough for API clients (JWT and OpenShift sha256~ tokens), htpasswd basic authentication, and Kubernetes service account token validation via the TokenReview API. Session state can be stored in encrypted cookies or externalized to Redis (standalone, Sentinel, or Cluster modes).

A key design goal is FIPS compliance — the binary is built with `CGO_ENABLED=1` and `GOEXPERIMENT=strictfipsruntime` to use OpenSSL-backed cryptography via Go's FIPS runtime mode. The project also integrates with Envoy's ext_authz protocol for use as an external authorization service in the RHOAI 3.x Gateway API ingress architecture, and includes MLflow SDK-specific error handling for improved developer experience.

## Architectural Analysis

kube-auth-proxy is architecturally a middleware-first reverse proxy. Every incoming request passes through a layered middleware chain before reaching the upstream proxy handler. The chain is divided into three phases: **pre-auth** (scope injection, HTTPS redirect, health checks, metrics), **session loading** (K8s TokenReview → OAuth/JWT bearer → basic auth → stored session cookie), and **headers** (request/response header injection from session claims). This chain ordering is significant — the K8s TokenReview loader runs first, ensuring service account tokens are validated before the OAuth/OIDC loaders attempt to process them. Each loader is a no-op if a previous loader already established a session, which allows multiple authentication mechanisms to coexist without conflict.

The project's lineage from oauth2-proxy is evident in its configuration system, which supports three modes: legacy CLI flags (compatible with the oauth-proxy used in RHOAI 2.x), TOML configuration files, and the newer "alpha" YAML configuration. All three can be combined, with explicit precedence rules. Environment variables use the `OAUTH2_PROXY_*` prefix for backward compatibility. This flexibility is architecturally intentional — it allows kube-auth-proxy to be a drop-in replacement for existing oauth-proxy sidecars without configuration changes.

The OpenShift OAuth provider implementation deserves attention. It uses a thread-safe discovery mechanism (`sync.Mutex`-protected) with retry logic that re-discovers OAuth endpoints on failure rather than caching them indefinitely. This handles OpenShift OAuth server certificate rotation and endpoint changes gracefully. The provider also implements `NewHTTPClient()` with a `sync.Map`-based client cache keyed by CA configuration, avoiding TLS handshake overhead for repeated API calls. TLS minimum version is hardcoded to 1.2 for OpenShift API connections (`providers/openshift.go:457`).

The ext_authz integration pattern is noteworthy. The `/oauth2/auth` endpoint is specifically designed for Envoy and nginx subrequest architectures — it returns HTTP 202 (accepted) on success rather than 200, and supports per-request authorization constraints via query parameters (`allowed_groups`, `allowed_emails`, `allowed_email_domains`). CSRF cookie failures on the OAuth callback now redirect to the sign-in page (configurable via `--disable-redirect-on-csrf-error`) instead of rendering an error page, because ext_authz returns raw 403 responses to clients that cannot render the error page HTML. This shows the proxy is being actively adapted for the RHOAI 3.x Gateway API + Envoy architecture.

The MLflow SDK integration (`pkg/authdeny/mlflow.go`) is a targeted UX improvement. When the proxy detects a `mlflow-python-client/` User-Agent, it returns structured JSON errors (`{"error_code": "UNAUTHENTICATED", ...}`) with actionable guidance about `oc login` and `MLFLOW_TRACKING_AUTH=kubernetes-namespaced`. This pattern — detecting SDK User-Agents and returning SDK-native error formats — could be extended to other RHOAI data science tools.

A potential concern is the `InsecureSkipVerify` option, which is configurable for upstream connections, Redis TLS, and OpenShift provider connections. While this is a legitimate operational need (e.g., self-signed certs in development environments), it is marked with `#nosec G402` annotations, indicating the team is aware of the security implications. The option is disabled by default and requires explicit opt-in.
