## Purpose

**Short**: A FIPS-compliant authentication and authorization reverse proxy for RHOAI, providing OIDC, OpenShift OAuth, and Kubernetes RBAC-based access control as a sidecar container.

**Detailed**: kube-auth-proxy is a focused, FIPS-compliant authentication proxy specifically designed for Red Hat OpenShift AI (RHOAI) and Open Data Hub (ODH) environments. It is derived from oauth2-proxy but has been streamlined to support only the authentication methods required for enterprise Kubernetes environments: OIDC (any standards-compliant provider) and OpenShift OAuth (the platform's built-in authentication system). The project serves as a drop-in replacement for existing oauth-proxy sidecars, maintaining argument and header compatibility while adding FIPS compliance.

The container image ships three binaries selected at runtime via an entrypoint multiplexer: **kube-auth-proxy** handles authentication (OIDC/OAuth flows, JWT bearer tokens, Kubernetes service account tokens, session management), while **kube-rbac-proxy** handles authorization (Kubernetes SubjectAccessReview-based RBAC enforcement). This dual-binary architecture separates authentication concerns (who are you?) from authorization concerns (what can you do?), allowing platform operators to choose the appropriate mode per component. The entrypoint binary selects between them based on the `PROXY_MODE` environment variable or a CLI argument, defaulting to `rbac` mode.

The project is maintained in both the `opendatahub-io` and `red-hat-data-services` GitHub organizations with commit synchronization between them. It is a critical security component in the RHOAI 3.x ingress architecture, where it replaces the legacy oauth-proxy sidecar pattern with a FIPS-compliant alternative that supports both the Gateway API (via Envoy ext_authz integration) and traditional sidecar proxy patterns.

## Architectural Analysis

kube-auth-proxy follows a **dual-binary single-image** architecture that is unusual but well-suited to its role in RHOAI. By packaging both an authentication proxy (kube-auth-proxy, derived from oauth2-proxy) and an authorization proxy (kube-rbac-proxy, derived from brancz/kube-rbac-proxy) into a single container image with an entrypoint multiplexer, the platform can use a single image reference for two distinct security functions. The `PROXY_MODE` environment variable selects the binary at container startup, defaulting to `rbac` mode. This design reduces image management overhead while maintaining clear separation of authentication (authn) and authorization (authz) concerns.

The authentication proxy (auth mode) implements a comprehensive middleware chain pattern using the `justinas/alice` library. The chain processes requests through: scope injection (request ID, reverse proxy detection) -> health/readiness checks -> request metrics -> session loading (K8s token -> OAuth token -> JWT token -> basic auth -> stored session) -> header injection -> upstream proxying. This layered design allows each authentication method to attempt validation independently, with the first successful method establishing the session. Notably, Kubernetes ServiceAccount token validation runs first in the chain, before provider-specific auth, ensuring cluster-internal service-to-service communication works regardless of the configured OAuth/OIDC provider.

The authorization proxy (rbac mode) uses a three-tier authorization strategy: (1) a hardcoded authorizer that always allows `system:serviceaccount:openshift-monitoring:prometheus-k8s` to GET `/metrics` (ensuring Prometheus can always scrape metrics), (2) a static authorizer for pattern-based rules configured via YAML, and (3) a SubjectAccessReview-based authorizer that delegates decisions to the Kubernetes API server. The union authorizer returns "allow" if any tier allows the request, implementing a permissive-union policy. Authorization results are cached (5-minute positive, 30-second negative) to reduce API server load.

Session management supports two storage backends: cookie-based (default, using AES-CFB encryption with HMAC-SHA256 signing) and Redis-based (for distributed deployments). The Redis backend supports standalone, Sentinel, and Cluster topologies, with distributed locking to prevent concurrent session refresh races. The cookie store implements automatic splitting for cookies exceeding the 4000-byte browser limit. The persistence layer uses a ticket system where the cookie stores only a session ID, and the actual session data (encrypted with AES-GCM using a per-session secret) resides in Redis.

A significant architectural consideration is the FIPS compliance strategy. Both binaries are built with `CGO_ENABLED=1`, `GOEXPERIMENT=strictfipsruntime`, and `-tags strictfipsruntime`, ensuring Go's crypto operations are delegated to the system OpenSSL library rather than Go's pure-Go crypto implementations. The UBI 9 Minimal base image provides a FIPS-validated OpenSSL. All cryptographic operations in the codebase (AES-CFB, AES-GCM, HMAC-SHA256, SHA-256) use FIPS-approved algorithms, and no non-FIPS crypto primitives (MD5, RC4, DES) were detected. The kube-rbac-proxy component uses Kubernetes' `k8sapiflag` package for TLS version and cipher suite parsing, which provides secure defaults aligned with OpenShift's security requirements.

The ext_authz compatibility endpoint (`/oauth2/auth`) is architecturally important for the RHOAI 3.x Gateway API pattern. When deployed behind Envoy (the data plane for Gateway API), Envoy can send subrequests to this endpoint. A 202 response means the user is authenticated and Envoy forwards the request; a 401 response triggers the authentication flow. This pattern allows kube-auth-proxy to work as an external authorization service rather than an inline reverse proxy, enabling the Gateway API ingress architecture where Envoy handles traffic routing and kube-auth-proxy handles authentication decisions.
