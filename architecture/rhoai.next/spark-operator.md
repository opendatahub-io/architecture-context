## Purpose

**Short**: Kubernetes operator that automates the lifecycle management of Apache Spark applications, scheduled Spark jobs, and Spark Connect servers on OpenShift/Kubernetes clusters.

**Detailed**: The Spark Operator (originally Kubeflow Spark Operator) provides a declarative, Kubernetes-native interface for running Apache Spark workloads. Users create `SparkApplication`, `ScheduledSparkApplication`, or `SparkConnect` custom resources, and the operator handles the full lifecycle: validating configurations, submitting jobs via `spark-submit`, creating driver/executor pods, exposing the Spark UI, monitoring execution state, managing retries, and cleaning up resources on completion.

The operator is deployed as two separate components sharing the same container image: a **controller** deployment that watches CRs and manages Spark workloads, and a **webhook** deployment that intercepts Pod and SparkApplication creation requests to inject Spark configuration (volumes, environment variables, sidecar containers, security context, resource limits, monitoring, and scheduler annotations). The webhook also validates SparkApplications against naming constraints, resource quotas, and Spark version compatibility. The operator integrates with batch schedulers (Volcano, YuniKorn, Kubernetes scheduler-plugins) for gang scheduling, Prometheus for metrics collection, and the OpenShift TLS security profile for cluster-aligned TLS configuration.

## Architectural Analysis

The Spark Operator follows a dual-deployment architecture where a single container image runs in two distinct modes: `spark-operator controller start` for the reconciliation controllers, and `spark-operator webhook start` for the admission webhook server. This separation is architecturally significant -- the controller manages long-running reconciliation loops with spark-submit subprocess management, while the webhook handles latency-sensitive admission requests that directly gate pod creation in the cluster. Separating them allows independent scaling, different resource limits, and failure isolation (a stuck spark-submit process doesn't block pod admissions).

The SparkApplication state machine is the operator's most complex subsystem, with 10+ states (New, Submitted, Running, Succeeding, Failing, Completed, Failed, FailedSubmission, PendingRerun, Invalidating, Suspending, Suspended, Resuming). The state machine drives retry logic with configurable backoff, TTL-based cleanup of completed applications, and a sophisticated spec-change detection mechanism via the `Invalidating` state. The `PartialRestart` feature gate (alpha, default off) allows certain executor fields (PriorityClassName, NodeSelector, Tolerations, Affinity, SchedulerName) to be changed without a full application restart -- these fields are applied via webhook mutation rather than requiring a new spark-submit.

The webhook's SparkPod defaulter is particularly complex, performing 22 sequential mutation steps on every Spark driver and executor pod. This design moves configuration injection from the spark-submit command line to Kubernetes admission control, which has advantages (the operator doesn't need to know all Spark configuration options) but creates a tight coupling between the webhook and pod creation -- if the webhook is down, no Spark pods can be created (failurePolicy: Fail). The webhook also integrates with OpenShift's cluster-wide TLS security profile, reading the `config.openshift.io/v1 APIServer` CR to dynamically configure its own TLS minimum version and cipher suites, which is a RHOAI-specific enhancement over the upstream Kubeflow operator.

The operator uses self-signed certificates for webhook TLS by default, with dedicated controllers (MutatingWebhookConfiguration and ValidatingWebhookConfiguration controllers) that watch for changes and re-inject the CA bundle. This self-managed PKI avoids a hard dependency on cert-manager while still supporting it as an optional backend. The certificate.Provider abstraction handles key generation, Secret synchronization, and file writing to the webhook container's cert directory.

Network isolation follows the ODH/RHOAI pattern: a NetworkPolicy restricts ingress to Spark pods, allowing internal Spark communication (driver-rpc 7078, blockmanager 7079, UI 4040) from same-namespace pods and OpenShift ingress, while SparkConnect gRPC (15002) is additionally allowed from workbench namespaces identified by the `opendatahub.io/dashboard` and `opendatahub.io/workbenches` labels. This aligns with the platform's multi-tenant workbench model where users in Data Science Projects interact with shared Spark Connect servers.
