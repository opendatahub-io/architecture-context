## Purpose

**Short**: Kubernetes operator that automates the submission, scheduling, monitoring, and lifecycle management of Apache Spark applications on OpenShift/Kubernetes clusters.

**Detailed**: The Spark Operator (originally Kubeflow Spark Operator) manages Apache Spark workloads through three Custom Resource Definitions: SparkApplication for one-off Spark jobs, ScheduledSparkApplication for cron-scheduled recurring jobs, and SparkConnect for persistent Spark Connect server sessions. The operator watches these CRDs and reconciles the desired state by creating driver pods, managing executor pods via Spark's native Kubernetes scheduler, exposing web UIs through Services and Ingresses, and configuring Prometheus monitoring.

The operator is deployed as two separate components — a controller deployment and a webhook deployment — sharing the same container image. The controller handles the reconciliation loop (watching CRDs, creating Pods, Services, Ingresses, ConfigMaps), while the webhook handles admission control (mutating Spark pods with volumes, environment variables, and security contexts; validating SparkApplication specs; and enforcing resource quotas). Both components support leader election for HA deployments.

The operator integrates with three pluggable batch schedulers (Volcano, YuniKorn, Kube-scheduler with scheduler-plugins) for gang scheduling of Spark driver and executor pods, and supports the OpenShift TLS security profile API for cluster-consistent cipher suite configuration.

## Architectural Analysis

The Spark Operator follows a clean split-process architecture where the controller and webhook run as separate Deployments sharing the same container image. The `cmd/operator/main.go` entry point dispatches to either `controller start` or `webhook start` subcommands. This separation ensures that webhook availability (critical for pod admission) is independent of controller reconciliation load, and each component can scale its resource limits independently. Both deployments use leader election for HA, with separate lease names (`spark-operator-controller-lock` and `spark-operator-webhook-lock`).

The webhook component is architecturally sophisticated. The SparkPod defaulter (`internal/webhook/sparkpod_defaulter.go`) applies 19 sequential mutation functions to every Spark pod at admission time, effectively turning the webhook into the primary configuration injection point. This means that Spark pod configuration (volumes, environment variables, GPU resources, security contexts, init containers, sidecar containers, DNS config, scheduling constraints) is not embedded in the driver pod spec at submission time — it is applied dynamically by the webhook based on the SparkApplication spec. This design enables partial restarts (via the `PartialRestart` feature gate): changing executor-only fields like `NodeSelector` or `Tolerations` in the SparkApplication spec causes new executor pods to pick up the changes via the webhook without restarting the driver.

Certificate management deserves attention. The operator implements its own self-signed CA infrastructure (`pkg/certificate/certificate.go`) with RSA-2048 keys, 10-year CA validity, and automatic regeneration when certificates are within 180 days of expiry. Two dedicated controllers (`MutatingWebhookConfiguration` and `ValidatingWebhookConfiguration` controllers) watch their respective webhook configurations and inject the current CA bundle. This self-contained certificate lifecycle avoids a hard dependency on cert-manager while supporting it as an optional alternative. The webhook TLS configuration integrates with the OpenShift cluster TLS security profile (`config.openshift.io/v1 APIServer`), reading cipher suites and minimum TLS version from the cluster-level setting and mapping OpenSSL cipher names to Go `crypto/tls` constants.

The scheduler integration layer (`internal/scheduler/`) uses a registry pattern to support three batch schedulers without coupling the controller to any specific one. Volcano creates `PodGroup` CRDs with min-member counts; Kube-scheduler plugins uses the `scheduling.x-k8s.io` PodGroup API; YuniKorn uses annotations (no CRDs). This pluggability is important for RHOAI deployments where the platform operator controls which scheduling infrastructure is available. The Kueue integration is label-based (`pkg/common/kueue.go`), adding `kueue.x-k8s.io/queue-name` labels to driver pods without any direct API interaction.

One crypto concern: `pkg/util/sparkapplication.go` imports `crypto/md5` to generate submission IDs by hashing the SparkApplication spec. This is a non-cryptographic use (idempotency key generation), so it does not compromise FIPS compliance, but it will trigger static analysis tools that flag MD5 imports.
