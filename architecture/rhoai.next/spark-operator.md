## Purpose

**Short**: Kubernetes operator for managing Apache Spark applications, scheduled Spark jobs, and Spark Connect servers on Kubernetes/OpenShift clusters.

**Detailed**: The Spark Operator is a Kubeflow-originated Kubernetes operator that automates the full lifecycle of Apache Spark workloads on Kubernetes. It provides three Custom Resource Definitions (CRDs): `SparkApplication` for one-shot Spark jobs, `ScheduledSparkApplication` for cron-scheduled recurring Spark jobs, and `SparkConnect` (alpha) for persistent Spark Connect server instances. The operator handles application submission via `spark-submit`, driver/executor pod lifecycle management, automatic retries, TTL-based garbage collection, and optional batch scheduler integration (Volcano, Yunikorn, kube-scheduler plugins).

The operator is deployed as two separate processes within the RHOAI platform: a controller manager that reconciles Spark CRDs and manages pod lifecycles, and a webhook server that mutates and validates Spark resources at admission time. The webhook server performs extensive pod mutation — injecting volumes, environment variables, sidecar containers, Prometheus monitoring configuration, security contexts, and scheduling constraints into Spark driver and executor pods. Both processes expose Prometheus metrics and support leader election for high availability. The operator also creates Kubernetes Ingress resources for Spark Web UI access and manages its own TLS certificates (either self-signed or via cert-manager).

## Architectural Analysis

The Spark Operator follows a clean separation-of-concerns pattern by running the controller and webhook as separate Deployment processes. This is unusual compared to many Kubernetes operators that run webhooks within the same process as controllers. The benefit is independent scaling and failure isolation — a webhook outage does not affect running reconciliation, and vice versa. Both processes share the same container image (the `entrypoint.sh` dispatches to different subcommands: `controller start` vs `webhook start`), keeping the build pipeline simple despite the dual-deployment topology.

The webhook server performs remarkably extensive pod mutation through the `SparkPodDefaulter`. Rather than a simple label injection, this webhook reconstructs significant portions of the pod spec — injecting volumes (Spark config, Hadoop config, Prometheus config, local scratch directories), environment variables, sidecar containers, init containers, GPU resource limits, security contexts, affinity rules, tolerations, DNS configuration, and lifecycle hooks. This design centralizes pod configuration in the webhook rather than spreading it across the controller's reconcile logic, which means the Spark driver pods submitted via `spark-submit` are enriched at admission time with all platform-specific concerns.

The operator supports three batch scheduler integrations (Volcano, Yunikorn, kube-scheduler-plugins) through a clean `Scheduler` interface with a global registry pattern. Each scheduler implementation creates scheduler-specific resources (PodGroups for Volcano and kube-scheduler-plugins, annotations for Yunikorn) before the Spark application is submitted, enabling gang scheduling of driver and executor pods. The Volcano integration includes a runtime check for the PodGroup CRD existence via the API extensions API, providing graceful degradation when the scheduler is not installed. The Yunikorn integration is annotation-based, requiring no CRD dependencies.

The `SparkConnect` controller (v1alpha1) represents a newer architectural direction — persistent Spark sessions rather than one-shot jobs. It manages a ConfigMap (executor pod template), a server Pod, and a multi-port Service (driver-rpc:7078, blockmanager:7079, web-ui:4040, spark-connect:15002) using a CreateOrUpdate pattern. The executor pod tracking is done via label-filtered watches rather than owner references, since executors are launched by the Spark Connect server process rather than by the controller directly. This is a notable design choice that enables the operator to monitor executor lifecycle without needing to create executors itself.

The kustomize overlay structure for ODH and RHOAI is nearly identical, differing primarily in namespace (opendatahub vs redhat-ods-applications) and the `app.kubernetes.io/part-of` label. Both overlays add a NetworkPolicy (`spark-operator-allow-internal`) that permits ingress to Spark pods on ports 7078, 7079, and 4040 from within the namespace and from OpenShift ingress. The overlays delete the Namespace resource from the base, delegating namespace management to the parent platform operator. Image parameterization flows through a `configMapGenerator` with `behavior: replace` in overlays, allowing each distribution to pin different image versions.
