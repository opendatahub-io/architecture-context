## Purpose

**Short**: Kubernetes operator that automates the lifecycle management of Apache Spark applications, scheduled Spark jobs, and Spark Connect servers on OpenShift/Kubernetes clusters.

**Detailed**: The Spark Operator (originally from the Kubeflow project) provides a declarative way to submit, schedule, and manage Apache Spark workloads on Kubernetes. It defines three Custom Resource Definitions — `SparkApplication`, `ScheduledSparkApplication`, and `SparkConnect` — that users create to describe their Spark jobs, cron-scheduled jobs, and persistent Spark Connect servers respectively.

The operator consists of two separately deployed components sharing the same container image: a **controller** that reconciles the three CRDs (managing the full lifecycle of Spark driver/executor pods, services, ingresses, and monitoring configuration) and a **webhook** server that validates and defaults CRs at admission time and mutates Spark pods to inject operator-managed configuration (environment variables, volumes, scheduling, security contexts, and Prometheus monitoring).

Within the RHOAI and ODH platforms, the Spark Operator is deployed by the platform operator (rhods-operator or opendatahub-operator) into the platform namespace. It watches user namespaces for SparkApplication CRs and orchestrates Spark workloads using the cluster's existing Spark container images, integrating with batch schedulers (Volcano, Yunikorn, kube-scheduler plugins), Prometheus monitoring, and Kubernetes RBAC.

## Architectural Analysis

The Spark Operator follows a clean separation of concerns between its two deployment components. The **controller** handles all reconciliation logic — it never serves webhook traffic — while the **webhook** handles admission control and never reconciles CRDs. Both share the same container image but are started with different subcommands (`controller start` vs `webhook start`). This split enables independent scaling and failure isolation: a webhook outage blocks new CR submissions but doesn't affect running Spark applications, and vice versa.

The **SparkApplication controller** implements a 13+ state machine that faithfully models the Spark job lifecycle. The state machine handles a rich set of transitions including submission retries with configurable backoff, suspension/resumption (via `spec.suspend`), TTL-based cleanup after completion, and invalidation-triggered reruns. A notable design decision is that the controller does not directly create driver/executor pods — it invokes the `spark-submit` binary as a local process, letting Spark's native Kubernetes scheduler backend handle pod creation. This preserves full Spark compatibility but means the operator image must include the Spark distribution (Java 17, PySpark 4.0.1) alongside the Go binary, resulting in a large image.

The **pod mutating webhook** is architecturally critical. When Spark creates driver/executor pods, the webhook intercepts them (identified by the `sparkoperator.k8s.io/launched-by-spark-operator` label) and applies 25+ mutation operations in a fixed order — including environment variables, volumes, container ports, init/sidecar containers, GPU resources, Prometheus monitoring agents, security contexts, scheduling configuration (priority class, scheduler name, affinity, tolerations), and DNS config. This design means the SparkApplication spec fields that affect pod shape are applied at admission time rather than by the controller, which is elegant but creates a strong coupling between the webhook and the controller's submission logic.

The **batch scheduler integration** uses an extensible factory/registry pattern supporting three schedulers: Volcano (creates `PodGroup` CRs with queue/priority support, handles both client and cluster deployment modes), kube-scheduler plugins (creates `PodGroup` CRs from the scheduler-plugins API), and Yunikorn (annotation-based, no external CRDs, includes sophisticated memory overhead calculations for Java vs Python apps). The registry is thread-safe and can be extended without modifying controller code. Scheduler cleanup is invoked on application termination to remove PodGroup resources.

The **certificate management** system supports dual modes — internal self-signed certificates (with RSA key generation, 10-year validity, and proactive validation 180 days ahead of expiry) or cert-manager integration. Dedicated controllers for MutatingWebhookConfiguration and ValidatingWebhookConfiguration keep the CA bundle synchronized. The TLS configuration uses Go's default cipher suites (no hardcoded restrictions), and HTTP/2 is explicitly disabled by default to mitigate CVE-2023-44487 (HTTP/2 Rapid Reset). One minor crypto note: MD5 is used in `pkg/util/sparkapplication.go` for DNS name hash truncation — this is non-cryptographic use (cosmetic name shortening) and does not affect FIPS compliance.

The **kustomize deployment structure** demonstrates mature multi-distribution support. The base configuration (`config/default`) uses `configMapGenerator` and `replacements` for image parameterization, with overlays for ODH (namespace: opendatahub) and RHOAI (namespace: redhat-ods-applications) that add NetworkPolicies, monitoring labels, and distribution-specific labels while removing the base Namespace resource (since the platform operator manages namespace creation). Both overlays include identical NetworkPolicies allowing Spark pod communication on ports 7078 (driver-rpc), 7079 (blockmanager), and 4040 (spark-ui) from same-namespace pods and OpenShift ingress.
