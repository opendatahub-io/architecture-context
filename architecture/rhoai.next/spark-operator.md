## Purpose

**Short**: Kubernetes operator that automates the lifecycle management of Apache Spark applications, scheduled Spark applications, and Spark Connect servers on OpenShift/Kubernetes clusters.

**Detailed**: The Spark Operator (originally from Kubeflow, forked by Red Hat) watches for SparkApplication, ScheduledSparkApplication, and SparkConnect custom resources and manages their complete lifecycle. For SparkApplication CRs, it orchestrates a 13-state reconciliation state machine that handles submission via `spark-submit`, driver/executor pod monitoring, web UI service exposure, Prometheus metrics integration, and TTL-based cleanup. For ScheduledSparkApplication CRs, it provides cron-based scheduling with concurrency policies (Allow/Forbid/Replace). For SparkConnect CRs (v1alpha1), it manages long-running Spark Connect servers with executor pools.

The operator deploys as two separate components: a **controller manager** that reconciles CRs and manages Spark workloads, and a **webhook server** that validates and mutates SparkApplication resources and injects configuration into Spark driver/executor pods. It integrates with multiple batch scheduling systems (Volcano, Yunikorn, kube-scheduler PodGroups) and provides comprehensive Prometheus metrics for application lifecycle tracking. The operator is deployed via kustomize manifests with overlays for both ODH (opendatahub namespace) and RHOAI (redhat-ods-applications namespace).

## Architectural Analysis

The Spark Operator implements a two-component deployment architecture that cleanly separates the control plane (controller manager) from the admission control plane (webhook server). Both components share a single container image but are started with different subcommands (`controller start` vs `webhook start`). This design enables independent scaling and failure isolation — a webhook failure doesn't prevent the controller from managing existing workloads, and a controller failure doesn't prevent new resources from being validated.

The SparkApplication controller implements a sophisticated 13-state finite state machine (New → Submitted → Running → Succeeding/Failing → Completed/Failed, with branches for PendingRerun, Invalidating, Suspending, Suspended, Resuming, and FailedSubmission). This is notably more complex than typical Kubernetes operator reconciliation loops and reflects the inherent complexity of Spark application lifecycle management. The state machine handles submission retries with exponential backoff, TTL-based cleanup, and suspension/resumption — a pattern that allows batch workloads to yield cluster resources and resume later. The controller creates resources dynamically (Services, Ingresses, ConfigMaps) based on the application specification rather than from static manifests, making the network architecture data-driven.

The webhook server performs extensive pod mutation that is critical to the operator's functioning. When Spark creates driver and executor pods (via spark-submit), the mutating webhook intercepts pod creation and injects up to 26 categories of configuration including environment variables, volume mounts, ConfigMap references, security contexts, scheduler names, sidecar containers, and Prometheus annotations. This pod mutation design avoids the need for a custom Spark scheduler backend — the operator uses standard `spark-submit` and relies on webhooks to customize the resulting pods. The label selector `sparkoperator.k8s.io/launched-by-spark-operator=true` gates which pods are mutated, preventing interference with non-operator Spark deployments.

The batch scheduler abstraction is well-designed with a registry/factory pattern supporting three implementations (Volcano, Yunikorn, kube-scheduler). Each scheduler creates PodGroup-equivalent resources to enable gang scheduling — ensuring that a Spark application's driver and all executors are scheduled atomically or not at all. This prevents resource deadlocks where a driver holds resources but executors can't be scheduled. The Volcano integration auto-detects CRD availability at startup, gracefully degrading if Volcano is not installed.

A notable FIPS compliance concern exists: the operator is built with `CGO_ENABLED=0`, producing a statically linked Go binary that uses pure-Go crypto instead of OpenSSL. This will fail Red Hat's check-payload FIPS validation, which requires dynamically linked binaries using system OpenSSL. The `crypto/md5` import in `pkg/util/sparkapplication.go` is used solely for non-cryptographic name hashing (truncating DNS-safe names to 63 characters) and is not a security concern, but the static linking issue needs resolution for FIPS-compliant RHOAI deployments.
