## Purpose

**Short**: REST API service for orchestrating LLM evaluation jobs, managing evaluation providers, benchmark collections, and tracking results via MLflow integration.

**Detailed**: Eval Hub is the backend orchestration service for the RHOAI evaluation platform. It provides a REST API (OpenAPI 3.1) for managing the lifecycle of LLM evaluation jobs — creating, monitoring, and cancelling evaluation runs across configurable benchmark providers (Lighteval, lm-evaluation-harness, Garak, GuideLLM, IBM CLEAR). The service supports both Kubernetes-native and local execution runtimes: in Kubernetes mode, it creates Batch Jobs with native sidecar containers (KEP-753) that proxy requests to the eval-hub API, MLflow tracking server, and OCI registries; in local mode, it spawns shell subprocesses for development and testing.

The platform uses a pluggable provider/benchmark architecture where evaluation providers define available benchmarks (with metrics, pass criteria, and runtime configurations), and collections group benchmarks into reusable evaluation suites. Results are tracked via MLflow experiment integration with workspace-scoped multi-tenancy. The service implements Kubernetes-native authentication (TokenReview) and authorization (SubjectAccessReview) with configurable RBAC rules mapped to API endpoints, and supports multi-tenant data isolation via the X-Tenant header.

The repository also ships a Model Context Protocol (MCP) server (`evalhub-mcp`) for AI agent integration, a Python wheel packaging layer for distributing the Go binary as a pip-installable package, and comprehensive OpenTelemetry and Prometheus observability.

## Architectural Analysis

Eval Hub implements a clean three-tier architecture: HTTP handler layer, storage abstraction, and runtime abstraction. The handler layer uses Go 1.22+ `http.ServeMux` with pattern-based routing (no external router dependency). The storage layer abstracts over SQLite and PostgreSQL via a `Storage` interface with context-based tenant scoping (`WithTenant`, `WithOwner`), enabling multi-tenant data isolation without modifying query logic. The runtime layer provides a `Runtime` interface with Kubernetes and local implementations, allowing the same codebase to serve production cluster deployments and local developer testing.

The Kubernetes runtime creates one Batch Job per benchmark within an evaluation, each with a consistent resource structure: a ConfigMap containing `job.json` and `sidecar_config.json`, an optional init container for S3 test data, a native sidecar container (KEP-753) for API proxying, and the evaluation adapter container from the provider's configured image. The sidecar uses the `RestartPolicy: Always` init container pattern to ensure the proxy is available before the adapter starts (with a startup probe of 30x2s intervals). This design means the adapter never needs direct network access to eval-hub, MLflow, or OCI registries — all communication goes through the sidecar on localhost, which handles token injection, TLS verification, and workspace scoping.

The authentication/authorization system is notably sophisticated for a non-operator service. It uses Kubernetes-native TokenReview for authentication (with 2-minute cache) and SubjectAccessReview for authorization (5-minute allow cache, 30-second deny cache), with authorization rules defined in a YAML config file (`auth.yaml`) that maps API paths and HTTP methods to Kubernetes resource attributes. The template system supports dynamic namespace extraction from the `X-Tenant` header and verb derivation from HTTP methods, effectively bridging REST API authorization to Kubernetes RBAC. This design allows cluster administrators to control eval-hub access using standard Kubernetes RBAC tools (ClusterRoles, RoleBindings) without custom admission webhooks.

The configuration system uses a cascading priority model: YAML config file < environment variable bindings < operator-mounted config overrides < secrets from files. The `config.Watcher` monitors provider and collection config directories using `fsnotify`, with debouncing for Kubernetes ConfigMap symlink swap patterns (atomic updates). This allows the operator to update evaluation providers and benchmark collections at runtime without restarting the service. Secret values are automatically redacted in logs, and URLs in secrets have passwords stripped while preserving usernames for debugging.

The MCP server (`evalhub-mcp`) is architecturally independent — it uses the `pkg/evalhubclient` REST client to access eval-hub rather than sharing internal packages. This clean separation means the MCP server can run anywhere with network access to the eval-hub API, either as a stdio server embedded in AI agent toolchains or as an HTTP server for multi-client access. The Python wheel packaging (`python-server/`) embeds the compiled Go binary inside a pip-installable package, enabling `pip install eval-hub-server` for local development without Go toolchain setup.
