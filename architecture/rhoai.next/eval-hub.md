## Purpose

**Short**: Evaluation Hub is a REST API service for orchestrating LLM benchmark evaluations across multiple evaluation providers (LM Evaluation Harness, Garak, Lighteval, GuideLLM) with integrated experiment tracking via MLflow.

**Detailed**: Eval-hub provides a centralized API for managing LLM evaluation workflows on Red Hat OpenShift AI. It allows users to define evaluation jobs specifying a model endpoint, benchmark suite, and evaluation provider, then orchestrates the execution as Kubernetes batch Jobs with sidecar containers for telemetry, authentication proxying, and result collection.

The service manages three core resource types: **providers** (evaluation framework definitions with benchmark catalogs), **collections** (curated benchmark suites with weighted scoring), and **evaluation jobs** (execution instances). Each evaluation job spawns Kubernetes Jobs containing an adapter container (the evaluation runtime), a native sidecar for proxy/auth (eval-runtime-sidecar), and optionally an init container for S3 test data download (eval-runtime-init). Results are reported back via HTTP callbacks, optionally tracked in MLflow experiments, and optionally exported to OCI registries.

The architecture supports multi-tenancy through Kubernetes RBAC-based authentication and authorization using SubjectAccessReview (SAR), with tenant scoping via the X-Tenant header mapped to Kubernetes namespaces. The service supports both PostgreSQL and SQLite backends, hot-reloads provider/collection configurations from disk, and provides comprehensive observability through OpenTelemetry tracing, Prometheus metrics, and structured logging.

## Architectural Analysis

Eval-hub follows a clean multi-binary architecture where a single Dockerfile produces three binaries (`eval-hub`, `eval-runtime-sidecar`, `eval-runtime-init`) that serve different roles in the evaluation workflow. The main API server creates Kubernetes Jobs that use the sidecar binary as a native sidecar container (KEP-753) and the init binary as an init container. This design avoids separate container images while keeping concerns separated at the binary level.

The sidecar proxy pattern is particularly noteworthy. Rather than having the adapter container directly authenticate with eval-hub, MLflow, and OCI registries, all outbound HTTP calls are routed through the sidecar on localhost:8080. The sidecar resolves authentication tokens from projected ServiceAccount tokens, Docker config secrets, and the OCI registry challenge-response flow. This decouples evaluation adapters from authentication concerns entirely — adapters only need to know about localhost URLs. Token caching with configurable TTL (default 5 minutes) using atomic pointers prevents unnecessary file reads and auth round-trips.

The multi-tenancy model uses an interesting approach: rather than implementing its own RBAC, eval-hub delegates authentication to the Kubernetes API server via `DelegatingAuthenticatorConfig` (TokenReview) and authorization via `DelegatingAuthorizerConfig` (SubjectAccessReview). The auth config maps HTTP endpoints to Kubernetes resource attributes with API groups like `trustyai.opendatahub.io` and `mlflow.kubeflow.org`, allowing platform administrators to control access using standard Kubernetes RBAC (ClusterRoles/RoleBindings). The X-Tenant header maps directly to a Kubernetes namespace for SAR checks, providing namespace-scoped multi-tenancy.

The storage layer uses a dual-backend strategy with PostgreSQL (via pgx) for production and SQLite (via modernc.org pure-Go implementation) for development and testing. Entity data is stored as JSONB (PostgreSQL) or JSON TEXT (SQLite), allowing flexible schema evolution without database migrations. Query filtering supports JSON path operations for both backends. The configuration system supports a layered cascade (config files -> CONFIG_PATH override -> environment variables -> secret files) with hot-reloading via filesystem watchers that handle Kubernetes ConfigMap atomic symlink swaps with 500ms debounce.

The evaluation job lifecycle is well-designed with a clear state machine (pending -> running -> completed/failed/partially_failed/cancelled). Each benchmark within a job tracks its own state independently, and the overall job state is computed from benchmark states using a weighted scoring algorithm. Pass/fail criteria support both lower-is-better and higher-is-better metrics with configurable thresholds at the benchmark, collection, and job levels. The test results computation uses weighted averages across benchmarks for an overall job score.
