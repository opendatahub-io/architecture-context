## Purpose

**Short**: A lightweight REST API service for orchestrating LLM evaluations across multiple backends, with an MCP server interface for AI-native tooling integration.

**Detailed**: EvalHub is the evaluation orchestration service for Red Hat OpenShift AI. It provides a REST API (`/api/v1/evaluations/`) for managing LLM evaluation jobs, benchmark providers (lm-evaluation-harness, RAGAS, Garak, GuideLLM, LightEval), and benchmark collections. When deployed on Kubernetes, the service creates batch Jobs for each evaluation benchmark, using a sidecar proxy pattern for credential injection and an init container for S3 test-data download. Evaluation results are tracked through optional MLflow experiment integration. The service also supports a local execution mode for development without a Kubernetes cluster.

EvalHub is managed by the TrustyAI Service Operator via the `EvalHub` custom resource (`trustyai.opendatahub.io/v1alpha1`). The operator handles deployment of the eval-hub API pod (fronted by kube-rbac-proxy for authentication), the MCP server pod, and all associated networking. The MCP server (`evalhub-mcp`) exposes evaluation capabilities through the Model Context Protocol, enabling AI coding assistants (Cursor, VS Code, Claude Code) to discover providers, submit evaluations, and monitor job progress.

The service uses Go's standard `net/http` router, structured logging via zap/slog, Prometheus metrics on a dedicated port, OpenTelemetry distributed tracing, and a pluggable SQL storage layer (SQLite for development, PostgreSQL for production). Provider and benchmark configurations are declared in YAML files shipped with the container image and hot-reloaded via filesystem watchers.

## Architectural Analysis

EvalHub follows a clean separation between its HTTP API layer and its evaluation execution layer. The API server uses Go's standard `net/http.ServeMux` without any web framework, achieving simplicity and control over request handling. The `ExecutionContext` pattern provides type-safe per-request state (tenant, user, request ID, logger) without relying on globals or middleware-injected context values. This pattern propagates cleanly through the handler -> storage -> runtime call chain, enabling consistent multi-tenant isolation at every layer.

The credential injection architecture is notably well-designed. Rather than mounting real model credentials into the evaluation adapter container, the K8s runtime creates ephemeral "ref" secrets where real API keys are replaced with reference tokens (e.g., `api-key:ref`). The sidecar proxy intercepts outbound model calls, detects ref tokens in the Authorization header, and substitutes real credentials from its own Secret mount. This means the adapter container (which runs untrusted community evaluation code) never has access to real credentials. The sidecar also handles multi-model scenarios where different models require different endpoints and credentials, using `{key}_url` files in Secret mounts to route requests dynamically.

The MCP server (`evalhub-mcp`) adds an AI-native interface layer on top of the REST API. It supports three transports (stdio for IDE integration, Streamable HTTP for web clients, and legacy HTTP+SSE for older MCP clients), implements cached auto-completion for resource IDs with a 30-second TTL, and provides structured prompts for evaluation workflows (EDD, model evaluation, run comparison). The MCP server operates as an independent deployment, communicating with the eval-hub API over HTTP with its own kube-rbac-proxy sidecar. This separation allows independent scaling and lifecycle management.

The storage layer uses a JSONB-first design where evaluation job entities (config, status, results) are stored as JSON documents in PostgreSQL (or SQLite for development). This provides schema flexibility for the rapidly evolving evaluation API while maintaining SQL query capabilities for filtering and pagination. The `ReadCommitted` isolation level for PostgreSQL was a deliberate choice to avoid serialization conflicts (`SQLSTATE 40001`) under concurrent status updates from multiple benchmark sidecar pods reporting progress simultaneously.

One architectural risk is the absence of RPM-level lock files (`rpms.lock.yaml`) and Hermeto prefetch integration on the main branch. While Go dependencies are hermetically locked via `go.sum`, the UBI9 base image's RPM packages are not pinned, creating a supply chain reproducibility gap. This is likely addressed on downstream release branches but should be verified for each release cut.
