## Purpose

**Short**: Eval Hub is a centralized evaluation orchestration service for running LLM benchmarks across multiple evaluation frameworks (LM Evaluation Harness, Lighteval, Garak, GuideLLM) as Kubernetes Jobs, with result tracking via MLflow and artifact export to OCI registries.

**Detailed**: Eval Hub provides a REST API for managing evaluation jobs, benchmark providers, and benchmark collections for Large Language Models. It orchestrates evaluation workloads by creating Kubernetes Jobs with adapter containers (evaluation runtimes like lm-eval or lighteval) and a sidecar proxy container that handles authenticated communication back to the Eval Hub API, MLflow experiment tracking, and OCI registry artifact export.

The service supports multi-tenancy through Kubernetes TokenReview and SubjectAccessReview-based authentication and authorization, scoping all resources (evaluations, providers, collections) to tenant namespaces. It stores evaluation metadata in either SQLite (for development) or PostgreSQL (for production), supports hot-reloading of provider and collection configurations from ConfigMap-mounted YAML files, and exposes Prometheus metrics and OpenTelemetry tracing for observability.

Eval Hub also includes an MCP (Model Context Protocol) server that exposes evaluation capabilities as tools, resources, and prompts for AI agent integration, allowing models to programmatically submit evaluations, query job status, and browse available benchmarks through the standardized MCP interface.

## Architectural Analysis

Eval Hub follows a clean hub-and-spoke architecture where the central API server orchestrates evaluation workloads across multiple runtime providers. The design decouples the evaluation orchestration logic from the actual benchmark execution by using Kubernetes Jobs with pluggable adapter containers. Each provider (lm-eval, lighteval, garak, guideLLM) is defined declaratively in YAML configuration files with its container image, entrypoint, resource requirements, and benchmark catalog. The eval-hub server creates a Job spec with the adapter container and injects a sidecar proxy that handles all authenticated communication — the adapter only needs to make unauthenticated localhost calls to the sidecar.

The sidecar proxy pattern is architecturally significant. It solves three authentication challenges simultaneously: (1) injecting ServiceAccount tokens for eval-hub API callbacks, (2) projecting ROSA/STS-compatible tokens for MLflow Kubernetes-auth RBAC, and (3) performing Docker config.json-based OCI registry authentication for artifact export. The sidecar uses KEP-753 native sidecars (init container with `restartPolicy=Always`) to ensure it starts before and outlives the adapter container. This approach cleanly separates security concerns from evaluation runtime concerns.

Multi-tenancy is enforced at multiple layers. The HTTP middleware performs Kubernetes TokenReview to authenticate requests and SubjectAccessReview with configurable resource attribute mapping to authorize them. The database layer scopes all queries with tenant filtering, and MLflow experiments are namespaced to tenant namespaces for workspace-level RBAC. System-provided resources (built-in providers and collections) are visible to all tenants but owned by the "system" identity.

The configuration hot-reload system using `fsnotify` with Kubernetes ConfigMap symlink-swap detection is well-designed for operator-managed deployments. Provider and collection definitions can be updated without pod restarts, and the debounce mechanism (500ms) handles the rapid file events that occur during atomic ConfigMap updates. This enables the RHOAI operator to manage provider catalogs dynamically.

The MCP server is an interesting addition that bridges eval-hub into the AI agent ecosystem. It exposes evaluation capabilities as standardized MCP tools (submit, cancel, status), resources (providers, benchmarks, collections, jobs), and prompted workflows (evaluate_model, compare_runs). This allows AI agents to programmatically run evaluations and monitor results through the MCP protocol, either via stdio (for local development) or HTTP transport.
