## Purpose

**Short**: A lightweight REST API service for orchestrating LLM evaluations across multiple backends, tracking experiments via MLflow, and running natively on OpenShift.

**Detailed**: EvalHub is a Go-based REST API server that serves as the orchestration hub for LLM evaluation workflows within the Red Hat OpenShift AI platform. It receives evaluation job requests from users or AI agents, validates them against registered providers and benchmark collections, then dispatches evaluation workloads as Kubernetes Jobs with sidecar containers that proxy authentication and route results. The service supports multiple evaluation frameworks—including lm-evaluation-harness (167 benchmarks), RAGAS, Garak, GuideLLM, LightEval, and MTEB—through a pluggable provider architecture where frameworks are declared in YAML configuration files.

The service is deployed and managed by the TrustyAI Service Operator via the `EvalHub` custom resource (`trustyai.opendatahub.io/v1alpha1`). In cluster mode, authentication is enforced by kube-rbac-proxy as a sidecar, which sets `X-Tenant` and `X-User` headers before forwarding requests to the eval-hub process. Evaluation job pods include a sidecar (`eval-runtime-sidecar`) that reverse-proxies requests to the eval-hub API, MLflow tracking server, OCI registries, and model inference endpoints, handling authentication token resolution at each hop. An init container (`eval-runtime-init`) downloads test data from S3-compatible storage when required. A fourth binary (`evalhub-mcp`) implements the Model Context Protocol for AI agent integration, exposing evaluation tools, resources, and workflow prompts via stdio or HTTP transports.

## Architectural Analysis

EvalHub implements a clean separation between the API server and the evaluation execution environment through a pluggable runtime abstraction. The `abstractions.Runtime` interface (`RunEvaluationJob` and `DeleteEvaluationJobResources`) decouples the HTTP handler layer from the execution backend. Two implementations exist: `K8sRuntime` for cluster deployments and `LocalRuntime` for development/CLI usage. The Kubernetes runtime creates Jobs with a three-container pod architecture: an optional init container for S3 test data, a native sidecar (KEP-753 `restartPolicy: Always` init container) for proxying, and the evaluation adapter container. This pattern ensures the sidecar is fully ready (health-checked via startup probe) before the adapter begins execution.

The credential injection system is a notable security design. When a model authentication secret contains proxy-injectable keys (`api-key`, `*_api-key`, `*_url`), the k8s runtime creates an ephemeral "internalModelRef" secret containing synthetic ref-token values (e.g., `api-key:ref`). The adapter container receives this synthetic secret via a projected volume, while the sidecar receives the real credentials secret. When the adapter makes model API calls with a ref-token Bearer header, the sidecar's model proxy intercepts the request, replaces the ref-token with the actual credential from its mount, and forwards the request to the real model endpoint. This ensures the adapter framework code never sees real credentials. For secrets that only contain passthrough keys (e.g., `hf-token`, `ca_cert`), the adapter mounts the real secret directly without credential injection.

The storage layer uses a SQL abstraction over SQLite (development) and PostgreSQL (production) with dialect-specific implementations under `storage/sql/sqlite` and `storage/sql/postgres`. Evaluation jobs, providers, and collections are stored as JSON documents in SQL tables, supporting tenant-scoped multi-tenancy via `WithTenant` query chaining. The config watcher (`config.SetupWatcher`) uses `fsnotify` to watch provider and collection YAML files for changes, enabling live reloads of system-scoped resources without service restart.

The MCP server is architecturally independent — it uses the `pkg/evalhubclient` public Go client to communicate with the eval-hub API over HTTP, rather than directly importing internal packages. This means it can run as a standalone process pointing at any eval-hub instance, or be deployed as a sidecar within the same pod. The three MCP prompts (`edd_workflow`, `evaluate_model`, `compare_runs`) encode domain expertise about evaluation methodology, making the MCP server more than a simple API wrapper — it provides structured guidance for AI agents performing evaluations. Completion caching with a 30-second TTL keeps the MCP experience responsive while avoiding stale data.

The Kueue integration is lightweight but meaningful: when an evaluation job specifies a `queue` with `kind: kueue`, the Job label `kueue.x-k8s.io/queue-name` is set and nodeSelector is intentionally omitted. This delegates GPU scheduling to Kueue's ResourceFlavor admission, allowing the cluster's Kueue configuration to control which nodes receive evaluation workloads based on available accelerators and quotas.
