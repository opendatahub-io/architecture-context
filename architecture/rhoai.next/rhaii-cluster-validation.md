## Purpose

**Short**: Preflight validation probe for xKS clusters that verifies GPU availability, RDMA connectivity, network bandwidth, required CRDs, and inference readiness before AI workload deployment.

**Detailed**: RHAII Cluster Validation is a kubectl plugin (`kubectl rhaii-validate`) and standalone CLI tool that orchestrates comprehensive GPU cluster readiness checks. It auto-detects cloud platforms (AKS, EKS, CoreWeave, OpenShift) and applies platform-specific thresholds, then deploys ephemeral Kubernetes Jobs onto GPU nodes to validate hardware and network infrastructure.

The tool operates in a three-tier validation model: (1) Cluster-level checks via the Kubernetes API (CRD presence, operator health), (2) Per-node checks via privileged Job pods (GPU driver version, ECC status, RDMA device discovery, GPU-NIC-NUMA-PCIe topology mapping), and (3) Multi-node network tests (TCP bandwidth via iperf3, TCP latency via built-in echo server, RDMA bandwidth via ib_write_bw with GPUDirect, and RDMA connectivity mesh via ibv_rc_pingpong). Results are stored in a ConfigMap as JSON and printed as a human-readable table or machine-parseable JSON, with PASS/WARN/FAIL/SKIP status per check.

The component ships two container images: a Go binary (the validator agent) and a tools image (iperf3, perftest, CUDA runtime for RDMA bandwidth testing). The validator agent runs both as the CLI orchestrator (creating Jobs, collecting logs, reporting results) and as the per-node check agent (deployed inside Job pods via the `run` subcommand). The tools image is deployed by the orchestrator for iperf3, RDMA bandwidth, and pingmesh tests.

## Architectural Analysis

The RHAII Cluster Validation tool follows a unique "self-deploying agent" pattern where the same Go binary serves dual roles: as a CLI orchestrator running on the user's workstation (or in CI), and as a per-node agent running inside privileged Kubernetes Job pods. The CLI creates Jobs that reference the same container image with the hidden `run` subcommand, effectively cloning itself onto each GPU node. This eliminates the need for a separate agent image or daemon, simplifying the deployment model to a single binary that can both manage and execute checks.

The GPU-NIC topology discovery system is sophisticated, implementing four distinct pairing strategies (PCIe distance, NUMA affinity, NUMA load balancing, and intra-host bandwidth probing). The bandwidth probe is particularly notable: when PCIe topology is flat (common in cloud VMs), the tool deploys additional privileged Jobs that run `ib_write_bw` in loopback mode to empirically determine the fastest GPU-NIC pairs. This data-driven approach avoids making assumptions about virtual PCIe topologies that may not reflect actual hardware performance. The topology results are stored in a ConfigMap and reused by subsequent rdma-ping and rdma-bandwidth runs, allowing incremental validation without re-running the expensive topology discovery.

The pingmesh RDMA connectivity testing uses a round-robin tournament scheduling algorithm (`roundRobinSchedule`) to test all N-choose-2 node pairs while maximizing parallelism. Disjoint pairs within each round execute concurrently, and each pair gets up to 3 retry attempts. Results are classified into "rail" (same NIC index across nodes) and "cross-rail" (different NIC indices) categories, enabling diagnosis of rail-specific fabric issues. Failed pairs are stored in a separate ConfigMap for detailed debugging.

Platform detection relies on a priority-ordered heuristic: OpenShift is checked first (via `node.openshift.io/os_id` label) because OCP can run on AWS/Azure infrastructure, then CoreWeave (via label substring matching), then cloud provider detection via node `providerID` prefixes (`azure://`, `aws://`). Each platform has embedded YAML configuration with appropriate default thresholds (e.g., OCP has RDMA PD pass threshold of 180 Gbps vs. the more conservative defaults for other platforms).

A key architectural decision is the separation of "check images" from "tools images" -- the validator binary runs GPU and RDMA discovery checks (using nvidia-smi/amd-smi and sysfs), while the heavier tools image (with CUDA runtime and perftest) is only deployed for bandwidth and connectivity tests. This keeps the primary agent image small and fast to pull, while the larger tools image (~756K of CUDA runtime plus compiled perftest) is only needed when RDMA bandwidth testing is requested. The TCP latency test notably avoids the tools image entirely by implementing a custom Go echo server/client pair (port 12865), replacing the earlier netperf dependency that required EPEL packages incompatible with Red Hat production images.
