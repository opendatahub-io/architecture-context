## Purpose

**Short**: Preflight validation tool for GPU clusters, verifying GPU hardware, RDMA networking, and cross-node bandwidth readiness before AI/ML workload deployment.

**Detailed**: RHAII Cluster Validation is a kubectl plugin (`kubectl rhaii-validate`) that performs hardware-level preflight checks on GPU clusters to ensure they are ready for AI/ML inference and training workloads. It operates as a two-tier validation system: Tier 1 (API checks) validates required CRDs (Gateway API, InferencePool, LeaderWorkerSet, cert-manager) and operator health (cert-manager, Istio, LeaderWorkerSet) via the Kubernetes API; Tier 2 (hardware checks) deploys privileged per-node Job pods that execute GPU driver validation (NVIDIA/AMD), ECC error detection, GPU-NIC-NUMA topology discovery, RDMA device enumeration, and NIC status verification via `chroot /host`.

The tool also orchestrates multi-node network tests using ring, star, and pairwise topologies. TCP bandwidth is measured via iperf3, TCP latency via a built-in Go echo server/client, RDMA connectivity via ibv_rc_pingpong pingmesh, and RDMA bandwidth via ib_write_bw with GPUDirect CUDA support. For flat PCIe topologies where GPU-NIC pairing cannot be determined from the PCIe tree, an intra-host loopback bandwidth probe measures all GPU×NIC combinations to find optimal pairings. Results are stored as JSON in a ConfigMap for persistence and reported as table or JSON output. The tool supports multi-platform deployment across OpenShift (OCP), AKS, EKS, and CoreWeave with auto-detected platform-specific defaults.

## Architectural Analysis

The rhaii-cluster-validation tool follows a controller pattern where a single Go binary serves dual roles: as a CLI controller on the user's machine (orchestrating Kubernetes Jobs) and as the per-node agent inside those Jobs (executing hardware checks). This dual-use design is enabled by the `run` subcommand, which is hidden from users but invoked by the Job template's entrypoint. The same binary also embeds a TCP latency echo server/client (`tcp-lat-server`/`tcp-lat-client`), replacing the previous netperf dependency that required EPEL (incompatible with Red Hat production images).

Platform auto-detection is notably defensive: it scans all nodes rather than just the first (since master nodes may lack platform labels), and prioritizes OpenShift label detection before cloud provider detection because OCP can run on AWS/Azure. The embedded platform config YAML files provide reasonable defaults for each platform, but users can override any value via a ConfigMap (`rhaii-validate-config`) that the controller creates on first run. This ConfigMap-as-config pattern means changes persist across validation runs without requiring re-detection.

The GPU-NIC topology discovery is particularly sophisticated. On standard PCIe hierarchies, GPUs and NICs are paired by walking the PCIe tree to find the shortest hop count. On flat topologies (where all devices share a single PCIe root complex, common in some cloud VMs), this approach fails — the tool falls back to an intra-host loopback bandwidth probe that measures actual ib_write_bw throughput for every GPU×NIC combination, then uses the measurements to determine optimal pairings. This probe can take ~10 minutes per node for an 8-GPU × 8-NIC matrix (64 tests × 34s each). The topology data is persisted in the report ConfigMap so that subsequent RDMA bandwidth and pingmesh runs can reuse it without re-running node checks.

The two-image architecture (validator + tools) is driven by image size and security constraints. The validator image is a lean UBI9 image (~100MB) containing just the Go binary and basic InfiniBand utilities (ibv_devices, ibstat). The tools image is significantly larger because it compiles perftest (ib_write_bw, ibv_rc_pingpong) from source with CUDA GPUDirect support, requiring the CUDA development toolkit during build. At runtime, only libcudart (~756K) is copied from the builder stage — the GPU driver library (libcuda.so.1) is mounted by the NVIDIA Container Toolkit. This split means the validator binary can run on any architecture (x86_64, arm64) while the tools image is currently x86_64-only due to CUDA/perftest dependencies.

Security-wise, the per-node Jobs require privileged containers with host root mounted at `/host` and hostPID access. This is necessary because GPU driver queries (nvidia-smi), PCIe topology discovery, and RDMA device enumeration require access to host sysfs, procfs, and device files. On OpenShift, the tool dynamically creates a ClusterRoleBinding to grant the `system:openshift:scc:privileged` SCC. The RBAC permissions are relatively narrow (nodes, pods, pods/log, jobs, CRDs) and are cleaned up after validation completes. FIPS compliance is enforced via `GOEXPERIMENT=strictfipsruntime` with CGO_ENABLED=1 in the Konflux Dockerfiles.
