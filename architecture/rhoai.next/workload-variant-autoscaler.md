## Purpose

**Short**: A Kubernetes controller that performs intelligent autoscaling for LLM inference model servers based on saturation, queueing theory, and cost optimization across GPU variant configurations.

**Detailed**: The Workload Variant Autoscaler (WVA) is a global autoscaler for inference model servers serving Large Language Models (LLMs). It works alongside standard Kubernetes HPA and external autoscalers like KEDA to drive the scale subresource of inference deployments. WVA determines optimal replica counts by continuously monitoring request rates and server performance via Prometheus metrics, using KV cache utilization, queue depth, and throughput analysis to make scaling decisions.

WVA introduces the concept of "variants" -- multiple model servers in an InferencePool that all serve the same base model but differ in hardware configuration (GPU type), serving configuration (tensor parallelism, max batch size, quantization), or both. This enables use cases such as prefill/decode disaggregation, batch vs. interactive workload separation, and cost-optimized accelerator selection. The controller emits optimization metrics to Prometheus which are consumed by HPA or KEDA to actuate the scaling decisions.

The controller supports three scaling analysis paths: V1 percentage-based saturation, V2 token-based saturation with universal threshold post-step, and queueing model analysis using M/M/c queueing theory for SLO-aware scaling. Configuration is namespace-aware with three-tier resolution (namespace-local > global > system defaults) managed through ConfigMaps that can be updated at runtime without controller restart.

## Architectural Analysis

WVA's architecture reflects a sophisticated separation of concerns between *analysis* (determining desired replicas), *actuation* (applying the scaling decision), and *enforcement* (applying safety constraints). The controller does not directly scale workloads in the saturation path -- instead, it emits `wva_desired_replicas` as a Prometheus metric that HPA or KEDA reads as an external metric signal. This indirection provides several benefits: HPA/KEDA handle stabilization windows, rate limiting, and scale target interaction natively; WVA can operate without `update` permission on the scale subresource (except for scale-from-zero); and operators can tune HPA behavior independently of WVA's analysis. The scale-from-zero path is the exception -- it directly updates the scale subresource because a zero-replica deployment has no HPA metric source to trigger scale-up.

The three-analyzer architecture (V1 saturation, V2 token-based saturation, queueing model) represents an evolution of the scaling algorithm. V1 uses percentage-based KV cache utilization with per-replica thresholds, which is simple but doesn't account for heterogeneous replica capacities. V2 operates on absolute token counts (TotalDemand vs TotalCapacity), enabling accurate scaling across mixed GPU types. The queueing model path applies M/M/c queueing theory with TTFT/ITL SLO targets, providing a latency-aware alternative. The analyzer selection is dynamic per optimization cycle, driven by ConfigMap configuration, which allows operators to switch algorithms without redeployment. The V2 path also supports a multi-analyzer pipeline where the saturation analyzer result is combined with supplementary analyzers (e.g., throughput) via weighted scoring.

The namespace-aware configuration system is architecturally notable. WVA uses a three-tier resolution hierarchy (namespace-local ConfigMap > global ConfigMap > system defaults) that enables per-namespace tuning without controller restart. This is implemented through the ConfigMapReconciler which watches for well-known ConfigMap names (`wva-saturation-scaling-config`, `wva-model-scale-to-zero-config`, `wva-qm-analyzer-config`) across all tracked namespaces. Namespace tracking itself is automatic -- when a VA, managed HPA, or managed ScaledObject is created in a namespace, that namespace is registered in the Datastore and becomes eligible for namespace-local ConfigMap watching. This design avoids polling all namespaces while still supporting dynamic namespace discovery.

The VariantAutoscaling CRD is explicitly deprecated (via kubebuilder deprecation warning) in favor of annotation-based discovery. The annotation path (`llm-d.ai/managed=true` on HPA or ScaledObject) is now the primary integration mechanism. WVA synthesizes virtual VariantAutoscaling objects from annotated HPAs/ScaledObjects, using annotations like `llm-d.ai/variant-cost` and `llm-d.ai/model-id` to populate scaling parameters. This transition reduces CRD sprawl and lets users manage scaling through native Kubernetes resources rather than custom CRDs.

The experimental Coordinator subsystem (off by default, gated by `EXPERIMENTAL_COORDINATOR_ENABLED`) adds a cluster-wide post-scaling optimization layer. Its GPU rebalance plugin analyzes GPU allocation efficiency across variants within a model and recommends rebalancing when some variants are saturated while others are idle. The coordinator operates on a separate loop from the main saturation engine, dispatching registered plugins in declaration order. This plugin architecture is designed for extensibility -- new optimization strategies can be added without modifying the core engine.
