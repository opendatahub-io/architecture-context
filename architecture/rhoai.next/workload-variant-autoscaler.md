## Purpose

**Short**: A Kubernetes controller that performs intelligent autoscaling for LLM inference model servers based on saturation metrics, cost optimization, and queueing theory.

**Detailed**: The Workload Variant Autoscaler (WVA) is a global autoscaler for inference model servers serving Large Language Models (LLMs) on Kubernetes. It works alongside standard Kubernetes HPA and external autoscalers like KEDA, emitting Prometheus metrics that drive scaling decisions for Deployments, StatefulSets, and LeaderWorkerSets.

WVA introduces the concept of a "variant" — a specific way of serving a given model with a particular combination of hardware (GPUs), runtimes, and serving configuration. When multiple variants serve the same base model (e.g., on different accelerators like A100, H100, L4), WVA chooses which variant to scale (e.g., adding capacity on the cheapest variant, removing from the most expensive). The controller continuously monitors request rates and server performance via Prometheus metrics from vLLM inference servers, uses saturation analysis (KV cache utilization, queue depth) or queueing theory models (M/M/1) to determine optimal replica counts, and emits `wva_desired_replicas` / `wva_desired_ratio` metrics that HPA or KEDA consume to perform the actual scaling.

The controller manages a single CRD (`VariantAutoscaling`) and integrates with the Gateway API Inference Extension (`InferencePool`) for scale-from-zero detection. It supports scale-to-zero, GPU-aware cost optimization, multi-controller isolation, namespace-scoped configuration overrides, and configurable saturation thresholds via ConfigMaps.

## Architectural Analysis

**Engine-Controller Separation Pattern**: WVA uses an unusual but effective architecture where the optimization engines (saturation, scale-from-zero, queueing model) run as independent polling loops separate from the Kubernetes reconcile loop. The engines write decisions to an in-memory `DecisionCache` and trigger reconciliation via a Go channel (`DecisionTrigger`). This decouples the optimization frequency (30s for saturation, 100ms for scale-from-zero) from the Kubernetes API interaction. The reconciler's sole job is to apply cached decisions to VA status — it never directly queries Prometheus or performs optimization logic. This avoids the common anti-pattern of doing expensive work inside a reconcile loop and allows different engines to run at different frequencies.

**Multi-Analyzer Architecture**: The saturation engine supports three analyzer implementations selectable via ConfigMap: V1 (percentage-based KV cache/queue thresholds — being deprecated), V2 (token-based capacity analysis using cache_config_info for actual KV token counts), and queueing model (M/M/1 queueing theory). The queueing model analyzer takes priority when its ConfigMap exists, otherwise V2 is selected via `analyzerName: "saturation"` in the saturation config. Each analyzer produces different intermediate types but all converge to `VariantDecision` structs that flow through the same enforcer (scale-to-zero) and optimizer (CostAware or GreedyByScore) pipeline. The V2 and queueing model paths use a two-phase architecture: collect `ModelScalingRequest` for all models, then call the optimizer once globally — enabling cross-model GPU-aware decisions.

**Scale-From-Zero via EPP Pod Scraping**: Unlike the saturation engine which reads from Prometheus, the scale-from-zero engine scrapes metrics directly from EPP (Endpoint Picker) pods. It polls at 100ms intervals looking for non-zero `inference_extension_flow_control_queue_size` per model. When pending requests are detected for an idle variant (replicas=0), it directly scales the Deployment/LWS via the scale subresource — the only place WVA directly mutates workload replicas. This bypass of HPA is intentional: HPA cannot act on a zero-replica deployment since there are no pods to emit metrics. The engine discovers EPP endpoints through the InferencePool reconciler which syncs pool metadata into the in-memory datastore.

**Namespace-Scoped Configuration**: WVA supports hierarchical configuration: global ConfigMaps in the controller namespace provide defaults, while namespace-local ConfigMaps with the same well-known names override specific settings per namespace. Namespace tracking is automatic (any namespace with a VariantAutoscaling resource gets tracked) or explicit (via `wva.llmd.ai/config-enabled` label). Namespaces can be excluded entirely via `wva.llmd.ai/exclude` annotation. The ConfigMap reconciler bootstraps all known ConfigMaps at startup before the manager starts, ensuring dynamic configuration is available before the first optimization loop runs.

**Cost-Aware Multi-Variant Optimization**: When multiple VariantAutoscaling resources target the same model ID but different hardware configurations, WVA groups them by model and optimizes globally. The `variantCost` field (e.g., H100=10.0, A100=8.0, L4=5.0) drives the CostAwareOptimizer to prefer scaling cheaper variants first and removing expensive ones first. In limited mode (GPU-constrained clusters), the GreedyByScoreOptimizer uses GPU inventory from node labels to constrain decisions within available hardware. The `GPUsPerReplica` value is extracted from the pod template's resource requests, enabling accurate accounting for multi-GPU serving configurations.
