## Purpose

**Short**: A Kubernetes controller that performs intelligent autoscaling for LLM inference model servers based on saturation, throughput, and queueing model analysis.

**Detailed**: The Workload Variant Autoscaler (WVA) is a GPU-aware global autoscaler for LLM inference workloads running on Kubernetes. It works alongside standard Kubernetes HPA and KEDA ScaledObject resources to drive the scale subresource of inference Deployments and LeaderWorkerSets. WVA continuously monitors vLLM server metrics (KV cache utilization, queue depth, request rates, latency distributions) via Prometheus, runs saturation and throughput analysis engines, and emits `wva_desired_replicas` as an external metric that HPA/KEDA consumes to execute scaling decisions.

WVA introduces the concept of "variants" — multiple model servers in a Gateway API InferencePool that all serve the same base model but differ in hardware configuration (GPU type), serving configuration (tensor parallelism, max batch size, quantization), or role (prefill vs. decode in P/D disaggregation). The controller supports three analysis paths: V1 percentage-based saturation, V2 token-based capacity modeling, and SLO-driven queueing theory. It also provides scale-from-zero capability for idle models and an experimental coordinator with GPU rebalance plugin for cross-model GPU quota allocation. The VariantAutoscaling CRD is deprecated in favor of an annotation-based path (`llm-d.ai/managed=true` on HPA/ScaledObject resources).

## Architectural Analysis

The Workload Variant Autoscaler represents a sophisticated, metrics-driven autoscaling architecture specifically designed for LLM inference workloads. Its most notable design decision is the separation between "deciding" and "actuating": WVA computes optimal replica counts and publishes them as Prometheus metrics (`wva_desired_replicas`), but relies entirely on existing Kubernetes scaling primitives (HPA or KEDA) to actually execute the scaling. This design avoids reinventing the scaling control loop and integrates naturally with cluster admission, quota enforcement, and scaling stabilization windows that HPA/KEDA already provide.

The multi-analyzer pipeline architecture is well-conceived. Three fundamentally different approaches to capacity modeling — percentage-based saturation (V1), token-based capacity (V2), and queueing-theoretic SLO estimation — coexist behind a common interface. Each analyzer produces a `VariantCapacity` (supply/demand pair), and a universal threshold post-step converts capacity into concrete scale-up/down decisions. This decouples the scaling signal generation from the optimization strategy, enabling new analyzers to be added without modifying the decision pipeline. The throughput analyzer's two-tier OLS fitting approach (full OLS when enough observations exist, constrained OLS with pinned baseline otherwise) shows careful attention to cold-start and data-sparse scenarios.

The CRD deprecation in favor of annotation-based discovery (`llm-d.ai/managed=true` on HPAs and ScaledObjects) is an interesting evolution that reduces the API surface. The annotation-based path allows WVA to "shadow" existing HPA/KEDA objects rather than requiring users to create a parallel CRD. The controller synthesizes internal VariantAutoscaling objects from annotated HPAs, preserving compatibility while simplifying the user experience. The dual-mode ScaledObject support (KEDA CRD presence detected at startup) demonstrates good progressive enhancement.

Configuration management is notably sophisticated for a single-controller system. The hierarchical ConfigMap architecture (global defaults → per-model overrides → namespace-local ConfigMaps with include/exclude lists) provides multi-tenant isolation without requiring per-namespace controller instances. The `configMapReconciler` bootstraps configuration eagerly at startup using the API reader (bypassing the informer cache, which isn't started yet) and maintains live synchronization thereafter. The unified Config object with RWMutex protection ensures thread-safe access from the concurrent optimization engine goroutines.

A potential concern is the reliance on Prometheus as the sole metrics transport. The `SourceRegistry` abstraction suggests extensibility (Pod scraping, NoOp sources exist), but in practice all production paths go through Prometheus. If Prometheus is slow or unavailable, the entire scaling pipeline stalls — the controller validates Prometheus connectivity at startup and exits on failure. The metrics cache layer (30s TTL, freshness/staleness/unavailable thresholds) provides resilience against transient scrape delays but cannot compensate for extended Prometheus outages. The controller does emit `wva_errors_total` and `wva_metrics_freshness_status` to surface these conditions, but there is no circuit-breaker or fallback scaling path.
