## Purpose

**Short**: A Kubernetes controller that performs intelligent autoscaling for LLM inference model servers based on saturation metrics and queueing theory.

**Detailed**: The Workload Variant Autoscaler (WVA) is a Kubernetes-based global autoscaler for inference model servers serving Large Language Models (LLMs). It operates as a controller-runtime operator that watches `VariantAutoscaling` custom resources and continuously monitors vLLM inference server metrics via Prometheus to determine optimal replica counts for model serving deployments. WVA considers constraints such as GPU count (cluster resources), energy budgets, and performance budgets (latency/throughput) to make intelligent scaling decisions.

WVA introduces the concept of a "variant" — a specific way of serving a given model with a particular combination of hardware (e.g., A100, H100, L4), runtimes, parallelism, and serving approach. When multiple variants serve the same model, WVA performs cost-aware optimization to choose which variant to scale (e.g., add capacity on the cheapest variant, remove from the most expensive). The autoscaler works alongside standard Kubernetes HPA and external autoscalers like KEDA by emitting custom Prometheus metrics (`wva_desired_replicas`, `wva_desired_ratio`) that HPA/KEDA consumes to perform actual scaling.

The controller supports three scaling analysis engines: V1 (percentage-based saturation analysis using KV cache utilization and queue depth), V2 (token-based saturation analysis with capacity knowledge), and a queueing model analyzer (using queueing theory for latency/throughput prediction). It also supports scale-to-zero and scale-from-zero capabilities by monitoring Gateway API Inference Extension EPP (Endpoint Picker) flow control queue metrics. Configuration is managed through a combination of command-line flags, environment variables, config files, and Kubernetes ConfigMaps with namespace-aware overrides.

## Architectural Analysis

The WVA follows a decoupled engine-controller architecture where scaling decisions are made asynchronously by engine loops and communicated to the reconciler via an in-memory decision cache and Go channel trigger. This design avoids direct API server writes from the engine (which runs every 30 seconds), reducing API server load and preventing rate-limit issues. The reconciler is the sole writer to VA status, applying decisions from the cache and patching status with optimized allocation targets. This pattern is well-suited for high-frequency optimization loops that need to be decoupled from Kubernetes API server latency.

The three-engine architecture (V1 saturation, V2 saturation, queueing model) reflects an evolving scaling strategy. V1 uses percentage-based KV cache saturation thresholds — simple and proven. V2 introduces token-based capacity modeling with a persistent CapacityKnowledgeStore that learns replica capacity over time. The queueing model analyzer applies queueing theory (M/M/1/k, M/G/1 models) for more accurate latency/throughput prediction. Engine selection is dynamic per-cycle based on ConfigMap presence and configuration, allowing operators to switch strategies without redeployment. The V1 path is slated for deprecation once V2 is fully validated.

The scale-from-zero capability is architecturally distinct from the saturation engine. It polls at 100ms intervals (vs. 30s for saturation) because it must react quickly to incoming requests for dormant models. It uses a different metrics source — EPP flow control queue size scraped directly from pods rather than Prometheus — because dormant models have no vLLM metrics to query. The DirectActuator bypasses the normal HPA flow and directly updates the Deployment/LeaderWorkerSet scale subresource, providing immediate scaling response. This design trade-off sacrifices HPA's stabilization window for latency-sensitive cold-start scenarios.

Configuration management uses a layered namespace-aware approach: global ConfigMaps in the controller's system namespace provide defaults, while namespace-local ConfigMaps override specific models or parameters per-namespace. The ConfigMap reconciler bootstraps initial configuration before the manager starts using an uncached client, ensuring scaling parameters are available from the first optimization cycle. Namespace tracking is automatic (driven by VA existence) with opt-in labels and exclusion annotations for fine-grained control. The multi-controller isolation feature (via CONTROLLER_INSTANCE env var and wva.llmd.ai/controller-instance labels) enables running multiple WVA instances in the same cluster, each managing a disjoint set of VariantAutoscaling resources.

The GPU limiter pipeline demonstrates a modular scaling constraint architecture. The DefaultLimiter composes a TypeInventory (which discovers GPU capacity via node labels) with a GreedyBySaturation algorithm. When enabled, the GreedyByScoreOptimizer replaces the CostAwareOptimizer, constraining scaling decisions to available GPU resources. This pluggable optimizer/limiter pattern allows new constraint types (energy budget, cost caps) to be added without modifying the core engine.
