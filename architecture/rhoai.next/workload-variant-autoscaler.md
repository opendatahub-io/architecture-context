## Purpose

**Short**: A Kubernetes-based global autoscaler that determines optimal replica counts for LLM inference model servers by analyzing saturation metrics, KV cache utilization, and queueing theory models.

**Detailed**: The Workload Variant Autoscaler (WVA) is a controller-runtime based Kubernetes operator purpose-built for scaling LLM inference workloads. It monitors inference server saturation metrics (KV cache utilization, request queue depth) from Prometheus and computes optimal replica counts for model serving deployments. WVA introduces the concept of "variants" — different configurations for serving the same model (e.g., different GPU types, parallelism settings) — and performs cost-aware cross-variant optimization.

WVA operates in a two-stage autoscaling architecture: it computes desired replica counts and emits custom metrics, which external autoscalers (HPA or KEDA) consume to perform the actual scaling. This decoupled design allows WVA to coexist with standard Kubernetes autoscaling primitives. For scale-to-zero scenarios, WVA includes a direct actuator that bypasses the HPA path for sub-second cold-start response.

The controller manages `VariantAutoscaling` custom resources (CRD group: `llmd.ai`), watches InferencePool resources from the Gateway API Inference Extension, supports both Deployment and LeaderWorkerSet scale targets, and integrates with Prometheus Operator ServiceMonitors for metrics collection. It features three concurrent analyzer paths: V1 percentage-based saturation (legacy), V2 token-based capacity analysis (primary), and a queueing model analyzer using Kalman filter learning.

## Architectural Analysis

The Workload Variant Autoscaler follows a clean separation between decision-making (engines) and decision-application (controller reconciler). The saturation engine runs as a background goroutine polling every 30 seconds, computing scaling decisions independently from the Kubernetes control loop. Decisions are communicated to the reconciler via an in-memory `DecisionCache` and `DecisionTrigger` channel, avoiding API server interaction in the hot optimization path. This architecture prevents the reconciler from becoming a bottleneck during high-frequency scaling decisions and allows the engine to continue operating even when API server latency is elevated.

The three-analyzer design (V1 percentage-based, V2 token-based, queueing model) reflects an incremental evolution from simple threshold-based scaling toward ML-driven capacity planning. The V2 analyzer introduces the concept of token capacity (k1/k2), where k1 represents maximum KV cache tokens per replica and k2 represents per-inference token demand. This enables workload-aware scaling that accounts for variable input/output lengths and prefix cache hit rates. The queueing model analyzer goes further with a Kalman filter that learns per-replica arrival rates (alpha, beta, gamma parameters) online, computing required replicas from M/M/c queue theory with configurable SLO targets.

The scale-from-zero engine operates on a much tighter 100ms polling loop, reflecting the need for sub-second cold-start response. It uses a separate `DirectActuator` that scales via the Kubernetes `/scale` subresource, bypassing the HPA path entirely. This dual-actuation pattern (metrics emission for steady-state, direct scaling for cold-start) is architecturally interesting — it trades consistency for latency, relying on the saturation engine to take over scaling once the first replica is warm.

Configuration management uses a sophisticated three-tier namespace strategy: global ConfigMaps in the controller namespace provide defaults, namespace-local ConfigMaps override per-namespace, and per-model entries within ConfigMaps override per-variant. The ConfigMapReconciler performs a blocking bootstrap before the engine starts, ensuring all configuration is loaded before any scaling decisions are made. This prevents the cold-start problem where the engine might scale incorrectly with default config. The namespace tracking in the Datastore coordinates which namespaces are actively watched for ConfigMap changes, with automatic opt-in when a VariantAutoscaling resource is created.

The GPU-aware limited mode (enabled via `WVA_LIMITED_MODE`) adds a constraint layer between analyzer output and scaling decisions. The `DefaultLimiter` queries the cluster's GPU inventory (via node labels from the GPU operator), computes per-accelerator-type availability, and uses a `GreedyBySaturation` algorithm to distribute scarce GPU resources across competing variants. Variants are sorted by spare capacity (ascending), ensuring the most saturated variants get resources first. This is particularly relevant for RHOAI deployments where GPU budgets are finite and cost optimization across model variants is critical.
