## Purpose

**Short**: Payload processing service for the AI Gateway that handles request/response mutation, API format translation, credential injection, and dynamic routing for external LLM providers via an Envoy ext_proc plugin architecture.

**Detailed**: The AI Gateway Payload Processing component is an Inference Payload Processor (IPP) that runs as a gRPC external processing filter attached to an Istio/Envoy-based gateway. It processes inference requests flowing through the RHOAI AI Gateway (Model-as-a-Service), enabling users to access external LLM providers (OpenAI, Anthropic, Azure OpenAI, AWS Bedrock, Google Vertex AI) through a unified OpenAI-compatible API. The component registers six custom plugins with the upstream llm-d IPP framework: `maas-headers-guard` (strips internal `x-maas-*` headers), `model-provider-resolver` (resolves model names to provider endpoints via CRD watches), `api-translation` (translates between OpenAI Chat Completions and provider-native formats), `apikey-injection` (injects provider credentials from Kubernetes Secrets), and `nemo-request-guard`/`nemo-response-guard` (optional NeMo Guardrails integration for content safety).

In addition to the ext_proc plugin pipeline, the component runs three Kubernetes controllers that dynamically create networking resources: the ExternalProvider controller creates an ExternalName Service, Istio ServiceEntry, and DestinationRule for each external provider endpoint (establishing mesh-external TLS egress); the ExternalModel controller creates Gateway API HTTPRoutes that route client traffic through the gateway to the correct provider; and the legacy migration controller watches `maas.opendatahub.io/v1alpha1` ExternalModel CRs and creates corresponding `inference.opendatahub.io/v1alpha1` ExternalProvider and ExternalModel CRs to facilitate API migration.

### Plugin Pipeline Configuration

The Helm chart configures the IPP plugin pipeline in `values.yaml`. The request pipeline executes sequentially:

1. `maas-headers-guard` — strip internal x-maas-* headers
2. `body-field-to-header` (model-extractor) — extract model field from body to X-Gateway-Model-Name header (upstream plugin)
3. `model-provider-resolver` — resolve model to provider info
4. `api-translation` — translate request body/headers to provider format
5. `apikey-injection` — inject provider credentials

The response pipeline:
1. `api-translation` — translate response body back to OpenAI format

## Architectural Analysis

The AI Gateway Payload Processing component implements a dual-layer architecture: an Envoy external processing (ext_proc) gRPC service for request/response mutation, and Kubernetes controllers for dynamic networking resource management. This separation cleanly divides data-plane concerns (payload translation, credential injection) from control-plane concerns (routing, egress configuration). The upstream `llm-d-inference-payload-processor` framework provides the ext_proc gRPC server, plugin lifecycle management, and controller-runtime integration. This component registers custom plugins and controllers with that framework rather than implementing its own gRPC server, following a framework-extension pattern that keeps the codebase focused on RHOAI-specific business logic.

The plugin pipeline design is notable for its use of `CycleState` as a shared in-process context bag. Each plugin in the request pipeline writes resolved data (provider name, target model, API format, auth type, credentials reference) that downstream plugins consume. This avoids repeated CRD lookups and keeps the pipeline efficient. The `model-provider-resolver` plugin maintains an in-memory store populated by two reconcilers (one for ExternalProvider, one for ExternalModel) that mirrors CRD state. This store-and-reconcile pattern means the ext_proc hot path never touches the Kubernetes API — all resolution is O(1) map lookups against pre-cached data. The store supports weighted random selection across multiple provider refs per model, enabling simple load distribution or A/B testing across providers.

The API translation layer implements a Translator interface with provider-specific implementations. Most providers (OpenAI, Azure, Bedrock, Vertex OpenAI-compatible) only need path rewriting since they accept OpenAI-format payloads. The Anthropic translator performs full request/response body transformation between OpenAI Chat Completions and the Anthropic Messages API, including system prompt separation, tool call translation, and multimodal image content conversion. A `passthrough` optimization bypasses translation entirely when the client's input format matches the upstream provider's format (except for OpenAI, which always needs path rewriting to strip the model prefix). The `ResponseFieldStripper` utility provides configurable removal of provider-specific response fields (e.g., Azure's `prompt_filter_results`, Vertex's `usage.extra_properties`) to normalize responses to clean OpenAI format.

The controller-managed networking resources form a complete Istio egress path for each external provider. The ExternalProvider controller creates three resources: (1) an ExternalName Service pointing to the provider FQDN, enabling Kubernetes DNS resolution within the mesh; (2) an Istio ServiceEntry declaring the endpoint as MESH_EXTERNAL with DNS resolution and HTTPS protocol; (3) an Istio DestinationRule with `tls.mode: SIMPLE` for TLS origination. The ExternalModel controller creates HTTPRoutes with two match rules per model: a path-prefix match (`/{namespace}/{model}`) and a header match (`X-Gateway-Model-Name: {targetModel}`), both referencing the platform's gateway as parent and the provider's Service as backend. This dual-match design supports both URL-based routing (direct path access) and header-based routing (after the body-field-to-header plugin extracts the model name). The route timeout defaults to 300 seconds, appropriate for long-running LLM inference requests.

The NeMo Guardrails integration follows a fail-closed security model: unknown statuses from NeMo are treated as blocked, and HTTP errors from the NeMo service return 503 (Service Unavailable) rather than silently passing traffic. The guard plugins support both OpenAI chat format and MCP JSON-RPC format, extracting user-supplied text from either `messages` arrays or `params.arguments` objects. This positions the component for future MCP tool-use guardrailing alongside traditional chat inference.
