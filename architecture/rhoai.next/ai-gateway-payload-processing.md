## Purpose

**Short**: A Body-Based Routing (BBR) plugin host that provides payload processing for AI Gateway, enabling model-to-provider resolution, API format translation, credential injection, and guardrail enforcement for external LLM provider traffic flowing through the Kubernetes Gateway API inference extension.

**Detailed**: AI Gateway Payload Processing is a Go service that extends the Kubernetes Gateway API Inference Extension's pluggable BBR (Body-Based Routing) framework with custom plugins for routing inference requests to external LLM providers. It runs as an Envoy external processing (ext_proc) filter attached to an Istio Gateway, intercepting HTTP request/response bodies to perform provider-specific mutations before traffic reaches external endpoints.

The component defines two Custom Resource Definitions — `ExternalModel` and `ExternalProvider` — in the `inference.opendatahub.io` API group. An `ExternalProvider` CR represents a connection to an external LLM service (endpoint FQDN + credential Secret reference), and the ExternalProvider controller creates the Istio networking resources (ExternalName Service, ServiceEntry, DestinationRule with TLS origination) needed to route mesh traffic to that provider. An `ExternalModel` CR maps a client-facing model name to a specific provider and target model, and its controller creates an HTTPRoute (Gateway API) that routes traffic from the platform's ingress Gateway to the provider's Service.

At request time, the BBR plugin pipeline resolves the model name from the request body (model-provider-resolver), translates the request/response format between OpenAI Chat Completions and provider-native APIs (api-translation for Anthropic, Azure OpenAI, Bedrock, Vertex AI), injects provider credentials from Kubernetes Secrets (apikey-injection), and optionally enforces content safety guardrails via NVIDIA NeMo Guardrails (nemo-request-guard, nemo-response-guard). The component is deployed via a Helm chart that wraps the upstream `body-based-routing` chart and must be installed in the same namespace as the Gateway for Istio EnvoyFilter targeting to work.

## Architectural Analysis

The AI Gateway Payload Processing component implements a sophisticated plugin-based architecture built on top of the upstream Kubernetes Gateway API Inference Extension's BBR framework. The key architectural decision is to use Envoy's ext_proc (External Processing) protocol rather than a sidecar proxy or middleware pattern. This allows the component to intercept and mutate both request and response bodies at the Gateway level without modifying the data plane proxy configuration for each backend — a critical design choice for a multi-tenant, multi-provider inference gateway.

The component has a dual-role architecture: it functions both as a runtime request processor (BBR plugins) and as a Kubernetes controller (ExternalModel and ExternalProvider reconcilers). The controllers create the "data plane plumbing" — HTTPRoutes, Services, ServiceEntries, and DestinationRules — while the plugins handle "request plane logic" — model resolution, API translation, credential injection, and guardrails. This separation means the networking topology (which providers are reachable, how traffic is routed) is declaratively managed through CRDs, while the request-level behavior (what happens to each request body) is handled imperatively by the plugin pipeline. The two systems communicate through shared Kubernetes resources: the controller creates an ExternalName Service for a provider, and the HTTPRoute references that Service as a backend.

The model-provider-resolver plugin watches `maas.opendatahub.io` ExternalModel CRDs (from the MaaS controller, a separate component) and maintains a thread-safe in-memory store mapping model names to provider info. This is notably a different CRD group than the `inference.opendatahub.io` ExternalModel CRD that this repo's own controllers manage — the repo contains two parallel model abstractions: one for the MaaS integration path (watched by the plugin) and one for the controller path (managed by the ExternalModel controller). The ExternalModel controller creates HTTPRoutes with two match rules: path-prefix matching (`/{namespace}/{model-name}`) and header matching (`X-Gateway-Model-Name`), providing flexible routing modes. The "body-field-to-header" upstream BBR plugin extracts the model name from the request body and promotes it to the `X-Gateway-Model-Name` header, enabling header-based routing without requiring clients to set headers explicitly.

The API translation layer implements full bidirectional translation for five provider formats. The Anthropic translator is the most complex, handling OpenAI Chat Completions ↔ Anthropic Messages API conversion including system prompt separation, tool calling (function_call ↔ tool_use), multimodal content (image_url → base64 source), and error format mapping. Azure OpenAI and Bedrock OpenAI translators are thin wrappers that primarily rewrite paths since these providers offer OpenAI-compatible endpoints. The Vertex AI translator comes in two variants: a full GenerateContent format translator (currently unused) and an OpenAI-compatible endpoint translator that only rewrites paths. All translators strip the `authorization` header after translation, relying on the downstream apikey-injection plugin to set provider-specific credentials.

The NeMo guardrail integration is notable for its protocol-aware design. The request guard plugin supports both OpenAI chat format (extracting all messages) and MCP JSON-RPC format (extracting string arguments from `params.arguments`). The response guard similarly handles both OpenAI choices and MCP result content. Both plugins are stateless HTTP clients with configurable timeouts and fail-open semantics (returning errors to the caller rather than silently passing blocked content). The NeMo endpoint is expected to return HTTP 200 for both allow and block decisions, with the actual verdict conveyed in the response body's `status` field.

Security-wise, the component follows the RHOAI credential separation pattern: provider API keys are stored in Kubernetes Secrets labeled with `inference.networking.k8s.io/bbr-managed=true`, and the apikey-injection plugin's reconciler only watches Secrets with this label. This prevents the component from accidentally caching unrelated Secrets. The ExternalProvider controller validates that the referenced Secret exists during reconciliation but does not read the Secret data — only the apikey-injection plugin does, and only at request time from its in-memory cache. TLS to external providers is handled entirely by Istio's DestinationRule with SIMPLE TLS origination mode, meaning the BBR process itself never terminates or initiates TLS connections to providers.
