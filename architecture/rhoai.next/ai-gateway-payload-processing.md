## Purpose

**Short**: Payload processing service that provides pluggable Body Based Routing (BBR) plugins for the AI Gateway, enabling model resolution, multi-provider API translation, API key injection, and NeMo guardrails enforcement on inference requests flowing through the Kubernetes Gateway API.

**Detailed**: The AI Gateway Payload Processing service is a Go binary that hosts a set of BBR (Body Based Routing) plugins built on the upstream Kubernetes Inference Gateway (gateway-api-inference-extension) framework. It runs as a gRPC External Processing (ExtProc) server attached to an Envoy proxy via an Istio EnvoyFilter, intercepting and mutating inference request and response payloads in the Gateway API data plane.

The service implements a plugin pipeline that processes each inference request through four ordered stages: (1) model-provider-resolver watches ExternalModel CRDs and resolves the client-facing model name to a specific external LLM provider (OpenAI, Anthropic, Azure OpenAI, Bedrock, Vertex AI); (2) api-translation converts the request body from the unified OpenAI Chat Completions format to the provider's native API format, and translates responses back; (3) apikey-injection watches labeled Kubernetes Secrets and injects provider-specific authentication headers; (4) optionally, NeMo guardrail plugins call an external NeMo Guardrails service to enforce input/output content policies. The service also supports MCP JSON-RPC payload inspection through the NeMo guard plugins.

The component defines two CRDs in the `inference.opendatahub.io` API group — `ExternalModel` and `ExternalProvider` — which represent the user-facing model abstraction and the backend provider connection respectively. These CRDs are watched by the model-provider-resolver plugin's controller-runtime reconciler. The service is deployed via a Helm chart that wraps the upstream `body-based-routing` chart from the Kubernetes Inference Gateway project, adding RBAC for ExternalModel/Secret access and configuring the plugin pipeline.

## Architectural Analysis

The AI Gateway Payload Processing service implements an extensible plugin pipeline pattern built on the upstream Kubernetes Inference Gateway BBR (Body Based Routing) framework. Rather than implementing a standalone HTTP proxy, the service leverages Envoy's External Processing (ExtProc) gRPC protocol to intercept and mutate request/response bodies inline within the Gateway API data plane. This design has two key advantages: it avoids an additional network hop (ExtProc runs as a sidecar-adjacent service in the same namespace as the Gateway), and it inherits Envoy's connection management, TLS termination, and load balancing without reimplementing them.

The plugin pipeline uses CycleState — a per-request shared memory map provided by the BBR framework — to pass data between plugins without modifying the request itself. The model-provider-resolver writes provider identity and credential references into CycleState, which downstream plugins (api-translation, apikey-injection) read. This decoupled design means plugins can be independently added, removed, or reordered in the Helm values without code changes. The NeMo guardrail plugins are notably independent of the provider pipeline — they operate on the raw message content and can be inserted at any point in the chain.

The controller-runtime reconcilers in model-provider-resolver and apikey-injection are notable for being embedded within the BBR plugin framework rather than running as separate controllers. Each plugin registers its reconciler via `handle.ReconcilerBuilder()` at factory time, sharing the BBR process's controller-runtime manager. This means the ExternalModel reconciler and Secret reconciler run in the same process as the ExtProc server, using in-memory stores (modelInfoStore, secretStore) with RWMutex synchronization. This avoids external cache dependencies at the cost of non-shared state — each replica maintains its own view. The stores are eventually consistent with the Kubernetes API via standard watch/list semantics.

The API translation layer supports five provider formats (OpenAI, Anthropic, Azure OpenAI, Bedrock OpenAI-compatible, Vertex AI OpenAI-compatible), plus a dormant native Vertex AI GenerateContent translator. The Anthropic and native Vertex translators perform full body transformation (OpenAI ↔ provider-native format), while Azure, Bedrock, and Vertex OpenAI-compatible translators only rewrite the `:path` pseudo-header and strip provider-specific response fields, since these providers accept OpenAI-format requests natively. The translation also strips and re-injects the `authorization` header to prevent client credentials from leaking to providers — the apikey-injection plugin provides the correct provider credential afterward.

A notable security consideration is that NeMo guardrail calls use plaintext HTTP (`http.Client` with no TLS config), relying on network-level isolation rather than transport encryption. This is appropriate for intra-cluster communication but should be documented for environments requiring end-to-end encryption. The NeMo response size is capped at 1 MiB (`maxNemoResponseBytes`) to prevent denial-of-service via large responses.

The CRD design reveals a phased roadmap: `ExternalModel.spec.externalProviderRefs` is currently limited to `MaxItems=1`, with a comment indicating Phase 2 will increase this for traffic splitting across multiple providers. The `ExternalProvider` CRD captures provider-agnostic connection details (endpoint FQDN, auth secret reference, provider-specific config map), enabling multi-tenant credential management via standard Kubernetes RBAC on Secret resources.
