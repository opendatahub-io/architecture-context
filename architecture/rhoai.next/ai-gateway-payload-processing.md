## Purpose

**Short**: A pluggable Body Based Routing (BBR) ext_proc service that performs payload-level mutations on AI inference requests and responses flowing through the AI Gateway, enabling multi-provider model routing, API translation, credential injection, and content guardrails.

**Detailed**: AI Gateway Payload Processing is a Go service that runs as a set of plugins within the Kubernetes Gateway API Inference Extension's BBR (Body Based Routing) framework. It is deployed as an Envoy external processing (ext_proc) filter attached to an Istio Gateway via EnvoyFilter, intercepting inference traffic at the HTTP body level. The service provides five production plugins: model-provider-resolver (which watches ExternalModel CRDs to map model names to cloud providers), api-translation (which translates between OpenAI Chat Completions format and provider-native APIs like Anthropic Messages, Azure OpenAI, Bedrock, and Vertex AI), apikey-injection (which reads Kubernetes Secrets and injects provider-specific authentication headers), and two NeMo guardrail plugins for input and output content filtering.

The component is part of the RHOAI Models-as-a-Service (MaaS) stack. It sits on the data path between the client and backend model endpoints, enabling a unified OpenAI-compatible API surface over heterogeneous providers. The service uses controller-runtime reconcilers (not a full operator) to watch ExternalModel CRDs and labeled Secrets, maintaining in-memory caches that the request/response processing plugins read from at inference time. Deployment is via a Helm chart that wraps the upstream `body-based-routing` chart from the Gateway API Inference Extension project, adding RBAC for ExternalModel and Secret access.

## Architectural Analysis

The AI Gateway Payload Processing service implements a plugin pipeline architecture built on the upstream Kubernetes Gateway API Inference Extension's BBR framework. Rather than being a standalone operator or service, it acts as a plugin host — registering five custom plugins with the framework's plugin registry at startup (`cmd/main.go` lines 50-55) and delegating all gRPC ext_proc server lifecycle, health checking, and Envoy integration to the upstream `runner.NewRunner()`. This design minimizes boilerplate and ensures compatibility with the evolving Gateway API Inference Extension ecosystem.

The plugin chain follows a deliberate ordering: `body-field-to-header` (upstream, extracts model name for routing) → `model-provider-resolver` (resolves model to provider via ExternalModel CRD) → `api-translation` (transforms request body to provider-native format) → `apikey-injection` (injects credentials). Inter-plugin communication uses the framework's `CycleState` — a per-request key-value store that carries provider name, model name, and credential reference between plugins without modifying request headers. This is a clean separation of concerns: each plugin reads from and writes to CycleState rather than coupling to other plugins' internal state. The state keys are defined in `pkg/plugins/common/state/state-keys.go`.

The reconciler pattern used by `model-provider-resolver` and `apikey-injection` is notable: both use controller-runtime's `builder.Builder` to watch Kubernetes resources (ExternalModel CRDs and labeled Secrets respectively) but do not run as a traditional operator. Instead, the BBR framework's `Handle` provides a reconciler builder that the plugins use to register their own watchers. The reconcilers populate thread-safe in-memory stores (`modelInfoStore` and `secretStore`) that the request processing hot path reads from with RWMutex locking. This avoids per-request API server calls and keeps inference latency low. The `apikey-injection` reconciler uses a label predicate (`inference.networking.k8s.io/bbr-managed=true`) to limit the watch scope to relevant Secrets only.

The API translation layer supports five providers with varying translation complexity. OpenAI and Bedrock OpenAI are essentially pass-through (path rewrite only), Azure OpenAI adds response field stripping for provider-specific fields (`prompt_filter_results`, `choices[].content_filter_results`), Vertex OpenAI rewrites the path to include GCP project/location/endpoint, and the Anthropic translator performs full body transformation between OpenAI Chat Completions and Anthropic Messages API format — including system message separation, tool call translation, image content block conversion, and usage field mapping. The Vertex native GenerateContent translator exists in code but is commented out in the plugin registration, indicating it is planned but not yet shipped in the 3.4 release.

The NeMo guardrail plugins are architecturally distinct from the other plugins: they make synchronous outbound HTTP calls during request/response processing rather than reading from in-memory state. This introduces a latency dependency on the NeMo service availability and response time (configurable timeout, default 360 seconds). The plugins support both OpenAI chat format and MCP JSON-RPC format for input extraction, suggesting forward-looking MCP tool integration. The NeMo service always returns HTTP 200 regardless of block/allow decisions — the actual decision is conveyed via the response body's `status` field, which the plugin interprets.

A security consideration worth noting: the ext_proc gRPC service on port 9004/TCP runs without TLS or authentication. Security relies entirely on network isolation — the service must be deployed in the same namespace as the Gateway so the EnvoyFilter `targetRefs` work, and only the Envoy proxy should connect. The API keys held in the in-memory `secretStore` are sensitive credentials for cloud providers; they persist only in process memory and are refreshed via the Secret reconciler. The FIPS compliance is enforced at build time via `GOEXPERIMENT=strictfipsruntime` in the Konflux Dockerfile.
