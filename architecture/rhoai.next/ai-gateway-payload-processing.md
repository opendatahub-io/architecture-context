## Purpose

**Short**: An Envoy External Processing (ext_proc) filter that intercepts inference requests at the AI Gateway, resolves external model/provider mappings, translates API formats across cloud LLM providers, injects authentication credentials, and enforces NeMo guardrails.

**Detailed**: AI Gateway Payload Processing is a pluggable Inference Payload Processor (IPP) built on the [llm-d](https://github.com/llm-d/llm-d-inference-payload-processor) framework. It runs as an Envoy ext_proc sidecar attached to the platform's AI Gateway (Istio EnvoyFilter) and intercepts every inference request flowing through the gateway. The processor resolves which external LLM provider should handle a request by watching `ExternalModel` and `ExternalProvider` CRDs, then translates the request from OpenAI Chat Completions format into the provider's native API format (Anthropic Messages, Vertex AI, Azure OpenAI, AWS Bedrock), injects the correct API key or SigV4 authentication headers from Kubernetes Secrets, and optionally enforces content safety via NeMo Guardrails.

The component defines two custom CRDs (`ExternalModel` and `ExternalProvider`) in the `inference.opendatahub.io` API group. Three embedded controllers reconcile these CRDs: the ExternalProvider controller creates the Istio networking stack (ExternalName Service, ServiceEntry, DestinationRule) for each provider endpoint; the ExternalModel controller creates Gateway API HTTPRoutes that route client traffic to the correct provider service; and a legacy migration controller watches `maas.opendatahub.io` ExternalModel CRs from the previous API version and creates corresponding `inference.opendatahub.io` resources. The plugin pipeline processes requests in order: strip internal `x-maas-*` headers, extract the model name to a routing header, resolve the model to a provider via the in-memory store, translate the API format, and inject authentication credentials.

## Architectural Analysis

The AI Gateway Payload Processing component follows a distinctive hybrid architecture: it is simultaneously an Envoy External Processing (ext_proc) gRPC service and a Kubernetes controller-runtime application. The llm-d framework provides the ext_proc gRPC server and plugin lifecycle, while the component registers its own controllers and plugin implementations. This means a single binary watches CRDs (ExternalModel, ExternalProvider), maintains in-memory stores, and processes live traffic -- a design that avoids the complexity of a separate operator deployment but couples the data plane (request processing) with the control plane (CRD reconciliation).

The plugin pipeline architecture is noteworthy for its use of `CycleState` as a shared request-scoped key-value store. The `model-provider-resolver` plugin writes provider metadata (endpoint, auth type, credentials reference, API format, model config) into CycleState, and downstream plugins (`api-translation`, `apikey-injection`) read from it. This decoupled design means plugins can be reordered or removed without code changes -- the Helm values define the pipeline order. The `maas-headers-guard` plugin runs first and strips any `x-maas-*` headers from incoming requests, preventing clients from injecting internal routing headers.

The API translation layer supports five cloud LLM providers with two translation approaches: full body translation (Anthropic -- converting between OpenAI Chat Completions and Anthropic Messages API including tool calls, vision, and system prompts) and path-only rewriting (OpenAI, Azure, Bedrock, Vertex OpenAI -- where the upstream endpoint accepts OpenAI format natively but needs different URL paths and headers). A passthrough optimization skips translation entirely when the input and output API formats match (e.g., Anthropic-to-Anthropic), except for OpenAI-to-OpenAI where path rewriting is always required to strip the model prefix path. The `ResponseFieldStripper` utility enables provider-specific response cleanup (removing Azure's `prompt_filter_results` or Vertex's `usage.extra_properties`) without per-provider response translation logic.

The networking architecture relies heavily on Istio service mesh. Each ExternalProvider gets three resources: an ExternalName Service (DNS mapping), a ServiceEntry (mesh-external registration), and a DestinationRule (TLS origination with mode SIMPLE). This ensures all egress to external LLM providers flows through the mesh with proper TLS. The ExternalModel controller creates HTTPRoutes with two match rules per model: a path-prefix match (`/{namespace}/{model}`) and a header match (`X-Gateway-Model-Name: {targetModel}`), with the path-prefix being a temporary pattern until a unified entrypoint is wired (per RHAISTRAT-1540). The weighted traffic splitting feature uses simple weighted random selection across multiple provider refs, enabling A/B testing or cost optimization across providers.

A security concern worth noting is that the NeMo Guardrails integration uses a plain `http.Client` with no TLS configuration, relying on network-level trust within the cluster. The `nemoURL` is fully configurable but there is no mTLS or certificate validation -- this is acceptable for in-cluster communication but would be a risk if NeMo were deployed externally. Additionally, the Secret reconciler for API key injection only watches Secrets with the `inference.networking.k8s.io/bbr-managed=true` label, which limits the blast radius but means users must explicitly label their credential Secrets.
