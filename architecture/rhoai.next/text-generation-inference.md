## Purpose

**Short**: TGIS is a high-performance text generation inference server that serves large language models (LLMs) via gRPC and HTTP APIs with batching, quantization, and multi-GPU tensor parallelism.

**Detailed**: Text Generation Inference Server (TGIS) is a fork of Hugging Face's text-generation-inference, customized and hardened for Red Hat OpenShift AI. It provides a production-grade serving runtime for decoder-only (causal LM) and encoder-decoder (seq2seq) transformer models. The server exposes two gRPC APIs — an internal `TextGenerationService` protocol for shard-level batch management, and a public `GenerationService` (FMaaS) API for client-facing text generation, streaming, tokenization, and model info queries.

The system follows a three-tier process architecture: a Rust **launcher** process that orchestrates the lifecycle of one or more Python **shard** processes (one per GPU) and a single Rust **router** process. The router handles client-facing gRPC/HTTP traffic, implements request batching, queuing, validation, and metrics, and communicates with Python shards via Unix domain sockets using the internal gRPC protocol. The Python shards load the model, manage GPU memory, run inference forward passes, and support advanced features including flash attention, paged attention, PyTorch 2 graph compilation, GPTQ/bitsandbytes quantization, prompt prefix caching, and memory scaling estimation. TGIS is deployed as a single-container image on OpenShift with NVIDIA GPU resources, typically managed by KServe's ServingRuntime CR for model serving workloads.

## Architectural Analysis

TGIS implements a three-tier process architecture within a single container, separating concerns across a Rust launcher, Rust router, and Python inference shards. The launcher (`text-generation-launcher`) acts as a process supervisor: it converts the tokenizer to fast format, resolves model configuration, spawns one Python shard process per GPU, waits for all shards to signal readiness via Unix domain socket creation, then starts the router. This design allows the GPU-bound Python inference code to remain isolated per-GPU while the Rust router handles all client-facing I/O with high concurrency.

The router implements a sophisticated batching engine with configurable weight limits, padding thresholds, and request reordering. Requests are validated (temperature, top_k, top_p, sequence lengths, stop sequences) before being enqueued. The batcher continuously drains the queue, forming optimal batches by analyzing both prefill and next-token weight costs. It supports request cancellation (checked every 16 tokens for unary, every token for streaming), time-limited generation, and graceful batch pruning. The router exposes the public `fmaas.GenerationService` gRPC API while internally communicating with shards via the `generate.v1.TextGenerationService` protocol — a clean separation between client-facing and internal APIs.

The Python server supports multiple inference engines (`tgis_native`, `hf_transformers`, `ibm_fms`, `onnx`) and model architectures (Llama, Falcon/RefinedWeb, BLOOM, T5, GPT BigCode, BART, IBM Megatron). Advanced GPU memory management is a central feature: the `Estimator` class in `memory_characterizer.py` runs empirical GPU memory profiling at startup, fitting linear and quadratic curves to predict memory usage as a function of batch size and sequence length. This `MemoryScalingModel` is communicated to the router via the `ModelInfo` RPC, enabling the router's batcher to make weight-aware batching decisions that avoid GPU OOM errors.

Security is minimal at the TGIS level — optional TLS for gRPC with client CA certificate validation is the only built-in mechanism. The container runs as non-root user `tgis` (UID 2000) with `allowPrivilegeEscalation: false`, `readOnlyRootFilesystem` support via kustomize patches, and `seccompProfile: RuntimeDefault`. In production RHOAI deployments, authentication and authorization are enforced at the KServe/platform ingress layer (kube-rbac-proxy sidecar, Gateway API HTTPRoute) rather than within TGIS itself. The prompt prefix cache supports PEFT adapter loading from a shared filesystem path, enabling efficient prompt tuning without full model reloading.

The deployment manifests use a kustomize base/overlay pattern where the base defines a generic inference server Deployment and headless Service, and per-model overlays specify the model name, GPU count, memory limits, and optional features (flash attention, PT2 compile, shared memory for NCCL). This pattern is well-suited for KServe's ServingRuntime model, where each model deployment gets its own kustomize overlay. The headless Service (clusterIP: None) is designed for direct pod-to-pod communication rather than load-balanced access, consistent with KServe's internal routing model.
