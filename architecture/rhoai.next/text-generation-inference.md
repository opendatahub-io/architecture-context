## Purpose

**Short**: A high-performance text generation inference server that serves large language models via gRPC and HTTP APIs with dynamic batching, tensor parallelism, and multiple inference engine backends.

**Detailed**: Text Generation Inference Server (TGIS) is an early fork of HuggingFace's text-generation-inference, developed by IBM and adapted for Red Hat OpenShift AI (RHOAI). It provides a production-grade serving runtime for causal language models (GPT-family, LLaMA, Falcon, etc.) and sequence-to-sequence models (T5, FLAN-UL2, etc.), optimized for GPU inference with CUDA 12.1.

The system has a three-tier architecture: a Rust **launcher** that orchestrates process lifecycle, one or more Python **server shards** that load model weights and execute inference on GPUs, and a Rust **router** that accepts client requests via gRPC and HTTP, validates inputs, manages batching/queuing, and distributes work to shards via Unix domain sockets. The router implements a sophisticated continuous batching algorithm with dynamic batch formation, prefill weight limits, and granular request scheduling to maximize GPU throughput while maintaining latency guarantees.

TGIS supports multiple inference backends (HuggingFace Transformers, TGIS Native with Flash Attention V2, IBM Foundation Model Stack, ONNX Runtime, DeepSpeed), GPTQ/bitsandbytes quantization, tensor parallelism across multiple GPUs via NCCL, paged KV cache for memory efficiency, PyTorch 2 compilation, TLS/mTLS for secure communication, OpenTelemetry distributed tracing, and comprehensive Prometheus metrics. It is deployed as a single-container inference server managed by KServe in RHOAI.

## Architectural Analysis

TGIS implements a three-tier process architecture optimized for GPU inference throughput. The Rust launcher (`text-generation-launcher`) serves as an init-like process supervisor, spawning Python shards and the router, then monitoring health via process polling. This design allows the heavy Python ML runtime to crash and be detected without affecting the lightweight Rust router. The launcher writes panic messages to `/dev/termination-log` for Kubernetes pod termination message integration, and handles SIGBUS errors specially for multi-GPU deployments where `/dev/shm` exhaustion is a common failure mode.

The continuous batching algorithm in the router is particularly sophisticated. Rather than simple first-come-first-served batching, it implements a weight-aware batch formation strategy that considers: (1) the total "weight" of a batch (input tokens × batch size including padding), (2) a configurable maximum padding ratio to prevent waste from mismatched input lengths, (3) a "queue jumping" mechanism where shorter requests can be inserted into a batch ahead of longer queued requests if they fit within weight constraints, and (4) mid-inference batch concatenation where new requests can be added to an already-running decode loop during the waiting period between prefill operations. This is tracked via BTreeSet-based granular analysis and reported through dedicated metrics (`tgi_queue_jump`, `tgi_granular_batch_addition`, `tgi_prefill_weight_limit_exceeded`).

The inference engine abstraction in the Python server is a key extensibility point. The `DEPLOYMENT_FRAMEWORK` environment variable selects from seven backends: `hf_transformers` (default, general-purpose), `tgis_native` (required for Flash Attention, paged attention, and tensor parallelism), `ibm_fms` (IBM Foundation Model Stack for megatron-style models), `hf_accelerate` (device mapping), `hf_optimum_ort` (ONNX Runtime), `hf_optimum_bt` (BetterTransformer), and `ds_inference` (DeepSpeed). The `tgis_native` engine is the most feature-rich, supporting Flash Attention V2 for memory-efficient attention, paged KV cache management for dynamic memory allocation, ExLlama v1/v2 kernels for GPTQ-quantized models, and custom CUDA kernels built at image compile time.

The dual-TLS stack in the router (both `rustls` and `openssl`/`native-tls` dependencies) warrants attention from a FIPS perspective. While the UBI 9.5 base image provides FIPS-validated OpenSSL, the Rust router uses `rustls` (which relies on the `ring` crate) for gRPC TLS via Tonic, and also depends on `native-tls` and `openssl` crates. This means TLS connections may not use the FIPS-validated OpenSSL path depending on which TLS backend Tonic selects at runtime. No explicit FIPS mode configuration or cipher suite restrictions were found in the codebase.

The gRPC API design exposes two separate protobuf services: an external `fmaas.GenerationService` (in `generation.proto`) used by clients with a clean request/response API supporting batch generation, streaming, tokenization, and model info; and an internal `generate.v1.TextGenerationService` (in `generate.proto`) used for router-to-shard communication with lower-level batch management primitives (Prefill, NextToken, PruneBatch, ClearCache, PrefixLookup). This separation allows the router to manage batching and scheduling without exposing those internals to clients.
