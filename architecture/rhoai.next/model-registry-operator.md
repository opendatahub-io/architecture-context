## Purpose

**Short**: Kubernetes operator that manages the lifecycle of Model Registry and Model Catalog instances on Red Hat OpenShift AI.

**Detailed**: The Model Registry Operator is a controller-runtime based Kubernetes operator that reconciles `ModelRegistry` custom resources (API group `modelregistry.opendatahub.io`) to deploy and manage Model Registry REST API services with backing database storage (PostgreSQL or MySQL). It handles the complete lifecycle including deployment creation, service exposure, database provisioning, authentication proxy injection (kube-rbac-proxy sidecar), RBAC configuration, and OpenShift Route creation.

The operator also manages a Model Catalog component, deployed alongside model registries, which serves as a centralized catalog of available ML models with metadata sourced from ConfigMaps and init containers. The catalog shares the same authentication pattern (kube-rbac-proxy sidecar) and database infrastructure (PostgreSQL) as the model registry instances.

The operator supports API version migration from v1alpha1 (deprecated, with Istio/OAuth proxy patterns) to v1beta1 (current, with kube-rbac-proxy patterns), including automatic conversion webhooks and OAuth-to-kube-rbac-proxy migration during reconciliation. It integrates with the broader RHOAI platform through cross-component CRD watches (`components.platform.opendatahub.io/ModelRegistry`, `services.platform.opendatahub.io/Auth`) and is deployed by the platform operator via kustomize overlays.

## Architectural Analysis

The Model Registry Operator follows a standard controller-runtime pattern with two primary controllers: `ModelRegistryReconciler` for managing individual registry instances, and `ModelCatalogReconciler` for the centralized catalog service. Both controllers use Go template-based resource generation from `.yaml.tmpl` files in `internal/controller/config/templates/`, which provides a clean separation between the reconciliation logic and the Kubernetes resource definitions. The `banzaicloud/k8s-objectmatcher` library is used for comparing desired vs. actual state, avoiding unnecessary updates.

The authentication architecture has undergone a significant migration from OAuth proxy (v1alpha1 pattern) to kube-rbac-proxy (v1beta1 pattern). The operator includes automatic migration logic: the mutating webhook calls `MigrateOAuthProxyToKubeRBACProxy()` to convert existing OAuth proxy configurations to kube-rbac-proxy on create/update, and the controller reconcile loop at line 181 handles runtime migration for existing deployments. The kube-rbac-proxy sidecar uses Kubernetes SubjectAccessReview to authorize requests — users must have `get` permission on the Service resource to access the registry API. This is a simpler and more Kubernetes-native approach than OAuth proxy, which required session management and cookie handling.

The catalog controller is notably more complex than it might appear from a quick glance. It dynamically discovers model source ConfigMaps via label selectors (`opendatahub.io/catalog-source`), integrates with the platform Auth CR for admin group resolution (lines 1244-1272 of `modelcatalog_controller.go`), and deploys init containers to populate catalog data from container images. The catalog also manages its own PostgreSQL instance independently of any registry's database, with generated credentials stored in a separate Secret.

Network security is implemented through a layered approach: NetworkPolicies restrict database access to only the associated registry/catalog pods (port 5432 ingress only from pods with matching labels), while kube-rbac-proxy NetworkPolicies allow ingress only from the OpenShift ingress controller namespace (`network.openshift.io/policy-group: ingress`). This means direct pod-to-pod access bypassing the Route/kube-rbac-proxy is blocked by default. The PostgreSQL network policies are particularly well-designed — they use precise label selectors to ensure only the owning registry pod can reach its database, preventing cross-registry database access within the same namespace.

The CRD API version migration deserves attention. The operator supports StorageVersionMigration (via `migration.k8s.io` API) to automatically migrate stored resources from v1alpha1 to v1beta1. This is driven by environment variables `STORAGE_MIGRATION_SOURCE_VERSION` and `STORAGE_MIGRATION_TARGET_VERSION` in the `internal/migration/` package, with both manual (direct update) and SVM (StorageVersionMigration CR) strategies available. The conversion webhook in `api/v1beta1/modelregistry_conversion.go` handles runtime conversion between versions, but storage migration ensures all persisted resources use the hub version.
