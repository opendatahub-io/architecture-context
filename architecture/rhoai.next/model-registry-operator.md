## Purpose

**Short**: Kubernetes operator that manages the lifecycle of Model Registry and Model Catalog instances on OpenShift/Kubernetes clusters.

**Detailed**: The Model Registry Operator is a controller-runtime based Kubernetes operator that reconciles `ModelRegistry` custom resources to deploy and manage Model Registry REST API services, PostgreSQL database backends, authentication proxies (kube-rbac-proxy), and associated networking/RBAC resources. It operates in the `modelregistry.opendatahub.io` API group with two API versions: v1alpha1 (deprecated) and v1beta1 (storage version).

The operator implements two reconciliation controllers: (1) the **ModelRegistryReconciler**, which manages individual Model Registry instances including their REST API deployments, database provisioning, kube-rbac-proxy authentication sidecars, OpenShift Routes, NetworkPolicies, and RBAC resources; and (2) the **ModelCatalogReconciler**, which manages a singleton Model Catalog service with its own database, catalog source discovery via labeled ConfigMaps, and admin group authorization from the platform Auth CR.

The operator detects cluster capabilities at startup (OpenShift vs vanilla Kubernetes, BYOIDC mode, Auth API availability) and conditionally creates platform-specific resources. It includes a storage version migration system that handles CRD version transitions from v1alpha1 to v1beta1 using either the Kubernetes StorageVersionMigration API or a manual re-read/re-write fallback strategy. The operator also handles API deprecation migrations: Istio config is automatically migrated to OAuthProxy, and OAuthProxy is migrated to KubeRBACProxy via defaulting webhooks.

## Architectural Analysis

The Model Registry Operator exhibits a clean separation between its two controllers. The `ModelRegistryReconciler` manages individual ModelRegistry CRs following a standard owner-reference pattern where each CR owns its Deployment, Service, NetworkPolicy, and RBAC resources. The `ModelCatalogReconciler`, in contrast, operates as a singleton manager that reconciles a fixed set of resources in a target namespace, using the platform's `ModelRegistry` CR as an owner reference to ensure cascading deletion when the platform component is removed. This dual-controller architecture allows the operator to manage both user-created registry instances and the platform's catalog service through a single binary.

The operator demonstrates sophisticated cluster capability detection at startup via the Kubernetes discovery API. It detects four independent capabilities: OpenShift (`route.openshift.io`), User API (`user.openshift.io`), Config API (`config.openshift.io`), and Auth API (`services.platform.opendatahub.io`). The absence of the User API on an OpenShift cluster is specifically detected as "BYOIDC mode" (Bring Your Own Identity Provider, available in OpenShift 4.20+), which causes the operator to skip `user.openshift.io/Group` creation and instead reference external IdP groups. This capability-based approach allows a single operator binary to work across vanilla Kubernetes, traditional OpenShift, and BYOIDC OpenShift clusters without configuration changes.

The API evolution path is well-documented in the codebase: Istio config (v1alpha1) is automatically migrated to OAuthProxy at the defaulting webhook stage, and OAuthProxy is further migrated to KubeRBACProxy in v1beta1. The conversion webhook handles v1alpha1-to-v1beta1 transitions including field mapping (Istio -> OAuthProxy -> KubeRBACProxy) and field dropping (v1beta1's `GenerateDeployment` and `KubeRBACProxy` fields are discarded when converting back to v1alpha1). The storage version migration system adds a background goroutine that monitors the CRD's `Status.StoredVersions` and triggers migration using either the Kubernetes-native `StorageVersionMigration` API or a manual re-read/re-write fallback.

Resource management uses Go template rendering for all Kubernetes objects, with templates embedded in the binary and rendered with dynamic parameters (image references, ports, resource requirements). The operator uses the banzaicloud `k8s-objectmatcher` library for intelligent diff detection, only issuing updates when resource specs actually change. Immutable field conflicts (e.g., changing a Deployment's selector) are detected and handled by deleting and re-creating the resource rather than failing. The catalog controller implements a particularly interesting pattern for ConfigMap discovery: it watches for ConfigMaps labeled `opendatahub.io/catalog-source=true` in the target namespace, validates their names as DNS-compliant labels, and mounts up to 9 discovered sources as volumes in the catalog deployment. The catalog also generates PostgreSQL credentials idempotently, only creating passwords on initial Secret creation and preserving existing passwords during subsequent reconciliations.

The security model centers on kube-rbac-proxy as the authentication enforcement point. When enabled, the operator creates a ConfigMap containing kube-rbac-proxy authorization rules, a ClusterRoleBinding granting `system:auth-delegator` to the registry's ServiceAccount, and (on OpenShift) a Route and NetworkPolicy for external HTTPS access. The proxy validates tokens via Kubernetes TokenReview and authorization via SubjectAccessReview before proxying to the REST API. NetworkPolicies are created for both the REST API (allowing only kube-rbac-proxy traffic) and PostgreSQL (allowing only traffic from registry pods), providing defense-in-depth.
