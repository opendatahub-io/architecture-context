## Purpose

**Short**: Manages the lifecycle of Model Registry and Model Catalog instances on OpenShift/Kubernetes, including their backing databases, authentication proxies, network policies, and ingress routing.

**Detailed**: The model-registry-operator is a Kubernetes operator built with controller-runtime (kubebuilder v4 scaffolding) that reconciles `ModelRegistry` custom resources. For each ModelRegistry CR, the operator dynamically creates and manages a complete application stack: a model-registry REST service deployment, PostgreSQL or MySQL database connectivity (with optional auto-provisioned PostgreSQL), kube-rbac-proxy sidecar for authentication, OpenShift Routes for HTTPS access, Gateway API HTTPRoutes for Data Science Gateway integration, RBAC resources (Roles, RoleBindings, ClusterRoleBindings), NetworkPolicies, and OpenShift user Groups.

The operator also includes a dedicated `ModelCatalogReconciler` that manages a singleton model catalog instance. The catalog provides a curated collection of model metadata with its own PostgreSQL database, kube-rbac-proxy authentication, configurable catalog sources via ConfigMaps, and admin group management via the platform Auth CR. The operator supports multiple OpenShift authentication modes including traditional OpenShift (user.openshift.io), BYOIDC (Bring Your Own Identity Provider) mode on OpenShift 4.20+, and plain Kubernetes. It includes a CRD storage version migration framework that supports both the Kubernetes StorageVersionMigration API and a manual re-read/re-write fallback strategy.

## Architectural Analysis

The model-registry-operator follows a template-driven reconciliation pattern where Go templates (embedded via `//go:embed`) define every Kubernetes resource the operator creates. This approach cleanly separates resource definitions from controller logic — the `TemplateApplier` renders YAML from templates with `ModelRegistryParams` or `ModelCatalogParams`, then `ResourceManager` calculates diffs using k8s-objectmatcher before applying changes. This architecture makes resource definitions auditable without reading Go code, but creates an implicit coupling: template file names are string constants scattered across controller methods, so renaming a template requires updating the corresponding controller call.

The operator manages two distinct controller loops. The `ModelRegistryReconciler` is the primary controller, creating one complete application stack per ModelRegistry CR. Each stack includes a Deployment (with optional kube-rbac-proxy sidecar), Service, ServiceAccount, Role, RoleBinding, OpenShift Group, and optionally an OpenShift Route, Gateway API HTTPRoute with ReferenceGrant, NetworkPolicy, and auto-provisioned PostgreSQL (Deployment, Service, PVC, Secret, NetworkPolicy). The `ModelCatalogReconciler` manages a singleton catalog instance using a fixed reconcile request pattern — all watched resources map to the same `{name: "model-catalog", namespace: targetNamespace}` request for workqueue deduplication. This prevents reconcile storms when multiple owned resources change simultaneously.

The authentication architecture has undergone significant migration. The v1alpha1 CRD API supported Istio service mesh with Authorino, VirtualService, DestinationRule, and Gateway configurations. The v1beta1 API replaced this with kube-rbac-proxy sidecar injection and direct OpenShift Route/Gateway API HTTPRoute exposure. The operator includes automatic migration logic: the `MigrateOAuthProxyToKubeRBACProxy` method transparently converts existing OAuthProxy configurations to KubeRBACProxy settings during webhook defaulting and controller reconciliation. The kube-rbac-proxy enforces authorization via Kubernetes SubjectAccessReview — users must have `get` permission on the registry's Service resource in its namespace, providing fine-grained namespace-scoped access control.

The Gateway API integration creates cross-namespace resources: HTTPRoutes are created in the `HTTPROUTE_NAMESPACE` (default: `redhat-ods-applications`) referencing Services in the registry's namespace, with a shared ReferenceGrant per namespace allowing this cross-namespace reference. The `ReferenceGrant` is intentionally created with `createIfNotExists` semantics and never deleted when individual registries are removed, because it is shared across all registries in the namespace. The operator also supports a dual-ingress mode where both OpenShift Routes and Gateway API HTTPRoutes coexist for backward compatibility.

Platform awareness is a key design decision. The `DetectClusterCapabilities` function probes the Kubernetes discovery API at startup to detect which platform APIs are available (route.openshift.io, user.openshift.io, config.openshift.io, services.platform.opendatahub.io). This enables graceful degradation: on plain Kubernetes, no Routes/Groups are created; on BYOIDC OpenShift 4.20+, Group creation is skipped while Routes still work; on traditional OpenShift, all resources are created. The TLS security profile integration is particularly notable — the operator watches the cluster's `APIServer` CR for TLS profile changes and triggers a graceful shutdown/restart to reload the new profile, ensuring the operator's webhook and metrics servers always comply with the cluster's crypto policy.
