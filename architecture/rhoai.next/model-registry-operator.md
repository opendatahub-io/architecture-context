## Purpose

**Short**: Kubernetes operator that manages the lifecycle of Model Registry and Model Catalog instances, including database provisioning, authentication proxy sidecars, network policies, and storage version migration.

**Detailed**: The Model Registry Operator is a controller-runtime-based Kubernetes operator that watches `ModelRegistry` custom resources (CRD group `modelregistry.opendatahub.io`) and reconciles a complete stack of Kubernetes resources for each instance — including a REST API deployment with a kube-rbac-proxy sidecar for authentication, PostgreSQL or MySQL database connectivity (with optional auto-provisioned PostgreSQL), OpenShift Routes for external access, NetworkPolicies for isolation, and RBAC bindings for authorization delegation.

The operator also includes a Model Catalog controller that manages a cluster-wide catalog service with its own PostgreSQL database, kube-rbac-proxy sidecar, and catalog source ConfigMap discovery. The catalog aggregates model metadata from multiple sources (default sources, user-managed sources, labeled ConfigMaps) and exposes them through a REST API.

A storage version migration subsystem automatically migrates `ModelRegistry` resources from v1alpha1 to v1beta1 storage format using either the native Kubernetes `StorageVersionMigration` API or a manual re-read/re-write fallback strategy. The operator detects OpenShift cluster capabilities at startup and adapts behavior accordingly — creating Routes on OpenShift, managing user Groups (except in BYOIDC mode), and reading the cluster ingress domain from `config.openshift.io`.

## Architectural Analysis

The Model Registry Operator follows a mature kubebuilder v4 operator pattern with notable sophistication in several areas. The two-controller design — ModelRegistryReconciler for per-instance lifecycle and ModelCatalogReconciler for the singleton catalog — cleanly separates concerns while sharing infrastructure (templates, capabilities detection, kube-rbac-proxy configuration).

**Authentication migration path**: The operator implements a three-generation authentication evolution: Istio/Authorino (v1alpha1, fully removed) → OAuth proxy (v1alpha1/v1beta1, deprecated) → kube-rbac-proxy (v1beta1, current). The migration is fully automated — the defaulting webhook detects OAuth proxy configuration and transparently converts it to kube-rbac-proxy settings. Shared resource names between proxy types (e.g., `{name}-auth-delegator` ClusterRoleBinding) enable seamless in-place migration without orphaned resources. The kube-rbac-proxy authorization model uses `SubjectAccessReview` against `services/get` permission on the registry's own Service resource, which integrates naturally with Kubernetes RBAC rather than requiring a separate identity provider.

**Template-driven resource management**: Rather than constructing Kubernetes objects in Go code, the operator uses embedded Go templates (`templates/*.yaml.tmpl`) that are rendered with the ModelRegistry spec as context. This approach makes the generated resources readable and auditable but introduces a maintainability trade-off — template rendering errors surface at runtime rather than compile time. The `k8s-objectmatcher` library provides smart diff calculation for `CreateOrUpdate`, preventing unnecessary updates when specs haven't changed.

**BYOIDC awareness**: The operator detects "Bring Your Own Identity Provider" OpenShift configurations at startup by checking for the `user.openshift.io` API group. In BYOIDC mode, user Group management is delegated to the external IdP, and the operator skips creating `{name}-users` Group resources. This is a clean capability-detection pattern that avoids feature flags.

**Storage version migration**: The dual-strategy migration system (native SVM API with manual fallback) is a noteworthy pattern for operators managing CRD version transitions. The manual strategy leverages conversion webhooks by reading resources in the target version and writing them back, triggering automatic conversion. The 30-second initial delay and annotation-based duplicate prevention show production-hardened design. However, the migration monitor runs as a background goroutine rather than a separate controller, so its lifecycle is tied to the operator process rather than being independently observable.

**Network isolation model**: Each ModelRegistry instance gets its own NetworkPolicy restricting kube-rbac-proxy port access to OpenShift ingress namespaces (`network.openshift.io/policy-group: ingress`) and ODH-generated namespaces (`opendatahub.io/generated-namespace: true`). Auto-provisioned PostgreSQL databases get a separate NetworkPolicy restricting port 5432 access to only the registry pod itself. This provides defense-in-depth even within the same namespace.
