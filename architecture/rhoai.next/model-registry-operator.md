## Purpose

**Short**: Manages the lifecycle of Model Registry and Model Catalog instances on OpenShift AI, deploying REST services, authentication proxies, database backends, and Gateway API routes.

**Detailed**: The Model Registry Operator is a Kubernetes operator built with controller-runtime that manages two primary controllers: `ModelRegistryReconciler` and `ModelCatalogReconciler`. For each `ModelRegistry` custom resource, the operator provisions a complete deployment stack including a REST API service (model-registry), an optional kube-rbac-proxy sidecar for authentication, per-instance PostgreSQL or MySQL database connections, OpenShift Routes, Gateway API HTTPRoutes, RBAC resources (Roles, RoleBindings, ClusterRoleBindings, Groups), NetworkPolicies, and ServiceAccounts.

The operator supports two authentication patterns with automatic migration between them: the legacy Istio/Service Mesh pattern (v1alpha1, deprecated), the OAuth proxy pattern (v1alpha1, being migrated), and the current kube-rbac-proxy pattern (v1beta1). The defaulting webhook automatically migrates Istio configurations to OAuth proxy, and OAuth proxy configurations to kube-rbac-proxy, ensuring seamless upgrades across operator versions.

The Model Catalog controller manages a singleton model-catalog deployment that aggregates model metadata from multiple ConfigMap sources, supports admin group RBAC via the platform Auth CR, auto-provisions PostgreSQL, and exposes its API through both OpenShift Routes and Gateway API HTTPRoutes. The operator also handles CRD storage version migration (v1alpha1 → v1beta1) using either the Kubernetes StorageVersionMigration API or a manual re-read/re-write fallback strategy.

## Architectural Analysis

The Model Registry Operator follows a multi-generational authentication architecture that reflects the platform's evolution from Istio Service Mesh (RHOAI 2.x) through OAuth proxy to the current kube-rbac-proxy pattern (RHOAI 3.x). The code preserves all three patterns with automatic migration: the v1alpha1 API's `Istio` and `Gateway` config fields are converted to `OAuthProxy` during defaulting, and `OAuthProxy` is further migrated to `KubeRBACProxy` in the v1beta1 webhook. This chain means existing CRs from any era are transparently upgraded without user intervention, which is a significant operational convenience but adds complexity to the codebase. The `MigrateOAuthProxyToKubeRBACProxy()` method in the webhook and the reconciler's `if modelRegistry.Spec.OAuthProxy != nil` check are critical migration codepaths that must remain until all clusters have completed the transition.

The dual-controller architecture (ModelRegistryReconciler + ModelCatalogReconciler) represents a deliberate design split: per-instance registries are CR-driven with owner references and finalizers, while the singleton model-catalog uses a fixed reconcile request with deduplication to prevent storms from its many watched resources (ConfigMaps, Secrets, Deployments, etc.). The catalog controller's `mapToFixedCatalogRequest` pattern is worth noting — all watched objects map to the same `NamespacedName{Name: "model-catalog", Namespace: targetNamespace}` key, relying on controller-runtime's workqueue deduplication to coalesce events. This is an elegant solution to the N-to-1 mapping problem.

The Gateway API integration is notable for its cross-namespace design: HTTPRoutes are created in `HTTPROUTE_NAMESPACE` (default: `redhat-ods-applications`) while backend Services live in per-registry namespaces. This requires `ReferenceGrant` CRs in the Service namespace to authorize cross-namespace references. The ReferenceGrant is namespace-scoped (one per registries namespace) and intentionally NOT deleted when individual registries are removed — it covers all registries in the namespace. The HTTPRoute path pattern `/model-registry/{name}/` with `URLRewrite` to strip the prefix enables path-based routing through a shared Gateway.

The cluster capability detection system (`DetectClusterCapabilities`) handles three platform modes: traditional OpenShift (all APIs), BYOIDC OpenShift 4.20+ (route API present but user API absent), and plain Kubernetes. The operator gracefully handles partial discovery failures, distinguishing between expected BYOIDC scenarios (user.openshift.io API missing) and actual errors (route.openshift.io or config.openshift.io failing). In BYOIDC mode, the operator skips Group creation and expects external IdP group management, while still creating RoleBindings that reference the expected group names. The TLS security profile integration via `openshift/controller-runtime-common` ensures the operator's webhook and metrics servers use the cluster-configured cipher suites and TLS versions, with automatic restart on profile changes.
