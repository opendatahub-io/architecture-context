RBAC Summary:
- No Kubernetes RBAC resources defined
- The orchestrator runs as a standard service without Kubernetes API access
- No ClusterRoles, Roles, or RoleBindings

TLS Configuration:
- Server TLS: Optional, configured via CLI args (--tls-cert-path, --tls-key-path)
- Server mTLS: Optional client CA verification (--tls-client-ca-cert-path)
- Client TLS: Per-service named TLS profiles in config.yaml
  - Each downstream service references a TLS profile by name
  - Profiles define: cert_path, key_path, client_ca_cert_path
  - "insecure" mode available per-service for development (skips cert verification)
- TLS Implementation: rustls with ring backend (no OpenSSL dependency)

Authentication Summary:
- Client → Orchestrator: Passthrough headers (configurable set forwarded to downstream)
  - Optional X-Forwarded-Access-Token to Bearer header rewrite
- Orchestrator → gRPC services: mTLS client certificates + optional Bearer token (from api_token env var)
- Orchestrator → HTTP services: TLS + optional Bearer token (from api_token env var) + passthrough headers
- Health endpoints (/health, /info on port 8034): No authentication

Secrets:
- TLS server certificate (kubernetes.io/tls) - Server TLS termination, provisioned by platform operator/cert-manager
- TLS client CA certificate (Opaque) - mTLS client verification, provisioned by platform operator/cert-manager
- Per-service TLS certificates (Opaque) - Client mTLS to downstream services, provisioned by platform operator
- API tokens (Opaque, env vars) - Bearer tokens for downstream service auth, provisioned by platform operator

Concurrency & Rate Limiting:
- detector_concurrent_requests: 5 (default) - parallel detector calls per request
- chunker_concurrent_requests: 5 (default) - parallel chunker calls per request
- No explicit rate limiting (relies on upstream load balancer/gateway)
