TLS Configuration:
- Server TLS: Optional, configured via CLI args (--tls-cert-path, --tls-key-path)
- Server mTLS: Optional client CA verification (--tls-client-ca-cert-path)
- Client TLS: Per-service named TLS profiles in config.yaml
  - Pattern: TLS profiles defined in `tls` section with named IDs
  - Services reference TLS profiles by name (avoids cert path duplication)
  - HTTP: hyper-rustls connectors (reqwest)
  - gRPC: tonic native TLS
- Insecure Mode: Per-service TLS verification can be disabled (development only)
- Implementation: rustls with ring backend (no OpenSSL dependency)

RBAC Summary:
- No Kubernetes RBAC resources defined
- The orchestrator runs as a standard service without Kubernetes API access
- No ClusterRoles, Roles, or RoleBindings
- No ServiceAccount-specific permissions beyond default

Secrets:
- TLS server certificate (kubernetes.io/tls) - Server TLS termination
  - Provisioned by: Platform operator / cert-manager
  - Args: --tls-cert-path, --tls-key-path
- TLS client CA certificate (Opaque) - mTLS client verification
  - Provisioned by: Platform operator / cert-manager
  - Args: --tls-client-ca-cert-path
- Per-service TLS certificates (Opaque) - Client mTLS to downstream services
  - Provisioned by: Platform operator
  - Config: cert_path, key_path, client_ca_cert_path in config.yaml
- API tokens (Opaque, env vars) - Bearer tokens for downstream authentication
  - Provisioned by: Platform operator
  - Config: api_token field referencing env vars per service

Authentication Flow:
- Client → Orchestrator: Passthrough headers (configurable set forwarded as-is)
  - Optional: X-Forwarded-Access-Token rewritten to Authorization: Bearer
- Orchestrator → Downstream gRPC: mTLS client certificates + optional Bearer token
- Orchestrator → Downstream HTTP: TLS + optional Bearer token from api_token env var
- Health endpoints: No authentication required

Concurrency & Rate Limiting:
- Detector concurrent requests: configurable (default: 5)
- Chunker concurrent requests: configurable (default: 5)
- Detection tasks parallelized via futures::stream buffer_unordered
- Streaming: broadcast channel fan-out to multiple detector workers
