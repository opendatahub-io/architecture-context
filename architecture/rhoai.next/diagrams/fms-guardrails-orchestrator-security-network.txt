RBAC Summary:
- No Kubernetes RBAC defined — standalone service binary
- RBAC is managed by the deploying platform operator (e.g., TrustyAI or KServe)
- No ClusterRoles, Roles, or RoleBindings in this repository

Authentication & Authorization Summary:
- /api/v1/*, /api/v2/*: Header passthrough (configurable allowed_headers list)
  - Headers matching passthrough_headers config forwarded to downstream services
  - Optional X-Forwarded-Access-Token → Authorization: Bearer rewrite
- /health, /info: Unauthenticated (separate port 8034, plaintext)
- TLS/mTLS: Optional server TLS with optional client cert verification (WebPKI)
  - Configured via --tls-cert-path, --tls-key-path, --tls-client-ca-cert-path
- Downstream auth: Per-service configurable Bearer tokens (api_token field from env vars)

Secrets:
- TLS server cert/key (kubernetes.io/tls) - TLS termination, platform operator/cert-manager, no auto-rotate
- TLS client CA cert (Opaque) - mTLS client verification, platform operator, no auto-rotate
- Downstream TLS certs (Opaque) - Client certs for detectors/chunkers/generation, platform operator, no auto-rotate
- API tokens (Opaque, env var) - Bearer tokens for OpenAI-compatible and detector services, platform operator, no auto-rotate

FIPS Compliance Notes:
- WARNING: Application uses rustls with ring crypto provider (NOT FIPS-validated)
- OpenSSL is installed in the container image (satisfying check-payload gate)
- However, the application's TLS bypasses FIPS-validated OpenSSL entirely
- ring is a pure-Rust/asm crypto library — potential FIPS gap

Build Hermeticity:
- Konflux hermetic builds with cargo + RPM prefetch
- Cargo.lock present (language dependency pinning)
- rpms.lock.yaml present (OS package pinning)
- Multi-architecture: x86_64, ppc64le, arm64, s390x
