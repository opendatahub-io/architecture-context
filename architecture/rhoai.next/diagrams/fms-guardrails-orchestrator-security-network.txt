TLS Configuration:
- Server TLS: rustls with ring crypto backend
  - Certificate: PEM file via --tls-cert-path CLI arg
  - Key: PEM file via --tls-key-path CLI arg
  - mTLS: WebPkiClientVerifier via --tls-client-ca-cert-path CLI arg
  - Supports TLS 1.2+
- Client TLS: Named profiles in YAML config
  - Per-service TLS configuration (cert_path, key_path, ca_cert_path)
  - Insecure mode: NoVerifier disables all cert verification (dev only)
  - Supports legacy signature schemes (RSA_PKCS1_SHA1, ECDSA_SHA1_Legacy)
  - System trust store (native-roots) or webpki-roots

RBAC Summary:
- No RBAC manifests. This service does not interact with the Kubernetes API.
- No ClusterRoles, Roles, or RoleBindings defined.

Secrets:
- TLS server cert/key (kubernetes.io/tls) - Server-side TLS for guardrails API
  - Provisioned manually via CLI args (--tls-cert-path/--tls-key-path)
  - No auto-rotation
- TLS client CA cert (Opaque) - mTLS client verification
  - Provisioned manually via CLI arg (--tls-client-ca-cert-path)
  - No auto-rotation
- Per-service TLS certs (Opaque) - Client TLS for downstream gRPC/HTTP services
  - Provisioned manually via YAML config tls section
  - No auto-rotation
- API tokens (Opaque) - Bearer auth for downstream HTTP services
  - Read from environment variables (api_token config field names env var)
  - No auto-rotation

Authentication & Authorization:
- Orchestrator itself performs NO authentication/authorization
- Header passthrough: X-Forwarded-Access-Token rewritten to Bearer header
- Headers matching passthrough_headers config forwarded to downstream services
- Detector/OpenAI endpoints use optional Bearer tokens from env vars
- All auth enforcement is delegated to downstream services

FIPS Compliance Notes:
- Uses ring crypto library (NOT OpenSSL FIPS module) for all TLS operations
- OpenSSL is installed in container image (satisfies check-payload for base image)
- Rust binary uses ring via rustls - potential FIPS compliance gap
- NoVerifier struct supports legacy SHA1 signature schemes (non-FIPS)
