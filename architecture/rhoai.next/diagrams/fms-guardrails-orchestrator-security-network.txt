RBAC Summary:
- No RBAC resources defined in this repository
- RBAC is managed by the platform deployment (rhods-operator)

Secrets:
- TLS cert/key pair (server) - TLS certificate + private key for guardrails API server
  Provisioned by: External (mounted via --tls-cert-path, --tls-key-path)
  Auto-rotate: No (file-based)

- TLS client CA cert (server) - CA certificate for mTLS client verification
  Provisioned by: External (mounted via --tls-client-ca-cert-path)
  Auto-rotate: No (file-based)

- Client TLS configs (per-service) - cert + key + CA cert per named TLS config
  Provisioned by: External (paths in config YAML)
  Auto-rotate: No (file-based)

- API token (per-service) - Bearer token for OpenAI-compatible backends
  Provisioned by: External (env var name in config YAML, resolved at startup)
  Auto-rotate: No

FIPS Compliance Notes:
- Container image includes OpenSSL (openssl-libs) satisfying container-level checks
- Application uses rustls with ring crypto backend (NOT FIPS-validated)
- ring uses BoringSSL-derived crypto, not system OpenSSL
- FIPS GAP: Binary passes container-level FIPS checks but does NOT use
  FIPS-validated cryptography at the application level
- NoVerifier struct disables all cert verification when insecure: true
  is set in per-service TLS config (security risk for production)

Authentication Notes:
- No built-in authentication on any endpoint
- Header passthrough mechanism: headers matching passthrough_headers config
  are forwarded to downstream services
- X-Forwarded-Access-Token header is rewritten to Authorization: Bearer
  (potential token injection risk - must be secured)
- Health endpoints (/health, /info) are unauthenticated on plaintext port
