RBAC Summary:
- No Kubernetes RBAC roles defined (NeMo Guardrails is a stateless service,
  not an operator/controller)
- No CRDs defined
- No ClusterRoles or RoleBindings
- Deployment RBAC managed by platform operator (RHOAI)

Secrets:
- LLM API keys (Opaque) - API keys for upstream LLM providers
  - OPENAI_API_KEY
  - ANTHROPIC_API_KEY
  - AZURE_OPENAI_API_KEY
  - COHERE_API_KEY
  - Provisioned by: Manual / platform operator
  - Auto-rotate: No

Authentication Model:
- NeMo Guardrails has NO built-in authentication or authorization
- Header Forwarding: X-* headers and Authorization header are forwarded
  to upstream LLM providers
- Auth Token Redaction: Authorization header values are redacted in
  forwarded X-* headers (security measure in fork)
- Access control to the guardrails service itself must be enforced by:
  - Service mesh (Istio/OSSM) policies
  - Network policies
  - External gateway/ingress authentication

Guardrail Filtering (Build-time Security):
- Profile-based filtering via scripts/filter_guardrails.py
- "opensource" profile (RHOAI default): removes 15 proprietary guardrail
  modules (ActiveFence, AI Defense, Cleanlab, etc.)
- Ensures no proprietary API dependencies in shipped container
- Remaining open-source guardrails: content safety, factchecking,
  hallucination detection, injection detection, regex filtering,
  self-check, sensitive data (Presidio), topic safety, attention-based,
  GLiNER NER, Guardrails AI, Llama Guard

Network Policy Recommendations:
- INGRESS: Allow port 8000/TCP from platform namespace only
- EGRESS: Allow port 443/TCP to configured LLM provider endpoints
- EGRESS: Allow port 4317-4318/TCP to OTEL Collector (if configured)
- DENY: All other ingress/egress traffic
