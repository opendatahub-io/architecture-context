RBAC Summary:
- No Kubernetes RBAC resources defined
- This is a stateless service container, not a Kubernetes operator
- No ClusterRoles, Roles, or RoleBindings

Authentication & Authorization:
- No built-in authentication mechanism
- Authorization header passthrough: incoming Bearer tokens forwarded to upstream LLM providers
- Relies entirely on platform-level auth enforcement (kube-rbac-proxy or Gateway API)
- Auth token redaction in logs (logging/callbacks.py) prevents credential leakage

Secrets:
- LLM API keys (Opaque, env vars) — API authentication to upstream LLM providers
  - Provisioned by: Manual / External secret management
  - Auto-rotate: No
  - Forwarding pattern: Auth headers from client requests OR env var-based shared keys
  - api_key_env_var model config for shared-key deployments

FIPS Compliance:
- Python service on UBI9 (FIPS-validated OpenSSL)
- cryptography==44.0.3 uses system OpenSSL backend
- MD5 usage: hashlib.md5 for cache keys (non-security use)
  - utils.py (line 410-416): FIPS-aware fallback to SHA-256
  - embeddings/cache.py (line 71): No FIPS fallback (cache indexing only)

Build Hermeticity:
- poetry.lock + uv-compiled requirements.txt: Python deps locked
- artifacts.lock.yaml: spaCy model locked with SHA-256
- Tekton PipelineRun: hermetic=false (not fully hermetic)
- RPM locks absent on main branch (expected on release branches)

Network Security Notes:
- Internal service port 8000/TCP is HTTP (no TLS)
- All egress to LLM providers uses HTTPS/TLS 1.2+
- No service mesh sidecar injection defined in this component
- Platform is expected to provide ingress TLS and auth enforcement
- Header forwarding enables per-request auth delegation without storing keys
