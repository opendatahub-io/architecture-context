RBAC Summary:
- Not applicable. NeMo Guardrails is a standalone service, not a Kubernetes operator.
- No ClusterRoles, Roles, or RoleBindings defined.

Authentication Summary:
- No server-side authentication enforcement
- Bearer tokens from client requests are forwarded to upstream LLM providers
- Auth token redaction in forwarded headers (fork customization)
- Unauthenticated endpoints: /v1/rails/configs, /v1/challenges, / (health), /v1/actions/*

Secrets:
- OPENAI_API_KEY (env var) - OpenAI API auth, manual provisioning, no auto-rotation
- AZURE_OPENAI_API_KEY (env var) - Azure OpenAI API auth, manual provisioning, no auto-rotation
- ANTHROPIC_API_KEY (env var) - Anthropic API auth, manual provisioning, no auto-rotation
- COHERE_API_KEY (env var) - Cohere API auth, manual provisioning, no auto-rotation
- Redis credentials (env var/URL, optional) - Redis auth for thread storage, manual, no auto-rotation

FIPS Compliance Notes:
- Python runtime inherits OpenSSL from UBI9 base image
- RHEL FIPS mode: Python ssl/hashlib auto-use FIPS-validated OpenSSL
- KNOWN GAP: MD5 used in utils.py (get_hash) and embeddings/cache.py for cache keys
  - Will raise ValueError under FIPS strict mode unless usedforsecurity=False is set
  - Usage is non-security-critical (cache key generation)
- No custom CA bundle or cert pinning; system trust store only
- Server runs plaintext HTTP; TLS termination expected at ingress layer

Security Observations:
- No server-side auth: Any client with network access can invoke guardrail checks
- Plaintext HTTP: All internal traffic is unencrypted; relies on platform network policies
- Redis connection: No TLS validation; credentials optional
- Actions server: Completely unauthenticated; can execute arbitrary registered actions
- Bearer token forwarding: Sensitive tokens traverse plaintext HTTP within cluster
