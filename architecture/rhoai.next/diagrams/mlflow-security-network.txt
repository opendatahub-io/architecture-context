RBAC Summary:
- ClusterRole: mlflow-executor (required for K8s project backend)
  - API Groups: batch
  - Resources: jobs
  - Verbs: create, get, list, watch, delete

Authentication & Authorization:
- Primary AuthN: kube-rbac-proxy sidecar validates Kubernetes SA tokens
- Secondary AuthN: MLflow kubernetes-namespaced auth provider resolves workspace
- AuthZ: MLflow internal permission system (READ, USE, EDIT, MANAGE per resource)
- Workspace Isolation: Kubernetes namespace → MLflow workspace mapping
- CORS: Cross-origin state change blocking enabled
- DNS Rebinding: Host header validation on all state-changing requests

Secrets:
- MLFLOW_CRYPTO_KEK_PASSPHRASE (Opaque env var) - KEK for gateway secrets envelope encryption, manual provisioning, no auto-rotate (manual via `mlflow crypto rotate-kek`)
- MLFLOW_FLASK_SERVER_SECRET_KEY (Opaque env var) - Flask session cookie signing, manual provisioning, no auto-rotate
- Database credentials (Opaque connection string) - PostgreSQL user/pass in backend store URI, manual or operator provisioned, no auto-rotate
- AI Gateway provider API keys (Opaque, encrypted in DB) - LLM provider API keys (OpenAI, Anthropic, etc.), admin provisioned via Gateway API, KEK rotation available

Encryption at Rest:
- AI Gateway secrets: Envelope encryption with KEK passphrase (MLFLOW_CRYPTO_KEK_PASSPHRASE)
- KEK rotation: `mlflow crypto rotate-kek` CLI command
- Database: Relies on PostgreSQL encryption settings
- Artifacts: Relies on storage backend encryption (S3 SSE, Azure SSE, GCS)

Network Notes:
- MLflow server listens on HTTP (plaintext) port 5000 only
- TLS termination handled by kube-rbac-proxy (8443) and platform Gateway (443)
- Pod-local traffic between kube-rbac-proxy and MLflow is unencrypted (same pod network namespace)
- All egress to external services uses TLS 1.2+
- No service mesh sidecar (relies on kube-rbac-proxy for auth boundary)
