RBAC Summary:
- No ClusterRoles or RoleBindings defined
- ServiceAccounts used for pod identity only:
  - {release}-batch-gateway-apiserver
  - {release}-batch-gateway-processor
  - {release}-batch-gateway-gc
- No Kubernetes API server access required (no CRDs, no watches)
- Authorization is application-level only (tenant header isolation)

Secrets:
- batch-gateway-secrets (Opaque) — redis-url, postgresql-url, inference-api-key,
  s3-secret-access-key. Provisioned manually. No auto-rotation.
- {release}-batch-gateway-apiserver-tls (kubernetes.io/tls) — TLS cert/key for apiserver.
  Provisioned by cert-manager (when enabled) or manually. Auto-rotates with cert-manager.

Authentication Model:
- batch-gateway does NOT perform authentication
- Tenant identity extracted from configurable HTTP header (default: X-MaaS-Username)
- Relies on upstream auth proxy (kube-rbac-proxy, external auth service) to validate
  identity and inject tenant header
- All data queries scoped by tenant ID
- File storage paths include SHA256-derived tenant-scoped folder names

TLS Configuration:
- Apiserver: Optional TLS with tls.Config{MinVersion: tls.VersionTLS12}
- Inference client: Configurable TLS with custom CA bundle, client certs, InsecureSkipVerify
- Redis: Configurable TLS/mTLS support
- PostgreSQL: TLS configurable via connection URL parameters
- All crypto: stdlib crypto/tls only (no third-party crypto libraries)

FIPS Status:
- Currently builds with CGO_ENABLED=0 (static linking — would fail check-payload)
- Dockerfiles expose CGO_ENABLED as build ARG for downstream override to CGO_ENABLED=1
- No non-FIPS crypto dependencies detected
- Planned migration from ubi-micro to distroless base images
