Service Mesh Configuration:
- KServe-deployed detectors (HF, LLM Judge):
  - sidecar.istio.io/inject: 'true' (Istio sidecar injection)
  - serving.knative.openshift.io/enablePassthrough: 'true'
  - security.opendatahub.io/enable-auth: 'true' (KServe auth delegation)
  - Transport: mTLS between services via Istio sidecar
- Built-in detector:
  - No Istio sidecar
  - Relies on network-level isolation within cluster

Authentication & Authorization:
- Application-level: None (no auth in detector code)
- Platform-level: KServe + Istio sidecar enforces auth for HF and LLM Judge detectors
- Built-in detector: Network isolation only (no auth enforcement)

Security Notes:
- Custom Python detectors in built-in detector use AST-based static analysis sandbox
  - Blocked imports: os, subprocess, sys, shutil
  - Blocked calls: eval, exec, open, compile, input
  - Limitation: Static analysis can be circumvented by obfuscation
- All detector endpoints expose plaintext HTTP (TLS/auth delegated to platform)
- HuggingFace Hub access (443/TCP HTTPS) is optional and only used during init
