NETWORK SUMMARY:
===============

Ingress:  None (ai4rag is a library, not a network service)
Egress:   OGX Server only (all API calls via ogx-client over HTTPS)

TRUST BOUNDARIES:
=================

1. In-Process Boundary:
   - All ai4rag components run in the same Python process
   - ChromaDB runs in-memory (no network, ephemeral)
   - LangChain, Unitxt, scikit-learn, pygam all run in-process
   - No inter-process or inter-container communication

2. Network Boundary (Egress Only):
   - ai4rag → OGX Server via HTTPS/TLS 1.2+
   - Authentication: API Key (Bearer token)
   - Client: ogx-client SDK wrapping httpx
   - Certificate validation: System trust store

RBAC SUMMARY:
=============

None. ai4rag is a Python library with no Kubernetes RBAC requirements.
RBAC is the responsibility of the consuming service (e.g., KFP component pod).

SECRETS:
========

| Secret              | Type      | Storage                    | Rotation    |
|---------------------|-----------|----------------------------|-------------|
| OGX API Key         | API Key   | User-provisioned (.env)    | Manual (No) |

- The API key is passed to OgxClient(api_key=...) at construction time
- No secrets are stored or managed by ai4rag itself
- In KFP context, the API key would typically come from a Kubernetes Secret
  mounted as an environment variable in the pipeline component pod

SERVICE MESH:
=============

Not applicable. ai4rag is a library, not a deployed service.
Service mesh participation depends on the consuming service's deployment.

TLS CONFIGURATION:
==================

- ai4rag does not configure TLS directly
- TLS behavior is inherited from ogx-client → httpx → Python ssl module
- Certificate trust store: System default (inherited from container/host)
- Minimum TLS version: Depends on Python ssl module configuration
- FIPS compliance: Depends on the system OpenSSL and Python build
