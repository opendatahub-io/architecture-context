NETWORK SURFACE SUMMARY:
═══════════════════════════════════════════════════════════════════════════

Ingress:  NONE (ai4rag exposes no network listeners, ports, or services)
Egress:   OGX Server only (HTTPS, TLS 1.2+, API Key auth)

RBAC Summary:
  NONE - ai4rag is a Python library, not a Kubernetes operator.
  No ClusterRoles, Roles, RoleBindings, or ServiceAccounts are created.
  RBAC is the responsibility of the consuming service.

Service Mesh Configuration:
  N/A - ai4rag does not deploy pods or participate in service mesh.
  mTLS and PeerAuthentication depend on the consuming service's namespace config.

Secrets:
  NONE managed by ai4rag directly.
  - OGX API Key: Passed programmatically via OgxClient(api_key=...) constructor.
    The consuming service is responsible for sourcing this from a Kubernetes Secret
    or environment variable.

TLS / Crypto:
  - No custom TLS configuration in ai4rag source code
  - No direct crypto library imports (no hashlib for auth, no custom ciphers)
  - hashlib.sha256 used ONLY for document content deduplication (FIPS-safe)
  - All TLS handled by ogx-client → httpx → system OpenSSL
  - Certificate validation: httpx defaults (system trust store)
  - FIPS compliance: Inherited from consuming service's container image
    and Python runtime (ai4rag has no compiled binaries)

FIPS Compliance:
  - Build-time: N/A (pure Python library, no container image, no Konflux pipeline)
  - Runtime: Delegated to consuming service's container image and OpenSSL
  - No non-FIPS crypto risks detected in ai4rag source code

Container Image:
  N/A - ai4rag is distributed as a PyPI package, not a container image.
  The consuming service packages ai4rag into its own container.
