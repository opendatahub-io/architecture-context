RBAC Summary:
- No RBAC resources — this is a standalone service, not a Kubernetes operator
- No ClusterRoles, Roles, or RoleBindings defined in this repository
- Deployment RBAC is managed by the deploying operator (e.g., TrustyAI operator)

Secrets:
- TLS cert (/etc/tls/private/tls.crt)
  Type: kubernetes.io/tls
  Purpose: mTLS client certificate for authenticating to orchestrator
  Provisioned by: Platform/cert-manager (external)
  Auto-rotate: Unknown

- TLS key (/etc/tls/private/tls.key)
  Type: kubernetes.io/tls
  Purpose: mTLS private key for orchestrator communication
  Provisioned by: Platform/cert-manager (external)
  Auto-rotate: Unknown

- CA cert (/etc/tls/ca/service-ca.crt)
  Type: Opaque
  Purpose: Custom CA certificate for trusting orchestrator's server certificate
  Provisioned by: Platform/service-ca-operator (external)
  Auto-rotate: Unknown

Authentication & Authorization:
- Endpoint: /{route_name}/v1/chat/completions (POST)
- Auth mechanism: Passthrough (forwards Authorization header to orchestrator)
- Enforcement point: FMS Guardrails Orchestrator (not gateway)
- No local auth enforcement; gateway transparently forwards client
  Authorization and x-forwarded-* headers

FIPS Compliance:
- Rust binary dynamically links against system OpenSSL (openssl crate v0.10.73)
- Uses native-tls for TLS transport (delegates to system OpenSSL)
- On FIPS-enabled RHEL/OpenShift nodes, system OpenSSL operates in FIPS mode
- No hardcoded ciphers or non-FIPS algorithms detected
- danger_accept_invalid_hostnames(true) scoped to localhost only (dev/test)

TLS Configuration:
- Client identity: PEM cert/key → PKCS#12 conversion → reqwest native-tls
- Cipher suites: System OpenSSL defaults (no hardcoded selection)
- TLS version: Determined by system OpenSSL configuration
