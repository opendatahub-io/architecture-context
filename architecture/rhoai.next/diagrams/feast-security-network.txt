TLS / Certificate Configuration:
- OpenShift service-serving-cert-signer: Auto-provisions TLS certs for Feast services (auto-rotate)
- Manual TLS: User-provided kubernetes.io/tls Secrets supported
- ODH CA Bundle: Auto-detected from odh-ca-bundle.crt ConfigMap
- Go feature server: TLS 1.2+ minimum, P256 + X25519MLKEM768 curves
- Operator webhook: HTTP/2 disabled in TLS config

Secrets:
- feast-{name}-{service}-tls (kubernetes.io/tls) - Service TLS certs, auto/manual, auto-rotate on OpenShift
- feast-{name}-intra-communication (Opaque) - HMAC-SHA256 JWT for operator↔registry, auto-generated
- (OIDC secret ref) (Opaque) - OIDC client credentials, manual provisioning
- (DB credential secrets) (Opaque) - Database connection credentials, manual provisioning
- (OpenLineage API key) (Opaque) - OpenLineage collector API key, manual provisioning

Security Concerns:
⚠ InsecureSkipVerify: true in operator→registry REST client (cluster-internal only)
⚠ Redis TLS uses empty tls.Config{} (inherits platform defaults, no explicit hardening)
⚠ Intra-communication JWT uses HMAC-SHA256 with empty key (anti-replay, not strong auth)
⚠ MySQL connections have no TLS option
⚠ No OLM fips-compliant annotation in CSV manifest

FIPS Compliance:
- Operator: CGO_ENABLED=1, GOEXPERIMENT=strictfipsruntime, -tags strictfipsruntime
- Feature server: Inherits FIPS from UBI9 base image OpenSSL
- Go feature server: TLS 1.2+ with FIPS-compliant cipher suites via strictfipsruntime
