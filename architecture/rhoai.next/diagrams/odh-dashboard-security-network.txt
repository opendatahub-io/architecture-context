AUTHORIZATION CHECKPOINTS (3-layer model):
  1. Gateway → kube-rbac-proxy: SubjectAccessReview (can user list projects?)
  2. Fastify Backend: secureAdminRoute() for admin endpoints, secureRoute() for
     authenticated endpoints, user token passthrough for K8s API
  3. BFF Middleware: SubjectAccessReview (can user access target service in namespace?)

RBAC Summary:
  - ClusterRole: odh-dashboard
    - API Groups: storage.k8s.io, machine.openshift.io, config.openshift.io,
      operators.coreos.com, route.openshift.io, user.openshift.io,
      rbac.authorization.k8s.io, kubeflow.org, datasciencecluster.opendatahub.io,
      dscinitialization.opendatahub.io, serving.kserve.io,
      modelregistry.opendatahub.io, services.platform.opendatahub.io,
      llamastack.io, trustyai.opendatahub.io, feast.dev, mlflow.opendatahub.io,
      authentication.k8s.io, authorization.k8s.io
    - Verbs: get, list, watch, create, update, patch, delete (varies by resource)
  - ClusterRoleBinding: odh-dashboard → SA/odh-dashboard
  - ClusterRoleBinding: odh-dashboard-auth-delegator → SA/odh-dashboard
    (system:auth-delegator for SubjectAccessReview/TokenReview)
  - ClusterRoleBinding: odh-dashboard-monitoring → SA/odh-dashboard
    (cluster-monitoring-view for Prometheus access)

Secrets:
  - dashboard-proxy-tls (kubernetes.io/tls)
    - Purpose: TLS certificate for kube-rbac-proxy and BFF sidecars
    - Provisioned by: OpenShift service-serving-cert-signer
    - Annotation: service.beta.openshift.io/serving-cert-secret-name
    - Auto-Rotate: Yes (auto-rotated by OpenShift)

FIPS Compliance:
  - Host (Node.js): FIPS-compliant OpenSSL via UBI 9 base image
  - BFF Sidecars (Go): CGO_ENABLED=1, -tags strictfipsruntime
  - TLS: 1.3 minimum for BFF upstream connections
  - esbuild Go binaries explicitly removed from production image

Network Policy:
  - Ingress: Restricted to configured ports from openshift-ingress namespace
    and same-namespace pods
  - Egress: All in-cluster egress allowed (supports inter-BFF and upstream calls)

CA Certificate Loading (5 paths at startup):
  - System trust store
  - Kubernetes ServiceAccount CA bundle
  - ODH custom CA bundles (configurable via BUNDLE_PATHS)
