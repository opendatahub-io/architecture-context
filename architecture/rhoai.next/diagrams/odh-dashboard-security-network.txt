Authentication Architecture:
  - Entry Point: kube-rbac-proxy (port 8443)
  - AuthN Method: TokenReview (validates OAuth Bearer token against K8s API)
  - AuthZ Method: SubjectAccessReview (user must be able to "list projects")
  - Unauthenticated Paths: /api/health, /api/quickstarts, /api/docs
  - Admin Paths: /api/config (PATCH), /api/modelRegistries/*, /api/cluster-settings
  - Admin Check: secureAdminRoute middleware verifies admin group membership
  - User Path Isolation: secureRoute middleware checks namespace ownership + username

Secrets:
  - dashboard-proxy-tls (kubernetes.io/tls) - TLS cert for kube-rbac-proxy
    Provisioned By: OpenShift service-ca (annotation-based)
    Auto-Rotate: Yes (service-ca managed)

NetworkPolicy:
  Ingress:
    - Allow from: OpenShift ingress controller (namespace: openshift-ingress)
    - Allow from: Same namespace pods
    - Ports: 8443/TCP (kube-rbac-proxy), 8043-8743/TCP (module ports)
  Egress:
    - All in-cluster traffic: ALLOWED (unrestricted)
    - External: 80/TCP, 443/TCP (HTTP/HTTPS)

Security Observations:
  - Defense-in-depth: kube-rbac-proxy (RBAC) → app-level authZ → upstream auth
  - Dashboard never stores credentials - always delegates to platform
  - User tokens are propagated (not exchanged) to upstream services
  - Several upstream services use plaintext HTTP (in-cluster assumed trusted)
  - FIPS compliance: Go BFFs built with -tags strictfipsruntime
  - Pod runs 9+ containers sharing a single ServiceAccount
  - Egress NetworkPolicy is permissive (all in-cluster allowed)
