  Note: automountServiceAccountToken: false prevents automatic token mounting.
        Separate projected volumes isolate permissions between main and BFF containers.

Authentication Flow:

  1. Platform Gateway (Envoy) terminates external TLS
  2. kube-rbac-proxy validates identity, injects headers:
     - X-Forwarded-Access-Token: OAuth access token
     - X-Auth-Request-User: authenticated username
     - X-Auth-Request-Groups: user group memberships
  3. Dashboard backend applies secureRoute (user auth) or secureAdminRoute (admin check)
  4. BFF sidecars extract user token from x-forwarded-access-token header
  5. BFFs perform Kubernetes SubjectAccessReview (SAR) for service-level authorization
  6. WebSocket auth uses OAuth token encoded in WebSocket subprotocol (base64 Bearer)

FIPS Compliance:

  - Main dashboard (Node.js): OpenSSL via UBI9 nodejs-22 base, esbuild binaries removed
  - BFF modules (Go): CGO_ENABLED=1, -tags strictfipsruntime, dynamic linking
  - TLS: BFF servers enforce MinVersion TLS 1.3; system CA pool + custom bundles
  - Risk: INSECURE_SKIP_VERIFY flag in BFFs (dev only), rejectUnauthorized=false in Feast paths

Security Risks:

  - INSECURE_SKIP_VERIFY flag available in all Go BFFs (intended dev-only)
  - rejectUnauthorized=false in feature store integration paths
  - Per-instance token caching (no cross-replica sharing) may lead to stale tokens
  - MCP servers use variable encryption (HTTP/HTTPS) per-server configuration
