RBAC Summary:
- No RBAC permissions required
- rest-proxy does not interact with the Kubernetes API
- ServiceAccount inherited from parent ModelMesh Serving pod
- No ClusterRoles, Roles, ClusterRoleBindings, or RoleBindings

Secrets:
- TLS certificate (MM_TLS_KEY_CERT_PATH)
  - Type: TLS key/cert files
  - Purpose: TLS encryption for HTTP listener and/or gRPC upstream
  - Provisioned by: ModelMesh Serving / cert-manager
  - Auto-Rotate: Depends on provisioner

Authentication & Authorization:
- No direct authentication enforcement at rest-proxy level
- All REST endpoints (GET, POST) have no auth mechanism
- Auth is expected to be enforced by:
  - ModelMesh Serving's own auth layer
  - Platform ingress configuration
  - Service mesh (if deployed with Istio/OSSM)

TLS Configuration:
- HTTP Listener TLS:
  - Env: MM_TLS_KEY_CERT_PATH, MM_TLS_PRIVATE_KEY_PATH
  - Optional: Disabled by default
- gRPC Upstream TLS:
  - Env: REST_PROXY_USE_TLS (enables TLS to localhost:8033)
  - Option: REST_PROXY_TLS_SKIP_VERIFY (InsecureSkipVerify)
  - Cert: Reuses MM_TLS_KEY_CERT_PATH

Network Topology:
- All traffic is pod-local (localhost / 127.0.0.1)
- No external egress from rest-proxy
- No ingress resources defined by rest-proxy
- Traffic enters via ModelMesh Serving's Kubernetes Service
- No service mesh sidecar interaction (unless deployed by parent)

FIPS Compliance:
- Konflux build: GOEXPERIMENT=strictfipsruntime, CGO_ENABLED=1
- Base image: ubi9/ubi-minimal (FIPS-validated crypto)
- Community build: CGO_ENABLED=0, no FIPS enforcement
