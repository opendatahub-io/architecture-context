NETWORK SUMMARY:
- All traffic between rest-proxy and ModelMesh is localhost (pod-local)
- No cross-namespace or cross-pod network communication from rest-proxy
- External access to rest-proxy is via ModelMesh Serving's Kubernetes Service
- No ingress resources defined by rest-proxy itself

TLS CONFIGURATION:
- HTTP Listener TLS:
  - Enabled when MM_TLS_KEY_CERT_PATH and MM_TLS_PRIVATE_KEY_PATH are set
  - Certificate provisioned by ModelMesh Serving / cert-manager
  - Rotation depends on provisioner (cert-manager supports auto-rotation)
- gRPC Upstream TLS:
  - Enabled when REST_PROXY_USE_TLS=true
  - Can skip certificate verification via InsecureSkipVerify (dev/test only)
  - Uses same MM_TLS_KEY_CERT_PATH for CA trust

RBAC Summary:
- No RBAC required
- rest-proxy does not interact with Kubernetes API
- No ClusterRoles, Roles, or RoleBindings
- No ServiceAccount permissions needed

Secrets:
- TLS certificate (MM_TLS_KEY_CERT_PATH) - TLS key/cert files
  - Purpose: TLS encryption for HTTP listener and/or gRPC upstream
  - Provisioned By: ModelMesh Serving / cert-manager
  - Auto-Rotate: Depends on provisioner

Authentication & Authorization:
- No direct authentication enforcement by rest-proxy
- All endpoints accept unauthenticated requests
- Auth is expected to be enforced by:
  - ModelMesh Serving's auth layer
  - Platform ingress (e.g., Istio, OpenShift Route)
  - Service mesh policies (if deployed with service mesh)

FIPS Compliance:
- Konflux build: GOEXPERIMENT=strictfipsruntime, CGO_ENABLED=1
- Base images: ubi9/go-toolset (builder), ubi9/ubi-minimal (runtime)
- Required for RHOAI FedRAMP / FIPS 140-2 compliance
