RBAC Summary:
- OVMS does not define its own RBAC resources
- Pod runs as non-root user (UID 5000, GID 5000, user "ovms")
- No Kubernetes API access from the OVMS container
- kube-rbac-proxy sidecar performs TokenReview and SubjectAccessReview
  against the Kubernetes API for authentication and authorization

Authentication & Authorization:
- /v3/* (OpenAI API): Optional API key (Bearer token in header)
  - Enforcement: OVMS HTTP handler (isAuthorized() function)
  - Policy: Configured via server settings
- /v1/*, /v2/*: No application-level auth
  - Platform-level auth (kube-rbac-proxy, Istio) applied externally
- gRPC: No application-level auth
  - Platform-level auth applied externally
- All endpoints (RHOAI): kube-rbac-proxy sidecar enforces RBAC

Secrets:
- HF_TOKEN (env var) - HuggingFace authentication, user-provisioned, no auto-rotate
- AWS credentials (env vars / IAM role) - S3 access, user-provisioned / IRSA
- GCS credentials (service account JSON) - GCS access, user-provisioned, no auto-rotate
- Azure connection string (env var) - Azure Blob access, user-provisioned, no auto-rotate

FIPS Compliance Notes:
- No TLS termination in OVMS itself (delegated to platform)
- Uses system OpenSSL (UBI 9) which respects FIPS mode when enabled
- openssl/md5.h used in src/filesystem.hpp for file content hashing (dedup, not security)
- MD5 usage would trigger warnings in strict FIPS audits
- CURL SSL configuration delegates to system OpenSSL (CURLSSLOPT_NATIVE_CA)

Security Design Philosophy:
- OVMS intentionally does not implement TLS or strong auth internally
- gRPC: InsecureServerCredentials (plaintext)
- HTTP: Drogon listens on plaintext
- Security enforced at platform level:
  - kube-rbac-proxy (authentication/authorization)
  - Istio mTLS (encryption)
  - Gateway API / HTTPRoute (ingress)
