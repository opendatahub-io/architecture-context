RBAC Summary:
- OVMS does NOT define its own RBAC resources (ClusterRoles, RoleBindings)
- RBAC is managed by the deploying operator (KServe, RHOAI Operator)
- kube-rbac-proxy sidecar enforces Kubernetes RBAC via SubjectAccessReview

Authentication Summary:
- REST API: Optional API Key via Authorization header (--api_key_file / API_KEY env)
- gRPC API: No built-in auth (InsecureServerCredentials)
- Health endpoints (/v2/health/live, /v2/health/ready): Auth bypassed
- Platform-level: OAuth/OIDC via RHOAI Gateway + kube-rbac-proxy enforcement

TLS Summary:
- OVMS does NOT implement TLS termination
- gRPC server uses InsecureServerCredentials
- HTTP server binds without TLS
- TLS is terminated by RHOAI Gateway (Envoy) at the ingress layer
- System trust store: /etc/ssl/certs/ca-bundle.crt → ca-certificates.crt
- OpenSSL: Provided by UBI 9 base image (dynamic linking)

FIPS Compliance:
- C++ binary (not Go — check-payload Go-specific checks N/A)
- Dynamically links against system OpenSSL via UBI 9
- MD5 used only for non-cryptographic filesystem cache checksums
- BoringSSL referenced only for Windows gRPC builds (not Linux)
