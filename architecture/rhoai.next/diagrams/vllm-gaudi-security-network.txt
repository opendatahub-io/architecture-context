RBAC Summary:
- No RBAC resources defined in this repository (container image, not operator)
- RBAC is managed by KServe and the RHOAI operator
- kube-rbac-proxy sidecar enforces authorization at the pod level

Authentication & Authorization:
- /v1/* endpoints: Bearer Token (JWT) or None, enforced by kube-rbac-proxy sidecar (RHOAI 3.x) or oauth-proxy (RHOAI 2.x)
- /health, /metrics: Unauthenticated (direct access, no sidecar interception)

Secrets:
- Model storage credentials (Opaque) — S3/MinIO access for model weight download, provisioned by User/Dashboard, no auto-rotation
- HuggingFace token (Opaque) — HuggingFace Hub auth for gated models, provisioned by User, no auto-rotation

FIPS Compliance:
- WARNING: openssl-fips-provider-so is explicitly REMOVED in Dockerfile (line 63)
- Container does NOT run in FIPS-compliant mode
- Likely intentional due to Habana SynapseAI runtime compatibility
- Deployments requiring FIPS compliance should be aware of this gap

Build Hermeticity:
- Builds are NON-HERMETIC (hermetic: false in Tekton PipelineRun)
- No lock files present (no rpms.lock.yaml, no pip lock file, no artifacts.lock.yaml)
- Dockerfile clones from GitHub and installs from PyPI/vault.habana.ai at build time

Security Observations:
1. HCCL inter-worker communication is plaintext with no authentication
2. NIXL side channel and RDMA transfers are plaintext with no authentication
3. Ray distributed orchestration has no encryption or authentication
4. All intra-cluster communication relies on network policy for isolation
5. FIPS provider explicitly removed — non-FIPS crypto in use
