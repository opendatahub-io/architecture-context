RBAC Summary:
- No Kubernetes RBAC defined — ModelMesh is a sidecar service that does not
  interact with the Kubernetes API
- RBAC is managed by the modelmesh-serving controller

Secrets:
- model-mesh-tls-secret (kubernetes.io/tls)
  - Purpose: TLS certificate and private key for external gRPC + inter-instance
  - Provisioned by: modelmesh-serving controller / manual
  - Auto-Rotate: No

- model-mesh-icd-etcd (Opaque)
  - Purpose: etcd connection configuration (JSON-format connection string)
  - Provisioned by: modelmesh-serving controller / manual
  - Auto-Rotate: No

- (trust certs) (Opaque)
  - Purpose: CA certificates for mTLS client authentication
  - Provisioned by: Manual
  - Auto-Rotate: No

Authentication & Authorization:
- gRPC external (8033):
  - Auth: Optional mTLS (client certificates)
  - Enforcement: Netty/gRPC TLS handler
  - Policy: ClientAuth.NONE / REQUIRE / OPTIONAL (MM_TLS_CLIENT_AUTH)

- gRPC inter-instance (8033):
  - Auth: TLS with shared certificates
  - Enforcement: litelinks/Netty
  - Policy: Mutual auth between pods using same TLS certs

- gRPC runtime (8085/UDS):
  - Auth: None
  - Enforcement: N/A (localhost only)
  - Policy: Pod-internal, no auth required

- Metrics (2112):
  - Auth: None
  - Encryption: Self-signed TLS (BouncyCastle)
  - Note: Self-signed cert is NOT FIPS-validated

- Health probes (8089):
  - Auth: None
  - Encryption: Plaintext HTTP
  - Policy: No auth

FIPS Compliance Notes:
- Konflux Dockerfile configures SunPKCS11 with NSS (FIPS-capable)
- Non-Konflux Dockerfile explicitly DISABLES FIPS (security.useSystemPropertiesFile=false)
- Metrics endpoint self-signed cert (BouncyCastle) operates OUTSIDE FIPS boundary
- Netty OpenSSL preferred over JDK crypto (FIPS-capable when host in FIPS mode)
