RBAC Summary:
- No RBAC roles defined — ModelMesh runs as a sidecar and does not directly
  interact with the Kubernetes API. RBAC is managed by the modelmesh-serving
  controller that deploys it.
- Kubernetes Downward API used for: Pod name (WKUBE_POD_NAME),
  Pod IP (WKUBE_POD_IPADDR), Host IP (MM_LOCATION)

Network Policy:
- NetworkPolicy defined in config/base/networkpolicy.yaml
- Restricts ingress to expected ports (8033, 8080, 8089, 8090, 2112)

Secrets:
- model-mesh-tls-secret (kubernetes.io/tls)
  - TLS cert + private key for external gRPC and internal Litelinks
  - Provisioned: Manual or cert-manager
  - Auto-Rotate: No

- model-mesh-icd-etcd (Opaque)
  - etcd connection config (JSON: endpoint, credentials, TLS certs)
  - Provisioned: Manual
  - Auto-Rotate: No

TLS Configuration:
- External gRPC (8033): Netty SslContext with BoringSSL-static
  - ClientAuth modes: NONE, REQUIRE, OPTIONAL (via MM_TLS_CLIENT_AUTH)
- Internal Litelinks (8080): Separate TLS key/cert path
  - ClientAuth: Optional (via MM_INTRACLUSTER_CLIENTAUTH)
- Prometheus metrics (2112): Self-signed cert (BouncyCastle generated)
- Model runtime sidecar (8085/UDS): Plaintext (localhost isolation)

FIPS Compliance:
- Java FIPS mode EXPLICITLY DISABLED: -Dcom.redhat.fips=false
- Reason: Incompatibility with TLS when NSS/PKCS11 provider reconfigured
- TLS uses BoringSSL-static (NOT FIPS-validated) via Netty tcnative
- Self-signed certs generated via BouncyCastle (NOT FIPS-validated)
- SIGNIFICANT COMPLIANCE GAP for FIPS-enabled OpenShift clusters

Authentication Summary:
- External gRPC (8033): mTLS optional (MM_TLS_CLIENT_AUTH=REQUIRE for enforcement)
- Inter-pod Litelinks (8080): mTLS optional (MM_ENABLE_SSL + MM_INTRACLUSTER_CLIENTAUTH)
- Model Runtime (8085/UDS): None (localhost/UDS isolation, NetworkPolicy enforced)
- Prometheus (2112): None (HTTPS with self-signed cert only)
- Health probes (8089): None (HTTP)
- PreStop hook (8090): None (HTTP)
