RBAC Summary:
- ModelMesh itself does NOT define RBAC resources
- Runs as a sidecar with no direct Kubernetes API access
- RBAC is managed entirely by the modelmesh-serving controller
- ServiceAccount: inherited from pod (set by modelmesh-serving controller)

Secrets:
- model-mesh-tls-secret (kubernetes.io/tls)
  - Purpose: TLS cert/key for external gRPC (8033) and internal Litelinks (8080)
  - Provisioned by: modelmesh-serving controller / manual
  - Auto-Rotate: No
- model-mesh-icd-etcd (Opaque, JSON)
  - Purpose: etcd connection configuration (endpoints, credentials, TLS)
  - Provisioned by: Platform operator / manual
  - Auto-Rotate: No

FIPS Compliance Notes:
- Java FIPS EXPLICITLY DISABLED: -Dcom.redhat.fips=false in start.sh
- SunPKCS11 security provider reordered in java.security
- BouncyCastle 1.78 used for Prometheus self-signed cert generation
- Netty tcnative-boringssl-static used for native TLS offload
- BoringSSL may NOT use FIPS-validated cryptographic modules
- RISK: Non-FIPS compliant in environments requiring strict FIPS validation

Network Policy:
- NetworkPolicy defined in config/base/networkpolicy.yaml
- Headless Service (clusterIP: None) for direct pod addressing

Service Configuration:
- Service Name: model-mesh
- Type: Headless ClusterIP (clusterIP: None)
- Port: 8033/TCP → grpc (8033)
- Purpose: Direct pod addressing for Litelinks service discovery
