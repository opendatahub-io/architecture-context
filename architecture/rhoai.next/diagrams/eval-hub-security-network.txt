Authentication & Authorization:
- All /api/v1/evaluations/* endpoints: kube-rbac-proxy enforces SubjectAccessReview
  - Inbound: Bearer Token (Kubernetes SA token) over TLS 1.2+
  - Injects X-Tenant and X-User headers to eval-hub API
- Health, OpenAPI, Docs endpoints: Unauthenticated
- Metrics (8081/TCP): Unauthenticated, cluster-internal only (no Route)
- MCP endpoints: Same kube-rbac-proxy pattern on separate deployment

Secrets:
- model-auth (Opaque, per Job) - Real model credentials (api-key, hf-token, ca_cert); user-provisioned, no auto-rotate
- internal-model-ref (Opaque, per Job) - Ephemeral ref tokens; auto-created by EvalHub, auto-deleted on Job completion
- test-data (Opaque, per Job) - S3 credentials (AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY); user-provisioned
- oci-credentials (kubernetes.io/dockerconfigjson, per Job) - OCI registry push credentials; user-provisioned
- mlflow-token (Opaque) - MLflow auth token; platform-provisioned, SA token rotation
- service-ca.crt (ConfigMap, projected) - OpenShift service CA cert; auto-rotated by service-ca-operator

Credential Injection Pattern:
- Real model credentials are NEVER mounted in the adapter container
- EvalHub creates ephemeral "internal-model-ref" secrets with ref tokens (e.g., "api-key:ref")
- Adapter sends requests with ref tokens to the sidecar proxy (pod-local HTTP/8080)
- Sidecar detects ref tokens, substitutes real credentials from its own Secret mount
- This isolates untrusted evaluation code from production credentials

FIPS Compliance:
- Build: GOEXPERIMENT=strictfipsruntime, CGO_ENABLED=1 (Konflux build only)
- TLS: MinVersion TLS 1.2, MaxVersion TLS 1.3, FIPS-safe cipher suites via OpenSSL
- No non-FIPS crypto detected (no MD5, RC4, DES usage)
- InsecureSkipVerify available but defaults to false

CRD:
- EvalHub (trustyai.opendatahub.io/v1alpha1, Namespaced) - Managed by TrustyAI Service Operator
- HardwareProfile (infrastructure.opendatahub.io/v1, read-only) - Resolves GPU/resource requirements
