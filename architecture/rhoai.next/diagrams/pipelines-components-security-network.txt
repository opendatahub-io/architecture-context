RBAC Summary:
- No RBAC manifests defined by pipelines-components
- Components rely on KFP execution ServiceAccount permissions managed by DSPO
- Deployment components use in-cluster Kubernetes client with pod's mounted SA token
- Required permissions (inherited from KFP SA):
  - serving.kserve.io: inferenceservices, servingruntimes (create, get, delete)
  - ray.io: rayjobs (create, get, delete)
  - core: secrets (get), namespaces (get)
  - hardwareprofile.opendatahub.io: hardwareprofiles (get, list)

Secrets Summary:
- s3-secret (Opaque) - AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_S3_ENDPOINT, AWS_DEFAULT_REGION | User/Admin provisioned | No auto-rotate
- hf-token (Opaque) - HuggingFace Hub token for gated model downloads | User provisioned | No auto-rotate
- kubernetes-credentials (Opaque) - K8s API server URL + auth token (out-of-cluster) | User/Admin provisioned | No auto-rotate
- oci-pull-secret-model-download (Opaque) - OCI image pull credentials | User provisioned | No auto-rotate
- minio-secret (Opaque) - MinIO access_key + secret_key | User/Admin provisioned | No auto-rotate
- ogx-secret (Opaque) - OGX API key + base URL | User/Admin provisioned | No auto-rotate
- llm-credentials (Opaque) - LLM API key for SDG | User provisioned | No auto-rotate

⚠ None of the 7 secrets have auto-rotation configured.

Security Risks:
1. verify=False SSL fallback in S3 and OGX clients — disables certificate validation (MITM risk)
2. Model Registry uses plaintext HTTP (port 80) — data-in-transit risk for model metadata
3. Eval Hub verify_tls defaults to False — certificate validation bypassed
4. No RBAC manifests — relies entirely on KFP SA permissions (shared privilege scope)
5. All 7 secrets lack auto-rotation — long-lived credentials increase compromise window

FIPS Notes:
- Python components inherit FIPS from underlying RHEL/UBI9 OpenSSL
- cryptography==49.0.0 delegates to system OpenSSL (FIPS-capable)
- verify=False pattern weakens certificate chain validation regardless of FIPS mode
