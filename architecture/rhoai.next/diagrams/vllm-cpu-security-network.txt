RBAC Summary:
- No RBAC — vLLM CPU is a containerized service, not a Kubernetes operator.
- RBAC is managed by KServe, which deploys vLLM as an InferenceService runtime.

Secrets:
- SSL certificate/key (kubernetes.io/tls) - TLS termination, cert-manager or manual, auto-rotate via --enable-ssl-refresh
- API key (Opaque) - Bearer token auth for /v1/* paths (VLLM_API_KEY env or --api-key), no auto-rotation
- HuggingFace token (Opaque) - Model repo access (HF_TOKEN env), no auto-rotation

Authentication & Authorization:
- /v1/* endpoints: Bearer Token (SHA256-hashed, constant-time comparison)
- Health/operational endpoints: Unauthenticated (open access)
- CORS: Defaults to allow all origins/methods/headers (should be restricted in production)
- OPTIONS preflight: Exempted from auth

FIPS Crypto Considerations:
- Base image: UBI 9 (system OpenSSL for FIPS)
- TLS: Python ssl module delegates to system OpenSSL
- Config hashing: MD5 default with SHA256 FIPS fallback (safe_hash())
- Multimodal hashing: blake3 default, set VLLM_MM_HASHER_ALGORITHM=sha256 for FIPS
- OTLP exporter: Insecure gRPC channel by default (configure TLS for FIPS)

Build Hermeticity:
- hermetic: false (explicitly disabled in Tekton pipeline)
- No lock files (requirements/*.txt specify version ranges)
- Dependencies resolved at build time from PyPI, IBM DevPI, RPM repos
