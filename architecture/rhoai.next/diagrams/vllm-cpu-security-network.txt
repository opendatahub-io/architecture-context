RBAC Summary:
  N/A — vLLM CPU is a user-space application container, not an operator.
  It does not define Kubernetes RBAC resources (ClusterRoles, RoleBindings).
  Pod-level RBAC is managed by the deploying operator (KServe / RHOAI).

Authentication & Authorization:
  - /v1/* endpoints: Bearer API Key (optional, via --api-key or VLLM_API_KEY)
    Enforcement: FastAPI middleware rejects if key configured and request key mismatches
  - /health, /metrics, /ping: No auth (required for K8s probes and Prometheus)
  - /v1/realtime (WebSocket): API Key validated during WebSocket handshake
  - Platform level: kube-rbac-proxy sidecar (port 8443) enforces RBAC before
    proxying to vLLM (port 8000) when deployed by RHOAI

Secrets:
  - HF_TOKEN (Opaque env var) - HuggingFace Hub authentication for gated models
    Provisioned by: User/platform operator | Auto-rotate: No
  - VLLM_API_KEY (Opaque env var) - API key(s) for client request authentication
    Provisioned by: User/platform operator | Auto-rotate: No
  - SSL key/cert files (kubernetes.io/tls) - TLS termination for HTTPS serving
    Provisioned by: cert-manager or manual | Auto-rotate: Yes (--enable-ssl-refresh)

FIPS Compliance:
  - Python application — uses system OpenSSL from UBI 9 base image
  - TLS: Python stdlib ssl.SSLContext (FIPS via system OpenSSL on UBI 9)
  - Crypto: hashlib uses OpenSSL backend (FIPS-compliant on UBI 9)
  - MD5 usage: 3 locations, non-security (usedforsecurity=False on 2/3)
    ⚠ hf3fs_connector.py:1012 uses hashlib.md5 WITHOUT usedforsecurity=False

Service Mesh Configuration:
  - N/A — vLLM does not configure service mesh resources directly
  - Platform-level service mesh (if deployed) provides mTLS for cluster-internal traffic
  - gRPC port (50051) relies on network policies or service mesh for transport security
