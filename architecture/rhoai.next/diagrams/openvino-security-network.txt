RBAC Summary:
  Not applicable — OpenVINO is a native C++ library, not a Kubernetes workload.
  RBAC is managed by OVMS and the RHOAI platform operator.

Service Mesh Configuration:
  Not applicable — No network services exposed by OpenVINO.
  Service mesh configuration is handled at the OVMS pod level.

Secrets:
  Not applicable — OpenVINO does not manage secrets.
  Model storage credentials (S3, etc.) are managed by OVMS.

Cryptographic Operations:
  None — OpenVINO is a pure compute library that processes tensor data.
  No TLS, no encryption, no certificate handling.
  FIPS compliance is enforced at the OVMS container image level.

Security Considerations:
  1. DESERIALIZATION ATTACK SURFACE: Frontends parse untrusted model files using
     protobuf (3.21.12), flatbuffers, and pugixml. Malformed models could
     potentially exploit parsing vulnerabilities.
  2. DYNAMIC LIBRARY LOADING: Device plugins are loaded as .so files. Plugin
     paths must be controlled to prevent loading malicious shared libraries.
  3. JIT CODE GENERATION: CPU plugin uses xbyak for runtime code generation
     (JIT). This requires executable memory pages (W^X implications).
  4. SUPPLY CHAIN: 14 git submodules pinned by commit SHA. Conan.lock provides
     reproducible C++ dependency resolution. No rpms.lock.yaml on master branch.
