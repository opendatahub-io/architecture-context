Secrets:
- TLS server key (kubernetes.io/tls) - gRPC/HTTP server TLS termination, provisioned by platform operator / cert-manager, no auto-rotate
- TLS server cert (kubernetes.io/tls) - gRPC/HTTP server TLS certificate, provisioned by platform operator / cert-manager, no auto-rotate
- TLS client CA cert (kubernetes.io/tls) - Client certificate verification for mTLS, provisioned by platform operator, no auto-rotate
- S3 credentials (Opaque) - accessKey, secretKey, IAM_id, IAM_api_key for data source access, provisioned by user / platform, no auto-rotate
- OTEL TLS certs (Opaque) - CA, client key, client cert for trace exporter mTLS, provisioned by platform operator, no auto-rotate

Authentication & Authorization:
- gRPC Server: Mutual TLS (configurable) via grpc.ssl_server_credentials with require_client_auth - client cert required when CA configured
- HTTP Server: Mutual TLS (configurable) via Python ssl.SSLContext with CERT_REQUIRED - client cert required when CA configured
- /health endpoint: No auth - always accessible for Kubernetes probes
- ModelMesh Unix Socket: No auth - pod-local only, access restricted by pod network namespace
- Auth is DELEGATED to the platform (KServe/Istio/Service Mesh) - Caikit does not enforce user-level authorization

RBAC:
- No Kubernetes RBAC defined by Caikit
- RBAC is managed by the platform operator and KServe
- Caikit runs as a Python process inside a container; it does not interact with the Kubernetes API

Network Policy:
- No NetworkPolicy resources defined by Caikit
- Network isolation managed by platform (KServe, Service Mesh, OpenShift SDN)
