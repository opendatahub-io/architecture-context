SECURITY CONTROLS:

1. Trusted Runtimes Allowlist (Application-Level)
   - File: /etc/mlserver/trusted-runtimes.json (baked into image at build time)
   - Mode: PRODUCTION (when file exists)
   - Enforcement: Rejects model load if implementation not in allowlist
   - Validation points:
     a. ModelSettings Pydantic model_validator
     b. implementation property accessor (defense-in-depth)
     c. Startup security check (log_runtime_security_mode)
   - Blocks: Untrusted model implementations, custom environments,
             environment_tarball, environment_path (sys.path injection)
   - Allowed runtimes: mlserver_sklearn.SKLearnModel, mlserver_xgboost.XGBoostModel,
                       mlserver_lightgbm.LightGBMModel, mlserver_onnx.ONNXModel (RHOAI default)

2. CORS Validation (Application-Level)
   - Wildcard CORS origins blocked in production mode
   - Regex-based CORS origins blocked in production mode
   - Only explicit, static CORS origins allowed

3. Import Path Validation
   - Strict regex: requires explicit dotted module paths with capitalized class names
   - Prevents arbitrary code execution through crafted import paths

RBAC Summary:
- No Kubernetes RBAC defined (MLServer is a service, not an operator)
- Pod RBAC managed by KServe and rhods-operator
- Reads namespace from /var/run/secrets/kubernetes.io/serviceaccount/namespace
  (read-only, for CloudEvents headers only)

Secrets:
- No Kubernetes Secrets consumed directly by MLServer
- TLS certificates managed at KServe/platform ingress layer
- Model storage credentials (S3) managed by KServe storage initializer

FIPS Compliance:
- Base image: quay.io/aipcc/base-images/cpu (UBI-based, FIPS-capable OpenSSL)
- Crypto usage: hashlib.sha256 only (stdlib, delegates to OpenSSL on UBI)
- TLS: Not managed by MLServer (all ports plaintext)
- TLS termination: Platform layer (KServe + Istio/Gateway API)
- Risk: None - no non-FIPS crypto detected

Network Trust Model:
- MLServer trusts pod-internal network (all ports plaintext)
- TLS termination at KServe/Istio ingress layer
- Authentication at KServe ingress (kube-rbac-proxy or oauth-proxy sidecar)
- Standard sidecar-proxied workload trust model
