RBAC Summary:
- MLServer does NOT define Kubernetes RBAC roles or bindings
- Runs as non-root user (UID 1000) within the container
- No ServiceAccount-level RBAC permissions required
- All Kubernetes API interactions are handled by KServe controller

Runtime Security Summary:
- PRODUCTION mode: /etc/mlserver/trusted-runtimes.json enforces runtime allowlist
- Allowed runtimes: LightGBM, ONNX Runtime, scikit-learn, XGBoost
- Blocks: Arbitrary Python code execution via custom runtime import paths
- Blocks: Wildcard CORS origins (PRODUCTION mode)
- Blocks: Custom Python environments (PRODUCTION mode)
- Endpoint /v2/runtimes exposes current security mode and allowlist

Authentication & Authorization:
- All REST/gRPC endpoints: No application-level auth
- Auth enforcement: External (KServe/Istio/kube-rbac-proxy at platform ingress)
- /v2/runtimes: Trusted Runtimes Allowlist enforcement (PRODUCTION mode)
- No Kubernetes Secrets consumed directly (credentials managed by KServe)

Network Exposure:
- Port 8080/TCP (HTTP REST): Internal only (ClusterIP via KServe)
- Port 8081/TCP (gRPC): Internal only (ClusterIP via KServe)
- Port 8082/TCP (HTTP Metrics): Internal only (ClusterIP via KServe)
- All ports bind to 0.0.0.0 within pod
- External exposure managed entirely by KServe (Istio VirtualService / Gateway API)
