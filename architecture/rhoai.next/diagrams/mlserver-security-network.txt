RBAC Summary:
  MLServer does not define Kubernetes RBAC resources.
  RBAC is managed by the deploying platform (KServe / ModelMesh).
  - No ClusterRoles defined
  - No RoleBindings defined
  - No ServiceAccounts created

Application-Level Security:
  - Trusted Runtimes Allowlist: /etc/mlserver/trusted-runtimes.json
    - Only pre-approved Python import paths can be instantiated
    - Validates import paths syntactically (dotted, uppercase class name, no leading underscores)
    - Blocks custom environment_tarball and environment_path in production mode
    - RHOAI allowlisted runtimes: sklearn, xgboost, lightgbm, onnx
  - CORS: Production mode blocks wildcard origins; explicit origins required
  - No built-in authentication or authorization
  - No TLS termination (delegated to platform)

Secrets:
  MLServer does not manage Kubernetes secrets directly.
  TLS termination and authentication are handled by the platform ingress layer.

FIPS Compliance:
  - Python service on UBI9 base image
  - OpenSSL provided by UBI9 (FIPS-validated when host RHEL has FIPS enabled)
  - No explicit FIPS opt-in required for Python services on UBI9
  - Risk: OTel OTLP exporter uses insecure=True (plaintext gRPC to collector)
  - gRPC server uses add_insecure_port() (no TLS)

Build Hermeticity:
  - poetry.lock: Present (841 KB, locks upstream versions)
  - requirements-cpu.txt: SHA256 hash-pinned from Red Hat internal PyPI
  - Konflux Dockerfile uses pre-built RHOAI base image with pre-vetted packages
  - rpms.lock.yaml: Not present on main branch
  - artifacts.lock.yaml: Not present
