RBAC Summary:
- No Kubernetes RBAC defined (MLServer is a service workload, not an operator)
- RBAC for the pod is managed by KServe and the platform operator (rhods-operator)
- MLServer reads namespace from /var/run/secrets/kubernetes.io/serviceaccount/namespace
  for CloudEvents source headers only (no API calls)

Application-Level Security Controls:
- Trusted Runtimes Allowlist (/etc/mlserver/trusted-runtimes.json):
  - Only model implementations in the allowlist can be loaded in PRODUCTION mode
  - Validated at: ModelSettings Pydantic validator, implementation property, startup check
  - Import path regex: requires dotted module paths with capitalized class names
  - Example allowed: mlserver_sklearn.SKLearnModel, mlserver_xgboost.XGBoostModel
- Environment Blocking (production mode):
  - environment_tarball parameter: REJECTED (prevents sys.path injection)
  - environment_path parameter: REJECTED (prevents sys.path injection)
- CORS Restrictions (production mode):
  - Wildcard ("*") origins: BLOCKED
  - Regex-based origins: BLOCKED
  - Only explicit origin strings allowed

Secrets:
- No Kubernetes Secrets consumed directly
- TLS certificates: None (handled at KServe/platform ingress layer)
- Model storage credentials: Managed by KServe (volume provisioning)

FIPS Compliance:
- Base image: quay.io/aipcc/base-images/cpu (UBI-based, FIPS-capable OpenSSL)
- Crypto usage: hashlib.sha256 only (delegates to OpenSSL on FIPS-enabled systems)
- No TLS configured in application (all plaintext, relies on platform TLS)
- No non-FIPS crypto risks detected

Network Trust Model:
- MLServer assumes pod-internal network is trusted (no TLS between containers)
- All external-facing TLS is terminated at KServe ingress (Istio/Gateway API)
- Service mesh mTLS (if enabled) protects pod-to-pod traffic transparently
- OTLP trace export uses insecure gRPC (assumes cluster-internal collector)
