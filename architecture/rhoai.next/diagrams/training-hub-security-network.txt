RBAC Summary:
  N/A -- training-hub is a Python library and does not define Kubernetes RBAC resources.
  RBAC is determined by the container/pod ServiceAccount in which the library executes.

Service Mesh Configuration:
  N/A -- training-hub does not configure or require Istio/ServiceMesh.
  Distributed training (torchrun/NCCL) uses raw TCP without mTLS.

Secrets / Credentials:
  No Kubernetes Secrets defined. Credentials managed via environment variables:
  - HF_TOKEN (env var) - HuggingFace Hub access for gated model downloads
  - WANDB_API_KEY (env var) - WandB experiment tracking authentication
  - MLFLOW_TRACKING_URI + token (env var) - MLflow tracking server access
  Note: All credentials are passed by the calling environment, not managed by the library.

Supply Chain Risk:
  - litellm pinned to >=1.82.0,<1.82.7 due to TeamPCP supply chain attack
    (versions 1.82.7-1.82.8 compromised on 2026-03-24)
  - Documented in pyproject.toml line 76

Encryption Gaps:
  - torchrun inter-node communication: No encryption (TCP/NCCL)
  - torchrun rendezvous endpoint: No encryption
  - vLLM inference (ART backend): HTTP on localhost (acceptable - same process)
  - verl distributed workers: No encryption (TCP)
  Note: Distributed training typically runs within a trusted cluster network.
        If cross-node training data contains sensitive information, network-level
        encryption (e.g., WireGuard, IPsec) should be applied at the infrastructure layer.

Application-Level Crypto:
  - No direct TLS configuration; delegates to urllib3/requests defaults
  - No crypto library usage; relies on Python stdlib + system OpenSSL
  - Certificate handling via system trust store (requests/urllib3)
  - No non-FIPS crypto risks detected in training-hub source
