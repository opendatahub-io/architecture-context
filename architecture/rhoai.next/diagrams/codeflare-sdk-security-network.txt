RBAC Summary (User-Required Permissions):
- The SDK does NOT define its own RBAC resources
- It operates under the authenticated user's/service account's permissions
- Required API permissions:
  - ray.io / rayclusters: get, list, create, delete
  - workload.codeflare.dev / appwrappers: get, list, create, delete
  - kueue.x-k8s.io / localqueues: get, list
  - route.openshift.io / routes: get, list
  - "" (core) / secrets: get, list
  - networking.k8s.io / ingresses: get, list

Secrets:
- {cluster-name}-ca-secret-* (Opaque: ca.crt, ca.key) - CA cert/key for Ray mTLS
  - Provisioned by: KubeRay Operator / RHOAI
  - Auto-Rotate: No
- Local TLS files (tls-{cluster}-{ns}/) - Client cert/key for Ray connection
  - Provisioned by: SDK generate_cert module (cryptography library)
  - Auto-Rotate: No

Authentication Model (3 Tiers):
1. Kubernetes Cluster Auth: Bearer Token (OAuth) or kubeconfig
2. Ray Dashboard Auth: Bearer Token forwarded from K8s context (OpenShift only)
3. Ray Client Auth: Optional mTLS with SDK-generated client certificates

CA Certificate Path Discovery (3-tier fallback):
1. Explicit parameter
2. CF_SDK_CA_CERT_PATH environment variable
3. RHOAI workbench default: /etc/pki/tls/custom-certs/ca-bundle.crt

Trust Boundaries:
- User Workbench → K8s API: TLS 1.2+, Bearer Token (trusted)
- User Workbench → Ray Dashboard: TLS via OpenShift Route (trusted)
- User Workbench → Ray Client: Optional mTLS (configurable trust)
- Ray Dashboard (non-OpenShift): No auth, direct access (UNTRUSTED)
- Ray internal (GCS 6379): No encryption, cluster-internal only
