AUTHENTICATION SUMMARY:
- Server authentication: Bearer Token validated via three strategies:
  1. JWT/JWKS: Periodic key refresh from JWKS endpoint, local validation
  2. Token introspection: RFC 7662, client credentials to OAuth2 provider
  3. Custom endpoint: Forward token to external validation service
- Authorization: ABAC with Cedar-like policy rules
  - Attributes: Extracted from JWT claims (roles, teams, projects)
  - Rules: permit/forbid with when/unless conditions
  - Scope: Per-resource (models, datasets, vector-dbs, etc.)
- Public endpoints (no auth): /v1/health, /v1/version, /docs, /openapi.json
- Rate limiting: 100 req/day anonymous, 1000 req/day authenticated (configurable)

SECRETS:
- TLS key/cert pair (tls_keyfile, tls_certfile) - Server HTTPS, manual provision
- TLS CA certificate (tls_cafile) - mTLS client cert validation, manual
- Provider API keys (env vars) - OpenAI, Anthropic, etc., manual
- OAuth2 introspection client_secret (config) - Token introspection, manual
- PostgreSQL credentials (config) - pgvector/KV/SQL store, manual
- Redis password (config) - Redis auth, manual
- HuggingFace token (env var) - Model downloads, manual
- No auto-rotation configured for any secrets

FIPS COMPLIANCE:
- No Konflux Dockerfiles in repo (build infra is downstream)
- Python ssl module inherits FIPS mode from OS (RHEL/UBI base)
- hashlib.md5 used in 3 locations with usedforsecurity=False (PEP 452 compliant)
- Crypto libraries: python-jose, ecdsa, rsa (use OpenSSL backend)
- TLS config: ssl.create_default_context() for server and auth connections
- Actual FIPS posture depends on runtime base image's OpenSSL configuration
