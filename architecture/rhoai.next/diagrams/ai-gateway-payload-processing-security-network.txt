RBAC Summary:
- ClusterRole: {release}-externalmodels-reader
  - API Groups: "" (core), maas.opendatahub.io
  - Resources: secrets, externalmodels
  - Verbs: get, list, watch
- ClusterRoleBinding: {release}-externalmodels-reader
  - ServiceAccount: payload-processing-{release}
  - Scope: Cluster-wide (when bbr.multiNamespace=true) or namespace-scoped

CRDs Defined:
- inference.opendatahub.io/v1alpha1 ExternalModel (Namespaced)
  - Maps client-facing model name → external provider(s)
  - spec.externalProviderRefs: MaxItems=1 (Phase 1; Phase 2 enables traffic splitting)
- inference.opendatahub.io/v1alpha1 ExternalProvider (Namespaced)
  - Defines provider connection: endpoint FQDN, provider type, auth secret ref

Secrets:
- Provider API key Secrets (Opaque)
  - Label: inference.networking.k8s.io/bbr-managed=true
  - Data field: api-key
  - Provisioned by: User or MaaS controller
  - Auto-rotate: No
  - Usage: Cached in-memory by apikey-injection plugin

FIPS Compliance:
- Build: GOEXPERIMENT=strictfipsruntime, CGO_ENABLED=1
- Base: registry.access.redhat.com/ubi9/ubi-minimal (OpenSSL included)
- Crypto: stdlib crypto/tls only (OpenSSL backend via strictfipsruntime)
- No custom TLS config; NeMo HTTP calls use plaintext
- External provider TLS delegated to Envoy proxy

Security Observations:
⚠️  NeMo Guardrails communication uses plaintext HTTP with no authentication
    - Acceptable for intra-cluster with network policies
    - Risk: MITM within cluster if network isolation is not enforced
⚠️  gRPC ExtProc channel (9004/TCP) is plaintext
    - Acceptable: pod-local communication with Envoy
    - EnvoyFilter ensures only co-located Envoy connects
⚠️  Provider API keys stored in Kubernetes Secrets (no auto-rotation)
    - Manual rotation required by user/operator
