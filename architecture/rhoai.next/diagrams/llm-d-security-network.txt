SECURITY OBSERVATIONS:
======================

1. NO TLS within cluster:
   - Model server API (8000/TCP) is plaintext HTTP with no authentication
   - NIXL KV transfer (5600/TCP) is plaintext gRPC with no authentication
   - Mooncake gRPC services (50051, 50052) are plaintext with no authentication
   - TLS termination is delegated entirely to the Gateway/proxy layer

2. NO authentication at model server layer:
   - All /v1/* endpoints are unauthenticated
   - Only protection is network isolation (namespace scoping)
   - Gateway provider may add auth, but it's not enforced by default

3. RDMA data plane (Mooncake):
   - Direct memory access between nodes via RDMA
   - No encryption or authentication on RDMA transfers
   - Requires rdma/ib device resources on nodes

Service Mesh / Network Policy:
==============================

- No PeerAuthentication configured (no mTLS enforcement)
- No NetworkPolicies defined in manifests
- No AuthorizationPolicies defined in manifests
- Namespace isolation is the primary security boundary
- ServiceAccount has automountServiceAccountToken=false (model server)

FIPS Compliance:
================

- NOT FIPS compliant at application level
- No GOEXPERIMENT, strictfipsruntime, or CGO_ENABLED flags
- vLLM runs plaintext HTTP (no TLS at server level)
- UBI9 base (CUDA) includes OpenSSL but not configured for FIPS mode
- FIPS compliance requires Gateway/proxy layer to enforce TLS + FIPS cipher suites

Build Hermeticity:
==================

- No rpms.lock.yaml, go.sum, uv.lock, or artifacts.lock.yaml
- No Hermeto/cachi2 prefetch configuration
- Dependencies fetched at build time from public registries (non-hermetic)
- Version constraints via docker/constraints.txt (partial pinning only)
