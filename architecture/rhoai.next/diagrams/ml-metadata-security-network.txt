INGRESS:
  - No Kubernetes Ingress resources defined
  - Service exposed as ClusterIP (internal only)
  - External access managed by consuming deployment (e.g., kube-rbac-proxy)

RBAC Summary:
  - No Kubernetes RBAC resources (standalone gRPC service)
  - No ClusterRoles, Roles, or RoleBindings
  - No per-RPC authorization mechanism
  - Access control relies on:
    1. Network isolation (ClusterIP, no external exposure)
    2. Optional mTLS (client certificate verification via custom_ca)
    3. Upstream proxy (kube-rbac-proxy in RHOAI deployments)

Secrets:
  - DB credentials (mysql_config_password / postgres_config_password)
    Type: Opaque, Provisioned: Manual/platform operator, Auto-Rotate: No
  - SSL server key (ssl_config.server_key)
    Type: TLS key, Provisioned: Manual/cert-manager, Auto-Rotate: No
  - SSL server cert (ssl_config.server_cert)
    Type: TLS cert, Provisioned: Manual/cert-manager, Auto-Rotate: No
  - SSL custom CA (ssl_config.custom_ca)
    Type: CA cert, Provisioned: Manual, Auto-Rotate: No
  - MySQL SSL certs (sslcert, sslkey, sslrootcert)
    Type: TLS, Provisioned: Manual, Auto-Rotate: No
  - PostgreSQL SSL certs (sslcert, sslkey, sslrootcert)
    Type: TLS, Provisioned: Manual, Auto-Rotate: No

FIPS Compliance Notes:
  - Crypto library: BoringSSL (via gRPC/Bazel), NOT system OpenSSL
  - BoringSSL is FIPS-capable but FIPS mode NOT explicitly enabled in build
  - Binary is statically linked against BoringSSL, not dynamically against system OpenSSL
  - No GOEXPERIMENT or strictfipsruntime flags (C++ binary, not Go)
  - UBI9 base image provides system OpenSSL but server binary uses BoringSSL

Security Considerations:
  - No built-in per-RPC authorization — any client with network access can
    perform any operation including bulk deletes and schema modifications
  - New database connection created per RPC (no connection pooling)
  - Default mode is InsecureServerCredentials (no TLS) when ssl_config not provided
  - Relies on platform-level network policies and proxy for access control
