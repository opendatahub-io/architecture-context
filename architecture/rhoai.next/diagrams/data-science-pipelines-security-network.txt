  SERVICE MESH CONFIGURATION (Multi-user mode)
================================================================================

  - DestinationRule: mTLS ISTIO_MUTUAL for all DSP services
  - AuthorizationPolicy: Restricts access based on service account identity
  - AuthorizationPolicy: Header-based checks (kubeflow-userid)
  - VirtualService: Routes through kubeflow-gateway

================================================================================
  SECRETS
================================================================================

  - mysql-secret (Opaque)
    - Purpose: Database username/password for MySQL connection
    - Provisioned By: Manual / DSPA operator
    - Auto-Rotate: No

  - mlpipeline-minio-artifact (Opaque)
    - Purpose: S3/MinIO access key and secret key
    - Provisioned By: Manual / DSPA operator
    - Auto-Rotate: No

  - webhook-server-tls (kubernetes.io/tls)
    - Purpose: TLS certificate for cache mutation webhook
    - Provisioned By: Cache deployer (CSR) or cert-manager
    - Auto-Rotate: Yes (via cache deployer)

================================================================================
  POD SECURITY
================================================================================

  All pods:
  - runAsNonRoot: true
  - runAsUser: 1000 (removed in OpenShift overlay, SCC manages)
  - capabilities: drop ALL
  - seccompProfile: RuntimeDefault
  - FIPS: CGO enabled, GOEXPERIMENT=strictfipsruntime
  - Base Image: UBI9

================================================================================
  AUTHENTICATION FLOW
================================================================================

  1. External request → OpenShift Route (TLS edge termination)
  2. Route → Frontend Server (HTTP/3000)
  3. Frontend → API Server (HTTP/8888 or gRPC/8887) with Bearer Token
  4. API Server → Kubernetes TokenReview (validates token)
  5. API Server → Extracts user from x-goog-authenticated-user-email header
  6. API Server → Kubernetes SubjectAccessReview (multi-user RBAC check)
  7. Multi-user: Istio AuthorizationPolicy enforces service identity + headers
