RBAC Summary:
- No RBAC resources defined in this repository
- Component inherits service account permissions from the Llama Stack server pod
- Remote provider requires:
  - READ access to ConfigMaps (trustyai-service-operator-config) in
    redhat-ods-applications or opendatahub namespaces
  - READ access to Secrets (S3 credentials, KFP token) in deployment namespace

Secrets:
- (S3 credentials secret) (Opaque) - AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY,
  AWS_DEFAULT_REGION for S3 result storage - Manual provisioning, No auto-rotate
- kubeflow-pipelines-token (Opaque) - KUBEFLOW_PIPELINES_TOKEN for KFP API auth -
  Manual provisioning, No auto-rotate (handled as SecretStr in Pydantic config)
- llama-stack-inference-model-secret (Opaque) - INFERENCE_MODEL, VLLM_URL,
  VLLM_TLS_VERIFY, VLLM_API_TOKEN - Manual provisioning, No auto-rotate
- aws-credentials (Opaque) - AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY,
  AWS_DEFAULT_REGION - Manual provisioning, No auto-rotate

Authentication & Authorization:
- Kubeflow Pipelines API: Bearer Token (OpenShift OAuth or kubeconfig fallback)
  Enforcement: KFP API server validates token
- S3 storage: AWS IAM credentials injected via Kubernetes Secret
  Enforcement: AWS/MinIO server-side
- Kubernetes API: Service Account Token / kubeconfig
  Enforcement: kube-apiserver (in-cluster config preferred)
- Llama Stack Server: NO AUTHENTICATION
  ⚠️  No TLS, no auth enforcement at application level
  All Llama Stack API communication is plaintext HTTP

FIPS Compliance:
- Python-based (no Go FIPS build flags applicable)
- UBI9 base image provides OpenSSL with FIPS capability
- cryptography==46.0.2 (transitive) uses OpenSSL backend on UBI9
- No custom TLS configuration; relies on Python stdlib defaults
- No non-FIPS crypto detected (no MD5, RC4, DES, or custom crypto)

Security Concerns:
⚠️  Llama Stack API (8321/TCP): Plaintext HTTP with no authentication
   - Client → Llama Stack: evaluation requests unencrypted
   - KFP Pod → Llama Stack: dataset retrieval + inference calls unencrypted
   - Model responses and evaluation data traverse network without encryption
⚠️  Ollama/vLLM (11434/TCP): Plaintext HTTP by default
   - LLM completions and embedding vectors transmitted unencrypted
   - vLLM supports optional TLS (VLLM_TLS_VERIFY) but not enforced
