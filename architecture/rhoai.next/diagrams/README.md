## How to Use

### Mermaid Diagrams (.mmd files)
- **In GitHub/GitLab**: Paste into markdown with ` ```mermaid ` code blocks
- **Render to PNG locally**:
   ```bash
   PUPPETEER_EXECUTABLE_PATH=/usr/bin/google-chrome mmdc -i diagram.mmd -o diagram.png -s 3
   ```

### ASCII Diagrams (.txt files)
- View in any text editor
- Include directly in security documentation
- Perfect for SAR (Security Architecture Review) submissions

## Diagram Descriptions

### Component Dependency Graph
Shows how all 70 platform components depend on each other, organized by functional domain: Platform Operator, Core Infrastructure, Auth, Model Serving, Inference Runtimes, LLM Optimization (llm-d), AI Safety/Guardrails, Model Management, Notebooks, Training/Distributed Computing, Pipelines, Llama Stack, and External Services. The rhods-operator sits at the top managing 15 component operators. KServe is the central serving hub deploying 10 inference runtimes.

### Platform Network Topology
Complete network architecture showing all 8 ingress points (Gateway API HTTPRoute), service mesh communication (PERMISSIVE mTLS default, STRICT for model-registry and MaaS), and 9+ egress destinations. Includes exact port numbers, protocols, encryption, and authentication mechanisms across 6 namespaces.

**Two formats provided**:
- **Mermaid (.mmd)**: Visual diagram with color-coded trust zones. Great for presentations and architecture discussions.
- **ASCII (.txt)**: Precise text format with no ambiguity. Required for SAR (Security Architecture Review) submissions and compliance documentation.

Both formats contain the same information - use Mermaid for visual clarity, ASCII for security reviews.

### Cross-Component Workflows
Sequence diagrams showing 5 end-to-end workflows:
1. **Model Serving (KServe)**: InferenceService creation through inference request
2. **LLM Inference with llm-d**: KV-cache-aware scheduling via Envoy ext_proc
3. **Notebook Development**: OAuth-authenticated notebook creation and access
4. **Pipeline-Driven Registration**: Pipeline execution to model registry to auto-deployment
5. **Guardrailed Inference**: Input/output detection pipeline with vllm-orchestrator-gateway

### Platform Security Overview
Visual representation of the security architecture including:
- **9 authentication mechanisms** (OAuth2, kube-rbac-proxy, Istio AuthorizationPolicy, Kuadrant, JWT, OAuth proxy, MLflow kubernetes-auth, API key, mTLS) plus 6 components with no auth
- **12 key ServiceAccounts** with RBAC bindings
- **9 most privileged ClusterRoles** (color-coded: red = full CRUD, green = read-only)
- **7 critical secrets** (storage credentials, session keys, API keys, database creds)
- **9 webhook TLS certificates** and **6 metrics TLS endpoints** via kube-rbac-proxy
- **8 service-level TLS configurations**
- **FIPS compliance status** across Go (21 enabled, 5 missing), Java (1), Python/Rust (none)
- **Container security patterns** (non-root, UBI9, capability drops, hermetic builds)
- **Service mesh policies** (PeerAuthentication, AuthorizationPolicy)

### Platform Observability
Maps all monitoring endpoints:
- **22 Prometheus scrape targets**: 8 protected by kube-rbac-proxy, 2 by Bearer Token, 1 by OAuth proxy, 11 unauthenticated
- **25 health probes**: Standard K8s (healthz/readyz), Quarkus, gRPC health, V2 inference protocol, simple /health, custom
- **5 distributed tracing integrations**: OTLP gRPC/HTTP to OpenTelemetry Collector and Tempo

### Platform Maturity Dashboard
High-level metrics: 70 components (15 operators, 10 runtimes, 25 services), ~90 CRDs across 15+ API groups, ~150 shipped images. Security posture summary (FIPS gaps, mTLS coverage). Deployment readiness (disconnected support, multi-arch, HA patterns). Language distribution (Go ~35, Python ~25, Rust 4, Java 2).
