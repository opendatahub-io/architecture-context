## Diagram Descriptions

### Component Dependency Graph
Shows how 68 platform components depend on each other. The rhods-operator sits at the top managing 16+ sub-operators. Components are grouped by function: model serving (KServe + llm-d), ML pipelines, training & distributed compute, model management, guardrails, and dashboard. Central components (rhods-operator, kserve, kueue, kube-rbac-proxy) are highlighted with thicker borders. External dependencies (S3, LLM providers, container registries) are shown in orange.

### Platform Network Topology
Complete network architecture showing all ingress points (Gateway API HTTPRoutes + OpenShift Routes), the gateway authentication layer (kube-auth-proxy for OIDC, kube-rbac-proxy for SAR), operator-to-workload communication, and egress destinations. Organized by namespace: redhat-ods-operator, redhat-ods-applications, redhat-ods-monitoring, and user namespaces.

**Two formats provided**:
- **Mermaid (.mmd)**: Visual diagram with color-coded trust zones. Great for presentations and architecture discussions.
- **ASCII (.txt)**: Precise text format with no ambiguity. Required for SAR (Security Architecture Review) submissions and compliance documentation.

Both formats contain the same information -- use Mermaid for visual clarity, ASCII for security reviews.

### Cross-Component Workflows
Four sequence diagrams showing end-to-end user workflows:
1. **Model Serving (KServe InferenceService)**: User creates InferenceService, model loads from S3, client sends inference request through Gateway + kube-rbac-proxy to runtime
2. **ML Pipeline Execution**: Pipeline submitted via Dashboard, Argo orchestrates steps, ml-metadata tracks lineage, artifacts stored in S3
3. **LLM Inference with llm-d Scheduling**: Client request flows through Envoy ext-proc to EPP, which scores endpoints by KV-cache hit rate and queue depth
4. **Notebook Workbench Access**: User creates Notebook CR, kubeflow controller provisions StatefulSet, access via Gateway with OIDC + SAR auth

### Platform Security Overview
Visual representation of the defense-in-depth security architecture:
- **Authentication**: kube-auth-proxy (OIDC/OAuth2), kube-rbac-proxy (SAR), OAuth proxy, API keys, SA tokens, mTLS
- **RBAC**: 11 ClusterRoles mapped to ServiceAccounts (most privileged highlighted in red)
- **Secrets**: 11 critical secret types (credentials highlighted in yellow, TLS certs in lighter yellow)
- **TLS**: 7 TLS configuration patterns (webhooks, metrics, gateway, Ray mTLS) shown in blue
- **Container Security**: Non-root, read-only FS, drop ALL caps, FIPS crypto, SCC, network policies in green
- **Service Mesh**: Gateway API, mTLS support, network policies, FIPS cryptography

### Platform Observability
Maps all monitoring and observability endpoints:
- **Prometheus**: 16 scrape targets across operators and services, with auth mechanism per endpoint
- **Health Probes**: 10 components with liveness and readiness probe details (port, path)
- **Distributed Tracing**: 5 components sending OTLP to OpenTelemetry Collector and Tempo
- **Monitoring Stack**: Prometheus, ThanosQuerier, Tempo, OpenTelemetry Collector, Perses

### Platform Maturity Dashboard
High-level metrics about platform health:
- 68 total components (16 operators, 18 services, 14 runtimes, 8 libraries, 5 sidecars, 4 CLI/init, 3 frontends)
- 80+ CRDs, 90+ container images
- Security: MIXED mTLS, FIPS supported, non-root everywhere
- API maturity: mixed (v1 through v1alpha1)
- Deployment: disconnected supported, 4 architectures (amd64 full, others partial)
- Monitoring: 16 Prometheus endpoints, 10 health probes, 5 tracing components
