
- **VS Code**: Install "Mermaid Preview" extension for live rendering

## Diagram Descriptions

### Component Dependency Graph
Shows how all 70 platform components depend on each other, organized into functional groups: Platform Operator, Model Serving & Inference, llm-d Disaggregated Inference, Gateway & Auth Infrastructure, MaaS & Guardrails, Training & Scheduling, Pipelines, Registry/Experiments/Evaluation, Notebooks & Feature Stores, and External Dependencies. Central components (rhods-operator, KServe, Kueue) are highlighted with thicker borders. Color coding: purple for platform operator, blue for core controllers, green for services, red for llm-d stack, orange for gateway/auth, gray for external dependencies.

### Platform Network Topology
Complete network architecture showing all trust zones: External (untrusted), Ingress (Gateway API with Envoy), Platform Services (redhat-ods-applications), Operator namespace (redhat-ods-operator, OLM-managed), User namespaces (per-project, RBAC isolated), Service Mesh (istio-system), Knative Serving, and Egress (S3, container registries, K8s API, external LLM providers).

**Two formats provided**:
- **Mermaid (.mmd)**: Visual diagram with color-coded trust zones showing controllers, user workloads, mesh components, and egress targets.
- **ASCII (.txt)**: Precise text format with complete authentication mechanism inventory (7 patterns), webhook server listing (15+), RBAC summary (top 5 ClusterRoles), namespace table, service mesh configuration, and Gateway API migration status.

Both formats contain the same information — use Mermaid for visual clarity, ASCII for security reviews.

### Cross-Component Workflows
5 sequence diagrams showing end-to-end user workflows spanning multiple components:

1. **Model Deployment via Dashboard** — User login through Gateway/kube-auth-proxy, InferenceService creation, model download from S3, HTTPRoute setup by odh-model-controller, production inference request flow
2. **Distributed Training Job Submission** — PyTorchJob submission, Kueue quota evaluation and admission, Training Operator worker pod creation, distributed training with NCCL/Gloo, model registration
3. **ML Pipeline Execution** — KFP v2 pipeline submission, Argo Workflow DAG execution with driver/launcher sidecars, artifact upload to S3, MLMD lineage tracking, optional model auto-deployment
4. **LLM Inference with Disaggregated P/D (llm-d)** — OpenAI-compatible request through Gateway, EPP ext_proc scoring (prefix cache match, queue depth), prefill → KV cache transfer (NIXL V2) → decode token generation with SSE streaming, WVA autoscaling
5. **Guardrailed LLM Inference (MaaS + Guardrails)** — API key validation via Authorino, rate limiting via Limitador, AI Gateway payload processing with NeMo input/output rails (jailbreak detection, PII filtering, content safety), external LLM provider federation

### Platform Security Overview
Visual representation of the complete security architecture:
- **7 Authentication Mechanisms**: kube-rbac-proxy (SAR), kube-auth-proxy (OAuth2/OIDC), OAuth proxy, Bearer Token, Authorino, Istio mTLS, Application mTLS
- **Key ServiceAccounts**: 9 primary SAs with their namespace bindings
- **ClusterRoles**: Top 5 most privileged roles (highlighted in red)
- **Critical Secrets**: Webhook certs, S3 credentials, OAuth secrets, MaaS API keys (credentials highlighted in yellow)
- **TLS Configuration**: Webhooks (9443/TCP), metrics (8443/TCP), mesh (pod-to-pod mTLS), service-ca, Gateway termination (in blue)
- **Container Security**: Non-root (70%), UBI 9 (35%), distroless (9%), multi-stage (74%), digest-pinned (24), SCC enforcement (in green)
- **FIPS Compliance**: kube-auth-proxy, kube-rbac-proxy, odh-model-controller, UBI 9 OpenSSL
- **Known Risks**: 25% root containers, llm-d InsecureSkipVerify default, mixed image pinning, partial FIPS (highlighted in red)

### Platform Observability
Maps all monitoring and observability infrastructure:
- **Prometheus Scrape Targets (20+)**: Operator metrics (kube-rbac-proxy protected, :8443/TCP), service metrics (direct, :8080/TCP), inference runtime metrics (GPU util, token throughput), Istio mesh telemetry
- **Health Probes**: Liveness and readiness probes for all operators (:8081 healthz/readyz) and services (:8080 or application-specific)
- **Distributed Tracing**: Optional OpenTelemetry via OTel Operator, OTLP gRPC export from KServe, DSP, llm-d EPP, Guardrails Orchestrator
- **Visualization**: Perses dashboards (replacing Grafana), OpenShift Prometheus UI
- **ServiceMonitors**: Operator-level (created by rhods-operator) and per-model (created by odh-model-controller)

### Platform Maturity Dashboard
High-level metrics organized into categories:
- **Platform Metrics**: 70 components, 23 operators, 416 Dockerfiles, 137 CRDs, 15+ webhooks, 10+ runtimes
- **Security Posture**: Mixed mTLS, 70% non-root, 74% multi-stage builds, partial FIPS, 7 auth mechanisms
- **Deployment Readiness**: HA via leader election, disconnected support (RELATED_IMAGE), multi-arch (amd64 full, others partial), OCP v4.14-4.17
- **API Maturity**: v1beta1/v1alpha1/v1alpha2 (no v1 stable), Gateway API migration in progress
- **Inference Stack Tiers**: ModelMesh (multi-model CPU), KServe InferenceService (GPU), KServe LLMInferenceService + llm-d (disaggregated P/D)
- **Architectural Risks**: Monolithic operator (16+ subsystems), auth fragmentation (5 patterns), scheduling overlap (3 systems), 25% root containers
