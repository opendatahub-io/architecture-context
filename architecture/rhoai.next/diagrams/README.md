## Diagram Descriptions

### Component Dependency Graph
Shows how the 72 platform components depend on each other, focused on the most architecturally significant relationships. Central components highlighted:
- **rhods-operator** (root operator, deploys 16 component controllers)
- **KServe** (13 reverse dependencies, universal serving control plane)
- **ODH Dashboard** (14 forward dependencies, primary UI)
- **Kueue** (6 reverse dependencies, cross-cutting batch scheduling)
- **kube-rbac-proxy** (sidecar used by 8+ components)

Groups: Core Platform (dark blue), Model Serving (green), ML Lifecycle (steel blue), Training/Batch (purple), AI Safety (red), Feature Engineering (peach), External Dependencies (gray/orange).

### Platform Network Topology
Complete network architecture showing all trust boundaries:
- **Ingress Layer**: Gateway API (data-science-gateway) with HTTPRoutes for 7 component types + legacy OpenShift Routes for CodeFlare/MaaS
- **Operator Namespace**: rhods-operator with webhook (9443) and metrics (8443)
- **Application Namespace**: 16 controllers with their ports, ServiceAccounts, and webhook configurations
- **User Namespaces**: 11 workload types with detailed port/protocol/auth information
- **Egress**: Object Storage, Container Registry, Databases, NVIDIA NIM API

**Two formats provided**:
- **Mermaid (.mmd)**: Visual diagram with color-coded trust zones
- **ASCII (.txt)**: Precise text format with every port, protocol, auth mechanism, and namespace

### Cross-Component Workflows
Four sequence diagrams showing end-to-end platform workflows:
1. **Model Serving**: Dashboard → Model Registry → KServe → vLLM → client inference (single-model)
2. **ML Pipeline Execution**: Dashboard/Notebook → DSP API → Argo Workflows → artifact storage → MLMD
3. **Distributed Training**: CodeFlare SDK → AppWrapper → Kueue admission → KubeRay → mTLS Ray cluster
4. **LLM with Guardrails (Disaggregated)**: LLMInferenceService → llm-d scheduler → guardrails orchestrator → prefill/decode with KV-cache transfer

### Platform Security Overview
Comprehensive security architecture visualization:
- **6 auth mechanisms**: kube-rbac-proxy (dominant), Kuadrant AuthPolicy (next-gen), OAuth proxy (legacy), K8s RBAC, mTLS, HMAC-JWT
- **Top RBAC roles**: Kueue (36), MaaS (36), DSPO (34), KServe (30+), rhods-operator (platform-wide)
- **Critical secrets**: storage-config (S3 creds), webhook certs, database passwords, NIM API keys
- **TLS configuration**: Webhooks (9443, cert-manager), Metrics (8443, FIPS ciphers), service-ca, Gateway termination
- **Container security**: Non-root (72%), distroless, UBI 9, multi-stage, capability drops, FIPS compliance
- **Network isolation**: 28 NetworkPolicies across 12 components
- **Webhooks**: Kueue (21 total), KServe, CodeFlare, Notebooks, Training Operator, Spark Operator

### Platform Observability
Maps all monitoring and observability endpoints:
- **16 operator metrics endpoints** on port 8443 via kube-rbac-proxy
- **5 workload/data plane metrics** sources (DSP API, vLLM, WVA, ModelMesh, Ray)
- **13 controller liveness probes** on port 8081
- **13 controller readiness probes** on port 8081
- **7 workload health probes** (vLLM, Model Registry, MLflow, DSP API, MLMD, Guardrails, Feast)
- **ServiceMonitor resources** created by odh-model-controller, rhods-operator, and Kueue
- **must-gather diagnostics** covering 15+ components

### Platform Maturity Dashboard
High-level metrics organized by category:
- Component statistics (72 total, 20 operators, 432 images)
- API maturity (142 CRDs, 37 API groups, mixed v1/v1beta1/v1alpha1)
- Security posture (6 auth mechanisms, FIPS enforced, 72% non-root)
- Ingress migration status (Gateway API partial, Routes legacy)
- mTLS coverage (MIXED — 4 different implementations)
- Deployment readiness (HA, disconnected, multi-arch)
- New components in rhoai.next (LLMInferenceService, llm-d, Trainer, MLflow Op, Feast, MaaS, Eval Hub, WVA, AI Gateway BBR)
