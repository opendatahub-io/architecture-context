AUTHENTICATION SUMMARY:
- Primary: Bearer Token (OAuth2 JWT validated via JWKS or token introspection)
- Alternative: Custom auth provider (external endpoint validation)
- Token claims mapped to attributes: roles, teams, projects, namespaces
- Unauthenticated endpoints: /v1/health, /v1/version, /v1/inspect/routes,
  /docs, /redoc, /openapi.json

AUTHORIZATION SUMMARY:
- Cedar-like Attribute-Based Access Control (ABAC)
- Policy evaluation order: forbid rules first, then permit rules
- Default policy: DENY (deny by default)
- Supports: owner-based rules, wildcard resource matching
- Scopes: permit/forbid with when/unless conditions

RATE LIMITING:
- QuotaMiddleware with per-client rate limiting
- Separate quotas for authenticated vs anonymous clients
- Configurable window and max requests

SECRETS:
- TLS certificate (PEM) — Server HTTPS, manual/cert-manager, no auto-rotate
- TLS CA bundle (PEM) — mTLS client cert validation, manual/cert-manager
- Provider API keys (env var) — OpenAI, Anthropic, etc., manual, no auto-rotate
- JWKS keys (remote HTTPS) — JWT validation, auto-refreshed (key_recheck_period)
- Introspection credentials (config) — client_id/client_secret, manual

FIPS COMPLIANCE RISKS:
- python-jose uses pure-Python ecdsa library (not OpenSSL-backed) for JWT verification
- No Konflux Dockerfiles in repository (main branch)
- UI Containerfile uses python:3.12-slim (Debian, not UBI)
- No explicit FIPS-mode enforcement on TLS connections
- Downstream release branches expected to add FIPS-compliant build artifacts

INTERNAL STORAGE (no network egress):
- SQLite — Metadata store, KV store, vector store (sqlite-vec), local file
- FAISS — In-memory vector similarity search, in-process library
