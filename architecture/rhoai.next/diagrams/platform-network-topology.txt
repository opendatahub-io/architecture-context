================================================================================
SERVICE MESH CONFIGURATION
================================================================================

Ingress Gateway:
  - Implementation: Istio Gateway API (data-science-gateway)
  - GatewayClass: istio
  - EnvoyFilter: ext_authz + Lua for auth and token forwarding
  - Auth proxy: kube-auth-proxy (auto-detects OAuth vs OIDC)

Service-to-Service TLS:
  - Pattern: OpenShift service-ca annotations
  - Components: DSPO, Model Registry, KServe webhook, TrustyAI, MLflow, Dashboard
  - Certificate source: service.beta.openshift.io/serving-cert-secret-name

Pod-to-Pod TLS:
  - Pattern: cert-manager CA chains (per-component)
  - Components: KubeRay (mTLS per-cluster CA), CodeFlare (RSA 2048-bit per-cluster CA)
  - agents-operator: SharedTrust CA for AuthBridge

mTLS Enforcement:
  - Mode: MIXED (per-component opt-in, no platform-wide mesh-level enforcement)
  - KubeRay: STRICT mTLS per RayCluster with dedicated CA
  - CodeFlare: mTLS with init container cert injection
  - agents-operator: SharedTrust CA mode available

gRPC Communication:
  - Load balancing: Client-side DNS polling
  - Components: Caikit-TGIS (GRPCLoadBalancerProxy), ModelMesh, DSPO metadata Envoy
  - Total: 106 gRPC service definitions across 15 components

Protocol Translation:
  - Envoy ext_proc: AI Gateway (OpenAI <-> provider formats), llm-d-router
  - REST proxy sidecar: ModelMesh (REST <-> gRPC)
