================================================================================
AUTHENTICATION SUMMARY
================================================================================

Pattern 1: Gateway API + kube-auth-proxy (User-facing services)
  Flow: Client -> HTTPS/443 -> Envoy Gateway -> ext_authz -> kube-auth-proxy
        -> OIDC/OpenShift OAuth -> Bearer token injected -> backend
  Used by: Dashboard, Notebooks, Model Registry UI, MLflow UI, Serving endpoints

Pattern 2: kube-rbac-proxy sidecar (Metrics + Internal APIs)
  Flow: Prometheus -> HTTPS -> kube-rbac-proxy -> TokenReview + SAR -> upstream
  Used by: KubeRay, Model Registry, Notebooks, EvalHub, Dashboard,
           odh-model-controller, TrustyAI, Kueue, DSPO, Spark Operator
  Always allows: system:serviceaccount:openshift-monitoring:prometheus-k8s GET

Pattern 3: Kuadrant AuthPolicy + Authorino (Model Serving)
  Flow: Client -> Gateway -> Authorino validates -> KServe backend
  Used by: LLMInferenceService, MaaS model endpoints
  Managed by: odh-model-controller, MaaS Controller

Pattern 4: API Key Authentication (MaaS)
  Flow: Client -> API key header -> Authorino -> maas-api callback -> validate
  Used by: MaaS subscription-based model access

Pattern 5: mTLS (Service-to-Service)
  Flow: Service A -> Istio DestinationRule TLS origination -> Service B
  Used by: KServe backends, CodeFlare/KubeRay inter-pod
  Note: Not all inter-service communication enforces mTLS (MIXED posture)

Pattern 6: OAuth Proxy Sidecar (Legacy)
  Flow: Client -> oauth-proxy (port 8443) -> OpenShift OAuth -> backend
  Used by: ModelMesh Serving (being replaced by Gateway API pattern)

================================================================================
NAMESPACE NETWORK POLICIES
================================================================================

Namespace: redhat-ods-applications
  - Baseline policies deployed by rhods-operator
  - Component-specific policies from: KServe, Notebooks, OGX, KubeRay,
    Model Registry, Kueue, MaaS, DSPO
  - Allows ingress from: Gateway namespace, monitoring namespace,
    operator namespace

Namespace: redhat-ods-monitoring
  - Prometheus scrape access to all platform + user namespaces
  - Thanos Querier access to Prometheus

User Namespaces:
  - Per-component NetworkPolicies (Notebooks, InferenceServices, DSPA, etc.)
  - Allows ingress from: Gateway (for HTTPRoute traffic),
    monitoring (for metric scraping), operator namespace (for reconciliation)

================================================================================
DISCONNECTED (AIR-GAPPED) DEPLOYMENT
================================================================================

Mechanism: RELATED_IMAGE_* environment variables
  - All 120+ images referenced by SHA256 digest in operator CSV relatedImages
  - rhods-operator reads image refs from env vars, not hardcoded registries
  - oc-mirror tool mirrors all required images to private registry
  - kserve-storage-initializer supports private S3 + private OCI registries
  - odh-model-controller NIM integration has air-gapped mode (skip external API)

================================================================================
MULTI-ARCHITECTURE SUPPORT
================================================================================

  amd64 (x86_64):  FULL    - All components, including GPU workloads
  arm64 (aarch64): PARTIAL - Go operators + CPU workloads only (no CUDA)
  ppc64le:         PARTIAL - Select Go operators only (no GPU)
  s390x:           PARTIAL - Select Go operators only (no GPU)
