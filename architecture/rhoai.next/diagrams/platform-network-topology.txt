================================================================================
AUTHENTICATION ARCHITECTURE
================================================================================

Layer 1 — Gateway Authentication (kube-auth-proxy):
  Scope: All Gateway API HTTPRoute-based ingress
  Mechanism: OAuth2/OIDC → Kubernetes TokenReview
  Replaces: Per-component oauth-proxy sidecars (RHOAI 2.x)

Layer 2 — Service Authorization (kube-rbac-proxy):
  Scope: Per-service sidecar on 8+ components
  Mechanism: Bearer token → Kubernetes SubjectAccessReview
  Components: odh-dashboard, Model Registry Operator, CodeFlare Operator,
    KServe, DSPO, TrustyAI, MaaS, OpenVINO Model Server

Layer 3 — API Policy (Kuadrant/Authorino):
  Scope: KServe InferenceServices, MaaS endpoints
  Mechanism: API keys (HTTP callback), OIDC JWT, TokenReview
  Enforcement: OPA policies for group membership, subscription validity
  Rate limiting: Limitador per-subscription quotas

Layer 4 — Service Mesh mTLS:
  Scope: KServe/Knative workloads
  Mechanism: Istio STRICT mTLS, PeerAuthentication
  Certificate: Istio CA auto-rotation

Special Cases:
  - CodeFlare: oauth-proxy sidecar via mutating webhook (per-RayCluster)
  - Ray clusters: Self-signed per-cluster CA mTLS (init containers)
  - ModelMesh: etcd-based authorization + Istio mTLS
  - Inference runtimes: Unauthenticated — auth delegated to serving layer
