## Purpose

**Short**: Argo Workflows is the workflow execution engine that powers Data Science Pipelines in Red Hat OpenShift AI, providing Kubernetes-native workflow orchestration with DAG and step-based execution models.

**Detailed**: Argo Workflows is a container-native workflow engine for orchestrating parallel jobs on Kubernetes. In the RHOAI context, it serves as the backend execution engine for Data Science Pipelines, managing the lifecycle of ML pipeline runs as Kubernetes-native Workflow custom resources. The system consists of two primary components shipped for RHOAI: a workflow controller that watches Workflow CRDs and orchestrates pod creation for each pipeline step, and an executor (argoexec) that runs as a sidecar/init container in each workflow pod to manage artifact upload/download, parameter extraction, and container lifecycle coordination.

The upstream Argo Workflows project also includes an API server (argo-server) with a React UI and gRPC/REST API, but only the workflow-controller and argoexec binaries are built and shipped via Konflux for RHOAI. The controller implements sophisticated orchestration logic including DAG scheduling, step sequencing, cron scheduling, artifact garbage collection, workflow archival to SQL databases, and leader election for high availability. The executor supports multiple artifact storage backends (S3/Minio, GCS, Azure Blob, Git, HDFS, HTTP, OSS) and uses an emissary pattern for inter-container coordination within workflow pods.

## Architectural Analysis

Argo Workflows in the RHOAI context is a focused downstream fork that ships only two of the three upstream binaries: the workflow-controller and argoexec. The argo-server (API/UI) is intentionally excluded from Konflux builds, as Data Science Pipelines provides its own API layer. This architectural choice means the controller operates headlessly, driven entirely by CRD watches rather than user-facing API calls.

The controller uses Kubernetes client-go informers directly rather than controller-runtime, which is notable because most RHOAI components use the higher-level controller-runtime framework. This gives Argo Workflows fine-grained control over informer indexing, work queue management, and reconciliation filtering. The controller maintains 10+ custom informer indexes (by UID, phase, semaphore config, labels, conditions) to efficiently query the in-memory cache without hitting the Kubernetes API. The worker pool architecture is configurable with separate goroutine pools for workflow reconciliation (default 32), cron scheduling (8), pod cleanup (4), TTL workers (4), and archive workers (8).

The executor (argoexec) uses an emissary pattern for inter-container coordination: it shares a `/var/run/argo/` emptyDir volume with the main container, writing stdout/stderr captures, output parameter files, and artifact tarballs to well-known paths. This avoids the need for Docker socket access or container runtime APIs, making it compatible with any OCI-compliant runtime. The executor builds two binaries in the Konflux Dockerfile — a FIPS-compliant binary (`argoexec-fips` with CGO_ENABLED=1 and strictfipsruntime) and a standard static binary (`argoexec` with CGO_ENABLED=0). However, the container entrypoint defaults to the non-FIPS binary, which may be a gap worth investigating for FIPS compliance. The workflow-controller Konflux build, by contrast, only builds a single FIPS-compliant binary.

The persistence layer is optional and supports both PostgreSQL and MySQL through the upper/db ORM library. When enabled, it provides two features: workflow archival (storing completed workflow JSON for long-term retention) and node status offloading (moving large node status objects out of the Workflow CR to keep etcd object sizes manageable). The migration system uses sequential versioned migrations with database-specific SQL variants. Connection credentials are always retrieved from Kubernetes Secrets, never from environment variables directly.

The artifact subsystem implements a driver interface pattern supporting 8 storage backends (S3, GCS, Azure Blob, Git, HDFS, HTTP/Artifactory, OSS, Raw inline data). Each driver handles authentication via Kubernetes Secrets, supports transient error retry with exponential backoff, and implements both streaming and file-based operations. The S3 driver is the primary backend used with Data Science Pipelines, typically connecting to a Minio instance deployed alongside the pipeline components. All drivers implement the same `ArtifactDriver` interface, making them interchangeable at runtime based on the artifact spec in the Workflow CR.
