## Purpose

**Short**: Argo Workflows is the workflow execution engine that powers Data Science Pipelines in Red Hat OpenShift AI, providing Kubernetes-native workflow orchestration through a controller and executor architecture.

**Detailed**: Argo Workflows is an open-source container-native workflow engine for orchestrating parallel jobs on Kubernetes. In the RHOAI context, it serves as the underlying execution engine for Data Science Pipelines (DSP), managing the lifecycle of ML pipeline runs as Kubernetes-native Workflow custom resources.

The system consists of two primary components: the **workflow-controller** which watches Workflow CRDs, creates pods for each workflow step, manages state transitions, handles garbage collection, and optionally archives workflow history to a database; and the **argoexec** executor which runs as an init/sidecar container within each workflow step pod, handling artifact upload/download, output parameter capture, and inter-step communication.

The controller supports both cluster-scoped and namespace-scoped installation modes, implements leader election for high availability, and provides a comprehensive set of CRDs for workflow definitions, templates, scheduling (CronWorkflow), and event-driven triggering. While the upstream Argo Workflows project also includes an API server (argo-server), the RHOAI distribution focuses on the controller and executor components, with the API server not being built via Konflux for production use.

## Architectural Analysis

The Argo Workflows codebase in this RHOAI distribution is a fork of the upstream argoproj/argo-workflows project, adapted for Red Hat's managed AI platform. The fork maintains upstream feature parity while adding RHOAI-specific Dockerfiles (both legacy UBI8-based and current Konflux UBI9-based), FIPS compliance through the `strictfipsruntime` Go build tag, and Red Hat container labeling.

**Controller Architecture**: The workflow-controller uses a raw client-go informer pattern rather than the higher-level controller-runtime framework common in Kubernetes operators. This gives fine-grained control over work queue management, rate limiting, and resource caching. The controller runs multiple worker pools (32 workflow workers, 4 TTL workers, 4 pod cleanup workers, 8 cron workers, 8 archive workers by default) with a custom rate limiter (20 QPS, burst 30) against the Kubernetes API. Leader election uses Lease-based locking with configurable durations.

**Executor Design**: The argoexec binary runs inside every workflow step pod in two phases — as an init container (preparing the execution environment and downloading input artifacts) and as a wait container (uploading output artifacts and reporting results). It communicates back to the controller via Kubernetes API annotations and the WorkflowTaskResult CRD. The Konflux build produces both a standard binary and a FIPS-compliant binary (`argoexec-fips`) with CGO enabled and the `strictfipsruntime` build tag, allowing runtime selection based on the cluster's FIPS requirements.

**Hydration/Dehydration Pattern**: Large workflows can exceed Kubernetes etcd object size limits. The controller implements a hydration/dehydration pattern that compresses node status data or offloads it to an external database. Compression is attempted first (using gzip), with database offloading as a fallback for workflows exceeding ~150KB. This pattern uses retry strategies with exponential backoff (5 attempts, 100ms-3s for reads, 1s-31s for writes).

**RHOAI Integration Model**: In the RHOAI platform, Argo Workflows is not deployed standalone — it is managed by the Data Science Pipelines Operator (DSPO). DSPO handles image references, namespace placement, RBAC setup, database configuration, and artifact storage configuration. The workflow-controller operates purely as an internal execution engine; end users interact with the Data Science Pipelines API rather than directly with Argo Workflows CRDs or the argo-server. This is why only the workflow-controller and argoexec components are built via Konflux — the argo-server is not needed in production RHOAI deployments.

**Security Posture**: Both containers run as non-root with read-only root filesystems (where possible), drop all Linux capabilities, and use dedicated UIDs (8737 for controller, 2000 for executor). The controller has restricted secret access — it can only read the specific `argo-workflows-agent-ca-certificates` secret by name. The aggregate RBAC roles (view/edit/admin) integrate with Kubernetes' built-in role aggregation for seamless RBAC extension.
