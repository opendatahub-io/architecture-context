## Purpose

**Short**: Argo Workflows is a container-native workflow engine for orchestrating parallel jobs on Kubernetes, used as the execution backend for Data Science Pipelines in RHOAI.

**Detailed**: Argo Workflows provides the core workflow execution engine that powers Data Science Pipelines (DSP) in Red Hat OpenShift AI. It defines a set of Custom Resource Definitions (CRDs) — Workflow, WorkflowTemplate, CronWorkflow, ClusterWorkflowTemplate, and others — that represent directed acyclic graphs (DAGs) of containerized steps. When DSP compiles a pipeline into a Workflow CR, Argo's workflow-controller reconciles it by creating Kubernetes Pods for each step, managing dependencies between steps, handling retries, timeouts, and artifact passing between steps.

The repository produces two Konflux container images for RHOAI: the **workflow-controller** (the Kubernetes controller that reconciles Workflow CRs and orchestrates pod execution) and **argoexec** (the executor sidecar injected into each workflow step pod to manage artifact I/O, progress reporting, and container lifecycle). In the RHOAI context, the upstream Argo Server (API/UI) component is not shipped — only the controller and executor are used, with DSP providing its own API layer. The workflow-controller builds with FIPS-compliant crypto (`GOEXPERIMENT=strictfipsruntime`), while argoexec ships both a FIPS and non-FIPS binary to support flexible deployment.

## Architectural Analysis

Argo Workflows in the RHOAI context is a focused deployment of the upstream Argo Workflows project, stripped down to just the workflow execution engine (controller + executor). The upstream project is a full-featured platform with a UI, API server, SSO, and CLI — but in RHOAI, only the controller and executor are shipped as Konflux images, with Data Science Pipelines providing its own API and persistence layer on top. This is a deliberate architectural choice: DSP owns the user-facing experience while Argo provides the low-level pod orchestration.

The workflow-controller uses a multi-queue, multi-worker architecture with configurable concurrency: 32 workflow workers, 4 TTL workers, 4 pod cleanup workers, 8 cron workers, and 8 archive workers by default. This provides high throughput for parallel pipeline execution while keeping different concerns (GC, archiving, scheduling) isolated. Leader election via Kubernetes Leases ensures only one controller instance is active at a time, with configurable lease duration/renew/retry periods. The controller's health check (`/healthz` on port 6060) is intelligent — it checks whether any workflows have been unreconciled for longer than `HEALTHZ_AGE` (default 5 minutes) and only reports healthy when the leader is keeping up.

The argoexec executor is architecturally interesting because the Konflux Dockerfile ships **two binaries** — `/bin/argoexec` (non-FIPS, statically linked with CGO_ENABLED=0) and `/bin/argoexec-fips` (FIPS-compliant, dynamically linked with CGO_ENABLED=1 and strictfipsruntime). The ENTRYPOINT defaults to the non-FIPS binary, meaning the consuming deployment (DSP Operator) must explicitly override the command to use the FIPS variant in FIPS-enforced clusters. This dual-binary pattern is unusual and creates a gap where FIPS compliance depends on correct deployment configuration rather than being baked into the image default. The workflow-controller Konflux image, by contrast, only ships a single FIPS binary.

The artifact subsystem is a significant piece of the architecture, supporting 10 storage backends (S3, GCS, HDFS, OSS, Azure, HTTP, Git, Raw, Resource, Logging). In RHOAI/DSP deployments, S3-compatible storage (typically MinIO or AWS S3) is the primary backend. Artifact credentials are loaded from Kubernetes Secrets and ConfigMaps, with memoization in the executor to avoid repeated API calls. The executor uses a shared filesystem pattern (`/var/run/argo/`) for inter-container communication — init containers write artifacts and scripts to shared volumes, the main container executes, and the wait/emissary sidecar reads outputs and uploads them.

The TLS configuration deserves attention: the metrics endpoint defaults to TLS-enabled with a self-signed certificate generated at startup using ECDSA P-256. The `InsecureSkipVerify: true` setting in the self-signed TLS config (util/tls/tls.go:127) is intentional for the local self-signed case, but the same pattern appears in the SSO OIDC provider config where it's a configurable option — this is a potential security concern if enabled in production.
