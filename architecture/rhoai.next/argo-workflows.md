## Purpose

**Short**: Argo Workflows is a container-native workflow engine for orchestrating parallel jobs on Kubernetes, used as the execution backend for Data Science Pipelines in RHOAI.

**Detailed**: Argo Workflows provides the core workflow execution engine for the Data Science Pipelines component within Red Hat OpenShift AI. It defines and manages Custom Resource Definitions (CRDs) for Workflows, WorkflowTemplates, CronWorkflows, and ClusterWorkflowTemplates, enabling users to define complex multi-step pipelines as directed acyclic graphs (DAGs) or sequential steps.

The system consists of three primary components: a **workflow controller** that watches Workflow CRs and orchestrates their execution by creating and managing Kubernetes pods; an **API server** (argo-server) that exposes gRPC and REST APIs for workflow management, artifact serving, and a web UI; and an **executor** (argoexec) that runs as a sidecar/init container within workflow pods to handle artifact I/O, container lifecycle management, and result reporting.

In the RHOAI context, Argo Workflows is deployed by the Data Science Pipelines Operator (DSPO) and provides the underlying execution engine that runs pipeline steps as Kubernetes pods. The workflow controller handles scheduling, parallelism control, artifact management, retry logic, and workflow archival to a database (PostgreSQL or MySQL). The system supports multiple artifact storage backends including S3/MinIO, GCS, Azure Blob Storage, and others, which are critical for persisting pipeline artifacts between steps.

## Architectural Analysis

Argo Workflows is a mature, feature-rich workflow engine that has been adapted for Red Hat's product requirements. The codebase is a fork of the upstream `argoproj/argo-workflows` with RHOAI-specific modifications focused on security hardening (FIPS compliance) and container build pipelines (Konflux Dockerfiles).

**Dual-binary FIPS strategy for argoexec**: The argoexec Dockerfile uniquely builds two binaries — a FIPS-compliant dynamically-linked binary (`argoexec-fips`) and a standard statically-linked binary (`argoexec`). Both are included in the final image. This approach allows the same image to serve both FIPS-mandated and non-FIPS environments. The workflow-controller, by contrast, only builds the FIPS variant. The `go.mod` file also sets `godebug fips140=on` at the module level, indicating migration to Go's native FIPS 140-3 module (introduced in Go 1.24+), moving away from the older OpenSSL-based FIPS approach. Recent commits confirm this transition: "Switch from OpenSSL FIPS to native Go FIPS 140-3 module (#67)".

**Controller architecture without controller-runtime**: Unlike most RHOAI operators that use `sigs.k8s.io/controller-runtime`, Argo Workflows builds its controller directly on `k8s.io/client-go` with hand-rolled informers, work queues, and reconciliation loops. This gives it fine-grained control over concurrency (configurable worker pools per subsystem), rate limiting (custom `fixedItemIntervalRateLimiter`), and resource lifecycle management. The downside is significantly more complex code compared to controller-runtime based operators.

**Rich subsystem decomposition**: The workflow-controller is internally composed of multiple cooperating subsystems: the main workflow reconciler, a cron controller, a GC controller, a pod cleanup controller, an artifact GC controller, and a synchronization manager (semaphores/mutexes). Each has its own work queue with configurable worker counts. Leader election ensures only one replica processes workflows, with followers running a dummy metrics server for monitoring continuity.

**Database-backed persistence**: The optional persistence layer (PostgreSQL or MySQL) serves two purposes: workflow archiving (storing completed workflow YAML for long-term retrieval) and node status offloading (moving large workflow status data from etcd to the database to prevent etcd size limits). This is critical in DSP environments where pipeline runs can have hundreds of steps. The database layer uses `upper/db/v4` as an abstraction with a custom migration system.

**TLS security considerations**: The codebase uses `InsecureSkipVerify: true` in several places — the SSO OIDC client (configurable via `insecureSkipVerify` in SSO config), the self-signed TLS configuration, and the agent executor's HTTP client. While these are configurable or limited to internal communication, they represent potential FIPS compliance gaps if not properly configured in production. The TLS minimum version defaults to 1.2 and is configurable via the `TLS_MIN_VERSION` environment variable.
