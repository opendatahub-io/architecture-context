## Purpose

**Short**: Argo Workflows is a Kubernetes-native workflow engine that orchestrates parallel jobs on Kubernetes, used as the execution backend for Data Science Pipelines in RHOAI.

**Detailed**: Argo Workflows provides a container-native workflow engine for orchestrating parallel jobs on Kubernetes. It implements workflows as Kubernetes Custom Resource Definitions (CRDs), where each step in a workflow runs as a container. The engine supports DAG-based and step-based workflow execution, cron scheduling, artifact management, and event-driven triggers.

In the RHOAI (Red Hat OpenShift AI) platform, Argo Workflows serves as the execution backend for Data Science Pipelines (DSP). The DSP operator deploys the workflow controller and argoexec components to run ML pipeline steps. The Argo Server (API server + UI) is not shipped in the RHOAI distribution — only the workflow-controller and argoexec images are built via Konflux for production use. The fork maintained under `red-hat-data-services` adds FIPS compliance, UBI-based container images, and Konflux CI/CD integration while tracking upstream Argo Workflows.

The system consists of three main binaries: the **workflow-controller** (a Kubernetes controller that watches Workflow CRDs and creates Pods for each step), **argoexec** (an executor sidecar injected into workflow pods to manage artifacts, capture outputs, and report results), and **argo** (a CLI and API server — not shipped in RHOAI). The controller creates and manages Pods, PVCs, ConfigMaps, and PodDisruptionBudgets dynamically based on workflow specifications.

## Architectural Analysis

Argo Workflows is a mature, large-scale Go project (4000+ lines in `operator.go` alone) that functions as a full workflow orchestration engine. The Red Hat fork adds minimal changes — primarily FIPS-compliant build configuration and UBI-based container images — while tracking upstream closely. The RHOAI distribution ships only two of the three binaries (workflow-controller and argoexec), as the Argo Server (API + UI) is replaced by the DSP API server in the platform stack.

The controller architecture follows a traditional Kubernetes informer pattern rather than controller-runtime. It uses shared informers with 20-minute resync periods for CRDs and 30-minute resync for Pods. The main reconciliation loop in `operator.go` is extensive, handling template resolution, pod creation, artifact management, retry logic, DAG/step execution ordering, synchronization locks, and garbage collection in a single large file. This monolithic operator pattern is characteristic of pre-controller-runtime Kubernetes controllers and makes the reconciliation flow complex but self-contained.

The argoexec component uses an "emissary" pattern where it acts as a process supervisor inside workflow pods. It replaces the original container command, manages I/O through a shared `/var/run/argo` volume, captures stdout/stderr, compresses artifacts into tarballs, and reports results back to the controller via the WorkflowTaskResult CRD. The dual-binary approach in the Konflux Dockerfile (both FIPS and non-FIPS binaries) is noteworthy — the non-FIPS binary uses `CGO_ENABLED=0` for a fully static build (smaller, no OpenSSL dependency), while the FIPS binary requires dynamic linking. The DSP operator selects the appropriate binary at runtime.

The FIPS story is well-implemented at the build level with the newer `godebug fips140=on` approach in go.mod (replacing the older `GOEXPERIMENT=strictfipsruntime` pattern used in the Dockerfiles — both are present for belt-and-suspenders compliance). However, at the application level, `InsecureSkipVerify: true` appears in several TLS configurations (self-signed cert generation, SSO client when configured). These are configurable and expected for development scenarios, but architects should verify that production deployments of the Argo Server (if used) supply proper TLS certificates. The workflow-controller itself does not serve TLS — it only makes outbound TLS connections to the Kubernetes API and artifact stores, inheriting the cluster's trust chain.

The integration with the RHOAI platform is lightweight: the DSP operator deploys these images and manages the workflow-controller ConfigMap. The controller's egress is primarily to the Kubernetes API (for pod management) and optionally to external artifact stores and databases. There are no ingress requirements for the controller or executor — the Argo Server's network exposure (port 2746) is only relevant if deployed independently, which is not the case in the standard RHOAI deployment.
