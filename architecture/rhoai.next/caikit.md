## Purpose

**Short**: Caikit is a Python AI toolkit and runtime framework that enables users to manage, serve, and train AI models through task-specific APIs over gRPC and HTTP.

**Detailed**: Caikit provides a modular framework for wrapping AI models as "modules" that implement well-defined "tasks" with typed data model inputs and outputs. The framework auto-generates gRPC and REST service definitions from module signatures, providing a consistent API layer regardless of the underlying model implementation (transformers, sklearn, tensorflow, etc.). This abstraction allows application developers to consume AI models through structured data APIs without understanding the model's internal tensor representations.

The runtime component serves as a dual-protocol server (gRPC and HTTP/REST via FastAPI) that can load, serve, and manage multiple models simultaneously. It integrates with IBM's ModelMesh for model lifecycle management (loading, unloading, sizing) in Kubernetes environments, and supports features like model batching, request metering, distributed tracing (OpenTelemetry), TLS/mTLS, and server-sent events for streaming inference. Caikit also provides a health probe utility, a pluggable model management system with configurable finders, initializers, and trainers, and domain-specific interfaces for NLP, time series, and vision tasks.

Within the RHOAI platform, caikit serves as the core model-serving framework used by downstream components (e.g., caikit-nlp, caikit-tgis-serving) to expose AI model inference and training APIs through KServe's model serving infrastructure.

## Architectural Analysis

Caikit's architecture follows a **decorator-driven registration pattern** where AI model implementations ("modules") are registered at import time via the `@module` decorator and associated with well-defined "tasks" via `@TaskBase.taskmethod()`. This two-tier abstraction (task ↔ module) is the core design decision — it decouples the API surface (defined by tasks) from the implementation (provided by modules), allowing multiple modules to serve the same task with different underlying technologies. The runtime auto-generates gRPC service descriptors and HTTP routes dynamically from discovered module signatures using `py-to-proto`, which means adding a new model type requires zero changes to the serving infrastructure.

The **data model framework** is particularly sophisticated. The `DataBase` metaclass generates protobuf descriptors from Python class definitions, providing seamless serialization between Python objects, protobuf messages, JSON, and binary formats. The `@dataobject` decorator extends this to dataclass-backed definitions, and the backend system (e.g., `PandasTimeSeriesBackend`, `ImagePilBackend`) enables lazy loading and transparent format conversion. This allows the same data model class to be backed by different storage representations depending on the runtime context.

The **dual-protocol serving** (gRPC + HTTP) shares a common thread pool and servicer layer, with the HTTP server (FastAPI/Uvicorn) acting as a translation layer that converts HTTP requests into the same servicer calls used by gRPC. Streaming inference is supported via gRPC server-streaming RPCs and HTTP Server-Sent Events (SSE). The runtime supports optional TLS and mutual TLS for both protocols, with a shared configuration (`runtime.tls.*`). The health probe (`caikit_health_probe`) is a separate process that can probe both protocols and handles an mTLS edge case where an SSL rejection is treated as a passing health check (the server is "healthy enough to reject").

The **ModelMesh integration** is implemented via the `ModelRuntimeServicer`, which implements the `mmesh.ModelRuntime` gRPC service over a Unix socket (`/tmp/mmesh/grpc.sock`). This sidecar pattern allows ModelMesh to manage model loading/unloading decisions while caikit handles the actual model lifecycle. The `model_management` subsystem uses a factory/plugin pattern with configurable finders, initializers, and trainers, making it extensible for different storage backends and model formats. The local model directory feature supports both eager and lazy loading with configurable polling and write detection to avoid loading partially-uploaded models.

A notable architectural risk is the **absence of built-in authentication** — all endpoints are unauthenticated at the application level, with security fully delegated to the deployment platform (kube-rbac-proxy sidecars, Istio mTLS, etc.). This is architecturally sound for a library but means that any deployment without proper platform-level auth is completely open. The `verify=False` usage in the health probe is a controlled workaround for mTLS detection, not a security vulnerability, as it only affects the health check path.
