## Purpose

**Short**: Caikit is an AI toolkit framework that provides a modular, task-based abstraction layer for building, serving, and managing AI/ML models through stable gRPC and HTTP APIs.

**Detailed**: Caikit serves as the foundational runtime framework for AI model serving in Red Hat OpenShift AI. It enables AI model authors to wrap model implementations into "modules" that expose standardized task-based APIs, decoupling application developers from the underlying model architecture. The framework dynamically generates gRPC and HTTP service endpoints from registered modules and tasks, allowing models to be served without writing custom server code.

The framework provides a dual-protocol runtime server (gRPC + HTTP/REST via FastAPI) with support for unary and streaming inference, model training, model lifecycle management (load/unload/deploy), and integration with IBM's Model Mesh for multi-model serving at scale. It includes a comprehensive data model system built on Protocol Buffers, pluggable module backends for execution flexibility, lazy data streaming with functional transformations, and a health probe binary for Kubernetes liveness/readiness checks.

Caikit is not deployed directly as a container in RHOAI — it is a Python library dependency consumed by higher-level serving runtimes such as `caikit-nlp`, `caikit-tgis-serving`, and other model-specific serving images. Those runtimes register their modules with the Caikit framework, which then generates and serves the appropriate API endpoints.

## Architectural Analysis

Caikit's architecture is centered on a **task-module-runtime** triad. Tasks define abstract AI capabilities (e.g., text generation, embedding, reranking) with explicit input/output type contracts. Modules are concrete implementations that register against one or more tasks and wrap a specific model or algorithm. The runtime dynamically generates gRPC and HTTP API endpoints from the registered task-module mappings, so adding a new model type requires zero server code changes — only a new module registration.

The **dynamic service generation** pipeline is the most architecturally distinctive feature. At startup, the runtime scans the module registry, creates `TaskPredictRPC` objects for each task and `ModuleClassTrainRPC` for each module's training method, and then generates both protobuf service descriptors (for gRPC) and Pydantic models (for HTTP/FastAPI) from the same source of truth (`caikit/runtime/service_generation/`). This ensures the gRPC and REST APIs are always in sync and accurately reflect the registered modules. The HTTP layer uses a custom `PydanticWrapper` (`caikit/runtime/http_server/pydantic_wrapper.py`) that bridges between FastAPI's Pydantic validation and caikit's protobuf-backed DataBase objects, handling nested type conversion, multipart form data with dot-notation for nested fields, and streaming SSE responses.

The **data model system** (`caikit/core/data_model/`) is protobuf-first: all data structures derive from `DataBase`, a metaclass-driven base that parses protobuf descriptors and dynamically creates Python properties with lazy backend delegation. The `@dataobject` decorator allows defining data models as Python dataclasses that automatically get proto serialization via `py-to-proto`, eliminating the need for manually maintained `.proto` files. The backend pattern enables domain-specific data access — time series data can transparently use Pandas or Spark DataFrames, and vision data uses PIL — without changing the serialization layer.

The **Model Mesh integration** deserves special attention. Caikit implements the `ModelRuntime` gRPC interface (defined in `caikit/runtime/protobufs/protos/model-runtime.proto`) that allows IBM's Model Mesh to manage model lifecycle externally. Model Mesh communicates via a Unix domain socket (`/tmp/mmesh/grpc.sock`) and controls loading, unloading, and sizing of models. This is the primary deployment pattern in RHOAI, where Model Mesh handles multi-model scheduling across GPU nodes while caikit handles the actual model serving. The `inference_plugin.model_mesh` configuration section (`config.yml:259-291`) contains Model Mesh-specific settings like capacity limits, loading timeouts, and autoscaling parameters.

The **health probe binary** (`caikit_health_probe/`) is a separate process designed for Kubernetes pod health checks. It checks both liveness (is the caikit runtime process running, via `psutil`) and readiness (can the gRPC/HTTP servers respond, by making actual health check calls). It correctly handles mTLS scenarios by loading the same TLS configuration as the main runtime. This separate binary approach is necessary because the caikit runtime process itself doesn't expose a simple exit-code-based health check suitable for Kubernetes exec probes.
