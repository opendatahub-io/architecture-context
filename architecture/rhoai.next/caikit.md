## Purpose

**Short**: Caikit is a Python AI toolkit that provides a modular framework for building, training, and serving AI models through task-specific APIs over gRPC and HTTP.

**Detailed**: Caikit serves as the foundational model-serving runtime used within the Red Hat OpenShift AI (RHOAI) platform. It abstracts AI model consumption behind stable, task-oriented APIs so that application developers can interact with models using domain-specific data structures (e.g., text, embeddings, time series) rather than raw tensors. This decouples applications from the underlying model implementation, enabling models and applications to evolve independently.

The framework consists of three main layers: (1) a **core module system** that provides a decorator-driven framework for defining AI modules with standardized load/save/run/train interfaces and pluggable execution backends; (2) an **interfaces layer** that defines domain-specific tasks and data models for NLP, time series, and vision use cases; and (3) a **runtime layer** that dynamically generates gRPC and HTTP server endpoints from registered modules, integrates with ModelMesh for model lifecycle management, and provides health checking, metrics, tracing, batching, and metering capabilities.

Caikit is deployed as the serving runtime within KServe/ModelMesh pods in RHOAI. It processes inference and training requests routed through the platform's ingress stack, loading models from storage and executing them via the appropriate module backend. The `caikit-health-probe` utility is used as a Kubernetes liveness/readiness probe for health checking of the runtime process.

## Architectural Analysis

Caikit's architecture follows a **decorator-driven registration pattern** that is central to its extensibility. The `@module` decorator registers AI model implementations against tasks, the `@task` decorator defines abstract ML operations with typed input/output contracts, and the `@dataobject` decorator auto-generates protobuf-backed data classes from Python dataclasses. At import time, these decorators populate global registries (`MODULE_REGISTRY`, `MODULE_BACKEND_REGISTRY`) that the runtime layer introspects to dynamically generate gRPC and HTTP service definitions. This means adding a new model or task requires only Python code; no manual protobuf or endpoint definitions are needed.

The **dual-protocol runtime** (gRPC + HTTP) shares a common thread pool and prediction pipeline through `RuntimeServerBase`. The gRPC server uses `CaikitRuntimeServerWrapper` to intercept and route RPCs to `GlobalPredictServicer`, which handles both unary and streaming inference patterns. The HTTP server (FastAPI/Uvicorn) generates equivalent REST endpoints and uses `sse-starlette` for server-sent event streaming. The `ServicePackageFactory` dynamically builds service packages by introspecting registered modules and tasks, grouping them by task type, and generating request/response message types via `py-to-proto`. This approach ensures API parity across protocols without code duplication.

The **ModelMesh integration** is implemented through the `ModelRuntime` gRPC service, which implements the ModelMesh sidecar protocol (`loadModel`, `unloadModel`, `predictModelSize`, `modelSize`, `runtimeStatus`). Communication with ModelMesh happens over a Unix domain socket at `/tmp/mmesh/grpc.sock`, providing pod-local communication without network overhead. The `ModelRuntimeServicerImpl` translates ModelMesh requests into Caikit's model management layer, handling concurrency configuration (`max_model_concurrency`), capacity reporting, and model sizing. This tight integration makes Caikit a first-class ModelMesh runtime.

The **module backend system** enables the same module interface to run on different execution engines. `LocalBackend` provides in-process execution, while external backends (registered via `register_backend_type`) can provide distributed or accelerated execution. The `LocalModelInitializer` iterates backends in priority order, checking `SUPPORTED_LOAD_BACKENDS` on each module class, and passes the backend instance to `Module.load()` if the signature supports it. This pattern supports heterogeneous deployment where different models may use different backends within the same runtime instance.

A notable design decision is the **health probe as a separate CLI** (`caikit-health-probe`) rather than an in-process endpoint. The probe runs as a separate process invoked by Kubernetes liveness/readiness checks, connecting back to the runtime's HTTP and gRPC servers to verify they are responsive. This provides true external health validation and supports TLS/mTLS configurations, Unix socket probing, and configurable timeouts. The probe uses `psutil` for liveness checks (verifying the runtime process exists) and protocol-specific checks for readiness.
