## Purpose

**Short**: Caikit is an AI toolkit and runtime framework that enables users to manage, serve, and train AI models through task-specific, developer-friendly gRPC and HTTP APIs.

**Detailed**: Caikit provides a modular, extensible framework for wrapping AI/ML models with stable, task-oriented APIs. Rather than exposing raw tensor interfaces, Caikit defines domain-specific tasks (text generation, classification, embeddings, time series forecasting, etc.) and lets model authors implement modules against those task definitions. The framework then automatically generates gRPC and REST service endpoints from the registered module implementations.

The runtime layer provides dual-protocol serving (gRPC on port 8085 and HTTP/FastAPI on port 8080 by default), with full TLS/mTLS support, Model Mesh integration for multi-model serving, distributed tracing via OpenTelemetry, Prometheus metrics, request batching, and training job management. A dedicated health probe binary (`caikit-health-probe`) supports Kubernetes liveness and readiness checks for both protocols.

As an upstream Python SDK, caikit does not ship container images directly from this repository. It is consumed as a Python dependency by downstream runtime components (e.g., `caikit-tgis-serving`, `caikit-nlp`) which build the actual deployable container images for RHOAI. The library defines the core abstractions (Module, Task, DataModel) and the runtime serving infrastructure that those downstream projects rely on.

## Architectural Analysis

Caikit's architecture is built around a **task-module-datamodel triad** that decouples AI model implementations from their serving interfaces. Tasks define abstract operations (e.g., `TextGenerationTask`, `EmbeddingTask`) with typed input/output contracts. Modules implement one or more tasks using concrete ML frameworks (transformers, sklearn, etc.). Data models provide the serialization layer, automatically generating protobuf descriptors from Python dataclasses via the `@dataobject` decorator and `py-to-proto`. This design means adding a new AI capability requires only implementing a Module class with the right task decorator -- the runtime automatically generates both gRPC and REST endpoints from the registration.

The **dynamic service generation** system (`ServicePackageFactory`) is architecturally distinctive. At startup, the runtime scans all registered modules, groups them by task, and generates gRPC service descriptors and HTTP routes on-the-fly. This eliminates the need for hand-written `.proto` files for inference and training RPCs. The generated services support both unary and server-streaming patterns, with HTTP streaming implemented via Server-Sent Events (SSE). Configuration allows fine-grained control over which modules and tasks are included/excluded, enabling operators to tailor the API surface to deployed models.

The **Model Mesh integration** is a key deployment pattern for RHOAI. Caikit implements the `mmesh.ModelRuntime` gRPC service (defined in `model-runtime.proto`), which is the sidecar contract for IBM's Model Mesh framework. Model Mesh handles multi-model orchestration (scheduling which models are loaded where), while caikit handles the actual model loading and inference. Communication uses a Unix domain socket (`/tmp/mmesh/grpc.sock`) for co-located sidecar communication, avoiding network overhead. The `ModelRuntimeServicer` translates Model Mesh's `loadModel`/`unloadModel` RPCs into caikit's internal `ModelManager` operations.

The runtime's **dual-protocol serving** (gRPC + HTTP on separate ports) with shared servicer implementations is well-structured. Both servers share the same `GlobalPredictServicer` and `GlobalTrainServicer`, ensuring consistent behavior. The HTTP layer adds pydantic validation, multipart form data support (for file uploads in training), and SSE streaming. The gRPC layer adds interceptors for metrics collection and a `CaikitRuntimeServerWrapper` that routes dynamically-generated RPC methods through the global predict servicer. Both servers support independent TLS configuration with mutual TLS for client certificate verification.

The **health probe** architecture deserves note: rather than embedding health checks only in the serving process, caikit ships a separate `caikit-health-probe` binary that externally probes both the gRPC and HTTP servers. This is used as the Kubernetes liveness and readiness probe command. The liveness probe checks process existence via `psutil`, while the readiness probe makes actual gRPC health check and HTTP GET requests to the servers, including full TLS/mTLS handshake verification. This external probe approach is more robust than in-process checks because it validates the full network path.
