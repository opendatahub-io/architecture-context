## Purpose

**Short**: Caikit is an AI toolkit and runtime framework that provides modular, task-oriented APIs for loading, serving, training, and managing AI/ML models over gRPC and HTTP.

**Detailed**: Caikit is a Python framework that abstracts AI model consumption through stable, task-specific APIs. Rather than requiring consumers to work with raw tensors or model-specific interfaces, Caikit provides a declarative system where AI model authors implement "modules" (via `@module` decorator) that conform to well-defined "tasks" (via `@task` decorator), and the framework automatically generates gRPC and HTTP service interfaces from those definitions.

The framework consists of three primary layers: (1) a **core library** providing module/task/data-model abstractions, model lifecycle management, and a pluggable backend system; (2) a **runtime** that serves models over dual-protocol gRPC and HTTP (FastAPI) servers with dynamic service generation, request batching, OpenTelemetry tracing, and Prometheus metrics; and (3) **interface definitions** for NLP, time series, and vision domains that define standard data models and task types. Caikit integrates with KServe/ModelMesh via its ModelRuntime gRPC sidecar protocol, making it a key component in the RHOAI model serving stack.

The downstream Red Hat fork is used as the model serving runtime within Red Hat OpenShift AI, where it runs as a container alongside KServe and ModelMesh to serve foundation models and other AI workloads.

## Architectural Analysis

Caikit's architecture is built around a declarative **module/task** pattern that separates AI model implementation from serving infrastructure. The `@module` decorator registers a Python class as a servable AI module, and the `@task` decorator defines the input/output contract. At runtime, the framework introspects all registered modules and tasks, then dynamically generates gRPC service definitions and HTTP routes — no hand-written proto files or route tables are needed for inference and training endpoints. This is a powerful pattern for scaling to many model types without duplicating serving code, but it means the API surface is only fully known at runtime after all modules are loaded.

The dual-protocol runtime (gRPC + HTTP) is notable for its clean separation. Both servers share the same underlying servicers (GlobalPredictServicer, GlobalTrainServicer, etc.) and model management layer, but their startup is independent — either or both can be enabled via configuration. The HTTP server uses FastAPI with uvicorn, supporting Server-Sent Events (SSE) for streaming inference. The gRPC server supports unix sockets for ModelMesh sidecar communication, which avoids network overhead when running as a co-located container. TLS/mTLS is configurable for both protocols with support for file-based or inline certificate content.

The **ModelMesh integration** is architecturally significant. Caikit implements the `ModelRuntime` gRPC sidecar protocol (defined in `model-runtime.proto`), which ModelMesh uses to orchestrate model loading/unloading across a fleet of serving pods. This positions Caikit as the "inner runtime" within a ModelMesh-managed pod — ModelMesh handles routing and scaling while Caikit handles the actual model execution. The `ModelRuntimeServicer` reports runtime capacity, model sizes, and loading status, enabling ModelMesh to make intelligent placement decisions.

The configuration system (via `alchemy-config`) uses a deep-merge strategy with a well-defined precedence chain: base config → YAML overrides → dict overrides → config files (from `CONFIG_FILES` env var) → environment variables. This makes Caikit highly configurable in containerized environments where environment variables are the primary configuration mechanism. The health probe system (`caikit_health_probe`) is a standalone utility that probes both gRPC (via health check protocol) and HTTP (via `/health` endpoint) servers, with full support for TLS and mTLS — critical for Kubernetes liveness and readiness probes in production.

The interface layer defines domain-specific data models and tasks for NLP (11 tasks including text generation, embedding, reranking), time series (4 tasks including forecasting and anomaly detection), and vision (image processing). These are designed to be extensible — downstream libraries (like `caikit-nlp` or `caikit-tgis-serving`) register their own modules that implement these task interfaces. The data model system includes automatic protobuf generation from Python dataclasses via the `py-to-proto` library, maintaining a single source of truth for the API schema.
