## Purpose

**Short**: llm-d is a high-performance distributed inference serving stack for LLMs on Kubernetes, delivered as a set of configuration-driven deployment blueprints (Dockerfiles, Kustomize overlays, Helm values) that orchestrate upstream components (vLLM, SGLang, Gateway API, NIXL) to serve models at scale across multiple hardware accelerators.

**Detailed**: llm-d is a CNCF sandbox project, founded by Red Hat, Google Cloud, IBM Research, CoreWeave, and NVIDIA. Unlike traditional operator or service repositories, llm-d is primarily a **configuration repository** — it does not compile application binaries. Instead, it delivers: (1) **Container images** for running vLLM model servers on NVIDIA CUDA, AMD ROCm, Intel Gaudi (HPU), Intel XPU, Google TPU, and CPU platforms, each with optimized communication libraries (NVSHMEM, RIXL, UCX, NIXL); (2) **Well-lit path guides** — tested, benchmarked Kustomize-based deployment recipes covering key inference patterns: optimized baseline routing, prefill/decode disaggregation, wide expert-parallelism (MoE), tiered KV cache offloading, predicted-latency scheduling, flow control, workload autoscaling, and batch processing; (3) **Gateway recipes** for integrating with Kubernetes Gateway API via Istio, AgentGateway, or kgateway implementations.

The stack is organized in four architectural layers: a **Proxy** layer (Envoy or Gateway API) that accepts user requests; an **Endpoint Picker (EPP)** scheduling brain (from `llm-d-inference-scheduler`) that selects optimal model server pods via prefix-cache affinity, load-aware balancing, and queue depth signals; an **InferencePool** CRD (from Gateway API Inference Extension) that groups model server pods; and **Model Servers** (vLLM or SGLang) executing models on accelerators. Advanced patterns layer on top: P/D disaggregation with NIXL-based KV transfer, KV-cache indexing for precise prefix cache routing, wide expert-parallelism using LeaderWorkerSet for MoE models, hierarchical KV offloading to CPU/disk, predicted-latency scheduling via XGBoost sidecars, and SLO-aware workload autoscaling.

## Architectural Analysis

**Configuration-driven architecture**: llm-d is architecturally unique among RHOAI/ODH components in that it ships no compiled binaries. The repository is a composition engine — it defines how upstream inference engines (vLLM, SGLang) and Kubernetes-native infrastructure (Gateway API, Helm charts, Kustomize) are assembled into production-grade serving topologies. This means the repository's "source code" is primarily Dockerfiles, YAML manifests, Helm values, and shell scripts. The actual scheduling logic lives in `llm-d-inference-scheduler` (Go), and the routing proxy lives in `llm-d-routing-sidecar` (Go) — both are separate repositories consumed as container images.

**Layered Kustomize composition**: The guides follow a disciplined three-tier Kustomize pattern. Recipe bases (`guides/recipes/`) define reusable Deployment, ServiceAccount, and monitoring templates. Each guide's accelerator-specific overlay references the appropriate base, applies `namePrefix` for uniqueness, sets accelerator labels (`llm-d.ai/accelerator-variant`, `llm-d.ai/guide`), and provides `patch-vllm.yaml` files with model-specific vLLM arguments. This pattern enables adding new accelerator variants without duplicating deployment logic — a new accelerator needs only a kustomization.yaml referencing the base and a patch file for its specific arguments.

**Multi-accelerator support with divergent image strategies**: The five Dockerfiles in `docker/` reveal fundamentally different image construction strategies per accelerator platform. CUDA images use a two-stage builder/runtime pattern from NVIDIA's official UBI9-based CUDA images, compiling NVSHMEM, UCX, DeepEP, DeepGEMM, and FlashInfer from source. The ROCm image layers RIXL on top of vLLM's official ROCm image. The HPU image starts from Habana's PyTorch installer. The CPU image is Ubuntu-based with multi-arch support (amd64/arm64). This divergence reflects the reality that accelerator ecosystems have incompatible toolchains and base images. The `docker/common-versions` file provides centralized version pinning across these divergent build paths.

**NIXL as the universal KV transfer layer**: A critical architectural decision is the use of NIXL (v1.0.0) as the unified communication abstraction for KV cache transfer in prefill/decode disaggregation. NIXL abstracts over NVSHMEM (NVIDIA), RIXL (AMD ROCm), and UCX (CPU/generic) to provide a single API for KV transfer regardless of accelerator. This is exposed via port 5600/TCP on both prefill and decode pods, and configured via `--kv_transfer_config '{"kv_connector":"NixlConnector"}'` in vLLM arguments. The routing sidecar (`llm-d-routing-sidecar`) coordinates the transfer by selecting the optimal decode pod and triggering KV movement.

**Security posture is deliberately minimal at the model server layer**: The model servers run plaintext HTTP on port 8000 with no authentication or TLS. This is an intentional architectural decision — security enforcement is delegated to the Gateway/proxy layer (Istio, AgentGateway, or kgateway). The ServiceAccounts use `automountServiceAccountToken: false`, and OpenShift gateway configurations apply comprehensive security contexts (non-root, read-only FS, capability dropping). However, the wide-EP LeaderWorkerSet manifests run as root (UID 0) with `IPC_LOCK` and `SYS_RAWIO` capabilities, which is required for RDMA/NVSHMEM operations but represents a notable security trade-off for multi-node GPU communication.
