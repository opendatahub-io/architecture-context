## Purpose

**Short**: llm-d is the umbrella repository for the llm-d distributed inference serving stack, providing container image build definitions for vLLM model server images (CUDA, CPU, ROCm, XPU), Kustomize-based deployment guides for production inference patterns, and comprehensive architecture documentation.

**Detailed**: llm-d is a Kubernetes-native distributed inference serving stack, a CNCF Sandbox project founded by Red Hat, Google Cloud, IBM Research, CoreWeave, and NVIDIA. This repository serves as the central coordination point for the llm-d project — it does not contain the core runtime code (which lives in separate repositories like llm-d-router and the upstream vLLM/SGLang projects), but instead provides the complete build and deployment infrastructure.

The repository contains Dockerfiles for building optimized vLLM container images across multiple hardware accelerators (NVIDIA CUDA, AMD ROCm, Intel XPU, CPU), with support for advanced networking libraries (NVSHMEM, UCX, UCCL, NIXL, DeepEP, LMCache, FlashInfer) required for disaggregated serving and expert parallelism. Each Dockerfile produces a production-ready inference container that packages a vLLM build with accelerator-specific optimizations.

The guides/ directory provides a comprehensive set of Kustomize-based "well-lit path" deployment recipes covering the full llm-d deployment topology: Gateway API infrastructure, model server deployments (with variants for each accelerator and serving mode), router/EPP configuration via Helm charts, and observability setup. These guides are tested in CI across GKE, OpenShift (CKS), CoreWeave, and other providers. The deployment model centers on three core abstractions: a Kubernetes Gateway (using Gateway API) for ingress, an InferencePool (from the Gateway API Inference Extension) for endpoint discovery, and model server Deployments with label-based membership.

### llm-d-cuda

**Dockerfile**: `docker/Dockerfile.cuda`
**Intent**: Primary production inference image for NVIDIA GPUs. Builds vLLM from source (or precompiled wheel) with full accelerator stack including NVSHMEM for multi-node communication, UCX/UCCL for high-performance networking, NIXL for KV-cache transfer, DeepEP for expert parallelism kernels, LMCache for tiered caching, and FlashInfer for attention kernels. Supports both standard and GB200 (Blackwell) targets via `--target runtime-gb200`.
**Image**: `ghcr.io/llm-d/llm-d-cuda`
**Language**: Python (vLLM runtime), C/C++/CUDA (compiled libraries)

TLS termination is expected to be handled by the Gateway/proxy layer (Envoy, Istio, etc.) external to the model server. For RHOAI productization, FIPS-compliant builds would require Konflux Dockerfiles with appropriate build flags and verified crypto configuration at the proxy layer._

## Architectural Analysis

The llm-d repository is architecturally unique in that it functions as a "meta-component" — it does not contain runtime source code but instead defines the complete build and deployment surface for the llm-d inference stack. The Dockerfiles are the component's primary build artifacts, while the Kustomize guides compose external components (vLLM, llm-d-router, Gateway API implementations, Mooncake) into tested deployment topologies. This design follows the principle of "composition over configuration" stated in the project's principles.

The Dockerfile architecture follows a consistent pattern: each accelerator variant (CUDA, CPU, ROCm, XPU) builds vLLM from source or a precompiled wheel, then layers accelerator-specific networking and optimization libraries. The CUDA Dockerfile is by far the most complex — it is a multi-stage build that compiles 8+ native libraries (UCX, UCCL, NVSHMEM, NIXL, gdrcopy, DeepEP, DeepGEMM, FlashInfer, LMCache) before assembling the runtime image. The build uses aggressive caching (`--mount=type=cache`) and sccache integration for CI performance. A notable design choice is the GB200-specific runtime target (`--target runtime-gb200`) that swaps the DeepEP wheel for a Blackwell-optimized variant, rather than parameterizing the base build — this keeps the Dockerfile linear while supporting divergent hardware configurations.

The deployment guide structure is well-organized around a recipe/overlay pattern. Base manifests in `guides/recipes/` define reusable building blocks (model server Deployments, ServiceAccounts, image components, Gateway CRs), while each well-lit path guide (optimized-baseline, pd-disaggregation, etc.) composes these bases with guide-specific overlays and Helm values. This means deployment configuration is DRY — changing the base vLLM image or router chart version propagates to all guides. The router deployment is handled via Helm charts (standalone or gateway mode) from the llm-d-router repository, referenced by OCI chart URLs in `guides/env.sh`.

A notable security consideration is that the model server pods expose their API on plaintext HTTP (port 8000) with no authentication. The design explicitly delegates TLS termination and authentication to the Gateway/proxy layer. This is appropriate for cluster-internal traffic patterns but means that Kubernetes NetworkPolicies or service mesh mTLS are the only mechanisms preventing unauthorized access to inference endpoints within the cluster. The mooncake gRPC services (ports 50051, 50052) similarly use plaintext without authentication, relying on namespace isolation. For RHOAI productization, these would need to be hardened with mTLS and potentially kube-rbac-proxy sidecars.

The absence of Konflux Dockerfiles is architecturally significant for RHOAI. The current Dockerfiles use upstream base images (NVIDIA CUDA from NGC, Ubuntu, upstream vLLM ROCm) and fetch dependencies from public registries at build time. RHOAI productization would require creating `Dockerfile.konflux.*` variants that use AIPCC base images, add RPM lockfiles for hermeticity, enable FIPS-compliant build flags, and integrate with Hermeto for dependency prefetching. The `docker/constraints.txt` pinning (torch==2.11.0, nixl==1.2.0) provides partial reproducibility but falls short of full hermetic builds.
