## Project Overview

llm-d is a production-ready, Kubernetes-native distributed inference serving stack for large language models. It is primarily a **configuration-driven** project (YAML, Kustomize, Helm, Bash, Dockerfiles) rather than a compiled application. The core deliverable is a set of "well-lit path" deployment guides that orchestrate upstream components -- vLLM, SGLang, Kubernetes Gateway API, NIXL -- to serve models at scale across diverse accelerator hardware (NVIDIA, AMD, Google TPU, Intel XPU/Gaudi, CPU).

The system aims to eliminate operational toil for teams serving LLMs at scale by providing:

- Well-lit paths for anyone to serve LLMs at scale
- An extensible inference scheduler with prefix-cache-aware routing, load balancing, and flow control
- Support for advanced serving patterns: P/D disaggregation, wide expert parallelism, tiered KV cache, autoscaling
- Multi-accelerator support across NVIDIA, AMD, Intel, Google TPU, and CPU
- Kubernetes Gateway API integration for standard L7 networking

## System Architecture

The stack is organized into four layers, from user-facing to hardware:

```
                    Client Requests (OpenAI-compatible API)
                              │
                              ▼
                ┌─────────────────────────────┐
                │     1. PROXY LAYER          │
                │  (Envoy / Gateway API)      │
                │  Accepts requests, consults │
                │  EPP via ext-proc protocol  │
                └──────────┬──────────────────┘
                           │ ext-proc gRPC
                           ▼
                ┌─────────────────────────────┐
                │   2. ENDPOINT PICKER (EPP)  │
                │  The scheduling brain:      │
                │  - Filter → Score → Pick    │
                │  - Flow control & fairness  │
                │  - P/D disaggregation       │
                │  - Prefix cache affinity    │
                └──────────┬──────────────────┘
                           │ endpoint selection
                           ▼
                ┌─────────────────────────────┐
                │   3. INFERENCE POOL         │
                │  (Kubernetes CRD)           │
                │  Groups model server pods,  │
                │  analogous to a Service     │
                └──────────┬──────────────────┘
                           │ label-based discovery
                           ▼
                ┌─────────────────────────────┐
                │   4. MODEL SERVERS          │
                │  (vLLM / SGLang)            │
                │  Execute models on          │
                │  accelerators               │
                └─────────────────────────────┘
                     │          │         │
                  NVIDIA     AMD      Google TPU
                  Intel XPU  Intel HPU   CPU
```

### Request Flow

1. Client sends an OpenAI-compatible API request to the proxy
2. Proxy invokes EPP via Envoy ext-proc gRPC protocol
3. EPP runs its Filter → Score → Pick pipeline to select the optimal model server endpoint
4. Proxy forwards the request to the selected model server
5. Response is streamed back through the proxy to the client

### Deployment Modes

- **Standalone Mode**: EPP runs with an Envoy sidecar in the same pod (no Gateway API required)
- **Gateway API Mode**: EPP integrates with Kubernetes Gateway API (Istio, GKE Gateway, agentgateway) for production L7 networking

## Core Components

### 1. Proxy Layer

The proxy accepts inference requests from clients and routes them to optimal model server endpoints. It uses Envoy's **External Processing (ext-proc)** protocol to consult the EPP for every routing decision.

**Supported Gateway Implementations:**

| Gateway | Status | Notes |
|---------|--------|-------|
| agentgateway | Preferred | Self-installed, recommended for new deployments |
| Istio | Tested | IngressGateway-compatible |
| GKE Gateway | Tested | Cloud-managed (regional internal/external LB) |
| kgateway | Deprecated | Migration path to agentgateway |

**Base Gateway Resource** (from `guides/recipes/gateway/base/gateway.yaml`):
```yaml
apiVersion: gateway.networking.k8s.io/v1
kind: Gateway
metadata:
  name: llm-d-inference-gateway
spec:
  listeners:
    - port: 80
      protocol: HTTP
      name: default
      allowedRoutes:
        namespaces:
          from: All
```

### 2. Endpoint Picker (EPP)

The EPP is the scheduling brain of llm-d. It lives in the separate `llm-d/llm-d-inference-scheduler` repository and is deployed as a pod alongside the proxy. The EPP has a five-layer internal architecture:

#### EPP Internal Architecture

```
Request → [Ext-Proc Server] → [Request Handler] → [Flow Control] → [Scheduler] → Endpoint
               │                     │                    │               │
          Protocol Layer        Parser Plugin       Queuing & Fairness  Filter→Score→Pick
```

**A. Ext-Proc Server** -- Standard Envoy external processing protocol (bidirectional streaming, `FULL_DUPLEX_STREAMED` mode). Not extensible.

**B. Request Handler** -- Extensible parsers convert incoming requests to internal format. Default: OpenAI-compatible API parser. Also supports vLLM gRPC and custom protocols.

**C. Data Layer** -- Continuously gathers data from three source types:

| Source Type | Input | Output |
|-------------|-------|--------|
| Polling Data Sources | Prometheus metrics from model servers | Endpoint attributes (queue depth, KV cache %) |
| Notification Sources | Kubernetes API (Pods, CRDs) | Endpoint lifecycle events |
| Endpoint Event Sources | EPP endpoint add/remove events | Setup/cleanup triggers |

Key metrics extracted: `TotalQueuedRequests`, `TotalRunningRequests`, `KVCacheUtilization`, `BlockSize`, `NumGPUBlocks`. Supports multiple inference engines (vLLM, SGLang, TensorRT-LLM, Triton) via engine-type labels.

**D. Flow Control** -- Prevents backend overload by managing request queuing at the gateway level:

- **Priority Bands**: Premium, Standard, Best-Effort traffic classes
- **3-Tier Dispatch**: Priority (highest band first) → Fairness (round-robin across tenants) → Ordering (FCFS/EDF/SLO-deadline within flow)
- **Saturation Detection**: Concurrency-based (open-loop) and utilization-based (closed-loop via KV cache + queue telemetry)
- **Fairness Policies**: `round-robin-fairness-policy`, `global-strict-fairness-policy`
- **Ordering Policies**: `fcfs-ordering-policy`, `edf-ordering-policy`, `slo-deadline-ordering-policy`
- **Client Integration** via headers: `x-gateway-inference-fairness-id`, `x-gateway-inference-objective`

**E. Scheduler** -- Extensible Filter → Score → Pick pipeline:

**Filters** (narrow candidates):
- `prefix-cache-affinity-filter` -- Prefer endpoints with high cache match
- `slo-headroom-tier-filter` -- Filter by SLO tier
- `prefill-endpoints-filter` / `decode-endpoints-filter` -- Role-based filtering
- `label-selector-filter` -- Custom label matching

**Scorers** (rate 0.0-1.0, weighted):
- `kv-cache-utilization-scorer` -- Lower utilization = higher score
- `queue-depth-scorer` -- Shorter queue = higher score
- `prefix-scorer` -- Score by prefix match length
- `precise-prefix-cache-scorer` -- Real-time ZMQ-based cache locality
- `lora-affinity-scorer` -- Prefer endpoints with LoRA already loaded
- `latency-scorer` -- Based on predicted latency headroom
- `session-affinity-scorer` -- Sticky to previous endpoint
- `token-load-scorer` -- Total token load (input + output)

**Pickers** (select endpoint):
- `max-score-picker` -- Highest score wins
- `random-picker` -- Random selection
- `weighted-random-picker` -- Weighted lottery

**Profile Handlers** (orchestrate profiles):
- `single-profile-handler` -- One profile for basic routing
- `disagg-profile-handler` -- Two profiles (prefill + decode) for P/D disaggregation

### 3. InferencePool

A Kubernetes CRD (from Gateway API Inference Extension) that defines a pool of model server pods. It serves two roles:

1. **Endpoint Discovery (EPP)**: Selector-based pod discovery via Kubernetes labels with dynamic membership as pods scale
2. **Gateway Integration (Controller)**: Links to EPP service via `endpointPickerRef`, configures proxy ext-proc filter, defines failure mode (FailOpen/FailClose)

Relationships: One InferencePool maps to one logical model deployment and one EPP deployment. Namespace-scoped for security isolation.

### 4. Model Servers

Supported engines: **vLLM** (primary) and **SGLang**.

Responsibilities:
- Load models onto accelerators and serve OpenAI-compatible API (`/v1/completions`, `/v1/chat/completions`)
- Expose Prometheus metrics for EPP scheduling decisions
- Manage KV-cache with prefix caching
- Support parallelism strategies (Tensor Parallel, Data Parallel, Expert Parallel)
- Support disaggregated serving via KV transfer protocol (NIXL)

**Base Deployment Template** (from `guides/recipes/modelserver/base/single-host/default/decode-deployment.yaml`):
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: decode
  labels:
    llm-d.ai/role: decode
spec:
  selector:
    matchLabels:
      llm-d.ai/role: decode
  template:
    spec:
      serviceAccountName: sa
      containers:
        - name: modelserver
          image: REPLACE_MODEL_SERVER_IMAGE
          ports:
            - name: modelserver
              containerPort: 8000
              protocol: TCP
          livenessProbe:
            httpGet:
              path: /health
              port: modelserver
          readinessProbe:
            httpGet:
              path: /v1/models
              port: modelserver
          startupProbe:
            httpGet:
              path: /v1/models
              port: modelserver
```

**Health Check Probes:**

| Probe | Endpoint | Purpose |
|-------|----------|---------|
| Startup | `GET /v1/models` | Model loaded and ready (long timeout for large models) |
| Liveness | `GET /health` | Process alive |
| Readiness | `GET /v1/models` | Model available for inference |

## Advanced Serving Patterns

### Prefill/Decode Disaggregation

Splits inference into separate prefill (FLOPs-bound) and decode (memory-bandwidth-bound) phases on different instances, enabling independent scaling and specialization.

**Request Flow:**
```
Client → Proxy → EPP (selects P & D worker pair)
    → Routing Sidecar → Prefill Worker (max_tokens=1)
    → KV cache transfer via NIXL RDMA
    → Decode Worker → Response
```

**KV Transfer via NIXL:** Abstracts multiple backends (UCX, UCCL, libfabric) across transports (NVLink, RoCE, InfiniBand, EFA) with direct GPU memory registration (GPUDirect RDMA).

**Port Configuration:**
- 8000: Prefill vLLM / routing sidecar frontend
- 8200: Decode vLLM
- 5600: NIXL KV cache transfer

**Default Configuration:** Model `openai/gpt-oss-120b`, 4 TP=1 Prefill workers, 1 TP=4 Decode worker, RDMA networking required.

### Wide Expert Parallelism

Expert parallelism (EP=16) with LeaderWorkerSets for Mixture-of-Experts models (e.g., DeepSeek-R1-0528) at scale. Uses 32+ H200 or B200 GPUs with InfiniBand/RoCE RDMA for all-to-all connectivity.

**Deployment:** Uses Kubernetes `LeaderWorkerSet` CRD for multi-node coordination. Platform-specific overlays for GKE (H200/B200) and CoreWeave.

### Tiered Prefix Cache

Hierarchical prefix cache offloading across storage tiers:

| Tier | Speed | Capacity | Implementation |
|------|-------|----------|---------------|
| HBM (GPU memory) | Fastest | Smallest | Native vLLM |
| CPU RAM | Medium | Medium | CPU offloading connector |
| Local Disk | Slower | Larger | File system connector |
| Shared Storage | Slowest | Largest | LMCache / Lustre / CephFS |

Sub-guides: `cpu/` (CPU RAM offloading) and `storage/` (shared storage offloading with LMCache or filesystem connectors).

### Precise Prefix Cache-Aware Routing

Real-time KV-cache event-driven prefix cache awareness using ZMQ:
- vLLM publishes KV-cache block events via ZMQ
- Scheduler indexes block hashes in real-time
- Routes requests to the pod with the highest prefix-block residency
- UDS tokenizer sidecar for token hashing

**Active-Active HA:** Multi-scheduler support with ZMQ redistribution. Each scheduler has its own Envoy gateway sidecar.

### Predicted Latency-Based Scheduling

**Experimental.** ML-based latency prediction for SLO-aware routing:
- In-pod latency predictor sidecars (training + prediction servers)
- Predicts p90 TTFT and TPOT given request features and load
- SLO scorer compares predictions against request SLO headers
- Request shedding if no pod can meet SLOs

**Client Headers:** `x-prediction-based-scheduling: true`, `x-slo-ttft-ms`, `x-slo-tpot-ms`

### Workload Autoscaling

Two strategies:

| Feature | HPA + EPP Metrics | Workload Variant Autoscaler (WVA) |
|---------|-------------------|----------------------------------|
| Complexity | Low | Medium |
| Multi-variant | No | Yes (optimizes across hardware tiers) |
| Scale-to-zero | Yes | Yes |
| SLO-driven | No | Yes (experimental) |
| Metrics source | Prometheus Adapter | KV cache utilization, queue depth |

### Asynchronous Processing

**Experimental.** Queue-based asynchronous inference for latency-insensitive workloads. Decouples request submission from execution, filling slack capacity in inference pools. Supports GCP Pub/Sub and Redis Sorted Set queue backends.

### Guide Overlay Pattern

Each guide customizes the shared recipe bases:

```yaml
# Example: guides/optimized-baseline/modelserver/gpu/vllm/kustomization.yaml

kind: Kustomization
resources:
  - ../../../../recipes/modelserver/base/single-host/default

namePrefix: optimized-baseline-nvidia-gpu-vllm-

  - name: REPLACE_MODEL_SERVER_IMAGE
    newName: vllm/vllm-openai
    newTag: v0.19.1

labels:
  - pairs:
      llm-d.ai/model: Qwen3-32B
      llm-d.ai/guide: optimized-baseline
      llm-d.ai/accelerator-variant: gpu
      llm-d.ai/accelerator-vendor: nvidia
    includeSelectors: true
    includeTemplates: true

patches:
  - path: patch-vllm.yaml    # Model-specific vLLM args, resources, volumes
```

The `patch-vllm.yaml` files contain guide-specific configuration: replica count, vLLM command args (model name, tensor-parallel-size), GPU resource requests/limits, volume mounts (shared memory, torch compile cache), and probe timing.

Gateway recipes follow a base + provider-specific patch pattern. The base defines the core `Gateway` resource, and provider overlays patch in gateway-class-specific parameters. Guides reference the appropriate gateway recipe based on their target platform.

## Container Image Build System

### Dockerfile Architecture

The primary CUDA Dockerfile uses a **two-stage multi-arch build**:

**Stage 1 -- Builder** (`cuda:${VERSION}-devel`):
- Installs compilation toolchain (gcc, cmake, ninja)
- Builds specialized libraries in dependency order: gdrcopy → UCX → NVSHMEM → NIXL → LMCache
- Compiles kernel libraries: DeepEP, DeepGEMM, FlashInfer
- Generates Python wheels in `/wheels`
- Uses sccache (S3-backed) for distributed compilation caching

**Stage 2 -- Runtime** (`cuda:${VERSION}-runtime`):
- Minimal base with only runtime CUDA libraries
- Copies pre-built wheels and libraries from builder
- Installs vLLM and OpenTelemetry instrumentation
- Creates non-root user (uid 2000, vllm) with OpenShift compatibility
- Sets `ENTRYPOINT: python -m vllm.entrypoints.openai.api_server`

**Other Dockerfiles:**
- `Dockerfile.cpu` -- Multi-stage Ubuntu-based, ARM64/AMD64 support, ccache for compilation
- `Dockerfile.hpu` -- Habana base image, vLLM + vllm-gaudi plugin installation
- `Dockerfile.rocm` -- Extends vLLM ROCm image, adds RIXL + UCX for RDMA

| `DEVICE` | `cuda` | Target device (cuda, xpu, cpu, hpu, rocm) |
| `BUILD_TYPE` | `dev` | Build type (dev, prod) |
| `BUILD_DEBUG` | `false` | Include debug symbols |
| `ENABLE_EFA` | `false` | AWS Elastic Fabric Adapter (CUDA only) |
| `BASE_IMAGE_SUFFIX` | `ubi9` | OS base (ubi9 for RHEL, ubuntu24.04) |
| `VLLM_PREBUILT` | `0` | Use production wheels |
| `VLLM_USE_PRECOMPILED` | `1` | Use precompiled shared libraries |

**vLLM Installation Modes:**
1. Fully from source (clone + build)
2. Precompiled binaries with editable source (fast dev builds)
3. Prebuilt wheel only (production, immutable)

### Image Tagging and Publication

**Registry:** `ghcr.io/llm-d`

| Build Type | Tag Pattern | Example |
|-----------|-------------|---------|
| Dev (per-commit) | `llm-d-{device}-dev:sha-{SHORT_SHA}` | `llm-d-cuda-dev:sha-abc1234` |
| Dev (PR) | `llm-d-{device}-dev:pr-{PR_NUMBER}` | `llm-d-cuda-dev:pr-42` |

**Multi-arch strategy:** Per-platform jobs push by digest, then a manifest job combines digests into a multi-arch manifest list. Supports linux/amd64 and linux/arm64 for CUDA (both RHEL and Ubuntu variants).

**Image Variants:** `-ubuntu` (Ubuntu base), `-debug` (debug symbols), `-aws` (EFA support)

## CI/CD Pipeline

The project uses **55 GitHub Actions workflows** organized into:

### PR Validation
- **`ci-pr-checks.yaml`** -- Pre-commit hooks (shellcheck, hadolint, yamllint, markdownlint, custom env var linters)
- **`ci-pr-test.yaml`** -- Smart change detection triggers relevant E2E tests. Includes fork gate for security
- **`ci-kustomize-dry-run.yaml`** -- Creates a Kind cluster, installs CRDs (Gateway API, monitoring, LWS, GKE), runs `kustomize build` + `kubectl apply --dry-run=server` on all changed guides
- **`ci-signed-commits.yaml`** -- DCO sign-off verification

### Image Building
- **`build-image.yml`** -- Reusable workflow called by ci-pr-test. Path-based filtering detects which platforms changed. Matrix builds for CUDA (4 combos: 2 architectures x 2 OS), AWS, XPU, CPU, HPU, ROCm
- **`nightly-build-image.yaml`** -- Nightly builds
- **`ci-release.yaml`** -- Triggered by git tags (`v*`). Builds all platforms, creates multi-arch manifests, runs Trivy security scanning

### E2E Testing
- **Accelerator-specific:** `e2e-optimized-baseline-xpu.yaml`, `e2e-pd-hpu.yaml`, etc.
- **Nightly suites:** Per-guide tests on GKE, OpenShift, CKS for each well-lit path
- **Simulated:** CPU-based testing without real accelerators

### Security
- **Trivy vulnerability scanning** (CRITICAL/HIGH/MEDIUM, SARIF upload to GitHub Security tab)
- **Fork gate** prevents unauthorized builds from external PRs
- **DCO sign-off** required on all commits

| check-added-large-files | `*` | Size limit: 1000KB |

### Shell Script Env Var Standard

All scripts in `docker/scripts/` must declare required environment variables in a header block:
```bash
#!/bin/bash
set -Eeuo pipefail
# Required environment variables:
# - VAR_NAME: description
```

The `scripts/lint-envvars.py` linter validates used variables are declared. The companion `scripts/lint-dockerfile-envvars.py` validates that Dockerfiles provide all variables required by executed scripts. Both run in pre-commit and CI.

## Labeling and Naming Conventions

### Resource Labels

All Kubernetes resources use a consistent labeling scheme:

```yaml
labels:
  llm-d.ai/model: <model-name>              # e.g., Qwen3-32B, gpt-oss-120b
  llm-d.ai/guide: <guide-name>              # e.g., optimized-baseline
  llm-d.ai/accelerator-variant: <variant>   # gpu, cpu, hpu, xpu, tpu-v6, tpu-v7
  llm-d.ai/accelerator-vendor: <vendor>     # nvidia, amd, intel, google
  llm-d.ai/inference-serving: "true"        # Selector for InferencePool
  llm-d.ai/role: <role>                     # decode, prefill, prefill-decode
```

### Resource Naming

- **namePrefix pattern:** `<guide>-<vendor>-<accelerator>-<framework>-` (e.g., `optimized-baseline-nvidia-gpu-vllm-`)
- **Gateway resource:** Standard name `llm-d-inference-gateway` across all guides
- **LeaderWorkerSet names:** Explicit (e.g., `wide-ep-llm-d-decode`, `wide-ep-llm-d-prefill`)

### Port Conventions

| Port | Service |
|------|---------|
| 8000 | vLLM API (default) / Prefill vLLM / Routing sidecar frontend |
| 8200 | Decode vLLM (in disaggregation) |
| 5600 | NIXL KV cache transfer |
| 80 | Gateway HTTP |
| 9002/9090 | EPP Prometheus metrics |
| 5556 | ZMQ (precise-prefix-cache-aware) |
