## Purpose

**Short**: llm-d is a Kubernetes-native distributed inference serving stack that provides optimized container images, deployment recipes, and orchestration patterns for production-scale LLM serving across multiple hardware accelerators.

**Detailed**: llm-d serves as the integration layer for a high-performance distributed inference platform. Rather than containing application source code directly, it provides: (1) multi-stage Dockerfiles that build optimized vLLM-based container images for CUDA, CPU, HPU (Intel Gaudi), ROCm (AMD), and RDMA-tools use cases, bundling networking libraries (NIXL, UCX, NVSHMEM), ML kernels (DeepEP, DeepGEMM, FlashInfer, LMCache), and observability instrumentation (OpenTelemetry); (2) a library of "well-lit path" deployment guides with Kustomize manifests, Helm values, and gateway configurations covering optimized baseline routing, prefill/decode disaggregation, wide expert-parallelism, tiered KV-cache offloading, predicted-latency scheduling, flow control, and workload autoscaling; and (3) CI/CD workflows for nightly end-to-end testing across OCP, GKE, and CKS clusters.

llm-d does not ship controllers or operators — the Kubernetes Inference Gateway (via the `llm-d-inference-scheduler` EPP), vLLM model servers, and gateway providers (Istio, kgateway/agentgateway, GKE) are the runtime components. This repository integrates, configures, and tests those components as a cohesive stack. For RHOAI, llm-d container images are the basis for downstream model-serving images.

## Architectural Analysis

llm-d follows a **composition-over-configuration** architecture that integrates multiple upstream open-source projects into a cohesive inference stack without forking any of them. The core runtime components — vLLM (model server), Kubernetes Gateway API (control plane), and Kubernetes Inference Gateway Extension (intelligent routing) — are developed in their respective upstream projects. llm-d's value-add is the integration layer: optimized container images, battle-tested deployment recipes, and end-to-end CI validation across multiple cloud platforms and accelerator types.

The **container image architecture** is notable for its multi-stage builds with aggressive dependency compilation. The CUDA Dockerfile compiles six major C/C++ libraries from source (NIXL, UCX, GDRCopy, NVSHMEM, DeepEP, DeepGEMM, FlashInfer, LMCache/InfiniStore) in the builder stage, then installs vLLM with optional precompiled binaries from the vLLM wheel index in the runtime stage. Version pinning is centralized in `docker/common-versions`, which is included by the Makefile and consumed as Dockerfile ARGs. The GB200 variant (`runtime-gb200` target) demonstrates how hardware-specific build variants are handled — it swaps the DeepEP wheel for a GB200-optimized version without rebuilding the entire image.

The **deployment recipe system** uses Kustomize bases and overlays to create a composable set of building blocks. A base deployment in `guides/recipes/modelserver/base/` defines the minimal vLLM Deployment, and each guide (optimized-baseline, pd-disaggregation, wide-ep-lws, etc.) adds guide-specific overlays with model configuration, resource requests, labels for InferencePool selection, and accelerator-specific patches. Gateway recipes follow the same pattern — a base Gateway CR is overlaid with provider-specific GatewayClass and parameters. This enables users to mix-and-match accelerator variants, gateway providers, and serving topologies without duplicating manifests.

The **networking architecture** has three distinct planes: (1) an HTTP control plane where the Gateway routes client requests through the EPP for intelligent backend selection, using scoring plugins (queue depth, KV cache utilization, prefix cache hit rate) and scheduling profiles; (2) a data plane where vLLM instances communicate via RDMA/InfiniBand for KV cache transfer (NIXL on port 5600) and collective communication (NVSHMEM for expert parallelism); and (3) an observability plane with Prometheus metrics scraping and OpenTelemetry trace export. The P/D disaggregation topology is particularly interesting — the routing-sidecar runs as an init container with `restartPolicy: Always` on decode pods, fronting vLLM on port 8000 while vLLM listens on port 8200, and coordinates the prefill-to-decode KV transfer via NIXL.

The **security posture** is minimal by design — llm-d is an upstream project that explicitly documents its threat model as a work-in-progress. vLLM serves plaintext HTTP, RDMA transport is unencrypted (operating at L2), and there is no built-in authentication. Security is expected to be layered on by the deployment environment: network policies, gateway-level auth (Istio AuthorizationPolicy, etc.), and platform-specific controls (RHOAI adds kube-rbac-proxy, mTLS, etc.). The ServiceAccount configuration (`automountServiceAccountToken: false`) and non-root user setup (UID 2000, group 0) in the CUDA Dockerfile follow OpenShift security conventions.

A noteworthy architectural decision is the **EPP plugin chain** configuration via Helm values rather than CRDs. The `EndpointPickerConfig` is passed as a YAML file through `pluginsCustomConfig` in the Helm values, defining ordered plugin chains with weights for different scheduling profiles (prefill vs decode). This allows rapid iteration on routing algorithms without CRD schema changes, but means the routing configuration is not natively declarative in the Kubernetes API — it's embedded in the EPP pod's ConfigMap.
