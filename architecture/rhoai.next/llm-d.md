## Purpose

**Short**: llm-d is a Kubernetes-native distributed inference serving stack that provides configuration-driven deployment blueprints, container images, and Kustomize-based guides for serving large language models at scale across multiple accelerators.

**Detailed**: llm-d is the meta-repository for the llm-d project, a CNCF sandbox project founded by Red Hat, Google Cloud, IBM Research, CoreWeave, and NVIDIA. Unlike traditional application repositories, llm-d is primarily a **configuration-driven** project — its core deliverables are container images packaging vLLM with optimized communication libraries (NIXL, UCX, NVSHMEM, DeepEP), and a set of "well-lit path" Kustomize deployment guides that orchestrate upstream components to serve models at production scale.

The stack implements a 4-layer architecture: (1) a Proxy layer (Envoy/Gateway API) that accepts inference requests, (2) an Endpoint Picker (EPP) that makes intelligent routing decisions using prefix-cache affinity, load awareness, and predicted latency, (3) an InferencePool CRD that bridges gateway routing to model server pods, and (4) Model Servers (vLLM, SGLang) executing inference on accelerators. The llm-d repository produces the model server container images and provides the Kustomize deployment blueprints for all four layers. The EPP and routing logic reside in sibling repositories (`llm-d/llm-d-inference-scheduler`). Advanced patterns include prefill/decode disaggregation, wide expert parallelism for MoE models, tiered KV-cache offloading, predicted latency-based scheduling, flow control for multi-tenant fairness, and workload autoscaling.

**Dockerfile**: `docker/Dockerfile.cuda`
**Intent**: Primary model server image for NVIDIA GPU inference. ## Architectural Analysis

llm-d represents an unusual architecture pattern in the Kubernetes ecosystem: a meta-repository whose primary deliverables are container images and deployment configuration rather than application code. The repository contains no Go, Python, or Rust source code of its own — instead, it assembles upstream components (vLLM, NIXL, UCX, NVSHMEM, DeepEP) into production-optimized container images through sophisticated multi-stage Docker builds, and provides Kustomize-based deployment blueprints that compose the full inference stack.

The Kustomize structure follows a well-designed 4-layer composition pattern: shared bases (recipes/modelserver/base/) define the core Kubernetes resources, component layers (recipes/modelserver/components/) add cross-cutting concerns like monitoring, guide overlays (guides/optimized-baseline/, guides/pd-disaggregation/, etc.) compose specific serving patterns, and accelerator variants (gpu/, amd/, xpu/, hpu/, tpu-v6/, cpu/) handle hardware-specific adaptation. This layering achieves excellent reuse — adding a new accelerator variant typically requires only a kustomization.yaml pointing to a base, an image reference override, and a patch for accelerator-specific vLLM arguments. The labels system (llm-d.ai/guide, llm-d.ai/accelerator-variant, llm-d.ai/role) provides consistent metadata for monitoring, autoscaling, and debugging across all variants.

The CUDA Dockerfile is the most complex build artifact, with a builder stage that compiles 7+ native libraries from source (GDRCopy, UCX, NVSHMEM, NIXL, LMCache, DeepEP, DeepGEMM, FlashInfer) and a runtime stage that installs vLLM in one of three modes: full prebuilt wheel, precompiled binaries with editable Python, or full source compilation. The S3-backed sccache integration significantly accelerates compilation. The GB200 variant demonstrates architectural flexibility — it uses Docker multi-stage target selection to swap DeepEP wheels without duplicating the entire build. Notably, the CUDA image supports both RHEL (UBI9) and Ubuntu base images via build arguments, enabling the same Dockerfile to produce images for both RHOAI (UBI-based) and upstream CNCF (Ubuntu-based) distributions.

The security model is minimal by design: model server pods run as non-root user (UID 2000), the ServiceAccount has automountServiceAccountToken disabled, and authentication/authorization is delegated entirely to the upstream proxy layer (Gateway API + EPP). This is appropriate for an inference workload but means that the container images themselves provide no authentication. For RHOAI distribution, the kube-rbac-proxy sidecar pattern used by other RHOAI components would need to be integrated, and the base images would need migration from upstream NVIDIA/Ubuntu to AIPCC base images to meet Red Hat's product security requirements for FIPS-compliant crypto and hermetic builds. Currently, there are no Konflux Dockerfiles, no lock files, and no hermeto/cachi2 prefetch integration — all of which would be required for the Konflux build pipeline used by RHOAI.

The project's investment in operational tooling is notable: custom linters enforce environment variable documentation standards across all build scripts, the CI/CD pipeline validates kustomize builds against kind clusters, nightly E2E tests run across three cloud platforms (OpenShift, CKS, GKE) for seven guide types, and a comprehensive smoke test with structured JSON output enables automated post-deployment verification. The release process supports release candidates with explicit cleanup procedures for GHCR's auto-generated semantic version tags.
