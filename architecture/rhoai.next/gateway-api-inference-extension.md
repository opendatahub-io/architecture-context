## Purpose

**Short**: Gateway API Inference Extension extends Envoy-based gateways with intelligent inference-aware routing, load balancing, and flow control for LLM serving workloads on Kubernetes.

**Detailed**: The Gateway API Inference Extension (also known as the Inference Gateway or IGW) is a Kubernetes-native system that upgrades any Gateway API-compatible proxy supporting Envoy's External Processing (ext-proc) protocol into an inference-aware gateway. It is developed upstream at `sigs.k8s.io/gateway-api-inference-extension` under the Kubernetes SIG Network umbrella. Red Hat ships it as part of Red Hat OpenShift AI to enable intelligent routing for self-hosted generative AI workloads.

The system consists of two primary gRPC servers — the Endpoint Picker (EPP) and the Body Based Router (BBR) — that run as ext-proc sidecars alongside an Envoy-based gateway (e.g., Istio, Envoy Gateway, GKE Gateway). The EPP implements a sophisticated scheduling framework with pluggable filters, scorers, and pickers that select optimal model server endpoints based on real-time metrics such as KV-cache utilization, running request counts, queue depth, prefix cache affinity, and LoRA adapter availability. The BBR parses HTTP request bodies to extract model names and converts them into headers that the gateway can route on before the request reaches the EPP.

The project defines Kubernetes CRDs — `InferencePool` (GA v1), `InferenceObjective`, `InferenceModelRewrite`, and `InferencePoolImport` (experimental) — that provide declarative APIs for platform administrators and workload owners to manage inference pools, set priority-based traffic policies, configure model name rewriting and A/B traffic splitting, and enable multi-cluster pool federation. An optional latency predictor subsystem (Python-based training and prediction servers deployed as sidecars or standalone pods) uses XGBoost models to predict per-request latency for SLO-aware scheduling.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| Endpoint Picker (EPP) | Go gRPC Service (ext-proc) | Core inference scheduling engine — intercepts Envoy requests/responses via ext-proc, selects optimal model server endpoints using pluggable scheduling profiles, manages flow control with priority-based queuing, and records per-request metrics |
| Body Based Router (BBR) | Go gRPC Service (ext-proc) | Parses HTTP request bodies to extract model names and LoRA adapter info, sets routing headers (X-Gateway-Model-Name, X-Gateway-Base-Model-Name) for gateway-level routing before the request reaches EPP |
| Latency Prediction Server | Python FastAPI Service | Serves XGBoost-based latency predictions via HTTP REST API (port 8001), used by EPP's predicted-latency scorer plugin to enable SLO-aware endpoint selection |
| Latency Training Server | Python FastAPI Service | Continuously trains XGBoost latency prediction models from live request telemetry (port 8000), stores trained models on shared persistent storage for prediction servers to consume |
| Latency Predictor Async Client | Go Sidecar Library | In-process Go client embedded in EPP that asynchronously batches training data to the training server and dispatches prediction requests with coalescing for high throughput |
| Controller Reconcilers | Go Controllers (controller-runtime) | Four reconcilers (InferencePool, InferenceObjective, InferenceModelRewrite, Pod) watch Kubernetes resources and maintain an in-memory datastore that drives scheduling decisions |

## Architectural Analysis

The Gateway API Inference Extension represents a sophisticated extension of the Kubernetes Gateway API model into the domain of LLM inference serving. Its architecture follows the Envoy external processing pattern, which allows it to work with any Gateway API implementation that supports ext-proc (Istio, Envoy Gateway, kgateway, GKE Gateway) without tight coupling to any specific gateway controller. This design is intentional — the project is a Kubernetes SIG effort aimed at broad ecosystem compatibility rather than vendor lock-in.

The scheduling framework is the most architecturally interesting component. It uses a plugin-based design inspired by the Kubernetes scheduler framework, with extension points for filtering, scoring, and picking endpoints. Scheduling profiles allow different scheduling strategies to be composed declaratively via the `EndpointPickerConfig` CRD. The framework supports a `ProfileHandler` that iteratively selects profiles to run based on prior results, enabling multi-stage scheduling pipelines. Built-in plugins include KV-cache utilization scoring, prefix cache affinity, LoRA adapter affinity, queue depth scoring, running request counting, predicted latency scoring, and weighted random/max-score pickers. This extensibility is critical for the project's partnership with llm-d, which customizes the scheduling pipeline for disaggregated serving.

The flow control subsystem adds priority-based request queuing and admission control on top of scheduling. It uses a sharded worker pool architecture (`FlowController` → `ShardProcessor`) where requests are distributed to shards managed by a `FlowRegistry`. Each priority band can have independent capacity limits (max bytes, max requests), fairness policies (global-strict, round-robin), and ordering policies (FCFS, EDF, SLO-deadline). A `SaturationDetector` plugin determines when the pool is saturated and flow control should activate. This design enables workload isolation — high-priority production traffic can be guaranteed capacity while best-effort traffic is queued or shed under load.

The data layer architecture (`pkg/epp/datalayer/`) introduces a dependency graph validation system (`ValidateAndOrderDataDependencies`) that topologically sorts plugins based on their producer/consumer data dependencies and validates that the execution order respects layer boundaries (FlowControl → RequestControl → Scheduling). This prevents configuration errors where a scheduling plugin depends on data from a flow control plugin that hasn't run yet. The data layer also manages asynchronous metrics collection from model servers, with configurable data sources (HTTP scraping, Kubernetes notifications) and extractors that parse vendor-specific metric formats (vLLM, SGLang, Triton).

A notable security observation is that the model server metrics scraping defaults to `InsecureSkipVerify: true` for HTTPS connections. While this is pragmatic for development environments where model servers use self-signed certificates, it represents a gap for production RHOAI deployments where certificate verification should be enforced. The ext-proc gRPC connection between Envoy and EPP/BBR can optionally use TLS with self-signed certificates generated at startup (4096-bit RSA, 10-year validity), but this is disabled by default. In Istio deployments, mTLS between services provides transport security without application-level TLS.

The deployment topology uses a `Recreate` strategy rather than `RollingUpdate` for the EPP Deployment. This is a deliberate architectural choice documented in the Helm chart — the EPP maintains stateful in-memory data (prefix cache state, metrics snapshots, flow control queues) that benefits from single-active-replica operation. For HA, leader election is supported with multiple replicas, where the health check differentiates between liveness (always passes) and readiness (only passes on the leader), ensuring Envoy only routes ext-proc traffic to the active leader.
