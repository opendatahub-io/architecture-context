## Purpose

**Short**: An Envoy External Processing (ext-proc) extension that upgrades any Gateway API-compatible proxy into an inference gateway for optimized LLM request scheduling, routing, and load balancing on Kubernetes.

**Detailed**: The Gateway API Inference Extension (also called Inference Gateway or IGW) provides intelligent request routing and load balancing for self-hosted generative AI model servers on Kubernetes. It implements the Envoy ext-proc protocol to intercept inference requests at the gateway layer and make scheduling decisions based on real-time model server metrics such as KV-cache utilization, queue depth, running requests, and prefix cache affinity.

The project consists of two primary services: the Endpoint Picker (EPP) and the Body Based Router (BBR). The EPP is the main scheduling component that watches InferencePool CRDs to discover model server pods, scrapes their Prometheus metrics, and uses a pluggable framework of filters, scorers, and pickers to select the optimal endpoint for each request. The BBR is an optional companion ext-proc server that parses HTTP request bodies to extract model names and LoRA adapter information, placing them into headers for gateway-level routing before requests reach the EPP.

The extension also defines Kubernetes CRDs for InferencePool (stable v1), InferenceObjective, InferenceModelRewrite, and InferencePoolImport (experimental), enabling declarative management of inference workload routing, priority, model name rewriting, and multi-cluster scenarios. An optional latency prediction subsystem (Python-based training and prediction servers with a Go async client sidecar) provides ML-based request latency estimation for scheduling decisions.

## Architectural Analysis

The Gateway API Inference Extension follows a clean separation between the data plane (Envoy proxy) and the control/scheduling plane (EPP/BBR). By implementing the Envoy ext-proc protocol rather than building a custom proxy, the project inherits all of Envoy's networking capabilities (TLS termination, connection pooling, load balancing, observability) while focusing exclusively on intelligent endpoint selection. This design means the extension works with any ext-proc-capable gateway implementation (Envoy Gateway, Istio, GKE Gateway, kgateway, nginx Gateway Fabric), avoiding vendor lock-in.

The EPP's plugin framework is the architectural centerpiece. It defines five scheduling extension points (ProfilePicker, Filter, Scorer, Picker, ProcessProfilesResults) and five request control extension points (PrepareData, AdmitRequest, PreRequest, ResponseHeader, ResponseBody). This mirrors the Kubernetes scheduler's plugin architecture (kube-scheduler framework), which is unsurprising given the project's home in kubernetes-sigs. The framework supports weighted multi-profile scheduling, where different scoring profiles can be composed for different traffic patterns. Built-in scorers cover KV-cache utilization, queue depth, prefix cache affinity, LoRA adapter affinity, token load, and predicted latency -- representing the key dimensions of LLM serving optimization.

The flow control subsystem is particularly sophisticated, using a sharded supervisor pattern with lock-free request hot paths. The `FlowController` implements "Join-Shortest-Queue-by-Bytes" (JSQ-Bytes) distribution across sharded processors, with two-phase distribution (fast non-blocking failover + single blocking fallback). The `FlowRegistry` uses `sync.Map` for the request path and mutex-protected admin paths, with just-in-time provisioning of flows and priority bands. This design targets high-throughput scenarios where thousands of inference requests per second must be prioritized and queued without adding significant scheduling latency.

The latency prediction subsystem (training server + prediction servers + async Go client) represents an ML-within-ML pattern: using machine learning (XGBoost, LightGBM, or Bayesian Ridge regression) to predict the latency of machine learning inference requests. The async client implements request coalescing (batching concurrent prediction requests into a single HTTP call within a configurable time window) and multi-server load balancing, indicating that prediction itself can become a throughput bottleneck at scale. The coalescer design (channel-based with configurable window and row caps) is production-hardened for high-QPS scenarios.

A notable security consideration is that the EPP defaults to `InsecureSkipVerify=true` when scraping model server metrics over HTTPS. This is pragmatic for development (model servers often use self-signed certs) but should be hardened in production RHOAI deployments. The self-signed TLS certificates generated for the ext-proc servers use RSA-4096 with a 10-year validity period and no client certificate verification, meaning the TLS provides encryption but not mutual authentication -- Envoy trusts the EPP based on network proximity rather than certificate validation.
