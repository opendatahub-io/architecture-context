## Purpose

**Short**: Provides intelligent request routing for LLM inference workloads by extending the Kubernetes Gateway API with endpoint picking and body-based routing capabilities via the Envoy external processing (ext-proc) protocol.

**Detailed**: The Gateway API Inference Extension (GIE) is a Kubernetes-native extension to the Gateway API that enables smart routing of inference requests to model-serving backends. It implements an Envoy ext-proc gRPC service that intercepts HTTP requests at the gateway, parses inference request bodies (OpenAI-compatible, vLLM gRPC, or passthrough), evaluates model server metrics (KV cache utilization, queue depth, running requests, LoRA adapter status), and selects the optimal backend endpoint using a pluggable scheduling framework.

The project consists of two primary binaries: the **Endpoint Picker Proxy (EPP)**, which handles intelligent request routing with scheduling, flow control, and model rewriting; and the **Body-Based Router (BBR)**, which provides a lighter-weight filter that extracts request body fields into HTTP headers for upstream routing decisions. Both run as Envoy ext-proc gRPC servers and are designed to work with any Gateway API-compatible gateway implementation (Envoy Gateway, Istio, GKE, nginx Gateway Fabric).

The extension defines custom resources (InferencePool, InferenceObjective, InferenceModelRewrite, InferencePoolImport) that let platform operators configure inference endpoint pools, priority-based request handling, model name rewriting with weighted traffic splitting, and multi-cluster pool federation. It is an upstream Kubernetes SIG project (`sigs.k8s.io/gateway-api-inference-extension`) that Red Hat forks for RHOAI integration.

## Architectural Analysis

The Gateway API Inference Extension implements a sophisticated request routing system for LLM inference workloads using the Envoy ext-proc (External Processing) protocol. The architecture follows a clean separation between the request interception layer (gRPC ext-proc streaming), the decision layer (pluggable scheduling framework), and the data layer (metrics scraping and CRD reconciliation).

**Plugin Architecture**: The EPP implements a Kubernetes scheduler-inspired plugin framework with multiple extension points: Filters (endpoint eligibility), Scorers (endpoint ranking with configurable weights), Pickers (final selection from scored candidates), and ProfileHandlers (multi-profile orchestration). Default plugins score endpoints based on queue depth, KV cache utilization, and prefix cache hit ratio. The framework supports out-of-tree plugins registered at compile time, and all plugin configuration is expressed via the `EndpointPickerConfig` CRD type (config/v1alpha1). The BBR uses a simpler plugin model with RequestProcessor and ResponseProcessor interfaces — primarily extracting body fields into headers for routing decisions.

**Flow Control**: An experimental flow control layer (feature-gated) adds priority-based queuing with configurable fairness policies (global strict, round-robin), ordering policies (FCFS, EDF, SLO-deadline), and usage limit enforcement. This transforms the EPP from a simple request router into a request admission controller that can queue, prioritize, and shed requests based on system saturation. The saturation detector is itself pluggable (concurrency-based or utilization-based).

**Data Layer**: The data layer manages endpoint metrics collection with two operational modes: a legacy polling mode (scraping Prometheus metrics from model servers at configurable intervals) and a newer plugin-based mode using DataSource and Extractor plugins. The data layer feeds the scheduling framework with per-endpoint metrics (queue depth, KV cache utilization, running requests, LoRA adapter status, prefix cache statistics) that scorers use for intelligent placement.

**Model Rewriting**: The InferenceModelRewrite CRD enables weighted traffic splitting across model backends. The Director rewrites the model field in request bodies before dispatching to the selected endpoint, and rewrites it back to the original model name in responses. Rules have precedence (exact match > generic, oldest creation timestamp wins ties), and targets have weights for probabilistic selection — enabling canary deployments and A/B testing of model versions.

**Latency Prediction**: An optional latency prediction subsystem deploys Python-based training and prediction servers as sidecar containers. The async Go client uses a coalescing pattern — multiple concurrent prediction requests are batched into a single HTTP call, reducing overhead. Supported models include Bayesian Ridge (linear regression from cached coefficients), XGBoost (native tree evaluation or HTTP), and LightGBM (HTTP only). Training data (observed TTFT/TPOT with features like KV cache utilization, input length, queue depth) is periodically flushed to the training server.

**Deployment Strategy**: The Helm charts deploy EPP with a Recreate strategy (not RollingUpdate) because the EPP maintains stateful in-memory caches (prefix cache index, endpoint metrics) that benefit from a single active replica. For HA, leader election via Kubernetes leases is supported with multiple replicas, where only the leader processes ext-proc requests while standbys maintain warm caches. The 130-second termination grace period mirrors the expected longest request duration in the model server pool.
