## Purpose

**Short**: Extends any Gateway API-compatible proxy into an inference gateway by providing intelligent, KV-cache-aware endpoint selection and body-based routing for Kubernetes-hosted generative AI model servers.

**Detailed**: Gateway API Inference Extension (IGW) optimizes self-hosting of generative AI models on Kubernetes by leveraging Envoy's External Processing (ext-proc) protocol. It provides two primary services: the **Endpoint Picker (EPP)**, which implements a pluggable scheduling framework that selects optimal backend model server pods based on metrics such as KV-cache utilization, queue depth, running requests, prefix cache affinity, and LoRA adapter availability; and the **Body Based Router (BBR)**, which parses HTTP request bodies to extract model names and route requests before they reach the EPP.

The project defines Kubernetes-native CRDs—`InferencePool`, `InferenceObjective`, `InferenceModelRewrite`, and `InferencePoolImport`—that allow platform teams to declaratively manage pools of model servers, set request priorities and SLO objectives, split traffic between model versions (A/B testing, canary rollouts), and import pools across clusters. The EPP watches these CRDs and pods matching the pool selector, scrapes Prometheus metrics from model server pods, and uses a profile-based scheduling pipeline (Filter → Score → Pick) to select the best endpoint for each inference request. This architecture supports multiple model server backends including vLLM, SGLang, Triton TensorRT-LLM, and trtllm-serve.

## Architectural Analysis

The Gateway API Inference Extension follows a clean separation of concerns between body parsing (BBR) and endpoint selection (EPP), both implemented as Envoy ext-proc services. This architecture allows the BBR to be deployed independently when body-based routing is needed without coupling it to the EPP's scheduling logic. The ext-proc protocol provides a non-invasive integration path — any Gateway API-compatible proxy supporting ext-proc can become an inference gateway without proxy modifications.

The EPP's scheduling framework is modeled after Kubernetes scheduler's plugin architecture with Filter → Score → Pick extension points organized into named profiles. The `EndpointPickerConfig` CRD (v1alpha1) allows runtime configuration of plugin chains and scoring weights. Default plugins include queue-scorer (weight 2), kv-cache-utilization-scorer (weight 2), and prefix-cache-scorer (weight 3), reflecting the project's emphasis on KV cache-aware routing. When latency prediction is enabled, the profile switches to a more sophisticated pipeline with predicted-latency scoring, prefix-cache affinity filtering (two-gate: strict 0.99 threshold then loose 0.80), SLO headroom tier filtering, and weighted random picking.

The BBR's plugin framework (`pkg/bbr/framework/`) provides a simpler extension model: `RequestProcessor` and `ResponseProcessor` interfaces with a factory registry. The built-in `body-field-to-header` plugin extracts a JSON field (default: "model") to a header (default: `X-Gateway-Model-Name`), and the `base-model-to-header` plugin resolves LoRA adapter names to base models via ConfigMap-backed lookup tables. The ConfigMap reconciler watches for ConfigMaps labeled `inference.networking.k8s.io/bbr-managed=true`, parsing `baseModel` and `adapters` keys to build an in-memory adapter-to-base-model mapping. This design allows platform teams to manage model routing configurations via standard Kubernetes resources.

A notable design decision is the EPP's deployment strategy: `Recreate` rather than `RollingUpdate`. The README and Helm values explain this is because EPP maintains stateful information (prefix cache state, scheduling context) that is best served by a single active replica. For HA deployments, leader election via `coordination.k8s.io/Lease` is supported, with the health check service differentiating between liveness (always passing for running pods) and readiness (only true on the elected leader with a synced pool). The 130-second termination grace period mirrors the expected longest grace period of model server pods in the pool.

TLS is handled pragmatically: the ext-proc servers generate self-signed RSA 4096-bit certificates in memory when no `--cert-path` is specified, with optional hot-reloading via fsnotify for production deployments. The body chunking implementation conservatively limits chunks to 62KB (below Envoy's 64KB limit) to avoid edge cases. The `InferenceModelRewrite` CRD enables sophisticated traffic management with weighted model name rewriting, supporting canary rollouts and A/B testing at the inference layer without requiring changes to client applications.
