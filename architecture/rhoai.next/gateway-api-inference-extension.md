## Purpose

**Short**: Extends any Gateway API-compatible proxy into an inference gateway by providing intelligent, KV-cache-aware endpoint selection and request routing for self-hosted generative AI models on Kubernetes.

**Detailed**: Gateway API Inference Extension (IGW) implements Envoy's External Processing (ext-proc) protocol to intercept inference requests at the proxy layer and make optimal routing decisions. It consists of two primary components: the Endpoint Picker (EPP), which performs request scheduling with pluggable scoring, filtering, and flow control; and the Body-Based Router (BBR), which parses HTTP request bodies to extract model names and maps LoRA adapters to base models for upstream routing.

The EPP watches Kubernetes CRDs (InferencePool, InferenceObjective, InferenceModelRewrite) and Pod resources to maintain an in-memory datastore of available model server endpoints and their metrics. It scrapes Prometheus metrics from model server pods (KV cache utilization, queue depth, running requests, LoRA adapter status) and uses this data to score endpoints via a pluggable scheduling framework. The scheduling pipeline supports filters (prefix cache affinity, SLO headroom), scorers (KV cache utilization, queue depth, LoRA affinity, predicted latency), and pickers (max score, random, weighted random). An experimental flow control layer provides priority-based queuing with fairness, eviction, and saturation detection.

The project originated as a Kubernetes SIG project (kubernetes-sigs/gateway-api-inference-extension) and is integrated into RHOAI as part of the llm-d inference stack. It supports multiple gateway implementations (Envoy Gateway, kgateway, GKE Gateway, Istio, nginx Gateway Fabric) and model servers (vLLM, SGLang, Triton TensorRT-LLM, trtllm-serve).

## Architectural Analysis

The Gateway API Inference Extension follows a sophisticated plugin-based architecture that separates concerns across multiple extension points. The EPP is essentially a scheduler for inference requests, drawing design inspiration from the Kubernetes scheduler's plugin framework. The scheduling pipeline operates in three phases: filtering (eliminating unsuitable endpoints), scoring (ranking remaining endpoints by weighted criteria), and picking (selecting the best endpoint from scored candidates). This design enables operators to customize routing behavior without modifying core code — for example, the llm-d project extends the scheduling framework with disaggregated serving plugins.

The ext-proc integration pattern is architecturally significant. Rather than implementing a full proxy, IGW relies on Envoy's ext-proc filter to intercept requests at the gateway layer. This means the EPP never handles actual inference traffic — it only receives request/response metadata and body content via gRPC streams, returning routing decisions (target endpoint headers, body mutations). The gateway itself handles the actual data plane forwarding. This separation keeps the EPP stateless from a traffic perspective while maintaining full control over routing decisions. The BBR operates similarly but earlier in the filter chain, focusing solely on body parsing and header injection.

The flow control subsystem is a particularly complex piece of architecture. It implements a Join-Shortest-Queue-by-Bytes (JSQ-Bytes) distribution strategy with sharded processing, just-in-time provisioning of priority bands and flows, pluggable fairness and ordering policies, and an eviction system for overload protection. The flow control registry uses a dual-concurrency model: lock-free atomics for the hot path (request processing) and mutex-based locking for the cold path (topology changes). Flows and priority bands are provisioned on-demand rather than pre-configured, and garbage-collected when idle. This design scales to many concurrent flows without paying the cost of pre-allocating infrastructure for flows that may never appear.

The data layer architecture underwent a significant redesign from a polling-based metrics collector to a push-based pluggable system. The old approach directly scraped model server Prometheus endpoints at fixed intervals. The new data layer uses a source/extractor pattern where data sources (HTTP metrics, Kubernetes notifications, endpoint notifications) feed data to extractors that parse and normalize it into the scheduling framework's endpoint metadata model. This supports custom data sources — for example, the llm-d project adds a notification-based data source that receives metrics pushed from the model server rather than polling them.

One notable security observation is that the model server metrics scraping defaults to `InsecureSkipVerify=true` for HTTPS connections. This is pragmatic for in-cluster communication where model servers may use self-signed certificates, but it represents a potential concern in environments with strict mTLS requirements. The EPP's own gRPC endpoint uses server-side TLS with either self-signed certificates (generated at startup with 4096-bit RSA keys and 10-year validity) or operator-provided certificates with optional file-watcher-based reloading. No client certificate verification is performed — the assumption is that only the co-located gateway proxy connects to the ext-proc service.
