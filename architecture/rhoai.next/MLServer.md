## Purpose

**Short**: MLServer is a Python-based multi-model inference server implementing the KServe v2 dataplane protocol, serving machine learning models over REST, gRPC, and Kafka interfaces with a pluggable runtime system.

**Detailed**: MLServer provides a production-grade inference serving platform that implements the open KServe v2 inference protocol (also known as the Open Inference Protocol). It exposes three concurrent server interfaces — a FastAPI-based REST server on port 8080, a gRPC server on port 8081, and an optional Kafka consumer/producer — all routing requests through a unified DataPlane handler. The server supports multi-model serving with version management, adaptive request batching, response caching, and multiprocess parallel inference execution via worker pools.

The runtime system is plugin-based: model-specific inference logic is provided by interchangeable runtime packages (e.g., sklearn, xgboost, lightgbm, onnx) that extend the `MLModel` base class. In RHOAI, the shipped image includes four runtimes (sklearn, xgboost, lightgbm, onnx) locked to a trusted-runtimes allowlist at `/etc/mlserver/trusted-runtimes.json` to prevent arbitrary code execution. The server also ships a codec system for automatic type conversion between wire formats (JSON, protobuf, raw bytes) and Python types (numpy arrays, pandas DataFrames, strings, datetime), and comprehensive observability via Prometheus metrics, OpenTelemetry tracing, and structured JSON logging.

MLServer is deployed as a container image within KServe `InferenceService` pods. It is not deployed as a standalone Kubernetes workload — KServe's controller manages the pod lifecycle, networking (via Knative or raw Kubernetes), and autoscaling. The MLServer container reads model artifacts from a models directory (typically `/mnt/models`, populated by a storage initializer init container) and begins serving immediately on startup.

### Trusted Runtime Security

MLServer implements a runtime trust model to prevent arbitrary code execution in production:

- **Production mode** (trusted-runtimes.json exists at /etc/mlserver/): Only runtimes listed in the allowlist can be loaded. Custom environment_tarball and environment_path are blocked. Wildcard CORS origins and CORS regex patterns are blocked. Runtime import paths are validated at both load time and access time.
- **Development mode** (trusted-runtimes.json absent): Any runtime can be loaded dynamically. Custom environments are supported. Wildcard CORS is allowed.

## Architectural Analysis

MLServer's architecture follows a clean separation between protocol adapters and inference logic. The three server interfaces (REST, gRPC, Kafka) are thin protocol-specific adapters that all converge on the same `DataPlane` handler. This design means adding a new protocol (e.g., WebSocket) requires only a new adapter — the inference, batching, caching, and model management logic is shared. The REST server is built on FastAPI with Uvicorn, providing both high-performance async I/O and automatic OpenAPI documentation. The gRPC server uses `grpc.aio` with interceptors for metrics and tracing. The Kafka integration uses aiokafka for fully async message processing.

The runtime plugin system is MLServer's most distinctive architectural pattern. Each runtime (sklearn, xgboost, etc.) is a separate Python package that extends `MLModel` and implements `load()` and `predict()`. The server discovers runtimes via Python's import system — the `implementation` field in model-settings.json specifies a dotted import path (e.g., `mlserver_sklearn.SKLearnModel`). In production (RHOAI), this is locked down by the trusted-runtimes mechanism: a JSON allowlist baked into the container image at `/etc/mlserver/trusted-runtimes.json` that blocks any unlisted runtime from loading. This is a critical security boundary — without it, a malicious model-settings.json could execute arbitrary Python code. The Dockerfile.konflux generates this allowlist from the `TRUSTED_RUNTIMES` build arg, validating each import path at build time.

The adaptive batching subsystem (`mlserver/batching/`) implements a time-and-size-bounded request aggregation strategy. When `max_batch_size > 1` and `max_batch_time > 0` are configured on a model, the `AdaptiveBatcher` queues incoming inference requests and waits until either the batch is full or the time window expires, then merges all inputs into a single `InferenceRequest`, executes a single `predict()` call, and splits the response back into individual replies. The merge/split logic in `BatchedRequests` handles shape tracking per minibatch to correctly partition output tensors. This is essential for GPU-backed runtimes where batched inference is dramatically more efficient than sequential calls.

The parallel execution system (`mlserver/parallel/`) provides multiprocess inference via worker pools. Each worker is a `multiprocessing.Process` running its own asyncio event loop with uvloop. Communication uses `multiprocessing.Queue` with a custom message protocol (`ModelRequestMessage`/`ModelResponseMessage`). The `Dispatcher` uses round-robin scheduling to distribute requests across workers. The `InferencePoolRegistry` supports environment-based pool isolation — models with different Python environments (via conda tarballs or custom paths) get separate worker pools, identified by environment hash. This enables serving models with conflicting dependencies in the same MLServer instance.

The codec system deserves architectural attention for its sophistication. Rather than requiring runtimes to manually serialize/deserialize inputs, MLServer provides a registry of `InputCodec` and `RequestCodec` implementations that handle conversion between wire formats and Python types. The `@decode_args` decorator on `predict()` methods enables fully transparent type conversion based on Python type hints — a method annotated `async def predict(self, data: np.ndarray) -> np.ndarray` will automatically decode inputs from the v2 protocol format and encode outputs back. This reduces runtime implementation to pure inference logic with no protocol awareness.

One notable security consideration: MLServer binds all three ports (8080, 8081, 8082) without TLS and without authentication. This is by design — TLS termination and auth are delegated to the platform layer (kube-rbac-proxy sidecars in RHOAI 3.x, Istio mTLS, or KServe's networking layer). However, the gRPC server explicitly uses `add_insecure_port()` (mlserver/grpc/server.py:84), meaning there is no option to enable TLS at the application level for gRPC, even in environments where platform-level TLS is not available. For REST, Uvicorn theoretically supports TLS via ssl_keyfile/ssl_certfile, but MLServer does not expose these settings.
