## Purpose

**Short**: MLServer is an inference server implementing the Open Inference Protocol (V2 Dataplane API) that serves machine learning models over REST and gRPC with a pluggable runtime architecture.

**Detailed**: MLServer is a production-ready ML inference server originally developed by Seldon Technologies and forked by Red Hat for use in Red Hat OpenShift AI (RHOAI). It implements the KServe V2 Inference Protocol (also known as the Open Inference Protocol), providing standardized REST (HTTP) and gRPC endpoints for model inference, health checking, model metadata, and model repository management.

The server features a pluggable runtime architecture where different ML frameworks (scikit-learn, XGBoost, LightGBM, ONNX Runtime, HuggingFace Transformers, MLflow, CatBoost, Alibi Detect, Alibi Explain, and Spark MLlib) are loaded as separate runtime modules. The RHOAI Konflux build ships a curated subset of runtimes (LightGBM, ONNX, scikit-learn, XGBoost) with a trusted-runtimes allowlist enforced at startup in PRODUCTION mode, preventing arbitrary code execution via untrusted runtime import paths.

MLServer also provides adaptive batching (time-based and size-based), response caching, parallel inference via process pools, OpenTelemetry tracing, Prometheus metrics export, Kafka event streaming, custom model handlers, and Server-Sent Events (SSE) for streaming inference. It runs as a containerized service deployed by KServe's `InferenceService` CRD within the RHOAI platform.

## Architectural Analysis

MLServer follows a **dual-protocol, plugin-based architecture** where the server core provides standardized V2 Dataplane API endpoints over both REST (FastAPI/Uvicorn on port 8080) and gRPC (grpcio on port 8081), with full feature parity between both protocols. All inference requests flow through a shared `DataPlane` handler layer, ensuring consistent behavior regardless of protocol. The `InferenceMiddleware` chain (including CloudEvents processing) is applied uniformly before reaching model-specific predict logic. This design means that runtime plugins only need to implement the `MLModel` abstract base class (`load()`, `predict()`, optionally `predict_stream()`) and the server handles protocol translation, batching, caching, and metrics automatically.

The **runtime security model** is a critical RHOAI hardening feature. In PRODUCTION mode (detected by the presence of `/etc/mlserver/trusted-runtimes.json`, baked into the container image at build time), MLServer enforces a strict allowlist of model implementation import paths. This prevents users from loading arbitrary Python code via `model-settings.json`. The Konflux Dockerfile generates this file during the build, restricting to only the four shipped runtimes (LightGBM, ONNX, scikit-learn, XGBoost). DEVELOPMENT mode disables this check and additionally allows wildcard CORS origins and custom Python environments — both blocked in PRODUCTION mode via Pydantic validators. The `/v2/runtimes` endpoint and `RuntimeSecurity` gRPC RPC expose the current mode and allowlist, enabling external monitoring of the security posture.

The **adaptive batching** system uses an `AdaptiveBatcher` per model that collects incoming inference requests into a queue, flushing either when `max_batch_size` items are collected or when `max_batch_time` seconds elapse. The `BatchedRequests` class merges individual `InferenceRequest` objects by concatenating tensor inputs along the batch dimension, invokes the model once on the batched tensor, and then splits the response back into individual responses. This is transparent to the model runtime — the model sees a single batched request. The **parallel inference** system spawns worker processes via `InferencePool`, each loading the model independently, with a round-robin `Dispatcher` distributing requests across workers via `multiprocessing.Queue`. Workers can optionally run in isolated Python environments (extracted from conda-pack tarballs), enabling per-model dependency isolation.

The **observability stack** includes three pillars: OpenTelemetry tracing (configured via `tracing_server` setting, exports to a collector over insecure gRPC on port 4317), Prometheus metrics (via `starlette-exporter` for REST and `py_grpc_prometheus` for gRPC, exposed on a dedicated metrics server on port 8082), and structured JSON logging (optional, via `use_structured_logging`). Health endpoint access logs are filtered out to reduce noise. The metrics server runs on its own Uvicorn instance to avoid interference with the inference server. Context variables (`model_name_var`, `model_version_var`) propagate model identity through the async call stack for accurate per-model metric labeling.

The **codec system** provides a registry-based approach to type conversion between Python objects and the Open Inference Protocol tensor format. Codecs for NumPy arrays, Pandas DataFrames, Base64 bytes, strings, JSON, datetime, and lists are registered centrally. Runtimes declare their preferred input/output codecs (e.g., `NumpyRequestCodec` for scikit-learn, `PandasCodec` for MLflow) and the server automatically handles serialization. The decorator-based `@register_input_codec` and `@register_request_codec` patterns allow runtimes to extend the codec registry with custom implementations. The codec system is complemented by content-type negotiation via the `content_type` parameter in model settings.
