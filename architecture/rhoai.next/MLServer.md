## Purpose

**Short**: MLServer is a Python-based machine learning inference server implementing the V2 Inference Protocol (KFServing / Open Inference Protocol) over both REST and gRPC, used as the default multi-model serving runtime in RHOAI for traditional ML frameworks.

**Detailed**: MLServer serves as the inference backbone for traditional machine learning models within the Red Hat OpenShift AI platform. It provides a standards-compliant inference API (V2/Open Inference Protocol) exposed via both REST (FastAPI/Uvicorn on port 8080) and gRPC (grpcio on port 8081), with an additional dedicated Prometheus metrics server on port 8082. The server supports multi-model serving, where multiple models can be loaded, versioned, and served simultaneously from a single process or across parallel worker processes.

The server ships with a pluggable runtime architecture, where each ML framework (scikit-learn, XGBoost, LightGBM, ONNX, etc.) is packaged as a separate Python package that implements the `MLModel` abstract class. The RHOAI Konflux image bundles four default runtimes (sklearn, xgboost, lightgbm, onnx) and enforces a trusted-runtimes allowlist at `/etc/mlserver/trusted-runtimes.json` to restrict which model implementations can be loaded in production mode. This security mechanism prevents arbitrary code execution through untrusted model implementations.

MLServer also provides adaptive batching (grouping multiple inference requests into a single prediction call), parallel inference via multiprocessing workers (to bypass the Python GIL), response caching, OpenTelemetry tracing, CloudEvents middleware for event-driven integration, and optional Kafka-based inference for asynchronous workloads. In the RHOAI platform, MLServer containers run behind a KServe InferenceService, which handles ingress routing, scaling, and lifecycle management.

## Architectural Analysis

MLServer follows a layered architecture with clear separation between protocol handling (REST/gRPC/Kafka), business logic (DataPlane handlers), and model runtime execution (MLModel abstract class). The DataPlane class serves as the central orchestration point, shared by all three server types (REST, gRPC, Kafka). This shared handler pattern ensures consistent behavior regardless of which protocol a client uses, including identical metrics counters (`model_infer_request_success`, `model_infer_request_failure`, `model_infer_request_duration`) and CloudEvents middleware applied uniformly.

The parallel inference architecture is particularly noteworthy. To work around Python's GIL, MLServer spawns separate worker processes via the `multiprocessing` module, each running its own asyncio event loop with uvloop. The main process dispatches inference requests to workers via `multiprocessing.Queue` and collects responses asynchronously through a `Dispatcher` that monitors response queues in a background thread. Workers are self-healing: if a worker process dies unexpectedly, the `InferencePool` detects it, spawns a replacement, and replays all model load operations to restore the worker to a consistent state. The `InferencePoolRegistry` manages multiple pools, each potentially with its own Python environment (via `conda-pack` tarballs), enabling heterogeneous model serving in a single server instance.

The trusted-runtimes security model is a defense-in-depth approach specific to the RHOAI deployment. At image build time, the Dockerfile generates `/etc/mlserver/trusted-runtimes.json` containing an explicit allowlist of model implementation import paths (e.g., `mlserver_sklearn.SKLearnModel`). When this file exists, the server operates in "PRODUCTION" mode: any attempt to load a model whose implementation is not in the allowlist is rejected with a `ValueError`. This validation happens at three points: during `ModelSettings` Pydantic validation (via `model_validator`), at the `implementation` property accessor (defense-in-depth against post-validation mutation), and during initial startup security check (`log_runtime_security_mode`). In production mode, custom environments (`environment_tarball`, `environment_path`) and wildcard CORS origins are also blocked, preventing sys.path injection attacks and overly permissive CORS. The import path itself is validated against a strict regex pattern requiring explicit dotted module paths with capitalized class names.

The adaptive batching system (`AdaptiveBatcher`) implements a time-window/size-limit batching strategy. When enabled on a per-model basis via `max_batch_size` and `max_batch_time` settings, individual inference requests are queued and merged into a single `BatchedRequests` object. The merged request is passed to `model.predict()`, and the response is then split back into individual responses matched by internal UUIDs. This is transparent to clients and can significantly improve throughput for models that benefit from batched computation (e.g., GPU-backed models).

One architectural consideration is that MLServer itself provides no TLS or authentication. All three listening ports (8080, 8081, 8082) use plaintext protocols. This is by design: in the RHOAI deployment topology, MLServer runs inside a KServe InferenceService pod where TLS termination and authentication are handled at the platform ingress layer (via Istio/Envoy sidecar or Gateway API with kube-rbac-proxy). The OTLP trace exporter is also configured with `insecure=True` by default, assuming the collector is co-located or reachable within the cluster network. This trust-the-network approach is standard for sidecar-proxied workloads but should be documented for security auditors.
