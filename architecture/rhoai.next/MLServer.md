## Purpose

**Short**: MLServer is a multi-model inference server implementing the KServe V2 Inference Protocol (dataplane), serving ML models over REST (HTTP) and gRPC with pluggable runtime backends.

**Detailed**: MLServer provides a production-grade model serving platform that exposes ML models via both REST (FastAPI/Uvicorn) and gRPC protocols simultaneously, following the KServe V2 dataplane specification. It supports multiple ML frameworks through a pluggable runtime system — the RHOAI distribution ships runtimes for scikit-learn, XGBoost, LightGBM, and ONNX. Additional runtimes (HuggingFace, MLflow, Alibi-Explain, Alibi-Detect, CatBoost) exist in the upstream codebase but are not included in the RHOAI container image.

The server features adaptive request batching, in-memory response caching, multiprocessing-based parallel inference workers, Prometheus metrics collection on a dedicated port, OpenTelemetry distributed tracing, optional Kafka-based streaming inference, and CloudEvents metadata injection for Kubernetes-native event-driven architectures. A trusted runtime allowlist enforced via `/etc/mlserver/trusted-runtimes.json` restricts which model implementations can load in production mode, preventing arbitrary code execution through model configuration.

MLServer is deployed as a container within KServe `InferenceService` pods on Red Hat OpenShift AI, where it serves as the inference runtime for traditional ML models. The platform operator (KServe) manages its lifecycle, routing, and scaling.

## Architectural Analysis

MLServer's architecture follows a layered, dual-protocol design centered around the KServe V2 dataplane specification. The core server (`mlserver/server.py`) orchestrates four independent protocol servers — REST (FastAPI/Uvicorn on port 8080), gRPC (grpc.aio on port 8081), metrics (Prometheus HTTP on port 8082), and optionally Kafka — all sharing a single `DataPlaneHandlers` instance that implements the business logic. This shared-handler pattern ensures consistent behavior across protocols while allowing protocol-specific middleware (CORS, gzip for REST; interceptors for gRPC).

The runtime plugin system is MLServer's most architecturally significant feature. Each ML framework (sklearn, xgboost, lightgbm, onnx, etc.) is a separate Python package that implements the `MLModel` abstract base class with `load()`, `predict()`, and `unload()` lifecycle methods. In production mode, a trusted runtime allowlist baked into the container image at `/etc/mlserver/trusted-runtimes.json` restricts which import paths can be loaded. This is a critical security boundary — without it, an attacker who can control model configuration files could execute arbitrary Python code. The validation occurs early in the settings parsing pipeline (`ModelSettings.validate_trusted_runtime()` at `settings.py:642-649`), blocking untrusted runtimes before any model artifacts are touched. Production mode also disables custom environment tarballs, dynamic sys.path injection, and wildcard CORS — a defense-in-depth approach that trades development flexibility for security.

The parallel inference subsystem uses Python multiprocessing to scale model inference across worker processes. Each `InferencePool` manages a pool of worker processes, with the `Dispatcher` implementing round-robin request distribution via `multiprocessing.Queue`. Workers run independent `MultiModelRegistry` instances, meaning each worker loads its own copy of the model. Environment isolation is supported — each pool can run in a separate Python environment (extracted from tarballs), enabling per-model dependency isolation. This design enables horizontal scaling within a single pod but has significant memory implications, as model weights are duplicated across workers.

The codec system (`mlserver/codecs/`) provides a type-safe bridge between the KServe V2 wire format (tensor-based) and high-level Python types (numpy arrays, pandas DataFrames, strings, bytes). The `@decode_args` decorator enables runtime methods to declare their expected input/output types via Python type hints, and the codec system automatically handles serialization. This is a clean separation of concerns — runtime implementors focus on model logic while the codec layer handles protocol translation.

The RHOAI distribution makes specific choices about which runtimes to include in the container image. Only sklearn, xgboost, lightgbm, and onnx are shipped (as defined in `Dockerfile.konflux` line 2 and `hack/requirements-config.json`). This is a deliberate constraint — these four runtimes cover the most common traditional ML use cases while keeping the image size manageable. The `requirements/requirements-cpu.txt` file pins all transitive dependencies with SHA-256 hashes against the RHEL AI PyPI, providing supply chain integrity even though the Dockerfile.konflux uses a simpler `pip install` approach that relies on the AIPCC base image's pre-configured index.
