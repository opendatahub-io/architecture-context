## Purpose

**Short**: MLServer is a multi-model inference server implementing the V2 Inference Protocol (KServe/Open Inference Protocol) over REST and gRPC, supporting pluggable ML runtimes for serving scikit-learn, XGBoost, LightGBM, ONNX, and other model frameworks.

**Detailed**: MLServer provides a production-grade model serving infrastructure for Red Hat OpenShift AI. It implements the KServe V2 Inference Protocol (also known as the Open Inference Protocol) which defines a standardized REST and gRPC API for model inference, health checking, model metadata, and model repository management. The server runs as a standalone Python process inside a container, listening on three ports: HTTP/REST (8080), gRPC (8081), and Prometheus metrics (8082).

The architecture follows a pluggable runtime pattern where ML framework-specific model loading and prediction logic is encapsulated in runtime modules (e.g., `mlserver-sklearn`, `mlserver-xgboost`). Each runtime inherits from the `MLModel` base class and implements `load()` and `predict()` methods. The server supports multi-model serving, adaptive request batching, parallel inference via multiprocessing worker pools, and streaming inference via Server-Sent Events (REST) and bidirectional streaming (gRPC).

For RHOAI deployments, MLServer runs in production mode with a trusted runtimes allowlist baked into the container image at `/etc/mlserver/trusted-runtimes.json`. This prevents loading of arbitrary Python code via model settings — only pre-approved runtime import paths can be instantiated. The RHOAI image ships four runtimes: scikit-learn, XGBoost, LightGBM, and ONNX. The server also supports optional Kafka-based async inference, OpenTelemetry distributed tracing, and CloudEvents middleware for event-driven architectures.

## Architectural Analysis

MLServer's architecture is built around a clean separation between the protocol layer and the ML framework layer. The protocol layer (REST via FastAPI, gRPC via grpc.aio) handles serialization, routing, health checks, and metrics, while the ML framework layer is encapsulated in pluggable runtime modules that inherit from the `MLModel` base class. This separation allows the same server binary to serve models from completely different ML frameworks without any framework-specific code in the core server.

The trusted runtimes security model is a notable design decision for production hardening. Rather than relying on network-level isolation alone, MLServer implements an application-level allowlist at `/etc/mlserver/trusted-runtimes.json`. This file is generated at container image build time (in both `Dockerfile` and `Dockerfile.konflux`) and contains the exact Python import paths that are permitted to be instantiated. In production mode (when this file exists), any attempt to load a model with an implementation not in the allowlist is rejected with a clear error. This defense-in-depth approach prevents code injection through malicious `model-settings.json` files that could specify arbitrary Python classes. The system also validates import paths syntactically (must be dotted, uppercase class name, no leading underscores) and blocks custom `environment_tarball` and `environment_path` in production mode to prevent sys.path injection and arbitrary tarball extraction.

The parallel inference subsystem uses Python multiprocessing with a round-robin dispatcher pattern. Each worker process runs its own async event loop (with uvloop) and maintains independent copies of loaded models. Communication between the main process and workers happens via multiprocessing Queues with a future-based response correlation mechanism. Worker failures are detected and trigger automatic restart with model reloading. Prometheus metrics are shared across processes via filesystem-based multiprocess collection in a configurable metrics directory.

The Dockerfile.konflux differs significantly from the standard Dockerfile. The standard Dockerfile uses a multi-stage build with Poetry-based wheel compilation from source, while the Konflux Dockerfile uses a pre-built RHOAI base image (`quay.io/aipcc/base-images/cpu:3.5.0-ea.1`) and installs MLServer and runtimes via `pip install` from packages pre-baked in the base image. This implies that the Konflux/RHOAI supply chain pre-builds and vets all Python packages before they reach the MLServer image build, adding another layer of supply chain security. The RHOAI image ships only four runtimes (sklearn, xgboost, lightgbm, onnx) compared to the full set of nine available in the upstream project.

The server exposes no authentication or TLS termination natively — the gRPC server explicitly uses `add_insecure_port()` and the REST server runs Uvicorn without SSL configuration. This is by design: in RHOAI deployments, authentication (via kube-rbac-proxy sidecar or Istio) and TLS termination (at the Gateway/Route level) are handled by the platform infrastructure. MLServer is intended to sit behind these layers, never directly exposed to external traffic. CORS settings include a production-mode validator that blocks wildcard origins when the trusted runtimes file is present.
