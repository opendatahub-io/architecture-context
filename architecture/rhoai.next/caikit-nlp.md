## Purpose

**Short**: A Python library providing NLP capabilities (text generation, embeddings, reranking, classification) built on top of the caikit AI framework.

**Detailed**: Caikit-NLP is a modular Python library that extends the [caikit](https://github.com/caikit/caikit) framework with Natural Language Processing capabilities. It implements caikit's "task" and "module" abstractions to provide a consistent interface for various NLP operations including text generation (via local HuggingFace models or remote TGIS inference servers), text embeddings (via sentence-transformers), document reranking (via both bi-encoder and cross-encoder approaches), sentence similarity, text classification, and tokenization.

The library is not deployed as a standalone service. Instead, it is packaged as a Python wheel and installed into the `caikit` runtime container image, which exposes the modules' capabilities through auto-generated HTTP (port 8080) and gRPC (port 8085) APIs. The caikit runtime discovers and loads caikit-nlp modules via the `RUNTIME_LIBRARY=caikit_nlp` environment variable, at which point all registered task methods become available as API endpoints.

Caikit-NLP supports two model execution backends: **local** (loading HuggingFace transformers models directly into GPU/CPU memory for inference and fine-tuning) and **remote TGIS** (delegating inference to a Text Generation Inference Server via gRPC). The TGIS backend enables prompt tuning workflows where lightweight prompt vectors are trained locally and then deployed alongside a shared base model running in TGIS. The library also provides a `TGISAutoFinder` model finder that enables automatic discovery and connection to remote TGIS-hosted models.

## Architectural Analysis

Caikit-NLP follows the caikit framework's modular architecture pattern where each NLP capability is implemented as a `@module` class that declares its supported `tasks` and exposes `@taskmethod` decorated functions. The caikit runtime auto-generates HTTP and gRPC APIs from these task methods, meaning caikit-nlp never directly handles network I/O — it is purely a library of NLP modules.

The dual-backend design (local vs. TGIS) is the most architecturally significant pattern. For text generation, each module has a "local" variant (`TextGeneration`, `PeftPromptTuning`) that loads HuggingFace models into memory and a "TGIS" variant (`TextGenerationTGIS`, `PeftPromptTuningTGIS`) that delegates to a remote TGIS server via gRPC. The TGIS variants use the `@module(backend_type=TGISBackend.backend_type, base_module=...)` decorator to register as backend-specific alternatives to their local counterparts. This allows the caikit runtime to transparently select the appropriate module based on the configured backend. The `TGISAutoFinder` extends this further by enabling model discovery — when a model path doesn't match a local model, it checks if TGIS can serve it and auto-configures a `TextGenerationTGIS` module.

The embedding modules (`EmbeddingModule`, `CrossEncoderModule`) use a different approach: they wrap `sentence-transformers` models with custom `SentenceTransformerWithTruncate` and `CrossEncoderWithTruncate` classes that add sophisticated token truncation handling and thread-safe tokenization. A notable implementation detail is the per-thread tokenizer cloning (`self.tokenizers` dict keyed by `threading.get_ident()`) to avoid "Already borrowed" errors when the fast tokenizer is used concurrently with different truncation settings. This is critical for production serving where multiple requests are processed simultaneously.

The training pipeline supports multi-GPU distributed training via PyTorch's `elastic_launch` (torchrun) for full fine-tuning (`TextGeneration.train()`), while prompt tuning (`PeftPromptTuning.train()`) uses HuggingFace Accelerate for mixed-precision training with gradient accumulation. The prompt tuning flow produces lightweight prompt vectors (encoder.pt / decoder.pt files) that can be loaded into TGIS for serving alongside a shared base model — enabling the n-prompts-to-1-base-model pattern that is economically efficient for multi-tenant deployments.

The container image (`Dockerfile.konflux`) is built on `ubi9/ubi-minimal` and uses a multi-stage build: first building the Python wheel via tox, then installing it into a virtual environment in the deploy stage. The container runs as a non-root user (caikit, UID 1001) and does not set `CMD` to start the caikit runtime — instead it defaults to `CMD ["python"]`, indicating the image is designed to be used as a base or with an explicit command override by the platform operator's ServingRuntime configuration.
