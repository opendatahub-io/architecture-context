## Purpose

**Short**: Caikit-NLP is a Python library providing NLP capabilities (text generation, embeddings, reranking, classification, tokenization) built on top of the Caikit AI framework, serving as a runtime module for RHOAI model serving.

**Detailed**: Caikit-NLP implements the NLP domain modules for the Caikit AI runtime framework. It provides a comprehensive set of NLP capabilities including text generation (local and remote via TGIS), prompt tuning (PEFT-based), text embeddings, cross-encoder reranking, sentence similarity, text classification, token classification, and tokenization. Each capability is implemented as a Caikit module that can be loaded, trained, saved, and served through Caikit's gRPC and HTTP runtime servers.

The library integrates with two primary inference backends: local PyTorch-based inference using HuggingFace Transformers models, and remote inference via the Text Generation Inference Server (TGIS) gRPC backend. For fine-tuning, it supports PEFT prompt tuning with multi-GPU distributed training via PyTorch elastic launch and FSDP. The embedding modules leverage sentence-transformers with optional Intel IPEX and PyTorch compile optimizations. When deployed as part of RHOAI, caikit-nlp is packaged as a container image (`odh-caikit-nlp-rhel9`) and loaded as the `RUNTIME_LIBRARY` in the Caikit runtime server, which exposes model inference via gRPC (port 8085) and an optional REST gateway (port 8080).

## Architectural Analysis

Caikit-NLP follows the Caikit framework's module pattern, where each NLP capability is encapsulated as a decorated `@module` class with standard lifecycle methods (`load`, `save`, `train`, `run`). This provides a uniform interface for model management and serving, allowing the Caikit runtime to automatically expose gRPC and HTTP endpoints for each registered task. The library registers 9 distinct modules across 5 NLP domains (text generation, text embedding, text classification, token classification, tokenization), each mapped to one or more Caikit tasks via `@taskmethod` decorators.

The most architecturally significant pattern is the local/remote duality in text generation. Each generation module has a "local" variant (`TextGeneration`, `PeftPromptTuning`) that loads HuggingFace models directly into GPU memory, and a "TGIS" variant (`TextGenerationTGIS`, `PeftPromptTuningTGIS`) that delegates inference to a remote Text Generation Inference Server via gRPC. The TGIS variants are registered using `backend_type=TGISBackend.backend_type` and share the same task interface, allowing transparent switching between local and remote inference at deployment time. The `TGISAutoFinder` model finder automatically discovers models available via TGIS connections, enabling dynamic model routing.

The PEFT prompt tuning implementation (`PeftPromptTuning`) is the only module that supports training within caikit-nlp. It stores trained prompt vectors (encoder and/or decoder) as separate artifacts that can be loaded independently of the base model. When used with TGIS (`PeftPromptTuningTGIS`), prompt artifacts are uploaded to the TGIS server's prompt cache, allowing efficient inference without modifying the base model weights. Training supports multi-GPU distribution via PyTorch elastic launch with Fully Sharded Data Parallel (FSDP) for large models.

The embedding stack (`EmbeddingModule`, `CrossEncoderModule`) is completely local — there is no remote backend integration for embeddings. These modules support optional performance optimizations including Intel IPEX, PyTorch `torch.compile()`, and autocast (bfloat16). Thread-safe tokenization is achieved by creating per-thread copies of the tokenizer. The embedding module registers the most tasks of any module (7 tasks: embedding, batch embeddings, similarity, batch similarity, rerank, batch rerank, tokenization), making it the most versatile component.

Security in caikit-nlp is deliberately minimal at the library level. TLS configuration for the gRPC and HTTP servers is handled via environment variables (`RUNTIME_TLS_SERVER_KEY`, `RUNTIME_TLS_SERVER_CERT`, `RUNTIME_TLS_CLIENT_CERT`), and TGIS backend connections support optional mTLS via config file paths (`ca_cert_file`, `client_cert_file`, `client_key_file`). Model download from HuggingFace Hub is disabled by default (`allow_downloads: false`) as a security measure, requiring explicit opt-in. In production RHOAI deployments, authentication is enforced externally by `kube-rbac-proxy` sidecars rather than within the library itself.
