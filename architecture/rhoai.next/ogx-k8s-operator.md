## Purpose

**Short**: Kubernetes operator that manages the lifecycle of OGX (formerly Llama Stack) AI distribution servers on OpenShift and Kubernetes clusters.

**Detailed**: The OGX K8s Operator is a controller-runtime based Kubernetes operator that automates the deployment, configuration, and lifecycle management of OGX server instances. Each OGXServer custom resource declaratively specifies a complete AI server stack: the distribution image to deploy, provider configurations for inference (vLLM, OpenAI, Azure, Bedrock, Vertex AI, watsonx), vector I/O (pgvector, Milvus, Qdrant), tool runtimes (Brave Search, Tavily, MCP), file storage (S3, local FS), batches, and responses APIs.

The operator renders Kubernetes manifests via an embedded kustomize pipeline, creating Deployments, Services, ServiceAccounts, PersistentVolumeClaims, NetworkPolicies, Ingresses, HorizontalPodAutoscalers, PodDisruptionBudgets, and RoleBindings per OGXServer instance. It manages TLS CA bundle aggregation (including auto-detection of ODH trusted CA bundles), operator-level image override configuration via ConfigMaps, and health/readiness probing of the deployed OGX server's `/v1/health`, `/v1/providers`, and `/v1/version` endpoints. The operator also supports legacy resource adoption from the previous LlamaStackDistribution (v1alpha1) CRD, enabling zero-downtime migration of PVCs, Services, and Ingresses to the new OGXServer (v1beta1) API.

The operator is distributed for both Open Data Hub (ODH) and Red Hat OpenShift AI (RHOAI) platforms, with kustomize overlays setting the target namespace (`opendatahub` vs `redhat-ods-applications`) and injecting platform-specific image references via `RELATED_IMAGE_*` environment variables.

## Architectural Analysis

The OGX K8s Operator follows a clean single-CRD-per-controller pattern, where one OGXServer CR maps to a complete set of Kubernetes resources for a single AI server instance. The operator uses an embedded kustomize pipeline rather than hardcoded Go resource construction, which keeps manifest templates in YAML (under `controllers/manifests/base/`) while using Go plugins for dynamic field injection, namespace setting, and NetworkPolicy transformation. This design makes it straightforward for platform teams to audit the exact resources created per instance.

The provider type system is notably well-structured. Rather than accepting arbitrary JSON configurations, the CRD defines typed provider specs for each supported backend (VLLMProvider, OpenAIProvider, BedrockProvider, PgvectorProvider, etc.) with per-field validation rules. This enables CEL-based admission validation at the API server level and provides clear schema documentation. The `CustomProvider` escape hatch allows users to configure providers not yet covered by typed specs, using a `remote::`/`inline::` prefix convention. Provider IDs are derived deterministically from types but can be overridden, and the webhook enforces global uniqueness across all provider categories.

The legacy adoption mechanism is architecturally interesting: it provides a migration path from the old `LlamaStackDistribution` (v1alpha1) CRD to the new `OGXServer` (v1beta1) without data loss. Annotations on the OGXServer CR (`ogx.io/adopt-storage`, `ogx.io/adopt-networking`) trigger a multi-step process: the operator scales down the legacy Deployment, waits for pods to terminate (to release RWO PVC locks), strips the old controller ownerRef from the PVC, and labels it for discovery by the new controller. Networking adoption (Service and Ingress) similarly transfers ownerRefs. Safety checks prevent same-name adoption (which would cause Deployment name conflicts) and verify that the existing controller ownerRef matches the expected legacy LLSD type.

The CA bundle management subsystem demonstrates defense-in-depth: certificates are PEM-decoded, validated as X.509 via `x509.ParseCertificate`, and re-encoded to strip non-certificate blocks (private keys, public keys). Size limits (10MB total, 1000 certificates max) prevent resource exhaustion. The operator auto-detects the well-known `odh-trusted-ca-bundle` ConfigMap that the ODH/RHOAI platform injects into component namespaces, merging it with any user-specified CA certificates into a single managed ConfigMap that is mounted into the OGX server pod.

The operator's cache configuration in `main.go` is carefully tuned: it uses label selectors to limit the informer cache to only resources managed by the operator (`app.kubernetes.io/managed-by: ogx-operator`), reducing memory pressure in namespaces with many unrelated resources. ConfigMaps are cached only when labeled `ogx.io/watch: true`, while user-referenced ConfigMaps without this label are read via a direct (non-cached) API client during reconciliation.
