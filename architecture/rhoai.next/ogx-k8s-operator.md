## Purpose

**Short**: Kubernetes operator that deploys and manages OGX (Open GenAI Stack) inference server instances via the OGXServer custom resource.

**Detailed**: The OGX K8s Operator manages the full lifecycle of OGX inference servers on Kubernetes and OpenShift clusters. It watches OGXServer custom resources (ogx.io/v1beta1) and reconciles a complete deployment stack for each instance: Deployment, Service, ServiceAccount, PersistentVolumeClaim, NetworkPolicy, PodDisruptionBudget, HorizontalPodAutoscaler, and Ingress. The operator supports a declarative provider configuration model where users specify inference providers (VLLM, OpenAI, Azure, Bedrock, etc.), vector stores, tool runtimes, and storage backends — the operator generates a complete config.yaml from these specifications.

The operator also includes a legacy adoption system for migrating from the predecessor LlamaStackDistribution CRD (llamastack.io/v1alpha1) to OGXServer. This allows zero-downtime migration of PVCs, Services, and Ingresses from old controller-managed resources to the new OGXServer controller. Distribution images are resolved through a layered system: embedded distributions.json → runtime ConfigMap overrides → per-CR image overrides.

The repository also contains `ogx-module`, a scaffolded ODH/RHOAI platform module operator that manages the OGX component CRD (components.platform.opendatahub.io/OGX). This module is the integration bridge between the ODH/RHOAI platform orchestrator and the OGX operator — it receives the platform-level OGX CR and deploys the OGX operator's kustomize manifests into the target namespace. The module is in early scaffolding and does not yet contain reconciler logic.

## Architectural Analysis

The OGX K8s Operator follows a kustomize-based manifest rendering pipeline that is both flexible and deterministic. Rather than constructing Kubernetes resources programmatically in Go structs, the operator maintains base YAML templates in `controllers/manifests/base/` and applies a sequence of five Go-based transformer plugins (NamePrefix, Namespace, FieldMutator, NetworkPolicyTransformer, and HPA replica removal) to customize them per-instance. The FieldMutator plugin uses RFC 6901 JSON Pointer paths to surgically set fields in the rendered manifests, avoiding the brittleness of string templating while keeping the base manifests readable. This design means adding new fields to the deployment requires only a new FieldMapping entry, not a rewrite of resource construction code.

The dual-mode configuration system is a notable design choice. Users can either specify providers declaratively (inference, vector I/O, tool runtimes, files, batches, file processors, responses) and have the operator generate config.yaml from those specs, or provide a complete override ConfigMap. The declarative mode resolves a base config from either a referenced ConfigMap or OCI image labels on the distribution container, then layers user-specified providers on top. Generated ConfigMaps use content-hash naming and are immutable, with a retention policy that keeps the 2 most recent plus any still referenced by in-flight rollouts. This prevents accidental data corruption and enables safe rollbacks.

The legacy adoption system for migrating from LlamaStackDistribution to OGXServer is carefully designed to handle the RWO PVC constraint. When adopting storage, the operator scales the legacy Deployment to zero replicas and waits for all pods to terminate before detaching the PVC — this prevents the multi-attach deadlock that would occur if both old and new Deployments tried to mount the same RWO volume simultaneously. Adopted PVCs intentionally do not get owner references, preventing data loss when the OGXServer CR is deleted. The adoption system also transfers Service and Ingress ownership, updating selectors to route traffic to new pods for a seamless cutover.

The NetworkPolicy design is thoughtful: the operator creates a default policy allowing ingress from same-namespace pods, operator-namespace pods, and OpenShift router pods. When users provide custom ingress or egress rules via the CR, the operator replaces default rules with custom ones and auto-injects kube-dns egress rules for proper DNS resolution. The entire NetworkPolicy feature can be disabled via a ConfigMap-based feature flag for clusters without network policy support. The operator uses Server-Side Apply (SSA) with field owner `ogx-operator` and force ownership to cleanly manage resource fields, with a full-replacement fallback for Deployments that have stale volumes from previous field managers.

The ogx-module sub-project represents the ODH/RHOAI platform integration pattern: a cluster-scoped OGX CRD (singleton, must be named "default-ogx") acts as the handshake between the platform orchestrator and the component operator. The module uses `get-manifests.sh` to fetch the OGX operator's kustomize manifests from a pinned commit SHA, ensuring reproducible deployments. The init container pattern copies these manifests from the operator image into an emptyDir volume, making them available to the manager at runtime. This architecture allows the OGX operator to be independently versioned while the module pins to specific operator versions per platform release.
