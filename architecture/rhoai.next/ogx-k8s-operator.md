## Purpose

**Short**: Kubernetes operator that deploys and manages OGX (Open GenAI Stack) server workloads including Deployments, Services, PVCs, NetworkPolicies, PDBs, HPAs, and Ingresses.

**Detailed**: The OGX Kubernetes Operator manages the full lifecycle of OGX server instances on Kubernetes and OpenShift clusters. It watches a single CRD (`OGXServer`, group `ogx.io`, version `v1beta1`) and reconciles the desired state by rendering kustomize-based manifests through a Go-based plugin pipeline, then applying them via server-side apply (SSA). The operator handles distribution image resolution (via embedded JSON, operator ConfigMap overrides, or direct image references), declarative config generation from provider/resource/storage specs, CA bundle aggregation (explicit and auto-detected ODH trusted CA bundles), persistent storage with PVC lifecycle management, network isolation via configurable NetworkPolicies, external access via Kubernetes Ingress resources, horizontal pod autoscaling, pod disruption budgets, and health checking via the OGX server's `/v1/health`, `/v1/providers`, and `/v1/version` endpoints.

The project was renamed from "LlamaStack" to "OGX". The legacy `LlamaStackDistribution` CRD (v1alpha1) is retained read-only for resource adoption: the controller supports annotation-driven migration of PVCs, Services, and Ingresses from legacy LlamaStackDistribution CRs to OGXServer CRs, including scaling down legacy deployments to release RWO PVCs before adoption.

## Architectural Analysis

The OGX operator follows a single-controller pattern where one `OGXServerReconciler` manages all resource types for each `OGXServer` CR. Rather than using separate controllers for Deployments, Services, NetworkPolicies, etc., the reconciler orchestrates everything in a single pass. This design keeps the codebase simple but means the reconcile function is complex, with multiple sequential phases: legacy adoption, ConfigMap reconciliation, manifest rendering, kustomize transformation, resource application, ingress reconciliation, and status updates. The controller uses sentinel error types (`requeueError` and `terminalError`) to distinguish retriable failures from permanent ones.

The manifest rendering pipeline is notable: base YAML templates in `controllers/manifests/base/` are processed through kustomize and then through a chain of Go-based transformer plugins (`NamePrefixPlugin`, `NamespacePlugin`, `FieldMutatorPlugin`, `NetworkPolicyTransformerPlugin`). The `FieldMutatorPlugin` performs JSON Pointer-based field injection to customize each resource with per-instance values (name, namespace, replicas, ports, storage size). This approach avoids Go template complexity while allowing dynamic resource customization. The Deployment's pod spec is built entirely in Go code (container spec, volumes, environment variables, probes) and then injected into the rendered manifest as an unstructured map.

The ConfigMap cache design is architecturally interesting: the operator's informer cache filters ConfigMaps by the label `ogx.io/watch: "true"`. Operator-managed ConfigMaps get this label automatically. User ConfigMaps that need instant change detection must also carry this label. ConfigMaps without the label (including the operator's own config ConfigMap) are read via a non-cached direct client, with the reconciler's 5-minute periodic requeue providing eventual consistency. This reduces cache memory pressure while still offering opt-in real-time reactivity. Secrets follow the same pattern.

The legacy adoption mechanism demonstrates careful handling of Kubernetes resource ownership semantics. When migrating from LlamaStackDistribution to OGXServer, the operator: (1) validates the adoption annotation is not self-referential, (2) scales down the legacy Deployment to zero and waits for pod termination (avoiding RWO PVC multi-attach deadlock), (3) strips the legacy controller's ownerRef from the PVC while intentionally not setting a new one (PVCs must outlive CR deletion), (4) sets adoption audit labels and timestamps, and (5) transfers Service/Ingress ownership by replacing the controller ownerRef. This entire flow is idempotent and requeue-safe, using label-based discovery as a fallback when annotations are removed post-adoption.

The declarative config generation pipeline resolves a base `config.yaml` from either a referenced ConfigMap or OCI image labels on the distribution image, then merges it with declarative provider, resource, storage, and disabledAPIs specs from the CR. The generated ConfigMap is named with a content hash for immutability and safe rollback, with retention of the last 2 generated ConfigMaps plus any still referenced by active ReplicaSets.
