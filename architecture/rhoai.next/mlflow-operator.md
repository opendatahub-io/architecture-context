## Purpose

**Short**: A Kubernetes operator that manages the lifecycle of MLflow tracking server deployments, including database migrations, ingress routing via Gateway API, and garbage collection of soft-deleted resources.

**Detailed**: The MLflow Operator is a controller-runtime-based Go operator that reconciles a cluster-scoped `MLflow` custom resource (API group `mlflow.opendatahub.io/v1`) into a fully functional MLflow tracking server deployment. Rather than generating Kubernetes manifests directly in Go code, the operator embeds a Helm chart (`charts/mlflow/`) and renders it at reconciliation time, converting `MLflow` CR spec fields into Helm values. The rendered objects are applied to the cluster using Server-Side Apply.

The operator manages a sophisticated deployment lifecycle including: operator-managed database migrations (via Kubernetes Jobs that run `mlflow db upgrade` before scaling up new deployments), automatic CA bundle aggregation (platform and custom CA certificates combined via init container and sidecar watcher), optional garbage collection (via CronJobs that permanently delete soft-deleted MLflow resources), and integration with the RHOAI Gateway API ingress stack (creating HTTPRoute resources that reference the platform's `data-science-gateway`). It also creates OpenShift ConsoleLink resources for UI integration and ServiceMonitor resources for Prometheus metrics collection.

The operator deploys into the `redhat-ods-applications` namespace (RHOAI) or `opendatahub` namespace (ODH) and manages MLflow server instances in the same namespace. It supports multiple metadata store backends (SQLite, PostgreSQL, MySQL) and multiple artifact storage backends (local filesystem, S3, GCS, Azure Blob) with configurable TLS, CORS, and network policies. The MLflow server itself uses Kubernetes-native authentication via `SelfSubjectAccessReview`, enforced through the `--app-name=kubernetes-auth` flag and RBAC pseudo-resources in the `mlflow.kubeflow.org` API group.

## Architectural Analysis

The MLflow Operator takes an unusual hybrid approach to Kubernetes resource management: rather than generating manifests in Go code (the typical operator pattern), it embeds a Helm chart and renders it at reconciliation time using the Helm v3 library. The `HelmRenderer` in `internal/controller/helm.go` translates MLflow CR spec fields into Helm values, then calls `engine.Render()` to produce unstructured Kubernetes objects. These objects are applied via Server-Side Apply with `mlflow-operator` as the field owner. This pattern provides the templating expressiveness of Helm while maintaining the reconciliation guarantees of controller-runtime. The trade-off is complexity -- the operator must maintain two layers of abstraction (Go values mapping and Helm templates) and the Helm chart must be distributed as a baked-in directory in the operator container image.

The database migration system is particularly sophisticated. When a version upgrade or spec change is detected, the operator orchestrates a zero-downtime migration by: (1) rendering all objects with the Deployment scaled to zero replicas, (2) waiting for existing pods to drain, (3) creating a migration Job derived from the Deployment's pod spec (reusing the same container image, volumes, CA bundles, and env vars but with a Python migration script injected), (4) monitoring the Job for completion with structured exit code handling, and (5) restoring the Deployment to its desired replica count only after migration succeeds. Terminal failures (version mismatch, unsupported backend schemes, revision conflicts) are distinguished from retryable failures via explicit exit codes, and the operator exposes a `mlflow.opendatahub.io/force-migrate` annotation for manual rerun after fixes.

The RBAC model has three distinct layers: (1) the operator's own ClusterRole (`manager-role`) and namespace-scoped Role for managing Kubernetes resources, (2) the MLflow server's shared ClusterRole (`mlflow`) for namespace enumeration and artifact credential access, and (3) aggregate ClusterRoles (`mlflow-view`, `mlflow-edit`, `mlflow-integration`) that define authorization pseudo-resources in the `mlflow.kubeflow.org` API group. These pseudo-resources (experiments, datasets, registeredmodels, gatewayendpoints, gatewaysecrets) are not actual CRDs -- they are evaluated via `SelfSubjectAccessReview` by the MLflow server's Kubernetes auth plugin to make RBAC decisions. This is an elegant pattern that reuses Kubernetes RBAC for application-level authorization without introducing custom auth middleware.

The operator's conditional CRD handling is noteworthy: at startup, it uses the Kubernetes discovery API to check for HTTPRoute, ConsoleLink, and ServiceMonitor CRD availability, then conditionally configures cache watches and enables/disables reconciliation of those resources. This allows the same operator binary to run on vanilla Kubernetes (without Gateway API, OpenShift, or Prometheus Operator) and progressively enable features based on cluster capabilities. The RBAC watch architecture uses two separate caches -- the main manager cache for the shared `mlflow` ClusterRole/ClusterRoleBinding (with `metadata.name` field selectors for tight RBAC), and a dedicated `GCRBACWatchCache` for the `mlflow-gc` RBAC objects -- to avoid widening label-scoped watches that would require broader RBAC permissions.

Network security is enforced through NetworkPolicies that the operator renders for both the MLflow server and migration Jobs. The server's NetworkPolicy allows ingress only on port 8443/TCP from any namespace (since the Gateway, dashboard, and Prometheus may reside in different namespaces) and restricts egress to DNS, HTTPS, PostgreSQL, MySQL, and S3-compatible storage ports. The migration Job gets a separate, tighter NetworkPolicy that only allows DNS and database egress. Both NetworkPolicies are configurable via the CR spec (`networkPolicyEgressRules`, `networkPolicyAdditionalEgressRules`).
