## Purpose

**Short**: Kubernetes operator that manages the lifecycle of MLflow tracking server deployments using Helm chart rendering and Gateway API integration.

**Detailed**: The MLflow Operator is a controller-runtime based Kubernetes operator that reconciles cluster-scoped `MLflow` custom resources to deploy and manage MLflow tracking server instances. Rather than using static manifests, the operator renders an embedded Helm chart (`charts/mlflow`) at reconciliation time, converting CRD spec fields into Helm values and applying the rendered Kubernetes objects via Server-Side Apply.

The operator manages a comprehensive set of resources including Deployments, Services, ServiceAccounts, PersistentVolumeClaims, NetworkPolicies, CronJobs (for garbage collection), ClusterRoles, ClusterRoleBindings, HTTPRoutes (Gateway API), ConsoleLinks (OpenShift), and ServiceMonitors (Prometheus). It performs runtime capability detection via the Kubernetes discovery API to conditionally create resources only when the required CRDs are present (e.g., HTTPRoute, ConsoleLink, ServiceMonitor).

A key architectural feature is the operator-managed database migration system. Before scaling up the MLflow Deployment after version upgrades, the operator scales the Deployment to zero, runs a migration Job using an embedded Python script, and only restores replicas after successful completion. The migration system supports automatic retries, force-rerun via annotations, terminal failure detection with specific exit codes, and TTL-based Job cleanup. The operator integrates with the RHOAI/ODH platform by creating HTTPRoute resources referencing the `data-science-gateway` Gateway for external access, and manages CA bundle composition from platform and custom sources via init containers and sidecar watchers.

## Architectural Analysis

The MLflow Operator employs a distinctive hybrid architecture: it is a standard controller-runtime operator for lifecycle management but uses Helm as a library for resource rendering. Rather than maintaining separate Go structs for each managed Kubernetes resource, the operator embeds a Helm chart (`charts/mlflow`) and translates the MLflow CRD spec into Helm values at reconciliation time. This approach centralizes resource templates in familiar Helm YAML while retaining the full power of controller-runtime's watch/reconcile/status loop. The rendered objects are applied via Server-Side Apply with `mlflow-operator` as the field owner, providing clean conflict resolution without manual three-way merges.

The runtime capability detection pattern is noteworthy. At startup, the operator uses the Kubernetes discovery API to check whether HTTPRoute (Gateway API), ConsoleLink (OpenShift), and ServiceMonitor (Prometheus) CRDs are present. This result configures both the informer cache (to avoid watch errors on missing CRDs) and the reconciler behavior (skipping resource creation when CRDs are absent). This makes the operator portable across OpenShift (with full Gateway API + ConsoleLink integration), vanilla Kubernetes with Gateway API, and minimal Kubernetes clusters. The detection runs once at startup — CRDs installed after operator boot require a restart.

The migration orchestration system is the most complex subsystem. When the operator detects a version mismatch between `status.version` and the build-time-injected `SupportedMLflowVersion`, or when `spec.migration.mode=Always`, it enters a multi-phase flow: (1) scale the Deployment to zero replicas, (2) wait for all pods to terminate, (3) create a migration Job derived from the Deployment's pod template (reusing the same image, volumes, CA bundles, and environment), (4) monitor the Job for success or failure, (5) on success, restore the Deployment and update `status.version`. Failed Jobs are classified as terminal (version mismatch, unsupported backend) or retryable (transient database errors), with automatic retry after a 2-minute delay for the latter. The `mlflow.opendatahub.io/force-migrate` annotation provides an escape hatch for administrators to trigger reruns after fixing issues. Migration Jobs are named deterministically based on MLflow version and CR generation, enabling idempotent reconciliation.

The RBAC architecture separates operator permissions from server permissions and provides user-facing aggregate roles. The operator uses a ClusterRole for cluster-scoped resources (CRDs, ConsoleLinks, HTTPRoutes, shared RBAC) and a namespace-scoped Role for managed workload resources (Deployments, Services, etc.). The MLflow server gets its own shared ClusterRole (`mlflow`) for namespace enumeration, MLflowConfig reads, and artifact-connection secret access. Aggregate ClusterRoles (`mlflow-view`, `mlflow-edit`) compose into Kubernetes' standard view/edit/admin roles, while `mlflow-integration` provides a minimal role for service accounts that need data-plane access (experiments, models) without administrative permissions. Pseudo-resources under `mlflow.kubeflow.org` (datasets, experiments, registeredmodels, gatewayendpoints, etc.) are not actual CRDs — they are synthetic API groups checked via SelfSubjectAccessReview by MLflow's Kubernetes auth plugin to enforce fine-grained access control.

The network security posture is well-defined. Each MLflow instance gets a dedicated NetworkPolicy restricting ingress to port 8443 from any namespace (since the gateway, dashboard, and Prometheus may be in different namespaces) and restricting egress to DNS, HTTPS, PostgreSQL, MySQL, and S3-compatible storage ports. A separate migration NetworkPolicy limits migration Job egress to only DNS and database ports. TLS is mandatory — the MLflow server runs uvicorn with `--ssl-keyfile` and `--ssl-certfile` from OpenShift service-ca provisioned certificates, and the `MLFLOW_SERVER_DISABLE_SECURITY_MIDDLEWARE` environment variable is hardcoded to `false` to prevent bypassing security middleware.
