## Purpose

**Short**: Kubernetes operator that manages the lifecycle of MLflow tracking server deployments, including artifact storage, workspace management, and Gateway API routing on RHOAI/ODH platforms.

**Detailed**: The MLflow Operator is a controller-runtime based Go operator that watches cluster-scoped `MLflow` custom resources and reconciles the desired state into a full MLflow deployment stack. Upon receiving an `MLflow` CR, the operator renders an embedded Helm chart to produce Kubernetes-native resources (Deployment, Service, ServiceAccount, NetworkPolicy, PVC, ClusterRole/ClusterRoleBinding) and applies them using server-side apply.

The operator integrates with the RHOAI 3.x platform ingress pattern by dynamically creating `HTTPRoute` resources that reference the platform's `data-science-gateway`, enabling MLflow to be accessed through the centralized Gateway API entry point. On OpenShift clusters, the operator also creates `ConsoleLink` resources for integration with the OpenShift web console's application menu. The MLflow server itself runs with TLS enabled (via OpenShift service-ca certificate provisioning), Kubernetes RBAC-based authorization (using SelfSubjectAccessReview), and a configurable CA bundle system for trust chain management.

The operator supports flexible backend storage (PostgreSQL, MySQL, SQLite), artifact storage (S3-compatible, GCS, local filesystem), multi-workspace namespacing via the Kubernetes workspace provider, and Prometheus metrics integration via ServiceMonitor resources.

## Architectural Analysis

The MLflow Operator follows a Helm-chart-as-template rendering pattern rather than the more common declarative kustomize or hand-built controller approach. The operator embeds a full Helm chart at `charts/mlflow/` and uses the Helm SDK to render templates at reconciliation time, converting CR spec fields into Helm values. This design gives the operator the expressiveness of Helm templating (conditional sidecars, dynamic volume mounts, configurable network policies) while retaining the advantages of a controller-runtime operator (watches, status management, leader election). However, it means the operator binary must bundle the chart directory alongside the Go binary in the container image (`COPY --from=builder /workspace/charts/mlflow charts/mlflow`).

The operator's approach to cluster API availability is notably careful: it probes the Kubernetes discovery API at startup to check for the presence of ConsoleLink, HTTPRoute, and ServiceMonitor CRDs, then conditionally configures its informer cache and watch list. This makes the operator portable across vanilla Kubernetes (no Gateway API, no OpenShift) and fully-featured OpenShift/RHOAI clusters without requiring feature flags or separate deployment configurations. The cache is further optimized with label selectors (`app: mlflow`) and namespace scoping to minimize memory footprint.

The security architecture is multi-layered. The MLflow server itself runs with Kubernetes RBAC-based authorization (`--app-name=kubernetes-auth`), performing `SelfSubjectAccessReview` checks against pseudo-resources in the `mlflow.kubeflow.org` API group (e.g., `experiments`, `registeredmodels`, `gatewayendpoints`). This means access control is delegated to standard Kubernetes RBAC — the aggregate ClusterRoles (`mlflow-view`, `mlflow-edit`, `mlflow-integration`) define what users can do, and MLflow enforces it at the application layer. The `MLFLOW_SERVER_DISABLE_SECURITY_MIDDLEWARE` environment variable is hardcoded to `"false"` in the deployment template and a CEL validation rule on the CRD prevents users from overriding it via the `env` field, ensuring the security middleware cannot be accidentally or intentionally disabled.

The CA bundle management system is sophisticated: an init container concatenates system CAs, platform CAs (from `odh-trusted-ca-bundle`), and optional user-provided CAs into a single combined bundle. A sidecar container then watches for ConfigMap changes and regenerates the bundle on a 30-second polling interval. The combined bundle is injected into the MLflow container via environment variables that cover Python's `requests` library, `curl`, `boto3` (AWS), `psycopg2` (PostgreSQL SSL), and `mysql` (MySQL SSL), ensuring consistent TLS verification across all backend integrations.

The NetworkPolicy is permissive on ingress (any namespace can reach the MLflow pod on port 8443) but scoped on egress, explicitly allowing only DNS, HTTPS, Kubernetes API, PostgreSQL, MySQL, and S3-compatible storage ports. Users can extend egress rules via the `networkPolicyAdditionalEgressRules` field on the CR, with CEL validation ensuring each rule has at least one port or destination specified.
