## Purpose

**Short**: Kubernetes operator that manages MLflow tracking server deployments on OpenShift/Kubernetes, handling lifecycle, networking, TLS, CA bundles, and integration with the ODH/RHOAI platform gateway.

**Detailed**: The MLflow Operator is a controller-runtime-based Kubernetes operator that reconciles a cluster-scoped `MLflow` custom resource (API group `mlflow.opendatahub.io/v1`) into a fully configured MLflow tracking server deployment. It uses an embedded Helm chart as a templating engine to render Kubernetes resources (Deployment, Service, ServiceAccount, PVC, NetworkPolicy, ClusterRole, ClusterRoleBinding, ServiceMonitor) and applies them via Server-Side Apply.

The operator integrates with the RHOAI 3.x platform ingress architecture by creating Gateway API `HTTPRoute` resources that reference the platform's `data-science-gateway` Gateway, exposing MLflow through the shared platform hostname with path-prefix-based routing (e.g., `/mlflow/api`, `/mlflow/v1`). On OpenShift clusters, it also creates `ConsoleLink` resources for OpenShift console integration and leverages service-serving-cert annotation for automatic TLS certificate provisioning.

The operator manages CA bundle composition through an init container and sidecar pattern, combining system CAs, platform-injected CA bundles (`odh-trusted-ca-bundle`), and user-provided custom CA ConfigMaps into a single PEM file consumed by the MLflow server for TLS verification with backend stores (PostgreSQL, MySQL, S3). It also deploys and distributes a second CRD (`MLflowConfig`, API group `mlflow.kubeflow.org/v1`) that enables namespace-scoped artifact storage configuration for the MLflow workspaces feature.

## Architectural Analysis

The MLflow Operator follows a clean Helm-as-templating-engine pattern where the operator owns the lifecycle but delegates manifest generation to an embedded Helm chart (`charts/mlflow/`). This is a pragmatic design choice: the operator handles the CRD-driven lifecycle, status management, and platform-specific resource creation (HTTPRoute, ConsoleLink) in Go code, while the repetitive Kubernetes manifests (Deployment, Service, NetworkPolicy, PVC, RBAC) are maintained as Helm templates. The Helm chart is never installed via Helm — it's loaded and rendered in-process by the `HelmRenderer` type and applied via Server-Side Apply. This avoids Helm release management overhead while retaining template reuse.

The CA bundle management is notably well-designed. Rather than requiring the MLflow container to understand multiple CA sources, the operator deploys an init container and sidecar that combine system CAs (`/etc/pki/tls/certs/ca-bundle.crt`), platform CAs (`odh-trusted-ca-bundle` ConfigMap), and user-provided CAs into a single PEM file at `/etc/pki/tls/certs/combined/ca-bundle.crt`. The sidecar performs periodic checksum-based change detection (every 30 seconds by default) and regenerates the combined bundle when source ConfigMaps are updated by kubelet's ConfigMap auto-refresh. This sidesteps the need for pod restarts on CA rotation. The operator also sets multiple environment variables (`SSL_CERT_FILE`, `REQUESTS_CA_BUNDLE`, `CURL_CA_BUNDLE`, `AWS_CA_BUNDLE`, `PGSSLROOTCERT`) to ensure all Python libraries in the MLflow server use the combined bundle, regardless of which library performs the TLS handshake.

The operator's resource ownership model is interesting for the shared `mlflow` ClusterRole. Since the ClusterRole is shared across all MLflow instances (there can be multiple MLflow CRs, each creating a ClusterRoleBinding to the same ClusterRole), the operator uses `appendOwnerReference` instead of `SetControllerReference`. This allows multiple MLflow CRs to list themselves as owners without conflicting on the controller reference. The tradeoff is that the ClusterRole requires explicit garbage collection logic — the `Watches` handler with `clusterRoleToMLflowRequests` ensures changes to the shared ClusterRole trigger reconciliation for all owning MLflow instances.

The security model is Kubernetes-native: the MLflow server runs with `--app-name=kubernetes-auth`, which activates MLflow's Kubernetes authorization plugin. This plugin intercepts API requests and performs `SelfSubjectAccessReview` checks against pseudo-resources in the `mlflow.kubeflow.org` API group (e.g., `experiments`, `datasets`, `registeredmodels`). Access is controlled via the aggregate ClusterRoles (`mlflow-view`, `mlflow-edit`, `mlflow-integration`) that grant permissions on these pseudo-resources. This design leverages Kubernetes RBAC as the authorization backend rather than implementing a separate auth system. The `MLFLOW_SERVER_DISABLE_SECURITY_MIDDLEWARE` environment variable is hardcoded to `"false"` in the Deployment template and blocked from user override via CEL validation on the CRD, preventing users from accidentally disabling the security middleware.

The operator's RHOAI 3.x ingress integration creates HTTPRoutes with URL rewriting rules to strip the `/mlflow` prefix before forwarding to the MLflow backend. Three route rules handle different path patterns: `/mlflow/api/*` is rewritten to `/api/*`, `/mlflow/v1/*` to `/v1/*`, and `/mlflow/*` passes through as-is for static assets. All routes target the backend service on port 8443/TCP, where TLS is terminated by uvicorn using the service-ca-provisioned certificate. The `UVICORN_SSL_CIPHERS=PROFILE=SYSTEM` setting on OpenShift ensures the server uses the platform's crypto policy for cipher selection, which is FIPS-appropriate.
