## Purpose

**Short**: Kubernetes operator that manages the lifecycle of MLflow server deployments for experiment tracking, model registry, and AI Gateway on OpenShift AI.

**Detailed**: The MLflow Operator is a controller-runtime based Kubernetes operator that reconciles a cluster-scoped `MLflow` custom resource into a fully functional MLflow server deployment. It uses an embedded Helm chart to render the deployment manifests (Deployment, Service, NetworkPolicy, PVC, ServiceAccount, ClusterRole/ClusterRoleBinding, ServiceMonitor) and then applies them via Server-Side Apply. The operator dynamically creates Gateway API HTTPRoute resources to expose MLflow through the RHOAI data-science-gateway, and OpenShift ConsoleLink resources for console integration.

The operator supports multiple storage backends (SQLite, PostgreSQL, MySQL) for experiment metadata, and multiple artifact storage systems (local filesystem via PVC, S3, GCS, HDFS). It manages TLS certificates via OpenShift's service-ca operator, builds combined CA bundles from platform and custom ConfigMaps using init containers and sidecar watchers, and enforces network security through a comprehensive NetworkPolicy. The MLflow server itself uses Kubernetes-native RBAC authorization via SelfSubjectAccessReview, with aggregate ClusterRoles that integrate into standard Kubernetes admin/edit/view role hierarchies.

The operator deploys into the `redhat-ods-applications` namespace (RHOAI) or `opendatahub` namespace (ODH) and manages MLflow resources in the same namespace. It is built and shipped via Konflux as the `odh-mlflow-operator-rhel9` container image.

## Architectural Analysis

The MLflow Operator follows a Helm-in-operator pattern: rather than managing individual Kubernetes resources through imperative Go code, it embeds a complete Helm chart (`charts/mlflow/`) and renders it at reconcile time using the Helm Go SDK. The operator converts the `MLflow` CR spec into Helm values (in `helm.go`), renders templates, and applies the resulting unstructured objects via Server-Side Apply with field ownership set to `mlflow-operator`. This approach gives the benefits of declarative Helm templating (easy to understand, standard Kubernetes patterns) while providing the reconciliation guarantees of a Kubernetes operator (drift detection, status management, owner references for garbage collection).

The operator's resource naming scheme supports multiple MLflow instances via a suffix mechanism: the default CR name `mlflow` produces resources named `mlflow`, while any other name produces `mlflow-<name>`. This carries through to all resources including the HTTPRoute path prefix (`/mlflow` vs `/mlflow-<name>`), ConsoleLink name, and ClusterRoleBinding. The shared `mlflow` ClusterRole uses a non-controller owner reference approach (`appendOwnerReference`) that allows multiple MLflow instances to share the same ClusterRole without conflicting controller references — a deliberate design choice to avoid permission duplication.

The TLS architecture is notable for its comprehensiveness. The MLflow server runs uvicorn with direct SSL termination (not behind a proxy), using certificates provisioned by OpenShift's service-ca operator. For outbound TLS (to databases and artifact stores), the operator builds a combined CA bundle from three sources: system CAs, platform CAs (`odh-trusted-ca-bundle`), and user-provided CAs (`caBundleConfigMap` spec field). An init container creates the initial bundle, and a sidecar watcher regenerates it every 30 seconds when source ConfigMaps change — without requiring pod restarts since kubelet auto-updates mounted ConfigMaps. The combined bundle is injected via environment variables into multiple libraries (`SSL_CERT_FILE`, `REQUESTS_CA_BUNDLE`, `CURL_CA_BUNDLE`, `AWS_CA_BUNDLE`, `PGSSLROOTCERT`, `MLFLOW_MYSQL_CA`).

The authorization model is Kubernetes-native: the MLflow server uses the `kubernetes-auth` app module with `SelfSubjectAccessReview`-based authorization. API operations are mapped to RBAC verbs on pseudo-resources in the `mlflow.kubeflow.org` API group (e.g., `experiments`, `datasets`, `registeredmodels`, `gatewayendpoints`). Aggregate ClusterRoles (`mlflow-view`, `mlflow-edit`) automatically integrate into the standard Kubernetes admin/edit/view role hierarchy via `rbac.authorization.k8s.io/aggregate-to-*` labels, plus a dedicated `mlflow-integration` role provides scoped access for service accounts of other platform components. The `MLFLOW_SERVER_DISABLE_SECURITY_MIDDLEWARE` environment variable is hardcoded to `false` and explicitly blocked in the CR spec validation, preventing users from accidentally disabling auth.

The NetworkPolicy is carefully designed with defense-in-depth: ingress is restricted to port 8443 from any namespace (necessary since the gateway, dashboard, and Prometheus reside in varying namespaces), while egress is allowlisted for specific protocols and ports (DNS, HTTPS, K8s API, PostgreSQL, MySQL, MinIO, SeaweedFS). Users can extend egress via the `networkPolicyAdditionalEgressRules` spec field for non-standard ports. The policy includes both port 53 and 5353 for DNS — the latter required because OpenShift's OVN-Kubernetes evaluates egress rules after DNAT, where CoreDNS listens on 5353.
