## Purpose

**Short**: Kubernetes operator that creates and manages Llama Stack AI inference server deployments on OpenShift/Kubernetes clusters.

**Detailed**: The llama-stack-k8s-operator is a controller-runtime based Kubernetes operator that manages the lifecycle of Llama Stack distribution servers. It introduces the `LlamaStackDistribution` (shortName: `llsd`) Custom Resource Definition, which allows users to declaratively specify a Llama Stack inference server deployment, including the distribution image, replicas, autoscaling, persistent storage, custom CA bundles, user configuration, and network access controls.

The operator reconciles each `LlamaStackDistribution` CR by rendering Kubernetes manifests (Deployment, Service, ServiceAccount, PVC, NetworkPolicy, PDB, HPA, RoleBinding, Ingress) through an embedded kustomize pipeline with Go-based transformer plugins. It manages the full stack of resources needed to run a Llama Stack server: deploying the container with the correct distribution image, configuring environment variables for uvicorn workers and ports, mounting user-provided ConfigMaps for server configuration, aggregating CA bundle certificates from explicit and auto-detected ODH trusted CA bundles, and optionally exposing the service externally via Ingress. The operator probes the running server's `/v1/health`, `/v1/providers`, and `/v1/version` endpoints to populate rich status information including provider health, server version, and distribution configuration.

In the RHOAI context, the operator is deployed into the `redhat-ods-applications` namespace via kustomize overlays. It supports a Red Hat-specific distribution image (`RELATED_IMAGE_RH_DISTRIBUTION` environment variable) and integrates with the ODH trusted CA bundle ConfigMap (`odh-trusted-ca-bundle`) for automatic certificate injection. The operator uses feature flags (stored in a ConfigMap) to control NetworkPolicy creation and supports image mapping overrides for distribution images.

## Architectural Analysis

The llama-stack-k8s-operator follows a clean single-CRD operator pattern built on kubebuilder v4 scaffolding with controller-runtime v0.22. The operator embeds its per-instance resource manifests (Deployment, Service, PVC, NetworkPolicy, PDB, HPA, RoleBinding, ServiceAccount) in `controllers/manifests/base/` and renders them at reconciliation time using an in-process kustomize pipeline. This avoids the typical pattern of constructing Kubernetes objects in Go code, instead using YAML templates with Go-based kustomize transformer plugins for field mutation (`FieldMutator`), name prefixing (`NamePrefix`), namespace setting (`Namespace`), and NetworkPolicy rule generation (`NetworkPolicyTransformer`). The Deployment name is kept equal to the CR name (excluded from prefix) for backward compatibility, while all other resources are prefixed with the instance name.

The CA bundle handling is notably sophisticated. The operator supports three certificate sources: (1) explicit CA bundle ConfigMaps referenced in `spec.server.tlsConfig.caBundle`, (2) the well-known `odh-trusted-ca-bundle` ConfigMap auto-detected in the CR's namespace, and (3) a combination of both. Certificates are validated through full PEM parsing and X.509 certificate validation, with security limits on total size (10MB) and certificate count (1000). All sources are merged into a single operator-managed ConfigMap (`{instance}-ca-bundle`) mounted at `/etc/ssl/certs/ca-bundle/ca-bundle.crt` with the `SSL_CERT_FILE` environment variable pointing to it. This design enables transparent CA certificate management for Llama Stack servers connecting to TLS-protected inference backends.

The operator implements a dual-client pattern: a cached client (via controller-runtime's informer cache) for watching managed resources with label selectors, and a direct (non-cached) API client for reading user-referenced ConfigMaps that lack operator labels. The cache is configured with label selectors filtering on `app.kubernetes.io/managed-by=llama-stack-operator` for most resources and `llamastack.io/watch=true` for ConfigMaps, minimizing memory usage. User-supplied ConfigMaps that have the `llamastack.io/watch=true` label are watched for changes and trigger reconciliation of referencing CRs, while those without the label are read on each reconciliation cycle (with a 5-minute requeue interval for eventual consistency).

The NetworkPolicy feature is controlled by a feature flag (`enableNetworkPolicy`) stored in the operator's configuration ConfigMap. When enabled, it creates per-instance NetworkPolicies that restrict ingress to the service port from: (1) all pods in the same namespace, (2) the operator namespace, (3) explicitly listed namespaces, (4) namespaces matching specified labels, and (5) OpenShift router namespaces (via `network.openshift.io/policy-group=ingress` label). The wildcard `*` in the namespaces list disables restrictions by using an empty namespaceSelector. The operator also supports HPA-driven autoscaling with CPU/memory utilization targets, and automatically strips `spec.replicas` from the Deployment when autoscaling is enabled to avoid conflicts with the HPA controller. When persistent storage is configured, the Deployment strategy is set to Recreate to avoid RWO PVC multi-attach deadlocks during rolling updates.
