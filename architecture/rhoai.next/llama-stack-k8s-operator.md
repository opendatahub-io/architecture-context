## Purpose

**Short**: Kubernetes operator that manages the lifecycle of OGX (formerly Llama Stack) AI inference servers, including deployment, networking, autoscaling, and provider configuration.

**Detailed**: The OGX K8s Operator (renamed from llama-stack-k8s-operator) is a controller-runtime based Kubernetes operator that automates the deployment and management of OGX AI inference server instances. It watches OGXServer custom resources (v1beta1, API group ogx.io) and reconciles the desired state by creating and managing Deployments, Services, NetworkPolicies, PersistentVolumeClaims, HorizontalPodAutoscalers, PodDisruptionBudgets, Ingresses, and ConfigMaps.

The operator supports a rich provider model where users can configure multiple AI service providers (vLLM, OpenAI, Azure, Bedrock, VertexAI, Watsonx, PGVector, Milvus, Qdrant, S3, BraveSearch, TavilySearch, MCP) through the CRD spec. It generates a server configuration (config.yaml) from the CR spec and mounts it into the OGX server container. The operator uses an embedded kustomize engine to render deployment manifests from base templates, applying per-instance customizations through a plugin pipeline (namespace, name prefix, field mutation, network policy transformation).

The operator also includes a legacy adoption system for migrating from the older LlamaStackDistribution (v1alpha1, API group llamastack.io) CRD to the new OGXServer CRD, supporting zero-downtime migration of PVCs, Services, and Ingresses. It integrates with OpenShift security features (SCC anyuid for init containers), supports TLS server configuration for the managed OGX servers, CA bundle injection for outbound TLS trust, and Prometheus metrics monitoring.

## Architectural Analysis

The OGX K8s operator follows the standard Kubernetes operator pattern with controller-runtime, implementing a single reconciler (OGXServerReconciler) that manages the full lifecycle of OGX inference server deployments. A notable architectural decision is the use of an **embedded kustomize engine** for manifest rendering rather than Go templates or raw object construction. The operator bundles base manifest templates in `controllers/manifests/base/` (Deployment, Service, NetworkPolicy, PVC, HPA, PDB, ServiceAccount, RoleBinding) and applies a Go plugin pipeline at runtime to customize them per-instance. This pipeline includes namespace injection, name prefixing, field mutation via JSON Pointers, network policy transformation, and autoscaling configuration. This design cleanly separates the manifest structure from per-instance customization logic and makes the base resources easy to audit independently.

The **provider model** is architecturally significant — the CRD spec supports a rich hierarchy of AI providers (inference, vector I/O, tool runtime, files, batches, responses) with both "remote" providers (external services like vLLM, OpenAI, PGVector) and "inline" providers (bundled implementations like file search, local FS). Each provider type implements a `Provider` interface with `DeriveID()` for automatic identifier generation. The operator generates a `config.yaml` from the CR spec and mounts it into the OGX server container, with ConfigMap hash annotations on the pod template to trigger rolling restarts when configuration changes. The webhook performs validation of provider ID uniqueness and cross-references (e.g., model provider references must point to configured inference providers), and CEL rules on the CRD enforce mutual exclusivity between `overrideConfig` mode and individual provider fields.

The **legacy adoption system** (`controllers/legacy_adoption.go`) is a well-designed migration path from the v1alpha1 LlamaStackDistribution CRD to the v1beta1 OGXServer CRD. It supports adopting PVCs (detaching from old controller, relabeling), Services (updating selectors to route to new pods), and Ingresses (transferring ownership) from legacy deployments. The adoption is triggered by annotations (`ogx.io/adopt-storage`, `ogx.io/adopt-networking`) and includes safeguards: it scales the legacy Deployment to zero, waits for pods to terminate before completing PVC adoption, validates annotation format (RFC 1123 DNS label), and records audit timestamps. Notably, adopted PVCs intentionally do not get a new ownerReference — they must outlive the OGXServer CR to preserve data.

The operator's **network architecture** includes per-instance NetworkPolicy support with a sophisticated transformer plugin. The transformer auto-injects kube-dns egress rules when egress restrictions are configured (preventing DNS breakage), adds default peers for same-namespace and operator-namespace pods, and supports router peers for OpenShift ingress namespaces. External access is provided via Kubernetes Ingress resources rather than OpenShift Routes or Gateway API, keeping the operator Kubernetes-portable.

A noteworthy risk area is the **image override system** (`controllers/ogxserver_controller.go:1564-1595`), which reads image mappings from an operator-managed ConfigMap. This provides a decoupled upgrade path where distribution images can be updated without modifying the operator binary, but also introduces a configuration surface that could be misused to inject arbitrary images. The operator validates image references using go-containerregistry but does not enforce image signing or provenance checks at this layer.
