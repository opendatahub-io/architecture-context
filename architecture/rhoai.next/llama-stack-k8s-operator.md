## Purpose

**Short**: A Kubernetes operator that manages the lifecycle of Llama Stack distribution server deployments, providing declarative configuration for AI inference stack instances on OpenShift/Kubernetes clusters.

**Detailed**: The llama-stack-k8s-operator is a controller-runtime-based Kubernetes operator that reconciles `LlamaStackDistribution` custom resources to deploy and manage Llama Stack server instances. Llama Stack is Meta's framework for building AI applications, and this operator automates provisioning of the server components including container deployments, persistent storage, network policies, horizontal pod autoscaling, and ingress configuration.

The operator uses kustomize-based manifest rendering with a custom plugin pipeline to transform base manifests into instance-specific Kubernetes resources. It supports multiple distribution images (starter, remote-vllm, meta-reference-gpu, postgres-demo) and can be overridden via operator-level ConfigMap configuration. The operator also manages CA bundle aggregation from multiple sources (explicit user bundles and auto-detected ODH trusted CA bundles), user configuration injection via ConfigMap mounts, and feature-flag-driven behavior (e.g., NetworkPolicy creation).

For RHOAI, the operator is deployed into the `redhat-ods-applications` namespace and supports RELATED_IMAGE environment variable overrides for disconnected/air-gapped environments. It creates per-instance resources including Deployments, Services, ServiceAccounts, RoleBindings, PersistentVolumeClaims, NetworkPolicies, PodDisruptionBudgets, HorizontalPodAutoscalers, and optionally Ingress resources for external access.

## Architectural Analysis

The llama-stack-k8s-operator follows a well-structured single-CRD operator pattern with a clean separation between manifest rendering (kustomize + plugin pipeline) and resource reconciliation. The kustomize plugin pipeline (`pkg/deploy/plugins/`) applies four transformations in sequence: name prefixing (with Deployment excluded for backward compatibility), namespace setting, field mutation (instance-specific values like ports, replicas, storage sizes), and NetworkPolicy transformation. This approach is more maintainable than hardcoding resource construction in Go, as the base YAML manifests in `controllers/manifests/base/` can be read and understood independently.

A notable design decision is the dual-client architecture: the controller uses both a cached client (via the informer cache, filtered by `app.kubernetes.io/managed-by=llama-stack-operator` labels) and a non-cached "DirectClient" for reading ConfigMaps that lack operator labels. This prevents the operator from caching all ConfigMaps in the cluster while still being able to read user-referenced ConfigMaps and the operator configuration ConfigMap. The cache also strips managed fields (`TransformStripManagedFields`) to reduce memory usage. The ConfigMap watch is further narrowed by requiring a `llamastack.io/watch=true` label on user ConfigMaps for instant reconciliation on changes; without this label, changes are detected only on the 5-minute periodic requeue.

The CA bundle management is particularly sophisticated. The operator supports three sources of CA certificates: explicit user-provided bundles (via `spec.server.tlsConfig.caBundle`), auto-detected ODH/RHOAI platform trusted CA bundles (`odh-trusted-ca-bundle` ConfigMap), and combinations of both. All certificates are validated at the PEM/X.509 level with security limits (10MB max size, 1000 max certificates), concatenated into a single managed ConfigMap (`{instance}-ca-bundle`), and mounted at `/etc/ssl/certs/ca-bundle/ca-bundle.crt` with `SSL_CERT_FILE` set accordingly. This three-layer approach (source → validation → managed ConfigMap → pod mount) ensures certificate integrity while supporting dynamic updates.

The operator includes an upgrade cleanup mechanism (`cluster.PerformUpgradeCleanup`) that removes legacy ClusterRoleBindings from previous versions when the operator migrated from cluster-scoped to namespace-scoped RBAC. The `deploymentNeedsFullReplacement` function handles another upgrade path where SSA (Server-Side Apply) cannot remove volumes that were originally created by `cli.Create`, falling back to full replacement. These pragmatic migration handlers demonstrate the maturity of the codebase.

One architectural observation: the operator communicates with its managed Llama Stack servers via plain HTTP on port 8321, which is acceptable for cluster-internal traffic but worth noting in security-sensitive environments. The NetworkPolicy feature (enabled by default) provides namespace-level network isolation, with configurable allow rules for additional namespaces or label-based selection.
