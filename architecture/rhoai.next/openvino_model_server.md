## Purpose

**Short**: OpenVINO Model Server is a high-performance C++ inference server that serves AI models via gRPC and REST APIs, compatible with OpenAI, KServe, and TensorFlow Serving protocols.

**Detailed**: OpenVINO Model Server (OVMS) is an Intel-developed, C++-based model serving system optimized for deployment on Intel architectures (CPU, GPU, NPU). It provides a comprehensive inference platform that supports traditional deep learning models (classification, detection, segmentation) as well as modern generative AI workloads including large language models (LLMs), visual language models (VLMs), text embeddings, document reranking, image generation, speech-to-text, and text-to-speech.

The server exposes multiple API protocols concurrently: TensorFlow Serving v1 API for backward compatibility, KServe v2 Inference Protocol for cloud-native ML serving, and an OpenAI-compatible v3 API for generative AI workloads (chat completions, embeddings, reranking, image generation, audio). It uses the OpenVINO runtime and OpenVINO GenAI libraries for optimized inference execution, supporting continuous batching for LLM workloads. Models can be loaded from local storage, Azure Blob Storage, Google Cloud Storage, S3, or pulled directly from HuggingFace Hub.

Within the RHOAI platform, OVMS is deployed as the inference runtime for KServe `InferenceService` resources, providing the actual model serving capability for deployed models. It runs as a container within model serving pods, receiving traffic through the platform's ingress infrastructure.

## Architectural Analysis

OpenVINO Model Server follows a modular architecture centered around a singleton `Server` class that orchestrates independent modules in a dependency-aware startup sequence. The nine-module system (PythonInterpreter, Profiler, Metrics, ServableManager, CAPI, gRPC, HTTP, HfPullModel, ServablesConfigManager) provides clean separation of concerns, with modules implementing a common `Module` interface with lifecycle methods (`start`/`shutdown`). The gRPC server intentionally starts before models are loaded to provide early liveness probe responses, which is critical for Kubernetes pod scheduling in RHOAI deployments.

The API layer implements a multi-protocol design where three distinct API families (TFS v1, KServe v2, OpenAI-compatible v3) share the same underlying model management and inference execution infrastructure. The v3 API is particularly noteworthy — it routes through MediaPipe graph execution even for simple chat completions, meaning all generative AI requests flow through the MediaPipe graph engine as the unified execution framework. This design allows composing complex inference workflows (e.g., retrieval-augmented generation pipelines, multi-model chains) using the same infrastructure as simple single-model inference.

The LLM subsystem is sophisticated, with model-specific I/O processing plugins (`src/llm/io_processing/`) for popular model families (Llama 3, Mistral, Hermes 3, Phi-4, Qwen 3, DevStral). Each plugin implements custom output parsing for tool calls and reasoning tokens, reflecting the reality that different LLM families use different special token conventions for structured output. The continuous batching mode uses OpenVINO GenAI's pipeline infrastructure for efficient multi-request LLM serving. Chat template rendering relies on Python Jinja2, which is why the Python interpreter module is loaded even though the core server is C++.

From a security perspective, OVMS provides minimal built-in authentication — just an optional API key file for REST endpoints. In RHOAI deployments, authentication is enforced by the kube-rbac-proxy sidecar injected by the platform operator. The server uses system OpenSSL for all TLS operations (via libcurl and libgit2), inheriting FIPS compliance from the UBI 9 base image's OpenSSL. However, the use of `openssl/md5.h` for file checksumming in `src/filesystem.hpp` is a minor FIPS concern, as MD5 is not a FIPS-approved algorithm — though in this context it's used for cache key computation rather than cryptographic security.

The build system is Bazel-based with extensive configuration flags controlling feature inclusion (MediaPipe, Python support, GPU drivers). The Konflux build uses an 8-hour timeout reflecting the substantial compilation time for the C++ codebase plus OpenVINO source builds. Build configuration is parameterized via `.konflux/build-args.conf`, which pins to specific `opendatahub-io` fork branches of OpenVINO, OpenVINO Tokenizers, and OpenVINO GenAI at the `releases/2026/1` branch with LTO (Link-Time Optimization) enabled for production binaries.
