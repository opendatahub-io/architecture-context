## Purpose

**Short**: High-performance AI model inference server supporting multiple API standards (KServe, TFS, OpenAI) with LLM continuous batching, embeddings, reranking, audio, and image generation capabilities.

**Detailed**: OpenVINO Model Server (OVMS) is a production-grade, high-performance system for serving AI models. Implemented in C++ for scalability and optimized for deployment on Intel architectures, it uses the OpenVINO runtime for inference execution. The server exposes models via gRPC and REST APIs, supporting TensorFlow Serving (TFS) v1 protocol, KServe (KFS) v2 inference protocol, and OpenAI-compatible chat/completions/embeddings APIs.

OVMS supports a wide range of AI workloads beyond traditional model inference: LLM text generation with continuous batching and streaming, text embeddings, document reranking, speech-to-text (Whisper), text-to-speech, and image generation (Stable Diffusion). It includes a DAG (Directed Acyclic Graph) pipeline scheduler for chaining models and custom processing nodes, as well as MediaPipe graph integration for complex inference pipelines.

The server supports loading models from local filesystems, Amazon S3, Google Cloud Storage, Azure Blob Storage, and HuggingFace Hub. It provides model versioning, hot reload, dynamic batching, and dynamic shape handling. The RHOAI distribution packages OVMS as a UBI 9-based container image built via Konflux, primarily used as the inference runtime for KServe InferenceService deployments.

## Architectural Analysis

**Module-Based Server Architecture**: OVMS follows a singleton server pattern with a pluggable module system. The `Server` class manages an ordered collection of modules (GRPCServerModule, HTTPServerModule, ServableManagerModule, MetricModule, HfPullModelModule, PythonInterpreterModule, etc.) that are started in dependency order and shut down in reverse. This design allows clean separation of concerns — the gRPC and HTTP frontends are independent modules that delegate to the shared ModelManager for inference execution. The HTTP server uses the Drogon framework with a dedicated thread pool for streaming (LLM completions) requests, preventing long-running SSE connections from starving non-streaming inference requests.

**Multi-Protocol API Surface**: A distinctive architectural choice is supporting three API standards simultaneously: TFS v1 (for backward compatibility), KServe v2 (for Kubernetes-native inference), and OpenAI-compatible v3 (for LLM/generative AI workloads). The v3 endpoints are routed through MediaPipe calculator graphs, which provides a flexible execution framework for complex inference pipelines (e.g., tokenize → generate → detokenize → parse tools). The v1/v2 endpoints use a more direct path through ModelManager and ModelInstance. This tri-protocol approach enables OVMS to serve as a drop-in replacement for multiple inference servers.

**LLM Serving with Continuous Batching**: The LLM subsystem (`src/llm/`) implements a sophisticated continuous batching engine that overlaps prompt processing and token generation across multiple concurrent requests. The architecture supports four decoding methods (standard, speculative decoding, EAGLE3, prompt lookup) and seven model-specific tool parsers for function calling (Llama3, Mistral, Hermes3, Devstral, Phi4, GPT-4o/o1, Qwen3). Visual language models (VLMs) extend the LLM servable to handle multimodal inputs (text + images). This makes OVMS competitive with dedicated LLM serving systems like vLLM while also supporting traditional inference workloads.

**Security Boundary Design**: OVMS intentionally does not implement TLS or strong authentication internally. The gRPC server uses `InsecureServerCredentials` and the HTTP server listens on plaintext. This is a deliberate architecture decision — in RHOAI deployments, security is enforced at the platform level via kube-rbac-proxy sidecars (for authentication), Istio mTLS (for encryption), and Gateway API/HTTPRoute (for ingress). The only application-level auth is optional API key validation for v3 (OpenAI) endpoints. The `extras/nginx-mtls-auth` sidecar demonstrates an alternative mTLS approach for non-platform deployments. One crypto observation: `src/filesystem.hpp` imports `openssl/md5.h` for file content hashing — this is MD5 used for deduplication, not security, but it would trigger warnings in strict FIPS audits.

**Build System and Hermeticity**: The project uses Bazel as its build system, which provides reproducible builds with pinned dependencies via the WORKSPACE file. The Konflux Dockerfile is notably complex (~480 lines) because it builds OpenVINO from source (when `ov_use_binary=0`), compiles OpenVINO Tokenizers, builds OpenVINO GenAI, and then builds OVMS itself — all in a multi-stage Docker build. The `.konflux/build-args.conf` pins specific commits for all OpenVINO components (ov_source_branch, ov_tokenizers_branch, ov_genai_branch all pointing to `releases/2026/0` on the `opendatahub-io` fork). This provides source-level pinning, but the build is not fully hermetic — Bazel fetches external dependencies during build, and the Dockerfile uses wget/git clone for dependencies. The Konflux build system compensates by controlling network access at the pipeline level.
