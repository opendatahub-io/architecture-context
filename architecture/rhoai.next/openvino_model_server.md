## Purpose

**Short**: OpenVINO Model Server is a high-performance C++ inference serving system that hosts AI models and exposes them via gRPC and REST APIs compatible with TensorFlow Serving, KServe v2, and OpenAI protocols.

**Detailed**: OpenVINO Model Server (OVMS) is a production-grade model serving solution developed by Intel and adapted for Red Hat OpenShift AI (RHOAI). It provides a unified inference endpoint for deploying machine learning models optimized through the OpenVINO toolkit. The server supports multiple model formats (OpenVINO IR, ONNX, PaddlePaddle, TensorFlow, TFLite) and can load models from local storage, Amazon S3, Google Cloud Storage, Azure Blob Storage, or HuggingFace Hub.

OVMS exposes three API families: TensorFlow Serving v1 (predict/classify/regress), KServe v2 (model inference with streaming), and OpenAI-compatible endpoints (chat completions, embeddings, reranking, image generation, speech-to-text, text-to-speech). The server supports advanced LLM serving capabilities including continuous batching with KV cache management, speculative decoding (EAGLE-3), tool-guided generation, extended reasoning, and prefix caching. Complex inference pipelines can be constructed using Directed Acyclic Graphs (DAGs) or MediaPipe graph definitions, with support for custom C/C++ nodes and embedded Python execution.

Within the RHOAI platform, OVMS serves as a KServe-compatible model runtime that can be deployed as an InferenceService. It is built via Konflux from the `red-hat-data-services` fork with RHOAI-specific base images (UBI 9), GPU driver support, and LTO optimizations. The single Konflux Dockerfile produces the `odh-openvino-model-server-rhel9` container image.

## Architectural Analysis

OpenVINO Model Server implements a modular server architecture where the main `Server` class orchestrates a set of independently-startable modules. The startup sequence is dependency-aware: MetricsModule → PythonInterpreterModule (optional) → ProfilerModule (optional) → ServableManagerModule → CAPIModule → GRPCServerModule → HTTPServerModule. This modular design enables features to be compiled in or out via Bazel build flags (`--config=mp_on_py_on`, `--config=mp_off_py_off`), producing different binary variants from the same codebase.

The dual HTTP framework approach is notable: OVMS uses Drogon as its async HTTP server, with a split threading model where regular unary requests are handled by Drogon's listener threads while streaming text generation requests are dispatched to a separate MediaPipe ThreadPool. This ensures that long-running LLM streaming responses don't starve short-lived inference requests. The gRPC layer supports multiple server instances (configurable via `GRPC_SERVERS` environment variable), each with independent thread pools, which allows scaling gRPC handling across cores. Both gRPC and REST share a maximum message size of 1GB.

The LLM serving architecture offers two pipeline engines: a legacy sequential `LLMPipeline` that processes one request at a time (with better NPU device support), and a `ContinuousBatchingPipeline` that batches multiple requests at the token level for higher throughput. Both use a dedicated executor thread that runs in a condition-variable-gated loop, waking when requests arrive and calling `pipeline.step()` to generate tokens. The continuous batching pipeline supports advanced features including prefix caching (reuse tokenized prompt KV cache across requests), speculative decoding with EAGLE-3 draft models, sparse attention (TriShape and XAttention modes), and dynamic KV cache eviction. All LLM features are exposed through MediaPipe calculator nodes configured via protobuf options, making them composable with other calculators.

The storage backend architecture uses a clean factory pattern: `ModelManager::getFilesystem()` inspects the URL prefix of a model path (s3://, gs://, az://, azurefs://, or local) and returns the appropriate `FileSystem` implementation. All backends implement the same interface for listing, downloading, and syncing model directories. The `HfPullModelModule` adds a parallel download path for HuggingFace Hub repositories using libgit2 for git clones and curl for GGUF file downloads, with proxy and custom endpoint support for air-gapped environments.

Security in OVMS is primarily externalized. The server provides optional API key authentication via Bearer tokens (configurable through `--api_key_file` or `API_KEY` environment variable), but has no built-in mTLS or OIDC support. For production deployments in RHOAI, authentication is handled by the platform's kube-rbac-proxy sidecar (RHOAI 3.x) or oauth-proxy sidecar (RHOAI 2.x), and network isolation is provided by Kubernetes NetworkPolicies. The `extras/nginx-mtls-auth/` directory provides a reference implementation for wrapping OVMS with an nginx mTLS proxy for standalone deployments. Path traversal prevention (`isPathEscaped()`) is the main in-process security control, blocking `../` sequences in model paths to prevent unauthorized file access.
