## Purpose

**Short**: High-performance AI model inference server that exposes models via gRPC and REST APIs, supporting KServe v2, TensorFlow Serving, OpenAI-compatible, and Cohere-compatible protocols.

**Detailed**: OpenVINO Model Server (OVMS) is a C++ inference serving system built on the OpenVINO runtime for optimized deployment of AI models on Intel architectures. It provides a unified serving layer that exposes models through multiple standard API protocols — KServe v2 (gRPC and REST), TensorFlow Serving (gRPC and REST), OpenAI-compatible chat/completion/embeddings APIs, and Cohere-compatible reranking API — making it a drop-in replacement for common inference endpoints.

The server supports a wide variety of AI workloads including traditional vision/NLP models, large language models (LLMs) with continuous batching and speculative decoding, vision-language models (VLMs), text embeddings, document reranking, speech-to-text, text-to-speech, and image generation via Stable Diffusion pipelines. Models can be loaded from local storage, Amazon S3, Google Cloud Storage, Azure Blob Storage, or HuggingFace Hub. The server features a modular architecture with a DAG pipeline scheduler for chaining multiple models, MediaPipe graph integration for complex processing flows, and an embedded Python interpreter for custom compute nodes.

Within RHOAI, OVMS is deployed as a model serving runtime container that KServe orchestrates for inference workloads. It is one of the primary inference backends available to users alongside other runtimes like vLLM and TGI.

## Architectural Analysis

OpenVINO Model Server follows a modular, plugin-based architecture centered around a singleton `Server` instance that orchestrates multiple independent modules: gRPC server, HTTP server, model manager, Python interpreter, HuggingFace pull, and C API. Each module implements a common lifecycle interface (start/shutdown) and can be independently enabled or disabled. The server supports simultaneous gRPC and REST endpoints on configurable ports, with the REST layer built on the Drogon HTTP framework and gRPC using the standard C++ gRPC library with support for multiple server instances (via `--grpc_workers`) for high-throughput scenarios.

The inference pipeline has three execution paths: (1) Direct model inference via the ModelManager for simple request-response workloads using the KServe v2 or TFS protocols, (2) DAG pipeline execution through the PipelineFactory for multi-model chaining with custom C++ plugin nodes, and (3) MediaPipe graph execution for advanced workloads like LLMs, embeddings, reranking, audio, and image generation. The MediaPipe path is particularly significant as it powers all the generative AI capabilities — the LLM calculator integrates with OpenVINO GenAI's ContinuousBatchingPipeline for token-level batching, speculative decoding (EAGLE3), prefix caching, sparse attention, and cache eviction strategies. Each calculator (LLM, embeddings, rerank, S2T, T2S, image gen) is configured via its own protobuf options message and manages its own servable lifecycle.

A notable architectural decision is that OVMS provides **no built-in TLS** — both gRPC and REST servers use plaintext connections (`InsecureServerCredentials` for gRPC, standard HTTP for REST). This is by design: in RHOAI deployments, TLS termination and authentication are handled externally by kube-rbac-proxy sidecars injected by the platform operator. The only built-in authentication mechanism is an optional API key file (`--api_key_file` or `API_KEY` env var) that enables Bearer token validation on generative endpoints (/v3/*). This separation of concerns keeps the inference server focused on performance while delegating security to the platform infrastructure.

The storage abstraction layer (`FileSystem` interface with S3, GCS, Azure, and local implementations) combined with the HuggingFace pull module provides flexible model sourcing. The filesystem factory routes based on URL prefix (`s3://`, `gs://`, `az://`, `abfs://`), and all remote backends support selective model version downloading to minimize storage usage. Path traversal protection is enforced across all backends via `isPathEscaped()` validation. The HuggingFace integration uses libgit2 for repository cloning with custom SSL certificate support, and includes a dedicated GGUF downloader for quantized LLM models.

The build system uses Bazel for the C++ core, which provides hermetic dependency management through pinned external repositories in the WORKSPACE file. The Konflux build (Dockerfile.konflux) is a complex multi-stage build that compiles OpenVINO from source (not binary), builds OpenVINO Tokenizers and GenAI from source, compiles OpenCV, and then builds OVMS with Bazel. The resulting release image is based on UBI 9 minimal and includes Python 3.12 for the embedded interpreter and demo applications. LTO (Link-Time Optimization) is enabled in the Konflux build for performance optimization.
