## Purpose

**Short**: High-performance AI model inference server built on OpenVINO, exposing TensorFlow Serving, KServe v2, and OpenAI-compatible APIs over gRPC and REST.

**Detailed**: OpenVINO Model Server (OVMS) is a C++-based inference serving system developed by Intel and packaged for Red Hat OpenShift AI. It hosts machine learning models and makes them accessible over standard network protocols (gRPC and REST), enabling remote inference for client applications written in any language. OVMS supports multiple model frameworks including OpenVINO IR, ONNX, TensorFlow, PaddlePaddle, and TFLite, with hardware acceleration for Intel CPUs, GPUs, and NPUs.

Beyond traditional model serving, OVMS provides a comprehensive generative AI platform with OpenAI-compatible chat completions, text embeddings (OpenAI API), reranking (Cohere API), image generation, speech-to-text (Whisper), and text-to-speech capabilities. It uses OpenVINO GenAI for LLM inference with continuous batching, speculative decoding, and KV cache management. The server supports model repositories on local filesystems, Amazon S3, Google Cloud Storage, and Azure Blob/File Storage, with Hugging Face Hub integration for direct model pulling and OpenVINO format conversion.

The server is built with Bazel and uses a modular architecture with MediaPipe as its computational graph framework for composing inference pipelines. It includes DAG (Directed Acyclic Graph) scheduling for multi-model pipelines, custom node support via C API and Python, and Prometheus-compatible metrics. In the RHOAI platform, OVMS runs as a serving runtime container deployed by KServe InferenceService CRs or standalone in pods.

## Architectural Analysis

OpenVINO Model Server follows a **modular monolith** architecture where the single `ovms` binary contains all functionality but is organized as independently startable modules with explicit dependency ordering. The server startup sequence is carefully orchestrated: the Servable Manager Module is started last (after gRPC and HTTP servers) so that health probes respond immediately while models are still loading. This design is critical for Kubernetes deployments where slow model loading could otherwise cause pod restarts due to failed liveness probes. The module shutdown sequence is reversed — network-facing modules shut down first to stop accepting requests before the servable manager drains in-flight inference.

The dual API surface (TFS + KServe v2 + OpenAI) is implemented through separate frontend modules that share a common backend. The HTTP REST handler uses regex-based URL routing to dispatch requests to the appropriate handler function, while the gRPC server registers three separate service implementations on the same port. The OpenAI-compatible v3 endpoints are routed through MediaPipe graph execution, which provides a flexible composition framework for chaining calculators (HttpLLMCalculator, EmbeddingsCalculator, RerankCalculator, ImageGenCalculator, S2TCalculator, T2SCalculator) with shared side packets for model servables. This approach allows the same model to be accessed through multiple API protocols without code duplication.

The storage backend abstraction layer (`FileSystem` interface with `FileSystemFactory`) provides transparent access to local filesystem, S3, GCS, and Azure storage through a URL-prefix-based factory pattern. The Model Manager runs a watcher thread that polls for configuration changes at a configurable interval (default 1 second), enabling hot-reload of model configurations without server restart. Model versions are managed through a directory structure convention (`{model_name}/{version}/`), and the system supports multiple versioning policies (latest, all, specific ranges). Cloud-stored models are downloaded to a local cache directory before loading, with MD5-based cache validation.

The LLM subsystem is the most architecturally complex feature, built on OpenVINO GenAI's `ContinuousBatchingPipeline` for high-throughput text generation. Chat template processing uses an embedded Python interpreter (pybind11) to execute Jinja2 templates, which is an unusual design choice for a C++ server but enables compatibility with the Hugging Face ecosystem's template format. Tool calling support is implemented through model-family-specific output parsers (Llama 3, Hermes3, Qwen3, Phi-4, Devstral) that detect and extract function call JSON from streaming token output. The streaming response follows the SSE (Server-Sent Events) protocol with `data: {json}\n\n` framing.

A notable security consideration is that OVMS does not implement TLS termination — the gRPC server uses `InsecureServerCredentials` and the HTTP server binds without TLS. In RHOAI deployments, TLS is terminated by the platform's ingress layer (Envoy Gateway) and authentication is enforced by kube-rbac-proxy sidecars. The built-in API key authentication mechanism (`--api_key_file` / `API_KEY` env var) provides a simple bearer token check but is not a substitute for the platform's OAuth/OIDC-based authentication. This architectural decision makes OVMS simpler but dependent on the surrounding platform for production security.
