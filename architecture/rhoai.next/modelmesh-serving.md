## Purpose

**Short**: Kubernetes operator that manages the lifecycle of ModelMesh Serving runtime deployments for multi-model inference on OpenShift AI.

**Detailed**: ModelMesh Serving is a controller-runtime based Kubernetes operator that reconciles ServingRuntime, Predictor, and InferenceService custom resources to deploy and manage ModelMesh-based multi-model serving infrastructure. The controller dynamically constructs multi-container runtime Deployments composed of a core ModelMesh container (gRPC inference engine), a REST proxy sidecar, a model puller sidecar for storage access, an OAuth proxy for authentication, and user-defined ML runtime containers (TensorFlow Serving, Triton, OVMS, TorchServe, MLServer).

The operator coordinates with an external etcd cluster for model registry state management, enabling ModelMesh's distributed model placement and routing across runtime pods. It supports both cluster-scoped and namespace-scoped deployment modes, optional HPA-based autoscaling (including scale-to-zero), TLS/mTLS for inter-component communication, and integration with KServe InferenceService CRDs for compatibility with the broader KServe ecosystem. The controller uses the manifestival library for templated Kubernetes resource management and maintains a gRPC client connection to the ModelMesh service for virtual model lifecycle operations (register, load, delete, status).

## Architectural Analysis

ModelMesh Serving follows a multi-controller architecture where three reconcilers (Service, ServingRuntime, Predictor) operate on orthogonal concerns but share state through an in-memory `MMServiceMap` and configuration provider. The ServiceReconciler manages the Kubernetes Service and ServiceMonitor per namespace, the ServingRuntimeReconciler constructs multi-container Deployments for each runtime, and the PredictorReconciler bridges the Kubernetes CRD world with the ModelMesh gRPC API for virtual model lifecycle management. This separation allows independent scaling of concerns but creates a tight coupling through the shared state map.

The deployment template system is noteworthy: runtime Deployments are constructed from a Go template (`config/internal/base/deployment.yaml.tmpl`) that defines the base pod specification including the ModelMesh container, OAuth proxy sidecar, and volume mounts. The controller then applies a chain of transformation functions (via manifestival) to inject additional containers (REST proxy, puller, built-in adapter), volumes (etcd, TLS, PVC), environment variables, and security context settings. This template-plus-transform approach provides flexibility but makes the final pod specification difficult to determine statically — one must trace through the transformation chain in `modelmesh.go`, `proxy.go`, `puller.go`, `runtime.go`, `etcd.go`, `tls.go`, and `constraints.go` to understand the complete pod composition.

The etcd integration is a distinguishing architectural feature. ModelMesh uses etcd as a distributed model registry and placement coordination store, not just for leader election. The `ModelMeshEventStream` watches etcd key ranges for model and virtual model state changes, translating them into Kubernetes reconciler events. This creates a bidirectional data flow: Kubernetes CRD changes flow through the controller to ModelMesh via gRPC, while ModelMesh state changes flow back through etcd watches to update CRD status. The per-namespace etcd secret isolation (with namespace-prefixed root paths) enables multi-tenancy within a shared etcd cluster.

The dual-scope deployment model (cluster-scoped vs. namespace-scoped) adds complexity. In cluster-scope mode, the controller watches all namespaces and uses Namespace resources as reconciliation triggers with a `modelmesh-enabled` label. In namespace-scope mode, it operates only within its own namespace. This bifurcation propagates through the RBAC manifests (separate ClusterRole vs. Role), the controller setup (different cache configurations), and the kustomize overlays (separate rbac directories). The ODH/RHOAI overlay defaults to cluster-scope mode with the ClusterRole binding.

A potential security concern is the `InsecureSkipVerify: true` setting on the ServiceMonitor TLS configuration for Prometheus scraping. While this is a common pattern for internal metrics endpoints, it means the Prometheus scraper does not validate the certificate presented by the metrics endpoint, which could be exploited in a compromised-network scenario. The TODO comment in `service_controller.go:394` acknowledges this should be updated to use a proper CA certificate.
