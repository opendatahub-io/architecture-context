## Purpose

**Short**: Kubernetes operator that manages the lifecycle of ModelMesh Serving deployments, enabling multi-model serving with intelligent model placement, scaling, and routing via an etcd-backed state store.

**Detailed**: The ModelMesh Serving Controller is a Kubernetes operator built on controller-runtime that orchestrates the deployment and management of model serving infrastructure. It manages ServingRuntime and ClusterServingRuntime custom resources to define runtime environments (such as Triton, MLServer, OVMS, and TorchServe), and Predictor/InferenceService resources to deploy trained ML models into those runtimes.

The controller dynamically constructs Deployment specifications by rendering Go templates and applying a transformation pipeline that injects sidecar containers (model puller, REST proxy, built-in adapter, oauth-proxy), configures TLS, mounts etcd secrets, and sets up type constraints. It integrates with etcd as a distributed key-value store for model registry state and uses a custom gRPC resolver for Kubernetes service discovery to communicate with the ModelMesh data plane.

The operator supports both cluster-scoped and namespace-scoped modes, scale-to-zero with configurable grace periods, horizontal pod autoscaling via HPA, and optional REST proxy for HTTP-based inference alongside the primary gRPC interface. It validates ServingRuntime configurations through a validating webhook and bridges KServe InferenceService resources to the ModelMesh Predictor abstraction.

## Architectural Analysis

The ModelMesh Serving Controller follows a template-based deployment construction pattern that is distinct from most Kubernetes operators. Rather than imperatively building Deployment specs in Go code, the controller loads a Go template (`config/internal/base/deployment.yaml.tmpl`) and applies a 14-step transformation pipeline to produce the final Deployment. This pipeline includes volume injection, sidecar container injection (puller, REST proxy, built-in adapter), etcd secret mounting, TLS configuration, and pod metadata merging. This approach provides flexibility but makes the resulting deployment topology non-obvious from static manifest analysis alone — the actual deployed pod spec depends on runtime configuration and ServingRuntime spec fields.

The etcd integration is a defining architectural characteristic. Unlike operators that rely solely on Kubernetes CRDs for state, ModelMesh uses etcd as a distributed model registry with per-namespace key prefix isolation (`{rootPrefix}/mm_ns/{namespace}`). The controller copies etcd connection secrets from its own namespace into user namespaces with adjusted root prefixes, establishing namespace isolation at the etcd level. The `ModelEventStream` component watches etcd for model and vmodel state changes, translating them into controller-runtime `GenericEvent` objects that trigger Predictor reconciliation. This creates a bidirectional data flow: the controller pushes model lifecycle commands to ModelMesh via gRPC, and ModelMesh pushes state updates back via etcd.

The controller's multi-source predictor registry design (`PredictorSource` interface) enables pluggable model input sources. The two built-in sources — native Predictor CRs and KServe InferenceService CRs — are supplemented by a plugin interface that allows external systems to feed models into the serving mesh. The `InferenceServiceRegistry` performs a complex translation from KServe's framework-specific spec (XGBoost, LightGBM, SKLearn, TensorFlow, ONNX, PyTorch, Triton, PMML) to the generic ModelMesh storage format, handling multiple storage URI schemes (s3://, gs://, pvc://, http/https including Azure Blob).

The deployment template includes an oauth-proxy sidecar with an OpenShift-specific SAR (SubjectAccessReview) policy that gates inference access based on the ability to `get services` in the runtime namespace. This provides namespace-level access control for REST inference endpoints. The gRPC path bypasses oauth-proxy and relies on optional TLS client certificates for authentication. This asymmetry means REST and gRPC endpoints have different security postures — REST has OAuth-enforced access control while gRPC relies on network policy and optional mTLS.

The scale-to-zero feature implements a grace period mechanism in the ServingRuntime reconciler. When a runtime has no assigned predictors, the controller tracks the transition time and requeues reconciliation after the configured grace period (default 60 seconds). The `runtimeInfoMap` maintains per-runtime state with timestamps to prevent premature scale-down. Combined with HPA support (CPU and memory metrics, validated via the webhook), this provides a two-tier autoscaling strategy: HPA handles load-based scaling within bounds, while scale-to-zero handles idle runtime cleanup.
