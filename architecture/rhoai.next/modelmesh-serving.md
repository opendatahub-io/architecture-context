## Purpose

**Short**: Kubernetes operator that manages the lifecycle of ModelMesh Serving deployments, orchestrating multi-model inference runtimes with scale-to-zero, autoscaling, and dynamic model placement.

**Detailed**: The ModelMesh Serving Controller is a Kubernetes operator built on controller-runtime that reconciles ServingRuntime, Predictor, and InferenceService custom resources into running ModelMesh inference deployments. It manages three primary controllers: the ServingRuntime controller (which creates and manages Deployments for each runtime with ModelMesh, REST proxy, and model puller sidecar containers), the Predictor controller (which communicates with the ModelMesh gRPC API to register, load, and manage virtual models), and the Service controller (which creates Kubernetes Services exposing inference endpoints and manages Prometheus ServiceMonitors).

The operator supports both cluster-scoped and namespace-scoped deployment modes, dynamically discovers available CRDs (InferenceService, ClusterServingRuntime, ServiceMonitor), and uses etcd as a backing store for model registry state. It integrates with the KServe ecosystem by consuming ServingRuntime and ClusterServingRuntime CRDs from KServe, and optionally watches InferenceService CRs which are internally converted to Predictor resources. The controller implements scale-to-zero with configurable grace periods, HPA-based autoscaling, and a type-constraints system that maps models to compatible runtimes.

For RHOAI deployments, the operator is deployed alongside etcd (for model registry), and the generated runtime Deployments include an oauth-proxy sidecar for authentication on the REST endpoint. Configuration is driven by a ConfigMap (`model-serving-config`) that controls images, resource limits, TLS settings, and scaling behavior.

## Architectural Analysis

The ModelMesh Serving Controller follows a multi-controller pattern where three controllers (Service, Predictor, ServingRuntime) collaborate through shared data structures and event streams. The ServiceReconciler manages the Kubernetes Service and gRPC connection lifecycle, the PredictorReconciler translates Predictor CRs into ModelMesh vmodel operations, and the ServingRuntimeReconciler generates complete Deployment specs from templated manifests. This separation of concerns allows each controller to operate independently while sharing the MMServiceMap for gRPC connection pooling across namespaces.

The deployment generation system uses a Go template (`deployment.yaml.tmpl`) processed through the Manifestival library, which is then programmatically augmented with runtime containers, sidecar containers (REST proxy, puller, oauth-proxy), TLS volumes, and storage mounts. This hybrid template+code approach allows the base deployment structure to be defined declaratively while container composition logic remains in Go code. The template includes an oauth-proxy sidecar (on port 8443) that fronts the REST proxy for RHOAI deployments, enforcing OpenShift SAR-based authorization. The ModelMesh container is always placed last in the container list to avoid blocking the pod with image pulls.

The type-constraints system (`cluster_config.go`, `constraints.go`, `model_type_labels.go`) is a sophisticated model placement engine. It generates a JSON constraint configuration stored in a ConfigMap (`tc-config`) that defines which models can run on which runtimes based on model type, protocol version, and runtime capabilities. Labels like `rt:runtime-name`, `mt:model-type`, and `pv:protocol-version` encode placement rules. The `_no_runtime` sentinel acts as an "unsupported" marker for default constraints, ensuring models are only placed on compatible runtimes.

Scale-to-zero is implemented through the ServingRuntime controller's `determineReplicasAndRequeueDuration` function, which tracks when a runtime transitions to having no active predictors. After a configurable grace period (default 60 seconds), the controller scales the Deployment to zero replicas. The HPA reconciler coexists with this logic — when scale-to-zero triggers, the HPA is deleted rather than set to minReplicas=0, and it is recreated when predictors are assigned back to the runtime. The autoscaler type is determined from the `autoscaler.knative.dev/class` annotation on the ServingRuntime metadata.

The etcd integration provides three capabilities: model registry state (EtcdRangeWatcher watches `registry` and `vmodels` key prefixes), event streaming (ModelMeshEventStream emits Kubernetes GenericEvents when model state changes in etcd), and per-namespace isolation (EtcdSecret creates individual Secrets with namespace-scoped prefixes). The controller manages a custom `kube://` gRPC resolver scheme that watches Kubernetes Endpoints resources to dynamically resolve ModelMesh service addresses, avoiding DNS-based discovery in favor of direct endpoint tracking.

A notable security concern is the `InsecureSkipVerify: true` setting in the ServiceMonitor TLS configuration (service_controller.go:346), which has a TODO comment indicating it needs to be updated to use the CA certificate. This means Prometheus metrics scraping does not validate the server certificate, which is a TLS verification gap in production.
