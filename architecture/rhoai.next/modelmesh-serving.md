## Purpose

**Short**: Kubernetes operator that manages the lifecycle of ModelMesh serving runtime deployments for multi-model inference serving.

**Detailed**: ModelMesh Serving Controller is a Kubernetes operator built on controller-runtime that orchestrates ML model inference serving using the ModelMesh distributed model serving framework. It manages three primary resources: ServingRuntimes (which define model serving containers like Triton, MLServer, OVMS, and TorchServe), Predictors (which represent individual models to be served), and InferenceServices (KServe-compatible model serving requests).

The controller dynamically creates and manages Deployments for each ServingRuntime, injecting sidecar containers (ModelMesh, REST proxy, storage puller, oauth-proxy) as needed. It communicates with ModelMesh instances via gRPC to register and manage model lifecycle (loading, unloading, transitioning between model versions). An etcd cluster serves as the coordination backend for ModelMesh instances, enabling distributed model placement and state synchronization across pods.

The operator supports both namespace-scoped and cluster-scoped modes, enabling multi-tenant model serving. It features scale-to-zero for unused runtimes, HPA-based autoscaling, TLS encryption for inference endpoints, and Prometheus metrics integration. In RHOAI deployments, it is deployed by the platform operator (rhods-operator) as one of the model serving components alongside KServe.

## Architectural Analysis

ModelMesh Serving follows a sophisticated multi-layer architecture that separates concerns between the Kubernetes control plane (the controller), the distributed model serving data plane (ModelMesh Java runtime), and the storage plane (puller sidecar). The controller does not serve inference traffic directly — it orchestrates Deployments whose pods contain a constellation of sidecar containers that collectively implement model serving.

The deployment template pipeline (`controllers/modelmesh/modelmesh.go`) applies a chain of 14 transformation functions to a base Deployment YAML template, progressively adding containers, volumes, environment variables, TLS configuration, and pod metadata. This design pattern — a functional transformation chain over a Kubernetes manifest — enables composable, testable deployment construction. Each transformation function has a single responsibility (e.g., `addRESTProxyToDeployment`, `configureMMDeploymentForTLSSecret`), and the chain order matters: TLS configuration must occur after container injection, and the ModelMesh container is explicitly moved to the last position to prevent large image pull times from blocking startup probes.

The etcd integration is architecturally noteworthy. The controller maintains a persistent watch on etcd via `EtcdRangeWatcher`, which implements a mirror-and-watch pattern with compaction recovery. Two independent watchers run per namespace: one for the virtual model registry (`vmodels/` prefix) that reads full values to extract owner metadata, and one for the concrete model registry (`registry/` prefix) that uses keys-only mode for efficiency. This dual-stream architecture allows the controller to react to both virtual model ownership changes and concrete model state transitions (loading, loaded, failed). The namespace isolation pattern — injecting `.{controller-namespace}` into etcd endpoint hostnames and appending `/mm_ns/{namespace}` to the root prefix — enables multi-tenant operation over a shared etcd cluster without requiring per-namespace etcd instances.

The InferenceService-to-Predictor adaptation layer (`pkg/predictor_source/inferenceservice_registry.go`) is a significant integration bridge. It converts KServe's InferenceService CRD (which supports single-model serving with Knative) into ModelMesh's Predictor representation (which supports multi-model serving). The conversion handles storage URI parsing for multiple backends (S3, GCS, Azure Blob, PVC, HTTP), model framework detection (XGBoost, LightGBM, SKLearn, TensorFlow, ONNX, PyTorch, Triton, PMML), and bidirectional status synchronization. This enables ModelMesh to participate in KServe's ecosystem while maintaining its own multi-model serving semantics.

The scale-to-zero implementation uses a grace-period-based approach rather than metric-driven scaling. When a runtime has no Predictors assigned to it, the `ServingRuntimeReconciler` records the transition time and requeues with the remaining grace period duration. After the grace period elapses, it scales the Deployment to zero replicas. This is distinct from the HPA-based autoscaling (which handles load-based scaling for active runtimes) and serves as a resource conservation mechanism for completely idle runtimes.
