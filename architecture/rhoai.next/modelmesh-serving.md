## Purpose

**Short**: Kubernetes controller that manages the lifecycle of ModelMesh Serving inference runtimes, orchestrating multi-model serving deployments with intelligent model placement, autoscaling, and etcd-based state coordination.

**Detailed**: ModelMesh Serving Controller is a controller-runtime based Kubernetes operator that manages the full lifecycle of model serving infrastructure. It reconciles three primary custom resources — ServingRuntime, Predictor, and InferenceService — to create and manage multi-container Deployments that host ML model inference workloads. Each runtime Deployment includes a ModelMesh sidecar (the Java-based model management engine), optional REST proxy, storage helper (puller), and the actual model runtime container (e.g., Triton, OVMS, MLServer, TorchServe).

The controller communicates with ModelMesh's gRPC API to register and manage virtual models (vmodels), enabling zero-downtime model updates through an active/target model transition pattern. It uses etcd as a distributed key-value store for model placement state, enabling intelligent routing of inference requests to the correct runtime pod. The controller supports both cluster-scoped and namespace-scoped operation modes, horizontal pod autoscaling (HPA) with scale-to-zero capability, and integrates with Prometheus for metrics collection. In RHOAI deployments, it includes an oauth-proxy sidecar on runtime pods for authentication enforcement.

## Architectural Analysis

ModelMesh Serving follows a multi-controller architecture where three controllers — ServiceReconciler, PredictorReconciler, and ServingRuntimeReconciler — cooperate to manage the full model serving lifecycle. The controllers interact through shared state (the `MMServiceMap` and `ModelEventStream`) rather than through CRD status fields, which is an unusual pattern that keeps the reconciliation loops tightly coupled but highly responsive. The ModelMeshEventStream watches etcd key ranges for vmodel and model registry changes, translating etcd events into controller-runtime generic events that trigger Predictor reconciliation. This gives sub-second model state propagation without polling.

The deployment template system (`config/internal/base/deployment.yaml.tmpl`) is notable for its complexity. Runtime Deployments are built by applying a chain of transformation functions to a base YAML template, adding containers (ModelMesh, puller, REST proxy, oauth-proxy, runtime), volumes (models-dir, domain-socket, storage-config, etcd-config, TLS certs, PVCs), and environment variables programmatically. This approach allows the controller to produce highly customized Deployments based on the ServingRuntime spec and cluster configuration, but makes it difficult to predict the final pod spec without reading the Go transformation code. The oauth-proxy sidecar is baked into the deployment template (not added dynamically), indicating RHOAI-specific authentication is a first-class concern in this fork.

The controller supports two operational modes — cluster-scoped and namespace-scoped — controlled by the `NAMESPACE_SCOPE` environment variable. In cluster scope, the controller watches all namespaces for the `modelmesh-enabled=true` label and creates namespace-specific Services owned by the Namespace resource. In namespace scope, it operates only within its own namespace with Services owned by the controller Deployment. This dual-mode design adds complexity to the RBAC configuration (separate ClusterRole and Role manifests) but enables flexible deployment topologies. The etcd secret is copied per-namespace with a namespace-specific root prefix (`mm_ns/{namespace}`), ensuring model state isolation across namespaces while sharing a single etcd cluster.

The predictor source abstraction (`pkg/predictor_source/`) is a well-designed plugin system that enables the controller to reconcile models from multiple sources — native Predictor CRs, KServe InferenceServices, and potentially external registries. The InferenceService registry converts KServe InferenceService CRs into the internal Predictor representation, supporting storage URI schemes (pvc://, s3://, gs://, https://) and framework-specific extraction (XGBoost, SKLearn, ONNX, PyTorch, TensorFlow, Triton). The caching layer (`CachedPredictorSource`) provides optimistic concurrency control with resource version tracking and a 20-second deletion grace period, preventing premature cleanup of transitioning models.

The ServingRuntime webhook validation (`servingruntime_webhook.go` and `servingruntime_validator.go`) enforces strict isolation between user-provided runtime container specs and internal ModelMesh infrastructure. Reserved container names (`mm-*`), volume names (`config-storage`, `etcd-volume`, `internal-config`, `socket-volume`, `models-dir`), and port ranges (8080, 8085, 8089, 8090, 11881-11899) are blocked to prevent conflicts with the ModelMesh sidecar, puller, and adapter containers that the controller injects. This defense-in-depth approach prevents users from accidentally breaking the multi-container pod orchestration.
