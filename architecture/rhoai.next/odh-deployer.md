## Purpose

**Short**: Deployment init-container that bootstraps the RHODS platform by creating KfDef custom resources, CRDs, monitoring stack, network policies, and dashboard configuration on OpenShift clusters.

**Detailed**: The odh-deployer is a container image designed to run alongside the opendatahub operator (rhods-operator). When executed, its `deploy.sh` shell script performs the complete initial setup of a Red Hat OpenShift Data Science (RHODS) installation. It creates namespaces (`redhat-ods-applications`, `redhat-ods-monitoring`, `rhods-notebooks`), applies CRDs for the dashboard, deploys KfDef custom resources that trigger the operator to install individual RHODS components (Dashboard, Notebook Controller, ModelMesh, Data Science Pipelines Operator, Notebooks, Monitoring, Anaconda), and sets up the full monitoring stack with Prometheus, Alertmanager, and Blackbox Exporter.

The deployer handles both managed service (OpenShift Dedicated / OSD) and self-managed installation modes. In managed mode, it configures Dead Man's Snitch, PagerDuty, and SMTP alerting integrations. In self-managed mode, it creates the `rhods-admins` group and applies simplified configurations. It also deploys ISV (Independent Software Vendor) application tiles for the dashboard, serving runtime templates for model serving, network policies for inter-namespace communication, and pod security RBAC bindings.

This component represents an earlier RHODS deployment pattern (pre-2.x). In newer RHOAI versions, much of this functionality has been absorbed into the rhods-operator itself. The deployer uses a simple `oc apply` approach with kustomize overlays and KfDef CRs, which the operator then reconciles to deploy actual component workloads from manifests bundled at `/opt/manifests/odh-manifests.tar.gz` inside the operator image.

## Architectural Analysis

The odh-deployer follows a "deployment script in a container" pattern that was common in early Kubernetes operator ecosystems before operators fully subsumed deployment logic. Rather than embedding all deployment logic in the operator's reconcile loops, the deployer runs as a one-shot init container that uses the `oc` CLI to imperatively create resources. This design has the advantage of simplicity and transparency (the entire deployment sequence is readable in a single bash script), but it creates a gap in the operator's ability to reconcile drift: resources created by `deploy.sh` are not continuously watched or repaired by the operator unless they happen to be KfDef-managed.

The monitoring architecture is notably self-contained. The deployer creates a complete Prometheus + Alertmanager + Blackbox Exporter stack in the `redhat-ods-monitoring` namespace, separate from the cluster's built-in monitoring. This dedicated monitoring instance federates selected metrics from `prometheus-k8s.openshift-monitoring.svc` (HAProxy backend responses, controller reconcile totals, container CPU/memory, kubelet volume stats, pod status, build status, statefulset replicas) and directly scrapes RHODS component metrics endpoints. The SLO-based alerting uses multi-window burn rate calculations (5m/1h, 30m/6h, 2h/1d, 6h/3d windows) across dashboard, notebook controller, model controller, modelmesh controller, and data science pipelines operator. All externally-exposed monitoring endpoints use the OpenShift `oauth-proxy` sidecar pattern with Subject Access Review (SAR) authorization, while internal `/metrics` paths skip authentication via regex.

The deployer has a clear separation between managed service (OSD) and self-managed deployment paths. The managed path configures SRE alerting (PagerDuty for critical alerts, email via SMTP for user-facing notifications like PVC usage warnings, Dead Man's Snitch for monitoring liveness). It reads multiple pre-provisioned secrets from the `redhat-ods-operator` namespace that are injected by the OSD addon framework. It also applies environment-specific blackbox exporter configurations (external vs internal TLS verification). The self-managed path omits SRE alerting, creates the `rhods-admins` group instead of using `dedicated-admins`, and uses a simplified ConsoleLink title. This conditional logic is handled entirely in bash with `if/else` blocks rather than kustomize overlays.

The network policy design uses a two-tier approach: port-based ingress rules allow traffic on specific service ports from any source, while namespace-selector rules allow any traffic from namespaces labeled with `opendatahub.io/generated-namespace=true`. The deployer labels all three RHODS namespaces with this label during creation. The operator namespace policy is more restrictive, only allowing ingress from labeled namespaces to pods with `name=rhods-operator` label, with no port restrictions. This design provides basic network isolation while allowing intra-platform communication. Note that there are no egress policies, meaning outbound traffic from RHODS pods is unrestricted.

The KfDef CR pattern used by the deployer references manifests from a tarball (`file:///opt/manifests/odh-manifests.tar.gz`) bundled inside the operator image. Each KfDef specifies a `repoRef` path within that tarball and optionally passes `RELATED_IMAGE_*` environment variables as kustomize parameters. This allows the operator to use OLM-managed image references (for disconnected/air-gapped installations) while letting the manifests define the actual deployment topology. The deployer creates 8 distinct KfDef CRs covering dashboard, notebook controller, model mesh, DSPO, notebooks, monitoring, modelmesh-monitoring, and anaconda, plus a 9th (rhods-osd-configs) in managed service mode only.
