## Purpose

**Short**: A deployment container that bootstraps the RHOAI (Red Hat OpenShift AI) platform by creating namespaces, applying KfDef custom resources, configuring monitoring, network policies, and dashboard resources.

**Detailed**: The odh-deployer is a container image designed to be run alongside the opendatahub operator (rhods-operator). When executed, it performs the initial platform bootstrapping sequence: creating the required namespaces (`redhat-ods-applications`, `redhat-ods-monitoring`, `rhods-notebooks`), applying namespace labels for pod security and monitoring, deploying KfDef custom resources to trigger component installations (dashboard, notebook controller, model mesh, data science pipelines, notebooks, monitoring, anaconda), configuring the Prometheus-based monitoring stack with alerting (PagerDuty, Dead Man's Snitch, SMTP email), setting up network policies across all platform namespaces, applying RBAC for pod security, and creating ConsoleLink and dashboard configuration resources.

The deployer distinguishes between managed service (OSD/addon) and self-managed environments, applying different configurations for ISV application tiles, monitoring alerting pipelines, and RBAC groups based on the detected environment. It is a legacy RHOAI 1.x/2.x deployment mechanism — the platform has been transitioning away from KfDef-based deployment toward the DSCInitialization/DataScienceCluster CRD-based approach managed directly by the operator in RHOAI 2.x+.

## Architectural Analysis

The odh-deployer is a legacy deployment mechanism from the RHOAI 1.x era that predates the DSCInitialization/DataScienceCluster CRD approach introduced in RHOAI 2.x. It uses a "deploy-once" pattern: a container image packaging the `oc` CLI alongside YAML manifests and a bash script that imperatively bootstraps the entire RHOAI platform. This design creates a tight coupling between the deployer's release cadence and the component versions it references through `RELATED_IMAGE_*` environment variables in the KfDef specifications.

The monitoring architecture is notably comprehensive for a shell-script deployer. It deploys a dedicated Prometheus instance in `redhat-ods-monitoring` that federates cluster-level metrics from OpenShift's built-in Prometheus while also directly scraping component-specific metrics endpoints. The SLO burn-rate recording rules and multi-window alerting (5m/1h, 30m/6h, 2h/1d, 6h/3d windows) follow Google SRE practices. The alerting pipeline has three tiers: PagerDuty for critical SRE alerts, email for user-facing notifications (PVC usage), and Dead Man's Snitch for pipeline liveness. The entire monitoring stack is secured with OAuth proxy sidecars using OpenShift SAR (SubjectAccessReview) for authorization.

A significant architectural concern is the environment detection mechanism: the script checks for the existence of `addon-managed-odh-catalog` CatalogSource to determine whether to apply managed-service or self-managed configurations. This bifurcation affects monitoring alerting routes (PagerDuty receivers are only wired for managed service), ISV application tiles (managed gets NVIDIA/RHOAM/Starburst; self-managed gets Starburst Enterprise), and admin group configuration (`dedicated-admins` vs `rhods-admins`). The blackbox exporter also has two variants — external (with `insecure_skip_verify: true`) for non-redhat.com clusters and internal (with proper TLS verification) for internal clusters — which is a pragmatic but security-relevant distinction.

The network policy design follows a namespace-isolation pattern with explicit port allowlists. The `redhat-ods-applications` namespace allows ingress on ports 8443, 8080, 8081, 5432, 8082, 8099, and 8181 from any source, plus unrestricted ingress from namespaces labeled `opendatahub.io/generated-namespace=true`. The monitoring namespace similarly restricts to ports 443, 9115, 8443, 9091, and 10443. The operator namespace restricts to pods labeled `name: rhods-operator` from generated namespaces only. This is a coarse-grained network segmentation — it allows any pod within the namespace to receive traffic on the listed ports regardless of the specific service.

The pod security model grants `anyuid` SCC (Security Context Constraints) to specific service accounts in the applications namespace (modelmesh-controller, notebook-controller, odh-model-controller, rhods-dashboard, modelmesh) and broadly to all service accounts in the monitoring namespace. The Prometheus deployment runs as UID 1000/GID 2000 with `runAsNonRoot: true` and `fsGroup: 2000`, which is appropriate for the persistent volume access pattern.
