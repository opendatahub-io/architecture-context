## Purpose

**Short**: A containerized shell-script deployer that bootstraps Red Hat OpenShift Data Science (RHODS/RHOAI) platform components by creating namespaces, KfDef custom resources, monitoring infrastructure, network policies, dashboard configuration, and ISV application tiles on an OpenShift cluster.

**Detailed**: The odh-deployer is a deployment orchestration container designed to run alongside the OpenDataHub/RHODS operator. It packages a Bash script (`deploy.sh`) together with KfDef manifests, monitoring configuration, network policies, dashboard CRDs/configs, and partner integrations into a UBI-based container image that includes the `oc` CLI tool. When executed, the container's entrypoint script creates the required namespaces (`redhat-ods-applications`, `redhat-ods-monitoring`, `rhods-notebooks`), applies labels, deploys KfDef custom resources to trigger component installation via the operator, sets up a complete Prometheus/Alertmanager monitoring stack with SLO-based alerting, configures network policies for namespace isolation, installs dashboard ISV application tiles, and provisions serving runtime templates for model serving.

The deployer differentiates between managed service (OSD) and self-managed environments. In managed environments, it configures Dead Man's Snitch, PagerDuty, SMTP alerting, and dedicated-admins RBAC. In self-managed environments, it creates a `rhods-admins` group and applies a simplified configuration. This is a legacy RHODS 1.x deployment mechanism that predates the newer operator-driven DSC/DSCI approach used in RHOAI 2.x+.

## Architectural Analysis

The odh-deployer represents an early-generation deployment pattern for the RHODS/RHOAI platform — a "deployer pod" approach where a container image bundles scripts and manifests, then executes `oc apply` commands imperatively against the cluster API. This pattern was common before the platform adopted a declarative operator-driven model with DataScienceCluster (DSC) and DSCInitialization (DSCI) custom resources in RHOAI 2.x+. The deployer is tightly coupled to the KfDef CRD from Kubeflow, which served as the primary component lifecycle mechanism in the ODH 1.x era.

The monitoring architecture is notably self-contained — the deployer provisions its own Prometheus and Alertmanager deployment rather than relying on the OpenShift User Workload Monitoring stack. This standalone monitoring stack federates metrics from the cluster Prometheus, scrapes component metrics endpoints directly, and uses a blackbox exporter to probe user-facing service availability. The SLO framework is well-developed with multi-window burn rate calculations (5m, 30m, 1h, 2h, 6h, 1d, 3d) for all major components. However, the monitoring configuration is highly imperative — `deploy.sh` uses `sed -i` to template secrets, hostnames, and tokens directly into YAML files at deployment time, making the configuration non-idempotent and fragile during upgrades.

The managed-vs-self-managed bifurcation is a significant architectural concern. The deployer detects the environment by checking for the `addon-managed-odh-catalog` CatalogSource in the operator namespace. In managed mode, it configures PagerDuty, Dead Man's Snitch, SMTP alerting, and OSD-specific RBAC (dedicated-admins). In self-managed mode, it creates a `rhods-admins` group and skips the external alerting integrations. The ISV application tiles also differ: managed service gets NVIDIA, RHOAM, and Starburst Galaxy, while self-managed gets Starburst Enterprise. This two-path approach, while functional, creates maintenance burden as configurations drift between variants.

Security-wise, the deployer grants `anyuid` SCC to several service accounts across the applications and monitoring namespaces via RoleBindings, which is a permissive posture necessary for components like Prometheus that require specific UIDs. Network policies are applied to all three namespaces (applications, monitoring, operator) with ingress restricted to pods from namespaces labeled `opendatahub.io/generated-namespace=true`. The Prometheus deployment runs as non-root (UID 1000, GID 2000) with fsGroup security context, and both Prometheus and Alertmanager UIs are fronted by OpenShift OAuth proxies that enforce SubjectAccessReview authorization.

The deployer contains several upgrade-specific code paths (e.g., removing old ClusterRoleBindings, deleting legacy DSPO deployments, cleaning up buildconfigs) marked with TODO comments for removal in future releases. These migration shims accumulate technical debt and suggest that the deployer was historically the primary mechanism for handling version-to-version state transitions, a responsibility that has since moved to the operator's upgrade reconciliation logic.
