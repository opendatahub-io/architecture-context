## Purpose

**Short**: A deployment container that bootstraps RHODS (Red Hat OpenShift Data Science) platform components by creating KfDef custom resources and configuring monitoring, networking, dashboard ISVs, and partner integrations.

**Detailed**: odh-deployer is a container image that runs alongside the opendatahub operator (rhods-operator) to perform initial platform setup. When executed, its `deploy.sh` script creates the necessary Kubernetes namespaces (`redhat-ods-applications`, `redhat-ods-monitoring`, `rhods-notebooks`), applies CRDs for the ODH Dashboard ecosystem, creates KfDef custom resources to trigger component installations (dashboard, notebook controller, model mesh, data science pipelines, monitoring, anaconda), and configures the complete monitoring stack (Prometheus, Alertmanager, blackbox exporter).

The deployer differentiates between managed service (OSD/addon) and self-managed installations. In managed service mode, it configures Dead Man's Snitch, PagerDuty, SMTP alerting, and OSD-specific RBAC. In self-managed mode, it skips SRE alerting and creates the `rhods-admins` group. The deployer also manages ISV (Independent Software Vendor) application tiles for the dashboard, serving runtime templates for model serving, network policies across all namespaces, and pod security RBAC bindings.

This component belongs to the legacy RHODS deployment model (pre-RHOAI 2.x/3.x). In newer versions of the platform, much of this functionality has been absorbed into the rhods-operator itself. The deployer uses the KfDef (Kubeflow Definition) API to trigger component installations through the opendatahub operator's reconciliation loops.

## Architectural Analysis

The odh-deployer represents an early-generation deployment pattern for RHODS where platform bootstrap was handled by a separate container rather than being integrated into the operator itself. The `deploy.sh` script is effectively a procedural orchestrator — it creates namespaces, applies CRDs, creates KfDef resources, and configures monitoring in a linear sequence. This contrasts with the declarative, reconciliation-based approach used in newer RHOAI versions where the operator directly manages component lifecycle through DSCInitialization and DataScienceCluster CRs.

The monitoring stack is the most architecturally significant part of this repository. It deploys a dedicated Prometheus instance in `redhat-ods-monitoring` that federates metrics from the platform Prometheus and directly scrapes component metrics endpoints. The SLO-based alerting rules implement multi-window, multi-burn-rate error budget tracking — a sophisticated pattern that measures availability across 5m/30m/1h/2h/6h/1d/3d windows and fires alerts at different severity levels based on burn rate thresholds. This is applied consistently to the dashboard, notebook spawner, model controller, ModelMesh controller, and DSPO. The Prometheus and blackbox exporter deployments both use init containers that block startup until all upstream services are available, creating an implicit dependency chain.

The security model relies heavily on OpenShift-specific constructs. OAuth proxy sidecars protect Prometheus (port 9091), Alertmanager (port 10443), and the blackbox exporter (port 9114) with SAR-based authorization. The `/metrics` endpoints are explicitly excluded from auth via `skip-auth-regex`, which is notable as it exposes operational metrics without authentication. Network policies use the `opendatahub.io/generated-namespace=true` label as a trust boundary — only namespaces with this label can reach RHODS services. The pod security RoleBindings grant `system:openshift:scc:anyuid` broadly to application service accounts, which is a permissive security posture justified by the legacy container requirements of some RHODS components.

A notable design decision is the managed-vs-self-managed bifurcation. The deployer detects the environment by checking for the `addon-managed-odh-catalog` CatalogSource in the operator namespace. This single check gates the entire alerting configuration (PagerDuty, SMTP, Dead Man's Snitch), ISV selection (managed-service ISVs vs on-prem ISVs), and admin group configuration (`dedicated-admins` vs `rhods-admins`). The sed-based template substitution for monitoring configuration (replacing placeholders like `<pagerduty_token>`, `<smtp_host>`, `<snitch_url>`) is fragile — errors in secret extraction silently produce invalid configurations.

The KfDef resources reference manifests from `file:///opt/manifests/odh-manifests.tar.gz`, indicating that the operator bundles component manifests as a tarball. Image references use `RELATED_IMAGE_*` environment variables, which is the standard OLM pattern for supporting disconnected/air-gapped clusters. The OVMS serving runtime templates define multi-model serving configurations supporting OpenVINO IR, ONNX, and TensorFlow model formats, with both CPU-only and GPU-enabled variants.
