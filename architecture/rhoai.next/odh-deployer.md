## Purpose

**Short**: Bootstrap container that deploys RHODS (Red Hat OpenShift Data Science) platform components by creating KfDef custom resources and configuring monitoring, networking, and dashboard resources on an OpenShift cluster.

**Detailed**: odh-deployer is a container image designed to run as a one-shot deployment job alongside the opendatahub/rhods operator. It packages a comprehensive shell script (`deploy.sh`) together with Kubernetes manifests (KfDef CRs, CRDs, NetworkPolicies, RBAC bindings, monitoring stack, dashboard configurations, and ISV application tiles) into a UBI8-based container image.

When executed, the deployer creates the required OpenShift projects (`redhat-ods-applications`, `redhat-ods-monitoring`, `rhods-notebooks`), applies pod security labels and RBAC bindings, deploys Dashboard CRDs and ISV application tiles, creates KfDef resources to trigger the opendatahub operator to install components (Dashboard, Notebook Controller, Model Mesh, Data Science Pipelines Operator, Notebooks, Monitoring, Anaconda), and configures the full Prometheus/Alertmanager monitoring stack with SLO-based alerting, PagerDuty integration, Dead Man's Snitch, blackbox endpoint probing, and email notifications.

The deployer distinguishes between managed service (OSD/addon) and self-managed environments via the presence of the `addon-managed-odh-catalog` CatalogSource, applying different ISV tiles, RBAC policies, monitoring configurations, and alerting routes accordingly. This is a legacy RHOAI 1.x/2.x deployment pattern; newer RHOAI 3.x releases use the rhods-operator directly for component lifecycle management.

## Architectural Analysis

The odh-deployer represents a **legacy RHOAI 1.x/2.x deployment pattern** where platform component lifecycle was managed through a combination of an init container (this deployer) and the KfDef custom resource. The deployer acts as a bootstrap orchestrator — it creates namespaces, applies CRDs, and creates KfDef resources that signal the opendatahub operator to pull and apply component manifests from the `odh-manifests` tarball bundled at `/opt/manifests/odh-manifests.tar.gz`. This pattern has been largely superseded in RHOAI 3.x where the rhods-operator directly manages component lifecycle through DSCInitialization and DataScienceCluster CRDs.

A notable architectural decision is the **tight coupling between monitoring initialization and component availability**. The Prometheus and Blackbox Exporter deployments include init containers that actively poll for the availability of 5 component services (dashboard, notebook controller, model controller, modelmesh controller, DSPO) with a 230-iteration retry loop at 10-second intervals (approximately 38 minutes maximum wait). This creates a hard dependency chain where monitoring cannot start until all components are healthy, but also means monitoring startup failures can cascade from any single component failure.

The **monitoring architecture is sophisticated for a shell-deployed component**. It deploys a full Prometheus stack with OAuth proxy sidecars (2.x auth pattern using ose-oauth-proxy), Alertmanager with PagerDuty/DMS/SMTP receivers, and a Blackbox Exporter for endpoint probing. SLO burn rate alerting is implemented for all major components (Dashboard, Notebook Controller, Model Controller, Modelmesh, DSPO) at multiple time windows (5m, 30m, 1h, 2h, 6h, 1d, 3d). The Prometheus configuration federates from the cluster-level Prometheus to obtain HAProxy backend metrics and Kubernetes resource metrics.

The **managed vs self-managed bifurcation** is a core design pattern. The deployer checks for `addon-managed-odh-catalog` CatalogSource in the operator namespace to determine the environment type. Managed environments get full SRE alerting (PagerDuty, Dead Man's Snitch, SMTP notifications, dedicated-admins RBAC), while self-managed environments get a minimal monitoring stack with a custom `rhods-admins` group. An additional check for "aisrhods" in the console route hostname disables SRE alerting on engineering/test clusters.

**Security observations**: The deployer grants `anyuid` SCC to multiple service accounts in the applications namespace, which is permissive but required for the monitoring and modelmesh workloads. The Prometheus/Alertmanager secrets are generated at deploy time using `openssl rand -hex 32`, which is cryptographically sound but means secrets change on every redeployment. The Segment.io analytics key is committed to the repository in base64 (not truly secret). The blackbox exporter's internal configuration uses `insecure_skip_verify: true`, which is acceptable for internal cluster probing but would be a concern if used externally. The `oc.tar.gz` download in the Dockerfile uses an unpinned "latest" URL without checksum verification, which is a supply chain risk.
