## Purpose

**Short**: Asynchronous queue-based worker that pulls inference jobs from message queues and dispatches them to an inference gateway, decoupling job submission from immediate execution.

**Detailed**: The Async Processor (AP) is a composable component within the llm-d ecosystem that enables efficient processing of latency-insensitive inference workloads. It functions as an asynchronous worker that reads jobs from a message queue (Redis Sorted Sets, Redis Pub/Sub, or GCP Pub/Sub), applies configurable dispatch gating based on system capacity (via Prometheus metrics or Redis-based budgets), and forwards requests to an inference gateway (IGW) over HTTP. Results are written back to a result queue for downstream consumers.

The AP follows a "Bring Your Own Queue" (BYOQ) design principle — all prioritization, routing, retries, and scaling are decoupled from the message queue implementation. The system supports multiple concurrent queues with per-queue dispatch gates, configurable concurrency, exponential backoff retries with deadline awareness, and per-attribute quota management (rate limiting and concurrency limits). It integrates with Prometheus for metric-based flow control gates that respond to inference pool saturation and EPP queue depth, ensuring async workloads only consume slack capacity without impacting online serving.

The repository also provides a Go SDK (`producer/` package) that external applications use to submit requests to and retrieve results from the Redis sorted-set queue, making it a client library as well as a deployable service.

## Architectural Analysis

The Async Processor follows a clean pipeline architecture with three pluggable layers: message queue transport (Flow interface), dispatch gating (DispatchGate/AttributeGate interfaces), and inference dispatch (InferenceClient interface). The `pipeline` package defines the contracts, and concrete implementations live in `pkg/redis`, `pkg/pubsub`, and `pkg/async/inference/flowcontrol`. This separation allows switching between Redis and GCP Pub/Sub without changing worker logic, and adding new gate types without modifying queue code.

The dispatch gate system is particularly sophisticated. The `GateFactory` creates gates from configuration, supporting seven gate types: constant, redis, redis-quota, prometheus-saturation, prometheus-budget, prometheus-query, and composite (which nests other gates). The `prometheus-budget` gate implements a cascade pattern using `CascadeMetricSource` — it first tries EPP flow control queue size metrics, and falls back to vLLM running request counts when EPP metrics are unavailable. Both sources compute `max_SYS = ready_pods × max_concurrency` dynamically from `inference_pool_ready_pods`, and the gate closes when the dispatch budget `D ≤ baseline`. The `CachedMetricSource` wrapper adds a TTL cache (default 5s) to reduce Prometheus query load, which is essential given that budget is checked on every poll cycle.

The worker design uses a fan-out pattern: `N` goroutines (configurable via `--concurrency`, default 8) read from a single merged channel produced by the `RandomRobinPolicy`, which uses `reflect.Select` to randomly multiplex across multiple queue channels. Each worker independently manages request timeouts (bounded by both `--request-timeout` and the message deadline), retry decisions, and result publishing. The retry mechanism uses exponential backoff with equal jitter (`[temp/2, temp)`) capped at 60 seconds, and respects server-specified `Retry-After` headers. For the Redis sorted-set implementation, the retry worker uses exponential backoff polling (1s base, 30s max) to avoid hammering Redis when idle.

Graceful shutdown is carefully implemented across all flow implementations. In-flight requests are re-enqueued to Redis on context cancellation, the retry worker flushes pending retries, and the result worker drains its channel before exiting. The `drainWg` sync.WaitGroup ensures all background goroutines complete before the process exits, and the 130-second `terminationGracePeriodSeconds` in the Helm chart provides adequate time for draining. The sorted-set flow additionally stores active quota releases in a `sync.Map` and releases them during result processing to prevent quota leaks.

The producer SDK (`producer/` package) is a separate Go module designed for external applications to submit requests and retrieve results. It uses the same Redis sorted-set structure as the consumer side, with deadline-based scoring for priority ordering. The `WithRedisClient` option allows callers to inject instrumented Redis clients (e.g., with OpenTelemetry hooks). Notably, the producer module has its own `go.mod` and `go.sum`, making it independently importable without pulling in the full async-processor dependency tree.
