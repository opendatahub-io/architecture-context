## Purpose

**Short**: Kubernetes operator that manages distributed training jobs across multiple ML frameworks (PyTorch, TensorFlow, MPI, JAX, XGBoost, PaddlePaddle).

**Detailed**: The Kubeflow Training Operator (KFTO) is a Kubernetes-native controller that automates the lifecycle management of distributed machine learning training workloads. It provides six Custom Resource Definitions (CRDs) â€” `PyTorchJob`, `TFJob`, `MPIJob`, `JAXJob`, `XGBoostJob`, and `PaddleJob` â€” each tailored to the distributed communication patterns of their respective ML frameworks.

For each job type, the operator handles pod creation with framework-specific environment variables (e.g., `TF_CONFIG` for TensorFlow, `MASTER_ADDR`/`WORLD_SIZE` for PyTorch), headless service creation for pod-to-pod DNS discovery, gang scheduling integration with Volcano and scheduler-plugins, elastic training support (PyTorch and PaddlePaddle), and job lifecycle management including failure detection, restart policies, and cleanup. The MPI controller additionally creates ServiceAccounts, Roles, RoleBindings, and ConfigMaps to enable launcher-to-worker `kubectl exec` communication.

In the RHOAI distribution, the operator is deployed into the `opendatahub` namespace with the `kubeflow-` name prefix, FIPS-compliant builds via `GOEXPERIMENT=strictfipsruntime`, and Prometheus metrics monitoring via PodMonitor. The operator uses webhook-based validation for all job types (except MPI) to enforce schema correctness at admission time.

## Architectural Analysis

The training-operator follows a unified controller architecture with a shared base (`JobController` in `pkg/controller.v1/common/`) that provides common reconciliation logic â€” pod lifecycle management, service creation, gang scheduling, status tracking, and expectation-based consistency â€” while each framework controller implements a `ControllerInterface` to inject framework-specific behavior. This design cleanly separates the generic Kubernetes resource management from framework-specific concerns like environment variable injection, init container configuration, and success criteria determination.

The MPI controller stands out architecturally from the other five controllers. While PyTorch, TensorFlow, JAX, XGBoost, and PaddlePaddle use headless services for pod-to-pod DNS discovery and direct network communication, MPI uses a fundamentally different pattern: a launcher pod that executes commands in worker pods via `kubectl exec`. This requires the MPI controller to create additional RBAC resources (ServiceAccount, Role, RoleBinding) and a ConfigMap containing hostfiles and shell scripts per job. The launcher pod receives a `kubectl-delivery` init container that copies the kubectl binary into a shared volume. This file-based communication model is inherent to MPI's `mpirun` execution pattern.

The operator implements a label-filtered informer cache (introduced in `main.go` lines 81-115) that restricts the controller-runtime cache to only watch Pods, Services, ConfigMaps, ServiceAccounts, Secrets, Roles, and RoleBindings carrying the `training.kubeflow.org/operator-name` label. This is a critical memory optimization â€” without it, the cache would load all cluster-wide resources of these types, causing unbounded memory growth in large clusters. ConfigMaps and Secrets are additionally excluded from the client cache (`DisableFor`) to ensure direct API reads for configuration lookups, avoiding stale cached data.

Gang scheduling is implemented as a pluggable strategy via `GangSchedulingSetupFunc` callbacks. The operator supports two backends â€” Volcano (`scheduling.volcano.sh/v1beta1.PodGroup`) and scheduler-plugins (`scheduling.x-k8s.io/v1alpha1.PodGroup`) â€” with distinct behavior: Volcano delays pod creation until the PodGroup transitions out of "Pending" phase, while scheduler-plugins does not delay. The gang scheduler is selected at startup via the `--gang-scheduler-name` flag, and the operator validates the corresponding CRD exists before proceeding. PodGroup resource calculations sort replicas by PriorityClass value and accumulate resource requests up to the `MinMember` threshold.

Elastic training is supported for PyTorch (via `ElasticPolicy` with C10d/etcd rendezvous and HPA integration) and PaddlePaddle (via `PaddleElasticPolicy`). The PyTorch elastic implementation is the most sophisticated: it configures the `torch.distributed.elastic` (TorchElastic) environment with rendezvous backend, endpoint, and scaling bounds, and optionally creates an HPA targeting custom metrics specified in the ElasticPolicy. TensorFlow supports a lighter-weight "dynamic worker" mode where workers can be added or removed without restarting the job, implemented via sparse `TF_CONFIG` JSON with dictionary-keyed worker entries instead of positional arrays.
