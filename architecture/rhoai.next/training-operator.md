## Purpose

**Short**: Kubernetes operator that manages distributed training jobs across six ML frameworks (PyTorch, TensorFlow, MPI, XGBoost, PaddlePaddle, JAX).

**Detailed**: The Kubeflow Training Operator (KFTO) is a Kubernetes-native controller that orchestrates distributed machine learning training workloads. It defines six Custom Resource Definitions (CRDs) — PyTorchJob, TFJob, MPIJob, XGBoostJob, PaddleJob, and JAXJob — each tailored to a specific ML framework's distributed training semantics. When a user creates a training job CR, the operator reconciles the desired state by creating worker Pods, headless Services for inter-pod DNS discovery, and optionally PodGroups for gang scheduling.

The operator follows a shared controller architecture where all six framework controllers inherit from a common `JobController` base that handles pod lifecycle, service management, status tracking, and gang scheduling integration. Each framework-specific controller adds custom environment variable injection (e.g., `TF_CONFIG` for TensorFlow, `MASTER_ADDR`/`WORLD_SIZE` for PyTorch), init containers for master discovery, elastic scaling support, and framework-appropriate success/failure semantics.

In the RHOAI distribution, the operator is deployed into the `opendatahub` namespace with a `kubeflow-` name prefix, configured via kustomize overlays. It integrates with OpenShift monitoring via PodMonitor for Prometheus metrics scraping, creates NetworkPolicies for PyTorchJob pod-to-pod isolation, and supports optional gang scheduling through Volcano or scheduler-plugins PodGroups.

## Architectural Analysis

The training-operator follows a **composable controller pattern** where all six framework controllers inherit from a shared `JobController` base class (`pkg/controller.v1/common/job_controller.go`). This base class handles the common lifecycle: pod creation/deletion, service management, status tracking, gang scheduling integration, and expectation-based consistency guarantees. Each framework-specific controller implements a small interface (`Controller`) that customizes environment variable injection (`SetClusterSpec`), port extraction (`GetPortsFromJob`), status determination (`UpdateJobStatus`), and master role identification (`IsMasterRole`). This pattern keeps the codebase DRY — approximately 80% of the logic is shared — while allowing each framework's unique distributed training semantics to be expressed clearly.

The **MPI controller is architecturally distinct** from the other five controllers. While PyTorch, TensorFlow, XGBoost, Paddle, and JAX all follow a symmetric pod pattern (create N pods, inject environment variables, let the framework handle coordination), MPI uses an asymmetric launcher/worker pattern. The launcher pod receives an init container that delivers `kubectl`, a ConfigMap with `kubexec.sh` (a script that uses `kubectl exec` to run commands on worker pods), and a `hostfile` listing worker DNS names. The launcher runs `mpirun` which orchestrates distributed execution by exec-ing into workers via the Kubernetes API. This requires additional RBAC setup (ServiceAccount, Role, RoleBinding per job) that the other controllers don't need — the MPI controller creates these resources dynamically during reconciliation.

The operator uses **expectation-based consistency** (`pkg/controller.v1/expectation/`) to handle the inherent race condition between issuing a Kubernetes API create/delete request and seeing the result reflected in the informer cache. When the controller creates a pod, it raises an "expectation" that is only satisfied when the informer reports the new pod. Reconciliation is skipped while expectations are outstanding, preventing duplicate resource creation. A TTL mechanism ensures stale expectations don't block progress indefinitely if a create/delete event is lost.

A notable RHOAI-specific feature is the **PyTorchJob NetworkPolicy** (`pkg/controller.v1/pytorch/pytorchjob_controller.go`, lines 540-564). For each PyTorchJob, the controller creates a NetworkPolicy that restricts ingress to only pods belonging to the same job (for inter-worker training traffic) and pods from the `openshift-monitoring` namespace on port 8080 (for Prometheus metrics scraping). This provides workload isolation without breaking monitoring — a pattern that other framework controllers do not implement. The asymmetry suggests this was added for PyTorchJob first due to its prevalence in RHOAI workloads and may be extended to other job types in the future.

The operator's **memory optimization** is noteworthy: the informer cache (`newCacheOptions` in `main.go`, lines 81-115) uses label selectors to restrict the cache to only operator-managed resources (those with the `training.kubeflow.org/operator-name` label). Without this filtering, the cache would load all Pods, Services, and ConfigMaps cluster-wide, causing unbounded memory growth in multi-tenant clusters. Additionally, ConfigMaps and Secrets are excluded from the cache entirely (`DisableFor`) to ensure direct API reads for configuration lookups, avoiding stale data.
