## Purpose

**Short**: Kubernetes operator that manages distributed training jobs across multiple ML frameworks (PyTorch, TensorFlow, XGBoost, MPI, PaddlePaddle, JAX).

**Detailed**: The Kubeflow Training Operator (KFTO) is a controller-runtime based Kubernetes operator that provides Custom Resource Definitions (CRDs) for orchestrating distributed machine learning training workloads. It supports six ML framework types—PyTorchJob, TFJob, XGBoostJob, MPIJob, PaddleJob, and JAXJob—each with its own CRD, controller, and validation webhook. The operator manages the full lifecycle of training jobs: creating worker and parameter server pods with appropriate environment variables for distributed communication, creating headless services for DNS-based pod discovery, managing pod cleanup policies, and tracking job status through conditions.

The operator integrates with gang scheduling systems (Volcano and scheduler-plugins) to ensure all training pods are co-scheduled, avoiding deadlocks in distributed training. For RHOAI, the operator is deployed into the `opendatahub` namespace with Konflux-built FIPS-compliant images. The PyTorch controller additionally supports elastic training with HPA-based autoscaling and NetworkPolicy creation for pod isolation. The MPI controller implements a distinct launcher/worker pattern with kubectl-delivery init containers and RBAC resources for pod exec capabilities.

## Architectural Analysis

The Kubeflow Training Operator follows a well-established plugin architecture pattern where a common controller framework (`pkg/controller.v1/common/`) provides shared reconciliation logic for pod lifecycle, service management, gang scheduling, and status tracking. Each ML framework (PyTorch, TensorFlow, XGBoost, MPI, PaddlePaddle, JAX) implements the `ControllerInterface` to define framework-specific behavior: cluster specification (environment variables for distributed communication), port configuration, master role identification, and status update policies. This design keeps the 6 controllers relatively thin—each one is 300-600 lines of framework-specific code—while the common framework handles the ~2000 lines of shared logic.

The MPI controller is architecturally distinct from the other five. While PyTorch, TensorFlow, XGBoost, PaddlePaddle, and JAX all use a master/worker pattern where each pod runs training code and communicates via DNS-resolvable headless services, MPI uses a launcher/worker pattern where the launcher pod executes `mpirun` which uses `kubectl exec` to run commands on sleeping worker pods. This requires the MPI controller to create additional resources (ConfigMaps for hostfile and kubexec.sh, ServiceAccount, Role, and RoleBinding for the launcher) and to override the common service reconciliation to avoid creating unnecessary services. The MPI controller also uniquely does not set environment variables via `SetClusterSpec()`, instead relying on the ConfigMap-mounted scripts for worker discovery. Notably, MPI is the only framework that has no validating webhook (the webhook setup uses a `scaffold` no-op function).

A significant optimization is the label-filtered informer cache (`newCacheOptions` in main.go). Without it, the controller-runtime cache would load all Pods, Services, ConfigMaps, Secrets, ServiceAccounts, Roles, and RoleBindings cluster-wide into memory, causing unbounded memory growth in large clusters. The operator filters these using the `training.kubeflow.org/operator-name` label that is already applied to all resources it creates. ConfigMaps and Secrets are additionally configured for direct API reads (bypassing cache) to avoid stale configuration data. The GOMEMLIMIT environment variable is set to 460MiB (within the 512Mi memory limit) to ensure the Go garbage collector manages memory aggressively.

The PyTorch controller is the most feature-rich, uniquely supporting elastic training through `ElasticPolicy` with HPA (HorizontalPodAutoscaler) integration, PyTorch rendezvous configuration (c10d, etcd backends), and NetworkPolicy creation for pod-to-pod isolation within a job. It also supports `NprocPerNode` configuration for multi-GPU nodes and init containers that wait for the master pod DNS to become resolvable before starting workers. The operator supports Kueue's MultiKueue integration through the `managedBy` field in `RunPolicy`—when set to `kueue.x-k8s.io/multikueue`, the training-operator skips reconciliation and defers to Kueue.

For RHOAI deployment, the kustomize overlay adds a `kubeflow-` name prefix to all resources, sets the namespace to `opendatahub`, configures a PodMonitor for Prometheus metrics collection, generates aggregate ClusterRoles (`training-edit` and `training-view`) that aggregate into the standard Kubernetes `admin`/`edit`/`view` roles, and patches the deployment to use the Konflux-built image from params.env as both the operator image and the PyTorch init container image. The monitoring-port (8080) is removed from the Service in the RHOAI overlay, leaving only the webhook port (443→9443) exposed.
