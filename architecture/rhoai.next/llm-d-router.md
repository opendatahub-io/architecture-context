## Purpose

**Short**: Intelligent inference request router that provides LLM load-aware, prefix-cache-aware routing, request prioritization, and disaggregated inference coordination for model-serving pools.

**Detailed**: The llm-d Router is the intelligent entry point for inference traffic in the llm-d platform. It consists of two main binaries: the Endpoint Picker (EPP) and the Disaggregation Sidecar (pd-sidecar). The EPP integrates with production-grade L7 proxies (Envoy, AgentGateway) via the Envoy ext_proc gRPC protocol, injecting real-time scheduling decisions into the data plane. It evaluates incoming requests against the current state of an InferencePool — considering KV-cache locality, endpoint load, prefix cache hits, LoRA adapter affinity, session stickiness, and priority — to make optimal placement decisions.

The EPP operates as a Kubernetes-native service using controller-runtime to watch InferencePool, Pod, InferenceObjective, and InferenceModelRewrite resources. It exposes a rich plugin framework with 60+ built-in plugins across scheduling (filters, scorers, pickers, profile handlers), flow control (fairness, ordering, saturation detection, eviction), request control (admission, data production, pre-admission), data layer (polling sources, extractors, file-based discovery), and request handling (parsers for OpenAI, Anthropic, VertexAI, vLLM HTTP/gRPC, passthrough formats).

The pd-sidecar runs alongside decode model servers and orchestrates multi-stage disaggregated inference lifecycles — Prefill/Decode (P/D) and Encode/Prefill/Decode (E/P/D) — by coordinating with prefill and encode workers to manage KV-cache and embedding transfers via pluggable connector protocols (nixlv2, shared-storage, sglang, mooncake for KV; ec-example, ec-nixl for embeddings).

_Note: The EPP Konflux Dockerfile uses CGO_ENABLED=1 but does not include -tags strictfipsruntime. The sidecar Dockerfile does include the tag. Both images use UBI 9 base which provides OpenSSL. The EPP binary will use Go's stdlib crypto linked against system OpenSSL via CGO, which is FIPS-compliant on RHEL/UBI, but the missing strictfipsruntime tag means the Go runtime does not enforce FIPS-only crypto at build time. The InsecureSkipVerify=true default on metrics scraping clients is a TLS verification gap (not a FIPS compliance issue per se, but a security concern)._

## Architectural Analysis

The llm-d-router represents a sophisticated scheduling-as-a-sidecar architecture where the routing intelligence is completely decoupled from the data plane. The EPP does not handle any inference traffic directly — it operates purely as an ext_proc advisory service to Envoy/AgentGateway, injecting routing decisions via header mutations (`llm-d-destination-endpoint`) and Envoy dynamic metadata. This separation enables the EPP to be upgraded, restarted, or scaled independently of the proxy, and allows the same EPP to work with different proxy implementations (Envoy, AgentGateway, or cloud-managed load balancers like Google Cloud ALB).

The plugin framework is the EPP's most distinctive architectural feature. With 60+ built-in plugins spanning 7 extension points (pre-admission, admission, data production, scheduling filters, scoring, picking, and response processing), the system achieves extreme configurability without code changes. Plugins declare typed data dependencies via Produces/Consumes interfaces, and the framework validates the dependency DAG at startup. This design enables complex routing strategies — for example, prefix-cache-aware routing requires the tokenizer data producer to run before the approximate-prefix-cache producer, which in turn feeds the prefix-cache-scorer. The framework handles this ordering automatically. The flyweight pattern for flow control plugins (single instance, per-band state) is notable for enabling horizontal scaling without shared state synchronization.

The disaggregation sidecar (pd-sidecar) implements a multi-stage inference orchestration pattern that is increasingly important for large model deployments. By separating prefill (compute-intensive, parallelizable) from decode (memory-bound, sequential), deployments can use heterogeneous hardware and scale each stage independently. The sidecar supports four KV-cache transfer protocols (nixlv2, shared-storage, sglang, mooncake) and two encoder connector protocols (ec-example, ec-nixl), reflecting the rapidly evolving landscape of disaggregated inference. The experimental MoRI-IO WRITE-mode support (feature-gated off) hints at future RDMA-based KV-cache transfer for low-latency disaggregation. The sidecar's SSRF protection mechanism — using the InferencePool as an allowlist to validate prefiller/encoder targets — is a pragmatic security boundary for multi-tenant environments.

A notable security gap is the default InsecureSkipVerify=true on metrics scraping HTTP clients. While pragmatic (model servers rarely have proper TLS certificates), this opens a TLS verification bypass that could be exploited in adversarial network environments. The EPP's own gRPC server defaults to TLS with self-signed certificates, which provides encryption but no authentication. The sidecar has more robust TLS configuration with explicit per-stage controls and modern cipher suites (TLS 1.2+ with ECDHE). The FIPS posture is inconsistent: the sidecar Dockerfile includes `-tags strictfipsruntime` for Go FIPS enforcement, but the EPP Dockerfile omits this tag despite using CGO_ENABLED=1 — this means the EPP binary relies on the UBI 9 base image's OpenSSL via CGO linking but does not enforce FIPS-only crypto paths at the Go level.

The deployment model offers three well-defined patterns: gateway mode (shared infrastructure via Gateway API), standalone mode (self-contained with embedded Envoy/AgentGateway), and disaggregated mode (sidecar alongside model servers). The Helm charts use a library chart pattern (routerlib) that cleanly separates shared templates from mode-specific application charts. The Envoy configuration in standalone mode is comprehensive — it includes HTTP/2 settings (64KB stream windows, 1MB connection windows), circuit breakers (40K limits across connections, requests, and pending), ext_proc health checking, and full-duplex streaming for request/response body interception. The 24-hour route timeout (86400s) and 1000-second ext_proc message timeout accommodate long-running inference requests with extended token generation.
