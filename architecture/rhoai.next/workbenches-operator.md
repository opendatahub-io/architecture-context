## Purpose

**Short**: Kubernetes operator that manages the lifecycle of the Workbenches component (notebook environments) for the Open Data Hub / Red Hat OpenShift AI platform.

**Detailed**: The workbenches-operator is a controller-runtime based Kubernetes operator that reconciles a cluster-scoped `Workbenches` custom resource (singleton named "default"). It manages the provisioning and health monitoring of workbench (Jupyter notebook) infrastructure within the platform. The operator's primary responsibilities are: ensuring the workbench namespace exists (with appropriate ownership labels), computing kustomize parameters for manifest rendering (section title, MLflow integration, gateway domain), and monitoring deployment readiness within the workbench namespace.

The operator follows a two-state management model: `Managed` (actively reconciling and monitoring) and `Removed` (component teardown). It exposes a rich set of status conditions (`Ready`, `ProvisioningSucceeded`, `Degraded`, `DeploymentsAvailable`) that the platform orchestrator uses to determine overall platform health. The spec includes orchestrator-projected fields (`gatewayDomain`, `platform`, `mlflowEnabled`) that the platform operator propagates from higher-level platform configuration.

This is a newly scaffolded operator (kubebuilder v4 layout) in early development. It does not yet apply kustomize manifests directly -- it computes parameters and checks deployment health, but the actual manifest application is expected to be integrated in future iterations. The operator is part of RHOAI's component operator decomposition, where each platform component gets its own dedicated operator managed by a central orchestrator.

## Architectural Analysis

The workbenches-operator is in early-stage development, freshly scaffolded using kubebuilder v4. The codebase is remarkably small (9 non-test Go files, ~550 lines of application code) and follows a clean separation of concerns: API types in `api/v1alpha1/`, a single controller in `internal/controller/`, and helper packages for platform constants (`internal/platform/`), metadata labels/annotations (`internal/metadata/`), and GVK references (`internal/gvk/`).

The operator currently has a gap between its intended design and its implementation. The `computeKustomizeParams()` method builds a parameter map (`section-title`, `mlflow-enabled`, `gateway-url`) that suggests future integration with a kustomize-based manifest application pipeline, but the operator does not yet apply any manifests. It only ensures the workbench namespace exists and monitors deployments that other processes create. This makes it a "status aggregator" today -- it reports on health rather than driving deployment. The TODO in `config/manager/kustomization.yaml` about NetworkPolicy (referencing CWE-284) suggests active security awareness but deferred implementation.

The `internal/gvk/` package defines GroupVersionKind constants for `Notebook` (kubeflow.org/v1), `ImageStream` (image.openshift.io/v1), and `HardwareProfile` (infrastructure.platform.opendatahub.io/v1). These are not currently used by the controller but indicate the operator's planned scope: managing the full workbench lifecycle including notebook instances, container images via ImageStreams, and GPU/accelerator hardware profiles. The annotation constants in `internal/metadata/annotations.go` (connection annotations, hardware profile annotations, accelerator annotations, last size selection) further confirm this intent.

The operator implements a robust status condition model with four condition types (`Ready`, `ProvisioningSucceeded`, `Degraded`, `DeploymentsAvailable`) and a phase field (`Ready`/`Not Ready`). The exponential backoff rate limiter (5s base, 5m max) prevents tight retry loops on persistent failures. The `workbenchNamespace` field is immutable (CEL validation rule `self == oldSelf`), and the CR name is locked to "default" via CEL validation -- both are solid patterns for singleton cluster-scoped resources. The operator supports two platforms (`OpenDataHub` with default namespace `opendatahub`, `SelfManagedRhoai` with default namespace `rhods-notebooks`), with the platform type projected by the orchestrator.

No Konflux Dockerfiles exist yet in this repository. The regular Dockerfile includes FIPS-compliant build flags (`CGO_ENABLED=1`, `-tags strictfipsruntime`) and runs as non-root (UID 65532), demonstrating security-first defaults. The container security context in the deployment manifest enforces `readOnlyRootFilesystem`, drops all capabilities, disallows privilege escalation, and uses `RuntimeDefault` seccomp profile. The metrics endpoint is secured via HTTPS with Kubernetes SubjectAccessReview authentication/authorization. HTTP/2 is disabled by default to mitigate HTTP/2 rapid-reset attacks (CVE-2023-44487 mitigation pattern). The `contracts/schemas/workbenches.v1alpha1.json` file provides a JSON Schema for the CRD, enabling contract-based validation between the orchestrator and this component operator.
