## Purpose

**Short**: Python SDK providing an intuitive interface for requesting, managing, and interacting with Ray clusters and batch workloads on Kubernetes/OpenShift.

**Detailed**: The CodeFlare SDK is a Python client library that abstracts the complexity of creating and managing distributed compute resources (Ray clusters) on Kubernetes and OpenShift platforms. It provides a programmatic interface for data scientists and ML engineers to request GPU/CPU resources, submit Ray jobs, monitor cluster status, and manage workload queuing via Kueue integration.

The SDK operates as a client-side library installed in user notebooks (Jupyter/RHOAI workbenches) rather than a deployed server-side service. It communicates with the Kubernetes API server to create and manage RayCluster and AppWrapper custom resources, interacts with the Ray Dashboard for job submission, and queries Kueue LocalQueues for workload scheduling. The SDK also provides Jupyter widget-based UI components for interactive cluster management in notebook environments.

The library integrates with the broader RHOAI/ODH platform by leveraging KubeRay for Ray cluster orchestration, CodeFlare AppWrappers (v1beta2) for workload wrapping, and Kueue for fair-share scheduling. It supports both OpenShift (Route-based) and vanilla Kubernetes (Ingress-based) cluster topologies for Ray Dashboard access.

## Architectural Analysis

The CodeFlare SDK follows a client-library architecture pattern where all logic runs in the user's Python environment (typically a Jupyter notebook on RHOAI/ODH). It is not a deployed service — it has no persistent process, no server-side state, and no dedicated infrastructure. This design keeps the SDK lightweight but means all Kubernetes API interactions happen with the user's credentials and network context.

The SDK's core abstraction is the `Cluster` class (`src/codeflare_sdk/ray/cluster/cluster.py`), which wraps a `ClusterConfiguration` dataclass and translates it into a RayCluster YAML manifest. The YAML generation pipeline reads a base template (`base-template.yaml`), applies user-specified resource configurations (CPU, memory, GPU, extended resources), and optionally wraps the RayCluster in a CodeFlare AppWrapper for Kueue-based scheduling. This template-based approach provides a simple extension point but limits flexibility compared to a programmatic builder pattern.

A notable architectural decision is the dual deployment path: the SDK can create either a bare RayCluster CR (when `appwrapper=False`) or an AppWrapper-wrapped RayCluster (when `appwrapper=True`). The AppWrapper path integrates with Kueue for fair-share scheduling, which is the recommended path in RHOAI. The Kueue integration (`src/codeflare_sdk/common/kueue/kueue.py`) discovers the default LocalQueue via the `kueue.x-k8s.io/default-queue` annotation and applies the `kueue.x-k8s.io/queue-name` label to workloads.

The SDK handles OpenShift vs. vanilla Kubernetes detection dynamically at runtime by checking for the `route.openshift.io/v1` API group (`generate_yaml.py:is_openshift_cluster()`). This affects dashboard URL discovery (OpenShift Routes vs. Kubernetes Ingresses) and authentication headers (Bearer token injection for OpenShift). The TLS certificate generation module (`generate_cert.py`) uses the `cryptography` library to create CA-signed client certificates for Ray mTLS, reading the CA secret provisioned by the KubeRay operator. The `verify_tls=False` option is a security risk that should be used only in development — it disables all TLS verification.

The widget system (`common/widgets/widgets.py`) is tightly coupled to the Jupyter notebook environment, detecting the notebook context via `PYDEVD_IPYTHON_COMPATIBLE_DEBUGGING` and `JPY_SESSION_NAME` environment variables. This provides a useful interactive experience but means the cluster management UI is unavailable in plain Python scripts. The widget code uses a class-based pattern (`RayClusterManagerWidgets`) that manages state across toggle buttons, delete operations, and dashboard navigation.
