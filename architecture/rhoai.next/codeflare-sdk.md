## Purpose

**Short**: Python SDK for managing Ray clusters and jobs on Kubernetes, providing high-level abstractions over KubeRay CRDs with Kueue queue integration.

**Detailed**: The CodeFlare SDK is a Python client library that simplifies the lifecycle management of distributed computing resources on Kubernetes. It provides intuitive interfaces for creating and managing Ray clusters via the KubeRay operator, submitting and monitoring Ray jobs, and integrating with the Kueue job queuing system for resource quota management. The SDK is designed to be used from Jupyter notebooks and Python scripts within the Open Data Hub (ODH) and Red Hat OpenShift AI (RHOAI) platforms.

The SDK abstracts the complexity of Kubernetes Custom Resources by exposing Python dataclasses (`ClusterConfiguration`, `ManagedClusterConfig`) that generate `RayCluster` and `RayJob` CRs. It handles TLS certificate generation for mTLS connections to Ray clusters, supports multiple authentication methods via the `kube-authkit` library (OIDC, OpenShift OAuth, token-based, kubeconfig), and provides automatic dashboard URL resolution across multiple ingress mechanisms (Gateway API HTTPRoute, OpenShift Route, Kubernetes Ingress). The SDK also includes Jupyter notebook widgets for interactive cluster management and a vendored KubeRay Python client for direct CRD operations.

As a client-side library, the CodeFlare SDK does not deploy any server-side components itself. It interacts with existing platform infrastructure (KubeRay operator, Kueue controller, Gateway API, OpenShift Routes) to manage Ray workloads. The SDK is distributed as a PyPI package (`codeflare-sdk`) and is pre-installed in RHOAI notebook images.

## Architectural Analysis

The CodeFlare SDK follows a layered architecture pattern with clear separation between user-facing APIs, Kubernetes resource construction, and platform integration. The `ray.cluster` module provides the `Cluster` class as the primary abstraction for RayCluster lifecycle management, while `ray.rayjobs` provides the `RayJob` class for job submission. Both modules delegate Kubernetes API calls to the vendored KubeRay Python client and the `common.kubernetes_cluster` authentication layer. Configuration is handled through Python dataclasses (`ClusterConfiguration`, `ManagedClusterConfig`) that validate inputs and generate Kubernetes resource specifications.

A notable design decision is the multi-path dashboard URL resolution. The SDK searches for dashboard access in order: Gateway API HTTPRoute (RHOAI 3.x pattern), OpenShift Routes (RHOAI 2.x / legacy), and Kubernetes Ingress (KinD / non-OpenShift). For HTTPRoute discovery, the SDK searches across five namespaces (cluster namespace, `redhat-ods-applications`, `opendatahub`, `default`, `ray-system`) using label selectors (`ray.io/cluster-name`, `ray.io/cluster-namespace`), then resolves the parent Gateway to construct the full URL. This approach allows the SDK to work across multiple platform versions and deployment topologies without requiring configuration changes.

The TLS certificate management subsystem in `generate_cert.py` implements a full client certificate lifecycle. It polls for a CA Secret created by the KubeRay operator (labeled `ray.openshift.ai/cluster-name`), generates RSA 3072-bit client certificates with SAN entries for the cluster's head service DNS names, stores them in the XDG-compliant directory `~/.local/share/codeflare/tls/` with strict file permissions (0600), and sets `RAY_USE_TLS=1` and related environment variables for Ray mTLS. The certificate management includes expiry tracking, cleanup of expired certificates, and force-regeneration for CA rotation scenarios.

The Kueue integration is woven throughout both the cluster and job modules. When a `local_queue` is specified (or auto-detected via the `kueue.x-k8s.io/default-queue` annotation), the SDK applies the `kueue.x-k8s.io/queue-name` label to the resource, which signals the Kueue mutating webhook to manage admission. The SDK tracks Workload admission status for RayClusters and validates WorkloadPriorityClass existence for RayJobs. The Kueue integration supports the `suspend/unsuspend` workflow where Kueue sets `suspend: true` on the CR until quota is available.

The SDK includes a vendored copy of the KubeRay Python client (`src/codeflare_sdk/vendored/python_client/`) which provides `RayClusterApi` and `RayjobApi` classes for direct CRD operations. This vendoring ensures version compatibility and allows the SDK to control the KubeRay client behavior without depending on an external package. The vendored client uses `kubernetes.client.CustomObjectsApi` for all CRD operations with the `ray.io` API group.
