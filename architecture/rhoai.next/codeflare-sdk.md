## Purpose

**Short**: A Python SDK that provides an intuitive interface for creating, managing, and monitoring Ray clusters on Kubernetes, with support for AppWrapper-based resource scheduling via Kueue.

**Detailed**: The CodeFlare SDK is a client-side Python library (not a deployed service) that enables data scientists and ML engineers to programmatically request, create, monitor, and tear down Ray clusters on Red Hat OpenShift AI (RHOAI) and Open Data Hub (ODH) platforms. It abstracts the complexity of Kubernetes Custom Resource management by providing a high-level `Cluster` object that generates RayCluster or AppWrapper YAML specifications from a `ClusterConfiguration` dataclass, then applies them to the cluster via the Kubernetes API.

The SDK integrates deeply with the RHOAI/ODH ecosystem: it generates RayCluster CRs (ray.io/v1), optionally wraps them in AppWrapper CRs (workload.codeflare.dev/v1beta2) for multi-resource scheduling, labels them for Kueue local queue integration (kueue.x-k8s.io/v1beta1), discovers OpenShift Routes and Kubernetes Ingresses for dashboard access, supports TLS certificate generation for Ray client connections, and provides Jupyter Notebook widgets (ipywidgets) for interactive cluster management. It also wraps the Ray Job Submission Client for job lifecycle management against running Ray clusters.

The SDK is distributed as a pip-installable Python package (`codeflare-sdk`) built with Poetry and does not produce a container image that runs as a service. Instead, it runs within user workspaces (Jupyter Notebooks, workbenches) as a library dependency.

## Architectural Analysis

The CodeFlare SDK follows a **client-library pattern** rather than a microservice architecture — it does not deploy as a pod or expose any endpoints. Instead, it runs within user workspaces (typically RHOAI/ODH Jupyter Notebook workbenches) and interacts with the Kubernetes API server to manage Ray cluster lifecycle. This design means the SDK inherits the permissions and network context of whatever environment it runs in, which has important security implications: the SDK can only do what the user's service account or token allows.

The **YAML generation pipeline** is the core of the SDK's architecture. It reads a base RayCluster template (`base-template.yaml`), applies user configuration from `ClusterConfiguration`, injects resource requests/limits, GPU/accelerator mappings, environment variables, image references, and Kueue labels, then optionally wraps the result in an AppWrapper CR. The template includes ODH-specific volume mounts for trusted CA bundles (`odh-trusted-ca-bundle`, `odh-ca-bundle.crt`), which are mounted at standard certificate paths in the generated pods. The SDK supports a configurable `extended_resource_mapping` that maps Kubernetes device plugin resources (e.g., `nvidia.com/gpu`, `habana.ai/gaudi`, `aws.amazon.com/neuroncore`) to Ray resource names, enabling multi-accelerator support.

The **authentication model** has three tiers: (1) Kubernetes cluster authentication via token or kubeconfig, (2) Ray dashboard access via Bearer token forwarded from the K8s auth context (on OpenShift), and (3) optional mTLS for Ray client connections. The SDK includes a certificate generation module (`generate_cert.py`) that reads CA secrets from the cluster, generates a local CA-signed client certificate, and exports TLS environment variables (`RAY_USE_TLS`, `RAY_TLS_SERVER_CERT`, etc.) for secure Ray client connections. The CA cert path discovery has a three-tier fallback: explicit parameter → `CF_SDK_CA_CERT_PATH` env var → RHOAI workbench default path (`/etc/pki/tls/custom-certs/ca-bundle.crt`).

The **Kueue integration** is notable: the SDK discovers LocalQueues in the target namespace, resolves the default queue (via `kueue.x-k8s.io/default-queue` annotation), validates that specified queues exist, and injects `kueue.x-k8s.io/queue-name` labels into generated CRs. This tight coupling with Kueue means the SDK participates in the platform's resource quota and fairness system, but it also means the SDK must have read access to LocalQueue CRDs in the target namespace.

The **Jupyter widget system** (`common/widgets/`) provides an interactive cluster management UI when running in notebook environments. It detects notebook contexts via `PYDEVD_IPYTHON_COMPATIBLE_DEBUGGING` and `JPY_SESSION_NAME` environment variables, and renders ipywidgets-based panels showing cluster status, worker specs, and action buttons (create, delete, view jobs, open dashboard). The widget layer uses pandas DataFrames to structure cluster data for HTML table rendering.
