## Purpose

**Short**: Python SDK that provides an intuitive interface for creating, managing, and submitting jobs to Ray clusters on Kubernetes/OpenShift, with Kueue and AppWrapper integration.

**Detailed**: The CodeFlare SDK is a client-side Python library that simplifies the lifecycle management of Ray clusters running on Kubernetes and OpenShift. It abstracts the complexity of generating RayCluster and AppWrapper custom resource YAML, authenticating to Kubernetes clusters, and interacting with Ray's job submission API.

The SDK is designed to be used from within Jupyter notebooks running on RHOAI/ODH workbenches. It provides a `Cluster` object that users instantiate with a `ClusterConfiguration` specifying resource requirements (CPU, memory, GPUs, extended accelerator resources). The SDK generates the appropriate Kubernetes custom resources (RayCluster or AppWrapper), submits them to the cluster, monitors their status, and exposes the Ray dashboard and job submission endpoints. It integrates with Kueue for workload queuing by discovering and attaching LocalQueue labels.

The SDK also provides Jupyter notebook widgets (via ipywidgets) for interactive cluster management — including cluster up/down buttons, cluster selection toggle buttons, and status display panels. A certificate generation utility enables mTLS communication between local Ray clients and remote Ray clusters. The library is published to PyPI as `codeflare-sdk` and is consumed by data scientists and ML engineers working on the RHOAI/ODH platform.

## Architectural Analysis

The CodeFlare SDK is a pure client-side Python library with no server component, no container image shipped via Konflux, and no Kubernetes deployment manifests. It occupies a unique position in the RHOAI/ODH stack as the primary user-facing interface for distributed computing workloads — data scientists interact with the SDK from Jupyter notebooks to provision and manage Ray clusters.

The SDK's architecture follows a layered design with clear separation of concerns. The `ray.cluster` module handles the core lifecycle (create/monitor/delete), `ray.cluster.generate_yaml` handles YAML rendering from templates, `ray.client` wraps Ray's job submission API, `common.kubernetes_cluster.auth` manages authentication, and `common.kueue` handles workload queue integration. This separation allows each layer to be tested independently, and the codebase includes co-located unit tests in each module.

A notable design pattern is the template-based YAML generation approach. Rather than constructing RayCluster manifests programmatically with Kubernetes client models, the SDK reads a YAML template file (`base-template.yaml`), patches it with user-specified resource values, and optionally wraps it in an AppWrapper. This makes the generated resources easy to inspect and debug but introduces coupling to the template's structure. The default Ray images are digest-pinned references to `quay.io/modh/ray` images, indexed by Python version (3.9 and 3.11), ensuring reproducibility.

The authentication module supports three modes: token-based authentication (common in CI/CD and programmatic access), kubeconfig file authentication (common in local development), and in-cluster service account authentication (used when running inside a pod on the cluster, such as in RHOAI workbenches). The `CF_SDK_CA_CERT_PATH` environment variable and the workbench default path `/etc/pki/tls/custom-certs/ca-bundle.crt` enable the SDK to work with custom CA bundles deployed by the ODH platform. The `skip_tls` option exists for development convenience but disables TLS verification entirely — this should never be used in production.

The Kueue integration is carefully designed to be backwards-compatible: the SDK checks whether LocalQueue CRDs exist before attempting to use them (gracefully handling 404/403 errors), discovers the default queue via annotations, and validates queue existence before labeling resources. This makes the SDK functional both with and without Kueue installed on the cluster.

The Jupyter widget layer (`common.widgets`) provides an interactive UI within notebooks that includes cluster selection, status display with HTML-formatted tables, and buttons for cluster operations (up/down, view jobs, open dashboard). The `is_notebook()` detection checks for environment variables specific to VSCode Jupyter and RHOAI/ODH workbench environments. This integration makes the SDK immediately accessible to data scientists who prefer visual interactions over code-only workflows.
