## Purpose

**Short**: Python SDK that provides an intuitive interface for requesting, managing, and monitoring Ray clusters and batch workloads on Kubernetes/OpenShift.

**Detailed**: The CodeFlare SDK is a client-side Python library that simplifies the lifecycle management of Ray clusters running on Kubernetes and OpenShift. It abstracts the complexity of creating and managing Kubernetes custom resources (RayCluster, AppWrapper) by providing a high-level Python API. Users configure cluster resources (CPU, memory, GPUs, extended accelerator resources) through a `ClusterConfiguration` dataclass, and the SDK generates the appropriate YAML manifests, submits them to the Kubernetes API, and monitors their status.

The SDK integrates with the Kueue batch scheduling system for workload queuing via LocalQueues, and with the CodeFlare AppWrapper (workload.codeflare.dev) for multi-resource workload wrapping. It also provides a wrapper around the Ray Job Submission Client for submitting and managing Ray jobs on running clusters. When used in Jupyter Notebook environments (such as RHOAI workbenches), the SDK provides interactive ipywidgets-based UI components for cluster management, including cluster creation/deletion buttons, status dashboards, and Ray dashboard links.

The SDK supports both OpenShift-native features (Route-based dashboard access, OAuth-aware ingress) and vanilla Kubernetes (Ingress-based dashboard access). It includes utilities for TLS certificate generation for secure Ray cluster communication and handles custom CA certificate bundles used in ODH/RHOAI environments.

## Architectural Analysis

The CodeFlare SDK follows a layered architecture centered around the `Cluster` class as the primary user-facing abstraction. The design separates concerns cleanly: `ClusterConfiguration` (dataclass) handles resource specification and validation, `generate_yaml` handles YAML template rendering, `auth` handles Kubernetes authentication, and the `Cluster` class orchestrates these into a coherent lifecycle management API. This separation allows the SDK to support multiple deployment patterns (AppWrapper-wrapped vs bare RayCluster, OpenShift vs vanilla Kubernetes) without exposing complexity to users.

The SDK uses a template-based approach for manifest generation rather than constructing YAML programmatically. The base template (`src/codeflare_sdk/ray/templates/base-template.yaml`) is a full RayCluster YAML that gets mutated in-place by `generate_yaml.py`. This approach is pragmatic but fragile — changes to the RayCluster API schema or KubeRay conventions require manual template updates. The template already includes ODH/RHOAI-specific volume mounts for trusted CA bundles (`odh-trusted-ca-bundle`, `odh-ca-cert`), tightly coupling the template to the platform. The hard-coded Ray image digests in `SUPPORTED_PYTHON_VERSIONS` (pointing to `quay.io/modh/ray`) further tie the SDK to the RHOAI ecosystem.

The authentication layer has a notable design pattern: it uses Python global variables (`api_client`, `config_path`) as module-level singletons to track authentication state. This works for single-user notebook contexts but would be problematic in multi-tenant or concurrent usage scenarios. The `config_check()` function implements a priority chain: explicit API client > explicit config path > ~/.kube/config > in-cluster config, with the workbench CA cert path (`/etc/pki/tls/custom-certs/ca-bundle.crt`) as an additional fallback for TLS verification.

The Kueue integration is implemented as label-based queue assignment (`kueue.x-k8s.io/queue-name` label) rather than direct API interaction with the Kueue admission webhook. The SDK validates that the specified LocalQueue exists in the target namespace before labeling resources, providing early error detection. The AppWrapper support uses the `workload.codeflare.dev/v1beta2` API, wrapping the RayCluster YAML as a component within the AppWrapper spec — this is the CodeFlare v2 AppWrapper pattern used in RHOAI for multi-resource workload management.

The `cryptography` library usage in `generate_cert.py` is security-sensitive: it generates RSA-2048 keys and self-signed X.509 certificates for Ray cluster TLS. The certificate handling uses SHA256 for signing (FIPS-approved) and generates certificates with limited validity (default 30 days). The `export_env` function sets `RAY_USE_TLS=1` and associated environment variables to enable Ray's built-in TLS support. On RHEL/UBI, the `cryptography` library uses the system OpenSSL, which is FIPS-validated when FIPS mode is enabled at the OS level.
