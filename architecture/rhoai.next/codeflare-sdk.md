## Purpose

**Short**: Python SDK providing an intuitive interface for requesting, managing, and monitoring Ray clusters and batch computing resources on Kubernetes/OpenShift.

**Detailed**: CodeFlare SDK is a Python client library that abstracts the complexity of managing distributed computing resources on Kubernetes and OpenShift clusters. It provides high-level Python objects (`Cluster`, `ClusterConfiguration`, `RayJobClient`, `AWManager`) that enable data scientists and ML engineers to request Ray clusters, submit jobs, and monitor cluster status without directly interacting with Kubernetes APIs.

The SDK generates RayCluster custom resource YAML from user-specified configurations (CPU, memory, GPU, worker count), optionally wrapping them in AppWrapper CRs for Kueue-based job queuing. It supports both OpenShift (using Routes for ingress) and vanilla Kubernetes (using Ingress resources) platforms. The SDK includes Jupyter notebook widgets (via ipywidgets) for interactive cluster management in notebook environments, TLS certificate generation for secure Ray client connections, and integration with Kueue local queues for resource quota management.

CodeFlare SDK is not deployed as a service — it runs client-side within user notebooks or Python environments (such as RHOAI workbenches) and communicates with the Kubernetes API server and Ray cluster head nodes. It is distributed via PyPI as the `codeflare-sdk` package.

## Architectural Analysis

CodeFlare SDK follows a classic client-library pattern: it provides Pythonic abstractions over Kubernetes custom resource operations, hiding the YAML generation and API mechanics from end users. The core design centers on the `Cluster` object, which couples a `ClusterConfiguration` dataclass (resource specifications) with a YAML generation pipeline. The `generate_appwrapper()` function reads a base RayCluster template, patches it with user-specified resources (CPU, memory, GPU, workers), optionally wraps it in an AppWrapper for Kueue integration, and either writes it to disk or holds it in memory for direct API submission.

A notable design decision is the dual-path platform support. The SDK detects OpenShift clusters at runtime by querying for the `route.openshift.io/v1` API group (`is_openshift_cluster()`). On OpenShift, it reads Route resources for dashboard URL discovery and passes Bearer tokens in headers for authenticated dashboard access. On vanilla Kubernetes, it falls back to Ingress resources. This branching appears in multiple places — `cluster_dashboard_uri()`, `_map_to_ray_cluster()`, `_get_ingress_domain()` — creating some duplication that could benefit from a platform abstraction layer.

The TLS/certificate management in `generate_cert.py` is architecturally significant. The SDK generates a self-signed CA and uses it to create client certificates for mTLS connections to Ray clusters. The CA secret is stored as a Kubernetes Secret (labeled with `ray.openshift.ai/cluster-name`), and the SDK retrieves it to generate local TLS certificates. This enables secure Ray client connections (port 10001) but relies on the user calling `generate_tls_cert()` and `export_env()` manually — there is no automatic TLS setup.

The SDK has several deprecating configuration fields (`head_gpus`, `num_gpus`, `head_cpus`, `min_cpus`, `max_cpus`, `head_memory`, `min_memory`, `max_memory`) that are mapped to their replacements in `ClusterConfiguration.__post_init__()`. The migration to `head_extended_resource_requests` and `worker_extended_resource_requests` provides better support for heterogeneous accelerators (NVIDIA GPU, Intel GPU, AMD GPU, Habana Gaudi, AWS Neuron, Google TPU) through a configurable resource mapping (`DEFAULT_RESOURCE_MAPPING`).

The Jupyter widget integration (`widgets.py`) is well-designed for the RHOAI workbench use case. It provides interactive toggle buttons for cluster selection, delete/view-jobs/dashboard buttons, and auto-refreshes cluster data on selection change. The `is_notebook()` detection checks for `PYDEVD_IPYTHON_COMPATIBLE_DEBUGGING` (VS Code) and `JPY_SESSION_NAME` (JupyterLab) environment variables, which covers the primary notebook environments but could miss other interactive Python environments.

The default Ray images (`SUPPORTED_PYTHON_VERSIONS` in `generate_yaml.py`) are pinned to SHA256 digests on `quay.io/modh/ray`, supporting Python 3.9 and 3.11. The SDK automatically selects the image matching the user's Python version, which is a nice ergonomic touch but means the image pins must be manually updated when new Ray versions are released.
