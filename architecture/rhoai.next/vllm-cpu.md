## Purpose

**Short**: CPU-optimized build of vLLM providing high-throughput, OpenAI-compatible LLM inference serving without GPU hardware requirements.

**Detailed**: vLLM-CPU is a Red Hat downstream fork of the vLLM project, specifically configured and optimized for CPU-only inference workloads on ppc64le and s390x architectures. It provides an OpenAI-compatible RESTful API server (FastAPI/Uvicorn) that supports chat completions, text completions, embeddings, and the newer OpenAI Responses API, along with Anthropic Messages API compatibility.

The component is built using Konflux CI/CD pipelines and packaged as a UBI9-based container image (`odh-vllm-cpu-rhel9`). It leverages architecture-specific optimizations including OpenBLAS, numactl, and Intel OpenMP (for x86_64), with platform-specific build scripts for ppc64le and s390x. The server supports both HTTP (via FastAPI/Uvicorn) and gRPC entrypoints, with optional TLS termination, API key authentication, CORS, and Prometheus metrics instrumentation.

vLLM-CPU fits into the RHOAI platform as a model serving runtime deployed by KServe's `ServingRuntime` CRD, enabling inference on models from Hugging Face Hub without requiring GPU resources. It supports PagedAttention for efficient memory management, continuous batching, speculative decoding, LoRA adapters, multimodal models, and structured output generation (JSON schema, grammar constraints).

## Architectural Analysis

The vLLM-CPU repository is a Red Hat downstream fork that maintains the full vLLM codebase while providing CPU-specific build infrastructure for ppc64le and s390x architectures. The key architectural distinction from upstream vLLM is the removal of GPU dependencies and the use of CPU-optimized linear algebra libraries (OpenBLAS, numactl) compiled from source during the container build process. The Konflux Dockerfile (`Dockerfile.konflux.cpu`) uses a multi-stage build with UBI9-minimal as the base, compiling architecture-specific dependencies via `build_vllm_ppc64le.sh` and `build_vllm_s390x.sh` scripts.

The server architecture follows a FastAPI-based design with modular API routers. The main entrypoint (`vllm.entrypoints.openai.api_server`) constructs a FastAPI app that conditionally registers route handlers based on the model's supported tasks (generate, embed, classify, score, transcription, realtime). The router registration pattern is clean and extensible — each API domain (chat completion, completion, responses, models, Anthropic, pooling, etc.) has its own `api_router.py` with an `attach_router(app)` function. The generate task group registers four distinct API flavors: OpenAI Chat, OpenAI Completion, OpenAI Responses, and Anthropic Messages.

The V1 engine architecture uses `AsyncLLM` as the primary engine class, which runs inference asynchronously with support for multiprocessing (controlled by `VLLM_ENABLE_V1_MULTIPROCESSING` and `VLLM_WORKER_MULTIPROC_METHOD`). For CPU inference, the default worker multiprocessing method is set to `fork` in the Dockerfile (`VLLM_WORKER_MULTIPROC_METHOD=fork`). The environment variable `VLLM_USE_V1=1` is explicitly set in the Dockerfile, indicating this deployment always uses the V1 engine path.

Security is implemented at the application layer with API key authentication via middleware (`AuthenticationMiddleware`), optional TLS with certificate hot-reloading (`SSLCertRefresher` using `watchfiles`), and CORS configuration. The container runs as a non-root user (UID 2000, GID 0) for OpenShift compatibility. Notably, the gRPC server currently only supports insecure ports — TLS for gRPC is not yet implemented. The `VLLM_NO_USAGE_STATS=1` environment variable is set in the Dockerfile, ensuring no telemetry data is sent to the upstream vLLM stats server in production deployments.

The build system for CPU targets is complex, using architecture-specific shell scripts (`build_vllm_ppc64le.sh`, `build_vllm_s390x.sh`) that compile OpenBLAS, numactl, LAPACK, and potentially LLVM from source. The final image uses `uv` as the Python package installer with cache-mounted build stages to optimize rebuild times. The Tekton pipeline for Konflux builds (`vllm-cpu-pull-request.yaml`) targets `linux/s390x` and `linux/ppc64le` architectures with an 8-hour pipeline timeout, reflecting the lengthy cross-compilation process.
