## Purpose

**Short**: CPU-optimized build of vLLM, a high-throughput inference and serving engine for large language models, providing an OpenAI-compatible API server.

**Detailed**: vLLM CPU is the Red Hat downstream fork of the vLLM project, specifically configured and optimized for CPU-only inference on x86_64, ppc64le (IBM POWER), and s390x (IBM Z) architectures. It provides an OpenAI-compatible HTTP/HTTPS API server and a gRPC server for serving large language models without GPU hardware. The component supports features such as continuous batching, PagedAttention for efficient KV cache management, prefix caching, speculative decoding, LoRA adapters, and multimodal inputs (vision, audio, speech-to-text).

As part of the RHOAI platform, vLLM CPU is deployed as a model-serving runtime container image (odh-vllm-cpu-rhel9) that can be selected by KServe InferenceService or ModelMesh for serving models on CPU-only worker nodes. It exposes OpenAI-compatible chat completion, text completion, embedding, and scoring APIs, plus an Anthropic-compatible messages API, SageMaker-compatible invocation endpoints, and a gRPC inference interface. The component is built for multiple architectures through Konflux pipelines using architecture-specific build scripts.

## Architectural Analysis

vLLM CPU is a large, complex Python application (~1700 non-test Python files) forked from the upstream vllm-project/vllm repository with Red Hat-specific patches for multi-architecture support and RHOAI integration. The codebase is dominated by the inference engine, which uses PyTorch for CPU-based tensor computation with custom C++ attention kernels for performance. The architecture follows a clear separation between the API layer (FastAPI for HTTP, grpcio for gRPC), the async engine core (scheduler, batch assembly), and the model executor (weight loading, forward pass execution).

The most architecturally significant aspect of the CPU variant is its multi-architecture build strategy. The Konflux Dockerfile (`Dockerfile.konflux.cpu`) uses a runtime architecture detection pattern (`uname -m`) to dispatch to platform-specific build scripts (`build_vllm_ppc64le.sh`, `build_vllm_s390x.sh`). Each script builds a substantial number of dependencies from source — including PyTorch itself on ppc64le and s390x — because pre-built wheels are not available for these architectures on PyPI. This results in very long build times (8-hour pipeline timeout) and non-hermetic builds that depend on external package indices at build time.

The gRPC server implementation (`vllm/entrypoints/grpc_server.py`) binds to an insecure port (50051) with no TLS support. In RHOAI deployments, network-level security is expected to be provided by the platform's service mesh or network policies. The HTTP server supports full TLS with hot-reloading of certificates via the `SSLCertRefresher` class, which watches certificate files for changes and reloads them without server restart — this is well-suited for cert-manager-managed certificates in Kubernetes.

The component does not yet use AIPCC base images, which is a notable gap. The current build starts from bare `ubi9/ubi-minimal` and manually installs Python, creates a virtual environment, and configures package indices. Migration to the AIPCC CPU base image would standardize the Python environment, provide pre-configured access to the RHEL AI Python Package Index, and align with Red Hat's secure build requirements. The `AIPCC-13600` Jira reference in `requirements/common.txt` (disabling OpenTelemetry on s390x due to segfaults) shows the team is already interacting with the AIPCC ecosystem.

The V1 engine architecture represents a significant modernization with async request processing, ZeroMQ-based IPC between the API frontend and engine backend, and support for advanced features like data parallelism, tensor parallelism, pipeline parallelism, expert parallelism (MoE), and speculative decoding. While these distributed features are primarily designed for GPU deployments, the CPU variant inherits the full configuration surface, making the `--help` output extensive with many GPU-specific options that are not applicable.
