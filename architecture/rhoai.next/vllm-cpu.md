## Purpose

**Short**: CPU-only build of the vLLM high-throughput LLM inference and serving engine, optimized for running model inference without GPU hardware on ppc64le and s390x architectures.

**Detailed**: vLLM CPU is a downstream fork of the upstream vLLM project, maintained by Red Hat Data Services specifically for CPU-based inference serving within RHOAI. The component provides an OpenAI-compatible HTTP API server (and optional gRPC server) for serving large language models on CPU hardware, supporting multiple architectures including IBM Power (ppc64le) and IBM System z (s390x).

The component implements a modern V1 engine architecture with an async-first design, supporting continuous batching, PagedAttention-style KV cache management, tensor/data/pipeline parallelism via Gloo backend, and multi-modal model capabilities. It serves as the inference runtime backing KServe InferenceService deployments when GPU hardware is unavailable or when workloads target IBM hardware platforms.

The Konflux-based build pipeline produces multi-architecture container images (ppc64le and s390x) using architecture-specific build scripts that handle dependencies like Apache Arrow (built from source on s390x) and IBM's DevPI wheel index for ppc64le-specific PyTorch wheels.

## Architectural Analysis

The vLLM CPU component represents a significant architectural investment in making high-throughput LLM inference available on non-GPU hardware, particularly IBM Power (ppc64le) and IBM System z (s390x) architectures. The codebase is a downstream fork of the upstream vLLM project, carrying platform-specific patches and build infrastructure while tracking upstream releases (currently at v0.18.0 equivalent).

**V1 Engine Architecture**: The codebase has completed a full migration from the V0 to V1 engine architecture. The V0 engine files (`vllm/engine/llm_engine.py`, `vllm/engine/async_llm_engine.py`) are now thin aliases pointing to V1 implementations. The V1 architecture follows a modular pipeline design: `InputProcessor → EngineCore (Scheduler + Executor) → OutputProcessor`. The `EngineCore` (82K+ bytes) is the central loop managing KV cache allocation, request scheduling, and batch formation. The `EngineCoreClient` abstraction (68K+ bytes) supports three deployment modes: in-process (`InprocClient`), synchronous multiprocess (`SyncMPClient`), and asynchronous multiprocess (`AsyncMPClient`), enabling flexible deployment from single-container to distributed configurations.

**CPU-Specific Adaptations**: The CPU model runner (`vllm/v1/worker/cpu_model_runner.py`) wraps the GPU model runner with CUDA API placeholders, swapping GPU tensor operations for CPU equivalents. The CPU platform module (`vllm/platforms/cpu.py`, 19K bytes) performs architecture detection using `lscpu` JSON output and enables architecture-specific optimizations: NUMA-aware memory binding, OpenMP thread pinning (`VLLM_CPU_OMP_THREADS_BIND`), and AMD Zen-specific weight prepacking via `zentorch`. A notable workaround is the `LD_PRELOAD=/usr/lib64/libtcmalloc.so.4` (Konflux) and `LD_PRELOAD=/usr/lib64/libomp.so` (UBI) settings that prevent soft hangs with >16 OpenMP threads (upstream issue #32651).

**Multi-Architecture Build Complexity**: The Konflux Dockerfile uses a dynamic build strategy (`sh ./build_vllm_$(uname -m).sh`) that selects architecture-specific build scripts at build time. The ppc64le script (`build_vllm_ppc64le.sh`, 12K) installs dependencies from IBM's DevPI wheel index and EPEL, while the s390x script (`build_vllm_s390x.sh`, 8K) builds Apache Arrow from source since no pre-built s390x packages exist. This adds significant build time (4+ hour pipeline timeout) and introduces non-hermetic dependency resolution — the Tekton pipeline explicitly sets `hermetic: false`. The build platforms are currently limited to `linux/s390x` and `linux/ppc64le` in the Konflux pipeline, with x86_64 CPU builds handled separately.

**FIPS and Crypto Considerations**: As a Python service on UBI 9, vLLM inherits OpenSSL from the base image and the Python `ssl` module delegates to system crypto. However, application-level crypto has gaps: `hashlib.md5` is the default config hasher (with SHA256 fallback in FIPS environments, via `safe_hash()` in `vllm/utils/hashing.py`), `blake3` is the default multimodal content hasher (configurable to SHA256 via `VLLM_MM_HASHER_ALGORITHM`), and the OTLP gRPC exporter defaults to insecure channels. For FIPS-compliant deployments, operators should set `VLLM_MM_HASHER_ALGORITHM=sha256` and ensure the OTLP endpoint uses TLS.

**Security Surface**: The API server exposes a broad surface — 40+ HTTP endpoints plus a gRPC service. Authentication (Bearer token with SHA256 hashing and constant-time comparison) only protects `/v1/*` paths; operational endpoints (`/pause`, `/resume`, `/sleep`, `/wake_up`, `/update_weights`, `/collective_rpc`) are unauthenticated by design. In production, network policies or service mesh authorization should restrict access to these operational endpoints. The CORS configuration defaults to allowing all origins, methods, and headers, which should be restricted in production deployments.
