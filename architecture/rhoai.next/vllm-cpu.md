## Purpose

**Short**: CPU-only build of vLLM, a high-throughput inference and serving engine for Large Language Models, optimized for running on x86_64, ppc64le, and s390x architectures without GPU hardware.

**Detailed**: vllm-cpu is a downstream fork of the upstream vLLM project, specifically built and packaged by Red Hat for RHOAI (Red Hat OpenShift AI). It provides an OpenAI-compatible API server for serving LLM inference on CPU-only hardware. The component is deployed as a container image (`odh-vllm-cpu-rhel9`) built via Konflux pipelines targeting the `linux/s390x` and `linux/ppc64le` architectures.

The server exposes a FastAPI-based HTTP API compatible with the OpenAI Chat Completions, Completions, Embeddings, and Responses APIs, as well as an Anthropic Messages API endpoint. It also supports a gRPC server mode for high-performance inference. The engine uses PyTorch for CPU-based tensor computation with architecture-specific optimizations including Intel Extension for PyTorch on x86_64, OpenBLAS on ppc64le, and LLVM-based optimizations on s390x. vLLM's V1 engine architecture is enabled by default (`VLLM_USE_V1=1`), providing improved scheduling, memory management, and throughput compared to the legacy engine.

In the RHOAI platform, vllm-cpu is typically deployed as a KServe InferenceService runtime, providing model serving capabilities for users who need LLM inference without GPU resources. It integrates with the platform's model serving infrastructure, supporting model loading from Hugging Face Hub or S3-compatible storage.

## Architectural Analysis

vllm-cpu is a downstream packaging of the upstream vLLM project, focused specifically on CPU-only inference. The architecture centers around a FastAPI application that wraps vLLM's V1 async engine (`AsyncLLM`), exposing a comprehensive set of REST APIs compatible with both OpenAI and Anthropic client libraries. The V1 engine is the default (`VLLM_USE_V1=1` in the Dockerfile), providing improved scheduling and throughput over the legacy engine.

The build system is notably complex due to multi-architecture support. The `Dockerfile.konflux.cpu` uses architecture-conditional logic (`uname -m` checks) to handle three distinct CPU architectures: x86_64 (with Intel Extension for PyTorch), ppc64le (with source-built OpenBLAS, numactl, and LAPACK), and s390x (with source-built LLVM and additional system libraries). Each architecture has its own build script (`build_vllm_ppc64le.sh`, `build_vllm_s390x.sh`) that compiles architecture-specific dependencies from source. The final image uses `uv` for Python package management and `gcc-toolset-14` for compilation. A notable optimization is `LD_PRELOAD=/usr/lib64/libtcmalloc.so.4` which preloads Google's tcmalloc for improved memory allocation performance.

The API surface is extensive — beyond the core OpenAI-compatible `/v1/chat/completions` and `/v1/completions`, it includes the Anthropic Messages API (`/v1/messages`), the OpenAI Responses API (`/v1/responses`), pooling/embedding/reranking endpoints, speech-to-text transcription, WebSocket-based real-time streaming, MCP tool server integration, and operational endpoints for LoRA adapter hot-loading, profiling, sleep/wake management, and cache control. Authentication is optional and uses a simple API key mechanism via the `AuthenticationMiddleware`. TLS is also optional but fully configurable, including dynamic certificate rotation via the `SSLCertRefresher` that uses `watchfiles` to monitor certificate file changes.

The gRPC server mode (`vllm.entrypoints.grpc_server`) provides an alternative high-performance interface using the `smg-grpc-servicer` package. It currently only supports insecure ports (`add_insecure_port`), which is a security consideration for production deployments — TLS would need to be provided at the network layer (e.g., via service mesh or load balancer). The gRPC server uses reflection for discoverability and configures keepalive parameters with a 10-second minimum ping interval to handle long-running inference requests.

The Konflux build pipeline (defined in `.tekton/`) is non-hermetic (`hermetic: false`), which means dependencies are fetched at build time from external sources (PyPI, IBM DevPI for ppc64le wheels, etc.). This is a supply chain risk that is typical for early-access branches but should be tightened for GA releases. The pipeline targets `linux/s390x` and `linux/ppc64le` architectures and publishes to `quay.io/rhoai/odh-vllm-cpu-rhel9`. The container runs as non-root user 2000 in group 0, following OpenShift security best practices.
