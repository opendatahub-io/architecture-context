## Purpose

**Short**: CPU-optimized build of the vLLM inference engine, providing an OpenAI-compatible API server for large language model serving without GPU hardware.

**Detailed**: vllm-cpu is a Red Hat downstream fork of the vLLM project, specifically configured and built for CPU-only inference on x86_64, ppc64le, and s390x architectures. The component ships as a container image (`odh-vllm-cpu-rhel9`) that exposes an OpenAI-compatible HTTP API server and an optional gRPC server for serving LLM inference requests. It supports a wide range of model architectures from HuggingFace Hub, including quantized models (AWQ, GPTQ, INT4), and provides features such as prefix caching, speculative decoding, continuous batching, and LoRA adapter hot-loading.

The CPU backend includes platform-specific optimizations: Intel Extension for PyTorch on x86_64, OpenMP thread binding, SGL kernels for small-batch inference, and ZenDNN weight prepacking for AMD Zen CPUs. The container is built via Konflux pipelines targeting UBI9 minimal base images and is designed for deployment as a KServe InferenceService or standalone serving container in RHOAI environments. The entrypoint runs the OpenAI-compatible API server on port 8000 by default, with health endpoints for Kubernetes liveness/readiness probes.

## Architectural Analysis

The vllm-cpu component is a substantial downstream fork of the upstream vLLM project, adapted for CPU-only inference across three architectures (x86_64, ppc64le, s390x). The fork's primary value is in the Konflux build infrastructure and architecture-specific build scripts (`build_vllm_ppc64le.sh`, `build_vllm_s390x.sh`) that compile native dependencies from source on non-x86 platforms. The s390x build is particularly complex — it builds LLVM 15, Apache Arrow, numactl, and llvmlite from source because pre-built packages are unavailable. This adds significant build time (8-hour Tekton pipeline timeout) and maintenance burden.

The API surface is extremely broad: the OpenAI-compatible HTTP server exposes chat completions, raw completions, embeddings, responses, speech-to-text, tokenization, LoRA management, Anthropic Messages API compatibility, and SageMaker compatibility endpoints — all on a single port (8000). The gRPC server on port 50051 provides an alternative interface via the `smg-grpc-servicer` package. Health endpoints (`/health`, `/ping`) are unauthenticated and suitable for Kubernetes probes. The Prometheus metrics endpoint at `/metrics` provides inference-specific metrics (token throughput, batch sizes, cache utilization).

The CPU backend has dedicated optimizations in `csrc/cpu/`: custom C++ kernels for attention (`cpu_attn.cpp`), MoE fusion (`cpu_fused_moe.cpp`), quantization (`cpu_wna16.cpp`), and Intel DNNL integration (`dnnl_kernels.cpp`). SGL kernels in `csrc/cpu/sgl-kernels/` provide small-batch optimized GEMM operations in INT4, INT8, and FP8 precisions. The engine supports extensive configuration via 200+ environment variables, covering KV cache sizing, OpenMP thread binding, quantization kernel selection, and distributed inference parameters.

A significant architectural gap is the absence of AIPCC base images. The Konflux Dockerfile uses `registry.access.redhat.com/ubi9/ubi-minimal` directly instead of the AIPCC CPU base image (`quay.io/aipcc/base-images/cpu`). This means the component does not benefit from the RHEL AI Python Package Index, pre-configured pip/uv, or the standardized AIPCC container layout. Migration to AIPCC base images is required for compliance with Red Hat's secure build requirements and would simplify dependency management by leveraging the curated package index.

The build is explicitly non-hermetic (`hermetic: "false"` in Tekton pipelines), with dependencies fetched from PyPI, the PyTorch CPU index, and IBM DevPI at build time. No lock files (rpms.lock.yaml, uv.lock) are present on the main branch. This creates a supply chain risk: builds are not reproducible and depend on the availability and integrity of external package indices at build time. Downstream release branches are expected to add lock files during release hardening.
