## Purpose

**Short**: The rhods-operator is the platform operator for Red Hat OpenShift AI (RHOAI), responsible for deploying, configuring, and lifecycle-managing all platform components, services, ingress infrastructure, authentication, monitoring, and RBAC.

**Detailed**: The rhods-operator is the central orchestration layer for Red Hat OpenShift AI. It manages the full platform lifecycle through two primary custom resources: `DataScienceCluster` (DSC) which defines which AI/ML components to deploy (Dashboard, Workbenches, KServe, Kueue, Ray, TrustyAI, Model Registry, Training Operator, Trainer, Feast, OGX, MLflow, Spark, ModelsAsService, ModelController), and `DSCInitialization` (DSCI) which configures platform-wide infrastructure (namespaces, networking, CA bundles, monitoring, authentication, gateway).

The operator follows a hierarchical controller pattern: the DSC controller creates per-component CRs (e.g., `Dashboard`, `Kserve`, `Kueue`), each managed by a dedicated component controller that renders and deploys kustomize manifests from upstream component repositories. Service controllers manage cross-cutting concerns: the Gateway controller deploys the platform's entire ingress stack (Gateway API, Envoy, EnvoyFilter, kube-auth-proxy, dashboard redirects), the Auth controller manages RBAC group bindings, and the Monitoring controller deploys Prometheus, Tempo, OpenTelemetry, and Perses observability infrastructure.

On non-OpenShift Kubernetes (xKS) platforms such as Azure AKS and CoreWeave, a separate `cloudmanager` binary handles dependency bootstrapping (cert-manager, Istio/Sail, LeaderWorkerSet, Gateway API CRDs) via Helm charts before the main operator reconciles.

## Architectural Analysis

### Hierarchical Controller Pattern

The operator employs a three-tier controller hierarchy. At the top, the **DSCInitialization controller** creates platform infrastructure (namespaces, network policies, CA bundles) and triggers creation of three singleton service CRs: Auth, GatewayConfig, and Monitoring. The **DataScienceCluster controller** sits at the second tier, iterating over the component registry and creating per-component CRs (e.g., Kserve, Dashboard) based on the user's `managementState` settings. At the third tier, **individual component controllers** each manage their own reconciliation loop — rendering kustomize manifests from upstream component repositories, deploying them, and reporting status back to the DSC.

This design enables independent component lifecycle management, horizontal scalability of the codebase (each component is a self-contained package), and a clean separation between platform-level concerns and component-level concerns. The component handler interface (`Init`, `NewCRObject`, `NewComponentReconciler`, `UpdateDSCStatus`, `IsEnabled`) provides a consistent contract that all 15 components implement.

### Action Pipeline Reconciler Framework

Rather than monolithic reconcile functions, the operator uses a sequential **action pipeline** pattern (defined in `pkg/controller/reconciler/`). Each controller defines an ordered list of actions (e.g., `renderKustomizeManifests` → `deployResources` → `trackDeploymentStatus` → `garbageCollect`), and the reconciler executes them in sequence, stopping on error. This approach makes reconciliation logic composable and testable — new steps can be inserted without modifying existing code. The framework also supports **preconditions** (evaluated before the pipeline runs), **finalizers** (run during deletion), and **dynamic ownership** (runtime GVK registration for garbage collection).

### Multi-Mode Ingress Architecture

The Gateway controller supports two distinct ingress modes, automatically detected based on cluster capabilities. In **LoadBalancer mode** (default on xKS), a Gateway API `Gateway` resource is created with explicit HTTPS listeners and a cloud load balancer provisions an external IP. In **OcpRoute mode** (OpenShift clusters where LoadBalancer is unavailable or undesired), the Gateway uses a ClusterIP service with an OpenShift Route providing external access via re-encrypt TLS termination. Both modes share the same authentication stack (EnvoyFilter + kube-auth-proxy), but differ in TLS handling and external exposure mechanism.

The authentication architecture itself supports three modes: **IntegratedOAuth** (using OpenShift's built-in OAuth server with an OAuthClient CR), **OIDC** (using an external OIDC provider configured via GatewayConfig), and **None** (no authentication, gateway deployed without auth proxy). The dual-deployment pattern (`kube-auth-proxy-oauth-deployment` vs `kube-auth-proxy-oidc-deployment`) ensures the correct OAuth2 proxy configuration is used for each mode.

### Dashboard Redirect Infrastructure

A separate nginx-based redirect service handles backward compatibility for legacy dashboard and gateway URLs. When the gateway hostname changes (e.g., during migration from RHOAI 2.x Routes to 3.x Gateway API), the redirect infrastructure creates OpenShift Routes on the old hostnames pointing to an nginx deployment that returns 301 redirects to the new gateway-hosted URLs. This is deployed as its own Deployment + Service + ConfigMap + Route stack, independent of the main gateway, and can be disabled via the `DISABLE_DASHBOARD_REDIRECTS` environment variable.

### Cloud Manager for xKS Platforms

The `cloudmanager` binary provides a separate operator for non-OpenShift Kubernetes platforms (Azure AKS, CoreWeave). It uses the `AzureKubernetesEngine` and `CoreWeaveKubernetesEngine` CRDs to manage dependency installation via Helm charts: cert-manager operator, Sail operator (Istio), LeaderWorkerSet operator, and Gateway API CRDs. The cloud manager includes hooks for working around known issues in upstream operators (e.g., annotating Istio webhooks with `sailoperator.io/ignore=true` for OSSM-12397) and supports Managed/Unmanaged states per dependency for flexible deployment control.

### Comprehensive Webhook System

The operator registers 12 active admission webhooks covering validation, defaulting, and resource injection. Notable patterns include: **singleton enforcement** (only one DSC and one DSCI allowed per cluster), **API version migration guards** (preventing v1 API updates when v2-only components are enabled, avoiding data loss), **hardware profile injection** (dynamically applying GPU/CPU resource requirements, tolerations, nodeSelector, and Kueue labels to Notebooks and InferenceServices), and **connection injection** (validating user permissions via SubjectAccessReview before injecting secret references into notebook and serving workloads). The Kueue validation webhook exists but is currently disabled.
