## Purpose

**Short**: Platform operator that manages the full lifecycle of Red Hat OpenShift AI (RHOAI), deploying and orchestrating 16+ data science components, platform services, ingress infrastructure, monitoring, and RBAC across the cluster.

**Detailed**: The rhods-operator (also known as opendatahub-operator) is the central platform operator for Red Hat OpenShift AI (RHOAI). It uses a hierarchical CRD model where `DSCInitialization` (DSCI) provisions platform infrastructure (namespaces, networking, monitoring, auth, gateway) and `DataScienceCluster` (DSC) manages the lifecycle of individual data science components (Dashboard, KServe, Kueue, Workbenches, Model Registry, Data Science Pipelines, TrustyAI, Ray, Training Operator, Trainer, Feast Operator, LlamaStack Operator, MLflow Operator, Spark Operator, Model Controller, and Models-as-a-Service).

The operator implements a sophisticated ingress architecture using Kubernetes Gateway API with an Envoy-based data plane, kube-auth-proxy for OAuth2/OIDC authentication, EnvoyFilter CRs for traffic shaping, and OpenShift Routes for legacy URL redirects. It supports three authentication modes (integrated OpenShift OAuth, external OIDC, and no-auth/external management) and two ingress modes (OCP Route and LoadBalancer). The operator also deploys a complete monitoring stack including MonitoringStack, Prometheus, ThanosQuerier, Tempo, OpenTelemetry Collector, and Perses for observability.

Each component is managed through a plugin-like registry pattern with its own CRD (e.g., `Dashboard`, `Kserve`, `Kueue`), reconciler, and manifest deployment pipeline. Components can be independently enabled, disabled, or set to unmanaged state. The operator uses a modular action-based reconciliation framework with template rendering, server-side apply, and garbage collection. It also includes a `cloudmanager` binary for managing cloud-specific infrastructure (Azure AKS, CoreWeave).

## Architectural Analysis

The rhods-operator demonstrates a sophisticated hierarchical controller architecture with clear separation between platform infrastructure (DSCI) and component lifecycle (DSC). The DSCI controller provisions foundational resources - namespaces with correct labels, network policies, CA bundle injection, and creates singleton CRs for Auth, GatewayConfig, and Monitoring services. The DSC controller then iterates through a component registry, creating individual component CRs that trigger their respective controllers. This two-tier model enables the platform to bootstrap itself: DSCI ensures infrastructure exists before DSC deploys workloads.

The gateway controller is the most architecturally complex piece, implementing a full ingress stack that rivals dedicated ingress controllers. It manages 14+ Kubernetes resource types across two namespaces, supports three authentication modes (integrated OAuth, external OIDC, no-auth), and two ingress modes (OCP Route, LoadBalancer). The authentication flow uses a three-layer proxy chain: Envoy (via Gateway API) receives traffic, delegates authentication to kube-auth-proxy via ext_authz EnvoyFilter, and a Lua filter then extracts and forwards access tokens as Bearer headers to backend services. The cookie management is particularly intricate - the Lua filter strips `_oauth2_proxy` cookies from forwarded requests (including split cookie variants) to prevent them from leaking to backend services while preserving them for the auth flow itself.

The monitoring architecture follows an atomic deployment pattern where missing CRDs cause all related conditions to be set to False rather than partially deploying. The controller manages MonitoringStack (Prometheus), ThanosQuerier, Tempo (distributed tracing), OpenTelemetry Collector, and Perses (dashboards) as a cohesive observability stack. Each component contributes Prometheus rules via its monitoring/ subdirectory, and the monitoring controller dynamically adds/removes these rules based on component enable/disable state. The custom metrics exporters feature (via OTEL Collector) allows forwarding metrics and traces to external observability tools.

A notable design decision is the component registry pattern, where all 16 components and 5 services implement handler interfaces and are registered in a central map. Components can be suppressed at startup via CLI flags (useful for xKS/cloud deployments where not all components are needed). Each component's Init() function sets up image parameter maps (RELATED_IMAGE_* env vars) and manifest paths, while NewComponentReconciler() creates the controller-runtime reconciler. The action-based reconciliation framework (initialize → template → deploy → gc) provides consistent behavior across all controllers. The `deploy.NewAction()` uses server-side apply with cache optimization, and `gc.NewAction()` garbage-collects resources that were removed from the reconciliation request.

The operator is increasingly multi-platform: alongside traditional OpenShift deployments, it supports xKS (cross-Kubernetes) mode where DSCI is disabled, the application namespace is injected via environment variable, and version detection bypasses OLM/CSV. The cloudmanager binary manages AzureKubernetesEngine and CoreWeaveKubernetesEngine CRDs for multi-cloud infrastructure. The build uses Go build tags (`-tags rhoai`) to compile platform-specific code paths, particularly around the DSCI spec (immutable namespace for RHOAI vs configurable for ODH) and default admin groups.
