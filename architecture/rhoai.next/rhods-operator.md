## Purpose

**Short**: Platform operator that orchestrates the full lifecycle of Red Hat OpenShift AI (RHOAI), managing 16+ data science components, platform services (gateway, auth, monitoring), and multi-cloud infrastructure.

**Detailed**: The rhods-operator is the central control plane for Red Hat OpenShift AI. It defines and reconciles two primary CRDs — `DataScienceCluster` (DSC) and `DSCInitialization` (DSCI) — which together declare the desired state of the entire AI/ML platform. The DSC controller provisions individual component CRs (Dashboard, KServe, Kueue, Workbenches, etc.) based on the user's enablement choices, while the DSCI controller bootstraps platform infrastructure: application namespaces, network policies, monitoring, authentication, and the Gateway API ingress stack.

The operator deploys a complete ingress architecture using Gateway API (RHOAI 3.x pattern): a `Gateway` CR with Envoy as the data plane, `EnvoyFilter` CRs for authentication via ext_authz, `kube-auth-proxy` (OAuth2 proxy) for OpenShift OAuth or external OIDC, `HTTPRoute` CRs for per-component routing, and `NetworkPolicy` resources for traffic isolation. Legacy OpenShift Routes are maintained for redirect purposes (301 redirects from old hostnames to new Gateway API hostnames) and OAuth callback flows.

The operator also includes a Cloud Manager subsystem that provisions dependency operators (cert-manager, Istio/OSSM, LeaderWorkerSet, Gateway API CRDs) via Helm charts on non-OpenShift Kubernetes clusters (AWS EKS, Azure AKS, CoreWeave), and a comprehensive webhook system that enforces singleton constraints, deprecation policies, hardware profile injection, connection secret injection, and monitoring label propagation.

## Architectural Analysis

The rhods-operator is one of the most architecturally complex components in the RHOAI platform. It serves as the single orchestration point for the entire AI/ML platform, managing 16+ components, 3 platform services, and a multi-cloud dependency management system — all from a single binary.

**Multi-level controller hierarchy**: The operator uses a three-level controller pattern. The DSCInitialization controller bootstraps platform infrastructure (namespaces, network policies, monitoring, auth, gateway). The DataScienceCluster controller orchestrates component CRs based on user enablement choices. Each component CR is reconciled by a dedicated component controller that renders and deploys component-specific manifests. This separation enables independent component lifecycle management while maintaining platform-wide coordination. A fourth level — the modules controller — handles operator-level dependencies (external operators deployed as modules) using either Helm or Kustomize rendering.

**Gateway API ingress stack (RHOAI 3.x)**: The gateway controller deploys a complete multi-layer ingress stack: Gateway CR → Envoy proxy → EnvoyFilter (ext_authz + Lua) → kube-auth-proxy (OAuth2 proxy) → component backends. This replaces the RHOAI 2.x pattern (Routes + oauth-proxy sidecars) with a centralized auth proxy. The EnvoyFilter Lua script deserves attention — it strips OAuth2 proxy cookies from upstream requests after authentication (preventing cookie bloat) and injects Bearer tokens into the Authorization header, enabling downstream components to receive standard OAuth tokens without needing their own auth sidecars. The legacy dashboard and gateway hostnames are handled via nginx redirect Deployments that 301-redirect to the new Gateway hostname.

**TLS profile compliance**: The gateway_tls.go code demonstrates careful FIPS compliance integration. It reads the OpenShift APIServer CR to get the cluster-wide TLS security profile, converts OpenSSL cipher suite names to IANA names (using openshift/library-go), floors unsupported configurations to the Intermediate profile, and injects the resulting TLS parameters into kube-auth-proxy Deployments. This ensures the auth proxy's TLS configuration automatically aligns with cluster-wide FIPS policy without manual configuration.

**Platform detection and conditional behavior**: The operator detects its running platform at startup (SelfManagedRhoai, ManagedRhoai, OpenDataHub, or XKS) and adjusts behavior accordingly. This affects admin group names, telemetry configuration, namespace labels, manifest overlays, and even which controllers are enabled. The XKS (external Kubernetes) mode disables DSC/DSCI entirely and uses a Platform CR for module enablement — a different operational model for non-OpenShift deployments.

**Cloud Manager subsystem**: The cloudmanager binary provides a separate control plane for provisioning dependency operators on non-OpenShift clusters. It manages AWSKubernetesEngine, AzureKubernetesEngine, and CoreWeaveKubernetesEngine CRs, deploying cert-manager, Istio (Sail Operator), LeaderWorkerSet, Gateway API CRDs, and Kyverno via Helm charts. The common/hooks.go file contains interesting workarounds — notably annotating Istio webhooks with `sailoperator.io/ignore=true` to prevent an infinite reconciliation loop (OSSM-12397), and skipping CRD installation when CRDs are already managed by another operator (preventing SSA conflicts).

**Comprehensive webhook system**: The 14 webhook types serve distinct architectural purposes. The singleton enforcement webhooks prevent multiple DSC/DSCI instances. The deprecation webhooks enforce API migration (AcceleratorProfile → HardwareProfile). The hardware profile webhook performs sophisticated resource injection — applying CPU/memory/GPU limits, node selectors, tolerations, and Kueue labels to Notebooks, InferenceServices, and LLMInferenceServices. The connection webhooks inject OCI pull secrets, S3 storage credentials, and URI references into workloads — with SubjectAccessReview validation to ensure users have permission to reference the secrets they specify.
