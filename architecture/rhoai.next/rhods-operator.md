## Purpose

**Short**: The RHOAI Operator is the central platform operator that manages the complete lifecycle of Red Hat OpenShift AI, including component deployment, ingress infrastructure, authentication, monitoring, and RBAC across the entire platform.

**Detailed**: The rhods-operator (also known as opendatahub-operator) is the top-level Kubernetes operator for the Red Hat OpenShift AI (RHOAI) platform. It uses two primary CRDs — `DSCInitialization` (DSCI) for platform-level initialization (namespaces, networking, monitoring, auth, gateway) and `DataScienceCluster` (DSC) for managing individual AI/ML components (Dashboard, KServe, Kueue, Ray, TrustyAI, Model Registry, Training Operator, Trainer, Data Science Pipelines, Feast, LlamaStack, MLflow, Spark, Models-as-a-Service, and Workbenches).

The operator deploys the platform's entire ingress infrastructure through its Gateway controller, which manages Gateway API resources (GatewayClass, Gateway, HTTPRoute), an Envoy proxy data plane, EnvoyFilter CRs for authentication enforcement, a kube-auth-proxy deployment for OAuth2/OIDC authentication, DestinationRules for mTLS, NetworkPolicies, and nginx-based redirect services for legacy URL migration. It also manages a comprehensive monitoring stack (Prometheus via MonitoringStack, Thanos, OpenTelemetry Collector, Tempo for traces, and Perses for visualization) and platform-wide RBAC through its Auth controller.

The operator supports multiple platforms: OpenShift (SelfManagedRhoai, ManagedRhoai), Open Data Hub (ODH), and cross-Kubernetes (XKS) via its CloudManager controllers (Azure Kubernetes Engine, CoreWeave Kubernetes Engine). Component manifests are fetched from upstream repositories via `get_all_manifests.sh` and deployed using kustomize with platform-specific overlays. The operator produces two binaries: the main `manager` and a `cloudmanager` for cloud-specific infrastructure on non-OpenShift platforms.

## Architectural Analysis

The rhods-operator is one of the most architecturally complex operators in the RHOAI/ODH ecosystem, serving as the single control plane for the entire AI/ML platform. Its design reflects several key architectural decisions worth noting.

**Component Registry Pattern**: The operator uses a registry-based pattern (`internal/controller/components/registry/`) where each AI/ML component is registered as a `ComponentHandler` during startup in `cmd/main.go`. The DSC controller iterates this registry to provision components, allowing new components to be added with minimal boilerplate. Each component handler follows a consistent interface: `Init()`, `GetManagementState()`, `NewReconciler()`, and provides manifest paths and container image env var mappings. This pattern has scaled well to 16+ components but creates a monolithic operator binary that manages an extremely broad set of resources.

**Action Chain Reconciliation**: Controllers compose reconciliation logic as chains of typed "actions" — small, focused functions executed sequentially during reconcile. For example, the Gateway controller executes 10 actions in order: `createGatewayInfrastructure → createKubeAuthProxyInfrastructure → createEnvoyFilter → createNetworkPolicy → createOCPRoutes → createDashboardRedirects → template.NewAction → deploy.NewAction → syncGatewayConfigStatus → gc.NewAction`. This pattern provides clear separation of concerns within a single controller and makes the reconciliation flow auditable, but can make error handling complex when early actions fail.

**Multi-Layer Ingress Architecture (RHOAI 3.x)**: The Gateway controller deploys a sophisticated multi-layer ingress stack that replaces the 2.x pattern of per-component OpenShift Routes + oauth-proxy sidecars. The new architecture uses Gateway API (GatewayClass → Gateway → HTTPRoutes) with an Envoy data plane, EnvoyFilter CRs for ext_authz integration, a centralized kube-auth-proxy deployment for OAuth2/OIDC, DestinationRules for backend mTLS, and NetworkPolicies restricting access. This centralization reduces the per-component auth burden but creates a critical single point of failure — if the gateway controller or auth proxy goes down, all platform ingress is affected. The operator supports two ingress modes: `LoadBalancer` (primary, creates Gateway with external LB) and `OcpRoute` (creates Routes backed by the OpenShift router), with automatic detection based on existing infrastructure.

**Legacy URL Migration**: The operator includes dedicated infrastructure for 301-redirecting legacy URLs. When migrating from RHOAI 2.x to 3.x, the dashboard and gateway hostnames change. The controller deploys nginx-based redirect services (ConfigMap with nginx.conf → Deployment → Service → Route) that redirect old URLs to the new Gateway API hostname. This ensures users with bookmarked URLs are seamlessly redirected. The redirect infrastructure is deployed conditionally and can be disabled via the `DISABLE_DASHBOARD_REDIRECTS` environment variable.

**Platform-Aware Design**: The operator detects its runtime platform (SelfManagedRhoai, ManagedRhoai, ODH, XKS) via `cluster.Init()` and adapts behavior accordingly. On OpenShift, it uses OAuth, Routes, and OLM. On XKS (cross-Kubernetes), it requires explicit `RHAI_APPLICATIONS_NAMESPACE` and `RHAI_VERSION` configuration, disables DSCI, and relies on the cloudmanager binary for infrastructure prerequisites. The cloudmanager deploys cert-manager, Istio/Sail, LeaderWorkerSet, and Gateway API CRDs via Helm charts with pre/post-apply hooks (e.g., annotating Istio webhooks to work around sail-operator bugs). This multi-platform support adds significant complexity but enables RHOAI to run beyond OpenShift.

**Monitoring Stack Depth**: The monitoring controller deploys a production-grade observability stack that goes well beyond basic Prometheus. It manages MonitoringStack (Prometheus with web TLS, alerting, and retention), ThanosQuerier (for federated queries), OpenTelemetryCollector (with TargetAllocator for dynamic pod discovery), Tempo (for distributed tracing), and Perses (for visualization dashboards). Each component has conditional deployment — metrics, traces, and alerting can be independently enabled. The monitoring stack exposes Prometheus behind kube-rbac-proxy sidecars (namespace-proxy and cluster-proxy) with NetworkPolicies restricting ingress, providing multi-tenant metrics access with Kubernetes RBAC enforcement.
