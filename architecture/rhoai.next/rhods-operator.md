## Purpose

**Short**: Platform operator that deploys, configures, and manages the entire Red Hat OpenShift AI (RHOAI) platform, including all AI/ML components, ingress infrastructure, authentication, monitoring, and cloud manager controllers.

**Detailed**: The rhods-operator (also known as opendatahub-operator) is the central platform operator for Red Hat OpenShift AI. It manages the full lifecycle of the AI/ML platform through two primary Custom Resources: `DSCInitialization` (DSCI) for platform-level infrastructure setup (namespaces, networking, auth, monitoring, gateway) and `DataScienceCluster` (DSC) for enabling/disabling individual AI/ML components (Dashboard, KServe, Kueue, Ray, Workbenches, Model Registry, Data Science Pipelines, TrustyAI, and more).

The operator deploys a sophisticated multi-layer ingress architecture using Gateway API, Envoy proxy with EnvoyFilter CRs for authentication, kube-auth-proxy for OAuth/OIDC enforcement, and OpenShift Routes for backward compatibility. It manages 16+ component controllers, platform services (auth, gateway, monitoring, cert config map generator), webhook validators/mutators, and cloud manager controllers for AWS, Azure, and CoreWeave Kubernetes deployments.

The operator supports dual-mode operation: DSC mode (OpenShift/ODH with full DSC + DSCI context) and Platform mode (vanilla Kubernetes with Platform CR). It uses a registry-based handler pattern where each component implements a standard interface for lifecycle management, status aggregation, and image override injection via RELATED_IMAGE_* environment variables from the operator CSV.

## Architectural Analysis

The rhods-operator is one of the most architecturally complex components in the RHOAI platform, functioning as both a meta-operator (managing 16+ sub-component lifecycle) and an infrastructure operator (deploying the platform's entire ingress, auth, and monitoring stack). Its design reflects several key architectural decisions:

**Registry-Based Component Management**: All components implement a standard `ComponentHandler` interface (`GetName()`, `Init()`, `NewCRObject()`, `IsEnabled()`, `UpdateDSCStatus()`). The DSC controller iterates a registry to provision components, making it straightforward to add new components without modifying orchestration logic. Each component controller independently manages its own reconciliation pipeline (preconditions → initialize → kustomize → deploy → gc) using an action-chain pattern from `pkg/controller/reconciler`.

**Multi-Layer Ingress Architecture**: The Gateway controller deploys a sophisticated ingress stack: GatewayClass → Gateway → Envoy proxy → EnvoyFilter (ext_authz + Lua) → kube-auth-proxy → backend applications. This architecture supports both OAuth (OpenShift-integrated) and OIDC (external provider) authentication modes, with automatic detection via `cluster.GetClusterAuthenticationMode()`. The Lua filter performs critical token forwarding and cookie stripping between the auth proxy and backend applications. Dashboard redirect resources (nginx Deployment + Service + Route) handle backward compatibility when hostnames change.

**Dual-Mode Operation**: The operator supports both DSC mode (OpenShift with DSCI + DSC) and Platform mode (vanilla Kubernetes with Platform CR). This is controlled by feature flags (`flags.IsDSCEnabled()`, `flags.IsDSCIEnabled()`) and affects which controllers are registered at startup. Cloud manager controllers (AWS, Azure, CoreWeave) only activate in Platform mode to manage dependency operators via Helm charts.

**Dynamic Resource Ownership**: The operator uses dynamic ownership and garbage collection patterns. Resources are owned by their parent CRs via `controllerutil.SetControllerReference`. The GC action verifies ownership before deletion, and watches are registered dynamically based on CRD existence (e.g., EnvoyFilter and DestinationRule are only watched when their CRDs exist on the cluster). This prevents errors when optional dependencies aren't installed.

**Webhook Architecture**: The operator registers 12 webhooks covering validation (singleton enforcement for DSC/DSCI, deprecation blocking for old API versions), mutation (hardware profile injection into Notebooks/InferenceServices, monitoring label injection, connection secret injection), and conversion (DSC v1↔v2, DSCI v1↔v2). The hardware profile webhook is particularly complex — it handles resource requirement injection, node scheduling, Kueue LocalQueue labels, and profile change detection with merge-vs-replace semantics.
