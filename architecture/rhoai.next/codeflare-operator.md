## Purpose

**Short**: Manages the lifecycle of RayCluster and AppWrapper resources, providing OAuth proxy integration, mTLS certificate management, and network policy enforcement for distributed AI/ML workloads on OpenShift.

**Detailed**: The CodeFlare operator is a controller-runtime based Kubernetes operator that reconciles RayCluster resources (from the KubeRay project) and optionally AppWrapper resources (from the CodeFlare AppWrapper project). Its primary role is to augment RayCluster deployments with enterprise security features required by the RHOAI platform.

When a RayCluster is created on OpenShift, the operator injects an OAuth proxy sidecar container into the head pod (via a mutating webhook), creates OpenShift Routes for both the Ray Dashboard and RayClient endpoints, provisions ServiceAccounts with OAuth redirect annotations, generates TLS certificates for intra-cluster mTLS communication between head and worker nodes, and deploys NetworkPolicies to restrict traffic flow. On vanilla Kubernetes, it instead creates Ingress resources for dashboard and client access.

The operator also embeds an optional AppWrapper controller (from the `project-codeflare/appwrapper` library) that integrates with Kueue for workload queuing and gang scheduling. The AppWrapper controller manages the lifecycle of wrapped Kubernetes resources (Deployments, Jobs, RayClusters, PyTorchJobs, etc.) and provides quota management and fault tolerance. Both controllers are protected by mutating and validating webhooks that ensure the operator-injected sidecar containers, volumes, and environment variables cannot be tampered with by users.

## Architectural Analysis

The CodeFlare operator follows a pragmatic design where it augments another operator's (KubeRay) resources rather than defining its own application-specific CRDs for Ray clusters. The operator watches `RayCluster` CRs owned by KubeRay and layers on enterprise security features — OAuth proxy, mTLS, and NetworkPolicies — that the upstream KubeRay operator does not provide. This "sidecar operator" pattern avoids CRD proliferation but creates a tight coupling: the CodeFlare operator must start *after* the KubeRay CRD is established, which it handles via a CRD watch-and-wait mechanism in `main.go` (lines 259-497).

The security architecture uses the RHOAI 2.x pattern: OpenShift Routes with oauth-proxy sidecars rather than the newer 3.x HTTPRoute + kube-rbac-proxy pattern. Each RayCluster gets a full set of per-cluster resources (OAuth Service, Route, ServiceAccount, ClusterRoleBinding, Secret, CA Secret, two NetworkPolicies) — this means cluster count directly scales RBAC object count, which could become a concern at high RayCluster density. The webhook validates that users cannot tamper with the injected sidecar containers, volumes, or environment variables, enforcing a "operator-managed security" model.

The mTLS implementation is notable: the operator generates a self-signed RSA 2048-bit CA per RayCluster, stores it in a Secret, and injects init containers into both head and worker pods that generate per-pod server certificates signed by this CA using the `openssl` CLI (from a UBI 9 image). The certificates are valid for 365 days with no automated rotation — the CA Secret persists for the lifetime of the RayCluster. This design means each RayCluster has its own trust domain, isolating mTLS per cluster, but lacks automated cert renewal for long-running clusters.

The AppWrapper controller is embedded as a Go library import from `project-codeflare/appwrapper`, not as a separate deployment. When enabled via the `codeflare-operator-config` ConfigMap, the operator calls `awctrl.SetupControllers()` and `awctrl.SetupWebhooks()` to register the AppWrapper controller and webhooks in the same manager process. This monolithic embedding simplifies deployment but means the AppWrapper feature lifecycle is tied to the operator's release cycle. The AppWrapper's RBAC footprint is notably wide — it requires full CRUD on pods, services, deployments, statefulsets, jobs, pytorchjobs, rayclusters, rayjobs, jobsets, and podgroups — because it must create and manage arbitrary wrapped resources.

The operator discovers its operational namespace dynamically through two mechanisms: the `NAMESPACE` environment variable (set via downward API in the Deployment) and the DSCInitialization CR (to find where KubeRay runs). The fallback chain for KubeRay namespace is: DSCI → {opendatahub, redhat-ods-applications} → pod label query → "ray-system" default. This makes the operator resilient to different installation configurations but adds complexity to NetworkPolicy rule generation, which must allow traffic from the correct KubeRay operator namespace.
