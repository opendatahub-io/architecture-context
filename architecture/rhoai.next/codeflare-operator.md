## Purpose

**Short**: Manages the lifecycle of RayCluster resources by injecting OAuth proxy sidecars, mTLS certificate generation, network policies, and OpenShift Routes for secure Ray Dashboard access, and optionally embeds the AppWrapper controller for Kueue-integrated batch workload management.

**Detailed**: The CodeFlare Operator is a controller-runtime based Kubernetes operator that watches `RayCluster` custom resources (from the KubeRay project) and augments them with platform-specific security and networking infrastructure. On OpenShift, it injects an `oauth-proxy` sidecar container into Ray head pods to provide authenticated access to the Ray Dashboard via OpenShift Routes with TLS reencrypt termination. It also creates a RayClient Route with TLS passthrough for direct Ray client connections. On vanilla Kubernetes, it creates Ingress resources instead.

When mTLS is enabled (the default), the operator generates a per-cluster CA certificate and injects init containers into both head and worker pods that generate per-node TLS certificates signed by the cluster CA. This ensures all Ray inter-node communication (GCS, object store, etc.) is encrypted. The operator also creates NetworkPolicies to restrict traffic to Ray head and worker pods, allowing only intra-cluster communication and access from the KubeRay operator namespace and OpenShift monitoring.

The operator optionally embeds the AppWrapper controller (from `project-codeflare/appwrapper`), which integrates with Kueue for quota-aware batch scheduling. When enabled, AppWrappers wrap Kubernetes workload resources (Deployments, Jobs, RayClusters, PyTorchJobs, JobSets) and manage their lifecycle with quota reservation and health monitoring. Both the RayCluster and AppWrapper controllers are protected by mutating and validating admission webhooks that enforce immutability of security-critical configuration.

## Architectural Analysis

The CodeFlare Operator follows a lean architecture pattern — rather than defining its own CRDs for its core function, it acts as a secondary controller for the KubeRay operator's `RayCluster` CRD. This avoids CRD proliferation while still enabling platform-specific security augmentation. The operator uses kubebuilder webhooks to inject security configuration at admission time (oauth-proxy sidecar, mTLS init containers, TLS env vars) and a reconcile loop to manage the surrounding infrastructure (Routes, Services, NetworkPolicies, Secrets). This split ensures that security configuration is applied before pods are created, while supporting infrastructure is managed declaratively.

The mTLS implementation is noteworthy: the operator generates a per-cluster RSA 2048-bit CA certificate and stores it as a Kubernetes Secret, then injects init containers into both head and worker pods that generate per-node certificates signed by this CA. The init containers use OpenSSL CLI commands (via a UBI9 image) rather than Go crypto libraries, which keeps the init container lightweight but means certificate generation depends on the OpenSSL binary being present in the cert generator image. The generated certificates include SANs for localhost, pod IP, Route hostname, and service DNS name, enabling both intra-cluster and external mTLS connections.

The operator's platform detection logic (`isOpenShift()`) dynamically switches behavior between OpenShift and vanilla Kubernetes. On OpenShift, it creates Routes and oauth-proxy sidecars; on Kubernetes, it creates Ingresses without authentication. This dual-mode approach is important for maintaining upstream compatibility while providing enterprise security features on OpenShift/RHOAI. The operator also dynamically discovers the KubeRay operator namespace by querying pods with the `app.kubernetes.io/component=kuberay-operator` label, which is used in NetworkPolicy rules to allow KubeRay operator access to Ray head pods.

The AppWrapper controller embedding is a design decision worth noting — rather than deploying a separate operator for AppWrapper/Kueue integration, the CodeFlare Operator embeds the AppWrapper controller and webhooks as a library dependency. This is controlled by the `appwrapper.enabled` configuration flag in the `codeflare-operator-config` ConfigMap. When disabled (the default), the operator applies label-based cache filtering to reduce memory usage, only caching Secrets, Services, ServiceAccounts, Ingresses, NetworkPolicies, and ClusterRoleBindings that carry the `ray.openshift.ai/cluster-name` label. When AppWrapper is enabled, this filtering is removed since the AppWrapper controller needs broader access to manage wrapped resources.

A potential risk area is the SHA1 usage for cookie secret generation (`sha1.New()` in `desiredOAuthSecret`). While SHA1 is not used for cryptographic authentication here (only for session cookie hashing), its use in security-adjacent code could trigger FIPS compliance audit findings. The comment `// REVIEW is SHA1 okay here?` in the source code indicates the team is aware of this concern.
