## Purpose

**Short**: Manages the lifecycle of RayCluster resources by injecting OAuth proxy sidecars, mTLS certificates, network policies, and OpenShift Routes to secure Ray Dashboard access and inter-node communication.

**Detailed**: The CodeFlare Operator is a Kubernetes operator built with controller-runtime and the Kubebuilder framework. It watches `RayCluster` custom resources (owned by the KubeRay operator) and reconciles a set of supporting infrastructure around each RayCluster to integrate it with the OpenShift platform's authentication and networking model.

On OpenShift, the operator injects an `oauth-proxy` sidecar container into the RayCluster head pod via a mutating webhook, creates OpenShift Routes for the Ray Dashboard (with TLS reencrypt termination) and the Ray client endpoint (with TLS passthrough), provisions ServiceAccounts and ClusterRoleBindings for OAuth delegation, and generates cookie secrets for session management. When mTLS is enabled (the default), it generates a CA certificate per cluster and injects init containers into both head and worker pods that create per-pod TLS certificates signed by that CA, enabling encrypted inter-node Ray communication.

The operator also embeds the AppWrapper controller (from the `project-codeflare/appwrapper` project), which can be optionally enabled to provide Kueue-integrated workload queuing for batch AI/ML jobs. The AppWrapper controller manages `AppWrapper` custom resources that wrap Kubernetes workloads (Deployments, Jobs, RayClusters, PyTorchJobs, etc.) with quota-aware scheduling semantics. Both the RayCluster and AppWrapper controllers register mutating and validating webhooks. The operator enforces network segmentation through NetworkPolicies that restrict worker pod ingress to same-cluster pods only, and head pod ingress to cluster pods, the KubeRay operator namespace, and OpenShift monitoring.

## Architectural Analysis

The CodeFlare Operator follows a clear separation between its two primary subsystems: the RayCluster infrastructure controller and the optional embedded AppWrapper controller. The RayCluster controller does not own the RayCluster CRD itself (that belongs to the KubeRay operator) but instead watches it and augments each RayCluster with platform-specific security infrastructure. This "sidecar operator" pattern is effective but creates a tight coupling: the operator must wait for the KubeRay `rayclusters.ray.io` CRD to be installed before it can start its controller, which it handles via a retry watcher that polls for CRD availability at startup (`waitForRayClusterAPIandSetupController` in `main.go`, lines 259-267).

The operator's security model is notably thorough. On OpenShift, it creates a full OAuth proxy chain per RayCluster: a dedicated ServiceAccount with an OAuthRedirectReference annotation, a ClusterRoleBinding to `system:auth-delegator`, a Service with the serving-cert-signer annotation for automatic TLS provisioning, a Route with reencrypt TLS termination, and a cookie Secret for session management. The mutating webhook ensures the oauth-proxy sidecar is always injected, and the validating webhook prevents users from removing or modifying the injected containers, volumes, and environment variables. This makes the OAuth infrastructure effectively immutable once a RayCluster is created. The operator uses a unique hash suffix scheme (`RCCUniqueName`) for all dynamically-created resource names to avoid collisions, with a backward-compatibility path for clusters created before this naming scheme was introduced (detected via the `ray.openshift.ai/version` annotation).

The mTLS implementation generates a self-signed CA per RayCluster and distributes it to both head and worker pods via init containers. The init containers use OpenSSL to generate per-pod certificates signed by the cluster CA, with SANs covering localhost, the pod IP, the Ray client route hostname, and the head service FQDN. This enables encrypted inter-node communication via Ray's built-in TLS support (`RAY_USE_TLS=1`). The CA certificates have a 1-year validity and are not automatically rotated, which could be a concern for long-lived clusters. The CA private key is stored in a Kubernetes Secret alongside the certificate, accessible to all pods in the namespace.

The NetworkPolicy design implements defense-in-depth: worker pods only accept traffic from pods in the same RayCluster (labeled `ray.io/cluster`), while head pods have a more permissive policy allowing traffic from the cluster's own pods, pods in the same namespace on the Ray client port (10001) and dashboard port (8265), the KubeRay operator namespace, OpenShift monitoring on port 8080, and any source on the secured ports (8443 for OAuth, 10001 for mTLS-protected Ray client). The KubeRay operator namespace is dynamically discovered by looking up pods with the `app.kubernetes.io/component=kuberay-operator` label, with fallback to the DSCI-configured applications namespace or well-known defaults (`opendatahub`, `redhat-ods-applications`).

The operator configuration is loaded from a ConfigMap (`codeflare-operator-config`) at startup, and if the ConfigMap doesn't exist, the operator creates it with defaults (OAuth enabled, mTLS enabled, AppWrapper disabled). This design means the operator is self-configuring on first deployment. The operator also detects whether it's running on OpenShift vs vanilla Kubernetes by checking for API groups ending in `.openshift.io`, and adjusts its behavior accordingly (Routes vs Ingresses, OAuth vs no auth).
