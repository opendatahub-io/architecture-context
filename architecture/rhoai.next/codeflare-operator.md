## Purpose

**Short**: Kubernetes operator that manages RayCluster lifecycle with OAuth proxy injection, mTLS certificate generation, and optional embedded AppWrapper controller for batch workload scheduling.

**Detailed**: The CodeFlare Operator is a controller-runtime-based Kubernetes operator built with the Kubebuilder framework. It watches RayCluster custom resources (from the KubeRay project) and augments them with enterprise security features. On OpenShift, it injects an OAuth proxy sidecar container into RayCluster head pods to protect the Ray Dashboard with OpenShift OAuth authentication, creates associated Routes for dashboard and Ray client access, and manages the full OAuth lifecycle (ServiceAccounts, Secrets, ClusterRoleBindings). On vanilla Kubernetes, it creates Ingress resources instead of Routes for dashboard and client access.

The operator also manages mutual TLS (mTLS) between Ray head and worker nodes by generating CA certificates, creating init containers that produce per-node TLS certificates, and injecting TLS environment variables into Ray containers. This ensures all inter-node Ray communication is encrypted.

Additionally, the operator embeds an optional AppWrapper controller (from the project-codeflare/appwrapper project) that integrates with Kueue for quota-aware batch workload scheduling. When enabled, AppWrappers provide a higher-level abstraction for submitting and managing multi-resource workloads with quota enforcement. The operator creates per-RayCluster NetworkPolicies to restrict traffic to only necessary communication paths.

## Architectural Analysis

The CodeFlare Operator follows a dual-platform architecture pattern, adapting its behavior based on whether it runs on OpenShift or vanilla Kubernetes. On OpenShift, it creates Routes with TLS reencrypt/passthrough termination and injects an OAuth proxy sidecar for authentication. On Kubernetes, it creates nginx Ingress resources without authentication. This platform detection happens at startup via API group discovery (`isOpenShift` function, `main.go:403-418`) and affects both the controller reconciliation path and the controller-manager cache configuration (only caching Routes on OpenShift).

The operator's security model for RayCluster access is notable: rather than relying on platform-level authentication (like RHOAI 3.x HTTPRoute + kube-rbac-proxy pattern), it uses the older OAuth proxy sidecar pattern (RHOAI 2.x). The OAuth proxy on port 8443 fronts the Ray Dashboard on port 8265, and authentication is delegated to OpenShift OAuth with a SubjectAccessReview check (`pods:get` in the RayCluster's namespace). This means any user who can read pods in the namespace can access the Ray Dashboard. The webhook enforces immutability of the OAuth proxy container, volume, and service account to prevent users from bypassing this security layer.

The mTLS implementation generates a self-signed CA certificate per RayCluster (stored in a Kubernetes Secret) and uses init containers that invoke `openssl` to generate per-node certificates at pod startup. The head node's certificate includes SANs for the Ray client Route hostname and the service DNS name, enabling external mTLS client access. Worker certificates only include loopback and pod IP SANs. This certificate management is entirely self-contained — no dependency on cert-manager or external PKI. The CA certificates have a 1-year validity period and are not auto-rotated, which is acceptable since RayClusters are typically short-lived.

The NetworkPolicy implementation creates two policies per RayCluster: one for workers (only accepts traffic from pods in the same cluster) and one for the head node (accepts traffic from cluster pods, from any pod on ports 10001 and 8265, from the KubeRay operator namespace, from openshift-monitoring on port 8080, and from any source on the secured ports 8443 and optionally 10001). The KubeRay operator namespace is discovered dynamically by querying for pods with the `app.kubernetes.io/component=kuberay-operator` label, falling back to the well-known namespaces (`opendatahub` or `redhat-ods-applications`).

The embedded AppWrapper controller is a significant architectural feature — rather than deploying AppWrapper as a separate operator, the codeflare-operator embeds it directly. This is controlled via the `codeflare-operator-config` ConfigMap (loaded at startup from `main.go:104,336-353`). When enabled, the AppWrapper controller manages wrapped resources (Deployments, StatefulSets, Jobs, RayClusters, RayJobs, PyTorchJobs, JobSets, PodGroups) and integrates with Kueue for quota-aware admission. The broad RBAC permissions in `config/rbac/role.yaml` reflect this — many API groups (kubeflow.org, jobset.x-k8s.io, scheduling.sigs.k8s.io) are needed only for the AppWrapper controller's wrapped resource management.
