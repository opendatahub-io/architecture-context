## Purpose

**Short**: OpenVINO is Intel's open-source toolkit for optimizing and deploying deep learning inference across Intel hardware (CPU, GPU, NPU), providing the core inference runtime used by OpenVINO Model Server (OVMS) in the RHOAI platform.

**Detailed**: OpenVINO (Open Visual Inference and Optimization) is a C++ inference framework that provides model conversion, optimization, and execution capabilities for deep learning models. It supports multiple model formats (ONNX, TensorFlow, PyTorch, PaddlePaddle, TensorFlow Lite, and its own IR format) through a frontend architecture, and targets Intel CPUs (via oneDNN/AVX/AMX), Intel GPUs (via OpenCL), and other accelerators through a plugin system.

Within the RHOAI platform, OpenVINO is not deployed as a standalone Kubernetes service. Instead, it is a build-time dependency of the OpenVINO Model Server (OVMS), which links against OpenVINO's C++ shared libraries to perform model inference. The `red-hat-data-services/openvino` fork exists to maintain a supply-chain-controlled copy of the upstream `openvinotoolkit/openvino` repository for RHOAI image builds.

This repository produces shared libraries (`libopenvino.so`, plugin `.so` files) and Python wheels that are consumed during the OVMS container image build. It does not define CRDs, deploy Kubernetes resources, or expose network services directly.

## Architectural Analysis

OpenVINO's architecture follows a classic plugin-based framework pattern with three major layers: **frontends** (model format readers), **core** (graph representation and transformations), and **plugins** (hardware-specific execution engines). This separation allows the framework to support new model formats and hardware targets independently.

The **frontend architecture** is notable for its breadth — six distinct frontends (ONNX, TensorFlow, TensorFlow Lite, PyTorch, PaddlePaddle, IR) each parse a different serialization format and convert to a unified `ov::Model` graph representation. The TensorFlow frontend alone carries 37 protobuf definitions. This creates a large dependency surface (protobuf 3.21, flatbuffers, ONNX) but enables OVMS to serve models from any major training framework without format conversion.

The **plugin system** is the key extensibility mechanism. Each hardware backend (CPU, GPU, Auto, Hetero, AutoBatch) is a dynamically loaded shared library that implements the `InferenceEngine::IInferencePlugin` interface. The CPU plugin is the most complex, pulling in oneDNN (oneAPI Deep Neural Network Library) and xbyak (JIT assembler) for kernel code generation targeting specific instruction sets (AVX2, AVX-512, AMX for Intel AMX accelerators, and ARM Compute Library for AArch64). The GPU plugin uses OpenCL and a separate oneDNN fork for GPU-optimized primitives.

From a supply chain perspective, this repository is unusual in the RHOAI stack because it is a **native C++ toolkit, not a Kubernetes component**. It has no Dockerfiles (Konflux or otherwise), no CRDs, no RBAC, and no network exposure. Its value to RHOAI is entirely as a build dependency — the shared libraries produced here are copied into the OVMS container image during the Konflux build pipeline. The `red-hat-data-services` fork provides a controlled copy of the upstream `openvinotoolkit/openvino` repository, allowing Red Hat to pin specific commits and apply downstream patches for RHOAI releases. The 14 git submodules (oneDNN, protobuf, OpenCL headers, etc.) and the Conan/vcpkg lock files provide the primary dependency pinning mechanisms. The absence of `rpms.lock.yaml` and Konflux Dockerfiles on this `master` branch is expected — this branch tracks upstream, and RHOAI release branches likely add the Konflux build infrastructure.

The codebase is massive: ~1,413 public header files and ~2,819 C++ source files in `src/` alone (excluding tests), plus ~1,679 Python files across bindings and tools. The operation set headers (`src/core/include/openvino/op/`) alone total ~13,700 lines, defining the complete set of neural network operations that OpenVINO can represent and execute.
