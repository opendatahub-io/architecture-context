## Purpose

**Short**: Kubeflow Pipelines backend for defining, scheduling, and executing ML workflows as Argo Workflows on Kubernetes.

**Detailed**: Data Science Pipelines (DSP) is the Red Hat downstream build of Kubeflow Pipelines (KFP), providing the backend infrastructure for ML pipeline orchestration in Red Hat OpenShift AI. The component comprises five independently deployable Go services that collectively manage the full pipeline lifecycle — from definition and scheduling through execution and result caching.

The system follows a hub-and-spoke architecture: the API server is the central service exposing gRPC and REST APIs for pipeline CRUD operations, run management, and experiment tracking. It compiles pipeline specs (IR YAML produced by the KFP Python SDK) into Argo Workflow CRs and submits them to Kubernetes. The persistence agent watches Argo Workflow execution status and reports it back to the API server for database persistence. The scheduled workflow controller manages recurring pipeline runs via the ScheduledWorkflow CRD.

Pipeline execution uses a driver/launcher pattern: the driver runs as an Argo step before each task container, resolving inputs, checking the execution cache, and generating Kubernetes pod spec patches. The launcher then wraps the user container, downloading artifacts, executing user code, and publishing outputs to ML Metadata (MLMD) and object storage. This separation enables features like execution caching, workspace PVCs for inter-task data sharing, and dynamic Kubernetes configuration (node affinity, tolerations, volumes) without recompilation.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| API Server | Go Service | Central REST/gRPC API for pipeline management, run orchestration, and experiment tracking |
| Driver | Go CLI (Argo step) | Pre-execution orchestration: input resolution, cache checking, pod spec patching |
| Launcher | Go CLI (container wrapper) | In-container execution wrapper: artifact download, user code execution, output publishing |
| Persistence Agent | Go Service | Watches Argo Workflow status and persists run state to the API server database |
| Scheduled Workflow Controller | Go Controller | Manages ScheduledWorkflow CRDs to create Argo Workflows on cron/periodic schedules |

## Architectural Analysis

The Data Science Pipelines architecture follows a clear separation between control plane (API server, scheduled workflow controller, persistence agent) and data plane (driver, launcher) components. This separation is architecturally significant: the control plane components are long-running services deployed once per namespace, while the data plane components are ephemeral CLIs injected into every pipeline task pod. This design means the driver and launcher binaries must be lightweight, stateless, and self-contained — they carry no persistent state and communicate exclusively via MLMD and the API server's gRPC interface.

The driver/launcher split mirrors a sidecar pattern but operates as sequential Argo Workflow steps rather than concurrent containers. The driver runs as an init step, generates a Kubernetes strategic merge patch for the pod spec, and writes it to a file. Argo then applies this patch when creating the main task container. This indirection enables dynamic pod configuration (node affinity, tolerations, volumes, security contexts) without requiring recompilation of the pipeline spec. The launcher then wraps the user's container command, handling artifact download/upload and MLMD metadata publishing. This two-phase approach is elegant but creates a dependency chain: if the driver fails to write the pod spec patch, the launcher will not have the correct executor input.

The caching system operates at two layers. For KFP v2 pipelines, the driver performs cache lookups before task scheduling — if a cache hit is found, the task container never launches. For legacy KFP v1 pipelines, a Kubernetes ValidatingAdmissionWebhook intercepts pod creation and injects cached outputs via JSON patch. This dual-layer design reflects the evolution from v1 to v2 while maintaining backward compatibility. The cache key fingerprint includes input parameters, output definitions, container image, and command arguments, providing deterministic cache hits.

The API server supports two pipeline storage modes: database (default, using MySQL/PostgreSQL) and Kubernetes-native (using Pipeline and PipelineVersion CRDs). The Kubernetes-native mode is a newer pattern that stores pipeline specs as Kubernetes custom resources, enabling Kubernetes-native workflows and GitOps patterns. When this mode is enabled, the API server registers admission webhooks for PipelineVersion validation and mutation. The webhook validates pipeline spec YAML/JSON parsing, enforces naming conventions (alphanumeric first character, then alphanumeric/dot/underscore/hyphen), and checks immutability of spec fields on updates (only DisplayName and Tags are mutable).

A notable security finding: the test client utility at `backend/src/common/client/api_server/util.go` uses `InsecureSkipVerify: true` for TLS configuration. This code path is used by integration tests and the common API client library, not by the production server code. However, if any downstream consumer reuses this client factory in production, it would disable TLS certificate verification. The production API server and driver/launcher components use proper TLS configuration with configurable CA certificate paths.
