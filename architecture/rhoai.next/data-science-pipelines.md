## Purpose

**Short**: Kubeflow Pipelines backend providing ML pipeline definition, scheduling, execution, caching, and artifact management on Kubernetes via Argo Workflows.

**Detailed**: Data Science Pipelines (DSP) is the RHOAI distribution of Kubeflow Pipelines v2. It provides a complete ML pipeline orchestration platform consisting of five shippable container images: an API server that exposes gRPC and REST endpoints for pipeline CRUD operations, a v2 driver that orchestrates individual pipeline step execution, a v2 launcher that manages container lifecycle and artifact I/O within user pods, a persistence agent that synchronizes Argo Workflow state back to the API server database, and a scheduled workflow controller that triggers recurring pipeline runs on cron or periodic schedules.

The system compiles KFP v2 pipeline specifications (defined via the Python SDK) into Argo Workflow DAGs. At runtime, the driver runs as a sidecar step before each user container, resolving inputs, checking the execution cache, and generating Kubernetes pod spec patches. The launcher then runs inside the user container, executing user code, uploading artifacts to S3-compatible object storage, and recording execution lineage in ML Metadata (MLMD). The API server stores pipeline definitions in either a relational database (MySQL/PostgreSQL) or as Kubernetes CRDs, and manages the lifecycle of experiments, runs, recurring runs, and artifacts. A mutating admission webhook (cache server) intercepts pod creation to short-circuit cached pipeline steps, replacing them with a no-op container.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| API Server | Go Service | Central REST/gRPC API for pipeline management, run submission, artifact access, and webhook validation |
| Driver | Go CLI (sidecar) | Pre-execution orchestrator that resolves inputs, checks cache, and patches pod specs in Argo Workflow steps |
| Launcher | Go CLI (sidecar) | In-container executor that runs user code, manages artifact upload/download, and publishes execution results to MLMD |
| Persistence Agent | Go Service | Watches Argo Workflows and syncs execution status back to the API server database |
| Scheduled Workflow Controller | Go Controller | Reconciles ScheduledWorkflow CRDs to trigger pipeline runs on cron or periodic schedules |
| Cache Server | Go Webhook | Mutating admission webhook that intercepts pod creation to reuse cached pipeline step outputs |
| Viewer Controller | Go Controller | Reconciles Viewer CRDs to create Tensorboard Deployments and Services |

## Architectural Analysis

Data Science Pipelines follows a **driver/launcher sidecar pattern** within Argo Workflow steps. Rather than the user container directly interacting with platform services, every pipeline step is wrapped with two injected binaries. The driver runs first as a separate Argo template step, resolving inputs from the MLMD execution graph, checking the cache, and generating a JSON pod spec patch. The launcher is then injected into the user container via an init container that copies the binary, wrapping the user's command to handle artifact I/O and execution recording. This pattern cleanly separates pipeline orchestration concerns from user code, but creates a deep dependency chain: Argo Workflow → driver template → pod spec patch → launcher wrapper → user code → launcher artifact upload → MLMD recording.

The **caching architecture spans two layers**. For KFP v1 pipelines, the cache server operates as a Kubernetes mutating admission webhook that intercepts pod creation, computes a SHA256 fingerprint of the step template, and replaces the entire container spec with a busybox no-op if a cache hit is found. For KFP v2 pipelines, caching is handled entirely within the driver — the v2 driver checks cache before generating the pod spec patch and sets the cached decision output, bypassing the webhook entirely. The webhook explicitly skips pods with the `pipelines.kubeflow.org/v2_component=true` annotation. This dual-path design means cache invalidation and staleness configuration must be considered at both layers.

The **launcher ships both FIPS and non-FIPS binaries** in the same container image (`launcher-v2` statically linked without FIPS, `launcher-v2-fips` dynamically linked with `strictfipsruntime`). The Dockerfile entrypoint defaults to the non-FIPS binary. The selection mechanism for FIPS mode at runtime (presumably controlled by the DSPO or the API server's `COMPILED_PIPELINE_SPEC_PATCH` environment variable) is not visible in this repository — it lives in the operator that deploys these images. This is a potential FIPS compliance gap: if the operator does not correctly switch to the FIPS binary, pipeline steps will run with non-FIPS crypto.

The **API server supports dual storage backends** for pipeline definitions: a traditional relational database (MySQL/PostgreSQL) and Kubernetes CRDs (`Pipeline` and `PipelineVersion` in `pipelines.kubeflow.org/v2beta1`). When CRD storage is enabled, the server registers validating and mutating webhooks that enforce spec immutability, auto-label PipelineVersions with their parent Pipeline UID, and set OwnerReferences for cascading deletion. The two modes are mutually exclusive per deployment but use the same gRPC/REST API surface, with the health endpoint reporting which store is active.

The **persistence agent** and **scheduled workflow controller** both use projected service account tokens with a 1-hour expiry and `pipelines.kubeflow.org` audience for authenticating to the API server's gRPC endpoint. This is a stronger security posture than static tokens, as the tokens are automatically rotated by the Kubernetes kubelet and can be validated by the API server via TokenReview. Both controllers use rate-limited work queues with exponential backoff, and the scheduled workflow controller re-queues after 10 seconds on successful submission to catch rapid-fire schedules.
