## Purpose

**Short**: Kubeflow Pipelines backend providing API, execution engine, and scheduling for ML pipeline workflows on Red Hat OpenShift AI.

**Detailed**: Data Science Pipelines is the RHOAI downstream fork of Kubeflow Pipelines, providing a complete ML pipeline orchestration platform. The system enables users to define, compile, submit, schedule, and monitor multi-step machine learning workflows that execute as Argo Workflows on Kubernetes.

The component comprises five deployable sub-components: (1) an API server exposing REST and gRPC endpoints for pipeline CRUD, experiment management, run execution, and artifact retrieval; (2) a v2 driver that orchestrates individual pipeline steps within Argo Workflow DAGs; (3) a v2 launcher that executes user containers, handles artifact I/O, and publishes metadata; (4) a persistence agent that watches Argo Workflow executions and syncs their state back to the API server; and (5) a scheduled workflow controller that manages the ScheduledWorkflow CRD to create recurring pipeline runs on cron or periodic schedules.

The platform integrates with relational databases (MySQL/PostgreSQL) for pipeline metadata, object storage (S3/MinIO/GCS) for pipeline artifacts and specs, ML Metadata (MLMD) for lineage tracking, and optionally with MLflow for experiment tracking. In RHOAI, the Data Science Pipelines Operator (DSPO) manages deployment of these components per-namespace via the DataSciencePipelinesApplication CRD.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| API Server (`ml-pipeline`) | Go Service (REST + gRPC) | Central control plane exposing pipeline, experiment, run, and artifact APIs over HTTP (8888/TCP) and gRPC (8887/TCP), plus webhook validation (8443/TCP) |
| Driver (`driver`) | Go CLI (Argo step) | Orchestrates individual pipeline steps within Argo Workflows — resolves inputs, manages caching, creates MLMD executions, patches pod specs |
| Launcher (`launcher-v2`) | Go CLI (Argo step) | Executes user containers, downloads/uploads artifacts from object storage, publishes execution metadata to MLMD |
| Persistence Agent (`persistence_agent`) | Go Agent (informer-based) | Watches Argo Workflow and ScheduledWorkflow resources via Kubernetes informers, syncs execution state to API server via gRPC |
| Scheduled Workflow Controller | Go Controller (informer-based) | Manages ScheduledWorkflow CRD lifecycle — evaluates cron/periodic schedules and creates Argo Workflows or API Runs |
| Viewer Controller | Go Controller (controller-runtime) | Manages Viewer CRD — creates TensorBoard Deployment and Service resources for pipeline visualization |
| Cache Server | Go Webhook Server | Mutating admission webhook that intercepts Argo Workflow pods and injects cached outputs when cache hits are found |
| Pipeline Compiler | Go CLI | Compiles KFP PipelineJob specs into Argo Workflow YAML (used at build time, not deployed) |

## Architectural Analysis

Data Science Pipelines follows a **hub-and-spoke architecture** centered on the API server. The API server is the single source of truth for pipeline metadata, with all other components either feeding data into it (persistence agent, scheduled workflow controller) or reading from it (driver, launcher, frontend). This centralized design simplifies consistency guarantees but makes the API server a single point of failure — all components have init containers that block startup until the API server is reachable.

The **dual-binary launcher** pattern (`launcher-v2` + `launcher-v2-fips`) in a single image is an interesting FIPS compliance strategy. The non-FIPS binary is statically linked (`CGO_ENABLED=0`) for maximum portability and minimal attack surface, while the FIPS binary is dynamically linked against OpenSSL. The API server selects which binary to inject into compiled workflow specs via the `V2_LAUNCHER_IMAGE` environment variable and entrypoint override. This allows FIPS-mandated clusters to use the dynamically linked binary while non-FIPS clusters benefit from the simpler static binary. However, shipping both binaries in the same image means the non-FIPS binary will be flagged by check-payload validation — this is a known trade-off.

The **v1/v2 pipeline coexistence** is managed through feature flags (`BLOCK_V1_PIPELINES`, `V1_ALLOWED_NAMESPACES`) that allow gradual migration. V1 pipelines embed Argo Workflow specs directly in the ScheduledWorkflow CRD, while v2 pipelines use the API server's RunService to create runs, which then compile to Argo Workflows server-side. The scheduled workflow controller handles both paths — checking whether the ScheduledWorkflow contains a v2 pipeline reference (via labels) and routing accordingly.

The **authentication chain** uses Kubernetes-native mechanisms: projected ServiceAccount tokens with the `pipelines.kubeflow.org` audience are mounted into persistence agent and scheduled workflow controller pods, then validated by the API server via TokenReview. For end-user requests, the API server supports two authenticators in order: HTTP header extraction (for identity-aware proxies like Istio) and TokenReview (for bearer tokens). Authorization is enforced via SubjectAccessReview against the `pipelines.kubeflow.org` API group, leveraging Kubernetes RBAC for namespace-scoped access control. In single-user mode, all authorization is bypassed.

The **plugin system** (currently only MLflow) hooks into the run lifecycle via the PluginDispatcher interface. The MLflow plugin resolves per-namespace configuration from the `kfp-launcher` ConfigMap (merged with global config), creates MLflow experiments and parent runs, and injects tracking environment variables into user containers. The 3-second timeout for MLflow API calls ensures that plugin failures don't block pipeline execution. This plugin architecture is extensible but currently tightly coupled to the API server — driver/launcher plugins use a separate registry pattern.
