## Purpose

**Short**: Data Science Pipelines (DSP) is the Kubeflow Pipelines v2 backend that provides ML pipeline definition, compilation, execution, scheduling, caching, and metadata tracking for Red Hat OpenShift AI.

**Detailed**: Data Science Pipelines is a downstream distribution of Kubeflow Pipelines that enables data scientists to author, compile, and execute ML workflows as directed acyclic graphs (DAGs) on OpenShift. The platform consists of multiple Go microservices: an API server exposing gRPC and REST endpoints for pipeline/run/experiment management, a persistence agent that syncs Argo Workflow execution state to a relational database, a scheduled workflow controller that manages cron-based and periodic recurring runs, a cache server implementing a mutating admission webhook for step-level caching, and a V2 engine with driver/launcher binaries that orchestrate container execution within Argo Workflow steps.

The system stores pipeline definitions in either a SQL database (MySQL/PostgreSQL) or as Kubernetes CRDs, stores artifacts in S3-compatible object storage (typically SeaweedFS), and tracks ML metadata via a gRPC metadata service. The frontend is a Node.js BFF serving a React UI that proxies API requests to the backend. DSP is deployed by the RHOAI platform operator (via `DataSciencePipelinesApplication` CRD) as a per-namespace instance, supporting multi-tenant isolation in multi-user mode.

| Persistence Agent | Go Service | Watches Argo Workflows and ScheduledWorkflows, syncs execution status and metrics to the database via gRPC |
| Scheduled Workflow Controller | Go Controller | Manages ScheduledWorkflow CRDs, creates pipeline runs on cron/periodic schedules |
| Viewer CRD Controller | Go Controller (controller-runtime) | Watches Viewer CRDs and creates TensorBoard Deployment + Service instances |

| Cache Deployer | Go Service | Manages webhook certificates and MutatingWebhookConfiguration lifecycle for cache server |
| V2 Driver | Go CLI | Argo Workflow sidecar step that orchestrates pipeline execution — resolves inputs, manages metadata, computes cache keys, generates pod spec patches |
| V2 Launcher | Go CLI | Runs inside user containers to execute pipeline components, download/upload artifacts, and publish results to ML Metadata |
| Metadata Writer | Go/Python Service | Watches completed Argo Workflow pods and propagates metadata annotations |
| Frontend (UI) | Node.js/TypeScript BFF + React | Express-based backend-for-frontend serving React UI, proxying API requests, and handling artifact retrieval from object storage |
| Python SDK | Python Library | Client library for authoring, compiling, and submitting pipeline definitions (not a deployed service) |

## Architectural Analysis

Data Science Pipelines follows a classic microservices architecture with clear separation of concerns across its components. The API server is the central hub, providing both REST (via gRPC-Gateway transcoding) and native gRPC interfaces. This dual-protocol design enables the frontend to use REST while internal components (persistence agent, scheduled workflow controller) communicate more efficiently via gRPC. The API surface spans two generations — v1beta1 (legacy, being deprecated) and v2beta1 (current) — reflecting the upstream Kubeflow evolution from embedded Argo Workflow specs to the V2 pipeline IR format.

The V2 execution engine implements a sophisticated two-phase execution model. The driver runs as an Argo Workflow sidecar step before the user container, performing input resolution, metadata creation, and cache key computation. It generates a `PodSpecPatch` that Argo applies to customize the task pod (environment variables, volume mounts, resource requests, Modelcar OCI sidecars). The launcher then runs inside the user container to handle artifact I/O and publish results. This separation means the driver can decide to skip execution entirely on a cache hit, replacing the container spec with a lightweight busybox image. The cache system operates at two levels: a V2 fingerprint-based cache (SHA256 of inputs + container image + command) stored in MLMD, and a V1 pod-mutation cache (mutating admission webhook) stored in a SQL table.

A notable design decision is the support for dual pipeline storage backends. Pipelines can be stored in the SQL database (traditional approach) or as Kubernetes CRDs (Pipeline and PipelineVersion resources). The CRD-based approach, gated by the `-pipelinesStoreKubernetes` flag, adds validating and mutating webhooks to enforce spec immutability and maintain ownership references. This enables GitOps-style pipeline management where pipeline definitions are Kubernetes-native resources. The mutating webhook automatically sets Pipeline→PipelineVersion owner references and labels for efficient querying.

The scheduled workflow controller has a clean V1/V2 split. For V1 pipelines, it creates Argo Workflow CRs directly from the embedded spec. For V2 pipelines, it calls the API server's `CreateRun` gRPC endpoint, delegating workflow submission to the API server. This allows V2 runs to benefit from the full API server pipeline (validation, metadata recording, plugin hooks) while V1 runs take the legacy fast path. The `BLOCK_V1_PIPELINES` feature flag provides a migration mechanism to enforce V2-only usage.

Security-wise, the system uses projected service account tokens with explicit audience (`pipelines.kubeflow.org`) and 1-hour expiry for inter-service authentication. All Go binaries are built with `GOEXPERIMENT=strictfipsruntime` and `CGO_ENABLED=1` for FIPS compliance, with the go.mod explicitly setting `godebug fips140=on`. The launcher uniquely ships two binaries — a static non-FIPS build and a dynamic FIPS build — to support both FIPS-required (RHOAI) and non-FIPS (upstream) deployment contexts. All containers run as non-root (UID 1000) with `seccompProfile: RuntimeDefault` and dropped capabilities, following OpenShift security best practices. No NetworkPolicies are defined in the base manifests; network isolation is expected to be provided by the DSPA operator or platform-level configuration.
