## Purpose

**Short**: Data Science Pipelines provides an ML pipeline orchestration platform that compiles, schedules, executes, and tracks machine learning workflows on Kubernetes using Argo Workflows.

**Detailed**: Data Science Pipelines is the Red Hat downstream distribution of Kubeflow Pipelines (KFP) v2, a platform for building and deploying portable, scalable ML workflows. It provides a comprehensive REST/gRPC API server for managing pipelines, experiments, runs, and recurring schedules; a compiler that transforms Python-defined pipelines into Argo Workflow manifests; and a driver/launcher execution model that orchestrates individual pipeline steps with full ML Metadata (MLMD) lineage tracking.

The platform consists of 5 Konflux-built container images (API server, driver, launcher, persistence agent, scheduled workflow controller) plus upstream community images for the frontend UI, cache server, visualization server, metadata writer, and viewer controller. It supports multi-user isolation via Kubernetes RBAC and Istio service mesh, pluggable object storage backends (S3, MinIO, GCS), and both MySQL and PostgreSQL databases. Pipelines are defined as Python code using the KFP SDK, compiled to Argo Workflow specs, and executed with artifact caching, workspace persistence, and OCI modelcar support.

Within RHOAI, this component is deployed by the Data Science Pipelines Application (DSPA) operator (`data-science-pipelines-operator`), which creates and manages the individual service deployments per-namespace. The kustomize manifests in this repo define the upstream deployment topology that the DSPA operator templates from.

## Architectural Analysis

Data Science Pipelines implements a three-tier execution model that cleanly separates orchestration from execution. The **Driver** runs as an init container in each Argo Workflow task pod, resolving inputs from upstream tasks via MLMD queries, checking the execution cache, and generating a strategic merge patch for the pod spec. The **Launcher** then runs alongside the user's container, handling artifact download/upload and publishing results back to MLMD. This separation allows the platform to manage execution state independently from user code, enabling features like cache-based step skipping, workspace persistence, and OCI modelcar support without requiring changes to user containers.

The caching architecture is particularly noteworthy: it operates at two levels. The **cache mutation webhook** intercepts pod creation (for v1 workflows) and replaces the entire container with a lightweight busybox image when a cache hit is found, injecting the cached output via pod annotations. This approach is transparent to Argo Workflows — the pod still completes, but does minimal work. For v2 workflows, caching is handled by the Driver at the orchestration level before the user container is even created. The cache key is a SHA256 hash of the Argo template skeleton (image, command, args, env, volumeMounts, inputs, outputs), and staleness is configurable per-pod and globally.

The API server serves dual protocols (gRPC on 8887 and HTTP REST on 8888 via grpc-gateway) with both v1beta1 and v2beta1 API versions. The v2beta1 API is the primary interface, with v1beta1 maintained for backward compatibility. The server also hosts an optional Kubernetes webhook server on port 8443 for PipelineVersion CRD validation and mutation. The PipelineVersion webhook enforces spec immutability (only `displayName` and `tags` are mutable after creation) and automatically sets up ownerReferences and labels linking PipelineVersions to their parent Pipeline CRDs.

The RBAC model uses principle of least privilege with 10 namespace-scoped Roles and 4 ClusterRoles. The `pipeline-runner` Role is notably broad (wildcards on pods, services, deployments, kubeflow.org resources, and even Seldon deployments), because it represents the service account used by actual pipeline step pods that may need to create arbitrary Kubernetes resources. Multi-user mode adds aggregation ClusterRoles (`kubeflow-pipelines-edit` and `kubeflow-pipelines-view`) that are composed from fine-grained sub-roles, following the Kubernetes RBAC aggregation pattern.

All container images are built with FIPS-compliant cryptography (`GOEXPERIMENT=strictfipsruntime`) on UBI9 base images, using Go 1.25 with CGO enabled. The launcher uniquely builds both standard and FIPS binaries in the same Dockerfile, allowing runtime selection based on cluster requirements. Pod security follows restricted standards: `runAsNonRoot: true`, `runAsUser: 1000`, `drop ALL` capabilities, and `seccompProfile: RuntimeDefault` across all deployments. The OpenShift overlay removes explicit `runAsUser` values, deferring to OpenShift's SCC mechanism.
