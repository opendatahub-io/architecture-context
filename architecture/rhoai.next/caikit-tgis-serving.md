## Purpose

**Short**: Caikit-TGIS-Serving is a container image that provides Large Language Model (LLM) inference by combining the Caikit AI toolkit runtime with a TGIS (Text Generation Inference Server) backend.

**Detailed**: Caikit-TGIS-Serving packages the Caikit runtime with the caikit-nlp and caikit-tgis-backend Python libraries into a single container image. This image serves as the "transformer-container" in a KServe multi-container ServingRuntime, where it runs alongside a separate TGIS container that hosts the actual model weights and performs GPU-accelerated inference. The Caikit runtime translates incoming HTTP (port 8080) or gRPC (port 8085) requests into calls to the co-located TGIS backend over a local gRPC connection (localhost:8033).

This repository does not contain an operator or controller — it is a **container image definition** (Dockerfile + configuration + dependency management). The image is consumed by KServe ServingRuntime custom resources that define multi-container inference pods. The runtime supports the Caikit model format and provides a standardized API surface for text generation tasks. Models are loaded from `/mnt/models/` (typically backed by a PVC or S3-compatible object storage), and the Caikit runtime lazily loads local models and delegates inference to the TGIS backend. The repository also includes demo manifests, OpenShift Templates, and smoke tests for validating the deployment on KServe with Knative and Istio service mesh.

## Architectural Analysis

This repository is architecturally minimal — it is a **container image definition**, not a service or operator. The repository contains no application source code of its own; instead, it assembles three upstream Python packages (`caikit`, `caikit-nlp`, `caikit-tgis-backend`) into a container image with a configuration file (`caikit.yml`) and a model conversion utility (`utils/convert.py`). The actual inference logic, API surface, health probes, and model management are all provided by the upstream `caikit` runtime (invoked as `python -m caikit.runtime`). This design means that any API, performance, or security changes to the serving runtime are driven by dependency version bumps in `pyproject.toml`, not by code changes in this repository.

The multi-container pod architecture is a deliberate design choice that separates concerns: the Caikit "transformer-container" handles API translation, model format management, and request routing, while the TGIS "kserve-container" handles GPU-accelerated model inference. Communication between them happens over a localhost gRPC connection (port 8033), avoiding network overhead. This pattern allows each container to be independently versioned, scaled, and updated. The repository also provides demo manifests for a "caikit-standalone" mode where Caikit runs without TGIS for simpler NLP tasks (using the `caikit-nlp` image directly).

The Konflux Dockerfile (`Dockerfile.konflux`) is nearly identical to the upstream `Dockerfile`, with both using a two-stage build (poetry-builder + deploy) on UBI 9 minimal. Notably, the Konflux builds are **not hermetic** (`hermetic: false` in the Tekton PipelineRun) and lack cachi2/Hermeto prefetch integration. The `poetry.lock` file provides Python dependency pinning, but OS-level packages are installed without version pinning. The Tekton pipeline builds multi-architecture images (linux/x86_64 and linux-m2xlarge/arm64) and pushes to `quay.io/rhoai/`. Security is delegated almost entirely to the infrastructure layer — Istio provides mTLS, KServe manages routing and access, and the UBI 9 base image provides FIPS-validated system libraries.

The demo manifests in `demo/kserve/` are extensive and document the full stack deployment including Istio ServiceMeshControlPlane, Knative Serving, Istio Gateways, PeerAuthentication, NetworkPolicies, and ServiceMonitors. While these are not deployed by this component directly, they serve as the reference architecture for how caikit-tgis-serving operates within the RHOAI platform. The `mTLS: STRICT` PeerAuthentication with `PERMISSIVE` on port 8086 is a notable pattern — it allows Prometheus to scrape metrics without requiring mTLS certificates while keeping inference traffic encrypted.
