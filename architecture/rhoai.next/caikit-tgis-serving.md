## Purpose

**Short**: Caikit-TGIS-Serving is a container image that provides a Caikit AI runtime for Large Language Model (LLM) inference, connecting the Caikit API layer with a TGIS (Text Generation Inference Server) backend.

**Detailed**: Caikit-TGIS-Serving packages a Python runtime environment that bridges the Caikit AI toolkit with the Text Generation Inference Server (TGIS). The container runs `caikit.runtime` as its entry point, which exposes both HTTP (port 8080) and gRPC (port 8085) inference APIs to clients. Caikit provides a developer-friendly API abstraction layer, while TGIS handles the actual model loading and GPU-accelerated inference execution.

This component is deployed as the `transformer-container` within a KServe `ServingRuntime` custom resource, alongside a TGIS `kserve-container`. The two containers form a multi-container pod orchestrated by KServe: TGIS loads and serves the model on port 8033 (gRPC), while Caikit connects to TGIS as its inference backend and exposes the unified Caikit API to external consumers. The serving stack integrates with Knative Serving for autoscaling, Istio service mesh for mTLS and traffic management, and KServe for model lifecycle management (storage access, deployment orchestration, and networking setup).

The repository itself contains no application source code — it is a packaging repository that assembles three upstream Python libraries (`caikit`, `caikit-nlp`, `caikit-tgis-backend`) via Poetry dependency management, adds a Caikit YAML configuration file, and produces a container image. The repository also provides extensive demo manifests, installation scripts, and deployment templates for the full KServe/Caikit/TGIS stack on OpenShift.

## Architectural Analysis

Caikit-TGIS-Serving is architecturally a **packaging repository** rather than a source code repository. It contains zero lines of application logic — the entire runtime behavior is inherited from three upstream Python packages (`caikit`, `caikit-nlp`, `caikit-tgis-backend`) assembled via Poetry. The `caikit.yml` configuration file is the sole behavioral artifact, configuring the runtime to use `caikit_nlp` as its library and to discover models via a TGIS-AUTO finder with a localhost gRPC connection to port 8033. This means all API surface, protocol handling, model management, and inference logic are defined upstream and merely packaged here.

The **multi-container pod architecture** is a defining pattern of this stack. The Caikit container acts as a transformer/frontend that receives client requests and delegates inference to the co-located TGIS container via localhost gRPC. This separation of concerns allows the API layer (Caikit) and the inference engine (TGIS) to be versioned, scaled, and updated independently within the same pod. The `ServingRuntime` CR defines both containers: `kserve-container` runs TGIS with `text-generation-launcher`, and `transformer-container` runs `python -m caikit.runtime`. KServe manages the pod lifecycle including storage initialization, health probing, and networking.

The repository provides **two operational modes** configurable via environment variables: HTTP mode (port 8080, `RUNTIME_HTTP_ENABLED=true`) and gRPC mode (port 8085 with h2c, `RUNTIME_GRPC_ENABLED=true`). These are mutually exclusive in the demo manifests, with separate ServingRuntime definitions for each. Health probes use the `caikit_health_probe` Python module for both readiness and liveness checks, executed as subprocess commands rather than HTTP endpoints — an important detail for environments where the HTTP server may not be enabled.

The **security posture** is entirely delegated to infrastructure components. The container itself runs as a non-root user (UID 1001, `caikit`), but provides no TLS termination, authentication, or authorization. These are handled by the Istio service mesh (mTLS via PeerAuthentication with STRICT mode), Knative Serving (traffic routing and passthrough TLS), and KServe (model access control). Metrics collection uses a PeerAuthentication port-level override on port 8086 to allow PERMISSIVE mTLS, enabling plaintext Prometheus scrapes from the `openshift-user-workload-monitoring` namespace, constrained by a NetworkPolicy. This layered security approach means a vulnerability assessment of the inference path must span multiple components.

The **build pipeline** uses Konflux with a Tekton PipelineRun that references `konflux-central` for the multi-arch container build pipeline. Builds target both `linux/x86_64` and `linux-m2xlarge/arm64` platforms. The `hermetic: false` parameter in the PR pipeline indicates that pull request builds are not hermetic — full hermeticity is likely enforced only on release branches. The Dockerfile uses a two-stage build: a Poetry builder stage on UBI 9 minimal installs dependencies into a virtualenv, and the deploy stage copies only the virtualenv and config, producing a minimal runtime image.
