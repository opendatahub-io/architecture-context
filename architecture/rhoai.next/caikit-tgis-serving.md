## Purpose

**Short**: Caikit-TGIS-Serving is a container image that provides a Caikit AI runtime for Large Language Model (LLM) inference, acting as a transformer layer between client requests and the TGIS (Text Generation Inference Server) backend.

**Detailed**: Caikit-TGIS-Serving packages the Caikit AI toolkit runtime with the caikit-nlp and caikit-tgis-backend Python libraries into a container image that is deployed as part of a KServe ServingRuntime. The image runs `python -m caikit.runtime`, which starts the Caikit runtime server that exposes both HTTP (port 8080) and gRPC (port 8085) endpoints for LLM inference.

The component operates as the "transformer container" in a multi-container KServe ServingRuntime pod. It sits alongside a TGIS container (the inference engine that loads and runs the actual model). Caikit handles model management, API abstraction (providing the Caikit NLP API surface), and delegates the actual model inference to TGIS over a local gRPC connection on port 8033. Models are loaded from `/mnt/models/` which is populated by KServe's storage initializer from S3, PVC, or other storage backends.

This repository itself contains no application source code — it is a packaging/integration repository. The actual runtime logic comes entirely from the `caikit`, `caikit-nlp`, and `caikit-tgis-backend` Python packages declared in `pyproject.toml` and locked in `poetry.lock`. The repository provides the Dockerfile, configuration file (`caikit.yml`), demo/example manifests for deploying on KServe with OpenShift AI, and a utility script for converting HuggingFace models to Caikit format.

## Architectural Analysis

Caikit-TGIS-Serving is a "packaging repository" rather than a source code repository. It contains no application logic — its entire value proposition is assembling three upstream Python packages (caikit, caikit-nlp, caikit-tgis-backend) with a configuration file and Dockerfile into a deployable container image. This architectural choice means that the component's behavior, API surface, security properties, and performance characteristics are entirely inherited from its upstream dependencies. Any security audit or architecture review must trace through to those upstream packages for meaningful analysis.

The two-container ServingRuntime pattern (Caikit transformer + TGIS inference engine) is a separation-of-concerns design. Caikit handles the API layer, model management, and task abstraction, while TGIS handles the actual GPU-accelerated model inference. They communicate over a localhost gRPC connection on port 8033, which avoids network overhead but means they must be co-scheduled on the same node. The `caikit.yml` configuration file hardcodes `hostname: localhost:8033` for the TGIS connection with `test_connections: true`, meaning Caikit will fail its health probes if TGIS is not ready — providing a natural readiness dependency chain.

The build pipeline is notably non-hermetic. The Dockerfile installs Poetry and all Python dependencies directly from the network (`pip3.11 install poetry && poetry install`), and the Tekton pipeline explicitly sets `hermetic: false`. While `poetry.lock` provides version pinning, the lack of Hermeto prefetch means builds are not reproducible in air-gapped environments and are vulnerable to upstream package repository compromise. This is a supply chain risk that should be addressed for production release branches. Additionally, the base image (`ubi9/ubi-minimal:latest`) is unpinned — using a floating `latest` tag rather than a digest-pinned reference, which can cause build non-determinism.

The component does not use AIPCC base images, which is a gap for a component that packages AI/ML inference libraries. Migration to AIPCC base images would provide access to the RHEL AI Python Package Index with securely-built wheels, Python 3.12 (the component currently uses 3.11), and a standardized container layout. However, migration would require updates to the upstream caikit packages to support Python 3.12, which may be a blocking dependency. The repository also uses Poetry for dependency management while AIPCC images ship with pip and uv pre-configured, so the build process would need adaptation.
