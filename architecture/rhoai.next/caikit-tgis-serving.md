## Purpose

**Short**: Container image that packages the Caikit AI runtime with TGIS backend support to enable Large Language Model (LLM) inference via KServe ServingRuntimes.

**Detailed**: Caikit-TGIS-Serving is a container image that bundles the Caikit AI toolkit runtime (`caikit`), the Caikit NLP module (`caikit-nlp`), and the Caikit TGIS backend connector (`caikit-tgis-backend`) into a single deployable image. The image runs `python -m caikit.runtime`, which starts the Caikit runtime server exposing both HTTP (port 8080) and gRPC (port 8085) interfaces for model inference.

In the RHOAI/ODH platform, this image is deployed as the `transformer-container` within a KServe `ServingRuntime` custom resource, running alongside a TGIS (Text Generation Inference Server) container that handles the actual model loading and GPU inference. The Caikit container acts as a front-end API layer: it receives inference requests from clients via HTTP or gRPC, translates them into the Caikit model management protocol, and forwards them to the co-located TGIS backend over gRPC on port 8033. This two-container architecture separates the API/model management layer (Caikit) from the inference engine (TGIS), allowing them to be versioned and scaled independently within the same pod.

The repository itself contains no application source code — it is a container image build repository. The primary artifacts are Dockerfiles, a `pyproject.toml` declaring Python dependencies (managed via Poetry), a `caikit.yml` runtime configuration file, and extensive demo/test manifests showing how to deploy the stack on KServe with OpenShift AI infrastructure (Service Mesh, Serverless, Authorino).

## Architectural Analysis

Caikit-TGIS-Serving is architecturally a "thin wrapper" container image — it contains no application source code in this repository. The Python dependencies (`caikit`, `caikit-nlp`, `caikit-tgis-backend`) are pulled from PyPI via Poetry, and the entire runtime logic lives in those upstream packages. The repository's value is in the Dockerfile build definition, the `caikit.yml` configuration file that wires the runtime to use TGIS as its inference backend, and the extensive demo manifests that document the full deployment stack.

The two-container pod architecture (Caikit + TGIS) is a deliberate separation of concerns. The TGIS container (`kserve-container`) handles GPU-intensive model loading and inference, while the Caikit container (`transformer-container`) provides the user-facing API layer with model management, format detection (via the `TGIS-AUTO` finder), and health probing (`caikit_health_probe` readiness/liveness probes). Communication between the two containers happens entirely over localhost gRPC (port 8033), avoiding network overhead and keeping the inference path latency-minimal. The `caikit.yml` configuration specifies `test_connections: true`, meaning the Caikit runtime validates TGIS connectivity at startup before accepting traffic.

The Konflux build configuration (`.tekton/caikit-tgis-serving-pull-request.yaml`) shows that the image is built for multi-architecture (`linux/x86_64` and `linux-m2xlarge/arm64`), with non-hermetic builds (`hermetic: false`). The Dockerfile uses a two-stage build: a `poetry-builder` stage that installs the full Poetry toolchain and resolves dependencies into a virtualenv, and a `deploy` stage that copies only the virtualenv into a minimal UBI9 image. The final image runs as a non-root user (UID 1001, group `caikit`), following container security best practices.

A notable gap is the absence of AIPCC Ecosystems base images. The image builds from `registry.access.redhat.com/ubi9/ubi-minimal:latest` and installs Python packages from public PyPI. Since caikit-tgis-serving delegates GPU operations to the TGIS container, the Caikit container itself likely only needs the CPU variant of the AIPCC base image. However, migration to AIPCC would provide access to the RHEL AI Python Package Index with Red Hat-built secure wheels, improving supply chain security. The lack of `rpms.lock.yaml` and Hermeto prefetch integration means the build is not fully hermetic — RPM packages are installed without version pinning.
