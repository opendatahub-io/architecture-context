## Purpose

**Short**: Caikit-TGIS-Serving is a container image that packages the Caikit AI runtime with the caikit-nlp and caikit-tgis-backend libraries, enabling Large Language Model (LLM) inference on Red Hat OpenShift AI via KServe ServingRuntimes.

**Detailed**: This repository produces a container image that bundles the Caikit AI framework (`caikit` 0.28.1), the Caikit NLP module (`caikit-nlp` 0.5.14), and the TGIS backend connector (`caikit-tgis-backend` 0.1.39) into a single deployable image. The image runs `python -m caikit.runtime` as its entrypoint, which starts the Caikit runtime server exposing both HTTP (port 8080) and gRPC (port 8085) inference endpoints.

In a KServe deployment, this image serves as the "transformer-container" in a multi-container ServingRuntime pod. The companion container runs the Text Generation Inference Server (TGIS) which loads the actual LLM model and provides the inference engine on port 8033. Caikit acts as the API layer, receiving client requests via HTTP or gRPC and forwarding inference calls to the co-located TGIS backend over localhost gRPC. The Caikit runtime also handles model management, health probing, and metrics export (port 8086).

The repository itself contains no operator code or CRD definitions. It is a packaging/configuration repository: a Dockerfile, a `caikit.yml` configuration file, dependency specifications in `pyproject.toml` with a `poetry.lock`, a utility script for model format conversion, and an extensive set of demo manifests showing how to deploy the stack with KServe, Istio service mesh, and Knative Serverless on OpenShift.

## Architectural Analysis

Caikit-TGIS-Serving is architecturally a thin packaging layer rather than a standalone service. The repository's primary artifact is a container image that bundles three upstream Python libraries (caikit, caikit-nlp, caikit-tgis-backend) with a configuration file. The image contains no custom application code — the entrypoint is simply `python -m caikit.runtime`, which bootstraps the Caikit framework's built-in HTTP and gRPC servers. This design means the runtime's API surface, model management logic, and health probing are entirely defined by the upstream caikit libraries, and the repository's role is version pinning and image packaging.

The deployment architecture follows a multi-container pod pattern via KServe ServingRuntime. The caikit-tgis-serving image runs as the "transformer-container" and handles the external-facing API, while a co-located TGIS container ("kserve-container") loads the actual LLM model into GPU memory and serves inference via gRPC on port 8033. The Caikit runtime connects to TGIS over localhost, making this a sidecar-like pattern within a single Kubernetes pod. This separation allows independent scaling and versioning of the API layer (Caikit) and the inference engine (TGIS). The repository also provides a "caikit-standalone" variant that runs without TGIS, using HuggingFace Transformers directly for smaller models.

The networking stack depends heavily on the Istio service mesh and Knative Serverless. All inter-service traffic is encrypted via Istio mTLS (PeerAuthentication set to STRICT), with an explicit exception for the metrics port 8086 (set to PERMISSIVE) to allow Prometheus scraping without client certificates. A NetworkPolicy allows ingress from the `openshift-user-workload-monitoring` namespace to enable User Workload Monitoring. The demo manifests include a complete Istio Gateway and KnativeServing configuration, showing the full ingress chain from external HTTPS through Istio Gateway to Knative queue-proxy to the Caikit container. The `serving.knative.openshift.io/enablePassthrough: "true"` annotation enables TLS passthrough for gRPC streaming.

The Konflux build pipeline (`Dockerfile.konflux` and `.tekton/caikit-tgis-serving-pull-request.yaml`) produces multi-architecture images (linux/x86_64 and linux/arm64) and runs in the `rhoai-tenant` namespace. Notably, the Tekton PipelineRun references `Dockerfile` (not `Dockerfile.konflux`) as its build file, and both Dockerfiles are currently identical in content, suggesting the Konflux variant was added as a placeholder for future differentiation. The pipeline uses the centralized `konflux-central` repository for shared pipeline definitions.

The `caikit.yml` configuration reveals the model management architecture: Caikit uses a TGIS-AUTO model finder that automatically discovers models compatible with TGIS, and a LOCAL initializer that prioritizes the TGIS backend with connection testing enabled. Models are expected at `/mnt/models/` (mounted via PVC or storage initializer) and loaded lazily (`lazy_load_local_models: true`), which reduces startup time for pods that may serve multiple models.
