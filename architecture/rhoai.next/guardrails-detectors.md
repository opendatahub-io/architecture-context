## Purpose

**Short**: A collection of detector algorithm microservices for the FMS Guardrails Orchestrator that analyze text content for safety, PII, file-type validation, and LLM-as-a-judge evaluation.

**Detailed**: Guardrails Detectors provides three independent detector microservices that integrate with the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator). Each detector runs as a standalone FastAPI/uvicorn HTTP service exposing the `/api/v1/text/contents` endpoint (and additional endpoints for the LLM Judge detector). The detectors are designed to be deployed either as standalone containers or as KServe InferenceServices.

The three detector types are: (1) **Built-in detectors** — lightweight, heuristic-based detectors including regex-based PII detection (email, credit card, SSN, phone numbers, IP addresses), file-type validation (JSON, XML, YAML with optional schema validation), and user-definable custom Python detectors with static code analysis sandboxing; (2) **HuggingFace detector** — a generic ML model-based detector compatible with `AutoModelForSequenceClassification`, `AutoModelForTokenClassification`, and `GraniteForCausalLM` (Granite Guardian) models, providing content safety classification using PyTorch inference; (3) **LLM Judge detector** — an LLM-as-a-judge based detector using the `vllm_judge` library to evaluate content against configurable metrics (safety, toxicity, accuracy, helpfulness, etc.) by delegating to an external vLLM-compatible LLM server.

All detectors share a common framework (`detectors/common/`) providing the FastAPI application base class, Pydantic request/response schemas, Prometheus metrics instrumentation (detections, requests, errors, runtime counters), and structured logging. The project is maintained by the TrustyAI team and built for multi-architecture support (x86_64, ppc64le, arm64, s390x) via Konflux.

## Architectural Analysis

The Guardrails Detectors repository follows a clean shared-framework pattern where three independent detector implementations share a common FastAPI base (`DetectorBaseAPI`), Pydantic schema definitions, and Prometheus instrumentation code. The `DetectorBaseAPI` class provides standardized error handling, health endpoints, and a detector registry pattern that allows each detector type to register itself at startup and be looked up at request time. This design means all detectors expose a uniform API contract (`/api/v1/text/contents`) while internally implementing completely different detection strategies.

The built-in detector's custom detector subsystem is architecturally notable for its security sandboxing approach. It performs static code analysis (AST parsing) on user-provided Python files before loading them, blocking dangerous imports (`os`, `subprocess`, `sys`, `shutil`) and function calls (`eval`, `exec`, `open`, `compile`, `input`). It also injects utility decorators (`@use_instruments`, `@non_blocking`) into the user code's namespace, allowing custom detectors to register Prometheus metrics and run asynchronously without importing anything. This is a pragmatic approach to user-extensibility that trades perfect isolation for practical usability — a compromised custom detector could still perform arbitrary computation, but cannot directly access the filesystem or execute system commands.

The HuggingFace detector implements a multi-architecture model loading strategy that auto-detects the model type from the HuggingFace config (`ForSequenceClassification`, `ForTokenClassification`, `GraniteForCausalLM`) and routes to the appropriate processing pipeline. The token classification path is particularly well-designed: it uses offset mapping from the fast tokenizer to produce character-level span annotations (not just token-level), merging adjacent tokens with the same label into contiguous spans. This enables precise PII entity boundary detection when backed by a NER model.

The LLM Judge detector is the newest addition and differs architecturally from the other two in that it doesn't perform any local computation — it delegates entirely to an external vLLM-compatible server via the `vllm_judge` library. This makes it lightweight (no GPU requirements, minimal memory) but introduces an external runtime dependency. The detector exposes an additional `/api/v1/text/generation` endpoint for prompt-response pair evaluation, which the other detectors don't offer. Notably, the LLM Judge detector has no Konflux Dockerfile, meaning it is not currently built through the RHOAI production pipeline — it appears to be in upstream/incubation status.

Multi-architecture support is a significant engineering investment in this repo. The Konflux HuggingFace Dockerfile conditionally builds PyTorch from source for ppc64le and s390x architectures, including OpenBLAS compilation for ppc64le. The built-in detector is simpler (no PyTorch) and uses the same multi-stage pattern with fewer conditional build steps. Both Tekton PipelineRun configurations target four architectures: x86_64, ppc64le, arm64, and s390x.
