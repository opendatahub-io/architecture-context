## Purpose

**Short**: A collection of text detection microservices for the FMS Guardrails Orchestrator that provide PII detection, content validation, ML-based content classification, and LLM-as-a-Judge evaluation.

**Detailed**: The guardrails-detectors repository provides three independent detector microservices that serve as backends for the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator). Each detector is packaged as an independent container image and exposes a REST API conforming to the IBM Detector API specification.

The **built-in detector** bundles lightweight, heuristic detection functions (regex-based PII detection, file-type validation, and user-defined custom Python detectors) into a single FastAPI service. The **HuggingFace detector** loads a pre-trained model (supporting `AutoModelForSequenceClassification`, `AutoModelForTokenClassification`, and `GraniteForCausalLM` architectures) and runs inference on user-supplied text to produce content analysis results. The **LLM Judge detector** delegates evaluation to an external OpenAI-compatible LLM server via the [vllm_judge](https://github.com/trustyai-explainability/vllm_judge) library, supporting built-in metrics like safety, toxicity, accuracy, and custom natural-language criteria.

All three detectors share a common library (`detectors/common/`) providing the FastAPI base application class, Pydantic request/response schemas, Prometheus instrumentation, and structured logging. The HuggingFace and LLM Judge detectors are designed to be deployed as KServe InferenceServices with associated ServingRuntimes, while the built-in detector runs as a standalone deployment alongside the Guardrails Orchestrator.

## Architectural Analysis

The guardrails-detectors repository follows a clean multi-component microservice pattern with a shared common library. All three detectors extend the same `DetectorBaseAPI` FastAPI base class, which provides standardized error handling, health check endpoints, Prometheus metrics instrumentation, and a detector registry pattern. This common foundation ensures API consistency across all detector types while allowing each to implement its own detection logic independently.

The **instrumentation architecture** is notable. The `InstrumentedDetector` base class provides automatic Prometheus Counter metrics for four dimensions: `detections`, `requests`, `errors`, and `runtime`. These are labeled by `detector_kind` and `detector_name`, enabling per-detector-function observability. The custom detector wrapper extends this further by allowing user-defined Python code to register its own Prometheus instruments via the `@use_instruments` decorator, and to run detection logic non-blockingly via the `@non_blocking` decorator (which uses a `ThreadPoolExecutor` for background execution).

The **custom detector subsystem** (built-in detector) implements a security-conscious dynamic code loading pattern. Before loading user-provided Python code from `custom_detectors/custom_detectors.py`, the `CustomDetectorRegistry` runs a static code analysis (`static_code_analysis`) using Python's `ast` module to check for forbidden imports (`os`, `subprocess`, `sys`, `shutil`) and forbidden function calls (`eval`, `exec`, `open`, `compile`, `input`). An exception is carved out for `os.environ` and `os.getenv` access. Utility functions (`use_instruments`, `non_blocking`) are injected into the module namespace before execution, enabling custom detectors to use platform instrumentation without importing it directly.

The **HuggingFace detector** supports three distinct model architectures through a runtime-dispatching pattern: `AutoModelForSequenceClassification` (binary/multi-class text classification), `AutoModelForTokenClassification` (NER-style span detection with offset mapping), and `GraniteForCausalLM` (IBM Granite Guardian causal language models). The token classifier path is particularly sophisticated, performing per-token probability analysis, filtering by configurable safe labels, and grouping adjacent tokens with the same label into contiguous character spans. The model architecture is auto-detected from the HuggingFace config at initialization time.

The **Konflux build pipeline** supports multi-architecture builds (x86_64, ppc64le, arm64, s390x). The HuggingFace Konflux Dockerfile is notably complex for ppc64le and s390x: it builds PyTorch from source (v2.6.0), compiles OpenBLAS from source (v0.3.30 for ppc64le), and builds cmake from source. This reflects the lack of pre-built PyTorch wheels for these architectures. The Dockerfile also pins the OpenSSL version (`3.5.1-5.el9_7`) across all architectures for reproducibility. Only two Konflux Dockerfiles exist (built-in and HuggingFace); the LLM Judge has no Konflux variant yet, suggesting it may be earlier in its productization lifecycle.
