## Purpose

**Short**: A collection of detector microservices for the FMS Guardrails Orchestrator that analyze text content for PII, content safety, file type validation, and LLM-as-a-judge evaluations.

**Detailed**: The guardrails-detectors repository provides three distinct detector microservices designed to integrate with the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator), an IBM-led open source project for invoking detectors on text generation input and output. Each detector is packaged as a standalone container image exposing a FastAPI-based REST API conforming to the IBM Detector API specification.

The repository produces three detector types: (1) a **built-in detector** providing lightweight, rule-based detection functions including regex-based PII detection (email, credit cards, SSN, phone numbers, IP addresses), file type validation (JSON, XML, YAML with optional schema validation), and user-defined custom Python detectors with static code analysis sandboxing; (2) a **HuggingFace detector** that loads any AutoModelForSequenceClassification, AutoModelForTokenClassification, or GraniteForCausalLM model for ML-based content classification (toxicity, prompt injection, hate speech); and (3) an **LLM Judge detector** that uses the vLLM Judge library to perform LLM-as-a-judge evaluations against an external OpenAI-compatible LLM server.

The detectors are designed for deployment on Red Hat OpenShift AI via KServe InferenceService and ServingRuntime custom resources. The HuggingFace and LLM Judge detectors include example deployment manifests for KServe with Istio sidecar injection and auth annotations. The project is maintained by the TrustyAI team.

## Architectural Analysis

The guardrails-detectors repository follows a microservice architecture where each detector type is a standalone FastAPI application sharing a common base framework (`detectors/common/`). The common layer provides the `DetectorBaseAPI` class with standardized health endpoints, exception handling, Prometheus metrics counters (requests, detections, errors, runtime), and a detector registry pattern. All three detector types implement the same `/api/v1/text/contents` endpoint with the `ContentAnalysisHttpRequest`/`ContentsAnalysisResponse` schema, making them interchangeable from the orchestrator's perspective. This pluggable design allows new detector types to be added without changing the orchestrator.

The built-in detector's custom detector feature is architecturally noteworthy. It dynamically loads user-defined Python functions from a `custom_detectors.py` file, performing static code analysis (AST-based) to block forbidden imports (`os`, `subprocess`, `sys`, `shutil`) and dangerous calls (`eval`, `exec`, `open`, `compile`, `input`) before execution. This provides a sandboxed extension mechanism, though the sandboxing is limited to static analysis — a sophisticated attacker could bypass AST-based checks via metaclasses, decorators, or `__import__`. The custom detector framework also supports `@non_blocking` decorators for background execution and `@use_instruments` for user-defined Prometheus metrics, showing a mature extensibility design.

The HuggingFace detector supports three distinct model architectures (sequence classification, token classification, and causal LM) through runtime auto-detection via `AutoConfig`. The token classification path includes offset mapping for character-level span detection and adjacent-token grouping into contiguous spans — a non-trivial implementation that enables NER-style PII detection. The causal LM path specifically targets IBM Granite Guardian models, using template-based chat formatting to evaluate multiple risk categories (harm, social bias, jailbreak, profanity, etc.) in a single pass. Per-request parameter resolution (`_resolve_params`) provides a layered configuration model: environment defaults → per-label thresholds → request-time overrides.

A notable gap is the absence of a Konflux Dockerfile for the LLM Judge detector. Only a standard `Dockerfile.judge` exists, and no Tekton PipelineRun is defined for it. This means the LLM Judge detector is not built through the Konflux secure supply chain pipeline, making it unsuitable for production RHOAI releases until a Konflux variant is added. Similarly, the non-hermetic build configuration (hermetic=false in Tekton pipelines) and lack of dependency lock files represent supply chain risks that should be addressed before GA. The HuggingFace Konflux Dockerfile's approach of building PyTorch from source on ppc64le/s390x architectures (cloning from GitHub at build time) is particularly concerning for hermetic build requirements.

The Prometheus metrics implementation uses the `PROMETHEUS_MULTIPROC_DIR` pattern required for pre-fork servers (uvicorn with multiple workers), with a dedicated `/metrics` endpoint that aggregates counters across worker processes. The metric naming follows the `trustyai_guardrails_` prefix convention, with labels for `detector_kind` and `detector_name` to enable per-detector monitoring. The built-in detector intentionally limits metric cardinality by using generic labels like `custom_regex` instead of publishing user-provided regex patterns as label values.
