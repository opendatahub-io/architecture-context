## Purpose

**Short**: A collection of text detection microservices for the FMS Guardrails Orchestrator, providing regex-based PII detection, HuggingFace model inference, and LLM-as-a-Judge evaluation.

**Detailed**: The guardrails-detectors repository implements the detector backend for the FMS (Foundation Model Stack) Guardrails Orchestrator. The Guardrails Orchestrator intercepts text generation input/output and routes content to these detector microservices for analysis. Each detector variant exposes a standard REST API (`/api/v1/text/contents`) that the orchestrator calls to evaluate text content against various safety and compliance criteria.

The repository produces three distinct detector microservices, each shipped as a separate container image: (1) the **built-in detector** — a lightweight, regex-based engine for PII detection (emails, SSNs, credit cards, phone numbers, IP addresses) and file-type validation (JSON, XML, YAML with optional schema enforcement), plus a custom detector framework allowing arbitrary Python-coded guardrails; (2) the **HuggingFace detector** — a PyTorch-based runtime that loads any HuggingFace `AutoModelForSequenceClassification`, `AutoModelForTokenClassification`, or `GraniteForCausalLM` model for content classification tasks such as toxicity, prompt injection, or safety evaluation; (3) the **LLM Judge detector** — a stateless proxy that delegates evaluation to an external vLLM-compatible LLM server using the `vllm_judge` library, supporting built-in metrics (safety, accuracy, helpfulness, etc.) and custom natural-language evaluation criteria.

All three variants are designed for deployment as KServe InferenceService predictor containers within Red Hat OpenShift AI, using the platform's serving infrastructure for model serving, autoscaling, and traffic management. The built-in detector is co-deployed with the Guardrails Orchestrator as an out-of-the-box detector, while the HuggingFace and LLM Judge detectors are deployed as user-configured inference services.

## Architectural Analysis

The guardrails-detectors repository follows a microservice-per-detector-type pattern where each detector variant is a separate FastAPI application sharing a common base framework. The shared `DetectorBaseAPI` class (in `detectors/common/app.py`) provides Prometheus instrumentation (four counter families: detections, requests, errors, runtime), a health endpoint, and consistent error handling. Each detector variant extends this base and registers its detector implementation at startup via the FastAPI lifespan pattern.

The built-in detector implements a registry-of-registries pattern: three `BaseDetectorRegistry` subclasses (RegexDetectorRegistry, FileTypeDetectorRegistry, CustomDetectorRegistry) are registered at startup, and incoming requests specify which registry and which specific detector functions to invoke via the `detector_params` dictionary. This design allows a single request to invoke multiple detectors across multiple registries, and the custom detector framework enables runtime-extensible guardrails by loading Python functions from a mounted file. The security sandbox (`static_code_analysis`) provides a first-pass defense against dangerous code in custom detectors, though it relies on AST analysis and could be bypassed by sufficiently motivated adversaries — this is a known trade-off for functionality.

The HuggingFace detector supports three distinct model architectures through runtime auto-detection (`AutoConfig.from_pretrained`): sequence classification for binary/multi-label toxicity models, token classification for NER-style PII detection with character-level span output, and causal LM specifically for IBM Granite Guardian models. The `_resolve_params` method provides a layered configuration system where environment variables set defaults that can be overridden per-request via `detector_params`, supporting both global thresholds and per-label thresholds. The token classification path is architecturally notable — it aggregates adjacent tokens with the same label into contiguous spans with averaged scores, producing output compatible with the same `ContentAnalysisResponse` schema used by the regex detectors.

A notable architectural gap is that the LLM Judge detector has no Konflux Dockerfile, meaning it is not built through the RHOAI Konflux pipeline and is only available as an upstream community image (`quay.io/trustyai/guardrails-detector-llm-judge`). The built-in and HuggingFace detectors are built for four architectures (x86_64, ppc64le, arm64, s390x) through multi-arch Konflux builds. The HuggingFace Dockerfile (`Dockerfile.konflux.hf`) handles the significant complexity of building PyTorch from source on ppc64le and s390x, including compiling OpenBLAS for ppc64le — this is a substantial build pipeline that takes considerable time and resources. The non-hermetic build mode (`hermetic: false`) in the Tekton pipelines is a supply chain risk — dependencies are fetched from PyPI at build time without lockfile-based verification.
