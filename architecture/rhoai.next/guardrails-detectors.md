## Purpose

**Short**: A collection of detector algorithm microservices for the FMS Guardrails Orchestrator, providing text content analysis for safety, PII, file validation, and LLM-as-a-judge evaluation.

**Detailed**: The guardrails-detectors repository provides three distinct detector microservices that integrate with the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator), an open source project led by IBM for invoking detectors on text generation input/output. Each detector is packaged as a separate container image, served by FastAPI/Uvicorn, and exposes the IBM Detector API (`/api/v1/text/contents`).

The three detector types serve different use cases: (1) the **built-in detector** provides lightweight, heuristic detection functions (regex-based PII detection, file type validation, and custom Python-defined detectors) that run without ML models; (2) the **HuggingFace detector** wraps HuggingFace transformer models (AutoModelForSequenceClassification and GraniteForCausalLM) to perform ML-based content classification such as toxicity or prompt injection detection; and (3) the **LLM Judge detector** leverages an external OpenAI-compatible LLM server via the vLLM Judge library to perform LLM-as-a-judge content assessment with configurable metrics and criteria. The HuggingFace and LLM Judge detectors are designed for deployment as KServe InferenceServices on RHOAI, while the built-in detector is deployed alongside the Guardrails Orchestrator.

## Architectural Analysis

The guardrails-detectors repository follows a **microservice-per-detector-type** pattern, where each of the three detector categories (built-in, HuggingFace, LLM Judge) is packaged as an independent container image with its own Dockerfile and dependency set. All three share a common library (`detectors/common/`) that provides the FastAPI base application class (`DetectorBaseAPI`), Pydantic request/response schemas conforming to the IBM Detector API, and a Prometheus instrumentation framework (`InstrumentedDetector`). This shared foundation ensures consistent API contracts and metrics across all detector types, which is critical for the Guardrails Orchestrator to treat them uniformly.

The **built-in detector** uses a registry pattern (`BaseDetectorRegistry`) where detector functions are organized by type (regex, file_type, custom). The custom detector subsystem is particularly notable: it dynamically loads user-provided Python code from `custom_detectors/custom_detectors.py` at startup, but performs static code analysis via AST inspection to block dangerous imports (`os`, `subprocess`, `sys`, `shutil`) and function calls (`eval`, `exec`, `open`, `compile`, `input`) before execution. This sandboxing approach provides a usability–security trade-off, though it relies on static analysis which can be circumvented by obfuscation. The custom detector system also supports decorator-based Prometheus metric registration (`@use_instruments`) and non-blocking background execution (`@non_blocking`).

The **HuggingFace detector** auto-detects the model architecture at startup by inspecting the model config's `architectures` field, supporting `AutoModelForSequenceClassification` (standard classification) and `GraniteForCausalLM` (IBM Granite Guardian causal LM with risk-name-based prompting). The causal LM path iterates over seven predefined risk categories (harm, social_bias, jailbreak, profanity, etc.) and computes risk probabilities from log probabilities. The `SAFE_LABELS` environment variable provides a mechanism to exclude specific model output labels from being flagged, enabling per-deployment tuning without code changes. The Konflux Dockerfile for this component is notably complex, with conditional multi-architecture build logic that compiles PyTorch and OpenBLAS from source for ppc64le and s390x platforms.

The **LLM Judge detector** is architecturally distinct from the other two: it has no local model and instead delegates all evaluation to an external LLM server via the `vllm_judge` library. This makes it lightweight (no GPU required) but introduces a synchronous network dependency on the LLM server. It exposes an additional endpoint (`/api/v1/text/generation`) not present in the other detectors, supporting prompt+response pair evaluation. The detector supports 31+ built-in evaluation metrics from vLLM Judge, plus custom natural-language criteria with Jinja2 templating, making it highly flexible for diverse evaluation scenarios.

All detectors expose plaintext HTTP endpoints without built-in TLS or authentication. When deployed on RHOAI as KServe InferenceServices, security is delegated to the platform: Istio sidecar injection provides mTLS between services, and the `security.opendatahub.io/enable-auth: 'true'` annotation enables KServe-level authentication. The built-in detector, which is deployed directly (not via KServe), relies on network-level isolation within the cluster. This "no auth at the application layer" design is consistent with the RHOAI platform's security model where the service mesh handles transport security.
