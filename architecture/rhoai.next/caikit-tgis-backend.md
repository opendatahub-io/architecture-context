## Purpose

**Short**: Python library providing a Caikit module backend that manages connections to TGIS (Text Generation Inference Service) for text generation inference.

**Detailed**: caikit-tgis-backend is a Python library (not a standalone service or operator) that extends the Caikit AI framework's backend system to provide integration with TGIS, an inference engine for large language models. It is distributed as a pip-installable Python package and is embedded into runtime container images that use the Caikit serving framework.

The library registers a `TGIS` backend type with Caikit's backend registry, enabling Caikit-based model runtimes to delegate text generation, streaming generation, tokenization, and model info queries to a TGIS server over gRPC. It supports both remote TGIS connections (with TLS/mTLS, client-side load balancing, and DNS-based endpoint discovery) and local TGIS subprocess management for development/testing scenarios. The library includes generated protobuf stubs for the `fmaas.GenerationService` gRPC API and a custom load balancer proxy that polls DNS for endpoint changes and transparently reconnects gRPC channels.

This component does not define any CRDs, deploy any Kubernetes resources, or expose any network services on its own. It is consumed as a dependency by higher-level components such as caikit-nlp and caikit-tgis-serving, which build container images that include this library.

## Architectural Analysis

caikit-tgis-backend follows a clean plugin architecture, registering itself as a named backend type ("TGIS") with Caikit's backend registry at module import time (line 258 of `tgis_backend.py`). This allows any Caikit-based runtime to discover and use TGIS backends through Caikit's standard configuration system without hard-coding the integration. The `TGISBackend` class extends `BackendBase` and provides two operating modes: remote (connecting to an external TGIS server) and local (managing a TGIS subprocess), selected automatically based on whether a connection hostname is configured.

The gRPC load balancing implementation (`GRPCLoadBalancerProxy`) is notably sophisticated for a library of this size. Rather than relying on gRPC's built-in name resolution, it implements custom DNS polling on a configurable interval (default 10 seconds) using `socket.getaddrinfo()`. When new IP addresses are detected, it transparently reconnects the gRPC channel by reinitializing the client stub's internal state — a technique that preserves external references to the client object while swapping the underlying channel. This is critical in Kubernetes environments where headless Service DNS records change as TGIS pods scale.

The TLS configuration supports a full certificate chain: CA certificate for server validation, and optional client certificate + key pair for mTLS. Certificate files are validated for existence at initialization time, and the library enforces that client TLS cannot be set without a CA certificate, and that client cert and key must be provided as a pair. This validation occurs in `TGISConnection.from_config()` (lines 82-183 of `tgis_connection.py`). All crypto is delegated to gRPC's native SSL channel credentials, meaning FIPS compliance is inherited from the underlying OpenSSL library in the container.

The `ManagedTGISSubprocess` class implements a robust lifecycle manager with auto-recovery. When TGIS crashes or becomes unhealthy, the `_FaultDetectingRPC` wrapper detects specific gRPC error codes (ABORTED, DEADLINE_EXCEEDED, INTERNAL, UNAVAILABLE, UNKNOWN) and triggers an auto-recovery check in a background thread. This design is primarily for development and testing scenarios — production deployments use remote TGIS connections managed by Kubernetes.

One architectural limitation worth noting: the library currently does not support multiple locally-managed TGIS instances (each TGIS process serves one model), though there are TODO comments indicating this is a known gap. For remote connections, multi-model support is handled via per-model connection configurations in the `remote_models` config section, with each model potentially having a different TGIS hostname.
