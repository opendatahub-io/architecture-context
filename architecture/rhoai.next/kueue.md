## Purpose

**Short**: Kueue is a Kubernetes-native job queueing system that manages when workloads should be admitted to start and when they should be preempted, providing quota management, fair sharing, and priority-based scheduling across cluster resources.

**Detailed**: Kueue implements a job-level manager for Kubernetes that controls resource quota allocation and admission for batch workloads. It introduces a hierarchical queueing model with ClusterQueues (cluster-scoped resource pools), LocalQueues (namespace-scoped submission points), and Workloads (the internal representation of resource requests). The system supports multiple scheduling strategies including StrictFIFO and BestEffortFIFO, with sophisticated preemption policies that allow borrowing and lending resources between ClusterQueues organized in Cohorts.

The component integrates with a wide range of AI/ML workload frameworks including Kubernetes batch Jobs, Kubeflow training jobs (PyTorchJob, TFJob, MPIJob, PaddleJob, XGBoostJob), Ray clusters and jobs, JobSets, AppWrappers (CodeFlare), LeaderWorkerSets, and plain Pods. Each framework integration is implemented via mutating and validating webhooks that intercept workload creation to inject queue annotations and enforce admission control. Kueue also provides advanced features like Topology-Aware Scheduling (TAS) for optimizing pod placement on closely-located nodes, MultiKueue for distributing workloads across multiple clusters, and Provisioning Request support for cluster autoscaler integration.

In RHOAI, Kueue is deployed into the `opendatahub` namespace with the `kueue-` name prefix via a kustomize overlay. The RHOAI configuration disables VisibilityOnDemand and MultiKueue, enables LocalQueueMetrics, and restricts certain webhook namespaces to avoid interfering with OpenShift system namespaces. The controller is built with FIPS-compliant crypto (GOEXPERIMENT=strictfipsruntime) for Red Hat product requirements.

## Architectural Analysis

Kueue follows a sophisticated multi-layer queueing architecture that separates resource allocation concerns from workload execution. The core abstraction is the Workload CRD, which acts as a framework-agnostic representation of resource requests. Each supported job framework (batch/v1 Job, Kubeflow training jobs, Ray, etc.) has a dedicated integration that translates framework-specific job specifications into Workload objects and manages the suspend/unsuspend lifecycle. This abstraction allows Kueue to support new frameworks by implementing a simple `GenericJob` interface and registering webhook handlers, without modifying the core scheduling logic.

The scheduling algorithm operates in a three-phase cycle: (1) pending workloads are sorted by priority and creation time within their ClusterQueue's queueing strategy (StrictFIFO or BestEffortFIFO), (2) the scheduler attempts to find a valid flavor assignment that fits within the ClusterQueue's nominal quota or borrowable capacity from cohort members, and (3) if no fit is found, the preemption engine evaluates whether lower-priority workloads can be evicted to make room. The preemption logic supports four configurable policies: `Never`, `LowerPriority`, `LowerOrNewerEqualPriority`, and `Any`, with separate controls for within-queue and cross-cohort preemption. The Fair Sharing extension adds usage-weighted distribution across ClusterQueues in a Cohort, using DRF (Dominant Resource Fairness) shares.

The RHOAI deployment makes several architectural choices that differ from upstream defaults. MultiKueue is disabled (the RHOAI platform does not currently use multi-cluster workload distribution), VisibilityOnDemand is disabled (the legacy visibility API is deprecated), and LocalQueueMetrics is enabled for better per-namespace monitoring. Webhook namespace selectors are carefully configured to exclude OpenShift system namespaces (kube-system, openshift-*, etc.) to prevent Kueue from interfering with platform operations. The Job webhook uses a `NotIn` operator to exclude system namespaces, while Pod and Deployment webhooks use a `matchLabels` selector requiring `kueue-managed: "true"`, providing explicit opt-in semantics.

The PrometheusRule configuration in RHOAI defines three alert categories: a critical `KueuePodDown` alert (fires after 5 minutes of pod unreadiness), and informational alerts for `LowClusterQueueResourceUsage` (below 20% utilization for 1 day), `ResourceReservationExceedsQuota` (10x over-reservation for 10 minutes), and `PendingWorkloadPods` (pods pending for 3+ days). These alerts include runbook URLs pointing to the opendatahub-io/runbooks repository.

The controller concurrency is explicitly configured per resource type: 5 concurrent reconcilers for Jobs, Pods, and Workloads, but only 1 for LocalQueues, Cohorts, ClusterQueues, and ResourceFlavors. This design reflects the fact that queue and quota operations require serialized access to prevent race conditions in admission decisions, while workload reconciliation benefits from parallelism. The rate limiter is configured at 50 QPS with burst of 100, applied globally across all API types to prevent overwhelming the API server.
