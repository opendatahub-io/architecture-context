## Purpose

**Short**: Kueue is a Kubernetes-native job queueing system that manages when jobs should be admitted to start and when they should stop, based on resource quotas, priorities, and fair sharing policies.

**Detailed**: Kueue provides a set of APIs and controllers for job-level resource management in Kubernetes clusters. It introduces the concepts of ClusterQueues (cluster-scoped resource pools with quotas), LocalQueues (namespace-scoped queue endpoints for users), ResourceFlavors (hardware variants like GPU models or pricing tiers), and Workloads (the unit of admission representing a job's resource requirements). The scheduler evaluates pending workloads against available quota, considering priorities, preemption policies, borrowing/lending limits within cohorts, and fair sharing weights.

Kueue integrates with multiple job frameworks including batch/v1 Jobs, JobSets, Kubeflow training jobs (PyTorch, TensorFlow, XGBoost, Paddle, MPI), Ray jobs/clusters, AppWrappers, Deployments, StatefulSets, LeaderWorkerSets, and bare Pods. Advanced features include MultiKueue for multi-cluster workload federation, Provisioning integration with cluster autoscaler via ProvisioningRequests, Topology-Aware Scheduling (TAS) for placing pods within specific topology domains (racks, blocks), and a visibility API for introspecting pending workload queues. In RHOAI, Kueue is deployed with specific feature gates (VisibilityOnDemand=false, LocalQueueMetrics=true, MultiKueue=false) and configured to manage batch jobs, Kubeflow training jobs, Ray jobs, JobSets, and AppWrappers.

## Architectural Analysis

Kueue follows a classic Kubernetes operator pattern built on controller-runtime, but with several distinctive architectural choices. The core scheduling logic is not a standard reconciler — instead, the `scheduler.Scheduler` implements `manager.Runnable` and runs as a continuous scheduling loop. It takes snapshots of the cache state, evaluates pending workloads from all ClusterQueues, assigns resource flavors considering borrowing limits and preemption, and admits workloads by updating their status. This separation between the reactive controllers (which maintain cache state) and the proactive scheduler (which makes admission decisions) is a key architectural pattern.

The controller communication architecture uses an observer pattern with typed watcher interfaces (`ClusterQueueUpdateWatcher`, `WorkloadUpdateWatcher`, `AdmissionCheckUpdateWatcher`, `ResourceFlavorUpdateWatcher`). When a ResourceFlavor changes, the ResourceFlavor controller notifies the ClusterQueue controller via a channel, which then re-evaluates all ClusterQueues using that flavor. This event-driven cascade avoids polling and enables responsive updates across the dependency graph. The `core.SetupControllers` function in `pkg/controller/core/core.go` orchestrates the wiring of these watchers during initialization.

The job framework (`pkg/controller/jobframework`) provides a plugin architecture for integrating diverse job types. Each integration implements the `GenericJob` interface with methods like `Suspend`, `RunWithPodSetsInfo`, `PodSets`, and `IsActive`. Optional interfaces (`JobWithReclaimablePods`, `JobWithCustomStop`, `ComposableJob`) extend behavior for specific job patterns. The `IntegrationManager` registers all integrations at init time, and `SetupControllers` dynamically creates controllers and webhooks based on the configured frameworks list. This design allows adding new job integrations without modifying the core scheduler.

The MultiKueue subsystem is particularly notable for its remote cluster client management pattern. The `ClustersReconciler` maintains a map of remote Kubernetes clients, each connected via kubeconfig (from Secrets or filesystem paths). It uses `fsnotify` for filesystem-based kubeconfig watching and implements garbage collection to clean up orphaned remote workloads. The `WlReconciler` synchronizes workload state between local and remote clusters, copying admission decisions and job status. Connection resilience is achieved through exponential backoff retry (0-320 seconds) and a configurable `workerLostTimeout` for detecting cluster disconnections.

The RHOAI distribution overlay in `config/rhoai/` is significant for its security additions: a NetworkPolicy restricting ingress to the webhook port (9443/TCP), a dedicated ServiceAccount and Secret for Prometheus metrics authentication, and PrometheusRule alerts for operational monitoring. The feature gate configuration disables VisibilityOnDemand and MultiKueue while enabling LocalQueueMetrics, reflecting RHOAI's conservative approach to alpha features. The FIPS-compliant build (`GOEXPERIMENT=strictfipsruntime`, `CGO_ENABLED=1`) ensures the binary uses OpenSSL for cryptographic operations instead of Go's standard library crypto, meeting Red Hat's FIPS certification requirements.
