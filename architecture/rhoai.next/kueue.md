## Purpose

**Short**: Kueue is a Kubernetes-native job queueing system that manages when workloads should be admitted to start based on quota, priority, and fair sharing policies.

**Detailed**: Kueue provides a set of APIs and controllers for job-level resource management in Kubernetes clusters. It implements a hierarchical quota management system through ClusterQueues and LocalQueues, enabling administrators to define resource quotas with multiple flavors (representing different hardware types or pricing models) and organize them into cohorts for resource sharing. The system decides when jobs should be admitted to start (allowing pod creation) and when they should stop (triggering pod deletion), acting as a layer between job submission and the Kubernetes scheduler.

Kueue integrates with multiple batch and ML training frameworks including Kubernetes Jobs, JobSets, Kubeflow training operators (PyTorchJob, TFJob, MPIJob, XGBoostJob, PaddleJob), Ray (RayJob, RayCluster), and CodeFlare AppWrappers. It supports advanced scheduling features including preemption policies, fair sharing across cohorts, topology-aware scheduling (TAS) for GPU locality optimization, provisioning request integration with cluster autoscaler, and multi-cluster workload distribution via MultiKueue. In the RHOAI distribution, Kueue is deployed in the `opendatahub` namespace with specific feature gates configured for the platform.

## Architectural Analysis

Kueue's architecture follows a classic Kubernetes operator pattern but with notably high internal complexity due to its role as a cross-cutting resource management system. The core design revolves around a three-layer queue hierarchy: Workloads (representing individual job resource requests) are submitted to LocalQueues (namespace-scoped), which reference ClusterQueues (cluster-scoped) that define actual resource quotas. ClusterQueues can be organized into Cohorts for resource sharing and borrowing. This hierarchy enables multi-tenant resource management without requiring direct cluster-level access for job submitters.

The controller architecture uses a sophisticated watcher pattern where controllers notify each other of state changes through interfaces (e.g., `ClusterQueueUpdateWatcher`, `WorkloadUpdateWatcher`) rather than relying solely on Kubernetes watch events. This is essential because Kueue's admission decisions depend on aggregate state (total queue utilization, cohort-level borrowing) that cannot be efficiently derived from individual resource watches. The internal `cache.Cache` and `queue.Manager` components maintain in-memory state that is synchronized across controllers, with the `scheduler.Scheduler` running as a separate goroutine making admission decisions based on this cached state.

The job framework integration layer is designed as a plugin system through the `jobframework.IntegrationCallbacks` interface. Each supported job type (Job, JobSet, PyTorchJob, etc.) registers itself via Go `init()` functions in the `pkg/controller/jobs/` directory. The generic reconciler (`jobframework.JobReconciler`) handles the common lifecycle (suspend/resume, Workload creation, status synchronization) while delegating job-specific behavior to the integration callbacks. This allows adding new job type support with minimal code changes. Notably, the RHOAI deployment configures 10 framework integrations by default, covering all major ML training frameworks.

The admission check system is architecturally extensible. Two built-in controllers demonstrate the pattern: the **Provisioning** controller creates `ProvisioningRequest` objects to trigger cluster autoscaler node provisioning, with exponential backoff retry logic for failed provisioning attempts. The **MultiKueue** controller implements multi-cluster workload distribution by maintaining connections to remote Kueue clusters via kubeconfig Secrets, with garbage collection of orphaned remote workloads and file system watching for kubeconfig updates. Both are disabled in the RHOAI deployment (`MultiKueue=false`, though `ProvisioningACC` is enabled by default at beta). The admission check pattern allows third-party controllers to gate workload admission without modifying Kueue itself.

Security follows a defense-in-depth approach: the webhook endpoint uses TLS certificates managed either internally or by cert-manager, the metrics endpoint enforces authentication and authorization via controller-runtime filters, and a NetworkPolicy restricts webhook ingress to port 9443. The Konflux build uses `GOEXPERIMENT=strictfipsruntime` with `CGO_ENABLED=1` for FIPS compliance. The controller runs as non-root (UID 65532) with `runAsNonRoot`, `RuntimeDefault` seccomp profile, and all capabilities dropped. The RHOAI overlay adds dedicated metrics reader ServiceAccount and Secret for secure Prometheus scraping.
