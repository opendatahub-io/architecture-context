## Purpose

**Short**: Kueue is a Kubernetes-native job queueing system that manages when workloads should be admitted to start based on quota, priority, and fair sharing policies.

**Detailed**: Kueue provides a set of APIs and controllers for job-level resource management in Kubernetes clusters. It acts as a gatekeeper between job submission and pod creation, deciding when a job should be admitted (allowing its pods to be created) and when it should be stopped (deleting active pods) based on resource quotas, priorities, and fair sharing policies.

The system introduces ClusterQueues (cluster-scoped resource pools with quotas), LocalQueues (namespace-scoped entry points for workload submission), and Workloads (the internal representation of a unit of work). Kueue integrates with multiple job frameworks including Kubernetes batch/v1 Jobs, Kubeflow Training Operator jobs (PyTorchJob, TFJob, XGBoostJob, PaddleJob), MPI Operator jobs, KubeRay jobs and clusters, JobSets, AppWrappers, LeaderWorkerSets, Deployments, StatefulSets, and raw Pods.

Advanced features include MultiKueue for multi-cluster workload distribution, Topology-Aware Scheduling (TAS) for hardware-aware pod placement, Provisioning Request integration with cluster autoscalers, hierarchical Cohorts for shared quota management, and fair sharing for equitable resource distribution across queues.

## Architectural Analysis

Kueue follows the Kubernetes operator pattern using controller-runtime, with a single binary (`kueue-controller-manager`) hosting all controllers, webhooks, the admission scheduler, and the visibility API server. The architecture is built around a plugin-based job framework where each supported workload type (Job, PyTorchJob, RayJob, etc.) registers integration callbacks for controller setup, webhook setup, and workload construction. This design allows adding new job types without modifying core scheduling logic.

The core scheduling loop operates as a periodic cycle rather than event-driven admission. The scheduler pulls pending workloads from the queue manager, attempts to assign resource flavors and quotas from the in-memory cache, and admits workloads by updating their status. The cache maintains a live snapshot of ClusterQueue quotas, flavor assignments, and workload states, synchronized through a watcher interface pattern where controllers notify each other of state changes via Go channels. This in-memory coordination avoids expensive API server round-trips during scheduling decisions but requires leader election to prevent split-brain scheduling across replicas.

The MultiKueue subsystem implements a federated scheduling model where a management cluster distributes workloads to worker clusters. Each worker cluster connection is managed by a `remoteClient` that maintains a watch on remote Workload objects labeled with a `multikueue-origin` label. Job framework adapters implement a `MultiKueueAdapter` interface to sync job-type-specific resources between clusters, creating remote copies with a `prebuilt-workload-label` that tells the remote Kueue instance to skip workload creation. This dual-cluster model supports graceful degradation when workers become unreachable, with configurable worker-lost timeouts and periodic garbage collection of orphaned remote workloads.

The Topology-Aware Scheduling (TAS) subsystem extends Kueue's scheduling with hardware topology awareness. Topology definitions map node labels to hierarchical levels (e.g., rack → host), and ResourceFlavors reference topologies to enable topology-constrained scheduling. The `topologyUngater` controller handles the critical post-admission step: after the scheduler assigns topology domains to workload pod sets, the ungater removes `TopologySchedulingGate` from pods and injects precise `nodeSelector` constraints matching the assigned topology domain. This two-phase approach (gate → schedule → ungate with selectors) avoids conflicts with the Kubernetes scheduler while ensuring topology-aware placement.

The webhook system uses a custom `WebhookBuilder` that enhances controller-runtime's default builder with lossless defaulting — it drops JSON patch "remove" operations for fields not present in the Go type, enabling forward compatibility when newer API versions add fields unknown to the current binary. Every supported job type gets both mutating (queue name defaulting, suspension injection, managedBy field) and validating (queue name immutability, topology request validation) webhooks. The RHOAI distribution adds a NetworkPolicy restricting ingress to the webhook port (9443/TCP), providing defense-in-depth for the admission control path.
