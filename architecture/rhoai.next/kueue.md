## Purpose

**Short**: Kueue is a Kubernetes-native job queueing system that manages when workloads should be admitted to start based on quota, priority, and fair sharing policies.

**Detailed**: Kueue provides a set of APIs and controllers for job-level resource management in Kubernetes clusters. It acts as a job-level manager that decides when a job should be admitted to start (allowing pod creation) and when it should stop (deleting active pods). Kueue introduces a resource model based on ClusterQueues that define quotas for named ResourceFlavors (representing different hardware tiers like GPU models or CPU architectures), and LocalQueues that provide namespace-scoped entry points for workload submission.

The system supports a pluggable job framework architecture that integrates with multiple workload types: Kubernetes batch/Job, JobSet, Kubeflow Training Operator jobs (PyTorchJob, TFJob, XGBoostJob, PaddleJob, MPIJob), Ray (RayJob, RayCluster), CodeFlare AppWrapper, and Kubernetes-native Pods/Deployments/StatefulSets. Advanced features include multi-cluster workload distribution via MultiKueue, topology-aware scheduling (TAS) for GPU/accelerator locality, provisioning request integration with cluster-autoscaler, hierarchical cohorts for resource sharing, fair sharing with preemption strategies, and partial admission for elastic workloads.

Within RHOAI, Kueue is deployed as a component managed by the rhods-operator, providing job queue management for AI/ML training and inference workloads across the platform.

## Architectural Analysis

Kueue implements a sophisticated job-level resource management system built around a clean separation between the queueing/admission decision layer and the job framework integration layer. The core architecture consists of three main subsystems: (1) the queue management and caching layer that maintains an in-memory representation of all ClusterQueues, Cohorts, and their quotas, (2) the scheduler that periodically evaluates pending workloads against available quota, and (3) the pluggable job framework that abstracts away differences between job types behind the `GenericJob` interface.

The job framework's plugin architecture is one of Kueue's most notable design decisions. Each job type (batch/Job, JobSet, Kubeflow training jobs, Ray, etc.) registers itself via an `IntegrationCallbacks` struct that provides schema registration, controller/webhook factories, and MultiKueue adapters. This allows the operator to support 13+ job types without tight coupling. The framework provides shared reconciliation logic — creating Workload objects for jobs, managing suspension/unsuspension, injecting node affinity from ResourceFlavor assignments, and handling eviction/preemption. Job-specific controllers only need to implement the `GenericJob` interface and optional extension interfaces like `JobWithCustomStop` or `JobWithManagedBy`.

The MultiKueue subsystem adds a multi-cluster dimension to the architecture. It uses an admission check controller pattern where a `MultiKueueConfig` references a set of `MultiKueueCluster` resources, each containing a kubeconfig reference (stored in Secrets). When a workload is admitted with a MultiKueue admission check, the controller creates a remote copy of both the Workload and the owning Job on a worker cluster, then synchronizes status back. This is implemented via job-specific `MultiKueueAdapter` implementations that know how to clone and sync each job type. The system includes a file system watcher for kubeconfig changes and garbage collection for orphaned remote resources.

The Topology-Aware Scheduling (TAS) feature adds hardware-locality awareness to the scheduling decisions. It introduces a `Topology` CRD that defines hierarchical levels (e.g., block → rack → hostname), and the TAS ResourceFlavor controller continuously scrapes node labels to maintain a topology map. During admission, the scheduler can assign workload pods to specific topology domains, and the TAS ungater controller removes scheduling gates from pods and injects the appropriate node selectors. This is critical for GPU workloads where pod-to-pod communication latency matters.

The RHOAI deployment variant (config/rhoai) adds production hardening: a NetworkPolicy restricting webhook ingress to port 9443, a dedicated ServiceAccount with a token Secret for metrics authentication, PrometheusRule alerts, and image parameterization via params.env. The controller image is built with FIPS-compliant Go flags (`GOEXPERIMENT=strictfipsruntime`, `CGO_ENABLED=1`) on a UBI 9 minimal base image, ensuring compliance with Red Hat's FIPS requirements for government deployments.
