## Purpose

**Short**: KServe model server runtime for serving AutoGluon TabularPredictor and TimeSeriesPredictor models via the KServe inference protocol (v1 and v2).

**Detailed**: The kserve-autogluon-server is a downstream Red Hat build of the AutoGluon serving runtime for KServe. It packages the AutoGluon ML framework into a container image that can be referenced by a `ClusterServingRuntime` CR, enabling `InferenceService` instances to serve AutoGluon models on Red Hat OpenShift AI. The server auto-detects whether a saved model artifact is a TabularPredictor or TimeSeriesPredictor and delegates to the appropriate model handler.

This repository is a fork of the upstream kserve/kserve repository but is scoped specifically to building the autogluon server image via Konflux. The Python source in `python/autogluonserver/` implements the serving logic, while the kserve SDK (`python/kserve/`) and storage library (`python/storage/`) are vendored dependencies that provide the HTTP/gRPC server framework and model storage download capabilities. The Konflux Dockerfile (`Dockerfile.konflux.autogluon`) builds the image hermetically using a Red Hat-hosted PyPI index (`console.redhat.com/api/pypi/public-rhai/rhoai/3.4/cpu-ubi9/simple/`) for dependency resolution with pinned AutoGluon versions (`1.5.0+rhaiv.2`).

## Architectural Analysis

The kserve-autogluon-server follows a clean delegation pattern: the `AutoGluonDetectedModel` factory class attempts to load the model directory as a `TimeSeriesPredictor` first, then falls back to `TabularPredictor`. This try-load detection (`predictor_detect.py`) means users don't need to declare the model type explicitly — the `storageUri` just points at the AutoGluon save directory and the server figures it out. The factory pattern is well-isolated, with `AutoGluonTabularModel` and `AutoGluonTimeSeriesModel` each implementing the KServe `Model` interface independently.

The v2 inference protocol support is asymmetric: tabular models fully support v2 tensor payloads (with careful feature-name mapping from `InferRequest` tensor names to DataFrame column names), while time series explicitly rejects v2 requests and only supports v1 JSON. This is a pragmatic design choice — time series data with its hierarchical structure (multiple series, timestamps, known covariates) maps poorly to flat tensor inputs. The `_infer_request_to_dataframe` function in `tabular_model.py` handles the complex mapping from v2 per-feature tensors to a pandas DataFrame, including shape validation and byte decoding.

The `runtime_paths.py` module addresses a container-specific concern: AutoGluon's `SeasonalNaive` fallback internally calls `setup_outputdir` which tries to create directories under the current working directory. In containers running as arbitrary UIDs (OpenShift's default security model), the default cwd `/` is not writable. The `ensure_autogluon_runtime_paths` function proactively changes the cwd to a writable location (`/tmp`, `$HOME`, or `/home/kserve`) before any AutoGluon operations occur. This is a Red Hat-specific hardening that prevents runtime crashes in restricted security contexts.

The build hermeticity story is notably strong for a Python ML service. The `autogluon-all-requirements.txt` file (596 lines) pins every transitive dependency, sourced from a Red Hat-curated PyPI index (`console.redhat.com/api/pypi/public-rhai/rhoai/3.4/cpu-ubi9/simple/`). The Tekton pipeline configures Hermeto pip prefetch with `hermetic: "true"`, and the build targets multi-arch (x86_64, ppc64le, s390x, arm64). The AutoGluon version itself carries a Red Hat version suffix (`1.5.0+rhaiv.2`), indicating it's a Red Hat-patched build of the upstream AutoGluon 1.5.0 release.

The `predict_proba` support (enabled via `PREDICT_PROBA=true` environment variable) is particularly well-implemented for the v2 protocol. When enabled on classification models, the server returns one output tensor per class label (named `proba_{sanitized_label}`), with careful handling of label sanitization to ensure output names are valid identifiers. The `_get_proba_output_names` function handles edge cases like numeric labels, special characters, and collisions.
