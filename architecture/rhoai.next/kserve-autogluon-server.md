## Purpose

**Short**: KServe model server for serving AutoGluon TabularPredictor and TimeSeriesPredictor models via REST v1/v2 and gRPC inference protocols.

**Detailed**: The kserve-autogluon-server is a downstream Red Hat fork of the KServe project, specifically producing the AutoGluon serving runtime container image (`odh-kserve-autogluon-server`). It enables users to deploy AutoGluon models as KServe InferenceService resources on Red Hat OpenShift AI.

The server auto-detects whether a model artifact is a TabularPredictor or TimeSeriesPredictor at startup by attempting to load each type in sequence. Tabular models support both KServe v1 REST and v2 inference protocols (including probability output mode via `PREDICT_PROBA`), while time series models support REST v1 JSON only. The server integrates with KServe's model storage system to download models from cloud storage (S3, GCS, Azure Blob) or local paths before serving.

This repository is a multi-component KServe fork containing Go operator code, Python SDK, and multiple Python model servers, but the Konflux build configuration produces only the AutoGluon server image. The autogluon server depends on the KServe Python SDK (`kserve`) and storage module (`kserve-storage`) as local path dependencies.

## Architectural Analysis

The kserve-autogluon-server is architecturally simple: a thin Python wrapper around AutoGluon's TabularPredictor and TimeSeriesPredictor, using the KServe Python SDK as the serving framework. The server's primary design decision is auto-detection -- rather than requiring users to specify the model type in the InferenceService YAML, the server tries loading as TimeSeriesPredictor first, then falls back to TabularPredictor. This is implemented through the `AutoGluonDetectedModel` facade pattern in `predictor_factory.py`, which delegates to the appropriate model class after detection.

The version compatibility layer (`version_compat.py`) is a notable defensive pattern: it reads AutoGluon's saved version file and compares major.minor versions before loading, allowing patch-level mismatches (e.g., `1.5.0` vs `1.5.0+rhaiv.3`) while rejecting major/minor version differences. This is critical because the RHOAI fork uses custom AutoGluon builds with the `+rhaiv.N` local version suffix, and saved models from slightly different patch versions must still load correctly.

The tabular model implementation (`tabular_model.py`) is the most complex file, handling both v1 (JSON dict) and v2 (tensor) inference protocols. It includes sophisticated type mapping from AutoGluon's feature metadata to v2 tensor datatypes, and supports optional probability output mode (`PREDICT_PROBA`) for classification models. The time series model (`timeseries_model.py`) only supports v1 JSON, using a column metadata system (`predictor_metadata.json` plus environment variable overrides) to map between AutoGluon's internal column names and the user's JSON payload.

The container runtime paths module (`runtime_paths.py`) addresses a practical container constraint: AutoGluon's SeasonalNaive fallback creates temporary directories under the current working directory, which fails in non-root containers where the default cwd is `/`. The module changes the working directory to a writable location at startup -- a pattern specific to running AutoGluon in constrained OpenShift environments.

The dependency footprint is substantial: the `autogluon-all-requirements.txt` lists ~150 transitive packages including PyTorch, transformers, spacy, and scikit-learn. This makes the container image large but necessary for full AutoGluon backend coverage (LightGBM, XGBoost, CatBoost, torch NN, fastai). The AIPCC CPU base image provides a secure foundation, and the RHEL AI Python Package Index (`console.redhat.com/api/pypi/public-rhai/rhoai/3.5-EA2/cpu-ubi9/simple/`) ensures all wheels are from Red Hat's secure build pipeline.
