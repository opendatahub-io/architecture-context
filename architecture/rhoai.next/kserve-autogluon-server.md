## Purpose

**Short**: A KServe-compatible model serving runtime that serves AutoGluon TabularPredictor and TimeSeriesPredictor models via REST and gRPC inference APIs.

**Detailed**: The KServe AutoGluon Server is a Python-based model serving runtime built on the KServe SDK framework. It provides inference capabilities for AutoGluon machine learning models, specifically supporting two predictor types: TabularPredictor for tabular data prediction (classification, regression, quantile regression) and TimeSeriesPredictor for time series forecasting.

The server auto-detects which predictor type a given model directory contains and delegates inference accordingly. It supports both the KServe v1 REST predict API and the v2 Open Inference Protocol (OIP), including tensor-based payloads. The tabular model supports probability predictions via the `PREDICT_PROBA` environment variable and exposes v2 model metadata (input/output schemas). The time series model currently supports v1 REST JSON only.

This repository is a downstream fork of the upstream kserve/kserve repository, narrowed to build only the AutoGluon serving container image for RHOAI. It uses Konflux hermetic builds with a custom Red Hat PyPI index (`console.redhat.com/api/pypi/public-rhai/rhoai/3.4/cpu-ubi9/simple/`) for dependency resolution. The AutoGluon version is pinned to `1.5.0+rhaiv.2`, a Red Hat internal variant.

## Architectural Analysis

The KServe AutoGluon Server follows the standard KServe model serving runtime pattern: it implements the `kserve.Model` interface and delegates server lifecycle to `kserve.ModelServer`. The KServe SDK provides the FastAPI-based HTTP server (v1 and v2 endpoints) and gRPC server out of the box, so the autogluon server code focuses solely on model loading, input parsing, and inference logic. This is a clean separation of concerns — the serving framework handles protocol compliance, health checks, and metrics, while the runtime handles ML-specific logic.

The auto-detection mechanism in `predictor_factory.py` and `predictor_detect.py` is noteworthy: when a model directory is provided, the server attempts to load it as a `TimeSeriesPredictor` first, then falls back to `TabularPredictor`. This try-load pattern means users don't need to specify the predictor type at deployment time. The `AutoGluonModelRepository` class extends this to support multi-model serving, where the server scans a model directory for subdirectories and loads each as a separate model.

The tabular model implementation (`tabular_model.py`) is the most feature-complete, with full support for v2 tensor-based payloads. It handles the mapping between v2 tensor inputs (named tensors with shapes) and AutoGluon's DataFrame-based input format, including feature name resolution, type mapping, and column ordering. It also supports `predict_proba` mode for classification models, where per-class probability outputs are returned as separate named tensors. The time series model (`timeseries_model.py`) currently only supports v1 REST JSON, which is appropriate for its more complex input structure (history rows + optional known covariates).

The container image is built hermetically via Konflux using a custom Red Hat PyPI index. The base image is `registry.redhat.io/rhai/base-image-cpu-rhel9`, and the final image runs as non-root UID 1001. The `runtime_paths.py` module handles a specific OpenShift constraint: AutoGluon's `SeasonalNaive` fallback writes to the current working directory, which is read-only in many container configurations. The module ensures a writable working directory exists (`/tmp` or `$HOME`) before inference begins. The `MPLCONFIGDIR` environment variable is also set to `/tmp/matplotlib` to prevent matplotlib configuration writes to read-only paths.

A key architectural observation is that this repository is a full fork of the kserve/kserve repo (containing the entire Go operator, Python SDK, router, and all serving runtimes), but the Konflux build only produces the autogluon server image. The other Dockerfiles (sklearn, xgboost, huggingface, etc.) are present but not built by this repository's Konflux pipeline. Each serving runtime has its own separate downstream repository for RHOAI builds.
