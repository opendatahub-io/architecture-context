## Purpose

**Short**: A KServe model server that serves AutoGluon TabularPredictor and TimeSeriesPredictor models for inference via REST v1/v2 and gRPC protocols.

**Detailed**: The kserve-autogluon-server is a downstream (Red Hat) fork of the kserve repository, narrowly focused on producing a single container image: the AutoGluon model server. This server auto-detects whether a saved model artifact is a TabularPredictor or TimeSeriesPredictor at startup and delegates inference accordingly. Tabular models support both KServe v1 (JSON) and v2 (tensor) inference protocols, while time series models currently support only REST v1 JSON.

The server is deployed as a predictor container inside KServe `InferenceService` custom resources. When a user creates an InferenceService with `modelFormat.name: autogluon`, the KServe controller selects the appropriate `ClusterServingRuntime` that references this image. The container downloads the model from the configured `storageUri` (S3, GCS, Azure Blob, or local), loads it via AutoGluon, and exposes HTTP and gRPC inference endpoints.

This repository is a full fork of upstream kserve (including Go operator code, CRDs, and all Python server packages), but only the `python/autogluonserver` directory and `Dockerfile.konflux.autogluon` are relevant to the shipped component. The rest of the repository provides build-time dependencies (the `kserve` Python SDK and `kserve-storage` packages) that are compiled into the image.

## Architectural Analysis

The kserve-autogluon-server follows the standard KServe model server pattern: a Python process implementing the `kserve.Model` interface, served by the kserve SDK's `ModelServer` class which provides both HTTP (FastAPI/uvicorn) and gRPC endpoints. The server's distinguishing feature is its auto-detection mechanism: at startup, `predictor_factory.py` instantiates an `AutoGluonDetectedModel` which tries loading the model directory first as a `TimeSeriesPredictor`, then as a `TabularPredictor`. This means a single container image and `ClusterServingRuntime` can serve both model types without user configuration beyond `modelFormat.name: autogluon`.

The tabular model implementation (`tabular_model.py`) is notably more sophisticated than the time series side. It implements full KServe v2 tensor protocol support with detailed type mapping from AutoGluon feature metadata to v2 tensor datatypes (INT64, FP64, BOOL, BYTES). The v2 contract requires one tensor per feature with the tensor name matching the feature name -- a design that exposes model schema through `GET /v2/models/{name}`. The `PREDICT_PROBA` environment variable toggle enables a classification-specific mode where probabilities per class are returned instead of hard labels, with v2 output tensors named `proba_{class_label}`. The time series implementation (`timeseries_model.py`) is currently v1-only, accepting JSON payloads with `instances` (history) and optional `known_covariates` arrays.

A notable design decision is the `version_compat.py` module, which implements patch-level version tolerance for AutoGluon model loading. AutoGluon writes its version to `version.txt` inside saved predictors and refuses to load across version mismatches. The server relaxes this to allow patch-level differences (e.g., 1.5.0 vs 1.5.0+rhaiv.3, the Red Hat internal version suffix) while still blocking major/minor mismatches. This is critical for RHOAI where the AutoGluon packages carry `+rhaiv.N` local version suffixes from Red Hat's secure build pipeline.

The repository is a full fork of upstream kserve, containing the Go operator, CRDs, and all Python server packages (huggingfaceserver, sklearnserver, etc.), but only the autogluon server image is built and shipped from this fork. The `Dockerfile.konflux.autogluon` copies the `kserve` SDK and `kserve-storage` packages as local path dependencies and installs them alongside the autogluon server. All Python dependencies are resolved against the RHEL AI Python Package Index (`console.redhat.com/api/pypi/public-rhai/rhoai/3.5-EA2/cpu-ubi9/simple/`) via a hash-pinned requirements file (`autogluon-all-requirements.txt`), which provides supply chain integrity for the ~150 transitive dependencies including torch, transformers, and the AutoGluon stack. The `runtime_paths.py` module addresses a container-specific concern: AutoGluon's SeasonalNaive fallback (for short time series histories) tries to create directories under cwd, which fails when running as non-root UID 1001 with cwd at `/`. The module relocates cwd to a writable directory at startup.
