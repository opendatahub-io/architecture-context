## Purpose

**Short**: A KServe-compatible model server for serving AutoGluon TabularPredictor and TimeSeriesPredictor models via REST and gRPC inference APIs.

**Detailed**: The kserve-autogluon-server is a Python-based inference server that wraps AutoGluon's TabularPredictor and TimeSeriesPredictor within the KServe model serving framework. It runs as a container in a KServe InferenceService pod, exposing the standard KServe v1 and v2 (Open Inference Protocol) REST endpoints and gRPC inference endpoints on ports 8080/TCP and 8081/TCP respectively.

The server auto-detects whether a saved model directory contains a tabular or time series predictor and delegates to the appropriate model implementation. For tabular models, it supports both v1 JSON and v2 tensor-based inference payloads, including predict and predict_proba modes. For time series models, it supports v1 JSON payloads with time series history and optional known covariates. The server is built on the KServe Python SDK which provides FastAPI-based REST serving, gRPC serving, Prometheus metrics, and model repository management.

This component is part of the RHOAI KServe serving runtime ecosystem. It is built as a RHOAI-specific container image (`odh-kserve-autogluon-server-rhel9`) using Konflux pipelines, pulling AutoGluon packages from Red Hat's internal PyPI index with pinned versions (`autogluon.tabular==1.5.0+rhaiv.2`, `autogluon.timeseries==1.5.0+rhaiv.2`).

## Architectural Analysis

The kserve-autogluon-server follows the standard KServe model server pattern: a thin model-specific adapter layer on top of the KServe Python SDK. The autogluon-specific code is minimal (~700 lines across 7 files), with the heavy lifting done by the KServe SDK (FastAPI serving, gRPC serving, v1/v2 protocol handling, model repository, metrics) and AutoGluon itself (model loading, prediction).

The auto-detection design (`predictor_factory.py` / `predictor_detect.py`) is notable — the server tries to load the model directory as a TimeSeriesPredictor first, then falls back to TabularPredictor. This allows a single container image to serve both model types without requiring the user to specify which predictor type at deployment time. The `AutoGluonDetectedModel` class wraps this pattern cleanly, delegating to the appropriate implementation after detection.

The tabular model implementation (`tabular_model.py`) has sophisticated v2 tensor handling. It maps between v2's tensor-per-feature format and AutoGluon's DataFrame-based interface, including feature name resolution, datatype mapping (AutoGluon raw types → v2 datatypes like INT64, FP64, BYTES), and comprehensive error messages with contract hints. The time series model (`timeseries_model.py`) is more constrained — it explicitly rejects v2 InferRequest payloads and only supports v1 JSON with `instances` (history) and optional `known_covariates`.

The `runtime_paths.py` module addresses a practical container runtime concern: AutoGluon's SeasonalNaive fallback tries to create `./AutogluonModels` in the current working directory, which fails when running as a non-root user with cwd set to `/`. The workaround changes cwd to a writable directory (`/tmp`, `$HOME`, or `/home/kserve`). Similarly, the `MPLCONFIGDIR` environment variable is set to prevent matplotlib cache directory issues. These are pragmatic fixes for running ML frameworks in restrictive OpenShift pod security contexts.

The Konflux build (`Dockerfile.konflux.autogluon`) uses a Red Hat UBI9 base image (`registry.redhat.io/rhai/base-image-cpu-rhel9`) and installs all Python dependencies from Red Hat's internal PyPI index with RHOAI-specific AutoGluon builds (`1.5.0+rhaiv.2`). The build is multi-stage: a builder stage installs all dependencies into a virtualenv, generates third-party license information, and a prod stage copies the result. The `PREDICT_PROBA` environment variable controls whether tabular classification models return probability distributions per class or just the predicted label.
