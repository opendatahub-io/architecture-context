## Purpose

**Short**: Thin container image wrapper that extends the AIPCC vLLM ROCm base image for AMD GPU inference on RHOAI.

**Detailed**: The vllm-rocm repository produces a container image for running vLLM large language model inference on AMD ROCm GPUs within Red Hat OpenShift AI. The repository contains no application source code — its sole artifact is a Konflux Dockerfile (`Dockerfile.konflux.rocm`) that extends the RHAIIS (Red Hat AI Inference Server) vLLM ROCm base image (`registry.redhat.io/rhaiis/vllm-rocm-rhel9`).

The base image from AIPCC Ecosystems provides the complete vLLM runtime, ROCm GPU libraries, and the `vllm_tgis_adapter` Python module. This repository's Dockerfile adds only environment variable defaults (ports, speculative decoding configuration), a non-root user, container metadata labels, and the TGIS adapter entrypoint. The resulting image is deployed as an inference runtime container by KServe's `ServingRuntime` CRD on RHOAI clusters with AMD Instinct GPUs.

The upstream vLLM README and pre-commit configuration are included in the repository but are inherited from the upstream vLLM project and are not used in the RHOAI build process.

## Architectural Analysis

The vllm-rocm repository follows the thinnest possible "container image wrapper" pattern: a single-stage, 24-line Dockerfile that extends an AIPCC Ecosystems base image with only environment variables, labels, and an entrypoint. There is no application source code, no build logic, no dependency installation, and no configuration files beyond the Dockerfile itself. All runtime content — vLLM, the TGIS adapter, ROCm libraries, Python, and system packages — is inherited entirely from `registry.redhat.io/rhaiis/vllm-rocm-rhel9`.

This architecture means that the security posture, FIPS compliance, and dependency management of the resulting image are almost entirely determined by the AIPCC base image. The vllm-rocm repo itself adds only the TGIS adapter entrypoint (`python3 -m vllm_tgis_adapter --uvicorn-log-level=warning`) and two configuration environment variables (GRPC_PORT=8033, PORT=8000, DISABLE_LOGPROBS_DURING_SPEC_DECODING=false). The `USER 2000` directive ensures non-root execution.

The dual-protocol design (HTTP on port 8000 for the OpenAI-compatible API, gRPC on port 8033 for the TGIS-compatible API) is a key pattern for RHOAI inference runtimes. The vllm_tgis_adapter module bridges vLLM's native OpenAI API with the TGIS gRPC interface that KServe and the model-serving operator expect. This allows the same container to serve both protocol families without separate images. The commit history shows the TGIS entrypoint and triton installation were added as distinct concerns (commits 2b145d6, a9202705), and the AIPCC integration was a later migration (commit 7c21195) consolidating the build onto the RHAIIS base image pipeline.

The build is explicitly non-hermetic (`hermetic: false` in the Tekton pipeline), which is acceptable for a thin wrapper that installs no packages. The Tekton pipeline builds only for `linux/amd64` architecture, consistent with AMD ROCm's x86_64-only support as documented in overlay 0017. The pipeline uses extended timeouts (8h pipeline, 4h task) reflecting the large image size typical of GPU inference runtimes.
