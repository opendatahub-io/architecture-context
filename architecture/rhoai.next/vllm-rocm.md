## Purpose

**Short**: GPU-accelerated vLLM container image built with AMD ROCm support for high-performance LLM inference on AMD GPUs, wrapped with the vllm-tgis-adapter for TGIS gRPC protocol compatibility.

**Detailed**: This repository produces a container image for running vLLM inference workloads on AMD ROCm-enabled GPUs within the Red Hat OpenShift AI (RHOAI) platform. The image is a thin overlay on the Red Hat AI Inference Server (RHAIIS) `vllm-rocm-rhel9` base image, adding a customized entrypoint that launches the `vllm_tgis_adapter` module. This adapter provides both an OpenAI-compatible HTTP API (port 8000) and a TGIS-compatible gRPC interface (port 8033), enabling seamless integration with KServe's ServingRuntime model serving infrastructure.

The repository contains no application source code — the vLLM engine, ROCm libraries, and vllm-tgis-adapter are all provided by the upstream RHAIIS base image. The repository's role is to define the Konflux build pipeline configuration, the Dockerfile overlay, and the container image metadata (labels, entrypoint, environment variables) that produce the final shippable RHOAI artifact. The build is orchestrated by Konflux via Tekton PipelineRuns synced from the `konflux-central` repository.

## Architectural Analysis

This repository exemplifies the RHOAI "thin overlay" pattern for inference runtimes. Rather than building vLLM from source, the repository inherits the entire vLLM stack — including the ROCm-accelerated engine, PagedAttention memory management, continuous batching, and the vllm-tgis-adapter — from the pre-built RHAIIS `vllm-rocm-rhel9:3.2.1` base image. The overlay Dockerfile adds only entrypoint configuration, environment variable defaults, and container metadata labels. This pattern enables RHOAI to ship patched and hardened inference runtimes while minimizing the build-time delta from the upstream RHAIIS product.

The dual-port architecture (HTTP on 8000, gRPC on 8033) is a critical design decision. The HTTP port exposes an OpenAI-compatible API that enables direct client access and broad ecosystem compatibility. The gRPC port exposes the TGIS (Text Generation Inference Server) protocol, which is KServe's native inference protocol for text generation. This dual-protocol approach allows the same container to serve both KServe-orchestrated workloads (via gRPC) and direct API consumers (via HTTP) without requiring a protocol translation layer.

The `DISABLE_LOGPROBS_DURING_SPEC_DECODING=false` environment variable override is noteworthy — upstream vLLM disables log probabilities during speculative decoding as a performance optimization, but RHOAI re-enables them to avoid surprising behavior for users of hosted models that may use speculative decoding transparently. This prioritizes API consistency over raw throughput, which is appropriate for a managed platform.

Security at the container level is intentionally minimal — the container runs as non-root user (UID 2000) but does not implement TLS or authentication natively. Instead, it relies on the RHOAI platform's ingress stack (Gateway API with kube-rbac-proxy sidecars in RHOAI 3.x) to enforce encryption and authorization before traffic reaches the container. This separation of concerns keeps the inference runtime focused on model serving while delegating security enforcement to the platform layer.

The Konflux build pipeline is the sole CI/CD mechanism, running in the `rhoai-tenant` namespace with generous timeouts (8 hours pipeline, 4 hours per task) reflecting the size and complexity of ROCm-based container builds. Pipeline definitions are centrally managed in `konflux-central` and automatically synced to this repository's `.tekton/` directory.
