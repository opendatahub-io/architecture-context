## Purpose

**Short**: Builds a ROCm (AMD GPU) variant of the vLLM inference serving runtime for Red Hat OpenShift AI, wrapping the upstream RHAIIS vllm-rocm base image with TGIS adapter configuration.

**Detailed**: The vllm-rocm repository is a downstream container image build repository that produces the `odh-vllm-rocm-rhel9` container image for RHOAI. It contains no application source code — the entire vLLM inference engine, ROCm GPU libraries, and vllm-tgis-adapter are provided by the base image `registry.redhat.io/rhaiis/vllm-rocm-rhel9`. This repo's role is to configure the entrypoint, set runtime environment variables (gRPC/HTTP ports, speculative decoding behavior), enforce non-root execution, and apply Red Hat container labels.

The resulting container is deployed as an inference serving runtime within KServe on OpenShift AI. It provides both an OpenAI-compatible HTTP API (port 8000) and a TGIS-compatible gRPC API (port 8033) for serving large language models on AMD ROCm GPUs. The `vllm_tgis_adapter` Python module wraps vLLM's inference engine to expose the TGIS gRPC interface alongside the native OpenAI-compatible HTTP endpoint, enabling compatibility with both the KServe model serving framework and Red Hat's TGIS-based inference pipeline.

## Architectural Analysis

The vllm-rocm repository represents a minimal "image wrapper" pattern commonly used in RHOAI for downstream container builds. Rather than containing application source code, it layers configuration on top of a pre-built RHAIIS base image. This architectural pattern has several implications:

**Dependency on RHAIIS base image**: The entire software stack — vLLM inference engine, ROCm GPU libraries, Python runtime, vllm-tgis-adapter, and all transitive dependencies — is packaged in the `registry.redhat.io/rhaiis/vllm-rocm-rhel9` base image. This creates a hard dependency on the RHAIIS product pipeline for all code changes. The `RHAIIS_VERSION` build argument (defaulting to 3.2.1) pins the base image version, providing version control at the image layer rather than at the source code layer. This is the AIPCC (AI Platform Common Components) integration pattern — as noted in commit `7c2119552`, the repository was updated to "consume vLLM ROCm from RHAIIS releases" rather than building from source.

**Dual-protocol serving**: The entrypoint configures the `vllm_tgis_adapter` module, which wraps vLLM to expose two serving interfaces simultaneously: the native OpenAI-compatible HTTP API on port 8000 and the TGIS (Text Generation Inference Service) gRPC protocol on port 8033. This dual-protocol design enables backward compatibility — existing TGIS-based KServe configurations continue to work (gRPC) while new deployments can use the industry-standard OpenAI API (HTTP). The `DISABLE_LOGPROBS_DURING_SPEC_DECODING=false` environment variable overrides vLLM's default optimization to ensure consistent API behavior regardless of whether speculative decoding is enabled.

**Security posture**: The container runs as non-root user 2000, following OpenShift security best practices. However, the build is explicitly non-hermetic (`hermetic: false` in the Tekton pipeline), delegating supply chain security entirely to the RHAIIS base image build process. No FIPS-specific configuration exists in this repository — FIPS compliance is inherited from the UBI9 base image's system OpenSSL when the OpenShift node has FIPS mode enabled. No TLS is configured at the application level; encryption is handled by the platform's ingress stack (KServe gateway, kube-rbac-proxy sidecars).

**Build infrastructure**: The Tekton pipeline runs through Konflux with extended timeouts (8h pipeline, 4h per task), reflecting the large size of GPU-enabled container images. The pipeline references a shared definition from `konflux-central`, builds only for `linux-extra-fast/amd64` (no ARM support — expected for ROCm workloads), and includes Clair vulnerability scanning and ecosystem cert preflight checks with elevated resource allocations (16Gi memory).
