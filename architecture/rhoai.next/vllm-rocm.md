## Purpose

**Short**: Repackages the upstream RHAIIS vLLM ROCm container image for AMD GPU inference serving within Red Hat OpenShift AI, adding TGIS adapter integration.

**Detailed**: The vllm-rocm repository is a container image repackaging project that takes the upstream Red Hat AI Infrastructure Services (RHAIIS) vLLM ROCm image (`registry.redhat.io/rhaiis/vllm-rocm-rhel9`) and customizes it for use as an inference serving runtime in RHOAI. The repository contains no application source code — its sole artifact is a Konflux Dockerfile that layers configuration on top of the RHAIIS base image.

The key customization is the entrypoint, which wraps vLLM with the `vllm_tgis_adapter` module. This adapter enables vLLM to serve models using the TGIS (Text Generation Inference Server) gRPC protocol in addition to the standard OpenAI-compatible HTTP API. The container exposes two ports: 8000/TCP for the OpenAI-compatible HTTP API and 8033/TCP for the TGIS gRPC interface. The image runs as non-root user (UID 2000) and disables logprobs during speculative decoding to ensure predictable behavior for hosted model users.

This component is deployed by KServe as a ServingRuntime on RHOAI clusters equipped with AMD ROCm-capable GPUs. It is the AMD GPU counterpart to the NVIDIA CUDA-based vLLM image produced by the sibling `vllm` repository.

## Architectural Analysis

The vllm-rocm repository follows a minimal "image repackaging" pattern common in RHOAI's container image supply chain. Rather than containing application source code, it applies a thin configuration layer atop the RHAIIS base image. This architecture centralizes the complex build process (compiling vLLM with ROCm support, installing AMD GPU libraries, packaging the TGIS adapter) in the upstream RHAIIS image pipeline, while giving the RHOAI team a separate Konflux-managed build point for layering RHOAI-specific defaults.

The TGIS adapter entrypoint (`python3 -m vllm_tgis_adapter`) is the most architecturally significant choice. It bridges vLLM's native OpenAI-compatible API with the TGIS gRPC protocol that KServe's model serving infrastructure expects. This dual-protocol exposure (HTTP on 8000, gRPC on 8033) allows the same container image to serve both OpenAI API consumers and the platform's internal gRPC-based routing.

The `DISABLE_LOGPROBS_DURING_SPEC_DECODING=false` environment variable override is notable — it reverses vLLM's upstream default to ensure logprob availability even when speculative decoding is enabled. This reflects RHOAI's requirement that hosted models behave predictably regardless of backend optimizations.

The non-hermetic build (`hermetic: false` in Tekton pipeline) is a calculated tradeoff: since the Dockerfile performs no package installation or source compilation (it only sets environment variables and labels on the base image), there are no untrusted dependencies to isolate. The RHAIIS base image itself is expected to be hermetically built in its own pipeline. The `RHAIIS_VERSION` ARG (defaulting to `3.2.1`) pins the base image version, providing build reproducibility.

The Tekton pipeline is synchronized from `konflux-central`, indicating this component participates in RHOAI's centralized CI/CD orchestration. The pipeline builds for `linux-extra-fast/amd64` platform only, with substantial resource allocations (8-16 CPU, 16-32Gi RAM) for the Clair vulnerability scan and ecosystem certification preflight checks, reflecting the large image size typical of ROCm-based containers.
