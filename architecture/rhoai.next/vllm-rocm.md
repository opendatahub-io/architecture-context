## Purpose

**Short**: Thin container image wrapper that repackages the AIPCC-built vLLM ROCm base image for RHOAI deployment with the vllm-tgis-adapter entrypoint.

**Detailed**: The vllm-rocm repository produces a container image for running vLLM inference serving on AMD ROCm GPUs within the Red Hat OpenShift AI platform. The repository itself contains no application source code — it consists of a single Konflux Dockerfile that extends the AIPCC Ecosystems base image (`registry.redhat.io/rhaiis/vllm-rocm-rhel9`) by setting environment variables, configuring a non-root user, and defining the entrypoint for the `vllm_tgis_adapter` serving process.

The vllm-tgis-adapter provides a compatibility layer that wraps vLLM's inference engine with a TGIS-compatible gRPC interface (port 8033) alongside vLLM's native OpenAI-compatible HTTP API (port 8000). This allows the component to be used as a KServe/ModelMesh serving runtime where existing TGIS clients can communicate via gRPC while new clients can use the OpenAI-compatible REST API. The image is built for AMD ROCm accelerators (x86_64 only) and is deployed as an inference serving container within KServe InferenceService pods.

## Architectural Analysis

This repository exemplifies the "thin wrapper" pattern used in RHOAI's inference serving stack. The entire repository consists of a single Dockerfile that adds no packages, patches, or source code — it only sets environment variables, a non-root user, an entrypoint, and OCI labels on top of the AIPCC Ecosystems base image. All actual functionality (vLLM inference engine, ROCm GPU libraries, the vllm-tgis-adapter gRPC bridge, and Python runtime) comes from the upstream `registry.redhat.io/rhaiis/vllm-rocm-rhel9` image built by the AIPCC Ecosystems team.

The `RHAIIS_VERSION=3.2.1` build argument pins the base image version, creating a clear version coupling between this repository and the AIPCC release cadence. The commit history shows this integration was introduced in commit `7c2119552` ("Integrate AIPCC to consume vLLM ROCm from RHAIIS releases"), representing a migration from a previous build approach where the Dockerfile likely compiled vLLM from source. This migration simplifies the build substantially but means all Python dependency management, ROCm library versions, and security patching are now entirely upstream responsibilities.

The dual-port design (HTTP 8000 + gRPC 8033) is architecturally significant: port 8000 serves the OpenAI-compatible REST API (used by KServe's v2 inference protocol and direct clients), while port 8033 serves the TGIS-compatible gRPC protocol (used by ModelMesh and legacy TGIS clients). The `DISABLE_LOGPROBS_DURING_SPEC_DECODING=false` environment variable overrides a vLLM performance optimization to maintain API compatibility — speculative decoding normally disables logprobs for efficiency, but hosted model users expect logprobs to always be available. This is a notable UX-over-performance trade-off documented directly in the Dockerfile.

The non-hermetic build (`hermetic: false` in Tekton) is appropriate for this wrapper pattern — since the Dockerfile downloads nothing and installs nothing, hermeticity is a property of the base image build, not this layer. However, the absence of `rpms.lock.yaml` even on the main branch means there is no lockfile-based audit trail for OS package versions in the final image. This is acceptable because the image inherits everything from the pinned AIPCC base image, which has its own hermeticity controls.
