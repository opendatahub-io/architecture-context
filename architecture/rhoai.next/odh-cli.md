## Purpose

**Short**: CLI tool for validating, managing, and migrating Red Hat OpenShift AI (RHOAI) deployments on Kubernetes.

**Detailed**: The `odh-cli` (shipped as `rhai-cli` / `kubectl-odh`) is a comprehensive command-line tool that provides operational capabilities for Red Hat OpenShift AI deployments. It operates as a kubectl plugin, connecting to the Kubernetes API server to perform read-only validation (lint), health assessment (status), resource inspection (get/components), dependency management (deps), workload backup, log streaming, and migration orchestration.

The tool is built in Go with a modular command architecture based on Cobra. It integrates deeply with the RHOAI platform CRDs (DataScienceCluster, DSCInitialization) and OLM (Operator Lifecycle Manager) to discover installed components, detect versions, check operator dependencies, and validate upgrade readiness. The lint subsystem alone implements 37 diagnostic checks across five categories — platform, dependencies, components, workloads, and services — covering upgrade paths from RHOAI 2.x to 3.x.

The CLI is distributed as a multi-architecture container image (`quay.io/rhoai/rhoai-upgrade-helpers-rhel9`) based on `ose-cli-rhel9`, which bundles the Go binary alongside `kubectl`, `oc`, `jq`, Python, and shell-based upgrade helper scripts. It can also be installed as a standalone Go binary or used as a kubectl plugin (`kubectl odh`).

## Architectural Analysis

The odh-cli exhibits a mature, well-structured architecture that balances extensibility with operational safety. The command lifecycle pattern (AddFlags → Complete → Validate → Run) is consistently applied across all 8 top-level commands and their subcommands, creating a predictable developer experience for adding new commands. The separation of `cmd/` (Cobra wiring) from `pkg/` (business logic) is clean and thorough — the cmd layer contains no business logic, only flag registration and error handling delegation.

The lint subsystem is the most architecturally significant component, implementing a full diagnostic framework with 37 checks organized into five execution groups (Dependency → Service → Platform → Component → Workload). The check framework uses composition-based metadata (`BaseCheck` embedding), functional options for condition construction, and fluent validation builders (`validate.Component()`, `validate.Workloads[T]()`) that dramatically reduce boilerplate. Each check produces Kubernetes CR-like `DiagnosticResult` structures with typed conditions, making the output machine-parseable while maintaining human readability. The version-aware `CanApply()` mechanism allows checks to self-select based on the current and target RHOAI versions, enabling the same codebase to handle both current-state validation and upgrade readiness assessment.

The client architecture demonstrates a sophisticated Reader/Writer split where lint checks enforce read-only access at the type level — checks accept `client.Reader` and cannot accidentally mutate cluster state. The multi-client composition (Dynamic + Discovery + APIExtensions + OLM + Metadata + CoreV1) with a cached REST mapper handles the diverse API surface efficiently. Error handling is particularly well-designed: permission errors are treated as non-fatal (the CLI degrades gracefully when RBAC is insufficient), resource-not-found errors emit friendly messages when CRDs aren't installed, and the 6-level exit code system (0-5) enables CI/CD pipelines to distinguish between success, warnings, errors, validation failures, auth problems, and connection issues.

The migration framework (`pkg/migrate/`) introduces a pluggable two-phase action pattern (Prepare → Run) with hierarchical step recording, enabling complex multi-step migrations like the Kueue RHBOK transition to be tracked, resumed, and audited. The conflict-resolution strategy uses exponential backoff with jitter for DSC patching, correctly handling concurrent modifications. The dependency management system (`pkg/deps/`) embeds odh-gitops manifests at build time but supports runtime refresh, with security validation that restricts OLM catalog sources to an allow-list (CWE-829 mitigation) and rejects system namespace targets (CWE-269 mitigation).

The container image strategy is notable: the image bundles Go, Python, and shell tooling into a single `ose-cli-rhel9` base, providing a self-contained operational environment. The FIPS compliance story is complete at the build level (CGO_ENABLED=1, strictfipsruntime, dynamic linking against UBI9 OpenSSL) with no application-level crypto gaps detected. Build hermeticity is strong with RPM locks (rpms.lock.yaml), Go module locks (go.sum), and Python dependency pinning (pip-compile), though Hermeto/cachi2 prefetch integration is absent.
