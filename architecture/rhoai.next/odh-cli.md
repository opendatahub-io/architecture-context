## Purpose

**Short**: CLI tool for validating, managing, and migrating Red Hat OpenShift AI (RHOAI) deployments on Kubernetes clusters.

**Detailed**: odh-cli (also known as `rhai-cli` or `kubectl-odh`) is a command-line interface that provides operational tooling for RHOAI platform administrators. Its primary function is upgrade readiness validation through a comprehensive lint framework that executes 37 diagnostic checks across platform resources, components, dependencies, and user workloads. The tool evaluates whether a cluster can safely upgrade between RHOAI versions (particularly 2.x → 3.x transitions) by inspecting DataScienceCluster configurations, OLM operator subscriptions, and workload compatibility.

Beyond validation, odh-cli provides resource management capabilities including component lifecycle management (enable/disable via DataScienceCluster patches), workload backup with dependency resolution (ConfigMaps, Secrets, PVCs), resource querying with ODH-aware formatting, dependency status reporting via OLM subscription inspection, and automated migrations (e.g., Red Hat Build of Kueue operator installation). The tool ships as both a standalone binary and a container image (`quay.io/rhoai/rhoai-upgrade-helpers-rhel9`) that bundles `oc`, `kubectl`, `jq`, `yq`, and Python-based upgrade helper scripts. The container image is built with FIPS-compliant Go using `strictfipsruntime` build tags.

## Architectural Analysis

odh-cli follows a well-structured plugin architecture centered around three extensibility registries: the **CheckRegistry** for lint validation, the **ActionRegistry** for migrations, and the **DependencyResolverRegistry** for backup operations. All three use the same pattern — explicit registration at startup (no global state), interface-based dispatch, and version-aware applicability filtering. This design makes it straightforward to add new checks, migrations, or backup resolvers without modifying core framework code. The 37 registered lint checks span five groups (dependency, service, platform, component, workload) and execute in canonical order, with each check receiving a read-only Kubernetes client to enforce the principle that validation never mutates cluster state.

The Kubernetes client layer is notably sophisticated for a CLI tool. It configures elevated QPS (50) and burst (100) compared to kubectl defaults (5/100), reflecting the tool's need for parallel resource listing across many CRD types. The client interface cleanly separates `Reader` and `Writer` contracts, allowing lint checks to receive only `Reader` access. The OLM integration uses a nil-safe pattern — when OLM is unavailable, all OLM queries return empty results rather than errors, enabling graceful degradation on clusters without OLM. All resource types are centralized in `pkg/resources/types.go` using a `ResourceType` struct that provides `GVK()`, `GVR()`, and `CRDFQN()` methods, eliminating inline schema construction throughout the codebase.

The lint framework's condition model is particularly well-designed, adopting a Kubernetes CR-inspired pattern with multi-outcome conditions rather than simple pass/fail. Each check produces a `DiagnosticResult` containing conditions with impact levels (Prohibited, Blocking, Advisory, None), which enables nuanced upgrade guidance. The fluent validation builders (`validate.Component()`, `validate.Workload[T]()`, etc.) eliminate significant boilerplate by automatically fetching platform resources, checking management states, and populating result annotations. The workload builder uses Go generics to support both full `*unstructured.Unstructured` objects (when spec inspection is needed) and lightweight `*metav1.PartialObjectMetadata` (for count-only checks), optimizing API server load.

The backup system implements a three-stage pipeline architecture (discovery → resolution → writing) with concurrent worker processing. The dependency resolver pattern is particularly useful for DSPA backups, where it traces references through jq paths to discover associated Secrets (S3 credentials, database passwords), ConfigMaps (CA bundles, custom configs), and PVCs (MariaDB/Minio storage). Resources are written with cluster-specific metadata stripped (`.status`, `.metadata.uid`, `.metadata.resourceVersion`, etc.) to enable portable restoration. The migration framework for Red Hat Build of Kueue (RHBOK) demonstrates the most complex write path in the tool — it creates OLM Subscriptions, polls for CSV readiness with exponential backoff, patches the DataScienceCluster with conflict retry, and verifies resource preservation post-migration, all with full dry-run support.

The container image build (Dockerfile.konflux) is noteworthy for its FIPS compliance approach — both the main binary and the vendored yq are built with `CGO_ENABLED=1 GOEXPERIMENT=strictfipsruntime` and `-tags strictfipsruntime`. The runtime image is based on `ose-cli-rhel9:v4.21.0`, which provides `oc` and `kubectl` natively, while the non-Konflux Dockerfile uses `ubi9:latest` and downloads these tools manually. Python dependencies (`kubernetes`, `PyYAML`) are installed for the `ray_cluster_migration.py` helper script. The image sets `KUBECONFIG=/kubeconfig` as a convention for volume-mounted kubeconfig files.
