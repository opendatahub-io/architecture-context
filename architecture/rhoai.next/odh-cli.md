## Purpose

**Short**: CLI tool for interacting with RHOAI (Red Hat OpenShift AI) deployments on Kubernetes, providing upgrade readiness assessment, cluster migration automation, workload backup, and operational diagnostics.

**Detailed**: odh-cli (shipped as `kubectl-odh` / `rhai-cli`) is a comprehensive command-line tool for Red Hat OpenShift AI platform operators and administrators. It functions as a kubectl plugin and is also distributed as a container image (`quay.io/rhoai/rhoai-upgrade-helpers-rhel9`). The tool serves three primary roles:

First, it provides **upgrade readiness assessment** through its `lint` command, which runs 38 diagnostic checks organized into five groups (platform, components, dependencies, workloads, services). These checks validate cluster state against target version requirements, identifying prohibited, blocking, and advisory findings that affect upgrade feasibility from RHOAI 2.x to 3.x and between 3.x minor versions.

Second, it automates **cluster migrations** through its `migrate` command framework, which supports 19 migration actions across AI Pipelines, KServe model serving, Kueue, RayCluster, TrustyAI, LlamaStack, Training Operator, and Workbenches. Each action follows a prepare/run lifecycle with dry-run support, hierarchical step recording, and RBAC pre-validation. Third, it provides **operational tooling** including workload backup with dependency resolution, component lifecycle management (enable/disable via DataScienceCluster), dependency installation via OLM, platform health status reporting, log aggregation, and event streaming.

## Architectural Analysis

odh-cli is architecturally significant as the **only official CLI tool in the RHOAI platform**, serving as the primary interface for upgrade validation and migration automation. Unlike most components in the platform which are operators running continuously, odh-cli is a stateless, one-shot CLI that performs complex orchestration across many platform components. Its design reflects three key architectural decisions worth noting.

**Comprehensive Kubernetes API surface**. The tool interacts with over 40 distinct Kubernetes resource types spanning ODH/RHOAI CRDs, OLM resources, Kubeflow workloads, KServe serving resources, Kueue scheduling resources, and core Kubernetes types. All GVK/GVR definitions are centralized in `pkg/resources/types.go` following the project's "Principle VIII" (centralized resource type definitions). This centralization prevents drift between different parts of the codebase but creates a single point of maintenance when API versions change. The tool uses dynamic client (`client-go` dynamic) and controller-runtime client for most operations, with dual-version discovery (v2 first, v1 fallback) for DataScienceCluster and DSCInitialization to support both RHOAI 2.x and 3.x clusters.

**Security-aware design with notable trade-offs**. The tool performs RBAC pre-validation via SelfSubjectAccessReview before executing migrations, preventing partial failures from insufficient permissions. The dependency installation subsystem validates OLM catalog sources against an allowlist (redhat-operators, community-operators, certified-operators, redhat-marketplace) to prevent supply-chain attacks. Event output sanitizes ANSI escape sequences to prevent terminal injection. However, the TrustyAI metrics HTTP client uses `InsecureSkipVerify: true` for TLS — this is documented as matching the original `curl -sk` behavior for accessing in-cluster OpenShift Routes with self-signed certificates, but it does bypass certificate validation entirely.

**Three-phase migration lifecycle**. The migration framework uses a prepare/run/verify pattern where each migration action can implement both a prepare task (backup, pre-flight checks) and a run task (actual mutation). The hierarchical step recorder provides an audit trail with color-coded real-time output. Version-aware filtering ensures actions only execute when applicable to the current→target version transition. The stdin input support (`--from-stdin`) enables CI/CD integration by accepting JSON/YAML configuration. API throttling defaults (QPS=50, Burst=100) are 10x higher than kubectl defaults to support the tool's parallel operations (backup pipeline workers, concurrent CSV polling, event label lookups).

**Lint engine design**. The lint system implements a pattern-based check registry with glob matching (e.g., `components.*`, `*dashboard*`) and canonical execution ordering (dependency → service → platform → component → workload). Impact levels (Prohibited > Blocking > Advisory > None) map directly to exit codes, enabling shell scripting: exit 0 = all passed, exit 1 = advisory only, exit 2 = blocked. The fluent validation builders (`validate.Component()`, `validate.Workloads()`, etc.) eliminate significant boilerplate by auto-fetching DataScienceCluster, checking management state preconditions, and populating diagnostic results with impacted object references.
