## Purpose

**Short**: CLI tool for validating, diagnosing, backing up, and migrating Red Hat OpenShift AI (RHOAI) deployments on Kubernetes.

**Detailed**: odh-cli (also known as `kubectl-odh` or `rhai-cli`) is a kubectl plugin that provides operational tooling for RHOAI platform administrators. It connects to live Kubernetes clusters to perform diagnostic validation (lint), workload backup, component management, dependency checking, resource listing, and automated migrations. The CLI is designed for both interactive use and CI/CD integration, supporting table, JSON, and YAML output formats.

The tool's primary value is upgrade readiness assessment. It ships 37 diagnostic checks that evaluate whether a cluster can safely upgrade between RHOAI versions (particularly the 2.x→3.x migration). These checks inspect DataScienceCluster, DSCInitialization, component CRs, OLM operators, and user workloads (Notebooks, InferenceServices, RayClusters, etc.) to detect blocking conditions, deprecated configurations, and required pre-migration steps.

The CLI is distributed as both a standalone Go binary (via goreleaser) and a container image (`quay.io/rhoai/rhoai-upgrade-helpers-rhel9`) that bundles kubectl, oc, jq, yq, Python, and upgrade helper scripts. The container image is built by Konflux with FIPS-compliant crypto (CGO_ENABLED=1, GOEXPERIMENT=strictfipsruntime) and supports linux/amd64, linux/arm64, and linux/ppc64le architectures.

## Architectural Analysis

### Command Architecture and Extensibility

odh-cli follows a well-structured Cobra command hierarchy with a consistent four-phase lifecycle (AddFlags → Complete → Validate → Run) enforced via the `pkg/cmd.Command` interface. This pattern cleanly separates flag parsing from client creation, validation, and execution, enabling comprehensive unit testing without Cobra coupling. Each command package (`pkg/lint`, `pkg/backup`, `pkg/components`, `pkg/deps`, `pkg/get`, `pkg/migrate`) is self-contained with its own types, but shares infrastructure through `pkg/util/client`, `pkg/printer`, and `pkg/output/envelope.go`. The envelope pattern (`cli.opendatahub.io/v1` APIVersion) makes JSON/YAML output self-describing with timestamps, command metadata, and status summaries.

### Lint Framework: A Production-Grade Diagnostic Engine

The lint system is the most architecturally sophisticated part of the codebase. It implements a plugin-style check framework with 37 checks across 5 groups (dependency, service, platform, component, workload), executed sequentially for deterministic output. Each check implements a `Check` interface and returns Kubernetes CR-inspired `DiagnosticResult` objects with multi-condition status reporting, impact levels (Prohibited > Blocking > Advisory > None), and impacted object tracking. The fluent builder APIs (`validate.Component()`, `validate.Workloads()`, `validate.Operator()`) reduce typical check implementations from 50+ lines to 5-10 lines while handling DSC singleton fetching, management state preconditions, and annotation population automatically.

The check applicability system is version-aware: most checks gate on `IsUpgradeFrom2xTo3x()`, meaning they only fire during 2.x→3.x upgrade assessment. The Kueue data integrity check is notable for its sophistication — it builds an ownership graph across Pods, Deployments, StatefulSets, and tracked CRs to validate three consistency invariants for queue-name label propagation across the entire ownership tree.

### Backup Pipeline: Concurrent Dependency Resolution

The backup system uses a three-stage concurrent pipeline (Discovery → Resolver → Writer) with channel-based communication and errgroup-managed worker pools. The dependency resolution is workload-type-aware: the Notebook resolver extracts ConfigMap, Secret, and PVC references from pod spec volumes and container env vars (filtering out trusted-ca-bundle), while the DSPA resolver queries 9 distinct JQ paths for external storage credentials, database passwords, CA bundles, and custom configurations. This pluggable resolver pattern (Strategy pattern via a registry) makes adding new workload types straightforward.

### Resource Type Registry: Centralized GVK/GVR Management

The `pkg/resources/types.go` file (402 lines) serves as the single source of truth for all Kubernetes resource type definitions — 40+ resource types spanning ODH/RHOAI, Kubeflow, KServe, Kueue, Ray, OLM, and core Kubernetes resources. This "Constitution Principle VIII" approach eliminates scattered inline GVR construction and ensures consistency across all commands. The companion `component_crs.go` maps DSC component names (e.g., "aipipelines", "dashboard", "kserve") to their v1alpha1 component CR types for health queries.

### Migration Framework: Two-Phase Execution with Progress Tracking

The migration system (currently implementing Kueue → Red Hat Build of Kueue migration) uses a two-phase Action/Task pattern: `Prepare` backs up ClusterQueues, LocalQueues, and ConfigMaps to disk, while `Run` installs the RHBOK operator via OLM Subscription creation, annotates ConfigMaps to prevent DSC reconciliation, patches the DataScienceCluster managementState, and verifies resource preservation. The hierarchical StepRecorder provides real-time progress output with tree-structured steps, making long-running migrations observable. Dry-run support is integrated at every level.

### Client Design: Higher Throughput for Parallel CLI Operations

The Kubernetes client wrapper (`pkg/util/client`) sets QPS=50 and Burst=100 (10x kubectl defaults) to support parallel backup workers and concurrent health queries without client-side throttling. The Reader/Writer interface split enforces that lint checks can only read (receiving `client.Reader`), while mutations (component enable/disable, migration patches) require the full `Client`. The nil-safe OLM reader pattern gracefully degrades on clusters without OLM installed, returning empty results instead of errors.
