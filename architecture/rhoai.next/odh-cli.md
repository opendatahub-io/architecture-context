## Purpose

**Short**: CLI tool for inspecting, validating, and migrating RHOAI (Red Hat OpenShift AI) deployments on OpenShift clusters.

**Detailed**: odh-cli (shipped as `rhai-cli` / `kubectl-odh`) is a diagnostic and migration CLI that provides RHOAI platform administrators with comprehensive cluster inspection, upgrade readiness validation, and automated migration capabilities. It operates as both a standalone container tool and a kubectl plugin, connecting to Kubernetes API servers to read and (for migrations) mutate cluster state.

The tool's primary subsystems are: (1) a **lint engine** with 38 registered upgrade-readiness checks organized by category (dependency, service, platform, component, workload) that validates whether a cluster can safely upgrade between RHOAI versions; (2) a **migrate engine** with 19 registered migration actions covering AI Pipelines, model serving, Kueue, RayClusters, TrustyAI, workbenches, and LlamaStack that automate pre-upgrade and post-upgrade data and configuration transformations; (3) **inspection commands** (`status`, `get`, `components`, `deps`, `events`, `logs`, `backup`) that provide platform health dashboards, resource listing, dependency graphs, event monitoring, and workload backup capabilities.

The CLI interacts with dozens of Kubernetes API groups spanning the RHOAI platform (opendatahub.io, kubeflow.org, serving.kserve.io, ray.io, trustyai.opendatahub.io, kueue.x-k8s.io, etc.) as well as OpenShift-specific APIs (config.openshift.io, route.openshift.io, operators.coreos.com) and Gateway API (gateway.networking.k8s.io). It embeds a dependency manifest from the odh-gitops repository for offline dependency checking and produces structured output in table, JSON, and YAML formats with embedded JSON Schema support.

## Architectural Analysis

The odh-cli is architected as a **modular CLI framework** built on cobra and the Kubernetes client-go ecosystem. Its design follows several notable patterns that merit discussion.

**Reader/Writer separation for safety**: The client layer (`pkg/util/client/interfaces.go`) enforces a Reader/Writer interface split. Lint checks receive only a `Reader` interface, making it structurally impossible for validation logic to accidentally mutate cluster state. Migration actions receive the full `Client` interface with both read and write capabilities. This separation is a deliberate safety mechanism — the lint subsystem is designed to be run frequently and risk-free, while migration actions require explicit opt-in (`--yes` flag or interactive confirmation).

**Registry-based extensibility**: Both the lint and migrate subsystems use a registry pattern where checks and actions are self-registering Go structs implementing well-defined interfaces (`Check` for lint, `Action` for migrate). Each check/action declares its own metadata (ID, name, group, target version range, applicable conditions). This makes it straightforward to add new checks without modifying framework code — just define a new struct, register it in the appropriate command file, and the framework handles discovery, selection, execution, and output formatting. The lint registry currently holds 38 checks; the migrate registry holds 19 actions.

**Version-aware targeting system**: The lint check system is version-aware at multiple levels. Checks declare the version transitions they apply to (e.g., "2.x→3.x" or "3.4→3.5"), and the framework auto-detects both the current cluster version (from CSV, DSC, or DSCI) and the user-specified target version. The `CanApply` method on each check then evaluates whether it should run, considering component management state, CRD availability, and version predicates. This prevents irrelevant checks from cluttering output and allows the same binary to service multiple upgrade paths.

**Unstructured dynamic client strategy**: Rather than importing typed client libraries for every RHOAI component (which would create massive dependency trees and version coupling), odh-cli uses `k8s.io/client-go/dynamic` with `unstructured.Unstructured` objects throughout. Resource types are defined centrally in `pkg/resources/types.go` as `ResourceType` structs containing Group, Version, Kind, and Resource strings. This design allows the CLI to interact with any CRD version without importing the component's Go types, at the cost of losing compile-time field validation. The centralized resource type registry (`pkg/resources/types.go`) catalogs 40+ resource types spanning 15+ API groups.

**Embedded dependency manifest**: The deps subsystem embeds the odh-gitops Helm chart's `values.yaml` and `Chart.yaml` at build time via `go:embed`. This enables offline dependency checking — the CLI can assess whether required operators are installed and at compatible versions without network access to the gitops repository. The gitops commit is pinned in the Makefile (`ODH_GITOPS_COMMIT`) for reproducible builds.

**Multi-format structured output**: All commands produce output through a unified envelope system (`pkg/output/envelope.go`) using `apiVersion: cli.opendatahub.io/v1` with metadata, spec, and status sections — mirroring the Kubernetes resource convention. The output system supports table (default), JSON, and YAML formats, with JSON Schema generation capabilities for API documentation. This enables both human-readable CLI usage and machine-parseable integration with automation tools.
