## Purpose

**Short**: Downstream packaging repository that wraps the RHAIIS vLLM CUDA image with a TGIS (Text Generation Inference Server) adapter entrypoint for deployment in Red Hat OpenShift AI.

**Detailed**: This repository is a thin downstream packaging layer in the vLLM container image supply chain. It does not contain application source code. Instead, it takes the pre-built `registry.redhat.io/rhaiis/vllm-cuda-rhel9` image — which itself is built from AIPCC (AI Platform Core Components) CUDA base images with the full vLLM inference engine pre-installed — and adds a TGIS adapter entrypoint (`python3 -m vllm_tgis_adapter`). The TGIS adapter provides a gRPC-compatible interface alongside the standard OpenAI-compatible HTTP API, enabling integration with the KServe model serving framework used by RHOAI.

The resulting container image (`rhoai/odh-vllm-cuda-rhel9`) is built by Konflux for both `linux/amd64` and `linux/arm64` architectures. It exposes two network ports: HTTP on port 8000 (OpenAI-compatible API) and gRPC on port 8033 (TGIS protocol). The image runs as non-root user (UID 2000) and is designed to be deployed as an InferenceService runtime container in KServe-managed pods on OpenShift clusters with NVIDIA GPU nodes.

## Architectural Analysis

This repository represents the thinnest possible downstream packaging pattern in the RHOAI ecosystem. The entire repository consists of a single 25-line Dockerfile, a Tekton pipeline definition synchronized from `konflux-central`, and metadata files. No application source code, no deployment manifests, no tests, and no configuration scripts are present. The architectural weight — vLLM engine, PyTorch, CUDA libraries, the TGIS adapter, and hundreds of Python dependencies — lives entirely in the RHAIIS base image (`registry.redhat.io/rhaiis/vllm-cuda-rhel9:3.2.2`).

The key architectural decision visible here is the addition of the TGIS adapter entrypoint. While the RHAIIS base image ships the vanilla vLLM server, this downstream image overrides the entrypoint to `python3 -m vllm_tgis_adapter --uvicorn-log-level=warning`. The TGIS adapter wraps vLLM's HTTP server and adds a gRPC interface (port 8033) implementing the Text Generation Inference Server protocol. This gRPC interface is the primary integration point with KServe's model serving infrastructure in RHOAI, which expects TGIS-protocol-compatible runtimes. The HTTP API (port 8000) provides the OpenAI-compatible REST interface for direct client access.

The `DISABLE_LOGPROBS_DURING_SPEC_DECODING=false` environment variable reveals a product-level ergonomic concern: upstream vLLM disables log probabilities during speculative decoding for performance, but in a hosted inference service context, users expect logprobs to always be available regardless of backend optimization strategy. This is a deliberate divergence from upstream defaults to maintain API contract reliability for RHOAI customers.

The build pipeline produces multi-architecture images (amd64 and arm64) with 10-hour pipeline timeouts, reflecting the large image size and GPU-heavy test requirements. The `hermetic: false` setting in the Tekton pipeline is architecturally correct — this Dockerfile performs no dependency resolution, so hermeticity enforcement is unnecessary at this layer. All supply chain security properties (reproducibility, dependency locking, FIPS compliance) are delegated to the RHAIIS and AIPCC build pipelines.

The absence of ROCm, CPU, Gaudi, and other accelerator variants on the `main` branch is notable. Historical branch names (`rhoai-2.16-gaudi`, `appstudio-vllm-rocm-217`, `appstudio-vllm-cpu-v2-19`) indicate these variants existed on version-specific branches. As the AIPCC ecosystem matures (per overlay 0017), variant management is increasingly centralized in the RHAIIS product build, reducing the need for per-variant branches in this downstream repo.
