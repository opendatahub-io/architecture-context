## Purpose

**Short**: Thin container image wrapper that adapts the RHAIIS vLLM CUDA product image for RHOAI deployment by adding the TGIS gRPC adapter entrypoint.

**Detailed**: This repository produces a single container image (`odh-vllm-cuda-rhel9`) by extending the Red Hat AI Inference Server (RHAIIS) vLLM CUDA product image (`registry.redhat.io/rhaiis/vllm-cuda-rhel9`). The repository contains no application source code — the entire "codebase" is a single Dockerfile that sets environment variables, configures the container to run as a non-root user, and overrides the entrypoint to use the `vllm_tgis_adapter` Python module instead of the default vLLM OpenAI-compatible server.

The vLLM TGIS adapter provides a gRPC interface implementing the Text Generation Inference Server (TGIS) protocol alongside the standard vLLM OpenAI-compatible HTTP API. This enables RHOAI to serve LLM inference requests through both the OpenAI API (HTTP on port 8000) and the TGIS gRPC protocol (on port 8033), maintaining backward compatibility with KServe's model serving infrastructure that expects the TGIS interface. The base image from RHAIIS already contains vLLM, the TGIS adapter, NVIDIA CUDA drivers, and all Python dependencies — this wrapper only configures runtime behavior.

## Architectural Analysis

This repository represents the thinnest possible wrapper pattern in the RHOAI component ecosystem. The entire build artifact is a 24-line Dockerfile that extends an RHAIIS product image (`registry.redhat.io/rhaiis/vllm-cuda-rhel9:3.2.2`) with three changes: setting environment variables for port configuration and speculative decoding behavior, switching to a non-root user (UID 2000), and overriding the entrypoint to use the `vllm_tgis_adapter` Python module. No packages are installed, no source code is compiled, and no files are copied into the image.

The key architectural decision captured here is the entrypoint override: RHAIIS ships vLLM with its default OpenAI-compatible server, but RHOAI needs the TGIS gRPC adapter as the primary entrypoint. The TGIS adapter (`vllm_tgis_adapter`) wraps the vLLM engine and exposes both the TGIS gRPC protocol (port 8033) and the OpenAI-compatible HTTP API (port 8000) simultaneously. This dual-protocol capability is essential for RHOAI because KServe's model serving infrastructure historically uses the TGIS protocol for inference routing, while newer clients expect the OpenAI-compatible API.

The repository's commit history reveals a significant architectural transition: prior to commit `e927dff02` (September 2025), this repo contained the full vLLM source code, build scripts, and dependencies, building vLLM from source in a multi-stage Dockerfile. The "Integrate AIPCC to consume vLLM CUDA from RHAIIS releases" commit replaced all of that with the current thin wrapper pattern, delegating the full build to the RHAIIS/AIPCC ecosystem. This means that vLLM version management, CUDA compatibility, Python dependency resolution, and FIPS compliance are all handled upstream in the RHAIIS build pipeline — this repository's only responsibility is the TGIS adapter configuration.

The Tekton pipeline configuration (synced from `konflux-central`) builds for both `amd64` and `arm64` architectures, with extended timeouts (10h pipeline, 8h tasks) reflecting the large image size. The non-hermetic build flag (`hermetic: false`) is architecturally appropriate since no packages are installed at this layer. The build is triggered by comments (`/build-cuda`) or labels (`kfbuild-all`, `kfbuild-cuda`) on pull requests, indicating this component is part of a multi-component build orchestration system.

The `DISABLE_LOGPROBS_DURING_SPEC_DECODING=false` environment variable override (line 13) is a notable runtime configuration choice: upstream vLLM disables log probabilities during speculative decoding as a performance optimization, but RHOAI overrides this to `false` to ensure users of hosted models always receive log probabilities, regardless of whether speculative decoding is enabled. This reflects RHOAI's priority of API behavior consistency over raw throughput.
