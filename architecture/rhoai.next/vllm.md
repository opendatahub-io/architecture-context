## Purpose

**Short**: Downstream Konflux build wrapper that repackages the RHAIIS vLLM CUDA base image with the TGIS gRPC adapter as the entrypoint for RHOAI model serving.

**Detailed**: This repository is a thin downstream build wrapper — it contains no application source code. Its sole artifact is a `Dockerfile.konflux.cuda` that layers on top of the pre-built `registry.redhat.io/rhaiis/vllm-cuda-rhel9` base image from the Red Hat AI Infrastructure & Services (RHAIIS) product. The Dockerfile sets environment variables, configures the entrypoint to use the `vllm_tgis_adapter` Python module (a gRPC-to-HTTP bridge that exposes vLLM's OpenAI-compatible HTTP API via a TGIS-compatible gRPC interface), and applies Red Hat container labels for OCI compliance.

The actual vLLM inference engine, CUDA runtime, and TGIS adapter code are all provided by the RHAIIS base image. This repository's role is strictly to configure the container entrypoint and metadata for consumption by RHOAI's model serving infrastructure (KServe `ServingRuntime`). The Tekton pipeline in `.tekton/` orchestrates multi-architecture builds (amd64/arm64) via Konflux.

## Architectural Analysis

This repository exemplifies the RHOAI pattern of separating build configuration from application code. The `red-hat-data-services/vllm` repository is not a fork of vLLM — it is a pure build wrapper that produces a redistributable container image. All application logic, including the vLLM inference engine and the TGIS gRPC adapter, is inherited from the RHAIIS base image (`registry.redhat.io/rhaiis/vllm-cuda-rhel9`). This layered approach means that vLLM version updates happen by bumping the `RHAIIS_VERSION` build argument (currently `3.2.2`), and the Dockerfile itself rarely changes.

The TGIS adapter pattern is architecturally significant. vLLM natively exposes an OpenAI-compatible HTTP API on port 8000. The `vllm_tgis_adapter` module, configured as the container entrypoint, adds a gRPC interface on port 8033 that implements the TGIS (Text Generation Inference Server) protocol. This dual-protocol exposure allows the same container to serve both OpenAI-compatible clients (via HTTP) and legacy TGIS clients (via gRPC), which is important for backward compatibility in RHOAI's model serving stack. The `DISABLE_LOGPROBS_DURING_SPEC_DECODING=false` environment variable overrides a vLLM optimization that would break user expectations when speculative decoding is enabled on hosted models.

The build infrastructure is noteworthy: the `.tekton/` directory is auto-synchronized from `konflux-central` (not manually maintained), and builds target both amd64 and arm64 architectures. The pipeline is explicitly non-hermetic (`hermetic: false`), which is acceptable because this Dockerfile performs no package installation — it only sets metadata and entrypoint on a pre-built base image. Security scanning (Clair vulnerability scan, ecosystem-cert-preflight-checks) is performed as part of the Konflux pipeline. The container runs as non-root user 2000, following OpenShift security best practices.

A notable gap is the absence of `argfile.conf` on the `main` branch, despite the Tekton pipeline referencing it as `build-args-file`. This suggests that build arguments (such as `RHAIIS_VERSION`) may be managed differently on release branches, or the file was recently removed. The recent commit history shows the RHAIIS version being bumped to 3.2.2, registry URL fixes, and Konflux pipeline synchronization — all consistent with a build wrapper repository's lifecycle.
