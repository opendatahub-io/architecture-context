## Purpose

**Short**: Container image wrapper that packages the RHAIIS vLLM CUDA runtime with the TGIS (Text Generation Inference Service) gRPC adapter for deployment as an LLM serving runtime on RHOAI.

**Detailed**: This repository is a minimal downstream container image build repo that produces the `odh-vllm-cuda-rhel9` image for Red Hat OpenShift AI. It does not contain application source code — instead, it layers configuration on top of the pre-built `registry.redhat.io/rhaiis/vllm-cuda-rhel9` base image from Red Hat AI Infrastructure Services (RHAIIS).

The image configures vLLM to run with the `vllm_tgis_adapter` Python module as its entrypoint, which exposes both an OpenAI-compatible HTTP API (port 8000) and a gRPC interface (port 8033) for model inference. vLLM itself provides GPU-accelerated LLM inference using NVIDIA CUDA, featuring PagedAttention for efficient memory management, continuous batching, speculative decoding, tensor/pipeline/data parallelism, and support for a wide range of open-source models from HuggingFace.

The build is managed by Konflux via Tekton PipelineRuns defined in the `konflux-central` repository. The pipeline builds multi-architecture images (amd64 and arm64) and publishes them to the RHOAI container registry. This image is consumed by KServe ServingRuntime custom resources to deploy vLLM as a model serving backend on RHOAI clusters.

## Architectural Analysis

This repository exemplifies the RHOAI pattern of separating container image composition from upstream source code. Rather than building vLLM from source, it consumes the pre-built RHAIIS base image (`registry.redhat.io/rhaiis/vllm-cuda-rhel9:3.2.2`) and applies a thin configuration layer. This architecture has several implications:

**Layered image composition**: The Dockerfile is a single-stage, single-`FROM` build with no `COPY`, `RUN`, or `ADD` instructions — it only sets environment variables, the runtime user, and the entrypoint. All application code, Python packages, CUDA libraries, and the TGIS adapter are inherited from the RHAIIS base image. This means the RHAIIS team controls the actual software bill of materials, and this repo controls only the runtime configuration and entrypoint behavior.

**Dual-protocol serving**: The container exposes both an OpenAI-compatible REST API (port 8000) and a TGIS-compatible gRPC interface (port 8033). The entrypoint runs `vllm_tgis_adapter`, which is a Python module bundled in the RHAIIS base image that wraps vLLM's serving engine to provide the TGIS gRPC protocol alongside the standard vLLM HTTP API. The `DISABLE_LOGPROBS_DURING_SPEC_DECODING=false` environment variable overrides a vLLM optimization that would suppress log probabilities during speculative decoding — this is necessary because hosted model users expect logprobs to always be available regardless of internal optimization strategies.

**Security posture**: The container runs as non-root user 2000, which is a standard hardening practice. However, the container itself serves plaintext HTTP and gRPC — TLS termination and authentication are handled externally by the RHOAI platform (kube-rbac-proxy sidecars or Gateway API ingress). This is the expected pattern for RHOAI 3.x serving runtimes, where security is enforced at the platform layer rather than the application layer.

**Build pipeline**: The Tekton pipeline is centrally managed in `konflux-central` and synced to this repo automatically. Builds are multi-architecture (amd64 + arm64), non-hermetic (`hermetic: false`), and run in the `rhoai-tenant` namespace. The pipeline includes Clair vulnerability scanning and ecosystem certification preflight checks, each allocated substantial compute resources (8-16 CPU, 16-32 GiB memory), indicating the base image is large due to CUDA libraries.

**Version coupling**: The `RHAIIS_VERSION=3.2.2` build argument pins the base image version. This is the sole version control mechanism — there are no dependency lock files, no source code versioning, and no semantic versioning of the output image independent of the RHAIIS base. Version upgrades are accomplished by changing this single ARG value, as seen in commit `562821ebc` ("Update vllm-cuda to 3.2.2").
