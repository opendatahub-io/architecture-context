## Purpose

**Short**: GPU-accelerated container image packaging the upstream vLLM inference engine with a TGIS-compatible gRPC adapter for RHOAI model serving.

**Detailed**: This repository builds a container image that wraps the upstream vLLM project — a high-performance LLM inference and serving engine — into a Red Hat-supported CUDA-based container for Red Hat OpenShift AI (RHOAI). The image is built on top of the RHAIIS (Red Hat AI Inference Server) base image `registry.redhat.io/rhaiis/vllm-cuda-rhel9`, which already contains the vLLM runtime, CUDA libraries, and Python dependencies.

The container adds the `vllm_tgis_adapter` as its entry point, which provides a gRPC interface compatible with the Text Generation Inference Server (TGIS) protocol. This adapter enables vLLM to integrate with KServe's model serving infrastructure in RHOAI, where inference requests arrive via both an OpenAI-compatible HTTP API (port 8000) and a TGIS-compatible gRPC API (port 8033). The image is built for both amd64 and arm64 architectures via Konflux multi-arch pipelines.

The repository itself contains no application source code — the entire "source" is a single Konflux Dockerfile that layers configuration and the adapter entry point on top of the pre-built RHAIIS base image. Build configuration, Tekton pipeline definitions, and CI/CD are managed centrally through the `konflux-central` repository.

## Architectural Analysis

This repository follows the **thin wrapper image** pattern common in RHOAI: the repository contains no application source code. Its sole purpose is to layer a specific entry point configuration (`vllm_tgis_adapter` with appropriate defaults) on top of the pre-built RHAIIS base image. All actual vLLM code, CUDA libraries, Python dependencies, and the TGIS adapter are inherited from `registry.redhat.io/rhaiis/vllm-cuda-rhel9:${RHAIIS_VERSION}`.

The architectural significance of this component lies in its dual-protocol serving capability. By using `vllm_tgis_adapter` as the entry point, the container exposes both an OpenAI-compatible HTTP API on port 8000 (vLLM's native API) and a TGIS-compatible gRPC API on port 8033. This dual-protocol design allows RHOAI to support both the modern OpenAI API standard and the legacy TGIS protocol that KServe's model mesh and earlier RHOAI versions depend on. The `DISABLE_LOGPROBS_DURING_SPEC_DECODING=false` environment variable is notable — it overrides a vLLM default optimization to ensure consistent behavior for hosted model users who expect logprobs to be available even when speculative decoding is enabled.

The container runs as non-root user (UID 2000), following RHOAI's security posture. Authentication and TLS termination are explicitly not handled by the container — these are delegated to the platform's ingress infrastructure (RHOAI Gateway with HTTPRoute, kube-rbac-proxy sidecar injection). This separation of concerns means the container focuses purely on inference performance while security is enforced at the platform layer.

The build infrastructure uses Konflux with multi-architecture support (amd64 and arm64), managed centrally via the `konflux-central` repository. The Tekton pipeline runs in the `rhoai-tenant` namespace with significant resource allocations (up to 16 CPU, 32Gi memory) for security scanning tasks (Clair, preflight checks), reflecting the heavyweight nature of GPU runtime images. The `argfile.conf` mechanism (referenced but not present on this branch) allows build-time parameterization without Dockerfile changes.
