## Purpose

**Short**: ML Metadata (MLMD) is a gRPC server that records and retrieves metadata associated with ML workflows, tracking artifacts, executions, and their lineage relationships in a relational database.

**Detailed**: ML Metadata is a library and gRPC service originally developed by Google as part of the TFX (TensorFlow Extended) ecosystem. It provides a metadata store for recording artifacts (datasets, models, metrics), executions (training runs, transformations), and contexts (pipelines, experiments) produced by ML workflows. The relationships between these entities — events, attributions, associations, and parent-child context hierarchies — enable full lineage tracking and provenance queries.

In the RHOAI platform, the MLMD gRPC server (`odh-mlmd-grpc-server`) serves as the backend metadata store for ML pipeline systems. It exposes the `MetadataStoreService` gRPC API on port 8080, backed by MySQL, PostgreSQL, or SQLite databases. The server handles schema migration, optimistic concurrency control, and supports SSL/TLS for both client connections and database backends. The Red Hat fork adds UBI9-based container images built via Konflux and patches for ZetaSQL compatibility.

## Architectural Analysis

ML Metadata follows a straightforward stateless gRPC server architecture. The server binary (`metadata_store_server`) accepts connections on a single gRPC port (default 8080) and delegates all persistence to a pluggable relational database backend. The `MetadataStoreServiceImpl` creates a new database connection for every RPC call rather than maintaining a connection pool (noted as a TODO in the source: `b/123039345`), which may impact performance under high concurrency. Each RPC handler follows the same pattern: connect to the database, execute the operation via the `MetadataStore` library, convert the absl::Status to gRPC status, and return.

The data model is centered on three core entity types — Artifacts, Executions, and Contexts — connected by Events (artifact↔execution), Attributions (artifact↔context), Associations (execution↔context), and ParentContext relationships. Each entity type is further categorized by user-defined or system-defined Types that define a schema of typed properties. This graph structure enables rich lineage queries through the `GetLineageSubgraph` API, which performs configurable graph traversal with depth limits, direction control, and ending conditions.

The database layer supports three backends: SQLite (for development/testing), MySQL/MariaDB, and PostgreSQL. Each backend has its own query configuration defined in `metadata_source.proto`, with parameterized SQL templates and a migration system that supports both upgrade and downgrade across schema versions. The migration system compares the stored schema version in the `MLMDEnv` table with the library's schema version and applies incremental migrations. The MariaDB client library source is copied into the built image's `third_party` directory for licensing compliance.

The Red Hat fork (`red-hat-data-services/ml-metadata`) primarily adds downstream container image builds. The Konflux Dockerfile (`Dockerfile.konflux`) builds on UBI9 and produces the `odh-mlmd-grpc-server` image with digest-pinned base images. The Tekton pipeline definitions in `.tekton/` configure Konflux CI/CD with the `Dockerfile.redhat` variant for the push pipeline, targeting `quay.io/opendatahub/mlmd-grpc-server:odh-stable`. The server runs as non-root user 65534:65534 in production. Notably, the ZetaSQL dependency requires a specific commit patch for UBI9 compatibility, which is applied either in the Dockerfile.redhat (via sed) or pre-configured in the WORKSPACE file.

A significant architectural consideration is the lack of built-in authentication beyond optional mTLS. The server relies entirely on network-level isolation or an upstream proxy (such as kube-rbac-proxy in RHOAI) for access control. There is no per-RPC authorization — any client with network access (and valid TLS credentials if configured) can perform any operation including bulk deletes and schema modifications.
