## Purpose

**Short**: A gRPC server that records and retrieves metadata (artifacts, executions, contexts, and lineage) associated with ML workflows.

**Detailed**: ML Metadata (MLMD) is a library and server originally developed by Google as part of the TFX (TensorFlow Extended) ecosystem. It provides a centralized metadata store for tracking the lifecycle of ML artifacts (datasets, models, metrics), executions (training runs, transformation steps), and contexts (pipelines, experiments). The server exposes a comprehensive gRPC API for CRUD operations on these entities plus lineage graph traversal.

In the RHOAI platform, MLMD serves as the persistence layer for ML pipeline metadata. It is deployed as a standalone gRPC server (`metadata_store_server`) that connects to a relational database backend (MySQL, PostgreSQL, or SQLite). Pipeline orchestrators (such as Kubeflow Pipelines / Data Science Pipelines) use the MLMD gRPC API to record artifact provenance, execution state, and lineage relationships. The server is built from C++ source using Bazel and packaged as a container image (`odh-mlmd-grpc-server`) on UBI9.

The repository is a downstream fork of `google/ml-metadata` maintained by Red Hat Data Services. It adds Red Hat-specific Dockerfiles (UBI-based, Konflux-compatible), Tekton pipeline definitions for CI/CD, and Renovate-managed base image updates. The upstream C++ and protobuf code is preserved largely unchanged.

## Architectural Analysis

ML Metadata is a stateless gRPC server that creates a new database connection per RPC call. Every method in `MetadataStoreServiceImpl` follows an identical pattern: connect to the database, execute the operation, return the result. There is no connection pooling — the code includes a TODO comment (`b/123039345`) acknowledging this gap. While this simplifies concurrency (the service implementation notes that concurrent calls are serialized), it introduces latency and connection overhead per request in production deployments with external databases.

The server supports three database backends (MySQL, PostgreSQL, SQLite) with SQL dialect abstraction through `MetadataSourceQueryConfig`, which contains parameterized SQL templates for each operation. Schema migration is built into the server startup: it compares the library's expected schema version against the database's stored version and can automatically upgrade (or downgrade) the schema. This migration system is critical for RHOAI upgrades where the MLMD version may change between platform releases.

The Konflux Dockerfile builds the C++ binary using Bazel on UBI9, producing a single statically-linked `metadata_store_server` binary that is then copied into a minimal UBI9 runtime image. The binary runs as non-root (UID 65534) and exposes a single gRPC port. The Tekton pipelines build for three architectures (x86_64, arm64, ppc64le), reflecting RHOAI's multi-architecture requirements. Build hermeticity is not fully enforced on this branch — `hermetic: false` is set in the PipelineRun, and Bazel fetches dependencies at build time rather than from pre-fetched sources.

TLS for the gRPC endpoint is fully configurable via the `MetadataStoreServerConfig.SSLConfig` protobuf, supporting server-side TLS with optional mutual TLS (client certificate verification). However, the default configuration uses `InsecureServerCredentials()` — in RHOAI deployments, the platform operator typically handles TLS termination or configures the SSLConfig appropriately. Database connections also support SSL/TLS independently via `MySQLDatabaseConfig.SSLOptions` and `PostgreSQLDatabaseConfig.SSLOptions`, allowing end-to-end encryption from gRPC clients through to the database.

The protobuf API is extensive (46 RPCs) and covers the complete lifecycle of ML metadata: type registration, artifact/execution/context CRUD with optimistic concurrency control, event recording with typed relationships (INPUT, OUTPUT, DECLARED_INPUT, etc.), hierarchical context management, and lineage graph traversal with configurable depth, direction, and filtering. The `ListOperationOptions` proto supports SQL-like filtering syntax for complex queries on node attributes, properties, and relationships — this is powered by the ZetaSQL dependency for query parsing.
