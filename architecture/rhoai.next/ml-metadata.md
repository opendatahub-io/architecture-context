## Purpose

**Short**: ML Metadata (MLMD) is a gRPC server and client library for recording and retrieving metadata associated with ML workflows, including artifacts, executions, contexts, and their lineage relationships.

**Detailed**: ML Metadata provides a metadata store service that tracks the lifecycle of ML artifacts, executions, and contexts. It is originally a Google project (google/ml-metadata) and is forked and adapted by Red Hat for use as a component in the RHOAI (Red Hat OpenShift AI) platform, specifically as part of the ML pipelines infrastructure. The MLMD gRPC server acts as the metadata backbone for Kubeflow Pipelines, storing information about pipeline runs, artifacts produced and consumed by pipeline steps, and the lineage relationships between them.

The repository contains both a C++ gRPC server binary (`metadata_store_server`) and a Python client library (`ml-metadata` PyPI package). The server supports multiple database backends including MySQL, PostgreSQL, and SQLite, and exposes a rich gRPC API defined in Protocol Buffers for CRUD operations on artifacts, executions, contexts, events, and lineage graphs. For RHOAI, the server is built using Bazel on UBI9 and packaged as a minimal container image using a multi-stage Dockerfile built by Konflux.

## Architectural Analysis

ML Metadata follows a classic three-tier architecture: a gRPC service layer, a business logic layer (MetadataStore), and a pluggable database access layer supporting MySQL, PostgreSQL, and SQLite backends. The C++ server binary (`metadata_store_server_main.cc`) uses gflags for configuration, supporting both a protobuf text config file (`--metadata_store_server_config_file`) and individual CLI flags for database connection parameters (`--mysql_config_*`, `--postgres_config_*`). The `--metadata_source_config_type` flag controls which backend is used (default, config_file, mysql, postgresql).

The gRPC service implementation (`MetadataStoreServiceImpl`) creates a fresh database connection per RPC call, which means there is no connection pooling at the service layer. Each method follows the same pattern: connect to the database, delegate to the `MetadataStore` business logic, convert the absl::Status to grpc::Status, and return. This design ensures thread-safety at the cost of connection overhead. A TODO in the code (b/123039345) notes that a connection pool should be added.

The data model is centered on three node types (Artifact, Execution, Context) connected by three edge types (Event, Attribution, Association) plus a ParentContext hierarchy. Each node type has a corresponding Type schema (ArtifactType, ExecutionType, ContextType) with typed properties. This design closely mirrors the W3C PROV data model and is optimized for ML pipeline lineage tracking. The server supports schema migration via `MigrationOptions`, allowing both upgrade and downgrade of the database schema across library versions.

For RHOAI, this component is built using the Konflux CI system (`Dockerfile.konflux`) targeting multi-architecture (x86_64, arm64, ppc64le). The builder stage uses UBI9 with Bazel 5, and the runtime image uses UBI9-minimal with only the compiled server binary. The server runs as non-root (UID 65534:65534) for security. The Tekton PipelineRun configurations in `.tekton/` define both pull-request and push pipelines, with the RHOAI build producing images to `quay.io/rhoai/pull-request-pipelines` and the ODH build to `quay.io/opendatahub/mlmd-grpc-server`. Environment variables `GRPC_PORT` (default 8080) and `METADATA_STORE_SERVER_CONFIG_FILE` control the server configuration at runtime.

The Python client library supports both direct database connections (via pybind11-wrapped C++ code) and gRPC connections to a remote server. It includes retry logic with exponential backoff for transient Aborted errors, configurable via `RetryOptions.max_num_retries`. The client supports all 40+ gRPC methods and additionally provides list pagination helpers via `ListOptions` with filtering (`filter_query`), ordering, and page-size control.
