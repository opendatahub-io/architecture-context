## Purpose

**Short**: ML Metadata (MLMD) is a gRPC server and client library for recording and retrieving metadata associated with ML workflows, tracking artifacts, executions, contexts, and their lineage relationships.

**Detailed**: ML Metadata is a Google-originated library that provides a structured metadata store for machine learning pipelines. It defines a data model centered on three core entity types -- Artifacts (data objects like datasets and models), Executions (pipeline steps or tasks), and Contexts (groupings like pipeline runs) -- along with Events that link artifacts to executions and lineage graphs that trace data provenance. The server exposes a comprehensive gRPC API (MetadataStoreService) with ~45 RPCs for creating, querying, and managing these entities.

In the RHOAI platform, MLMD serves as the backend metadata store for Kubeflow Pipelines (KFP). Pipeline runs record their artifacts, executions, and lineage through MLMD's gRPC interface, enabling experiment tracking, artifact lineage tracing, and pipeline provenance. The RHOAI distribution ships a Konflux-built container image (`odh-mlmd-grpc-server`) based on UBI9, which compiles the C++ gRPC server binary from source using Bazel and exposes it on port 8080.

The component also provides a Python client library (`ml-metadata` PyPI package) that can connect to the gRPC server or directly to a database backend (MySQL, PostgreSQL, or SQLite) via a C++ pybind11 extension module. The Python client is used by KFP SDK and TFX for programmatic metadata access.

## Architectural Analysis

ML Metadata follows a clean separation between its gRPC transport layer and its storage layer. The `MetadataStoreServiceImpl` class implements each gRPC RPC by creating a fresh database connection per request (via `ConnectMetadataStore`), executing the corresponding `MetadataStore` method, and returning the result. This per-request connection model is simple but creates a new connection for every call, which the code acknowledges as a known limitation with a TODO for connection pooling (`b/123039345`). In production, this means the database connection pool behavior is entirely determined by the underlying database driver, not by MLMD itself.

The server supports three database backends -- MySQL, PostgreSQL, and SQLite -- abstracted through a `MetadataSource` interface with backend-specific query configurations defined in `MetadataSourceQueryConfig`. The PostgreSQL support uses a patch-based approach with ZetaSQL SQL dialect parsing, and the Dockerfile.redhat includes environment variable patching for UBI9 compatibility with the ZetaSQL commit hash. The Konflux Dockerfile does not perform this patching, suggesting the WORKSPACE already uses the UBI9-compatible ZetaSQL commit on the main branch.

The gRPC API is designed around optimistic concurrency control (OCC) for concurrent updates, using `last_update_time_since_epoch` comparison. The `PutExecution` and `PutLineageSubgraph` RPCs provide atomic multi-entity operations that are critical for pipeline systems -- they allow recording an execution along with its input/output artifacts, events, and context associations in a single transaction. The API also supports field-mask-based partial updates for artifacts, executions, and contexts, which enables efficient incremental updates.

From a security perspective, MLMD has no built-in authentication or authorization. It supports optional TLS via a `SSLConfig` protobuf message that configures server certificates and optionally requires client certificates (mTLS). However, when deployed in RHOAI, MLMD typically runs without TLS as an in-cluster service, relying on Kubernetes network policies and the Data Science Pipelines Operator for access control. The server runs as a non-root user (UID 65534) in the Konflux container image. Database credentials are passed via command-line flags or a protobuf config file, with no secrets management integration. The `GRPC_PORT` and `METADATA_STORE_SERVER_CONFIG_FILE` environment variables control the server's runtime configuration.

The build uses Bazel with all C++ dependencies pinned by SHA256 in the WORKSPACE file, providing strong reproducibility for the C++ layer. However, the Konflux Dockerfile installs OS packages via `dnf` without lock files, and the BoringSSL dependency is not built in FIPS mode, which creates a potential compliance gap for FIPS-required deployments.
