## Purpose

**Short**: A centralized library of reusable Kubeflow Pipelines (KFP) components and pre-composed ML pipelines for data processing, training, evaluation, and model deployment workflows in Red Hat OpenShift AI.

**Detailed**: The pipelines-components repository provides a curated collection of KFP v2 components and pipelines organized by ML lifecycle stage: data processing, training, evaluation, and deployment. Components are individual, reusable building blocks decorated with `@dsl.component`, while pipelines compose multiple components into end-to-end ML workflows decorated with `@dsl.pipeline`.

The repository ships three Konflux-built container images: `odh-automl` (AutoGluon tabular/timeseries training), `odh-autorag` (RAG optimization with document extraction, indexing, and evaluation), and `odh-pipelines-components` (an init container that compiles managed pipelines at build time and copies pre-compiled pipeline YAMLs to a shared volume at runtime). The init container (`odh-pipelines-components`) generates a `managed-pipelines.json` manifest and pre-compiled pipeline YAML specs during the image build, then at container startup copies them to `/config/managed-pipelines/` for consumption by the KFP API server.

The component library spans multiple ML domains: LLM fine-tuning (LoRA, SFT, OSFT), AutoML (AutoGluon tabular and timeseries), RAG optimization (document discovery, extraction, indexing, and template optimization via ai4rag/OGX), synthetic data generation (SDG Hub), model evaluation (lm-eval with vLLM backend), and model registration (Kubeflow Model Registry). All components are designed for execution on Kubeflow Pipelines within an OpenShift AI cluster, with PVC-based workspace sharing between pipeline stages.

## Architectural Analysis

The pipelines-components repository follows a clear **component-pipeline separation pattern** where individual KFP components (`@dsl.component`) serve as atomic building blocks and pipelines (`@dsl.pipeline`) compose them into end-to-end workflows. This architecture enables maximum reuse: the `dataset_download` component is shared across all fine-tuning pipelines (LoRA, SFT, OSFT), and the `documents_discovery` and `text_extraction` components are shared between the indexing and optimization AutoRAG pipelines.

The **image architecture** is notable for its runtime injection pattern. Components reference base images via environment variables (`RELATED_IMAGE_ODH_AUTOML_IMAGE`, `RELATED_IMAGE_ODH_AUTORAG_IMAGE`) with fallback defaults. This enables the RHOAI operator to inject production-grade, Konflux-built images at deployment time while allowing developers to use upstream defaults during development. The two specialized task images (`odh-automl`, `odh-autorag`) bundle heavy ML dependencies (AutoGluon, docling models) that would be impractical to install at pipeline runtime, while simpler components use the default KFP image or inline `packages_to_install`.

The **managed pipeline delivery mechanism** is architecturally interesting. The `odh-pipelines-components` image serves as a build-time pipeline compiler and runtime init container. During the Docker build, `generate_managed_pipelines.py` discovers all pipelines with `managed: true` in their metadata, compiles them via the KFP compiler, and produces both compiled YAML specs and a JSON manifest. At runtime, `init_managed_pipelines.py` simply copies these pre-compiled artifacts to `/config/managed-pipelines/` on a shared volume. This two-phase approach ensures pipelines are validated at build time (compilation errors break the image build) and are available immediately at pod startup without runtime compilation.

The **fine-tuning pipeline family** (LoRA, SFT, OSFT and their minimal variants) shares a consistent 4-stage pattern: dataset download → training → evaluation → model registry. Each variant uses a different training backend (unsloth for LoRA, instructlab-training for SFT with FSDP multi-GPU support, mini-trainer for OSFT) but shares the same dataset download, evaluation, and model registry components. The minimal variants reduce the parameter surface for simpler use cases while maintaining the same pipeline topology.

The **AutoRAG subsystem** is the most complex, spanning 5 pipeline stages with dependencies on external services (OGX API, Milvus, S3). It uses the ai4rag framework for iterative RAG template optimization, evaluating combinations of embedding models, generation models, chunking strategies, and distance metrics. The `prepare_responses_api_requests` component generates ready-to-use OGX `/v1/responses` API request bodies from optimized RAG patterns, bridging the gap between optimization results and deployment.

A notable **security consideration** is that the AutoRAG components support a `OGX_TLS_INSECURE=1` environment variable for development, which disables TLS certificate verification. This is documented in the code but could be a risk if accidentally enabled in production. The `InsecureSkipVerify` pattern appears in the `documents_indexing` component's SSL fallback logic (lines 66-82), where it catches SSL errors and retries with relaxed verification.
