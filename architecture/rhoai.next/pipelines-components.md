## Purpose

**Short**: A centralized Python library of reusable Kubeflow Pipelines (KFP) v2 components and pre-assembled pipelines for AI/ML workflows on Red Hat OpenShift AI.

**Detailed**: The pipelines-components repository provides a structured collection of KFP v2 components and pipelines organized into four functional categories: data processing, training, evaluation, and deployment. Components are defined as Python functions decorated with `@dsl.component` or `@dsl.container_component` and compiled into pipeline YAML specifications that Kubeflow Pipelines executes as containerized steps on Kubernetes.

The repository produces three container images via Konflux: (1) `odh-pipelines-components` — an init container that stages compiled managed pipeline YAMLs into a shared volume for the KFP API server, (2) `odh-automl` — a runtime image bundling the kfp-components wheel with AutoGluon dependencies for AutoML tabular and timeseries training, and (3) `odh-autorag` — a runtime image bundling document processing dependencies (Docling, ai4rag, OGX client) with pre-embedded ML models for RAG optimization workflows. These images are consumed by the RHOAI Pipelines operator to provide managed pipeline capabilities out of the box.

The component library covers the complete ML lifecycle: downloading datasets from S3/HuggingFace, fine-tuning LLMs with LoRA/OSFT/SFT, automated ML with AutoGluon, document-based RAG optimization with AI4RAG, model evaluation with lm-eval and Eval Hub, model deployment via KServe InferenceService, and model registration with Kubeflow Model Registry. Each component manages its own Kubernetes interactions (secret injection, PVC mounts, KServe CRD operations) while the pipelines compose them into end-to-end workflows with shared workspaces.

## Architectural Analysis

The pipelines-components repository follows a modular architecture where each KFP component is a self-contained Python function that encapsulates its own Kubernetes interactions. This design means that complex infrastructure operations — creating KServe InferenceServices, submitting RayJobs, polling for training completion — are embedded within individual component code rather than managed by an external controller. The trade-off is that each component needs its own error handling, timeout logic, and cleanup routines. The `evalhub_evaluator_kserve` component exemplifies this pattern well: it creates ephemeral KServe resources, polls for readiness, submits evaluation jobs, and cleans up in a `finally` block — all within a single `@dsl.component` function.

The image architecture uses a three-tier base image strategy. The `odh-pipelines-components` image uses a plain UBI9 Python 3.12 base and runs only at init time to stage pipeline YAMLs. The `odh-automl` and `odh-autorag` images use AIPCC CPU base images (`registry.redhat.io/rhai/base-image-cpu-rhel9`) which provide pre-configured pip access to the RHEL AI Python Package Index. The autorag image additionally embeds Docling ML model weights from two modelcar images at build time, enabling offline document text extraction without downloading models at runtime. The `RELATED_IMAGE_*` environment variable pattern allows the RHOAI operator to override base images at deployment time, which is critical for disconnected/air-gapped environments.

A noteworthy pattern is the `component_stage_map_publisher` component used in both AutoML and AutoRAG pipelines. This component runs first in a pipeline and publishes a JSON manifest describing the expected pipeline stages and their relationships. RHOAI dashboards consume this artifact to render pipeline progress before all stages have executed. This is an interesting inversion — rather than the dashboard discovering pipeline structure from KFP metadata, the pipeline explicitly declares its UI contract.

The finetuning pipelines (LoRA, OSFT, SFT) share a common four-stage structure (dataset download → training → evaluation → model registry) but use different training backends: unsloth for LoRA, mini-trainer for OSFT, and instructlab-training for SFT. Each variant has a "minimal" version that omits the evaluation stage, and a "finetuning_evalhub" version that replaces lm-eval with Eval Hub + KServe-based evaluation. This combinatorial explosion (3 methods × 3 variants = 9 distinct pipeline definitions) suggests an opportunity for pipeline composition refactoring, though the current approach ensures each pipeline is independently compilable and deployable.

The SSL `verify=False` fallback pattern appears across many data processing components as a convenience for development environments with self-signed certificates. While not a production security risk (it's a retry fallback, not the default), it represents a gap that could be addressed with proper CA bundle injection via Kubernetes Secrets or ConfigMaps. Similarly, the Milvus integration uses plaintext gRPC (port 19530) which may be acceptable for in-cluster communication but should be documented as a network security consideration.
