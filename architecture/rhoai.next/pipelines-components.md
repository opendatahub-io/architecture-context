## Purpose

**Short**: A centralized library of reusable Kubeflow Pipelines (KFP) v2 components and pre-composed pipelines for AI/ML workflows including AutoML, AutoRAG, fine-tuning, evaluation, and model deployment on Red Hat OpenShift AI.

**Detailed**: The pipelines-components repository provides a structured collection of KFP v2 components (individual workflow steps) and pre-composed pipelines (multi-step workflows) that cover the full AI/ML lifecycle. Components are organized into four categories: data processing (dataset download, S3 interaction, synthetic data generation, document discovery/extraction/indexing), training (AutoGluon tabular/timeseries AutoML, AutoRAG optimization, LoRA/OSFT/SFT fine-tuning), evaluation (lm-eval harness integration), and deployment (Kubeflow Model Registry registration, OpenAI Responses API request generation).

The repository ships three Konflux-built container images: (1) the main `pipelines-components` init container that compiles managed pipelines at build time and copies them to a shared volume at runtime for the KFP API server to discover, (2) an `odh-automl` runtime image with AutoGluon and ML dependencies pre-installed for AutoML pipeline execution, and (3) an `odh-autorag` runtime image with Docling, LangChain, and ChromaDB pre-installed for AutoRAG document processing. The repository also functions as an installable Python package (`kfp-components`) that users can import to compose custom pipelines from the component library.

## Architectural Analysis

The pipelines-components repository follows a modular, category-based organization pattern aligned with the AI/ML lifecycle. Components are individual KFP v2 steps defined as Python functions decorated with `@dsl.component` or `@dsl.container_component`, while pipelines compose multiple components into end-to-end workflows using `@dsl.pipeline`. Each component and pipeline carries a `metadata.yaml` file that declares stability level (experimental/alpha/beta/stable), dependencies (KFP SDK version, external services), and management status. This metadata-driven approach enables automated validation, discovery, and lifecycle management through the extensive CI/CD tooling in the `scripts/` directory.

The repository employs a "managed pipelines" pattern for RHOAI integration: pipelines marked `managed: true` in their metadata are compiled at image build time by `generate_managed_pipelines.py`, which discovers them via directory traversal, validates metadata, compiles Python pipeline definitions into KFP YAML using the KFP compiler, and generates a `managed-pipelines.json` manifest. At runtime, the init container (`init_managed_pipelines.py`) copies these compiled YAMLs to a shared volume at `/config/managed-pipelines/`, where the KFP API server discovers and registers them automatically. This design avoids runtime compilation and ensures pipeline definitions are immutable and reproducible.

The base image governance system is notable: `utils/consts.py` defines environment-overridable image references (`RELATED_IMAGE_MPI_AUTOML_RUNTIME`, `RELATED_IMAGE_MPI_AUTORAG_RUNTIME`) following the Kubernetes operator pattern for image management. The `base_image_allowlist.yaml` restricts which container registries may be used (Red Hat registry, Kubeflow GHCR, OpenDataHub Quay), and `validate_base_images.py` enforces this during CI. The `override_base_images.py` script uses Python AST analysis to surgically replace image tags in source files during release builds, ensuring consistent versioning without manual edits. This multi-layer image governance prevents supply chain risks while supporting both ODH (community) and RHOAI (enterprise) distribution channels.

Fine-tuning pipelines (LoRA, OSFT, SFT) share a common 4-stage architecture: (1) dataset download supporting multiple protocols (HF Hub, S3, HTTPS, PVC, OCI), (2) model training using pluggable backends (unsloth for LoRA, mini-trainer for OSFT, instructlab for SFT), (3) evaluation via lm-eval harness, and (4) optional model registration in Kubeflow Model Registry. Shared utilities in `components/training/finetuning/shared/` handle dataset resolution, OCI model downloads, output persistence, and training setup. Each pipeline offers both a full variant (70+ configurable parameters) and a minimal variant (reduced parameters with sensible defaults) for quick testing. All fine-tuning pipelines use 50Gi PVCs with nfs-csi storage class and ReadWriteMany access to support multi-pod training, while AutoML pipelines use smaller 12Gi PVCs with ReadWriteOnce access due to their single-writer model.

The dual-distribution build system (ODH and RHOAI) is managed through separate Tekton pipeline definitions in `.tekton/`. RHOAI builds use Konflux with hermetic builds enabled, dependency prefetch from Red Hat's managed PyPI mirror (`console.redhat.com/api/pypi/public-rhai/rhoai/3.4/cpu-ubi9/simple/`), and Hermeto generic fetchers for non-PyPI artifacts (Docling ML models, SQLite sources). ODH builds use the same Konflux infrastructure but push to different registries (`quay.io/opendatahub/` vs `quay.io/rhoai/`) and may use non-Konflux Dockerfiles. Multi-architecture support spans x86_64, arm64, ppc64le, and s390x, with platform-specific dependency handling (e.g., triton excluded on s390x).
