## Purpose

**Short**: A centralized library of reusable Kubeflow Pipelines (KFP) components and managed pipelines for ML workflows including fine-tuning, AutoML, RAG optimization, and data processing on Red Hat OpenShift AI.

**Detailed**: The pipelines-components repository provides a structured collection of KFP v2 Python components and complete pipeline definitions organized across four functional categories: data processing, training, evaluation, and deployment. Components are designed as modular building blocks that compose into end-to-end ML pipelines — for example, a supervised fine-tuning pipeline chains dataset download → model training → evaluation → model registry steps, while a RAG optimization pipeline chains document discovery → text extraction → embedding → vector indexing → hyperparameter search.

The repository produces three container images via Konflux: a primary `odh-pipelines-components` init container that pre-compiles managed pipeline YAML manifests and stages them for the KFP API server, an `odh-automl` image bundling AutoGluon-based tabular and time-series training components, and an `odh-autorag` image bundling RAG pipeline components with embedded Docling models for offline text extraction. At runtime, the init container copies compiled pipeline specs to a shared volume so that the Data Science Pipelines Operator (DSPO) can register managed pipelines in the KFP API. Components themselves execute as KFP task pods using configurable base images, with Kubernetes-native secret injection, PVC mounts, and GPU scheduling.

## Architectural Analysis

The pipelines-components repository follows a clean modular architecture where individual KFP components are composed into complete pipelines through Python imports and DAG declarations. The four-category organization (data_processing, training, evaluation, deployment) maps directly to the stages of a typical ML lifecycle, and the consistent `@dsl.component` / `@dsl.pipeline` decorator pattern makes adding new pipeline variants straightforward — the six finetuning pipeline variants (SFT/LoRA/OSFT × full/minimal, plus Eval Hub variants) demonstrate this composability.

The managed pipeline system is particularly noteworthy. The `generate_managed_pipelines.py` script discovers pipelines with `managed: true` metadata, compiles them to KFP YAML at image build time, and packages them into the init container. At runtime, `init_managed_pipelines.py` checks for `RELATED_IMAGE_*` environment variables — if present (as set by DSPO), it recompiles all managed pipelines with overridden base images to match the operator's configured container digests. This two-phase approach (prebuilt for speed, recompile for customization) is a pragmatic design that avoids shipping a full Python SDK in the KFP API server while still supporting operator-managed image pinning.

A security concern is the widespread use of `verify=False` SSL fallback in S3 and OGX client connections. While this accommodates self-signed certificates common in development/air-gapped environments, it silently disables certificate chain validation in production. The pattern — try with verification, catch SSLError, retry without — means that a man-in-the-middle attack would succeed on the second attempt. The Model Registry client also communicates over plaintext HTTP by default, which is a data-in-transit risk for model metadata including training hyperparameters and evaluation metrics. Both patterns should be reviewed for production hardening.

The AutoML and AutoRAG images use AIPCC CPU base images with the RHEL AI PyPI index, which is the correct pattern for RHOAI components. However, the primary `odh-pipelines-components` image uses a standard UBI9 Python image and pins packages from the RHEL AI PyPI via `--index-url` in requirements.txt. This achieves similar dependency sourcing but lacks the full AIPCC environment metadata and tooling (`/etc/rhaipcc/env`, `/usr/libexec/rhaipcc/dnf`). The AutoRAG image additionally bundles Docling ML models from RHAI modelcar images (`registry.redhat.io/rhai/modelcar-docling-project-*`) for offline text extraction, avoiding runtime downloads in air-gapped environments.

The pipeline parameter naming convention (`phase_NN_component_[man|opt]_param_name`) is a strong usability pattern that groups parameters by lifecycle stage and communicates mandatory vs. optional clearly to users. Combined with universal caching disabled (`set_caching_options(False)`) and explicit resource constraints on every task, the pipelines are designed for production reliability rather than development convenience.
