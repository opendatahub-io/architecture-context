## Purpose

**Short**: A centralized library of reusable Kubeflow Pipelines (KFP) components and end-to-end pipelines for AI/ML workflows on Red Hat OpenShift AI, covering data processing, model training (AutoML, AutoRAG, fine-tuning), evaluation, and deployment.

**Detailed**: The pipelines-components repository implements the KFP component library for RHOAI's managed pipeline experience. It provides over 25 reusable KFP components organized into four categories (data processing, training, evaluation, deployment) and 11 end-to-end pipelines that compose these components into complete AI workflows. Components are Python functions decorated with `@dsl.component` that run as containerized steps in Kubeflow Pipelines, each with standardized metadata, input/output contracts, and base image configurations.

The repository produces three Konflux-built container images: `odh-pipelines-components` (the primary init container that compiles and stages managed pipeline YAMLs for the KFP API server), `odh-automl` (a runtime image with AutoGluon dependencies for AutoML tabular and time-series components), and `odh-autorag` (a runtime image with ai4rag, Docling, and Llama Stack dependencies for RAG optimization components). The init container runs at platform startup to populate the KFP API server's managed pipelines directory, making curated pipelines available to users out of the box.

The component library integrates with multiple external services including AWS S3 (data storage), HuggingFace Hub (datasets and models), Llama Stack (RAG vector stores and inference), Kubeflow Model Registry (model versioning), and the Kubeflow Trainer API (distributed training jobs on Kubernetes). Fine-tuning components support LoRA (via Unsloth), OSFT (via mini-trainer), and SFT (via InstructLab Training) backends with GPU-accelerated distributed training.

## Architectural Analysis

The pipelines-components repository follows a library-first architecture where reusable KFP components are the atomic building blocks, and pipelines are compositions of these components. The project structure enforces a strict category taxonomy (data_processing, training, evaluation, deployment) with each component containing a standardized set of files: `component.py` (the `@dsl.component`-decorated function), `metadata.yaml` (standardized metadata with stability classification and dependency declarations), and optional `example_pipelines.py`. This structure is enforced by automated validation scripts and skeleton generators (`make component`, `make pipeline`) that produce consistent scaffolding.

A significant architectural decision is the "managed pipeline" system. Pipelines with `managed: true` in their metadata.yaml are automatically compiled to KFP YAML by the `generate_managed_pipelines.py` script during the `odh-pipelines-components` container build. The init container (`init_managed_pipelines.py`) then copies these compiled YAMLs to a shared volume that the KFP API server reads from. This creates a "batteries-included" experience where users see curated, pre-built pipelines without needing to write or compile pipeline code themselves. The stability classification (alpha/beta/stable, mapped to "Development Preview"/"Technology Preview"/"General Availability") provides lifecycle management for these managed pipelines.

The fine-tuning subsystem (LoRA, OSFT, SFT) is notably complex. Rather than running training directly in the KFP pod, these components use the Kubeflow Trainer API to submit distributed TrainingJobs to the cluster. The component pod acts as a control plane — initializing the Kubernetes client, submitting the TrainingJob CR, streaming logs from worker pods, and waiting for completion. This two-level indirection (KFP step -> TrainingJob -> worker pods) enables multi-GPU and multi-node training while keeping the pipeline orchestration lightweight. Each fine-tuning variant (LoRA via Unsloth, OSFT via mini-trainer, SFT via InstructLab Training) follows the same pattern but with different training backends and hyperparameter surfaces.

The component image strategy uses a three-tier approach: (1) the `odh-pipelines-components` init container uses a minimal UBI9/python-311 base with only kfp and utility dependencies, (2) the `odh-automl` image uses `base-image-cpu-rhel9` with heavy AutoGluon/pandas/scikit-learn packages pre-installed, and (3) the `odh-autorag` image uses the same base with ai4rag, Docling, and Llama Stack packages plus pre-cached ML model weights. Components reference their base image via the `AUTOML_IMAGE` and `AUTORAG_IMAGE` constants from `utils/consts.py`, which support environment variable overrides (`RELATED_IMAGE_MPI_AUTOML_RUNTIME`, `RELATED_IMAGE_MPI_AUTORAG_RUNTIME`) for operator-managed image substitution — a standard RHOAI pattern for disconnected/air-gapped deployments.

A notable security concern is the pervasive SSL verification fallback pattern across multiple components. When boto3 or httpx encounters an SSL error (typically self-signed certificates in development environments), the code catches the exception and retries with `verify=False`. While this is a pragmatic development convenience, it means components in non-production environments operate with weakened TLS. The `build_responses_request_bodies` component generates scripts that properly handle custom CA bundles via `REQUESTS_CA_BUNDLE` and `SSL_CERT_FILE` environment variables, but the runtime components themselves rely on the catch-and-retry fallback rather than proactive CA bundle configuration.
