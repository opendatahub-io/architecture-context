## Purpose

**Short**: A comprehensive platform for Models as a Service (MaaS) that provides subscription-based access control, API key management, token rate limiting, and policy-driven governance for LLM inference services on OpenShift.

**Detailed**: Models as a Service (MaaS) implements a multi-tenant governance layer on top of KServe LLMInferenceService and external model providers. It consists of two primary components: a Kubernetes operator (`maas-controller`) that reconciles custom resources representing tenants, model references, subscriptions, and authentication policies; and a REST API server (`maas-api`) that provides OpenAI-compatible API key management, model discovery, and subscription selection endpoints.

The operator dynamically creates Kuadrant AuthPolicy and TokenRateLimitPolicy resources to enforce per-model authentication and rate limiting through the Gateway API data plane. It also manages an Envoy-based payload processing sidecar (BBR ext_proc) that handles request body inspection, model routing, API translation, and credential injection for external model providers. The system integrates with Kuadrant/Authorino for authentication (supporting API keys, OpenShift tokens, and external OIDC), Istio for traffic telemetry, and PostgreSQL for durable API key storage.

MaaS is deployed as a component of both Open Data Hub (ODH) and Red Hat OpenShift AI (RHOAI), controlled by the `modelsAsService` field on the DataScienceCluster CR. The Tenant reconciler mirrors the ODH operator's component deploy pattern, using embedded kustomize manifests rendered and applied via server-side apply.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| maas-controller | Go Operator (controller-runtime) | Reconciles Tenant, MaaSModelRef, MaaSSubscription, MaaSAuthPolicy, and ExternalModel CRs; deploys maas-api and payload-processing via kustomize pipeline; creates Kuadrant policies and Istio resources dynamically |
| maas-api | Go REST Service (Gin) | OpenAI-compatible API for key management, model discovery, and subscription selection; PostgreSQL-backed; provides internal callback endpoints for Authorino |
| payload-processing | Go gRPC Service (BBR ext_proc) | Envoy external processor that inspects request bodies to extract model names, resolves model providers, translates API formats, and injects backend credentials |

## Architectural Analysis

The MaaS architecture follows a governance-aggregation pattern where multiple user-facing CRs (MaaSSubscription, MaaSAuthPolicy) are reconciled into a smaller number of Kuadrant policy objects (one AuthPolicy and one TokenRateLimitPolicy per model). This aggregation is a deliberate design choice that accommodates Kuadrant's constraint of allowing only one policy per HTTPRoute target. Each controller independently aggregates its policy type, then the MaaSModelRefReconciler acts as a status coordinator that tracks whether a model has both governance dimensions (subscription + auth policy) attached before marking it as Ready. This "governance pairing" pattern ensures models cannot serve traffic without complete access control.

The Tenant reconciler is architecturally distinct from the per-model controllers. It functions as a mini-operator-within-an-operator, embedding a full kustomize rendering and server-side apply pipeline. This mirrors how the ODH platform operator deploys components, but scoped to the MaaS operands (maas-api, payload-processing, networking, policies). The pipeline includes a post-render phase that dynamically transforms AuthPolicies with gateway references, OIDC configuration, and telemetry resources based on the Tenant CR spec. The Config CR serves as a garbage collection anchor: resources owned by Config are cleaned up when MaaS is disabled, even though the Tenant CR is the user-facing configuration object.

The payload-processing component introduces an Envoy ext_proc (external processing) pattern that is architecturally significant. Rather than handling request body inspection in the gateway's Wasm plugins (which have memory and CPU constraints), MaaS delegates body-aware routing to an out-of-process gRPC service. This service runs a plugin pipeline: body-field-to-header extracts the `model` field from OpenAI-format request bodies into an `X-Gateway-Model-Name` header, model-provider-resolver maps model names to external provider endpoints, api-translation transforms between API formats (e.g., OpenAI to Anthropic), and apikey-injection retrieves backend credentials from Kubernetes Secrets and injects them into the proxied request. This design isolates complex payload manipulation from the gateway's critical path while maintaining fail-closed semantics (`failure_mode_allow: false`).

Security is enforced at multiple layers. The gateway-level default-deny policies (AuthPolicy returning 401 and TokenRateLimitPolicy with 0 tokens/minute) ensure that any model path without an explicit per-model policy is blocked. Per-model AuthPolicies use CEL expressions to extract user identity from three authentication sources (API keys, K8s tokens, OIDC JWTs) into a unified user/group/subscription tuple. API keys use an `sk-oai-*` format with SHA-256 hash-based storage, constant-time comparison, and subscription binding at mint time (immutable). The Authorization header is stripped after authentication to prevent credential exfiltration to model backends, a security measure that required Kuadrant 1.4.2+. Internal endpoints (`/internal/v1/*`) are protected by Kubernetes NetworkPolicies rather than application-level auth, restricting access to Authorino pods and cleanup CronJobs only.

The conflict detection system (`conflict_detection.go`) monitors for "rogue" AuthPolicies — non-MaaS AuthPolicies that target the same HTTPRoutes as MaaS-managed models. Since Kuadrant enforces single-policy-per-target semantics, a conflicting AuthPolicy would silently override MaaS access controls. The controller detects these conflicts, emits Kubernetes events, and degrades the affected MaaSAuthPolicy's status, providing administrators with clear visibility into policy conflicts.
