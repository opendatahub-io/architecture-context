## Purpose

**Short**: Kubernetes-native platform for managing inference model endpoints with policy-based access control, API key management, and token-based rate limiting via Gateway API and Kuadrant.

**Detailed**: Models as a Service (MaaS) provides a comprehensive platform for exposing and governing AI/ML inference endpoints. It consists of two primary components: a Kubernetes controller (maas-controller) that manages custom resources for model references, auth policies, subscriptions, and tenants; and an HTTP API service (maas-api) that provides OpenAI-compatible model discovery, API key lifecycle management, and subscription selection endpoints.

The controller reconciles a declarative model governance stack: MaaSModelRef CRs map to backend inference services (KServe LLMInferenceService or ExternalModel providers), MaaSAuthPolicy CRs define user/group access rules that translate to Kuadrant AuthPolicy resources, and MaaSSubscription CRs bind rate-limited entitlements to models via Kuadrant TokenRateLimitPolicy. The Tenant CR drives a full platform reconcile pipeline that deploys the maas-api service, networking (HTTPRoutes, DestinationRules, EnvoyFilters), observability (TelemetryPolicy, Istio Telemetry), and security (AuthPolicy, NetworkPolicies) via embedded kustomize manifests with a post-render stage for per-tenant parameterization.

Traffic flows through a Gateway API Gateway (maas-default-gateway) in openshift-ingress, where Envoy processes requests through a payload-processing sidecar pipeline (model extraction, provider resolution, API translation, API key injection) before and after Authorino-based authentication. The maas-api service handles API key validation, subscription selection, and model discovery as HTTP callbacks during the AuthPolicy evaluation chain.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| maas-controller | Go Operator (controller-runtime) | Manages MaaS CRDs (Tenant, AITenant, MaaSModelRef, MaaSAuthPolicy, MaaSSubscription, ExternalModel, Config), reconciles Kuadrant policies, deploys maas-api platform workloads via Tenant reconciler |
| maas-api | Go HTTP Service (Gin) | OpenAI-compatible model discovery API, API key lifecycle management (PostgreSQL-backed), subscription selection, Authorino callback endpoints |
| payload-processing | Envoy ext_proc gRPC Service | Request/response processing pipeline (model-to-header extraction, provider resolution, API translation, API key injection) deployed as EnvoyFilter stages |

## Architectural Analysis

The Models as a Service platform implements a layered governance architecture where declarative CRDs (MaaSModelRef, MaaSAuthPolicy, MaaSSubscription) are reconciled into executable Kuadrant policy resources (AuthPolicy, TokenRateLimitPolicy). This abstraction insulates model administrators from the complexity of Kuadrant's CEL-based AuthPolicy expressions while providing a domain-specific API. The MaaSAuthPolicy→AuthPolicy translation is particularly sophisticated: the controller generates CEL expressions that match user/group memberships against authorized subjects, embedding the logic directly into the AuthPolicy spec rather than requiring an external policy evaluation service.

The Tenant reconciler is the most complex component, implementing a full platform deploy pipeline (initialize → dependencies → prerequisites → gateway → params → kustomize → post-render → apply → deployment status) modeled after the ODH operator's component reconcile pattern. It embeds kustomize manifests into the controller container image (via Dockerfile COPY) and renders them at runtime with a PostRender stage that injects per-tenant namespace, image references, gateway configuration, and OIDC settings. This architecture enables the controller to self-deploy the entire maas-api stack — Deployments, Services, HTTPRoutes, CronJobs, AuthPolicies, EnvoyFilters, and observability resources — from a single Tenant CR, making the operational footprint minimal. The multi-tenant extension (AITenant) adds a bootstrap layer that creates tenant namespaces, Tenant CRs, and RBAC bindings, with per-tenant resource naming via 41-character-limited tenant identifiers.

The request processing pipeline uses a dual-stage Envoy ext_proc pattern: the payload-processing service is injected into the Gateway's Envoy via EnvoyFilter as both a pre-auth and post-auth external processor. Pre-auth handles model name extraction from request bodies (setting X-Gateway-Model-Name header), while post-auth handles provider resolution, API format translation, and API key injection. This split is critical because the pre-auth stage must run before Authorino evaluates the request (Authorino needs the model name for per-model auth policies), while post-auth handles credential injection that requires the authenticated identity.

Security is defense-in-depth: NetworkPolicies restrict which pods can reach the maas-api internal endpoints (only Authorino and cleanup CronJob pods), API keys are stored as SHA-256 hashes with per-key salt (key_id + null byte + secret), SubjectAccessReview-based admin checks are cached with asymmetric TTLs (30s positive, 2s negative), group names are validated against a regex allowlist to prevent CEL injection, and IDOR protection returns 404 instead of 403 for unauthorized key access. The gateway-default-deny TokenRateLimitPolicy acts as a safety net — any model inference path without an explicit per-route TokenRateLimitPolicy gets zero tokens (denied), ensuring no model is accidentally exposed without rate limiting.

The multi-tenant database isolation model uses the TENANT_NAME environment variable to scope all PostgreSQL queries. Each maas-api instance binds to a single tenant, and the Tenant reconciler injects the correct tenant name during deployment. API key creation enforces tenant mismatch validation at the service layer, preventing cross-tenant key issuance. This architectural choice means each tenant gets its own maas-api Deployment (rather than a shared multi-tenant service), simplifying isolation at the cost of per-tenant resource overhead.
