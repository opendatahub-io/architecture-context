## Purpose

**Short**: A multi-tenant platform for exposing LLM models as governed, subscription-based services with API key management, token rate limiting, and policy-driven authorization through Gateway API and Kuadrant.

**Detailed**: Models as a Service (MaaS) provides a comprehensive platform for managing access to large language models (LLMs) across both internal KServe-served models and external providers (OpenAI, Anthropic, etc.). The platform consists of two primary components: a Kubernetes operator (`maas-controller`) that reconciles custom resources for tenants, subscriptions, authorization policies, and model references into Gateway API HTTPRoutes, Kuadrant AuthPolicies, and TokenRateLimitPolicies; and a REST API service (`maas-api`) that manages API keys with zero-knowledge hash storage in PostgreSQL, provides OpenAI-compatible model listing, and serves as a callback target for Authorino authentication.

The platform implements multi-tenancy through the AITenant/Tenant CRD hierarchy. AITenant CRs bootstrap per-tenant namespaces with dedicated Gateway references, RBAC, and OIDC configuration, while Tenant CRs drive the platform reconciliation pipeline that deploys maas-api instances, networking, and observability per tenant. Governance is enforced through a dual-resource model: MaaSSubscription defines token rate limits per model per user group, while MaaSAuthPolicy grants model access to specific users/groups. A MaaSModelRef only reaches "Ready" phase when it has both an active subscription and an auth policy — ensuring no model is exposed without governance.

A payload processing pipeline (deployed as Envoy ext_proc sidecars in the gateway namespace) handles request/response transformation including model name extraction, provider-specific API translation, and credential injection for external providers. The architecture integrates deeply with Kuadrant (Authorino for authentication, Limitador for rate limiting), Istio service mesh (for mTLS, DestinationRules, EnvoyFilters), and Gateway API (for HTTPRoute-based traffic routing).

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| maas-controller | Go Operator (controller-runtime) | Reconciles MaaS CRDs (AITenant, Tenant, MaaSModelRef, MaaSSubscription, MaaSAuthPolicy, ExternalModel) into Gateway API HTTPRoutes, Kuadrant policies, Istio resources, and maas-api deployments |
| maas-api | Go REST Service (Gin framework) | Manages API keys (CRUD + validation), lists models in OpenAI-compatible format, handles subscription selection, serves as Authorino callback target |
| payload-processing | Go gRPC Service (ext_proc) | Envoy external processor for request/response transformation: model name extraction, API translation, provider credential injection |
| payload-pre-processing | Go gRPC Service (ext_proc) | Lightweight pre-auth variant extracting model name from request body into X-Gateway-Model-Name header before Authorino evaluation |

### maas-api

**Dockerfile**: `maas-api/Dockerfile.konflux`
**Intent**: REST API service deployed per tenant by the Tenant reconciler. Manages API keys with zero-knowledge hash storage in PostgreSQL, lists models in OpenAI-compatible format, validates API keys for Authorino callbacks, and selects subscriptions for gateway authorization. Deployed as a Deployment in each tenant's infrastructure namespace.
**Image**: quay.io/opendatahub/maas-api:odh-stable
**Language**: Go 1.25
**Port**: 8080/TCP (HTTP) or 8443/TCP (HTTPS with TLS overlay)

## Architectural Analysis

The MaaS platform implements a layered governance model that separates concerns between tenancy, model exposure, access control, and rate limiting. The dual-CRD governance requirement — where a MaaSModelRef reaches "Ready" only when paired with both a MaaSSubscription and a MaaSAuthPolicy — is an elegant design that prevents accidental model exposure. This is enforced in the MaaSModelRefReconciler's `checkGovernanceAttached()` function, which queries both resource types before setting the GovernanceAttached condition. The phase semantics (Pending → Ready/Unhealthy) cleanly distinguish between "waiting for governance" and "governed but backend failure."

The tenant reconciliation pipeline is architecturally sophisticated: the controller embeds kustomize manifests inside the controller container image (COPY statements in Dockerfile.konflux), then renders them at runtime with per-tenant parameters, applies post-render transformations (service naming, HTTPRoute generation), and uses server-side apply for idempotent resource management. This "operator deploys operator" pattern means the maas-controller effectively manages full application deployments (maas-api, payload-processing, networking, observability) as operands — similar to how the platform operator manages components, but at a finer grain. The pipeline stages (prerequisites → params → kustomize → post-render → SSA → deployment status) provide clear failure isolation: a missing Kuadrant CRD blocks at stage 1 before any cluster mutations.

The API key security model demonstrates defense-in-depth: keys use SHA-256(key_id + "\x00" + secret) with constant-time comparison, the database stores only hashes (zero-knowledge), tenant isolation is enforced at every query via composite indexes, and the API returns 404 instead of 403 to prevent key enumeration (IDOR protection). The ephemeral key cleanup CronJob with a 30-minute grace period and the NetworkPolicy restricting cleanup pod egress to only the maas-api service show careful blast-radius minimization.

The payload processing architecture uses a two-stage Envoy ext_proc design: the pre-processing stage (before Authorino) extracts the model name from the request body into a header (X-Gateway-Model-Name) so Authorino can make model-aware authorization decisions; the post-processing stage (after Authorino) performs API translation, provider credential injection, and model resolution. This split is necessary because Authorino evaluates headers — it cannot inspect request bodies. The pre-processing stage is configured with `failure_mode: ALLOW` (non-critical) while post-processing uses `failure_mode: DENY` (critical), reflecting their different roles. Route exclusions for `/v1/models` and `/maas-api/*` paths prevent the inference-specific pipeline from processing management API traffic.

The multi-tenancy architecture progresses through two CRDs: AITenant bootstraps infrastructure (namespace, Gateway reference, RBAC roles, OIDC config) in a central `ai-tenants` namespace, while Tenant is the per-namespace runtime configuration that the TenantReconciler uses to deploy the full stack. The controller uses namespace label discovery (`ai-gateway.opendatahub.io/tenant`, `maas.opendatahub.io/managed-by-aitenant`) to find AITenant-managed namespaces, with guards against reconciling unlabeled foreign namespaces. The Config/default singleton serves as a garbage collection anchor — all platform operands reference it as a non-controller owner, enabling cascade cleanup when the platform is uninstalled. The LifecycleReconciler ensures this singleton exists and strips legacy finalizers during upgrades, showing thoughtful lifecycle management.
