## Purpose

**Short**: Provides a comprehensive platform for Models as a Service with real-time policy management, API key authentication, token-based rate limiting, and multi-model access control for KServe and external LLM providers.

**Detailed**: Models as a Service (MaaS) is a Kubernetes-native platform that enables organizations to expose AI/ML models (both internal KServe-hosted models and external third-party LLM providers) as managed API endpoints with fine-grained access control. The platform consists of two primary components: a Go controller (`maas-controller`) that manages the lifecycle of model references, authentication policies, subscription-based rate limiting, and platform infrastructure; and a Go API service (`maas-api`) that handles API key CRUD operations, model discovery, subscription selection, and serves as a callback target for Authorino authentication validation.

The controller implements a model-centric aggregation pattern where multiple access policies and subscriptions referencing the same model are aggregated into single Kuadrant AuthPolicy and TokenRateLimitPolicy resources, satisfying Kuadrant's one-policy-per-target constraint. The platform integrates deeply with the Gateway API for traffic routing, Kuadrant/Authorino for authentication and authorization, Limitador for distributed token-based rate limiting, and Istio for service mesh networking to external providers. A third sub-component, the payload-processing service (BBR ext-proc), intercepts inference requests via an Envoy external processor to extract model names, resolve providers, translate API formats, and inject upstream credentials.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| maas-controller | Go Operator (controller-runtime) | Manages CRD lifecycle for Tenant, MaaSModelRef, MaaSAuthPolicy, MaaSSubscription, ExternalModel; deploys platform infrastructure via kustomize; creates Kuadrant policies and Istio resources |

## Architectural Analysis

The Models-as-a-Service platform implements a sophisticated model-centric aggregation pattern that transforms user-friendly multi-model access policies into the single-policy-per-target constraint required by Kuadrant. When multiple MaaSAuthPolicy resources reference the same model, the controller collects all subjects (groups and users) across every referencing policy, then generates a single Kuadrant AuthPolicy targeting that model's HTTPRoute. This same aggregation pattern applies to MaaSSubscription → TokenRateLimitPolicy. The controller tracks which user-facing CRs contributed to each generated policy via annotations (`maas.opendatahub.io/auth-policies` and `maas.opendatahub.io/subscriptions`), enabling accurate status propagation back to each contributing CR. This design allows platform teams to express access control in familiar terms (grant team X access to models A, B, C) while the controller handles the mechanical translation to gateway-level enforcement.

The security architecture employs defense-in-depth with three distinct layers. First, Authorino handles authentication (API keys validated via HTTP callback to maas-api, Kubernetes tokens via TokenReview, and optional OIDC JWT validation) and authorization (OPA policies checking group membership and subscription validity). Second, Limitador enforces per-user token-based rate limits, with rates aggregated from subscriptions and counted against `auth.identity.userid`. Third, the BBR payload-processing ext-proc strips the original Authorization header before it reaches model backends (preventing credential exfiltration to potentially untrusted models), then injects model-specific credentials from Kubernetes Secrets for external providers. The gateway-level default policies (gateway-default-auth and gateway-default-deny) ensure fail-closed behavior: any model without an explicit policy gets a 401/403 from auth and a 0-token rate limit.

The ExternalModel subsystem extends the platform beyond internal KServe models to third-party LLM providers (OpenAI, Anthropic, etc.) by creating a stack of Istio resources (ServiceEntry for mesh registration, DestinationRule for TLS origination, ExternalName Service for DNS mapping) and a Gateway API HTTPRoute with dual-match rules (path-based `/{namespace}/{name}` and header-based `X-Gateway-Model-Name`). This enables the same authentication, authorization, rate limiting, and observability infrastructure to apply uniformly to both internal and external models. All ExternalModel-created resources use OwnerReferences for automatic garbage collection, avoiding the need for finalizers.

The Tenant reconciler implements a full platform lifecycle manager using an embedded kustomize pipeline. It renders deployment manifests from overlays baked into the controller image, applies post-render transformations (gateway configuration, OIDC injection, telemetry enablement), and uses server-side apply with a dedicated field manager (`maas-controller`) to take ownership from any prior manager (enabling migration from ODH operator-managed deployments). The reconciler checks prerequisites (database secret, Authorino CRDs, User Workload Monitoring) and reports granular status conditions (DependenciesAvailable, MaaSPrerequisitesAvailable, DeploymentsAvailable) so operators can diagnose deployment issues. Optional features like Perses dashboards are handled via graceful degradation — if the CRD isn't installed, the resource is silently skipped, and a CRD watch triggers re-reconciliation when it appears.

The API key design prioritizes security through several mechanisms: keys use a `sk-oai-{key_id}_{secret}` format where the key_id serves as a per-key salt for SHA-256 hashing (with a null byte delimiter to prevent length-extension attacks), plaintext keys are shown exactly once at creation and never stored, subscription binding is immutable at creation time (preventing subscription-swap attacks), group snapshots are frozen at key creation (preventing privilege escalation through group changes), and all authorization failures return HTTP 404 instead of 403 to prevent key enumeration (IDOR protection). The API key validation endpoint uses constant-time comparison and updates `last_used_at` asynchronously to avoid timing side-channels.
