## Purpose

**Short**: A multi-tenant LLM inference gateway that provides model discovery, API key management, authentication, authorization, and token-based rate limiting via Gateway API and Kuadrant.

**Detailed**: Models as a Service (MaaS) is a Kubernetes-native platform for exposing LLM inference endpoints as managed services with fine-grained access control and rate limiting. It consists of two primary components: a Go controller (`maas-controller`) that reconciles custom resources into Gateway API HTTPRoutes, Kuadrant AuthPolicies/TokenRateLimitPolicies, and Istio networking resources; and a REST API server (`maas-api`) that provides OpenAI-compatible model listing, subscription management, and cryptographically secure API key lifecycle management backed by PostgreSQL.

The platform integrates with KServe LLMInferenceService for internal model serving and supports external model providers (OpenAI, Anthropic, etc.) via ExternalModel CRDs. Authentication is handled through Kuadrant/Authorino with three methods: API keys (`sk-oai-*`), Kubernetes TokenReview, and external OIDC. Rate limiting uses Kuadrant TokenRateLimitPolicy with per-subscription token quotas. A payload processing sidecar (BBR) runs as an Envoy external processor to extract model names from request bodies, resolve providers, translate API formats, and inject credentials for external models.

The architecture follows a CRD-driven design where administrators define MaaSModelRef, MaaSAuthPolicy, and MaaSSubscription resources, and the controller generates the corresponding Gateway API and Kuadrant policies. A Tenant singleton CRD configures platform-wide settings including gateway reference, OIDC configuration, API key policies, and telemetry dimensions.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| maas-controller | Go Operator (controller-runtime) | Reconciles MaaS CRDs into HTTPRoutes, AuthPolicies, TokenRateLimitPolicies, and Istio resources; manages Tenant lifecycle and platform deployment via kustomize |

## Data Flows

### Flow 1: Model Inference Request (Internal Model via LLMInferenceService)

| Step | Source | Destination | Port | Protocol | Encryption | Auth |
|------|--------|-------------|------|----------|------------|------|

## Architectural Analysis

The MaaS platform implements a policy-as-code pattern where user-facing CRDs (MaaSModelRef, MaaSAuthPolicy, MaaSSubscription) are reconciled into lower-level Kuadrant and Gateway API resources. This two-tier CRD model provides a clean abstraction: administrators think in terms of "which users can access which models at what rate," while the controller translates this into AuthPolicies with OPA rules, TokenRateLimitPolicies with per-subscription counters, and HTTPRoutes with appropriate backend references. The aggregation pattern is notable -- multiple MaaSAuthPolicies targeting the same model are merged into a single Kuadrant AuthPolicy, and multiple MaaSSubscriptions into a single TokenRateLimitPolicy, avoiding policy explosion at the gateway level.

The authentication architecture supports three credential types (API keys, Kubernetes tokens, external OIDC) through a unified Authorino pipeline. API key validation is particularly sophisticated: keys follow the `sk-oai-{key_id}_{secret}` format with SHA-256 hashing using the key_id as a per-key salt, constant-time comparison, and a show-once security model. The validation flow is entirely fail-closed -- missing subscriptions, expired keys, or unreachable model endpoints all result in access denial. The Authorino AuthPolicy strips the Authorization header before forwarding to model backends, preventing credential exfiltration to potentially untrusted model services.

The Tenant reconciler is the platform orchestration component -- it manages the lifecycle of maas-api, payload-processing, networking resources, and observability configuration using embedded kustomize manifests. The reconciliation pipeline (dependencies -> prerequisites -> gateway -> params -> kustomize -> post-render -> apply) uses server-side apply with "maas-controller" as the field owner, enabling declarative management of the entire MaaS stack. This design allows the ODH/RHOAI operator to simply create a Tenant CR and delegate all platform deployment to maas-controller.

External model support (ExternalModel CRD) demonstrates the platform's extensibility. The controller creates ExternalName Services, Istio ServiceEntries, and DestinationRules with TLS origination to route traffic to providers like OpenAI and Anthropic. The BBR payload processing pipeline intercepts requests at the Envoy level via ext_proc to extract model names from request bodies, resolve the correct provider, translate API formats, and inject provider-specific credentials from Kubernetes Secrets. This design keeps credential management within the cluster while supporting heterogeneous model backends transparently.

The observability stack is comprehensive: TelemetryPolicy exports per-request metadata (user, subscription, cost center, organization, model) for billing and analytics, Istio Telemetry adds subscription dimensions to REQUEST_DURATION metrics, and ServiceMonitors scrape Authorino, Limitador, and gateway metrics. The optional Perses dashboard integration (gated on CRD existence) shows forward-looking architecture for the Cluster Observability Operator migration from Grafana.
