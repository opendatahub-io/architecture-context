## Purpose

**Short**: A Kubernetes operator that manages the lifecycle of TrustyAI explainability services, LM evaluation jobs, guardrails orchestration, NeMo guardrails, and evaluation hub services within the Red Hat OpenShift AI platform.

**Detailed**: The TrustyAI Service Operator is a multi-controller operator built with controller-runtime that manages five distinct services within the AI/ML platform. It provides model explainability through the TrustyAI Service (TAS), language model evaluation through LMEvalJob (LMES), AI safety guardrails through GuardrailsOrchestrator (GORCH) and NemoGuardrails, and centralized evaluation management through EvalHub.

The operator uses a modular controller architecture where each controller is independently registerable and can be selectively enabled via the `--enable-services` flag. The RHOAI overlay enables all controllers: TAS, LMES, GORCH, NEMO_GUARDRAILS, and EVALHUB. Each controller manages its own CRD and creates the necessary Kubernetes resources (Deployments, Services, Routes, ConfigMaps, RBAC) dynamically at reconciliation time. The operator integrates with KServe InferenceServices for model serving, Kueue for job scheduling, Istio for service mesh traffic management, and OpenShift for TLS certificate generation and Routes.

The operator ships two container images: the main operator binary (`Dockerfile.konflux`) and a driver binary for LM evaluation jobs (`Dockerfile.konflux.driver`). All managed component images (TrustyAI service, guardrails orchestrator, EvalHub, etc.) are configurable via a central ConfigMap and are not built from this repository.

## Architectural Analysis

The TrustyAI Service Operator exhibits a well-structured modular controller architecture where each of its six controllers is registered via a plugin-style `registerService()` pattern in `controllers/controllers.go`. The operator binary accepts an `--enable-services` flag that selectively enables controllers, allowing the same binary to serve different deployment profiles (LMES-only, MCP guardrails, full RHOAI). This design enables fine-grained deployment control without requiring separate builds.

A significant architectural pattern is the consistent use of Go template files (`.tmpl.yaml`) embedded via `//go:embed` directives for all Kubernetes resource creation. Each controller group (TAS, GORCH, NemoGuardrails) maintains its own `templates/` directory with deployment, service, route, and kube-rbac-proxy configuration templates. This separation of resource definitions from reconciliation logic improves maintainability and makes the exact resource shape visible without reading Go code. The kube-rbac-proxy sidecar pattern is consistently applied across TAS, GORCH, EvalHub, and NemoGuardrails, with authorization enforced via SubjectAccessReview checking the `get` verb on the service's `services` resource — a lightweight RBAC-based authentication model.

The LMES controller is architecturally distinct from the other controllers: it creates bare Pods rather than Deployments, uses a driver sidecar pattern where an init container copies the driver binary to a shared EmptyDir volume, and the driver process coordinates evaluation execution and progress monitoring via pod exec commands. This Pod-based design supports ephemeral batch workloads that integrate with Kueue's admission control through the separate JobMgr controller, which implements the Kueue `GenericJob` interface by wrapping `LMEvalJob` resources.

The GORCH controller implements a sophisticated auto-configuration system that watches InferenceServices and ServingRuntimes to dynamically generate orchestrator configuration. It discovers generation services by name and detector services by label selector (`trustyai/guardrails: true`), extracts TLS secrets, and generates YAML configuration files in ConfigMaps. A resource version hash mechanism (`trustyai.opendatahub.io/autoconfig-resource-hash` annotation) prevents unnecessary redeployments while ensuring configuration stays synchronized with the underlying InferenceService topology. The controller also supports up to three kube-rbac-proxy sidecars (orchestrator, gateway, built-in detector), each with independent ports and health endpoints.

The operator disables controller-runtime's client cache for ConfigMaps, Secrets, Pods, and Services (`client.CacheOptions.DisableFor` in `cmd/main.go`) to prevent OOM from cluster-wide informer caches. Similarly, the GORCH controller watches ConfigMaps with `OnlyMetadata` to avoid caching full ConfigMap data. These memory optimizations reflect operational lessons from running in memory-constrained environments (the operator is limited to 700Mi with a GOMEMLIMIT of 630MiB). Input validation in the LMES controller is security-hardened with regex-based pattern matching and explicit blocking of shell metacharacters (`$(`, backtick, `&&`, `||`, `;`) in user-provided values, preventing command injection via LMEvalJob specs.
