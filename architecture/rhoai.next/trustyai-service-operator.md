## Purpose

**Short**: Multi-controller Kubernetes operator that manages TrustyAI explainability services, LLM evaluation jobs, guardrails orchestration, NeMo guardrails, and an evaluation hub for Red Hat OpenShift AI.

**Detailed**: The TrustyAI Service Operator is a comprehensive operator that manages five distinct controller domains within a single binary. The **TrustyAIService (TAS)** controller deploys AI model explainability services that intercept inference payloads from KServe/ModelMesh InferenceServices to compute fairness and bias metrics (SPD, DIR). The **LMEvalJob (LMES)** controller orchestrates language model evaluation jobs using the lm-evaluation-harness framework, supporting Unitxt task recipes, offline evaluation, OCI artifact publishing, and Kueue-based workload scheduling. The **EvalHub** controller manages a centralized evaluation hub API server with multi-tenant job execution, provider/collection plugin systems, MCP (Model Context Protocol) server integration, and Kueue workload failure monitoring. The **GuardrailsOrchestrator (GORCH)** controller deploys guardrails orchestration services that auto-discover KServe InferenceServices to configure generators and detectors for content safety guardrailing. The **NemoGuardrails** controller manages NVIDIA NeMo Guardrails server deployments with optional Istio EnvoyFilter integration for MCP gateway SSE response handling.

All controllers share a common pattern of kube-rbac-proxy sidecars for authentication, OpenShift Route creation, ServiceMonitor integration for Prometheus metrics, and CA bundle management for secure in-cluster HTTPS communication. The operator uses a modular kustomize component system that allows individual controllers to be enabled or disabled via `--enable-services` flags and kustomize overlays.

### odh-ta-lmes-driver

**Dockerfile**: `Dockerfile.konflux.driver`
**Intent**: Init container image for LMEvalJob Pods. The driver binary is copied into a shared emptyDir volume at Pod start and then executed by the main evaluation container. It manages the lifecycle of `python -m lm_eval` subprocesses, tracks progress bars, handles graceful shutdown, and uploads results to S3 or OCI registries.
**Image**: odh-ta-lmes-driver
**Language**: Go 1.26
**Port**: 18080/TCP (default, configurable — driver HTTP control port for GetStatus/Shutdown)

## Architectural Analysis

The TrustyAI Service Operator exemplifies a "multi-controller monolith" pattern — five functionally distinct controllers packaged into a single binary with selective activation. This design trades deployment simplicity (one Deployment, one image, one set of RBAC) for operational flexibility: overlays can enable any subset of controllers via `--enable-services`. The shared `controllers.go` registration pattern with `registerService()` / `SetupControllers()` provides clean separation while allowing shared infrastructure (metrics, logging, CA bundle management, generic reconciliation utilities).

The kustomize component system (`config/components/`) is particularly well-designed. Each controller's CRDs and RBAC are isolated into independent kustomize Components, and overlays compose them additively. This means RBAC follows least-privilege per overlay — the LMES-only overlay gets only LMES RBAC, not the full TAS or GORCH permissions. The `hack/generate-components.sh` script auto-generates component kustomizations from controller-gen output, ensuring RBAC definitions stay synchronized with kubebuilder markers.

The authentication pattern is notably flexible. All controllers support an annotation-driven auth toggle (`security.opendatahub.io/enable-auth`). When enabled, kube-rbac-proxy sidecars are injected with per-controller RBAC proxy configurations (distinct ConfigMaps per port/upstream). EvalHub goes further with SAR-based multi-tenant authorization: the kube-rbac-proxy config maps X-Tenant headers to namespace-scoped SubjectAccessReview checks, enabling fine-grained per-namespace access control without application-level auth code. The GORCH controller has three separate kube-rbac-proxy configurations (orchestrator, gateway, built-in detector) each on different ports.

The EvalHub controller introduces a sophisticated multi-tenancy pattern. Namespaces labeled with `evalhub.trustyai.opendatahub.io/tenant` are dynamically discovered, and per-namespace RBAC resources (Roles, RoleBindings, service CA ConfigMaps, discovery ConfigMaps) are reconciled. The discovery ConfigMap pattern (`evalhub-discovery` with `<instance>.url` keys) provides service discovery for evaluation job pods without requiring DNS or hardcoded URLs. The job failure reconciler and Kueue workload reconciler add observability by POSTing failure events back to the EvalHub API using bearer token auth and service CA trust.

A notable architectural risk is the operator's memory management strategy. The `cmd/main.go` explicitly disables informer caching for ConfigMaps, Secrets, Pods, and Services to prevent OOM from cluster-wide informer cache flooding. This trades latency (~50ms API server reads vs ~1ms cache reads) for memory safety, but means the operator performs more API server calls than a typical controller-runtime operator. The GOMEMLIMIT of 630MiB with 700Mi memory limit provides tight bounds.

The NemoGuardrails controller's EnvoyFilter integration for MCP gateway SSE response stripping is architecturally interesting — it auto-discovers MCPGatewayExtension resources (Kuadrant CRD), resolves the target gateway, and injects Envoy Lua filters for response header manipulation. This bridges the NeMo guardrails domain with the Istio/Envoy data plane in a way that's specific to the MCP Server-Sent Events protocol.
