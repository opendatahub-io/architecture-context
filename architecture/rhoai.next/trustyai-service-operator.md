## Purpose

**Short**: Kubernetes operator managing AI trustworthiness services including model explainability, language model evaluation, and guardrails orchestration for RHOAI.

**Detailed**: The TrustyAI Service Operator is a multi-controller Kubernetes operator built with controller-runtime that manages five distinct AI trustworthiness services within the Red Hat OpenShift AI platform. It uses a plugin-based controller registry where each service (TAS, LMES, GORCH, NEMO_GUARDRAILS, EVALHUB) is independently registered and selectively enabled via command-line flags.

The operator manages the lifecycle of TrustyAI explainability services (model bias and fairness monitoring integrated with KServe InferenceServices), LMEval evaluation jobs (language model evaluation using lm-evaluation-harness with Kueue integration), GuardrailsOrchestrator (Rust-based guardrails enforcement with auto-configuration from KServe InferenceServices), NemoGuardrails (NVIDIA NeMo guardrails deployment with CA bundle management), and EvalHub (centralized evaluation management with multi-tenant namespace support and provider/collection configuration).

Each controller creates and manages Kubernetes resources dynamically including Deployments, Services, Routes, ConfigMaps, ServiceMonitors, and RBAC resources. The operator enforces security through kube-rbac-proxy sidecars for authentication, OpenShift serving certificates for TLS, and FIPS 140-2 compliant cipher suites across all proxy configurations.

## Architectural Analysis

The TrustyAI Service Operator exemplifies a multi-service operator pattern where a single binary manages five distinct CRD controllers, each with its own reconciliation loop, RBAC requirements, and resource graph. The `EnabledServices` mechanism (`controllers/controllers.go`) uses a global service registry populated via Go `init()` functions, allowing each controller to self-register and be selectively enabled through command-line flags. This design enables flexible deployment profiles -- RHOAI enables all five services, while LMES-only or MCP-guardrails overlays run subsets.

A notable architectural decision is the consistent use of kube-rbac-proxy sidecars across TAS, GORCH, NemoGuardrails, and EvalHub for authentication enforcement. All proxy configurations share the same FIPS 140-2 compliant cipher suite (6 approved ciphers) and TLS 1.2 minimum. The proxy validates bearer tokens via Kubernetes TokenReview/SubjectAccessReview, checking whether the caller can GET the associated Service resource in the target namespace. This provides Kubernetes-native RBAC without requiring Istio or external identity providers. The GORCH controller is particularly complex -- it configures up to three separate kube-rbac-proxy instances (orchestrator on 8432, gateway on 8490, built-in detector on 8480), each with its own health check port.

The GORCH auto-configuration system (`config_generation.go`, ~800 lines) represents the most sophisticated reconciliation logic in the operator. It watches KServe InferenceServices and ServingRuntimes, extracts hostnames, ports, and TLS requirements, then generates YAML configuration ConfigMaps for the Rust-based orchestrator and optional gateway containers. Configuration changes are detected via SHA256 hashing of ConfigMap data, with hash annotations on Deployment pod templates triggering automatic rollouts. This creates a dynamic, zero-touch integration where deploying a new InferenceService with the label `trustyai/guardrails=true` automatically reconfigures the guardrails pipeline.

The EvalHub controller introduces a multi-tenant namespace architecture unique among the controllers. It watches namespaces with a `evalhub.trustyai.opendatahub.io/tenant` label and provisions ServiceAccounts, Roles, RoleBindings, and service CA ConfigMaps in each tenant namespace. The evaluation job failure reconciler monitors batch/v1 Jobs and Kueue Workloads across tenant namespaces, detecting operator-only failures (image pull errors, OOM kills, container crashes) that the evaluation sidecar cannot report, and POSTing structured failure events back to the EvalHub API with `X-Tenant` headers for namespace isolation. This compensates for the inherent limitation that crashed containers cannot call back to report their own failure.

The LMEval controller uses an unusual pod-as-job pattern rather than Kubernetes Jobs. It creates bare Pods with `restartPolicy: Never`, managed via ownerReferences and finalizers. The driver binary is injected via an init container (copy from `/bin/driver` to a shared EmptyDir volume), and a tightly sandboxed execution environment enforces offline-first operation (all HuggingFace telemetry and online access disabled by default), with explicit opt-in through both job spec flags and operator-level DSC config overrides. Input validation is comprehensive -- regex patterns block shell metacharacters, path traversal, and invalid model/task names, while the pod security context drops all capabilities with seccomp RuntimeDefault.
