## Purpose

**Short**: Kubernetes operator that manages AI trustworthiness, model evaluation, and guardrails infrastructure for Red Hat OpenShift AI.

**Detailed**: The TrustyAI Service Operator is a multi-controller Kubernetes operator built with kubebuilder v4 and controller-runtime. It manages five distinct CRDs under the `trustyai.opendatahub.io` API group, each addressing a different aspect of AI/ML trustworthiness:

1. **TrustyAIService (TAS)** — Deploys the TrustyAI explainability and fairness service alongside KServe/ModelMesh inference services, injecting payload processors and monitoring model predictions for bias and drift.
2. **LMEvalJob (LMES)** — Orchestrates language model evaluation jobs using lm-evaluation-harness and Unitxt frameworks, with support for offline execution, S3/OCI artifact management, and Kueue-based queue scheduling.
3. **GuardrailsOrchestrator (GORCH)** — Deploys the FMS Guardrails orchestrator with auto-discovery of InferenceServices and detector services, enabling content moderation and safety guardrails for LLM inference.
4. **NemoGuardrails** — Deploys NVIDIA NeMo Guardrails instances with multi-source CA bundle management and configurable guardrail configurations.
5. **EvalHub** — Deploys a centralized evaluation hub with multi-tenant namespace support, provider/collection management, database backends, MLFlow integration, and Kueue workload monitoring.

The operator uses a modular architecture where each controller is independently selectable via `--enable-services` flags, allowing distribution-specific overlays (ODH, RHOAI, LMES-only, etc.) to compose the exact set of capabilities needed.

## Architectural Analysis

The TrustyAI Service Operator follows a modular multi-controller pattern where a single operator binary hosts multiple independent controllers, each selectable at startup via the `--enable-services` flag. This design enables distribution-specific deployment profiles — RHOAI and ODH enable all controllers, while standalone profiles like LMES-only or MCP-Guardrails can be composed for edge use cases. The kustomize overlay structure mirrors this modularity: each controller's CRD, RBAC, and ConfigMaps are packaged as kustomize components that overlays compose selectively.

The authentication architecture consistently uses kube-rbac-proxy as a sidecar container across all user-facing services (TAS, GORCH, NemoGuardrails, EvalHub). Each controller checks for an auth annotation on the CR and conditionally injects the proxy sidecar, forwarding authenticated traffic to the upstream application container. The GORCH controller is the most sophisticated here, deploying up to three kube-rbac-proxy instances (orchestrator, gateway, built-in detector) with distinct upstream/downstream port mappings and OAuth header forwarding for the orchestrator path.

The LMES controller implements a full job lifecycle state machine (New → Scheduled → Running → Complete, with Cancelled and Suspended branches) and manages pods directly rather than using Jobs or batch primitives. This gives fine-grained control over the evaluation lifecycle, including init container-based driver binary distribution, pod-exec-based status polling, and configurable offline/online modes. The integration with Kueue via the JOB_MGR controller implements the `jobframework.GenericJob` interface, mapping LMEvalJobs to Kueue Workloads for queue-based admission without changing the pod management logic.

The EvalHub controller is the most complex in terms of RBAC and multi-tenancy. It discovers tenant namespaces by watching Namespace labels, provisions per-tenant service accounts and role bindings, and generates SubjectAccessReview policies mapped from HTTP paths and X-Tenant headers. The job failure monitoring subsystem includes two secondary reconcilers — one watching batch/v1 Jobs for container failures and another watching Kueue Workloads for admission failures — both posting structured events to the EvalHub API.

A notable design decision is the conditional Istio integration in the TAS controller. The controller checks at runtime whether the DestinationRule and VirtualService CRDs exist in the cluster before attempting to create these resources, making the service mesh integration gracefully optional. However, the VirtualService CRD existence check references the DestinationRule CRD name (`destinationrules.networking.istio.io`) rather than the VirtualService CRD name — this appears to be a copy-paste bug in `virtual_service.go` that could cause the VirtualService to not be created even when the CRD is available if DestinationRules are absent.
