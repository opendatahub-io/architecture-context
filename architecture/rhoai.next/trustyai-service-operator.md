## Purpose

**Short**: Multi-CRD Kubernetes operator that manages TrustyAI explainability services, AI model evaluation (LMEval/EvalHub), guardrails orchestration, and NeMo Guardrails deployments for responsible AI on OpenShift AI.

**Detailed**: The TrustyAI Service Operator is a controller-runtime based Go operator that manages five distinct custom resources spanning AI trustworthiness capabilities. It operates as a single binary with feature-gated controllers enabled via `--enable-services` flags (TAS, LMES, GORCH, NEMO_GUARDRAILS, EVALHUB).

The operator's core function is lifecycle management of AI governance infrastructure. The **TrustyAIService** controller deploys explainability/fairness monitoring services that intercept KServe InferenceService predictions via InferenceLogger. The **LMEvalJob** controller orchestrates language model evaluation jobs using lm-evaluation-harness, with Kueue integration for GPU quota management. The **EvalHub** controller provides a centralized evaluation management API with multi-tenant namespace distribution, MCP (Model Context Protocol) server support, and provider/collection discovery. The **GuardrailsOrchestrator** controller deploys guardrails infrastructure with auto-configuration from KServe InferenceServices. The **NemoGuardrails** controller manages NVIDIA NeMo Guardrails server deployments with MCP gateway integration and Istio EnvoyFilter support.

All controllers share a common pattern of kube-rbac-proxy sidecar injection for authentication, OpenShift Route creation for external access, ServiceMonitor creation for Prometheus metrics collection, and CA bundle management for TLS trust chains. The operator reads image references and configuration from a central ConfigMap (`trustyai-service-operator-config`), enabling the platform operator (rhods-operator) to control versions and settings.

## Architectural Analysis

The TrustyAI Service Operator is architecturally notable for consolidating five distinct functional domains — explainability monitoring, model evaluation, evaluation management, guardrails orchestration, and NeMo guardrails — into a single operator binary with feature-gated controllers. This "multi-personality" design is driven by organizational ownership: the TrustyAI team owns all responsible AI capabilities in RHOAI, and packaging them as a single operator simplifies lifecycle management and version coordination. The `--enable-services` flag allows the platform operator to activate only the needed controllers, and the RHOAI overlay enables all five by default (`TAS,LMES,GORCH,NEMO_GUARDRAILS,EVALHUB`).

A consistent security pattern runs across all controllers: every user-facing service gets a kube-rbac-proxy sidecar container that terminates TLS (using OpenShift service-ca-provisioned certificates) and enforces SubjectAccessReview-based authorization before proxying to the application container on localhost. This pattern appears in TrustyAI (port 8443), EvalHub (port 8443), EvalHub MCP (port 8443), Guardrails Orchestrator (ports 8432/8490/8480), and NeMo Guardrails (port 8443). The EvalHub controller takes this further with a sophisticated multi-resource SAR policy: different API endpoints map to different virtual Kubernetes resources (providers, collections, status-events, evalhubs/proxy), enabling fine-grained per-endpoint authorization without a separate policy engine.

The EvalHub controller introduces multi-tenancy via namespace labels (`evalhub.trustyai.opendatahub.io/tenant`). When a namespace carries this label, the controller automatically provisions ServiceAccounts, Roles, RoleBindings, service CA ConfigMaps, and a shared `evalhub-discovery` ConfigMap containing service URLs. This cross-namespace resource distribution cannot use owner references (which are namespace-scoped), so the controller uses label-based tracking with normalized names to prevent collisions across multiple EvalHub instances. The job failure reconciler watches batch Jobs and Kueue Workloads in tenant namespaces and reports failures back to EvalHub via HTTPS using the operator's own ServiceAccount token — a callback pattern that avoids giving job pods direct API access.

The GuardrailsOrchestrator controller features a unique auto-configuration system that watches KServe InferenceServices and ServingRuntimes. When a user specifies `autoConfig.inferenceServiceToGuardrail`, the controller discovers detector services by label matching, extracts TLS serving certificates from secrets, generates orchestrator and gateway ConfigMaps, and triggers deployment updates only when a content hash changes. This reactive pattern — watching external CRDs and synthesizing configuration — is more complex than typical operator reconciliation and requires careful ordering to avoid race conditions between ISVC readiness and config generation.

The NeMo Guardrails controller introduces Istio integration via EnvoyFilter CRs, specifically an `mcp-sse-strip` filter that manipulates SSE (Server-Sent Events) headers for MCP protocol compatibility. It also integrates with Kuadrant's MCPGatewayExtension CRD for API gateway discovery, making it the most externally-coupled controller. The CA bundle handling across NeMo and Guardrails controllers merges three trust sources (ODH trusted CA, OpenShift service CA, user-provided CA) via an init container that concatenates PEM files — a practical but fragile approach that could break if any source ConfigMap is missing or malformed.
