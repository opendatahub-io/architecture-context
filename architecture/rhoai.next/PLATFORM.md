## Platform Overview

Red Hat OpenShift AI (RHOAI) is an enterprise MLOps and AI platform deployed on OpenShift Container Platform via the Operator Lifecycle Manager (OLM). The platform is orchestrated by a meta-operator (`rhods-operator`) that manages 29 sub-operators and services through two primary Custom Resources: `DataScienceCluster` (DSC) controls which components are enabled, and `DSCInitialization` (DSCI) configures platform-wide settings including monitoring, service mesh, and trusted CA bundles. The operator uses a registry-based component management pattern where each component implements a `ComponentHandler` interface, enabling independent reconciliation pipelines (preconditions → initialize → kustomize → deploy → garbage-collect) without modifying orchestration logic.

The platform's ingress architecture centers on Gateway API with Istio as the implementation. The `rhods-operator` Gateway controller deploys a multi-layer stack: GatewayClass → Gateway → Envoy proxy → EnvoyFilter (ext_authz + Lua) → kube-auth-proxy → backend applications. This architecture supports both OAuth (OpenShift-integrated) and OIDC (external identity provider) authentication modes, with automatic detection via cluster introspection. OpenShift Routes provide backward-compatible ingress for components that predate the Gateway API migration, and several components (KubeRay, MaaS, Model Registry) use HTTPRoute resources with cross-namespace ReferenceGrants for secure service exposure through the platform's central `data-science-gateway`.

The security posture is defense-in-depth: kube-rbac-proxy enforces SubjectAccessReview authorization on 9+ components, Authorino/Kuadrant provides API-level authentication for model serving and MaaS endpoints, NetworkPolicies segment traffic for 15+ components, and pod-to-pod TLS (via OpenShift service-ca or cert-manager) encrypts internal communication. Container images overwhelmingly use Red Hat UBI base images (198 of 474 Dockerfiles) and run as non-root users (313 Dockerfiles). The platform ships 192 CRDs across 50+ API groups, reflecting a deeply Kubernetes-native architecture.

### Central Components

**rhods-operator** is the platform root: it deploys and manages the lifecycle of all other components through DSC/DSCI reconciliation. It owns 209 controller watches and 27 CRDs, functioning as both a meta-operator (component lifecycle) and infrastructure operator (Gateway, monitoring, service mesh, trusted CA).

**KServe** is the most depended-upon component (18 reverse dependencies). It provides the model serving control plane, managing InferenceService, LLMInferenceService, LocalModel, and InferenceGraph CRDs. All inference runtimes (vLLM, Caikit-TGIS, MLServer, OVMS, ModelMesh) deploy through KServe's serving infrastructure. The v1alpha2 LLMInferenceService controller represents a major evolution toward disaggregated serving with llm-d integration.

**Kueue** is the platform's resource quota and scheduling backbone (10 reverse dependencies). It provides framework-agnostic workload queuing for distributed training (Kubeflow Training Operator), Ray clusters (CodeFlare/KubeRay), and Spark jobs. Its ClusterQueue/LocalQueue/Cohort hierarchy enables multi-tenant resource sharing with priority-based preemption and fair sharing.

**odh-model-controller** bridges KServe's core inference infrastructure with RHOAI platform services, adding OpenShift Routes, Prometheus monitoring, Kuadrant AuthPolicies, and KEDA TriggerAuthentications to InferenceService and LLMInferenceService resources.

**kube-rbac-proxy** is the platform-wide authentication sidecar, used by 9+ components to enforce Kubernetes SubjectAccessReview authorization at the HTTP layer.

## Deployment Architecture

### Deployment Topology

The platform follows a three-tier namespace model. The **operator tier** (redhat-ods-operator or openshift-operators namespace) contains the rhods-operator meta-operator, which is installed via OLM and manages the lifecycle of all platform components. The **platform tier** (redhat-ods-applications or opendatahub namespace) hosts the control plane: component operators, the dashboard, model controllers, and shared infrastructure like Authorino and kube-auth-proxy. The **user tier** spans multiple user-created namespaces where workloads run — InferenceService pods, Notebooks, RayCluster workers, training jobs, and pipeline runs — each isolated by RBAC and NetworkPolicies.

Istio provides the service mesh layer, with the data-science-gateway Gateway deployed in the platform namespace. Components that require external access (dashboard, model endpoints, Ray dashboards, Model Registry) create HTTPRoute resources targeting this gateway, with cross-namespace ReferenceGrants authorizing the routing. An EnvoyFilter injects ext_authz and Lua filter chains for authentication and token forwarding.

#### Manifest Consumption Pipeline

Component manifests flow from upstream repositories through a multi-stage pipeline. Each component repository maintains its own kustomize manifests with base + distribution overlays. The `get_all_manifests.sh` script in the rhods-operator repository clones component repos at pinned commits and copies their manifests into the operator's `opt/manifests/` directory. During build, these manifests are embedded into the operator binary. At runtime, the operator renders kustomize manifests with runtime parameters (namespace, image references from RELATED_IMAGE env vars, feature flags from DSC/DSCI CRs) and applies them to the cluster. This pipeline enables reproducible builds while allowing each component team to own their manifest definitions.

### Disconnected Support

RHOAI supports air-gapped/disconnected deployments through the `RELATED_IMAGE` environment variable pattern mandated by OLM. The rhods-operator and component operators declare all container image references as `RELATED_IMAGE_*` env vars in their ClusterServiceVersion (CSV), enabling OLM to catalog and mirror all required images. Operators resolve image references at runtime from these env vars rather than hardcoding registry URLs, so mirrored registries are used transparently. The `oc-mirror` tool supports mirroring the complete RHOAI image set from Red Hat's registry to a disconnected registry. All image references use digest pinning (`@sha256:...`) rather than mutable tags, ensuring reproducibility and preventing supply-chain drift in disconnected environments.

## Platform Architectural Analysis

RHOAI's architecture reveals several cross-cutting patterns and tensions that emerge only when examining all 92 components together.

**The meta-operator pattern creates a single point of orchestration complexity.** The rhods-operator manages 29 sub-components through a single reconciliation loop with 209 controller watches. While the `ComponentHandler` registry pattern makes adding components mechanical, the operator's surface area is enormous: it owns the Gateway infrastructure, monitoring stack, trusted CA distribution, service mesh configuration, and component lifecycle simultaneously. Any disruption to the operator's reconciliation blocks the entire platform. The DSC/DSCI two-CR model (one for component enablement, one for platform-wide settings) is a clean separation, but in practice the operator must reason about cross-component dependencies (e.g., KServe requires Istio, DSPO requires Argo Workflows, CodeFlare requires KubeRay) during reconciliation ordering.

**Authentication is fragmented across four distinct mechanisms.** kube-rbac-proxy (SubjectAccessReview) is used by 9 components, Kuadrant/Authorino (AuthPolicy CRDs) by 8 components, OAuth2 proxy by 2 components, and the agents-operator AuthBridge by AI agent workloads. Each mechanism has different configuration surfaces, failure modes, and debugging approaches. The Kuadrant/Authorino path is the strategic direction (it supports Gateway API natively and provides declarative policy), but the migration is incomplete — many components still use kube-rbac-proxy which requires per-component sidecar configuration. The agents-operator introduces a fifth authentication layer (OPA policy bundles + JWT validation + mTLS transport) that operates independently of the platform's other auth mechanisms.

**The model serving stack has three generations of architecture coexisting.** ModelMesh (multi-model, gRPC-based, etcd-coordinated) represents generation 1; KServe InferenceService (single-model-per-pod, Knative/raw deployment) represents generation 2; and KServe LLMInferenceService + llm-d (disaggregated serving with prefix-cache-aware routing) represents generation 3. Each generation has different scaling models, networking requirements, and operational characteristics. ModelMesh uses sidecar injection (runtime adapter, REST proxy, puller, oauth-proxy) for a total of 5+ containers per pod. KServe uses storage initializer init containers. llm-d uses ext_proc advisory routing with 60+ plugins. The odh-model-controller must understand all three models to provide consistent platform integration (Routes, monitoring, auth).

**Go build tags create a hidden distribution layer.** KServe, KubeRay, odh-model-controller, and other components use `_odh.go` / `_default.go` file suffixes with Go build tags (`distro`, `odh`) to compile different behavior for upstream vs. RHOAI. This pattern keeps upstream compatibility but makes it difficult to understand what code actually runs in production without checking the Dockerfile's `GOTAGS` argument. It also means that CI running without the right build tags tests different code paths than production.

**The llm-d stack introduces a fundamentally different serving paradigm.** Unlike traditional model serving (one model per pod, static routing), llm-d separates the routing decision (llm-d-router EPP with 60+ scheduling plugins) from the data plane (Envoy/AgentGateway), enables KV-cache disaggregation (separate prefill and decode phases), and supports prefix-cache-aware routing across a fleet of vLLM instances. This architecture trades simplicity for performance optimization, requiring coordination between 6+ components (llm-d, llm-d-router, llm-d-kv-cache, llm-d-routing-sidecar, llm-d-latency-predictor, Gateway API Inference Extension) for a single inference request.

**Container image provenance follows two distinct patterns.** The majority of components build from source in multi-stage Dockerfiles. However, a growing number of components (vllm, vllm-gaudi, vllm-spyre) use a "thin wrapper" pattern where the Dockerfile extends a product image from RHAIIS (Red Hat AI Infrastructure Services) with minimal changes — typically just entrypoint overrides and environment variable configuration. This pattern reduces build complexity and leverages RHAIIS's investment in GPU runtime optimization, but creates a hard dependency on RHAIIS release cadence and image availability.

**CRD API maturity is heavily skewed toward alpha.** Of 192 CRDs, 93 are v1alpha1 and 10 are v1alpha2, meaning over half the platform's API surface is explicitly marked as unstable. Only 51 CRDs have reached v1. This reflects the rapid pace of feature development (llm-d, agents, MaaS, Feast, Training V2) but poses a significant upgrade burden: alpha CRDs may change without deprecation guarantees, requiring coordinated migrations across the platform.

**Observability is consistently implemented but not centrally enforced.** Most operators emit Prometheus metrics via `/metrics` endpoints on port 8443 (behind kube-rbac-proxy) or 8080 (direct), and the rhods-operator deploys ServiceMonitor resources for each component. However, distributed tracing (OpenTelemetry) adoption is uneven — agents-operator, AI Gateway, and llm-d-router support OTLP export, while most other components do not. Log aggregation relies on pod-level stdout/stderr with no structured logging standard enforced across components.
