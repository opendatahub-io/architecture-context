## Platform Overview

Red Hat OpenShift AI (RHOAI) is an enterprise AI/ML platform deployed on OpenShift Container Platform via OLM (Operator Lifecycle Manager). The central `rhods-operator` manages 16 component controllers and 5 service controllers, deploying and reconciling the full platform stack through kustomize-based manifests fetched from component repositories via `get_all_manifests.sh`. The platform spans model development (Jupyter notebooks, training operators), model serving (KServe, ModelMesh, vLLM), model management (Model Registry, MLflow), inference optimization (llm-d, Kueue), and AI safety (TrustyAI, guardrails orchestration).

Ingress architecture is transitioning from Istio VirtualService/OpenShift Routes to Gateway API HTTPRoutes with Envoy-based gateways. Authentication is enforced through a layered approach: `kube-auth-proxy` provides OAuth2/OIDC at the gateway level, `kube-rbac-proxy` sidecars protect metrics and API endpoints via Kubernetes TokenReview/SubjectAccessReview, and Istio AuthorizationPolicies enforce service mesh-level access control. The platform supports both managed (SRE-operated) and self-managed deployment modes.

The security posture combines FIPS-compliant Go binaries (using `GOEXPERIMENT=strictfipsruntime` with `CGO_ENABLED=1`), UBI9 base images across all components, non-root container execution, and capability drops. However, FIPS compliance is inconsistent: Python and Rust components lack FIPS-validated crypto libraries, and several Go components have disabled FIPS flags. TLS is enforced at webhook and metrics endpoints, with service mesh mTLS available for inter-service communication.

### Central Components

- **rhods-operator**: Central platform operator managing 16 component controllers and 5 service controllers; all components depend on it for deployment and lifecycle management
- **kserve**: Core model serving platform; 12 CRDs; depended upon by all inference runtimes (vLLM, OVMS, TGIS, MLServer, Caikit, AutoGluon)
- **kube-rbac-proxy**: Universal metrics/API authorization sidecar; used by 10+ operators and controllers
- **Gateway API / Istio**: Ingress and service mesh foundation; used by 7+ components for HTTPRoute routing
- **odh-dashboard**: Primary user interface; BFF sidecars integrate with Model Registry, MLflow, MaaS, EvalHub, and Gen AI APIs
- **kueue**: Job scheduling and resource quota management; integrates with batch Jobs, KubeRay, training operators, and AppWrappers
- **model-registry**: Central model artifact tracking; backed by ml-metadata; consumed by dashboard, pipelines, and serving components

## Deployment Architecture

### Deployment Topology

The platform uses a layered namespace model. The `rhods-operator` deploys in `redhat-ods-operator` and manages all component controllers in `redhat-ods-applications`. User workloads (notebooks, inference services, training jobs, pipelines) run in user-created namespaces. The monitoring stack (Prometheus, Alertmanager) deploys in `redhat-ods-monitoring` for managed clusters. Istio control plane components occupy `istio-system`, and optional Knative infrastructure lives in `knative-serving`. Gateway infrastructure (Envoy pods, kube-auth-proxy) spans the platform namespace and istio-system based on gateway configuration.

#### Manifest Consumption Pipeline

Component repositories maintain their own kustomize bases and overlays. The `rhods-operator` uses `get_all_manifests.sh` to fetch manifests from each component repository at a pinned commit, storing them in `./opt/manifests/` within the operator image. At runtime, the operator's component controllers apply these manifests to the cluster using Server-Side Apply, with distribution-specific overlays selected based on the `DataScienceCluster` configuration. This creates a two-stage pipeline: component repos define the desired state, and the operator materializes it on the cluster.

### Disconnected Support

The platform supports air-gapped (disconnected) deployments through the `RELATED_IMAGE_*` environment variable pattern. The `rhods-operator` declares all component images in its ClusterServiceVersion (CSV) `relatedImages` section with digest-pinned references, enabling OLM to generate the `RELATED_IMAGE_*` environment variables. Images can be mirrored to a local registry using `oc-mirror` or similar tooling. Component controllers read image references from `RELATED_IMAGE_*` environment variables or params.env ConfigMaps, ensuring no external registry access is required at runtime. Notebook images use ImageStream tags that can be overridden for disconnected environments.

## Platform Architectural Analysis

The RHOAI platform exhibits several cross-cutting architectural patterns and tensions that emerge from analyzing all 70 components together.

**Ingress Architecture Transition**: The platform is mid-migration from three ingress models (Istio VirtualService, OpenShift Routes, Knative) to Gateway API HTTPRoutes. KServe supports all three modes, while newer components (models-as-a-service, llm-d) target Gateway API exclusively. This creates operational complexity during the transition period, as different components may use different ingress mechanisms in the same cluster. The kube-auth-proxy component bridges Gateway API and legacy patterns by supporting both OAuth2 proxy and Envoy ext_authz modes.

**Authentication Layering**: The platform uses at least six distinct authentication patterns: kube-rbac-proxy sidecars (10+ components), OAuth2/OIDC via kube-auth-proxy (gateway), Istio AuthorizationPolicy (model-registry, MaaS), Kuadrant/Authorino (MaaS), application-level JWT (llama-stack), and MLflow's kubernetes-auth plugin. This heterogeneity means debugging auth failures requires understanding which layer is enforcing. Notably, 6 components (guardrails-regex-detector, guardrails-detectors, llm-d-inference-scheduler, llm-d-routing-sidecar, NeMo-Guardrails, rhds-llama-stack-distribution) have zero application-level authentication, relying entirely on NetworkPolicy for isolation.

**FIPS Compliance Inconsistency**: FIPS compliance is a significant concern. While 21 Go components correctly use `GOEXPERIMENT=strictfipsruntime` with `CGO_ENABLED=1`, the remaining components present risks: spark-operator and workload-variant-autoscaler build with `CGO_ENABLED=0` (would fail FIPS check-payload); llm-d-inference-scheduler EPP lacks strictfipsruntime while its sidecar has it (asymmetric FIPS in the same product); vllm-gaudi explicitly removes the OpenSSL FIPS provider; Python components (llama-stack's python-jose, lm-evaluation-harness) use pure-Python crypto that bypasses system OpenSSL; and Rust components (fms-guardrails-orchestrator's rustls/ring, text-generation-inference's dual TLS stack) may not use FIPS-validated crypto paths. This creates a patchwork where some request paths traverse FIPS-compliant components while others do not.

**Build Hermeticity Gap**: Go components consistently achieve hermetic builds via Hermeto gomod prefetch in Konflux pipelines. Python components lag significantly — llama-stack, llama-stack-provider-ragas, and several notebook images lack lock files or prefetch mechanisms. The GPU inference runtimes (vllm, vllm-rocm, vllm-spyre, vllm-gaudi, openvino_model_server) explicitly set `hermetic: false` in their Konflux pipelines due to large binary dependencies (CUDA, ROCm, SynapseAI) that cannot be prefetched. This creates a two-tier supply chain: Go components have strong provenance guarantees while Python/GPU components have weaker controls.

**CRD Governance at Scale**: The platform ships approximately 90+ CRDs across 15+ API groups. The rhods-operator alone defines ~23 CRDs for platform lifecycle management. Several CRDs are at alpha stability (v1alpha1) including critical inference path components (LLMInferenceService, InferencePool, VariantAutoscaling, Tenant). The overlap between kserve's InferencePool/InferenceModel and gateway-api-inference-extension's identical CRDs suggests upstream alignment in progress. CRD version migration (e.g., model-registry-operator's v1alpha1->v1beta1 conversion webhook) adds operational complexity.

**Distributed State Dependencies**: The platform has several critical stateful dependencies: etcd (ModelMesh coordination), PostgreSQL/MySQL (data-science-pipelines, mlflow, ml-metadata, model-registry, rhds-llama-stack-distribution), and S3-compatible storage (model artifacts, pipeline artifacts, evaluation results). etcd is a single point of failure for ModelMesh — if etcd becomes unavailable, model routing breaks across all ModelMesh instances. The trend toward Gateway API + llm-d scheduling may reduce the etcd dependency, but the migration path for existing ModelMesh deployments is unclear.

**Inference Runtime Proliferation**: The platform ships 10 inference runtimes (vllm, vllm-cpu, vllm-gaudi, vllm-rocm, vllm-spyre, openvino_model_server, MLServer, text-generation-inference, caikit-tgis-serving, kserve-autogluon-server) plus the Llama Stack distribution. While this provides broad hardware and framework coverage, it also means maintaining 10+ container images with different security profiles, update cadences, and API compatibility levels. The trend toward OpenAI-compatible APIs (adopted by 9+ components) provides a unifying contract, but gRPC protocol support varies (TGIS, V2/OIP, custom).

**Security Boundary Concerns**: Several components expose unauthenticated operational endpoints that could be exploited if NetworkPolicy is misconfigured: vllm-cpu exposes /pause, /resume, /update_weights, /sleep on its serving port; llm-d-routing-sidecar's SSRF protection relies on timely InferencePool pod IP updates (race conditions during scale events could permit requests to stale IPs); odh-model-controller generates self-signed Ray TLS certificates with 3-year validity and no automatic rotation; and llm-d-kv-cache defaults to InsecureSkipVerify=true for EPP metrics scraping.
