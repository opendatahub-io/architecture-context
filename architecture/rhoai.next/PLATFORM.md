## Platform Overview

Red Hat OpenShift AI (RHOAI) is an enterprise AI/ML platform deployed as an OLM-managed operator on OpenShift Container Platform. The platform operator (`rhods-operator`) implements a three-tier controller hierarchy: a `DSCInitialization` controller bootstraps platform infrastructure (namespaces, network policies, CA bundles, and singleton service CRs for Auth, GatewayConfig, and Monitoring), a `DataScienceCluster` controller manages the lifecycle of 16 individual component controllers, and each component controller reconciles its own kustomize-rendered manifests. This hierarchical design enables independent component lifecycle management and a clean separation between platform-level and component-level concerns.

The platform's ingress architecture is in active transition from OpenShift Routes to Kubernetes Gateway API. The rhods-operator auto-detects cluster capabilities and configures either an Istio-based service mesh with `data-science-gateway` Gateway resources and HTTPRoutes, or falls back to legacy OpenShift Routes. Components like KServe, Kubeflow Notebook Controllers, KubeRay, and Model Registry have migrated to Gateway API with cross-namespace HTTPRoute patterns, while others (CodeFlare Operator, odh-deployer) retain the Route-based model. Authentication is similarly in transition from OAuth proxy sidecars (RHOAI 2.x pattern) to kube-rbac-proxy sidecars with Kubernetes TokenReview/SubjectAccessReview (RHOAI 3.x pattern), with Kuadrant AuthPolicy integration emerging as the next-generation approach for Gateway API workloads.

The security posture is defense-in-depth: 72% of production container images run as non-root users (UID 1001 or distroless 65532), NetworkPolicies restrict inter-component traffic for critical services (ModelMesh, DSP, MaaS, ODH Dashboard), kube-rbac-proxy sidecars enforce authentication on operator metrics and service endpoints, and TLS is enforced throughout via OpenShift service-ca certificates, operator-managed self-signed CAs, or cert-manager integration. FIPS 140-2 compliant cipher suites are hardcoded across kube-rbac-proxy configurations.

### Central Components

**KServe** (13 reverse dependencies) is the most depended-upon component, serving as the universal model serving control plane. It manages InferenceService lifecycles for vLLM (CUDA, ROCm, Gaudi, CPU, Spyre), Caikit-TGIS, MLServer, OpenVINO, and AutoGluon runtimes. KServe also underpins the newer LLMInferenceService CRD for disaggregated prefill/decode topologies via llm-d.

**rhods-operator** (9+ reverse dependencies) is the platform root operator, deployed via OLM. It creates the DSCInitialization and DataScienceCluster CRs that orchestrate all component deployments. Every operator-based component depends on rhods-operator for lifecycle management, namespace creation, and platform configuration propagation.

**Kueue** (6 reverse dependencies) is the cross-cutting batch scheduling system. It provides quota-aware workload admission for CodeFlare/AppWrapper, KubeRay (RayCluster/RayJob), Kubeflow Training Operator (PyTorchJob, etc.), TrustyAI LMEval jobs, and Eval Hub evaluations. Its three-tier queue hierarchy (Workload -> LocalQueue -> ClusterQueue -> Cohort) enables multi-tenant GPU resource management.

**ODH Dashboard** (14 forward dependencies) is the primary user-facing component, integrating with nearly every other component through its micro-frontend plugin architecture. It depends on KServe, Data Science Pipelines, Model Registry, MLflow, Feast, TrustyAI, NeMo Guardrails, Eval Hub, OGX, and rhods-operator for UI surface area.

**kube-rbac-proxy** (sidecar, used by 8+ components) is the authentication enforcement point across the platform. TrustyAI Service Operator, Model Registry Operator, Kubeflow Notebook Controllers, KServe, ODH Dashboard, OpenVINO, vllm-spyre, and kserve-autogluon-server all use kube-rbac-proxy sidecars for metrics and API endpoint authentication.

### Authentication Mechanisms

| Pattern | Components Using | Enforcement Point |
|---------|------------------|-------------------|
| kube-rbac-proxy sidecar (TokenReview + SubjectAccessReview) | TrustyAI Service Operator, Model Registry Operator, Kubeflow Notebook Controllers, ODH Dashboard, KServe (auth proxy), OpenVINO, vllm-spyre, kserve-autogluon-server, Eval Hub | Sidecar container on ports 8443/8444, validates bearer tokens via Kubernetes API, checks `services/get` or namespace-scoped permissions |
| Kuadrant AuthPolicy (Gateway API) | odh-model-controller, KServe (LLMInferenceService), Models as a Service | AuthPolicy CRs target Gateway or HTTPRoute resources, delegate to Authorino for token validation with CEL-based authorization expressions |
| OAuth proxy sidecar (legacy) | CodeFlare Operator (RayCluster Dashboard) | OAuth proxy on port 8443 delegates to OpenShift OAuth, SubjectAccessReview for `pods:get` in namespace |
| Kubernetes RBAC (application-level) | MLflow (SelfSubjectAccessReview against mlflow.kubeflow.org pseudo-resources), Feast (Python permissions framework with OIDC + K8s token review) | Application code performs access checks against Kubernetes RBAC, using aggregate ClusterRoles (view/edit/admin) |
| mTLS (mutual TLS) | KubeRay (cert-manager CA hierarchy), CodeFlare (self-signed openssl CA), ModelMesh (etcd client certs), KServe (Istio mTLS) | Pod-to-pod encryption and identity verification for distributed workloads |
| HMAC-SHA256 JWT (internal) | Feast (operator-to-registry intra-communication token) | Lightweight internal authentication between operator and feature registry, prevents per-user auth on internal calls |

## Deployment Architecture

### Deployment Topology

The platform follows a hub-and-spoke deployment model. The **operator namespace** (`redhat-ods-operator`) contains the OLM-managed rhods-operator Deployment which serves as the single entry point for platform installation and lifecycle management. The **application namespace** (`redhat-ods-applications`) hosts all platform controllers, the ODH Dashboard (with its multi-container pod of BFF sidecars and kube-rbac-proxy), and shared infrastructure services. An optional **monitoring namespace** (`redhat-ods-monitoring`) hosts platform-specific Prometheus and monitoring resources.

User workloads are deployed in **user namespaces** where the platform controllers create per-workload resources: Notebooks as StatefulSets, InferenceServices as Deployments with multi-container pods (model server + sidecar constellation), DSPA instances with their full pipeline stack (API Server, Persistence Agent, Scheduled Workflow, Workflow Controller, MLMD, MariaDB/PostgreSQL), RayClusters with head and worker pods, and FeatureStore instances with offline/online/registry/UI deployments.

The **ingress layer** operates through the `data-science-gateway` Gateway resource in the application namespace, with HTTPRoutes created per-workload and ReferenceGrants enabling cross-namespace service references. Components that haven't migrated to Gateway API create OpenShift Routes directly in the workload namespace.

| ModelMesh Serving | `config/default/` | `config/rhoai/` | RHOAI overlay adds OpenShift Route support, monitoring integration, and different default image references |

#### Manifest Consumption Pipeline

Component manifests flow through a multi-stage pipeline: component repositories maintain kustomize bases and overlays in their `config/` or `manifests/` directories. The `get_all_manifests.sh` script in the operator repository clones component repositories at pinned commits and copies their manifests into the operator's vendored manifest tree. At reconciliation time, rhods-operator loads these vendored manifests, applies runtime transformations (image overrides from `RELATED_IMAGE_*` environment variables, namespace injection, owner references), and applies them to the cluster via manifestival. For Helm-based components (MLflow Operator), the operator embeds the Helm chart and renders templates at reconciliation time using the Helm SDK.

### Disconnected Support

RHOAI supports air-gapped (disconnected) OpenShift deployments through several mechanisms. The rhods-operator CSV declares `RELATED_IMAGE_*` environment variables for every container image used by the platform, enabling OLM's `relatedImages` mechanism to catalog all required images for mirroring. The `oc-mirror` tool uses this catalog to create mirror registries. At runtime, the operator reads `RELATED_IMAGE_*` environment variables and overrides image references in all rendered manifests, transparently redirecting from public registries to the mirror registry. For NIM (NVIDIA Inference Microservice) integration, the odh-model-controller detects `spec.components.kserve.nim.airGapped` on the DataScienceCluster CR and skips external API calls, using locally cached model catalogs instead. The DSPO managed pipelines feature uses OCI image layers as the distribution mechanism, supporting registry mirroring for pipeline definitions.

## Platform Architectural Analysis

RHOAI exhibits a consistent **operator-of-operators** architecture where the root operator (rhods-operator) manages 16 component controllers that each manage their own CRD-defined resources. This three-tier hierarchy (DSCInitialization -> DataScienceCluster -> component CRs) provides clean lifecycle management but creates a deep dependency chain: a change to DSCInitialization can cascade through all components. The action pipeline reconciler framework used by rhods-operator — sequential composable actions with preconditions and finalizers — is a mature pattern that several component operators (DSPO, TrustyAI Service Operator) have independently adopted in simplified forms.

**Authentication is the platform's most architecturally fragmented subsystem.** At least six distinct authentication mechanisms coexist: kube-rbac-proxy sidecars (the dominant RHOAI 3.x pattern), OAuth proxy sidecars (CodeFlare, legacy), Kuadrant AuthPolicy (odh-model-controller, KServe LLMInferenceService), application-level Kubernetes RBAC (MLflow SelfSubjectAccessReview), Python-level OIDC/K8s token review (Feast), and Istio mTLS (KServe Serverless mode). The Model Registry Operator documents a three-generation migration path (Istio/Authorino -> OAuth proxy -> kube-rbac-proxy), suggesting the platform is converging on kube-rbac-proxy for service endpoints and Kuadrant AuthPolicy for Gateway API workloads, but the transition is incomplete. This fragmentation means operators troubleshooting authentication failures must understand which mechanism each component uses — there is no single authentication enforcement point.

**The ingress transition from OpenShift Routes to Gateway API is half-complete**, creating a dual-path routing architecture. Components using Gateway API (KServe, Kubeflow Notebooks, KubeRay, Model Registry, MLflow, Feast) place HTTPRoutes in the platform namespace with ReferenceGrants in user namespaces, providing centralized routing control. Components still on Routes (CodeFlare, odh-deployer) create routes in user namespaces. The cross-namespace HTTPRoute pattern is architecturally elegant — it prevents users from modifying their own ingress, centralizes TLS termination, and integrates with Kuadrant for AuthPolicy — but it requires careful namespace-scoped RBAC and creates a reference counting problem for shared ReferenceGrants (solved independently by Kubeflow Notebooks and KubeRay with slightly different implementations).

**The model serving stack has significant architectural depth**, with three distinct serving architectures coexisting: KServe single-model serving (one model per InferenceService, dominant pattern), ModelMesh multi-model serving (distributed model cache with etcd coordination, Java runtime with Go controller), and llm-d disaggregated serving (prefill/decode separation with KV-cache-aware scheduling). Each architecture has different scaling characteristics, resource efficiency profiles, and operational complexity. The llm-d stack introduces a particularly sophisticated scheduling pipeline through the Gateway API Inference Extension's ext-proc protocol, with five scheduling extension points and five request control extension points mirroring the Kubernetes scheduler plugin architecture.

**Template-driven deployment is the dominant manifest generation pattern**, but implementations diverge significantly. DSPO uses Go templates with a 950-line params struct rendered via manifestival. MLflow Operator embeds a full Helm chart and uses the Helm SDK at reconciliation time. Model Registry Operator uses Go templates with k8s-objectmatcher for smart diff. ModelMesh Serving applies a chain of 14 transformation functions to a base Deployment YAML. rhods-operator renders kustomize manifests from vendored component trees. This diversity means each operator has its own manifest debugging workflow, and there is no unified template validation or preview mechanism across the platform.

**CA bundle and TLS certificate management is a cross-cutting concern handled independently by multiple components.** DSPO aggregates certificates from three sources (odh-trusted-ca-bundle, user-provided CA, service-ca) into a per-instance merged ConfigMap. KServe uses OpenShift service-ca annotations for controller certificates and Istio-managed mTLS for data plane traffic. KubeRay uses cert-manager with a two-level CA hierarchy per RayCluster. CodeFlare generates self-signed certificates via openssl in init containers. Feast uses service-serving certificates on OpenShift. The odh-trusted-ca-bundle ConfigMap (injected by OpenShift's network operator) is the closest thing to a platform-wide trust anchor, referenced by 4+ components, but each component handles TLS termination and certificate rotation differently.

**The batch scheduling integration through Kueue is architecturally clean** — the `jobframework.IntegrationCallbacks` plugin system allows each job type (Job, RayCluster, RayJob, PyTorchJob, AppWrapper) to register via Go init() functions with minimal code. However, the interaction between Kueue, CodeFlare's AppWrapper, and KubeRay creates a three-layer scheduling abstraction (AppWrapper -> Kueue Workload -> RayCluster) that can be difficult to debug when admission decisions are rejected. The platform configures 10 framework integrations by default.

**Container image security follows two dominant patterns** that reflect different upstream communities: Red Hat-originated components use UBI 9 minimal base images with UID 1001, while CNCF/Kubernetes-originated components (KServe, Kueue, Gateway API Inference Extension, Training Operator) use distroless images with UID 65532. Both patterns achieve non-root execution, but 120 out of 432 Dockerfiles lack a USER directive entirely (defaulting to root), primarily in development, test, and upstream-only Dockerfiles not shipped in production. The Konflux build system adds an additional overlay dimension, with `.konflux` suffixed Dockerfiles providing reproducible, hermetic builds for Red Hat's CI/CD pipeline.
