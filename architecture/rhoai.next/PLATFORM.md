## Platform Overview

Red Hat OpenShift AI (RHOAI) is a comprehensive AI/ML platform deployed on OpenShift Container Platform via the Operator Lifecycle Manager (OLM). The platform is orchestrated by a single monolithic operator (rhods-operator) that manages the lifecycle of 16+ component subsystems spanning model serving, training, pipelines, notebooks, feature stores, model registry, guardrails, and observability. The operator uses a component registry pattern where each subsystem registers a `ComponentHandler` during startup, enabling declarative enablement via the `DataScienceCluster` CRD.

Ingress architecture has undergone a major evolution in RHOAI 3.x, moving from per-component OpenShift Routes with oauth-proxy sidecars to a centralized Gateway API architecture. A shared Envoy-based Gateway (`data-science-gateway`) serves as the single ingress point, with per-component HTTPRoutes directing traffic to backend services. Authentication is centralized through kube-auth-proxy, which handles OAuth2/OIDC and integrates with the Envoy data plane via EnvoyFilter ext_authz. This replaces the distributed oauth-proxy sidecar pattern, reducing the number of TLS certificates and authentication endpoints from dozens to a centralized pair.

The security posture follows defense-in-depth principles across multiple layers: Gateway-level authentication via kube-auth-proxy, per-service authorization via kube-rbac-proxy sidecars (SubjectAccessReview), Istio mTLS for pod-to-pod encryption, NetworkPolicies for pod-level network segmentation, and OpenShift SecurityContextConstraints for container privilege restriction. Container images are predominantly built on Red Hat UBI 9 base images with non-root execution (70% of Dockerfiles specify non-root USER directives), and the platform supports FIPS-compliant cryptographic configurations across key components.

## Deployment Architecture

### Deployment Topology

The platform follows a hub-and-spoke namespace topology. The operator namespace (`redhat-ods-operator`) contains the OLM-managed platform operator. The applications namespace (`redhat-ods-applications`) hosts all platform controllers, the shared Gateway, Dashboard, and centralized authentication infrastructure. User workloads (notebooks, inference services, training jobs, pipelines, feature stores) are deployed into project namespaces with per-namespace isolation via RBAC, NetworkPolicies, and Istio PeerAuthentication. KServe creates per-InferenceService Knative Services or Deployments in user namespaces, while the KServe controllers remain in the platform namespace.

The RHOAI 3.x Gateway architecture deploys a shared Envoy-based Gateway in the applications namespace with GatewayClass, Gateway, and per-component HTTPRoutes. Components that previously created their own OpenShift Routes (Dashboard, MLflow, KubeRay, Model Registry) now create HTTPRoutes referencing the shared Gateway. Cross-namespace service references use ReferenceGrants to authorize traffic flow from the platform namespace to user namespaces.

#### Manifest Consumption Pipeline

Component source repositories ship kustomize bases and overlays in their `config/` or `manifests/` directories. The `get_all_manifests.sh` script in the rhods-operator repository clones each component repository at a pinned commit, copies the relevant overlay into the operator's `odh-manifests/` directory, and bundles them into the operator container image. At runtime, rhods-operator reads the `DataScienceCluster` CR, iterates through enabled components via the component registry, and applies each component's kustomize overlay using `kustomize build` with image substitutions from the operator's environment variables (set by OLM from the CSV).

### Disconnected Support

The platform supports air-gapped (disconnected) deployments through the `RELATED_IMAGE` environment variable pattern. Each operator's ClusterServiceVersion (CSV) declares `relatedImages` with digest-pinned references for all container images it deploys. The `oc-mirror` tool creates mirror registries from these references. At runtime, operators read image references from `RELATED_IMAGE_*` environment variables (set by OLM from CSV relatedImages), enabling automatic image source redirection without manifest modifications. Components that download models at runtime (vLLM, TGIS, Caikit) support offline model loading from PVC-mounted storage, and the llm-d KV cache system uses local filesystem backends. The Llama Stack K8s Operator detects and aggregates platform CA certificates from the `odh-trusted-ca-bundle` ConfigMap for custom registry trust.

## Platform Architectural Analysis

RHOAI's architecture reveals several cross-cutting patterns and tensions that emerge from examining all 70 components together.

**Monolithic Operator, Distributed Execution**: The rhods-operator is architecturally unique as a single binary that manages 16+ component subsystems via a component registry pattern. This creates a clean single-CRD control plane (DataScienceCluster) but means that any component change requires an operator release. The action chain reconciliation pattern (sequential action functions per controller) provides auditability but creates long reconciliation cycles when many components are enabled. The operator's role has expanded from component deployment to include Gateway infrastructure, authentication proxy deployment, EnvoyFilter management, and observability integration — responsibilities that in other platforms would be split across multiple operators.

**Authentication Architecture Fragmentation**: Despite the RHOAI 3.x centralization effort, the platform exhibits five distinct authentication patterns: (1) kube-rbac-proxy sidecar with SubjectAccessReview, (2) kube-auth-proxy centralized OAuth2/OIDC, (3) OpenShift oauth-proxy sidecar, (4) Authorino gateway-level auth with API keys, and (5) Istio mTLS. The migration from oauth-proxy to kube-rbac-proxy is partially complete (Model Registry Operator includes automatic migration logic), and the Gateway-based kube-auth-proxy coexists with per-service kube-rbac-proxy sidecars. This creates a complex authentication topology where a single request may traverse three authorization checkpoints (Gateway → kube-auth-proxy → BFF middleware in Dashboard, or Gateway → Authorino → kube-rbac-proxy in MaaS). The tension between centralized auth (simpler operations) and per-service auth (finer-grained control) is a recurring architectural decision point.

**Inference Stack Stratification**: The model serving architecture has three distinct tiers that reflect evolutionary layering: (1) ModelMesh Serving for multi-model, CPU-optimized serving with etcd-based distributed LRU cache, (2) KServe InferenceService for single-model GPU serving with Knative serverless scaling, and (3) KServe LLMInferenceService with llm-d for LLM-specific features (disaggregated P/D, prefix cache-aware routing, InferencePool-based scheduling). Each tier has its own controller, CRDs, and operational model. The llm-d stack introduces Gateway API Inference Extension as a fourth integration point, with ext-proc-based scheduling that bypasses KServe's traditional Knative routing entirely. This stratification means operators must understand which serving tier to use for a given workload — a knowledge gap the Dashboard UI attempts to bridge but cannot fully abstract.

**Template vs. Imperative Resource Generation**: Components are split between two reconciliation patterns. Template-based operators (DSPO, ModelMesh Serving, Model Registry Operator, MLflow Operator) render Kubernetes manifests from Go/Helm templates with CR-derived parameters, providing clear separation of concerns but making the deployed topology opaque to static analysis. Imperative operators (KServe, Kueue, Training Operator, CodeFlare) build resources programmatically, which is easier to trace in code but harder to customize. The rhods-operator itself uses a hybrid approach — kustomize overlays for component deployment but imperative Go code for Gateway infrastructure. This inconsistency makes platform debugging harder, as operators must understand each component's specific resource generation strategy.

**Distributed Scheduling Convergence**: Three independent scheduling systems operate within the platform: Kueue (job-level quota and admission), llm-d-inference-scheduler (per-request inference routing via ext-proc), and WVA (model-level autoscaling via Prometheus metrics). Kueue integrates with Training Operator, Spark, KubeRay, and JobSet via a plugin framework. The llm-d scheduler implements its own three-phase pipeline (filter → score → pick) inspired by the Kubernetes scheduler. WVA watches InferencePool CRs and EPP metrics to scale model server replicas. These systems operate at different granularities (job vs. request vs. replica) but share no common scheduling framework, creating potential for conflicting scaling decisions when all three are active on the same workload.

**Observability Stack Complexity**: The platform deploys observability infrastructure across multiple layers: Prometheus ServiceMonitors for operator metrics, Istio telemetry for service mesh metrics, OpenTelemetry for distributed tracing (optional, via OTel Operator), Perses for visualization dashboards, and per-component metrics endpoints. Components expose metrics inconsistently — some via dedicated metrics ports with kube-rbac-proxy sidecars, others via application ports, and others via Istio sidecar-injected endpoints. The transition from Grafana to Perses for observability dashboards adds another migration vector.

**Security Posture Asymmetry**: While 70% of Dockerfiles specify non-root execution, 105 (25%) lack USER directives entirely (defaulting to root), and 19 explicitly run as root. The llm-d-inference-scheduler defaults `InsecureSkipVerify` to true for model server metrics scraping, meaning the EPP doesn't validate pod identities in production by default. Container images use a mix of digest-pinned (24), tag-pinned, and unpinned base images. The platform's FIPS compliance is partial — kube-auth-proxy, kube-rbac-proxy, and odh-model-controller enforce FIPS cipher suites, but many Python-based services rely on system-level OpenSSL FIPS configuration without application-level enforcement.

**Evaluation and Guardrails Ecosystem**: The platform has developed a rich but architecturally complex evaluation stack: TrustyAI Service Operator manages LMEval jobs and guardrails orchestrators, Eval Hub orchestrates evaluation providers, lm-evaluation-harness runs batch evaluations, Llama Stack providers (Ragas, Garak) bridge to Kubeflow Pipelines for distributed evaluation, and MLflow tracks experiment results. This creates multiple evaluation paths with different execution models (direct Job, KFP pipeline, eval-hub adapter), making it difficult to provide a unified evaluation experience.
