## Platform Overview

Red Hat OpenShift AI (RHOAI) is an enterprise AI/ML platform deployed on OpenShift Container Platform via a single OLM-managed operator (`rhods-operator`). The operator uses a three-level controller hierarchy — DSCInitialization bootstraps platform infrastructure (namespaces, networking, monitoring, auth, gateway), DataScienceCluster orchestrates 16+ component CRs based on user enablement choices, and per-component controllers render and deploy component-specific manifests via Kustomize. A fourth "modules" level manages external operator dependencies (cert-manager, Istio/Sail, LeaderWorkerSet) via Helm. The platform supports three operational modes: SelfManagedRhoai (on-premises OpenShift), ManagedRhoai (cloud-hosted), and XKS (non-OpenShift Kubernetes via a Platform CR).

Ingress follows a Gateway API architecture introduced in RHOAI 3.x, replacing the legacy Routes + oauth-proxy sidecar pattern. The rhods-operator deploys a centralized ingress stack: Gateway CR → Envoy proxy → EnvoyFilter (ext_authz + Lua cookie/token handling) → kube-auth-proxy (OIDC/OpenShift OAuth) → component backends. This provides unified authentication at the gateway layer, with an Envoy Lua script that strips OAuth cookies from upstream requests and injects Bearer tokens into the Authorization header. Legacy hostnames are handled via nginx 301-redirect Deployments.

The security posture is defense-in-depth: kube-rbac-proxy sidecars protect metrics and API endpoints via Kubernetes TokenReview/SubjectAccessReview, the gateway stack enforces authentication before traffic reaches backends, and NetworkPolicies restrict inter-component communication. Container images are built on UBI 9 with FIPS-compliant Go crypto (CGO_ENABLED=1 + strictfipsruntime build tag), though several components have identified FIPS gaps. The platform supports air-gapped (disconnected) deployments via the `RELATED_IMAGE_*` pattern for digest-pinned images and OLM relatedImages for image mirroring.

### Central Components

**rhods-operator** is the platform root — every operator-managed component depends on it for deployment, RBAC, networking, and monitoring. It owns 16+ component CRs and 3 service CRs (Auth, Gateway, Monitoring), making it the single coordination point for the entire platform.

**KServe** is the serving hub — it manages InferenceService and LLMInferenceService CRDs that virtually every serving-related component interacts with. KServe is depended on by vLLM, odh-model-controller, TrustyAI, Model Registry, MaaS, WVA, the llm-d inference scheduler, and the dashboard.

**odh-dashboard** is the user-facing integration point — it aggregates APIs from nearly every other component (pipelines, notebooks, model registry, serving, evaluation, GenAI, MaaS, MLflow) through its module federation architecture with 10+ BFF (Backend-for-Frontend) servers.

**odh-model-controller** is the serving infrastructure controller — it bridges KServe with the platform's Gateway API ingress, authentication (Kuadrant/Authorino), metrics (Prometheus), and autoscaling (KEDA) systems.

**Kueue** is the resource scheduling backbone — it manages workload admission for training jobs (via Kubeflow Training Operator, Kubeflow Trainer), Ray clusters (via CodeFlare/KubeRay), and evaluation jobs (via TrustyAI/EvalHub).

**kube-rbac-proxy** is the ubiquitous security sidecar — deployed alongside nearly every operator and service to protect metrics endpoints and APIs via Kubernetes RBAC.

## Deployment Architecture

### Deployment Topology

The platform deploys across three primary namespace tiers. The **operator namespace** (`redhat-ods-operator`) runs the rhods-operator manager and cloudmanager binaries. The **platform namespace** (`redhat-ods-applications`) hosts all controllers, the dashboard, monitoring infrastructure, and shared services. **User namespaces** (data science projects) contain per-tenant workloads: Notebook StatefulSets, InferenceService Deployments, DSPA pipeline infrastructure, RayClusters, FeatureStores, MLflow instances, and OGX servers. A separate **monitoring namespace** (`redhat-ods-monitoring`) isolates Prometheus, Thanos Querier, OpenTelemetry Collector, and Perses.

The Gateway infrastructure spans the platform namespace with an Envoy proxy Deployment, kube-auth-proxy Deployment, and EnvoyFilter configurations. HTTPRoutes from user namespaces reference the platform Gateway for centralized ingress. NetworkPolicies in each namespace restrict ingress to authorized sources (Gateway, monitoring, operator namespaces).

#### Manifest Consumption Pipeline

Component manifests flow from upstream component repositories through the `get_all_manifests.sh` script in rhods-operator, which clones specific commits/tags of each component repo and copies their `config/` or `manifests/` directories into the operator's manifest tree. The operator build then embeds these manifests into the container image. At runtime, component controllers read manifests from the embedded filesystem, render them via Kustomize with `RELATED_IMAGE_*` environment variable substitution, and apply the rendered resources to the cluster. Konflux CI builds pin all component versions via `params.env` files containing image digests.

### Disconnected Support

RHOAI supports air-gapped (disconnected) deployments through several mechanisms. The `RELATED_IMAGE_*` environment variable pattern provides digest-pinned image references for every component image, injected by the OLM CSV's `relatedImages` section. The `oc-mirror` tool can mirror all required images to a private registry. The rhods-operator reads image references from environment variables rather than hardcoded registries, allowing transparent image redirection. Specific components (odh-model-controller NIM integration) have air-gapped modes that skip external API calls and use locally-provided configuration. The kserve-storage-initializer supports pulling models from private S3-compatible storage and private OCI registries.

## Platform Architectural Analysis

The RHOAI platform reveals several cross-cutting architectural patterns and tensions that emerge from reading all 76 component analyses together.

**Converging on Gateway API, but the transition is incomplete.** The platform is in an active migration from the legacy Routes + oauth-proxy sidecar model to a centralized Gateway API ingress stack. The rhods-operator deploys the new Gateway infrastructure, KServe creates HTTPRoutes for InferenceServices, odh-dashboard uses HTTPRoutes, MLflow Operator creates HTTPRoutes, and Model Registry Operator creates HTTPRoutes. However, ModelMesh Serving still uses OpenShift Routes with oauth-proxy sidecars, and the rhods-operator maintains nginx redirect Deployments for legacy hostname compatibility. This creates a period where both ingress patterns coexist, with different security enforcement points.

**Three distinct authentication paradigms coexist.** The platform uses kube-rbac-proxy sidecars (TokenReview + SAR) for metrics and internal APIs, the centralized Gateway ext_authz + kube-auth-proxy for user-facing services, and Kuadrant AuthPolicy + Authorino for fine-grained model serving authentication. MaaS adds a fourth layer with application-level API key validation. While each pattern serves a legitimate purpose, the diversity increases operational complexity — troubleshooting an authentication failure requires understanding which of the four mechanisms applies to the specific request path. The kube-auth-proxy is positioned as a FIPS-compliant drop-in replacement for oauth-proxy, suggesting the legacy oauth-proxy sidecars in ModelMesh will eventually be migrated.

**The Envoy ext_proc pattern is proliferating rapidly.** Four independent implementations use Envoy's External Processing protocol: the llm-d Inference Scheduler (EPP) for intelligent inference routing, the Gateway API Inference Extension (EPP + BBR) for body-based routing, the AI Gateway Payload Processing service for API translation and credential injection, and the MaaS payload-processing/payload-pre-processing services for model name extraction and auth preparation. While they share the same protocol, each is implemented independently with different lifecycle management. This suggests a potential convergence opportunity toward a shared ext_proc framework, or at minimum, standardized operational patterns for deploying and monitoring ext_proc services.

**Inference scheduling is the most architecturally sophisticated subsystem.** The llm-d stack (llm-d containers, inference scheduler, routing sidecar, KV-cache indexer, WVA autoscaler, planner) represents the most complex subsystem in the platform. It implements disaggregated prefill/decode serving, KV-cache-aware routing via ZMQ event streams, priority-based flow control with queuing theory, latency prediction via ML models, and session affinity with SSRF protection. This subsystem touches every layer of the stack (container runtime, networking, scheduling, monitoring) and introduces its own CRDs (InferencePool, InferenceObjective, InferenceModelRewrite). The architectural tension here is between the upstream Gateway API Inference Extension (community-driven) and the llm-d Inference Scheduler (which extends and diverges from it).

**Module federation creates a scalable UI architecture, but with operational complexity.** The odh-dashboard's module federation architecture (10+ React micro-frontends + Go BFFs) elegantly solves the problem of feature velocity across 10+ teams. However, the sidecar deployment mode (all BFFs in one pod with 8+ ports) creates a "blast radius" concern — a crash in any BFF sidecar affects the entire dashboard pod. The standalone deployment mode mitigates this but increases the number of Deployments, Services, and RBAC resources to manage. The dashboard-operator's dependency resolution algorithm (evaluating DSC component states, inter-module dependencies, and admin overrides) adds a layer of indirection that can make it difficult to understand why a particular module is or isn't deployed.

**FIPS compliance is mostly achieved but has documented gaps.** The platform has standardized on `CGO_ENABLED=1` + `-tags strictfipsruntime` for FIPS-compliant Go crypto in UBI-based images. However, three specific gaps are documented: (1) KServe's kserve-module-controller uses `CGO_ENABLED=0`, (2) the llm-d EPP Dockerfile omits `-tags strictfipsruntime`, and (3) Python-based components (MLflow, Feast, vLLM, Caikit, OGX) rely on system OpenSSL via the UBI base image but don't explicitly verify FIPS mode at the application level. These gaps don't prevent `check-payload` CI gates from passing but may result in non-FIPS-validated crypto operations at runtime.

**The operator-of-operators pattern introduces deployment ordering challenges.** The rhods-operator deploys component operators, which in turn deploy user-facing services. This creates a three-level deployment chain: rhods-operator → component operator → workload. Each level has its own health checking, status reporting, and error handling. The cloudmanager adds a fourth level for non-OpenShift deployments by deploying dependency operators (cert-manager, Istio, LWS) via Helm before the main operator chain starts. Status propagation across these levels is handled by conditions on CRs, but the lack of a unified status aggregation mechanism means that a failure deep in the chain (e.g., a CRD dependency missing for a sub-component) may be difficult to diagnose from the top-level DSC status.

**Training and evaluation are converging on a unified SDK model.** The Kubeflow SDK provides a single Python entry point for training (Trainer), hyperparameter optimization (Katib), Spark jobs, and model registry operations. The training-hub library adds algorithm-specific backends (SFT, OSFT, LoRA, GRPO) with pluggable execution engines (InstructLab, Unsloth, verl). EvalHub provides evaluation orchestration with framework adapters (lm-evaluation-harness, Garak). The RHOAI MCP Server wraps all of these as MCP tools for AI agents. This layered composition pattern enables progressive complexity — simple training via `kubeflow.trainer`, advanced algorithms via training-hub, and autonomous workflows via MCP — but the abstraction layers can make it difficult to debug failures at the execution layer.
