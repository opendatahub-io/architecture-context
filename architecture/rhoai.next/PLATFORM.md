## Platform Overview

Red Hat OpenShift AI (RHOAI) is an enterprise ML/AI platform built on OpenShift Container Platform, managed by the central `rhods-operator`. The operator uses a hierarchical CRD model where `DSCInitialization` provisions platform infrastructure (namespaces, networking, monitoring, authentication, gateway) and `DataScienceCluster` manages the lifecycle of 16+ data science components including Dashboard, KServe, Kueue, Workbenches, Model Registry, Data Science Pipelines, TrustyAI, Ray, Training Operator, Trainer, Feast, LlamaStack, MLflow, Spark, Model Controller, and Models-as-a-Service. Each component can be independently enabled, disabled, or set to unmanaged via its own component CRD under the `components.platform.opendatahub.io` API group.

The platform deploys a Gateway API-based ingress architecture with Envoy as the data plane, kube-auth-proxy for OAuth2/OIDC authentication at the gateway level, and EnvoyFilter CRs for traffic shaping. Three authentication modes are supported: integrated OpenShift OAuth, external OIDC, and no-auth (external management). Model serving uses KServe with InferenceService and LLMInferenceService CRDs, supporting Knative serverless, raw Kubernetes, and Gateway API ingress modes. The distributed inference stack (llm-d) provides disaggregated prefill/decode with KV-cache-aware routing, while the Models-as-a-Service subsystem adds multi-tenant LLM gateway capabilities with Kuadrant-based rate limiting and API key management.

The security posture follows a defense-in-depth model with kube-rbac-proxy sidecars for SubjectAccessReview-based authentication on component endpoints, OpenShift service-serving certificates for automatic TLS provisioning, optional Istio service mesh with mTLS PeerAuthentication, and FIPS compliance via `GOEXPERIMENT=strictfipsruntime` in Go builds. The monitoring stack includes MonitoringStack (Prometheus), ThanosQuerier, Tempo for distributed tracing, OpenTelemetry Collector, and Perses for observability dashboards. All components are built via the Konflux CI/CD system using UBI9 base images with hermetic builds and cachi2 dependency prefetch.

### Central Components

- **rhods-operator**: Central orchestrator — all other components depend on it for deployment and lifecycle management
- **kserve**: Core model serving platform — inference runtimes, model controller, llm-d stack, and dashboard all depend on it
- **kube-rbac-proxy**: Authentication sidecar — used by Dashboard, DSP operator, TrustyAI, model serving, operator metrics endpoints
- **kube-auth-proxy**: Gateway authentication — all externally-accessible services route through it
- **kueue**: Job scheduling — Training Operator, Trainer, KubeRay, DSP, and TrustyAI LMES integrate with it for quota management
- **odh-dashboard**: User interface — proxies to nearly all other platform services

## Deployment Architecture

### Deployment Topology

The platform follows a namespace-separated deployment topology. The central `rhods-operator` runs in `redhat-ods-applications` as a single-replica OLM-managed Deployment, deploying all component operators into the same namespace. The monitoring stack (MonitoringStack, ThanosQuerier, Tempo, OTEL Collector, Perses) is deployed into `redhat-ods-monitoring`. The gateway infrastructure (Envoy data-science-gateway, kube-auth-proxy, dashboard-redirect nginx) runs in `openshift-ingress`. User workloads (notebooks, InferenceServices, DSPAs, training jobs) run in user-created namespaces. The MaaS subsystem uses `models-as-a-service` for its CRs and `kuadrant-system` for rate limiting infrastructure.

- **Operators**: 20+ (rhods-operator, kserve [4 controllers], kueue, kuberay, codeflare-operator, data-science-pipelines-operator, trustyai-service-operator, model-registry-operator, models-as-a-service, training-operator, trainer, feast, llama-stack-k8s-operator, mlflow-operator, spark-operator, kubeflow [2 controllers], modelmesh-serving, odh-model-controller)

#### Manifest Consumption Pipeline

Component manifests flow from individual component repositories through the `get_all_manifests.sh` script (run during `rhods-operator` image build) into the operator's `/opt/manifests/` directory. Each component specifies a Git repository URL and commit SHA in the manifest script. At runtime, the rhods-operator reads manifests from `/opt/manifests/{component}/`, renders them with kustomize (or Helm for chart-based components), applies image parameter substitutions from RELATED_IMAGE environment variables (set in the OLM CSV), and deploys them to the cluster using server-side apply. This pipeline ensures all component manifests are pinned to specific commits and embedded in the operator image, enabling fully disconnected deployment without network access to Git repositories.

### Disconnected Support

All RHOAI components support disconnected (air-gapped) deployment through the RELATED_IMAGE pattern. Each operator's OLM ClusterServiceVersion (CSV) declares all container image references as `RELATED_IMAGE_*` environment variables, which are included in the `relatedImages` section of the CSV. The `oc-mirror` tool uses these references to mirror all required images to a local registry. All images are referenced by SHA256 digest (not tags) to ensure immutable, reproducible deployments. Component manifests are embedded in the operator image at build time via `get_all_manifests.sh`, eliminating runtime Git access requirements. The Konflux build system produces hermetic builds with cachi2 dependency prefetch, ensuring all build dependencies are captured and reproducible.

## Platform Architectural Analysis

The RHOAI platform exhibits several cross-cutting architectural patterns and tensions that emerge from reading all 69 component analyses together.

**Hierarchical Operator Pattern**: The rhods-operator implements a sophisticated two-tier CRD model (DSCInitialization → DataScienceCluster → per-component CRDs) that enables clean separation between infrastructure provisioning and component lifecycle. This pattern allows individual components to be independently enabled, disabled, or set to unmanaged state, providing flexibility for different deployment scenarios (full platform, model serving only, training only, xKS cloud deployments). The component registry pattern with handler interfaces enables a plugin-like architecture that scales well as new components are added.

**Authentication Inconsistency**: The platform uses at least five distinct authentication mechanisms: kube-rbac-proxy (SubjectAccessReview) for most operator endpoints, kube-auth-proxy (OAuth2/OIDC) at the gateway level, Kuadrant/Authorino for MaaS API key and OIDC validation, platform-delegated (no auth) for inference runtime containers, and API key middleware in vllm-cpu. This heterogeneity reflects the platform's evolution from multiple upstream projects (KServe, Kubeflow, IBM Watson derivatives, Meta LlamaStack) that each brought their own security models. While the gateway-level authentication provides a unifying layer, the internal authentication landscape requires careful documentation for operators.

**Dual-Protocol Inference**: The consistent pattern of HTTP (OpenAI-compatible on port 8000) + gRPC (TGIS on port 8033) across vLLM variants, TGIS, and Caikit-TGIS-Serving demonstrates strong architectural alignment for model serving. The `vllm_tgis_adapter` bridges vLLM's native API to the TGIS protocol in a single process, enabling backward compatibility with IBM's Watson-era inference stack while providing modern OpenAI API compatibility. This dual-protocol pattern is a pragmatic compromise between ecosystem standards and legacy platform requirements.

**Thin Wrapper Container Pattern**: Five vLLM variants (CUDA, ROCm, CPU, Gaudi, Spyre) follow a "thin wrapper" pattern where the repository contains only a Dockerfile and no application source code. All logic resides in upstream RHAIIS base images. This pattern minimizes maintenance overhead and keeps the build-only repositories focused on Konflux pipeline integration, labeling, and environment variable defaults. However, it means that understanding the actual runtime behavior requires access to the RHAIIS base image contents, which are not inspectable from the component repositories alone.

**Gateway Architecture Complexity**: The data-science-gateway implementation in rhods-operator is remarkably complex, managing 14+ Kubernetes resource types across two namespaces. The three-layer proxy chain (Envoy → kube-auth-proxy ext_authz → Lua filter → backend) provides powerful authentication capabilities but creates debugging challenges. The cookie management logic (stripping `_oauth2_proxy` cookies including split variants) and the dual-mode support (OCP Route vs LoadBalancer) add significant operational complexity. The addition of Envoy ext_proc for MaaS BBR and inference routing adds a fourth processing layer.

**Module Federation UI Architecture**: The odh-dashboard's Module Federation approach with 7 independently deployable Go BFF sidecars is architecturally novel for Kubernetes platforms. Each module ships as a separate container with its own BFF, enabling independent development and deployment cycles. However, this creates a complex pod with 9+ containers (dashboard, kube-rbac-proxy, 7 BFF modules), each with its own TLS certificates and upstream service dependencies. The inter-BFF communication pattern (e.g., gen-ai BFF calling maas BFF for token issuance) adds additional complexity.

**Kustomize + params.env Uniformity**: The ConfigMapGenerator + params.env pattern is used consistently across nearly all operator components, providing a clean mechanism for image reference management. This pattern enables the RELATED_IMAGE disconnected deployment flow and provides a single point of change for image updates. The distribution variants (ODH vs RHOAI) are cleanly handled through kustomize overlays that change namespace, RBAC, and monitoring configurations without code duplication.

**Training Stack Transition**: The platform ships both the legacy Kubeflow Training Operator (6 job types) and the next-gen Kubeflow Trainer (TrainJob/TrainingRuntime with plugin framework). This dual-shipping indicates an ongoing migration where the Trainer will eventually replace the Training Operator. The Trainer introduces new patterns (plugin framework, RHAI progression tracking, NetworkPolicy per TrainJob, 16 ClusterTrainingRuntime variants) that represent a significant architectural evolution.

**Distributed LLM Inference Stack**: The llm-d ecosystem (llm-d, llm-d-inference-scheduler, llm-d-kv-cache, llm-d-routing-sidecar, workload-variant-autoscaler, gateway-api-inference-extension) represents the most architecturally sophisticated subsystem, implementing disaggregated prefill/decode with KV-cache-aware routing, flow control with scale-from-zero, and queueing-theory-based autoscaling. The integration with KServe's LLMInferenceService controller and Gateway API InferencePool/InferenceModel CRDs creates a complete inference lifecycle from deployment through routing to autoscaling.

**Observability Stack Depth**: The monitoring controller in rhods-operator deploys a full observability stack (MonitoringStack/Prometheus, ThanosQuerier, Tempo, OTEL Collector, Perses) as an integrated subsystem. This is more comprehensive than most Kubernetes platforms and reflects the need for deep visibility into ML/AI workloads. However, the atomic deployment pattern (all-or-nothing based on CRD availability) could limit flexibility for operators who want partial observability deployment.
