## Platform Overview

Red Hat OpenShift AI (RHOAI) is a comprehensive AI/ML platform deployed on OpenShift Container Platform via OLM (Operator Lifecycle Manager). The platform is orchestrated by a single meta-operator (`rhods-operator`) that uses a component registry pattern to provision and lifecycle-manage 16+ sub-components through two primary custom resources: `DataScienceCluster` (DSC) for component enablement and `DSCInitialization` (DSCI) for platform-wide configuration including monitoring, service mesh, and trusted CA bundles. The operator deploys a sophisticated multi-layer ingress architecture based on Gateway API (GatewayClass → Gateway → HTTPRoutes) with an Envoy data plane, replacing the RHOAI 2.x pattern of per-component OpenShift Routes and oauth-proxy sidecars.

The platform spans the full AI/ML lifecycle: interactive development (Jupyter notebooks via Kubeflow Notebook Controllers), data pipelines (Data Science Pipelines based on Kubeflow Pipelines v2), distributed training (Kubeflow Training Operator, CodeFlare/KubeRay for Ray clusters), model serving (KServe for single-model, ModelMesh for multi-model, vLLM/TGIS/OpenVINO as runtimes), model registry (ML Metadata-backed registry with operator), model evaluation and safety (TrustyAI, LM Evaluation Harness, FMS Guardrails Orchestrator), and LLM-specific infrastructure (llm-d disaggregated inference, Gateway API Inference Extension for intelligent routing, Models-as-a-Service for multi-tenant API gateway). Workload scheduling is unified through Kueue, which provides quota management and fair-sharing across all job types.

The security posture follows defense-in-depth: centralized authentication via kube-auth-proxy (OAuth2/OIDC) at the gateway layer, kube-rbac-proxy sidecars for per-service SubjectAccessReview authorization, NetworkPolicies for pod-to-pod isolation, OpenShift service-ca for internal TLS, and FIPS-compliant cryptographic libraries across all Go components (via BoringCrypto build tags). Container images are predominantly UBI-based with non-root execution (USER 1001 or 65532), and builds use Konflux/cachi2 for hermetic, reproducible supply chains.

### Central Components

**rhods-operator (RHOAI Operator)**: The platform's meta-operator and single control plane. Uses a component registry pattern to provision 16+ sub-components via DSC/DSCI custom resources. Deploys and manages: KServe, Kueue, Kubeflow Training Operator, Data Science Pipelines Operator, CodeFlare Operator, Notebooks, odh-dashboard, Model Registry Operator, Feast, Spark Operator, TrustyAI Service Operator, odh-model-controller, kube-auth-proxy, Models-as-a-Service, and others. Owns 28 CRDs — more than any other component.

**KServe**: The primary model serving control plane, depended on by 11+ components. Manages InferenceService, LLMInferenceService, and LocalModelCache CRDs through four separate controller binaries. All serving runtimes (vLLM, TGIS, MLServer, OpenVINO, Caikit) deploy as KServe InferenceService pods. Uses Go `//go:build distro` tags to cleanly separate OpenShift-specific behavior.

**odh-dashboard**: The user-facing frontend and API gateway, with 18 internal dependencies — the most dependent component. Implements a Webpack Module Federation architecture with BFF (Backend-for-Frontend) sidecars for domain-specific modules (model registry, gen-ai, MaaS). Three-layer authentication: Gateway → kube-rbac-proxy → BFF middleware.

**Kueue**: Central workload scheduling and quota management. Integrates with KubeRay, Kubeflow Training Operator, and TrustyAI's LMES controller via a plugin-based job framework. Provides cluster-level resource fairness across all AI/ML job types.

**Models-as-a-Service (MaaS)**: The multi-tenant API gateway layer. Aggregates user-facing access policies into Kuadrant AuthPolicy/RateLimitPolicy resources. Implements three-layer security: Authorino (authn/authz), Limitador (rate limiting), and BBR payload processing (credential injection/stripping).

## Deployment Architecture

### Deployment Topology

The platform uses a three-tier namespace topology. The **operator tier** (`redhat-ods-operator`) contains the OLM-managed rhods-operator deployment, which reconciles DSC/DSCI custom resources. The **platform tier** (`redhat-ods-applications`) hosts all controller deployments, the centralized Gateway (Envoy), kube-auth-proxy, odh-dashboard, and platform-wide services like Kueue and Model Registry Operator. The **workload tier** spans user-created namespaces where notebooks, pipeline runs, training jobs, inference services, and Ray clusters execute. Some components bridge tiers: KServe controllers run in the platform namespace but manage InferenceService pods in workload namespaces; DSPO runs in the platform namespace but creates pipeline infrastructure (API server, MariaDB, Argo, MLMD) per-DSPA in workload namespaces.

The Gateway API architecture in RHOAI 3.x deploys a shared Envoy-based Gateway in the platform namespace with HTTPRoutes providing path-based routing to individual services. EnvoyFilter CRs integrate ext_authz with kube-auth-proxy for centralized OAuth2/OIDC authentication. This replaces the 2.x model where each component had its own OpenShift Route and oauth-proxy sidecar, reducing the total number of TLS-terminating proxies and simplifying certificate management.

#### Manifest Consumption Pipeline

Component manifests flow from upstream repositories through a multi-stage pipeline. Each component repository contains its own `config/` or `manifests/` directory with Kustomize bases and overlays. The `get_all_manifests.sh` script in the operator repository clones component repos at pinned commits and copies manifests into the operator's embedded resource directory. During operator build, these manifests are embedded in the binary (Go `//go:embed` or COPY in Dockerfile). At runtime, the rhods-operator renders component manifests using the manifestival library or direct `kubectl apply`-equivalent logic, injecting distribution-specific values (image references via `RELATED_IMAGE_*` environment variables, namespace names, monitoring configuration) from the DSC/DSCI custom resources.

### Disconnected Support

RHOAI supports air-gapped (disconnected) deployments through the `RELATED_IMAGE_*` pattern. Each operator declares all container image references as `RELATED_IMAGE_` environment variables in its deployment manifest, which OLM extracts into the ClusterServiceVersion's `relatedImages` field. This enables `oc-mirror` to discover and mirror all required images to a private registry. At runtime, operators resolve image references from these environment variables rather than hardcoding registry URLs, allowing transparent image source substitution. Components that dynamically reference images (DSPO for pipeline runtime images, KServe for storage initializers, TrustyAI for evaluation containers) also use RELATED_IMAGE variables to ensure all transitive image dependencies are captured for mirroring. 13 components explicitly implement this pattern.

## Platform Architectural Analysis

**Operator Sprawl and the Meta-Operator Pattern**: RHOAI's most distinctive architectural characteristic is the meta-operator pattern — a single `rhods-operator` that provisions 16+ sub-operators through a component registry. This creates a two-level operator hierarchy where the platform operator manages the lifecycle of domain-specific operators (KServe, DSPO, Kueue, etc.), which in turn manage user workloads. The pattern provides a unified control plane and consistent component enablement, but introduces a monolithic binary that must understand the deployment requirements of every component. With 28 CRDs owned by rhods-operator alone and 135 CRDs across the platform, the CRD surface area is substantial. The action-chain reconciliation pattern within rhods-operator (e.g., the Gateway controller's 10-step sequence) provides auditability but means failures in early actions can cascade.

**Ingress Architecture Migration (2.x → 3.x)**: The shift from per-component OpenShift Routes with oauth-proxy sidecars to centralized Gateway API with kube-auth-proxy represents a fundamental architectural transition. The 3.x model reduces the number of TLS-terminating proxies, centralizes authentication, and enables policy-based access control via Kuadrant. However, the migration creates a period where both patterns coexist — some components still support Route-based ingress for backward compatibility, and the `odh-model-controller` manages both OpenShift Routes (RHOAI 2.x) and HTTPRoutes + AuthPolicies (RHOAI 3.x). This dual-mode operation increases controller complexity and RBAC surface area.

**Inconsistent Authentication Patterns**: Despite the centralization trend, authentication mechanisms vary significantly across the platform. KServe InferenceServices use Kuadrant AuthPolicy (Authorino) for Gateway API-based auth; the dashboard uses kube-rbac-proxy for SubjectAccessReview; ModelMesh relies on Istio mTLS plus its own etcd-based authorization; CodeFlare injects oauth-proxy sidecars via webhook; and inference runtimes (vLLM, TGIS, MLServer) expose completely unauthenticated endpoints, delegating auth to the serving layer above them. MaaS adds yet another pattern with Authorino API key validation via HTTP callback. This heterogeneity means security audits must understand 5+ distinct authentication flows.

**Model Serving Duality (KServe vs. ModelMesh)**: The platform maintains two fundamentally different model serving architectures. KServe follows a one-pod-per-model pattern with Knative autoscaling (including scale-to-zero), while ModelMesh implements a shared-pod multi-model pattern with etcd-based distributed state. These architectures have different performance characteristics, different failure modes, and different operational patterns. The `odh-model-controller` and dashboard abstract over this duality, but the underlying complexity surfaces in debugging, capacity planning, and networking configuration.

**LLM-Specific Infrastructure Layer**: The platform has developed a significant infrastructure layer specifically for LLM workloads that sits alongside the traditional ML serving stack. This includes: the llm-d project (disaggregated inference with prefill/decode separation, KV cache management, NIXL-based memory transfer), the Gateway API Inference Extension (intelligent request routing based on prefix cache hits, queue depth, and KV utilization), the vLLM orchestrator gateway (guardrails-aware routing), and the LLMInferenceService CRD (multi-node, disaggregated P/D). This layer introduces 5+ additional sidecars and services per LLM deployment, creating a substantially different deployment topology from traditional model serving.

**Guardrails Pipeline Complexity**: The FMS Guardrails Orchestrator implements a sophisticated streaming pipeline where inference responses are chunked, fanned out to multiple detector services in parallel, aggregated with ordering guarantees, and returned to clients. The orchestrator's dual-protocol architecture (gRPC for generation backends, HTTP for detectors) and its streaming detection pipeline (with `DashMap` for lock-free concurrent state, actor-pattern batcher management) represent some of the most complex runtime logic in the platform. The vllm-orchestrator-gateway adds another layer by wrapping the orchestrator with gateway-aware routing. Each layer in this chain (BBR → EPP → vLLM → Orchestrator → Detectors) adds latency and failure modes.

**Build System Divergence**: Component build systems span three distinct patterns: (1) Konflux/cachi2 hermetic builds with prefetched dependencies and Tekton pipelines (notebooks, vLLM, most container images), (2) standard multi-stage Dockerfiles with `go build` or `pip install` (Go operators, Python services), and (3) build-wrapper repositories that layer on top of upstream base images without compiling application code (vLLM downstream wrapper, vllm-rocm, vllm-spyre). The Konflux pattern provides the strongest supply chain guarantees but requires maintaining lock files (`rpms.lock.yaml`, `pylock.*.toml`) alongside `requirements.txt`, creating dual dependency management.

**Controller-Runtime Dominance with Notable Exceptions**: 20 components use controller-runtime as their primary framework, making it the de facto standard. However, several components deviate: ModelMesh Serving uses etcd as a coordination layer alongside controller-runtime, creating a bidirectional data flow where the controller pushes to ModelMesh via gRPC and ModelMesh pushes back via etcd watches. Kueue's scheduler implements `manager.Runnable` as a continuous scheduling loop rather than a standard reconciler. The TrustyAI LMES controller manages pods directly rather than using Jobs, implementing its own lifecycle state machine. These deviations are architecturally motivated but increase the operational knowledge required to debug the platform.

**Multi-Tenancy Enforcement Gaps**: Multi-tenancy is enforced through different mechanisms at different layers. Kueue provides cluster-level quota fairness; NetworkPolicies provide pod-to-pod isolation (23 policies across the platform); RBAC provides API-level access control (674 roles). However, ModelMesh's etcd-based namespace isolation depends on correct root prefix configuration, and the Models-as-a-Service gateway relies on Authorino correctly evaluating OPA policies for group membership. The EvalHub controller in TrustyAI discovers tenant namespaces by watching labels, creating an implicit trust model where namespace labeling controls evaluation access.
