## Platform Overview

Red Hat OpenShift AI (RHOAI) is an enterprise AI/ML platform deployed on OpenShift Container Platform via Operator Lifecycle Manager (OLM). The platform is managed by a central operator (rhods-operator) that uses a hierarchical CRD model -- DSCInitialization provisions platform infrastructure (namespaces, networking, monitoring, auth, gateway) and DataScienceCluster manages the lifecycle of 16+ data science components. Components can be independently enabled, disabled, or set to unmanaged state, allowing flexible deployment topologies.

The platform employs a Gateway API-based ingress architecture with an Envoy data plane, kube-auth-proxy for OAuth2/OIDC authentication, and OpenShift Routes for legacy compatibility. Three authentication modes are supported: integrated OpenShift OAuth, external OIDC, and unmanaged (external management). Traffic flows through a central data-science-gateway (Gateway CR) that routes to component backends via HTTPRoutes, with kube-rbac-proxy sidecars enforcing Kubernetes RBAC at the pod boundary.

The security posture follows defense-in-depth: TLS encryption (via OpenShift service-ca and cert-manager), RBAC enforcement at every API endpoint, network policies restricting inter-component communication, and FIPS-compliant cryptographic libraries in critical components. The platform ships a complete monitoring stack including MonitoringStack, Prometheus, ThanosQuerier, Tempo, OpenTelemetry Collector, and Perses for observability. Disconnected (air-gapped) deployments are supported via RELATED_IMAGE environment variables and OLM relatedImages for image mirroring.

### Central Components

The following components have the most dependencies across the platform and serve as core infrastructure:

1. **rhods-operator**: Central platform operator managing 16+ component lifecycles, gateway infrastructure, monitoring, and RBAC. All components depend on it for deployment.
2. **kserve**: Core model serving platform providing InferenceService and LLMInferenceService CRDs. Used by all model serving runtimes (vLLM, TGIS, OVMS, Caikit, MLServer).
3. **kube-rbac-proxy**: Authentication/authorization sidecar used by nearly every component to enforce OpenShift RBAC at pod boundaries.
4. **kube-auth-proxy**: Platform-level OIDC/OAuth2 authentication proxy for the data-science-gateway.
5. **kueue**: Job scheduling system that manages workload admission for training jobs, Ray clusters, and batch workloads across the platform.
6. **odh-dashboard**: Central management UI that integrates with nearly every platform service via REST API proxying and Module Federation.
7. **data-science-pipelines-operator**: Manages pipeline infrastructure used by evaluation, fine-tuning, and AutoML workflows.

## Deployment Architecture

### Deployment Topology

The platform is deployed across multiple namespaces following a separation-of-concerns model. The rhods-operator runs in the `redhat-ods-operator` namespace and manages all component deployments in the `redhat-ods-applications` namespace. Monitoring infrastructure (Prometheus, Thanos, Tempo) is deployed in `redhat-ods-monitoring`. Gateway infrastructure (Envoy, kube-auth-proxy) runs in the ingress namespace (typically `istio-system` or alongside the service mesh control plane).

User workloads (notebooks, inference services, pipelines, training jobs, Ray clusters, feature stores) run in user-created namespaces. Each user namespace can contain DSPA (pipeline), InferenceService, Notebook, RayCluster, FeatureStore, and TrainJob resources. The platform operator watches all namespaces for relevant CRDs and creates supporting infrastructure (Routes, HTTPRoutes, RBAC, monitoring) as needed.

#### Manifest Consumption Pipeline

Component manifests flow from individual component repositories through a centralized `get_all_manifests.sh` script in the rhods-operator repository. This script clones specific branches/tags of each component repo and copies their kustomize manifests into the operator's `opt/manifests/` directory structure. During OLM bundle builds, these manifests are embedded into the operator container image. At runtime, the operator reads these bundled manifests, applies kustomize overlays (selecting ODH or RHOAI paths based on build tags), renders any templates with runtime parameters (namespace, image references, feature flags), and deploys them via server-side apply to the cluster.

### Disconnected Support

The platform supports air-gapped (disconnected) deployments through several mechanisms. All operator container images are referenced via `RELATED_IMAGE_*` environment variables in the ClusterServiceVersion (CSV), enabling OLM's `relatedImages` feature for automatic image mirroring via `oc-mirror`. Component images are pinned by SHA256 digest rather than mutable tags. The `oc-mirror` tool can be used to create a complete mirror of all platform images to an internal registry. The rhods-operator supports configuring a custom trusted CA bundle (via DSCInitialization) that is propagated to all component namespaces for internal registry access. Notebook images are delivered via ImageStreams that can be retargeted to internal registries.

## Platform Architectural Analysis

The RHOAI platform exhibits several cross-cutting architectural patterns and tensions that emerge from analyzing 68 components together.

**Unified Ingress Architecture**: The platform is converging on Gateway API as the standard ingress mechanism, with a central data-science-gateway (Envoy) routing traffic to all components via HTTPRoutes. However, many components still maintain OpenShift Route-based ingress paths for backward compatibility, creating dual-path ingress that must be maintained and tested. The trustyai-service-operator, data-science-pipelines-operator, model-registry-operator, and codeflare-operator still primarily use OpenShift Routes, while newer components (mlflow-operator, models-as-a-service, kserve LLMInferenceService) are Gateway API-native.

**Authentication Heterogeneity**: Despite the centralized kube-auth-proxy at the gateway level, authentication is implemented inconsistently across components. Some use kube-rbac-proxy sidecars (SubjectAccessReview), others use OAuth proxy, the MaaS component uses Kuadrant AuthPolicy with API keys, and Llama Stack uses application-level API keys. This creates a complex security surface where different endpoints require different authentication flows, making unified access management challenging.

**Operator Proliferation**: With 16+ operators, the platform has significant operational complexity. The rhods-operator mitigates this by managing operator lifecycle, but each operator brings its own CRD schema, reconciliation loop, webhook server, leader election, and RBAC requirements. The emerging Kubeflow Trainer (trainer) coexists with the legacy Training Operator (training-operator), representing an in-progress migration that increases the component count.

**LLM-Specific Innovation Layer**: The llm-d stack (inference-scheduler, kv-cache, routing-sidecar) and supporting components (gateway-api-inference-extension, workload-variant-autoscaler, batch-gateway, vllm-orchestrator-gateway) represent a rapidly evolving LLM inference optimization layer. These components use advanced patterns (Envoy ext-proc, KV-cache-aware routing, disaggregated prefill/decode) that are architecturally distinct from the traditional ML model serving path (KServe/ModelMesh). The coexistence of these two serving architectures (KServe InferenceService for traditional ML, LLMInferenceService for LLMs) creates deployment complexity but allows independent evolution.

**Module Federation in Dashboard**: The odh-dashboard's adoption of Webpack Module Federation with independently deployable Go BFF sidecars is architecturally ambitious -- enabling 7+ teams to independently develop and deploy UI modules. However, this creates a large pod footprint (1 main container + kube-rbac-proxy + up to 7 sidecar containers) and tight coupling at the pod scheduling boundary.

**Storage and State Management**: Components use diverse persistence backends: PostgreSQL (model-registry, mlflow, data-science-pipelines), MariaDB/MySQL (data-science-pipelines, trustyai), etcd (modelmesh), S3-compatible storage (pipelines artifacts, MLflow artifacts, batch-gateway), SQLite (llama-stack), and PVC (workbenches, local model cache). There is no platform-wide storage abstraction, requiring per-component database and storage provisioning.

**Guardrails Architecture Fragmentation**: Content safety is served by multiple overlapping components: fms-guardrails-orchestrator (Rust, REST orchestration), NeMo-Guardrails (Python, programmable rails), guardrails-detectors (Python microservices), guardrails-regex-detector (Rust), and vllm-orchestrator-gateway (Rust, OpenAI gateway with detector routing). The trustyai-service-operator manages both GuardrailsOrchestrator and NemoGuardrails CRDs, but the runtime architectures are fundamentally different, suggesting ongoing consolidation.

**Build and Distribution Pipeline**: The get_all_manifests.sh pattern for collecting component manifests into the operator is pragmatic but creates tight coupling between component repository structure and the operator's build process. Any change to a component's kustomize layout requires corresponding updates to the manifest collection script. The RELATED_IMAGE pattern for disconnected support works well but requires careful coordination across 90+ container images during releases.
