## Purpose

**Short**: Builds and publishes container images for interactive data science workbench environments (Jupyter, Code Server, RStudio) and pipeline runtime images used in Red Hat OpenShift AI.

**Detailed**: The notebooks-downstream repository is the RHOAI downstream fork of the upstream opendatahub-io/notebooks repository. It produces a comprehensive set of container images that serve as the interactive development environments for data scientists on the OpenShift AI platform. These images are launched by the ODH Notebook Controller as Kubernetes pods when users create workbenches through the RHOAI Dashboard.

The repository builds images across three IDE families — JupyterLab, VS Code (code-server), and RStudio Server — each with multiple Python versions (3.11 and 3.12) and hardware acceleration variants (CPU-only, NVIDIA CUDA 12.6.3 for GPU compute, and AMD ROCm 6.2.4 for AMD GPU compute). Additionally, it produces lightweight runtime images used by the Elyra pipeline engine to execute individual pipeline nodes outside of the interactive workbench context.

All images share a common UBI9-based foundation layer that includes the OpenShift CLI (`oc`), `micropipenv` for reproducible Python dependency installation from `Pipfile.lock`, and common OS packages. The images are published to `quay.io/modh/` registries and deployed to clusters via OpenShift ImageStream resources managed through kustomize manifests, which the platform operator (`rhods-operator`) consumes.

## Architectural Analysis

The notebooks-downstream repository follows a layered container image architecture where all images share a common UBI9 Python base stage. This base stage installs the `oc` CLI, `micropipenv`, and essential OS packages, ensuring consistency across all workbench types. The Jupyter image family uses a multi-stage build chain: `base → jupyter-minimal → jupyter-datascience → {pytorch, tensorflow, trustyai}`, where each stage adds incremental capabilities. This pattern minimizes image size through layer sharing but creates tight coupling — a base layer change forces rebuilds of all downstream images.

For GPU-accelerated variants, a separate `cuda-base` intermediate stage is inserted between `base` and the IDE-specific stages. This stage installs the complete CUDA 12.6.3 toolkit (runtime, development, and cuDNN libraries) from NVIDIA's repository. The AMD ROCm variants follow a similar pattern with a `rocm-base` stage installing ROCm 6.2.4. Multi-architecture support (amd64/arm64) is handled at the Dockerfile level through `TARGETARCH`-conditional build stages, though CUDA/ROCm images are amd64-only due to hardware constraints.

The code-server and RStudio images use a distinct architecture from Jupyter notebooks: both run NGINX as a reverse proxy in front of the actual IDE process, with supervisord managing the process tree. This NGINX layer serves two purposes — it provides health check endpoints (`/api/status`) compatible with Kubernetes probes, and it maintains activity logs used by the notebook culler to detect idle workbenches. The Jupyter images don't need this pattern because JupyterLab natively supports Kubernetes health checks and activity tracking via its REST API.

The kustomize manifest structure is designed for versioned image lifecycle management. Each ImageStream maintains up to 6 version tags (N through N-5), where N is the current release (2025.1) and older versions are progressively marked with `opendatahub.io/image-tag-outdated: "true"`. Image references are not hardcoded in the ImageStream YAML — they use placeholder values that are substituted at deploy time via kustomize `replacements` sourced from `params.env` ConfigMaps. This allows the platform operator to pin exact image digests (SHA256) for reproducibility while the ImageStream YAML remains version-controlled. The `commit.env` files track which Git commit produced each image, exposed as the `opendatahub.io/notebook-build-commit` annotation for traceability.

The CI/CD pipeline is transitioning to Konflux (Tekton-based). The `.tekton/` directory contains per-component `PipelineRun` definitions that reference a shared `multiarch-pull-request-pipeline.yaml`. Each PipelineRun specifies the Dockerfile path and build context for a single image variant. This is a recent addition (RHOAIENG-30439) and currently covers runtime images and some workbench images, with more being added incrementally. The RStudio images uniquely retain an OpenShift `BuildConfig` strategy as well, referencing a RHEL subscription secret for builds that require Red Hat package access.
