## Purpose

**Short**: Builds and publishes container images for notebook workbenches (Jupyter, Code Server, RStudio) and pipeline runtime environments used in Red Hat OpenShift AI.

**Detailed**: This repository is the downstream (RHOAI) fork of the OpenDataHub notebooks project. It produces approximately 35 container images spanning four workbench types (Jupyter notebooks, VS Code via code-server, RStudio Server, and pipeline runtimes) across three hardware acceleration targets (CPU, NVIDIA CUDA, AMD ROCm) and two Python versions (3.11, 3.12). The images are built via Dockerfiles using a layered multi-stage build pattern where a common UBI9/C9S base image is extended with progressively specialized software stacks (minimal → data science → ML framework).

The repository does not deploy an operator or controller. Instead, it produces OpenShift ImageStream manifests (via kustomize) that the platform operator (`rhods-operator` / `opendatahub-operator`) consumes to present available workbench images to users through the RHOAI dashboard. When a user creates a workbench, the odh-notebook-controller instantiates a StatefulSet using one of these images. The images are built by Konflux CI via Tekton PipelineRun definitions in the `.tekton/` directory, which trigger on pull requests affecting specific Dockerfile paths.

## Architectural Analysis

The notebooks-downstream repository follows a layered container image architecture that mirrors the software stack specialization pattern. At the base is a common UBI9 Python image from Red Hat's container catalog. On top of this, a `base` stage installs system utilities (mesa-libGL, skopeo, oc CLI) and micropipenv. The `jupyter-minimal` stage adds JupyterLab, PDF export dependencies (TexLive, Pandoc), and JupyterLab configuration. The `jupyter-datascience` stage further adds database clients (MongoDB CLI, MSSQL tools, PostgreSQL), additional Python data science packages, and Elyra pipeline integration. Finally, framework-specific stages (`jupyter-pytorch`, `jupyter-tensorflow`, `jupyter-trustyai`) layer on their respective ML libraries. This layering means each image inherits all capabilities of its parent, and a Pipfile.lock at each layer pins the specific Python packages for that specialization.

The hardware acceleration architecture uses a branching strategy rather than a linear one. For CUDA images, a separate `cuda-base` multi-stage build configures NVIDIA repositories, installs CUDA runtime, development, and cuDNN libraries, and sets required environment variables (NVIDIA_VISIBLE_DEVICES, NVIDIA_DRIVER_CAPABILITIES, XLA_FLAGS). The CUDA base itself is multi-architecture, with conditional stages for amd64 (includes nvprof) and arm64/sbsa. For ROCm images, a `rocm-base` stage configures AMD repositories and installs the ROCm SDK. Both GPU base stages are then combined with the Jupyter/runtime specialization stages, creating a matrix of (workbench-type × acceleration × python-version) images.

The Code Server and RStudio workbenches follow a different deployment pattern from Jupyter. Both use NGINX 1.24 as a reverse proxy (port 8585) with supervisor managing the dual-process lifecycle (IDE server + NGINX). This architecture provides uniform health check endpoints and consistent port mapping regardless of the backend IDE. RStudio images additionally install R 4.4.3 from EPEL/CRB repositories, with the RHEL9 variant requiring subscription-manager integration for build-time access to Red Hat repos.

The build system uses a `sandbox.py` script that analyzes Dockerfile `COPY` directives via a Go-based `buildinputs` tool to determine the minimal set of files needed for each image build. This creates a temporary build context containing only the necessary files, preventing unnecessary cache invalidation and reducing build context size. The Konflux CI integration is defined through Tekton PipelineRun resources in `.tekton/`, with each image variant having its own PipelineRun that triggers on changes to relevant paths (e.g., changes to `jupyter/minimal/ubi9-python-3.11/**` trigger the minimal image build). The pipeline uses PipelinesAsCode CEL expressions for fine-grained trigger control.

The manifest system uses kustomize with a sophisticated replacement strategy. The `manifests/base/kustomization.yaml` defines 86+ replacement rules that map ConfigMap entries (image digests from `params.env` and commit hashes from `commit.env`) to specific ImageStream tag references. This allows updating all image references by simply editing the `.env` files, which is done automatically by the `update-commit-latest-env.py` script when new images are built. The overlay at `manifests/overlays/additional/` extends the base with Python 3.12 variants without duplicating the base configuration.
