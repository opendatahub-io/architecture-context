## Purpose

**Short**: Builds and publishes container images for Jupyter notebooks, code-server, RStudio workbenches, and Elyra pipeline runtime environments used in Red Hat OpenShift AI.

**Detailed**: The notebooks-downstream repository is the downstream (RHOAI) fork of the opendatahub-io/notebooks project. It produces approximately 32 container images spanning three categories: **workbench images** (interactive development environments including JupyterLab, VS Code via code-server, and RStudio), **GPU-accelerated workbench variants** (CUDA for NVIDIA GPUs, ROCm for AMD GPUs), and **pipeline runtime images** (headless Python environments for Elyra pipeline execution). Each image is built as a layered Dockerfile, starting from a UBI9 or CentOS Stream 9 base, adding Python tooling (micropipenv, JupyterLab, data science libraries), and optionally adding GPU compute stacks (CUDA 12.6.3, ROCm 6.2.4). The repository does not contain application source code — it composes existing open-source tools into pre-configured, OpenShift-optimized container images with proper permissions (UID 1001, GID 0), security tools (skopeo, oc client), and PDF export capabilities (TeX Live, pandoc).

The images are deployed to OpenShift clusters via OpenShift ImageStream resources managed through kustomize manifests in the `manifests/` directory. The platform operator (rhods-operator) consumes these manifests to register workbench images in the RHOAI dashboard, where users can select them when creating notebook workbenches. Image references are pinned by SHA256 digest in `params.env` and `params-latest.env` files, with kustomize replacements injecting them into ImageStream tags. The repository supports a rolling version window (N through N-5) of image tags per workbench type, allowing users to use older image versions while gradually migrating to newer ones.

## Architectural Analysis

The notebooks-downstream repository follows a **layered Dockerfile composition** pattern where images build upon each other in a well-defined hierarchy. The base layer (`base` stage) is shared across all images within a Python version and provides UBI9 Python, micropipenv, skopeo, and the OpenShift CLI. Accelerator layers (CUDA, ROCm) add GPU runtime libraries on top of this base. Application layers (minimal, data science, PyTorch, TensorFlow, TrustyAI) add progressively more Python packages. This design maximizes Docker layer sharing — a PyTorch CUDA image shares all layers with the CUDA base and data science layers, reducing registry storage and pull times.

The **multi-architecture support** for CUDA images is noteworthy. The Dockerfiles use Docker BuildKit's `TARGETARCH` build argument with conditional `FROM` stages (`cuda-base-amd64` and `cuda-base-arm64`) to produce images for both x86_64 and ARM64 architectures. ROCm images are x86_64 only due to AMD's platform constraints.

The **process management architecture** differs significantly between image types. Jupyter notebooks use a simple shell entrypoint (`start-notebook.sh`) that launches JupyterLab directly. Code-server and RStudio images use a more complex supervisord-managed setup with nginx as a reverse proxy. Nginx provides health check endpoints at well-known paths (e.g., `/api/kernels`) that the Kubernetes liveness/readiness probes can query, since code-server and RStudio don't natively expose Jupyter-compatible API endpoints. The supervisord configuration also manages fcgiwrap for CGI script execution.

The **version window management** (N through N-5) in the kustomize manifests is a notable pattern. Each workbench type maintains up to 6 tagged versions in its ImageStream, with the latest (`N`) marked as recommended and older versions progressively marked as outdated. This allows users running long-lived notebook sessions to continue using their current image version while new users get the latest. Image references are SHA256-pinned in `params.env` (historical versions) and `params-latest.env` (current build), with automated CI updating these files as new images are built.

A **security concern** is that the notebook images intentionally disable JupyterLab authentication (`--ServerApp.token='' --ServerApp.password=''`). This is by design — authentication is expected to be enforced by the sidecar proxy (kube-rbac-proxy in RHOAI 3.x) injected by the odh-notebook-controller. However, if these images were deployed without the controller (e.g., directly via the per-image kustomize manifests in `jupyter/*/kustomize/base/`), the notebooks would be accessible without authentication. The per-image kustomize manifests do not include a sidecar container.
