## Purpose

**Short**: Builds and publishes container images for interactive data science workbenches (Jupyter, RStudio, code-server) and Elyra pipeline runtime environments used in Red Hat OpenShift AI.

**Detailed**: The notebooks-downstream repository is the downstream (Red Hat) fork of the OpenDataHub notebooks project. It produces ~35 container images that serve as workbench environments for data scientists on the RHOAI platform. These images are launched by the ODH Notebook Controller as individual user workspaces (StatefulSets) and are presented to users in the RHOAI Dashboard as selectable workbench types.

The repository is organized into four major image families: **Jupyter notebooks** (minimal, datascience, PyTorch, TensorFlow, TrustYAI, ROCm variants), **code-server** (VS Code in browser), **RStudio Server** (R language IDE), and **pipeline runtimes** (headless Python environments for Elyra/Data Science Pipelines). Each family supports multiple Python versions (3.11, 3.12) and GPU accelerator backends (NVIDIA CUDA 12.6.3, AMD ROCm 6.2.4). Images are built on Red Hat UBI 9 (or CentOS Stream 9 for RStudio community variants) and include the OpenShift `oc` CLI and `skopeo` for container operations.

The deployment model uses OpenShift ImageStreams with kustomize-managed manifests. The platform operator (`rhods-operator`) applies these manifests to make images available in the dashboard. Each ImageStream maintains up to 6 tagged versions (N through N-5) for rolling upgrades with backward compatibility. Konflux (via Tekton PipelineRuns) builds multi-architecture images on pull requests.

## Architectural Analysis

The notebooks-downstream repository follows a **layered image composition** pattern where more specialized images build on top of less specialized ones within multi-stage Dockerfiles. The minimal Jupyter image is the foundation: UBI 9 Python base → system packages (mesa-libGL, skopeo) → micropipenv → OpenShift oc CLI → JupyterLab from Pipfile.lock. The datascience image extends minimal with database tools (MongoDB CLI built from Go, MSSQL tools, PostgreSQL client), Elyra notebook pipeline support, and a richer set of Python libraries. The framework images (PyTorch, TensorFlow) extend datascience with GPU-specific layers. TrustYAI extends datascience with Java 17 for the TrustYAI explainability library. This layering is logical but not implemented as Docker multi-stage composition between image types — each Dockerfile repeats the full build chain. The GPU layers (CUDA, ROCm) introduce significant complexity: CUDA images use a 4-stage build with architecture-conditional stages (`cuda-base-amd64` vs `cuda-base-arm64`) to handle the different NVIDIA package repositories for x86_64 and aarch64.

The **image versioning strategy** is notable: each ImageStream maintains 6 tagged versions (2025.1, 2024.2, 2024.1, 2023.2, 2023.1, 1.2) with the latest marked `recommended` and older ones marked `outdated`. Image references are stored as SHA256 digests in `params.env` / `params-latest.env` files and injected into ImageStream manifests via kustomize replacements. This creates an N/N-1/N-2 upgrade pattern where the platform always has the current and two previous versions available, enabling controlled rollouts. The parameterization uses a naming convention (`odh-workbench-{type}-{feature}-{accel}-py{ver}-{os}-n` through `n-5`) that maps directly to the kustomize replacement targets.

The **build infrastructure** spans two systems: GitHub Actions (referenced in README for Pipfile.lock renewal) and Konflux (Tekton pipelines in `.tekton/`). Konflux is the production build system — each image variant has a dedicated PipelineRun YAML that triggers on pull requests when relevant source files change (using CEL expressions like `"jupyter/minimal/ubi9-python-3.11/**".pathChanged()`). The common pipeline (`multiarch-pull-request-pipeline.yaml`) orchestrates: git clone → buildah build → SBOM generation → image index creation. A Python script (`scripts/generate_pull_request_pipelineruns.py`) auto-generates per-component PipelineRun definitions to avoid manual maintenance of 35+ pipeline files.

The **security model** deliberately delegates authentication to the platform. All workbench images disable built-in authentication tokens (`--ServerApp.token=''` for JupyterLab) because the ODH Notebook Controller injects authentication sidecars (kube-rbac-proxy in RHOAI 3.x, oauth-proxy in RHOAI 2.x) that handle authentication before traffic reaches the workbench container. This is a clean separation of concerns but means these images must NEVER be deployed without the Notebook Controller's sidecar injection — direct exposure would be unauthenticated. The images also run as non-root user (UID 1001) with extensive `chmod`/`chgrp` operations to support OpenShift's random UID security model where containers run as arbitrary UIDs within GID 0.

A potential concern is the **non-hermetic OS package layer**: while Python dependencies are fully locked via Pipfile.lock files, system packages installed via `dnf`/`yum` in Dockerfiles pull whatever version is currently in the UBI/CentOS/RHEL repositories. This means rebuilding the same Dockerfile at different times may produce different OS package versions. The CUDA and ROCm repositories compound this — they're added via `.repo` files pointing to external NVIDIA/AMD mirrors with no version pinning beyond the major CUDA/ROCm version (e.g., `cuda-12-6` or `rocm-6.2.4`). Downstream release branches likely address this with `rpms.lock.yaml` files, but the main branch analyzed here does not have them.
