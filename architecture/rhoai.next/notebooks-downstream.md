## Purpose

**Short**: Builds and maintains container images for Jupyter, RStudio, and Code Server workbenches plus Elyra runtime images used in Red Hat OpenShift AI (RHOAI).

**Detailed**: The notebooks-downstream repository is the downstream (RHOAI-specific) fork of opendatahub-io/notebooks. It produces 32+ container images that serve as interactive data science environments (workbenches) and pipeline execution environments (runtimes) for the RHOAI platform. The repository does not contain operators or services — it is a pure container image build factory.

Workbench images include JupyterLab, RStudio Server, and VS Code (code-server) environments pre-loaded with data science libraries. Each workbench type ships in multiple variants covering different Python versions (3.11, 3.12), accelerator stacks (CPU, NVIDIA CUDA 12.6, AMD ROCm 6.2), and library sets (minimal, data science, PyTorch, TensorFlow, TrustyAI). Runtime images are lightweight containers designed for Elyra pipeline execution with minimal dependencies.

The repository integrates with the RHOAI platform through OpenShift ImageStream manifests (consumed by the platform operator) and Tekton/Konflux CI pipelines for multi-architecture builds. Images are published to `quay.io/opendatahub/workbench-images` (upstream) and `quay.io/modh/` registries (downstream). The notebook controller (`odh-notebook-controller`) is responsible for deploying these images as StatefulSets when users create workbench instances through the RHOAI dashboard.

#### Application-Level Crypto

| Aspect | Value | Source |
|--------|-------|--------|
| **TLS configuration** | No explicit TLS configuration — workbench servers listen on plaintext HTTP; TLS is terminated by platform ingress | StatefulSet manifests (port 8888 HTTP) |
| **Crypto libraries** | Python `cryptography` library inherited from dependencies; JupyterLab uses stdlib `ssl` | Pipfile.lock transitive dependencies |
| **Certificate handling** | System trust store (UBI9 ca-certificates) | Base image default |
| **Non-FIPS crypto risks** | Minimal images (cpu/cuda/rocm variants) do NOT compile mongocli with strictfipsruntime — they have no Go binaries. Python crypto relies on UBI9 OpenSSL which is FIPS-capable when the host FIPS mode is enabled. No explicit FIPS opt-in for Python processes. | `jupyter/minimal/ubi9-python-3.11/Dockerfile.cpu` (no FIPS flags) |

_FIPS compliance for these images depends primarily on the host OpenShift cluster having FIPS mode enabled, which forces UBI9's OpenSSL into FIPS mode. The mongocli binary in data science images is built with `strictfipsruntime` to ensure its Go crypto uses FIPS-approved algorithms. Minimal images contain no Go binaries and rely entirely on Python → OpenSSL → host FIPS mode._

## Architectural Analysis

The notebooks-downstream repository follows a layered image build architecture where each workbench type builds incrementally on top of the previous layer. The base layer is always a Red Hat UBI9 Python image (`registry.access.redhat.com/ubi9/python-3xx`), upon which the minimal Jupyter layer adds JupyterLab and basic tooling. The data science layer extends minimal with comprehensive ML libraries, database connectors, and a FIPS-compliant MongoDB CLI. Framework layers (PyTorch, TensorFlow, TrustyAI) extend data science further with specialized libraries. GPU layers (CUDA, ROCm) add accelerator stacks at the appropriate point in the chain. This layering is implemented within single multi-stage Dockerfiles rather than through separate base images, meaning each downstream image re-runs all upstream build steps.

The Makefile-based build system uses a novel sandboxing approach (`scripts/sandbox.py` + `scripts/buildinputs/buildinputs.go`): a Go tool performs static analysis of Dockerfiles to determine which files each image actually needs (COPY sources and build context), then a Python sandbox script creates a minimal temporary directory containing only those files before invoking the container engine. This ensures builds are reproducible and prevents accidental inclusion of unrelated files in the build context. The buildinputs Go tool performs Dockerfile parsing at compile time, not runtime, which is unusual for this type of build optimization.

A significant architectural concern is the absence of Konflux-specific Dockerfiles. While the `.tekton/` directory contains 36 Tekton PipelineRun definitions pointing at the standard Dockerfiles, there are no `Dockerfile.konflux*` variants. This means the same Dockerfiles are used for both local/CI builds and Konflux production builds. The Tekton pipelines are configured for multi-architecture builds (x86_64, arm64, s390x for some images), but the Dockerfiles themselves handle multi-arch only for CUDA (which has separate base image references for amd64 and arm64 via build stages). The pipeline configuration uses `hermetic: false` by default and has no `prefetch-input` set, indicating that Konflux builds are not currently hermetic — they pull dependencies from the internet at build time.

The image versioning system is sophisticated: each ImageStream tracks 6 versions (N through N-5) with SHA256-pinned image references stored in `params.env`/`params-latest.env` files, plus corresponding git commit hashes in `commit.env`/`commit-latest.env`. The kustomization uses ConfigMap generators and replacements to inject these values into ImageStream tags. This allows the platform operator to deploy multiple image versions simultaneously, enabling users to choose their preferred version while older versions are gradually marked as outdated via `opendatahub.io/image-tag-outdated: "true"` annotations. The dashboard reads these annotations to inform users when they should upgrade.

The separation between workbench images and runtime images is architecturally clean: workbenches are interactive environments (JupyterLab, RStudio, code-server) with rich UI dependencies, while runtimes are minimal execution environments for Elyra pipeline nodes. Runtimes expose port 8080 and are designed to install additional packages at runtime via `requirements.txt`, while workbenches expose their native ports (8888, 8585, 8787) and come pre-loaded with all dependencies. Both types are deployed as pods, but workbenches run as StatefulSets (persistent user sessions) while runtimes run as ephemeral pipeline step pods.
