## Purpose

**Short**: The unified web UI and API gateway for Red Hat OpenShift AI, providing a React-based dashboard with modular federated plugin architecture and Go backend-for-frontend (BFF) services.

**Detailed**: odh-dashboard is the primary user-facing component of both Open Data Hub (ODH) and Red Hat OpenShift AI (RHOAI). It delivers a single-page React application built on PatternFly 6, a Node.js (Fastify) backend that proxies Kubernetes API requests and serves the frontend, and a growing set of Go BFF sidecar modules that provide domain-specific API layers for features like generative AI, model-as-a-service, model registry, MLflow experiment tracking, evaluation hub, AutoML, AutoRAG, and agent operations.

The repository is structured as an npm workspace monorepo with a modular architecture pattern: a host application (`frontend/` + `backend/`) renders the shell UI (navigation, header, routing), while feature-specific packages (`packages/*/`) are built as Webpack Module Federation remotes. Each package contains a React frontend and a Go BFF server. In production, these BFF modules run as sidecar containers within the same pod as the host dashboard, sharing a single Kubernetes Service on different ports (8043-8843). A dedicated Go operator (`dashboard-operator/`) manages the lifecycle of the dashboard deployment, including module enablement, kustomize rendering, and platform-specific overlays for ODH vs RHOAI.

The security model uses kube-rbac-proxy as a TLS-terminating authentication sidecar (port 8443), forwarding authenticated requests to the Node.js backend (port 8080). BFF sidecars receive user tokens via the `x-forwarded-access-token` header and use Kubernetes impersonation to call upstream APIs on behalf of the authenticated user.

## Architectural Analysis

The odh-dashboard represents one of the more architecturally sophisticated components in the RHOAI platform, implementing a full micro-frontend architecture using Webpack Module Federation. The host application (`frontend/` + `backend/`) provides the shell (navigation, routing, header) while feature-specific packages are loaded as federated remotes at runtime. This design enables independent deployment and development of feature areas like generative AI, model-as-a-service, and model registry without requiring a full dashboard rebuild.

The multi-sidecar deployment pattern is particularly noteworthy: up to 10 containers (1 Node.js host + 1 kube-rbac-proxy + 8 Go BFF sidecars) run in a single pod, sharing a Kubernetes Service with 9 exposed ports (8443 primary + 8043-8843 for modules). Each BFF sidecar serves both its React frontend bundle (as static files) and a domain-specific REST API. The Node.js backend acts as a routing hub, proxying `/_mf/{name}/*` requests to the appropriate sidecar port. This architecture trades pod density for deployment simplicity — all components scale together and share network namespace.

The authentication chain is well-layered: kube-rbac-proxy provides the first gate (verifying the user can list OpenShift projects), then forwards authenticated requests with `X-Auth-Request-User` and `X-Auth-Request-Groups` headers. The Node.js backend extracts the user's bearer token from `x-forwarded-access-token` and either uses it directly for K8s API calls (via impersonation) or forwards it to BFF sidecars. The BFF sidecars then use the same token to call upstream services on behalf of the user. The `RequireAccessToService` middleware in Go BFFs validates the token against the K8s API before processing requests.

The dashboard-operator introduces a controller-based lifecycle management layer using controller-runtime. It reconciles a singleton `Dashboard` CR, rendering platform-specific kustomize overlays (ODH vs RHOAI) and managing module enablement based on component dependencies. The module registry supports 10 modules with dependency resolution — for example, `mlflowEmbedded` depends on `mlflow`, and `modelRegistry` requires the `modelregistry` component to be enabled in the DataScienceCluster. The operator also handles cross-namespace cleanup (e.g., Perses resources in the monitoring namespace) and extracts the dashboard URL from admitted OpenShift Routes.

A notable security consideration is the ServiceAccount token isolation in the modular architecture deployment: `automountServiceAccountToken: false` is set on the pod, with manual projected volumes providing separate tokens — `dashboard-sa-token` for the core containers and `modules-sa-token` (from a separate Secret) for the BFF sidecars. This limits the blast radius if a BFF sidecar is compromised. The SSRF protection in the core BFF's K8s proxy (`internal/proxy/k8s_proxy.go`) restricts DNS resolution targets to the K8s API host, preventing proxy abuse.

The inter-BFF communication pattern (gen-ai BFF calling MaaS BFF for token management) demonstrates early signs of service-to-service dependencies within the sidecar constellation. Currently limited to token management, this pattern could expand as feature areas become more intertwined. The BFF client factory (`bffclient/`) provides a reusable framework for these inter-sidecar calls with configurable auth methods (`internal` vs `user_token`).
