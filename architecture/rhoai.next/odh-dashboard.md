## Purpose

**Short**: Web-based management console for Red Hat OpenShift AI (RHOAI) and Open Data Hub (ODH) that provides a unified UI for managing AI/ML workloads, model serving, pipelines, notebooks, and platform configuration.

**Detailed**: The ODH Dashboard is a monorepo producing 9 container images via a micro-frontend architecture. The host application is a React 18 frontend served by a Node.js (Fastify) backend that acts as an API gateway to Kubernetes and downstream ML services. Seven federated plugin modules (gen-ai, maas, model-registry, automl, autorag, eval-hub, mlflow) are deployed as Go-based BFF sidecar containers within the same pod, each providing domain-specific UI and API proxy capabilities.

The dashboard mediates all user interactions with the RHOAI/ODH platform: creating and managing data science projects, launching Jupyter notebooks, deploying models via KServe, running ML pipelines via Data Science Pipelines, managing model registries, interacting with GenAI services (LlamaStack, MCP servers, guardrails), and administering platform-wide settings like hardware profiles, connection types, and RBAC policies. Authentication flows through a kube-rbac-proxy sidecar that injects user identity headers, and the backend enforces namespace-scoped authorization for multi-tenant isolation.

The modular architecture uses Webpack Module Federation to dynamically load plugin frontends at runtime, while the Go BFF containers proxy API calls to their respective upstream services (LlamaStack, MaaS controller, MLflow tracking server, EvalHub, Pipeline Server, Model Registry API, Model Catalog API). Inter-BFF communication (e.g., Gen-AI BFF calling MaaS BFF for token delegation) uses in-cluster service discovery.

| autorag BFF | Go HTTP Server (BFF) | Orchestrates AutoRAG pipeline runs with LlamaStack integration for RAG optimization |
| eval-hub BFF | Go HTTP Server (BFF) | Proxies EvalHub evaluation framework for model evaluation jobs, collections, and providers |
| mlflow BFF | Go HTTP Server (BFF) | Proxies MLflow tracking server for experiment management and tracking |
| plugin-core | TypeScript Package | Shared extension point definitions and plugin store for module federation plugin system |

## Architectural Analysis

The ODH Dashboard represents a sophisticated micro-frontend architecture that has evolved from a monolithic dashboard into a federated plugin system. The most notable architectural decision is the single-pod, multi-container deployment model: the main dashboard container (Node.js) runs alongside a kube-rbac-proxy sidecar and up to 7 Go-based BFF sidecar containers, all sharing the same pod network and TLS certificates. This co-location eliminates network hops for module federation asset loading and API proxying, while the BFF pattern provides strong domain isolation between feature areas.

The module federation topology is hub-and-spoke: the host application loads `remoteEntry.js` from each BFF container's static assets directory, then dynamically resolves `./extensions` and `./extension-points` exports. Each plugin registers its routes, navigation items, and feature areas through a typed extension point system (plugin-core package). The PluginStore filters extensions based on feature flags derived from the DashboardConfig CR and user admin status, enabling fine-grained feature gating without code changes. The `@odh-dashboard/plugin-core` and `@odh-dashboard/internal` packages are shared as singletons across module boundaries to ensure type compatibility and avoid duplicate React instances.

The authentication architecture is layered: kube-rbac-proxy handles the initial authentication (extracting user identity from Gateway/OAuth headers), the Node.js backend enforces namespace-scoped authorization via `secureRoute`/`secureAdminRoute` wrappers, and the Go BFFs perform Kubernetes SubjectAccessReview checks for service-level authorization. Service account token isolation is particularly well-designed -- `automountServiceAccountToken: false` prevents automatic token mounting, and two separate projected volumes (`dashboard-sa-token` and `modules-sa-token`) provide the main container and BFF containers with different permission sets. This limits the blast radius of a container compromise.

The inter-BFF communication pattern (e.g., Gen-AI BFF calling MaaS BFF for token delegation) uses in-cluster service discovery with configurable auth methods. This allows the Gen-AI playground to mint MaaS API keys on behalf of users without exposing MaaS credentials to the frontend. The token caching strategy (in-memory with TTL from token expiry) balances performance with security, though the per-instance cache means tokens cannot be shared across pod replicas.

A notable risk area is the `INSECURE_SKIP_VERIFY` flag available in all Go BFFs and the `rejectUnauthorized=false` usage in the feature store integration. While both are intended for development only, their presence in production code paths requires careful operational controls. The esbuild binary removal in the Dockerfile (lines 41-45) is a FIPS compliance measure worth noting -- transitive build dependencies can introduce non-compliant binaries that must be explicitly cleaned up.
