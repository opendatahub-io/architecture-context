## Purpose

**Short**: The ODH Dashboard is the web-based management console for Red Hat OpenShift AI (RHOAI) and Open Data Hub (ODH), providing a unified interface for managing data science workloads, model serving, pipelines, model registries, and platform administration.

**Detailed**: The dashboard is built as a monorepo using npm workspaces and Turborepo, producing 9 deployable container images via Konflux. The primary image contains a React/TypeScript frontend (host application) served by a Node.js/Fastify backend, while 7 additional images are federated plugin modules, each containing a React frontend and a Go BFF service. A ninth image (sealights) is a build variant of the primary for test instrumentation.

The architecture follows a module federation pattern where the host application loads plugin frontends at runtime via Webpack Module Federation. Each plugin's Go BFF runs as a sidecar container in the same pod, providing domain-specific API endpoints that proxy to upstream platform services (Kubernetes API, Pipeline Server, Model Registry, LlamaStack, MaaS, MLflow, EvalHub, etc.). The Fastify backend handles authentication passthrough from kube-rbac-proxy, Kubernetes API proxying, WebSocket watch streams, and module federation proxy routing to the BFF sidecars.

All traffic enters through kube-rbac-proxy (port 8443/HTTPS) which authenticates users via OpenShift RBAC before forwarding to the Fastify backend (port 8080/HTTP). The backend then routes module-federation API requests to the appropriate BFF sidecar over HTTPS using the forwarded user token. The deployment is fronted by a Gateway API HTTPRoute for external access.

## Architectural Analysis

The ODH Dashboard represents a sophisticated micro-frontend architecture built on Webpack Module Federation. The host application (React + Fastify) acts as both a user-facing shell and an API gateway, while domain-specific functionality is delivered through federated plugin modules. Each plugin is a self-contained unit pairing a React frontend with a Go BFF service, deployed as sidecar containers in the same Kubernetes pod. This sidecar pattern is notable: rather than separate Deployments per module, all BFFs share the pod's network namespace and TLS certificates, communicating with the host's Fastify backend over localhost HTTPS. The deployment manifest uses kustomize JSON patches to inject sidecar containers and Service ports for each module.

The authentication architecture follows a layered trust model. External traffic enters through Gateway API (HTTPRoute), hits kube-rbac-proxy which performs a SubjectAccessReview (checking if the user can list OpenShift projects), then forwards the authenticated request to Fastify with identity headers (x-forwarded-access-token, x-auth-request-user, x-auth-request-groups). The Fastify backend either handles the request directly (for K8s API passthrough, admin routes, service proxying) or routes module federation requests (/_mf/:name/* and proxy paths like /model-registry/api/*) to the appropriate BFF sidecar over HTTPS, forwarding the user token. Each BFF then performs its own authorization checks via SubjectAccessReview before proxying to upstream services. This creates three authorization checkpoints: Gateway → kube-rbac-proxy → BFF middleware.

The inter-BFF communication pattern (seen in gen-ai BFF calling maas BFF for token delegation) is architecturally significant. Rather than each BFF independently calling upstream services, they can delegate to sibling BFFs via Kubernetes service DNS (odh-dashboard.{namespace}.svc.cluster.local:{port}). This creates a service mesh within the pod, with the federation-config ConfigMap serving as the service registry. The network policy explicitly allows all in-cluster egress to support this pattern, while restricting ingress to only the configured ports from the OpenShift ingress namespace and same-namespace pods.

The codebase demonstrates a clear progression from a monolithic dashboard toward a plugin-based architecture. The host application still contains substantial domain logic (pipelines, notebooks, model serving) in its frontend and backend, while newer features (AutoML, AutoRAG, Gen AI, MaaS, MLflow, Eval Hub) are built as federated modules. The plugin-core package defines the extension point contract (route extensions, navigation items, tab pages, status providers, task groups), and each module implements these extension points. The @odh-dashboard/internal package (aliased via module federation shared config) provides shared frontend utilities from the host app to modules without circular dependencies. This architecture enables independent team ownership and deployment cadence for each module while maintaining a unified user experience.

The FIPS compliance story is split between the Node.js host (which inherits FIPS-compliant OpenSSL from the UBI 9 base image) and the Go BFF sidecars (which use CGO_ENABLED=1 with -tags strictfipsruntime for FIPS-compliant crypto). The Dockerfile explicitly removes esbuild Go binaries from the production image as a safety net against non-FIPS binaries leaking through transitive build dependencies. The Go BFFs enforce TLS 1.3 minimum for upstream connections and support custom CA bundles via the BUNDLE_PATHS configuration, enabling integration with enterprise PKI infrastructure.
