## Purpose

**Short**: The RHOAI/ODH Dashboard is the web-based management console for Red Hat OpenShift AI, providing a unified interface for managing data science projects, model serving, notebooks, pipelines, and AI/ML platform features.

**Detailed**: odh-dashboard is a large frontend + backend-for-frontend (BFF) monorepo that produces the primary user interface for the Red Hat OpenShift AI (RHOAI) and Open Data Hub (ODH) platforms. The architecture follows a module federation pattern: a host React application (built with PatternFly 6) serves as the shell, while feature-specific UI modules (gen-ai, model-registry, maas, mlflow, eval-hub, automl, autorag, agent-ops) are independently built and deployed as federated remotes, each with its own Go-based BFF HTTP server.

The Node.js backend (Fastify) acts as the primary BFF for the host application, proxying Kubernetes API calls, managing WebSocket connections for real-time resource watching, and routing requests to upstream data science services (Pipelines, TrustyAI, Feast, Model Serving). Each federated module's Go BFF serves its own API surface and proxies to its respective upstream services (LlamaStack, MLflow, MaaS Gateway, Kubeflow Pipelines, EvalHub, OGX).

A Go-based dashboard-operator (controller-runtime) manages the Dashboard custom resource, handling kustomize-based manifest rendering, module lifecycle (dependency resolution, enablement/disablement), and status reporting. The operator reconciles the complete deployment topology including the host application, sidecar BFF containers, RBAC, networking, and ingress configuration.

## Architectural Analysis

The odh-dashboard architecture is a sophisticated module federation monorepo that has evolved from a monolithic React+Node.js application into a distributed, plugin-based platform. The core insight is that each feature domain (GenAI, Model Registry, MaaS, MLflow, AutoML, etc.) is packaged as an independently deployable "module" consisting of a React micro-frontend (loaded via Webpack Module Federation's `remoteEntry.js`) and a Go BFF HTTP server. This allows features to be enabled/disabled at deployment time without rebuilding the host application.

The deployment topology is particularly noteworthy. In the default "sidecar mode," all BFF modules run as additional containers within the same pod as the host dashboard — sharing a single Kubernetes Service with different ports (8043, 8143, 8243, 8343, 8543, 8643, 8743, 8843). The kube-rbac-proxy sidecar guards the external entry point (8443), forwarding authenticated requests with identity headers to the Node.js backend (8080), which in turn proxies BFF API calls to the appropriate sidecar port based on federation config. An alternative "standalone mode" deploys each module as its own Deployment+Service, primarily used for the automl, autorag, eval-hub, gen-ai, maas, mlflow, and model-registry modules when the Dashboard operator determines they should run independently.

The dashboard-operator introduces a CRD-driven lifecycle model for modules. The `Dashboard` CR (cluster-scoped, singleton-validated via admission webhook) carries component availability signals projected by the platform operator (`spec.components`) and optional per-module overrides (`spec.modules`). A two-pass dependency resolution algorithm determines which modules should be `Deployed`, `NotDeployed`, or `Disabled`, supporting both DSC component dependencies (e.g., model-registry requires `modelregistry` component) and inter-module dependencies (e.g., `mlflowEmbedded` requires `mlflow`). This decouples the dashboard from the platform operator's versioning while maintaining correct feature gating.

Security follows a defense-in-depth pattern with three layers: (1) kube-rbac-proxy enforces that the user can list OpenShift projects before any request reaches the backend; (2) the Node.js backend applies `secureRoute`/`secureAdminRoute` middleware that checks group membership for admin operations; (3) each Go BFF performs Kubernetes SubjectAccessReview calls to verify per-namespace access to the specific resources being requested. Service account isolation is carefully managed — the main dashboard containers use the `odh-dashboard` SA (with broad RBAC including notebook and model registry CRUD), while BFF modules use a separate `odh-dashboard-modules` SA with a narrower ClusterRole scoped to discovery-level access (MLflow CRs, ingresses, SAR creation, user impersonation). Inter-BFF communication (e.g., gen-ai calling maas for token management) forwards the original user's access token, maintaining the security context across the call chain.

The `OdhDashboardConfig` CRD serves as the feature flag mechanism, with a comprehensive set of `disable*` and enable boolean fields that control UI visibility of platform features. This CR is read by the frontend at startup and cached, allowing administrators to hide features without redeployment. The system supports over 30 independently toggleable features spanning model serving, pipelines, notebooks, model catalog, GenAI studio, guardrails, AutoRAG, AutoML, and more.
