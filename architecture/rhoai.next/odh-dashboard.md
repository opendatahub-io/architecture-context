## Purpose

**Short**: Web-based management console for Red Hat OpenShift AI (RHOAI) that provides a unified UI for managing ML/AI workloads, notebooks, model serving, pipelines, and platform configuration.

**Detailed**: The odh-dashboard is a React-based single-page application with a Node.js/Fastify backend-for-frontend (BFF) that serves as the primary management interface for the RHOAI platform. It provides administrators and data scientists with capabilities to manage Jupyter notebooks and workbenches, configure model serving via KServe, interact with ML pipelines (Kubeflow Pipelines), manage model registries, configure hardware/accelerator profiles, and control platform-wide settings through feature flags stored in the OdhDashboardConfig CRD.

The dashboard employs a modular architecture using Webpack Module Federation, where the core dashboard shell hosts up to 7 independently deployable federated UI modules (Gen AI, MaaS, AutoML, AutoRAG, Eval Hub, MLflow, Model Registry). Each federated module ships as a separate container with its own Go-based BFF backend, running as sidecar containers within the same pod. The Go BFFs proxy requests to upstream services (LlamaStack, MaaS API, MLflow, Pipeline Server, Model Registry API) while the core Node.js BFF proxies requests to the Kubernetes API server and platform services. Authentication is enforced at the pod boundary by kube-rbac-proxy, which validates OpenShift RBAC before forwarding requests with identity headers to the backend containers.

The repository supports both ODH (Open Data Hub) and RHOAI distributions through a kustomize overlay system, with RHOAI further split into addon (SaaS) and on-premise variants. Distribution-specific branding, feature flags, and application catalogs are injected via kustomize parameterization without code duplication.

## Architectural Analysis

The odh-dashboard represents one of the most architecturally sophisticated components in the RHOAI platform, employing a Module Federation pattern that enables independent development and deployment of feature-specific UI modules while maintaining a unified user experience. The core dashboard shell (React/TypeScript) acts as a host application that dynamically loads federated remote modules at runtime from sidecar containers running within the same pod. This design allows teams to develop features like Gen AI Studio, MaaS, and AutoML independently, each with their own Go-based BFF backend, without requiring coordination on the core dashboard release cycle.

The authentication architecture follows a defense-in-depth pattern: kube-rbac-proxy sits at the pod boundary on port 8443, enforcing OpenShift RBAC by validating that the user can "list projects" before allowing any request through. Once past kube-rbac-proxy, the user's identity is injected via `x-forwarded-access-token` and `x-auth-request-user` headers. The Node.js backend then applies application-level authorization (admin checks, namespace ownership, resource name validation) before proxying to upstream services with the user's token. This means the dashboard never stores credentials -- it always delegates authentication to the platform and propagates user identity to downstream services.

The Go BFF modules share a remarkably consistent architecture: all use httprouter for routing, implement the same middleware stack (CORS, request identity injection, panic recovery, telemetry), and support identical authentication configuration (user_token, internal, disabled modes). They differ primarily in which upstream services they proxy to and what domain-specific API routes they expose. The gen-ai BFF is the most complex, integrating with LlamaStack, MaaS, MLflow, and MCP, while the mlflow BFF is the simplest with just status and experiment listing. Notably, the gen-ai BFF communicates with the MaaS BFF via inter-BFF calls for token issuance, creating a dependency between sidecar containers.

The deployment manifests demonstrate a well-designed kustomize composition pattern that supports four distribution variants (ODH, RHOAI Shared, RHOAI Addon, RHOAI OnPrem) from a single base set of resources. RHOAI variants apply JSON6902 patches to rename all resources from `odh-dashboard` to `rhods-dashboard` and inject additional applications. The modular architecture layer adds 7 sidecar containers and a federation ConfigMap that maps module names to their service ports, proxy rules, and auth requirements. The NetworkPolicy is notably permissive for egress (all in-cluster traffic allowed, external on 80/443), reflecting the dashboard's role as a universal proxy that needs to reach many platform services. The ingress side is more restrictive, allowing only the OpenShift ingress controller and same-namespace pods.

A notable architectural risk is the pod's resource footprint: the deployment runs 9+ containers (dashboard, kube-rbac-proxy, and 7 BFF sidecars) each requesting 300-500m CPU and 512Mi-1Gi memory. With 2 replicas and pod anti-affinity across zones, this requires significant cluster resources. The Module Federation ConfigMap enables runtime configuration of which modules are loaded, allowing operators to disable unused modules, but the sidecar containers still run regardless of feature flag state. The FIPS compliance requirement is addressed by building Go BFFs with `-tags strictfipsruntime` and removing esbuild binaries from the Node.js image, ensuring all cryptographic operations use FIPS-validated implementations.
