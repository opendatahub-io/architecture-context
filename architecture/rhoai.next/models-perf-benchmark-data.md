## Purpose

**Short**: Packages Red Hat AI validated model performance benchmark data into a container image consumed as an init container by the model-catalog deployment.

**Detailed**: models-perf-benchmark-data is a data-only container that provides a structured catalog of AI model performance, evaluation, and validation data. The container image bundles static JSON and NDJSON files organized in a `provider/model` directory hierarchy (e.g., `meta-llama/Llama-3.1-8B-Instruct/`) under `/app/benchmarks`. This data includes model metadata, provider information, performance metrics (latency, throughput), evaluation scores (accuracy, F1), and RHOAI validation results.

The container is deployed as an init container within the `model-catalog` deployment in the `odh-model-registries` namespace. It populates a shared volume with benchmark data that the model-catalog service then serves. The container itself runs `sleep infinity` as its default command, but when used as an init container in Kubernetes, the consuming deployment overrides the command to copy data into the shared volume before the main container starts.

The repository also contains a Kafka processor tool (`kafka-processor/`) used during development to ingest model benchmark data from Kafka topics (CDC envelope format) and transform it into the structured file hierarchy that gets packaged into the container image. This tool is not shipped in the production image.

## Architectural Analysis

This component follows a data-as-code pattern where benchmark data is versioned in git and packaged into a container image for deployment. The separation between data preparation tooling (kafka-processor) and the production artifact (Dockerfile.konflux) is clear: the kafka-processor consumes CDC events from Kafka topics, transforms them through a pipeline (entity detection, field mapping, post-processing transforms), and writes structured files to disk. These files are then committed to the repository and baked into a minimal UBI9 container.

The kafka-processor implements a CDC (Change Data Capture) envelope pattern supporting both a simple format (`{"op": "CREATE", "data": {...}}`) and a legacy Debezium format. Entity types are detected via a priority chain: topic name matching first, then explicit `entity_type`/`type` fields, then heuristic field-based detection. This multi-layer detection provides flexibility but creates ambiguity risk -- a message with both performance and evaluation fields could be misclassified. The processor supports five entity types: model, model-provider, model-performance, model-evaluation, and model-validation, plus model-variant-group.

The post-processing transform system (`transforms.py`) is well-designed with an abstract base class supporting two target file types (metadata.json and performance.ndjson). Currently two transforms exist: `OverallAccuracyTransform` (computes mean evaluation score) and `HardwareConfigurationTransform` (combines hardware_type and hardware_count into a display string). The plugin architecture makes it easy to add new enrichments.

A notable architectural decision is using the production container's default `CMD ["sleep", "infinity"]` -- this allows the image to work both as a standalone container for debugging (keeping it alive for file inspection) and as an init container where the consuming deployment overrides the command. The README documents the deployment pattern: the image is patched into the model-catalog deployment's init container array at index 1.

The repository is marked as confidential and private, containing proprietary benchmark data for Red Hat AI validated models. This is consistent with its role providing curated, validated performance data for the RHOAI model catalog rather than community-sourced metrics.
