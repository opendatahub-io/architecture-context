## Purpose

**Short**: Provides a catalog of Red Hat AI validated model performance benchmark data as a container image, used as an init container by the model-catalog deployment.

**Detailed**: models-perf-benchmark-data is a data-only container that packages model performance benchmark data (metadata, evaluations, performance metrics, and validation results) for AI/ML models validated by Red Hat. The container image is built from `Dockerfile.konflux` on a UBI9-minimal base and contains no runtime code — it simply holds static JSON/NDJSON data files at `/app/benchmarks` and runs `sleep infinity` to keep the container alive as a data volume source.

The repository also contains two Python data-ingestion utilities that populate the `data/` directory before container builds. The **kafka-processor** consumes CDC (Change Data Capture) messages from Kafka topics to produce the structured benchmark data files. The **api-client** fetches the same data from a REST API (model-validation-hub) as an alternative ingestion path. Both utilities produce an identical output structure: provider directories containing model subdirectories with `metadata.json`, `performance.ndjson`, `evaluations.ndjson`, and `validation.ndjson` files plus a top-level `manifest.json` index. A parity validation script (`validate-parity.sh`) ensures both ingestion paths produce equivalent output. Additionally, a one-off enrichment script (`enrich_models.py`) merges cold-start time, minimum VRAM, and modelcar image size data into existing model metadata files.

In the RHOAI platform, this container is used as an init container in the `model-catalog` deployment within the `odh-model-registries` namespace, where it provides the benchmark data volume that the model-catalog service serves to the dashboard.

## Architectural Analysis

This repository follows a **data packaging pattern** common in Kubernetes deployments where static data needs to be shipped alongside application containers. The primary artifact — the Konflux-built container image — contains no executable code; it is a pure data container based on UBI9-minimal that packages JSON/NDJSON files at `/app/benchmarks`. In the platform deployment, it runs as an init container alongside the model-catalog service, copying benchmark data into a shared volume. The `CMD ["sleep", "infinity"]` is the image default but is overridden in practice by the Kubernetes init container spec.

The data ingestion layer is intentionally separated from the shipped artifact. Two parallel ingestion paths exist: a **Kafka consumer** (kafka-processor) that processes CDC envelope messages from five `dev.aihub-public-*` topics, and a **REST API client** (api-client) that fetches the same data from the model-validation-hub. Both produce an identical directory structure organized by provider/model hierarchy with JSON metadata and NDJSON time-series files. The `validate-parity.sh` and `find-missing-records.sh` scripts verify that both paths produce equivalent output, suggesting this dual-path design serves as both a reliability measure and a migration strategy from one data source to another.

The kafka-processor implements a pluggable transform architecture (`Transform` ABC with `OverallAccuracyTransform` and `HardwareConfigurationTransform`) that enriches data during post-processing. Transforms target either `metadata.json` (model-level) or `performance.ndjson` (record-level), applied after all messages are consumed and deduplicated. The api-client implements equivalent post-processing in `postprocess.py` but without the formal transform framework. The CDC envelope parser also includes temporary Debezium format support for legacy data migration.

A notable security consideration is the api-client's `_CertNoHostnameAdapter` class, which creates a custom SSL context that trusts a specific CA certificate but skips hostname verification (`check_hostname = False`, `assert_hostname = False`). This is documented as being necessary for the model-validation-hub's self-signed certificate which lacks a SAN for the server hostname. While this does not break FIPS compliance, it reduces TLS security and should be addressed when the upstream service provisions a proper certificate.

The `cold-start-vram-data/` directory contains enrichment data and a one-off script that merges cold-start timing, minimum VRAM requirements, and modelcar image sizes into existing model metadata files. This enrichment data is maintained separately from the main ingestion pipelines and appears to be sourced from manual benchmarking or external analysis, not from Kafka or the REST API.
