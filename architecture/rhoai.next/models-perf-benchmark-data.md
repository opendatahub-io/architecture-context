## Purpose

**Short**: Container image that provides static benchmark data (performance metrics, evaluation scores, validation results) for AI models in the Red Hat OpenShift AI Model Catalog.

**Detailed**: models-perf-benchmark-data is a data-only container image used as an init container in the model-catalog deployment within RHOAI. It packages pre-processed model performance benchmark data into a minimal UBI9 container that exposes a volume at `/app/benchmarks`. When deployed, the init container populates a shared volume with JSON/NDJSON files organized by provider and model, which the model-catalog service then reads to display model performance data, evaluation scores, and validation status in the dashboard.

The repository also contains two Python data ingestion tools that are NOT shipped as part of the product image: a **kafka-processor** that consumes model benchmark data from Kafka topics and writes it to the filesystem, and an **api-client** that fetches the same data from a REST API (model-validation-hub). Both tools produce an identical directory structure of JSON and NDJSON files. The output from either tool is committed to the `data/` directory in the repo, which is then baked into the container image by Konflux. The kafka-processor has its own Dockerfile (not Konflux-based) for development/testing use; only the root `Dockerfile.konflux` produces the shipped product image.

## Architectural Analysis

This repository follows a "data-as-container" pattern common in Kubernetes: static data is baked into a container image at build time and delivered to the consuming service at runtime via an init container and shared volume. The shipped artifact is remarkably simple -- a UBI9-minimal image containing only JSON and NDJSON files with `sleep infinity` as its entrypoint. The real complexity lives in the data ingestion pipeline, which exists solely to populate the `data/` directory that gets committed to the repository.

The dual-ingestion architecture (Kafka consumer vs. REST API client) is notable. Both tools produce structurally identical output, which is verified by the `validate-parity.sh` and `find-missing-records.sh` scripts. The kafka-processor handles CDC (Change Data Capture) envelopes including legacy Debezium format, while the api-client uses the model-validation-hub's v1 REST API. The api-client was added later (commit `228141f`) as the v1 API became available, and both paths coexist for flexibility. The shared post-processing logic (deduplication, overall accuracy calculation, hardware configuration enrichment) is implemented independently in each tool rather than being shared as a library, which creates a maintenance risk for drift.

The data schema is well-structured: models are organized by HuggingFace repo name (creating a natural `provider/model` directory hierarchy), with separate NDJSON append files for time-series data (performance metrics, evaluation scores) and JSON files for entity metadata. Validation data is merged directly into `metadata.json` rather than stored separately, which is a pragmatic choice given that validation status is a property of the model rather than a time-series observation. The `cold-start-vram-data` enrichment script adds hardware-specific fields (minimum VRAM, modelcar image size, cold start matrix with GPU-specific runtime commands) to existing metadata files from a separate JSON dataset.

From a security perspective, the shipped image is minimal-risk: it contains no executables, no network services, no credentials, and runs as a non-root user (UID 1001). The development tools handle credentials via environment variables (Kafka SASL, TLS certs) and support `.env` files for local configuration. The api-client's TLS implementation includes a custom adapter that trusts a specific CA cert while skipping hostname verification -- this is explicitly designed for the self-signed certificate on model-validation-hub.us-east4.gcp.redhat.com and would be a concern if the pattern were used more broadly. The repository README warns that it contains confidential benchmark data and must remain private.
