## Purpose

**Short**: Implements Ragas evaluation as an out-of-tree Llama Stack provider, enabling LLM evaluation via Llama Stack's eval API with both in-process (inline) and Kubeflow Pipelines (remote) execution modes.

**Detailed**: This component provides a Ragas-based evaluation provider for the Llama Stack framework. Ragas is an open-source evaluation framework for Retrieval-Augmented Generation (RAG) systems. The provider bridges Ragas evaluation capabilities into Llama Stack's standardized eval API, enabling users to evaluate LLM outputs using metrics such as answer relevancy, context precision, faithfulness, and context recall.

The provider ships two execution modes. The **inline** provider runs Ragas evaluation in the same process as the Llama Stack server, wrapping the Llama Stack inference API (for LLM completions and embeddings) with LangChain-compatible adapters that Ragas can consume. The **remote** provider offloads evaluation to Kubeflow Pipelines, submitting multi-step pipeline runs that retrieve datasets from Llama Stack, execute Ragas evaluation in isolated containers, and persist results to S3-compatible storage. The remote mode uses the `llama-stack-client` SDK to communicate with the Llama Stack server from within Kubeflow pipeline components.

The component is published to PyPI as `llama-stack-provider-ragas` and is also built as a container image via Konflux for use as the base image in Kubeflow pipeline components. The image resolution for pipeline components follows a cascade: environment variable override, then a Kubernetes ConfigMap (`trustyai-service-operator-config`) in platform namespaces (`redhat-ods-applications`, `opendatahub`), then a hardcoded default.

## Architectural Analysis

The llama-stack-provider-ragas component follows the Llama Stack external provider pattern, implementing the `Eval` API interface to plug Ragas evaluation capabilities into any Llama Stack distribution. The dual-mode architecture (inline vs. remote) is a deliberate design choice that trades deployment complexity for scalability. The inline mode is suitable for development and small-scale evaluation, running synchronously within the Llama Stack process. The remote mode decouples evaluation execution from the server by delegating to Kubeflow Pipelines, enabling longer-running evaluations without blocking the server and providing Kubernetes-native job lifecycle management.

The adapter pattern used to bridge Llama Stack's inference API to Ragas' LangChain-based interfaces is architecturally significant. The inline wrappers (`LlamaStackInlineLLM`, `LlamaStackInlineEmbeddings`) use the Llama Stack server's internal Python API objects directly, while the remote wrappers (`LlamaStackRemoteLLM`, `LlamaStackRemoteEmbeddings`) use the `llama-stack-client` SDK for HTTP-based communication. This means the remote pipeline components need network access back to a Llama Stack server for inference -- the Llama Stack URL must be accessible from within Kubeflow pods, which has networking implications for cluster deployments.

The base image resolution cascade (`KUBEFLOW_BASE_IMAGE` env var -> `trustyai-service-operator-config` ConfigMap -> hardcoded default) in `components.py` is a pragmatic integration pattern with the TrustyAI operator. It allows platform operators to control which container image runs in evaluation pipelines without modifying the provider configuration, while still working standalone with a default image. The ConfigMap lookup scans both `redhat-ods-applications` (RHOAI) and `opendatahub` (ODH) namespaces, maintaining compatibility across distributions.

The compatibility layer (`compat.py`) handles the ongoing migration between `llama_stack` and `llama_stack_api` package namespaces. This is a transitional concern -- the codebase wraps all Llama Stack imports with try/except fallbacks, which adds import overhead but ensures the provider works across multiple Llama Stack versions. This pattern suggests the upstream API surface is still stabilizing.

A notable security consideration is that the remote provider communicates with both the KFP API and S3 storage using credentials that must be pre-provisioned as Kubernetes Secrets. The KFP token extraction (`_get_kfp_token`) attempts to use a configured `pipelines_api_token` (which is properly typed as `SecretStr` to prevent log leakage) and falls back to extracting the bearer token from the local kubeconfig. The S3 credentials are injected into KFP pod environments via `kubernetes.use_secret_as_env()`, which is the standard KFP pattern but means credentials are visible as environment variables within the pod.
