## Purpose

**Short**: An out-of-tree Llama Stack evaluation provider that implements Ragas-based LLM evaluation, supporting both inline (in-process) and remote (Kubeflow Pipelines) execution modes.

**Detailed**: This component implements the [Ragas](https://github.com/explodinggradients/ragas) evaluation framework as an external provider for [Meta's Llama Stack](https://github.com/meta-llama/llama-stack) platform. It plugs into Llama Stack's evaluation API (`/alpha/eval`) to provide LLM evaluation capabilities using Ragas metrics such as answer relevancy, context precision, faithfulness, and context recall.

The provider ships in two execution modes. The **inline** provider runs Ragas evaluation in the same process as the Llama Stack server, using wrapper classes (`LlamaStackInlineLLM`, `LlamaStackInlineEmbeddings`) that adapt Llama Stack's inference API to Ragas' `BaseRagasLLM` and `BaseRagasEmbeddings` interfaces. The **remote** provider offloads evaluation to Kubeflow Pipelines on OpenShift, constructing a two-step KFP pipeline (data retrieval → Ragas evaluation) with results persisted to S3. The remote provider uses a separate set of wrappers (`LlamaStackRemoteLLM`, `LlamaStackRemoteEmbeddings`) that communicate with a Llama Stack server via HTTP using `llama_stack_client`.

The component is deployed as a container image built by Konflux (`Dockerfile.konflux`) based on UBI9 Python 3.12. On RHOAI, the base image for Kubeflow pipeline components is resolved from a Kubernetes ConfigMap (`trustyai-service-operator-config`) in the `redhat-ods-applications` or `opendatahub` namespace, falling back to a default Quay image. The provider registers itself with Llama Stack via the provider spec mechanism, exposing `inline::trustyai_ragas` and `remote::trustyai_ragas` provider types.

## Architectural Analysis

The llama-stack-provider-ragas component follows Meta's Llama Stack external provider pattern, implementing the `Eval` and `BenchmarksProtocolPrivate` interfaces. The provider registers itself via a `get_provider_spec()` function that returns `InlineProviderSpec` and `RemoteProviderSpec` objects, allowing Llama Stack to discover and load it at runtime. This is a plugin architecture — the provider runs inside the Llama Stack server process and is configured via the distribution YAML file (`distribution/run.yaml`).

The dual inline/remote architecture is a key design decision. The inline provider is simpler — it adapts Llama Stack's async inference API to Ragas' `BaseRagasLLM`/`BaseRagasEmbeddings` interfaces using wrapper classes, then calls `ragas.evaluate()` in a thread pool (`asyncio.to_thread`). The remote provider is more complex — it constructs a Kubeflow Pipeline with `@dsl.component` decorated functions, submits it via the KFP SDK, and asynchronously polls for completion. A notable pattern is the base image resolution chain: environment variable → Kubernetes ConfigMap (`trustyai-service-operator-config`) → hardcoded default. This three-level fallback integrates with the TrustyAI Service Operator's configuration mechanism on RHOAI.

The compatibility layer (`compat.py`) is worth noting — it implements a try/except import pattern to support both the legacy `llama_stack` package and the newer `llama_stack_api` package. This suggests the codebase is tracking an upstream API migration. The metrics are currently limited to four Ragas metrics (`answer_relevancy`, `context_precision`, `faithfulness`, `context_recall`), with TODOs indicating planned expansion to `answer_correctness`, `factual_correctness`, and others.

A security consideration is that the remote provider's KFP pipeline components call back to the Llama Stack server over HTTP without TLS or authentication. The `KUBEFLOW_LLAMA_STACK_URL` must be accessible from KFP pods, and the connection is unencrypted. AWS credentials for S3 are properly injected via Kubernetes Secrets into KFP pods using `kubernetes.use_secret_as_env()`. The KFP authentication token can be provided via environment variable or extracted from the local kubeconfig, which handles both CI and interactive use cases.
