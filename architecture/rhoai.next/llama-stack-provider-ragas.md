## Purpose

**Short**: An out-of-tree Llama Stack evaluation provider that implements Ragas (Retrieval Augmented Generation Assessment) metrics for evaluating LLM applications through both inline (in-process) and remote (Kubeflow Pipelines) execution modes.

**Detailed**: This component implements the Llama Stack Eval API by wrapping the [Ragas](https://github.com/explodinggradients/ragas) evaluation framework as a provider plugin. It enables users to run standardized RAG (Retrieval Augmented Generation) evaluation metrics — including answer relevancy, context precision, faithfulness, and context recall — against LLM-generated outputs.

The provider ships in two execution modes. The **inline** mode runs evaluations within the same process as the Llama Stack server, using adapter wrappers (`LlamaStackInlineLLM`, `LlamaStackInlineEmbeddings`) that bridge the Llama Stack inference API to the Ragas/LangChain interface. The **remote** mode offloads evaluation to Kubeflow Pipelines on OpenShift, constructing a two-step KFP pipeline that retrieves data from a Llama Stack server and then runs Ragas evaluation in a containerized pod. Results are stored in S3 and fetched back via pandas. The remote mode supports the TrustyAI evaluation workflow in RHOAI, integrating with Data Science Pipelines (DSP) for scalable, asynchronous evaluation jobs.

The component is deployed as a container image (`odh-trustyai-ragas-lls-provider-dsp`) built from `Dockerfile.konflux` on a UBI9 Python 3.12 base. It reads base image configuration from a `trustyai-service-operator-config` ConfigMap managed by the TrustyAI operator, and authenticates to Kubeflow Pipelines via bearer tokens sourced from OpenShift kubeconfig or explicit environment variables.

## Architectural Analysis

The llama-stack-provider-ragas component follows the Llama Stack out-of-tree provider pattern, implementing the `Eval` and `BenchmarksProtocolPrivate` interfaces to plug into any Llama Stack server distribution. The architecture cleanly separates two execution modes behind a unified API surface: inline execution for development and small-scale evaluation, and remote execution for production-scale workloads on OpenShift.

The **compatibility layer** (`compat.py`) is a notable architectural choice. It implements a try/except import pattern to bridge two different Llama Stack package layouts (`llama_stack` vs. `llama_stack_api`), allowing the provider to work across Llama Stack versions 0.2.x and 0.5.x without code changes. This pattern trades runtime import overhead for version flexibility, which is pragmatic for a component that must track a rapidly evolving upstream API.

The **remote provider's Kubeflow integration** is the most architecturally significant aspect. The provider constructs KFP pipelines with two components: `retrieve_data_from_llama_stack` (data fetching) and `run_ragas_evaluation` (metric computation). Both components use a dynamically resolved base image — first checking an environment variable, then falling back to a ConfigMap (`trustyai-service-operator-config`) in candidate namespaces (`redhat-ods-applications`, `opendatahub`), and finally defaulting to `quay.io/trustyai/llama-stack-provider-ragas:latest`. This three-tier resolution enables operator-managed image updates without requiring provider reconfiguration.

A notable security consideration is that the **KFP token handling** uses Pydantic's `SecretStr` for the `pipelines_api_token` field, ensuring tokens are redacted in logs and serialization. The fallback token extraction from kubeconfig (`_load_kube_config()`) supports both in-cluster and out-of-cluster operation. However, the S3 credentials are mounted as plain environment variables in KFP pods via `kubernetes.use_secret_as_env()`, which is the standard KFP pattern but means credentials are visible in pod environment inspection.

The inline provider is deliberately constrained with `max_workers=1` in the Ragas `RunConfig`, which serializes all LLM and embedding calls. This avoids overwhelming the inference backend but limits throughput for large evaluation datasets. The remote provider has no such constraint since each KFP pod runs independently. The provider also uses `asyncio.to_thread()` for the inline Ragas evaluation call, preventing the synchronous Ragas `evaluate()` function from blocking the Llama Stack server's async event loop.
