## Purpose

**Short**: An out-of-tree Llama Stack evaluation provider that implements Ragas evaluation metrics, supporting both inline (in-process) and remote (Kubeflow Pipelines) execution modes.

**Detailed**: This component implements [Ragas](https://github.com/explodinggradients/ragas) as an external provider for the [Llama Stack](https://github.com/meta-llama/llama-stack) evaluation API. It bridges the Ragas evaluation framework with Llama Stack's standardized eval, inference, dataset, and benchmark APIs, enabling users to run LLM evaluation metrics (faithfulness, answer relevancy, context precision, context recall) through the Llama Stack interface.

The provider ships in two execution modes. The **inline** provider runs Ragas evaluation directly within the Llama Stack server process, using the server's inference API for LLM and embedding calls. The **remote** provider offloads evaluation to Kubeflow Pipelines, constructing a two-step KFP pipeline (data retrieval + Ragas evaluation) that runs on a Kubernetes cluster, stores results in S3, and reports status back through the Llama Stack job API. This separation allows lightweight local evaluations via inline mode and scalable, distributed evaluations via remote mode on OpenShift AI clusters.

The component is published to PyPI as `llama-stack-provider-ragas` and is also built as a container image via Konflux for deployment within RHOAI. It integrates with the TrustyAI ecosystem and references the `trustyai-service-operator-config` ConfigMap for base image resolution when running remote evaluations.

## Architectural Analysis

The llama-stack-provider-ragas component follows a clean plugin architecture, implementing the Llama Stack `Eval` and `BenchmarksProtocolPrivate` interfaces. The dual-mode design (inline vs. remote) is the most notable architectural decision. The inline provider is straightforward -- it wraps Ragas' `evaluate()` function with Llama Stack-compatible LLM and embedding wrappers (`LlamaStackInlineLLM`, `LlamaStackInlineEmbeddings`) that translate between Ragas' LangChain-based interfaces and Llama Stack's OpenAI-compatible inference API. The remote provider is more complex: it constructs a Kubeflow Pipelines DAG with two steps, each running as a containerized component using a separate set of wrappers (`LlamaStackRemoteLLM`, `LlamaStackRemoteEmbeddings`) that use the `llama_stack_client` SDK to make HTTP calls back to the Llama Stack server from within the KFP pod.

A key architectural concern is the wrapper duplication between inline and remote modes. Both modes implement essentially the same LLM and embedding adapters, but with different underlying transport mechanisms. The inline wrappers use the in-process Llama Stack inference API (async), while the remote wrappers use the `llama_stack_client` HTTP client (sync and async). This duplication is a conscious design trade-off: the inline wrappers operate within the server's asyncio event loop and use internal API objects, while the remote wrappers must be serializable as KFP component code and operate against a network endpoint. The `compat.py` module adds another layer, providing backward compatibility between the legacy `llama_stack` package layout and the newer `llama_stack_api` module structure.

The remote provider's base image resolution strategy is notable: it first checks the `KUBEFLOW_BASE_IMAGE` environment variable, then falls back to reading a ConfigMap (`trustyai-service-operator-config`) from either `redhat-ods-applications` or `opendatahub` namespaces, and finally defaults to a public quay.io image. This three-tier fallback ensures the provider works in development, managed RHOAI, and ODH environments without explicit image configuration. The ConfigMap approach ties the provider to the TrustyAI operator ecosystem, which manages the image reference centrally.

The component has a meaningful supply chain gap: the Konflux Dockerfile uses `pip install` rather than `uv sync --frozen`, so while `uv.lock` exists in the repository, it is not enforced during the container build. This means the production image may install different dependency versions than what was tested locally. Additionally, the absence of `rpms.lock.yaml` and Hermeto prefetch integration means OS-level packages and non-PyPI artifacts are not locked. The component also does not use AIPCC base images, which is a compliance gap for RHOAI product images. Migration to the AIPCC CPU base image would address both the secure build requirement and the pip/uv configuration gap (AIPCC images ship with pre-configured `pip.conf` and `uv.toml` pointing to the RHEL AI Python Package Index).
