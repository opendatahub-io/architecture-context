## Purpose

**Short**: An out-of-tree Llama Stack evaluation provider that implements Ragas (Retrieval Augmented Generation Assessment) metrics over the Llama Stack Eval API, with both inline (in-process) and remote (Kubeflow Pipelines) execution modes.

**Detailed**: This component bridges the Ragas evaluation framework with Meta's Llama Stack platform by implementing the Llama Stack `Eval` API as an external provider plugin. It enables model evaluation using Ragas metrics (answer relevancy, context precision, faithfulness, context recall) through the standard Llama Stack eval interface, abstracting away the Ragas-specific evaluation machinery behind a unified API.

The provider ships in two execution modes. The **inline provider** runs Ragas evaluation directly within the Llama Stack server process, wrapping the Llama Stack inference API into LangChain-compatible LLM and embedding wrappers that Ragas can consume. The **remote provider** offloads evaluation to Kubeflow Pipelines, constructing a two-step KFP pipeline (data retrieval from Llama Stack, then Ragas evaluation) that runs as Kubernetes pods. Remote evaluation results are persisted to S3 and fetched on demand.

The component is deployed as a container image built on UBI9/Python 3.12 for RHOAI, and is also published as a standalone PyPI package (`llama-stack-provider-ragas`) for upstream use. It integrates with the TrustyAI ecosystem via the `trustyai-service-operator-config` ConfigMap for image resolution, and with Kubeflow Pipelines for remote execution orchestration.

## Architectural Analysis

The llama-stack-provider-ragas component follows a clean provider plugin architecture that separates concerns between the evaluation orchestration layer (Llama Stack's Eval API) and the evaluation execution engine (Ragas). The dual-mode design (inline vs. remote) is the most significant architectural decision: the inline provider runs everything in-process within the Llama Stack server, while the remote provider delegates execution to Kubeflow Pipelines pods. This is achieved through a shared constant/configuration layer (`config.py`, `constants.py`) and separate provider implementations that both implement the same `Eval` + `BenchmarksProtocolPrivate` interfaces. The compatibility layer (`compat.py`) adds resilience against Llama Stack API restructuring by supporting both the legacy `llama_stack` and newer `llama_stack_api` import paths.

The LLM and embedding wrapper pattern is noteworthy. Ragas expects LangChain-compatible LLM and embedding interfaces (`BaseRagasLLM`, `BaseRagasEmbeddings`), but Llama Stack exposes its own async inference API. The inline wrappers (`wrappers_inline.py`) bridge this by calling the in-process Llama Stack inference API and translating responses to LangChain `Generation`/`LLMResult` objects. The remote wrappers (`wrappers_remote.py`) do the same but use the `llama-stack-client` SDK to make HTTP calls to an external Llama Stack server. Both wrapper sets handle sync-to-async bridging — the inline wrappers use `asyncio.get_event_loop().run_until_complete()` for sync methods (a pattern that can cause issues in nested event loops), while the remote wrappers provide both native sync and async implementations via separate `LlamaStackClient` and `AsyncLlamaStackClient` instances.

The Kubeflow pipeline architecture (`remote/kubeflow/`) reveals a practical integration pattern with RHOAI's Data Science Pipelines. The pipeline is two steps: `retrieve_data_from_llama_stack` fetches the evaluation dataset via the Llama Stack client SDK, and `run_ragas_evaluation` executes the Ragas evaluation using the remote wrappers. AWS credentials for S3 result storage are injected into the evaluation pod via `kubernetes.use_secret_as_env()`. Base image resolution follows a three-tier fallback: environment variable, then Kubernetes ConfigMap (`trustyai-service-operator-config` in `redhat-ods-applications` or `opendatahub` namespaces), then a hardcoded default. This ties the component into the TrustyAI operator's configuration ecosystem. The evaluation runs are max_workers=1 in both modes, suggesting serial metric computation to avoid resource contention.

A notable security consideration is that all Llama Stack API communication (both the hosting server and the Llama Stack URL used by remote KFP pods) defaults to plaintext HTTP on port 8321 with no authentication. The KFP pipeline pods reach back to the Llama Stack server over HTTP, meaning evaluation data and model responses traverse the network unencrypted. The `KUBEFLOW_PIPELINES_TOKEN` is properly handled as a `SecretStr` in Pydantic configuration to prevent log leakage, and S3 credentials are mounted via Kubernetes Secrets rather than hardcoded — but the Llama Stack server itself has no TLS or auth enforcement at the application level.
