## Purpose

**Short**: Manages lifecycle, networking, authentication, and integrations for Jupyter notebook workbenches on Red Hat OpenShift AI.

**Detailed**: This repository contains two cooperating Kubernetes controllers that manage Jupyter notebook instances (workbenches) within the RHOAI platform. The upstream **notebook-controller** (from Kubeflow) handles core notebook lifecycle — creating StatefulSets, Services, and optional Istio VirtualServices, plus idle notebook culling. The downstream **odh-notebook-controller** (Open Data Hub extension) layers RHOAI-specific capabilities on top: kube-rbac-proxy sidecar injection for authentication, Gateway API HTTPRoute creation for ingress, ReferenceGrant management for cross-namespace routing, NetworkPolicy enforcement, and integrations with Data Science Pipelines (DSPA), MLflow, and Feast.

The two controllers run as separate Deployments but operate on the same `Notebook` CRD (`kubeflow.org/v1`). The notebook-controller manages the core pod lifecycle (StatefulSet, Service, status mirroring, culling), while the odh-notebook-controller adds platform-specific mutations via webhooks and reconciles additional resources (HTTPRoutes, NetworkPolicies, RBAC bindings, integration secrets). A shared `common` library provides reconciliation helpers for Deployments, Services, StatefulSets, and VirtualServices.

**Dockerfile**: `components/notebook-controller/Dockerfile.konflux`
**Intent**: The upstream Kubeflow notebook controller adapted for RHOAI. ## Architectural Analysis

The kubeflow repository implements a **dual-controller pattern** where two independent Go operators cooperate on the same Notebook CRD. The upstream `notebook-controller` handles core Kubernetes lifecycle management (StatefulSet, Service, culling), while the downstream `odh-notebook-controller` layers RHOAI platform integrations on top via webhooks and additional reconciliation. This separation preserves upstream compatibility — the notebook-controller can be rebased from the Kubeflow upstream with minimal conflicts, while all RHOAI-specific logic lives in the ODH controller. The `common` library provides shared reconciliation helpers to avoid code duplication.

The **ingress architecture follows the RHOAI 3.x Gateway API pattern**. For each notebook, the ODH controller creates an HTTPRoute in the controller's central namespace (not the notebook's namespace) with a cross-namespace backend reference to the notebook's Service. This cross-namespace reference requires a ReferenceGrant in the notebook's namespace. The controller intelligently manages ReferenceGrants at namespace scope — one grant per namespace shared by all notebooks, with cleanup only when the last notebook in a namespace is deleted. When authentication is enabled (via the `notebooks.opendatahub.io/inject-auth` annotation), the HTTPRoute backend switches from the notebook Service (port 8888) to the kube-rbac-proxy Service (port 8443), and conflicting routes from the alternate mode are cleaned up to prevent routing ambiguity.

The **kube-rbac-proxy sidecar injection** is particularly sophisticated. The mutating webhook injects a full kube-rbac-proxy container into notebook pods with: TLS certificate mounting, authorization config via ConfigMap (checking `get` verb on `notebooks` resource), health probes on a separate port (8444), and configurable resource requests/limits via annotations. The controller also creates the supporting infrastructure: a ServiceAccount, a ClusterRoleBinding to `system:auth-delegator`, a dedicated Service, and NetworkPolicies. A finalizer chain ensures cross-namespace cleanup (ClusterRoleBinding, HTTPRoute, ReferenceGrant) when notebooks are deleted, since owner references cannot span namespaces.

The **webhook-based mutation strategy** addresses a critical race condition (RHOAIENG-24545): pipeline runtime ConfigMaps and DSPA secrets are synced in the mutating webhook during notebook creation, before the first pod starts. Without this, the pod could start without mounted pipeline configuration. The validating webhook prevents removal of MLflow annotations on running notebooks, requiring users to stop the notebook first. The webhook also handles ImageStream-based container image resolution, CA bundle mounting, cluster proxy env var injection, and Feast config mounting.

The **idle culling mechanism** is noteworthy for its design: the culling controller runs as a separate reconciler that periodically queries each notebook's Jupyter HTTP API (`/api/kernels` and `/api/terminals`) to detect activity. It tracks last activity via annotations on the Notebook CR, then sets the `kubeflow-resource-stopped` annotation when idle time exceeds the threshold. The main notebook-controller detects this annotation and scales the StatefulSet to zero replicas. This annotation-based coupling between controllers is elegant — it avoids tight coupling while maintaining clear responsibility boundaries.
