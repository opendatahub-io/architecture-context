## Purpose

**Short**: Manages the lifecycle of Jupyter Notebook workbenches on OpenShift, providing creation, routing, authentication, idle detection, and integration with RHOAI platform services.

**Detailed**: This repository contains two complementary Kubernetes controllers that together manage the full lifecycle of Notebook workbenches in Red Hat OpenShift AI (RHOAI). The upstream **Kubeflow Notebook Controller** (`notebook-controller`) handles core notebook lifecycle — creating StatefulSets, Services, and optionally Istio VirtualServices for each Notebook CR, along with idle detection and automatic culling of unused notebooks. The **ODH Notebook Controller** (`odh-notebook-controller`) extends this with RHOAI-specific functionality: Gateway API HTTPRoute-based ingress, kube-rbac-proxy sidecar injection for authentication, Network Policy enforcement, CA bundle management, and integration with platform services like MLflow, Data Science Pipelines (Elyra), OpenShift ImageStreams, and Feast.

Both controllers watch the same `kubeflow.org/v1.Notebook` CRD but handle different aspects of the notebook lifecycle. The ODH controller uses mutating and validating admission webhooks to inject configuration at notebook creation time (image resolution, CA certificates, proxy settings, authentication sidecars, MLflow environment variables) and prevent unsafe state transitions (MLflow annotation removal on running notebooks). The deployment model follows the RHOAI 3.x pattern: HTTPRoutes reference a central `data-science-gateway` Gateway in `openshift-ingress`, with cross-namespace ReferenceGrants enabling routing from the central namespace to user-namespace Services.

## Architectural Analysis

The kubeflow repository implements a dual-controller pattern where two independent operators watch the same CRD (`kubeflow.org/v1.Notebook`) but manage complementary sets of resources. The upstream Kubeflow Notebook Controller handles the "inner loop" — creating the StatefulSet that runs the notebook pod and the ClusterIP Service that provides stable DNS — while the ODH Notebook Controller handles the "outer loop" of RHOAI-specific concerns: ingress routing, authentication, network isolation, and platform service integration. This separation preserves upstream compatibility while allowing RHOAI to layer on enterprise features without forking the core controller.

The RHOAI 3.x ingress architecture uses Gateway API HTTPRoutes created in the controller's central namespace (not the user's namespace), referencing a platform-wide `data-science-gateway` Gateway in `openshift-ingress`. Cross-namespace Service references are authorized by `ReferenceGrant` CRs in each user namespace. This is a security improvement over the 2.x pattern (OpenShift Routes created in user namespaces with oauth-proxy sidecars) — the user cannot modify the HTTPRoute or the authentication proxy configuration. The kube-rbac-proxy sidecar enforces per-notebook RBAC by performing Kubernetes TokenReview and SubjectAccessReview calls, checking that the request user has `get` permission on the specific `kubeflow.org/notebooks` resource. Each notebook gets its own ClusterRoleBinding to `system:auth-delegator`, which is a cluster-scoped resource requiring explicit finalizer-based cleanup.

The mutating webhook is critical to the initialization flow. It sets a "reconciliation lock" annotation (`kubeflow.io/stop-notebook=odh-notebook-controller-lock`) at creation time, which the upstream controller interprets as a stop signal — setting StatefulSet replicas to 0. This prevents the notebook pod from starting before the ODH controller has finished setting up authentication resources (kube-rbac-proxy Service, ConfigMap, ClusterRoleBinding, HTTPRoute). Once initialization completes, the ODH controller removes the lock annotation, and the upstream controller scales the StatefulSet back to 1. This handshake pattern is elegant but introduces a tight coupling between the two controllers via annotations.

The webhook also handles image resolution (ImageStream tag to pinned digest), CA bundle mounting (merging ODH, cluster root, and service CA certificates into a single `workbench-trusted-ca-bundle` ConfigMap), and MLflow environment variable injection (constructing the tracking URI from the Gateway's public hostname). The validating webhook prevents a specific unsafe state transition: removing the `opendatahub.io/mlflow-instance` annotation from a running notebook. Since the RoleBinding is deleted immediately but environment variables persist until pod restart, this would leave the MLflow client configured but unauthorized — the webhook requires the notebook to be stopped first.

Network isolation follows a defense-in-depth approach. Each notebook gets two NetworkPolicies: one restricting ingress on port 8888/TCP (notebook) to traffic from the controller namespace (where the Gateway proxy lives), and another for port 8443/TCP (kube-rbac-proxy). The kube-rbac-proxy NetworkPolicy allows all ingress to port 8443 (since the proxy itself enforces auth), while the notebook port policy ensures only the ingress controller can reach the unprotected notebook container directly. This means even if an attacker can reach the notebook pod, they cannot bypass the authentication proxy.
