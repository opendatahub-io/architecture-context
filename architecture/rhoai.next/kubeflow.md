## Purpose

**Short**: Manages the lifecycle of Jupyter Notebook workloads on OpenShift, including pod creation, idle culling, authentication via kube-rbac-proxy sidecars, network routing via Gateway API HTTPRoutes, and integrations with Data Science Pipelines, MLflow, and Feast.

**Detailed**: This repository is the OpenDataHub/RHOAI fork of upstream kubeflow/kubeflow, containing two Go-based Kubernetes controllers that together manage Jupyter Notebook workloads:

1. **notebook-controller** (upstream Kubeflow): Watches `Notebook` CRs (kubeflow.org/v1) and reconciles them into StatefulSets and Services. It also implements idle notebook culling by querying the Jupyter `/api/kernels` and `/api/terminals` endpoints to detect inactivity, then scaling down idle notebooks to zero replicas via a stop annotation. When Istio is enabled, it creates VirtualService resources for notebook routing.

2. **odh-notebook-controller** (ODH/RHOAI-specific): Extends the upstream controller with OpenShift and RHOAI-specific functionality. It injects kube-rbac-proxy sidecars for authentication, creates Gateway API HTTPRoutes (RHOAI 3.x pattern) in a centralized namespace for secure notebook routing, manages NetworkPolicies, handles TLS certificate bundles, integrates with Data Science Pipelines (Elyra runtime config), resolves notebook images from OpenShift ImageStreams, integrates with MLflow tracking, and mounts Feast configuration. It uses mutating and validating admission webhooks to inject sidecars and configuration at notebook creation/update time.

Both controllers run as separate Deployments in the same namespace. The upstream controller manages the core StatefulSet lifecycle, while the ODH controller handles all platform-specific concerns including security, networking, and integrations.

## Architectural Analysis

The kubeflow repository implements a two-controller architecture where the upstream Kubeflow notebook-controller handles the core lifecycle (StatefulSet/Service/VirtualService management) while the ODH notebook-controller adds all platform-specific concerns. This separation preserves upstream compatibility while enabling deep OpenShift integration. The controllers share the same CRD (kubeflow.org Notebook) and coordinate via annotations -- notably the `kubeflow-resource-stopped` annotation that the culling controller sets and the upstream controller reads to scale replicas to zero.

The RHOAI 3.x ingress pattern is fully implemented via Gateway API. HTTPRoutes are created in the controller's own namespace (a centralized "protected" namespace) rather than in user namespaces, which is a security-conscious design: users cannot modify or delete their own HTTPRoutes. Cross-namespace backend references are enabled via ReferenceGrant CRs created in each user namespace. The controller manages the complete lifecycle of these cross-namespace resources using Kubernetes finalizers (`notebook.opendatahub.io/httproute-cleanup`, `notebook.opendatahub.io/referencegrant-cleanup`, `notebook.opendatahub.io/kube-rbac-proxy-cleanup`) since standard owner references cannot span namespaces. The controller also handles switching between auth-enabled (kube-rbac-proxy → port 8443) and non-auth (direct → port 8888) HTTPRoute backends, cleaning up conflicting routes during mode transitions.

The kube-rbac-proxy sidecar injection is implemented entirely in the mutating webhook. When the `notebooks.opendatahub.io/inject-auth` annotation is set, the webhook injects a sidecar container that listens on port 8443 with TLS (using service-ca-issued certs), performs TokenReview and SubjectAccessReview against the Kubernetes API, and proxies authenticated traffic to the notebook container on port 8888. The authorization policy is resource-based: users need `get` permission on the specific `notebooks.kubeflow.org` resource, enforced via a ConfigMap-mounted kube-rbac-proxy config file. Each notebook gets its own ClusterRoleBinding to `system:auth-delegator` for the token review/SAR capability.

The webhook implements a sophisticated restart-avoidance mechanism (`maybeRestartRunningNotebook`): when webhook-injected changes (e.g., kube-rbac-proxy image update) would modify the pod template of a running, non-stopped notebook, those changes are blocked and a `notebooks.opendatahub.io/update-pending` annotation is set with a human-readable diff explaining what changed. The updates are only applied when the notebook is stopped and restarted. This prevents unexpected pod restarts from infrastructure updates.

The TLS security posture is well-considered. The ODH controller integrates with OpenShift's `config.openshift.io/v1` APIServer TLS profile via `openshift/controller-runtime-common`, applying the cluster-wide TLS security profile to both the webhook server and metrics server. On non-OpenShift clusters, it falls back to the Mozilla Intermediate cipher suite (TLS 1.2 minimum, ECDHE+AES-GCM and CHACHA20-POLY1305). A `SecurityProfileWatcher` controller watches for TLS profile changes and triggers a graceful shutdown to reload the configuration. This is notable for FIPS compliance -- the cipher suite is explicitly controlled and limited to FIPS-approved algorithms.
