## Purpose

**Short**: Kubernetes operator that manages the lifecycle, networking, authentication, and integrations for Jupyter notebook workbenches on OpenShift AI.

**Detailed**: The kubeflow repository contains two cooperating Go controllers that together manage Kubeflow Notebook custom resources on Red Hat OpenShift AI (RHOAI). The upstream **notebook-controller** (kf-notebook-controller) handles core lifecycle management -- creating StatefulSets, Services, and optional Istio VirtualServices for each Notebook CR, plus an idle-culling controller that monitors Jupyter kernel/terminal activity and stops idle notebooks. The downstream **odh-notebook-controller** extends this with OpenShift-specific functionality: Gateway API ingress via HTTPRoute, kube-rbac-proxy sidecar injection for authentication, NetworkPolicy enforcement, CA certificate trust chain management, and integrations with Data Science Pipelines (DSPA/Elyra), MLflow, Feast, and OpenShift ImageStreams for runtime image resolution.

Both controllers run as separate Deployments but operate on the same Notebook CRD (`kubeflow.org/v1`). The kf-notebook-controller creates the underlying StatefulSet and Service, while the odh-notebook-controller intercepts Notebook creation/updates via mutating and validating webhooks to inject sidecars, mount configuration, and enforce update policies. The odh-notebook-controller also manages cross-namespace resources -- HTTPRoutes in the central controller namespace, ClusterRoleBindings for auth delegation, and ReferenceGrants for Gateway API cross-namespace service references.

## Sub-Component Details

### odh-kf-notebook-controller

**Dockerfile**: `components/notebook-controller/Dockerfile.konflux`
**Intent**: The upstream Kubeflow notebook controller, adapted for RHOAI. It is the primary controller that watches Notebook CRs and creates the core infrastructure (StatefulSet with 1 replica, ClusterIP Service, optional VirtualService). It also runs an optional culling controller that polls Jupyter `/api/kernels` and `/api/terminals` endpoints to detect idle notebooks and stop them after a configurable timeout. Deployed as a separate Deployment alongside the odh-notebook-controller.
**Image**: rhoai/odh-kf-notebook-controller-rhel9

### odh-notebook-controller

**Dockerfile**: `components/odh-notebook-controller/Dockerfile.konflux`
**Intent**: The OpenShift-specific extension controller that adds enterprise features to the upstream notebook controller. It injects kube-rbac-proxy sidecars for authentication, creates Gateway API HTTPRoutes for external ingress, manages NetworkPolicies for pod isolation, handles CA certificate trust chains, and integrates with platform services (DSPA, MLflow, Feast). It also runs mutating and validating webhooks that intercept Notebook CR creation and updates. Always deployed alongside the kf-notebook-controller.
**Image**: rhoai/odh-notebook-controller-rhel9
**Language**: Go 1.26
**Port**: 8080/TCP (metrics), 8081/TCP (health), 8443/TCP (webhook)

## Architectural Analysis

The kubeflow notebook controller architecture follows a dual-controller pattern that separates upstream Kubeflow concerns from downstream OpenShift-specific extensions. The kf-notebook-controller is a relatively straightforward Kubernetes operator that maps each Notebook CR to a StatefulSet (always 1 replica, controlled via stop annotation), a ClusterIP Service, and optionally an Istio VirtualService. The odh-notebook-controller layers on top with webhook-based mutation and a parallel reconciliation loop that manages the ingress, authentication, and integration infrastructure. This separation allows RHOAI to track upstream Kubeflow changes while maintaining its own feature set.

The RHOAI 3.x ingress pattern implemented here is notably sophisticated. Each notebook gets an HTTPRoute in the controller's central namespace (not the user namespace) that references the user's Service cross-namespace via Gateway API's ReferenceGrant mechanism. This architectural choice centralizes routing in the platform namespace while allowing the Gateway to route to services in arbitrary user namespaces. The kube-rbac-proxy sidecar provides authentication enforcement at the pod level -- it performs SubjectAccessReview checks against the Kubernetes API to verify the caller has `get` permission on the specific Notebook resource, effectively delegating authorization to Kubernetes RBAC. The TLS certificates for kube-rbac-proxy are auto-generated by OpenShift's service-ca controller via the `service.beta.openshift.io/serving-cert-secret-name` annotation, which provides automated certificate rotation.

The mutating webhook is the most complex piece of the architecture. It performs eight distinct injection/transformation operations during notebook creation and updates: reconciliation lock injection (prevents race conditions during initial setup), container image resolution from ImageStreams, CA certificate bundle mounting, pipeline runtime image syncing, Elyra pipeline config secret mounting, Feast config mounting, MLflow environment variable injection, kube-rbac-proxy sidecar injection, and cluster proxy environment variable injection. The webhook also implements an update-blocking mechanism -- if a running notebook would have its pod template changed by webhook mutations (e.g., new CA certs, updated sidecar image), the webhook preserves the old template and sets an `update-pending` annotation. The Dashboard reads this annotation and prompts the user to stop the notebook before applying changes, preventing unexpected pod restarts.

The controller manages resources across three scopes: cluster-scoped ClusterRoleBindings (one per notebook for auth-delegator), central-namespace HTTPRoutes (cross-namespace ingress), and user-namespace resources (Service, ServiceAccount, ConfigMaps, Secrets, NetworkPolicies, RoleBindings). Since Kubernetes does not support cross-namespace owner references, the controller uses label-based tracking (`notebook-name`, `notebook-namespace`) plus finalizers (`notebook.opendatahub.io/httproute-cleanup`, `notebook.opendatahub.io/kube-rbac-proxy-cleanup`, `notebook.opendatahub.io/referencegrant-cleanup`) to ensure cleanup of cross-namespace and cluster-scoped resources when a notebook is deleted. The ReferenceGrant is shared across all notebooks in a namespace and is only deleted when the last notebook in that namespace is removed.

The TLS security profile integration with OpenShift's APIServer configuration is worth noting. The odh-notebook-controller reads the cluster's TLS security profile at startup and configures both the webhook server and metrics server accordingly. If the profile changes at runtime, a SecurityProfileWatcher triggers a graceful shutdown and restart to reload the new TLS configuration. On non-OpenShift clusters, the controller falls back to Mozilla Intermediate cipher suites (TLS 1.2+, 6 FIPS-approved ciphers). This is a defense-in-depth approach that ensures the controller's TLS posture matches the cluster's security requirements.
