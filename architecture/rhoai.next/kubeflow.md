## Purpose

**Short**: Two Kubernetes controllers that manage Jupyter notebook workloads on OpenShift, providing lifecycle management, Gateway API ingress, and authentication proxy injection.

**Detailed**: This repository is the Red Hat downstream fork of kubeflow/kubeflow, focused exclusively on the Notebook subsystem. It produces two deployable controller binaries that work in concert: the upstream Kubeflow Notebook Controller (kf-notebook-controller) and the ODH Notebook Controller (odh-notebook-controller).

The upstream controller handles core notebook lifecycle: it watches `Notebook` custom resources (`kubeflow.org/v1`) and reconciles them into StatefulSets with one replica, ClusterIP Services, and optionally Istio VirtualServices. It also implements idle culling by querying the Jupyter API for kernel/terminal activity and scaling notebooks to zero after configurable idle periods.

The ODH controller extends this with RHOAI-specific capabilities. Its mutating webhook injects kube-rbac-proxy sidecar containers for authentication, mounts CA certificate bundles, configures Elyra pipeline runtime secrets, and resolves container images from OpenShift ImageStreams. Its reconciler creates Gateway API HTTPRoutes (cross-namespace, centralized in the controller namespace), NetworkPolicies, ReferenceGrants, and manages integration with Data Science Pipelines (DSPA), MLflow, and Feast feature stores.

## Architectural Analysis

**Dual-Controller Architecture**: The repository implements a clear separation-of-concerns pattern between upstream Kubeflow and RHOAI-specific functionality. The upstream kf-notebook-controller handles core lifecycle (StatefulSet, Service, culling) and is kept as close to upstream as possible to minimize rebase conflicts. The odh-notebook-controller adds all platform-specific extensions (Gateway API, kube-rbac-proxy, DSPA, MLflow, Feast) without modifying upstream code. Both controllers watch the same `Notebook` CRD (`kubeflow.org/v1`) and run as separate processes in the same namespace. This design allows independent scaling and failure isolation, though it introduces a coordination challenge: the ODH controller must handle the case where the upstream controller hasn't yet created the StatefulSet or Service.

**Cross-Namespace HTTPRoute Pattern**: A distinctive architectural choice is that HTTPRoutes are created in the controller's namespace (typically `opendatahub`) rather than the user's namespace. This centralizes routing configuration and prevents users from modifying or deleting their own ingress routes, but it requires ReferenceGrant resources in each user namespace to permit cross-namespace Service references. The controller manages one ReferenceGrant per user namespace (shared across all notebooks in that namespace) and uses finalizers (not owner references) for cleanup since owner references cannot span namespaces. This pattern also means HTTPRoute names must be globally unique, which is achieved by encoding both namespace and notebook name in the route name (`nb-{ns}-{name}`), with a fallback hash-based naming for names exceeding the 63-character Kubernetes limit.

**Webhook-Driven Sidecar Injection**: The mutating admission webhook serves as the primary extension point for the ODH controller. Rather than modifying notebook pods after creation, the webhook intercepts CREATE/UPDATE operations on Notebook CRs and injects the kube-rbac-proxy sidecar container, CA certificate volumes, Elyra secret mounts, Feast configuration, and MLflow environment variables directly into the PodSpec template. This is architecturally significant because it means the controller doesn't need to patch running StatefulSets — mutations happen before the upstream controller ever sees the resource. The webhook also includes a reconciliation lock mechanism (annotation `notebooks.opendatahub.io/odh-notebook-controller-lock`) that temporarily pauses the upstream culling controller during notebook creation, preventing a race condition where the culler could set a stop annotation before the ODH controller finishes its reconciliation.

**Idle Culling via Jupyter API Polling**: The culling controller implements an unconventional activity detection pattern by directly querying the Jupyter notebook API endpoints (`/api/kernels` and `/api/terminals`) via HTTP within the cluster network. This avoids the need for a dedicated metrics or activity agent inside the notebook pod. Activity timestamps from kernel/terminal responses are compared against annotation-stored values with backward-clock protection (timestamps are only updated forward). The culling mechanism uses annotations (`kubeflow-resource-stopped`) rather than direct StatefulSet scaling, allowing the upstream controller's normal reconciliation to handle the actual scale-to-zero operation. This decouples the culling decision from the scaling implementation.

**RHOAI 3.x Migration Artifacts**: The codebase contains legacy OAuth proxy code (notebook_oauth.go) that only runs as a cleanup finalizer for RHOAI 2.x to 3.0 migration. The current RHOAI 3.x pattern uses kube-rbac-proxy (SubjectAccessReview-based) instead of oauth-proxy (OAuth token-based). The OAuth cleanup code deletes orphaned OAuthClient resources when notebooks are deleted, ensuring a clean migration path. The Route API (route.openshift.io) is still watched (read-only) for fallback hostname resolution but no Routes are actively created — all new ingress uses Gateway API HTTPRoutes.
