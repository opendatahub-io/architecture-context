## Purpose

**Short**: Kubeflow Trainer V2 is a Kubernetes operator that orchestrates distributed machine learning training jobs by reconciling TrainJob custom resources against pluggable TrainingRuntime and ClusterTrainingRuntime templates.

**Detailed**: The Kubeflow Trainer (formerly Kubeflow Training Operator) manages the lifecycle of distributed ML training workloads on Kubernetes. Users define training jobs via the `TrainJob` custom resource, which references a `TrainingRuntime` (namespace-scoped) or `ClusterTrainingRuntime` (cluster-scoped) that specifies the training framework (PyTorch, DeepSpeed, MPI, PlainML), container images, and ML-specific policies (number of nodes, processes per node, elastic training).

The controller reconciles TrainJob resources by resolving the referenced runtime template, applying ML policy enforcement through a plugin-based framework, and creating downstream Kubernetes resources — primarily `JobSet` objects (from the `jobset.x-k8s.io` API) with optional `PodGroup` resources for gang-scheduling via Kueue/scheduler-plugins or Volcano. The operator also manages dataset and model initializer containers, SSH key generation for MPI-based training, and webhook-based admission validation.

For RHOAI, the operator includes Red Hat-specific enhancements: progression tracking (polling training pod metrics endpoints during training for real-time progress reporting), NetworkPolicy creation for pod-level security isolation of metrics endpoints, and pre-configured ClusterTrainingRuntimes with Red Hat-built training images for CUDA, ROCm, and CPU workloads. OpenShift ImageStreams provide versioned training hub images for workbench integration.

## Architectural Analysis

The Kubeflow Trainer V2 represents a significant architectural evolution from the original Training Operator (V1). The V1 operator had separate controllers for each framework (TFJob, PyTorchJob, MPIJob, etc.) with distinct CRDs and reconciliation loops. V2 consolidates these into a single `TrainJob` CRD that delegates framework-specific behavior to a runtime template system (`TrainingRuntime`/`ClusterTrainingRuntime`), dramatically reducing CRD proliferation and simplifying the user-facing API.

The plugin-based runtime framework is the architectural centerpiece. The `Framework` struct in `pkg/runtime/framework/core/framework.go` orchestrates a pipeline of typed plugins: `EnforceMLPolicyPlugin` (Torch, MPI, PlainML) handles framework-specific distributed training configuration, `EnforcePodGroupPolicyPlugin` (CoScheduling, Volcano) manages gang-scheduling, `ComponentBuilderPlugin` (JobSet, MPI, CoScheduling, Volcano) builds Kubernetes resources, and `CustomValidationPlugin` provides webhook validation. This design allows new training frameworks to be added by implementing the appropriate interfaces without modifying the core controller logic. The plugin registry in `pkg/runtime/framework/plugins/registry.go` maps plugin names to constructors, enabling runtime discovery.

The RHOAI-specific enhancements are cleanly layered via the `pkg/rhai/` package and kustomize overlays. The progression tracking feature (enabled via `ENABLE_RHAI_FEATURES=true`) polls training pods' HTTP metrics endpoints on port 2113 during reconciliation to report real-time training progress in the TrainJob status. This is secured by automatically creating NetworkPolicies per TrainJob that restrict ingress to the metrics port to only controller manager pods, preventing unauthorized access to training metrics. The progression tracking uses the API reader (direct API server reads) rather than the cached client to avoid the need for pod watches, which is a deliberate design choice to minimize RBAC scope and cache memory.

The RHOAI overlay also introduces a rich set of pre-configured ClusterTrainingRuntimes covering multiple hardware targets: CUDA (NVIDIA GPU), ROCm (AMD GPU), and CPU-only variants. These are parameterized via `params.env` with SHA256-pinned images from `registry.redhat.io`, ensuring reproducible and auditable deployments. The runtime matrix includes both "floating" runtimes (e.g., `torch-distributed` which tracks the latest recommended image) and version-pinned runtimes (e.g., `torch-distributed-cuda128-torch29-py312`). OpenShift ImageStreams provide versioned training hub images that integrate with the RHOAI workbench UI, allowing users to select training images through the dashboard.

The operator's certificate management is noteworthy: it uses the `open-policy-agent/cert-controller` library to manage webhook TLS certificates internally, generating self-signed certificates and storing them in a Kubernetes Secret. This avoids an external dependency on cert-manager while still providing automatic certificate rotation. The readiness probe blocks until certificates are ready, preventing the operator from advertising availability before webhooks can serve traffic.
