## Purpose

**Short**: Kubernetes operator that manages distributed machine learning training jobs using a plugin-based runtime framework supporting PyTorch, MPI, and TorchTune.

**Detailed**: Kubeflow Trainer (formerly Training Operator) is a Kubernetes operator built on controller-runtime that orchestrates distributed ML training workloads. It introduces a declarative API through three CRDs: TrainJob (the user-facing workload), TrainingRuntime (namespace-scoped runtime templates), and ClusterTrainingRuntime (cluster-scoped runtime templates). Users create a TrainJob that references a runtime, and the operator's plugin framework composes the appropriate Kubernetes resources (JobSets, PodGroups, Secrets, ConfigMaps) based on the ML framework (PyTorch/MPI/TorchTune) and scheduling policy (Volcano/CoScheduling).

The RHOAI distribution adds several enhancements: progression tracking (RHAI) that polls training pods for real-time metrics and stores them as TrainJob annotations, NetworkPolicy creation for pod isolation and secure metrics access, pre-built ClusterTrainingRuntime definitions for CUDA/ROCm/CPU configurations across PyTorch and OpenMPI, and OpenShift ImageStream resources for Training Hub workbench images. The operator deploys as a single controller manager pod with a webhook server for admission validation of all three CRDs.

## Architectural Analysis

The Kubeflow Trainer architecture is built around a sophisticated plugin framework that cleanly separates ML framework concerns from scheduling and infrastructure concerns. The `Runtime` interface provides a well-defined contract: `NewObjects()` creates Kubernetes resources, `RuntimeInfo()` builds a metadata pipeline, and the framework orchestrates plugin execution through typed chains (EnforceMLPolicy, EnforcePodGroupPolicy, PodNetwork, ComponentBuilder). This design enables adding new ML frameworks (e.g., JAX, TensorFlow) as plugins without modifying the controller core. The `Info` object serves as a data pipeline through the plugin chain -- created once, mutated by each plugin in sequence, then consumed by ComponentBuilder plugins to generate concrete Kubernetes resources.

The TrainJob-to-Runtime reference pattern provides a clean separation between "what to train" (TrainJob) and "how to train" (TrainingRuntime/ClusterTrainingRuntime). The multi-level override system (RuntimeRef, Initializer, Trainer, PodTemplateOverrides) allows users to progressively customize without duplicating runtime templates. The immutability of `RuntimeRef` and `managedBy` fields prevents dangerous mid-flight changes. Finalizer-based protection on runtimes prevents deletion of in-use templates, with controller-to-controller notification via typed channels (capacity 10, 1-second dequeue delay) ensuring timely finalizer lifecycle management.

The RHOAI-specific RHAI progression tracking is architecturally notable: rather than watching pods (which would consume controller memory with informer caches), it uses `APIReader` for direct API calls and HTTP polling to the training pod's metrics endpoint. NetworkPolicy resources are created per-TrainJob to restrict metrics endpoint access to only the controller pod, providing security without service mesh overhead. The progression system supports both live HTTP polling and post-termination message capture, with configurable poll intervals (5s-300s, default 30s) and graceful degradation when pods become unavailable.

The MPI plugin's SSH key management follows an immutable-secret pattern: fresh ECDSA P521 key pairs are generated per TrainJob and stored as immutable Kubernetes Secrets. This prevents accidental key rotation that could break running training jobs. Similarly, the JobSet plugin avoids updating JobSets that are already running and not suspended, preventing accidental disruption of active training. The resource ownership model (all resources owned by TrainJob via OwnerReferences) ensures clean garbage collection when TrainJobs are deleted.

The RHOAI kustomize overlay provides 16 ClusterTrainingRuntime variants covering CUDA 12.8/13.0, ROCm 6.4, and CPU configurations across PyTorch 2.9/2.9.1 versions, plus OpenMPI CUDA. Image references are parameterized through a ConfigMap generator (`rhoai-config`) with kustomize replacements, enabling centralized image management across all runtimes. The overlay also adds OpenShift ImageStreams for Training Hub workbench images, integrating with the OpenShift developer console for image selection.
