## Purpose

**Short**: Kubernetes-native operator for managing distributed machine learning training jobs across PyTorch, MPI/OpenMPI, TorchTune, and other frameworks.

**Detailed**: Kubeflow Trainer (v2) is a controller-runtime based Kubernetes operator that orchestrates distributed ML training workloads on OpenShift. It provides a declarative API through three CRDs: `TrainJob` (user-facing training job specification), `TrainingRuntime` (namespace-scoped runtime templates), and `ClusterTrainingRuntime` (cluster-scoped runtime templates). The operator reconciles TrainJob resources by resolving them against their referenced training runtimes, building JobSet resources for the underlying workload orchestration, and managing the lifecycle of distributed training across multiple nodes.

In the RHOAI distribution, the operator includes Red Hat AI (RHAI) extensions: progression tracking (real-time training metrics polling via HTTP from training pods), NetworkPolicy-based pod isolation for training jobs, and pre-configured ClusterTrainingRuntime resources for PyTorch distributed (CPU, CUDA, ROCm), OpenMPI with CUDA, and Training Hub runtimes across multiple Python/PyTorch/CUDA version combinations. The operator integrates with JobSet for workload orchestration, supports gang-scheduling via Volcano and Kubernetes scheduler-plugins (Coscheduling), and provides validation webhooks for all three CRDs.

## Architectural Analysis

The Kubeflow Trainer v2 architecture represents a significant redesign from the v1 Training Operator. Instead of having separate controllers per framework (PyTorchJob, MPIJob, TFJob), v2 unifies everything under a single `TrainJob` CRD with a pluggable runtime framework. The `TrainingRuntime` / `ClusterTrainingRuntime` CRDs provide the template system, and the `RuntimeRef` in TrainJob selects which runtime to use. This design separates the concern of "what to train" (TrainJob) from "how to train" (TrainingRuntime), enabling platform administrators to provide pre-configured runtimes while users focus on their training code.

The runtime framework (`pkg/runtime/framework/`) follows a plugin-based architecture inspired by the Kubernetes scheduler framework. Plugins implement specific interfaces (`EnforceMLPolicyPlugin`, `EnforcePodGroupPolicyPlugin`, `PodNetworkPlugin`, `ComponentBuilderPlugin`, `TrainJobStatusPlugin`, `CustomValidationPlugin`, `WatchExtensionPlugin`) and are composed into a processing pipeline during reconciliation. This design makes it straightforward to add new training frameworks or scheduling integrations without modifying the core reconciliation logic.

The RHAI (Red Hat AI) extensions demonstrate a downstream customization pattern that adds capabilities without forking the upstream codebase. The progression tracking feature (`pkg/rhai/`) adds real-time training metrics polling by directly HTTP-calling training pods' metrics endpoints. This is architecturally interesting because it creates an unusual data flow where the controller makes outbound HTTP calls to workload pods, secured by a dynamically-created NetworkPolicy that restricts the metrics port to only the controller's pods (identified by namespace and labels). The progression status is stored in TrainJob annotations, providing a lightweight real-time status mechanism without requiring a separate database or status API.

The certificate management approach uses the OPA cert-controller library for automatic webhook certificate rotation, avoiding a hard dependency on cert-manager. The controller generates a self-signed CA and server certificates, stores them in a Kubernetes Secret, and automatically rotates them. This is reflected in the readiness probe design: the controller reports as not-ready until certificates are generated, ensuring no webhook calls are rejected during startup.

A notable security consideration is the plaintext HTTP communication between the controller and training pod metrics endpoints (port 28080). While this is mitigated by the NetworkPolicy that restricts access to the controller's namespace, it means metrics data traverses the pod network unencrypted. For environments with strict security requirements, this could be a gap worth addressing with mTLS or at minimum TLS for the metrics endpoint.
