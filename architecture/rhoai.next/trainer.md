## Purpose

**Short**: Kubeflow Trainer is a Kubernetes operator that orchestrates distributed machine learning training workloads by managing TrainJob, TrainingRuntime, and ClusterTrainingRuntime custom resources.

**Detailed**: Kubeflow Trainer provides a declarative API for running distributed training jobs on Kubernetes. Users define training runtimes (ClusterTrainingRuntime for cluster-wide or TrainingRuntime for namespace-scoped) that specify ML framework configuration (PyTorch, MPI), scheduling policies (CoScheduling, Volcano), and container templates. Users then create TrainJob resources that reference a runtime and optionally override trainer settings like image, node count, and resource requirements.

The operator's controller reconciles TrainJob resources by resolving the referenced runtime, executing a plugin framework that enforces ML policies (distributed environment variables, process counts), builds gang-scheduling PodGroups, and generates JobSet resources that orchestrate the actual training pods. The plugin architecture supports PyTorch (torchrun and torchtune), OpenMPI, and a generic PlainML fallback. RHOAI adds progression tracking (real-time training metrics polling) and NetworkPolicy enforcement for pod isolation.

The component also includes a Rust-based data cache system (head-worker architecture using Apache Arrow Flight) for distributed dataset caching, and Go-based initializer containers for dataset and model preparation. RHOAI ships 16 pre-configured ClusterTrainingRuntimes covering CPU, CUDA, and ROCm accelerator variants with multiple PyTorch/Python version combinations, plus 3 ImageStreams for universal training workbench images.

### data-cache

**Dockerfile**: `cmd/data_cache/Dockerfile`
**Intent**: Optional distributed data caching system for training workloads. Deployed as a head-worker topology using LeaderWorkerSet (LWS) when data caching is enabled on a ClusterTrainingRuntime. The head node coordinates data file distribution across workers using Apache Arrow Flight; workers serve cached data to training pods. Not deployed by default.
**Image**: Built from Rust source in `pkg/data_cache/`
**Language**: Rust 1.85

### model-initializer

**Intent**: Init container that downloads and prepares pre-trained model weights before the main training container starts. Deployed as part of a TrainJob's JobSet when `spec.initializer.model` is configured. Runs as a one-shot init container and exits.

### torchtune trainer

**Intent**: Utility image providing TorchTune CLI for LLM fine-tuning. Used as the trainer container image in TorchTune-based ClusterTrainingRuntimes. Supports recipe selection (full finetune, LoRA, QLoRA) across single-device, multi-device, and multi-node configurations for Llama 3.2 (1B, 3B) and Qwen 2.5 (1.5B) models.
**Language**: Go (wrapper), Python (TorchTune)

## Architectural Analysis

The Kubeflow Trainer operator follows a sophisticated plugin-based architecture that cleanly separates ML framework concerns from Kubernetes orchestration. The core design centers on three custom resources forming a template-instance pattern: ClusterTrainingRuntime/TrainingRuntime act as templates defining the training infrastructure recipe, while TrainJob instances reference these templates and optionally override specific parameters (image, node count, resources). This separation enables platform administrators to define approved training configurations while giving data scientists a simplified API surface.

The plugin framework is the architectural highlight, implementing seven distinct plugin interfaces (CustomValidation, EnforceMLPolicy, EnforcePodGroupPolicy, PodNetwork, ComponentBuilder, TrainJobStatus, WatchExtension) that execute in a defined sequence during reconciliation. This design allows the operator to support PyTorch distributed training (torchrun/torchtune), OpenMPI, CoScheduling, and Volcano scheduling without any of these concerns leaking into the core controller logic. The JobSet plugin acts as the final component builder, translating the accumulated configuration into a concrete JobSet resource. This plugin chain is extensible — adding a new ML framework requires implementing only the relevant plugin interfaces.

RHOAI extends the upstream operator with two significant features behind the `ENABLE_RHAI_FEATURES` flag. First, progression tracking enables real-time monitoring of training jobs by polling an HTTP metrics endpoint (port 28080) on the rank-0 training pod during reconciliation. The controller reads training progress, step counts, epoch progress, and loss metrics, storing them as JSON annotations on the TrainJob resource. This design avoids requiring a separate metrics infrastructure by piggybacking on the controller's reconciliation loop. Second, NetworkPolicy enforcement creates per-TrainJob network isolation: training pods can communicate with each other (for NCCL/MPI/gRPC), and the controller pod can access the metrics endpoint, but no other traffic is permitted. The controller identifies itself via label selectors, and the NetworkPolicy uses OwnerReference for automatic cleanup when the TrainJob is deleted.

The data cache subsystem is architecturally distinct — it's a Rust-based distributed caching layer using Apache Arrow Flight protocol with Iceberg table metadata integration. The head-worker topology (managed by LeaderWorkerSet) uses greedy load balancing to partition data files across workers, with each worker adding a `cache_index` column for global row ordering. This design enables training pods to request specific data partitions without reading the full dataset, reducing I/O bottleneck in multi-node training. The data cache is optional and deployed only when enabled on a ClusterTrainingRuntime.

A notable design decision is the finalizer pattern on TrainingRuntime and ClusterTrainingRuntime resources — the controller adds a `ResourceInUse` finalizer when any TrainJob references the runtime and removes it only when all referencing TrainJobs are deleted. This prevents accidental deletion of runtimes that are actively in use. The cross-resource notification happens through channel-based event sources with 1-second delays, balancing responsiveness with API server load.

The RHOAI distribution ships 16 ClusterTrainingRuntimes covering a matrix of accelerator (CPU, CUDA 12.8, CUDA 13.0, ROCm 6.4) and framework version (PyTorch 2.9, 2.10, 2.9.1) combinations, plus 3 "training-hub" variants that include additional libraries (transformers, accelerate, peft, trl, deepspeed). All training images in `params.env` are SHA256-pinned for reproducibility. The kustomize overlay uses `replacements` (not deprecated `vars`) for image substitution, mapping ConfigMap entries to ClusterTrainingRuntime container image fields.
