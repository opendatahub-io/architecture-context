## Purpose

**Short**: Llama Stack is an extensible AI/ML platform server that provides a unified API for inference, agents, safety, vector I/O, evaluation, and related AI capabilities through a pluggable provider architecture.

**Detailed**: Llama Stack (ogx) is Red Hat's downstream fork of Meta's Llama Stack project. It defines a comprehensive set of protocol-based APIs for AI/ML operations — including LLM inference, agentic workflows, content safety, vector database operations, model evaluation, dataset management, telemetry, and tool execution — served through a FastAPI HTTP server on port 8321/TCP.

The architecture is built around a provider abstraction layer: each API can be backed by one or more "inline" (in-process) or "remote" (external service adapter) providers. For example, the Inference API can route to vLLM, Ollama, OpenAI, Anthropic, or 25+ other backends through a unified interface. Providers are composed into "distributions" — pre-configured stacks defined by YAML templates (e.g., `starter`, `remote-vllm`, `meta-reference-gpu`). The server supports both HTTP client mode (remote requests) and library client mode (direct Python API, no network).

In the RHOAI context, ogx provides the Llama Stack server component that can be deployed as a containerized service, offering multi-model inference routing, safety guardrails (Llama Guard, Prompt Guard), RAG capabilities via vector stores, agentic orchestration, and OpenAI-compatible endpoints alongside native Llama Stack APIs.

### Authentication & Authorization

| Endpoint | Methods | Auth Mechanism | Enforcement Point | Policy |
|----------|---------|----------------|-------------------|--------|
| All API endpoints (`/v1/*`) | ALL | Bearer token (OAuth2 JWT or Custom) | AuthenticationMiddleware (ASGI) | Configurable via `server.auth` in run config |
| All API endpoints (`/v1/*`) | ALL | Attribute-based access control (ABAC) | AccessControl module | Cedar-like permit/forbid rules on principal, actions (CRUD), resources (type::id) with conditions |
| `/health`, `/version`, `/docs` | GET | None | N/A | Public endpoints, bypass auth |

**Authentication Flow**:
1. `AuthenticationMiddleware` extracts Bearer token from `Authorization` header
2. Token validated via configured provider (OAuth2 JWT with JWKS/introspection, or custom endpoint)
3. User principal and attributes (roles, teams, projects, namespaces) extracted from claims
4. Attributes stored in request scope for downstream access control
5. `QuotaMiddleware` (optional) enforces per-client rate limits (separate limits for authenticated vs anonymous)
6. Route handler applies ABAC rules: permit/forbid based on principal, action, resource, and attribute conditions

**Access Control Model (Cedar-like)**:
- Rules: `permit` or `forbid` scopes with optional `when`/`unless` conditions
- Principals: user identifiers matched against `sub`/`username` JWT claims
- Actions: `create`, `read`, `update`, `delete`
- Resources: typed identifiers (`model::my-model`, `vector_db::*`)
- Conditions: `user with <value> in <attr>`, `user is owner`, `user in owners <attr>`
- Default policy: owners have full access (backward compatibility)

## Architectural Analysis

**Provider-based Extensibility**: The most distinctive architectural pattern in Llama Stack is the provider abstraction. Every API is defined as a Python `Protocol` with `@runtime_checkable` and `@webmethod` decorators. Each webmethod maps to an HTTP route. Providers implement these protocols and are resolved via dependency injection with topological sorting — providers declare their API dependencies, and the resolver instantiates them in the correct order. This means adding a new inference backend requires only implementing the protocol and registering it, with zero changes to server code. The 60 existing providers (19 inline, 41 remote) demonstrate this pattern's effectiveness.

**Distribution Composition System**: Rather than having a monolithic configuration, Llama Stack uses a template-based distribution system. Each distribution template (e.g., `starter`, `remote-vllm`) defines which providers back which APIs, what models to register, and what environment variables to expose. The `DistributionTemplate` class generates both `build.yaml` (for container image building) and `run.yaml` (runtime configuration) from a single Python definition. Environment variable substitution (`${env.VAR:default}`) makes the same template usable across development, CI, and production environments. This is architecturally elegant but means the actual deployment topology is configuration-driven, not manifest-driven — the same container image serves radically different purposes depending on the run config.

**Security Architecture**: Authentication is cleanly layered as ASGI middleware. The OAuth2 provider supports both JWKS (asymmetric key validation) and token introspection (RFC 7662), with configurable claim-to-attribute mapping. The access control system implements a Cedar-like policy language (permit/forbid rules with conditions), which is notably more expressive than simple RBAC. However, authentication is entirely optional — there is no default-on security posture. This is appropriate for development but requires explicit configuration for production deployments. Rate limiting via `QuotaMiddleware` is also optional, with separate limits for authenticated and anonymous clients.

**Streaming and Async Design**: The server makes heavy use of Python's async capabilities. Streaming responses use Server-Sent Events (SSE) via FastAPI's `StreamingResponse`, with careful context variable preservation across async generators (see `preserve_contexts_async_generator()`). The library client mode uses `ThreadPoolExecutor` to bridge sync-to-async for notebook usage. W3C trace context (`traceparent`, `tracestate`) is extracted from headers and propagated through the request lifecycle, enabling distributed tracing across Llama Stack and its backend providers.

**RHOAI Deployment Considerations**: This repository contains no Konflux Dockerfiles, no kustomize manifests, and no Kubernetes-specific deployment resources. It is a pure Python service that can run as a standalone process or inside a container. For RHOAI integration, the container image build and deployment configuration must be provided externally — likely via a separate deployment repository or operator configuration. The `remote-vllm` and `vllm-gpu` templates are the most RHOAI-relevant distributions, as vLLM is the primary inference serving technology in the platform. The lack of build-time FIPS configuration in this repository means FIPS compliance depends entirely on the container base image and build pipeline applied downstream.
