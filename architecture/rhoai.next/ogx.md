## Purpose

**Short**: A pluggable AI/LLM application framework that provides a unified REST API for inference, agents, safety, evaluation, and tool orchestration across multiple model providers.

**Detailed**: ogx is a downstream fork of Meta's [llama-stack](https://github.com/meta-llama/llama-stack), providing a comprehensive platform for building and deploying AI applications. It exposes a unified HTTP API (76 endpoints across 19 API modules) that abstracts over dozens of backend providers — including vLLM, Ollama, OpenAI, Anthropic, Bedrock, TGI, and many others — through a pluggable provider architecture.

The framework supports two provider modes: **inline providers** that run within the Llama Stack process (e.g., meta-reference inference, SQLite vector store, local file storage) and **remote providers** that proxy to external services (e.g., vLLM inference endpoints, Chroma vector DBs, HuggingFace datasets). Providers are composed into "distributions" — pre-configured stacks defined by YAML templates (24 built-in templates) that wire together providers for specific deployment scenarios.

The server includes enterprise-grade features: OAuth2/JWT and custom authentication, Cedar-like attribute-based access control (ABAC), per-client rate limiting with configurable quotas, TLS/mTLS support, OpenTelemetry-based distributed tracing, and SSE streaming for real-time inference responses. It also provides OpenAI-compatible API endpoints (`/openai/v1/chat/completions`, `/openai/v1/embeddings`, `/openai/v1/responses`) enabling drop-in replacement for OpenAI API consumers.

## Architectural Analysis

ogx (Llama Stack) is a **provider-aggregation framework** — its core value proposition is abstracting over a wide variety of AI infrastructure backends through a single API surface. The architecture follows a clear layered pattern: API protocol definitions → routers → routing tables → provider implementations. The `@webmethod` decorator-driven API protocol design means API surface is defined via Python Protocol classes in `llama_stack/apis/`, which are dynamically introspected at server startup to register FastAPI routes. This is powerful but unusual — no explicit route decorators exist in the server; instead, `create_dynamic_typed_route()` in `server.py` generates FastAPI handlers by reflecting on protocol method signatures.

The **provider resolution pipeline** (`resolver.py`) is architecturally significant: it performs topological sorting on provider dependency graphs, validates protocol compliance via reflection, and supports both compile-time (build config) and runtime (run config) provider composition. The auto-routing system (7 paired APIs — e.g., `models`↔`inference`, `shields`↔`safety`) means that registering a model automatically enables routing inference requests to the correct provider. This is elegant but creates implicit coupling between resource registration and request routing that may be non-obvious to operators.

The **authentication stack** is modular and well-designed: `AuthenticationMiddleware` (ASGI middleware) validates Bearer tokens via pluggable providers (`OAuth2TokenAuthProvider` with JWKS or introspection, `CustomAuthProvider` with external endpoint). Token claims are mapped to user attributes (`roles`, `teams`, `projects`, `namespaces`) via configurable `claims_mapping`, which then feed into the Cedar-like ABAC policy engine (`access_control/`). The policy language supports `permit`/`forbid` scopes with `when`/`unless` conditions, owner-based rules, and wildcard resource matching. Rate limiting (`QuotaMiddleware`) operates above auth with separate anonymous and authenticated quotas.

A notable **gap in the current repository** is the absence of Konflux Dockerfiles. As a downstream RHOAI component, production container images are expected to be built via Konflux with UBI base images, FIPS-compliant crypto, and hermetic builds — but none of these exist on the `main` branch. The existing `Containerfile` files are for the Streamlit UI demo (`python:3.12-slim` base, port 8501) and CI test infrastructure (`ollama/ollama` base). This means FIPS compliance, RPM lock files, and Hermeto prefetch integration would need to be added on downstream release branches. The `python-jose` library's use of the pure-Python `ecdsa` package for JWT verification is a FIPS risk worth tracking — in a FIPS-enforced environment, JWT verification should use OpenSSL-backed crypto.

The **template system** (24 built-in templates in `llama_stack/templates/`) provides pre-configured provider stacks for various deployment scenarios. The `starter` template is particularly notable — it pre-configures 100+ models across 8 inference providers (OpenAI, Fireworks, Together, Ollama, Anthropic, Gemini, Groq, SambaNova) plus multiple vector DB, safety, and tool providers. Environment variable substitution (`${env.VAR:default}`) in YAML configs enables configuration injection without template modification. The `llama` CLI provides `build` and `run` subcommands that can create container images, conda environments, or virtual environments from these templates.
