## Purpose

**Short**: llm-d Planner is an LLM deployment planning service that guides users from business requirements to production-ready GPU-backed inference deployments through conversational intent extraction, SLO-driven capacity planning, and Kubernetes manifest generation.

**Detailed**: llm-d Planner unifies the fragmented landscape of model selection, GPU sizing, performance estimation, and Kubernetes configuration into a single platform. It provides three core capabilities: (1) **Planner** — the main workflow that guides users from natural-language business requirements through SLO-driven recommendations to one-click Kubernetes deployment; (2) **Capacity Planner** — GPU memory estimation for any HuggingFace model including model weights, KV cache, activation memory, and system overhead; and (3) **GPU Recommender** — inference performance estimation (TTFT, ITL, throughput) across GPU types without running actual benchmarks, powered by BentoML's roofline model.

The system operates as a multi-tier architecture with a FastAPI backend serving a REST API, a Streamlit frontend providing a conversational UI, PostgreSQL for benchmark data storage, and Ollama for LLM-powered intent extraction. It integrates with the RHOAI Model Catalog for model metadata and benchmarks when deployed on OpenShift, and can deploy generated KServe/vLLM configurations directly to Kubernetes clusters. A vLLM simulator enables GPU-free development and testing.

## Architectural Analysis

### Layered Architecture with Clear Separation of Concerns

The llm-d Planner follows a well-structured layered architecture. The API layer (`src/planner/api/routes/`) strictly delegates to service modules, using FastAPI dependency injection to provide singleton service instances. The `RecommendationWorkflow` orchestrator in `src/planner/orchestration/workflow.py` coordinates the multi-step recommendation pipeline (intent extraction → specification → capacity planning → scoring → ranking) while keeping each engine independently testable. This clean separation means the CLI (`src/planner/cli/planner_cli.py`) can access the same capacity planning and GPU estimation logic without the API layer, and the service modules have no awareness of their HTTP transport.

### Dual Data Source Strategy

A notable architectural decision is the dual data source strategy controlled by `PLANNER_BENCHMARK_SOURCE`. In standalone/development mode (`postgresql`), the system uses a local PostgreSQL database seeded with BLIS benchmark data and estimated performance data. In RHOAI platform mode (`model_catalog`), a background sync thread (`model_catalog_sync.py`) periodically pulls models and artifacts from the RHOAI Model Catalog API, translates them into the internal benchmark schema, and upserts them into PostgreSQL. This means PostgreSQL remains the query layer regardless of data source — the Model Catalog client acts as an ETL pipeline, not a direct query backend. The `ModelCatalogClient` implements retry logic with exponential backoff, response caching (3600s TTL), and graceful fallback to ServiceAccount token authentication when no explicit bearer token is configured.

### Security Gaps in Current State

The most significant security observation is the complete absence of authentication on all API endpoints. The backend exposes 27 REST endpoints including destructive operations (`POST /api/v1/db/reset`, `DELETE /api/v1/deployments/{id}`, `POST /api/v1/deploy-to-cluster`) with no auth mechanism. The OpenShift Routes provide TLS edge termination but no authentication enforcement. CORS is configured to allow all origins (`allow_origins=["*"]`). For an incubation project this is expected, but the `README.md` acknowledges "Security Hardening — YAML validation, RBAC, network policies" as future work. The `KubernetesClusterManager` executes `kubectl` as subprocess calls (`subprocess.run(["kubectl", ...])`) using whatever ServiceAccount the pod runs as, meaning anyone who can reach the backend API can deploy arbitrary InferenceServices to the cluster.

### Incubation Status and Build Infrastructure

This component is clearly in early incubation stage — there are no Konflux Dockerfiles, no lock files (not even `uv.lock` despite the Dockerfile using `uv sync --frozen`), no kustomize overlays, and the base images use upstream Debian (`python:3.14-slim`) rather than UBI. The PostgreSQL image in the Kubernetes manifests uses `registry.redhat.io/rhel9/postgresql-16:latest`, suggesting partial RHOAI alignment. For production RHOAI deployment, the component would need: UBI-based images, Konflux onboarding, hermetic builds with `uv.lock` + `rpms.lock.yaml`, and proper authentication/authorization on API endpoints.

### Performance Estimation Fallback Pattern

The recommendation engine implements an elegant fallback pattern: when the PostgreSQL benchmark database lacks data for a (model, GPU, traffic profile) combination, the system generates synthetic benchmarks using BentoML's roofline model via the `GPURecommender`. These estimated results are visually differentiated in the UI as "Estimated" vs "Benchmarked" and are persisted back to PostgreSQL with `confidence_level='estimated'`. This means the benchmark database grows over time as users explore configurations not covered by original benchmark runs. The estimation is bounded by `PLANNER_ESTIMATED_MAX_MODELS` (default 5) and `PLANNER_ESTIMATED_TIMEOUT_S` (default 60) to prevent long-running recommendation requests.
