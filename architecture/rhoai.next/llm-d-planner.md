## Purpose

**Short**: llm-d Planner is an AI-powered capacity planning and deployment automation service that guides users from business requirements to production-ready LLM deployments on Kubernetes.

**Detailed**: llm-d Planner unifies model selection, GPU sizing, performance estimation, and Kubernetes configuration generation into a single conversational platform. It addresses the complexity of deploying LLMs in production by translating high-level business requirements (e.g., "I need a chatbot for 1000 users with low latency") into concrete infrastructure specifications including model selection, GPU type and count, SLO targets, and deployment topology.

The system operates through a 4-stage conversational flow: (1) extract deployment intent from natural language using LLM-powered analysis, (2) generate traffic profiles and SLO targets based on use-case templates and GuideLLM benchmark configurations, (3) find optimal model+GPU configurations through multi-criteria scoring (accuracy, price, latency, complexity) against a benchmark database, and (4) generate validated KServe/vLLM YAML and deploy to Kubernetes clusters. The platform includes three capabilities: the main Planner workflow for end-to-end deployment planning, a Capacity Planner for GPU memory estimation, and a GPU Recommender for cross-GPU performance estimation via roofline modeling.

llm-d Planner integrates with the RHOAI ecosystem through the Model Catalog API for model and benchmark data synchronization, and supports pluggable LLM providers (Ollama, Vertex AI/Claude, OpenAI-compatible APIs) for intent extraction. It deploys as a multi-service stack on OpenShift with a FastAPI backend, Streamlit UI frontend, PostgreSQL benchmark database, and optional Ollama LLM service.

## Architectural Analysis

The llm-d Planner exhibits a well-structured layered architecture with clear separation of concerns. The FastAPI backend is organized into distinct functional modules: intent extraction (LLM-powered), specification generation (template-driven), recommendation (multi-criteria scoring), configuration generation (Jinja2 templates), and cluster management (kubectl subprocess). All major services are initialized as singletons during application startup and injected via FastAPI's dependency injection system, which provides clean testability. The use of a Protocol-based LLM client interface (structural typing, not inheritance) enables swapping between Ollama, Vertex AI, and OpenAI providers without modifying business logic.

The recommendation engine represents the core innovation: it queries a PostgreSQL benchmark database using exact-match on (model, GPU, traffic_profile) tuples, filters by SLO compliance with configurable near-miss tolerance, then scores candidates across four dimensions (accuracy from Artificial Analysis benchmarks, price from GPU catalog, latency headroom, and deployment complexity). When benchmarks are missing, a roofline estimation fallback via BentoML's llm-optimizer fills gaps, with results persisted back to PostgreSQL and visually differentiated as "Estimated" in the UI. The 5-ranked-list output (best_accuracy, lowest_cost, lowest_latency, simplest, balanced) gives users multiple optimization perspectives.

From a deployment perspective, Planner runs as a multi-service stack in a dedicated `planner` namespace. The deploy-all.sh script uses sed-based manifest templating (acknowledged as a stopgap before Kustomize migration) and supports conditional deployment: Ollama is only deployed when `LLM_PROVIDER=ollama`, the Model Catalog NetworkPolicy is only applied when `PLANNER_BENCHMARK_SOURCE=model_catalog`, and GPU resource requests on Ollama can be toggled via `PLANNER_ENABLE_GPU`. The cross-namespace NetworkPolicy for Model Catalog access (`rhoai-model-registries` namespace) demonstrates awareness of RHOAI's network isolation requirements, though the backend's own namespace currently lacks a NetworkPolicy restricting ingress.

A notable security gap is the absence of authentication on most API endpoints — the backend relies entirely on network-level isolation (namespace, OpenShift Router). Only the database admin endpoints (`/db/upload-benchmarks`, `/db/reset`) have optional HMAC password protection. The CORS middleware is configured with `allow_origins=["*"]` which is appropriate for development but should be restricted in production. The Kubernetes cluster interaction uses kubectl subprocess calls rather than the Python kubernetes SDK (except for GPU detection), which means error handling depends on parsing command output rather than structured exceptions. The Vertex AI credential handling (writing GCP_CREDENTIALS_JSON to a temp file with atexit cleanup) is pragmatic but creates a brief window where credentials exist on the filesystem.

The repository currently lacks Konflux integration — there are no Konflux Dockerfiles, no rpms.lock.yaml, and no Hermeto prefetch. The base images use upstream `python:3.14-slim` (Debian) rather than UBI, which means the images cannot pass check-payload validation for FIPS compliance. Migration to Konflux builds with UBI base images would be required for RHOAI productization. The uv.lock file provides good language-level dependency pinning, but the `llm-optimizer` dependency is installed from a git repository reference rather than a pinned version, which poses a reproducibility risk.
