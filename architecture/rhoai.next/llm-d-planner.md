## Purpose

**Short**: SLO-driven capacity planning and deployment automation platform that guides users from business requirements to production-ready LLM deployments on Kubernetes.

**Detailed**: llm-d Planner is a multi-component Python application that unifies LLM deployment planning into a single conversational workflow. It consists of a FastAPI backend API, a Streamlit web UI, and a vLLM simulator for GPU-free development. The system extracts user intent via a local LLM (Ollama), maps business requirements to technical specifications (traffic profiles, SLO targets), queries a PostgreSQL benchmark database or generates roofline-based performance estimates, and produces ranked GPU+model configurations scored across accuracy, price, latency, and complexity dimensions.

The platform generates production-ready Kubernetes YAML (KServe InferenceService, HPA, ServiceMonitor) from Jinja2 templates and can deploy directly to clusters via kubectl. It integrates with the RHOAI Model Catalog for benchmark data synchronization and supports both benchmarked and estimated performance data. Three user-facing capabilities are exposed: the main Planner workflow (conversational recommendations), a standalone Capacity Planner (GPU memory estimation for any HuggingFace model), and a GPU Recommender (cross-GPU performance estimation without benchmarks).

## Architectural Analysis

The llm-d Planner follows a clean **layered architecture** with well-separated concerns: a stateless Streamlit frontend communicates exclusively via HTTP with a FastAPI backend, which orchestrates multiple internal Python modules (intent extraction, specification generation, recommendation scoring, configuration templating, cluster management). The backend API acts as a gateway — all external integrations (Ollama, PostgreSQL, Kubernetes, HuggingFace, Model Catalog) are encapsulated behind service classes with dependency injection via FastAPI's lifespan context manager. This design means the UI has zero direct dependencies on infrastructure services.

The **recommendation engine** uses a sophisticated multi-criteria scoring system with four dimensions (accuracy 40%, price 40%, latency 10%, complexity 10%) and generates five ranked views. The scoring includes non-linear formulas (price uses a power function, latency uses a capped-range approach), task-specific bonuses for specialized models, and a scalability penalty for high replica counts. The system supports two data sources: PostgreSQL benchmarks (real performance data from BLIS simulator) and roofline estimation (via BentoML's llm-optimizer library) as a fallback for models without benchmarks. The two-source approach provides broad coverage while clearly distinguishing "benchmarked" from "estimated" confidence levels in the UI.

The **capacity planner** module is remarkably thorough in its GPU memory estimation. It handles all major attention architectures (MHA, GQA, MQA, and the newer MLA used by DeepSeek models), fetches real parameter counts from HuggingFace safetensors metadata, and uses empirically validated activation memory profiles per model architecture. The KV cache calculation correctly accounts for the difference between standard key-value caching and latent attention compression. This precision allows the system to determine valid tensor parallelism values and maximum concurrent request capacity for specific GPU configurations.

A notable **security gap** is the complete absence of authentication on the backend API — all 29 endpoints are publicly accessible. The deployment manifests rely on OpenShift Routes with TLS edge termination for external access, but internal pod-to-pod communication (backend↔UI, backend↔Ollama, backend↔PostgreSQL) uses unencrypted HTTP/PostgreSQL. The CORS middleware allows all origins with credentials enabled, which is appropriate for development but would need tightening for production. The Model Catalog integration is the most security-conscious component, supporting configurable SSL verification and OpenShift service-serving CA bundles for internal HTTPS.

The **deployment topology** includes a Kubernetes Job (`db-init`) that initializes the database schema and loads three sets of benchmark data (BLIS, interpolated, estimated) before the backend starts. The backend uses the same container image as the db-init Job, which is efficient but means the backend image carries database scripts and benchmark JSON files that are only needed at initialization time. The Ollama deployment requests a GPU resource (`nvidia.com/gpu: 1`) and a 20Gi PVC for model weights, making it the most resource-intensive infrastructure component.
