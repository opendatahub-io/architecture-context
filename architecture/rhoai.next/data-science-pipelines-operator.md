## Purpose

**Short**: Manages the lifecycle of Data Science Pipelines (DSP) infrastructure by reconciling DataSciencePipelinesApplication custom resources into a complete ML pipeline execution environment.

**Detailed**: The Data Science Pipelines Operator (DSPO) is a Kubernetes operator that deploys and manages the full stack of components required to run Kubeflow Pipelines v2 on Red Hat OpenShift AI. When a user creates a DataSciencePipelinesApplication (DSPA) custom resource in a namespace, the operator reconciles it into a set of cooperating deployments: a KFP API server (the central pipeline management endpoint), an Argo Workflow Controller (the pipeline execution engine), a Persistence Agent (syncs workflow state back to the API server), a Scheduled Workflow controller (cron-based pipeline triggers), and ML Metadata services (GRPC server + Envoy proxy for artifact and execution lineage tracking).

The operator also manages backing infrastructure: a MariaDB database for pipeline metadata and a MinIO object store for artifact storage (both optional, with external alternatives supported). Security is enforced through kube-rbac-proxy sidecars for authentication, NetworkPolicies for pod-to-pod isolation, optional TLS for inter-component communication, and RBAC roles scoped per DSPA instance. The operator supports managed pipelines, which are pre-compiled pipeline definitions fetched from OCI container images and automatically imported into the pipeline server on deployment.

DSPO integrates with the broader RHOAI platform through the platform operator (rhods-operator), which manages the operator's own deployment. It also integrates with MLflow (for experiment tracking), KServe (for model serving), Ray (for distributed computing), and CodeFlare AppWrappers (for workload queuing).

## Architectural Analysis

The Data Science Pipelines Operator follows a **template-driven reconciliation pattern** that distinguishes it from operators that use programmatic resource construction. All child resources (Deployments, Services, ConfigMaps, Roles, etc.) are defined as Go text/template YAML files in `config/internal/`, rendered with DSPA parameters, and applied via the manifestival library. This approach provides readability and auditability but means that understanding what the operator deploys requires reading both controller Go code (which decides *when* to apply templates) and the template files (which define *what* is applied). The controller code in `controllers/apiserver.go`, `controllers/database.go`, etc. each contain a list of template directories to apply, with conditional logic for optional features (PodToPodTLS, Routes, sample pipelines, managed pipelines).

The **security architecture** uses a layered defense model. External access to the API server and MLMD Envoy is gated by kube-rbac-proxy sidecars that perform SubjectAccessReview checks against the `datasciencepipelinesapplications/api` subresource, ensuring that only users with explicit RBAC grants on the specific DSPA instance can access its APIs. NetworkPolicies provide a second layer, restricting which pods can reach internal ports (8888, 8887, 8080) to only DSP component pods, monitoring namespaces, and workbenches. Pod-to-Pod TLS (optional, default-on) adds encryption for internal communication using OpenShift service-ca generated certificates.

The operator implements **comprehensive health checking** with two validation layers. Pre-deployment health checks verify database and object store connectivity before deploying dependent components, preventing cascading failures. Post-deployment readiness tracking via DSPA status conditions provides fine-grained visibility into each component's state (DatabaseAvailable, ObjectStoreAvailable, APIServerReady, etc.), with an aggregate CrReady condition that considers non-blocking conditions (ManagedPipelinesFetchError, NotApplicable states for disabled components). The metrics subsystem exposes Prometheus gauges for each component's readiness state.

The **managed pipelines** feature represents a unique integration pattern: the operator fetches pipeline definitions from OCI container images at reconcile time, validates pipeline names against the manifest, and injects an init container into the API server Deployment that loads the compiled pipelines on startup. This includes a 10-minute TTL cache per image digest, registry allowlist enforcement, and file size limits (1 MiB per file). The feature uses upload tags (`managed=true,rhoai-version={version}`) to track which pipelines were auto-imported.

The **MLflow integration** uses an auto-detect pattern: the operator watches for `mlflow.opendatahub.io/MLflow` CRs in the DSPA namespace and, if found, configures the API server with the MLflow tracking endpoint. This uses a 45-second TTL cache for endpoint resolution and bypasses the controller-runtime cache for MLflow CRD reads, ensuring the operator works regardless of whether the MLflow CRD was installed before or after the operator started. The integration mode is configurable per DSPA (AUTODETECT or DISABLED).
