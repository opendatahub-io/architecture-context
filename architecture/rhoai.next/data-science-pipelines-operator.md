## Purpose

**Short**: Manages the lifecycle of Data Science Pipelines (Kubeflow Pipelines v2) instances on OpenShift by reconciling `DataSciencePipelinesApplication` (DSPA) custom resources.

**Detailed**: The Data Science Pipelines Operator (DSPO) is a Kubernetes operator built with controller-runtime that watches `DataSciencePipelinesApplication` (DSPA) custom resources. For each DSPA instance, it deploys and manages a complete Kubeflow Pipelines v2 stack including: the KFP API server, a persistence agent, a scheduled workflow controller, an Argo workflow controller, ML Metadata (MLMD) gRPC server with an Envoy proxy, and backing infrastructure (MariaDB database and Minio object storage when not using external services).

The operator uses the manifestival library to render Go templates into Kubernetes manifests and apply them to the cluster. It performs health checks against the database and object storage before deploying pipeline components, manages TLS certificates for pod-to-pod communication via OpenShift service-ca, deploys kube-rbac-proxy sidecars for authenticated external access via OpenShift Routes, creates NetworkPolicies to restrict inter-component communication, and exposes Prometheus metrics for component readiness monitoring. The operator also supports managed pipelines — pre-compiled pipeline definitions bundled in OCI container images that are validated against a manifest and automatically imported into the API server.

## Architectural Analysis

The Data Science Pipelines Operator follows a template-driven reconciliation pattern using the manifestival library rather than programmatic resource construction. All managed sub-component manifests live in `config/internal/` as Go text templates (`.yaml.tmpl` files) that are rendered with `DSPAParams` — a large parameter struct populated during `ExtractParams()`. This approach trades compile-time type safety for deployment flexibility: images, ports, TLS settings, resource limits, and proxy configuration are all injectable without code changes. The operator applies resources via `manifestival.Apply()` with owner injection and label transformers, ensuring proper garbage collection when DSPAs are deleted.

The security architecture centers on pod-to-pod TLS and kube-rbac-proxy. When `PodToPodTLS` is enabled (default: true), the operator annotates Services with `service.beta.openshift.io/serving-cert-secret-name` to trigger OpenShift service-ca certificate generation, then mounts these certs into the API server, MariaDB, and MLMD gRPC containers. External access via OpenShift Routes is protected by kube-rbac-proxy sidecars that perform Kubernetes SubjectAccessReview against the DSPA's `api` subresource — effectively using Kubernetes RBAC to control who can interact with each pipeline instance. NetworkPolicies provide defense-in-depth by restricting direct pod-to-pod communication to only the DSP component set, monitoring namespaces, pipeline execution pods, and workbench pods.

The operator manages a multi-layered health check system. Before deploying any pipeline components, it actively probes database connectivity (MySQL protocol with configurable TLS) and object storage availability (MinIO client `StatObject` call). These health checks use configurable timeouts and support custom CA bundles for TLS verification. Component readiness is tracked through Kubernetes Deployment conditions and published as Prometheus gauges per DSPA instance, enabling monitoring dashboards to track individual pipeline stack health. The managed pipelines feature adds an additional validation layer: the operator fetches `managed-pipelines.json` from an OCI image (using go-containerregistry), caches results per-digest with a 10-minute TTL, and validates CR-specified pipeline names against the manifest before allowing API server deployment. This gate prevents misconfigured managed pipelines from reaching the API server.

The DSPA CRD is notable for its "batteries-included" design — a minimal DSPA spec deploys a complete pipeline stack with MariaDB, Minio, and all pipeline components using sensible defaults. Users can selectively replace any infrastructure component (e.g., point to an external RDS database or S3 bucket) while keeping operator management of the remaining components. The operator intelligently handles transitions: when a user switches from managed to external database, only the relevant templates are skipped. Resource requirements have per-component defaults (defined in `config/defaults.go`) that can be overridden at the CR level. The `DSPVersion` field (currently only "v2" is supported) gates the entire reconciliation — unsupported versions produce clear error conditions without deploying any resources. The finalizer-based cleanup handles both namespaced resources (via owner references) and cluster-scoped resources (ClusterRoleBindings removed explicitly via `cleanUpResources`).
