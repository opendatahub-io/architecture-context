## Purpose

**Short**: Manages the lifecycle of Data Science Pipelines (Kubeflow Pipelines v2) custom resources and all associated Kubernetes workloads on OpenShift AI.

**Detailed**: The Data Science Pipelines Operator (DSPO) is a Kubernetes operator built on controller-runtime that watches `DataSciencePipelinesApplication` (DSPA) custom resources. When a user creates a DSPA in a namespace, DSPO deploys the entire Kubeflow Pipelines v2 stack including the API server, persistence agent, scheduled workflow controller, Argo Workflow controller, ML Metadata (MLMD) gRPC server, and an Envoy proxy for MLMD access. It optionally deploys MariaDB for metadata storage and MinIO for artifact object storage, or connects to external database and S3-compatible storage.

DSPO uses a manifestival-based templating system: Go templates in `config/internal/` are rendered with parameters extracted from each DSPA CR and applied to the cluster. The operator manages per-DSPA NetworkPolicies, OpenShift Routes with kube-rbac-proxy sidecars for authentication, TLS certificates via OpenShift service-ca, and health checks against the database and object storage backends. It supports managed pipelines (pre-compiled pipelines from OCI images), pod-to-pod TLS, proxy configuration, custom CA bundles, FIPS-compliant builds, and the Kubernetes-native pipeline store with admission webhooks for PipelineVersion validation.

The operator is deployed by the platform operator (rhods-operator / opendatahub-operator) via kustomize overlays that inject image references and configuration through a `dspo-parameters` ConfigMap. Each DSPA instance creates a complete, isolated pipeline stack within its namespace, supporting multi-tenancy through namespace-scoped resources and RBAC.

## Architectural Analysis

The Data Science Pipelines Operator follows a **single-controller, multi-template** architecture. Rather than having separate controllers for each sub-component, a single `DSPAReconciler` orchestrates the entire DSP stack through a sequential reconciliation loop. This design simplifies ordering — database and object storage must be healthy before the API server can start — but means a single DSPA failure blocks all component reconciliation. The `MaxConcurrentReconciles` setting (default 10) allows parallel reconciliation across different DSPA instances.

The **manifestival templating system** is central to DSPO's architecture. All Kubernetes resources created by the operator are defined as Go templates in `config/internal/`, rendered with `DSPAParams` extracted from the DSPA CR, and applied via the manifestival library. This provides a clear separation between the operator's reconciliation logic and the resource definitions. The operator injects `dsp-version` labels on all managed resources and uses owner references for garbage collection. The `config/internal/` directory is embedded in the operator image at build time (COPY in Dockerfile.konflux), making the templates immutable per operator version.

The **security model** uses a defense-in-depth approach: NetworkPolicies restrict pod-to-pod communication to only DSP components and authorized workloads (pipeline pods, monitoring, workbenches); kube-rbac-proxy sidecars enforce SubjectAccessReview-based authorization on external Routes; and pod-to-pod TLS (enabled by default) encrypts all internal traffic using OpenShift service-ca certificates. The auth policy for external access checks whether the caller has access to the DSPA subresource `api`, effectively tying pipeline API access to the same RBAC as the DSPA CR itself. This is a notable design choice — it means a single RBAC rule controls both CR management and pipeline API usage.

The **managed pipelines** feature introduces an OCI registry dependency at reconciliation time. DSPO fetches `managed-pipelines.json` from an OCI image (specified by `IMAGES_PIPELINES_COMPONENTS` or per-CR override), validates CR-requested pipeline names against the manifest, and blocks API server deployment on validation failures. The fetcher uses a 10-minute in-memory cache keyed by resolved image digest, with a 1 MiB size limit on manifest extraction. This creates a transient vs. permanent error distinction: registry unavailability is transient (deployment proceeds with status warning), while invalid pipeline names are permanent (deployment blocks). The `RELATED_IMAGE_*` environment variables are forwarded from the operator process to the init container, enabling Konflux-style image pinning for pipelines that reference container images.

The **health check system** probes both database (MySQL `SELECT 1`) and object storage (MinIO `StatObject`) with configurable 15-second timeouts before proceeding with component deployment. These checks run during every reconciliation, and failures set status conditions and trigger 20-second requeue. The operator also monitors pod status for CrashLoopBackOff conditions, providing detailed per-container failure reporting in DSPA status conditions. Nine Prometheus gauge metrics track component readiness across all DSPAs.
