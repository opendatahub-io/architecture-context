## Purpose

**Short**: Manages the full lifecycle of Data Science Pipelines (Kubeflow Pipelines v2) infrastructure on OpenShift, deploying and reconciling API servers, metadata stores, workflow controllers, and supporting services per namespace.

**Detailed**: The Data Science Pipelines Operator (DSPO) is a Kubernetes operator built with controller-runtime that watches `DataSciencePipelinesApplication` (DSPA) custom resources and deploys a complete Kubeflow Pipelines v2 stack per namespace. Each DSPA instance provisions an API server, persistence agent, scheduled workflow controller, Argo Workflows controller, ML Metadata (MLMD) gRPC server with Envoy proxy, and optionally a MariaDB database and MinIO object store. The operator uses manifestival with Go templates to render Kubernetes resources dynamically from templates in `config/internal/`, enabling per-instance customization of images, resource limits, TLS settings, and proxy configuration.

DSPO integrates deeply with the OpenShift platform through service-ca certificate generation, OpenShift Routes for external access, kube-rbac-proxy sidecars for API authentication/authorization, and NetworkPolicies for traffic isolation. It also integrates with the MLflow operator for experiment tracking, supports managed pipeline bundles fetched from OCI registries, and provides Kubernetes-native pipeline storage via Pipeline and PipelineVersion CRDs with validating/mutating webhooks. The operator performs health checks against database and object storage backends before deploying dependent components, and exposes Prometheus metrics for all component readiness states.

## Architectural Analysis

The Data Science Pipelines Operator follows a template-driven reconciliation pattern that is distinctive among ODH/RHOAI operators. Rather than constructing Kubernetes resources programmatically in Go, DSPO uses the manifestival library to apply Go templates from `config/internal/` directories. This provides significant flexibility — each DSPA instance can customize images, resource limits, TLS settings, proxy configuration, and database/storage backends through the CR spec, and these are resolved into templates via a central `DSPAParams` struct. The tradeoff is that the actual resource structure is split across Go controller logic (which decides what templates to apply) and YAML template files (which define the resources), requiring cross-referencing both to understand the full deployment.

The security architecture is well-layered. External access flows through OpenShift Routes with TLS reencrypt termination, through kube-rbac-proxy sidecars that perform SubjectAccessReview against the DSPA resource (checking if the caller has permission on the `datasciencepipelinesapplications/api` subresource), and finally to the application containers. Internal pod-to-pod communication can optionally use TLS via OpenShift service-ca certificates (controlled by `podToPodTLS`, defaulting to true). NetworkPolicies on MariaDB and MLMD gRPC restrict ingress to only the specific pods that need access. The consolidated CA bundle mechanism (`dsp-trusted-ca-{name}` ConfigMap) aggregates certificates from three sources: the global ODH trusted CA bundle, user-provided CA bundles, and service-ca certificates, ensuring all components can verify TLS connections to external services.

The operator's RBAC footprint is notably broad — it requires wildcard permissions on `kubeflow.org/*` and `argoproj.io/workflows`, plus access to multiple ML framework CRDs (KServe, Ray, Seldon, CodeFlare). This is because the pipeline-runner ServiceAccount (created per DSPA) needs these permissions for pipeline steps to create and manage ML workloads. The aggregate ClusterRoles (`aggregate-dspa-admin-edit`, `aggregate-dspa-admin-view`) enable OpenShift's role aggregation to automatically grant DSPA and Pipeline access to users with admin/edit/view roles.

The managed pipelines feature introduces an interesting init-container pattern: an OCI image containing compiled pipeline YAML is validated by the operator (checking pipeline names against a manifest and enforcing registry allowlists), then deployed as an init container that writes pipeline files to a shared emptyDir volume consumed by the API server. This enables platform-managed "golden" pipelines to be delivered as container images, versioned independently of the operator, with a supply chain that runs through the standard Konflux build process.

One notable risk area is the database health check in `database.go`, which uses `InsecureSkipVerify: true` when constructing the TLS config for MySQL connectivity verification. While this is a health check (not production traffic), it means the operator does not verify the database server's certificate during pre-deployment checks, which could be exploited in a MITM scenario. The object storage health check in `storage.go` is more careful, setting `MinVersion: tls.VersionTLS12` explicitly.
