## Purpose

**Short**: Manages the lifecycle of Data Science Pipelines (Kubeflow Pipelines v2) deployments on OpenShift, orchestrating API servers, workflow controllers, metadata services, databases, and object storage.

**Detailed**: The Data Science Pipelines Operator (DSPO) is a Kubernetes operator built with controller-runtime that reconciles `DataSciencePipelinesApplication` (DSPA) Custom Resources. Each DSPA instance defines a complete pipeline execution environment consisting of up to eight interconnected components: an API server for pipeline management, a persistence agent for artifact tracking, a scheduled workflow controller for cron-based execution, an Argo Workflow controller for orchestration, an ML Metadata (MLMD) service with Envoy proxy, a MariaDB database for pipeline metadata, MinIO for S3-compatible object storage, and a Kubernetes webhook for PipelineVersion validation.

The operator uses Go templates (via the manifestival library) to render and apply Kubernetes manifests for each component, supporting extensive configuration including Pod-to-Pod TLS (via OpenShift Service CA), custom CA bundles, proxy configuration, FIPS-compliant builds, and managed pipeline auto-import from OCI registries. It creates NetworkPolicies for inter-component isolation, RBAC for each component's service account, and Prometheus ServiceMonitors for observability. The operator supports both internal (MariaDB/MinIO) and external (any MySQL-compatible DB / S3-compatible storage) backends.

DSPO integrates with the broader RHOAI/ODH platform through the `odh-trusted-ca-bundle` ConfigMap contract, kube-rbac-proxy sidecars for API server authentication, and kustomize overlays that allow the platform operator (`rhods-operator` or `opendatahub-operator`) to deploy it with distribution-specific image references and configuration.

## Architectural Analysis

The Data Science Pipelines Operator follows a template-driven deployment pattern using the manifestival library. Rather than constructing Kubernetes resources programmatically in Go, the operator maintains a large set of Go template files in `config/internal/` that are rendered with a params struct (`DSPAParams`) populated during each reconciliation cycle. This approach makes deployment manifests auditable as standalone YAML templates while still supporting complex conditional logic (e.g., TLS configuration, proxy settings, managed pipelines). The trade-off is that the params struct in `dspipeline_params.go` is very large (~950 lines), acting as a data transfer object between the CRD spec, environment variables, viper configuration, and template rendering.

The operator's CA bundle management is notably sophisticated. It aggregates certificates from three sources: the `odh-trusted-ca-bundle` platform ConfigMap (injected by the OpenShift network operator), a user-provided custom CA bundle ConfigMap, and the OpenShift service-ca ConfigMap. These are merged into a per-DSPA `dsp-trusted-ca-bundle-{name}` ConfigMap that is mounted into all components needing outbound TLS trust. The operator watches the platform CA bundle ConfigMap and triggers reconciliation across all DSPAs in the namespace when it changes, ensuring certificate updates propagate without manual intervention.

The managed pipelines feature represents an interesting OCI-native pattern: pipeline definitions are shipped as layers in container images (via the `pipelines-components` image), and the operator's fetcher (`managed_pipelines_fetcher.go`) extracts a `managed-pipelines.json` manifest by scanning image layers from newest to oldest with OCI whiteout marker support. Results are cached by content digest with a 10-minute TTL. The operator validates user-requested pipeline names against the manifest, distinguishing permanent errors (invalid image, disallowed registry) from transient ones (network failure), allowing deployment to proceed with stale data during transient failures. An OCI registry allowlist (`MANAGED_PIPELINES_ALLOWED_REGISTRIES`) provides supply chain security. RELATED_IMAGE_* environment variables pass through from the operator to the init container, enabling disconnected/air-gapped environments to override component images.

The NetworkPolicy design creates a defense-in-depth topology: MariaDB only accepts connections from the operator pod (in the DSPO namespace), the API server pod, and the MLMD gRPC pod. The API server's 8443 proxy port is open to all sources (for route traffic), while the internal 8888/8887 ports are restricted to DSP component pods, pipeline step pods (`pipelines.kubeflow.org/v2_component: true`), monitoring namespaces, and workbench pods. MLMD gRPC is restricted to pipeline step pods and DSP component pods. This ensures that even within the namespace, only authorized pods can reach backend services.

The webhook component has a unique lifecycle: it deploys in the DSPO namespace (not the DSPA namespace) and is shared across all DSPAs that use `pipelineStore: kubernetes`. The `CleanUpWebhookIfUnused` function checks whether any remaining DSPAs still need the webhook before deleting it, preventing premature cleanup. The webhook deployment uses the operator's own deployment as an owner reference, tying its lifecycle to the operator rather than any individual DSPA.
