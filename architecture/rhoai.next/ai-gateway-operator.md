## Purpose

**Short**: A module operator that manages AI Gateway sub-components (currently batch-gateway-operator) for the Open Data Hub / Red Hat OpenShift AI platform.

**Detailed**: The ai-gateway-operator is a Kubernetes operator built with controller-runtime that runs as a _module operator_ under the opendatahub-operator. It reconciles a cluster-scoped `AIGateway` custom resource (`components.platform.opendatahub.io/v1alpha1`) whose spec controls the lifecycle of AI gateway sub-components. Currently, the only sub-component is the **batch-gateway-operator** (from the `llm-d-batch-gateway-operator` repository), which is deployed when `spec.batchGateway.managementState` is set to `Managed`. The operator vendors sub-component manifests into `config/manifests/` at build time and renders them via kustomize during reconciliation.

The operator follows the ODH modular architecture pattern: the platform operator (`opendatahub-operator`) deploys the module operator itself, and the module operator then deploys its sub-components. This two-level architecture allows the AI Gateway feature area to evolve independently from the platform operator while still being managed through the same DSC/DSCI lifecycle. The operator reports detailed status including its own version, build source, detected platform identity, and the manifest sources it rendered, enabling platform-level observability of module health. A future `MaaS` (Models as a Service) sub-component is planned but not yet implemented.

## Architectural Analysis

The ai-gateway-operator exemplifies the ODH modular operator architecture where a domain-specific "module operator" manages its own sub-components independently from the platform operator. The key design insight is the two-level deployment: `opendatahub-operator` → `ai-gateway-operator` → `batch-gateway-operator`. This allows the AI Gateway team to iterate on their operator's reconciliation logic and sub-component management without requiring changes to the platform operator. The module operator pattern is documented in the [ODH Modularisation Guide](https://docs.google.com/document/d/1FgN_U-6XH8M-Mu6XNeldUlTPsnw7UyPCWg5NVJJdYnw/) and implemented using shared reconciler actions from `opendatahub-operator/v2/pkg/controller/actions/`.

The operator uses a **manifest vendoring** strategy: sub-component manifests (from `llm-d-batch-gateway-operator`) are fetched at build time via `hack/scripts/get-manifests.sh` pinned to specific commit SHAs, stored in `config/manifests/`, and baked into the container image. An init container (`copy-manifests`) copies them from the image into an emptyDir volume that the main manager container reads at runtime. This ensures manifests are immutable per release while allowing the module operator image to be the sole source of truth for what gets deployed. The image parameterization via `params.env` + kustomize `replacements` follows the same pattern used across all ODH components, enabling the platform operator to inject digest-pinned `RELATED_IMAGE_*` references.

The RBAC configuration deserves attention: the operator's ClusterRole includes permissions not just for its own CRD (`components.platform.opendatahub.io/aigateways`) but also for all resources that the batch-gateway-operator needs to manage (`batch.llm-d.ai/llmbatchgateways`, `apps/deployments`, `gateway.networking.k8s.io/httproutes`, `cert-manager.io/certificates`, `monitoring.coreos.com/*`). This is an RBAC escalation pattern — the module operator needs at least the same permissions as its sub-components to create their ServiceAccounts, ClusterRoles, and ClusterRoleBindings. The operator is careful to use `resourceNames` constraints where possible (e.g., only allowing modification of specific ClusterRoles like `llm-d-batch-gateway-operator`).

The upgrade mechanism (`aigateway_upgrade.go`) is currently a no-op stub that detects when either the module version or platform version has changed and calls an empty `upgrade()` function. This provides a hook for future migration logic (e.g., when LLMBatchGateway API changes between versions) without requiring code restructuring. The condition management in `overWriteCondition` shows a pragmatic approach: when no sub-module is `Managed`, the `DeploymentsAvailable` condition is set to `False` with `Info` severity rather than `Error`, preventing the platform operator from flagging the AIGateway component as unhealthy when it simply has nothing to deploy.
