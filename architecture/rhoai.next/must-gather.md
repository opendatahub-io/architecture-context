## Purpose

**Short**: must-gather is a diagnostic data collection tool for Red Hat OpenShift AI (RHOAI) and Red Hat AI Inference (RHAII) that captures cluster state, logs, and custom resources across all RHOAI components and their dependencies.

**Detailed**: The must-gather tool provides a standardized mechanism for collecting diagnostic information from clusters running RHOAI or RHAII. It is packaged as a container image and invoked via `oc adm must-gather` on OpenShift or as a Kubernetes Job on non-OpenShift platforms (AKS, CKS). The tool automatically detects the Kubernetes distribution (OpenShift, AKS, CKS, or generic Kubernetes) and adapts its collection strategy accordingly.

The tool supports collecting data for all RHOAI platform components — Dashboard, AI Pipelines, KServe, KubeRay, Kueue, Kubeflow Training Operator, Workbenches, Model Registry, TrustyAI, Feast Operator, Llama-stack Operator, MLflow Operator, Spark Operator, Models as a Service — either individually via the `COMPONENT` environment variable or all at once (default). On non-OpenShift platforms, collection is scoped to LLM-D/RHAII resources. When collecting all components, the tool runs per-component gather scripts in parallel for efficiency, then produces a summary report listing any failures.

For LLM-D workloads, the tool also collects dependency operator state (cert-manager, Sail/Istio, LeaderWorkerSet), cluster-level node information, observability resources (Prometheus Operator CRDs or Azure Managed Prometheus), and optionally workload-variant-autoscaler (WVA) resources controlled by the `ENABLE_WVA` environment variable. The tool supports Helm release value extraction for RHOAI and RHAII Helm chart deployments.

## Architectural Analysis

must-gather follows a modular, parallel collection architecture. The main entry point (`gather.sh`) acts as an orchestrator that first detects the Kubernetes distribution, then either collects all components in parallel or dispatches to a single component based on the `COMPONENT` environment variable. Each component has its own gather script that follows a consistent pattern: source `common.sh`, define an array of CRD resource types, discover namespaces where those resources exist via `get_all_namespace`, and then call `run_mustgather` (OpenShift) or `run_k8sgather` (xKS) to inspect those namespaces. This pattern is highly extensible — adding a new component requires only a new gather script and a case entry in `gather.sh`.

The xKS support (`xks_util.sh`) is architecturally significant because it reimplements `oc adm inspect` functionality using pure `kubectl` commands. The `kubectl_inspect` function auto-discovers all namespaced resource types via `kubectl api-resources`, collects each one, and then iterates over pods to collect logs for each container (including previous logs). This makes the tool portable to CKS, AKS, and generic Kubernetes without depending on OpenShift tooling. The distribution detection (`detect_k8s_distro`) uses a layered approach: first checking for OpenShift's `infrastructures.config.openshift.io` CRD, then falling back to kernel version (CKS) or provider ID (AKS), and finally checking the OS image.

A default set of Gateway API and Istio resources (`DEFAULT_RESOURCES` in `common.sh`) is automatically collected for every namespace inspected on OpenShift — this includes `gateways.gateway.networking.k8s.io`, `httproutes`, `grpcroutes`, `referencegrants`, `envoyfilters`, and `destinationrules`. This ensures that RHOAI 3.x ingress infrastructure is captured without requiring component-specific scripts to enumerate these resources individually.

The LLM-D collection path is more sophisticated than other components. It runs dependency collection scripts (cert-manager, Sail/Istio, LeaderWorkerSet) in parallel, performs full namespace inspection for all namespaces containing LLMInferenceService CRs, and collects cluster-level information (node details, API resources, cluster-scoped resources). The observability collection (`gather_o11y.sh`) adapts to the monitoring stack: self-hosted (kube-prometheus-stack with Prometheus Operator CRDs) or Azure Managed Prometheus (ama-metrics pods in kube-system). The WVA collection is opt-in (`ENABLE_WVA=true`) and adds KEDA resources on top of observability.

The Helm integration (`collect_helm_releases` in `common.sh`) is noteworthy — it captures both the user-supplied values and the fully rendered manifest for the RHOAI or RHAII Helm release. The default release name differs by platform: `rhoai` on OpenShift, `rhaii` on xKS. This Helm data is critical for debugging configuration issues in installations that use the Helm deployment model (RHOAI 3.x, RHAII).

The image build uses a multi-stage Konflux Dockerfile that pins both the base image and builder stage by SHA256 digest. The kubectl binary comes from the `ose-cli-rhel9` builder image, and the helm binary is checked into the repository via Git LFS to ensure version consistency. The Tekton PipelineRun builds for four architectures (x86_64, ppc64le, s390x, arm64) and runs hermetically.
