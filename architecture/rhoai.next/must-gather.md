## Purpose

**Short**: Collects diagnostic data from Red Hat OpenShift AI (RHOAI) and Red Hat AI Inference (RHAII) clusters for troubleshooting and support.

**Detailed**: The must-gather component is a container image that, when executed via `oc adm must-gather` on OpenShift or as a Kubernetes Job on non-OpenShift platforms (CKS, AKS, EKS), collects comprehensive diagnostic information about RHOAI/RHAII deployments. It gathers namespace-level resources (pods, logs, events, deployments, services), custom resource instances for every RHOAI component, Gateway API and Istio networking resources, Helm release values, cluster-scoped resources, and monitoring/observability data.

The tool supports component-specific collection via the `COMPONENT` environment variable, allowing users to target a single component (e.g., `kserve`, `dashboard`, `llm-d`) rather than collecting everything. When run without a component filter on OpenShift, all 13 component-specific collection scripts execute in parallel. On non-OpenShift Kubernetes distributions (xKS), only the `llm-d` component is collected by default, with optional WVA (workload-variant-autoscaler) and batch-gateway collection gated behind feature flags.

The must-gather image extends the OpenShift `ose-must-gather` base image (for Konflux/downstream builds) or the upstream `origin-must-gather` image (for community builds). It bundles `kubectl` and `helm` CLIs alongside the collection scripts, enabling cross-platform resource collection. The collection scripts auto-detect the Kubernetes distribution (OpenShift, AKS, EKS, CKS) and adapt their behavior accordingly — using `oc` on OpenShift and `kubectl` with custom inspection functions on xKS platforms.

## Architectural Analysis

The must-gather component follows a hierarchical dispatch architecture centered on platform detection. The entry point (`gather.sh`) first sources the xKS distribution detection utility (`xks_util.sh`), which probes cluster metadata (node kernel versions, provider IDs, and OpenShift infrastructure CRDs) to classify the target as OpenShift, AKS, EKS, CKS, or unknown. This classification drives two divergent code paths: OpenShift clusters run the full RHOAI collection with `oc adm inspect` for namespace-level data, while xKS clusters default to `llm-d`-only collection using the `kubectl_inspect` replacement functions.

The cross-platform abstraction layer in `xks_util.sh` is noteworthy. Since `oc adm inspect` is an OpenShift-specific command that collects all resources, pods, logs, and events in a namespace with a single call, the xKS path must replicate this behavior using raw `kubectl` commands. The `kubectl_inspect` function implements this by: (1) dumping the namespace YAML, (2) auto-discovering all namespaced resource types via `kubectl api-resources`, (3) fetching each resource type as YAML, (4) collecting pod logs (current and previous) for all containers — all with parallel pod log collection via background processes. The `auto_discover_resources` function is particularly broad: it iterates over every API resource type registered with the API server and attempts to fetch instances, rather than maintaining a fixed list. This ensures completeness but may be slow on clusters with many CRD types.

The component collection scripts follow a consistent pattern: define an array of CRD resource types, call `get_all_namespace` to discover which namespaces contain those resources, then call `run_mustgather` (OpenShift) or `run_k8sgather` (xKS) to collect them. A shared `DEFAULT_RESOURCES` array in `common.sh` ensures that Gateway API resources (Gateway, HTTPRoute, GRPCRoute, ReferenceGrant) and Istio resources (EnvoyFilter, DestinationRule) are collected from every namespace visited — not just by the serving script. This reflects the RHOAI 3.x networking architecture where Gateway API and Istio resources exist alongside component workloads.

The `llm-d` collection path is significantly more complex than the OpenShift-only component scripts. It collects cluster-level metadata (API resources, node details, cluster-scoped resources), runs three dependency collection scripts in parallel (cert-manager, Sail/Istio, LeaderWorkerSet), inspects all namespaces containing LLMInferenceService CRs, and collects an extensive set of Gateway API, KServe, and Gateway API Inference Extension resources. Two optional collection modules (WVA and batch-gateway) are gated behind `ENABLE_WVA` and `ENABLE_BATCH_GATEWAY` environment variables, defaulting to disabled. The WVA module also chains into the observability collection (`gather_o11y.sh`), which supports both self-hosted kube-prometheus-stack and Azure Managed Prometheus (AMP) monitoring backends.

The build architecture uses a two-stage Konflux Dockerfile: the first stage (`ose-cli-rhel9`) provides `kubectl`, and the second stage (`ose-must-gather-rhel9`) provides the OpenShift must-gather framework. The `helm` binary is checked into the repository via Git LFS (`bin/helm`, 62 MB) to avoid network downloads during hermetic builds. The Tekton pipeline builds multi-architecture images (x86_64, ppc64le, s390x, arm64) with `hermetic: true`. The community `Containerfile` downloads `kubectl` and `helm` at build time via `curl`, which is suitable for development but would fail hermetic builds — hence the separate Konflux Dockerfile.
