## Purpose

**Short**: A must-gather diagnostic image that collects cluster state, logs, and custom resource data for all Red Hat OpenShift AI (RHOAI) components to facilitate support and troubleshooting.

**Detailed**: The must-gather component is a container image designed for use with `oc adm must-gather` (on OpenShift) or as a Kubernetes Job (on non-OpenShift distributions). It collects comprehensive diagnostic information from clusters running RHOAI or RHAII (Red Hat AI Inference) workloads.

The image contains a collection of Bash scripts, each specialized for a specific RHOAI component (KServe, Dashboard, Data Science Pipelines, Kueue, KubeRay, Workbenches, Model Registry, TrustyAI, Feast, Llama-stack, MLflow, Spark Operator, Models as a Service, and LLM-D). These scripts use `oc adm inspect` (on OpenShift) or a custom `kubectl_inspect` function (on xKS platforms) to gather namespace resources, pod logs, custom resource instances, and cluster-scoped information. The main gather script auto-detects the Kubernetes distribution (OpenShift, AKS, EKS, CKS) and adjusts collection behavior accordingly.

The must-gather also supports selective component collection via the `COMPONENT` environment variable, parallel execution of all component gatherers for full collection, configurable log time-range filtering via `--since` / `--since-time`, and optional sub-features such as workload-variant-autoscaler (WVA) and batch-gateway collection for LLM-D workloads. It also collects Helm release values and manifests for Helm-deployed configurations (RHAII).

## Architectural Analysis

The must-gather component follows a modular, script-per-component architecture. The main `gather.sh` entry point acts as a dispatcher: it detects the Kubernetes distribution, configures namespaces, collects platform-level resources (DSCI, DSC, platform CRs), and then either runs a single component gatherer (when `COMPONENT` is set) or spawns all 13 component gatherers in parallel background processes. This parallel execution pattern (`job_pids` tracking with per-job failure reporting) is critical for performance — a full RHOAI cluster has resources spread across many namespaces and component types.

The dual-platform support (OpenShift vs. xKS) is a significant architectural feature. On OpenShift, the scripts rely on `oc adm inspect` for comprehensive resource collection. On non-OpenShift platforms (AKS, EKS, CKS), the `xks_util.sh` library provides a `kubectl_inspect` function that reimplements the core inspect functionality using standard `kubectl` commands: it auto-discovers all API resources in a namespace, serializes them to YAML, and collects pod logs for each container (including previous-container logs). The Kubernetes distribution detection (`detect_k8s_distro`) uses a waterfall of heuristics — checking for OpenShift `infrastructures.config.openshift.io`, CoreWeave kernel version strings, Azure/AWS provider IDs, and EKS version markers.

The LLM-D collection path (`gather_llmd.sh`) is notably more complex than other component gatherers because it must also collect dependency operator state (cert-manager, Sail/Istio, LeaderWorkerSet) and supports optional sub-features (WVA autoscaling via KEDA, batch-gateway). The dependency scripts run in parallel within the LLM-D gatherer, adding a second layer of parallelism. The LLM-D path also collects Gateway API Inference Extension resources (`InferencePool`, `InferenceObjective`) which are not collected by any other gatherer, reflecting the newer inference-specific routing architecture.

A common pattern across all component scripts is the `DEFAULT_RESOURCES` array in `common.sh`, which ensures Gateway API resources (`Gateway`, `HTTPRoute`, `GRPCRoute`, `ReferenceGrant`), `EnvoyFilter`, and `DestinationRule` CRs are collected in every namespace alongside component-specific resources. This reflects the RHOAI 3.x architecture where Gateway API and Istio resources are pervasive. The `run_mustgather` and `run_k8sgather` functions both iterate through resources per namespace, but the former uses `oc adm inspect` while the latter uses the custom `kubectl_inspect` — a clean abstraction that keeps component scripts distribution-agnostic.

The Helm integration (`collect_helm_releases`) supports the RHAII (Red Hat AI Inference) deployment model where components are installed via Helm charts rather than OLM operators. The function collects both `helm get values` (user-supplied configuration) and `helm get manifest` (rendered Kubernetes manifests), providing a complete picture of the deployed state. The release name defaults to `rhoai` on OpenShift and `rhaii` on xKS platforms, reflecting the different product names.
