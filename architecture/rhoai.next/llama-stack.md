## Purpose

**Short**: Llama Stack is a standardized API server for building and deploying generative AI applications, providing unified interfaces for inference, agents, safety, RAG, evaluation, and tool use across pluggable backend providers.

**Detailed**: Llama Stack is Meta's open-source framework that defines and implements a comprehensive set of REST APIs for interacting with large language models and building AI-powered applications. The server provides a plugin-based architecture where each API capability (inference, safety, vector I/O, agents, evaluation, etc.) can be backed by different "providers" — either inline implementations running in-process or remote adapters connecting to external services like vLLM, Ollama, OpenAI, Anthropic, and many others.

The distribution system allows operators to compose custom "stacks" by selecting which providers to use for each API, configured via YAML files. Pre-built distribution templates (e.g., `starter`, `remote-vllm`, `ollama`) provide ready-to-use configurations for common deployment scenarios. The server runs on FastAPI/Uvicorn with support for TLS, OAuth2/JWT authentication, attribute-based access control (Cedar-inspired policies), rate limiting, and OpenTelemetry-based tracing.

Red Hat distributes Llama Stack as part of Red Hat OpenShift AI (RHOAI) to provide a standardized AI application development API layer. This fork (`red-hat-data-services/llama-stack`) tracks the upstream `meta-llama/llama-stack` repository and serves as the basis for RHOAI's Llama Stack integration, enabling enterprise customers to build agentic AI applications using a consistent, supported API surface.

### Authentication & Authorization

| Endpoint | Methods | Auth Mechanism | Enforcement Point | Policy |
|----------|---------|----------------|-------------------|--------|
| `/v1/*` (all API endpoints) | ALL | Bearer Token (OAuth2 JWT or Custom) | AuthenticationMiddleware (ASGI) | Configurable via `server.auth` in YAML config |
| `/v1/health` | GET | None | None | Public health check |
| `/v1/version` | GET | None | None | Public version info |
| `/docs`, `/redoc`, `/openapi.json` | GET | None | None | API documentation |
| All resource endpoints | ALL | Attribute-based access control (ABAC) | AccessControlMiddleware | Cedar-inspired rules: permit/forbid with principal, resource, and condition matching |

**Authentication Providers:**
- **OAuth2 Token (JWT)**: Validates tokens via JWKS URI or RFC 7662 token introspection. Extracts claims (sub, groups, team, project, tenant, namespace) mapped to access attributes (roles, teams, projects, namespaces). Supports configurable audience, issuer, and TLS verification.
- **Custom**: Delegates token validation to an external HTTP endpoint. Sends token + request context; receives principal and attributes.

**Access Control Model:**
- Cedar-inspired policy language with `permit`/`forbid` rules
- Scoped by `principal`, `actions` (create, read, update, delete), and `resource` (type-qualified, e.g., `model::my-model`)
- Conditions: `when`/`unless` clauses supporting attribute matching (`user with X in roles`), ownership checks (`user is owner`), and group membership (`user in owners teams`)
- Rules evaluated in order; first match wins; default deny if no rules match

**Rate Limiting:**
- QuotaMiddleware enforces per-client request quotas
- Separate limits for authenticated vs anonymous clients (default: 1000/100 per day)
- Authenticated clients identified by token; anonymous by IP address
- KV store backend (SQLite) for counter persistence

The `hashlib.md5` calls use `usedforsecurity=False` which is FIPS-compliant per PEP 456._

## Architectural Analysis

Llama Stack's architecture follows a **protocol-oriented provider pattern** where each AI capability (inference, safety, vector I/O, agents, etc.) is defined as a Python `Protocol` class decorated with `@webmethod` annotations that declare HTTP routes. The distribution system composes these protocols with concrete provider implementations at startup, using a YAML configuration to map each API to one or more providers. This creates a highly flexible deployment model — the same API surface can be backed by local inference (vLLM, meta-reference), cloud providers (OpenAI, Anthropic, Bedrock), or a mix of both simultaneously. The routing layer supports multi-provider configurations where different models can be served by different providers, with automatic routing based on model identifiers.

The server's middleware stack is well-layered: client version compatibility checking, authentication (OAuth2 JWT or custom), rate limiting (per-client quotas), and distributed tracing (W3C trace context) are all implemented as independent ASGI middleware components. The authentication system is particularly noteworthy — it supports both JWT validation (via JWKS or RFC 7662 introspection) and delegation to custom auth endpoints, with the resulting user attributes fed into a Cedar-inspired policy evaluation engine for fine-grained resource access control. This ABAC model allows platform operators to define rules like "users with admin role can access all models" or "users can only access resources they created" without modifying application code.

A significant architectural consideration for RHOAI integration is the **provider isolation model**. All providers run in-process within the same Python process — there is no sandboxing or resource isolation between providers. This means a misbehaving provider (e.g., one that leaks memory or blocks the event loop) affects the entire server. For production RHOAI deployments, the recommended pattern is to use `remote::` providers that delegate heavy computation to separate services (vLLM for inference, pgvector for vectors) while keeping only lightweight coordination logic in-process. The `inline::` providers (meta-reference inference, FAISS, SQLite-vec) are suitable for development and testing but may not meet production requirements for scalability and fault isolation.

The codebase shows active evolution toward OpenAI API compatibility — the `/openai/v1/*` endpoints for chat completions, files, models, and the Responses API provide a familiar interface for developers migrating from OpenAI. The Responses API (`/openai/v1/responses`) is particularly significant as it represents the agentic workflow entry point, supporting multi-turn conversations with tool use, code execution, and web search. This dual-API approach (native Llama Stack + OpenAI-compatible) allows gradual adoption while maintaining the richer Llama Stack API for advanced use cases like safety shields, evaluations, and custom scoring.

One area that warrants attention for RHOAI hardening is the JWT validation library: `python-jose` uses pure-Python RSA/ECDSA implementations (`rsa` and `ecdsa` packages) rather than linking to system OpenSSL via the `cryptography` package. In FIPS-enforcing environments, this means JWT signature verification may not use FIPS-validated cryptographic operations. A migration to `PyJWT` with `cryptography` backend would address this gap.
