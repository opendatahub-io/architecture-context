## Purpose

**Short**: Llama Stack is a modular AI application server providing standardized REST APIs for LLM inference, agents, RAG, safety, evaluation, and post-training with a pluggable provider architecture.

**Detailed**: Llama Stack is Meta's open-source framework for building production AI applications, forked by Red Hat for integration into RHOAI. It exposes a comprehensive set of REST APIs (OpenAPI 3.1.0) through a FastAPI/uvicorn server that abstracts over multiple LLM inference backends, vector databases, safety systems, and tool runtimes via a pluggable provider architecture.

The system operates on a "distribution" model where pre-configured templates (e.g., `remote-vllm`, `starter`, `ollama`) define which provider plugins are active for each API surface. Providers are split into two categories: **inline providers** that run in-process (e.g., vLLM local, FAISS, sentence-transformers) and **remote providers** that delegate to external services via HTTP adapters (e.g., remote vLLM, OpenAI, Anthropic, pgvector). This architecture allows a single Llama Stack server to simultaneously route requests to multiple inference backends, vector stores, and auxiliary services.

For RHOAI, Llama Stack serves as the AI application gateway layer, enabling standardized access to LLM capabilities through both native Llama Stack APIs and OpenAI-compatible endpoints. It integrates with vLLM for model serving, supports OAuth2/JWT authentication, provides attribute-based access control (ABAC), and includes quota/rate-limiting middleware. The server supports TLS with optional mutual TLS (mTLS) for production deployments.

## Architectural Analysis

Llama Stack's architecture is built around a **provider abstraction layer** that decouples API endpoints from their backend implementations. The `StackRunConfig` (defined in `llama_stack/distribution/datatypes.py`) is the central configuration object that maps each API surface (inference, safety, vector_io, etc.) to one or more provider instances. The resolver (`llama_stack/distribution/resolver.py`) reads this configuration at startup, dynamically imports provider modules, and constructs the provider dependency graph. This design allows the same Llama Stack server to be reconfigured for different environments simply by swapping a YAML configuration file — from a local development setup with Ollama and SQLite to a production deployment with remote vLLM, pgvector, and Redis.

The **middleware chain** is a notable architectural choice. Rather than embedding authentication and rate-limiting into route handlers, Llama Stack implements them as ASGI middleware layers that execute before the FastAPI routing takes effect. The order is: `ClientVersionMiddleware` → `AuthenticationMiddleware` → `QuotaMiddleware` → `TracingMiddleware` → route handlers. This means authentication and quota checks happen at the protocol level, before any request body parsing or validation. The `AuthenticationMiddleware` supports two authentication strategies — JWT validation via JWKS (with periodic key refresh) and token introspection (RFC 7662) — both configured declaratively in YAML. The access control system is implemented separately from authentication, using a Cedar-inspired policy language (`llama_stack/distribution/access_control/`) that evaluates rules like `"user in owners roles"` or `"user with admin in roles"` against user attributes extracted from JWT claims.

The **routing system** is dynamically constructed from Python protocol classes. Each API (e.g., `Inference`, `Safety`, `VectorIO`) is defined as a Python `Protocol` class where methods are decorated with `@webmethod(route=..., method=...)`. At startup, `get_all_api_routes()` in `routes.py` introspects these protocol classes to enumerate all routes, then `create_dynamic_typed_route()` wraps each provider implementation method into a FastAPI route handler. This means routes are not statically declared but are programmatically derived from the API type system. This approach ensures API definitions serve as both the Python interface contract and the HTTP endpoint specification.

The **streaming architecture** uses Server-Sent Events (SSE) for real-time responses. Inference completions, agent turns, and OpenAI-compatible endpoints all support `stream=True`, which causes the server to return a `StreamingResponse` with `media_type="text/event-stream"`. The SSE generator wraps async generators with context preservation (`preserve_contexts_async_generator`) to maintain trace context and provider data across async boundaries — a subtle but important detail for correct telemetry in streaming scenarios. The server also adds an `x-trace-id` response header to all requests via the `TracingMiddleware`, enabling end-to-end trace correlation.

A key risk area is the **lack of Konflux build infrastructure** in this repository. Unlike other RHOAI components, there are no `Dockerfile.konflux` files, no `rpms.lock.yaml`, and no Hermeto/cachi2 prefetch integration on the `main` branch. This means the container image build process, FIPS compliance enforcement, and supply chain hermeticity are entirely delegated to the downstream release branch and Konflux pipeline configuration. The `uv.lock` file provides Python-level dependency pinning, but OS-level packages are not locked. The `hashlib.md5` usage in three locations is flagged with `usedforsecurity=False` (PEP 452 compliant), but the broader crypto posture depends on the runtime base image's OpenSSL configuration.
