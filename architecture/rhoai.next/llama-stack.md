## Purpose

**Short**: Llama Stack is an extensible AI inference and agent orchestration server that provides a unified API layer for model inference, safety, retrieval-augmented generation (RAG), evaluation, and agentic workflows.

**Detailed**: Llama Stack provides a standardized REST API server built on FastAPI/Uvicorn that abstracts over multiple LLM inference backends (vLLM, Ollama, TGI, OpenAI, Anthropic, Bedrock, Fireworks, Together, NVIDIA, and many more) through a pluggable provider architecture. The server exposes both Llama Stack-native API endpoints and OpenAI-compatible endpoints for inference, chat completions, embeddings, agents, files, and responses.

The architecture follows a "distribution" model where a YAML configuration file selects which provider implementations to wire together for each API surface (inference, safety, vector I/O, evaluation, telemetry, etc.). Providers are categorized as "inline" (running in-process, e.g., meta-reference inference with PyTorch, FAISS vector store) or "remote" (delegating to external services via HTTP/gRPC adapters). This allows the same server binary to be deployed against local GPU hardware, cloud-hosted inference services, or hybrid configurations.

The server includes enterprise features critical for RHOAI deployment: OAuth2/JWT authentication with JWKS and token introspection, Cedar-inspired attribute-based access control (ABAC) with per-resource policies, per-client request quota/rate limiting, TLS/mTLS support, W3C distributed tracing, and OpenTelemetry integration. The CLI (`llama`) provides commands for stack building, running, model downloading, and template management.

| Component | Type | Purpose |

| Llama Stack Server | Python Service (FastAPI/Uvicorn) | HTTP API server exposing inference, agents, safety, RAG, eval, and telemetry endpoints with pluggable provider backends |
| Provider Router | Python Module | Routes API requests to the correct provider implementation based on model/resource registration and routing tables |
| Access Control Engine | Python Module | Cedar-inspired policy engine enforcing attribute-based access control on all resource operations |
| Authentication Middleware | Python Module | Validates Bearer tokens via OAuth2 JWT (JWKS) or custom auth endpoint, extracts user attributes |

| CLI (`llama`) | Python CLI (fire) | Command-line interface for stack build, run, model download, and template management |
| Distribution Templates | YAML Configuration | Pre-configured provider compositions for common deployment scenarios (vLLM, Ollama, Bedrock, etc.) |

## Architectural Analysis

Llama Stack follows a **distribution-based plugin architecture** that is distinct from typical Kubernetes operators or microservices. The entire system runs as a single Python process (FastAPI on Uvicorn) whose behavior is defined entirely by a YAML configuration file at startup. This configuration selects which "providers" to activate for each API surface, creating what the project calls a "distribution" — a named composition of providers. For example, the `remote-vllm` distribution configures vLLM as the inference provider and FAISS as the vector store, while the `bedrock` distribution uses AWS Bedrock for inference.

The **provider abstraction** is the core architectural pattern. Each API surface (inference, safety, vector I/O, agents, eval, scoring, telemetry, tools, files, datasets, post_training) defines a Python Protocol interface. Providers implement these interfaces either "inline" (running in-process with the server) or "remote" (delegating to external services via HTTP adapters). The router layer maps resource identifiers (model IDs, vector DB IDs, shield IDs) to the correct provider implementation through a routing table. This allows multiple providers for the same API to coexist — for example, different models can be served by different inference backends simultaneously.

The **security architecture** is layered and enterprise-grade for a Python service. Authentication supports OAuth2 JWT validation with JWKS key rotation (caching with configurable recheck period), RFC 7662 token introspection, and custom auth endpoints. Authorization uses a Cedar-inspired policy language with attribute-based access control: rules can permit or forbid actions (CREATE, READ, UPDATE, DELETE) scoped to specific principals, resources (qualified as `type::identifier`, e.g., `model::llama-3`), and conditions based on user attributes or resource ownership. The quota middleware provides per-client rate limiting backed by a persistent KV store. TLS termination is handled directly by Uvicorn with optional mutual TLS.

The **streaming design** uses Server-Sent Events (SSE) with `text/event-stream` for all streaming endpoints (chat completion, agent turns, etc.). The implementation uses async generators with explicit context variable preservation across event loop boundaries via `preserve_contexts_async_generator`, which maintains trace context and provider data across SSE event boundaries. This is a careful design choice that avoids common pitfalls with FastAPI's `StreamingResponse` and asyncio context variables.

For RHOAI deployment, the key concern is that Llama Stack is designed to be deployed as an opaque service behind the platform's ingress infrastructure. It does not define any Kubernetes-native resources (no CRDs, no kustomize manifests, no Helm charts). The deploying operator (rhods-operator or similar) must create the Deployment, Service, ConfigMap (for run configuration), and ingress resources (HTTPRoute or Route). The server's own authentication can be layered on top of or delegated to the platform's authentication (e.g., the platform's kube-rbac-proxy sidecar can handle OpenShift auth, while Llama Stack's built-in OAuth2 middleware handles application-level JWT validation).
