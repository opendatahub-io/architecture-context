## Purpose

**Short**: A unified Python SDK for managing ML training workloads across Kubeflow projects (Trainer, Katib, Spark Operator, Model Registry) with RHAI-specific extensions for Red Hat OpenShift AI.

**Detailed**: The Kubeflow SDK is a pure Python client library (no server-side component) that provides unified Pythonic APIs to interact with Kubeflow ecosystem services running on Kubernetes. It abstracts Kubernetes complexity, allowing data scientists to manage distributed training jobs, hyperparameter optimization, Spark data processing, and model registration through consistent Python APIs without needing to learn Kubernetes primitives.

This repository is the OpenDataHub midstream fork of the upstream `kubeflow/sdk`, containing Red Hat AI (RHAI) extensions that are not present upstream. The RHAI extensions add two specialized trainers — `TransformersTrainer` (for HuggingFace Transformers/TRL with auto-instrumentation for checkpointing, progression tracking, and cloud storage integration) and `TrainingHubTrainer` (for RHAI Training Hub algorithm integration with SFT, OSFT, LoRA SFT, and LoRA GRPO support). These extensions integrate with the OpenDataHub/RHOAI training controller ecosystem via custom annotations and HTTP metrics endpoints.

The SDK is not deployed as a service — it is installed via `pip install kubeflow` and runs client-side, communicating with Kubernetes API servers and Kubeflow operator CRDs. It supports three execution backends: KubernetesBackend (production, creates TrainJob CRs), ContainerBackend (local Docker/Podman), and LocalProcessBackend (Python subprocesses for quick prototyping).

## Architectural Analysis

The Kubeflow SDK employs a clean **strategy pattern** for backend abstraction. The `RuntimeBackend` ABC (`kubeflow/trainer/backends/base.py`) defines the contract, and three concrete implementations (Kubernetes, Container, LocalProcess) provide the same API surface across different execution environments. This allows users to develop and test locally with `ContainerBackendConfig()` and deploy to production with `KubernetesBackendConfig()` by changing a single constructor argument. The backend selection is fully driven by the config type — no factory pattern or registration needed.

The **RHAI extension architecture** is the most architecturally significant aspect of this midstream fork. The `kubeflow.trainer.rhai` module adds two trainer types (`TransformersTrainer` and `TrainingHubTrainer`) that extend the SDK's capabilities beyond what upstream provides. These trainers use a code injection pattern: they extract Python source code via `inspect.getsource()`, wrap it with instrumentation code (progression tracking HTTP servers, JIT checkpoint handlers, metrics file readers), and embed the combined code into the TrainJob's container command as a heredoc script. This avoids requiring any SDK dependencies inside the training container image — the injected code is fully self-contained. The `TransformersTrainer` monkey-patches `transformers.Trainer.__init__` to auto-inject callbacks for progression tracking and JIT checkpointing on SIGTERM, while `TrainingHubTrainer` generates wrapper scripts that call Training Hub algorithms (SFT, OSFT, LoRA SFT, LoRA GRPO) with embedded metrics reading and HTTP serving.

The **checkpoint storage abstraction** in `TransformersTrainer` supports three URI schemes: local filesystem paths, PVC URIs (`pvc://<name>/<path>`, which add PersistentVolumeClaim volume mounts to the TrainJob's pod template overrides), and S3 URIs (`s3://<bucket>/<prefix>`, which use ephemeral emptyDir volumes for local staging followed by async background uploads via fsspec). The S3 path includes sophisticated features: LIFO upload queues for checkpoint prioritization, per-rank incomplete markers for distributed consistency, parallel file uploads with retry logic, automatic checkpoint download on training resume, and SSL verification controls for self-signed certificates. Cloud storage credentials are injected from Kubernetes Data Connection secrets (read via `core_api.read_namespaced_secret`).

The **algorithm registry** (`kubeflow/trainer/algorithms.py`) is a centralized, extensible registry of supported training algorithms. Each `AlgorithmSpec` defines the algorithm's name, metrics file patterns, validation function, and entrypoint command. This design makes adding new algorithms a single-point change — add an entry to the `ALGORITHMS` dict and the rest of the SDK (metrics reading, entrypoint selection, pod metadata injection) derives its behavior from the spec. The current registry supports four algorithms: SFT, OSFT, LoRA SFT, and LoRA GRPO, each with different metrics file patterns and entrypoint strategies (some use `torchrun` directly, others use `python` and launch `torchrun` internally).

One notable **security consideration** is that the in-pod metrics HTTP server (port 28080) binds to `0.0.0.0` with no authentication. This is acceptable because the server runs inside training pods and is only accessible within the cluster network (polled by the RHOAI controller). However, in multi-tenant environments without NetworkPolicy enforcement, any pod in the namespace could read training metrics. The `verify_cloud_storage_ssl=False` option is another security-relevant feature — it's explicitly documented as a security risk and guarded by warnings, but its presence enables use with self-signed S3-compatible storage endpoints.
