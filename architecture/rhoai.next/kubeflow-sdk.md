## Purpose

**Short**: Unified Python SDK providing consistent APIs to manage ML workloads across Kubeflow ecosystem components (Trainer, Katib Optimizer, Spark, Model Registry) without requiring Kubernetes expertise.

**Detailed**: The Kubeflow SDK is a Python client library that abstracts Kubernetes complexity for AI/ML practitioners. It provides a pluggable backend architecture allowing the same training code to run on Kubernetes clusters (production), local containers (Docker/Podman for development), or local subprocesses (rapid prototyping). The SDK does not deploy any services itself -- it is a pip-installable package that communicates with existing Kubernetes operators (Kubeflow Trainer, Katib, Spark Operator) and services (Model Registry) via their CRD APIs and HTTP/gRPC endpoints.

This repository is an OpenDataHub (ODH) midstream fork of the upstream `kubeflow/sdk`. It carries downstream-only additions in the `kubeflow/trainer/rhai/` package, which adds RHOAI-specific trainers (`TransformersTrainer` for HuggingFace Transformers/TRL and `TrainingHubTrainer` for TrainingHub algorithms). These RHAI trainers add capabilities not present upstream: real-time training progression tracking via HTTP metrics servers, just-in-time (JIT) checkpoint management with PVC and S3 storage, cloud storage credential injection from Kubernetes Secrets, and algorithm-specific metrics parsing for SFT, OSFT, LoRA SFT, and LoRA GRPO training algorithms. The midstream follows a weekly upstream sync cadence with an automated rebase workflow and an independent RHAI release process (`vX.Y.Z+rhaiv.N` versioning).

## Architectural Analysis

The Kubeflow SDK follows a clean **backend abstraction pattern** that decouples training logic from execution infrastructure. The `TrainerClient` facade delegates all operations to a `RuntimeBackend` interface with three implementations: `KubernetesBackend` (production CRD-based), `ContainerBackend` (Docker/Podman for development), and `LocalProcessBackend` (subprocess-based for rapid prototyping). This allows users to iterate locally with `ContainerBackendConfig()` and deploy to production by simply switching to `KubernetesBackendConfig()` without changing training code. The same pattern extends to Spark (`SparkClient` with `KubernetesBackend`) and the Optimizer (`OptimizerClient` with `KubernetesBackend`), though these currently only support the Kubernetes backend.

The **RHAI extensions** (`kubeflow/trainer/rhai/`) represent the most architecturally significant downstream additions. Rather than modifying upstream code, they introduce two new trainer types (`TransformersTrainer` and `TrainingHubTrainer`) that generate instrumented Python code injected into training container commands. The instrumentation code is serialized as string literals and embedded in bash commands -- a code-as-data pattern that avoids binary patching but makes the generated code difficult to test and debug. The `TransformersTrainer` monkeypatches HuggingFace `transformers.Trainer` methods at runtime to intercept checkpointing and metrics, which creates tight coupling to specific Transformers API versions. The `TrainingHubTrainer` supports four algorithm types (SFT, OSFT, LoRA SFT, LoRA GRPO) with algorithm-specific metrics parsing logic.

The **checkpoint system** in `TransformersTrainer` implements a sophisticated SIGTERM-handling mechanism for JIT (just-in-time) checkpointing. When a training pod receives SIGTERM (e.g., during preemption or scale-down), a `CheckpointManager` intercepts the signal, saves the model state to a local ephemeral volume, and then uploads it to S3 via `fsspec`/`s3fs`. On resume, the system auto-detects the latest checkpoint. This design has an important nuance: cloud storage uploads happen synchronously during the SIGTERM grace period, which means large checkpoints on slow S3 connections could be truncated if the grace period expires. The SDK uses an "incomplete marker" file (`checkpoint-is-incomplete.txt`) to detect this scenario.

The **progression tracking** system injects an HTTP server into training pods that serves real-time metrics (progress percentage, estimated remaining time, current step/epoch, train/eval metrics) in a camelCase JSON format compatible with the Trainer controller's metrics polling. Annotations on the TrainJob CR (`trainer.opendatahub.io/progression-tracking`, `trainer.opendatahub.io/metrics-port`, `trainer.opendatahub.io/metrics-poll-interval`) signal the controller to poll these endpoints. Final metrics are also written to `/dev/termination-log` as a Kubernetes-native fallback mechanism for post-completion metrics capture.

The **upstream sync mechanism** is well-engineered. A weekly GitHub Actions workflow (`rebase-upstream.yaml`) rebases `main` onto the upstream `kubeflow/sdk` repository, with automated conflict detection and Snyk security scanning of incoming changes. The RHAI-specific code lives in `kubeflow/trainer/rhai/` which is unlikely to conflict with upstream changes. The release process uses `vX.Y.Z+rhaiv.N` versioning (PEP 440 local version) and pins all direct dependencies to exact versions from `uv.lock` during release, ensuring reproducible builds.

A notable architectural gap is that the SDK ships no Konflux Dockerfiles -- it is distributed purely as a pip package. This means it does not go through Konflux's check-payload FIPS validation or RPM-level supply chain locking. FIPS compliance depends entirely on the training container images where the SDK is installed. The SDK's `verify_cloud_storage_ssl=False` option is a potential security concern in FIPS-regulated environments, though it defaults to `True`.
