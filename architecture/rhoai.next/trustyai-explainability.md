## Purpose

**Short**: TrustyAI is a Java library and Quarkus-based REST service providing AI explainability (SHAP, LIME, Counterfactual), fairness metrics (SPD, DIR), and data drift detection for models served by ModelMesh and KServe on Red Hat OpenShift AI.

**Detailed**: TrustyAI Explainability is a multi-module Java project that provides Explainable AI (XAI) capabilities at both a library and service level. The core library implements fairness metrics (Disparate Impact Ratio, Statistical Parity Difference), explainability algorithms (SHAP, LIME, Counterfactual, Time Series Saliency), and drift detection methods (Kolmogorov-Smirnov, Fourier MMD, Meanshift).

The `explainability-service` module is the primary deployable artifact — a Quarkus 3.8.5 REST service that exposes 50+ HTTP endpoints for on-demand and scheduled metric computation, explanation generation, and data management. It integrates with ModelMesh via a payload consumer endpoint that captures inference data in real time, and with KServe-compatible model servers via gRPC for generating explanations. Computed metrics are published to Prometheus via Micrometer for dashboard integration.

The project also includes the `explainability-connectors` module (KServe V1/V2 HTTP and gRPC clients with protobuf definitions), the `explainability-core` module (core algorithms, data models, and metrics), and the `explainability-arrow` module (Apache Arrow serialization for Java-Python interoperability). The RHOAI distribution profile disables explainer endpoints (SHAP, LIME, Counterfactual, TS Saliency, global explainers) while enabling fairness and drift metrics.

## Architectural Analysis

TrustyAI Explainability follows a modular Maven architecture with clear separation of concerns. The `explainability-core` module is a pure Java library with no service dependencies, containing the algorithmic implementations for SHAP, LIME, Counterfactual, and Partial Dependence Plot explanations, along with fairness metrics (Disparate Impact Ratio, Statistical Parity Difference) and drift detection (Kolmogorov-Smirnov, Fourier MMD, Meanshift). This module uses OptaPlanner as a constraint solver for counterfactual explanations — a notable architectural choice that enables goal-directed search over feature spaces. The core data model (`Feature`, `Prediction`, `PredictionInput`, `PredictionOutput`, `Dataframe`) provides a consistent abstraction that all algorithms operate on, regardless of the underlying model serving protocol.

The `explainability-connectors` module bridges TrustyAI's internal data model to the KServe inference protocol. It implements both HTTP (V1 and V2) and gRPC (V2) prediction providers, with the gRPC client generated from the standard KServe `grpc_predict_v2.proto` protobuf definition. The connector uses a semaphore-based concurrency limiter for gRPC calls, which prevents overwhelming model servers during batch explanation generation. The `explainability-arrow` module adds Apache Arrow IPC support for zero-copy data exchange with the TrustyAI Python library, enabling hybrid Java-Python workflows.

The service layer's most distinctive design pattern is its **pluggable storage architecture** — the same `Storage` interface supports four backends: in-memory (development), CSV flat files on PVC (default production), Hibernate ORM with MariaDB/MySQL (database mode), and MinIO S3 (cloud-native mode). This flexibility means TrustyAI can adapt to different deployment topologies without code changes, driven entirely by the `SERVICE_STORAGE_FORMAT` environment variable. The production-default PVC mode stores inference data as CSV files on a 1Gi ReadWriteOnce volume, which inherently limits the service to single-replica deployments.

A key architectural observation is that TrustyAI itself implements **no authentication or authorization** — it relies entirely on external enforcement. In RHOAI deployments, the TrustyAI Service Operator injects a `kube-rbac-proxy` (RHOAI 3.x) or `oauth-proxy` (RHOAI 2.x) sidecar container to authenticate requests before they reach the Quarkus service. This explains why the manifest's Route has `tls: null` and all endpoints accept unauthenticated requests — the auth proxy handles TLS termination and token validation at the pod level. The ServiceMonitor configuration confirms this pattern: it uses `bearerTokenFile` to authenticate Prometheus scraping against the service.

The RHOAI profile (`%rhoai.*`) disables all explainer endpoints (SHAP, LIME, Counterfactual, Time Series Saliency, global explainers) while keeping fairness and drift metrics enabled. This suggests that explainability features are considered tech-preview or not yet certified for RHOAI, while fairness monitoring is the primary supported use case. The init container in the deployment manifest writes a `model-serving-config` ConfigMap that configures ModelMesh's `payloadProcessors` to route inference data to TrustyAI — this is the critical integration mechanism that enables transparent inference data capture without modifying model serving infrastructure.
