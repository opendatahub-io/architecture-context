## Purpose

**Short**: TrustyAI is a Java library and Quarkus-based service providing AI model explainability, fairness metrics, and data drift detection for ML models served via KServe/ModelMesh on OpenShift AI.

**Detailed**: TrustyAI Explainability is a multi-module Java project that provides both a reusable library for AI explainability algorithms and a deployable Quarkus service for the Red Hat OpenShift AI (RHOAI) platform. The core library implements model-agnostic explainability algorithms including LIME (Local Interpretable Model-agnostic Explanations), SHAP (SHapley Additive exPlanations), Counterfactual explanations, and Partial Dependence Plots (PDP), enabling data scientists and ML engineers to understand why AI models make specific predictions.

The service component exposes these capabilities via REST endpoints and integrates with the KServe/ModelMesh model serving infrastructure. It consumes inference payloads (both requests and responses) from model servers via HTTP and CloudEvents, stores observation data (supporting PVC, MariaDB/MySQL database, MinIO, and in-memory backends), computes fairness metrics (Statistical Parity Difference, Disparate Impact Ratio) and drift metrics (KS Test, FourierMMD, Meanshift), and publishes results to Prometheus for monitoring dashboards. The service connects to KServe-compatible model servers via both HTTP (v1/v2 protocols) and gRPC (v2 protocol) to invoke models during explanation generation.

In the RHOAI distribution, the explainer endpoints (LIME, SHAP, Counterfactual, TSSaliency, PDP, Aggregated LIME) are disabled via profile configuration, exposing only the fairness and drift metric endpoints. The ODH distribution enables all endpoints. The service is deployed per data science project namespace by the TrustyAI operator (trustyai-service-operator), which manages the lifecycle, storage, and integration with model serving infrastructure.

## Architectural Analysis

The TrustyAI Explainability project follows a clean multi-module Maven architecture separating concerns into distinct libraries and a service layer. The `explainability-core` module is a pure Java library implementing AI explainability algorithms (LIME, SHAP, Counterfactual via OptaPlanner, PDP) against a model-agnostic `PredictionProvider` interface. This design allows the core algorithms to work with any model serving infrastructure that can be wrapped behind the async `predictAsync` contract. The `explainability-connectors` module bridges this abstraction to the KServe ecosystem, providing HTTP and gRPC clients for both v1 (simple instance-based) and v2 (tensor-based) inference protocols, with protobuf definitions for the v2 gRPC service. The `explainability-arrow` module adds Apache Arrow columnar data format support for efficient bulk data interchange.

The service layer is a Quarkus 3.8.5 application using RESTEasy Reactive for REST endpoints, Quarkus Arc for CDI-based dependency injection, and a sophisticated pluggable storage architecture. Storage backends (PVC, MariaDB/H2 via Hibernate ORM, MinIO, in-memory) are selected at runtime via the `service.storage-format` configuration property using Quarkus `@LookupIfProperty` conditional bean resolution. This means only one storage implementation is ever active per deployment. The DATABASE mode includes an automatic migration path from PVC storage via `HibernateMigrator`, which runs asynchronously on startup and migrates existing flat-file data into the relational schema. This migration is event-driven using CDI async events (`MigrationEvent`).

A notable architectural pattern is the **dual ingestion path** for inference data. The primary path uses a direct HTTP endpoint (`/consumer/kserve/v2`) where ModelMesh sends inference request/response payloads as `InferencePartialPayload` objects — this is configured by the init container that writes a `model-serving-config` ConfigMap pointing ModelMesh at the TrustyAI consumer URL. The secondary path uses **Knative CloudEvents** via the Quarkus Funqy extension, where KServe InferenceServices emit `org.kubeflow.serving.inference.request` and `org.kubeflow.serving.inference.response` events. Both paths feed into `KServeInferencePayloadReconciler` which matches partial payloads (request ↔ response) before persisting complete observations.

The metrics system uses a two-tier approach: users schedule recurring metric computations (SPD, DIR, drift tests) via POST to `/metrics/*/request` endpoints, and the `PrometheusScheduler` (`@Scheduled` at configurable intervals, default 5s) iterates over all registered requests, computes metric values from stored observation data, and publishes results as Micrometer gauges with the `trustyai_` prefix. The `ServiceMonitor` CR then tells Prometheus to scrape these gauges, filtering for `trustyai_spd` and `trustyai_dir` patterns. This design decouples metric computation frequency from Prometheus scrape intervals.

The **RHOAI vs ODH endpoint differentiation** is implemented via Quarkus profiles and feature flags in `application.properties`. The `%rhoai` profile disables all explainer endpoints (LIME, SHAP, Counterfactual, TSSaliency, global explainers) while keeping fairness and drift endpoints enabled. Each endpoint class checks its corresponding feature flag at startup via `@IfBuildProperty` or runtime configuration. This allows the same codebase and container image to serve both distributions with profile-based configuration.

The Konflux build pipeline (`Dockerfile.konflux`) uses an interesting two-stage build where the Java application is **not built from source in Konflux**. Instead, PNC (Red Hat's internal build system) compiles the Java artifact, and the resulting zip is referenced in `artifacts.lock.yaml` and prefetched via Hermeto (cachi2) into the Konflux build context. The Dockerfile simply extracts the pre-built Quarkus app from the zip and layers it into a UBI9 `openjdk-17-runtime` base image. This is architecturally significant because it means Java dependency resolution and compilation happen in PNC's controlled environment, while Konflux handles only the container image assembly with locked RPM dependencies.
