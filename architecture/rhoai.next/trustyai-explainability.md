## Purpose

**Short**: TrustyAI is a Quarkus-based service that provides model fairness monitoring, data drift detection, and explainability (LIME, SHAP, counterfactual) for ML models served via KServe and ModelMesh on OpenShift AI.

**Detailed**: TrustyAI Explainability is a multi-module Java library and service that integrates with the Red Hat OpenShift AI platform to provide Responsible AI capabilities. It receives inference data (inputs and outputs) from ML models via CloudEvents (Knative) or REST endpoints, stores the data in configurable backends (PVC filesystem, MariaDB/MySQL via Hibernate, or MinIO S3-compatible storage), and continuously computes fairness metrics (Statistical Parity Difference, Disparate Impact Ratio) and data drift metrics (Kolmogorov-Smirnov test, Fourier MMD, Meanshift, Jensen-Shannon divergence) on a configurable schedule.

The service exposes computed metrics as Prometheus gauges (prefixed `trustyai_`) that are scraped by the platform's monitoring stack. It also provides on-demand local explainability (LIME, SHAP, counterfactual, time-series saliency) and global explainability (Partial Dependence Plots, aggregated LIME) endpoints that query models via KServe v2 gRPC to generate explanations for individual predictions or model behavior.

The project is organized as a Maven multi-module build: `explainability-core` (algorithms and data model), `explainability-connectors` (KServe v1/v2 gRPC client, protobuf definitions), `explainability-arrow` (Apache Arrow serialization), and `explainability-service` (Quarkus REST service, storage backends, Prometheus integration). The RHOAI distribution disables local explainer and global explainer endpoints, exposing only fairness and drift metrics endpoints.

## Architectural Analysis

TrustyAI's architecture follows a classic ingestion-storage-computation pipeline pattern. Inference data enters the system through two paths: CloudEvents from Knative (for KServe-deployed models) and REST POST from ModelMesh's payload processor. Both paths converge at the payload reconciler, which matches input/output halves by inference ID before persisting complete prediction records. The reconciler uses `synchronized` methods as a concurrency control mechanism — a simple but effective approach given that each TrustyAI instance handles a single namespace.

The storage abstraction is well-designed with four backends (PVC, MinIO, Hibernate, Memory) behind a common interface, but the backends have significantly different capability profiles. PVC supports batched reads and tag/ID filtering; Hibernate provides full query capabilities with JPA; MinIO has no filtering or batch support at all and reads entire objects. This means the service's behavior varies meaningfully depending on the storage backend, which could surprise operators. The Hibernate backend is the recommended and most capable option, and the service includes a migration pathway from PVC to database storage via the `HibernateMigrator` component.

A notable architectural decision is the feature-flag system for endpoints. The RHOAI Quarkus profile (`%rhoai`) disables all local explainer endpoints (LIME, SHAP, counterfactual, TSSaliency) and global explainer endpoints, exposing only fairness and drift metrics. This means RHOAI customers get monitoring capabilities but not on-demand explainability. The ODH profile enables everything. This bifurcation is implemented cleanly via Quarkus configuration profiles in `application.properties` rather than code-level conditionals.

The Prometheus integration deserves attention: the `PrometheusScheduler` runs on a configurable interval (default 5s in manifests) and iterates over all registered metric requests. For each request, it loads data from the storage backend, computes the metric, and publishes a gauge via Micrometer. With many models and metrics, this polling pattern could create significant database load. The scheduler determines the maximum batch size across all metric requests for a given model and loads data once, then tails the dataframe per-request — an optimization that reduces redundant data reads.

The Konflux build pipeline is noteworthy: the Java application is NOT built inside the Dockerfile. Instead, PNC (Project Newcastle) pre-builds the application and produces a ZIP artifact pinned in `artifacts.lock.yaml`. The Dockerfile.konflux simply extracts the pre-built Quarkus application from the PNC artifact using Hermeto's generic prefetch mechanism. This separates Java dependency resolution (PNC's domain) from container image construction (Konflux's domain), providing better supply chain control but making the build pipeline less self-contained. The build supports four architectures: x86_64, arm64, ppc64le, and s390x.
