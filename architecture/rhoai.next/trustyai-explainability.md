## Purpose

**Short**: TrustyAI Explainability is a Quarkus-based Java service that provides model fairness/bias monitoring, data drift detection, and explainability (LIME, SHAP, counterfactual) for ML models served by KServe and ModelMesh on the RHOAI platform.

**Detailed**: TrustyAI Explainability is a core component of Red Hat OpenShift AI that enables responsible AI practices by providing runtime monitoring and explanation capabilities for deployed machine learning models. The service consumes inference payloads (inputs and outputs) from KServe and ModelMesh model serving platforms via CloudEvents and REST endpoints, stores them in a pluggable storage backend (PVC, MinIO, or MariaDB/MySQL database), and computes fairness metrics (Statistical Parity Difference, Disparate Impact Ratio), data drift metrics (Meanshift, Fourier MMD), and identity metrics on scheduled intervals. These metrics are exposed via Prometheus for integration with OpenShift monitoring.

The service also provides on-demand local and global explainability capabilities including LIME (Local Interpretable Model-agnostic Explanations), SHAP (SHapley Additive exPlanations), counterfactual explanations, time series saliency, aggregated LIME, and partial dependence plots. The explainability endpoints communicate with the inference servers via KServe v1/v2 HTTP and v2 gRPC protocols to generate explanations. In the RHOAI distribution, local explainer and global explainer endpoints are disabled by default (feature-flagged off), while fairness and drift metrics remain enabled.

The project is structured as a Maven multi-module build with four main modules: `explainability-core` (the XAI library with algorithms), `explainability-connectors` (KServe/ModelMesh protocol integration including gRPC stubs), `explainability-arrow` (Apache Arrow data format support), and `explainability-service` (the Quarkus REST service that ties everything together). The service is deployed as a single-replica Kubernetes Deployment per monitored namespace, with an init container that dynamically registers the TrustyAI payload processor endpoint in the `model-serving-config` ConfigMap.

## Architectural Analysis

TrustyAI Explainability follows a **pluggable storage architecture** that cleanly separates the data ingestion, metric computation, and explainability layers from the underlying storage mechanism. The `StorageInterface` and `DataSource` abstractions allow transparent switching between PVC-based CSV files (default), MinIO S3 object storage, and Hibernate-managed relational databases (MariaDB/MySQL) via a single configuration property (`service.storage-format`). This design enables deployment flexibility: simple single-node setups use PVC storage, while production deployments requiring durability and scalability can switch to database-backed storage with migration support from PVC. The `HibernateMigrator` handles online migration between storage backends, publishing async events to coordinate the transition.

The **inference payload reconciliation pattern** is a key architectural element. Since inference payloads arrive as separate input and output messages (either via CloudEvents or REST), the `InferencePayloadReconciler` maintains an in-memory matching buffer that pairs inputs with outputs by correlation ID before persisting them as complete dataframe rows. This pattern has two implementations — `KServeInferencePayloadReconciler` for CloudEvents-based KServe payloads and `ModelMeshInferencePayloadReconciler` for REST-based ModelMesh payloads — each handling the protocol-specific deserialization before delegating to the common reconciliation logic. The ModelMesh reconciler also performs synthetic data detection (via the `BIAS_IGNORE_PARAM` parameter) to tag and filter out test/synthetic inferences from fairness calculations.

The **metrics scheduling system** uses Quarkus's `@Scheduled` mechanism with a configurable cron expression (defaulting to 5 seconds) to periodically recalculate all registered metric requests. Each metric request is identified by UUID and stored in a `ConcurrentHashMap`. The scheduler iterates over all registered requests, computes the metric against the current dataframe (with configurable batch windowing), and publishes the result as a Micrometer gauge prefixed with `trustyai_`. The ServiceMonitor is configured to scrape only `trustyai_spd` and `trustyai_dir` metrics, though the scheduler can publish any registered metric type. This 4-second scrape interval with a 5-second calculation interval means Prometheus will occasionally see stale values, which is acceptable for fairness monitoring use cases.

The **feature flag system** uses Quarkus profiles (`%odh.` and `%rhoai.`) to control which endpoint categories are enabled. In the RHOAI distribution, local explainers (LIME, SHAP, counterfactual, time series saliency) and global explainers (aggregated LIME, partial dependence plots) are disabled, leaving only fairness and drift metrics active. This reflects a product decision to limit the API surface in the supported RHOAI distribution while keeping the full capability available for ODH deployments. The data download endpoint is also gated, enabled only in test profiles.

The **Konflux build pipeline** is notable in that it does not compile the Java source code — instead, it consumes a pre-built artifact from PNC (Red Hat's managed build system) via `artifacts.lock.yaml`. The Dockerfile.konflux unpacks the pre-built Quarkus application from a cachi2/Hermeto generic dependency and copies the layered Quarkus JAR structure into the runtime image. This delegates Java build hermeticity and reproducibility to the PNC pipeline while Konflux handles only container image construction, RPM locking (for the `unzip` package), and multi-architecture support (x86_64, aarch64, ppc64le, s390x).
