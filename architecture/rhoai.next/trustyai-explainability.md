## Purpose

**Short**: TrustyAI Explainability is a Java/Quarkus service that provides AI fairness metrics, model explainability algorithms, and data drift detection for ML models served via KServe and ModelMesh on OpenShift AI.

**Detailed**: TrustyAI Explainability is a multi-module Maven project that produces a containerized Quarkus REST service for monitoring and explaining ML model behavior in production. The service integrates with KServe (v1/v2 HTTP and gRPC) and ModelMesh inference platforms to automatically ingest model inference data (input/output pairs), store it in configurable backends (PVC filesystem, MinIO/S3, MariaDB/MySQL database, or in-memory), and compute fairness metrics (Statistical Parity Difference, Disparate Impact Ratio), drift detection metrics (Kolmogorov-Smirnov, Fourier MMD, Meanshift), and local/global explanations (LIME, SHAP, Counterfactual, Time Series Saliency, Partial Dependence Plots).

The service exposes computed metrics to Prometheus via Micrometer gauges (prefixed `trustyai_*`), enabling dashboard visualization and alerting on model bias and data drift. It supports Kubeflow CloudEvents for serverless inference logging. The RHOAI distribution enables only fairness and drift endpoints, while ODH enables the full set including explainer endpoints. The component is deployed by the TrustyAI Operator (`trustyai-service-operator`) which manages per-namespace TrustyAI service instances alongside model serving infrastructure.

## Architectural Analysis

TrustyAI Explainability follows a well-decomposed multi-module Maven architecture where the core algorithms (`explainability-core`) are cleanly separated from the deployment runtime (`explainability-service`) and the inference protocol connectors (`explainability-connectors`). This separation allows the core library to be consumed independently (published to Maven Central) for non-Kubernetes use cases, while the service module adds Quarkus-specific CDI wiring, REST endpoints, storage management, and Prometheus integration. The `explainability-arrow` module provides an Apache Arrow bridge specifically for Python interoperability via the TrustyAI Python library, though this module is not included in the `service-minimal` build profile used for RHOAI container builds.

The storage backend pattern is a notable architectural choice. The service supports four pluggable storage backends (PVC filesystem, MinIO/S3, Hibernate/database, and in-memory) selected at startup via the `service.storage-format` configuration property using Quarkus CDI's `@LookupIfProperty` conditional bean selection. The PVC backend is the default and most common deployment mode, storing inference data as CSV files on a Kubernetes PersistentVolumeClaim. However, the MinIO and database backends have different capability profiles — MinIO lacks batch limiting, sliced reads, and tag-based filtering, making it less suitable for large-scale deployments that rely on these features. The database backend offers the richest feature set with full JPA/Hibernate querying capabilities including tag and ID filtering with `setMaxResults()` pagination.

The inference data ingestion pipeline uses a two-phase reconciliation pattern: partial payloads (input and output tensors) arrive separately and are stored in `PayloadStorage` (memory-backed or Hibernate-backed), then matched by inference ID in the `PayloadReconciler` to produce complete `Dataframe` records. This accommodates the asynchronous nature of KServe and ModelMesh payload processing, where request and response payloads may arrive at different times. The reconciler also handles synthetic data tagging — if a KServe inference includes a `bias.ignore` parameter, the data is tagged as `SYNTHETIC` and excluded from organic metric calculations, preventing test/calibration data from skewing fairness measurements.

The RHOAI distribution notably disables explainer endpoints (LIME, SHAP, Counterfactual, TSSaliency, PDP, Aggregated LIME) via Quarkus profile configuration (`%rhoai.endpoints.explainers.*=disable`), limiting the API surface to fairness metrics and drift detection only. This is a deliberate product scoping decision — the explainability algorithms require direct model inference access and can be computationally expensive, while fairness/drift monitoring operates passively on stored inference data. The recent addition of a `GzipRequestFilter` (scoped to `/data/upload` only) addresses a specific integration issue where the KServe agent sidecar in RawDeployment mode automatically gzip-compresses CloudEvent payloads sent to TrustyAI.

The Konflux build pipeline is notably different from typical Go-based components in the platform: rather than building from source in the container, the Dockerfile.konflux extracts a pre-built Maven artifact (produced by Red Hat's PNC/Indy build system) from the Hermeto generic prefetch cache. This artifact zip contains the pre-compiled Quarkus application JAR and its dependencies, which are simply copied into the UBI 9 OpenJDK 17 runtime image. This approach separates the Java build (PNC) from the container image build (Konflux), allowing each to enforce its own reproducibility and security constraints.
