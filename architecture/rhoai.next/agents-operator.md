## Purpose

**Short**: Kubernetes operator that automates deployment, discovery, identity, mutual authentication, and observability of AI agents and tools via Custom Resources.

**Detailed**: The Kagenti Operator (shipped as `agents-operator` in RHOAI) manages the lifecycle of AI agents running on Kubernetes. It provides a declarative API via `AgentRuntime` and `AgentCard` CRDs that enroll workloads into a platform of security and observability services. When a user creates an `AgentRuntime` CR pointing at a Deployment, StatefulSet, or Sandbox workload, the operator applies labels, computes a config hash, and triggers rolling updates. A mutating admission webhook intercepts Pod CREATE events and injects AuthBridge sidecar containers (envoy-proxy, authbridge-proxy, or authbridge-lite depending on mode) along with SPIRE identity volumes and proxy-init iptables containers.

The operator also provides agent discovery through the A2A protocol: the `AgentCardSync` controller auto-creates `AgentCard` CRs for labeled workloads, and the `AgentCard` controller periodically fetches agent card metadata from running agents, verifies JWS signatures using SPIRE X.509 trust bundles, evaluates SPIFFE identity binding, and optionally verifies Sigstore supply-chain attestations. NetworkPolicy enforcement, MLflow experiment tracking integration, Keycloak OAuth2 client registration, TLS bridge CA provisioning (via cert-manager), Kuadrant API gateway integration, and Istio ambient mesh enrollment are provided by additional controllers. A dedicated Bundle Service serves OPA authorization policy bundles to AuthBridge clients.

| Component | Type | Purpose |
|-----------|------|---------|
| kagenti-operator (manager) | Go Operator (controller-runtime) | Main operator binary managing AgentRuntime, AgentCard, NetworkPolicy, MLflow, Keycloak client registration, SharedTrust, TLS Bridge CA, AuthBridge config, Kuadrant, and TektonConfig controllers; hosts mutating and validating admission webhooks |

## Architectural Analysis

The Kagenti Operator follows a **multi-controller composition pattern** where the main binary hosts 10+ controllers and 3 webhooks, each activated conditionally based on CRD availability (cert-manager, Kuadrant, TektonConfig, DataScienceCluster, Sandbox) and feature flags. This design keeps the operator as a single binary with a wide surface area that gracefully degrades when optional dependencies are absent. The `SetupWithManager` calls in `cmd/main.go` check for CRD existence via API discovery before registering controllers, avoiding crash loops on clusters without optional CRDs.

The **AuthBridge sidecar injection webhook** is the operator's most complex component, implementing a 4-mode injection system (proxy-sidecar, envoy-sidecar, lite, waypoint) with a multi-layer configuration resolution chain: feature gates → namespace ConfigMap → workload labels → AgentRuntime CR. The webhook performs "port stealing" in proxy-sidecar/lite modes — it relocates the agent's original port to a new one and inserts the authbridge-proxy reverse proxy on the original port, making the proxy transparent to external callers. The inject decision flows through a precedence evaluator that supports global kill switches, per-sidecar feature gates, and per-workload label opt-outs. Reinvocation policy (`IfNeeded`) handles the case where the Keycloak client credentials Secret doesn't exist at initial admission but is created by the `ClientRegistration` controller shortly after.

The **mTLS transport security model** implements a three-mode progression (disabled → permissive → strict) for the proxy-sidecar and lite paths. When mTLS is non-disabled, SPIRE is automatically enabled: the webhook injects a CSI volume for the SPIRE workload API socket, creates a dedicated ServiceAccount (so the SPIFFE ID maps to the workload name), and configures the bundled spiffe-helper to write SVIDs to an emptyDir. The operator proactively triggers pod restarts before SVID expiry via a configurable grace period. The envoy-sidecar path uses Envoy's native TLS inspector with filter chains — mTLS strict mode adds a `DownstreamTlsContext` requiring client certificates validated against the SPIRE trust bundle.

The **TLS Bridge** feature (proxy-sidecar/lite only, gated by `tlsBridgeMode=enabled`) provides HTTPS interception for agent egress. The operator provisions a per-agent CA via cert-manager (self-signed Issuer → CA Certificate), mounts the keypair into the authbridge-proxy sidecar (which mints ephemeral leaf certificates at runtime), and injects CA trust environment variables (`SSL_CERT_FILE`, `NODE_EXTRA_CA_CERTS`, `REQUESTS_CA_BUNDLE`, etc.) into all application containers. This allows authbridge to decrypt, inspect, and re-encrypt HTTPS traffic without modifying application code.

The **trust evaluation chain** for AgentCards involves multiple verification mechanisms that compose together: (1) mTLS-authenticated fetch via SPIFFE extracts the attested agent SPIFFE ID from the peer certificate, (2) JWS x5c signature verification validates the card's cryptographic signature against the SPIRE trust bundle, (3) SPIFFE identity binding validates that the SPIFFE ID belongs to the configured trust domain, and (4) Sigstore bundle verification validates supply-chain attestations via Rekor transparency log entries. Each mechanism sets independent status conditions, and the `Verified` condition is a composite that takes the most authoritative result (mTLS attestation takes precedence over JWS). The NetworkPolicy controller uses the `Verified` condition to apply permissive (allow all) or restrictive (API server only) policies.

The **bundle service** is architecturally separate from the operator — it's a standalone Go HTTP server that watches `AuthorizationPolicy` CRs and builds OPA authorization policy bundles served to AuthBridge clients. It uses ETag-based caching (304 Not Modified) for efficient bundle distribution. The service is deployed via static manifests in `config/bundleservice/` rather than the Helm chart, targeting SRE operational use.
