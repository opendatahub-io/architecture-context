## Purpose

**Short**: A central repository and REST API for ML model developers to store, manage, and retrieve metadata about models, versions, artifacts, experiments, and serving environments.

**Detailed**: The Model Registry provides a metadata management layer for the AI/ML lifecycle in the OpenShift AI platform. It implements the Kubeflow Model Registry specification, offering a REST API (OpenAPI v1alpha3) for managing registered models, model versions, model artifacts, inference services, serving environments, experiments, and experiment runs. The service uses an embedded metadata store (EmbedMD) backed by PostgreSQL or MySQL, replacing the original C++ gRPC ml-metadata service with a Go-based implementation.

The repository is a multi-component monorepo producing five distinct deployable artifacts: (1) the core Model Registry REST proxy server, (2) a Model Catalog server for browsing curated model and MCP server catalogs from sources like HuggingFace, (3) a Kubernetes controller that synchronizes KServe InferenceService CRs with Model Registry metadata, (4) a CSI storage initializer that resolves `model-registry://` URIs for KServe model downloads, and (5) an async upload Job image for copying models between storage backends (S3, OCI, HuggingFace) with optional Sigstore signing. The repository also includes a React/TypeScript frontend UI with a Go BFF (Backend-For-Frontend) and Python client libraries, though these are shipped via separate build pipelines.

## Architectural Analysis

The Model Registry is a multi-component monorepo following a "hub and satellites" architecture pattern. The central hub is the Go-based REST proxy server that implements the Kubeflow Model Registry OpenAPI v1alpha3 specification, while satellite components (controller, CSI, async-upload job, UI) integrate with it via HTTP REST calls. This design provides clean separation of concerns — the proxy server owns the data model and persistence layer, while each satellite handles a specific integration concern.

The embedded metadata store (EmbedMD) is a notable architectural decision. Rather than depending on the original ml-metadata C++ gRPC service, the team reimplemented the metadata store in Go using GORM as the ORM layer, supporting both PostgreSQL and MySQL backends. This eliminates the gRPC dependency, simplifies deployment (single binary), and enables direct SQL database access with configurable TLS. The database schema uses a type system with runtime type registration (TypeMap), allowing extensible entity types without schema changes. Database migrations are embedded in the binary via Go's embed package and managed by golang-migrate.

The proxy server uses a "dynamic router" pattern for graceful startup — it begins serving HTTP immediately with a 503 handler while establishing the database connection in a background goroutine. Once the connection is ready, the router atomically swaps in the real OpenAPI handler. This ensures Kubernetes probes can reach the server during initialization while preventing premature traffic. Two distinct health endpoints serve different purposes: `/readyz/isDirty` checks only the database migration state (used for liveness), while `/readyz/health` performs a comprehensive check across all entity types (used for readiness).

The Model Catalog is a plugin-based system with leader election via PostgreSQL advisory locks (cirello.io/pglock). This allows multiple catalog server replicas while ensuring only one actively syncs from external sources like HuggingFace. The plugin architecture supports two catalog types — model catalogs and MCP (Model Context Protocol) server catalogs — each with their own loader, filter, and service layers. Source configurations are loaded from YAML files and monitored for changes via file watchers, allowing runtime reconfiguration without restarts.

The async-upload job represents a significant operational capability — it handles the complex workflow of downloading models from various sources (S3, OCI registries, HuggingFace Hub, HTTP URIs), optionally signing them with Sigstore, uploading to destination storage, and registering/updating the model artifact in the Model Registry. The Pydantic-based configuration system supports three intent types (update_artifact, create_model, create_version) with fast-fail validation before expensive I/O operations. This is the only Python component shipped via Konflux, and it uses the model-registry Python SDK internally.

Security at the service level relies on Istio for transport encryption (mTLS between pods) and access control (AuthorizationPolicy). The AuthorizationPolicy has two rules: one allowing traffic from the Istio ingress gateway (where users are already authenticated), and another allowing internal cluster traffic with a valid Kubernetes JWT but explicitly blocking kubeflow-userid header spoofing — a defense-in-depth measure against identity impersonation within the mesh. The controller uses a separate authentication path via ServiceAccount bearer tokens for its REST calls to the Model Registry API.
