## Purpose

**Short**: A central metadata repository for ML models, versions, artifacts, experiments, serving environments, and catalog browsing, providing REST APIs for model lifecycle management.

**Detailed**: The Model Registry (upstream: Kubeflow Hub) is a multi-component Go service that provides a REST API for recording and retrieving metadata associated with ML developer and data scientist workflows. It stores registered models, model versions, model artifacts, serving environments, inference services, experiments, and experiment runs in a relational database (MySQL or PostgreSQL) using an embedded metadata (EmbedMD) datastore pattern built on GORM.

The repository produces multiple deployable components: (1) the primary REST API proxy server exposing the Model Registry Service API at `/api/model_registry/v1alpha3`, (2) a catalog server providing model and MCP server browsing APIs at `/api/model_catalog/v1alpha1` and `/api/mcp_catalog/v1alpha1`, (3) an InferenceService controller that watches KServe InferenceService CRs and synchronizes them with model registry metadata, (4) a CSI storage initializer that resolves `model-registry://` URIs for KServe model serving, and (5) a Python-based Kubernetes Job for asynchronous model upload between storage backends (S3, OCI registries, HTTP/HuggingFace) with optional Sigstore signing. In RHOAI, the platform operator deploys instances of the model registry per namespace, with the controller and CSI components enabling integration with KServe inference serving.

## Architectural Analysis

The Model Registry follows a hub-and-satellite architecture where the proxy server is the central metadata store, and multiple satellite components (controller, CSI, catalog, async-upload) integrate with it via its REST API. The proxy server itself is a straightforward Go HTTP service using the chi router framework with OpenAPI-generated handlers, backed by GORM for database operations. The codebase uses a clean separation between the OpenAPI layer (`internal/server/openapi/`), the core business logic (`internal/core/`), the database models (`internal/db/models/`), and the datastore abstraction (`internal/platform/datastore/`). The "EmbedMD" datastore pattern is an embedded metadata store that replaces the original MLMD gRPC service with a direct database connection, supporting both MySQL and PostgreSQL.

The catalog server is a notable architectural addition that introduces a plugin system for extensible catalog sources. It uses PostgreSQL-based distributed leader election (via `cirello.io/pglock`) to ensure only one replica writes to the database while all replicas serve read requests. The plugin architecture supports hot-reloading of configuration files via filesystem watching, and the two built-in plugins (model catalog and MCP catalog) provide independent API surfaces mounted under different base paths. The RHOAI overlay adds a performance metrics init container that ships pre-computed benchmark data from a separate image.

The InferenceService controller is a lightweight controller-runtime operator that bridges KServe and Model Registry. It watches KServe InferenceService CRs for specific labels (`modelregistry.kubeflow.org/*`) and creates/updates corresponding InferenceService records in the model registry. The controller uses finalizers to update the model registry when ISVCs are deleted (setting desired state to UNDEPLOYED). The controller discovers model registry service endpoints dynamically by listing Services with the `component: model-registry` label, making it work across multiple model registry instances.

The TLS configuration is well-designed for FIPS compliance: the proxy server only allows FIPS-approved cipher suites from an explicit allowlist (no insecure ciphers), enforces TLS 1.2 minimum, and supports configurable server certificate verification. However, the controller has an `InsecureSkipVerify` option (disabled by default) that could be a concern in hardened environments. The async-upload job adds cryptographic complexity through Sigstore integration (model signing and verification) using the `rh-model-signing` and `sigstore` Python packages.

The kustomize manifest structure is modular with a base deployment and separate optional directories for the controller, catalog, UI, CSI, and Istio integration. Distribution-specific overlays (ODH vs RHOAI) primarily differ in catalog source configuration and the inclusion of performance benchmark data. The Istio integration provides mTLS between services and an AuthorizationPolicy that allows traffic from the Istio ingress gateway (authenticated users) and internal services with valid Kubernetes JWTs, while preventing `kubeflow-userid` header spoofing.
