## Purpose

**Short**: A central REST API service for recording and retrieving metadata associated with ML models, versions, artifacts, experiments, and inference services.

**Detailed**: The Model Registry (also known as Kubeflow Hub) provides a comprehensive REST API for managing ML model lifecycle metadata. It serves as the system of record for registered models, model versions, model artifacts (including documentation artifacts, datasets, metrics, and parameters), serving environments, inference services, and experiments with experiment runs. The service exposes a v1alpha3 OpenAPI-based REST API backed by either MySQL or PostgreSQL databases via the GORM ORM, using an embedded metadata (EmbedMD) datastore layer derived from the MLMD (ML Metadata) schema.

The repository also produces several auxiliary components: a Model Catalog server for curated model discovery with MCP (Model Context Protocol) server support, a Kubernetes controller that synchronizes KServe InferenceService resources with the Model Registry API, a CSI storage initializer plugin that enables KServe to download models indexed in the registry, and an async upload Kubernetes Job for transferring model artifacts between storage backends. A web UI frontend with a Go-based BFF (Backend for Frontend) provides a user interface for browsing and managing models. Together these components form the model management subsystem of the RHOAI/ODH platform.

## Architectural Analysis

The Model Registry follows a clean layered architecture with clear separation of concerns. The core service layer (`internal/core/`) implements business logic through a `ModelRegistryService` struct that depends on 14 repository interfaces, one per entity type. This design enables testability and allows the persistence layer to be swapped without affecting business logic. The OpenAPI-generated HTTP handler layer (`internal/server/openapi/`) delegates to a service implementation that maps between OpenAPI types and internal domain models. The EmbedMD datastore connector (`internal/datastore/embedmd/`) bridges the service layer to the GORM ORM, with database-specific drivers for MySQL and PostgreSQL. Notably, the database schema derives from the ML Metadata (MLMD) project, using generic tables (`context`, `artifact`, `execution`, `type`, `*_property`) with a type system rather than domain-specific tables — an unusual design that provides schema flexibility at the cost of query complexity.

The repository produces multiple deployable components from a single codebase, reflecting a "mono-repo with multiple entry points" pattern. The main binary (`main.go → cmd/proxy.go`) runs the REST API server, while `cmd/controller/main.go` runs the InferenceService controller as a separate process. The controller is conditionally enabled via the `INFERENCE_SERVICE_CONTROLLER=managed` environment variable, allowing the same binary to serve API-only or API+controller modes. The CSI storage initializer (`cmd/csi/main.go`) is a standalone CLI tool that implements the KServe storage provider interface for `model-registry://` URIs. This architectural decision means model download requests are resolved through a chain: KServe invokes the storage initializer → it queries Model Registry for the artifact URI → it delegates to the appropriate KServe provider (S3, HTTP, GCS) for actual download.

The experiment tracking subsystem (experiments, experiment runs, metric history) is a significant addition to the original model registry concept. It implements an automatic metric history mechanism where each metric artifact upserted to an experiment run also creates a timestamped copy in a `MetricHistory` table, enabling time-series tracking of experiment metrics. The timestamp-suffixed naming scheme (`{experimentRunId}:{metricName}__{timestamp}`) ensures uniqueness while the retrieval logic strips the suffix for clean display. The HuggingFace model validation in `serve_model.go` prevents deployment of gated or private models, a safeguard specific to the RHOAI/ODH ecosystem.

The async upload job is a Python-based component that handles model artifact transfers between storage backends. It supports multiple upload intents (update_artifact, create_model, create_version) and integrates with Sigstore for model signing. The job uses a priority-based configuration system (CLI args > env vars > credential files > defaults) and writes results to the container's termination-log for Kubernetes Job status reporting. The RHOAI overlay adds performance benchmark data via an init container that copies pre-computed model performance metrics into the catalog server.

Security is primarily delegated to infrastructure: Istio mTLS for in-mesh traffic, Istio AuthorizationPolicy for access control, and database-level authentication via Secrets. The service itself has no authentication logic — it relies on the mesh and network policies. The TLS configuration module (`internal/platform/tls/`) provides comprehensive TLS 1.2+ support for database connections with configurable cipher suites and insecure cipher rejection. The validation middleware protects against null byte injection attacks in query parameters and request bodies, but there is no rate limiting or API key authentication at the application level.
