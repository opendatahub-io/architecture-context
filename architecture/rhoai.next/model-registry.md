## Purpose

**Short**: A REST API service and supporting components for storing, managing, and serving ML model metadata, versions, and artifacts.

**Detailed**: Model Registry is the central metadata repository for the Kubeflow/RHOAI platform, providing a REST API (OpenAPI v1alpha3) for ML model developers and data scientists to register, version, and track machine learning models and their associated artifacts. It stores metadata about registered models, model versions, model artifacts, serving environments, inference services, experiments, and experiment runs in a relational database (MySQL or PostgreSQL).

The repository produces multiple deployable components: (1) the primary REST API proxy server that exposes the OpenAPI-defined endpoints on port 8080; (2) a Kubernetes controller that synchronizes KServe InferenceService CRDs with model registry metadata; (3) a CSI storage initializer that enables KServe to download model artifacts indexed in the registry; (4) a Python-based async upload Kubernetes Job for transferring models between storage backends (S3, OCI, HTTP) and registering them; and (5) a catalog sub-command that provides a unified REST API for querying models from multiple sources including HuggingFace and MCP (Model Context Protocol) servers.

The service uses the EmbedMD datastore pattern — an embedded metadata store backed by GORM that implements the MLMD (ML Metadata) type system with types like `kf.RegisteredModel`, `kf.ModelVersion`, `kf.ModelArtifact`, etc. Database migrations are managed automatically on startup, and the service supports both MySQL 8.3 and PostgreSQL 16+ as backends.

_The Konflux Dockerfile builds with full FIPS compliance flags (CGO_ENABLED=1, GOEXPERIMENT=strictfipsruntime, -tags strictfipsruntime). The application-level TLS is configurable with MinVersion TLS 1.2, an explicit allowlist of FIPS-approved cipher suites, and rejection of insecure ciphers. However, InsecureSkipVerify defaults to true for database connections unless explicitly overridden, which is a deployment-level concern rather than a crypto correctness issue. The async-upload job (Python) inherits OpenSSL from the UBI9 base image; its `cryptography==46.0.7` dependency uses OpenSSL bindings._

## Architectural Analysis

The Model Registry follows a multi-component architecture where the primary REST API server (`model-registry proxy`) is a Go service built on the chi router with OpenAPI-generated handlers. The service uses a "dynamic router" pattern (`internal/proxy/dynamic_router.go`) that starts serving 503 responses immediately while the database connection is established asynchronously in a background goroutine. This design ensures the HTTP server is always reachable for Kubernetes probes even during slow database startups — a practical pattern for environments with database connection retry loops (25 attempts with incremental backoff up to ~5 minutes).

The data layer uses an "EmbedMD" (Embedded Metadata) pattern that implements the MLMD (ML Metadata) type system on top of GORM. Rather than using Google's ml-metadata C++ gRPC server, this Go implementation embeds the metadata store directly, maintaining the same type taxonomy (`kf.RegisteredModel`, `kf.ModelVersion`, `kf.ModelArtifact`, etc.) but backing it with standard SQL databases. The repository pattern is generic — `GenericRepository[Entity, Schema, Properties, ListOptions]` — with concrete implementations for each entity type that provide mapping functions between OpenAPI models and database schema. This is a significant architecture choice: it eliminates the ml-metadata dependency while preserving API compatibility.

The controller sub-component demonstrates a clean separation of concerns. The `InferenceServiceReconciler` in `cmd/controller/` is a thin wrapper that delegates all logic to `pkg/inferenceservice-controller/controller.go`. The controller performs service discovery to find model registry instances — searching for services labeled `component: model-registry` with a port named `http-api`. This is a runtime discovery pattern rather than static configuration, enabling multi-registry deployments. The controller is gated by the `INFERENCE_SERVICE_CONTROLLER=managed` environment variable, so the same binary can be deployed with or without the controller enabled.

The TLS configuration (`internal/platform/tls/config.go`) is noteworthy for its explicit cipher suite allowlist — it maintains a map of 13 FIPS-approved cipher suites and actively rejects any cipher from Go's `InsecureCipherSuites()` list. However, the default `InsecureSkipVerify: !c.VerifyServerCert` setting means database connections skip TLS verification unless explicitly enabled, which is a deployment hardening concern. The Konflux Dockerfile builds with full FIPS flags (`GOEXPERIMENT=strictfipsruntime`, `CGO_ENABLED=1`, `-tags strictfipsruntime`), ensuring the binary uses OpenSSL for cryptographic operations.

The catalog sub-command represents a newer architectural addition — it introduces a PostgreSQL-only datastore (separate from the MySQL/PostgreSQL choice for the main registry), leader election for multi-replica coordination, file-watching for configuration hot-reload, and support for MCP (Model Context Protocol) servers. The catalog and the main registry share the same Go binary but operate independently with different database backends and API namespaces (`/api/model_catalog/` vs `/api/model_registry/`).
