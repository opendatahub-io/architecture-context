## Purpose

**Short**: A central repository for ML model developers to store and manage models, versions, artifacts, and serving metadata.

**Detailed**: Model Registry (formerly Kubeflow Model Registry, now part of Kubeflow Hub) provides a REST API for recording and retrieving metadata associated with ML developer and data scientist workflows. It stores registered models, model versions, model artifacts, serving environments, inference services, experiments, and experiment runs. The system supports multiple database backends (MySQL and PostgreSQL) and provides an OpenAPI-first REST interface.

The component consists of several sub-components: the core REST API proxy server (Go), a Kubernetes controller that reconciles KServe InferenceService CRs with model registry metadata, a Backend-For-Frontend (BFF) Go server that proxies requests from the React UI to the model registry and catalog APIs, a Model Catalog service that aggregates model metadata from multiple sources (including Hugging Face and internal databases), a CSI-compatible storage initializer for KServe that resolves `model-registry://` URIs to actual storage locations, and an async upload job (Python) that handles model file transfers between storage backends with signing support.

Model Registry is consumed by the RHOAI platform operator which deploys it as a managed component. It integrates with KServe for model serving, Istio for service mesh networking, and the RHOAI dashboard UI for user interaction.

## Architectural Analysis

Model Registry follows a multi-tier architecture with clear separation between the API layer, data access layer, and integration layer. The core design is **OpenAPI-first**: both the model registry API (v1alpha3) and the catalog API (v1alpha1) are defined as OpenAPI specs in `api/openapi/src/`, and Go server code is generated from these specs. This ensures API consistency and enables automatic client generation for the Python SDK, BFF, and CSI components.

The **database abstraction** is notably well-designed. The `internal/platform/` package introduces a connector pattern with factory-based driver registration, allowing MySQL and PostgreSQL to be used interchangeably. Database drivers register themselves via Go `init()` side-effect imports (`internal/platform/db/drivers/drivers.go`), and the connector uses incremental backoff retry (25 attempts over ~5 minutes) for resilient startup. The EmbedMD datastore layer wraps this with a repository pattern, providing type-safe access to 14+ entity repositories through reflection-based lookup. Schema migrations are tracked via `schema_migrations` and health-checked for dirty state.

The **controller sub-component** implements a label-based integration pattern with KServe InferenceServices. Rather than owning CRDs, the controller watches `serving.kserve.io/v1beta1.InferenceService` resources and uses configurable labels (`REGISTERED_MODEL_ID_LABEL`, `MODEL_VERSION_ID_LABEL`, etc.) to link ISVCs to model registry entities. When an ISVC with these labels is created, the controller creates a corresponding InferenceService record in model registry and sets a finalizer. On deletion, it marks the MR record as UNDEPLOYED. This loose coupling means the controller can be disabled entirely (`INFERENCE_SERVICE_CONTROLLER=""`) without affecting the core registry.

The **BFF** serves a dual role: it proxies REST API requests from the React frontend to both the model registry and catalog services, and it performs Kubernetes operations (namespace listing, RBAC checks, service discovery) that the browser cannot do directly. Authentication is mode-dependent — in `kubeflow` mode, user identity comes from `kubeflow-userid` headers injected by the mesh; in `user_token` mode, the BFF forwards the user's Bearer token to upstream services. This design centralizes auth handling while supporting multiple deployment topologies.

A notable architectural risk is that the core model registry API has **no built-in authentication** — it relies entirely on the service mesh (Istio mTLS + AuthorizationPolicy) or the BFF layer for access control. The `AuthorizationPolicy` is set to `ALLOW all` within the mesh, meaning any pod in the mesh can access the registry API. In production RHOAI deployments, the platform operator likely configures stricter policies, but the component's own manifests are permissive. The TLS configuration module (`internal/platform/tls/config.go`) is well-implemented with insecure cipher rejection and FIPS-compatible cipher suites, but `InsecureSkipVerify` defaults to `true` unless explicitly configured, which is a gap in defense-in-depth.
