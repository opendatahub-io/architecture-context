## Purpose

**Short**: Provides GPU-accelerated training runtime container images, Ray runtime images, and end-to-end integration tests for the Distributed Workloads vertical of Red Hat OpenShift AI.

**Detailed**: The distributed-workloads repository is a multi-purpose component that serves three primary functions within the RHOAI platform. First, it produces training runtime container images for PyTorch-based distributed training workloads, built in multiple accelerator variants (NVIDIA CUDA and AMD ROCm) across different PyTorch versions. These images are used as the base environment for PyTorchJob and Kubeflow Trainer v2 workloads, and include ML libraries such as PyTorch, Transformers, PEFT, DeepSpeed, Flash Attention, and Training Hub. Second, it produces Ray runtime images for KubeRay-based distributed computing workloads, also in CPU, CUDA, and ROCm variants. Third, it contains comprehensive Go-based end-to-end test suites that validate the Distributed Workloads stack (Kueue, Training Operator/KFTO v1, Kubeflow Trainer v2, KubeRay) on OpenShift clusters.

The repository also includes a "universal" training image line (prefixed `th06-`) that combines Jupyter workbench functionality with training runtime capabilities in a single image, supporting both interactive development and batch training modes. These universal images use AIPCC (AI Platform Container and Compute) base images from the Ecosystems team and install packages from the RHEL AI Python Package Index for secure, hermetic downstream builds. The older "runtime" training images use UBI9 base images with packages from public PyPI.

## Architectural Analysis

The distributed-workloads repository has an unusual architecture for a RHOAI component: rather than being a deployed service or operator, it is primarily an image factory and test harness. Its 13 Konflux Dockerfiles produce training runtime images that are consumed by reference in other components' CRs (PyTorchJob, TrainJob, RayCluster, Notebook). The images themselves expose no Kubernetes APIs and define no controllers — they are passive workload containers.

The repository is in the midst of a significant architectural transition. The older "runtime" training images (8 variants covering CUDA 12.1–12.8 and ROCm 6.2–6.4 with Python 3.11/3.12) use plain UBI9 base images and install GPU toolkits manually from vendor repos. The newer "universal" training images (3 TH06 variants) and "OpenMPI" training images (2 variants) represent the target architecture: they use AIPCC or RHOAI workbench base images, install Python packages from the RHEL AI PyPI index, and support hermetic builds via Hermeto (cachi2) prefetch. The coexistence of both patterns reflects the gradual migration path — the older images will eventually be retired as workloads migrate to the universal/OpenMPI line.

The OpenMPI images introduce a notable networking pattern: multi-node MPI training requires SSH connectivity between pods. The images embed an SSH server (port 2222) with relaxed security settings (`StrictModes no`, `StrictHostKeyChecking no`) to accommodate OpenShift's random UID assignment. The `uid_entrypoint.sh` and `mpirun_wrapper.sh` scripts handle the OpenShift UID registration in `/etc/passwd` at container startup, which is necessary because Python's `getpass.getuser()` and PyTorch cache directory setup require a resolvable UID. This is a well-documented OpenShift pattern but introduces a security trade-off — SSH host key verification is disabled for MPI peer communication.

The Go test code in `tests/` is architecturally significant as a platform integration validation layer. The tests exercise the full distributed workloads stack — creating PyTorchJobs, TrainJobs, RayJobs, and RayClusters with specific image references, verifying Kueue queue management, and validating training completion with S3 checkpoint storage. The test dependencies in `go.mod` reveal the full API surface this component interacts with: `kubeflow/trainer/v2`, `kubeflow/training-operator`, `ray-project/kuberay/ray-operator`, `sigs.k8s.io/kueue`, `sigs.k8s.io/jobset`, and `openshift/kueue-operator`.

The universal images implement a clever dual-mode pattern via `entrypoint-universal.sh`: when `NOTEBOOK_ARGS` is set (by the OpenShift workbench controller), the image starts JupyterLab; otherwise, it executes the provided command. This allows the same image to serve as both an interactive development environment and a batch training runtime, reducing the image matrix and ensuring consistency between development and production training environments.
