## Purpose

**Short**: Sidecar container image providing model downloading and runtime adaptation for ModelMesh Serving model server pods.

**Detailed**: The modelmesh-runtime-adapter repository produces a single multi-purpose container image containing five Go binaries that run as sidecar containers in ModelMesh Serving model server pods. These binaries act as intermediaries between the ModelMesh framework and third-party model-serving runtimes (Triton Inference Server, OpenVINO Model Server, MLServer, and TorchServe).

The image contains two categories of components: (1) the **model-serving-puller**, which downloads model artifacts from cloud storage backends (S3, Azure Blob, Google Cloud Storage, HTTP, PVC) to the local pod filesystem, and (2) four **runtime adapters** (triton, ovms, mlserver, torchserve) that implement the ModelMesh `ModelRuntime` gRPC interface and translate model load/unload requests into runtime-specific operations. Each adapter handles model layout adaptation — transforming downloaded model files into the directory structure expected by its target runtime.

The puller and adapters can operate independently or in an "embedded puller" mode where the adapter internally invokes the puller before loading models. All inter-process communication uses gRPC on localhost within the same pod. The image also includes Python 3.11 with TensorFlow for Keras-to-TensorFlow SavedModel conversion, used by the Triton adapter.

## Architectural Analysis

The modelmesh-runtime-adapter is architecturally interesting as a **multi-binary sidecar image** — a single container image producing five distinct binaries, each serving a different role. This design trades image size (one large image) for operational simplicity (one image to version, distribute, and patch across all ModelMesh runtimes). The Dockerfile.konflux builds all five binaries in a single multi-platform build stage and copies them into the runtime image alongside Python 3.11 and TensorFlow. This means even pods using only the OVMS adapter carry the TorchServe adapter binary and TensorFlow — a tradeoff of image size (~1GB+ due to TensorFlow) for build/release simplicity.

All adapters implement the same `mmesh.ModelRuntime` gRPC interface (defined in `internal/proto/mmesh/model-runtime.proto`), creating a uniform adapter pattern. Each adapter translates this common interface into runtime-specific operations: Triton uses native gRPC, OVMS uses HTTP REST with a batched actor model, MLServer uses gRPC with model-settings.json generation, and TorchServe uses separate management/inference gRPC connections with MAR file symlinks. The OVMS adapter is the most architecturally distinct — it employs an **actor pattern** (`modelmanager.go`, 598 lines) with request batching to coalesce rapid model load/unload requests into fewer config reload operations, preventing OVMS from thrashing on rapid configuration changes.

The **pullman** library is the shared storage abstraction layer, implementing a pluggable provider architecture with a client cache (1-hour TTL, 4-hour cleanup). Each storage provider (S3, Azure, GCS, HTTP, PVC) registers itself via `init()` functions. The PVC provider is notably different — it creates filesystem symlinks rather than downloading data, since PVC mounts are already local. The HTTP provider is the only one supporting mutual TLS (mTLS) with client certificates, making it suitable for enterprise model repositories with certificate-based auth.

Security-wise, all inter-process communication within the pod uses **insecure gRPC** (no TLS, no authentication), which is appropriate for localhost communication within a single pod. External storage access uses provider-appropriate TLS with credentials mounted from the `/storage-config` volume. The `util.SecureJoin()` function wrapping `cyphar/filepath-securejoin` is used consistently across all adapters to prevent directory traversal attacks in model paths — a critical security measure given that model paths originate from user-supplied LoadModelRequest data.

The model layout adaptation logic varies significantly by runtime: Triton requires `config.pbtxt` protobuf config files and versioned directories; OVMS needs JSON config lists with explicit target device settings; MLServer expects `model-settings.json` with implementation class mappings; TorchServe simply symlinks MAR archive files. The Triton adapter additionally supports **Keras-to-TensorFlow conversion** by invoking a Python script (`/opt/scripts/tf_pb.py`) with a semaphore limiting concurrent conversions to 2 by default.
