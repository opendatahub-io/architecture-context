## Purpose

**Short**: Sidecar container image that runs inside ModelMesh Serving model server pods, providing model downloading from cloud storage and runtime-specific adapter logic for multiple ML inference engines.

**Detailed**: The modelmesh-runtime-adapter repository produces a single multi-purpose container image containing five Go binaries: a model-serving puller and four runtime-specific adapters (Triton, MLServer, OVMS, TorchServe). Each binary runs as a separate process within the same pod.

The **model-serving-puller** acts as a gRPC intermediary between ModelMesh and the actual model runtime. It intercepts `LoadModel` requests, downloads model artifacts from remote storage (S3, GCS, Azure Blob, HTTP, PVC) to local disk using the `pullman` library, then forwards the modified request (with local filesystem paths) to the runtime adapter. It supports multiple authentication methods per storage backend and manages model lifecycle including cleanup on unload.

The **runtime adapters** (triton-adapter, mlserver-adapter, ovms-adapter, torchserve-adapter) each implement the `mmesh.ModelRuntime` gRPC service and translate between the ModelMesh model management protocol and the specific API of each inference engine. They handle model directory layout adaptation, configuration file generation (config.pbtxt, model-settings.json, etc.), and runtime health monitoring. The adapters communicate with their respective inference runtimes via gRPC or HTTP REST within the same pod.

## Architectural Analysis

The modelmesh-runtime-adapter follows a **proxy-chain sidecar pattern** where the model-serving-puller sits between ModelMesh and the runtime adapters in the gRPC call chain. When a `LoadModel` request arrives, the puller intercepts it, downloads the model from cloud storage, rewrites the model path to the local filesystem, strips sensitive credentials from the request, and then forwards the sanitized request to the downstream runtime adapter. This design cleanly separates storage concerns from runtime-specific model management. The puller uses the `pullman` library which provides a plugin-based storage provider system with automatic client caching (1-hour TTL) to avoid reconnecting to the same storage backends repeatedly.

Each runtime adapter implements the same `mmesh.ModelRuntime` gRPC interface but translates to completely different downstream protocols. Triton and MLServer use gRPC repository management APIs with similar patterns (RepositoryModelLoad/Unload), OVMS uses HTTP REST with a unique actor-pattern batch processor for configuration management, and TorchServe uses a dedicated Management gRPC API. The OVMS adapter stands out architecturally: instead of direct per-model API calls, it maintains an internal model configuration state machine, batches load/unload requests (configurable 100ms-3s window), writes a JSON config file, and triggers OVMS to reload. This design handles OVMS's limitation of only supporting full-config reloads rather than incremental model operations.

The model layout adaptation logic is a critical architectural concern. Each inference runtime expects a specific directory structure and configuration format. Triton requires a repository structure with `config.pbtxt` files describing model inputs/outputs. MLServer expects `model-settings.json` with framework-specific `implementation` mappings (e.g., `mlserver_sklearn.SKLearnModel`). OVMS uses a simpler directory-and-config-file approach. TorchServe requires Model Archive (.mar) files. The adapters create symlinks to the original model files rather than copying them, which avoids doubling disk usage. The Triton adapter additionally supports Keras-to-TensorFlow model conversion by invoking a Python subprocess (`/opt/scripts/tf_pb.py`) that uses TensorFlow to load .h5 files and re-export as SavedModel format.

All inter-process communication within the pod uses plaintext gRPC over localhost, which is appropriate for sidecar containers sharing a pod network namespace. Security is enforced at the pod boundary by the parent ModelMesh Serving deployment. The only external network traffic is egress to cloud storage providers, which uses TLS with provider-specific authentication. Storage credentials are mounted from Kubernetes Secrets at `/storage-config` and are intentionally stripped from the `LoadModelRequest.ModelKey` before forwarding to the runtime adapter (lines 220-221 in `puller.go`), ensuring credentials do not leak to the inference runtime. The `pullman` library also uses FNV-64a hashing for cache keys derived from secrets, avoiding storage of raw credentials in memory-resident cache structures.

The single Konflux Dockerfile builds all five Go binaries and includes Python 3.11 with TensorFlow as a runtime dependency. The image intentionally has no entrypoint — the orchestrating controller selects which binary to run based on the inference runtime type. The FIPS-compliant build uses `GOEXPERIMENT=strictfipsruntime` and `CGO_ENABLED=1` for FIPS-validated cryptographic operations. The base image is UBI 9 Minimal, consistent with RHOAI's Red Hat Enterprise Linux foundation.
