## Purpose

**Short**: Sidecar container that downloads ML models from cloud storage and adapts them for specific model serving runtimes (Triton, MLServer, OVMS, TorchServe) within ModelMesh Serving pods.

**Detailed**: The modelmesh-runtime-adapter is a multi-purpose container image that runs as a sidecar in ModelMesh Serving model server pods. It provides two key functions: (1) a model-serving-puller that downloads model artifacts from cloud object storage (S3, GCS, Azure Blob, HTTP/HTTPS) or PVC mounts into local storage, and (2) runtime-specific adapter servers that translate between ModelMesh's unified `ModelRuntime` gRPC interface and the native APIs of supported inference runtimes.

The adapter pattern enables ModelMesh to manage model lifecycle (load, unload, predict size, report status) across heterogeneous model serving backends through a single consistent interface. Each adapter handles model layout adaptation — restructuring downloaded model files into the directory layout expected by its target runtime (e.g., Triton's `config.pbtxt` model repository format, MLServer's `model-settings.json`, OVMS's version directory structure, or TorchServe's `.mar` archive symlinks).

The image produces five Go binaries compiled into a single container: `puller`, `triton-adapter`, `mlserver-adapter`, `ovms-adapter`, and `torchserve-adapter`. The consuming ModelMesh deployment selects which binary to run via the container command. The puller runs as an intermediary that intercepts `LoadModel` requests from ModelMesh, downloads the model, and forwards the request to the underlying runtime adapter. Python 3.11 with TensorFlow is included in the runtime image to support Keras-to-TensorFlow SavedModel conversion for Triton.

## Architectural Analysis

The modelmesh-runtime-adapter implements a clean adapter pattern where a single container image packages five independent Go binaries, each implementing the same `mmesh.ModelRuntime` gRPC interface but translating to different backend runtime APIs. This design allows ModelMesh to treat all model serving runtimes uniformly while the adapter handles the complexity of runtime-specific model layout, configuration, and lifecycle management. The adapter and puller communicate exclusively over localhost gRPC, with the puller intercepting load requests to download models before forwarding to the adapter.

The pullman storage framework is a notable architectural choice — it provides a pluggable provider system with connection caching (1-hour TTL), concurrent downloads (10 workers for S3, 4 for GCS/Azure), and secure path joining to prevent directory traversal attacks. Each storage provider registers itself via Go's `init()` mechanism, making the system extensible without modifying the core pull logic. The credential model is file-based: Kubernetes Secrets are mounted at `/storage-config` and each key maps to a JSON file containing provider-specific credentials, with optional per-request parameter overrides via dotpath notation.

The OVMS adapter stands out architecturally from the other three adapters. While Triton, MLServer, and TorchServe adapters use gRPC to communicate with their runtimes, the OVMS adapter uses HTTP REST (`/v1/config` and `/v1/config/reload`). More significantly, it implements an actor-pattern model manager (`OvmsModelManager`) that batches model load/unload requests to minimize expensive OVMS configuration reloads. This batching uses configurable wait times (100ms for loads, 3s for unloads) and deduplicates requests to the same model ID, which is critical because each OVMS reload reloads all model configurations simultaneously.

Each adapter's model layout adaptation is tailored to its runtime's requirements: Triton expects `config.pbtxt` + version directories, MLServer expects `model-settings.json` with implementation class names, OVMS expects numeric version directories with specific naming conventions, and TorchServe requires `.mar` archive files. All adapters use symlinks rather than copying files, enabling efficient model loading when combined with the puller's download mechanism. The Triton adapter additionally includes Keras-to-TensorFlow SavedModel conversion capability, invoking a Python script (`tf_pb.py`) with concurrency-limited semaphore to prevent resource exhaustion.

Security-wise, all adapter-to-runtime communication is plaintext on localhost, which is acceptable since both containers share the same pod. External cloud storage communication uses TLS with provider-specific authentication. The HTTP storage provider supports custom CA certificates and mTLS client certificates for enterprise environments. The FIPS compliance posture is strong: all Go binaries are compiled with `GOEXPERIMENT=strictfipsruntime` and `-tags strictfipsruntime` with dynamic linking (`CGO_ENABLED=1`), ensuring OpenSSL-backed crypto in FIPS mode. The `crypto/tls` usage in the HTTP provider delegates cipher suite selection to Go's defaults, which in FIPS mode are restricted to FIPS-approved algorithms.
