## Purpose

**Short**: Sidecar container that acts as an intermediary between ModelMesh and third-party model-serving runtime containers, handling model artifact storage retrieval and runtime-specific protocol adaptation.

**Detailed**: The modelmesh-runtime-adapter repository produces a single multi-purpose container image containing five Go binaries that run as sidecar containers inside ModelMesh Serving model server pods. The image serves two primary functions: (1) a model-serving-puller that downloads model artifacts from cloud storage backends (S3, GCS, Azure Blob, HTTP/HTTPS, PVC) into the pod's local filesystem, and (2) four runtime-specific adapters (Triton, OVMS, MLServer, TorchServe) that translate the ModelMesh internal gRPC protocol (`mmesh.ModelRuntime`) into the specific model management APIs of each supported inference runtime.

Each adapter implements the same `mmesh.ModelRuntime` gRPC service interface, which ModelMesh uses to load/unload models and query runtime status. The adapters then communicate with their respective model server containers (co-located in the same pod) using runtime-specific protocols: gRPC for Triton, MLServer, and TorchServe; HTTP/REST for OVMS. The puller component can operate standalone or be embedded within each adapter for unified model retrieval and loading. The image also includes a Python script and TensorFlow installation for converting Keras (.h5) models to TensorFlow SavedModel format at model-load time, specifically for the Triton adapter path.

The upstream project originated at IBM/KServe and is forked by Red Hat for RHOAI with FIPS-compliant builds via Konflux. The container image is referenced by the ModelMesh Serving controller via the `model-serving-config` ConfigMap's `storageHelperImage` field.

## Architectural Analysis

The modelmesh-runtime-adapter follows a clean adapter pattern architecture where a common interface (`mmesh.ModelRuntime` gRPC service) is implemented independently by each runtime adapter. This design enables ModelMesh to manage models across heterogeneous inference runtimes without knowing the specifics of each runtime's API. The shared interface defines five key operations — loadModel, unloadModel, predictModelSize, modelSize, and runtimeStatus — providing a consistent lifecycle contract regardless of the underlying model server.

A notable design decision is the "embedded puller" option (`USE_EMBEDDED_PULLER` flag), which allows each adapter to include model artifact retrieval directly rather than requiring the separate puller sidecar. This reduces the two-hop path (ModelMesh → puller → adapter → runtime) to a single-hop path (ModelMesh → adapter-with-puller → runtime), trading pod complexity for latency reduction. The puller and all adapters share the same `pullman` library, which implements a plugin-based storage provider architecture where each cloud backend is registered via Go `init()` functions. The pullman library maintains a TTL-based client cache (1-hour TTL, 4-hour cleanup) to avoid recreating storage SDK clients on every model load request.

The OVMS adapter stands out architecturally from the other three adapters: while Triton, MLServer, and TorchServe use gRPC for runtime communication, OVMS uses an HTTP/REST API with an actor-pattern model manager (`OvmsModelManager`) that batches configuration reload requests. This batching is necessary because OVMS reloads its entire model configuration atomically (via a single JSON config file), making per-model reload calls expensive. The batch window is configurable (100ms min to 3s max), allowing the adapter to amortize multiple concurrent model load/unload requests into a single OVMS reload.

The TorchServe adapter has unique characteristics: it communicates with two separate TorchServe endpoints (management on port 7071 and inference on port 7070), uses a higher default `MODELSIZE_MULTIPLIER` of 2.75 (vs 1.25 for others) reflecting TorchServe's higher memory overhead, and implements a separate `ModelSize` RPC that queries TorchServe's `DescribeModel` for actual worker memory usage with retry logic (1-second intervals up to 16 seconds). The Triton adapter performs the most complex model layout adaptation, including Keras-to-TensorFlow model conversion via the bundled Python script and config.pbtxt generation with schema injection for batch dimension handling.

All communication between sidecar containers within a pod is plaintext gRPC or HTTP over localhost, which is acceptable for pod-internal traffic. External egress to cloud storage uses TLS with support for custom CA certificates and mTLS client authentication (HTTP provider). The security boundary is at the pod level, with storage credentials delivered via mounted Kubernetes Secrets. Path traversal protection is consistently applied across all storage providers using the `cyphar/filepath-securejoin` library, and the model state manager serializes operations per-model using goroutines and channels (25-item buffer) to prevent race conditions.
