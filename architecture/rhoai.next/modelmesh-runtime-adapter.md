## Purpose

**Short**: Sidecar container that intermediates between ModelMesh and third-party model-server runtimes, handling model storage retrieval and runtime-specific protocol adaptation.

**Detailed**: The modelmesh-runtime-adapter repository produces a single multi-purpose container image containing five Go binaries: a model-serving puller and four runtime-specific adapters (Triton, MLServer, OVMS, TorchServe). These binaries run as sidecar containers within ModelMesh Serving model server pods, bridging the gap between the ModelMesh orchestration layer and diverse model inference runtimes.

The **model-serving-puller** implements the ModelRuntime gRPC interface as a proxy, intercepting LoadModel calls to download model artifacts from remote storage (S3, GCS, Azure Blob, HTTP, PVC) before delegating to the actual model runtime. This decouples storage access from inference runtime concerns. The puller uses the **pullman** library, a pluggable storage provider framework with client caching and support for multiple authentication mechanisms.

Each **runtime adapter** also implements the ModelRuntime gRPC interface and translates ModelMesh's generic model lifecycle operations (load, unload, predict size, status) into runtime-specific API calls — gRPC for Triton/MLServer/TorchServe and HTTP REST for OVMS. Adapters handle model layout adaptation (reorganizing downloaded model files into the directory structure each runtime expects), capacity management, and health checking. The Triton adapter additionally includes a Python script for converting Keras H5 models to TensorFlow SavedModel format, which is why the container image includes Python 3.11 and TensorFlow.

## Architectural Analysis

The modelmesh-runtime-adapter follows a **proxy/adapter pattern** where a single container image bundles multiple purpose-specific binaries, each implementing the same gRPC interface (`mmesh.ModelRuntime`) but targeting different backend runtimes. This design enables a clean separation between the ModelMesh orchestration layer and the diverse ecosystem of model inference servers. The puller sits in front as a transparent gRPC proxy, adding storage retrieval capability without any runtime-specific knowledge, while each adapter translates the generic interface to runtime-specific APIs behind it.

A notable architectural choice is the **all-in-one container image** approach: rather than producing separate images per adapter, all five binaries are built into a single image with no default entrypoint. The consuming deployment selects which binary to run via the container `command` field. This reduces image sprawl and simplifies version management but means every model serving pod carries unused binaries and the full Python/TensorFlow stack (even when Triton's Keras conversion isn't needed). The Python dependencies (tensorflow at ~1.8GB installed) significantly inflate the image size for a component that is fundamentally a Go gRPC service.

The **pullman library** is well-designed as an extensible storage framework with pluggable providers, automatic client caching (1-hour TTL), and per-provider authentication. Each provider registers itself via Go's `init()` pattern, making it easy to add new storage backends. The client cache uses a hash of credential-relevant configuration fields as the cache key, so different credential sets for the same provider type get distinct clients. However, there is no client-side rate limiting or retry logic in the storage providers — error handling is left to the SDK defaults.

The OVMS adapter is architecturally distinct from the other three adapters: while Triton, MLServer, and TorchServe adapters communicate with their runtimes via gRPC, the OVMS adapter uses HTTP REST and implements an **actor-pattern model manager** that batches model configuration changes. The model manager debounces rapid load/unload sequences (configurable 100ms–3s window) before issuing a single config reload to OVMS, which is important because OVMS config reloads are relatively expensive operations that involve re-reading the model configuration file.

All inter-container communication within the pod uses **plaintext gRPC on localhost** — there is no TLS or authentication between the puller, adapter, and runtime containers. This is a deliberate design decision: security is enforced at the pod boundary by ModelMesh and the platform's network policies, not within the pod. The only TLS usage is in outbound storage connections, where the HTTP provider supports custom CA certificates and mutual TLS client authentication, and the S3 provider supports custom CA bundles.
