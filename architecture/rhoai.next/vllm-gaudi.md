## Purpose

**Short**: Intel Gaudi (Habana) hardware plugin for vLLM that enables high-performance LLM inference on Gaudi 2 and Gaudi 3 accelerators.

**Detailed**: vllm-gaudi is a hardware plugin package for the vLLM inference engine that provides Intel Gaudi HPU (Habana Processing Unit) support. It follows vLLM's pluggable hardware architecture (RFC #11162 and RFC #19161), registering itself via Python entry points to replace GPU-specific operations with Gaudi-optimized implementations for attention, model execution, distributed communication, and memory management.

The plugin provides HPU-specific implementations of attention backends (paged attention, unified attention, MLA attention), custom ops for quantization (FP8, INC, AWQ, GPTQ, ModelOpt), fused MoE layers, LoRA adapters, and speculative decoding (EAGLE). It manages Gaudi-specific memory profiling, KV cache allocation with 128-block alignment, HPU graph compilation, and bucketing strategies for efficient batching. The container image bundles vLLM core with the Gaudi plugin, Habana SynapseAI drivers, and PyTorch for Habana, exposing an OpenAI-compatible API server as the entrypoint.

Within RHOAI, this image is consumed by KServe as a model serving runtime for deploying LLMs on Intel Gaudi accelerator nodes, providing an alternative to NVIDIA GPU-based vLLM serving.

## Architectural Analysis

The vllm-gaudi plugin follows a clean separation of concerns by leveraging vLLM's plugin architecture. Rather than forking vLLM, it registers itself via Python entry points (`vllm.platform_plugins`, `vllm.general_plugins`) and provides override implementations for HPU-specific behavior. The `HpuPlatform` class (in `platform.py`) is the central registration point, declaring supported quantization methods, attention backends, worker classes, and device communicators. This design allows the plugin to track upstream vLLM releases independently — the `VLLM_STABLE_COMMIT` file pins the last verified upstream commit, and the Dockerfile builds against that specific version.

The attention backend architecture is notably sophisticated, with four distinct backends: `HPUAttentionBackendV1` (standard paged attention), `HPUUnifiedAttentionBackend` (unified prefill+decode attention), `HPUMLAAttentionBackend` (Multi-head Latent Attention for DeepSeek-style models), and `HPUUnifiedMLABackend`. The runtime configuration system (`extension/config.py`, `extension/runtime.py`, `extension/features.py`) dynamically selects backends and features based on hardware detection (Gaudi 2 vs Gaudi 3), SynapseAI version, model type, and environment variables. This configuration system is unusually complex for a plugin — it implements a lazy-evaluation dependency graph where feature values can depend on other features, hardware capabilities, and version constraints.

Memory management is a critical concern for Gaudi accelerators. The plugin implements custom KV cache allocation with 128-block alignment (vs. typical GPU block sizes of 16-32), HPU-specific memory profiling (`HabanaMemoryProfiler`), graph compilation memory reservation (`VLLM_GRAPH_RESERVED_MEM`), and online defragmentation (`OnlineDefragmenter`). The worker implements sleep/wake-up cycles that move model weights between HPU and CPU to enable memory-efficient model serving. The bucketing system (`extension/bucketing/`) optimizes batch processing by grouping sequences into size buckets (exponential or linear strategies), reducing recompilation overhead for HPU graph mode.

The FIPS architecture deserves special attention: the Dockerfile explicitly removes the OpenSSL FIPS provider (`rpm -e --nodeps openssl-fips-provider-so`), which is unusual for RHOAI components. This is likely required for compatibility with Habana's proprietary libraries that may conflict with FIPS-mode OpenSSL. Since the vLLM server itself runs plain HTTP (no TLS) and all security is handled at the platform ingress layer, the practical FIPS risk is limited to the container's runtime crypto posture. However, this means the image would not pass check-payload FIPS validation if that were enforced.

The model support layer (`models/`) provides HPU-optimized implementations for specific architectures that need custom handling: Gemma3, Qwen2.5-VL, Qwen3-VL-MoE, Ovis, HunYuan, MiniMax, Pixtral, DotsOCR, SeedOss, DeepSeek-v2, and DeepSeek-OCR. These are registered as model overrides that replace the upstream vLLM implementations when running on Gaudi hardware, typically adding HPU-specific attention patterns, memory layouts, or compilation hints.
