## Purpose

**Short**: Hardware plugin that enables vLLM large language model inference on Intel Gaudi (Habana) accelerators.

**Detailed**: vllm-gaudi is the Intel Gaudi hardware plugin for the vLLM inference engine. It follows vLLM's pluggable hardware architecture (RFC #11162 and RFC #19161) to provide HPU-specific implementations of attention backends, model runners, workers, distributed communication, quantization support, and custom kernel operations. The plugin registers itself via Python entry points (`vllm.platform_plugins` and `vllm.general_plugins`) and replaces vLLM's default GPU-oriented code paths with Gaudi-optimized equivalents.

The plugin supports Intel Gaudi 2 and Gaudi 3 accelerators with BFloat16 as the primary compute dtype (FP16 and FP32 are automatically converted to BF16). It provides FP8 quantization support via Intel Neural Compressor (INC), AWQ, GPTQ, ModelOpt, and compressed-tensors quantization methods. Key optimizations include unified attention with chunked prefill, FlashAttention online merge, contiguous paged attention, NIXL-based KV cache transfer for disaggregated serving, EAGLE speculative decoding, and HPU-specific bucketing strategies for efficient graph compilation. The container image packages vLLM core plus this plugin and exposes the OpenAI-compatible API server as its entrypoint.

## Architectural Analysis

**Plugin Architecture**: vllm-gaudi exemplifies a well-designed hardware plugin system. It registers itself via three Python entry points: `vllm.platform_plugins` (registering `HpuPlatform`), `vllm.general_plugins` for utilities and custom ops, and model overrides. This clean separation means the plugin can override vLLM's behavior at multiple layers — platform detection, attention backends, worker implementation, distributed communication, and model-specific optimizations — without forking vLLM itself. The plugin also monkey-patches certain vLLM internals at import time (e.g., replacing `AsyncScheduler` with `HPUAsyncScheduler` in `utils.py:290`, and `make_tensor_with_pad` with an HPU-compatible implementation).

**Attention Backend Complexity**: The plugin provides at least four attention backend implementations — `HPUAttentionV1`, `HPUUnifiedAttentionBackend`, `HPUMLAAttentionBackend`, and `HPUUnifiedMLABackend` — selected dynamically based on runtime configuration (`unified_attn`, `use_mla`). The unified attention path (`extension/unified.py`) enables chunked prefill with online FlashAttention merge and FP32 softmax adjustments. This complexity is driven by Gaudi's unique execution model where operations must be compiled into HPU graphs, and dynamic shapes are handled through a sophisticated bucketing system (`extension/bucketing/`) that rounds sequence lengths and batch sizes to predetermined buckets to maximize graph reuse.

**Configuration System**: The `extension/` directory implements a full runtime configuration framework with lazy evaluation, environment variable overrides, hardware detection, and version-gated features. The `Config` class (`extension/config.py`) supports computed values with dependencies — for example, `unified_attn_softmax_fa2` is enabled only when the Synapse version is >=1.24.0.279, unified attention is on, the `softmax_fa2` kernel exists, the hardware is Gaudi 3, and chunked shared attention is disabled. This conditional feature system means the same plugin binary adapts its behavior based on the detected hardware, driver version, and user configuration, but adds significant complexity to understanding what code paths are active in any given deployment.

**Non-Hermetic Build Concerns**: The Dockerfile clones both `vllm-project/vllm` and `vllm-project/vllm-gaudi` from public GitHub at build time (lines 196-217), controlled only by build ARGs (`VLLM_GAUDI_COMMIT`, `VLLM_PROJECT_COMMIT`). The Tekton pipeline sets `hermetic: false`. While commit pinning provides reproducibility at the source level, the lack of dependency lock files means transitive Python dependencies can drift between builds. The explicit removal of `openssl-fips-provider-so` (line 63) is also noteworthy — it disables FIPS-mode OpenSSL, which may conflict with FIPS requirements on regulated clusters.

**Quantization Support**: The plugin supports six quantization methods: `compressed-tensors`, `fp8` (via INC), `inc`, `awq_hpu`, `gptq_hpu`, and `modelopt`. INC quantization requires a calibration step (documented in `calibration/`) that generates measurement files, and the plugin forces CPU weight loading when INC is detected to prevent OOM during the quantization-aware weight loading process (`platform.py:170-174`). The FP8 path includes sophisticated per-token and per-channel scaling with hardware-specific handling — Gaudi 2 uses `float8_e4m3fnuz` while Gaudi 3 uses `float8_e4m3fn` (`extension/ops.py:28-29`).

**NIXL Disaggregated Serving**: The plugin implements NIXL-based KV cache transfer (`distributed/kv_transfer/kv_connector/v1/`) for disaggregated prefill/decode architectures. It monkey-patches `torch.Tensor.data_ptr` globally to map virtual data pointers to physical HPU addresses via `htexp._data_ptr`, which is required for NIXL to correctly address HPU memory. The heterogeneous connector (`hetero_hpu_nixl_connector.py`) supports mixed tensor parallel sizes between prefill and decode workers, enabling asymmetric deployments where prefill uses more HPUs than decode.
