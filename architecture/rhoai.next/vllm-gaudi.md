## Purpose

**Short**: vLLM hardware plugin that enables high-performance large language model inference on Intel Gaudi (Habana) accelerators.

**Detailed**: The vllm-gaudi repository implements a vLLM hardware plugin that integrates Intel Gaudi AI accelerators (Gaudi2 and Gaudi3) with the vLLM inference engine. It follows vLLM's pluggable hardware architecture (RFC #11162 and #19161) to provide a modular interface that registers platform-specific implementations for attention backends, model runners, workers, distributed communication, quantization, and KV cache management.

The plugin is packaged as a Python library (`vllm_gaudi`) that registers itself via entry points (`vllm.platform_plugins` and `vllm.general_plugins`). When installed alongside upstream vLLM, it intercepts device selection and substitutes HPU-optimized implementations for attention computation, memory management, model execution, and distributed inference. The shipped container image (`odh-vllm-gaudi-rhel9`) bundles the Habana SynapseAI runtime, PyTorch for Gaudi, upstream vLLM, and this plugin, and exposes an OpenAI-compatible API server as its entrypoint.

The plugin supports multiple quantization formats (FP8, INC, GPTQ, AWQ, compressed-tensors, ModelOpt), advanced attention mechanisms (unified attention, MLA/Multi-head Latent Attention, speculative decoding with EAGLE), data parallelism, tensor parallelism, expert parallelism for MoE models, and KV cache offloading between HPU and CPU memory. It targets RHOAI as the serving runtime for LLM inference workloads on Intel Gaudi hardware.

## Architectural Analysis

The vllm-gaudi plugin demonstrates a well-structured hardware abstraction approach. By registering itself via Python setuptools entry points (`vllm.platform_plugins` for HpuPlatform, `vllm.general_plugins` for utilities, ops, and models), it can intercept vLLM's device selection at startup and substitute HPU-optimized implementations without forking the upstream codebase. This is architecturally significant because it allows the plugin to track upstream vLLM releases closely — the `VLLM_STABLE_COMMIT` file pins a specific upstream vLLM commit hash that has been validated against this plugin version.

The extension module (`vllm_gaudi/extension/`) is the most architecturally interesting subsystem. It implements a lazy-evaluated configuration framework with hardware detection, feature flags, and runtime validation. Over 150 environment variables control optimization behavior, from attention implementation selection (`VLLM_PROMPT_USE_FLEX_ATTENTION`, `VLLM_PROMPT_USE_FUSEDSDPA`) to bucketing strategies (`VLLM_EXPONENTIAL_BUCKETING`), quantization modes, and profiling controls. This design reflects the reality of accelerator optimization: performance tuning on specialized hardware requires fine-grained control over execution strategies, and the lazy-evaluation approach ensures only queried configuration values are computed.

The attention subsystem is the largest and most complex part of the plugin (~4,000+ lines across multiple files). It provides four attention backend variants: standard HPU attention, MLA (Multi-head Latent Attention for DeepSeek-style models), unified attention (an optimized batched approach with online merge for Flash-Attention-2 style incremental computation), and unified MLA. The unified attention path is particularly sophisticated — it splits attention into partial computations (causal, shared-cache, unique-cache) and merges results using an online merge algorithm to reduce peak memory usage. The bucketing system (`vllm_gaudi/extension/bucketing/`) pre-generates fixed-shape execution graphs for different batch size and sequence length combinations, which is essential for HPU graph compilation efficiency since Gaudi accelerators benefit significantly from static graph shapes.

Quantization support is comprehensive, covering six distinct quantization methods: FP8 per-tensor/per-channel/per-block, Intel Neural Compressor (INC), GPTQ (2/3/4/8-bit), AWQ (4-bit), compressed-tensors (W8A8, WNA16), and NVIDIA ModelOpt FP8. The FP8 scale handling includes hardware-specific alignment for Gaudi2 vs Gaudi3 devices, reflecting that these accelerators have different exponent bias ranges for FP8 operations. The `scales.py` module contains pre-computed lookup tables for hardware-aligned FP8 scales, which is a critical correctness requirement for quantized inference on these devices.

The container image build (`Dockerfile.konflux.gaudi`) uses a multi-stage approach: Stage 1 installs Habana system drivers and RPM packages on UBI 9.6, Stage 2 installs PyTorch from Habana's proprietary artifact repository, and Stage 3 clones and installs both upstream vLLM and the vllm-gaudi plugin. The final image runs as non-root user (UID 2000) for OpenShift compatibility and uses the vLLM OpenAI API server as its entrypoint. The Konflux build system is configured via `.konflux/build-args.conf` to pin both the vLLM and vllm-gaudi versions to v0.16.0.
