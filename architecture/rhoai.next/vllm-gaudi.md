## Purpose

**Short**: Hardware plugin for vLLM that enables high-performance large language model inference on Intel Gaudi (Habana) accelerators.

**Detailed**: vLLM-Gaudi is an out-of-tree (OOT) hardware plugin for the vLLM inference serving framework that adapts vLLM to run on Intel Gaudi 2 and Gaudi 3 AI accelerators. It follows vLLM's plugin architecture (`vllm.platform_plugins` and `vllm.general_plugins` entry points) to register HPU-specific implementations for attention backends, custom ops, model overrides, distributed communication, and worker management without modifying the upstream vLLM codebase.

The plugin ships as a single container image (`odh-vllm-gaudi-rhel9`) built on Red Hat UBI 9 with the Intel Habana SynapseAI runtime stack (v1.23.0), PyTorch 2.9.0 for Gaudi, and the upstream vLLM v0.16.0 server. The container runs the standard `vllm.entrypoints.openai.api_server` as its entrypoint, serving an OpenAI-compatible REST API. The vllm_gaudi plugin is loaded automatically via Python entry points, overriding platform detection, attention kernels, quantization methods, model forward passes, and worker lifecycle management with Gaudi-optimized implementations.

The component is deployed as a model serving runtime in Red Hat OpenShift AI (RHOAI), typically behind KServe as an InferenceService, providing high-throughput inference for large language models on Gaudi hardware. It supports tensor parallelism, data parallelism, pipeline parallelism, speculative decoding (Eagle), FP8/GPTQ/AWQ/INC quantization, LoRA adapters, vision-language models, and KV cache disaggregation via NIXL.

## Architectural Analysis

### Plugin Architecture and vLLM Integration

The vllm_gaudi plugin leverages vLLM's extensible plugin system to intercept and override platform-specific behavior at multiple levels. The plugin registers via three entry points defined in `setup.py` (lines 66-72): `vllm.platform_plugins` for the `HpuPlatform` class, and two `vllm.general_plugins` entries for custom ops and model overrides. This layered registration approach allows the plugin to progressively specialize vLLM's execution without touching upstream code.

The `HpuPlatform` class (platform.py:32-333) is the central integration point. It overrides attention backend selection, worker class resolution, compilation mode, cache block size defaults (128 blocks for HPU efficiency), and dtype enforcement (forcing bfloat16 over float16/float32). It also disables CUDA-specific features (CUDAGraph mode, pin memory) and configures HPU-specific execution modes (lazy vs eager, torch.compile settings).

### Unified Attention System

A distinctive architectural feature is the Unified Attention system (extension/unified.py, extension/unified_batch.py), which decomposes attention into three orthogonal computations: causal attention (current query tokens against their own KV), shared block attention (prefix/prompt cache blocks shared across requests), and unique block attention (per-request decode blocks). This decomposition enables more efficient batch processing by sharing computation across requests that share prompt prefixes — a common pattern in production LLM serving. The system supports both online and offline softmax merging strategies, with chunked computation for memory efficiency on long shared contexts.

### Multi-Level Quantization Support

The plugin supports six quantization methods (platform.py:39): compressed-tensors, fp8, inc, awq_hpu, gptq_hpu, and modelopt. Each method has a dedicated ops module (e.g., `ops/hpu_fp8.py`, `ops/hpu_gptq.py`) that registers HPU-optimized implementations of quantized matrix multiplication. The FP8 quantization path is particularly mature, with a full calibration pipeline in the `calibration/` directory (6 steps from dataset preparation through scale quantization) and support for both per-tensor and per-token scaling. The INC (Intel Neural Compressor) quantization path includes a safety feature: the platform forces CPU weight loading for INC quantized models (platform.py:170-174) to prevent OOM during the quantization-in-place weight transformation.

### Speculative Decoding

The plugin implements HPU-optimized speculative decoding with Eagle proposers (`v1/spec_decode/hpu_eagle.py`, `v1/spec_decode/hpu_eagle_unified.py`) and a vectorized rejection sampler (`v1/sample/hpu_rejection_sampler.py`). The rejection sampling is deliberately performed on CPU (line 53 of hpu_rejection_sampler.py) for faster processing, as the comparison logic is sequential and benefits from CPU optimization. Two Eagle variants are supported: standard (sequential draft generation) and unified (batch-mode draft generation using the unified attention system).

### KV Cache Management and Disaggregation

KV cache management is a critical architectural concern with three distinct systems: (1) local cache ops (extension/cache_ops.py) for block-level copy/swap within a single instance, (2) online defragmentation (extension/defragmentation.py) that monitors fragmentation and compacts cache blocks when a threshold is exceeded, and (3) cross-instance KV transfer via NIXL connectors (distributed/kv_transfer/). The NIXL integration includes heterogeneous support (hetero_hpu_nixl_connector.py) that handles block size conversion and layout transformation (NHD↔HND) between prefill and decode instances that may have different configurations — a sophisticated feature for production disaggregated serving.

### FIPS Compliance Gap

A significant security observation: the Dockerfile explicitly removes the OpenSSL FIPS provider (`rpm -e --nodeps openssl-fips-provider-so || true` at line 63). This means the container runs with standard (non-FIPS) OpenSSL, which may be a deliberate decision to avoid compatibility issues with the Habana SynapseAI runtime. Deployments requiring FIPS compliance should be aware of this gap.
