## Purpose

**Short**: Hardware plugin that enables vLLM inference serving on Intel Gaudi (Habana) accelerators.

**Detailed**: The vllm-gaudi repository provides an out-of-tree (OOT) hardware plugin for the vLLM inference engine that adds support for Intel Gaudi (Habana Labs) AI accelerators, including Gaudi 2 and Gaudi 3. The plugin follows vLLM's hardware-pluggable architecture ([RFC #11162](https://github.com/vllm-project/vllm/issues/11162)) and registers itself via Python entry points as both a platform plugin (`vllm.platform_plugins`) and general plugins for custom utilities, ops, and models.

The plugin replaces GPU-specific components in vLLM with HPU (Habana Processing Unit) implementations, including custom attention backends (FusedSDPA, unified attention, MLA), KV cache management with Gaudi-optimized block sizes (128), distributed communication via HCCL, speculative decoding with EAGLE, and hardware-specific quantization support (INC FP8, AWQ, GPTQ, ModelOpt, compressed-tensors). It also provides HPU-optimized model implementations for architectures like DeepSeek V2, Qwen 2.5-VL, Qwen 3 MoE, Gemma 3, Pixtral, Ovis, and others that require custom attention or MoE handling on Gaudi hardware.

The Konflux Dockerfile builds a multi-stage container image based on RHEL 9.6 UBI, installing Habana Synapse drivers (v1.23.0), PyTorch for Habana, upstream vLLM, and the vllm-gaudi plugin. The container runs the vLLM OpenAI-compatible API server (`vllm.entrypoints.openai.api_server`) as a non-root user (UID 2000) for OpenShift compatibility.

## Architectural Analysis

The vllm-gaudi plugin follows a clean separation of concerns enabled by vLLM's hardware-pluggable architecture. Rather than forking vLLM, it registers as an out-of-tree plugin via Python entry points, overriding only the hardware-specific components while relying on vLLM core for the API server, scheduler, tokenization, and request management. This design allows the plugin to track upstream vLLM releases (pinned via `VLLM_STABLE_COMMIT` and the `.konflux/build-args.conf` file) without maintaining a full fork. The current pin is at vLLM v0.16.0 with vllm-gaudi v0.16.0.

The plugin's architecture is organized around four registration hooks in `__init__.py`: `register()` for the platform class, `register_utils()` for utility monkey-patches, `register_ops()` for custom operator implementations, and `register_models()` for model architecture overrides. The platform class (`HpuPlatform`) overrides vLLM's platform abstraction to configure HPU-specific defaults: 128-block KV cache sizes (vs. smaller GPU defaults), bfloat16 as the mandatory dtype (float16/float32 are not supported on HPU), spawn-based multiprocessing (fork causes hangs), and torch.compile with the `hpu_backend`.

The attention subsystem is notably complex, with multiple backend implementations selectable at runtime: `HPUAttentionBackendV1` (default paged attention), `HPUUnifiedAttentionBackend` (fused prefill+decode in a single graph), `HPUMLAAttentionBackend` (Multi-Latent Attention for DeepSeek-style models), and `HPUUnifiedMLABackend`. The unified attention path is a Gaudi-specific optimization that batches prefill and decode operations into a single graph compilation, reducing the number of HPU graph recompilations. The backend selection is driven by a rich runtime configuration system (`vllm_gaudi/extension/config.py`, `features.py`, `environment.py`) that detects hardware type (Gaudi 2 vs 3), Synapse version, model architecture, and user environment variables to automatically select optimal configurations.

The distributed communication layer wraps HCCL (Habana Collective Communication Library) through the `HpuCommunicator` class, which implements `all_reduce`, `all_gather`, and MoE-specific `dispatch`/`combine` operations. A notable workaround is the explicit `mark_step()` calls before collective operations, required to flush the HPU lazy execution pipeline before synchronization. The NIXL connector support enables disaggregated prefill/decode serving, with HPU-specific adaptations for data pointer management (the `_hpu_data_ptr` function works around HPU tensor view limitations) and heterogeneous tensor parallel configurations.

A significant architectural risk is the FIPS compliance gap: the Dockerfile explicitly removes the OpenSSL FIPS provider (`rpm -e --nodeps openssl-fips-provider-so`), which means this image cannot operate in FIPS mode. This is likely driven by compatibility issues between Habana's proprietary drivers and FIPS-mode OpenSSL, but it blocks deployment on FIPS-enforcing OpenShift clusters. Additionally, the AIPCC Gaudi base image is currently disabled (AIPCC-3471), so this component builds its entire Gaudi stack from scratch on UBI 9.6 — a significant deviation from the RHOAI standard where other accelerator variants (CUDA, ROCm, Spyre) use AIPCC base images. The build is also non-hermetic: it clones vLLM and vllm-gaudi from GitHub during the Docker build and installs packages from the public internet, with no lock files or Hermeto prefetch on the `main` branch.
