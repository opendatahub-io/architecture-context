## Purpose

**Short**: Garak red-teaming evaluation adapter for the eval-hub platform, providing automated LLM vulnerability scanning as Kubernetes Jobs or Kubeflow Pipelines.

**Detailed**: This component integrates NVIDIA's Garak red-teaming framework with the RHOAI eval-hub evaluation platform. It provides a `FrameworkAdapter` implementation that reads a `JobSpec` from a mounted ConfigMap, builds and executes Garak CLI commands for LLM security scanning, parses JSONL/AVID-format results, and reports evaluation metrics back to eval-hub via sidecar callbacks. The adapter supports two execution modes: "simple" mode runs Garak as a subprocess directly inside the K8s Job pod, while "KFP" mode submits a six-step Kubeflow Pipeline (validate → taxonomy → SDG → prompts → scan → parse) that runs Garak in a separate KFP pod with S3-based artifact transfer.

The component provides predefined benchmark profiles mapped to security taxonomies (OWASP LLM Top 10, AVID, CWE, quality) as well as an "intents" benchmark that uses Synthetic Data Generation (SDG) via sdg-hub to generate adversarial prompts from policy taxonomy datasets. The intents pipeline uses TAPIntent (tree-of-attacks with persuasion) probes, MulticlassJudge detectors, and multi-model configuration (target, judge, attacker, evaluator) to perform targeted risk assessment. Results are persisted to OCI registries and MLflow for tracking, with HTML report generation via Jinja2 templates and Vega chart specifications.

## Architectural Analysis

The component follows a clean separation between framework-agnostic core logic and framework-specific adapters. The `core/` module contains pure functions for config resolution, command building, subprocess management, and pipeline steps that are intentionally decoupled from eval-hub and KFP specifics. This design enables the same business logic to serve multiple execution paths: inline eval-hub jobs, KFP pipelines, and potentially future Llama Stack providers (the repository name reflects its origin as a Llama Stack provider, though v0.5.0 removed the Llama Stack provider in favor of the eval-hub adapter pattern).

The configuration resolution system is notable for its layered merge approach: GarakCommandConfig defaults → predefined profile (OWASP, AVID, intents, etc.) → user's `garak_config` overrides → legacy flat-key overrides. This three-level deep-merge allows users to minimally customize scans while inheriting sensible defaults from curated profiles. The intents model configuration adds a further layer with the `intents_models` shorthand that fills empty slots in the garak config using a `x.get("key") or default` pattern, ensuring user-configured values are preserved.

API key management follows a defense-in-depth approach across execution modes. In simple mode, keys are read from mounted secrets via eval-hub SDK's `read_model_auth_key()` or environment variables. In KFP mode, keys use a `__FROM_ENV__` placeholder in the config dict that is resolved at pod level by `_resolve_config_api_keys()`, which reads from volume-mounted secrets at `/mnt/model-auth` or environment variables. This ensures config.json files uploaded to S3 never contain real credentials — they are always redacted via `redact_api_keys()` before persistence. The six-level key resolution cascade (role-specific env → role-specific file → generic env → generic file → `api-key` file → `"DUMMY"`) provides flexibility while maintaining security.

The intents benchmark pipeline is the most architecturally complex feature, requiring four distinct LLM endpoints (target, judge, attacker, evaluator) plus an optional SDG endpoint. The pipeline generates adversarial prompts via SDG from a policy taxonomy, then runs a multi-probe scan (SPOIntent, SPOIntentUserAugmented, SPOIntentSystemAugmented, SPOIntentBothAugmented, TranslationIntent, TAPIntent) with an EarlyStopHarness that filters stubs at each stage. The ART HTML report generation uses Vega chart specifications embedded as JSON resources with Jinja2 templating for visualization.

The Garak subprocess runner (`garak_runner.py`) implements robust process group management: it creates a new process group via `os.setsid`, streams stdout/stderr in real-time via daemon threads, handles timeouts with graceful SIGTERM → SIGKILL escalation across the entire process group, and caps output with a configurable tail buffer. This is critical because Garak scans can run for hours and may spawn child processes for parallel probe execution.
