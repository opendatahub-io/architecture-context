## Purpose

**Short**: Out-of-tree Llama Stack evaluation provider and eval-hub adapter that orchestrates NVIDIA Garak red-teaming scans against LLMs, with support for both local inline execution and remote Kubeflow Pipelines-based execution.

**Detailed**: This component integrates NVIDIA's Garak vulnerability scanning framework with two independent orchestration platforms: Meta's Llama Stack (as an out-of-tree evaluation provider) and Red Hat's eval-hub evaluation platform (as a `FrameworkAdapter`). It enables automated red-teaming of Large Language Models for prompt injection, jailbreaks, toxicity, bias, and other vulnerabilities, with support for compliance frameworks including OWASP LLM Top 10, AVID taxonomy, CWE, and custom intent-based risk assessments.

The provider supports three deployment modes: (1) **Total Remote** — runs entirely on OpenShift/Kubernetes with KFP pipelines orchestrating the scan; (2) **Partial Remote** — local Llama Stack server submits KFP pipelines to a cluster; (3) **Total Inline** — runs garak locally inside the Llama Stack server process. A key feature is the "Intents" benchmark, which uses Synthetic Data Generation (SDG) via sdg-hub to generate targeted adversarial prompts from a policy taxonomy, then scans the model with TAPIntent probes using tree-of-attacks with persuasion strategies. The component also provides shield testing capabilities, allowing users to measure guardrail effectiveness by wrapping model inference with input/output safety shields.

The component ships as a single container image (`odh-trustyai-garak-lls-provider-dsp`) built by Konflux, which serves as the base image for KFP pipeline steps and as the adapter pod image for eval-hub jobs. It does not expose any network services itself — it is consumed as a Python library by Llama Stack servers or as a standalone adapter pod by the eval-hub platform.

## Architectural Analysis

This component has a distinctive dual-integration architecture, serving two completely independent orchestration platforms (Llama Stack and eval-hub) that share core business logic but have fundamentally different entry points, APIs, and deployment models. The `core/` module provides the common layer — config resolution, command building, garak execution, pipeline step logic, and result parsing — while `inline/`, `remote/`, and `evalhub/` are platform-specific adapters. This design is reinforced by a deliberate dependency firewall: the eval-hub adapter imports only from `core/` and never touches Llama Stack types, while the Llama Stack providers use `base_eval.py` as their shared base class. The `compat.py` module adds another layer of resilience by providing a dual-import path for Llama Stack APIs (new `llama_stack_api` package vs. legacy `llama_stack` layout), enabling the provider to work across upstream API transitions.

The configuration system uses a deep-merge pattern throughout. Predefined benchmark profiles (OWASP LLM Top 10, AVID taxonomy, CWE, Intents, etc.) define base configurations that users can override at any granularity — from swapping a single probe to completely replacing the garak config. The `GarakCommandConfig` Pydantic model mirrors garak's native YAML config structure (system, run, plugins, reporting, CAS), ensuring type safety while remaining faithful to garak's configuration surface. For the Intents benchmark specifically, an `intents_models` shorthand provides a clean abstraction over the complex 4-model configuration (judge, attacker, evaluator, target), with a 1-or-3 validation rule and `__FROM_ENV__` placeholder pattern for Kubernetes Secret injection.

The KFP pipeline architecture differs significantly between the two platforms. The Llama Stack remote provider uses a simpler 4-step pipeline that uploads artifacts via the Llama Stack Files API. The eval-hub KFP pipeline uses a 6-step design (validate → resolve taxonomy → SDG → prepare prompts → scan → write KFP outputs) with S3 as the artifact transfer medium via Data Connection secrets. Both pipelines use the same container image as the base, and both inject secrets as environment variables via `kubernetes.use_secret_as_env`. The eval-hub pipeline additionally mounts the model auth secret as a volume at `/mnt/model-auth` with `optional=True`, enabling a flexible API key resolution chain: role-specific env var → role-specific file → generic env var → generic file → "DUMMY".

The Rust/Cargo configuration in this repository is worth noting from an architectural standpoint. The `Cargo.toml` defines a dummy binary with dependencies like `pyo3`, `fancy-regex`, and `bstr` — these are not used by the application directly but are required at build time for compiling native Python extensions (tokenizer libraries) that are transitive dependencies of garak. The `rpms.in.yaml` installs rust, cargo, gcc, and ninja-build into the Konflux build image specifically for this purpose. This is a common pattern in RHOAI Python components that depend on packages with Rust native extensions.

Security-wise, the component handles credentials through multiple channels: environment variables, Kubernetes Secrets (both as env vars and volume mounts), and the eval-hub adapter's `read_model_auth_key` utility. API keys are explicitly redacted from configs before S3 upload or logging via `redact_api_keys()`. TLS verification is configurable throughout, supporting boolean flags, string truthy/falsy values, and custom CA certificate paths — a necessary flexibility for OpenShift clusters with custom certificate authorities.
