## Purpose

**Short**: Garak-based LLM red-teaming and vulnerability scanning adapter for the eval-hub evaluation platform.

**Detailed**: This component provides a `FrameworkAdapter` implementation that integrates NVIDIA's Garak red-teaming framework with the eval-hub evaluation platform. It enables automated security scanning of Large Language Models for vulnerabilities including prompt injection, jailbreaks, toxicity, bias, and other safety concerns. The adapter supports predefined benchmark profiles aligned to industry frameworks (OWASP LLM Top 10, AVID taxonomy, CWE, quality) and a novel intents-based testing mode that combines Synthetic Data Generation (SDG) with multi-strategy probe pipelines.

The component runs as a Kubernetes Job orchestrated by the eval-hub service. It supports two execution modes: **simple mode** (runs Garak as a subprocess directly inside the Job pod) and **KFP mode** (submits a six-step Kubeflow Pipeline that runs Garak in a separate pod, with S3 for artifact transfer). In KFP mode, the pipeline steps handle validation, taxonomy resolution, SDG prompt generation, prompt normalization, Garak scanning, and result aggregation. Results are persisted to OCI registries, S3 buckets, and MLflow, and reported back to the eval-hub sidecar.

The package was originally a Llama Stack provider but has been refactored (v0.5.0) to be an eval-hub-only adapter, removing the Llama Stack provider layer and keeping only the eval-hub integration.

## Architectural Analysis

The component follows a layered architecture that cleanly separates framework-agnostic business logic (`core/`) from eval-hub-specific integration (`evalhub/`). The `core` package contains pure functions for config resolution, command building, Garak subprocess management, and pipeline step logic, while the `evalhub` package provides the `FrameworkAdapter` implementation and KFP pipeline definition. This separation was intentional to allow future reuse of the core logic by other integration frameworks beyond eval-hub.

The intents-based testing mode is architecturally the most complex feature. It implements a multi-model pipeline where up to six different LLM endpoints may be contacted during a single scan: the target model (being tested), an SDG model (generating adversarial prompts from a harm taxonomy), a judge model (MulticlassJudge detector for scoring responses), an attacker model (TAP probe for tree-of-attacks-with-pruning jailbreaking), an evaluator model (TAP evaluator for jailbreak scoring), and a translation model (multilingual TranslationIntent probe). API key management for these roles uses a hierarchical resolution pattern: role-specific env vars → generic fallback env var → volume-mounted secret files → "DUMMY" default. This complexity is managed through the `_resolve_config_api_keys` function that walks the nested config and replaces placeholder values at pod startup time.

The KFP pipeline uses a six-step DAG (validate → resolve_taxonomy → sdg_generate → prepare_prompts → garak_scan → write_kfp_outputs) where the S3 Data Connection secret is injected into every pod via `kubernetes.use_secret_as_env`, and the model auth secret is mounted as a volume (with `optional=True` to support unauthenticated local models). A notable design decision is that the adapter pod does NOT embed real API keys into the KFP pipeline arguments — instead, it writes `__FROM_ENV__` placeholders into the garak config, and the pipeline pods resolve them from the mounted secret at runtime. This prevents API keys from appearing in KFP run logs or pipeline metadata. After scan completion, `config.json` is also redacted (all `api_key` values replaced with `***`) before upload to S3 or OCI.

The Garak subprocess runner (`core/garak_runner.py`) demonstrates careful process management: it creates a new process group (`os.setsid`), streams stdout/stderr to the logger via daemon threads with bounded deque tails, and handles timeout with graceful shutdown (SIGTERM → 5s wait → SIGKILL to process group). This ensures that even hung garak processes with spawned subprocesses are fully cleaned up. The `shell=False` execution explicitly prevents shell injection via the config parameters.

Result parsing supports two distinct output formats: the standard per-probe AVID-enhanced aggregation (used for framework benchmarks like OWASP, AVID, CWE) and a stub-level intents aggregation (used for the intents benchmark). The intents aggregation groups results by `(intent, stub)` tuples and computes attack success rate as the percentage of unique stubs jailbroken by any probe — matching the ART (Automated Red Team) HTML report's `high_level_stats` computation. TBSA (Tier-Based Security Aggregate) scoring is available from garak>=0.14.0 as an overall security metric.
