## Purpose

**Short**: An out-of-tree Llama Stack evaluation provider and eval-hub adapter for automated LLM red-teaming and vulnerability scanning using Garak.

**Detailed**: This component implements Garak-based LLM security evaluation across two independent integration surfaces: (1) a Llama Stack provider registered with the Llama Stack framework, and (2) an eval-hub adapter for the RHOAI evaluation platform. Both surfaces share core Garak logic but have different entry points, APIs, and deployment models.

The provider supports three execution modes: **inline** (Garak runs as a subprocess inside the Llama Stack server), **remote KFP** (Garak runs as Kubeflow Pipelines components on Kubernetes), and **eval-hub** (Garak runs as a Kubernetes Job orchestrated by the eval-hub SDK). The remote/KFP modes enable a six-step pipeline: validate → taxonomy → SDG → prompts → scan → parse, supporting advanced intents-based benchmarking with synthetic data generation and tree-of-attacks probing.

The component ships as a single container image (`odh-trustyai-garak-lls-provider-dsp`) built via Konflux, containing all dependencies including Garak, KFP SDK, boto3, and the eval-hub SDK adapter. It is consumed by the Llama Stack server (as a registered provider) and by eval-hub (as a Kubernetes Job container). The component does not expose any network services itself — it acts as a client to other platform services (Llama Stack API, KFP, S3, model inference endpoints).

## Architectural Analysis

The llama-stack-provider-trustyai-garak component has an unusually clean dual-integration architecture. It serves two completely independent orchestration platforms — Llama Stack and eval-hub — through a shared `core/` module that contains all framework-agnostic business logic. The `evalhub/` module explicitly avoids importing any Llama Stack types, ensuring the eval-hub adapter has no transitive dependency on the Llama Stack framework. This separation is enforced by design: `evalhub/` only imports from `core/`, while `inline/` and `remote/` import from both `core/` and `base_eval.py` (which provides the Llama Stack `Eval` protocol implementation).

The configuration system uses a sophisticated multi-layer merging strategy. Framework profiles (predefined benchmark configurations like `trustyai_garak::owasp_llm_top10`) provide defaults, which are deep-merged with user-supplied `benchmark_config` overrides, which are then deep-merged with `intents_models` overlays. The `deep_merge_dicts()` function implements recursive dictionary merging where later values override earlier ones, but only at the leaf level — this allows users to override specific probe settings without clobbering the entire plugins configuration. The intents model overlay uses a particularly careful `x.get("key") or default` pattern that respects user-configured values while filling empty slots, and always forces `api_key` to `__FROM_ENV__` for Kubernetes Secret injection.

The KFP pipeline architecture is duplicated across two implementations — `remote/kfp_utils/` for Llama Stack and `evalhub/kfp_pipeline.py` for eval-hub — with identical six-step structure but different artifact transfer mechanisms. The Llama Stack pipeline uses the Llama Stack Files API for all artifact uploads/downloads, while the eval-hub pipeline uses S3 via Data Connection Secrets. Both pipelines mount model authentication secrets as volumes at `/mnt/model-auth` and resolve API keys per-role (TARGET, SDG, JUDGE, ATTACKER, EVALUATOR) with a sophisticated fallback chain: direct value → named secret file → environment variable → convention-based lookup → DUMMY placeholder. This duplication exists because the two platforms have fundamentally different artifact storage paradigms, though the core pipeline logic (`core/pipeline_steps.py`) is shared.

The Garak subprocess management (`core/garak_runner.py`) demonstrates careful production hardening: process group creation via `os.setsid()` enables reliable cleanup of entire process trees, graceful shutdown uses SIGTERM with a 5-second grace period before SIGKILL, real-time log streaming uses dedicated threads, and output capture uses a configurable tail buffer (default 500 lines) to prevent memory exhaustion on long-running scans. The inline adapter adds async concurrency control via `asyncio.Semaphore(max_concurrent_jobs)` to prevent resource exhaustion when multiple evaluations run simultaneously.

A notable security consideration is that the reference deployment manifests in `lsd_remote/` contain placeholder credentials and development-only configurations (default PostgreSQL password `llamastack`, emptyDir volume, TLS verification disabled for vLLM). These are clearly documented as reference/dev configurations, but the architecture design delegates all authentication to Kubernetes Secrets and ServiceAccount tokens in production, with the RBAC configuration following least-privilege principles (only `get` and `create` verbs on DSPA API resources).
