## Purpose

**Short**: Out-of-tree Llama Stack evaluation provider and eval-hub adapter that uses NVIDIA Garak for automated LLM red-teaming, vulnerability scanning, and compliance benchmarking.

**Detailed**: This component implements two completely independent integration surfaces for LLM security evaluation using NVIDIA's Garak red-teaming framework. The first surface is a Llama Stack provider registered via the Llama Stack plugin system, allowing users to invoke vulnerability scans through the standard `benchmarks.register` / `eval.run_eval` API. It supports inline execution (garak runs in-process) and remote execution (scans delegated to Kubeflow Pipelines). The second surface is an eval-hub `FrameworkAdapter` used by the RHOAI evaluation platform; it reads a `JobSpec` from a mounted ConfigMap and executes scans either as a local subprocess ("simple" mode) or via KFP pipeline submission ("kfp" mode).

A key capability is the "intents" benchmark, which combines synthetic data generation (SDG) with tree-of-attacks probing to assess model safety against a policy taxonomy. This six-step pipeline (validate -> taxonomy -> SDG -> prompts -> scan -> parse) runs only in KFP-based modes. The component ships predefined benchmark profiles for OWASP LLM Top 10, AVID taxonomy, CWE, and quality assessments, and supports deep-merge customization of profiles. Results include per-probe attack success rates, TBSA (Tier-Based Security Aggregate) scores, and HTML reports. Scan artifacts are persisted to Llama Stack Files API, S3, OCI registries, and MLflow depending on the execution mode.

## Architectural Analysis

This component exhibits a clean dual-integration architecture. The Llama Stack provider surface (`inline/`, `remote/`, `base_eval.py`) and the eval-hub adapter surface (`evalhub/`) share core business logic through the `core/` module but maintain strict dependency isolation: the eval-hub adapter never imports Llama Stack types, and the Llama Stack providers never import eval-hub types. This is enforced by the module dependency graph documented in `ARCHITECTURE.md`. The `compat.py` layer provides forward/backward compatibility across Llama Stack API package migrations (`llama_stack_api` vs legacy `llama_stack` layout), allowing the provider to work with both new and old Llama Stack versions.

The configuration system uses a deep-merge pattern that is central to usability. Predefined benchmark profiles (OWASP, AVID, CWE, intents) are defined as `GarakCommandConfig` instances in `config.py`, and users override specific fields via `metadata.garak_config`. The `deep_merge_dicts` function (implemented in both `base_eval.py` and `core/config_resolution.py`) recursively merges override values, so users can tune a single parameter (e.g., `soft_probe_prompt_cap`) without restating the entire profile. The eval-hub adapter additionally supports legacy flat-key overrides (`probes`, `probe_tags`, `detectors`) translated via `legacy_overrides_from_benchmark_config`.

API key management follows a layered resolution strategy designed for Kubernetes. In KFP mode, model auth secrets are mounted as volumes at `/mnt/model-auth`, and the `resolve_api_key(role)` function reads role-specific files (e.g., `JUDGE_API_KEY`) with fallback to generic `api-key` and environment variables. Configs are always redacted (`redact_api_keys`) before upload to S3, Files API, or OCI, so secrets never persist in object storage. The `_resolve_config_api_keys` function replaces placeholder values (`DUMMY`, `__FROM_ENV__`, `***`) with real keys at pod startup time. This design allows the same config JSON to be used across environments without embedded secrets.

The process management in `core/garak_runner.py` is notably robust for a batch job: garak runs in a new process group (`os.setsid`), output is streamed via daemon threads with bounded deques, and timeouts trigger SIGTERM-then-SIGKILL of the entire process tree. The inline adapter adds asyncio-level concurrency control via a semaphore (`max_concurrent_jobs`), while the remote adapter delegates concurrency to KFP. The Rust dependency (`Cargo.toml`) is not for this project's code -- it's a "dummy-project" that exists solely to trigger Hermeto/cachi2 prefetching of Rust crates that garak's tokenizer dependencies need at build time.
