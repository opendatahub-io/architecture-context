## Purpose

**Short**: A Python library and container image for supervised fine-tuning (SFT) of large language models using HuggingFace Transformers and TRL, with multi-GPU support via Accelerate/FSDP.

**Detailed**: fms-hf-tuning (Foundation Model Stack HuggingFace Tuning) is a training toolkit that wraps HuggingFace's SFTTrainer to provide a production-ready, containerized solution for fine-tuning causal language models and vision-language models. It is the primary training engine used by the Kubeflow Training Operator (KFTO) in Red Hat OpenShift AI to execute PyTorchJob-based fine-tuning workloads.

The library supports multiple PEFT methods (LoRA, activated LoRA/aLoRA, Prompt Tuning) and full-parameter fine-tuning. It integrates with the fms-acceleration framework for GPU-optimized training techniques including quantized LoRA (QLoRA via AutoGPTQ and BitsAndBytes), padding-free flash attention, multipack distributed packing, fused operations/kernels, ScatterMoE for parallelized Mixture of Experts, and Online Data Mixing (ODM). Configuration is provided via JSON (through environment variables or file paths), and the container image uses HuggingFace Accelerate's launch command to manage single-GPU and multi-GPU (FSDP) distributed training automatically.

The toolkit includes a pluggable trainer controller framework for dynamic training loop control (early stopping, loss thresholding, patience-based policies), a flexible data preprocessing pipeline supporting pretokenized data, Jinja templates, chat templates, multi-turn conversations, and vision-language datasets, and an extensible experiment tracking system with support for file logging, AimStack, MLflow, ClearML, and HF Resource Scanner.

## Architectural Analysis

The fms-hf-tuning codebase follows a layered architecture with clear separation between the container entrypoint, training logic, data processing, and extensibility frameworks. The `accelerate_launch.py` entrypoint in the `build/` directory acts as the container's CMD, reading configuration from a JSON file (via `SFT_TRAINER_CONFIG_JSON_PATH` or `SFT_TRAINER_CONFIG_JSON_ENV_VAR` environment variables), constructing `accelerate launch` arguments, and delegating to `tuning/sft_trainer.py` as a subprocess. This two-layer design allows the same training code to be invoked directly (via `python -m tuning.sft_trainer`) or through accelerate's distributed launcher, with the entrypoint automatically detecting GPU count and applying FSDP defaults when multiple GPUs are present.

The data processing pipeline (`tuning/data/`) implements a composable handler pattern where data transformations are registered as named `DataHandler` objects with typed operations (MAP, FILTER, REMOVE, SELECT, RENAME). The framework supports five distinct data formats: pretokenized data (passthrough), single-sequence with Jinja templates, chat-style with instruction/response templates, multi-turn conversation with chat templates, and vision-language multimodal data. Data configuration can be provided either through CLI arguments or a YAML/JSON data config file (`data_config_path`) that specifies multiple datasets with per-dataset handlers, sampling ratios, and split definitions. This design enables complex data mixing strategies and the Online Data Mixing (ODM) feature from fms-acceleration.

The trainer controller framework (`tuning/trainercontroller/`) provides a YAML-driven policy engine for dynamic training control. It uses HuggingFace's TrainerCallback hooks (on_step_end, on_epoch_end, on_evaluate, etc.) to evaluate user-defined rules against computed metrics, then execute operations like early stopping, logging, or custom actions. Rules are Python expressions evaluated safely via the `simpleeval` library with restrictions on imports and dunder access. The framework supports patience-based tolerance for noisy metrics. This is a distinctive feature that gives operators fine-grained control over training behavior without code changes.

The error handling strategy is notable for Kubernetes integration: the codebase distinguishes between user errors (exit code 1) and internal errors (exit code 203), writing structured termination logs to `/dev/termination-log`. This allows the Kubeflow Training Operator to surface meaningful error messages to users through the PyTorchJob status. The vLLM post-processing step in `accelerate_launch.py` handles the adapter compatibility pipeline for serving LoRA-tuned models with vLLM, including ScatterMoE checkpoint conversion, which bridges the training and inference lifecycle.

The container image build (`build/Dockerfile`) uses a multi-stage approach with a UBI9 base, CUDA 12.1 development layer, and a stripped release layer that removes Python development packages and dnf to minimize CVE surface. Optional features (AimStack, MLflow, ClearML, fms-acceleration plugins, flash-attention, mamba) are conditionally installed via build args, allowing customized images for different deployment scenarios. The Triton cache directories are explicitly configured via environment variables for qLoRA kernel compilation at runtime.
