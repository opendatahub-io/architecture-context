## Purpose

**Short**: Python library and container image for supervised fine-tuning (SFT) of large language models using Hugging Face Transformers, PEFT, and PyTorch FSDP.

**Detailed**: fms-hf-tuning provides a production-ready framework for fine-tuning foundation models (LLMs and vision-language models) using Hugging Face's `SFTTrainer` and PyTorch Fully Sharded Data Parallel (FSDP) for multi-GPU training. It is the core tuning engine used by Red Hat OpenShift AI for model customization workloads, typically deployed as a Kubernetes batch job via the Kubeflow Training Operator (`PyTorchJob`).

The library supports multiple tuning techniques including LoRA, Activated LoRA (aLoRA), Prompt Tuning, GPTQ-LoRA, and full fine-tuning. It features a pluggable acceleration framework (`fms-acceleration`) that integrates quantized LoRA, fused operations and kernels, padding-free flash attention, distributed packing (multipack), ScatterMoE for mixture-of-experts models, and Online Data Mixing (ODM). A sophisticated data preprocessing pipeline supports multiple input formats (pretokenized, single-sequence, chat/conversational, vision-language multimodal) with configurable data handlers, Jinja2 templating, and streaming datasets. Experiment tracking is provided through a pluggable tracker system supporting file logging, AimStack, MLflow, ClearML, and HF Resource Scanner. A trainer controller framework allows user-defined rules and metrics to control the training loop (e.g., early stopping, dynamic scaling).

The container image is built on Red Hat UBI 9 with CUDA 12.1 support and packages all dependencies for GPU-accelerated fine-tuning. Configuration is passed via JSON (file or base64-encoded environment variable), and the entrypoint (`accelerate_launch.py`) wraps `accelerate launch` to manage multi-GPU orchestration with automatic FSDP configuration.

## Architectural Analysis

**Architecture Pattern — Batch Job with Pluggable Pipelines**: fms-hf-tuning follows a pipeline architecture where configuration flows through parsing (JSON → dataclass hierarchy), data preprocessing (format detection → handler chain → collation), model loading (auto-detect vision vs language → optional acceleration framework augmentation), training (SFTTrainer with callbacks), and model saving. The entire flow is orchestrated by `accelerate launch` for multi-GPU support. This design decouples each stage cleanly, but the monolithic `train()` function in `sft_trainer.py` (lines 73-557) handles too many concerns — model loading, PEFT config, data processing, framework augmentation, callback registration, and training — making it a maintenance bottleneck.

**Acceleration Framework Integration**: The `fms-acceleration` integration is architecturally notable. The `AccelerationFrameworkConfig` class uses Python's `Annotated` type hints with custom `ConfigAnnotation` metadata to declaratively map CLI arguments to a YAML configuration consumed by the external `fms_acceleration.AccelerationFramework`. This is a creative use of the type system to bridge two configuration models, but it introduces complexity — the config is serialized to a temporary YAML file, then deserialized by the framework. The acceleration framework can modify both the model (custom loading, augmentation) and the training loop (callbacks), giving it deep hooks into the training pipeline.

**Data Preprocessing Extensibility**: The data preprocessing system is the most architecturally sophisticated component. It uses a `DataHandler` abstraction (map, filter, remove, select, rename operations) registered in a handler registry. Users can compose processing pipelines in YAML data config files. The system auto-detects four data formats (pretokenized, single-sequence, chat/conversational, vision-multimodal) and constructs appropriate handler chains. Online Data Mixing (ODM) is integrated as a special `ODMDataPreProcessor` subclass that produces per-dataset train/eval splits with dynamic sampling weights. The Jinja2 templating uses `SandboxedEnvironment` with `StrictUndefined` for safe template rendering.

**Security Concern — Pickle Deserialization**: The `txt_to_obj()` function in `tuning/utils/config_utils.py:167` falls back to `pickle.loads()` when JSON deserialization fails. This is a known deserialization vulnerability — an attacker who can control the `SFT_TRAINER_CONFIG_JSON_ENV_VAR` environment variable could execute arbitrary code. While the attack surface is limited (the env var is typically set by a trusted job orchestrator), this pattern should be replaced with JSON-only deserialization.

**Container Image Design**: The Dockerfile uses a multi-stage build with a notable "release-base" stage that strips Python, dnf, and crypto-policies-scripts from the final image to reduce CVE surface. The CUDA development libraries are only in the build stages; the final image copies only the installed Python packages from `~/.local/`. This reduces the image size but means the final image lacks `pip` — packages cannot be installed at runtime. The image runs as a non-root user (`tuning`, UID 1000). The conditional feature enablement via build args (`ENABLE_FMS_ACCELERATION`, `ENABLE_AIM`, etc.) allows building variants with different optional dependencies.
