## Purpose

**Short**: Python library and container image for supervised fine-tuning (SFT) of large language models using HuggingFace Transformers, TRL, and PyTorch FSDP.

**Detailed**: FMS HF Tuning (Foundation Model Stack HuggingFace Tuning) provides a production-ready fine-tuning framework for large language models. It wraps HuggingFace's `SFTTrainer` with extensive configuration management, multi-GPU distributed training via PyTorch FSDP and HuggingFace Accelerate, and support for parameter-efficient fine-tuning methods including LoRA, Activated LoRA, QLoRA (AutoGPTQ and BitsAndBytes), and Prompt Tuning.

The component is deployed as a container image (`quay.io/modh/fms-hf-tuning`) that runs as a Kubernetes job, typically orchestrated by the Kubeflow Training Operator (KFTO) as a `PyTorchJob`. It accepts training configuration via JSON config files or environment variables, loads models from HuggingFace Hub or local storage, and outputs trained model checkpoints. The framework includes a pluggable experiment tracking system supporting Aim, MLflow, ClearML, and file-based logging, as well as a rule-driven trainer controller framework for dynamic training loop control based on metrics.

The library also integrates with the FMS Acceleration framework for hardware-optimized training techniques including padding-free attention, fused LoRA operations, fast triton kernels, ScatterMoE expert parallelism, and online data mixing. A tuning configuration recommender (`fms-recommender.py`) provides automated hyperparameter optimization for launch commands.

## Architectural Analysis

FMS HF Tuning follows a layered architecture with clear separation between configuration, data processing, training orchestration, and experiment tracking. The entry point (`build/accelerate_launch.py`) acts as a container-native wrapper around HuggingFace Accelerate's launch mechanism, adding JSON-based configuration injection (via `SFT_TRAINER_CONFIG_JSON_PATH` or `SFT_TRAINER_CONFIG_JSON_ENV_VAR`), automatic GPU count detection, and FSDP defaults application. This design enables the same container image to transparently handle single-GPU and multi-GPU training without configuration changes — the wrapper auto-detects `torch.cuda.device_count()` and adjusts `num_processes` accordingly.

The data processing pipeline is notably sophisticated for a training library. The `DataPreProcessor` class supports a handler-chain pattern where datasets pass through configurable sequences of operations (map, filter, remove, select, rename) defined in YAML/JSON data config files. This enables complex preprocessing workflows including multi-modal vision-language data processing (via `VisionDataCollator`), chat template application with response masking for instruction-tuning, and online data mixing (ODM) with bandit-based sampling. The data config system supports base64-encoded chat templates for Kubernetes ConfigMap compatibility, which is a practical design for the PyTorchJob deployment pattern.

The trainer controller framework (`tuning/trainercontroller/`) implements a domain-specific language for training loop control using `simpleeval` for safe expression evaluation. This allows users to define rules like `"loss['eval_loss'] > 5.0 and loss['epoch'] > 2"` that trigger operations such as early stopping, forced evaluation, or custom logging. The system includes a patience mechanism for debouncing rule triggers, preventing premature action on transient metric spikes. Security is addressed through rule validation that rejects `__` patterns (preventing `__import__` exploits in the expression evaluator).

The container image build (`build/Dockerfile`) uses a multi-stage pattern with UBI 9 as the base, installing CUDA 12.1 in development stages and stripping Python/dnf packages in the release stage to reduce CVE surface. The release image intentionally removes Python 3 system packages and dnf itself (`rpm -e`), relying solely on the user-installed Python and pip packages. This aggressive stripping reduces image size and attack surface but means the runtime container has no package manager for debugging. The Tekton/Konflux pipeline builds from this Dockerfile and pushes to `quay.io/modh/fms-hf-tuning`, running in the `rhoai-tenant` namespace.

A notable architectural risk is the inter-process communication between FSDP workers: distributed training uses PyTorch's `c10d` static rendezvous on port 29500/TCP with plaintext TCP. In a multi-node training scenario, gradient synchronization data flows unencrypted between pods. This is standard practice for distributed ML training (the overhead of encrypting gradient tensors at training speeds would be prohibitive), but it means the training job relies on network-level isolation (Kubernetes NetworkPolicy or namespace isolation) rather than transport-level encryption for data protection during distributed training.
