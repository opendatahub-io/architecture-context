## Purpose

**Short**: OpenAI-compatible batch inference API gateway for llm-d that accepts, queues, processes, and manages asynchronous batch completion requests against LLM inference backends.

**Detailed**: The batch-gateway provides an OpenAI Batch API-compatible REST interface that enables users to submit large collections of inference requests (chat completions, embeddings, completions, moderations, responses) as batch jobs. It manages the full lifecycle of each batch — file upload, validation, priority-queued scheduling, concurrent execution against one or more inference gateway backends, progress tracking, output file assembly, and automatic expiration cleanup.

The system is composed of three independently deployable binaries sharing common libraries: (1) an **apiserver** that exposes the OpenAI-compatible `/v1/batches` and `/v1/files` REST API, (2) a **batch-processor** that dequeues jobs from a Redis/Valkey priority queue and executes inference requests against configured model gateways with per-model and global concurrency controls, and (3) a **batch-gc** garbage collector that periodically removes expired batch and file records. All three components share a pluggable storage backend (PostgreSQL or Redis/Valkey for metadata, filesystem or S3 for file content) and are deployed via a single Helm chart.

The batch-gateway is part of the llm-d ecosystem for Red Hat OpenShift AI, providing enterprise-grade batch inference with multi-tenancy (tenant isolation via configurable HTTP headers), SLO-based priority scheduling, distributed tracing (OpenTelemetry), Prometheus metrics with pre-built Grafana dashboards and PrometheusRule alerts, TLS support (including cert-manager integration), and crash-recovery for in-flight jobs.

## Architectural Analysis

The batch-gateway follows a classic producer-consumer architecture with clear separation of concerns across three binaries. The apiserver acts as the producer — it validates input, stores metadata, and enqueues work items into a Redis sorted set (priority queue) with SLO-derived priority scores. The processor acts as the consumer — it dequeues jobs, downloads input files, fans out inference requests with two-level concurrency control (global and per-model semaphores), and assembles output files. The GC runs independently as a periodic cleanup loop. This decomposition allows each component to scale independently: apiserver replicas scale with inbound request volume, processor replicas scale with inference throughput needs, and the GC typically runs as a single replica.

The concurrency model in the processor is notably sophisticated. A semaphore-driven polling loop avoids busy-waiting: workers block on semaphore acquisition rather than polling on a timer. The two-level concurrency system (global semaphore limits total in-flight inference requests across all models, per-model semaphore prevents any single model from monopolizing capacity) prevents starvation in multi-model deployments. Context separation ensures graceful shutdown preserves in-flight work — the polling context can be cancelled independently of job contexts, so `stopAccepting()` halts new intake without killing running jobs. Startup recovery scans the local workdir for incomplete jobs from previous crashes and either re-enqueues or completes them, providing crash resilience without external state.

Multi-tenancy is implemented at the application layer via HTTP header extraction (default: `X-MaaS-Username`). All database queries are scoped by tenant ID, and file storage paths use SHA256-derived folder names incorporating the tenant ID. However, the batch-gateway does **not** perform authentication itself — it expects an upstream auth proxy (such as kube-rbac-proxy or an external auth service) to validate identity and inject the tenant header. This is a deliberate design choice for the llm-d ecosystem, where the gateway sits behind a platform-level ingress stack.

The storage architecture is fully pluggable at two levels: metadata (PostgreSQL vs Redis/Valkey) and files (S3 vs filesystem). Both layers are wrapped with retry logic (exponential backoff with jitter) and OpenTelemetry tracing. The file store additionally implements `LimitedCountingReader` for streaming validation (enforcing both byte size and line count limits during upload without buffering the entire file). Redis operations use Lua scripts for atomicity (get-by-tags, get-by-expiry queries), and PostgreSQL uses pgx v5 with connection pooling and custom OTel span naming.

Observability is comprehensive. Both the apiserver and processor expose Prometheus metrics (HTTP request counters/histograms/in-flight gauges for the apiserver; job processing counters, queue wait histograms, token usage counters, worker utilization gauges, per-model request metrics, and E2E latency histograms for the processor). The Helm chart includes pre-built Grafana dashboards and PrometheusRule alerts for queue wait time, job failure rate, expired job rate, and worker saturation. OpenTelemetry tracing spans cover the full request lifecycle, with trace context propagated from the apiserver through batch tags into the processor's inference requests, enabling end-to-end distributed tracing across the batch job lifecycle.
