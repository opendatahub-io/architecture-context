## Purpose

**Short**: High-performance OpenAI-compatible batch inference gateway for processing large-scale batch jobs in Kubernetes environments alongside interactive workloads.

**Detailed**: Batch Gateway provides an OpenAI-compatible REST API (`/v1/batches` and `/v1/files`) for submitting, tracking, and managing batch inference jobs. It is designed for the llm-d ecosystem and RHOAI platform, enabling users to process up to 50,000 inference requests per batch job with model-aware scheduling, intelligent flow control, and SLO-driven prioritization.

The system consists of three independently scalable components: an API server that handles REST requests and file management, a batch processor that pulls jobs from a priority queue and dispatches inference requests to downstream gateways, and a garbage collector that cleans up expired jobs and files. The data layer is pluggable, supporting PostgreSQL for metadata storage, Redis/Valkey for priority queuing and event channels, and S3 or filesystem for file storage.

Authentication and authorization are delegated to the gateway layer (Kuadrant/Authorino or Envoy ext_authz), with multi-tenant isolation enforced at the application level via tenant-scoped data access. The processor supports mTLS for outbound inference connections and integrates with Gateway API InferenceExtension (GIE) for batch-vs-interactive workload balancing.

## Architectural Analysis

Batch Gateway follows a clean microservices architecture with three independently deployable components sharing a common data layer. The separation between API server (request acceptance), processor (async execution), and GC (lifecycle maintenance) enables independent scaling — processors can be horizontally scaled based on queue depth while the API server scales based on request rate. The GC intentionally runs as a single replica to avoid concurrent delete races.

The processor's concurrency model uses a three-tier semaphore hierarchy: worker tokens (job-level), global concurrency (total in-flight requests across all jobs), and per-model concurrency (per-model fairness). Acquisition order is per-model first, then global, to prevent starvation. A "double-release guard" callback detects semaphore violations and triggers graceful shutdown — this is a safety net against concurrency bugs that would otherwise lead to resource leaks. The context separation (`pollingCtx` for queue intake vs `jobBaseCtx` for in-flight work) ensures that shutdown halts new job acceptance while allowing in-flight jobs to complete within the `shutdownTimeout` deadline.

Crash recovery is architecturally significant. On startup, the processor scans its `workDir` for leftover job directories (evidence of a prior crash), inspects each job's database state, and makes phase-aware recovery decisions: finalizing jobs get their results re-uploaded, in-progress jobs with partial output get marked as failed with preserved results, and validating jobs get re-enqueued. This enables safe use of `emptyDir` volumes (which survive pod restarts but not rescheduling) and eliminates the class of "lost job" bugs common in queue-based systems.

The multi-tenant isolation model is application-level rather than infrastructure-level. The API server extracts the tenant ID from the last value of the configured HTTP header (exploiting Envoy's behavior of appending verified identity after client-supplied values), then scopes all database queries to that tenant. Cross-tenant access returns 404 rather than 403, preventing tenant enumeration attacks. This design delegates authentication to the gateway layer (Kuadrant/Authorino) while enforcing authorization in the application.

The inference client supports both global routing (single gateway for all models, typical for MaaS/multi-model platforms) and per-model routing (different gateways per model, typical for dedicated deployments). The client resolver deduplicates HTTP clients with identical configuration to avoid connection pool proliferation. SLO enforcement is end-to-end: the API server captures the SLO deadline at submission, the priority queue orders by SLO, and the processor injects `x-slo-ttft-ms` headers on inference requests to signal remaining time budget to the inference gateway.
