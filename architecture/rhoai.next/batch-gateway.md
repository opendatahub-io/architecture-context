## Purpose

**Short**: High-performance system for processing large-scale batch inference jobs in Kubernetes environments, providing an OpenAI-compatible API for submitting, tracking, and managing batch workloads.

**Detailed**: Batch Gateway is a multi-component service that enables efficient processing of batch inference workloads alongside interactive workloads in Kubernetes/OpenShift environments. It provides full schema parity with the OpenAI `/v1/batches` and `/v1/files` API endpoints, supporting up to 50,000 requests per batch job.

The system is composed of three independently scalable components: an API server for request handling, a batch processor for job execution with AIMD-based adaptive concurrency control, and a garbage collector for lifecycle management. It uses PostgreSQL for metadata persistence, Redis/Valkey for priority queueing, event channels, status tracking, and in-flight job monitoring, and S3 or filesystem backends for file storage.

The processor component implements model-aware scheduling with per-model execution plans, global and per-endpoint concurrency semaphores with AIMD (Additive Increase / Multiplicative Decrease) adaptive control, and fault-tolerant job recovery on container restart. Authentication and authorization are delegated to the gateway layer (Kuadrant/Authorino), while multi-tenant data isolation is enforced at the application level via tenant ID scoping on all data access.

## Architectural Analysis

The batch-gateway follows a clean separation-of-concerns architecture with three independently deployable components sharing a common data layer. The apiserver is stateless (horizontally scalable), the processor maintains local state only for plan files during execution (using emptyDir volumes), and the GC runs as a singleton. This design enables independent scaling of API throughput and batch processing capacity.

The concurrency control system is particularly sophisticated. The processor implements a three-tier semaphore hierarchy: a worker semaphore (controls job-level parallelism), a global concurrency semaphore (caps total in-flight inference requests), and per-endpoint adaptive semaphores with AIMD controllers. The AIMD implementation responds to 429 and 5xx signals from inference gateways by multiplicatively decreasing per-endpoint concurrency, then additively increasing after sustained success windows. This adaptive approach allows the processor to coexist with interactive workloads without requiring manual tuning of concurrency limits. The double-release guard on semaphores triggers a graceful shutdown rather than panicking, which is a notable defensive design choice.

The fault tolerance model is built around the GC reconciler and processor startup recovery. When a processor container crashes (OOM, panic), its in-flight jobs become orphans in Redis. The reconciler detects these via the `llmd_batch:inflight` hash, cross-references with the database, and either re-enqueues (if SLO is still active) or transitions to a terminal state. On restart, the processor scans its workdir for partial results from the previous incarnation and attempts to finalize them. This two-layer recovery (GC-driven orphan recovery + processor-local crash recovery) provides resilience without requiring external coordination.

Multi-tenancy is enforced at the application layer rather than through Kubernetes RBAC. Every data access operation is scoped by tenant ID, extracted from the `X-MaaS-Username` header (configurable). Cross-tenant requests return 404 rather than 403, preventing information leakage about other tenants' resources. File storage uses SHA256-hashed tenant IDs as folder names to prevent path traversal and ensure S3-compatible naming. Authentication and authorization are fully delegated to the gateway layer (Kuadrant/Authorino), which means the apiserver itself performs no auth — it trusts the gateway to have validated the request before it arrives.

The async dispatch mode (`dispatch_mode: async`) is an emerging integration with `llm-d-async`, where inference requests are submitted to a Redis sorted set and results are collected from a per-tenant list. This enables an external dispatcher to manage flow control and prioritization between batch and interactive workloads at a lower level than the processor's AIMD-based approach. The dual dispatch modes (sync and async) suggest the system is evolving toward a more decoupled architecture where the processor focuses on job lifecycle management while dispatching is handled externally.
