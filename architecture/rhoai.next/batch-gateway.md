## Purpose

**Short**: OpenAI-compatible batch inference API gateway for llm-d that manages large-scale batch job submission, processing, and lifecycle in Kubernetes environments.

**Detailed**: Batch Gateway is a high-performance system for processing large-scale batch inference jobs. It provides an OpenAI-compatible REST API (implementing `/v1/batches` and `/v1/files` endpoints) for submitting, tracking, and managing batch inference jobs that are dispatched to downstream LLM inference gateways.

The system is designed to facilitate efficient processing of batch workloads alongside interactive workloads. It minimizes interference with interactive workloads by implementing intelligent flow control via per-model and global concurrency semaphores, SLO-based priority scheduling, and downstream metrics monitoring. The processor supports both global (single gateway for all models) and per-model routing to inference endpoints.

The architecture comprises three independently-deployable Go binaries: an API server that accepts user requests, a batch processor that polls a priority queue and dispatches inference requests, and a garbage collector that periodically removes expired jobs and files. All three components share a pluggable data layer backed by PostgreSQL (metadata), Redis/Valkey (priority queue, events, status), and S3 or local filesystem (file storage). The system is deployed via a Helm chart with full support for Kubernetes, OpenShift, and RHOAI environments.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| batch-gateway-apiserver | Go HTTP Service | REST API server implementing OpenAI-compatible `/v1/batches` and `/v1/files` endpoints for batch job and file management |
| batch-gateway-processor | Go Background Worker | Polls priority queue, preprocesses batch input files into per-model execution plans, dispatches inference requests to downstream gateways with concurrency control, uploads results |
| batch-gateway-gc | Go Background Worker | Periodically scans for and removes expired batch jobs and files from both database and object storage |
| PostgreSQL | External Database | Persistent metadata storage for batch job and file records (spec, status, tags, tenant isolation) |

| S3 / Filesystem | External Storage | Stores batch input files (JSONL), output result files, and error files |

## Architectural Analysis

### Multi-Binary Decomposition with Shared Data Layer

The batch-gateway follows a clean separation of concerns across three binaries that share a common data layer. The API server is stateless and horizontally scalable, handling only request validation and metadata persistence. The processor is the compute-intensive component, performing file I/O, plan generation, and inference request dispatching. The garbage collector runs as a single-instance background process. All three share the same `internal/` packages for database, file storage, and utility code, composed via a builder-pattern `Clientset` that assembles the exact combination of clients each binary needs (e.g., the GC needs no queue or inference client).

The data layer is fully pluggable: metadata can be stored in PostgreSQL (default, recommended for production) or Redis/Valkey, while file storage supports S3-compatible backends or local filesystem with PVC. This is achieved through Go interfaces (`BatchDBClient`, `FileDBClient`, `BatchFilesClient`) with implementations wrapped in tracing and retry decorators using the decorator pattern. The Redis implementation uses embedded Lua scripts for atomic compound operations (tag-filtered queries, expiry checks), while PostgreSQL uses JSONB containment operators for tag queries.

### Processor Concurrency and Fault Tolerance

The processor implements a sophisticated concurrency control system using three layers of channel-based semaphores: a worker semaphore controlling how many jobs can be processed simultaneously, a global semaphore limiting total in-flight inference requests across all workers, and per-model semaphores preventing any single model from monopolizing resources. The acquisition order is deliberately local-first then global to prevent cross-model starvation. A double-release guard on each semaphore detects programming errors and triggers graceful shutdown rather than allowing undefined behavior.

Fault tolerance is built into the processor at multiple levels. On startup, a recovery phase scans the work directory for leftover artifacts from a previous crash, re-uploading partial results or re-enqueuing jobs as appropriate. During execution, a panic recovery handler with its own timeout (using a detached context) attempts to preserve partial results even when a goroutine panics. The finalization phase handles upload failures gracefully by falling back to a "failed" status while preserving whichever file uploads succeeded. Job re-enqueue on SIGTERM ensures no jobs are lost during rolling updates.

### Preprocessing and Plan-Based Execution

The processor uses a two-phase approach: preprocessing builds binary plan files (16-byte entries: 8-byte offset + 4-byte length + 4-byte FNV-32a prefix hash) sorted by system prompt hash, then execution reads these plans to dispatch requests. The prefix hash sorting groups requests with identical system prompts together, optimizing downstream inference engine KV cache utilization. This design means the processor can handle up to 50,000 requests per job while keeping a predictable memory footprint since only plan entries (not full request payloads) are held in memory.

### Security Model

Authentication is delegated to an external gateway (Kuadrant/Authorino) that authenticates users and injects a tenant header. The batch-gateway trusts this header and enforces tenant isolation at the data layer: every query is scoped to the tenant ID, and cross-tenant access returns 404 (not 403) to prevent information leakage. For inference gateway authentication, the processor supports multiple patterns: API key from a named file, API key from a Kubernetes secret mount at `/etc/.secrets/`, or service account token from the standard mount. mTLS with custom CA is supported for inference connections. Secrets are read via `os.OpenInRoot()` to prevent path traversal attacks. All pods run with read-only root filesystem, dropped capabilities, non-root user (UID 65532), and seccomp RuntimeDefault profile.

### Observability

The system provides comprehensive observability through three pillars. Prometheus metrics cover HTTP request rates, latencies, and in-flight counts for the API server, while the processor tracks job throughput by outcome, per-model execution durations, token usage, queue wait times, and worker utilization. Pre-built Grafana dashboards and PrometheusRule alerts (queue wait, failure rate, expired jobs, worker saturation) are included in the Helm chart. OpenTelemetry distributed tracing is integrated at all layers: HTTP handlers, database operations, file storage, and inference requests, with trace context propagated through batch job tags for cross-component correlation. The API server passes through OTel trace context into batch metadata, allowing the processor's inference requests to be correlated back to the original batch submission.
