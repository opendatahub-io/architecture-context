## Purpose

**Short**: OpenAI-compatible batch API gateway for llm-d that accepts, processes, and manages batch inference jobs.

**Detailed**: The batch-gateway implements the OpenAI Batch API specification for the llm-d inference platform. It provides a REST API for submitting batch inference jobs (JSONL files containing multiple inference requests), processes those jobs by dispatching individual requests to inference endpoints, and manages the full lifecycle including file upload/download, job queuing, progress tracking, cancellation, and garbage collection.

The system consists of three independently deployable binaries that share a common codebase: the **apiserver** handles REST API requests for batch and file CRUD operations, the **batch-processor** dequeues jobs from a priority queue and executes inference requests against model gateways with adaptive concurrency control (AIMD), and the **batch-gc** performs periodic garbage collection of expired batches and files plus orphan reconciliation. All three components share state through PostgreSQL (or Redis) for metadata and Redis for job queuing, events, and ephemeral status. File storage uses S3 (production) or local filesystem (development).

The batch-gateway is deployed via a Helm chart and supports Gateway API (HTTPRoute) for external ingress, TLS termination via cert-manager, and Prometheus monitoring via ServiceMonitor/PodMonitor resources. It integrates with llm-d inference endpoints and optionally with llm-d-async for asynchronous dispatch.

## Architectural Analysis

The batch-gateway follows a classic producer-consumer architecture with three independently scalable components sharing state through PostgreSQL/Redis and S3. The apiserver acts as the producer — validating requests, storing metadata, and enqueueing jobs onto a Redis sorted-set priority queue. The processor acts as the consumer — dequeuing jobs, downloading input files, dispatching inference requests, and uploading results. The GC runs independently to clean up expired resources and recover orphaned jobs. This separation enables horizontal scaling of processors without impacting API latency, and allows the GC to run as a singleton with different resource constraints.

The AIMD (Additive Increase / Multiplicative Decrease) concurrency control system is a noteworthy design pattern. Each inference endpoint gets an adaptive semaphore that starts at `per_endpoint` slots and adjusts dynamically: on 429/5xx responses, the limit is multiplicatively decreased by `backoff_factor` (default 0.5); after a full window of consecutive successes, it's additively increased by 1. The AIMD floor (`min`, default 5) prevents starvation. A global semaphore (`global`, default 100) caps total in-flight requests across all endpoints. This design adapts to inference backend capacity without manual tuning and provides automatic backpressure during overload.

Multi-tenancy is enforced at the application layer through a configurable HTTP header (default `X-MaaS-Username`). The tenant ID flows through request context into all database queries and file storage operations. Files are organized in S3 by tenant folder (SHA256 hash of tenant ID, prefixed with `t-`), providing namespace isolation at the storage level. Crucially, every retrieval and mutation operation validates that the resource's `tenant_id` matches the request's tenant — a mismatch returns 404 (not 403), hiding the existence of cross-tenant resources. This header-based approach assumes an upstream auth proxy (e.g., Gateway API with ext_authz or kube-rbac-proxy) has authenticated the user and injected the tenant header. The `LastHeaderValue` utility specifically handles the Envoy ext_authz pattern where the auth-injected header value is appended after any client-supplied value.

The dual-database strategy (PostgreSQL OR Redis for metadata) is interesting. Redis provides simpler deployment and lower latency via Lua scripts for atomic operations (CAS updates, filtered queries using JSONB-like tag containment). PostgreSQL offers ACID guarantees and native JSONB containment operators for tag-based queries. Both implementations conform to the same `DBClient` interface. The priority queue, event channels, status cache, and in-flight tracking always use Redis regardless of the metadata backend choice.

The dispatch mode flexibility (`sync` vs `async`) allows the processor to either call inference endpoints directly via HTTP or submit requests through the llm-d-async Redis queue system. In async mode, each model gateway must specify an `inference_pool_name` that maps to Redis sorted-set and list keys (`llm-d-async:requests:{pool}` and `llm-d-async:results:{pool}:$batch`). This supports scenarios where inference backends need request buffering or where the batch gateway should not hold connections open during long inference operations.
