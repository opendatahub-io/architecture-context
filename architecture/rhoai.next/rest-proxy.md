## Purpose

**Short**: A reverse-proxy server that translates KServe V2 RESTful HTTP inference requests into gRPC calls for ModelMesh Serving backends.

**Detailed**: The rest-proxy is a lightweight Go service that leverages [gRPC-Gateway](https://github.com/grpc-ecosystem/grpc-gateway) to provide a REST API facade over the KServe V2 gRPC Predict Protocol. It enables clients to send inference requests using the standard KServe V2 REST Predict Protocol (`/v2/models/{name}/infer`) to platforms that natively implement only the gRPC V2 Predict Protocol, such as ModelMesh Serving.

The proxy runs as a sidecar container alongside ModelMesh Serving pods. It listens on an HTTP port (default 8008) for incoming REST requests, translates them into protobuf gRPC messages, and forwards them to the local gRPC inference service (default port 8033). Response data is then marshaled back into JSON for the REST client. The proxy includes custom JSON marshaling/unmarshaling logic for handling tensor data types (BOOL, INT8-64, UINT8-64, FP16-64, BYTES) with support for multi-dimensional nested arrays, raw binary tensor contents, and base64-encoded byte data.

This component is not deployed standalone -- it is injected into ModelMesh Serving pods by the `modelmesh-serving` controller via the `restProxy.image` configuration in the `model-serving-config` ConfigMap. The upstream repository is `kserve/rest-proxy`; the Red Hat downstream fork (`red-hat-data-services/rest-proxy`) adds FIPS-compliant build configuration and Konflux CI integration.

## Architectural Analysis

The rest-proxy follows a clean sidecar proxy pattern. It is a single-purpose, stateless service that performs protocol translation between HTTP/JSON and gRPC/protobuf. The architecture is straightforward: the `proxy/main.go` file sets up a gRPC-Gateway mux that routes incoming HTTP requests to the appropriate gRPC methods on a co-located ModelMesh server. The generated gateway code in `gen/` handles the HTTP-to-gRPC translation mechanics, while the hand-written code in `proxy/` implements custom JSON marshaling for the KServe V2 tensor data model.

The custom marshaling logic (`proxy/marshaler.go`, `proxy/request.go`, `proxy/bytes.go`) is the most architecturally significant part of this codebase. The standard protobuf JSON marshaling is insufficient for the KServe V2 REST protocol because tensor data can be represented as flat arrays, nested multi-dimensional arrays, raw binary bytes, or base64-encoded strings. The `CustomJSONPb` marshaler handles all these cases, including flattening nested arrays on input and converting raw binary output contents into typed arrays. This custom marshaling is the primary source of complexity and the area most likely to have bugs (FP16 tensors are explicitly unsupported with a TODO comment).

The FIPS compliance story is well-structured for RHOAI: the `Dockerfile.konflux` uses `CGO_ENABLED=1` with `GOEXPERIMENT=strictfipsruntime` and `-tags strictfipsruntime`, ensuring the Go binary dynamically links against OpenSSL for FIPS-validated cryptography. However, the application-level TLS configuration includes a `REST_PROXY_SKIP_VERIFY` environment variable that can disable certificate verification on the gRPC upstream connection -- this is a security risk in production but is mitigated by the fact that the gRPC upstream is always `localhost` (same pod). The upstream `Dockerfile` notably uses `CGO_ENABLED=0` (static linking, no FIPS), confirming that FIPS compliance is a downstream-only concern.

The dependency on `sigs.k8s.io/controller-runtime` is worth noting -- it is used solely for the zap logger (`controller-runtime/pkg/log/zap`), which pulls in a significant dependency tree (k8s.io/apimachinery, etc.) for a non-Kubernetes service. This could be simplified by using `go.uber.org/zap` directly, though the indirect dependency footprint is the main cost rather than any runtime overhead.
