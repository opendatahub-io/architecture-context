## Purpose

**Short**: A reverse-proxy server that translates KServe V2 RESTful HTTP inference requests into gRPC calls for ModelMesh Serving backends.

**Detailed**: The rest-proxy component leverages the gRPC-Gateway library to create a reverse-proxy server that converts RESTful HTTP API calls conforming to the KServe V2 REST Predict Protocol into gRPC V2 Predict Protocol calls. This enables clients to send inference requests using standard HTTP/JSON to model-serving platforms that natively speak gRPC, such as ModelMesh Serving.

The proxy runs as a sidecar container within ModelMesh Serving deployments. It listens for incoming REST requests on a configurable port (default 8008/TCP), transforms the JSON payloads into protobuf messages using custom marshaling logic that handles tensor data types (BOOL, INT8-64, UINT8-64, FP16-64, BYTES), and forwards them as gRPC calls to the co-located inference service on a configurable port (default 8033/TCP). The response path performs the inverse transformation, converting gRPC protobuf responses back into JSON, including support for raw output byte arrays and base64-encoded binary tensor data.

The proxy supports optional TLS for both the inbound HTTP listener (using certificate/key files) and the outbound gRPC connection to the inference server. It is part of the KServe ecosystem and is maintained as a downstream fork by Red Hat for inclusion in RHOAI (Red Hat OpenShift AI).

## Architectural Analysis

The rest-proxy is a focused, single-purpose component with a clean architecture. It implements the gRPC-Gateway pattern: a `.proto` file (`grpc_predict_v2.proto`) defines the KServe V2 inference protocol, and `protoc` generates both the gRPC stubs and the HTTP-to-gRPC gateway code. The custom marshaling layer (`CustomJSONPb` in `proxy/marshaler.go` and `proxy/request.go`) sits on top of the generated gateway to handle KServe-specific JSON-to-protobuf transformations that the standard gRPC-Gateway marshaler cannot handle natively, such as multi-dimensional tensor data flattening and base64-encoded BYTES tensor support.

A notable design decision is that `sigs.k8s.io/controller-runtime` is imported solely for its `zap` logging package — the proxy has no controller or Kubernetes API interaction. This is an unnecessarily heavy dependency for a logging library and could be replaced with a direct `go.uber.org/zap` import to reduce the dependency graph. The component has no RBAC needs, no CRDs, and no Kubernetes client usage.

The TLS configuration has a potential security concern: the `REST_PROXY_SKIP_VERIFY` environment variable allows disabling TLS certificate verification on the gRPC upstream connection. While this is useful for development, it should be carefully controlled in production. Additionally, the `tls.Config` struct used for the gRPC connection has no `MinVersion` set, which defaults to TLS 1.0 in Go's standard library. In FIPS mode with `strictfipsruntime`, the Go runtime restricts to FIPS-approved TLS versions, mitigating this in RHOAI builds, but the upstream (non-Konflux) build does not have this protection. The FP16 tensor data type is explicitly unsupported and returns an error, which is a known limitation documented in the code with TODO comments.

The Konflux build (`Dockerfile.konflux`) differs significantly from the upstream `Dockerfile` in FIPS posture: the Konflux variant uses `CGO_ENABLED=1` with `GOEXPERIMENT=strictfipsruntime` and `-tags strictfipsruntime`, while the upstream uses `CGO_ENABLED=0` (static binary, no FIPS). The Konflux build also uses a digest-pinned `ubi9/go-toolset:1.23` builder and `ubi9/ubi-minimal` runtime, whereas the upstream uses `ubi9/ubi-micro:9.5`. The Tekton PipelineRun configures multi-architecture builds (x86_64, ppc64le, s390x, arm64) with hermetic gomod prefetch.
