## Purpose

**Short**: A reverse-proxy server that translates KServe V2 RESTful HTTP inference requests into gRPC V2 Predict Protocol calls for ModelMesh Serving.

**Detailed**: The rest-proxy is a lightweight Go service that leverages the grpc-gateway framework to provide a REST API facade over gRPC-based model inference servers. It implements the KServe V2 REST Predict Protocol, accepting JSON-formatted inference requests over HTTP and translating them into the corresponding gRPC V2 Predict Protocol messages before forwarding them to a co-located gRPC inference service (typically ModelMesh Serving).

The proxy runs as a sidecar container within ModelMesh Serving deployments, listening on a configurable HTTP port (default 8008) and forwarding requests to a localhost gRPC endpoint (default port 8033). It provides custom JSON marshaling/unmarshaling logic to handle the complex tensor data types defined in the KServe V2 inference protocol, including support for multi-dimensional nested arrays, raw byte tensor contents, base64-encoded BYTES tensors, and various numeric data types (BOOL, INT8-64, UINT8-64, FP16/32/64). The proxy supports optional TLS for both its HTTP listener and the upstream gRPC connection, with configurable certificate paths and TLS verification settings.

## Architectural Analysis

The rest-proxy follows a clean, single-purpose sidecar pattern. It exists solely to bridge the protocol gap between REST clients and the gRPC-native ModelMesh Serving infrastructure. The architecture is deliberately minimal — a single Go binary with no Kubernetes API interactions, no persistent state, and no controller logic. This makes it highly portable and easy to reason about from a security perspective.

The most architecturally significant aspect of this component is its custom JSON marshaling layer (`CustomJSONPb`, `request.go`, `bytes.go`). While grpc-gateway provides automatic REST-to-gRPC translation, the KServe V2 tensor data format requires specialized handling that the default protobuf JSON marshaler cannot provide. The custom marshaler handles: (1) multi-dimensional nested JSON arrays that must be flattened for protobuf wire format, (2) BYTES tensors that may be base64-encoded strings or raw numeric byte arrays, (3) raw output contents (`RawOutputContents`) that use binary-packed little-endian format with 4-byte size delimiters, and (4) inference parameters that map between JSON's dynamic types and protobuf's strongly-typed `InferParameter` oneof. This custom marshaling is the core value-add of the proxy beyond what grpc-gateway provides out of the box.

The TLS configuration is notable for its dual-mode design. The proxy can independently enable TLS on its HTTP listener (via `MM_TLS_KEY_CERT_PATH`/`MM_TLS_PRIVATE_KEY_PATH`) and on the upstream gRPC connection (via `REST_PROXY_USE_TLS`). The gRPC TLS mode also supports `InsecureSkipVerify` for environments where certificate verification is impractical (e.g., self-signed certs in development). The shared `MM_TLS_*` environment variable naming convention indicates tight coupling with ModelMesh Serving's TLS configuration.

The Konflux build pipeline (`Dockerfile.konflux`) enables FIPS-compliant builds via `GOEXPERIMENT=strictfipsruntime` and `CGO_ENABLED=1`, using the `ubi9/go-toolset` builder and `ubi9/ubi-minimal` runtime base images. This is critical for RHOAI's FedRAMP and FIPS 140-2 compliance requirements. The non-Konflux Dockerfile uses `CGO_ENABLED=0` and targets `ubi9/ubi-micro`, reflecting the upstream community build that does not require FIPS compliance. Multi-architecture builds (amd64, arm64, ppc64le, s390x) are supported in both the GitHub Actions workflow and the Konflux Tekton pipelines.

The configurable max gRPC message size (default 16 MB via `REST_PROXY_GRPC_MAX_MSG_SIZE_BYTES`) is an important operational parameter — large model inputs (especially image or audio tensors) can easily exceed the default gRPC 4 MB limit, and this proxy configuration propagates to the `grpc.MaxCallRecvMsgSize` dial option.
