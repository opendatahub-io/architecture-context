## Purpose

**Short**: A gRPC-to-REST reverse proxy that translates KServe V2 REST Predict Protocol requests into gRPC V2 Predict Protocol calls for ModelMesh Serving.

**Detailed**: The rest-proxy component implements a protocol translation layer between REST/HTTP clients and gRPC-based model serving backends. It uses the gRPC-Gateway library to automatically generate a reverse proxy from the KServe V2 Inference Protocol protobuf definition (`grpc_predict_v2.proto`), enabling clients to send inference requests using the standard KServe V2 REST API while the backend ModelMesh Serving infrastructure communicates over gRPC.

The proxy runs as a sidecar container alongside ModelMesh Serving pods, listening on HTTP port 8008 (configurable) and forwarding translated requests to the local gRPC inference service on port 8033 (configurable). It handles complex data type marshalling including multi-dimensional tensor data, raw byte content, base64-encoded BYTES tensors, and all standard numeric types (BOOL, INT8/16/32/64, UINT8/16/32/64, FP32, FP64). The proxy supports both plaintext and TLS modes for both its HTTP listener and its upstream gRPC connection.

This is the Red Hat downstream fork of the upstream `kserve/rest-proxy` repository, maintained as part of the ModelMesh Serving stack within Red Hat OpenShift AI (RHOAI). It is built by Konflux with FIPS-compliant build flags for Red Hat product delivery.

## Architectural Analysis

The rest-proxy is architecturally simple but fills a critical role in the ModelMesh Serving stack: it bridges the gap between REST-speaking clients and the gRPC-native ModelMesh inference infrastructure. The design leverages protoc-generated gRPC-Gateway code to minimize hand-written routing logic — the `.proto` file is the single source of truth for both the gRPC service interface and the REST-to-gRPC mapping.

The custom marshaling layer (`marshaler.go`, `request.go`, `bytes.go`) is the most complex part of the codebase. It handles the impedance mismatch between JSON tensor representations (nested arrays, base64-encoded byte strings, multi-dimensional shapes) and the flat protobuf tensor contents. This includes raw output content parsing (`readBytes`, `splitRawBytes`), multi-dimensional array flattening for nested JSON inputs, and base64 encoding/decoding for BYTES-type tensors. FP16 tensors are explicitly unsupported with TODO comments, which is a known limitation.

The proxy is designed as a stateless, single-purpose sidecar with minimal configuration (7 environment variables). It has no health check endpoint, no metrics endpoint, and no readiness/liveness probes of its own — it relies on the parent ModelMesh pod's lifecycle management. The `controller-runtime` dependency is used solely for its zap logger, which is an unusually heavyweight dependency for a non-controller application. The TLS configuration supports optional server-side TLS for the HTTP listener and optional client-side TLS for the gRPC upstream connection, with a configurable `InsecureSkipVerify` option that should be used cautiously in production.

The divergence between the upstream `Dockerfile` (CGO_ENABLED=0, no FIPS, `ubi-micro` runtime) and the Konflux `Dockerfile.konflux` (CGO_ENABLED=1, FIPS-compliant, `ubi-minimal` runtime) reflects the standard Red Hat downstream hardening pattern. The Konflux build produces a larger binary (dynamic linking) but gains FIPS-validated cryptography through the OpenSSL backend.
