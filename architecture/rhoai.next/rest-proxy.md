## Purpose

**Short**: A reverse-proxy server that translates KServe V2 RESTful HTTP API calls into gRPC, enabling REST clients to communicate with inference servers that speak the gRPC V2 Predict Protocol.

**Detailed**: The rest-proxy leverages [gRPC-Gateway](https://github.com/grpc-ecosystem/grpc-gateway) to create a reverse-proxy that accepts incoming HTTP/REST requests conforming to the [KServe V2 REST Predict Protocol](https://github.com/kserve/kserve/blob/master/docs/predict-api/v2/required_api.md#httprest) and translates them into gRPC calls following the [KServe V2 gRPC Predict Protocol](https://github.com/kserve/kserve/blob/master/docs/predict-api/v2/required_api.md#grpc). This allows inference clients that only support REST to interact with gRPC-native model serving backends.

The proxy runs as a sidecar container alongside ModelMesh Serving pods. It is configured via the `restProxy.image` setting in the `model-serving-config` ConfigMap used by [ModelMesh Serving](https://github.com/kserve/modelmesh-serving). The proxy handles custom JSON marshaling/unmarshaling for tensor data types (including BOOL, INT8-64, UINT8-64, FP16/32/64, and BYTES), supports base64-encoded binary content, and can operate with or without TLS for both the incoming HTTP listener and the outgoing gRPC connection to the inference service.

The component is originally from IBM (kserve/rest-proxy upstream) and is carried downstream by Red Hat for inclusion in RHOAI's ModelMesh Serving stack.

## Architectural Analysis

The rest-proxy is architecturally simple — a single Go binary acting as a protocol translation sidecar. Its primary value is enabling REST clients to interact with gRPC-only inference servers in the ModelMesh Serving stack. The architecture follows the sidecar pattern: the proxy runs alongside the inference server in the same pod, communicating over localhost, which eliminates network latency and the need for service discovery between proxy and backend.

The custom JSON marshaler/unmarshaler (`CustomJSONPb` in `proxy/marshaler.go` and `proxy/request.go`) is a significant piece of the codebase, handling the complex tensor data type conversions between JSON arrays and protobuf tensor representations. The code supports 13 tensor data types (BOOL, INT8-64, UINT8-64, FP16, FP32, FP64, BYTES) and handles both flat and nested array representations, raw byte content, and base64-encoded binary data. FP16 tensors are explicitly unsupported with a TODO marker. The byte handling (`proxy/bytes.go`) includes custom "raw" JSON parsing for performance, avoiding standard library JSON parsing for byte arrays.

The TLS configuration is notable for its dual-mode operation: the proxy can optionally enable TLS on both the inbound HTTP listener (via `MM_TLS_KEY_CERT_PATH` / `MM_TLS_PRIVATE_KEY_PATH`) and the outbound gRPC connection (via `REST_PROXY_USE_TLS`). The `REST_PROXY_SKIP_VERIFY` flag allows disabling certificate verification on the gRPC upstream, which is useful for development but should be avoided in production. The proxy does not implement any authentication itself — it relies on the platform's ingress layer and ModelMesh to handle auth.

The gRPC message size limit defaults to 16 MiB (`maxGrpcMessageSizeBytes = 16777216`) but is configurable via the `REST_PROXY_GRPC_MAX_MSG_SIZE_BYTES` environment variable. This is important for large inference payloads (e.g., image data). The `controller-runtime` dependency is used solely for its `zap` structured logging package — the component is not a Kubernetes operator and does not interact with the Kubernetes API.

The Konflux build (`Dockerfile.konflux`) is properly configured for FIPS compliance with `CGO_ENABLED=1`, `GOEXPERIMENT=strictfipsruntime`, and `-tags strictfipsruntime`. This contrasts with the upstream `Dockerfile` which uses `CGO_ENABLED=0` and produces a statically linked binary without FIPS support. The Konflux build also uses digest-pinned base images for reproducibility, while the upstream Dockerfile uses tag-based references.
