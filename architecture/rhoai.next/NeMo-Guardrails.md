## Purpose

**Short**: NeMo Guardrails is a programmable guardrails toolkit that adds safety controls (input/output filtering, content moderation, hallucination detection, sensitive data detection) to LLM-based conversational systems, exposed as an OpenAI-compatible API server.

**Detailed**: NeMo Guardrails is an open-source toolkit originally developed by NVIDIA for adding programmable guardrails to LLM-based conversational applications. This repository is the Red Hat downstream fork (via TrustyAI) that adds OpenShift-specific customizations: a UBI9-based container image, a fork-specific `/v1/guardrail/checks` endpoint for stateless message validation, request header forwarding to LLM providers with auth token redaction, and a guardrail profile system that filters closed-source guardrails from the build.

The server exposes an OpenAI-compatible chat completions API (`/v1/chat/completions`) that wraps LLM calls with configurable guardrails pipelines. Guardrails are defined using the Colang domain-specific language (v1.0 and v2.x), which specifies conversation flows, input/output validation rules, and action triggers. The toolkit includes 30+ built-in guardrail types in its library, covering content safety, hallucination detection, jailbreak injection detection, sensitive data detection (PII via Presidio), topic safety, regex-based filtering, and integrations with third-party moderation services (ActiveFence, LlamaGuard, etc.). The downstream build uses the "opensource" profile, which strips closed-source guardrails at build time via `filter_guardrails.py`.

The component runs as a stateless FastAPI service on port 8000, relying on external LLM providers (via LangChain) for inference. It pre-downloads embedding models (Sentence Transformers, FastEmbed, spaCy) at build time and caches them in the container image. Optional OpenTelemetry auto-instrumentation is supported via the entrypoint script when `OTEL_EXPORTER_OTLP_ENDPOINT` is set.

## Architectural Analysis

**Fork Architecture and Guardrail Profiles**: The Red Hat downstream fork introduces a profile-based guardrail filtering system (`scripts/filter_guardrails.py` + `scripts/provider-list.yaml`) that removes closed-source guardrail modules at Docker build time. The "opensource" profile strips 15 modules (ActiveFence, AI Defense, Clavata, Cleanlab, CrowdStrike, Fiddler, GCP, Pangea, PatronusAI, PolicyAI, PrivateAI, Prompt Security, Trend Micro, Autoalign, and jailbreak_detection due to upstream bugs). This means the RHOAI image ships approximately 14 open-source guardrail types: attention, content_safety, factchecking, gliner, guardrails_ai, hallucination, injection_detection, llama_guard, regex, self_check, sensitive_data_detection, topic_safety, and utility modules. This filtering is architecturally significant because it determines the security capabilities available in production.

**Header Forwarding and Auth Isolation**: The fork implements a careful header forwarding mechanism (`nemoguardrails/actions/llm/utils.py:62-88`) that distinguishes between infrastructure headers and application-level auth. The inbound `Authorization` header (typically K8s/proxy auth) is explicitly excluded from forwarding. Instead, `X-Authorization` is forwarded as `Authorization` to the LLM provider, but only for LLM instances that registered as needing runtime auth at init time. This design prevents accidental credential leakage between the platform auth layer and the upstream LLM provider. Auth tokens are redacted from callback logs (`nemoguardrails/logging/callbacks.py:41-53`) using a regex that replaces Bearer token values with `[REDACTED]`.

**Multi-Architecture Build Strategy**: The Konflux Dockerfile implements an elaborate multi-stage build with architecture-specific compilation stages for ppc64le support. Stages 2-6 conditionally build OpenBLAS, PyTorch, Tiktoken, ONNX Runtime, and hf_xet from source on ppc64le (since pre-built wheels are unavailable), while x86_64 uses standard PyPI wheels. This adds significant build complexity but enables the service to run on IBM Power Systems. The final image uses the same UBI9 Python 3.12 runtime regardless of architecture.

**Stateless Design with Baked-in Models**: The server pre-downloads embedding models (Sentence Transformers, FastEmbed, spaCy en_core_web_lg) at build time and caches them inside the container image (`/app/.cache/`). This makes the service fully air-gap capable once deployed — it does not need to download models at runtime. The spaCy model is locked via `artifacts.lock.yaml` and prefetched by Hermeto/cachi2. However, the LLM provider connection is still required at runtime; the guardrails evaluate against but do not replace the external LLM.

**MD5 FIPS-Aware Fallback**: The codebase uses `hashlib.md5` for cache key generation in two locations (`utils.py:411` and `embeddings/cache.py:71`). Notably, `utils.py` includes a FIPS-aware fallback: it attempts to call `hashlib.md5(b"")` and falls back to SHA-256 if MD5 raises a `ValueError` (which happens in FIPS mode). This is a good practice for non-cryptographic hash usage, though `embeddings/cache.py` lacks this fallback and would fail in strict FIPS mode.
