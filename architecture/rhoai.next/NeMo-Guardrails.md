## Purpose

**Short**: NeMo Guardrails is NVIDIA's open-source toolkit for adding programmable safety guardrails to LLM-based conversational systems, packaged as a FastAPI server for the Red Hat OpenShift AI platform.

**Detailed**: NeMo Guardrails provides a comprehensive framework for enforcing safety constraints on Large Language Model (LLM) applications. It implements a programmable guardrails pipeline that intercepts user inputs and model outputs, applying configurable safety checks including content moderation, jailbreak detection, PII masking, hallucination detection, and topic safety enforcement. The system uses a domain-specific language called Colang (v1.0 and v2.x) to define guardrail flows, and supports 27+ built-in guardrail types that integrate with external safety APIs (ActiveFence, Cisco AI Defense, CrowdStrike, Pangea, etc.) or use local ML models (Presidio, YARA, spaCy).

This repository is the Red Hat downstream fork (red-hat-data-services/NeMo-Guardrails) of the upstream NVIDIA NeMo Guardrails project, maintained via a multi-remote chain: upstream (NVIDIA) → trustyai-fork → midstream (opendatahub-io) → downstream (red-hat-data-services). The fork adds Red Hat-specific customizations including a UBI9-based container image built via Konflux, the `/v1/guardrail/checks` endpoint for standalone rail evaluation without LLM generation, API header forwarding for multi-tenant deployments, and a guardrail profile system that filters closed-source guardrails from the build. The server exposes an OpenAI-compatible API on port 8000 and ships with pre-downloaded NLP models (spaCy, sentence-transformers, fastembed) baked into the container image for air-gapped operation.

## Architectural Analysis

NeMo Guardrails follows a pipeline architecture where user messages flow through configurable input rails before reaching the LLM, and LLM responses pass through output rails before being returned to the client. The pipeline is defined using Colang, a domain-specific language with two versions: Colang v1.0 (imperative, `define`-based) and Colang v2.x (declarative, `import`-based). The runtime selects the appropriate engine based on configuration, and both versions share the same action system and LLM integration layer. This dual-version support adds complexity but enables backward compatibility with existing guardrail definitions while encouraging migration to the more expressive v2.x syntax.

The server is architecturally a thin FastAPI wrapper around the `LLMRails` engine class, which orchestrates configuration loading, runtime initialization, action dispatch, and LLM calls. The action system uses a decorator pattern (`@action()`) with filesystem-based discovery — actions are automatically loaded from the `nemoguardrails/library/`, `nemoguardrails/actions/`, and user-provided directories. This makes the guardrail library highly extensible but also means the container image must include all potential guardrail dependencies at build time. The `scripts/filter_guardrails.py` build-time script addresses this by removing closed-source guardrail directories based on a profile configuration (`opensource` vs `all`), ensuring the Red Hat build only ships open-source guardrails.

The fork-specific `/v1/guardrail/checks` endpoint is architecturally significant — it enables standalone safety evaluation without requiring an upstream LLM. This decouples the guardrails from the inference pipeline, allowing them to be used as a sidecar safety service that validates messages from any source. The endpoint routes messages based on role: `user`/`system` messages go through input rails, `assistant` messages through output rails, and `tool` messages through tool input rails. This role-based routing is implemented in `nemoguardrails/server/api.py` (lines 1026-1045) and enables flexible deployment patterns where guardrails can be applied at different stages of a multi-service pipeline.

The container build (`Dockerfile.konflux`) is a complex multi-stage build with architecture-specific stages for ppc64le support. Stages 2-6 build OpenBLAS, PyTorch, tiktoken, onnxruntime, and hf-xet from source for ppc64le (since pre-built wheels are unavailable for that architecture), while x86_64 and arm64 use pre-built wheels. The final image includes pre-downloaded NLP models (spaCy en_core_web_lg, sentence-transformers, fastembed models) baked in via `scripts/pre_download_required_models.py`, enabling air-gapped operation. The `artifacts.lock.yaml` locks the spaCy model download URL with a SHA256 checksum for supply chain integrity, and the Tekton pipeline uses Hermeto (cachi2) prefetch for this artifact. However, the overall build is not hermetic (`hermetic: false` in the PipelineRun), meaning pip install operations still reach the internet during build.

The server's security model relies entirely on the platform for authentication and TLS termination — the application listens on plaintext HTTP port 8000 with no built-in auth. In a RHOAI deployment, the platform operator would typically front this with a kube-rbac-proxy sidecar or place it behind the platform's Gateway API ingress with appropriate authentication. The CORS middleware is disabled by default and requires explicit opt-in via environment variables. Path traversal protections are implemented in the configuration loading code (regex validation and resolved-path checking), and thread IDs require a minimum 16-character length. Request headers are forwarded to upstream LLM providers to support multi-tenant API key routing, with auth token redaction in logs.
