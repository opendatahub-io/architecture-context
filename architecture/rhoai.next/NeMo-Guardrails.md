## Purpose

**Short**: NeMo Guardrails is an open-source toolkit for adding programmable guardrails (input/output safety, topic control, hallucination detection, sensitive data filtering) to LLM-based conversational systems, exposed as an OpenAI-compatible FastAPI server.

**Detailed**: NeMo Guardrails provides a programmable framework for controlling and securing interactions with large language models. It implements a pipeline of configurable "rails" — input rails that validate user messages before they reach the LLM, output rails that check LLM responses before they reach the user, and tool rails that validate tool calls and responses. The framework uses NVIDIA's Colang domain-specific language (v1.0 and v2.x) to define conversational flows and guardrail logic, backed by LangChain for LLM provider abstraction.

This repository is the Red Hat / TrustyAI downstream fork (`red-hat-data-services/NeMo-Guardrails`) of the NVIDIA upstream. Key fork-specific additions include: (1) a `/v1/guardrail/checks` endpoint for validating messages against rails without generating LLM responses, (2) header forwarding for X-* headers to LLM providers with auth token redaction, (3) UBI9-based Konflux container builds with baked-in embedding models, (4) open/closed-source guardrail filtering via build profiles, and (5) OpenTelemetry auto-instrumentation support via the entrypoint script.

The server ships as a single container image (`odh-nemo-guardrails`) built on Red Hat UBI9 with Python 3.12. At runtime, guardrail configurations are mounted via ConfigMaps specifying `config.yaml` (model/rail definitions) and `rails.co` (Colang flow definitions). The server exposes an OpenAI-compatible chat completions API on port 8000, enabling drop-in integration as a guardrails proxy in front of any LLM endpoint.

## Architectural Analysis

NeMo Guardrails implements a layered pipeline architecture centered on the `LLMRails` engine class. Each incoming request passes through a configurable sequence of rail types: input rails → dialog/retrieval rails → LLM generation → output rails. Each rail type can invoke the LLM itself (e.g., for self-check guardrails that ask the LLM "does this message contain harmful content?"), invoke local models (e.g., FastEmbed for similarity matching against known-bad patterns), or apply heuristic checks (e.g., regex-based injection detection). The pipeline is orchestrated by the Colang runtime, which interprets domain-specific flow definitions to determine which rails fire and in what order.

The fork introduces a significant architectural addition: the `/v1/guardrail/checks` endpoint, which decouples guardrail evaluation from LLM response generation. This allows RHOAI platform components to validate messages against safety rails without triggering the full chat completion pipeline — useful for pre-flight content checks where the caller already has an LLM response and only needs safety validation. The endpoint processes each message independently based on its role (user → input rails, assistant → output rails, tool → tool_input rails) and returns per-message and aggregate rail status.

The build system uses a profile-based guardrail filtering mechanism (`scripts/filter_guardrails.py` + `scripts/provider-list.yaml`) that removes closed-source guardrail modules at container build time. The `opensource` profile (default for RHOAI) strips 15 proprietary guardrails (ActiveFence, AI Defense, Cleanlab, etc.), leaving only open-source implementations. This ensures the shipped container contains no code requiring proprietary API keys or licenses. The remaining open-source guardrails include: content safety, factchecking, hallucination detection, injection detection, regex-based filtering, self-check (LLM-based), sensitive data detection (via Presidio), topic safety, attention-based checks, GLiNER NER-based detection, Guardrails AI integration, and LLama Guard.

The Konflux Dockerfile (`Dockerfile.konflux`) is a multi-stage build with 8 stages, primarily to support ppc64le architecture by building OpenBLAS, PyTorch, tiktoken, onnxruntime, and hf-xet from source. For x86_64, most wheels come from PyPI. The final image bakes in pre-downloaded embedding models (FastEmbed `all-MiniLM-L6-v2`, spaCy `en_core_web_lg`, NLTK punkt tokenizer) so the container can function in disconnected/air-gapped environments without network access to model registries. The entrypoint script (`scripts/entrypoint.sh`) validates that the required ConfigMap files (`config.yaml`, `rails.co`) exist before starting the server, and optionally wraps the server process with OpenTelemetry auto-instrumentation if `OTEL_EXPORTER_OTLP_ENDPOINT` is set.

The LLM provider abstraction is dual-layered: LangChain handles the actual LLM API calls (supporting OpenAI, Azure, Anthropic, NIM, Cohere, and any LangChain-compatible provider), while a separate `PROVIDERS` registry in `server/schemas/utils.py` handles the `/v1/models` endpoint for listing available models from the upstream provider. The server supports dynamic model selection via the `model` field in the request body, with the actual LLM engine determined by the `MAIN_MODEL_ENGINE` environment variable and base URL by `MAIN_MODEL_BASE_URL`. This allows a single guardrails server instance to proxy requests to different LLM backends without reconfiguration.
