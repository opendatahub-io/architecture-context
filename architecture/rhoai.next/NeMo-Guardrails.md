## Purpose

**Short**: NeMo Guardrails is an open-source toolkit for adding programmable safety guardrails to LLM-based conversational systems, deployed as a FastAPI server that intercepts, validates, and enforces policies on LLM interactions.

**Detailed**: NeMo Guardrails provides a programmable framework for controlling LLM-based applications through configurable input/output safety rails. The system acts as a middleware layer between clients and LLM providers, enforcing content safety policies, detecting jailbreak attempts, performing sensitive data detection, and validating both user inputs and model outputs against configurable rules written in NVIDIA's Colang domain-specific language (v1.0 and v2.x).

This is the Red Hat / TrustyAI fork of NVIDIA's upstream NeMo Guardrails, maintained at `red-hat-data-services/NeMo-Guardrails`. Key fork customizations include: a `/v1/guardrail/checks` endpoint for checking messages against rails without LLM generation, header forwarding with auth token redaction, build-time guardrail profile filtering (open-source vs. all), UBI9-based multi-stage container builds with pre-baked ML models, and multi-architecture support (x86_64, arm64, ppc64le). The server exposes an OpenAI-compatible chat completions API, enabling drop-in integration with existing LLM client tooling.

The component integrates with multiple upstream LLM providers (OpenAI, Azure OpenAI, Anthropic, Cohere, NVIDIA NIM, vLLM, HuggingFace) and embedding providers, and ships pre-downloaded ML models (spaCy, Sentence Transformers, FastEmbed, NLTK) for offline inference of safety checks. It supports OpenTelemetry-based distributed tracing with GenAI semantic conventions for observability.

## Architectural Analysis

NeMo Guardrails follows a middleware architecture pattern where it sits between LLM clients and LLM providers, intercepting requests and applying configurable safety policies. The core abstraction is the **rail** — a programmable check written in NVIDIA's Colang DSL that can inspect, modify, or block messages at various stages of the LLM interaction lifecycle. The server presents an OpenAI-compatible API surface, making it a transparent proxy that can be inserted into existing LLM pipelines without client-side changes.

The engine selection logic is notable: the `Guardrails` wrapper class (`guardrails/guardrails.py`) automatically chooses between two execution engines based on the configured rails. If all flows are I/O-only (pure content safety checks without dialogue management), it selects the lightweight `IORails` engine optimized for stateless validation. Otherwise, it falls back to the full `LLMRails` engine with Colang runtime support, dialogue state management, and multi-turn conversation handling. This bifurcation allows high-throughput content safety checking without the overhead of the full dialogue engine.

The build-time guardrail profiling system (`scripts/filter_guardrails.py` + `scripts/provider-list.yaml`) is a Red Hat fork customization that physically removes closed-source guardrail modules from the container image during build. The `opensource` profile excludes 15 proprietary providers (ActiveFence, AI Defense, CrowdStrike, Fiddler, etc.), leaving only open-source guardrails (content safety, self-check, regex, hallucination, sensitive data detection, topic safety, etc.). This ensures the RHOAI image ships only with freely redistributable code and models.

The model pre-caching strategy (`scripts/pre_download_required_models.py`) is architecturally significant: all ML models required by the selected guardrail profile are downloaded and embedded in the container image at build time. This eliminates runtime network dependencies for inference — the server can run air-gapped for content safety checks. Models are cached in five separate directories (HuggingFace, Transformers, Sentence Transformers, NLTK, FastEmbed) and cleaned of `.pyc`, `__pycache__`, and `.git` artifacts to reduce image size. The `artifacts.lock.yaml` file locks the spaCy `en_core_web_lg` model to a specific SHA256 checksum, and the model is fetched via Hermeto/cachi2 in Konflux builds for supply chain integrity.

The multi-architecture support in `Dockerfile.konflux` is complex: ppc64le requires building PyTorch, tiktoken, ONNX Runtime, OpenBLAS, and hf-xet from source (stages 2-6) because pre-built wheels are unavailable. The build uses 8 Docker stages with conditional execution — non-ppc64le architectures skip these source builds entirely and install pre-built wheels from PyPI. This architecture-conditional logic is implemented via `if [ "$TARGETARCH" = "ppc64le" ]` guards throughout the Dockerfile.

The OpenTelemetry integration follows library-only best practices: the application uses only the OpenTelemetry API (`opentelemetry-api`), never the SDK directly. Tracer configuration is deferred to the deployment environment. The entrypoint script (`scripts/entrypoint.sh`) conditionally wraps the server with `opentelemetry-instrument` auto-instrumentation when `OTEL_EXPORTER_OTLP_ENDPOINT` is set, providing zero-code-change observability. Custom span attributes follow the emerging GenAI semantic conventions (`gen_ai.request.model`, `gen_ai.usage.input_tokens`, etc.) alongside guardrails-specific attributes (`rail.type`, `rail.name`, `action.name`).
