## Purpose

**Short**: NeMo Guardrails is a programmable guardrails toolkit for adding safety, security, and content moderation to LLM-based conversational systems.

**Detailed**: NVIDIA NeMo Guardrails provides a framework for defining and enforcing input/output safety rails on LLM interactions. The toolkit intercepts user prompts and LLM responses, applying configurable checks such as content safety, jailbreak detection, topic safety, sensitive data detection, hallucination detection, and injection prevention. Rails are defined using Colang (v1.0 and v2.x), a domain-specific language for conversational flow control.

This repository is the Red Hat downstream fork (`red-hat-data-services/NeMo-Guardrails`) of the NVIDIA upstream, with TrustyAI-specific customizations. Key fork additions include a `/v1/guardrail/checks` endpoint for standalone guardrail validation without LLM generation, header forwarding for upstream LLM provider authentication, a UBI9-based multi-stage Dockerfile with baked-in models, guardrail profile filtering (open-source vs closed-source), and OpenTelemetry auto-instrumentation support. The server exposes an OpenAI-compatible REST API (FastAPI/Uvicorn) on port 8000 and supports both streaming and non-streaming responses.

The component is deployed as a standalone container within the RHOAI platform, typically configured via ConfigMaps containing Colang flow definitions and YAML configuration. It integrates with external LLM providers (OpenAI, NVIDIA NIM, Anthropic, Azure, etc.) and embedding services for semantic similarity-based guardrail evaluation.

## Architectural Analysis

NeMo Guardrails follows a layered pipeline architecture where user messages flow through a configurable sequence of input rails, LLM generation, and output rails. The system uses two runtime engines: the full **LLMRails** engine, which supports the complete Colang language (v1.0 and v2.x) with dialog management, knowledge base retrieval, and complex flow control; and the optimized **IORails** engine, which provides a fast path for configurations that only use input/output content safety checks without dialog rails. The `Guardrails` wrapper class auto-selects the appropriate engine based on the configured flows, preferring IORails when possible for reduced latency.

The fork-specific **`/v1/guardrail/checks`** endpoint is architecturally significant — it enables a "check-only" mode where messages are validated against rails without invoking LLM generation. This supports use cases where the platform needs to pre-validate user input or post-validate assistant output independently of the generation pipeline. The endpoint routes messages to input, output, or tool_input rails based on the message role, and returns a structured `RailsResult` with PASSED/MODIFIED/BLOCKED status.

The **guardrail library** is the most extensible part of the architecture, containing 28 guardrail type implementations organized as self-contained modules with Colang flow definitions (`.co` files) and Python action implementations. The build-time **profile filtering** (`scripts/filter_guardrails.py` + `scripts/provider-list.yaml`) removes closed-source guardrails from the image, leaving only open-source implementations (13 of 28 guardrails) in the default `opensource` profile. This is critical for RHOAI distribution where closed-source third-party integrations (ActiveFence, CrowdStrike, Trend Micro, Pangea, etc.) are excluded.

The **IORails engine** (`nemoguardrails/guardrails/`) is a fork-specific addition that provides a streamlined, production-optimized inference path. It consists of three components: `ModelManager` (manages per-model HTTP clients with retry), `RailsManager` (orchestrates sequential or parallel rail execution), and `ModelEngine`/`APIEngine` (raw HTTP clients to OpenAI-compatible endpoints and arbitrary REST APIs). This bypasses the full Colang runtime overhead for simple content safety configurations.

The **header forwarding pattern** is a critical security design decision — the server extracts Authorization headers from incoming requests and forwards them to upstream LLM providers, enabling per-request authentication delegation without storing API keys in the server. This coexists with the environment variable-based API key pattern (`api_key_env_var` in model config) for deployments where a shared key is acceptable. The `logging/callbacks.py` module intentionally redacts auth tokens from invocation parameter logs to prevent credential leakage.

The **multi-architecture build** (x86_64, arm64, ppc64le) in the Konflux Dockerfile is notable for its complexity. The ppc64le architecture requires building PyTorch, tiktoken, ONNX Runtime, OpenBLAS, and hf_xet from source due to missing pre-built wheels, adding 6 build stages beyond the standard 2-stage build. This makes the ppc64le image build significantly slower (4-hour pipeline timeout) but ensures full platform coverage.
