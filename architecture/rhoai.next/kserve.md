## Purpose

**Short**: KServe is a Kubernetes-native model serving platform that provides CRDs and controllers for deploying, scaling, and managing machine learning inference services with support for serverless, raw Kubernetes, and LLM-optimized deployment modes.

**Detailed**: KServe is the core model serving component of Red Hat OpenShift AI (RHOAI), providing a declarative API for deploying ML models as inference services on Kubernetes. It defines multiple CRDs including InferenceService (v1beta1) for traditional ML model serving, LLMInferenceService (v1alpha2) for large language model serving with disaggregated prefill/decode architectures, InferenceGraph (v1alpha1) for routing across multiple models, and LocalModelCache for node-local model caching.

The operator manages the full lifecycle of inference workloads: it creates Deployments or LeaderWorkerSets for model serving pods, configures autoscaling via HPA, KEDA, or the Workload Variant Autoscaler (WVA), sets up networking through Gateway API HTTPRoutes or Istio VirtualServices, injects storage initializer init containers for model downloading from S3/GCS/Azure/HuggingFace/OCI registries, and manages TLS certificates for internal communication. On OpenShift, it integrates with the platform's service CA, SecurityContextConstraints, and Istio service mesh via DestinationRules.

KServe ships as multiple container images: the main kserve-controller-manager (InferenceService controller), the llmisvc-controller-manager (LLMInferenceService controller), a localmodel controller, a localmodel node agent, a storage initializer (Python), a router for InferenceGraph, and an agent for logging/batching sidecars. The RHOAI distribution uses `distro` build tags to enable OpenShift-specific behavior including OAuth proxy injection, Route creation, and Istio DestinationRule management.

## Architectural Analysis

KServe's architecture reflects its evolution from a traditional ML model serving platform to a comprehensive inference infrastructure supporting both classical models and large language models. The codebase maintains two distinct controller binaries — `kserve-controller-manager` for InferenceService/InferenceGraph/TrainedModel and `llmisvc-controller-manager` for LLMInferenceService — each running as a separate Deployment. This separation is architecturally significant: the LLMInferenceService controller was designed from scratch with Gateway API and disaggregated serving as first-class concepts, while the InferenceService controller retains compatibility with Knative, Istio VirtualService, and Kubernetes Ingress.

The RHOAI distribution uses Go build tags (`distro`) to conditionally compile OpenShift-specific code paths. Files suffixed `_ocp.go` add Istio DestinationRule management, OpenShift service-ca certificate generation, SecurityContextConstraints role bindings, and Kuadrant AuthPolicy precondition checks. This pattern keeps the upstream codebase clean while allowing deep platform integration. The `constants_odh.go` file defines ODH-specific annotation keys (`security.opendatahub.io/enable-auth`, `opendatahub.io/route-enabled`) and the auth proxy configuration, which controls whether `oauth-proxy` or `kube-rbac-proxy` sidecars are injected into inference pods.

The LLMInferenceService controller implements a sophisticated configuration inheritance system through `LLMInferenceServiceConfig` CRs and `baseRefs`. Well-known default configs are automatically injected based on topology (single-node, multi-node, P/D disaggregated), and users can override specific fields. The controller supports three deployment patterns: single-node Deployments, multi-node LeaderWorkerSets, and disaggregated prefill/decode with independent scaling for each phase. Autoscaling flows through the WVA (Workload Variant Autoscaler) which publishes `wva_desired_replicas` to Prometheus; the controller then creates either HPA (reading the metric via Prometheus Adapter) or KEDA ScaledObjects (querying Prometheus directly) as the enforcement layer.

A notable FIPS compliance gap exists in the localmodel components: both `localmodel.Dockerfile.konflux` and `localmodel-agent.Dockerfile.konflux` use `CGO_ENABLED=0` with static linking, which will fail Konflux check-payload validation for FIPS. The main controller and other Go components correctly use `CGO_ENABLED=1` with `GOEXPERIMENT=strictfipsruntime`. Additionally, the LLMInferenceService monitoring stack uses `InsecureSkipVerify=true` in PodMonitor TLS config and Istio DestinationRules — while pragmatic for self-signed certificates, this bypasses certificate validation in the metrics scraping path.

The InferencePool API migration from `v1alpha2` to `v1` is handled through a one-way lock mechanism: the controller creates both versions simultaneously, checks if the Gateway accepts the v1 version, and once confirmed, annotates the resource with `AnnotationInferencePoolMigrated=v1` to prevent fallback. This pattern ensures zero-downtime migration without manual intervention.
