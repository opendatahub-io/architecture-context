## Purpose

**Short**: KServe is a Kubernetes-native platform for serving machine learning models, providing CRDs for InferenceService, LLMInferenceService, InferenceGraph, and LocalModel caching with support for both serverless (Knative) and raw deployment modes.

**Detailed**: KServe manages the complete lifecycle of ML model serving on Kubernetes/OpenShift. It provides multiple controllers that reconcile custom resources into Kubernetes Deployments, Services, ingress resources (HTTPRoutes via Gateway API, Istio VirtualServices, or Kubernetes Ingress), and autoscaling configurations. The primary controller handles `InferenceService` resources (v1beta1), which support predictor, transformer, and explainer components with automatic storage initialization, model downloading, and serving runtime selection.

The `LLMInferenceService` controller (v1alpha2) is a newer, LLM-focused controller that manages disaggregated serving (separate prefill/decode workloads), LoRA adapter management, multi-node inference via LeaderWorkerSet, and integration with llm-d components (inference scheduler, routing sidecar). It creates Gateway API HTTPRoutes, InferencePools, and supports KEDA-based autoscaling with OpenTelemetry metrics collection.

The platform also includes a `kserve-module` controller that integrates with the ODH platform operator pattern, managing KServe's own deployment lifecycle as a platform component. Additional controllers handle InferenceGraphs (multi-model routing DAGs), LocalModel caching (pre-downloading models to node-local PVs), and TrainedModels (A/B testing model variants). Pod mutation webhooks inject storage initializer init containers, inference agent sidecars, batcher containers, and metrics aggregation containers into model serving pods.

Downloads model artifacts from various cloud storage backends (S3, GCS, Azure Blob, HuggingFace Hub, HTTP/HTTPS, OCI) into an emptyDir volume shared with the model server container. Supports Kerberos authentication for HDFS. Also supports confidential model serving with JWE decryption.
**Image**: odh-kserve-storage-initializer

### inference-router

**Intent**: Go service deployed as the router container for InferenceGraph resources. Implements DAG-based routing with Sequence, Splitter (weighted), Ensemble (fan-out/combine), and Switch (conditional) patterns. Each InferenceGraph gets a router Deployment that orchestrates requests across multiple InferenceServices.

## Architectural Analysis

KServe follows a multi-controller architecture where each major feature area (InferenceService, LLMInferenceService, LocalModel, InferenceGraph) has its own dedicated controller binary. This separation allows independent scaling, deployment, and lifecycle management of each controller. The main `kserve-controller-manager` handles the original InferenceService and InferenceGraph CRDs, while the `llmisvc-controller-manager` handles the newer LLMInferenceService CRD. This split was introduced to decouple the LLM-specific logic (disaggregated serving, llm-d integration) from the general model serving controller.

The codebase extensively uses Go build tags (`distro`, `odh`) to compile different behavior for upstream KServe versus the ODH/RHOAI distribution. Files suffixed with `_odh.go` contain OpenShift-specific logic (Route creation, OpenShift service-ca integration, kube-rbac-proxy configuration), while `_default.go` files contain upstream behavior. This pattern appears consistently across controllers (e.g., `controller_setup_odh.go`, `router_platform_networking_odh.go`, `workload_permissions_odh.go`), webhooks, and API types. The `GOTAGS="distro"` build argument in Konflux Dockerfiles activates the ODH code paths.

The LLMInferenceService controller (v1alpha2) represents a significant architectural evolution. It manages a complex deployment topology: the controller can create a single-node Deployment, a multi-node LeaderWorkerSet, or a fully disaggregated setup with separate prefill and decode workloads. It integrates deeply with the llm-d ecosystem, deploying an inference scheduler (endpoint picker), routing sidecar containers, and UDS tokenizer alongside the main vLLM inference server. The controller also manages Gateway API HTTPRoutes and InferencePools for traffic routing, KEDA ScaledObjects for autoscaling, and OpenTelemetryCollectors for metrics collection. The router discovery logic (`router_discovery.go`) resolves Gateway references to determine service hostnames and construct URLs.

The pod mutation webhook is a critical architectural component. It intercepts pod creation for any pod labeled with `serving.kserve.io/inferenceservice` and injects up to four types of sidecar/init containers: (1) storage-initializer init container for downloading model artifacts, (2) inference agent sidecar for request logging and batching, (3) batcher container for request batching, and (4) metrics aggregation container for Prometheus metrics. The injectors examine the InferenceService annotations and configuration to determine what to inject, making the webhook a central point of extensibility.

The kserve-module controller (`kserve-module/`) represents the ODH platform integration layer. It implements the `common.PlatformObject` interface, allowing the ODH operator to manage KServe through a uniform lifecycle pattern. The controller checks for CRD dependencies (Istio networking CRDs, Gateway API CRDs, Knative serving CRDs) before deploying KServe manifests, and uses a kustomize-based resource manager to apply manifests from embedded templates. This is a separate Go module with its own `go.mod` and is built into a distinct container image.

A notable FIPS compliance gap exists: the kserve-module-controller is built with `CGO_ENABLED=0` (static linking) and does not use `strictfipsruntime` flags, which means it will fail the check-payload FIPS validation. All other Go binaries in the repository correctly use `CGO_ENABLED=1` with `GOEXPERIMENT=strictfipsruntime`. The Python storage-initializer uses the `cryptography` library which uses the OpenSSL backend, providing FIPS-compatible crypto on UBI9 images.
