## Purpose

**Short**: KServe is the Kubernetes-native model serving platform that provides CRD-driven lifecycle management for machine learning inference services, including standard model serving, serverless deployments, inference graphs, local model caching, and large language model (LLM) inference orchestration.

**Detailed**: KServe provides a declarative API for deploying, scaling, and managing ML/AI model inference workloads on Kubernetes. It defines several Custom Resource Definitions — InferenceService (v1beta1), InferenceGraph (v1alpha1), TrainedModel (v1alpha1), LocalModelCache (v1alpha1), LocalModelNamespaceCache (v1alpha1), and LLMInferenceService (v1alpha2) — each backed by dedicated controllers that reconcile the desired state into Kubernetes-native resources (Deployments, Services, HPAs, VirtualServices, HTTPRoutes, etc.).

The platform supports multiple deployment modes: Standard (raw Kubernetes Deployments), Knative (serverless autoscaling to zero), and ModelMesh (multi-model serving). For LLM workloads, the LLMInferenceService controller provides specialized capabilities including single-node and multi-node (LeaderWorkerSet) deployments, disaggregated prefill/decode architecture, Gateway API integration with InferencePools, and Workload Variant Autoscaling (WVA). The local model caching subsystem enables node-level model pre-caching via PersistentVolumes and download Jobs, reducing cold-start latency for large models.

In the RHOAI distribution, KServe is deployed by the rhods-operator/opendatahub-operator into the `opendatahub` namespace. The `distro` build tag activates OpenShift-specific features: Istio DestinationRule management, OpenShift Route creation for InferenceGraphs, service-ca TLS certificate injection, SecurityContextConstraints for LLM workloads, and kube-rbac-proxy/oauth-proxy sidecar injection for authentication. Images are parameterized via `params.env` and replaced at deployment time through kustomize replacements.

## Architectural Analysis

KServe's architecture reflects its evolution from a single-CRD model serving platform to a comprehensive ML/AI inference platform spanning three major workload categories: traditional model serving (InferenceService), LLM-specific workloads (LLMInferenceService), and local model caching (LocalModelCache). Each category has its own controller binary, CRDs, and RBAC, but they share the `serving.kserve.io` API group and the Go module's `pkg/` package tree.

**Multi-Controller Architecture**: The repository produces four distinct controller binaries (`cmd/manager`, `cmd/llmisvc`, `cmd/localmodel`, `cmd/localmodelnode`), each deployed as a separate Deployment or DaemonSet. This separation allows independent scaling and lifecycle management. The main kserve-controller handles InferenceService/InferenceGraph/TrainedModel, while the llmisvc-controller is purpose-built for the LLMInferenceService's more complex reconciliation (multi-node, disaggregated P/D, InferencePool, WVA scaling). The localmodel subsystem is split between a cluster-level controller (PV/PVC management) and a per-node agent (download Jobs, filesystem management). This design avoids a monolithic operator but introduces inter-controller coordination via shared CRDs (e.g., `LocalModelNode` is written by `localmodel-controller` and reconciled by `localmodelnode-agent`).

**Distribution Abstraction via Build Tags**: The `//go:build distro` mechanism cleanly separates OpenShift/RHOAI-specific behavior (Istio DestinationRules, OpenShift Routes, service-ca certificates, SecurityContextConstraints, SELinux MCS levels) from community KServe code. Files like `manager_options_ocp.go`, `controller_setup_ocp.go`, `controller_rbac_ocp.go`, `platform_ocp.go`, and `workload_permissions_ocp.go` provide the OpenShift layer. This avoids `#ifdef`-style conditionals in the main reconciliation logic while keeping all distribution code in the same repository. The Konflux Dockerfiles for controller and llmisvc-controller pass `GOTAGS="distro"` to activate this layer.

**Ingress Flexibility**: The InferenceService controller uses a factory pattern (`reconcilers/factory.go`) to select between three ingress backends: Istio VirtualService, Gateway API HTTPRoute, or Kubernetes Ingress. The LLMInferenceService controller exclusively uses Gateway API with InferencePools for load balancing, representing the modern architectural direction. Both controllers discover gateway readiness dynamically and handle the v1alpha2→v1 InferencePool API migration gracefully via a one-way annotation flag (`serving.kserve.io/inference-pool-migrated`). This migration-awareness pattern is noteworthy — it allows rolling upgrades without service disruption.

**FIPS Gap in LocalModel Components**: While the main controller, agent, and router binaries use `CGO_ENABLED=1` with `GOEXPERIMENT=strictfipsruntime` (FIPS-compliant dynamic linking), the `localmodel-controller` and `localmodelnode-agent` Dockerfiles use `CGO_ENABLED=0` (static linking). This means these binaries use Go's standard crypto library rather than OpenSSL, and they may fail the `check-payload` validation gate on FIPS-required clusters. This appears to be an oversight in the downstream Konflux Dockerfile configuration, as the upstream community Dockerfiles also use static linking — the downstream fork should have patched these to match the controller/agent/router pattern.

**LLMInferenceService Complexity**: The llmisvc controller is the most complex controller in the repository, managing a full stack: Deployments or LeaderWorkerSets (workload), Services (networking), HTTPRoutes + InferencePools (routing), Scheduler Deployments with RBAC (load balancing), VariantAutoscaling + HPA/KEDA (scaling), PodMonitors + ServiceMonitors (monitoring), and self-signed TLS certificates (security). The configuration composition system (`config_merge.go`) uses a WellKnownConfigResolver that pins config names to Status.Annotations for stability across upgrades — a pattern that prevents config drift when templates change but existing deployments should remain on their original config version.
