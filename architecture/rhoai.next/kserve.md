## Purpose

**Short**: KServe is a Kubernetes-native model serving platform that provides CRDs and controllers for deploying, scaling, and managing ML inference services with support for standard model serving, LLM inference, inference graphs, and local model caching.

**Detailed**: KServe is the core model serving infrastructure component for Red Hat OpenShift AI. It defines and manages the complete lifecycle of inference workloads through a set of Custom Resource Definitions (InferenceService, LLMInferenceService, InferenceGraph, TrainedModel, ServingRuntime, ClusterServingRuntime) and their associated controllers. The platform supports multiple deployment modes including raw Kubernetes deployments, Knative serverless, and ModelMesh distributed serving.

The repository produces eight container images via Konflux, organized across four controller binaries (kserve-controller-manager, llmisvc-controller, localmodel-controller, localmodelnode-agent), a sidecar agent, an inference router, a storage initializer, and a kserve-module controller that manages KServe's own deployment within the ODH/RHOAI platform operator. The LLMInferenceService controller (v1alpha2) is a newer addition specifically designed for large language model workloads, supporting features like LoRA adapter management, disaggregated prefill/decode, multi-node inference via LeaderWorkerSet, Dynamic Resource Allocation (DRA), Gateway API routing, and integration with the llm-d inference scheduler (Endpoint Picker Protocol).

KServe integrates deeply with the RHOAI platform through build-tagged distro files (`_odh.go` suffixes) that add OpenShift-specific functionality including OpenShift Route support, OAuth/kube-rbac-proxy authentication sidecars, OpenShift serving certificate integration, SELinux/MCS-aware local model storage, and Istio DestinationRule management for service mesh TLS origination.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| kserve-controller-manager | Go Operator | Main controller managing InferenceService, TrainedModel, and InferenceGraph CRDs with support for Knative, raw Kubernetes, and ModelMesh deployment modes |
| llmisvc-controller | Go Operator | Dedicated controller for LLMInferenceService CRD with LLM-specific features (LoRA, multi-node, DRA, inference scheduling, disaggregated serving) |
| localmodel-controller | Go Operator | Manages LocalModelCache and LocalModelNamespaceCache CRDs for pre-caching models on cluster nodes via PV/PVC and download jobs |
| localmodelnode-agent | Go DaemonSet Agent | Per-node agent that reconciles LocalModelNode CRs, manages model download jobs, and handles filesystem permissions |
| kserve-agent | Go Sidecar | Sidecar container injected into inference pods for model pulling, request logging, batching, and metrics aggregation |

| kserve-storage-initializer | Python Init Container | Init container that downloads model artifacts from various storage backends (S3, GCS, Azure, HuggingFace, OCI) |
| kserve-module-controller | Go Operator | Component controller for the ODH/RHOAI platform operator that manages the deployment of KServe, odh-model-controller, and WVA via kustomize manifests |

### localmodelnode-agent

**Dockerfile**: `Dockerfiles/localmodel-agent.Dockerfile.konflux`
**Intent**: Utility image — runs as a DaemonSet agent on each node. Reconciles LocalModelNode CRs, creates batch/v1 Jobs to download models to node-local storage, manages filesystem permissions (SELinux/MCS on ODH), and reports download status back to LocalModelNode status. **Image**: odh-kserve-localmodelnode-agent
**Language**: Go
**Port**: 8080/TCP (metrics), 8081/TCP (health)

## Architectural Analysis

KServe is a multi-controller architecture with clear separation of concerns across four controller binaries. The main kserve-controller-manager handles the foundational InferenceService, TrainedModel, and InferenceGraph CRDs, while the llmisvc-controller is a dedicated, separately-deployed controller for the newer LLMInferenceService CRD. This separation allows the LLM-specific controller to evolve independently with its own deployment lifecycle, TLS-secured metrics endpoint, and storage version migration logic. The localmodel-controller and localmodelnode-agent pair implements a split-brain architecture where the controller manages cluster-level resources (PV/PVC) and the DaemonSet agent handles per-node operations (download jobs, filesystem permissions).

The repository employs a sophisticated build-tag system for distribution-specific behavior. Files suffixed with `_odh.go` are compiled with `-tags distro` (set in Konflux Dockerfiles) and contain OpenShift-specific logic: OpenShift Route reconciliation, OAuth/kube-rbac-proxy sidecar injection, OpenShift serving certificate integration, Istio DestinationRule management, and SELinux/MCS-aware filesystem operations. The default (non-distro) files provide upstream-compatible behavior. This pattern avoids runtime feature flags and ensures dead code elimination for each distribution.

The LLMInferenceService controller represents the most architecturally complex part of the codebase. It manages a full inference stack per CR: Deployment or LeaderWorkerSet for the model workload, a separate Deployment for the EPP (Endpoint Picker Protocol) scheduler, Gateway API HTTPRoutes for ingress, InferencePool CRs for load balancing, DRA ResourceClaimTemplates for GPU allocation, and optional Istio DestinationRules for service mesh TLS origination (ODH). The controller supports three autoscaling backends (HPA, KEDA, WVA VariantAutoscaling) and creates monitoring resources (PodMonitor, ServiceMonitor) for Prometheus scraping. The v1alpha1→v1alpha2 CRD conversion is handled via conversion webhooks with a background storage version migration process.

The kserve-module-controller is an interesting meta-controller pattern — it deploys KServe itself. It reads kustomize manifests bundled into the container image (at `/opt/manifests-template/`), renders them with image parameters from `RELATED_IMAGE_*` environment variables (injected by the platform operator), applies post-render customizations (ConfigMap tuning, LLMInferenceServiceConfig versioning), and deploys the rendered resources. It manages three sub-components (KServe, odh-model-controller, WVA) with independent enable/disable states and validates 16+ platform dependencies (Istio CRDs, cert-manager CRDs, OLM subscriptions, operator health). Notably, this controller uses `CGO_ENABLED=0` in its Dockerfile, making it the only Go binary in the repo without FIPS-compliant crypto — a gap that should be addressed.

The webhook system is comprehensive: a pod mutating webhook intercepts pod creation and injects storage-initializer init containers, agent sidecars (for model pulling/logging/batching), metrics aggregation configuration, and GKE accelerator node selectors. Validating webhooks enforce business rules across all CRDs — model format priority consistency across ServingRuntimes, multi-node configuration validity, LocalModelCache storage URI uniqueness, and deletion protection when models are in use. The webhook reinvocation policy (`IfNeeded`) handles the complex multi-step injection chain where earlier injections may modify the pod spec in ways that require re-evaluation.
