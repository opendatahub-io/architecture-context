## Purpose

**Short**: KServe is a Kubernetes-native model serving platform that provides CRDs and controllers for deploying, scaling, and managing ML/LLM inference workloads with multi-protocol support (REST V1/V2, gRPC, OpenAI-compatible).

**Detailed**: KServe defines the core model serving abstractions for Red Hat OpenShift AI. It provides Custom Resource Definitions for InferenceService (traditional ML model serving), LLMInferenceService (LLM-optimized serving with disaggregated prefill/decode, Gateway API integration, and advanced autoscaling), InferenceGraph (multi-model orchestration pipelines), and LocalModel (node-local model caching). The project ships multiple Go controllers (kserve-controller, llmisvc-controller, localmodel-controller, localmodelnode-agent) that reconcile these CRDs into Kubernetes Deployments, Services, HTTPRoutes, and autoscaling resources. A Python SDK provides the model serving framework with V1/V2/OpenAI REST and gRPC inference protocols, as well as storage-initializer for downloading model artifacts from S3, GCS, Azure, HuggingFace Hub, and other backends.

The component integrates deeply with the RHOAI platform through OpenShift-specific features: OpenShift serving certificates (replacing cert-manager), OpenShift Routes for InferenceGraph, SecurityContextConstraints for multi-node workloads, Gateway API with Istio DestinationRules for LLM traffic, and kube-rbac-proxy for authentication on InferenceService endpoints. KServe supports Knative Serving for serverless deployment, raw Kubernetes Deployments for always-on serving, and Kubernetes Gateway API for LLM workloads.

## Architectural Analysis

KServe's architecture follows a multi-controller pattern where four separate controller binaries manage different CRD families: the main kserve-controller handles traditional InferenceService and InferenceGraph resources, the llmisvc-controller handles LLM-specific workloads, the localmodel-controller manages cluster-wide model caching, and the localmodelnode-agent runs as a DaemonSet on worker nodes. This separation allows independent scaling and deployment — the LLMInferenceService controller, for example, uses secure metrics (8443/TCP HTTPS) and has a more complex cache configuration with label-based filtering of child resources.

The LLMInferenceService controller represents a significant evolution in architecture. Unlike the traditional InferenceService which supports multiple deployment modes (Knative, raw Kubernetes, ModelMesh), LLMInferenceService is purpose-built for large language models with features like disaggregated prefill/decode using separate Deployments or LeaderWorkerSets, Gateway API integration via HTTPRoute and InferencePool CRDs, and a sophisticated autoscaling stack combining VariantAutoscaling (cost-aware), HPA, and KEDA as actuators. The controller creates Istio DestinationRules on OpenShift for mTLS between components and manages a complete scheduler stack including Endpoint Picker (EPP) Deployments with ServiceAccounts, Roles, and ClusterRoleBindings. The configuration system uses template-based LLMInferenceServiceConfig CRs with strategic merge patch semantics, allowing platform administrators to define default configurations for different accelerator types (NVIDIA CUDA, AMD ROCm, Intel Gaudi, IBM Spyre) and deployment patterns (single-node, multi-node, disaggregated P/D).

The webhook mutation system is a critical architectural component. The pod mutator webhook intercepts all pods in KServe-managed namespaces and injects storage-initializer init containers, agent sidecars, batcher sidecars, and metrics aggregation configuration based on InferenceService annotations and the inferenceservice-config ConfigMap. The storage-initializer injection is particularly complex — it detects the storage URI protocol, selects the appropriate ClusterStorageContainer, injects credentials from Kubernetes Secrets, mounts PVCs, and configures CA bundles. For RHOAI deployments, the deployment reconciler additionally injects kube-rbac-proxy containers with SubjectAccessReview ConfigMaps for authentication.

The kustomize deployment structure uses a component-based composition where `config/components/` defines reusable KServe, LLMInferenceService, and LocalModel components that are composed into overlays. The ODH overlay (`config/overlays/odh/`) is the primary RHOAI deployment path and makes extensive use of `replacements:` to inject image references from a `params.env`-generated ConfigMap into Deployments, LLMInferenceServiceConfigs, and the inferenceservice-config ConfigMap. This approach allows the platform operator to control all container images from a single ConfigMap, enabling coordinated image updates across the entire KServe deployment. The overlay also replaces cert-manager with OpenShift serving certificates, removes ClusterServingRuntimes (managed externally), and adds NetworkPolicies for pod-to-pod traffic isolation.

The Python SDK provides a dual-protocol model serving framework (REST via FastAPI/uvicorn on 8080/TCP and gRPC on 8081/TCP) with support for V1, V2, and OpenAI-compatible inference protocols. The model lifecycle is designed for the Kubernetes sidecar pattern — the storage-initializer downloads models before the server starts, and the agent sidecar handles dynamic model loading/unloading via the V2 repository API. The SDK supports a transformer pattern where a preprocessing service calls a separate predictor service, with configurable protocol selection (v1, v2, grpc-v2) and retry policies. The storage library supports 11 storage backends with configurable parallelism and pattern-based file filtering.
