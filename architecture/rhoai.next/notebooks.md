## Purpose

**Short**: Provides pre-built container images for interactive data science workbenches (Jupyter, RStudio, code-server) and pipeline runtime environments used in OpenDataHub and Red Hat OpenShift AI.

**Detailed**: The notebooks repository is a container image factory that produces two categories of images: **workbenches** and **pipeline runtimes**. Workbenches are interactive development environments (JupyterLab, RStudio Server, VS Code via code-server) pre-configured with data science libraries and tools. Pipeline runtimes are minimal execution environments used by Elyra pipelines to run notebook nodes as Kubernetes Jobs.

Each image variant is built for a specific accelerator target (CPU, NVIDIA CUDA, AMD ROCm) and comes with a curated set of Python packages for different use cases (minimal, data science, PyTorch, TensorFlow, TrustyAI, LLM compression). The repository uses a layered Dockerfile architecture where specialized images inherit from base images, adding libraries incrementally. All RHOAI images are built hermetically via Konflux using cachi2/Hermeto prefetching — no network access during build.

The images are deployed to OpenShift clusters as ImageStream resources managed by kustomize manifests. The platform operator (rhods-operator for RHOAI, opendatahub-operator for ODH) applies these manifests to create ImageStreams that the ODH Dashboard presents to users for workbench creation. The ODH Notebook Controller then creates StatefulSets from the selected image.

## Architectural Analysis

The notebooks repository is architecturally unusual within the RHOAI platform: it produces no running services or operators, only container images. Its "deployment" is the creation of ImageStream resources that catalog these images for the platform's workbench lifecycle. This makes the repository primarily a build system with sophisticated dependency management rather than a traditional Kubernetes application.

**Layered Image Hierarchy**: The Dockerfile architecture follows a strict inheritance chain: `base-image (CPU/CUDA/ROCm)` -> `minimal` -> `datascience` -> `{pytorch,tensorflow,trustyai,llmcompressor}`. Each layer adds Python packages incrementally. This design minimizes redundancy (shared base layers reduce registry storage and pull times) but creates a deep dependency chain — a CVE in the minimal layer requires rebuilding all downstream images. The CUDA and ROCm variants fork at the base-image level, creating three parallel tracks (CPU, CUDA, ROCm) that share the same Python package layering pattern.

**Hermetic Build System**: The repository implements full build hermeticity via Konflux/cachi2. All dependencies — RPMs, Python wheels, npm packages, Go modules, and even GPG keys — are prefetched and stored in lock files (`rpms.lock.yaml`, `pylock.*.toml`, `requirements.*.txt`). Dockerfiles reference `/cachi2/output/deps/` mounts with `--no-index` flags, ensuring zero network access during builds. This is critical for FIPS compliance and supply chain security. The `prefetch-input/` directory structure supports both ODH (upstream) and RHDS (downstream) RPM configurations, with separate lock files for each. A noteworthy supply-chain hardening measure is the `exclude-newer = "3 days"` setting in `pyproject.toml`, which refuses to resolve Python packages published less than 3 days ago — a defense against supply chain attacks that inject malicious versions.

**Multi-Architecture Support**: Images are built for 4 architectures (x86_64, aarch64, ppc64le, s390x) where vendor support allows. The `ppc64le` and `s390x` builds are notably complex — the `whl-cache` stage in the codeserver Dockerfile compiles Python C extensions from source distribution (sdist) because pre-built wheels aren't available for these architectures. RHOAI provides pre-compiled wheels for numpy, scipy, and other native packages on these platforms via the `requirements-rhoai.txt` index.

**Code-Server Complexity**: The code-server workbench is significantly more complex than the Jupyter images. It builds VS Code's code-server from source in a multi-stage Dockerfile with 5 stages: `rpm-base` (builds code-server from prefetched npm/node sources), `whl-cache` (compiles Python wheels for non-x86 architectures), `cpu-base` (installs runtime OS packages), `codeserver` (assembles final image with nginx proxy), and `tests` (smoke tests). The hermetic npm build requires URL rewriting (`rewrite-npm-urls.sh`) to redirect package-lock.json resolved URLs from `registry.npmjs.org` to local cachi2 paths.

**ImageStream Versioning Strategy**: The RHOAI manifests maintain 3 concurrent image versions per workbench type: N (current, tag `3.4`), N-1 (previous release, tag `2025.2`), and N-2 (oldest supported, tag `2025.1`, marked as outdated). This enables rolling upgrades — users on N-2 see a deprecation warning in the Dashboard, while N is marked as `opendatahub.io/default-image: "true"`. The kustomize `replacements` mechanism injects actual image digests from `params.env` into ImageStream tags, ensuring immutable image references via `@sha256:` digests in production.
