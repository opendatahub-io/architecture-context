## Purpose

**Short**: Builds and publishes container images for interactive data science workbenches (JupyterLab, Code-Server, RStudio) and Elyra pipeline runtime images used in Red Hat OpenShift AI.

**Detailed**: The Notebooks repository is a container image factory that produces approximately 20 distinct container images spanning four workbench types (JupyterLab, Code-Server, RStudio, pipeline runtimes) across three accelerator targets (CPU, NVIDIA CUDA, AMD ROCm). These images form the user-facing compute environment in RHOAI — when a data scientist creates a "workbench" in the ODH Dashboard, the platform pulls one of these images and runs it as a pod on OpenShift.

The repository does not contain any operator or controller code. Instead, it defines Dockerfiles (both standard and hermetic Konflux variants), Python dependency lockfiles, Kubernetes ImageStream manifests, and a Makefile-based build system. The images are consumed by the platform operator (`rhods-operator` / `opendatahub-operator`) via kustomize manifests that define ImageStream resources. The ODH Dashboard discovers available workbench images through ImageStream annotations, and the ODH Notebook Controller (in the `kubeflow` repo) handles pod lifecycle, sidecar injection, and idle culling.

Each workbench image bundles a complete data science environment: Python with locked dependencies (via `uv` and pylock), ML frameworks (PyTorch, TensorFlow, LLMCompressor), database connectors, the `oc` CLI, and IDE-specific tooling. Code-Server and RStudio images additionally include an nginx reverse proxy and CGI scripts that emulate the Jupyter `/api/kernels/` endpoint for idle culling compatibility.

## Architectural Analysis

### Image Hierarchy and Build Model

The repository implements a layered image hierarchy where each Jupyter flavor extends the previous one. The hierarchy is: `base-image (external) → jupyter/minimal → jupyter/datascience → {pytorch, tensorflow, trustyai, pytorch+llmcompressor}`. This is a conceptual hierarchy — each Dockerfile uses `FROM ${BASE_IMAGE}` and builds independently via multi-stage Dockerfiles. In ODH mode, base images are built from `base-images/` in this repo; in RHOAI mode, base images come from the AIPCC pipeline (e.g., `quay.io/aipcc/base-images/cuda-13.0-el9.6:3.5.0-ea.1-*`). The Makefile selects between standard Dockerfiles (`Dockerfile.cpu`) and Konflux hermetic Dockerfiles (`Dockerfile.konflux.cpu`) via the `KONFLUX` flag.

### Hermetic Build Architecture

RHOAI images are built entirely by Konflux using hermetic Dockerfiles that run without network access. All dependencies — RPMs, Python wheels, npm packages, Go modules — are prefetched by cachi2 and mounted at `/cachi2/output/deps/` during the build. Python dependencies use `uv pip install --no-index --find-links /cachi2/output/deps/pip` with hash verification via pylock TOML lockfiles. The Go `buildinputs` tool (`scripts/buildinputs/`) converts Dockerfiles to BuildKit's LLB intermediate representation to extract exactly which files from the build context are needed, enabling minimal build contexts for Konflux. This supply-chain security posture includes a 3-day `exclude-newer` window in `pyproject.toml` to prevent newly-published malicious packages from being picked up.

### Idle Culling Compatibility Layer

JupyterLab natively exposes `/api/kernels/` for idle detection. Code-Server and RStudio do not. This repository works around this by implementing a three-process stack inside each Code-Server/RStudio container: (1) nginx on port 80 serves as a reverse proxy with custom JSON access logging that records `last_activity` timestamps, (2) Apache httpd on port 8080 acts as a CGI gateway, and (3) a bash CGI script (`access.cgi`) implements the `/api/kernels/` endpoint. For Code-Server, the CGI polls `localhost:8787/codeserver/healthz` and converts the heartbeat to Jupyter kernel format. For RStudio, it parses nginx access logs and marks the workbench as idle after 10 minutes of inactivity. This architecture is acknowledged as fragile in the ARCHITECTURE.md and is planned for replacement with a single Go reverse proxy.

### Manifest Parameterization

The kustomize manifests use a sophisticated parameterization scheme: 114 replacement rules inject image references from `params.env` / `params-latest.env` ConfigMaps into ImageStream `spec.tags[N].from.name` fields, plus commit hash traceability into annotations. All production image references use sha256 digest pinning (`@sha256:...`) for immutable deployments. Each ImageStream maintains 3 tags representing version history: current (3.4), previous (2025.2), and older (2025.1, marked as outdated). The manifest tools in `manifests/tools/` (5 Python scripts) semi-automate generation of these manifests from registry data and pylock files.

### Multi-Architecture Support

The images support up to 4 architectures: x86_64, aarch64, ppc64le, and s390x. For ppc64le and s390x, where PyPI doesn't provide pre-built wheels for many packages (numpy, scipy, pyarrow), the Dockerfiles include a `whl-cache` build stage that compiles C-extension packages from source using RHOAI-provided wheels and exports them for the final stage. The `openblas-threads` RPM provides the `libopenblasp.so.0` runtime library that numpy/scipy link against. ROCm builds additionally run `de-vendor-torch.py` to remove vendored CUDA libraries from PyTorch wheels and `link-solibs.py` for ROCm library compatibility.
