## Purpose

**Short**: Builds container images for interactive data science workbenches (Jupyter notebooks, Code-Server) and pipeline runtime environments for OpenDataHub and Red Hat OpenShift AI.

**Detailed**: The notebooks repository is a container image factory that produces the workbench images users interact with when doing data science on the OpenDataHub (ODH) and Red Hat OpenShift AI (RHOAI) platforms. It does not deploy services or run controllers; instead, it defines Dockerfiles, Python dependency specifications, and kustomize manifests that produce container images consumed by the platform operator and notebook controller.

The repository produces three categories of images: (1) **Jupyter notebook workbenches** in multiple flavors (minimal, datascience, PyTorch, TensorFlow, TrustyAI) with CPU, CUDA, and ROCm GPU variants; (2) a **Code-Server workbench** (VS Code in the browser) with an nginx/httpd proxy stack for Jupyter-compatible idle culling; and (3) **pipeline runtime images** used by the Elyra JupyterLab extension to execute notebook pipeline nodes on Kubeflow Pipelines. Additionally, it maintains **GPU base images** (CUDA and ROCm) built on CentOS Stream 9 that serve as foundations for the GPU-accelerated workbench variants.

Images are built hermetically using Konflux (cachi2 prefetch) for RHOAI, and with standard Dockerfiles for ODH. The repository publishes OpenShift ImageStream manifests that the platform operator (rhods-operator/opendatahub-operator) deploys to clusters, making workbench images discoverable by the ODH Dashboard for users to launch.

## Architectural Analysis

The notebooks repository is architecturally unusual within the ODH/RHOAI ecosystem because it is not a service or operator but a **container image factory**. Its primary output is not running software but a set of container images and OpenShift ImageStream manifests that other platform components consume. This design makes it a critical supply chain component: every data scientist's workbench environment flows through this repo's Dockerfiles and dependency locks.

**Image Hierarchy and Build Graph**: The images form a strict inheritance chain (minimal -> datascience -> pytorch/tensorflow/trustyai), which means a change to the minimal base image cascades through all downstream flavors. The Makefile encodes this dependency graph and builds images in topological order. Each image directory (`jupyter/*/ubi9-python-3.12/`) is self-contained with its own `Dockerfile.*`, `pyproject.toml`, `uv.lock.d/pylock.*.toml`, and `build-args/*.conf`. This isolation means each image can be independently versioned and locked, but it also means dependency updates must propagate through the chain. The `refresh-lock-files` Makefile target automates this propagation.

**Dual Build Mode**: The repository supports two build paths: standard Dockerfiles (`Dockerfile.cpu`, `Dockerfile.cuda`, `Dockerfile.rocm`) for ODH upstream builds, and Konflux Dockerfiles (`Dockerfile.konflux.*`) for RHOAI downstream builds. The Konflux path is fully hermetic, using cachi2 to prefetch all dependencies (RPMs, Python wheels, npm packages, generic tarballs) before the build runs in a network-isolated environment. The `KONFLUX` Makefile variable switches between modes, and tests are aware of this distinction (manifests come from `manifests/odh/` or `manifests/rhoai/` depending on mode).

**Code-Server's Idle Culling Architecture**: The most architecturally complex component is the Code-Server workbench. Unlike Jupyter, which natively exposes a `/api/kernels/` endpoint for activity tracking, Code-Server has no equivalent. The repository solves this by deploying a three-process stack inside each Code-Server pod: nginx (port 80) as a reverse proxy with JSON access logging that records activity timestamps, httpd (port 8080) as a CGI gateway, and a bash CGI script (`access.cgi`) that polls code-server's `/codeserver/healthz` endpoint and translates the heartbeat into Jupyter-compatible kernel format. This is acknowledged as fragile in the ARCHITECTURE.md and is planned for replacement with a single Go reverse proxy.

**Supply Chain Hardening**: The repository demonstrates unusually thorough supply chain security for a container image repo. RPM locks (`rpms.lock.yaml`), Python locks (`pylock.*.toml`), Go locks (`go.sum`), npm locks (`package-lock.json`), and artifact locks (`artifacts.lock.yaml`) cover every dependency layer. The cachi2 integration rewrites all package URLs to point at prefetched local caches. The `uv` package manager is configured with `exclude-newer = "3 days"` to reject packages published within the last 3 days (a supply chain attack mitigation). FIPS compliance is enforced at the Go binary level (mongocli is built with `strictfipsruntime`) and at the image level (compat-openssl11 is installed for check-payload scanning).

**Multi-Architecture Complexity**: All images target four architectures (x86_64, aarch64, ppc64le, s390x). The ppc64le and s390x architectures present unique challenges: many Python packages don't publish pre-built wheels for these platforms, so the Dockerfiles include special handling to compile wheels from source (the `whl-cache` build stage in Code-Server) or use RHOAI-provided wheels from an internal wheel repository. The ROCm images include additional workarounds like `de-vendor-torch.py` (removes vendored ROCm libraries that conflict with system-installed ones) and `link-solibs.py` (creates version-specific symlinks for shared libraries).
