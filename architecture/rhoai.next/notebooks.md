## Purpose

**Short**: Builds container images for interactive data science workbenches (JupyterLab, Code-Server) and pipeline runtime images used by the RHOAI platform.

**Detailed**: This repository produces the complete set of container images that power user-facing data science workbenches in Red Hat OpenShift AI. The images span three categories: Jupyter notebook workbenches (minimal, datascience, PyTorch, TensorFlow, TrustyAI, and PyTorch+LLMCompressor flavors across CPU, CUDA, and ROCm accelerators), Code-Server (VS Code in the browser) workbenches, and Elyra pipeline runtime images. Each image is built as a self-contained multi-stage Dockerfile starting from an external base image — there are no inter-image build dependencies.

The repository does not deploy any services or operators itself. Instead, it produces container images and OpenShift ImageStream manifests. The platform operator (rhods-operator / opendatahub-operator) consumes these ImageStreams via kustomize and deploys them to the cluster. The ODH Dashboard then discovers the workbench images through ImageStream annotations and presents them to users. When a user launches a workbench, the notebook controller (kubeflow/odh-notebook-controller) handles pod lifecycle, OAuth proxy injection, CA bundle mounting, and Elyra runtime configuration.

For RHOAI builds, all images use AIPCC (AI Platform Core Components) Ecosystems base images from `quay.io/aipcc/base-images/`, which provide RHEL 9.6 with pre-configured Python 3.12 and accelerator-specific runtime libraries. The build system is fully hermetic — all dependencies (RPMs, Python wheels, Go modules, npm packages) are prefetched by Hermeto/Cachi2 and mounted at build time with no network access.

## Sub-Component Details

### Jupyter Workbenches (Minimal / DataScience / PyTorch / TensorFlow / TrustyAI / PyTorch+LLMCompressor)

**Dockerfiles**: `jupyter/*/ubi9-python-3.12/Dockerfile.konflux.*`
**Intent**: Primary interactive data science workbenches. Each flavor extends the previous layer conceptually: minimal → datascience → framework-specific (pytorch, tensorflow, trustyai). All images use a self-contained multi-stage Dockerfile pattern — each Dockerfile re-creates all ancestor stages internally rather than referencing another image.
**Language**: Python (JupyterLab + libraries)
**Port**: 8888/TCP (JupyterLab server, configurable via `NOTEBOOK_PORT`)

#### Startup Flow

| Step | Action | Source |
|------|--------|--------|
| 1 | `start-notebook.sh` executes as ENTRYPOINT | `jupyter/minimal/ubi9-python-3.12/start-notebook.sh` |
| 2 | Sources `utils/process.sh` (process management helpers) | `jupyter/utils/process.sh` |
| 3 | Sources `utils/setup-elyra.sh` if present (datascience+ images) | `jupyter/datascience/ubi9-python-3.12/setup-elyra.sh` |
| 4 | Sources `utils/setup-kale.sh` if present (datascience+ images) | `jupyter/datascience/ubi9-python-3.12/setup-kale.sh` |
| 5 | Builds NOTEBOOK_PROGRAM_ARGS from env vars (NOTEBOOK_PORT, NOTEBOOK_BASE_URL, NOTEBOOK_ROOT_DIR, NOTEBOOK_ARGS) | `start-notebook.sh:17-44` |
| 6 | Starts `jupyter lab` with constructed arguments | `start-notebook.sh:47-50` |

#### Configuration

|------|------|---------|---------|

### Jupyter ROCm Variants (Minimal / PyTorch / TensorFlow)

**Dockerfiles**: `jupyter/minimal/ubi9-python-3.12/Dockerfile.konflux.rocm`, `jupyter/rocm/pytorch/ubi9-python-3.12/Dockerfile.konflux.rocm`, `jupyter/rocm/tensorflow/ubi9-python-3.12/Dockerfile.konflux.rocm`
**Intent**: GPU-accelerated workbenches for AMD ROCm hardware. ROCm variants include a `de-vendor-torch.py` step (PyTorch) that removes vendored ROCm libs from the PyTorch wheel to use system ROCm instead, and `link-solibs.py` to create unversioned `.so` symlinks for ROCm libraries.
**Language**: Python
**Port**: 8888/TCP

### Code-Server Workbench

**Dockerfile**: `codeserver/ubi9-python-3.12/Dockerfile.konflux.cpu`
**Intent**: VS Code in the browser for users who prefer an IDE over Jupyter. Built from source (code-server v4.106.3) with a hermetic npm build. Includes a multi-process idle culling stack (nginx + httpd CGI + bash scripts) to satisfy the notebook controller's Jupyter-compatible kernel API expectations.
**Image**: `odh-workbench-codeserver-datascience-cpu-py312-rhel9`
**Language**: TypeScript/JavaScript (code-server), Python (workbench environment), Bash (CGI scripts)
**Port**: 8787/TCP (code-server), 80/TCP (nginx proxy), 8080/TCP (httpd CGI)

#### Multi-Process Architecture

| Process | Port | Purpose |
|---------|------|---------|
| code-server | 8787/TCP | VS Code IDE server |
| nginx | 80/TCP | Reverse proxy with JSON access logging for idle culling |
| httpd | 8080/TCP | Apache CGI gateway for kernel API emulation |
| access.cgi (bash) | via httpd | Implements `/api/kernels/` endpoint by polling code-server healthz |

#### Configuration

| Name | Type | Default | Purpose |
|------|------|---------|---------|
| Code-Server bind address | computed | `[::]:8787` (IPv6) or `0.0.0.0:8787` (IPv4) | Determined by IPv6 availability at startup |
| `--auth none` | flag | none | Authentication handled by kube-rbac-proxy sidecar, not code-server |
| `--disable-telemetry` | flag | true | Telemetry disabled |
| VS Code settings | JSON | Python interpreter, telemetry off, trust disabled | Written to `$CODE_SERVER_DATA_DIR/User/settings.json` at startup |

### Pipeline Runtime Images

**Dockerfiles**: `runtimes/*/ubi9-python-3.12/Dockerfile.konflux.*`
**Intent**: Execution environments for Elyra pipeline nodes. When a user defines a pipeline in JupyterLab's visual editor (Elyra), each node executes inside a runtime container. Runtime images are lighter than workbenches — no JupyterLab, no IDE — just Python with the necessary ML libraries. Includes `bootstrapper.py` from Elyra for pipeline orchestration.
**Language**: Python
**Port**: None (batch execution, not a server)

#### Configuration

| Name | Type | Default | Purpose |
|------|------|---------|---------|
| `ELYRA_INSTALL_PACKAGES` | env | false | Controls whether Elyra bootstrapper installs packages at pipeline node startup |

### mongocli (Build-Stage Binary)

**Dockerfile stage**: `mongocli-builder` in datascience+ Konflux Dockerfiles
**Intent**: MongoDB CLI tool compiled from a Git submodule (`prefetch-input/mongocli`) using Go with FIPS-compliant build flags. Included in all datascience-tier and above workbench images. Skipped on s390x (dummy binary created instead).
**Language**: Go
**Build flags**: `CGO_ENABLED=1 GOOS=linux -tags strictfipsruntime`

## Architectural Analysis

The notebooks repository is architecturally unusual compared to most RHOAI components: it produces no running services or controllers, only container images. Its complexity lies in the combinatorial explosion of image flavors (7 Jupyter flavors × 3 accelerator targets + Code-Server + 7 runtime images = 18+ distinct images) and the hermetic build infrastructure required to produce them at product quality.

**Self-contained multi-stage Dockerfile pattern**: Each leaf Dockerfile (e.g., PyTorch CUDA) rebuilds every ancestor stage (base → minimal → datascience → pytorch) within a single file. This eliminates inter-image build dependencies but creates significant duplication — the `cuda-base`, `cuda-jupyter-minimal`, and `cuda-jupyter-datascience` stages are nearly identical across pytorch, tensorflow, llmcompressor, and trustyai Dockerfiles. The trade-off is explicit: build independence and cacheability over DRY principles.

**Hermetic build infrastructure**: All Konflux Dockerfiles run without network access. Dependencies are prefetched by Hermeto/Cachi2 into `/cachi2/output/deps/{rpm,pip,generic,gomod}/` and mounted at build time. RPM GPG key imports, `rpms.lock.yaml` cache-busting, and `uv pip install --no-index --find-links /cachi2/output/deps/pip` patterns appear uniformly across all images. The `prefetch-input/` directory contains lock files for both ODH (`prefetch-input/odh/`) and RHOAI (`prefetch-input/rhds/`) distributions, plus yum repo configs for CentOS Stream, EPEL, and RHSM-Pulp. The `uv.lock` at the repo root locks the CI/test Python environment, while per-image `pylock.*.toml` files lock the workbench Python environments — two distinct dependency graphs with different resolution targets.

**Code-Server idle culling stack**: The Code-Server image requires a three-process architecture (code-server + nginx + httpd/CGI) solely to emulate Jupyter's `/api/kernels/` API for the notebook controller's idle culling logic. This is acknowledged as fragile in the existing ARCHITECTURE.md, with plans to replace it with a single Go reverse proxy. The CGI script (`access.cgi`) polls `localhost:8787/codeserver/healthz` and converts heartbeat responses to Jupyter kernel JSON format — a creative but brittle integration point.

**Build-args and version management**: The `versions_config.yml` file centralizes version information (release version, Python version, RHEL OS version, CUDA/ROCm accelerator versions), and `make sync-build-args-from-versions` propagates it to per-image `build-args/konflux.*.conf` files. Each conf file pins the exact AIPCC base image digest, ensuring reproducible builds. The `KONFLUX` Makefile variable selects the product variant (ODH vs RHOAI) by choosing different build-args files and manifest sets, not different Dockerfiles (Dockerfile paths unified since RHAIENG-4516).

**ImageStream versioning strategy**: Each RHOAI ImageStream carries 4 tags: N (current, 3.5), N-1 (3.4), N-2 (2025.2), and N-3 (2025.1). The N-3 tag carries `opendatahub.io/image-tag-outdated: "true"` to signal deprecation to the dashboard. Image digests are injected via kustomize `replacements` from a `notebook-image-params` ConfigMap generated from `params.env` — a complex but automated mechanism with 80+ replacement entries that maps each image key to its ImageStream tag.
