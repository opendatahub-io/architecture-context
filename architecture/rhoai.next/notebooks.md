## Purpose

**Short**: Builds and maintains the Jupyter notebook workbench and Data Science Pipelines runtime container images for OpenDataHub and Red Hat OpenShift AI.

**Detailed**: The notebooks repository is a container image factory that produces 18+ shippable container images spanning three product categories: JupyterLab-based interactive workbenches (minimal, data science, PyTorch, TensorFlow, TrustyAI, PyTorch+LLMCompressor), VS Code-based workbenches (code-server), and headless Data Science Pipeline runtime images. Each image category has multiple accelerator variants — CPU, NVIDIA CUDA (12.9/13.0), and AMD ROCm (7.1) — built from shared base images hosted in the `base-images/` directory.

The repository implements a layered Dockerfile architecture where base images provide OS packages, accelerator drivers, and Python virtual environments, while downstream images layer JupyterLab, ML frameworks (PyTorch, TensorFlow), and domain-specific libraries (TrustyAI, LLMCompressor, Elyra, Kale). All RHOAI production builds use Konflux Dockerfiles (`Dockerfile.konflux.*`) that enforce hermetic builds — every dependency (RPMs, Python wheels, Go modules, npm packages) is prefetched by Cachi2/Hermeto and installed offline. The repository also produces OpenShift ImageStream manifests (via kustomize) that the platform operator (`rhods-operator`) applies to make workbench images selectable in the RHOAI dashboard.

The repository uses the AIPCC Ecosystems team's Python Package Index for resolving Python packages, configured via `INDEX_URL` build arguments pointing to `packages.redhat.com` endpoints. Base images run the `aipcc.sh` script to install system-level dependencies aligned with the RHEL AI package set. Python dependency locking uses `uv` with per-image `pylock.toml` files (PEP 751 format) and `requirements.{flavor}.txt` files for Cachi2 prefetch compatibility.

## Architectural Analysis

The notebooks repository follows a **layered image hierarchy** pattern where base images form a reusable foundation and downstream images compose functionality through multi-stage Dockerfile inheritance. The CPU base image (`base-images/cpu/`) provides the Python 3.12 virtual environment, AIPCC package index configuration, and essential system libraries on CentOS Stream 9. The CUDA and ROCm bases extend the same foundation with GPU acceleration drivers and libraries. This hierarchy is critical for keeping image sizes manageable — a PyTorch CUDA image would be enormous if it installed CUDA from scratch rather than inheriting from the cuda-base layer.

The **hermetic build** architecture is the most distinctive engineering decision in this repository. Every Konflux Dockerfile uses Cachi2/Hermeto prefetched dependencies mounted at `/cachi2/output/deps/`, with `--no-index` flags ensuring zero network access during the build. This covers all dependency layers: RPM packages (via `rpms.lock.yaml`), Python wheels (via `pylock.*.toml`), Go modules (for mongocli), and npm packages (for code-server). The prefetch inputs are organized in `prefetch-input/` with separate `odh/` and `rhds/` directories for upstream and downstream RPM sets. This supply chain hardening is a Red Hat product security requirement — it ensures reproducible builds and prevents dependency confusion attacks.

The **AIPCC (AI Platform and Cloud Computing) Ecosystems** integration is pervasive. The `aipcc.sh` script installs a comprehensive set of system libraries aligned with the RHEL AI package set — including COPR-rebuilt packages (hdf5 with libhdf5.so.310, zeromq 4.3.5) that are newer than what UBI9 provides. The AIPCC Python Package Index at `packages.redhat.com` serves pre-built, vulnerability-scanned wheels for every accelerator variant (CPU, CUDA 12.9/13.0, ROCm 7.1). This centralized package management ensures that all notebook images use consistent, security-vetted Python packages.

The **mongocli binary** deserves attention: it is built from source with FIPS-compliant Go flags (`CGO_ENABLED=1 -tags strictfipsruntime`) using a `ubi9/go-toolset` builder stage, then copied into datascience and pytorch images. This is the only Go artifact in the repository; everything else is Python. The s390x architecture gets a dummy script (`echo "mongocli not supported on s390x"`) because the MongoDB Go driver doesn't support s390x. This FIPS-compliant build pattern ensures the binary passes check-payload validation.

The **versioning strategy** in the ImageStream manifests supports N/N-1/N-2/N-3 image versions (currently 3.5/3.4/2025.2/2025.1), allowing users to pin workbenches to older releases during migration. The ImageStreams use SHA256 digest-pinned references (`registry.redhat.io/rhoai/...@sha256:...`) for immutability. The kustomize replacements system maps ConfigMap entries to ImageStream tag references using over 1250 replacement rules, making the manifest structure complex but fully parameterized.
