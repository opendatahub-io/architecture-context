## Purpose

**Short**: MCP (Model Context Protocol) server that enables AI agents to interact with Red Hat OpenShift AI environments through programmatic tools, resources, and workflow prompts.

**Detailed**: RHOAI MCP Server implements the Model Context Protocol specification to expose RHOAI platform capabilities as MCP tools, resources, and prompts that AI agents (such as Claude Desktop, Lightspeed, or custom LLM agents) can invoke. The server acts as a bridge between AI agents and Kubernetes/OpenShift clusters running RHOAI, providing domain-specific operations for managing data science projects, workbenches (Jupyter notebooks), model serving (KServe InferenceServices), data science pipelines, data connections (S3 secrets), storage (PVCs), distributed training (Kubeflow Training Operator), model registry integration, and Neural Navigator model recommendations.

The server uses a pluggy-based plugin architecture where each RHOAI domain is implemented as an independent plugin that registers its tools, resources, and prompts via hook specifications. Plugins undergo health checks at startup to verify required CRDs exist in the cluster, enabling graceful degradation — if KServe CRDs are absent, the inference plugin is disabled but the server continues to serve other domains. The server supports three MCP transport modes (stdio, SSE, streamable-http), multiple Kubernetes authentication modes (auto-detect, kubeconfig, token), and configurable safety controls (read-only mode, dangerous operations gating).

## Architectural Analysis

The RHOAI MCP Server demonstrates a well-structured plugin-based architecture built on the pluggy framework. The separation between domain plugins (CRUD on specific resource types) and composite plugins (cross-cutting orchestration) is a clean design that allows independent evolution of each concern. The plugin health check mechanism at startup enables graceful degradation — a cluster without KServe CRDs still serves notebook and pipeline operations, which matches real-world deployment scenarios where not all RHOAI components are installed.

The server's approach to Kubernetes authentication is pragmatic. The `AuthMode.AUTO` mode tries in-cluster service account auth first, then falls back to kubeconfig, which works for both deployed containers and local development. The safety system with `read_only_mode` and `enable_dangerous_operations` flags provides defense-in-depth against unintended modifications, which is critical given that AI agents are the callers and could hallucinate destructive actions.

A notable architectural pattern is the verbosity system. Every list/get tool supports `minimal`, `standard`, and `full` verbosity levels, with minimal mode providing approximately 85% token savings. This is a practical optimization for LLM agents that have context window limits — the server adapts its response size to the caller's needs rather than always returning full resource details. The companion `ResponseBuilder` utility class centralizes response formatting across all domains.

The Model Registry domain showcases a sophisticated multi-backend design. It auto-discovers Model Registry instances from the cluster (via ModelRegistry CRDs and OpenShift Routes), supports two distinct API types (Kubeflow Model Registry v1alpha3 and Red Hat AI Model Catalog v1alpha1), and transparently falls back between them. The authentication layer supports three modes (none for in-cluster, OAuth via `oc whoami -t`, and explicit token), and the port-forwarding mechanism uses subprocess management with reference counting to tunnel into in-cluster services from external clients.

The Neural Navigator integration is the most externally-coupled part of the architecture. The `recommend_model` composite tool chains five sequential HTTP calls to the Neural Navigator backend to extract user intent, get SLO defaults, compute workload profiles, and produce ranked model recommendations. The `get_deployment_config` tool then generates ready-to-apply Kubernetes YAML (InferenceService, HPA, ServiceMonitor) via the deploy endpoint. This tight coupling means Neural Navigator availability directly affects these tools, though the plugin health check mechanism prevents startup failures.

The evaluation framework (docker-compose.eval.yml) reveals the intended deployment topology: the MCP server runs alongside Lightspeed Stack (which routes LLM queries through the MCP tools) and Llama Stack (which provides LLM inference). This three-service architecture enables end-to-end testing of AI agent workflows against a mock Kubernetes cluster without requiring real infrastructure.
