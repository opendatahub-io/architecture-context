## Purpose

**Short**: An MCP (Model Context Protocol) server that enables AI agents to interact with Red Hat OpenShift AI (RHOAI) environments through programmatic tools, resources, and workflow prompts.

**Detailed**: The RHOAI MCP Server replicates the capabilities of the OpenShift AI Dashboard through a programmatic interface designed for AI agents. It provides 60+ MCP tools organized into domain modules (projects, notebooks, inference, pipelines, connections, storage, training, model registry, quickstarts) and composite modules (cluster exploration, training workflow orchestration, tool discovery, Neural Navigator model recommendations). The server communicates with Kubernetes/OpenShift APIs to manage RHOAI resources and with external REST APIs (Model Registry, Model Catalog, Neural Navigator) for model discovery and recommendation.

The server is built on the FastMCP framework with a pluggy-based plugin architecture that enables modular registration of tools, resources, and prompts. It supports three transport modes (stdio, SSE, streamable-http) for integration with Claude Desktop, Claude Code, and HTTP-based clients. Safety features include read-only mode, dangerous operation gating, credential masking, and RBAC-aware project enumeration.

## Architectural Analysis

The RHOAI MCP Server follows a **domain-driven plugin architecture** that cleanly separates concerns. Each RHOAI resource type (notebooks, inference services, pipelines, etc.) is encapsulated in a self-contained domain module with its own client, models, tools, and CRD definitions. This design enables independent testing, optional feature gating via the `RHOAI_MCP_ENABLED_PLUGINS` configuration, and clean extension via external plugins through the `rhoai_mcp.plugins` entry point group.

**The composite layer is architecturally significant.** Rather than requiring AI agents to make multiple sequential tool calls across domains, composite tools like `explore_cluster()`, `prepare_training()`, and `diagnose_resource()` orchestrate multiple domain clients in a single call. This reduces token usage by 70-90% according to the verbosity system, which is critical for AI agent context window management. The `ResponseBuilder` utility enforces three verbosity levels (minimal/standard/full) across all domains, ensuring consistent token-efficient responses.

**The Model Registry domain is unique** in that it communicates via HTTP REST API rather than Kubernetes CRDs. It supports two API backends (Kubeflow Model Registry and Red Hat AI Model Catalog) with automatic API type probing and fallback. The auto-discovery mechanism queries the `ModelRegistry` component CRD to find the registries namespace, then scans for services, with intelligent port preference (8080 for direct REST, 8443 for kube-rbac-proxy). When running outside the cluster, it automatically sets up `oc port-forward` tunnels via the `PortForwardManager`, which uses reference counting and singleton patterns for connection reuse across MCP sessions.

**The Neural Navigator integration** adds AI-guided model selection capabilities. It calls a separate backend service to extract deployment intent from natural language, compute SLO defaults and workload profiles, get ranked model recommendations, and generate complete Kubernetes deployment YAML (InferenceService, autoscaling, ServiceMonitor). This represents an interesting "AI calling AI" pattern — an AI agent uses MCP tools to invoke another AI-powered recommendation engine.

**Security is layered but relies primarily on Kubernetes RBAC.** The MCP server itself has no authentication layer — it inherits permissions from the configured service account (in-cluster) or kubeconfig (external). Safety features are implemented at the application level: read-only mode blocks all mutations, dangerous operation gating requires explicit enablement of deletes, and a confirmation pattern requires `confirm=True` for destructive operations. Credential masking ensures S3 secret keys are never returned in full. However, the lack of MCP-level authentication means the server trusts any connected client, making network isolation (via the OpenShift Route) the primary access control mechanism for HTTP transports.

**The quickstarts domain introduces a different risk profile** by executing `git clone`, `helm upgrade`, and `oc apply` as subprocesses. While timeouts are enforced (60s for git, 300s for deployment), this pattern allows the MCP server to execute arbitrary Kubernetes manifests from external repositories. The `GIT_TERMINAL_PROMPT=0` setting prevents credential prompts but does not validate repository content.
