## Purpose

**Short**: An MCP (Model Context Protocol) server that enables AI agents to interact with Red Hat OpenShift AI environments, replicating the capabilities of the OpenShift AI Dashboard through programmatic tools.

**Detailed**: The RHOAI MCP Server is a Python service that implements the Model Context Protocol (MCP) specification to expose Red Hat OpenShift AI (RHOAI) platform capabilities as tools, resources, and prompts consumable by AI agents (such as Claude Desktop, Claude Code, or Lightspeed). It acts as a bridge between LLM-based agents and the Kubernetes API, providing structured access to RHOAI operations including Data Science Project management, Jupyter workbench lifecycle, KServe model serving, data connection management, Data Science Pipelines, Kubeflow Training Operator jobs, and Model Registry integration.

The server uses a pluggy-based plugin architecture that enables modular tool registration, health-check-based graceful degradation, and extensibility via external Python entry points. It supports three MCP transport modes (stdio, SSE, streamable-http), three Kubernetes authentication modes (auto/kubeconfig/token), and optional OIDC-based multi-user authentication with per-tool RBAC filtering via Kubernetes SubjectAccessReview. The architecture is organized into domain plugins (projects, notebooks, inference, connections, storage, pipelines, training, model_registry, quickstarts, prompts) and composite plugins (cluster summaries, training orchestration, meta/tool discovery, planner/model recommendations) that aggregate domain operations into context-efficient workflows optimized for AI agent token budgets.

## Architectural Analysis

The RHOAI MCP Server represents an innovative approach to platform interaction, positioning the MCP protocol as a programmatic interface layer that sits alongside the traditional web dashboard. By exposing RHOAI operations as MCP tools, the server enables AI agents to perform complex multi-step workflows (such as "set up a training project with GPU resources, create a data connection, and submit a fine-tuning job") that would otherwise require extensive Kubernetes API knowledge. The plugin architecture is particularly well-designed: each domain (projects, notebooks, inference, etc.) is a self-contained plugin that declares its CRD dependencies, tool permissions, and health checks. If a CRD is unavailable on the cluster (e.g., the Training Operator is not installed), the corresponding plugin degrades gracefully without affecting other domains.

The composite tool layer is architecturally significant. Rather than exposing 40+ individual tools that would overwhelm an AI agent's context window, composite tools like `cluster_summary`, `diagnose_resource`, and the unified `training` tool consolidate multi-step operations. The `suggest_tools` meta-tool provides intent-based tool discovery, enabling agents to find the right workflow without enumerating all available tools. This context-efficiency design (documented as achieving 70-90% token savings) reflects a deep understanding of how LLM-based agents consume and reason about available capabilities.

The authentication architecture supports two distinct deployment modes: a single-user mode (kubeconfig or ServiceAccount) where the server acts with the credentials of its own ServiceAccount, and a multi-user OIDC mode where each request carries a Bearer token that is validated via either JWKS-based JWT verification or Kubernetes TokenReview API. In multi-user mode, the server uses Kubernetes Impersonate-User/Impersonate-Group headers to delegate operations to the authenticated user's identity, ensuring that the platform's existing RBAC policies are enforced. The per-tool RBAC filtering (which patches FastMCP's lowlevel request handlers) ensures that users only see tools they have permission to use. This is implemented as a pure ASGI middleware to avoid BaseHTTPMiddleware's incompatibility with SSE streaming -- a subtle but important architectural decision.

The workflow token system (HMAC-signed, time-limited tokens embedded in tool responses) addresses a unique challenge in agent-based workflows: preventing agents from skipping prerequisite steps. For example, a `prepare_training` tool produces a workflow token that `create_training_job` requires, ensuring agents validate prerequisites before submitting jobs. This stateless approach (all state encoded in the token) avoids server-side session management while providing ordering guarantees.

The deployment manifests demonstrate a thoughtful layering strategy. The base kustomization provides hardened defaults (non-root, read-only rootfs, dropped capabilities, seccomp RuntimeDefault), while overlays progressively customize for each target environment. The RHOAI overlay defaults to read-only mode as a production safety measure, and the image injection mechanism (params.env → configMapGenerator → replacement) enables air-gapped deployments. One notable gap is the absence of a Konflux Dockerfile -- the component builds via a standard Containerfile with unpinned UBI base images, indicating it has not yet completed Konflux onboarding for the RHOAI product pipeline. The use of `hashlib.md5` for training job name generation, while not a security concern (it's used for uniqueness, not cryptography), could trigger static analysis warnings in FIPS-scanned environments.
