## Purpose

**Short**: A lightweight Rust HTTP service that detects PII and custom patterns in text using regex matching, designed as a detector plugin for the FMS Guardrails Orchestrator.

**Detailed**: The Guardrails Regex Detector is a stateless REST API service built with the Axum web framework in Rust. It accepts text content via HTTP POST requests and scans for personally identifiable information (PII) such as email addresses, Social Security numbers, and credit card numbers using predefined regex patterns. It also supports arbitrary custom regex patterns provided in the request payload.

This component operates as a detector backend within the FMS Guardrails ecosystem. The FMS Guardrails Orchestrator dispatches text content to this detector, which returns structured detection results including match positions, detected text, detection type classification (PII vs custom), and confidence scores. The service is designed to be deployed as a sidecar or standalone microservice alongside the orchestrator.

The detector is intentionally minimal — a single-binary Rust service with no database dependencies, no persistent state, and no TLS configuration. It relies on the platform's network infrastructure (service mesh, network policies) for transport security and on the orchestrator for authentication and request routing.

## Architectural Analysis

The Guardrails Regex Detector is an intentionally minimal, single-purpose microservice. The entire codebase consists of two Rust source files (~210 lines total including tests): `main.rs` for HTTP server setup and `detectors.rs` for detection logic. This extreme simplicity is a deliberate architectural choice — the service does one thing (regex matching on text) with zero external dependencies beyond the text content in each request.

The detection architecture uses a two-tier matching strategy. Three built-in detectors (email, SSN, credit card) are registered by name and can be invoked by passing their string keys (`"email"`, `"ssn"`, `"credit-card"`) in the `detector_params.regex` array. Any string that doesn't match a built-in detector name is treated as a raw regex pattern and compiled on-the-fly. This means every request that includes custom regex patterns incurs regex compilation overhead — there is no caching of compiled regex objects across requests. For high-throughput production use, this is a potential performance bottleneck, especially with complex patterns.

The service has no authentication, no TLS, no persistent state, and no configuration beyond two environment variables (`HTTP_PORT` and `HOST`). It is designed to run behind the FMS Guardrails Orchestrator, which handles client authentication and request routing. The absence of transport encryption means this service MUST be deployed within a network perimeter that provides encryption at the infrastructure layer (e.g., Istio mTLS, Kubernetes network policies). Exposing this service directly to untrusted networks would allow plaintext interception of PII data being scanned.

The Dockerfile uses a multi-stage build pattern with UBI 9 Minimal as the runtime base image. Notably, there is no Konflux-specific Dockerfile (`Dockerfile.konflux`), which suggests this component may not yet be fully onboarded to the Konflux build pipeline used by RHOAI. The `compat-openssl11` RPM is installed in the runtime image, likely to satisfy dynamic linking requirements, even though the application itself performs no TLS operations. The Dockerfile also lacks a `USER` directive, meaning the container runs as root by default — a security concern for production deployments.

The response format wraps all detections in a double-nested JSON array (`[[...]]`), which aligns with the FMS Guardrails detector API contract where the outer array corresponds to content items and the inner array to detections within each item. However, the current implementation flattens all detections into a single inner array regardless of which content item they matched against, which could make it difficult for the orchestrator to correlate detections back to specific input texts.
