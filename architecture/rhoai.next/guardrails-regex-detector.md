## Purpose

**Short**: A lightweight HTTP server that detects PII and custom patterns in text using regular expressions, serving as a detection backend for the FMS Guardrails Orchestrator.

**Detailed**: The Guardrails Regex Detector is a Rust-based microservice built with the Axum web framework that provides text content scanning via regex pattern matching. It exposes a single POST endpoint that accepts text content and a list of regex patterns (either built-in named patterns or custom regex strings), returning detection results with match positions and classification metadata.

The service ships with three built-in PII detection patterns — email addresses, Social Security numbers, and credit card numbers — each returning typed detection results (detection_type: "pii"). Custom regex patterns submitted by callers are also supported, returning results with detection_type: "custom". The service is designed to run as a sidecar or backend service alongside the FMS Guardrails Orchestrator, which routes content inspection requests to it.

The binary is compiled as a statically-linked Rust executable and packaged into a UBI 9 minimal container image for deployment in OpenShift environments. The service binds to `0.0.0.0:8080` by default but supports configuration via `HTTP_PORT` and `HOST` environment variables.

## Architectural Analysis

The Guardrails Regex Detector is intentionally minimal — a single-purpose microservice with approximately 170 lines of Rust code across two source files. The architecture follows a straightforward request-handler pattern: `main.rs` sets up the Axum router and TCP listener, while `detectors.rs` contains all detection logic. There is no database, no persistent state, and no outbound network calls. This simplicity is a deliberate design choice for a service that performs pure computation (regex matching) on request payloads.

The detection engine uses a registry pattern (`builtin_regex` HashMap) that maps string keys ("email", "ssn", "credit-card") to detector functions. Each detector function constructs a `RegexDetection` struct with a compiled regex pattern and metadata, then delegates to the shared `regex_match` function. Custom patterns bypass the registry and are compiled on every request. This means maliciously crafted regex patterns could cause ReDoS (Regular Expression Denial of Service) since there is no regex complexity limit or compilation timeout. The `regex` crate does provide some protection against catastrophic backtracking by default, but no explicit safeguards are present in the application code.

The service has no authentication, authorization, or TLS termination — it relies entirely on network-level isolation (Kubernetes NetworkPolicy or service mesh) for security. This is typical for an internal detection backend that sits behind the Guardrails Orchestrator, which handles external-facing authentication. The Dockerfile uses UBI 9 minimal as the base image and installs `compat-openssl11`, suggesting the compiled binary or a dependency requires OpenSSL 1.1 compatibility libraries at runtime.

The container build uses a multi-stage Dockerfile with dedicated stages for building, testing (`cargo test`), linting (`cargo clippy`), and formatting checks (`cargo fmt --check`). There is no Konflux-specific Dockerfile — the standard `Dockerfile` is the only build artifact. No kustomize manifests, Helm charts, or Kubernetes deployment manifests are included in this repository; deployment configuration is likely managed externally by the platform operator or the Guardrails Orchestrator's deployment manifests.
