## Purpose

**Short**: A lightweight HTTP server that detects PII and custom patterns in text using regex matching, serving as a detection backend for the FMS Guardrails Orchestrator.

**Detailed**: The Guardrails Regex Detector is a Rust-based HTTP microservice built with the Axum web framework. It provides a single detection endpoint that accepts text content and regex parameters, then returns structured detection results identifying matches with their positions, types, and scores.

The service includes three built-in PII detection patterns — email addresses, Social Security numbers, and credit card numbers — that can be invoked by name. It also supports arbitrary custom regex patterns passed directly in the request payload. All detections return a uniform response structure with start/end offsets, matched text, detection type classification, and a confidence score (always 1.0 for regex matches).

This component is designed to run as a sidecar or backend service within the FMS Guardrails architecture, where the Guardrails Orchestrator routes text content to various detectors (including this one) for content safety screening before or after LLM inference.

## Architectural Analysis

The Guardrails Regex Detector is intentionally minimal — a ~200-line Rust service with exactly two endpoints and zero external dependencies beyond text processing. This simplicity is a deliberate architectural choice: regex-based PII detection requires high throughput and low latency (every LLM request may pass through it), and Rust with Tokio's multi-threaded async runtime provides that without the overhead of garbage collection or interpreter startup.

The service follows a detector plugin pattern used across the FMS Guardrails ecosystem. The orchestrator dispatches text to multiple detector backends (regex, ML-based, custom) via a common API contract (`POST /api/v1/text/contents` with `contents` and `detector_params`). This detector implements the regex variant, where `detector_params.regex` can reference built-in named patterns (`email`, `ssn`, `credit-card`) or pass arbitrary regex strings for custom matching. The uniform `DetectionResponse` structure (start/end offsets, detection type, score) allows the orchestrator to merge results from heterogeneous detectors.

A notable architectural gap is the absence of any authentication or TLS on the HTTP endpoint. The service relies entirely on network-level isolation for security — it is assumed to be deployed within a pod network where only the orchestrator can reach it. There are no RBAC definitions, no service mesh configuration, and no network policies in this repository. These would be provided by the platform operator (rhods-operator) or the Guardrails orchestrator's deployment manifests. The service also has no rate limiting, request size validation, or timeout controls beyond what Axum provides by default, which could be a concern if exposed to untrusted input.

The container image is built on UBI9 minimal, which is appropriate for RHOAI compliance. However, the absence of a Konflux Dockerfile (`Dockerfile.konflux`) on this branch suggests the component either hasn't been onboarded to the Konflux build pipeline or the Konflux configuration exists only on downstream release branches. The `compat-openssl11` package is installed in the image but not used by the application code — it may be present for system-level compatibility or as a check-payload requirement.
