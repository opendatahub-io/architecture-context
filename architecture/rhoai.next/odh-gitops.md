## Purpose

**Short**: GitOps repository providing Kustomize manifests and Helm charts for deploying OpenDataHub (ODH) and Red Hat OpenShift AI (RHOAI) platform dependencies, operator configurations, and complete deployment stacks across OpenShift and vanilla Kubernetes environments.

**Detailed**: odh-gitops serves as the single source of truth for deploying the complete RHOAI/ODH dependency stack — the set of prerequisite operators and their configurations that the platform requires before it can function. This repository does not contain the platform operators themselves (rhods-operator or opendatahub-operator), but rather manages the surrounding ecosystem of dependency operators (cert-manager, Kueue, Istio/Sail, Gateway API, Leader Worker Set, Kuadrant/RHCL, etc.) and provides the scaffolding to install the main platform operator alongside them.

The repository supports two primary deployment paradigms. First, a **Kustomize-based approach** for OpenShift clusters with OLM (Operator Lifecycle Manager), where each dependency is a self-contained Kustomize component with Namespace, OperatorGroup, and Subscription resources that can be composed via layered `kustomization.yaml` files and applied with ArgoCD, Flux, or `kubectl`. Second, a **Helm chart approach** with two distinct charts: `rhai-on-openshift-chart` for OLM-based OpenShift deployments (managing 13 platform components and 11 dependency operators with profile-based configuration and tri-state dependency resolution), and `rhai-on-xks-chart` for non-OpenShift Kubernetes services (Azure AKS, CoreWeave, AWS EKS) that bundles the RHAI operator, cloud-specific managers, CRDs, and post-install hooks without relying on OLM. The Helm charts include dependency sub-charts (cert-manager-operator, gateway-api, lws-operator, sail-operator) extracted from OLM bundles for vanilla Kubernetes use.

## Architectural Analysis

### Layered Deployment Architecture

The repository implements a sophisticated layered deployment model that addresses three distinct deployment scenarios. The **Kustomize layer** provides the most granular control, with each operator packaged as a self-contained Kustomize Component (not a mere resource directory) that encapsulates its Namespace, OperatorGroup, and Subscription. These components are composed via overlay kustomization files in `dependencies/operators/`, with transitive dependency support (e.g., kueue-operator's kustomization includes cert-manager). The **configurations layer** sits atop the dependencies layer and creates the post-install CRs that activate the operators (Kueue CR, Kuadrant CR, etc.). This two-phase approach (`kubectl apply -k dependencies/` then `kubectl apply -k configurations/`) mirrors the natural OLM lifecycle where CRDs must exist before CRs can be created.

### Helm Chart Design Patterns

The `rhai-on-openshift-chart` exhibits several notable design patterns. The **tri-state dependency resolution** (auto/true/false) with transitive support means that enabling a single component like KServe automatically pulls in cert-manager, RHCL, and Leader Worker Set without explicit configuration. The **profile system** (`default` vs `rhaii`) provides curated deployment presets — the `rhaii` profile enables only KServe for inference-only deployments, while the `default` profile starts everything in Removed state. The **two-pass rendering** pattern handles the OLM chicken-and-egg problem: CRDs don't exist during the first `helm install`, so CR templates use `crdExists` guards that evaluate to false; a second `helm upgrade` after operators are running finds the CRDs and renders the CRs.

The `rhai-on-xks-chart` takes a fundamentally different approach by bundling everything needed for non-OLM environments. It deploys the RHAI operator directly (not via OLM), includes cloud-specific manager operators (Azure, CoreWeave, AWS) with exactly-one-enabled validation, and uses Helm post-install hooks to create CRs and Gateway resources in the correct order. The Gateway creation script (`create-gateway.sh`) is particularly complex — it waits for CRDs, creates CA ConfigMaps, optionally provisions TLS Certificates via cert-manager, waits for GatewayClass availability, and then creates the Gateway with configurable allowed routes and listener configuration.

### CI/CD and Validation Strategy

The Tekton pipeline infrastructure provisions **ephemeral AWS HyperShift clusters** for end-to-end testing, supporting both the Kustomize path (`make apply-and-verify-dependencies`) and the Helm chart path (`make helm-install-verify`). The pipeline includes custom OLM catalog injection, detailed OLM verification (CSV phase checking, pod readiness), and multi-stage Helm installation with intermediate CRD waiting. The `verify-dependencies.sh` script contains operator-specific health checks that go beyond simple pod readiness — for example, it checks multiple KEDA sub-components (operator, metrics-apiserver, admission-webhooks) and multiple RHCL sub-components (kuadrant-operator, authorino, limitador, dns-operator). The snapshot-based Helm testing (`chart-snapshots.sh`) provides regression detection for template rendering across 20+ test scenarios.

### Contract-Based Schema Validation

The `contracts/schemas/` directory contains 40+ JSON schemas that define the expected shape of CRDs consumed by the charts. These schemas serve as a contract between the Helm chart templates and the upstream CRD definitions. The KServe schema, for example, defines model cache configuration with mutual exclusivity validation between `nodeNames` and `nodeSelector` and conditional required fields (`cacheSize` required when `managementState` is `Managed`). The Platform schema defines the xKS-specific monitoring module configuration. These contracts enable chart-level validation without requiring a live cluster.

### Security Considerations

All operator deployments in the dependency charts follow a consistent security posture: `runAsNonRoot: true`, read-only root filesystem, `seccompProfile: RuntimeDefault`, and no privilege escalation. The `kube-linter` configuration enforces additional security checks (privileged containers, privileged ports, cluster-admin bindings, wildcard RBAC rules) with a custom CEL check preventing bindings to `system:authenticated` or `system:unauthenticated` groups. The xKS chart's extensive RBAC (1419 lines in the ClusterRole) reflects the broad permissions needed for a platform operator that manages the entire stack — this is a potential area for future RBAC narrowing once component boundaries stabilize.
