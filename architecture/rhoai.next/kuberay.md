## Purpose

**Short**: Manages the lifecycle of Ray clusters, jobs, and services on Kubernetes/OpenShift by reconciling RayCluster, RayJob, and RayService custom resources.

**Detailed**: KubeRay is a Kubernetes operator that automates the deployment and management of Ray — a distributed computing framework for ML workloads. It provides three CRDs: `RayCluster` for running persistent Ray clusters with head and worker pods, `RayJob` for submitting one-shot jobs to ephemeral or existing clusters, and `RayService` for deploying long-running Ray Serve applications with zero-downtime upgrade support.

The RHOAI downstream fork extends the upstream KubeRay operator with OpenShift-specific features: Gateway API integration for dashboard access via HTTPRoute (replacing direct Ingress/Route), kube-rbac-proxy and OAuth proxy sidecars for authenticated dashboard access, mTLS certificate management via cert-manager for intra-cluster communication, NetworkPolicy enforcement for per-cluster network isolation, and SecurityContextConstraints for running Ray pods as a dedicated user. The operator integrates with the RHOAI platform by creating HTTPRoutes in the platform namespace referencing the shared `data-science-gateway` Gateway, enabling centralized ingress management.

The operator also supports pluggable batch schedulers (Volcano, Yunikorn, scheduler-plugins) for gang scheduling of Ray workloads, autoscaling via Ray's built-in autoscaler, and GCS fault tolerance with Redis for durable cluster state.

## Architectural Analysis

### OpenShift Gateway API Integration Pattern

The RHOAI downstream extends KubeRay with a sophisticated Gateway API integration pattern that is architecturally distinct from the upstream Ingress/Route approach. On OpenShift, the mutating webhook enforces `enableIngress: false` and sets `odh.ray.io/secure-trusted-network: true`, which activates the Authentication Controller. This controller creates HTTPRoutes in the platform namespace (e.g., `redhat-ods-applications`) rather than the cluster namespace, requiring a ReferenceGrant in the cluster namespace to authorize cross-namespace service references. The ReferenceGrant uses a shared-per-namespace model with reference counting — one grant serves all authenticated RayClusters in a namespace, and it is only deleted when the last authenticated cluster is removed.

The HTTPRoute path structure (`/ray/{namespace}/{cluster}/`) provides natural multi-tenancy: each RayCluster gets a unique, namespace-scoped path prefix under the shared Gateway. URL rewriting strips this prefix before forwarding to the dashboard, and an exact-match rule handles the root path redirect to the dashboard's hash-based routing (`/#/`).

### Defense-in-Depth Security Model

The security architecture implements four layers of defense, all controlled by a single annotation (`odh.ray.io/secure-trusted-network`). The NetworkPolicy Controller creates per-cluster policies that restrict head pod access to intra-cluster traffic on all ports, namespace-local access on specific ports (10001, 8265), operator access for health checks, and gateway namespace access for ingress — while exposing only port 8443 (the auth proxy) to external traffic. The Authentication Controller injects either an OAuth proxy or kube-rbac-proxy sidecar that terminates TLS on 8443 and proxies authenticated requests to the dashboard on localhost:8265. The MTLS Controller manages a per-cluster CA hierarchy via cert-manager with 90-day node certificates and automatic renewal, securing all head-to-worker and worker-to-worker communication. This layered approach means that even if one layer is compromised, the others provide independent protection.

### Controller Decomposition Strategy

Rather than a single monolithic reconciler, the RHOAI KubeRay operator decomposes security concerns into three independent controllers (Authentication, MTLS, NetworkPolicy) that all watch the same RayCluster CRD but manage orthogonal resource sets. This reduces reconcile contention and allows each security feature to evolve independently. The Authentication Controller uses a finalizer (`ray.io/authentication-resources`) for cleanup of cross-namespace resources that cannot rely on Kubernetes garbage collection, while the MTLS and NetworkPolicy controllers use standard controller references for automatic cleanup.

### Batch Scheduler Plugin Architecture

The operator implements a plugin system for batch schedulers through the `BatchSchedulerFactory` interface, supporting Volcano (gang scheduling via PodGroup CRD), Yunikorn (gang scheduling via pod annotations with taskGroups), and scheduler-plugins (coscheduling via PodGroup). The plugin model cleanly separates scheduler-specific logic from core reconciliation — each plugin implements `DoBatchSchedulingOnSubmission()` (pre-pod creation), `AddMetadataToPod()` (per-pod annotation injection), and `ConfigureReconciler()` (controller watch setup). This allows the same operator binary to work with different cluster schedulers without code changes.

### RayService Zero-Downtime Upgrade Model

RayService implements a blue-green deployment pattern where a "pending" RayCluster is created alongside the "active" cluster. The controller pushes Serve configuration to the pending cluster, monitors application health via the HTTP proxy, and atomically switches the Kubernetes Service selector to the new cluster once all applications are healthy. An LRU cache tracks Serve configuration to avoid redundant API calls. This model guarantees that client traffic sees no interruption during upgrades, though it temporarily doubles resource consumption.
