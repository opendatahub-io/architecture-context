## Purpose

**Short**: Manages the lifecycle of Ray clusters, jobs, and services on Kubernetes/OpenShift.

**Detailed**: KubeRay is a Kubernetes operator that automates the deployment and management of Ray workloads. It provides three Custom Resource Definitions — RayCluster, RayJob, and RayService — which allow users to declaratively create Ray clusters with head and worker nodes, submit Ray jobs with automatic cluster provisioning, and deploy Ray Serve applications with zero-downtime upgrades. The operator handles pod lifecycle management, autoscaler integration, service creation, and batch scheduler coordination (Volcano, Yunikorn, scheduler-plugins).

In the RHOAI distribution, KubeRay is extended with significant OpenShift-specific features. The operator automatically injects authentication sidecars (kube-rbac-proxy for OIDC or oauth-proxy for integrated OAuth) into Ray head pods, creates HTTPRoute resources via Gateway API for secure cluster access through the platform's ingress gateway, manages mTLS certificates between Ray nodes using cert-manager, and enforces network isolation via Kubernetes NetworkPolicies. A mutating webhook defaults the security annotation `odh.ray.io/secure-trusted-network=true` on OpenShift clusters and blocks direct ingress/route creation, forcing all external access through the Gateway API pathway.

The operator integrates with the RHOAI platform by placing HTTPRoutes in the platform namespace (configured via `APPLICATION_NAMESPACE`) and referencing the platform's `data-science-gateway` Gateway resource. ReferenceGrants enable cross-namespace routing from the platform namespace back to Ray cluster services in user namespaces.

## Architectural Analysis

The KubeRay operator in the RHOAI distribution represents a significant extension of the upstream KubeRay project, adding a full security stack that transforms Ray from a development-oriented tool into an enterprise-grade platform component. The most architecturally notable aspect is how the operator bootstraps a complete authentication and ingress chain per RayCluster: on OpenShift, the mutating webhook automatically annotates every RayCluster with `odh.ray.io/secure-trusted-network=true` and blocks native ingress/route creation, then four separate controllers (RayCluster, Authentication, mTLS, NetworkPolicy) cooperate to build the security envelope around each cluster.

The authentication controller's cross-namespace design is particularly sophisticated. HTTPRoutes are placed in the platform namespace (not the user's namespace) to integrate with the platform's `data-science-gateway` Gateway resource, while ReferenceGrants in the user namespace permit the cross-namespace Service reference. The ReferenceGrant is shared per namespace with reference counting — it's only deleted when the last authenticated RayCluster in the namespace is removed. The authentication controller's finalizer (`ray.io/authentication-resources`) ensures cross-namespace cleanup even when owner references can't span namespaces.

The mTLS controller leverages cert-manager to build a two-level certificate hierarchy per RayCluster: a SelfSigned Issuer bootstraps a CA Certificate (RSA 4096-bit, 10-year validity), which then signs per-role certificates (90-day validity, 15-day renewal) for head and worker nodes. Worker certificates include wildcard SANs (`*.{svc}.{ns}.svc`) to accommodate dynamic worker scaling without certificate regeneration. The controller monitors pod IP changes and updates certificate SANs when the worker set changes, which is critical for environments where pod IPs are baked into certificates.

The NetworkPolicy controller creates a security perimeter that is tightly scoped: worker-to-worker traffic is unrestricted within a cluster, head node access is limited to cluster-internal pods, the operator namespace, the monitoring namespace, and the gateway namespace. Port 8443 (auth proxy) is intentionally left open from all sources since authentication is enforced at the application level by kube-rbac-proxy. The controller's namespace detection logic (`APPLICATION_NAMESPACE` → `POD_NAMESPACE` → hardcoded fallbacks) mirrors the authentication controller's gateway namespace resolution, ensuring consistent platform integration.

A design tension exists between the upstream ingress model and the RHOAI security model. Upstream KubeRay supports user-configured `enableIngress` to create Routes/Ingresses directly, but the RHOAI webhook forcefully overrides this to `false` and logs a warning, channeling all access through the Gateway API. This is enforced at admission time rather than reconcile time, which is a stronger guarantee — but it means users who set `enableIngress: true` in their manifests will see it silently overridden. The webhook also defaults the security annotation, making all RayClusters on OpenShift authenticated by default.
