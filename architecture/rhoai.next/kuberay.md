## Purpose

**Short**: Kubernetes operator that manages the lifecycle of Ray clusters, jobs, and serving applications on OpenShift.

**Detailed**: KubeRay Operator provides three Custom Resource Definitions (RayCluster, RayJob, RayService) that enable users to deploy and manage Ray distributed computing clusters on Kubernetes. The operator handles pod provisioning, autoscaling coordination, service exposure, and lifecycle management for Ray workloads including batch jobs and real-time serving applications.

The RHOAI distribution adds OpenShift-specific capabilities including Gateway API integration for HTTPRoute-based ingress, kube-rbac-proxy sidecar injection for OIDC authentication, mTLS certificate management via cert-manager, NetworkPolicy enforcement for pod isolation, and SecurityContextConstraints for non-root execution. The mutating webhook enforces OpenShift compliance by automatically enabling secure trusted network annotations and disabling direct Ingress creation in favor of Gateway API.

The operator runs six independent controllers: RayCluster (pod/service lifecycle), RayJob (job submission and tracking), RayService (zero-downtime serving upgrades), Authentication (OIDC proxy and HTTPRoute management), NetworkPolicy (pod network isolation), and mTLS (certificate chain management via cert-manager).

## Architectural Analysis

The KubeRay operator follows a multi-controller architecture where six independent controllers watch the same primary CRD (RayCluster) but manage orthogonal concerns. The RayCluster controller handles core pod and service lifecycle, while Authentication, NetworkPolicy, and mTLS controllers layer security features on top via annotation-based activation (`ray.openshift.ai/enable-secure-trusted-network=true`). This design allows the operator to run unchanged on vanilla Kubernetes (where the security controllers are no-ops) while enabling full RHOAI security features on OpenShift via the mutating webhook that sets the activation annotation.

The Authentication controller's cross-namespace HTTPRoute design is architecturally notable. It creates HTTPRoutes in the platform namespace (where the Gateway lives) rather than the user's namespace, then creates ReferenceGrants in the user's namespace to authorize the cross-namespace Service reference. This pattern ensures all Ray dashboard traffic flows through the platform's centralized ingress gateway with OIDC enforcement, while the ReferenceGrant provides namespace-scoped authorization that gets garbage collected when the RayCluster is deleted. The cleanup is security-ordered: ReferenceGrant is removed first (revoking access) before removing the HTTPRoute.

The mTLS controller implements a full certificate chain (self-signed Issuer -> CA Certificate -> CA Issuer -> head/worker Certificates) and solves a race condition where cert-manager must include pod IPs as Subject Alternative Names before the Ray GCS process loads the certificate. It achieves this by injecting a `wait-for-tls-ip-san` init container that blocks pod startup until the certificate contains the pod's IP. The controller periodically reconciles (every 60 seconds) to update certificates when pod IPs change due to scaling or restarts.

The RayService controller's zero-downtime upgrade architecture uses a dual-cluster model where a pending cluster is fully warmed up (Ray Serve applications deployed and healthy) before traffic is switched from the active cluster. The controller uses spec hashing (excluding autoscaling fields like replicas/minReplicas/maxReplicas) to detect meaningful spec changes that require a new cluster, and allows partial hash matching to enable adding new worker groups without triggering a full cluster recreation. A 60-second delay protects against premature deletion of dangling clusters from failed upgrades.

The operator exposes rich Prometheus metrics through three dedicated metrics managers that implement the `prometheus.Collector` interface. These emit cluster provisioning duration, job execution duration, deployment status, service readiness conditions, and per-cluster info metrics with labels for namespace, owner kind, and retry count. Combined with the external PodMonitors for Ray head/worker metrics and PrometheusRules for GCS availability alerting, this provides comprehensive observability across both the control plane and data plane.
