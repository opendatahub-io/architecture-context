## Purpose

**Short**: Manages the lifecycle of RayCluster, RayJob, and RayService custom resources and their associated Kubernetes infrastructure for distributed Ray computing on OpenShift AI.

**Detailed**: The KubeRay operator is a controller-runtime based Kubernetes operator that provides full lifecycle management for Ray distributed computing clusters on RHOAI. It defines three primary CRDs (RayCluster, RayJob, RayService) in the `ray.io` API group and implements six controllers: RayCluster (pod, service, and ingress management), RayJob (job submission and lifecycle), RayService (zero-downtime serving upgrades), Authentication (Gateway API HTTPRoute and OIDC proxy), mTLS (cert-manager certificate management), and NetworkPolicy (annotation-driven network isolation).

The operator integrates deeply with the RHOAI platform through Gateway API HTTPRoutes for dashboard access, kube-rbac-proxy sidecars for authentication, cert-manager for mTLS certificate provisioning, and OpenShift SecurityContextConstraints for pod security. It supports optional batch scheduling integration with Volcano, YuniKorn, and Kubernetes scheduler-plugins. The operator detects OpenShift environments via the Route API and adapts its behavior accordingly, including namespace detection through APPLICATION_NAMESPACE and POD_NAMESPACE environment variables set by the ODH/RHOAI platform operator.

## Architectural Analysis

The KubeRay operator follows a modular controller architecture with six independently reconciling controllers that coordinate through Kubernetes status conditions and shared annotations. The primary RayCluster controller manages the pod lifecycle while delegating security concerns to three specialized controllers: Authentication, mTLS, and NetworkPolicy. This separation of concerns enables the security features to be independently enabled and disabled via annotations without modifying the core cluster management logic.

The RHOAI-specific authentication architecture uses a cross-namespace pattern where HTTPRoutes are created in the platform namespace (typically `redhat-ods-applications` or `opendatahub`) while the target Services reside in user namespaces. This cross-namespace reference is secured by a shared ReferenceGrant resource that uses reference counting for cleanup - it is only deleted when the last authenticated RayCluster in a namespace is removed. The kube-rbac-proxy sidecar performs SubjectAccessReview checks, verifying that users have `get` permission on the head Service before proxying requests to the dashboard.

The mTLS implementation uses a cert-manager certificate chain (self-signed issuer -> CA certificate -> CA issuer -> workload certificates) with automatic SAN updates when pod IPs change. Head certificates include service DNS names with full FQDN variants, while worker certificates use wildcard patterns for dynamic pod matching. Certificate secrets are mounted read-only into pods with Ray-specific environment variables (RAY_USE_TLS, RAY_TLS_SERVER_CERT, etc.) to enable transparent TLS for all Ray internal communication.

The RayService controller implements a sophisticated zero-downtime upgrade pattern using active/pending cluster pairs. When the spec changes, a new "pending" RayCluster is created and configured with the new Serve deployment configuration. Only after the pending cluster's Serve endpoints are healthy does the controller switch the Service selectors to point to the new cluster and clean up the old active cluster. This is supported by an in-memory LRU cache that prevents redundant Serve configuration updates.

The operator's NetworkPolicy controller creates defense-in-depth network isolation with fine-grained ingress rules: intra-cluster pods get unrestricted access, in-namespace pods can reach the client and dashboard ports, the operator namespace gets management access, the monitoring namespace gets metrics scrape access, and the gateway namespace gets dashboard access. Port 8443 (kube-rbac-proxy) is open to all sources but requires OIDC authentication, effectively creating a public endpoint protected by authentication rather than network policy.
