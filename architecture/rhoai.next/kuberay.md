## Purpose

**Short**: Kubernetes operator that manages the lifecycle of Ray clusters, Ray jobs, and Ray Serve deployments on OpenShift and Kubernetes.

**Detailed**: The KubeRay operator provides a Kubernetes-native way to deploy and manage Ray clusters through three Custom Resource Definitions: RayCluster, RayJob, and RayService. It reconciles these CRDs into the underlying Kubernetes resources — pods, services, ingress rules, RBAC roles, and network policies — needed to run distributed Ray workloads.

For RHOAI, the operator includes significant OpenShift-specific enhancements: automatic detection of OpenShift clusters via Route API discovery, integration with Gateway API (HTTPRoute + ReferenceGrant) for external dashboard access, kube-rbac-proxy sidecar injection for OIDC-based authentication, annotation-driven network policy enforcement, and mutual TLS certificate management via cert-manager. A mutating webhook enforces secure defaults on OpenShift by setting `odh.ray.io/secure-trusted-network=true` and disabling direct Ingress/Route creation.

The operator supports batch scheduling integration with Volcano, Yunikorn, and scheduler-plugins for gang scheduling. It manages GCS fault tolerance with external Redis, provides zero-downtime upgrades for RayService via pending-cluster promotion, and emits Prometheus metrics for RayCluster, RayJob, and RayService resources.

## Architectural Analysis

The KubeRay operator follows a multi-controller architecture where six controllers watch the same primary CRD (RayCluster) but manage orthogonal concerns. The RayCluster controller handles pod lifecycle and service networking; the Authentication controller manages OIDC/OAuth sidecar injection and Gateway API resources; the NetworkPolicy controller enforces network segmentation; and the mTLS controller manages cert-manager certificate chains. This separation of concerns allows each security feature to be enabled independently via annotations (e.g., `odh.ray.io/secure-trusted-network`) rather than requiring monolithic configuration.

The authentication architecture demonstrates a thoughtful security-first design. On OpenShift, the mutating webhook enforces `odh.ray.io/secure-trusted-network=true` by default and disables direct Ingress/Route creation. The Authentication controller then creates HTTPRoutes in the platform namespace (not the user namespace) pointing at the head service via cross-namespace ReferenceGrants. This pattern ensures that dashboard access routes through the platform's central Gateway (data-science-gateway) with kube-rbac-proxy enforcing SubjectAccessReview authorization before proxying to the dashboard. The cleanup order is deliberately security-first: ReferenceGrants are deleted before HTTPRoutes to revoke cross-namespace trust before removing routing rules.

The mTLS implementation uses a cert-manager certificate chain (SelfSigned Issuer → CA Certificate → CA Issuer → leaf certificates) that provides per-cluster isolation without requiring a cluster-wide CA. The controller dynamically updates IP SANs as pods scale, and the RayCluster controller coordinates by waiting for certificate secrets to be ready before creating pods and injecting an init container that verifies the pod's IP appears in the certificate SANs. This tight coupling between the mTLS and RayCluster controllers is managed through shared annotation checks and condition-based signaling.

The RayService zero-downtime upgrade strategy creates a pending RayCluster alongside the active one, waits for all serve applications to reach RUNNING status on the pending cluster, then atomically switches the serve service selector to the new cluster. The serve config is cached per-cluster in an LRU cache to avoid redundant dashboard API calls. This design trades cluster resource duplication during upgrades for guaranteed availability.

A notable design choice is the operator's extensive use of environment variables for feature flags (`ENABLE_INIT_CONTAINER_INJECTION`, `ENABLE_GCS_FT_REDIS_CLEANUP`, `ENABLE_ZERO_DOWNTIME`, etc.) alongside Kubernetes feature gates (`RayClusterStatusConditions`, `RayJobDeletionPolicy`). The environment variables predate the feature gate system and are gradually being deprecated in favor of the structured feature gate mechanism, but both coexist in the current codebase.
