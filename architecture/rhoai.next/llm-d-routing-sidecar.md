## Purpose

**Short**: A reverse proxy sidecar that orchestrates prefill/decode disaggregated inference by routing incoming LLM requests to a remote prefiller before forwarding to the local decoder (vLLM).

**Detailed**: The llm-d-routing-sidecar implements the routing logic for prefill/decode (P/D) disaggregated LLM inference in the llm-d architecture. It runs as a sidecar container alongside a vLLM decoder instance, intercepting OpenAI-compatible completion requests. When a request includes an `x-prefiller-host-port` header (set by the upstream inference scheduler/EPP), the sidecar first forwards the request to the specified remote prefiller pod to perform KV-cache prefilling, then forwards the response metadata (including KV transfer parameters) to the local decoder to complete the generation.

The sidecar supports multiple P/D connector protocols: NIXLv2 (current/recommended), NIXLv1 (deprecated), and LMCache (deprecated). Each protocol implements a different mechanism for coordinating KV-cache transfer between prefiller and decoder. The sidecar also includes optional SSRF protection that watches Kubernetes `InferencePool` resources (Gateway API Inference Extension CRD) to maintain a dynamic allowlist of valid prefill targets based on pod IPs matching the pool's label selectors. This is an experimental component noted for removal in an upcoming iteration of the llm-d architecture.

## Architectural Analysis

The llm-d-routing-sidecar follows a classic sidecar proxy pattern, running alongside a vLLM decoder in the same pod. Its primary role is to implement the two-phase disaggregated inference protocol where prefilling (KV-cache population) happens on a remote worker and decoding (token generation) happens locally. This separation allows GPU-optimized scheduling where prefill and decode workloads can be assigned to different GPU types or pools.

The proxy architecture is built around Go's `net/http/httputil.ReverseProxy`. The decoder proxy is a single, statically-configured reverse proxy to localhost. Prefiller proxies are created dynamically and cached in an LRU cache (capacity 16) keyed by host:port. This design assumes a moderate number of distinct prefiller targets per decoder; the LRU eviction is acceptable because creating a new reverse proxy is cheap. For TLS connections to prefillers and the decoder, the sidecar supports optional `InsecureSkipVerify` flags, which is typical for development/internal traffic where certificate management is challenging.

The NIXL v2 protocol (the current recommended connector) works by: (1) cloning the incoming request and modifying it to set `max_tokens=1`, `stream=false`, and add `kv_transfer_params` with `do_remote_decode=true`, then sending it to the prefiller; (2) extracting the `kv_transfer_params` from the prefiller response (which now contains `remote_block_ids`, `remote_engine_id`, `remote_host`, `remote_port`); (3) creating a new request to the local decoder with the original parameters restored and the prefiller's KV transfer metadata attached. A UUID is generated per request and propagated via the `x-request-id` header to correlate prefill and decode phases. The NIXL v1 and LMCache protocols are deprecated and follow simpler variations of this pattern.

The SSRF protection feature is architecturally notable. It uses the Kubernetes dynamic client with `SharedInformer` patterns (not controller-runtime) to watch a specific `InferencePool` resource and then create secondary pod informers based on the pool's label selectors. The allowlist is rebuilt entirely on every pod add/update/delete event, which is safe for small-to-medium pools but could be a concern at scale. The allowlist tracks host IPs only (not ports), meaning any port on an allowed host is permitted — this is documented as intentional. The validator integrates at the request path level, checking the `x-prefiller-host-port` header value before the connector protocol runs.

The deployment manifests use a kustomize-with-envsubst pattern (not standard kustomize variable substitution), requiring `envsubst` to process `${PROJECT_NAME}`, `${NAMESPACE}`, etc. at deploy time. The StatefulSet deployment model with `serviceName` linking to a Service is somewhat unusual for a sidecar — in production, this component would typically be injected as an additional container in the vLLM decoder pod rather than deployed as a standalone StatefulSet. The standalone deployment manifests appear to be primarily for development and testing purposes.
