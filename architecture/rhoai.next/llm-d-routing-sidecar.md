## Purpose

**Short**: A reverse proxy sidecar that routes LLM inference requests between prefill and decode workers in a disaggregated prefill/decode (P/D) architecture.

**Detailed**: The llm-d-routing-sidecar is a lightweight HTTP/HTTPS reverse proxy designed to run as a sidecar container alongside vLLM decoder pods in the llm-d disaggregated inference architecture. Its primary function is to intercept incoming completion requests, identify the designated prefill worker from the `x-prefiller-host-port` request header, orchestrate the two-phase prefill-then-decode protocol, and return the final response to the client.

The sidecar supports three P/D connector protocols: NIXL v2 (current/default), NIXL v1 (deprecated), and LMCache (deprecated). For NIXL v2, it forwards the request to the prefill worker with `kv_transfer_params` containing the KV-cache transfer metadata, buffers the prefill response, then forwards the original request plus KV transfer parameters to the co-located decoder. For requests without the prefill header, the sidecar acts as a transparent pass-through proxy to the local decoder.

The sidecar includes an optional SSRF protection mechanism that watches a Gateway API `InferencePool` custom resource and builds a dynamic allowlist of permitted prefill targets from the pod IPs matching the pool's label selector. This prevents request forgery attacks where a malicious `x-prefiller-host-port` header could redirect traffic to unauthorized destinations. The component is explicitly marked as experimental and will be removed in a future iteration of the llm-d architecture.

## Architectural Analysis

The llm-d-routing-sidecar follows a classic sidecar proxy pattern -- it intercepts traffic at the pod network level and implements application-aware routing logic that would be impractical in a generic service mesh proxy. The key architectural decision is to run as a transparent reverse proxy that only intervenes on specific completion endpoints (`/v1/chat/completions` and `/v1/completions`), passing all other traffic through to the co-located decoder unchanged. This minimizes the sidecar's blast radius and keeps it focused on the P/D disaggregation concern.

The three connector protocol implementations (NIXL v2, NIXL v1, LMCache) reveal an evolving P/D protocol landscape. The NIXL v2 protocol encapsulates KV transfer parameters into a single `kv_transfer_params` JSON field, while v1 spreads them across top-level fields (`remote_block_ids`, `remote_engine_id`, `remote_host`, `remote_port`). The LMCache protocol is simpler -- it just sends the original request to the prefiller with `max_tokens=1` to trigger KV cache population, then sends the original request to the decoder. Both v1 and LMCache are deprecated, suggesting the architecture is consolidating on the v2 protocol structure. The prefill response is always buffered in memory (via `bufferedResponseWriter`), which means large prefill responses are held entirely in RAM -- a potential concern for very large context windows.

The SSRF protection subsystem is architecturally interesting because it uses Kubernetes informers (via the dynamic client) to maintain a real-time allowlist rather than performing API calls per-request. The `AllowlistValidator` watches a specific `InferencePool` CR, extracts its label selector, then starts a secondary pod informer filtered by those labels. Pod IPs and names are extracted into an in-memory set that is rebuilt atomically on every pod add/update/delete event. The 30-second resync period provides a safety net for missed watch events. One limitation: the allowlist only validates hosts/IPs (not ports), so any port on an allowed host is permitted. The allowlist is disabled by default for backward compatibility.

The TLS configuration is thorough -- the proxy server, decoder client, and prefiller client all use explicitly configured cipher suites (AES-GCM and CHACHA20-POLY1305 with ECDHE key exchange) with a TLS 1.2 minimum version. The self-signed certificate fallback (4096-bit RSA, 10-year validity) is clearly marked as a testing convenience. However, the `InsecureSkipVerify` options for both prefiller and decoder connections are a security concern in production -- they should be paired with proper CA configuration rather than disabled verification. The Konflux Dockerfile correctly uses `CGO_ENABLED=1` for FIPS compliance, but the absence of `GOEXPERIMENT=strictfipsruntime` means FIPS enforcement depends on the UBI9 system-level OpenSSL rather than Go's built-in FIPS module.

The deployment manifests use a kustomize-with-envsubst pattern rather than pure kustomize parameterization. Variables like `${PROJECT_NAME}`, `${NAMESPACE}`, and `${IMAGE_TAG_BASE}` are substituted at deployment time, which is less idiomatic than kustomize's built-in replacement mechanisms but provides flexibility for CI/CD pipelines. The StatefulSet deployment (rather than Deployment) is notable -- this is likely because the sidecar is co-deployed with a vLLM pod that requires stable network identity for NIXL KV-cache transfers between prefill and decode workers. The component is explicitly marked as experimental and scheduled for removal in a future llm-d iteration, suggesting the P/D routing logic may be absorbed into the EPP or vLLM itself.
