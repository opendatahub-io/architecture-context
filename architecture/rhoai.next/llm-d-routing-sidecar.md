## Purpose

**Short**: A reverse proxy sidecar that enables disaggregated prefill/decode (P/D) inference by routing incoming LLM requests to a remote prefiller before forwarding KV cache transfer parameters to the local decoder.

**Detailed**: The llm-d-routing-sidecar is a Go-based HTTP reverse proxy that runs as a sidecar container alongside a vLLM decoder instance in a Kubernetes pod. Its primary function is to enable disaggregated prefill/decode (P/D) inference — an architecture where the computationally expensive prefill stage of LLM inference is offloaded to a separate GPU worker (the "prefiller"), and only the KV cache transfer parameters are forwarded to the local decoder for token generation.

When a client sends a chat/completions request with an `x-prefiller-host-port` header, the sidecar intercepts the request, forwards it to the specified prefiller pod, extracts KV cache transfer metadata from the prefiller's response, and then sends a modified request to the local decoder with the KV transfer parameters attached. This two-phase proxy pattern supports three connector protocols: NIXLv2 (current/recommended), NIXLv1 (deprecated), and LMCache (deprecated).

The sidecar includes SSRF (Server-Side Request Forgery) protection that can optionally validate prefill targets against an allowlist derived from Kubernetes InferencePool resources. When enabled, the sidecar watches InferencePool CRs and their matching pods via Kubernetes dynamic informers, building a real-time allowlist of permitted prefill target IPs and hostnames. The sidecar supports TLS for both its own serving endpoint (default on, with self-signed cert fallback) and for connections to prefiller and decoder endpoints. This project is experimental and is expected to be removed in an upcoming iteration of the llm-d P/D disaggregation architecture.

## Architectural Analysis

The llm-d-routing-sidecar implements a two-phase reverse proxy pattern specifically designed for disaggregated prefill/decode LLM inference. The core architectural insight is that prefill (computing attention over the prompt) and decode (generating output tokens) can run on different GPU workers, with KV cache data transferred via NIXL (a high-performance GPU memory transfer library). The sidecar transparently orchestrates this split by intercepting OpenAI-compatible API requests, coordinating the prefill stage on a remote worker, extracting the `kv_transfer_params` from the prefiller response, and injecting them into the decode request sent to the local vLLM instance. This is functionally invisible to the client — the client sends a standard completions request with an extra header and receives a standard response.

The proxy supports three connector protocols, though two are deprecated. The NIXLv2 protocol (default) packages KV transfer metadata into a single `kv_transfer_params` object, while the older NIXLv1 protocol used individual fields (`remote_block_ids`, `remote_engine_id`, `remote_host`, `remote_port`). The LMCache protocol is simpler — it just forwards the request to the prefiller with `max_tokens=1` to warm the cache, then sends the original unmodified request to the decoder. The LRU cache of prefiller proxy handlers (capacity 16) is a pragmatic optimization that avoids creating a new `httputil.ReverseProxy` for every request to the same prefiller endpoint.

The SSRF protection subsystem is architecturally notable: rather than using a static allowlist, it dynamically builds one by watching Kubernetes InferencePool resources and their matching pods. This is significant because in a P/D disaggregation setup, the `x-prefiller-host-port` header comes from the external routing layer (e.g., the Gateway API Inference Extension EPP), and an attacker who controls this header could redirect prefill requests to arbitrary internal endpoints. The AllowlistValidator uses Kubernetes dynamic informers with a 30-second resync period, tracking pod IPs and hostnames. Notably, it validates only the host portion of the target (not ports), so any port on an allowed host is permitted.

The TLS configuration is well-considered for a sidecar: the proxy defaults to secure mode (`-secure-proxy=true`) with a self-signed certificate fallback for development, but supports external certificate provisioning via `-cert-path`. All three TLS contexts (server, decoder client, prefiller client) use identical FIPS-compatible cipher suites with TLS 1.2 minimum. The `InsecureSkipVerify` flags for prefiller and decoder connections are opt-in (disabled by default), which is appropriate for environments where internal service mesh TLS may use self-signed certificates. The absence of `GOEXPERIMENT=strictfipsruntime` in the Konflux build is a gap — while CGO_ENABLED=1 with UBI9 provides the OpenSSL backend, the strictfipsruntime flag would enforce FIPS-only crypto at the Go runtime level.

The project is explicitly marked as experimental and slated for removal in a future llm-d iteration. This likely means the P/D routing logic will be absorbed into the Gateway API Inference Extension EPP (Endpoint Picker) or into vLLM itself, eliminating the need for a separate sidecar. The deployment manifests use kustomize with envsubst-based parameterization rather than the more standard kustomize `vars` or `replacements` mechanisms, which is somewhat unusual but functional.
