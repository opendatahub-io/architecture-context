## Purpose

**Short**: A reverse proxy sidecar that routes LLM inference requests between prefill and decode workers in an llm-d prefill/decode (P/D) disaggregated serving architecture.

**Detailed**: The llm-d-routing-sidecar is a Go-based HTTP/HTTPS reverse proxy that runs as a sidecar container alongside a vLLM decoder instance. It intercepts OpenAI-compatible chat/completion API requests and orchestrates the prefill/decode disaggregation protocol. When a request arrives with an `x-prefiller-host-port` header, the sidecar first forwards the request to the specified remote prefill worker, collects KV cache transfer parameters from the prefill response, then forwards the enriched request to the local decoder. This two-phase protocol enables KV cache transfer between prefill and decode workers using NIXL (v1 and v2) or LMCache connectors.

The sidecar includes SSRF (Server-Side Request Forgery) protection that can be enabled via feature flag. When enabled, it watches Kubernetes `InferencePool` resources (from the Gateway API Inference Extension) and maintains a dynamic allowlist of valid prefill targets based on pod IPs matching the InferencePool's label selector. Requests to unauthorized prefill targets are rejected with HTTP 403.

This component is marked as experimental and is expected to be removed in an upcoming iteration of the llm-d P/D disaggregation architecture.

## Architectural Analysis

The llm-d-routing-sidecar implements a sidecar pattern for LLM inference disaggregation, separating the prefill and decode phases of transformer inference across different GPU workers. The architecture is intentionally simple: a single Go binary acting as an HTTP reverse proxy with protocol-aware request manipulation. The sidecar intercepts OpenAI-compatible API calls, splits them into a prefill phase (forwarded to a remote worker) and a decode phase (forwarded to the co-located vLLM instance), and stitches the results together by passing KV cache transfer parameters between the two phases.

The connector abstraction (`protocolRunner` function type) cleanly separates the three supported P/D protocols: NIXL v1 (deprecated), NIXL v2 (current), and LMCache (deprecated). Each connector implements the same two-phase flow but with different request/response field manipulation. NIXL v2 uses a `kv_transfer_params` envelope with fields for remote engine ID, block IDs, host, and port. NIXL v1 passes these fields at the top level of the request body. LMCache uses a simpler approach where the prefill is triggered by setting `max_tokens=1`, relying on shared cache for KV reuse. The LRU cache for prefiller proxy handlers (`capacity=16`) indicates the sidecar is expected to interact with a relatively small number of distinct prefill workers.

The SSRF protection mechanism is architecturally notable. Rather than relying on network policies alone, the sidecar uses the Kubernetes dynamic client to watch `InferencePool` resources (from the Gateway API Inference Extension, `inference.networking.x-k8s.io/v1alpha2`) and their matching pods. It builds an in-memory allowlist of pod IPs and names, updated in real-time via informer watch events with a 30-second resync period. This is a defense-in-depth measure: the `x-prefiller-host-port` header is attacker-controllable (it comes from the upstream routing layer), so without SSRF protection, a compromised or malicious router could redirect prefill traffic to arbitrary internal services. The allowlist validation checks host only (not port), which is a documented limitation.

TLS is handled with care throughout: the sidecar proxy itself defaults to TLS-enabled (`-secure-proxy=true`) with either externally provided certificates or auto-generated self-signed certs. The cipher suite configuration is explicitly hardened to ECDHE-only key exchange with AES-GCM or ChaCha20-Poly1305 symmetric ciphers, and `MinVersion` is set to TLS 1.2. However, the Konflux Dockerfile does not include `GOEXPERIMENT=strictfipsruntime`, so the binary may not use FIPS-validated cryptographic implementations even though `CGO_ENABLED=1` enables dynamic linking. The availability of `InsecureSkipVerify` flags for both prefiller and decoder connections is a pragmatic choice for development/testing but should be documented as a security consideration for production deployments.

The component is explicitly marked as experimental and expected to be superseded. This is consistent with the broader llm-d architecture evolution, where P/D disaggregation routing logic is likely to be absorbed into the Gateway API Inference Extension (EPP) or the vLLM serving layer itself.
