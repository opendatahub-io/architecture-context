## Purpose

**Short**: A reverse proxy sidecar that intercepts LLM inference requests and routes them through a disaggregated prefill/decode (P/D) pipeline, forwarding prefill work to a designated prefill worker before sending the decode request to the co-located vLLM decoder.

**Detailed**: The llm-d-routing-sidecar is a lightweight Go-based HTTP/HTTPS reverse proxy designed to run as a sidecar container alongside a vLLM decoder instance in a Kubernetes pod. Its primary role is to enable disaggregated prefill/decode (P/D) inference, where the computationally expensive prefill phase of LLM inference is offloaded to a separate worker pod, and only the decode phase runs on the local GPU.

When a request arrives with the `x-prefiller-host-port` header, the sidecar intercepts the OpenAI-compatible completion request, forwards it to the specified remote prefill worker with protocol-specific modifications (NIXL v2, deprecated NIXL v1, or deprecated LMCache), captures the prefill response containing KV transfer parameters, then constructs and forwards a decode request to the co-located vLLM instance on localhost. Requests without the prefill header are passed through directly to the local decoder. The sidecar supports TLS for both its listening endpoint (with self-signed or mounted certificates) and for upstream connections to prefillers and decoders.

The sidecar includes optional SSRF protection that watches a Kubernetes InferencePool custom resource to maintain a dynamic allowlist of valid prefill target hosts based on pod IPs matching the pool's selector. This prevents arbitrary server-side request forgery by ensuring only pods belonging to the designated InferencePool can be used as prefill targets. The project is marked as experimental and is planned for removal in a future iteration of the llm-d architecture.

## Architectural Analysis

The llm-d-routing-sidecar implements a remarkably focused sidecar pattern: its sole purpose is to orchestrate the two-phase prefill/decode pipeline for disaggregated LLM inference. The architecture is intentionally simple - a single Go binary with no operator framework, no CRDs, and no persistent state. The sidecar runs alongside a vLLM decoder in the same pod and acts as a transparent reverse proxy that only activates its P/D logic when the `x-prefiller-host-port` header is present in incoming requests.

The protocol implementation reveals a clear evolutionary path. Three connector protocols are supported - NIXL v1 (deprecated), NIXL v2 (current), and LMCache (deprecated). The NIXL v2 protocol (`connector_nixlv2.go`) wraps the original request with `kv_transfer_params` containing remote decode/prefill flags, engine IDs, block IDs, and host/port information. It sends a modified prefill request with `max_tokens=1` and streaming disabled to the remote prefiller, extracts the KV transfer parameters from the response, then constructs a decode request with those parameters for the local decoder. This two-hop design enables KV cache transfer between prefiller and decoder via the NIXL transport layer (outside the HTTP path).

The SSRF protection mechanism in `allowlist.go` is architecturally interesting. Rather than using a static configuration, it leverages Kubernetes informers to dynamically watch an InferencePool custom resource (from the Gateway API Inference Extension project at `inference.networking.x-k8s.io/v1alpha2`). When a pool's selector changes or pods are added/removed, the allowlist is rebuilt from pod IPs and names. This approach tracks only host identifiers (not ports), so any port on an allowed host is permitted. The implementation uses the Kubernetes dynamic client rather than typed clients, which avoids compile-time dependency on the InferencePool API types but sacrifices type safety.

The TLS configuration is well-considered for a sidecar. The proxy listener defaults to TLS enabled (`-secure-proxy=true`) with TLS 1.2 minimum and a strong cipher suite selection (ECDHE with AES-GCM and ChaCha20-Poly1305). When no certificate path is provided, it generates a self-signed RSA-4096 certificate at startup - practical for development but requiring explicit certificate mounting in production. The upstream connections to prefillers and the decoder support independent TLS configuration with separate `InsecureSkipVerify` flags, acknowledging that TLS verification may need to be relaxed in certain cluster networking configurations.

The project is explicitly marked as experimental with planned removal in a future llm-d iteration. This likely reflects an architectural evolution toward integrating routing logic directly into the gateway or inference infrastructure rather than maintaining a per-pod sidecar. The LRU cache for prefiller proxy handlers (capacity 16 in `proxy.go:113`) suggests the sidecar may see requests to many different prefillers, but the cache size is quite small, which could cause performance degradation with large prefill pools due to repeated proxy handler creation.
