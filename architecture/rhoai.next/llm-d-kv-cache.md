## Purpose

**Short**: Pluggable KV-cache management suite that enables KV-cache-aware routing and cross-node cache coordination for vLLM-based LLM inference platforms.

**Detailed**: The llm-d-kv-cache repository provides the core infrastructure for tracking and scoring KV-cache block residency across a fleet of model-serving pods (vLLM, SGLang). The primary component is a Go library — the **KV-Cache Indexer** — that maintains a near-real-time, globally consistent index of which KV-cache blocks reside on which pods and storage tiers (GPU, CPU). It ingests `KVEvents` streamed from model servers over ZeroMQ, and exposes a fast scoring API that ranks pods by how much of a request's prefix they already have cached. This library is designed to be embedded in a host process (the llm-d EPP is the reference consumer).

The repository also ships a **UDS Tokenizer** sidecar service (Python/gRPC) that performs tokenization and chat template rendering for the indexer, and two **KV connectors**: the `llmd-fs-backend` (a Python vLLM plugin for offloading KV-cache blocks between GPU and shared storage via POSIX/GDS/NIXL) and the `pvc-evictor` (a Python utility that manages PVC disk space by evicting stale KV-cache files). Together, these components form the KV-cache management layer of the llm-d inference platform used by Red Hat OpenShift AI.

## Architectural Analysis

The llm-d-kv-cache repository implements a sophisticated dual-key indexing design that decouples engine-internal block hashes from the indexer's own request-key hashes. This is architecturally significant because it means the indexer can score prompts without round-tripping to model servers — it reproduces the content-addressing scheme (chained FNV-64a over CBOR) locally. The bridge between engine keys and request keys is built on ingestion via `BlockStored` events (which carry token chunks), and used on eviction via `BlockRemoved` events (which carry only engine keys). This design enables the indexer to be completely decoupled from the inference engine's internal state while maintaining correct cache hit scoring.

The event processing pipeline uses FNV-1a hash-based sharding to guarantee per-pod event ordering without global locking. This is critical for correctness: KV-cache events must be applied in order per-pod (a BlockRemoved for key X must not be processed before the corresponding BlockStored), but events from different pods can be processed in parallel. The pool supports two delivery modes — centralized (single ZMQ endpoint) and pod-discovery (per-pod subscribers via Kubernetes reconciler) — enabling both simple single-replica and production multi-replica deployments.

The block index is pluggable with three backends: in-memory LRU (default, using `hashicorp/golang-lru`), cost-aware memory (using Ristretto with admission control), and Redis/Valkey. The in-memory backends are sized by entry count or byte budget, while Redis provides persistence and shared state. Notably, the codebase recommends in-memory for most deployments because each replica in pod-discovery mode independently converges from the event stream, making shared state unnecessary. The Valkey backend also supports experimental RDMA transport for ultra-low-latency lookups.

The UDS tokenizer sidecar pattern is a key architectural choice — tokenization is offloaded to a Python process running vLLM's rendering pipeline, communicating over a Unix Domain Socket via gRPC. This avoids embedding Python tokenization in the Go process while keeping the latency of a local socket. The shared `emptyDir` volume mount at `/tmp/tokenizer` provides the IPC channel. The tokenizer supports multimodal inputs (images with content hashes), LoRA adapters, and thinking tokens, making it compatible with the full range of vLLM serving modes. Internal in-process tokenization via the `tokenization.Pool` is deprecated in favor of this external approach.

The fs-backend connector demonstrates a layered storage architecture with POSIX filesystem, GPU Direct Storage (GDS), and NIXL object storage backends. The file organization uses a hierarchical hash-partitioned directory structure (3-hex/2-hex prefix directories) to avoid filesystem inode pressure, and supports content-addressable deduplication via block hash filenames. The PVC evictor complements this with an N+2 multi-process design (N crawlers + activator + deleter) using hysteresis-based disk management. The activator uses `statvfs()` for O(1) disk usage monitoring, and the crawlers use hex-modulo sharding for parallel file discovery — a pattern that mirrors the event pool's sharding approach.
