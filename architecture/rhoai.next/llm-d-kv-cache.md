## Purpose

**Short**: KV-Cache indexing library and supporting services that enable KV-cache-aware routing for distributed LLM inference on the llm-d platform.

**Detailed**: The llm-d-kv-cache repository provides a comprehensive suite of KV-Cache management capabilities for distributed LLM inference. The core component is a Go library (`pkg/kvcache`) that maintains a near-real-time, globally consistent view of KV-Cache block residency across a fleet of vLLM or SGLang inference engine pods. It subscribes to KV-Events emitted by model servers over ZeroMQ and exposes a fast scoring API that ranks pods by how much of a request's prefix they already have cached, enabling KV-cache-aware routing decisions.

The library is designed to be embedded in a host process — llm-d's Envoy Processing Policy (EPP) is the reference consumer. It is not a standalone service. Alongside the core library, the repository contains several supporting components: a gRPC tokenization sidecar service (`uds_tokenizer`) that handles prompt tokenization and chat template rendering via Unix Domain Socket, a Python storage backend (`llmd_fs_backend`) that offloads KV-cache blocks between GPU memory and shared storage (POSIX filesystem or S3 object store via NIXL), and a PVC evictor utility (`pvc_evictor`) that manages disk space for the offloading connector's KV-cache storage on PersistentVolumeClaims.

The system supports multiple index backends (in-memory LRU, cost-aware memory via Ristretto, Redis/Valkey), multimodal content hashing, LoRA adapter differentiation, hybrid model attention (HMA) group tracking, and speculative cache entries. It integrates with Prometheus for metrics, OpenTelemetry for distributed tracing, and Kubernetes for pod discovery.

## Architectural Analysis

The llm-d-kv-cache repository follows a library-first architecture pattern. The core KV-cache indexer is designed as an embeddable Go library rather than a standalone service, which eliminates a network hop on the critical scoring path. The EPP process embeds the indexer directly, gaining sub-millisecond scoring latency. This is architecturally significant because scoring happens on every incoming inference request, and even small latency additions would compound across the fleet.

The dual-key design (engine keys vs. request keys) is the most architecturally notable pattern. Engine keys are the internal content-address hashes that vLLM/SGLang use for their KV-cache blocks; request keys are the indexer's own hashes computed from token sequences. The two hash spaces are independent — the indexer bridges them on ingestion by recording `engineKey → requestKey` mappings. This decoupling means the indexer doesn't need to know engine internals at scoring time: it just computes request keys from prompt tokens and looks them up. The complexity cost is a secondary data structure that must be maintained and cleaned up on eviction.

The event processing pipeline demonstrates careful attention to ordering guarantees. The sharded worker pool uses FNV-1a hashing on pod identifiers to route all events from the same pod to the same worker queue. This preserves per-pod event ordering — critical because a `BlockStored` with tokens must be processed before a subsequent `BlockStored` without tokens (CPU offload) that references the same engine key. The design trades some parallelism (events from the same pod are serialized) for correctness.

The tokenization subsystem has undergone a significant architectural evolution. The original in-process tokenization pool (using CGO bindings to HuggingFace tokenizers) has been deprecated in favor of an external gRPC sidecar (`uds_tokenizer`). This separation allows the tokenizer to use Python with vLLM's full rendering pipeline (including chat templates and multimodal processing) while the core indexer remains pure Go. The UDS socket transport minimizes the latency cost of this separation. The `ScoreTokens` API (accepting pre-tokenized input) is now the primary interface, with the prompt-string APIs scheduled for removal.

The storage backend (`llmd_fs_backend`) has been upstreamed into vLLM v0.23 as the FS tier of TieringOffloadingSpec. The version in this repository (v0.23) is the final standalone release. This is architecturally relevant because future KV-cache offloading development will happen in the vLLM repository rather than here. The NIXL OBJ backend for S3 object storage is a notable addition that extends offloading beyond local/NFS storage.

The PVC evictor uses a multi-process architecture (N+2 processes) to achieve high eviction throughput. The hysteresis control (cleanup_threshold → target_threshold) prevents thrashing. The hex-modulo load balancing across crawler processes ensures even distribution without coordination. The use of `xargs rm` for batch deletion is pragmatic — it avoids Python's per-file overhead for deletion operations.

A notable gap is the lack of TLS/authentication on any network-facing interface. The ZMQ sockets, gRPC IndexerService, and health probe all operate without encryption or authentication. This is acceptable for pod-local UDS communication and cluster-internal services behind network policies, but the Redis/Valkey connection and OTLP export should use TLS in production. The codebase contains a comment noting this for OTLP (`// In production, use TLS`).
