## Purpose

**Short**: Pluggable KV-Cache indexing library and tooling for cache-aware routing and cross-node cache coordination in vLLM/SGLang-based LLM inference platforms.

**Detailed**: The llm-d-kv-cache project provides a high-performance KV-Cache Indexer library (Go) that maintains a near-real-time global view of KV-Cache block locality across a fleet of vLLM or SGLang inference pods. It ingests KV-Events via ZeroMQ pub/sub from model servers and builds an in-memory (or Redis/Valkey-backed) index mapping block hashes to pod locations. A scoring API allows schedulers to rank candidate pods by how much of a request's prompt prefix is already cached, enabling KV-cache-aware routing that improves Time To First Token (TTFT) and throughput.

The project also includes a UDS Tokenizer sidecar service (Python) that provides gRPC-based tokenization and chat template rendering using vLLM's CPU-mode OpenAIServingRender, a filesystem storage connector (`llmd_fs_backend`) for offloading KV-cache blocks between GPU and shared storage, and a PVC Evictor utility for managing disk space on KV-cache storage volumes. The indexer library is designed to be embedded in a host process — llm-d's Envoy Processing Plugin (EPP) is the reference consumer — rather than deployed as a standalone service.

## Architectural Analysis

The llm-d-kv-cache project follows a **library-first architecture** — the core indexing logic is designed to be embedded in a host process rather than deployed as a standalone service. This is a deliberate design choice that eliminates network hops on the critical scoring path, where latency directly impacts inference request routing. The reference consumer is llm-d's Envoy Processing Plugin (EPP), which calls `Indexer.ScoreTokens()` in-process during request routing decisions.

The **dual-key design** is the most architecturally significant pattern in the codebase. The indexer maintains two independent hash spaces: "engine keys" (content-addressing hashes used by vLLM/SGLang internally) and "request keys" (FNV-64a hashes computed from tokens by the indexer). The bridge between these spaces is built during event ingestion — `BlockStored` events carry both the engine key and the token chunk, allowing the worker to compute the corresponding request key and store the mapping. This decoupling means the scoring path never needs to contact an inference engine — it computes request keys from the prompt tokens and looks them up directly. The tradeoff is an additional mapping (`engineKey → requestKey`) that must be maintained and consulted during eviction.

The **sharded worker pool** in `kvevents.Pool` guarantees per-pod event ordering by using FNV-1a hashing of the pod identifier to deterministically assign messages to worker queues. This prevents race conditions where out-of-order processing could corrupt the index (e.g., an `Evict` arriving before the corresponding `Add`). The default concurrency of 4 workers with rate-limited work queues balances throughput against ordering guarantees.

The **pluggable backend architecture** for `kvblock.Index` supports four implementations (InMemory LRU, CostAwareMemory via Ristretto, Redis, Valkey) with a clear recommendation hierarchy: in-memory backends are preferred for most deployments because each EPP replica independently converges to the same index state from the shared event stream, making external storage unnecessary. The Redis/Valkey backend exists primarily for edge cases requiring persistent or shared index state.

The **tokenization sidecar** (`uds_tokenizer`) is notable for its use of vLLM's `OpenAIServingRender` in CPU mode — it runs vLLM's full prompt rendering pipeline (including chat template application, multimodal processing, and tokenization) without requiring GPU access. Communication via Unix Domain Socket eliminates network overhead and provides natural security isolation (filesystem permissions). The 100MB gRPC message size limit and separate timeouts for standard (5s) vs. multimodal (30s) requests reflect the reality that multimodal prompt rendering involves downloading and processing images.

The **fs_backend connector** bridges a gap between the indexing system and vLLM's OffloadingConnector API. While the indexer tracks _where_ blocks are cached, the fs_backend handles _physically moving_ blocks between GPU memory and shared storage. It supports two storage backends: POSIX filesystem (for NFS/PVC mounts) and object store via NIXL. The C++/CUDA code in `csrc/` handles GPU-to-host memory copies and direct storage I/O with GDS (GPU Direct Storage) support. The PVC Evictor complements this by managing disk space — its N+2 multi-process architecture (N crawlers + 1 activator + 1 deleter) with hex-modulo-based file partitioning enables efficient parallel filesystem scanning.

A notable architectural gap is the **absence of authentication** on the ZMQ event stream between inference pods and the indexer. Events are transmitted in plaintext msgpack over TCP with no authentication or encryption. This is acceptable when network policies restrict pod-to-pod communication within a trusted namespace, but should be documented as a security consideration for multi-tenant deployments.
