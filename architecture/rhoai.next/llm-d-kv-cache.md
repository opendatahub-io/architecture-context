## Purpose

**Short**: A pluggable KV-cache management system that enables cache-aware routing and cross-node cache coordination for distributed LLM inference on vLLM-based serving platforms.

**Detailed**: The llm-d-kv-cache component provides the KV-Cache Indexer, a high-performance library that maintains a global, near-real-time view of KV-Cache block locality across a fleet of vLLM inference engine pods. It ingests KV-Events streamed from vLLM via ZeroMQ, tracking which KV-blocks reside on which pods and on which device tier (GPU or CPU). This metadata enables intelligent, cache-aware routing so that inference schedulers can direct requests to pods that already have the relevant KV-cache blocks, significantly improving Time To First Token (TTFT) and overall throughput.

The component also includes a Python UDS Tokenizer Service (the only Konflux-built container image), a filesystem-based KV-cache offloading backend for shared storage (NFS/PVC), and a PVC evictor for autonomous disk space management. The Go core library is designed to be embedded into the llm-d inference scheduler, while the tokenizer service runs as a sidecar container providing gRPC-based tokenization and chat template rendering.

## Architectural Analysis

The llm-d-kv-cache system is a multi-language, multi-component architecture centered on a Go core library that provides real-time KV-cache locality tracking and pod scoring. The design reflects a clear separation of concerns: the Go packages handle the performance-critical indexing, scoring, and event processing, while Python handles the computationally-intensive but latency-tolerant tokenization and chat template rendering.

**Event-Driven Architecture with Sharded Processing**: The KV events subsystem (`pkg/kvevents/`) implements a particularly well-designed sharded worker pool pattern. Events from each vLLM pod are consistently routed to the same worker queue via FNV-1a hashing of the pod ID, guaranteeing in-order processing per pod while allowing parallel processing across pods. The engine adapter abstraction (`EngineAdapter` interface) cleanly separates the transport layer (ZMQ) from engine-specific parsing, with concrete implementations for both vLLM and SGLang. The ZMQ subscriber uses a 3-frame message format (topic, sequence number, payload) with automatic reconnection and 5-second backoff on connection loss.

**Token-to-Block Hash Compatibility**: A critical architectural constraint is hash compatibility with vLLM's prefix-cache algorithm. The `TokenProcessor` uses CBOR-encoded `[parentHash, tokenChunk, extraFeatures]` tuples hashed with FNV-64a, creating a chained hash where each block's key depends on all preceding blocks. This deterministic scheme requires `PYTHONHASHSEED` to be consistent across vLLM pods and the indexer. The `BlockExtraFeatures` system extends this to multimodal content, where image/video placeholder ranges "taint" block hashes to prevent cache pollution between text-only and multimodal requests for the same token sequences.

**Dual Tokenization Strategy**: The system supports two tokenization modes selected via build tags. The `embedded_tokenizers` build tag enables CGo-based Python embedding where the Go binary directly invokes Python tokenization through a C bridge (`cgo_functions.go` -> `tokenizer_wrapper.py`). Without this tag, only the UDS gRPC tokenizer is available. The `CompositeTokenizer` implements a fallback chain (local file -> UDS gRPC -> HuggingFace download), providing graceful degradation. The UDS tokenizer service itself uses `asyncio.to_thread()` to offload blocking tokenization to a cgroup-aware thread pool, with a 100MB gRPC message size limit to handle large multimodal payloads.

**Storage Offloading Design**: The filesystem backend (`llmd_fs_backend`) integrates with vLLM's `OffloadingSpec` plugin system to provide shared-storage KV-cache persistence. The `FileMapper` uses a deterministic hash-based directory hierarchy (`<model>/<block_config>/<parallel_config>/<rank>/<dtype>/<hex>/<hex>/<hash>.bin`) that distributes files across subdirectories to avoid filesystem metadata bottlenecks. The C++ `storage_offload` extension provides async I/O with NUMA-aware thread placement and optional NVIDIA GPU-Direct Storage (GDS) for zero-copy transfers, which is critical for performance on high-bandwidth storage fabrics.

**N+2 Process Eviction Model**: The PVC evictor implements a sophisticated multi-process architecture with hysteresis control. N crawler processes use hex-based load balancing (`hash % 16` ranges) for deterministic, non-overlapping filesystem traversal. The activator uses `statvfs()` polling (O(1) syscall) with a cleanup/target threshold pair (default 85%/70%) to prevent thrashing. Files are only eligible for deletion if their access time exceeds a configurable threshold (default 60 minutes), protecting recently-accessed "hot" cache entries. Batch deletion via `xargs -0 rm -f` handles special characters safely and provides I/O efficiency.
