## Purpose

**Short**: A pluggable KV-cache indexing and event-processing library that enables KV-cache-aware routing for distributed LLM inference, plus companion tokenizer sidecar and storage management utilities.

**Detailed**: The llm-d-kv-cache repository provides the core KV-cache indexing infrastructure for the llm-d distributed LLM inference platform. The primary component is a Go library (`pkg/`) that maintains a near-real-time, globally consistent view of KV-cache block residency across a fleet of vLLM/SGLang model-server pods. It subscribes to KV-Events emitted by model servers via ZeroMQ and exposes a fast scoring API that ranks pods by how much of a request's prefix they already have cached, enabling intelligent KV-cache-aware routing.

The repository also ships three companion components: (1) a Python gRPC **UDS Tokenizer** sidecar service that handles prompt tokenization and chat template rendering for the inference pipeline, (2) **llmd-fs-backend**, a Python/C++/CUDA vLLM plugin that offloads KV-cache blocks between GPU memory and shared storage (NFS/PVC/object store), and (3) **PVC Evictor**, a Python Kubernetes utility that automatically manages disk space on PVCs used for KV-cache offloading. The Go library is designed to be embedded in a host process (the llm-d EPP/router is the reference consumer); it is not deployed as a standalone service. The UDS Tokenizer is the only component with a Konflux Dockerfile and is shipped as a container image for RHOAI.

## Architectural Analysis

The llm-d-kv-cache repository follows a clear separation between library and service concerns. The core Go library (`pkg/`) is designed as an embeddable module, not a standalone service. This design decision means the library carries no HTTP/gRPC server of its own -- the host process (llm-d-router/EPP) provides the serving infrastructure and simply calls `Indexer.ScoreTokens()` as part of its routing logic. The `examples/kv_cache_index_service/` directory contains a reference gRPC server wrapping the indexer, but this is not the production deployment pattern.

The dual-key architecture (engine keys vs. request keys) is the most architecturally significant design choice. Model servers use content-addressing hashes that differ from the indexer's own FNV-64a-over-CBOR scheme. The bridge is built on ingestion: `BlockStored` events carry token chunks that let the indexer compute its own request keys and record the `engine_key -> request_key` mapping. This decoupling means the indexer's hash space is independent of any engine's internal implementation, but it adds complexity on eviction (where `BlockRemoved` events carry only engine keys) and on CPU offloading (where tokens are not re-sent). The index must maintain both mappings atomically.

The event processing pipeline uses a sharded work queue pattern (FNV-1a hash of pod ID routes messages to specific workers) that guarantees per-pod ordering while allowing parallelism across pods. This is critical because KV-cache events are causally ordered per-pod (a GPU `BlockStored` must precede a CPU offload `BlockStored` for the same block). The `EngineAdapter` interface cleanly abstracts vLLM vs. SGLang wire formats, supporting both array-encoded and map-encoded msgpack payloads.

The three index backends (in-memory LRU, cost-aware Ristretto, Redis/Valkey) use a decorator pattern for instrumentation: `Index -> InstrumentedIndex -> TracedIndex`. This composable wrapping adds Prometheus metrics and OpenTelemetry tracing transparently. The in-memory backend is the production default; Redis/Valkey is available but adds network latency and is rarely necessary since each EPP replica independently converges to the same state from the event stream.

A notable security gap is the complete absence of encryption or authentication on the ZMQ event transport. KV-cache events flow in plaintext between model servers and the indexer. While this is typical for intra-cluster communication in trusted networks, it means that any pod with network access to the ZMQ port can inject or observe cache metadata. The Redis backend also defaults to plaintext, though TLS is available via the `rediss://` URL scheme.

The UDS Tokenizer's use of Unix Domain Sockets rather than TCP for its primary interface is a deliberate performance optimization -- it eliminates network stack overhead when the tokenizer runs as a sidecar in the same pod as the EPP. The health check endpoint remains on TCP (port 8082) for Kubernetes probe compatibility. The llmd-fs-backend has been upstreamed into vLLM as of v0.23, making the version in this repository the final release. Future KV offloading will use vLLM's native multi-tier offloading connector.
