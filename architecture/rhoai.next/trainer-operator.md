## Purpose

**Short**: Module operator that reconciles the Trainer CR to deploy and manage Kubeflow Trainer v2 resources (controller, CRDs, ClusterTrainingRuntimes, ImageStreams) on the RHOAI/ODH platform.

**Detailed**: The trainer-operator is a Kubernetes operator built with controller-runtime that implements the ODH modular architecture pattern. It watches a singleton cluster-scoped `Trainer` CR (`components.platform.opendatahub.io/v1alpha1`) named `default-trainer` and deploys the upstream Kubeflow Trainer v2 controller along with pre-configured ClusterTrainingRuntimes for distributed PyTorch training across CPU, CUDA, and ROCm accelerator variants.

The operator follows a manifest-rendering pipeline: at build time, upstream Kubeflow Trainer manifests are fetched from the `opendatahub-io/trainer` (ODH) or `red-hat-data-services/trainer` (RHOAI) repository via `hack/get_trainer_manifests.sh` and baked into the container image. At runtime, the reconciler copies these templates to a writable work directory, applies image overrides from `RELATED_IMAGE_*` environment variables to `params.env` files, renders kustomize overlays, and applies resources via Server-Side Apply. The operator also manages ClusterTrainingRuntime definitions (torch-distributed and training-hub variants for CUDA 12.8/13.0, ROCm 6.4, and CPU) and OpenShift ImageStreams for Training Hub workbench images.

The operator uses `odh-platform-utilities` for shared platform capabilities including the `PlatformObject` interface, condition management, label-based garbage collection via the discovery API, cluster type detection (OpenShift vs vanilla Kubernetes), and kustomize rendering. It checks for required dependencies (JobSet Operator, JobSet CRD) before provisioning, and supports `Managed`/`Removed` management states with full lifecycle cleanup including finalizer-based garbage collection.

## Architectural Analysis

The trainer-operator follows the ODH modular architecture pattern where platform functionality is decomposed into independent module operators, each managing a single component CR. This design cleanly separates the platform orchestrator (rhods-operator) from component lifecycle management, allowing independent development and versioning of each module. The operator uses `odh-platform-utilities` extensively, which provides standardized patterns for condition management, garbage collection, cluster detection, and kustomize rendering across all module operators.

The manifest-rendering pipeline is a key architectural pattern: upstream Kubeflow Trainer manifests are fetched at build time (not runtime), baked into the container image at `/opt/manifests-template/`, then copied to a writable emptyDir volume (`/opt/manifests-work`) at startup. This approach ensures air-gapped deployability (no network fetch at runtime) while still allowing image overrides via `RELATED_IMAGE_*` environment variables at the OLM/CSV level. The `params.env` file acts as an indirection layer between environment variables and kustomize replacements, enabling the platform operator to pin specific image digests without modifying the upstream manifests.

The dependency-checking pattern is noteworthy: on OpenShift, the operator performs a three-stage dependency check before provisioning (JobSet Operator installed via OLM, JobSetOperator CR exists, JobSet CRD is established). If any dependency is missing, the operator sets `Degraded` and `ProvisioningSucceeded=False` conditions with descriptive messages and requeues after 60 seconds. This is more sophisticated than many operators that simply fail; it provides clear user-facing guidance about what needs to be installed. On vanilla Kubernetes, only the JobSet CRD check is performed since OLM is not available.

The operator's drift correction mechanism watches downstream resources (Deployments, Services, ConfigMaps, ValidatingWebhookConfigurations) filtered by the `platform.opendatahub.io/part-of: trainer` label selector. When any managed resource is modified externally, the watch triggers re-reconciliation via `cluster.EnqueueOwner()`, which uses annotations (`platform.opendatahub.io/instance-name`, `platform.opendatahub.io/instance-namespace`) stamped during rendering to map back to the owning Trainer CR. This provides self-healing for configuration drift without cluster-scoped watches on all resources.

The ClusterTrainingRuntime manifests define 15 variants across two framework categories (torch-distributed and training-hub) and three accelerator types (CPU, CUDA, ROCm), with both "floating" (latest-version) and pinned (specific PyTorch/CUDA/ROCm version) variants. Image references in these manifests are parameterized via kustomize `replacements` from a `configMapGenerator`, allowing the platform operator to override all training images centrally. This is important for air-gapped and disconnected environments where images must come from a local registry.
