## Purpose

**Short**: A batch-oriented runtime for executing language model evaluations using the lm-evaluation-harness framework, integrated with the EvalHub platform for RHOAI TrustyAI.

**Detailed**: lm-evaluation-harness is a downstream fork of EleutherAI's lm-evaluation-harness (v0.4.8), adapted for Red Hat OpenShift AI's TrustyAI evaluation pipeline. It provides a standardized, secure environment for running language model benchmarks as Kubernetes batch jobs. The component ships as a single container image (`odh-ta-lmes-job`) that receives evaluation job specifications via a mounted ConfigMap, executes benchmarks against model inference endpoints using OpenAI-compatible APIs, and reports results back to the EvalHub service.

The adapter layer (`main.py`) bridges the EvalHub job lifecycle with lm-evaluation-harness internals: it loads a `JobSpec` JSON (model name, benchmark ID, parameters), constructs an OpenAI-compatible completions client (`local-completions` backend), runs the evaluation with `simple_evaluate()`, and publishes results via callbacks (status updates, OCI artifact persistence, MLflow logging). The framework supports 6,500+ benchmark tasks defined as YAML configurations, 117 evaluation metrics, and multiple model backends (HuggingFace, vLLM, OpenAI, Anthropic, IBM watsonx.ai, etc.), though in RHOAI production the `local-completions` backend is the primary pathway.

## Architectural Analysis

The lm-evaluation-harness fork follows a **batch job architecture** pattern — it is designed as a one-shot execution container, not a long-running service. The trustyai-service-operator spawns Kubernetes Jobs from LMEvalJob CRD specifications, and each job runs a single evaluation lifecycle from start to finish. This design provides strong isolation between evaluations (each runs in its own pod), simplifies error handling (a failed job can be retried by the operator), and avoids the complexity of managing persistent state across evaluations.

The **EvalHub adapter layer** (`main.py`) is the key Red Hat-specific addition on top of the upstream EleutherAI framework. It introduces a structured job lifecycle with well-defined phases (INITIALIZING → LOADING_DATA → RUNNING_EVALUATION → POST_PROCESSING → PERSISTING_ARTIFACTS), callback-driven status reporting to the EvalHub service, and multi-backend artifact persistence (OCI registries, MLflow). The adapter always uses the `local-completions` model backend, meaning evaluation traffic always flows through an external OpenAI-compatible API endpoint — the job container itself never loads model weights, keeping GPU requirements limited to the inference serving layer. This separation of concerns is architecturally significant: the evaluation harness runs on CPU-only pods while model inference happens on dedicated GPU-equipped serving infrastructure.

The **model backend system** uses a registry pattern (`@register_model` decorator) where model backends register themselves by name and are instantiated via string-based configuration. The `TemplateAPI` base class in `api_models.py` provides a sophisticated HTTP client implementation with support for async concurrent requests (up to 128 via `num_concurrent`), configurable TLS (custom CA bundles, certificate verification), retry logic with exponential backoff, batched requests, and both synchronous (`requests`) and asynchronous (`aiohttp`) transport paths. The TLS implementation is notably security-conscious: symlink paths for CA certificates are explicitly rejected (line 136-137 of `api_models.py`), and the `ssl.create_default_context()` call ensures system-level cipher suite and protocol selection rather than hardcoded values.

The **task system** is one of the most extensive aspects of the codebase, with 6,500+ YAML task definition files spanning benchmarks from MMLU and HellaSwag to specialized medical, legal, and multilingual evaluation suites. Tasks are organized into groups, subtasks, and tags for flexible composition. The Unitxt integration adds a plugin architecture for dynamically defining evaluation tasks using the Unitxt framework's modular data preparation pipeline. The **offline mode** support (`HF_HUB_OFFLINE`, `HF_EVALUATE_OFFLINE`, etc.) is a critical production feature that enables evaluations in disconnected/air-gapped environments by pre-populating datasets and metrics from an S3-backed init container.

The **multi-architecture build** strategy in `Dockerfile.konflux.lmes-job` is notably complex: it uses 5 build stages to handle architecture-specific compilation of PyArrow (from source on s390x and ppc64le), PyTorch (from source on s390x and ppc64le), and OpenBLAS (from source on ppc64le). The x86_64 architecture uses pre-built wheels. This reflects RHOAI's requirement to support IBM Z (s390x) and IBM Power (ppc64le) in addition to x86_64, which requires building native C/C++ extensions from source on non-x86 architectures.
