## Purpose

**Short**: LMEval is a secure, batch-oriented runtime for one-shot language model evaluations using the lm-evaluation-harness framework, integrated with EvalHub for orchestrated benchmark execution.

**Detailed**: lm-evaluation-harness is a fork of EleutherAI's lm-evaluation-harness framework, adapted for Red Hat OpenShift AI's TrustyAI evaluation pipeline. The component runs as a Kubernetes Job container (`odh-ta-lmes-job-rhel9`) that executes language model benchmarks against OpenAI-compatible inference endpoints. It provides a standardized runtime for evaluating LLM performance across hundreds of benchmarks (MMLU, HellaSwag, ARC, BBH, GPQA, IFEval, etc.) and the Unitxt evaluation framework.

The adapter layer (`main.py`) integrates with the EvalHub SDK (`eval-hub-sdk[adapter]`), which handles job specification loading from `/meta/job.json`, status reporting via callbacks, credential management (model auth, HF tokens, CA certificates), and OCI artifact persistence of evaluation results via `skopeo`. The adapter supports air-gapped/disconnected operation by auto-detecting local tokenizer and dataset files under `/test_data` and configuring Hugging Face offline mode. The runtime uses the `local-completions` model backend to send prompts to external model endpoints, supporting concurrent inference requests with configurable batching and timeout parameters.

## Architectural Analysis

The lm-evaluation-harness component occupies a unique niche in the RHOAI architecture: it is a stateless, one-shot batch container rather than a long-running service or operator. The TrustyAI LMEval operator creates Kubernetes Jobs that instantiate this container image, inject job specifications via ConfigMap (`/meta/job.json`), and mount model credentials as secrets. The container runs a single evaluation, reports results, and terminates. This design cleanly separates orchestration (operator) from execution (harness).

The adapter layer in `main.py` is the critical Red Hat customization atop the upstream EleutherAI framework. It implements the `FrameworkAdapter` interface from `eval-hub-sdk`, translating EvalHub's job specification format into lm-evaluation-harness's `simple_evaluate()` parameters. The adapter always uses the `local-completions` model backend (OpenAI-compatible endpoint), meaning the evaluated model runs externally (e.g., via vLLM, TGI, or another serving runtime). This is architecturally important — the harness never loads model weights itself; it only tokenizes prompts and sends HTTP requests.

The air-gapped support is noteworthy. The `_seed_hf_offline_before_lm_eval_import()` function runs before any Hugging Face library import, detecting local tokenizer and dataset files under `/test_data`. When detected, it sets `HF_HUB_OFFLINE=1`, `HF_DATASETS_OFFLINE=1`, and redirects all HF cache directories to `/test_data`. The offline dataset loading in `lm_eval/api/task.py` supports DatasetDict bundles exported via `datasets.save_to_disk()`, with explicit error handling for PyArrow version mismatches between the export and runtime environments. This is essential for disconnected OpenShift clusters.

Security-wise, the adapter includes credential sanitization (`_sanitize_error_message`) that redacts Bearer tokens, API keys, passwords, and URL credentials from error messages before they are sent to EvalHub callbacks. This prevents accidental credential leakage through error reporting. The `verify_certificate` parameter in `api_models.py` validates CA bundle paths rigorously — rejecting symlinks, verifying file existence, and using `ssl.create_default_context()` for proper SSL context creation. The job spec path is also path-traversal protected via `_resolve_job_spec_path_for_read()`, which ensures the spec file resolves under `/meta` only.

The multi-architecture build is complex. The Konflux Dockerfile includes three parallel build stages: Arrow (PyArrow built from source for ppc64le and s390x), PyTorch (built from source for ppc64le and s390x with gcc-toolset-13), and OpenBLAS (built for ppc64le). On x86_64, pre-built wheels from PyPI are used. This cross-architecture support is critical for RHOAI's multi-platform certification requirements but makes the build pipeline significantly more complex than a typical Python project.

The evaluation error handling includes sophisticated HTTP error code mapping (`_evaluation_failure_for_evalhub`) that translates model endpoint errors (401, 403, 404, 408, 429, 5xx) into structured EvalHub message codes, and detects gated HuggingFace dataset errors with actionable guidance for users to configure HF_TOKEN.
