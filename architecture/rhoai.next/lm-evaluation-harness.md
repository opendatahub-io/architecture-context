## Purpose

**Short**: LMEval is a secure, batch-oriented runtime for executing language model evaluations using the lm-evaluation-harness framework, integrated with the EvalHub platform for job orchestration, credential management, and result persistence.

**Detailed**: This component is a Red Hat downstream fork of EleutherAI's lm-evaluation-harness (v0.4.8), extended with an EvalHub adapter (`main.py`) that transforms the evaluation framework into a containerized Kubernetes batch job. The adapter receives a job specification (JSON) from EvalHub, configures the evaluation against an OpenAI-compatible model endpoint, executes benchmark tasks, and reports results back through EvalHub callbacks including status updates, OCI artifact creation, and MLflow experiment tracking.

The container image (`odh-ta-lmes-job`) is built for four architectures (x86_64, arm64, ppc64le, s390x) with custom-compiled PyTorch and PyArrow for non-x86 platforms. It runs as a non-root user (UID 65532) and forces CPU-only evaluation, connecting to external model inference servers via HTTP(S) rather than loading models locally. The evaluation harness supports 150+ benchmark tasks (MMLU, GSM8K, HellaSwag, ARC, TruthfulQA, BBH, GPQA, etc.) with pluggable metrics, few-shot prompting, and the Unitxt evaluation framework.

The adapter implements security-conscious patterns including CWE-22 path traversal prevention for job spec loading, comprehensive credential redaction in error messages, custom CA certificate validation for model endpoints, and automatic air-gapped/offline mode detection for disconnected clusters.

## Architectural Analysis

The lm-evaluation-harness adapter follows a **batch job pattern** rather than a long-running service pattern. Each evaluation is a single-shot execution: the container starts, reads a job specification, executes the benchmark, reports results, and terminates. This makes it fundamentally different from most RHOAI components — it has no ingress, no persistent state, and no RBAC of its own. The TrustyAI operator is responsible for creating the Kubernetes Job, mounting the job spec ConfigMap, and managing the pod lifecycle.

The EvalHub adapter layer (`main.py`) is the primary Red Hat-specific addition on top of the upstream EleutherAI framework. It implements a clean separation: the upstream `lm_eval.simple_evaluate()` function handles all benchmark execution, while the adapter handles orchestration concerns — credential management, status reporting, error classification, and artifact persistence. This separation is well-designed for maintainability: upstream rebases only need to resolve conflicts in the `lm_eval/` directory, while the adapter remains stable. The adapter always uses the `local-completions` backend (OpenAI-compatible API), forcing all model communication through HTTP(S) to external inference servers rather than loading models into the job pod. This is architecturally significant: it means the evaluation pod itself is CPU-only and lightweight, with all GPU/accelerator requirements pushed to the model serving layer.

The security design is notably thorough for a batch component. The `_sanitize_error_message()` function (main.py:265-309) implements multi-pattern credential redaction covering Bearer tokens, key=value secrets, JSON-encoded secrets, and URL credentials before any error is sent to EvalHub callbacks. The job spec path validation (`_resolve_job_spec_path_for_read()`, main.py:51-72) prevents CWE-22 path traversal by resolving the path and checking it stays under `/meta`. The `verify_certificate` parameter in `api_models.py` includes symlink traversal prevention (line 135) — Kubernetes secret mounts expose keys as symlinks, and the code validates the path is not a symlink before resolving it. However, `main.py:529` deliberately resolves symlinks with `os.path.realpath()` for the CA cert path, working around this restriction for the K8s-mounted credential case. This asymmetry is intentional but worth noting.

The multi-architecture build in `Dockerfile.konflux.lmes-job` is the most complex aspect of the build system. For x86_64 and arm64, standard PyPI wheels are used. For ppc64le and s390x, PyTorch 2.6.0 and Apache Arrow must be compiled from source (stages 2-4), which significantly increases build time and introduces non-hermetic git clones. The s390x Arrow build uses a specific commit SHA (`7c3d4867e40dd0100542247a61cb83520369b2d4`) with a custom parquet patch, indicating ongoing platform-specific compatibility work. The OpenBLAS compilation for ppc64le targets POWER9 specifically.

The offline/air-gapped detection (main.py:154-213) is sophisticated: it reads the job spec _before_ importing lm_eval to set HuggingFace offline environment variables, since those libraries cache their configuration at import time. The detection heuristic checks if `parameters.tokenizer` points to a path under `/test_data` and whether `dataset_dict.json` files exist alongside the tokenizer — indicating that an init container has pre-populated the local data. This approach avoids requiring an explicit "offline" flag in the job spec.
