## Purpose

**Short**: LMEval is a secure, batch-oriented runtime for one-shot language model evaluations using the lm-evaluation-harness framework, integrated with the EvalHub platform.

**Detailed**: LM Evaluation Harness (LMEval Job) is a downstream fork of EleutherAI's `lm-evaluation-harness` framework, adapted for Red Hat OpenShift AI (RHOAI). It provides a containerized, batch-oriented execution environment for running standardized language model benchmark evaluations. The component is deployed as a Kubernetes Job (not a long-running service) and executes a single evaluation run per invocation, then exits.

The adapter layer (`main.py`) integrates with the EvalHub platform via the `eval-hub-sdk`. It reads a job specification (JSON) from a mounted ConfigMap, configures the lm-evaluation-harness to evaluate a model against a specified benchmark, reports progress and status updates to EvalHub via callbacks, and persists evaluation results as OCI artifacts to an OCI registry. The adapter always uses the `local-completions` model backend, connecting to an OpenAI-compatible inference endpoint (such as vLLM or TGI) to evaluate the model remotely rather than loading it locally.

The container image supports multiple architectures (x86_64, arm64, ppc64le, s390x) and includes custom build stages for PyTorch, Apache Arrow, and OpenBLAS to support non-x86 platforms. It bundles hundreds of pre-defined evaluation tasks (MMLU, HellaSwag, ARC, GPQA, BBH, IFEval, etc.) and supports the Unitxt framework for customizable evaluation pipelines. Results can be exported to OCI registries and MLflow for tracking.

## Architectural Analysis

The LMEval Job follows a **batch adapter pattern** — it wraps the upstream EleutherAI `lm-evaluation-harness` library with a thin adapter layer (`main.py`, ~565 lines) that bridges between the EvalHub job specification format and the lm-eval framework's `simple_evaluate()` API. The adapter reads a job spec from a mounted ConfigMap (`/meta/job.json`), translates it into lm-eval parameters (model backend, benchmark tasks, few-shot settings), executes the evaluation, and reports results back via EvalHub callbacks. This pattern keeps the upstream lm-eval library largely unmodified while adding RHOAI-specific integration.

The component **always evaluates models remotely** via the `local-completions` backend, which sends prompts to an OpenAI-compatible HTTP completions endpoint. This is a critical architectural decision: the LMEval container never loads model weights directly. Instead, it relies on a separate inference server (typically vLLM or TGI, already deployed in the cluster) to serve the model. This separation of concerns means the LMEval pod can run on CPU-only nodes with modest resource requirements, while the GPU-intensive inference runs on dedicated serving infrastructure. The adapter constructs the completions URL from `model.url` in the job spec and authenticates via `OPENAI_API_KEY` extracted from model credentials.

The **offline mode** (`parameters.offline=true`) is a notable feature for air-gapped deployments. When enabled, the adapter sets multiple HuggingFace environment variables (`HF_HUB_OFFLINE`, `HF_DATASETS_OFFLINE`, `HF_EVALUATE_OFFLINE`, `TRANSFORMERS_OFFLINE`) and redirects `HF_HOME` to `/test_data`, which is expected to be pre-populated by an init container that downloads data from S3. The `__main__.py` module also patches `evaluate.load()` to load metrics from a local `metrics/` directory when `HF_EVALUATE_OFFLINE=1`, which is critical since the bundled `metrics/` folder in the repository contains all 55+ evaluation metrics as local Python modules. This air-gapped support is comprehensive and well-engineered.

The **multi-architecture build** is notably complex. The Konflux Dockerfile (`Dockerfile.konflux.lmes-job`, 260 lines) uses a 5-stage build: (1) base builder with common tooling, (2) Apache Arrow from source for ppc64le/s390x, (3) PyTorch from source for ppc64le/s390x, (4) OpenBLAS from source for ppc64le, and (5) final image assembly. The s390x stage includes a custom parquet patch (`patches/s390x/parquet-support.patch`). For x86_64 and arm64, pre-built wheels from PyPI are used directly. The final image runs as non-root user `65532:65532` and the entrypoint is simply the Python interpreter, with the actual command provided by the Kubernetes Job spec.

The **concurrent request support** (`num_concurrent` parameter) allows the adapter to send multiple inference requests in parallel to the model endpoint, bounded by a maximum of 128 concurrent requests. This leverages the `TemplateAPI` base class's async HTTP support via `aiohttp` and can significantly speed up evaluations when the inference server has capacity. The adapter validates and bounds this parameter to prevent abuse.
