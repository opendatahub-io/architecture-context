## Purpose

**Short**: CLI tool and data container that collects, enriches, and serves AI model metadata catalogs and MCP server catalogs for the RHOAI dashboard.

**Detailed**: model-metadata-collection is a Go-based offline data pipeline that produces YAML catalog files describing Red Hat AI models and Model Context Protocol (MCP) servers. The primary binary (`model-extractor`) reads model index files (`data/models-index.yaml`) and MCP server index files, then performs a multi-stage enrichment pipeline: (1) fetching HuggingFace collections to discover validated models, (2) extracting model card metadata from OCI container image layers (modelcar format), (3) enriching metadata with HuggingFace API data (descriptions, licenses, languages, tags, tool-calling configurations), (4) looking up vLLM recommended serving configurations, and (5) generating deduplicated YAML catalog files.

The enriched catalogs (`data/models-catalog.yaml`, `data/validated-models-catalog.yaml`, `data/redhat-mcp-servers-catalog.yaml`, etc.) are committed to the repository by CI/CD. The container image (`Dockerfile.konflux` / `Dockerfile`) simply copies these pre-generated YAML files into a minimal UBI image that runs `sleep infinity`. At runtime, this container is deployed as an init container or sidecar, and its `/app/data/` volume is mounted by the RHOAI dashboard to read the model and MCP server catalog data. The component has no network services, no APIs, and no runtime processes — it is purely a data delivery mechanism.

A secondary binary (`metadata-report`) generates completeness reports analyzing which metadata fields are populated, what data sources they came from (modelcard YAML frontmatter, HuggingFace API, regex extraction, registry artifacts), and identifying gaps.

## Architectural Analysis

The model-metadata-collection component implements a clever separation between build-time data generation and runtime data serving. The Go CLI tools (`model-extractor`, `metadata-report`) are development/CI tools that perform the computationally expensive work of pulling OCI manifests, calling HuggingFace APIs, and running enrichment logic. The output — static YAML catalog files — is committed to the Git repository. The production container image then simply copies these pre-generated files into a minimal UBI9 image with no Go binaries, no runtime dependencies, and no network access requirements. This architecture means the container image is extremely small, starts instantly, and has a minimal attack surface.

The metadata enrichment pipeline uses a sophisticated multi-source priority system. Each metadata field (name, provider, description, license, tasks, etc.) can come from multiple sources with different priority levels: HuggingFace YAML frontmatter (highest), HuggingFace API data, modelcard YAML frontmatter, modelcard regex extraction, HuggingFace README text parsing, or generated defaults (lowest). The `MetadataSource` type tracks provenance for each field, enabling the `metadata-report` tool to produce detailed audits of where data came from. This is a mature data lineage system that enables confidence in catalog quality.

The component handles three distinct catalog types: (1) Red Hat AI model catalogs (validated models, other models) extracted from OCI modelcar containers, (2) MCP server catalogs (Red Hat, partner, community) with enrichment from OCI registries for architecture and timestamp data, and (3) supplemental static catalog entries manually curated in `input/supplemental-catalog.yaml`. The model family matching system (`internal/config/model_families.go`) prevents cross-family false matches during HuggingFace enrichment — for example, preventing a Granite container from matching a Llama HuggingFace entry. The concurrent processing (configurable via `--max-concurrent`, default 5) with semaphore-based rate limiting enables efficient batch processing of dozens of models.

The tool-calling configuration enrichment (`pkg/types/toolcalling.go`) is notable: it extracts structured tool-calling metadata from HuggingFace YAML frontmatter and converts chat template paths from HuggingFace's `examples/` convention to RHOAI's `opt/app-root/template/` runtime path. This path conversion bridges the gap between upstream HuggingFace model artifacts and the RHOAI serving runtime environment.

The dual CI system (GitHub Actions for ODH/upstream, Konflux/Tekton for RHOAI/downstream) reflects the component's dual distribution. The GitHub Actions workflow builds multi-arch images (amd64, arm64, ppc64le) and pushes to `quay.io/opendatahub/`. The Konflux pipeline builds for the `stable` branch and pushes to the RHOAI downstream registry. Both use the same `Dockerfile` (or `Dockerfile.konflux`) that performs no build — just `COPY data/*.yaml /app/data/`.
