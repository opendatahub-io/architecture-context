## Purpose

**Short**: A Go-based CLI tool and static data container that aggregates, enriches, and catalogs AI model metadata and MCP (Model Context Protocol) server metadata from container registries and HuggingFace APIs.

**Detailed**: Model Metadata Collection serves as the metadata backbone for Red Hat OpenShift AI's model catalog. It produces two primary artifacts: (1) a Go CLI tool (`model-extractor`) that processes model index YAML files, fetches metadata from OCI container registries and the HuggingFace API, enriches model cards with structured metadata (license, language, tasks, provider, vLLM configuration), and generates comprehensive catalog YAML files; and (2) a minimal data-only container image that bundles the pre-generated catalog YAML files for consumption by other platform components (such as the RHOAI dashboard).

The system operates as a batch pipeline rather than a long-running service. During CI/CD, the `model-extractor` binary processes models defined in index files (`data/models-index.yaml`, `data/validated-models-index.yaml`, `data/other-models-index.yaml`) and MCP server definitions (`data/*-mcp-servers-index.yaml`). It fetches OCI image manifests to extract architecture support and timestamps, queries HuggingFace APIs for model details and README content, applies confidence-based metadata merging with source provenance tracking, and writes catalog files. The final container image ships only the generated YAML data files—no Go binary is included in the shipped image. A secondary tool (`metadata-report`) generates completeness reports analyzing coverage of metadata fields across all cataloged models.

## Architectural Analysis

**Data-Only Container Pattern**: This component follows an unusual architectural pattern — it is a data-only container that runs `sleep infinity` and serves static YAML files via volume mounts. The Go binary (`model-extractor`) is exclusively a build-time tool used in CI/CD to generate the data files. The shipped container image is based on `ubi9-minimal` and contains no executable binaries, only YAML catalog files. This separation means the component has zero runtime attack surface beyond the base OS image.

**Multi-Source Metadata Enrichment with Provenance Tracking**: The enrichment pipeline implements a sophisticated multi-source metadata merging strategy with 7 priority levels (from `huggingface.yaml` at highest to `generated` at lowest). Each metadata field carries source attribution, enabling the `metadata-report` tool to produce completeness reports with full provenance. The confidence-based override logic (thresholds at 0.5 and 0.8 similarity scores) prevents cross-family model mismatches (e.g., preventing a Llama container from inheriting Granite model metadata) while allowing high-confidence enrichment to override low-quality names.

**Dual Build Pipeline (Konflux + GitHub Actions)**: The component is built through two parallel pipelines. The RHOAI downstream build uses Konflux with hermetic builds, multi-arch support (x86_64, ppc64le, s390x, arm64), and the `Dockerfile.konflux` which copies all data files via wildcard globs. The ODH upstream build uses GitHub Actions with Docker Buildx, targeting linux/amd64, linux/arm64, and linux/ppc64le, using the standard `Dockerfile` which explicitly lists each file. The RHOAI Konflux pipeline sets `hermetic: true` with Go module prefetch, though this is somewhat academic since the Dockerfile itself doesn't build any Go code — it only copies pre-generated data.

**MCP Server Catalog Extension**: The system was recently extended to support MCP (Model Context Protocol) server metadata alongside traditional AI model metadata. MCP servers have their own index format (`MCPServersIndex`), enrichment pipeline (OCI registry metadata), and catalog output, organized into three tiers: Red Hat, partner, and community. This mirrors the model catalog's tiered approach (Red Hat, validated, other) and suggests an evolving catalog that will encompass the broader AI tooling ecosystem beyond inference models.

**vLLM Configuration Recommendations**: The system integrates vLLM serving configuration recommendations alongside model metadata. Input files in `input/models/vllm-config/` define hardware-specific serving presets (online-serving with low-latency optimization, offline-serving with high-throughput) that are matched by exact model name and rendered into model card documentation via Go templates. This positions the catalog not just as a metadata index but as a deployment guidance system.
