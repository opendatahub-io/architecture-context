## Purpose

**Short**: Extracts, enriches, and catalogs AI model metadata and MCP server metadata from OCI container registries and HuggingFace, producing YAML catalog files shipped as a static data container image.

**Detailed**: Model Metadata Collection is a Go-based CLI tool (`model-extractor`) and reporting utility (`metadata-report`) that automates the creation and maintenance of structured metadata catalogs for Red Hat AI models and MCP (Model Context Protocol) servers. It operates as an offline build-time pipeline — not a runtime service — processing model data from multiple sources and producing pre-generated YAML catalog files that are baked into a container image.

The pipeline works in several stages: (1) discovery of validated model collections from HuggingFace via the HuggingFace API, (2) extraction of model card content from OCI container image layers (modelcar format), (3) enrichment of extracted metadata with HuggingFace README data, YAML frontmatter, repository tags, vLLM recommended configurations, and tool-calling metadata, (4) generation of structured catalog YAML files, and (5) packaging the resulting YAML files into a minimal `ubi9-minimal` container image.

The resulting container image (`odh-model-metadata-collection`) is consumed by the RHOAI dashboard as a data source — the dashboard mounts the container's `/app/data` volume to read the catalog YAML files, which populate the model catalog UI. The tool also processes MCP server metadata, enriching server entries with OCI registry data (architecture support, timestamps) and producing separate MCP server catalog files. A secondary `metadata-report` command generates completeness reports tracking data source provenance for each metadata field.

## Architectural Analysis

The model-metadata-collection component follows a **data pipeline architecture** rather than a service architecture. It is fundamentally a build-time ETL (Extract, Transform, Load) system that produces static data artifacts. The Go binary is never shipped in the final container image — only the YAML catalog files it produces are copied into a minimal `ubi9-minimal` container. This design means the component has zero runtime attack surface: no processes listen on ports, no code executes in the deployed container, and the image runs `sleep infinity` purely to keep the volume mountable.

The **metadata enrichment pipeline** implements a sophisticated multi-source priority system. Each metadata field (name, provider, description, license, language, tasks, tags) can come from one of six sources, with clear priority ordering: (1) HuggingFace YAML frontmatter (highest), (2) HuggingFace API, (3) HuggingFace README regex extraction, (4) HuggingFace repository tags, (5) container modelcard YAML frontmatter, (6) container modelcard regex extraction (lowest). This priority system is implemented in `internal/enrichment/update.go` and tracked in separate `enrichment.yaml` provenance files per model. The `metadata-report` tool then aggregates this provenance data into completeness reports, enabling quality auditing of the enrichment process.

A notable architectural pattern is the **model family guard** (`internal/config/model_families.go` and `internal/enrichment/enrichment.go`). When matching OCI registry models to HuggingFace models for enrichment, the system prevents cross-family matches (e.g., a Llama container matching a Granite HuggingFace entry) by extracting the model family name from both sides and requiring compatibility. The supported families list (`SupportedModelFamilies`) serves as a single source of truth used by both the enrichment matching logic and the name normalization regex in `pkg/utils/text.go`.

The **MCP server catalog pipeline** is a parallel track that shares infrastructure (OCI registry inspection, YAML marshaling, retry logic) but operates independently. MCP servers have their own index format (`MCPServersIndex`), metadata structure (`MCPServerMetadata` with tools, artifacts, runtime metadata, security indicators), and enrichment path (OCI architecture and timestamp extraction). The enrichment writes back directly to the input YAML files, making the process idempotent across runs. The three-tier MCP catalog structure (Red Hat, partner, community) supports distinct catalog files per provider tier.

The container image build strategy uses **two Dockerfiles**: `Dockerfile` (upstream/ODH, uses `ubi9-micro` with explicit file listing) and `Dockerfile.konflux` (downstream/RHOAI, uses `ubi9-minimal` with wildcard COPY). The Konflux variant includes `sample-data/` as benchmarks and uses a wildcard glob for catalog files, making it more maintainable as new catalog types are added. Both run as non-root user 1001 and expose volume mount points at `/app/data`.
