## Purpose

**Short**: Extracts, enriches, and catalogs AI model metadata and MCP server metadata from OCI container registries and HuggingFace, producing YAML catalog files shipped as a static data container image.

**Detailed**: model-metadata-collection is a build-time data pipeline that generates structured YAML catalog files describing Red Hat AI models (validated models, other models) and MCP (Model Context Protocol) servers. The pipeline runs as a CI/CD job — not as a runtime service — and its output is a container image containing only static YAML data files.

The tool operates in several phases: (1) it discovers model collections from HuggingFace's RedHatAI organization, (2) it fetches OCI container manifests from registries like `registry.redhat.io` to extract modelcard metadata from container layers, (3) it enriches extracted metadata with HuggingFace API data (descriptions, licenses, languages, tags, tasks, tool-calling configs, vLLM recommended configurations), and (4) it generates deduplicated, consolidated YAML catalog files. A separate MCP server catalog pipeline processes MCP server index files, enriches them with OCI registry metadata (architectures, timestamps), and produces MCP server catalogs.

The final container image (`odh-model-metadata-collection`) serves as a data volume: it runs `sleep infinity` and is mounted by other platform components (such as the RHOAI dashboard) to read the catalog YAML files at `/app/data/`. This pattern allows the model catalog data to be updated independently of the consuming services. A secondary `metadata-report` tool generates completeness reports tracking which metadata fields were populated and their data sources.

## Architectural Analysis

The model-metadata-collection component follows a **build-time data pipeline** pattern rather than the typical runtime service or operator pattern seen in most RHOAI components. The Go CLI tools (`model-extractor`, `metadata-report`) run during CI/CD to produce static YAML files, which are then packaged into a minimal container image. This design decouples catalog data updates from consuming service deployments — the dashboard can pick up new model catalog data simply by pulling a new version of this container image, without requiring a code change or redeployment.

The metadata enrichment pipeline implements a **multi-source priority hierarchy** for resolving metadata fields. Each field (name, provider, description, license, etc.) can come from multiple sources: container modelcard YAML frontmatter, modelcard markdown regex extraction, HuggingFace YAML frontmatter, HuggingFace API, HuggingFace repository tags, or generated fallbacks. The priority is generally: HuggingFace YAML frontmatter > HuggingFace API > modelcard YAML > modelcard regex > generated. This is tracked via `MetadataSource` structs that record both the value and its provenance (e.g., `"huggingface.yaml"`, `"modelcard.regex"`), enabling the metadata-report tool to audit data quality and source distribution. The `enrichment.yaml` sidecar files preserve this provenance for each model.

A notable design pattern is the **model family cross-matching prevention** in the enrichment module. The system maintains a centralized list of model families (`SupportedModelFamilies` in `internal/config/model_families.go`) and uses it to prevent incorrect matches — for example, preventing a Llama container from matching a Granite HuggingFace entry based on superficial name similarity. The `NormalizeModelName` function and `CalculateSimilarity` scoring work together with family extraction to ensure enrichment accuracy.

The MCP server catalog pipeline shares infrastructure with the model pipeline but follows a simpler path: index → OCI registry enrichment (architectures, timestamps) → catalog generation. MCP servers are organized into three tiers: Red Hat, Partner, and Community, each with its own index and catalog files. The enrichment phase writes updated metadata back to the input YAML files in-place (a side-effect pattern), which allows incremental enrichment without re-processing unchanged servers.

The container image build strategy is worth noting: the Konflux Dockerfile (`Dockerfile.konflux`) is a pure data copy — `FROM ubi9-minimal` followed by `COPY data/*.yaml /app/data/`. No Go compilation happens inside the container build. This means the shipped artifact contains no executable code beyond the UBI base, making it a low-risk security surface. The Go tools are development/CI dependencies only. The regular `Dockerfile` (non-Konflux) is similar but uses `ubi9-micro` and includes benchmark sample data in `/app/benchmarks/`.
