## Purpose

**Short**: A CLI tool and data container that extracts, enriches, and catalogs metadata for Red Hat AI models and MCP servers from OCI container registries and HuggingFace.

**Detailed**: Model Metadata Collection is a build-time data pipeline that produces YAML catalog files describing Red Hat AI validated models and Model Context Protocol (MCP) servers. The primary binary (`model-extractor`) reads a models index file listing OCI container image references, fetches each image's manifest and layers from container registries (primarily `registry.redhat.io`), extracts modelcard markdown files from annotated layers, parses metadata (name, provider, license, tasks, languages), and enriches the extracted data with information from the HuggingFace API (README YAML frontmatter, model details, collection membership). It also processes MCP server definitions from input YAML files, enriching them with OCI registry data (architectures, timestamps). A secondary binary (`metadata-report`) generates completeness reports showing which metadata fields are populated and from which sources.

The final output — a set of `*-catalog.yaml` and `*-index.yaml` files under `data/` — is baked into a minimal container image (`ubi9-minimal`) via Dockerfile.konflux. This container serves as a static data volume: it runs `sleep infinity` and exposes its `/app/data/` directory for other platform components (such as the RHOAI Dashboard) to mount and read model catalog information. The Go CLI is never shipped in the final container image; only the pre-generated YAML data files are included.

## Architectural Analysis

The model-metadata-collection repository implements a multi-stage metadata enrichment pipeline with a clear separation between build-time processing and runtime data delivery. The Go CLI tools (`model-extractor`, `metadata-report`) are never shipped — they run during CI/CD to generate YAML files that are checked into the repository. The Konflux Dockerfile then simply copies these pre-generated files into a minimal UBI9 container. This design eliminates runtime dependencies on external APIs and ensures the data container can be deployed without network access.

The metadata extraction pipeline follows a priority-based source resolution pattern. For each model, metadata can come from multiple sources: OCI container modelcard YAML frontmatter, modelcard regex parsing, HuggingFace YAML frontmatter, HuggingFace API responses, HuggingFace tags, and generated fallbacks. The `MetadataSource` type tracks provenance with explicit source labels (`modelcard.yaml`, `modelcard.regex`, `huggingface.yaml`, `huggingface.api`, `huggingface.tags`, `huggingface.regex`, `generated`). HuggingFace YAML frontmatter is generally treated as highest priority, allowing upstream model publishers to control their metadata. The enrichment system also supports cross-family matching prevention via `isCompatibleModelFamily()`, which uses a centralized list of model families (`SupportedModelFamilies` in `internal/config/model_families.go`) to prevent a Llama container from matching a Granite HuggingFace entry.

The codebase demonstrates careful handling of legacy format migration. The `internal/metadata/migration.go` file implements three-way format detection (`ExtractedMetadata`, `MixedTypeExtractedMetadata`, `LegacyExtractedMetadata`) to handle evolving YAML schemas without breaking existing data. Timestamps can appear as strings or int64 values, and the migration layer normalizes them transparently. The catalog generation (`internal/catalog/catalog.go`) includes deduplication logic that consolidates models with the same name, merging their artifacts while preserving the earliest create time and latest update time.

The MCP server catalog pipeline, added more recently, follows the same enrichment pattern but with simpler input: YAML files in `input/mcp_servers/` describe each server's metadata, and OCI registry enrichment adds architecture and timestamp information. The index file (`data/redhat-mcp-servers-index.yaml`) maps server names to their input file paths, with path traversal protection (`filepath.Clean`, rejection of absolute or `..` paths). Three tiers of MCP servers are supported: Red Hat, partner, and community, each with their own index and catalog output files.

The container image uses a non-root user (UID 1001) and exposes two volumes: `/app/data` for catalog YAML files and `/app/benchmarks` for sample model evaluation data. The Dockerfile.konflux explicitly copies each YAML file individually (rather than using globs) to maintain precise control over what ships in the image. The regular `Dockerfile` uses a glob pattern and ships to `quay.io/opendatahub/odh-model-metadata-collection` via GitHub Actions, while the Konflux Dockerfile ships to the Red Hat registry via Tekton pipelines.
