## Purpose

**Short**: MLflow is the experiment tracking, model registry, and AI gateway service for Red Hat OpenShift AI, deployed as a shared Kubernetes-native multi-tenant server.

**Detailed**: MLflow provides a centralized platform for the complete machine learning lifecycle, including experiment tracking, model versioning and registry, artifact management, and AI gateway capabilities. In the RHOAI distribution, MLflow is deployed as a single shared tracking service managed by the MLflow Operator. Two Kubernetes-aware extensions — the `kubernetes://` workspace provider and the `kubernetes-auth` application plugin — enable multi-tenancy by mapping Kubernetes namespaces to MLflow workspaces and enforcing Kubernetes RBAC for each API request.

The server exposes both REST APIs (Flask + FastAPI/ASGI) and a React-based web UI for interactive experiment management. It supports multiple backend store types (PostgreSQL, MySQL, SQLite) for metadata persistence, and multiple artifact store backends (S3, GCS, Azure Blob, HDFS, local filesystem) for model and data artifact storage. The Helm chart packages the deployment with optional TLS, NetworkPolicy, Prometheus monitoring, Ingress, and garbage collection CronJob resources.

MLflow integrates with the broader RHOAI platform through the ODH data science gateway (exposed at `/mlflow`), receiving traffic via HTTPRoute from the platform's Gateway API ingress. SDK clients authenticate using Kubernetes service account tokens, and workspace context is derived from the caller's namespace.

## Architectural Analysis

MLflow in the RHOAI distribution represents a significant downstream extension of the upstream MLflow open-source project. The fork maintains compatibility with the upstream codebase while adding Kubernetes-native multi-tenancy through two plugin mechanisms: the workspace provider and the auth plugin. This plugin-based architecture is a notable design decision — rather than forking the server core, the extensions use MLflow's existing entry point system (`mlflow.workspace_provider` and `mlflow.app`), which reduces the maintenance burden during upstream rebases.

The multi-tenancy model maps Kubernetes namespaces directly to MLflow workspaces, which is an elegant alignment with the RHOAI platform's existing project isolation model. The workspace provider is read-only — it reflects namespaces as workspaces rather than managing workspace lifecycle — which keeps MLflow stateless with respect to workspace provisioning and delegates that responsibility to the platform operator and Kubernetes itself. Authorization relies on `SelfSubjectAccessReview` against a pseudo-resource in the `mlflow.kubeflow.org` API group, reusing the platform's existing RBAC machinery rather than introducing a separate permission system.

The server architecture is a hybrid Flask/FastAPI stack. The Flask app provides backward compatibility with MLflow's extensive handler codebase (the `handlers.py` file alone is 266KB), while FastAPI wraps it via WSGI middleware to support ASGI features like async OTLP trace ingestion, job management APIs, and the AI gateway. This layered approach adds complexity but avoids rewriting the entire handler layer. The custom `_EfficientWSGIResponder` class addresses a performance issue in the standard WSGI-to-ASGI bridge by avoiding O(n^2) body buffering.

The security middleware is implemented twice — once for Flask and once for FastAPI — with shared utility functions. This dual implementation is necessary because Flask and FastAPI have different middleware patterns, but it creates a maintenance risk where security behavior could diverge between the two layers. The middleware provides host header validation (DNS rebinding protection), CORS blocking for state-changing requests, and security headers (X-Frame-Options, X-Content-Type-Options).

The Helm chart is well-structured with comprehensive security defaults (readOnlyRootFilesystem, non-root user, dropped capabilities, seccompProfile). The NetworkPolicy template covers DNS (both port 53 and OpenShift's CoreDNS port 5353), HTTPS, Kubernetes API, PostgreSQL, MySQL, and S3-compatible storage ports. The garbage collection CronJob provides automated cleanup of soft-deleted resources, which is essential for production deployments where storage costs matter.

The cryptographic implementation in `mlflow/utils/crypto.py` uses the `cryptography` library's AES-GCM for envelope encryption of gateway secrets, with PBKDF2HMAC for key derivation from a passphrase. The default passphrase is documented as development-only, and the code includes clear security warnings. The `hashlib.sha1` usage in `conda.py` explicitly sets `usedforsecurity=False`, which is a FIPS-safe pattern. Overall, the crypto hygiene is solid for a Python application — no custom crypto, no weak algorithms for security purposes, and proper use of the `cryptography` library which delegates to OpenSSL.
