## Purpose

**Short**: MLflow is a machine learning lifecycle platform providing experiment tracking, model versioning, model registry, artifact management, and LLM observability with Kubernetes-native workspace and authentication integration for RHOAI.

**Detailed**: MLflow is an open-source platform for managing the end-to-end machine learning lifecycle, forked by Red Hat Data Services from the upstream MLflow project. In the RHOAI deployment model, it runs as a single shared tracking service deployed by the MLflow Operator. The server exposes REST APIs, a GraphQL endpoint, an OTLP trace ingestion endpoint, and an AI Gateway proxy — all behind a hybrid Flask/FastAPI application stack.

The RHOAI-specific integration centers on two Kubernetes-aware extensions: a workspace provider that maps Kubernetes namespaces to MLflow workspaces (providing multi-tenant isolation without separate MLflow instances per namespace), and a `kubernetes-auth` application plugin that enforces Kubernetes RBAC for each API request using SelfSubjectAccessReview delegation. An optional namespace-scoped `MLflowConfig` CRD allows per-workspace artifact storage overrides. Users reach the shared MLflow service through the ODH data science gateway's `/mlflow` HTTPRoute endpoint.

The server is built as a single container image from a 3-stage Dockerfile (Node.js UI build, Python wheel build, UBI9-minimal runtime) and exposes port 5000/TCP. It supports PostgreSQL and MySQL for backend metadata storage, and pluggable artifact storage backends including S3, GCS, Azure Blob, and HDFS. The project also produces three Python sub-packages (`mlflow`, `mlflow-skinny`, `mlflow-tracing`) with progressively smaller dependency footprints for different consumption scenarios.

## Architectural Analysis

The MLflow codebase represents a substantial fork of the upstream Databricks MLflow project with significant Kubernetes-native extensions for RHOAI integration. The most architecturally noteworthy decision is the **hybrid Flask/FastAPI application stack**: legacy tracking and model registry APIs remain in Flask (mounted via WSGIMiddleware at the root path), while newer functionality — AI gateway, job management, OTLP trace ingestion, and assistant APIs — uses native FastAPI routes. This dual-framework approach allows incremental modernization without a risky wholesale migration, but it creates two distinct middleware chains (Flask `before_request` vs. FastAPI middleware) that must both enforce security policies consistently.

The **workspace isolation model** is the core RHOAI integration pattern. Rather than deploying separate MLflow instances per tenant, a single shared server uses request-scoped context variables to enforce workspace isolation at the data layer. The `X-MLFLOW-WORKSPACE` header drives workspace resolution, and a `workspace_aware_mixin` automatically filters all SQLAlchemy queries to the active workspace. This is elegant but creates a trust dependency: if the middleware fails to set workspace context, queries default to the default workspace (ID=0), which could expose cross-tenant data. The permission model has a defensive fallback — when workspaces are enabled and no explicit permission is found, access defaults to `NO_PERMISSIONS` rather than a permissive default.

The **Kubernetes auth plugin architecture** is pluggable via entry points (`mlflow.app` and `mlflow.app.client`), meaning the authentication mechanism can be swapped without modifying server code. The `kubernetes-auth` plugin uses Kubernetes service account tokens and delegates authorization to the API server via SelfSubjectAccessReview against pseudo-resources in the `mlflow.kubeflow.org` API group. This reuses the platform's existing RBAC infrastructure rather than introducing a separate permission system, but it means every authenticated MLflow API call generates at least one Kubernetes API call, creating a latency dependency on the API server.

The **AI gateway** subsection is notable for its secrets management: provider API keys are encrypted at rest using AES-256-GCM with a KEK (Key Encryption Key) derived via PBKDF2-HMAC-SHA256 with 600,000 iterations. The KEK passphrase is provided via environment variable. Gateway provider configurations support traffic splitting (weighted routing across model versions) and fallback chains (automatic retry with alternative models), both configured via database-stored endpoint configs. The provider abstraction (`get_provider()` factory) supports OpenAI, Anthropic, Mistral, Gemini, and LiteLLM backends with streaming response support.

The **build system** demonstrates strong supply chain security practices. The Konflux Dockerfile uses a 3-stage build (Node.js UI builder → Python wheel builder → UBI9-minimal runtime) with SHA256-pinned base images, `--require-hashes` for all pip installs, hermetic Cachi2 prefetch, and locked dependencies at all layers (RPM, Python, JavaScript). Multi-architecture support covers amd64, arm64, ppc64le, and s390x. The production image explicitly disables telemetry, job execution, assistant features, and AI gateway by default — these must be explicitly enabled, following a secure-by-default principle.
