## Purpose

**Short**: MLflow is a shared experiment tracking, model registry, and AI gateway service for Red Hat OpenShift AI, providing multi-tenant ML lifecycle management via Kubernetes-native workspace isolation.

**Detailed**: MLflow is deployed as a single shared tracking server on OpenShift AI by the MLflow Operator. It provides REST APIs and a React-based UI for experiment tracking, model versioning, artifact management, distributed tracing, AI gateway (LLM routing), and evaluation scoring. The server uses a PostgreSQL (or other SQL) backend for metadata and supports pluggable artifact stores (S3, GCS, Azure Blob, local filesystem, etc.).

In the RHOAI deployment model, MLflow extends its upstream capabilities with two Kubernetes-specific plugins: a `kubernetes://` workspace provider that maps Kubernetes namespaces to MLflow workspaces, and a `kubernetes-auth` application plugin that enforces Kubernetes RBAC via `SelfSubjectAccessReview` for every API request. This enables one shared MLflow deployment to serve many data science project namespaces with proper tenant isolation, without requiring per-namespace MLflow instances.

The server exposes its API surface through FastAPI (wrapping a legacy Flask app) with Protobuf-defined RPC contracts for tracking, model registry, and artifact services, plus native FastAPI routers for the AI Gateway, OpenTelemetry trace ingestion, and job execution APIs. The built-in React UI can be embedded in the ODH dashboard via Module Federation and communicates through the same API endpoints. TLS is terminated inside the MLflow pod, and traffic reaches the server via the ODH data science gateway's `/mlflow` HTTPRoute.

## Architectural Analysis

MLflow in the RHOAI context represents a unique deployment pattern: an upstream open-source project that has been extended with Kubernetes-native multi-tenancy rather than forked. The `mlflow-kubernetes-plugins` package provides the critical integration layer — mapping Kubernetes namespaces to MLflow workspaces and enforcing RBAC via `SelfSubjectAccessReview` — while the core MLflow codebase remains upstream-compatible. This approach minimizes downstream maintenance burden while delivering platform-grade tenant isolation.

The server architecture is in an active migration from Flask to FastAPI. The current implementation uses a FastAPI wrapper (`fastapi_app.py`) that mounts the Flask app via `WSGIMiddleware` at the root path, while new features (AI Gateway, OTLP trace ingestion, job execution, assistant) are implemented as native FastAPI routers that take precedence over the catch-all Flask mount. This dual-stack approach maintains backward compatibility while enabling async capabilities for new endpoints. The `kubernetes-auth` app plugin is loaded as a Flask application factory via the `mlflow.app` entry point system.

The secrets encryption system (`mlflow/utils/crypto.py`) implements envelope encryption with AES-256-GCM and PBKDF2-derived KEKs, following OWASP 2023 recommendations (600,000 PBKDF2 iterations). Each secret gets a unique DEK wrapped by the KEK, with Additional Authenticated Data (AAD) binding ciphertext to secret metadata to prevent substitution attacks. KEK rotation is supported without re-encrypting secret values (only DEK re-wrapping). However, the default passphrase fallback (`mlflow-default-kek-passphrase-for-development-only`) means production deployments must explicitly set `MLFLOW_CRYPTO_KEK_PASSPHRASE` — this is a deployment configuration concern rather than a code defect.

The RHOAI container image (`Dockerfile.konflux`) is built as a multi-stage, multi-architecture image (amd64, arm64, ppc64le, s390x) with UBI9 base images. The build is fully hermetic: RPMs are locked via `rpms.lock.yaml`, Python dependencies are locked via compiled requirements files and prefetched by cachi2, and the JS UI is built with Yarn in offline mode. Notable RHOAI-specific defaults include disabling telemetry (`MLFLOW_DISABLE_TELEMETRY=true`), job execution, the AI assistant, and the AI gateway UI — focusing the deployment on core tracking and model registry functionality. The startup command enables workspaces with `kubernetes://` as the workspace store and `kubernetes-auth` as the auth plugin, plus `--serve-artifacts` for proxied artifact access.

The workspace system enforces Kubernetes DNS naming conventions for workspace names (lowercase, 2-63 characters, no consecutive hyphens), which aligns with namespace naming since workspaces map 1:1 to namespaces. The `X-MLFLOW-WORKSPACE` request header carries workspace context, and the server resolves workspace state per-request using `ContextVar` for thread safety. This design enables a single server process to handle concurrent requests across many workspaces without workspace-level process isolation.
