## Purpose

**Short**: MLflow is an open-source ML experiment tracking, model registry, and artifact management platform adapted for Red Hat OpenShift AI with Kubernetes-native workspace isolation, authentication, and Module Federation UI embedding.

**Detailed**: MLflow provides the complete machine learning lifecycle management platform for RHOAI. It tracks experiments, runs, metrics, parameters, and artifacts; hosts a model registry with versioning, staging, and aliasing; serves an AI Gateway for proxying LLM provider calls; and provides distributed tracing integration via OpenTelemetry. The server exposes both REST (Protocol Buffer-backed) and GraphQL APIs, with a React/TypeScript frontend that can operate standalone or as a Module Federation remote embedded in the RHOAI dashboard.

The RHOAI adaptation adds several critical capabilities on top of upstream MLflow: Kubernetes-native workspace isolation that maps Kubernetes namespaces to MLflow workspaces for multi-tenancy, a `kubernetes-namespaced` auth provider that uses service account tokens and namespace context, and a Konflux-based build pipeline producing a single container image (`odh-mlflow-rhel9`) on UBI 9. The server is designed to run behind `kube-rbac-proxy` for authentication enforcement, delegating RBAC to the platform's ingress infrastructure rather than performing direct Kubernetes RBAC checks.

The codebase ships three Python packages (`mlflow`, `mlflow-skinny`, `mlflow-tracing`) plus a `mlflow-kubernetes-plugins` package for Kubernetes Job-based project execution. The frontend builds both a standalone React app and a federated bundle (`remoteEntry.js`) exposing experiment, prompt, and run components for the RHOAI dashboard to consume.

## Architectural Analysis

MLflow's RHOAI adaptation transforms a single-tenant ML tracking tool into a multi-tenant platform service. The workspace isolation model — mapping Kubernetes namespaces to MLflow workspaces — is the most significant architectural change from upstream. This is implemented through composite primary keys on most database tables (adding a `workspace` column), a request-scoped `ContextVar` for workspace propagation, and the `X-MLFLOW-WORKSPACE` HTTP header injected by the `kubernetes-namespaced` auth provider. The workspace boundaries extend to experiments, runs, models, evaluation datasets, gateway endpoints, secrets, and jobs. Workspace deletion supports three modes (RESTRICT, CASCADE, SET_DEFAULT), giving operators flexibility in tenant lifecycle management.

The server architecture is a hybrid Flask/FastAPI application. Legacy REST endpoints use Flask with Protocol Buffer message serialization (defining RPC-like handlers via `.proto` files), while newer features (jobs, gateway, assistant, OTLP traces) use FastAPI routers mounted on the same application. Flask is bridged into FastAPI via `WSGIMiddleware`, with FastAPI routes registered first to avoid Flask catching them. This dual-framework approach lets the team ship async-capable endpoints (SSE streaming, OTLP ingestion) without rewriting the extensive Protobuf-based handler system.

The security model relies on delegation rather than direct enforcement. MLflow itself implements an internal permission system (READ, USE, EDIT, MANAGE per resource), but in RHOAI deployments, authentication is handled by `kube-rbac-proxy` as a sidecar container that validates Kubernetes service account tokens before proxying requests to MLflow. The `kubernetes-namespaced` auth provider reads tokens from `/var/run/secrets/kubernetes.io/serviceaccount/token` (in-cluster) or kubeconfig (local development) and injects both `Authorization` and `X-MLFLOW-WORKSPACE` headers. Secrets management for AI Gateway provider API keys uses envelope encryption with a Key Encryption Key (KEK) passphrase, supporting rotation via `mlflow crypto rotate-kek`.

The frontend builds in two modes: a standalone React app (`yarn build`) and a Module Federation remote (`yarn build:federated`). The federated bundle exposes three wrapper components — `MlflowExperimentWrapper`, `MlflowPromptsWrapper`, and `MlflowRunTabsWrapper` — with PatternFly and OpenShift Dynamic Plugin SDK as shared singletons. The `src/odh/` directory contains RHOAI-specific integration code. React Router v6 is intentionally not shared with the dashboard (which uses v7), avoiding major-version incompatibility by providing an isolated BrowserRouter with a basename. The Dockerfile.konflux build produces both the standard and federated bundles, with `MLFLOW_ENABLE_ASSISTANT=false` and `MLFLOW_ENABLE_AI_GATEWAY=false` set at build time to exclude those features from the shipped UI.

The container image is built on UBI 9 Minimal with Python 3.12, runs on port 5000, and launches with `--app-name kubernetes-auth --enable-workspaces --workspace-store-uri kubernetes:// --serve-artifacts` by default. This means the shipped image expects Kubernetes workspace integration out of the box. The Konflux build pipeline supports multi-architecture builds (amd64, arm64, ppc64le, s390x) with hermetic builds and dependency prefetch for pip, yarn, and rpm packages.
