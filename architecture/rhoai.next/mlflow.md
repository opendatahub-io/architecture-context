## Purpose

**Short**: MLflow is an open-source ML experiment tracking, model registry, and AI gateway server deployed as a containerized Python service for Red Hat OpenShift AI.

**Detailed**: MLflow provides a comprehensive platform for managing the complete machine learning lifecycle. In its RHOAI deployment, it serves as the tracking server for recording experiment runs, parameters, metrics, and artifacts, as well as a model registry for versioning and staging ML models. The server exposes both a REST API (protobuf-based) and a React-based web UI for browsing experiments, comparing runs, and managing models.

The RHOAI distribution ships MLflow as a single container image (`odh-mlflow-rhel9`) built via Konflux with hermetic builds across four architectures (amd64, arm64, ppc64le, s390x). The image includes the MLflow tracking server, the built React UI, and integrations with Kubernetes workspaces for multi-tenant experiment isolation. The server supports multiple backend store options (PostgreSQL, MySQL, SQLite) and artifact storage backends (S3, local filesystem). A Helm chart is provided for Kubernetes deployment with configurable TLS, NetworkPolicy, RBAC, Prometheus metrics, and garbage collection CronJobs.

MLflow also includes an AI Gateway for proxying LLM provider API calls (OpenAI, Anthropic, Bedrock, Gemini, etc.) with rate limiting and budget management, an OpenTelemetry trace ingestion endpoint for distributed tracing, and a job execution system using a Huey task queue for server-side compute tasks.

## Architectural Analysis

MLflow's RHOAI deployment represents a unique packaging challenge: a large upstream open-source project (3.13.0 with 79+ Python sub-packages) repackaged as a hardened, multi-architecture container for OpenShift. The build strategy uses a three-stage Dockerfile with distinct UI builder (Node.js/Yarn), Python wheel builder, and minimal UBI9 runtime stages. This separation ensures the final image contains only runtime dependencies while the build toolchain stays in ephemeral stages.

The hybrid AIPCC/PyPI dependency strategy is noteworthy. Most Python packages (~150+ transitive) are sourced from the RHEL AI Python Package Index for supply chain security, but four packages (mlflow-kubernetes-plugins, psycopg2, prometheus-flask-exporter, starlette) must come from PyPI because they are not yet available on the AIPCC index. The `requirements/compile.py` script performs Docker-based per-architecture compilation to generate unified hash lock files across all four architectures, with an allowlist for packages missing on specific architectures (e.g., hf-xet on ppc64le/s390x).

The server architecture is a Flask application wrapped in a FastAPI/ASGI layer via a custom `_EfficientWSGIMiddleware` to avoid O(n^2) body buffering. This dual-framework approach allows the legacy protobuf-based REST API (Flask routes) to coexist with modern async endpoints (AI Gateway, OTLP traces, job execution) on the same server. Uvicorn is the default ASGI server with 4 workers, replacing the deprecated gunicorn path. The AI Gateway's envelope encryption system using AES-256-GCM with PBKDF2-derived KEKs (600,000 iterations) is well-designed with proper nonce management, AAD for anti-substitution, and KEK rotation support -- though the default passphrase for development is a production security risk if not overridden.

The Helm chart provides comprehensive operational controls including optional TLS (via Kubernetes Secrets), NetworkPolicy with explicit egress allowlists (DNS, HTTPS, databases, S3), Prometheus ServiceMonitor integration, and a garbage collection CronJob. Security hardening includes read-only root filesystem, non-root user (UID 1000), dropped capabilities, RuntimeDefault seccomp profile, and disabled service account token automounting by default. The Kubernetes workspace integration enables multi-tenant experiment isolation backed by namespace boundaries, which is critical for shared RHOAI deployments.

The React frontend uses a module federation architecture enabling dynamic plugin loading, which supports the embedded trace view (MlflowTraceDetailWrapper) and federated compare runs page for integration with other RHOAI components like the evaluation hub. The UI build produces both a standard build and a federated build with a configurable API base URL.
