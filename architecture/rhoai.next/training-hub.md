## Purpose

**Short**: A unified Python library providing a consistent API for fine-tuning large language models using multiple training algorithms (SFT, OSFT, LoRA, GRPO) with pluggable backend implementations.

**Detailed**: training-hub is a pure Python library distributed via PyPI (`pip install training-hub`) that abstracts away the complexity of LLM fine-tuning by providing a single, consistent interface across multiple training algorithms and backend engines. It implements an Algorithm-Backend plugin architecture where training algorithms (SFT, OSFT, LoRA+SFT, LoRA+GRPO, GRPO) are decoupled from their backend implementations (instructlab-training, mini-trainer, Unsloth, OpenPipe ART, verl), allowing users to switch between approaches without changing their code.

The library handles distributed training orchestration via PyTorch torchrun (DDP/FSDP), GPU memory profiling and estimation, training loss visualization, parameter-efficient fine-tuning (LoRA, QLoRA, DoRA), reinforcement learning from verifiable rewards (GRPO with tool-call verification), and Vision-Language Model (VLM) support. It integrates with experiment tracking systems (Weights & Biases, MLflow, TensorBoard) and supports multi-GPU/multi-node training configurations.

training-hub is not a deployed service, operator, or containerized application. It has no Kubernetes manifests, CRDs, network services, or container images. It is consumed as a library dependency by other projects and used directly by data scientists and ML engineers for model fine-tuning workflows.

## Architectural Analysis

### Algorithm-Backend Plugin Architecture

training-hub's central design pattern is the separation of training algorithms from their backend implementations through an `AlgorithmRegistry`. This allows the same high-level algorithm (e.g., LoRA+GRPO) to be executed by different engines (OpenPipe ART vs. verl) without any API change. The registry uses class-level dictionaries for algorithm and backend registration, with `create_algorithm()` serving as the factory method. This design is extensible -- new algorithms and backends can be added by subclassing `Algorithm`/`Backend` and calling `register_algorithm()`/`register_backend()`.

### Parameter Validation and Type Safety

The OSFT algorithm implementation includes a sophisticated runtime type validation system (`_validate_param_types()`, `_check_type()`) that handles Python typing generics (Union, Optional, Literal, list, dict) and provides clear error messages with expected vs. actual types. This is implemented only in OSFT; other algorithms rely on simpler validation or Pydantic models. Standardizing this across all algorithms would improve consistency.

### Subprocess Isolation for GRPO Backends

Both GRPO backends (ART and verl) use subprocess isolation rather than direct Python API calls. This is an intentional architectural decision driven by the complexity of multi-process training with vLLM inference servers. The ART backend spawns `art train` as a subprocess, while the verl backend spawns `verl.trainer.main_ppo`. Both backends prepare configuration files (JSON/YAML) and write reward function files to disk before launching subprocesses. This isolation prevents GPU memory conflicts between the training loop and the inference server.

### Distributed Training Orchestration

For SFT and OSFT, training-hub handles torchrun orchestration directly, including parameter resolution with a three-level hierarchy (explicit args > environment variables > defaults). The `get_torchrun_params()` utility in `utils.py` implements careful validation of mutually exclusive parameters (`master_addr` vs. `rdzv_endpoint`) with precedence-based conflict resolution and user-facing warnings. For LoRA and GRPO, distributed training is delegated to the backend frameworks.

### VLM Architecture Detection

The LoRA backend includes automatic Vision-Language Model detection through two methods: architecture-based matching (`_is_vlm_architecture()` checks for known VLM architectures like LlavaForConditionalGeneration, Qwen2VLForConditionalGeneration, etc.) and model-ID-based heuristic matching (`_is_vlm_model_id()` checks for patterns like "vision", "VL", "pixtral" in model names). This enables automatic configuration adjustments for VLM training without requiring explicit user flags.

### Memory Estimation Design

The profiling module provides four estimator classes (`BasicEstimator`, `OSFTEstimator`, `LoRAEstimator`, `QLoRAEstimator`) that calculate GPU memory requirements based on model parameter count, sequence length, batch size, and algorithm-specific factors. The `QLoRAEstimator` notably detects a pre-quantization memory bottleneck where loading the full-precision model before quantization temporarily requires more memory than the final training state. These estimators use HuggingFace model configs to determine parameter counts without downloading full model weights.

### Supply Chain Security Note

The `litellm` dependency is pinned to `<1.82.7` with an explicit comment referencing a supply chain attack (TeamPCP, 2026-03-24). This demonstrates supply chain awareness but also highlights the risk surface of having many optional ML framework dependencies.

### Timing Estimator Limitations

The `TimingEstimatorExperimental` class is explicitly marked as experimental and carries a warning that estimates "may vary greatly." It works by training on a small data subset (default 5%) and extrapolating, which inherently cannot account for GPU memory pressure changes, data complexity variations, or checkpoint overhead at scale. The class also has swapped docstrings (`_sft_log_parse` says "Parse the OSFT log file" and vice versa).
