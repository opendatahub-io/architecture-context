## Purpose

**Short**: A Python SDK providing an algorithm-focused interface for language model fine-tuning, continual learning, and reinforcement learning techniques.

**Detailed**: training-hub is a pure Python library published to PyPI that abstracts multiple fine-tuning backends behind a unified Algorithm-Backend architecture. It enables users to perform Supervised Fine-Tuning (SFT), Orthogonal Subspace Fine-Tuning (OSFT), LoRA-based SFT, and Group Relative Policy Optimization (GRPO) for reinforcement learning from verifiable rewards -- all through simple convenience functions like `sft()`, `osft()`, `lora_sft()`, `lora_grpo()`, and `grpo()`.

The library delegates actual training execution to specialized backend frameworks: instructlab-training for SFT, rhai-innovation-mini-trainer for OSFT, Unsloth/TRL for LoRA-based training, OpenPipe ART for single-GPU GRPO, and verl (Volcano Engine RL) for distributed multi-GPU GRPO. It also provides GPU memory estimation, training time estimation, loss curve visualization, and reward function utilities for tool-call and binary reward evaluation.

training-hub is not a deployed service or operator -- it has no container images, no Kubernetes resources, no CRDs, and no deployment manifests. It is consumed as a Python dependency by other components or used directly by researchers and engineers in interactive environments (Jupyter notebooks, CLI scripts).

## Architectural Analysis

### Algorithm-Backend Design Pattern

training-hub's core architecture follows a Strategy pattern with a Registry. The `Algorithm` abstract base class defines `validate()` and `run()` methods, while `Backend` defines `train()`. The `AlgorithmRegistry` maintains class-level dictionaries mapping string names to Algorithm and Backend classes. The `create_algorithm()` factory function constructs an Algorithm instance with a specified Backend, enabling users to swap execution engines without changing their training code.

Each Algorithm class owns the full parameter specification as a dataclass-like `__init__` with comprehensive type validation. The `_validate_param_types()` method in OSFTAlgorithm demonstrates thorough runtime type checking that handles Union types, Optional types, list/dict generics, and Literal values -- providing clear error messages when parameters don't match expected types.

### Backend Diversity and Isolation

The five backends span very different execution models:
- **InstructLabTrainingSFTBackend** and **MiniTrainerOSFTBackend** delegate directly to their respective libraries via Python function calls.
- **UnslothLoRABackend** orchestrates model loading (with VLM detection), LoRA adapter application, dataset formatting, and TRL's SFTTrainer -- all in-process.
- **ARTLoRAGRPOBackend** is the most complex: it runs in a subprocess for vLLM isolation, handles engine lifecycle management, creates seed LoRA checkpoints for vLLM V1 compatibility, and supports three reward modes (tool_call, generic, custom).
- **VeRLLoRAGRPOBackend** constructs a verl CLI command with Hydra-style config overrides and runs it as a subprocess, then post-processes checkpoints to normalize the output format.

This diversity means training-hub acts more as an orchestration layer than a training framework -- it provides a consistent interface while delegating the heavy lifting to specialized frameworks.

### GRPO Multi-Turn Decomposition

A notable architectural feature in the ART GRPO backend is multi-turn trace decomposition (`_decompose_multiturn_traces()`). When training on multi-turn conversations, the backend decomposes an N-turn trace into N independent per-turn training samples, each preserving the conversation context up to that turn. This allows standard single-turn GRPO training to learn from multi-turn interactions without requiring architectural changes to the training loop.

### GPU Memory Estimation Architecture

The profiling module implements a hierarchy of VRAM estimators, each specialized for an algorithm:
- `BasicEstimator` (SFT): model params + gradients + AdamW optimizer states + activations + output buffer
- `OSFTEstimator`: Adjusts for `unfreeze_rank_ratio` (only a fraction of parameters require gradients) and SVD decomposition overhead
- `LoRAEstimator`: Accounts for low-rank adapter matrices (AB_params = weight_size_total * lora_r) being the only trainable parameters
- `QLoRAEstimator`: Models float4 quantized storage, bounded by pre-quantized model size

All estimators use a `_ModelStorage` CSV cache to avoid repeated HuggingFace Hub API calls for model metadata.

### Supply Chain Security Awareness

The pinning of litellm to `<1.82.7` (with an inline comment referencing the TeamPCP supply chain attack of 2026-03-24) demonstrates active supply chain risk management. This is a notable security practice for a library that transitively depends on many packages across the ML ecosystem.

### VLM (Vision-Language Model) Support

The Unsloth backend includes VLM detection logic (`_is_vlm_architecture()`) that distinguishes between text-only models (ForCausalLM) and vision-language models (ForConditionalGeneration). This enables automatic selection of `FastModel` vs `FastLanguageModel` for optimal loading, supporting architectures like LLaVA, Qwen-VL, and similar multimodal models.

### Limitations and Experimental Features

The timing estimator (`TimingEstimatorExperimental`) is explicitly marked as experimental. It works by training on a small subset of data (default 5%) and extrapolating, which introduces significant variance. The class emits a warning on initialization. The OSFT and SFT log parsers use fragile string parsing of training logs, which could break if log formats change in upstream dependencies.
