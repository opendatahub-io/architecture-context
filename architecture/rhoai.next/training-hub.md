## Purpose

**Short**: An algorithm-focused Python SDK providing a unified interface for LLM fine-tuning techniques including SFT, OSFT, LoRA, and GRPO reinforcement learning.

**Detailed**: training-hub is a Python library (not a Kubernetes operator or deployed service) that provides an algorithm-first abstraction layer for large language model training. It implements a pluggable Algorithm-Backend architecture pattern where training algorithms (SFT, OSFT, LoRA+SFT, LoRA+GRPO, GRPO) are decoupled from their execution backends (instructlab-training, mini-trainer, Unsloth, OpenPipe ART, verl). This separation allows users to swap backends without changing their training code, and enables the library to support a growing matrix of algorithm-backend combinations.

The library covers the full training lifecycle: data preprocessing (tokenization, chat template formatting, Parquet conversion), distributed training orchestration via torchrun (PyTorch DDP/FSDP), parameter-efficient fine-tuning (LoRA, QLoRA with 4-bit/8-bit quantization, DoRA, RSLoRA), reinforcement learning from verifiable rewards (GRPO/Dr. GRPO for tool-calling agents), GPU memory and timing profiling, experiment tracking integration (WandB, MLflow, TensorBoard), and training metrics visualization. It is designed to be imported as a Python package (`from training_hub import sft, osft, lora_sft, lora_grpo, grpo`) and called programmatically, or driven from scripts.

training-hub has no Kubernetes manifests, Dockerfiles, CRDs, services, or deployment artifacts. It is a pure library dependency consumed by higher-level platform components or user scripts that orchestrate LLM training workloads on GPU infrastructure.

## Architectural Analysis

### Algorithm-Backend Pattern

training-hub's core architectural innovation is a clean separation between _what_ training algorithm to use and _how_ to execute it. The `Algorithm` ABC defines the training contract (`train()`, `get_required_params()`, `get_optional_params()`), while `Backend` defines the execution contract (`execute_training()`). The `AlgorithmRegistry` provides a factory pattern (`create_algorithm()`) that wires these together at runtime.

This design enables several key capabilities:
- **Backend swappability**: The same SFT algorithm can run on instructlab-training or a future backend without API changes
- **Parameter validation**: Each algorithm defines its full parameter surface with types, enabling early validation before expensive GPU operations
- **Extensibility**: New algorithms or backends can be registered without modifying existing code

### PEFT Composition

Rather than creating separate algorithm classes for every combination of base algorithm + PEFT technique, training-hub uses a composition pattern via `PEFTExtender`. The `LoRAPEFTExtender` defines LoRA/QLoRA/DoRA parameters and applies defaults, and this is composed with base algorithms (e.g., `LoRASFTAlgorithm` combines SFT + LoRA). This avoids combinatorial explosion as new PEFT techniques are added.

### Distributed Training Orchestration

training-hub does not implement distributed training primitives directly. Instead, it orchestrates existing frameworks:
- **SFT/OSFT**: Delegates entirely to instructlab-training/mini-trainer, which manage torchrun internally
- **LoRA+SFT**: Uses Unsloth's built-in DDP support with HuggingFace Trainer
- **LoRA+GRPO (ART)**: Runs in a subprocess with co-located vLLM inference (single-GPU)
- **LoRA+GRPO (verl)**: Launches verl as a subprocess with FSDP for training and vLLM for rollouts (multi-GPU)

The `get_torchrun_params()` utility centralizes distributed training parameter resolution with a clear precedence hierarchy (args > environment > defaults), handling the mutual exclusivity of `master_addr` and `rdzv_endpoint` with proper warnings.

### GRPO Multi-Turn Trace Decomposition

The LoRA+GRPO algorithm includes a sophisticated `_decompose_multiturn_traces()` function that converts N-turn agent conversations into N per-turn training samples. Each sample contains the conversation history up to that turn, the tool definitions, and the reward for that specific turn. This is architecturally significant because it enables training on multi-turn tool-calling traces without requiring the RL framework to natively support multi-turn rollouts.

### Subprocess Isolation Strategy

Both GRPO backends (ART and verl) execute training in subprocesses rather than in-process. This provides:
- **GPU memory isolation**: The training process can be fully cleaned up after completion
- **vLLM lifecycle management**: The inference engine is started and stopped within the subprocess
- **Error isolation**: Crashes in the training loop don't bring down the orchestrator

The ART backend additionally creates a seed LoRA checkpoint (`_create_seed_lora_checkpoint()`) for vLLM V1 compatibility, as vLLM V1 requires eager adapter validation at startup.

### Profiling Suite Design

The memory estimation hierarchy (`BasicEstimator` > `OSFTEstimator` > `LoRAEstimator` > `QLoRAEstimator`) models GPU memory usage analytically rather than empirically. Each estimator computes memory components: model parameters, gradients, optimizer states, activations, and algorithm-specific overhead (SVD buffers for OSFT, low-rank matrices for LoRA, quantized weights for QLoRA). A CSV-based model parameter cache (`model_storage.csv`) avoids redundant model downloads for repeated estimations.
