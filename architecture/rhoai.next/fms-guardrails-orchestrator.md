## Purpose

**Short**: A REST API orchestrator that coordinates AI text generation with content safety guardrails by managing multiple detector services, chunker services, and LLM backends.

**Detailed**: The FMS Guardrails Orchestrator (internally named `fms-guardrails-orchestr8`) serves as middleware between clients and language models, providing content safety detection, filtering, and secure text generation. It sits in front of generation backends (TGIS, Caikit NLP, or OpenAI-compatible/vLLM endpoints) and routes requests through configurable detector and chunker services before and/or after text generation.

The orchestrator implements both legacy v1 APIs (classification-with-generation) and modern v2 APIs including text content detection, chat detection, context-aware document detection, and OpenAI-compatible chat/completion endpoints with integrated safety detection. It supports both unary and streaming request modes, with Server-Sent Events (SSE) and ND-JSON streaming for real-time content safety analysis.

Built on the Axum web framework with Tokio async runtime, the orchestrator uses a dual-server architecture: a primary guardrails server handling all API requests, and a separate health server for liveness/readiness probes. It features comprehensive OpenTelemetry integration for distributed tracing and metrics, configurable TLS/mTLS support via rustls, and gRPC load balancing through ginepro for upstream service connections.

## Architectural Analysis

The FMS Guardrails Orchestrator follows a sophisticated middleware architecture that decouples content safety enforcement from both client applications and LLM backends. The service acts as a transparent proxy-with-enrichment: clients send generation or detection requests, the orchestrator routes them through a configurable pipeline of chunking and detection stages, then returns augmented responses with safety annotations. This design allows the same detector infrastructure to be shared across multiple LLM backends (TGIS, Caikit NLP, vLLM) without modifying any of them.

The dual-protocol client strategy is noteworthy: gRPC is used for high-throughput, low-latency connections to generation and chunker services (with ginepro providing DNS-based load balancing and connection pooling), while HTTP/REST is used for detector services that have simpler request/response patterns. The `GenerationClient` abstraction transparently wraps either TGIS or NLP backends behind a unified interface, with retry logic (exponential backoff on 502-506 errors) implemented only for the NLP provider — suggesting TGIS is considered more stable or handles retries internally.

The streaming architecture deserves particular attention. For OpenAI-compatible endpoints, the orchestrator implements a complex detection pipeline: generation chunks arrive via SSE from the upstream service, are accumulated per-choice in a `CompletionState` (using `DashMap` for lock-free concurrent access), routed to chunker services, then fanned out to multiple detector services in parallel. A `DetectionBatchStream` with pluggable batcher implementations (`CompletionBatcher` for multi-choice responses, `MaxProcessedIndexBatcher` for single-stream content) ensures detections from multiple detectors are properly ordered and aggregated before being sent to the client. This batching is implemented via an actor pattern (`DetectionBatcherManager`) that serializes access to the batcher state while allowing concurrent detector streams.

A significant FIPS compliance gap exists: while the container image includes OpenSSL (satisfying check-payload validation), all TLS operations use rustls with the ring crypto backend. Ring is a high-quality cryptographic library but is not FIPS 140-2/3 validated. The `NoVerifier` implementation in `src/utils/tls.rs` that completely bypasses certificate verification when `insecure: true` is configured is also a security concern in production environments. The DISA STIG remediation script (`scripts/remediation-script.sh`) sets the system crypto policy to FIPS, but this only affects OpenSSL-based applications, not the rustls-based orchestrator binary.

The configuration system is YAML-based with extensive environment variable overrides via clap's `env` attribute. Named TLS configurations can be shared across services, and detector configurations include per-detector default thresholds, health check endpoints, and chunker assignments. The `passthrough_headers` feature with optional `X-Forwarded-Access-Token` to `Authorization: Bearer` rewriting suggests this service is designed to operate behind an authentication proxy (like kube-rbac-proxy or oauth-proxy) that injects identity headers.
