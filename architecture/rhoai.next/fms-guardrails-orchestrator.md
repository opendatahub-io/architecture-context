## Purpose

**Short**: A Rust-based REST API orchestrator that coordinates AI text generation with content safety guardrails by managing detector, chunker, and LLM backend services.

**Detailed**: The FMS Guardrails Orchestrator serves as middleware between clients and language models to enforce content safety. It receives user requests for text generation, applies configurable input and output detections via external detector services, optionally chunks text through chunker services, and coordinates with LLM backends (TGIS, Caikit NLP, or OpenAI-compatible) for text generation. The orchestrator evaluates detector responses against configurable thresholds to block, warn about, or pass through content.

The service exposes a dual-server architecture: a main guardrails HTTP API (default port 8033) with TLS/mTLS support, and a separate plaintext health server (default port 8034). It supports both unary and streaming request patterns, including Server-Sent Events (SSE) for streaming generation-with-detection and OpenAI-compatible chat/completions endpoints. Configuration is YAML-driven, allowing operators to define generation backends, chunker services, detector services (with per-detector thresholds and types), and named TLS profiles.

The orchestrator is built with Tokio, Axum (HTTP), and Tonic (gRPC client stubs). It communicates with detector services over HTTP and with generation/chunker services over gRPC, using protobuf-generated client stubs compiled at build time. OpenTelemetry integration provides distributed tracing and metrics export via OTLP (gRPC or HTTP).

## Architectural Analysis

The FMS Guardrails Orchestrator follows a clean dual-server architecture: a main guardrails API server with optional TLS/mTLS, and a separate plaintext health server. This separation ensures health probes work independently of TLS configuration and prevents health check failures from cascading into the main API path. The health server provides both a simple `/health` endpoint and a richer `/info` endpoint that probes all downstream service health.

The orchestrator uses a task-based handler pattern (`Handle<Task>` trait) with distinct task types for each endpoint (e.g., `ClassificationWithGenTask`, `ChatCompletionsDetectionTask`). Each task encapsulates request validation, detector orchestration (chunking, detection, threshold filtering), and response construction. This pattern cleanly separates routing from business logic and allows handler-specific streaming behavior (SSE for classification, NDJSON for content detection, SSE for OpenAI-compatible endpoints).

A notable design decision is the generation provider abstraction: the `GenerationClient` enum wraps either a `TgisClient` (gRPC) or `NlpClient` (gRPC) with unified generate/tokenize interfaces, while the `OpenAiClient` (HTTP) handles OpenAI-compatible backends separately. This dual-path design reflects the service's role bridging IBM's internal AI stack (TGIS/Caikit) with the OpenAI-compatible ecosystem (vLLM). The generation client includes retry logic with exponential backoff for transient errors (502, 503, 504).

The TLS implementation uses `rustls` with the `ring` crypto backend throughout, both for server TLS (`src/server/tls.rs`) and client TLS (`src/utils/tls.rs`). The client TLS supports an `insecure` mode with a `NoVerifier` that disables all certificate verification - this is a security-sensitive feature that should be used only in development. The `NoVerifier` supports legacy signature schemes (SHA1) which may not be appropriate for production deployments requiring strict crypto compliance.

Configuration is deserialized from a YAML file (`config/config.yaml`) with a flexible named TLS profile system: TLS configurations can be defined once and referenced by name across multiple services (chunkers, detectors, generation). The `api_token` field uses an environment variable indirection pattern - the config value names an environment variable, and the actual token is read from that variable at startup. This avoids embedding secrets in config files.

The OpenAI-compatible endpoint support (`/api/v2/chat/completions-detection` and `/api/v2/text/completions-detection`) is conditionally enabled only when the `openai` section is present in the configuration, allowing deployments that only use TGIS/NLP backends to avoid exposing unused endpoints. The OpenAI client also supports path prefixes for service routing compatibility with llm-d deployments.
