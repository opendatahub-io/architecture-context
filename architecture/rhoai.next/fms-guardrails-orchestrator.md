## Purpose

**Short**: A REST API orchestrator that coordinates AI text generation with content safety guardrails by managing detectors, chunkers, and LLM backends.

**Detailed**: The FMS Guardrails Orchestrator serves as middleware between clients and language models, providing content safety detection, filtering, and secure text generation. It exposes a REST API that accepts text generation and detection requests, then orchestrates calls to multiple downstream services—content detectors (HAP, PII, etc.), text chunkers, and generation backends (TGIS, caikit-nlp, or OpenAI-compatible endpoints like vLLM).

The orchestrator supports both unary and streaming workflows. For streaming, it implements detection batching—collecting text chunks from a generation stream, broadcasting them to multiple detectors concurrently, and aggregating results before returning them to the client. It supports the OpenAI chat completions API format with pluggable input/output detection, enabling integration with any OpenAI-compatible serving backend.

The service is designed for deployment alongside AI model serving infrastructure in RHOAI. It runs as a standalone binary with configurable TLS (including mTLS), OpenTelemetry observability, and health checking for all downstream services. Configuration is driven by a YAML file that declares which detectors, chunkers, and generation backends to use, along with their connection parameters.

## Architectural Analysis

The FMS Guardrails Orchestrator follows a classic middleware/gateway pattern, sitting between AI clients and the model-serving infrastructure. Its primary architectural contribution is the orchestration of multi-service detection pipelines—coordinating chunkers, detectors, and generation backends into coherent request flows. The design is heavily asynchronous, using Tokio's multi-threaded runtime with broadcast channels and detection batchers to maximize concurrency when multiple detectors must evaluate the same text.

A notable architectural decision is the separation of the health server (port 8034) from the guardrails API server (port 8033). The health server runs on a dedicated plaintext listener without TLS, enabling Kubernetes health probes to work regardless of TLS configuration. The guardrails server optionally supports TLS and mTLS via `rustls` with the `ring` crypto provider—a deliberate choice for performance, though it introduces a FIPS compliance consideration since `ring` is not FIPS-validated (see Security section).

The client abstraction layer (`src/clients/`) uses a trait-based design with polymorphic health checking. Each client type (detector, chunker, TGIS, NLP, OpenAI) implements the `Client` trait, enabling the orchestrator to manage heterogeneous downstream services uniformly. The `ClientMap` uses unsafe downcasting for type-safe storage of different client implementations—a pattern that trades compile-time safety for runtime flexibility. The gRPC clients use `ginepro` for DNS-based load balancing with configurable resolution strategies (eager/lazy).

The streaming architecture is particularly sophisticated. For chat/text completions with detection, the orchestrator splits detectors into "chunk detectors" (invoked incrementally as text arrives) and "whole document detectors" (invoked after the stream completes). A `DetectionBatcher` trait with two implementations (`CompletionBatcher` and `MaxProcessedIndexBatcher`) manages the aggregation of detection results across multiple concurrent detector streams. This allows the orchestrator to return detection results with minimal latency for chunk-based detectors while still supporting whole-document analysis.

The configuration system (`config.yaml`) provides a declarative way to wire up the service mesh of detectors, chunkers, and generation backends. TLS configuration supports named profiles that can be shared across multiple services, reducing configuration duplication. The `passthrough_headers` mechanism allows the orchestrator to forward authentication tokens and other context from the original client request to downstream services, with an optional rewrite of `X-Forwarded-Access-Token` to `Authorization: Bearer` format—a pattern designed for integration with OpenShift's authentication proxy infrastructure.

The build system uses Konflux with hermetic builds. The Tekton pipeline configuration shows multi-architecture support (x86_64, ppc64le, arm64, s390x) with cargo and RPM prefetch for build reproducibility. The `rpms.lock.yaml` and `Cargo.lock` files provide full dependency pinning. The orchestrator binary is built with release optimizations including LTO and symbol stripping, producing a minimal container image based on UBI 9 minimal.
