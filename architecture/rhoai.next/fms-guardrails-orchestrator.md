## Purpose

**Short**: A REST API orchestrator that coordinates AI text generation with content safety guardrails by managing detector, chunker, and LLM backend services.

**Detailed**: The FMS Guardrails Orchestrator serves as middleware between clients and language models to enforce content safety. It receives text generation and chat completion requests, orchestrates parallel calls to multiple downstream services (text generation backends, content detectors, and text chunkers), and applies guardrail policies to both input and output text. When harmful content is detected above configured thresholds, it blocks or filters responses before returning them to clients.

The orchestrator supports multiple generation backends including TGIS (Text Generation Inference Server) via gRPC, Caikit NLP via gRPC, and OpenAI-compatible APIs (including vLLM) via HTTP. Detectors are HTTP services that analyze text for content safety issues (hate speech, profanity, etc.), while chunkers are gRPC services that tokenize text for detector processing. All downstream service connections support TLS/mTLS, and the orchestrator provides comprehensive OpenTelemetry-based observability with distributed tracing and metrics.

The service runs as a single Rust binary (`fms-guardrails-orchestr8`) built on the Axum HTTP framework with Tokio async runtime. It exposes two HTTP servers: a main API server (default port 8033) for guardrails endpoints and a separate health server (default port 8034) for liveness/readiness checks. The dual-server pattern allows health probes to remain responsive even under heavy API load.

## Architectural Analysis

The FMS Guardrails Orchestrator implements a sophisticated middleware pattern for AI safety, positioned between API consumers and LLM backends. Its core design philosophy is to be protocol-agnostic toward downstream services while exposing a unified REST API surface. The orchestrator speaks gRPC to generation and chunking services (TGIS, Caikit NLP, Caikit Chunkers) and HTTP to detectors and OpenAI-compatible APIs, abstracting this complexity behind a consistent REST interface for clients.

The orchestration engine makes heavy use of Tokio's async concurrency primitives. Detection tasks are spawned in parallel using `futures::stream::iter` with `buffer_unordered`, bounded by configurable concurrency limits (`detector_concurrent_requests` and `chunker_concurrent_requests`, both defaulting to 5). This means multiple detectors can analyze content simultaneously, with results aggregated and filtered by score thresholds. The streaming pipeline is particularly sophisticated: for streaming completions, the orchestrator uses a broadcast channel pattern where a single incoming generation stream is fanned out to multiple detector workers. Each detector receives chunks via bidirectional gRPC streaming to the chunker, processes them, and feeds results into a `DetectionBatchStream` that uses an actor-pattern batcher (`DetectionBatcherManager`) to coordinate ordered batch release across detectors and choices.

The dual-server architecture (API on port 8033, health on port 8034) is a deliberate design choice that isolates health checking from API load. Both servers share TLS configuration from CLI arguments, but the health server runs independently, ensuring Kubernetes liveness/readiness probes remain responsive during high-throughput detection workloads. The `/info` endpoint provides deep health checking by probing all configured downstream services (generation, chunker, detector, OpenAI) and caching results.

TLS configuration follows a name-reference pattern: TLS profiles are defined once in the `tls` section of `config.yaml` with named IDs (e.g., "caikit", "detector"), and individual service configurations reference these by name. This avoids certificate path duplication across services. Client-side mTLS is built on rustls with hyper-rustls connectors for HTTP and tonic's native TLS for gRPC. The server supports optional mTLS with client CA verification. An "insecure" mode is available for development where TLS certificate verification can be disabled per downstream service.

The codebase reveals careful attention to observability: every HTTP and gRPC client call is instrumented with OpenTelemetry spans, W3C traceparent/tracestate headers are propagated to all downstream services, and metrics are emitted for request counts, latencies, error rates (by HTTP status class), and streaming durations. This end-to-end distributed tracing is critical for debugging multi-service guardrail pipelines where a single user request may fan out to 5+ downstream service calls.
