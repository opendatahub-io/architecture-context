## Purpose

**Short**: A REST API orchestrator that coordinates AI text generation with content safety guardrails, acting as middleware between clients and language models.

**Detailed**: The FMS Guardrails Orchestrator serves as an intermediary service that intercepts client requests to large language models (LLMs) and applies configurable content safety detection before and/or after text generation. It manages connections to multiple downstream services — generation backends (TGIS via gRPC, caikit-nlp via gRPC, or OpenAI-compatible via HTTP), content detector services (via HTTP), and text chunker services (via gRPC) — through unified REST APIs.

The orchestrator exposes two generations of API endpoints: v1 endpoints for classification-with-text-generation workflows and v2 endpoints for more granular detection scenarios including content detection, chat detection, context document detection, generation-with-detection, and streaming content detection. When configured with an OpenAI-compatible backend, it also exposes chat completions and text completions detection endpoints that wrap the standard OpenAI API with guardrail detection capabilities.

The service operates with a dual-server architecture: a main guardrails API server (default port 8033) that handles all detection and generation requests with optional TLS/mTLS, and a separate health server (default port 8034) that runs over plaintext HTTP for health checks and service info queries. Both servers support graceful shutdown. The orchestrator is built with tokio for async runtime, axum for HTTP routing, and tonic for gRPC client communication, with comprehensive OpenTelemetry integration for distributed tracing and metrics export.

## Architectural Analysis

The FMS Guardrails Orchestrator follows a clean orchestration pattern where a lightweight Rust middleware sits between API consumers and multiple backend services. The dual-server architecture (guardrails on 8033, health on 8034) is a thoughtful design decision — it ensures health checks remain available over plaintext even when the main API enforces TLS, which is critical for Kubernetes liveness/readiness probes that may not have TLS credentials.

The client architecture is notable for its polymorphism: the `GenerationClient` enum wraps both TGIS (gRPC) and caikit-nlp (gRPC) behind a unified interface, while detector services use HTTP and chunkers use gRPC. The `ClientMap` provides a type-erased container that can hold different client types, retrieved by downcast at runtime. This allows dynamic configuration at startup based on the YAML config file, but introduces some runtime type-unsafety that is mitigated by careful construction in `create_clients()`.

A significant architectural concern is the TLS implementation. The service uses rustls with the ring crypto backend for all TLS operations — both server-side and client-side. While this provides high performance and memory safety, the ring library is not FIPS-validated. The container image includes OpenSSL (satisfying container-level checks), but the application binary does not use it for cryptographic operations. Additionally, the `NoVerifier` struct in `src/utils/tls.rs` completely disables certificate verification when `insecure: true` is set in per-service TLS config, which should be audited for production deployments. The X-Forwarded-Access-Token rewriting (converting to Authorization: Bearer) in `src/server/routes.rs:470-480` is a useful integration pattern but must be carefully secured to prevent token injection.

The OpenTelemetry integration is comprehensive, covering both traces and metrics with configurable OTLP export protocols (gRPC or HTTP) and endpoints. The tracing spans propagate `traceparent` headers across all downstream gRPC and HTTP calls, enabling end-to-end distributed tracing through the detection and generation pipeline. Custom metrics are emitted at multiple levels: incoming request counts, client response counts (with separate counters for 2xx, 4xx, 5xx), request durations, and streaming response metrics.

The streaming architecture supports multiple patterns: SSE (Server-Sent Events) for classification and chat completions streaming, ND-JSON for content detection streaming, and gRPC server streaming for generation backends. The streaming detection pipeline uses tokio channels (`mpsc`) to decouple the response stream consumption from the output formatting, allowing concurrent processing of detection results as generation tokens arrive.
