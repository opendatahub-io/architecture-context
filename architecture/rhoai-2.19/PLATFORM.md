## Platform Purpose

Red Hat OpenShift AI (RHOAI) is an enterprise MLOps platform built on OpenShift Container Platform that provides a comprehensive set of tools for the full AI/ML lifecycle. It enables data scientists and ML engineers to develop, train, serve, and monitor machine learning models at scale.

RHOAI 2.19 integrates 36 upstream and downstream components into a unified platform managed by the `rhods-operator`, which uses OLM (Operator Lifecycle Manager) for installation and lifecycle management. The platform is assembled via a central build configuration repository (RHOAI-Build-Config) that produces operator bundles and FBC (File-Based Catalog) fragments for distribution through the Red Hat catalog.

The platform provides:
- **Interactive Development**: JupyterLab, code-server, and RStudio workbench environments with GPU support (NVIDIA CUDA, AMD ROCm, Intel Gaudi)
- **Model Training**: Distributed training via Kubeflow Training Operator with PyTorch, TensorFlow, MPI, XGBoost, PaddlePaddle, and JAX support
- **Model Serving**: Single-model (KServe) and multi-model (ModelMesh) serving with multiple inference runtimes (vLLM, TGIS, OVMS, Caikit)
- **ML Pipelines**: Kubeflow Pipelines v2 with Argo Workflows backend for workflow orchestration
- **Distributed Computing**: Ray clusters via KubeRay with CodeFlare integration and Kueue job queuing
- **Model Management**: Model Registry for versioning and tracking, Feast feature store
- **AI Safety**: TrustyAI for fairness/explainability, FMS Guardrails Orchestrator for content safety, LM Evaluation Harness for benchmarking
- **LLM Fine-Tuning**: InstructLab pipeline for synthetic data generation and distributed training

## Data Flow Patterns

### Pattern 1: Model Serving (KServe Serverless)

```
Client → OpenShift Route → Istio Gateway → Knative Activator → KServe Pod
  ├── oauth-proxy sidecar (auth) → OpenShift OAuth
  ├── queue-proxy sidecar (Knative)
  └── Inference Runtime Container (vLLM/TGIS/OVMS/Caikit)
       └── Model weights from S3/PVC
```

### Pattern 2: Model Serving (KServe RawDeployment)

```
Client → OpenShift Route → KServe Pod
  ├── oauth-proxy sidecar (auth)
  └── Inference Runtime Container
       └── Model weights from S3/PVC
```

### Pattern 3: Model Serving (ModelMesh)

```
Client → OpenShift Route → modelmesh Pod (headless service)
  ├── oauth-proxy sidecar (auth, port 8443)
  ├── rest-proxy sidecar (REST→gRPC, port 8008)
  ├── modelmesh sidecar (gRPC, port 8033) → etcd coordination
  ├── modelmesh-runtime-adapter sidecar (puller + adapter)
  └── Inference Runtime Container (OVMS/Triton/MLServer)
       └── Model weights pulled by runtime-adapter from S3/PVC
```

### Pattern 4: Pipeline Execution (Data Science Pipelines)

```
User → RHOAI Dashboard → DSP API Server (REST 8888)
  → Argo Workflow Controller → Pipeline Task Pods
     ├── Driver container (pipeline logic)
     ├── Launcher container (artifact management)
     └── User container (pipeline step execution)
         └── Artifacts stored in S3/MinIO
  → ML Metadata (gRPC 8080) → MariaDB (3306)
```

### Pattern 5: Notebook Workbench

```
User → OpenShift Route → oauth-proxy sidecar → Notebook Pod
  └── JupyterLab/code-server/RStudio container (8888/8080/8787)
       ├── GPU access via NVIDIA/AMD/Intel device plugin
       └── PVC for user data persistence
```

### Pattern 6: Distributed Training

```
User → Kubeflow Training Operator → PyTorchJob CR
  → Master Pod ←→ Worker Pods (NCCL/Gloo communication)
       ├── GPU access via device plugin
       ├── Shared storage via PVC (ReadWriteMany)
       └── Kueue admission (queueing/quota)
```

### Pattern 7: InstructLab Fine-Tuning

```
DSPA → InstructLab Pipeline (KFP v2)
  1. Prerequisites check → K8s API + Teacher/Judge model servers
  2. SDG → Git clone taxonomy → Teacher model (KServe) → Synthetic data
  3. Training → PyTorchJob (multi-node multi-GPU) via Training Operator
  4. Evaluation → Local vLLM + Judge model (KServe) → MT-Bench/MMLU scores
  5. Upload → OCI Registry (ModelCar) + Model Registry
```

### Pattern 8: Guardrails Content Safety

```
Client → Guardrails Orchestrator (8033)
  → Input chunking (gRPC → Chunker server)
  → Input detection (HTTP → Detector servers)
  → Text generation (gRPC → TGIS/Caikit or HTTP → vLLM)
  → Output detection (HTTP → Detector servers)
  → Response to client
```
