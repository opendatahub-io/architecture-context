RBAC Summary:
- No RBAC definitions - vLLM is a containerized application, not a Kubernetes operator
- RBAC is managed by KServe or the deploying operator

Container Security:
- Run as UID: 2000 (non-root)
- Run as GID: 0 (root group, for OpenShift arbitrary UID support)
- Home directory: /home/vllm (writable)
- umask: 002 (group-writable)

Secrets:
- HuggingFace Token (Opaque, env var / mounted file) - Model downloads, User-provisioned, No auto-rotate
- S3 Credentials (Opaque, env vars S3_ACCESS_KEY_ID/S3_SECRET_ACCESS_KEY) - S3 model storage, User-provisioned, No auto-rotate
- SSL Key/Cert (kubernetes.io/tls) - TLS termination, User/cert-manager, Auto-rotate via --enable-ssl-refresh
- API Key (Opaque, env var VLLM_API_KEY or --api-key) - Bearer token auth for /v1/*, User-provisioned, No auto-rotate

Authentication & Authorization:
- /v1/* endpoints: Bearer Token (optional, enforced when VLLM_API_KEY is set)
- /health, /ping, /metrics, /version: Always publicly accessible (no auth)
- /tokenize, /detokenize: No auth enforcement (not under /v1/ path)

Internal IPC:
- ZeroMQ IPC: Unix domain socket (ipc://) - Frontend to backend engine
- NCCL: GPU Direct / TCP - Multi-GPU tensor parallelism
- PyTorch distributed: TCP (gloo/nccl) - Worker process coordination
