## Purpose
**Short**: KubeRay Operator manages the lifecycle of Ray clusters, jobs, and serving deployments on Kubernetes/OpenShift.

**Detailed**: KubeRay is a Kubernetes operator that simplifies the deployment and management of Ray applications on Kubernetes. It provides three Custom Resource Definitions (CRDs) -- RayCluster, RayJob, and RayService -- that allow users to declaratively manage Ray clusters and workloads.

The operator handles the full lifecycle of Ray clusters including creating head and worker pods, managing head services, serve services, headless worker services, and optional ingress (Kubernetes Ingress or OpenShift Routes). It supports autoscaling through integration with the Ray Autoscaler, batch scheduling via Volcano and Yunikorn schedulers, and fault tolerance through GCS (Global Control Store) with Redis-backed storage.

For RHOAI, the operator is deployed within the `opendatahub` namespace and uses OpenShift SecurityContextConstraints for pod security. It auto-detects OpenShift environments and creates Route resources instead of Kubernetes Ingress resources when ingress is enabled for a RayCluster. The operator is built using UBI8 base images and supports disconnected/air-gapped environments.

**Dynamic Services Created by Controllers**:
- **Head Service**: Created per RayCluster by the RayCluster controller. Default ports include GCS (6379), Dashboard (8265), Client (10001), Metrics (8080), Serve (8000). Service type is configurable via `spec.headGroupSpec.serviceType`.
- **Serve Service**: Created per RayCluster when `ray.io/enable-serve-service=true` annotation is set, or per RayService by the RayService controller. Exposes the Ray Serve port (default 8000).
- **Headless Worker Service**: Created for multi-host worker groups to enable DNS-based pod discovery.

### Ingress

| Name | Type | Hosts | Port | Protocol | Encryption | TLS Mode | Exposure |
|------|------|-------|------|----------|------------|----------|----------|
| {cluster}-ingress (dynamic) | Kubernetes Ingress | configurable | 8265/TCP | HTTP | None (configurable) | N/A | External (vanilla K8s) |
| {cluster}-head-route (dynamic) | OpenShift Route | auto-generated | 8265/TCP | HTTP | edge (OpenShift default) | edge | External (OpenShift) |

**Controller-managed Ingress**:
- When `spec.headGroupSpec.enableIngress=true`, the RayCluster controller auto-detects the cluster type:
  - **OpenShift**: Creates an OpenShift `Route` CR targeting the Ray Dashboard port (8265) on the head service
  - **Vanilla Kubernetes**: Creates a Kubernetes `Ingress` CR targeting the Ray Dashboard port
- The `USE_INGRESS_ON_OPENSHIFT=true` env var forces Kubernetes Ingress creation even on OpenShift
- Ingress class is configured via the `kubernetes.io/ingress.class` annotation on the RayCluster CR
