## Purpose
**Short**: Kubeflow Training Operator manages distributed machine learning training jobs on Kubernetes, supporting PyTorch, TensorFlow, MPI, XGBoost, PaddlePaddle, and JAX frameworks.

**Detailed**: The Kubeflow Training Operator (KFTO) is a Kubernetes-native operator that enables fine-tuning and scalable distributed training of AI/ML models. It provides Custom Resource Definitions (CRDs) for each supported ML framework: PyTorchJob, TFJob, MPIJob, XGBoostJob, PaddleJob, and JAXJob. Each CRD allows users to declaratively specify distributed training configurations including replica counts, resource requirements, and framework-specific settings like elastic training policies.

The operator runs as a single controller-runtime-based manager that registers per-framework reconcilers and validating webhooks. Each reconciler manages the lifecycle of training jobs by creating and monitoring Pods and headless Services for inter-worker communication. The PyTorch controller additionally supports elastic training with HorizontalPodAutoscaler integration and dynamically creates NetworkPolicy resources to isolate training job network traffic. The operator also supports gang scheduling through Volcano or scheduler-plugins PodGroups for coordinated pod scheduling.

The RHOAI distribution deploys the operator in the `opendatahub` namespace with a `kubeflow-` name prefix. It uses self-managed TLS certificates for webhook serving via the open-policy-agent cert-controller library, and includes Prometheus PodMonitor integration for metrics collection. A v2alpha1 API (TrainJob, TrainingRuntime, ClusterTrainingRuntime) is also present in manifests but represents the next-generation API surface currently under development.

### Dynamically Created Services (per training job)

The operator dynamically creates headless ClusterIP Services for each training job replica to enable inter-worker communication via DNS-based discovery. These services are created by the common `JobController.ReconcileJobs()` logic and are owned by the training job CR.

| Service Name Pattern | Type | Port | Protocol | Purpose |
|---------------------|------|------|----------|---------|
| `{job-name}-{replica-type}-{index}` | ClusterIP (headless) | Framework-specific | TCP | Inter-worker communication for distributed training |

### Network Policies (Dynamically Created)

The PyTorchJob controller dynamically creates NetworkPolicy resources per PyTorchJob to restrict network traffic:

| Policy | Ingress Rules | Purpose |
|--------|---------------|---------|
| `{pytorchjob-name}` | Allow from pods with same `training.kubeflow.org/job-name` label; Allow from `openshift-monitoring` namespace on port 8080/TCP | Isolates PyTorch training pods; allows monitoring |
