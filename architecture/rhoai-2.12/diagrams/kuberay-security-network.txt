═══════════════════════════════════════════════════════════════════════════
Security Considerations
═══════════════════════════════════════════════════════════════════════════

⚠️  HIGH RISK:
1. Ray GCS (6379/TCP) - UNENCRYPTED, NO AUTH
   - Internal Ray communication is plaintext
   - Recommendation: Use NetworkPolicies to restrict access
   - Never expose port 6379 externally

2. Ray Dashboard (8265/TCP) - UNENCRYPTED, NO AUTH
   - Web UI has no authentication by default
   - Recommendation: Place behind OAuth2/OIDC proxy
   - Restrict access via NetworkPolicies or ingress auth

3. Ray Serve (8000/TCP) - UNENCRYPTED, NO AUTH
   - Model serving endpoints have no auth by default
   - Recommendation: Implement application-level authentication
   - Use TLS at ingress layer

4. Ray Client (10001/TCP) - UNENCRYPTED, NO AUTH
   - Remote job submission has no auth
   - Recommendation: Restrict via NetworkPolicies
   - Only allow from trusted namespaces/pods

✅ SECURED:
1. Kubernetes API communication - TLS 1.3, ServiceAccount auth
2. Webhook communication - TLS 1.2+, K8s API auth
3. Container image pulls - TLS 1.2+, Pull Secret auth
4. Operator pod - Non-root (UID 65532), no privilege escalation

📋 RECOMMENDATIONS:
1. Enable NetworkPolicies to restrict Ray cluster communication
2. Deploy OAuth2 proxy for Ray Dashboard external access
3. Implement application-level auth for Ray Serve endpoints
4. Use TLS termination at ingress for all external endpoints
5. Rotate webhook TLS certificates regularly (cert-manager)
6. Monitor RBAC permissions and audit operator actions
7. Use FIPS-compliant images for compliance environments
8. Configure resource limits to prevent resource exhaustion
9. Enable leader election for operator high availability
10. Regularly update to latest RHOAI version for security patches
