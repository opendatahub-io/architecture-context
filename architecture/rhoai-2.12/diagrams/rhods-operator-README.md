## Component Overview

The RHODS Operator is the primary orchestration component for Red Hat OpenShift AI (RHOAI). It manages:
- **DataScienceCluster (DSC)**: Component deployment and lifecycle management
- **DSCInitialization (DSCI)**: Platform-wide configuration (Service Mesh, monitoring, CA bundles)
- **10 managed components**: Dashboard, KServe, ModelMesh, Data Science Pipelines, Workbenches, CodeFlare, Ray, Kueue, Training Operator, TrustyAI

## Key Features

- **Declarative Management**: Uses CRDs to define desired platform state
- **Manifest Fetching**: Pulls component manifests from GitHub repositories via Kustomize
- **Service Mesh Integration**: Configures Istio, Authorino, and mTLS for components
- **Monitoring Integration**: Deploys Prometheus, Alertmanager, and creates ServiceMonitors
- **Secret Management**: Generates component secrets and trusted CA bundles
- **RBAC**: Comprehensive cluster-wide permissions for managing all platform resources

## Network Architecture

### Exposed Endpoints
- **Webhook Server**: 9443/TCP HTTPS (mTLS) - Admission control for CRs
- **Metrics Endpoint**: 8443/TCP HTTPS - Prometheus metrics (via kube-rbac-proxy)
- **Health Probes**: 8081/TCP HTTP - Liveness (/healthz) and readiness (/readyz)

### Egress Connections
- **Kubernetes API Server**: 6443/TCP HTTPS (ServiceAccount Token)
- **GitHub**: 443/TCP HTTPS (fetch component manifests)

### Managed Namespaces
- `redhat-ods-operator` - Operator deployment
- `redhat-ods-applications` - Application components (Dashboard, Workbenches)
- `redhat-ods-monitoring` - Monitoring stack (Prometheus, Alertmanager)
- `istio-system` - Service Mesh Control Plane (conditional)
- `redhat-ods-applications-auth-provider` - Authorino instance (conditional)
- `knative-serving` - Knative Serving for KServe (conditional)

## Security

### Authentication & Authorization
- **API Server to Operator**: mTLS (API Server validates webhook cert)
- **Operator to API Server**: ServiceAccount Token (controller-manager SA)
- **Monitoring to Operator**: Bearer Token via kube-rbac-proxy
- **User Access**: OpenShift OAuth (for monitoring routes)

### RBAC Summary
The `controller-manager` ServiceAccount has cluster-wide permissions for:
- **ODH CRDs**: DataScienceCluster, DSCInitialization, FeatureTracker
- **Core Resources**: Namespaces, ConfigMaps, Secrets, ServiceAccounts, Services, Pods
- **Workload Resources**: Deployments, StatefulSets, Jobs, CronJobs
- **RBAC**: ClusterRoles, ClusterRoleBindings, Roles, RoleBindings
- **Networking**: Routes, NetworkPolicies, Ingresses
- **Service Mesh**: ServiceMeshControlPlanes, ServiceMeshMembers, Gateways, AuthorizationPolicies
- **Monitoring**: ServiceMonitors, PodMonitors, PrometheusRules
- **Component CRDs**: InferenceServices, KnativeServings, Tekton resources, Ray resources, etc.

### Secrets Management
- **webhook-server-cert**: TLS cert for webhooks (cert-manager/OLM)
- **prometheus-tls**: TLS cert for Prometheus (OpenShift serving-cert, auto-rotates)
- **odh-trusted-ca-bundle**: Cluster and custom CA certificates (auto-rotates on cluster CA changes)
- **Component secrets**: Generated by SecretGenerator controller (DB passwords, API tokens)

## Component Details

### Controllers
1. **DataScienceCluster Controller**: Manages component deployment based on DSC CR specifications
2. **DSCInitialization Controller**: Configures platform-wide settings (namespaces, Service Mesh, monitoring, CA bundles)
3. **SecretGenerator Controller**: Generates component-specific secrets
4. **CertConfigmapGenerator Controller**: Creates and distributes trusted CA bundles

### Component Reconcilers
The operator includes dedicated reconcilers for each managed component:
- Dashboard, KServe, ModelMesh, Data Science Pipelines
- Workbenches, CodeFlare, Ray, Kueue
- Training Operator, TrustyAI

Each reconciler:
- Fetches manifests from GitHub (via Manifest Fetcher)
- Applies Kustomize overlays
- Creates RBAC resources
- Configures monitoring (ServiceMonitors, PodMonitors)
- Updates FeatureTracker CRs for garbage collection

### Manifest Sources
Component manifests are fetched from upstream repositories:
- `opendatahub-io/odh-dashboard`
- `opendatahub-io/kserve`
- `opendatahub-io/modelmesh-serving`
- `opendatahub-io/data-science-pipelines-operator`
- `opendatahub-io/notebooks`
- `project-codeflare/codeflare-operator`
- `ray-project/kuberay`
- `kubernetes-sigs/kueue`
- `kubeflow/training-operator`
- `trustyai-explainability/trustyai-service-operator`
