## Purpose

**Short**: CPU-only build of vLLM, a high-throughput inference engine for Large Language Models, optimized for serving without GPU hardware.

**Detailed**: vLLM CPU is a downstream Red Hat fork of the vLLM inference engine, specifically built to run LLM inference on CPU architectures (ppc64le, s390x, and x86_64). It provides an OpenAI-compatible REST API server for serving large language models using techniques like PagedAttention for efficient KV cache memory management, continuous batching for high throughput, and support for tensor parallelism across CPU cores.

The CPU variant is built from a single Konflux Dockerfile (`Dockerfile.konflux.cpu`) that produces a multi-stage image. The final image contains two entry points: the primary `vllm-openai` stage that runs the OpenAI-compatible API server, and the `vllm-grpc-adapter` stage that wraps vLLM with the `vllm-tgis-adapter` to expose a gRPC interface compatible with the TGIS (Text Generation Inference Server) protocol used by KServe's ServingRuntime. This adapter enables vLLM to serve as a drop-in replacement for TGIS in RHOAI's model serving stack.

The component integrates with the RHOAI model serving infrastructure through KServe ServingRuntimes. Models are loaded from Hugging Face Hub or local storage, and inference requests are handled via the FastAPI-based OpenAI API or the gRPC TGIS adapter layer.

## Architectural Analysis

vLLM CPU is a specialized variant of the vLLM inference engine that strips away GPU dependencies and compiles CPU-optimized attention kernels. The architecture centers on a dual-engine design: V0 (synchronous, legacy) and V1 (asynchronous, default), with V1 enabled by default via `VLLM_USE_V1=True`. The V1 engine uses Python's asyncio event loop with per-request async generators, enabling efficient streaming of token-by-token responses through Server-Sent Events. The engine's PagedAttention mechanism manages KV cache in fixed-size blocks (default 16 tokens), which is critical for CPU inference where memory is plentiful but slower than GPU HBM.

The CPU kernel layer (`csrc/cpu/`) provides architecture-specific implementations for x86 (AVX/AVX-512 via `cpu_types_x86.hpp`), ARM (Neoverse via `cpu_types_arm.hpp`), IBM POWER (VSX via `cpu_types_vsx.hpp`), and IBM Z (VXE via `cpu_types_vxe.hpp`). The Konflux Dockerfile builds exclusively for ppc64le and s390x platforms (per `.tekton/odh-vllm-cpu-v3-0-push.yaml` `build-platforms`), with x86_64 builds handled by the separate `vllm` (GPU) repository. This architecture split means the CPU image targets mainframe and POWER systems where GPUs are unavailable or impractical.

The multi-stage Dockerfile design is architecturally significant: the `vllm-openai` stage provides the OpenAI-compatible REST API (FastAPI/uvicorn), while the `vllm-grpc-adapter` stage extends it by installing the `vllm-tgis-adapter` package (from opendatahub-io/vllm-tgis-adapter). This adapter translates TGIS gRPC protocol calls (used by KServe's ServingRuntime) into vLLM engine calls. The two stages share the same base image, with the gRPC adapter adding only the adapter wheel and environment variables (`GRPC_PORT=8033`, `PORT=8000`). This design allows a single container build to produce images for both protocol variants.

The build process is notably complex for CPU targets: the `build_vllm_ppc64le.sh` and `build_vllm_s390x.sh` scripts install development toolchains, compile OpenBLAS and LAPACK from source, install Rust (for tokenizers compilation), and build the vLLM wheel with architecture-specific optimizations. The ppc64le build uses gcc-toolset-14 for modern C++ support, while s390x requires source-built LLVM for certain compiler features. This complexity stems from the limited pre-built binary availability for these architectures on PyPI.

Security follows a defense-in-depth model when deployed on RHOAI: the vLLM server itself provides optional Bearer token authentication (AuthenticationMiddleware, only for `/v1/*` paths), while the platform adds kube-rbac-proxy sidecars for Kubernetes-native RBAC enforcement. TLS termination is typically handled by the platform's Gateway/Route infrastructure, though vLLM supports native TLS with certificate hot-reload (SSLCertRefresher using the `watchfiles` library for async file monitoring). The permissive default CORS configuration (`*` for origins/methods/headers) is appropriate for cluster-internal deployment where ingress controls are handled at the platform level.
