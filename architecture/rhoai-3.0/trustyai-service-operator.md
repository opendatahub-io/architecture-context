## Purpose

**Short**: A multi-controller Kubernetes operator that manages TrustyAI explainability services, LM evaluation jobs, and guardrails orchestration for AI/ML workloads on Red Hat OpenShift AI.

**Detailed**: The TrustyAI Service Operator is a single Go binary containing four independently-enableable controllers that manage the TrustyAI platform's three core capabilities: model explainability (TrustyAIService), language model evaluation (LMEvalJob), and AI guardrails orchestration (GuardrailsOrchestrator). Each controller is registered via a service registry pattern and enabled at startup via the `--enable-services` flag, allowing distribution-specific configurations — RHOAI enables all four services (TAS, LMES, JOB_MGR, GORCH), while ODH enables a subset with optional Kueue integration.

The operator watches three custom resource definitions under the `trustyai.opendatahub.io` API group and dynamically creates Deployments, Services, Routes, ConfigMaps, ServiceAccounts, ServiceMonitors, and PVCs. It integrates with KServe (InferenceService patching), Kueue (job queue management), Istio (VirtualService/DestinationRule for service mesh), Prometheus (ServiceMonitor for metrics collection), and OpenShift (Routes, serving certificates, kube-rbac-proxy authentication). The operator uses a kube-rbac-proxy sidecar pattern for both TrustyAIService and GuardrailsOrchestrator workloads to enforce RBAC-based authentication with FIPS 140-2 compliant TLS cipher suites.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| TrustyAIService Controller (TAS) | Go Controller | Manages TrustyAI explainability service deployments with kube-rbac-proxy sidecars, KServe/ModelMesh inference logging, and PVC/database storage |
| LMEvalJob Controller (LMES) | Go Controller | Orchestrates language model evaluation jobs using a state machine with pod-based execution and driver binary for progress monitoring |
| Job Manager Controller (JOB_MGR) | Go Controller | Wraps LMEvalJob with Kueue job framework for queue-based scheduling and resource management |
| GuardrailsOrchestrator Controller (GORCH) | Go Controller | Manages guardrails orchestrator deployments with optional gateway sidecar, built-in detectors, and auto-configuration from KServe InferenceServices |
| LMES Driver | Go CLI Binary | Execution engine that runs inside evaluation pods, manages lm-eval process lifecycle, reports progress via HTTP API, handles custom tasks and S3 downloads |

## Architectural Analysis

The TrustyAI Service Operator employs a modular multi-controller architecture where each controller is independently registerable via a service registry pattern (`controllers/controllers.go`). This design allows the same binary to serve different deployment profiles — RHOAI enables all four services (TAS, LMES, JOB_MGR, GORCH), while ODH might enable a subset. The registry pattern uses Go `init()` functions in separate files (`controllers/tas.go`, `controllers/lmes.go`, `controllers/gorch.go`, `controllers/job_mgr.go`) to register controllers, avoiding import cycles and enabling clean separation of concerns.

The LMES controller implements a sophisticated 7-state machine (New → Scheduled → Running → Complete, with Suspended and Cancelled branches) for job lifecycle management. Rather than creating Kubernetes Jobs, it creates bare Pods with an init container that copies the driver binary from a dedicated container image into a shared EmptyDir volume. The main container then runs the driver, which forks `python -m lm_eval` and monitors its progress by parsing tqdm output on stderr. The controller communicates with the driver via `kubectl exec` (SPDY protocol) rather than a network service, keeping the driver's HTTP API (port 18080) bound to localhost only. This design avoids network exposure while enabling real-time progress reporting. The input validation layer (`controllers/lmes/validation.go`) is notably thorough — it blocks environment variable injection by protecting 8 critical variables (TRUST_REMOTE_CODE, HF_DATASETS_TRUST_REMOTE_CODE, etc.) and validates all user inputs against regex patterns to prevent shell injection via model names, task names, Git URLs, and S3 paths.

The GuardrailsOrchestrator controller stands out for its auto-configuration capability. When `spec.autoConfig` is set, it discovers KServe InferenceServices in the namespace — identifying the generation service (the model to be guardrailed) and detector services (via a label match, defaulting to `trustyai/guardrails: true`). It then generates orchestrator and gateway ConfigMaps containing the service hostnames, ports, and TLS configurations. A hash-based change detection mechanism (`shouldRegenerateAutoConfig()`) avoids unnecessary regeneration when the underlying InferenceServices haven't changed. The controller also auto-detects TLS by checking InferenceService URL schemes and searching for `{isvc-name}-predictor-serving-cert` secrets, then dynamically mounts these secrets into the orchestrator deployment.

Both TAS and GORCH use a kube-rbac-proxy sidecar pattern for authentication, but with a critical difference in activation: TAS always deploys with kube-rbac-proxy, while GORCH only activates it when the annotation `security.opendatahub.io/enable-auth=true` is present. Both use identical FIPS 140-2 approved cipher suites in their proxy configurations. The proxy authorizes requests by performing a SubjectAccessReview against the kube-apiserver, checking whether the bearer token holder has `get` permission on the named Service resource — a lightweight but effective authorization pattern that leverages existing Kubernetes RBAC.

The Kueue integration (JOB_MGR) is architecturally clean — it implements Kueue's `jobframework.GenericJob` interface, wrapping LMEvalJob as a Kueue-schedulable workload. The integration converts Kueue's NodeSelector-based scheduling decisions into Kubernetes NodeAffinity rules on the pod spec, and the LMES controller's suspend/resume logic dovetails naturally with Kueue's admission control. When Kueue is not enabled, the LMES controller creates pods immediately; when enabled, it waits for the queue label and defers pod creation until Kueue sets `spec.suspend=false`.
