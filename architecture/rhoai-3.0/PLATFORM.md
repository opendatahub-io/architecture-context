## Platform Overview

Red Hat OpenShift AI (RHOAI) 3.0 is an enterprise AI/ML platform deployed on OpenShift Container Platform that provides a comprehensive suite of tools for the full AI/ML lifecycle: interactive development (workbenches), model training (distributed training, evaluation), model serving (real-time inference at scale), model management (registry, metadata, catalogs), AI safety (guardrails, explainability, fairness), and platform services (authentication, monitoring, pipelines, feature stores).

The platform is deployed and managed by a central OLM-based operator (`rhods-operator`) that orchestrates 11+ sub-components through `DataScienceCluster` and `DSCInitialization` custom resources. Each component is deployed via kustomize manifests bundled within the operator image. RHOAI 3.0 introduces a significant architectural shift from RHOAI 2.x: migration from OpenShift Routes + oauth-proxy to Kubernetes Gateway API + kube-rbac-proxy for ingress and authentication, and from ServiceMesh/Istio-managed ingress to operator-managed Gateway infrastructure.

### Key Architectural Patterns (RHOAI 3.x)

1. **Gateway API Ingress**: A central `data-science-gateway` Gateway (OpenShift Gateway Controller) replaces per-component OpenShift Routes. Components create HTTPRoute CRs referencing this shared Gateway.
2. **kube-rbac-proxy Authentication**: Replaces oauth-proxy sidecars. Validates users via Kubernetes SubjectAccessReview before proxying to backend services.
3. **EnvoyFilter + ext_authz**: The Gateway uses an Istio EnvoyFilter to call `kube-auth-proxy` (an OAuth2 proxy) for external authentication, with a Lua filter to inject Bearer tokens.
4. **FIPS Compliance**: Go components built with `GOEXPERIMENT=strictfipsruntime`; all components use UBI9 base images with FIPS-compliant crypto.
5. **Disconnected Support**: All 73 container images referenced via `RELATED_IMAGE` environment variables, enabling air-gapped installations.
6. **Konflux CI/CD**: All container images built via Konflux (Tekton-based) pipelines with multi-architecture support.

### Runtime Dependencies

```
User Request Flow (RHOAI 3.x):
  Client --> data-science-gateway (Envoy/443)
    --> EnvoyFilter ext_authz --> kube-auth-proxy (OAuth2/OIDC)
    --> HTTPRoute --> Backend Service (with kube-rbac-proxy sidecar)

Model Serving Flow:
  Client --> data-science-gateway (443)
    --> HTTPRoute --> KServe InferenceService
    --> ServingRuntime Container (vLLM / OVMS / TGIS)
    --> Model Storage (S3/PVC/HuggingFace)

LLM Serving Flow (LLMInferenceService):
  Client --> openshift-ai-inference Gateway (443)
    --> AuthPolicy (Kuadrant) --> HTTPRoute
    --> InferencePool (Gateway API Inference Extension)
    --> llm-d EPP (prefill/decode routing)
    --> vLLM pods

Pipeline Flow:
  Dashboard --> DSP API Server (8888)
    --> Argo Workflow Controller --> Workflow Pods
    --> ML Metadata (gRPC/8080) --> MariaDB/PostgreSQL

Notebook Flow:
  Dashboard --> ODH Notebook Controller --> Notebook Pod
    --> HTTPRoute --> data-science-gateway
    --> kube-rbac-proxy (8443) --> JupyterLab/CodeServer/RStudio (8888/8787)

Training Flow:
  User --> Training Operator --> PyTorchJob/TFJob/etc.
    --> Worker Pods (headless Service for peer discovery)
    --> Optional: Kueue (workload scheduling)
    --> Optional: Volcano (gang scheduling)
```

## Security Architecture

### Authentication Patterns

| Pattern | Components | Mechanism | Port | Description |
|---------|-----------|-----------|------|-------------|
| Gateway ext_authz | All Gateway-fronted services | kube-auth-proxy (OAuth2/OIDC) | 8443 | EnvoyFilter calls kube-auth-proxy for session validation |
| kube-rbac-proxy sidecar | Dashboard, Notebooks, Model Registry, TrustyAI, Guardrails | SubjectAccessReview | 8443 | Per-pod RBAC check before proxying to backend |
| AuthPolicy (Kuadrant) | LLMInferenceService | Gateway API policy | N/A | Auth policy targeting Gateway + HTTPRoute |
| API Key | vLLM, OVMS (optional) | Bearer Token | 8000 | Application-level API key validation |
| mTLS | ml-metadata, TGIS, KServe (optional) | Client certificates | varies | Mutual TLS for service-to-service |
| OAuth redirect | Dashboard | OpenShift OAuth | 443 | Browser-based login flow |

## Data Flows

### Flow 1: Platform Installation

```
Admin --> OLM (install rhods-operator from FBC catalog)
  --> OLM creates CSV, deploys operator in redhat-ods-operator namespace
  --> Admin creates DSCInitialization CR
  --> DSCI Controller: creates application namespace, NetworkPolicies, Auth CR, HardwareProfiles
  --> Gateway Controller: creates GatewayClass, Gateway, kube-auth-proxy, EnvoyFilter, OAuthClient
  --> Monitoring Controller: creates MonitoringStack, ThanosQuerier, Tempo, OpenTelemetry
  --> Admin creates DataScienceCluster CR
  --> DSC Controller: creates component sub-CRDs (Dashboard, Kserve, Ray, etc.)
  --> Component Controllers: render kustomize from /opt/manifests/, deploy to cluster
```

### Flow 2: User Authentication (RHOAI 3.x)

```
User browser --> data-science-gateway (Envoy/443/HTTPS)
  --> EnvoyFilter ext_authz --> kube-auth-proxy:8443
  --> IF no valid session: redirect to OpenShift OAuth / OIDC login
  --> IF valid session: Lua filter extracts access token, sets Authorization: Bearer header
  --> Envoy forwards to backend HTTPRoute target
  --> kube-rbac-proxy sidecar: validates user can list projects (SubjectAccessReview)
  --> Backend application receives request with X-Auth-Request-User/Groups headers
```

### Flow 3: Model Inference (KServe InferenceService)

```
Client --> data-science-gateway:443 --> HTTPRoute
  --> kube-rbac-proxy sidecar:8443 --> RBAC check
  --> Serving Runtime container (vLLM/OVMS/TGIS):8000/8033
  --> Model loaded from: PVC / S3 / HuggingFace
  --> Response returned via same path
```

### Flow 4: LLM Inference (LLMInferenceService + llm-d)

```
Client --> openshift-ai-inference Gateway:443
  --> AuthPolicy (Kuadrant) validation
  --> HTTPRoute --> InferencePool
  --> llm-d EPP (ext_proc/gRPC:9002): selects optimal pod (prefill vs decode)
  --> vLLM pod:8000 (chosen by EPP)
  --> llm-d routing sidecar: P/D transfer between prefill and decode pods
  --> Response returned to client
```

### Flow 5: ML Pipeline Execution

```
User --> Dashboard --> DSP API Server:8888 (POST /apis/v2beta1/runs)
  --> API Server creates Argo Workflow CR
  --> Argo Workflow Controller creates step Pods
  --> Each step records metadata via ML Metadata gRPC:8080
  --> ML Metadata --> MariaDB/PostgreSQL:3306/5432
  --> Artifacts stored in Minio/S3
  --> User queries pipeline results via Dashboard
```

### Flow 6: Notebook/Workbench Session

```
User --> Dashboard: select workbench image, create Notebook CR
  --> ODH Notebook Controller: creates Pod, Service, HTTPRoute, injects kube-rbac-proxy
  --> User --> data-science-gateway:443 --> HTTPRoute
  --> kube-rbac-proxy:8443 --> RBAC check
  --> JupyterLab/CodeServer/RStudio:8888/8787
```

### Flow 7: Model Fairness Monitoring

```
ModelMesh/KServe --> TrustyAI /consumer/kserve/v2 (inference payloads)
  OR KServe Logger --> CloudEvents --> TrustyAI
  --> InferencePayloadReconciler --> Storage (PVC/DB/MinIO)
  --> PrometheusScheduler: periodically computes SPD, DIR, drift metrics
  --> PrometheusPublisher: updates trustyai_* Micrometer gauges
  --> Prometheus scrapes /q/metrics
  --> Dashboard queries Thanos for trustyai_* metrics
```

### Flow 8: Guardrails-Protected Inference

```
Client --> vllm-orchestrator-gateway:8090/{route}/v1/chat/completions
  --> Gateway injects detector config for route
  --> fms-guardrails-orchestrator:8033 /api/v2/chat/completions-detection
  --> Orchestrator calls: LLM backend (vLLM), input/output detectors
  --> Detectors (regex, HuggingFace, LLM-judge) evaluate content
  --> IF detection triggered AND fallback configured: return fallback message
  --> ELSE: return LLM response
```

### Flow 9: Distributed Training

```
User --> Training Operator: create PyTorchJob CR
  --> Webhook validates job spec
  --> Controller creates: master Pod, worker Pods, headless Service, NetworkPolicy
  --> Workers discover peers via headless Service DNS
  --> Workers communicate via NCCL (29500/TCP) or framework-specific protocols
  --> Optional: Kueue manages workload scheduling
  --> Controller updates job status on completion
```
