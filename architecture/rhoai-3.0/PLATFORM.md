## Platform Overview

Red Hat OpenShift AI (RHOAI) 3.0 is a Kubernetes-native machine learning platform that provides a complete lifecycle for ML/AI workloads â€” from interactive development through model training, serving, evaluation, and monitoring. The platform is deployed on OpenShift Container Platform via a single OLM-managed operator (`rhods-operator`) that provisions and configures all 34 components through a two-tier CRD hierarchy: the cluster-scoped `DSCInitialization` bootstraps platform infrastructure (namespaces, network policies, monitoring, auth, CA bundles), and the cluster-scoped `DataScienceCluster` declares which components are enabled, triggering individual component reconcilers that deploy sub-operators and services from kustomize manifests.

RHOAI 3.0 introduces a centralized Gateway API ingress architecture that replaces per-component OpenShift Routes and oauth-proxy sidecars. A single `Gateway` CR in the `openshift-ingress` namespace terminates TLS and routes traffic through an Envoy ext_authz filter to `kube-auth-proxy`, which validates OAuth2/OIDC tokens and injects identity headers into upstream requests. Components receive pre-authenticated requests and no longer need individual auth sidecars. The platform supports three auth modes (IntegratedOAuth, OIDC, None) and three certificate types (OpenshiftDefaultIngress, SelfSigned, Provided), making it adaptable to both on-premise OpenShift and managed cloud environments.

The security posture is defense-in-depth: FIPS 140-2 compliance via `GOEXPERIMENT=strictfipsruntime` across all Go binaries and UBI9 base images, NetworkPolicies restricting pod-to-pod traffic, kube-rbac-proxy sidecars enforcing SubjectAccessReview-based authorization, Istio mTLS for service mesh components, and pod-to-pod TLS via OpenShift service-ca for internal communication. The platform supports disconnected (air-gapped) deployments through the `RELATED_IMAGE` environment variable pattern and OLM `relatedImages` for image mirroring.

## Deployment Architecture

### Deployment Topology

The platform follows a hub-and-spoke deployment model. The `rhods-operator` runs in the operator namespace (`redhat-ods-operator`) and is managed by OLM via a `Subscription` and `ClusterServiceVersion`. Upon installation, it creates the platform namespace (`redhat-ods-applications`) where all component operators and platform services are deployed. The Gateway infrastructure (Gateway CR, kube-auth-proxy, EnvoyFilters) is placed in `openshift-ingress` to co-locate with the OpenShift router. User workloads (notebooks, inference services, pipelines, training jobs, feature stores) run in user-created namespaces, with the platform operator distributing CA bundles and default configurations to all namespaces via the `CertConfigMapGenerator` controller.

#### Manifest Consumption Pipeline

Component manifests flow from individual component repositories (each with their own kustomize base and overlays) through the `get_all_manifests.sh` script during the rhods-operator build process. This script downloads manifests at pinned commit SHAs from each component repo, applies distribution-specific kustomize overlays, and bundles the rendered manifests into the operator container image. At runtime, component reconcilers in rhods-operator read these bundled manifests, apply additional runtime parameterization (image overrides via `RELATED_IMAGE_*` environment variables, namespace injection, feature flags from DSC/DSCI status), and deploy them via server-side apply. This pipeline ensures reproducible deployments while allowing each component team to own their manifest definitions.

### Disconnected Support

RHOAI supports air-gapped (disconnected) deployments through a multi-layered image reference strategy. All operator-managed container images are specified as `RELATED_IMAGE_*` environment variables in the operator deployment, which OLM captures in the `relatedImages` section of the ClusterServiceVersion. Cluster administrators use `oc-mirror` to mirror these images to a private registry, and the Operator Lifecycle Manager rewrites image references at deployment time. Component operators (DSPO, KServe, Model Registry Operator) propagate image overrides from their own `RELATED_IMAGE_*` environment variables into the workloads they deploy. The `CertConfigMapGenerator` controller distributes the cluster's CA bundle (`odh-trusted-ca-bundle` ConfigMap) to all namespaces, ensuring workloads can validate TLS certificates against enterprise CA chains. All base images use SHA256 digest references for reproducibility and security.

## Platform Architectural Analysis

RHOAI 3.0 exhibits a coherent macro-architecture centered on the Kubernetes operator pattern, but reveals several architectural tensions that emerge only when viewing the platform holistically.

**The Auth Transition Is Incomplete.** The platform is migrating from per-component oauth-proxy sidecars (RHOAI 2.x) to centralized Gateway API auth (3.x), but the transition is not uniform. LLMInferenceServices use the new Kuadrant AuthPolicy â†’ Authorino â†’ RBAC chain, while standard InferenceServices still rely on kube-rbac-proxy sidecars and OpenShift Routes. InferenceGraph uses neither â€” it creates Routes directly. The MaaS API adds yet another auth layer with tiered token issuance. This means a single cluster may have four distinct auth enforcement mechanisms for different types of model endpoints, complicating security audits and operational debugging. The field name `oauthProxy` in KServe's `inferenceservice-config` still references the old oauth-proxy while pointing to the new kube-rbac-proxy image â€” a deliberate backward-compatibility choice that obscures the actual implementation.

**CRD Proliferation and Overlap.** The platform defines 85 CRDs across 34 components, with notable overlaps: `InferenceService` is watched by KServe, odh-model-controller, Model Registry, and TrustyAI Service Operator â€” each adding its own annotations, labels, sidecars, and auxiliary resources. `Notebook` exists in three API versions (v1alpha1, v1beta1, v1) across two controllers. The rhods-operator alone manages 19 CRDs, creating a deep dependency tree where changes to platform-level CRDs cascade through multiple reconciliation loops. The `components.platform.opendatahub.io/v1alpha1` API group contains 13 component CRDs that are all cluster-scoped singletons â€” a design that ensures single-instance deployment but limits multi-tenancy at the platform level.

**The Two-Speed Serving Architecture.** KServe's InferenceService (v1beta1) and LLMInferenceService (v1alpha1) represent two fundamentally different serving paradigms sharing a controller binary but diverging in networking, auth, scaling, and multi-node support. InferenceService uses Deployments, kube-rbac-proxy, OpenShift Routes, and HPA/KEDA scaling. LLMInferenceService uses LeaderWorkerSet, InferencePool/InferenceModel (Gateway API Inference Extension), Kuadrant AuthPolicy, self-signed TLS, and custom EPP-based scheduling with prefix cache awareness. The llm-d stack (inference-scheduler + routing-sidecar) adds a third layer of routing intelligence on top. This divergence is architecturally motivated â€” LLM workloads have fundamentally different requirements (multi-GPU, P/D disaggregation, KV cache awareness) â€” but creates a bifurcated operational model where serving teams must understand two distinct infrastructure stacks.

**Template-Driven vs. Programmatic Deployment.** The platform uses two competing deployment strategies. DSPO renders Go template files via Manifestival, making deployed resources visible as `.yaml.tmpl` files â€” highly auditable but requiring template expertise to modify. rhods-operator uses kustomize with server-side apply and caching â€” leveraging the standard Kubernetes toolchain but distributing configuration across base/overlay/patch files. KServe constructs resources programmatically in Go code. This inconsistency means debugging deployment issues requires different skills depending on the component.

**Shared DestinationRule with insecureSkipVerify.** Both rhods-operator (for the platform Gateway) and KServe (for LLMInferenceService workloads) create DestinationRules with `insecureSkipVerify: true`. This effectively disables TLS certificate verification within the service mesh, relying on Istio mTLS for transport security and network isolation for trust. While pragmatic for self-signed service-serving certificates, it means a compromised pod within the mesh could impersonate any service without certificate validation â€” a deviation from zero-trust principles that should be tracked as a known security trade-off.

**The TrustyAI Service Operator as a Platform Seam.** The TrustyAI Service Operator has evolved into a multi-domain operator managing four loosely-related services: TrustyAI explainability (fairness, drift), LM evaluation (lm-eval jobs), Kueue job management (evaluation queue scheduling), and Guardrails Orchestrator (content safety). These services share a binary and RBAC but have independent CRDs, reconciliation loops, and dependency graphs. The Guardrails Orchestrator controller auto-discovers KServe InferenceServices â€” a cross-cutting concern that creates an implicit dependency on KServe's labeling conventions.
