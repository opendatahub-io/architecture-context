## Purpose

**Short**: A reverse proxy sidecar that routes LLM inference requests between prefill and decode workers in a disaggregated prefill/decode (P/D) architecture for vLLM serving.

**Detailed**: The llm-d-routing-sidecar is a critical component of the llm-d disaggregated serving architecture. It runs as a sidecar container alongside a vLLM decoder instance and intercepts incoming OpenAI-compatible chat completion and completion requests. When a request includes an `x-prefiller-host-port` header, the sidecar orchestrates a two-phase inference flow: first forwarding the request to the specified prefill worker to populate KV cache data, then forwarding the decode request (with KV transfer parameters) to the local vLLM decoder.

The sidecar supports multiple P/D connector protocols — NIXL v2 (current), NIXL v1 (deprecated), and LMCache (deprecated). It provides TLS termination for both inbound connections and outbound connections to prefillers/decoders, with configurable cipher suites enforcing TLS 1.2+ minimum. An SSRF protection feature uses Kubernetes informers to watch InferencePool resources (`inference.networking.x-k8s.io/v1alpha2`) and dynamically maintain an allowlist of valid prefill pod IPs, preventing request forgery attacks against arbitrary hosts. The project is described as experimental and slated for removal in a future iteration of the llm-d P/D architecture.

## Architectural Analysis

The llm-d-routing-sidecar follows a clean sidecar proxy pattern designed specifically for the prefill/decode disaggregation architecture in LLM serving. It sits between the external request flow (typically from the Endpoint Picker / Gateway API Inference Extension) and the local vLLM decoder instance. The architectural design is deliberately minimal — a single Go binary with no controller-runtime dependency, using only `client-go` dynamic clients when SSRF protection is needed.

The connector protocol abstraction (`protocolRunner` function type) allows the sidecar to support multiple P/D communication protocols. The NIXL v2 protocol (current default) introduces a `kv_transfer_params` field that encapsulates all transfer metadata in a single JSON object, simplifying the prefill→decode handoff compared to v1's flat field approach. The LMCache protocol is even simpler — it just forwards the request to the prefiller with `max_tokens=1` and then replays the original to the decoder, relying on shared cache state. Both v1 and LMCache are deprecated in favor of NIXL v2.

The SSRF protection mechanism is architecturally notable. Rather than using a static allowlist, it dynamically watches InferencePool CRDs (from the Gateway API Inference Extension project at `inference.networking.x-k8s.io/v1alpha2`) and derives allowed prefiller targets from pod IPs matching the pool's label selector. This creates a real-time, Kubernetes-native defense against request forgery — a critical concern since the `x-prefiller-host-port` header directly controls where the sidecar sends traffic. The allowlist validates by host/IP only (not port), which is a deliberate design tradeoff documented in the README. The informer-based approach with a 30-second resync period ensures the allowlist stays current as pods scale up/down.

TLS configuration is thorough but has notable flexibility risks. The proxy supports three separate TLS configurations: inbound (serving), outbound to decoder, and outbound to prefiller. Each can be independently configured, and both outbound connections offer `InsecureSkipVerify` flags. While disabled by default, these flags represent a FIPS compliance surface area that should be monitored. The custom cipher suite list is FIPS-compatible (ECDHE + AES-GCM/ChaCha20) but the self-signed certificate fallback (RSA 4096-bit, 10-year validity) is intended only for development — production deployments should use the `-cert-path` flag with properly provisioned certificates.

The deployment manifests use kustomize with envsubst-based variable substitution rather than kustomize's native `vars` or `replacements` mechanisms. This approach works but requires the `envsubst` tool at deploy time and is less declarative than pure kustomize parameterization. The StatefulSet deployment model (rather than Deployment) is unusual for a stateless proxy and likely inherited from earlier architecture — the sidecar maintains no persistent state beyond the in-memory LRU cache of prefiller proxy handlers (capacity 16). The project is explicitly marked as experimental and slated for removal in a future llm-d iteration, suggesting the routing logic will be absorbed into another component (possibly the EPP or the vLLM serving layer itself).
