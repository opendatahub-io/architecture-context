## Purpose
**Short**: An HTTP gateway that provides OpenAI-compatible chat completion endpoints with configurable detector-based content filtering, routing requests to the FMS Guardrails Orchestrator.

**Detailed**: The vLLM Orchestrator Gateway is a Rust-based HTTP service that acts as a routing and policy enforcement layer between clients and the FMS Guardrails Orchestrator. It exposes multiple OpenAI-compatible `/v1/chat/completions` endpoints, each configured with a specific set of content detectors (e.g., PII detection via regex). This allows organizations to offer guardrailed LLM inference endpoints as drop-in replacements for standard chat completion APIs.

The gateway reads a YAML configuration file that defines three key elements: the upstream orchestrator connection details, available detector configurations (with input/output filtering capabilities and detector parameters), and named routes that map to specific detector pipelines. When a request arrives at a route endpoint (e.g., `/pii/v1/chat/completions`), the gateway injects the corresponding detector specifications into the request payload and forwards it to the FMS Guardrails Orchestrator's `/api/v2/chat/completions-detection` endpoint. If a detection is triggered and a fallback message is configured for that route, the gateway replaces the LLM response with the fallback message.

The service supports mTLS communication with the orchestrator backend, loading TLS certificates from `/etc/tls/private/` and CA certificates from `/etc/tls/ca/`, with automatic scheme selection (HTTP vs HTTPS) based on certificate availability. Authorization headers and `x-forwarded-*` headers are forwarded to the upstream orchestrator.

### TLS Configuration

The gateway supports mTLS for orchestrator communication:

| Path | Purpose | Required |
|------|---------|----------|
| `/etc/tls/private/tls.crt` | Client TLS certificate | No (HTTP fallback) |
| `/etc/tls/private/tls.key` | Client TLS private key | No (HTTP fallback) |
| `/etc/tls/ca/service-ca.crt` | Custom CA certificate for orchestrator verification | No |

When TLS certificate and key are present, the gateway automatically switches to HTTPS scheme. When only the CA certificate is present, it is added as a trusted root but HTTP is still used. When connecting to `localhost` with a custom CA, invalid hostname verification is relaxed.

**Processing logic**:
1. Client sends OpenAI-compatible chat completion request to a named route endpoint
2. Gateway resolves configured detectors for the route and injects them into the request payload
3. Gateway forwards augmented request to orchestrator's `/api/v2/chat/completions-detection` endpoint
4. On response, if detections are found and a `fallback_message` is configured, the gateway replaces the response content with the fallback message
5. Response (with or without fallback) is returned to the client

### Configuration Structure

```yaml
orchestrator:
  host: <string>       # Orchestrator hostname (default: localhost)
  port: <uint16>       # Orchestrator port (default: 8032)
detectors:
  - name: <string>     # Detector name (must match orchestrator config)
    server: <string>   # Optional server override (defaults to name)
    input: <bool>      # Apply to input messages
    output: <bool>     # Apply to output messages
    detector_params: <object>  # Detector-specific parameters
routes:
  - name: <string>     # Route name (becomes /{name}/v1/chat/completions)
    detectors: [<string>]  # List of detector names to apply
    fallback_message: <string>  # Optional message when detection triggers
```

### Validation

The gateway validates configuration at startup:
- All detectors referenced in routes must be defined in the `detectors` section
- No duplicate input detector servers per route
- No duplicate output detector servers per route
- Panics on validation failure with detailed error messages
