## Purpose

**Short**: An OpenAI-compatible HTTP gateway that routes chat completion requests through configurable detector-based content filtering pipelines to the FMS Guardrails Orchestrator.

**Detailed**: The vllm-orchestrator-gateway is a lightweight Rust HTTP service built with the Axum web framework that sits between clients and the FMS Guardrails Orchestrator. It provides multiple OpenAI-compatible `/v1/chat/completions` endpoints, each backed by a configurable set of content detectors (e.g., PII regex matching). When a client sends a chat completion request to a named route (e.g., `/pii/v1/chat/completions`), the gateway injects the corresponding detector configuration into the request payload and forwards it to the orchestrator's `/api/v2/chat/completions-detection` endpoint.

The gateway serves as a policy enforcement layer that allows platform operators to define route-specific content filtering policies. Each route can specify which detectors to apply for input and/or output analysis, and an optional fallback message that replaces the model's response when a detection is triggered. This enables drop-in replacement of an unguarded chat completion endpoint with a guardrailed one — clients use the same OpenAI-compatible API contract, but traffic flows through content safety checks transparently.

The gateway supports mTLS for secure communication with the orchestrator via OpenShift service-serving certificates, and forwards authorization headers from clients to the orchestrator for pass-through authentication.

## Architectural Analysis

The vllm-orchestrator-gateway is a minimal, single-purpose HTTP proxy with a well-defined architectural boundary: it accepts OpenAI-compatible chat completion requests and transforms them into orchestrator-compatible detection requests. The codebase is remarkably small (~330 lines of Rust across three source files) with a clear separation between configuration parsing (`config.rs`), API types (`api.rs`), and request handling logic (`main.rs`).

The dynamic route registration pattern is the core architectural decision. Rather than hardcoding endpoints, routes are defined in a YAML configuration file and registered at startup via Axum's `Router::route()`. Each route creates a closure that captures its detector list and fallback message, enabling route-specific content filtering policies without code changes. This approach trades runtime flexibility (no hot-reloading of routes) for simplicity and type safety. A pod restart is required to change the routing configuration.

The TLS implementation follows the OpenShift service-serving certificate pattern. The gateway auto-detects TLS certificates at well-known paths (`/etc/tls/private/tls.crt`, `/etc/tls/private/tls.key`) and constructs an mTLS client identity when they are present. When certificates are absent, it falls back to plaintext HTTP. This dual-mode behavior is intentional — it allows the same binary to run in development (no certs) and production (service-mesh-injected certs) without configuration changes. The `danger_accept_invalid_hostnames(true)` call for `localhost` is a pragmatic workaround for development environments where the service-serving certificate's SAN (e.g., `svc.cluster.local`) doesn't match `localhost`.

The gateway does not enforce authentication itself — it forwards `Authorization` and `x-forwarded-*` headers transparently to the orchestrator. This is a deliberate architectural choice that delegates auth enforcement to the orchestrator and any platform-level auth proxies (e.g., kube-rbac-proxy sidecar). The gateway's only security function is the content filtering policy: when a detection is found in the orchestrator's response, the gateway replaces the model's output with a configured fallback message, preventing potentially harmful content from reaching the client.

One notable gap is the absence of health check endpoints (`/healthz`, `/readyz`). The service has no liveness or readiness probes, which limits Kubernetes' ability to manage the pod lifecycle. Similarly, there are no Prometheus metrics endpoints, meaning the gateway's behavior is only observable through structured tracing logs.
