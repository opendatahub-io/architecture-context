## Purpose

**Short**: Manages the lifecycle of Jupyter notebook workbenches on Red Hat OpenShift AI, including pod management, Gateway API ingress routing, authentication sidecars, and pipeline integration.

**Detailed**: This repository contains two cooperating Kubernetes controllers that together manage Jupyter notebook workbenches in RHOAI. The **kubeflow notebook controller** (kf-notebook-controller) is the upstream Kubeflow controller responsible for core notebook lifecycle — creating StatefulSets, Services, and optionally Istio VirtualServices for each Notebook custom resource, plus idle notebook culling. The **ODH notebook controller** extends this with enterprise OpenShift features: Gateway API HTTPRoute-based ingress in a central namespace, kube-rbac-proxy authentication sidecar injection via mutating webhook, NetworkPolicy enforcement, certificate bundle management, Data Science Pipelines (DSPA) integration for Elyra, and ImageStream-based runtime image resolution.

The two controllers run as separate Deployments in the same namespace and watch the same `Notebook` CRD (`kubeflow.org`). The kf-notebook-controller handles the data plane (StatefulSet/Pod/Service lifecycle and idle culling), while the ODH controller handles the control plane (ingress routing, auth, networking, pipeline config). In RHOAI 3.x, the ODH controller uses Gateway API HTTPRoutes created in the controller's namespace (central namespace pattern) with cross-namespace ReferenceGrants, replacing the RHOAI 2.x pattern of OpenShift Routes with oauth-proxy sidecars. Legacy OAuth cleanup code remains for 2.x→3.x migration support.

## Architectural Analysis

The kubeflow repository implements a **dual-controller architecture** where the upstream Kubeflow notebook controller and the ODH-specific controller operate on the same Notebook CRD but handle orthogonal concerns. This separation preserves upstream compatibility while adding enterprise features. The kf-notebook-controller owns the StatefulSet and Service lifecycle (data plane), while the ODH controller owns the ingress, authentication, networking, and pipeline integration (control plane). Both controllers use controller-runtime and share the same namespace, coordinating through annotations on the Notebook CR (e.g., `kubeflow-resource-stopped` for culling, `odh-notebook-controller-lock` for initialization sequencing).

The **central namespace HTTPRoute pattern** (introduced in RHOAIENG-38009) is a significant architectural decision. Rather than creating HTTPRoutes in each user's namespace, the ODH controller creates all HTTPRoutes in its own namespace (typically `opendatahub`). This provides security isolation — users cannot modify their own ingress routes — but requires cross-namespace ReferenceGrants for Gateway API to permit backend references to Services in user namespaces. The controller manages one shared ReferenceGrant per user namespace, tracks notebooks via labels (since owner references don't work across namespaces), and uses three finalizers per notebook for guaranteed cleanup of cross-namespace resources (HTTPRoute, ReferenceGrant, ClusterRoleBinding).

The **kube-rbac-proxy authentication pattern** replaces RHOAI 2.x's oauth-proxy approach. When annotation `notebooks.opendatahub.io/inject-auth=true` is set, the mutating webhook injects a kube-rbac-proxy sidecar container that intercepts all traffic on port 8443, validates bearer tokens via Kubernetes TokenReview, performs SubjectAccessReview to check if the user has `get` permission on the specific Notebook CR, and only then proxies to the actual Jupyter notebook on localhost:8888. This delegates authentication entirely to Kubernetes RBAC rather than requiring per-notebook OAuth clients. The controller creates supporting infrastructure (ServiceAccount, ConfigMap with authorization policy, Service with auto-generated TLS cert, ClusterRoleBinding for auth-delegator). Auth sidecar resource limits are configurable via annotations, enabling per-notebook tuning.

The **webhook update guard** is a noteworthy design pattern. When the webhook mutates a running notebook (e.g., because the kube-rbac-proxy image was updated), it detects that the change came from webhook logic rather than user intent and adds an `update-pending` annotation instead of applying the change. This prevents unexpected pod restarts for running notebooks — the user must explicitly stop the notebook for updates to apply. The comparison logic (in `notebook_webhook_utils.go`) deep-diffs the old and new specs to distinguish user-initiated changes from webhook-injected changes.

The **contracts/schemas** directory contains JSON schema definitions for all CRDs the ODH controller interacts with (Notebook v1/v1alpha1/v1beta1, DSPA v1/v1alpha1, Gateway v1/v1beta1, HTTPRoute v1/v1beta1, ReferenceGrant v1beta1, ImageStream v1). These schemas appear to be used for validation or contract testing, ensuring the controller handles all expected CRD shapes correctly.
