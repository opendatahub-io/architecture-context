## Purpose

**Short**: Argo Workflows is a Kubernetes-native workflow engine that orchestrates parallel jobs (DAGs and steps) on Kubernetes, used by Data Science Pipelines (DSP) in RHOAI to execute ML pipeline runs.

**Detailed**: Argo Workflows provides the workflow execution engine for RHOAI's Data Science Pipelines component. It defines Custom Resource Definitions (CRDs) for workflows, cron workflows, and workflow templates, and includes a workflow controller that watches these CRDs and manages pod lifecycle for each workflow step. The controller creates pods to execute individual workflow steps, manages artifact passing between steps via S3-compatible storage, handles retries and error handling, and supports DAG and step-based execution patterns.

In the RHOAI context, Argo Workflows is not deployed as a standalone platform — it is embedded within the Data Science Pipelines (DSP) stack. The DSP operator deploys the workflow-controller and argoexec images. The Argo Server (API server and UI) is generally not deployed in RHOAI; pipeline management goes through the DSP API server instead. The two Konflux-built images (`argo-workflowcontroller` and `argo-argoexec`) are the components shipped for RHOAI, both compiled with FIPS-compliant build flags.

The argoexec binary runs as an init/sidecar container within each workflow pod, responsible for artifact collection, script execution, resource management, and reporting task results back to the controller via the Kubernetes API.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| workflow-controller | Go Controller (client-go informers, no controller-runtime) | Watches Workflow CRDs and manages pod lifecycle, scheduling, DAG execution, cron scheduling, garbage collection, and artifact GC |
| argoexec | Go Executor (sidecar/init container) | Runs inside workflow pods to execute user containers, collect artifacts, manage resources, and report task results |
| argo (CLI + server) | Go CLI / API Server | CLI for workflow management and optional API server with gRPC + REST gateway + UI (not shipped in RHOAI Konflux images) |

## Architectural Analysis

Argo Workflows in the RHOAI context serves as the low-level workflow execution engine embedded within Data Science Pipelines. Unlike standalone Argo Workflows deployments, RHOAI ships only two container images — the workflow-controller and argoexec — without the Argo Server (API + UI), which is replaced by the DSP API server for user-facing workflow management. This architectural choice means the Argo Server's authentication stack (SSO/OIDC, webhook auth, Bearer token modes) is present in the codebase but irrelevant to the RHOAI deployment model.

The controller uses a traditional client-go informer/work-queue pattern rather than controller-runtime, which gives it fine-grained control over caching, indexing, and reconciliation at the cost of more boilerplate. It maintains eight distinct indexers on the Workflow informer (by ClusterWorkflowTemplate, CronWorkflow, WorkflowTemplate, SemaphoreConfig, WorkflowPhase, Conditions, and UID), enabling efficient lookups for cross-resource resolution. The controller also runs multiple sub-controllers internally: a CronWorkflow scheduler, a GC controller for TTL-based workflow cleanup, an archive controller for database persistence, and a pod cleanup controller — all sharing the same process but with independently configurable worker counts.

A notable architectural pattern is the argoexec "emissary" executor model. Rather than using Docker or Kubernetes APIs to monitor container execution, argoexec wraps the user's command as a subprocess, captures exit codes, manages artifact collection, and reports results via WorkflowTaskResult CRs. The emissary approach writes control signals and exit codes to a shared `/var/run/argo` volume mounted between the init/sidecar and user containers. This filesystem-based coordination avoids needing privileged access to container runtimes.

The FIPS compliance story has an interesting wrinkle: the argoexec Konflux Dockerfile builds two binaries — `argoexec-fips` (CGO_ENABLED=1, GOEXPERIMENT=strictfipsruntime) and `argoexec` (CGO_ENABLED=0, statically linked). Both are placed in the image, but the ENTRYPOINT uses the non-FIPS binary. The workflow-controller Konflux image, by contrast, builds only the FIPS variant. This asymmetry means FIPS compliance for the executor requires deployment-time intervention to select the correct binary. Additionally, `InsecureSkipVerify: true` is hardcoded in the self-signed TLS configuration (`util/tls/tls.go:127`), though this is only relevant when no external TLS certificate is provided to the Argo Server.

The artifact subsystem is one of the most architecturally complex parts — it supports seven different storage backends (S3, GCS, Azure, Git, HTTP, HDFS, OSS/Alibaba) with per-backend credential injection via Kubernetes Secrets. Credentials are mounted as volumes into workflow pods by the controller's `addCredentialsToVolumes` function, which inspects the artifact repository configuration to determine which secrets to mount. This tight coupling between the controller's pod creation logic and artifact configuration means adding new storage backends requires changes in both the controller and executor.
