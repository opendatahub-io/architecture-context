## Purpose

**Short**: Kubernetes operator that manages LlamaStack AI inference server deployments by reconciling LlamaStackDistribution custom resources.

**Detailed**: The LlamaStack K8s Operator (internally named `llama-stack-k8s-operator`) is a controller-runtime-based Kubernetes operator that automates the lifecycle of LlamaStack distribution servers on OpenShift/Kubernetes clusters. Users create `LlamaStackDistribution` (shortname: `llsd`) custom resources to declare desired LlamaStack server deployments, and the operator reconciles the full deployment stack: Deployment, Service, ServiceAccount, PersistentVolumeClaim, NetworkPolicy, and RoleBinding.

The operator supports multiple LlamaStack distributions (starter, ollama, bedrock, remote-vllm, tgi, together, vllm-gpu) mapped via an embedded `distributions.json` or overridden by the `RELATED_IMAGE_RH_DISTRIBUTION` environment variable for RHOAI builds. It handles user-provided configuration via ConfigMap references, CA bundle management (including auto-detection of the platform's `odh-trusted-ca-bundle` ConfigMap), persistent storage with configurable PVC sizes, and optional NetworkPolicy creation controlled by feature flags. The operator uses kustomize-based manifest rendering with custom Go plugins for namespace injection, name prefixing, and field mutation to dynamically generate the managed resources.

The operator is deployed into the `redhat-ods-applications` namespace (RHOAI) or `opendatahub` namespace (ODH) by the platform operator via kustomize overlays. It integrates with OpenShift Security Context Constraints (SCC) by creating RoleBindings to the `system:openshift:scc:anyuid` ClusterRole, and performs upgrade cleanup of legacy ClusterRoleBindings from earlier operator versions.

## Architectural Analysis

The LlamaStack K8s Operator follows a clean single-controller architecture using the Kubebuilder v4 scaffold with controller-runtime v0.17.2. The operator manages exactly one CRD (`LlamaStackDistribution`) and creates a fixed set of child resources per CR instance. The reconcile loop is well-structured with separate phases: fetch instance, reconcile ConfigMaps, render and apply manifests, update status. The status update phase is notably thorough — it queries the managed LlamaStack pod's HTTP API to gather provider information and version data, embedding rich observability into the CR status.

The manifest rendering system is architecturally interesting: rather than constructing Kubernetes objects programmatically in Go, the operator embeds kustomize base manifests (`controllers/manifests/base/`) in the container image and runs them through a kustomize pipeline with custom Go-based plugins (`pkg/deploy/plugins/`) at reconcile time. The plugins handle name-prefixing (with Deployment excluded for backward compatibility), namespace injection, and field mutation for instance-specific values like ports, storage sizes, and label selectors. This approach separates the "shape" of resources (YAML templates) from the "parameterization" (Go plugin logic), making it easier to add new resource types without writing boilerplate Go code. However, it also means the Deployment spec is built in two stages — first as a kustomize template, then enriched with the full PodSpec (containers, volumes, init containers) via the `ManifestContext` overlay in `kustomizer.go`.

The ConfigMap watching implementation is notably sophisticated. The operator creates field indexers for efficient reverse-lookup from ConfigMap to referencing LlamaStackDistribution CRs, with a manual fallback for Kubernetes versions that don't support custom field labels. When a referenced ConfigMap changes, the operator computes a content-based hash and updates the Deployment's pod template annotations, triggering a rolling restart. This pattern ensures configuration changes propagate to running pods without manual intervention.

The CA bundle management deserves attention: the operator supports both explicit CA bundle configuration (user-specified ConfigMap with multiple PEM keys) and auto-detection of the platform's `odh-trusted-ca-bundle` ConfigMap. For multi-key CA bundles, it creates an init container that concatenates PEM certificates from a source ConfigMap volume into a single file in an emptyDir volume, which the main container then mounts at `/etc/ssl/certs/ca-bundle.crt` via `SubPath`. The `SSL_CERT_FILE` environment variable is set to point to this mount path, ensuring Python-based LlamaStack servers trust the custom CA chain. This is critical for RHOAI deployments behind corporate proxies.

One architectural concern is the FIPS build discrepancy between `Dockerfile` and `Dockerfile.konflux`. The standard Dockerfile includes full FIPS flags (`GOEXPERIMENT=strictfipsruntime`, explicit OpenSSL installation), while the Konflux Dockerfile only sets `CGO_ENABLED=1` without the FIPS build tags or OpenSSL installation. Since RHOAI is built entirely via Konflux, this means the shipped binary may not include FIPS-compliant crypto unless the UBI base image provides OpenSSL and the Go toolchain defaults handle FIPS. This should be verified against `check-payload` results.
