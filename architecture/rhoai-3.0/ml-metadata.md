## Purpose

**Short**: ML Metadata (MLMD) is a gRPC server that records and retrieves metadata associated with ML workflows, tracking artifacts, executions, contexts, and their lineage relationships.

**Detailed**: ML Metadata (MLMD) is a library and service originally developed by Google as part of the TFX (TensorFlow Extended) ecosystem. It provides a structured metadata store for recording and querying metadata about machine learning workflows. The core data model consists of three primary entities ã Artifacts (datasets, models, metrics), Executions (training runs, transforms, evaluations), and Contexts (pipelines, experiments, runs) ã connected through Events (input/output relationships), Attributions (context-to-artifact), and Associations (context-to-execution). This forms a lineage graph that enables provenance tracking across ML pipelines.

In the RHOAI platform, MLMD serves as the metadata backend for ML pipeline orchestration. It is deployed as a standalone gRPC server (`metadata_store_server`) that exposes the `MetadataStoreService` API on port 8080/TCP. The server supports MySQL, PostgreSQL, and SQLite as storage backends, with PostgreSQL and MySQL being the production-relevant options. The Red Hat downstream fork ships the server as a container image (`odh-mlmd-grpc-server`) built on UBI 9 via Konflux, with the C++ binary compiled from source using Bazel. The server is consumed by pipeline components (e.g., Kubeflow Pipelines, Data Science Pipelines) that need to record experiment metadata, track model lineage, and query artifact provenance.

## Architectural Analysis

ML Metadata follows a straightforward client-server architecture with a single gRPC entry point and a pluggable database backend. The server binary (`metadata_store_server`) is entirely stateless ã it creates a new database connection for each gRPC request (observed in `metadata_store_service_impl.cc` where `ConnectMetadataStore` is called at the top of every RPC handler). This per-request connection pattern simplifies the server's concurrency model but may introduce performance overhead under high load due to connection setup costs. The service implementation is explicitly documented as thread-safe but sequential for concurrent calls, with a TODO noting the need for a connection pool (`b/123039345`).

The data model is well-designed for ML provenance tracking. The three core entities (Artifact, Execution, Context) form a flexible graph through Events (artifact-execution edges), Attributions (context-artifact edges), and Associations (context-execution edges). The `GetLineageSubgraph` API provides sophisticated graph traversal with configurable direction (upstream/downstream/bidirectional), maximum hops, ending node filters, and read masks for controlling response size. This replaces the deprecated `GetLineageGraph` API and represents a significant improvement in query flexibility.

The Red Hat downstream fork is minimal ã the commit history on `rhoai-3.0` consists almost entirely of Renovate bot updates to the UBI base image digests in the Dockerfiles. No functional code changes are visible in the recent 20 commits, indicating this is a stable upstream component that Red Hat consumes with minimal modification. The primary downstream value-add is the Konflux-based build pipeline (`Dockerfile.konflux`) that produces the container image on UBI 9 with multi-architecture support (x86_64, arm64, ppc64le).

A notable FIPS consideration: the binary uses BoringSSL (bundled at build time via Bazel) rather than system OpenSSL. While BoringSSL has a FIPS-validated module, the Bazel build does not explicitly enable FIPS mode (`-DFIPS=1` or `BORINGSSL_FIPS`). This means the server's TLS implementation may not use FIPS-validated cryptographic primitives even on a FIPS-enabled RHEL/UBI host. Since the binary is C++ (not Go), the standard check-payload Go FIPS validation does not apply. This is a gap worth investigating for FIPS-critical deployments.

The server supports schema migration (`MigrationOptions.enable_upgrade_migration`) which allows automatic database schema upgrades when the server version advances. Downgrade migration is also supported but requires explicit opt-in and is documented as potentially causing data loss. The server retries database connections up to 5 times (configurable via `--metadata_store_connection_retries`) to handle transient connection failures during startup.
