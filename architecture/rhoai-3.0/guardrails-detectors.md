## Purpose
**Short**: A collection of detector algorithm microservices for the FMS Guardrails Orchestrator, providing text content analysis for safety, PII detection, file-type validation, and LLM-as-a-judge evaluation.

**Detailed**: The Guardrails Detectors repository provides a suite of detector microservices that integrate with the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator). Each detector is a standalone Python FastAPI-based microservice that accepts text content and returns detection results with position information, confidence scores, and classification labels.

Three detector types are provided: (1) the **Built-in Detector**, which bundles lightweight heuristic detection functions such as regex-based PII detection (emails, credit cards, SSNs, phone numbers, IP addresses), file-type validation (JSON, XML, YAML with optional schema validation), and user-defined custom Python detectors; (2) the **HuggingFace Detector**, a generic detector compatible with HuggingFace `AutoModelForSequenceClassification` and `GraniteForCausalLM` models (e.g., IBM Granite Guardian) that performs ML-based content classification; and (3) the **LLM Judge Detector**, which uses the [vllm_judge](https://github.com/trustyai-explainability/vllm_judge) library to leverage any OpenAI-compatible LLM for content evaluation against built-in metrics (safety, toxicity, accuracy, helpfulness, etc.) and custom natural-language criteria.

All detectors expose a common `/api/v1/text/contents` REST API endpoint conforming to the IBM Detector API specification. The HuggingFace and LLM Judge detectors are designed for deployment as KServe InferenceServices, while the Built-in detector runs as a standalone microservice. Each detector includes Prometheus metrics instrumentation for monitoring detection counts, request counts, error counts, and runtime latencies.

### Custom Detector Security

The Built-in Detector includes a static code analysis security mechanism for user-provided custom Python detectors:

- **Forbidden imports**: `os`, `subprocess`, `sys`, `shutil`
- **Forbidden function calls**: `eval`, `exec`, `open`, `compile`, `input`
- Custom detector modules are scanned at startup via AST analysis; the application refuses to start if forbidden patterns are detected.
