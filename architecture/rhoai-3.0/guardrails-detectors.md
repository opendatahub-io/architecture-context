## Purpose

**Short**: A collection of detector microservices that analyze text content for the FMS Guardrails Orchestrator, providing PII detection, content safety classification, file type validation, custom code-based detection, and LLM-as-a-Judge evaluation.

**Detailed**: Guardrails Detectors is a set of Python-based microservices designed to work with the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator). Each detector runs as an independent FastAPI service exposing a REST API endpoint (`/api/v1/text/contents`) that accepts text content and returns detection results with position information, confidence scores, and evidence metadata.

The repository produces three distinct detector types, each packaged as a separate container image: (1) the **Built-in Detector** — a lightweight collection of heuristic algorithms including regex-based PII detection (email, credit card, SSN, phone numbers, IP addresses), file format validation (JSON, XML, YAML with optional schema enforcement), and a custom detector framework allowing user-defined Python functions with static code analysis security scanning; (2) the **HuggingFace Detector** — a GPU-capable runtime for running HuggingFace models (AutoModelForSequenceClassification and GraniteForCausalLM) that supports content safety classification across risk categories like harm, social bias, jailbreak, profanity, and violence; and (3) the **LLM Judge Detector** — an LLM-as-a-Judge evaluator that delegates to an external vLLM-compatible server using the `vllm_judge` library for metrics like safety, toxicity, accuracy, and helpfulness.

All detectors are deployed as KServe InferenceServices using `RawDeployment` mode on OpenShift AI, with Istio sidecar injection, Prometheus metrics instrumentation, and optional TLS support. The detectors share a common FastAPI base class (`DetectorBaseAPI`) that provides health checking, structured error handling, and Prometheus counters for detections, requests, errors, and runtime latency per detector function.

## Architectural Analysis

The guardrails-detectors repository follows a modular microservice architecture with three distinct detector types sharing a common FastAPI base framework. The `DetectorBaseAPI` class in `detectors/common/app.py` provides a uniform foundation including health endpoints, structured error handling, and Prometheus instrumentation, while each detector variant implements its own `lifespan` initialization and request handling logic. This design allows the Guardrails Orchestrator to treat all detectors uniformly through the `/api/v1/text/contents` API contract, regardless of whether the detection runs a regex, a local ML model, or an external LLM call.

The built-in detector is architecturally notable for its registry pattern — `BaseDetectorRegistry` is an abstract base that `RegexDetectorRegistry`, `FileTypeDetectorRegistry`, and `CustomDetectorRegistry` all inherit from. The custom detector system includes a static code analysis security gate (`static_code_analysis` in `custom_detectors_wrapper.py`, lines 51-88) that scans user-provided Python code for forbidden imports (`os`, `subprocess`, `sys`, `shutil`) and dangerous function calls (`eval`, `exec`, `open`, `compile`, `input`) before loading. This is a defense-in-depth measure against arbitrary code execution, though it operates on AST analysis alone and cannot catch all evasion techniques (e.g., `__import__`, `getattr` chains).

The HuggingFace detector demonstrates thoughtful multi-architecture support. The Konflux Dockerfile (`Dockerfile.konflux.hf`) includes conditional compilation of PyTorch 2.6.0 from source for ppc64le and s390x architectures, along with OpenBLAS compilation for ppc64le. This is a significant build complexity — the Dockerfile has 5 build stages with architecture-specific branching via `TARGETARCH` — but it enables the same detector image to run across all RHOAI-supported architectures.

The Prometheus instrumentation layer deserves attention. The `InstrumentedDetector` class in `detectors/common/instrumented_detector.py` tracks four metrics per detector function — `trustyai_guardrails_detections` (detection count), `trustyai_guardrails_requests` (request count), `trustyai_guardrails_errors` (error count), and `trustyai_guardrails_runtime` (cumulative latency). These are labeled by `detector_kind` and `detector_name`, which means each regex pattern, file type check, and custom function gets independent observability. The built-in detector is careful to avoid metric cardinality explosion by using generic labels like `custom_regex` and `json-with-schema` instead of publishing user-provided regex patterns or schemas as Prometheus labels.

A notable gap is the absence of any authentication or authorization at the application layer — all detector endpoints accept unauthenticated requests. Security is entirely delegated to the platform: Istio sidecar injection provides mTLS between pods, and the KServe InferenceService annotations (`security.opendatahub.io/enable-auth: "true"`, `serving.knative.openshift.io/enablePassthrough: "true"`) configure platform-level authentication. The LLM Judge detector's connection to the upstream vLLM server is also unauthenticated and unencrypted (plain HTTP within the cluster), relying on network-level isolation.
