## Purpose

**Short**: Data Science Pipelines (based on Kubeflow Pipelines v2) provides a platform for defining, deploying, scheduling, and monitoring multi-step ML/data science workflows on Kubernetes using Argo Workflows as the execution engine.

**Detailed**: Data Science Pipelines is a downstream distribution of Kubeflow Pipelines (KFP) v2, adapted for Red Hat OpenShift AI. It enables data scientists and ML engineers to define pipelines as Python code using the KFP SDK, compile them into portable IR (Intermediate Representation) YAML, and submit them for execution on Kubernetes clusters. The platform manages the full pipeline lifecycle: compilation, submission, scheduling (via cron or periodic triggers), execution orchestration (via Argo Workflows), artifact storage (via S3-compatible object stores), metadata tracking (via ML Metadata), and caching of repeated steps.

The system is composed of five backend Go binaries (API server, driver, launcher, persistence agent, scheduled workflow controller), a React-based frontend, a cache webhook server, and supporting infrastructure (ML Metadata gRPC service, object storage, relational database). In the RHOAI distribution, all Go binaries are built with FIPS-compliant cryptography via `GOEXPERIMENT=strictfipsruntime` and UBI9 base images. The platform operator (`rhods-operator` / `opendatahub-operator`) deploys and manages these components via the `DataSciencePipelinesApplication` CRD, which is defined externally in the `data-science-pipelines-operator` repository.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| API Server (`ml-pipeline`) | Go Service | Central REST/gRPC API for pipeline CRUD, run management, experiment tracking, artifact access, and webhook validation |
| Driver (`driver`) | Go CLI (Argo step) | Orchestrates pipeline DAG execution: resolves inputs, evaluates conditions, manages caching, patches pod specs for each step |
| Launcher (`launcher-v2`) | Go CLI (Argo step) | Executes individual pipeline steps: downloads input artifacts, runs user containers, uploads output artifacts, records metadata |
| Persistence Agent (`persistenceagent`) | Go Agent | Watches Argo Workflow and ScheduledWorkflow resources, syncs state back to the API server via gRPC |
| Scheduled Workflow Controller (`scheduledworkflow`) | Go Controller | Manages `ScheduledWorkflow` CRD to create Argo Workflows on cron/periodic schedules |
| Cache Server (`cache-server`) | Go Webhook Server | Mutating admission webhook for Argo Workflow pods to enable/disable cache reuse |
| Frontend (`ml-pipeline-ui`) | React/TypeScript App | Web UI for pipeline visualization, run management, experiment browsing, and artifact inspection |
| Viewer Controller (`viewer-crd`) | Go Controller | Manages `Viewer` CRD to deploy TensorBoard instances on demand |
| Visualization Server (`visualization`) | Python Service | Renders pipeline visualization artifacts |
| ML Metadata Service | Go/C++ gRPC Service | Tracks execution metadata, artifacts, and lineage (third-party component) |

## Architectural Analysis

Data Science Pipelines follows a **driver/launcher split execution model** that cleanly separates orchestration concerns from step execution. The **driver** runs as an Argo Workflow step that handles DAG-level logic: it resolves input parameters and artifacts from upstream tasks, evaluates CEL-based conditional expressions, checks the cache server for reusable results, and generates pod spec patches for the next step. The **launcher** then executes within the step's pod to handle artifact I/O: downloading inputs from object storage, running the user's container command, uploading outputs, and registering everything in ML Metadata. This two-phase architecture enables caching to skip entire step executions (the driver detects a cache hit and short-circuits before the launcher even runs) and provides clean separation between infrastructure concerns and user code execution.

The **launcher ships both FIPS and non-FIPS binaries** in the same container image (`launcher-v2` built with `CGO_ENABLED=0` and `launcher-v2-fips` built with `GOEXPERIMENT=strictfipsruntime`). This dual-binary approach is unique among the DSP components — all other binaries are FIPS-only. The rationale is that the launcher runs inside user-controlled pipeline pods where FIPS requirements may vary, and the non-FIPS static binary avoids OpenSSL dependency issues in arbitrary base images. The selection of which binary to use at runtime depends on the platform operator configuration.

The **persistence agent pattern** decouples workflow state tracking from the execution engine. Rather than having the API server poll Argo Workflows for status, the persistence agent runs as a dedicated controller that watches Workflow and ScheduledWorkflow resources via Kubernetes informers and reports state changes to the API server via gRPC. This uses projected ServiceAccount tokens with 1-hour expiry for authentication, with automatic refresh logic. The agent implements exponential backoff (1s to 360s) and classifies errors as transient (retry) or permanent (stop). Workflows are eligible for TTL-based cleanup after 86400 seconds (1 day) by default.

The **ScheduledWorkflow CRD** provides a Kubernetes-native scheduling primitive that supports both cron expressions (via `robfig/cron`) and fixed-interval periodic schedules. It enforces concurrency limits (`maxConcurrency` 1-10), supports catchup/no-catchup modes for missed schedules, and tracks workflow history up to a configurable `maxHistory` (0-100). The controller creates Argo Workflow CRs directly for execution, and for v2 pipelines, also creates Runs via the API server's gRPC interface. This dual creation path (Workflow CR + API Run) ensures both the execution engine and the metadata store are aware of scheduled runs.

The **cache server** operates as a Kubernetes mutating admission webhook that intercepts Argo Workflow pod creation. It injects cache-related annotations into pods, enabling the driver to check for cache hits before executing steps. The cache server connects to the same database as the API server to look up historical execution fingerprints. This webhook approach is architecturally significant because it operates at the pod admission level rather than at the workflow level, allowing per-step cache decisions without modifying the Argo Workflow controller itself.

The **network architecture** is notably flat within the namespace — all services communicate over plaintext HTTP/gRPC internally, relying on Kubernetes NetworkPolicy (`default-allow-same-namespace`) to restrict cross-namespace traffic. TLS is optional for most internal connections. The cache server is the exception, requiring TLS because it operates as a Kubernetes admission webhook (which mandates HTTPS). External access is provided through OpenShift Routes (for the frontend) or Istio VirtualServices (in multi-user mode). The RHOAI deployment via `data-science-pipelines-operator` may configure additional ingress mechanisms (HTTPRoute via Gateway API, kube-rbac-proxy sidecars) that are defined in the operator's controller code rather than in this repository's manifests.
