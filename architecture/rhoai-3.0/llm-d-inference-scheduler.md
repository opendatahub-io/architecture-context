## Purpose

**Short**: An inference request scheduler that makes optimized routing decisions for LLM inference requests via a Kubernetes Gateway API Inference Extension (GIE) Endpoint Picker (EPP).

**Detailed**: The llm-d-inference-scheduler is the core routing intelligence component of the llm-d inference framework. It implements an Endpoint Picker (EPP) that integrates with Envoy's External Processing (ext-proc) filter via gRPC to intercept and route incoming inference requests to optimal model-serving pods. The EPP receives requests from an Envoy-based Gateway, evaluates candidate pods using a configurable pipeline of filter and scorer plugins, and returns the selected pod endpoint back to Envoy for request forwarding.

The scheduler extends the upstream Gateway API Inference Extension (GIE) project with llm-d-specific plugins for disaggregated Prefill/Decode (P/D) scheduling, precise KV-cache-aware prefix scoring using real-time ZMQ event streams from vLLM instances, load-aware scoring, session affinity, and label-based pod filtering. The plugin architecture is fully configurable via YAML-based `EndpointPickerConfig` resources that define plugin instances, parameters, and scheduling profiles. Multiple scheduling profiles can be composed for different routing strategies — notably the dual-profile P/D configuration where prefill and decode stages are routed to separate specialized pods.

The component uses controller-runtime to watch Kubernetes resources (InferencePool, Pods, EndpointSlices) for dynamic pod discovery, and integrates with the llm-d-kv-cache-manager library for precise prefix cache scoring via ZMQ-based KV-cache event subscriptions. It produces a single container image deployed as a Deployment alongside an Envoy-based Gateway implementation (e.g., Istio).

## Architectural Analysis

The llm-d-inference-scheduler follows a clean plugin architecture built on top of the upstream Gateway API Inference Extension (GIE) framework. The main binary (`cmd/epp/main.go`) is remarkably thin — just 7 lines of non-boilerplate code that register llm-d-specific plugins and delegate to the GIE runner. This means the vast majority of the EPP infrastructure (gRPC ext-proc server, controller-runtime watches, scheduling framework, health probes, metrics) comes from the GIE dependency. The llm-d repo contributes only the custom plugins that implement llm-d-specific routing intelligence.

The plugin system supports four extension points: **Filters** (exclude pods from candidates), **Scorers** (rank candidate pods), **ProfileHandlers** (orchestrate multi-profile scheduling), and **PreRequest** hooks (inject metadata into headers). Each plugin type implements a factory pattern with JSON-based parameter configuration, enabling runtime composition via the `EndpointPickerConfig` YAML. This architecture allows the same EPP binary to serve fundamentally different routing strategies — from simple single-profile decode-only routing to complex P/D disaggregated scheduling — purely through configuration changes.

The **Prefill/Decode disaggregation** design is architecturally notable. The `PdProfileHandler` orchestrates a two-pass scheduling approach: it first runs the decode profile to select a decode pod, then conditionally runs the prefill profile if the prompt's uncached suffix exceeds a configurable threshold. The `PrefillHeaderHandler` then injects the selected prefill pod's address into the `x-prefiller-host-port` header, enabling the decode pod to coordinate with the prefill pod at the vLLM layer. This creates a loose coupling where the scheduler only needs to pick pods — the actual P/D coordination happens between the vLLM instances themselves.

The **precise prefix cache scorer** (`PrecisePrefixCacheScorer`) is the most complex plugin. Unlike the upstream GIE's estimation-based prefix cache scorer, it uses real-time KV-cache state from vLLM instances via ZMQ PUB/SUB event streams on port 5557. The `llm-d-kv-cache-manager` library maintains an in-process KV-cache index that is continuously updated from these events, enabling the scorer to calculate exact prefix cache hit ratios rather than relying on scheduling history. This requires CGO for the HuggingFace tokenizer bindings (`libtokenizers.a`) used to tokenize prompts into blocks matching vLLM's internal block structure.

The deployment architecture has a notable split between the inference-gateway component manifests (Deployment-based, used with Istio/kGateway) and the OpenShift overlay (StatefulSet-based). The OpenShift variant uses a StatefulSet with an attached OpenShift Route for external access, while the inference-gateway variant uses a standard Deployment behind a Gateway API Gateway. The OpenShift RBAC includes a broad `pods/exec` role bound to `system:authenticated`, which is likely intended for development/debugging and should be reviewed for production hardening.
