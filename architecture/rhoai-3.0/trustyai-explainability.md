## Purpose

**Short**: TrustyAI is a Java library and Quarkus REST service that provides AI fairness metrics, drift detection, and explainability algorithms for ML model monitoring on OpenShift AI.

**Detailed**: TrustyAI Explainability is the core computational engine for TrustyAI on Red Hat OpenShift AI. It is a multi-module Maven project that produces a single deployable Quarkus-based REST service (`explainability-service`) backed by a rich Java library (`explainability-core`) for Explainable AI (XAI). The service integrates with KServe and ModelMesh inference platforms via gRPC (KServe v2 protocol) and HTTP (KServe v1/v2), consuming inference payloads (inputs/outputs) and storing them for continuous fairness and drift monitoring.

The service exposes REST endpoints for computing fairness metrics (Statistical Parity Difference, Disparate Impact Ratio), data drift metrics (Meanshift, FourierMMD, KS-Test, Jensen-Shannon), and local/global explainability algorithms (LIME, SHAP, Counterfactual, Partial Dependence Plots, Time-Series Saliency). Computed metrics are published to Prometheus on a configurable schedule. The service supports multiple storage backends: PVC-based flat files, MinIO/S3-compatible object storage, and relational databases (MariaDB, MySQL) via Hibernate ORM.

In the RHOAI distribution, the service runs as a sidecar-like deployment alongside model serving components. The TrustyAI Service Operator (a separate repository) manages the lifecycle of TrustyAI service instances. The RHOAI profile disables local explainer and global explainer endpoints, keeping only fairness and drift monitoring active.

## Architectural Analysis

TrustyAI Explainability follows a clean multi-module Maven architecture where the core library (`explainability-core`) is framework-agnostic and can be used independently, while the Quarkus service layer (`explainability-service`) provides the REST API, scheduling, and storage integration. This separation allows the core algorithms to be consumed by the TrustyAI Python library (via Arrow serialization) and the service independently.

The storage layer uses a Strategy pattern implemented via Quarkus CDI's `@LookupIfProperty` annotation, allowing the storage backend (PVC, MinIO, DATABASE) to be selected at deployment time through the `service.storage-format` configuration property. The PVC backend provides the most complete feature set (batch reads, sliced reads, tag/ID filtering), while MinIO has limited functionality (no batch or tag-filtered reads). The DATABASE backend supports migration from PVC format via CDI events, indicating an evolutionary architecture where PVC was the original storage and database support was added later.

The inference payload reconciliation system is sophisticated — it handles two distinct ingestion pathways: ModelMesh sends Base64-encoded KServe v2 protobuf payloads via REST POST, while KServe InferenceServices emit Knative CloudEvents. Both pathways converge on the same `InferencePayloadReconciler` abstraction, which pairs incoming request/response payloads by ID, converts them to TrustyAI's internal `Dataframe` format, and persists them. The reconciler also handles synthetic data tagging via the `BIAS_IGNORE_PARAM` parameter, allowing the fairness metrics to distinguish between real inference data and data injected by explainability algorithms.

The RHOAI profile configuration is noteworthy: it disables all local and global explainability endpoints (LIME, SHAP, Counterfactual, TSSaliency, Aggregated LIME, PDP), keeping only fairness metrics (SPD, DIR) and drift detection (Meanshift, FourierMMD). This suggests that in the RHOAI distribution, the primary use case is continuous model monitoring rather than interactive explainability. The explainability features remain available in the ODH distribution. Feature flag toggling is implemented via a custom `@EndpointDisabled` annotation pattern.

The Prometheus integration deserves attention: metrics are not just passively exposed — the `PrometheusScheduler` actively computes metrics on a configurable schedule (default 5s) and publishes them as dynamically-tagged Micrometer gauges with the `trustyai_` prefix. This allows users to register recurring metric computations via the REST API (`/request` endpoints) that persist across service restarts. The scheduler reads the latest batch of inference data, computes all registered metrics, and publishes the results to Prometheus in each cycle.
