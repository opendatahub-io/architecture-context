## Purpose

**Short**: Kubernetes operator that manages distributed machine learning training jobs across multiple ML frameworks (PyTorch, TensorFlow, XGBoost, MPI, PaddlePaddle, JAX).

**Detailed**: The Kubeflow Training Operator (KFTO) is a controller-runtime based Kubernetes operator that provides Custom Resource Definitions for running distributed ML training workloads. It manages the full lifecycle of training jobs including pod creation, service creation for inter-worker communication, status tracking, and cleanup. Each supported framework (PyTorch, TensorFlow, XGBoost, MPI, PaddlePaddle, JAX) has a dedicated controller that understands the framework's distributed training semantics — such as parameter servers vs. workers in TensorFlow, elastic training in PyTorch, or launcher/worker patterns in MPI.

The operator integrates with gang scheduling systems (Volcano and scheduler-plugins) for coordinated pod scheduling, creates per-job NetworkPolicies for PyTorch workloads to isolate inter-worker traffic, and exposes Prometheus metrics for job lifecycle events. It uses validating webhooks to enforce job spec constraints at admission time. In the RHOAI distribution, the operator is built with FIPS-compliant cryptography (strictfipsruntime), deployed in the `opendatahub` namespace with a `kubeflow-` name prefix, and monitored via a PodMonitor for Prometheus scraping.

## Architectural Analysis

The training operator follows a clean plugin architecture where each ML framework (PyTorch, TensorFlow, MPI, XGBoost, PaddlePaddle, JAX) implements a common `ControllerInterface` and registers itself in `SupportedSchemeReconciler`. The `JobController` base type in `pkg/controller.v1/common/` provides shared reconciliation logic for pod and service management, while each framework controller implements framework-specific logic like environment variable injection (TF_CONFIG for TensorFlow, MASTER_ADDR/MASTER_PORT for PyTorch) and success/failure determination policies.

The MPI controller is architecturally distinct from the other controllers. While PyTorch, TensorFlow, XGBoost, PaddlePaddle, and JAX use the standard pod/service reconciliation pattern, the MPI controller overrides `ReconcileServices` (no-op) and `ReconcilePods` with a custom launcher/worker pattern. It creates per-job RBAC resources (ServiceAccount, Role, RoleBinding) so the launcher pod can `kubectl exec` into worker pods for SSH-less MPI communication. It also manages a ConfigMap containing the hostfile and kubexec script that replaces traditional SSH-based MPI launching.

The PyTorch controller has a notable RHOAI-specific addition: it creates a NetworkPolicy per job that restricts pod-to-pod ingress to only pods within the same training job, with an exception for the `openshift-monitoring` namespace to allow Prometheus scraping on port 8080. This provides network isolation between training jobs in shared namespaces. The PyTorch controller also supports HorizontalPodAutoscaler creation for elastic training scenarios.

The operator's webhook certificate management uses the `open-policy-agent/cert-controller` library rather than cert-manager, making it self-contained. The cert-rotator creates a self-signed CA and server certificate, stores them in a Secret, and automatically rotates them. The readiness probe blocks until certificates are ready, ensuring the operator doesn't accept traffic before webhooks are functional. Controllers are started in a goroutine that waits on a `certsReady` channel, decoupling controller startup from certificate generation.

For the RHOAI distribution, the kustomize overlay in `manifests/rhoai/` applies several important transformations: it sets the namespace to `opendatahub`, prefixes all resource names with `kubeflow-`, replaces the container image via configMapGenerator/replacements (including using the same image for the PyTorch init container), removes the metrics Service port in favor of a PodMonitor for Prometheus scraping, and adds aggregated ClusterRoles (`training-edit`, `training-view`) that integrate with OpenShift's standard RBAC aggregation labels (`rbac.authorization.k8s.io/aggregate-to-edit` and `aggregate-to-view`).
