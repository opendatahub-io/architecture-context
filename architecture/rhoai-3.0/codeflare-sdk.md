## Purpose

**Short**: Python SDK providing a high-level interface for managing Ray clusters and submitting distributed computing jobs on Kubernetes/OpenShift via the KubeRay operator.

**Detailed**: The CodeFlare SDK is a client-side Python library that abstracts the complexity of deploying and managing Ray distributed computing clusters on Kubernetes and OpenShift. It provides intuitive Python classes (`Cluster`, `ClusterConfiguration`, `RayJob`, `RayJobClient`) for requesting compute resources, configuring Ray clusters (including GPU/accelerator support), submitting jobs, and monitoring cluster and job status — all through KubeRay custom resources (RayCluster, RayJob).

The SDK integrates with key RHOAI/ODH platform components including Kueue for job queuing and resource management, Gateway API (HTTPRoute) for dashboard access in RHOAI 3.x, and OpenShift Routes for legacy dashboard routing. It handles mTLS certificate generation for secure Ray cluster communication, supports multiple authentication methods via kube-authkit (OIDC, OAuth, kubeconfig, in-cluster), and provides auto-detection of cluster topology (OpenShift vs vanilla Kubernetes).

As a client library, the SDK does not deploy as a service itself — it runs within Jupyter notebooks, workbenches, or standalone Python environments. It creates and manages Kubernetes resources remotely through the Kubernetes API, making it the primary user-facing interface for the CodeFlare distributed computing stack within Red Hat OpenShift AI.

## Architectural Analysis

The CodeFlare SDK follows a layered architecture with clean separation between user-facing abstractions and Kubernetes API interactions. The top layer (`Cluster`, `RayJob`) provides high-level lifecycle management, while the vendored KubeRay Python client handles low-level CRD CRUD operations. This layering means the SDK can evolve its user experience independently of the underlying KubeRay API version.

A notable design pattern is the multi-level fallback for Ray dashboard URL discovery. The SDK tries HTTPRoute (Gateway API, RHOAI 3.x), then OpenShift Route, then Kubernetes Ingress — making it compatible across RHOAI 3.x, RHOAI 2.x, and vanilla Kubernetes environments. The HTTPRoute discovery (`cluster.py:1261-1392`) searches multiple namespaces (`redhat-ods-applications`, `opendatahub`, `default`, `ray-system`) and extracts the hostname from Gateway listeners, OpenShift Routes matching the gateway name, or Gateway status addresses as a last resort. This resilience is critical because users may not know which ingress mechanism their platform uses.

The mTLS certificate management (`generate_cert.py`) implements a full certificate chain: it retrieves the KubeRay-generated CA secret (identified by label `ray.openshift.ai/cluster-name`), then generates a client TLS certificate signed by that CA with RFC 5280 extensions (BasicConstraints, KeyUsage, ExtendedKeyUsage for both serverAuth and clientAuth, and SAN including the head service DNS names). Certificates are stored locally at `~/.local/share/codeflare/tls/{cluster-name}-{namespace}/` with 0600 permissions and can be refreshed when the server CA rotates. The certificate lifecycle is tightly coupled to cluster lifecycle — certificates are generated on `apply()`, refreshed on demand, and cleaned up on `down()`.

The SDK's RayJob interface represents a newer, more declarative approach compared to the interactive `Cluster` + `RayJobClient` pattern. `RayJob` creates a Kubernetes RayJob CR directly, which the KubeRay operator manages. It can either reference an existing RayCluster or embed a `rayClusterSpec` to create a new one. Local files (entrypoint scripts, working directories) are uploaded as Kubernetes Secrets with owner references to the RayJob CR, ensuring garbage collection. The runtime environment processing handles local working directories by zipping them, storing as base64 in Secrets, and using an init container to unzip on the submitter pod.

One architectural risk is the vendored KubeRay client — it is a copy of the upstream KubeRay Python client embedded at `src/codeflare_sdk/vendored/python_client/`. This means security patches or API changes in the upstream client require manual synchronization. The vendored client has its own version (`1.1.0`) independent of the SDK version. Additionally, the SDK pins the Ray version tightly (`2.54.1`) and validates image compatibility at submit time (`validate_ray_version_compatibility`), which prevents silent version mismatches but may require coordinated upgrades across the stack.
