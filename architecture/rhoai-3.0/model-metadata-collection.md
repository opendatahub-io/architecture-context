## Purpose

**Short**: A Go CLI tool and data-only container that extracts, enriches, and catalogs AI/ML model metadata from OCI container registries and HuggingFace, producing YAML catalog files consumed by the RHOAI platform dashboard.

**Detailed**: The model-metadata-collection component is a build-time data pipeline that processes AI/ML model metadata from two primary sources: OCI container registries (registry.redhat.io) containing "modelcar" images, and the HuggingFace API (huggingface.co) for enrichment data. The pipeline extracts metadata from model cards embedded in OCI image layers, enriches it with additional information from HuggingFace (descriptions, licenses, languages, tags, tasks, download/like counts), and produces structured YAML catalog files.

The component ships as a data-only container image based on `ubi9-micro` that contains only the generated YAML catalog files (no binaries). The Go CLI tool (`model-extractor`) runs during CI/CD to produce these files, which are then baked into the container image. A secondary CLI tool (`metadata-report`) generates completeness reports for metadata quality tracking. The container image is consumed by other RHOAI platform components (primarily the dashboard) to display model catalog information to users.

The metadata pipeline implements a multi-source enrichment strategy with source tracking: each metadata field records where it came from (e.g., `modelcard.yaml`, `huggingface.api`, `huggingface.regex`). This allows downstream consumers to understand data provenance and confidence levels. The pipeline also supports static catalog entries for models with manually curated metadata, and merges them with dynamically extracted data while deduplicating by model name.

## Architectural Analysis

### Data Pipeline Pattern
The component follows a classic ETL (Extract-Transform-Load) pattern with three distinct phases:
1. **Extract**: Pull metadata from OCI container registries (modelcard layers) and HuggingFace API
2. **Transform**: Parse YAML frontmatter, apply regex-based extraction, normalize tasks/languages, enrich with HuggingFace data, track data sources
3. **Load**: Generate aggregated YAML catalog files and per-model metadata files

### Multi-Source Enrichment with Provenance
A key design decision is the metadata source tracking system. Every metadata field records its origin (e.g., `modelcard.yaml`, `huggingface.api`, `huggingface.regex`, `huggingface.tags`). This allows downstream consumers to assess confidence and enables debugging of metadata quality issues. The enrichment follows a priority hierarchy: YAML frontmatter from model cards takes precedence over HuggingFace YAML frontmatter, which takes precedence over API-level metadata, which takes precedence over regex-extracted values.

### Model Family Matching Safety
The HuggingFace enrichment pipeline implements model family matching (`isCompatibleModelFamily`) to prevent cross-contamination of metadata between different model families (e.g., matching a Llama model's HuggingFace data to a Granite registry entry). This is critical because many models share similar naming patterns and the similarity-based matching (threshold 0.5) could otherwise produce incorrect enrichment.

### Data-Only Container Pattern
The architectural choice to ship a data-only container (`sleep infinity` on `ubi9-micro`) rather than a running service is notable. This pattern:
- Eliminates runtime security surface (no processes, no network)
- Simplifies updates (rebuild container with new data, redeploy)
- Decouples data generation from data consumption
- Allows other components to use init containers or volume mounts to access the data

### Concurrent Processing
The CLI tool uses Go goroutines with a semaphore channel pattern for concurrent model processing (default 5 concurrent). This is important because OCI registry and HuggingFace API operations involve network I/O latency. The `sync.WaitGroup` ensures all models complete before catalog generation.

### Static Catalog Merging
The supplemental catalog (`input/supplemental-catalog.yaml`) contains 7 manually curated Granite and Prometheus model entries with full metadata including detailed README content. These are merged with dynamically extracted models during catalog generation, with deduplication by case-insensitive name comparison. This hybrid approach allows high-quality manual curation for important models while scaling to many models via automation.

### Legacy Format Migration
The codebase maintains backward compatibility with older metadata formats through `LoadExistingMetadata` in `internal/metadata/migration.go`, which tries three different format parsers in sequence (new format with structured OCI artifacts, mixed-type timestamps, then legacy format with string artifacts). This is important for incremental migration of existing metadata files.

### Logo Embedding
The catalog generation embeds base64-encoded SVG logo data URIs directly into the catalog YAML based on model tags. Models tagged as "validated" receive a Red Hat AI logo. This self-contained approach avoids external URL dependencies for logo display.
