## Purpose

**Short**: High-throughput LLM inference server optimized for Intel Gaudi (Habana) accelerators, providing OpenAI-compatible and gRPC APIs for serving large language models on RHOAI.

**Detailed**: vLLM Gaudi is a fork of the vLLM project customized for Intel Gaudi (Habana) HPU accelerators. It provides a FastAPI-based REST API server that implements the OpenAI Chat Completions, Completions, Embeddings, and Models APIs, enabling drop-in replacement for OpenAI endpoints. The server supports PagedAttention for efficient KV cache management, continuous batching, tensor parallelism, pipeline parallelism, and speculative decoding — all optimized for the Gaudi hardware platform.

In RHOAI, vLLM Gaudi serves as the model serving runtime for InferenceService workloads running on Intel Gaudi accelerators. It is deployed as a container within KServe-managed pods. The repository produces two container image variants: an OpenAI-compatible HTTP server (`vllm-openai`) and a gRPC adapter (`vllm-grpc-adapter`) that wraps vLLM with the TGIS (Text Generation Inference Service) gRPC protocol used by Caikit and the KServe data plane. The gRPC adapter enables compatibility with RHOAI's existing model serving infrastructure that expects the TGIS protocol.

## Architectural Analysis

vLLM Gaudi is a downstream fork of the upstream vLLM project, customized for Intel Gaudi (Habana) accelerators. The architecture follows the standard vLLM design with platform-specific adaptations at the worker, attention backend, and device communicator layers. The `HpuPlatform` class in `vllm/platforms/hpu.py` acts as the integration point, overriding device detection, worker selection, attention backend routing, and configuration defaults (e.g., block_size=128 vs. 16-32 on CUDA).

The Dockerfile produces two container image targets from a single multi-stage build: `vllm-openai` (the primary OpenAI-compatible REST API server) and `vllm-grpc-adapter` (which extends `vllm-openai` by adding the `vllm-tgis-adapter` package). This design means the gRPC adapter image includes all OpenAI API capabilities but uses the TGIS gRPC entrypoint instead. The TGIS adapter is critical for RHOAI because it enables compatibility with the Caikit runtime and KServe's gRPC data plane, which are the standard model serving protocols in the platform.

A notable architectural concern is the non-hermetic build. Unlike most RHOAI components that are transitioning to hermetic Konflux builds, vLLM Gaudi explicitly sets `hermetic: false`. This is likely driven by the complex dependency chain: the Habana vault base image, large binary dependencies (PyTorch, Triton, Habana runtime), and the `vllm-hpu-extension` git dependency don't easily fit Hermeto prefetch workflows. The dependency on `vllm-hpu-extension` is pinned to a specific git commit (29bc7a4), providing some reproducibility, but the broader dependency graph uses version ranges that can float between builds.

The distributed inference architecture supports both tensor parallelism and pipeline parallelism via an HPU-specific communicator (`HpuCommunicator`) that works around Habana's lazy collective behavior with explicit `mark_step()` synchronization. The worker selection logic automatically chooses between standard, multi-step, and speculative decoding workers based on the inference configuration. The V1 engine path (`VLLM_USE_V1=1`) is experimental on HPU and provides an alternative async execution model.

Security is handled primarily at the deployment layer rather than the application layer. vLLM provides optional API key authentication and TLS support (with certificate hot-reload), but in RHOAI deployments, authentication and authorization are typically enforced by KServe's infrastructure (kube-rbac-proxy sidecars, Gateway API routing). The container runs as a non-root user (UID 2000, GID 0) with OpenShift-compatible permissions, following RHOAI security practices.
