## Purpose

**Short**: Feast is an open-source feature store that manages and serves machine learning features for both training and online inference, deployed and managed on Kubernetes/OpenShift via the Feast operator.

**Detailed**: Feast (Feature Store) provides a centralized platform for managing, storing, and serving machine learning features. It enables data scientists and ML engineers to define, register, and retrieve features for both offline training (batch) and online inference (real-time serving) use cases. The component consists of multiple services: a feature registry for metadata management, an online store for low-latency feature retrieval, an offline store for historical feature access, and an optional UI for visualization.

The Feast operator (`feast-operator`) is a Kubernetes controller-runtime based operator that manages the full lifecycle of FeatureStore Custom Resources (CRs). When a user creates a FeatureStore CR, the operator deploys and configures all necessary Feast services (registry, online store, offline store, UI) as containers within a single Deployment, creates Kubernetes Services for network access, configures TLS (automatically on OpenShift via service serving certificates), sets up RBAC for Kubernetes-native or OIDC authorization, and manages CronJobs for periodic feature materialization. The operator supports multiple persistence backends including PostgreSQL, Redis, Snowflake, DuckDB, and file-based storage, with credentials managed through Kubernetes Secrets.

In the RHOAI distribution, Feast is deployed into the `redhat-ods-applications` namespace. The operator is built with FIPS-compliant crypto (GOEXPERIMENT=strictfipsruntime) and the feature server image is based on UBI9 Python 3.11. The operator also supports a Go-based feature server for high-performance online feature serving with HTTP and gRPC interfaces, OpenTelemetry tracing, and support for Redis, SQLite, and DynamoDB online stores.

## Architectural Analysis

The Feast component follows a hub-and-spoke architecture where the operator acts as the hub, managing the lifecycle of all Feast service spokes. A notable design decision is the single-Deployment multi-container pattern: the operator deploys registry, online store, offline store, and UI as separate containers within a single Kubernetes Deployment. This simplifies lifecycle management and ensures all services share the same `feature_store.yaml` configuration, but limits independent scaling of individual services. The DeploymentStrategy field allows users to configure rolling update or recreate strategies, but per-service scaling requires separate FeatureStore CRs.

The TLS architecture demonstrates thoughtful platform integration. On OpenShift, the operator leverages service serving certificates via annotations (`service.beta.openshift.io/serving-cert-secret-name`), which provides automatic certificate provisioning, rotation, and CA bundle injection without requiring external certificate management. On non-OpenShift Kubernetes, users must provide TLS secrets manually. The operator handles both modes transparently, adjusting port numbers (HTTP vs HTTPS variants) and Feast CLI arguments accordingly. The `tls.go` file manages this dual-mode logic, including configuring SANs for services that expose both gRPC and REST endpoints on the same certificate.

The authorization model supports two mutually exclusive modes: Kubernetes RBAC and OIDC. In Kubernetes RBAC mode, the operator creates a comprehensive set of Roles and ClusterRoles that allow the Feast ServiceAccount to perform TokenReview and SubjectAccessReview calls, enabling the Python Feast server to validate incoming requests against Kubernetes RBAC policies. Notably, the operator creates empty custom Roles (with no rules) based on user-specified role names, allowing administrators to define fine-grained Feast permission policies mapped to Kubernetes RBAC roles. The cluster-scoped `feast-token-review-cluster-role` is shared across all FeatureStore instances as a singleton.

The CronJob implementation uses an interesting exec-based approach rather than running Feast commands directly. CronJob pods use the `ose-cli` (kubectl) image and execute `kubectl exec` into the Feast Deployment pod to run commands like `feast apply` and `feast materialize-incremental`. This avoids duplicating the Feast SDK installation in the CronJob image but requires pods/exec RBAC permissions. The CronJob is created in a suspended state by default (schedule: `@yearly`, suspend: true), requiring explicit user configuration to activate.

The Go feature server provides a high-performance alternative to the Python feature server for online inference workloads. It supports HTTP and gRPC interfaces, Redis/SQLite/DynamoDB online stores, and includes comprehensive OpenTelemetry instrumentation with per-module tracers. However, it lacks the full Feast SDK functionality — it only serves online features and cannot manage the registry or offline store. The Go server's transformation pipeline supports remote gRPC transformation services, but notably uses insecure gRPC credentials for this connection, which is a potential security concern in production environments.

The namespace registry (`feast-configs-registry` ConfigMap) provides a cluster-wide service discovery mechanism for Feast instances. It maintains a JSON mapping of namespaces to their Feast client configurations, stored in the operator's namespace and readable by all authenticated users (`system:authenticated` group). This enables applications to discover available Feast services without knowing specific namespace details.
