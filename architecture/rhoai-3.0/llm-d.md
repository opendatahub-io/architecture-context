## Purpose

**Short**: llm-d is a Kubernetes-native distributed inference serving stack that provides optimized container images, deployment guides, and Helm/Kustomize recipes for high-performance LLM serving across multiple hardware accelerators.

**Detailed**: llm-d is the umbrella repository for the llm-d project, a CNCF sandbox project founded by Red Hat, Google Cloud, IBM Research, CoreWeave, and NVIDIA. Unlike typical service repositories, llm-d does not contain application source code. Instead, it serves as the integration point where vLLM (the inference engine), the llm-d Router (intelligent load balancer), and supporting infrastructure are composed into tested, benchmarked deployment patterns called "well-lit paths."

The repository produces container images for vLLM across four accelerator families (NVIDIA CUDA, AMD ROCm, Intel Gaudi HPU, and CPU) plus an RDMA diagnostic tools image. Each image bundles the complete inference runtime stack: vLLM, NIXL (for RDMA-based KV cache transfer), LMCache, UCX, and accelerator-specific libraries like NVSHMEM, DeepEP, DeepGEMM, and FlashInfer. The deployment guides in `guides/` provide Kustomize overlays and Helm values files that compose the llm-d Router, Gateway API (Istio or AgentGateway), and model server deployments into production-ready topologies supporting optimized baseline routing, prefill/decode disaggregation, wide expert parallelism, tiered KV cache offloading, flow control, and workload autoscaling.

llm-d integrates with the Kubernetes Inference Gateway (GAIE) project, using the InferencePool CRD and HTTPRoute resources to connect Gateway API infrastructure with the Endpoint Picker (EPP) for intelligent, LLM-aware request routing. The platform targets RHOAI (Red Hat OpenShift AI) and upstream Kubernetes deployments on OpenShift, GKE, and generic CKS clusters.

## Architectural Analysis

llm-d is architecturally unique among RHOAI components because it is a meta-repository rather than a traditional service or operator. It does not contain Go controllers, Python services, or TypeScript frontends. Instead, it is the integration hub where upstream open-source components (vLLM, NIXL, Istio, Gateway API) are composed into tested deployment patterns. This design reflects the project's principle of "composition over configurability" -- rather than building a monolithic inference platform, llm-d provides well-tested assembly instructions for combining best-of-breed components.

The container image architecture follows a layered build strategy with clear separation between the builder stage (compiling CUDA kernels, RDMA libraries, and vLLM extensions) and the runtime stage (minimal image with only runtime dependencies). The CUDA Dockerfile is the most complex, building from source: gdrcopy, UCX, NIXL, LMCache, InfiniStore, DeepEP, DeepGEMM, and FlashInfer before installing vLLM itself. The version pinning strategy uses Git commit SHAs in `docker/common-versions` rather than semantic version tags, providing exact reproducibility at the cost of readability. The GB200 (NVIDIA Blackwell) variant is supported as a separate Docker build target (`--target runtime-gb200`) that swaps in GB200-specific DeepEP wheels.

The deployment guide structure uses a composable Kustomize hierarchy where base manifests in `guides/recipes/` define the common deployment topology (decode Deployment, prefill Deployment, ServiceAccount, Gateway) and per-guide overlays in `guides/{optimized-baseline,pd-disaggregation,wide-ep-lws,...}/` apply accelerator-specific and pattern-specific patches. This structure allows a single base decode Deployment to be reused across all accelerators (NVIDIA, AMD, Intel, Google TPU) by applying different image and resource patches. The scheduler configuration follows a similar pattern: `guides/recipes/scheduler/base.values.yaml` provides the common EPP Helm values, while `pd.values.yaml` adds the disaggregation plugin pipeline.

The security posture is notably minimal for a production inference platform. Model server pods run as non-root user (UID 2000) and disable service account token auto-mounting, but the inference API itself (8000/TCP) has no authentication or TLS. RDMA KV cache transfers between pods (5600/TCP) are completely unencrypted and unauthenticated, relying entirely on network-level isolation (RDMA fabric segmentation, Kubernetes NetworkPolicies). This is an intentional performance trade-off -- adding encryption to the RDMA path would negate the latency benefits of disaggregated serving. For production RHOAI deployments, security would be enforced at the Gateway layer (TLS termination, authentication) and the OpenShift network layer (NetworkPolicies, RBAC), not within the model server pods.

The multi-cloud support is reflected in platform-specific overlay directories (CKS, OCP, GKE) primarily for the monitoring and autoscaling stack. The Prometheus endpoint configuration, TLS certificate handling, and ServiceMonitor labels differ across platforms. The gateway provider configuration similarly offers Istio (default), AgentGateway (formerly kgateway), and GKE L7 load balancer options, each with its own Helmfile and gateway manifests. This design ensures llm-d can be deployed on any Kubernetes distribution while taking advantage of platform-specific infrastructure.
