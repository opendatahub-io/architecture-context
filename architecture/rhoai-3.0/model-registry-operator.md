## Purpose

**Short**: Kubernetes operator that manages the lifecycle of Model Registry instances, deploying and configuring REST/gRPC services, database backends, and authentication proxies for ML model metadata management.

**Detailed**: The Model Registry Operator is a controller-runtime based Kubernetes operator that reconciles `ModelRegistry` custom resources (CRD group `modelregistry.opendatahub.io`). When a user creates a ModelRegistry CR, the operator provisions a complete model registry stack: a REST API server (model-registry), a gRPC metadata server (MLMD/mlmd-grpc-server), an optional auto-provisioned PostgreSQL database, and an optional kube-rbac-proxy sidecar for authentication and authorization.

The operator supports two API versions: v1alpha1 (deprecated, with Istio service mesh integration) and v1beta1 (current storage version, with kube-rbac-proxy support replacing Istio). A conversion webhook automatically migrates resources between versions, and the operator includes a storage version migration system (using both Kubernetes StorageVersionMigration API and a manual fallback strategy) to ensure all persisted resources are upgraded to the v1beta1 storage format.

In addition to per-instance model registries, the operator manages a shared "model catalog" subsystem. When enabled, the ModelCatalogReconciler deploys a catalog service with its own PostgreSQL database, kube-rbac-proxy, and data initialization from pre-built container images containing model metadata collections. The operator detects OpenShift clusters at startup and conditionally creates OpenShift-specific resources (Routes, Groups, ClusterRoleBindings) for platform integration.

## Architectural Analysis

The Model Registry Operator follows a standard kubebuilder/controller-runtime pattern with two reconcilers sharing a common template-based resource management layer. The `TemplateApplier` (in `internal/controller/util.go`) uses Go `text/template` to render Kubernetes manifests from `.yaml.tmpl` files at runtime, applying ModelRegistry spec fields as template parameters. This approach decouples resource definitions from Go code but means the actual deployed resources are not visible in static manifests alone -- they are generated dynamically from 28 template files in `internal/controller/config/templates/`.

The operator manages a notable API version migration path. The v1alpha1 API included Istio service mesh integration (Gateway, VirtualService, AuthorizationPolicy via Authorino), while v1beta1 replaces this with a simpler kube-rbac-proxy sidecar pattern. The defaulting webhook automatically migrates v1alpha1 Istio configurations to OAuthProxy (v1alpha1) or KubeRBACProxy (v1beta1) configurations. The conversion webhook handles structural field mapping between versions, and the StorageMigrationManager runs as a background goroutine to ensure all etcd-stored resources are re-written in the v1beta1 storage format. This three-layer migration (webhook defaulting, CRD conversion, storage migration) is architecturally significant because it means the operator must support both old and new auth patterns simultaneously during upgrades.

The kube-rbac-proxy sidecar pattern deserves attention. For each ModelRegistry instance with proxy enabled, the operator creates: (1) a ConfigMap defining SubjectAccessReview authorization policy (verb=get on the instance's Service resource), (2) a ClusterRoleBinding to `system:auth-delegator` for the instance ServiceAccount, (3) an OpenShift Route with `reencrypt` TLS termination pointing to the proxy port, and (4) a NetworkPolicy allowing ingress only from the OpenShift router namespace. This effectively gates REST API access behind Kubernetes RBAC -- users need `get` permission on the registry's Service object to access the API through the proxy.

The ModelCatalogReconciler operates differently from the main reconciler. Instead of watching a CR, it uses fixed resource names (`model-catalog`, `model-catalog-postgres`) and is triggered by changes to any of its owned resources. It manages two ConfigMaps for catalog sources: a "default sources" ConfigMap populated from a data init container, and a "user sources" ConfigMap that preserves user modifications. The reconciler carefully avoids overwriting user customizations while ensuring the default catalog entry can be removed if the user opts out. The catalog subsystem can be disabled entirely via the `ENABLE_MODEL_CATALOG` environment variable.

A security consideration is the auto-provisioned PostgreSQL deployment. When `spec.postgres.generateDeployment=true`, the operator creates a PostgreSQL pod with credentials stored in a Secret. The default catalog database uses hardcoded default credentials (`catalog_user`/`catalog_password_change_me`), which is noted in the code but represents a security gap for production deployments. The PostgreSQL container runs as non-root with the `registry.redhat.io/rhel9/postgresql-16` image, and storage is provided via PVC.
