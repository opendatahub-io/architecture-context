## Purpose

**Short**: Manages the lifecycle of Ray clusters, jobs, and serving applications on Kubernetes/OpenShift via custom resource definitions.

**Detailed**: KubeRay is a Kubernetes operator that provides first-class management of Ray distributed computing workloads. It defines and reconciles three primary CRDs — RayCluster, RayJob, and RayService — enabling users to declaratively create and manage Ray clusters, submit batch jobs, and deploy Ray Serve applications with zero-downtime upgrades. The operator handles the complete lifecycle including pod creation, service exposure, autoscaler integration, and cleanup.

In the RHOAI distribution, KubeRay includes significant OpenShift-specific enhancements: Gateway API integration with HTTPRoute-based ingress, kube-rbac-proxy sidecar injection for OIDC/OAuth authentication, NetworkPolicy-based isolation for secure clusters, mTLS certificate management via cert-manager, and SecurityContextConstraints for running Ray workloads under restricted policies. The operator creates cross-namespace ReferenceGrants to enable the platform's central Gateway to route traffic to Ray clusters in user namespaces.

The operator integrates with the broader RHOAI platform through the `data-science-gateway` Gateway CR managed by the rhods-operator, enabling authenticated external access to Ray dashboards. It also supports optional batch scheduling integration with Volcano, YuniKorn, and Kubernetes scheduler-plugins for gang scheduling of distributed Ray workloads.

## Architectural Analysis

### Dual Authentication Architecture

The KubeRay operator implements a sophisticated dual authentication model. On OpenShift, the mutating webhook automatically sets `odh.ray.io/secure-trusted-network=true` on all RayCluster CRs, triggering the authentication controller to provision a complete auth stack per cluster. This stack consists of a kube-rbac-proxy sidecar container (on port 8443/TCP) that fronts the Ray Dashboard, a ConfigMap with RBAC proxy configuration, a ServiceAccount for the proxy, and an HTTPRoute in the platform namespace pointing to the kube-rbac-proxy port. A shared ReferenceGrant per namespace allows the platform-namespace HTTPRoute to reference Services in user namespaces. The operator supports two authentication modes (ModeIntegratedOAuth and ModeOIDC), though currently `DetectAuthenticationMode()` returns `ModeIntegratedOAuth` for all clusters — both modes now use kube-rbac-proxy with OIDC configuration, unifying the implementation.

### Cross-Namespace Gateway API Pattern

The Gateway API integration follows a cross-namespace pattern: HTTPRoutes are created in the platform namespace (configured via `APPLICATION_NAMESPACE`) to colocate them with the `data-science-gateway` Gateway CR managed by rhods-operator. ReferenceGrants in each user namespace authorize these HTTPRoutes to reference the cluster's head service. This is a security-conscious design — the platform operator controls the Gateway, and per-namespace ReferenceGrants provide explicit cross-namespace authorization. The ReferenceGrant lifecycle uses reference counting: it is only deleted when the last cluster in a namespace disables authentication, preventing disruption to other clusters.

### mTLS Certificate Architecture

The mTLS controller creates a two-tier PKI per cluster: a self-signed Issuer bootstraps a root CA Certificate (10-year duration), which backs a CA Issuer that signs per-pod certificates (90-day duration, 15-day renewal). Head and worker certificates include dynamic SANs updated whenever pod IPs change. The controller polls every minute to detect certificate expiry and IP changes, a pragmatic approach that avoids watch-based complexity for cert-manager resources. Certificate secrets use a UID-based naming convention (`{cluster}-ca-secret-{first_8_chars_of_uid}`) to prevent name collisions when clusters are deleted and recreated.

### NetworkPolicy Isolation Model

When secure-trusted-network is enabled, the NetworkPolicy controller creates two policies per cluster: one for the head pod and one for worker pods. Head pod policies allow intra-cluster traffic on all ports, namespace-local traffic on ports 10001 and 8265 (client and dashboard), operator namespace traffic for management, optional monitoring namespace access on port 8080, optional gateway namespace access on port 8265, and auth-proxy traffic on port 8443 from anywhere. Worker policies are simpler, allowing only intra-cluster traffic. The operator detects its own namespace through a priority chain (APPLICATION_NAMESPACE → POD_NAMESPACE → defaults), with OpenShift-specific defaults (`redhat-ods-applications`) and vanilla K8s defaults (`ray-system`).

### Controller Interaction Pattern

The six controllers operate on the same RayCluster CR but are carefully coordinated. The RayCluster controller waits for the Authentication controller to create the ServiceAccount before creating pods, preventing race conditions where pods might start without auth sidecars. The mTLS controller operates independently via polling, creating certificates that the RayCluster controller checks before allowing pod creation. The NetworkPolicy controller is fully independent, creating policies based on annotations. This loose coupling allows individual controllers to be disabled via feature gates without breaking the overall system.
