## Purpose

**Short**: Implements Ragas evaluation as an out-of-tree Llama Stack provider, enabling LLM evaluation via the Llama Stack eval API with both inline (in-process) and remote (Kubeflow Pipelines) execution modes.

**Detailed**: The llama-stack-provider-ragas component bridges the Ragas evaluation framework with Meta's Llama Stack platform. It registers as an external provider for the Llama Stack `eval` API, allowing users to run standardized LLM evaluation metrics (faithfulness, answer relevancy, context precision, context recall) against models served through Llama Stack's inference API.

The provider ships in two execution modes. The **inline** provider runs evaluations directly within the Llama Stack server process, wrapping the Ragas library to use Llama Stack's inference and embedding APIs instead of LangChain's default providers. The **remote** provider offloads evaluation to Kubeflow Pipelines (Data Science Pipelines in RHOAI), constructing a two-step KFP pipeline: (1) retrieve the dataset from Llama Stack, then (2) run Ragas evaluation inside a dedicated container image. Results are written to S3 and fetched back through the Llama Stack eval job API.

The Konflux-built container image (`odh-trustyai-ragas-lls-provider-dsp-rhel9`) is the pipeline step image used by Kubeflow Pipelines for remote evaluations. It contains the provider package with remote dependencies (kfp, kubernetes client, s3fs) pre-installed. The base image resolution follows a three-tier fallback: environment variable `KUBEFLOW_BASE_IMAGE` > Kubernetes ConfigMap `trustyai-service-operator-config` in `redhat-ods-applications` or `opendatahub` namespaces > default public image.

## Architectural Analysis

The llama-stack-provider-ragas component follows a clean provider plugin architecture. The provider registers with the Llama Stack server at startup through `get_provider_spec()`, which returns either an `InlineProviderSpec` or `RemoteProviderSpec` (or both, depending on installed dependencies). The conditional import pattern in `__init__.py` gracefully degrades — if `kfp`, `kubernetes`, and `s3fs` are not installed, only the inline provider is available. This avoids heavyweight dependencies for simple local evaluation scenarios.

The wrapper layer (in `wrappers_inline.py` and `wrappers_remote.py`) adapts Llama Stack's inference API to Ragas' expected interfaces. Ragas inherits from LangChain's `BaseRagasLLM` and `BaseRagasEmbeddings` abstractions, so the wrappers translate between Llama Stack's OpenAI-compatible completion/embedding API and LangChain's `LLMResult`/`Generation` types. The inline wrappers use the in-process inference API directly (async), while the remote wrappers instantiate `LlamaStackClient` / `AsyncLlamaStackClient` HTTP clients to call back to the Llama Stack server from within KFP pods. This callback pattern means the Llama Stack server must be network-reachable from the Kubeflow namespace.

The remote execution model has a notable security consideration: the KFP pipeline pods receive AWS credentials via Kubernetes Secret injection (`kubernetes.use_secret_as_env` in `pipeline.py`), and the Llama Stack server URL is passed as a plain string parameter. There is no mTLS or token-based authentication between KFP pods and the Llama Stack server — traffic flows over HTTP. This is acceptable for intra-cluster communication but represents a risk if the Llama Stack server is exposed externally or across network boundaries. The KFP API itself is authenticated via bearer token (either from an environment variable or extracted from the local kubeconfig), with a pre-flight health check (`/apis/v2beta1/healthz`) before client initialization.

The build pipeline is explicitly non-hermetic (`hermetic: false` in the Tekton PipelineRun), meaning dependencies are fetched from PyPI at build time. While `uv.lock` and `requirements.txt` with hashes exist for reproducibility, the Dockerfile.konflux installs via `pip install -e ".[remote]"` which resolves from `pyproject.toml` rather than the lockfile. The `rpms.lock.yaml` is absent, meaning OS-level packages in the UBI9 base image are not version-locked. This is a supply chain gap worth noting for a component that handles evaluation data and credential secrets.
