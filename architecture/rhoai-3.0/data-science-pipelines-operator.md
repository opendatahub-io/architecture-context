## Purpose

**Short**: Manages the lifecycle of Kubeflow Data Science Pipelines (DSP) installations on OpenShift, deploying and configuring all sub-components (API server, persistence agent, scheduled workflow controller, workflow controller, ML Metadata, database, and object storage) based on a single DataSciencePipelinesApplication custom resource.

**Detailed**: The Data Science Pipelines Operator (DSPO) is a Kubernetes operator built with controller-runtime that watches `DataSciencePipelinesApplication` (DSPA) custom resources and reconciles the full stack of Kubeflow Pipelines v2 components into the target namespace. Each DSPA CR declares the desired state of a complete pipelines installation — including the API server, persistence agent, scheduled workflow controller, Argo workflow controller, ML Metadata (MLMD) with Envoy proxy, and backing infrastructure (MariaDB for metadata, MinIO or external S3 for artifact storage).

The operator uses a template-driven approach via the Manifestival library to render Go templates (`config/internal/`) into Kubernetes manifests, applying them with owner references for automatic garbage collection. It performs health checks against the database and object store before deploying dependent components, manages TLS certificates for pod-to-pod encryption via OpenShift's service-ca operator, and composes CA bundles from multiple sources (ODH trusted CA, user-provided certs, service-ca certs). The operator also deploys kube-rbac-proxy sidecars alongside the API server and MLMD Envoy proxy to enforce OpenShift RBAC on external-facing routes.

DSPO is deployed by the platform operator (rhods-operator for RHOAI) into the `redhat-ods-applications` namespace. Users create DSPA CRs in their data science project namespaces, and the operator deploys all pipeline components into those namespaces with appropriate network policies, RBAC, and monitoring configuration.

## Architectural Analysis

The Data Science Pipelines Operator follows a clean "operator deploys infrastructure" pattern where a single CRD (`DataSciencePipelinesApplication`) declaratively describes an entire Kubeflow Pipelines installation. The operator's reconciliation loop is sequential and dependency-aware: it first ensures database and object storage are accessible (with configurable health check timeouts), then deploys the API server, persistence agent, scheduled workflow controller, workflow controller, and MLMD stack in order. This sequential approach prevents components from starting before their dependencies are ready, avoiding crash loops during initial deployment.

The template-driven architecture using Manifestival is a distinctive design choice. Rather than constructing Kubernetes objects programmatically in Go code, the operator reads Go template files from `config/internal/`, renders them with reconciliation parameters (`DSPAParams`), and applies the resulting manifests. This makes the deployed resource structure highly visible — each `.yaml.tmpl` file directly shows what gets deployed. The template parameters struct (`DSPAParams`) contains 56+ fields that capture the entire reconciliation state, including database connection details, object storage endpoints, TLS configuration, CA bundles, and image references. The configuration hash mechanism (SHA256 of sample config + workspace JSON) ensures pods restart when their configuration changes.

The security model is layered and thorough. External access goes through kube-rbac-proxy sidecars that perform SubjectAccessReview checks against the `datasciencepipelinesapplications/api` subresource, meaning users must have RBAC permissions on the specific DSPA instance to access its API. Internal communication between components is protected by NetworkPolicies that restrict traffic to only the required pod-to-pod paths. Pod-to-pod TLS is enabled by default (`podToPodTLS: true`), leveraging OpenShift's service-ca operator to automatically generate and rotate TLS certificates. The CA bundle composition logic is notably complex — it merges certificates from three sources (ODH global trust, user-provided custom CAs, and service-ca certs) into a single `dsp-trusted-ca` ConfigMap, then sets `SSL_CERT_DIR` to include both the composed bundle and the system certificate path.

The FIPS compliance implementation spans both build-time (Go binary compiled with `GOEXPERIMENT=strictfipsruntime` and `CGO_ENABLED=1` against OpenSSL) and runtime behavior (selecting a FIPS-specific launcher command `launcher-v2-fips` when `FIPSEnabled` is true). The database TLS configuration offers multiple modes including `true` (proper CA validation), `skip-verify` (InsecureSkipVerify), and `preferred` — the latter two exist for development/migration scenarios but represent potential FIPS compliance gaps if used in production. The default TLS mode is `true` for external databases and `false` for internal MariaDB (which uses service-ca certs when PodToPodTLS is enabled).

The webhook subsystem (`controllers/webhook.go`) is conditionally deployed only when a DSPA uses `pipelineStore: kubernetes` (instead of the default `database` mode). This webhook intercepts PipelineVersion CRs for validation and mutation. The webhook lifecycle management includes cleanup logic that checks whether any remaining DSPAs in the cluster use Kubernetes-based storage before removing webhook resources — a careful handling of shared cluster-scoped resources.
