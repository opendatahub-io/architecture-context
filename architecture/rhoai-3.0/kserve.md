## Purpose

**Short**: KServe is a Kubernetes-native model serving platform that manages the full lifecycle of inference services, including model deployment, scaling, routing, and monitoring.

**Detailed**: KServe provides a set of Custom Resource Definitions (CRDs) and controllers for deploying and managing machine learning model inference on Kubernetes. The core InferenceService (v1beta1) CRD supports multiple deployment modes (Serverless via Knative, RawDeployment, and ModelMesh), with pluggable runtimes (ServingRuntime), multi-model serving (TrainedModel), and inference graphs (InferenceGraph) for complex multi-step inference pipelines.

KServe extends beyond traditional model serving with the LLMInferenceService (v1alpha1) CRD, purpose-built for large language model workloads. This CRD supports multi-node deployments via LeaderWorkerSet, prefill-decode disaggregation for optimized LLM serving, and integration with the Gateway API Inference Extension (InferencePool/InferenceModel) for intelligent request scheduling. The LLM serving pipeline includes a scheduler (Endpoint Picker Protocol) for load-aware routing, self-signed TLS for workload communication, and Prometheus monitoring integration.

On RHOAI, KServe defaults to RawDeployment mode with OpenShift-specific adaptations: kube-rbac-proxy for authentication (replacing oauth-proxy), OpenShift Routes for InferenceGraph exposure, SecurityContextConstraints for GPU workloads (IPC_LOCK, SYS_RAWIO), Gateway API integration for ingress routing, Kuadrant AuthPolicy enforcement, and Istio DestinationRules for self-signed TLS. The storage subsystem supports S3, GCS, Hugging Face Hub, PVC, and OCI model sources with a local model caching system for distributing models to nodes via DaemonSet agents.

## Architectural Analysis

KServe exhibits a layered architecture with three distinct serving paradigms evolving in parallel. The **InferenceService** (v1beta1) represents the stable, general-purpose model serving path, supporting Serverless (Knative), RawDeployment (Kubernetes native), and ModelMesh modes. The **LLMInferenceService** (v1alpha1) is a purpose-built layer for large language models, introducing fundamentally different orchestration patterns including multi-node deployments (LeaderWorkerSet), prefill-decode disaggregation, and Gateway API Inference Extension integration. The **InferenceGraph** (v1alpha1) provides a DAG-based composition layer for multi-step inference pipelines. These three paradigms share the same controller binary but have largely independent reconciliation paths, CRD hierarchies, and networking stacks.

The RHOAI distribution significantly transforms the upstream architecture. The ODH overlay switches the default deployment mode to RawDeployment, replaces cert-manager with OpenShift serving certificate injection, substitutes oauth-proxy with kube-rbac-proxy (commit `e442d5eb5`), and configures the ingress to use the OpenShift AI inference gateway (`openshift-ingress/openshift-ai-inference`). The LLM serving stack introduces OpenShift-specific SecurityContextConstraints that grant GPU workloads `IPC_LOCK` and `SYS_RAWIO` capabilities (required for GPU direct memory access), with a separate multi-node SCC that additionally permits `hostPath` volumes for shared memory across pods. The ODH overlay also removes ClusterServingRuntimes and local model management, indicating these are managed differently in the RHOAI distribution.

The LLMInferenceService controller demonstrates a sophisticated resource ownership hierarchy, creating approximately 15 distinct Kubernetes resource types per LLM deployment. The self-signed TLS subsystem (`workload_tls_self_signed.go`) generates 4096-bit RSA certificates with 10-year validity and 2-year renewal buffers, mounted into both workload pods and scheduler deployments. When Istio is detected as the gateway controller, DestinationRules with `InsecureSkipVerify=true` are created to allow the mesh to accept these self-signed certificates — a pragmatic but security-relevant design choice for internal workload traffic. The security-critical `router_validation_ocp.go` implements a defensive pattern where the controller proactively deletes HTTPRoutes if the Kuadrant AuthPolicy CRD is unavailable, preventing unauthorized exposure of LLM endpoints.

The webhook infrastructure implements a multi-stage pod mutation pipeline: storage initializer injection → Istio CNI security context → agent injection (puller/logger/batcher) → metrics aggregator → modelcar injection. This ordered pipeline is crucial for correct sidecar composition, as each stage may modify volumes, environment variables, and container specifications that subsequent stages depend on. The `ClusterStorageContainer` CRD enables extensible storage backends through URI pattern matching (prefix or regex), allowing the platform to support arbitrary model storage systems without controller code changes.

The codebase shows an active migration from RHOAI 2.x patterns (oauth-proxy, OpenShift Routes) to 3.x patterns (kube-rbac-proxy, Gateway API). The recent commit history (`e442d5eb5` — "Replace oauth-proxy with kube-rbac-proxy", `b06318bed` — "Migrate oauth-proxy to kube-rbac-proxy image") confirms this transition. However, OpenShift Routes remain in the codebase for InferenceGraph exposure, indicating that Gateway API adoption is not yet complete across all CRD types. The `inferenceservice-config` ConfigMap in the ODH overlay references `oauthProxy` with kube-rbac-proxy image, preserving the field name for backward compatibility while swapping the implementation.
