## Purpose

**Short**: ODH Dashboard is the web-based management console for Red Hat OpenShift AI (RHOAI), providing a unified UI for managing data science workbenches, model serving, pipelines, model registries, and platform configuration.

**Detailed**: The dashboard serves as the primary user interface for the OpenShift AI platform. It is a monorepo containing a React/TypeScript frontend, a Node.js/Fastify BFF backend, and multiple Go-based BFF modules deployed as sidecar containers via Webpack Module Federation. The frontend uses PatternFly 6 for its design system, Redux for state management, and the OpenShift Dynamic Plugin SDK for Kubernetes API integration.

The BFF backend proxies Kubernetes API calls, aggregates data from multiple platform services (Data Science Pipelines, Model Registry, TrustyAI, Prometheus/Thanos, Llama Stack), and enforces namespace-scoped authorization. It watches 9 custom resource types to maintain an in-memory cache of platform state. Authentication is handled by a kube-rbac-proxy sidecar that validates tokens and injects user identity headers (`X-Auth-Request-User`, `X-Auth-Request-Groups`) before requests reach the application.

The modular architecture pattern enables independent development and deployment of feature modules (Gen AI, Model Registry, LM Eval) as federated micro-frontends with their own Go BFF backends, all running as sidecar containers in the same pod. This design allows feature teams to ship independently while sharing the core dashboard infrastructure.

## Architectural Analysis

The ODH Dashboard exemplifies a modular micro-frontend architecture within a Kubernetes-native platform. The core design pattern is a monorepo producing multiple container images — one primary dashboard and multiple sidecar BFF modules — all deployed in a single pod. This co-location strategy simplifies service discovery (sidecars communicate via localhost) while the Module Federation runtime enables independent frontend builds that share React, PatternFly, and Redux as singletons.

The authentication architecture follows the RHOAI 3.x pattern: kube-rbac-proxy sits in front of the main application on port 8443, terminating TLS with OpenShift-issued serving certificates and validating Bearer tokens via TokenReview against the Kubernetes API. It then injects `X-Auth-Request-User` and `X-Auth-Request-Groups` headers into upstream requests. The backend trusts these headers without re-validating tokens — a deliberate design choice that avoids double-validation overhead but means the backend must never be exposed without the proxy. The NetworkPolicy restricts ingress to ports 8443, 8043, and 8143 only, ensuring port 8080 (unauthenticated) is never reachable from outside the pod.

The BFF pattern is central to the architecture. The backend does not simply proxy Kubernetes API calls — it aggregates data from 9 watched CRD types into an in-memory cache, resolves feature flags from OdhDashboardConfig and DataScienceCluster status, and implements multi-layer authorization (namespace ownership, admin group membership, resource-level permission checks via SelfSubjectAccessReview). Service proxying to downstream components (Model Registry, Pipelines, TrustyAI) uses dynamic service discovery: the backend looks up Kubernetes Services and reads annotations or status fields to determine proxy targets, with fallback to in-cluster DNS names.

The Go BFF modules (gen-ai, model-registry) represent an evolution toward more complex per-feature backends. The gen-ai BFF integrates with KServe, Llama Stack, OpenAI-compatible APIs, and Model Context Protocol servers — a significantly richer set of integrations than the main Node.js backend. Both Go BFFs are built with `CGO_ENABLED=1` and `-tags strictfipsruntime` for FIPS compliance, while the Node.js backend inherits FIPS from the UBI9 base image's OpenSSL. This dual-language stack (TypeScript for the core, Go for ML-focused modules) reflects the platform's ML ecosystem being Go-heavy (Kubernetes, KServe, Kubeflow).

The kustomize manifest structure demonstrates clear separation between base resources and distribution-specific overlays. The `core-bases/base` provides the fundamental deployment, while `modular-architecture` adds sidecar containers via JSON6902 patches. Distribution variants (ODH, RHOAI shared/addon/onprem) layer on branding, app catalog entries, and configuration without modifying the core. The HTTPRoute definition references a parameterized Gateway name (`$(gateway-name)` defaulting to `data-science-gateway`), aligning with the RHOAI 3.x Gateway API ingress pattern where a platform-level Gateway in `openshift-ingress` namespace serves as the entry point for all components.
