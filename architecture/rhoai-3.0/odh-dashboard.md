## Purpose
**Short**: The ODH Dashboard is the primary web-based user interface for Red Hat OpenShift AI (RHOAI), providing a single pane of glass for managing data science projects, notebooks, model serving, pipelines, and platform configuration.

**Detailed**: The ODH Dashboard is a full-stack web application consisting of a React/TypeScript frontend (built with PatternFly UI framework and Webpack module federation) and a Node.js/Fastify backend that serves as a Backend-for-Frontend (BFF). The backend acts as a secure API proxy to the Kubernetes API server, enforcing user authorization through token pass-through and RBAC checks. It runs behind a kube-rbac-proxy sidecar that handles TLS termination and authentication before forwarding requests to the application.

The dashboard provides user workflows for creating and managing data science projects, launching Jupyter notebooks (workbenches), deploying models via KServe and ModelMesh, configuring model registries, managing data connections, running ML pipelines, and administering platform settings. It uses a modular architecture with Webpack Module Federation to dynamically load UI plugins for Model Registry and Gen AI features from separate sidecar containers within the same pod.

In RHOAI 3.x, the dashboard is exposed externally via Gateway API (HTTPRoute) referencing the platform's `data-science-gateway` Gateway, replacing the legacy OpenShift Route + oauth-proxy pattern. Authentication is handled by the kube-rbac-proxy sidecar which validates that users can list OpenShift projects before granting access. The backend then passes through the user's token for fine-grained Kubernetes RBAC on every API call.

### Authentication & Authorization

| Endpoint | Methods | Auth Mechanism | Enforcement Point | Policy |
|----------|---------|----------------|-------------------|--------|
| / (all external) | * | kube-rbac-proxy: checks user can list project.openshift.io/projects | kube-rbac-proxy sidecar (port 8443) | Users must be able to list OpenShift projects |
| /api/* (all API) | * | User token passthrough via X-Forwarded-Access-Token header | Backend (route-security.ts) | Token passed to Kubernetes API for fine-grained RBAC |
| /api/dashboardConfig | PATCH | Admin check (isUserAdmin) | Backend (secureAdminRoute) | Only dashboard admin users can modify configuration |
| /api/notebooks | POST/PATCH/DELETE | Namespace + resource ownership check | Backend (requestSecurityGuard) | Users can only manage their own notebooks |
| /_mf/* | * | Bearer token authorization header | Backend proxy (setAuthorizationHeader) | Token forwarded to module federation sidecars |

The kube-rbac-proxy configuration (kube-rbac-proxy-config ConfigMap) performs an authorization check using `resourceAttributes`:
- API Group: `project.openshift.io`
- Resource: `projects`
- Verb: `list`

This ensures only authenticated OpenShift users who can list projects are granted access. After authentication, user identity is forwarded to the backend via `X-Auth-Request-User` and `X-Auth-Request-Groups` headers.
