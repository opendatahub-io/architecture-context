## Purpose

**Short**: Provides GPU-accelerated training runtime container images, integration tests, and example workloads for the Distributed Workloads stack in Red Hat OpenShift AI.

**Detailed**: The distributed-workloads repository is a multi-purpose component in the RHOAI platform that serves three distinct roles. First, it produces GPU-accelerated container images used as training runtimes — these images are the execution environments where PyTorchJob and Ray workloads run on the cluster. The images include NVIDIA CUDA and AMD ROCm GPU toolkits combined with Python ML frameworks (PyTorch, DeepSpeed, Transformers, Flash Attention, NCCL) on a UBI9 base. These are shipped as productized RHOAI container images via Konflux.

Second, the repository contains a comprehensive Go-based integration test suite that validates the distributed workloads stack (Kubeflow Training Operator, Kueue, KubeRay) running on OpenShift AI clusters. The tests exercise PyTorchJob training, Ray cluster operations, Kueue quota management, and the newer Trainer v2 CustomTrainingRuntime API. A containerized test runner image packages these tests for CI execution.

Third, the repository provides example workloads and workshops demonstrating distributed training patterns including LLM fine-tuning with DeepSpeed, Stable Diffusion DreamBooth training, hyperparameter optimization with Ray Tune, and supervised fine-tuning with PEFT/LoRA. These examples serve as reference implementations for platform users.

## Architectural Analysis

The distributed-workloads repository has an unusual architecture for a platform component — it is not a service or operator but rather a container image factory combined with a test suite. Its primary deliverable is a matrix of GPU-accelerated training runtime images spanning two GPU vendors (NVIDIA CUDA, AMD ROCm), three CUDA versions (12.1, 12.4, 12.8), two ROCm versions (6.2, 6.4), and three PyTorch versions (2.4.1, 2.5.1, 2.8.0). This combinatorial matrix produces 6 training image variants plus 2 Ray image variants, each requiring specialized build toolchains (CUDA nvcc compilation, ROCm HIP compilation, Flash Attention from-source builds). The build complexity is reflected in the Dockerfile sizes and the non-hermetic build configuration.

A notable architectural pattern is the divergence between RHOAI-branded images (built via Konflux from the `rhoai-3.0` branch using `Dockerfile.konflux` files and pushed to `quay.io/rhoai/`) and community images (built from `main` branch using standard `Dockerfile` files and pushed to `quay.io/modh/`). The Konflux-built training images use pinned UBI9 base images (`:latest` tag), while the community Ray images use version-tagged bases (`9.6-1755735361`). The Tekton pipelines for RHOAI training images are synchronized from `konflux-central` and explicitly set `hermetic: false`, acknowledging that GPU toolkit installation from vendor repositories cannot currently be made hermetic.

The Go test suite is architecturally significant as it serves as the integration contract between multiple distributed workload components. It validates end-to-end workflows spanning Training Operator (PyTorchJob creation, MNIST training, SFT with fms-hf-tuning), KubeRay (RayCluster lifecycle, Ray Tune HPO), Kueue (quota enforcement, queue admission), and the newer Trainer v2 API (CustomTrainingRuntime). The test framework uses Gomega matchers and provides reusable helpers for namespace management, RBAC setup, pod log collection, and Prometheus metric querying. The `.env-rhoai` file pins all test image references to digest-based SHA256 references, ensuring reproducible test execution.

The RHOAI 3.0 branch shows several evolutionary changes: removal of AppWrapper and CodeFlare Operator references (commit `97bc7e2`), addition of Trainer v2 CustomTrainingRuntime tests, transition from `inject-oauth` to `inject-auth` annotations reflecting the RHOAI 3.x authentication model shift from oauth-proxy to kube-rbac-proxy, and the addition of training-hub as a dependency in the latest PyTorch 2.8.0 image. The InfiniBand/RDMA networking packages in CUDA training images indicate support for multi-node distributed training over high-performance interconnects, which is critical for large-scale LLM training workloads.
