## Purpose

**Short**: High-performance AI model inference server implementing KServe, TensorFlow Serving, and OpenAI-compatible APIs on top of the OpenVINO runtime.

**Detailed**: OpenVINO Model Server (OVMS) is a model serving platform built in C++ that exposes trained AI/ML models over gRPC and REST APIs. It supports multiple inference protocols — TensorFlow Serving (v1), KServe Predict v2, and OpenAI-compatible generative AI APIs (v3) — allowing clients in any language to perform remote inference. OVMS is optimized for Intel architectures and leverages the OpenVINO toolkit for inference execution across CPU and GPU devices.

The server provides a rich feature set including model versioning with hot-reload, DAG pipeline scheduling for multi-model workflows, MediaPipe graph execution for complex processing pipelines, and a continuous batching LLM executor for efficient large language model serving. It supports model storage on local filesystems, AWS S3, Google Cloud Storage, Azure Blob Storage, and HuggingFace Hub. For RHOAI, OVMS is the primary inference runtime backing KServe InferenceService deployments and is used as the serving container for OpenVINO-optimized models.

The RHOAI distribution builds OVMS from source (including OpenVINO) on UBI 9 with GPU support enabled, LTO optimizations, and MediaPipe/Python support. The resulting container image (`odh-openvino-model-server-rhel9`) runs as a non-root user and exposes configurable gRPC and REST ports for inference traffic.

## Architectural Analysis

**Modular Server Architecture**: OVMS employs a well-structured modular architecture where each major subsystem (gRPC, HTTP, model management, metrics, Python, profiling, HuggingFace pull) is encapsulated as a `Module` with a standard lifecycle (`start`/`shutdown`). Modules are registered in a thread-safe map and initialized in a carefully ordered sequence to satisfy dependencies — for example, the ServableManager module is created before gRPC/HTTP modules so health probes can respond quickly, but model loading starts last. This design allows clean shutdown in reverse order and supports conditional compilation (MediaPipe and Python can be disabled at build time).

**Multi-Protocol API Surface**: The server exposes three distinct API families on the same ports: TensorFlow Serving v1 (`/v1/...`), KServe Predict v2 (`/v2/...`), and OpenAI-compatible generative AI (`/v3/...`). The REST layer dispatches requests via regex-based URL matching to handler callbacks. The gRPC layer registers three service implementations (TFS Predict, TFS Model, KFS Inference) on a single gRPC server. Notably, the HTTP REST handler delegates to the gRPC KFS implementation internally — REST requests are converted to protobuf, processed through the gRPC path, and serialized back to JSON. This ensures consistent behavior across protocols but adds serialization overhead.

**LLM Serving via Continuous Batching**: The generative AI subsystem uses OpenVINO GenAI's `ContinuousBatchingPipeline` to serve LLM models efficiently. A dedicated `LLMExecutor` thread continuously calls `pipeline->step()` to process batched requests. Streaming responses flow through the Drogon HTTP server's separate thread pool (detected via `/completions` path substring), ensuring that long-running text generation doesn't block short inference requests. The system supports OpenAI-compatible chat completions with tool calling (Hermes3 format), structured output, visual language models (VLM), and Server-Sent Events streaming.

**No Built-in TLS or Authentication**: OVMS intentionally runs without TLS or authentication. The gRPC server uses `InsecureServerCredentials()`, and the Drogon HTTP server listens on plaintext. In RHOAI deployments, security is provided by the `kube-rbac-proxy` sidecar injected by the platform operator, which terminates TLS and enforces RBAC-based authorization before proxying to OVMS. This architectural separation means OVMS itself has no FIPS crypto concerns in its serving path — all cryptographic operations are delegated to the sidecar and platform infrastructure. However, the model download path (S3, GCS, Azure, HuggingFace) does use TLS via system OpenSSL and cloud SDK libraries.

**Build Complexity**: The Dockerfile.konflux is one of the most complex in the RHOAI platform — it builds OpenVINO from source (C++ with CMake), compiles OpenVINO Tokenizers, builds OVMS itself via Bazel, creates a C API shared library, and packages everything into a minimal UBI 9 image. The build uses a multi-stage Docker pattern with 5 stages (base_build → build → capi-build → pkg → release). Build arguments are controlled via `.konflux/build-args.conf` which pins OpenVINO source branches, enables LTO, sets GPU driver versions, and configures Bazel flags. The lack of Hermeto/cachi2 prefetch means builds are not fully hermetic — they download sources and packages from the internet during build.
