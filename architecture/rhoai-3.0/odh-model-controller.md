## Purpose

**Short**: A Kubernetes controller that automates model serving infrastructure management for KServe InferenceServices, LLMInferenceServices, and NVIDIA NIM integrations in the Red Hat OpenShift AI platform.

**Detailed**: The odh-model-controller bridges the gap between the KServe model serving framework and the RHOAI platform by managing the operational lifecycle of model deployments. When users deploy machine learning models via KServe InferenceService or LLMInferenceService custom resources, this controller automatically provisions the surrounding infrastructure: OpenShift Routes for external HTTPS access, Prometheus monitoring (ServiceMonitors, PodMonitors, metrics dashboards), RBAC policies for authentication and authorization, KEDA ScaledObjects for Prometheus-based autoscaling, and storage credential aggregation.

The controller also implements the LLMInferenceService lifecycle, a RHOAI-specific CRD that extends KServe for large language model serving with Gateway API integration, Kuadrant AuthPolicy management (for both authenticated and anonymous access), Istio EnvoyFilter creation for SSL termination, and tier-based RBAC. Additionally, it provides a complete NVIDIA NIM (NVIDIA Inference Microservices) integration layer through the NIM Account CRD, managing API key validation, model catalog synchronization, container registry authentication, and OpenShift Template creation for NIM-based ServingRuntimes.

The controller operates in RawDeployment mode for KServe (no Knative dependency) and integrates with the platform's Gateway API ingress stack, Kuadrant for policy-driven auth, and Istio for EnvoyFilter-based traffic management.

## Architectural Analysis

The odh-model-controller is a multi-domain operator that spans three distinct functional areas — KServe model serving infrastructure, LLM inference services with Gateway API auth, and NVIDIA NIM integration — unified under a single controller-runtime manager binary. This design keeps the operational footprint small (one pod) while managing a broad surface area of Kubernetes resources.

**Controller Architecture and Reconciler Composition**: The InferenceService controller uses a sub-reconciler pattern where the main reconciler delegates to specialized reconcilers (route, ClusterRoleBinding, metrics, ServiceMonitor, KEDA, Model Registry). Each sub-reconciler uses a `DeltaProcessor` to compute create/update/no-op decisions, reducing unnecessary API writes. The LLM controller extends this with conditional sub-reconciler registration — AuthPolicy and EnvoyFilter reconcilers are only loaded if the corresponding CRDs exist on the cluster (`AuthPolicyCRDAvailable`, `EnvoyFilterCRDAvailable`). This graceful degradation means the controller operates correctly in environments without Kuadrant or Istio installed.

**Dual Auth Model (RHOAI 3.x)**: The controller implements two distinct auth patterns that coexist. For KServe InferenceServices in RawDeployment mode, it creates OpenShift Routes with edge TLS — authentication is optional and external. For LLMInferenceServices, it implements a policy-driven auth model using Kuadrant AuthPolicies targeting the platform Gateway. The Gateway-level AuthPolicy enforces Kubernetes TokenReview + SubjectAccessReview (checking `get` verb on `llminferenceservices`), while per-HTTPRoute AuthPolicies can override with anonymous access. The EnvoyFilter adds an Envoy cluster definition pointing to Authorino's gRPC endpoint (port 50051) with TLS via the service CA certificate. This layered approach (Gateway AuthPolicy → EnvoyFilter → Authorino → RBAC) provides defense-in-depth.

**NIM Integration as Handler Chain**: The NIM Account controller uses a handler chain pattern (`ValidationHandler → ConfigMapHandler → TemplateHandler → PullSecretHandler`) where each handler performs one step of the integration lifecycle. The `ApiKeyManager` uses a lazy singleton pattern with time-based refresh (configurable via `validationRefreshRate` and `nimConfigRefreshRate`, default 24h). The controller creates OpenShift Templates (not raw ServingRuntimes) to allow users to instantiate NIM models through the Template mechanism, which is an OpenShift-specific pattern.

**CA Bundle Aggregation**: The ConfigMap controller watches multiple CA certificate sources (odh-trusted-ca-bundle with cluster CA + custom CA, openshift-service-ca.crt with service CA) and aggregates them into a single `odh-kserve-custom-ca-bundle` ConfigMap per namespace. This ensures KServe model servers can trust both cluster-internal and external CA chains. This is a cross-cutting concern that affects all InferenceService deployments.

**Security Considerations**: The webhook layer provides admission control at multiple levels — Pod mutation for Ray TLS injection, InferenceService validation to prevent creation in the controller namespace, InferenceGraph validation for URL namespace isolation, NIM Account singleton enforcement, and LLMInferenceService tier validation. The tier system (`tier-to-group-mapping` ConfigMap in `maas-api` namespace) maps user groups to tiers, with Roles and RoleBindings created per tier to control which groups can access which LLM services.
