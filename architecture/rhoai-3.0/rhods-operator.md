## Purpose

**Short**: Platform operator that provisions, configures, and manages all Red Hat OpenShift AI (RHOAI) components, services, ingress infrastructure, monitoring, and authentication.

**Detailed**: The rhods-operator is the central control plane for Red Hat OpenShift AI 3.x. It implements a two-tier custom resource model: a `DSCInitialization` singleton configures platform-wide infrastructure (namespaces, network policies, monitoring, authentication, CA bundles, hardware profiles), while a `DataScienceCluster` singleton declares which AI/ML components are enabled and their configuration. The operator watches these top-level CRs and dynamically creates child component CRs (Dashboard, Workbenches, KServe, Kueue, Ray, TrustyAI, ModelRegistry, DataSciencePipelines, TrainingOperator, ModelController, FeastOperator, LlamaStackOperator), each with its own reconciler that deploys the component's kustomize manifests from embedded `prefetched-manifests/`.

In RHOAI 3.x, the operator also deploys the entire platform ingress stack via a Gateway controller: a Kubernetes Gateway API `Gateway` CR in `openshift-ingress`, a `kube-auth-proxy` Deployment for centralized OAuth2/OIDC authentication, Istio `EnvoyFilter` CRs for ext_authz and header manipulation, `DestinationRule` CRs for TLS policy, and `HTTPRoute` CRs for routing. This replaces the RHOAI 2.x pattern of per-component OpenShift Routes with `oauth-proxy` sidecars.

The operator also manages a full observability stack (MonitoringStack, ThanosQuerier, TempoMonolithic/TempoStack, OpenTelemetryCollector, Instrumentation) via the Monitoring service controller, distributes trusted CA bundles to all namespaces via CertConfigMapGenerator, and manages platform RBAC via the Auth service controller.

#### Gateway Ingress Traffic Flow

```
Client Browser
    |
    v (HTTPS, port 443)
Gateway CR (data-science-gateway, ns: openshift-ingress)
    |
    v (Envoy data plane)
EnvoyFilter: ext_authz → kube-auth-proxy:8443/oauth2/auth
    |
    ├── Auth success: x-auth-request-user, x-auth-request-email, x-auth-request-access-token headers set
    |
    v (Envoy)
EnvoyFilter: Lua filter → converts x-auth-request-access-token to Authorization: Bearer header
    |
    v (via HTTPRoute matching)
Backend Service (component service in redhat-ods-applications namespace)
```

## Architectural Analysis

The rhods-operator implements a sophisticated multi-tier reconciliation architecture. At the top level, `DSCInitialization` and `DataScienceCluster` are singleton cluster-scoped CRs that drive all platform behavior. The DSCI controller acts as the platform bootstrapper: it creates namespaces with proper labels and pod security policies, deploys network policies scoped by platform type (SelfManagedRhoai, ManagedRhoai, ODH), creates the Auth and Monitoring service CRs, distributes CA bundles, and deploys default hardware profiles. The DSC controller then iterates through enabled components, creating their respective CRs, which trigger individual component reconcilers.

The action-based reconciliation pattern (`WithAction()` chain) used throughout the codebase provides a clean pipeline model. Each reconciler defines an ordered sequence of actions (initialize, render templates, deploy, update status, garbage collect) that execute sequentially within a single reconciliation loop. This pattern enables consistent resource lifecycle management across 13+ component controllers and 5 service controllers while keeping each action unit-testable. The `deploy.NewAction(deploy.WithCache())` pattern ensures server-side apply with caching to reduce API server load.

The Gateway controller is the most architecturally significant service, implementing the complete RHOAI 3.x ingress stack. It replaces per-component Routes and oauth-proxy sidecars with a centralized Gateway API approach: a single `Gateway` CR in `openshift-ingress` terminates TLS, EnvoyFilter provides ext_authz integration that routes every request through kube-auth-proxy for cookie/token validation, and a Lua filter converts OAuth2-proxy response headers into standard `Authorization: Bearer` headers. This centralized auth model means components no longer need individual auth sidecars - they receive pre-authenticated requests with identity headers. The Gateway supports three auth modes (IntegratedOAuth, OIDC, None) and three certificate types (OpenshiftDefaultIngress, SelfSigned, Provided), making it adaptable to both on-premise OpenShift and managed cloud environments.

One notable security consideration is the DestinationRule `odh-tls-rule` which sets `insecureSkipVerify: true` on both ports 443 and 8443. This disables certificate verification within the service mesh, relying on network isolation rather than cryptographic verification for internal traffic. While acceptable for internal mesh traffic with self-signed service-serving certificates, it represents a deviation from zero-trust networking principles and should be documented in security reviews.

The Kueue controller is unique among component controllers: it operates exclusively in `Unmanaged` mode (attempting `Managed` state returns an error), meaning it doesn't deploy Kueue itself but manages the integration layer (default ClusterQueue, LocalQueues with dynamically-detected cluster resources, ResourceFlavors, namespace labels, and admin RBAC from Auth CR groups). This design reflects Kueue being installed separately via an external operator.

The monitoring architecture is highly flexible, with three distinct configurations based on platform type. ManagedRhoai forces monitoring enabled and manages prometheus-configs.yaml file updates for SRE integration. SelfManagedRhoai and ODH support optional monitoring with MonitoringStack, ThanosQuerier, Tempo, and OpenTelemetryCollector CRs. Component-specific PrometheusRules are dynamically added/removed based on component enablement status, creating a self-healing monitoring configuration.
