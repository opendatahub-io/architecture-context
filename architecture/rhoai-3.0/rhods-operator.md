## Purpose

**Short**: The RHODS Operator (Red Hat OpenShift AI Operator) is the central lifecycle manager for the Red Hat OpenShift AI platform, responsible for deploying, configuring, and managing all AI/ML platform components on OpenShift.

**Detailed**: The RHODS Operator is a Go-based Kubernetes operator built on controller-runtime that manages the complete lifecycle of the Red Hat OpenShift AI (RHOAI) 3.x platform. It operates through two primary custom resources — `DataScienceCluster` (DSC) and `DSCInitialization` (DSCI) — which serve as the user-facing APIs for configuring the platform.

The operator orchestrates deployment of 11+ AI/ML components including Dashboard, Workbenches (Jupyter notebooks), KServe (model serving), Data Science Pipelines, Ray, Kueue, TrustyAI, Model Registry, Training Operator, Feast Operator, and LlamaStack Operator. Each component is represented as a sub-CRD in the `components.platform.opendatahub.io` API group, and the operator manages their deployment via kustomize manifests bundled in `/opt/manifests/`.

In RHOAI 3.x, the operator deploys critical platform ingress infrastructure using the Kubernetes Gateway API. It creates a `data-science-gateway` Gateway with an OpenShift Gateway controller, an EnvoyFilter for external authentication via a `kube-auth-proxy` (OAuth2 proxy) deployment, and DestinationRule for TLS management. This replaces the legacy ServiceMesh/Istio-based ingress from RHOAI 2.x. The operator also manages platform-level services for authentication (RBAC groups/bindings), monitoring (MonitoringStack, OpenTelemetry, Tempo), and certificate distribution (trusted CA bundles).

**Gateway API Infrastructure (RHOAI 3.x — dynamically created by Gateway Controller)**:

The Gateway Controller (`internal/controller/services/gateway/`) deploys the following resources at runtime:

1. **GatewayClass** (`data-science-gateway-class`): References `openshift.io/gateway-controller/v1` controller
2. **Gateway** (`data-science-gateway`): Deployed to `openshift-ingress` namespace with HTTPS listener on port 443, TLS termination, and certificate from Secret; allows routes from `openshift-ingress` and the application namespace
3. **kube-auth-proxy Deployment**: OAuth2 proxy in `openshift-ingress` namespace with ports 4180/TCP (HTTP), 8443/TCP (HTTPS), 9000/TCP (metrics); supports both OpenShift OAuth and external OIDC
4. **kube-auth-proxy Service**: Exposes OAuth2 proxy on 8443/TCP (HTTPS) and 9000/TCP (metrics) with service-serving-cert TLS annotation
5. **kube-auth-proxy-creds Secret**: Contains OAuth2 client ID, client secret, and cookie secret
6. **OAuthClient** (`odh`): OpenShift OAuth client registration (IntegratedOAuth mode only) with redirect URI to `https://{domain}/oauth2/callback`
7. **HTTPRoute** (`oauth-callback-route`): Routes `/oauth2` path to kube-auth-proxy service via Gateway API
8. **EnvoyFilter** (`authn-filter`): Configures Envoy ext_authz filter to call `kube-auth-proxy:8443/oauth2/auth` for authentication; includes Lua filter to forward access tokens as Bearer authorization headers
9. **DestinationRule** (`odh-tls-rule`): Configures TLS SIMPLE mode for ports 8443 and 443 in `openshift-ingress` namespace

**Authentication Flow**: Incoming requests → Gateway (Envoy) → EnvoyFilter ext_authz → kube-auth-proxy (OAuth2 check) → if authenticated, request forwarded to backend with Bearer token; if not, redirect to OAuth login.

Component controllers (Dashboard, Workbenches, etc.) create their own HTTPRoute CRs referencing `data-science-gateway` as the parent Gateway.
