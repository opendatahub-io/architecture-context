## Purpose

**Short**: A central repository for ML model developers to store, manage, and serve model metadata including registered models, model versions, artifacts, inference services, and serving environments.

**Detailed**: Model Registry provides a REST API for managing the lifecycle of machine learning models within the RHOAI platform. It stores metadata about registered models, their versions, associated artifacts (model binaries, datasets, documentation), inference services, and serving environments. The core service exposes an OpenAPI v3-defined REST API and persists data to either MySQL or PostgreSQL databases using an embedded metadata (EmbedMD) datastore layer with GORM and golang-migrate for schema management.

The repository also includes an InferenceService controller that watches KServe `InferenceService` custom resources and synchronizes their state back into the Model Registry, creating a bidirectional link between deployed models and their registry entries. A CSI storage initializer enables KServe to resolve `model-registry://` URIs by looking up model artifacts in the registry and delegating the actual download to the appropriate KServe storage provider (S3, PVC, etc.). An asynchronous upload job (Python) handles copying models between storage backends (S3, OCI) and registering them in the Model Registry.

## Architectural Analysis

The Model Registry follows a layered architecture with clear separation between the REST API surface, the service/business logic layer, and the data persistence layer. The REST API is generated from an OpenAPI 3.0 specification (`api/openapi/model-registry.yaml`) using openapi-generator-cli, which produces both the server handler stubs (`internal/server/openapi/`) and the Go client SDK (`pkg/openapi/`). The chi router framework provides HTTP routing with CORS and logging middleware. A notable pattern is the "dynamic router" in `internal/proxy/dynamic_router.go`, which uses RWMutex-protected handler swapping to enable concurrent server startup — the HTTP server starts immediately while the database connection is established asynchronously, and handlers are swapped in once the datastore is ready.

The data layer uses an "EmbedMD" (Embedded Metadata) pattern that abstracts away the specific database backend. Both MySQL and PostgreSQL are supported through GORM with database-specific connectors (`internal/datastore/embedmd/mysql/` and `postgres/`). Schema migrations are embedded in the Go binary via `//go:embed` directives, and the golang-migrate library handles migration execution. The connection layer implements exponential backoff retry logic (up to 25 retries with incremental sleep) and mutex-protected singleton connections. TLS configuration for database connections is comprehensive, supporting custom certificates, cipher suite selection, and server certificate verification via the `internal/tls/config.go` module.

The InferenceService controller is architecturally interesting as a "sync agent" rather than a traditional operator that creates resources. It watches KServe InferenceService CRs and mirrors their state into Model Registry entries via the REST API, maintaining bidirectional references through Kubernetes labels (`inferenceServiceIDLabel`, `registeredModelIDLabel`, `modelVersionIDLabel`). The controller discovers Model Registry services via label selectors (`component: model-registry`) and constructs in-cluster DNS URLs. It uses a finalizer to clean up registry entries when InferenceServices are deleted, setting them to UNDEPLOYED state. One security concern is that the controller's HTTP client has `InsecureSkipVerify: true` hardcoded (configurable only via the `SKIP_TLS_VERIFY` env var at startup), meaning TLS certificate verification is globally disabled when that flag is set, not per-connection.

The CSI storage initializer provides a clean integration point with KServe's model serving infrastructure. It registers the `model-registry://` URI scheme via a ClusterStorageContainer CR and resolves model URIs through a chain of API calls: find registered model by name → find model version (by name or latest) → get model version artifacts → extract storage URI from the first model artifact → delegate to the appropriate KServe storage provider. This design means the storage initializer itself never downloads model files — it only resolves metadata and hands off to established KServe providers (S3, PVC, etc.).

The async-upload job represents a newer addition (referenced in issue #1108) that supports three "intents": `update_artifact` (update an existing artifact's URI after upload), `create_model` (register a new model with version and artifact), and `create_version` (add a version to an existing model). It uses configargparse with `MODEL_SYNC_` prefix for environment variable auto-mapping and loads credentials from mounted Kubernetes Secrets. The job supports S3, OCI, and generic URI sources, with S3 and OCI destinations, using skopeo for OCI image operations and the model-registry Python SDK for metadata management.
