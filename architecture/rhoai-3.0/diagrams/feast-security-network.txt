Secrets:
========
- {name}-tls (kubernetes.io/tls)
  - Purpose: TLS certificates for feast services
  - Provisioned By: User or cert-manager
  - Auto-Rotate: No (user) / Yes (cert-manager, 90d default)
  - Contains: tls.crt, tls.key

- {name}-client-tls (kubernetes.io/tls)
  - Purpose: Client TLS certificates for mTLS
  - Provisioned By: User or cert-manager
  - Auto-Rotate: No (user) / Yes (cert-manager)
  - Contains: tls.crt, tls.key

- oidc-secret (Opaque)
  - Purpose: OIDC authentication credentials
  - Provisioned By: User
  - Auto-Rotate: No
  - Contains: client_id, client_secret, etc.

- {storage}-credentials (Opaque)
  - Purpose: Storage backend credentials (S3/GCS/DB)
  - Provisioned By: User
  - Auto-Rotate: No
  - Contains: AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, DB passwords, etc.

- git-token (Opaque)
  - Purpose: Git repository access token for feature repo sync
  - Provisioned By: User
  - Auto-Rotate: No
  - Contains: token

Authentication Methods:
=======================
1. No Authentication (no_auth)
   - Default for local/development deployments
   - No token validation

2. Kubernetes RBAC (kubernetes)
   - Feature servers perform SubjectAccessReview checks
   - Validates caller has feast.dev permissions
   - Enforcement: Feature Server code

3. OIDC (oidc)
   - Bearer token validation against OIDC provider
   - JWT token validation
   - Configurable OIDC provider endpoint
   - Enforcement: Feature Server code

TLS Configuration:
==================
- All feast services support dual HTTP/HTTPS ports
- HTTP ports (6566, 8815, 6572, 8888): No encryption
- HTTPS ports (6567, 8816, 6573, 8443): TLS 1.2+
- Certificates: User-provided or cert-manager issued
- Custom CA bundles: Supported via odh-trusted-ca-bundle ConfigMap injection

Network Isolation:
==================
- Services use ClusterIP (internal only by default)
- External access via OpenShift Routes (optional)
- No NetworkPolicies in base deployment (user-configurable)
- Optional Service Mesh integration for mTLS and traffic management
