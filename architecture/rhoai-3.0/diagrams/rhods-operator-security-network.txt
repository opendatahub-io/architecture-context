AUTHENTICATION & AUTHORIZATION FLOW
═══════════════════════════════════════════════════════════════════════════════════════════════════

IntegratedOAuth Mode (OpenShift default):
  1. External user → Gateway (443/TCP HTTPS)
  2. EnvoyFilter ext_authz → kube-auth-proxy (8443/TCP HTTPS)
  3. kube-auth-proxy checks session cookie
  4. If no session: Redirect to OpenShift OAuth server
  5. User logs in → OAuth server issues code
  6. Callback to oauth-callback-route → kube-auth-proxy
  7. kube-auth-proxy exchanges code for access token
  8. kube-auth-proxy sets httpOnly, secure session cookie
  9. Future requests: kube-auth-proxy validates session cookie
  10. EnvoyFilter allows request → HTTPRoute → kube-rbac-proxy sidecar
  11. kube-rbac-proxy performs SubjectAccessReview (Kubernetes RBAC)
  12. If authorized → proxy to application container (localhost HTTP)

OIDC Mode (ROSA / External IdP):
  1-2. Same as IntegratedOAuth
  3. kube-auth-proxy checks session cookie
  4. If no session: Redirect to external OIDC provider (configured in GatewayConfig)
  5. User logs in → OIDC provider issues ID token + access token
  6. Callback to oauth-callback-route → kube-auth-proxy
  7. kube-auth-proxy validates OIDC tokens
  8. kube-auth-proxy sets httpOnly, secure session cookie
  9-12. Same as IntegratedOAuth

Per-Workload RBAC Enforcement:
  - Component controllers (e.g., odh-notebook-controller) inject kube-rbac-proxy sidecars
  - Sidecars listen on 8443/TCP (HTTPS)
  - On each request:
    a. kube-rbac-proxy extracts Kubernetes user identity from OAuth/OIDC headers
    b. Performs SubjectAccessReview API call to Kubernetes
    c. If authorized: proxies to application container (localhost:8080 HTTP)
    d. If denied: returns 403 Forbidden

SECURITY POSTURE
═══════════════════════════════════════════════════════════════════════════════════════════════════

Encryption:
  - All external traffic: TLS 1.3 (Gateway, Router)
  - Operator API calls: TLS 1.2+ (Kubernetes API)
  - Webhook server: TLS 1.2+ (9443/TCP)
  - kube-auth-proxy: TLS 1.3 (8443/TCP)
  - kube-rbac-proxy sidecars: TLS 1.3 (8443/TCP)
  - Monitoring: TLS 1.3 with optional mTLS
  - Internal component-to-component: Optional mTLS via Service Mesh

Authentication:
  - External users: OAuth 2.0 (OpenShift) or OIDC (external IdP)
  - Operator: Kubernetes ServiceAccount tokens (RBAC)
  - Webhooks: TLS client certificate validation
  - Component workloads: kube-rbac-proxy sidecars (Kubernetes RBAC)
  - Metrics scraping: Bearer tokens or no auth (metrics only)

Network Isolation:
  - Operator: redhat-ods-operator namespace (restricted SCC)
  - Gateway infrastructure: openshift-ingress namespace (DMZ)
  - Component workloads: redhat-ods-applications namespace (user workloads)
  - Monitoring: redhat-ods-monitoring namespace (observability)
  - NetworkPolicies: Optional, can be enforced for fine-grained isolation

Security Context:
  - Operator runs as non-root (UID 1001)
  - Restricted SCC (no privileged, no host access)
  - Read-only root filesystem where possible
  - Capabilities dropped (minimum required)

Certificate Management:
  - Webhook cert: OpenShift cert injection (service.beta.openshift.io/serving-cert-secret-name)
  - Gateway cert: References OpenShift default ingress cert (auto-rotates with cluster cert)
  - Component certs: cert-manager integration optional (custom certs)
  - Session cookies: Encrypted with cookieSecret, httpOnly, secure, time-limited

Trust Boundaries:
  1. External (untrusted) → Ingress (DMZ)
  2. Ingress (DMZ) → Application Layer (workload namespaces)
  3. Application Layer → Operator Control Plane (cluster-admin level)
  4. All layers → Kubernetes API Server (RBAC enforced)
