Network Flow Summary
═════════════════════════════════════════════════════════════════════════

Ingress Flows:
  1. External User → Kubernetes API (6443/TCP HTTPS TLS1.2+)
     - Create Notebook CR via kubectl/jupyter-web-app
     - Auth: User Bearer Token

  2. Prometheus → Metrics Endpoint (8443/TCP HTTPS TLS1.2+)
     - Scrape metrics via kube-rbac-proxy
     - Auth: TokenReview (RBAC: metrics-reader role)

  3. Kubernetes → Health Probes (8081/TCP HTTP)
     - Liveness/Readiness checks
     - Auth: None (unauthenticated)

  4. External User → Istio Gateway → VirtualService → Notebook Pod
     - Access Jupyter notebook UI
     - Auth: Configurable via Istio RequestAuthentication

Egress Flows:
  1. Notebook Controller → Kubernetes API (6443/TCP HTTPS TLS1.2+)
     - CRUD operations on resources
     - Auth: ServiceAccount Token

  2. CullingReconciler → Notebook Pods (8888/TCP HTTP)
     - Poll /api/kernels for idle detection
     - Auth: Bearer Token or Istio AuthorizationPolicy

Security Considerations
═════════════════════════════════════════════════════════════════════════

1. HTTP-Only Kernel Polling:
   - CullingReconciler polls /api/kernels over HTTP (port 8888)
   - No TLS encryption for this traffic
   - Requires Istio AuthorizationPolicy to secure access
   - Consider using mTLS via service mesh for production

2. Metrics Endpoint:
   - Default: HTTP on port 8080 (no authentication)
   - Production: HTTPS on port 8443 via kube-rbac-proxy (recommended)
   - Use kube-rbac-proxy for TokenReview-based authentication

3. Notebook Service Exposure:
   - Notebook Services (port 80→8888) are ClusterIP (internal only)
   - External access only via Istio VirtualService (when enabled)
   - Configure Istio RequestAuthentication for external access control

4. RBAC Permissions:
   - Controller has cluster-wide * (all) permissions on:
     - services (core)
     - statefulsets (apps)
     - notebooks (kubeflow.org)
     - virtualservices (networking.istio.io)
   - Principle of least privilege: Consider scoping to specific namespaces

5. Leader Election:
   - Default: Disabled (ENABLE_LEADER_ELECTION=false)
   - HA deployments should enable leader election to prevent split-brain
   - Leader election ConfigMap in opendatahub namespace

6. FIPS Compliance:
   - Container image built with Konflux (strictfipsruntime)
   - TLS 1.2+ enforced for Kubernetes API communication
   - FIPS-compliant cryptography for all TLS connections
