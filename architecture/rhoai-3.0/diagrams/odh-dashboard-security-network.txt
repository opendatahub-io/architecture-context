AUTHENTICATION & AUTHORIZATION FLOW
═══════════════════════════════════════════════════════════════════════════

1. User Access (OAuth2 Flow):
   User → Route (443/TCP) → kube-rbac-proxy (8443/TCP)
   ↓
   kube-rbac-proxy → OpenShift OAuth (redirect if no token)
   ↓
   OpenShift OAuth → User (login page)
   ↓
   User → OpenShift OAuth (credentials)
   ↓
   OpenShift OAuth → kube-rbac-proxy (access token)
   ↓
   kube-rbac-proxy validates token, injects headers:
   - X-Auth-Request-User: {username}
   - X-Auth-Request-Groups: {group1,group2,...}
   - X-Forwarded-Access-Token: {bearer-token}
   ↓
   kube-rbac-proxy → odh-dashboard backend (8080/TCP HTTP)
   ↓
   Backend trusts headers from sidecar (pod-local only)
   ↓
   Backend → Kubernetes API (with ServiceAccount token)

2. Backend Authorization:
   - Backend uses ServiceAccount token for Kubernetes API calls
   - User identity from X-Auth-Request-User used for audit/context
   - RBAC enforced at Kubernetes API level (odh-dashboard SA permissions)
   - Dashboard does NOT perform user-level RBAC itself (cluster-wide SA)

3. Health Checks (Unauthenticated):
   - /api/health (8080/TCP): Backend health (internal, no auth)
   - /healthz (8444/TCP): kube-rbac-proxy health (internal, no auth)
