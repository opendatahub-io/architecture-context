## Purpose

**Short**: REST API orchestration server that coordinates AI text generation with content safety guardrails by managing detector, chunker, and LLM backend services.

**Detailed**: The FMS Guardrails Orchestrator (`fms-guardrails-orchestr8`) is a Rust-based middleware service that sits between AI application clients and language model backends, enforcing content safety through configurable detector pipelines. It receives text generation requests (both unary and streaming), optionally runs input text through content safety detectors before forwarding to LLM backends, and then runs output through detectors before returning results to clients. The orchestrator manages multiple detector services, chunker services (for text segmentation), and generation backends (TGIS via gRPC or Caikit NLP via gRPC) through a unified REST API.

The service exposes two HTTP servers: a main guardrails API server (default port 8033/TCP) and a separate health check server (default port 8034/TCP). It supports TLS and mTLS for both inbound connections and outbound client connections to downstream services. The orchestrator implements OpenTelemetry-based observability with configurable traces and metrics export via OTLP (gRPC or HTTP). It also provides an OpenAI-compatible API layer (`/v1/chat/completions`, `/v1/completions`) for chat completion detection workflows, proxying requests to vLLM or compatible OpenAI API servers while injecting guardrail detections into the response.

The architecture follows a handler-per-endpoint pattern where each API endpoint maps to a distinct orchestration task type. Tasks coordinate the fan-out of detection requests across multiple detectors concurrently, aggregate results, and apply threshold-based filtering before constructing the response. Streaming endpoints use SSE (Server-Sent Events) or ND-JSON for real-time response delivery.

## Architectural Analysis

The FMS Guardrails Orchestrator is a well-structured Rust service that acts as a content safety middleware layer for AI text generation workflows. Its architecture follows several notable patterns:

**Handler-per-endpoint task pattern**: Each API endpoint maps to a dedicated task struct (e.g., `ClassificationWithGenTask`, `ChatCompletionsDetectionTask`, `StreamingContentDetectionTask`) that encapsulates the orchestration logic for that specific workflow. Tasks implement the `Handle` trait, and the `Orchestrator` struct dispatches them. This pattern cleanly separates routing concerns from orchestration logic, making it straightforward to add new endpoint types without modifying existing ones.

**Dual-server architecture**: The service runs two independent HTTP servers — a main guardrails API server on port 8033 and a health check server on port 8034. Only the guardrails server supports TLS/mTLS; the health server always runs plain HTTP. This design allows health probes from Kubernetes to function independently of TLS configuration while keeping the API surface secured. The health server also provides an `/info` endpoint that proactively checks the health of all configured downstream services.

**Multi-protocol client abstraction**: The orchestrator communicates with different backend types using different protocols — gRPC (via tonic) for TGIS and Caikit NLP generation servers and chunker services, and HTTP (via hyper/reqwest) for detector services and OpenAI-compatible servers. The `ClientMap` provides a type-erased container for all client types, and the `Client` trait provides a unified health check interface. The generation client abstraction (`GenerationClient`) supports both TGIS and NLP backends through an enum-based dispatch pattern with configurable retry logic and exponential backoff.

**TLS architecture concerns**: The orchestrator uses `rustls` with the `ring` crypto backend rather than OpenSSL for all TLS operations. While this provides good performance and memory safety, it creates a FIPS compliance gap: `ring` is not FIPS-certified, meaning all TLS connections (both inbound server TLS and outbound client TLS to detectors, generators, and chunkers) use non-FIPS-validated cryptography. The `NoVerifier` implementation in `src/utils/tls.rs` allows completely disabling certificate verification for outbound connections when `insecure: true` is set in the TLS config — this is useful for development but represents a security risk if used in production. Additionally, the supported verification schemes include legacy SHA1-based algorithms (`RSA_PKCS1_SHA1`, `ECDSA_SHA1_Legacy`), which are considered weak.

**Configuration-driven service mesh**: The orchestrator's entire service topology is defined through a single YAML configuration file. Detectors, chunkers, generation servers, and TLS configurations are all declared at startup and cannot be changed at runtime. Each detector is typed (`text_contents`, `text_generation`, `text_chat`, `text_context_doc`), and the orchestrator validates that each detector references a valid chunker and has a valid hostname. This static configuration model is simple but means the orchestrator must be restarted to pick up topology changes, unlike controller-based approaches that watch for CRD changes.
