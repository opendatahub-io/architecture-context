## Purpose

**Short**: MaaS API is a Go REST service that provides OpenAI-compatible model listing, tier-based subscription management, and ephemeral token issuance for KServe LLM inference services, fronted by Kuadrant-managed Gateway API policies for authentication, authorization, and tiered rate limiting.

**Detailed**: The Models as a Service (MaaS) API acts as the control plane for a tiered LLM inference platform built on top of KServe and OpenShift AI. It exposes three core capabilities: (1) an OpenAI-compatible `/v1/models` endpoint that discovers and lists `LLMInferenceService` and `InferenceService` CRs across all namespaces via the Kubernetes dynamic client, (2) a tier lookup service that maps user groups to subscription tiers (free/premium/enterprise) from a ConfigMap-based configuration, and (3) a token management service that issues time-limited Kubernetes ServiceAccount tokens scoped to tier-specific namespaces, enabling fine-grained access control to inference endpoints.

The MaaS API does not directly serve inference traffic. Instead, it integrates with the Kuadrant policy engine at the Gateway API level. A `gateway-auth-policy` AuthPolicy on the `maas-default-gateway` Gateway calls back to the MaaS API's `/v1/tiers/lookup` endpoint to enrich identity metadata with subscription tier information. This tier metadata then drives per-user rate limiting via `RateLimitPolicy` and `TokenRateLimitPolicy` CRs, enforced by Kuadrant's Authorino and Limitador components. The architecture separates the management plane (MaaS API) from the data plane (Gateway → KServe inference endpoints), with Kuadrant providing the policy enforcement layer between them.

The repository also contains prototype frontend (React) and backend (Node.js/Python) applications under `apps/`, as well as a QoS prioritizer module, but these are development-time utilities not shipped as production artifacts — only the `maas-api` Go service has a Konflux Dockerfile and Tekton pipeline for production builds.

## Architectural Analysis

The MaaS API follows a clear separation between the management plane and the data plane. The API itself handles only control-plane operations — model discovery, tier resolution, and token issuance — while all data-plane concerns (authentication, authorization, rate limiting, and traffic routing to inference endpoints) are delegated to the Kuadrant policy engine operating at the Gateway API level. This is a deliberate architectural choice that keeps the MaaS API stateless and focused, relying on Kubernetes-native primitives (ServiceAccounts, Namespaces, ConfigMaps) as its only persistence layer.

The tier-based access control model is implemented through an interesting callback pattern: the Kuadrant `gateway-auth-policy` AuthPolicy enriches each request's identity metadata by making an HTTP POST to the MaaS API's `/v1/tiers/lookup` endpoint during the authentication flow. The response (tier name) is then cached by Authorino for 300 seconds per user and used by downstream `RateLimitPolicy` and `TokenRateLimitPolicy` predicates. This means the MaaS API is both an API server (serving external clients) and an internal callback target (serving the policy engine). The tier lookup is currently unprotected by authentication since it's intended for cluster-internal use only — the `gateway-auth-policy.yaml` includes a TODO noting that a NetworkPolicy should be added to restrict access.

The token management design creates a namespace-per-tier topology: when a user requests a token, the MaaS API creates a namespace named `{instance}-tier-{tierName}` (e.g., `maas-default-gateway-tier-premium`) and a ServiceAccount named after a SHA1-hashed sanitization of the username. Tokens are issued via the Kubernetes TokenRequest API with configurable TTL (minimum 10 minutes, default 4 hours). Token revocation works by deleting and recreating the ServiceAccount, which invalidates all previously issued tokens. The `gateway-auth-policy` then validates these SA tokens via `kubernetesTokenReview` with audience `maas-default-gateway-sa`, and performs `kubernetesSubjectAccessReview` to authorize access to specific `LLMInferenceService` resources by namespace and name extracted from the request path.

A notable FIPS compliance gap exists: the Konflux Dockerfile builds with `CGO_ENABLED=0`, producing a statically linked binary that bypasses OpenSSL entirely. This will fail the `check-payload` FIPS gate if FIPS compliance is required for this component. The binary uses `crypto/sha1` for username-to-SA-name hashing, but this is a non-cryptographic use (collision avoidance). The application does not configure TLS directly — it listens on plaintext HTTP and relies on the Gateway for TLS termination, which is an appropriate pattern for cluster-internal services behind a TLS-terminating gateway.

The Tekton pipelines (`odh-maas-api-v3-0-push.yaml`) show that Konflux builds are hermetic (`hermetic: true`) with Go module prefetching, and the image is built for four architectures (x86_64, arm64, ppc64le, s390x). The output image is pushed to `quay.io/rhoai/odh-maas-api-rhel9`, following the RHOAI naming convention. The pipeline includes build-nudge support for automated dependency updates.
