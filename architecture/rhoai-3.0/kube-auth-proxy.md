## Purpose

**Short**: FIPS-compliant authentication and authorization reverse proxy for RHOAI components, supporting both OIDC and OpenShift OAuth providers with Kubernetes RBAC enforcement.

**Detailed**: kube-auth-proxy is a focused authentication proxy derived from oauth2-proxy, designed specifically for OpenShift Data Hub (ODH) and Red Hat OpenShift AI (RHOAI) environments. It provides secure authentication through external OIDC providers and OpenShift's internal OAuth service, serving as a drop-in replacement for existing oauth-proxy sidecars.

The repository produces a single container image containing three binaries: `kube-auth-proxy` (OAuth2/OIDC authentication proxy), `kube-rbac-proxy` (Kubernetes SubjectAccessReview authorization proxy), and `entrypoint` (a mode-selecting wrapper that routes to either binary based on configuration). This dual-proxy architecture allows RHOAI components to use either authentication-only mode (OIDC/OAuth) or authorization mode (Kubernetes RBAC via SAR) from the same container image.

The project was created in response to the RHOAI BYOIDC (Bring Your Own Identity Provider) initiative, consolidating authentication capabilities into a single FIPS-compliant image that can replace both oauth-proxy (for authentication) and kube-rbac-proxy (for authorization) sidecars across the platform. All non-OIDC providers (Google, GitHub, Azure, etc.) have been removed to reduce attack surface and focus on enterprise authentication requirements.

## Architectural Analysis

kube-auth-proxy represents a strategic consolidation of two previously separate authentication and authorization sidecars (oauth-proxy and kube-rbac-proxy) into a single container image with a mode-selecting entrypoint. This design is driven by the RHOAI BYOIDC initiative, which requires supporting arbitrary OIDC providers beyond OpenShift's built-in OAuth service. The entrypoint binary (`cmd/entrypoint/main.go`) uses `syscall.Exec` to replace itself with the selected proxy binary, avoiding process overhead — the container always runs exactly one proxy binary, never both simultaneously.

The kube-auth-proxy component (lines 1-1527 of `oauthproxy.go`) is a substantially modified fork of oauth2-proxy with all non-OIDC providers removed (no Google, GitHub, Azure, GitLab, etc.). Only two providers remain: a generic OIDC provider (`providers/oidc.go`) and an OpenShift-specific OAuth provider (`providers/openshift.go`). The OpenShift provider includes auto-discovery of OAuth endpoints via the Kubernetes API (`/.well-known/oauth-authorization-server`) and native service account token support, making it work seamlessly in OpenShift environments without manual endpoint configuration. The OIDC provider supports PKCE (S256 and plain), OIDC RP-Initiated Logout via discovery of the `end_session_endpoint`, and JWT Bearer token passthrough for CLI/API access.

The kube-rbac-proxy component (`kube-rbac-proxy/` subdirectory) is an embedded copy of brancz/kube-rbac-proxy v0.19.1 with its own `go.mod` and vendor directory. It provides Kubernetes-native RBAC enforcement via SubjectAccessReview, with a three-layer authorization chain: (1) a hardcoded authorizer that allows the OpenShift monitoring prometheus-k8s service account to scrape `/metrics` without SAR checks, (2) a static authorizer driven by YAML configuration for custom rules, and (3) the standard SAR authorizer that delegates to the Kubernetes API. This layered approach ensures OpenShift monitoring always works while allowing flexible authorization policies.

A notable architectural decision is the session management system inherited from oauth2-proxy. The proxy supports two session storage backends: encrypted cookies (default, using AES-CFB encryption with HMAC signing) and Redis (with standalone, sentinel, and cluster modes). The Redis backend uses per-session AES-GCM encryption keys stored as ticket references in cookies, providing stronger encryption at the cost of external state. The distributed locking mechanism (`pkg/sessions/redis/lock.go`) prevents concurrent session refresh storms in multi-replica deployments. The middleware chain architecture (`pkg/middleware/`) follows a classic responsibility chain pattern: scope → health check → logging → metrics → session loading → header injection → upstream proxy, with clean separation enabling independent testing and composition.

The TLS configuration across both proxy modes defaults to TLS 1.2 minimum with configurable cipher suites. The kube-rbac-proxy component can auto-generate self-signed certificates if none are provided, with certificate hot-reload support via `pkg/tls/reloader.go`. Both components produce Prometheus metrics — kube-auth-proxy exposes `oauth2_proxy_requests_total`, `oauth2_proxy_requests_in_flight`, and `oauth2_proxy_response_duration_seconds` histograms, while kube-rbac-proxy exposes standard Kubernetes apiserver authentication and authorization metrics.
