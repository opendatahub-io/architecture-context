## Purpose

**Short**: A FIPS-compliant authentication reverse proxy for OIDC and OpenShift OAuth, serving as a drop-in replacement for oauth-proxy sidecars in RHOAI environments.

**Detailed**: kube-auth-proxy is a focused authentication proxy derived from oauth2-proxy, streamlined specifically for OpenShift Data Hub (ODH) and Red Hat OpenShift AI (RHOAI) environments. It provides secure authentication through two provider types: external OIDC providers (any standards-compliant OpenID Connect provider) and OpenShift's internal OAuth service. The project intentionally removes all other authentication providers (Google, GitHub, Azure, etc.) to minimize attack surface and maintain focus.

The container image ships two binaries in a single image: `kube-auth-proxy` (the OAuth/OIDC authentication proxy) and `kube-rbac-proxy` (a Kubernetes RBAC authorization proxy based on brancz/kube-rbac-proxy v0.19.1). An `entrypoint` binary selects between the two at runtime based on the `PROXY_MODE` environment variable or a positional command-line argument (`auth` or `rbac`), defaulting to `rbac` mode. This dual-binary design allows the same container image to serve as either an authentication sidecar or an authorization sidecar depending on the deployment context.

FIPS compliance is a primary design requirement. The Konflux build (Dockerfile.konflux) uses `CGO_ENABLED=1`, `GOEXPERIMENT=strictfipsruntime`, and `-tags strictfipsruntime` to produce a FIPS-compliant binary on a UBI9 base image. The proxy handles authentication (authn) only in `auth` mode; it does not perform SubjectAccessReview (SAR) authorization. In `rbac` mode, the kube-rbac-proxy binary provides Kubernetes RBAC-based authorization with SAR checks.

## Security

### RBAC - Cluster Roles

No RBAC manifests are shipped with this repository. RBAC is configured by the deploying component (e.g., the RHOAI operator).

In **rbac mode** (kube-rbac-proxy), the proxy performs SubjectAccessReview checks against the Kubernetes API server using the caller's bearer token. The proxy's own ServiceAccount needs permission to create SubjectAccessReview resources.
