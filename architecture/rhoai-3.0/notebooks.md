## Purpose

**Short**: Builds and publishes multi-architecture container images for interactive data science workbenches (JupyterLab, RStudio, Code Server) and non-interactive Kubeflow pipeline runtime environments used in Red Hat OpenShift AI.

**Detailed**: The notebooks repository is a container image factory that produces the complete set of user-facing workbench and pipeline runtime images for RHOAI. It does not contain any operator or controller code — instead, it defines Dockerfiles for 16+ Konflux-built image variants spanning CPU, NVIDIA CUDA, and AMD ROCm accelerator stacks, with multi-architecture support for x86_64, aarch64, ppc64le, and s390x.

The repository follows a layered image composition pattern: base images (UBI9 or CentOS Stream 9 with Python 3.12) are extended by "minimal" notebook layers (JupyterLab core), then specialized by data science libraries, ML frameworks (PyTorch, TensorFlow), and explainability toolkits (TrustyAI). Pipeline runtime images follow a parallel but lighter-weight track — they share the same base and dependency stacks but omit interactive notebook infrastructure. The resulting images are consumed by the ODH Notebook Controller, which launches them as single-user workbench pods when users create Notebook CRs through the RHOAI Dashboard.

Deployment manifests in `manifests/base/` define OpenShift ImageStream resources that reference the built images by SHA digest. These ImageStreams are consumed by the platform operator (`rhods-operator`) via `get_all_manifests.sh` and make the workbench images available for selection in the RHOAI Dashboard UI. Kustomize `replacements` inject image references from `params.env` files and commit hashes from `commit.env` files into the ImageStream tags, enabling version management across N through N-6 image generations.

## Architectural Analysis

The notebooks repository follows a **layered image composition** pattern that is architecturally noteworthy. The Dockerfile inheritance chain — `base-images/cpu/ubi9 → jupyter/minimal → jupyter/datascience → jupyter/{pytorch,tensorflow,trustyai}` — mirrors a dependency tree where each layer adds specialized libraries without duplicating base infrastructure. The Konflux Dockerfiles (`Dockerfile.konflux.*`) reproduce this layering using multi-stage builds rather than `FROM` chaining, which allows Konflux to track build inputs precisely. Each Dockerfile shares common fragments injected by `scripts/dockerfile_fragments.py`, ensuring consistent installation of micropipenv, uv, OpenShift CLI, and security updates across all variants.

**Multi-architecture support is a first-class concern.** The DataScience and TrustyAI Dockerfiles contain extensive architecture-conditional logic: ppc64le builds use gcc-toolset-13 and compile OpenBLAS v0.3.30 and ONNX v1.19.0 from source; s390x builds install Rust for compiling native Python extensions and build pyarrow with S3 support disabled. The ROCm PyTorch runtime includes a `de-vendor-torch.sh` script that strips bundled ROCm libraries to reduce image size and use the system-installed ROCm stack. This architecture-aware build logic is necessary because many ML framework wheels are not available as pre-built binaries for non-x86 platforms.

**The runtime images are deliberately minimal.** They share dependency stacks with their workbench counterparts but omit JupyterLab, Elyra, PDF export utilities, and interactive tooling. This keeps pipeline step containers small and fast to pull. Runtime images have no ENTRYPOINT — they are designed to have commands specified by the Kubeflow pipeline orchestrator at launch time.

**The Kustomize manifests implement an N/N-1/.../N-6 image versioning scheme** via OpenShift ImageStream tags. Each ImageStream carries up to 7 tagged versions (e.g., `2025.2`, `2025.1`, `2024.2`, etc.), allowing users to select current or older workbench images. Image references in `params.env` and `params-latest.env` use SHA256 digests rather than mutable tags, ensuring reproducibility. Build provenance is tracked via `commit.env` files that map each image to its source git commit. The `opendatahub.io/notebook-image-recommended: 'true'` annotation on the N tag marks it as the default selection in the Dashboard UI.

**Build hermeticity is partial.** Python dependencies are locked via `pylock.toml` files (PEP 665 format) providing hash-verified installs, and Go build tools have `go.sum` files. However, the Tekton pipeline runs with `hermetic: "false"`, RPM-level lock files (`rpms.lock.yaml`) are absent, and Hermeto/cachi2 prefetch is not integrated. This means builds can access the network at runtime, which is typical for this branch — downstream release branches may add stricter hermetic build controls.
