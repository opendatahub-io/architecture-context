## Purpose

**Short**: A unified Python SDK for fine-tuning large language models using multiple algorithms (SFT, OSFT, LoRA, GRPO) with interchangeable backends.

**Detailed**: training-hub provides a single-package Python API that abstracts over several model fine-tuning strategies and their underlying training engines. It implements an Algorithm-Backend-Registry design pattern where each fine-tuning algorithm (SFT, OSFT, LoRA SFT, LoRA GRPO, full GRPO) is paired with one or more interchangeable backend implementations (instructlab-training, mini-trainer, Unsloth, ART, verl). Users call top-level convenience functions like `sft()`, `osft()`, `lora_sft()`, `lora_grpo()`, or `grpo()` with consistent keyword arguments, and the library handles backend selection, parameter translation, distributed training orchestration via PyTorch torchrun, and experiment tracking integration.

The library also provides GPU memory estimation (per-algorithm VRAM calculators for SFT, OSFT, LoRA, and QLoRA), training time estimation (via sample-based extrapolation), and loss visualization utilities. It is distributed as a PyPI package (`training-hub`) with optional extras for CUDA kernels, LoRA dependencies, and GRPO-specific packages. training-hub is not a Kubernetes service or operator; it is a client-side SDK intended to run on GPU-equipped machines or within notebook environments as part of the RHOAI model training workflow.

training-hub supports 14+ model architectures including Llama, Qwen2/3/3.5, Granite, Phi4, Gemma3, Mistral, Nemotron, and vision-language models (VLMs). It integrates with experiment tracking platforms (Weights & Biases, MLflow, TensorBoard) and includes a comprehensive model validation harness for testing convergence across architectures and algorithms.

## Architectural Analysis

### Algorithm-Backend-Registry Pattern

training-hub's core architecture uses a Strategy pattern implemented through three abstractions: `Algorithm` (defines required/optional parameters and dispatches to a backend), `Backend` (executes the actual training), and `AlgorithmRegistry` (class-level registry mapping `(algorithm_name, backend_name)` tuples to implementation classes). This allows adding new algorithms or backends independently. Each backend registration happens at module import time via decorators or explicit `AlgorithmRegistry.register_backend()` calls. The `create_algorithm()` factory function provides the entry point for programmatic algorithm construction.

### Parameter Translation Layer

Each backend implements significant parameter translation logic. For example, the SFT backend maps `max_tokens_per_gpu` to instructlab-training's `max_batch_len`, and the OSFT backend maps generic parameters to mini-trainer-specific arguments while also performing comprehensive type validation. This translation layer is a key design decision: users work with a consistent API while backends receive their native parameter formats. The LoRA backend handles the additional complexity of VLM detection (checking `ForConditionalGeneration` vs `ForCausalLM` in model architecture class names) to select appropriate model loading strategies.

### Distributed Training Orchestration

The `get_torchrun_params()` utility in `utils.py` implements a hierarchical precedence system for distributed training parameters: function arguments > environment variables > defaults. It handles the mutual exclusivity of `master_addr` and `rdzv_endpoint` with source-aware conflict resolution and warnings. The verl backend uses a different distributed strategy, constructing torchrun CLI commands directly and managing FSDP/vLLM worker allocation.

### GRPO Architecture (ART vs verl)

The GRPO implementation has two distinct backends with fundamentally different architectures:
- **ART backend**: Runs as a subprocess on a single GPU, handles rollout generation and reward computation in-process, and produces a seed LoRA checkpoint to work around vLLM V1 eager LoRA validation.
- **verl backend**: Generates a standalone Python reward function file, converts data to Parquet format, builds a complex CLI command with FSDP and vLLM rollout configuration, streams output while parsing step metrics live into `training_metrics.jsonl`, and performs post-training checkpoint normalization for vLLM adapter loading compatibility.

### Memory Estimation Design

The profiling module provides per-algorithm GPU VRAM calculators (`BasicEstimator` for SFT, `OSFTEstimator`, `LoRAEstimator`, `QLoRAEstimator`). These estimate memory requirements by computing: model parameters size, gradient memory, optimizer state memory (AdamW with 3x overhead), intermediate activations, and KV-cache contributions. Results include lower/expected/upper bounds with configurable overhead multipliers. Model metadata is cached to a local CSV file to avoid repeated HuggingFace Hub queries.

### Design Risks and Observations

1. **Timing estimator fragility**: The `TimingEstimatorExperimental` class parses log files using string splitting and ANSI escape code matching (`\x1b`), which is brittle and may break if backend log formats change. The docstrings for `_sft_log_parse` and `_osft_log_parse` are swapped (each describes the other's method).
2. **Temp file management**: The timing estimator writes `temp_data.jsonl` to the current working directory without namespacing, which could conflict in concurrent usage scenarios.
3. **litellm pinning**: The `litellm==1.63.11` pin in the grpo extra is explicitly noted as a supply chain security measure, suggesting awareness of dependency risk in the LLM tooling ecosystem.
4. **No container images**: Unlike most RHOAI components, training-hub is distributed solely via PyPI. This means it bypasses Konflux build pipelines, check-payload FIPS validation, and container-level supply chain controls. FIPS compliance depends entirely on the runtime environment.
5. **Backend isolation**: The ART GRPO backend runs training in a subprocess to ensure GPU memory isolation, an important design choice for RL workloads that alternate between training and inference.
