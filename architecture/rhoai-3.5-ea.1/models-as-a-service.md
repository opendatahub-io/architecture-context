## Purpose

**Short**: Models as a Service (MaaS) provides a comprehensive platform for managing LLM model access, authentication, rate limiting, and subscription-based billing through a Kubernetes-native control plane and REST API, integrating with KServe for model serving and Kuadrant/Authorino for policy enforcement.

**Detailed**: MaaS consists of three primary components: a Kubernetes controller (`maas-controller`) that manages custom resources (Tenant, MaaSModelRef, MaaSAuthPolicy, MaaSSubscription, ExternalModel) and reconciles them into Kuadrant AuthPolicies, TokenRateLimitPolicies, and Istio networking resources; a Go REST API service (`maas-api`) that provides OpenAI-compatible model listing, subscription selection, and API key lifecycle management backed by PostgreSQL; and a payload-processing component (BBR ext_proc server) that performs request transformations at the gateway level including model extraction, provider resolution, API translation, and credential injection.

The platform implements a multi-tenant, subscription-based access model where users and groups are granted model access through MaaSAuthPolicy resources, rate-limited through MaaSSubscription resources with per-token quotas, and can authenticate via OpenShift tokens, external OIDC providers, or programmatic API keys in OpenAI-compatible `sk-oai-*` format. The controller aggregates multiple policies per model into single Kuadrant AuthPolicy and TokenRateLimitPolicy resources attached to Gateway API HTTPRoutes, while a default-deny security posture at the gateway level ensures unconfigured models are inaccessible. The system supports both KServe LLMInferenceService backends and external model providers (e.g., OpenAI, Anthropic) through the ExternalModel CRD, with transparent proxy behavior enabled by the BBR payload-processing pipeline.

## Architecture Components

| Component | Type | Purpose |

## Architectural Analysis

The MaaS platform implements a sophisticated multi-layer architecture that separates policy definition from policy enforcement. The controller layer translates high-level intent (MaaSAuthPolicy, MaaSSubscription CRDs) into Kuadrant-native resources (AuthPolicy, TokenRateLimitPolicy) through an aggregation pattern where multiple MaaS policies targeting the same model are collapsed into a single Kuadrant policy per HTTPRoute. This prevents Kuadrant's constraint that only one policy can target a given HTTPRoute, while allowing fine-grained per-user and per-group access control at the MaaS layer.

The authentication architecture is notably multi-modal: users can authenticate via OpenShift tokens (validated through KubernetesTokenReview), external OIDC providers (configured per-Tenant), or programmatic API keys in the OpenAI-compatible `sk-oai-*` format. API keys use a hash-only storage model (SHA-256 with per-key salt derived from the key_id) with constant-time comparison, and are bound to a specific subscription at creation time to prevent lateral movement. The system implements a fail-closed design throughout: missing headers, unreachable probes, and unbound subscriptions all result in access denial. A particularly interesting security feature is the credential stripping: Authorino validates the incoming credential but the AuthPolicy response filter replaces or removes the Authorization header before the request reaches the model backend, preventing credential exfiltration to potentially untrusted model providers.

The Tenant reconciler follows a pipeline pattern for platform deployment, using kustomize build with post-render transformations to inject gateway references, OIDC configuration, and telemetry settings into the maas-api deployment manifests. This design means the controller image bundles the deployment manifests for maas-api (copied at Dockerfile build time), enabling the controller to deploy and manage maas-api's full lifecycle including its Service, CronJob, RBAC, and NetworkPolicy without requiring external manifest sources. The post-render phase patches the gateway reference, OIDC audience, and cache TTL values into the rendered manifests before applying via server-side apply (SSA).

The payload-processing component acts as a transparent middleware layer at the gateway, using the Envoy ext_proc protocol to perform request transformations before traffic reaches model backends. The plugin pipeline (model extraction -> provider resolution -> API translation -> API key injection) enables support for external model providers by extracting the model name from the request body, resolving the target provider and credentials from ExternalModel CRs, translating between API formats (e.g., OpenAI to Anthropic), and injecting provider-specific credentials from Kubernetes Secrets. This design decouples the authentication layer (Authorino validates the user's credential) from the authorization layer (credential replacement for upstream providers), which is critical for the security model that prevents user credential leakage.

The observability integration spans three layers: Prometheus metrics from the controller via PodMonitor, Kuadrant TelemetryPolicy for per-request metrics with rich dimensions (user, model, subscription, organization, cost center), and optional Perses dashboards for visualization. The TelemetryPolicy is particularly notable as it extracts dimensions from the Authorino auth identity, enabling billing and usage tracking per subscription without any application-level instrumentation in the model backends.
