## Purpose

**Short**: Models as a Service (MaaS) provides a comprehensive platform for managing AI/ML model endpoints with policy-based access control, rate limiting, subscription management, and API key lifecycle.

**Detailed**: MaaS consists of two primary components — a Kubernetes operator (`maas-controller`) and a RESTful API service (`maas-api`) — that together enable platform administrators to expose, govern, and meter access to AI/ML models served via KServe or external providers (OpenAI, Anthropic, etc.). The controller manages six CRDs in the `maas.opendatahub.io` API group: `Config`, `Tenant`, `MaaSModelRef`, `MaaSAuthPolicy`, `MaaSSubscription`, and `ExternalModel`. These CRDs compose a multi-tier governance model where model endpoints are registered, access policies are defined per group/user, and rate-limited subscriptions control consumption. The controller translates these high-level intents into Kuadrant `AuthPolicy` and `TokenRateLimitPolicy` resources, Istio `ServiceEntry`/`DestinationRule`/`EnvoyFilter` resources, and Gateway API `HTTPRoute` resources.

The `maas-api` service provides the data plane for API key management (hash-based, show-once pattern backed by PostgreSQL), model discovery with access validation, and subscription selection. It integrates with Kuadrant Authorino for authentication evaluation — Authorino calls back to maas-api internal endpoints for API key validation and subscription resolution. A third deployment, `payload-processing`, runs as an Envoy ext_proc filter in the gateway namespace to handle request/response transformation (model name extraction, provider credential injection, API translation).

The platform deploys its own Gateway API `Gateway` resource (`maas-default-gateway`) in `openshift-ingress` with HTTP/HTTPS listeners, and all model traffic flows through this gateway with per-model HTTPRoutes, AuthPolicies, and TokenRateLimitPolicies providing fine-grained governance.

## Architectural Analysis

The MaaS platform implements a layered governance model that separates model endpoint registration (MaaSModelRef), access control (MaaSAuthPolicy), and metering (MaaSSubscription) into distinct CRDs. This separation enables platform administrators to compose policies independently — a model can exist without governance (phase: Pending), and governance policies can reference models that aren't yet ready. The controller tracks this via a "governance pairing" concept where a model reaches Ready phase only when at least one active MaaSAuthPolicy AND MaaSSubscription pair exists. This deliberate coupling prevents models from being accidentally exposed without access controls, implementing a "secure by default" posture.

The controller's Tenant reconciler is architecturally notable for its kustomize-based deployment pipeline. Rather than imperatively creating resources, it renders kustomize overlays, applies post-render transformations (gateway references, OIDC configuration, telemetry settings), and uses server-side apply with the Config CR as the controller owner. This means the entire maas-api stack (Deployment, Service, HTTPRoute, AuthPolicy, NetworkPolicies, CronJob) is treated as a single declarative unit, with drift detection via SHA-256 hash of rendered manifests stored as an annotation. The Config CR serves as a GC anchor — when MaaS is disabled, deleting Config cascades to all platform resources via owner references.

The security architecture is particularly well-designed with defense-in-depth. External-facing endpoints (model inference, API management) are protected by Kuadrant AuthPolicy at the Gateway level. Internal endpoints (/internal/v1/*) used for Authorino callbacks are protected by NetworkPolicy restricting access to Authorino and CronJob pods only. The API key system uses a hash-based show-once pattern (SHA-256 with null-byte delimiter, constant-time comparison) where plaintext keys are never stored. The gateway itself has a deny-by-default posture via two gateway-level policies: `gateway-default-auth` which rejects requests to unconfigured model paths, and `gateway-default-deny` which rate-limits unconfigured paths to 0 tokens/min. The conflict detection system proactively warns when external (non-MaaS) AuthPolicies target the same HTTPRoutes, preventing policy conflicts.

The payload-processing component is architecturally interesting as it runs as an Envoy ext_proc filter registered via EnvoyFilter CR. It intercepts both request and response bodies in full-duplex streaming mode, running four plugins sequentially: extracting the model name from the request body into a header, resolving the model to its provider endpoint, translating between API formats (e.g., OpenAI-to-provider format), and injecting provider-specific API keys from Kubernetes Secrets. This design keeps credential management (ExternalModel credentialRef) in the controller plane while the data plane handles injection, preventing credentials from appearing in Gateway configuration.

The ExternalModel reconciler creates a full Istio routing stack per external provider: an ExternalName Service for DNS resolution, a ServiceEntry to register the external host in the mesh, a DestinationRule for TLS origination, and an HTTPRoute for path-based routing through the Gateway. This enables the platform to treat external LLM providers (OpenAI, Anthropic, etc.) identically to internal KServe models from a governance perspective — both get HTTPRoutes, AuthPolicies, and TokenRateLimitPolicies.
