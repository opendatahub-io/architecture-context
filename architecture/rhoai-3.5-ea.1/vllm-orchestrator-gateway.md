## Purpose

**Short**: An HTTP gateway that provides OpenAI-compatible chat completion endpoints with configurable detector-based content filtering, routing requests to the FMS Guardrails Orchestrator.

**Detailed**: The vLLM Orchestrator Gateway is a Rust-based HTTP service built on the Axum web framework that sits between API consumers and the FMS Guardrails Orchestrator. It enforces controlled access to the orchestrator's endpoints by providing a mechanism for configuring fixed detector pipelines, each exposed as a unique `/v1/chat/completions` endpoint per configured route.

Each route in the gateway's YAML configuration specifies which content detectors (e.g., PII regex detectors) should be applied to input and/or output, along with an optional fallback message returned when a detection is triggered. The gateway transforms standard OpenAI-compatible chat completion requests by injecting the appropriate detector specifications before forwarding them to the orchestrator's `/api/v2/chat/completions-detection` endpoint. This allows drop-in replacement of an unguardrailed chat completions model with a guardrailed one, without requiring client-side changes beyond updating the endpoint URL.

The gateway supports both streaming (SSE) and non-streaming responses, mTLS communication with the orchestrator, and forwards authorization and `x-forwarded-*` headers from clients. It is part of the TrustyAI stack within Red Hat OpenShift AI, providing AI safety guardrails for vLLM-served models.

## Architectural Analysis

The vLLM Orchestrator Gateway is a deliberately thin routing layer — its Rust codebase consists of only three source files (`main.rs`, `config.rs`, `api.rs`) totaling roughly 700 lines. This simplicity is a design choice: the gateway's sole responsibility is to map named routes to detector configurations and forward requests to the FMS Guardrails Orchestrator. It performs no inference, no detection logic, and no persistent state management. The YAML-driven configuration model allows new guardrail policies to be deployed by adding route entries without code changes.

The mTLS implementation in `build_orchestrator_client()` (lines 351-406 of `main.rs`) reveals an interesting pattern: the gateway constructs a PKCS#12 identity from PEM-encoded cert/key files using the `openssl` crate, then passes it to `reqwest` via the `native-tls` backend. This two-step conversion (PEM → OpenSSL X509/PKey → PKCS#12 DER → native-tls Identity) is necessary because `reqwest`'s `Identity::from_pkcs12_der` API expects DER-encoded PKCS#12, not raw PEM. The `danger_accept_invalid_hostnames(true)` call when the orchestrator host is `localhost` is a development convenience that should be audited before production use, though it only triggers in local development configurations.

The gateway's streaming implementation deserves attention: it parses SSE chunks from the orchestrator, deserializes each chunk as a `StreamingResponse`, checks for detections, and re-serializes with potential fallback message injection. This per-chunk inspection means detection-triggered fallback works for streaming responses, not just batch responses. However, the SSE parsing in `orchestrator_streaming_request()` uses a simple line-based `data: ` prefix parser (lines 526-543) — multi-line SSE data fields or fields other than `data:` are not handled. This is acceptable given the orchestrator's known response format but could be fragile if the orchestrator changes its SSE encoding.

The gateway exposes its HTTP listener on port 8090 by default (configurable via `HTTP_PORT` env var) without TLS termination — it listens in plaintext. TLS termination for client-facing traffic is expected to be handled by the platform's ingress layer (Gateway API / OpenShift Route / kube-rbac-proxy sidecar). The mTLS configuration only applies to the egress connection to the orchestrator, not to the gateway's own listener. This is a standard sidecar-friendly pattern in RHOAI where the platform injects auth proxies in front of application containers.

The build infrastructure uses Konflux for multi-architecture builds (x86_64, arm64, ppc64le, s390x) but is explicitly non-hermetic (`hermetic: false` in both push and PR Tekton pipelines). The Rust toolchain is pinned to 1.86.0 via `rust-toolchain.toml`, and the Dockerfile.konflux uses UBI 9 minimal as the base image with `compat-openssl11` installed alongside the binary. The absence of `rpms.lock.yaml` and Hermeto prefetch integration suggests this component has not yet undergone full supply chain hardening for RHOAI GA release.
