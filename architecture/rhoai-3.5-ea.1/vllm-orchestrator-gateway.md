## Purpose

**Short**: An OpenAI-compatible HTTP gateway that routes chat completion requests through configurable detector-based content filtering pipelines via the FMS Guardrails Orchestrator.

**Detailed**: The vllm-orchestrator-gateway is a Rust-based HTTP service that sits between client applications and the FMS Guardrails Orchestrator, providing a simplified OpenAI-compatible API for guardrailed LLM inference. It enables organizations to define named routes (e.g., `/pii/v1/chat/completions`) that automatically inject specific detector configurations (such as PII detection via regex patterns) into requests before forwarding them to the orchestrator's `/api/v2/chat/completions-detection` endpoint.

The gateway reads a YAML configuration file that defines the orchestrator connection, detector definitions (with input/output direction and parameters), and routes that map detector pipelines to named URL paths. Each route can specify a fallback message that replaces the LLM response when a detection is triggered, enabling drop-in replacement of an unguardrailed chat completions endpoint with a guardrailed one. The gateway supports both standard (non-streaming) and Server-Sent Events (SSE) streaming responses, forwarding authorization headers and `x-forwarded-*` headers to the orchestrator.

The service supports optional mTLS for secure communication with the orchestrator, loading TLS certificates and CA bundles from well-known Kubernetes secret mount paths (`/etc/tls/private/` and `/etc/tls/ca/`). It is part of the TrustyAI ecosystem within RHOAI, providing content safety guardrails for vLLM-served models.

## Architectural Analysis

The vllm-orchestrator-gateway follows a clean reverse-proxy / API gateway pattern implemented in Rust with the axum framework. Its primary architectural role is **policy injection**: it takes standard OpenAI-compatible chat completion requests and enriches them with detector configurations before forwarding to the FMS Guardrails Orchestrator. This design allows organizations to expose named endpoints (e.g., `/pii/v1/chat/completions`) that are drop-in replacements for direct LLM endpoints, transparently adding content safety guardrails.

The configuration model (`config.rs`) is intentionally simple — a single YAML file defines the orchestrator connection, detector definitions, and route-to-detector mappings. The `DetectorConfig` struct supports both input and output detection directions, custom detector parameters, and an optional `server` field that allows multiple detector configurations to share the same backend detector service. Route validation at startup (`validate_registered_detectors`) ensures all detectors referenced by routes exist in the config and prevents duplicate input/output detectors with the same server within a route, catching configuration errors early.

A notable design decision is the mTLS support implementation (`build_orchestrator_client` in `main.rs:351-406`). The gateway uses OpenSSL to construct a PKCS#12 identity from PEM-formatted certificates, then passes it to the reqwest HTTP client via `native-tls`. This approach bridges the Kubernetes convention of mounting TLS secrets as PEM files with reqwest's PKCS#12 identity API. The `danger_accept_invalid_hostnames(true)` flag for localhost connections is a development convenience that should not impact production deployments where the orchestrator's DNS name matches its certificate.

The streaming implementation (`handle_streaming_generation` in `main.rs:259-349`) deserializes each SSE chunk from the orchestrator, checks for detections, and applies the fallback message to the first chunk if detections are found. This chunk-level inspection means detection results must arrive within the SSE stream — the gateway cannot buffer the entire response before deciding. This is consistent with the orchestrator's streaming detection API contract.

From a security perspective, the gateway performs no authentication or authorization itself — it passes through `Authorization` and `x-forwarded-*` headers to the orchestrator. Authentication enforcement is expected at either the network level (via kube-rbac-proxy sidecar, HTTPRoute, or service mesh) or at the orchestrator. This is a deliberate architectural choice that keeps the gateway stateless and simple, but means it must be deployed behind an authenticating proxy in production.
