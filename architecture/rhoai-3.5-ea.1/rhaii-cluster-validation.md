## Purpose

**Short**: Preflight validation probe for GPU clusters that verifies GPU availability, RDMA connectivity, network bandwidth, required CRDs, and operator health before AI workload deployment.

**Detailed**: RHAII Cluster Validation (`rhaii-validator`) is a Go-based CLI tool and kubectl plugin (`kubectl rhaii-validate`) that performs comprehensive preflight validation of Kubernetes clusters intended for AI/ML workloads. It targets GPU clusters across multiple cloud platforms (OpenShift/OCP, AKS, EKS, CoreWeave) and validates hardware readiness at multiple tiers.

The tool operates in a two-tier architecture: Tier 1 performs API-level checks (CRD existence, operator health) directly from the CLI process, while Tier 2 deploys privileged Job pods to each GPU node for hardware-level validation (GPU driver version, ECC status, RDMA device discovery, GPU-NIC-NUMA-PCIe topology mapping). Multi-node network tests (TCP bandwidth via iperf3, TCP latency via built-in echo server, RDMA bandwidth via ib_write_bw, RDMA connectivity via ibv_rc_pingpong pingmesh) use a server/client Job pattern with ring, star, or pairwise topologies.

The validator produces a structured JSON report stored in a Kubernetes ConfigMap, enabling downstream automation to gate workload deployment on cluster readiness. It ships as two container images: the validator agent (Go binary with GPU/RDMA check logic) and a tools image (CUDA-enabled perftest and iperf3 binaries for bandwidth testing).

## Architectural Analysis

The RHAII Cluster Validation tool follows a two-phase architecture that cleanly separates API-level checks from hardware-level probes. Tier 1 checks (CRD existence, operator health) run in-process from the CLI, requiring only Kubernetes API access. Tier 2 checks deploy privileged Job pods directly to GPU nodes, mounting the host filesystem to access sysfs, GPU devices, and RDMA hardware. This design allows the same binary to work as both a user-facing kubectl plugin (orchestrating from a developer laptop) and an in-cluster agent (running inside Job pods via the hidden `run` subcommand).

The multi-node testing framework in `pkg/jobrunner` implements three topology patterns: ring (every node tested as both sender and receiver), star (one server, N clients), and pairwise (N-choose-2 with round-robin tournament scheduling for parallel execution of disjoint pairs). The pairwise mode, used primarily for RDMA pingmesh connectivity testing, includes retry logic with up to 3 attempts per pair and concurrent execution of non-conflicting pairs within each round. This is architecturally notable because it minimizes total test time on large clusters while ensuring deterministic ordering.

The GPU-NIC topology discovery in `pkg/checks/rdma/topology.go` implements three pairing strategies based on hardware topology: PCIe distance pairing (for hierarchical topologies with multi-segment PCIe paths), NUMA affinity pairing (for flat/virtual PCI buses), and NUMA load-balance pairing (when GPUs outnumber NICs). The PCIe distance calculation walks the sysfs device path to compute hop counts between GPU and NIC devices, enabling optimal GPU-NIC pairing for GPUDirect RDMA bandwidth tests. An explicit penalty system (100 for cross-NUMA, 50 for unknown paths) ensures correct ordering when topology information is incomplete.

Platform detection is auto-detected from node labels and provider IDs, with a priority order that correctly handles OpenShift-on-cloud scenarios (OCP checked before AWS/Azure to avoid misclassification). Each platform has embedded YAML defaults for resource requests, RDMA configuration, operator namespaces, and performance thresholds. Users can override any default via a ConfigMap that persists across validation runs.

The image resolution system uses the OLM `relatedImages` pattern: default images are embedded in `manifests/image-references/image-references.yaml` and can be overridden via environment variables (`RELATED_IMAGE_RHAII_CLUSTER_VALIDATOR`, `RELATED_IMAGE_RHAII_VALIDATOR_TOOLS`). This enables the RHOAI operator to inject SHA-pinned images at deployment time, matching the pattern used across other RHOAI components.

The RDMA device detection in `pkg/checks/rdma/devices.go` includes a sophisticated three-layer check to distinguish genuine RDMA-capable NICs from phantom verbs devices (e.g., Azure Accelerated Networking SR-IOV VFs that expose mlx5_ib entries but have no actual RDMA transport). It checks link layer type, then GID netdev associations, then falls back to a GID scan — ensuring correct behavior across cloud providers with different virtualization approaches.
