## Purpose

**Short**: Preflight validation tool that verifies GPU cluster readiness for AI/ML workloads by checking GPU hardware, RDMA connectivity, network bandwidth, required CRDs, and operator health across xKS clusters.

**Detailed**: RHAII Cluster Validation (rhaii-cluster-validation) is a Go-based CLI tool that functions as a kubectl plugin (`kubectl rhaii-validate`) to perform comprehensive preflight checks on Kubernetes clusters before deploying AI/ML inference workloads. It is designed for RHOAI (Red Hat OpenShift AI) and also supports AKS, EKS, and CoreWeave platforms.

The tool operates in a two-tier architecture: Tier 1 checks run directly via the Kubernetes API (CRD existence, operator health), while Tier 2 checks deploy privileged Job pods to GPU nodes for hardware-level validation. The controller orchestrates the full lifecycle — creating RBAC, deploying per-node check Jobs, collecting results from pod logs, running multi-node bandwidth tests, and performing RDMA connectivity mesh testing. Results are stored in a ConfigMap for persistence and output in table or JSON format.

The validation suite covers: GPU driver version and ECC memory errors (NVIDIA and AMD), RDMA device discovery and NIC status, GPU-NIC-NUMA-PCIe topology mapping with three pairing strategies (NUMA affinity, PCIe distance, NUMA load balance), TCP bandwidth via iperf3, TCP latency via a custom Go echo server, RDMA bandwidth via ib_write_bw with GPUDirect CUDA support, and RDMA connectivity mesh (pingmesh) using ibv_rc_pingpong. Platform-specific configurations (thresholds, RDMA resources, operator namespaces) are embedded per platform and customizable via ConfigMap.

## Architectural Analysis

The RHAII Cluster Validation tool follows a hub-and-spoke architecture where the CLI process (hub) orchestrates short-lived Kubernetes Jobs (spokes) across GPU nodes. This design avoids requiring a DaemonSet or persistent deployment — the validator creates resources on demand, collects results, and cleans up. The two-tier check model (Tier 1 API-level, Tier 2 node-level) allows lightweight dependency checks to run without deploying any pods, while hardware-level checks require privileged Job pods with host filesystem access.

The multi-node bandwidth testing subsystem supports three topologies: ring (default, every node tested as both server and client), star (one server, N clients), and pairwise (N-choose-2 with round-robin tournament scheduling for parallelism). The pairwise topology is used for RDMA pingmesh connectivity testing, where disjoint pairs within each round run concurrently. Each pair is retried up to 3 times, and results are classified as rail (same GPU-NIC index) or cross-rail for separate pass/fail reporting. This classification enables operators to distinguish between rail-aligned RDMA fabric issues and switch-level cross-rail problems.

The GPU-NIC topology discovery is a standout feature, implementing three pairing strategies: NUMA affinity (flat topology fallback), PCIe tree distance (1:1 matching by shortest PCIe hop count), and NUMA load balance (distributes GPUs across fewer NICs). The topology is stored in a ConfigMap-based report that persists across runs, allowing the more expensive `rdma-node` discovery to be run once and reused by subsequent `rdma-ping` and `rdma-bandwidth` runs. This incremental execution model lets operators diagnose issues step by step rather than running the full suite every time.

Platform detection is automatic based on node labels and provider IDs (OCP first to avoid misclassification on OpenShift-on-Azure/AWS, then CoreWeave by label, then AKS/EKS by provider ID). Each platform has embedded configuration defaults, particularly around RDMA resource names (which vary by device plugin vendor — `rdma/shared_ib` on AKS, custom on others), operator namespace overrides (cert-manager-operator vs cert-manager on OCP), and performance thresholds. The ConfigMap-based override system uses YAML merge semantics, so users only specify fields they want to change.

A notable design decision is the separation of validator and tools images. The validator image contains the Go binary with platform checks and the custom TCP latency server, while the tools image is a heavier image built from a CUDA base with compiled perftest binaries (ib_write_bw, ib_read_bw) and iperf3. This separation keeps the validator image small and avoids CUDA dependencies for non-RDMA checks. The two images are resolved via an embedded `image-references.yaml` that follows the RHOAI relatedImages pattern, allowing Konflux to substitute pinned image references at release time.
