## Purpose

**Short**: FIPS-compliant authentication and authorization reverse proxy for RHOAI, combining OIDC/OpenShift OAuth authentication (kube-auth-proxy) and Kubernetes RBAC authorization (kube-rbac-proxy) into a single container image.

**Detailed**: kube-auth-proxy is a focused authentication proxy derived from oauth2-proxy, designed specifically for OpenShift Data Hub (ODH) and Red Hat OpenShift AI (RHOAI) environments. It provides secure authentication through external OIDC providers, OpenShift's internal OAuth service, and Kubernetes service account tokens (via TokenReview API). The proxy acts as a sidecar container that intercepts requests to upstream applications, validates credentials, and forwards authenticated requests with identity headers.

The repository produces a single container image containing three binaries: the kube-auth-proxy (authentication proxy), kube-rbac-proxy (Kubernetes RBAC authorization proxy), and an entrypoint binary that selects between the two at runtime via the `PROXY_MODE` environment variable or CLI argument. This dual-mode design replaces the need for separate oauth-proxy and kube-rbac-proxy container images. The kube-auth-proxy handles authentication only (authn), while kube-rbac-proxy handles authorization via SubjectAccessReview (authz). The entrypoint defaults to RBAC mode, reflecting its primary use as a kube-rbac-proxy replacement in RHOAI 3.x.

The project was forked from oauth2-proxy and streamlined to support only OIDC and OpenShift OAuth providers, removing all other providers (Google, GitHub, Azure, etc.) to reduce attack surface and improve maintainability.

## Architectural Analysis

**Dual-Mode Architecture**: The most distinctive architectural decision in this component is the single container image serving two fundamentally different proxy personalities. The entrypoint binary (87 lines) acts as a runtime dispatcher, selecting between kube-auth-proxy (authentication) and kube-rbac-proxy (authorization) based on environment configuration. This collapses what was historically two separate container images (oauth-proxy and kube-rbac-proxy) into one, simplifying the RHOAI operator's container image management. The default mode is RBAC, reflecting that most RHOAI 3.x component pods use kube-rbac-proxy as a sidecar for SAR-based authorization.

**Authentication Chain Design**: The kube-auth-proxy implements a multi-tier authentication chain in `oauthproxy.go:407-449` that evaluates session loaders in strict priority order: (1) Kubernetes service account tokens via TokenReview API, (2) OpenShift OAuth tokens (sha256~ format), (3) JWT bearer tokens from OIDC issuers, (4) Basic auth via htpasswd, (5) Stored sessions from cookies. This ordering is intentional: machine-to-machine authentication (K8s SA tokens) is evaluated before human authentication flows, and stateless token validation precedes stateful session lookup. The K8s TokenReview integration (`pkg/authentication/k8s/tokenreview.go`) is notable as an addition not present in upstream oauth2-proxy, enabling pod-to-pod authentication without OAuth credentials.

**Session Management Trade-offs**: The component supports two session storage backends with meaningfully different trade-offs. Cookie-based sessions (default) are stateless and zero-dependency but limited to ~4KB per cookie, requiring multi-cookie splitting for large sessions (`pkg/sessions/cookie/session_store.go:174-204`). Redis sessions enable server-side state with distributed locking to prevent thundering herd during refresh (`pkg/sessions/redis/lock.go`), supporting standalone, Sentinel, and Cluster topologies. The persistence manager adds an indirection layer where cookies store only an opaque ticket ID, with the full session in Redis. A "minimal mode" strips OAuth tokens from cookie sessions when they're not needed downstream, reducing cookie size and improving security.

**kube-rbac-proxy Authorization Architecture**: The kube-rbac-proxy sub-component implements a three-tier union authorizer evaluated in order: (1) a hardcoded authorizer that unconditionally allows `system:serviceaccount:openshift-monitoring:prometheus-k8s` to GET `/metrics`, (2) a static authorizer that pattern-matches against user/verb/namespace/resource tuples from a config file, and (3) the SubjectAccessReview authorizer that delegates to the Kubernetes API server. The hardcoded Prometheus allowlist (`pkg/hardcodedauthorizer/metrics.go:26-59`) is an OpenShift-specific customization ensuring monitoring always works regardless of RBAC policy configuration errors. Token sanitization in logs (`cmd/kube-rbac-proxy/app/sanitazion.go`) prevents credential exposure in debug output.

**FIPS Compliance Strategy**: The build pipeline enforces FIPS at the build level via `GOEXPERIMENT=strictfipsruntime` and `CGO_ENABLED=1`, ensuring the Go binary uses OpenSSL for cryptographic operations rather than pure-Go crypto. The session encryption uses AES-CFB and AES-GCM (both FIPS-approved symmetric algorithms) with HMAC-SHA256 signatures. The TLS configuration defaults to TLS 1.2 minimum with a TLS 1.3 maximum, and cipher suites are configurable. The `InsecureSkipVerify` option exists for development but requires explicit opt-in. No non-FIPS cryptographic algorithms (MD5, RC4, DES) were detected in the codebase. The kube-rbac-proxy sub-component is also built with CGO_ENABLED=1 for FIPS compliance.
