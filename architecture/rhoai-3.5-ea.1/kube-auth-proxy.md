## Purpose

**Short**: A FIPS-compliant authentication and authorization reverse proxy that provides OIDC, OpenShift OAuth, and Kubernetes RBAC-based access control for RHOAI platform components.

**Detailed**: kube-auth-proxy is a unified container image that ships two distinct proxy binaries — **kube-auth-proxy** (an oauth2-proxy fork for OIDC/OpenShift OAuth authentication) and **kube-rbac-proxy** (a Kubernetes SubjectAccessReview-based authorization proxy) — selected at runtime by an entrypoint dispatcher. The kube-auth-proxy binary intercepts HTTP requests, authenticates users via external OIDC providers or OpenShift's built-in OAuth service, manages sessions (via cookies or Redis), and proxies authenticated traffic to upstream applications with identity-bearing headers. It is designed as a drop-in replacement for the legacy oauth-proxy sidecar used in RHOAI 2.x.

The kube-rbac-proxy binary provides Kubernetes-native RBAC authorization by delegating authentication to Kubernetes TokenReview and authorization to SubjectAccessReview APIs. It is typically deployed as a sidecar container on port 8443/TCP to protect metrics endpoints and component APIs. Together, these two binaries provide the full authentication and authorization sidecar infrastructure for RHOAI 3.x platform components.

## Architectural Analysis

**Unified Container Image with Dual Proxy Architecture**: kube-auth-proxy takes the unusual approach of shipping two distinct proxy implementations in a single container image, selected at runtime by a lightweight Go entrypoint dispatcher. This design consolidates the authentication (OIDC/OAuth) and authorization (K8s RBAC) sidecars that were previously separate images into one. The entrypoint defaults to `rbac` mode (kube-rbac-proxy), maintaining backward compatibility with existing RHOAI sidecar deployments. This consolidation reduces image registry footprint, simplifies operator deployment logic (one image reference instead of two), and ensures both proxies share identical FIPS-compliant build flags and base image versions.

**OAuth2-proxy Fork with Provider Consolidation**: The kube-auth-proxy binary is a focused fork of the upstream oauth2-proxy project that has intentionally removed all providers except OIDC and OpenShift OAuth. This provider consolidation (from ~15 providers to 2) significantly reduces attack surface and maintenance burden. The OpenShift OAuth provider was ported from the legacy openshift/oauth-proxy project and adds features not in upstream oauth2-proxy: auto-discovery from `/.well-known/oauth-authorization-server`, ServiceAccount-based client credentials, OpenShift user/group enrichment, and OAuthAccessToken deletion on logout. The K8s TokenReview integration (`pkg/authentication/k8s/tokenreview.go`) is a wholly new addition enabling service account tokens as a first-class authentication method alongside browser-based OAuth flows.

**Session Management Flexibility with Security Trade-offs**: The session subsystem supports two backends — cookie-based (default, encrypted with AES-CFB + HMAC signed, compressed with LZ4) and Redis-based (server-side storage with only a session ID cookie). Cookie-based sessions hit browser size limits (~4KB) and use a multi-cookie splitting strategy for large sessions. Redis-based sessions support standalone, Sentinel, and Cluster topologies with distributed locking for concurrent refresh. The session loading chain evaluates five session sources in priority order (K8s token → OAuth token → JWT → Basic Auth → Stored Session), meaning a single request may trigger multiple validation paths. The 30-second session expiry for K8s service account tokens ensures near-real-time token revocation checking at the cost of increased K8s API traffic.

**FIPS-First Build Strategy**: The Konflux Dockerfile builds all three binaries with `CGO_ENABLED=1 GOEXPERIMENT=strictfipsruntime -tags strictfipsruntime`, which forces Go to use OpenSSL via the FIPS-certified BoringCrypto module rather than Go's native crypto implementation. Both binaries configure TLS 1.2 as the minimum version by default, with TLS 1.3 available. The kube-rbac-proxy sub-component includes configurable cipher suite selection. No non-FIPS cryptographic algorithms (MD5, RC4, DES) were found in the codebase. The `InsecureSkipVerify` option exists but is configurable (not default), which is an acceptable operational trade-off documented in the providers.

**kube-rbac-proxy's SubjectAccessReview Caching**: The authorization layer caches allow decisions for 5 minutes and deny decisions for 30 seconds. This asymmetric caching provides a balance between reducing K8s API load (most requests are repeated allowed patterns from Prometheus scraping) and maintaining security responsiveness (denials are re-evaluated quickly). The hardcoded metrics authorizer bypasses SAR entirely for the `system:serviceaccount:openshift-monitoring:prometheus-k8s` service account accessing `/metrics`, which is a pragmatic optimization for the most common scraping pattern in OpenShift environments.
