## Purpose

**Short**: A Kubernetes controller that performs intelligent autoscaling for LLM inference model servers based on saturation metrics, KV cache utilization, and queueing theory.

**Detailed**: The Workload Variant Autoscaler (WVA) is a global autoscaler for inference model servers serving Large Language Models (LLMs). It introduces the concept of a "variant" — a way of serving a given model with a particular combination of hardware (GPU type), runtime, and serving approach. Each variant maps to a scale target (Deployment, StatefulSet, or LeaderWorkerSet) and is represented by a `VariantAutoscaling` custom resource.

WVA works alongside standard Kubernetes HPA and external autoscalers like KEDA. Rather than directly scaling deployments in most configurations, WVA continuously monitors request rates and server performance via Prometheus metrics, computes optimal replica counts using saturation-based capacity models (KV cache utilization, queue depth, spare capacity), and emits optimization metrics to Prometheus. External autoscalers (HPA/KEDA) then read these metrics and perform the actual scaling. WVA also supports direct actuation for scale-from-zero scenarios.

The controller supports multi-variant cost optimization — when multiple variants serve the same model (e.g., on different GPU types like A100 vs H100 vs L4), WVA uses a greedy saturation algorithm to allocate capacity to the cheapest variant first and remove it from the most expensive. Configuration is hierarchical: global defaults from ConfigMaps, namespace-level overrides, and per-VA settings via the CR spec.

## Architectural Analysis

The Workload Variant Autoscaler follows a **sensor-analyzer-actuator** pattern common in control systems. The controller separates concerns into distinct internal modules: the Collector layer (Prometheus source with caching), the Engine layer (saturation analysis and optimization), and the Actuator layer (metrics emission and direct scaling). This separation allows the optimization logic to evolve independently from metrics collection and actuation.

**Optimization Architecture**: WVA implements two distinct optimization engines that run as independent goroutine loops within the manager process. The primary **Saturation Engine** runs on a configurable interval (default 60s), collects metrics from Prometheus (with a multi-tier cache featuring TTL, freshness, and staleness thresholds), computes optimal replica counts using the `SaturationV2Analyzer`, and stores decisions in a shared `DecisionCache`. The secondary **Scale-from-Zero Engine** monitors InferencePool CRs for pending requests and directly patches Deployment/LWS replica counts when scaling from zero replicas. The engines use a pipeline pattern with pluggable analyzers, optimizers (greedy saturation algorithm), and limiters (cost-aware capacity limiting).

**Multi-Variant Cost Optimization**: WVA's distinguishing feature is cross-variant optimization. When multiple `VariantAutoscaling` CRs serve the same model (e.g., on different GPU types with different costs), the greedy saturation algorithm allocates capacity to the cheapest variant first and removes it from the most expensive. The `variantCost` field in the CR spec drives this — a lower cost means the variant is preferred for scale-up and last for scale-down. This enables cost-optimized GPU allocation across heterogeneous clusters.

**Configuration Hierarchy**: WVA uses a layered configuration system built on Viper. Configuration sources (in order of precedence): CLI flags > environment variables > ConfigMap values > file-based config > defaults. The `ConfigMapReconciler` watches for labeled ConfigMaps (`app.kubernetes.io/name: workload-variant-autoscaler`) and dynamically updates saturation thresholds, scale-to-zero settings, and queueing model parameters without requiring pod restarts. Namespaces can opt-in or opt-out of WVA configuration via labels (`workload-variant-autoscaler.llmd.ai/config-enabled`) and annotations (`workload-variant-autoscaler.llmd.ai/exclude`).

**LeaderWorkerSet Support**: WVA conditionally supports LeaderWorkerSet (LWS) as a scale target, checking CRD availability at startup via the discovery API. If the LWS CRD is not installed, WVA operates in Deployment-only mode. This design avoids hard dependencies on optional CRDs while supporting distributed inference topologies (e.g., tensor-parallel serving across multiple GPUs/nodes). The scheme registration, controller watches, and index functions all branch on the `lwsEnabled` flag.

**Security Design**: The metrics endpoint is secured with HTTPS (TLS, self-signed or cert-manager) and authenticated via `SubjectAccessReview` — Prometheus must present a valid token with `get` access to `/metrics`. Outbound connections to Prometheus use TLS 1.2+ with configurable CA certificates, optional mTLS client certificates, and `InsecureSkipVerify` as an opt-out for development environments. HTTP/2 is disabled by default to mitigate stream cancellation CVEs (GHSA-qppj-fm5r-hxr3, GHSA-4374-p667-p6c8). The Konflux build enforces FIPS compliance via `GOEXPERIMENT=strictfipsruntime` with dynamic linking (CGO_ENABLED=1).
