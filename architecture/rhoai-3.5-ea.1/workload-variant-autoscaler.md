## Purpose

**Short**: A Kubernetes operator that performs intelligent autoscaling for LLM inference model servers based on saturation metrics, queueing theory models, and GPU-aware cost optimization.

**Detailed**: The Workload Variant Autoscaler (WVA) is a controller-runtime based Kubernetes operator that manages `VariantAutoscaling` custom resources. Each VariantAutoscaling CR represents a "variant" — a specific deployment of a model on particular hardware (e.g., Llama-3.1-8B on A100 GPUs). WVA continuously monitors inference server metrics from Prometheus (KV cache utilization, queue depth, token throughput, latency) and computes optimal replica counts using multiple analysis engines: a percentage-based saturation analyzer (V1), a token-based capacity model (V2), and a queueing-theory model with online Kalman filter parameter learning.

WVA does not directly scale deployments in its default mode. Instead, it emits Prometheus metrics (`wva_desired_replicas`, `wva_current_replicas`, `wva_desired_ratio`) that external autoscalers (HPA, KEDA) consume to drive actual scaling decisions. This decoupled architecture allows WVA to coexist with existing Kubernetes autoscaling infrastructure. WVA also supports a "direct actuator" mode where it patches the scale subresource directly. The operator supports scale-to-zero for idle models, cost-aware multi-variant optimization (preferring cheaper hardware for scale-up and expensive hardware for scale-down), and GPU-constrained cluster-wide resource allocation via a pluggable limiter pipeline.

WVA integrates with the Gateway API Inference Extension's `InferencePool` CRD to discover endpoint picker (EPP) pods and collect flow-control queue metrics for scale-from-zero decisions. It supports both Deployments and LeaderWorkerSets as scale targets, with LWS support conditionally enabled via CRD discovery at startup.

## Architectural Analysis

The Workload Variant Autoscaler employs a sophisticated decoupled architecture where the optimization engine and the scaling actuator are intentionally separated. Rather than directly scaling Deployments (which would conflict with existing HPA/KEDA controllers), WVA computes desired replica counts and publishes them as Prometheus metrics. This design choice allows WVA to coexist with the standard Kubernetes autoscaling ecosystem — HPA reads `wva_desired_replicas` as an external metric and handles the actual scale subresource patching with its own stabilization windows and policies. The direct actuator mode exists as an alternative but is not the recommended deployment pattern.

The three-engine analysis pipeline (V1 percentage-based, V2 token-based, and queueing model) reflects an evolving approach to LLM workload characterization. The V1 analyzer uses simple KV cache and queue depth thresholds — effective but coarse. V2 introduces token-level capacity modeling with a `CapacityKnowledgeStore` that caches deployment-level capacity hints. The queueing model analyzer is the most sophisticated, using an Extended Kalman Filter from the `llm-inferno/kalman-filter` library to online-learn service parameters (alpha, beta, gamma) and predict maximum request rates that satisfy TTFT/ITL SLO targets. These analyzers are selected and configured per-model via ConfigMap-based configuration, with namespace-local overrides for multi-tenant isolation.

The cost-aware optimization pipeline is architecturally notable. When `enableLimiter` is true, WVA treats GPU resources as a finite cluster-wide budget, discovers actual GPU inventory from node GFD labels, and runs a greedy fair-share allocation algorithm that distributes GPUs to models based on priority-weighted scores. This transforms WVA from a per-model autoscaler into a cluster-wide GPU resource allocator — a significant architectural distinction that affects how multiple VariantAutoscaling CRs interact. Scale-up always targets the cheapest variant (lowest cost/capacity ratio), while scale-down removes replicas from the most expensive variant, with a policy that protects at least one replica on the cheapest variant.

The scale-from-zero integration with Gateway API Inference Extension's InferencePool is a critical path for cost optimization. The `ScaleFromZeroEngine` polls inactive VAs at 100ms intervals and queries EPP pod metrics for pending requests in the flow-control queue. This tight polling loop enables rapid reaction to incoming requests for scaled-to-zero models. The EPP metrics are collected either via Prometheus or direct pod scraping (using the `PodScrapingSource`), with bearer token authentication via a dedicated `epp-metrics-reader` ServiceAccount. The architecture supports both `inference.networking.k8s.io` (v1) and `inference.networking.x-k8s.io` (v1alpha2) InferencePool API groups, handled via the `POOL_GROUP` environment variable.

A notable architectural decision is the use of an in-memory `InternalDecisionCache` (with a `DecisionTrigger` channel of capacity 1000) to bridge the optimization engines and the reconciliation controller. The engines write decisions to this shared cache, and a channel notification triggers the reconciler to update VariantAutoscaling status. This avoids an API server round-trip for every optimization cycle and reduces reconciliation latency. However, this means scaling decisions are not persisted across controller restarts — a tradeoff that is acceptable given the short optimization intervals (30-60s).
