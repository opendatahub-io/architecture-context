## Purpose

**Short**: EvalHub is a lightweight REST API service for orchestrating LLM evaluations across multiple backends, tracking experiments via MLflow, and running natively on OpenShift.

**Detailed**: EvalHub provides a centralized evaluation orchestration layer for Red Hat OpenShift AI. It receives evaluation job requests through its REST API, dispatches benchmark executions to pluggable evaluation frameworks (lm-evaluation-harness, RAGAS, Garak, GuideLLM, LightEval, MTEB) via Kubernetes Jobs, and aggregates results with pass/fail scoring. The service supports multi-tenancy through Kubernetes namespace isolation, with authentication and authorization delegated to Kubernetes TokenAccessReview and SubjectAccessReview APIs.

The architecture follows a sidecar proxy pattern for evaluation job pods: an init container (`eval-runtime-init`) downloads test data from S3, the framework adapter runs the evaluation, and a sidecar proxy (`eval-runtime-sidecar`) mediates all communication back to the EvalHub API, MLflow tracking server, and OCI registries. EvalHub is managed on-cluster by the TrustyAI Service Operator via an `EvalHub` custom resource. A Model Context Protocol (MCP) server component (`evalhub-mcp`) enables Claude and other AI assistants to discover and interact with evaluation data.

The service uses Go's standard `net/http` router, structured logging with zap/slog, Prometheus metrics, OpenTelemetry tracing, and a pluggable storage layer (SQLite for development, PostgreSQL for production). Providers and benchmarks are declared in YAML configuration files shipped with the container image.

## Architectural Analysis

EvalHub follows a well-structured microservice architecture with clear separation of concerns. The core service is a pure Go HTTP server using the standard library router (`net/http.ServeMux`), deliberately avoiding heavyweight frameworks. This keeps the dependency footprint small and startup fast. The middleware chain (CORS -> Metrics -> Auth) is applied cleanly, and the service supports dynamic configuration reloading via `fsnotify` file watching -- a pattern that works well with Kubernetes ConfigMap mounts.

The sidecar proxy pattern (`eval-runtime-sidecar`) is a central architectural decision that decouples evaluation framework adapters from platform concerns. Framework adapters (lm-evaluation-harness, RAGAS, Garak, etc.) communicate only with `localhost:8080` (the sidecar), which handles TLS termination, token acquisition, token caching (5-minute TTL), and proxying to the correct upstream service (eval-hub API, MLflow, or OCI registry). This means framework adapters need zero knowledge of cluster networking, authentication, or TLS -- they just POST JSON to localhost. The OCI proxy is particularly sophisticated, implementing OAuth2 challenge-response token negotiation and response rewriting (stripping `WWW-Authenticate` headers, rewriting `Location` redirects).

The authentication architecture delegates entirely to Kubernetes RBAC. Rather than implementing its own auth layer, EvalHub uses `k8s.io/apiserver` authenticators and authorizers, mapping HTTP endpoints to Kubernetes `SubjectAccessReview` checks via a declarative `auth.yaml` configuration. This integrates naturally with OpenShift's RBAC model and enables namespace-scoped multi-tenancy -- the `X-Tenant` header maps directly to a Kubernetes namespace for SAR checks against custom API groups (`trustyai.opendatahub.io`, `mlflow.kubeflow.org`). Cache TTLs (2 min auth, 5 min allow, 30s deny) balance performance with security responsiveness.

The storage layer uses a builder pattern with SQL abstraction supporting both SQLite (dev/test) and PostgreSQL (production). Notably, the SQLite backend uses `modernc.org/sqlite` (pure Go) rather than cgo-based alternatives, which simplifies cross-compilation. All database operations are transactional, with special handling for SQLite concurrency -- transactional reads prevent deadlocks, and write retries handle database lock contention. The schema stores evaluation entities as JSON columns, enabling flexible schema evolution without database migrations.

The MCP server (`evalhub-mcp`) is architecturally independent -- it's a standalone binary that acts as a read-only client of the eval-hub REST API, exposing evaluation data as MCP resources for AI assistant consumption. This is a clean separation that avoids coupling the MCP protocol to the core service. The Python wheel distribution (`python-server`) is another distribution mechanism: it embeds the cross-compiled Go binary inside a Python package, enabling `pip install`-based distribution for local development workflows.

A notable design risk is that the Containerfile (community) and Dockerfile.konflux (RHOAI) diverge significantly: the community build uses `CGO_ENABLED=0` (static linking, no FIPS), while the RHOAI build uses `CGO_ENABLED=1` + `GOEXPERIMENT=strictfipsruntime` (dynamic linking, FIPS-compliant). This means community builds cannot be FIPS-compliant and may exhibit different runtime behavior due to different crypto implementations. The RHOAI build targets 4 architectures (x86_64, ppc64le, s390x, arm64) with hermetic gomod prefetch via Konflux pipelines.
