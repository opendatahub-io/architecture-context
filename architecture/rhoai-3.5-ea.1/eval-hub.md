## Purpose

**Short**: EvalHub is a REST API service for orchestrating LLM evaluations across multiple evaluation framework backends, tracking experiments via MLflow, and managing benchmark collections and providers.

**Detailed**: EvalHub serves as the central orchestration layer for LLM model evaluation within the Red Hat OpenShift AI platform. It exposes a versioned REST API (`/api/v1`) that allows users to submit evaluation jobs against configured providers (lm-evaluation-harness, RAGAS, Garak, GuideLLM, LightEval, MTEB), manage benchmark collections, and track evaluation experiments through MLflow integration. The service supports multi-tenancy with Kubernetes-native authentication (TokenReview) and authorization (SubjectAccessReview) mapped to custom RBAC resources under the `trustyai.opendatahub.io` API group.

The repository produces a single container image containing four binaries: (1) `eval-hub` — the primary HTTP API server with pluggable storage (SQLite/PostgreSQL), configurable runtimes (local process or Kubernetes Job), and OpenTelemetry/Prometheus observability; (2) `eval-runtime-sidecar` — a reverse proxy sidecar deployed alongside evaluation job pods to mediate authenticated access to the eval-hub API, MLflow tracking server, and OCI registries; (3) `eval-runtime-init` — an init container that downloads test data from S3-compatible storage before evaluation jobs run; and (4) `evalhub-mcp` — a Model Context Protocol (MCP) server providing tool, resource, and prompt interfaces for AI-assisted evaluation workflows. EvalHub is managed on-cluster by the TrustyAI Service Operator via an `EvalHub` custom resource (`trustyai.opendatahub.io/v1alpha1`).

## Architectural Analysis

EvalHub follows a classic Go service architecture with clean separation between HTTP handling, business logic, and storage. The `internal/eval_hub` package uses three core abstractions — `Storage`, `Runtime`, and `RuntimeStorage` — to decouple the API handlers from infrastructure concerns. This allows the same API code to work in both local development mode (SQLite + process-based runtimes) and production Kubernetes clusters (PostgreSQL + K8s Job runtimes) with only configuration changes. The `ExecutionContext` pattern threads request-scoped state (request ID, tenant, user, logger) through all handler operations, enabling consistent multi-tenant logging and authorization without global state.

The sidecar architecture is particularly well-designed for the evaluation job use case. Rather than requiring each evaluation adapter (lm-evaluation-harness, RAGAS, etc.) to implement authentication and TLS for multiple backend services, the sidecar handles all credential management centrally. The adapter container communicates with the sidecar over localhost HTTP, while the sidecar manages ServiceAccount token rotation, MLflow token acquisition, and OCI registry OAuth2 challenge flows. Token caching with configurable TTLs (default 5 minutes) reduces unnecessary credential lookups. The init container pattern for S3 test data download is cleanly separated from the sidecar, running as a standard Kubernetes init container with its own credential mount.

The authentication and authorization system maps HTTP endpoints to Kubernetes RBAC resources via a declarative `auth.yaml` configuration. This is a powerful design: instead of implementing custom RBAC, EvalHub delegates all access control decisions to the Kubernetes API server via TokenReview and SubjectAccessReview. The tenant namespace is extracted from the `X-Tenant` HTTP header and used as the SAR namespace, ensuring that users can only access evaluations in namespaces where they have appropriate Role/RoleBinding permissions. The RBAC resource names (`trustyai.opendatahub.io/evaluations`, `mlflow.kubeflow.org/experiments`) are virtual — they don't correspond to actual CRDs but allow fine-grained policy using standard Kubernetes RBAC tooling.

The Kubernetes Job builder (`internal/eval_hub/runtimes/k8s/job_builders.go`, 805 lines) is the most complex single file in the codebase. It constructs evaluation job pods with up to five containers (adapter, sidecar, init for test data) and six volume types (ConfigMap for job spec and sidecar config, emptyDir for shared data, projected SA tokens, Secrets for OCI/model auth and S3 credentials, service-ca bundle). Every container runs as non-root with `SeccompProfile: RuntimeDefault` and all capabilities dropped — a strong security baseline. The sidecar uses the KEP-753 pattern (`restartPolicy: Always` on an init container) rather than a traditional sidecar, which is the modern Kubernetes-native approach. Kueue integration is achieved through a simple label (`kueue.x-k8s.io/queue-name`) set on the Job metadata, allowing cluster administrators to manage evaluation job scheduling without any changes to EvalHub itself.

The MCP server (`evalhub-mcp`) represents a forward-looking integration surface for AI agent workflows. It exposes the full eval-hub API surface through standardized MCP tools (submit_evaluation, get_job_status, cancel_job), resources (providers, benchmarks, collections, jobs), and prompts (edd_workflow, evaluate_model, compare_runs). The prompt system includes embedded guidance YAML files with application-type-specific evaluation workflows (RAG, agent, safety, classifier). Notably, the MCP server is not included in the Konflux production build — it appears only in the development `Containerfile`, suggesting it is either distributed separately or intended for local/dev use with AI coding assistants.
