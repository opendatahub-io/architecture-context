## Purpose

**Short**: A KServe model serving runtime that serves AutoGluon TabularPredictor and TimeSeriesPredictor models via REST and gRPC inference protocols.

**Detailed**: The kserve-autogluon-server is a Python-based model serving image that provides inference capabilities for AutoGluon machine learning models within the Red Hat OpenShift AI (RHOAI) platform. It is deployed as a container within a KServe InferenceService pod, running as the predictor component alongside KServe's storage initializer and queue proxy sidecars.

The server supports two distinct AutoGluon predictor types from a single image: **TabularPredictor** (for structured/tabular data classification, regression, and quantile prediction) and **TimeSeriesPredictor** (for time series forecasting). The server auto-detects the predictor type at load time by attempting `TimeSeriesPredictor.load()` first, then falling back to `TabularPredictor.load()`. Tabular models support both KServe v1 and v2 inference protocols, while time series models support v1 REST JSON only. The server uses the kserve Python SDK (`python/kserve`) as its serving framework, which provides FastAPI-based REST endpoints and gRPC services. It also uses the kserve-storage module (`python/storage`) for downloading model artifacts from S3, GCS, Azure Blob, or PVC storage backends.

The component is built as a Red Hat downstream variant using AutoGluon version 1.5.0+rhaiv.2 (a Red Hat-customized build) and is shipped as a multi-architecture container image (x86_64, ppc64le, s390x, arm64) via Konflux CI/CD.

## Architectural Analysis

The kserve-autogluon-server is a thin adapter layer that bridges AutoGluon's Python ML prediction API to the KServe serving protocol. The server's core design follows a **factory + delegation pattern**: the `AutoGluonDetectedModel` class auto-detects whether a model directory contains a tabular or time series predictor via try-load (attempting `TimeSeriesPredictor.load()` first, then `TabularPredictor.load()`), then delegates all prediction calls to the appropriate model implementation (`AutoGluonTabularModel` or `AutoGluonTimeSeriesModel`). This eliminates the need for users to specify the predictor type in their InferenceService CR.

The tabular model implementation (`tabular_model.py`) handles both v1 and v2 inference protocols with significant effort devoted to correct v2 tensor-to-DataFrame conversion. The `_infer_request_to_dataframe` function maps v2 InferRequest tensors (one per feature, named by feature name) to pandas DataFrames, with validation against the predictor's expected feature list. The v2 output path (`_build_v2_outputs`) handles four distinct result shapes: regression (FP64 scalars), classification (labels with configurable datatype), probability distributions (one output tensor per class when `PREDICT_PROBA=true`), and quantile regression (2D FP64 tensors). This complexity reflects AutoGluon's multi-task design where the same TabularPredictor can serve classification, regression, or quantile problems depending on training configuration.

Time series support is v1-only in this release, with an explicit error message if v2 InferRequest is received. The time series payload format uses a custom JSON structure with `instances` (historical data rows) and optional `known_covariates` (horizon covariates), which differs from the standard v1 instances-as-rows convention. The `_dataframe_to_tsdf` function converts standard DataFrames to AutoGluon's `TimeSeriesDataFrame` format, validating required columns (id, timestamp, target). A notable implementation detail is that `use_cache=False` is passed to `predict()` to avoid writing prediction cache files to the read-only model directory mounted from the storage initializer.

The `runtime_paths.py` module addresses a practical container environment concern: AutoGluon's SeasonalNaive fallback model (used when time series history is too short) attempts to create an `AutogluonModels` directory under the current working directory. In non-root OpenShift containers, the default cwd (`/`) is not writable. The `ensure_autogluon_runtime_paths` function selects a writable working directory from a priority list (`$TMPDIR` → `$HOME` → `/home/kserve`) and also sets `MPLCONFIGDIR=/tmp/matplotlib` for matplotlib cache. This is a defense-in-depth measure for containerized deployments.

The build pipeline is fully hermetic via Konflux, pulling all Python dependencies from Red Hat's internal PyPI index at `console.redhat.com/api/pypi/public-rhai/rhoai/3.4/cpu-ubi9/simple/`. The AutoGluon version (`1.5.0+rhaiv.2`) includes Red Hat-specific patches. The image is built for four architectures (x86_64, ppc64le, s390x, arm64), which is notable given AutoGluon's heavy native dependencies (PyTorch, LightGBM, XGBoost). The `hf-transfer` dependency is explicitly excluded on s390x, and `triton` (NVIDIA GPU library) is excluded on s390x as well, reflecting architecture-specific capability constraints.
