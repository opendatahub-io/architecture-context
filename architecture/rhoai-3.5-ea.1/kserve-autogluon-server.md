## Purpose

**Short**: A KServe model serving runtime that serves AutoGluon TabularPredictor and TimeSeriesPredictor models for inference via REST v1/v2 and gRPC protocols.

**Detailed**: The kserve-autogluon-server is a Python-based model serving container that implements the KServe inference protocol for AutoGluon machine learning models. It supports two model types: **TabularPredictor** (classification, regression, quantile regression) and **TimeSeriesPredictor** (time series forecasting). The server auto-detects the model type at load time by attempting TimeSeriesPredictor.load first, then falling back to TabularPredictor.load.

The server extends the KServe Python SDK's `Model` base class and is deployed as a container within a KServe `InferenceService` pod. Tabular models support both KServe v1 REST JSON and v2 inference protocol (tensor-based), with optional `predict_proba` output for classification tasks. Time series models currently support REST v1 JSON only. The server uses the KServe storage handler (`kserve-storage`) to download models from S3, GCS, Azure Blob Storage, HuggingFace Hub, or PVC-backed storage before serving.

This component is built by Konflux as a multi-architecture image (x86_64, ppc64le, s390x, arm64) from a Red Hat UBI9-based container image. It is deployed as a `ClusterServingRuntime` resource that the KServe controller instantiates when an `InferenceService` requests the `autogluon` model format.

## Architectural Analysis

The kserve-autogluon-server is a relatively straightforward KServe serving runtime that follows the standard KServe Python SDK pattern: extend the `Model` base class, implement `load()` and `predict()`, and let the SDK handle HTTP/gRPC routing, health checks, and metrics. The architecture is simple by design — the complexity lives in AutoGluon itself, not in the serving layer.

The most notable architectural decision is the **auto-detection mechanism** (`predictor_detect.py` and `predictor_factory.py`). Rather than requiring users to specify whether their model is tabular or time series, the server attempts `TimeSeriesPredictor.load()` first, then falls back to `TabularPredictor.load()`. This simplifies the user experience at the cost of potentially loading the wrong predictor type first (and catching the exception). The `AutoGluonDetectedModel` wrapper delegates to the appropriate implementation (`AutoGluonTabularModel` or `AutoGluonTimeSeriesModel`) after detection, which is a clean delegation pattern.

The **v2 tensor protocol support for tabular models** (`tabular_model.py`, lines 211-261) is sophisticated — it maps named input tensors to DataFrame columns matching the model's feature names, with shape validation and bytes-to-string decoding. The output side (`_build_v2_outputs`, lines 263-380) handles the variety of AutoGluon prediction types (regression → FP64, classification → INT64/BYTES, probabilities → per-class FP64 tensors, quantile regression → 2D FP64). Time series models deliberately reject v2 tensor payloads (`timeseries_model.py:172-176`), limiting them to v1 JSON only, which is a pragmatic decision given the complex nested structure of time series requests (history + known covariates).

The `runtime_paths.py` module addresses a practical operational concern: AutoGluon's `SeasonalNaive` fallback creates directories under the current working directory, which fails in read-only container filesystems or when cwd is `/`. The `ensure_autogluon_runtime_paths()` function proactively sets cwd to a writable directory (`/tmp` preferred) and configures matplotlib's cache directory. This is the kind of container-runtime adaptation that is critical for running ML frameworks in restricted OpenShift environments.

The RHOAI build (`Dockerfile.konflux.autogluon`) uses a custom AutoGluon version (`1.5.0+rhaiv.2`) sourced from a Red Hat internal PyPI mirror (`console.redhat.com/api/pypi/public-rhai/rhoai/3.4/cpu-ubi9/simple/`), which ensures the ML framework itself is vetted and patched for the platform. The dependency list in `autogluon-all-requirements.txt` is comprehensive (150+ packages) and fully pinned, providing build reproducibility. The Tekton pipeline enforces hermetic builds with pip prefetch, ensuring no network access during the actual build step.
