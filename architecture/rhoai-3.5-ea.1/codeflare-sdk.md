## Purpose

**Short**: Python SDK for creating, managing, and interacting with Ray clusters and Ray jobs on Kubernetes, specifically within Red Hat OpenShift AI and Open Data Hub environments.

**Detailed**: The CodeFlare SDK is a Python client library that provides a high-level interface for data scientists and ML engineers to manage distributed computing workloads on Kubernetes. It wraps the KubeRay operator API, allowing users to create and manage RayCluster and RayJob custom resources from Jupyter notebooks or Python scripts without needing to write Kubernetes YAML directly.

The SDK handles the full lifecycle of Ray clusters: creating RayCluster CRs with configurable head/worker node resources (CPU, memory, GPUs via extended resources), waiting for cluster readiness, connecting to the Ray dashboard (via HTTPRoute on RHOAI 3.x or OpenShift Routes on RHOAI 2.x), and managing mTLS certificate generation for secure client connections. It integrates with Kueue for workload queuing and resource quota management by automatically detecting and applying LocalQueue labels.

For batch workloads, the SDK provides a RayJob interface that builds and submits RayJob CRs to the KubeRay operator, supporting both existing clusters and ephemeral managed clusters. It handles local file packaging (zipping working directories into Kubernetes Secrets), runtime environment configuration, and Kueue priority class validation. The SDK is designed to run inside RHOAI/ODH workbenches (Jupyter environments) and includes ipywidgets-based UI components for interactive cluster management.

## Architectural Analysis

The CodeFlare SDK follows a layered architecture with clear separation between user-facing API, Kubernetes resource generation, and platform-specific ingress discovery. The top-level `Cluster` and `RayJob` classes provide ergonomic Python APIs, while the internal `build_ray_cluster` module translates `ClusterConfiguration` dataclasses into Kubernetes-native objects using the official `kubernetes` Python client types (V1Container, V1PodSpec, V1ResourceRequirements, etc.). This approach avoids raw dict manipulation for most of the resource construction, improving type safety and IDE support.

The SDK has a sophisticated multi-strategy dashboard URL resolution system that reflects the platform's ingress evolution. On RHOAI 3.x, it queries HTTPRoute CRs via Gateway API labels (`ray.io/cluster-name`, `ray.io/cluster-namespace`), then resolves the parent Gateway to extract the hostname and constructs a path-based URL (`/ray/{namespace}/{cluster}`). On RHOAI 2.x, it falls back to OpenShift Routes. On vanilla Kubernetes (KinD), it falls back to Ingress resources. This cascade is implemented in `_get_dashboard_url_from_httproute()` and `cluster_dashboard_uri()`, with each layer handling its own error cases to avoid blocking the fallback chain.

The mTLS certificate management is a critical architectural component. KubeRay creates a CA secret (`{cluster}-ca-secret-*`) when the head pod starts. The SDK polls for this secret after `cluster.apply()`, then generates a client certificate signed by that CA with comprehensive SAN entries (localhost, 127.0.0.1, ::1, and the head service DNS names). Certificates use RSA-3072 with SHA-256 and include RFC 5280-compliant extensions (BasicConstraints, KeyUsage, ExtendedKeyUsage for both serverAuth and clientAuth, SubjectKeyIdentifier, AuthorityKeyIdentifier). The certificates are stored in a user-private directory (`~/.local/share/codeflare/tls/`) with 0600 permissions, and environment variables (`RAY_USE_TLS`, `RAY_TLS_SERVER_CERT`, etc.) are set to configure the Ray client. The SDK also provides `refresh_certificates()` for CA rotation scenarios and `cleanup_tls_cert()` for automatic cleanup on cluster deletion.

The RayJob subsystem introduces a "managed cluster" pattern where the SDK builds an inline RayCluster spec within the RayJob CR (via `ManagedClusterConfig.build_ray_cluster_spec()`), enabling ephemeral clusters that are created and destroyed with the job. It handles local file packaging by zipping working directories, base64-encoding them into Kubernetes Secrets with owner references (for automatic garbage collection), and configuring init containers to unzip files before the submitter pod runs. The Kueue integration is intentionally non-blocking: if Kueue is not installed, the queue label becomes harmless metadata and the job runs immediately. The SDK validates priority classes before submission but allows submission when verification is impossible (e.g., permission denied), deferring final validation to Kueue's admission webhook.

The vendored KubeRay Python client (`src/codeflare_sdk/vendored/python_client/`) provides low-level CRUD operations for RayCluster and RayJob CRs. This vendoring approach allows the SDK to maintain API compatibility independent of upstream KubeRay Python client releases. The vendored client uses its own Kubernetes config loading (kubeconfig with in-cluster fallback), separate from the main SDK's kube-authkit-based authentication, which creates a subtle dual-config pattern that users should be aware of.
