## Purpose

**Short**: Python SDK that provides an intuitive interface for creating, managing, and submitting jobs to Ray clusters on Kubernetes (OpenShift/RHOAI).

**Detailed**: The CodeFlare SDK is a Python client library that abstracts the complexity of managing Ray clusters and Ray jobs on Kubernetes. It enables data scientists and ML engineers to create RayCluster custom resources, submit RayJob CRs, monitor cluster/job status, and manage TLS certificates for mTLS communication -- all from within Jupyter notebooks or Python scripts running on OpenShift AI (RHOAI) or Open Data Hub (ODH) platforms.

The SDK provides two primary workflows: (1) **Interactive RayCluster management** via the `Cluster` and `ClusterConfiguration` classes, which generate RayCluster CRs, apply them via server-side apply, manage TLS certificates, and integrate with Kueue for resource queuing; and (2) **Declarative RayJob submission** via the `RayJob` and `ManagedClusterConfig` classes, which build RayJob CRs with optional embedded cluster specs, handle local file packaging via Kubernetes Secrets, and support Kueue priority classes. Both workflows support RHOAI v3.0+ Gateway API (HTTPRoute) for dashboard access, with fallback to OpenShift Routes and Kubernetes Ingresses for backward compatibility. Authentication is handled by the `kube-authkit` library, supporting OIDC, OpenShift OAuth, token-based, kubeconfig, and in-cluster service account authentication.

## Architectural Analysis

The CodeFlare SDK follows a layered architecture where the top-level `Cluster` and `RayJob` classes provide high-level workflows, delegating to lower-level builders (`build_ray_cluster.py`, `ManagedClusterConfig.build_ray_cluster_spec`) for Kubernetes resource construction. The authentication layer has been recently modernized from bespoke `TokenAuthentication`/`KubeConfigFileAuthentication` classes to the external `kube-authkit` library, with the old classes preserved as deprecated wrappers for backward compatibility. This migration path is well-documented and the old classes issue `DeprecationWarning` on instantiation.

The RayJob subsystem (`src/codeflare_sdk/ray/rayjobs/`) represents a significant architectural evolution from the interactive `Cluster` workflow. While `Cluster` uses server-side apply and manages TLS certificates directly, `RayJob` delegates cluster lifecycle to the KubeRay operator via `shutdownAfterJobFinishes`, packages local files into Kubernetes Secrets with ownerReferences for automatic cleanup, and supports a submitter pod pattern with init containers for unzipping working directories. The `ManagedClusterConfig` dataclass intentionally mirrors `ClusterConfiguration` but simplifies accelerator configuration and removes autoscaling support (not compatible with Kueue-managed jobs).

The dashboard URL resolution logic in `cluster.py` demonstrates careful multi-version compatibility. The SDK first attempts RHOAI v3.0+ Gateway API resolution by searching for HTTPRoutes with `ray.io/cluster-name` and `ray.io/cluster-namespace` labels, trying cluster-wide search first then falling back to namespace-specific searches across well-known platform namespaces (`redhat-ods-applications`, `opendatahub`, etc.). If no HTTPRoute is found, it falls back to OpenShift Routes, and finally to Kubernetes Ingresses. This layered fallback ensures the SDK works across RHOAI 2.x, RHOAI 3.x, and vanilla Kubernetes/Kind clusters.

The TLS certificate management (`generate_cert.py`) generates RFC 5280-compliant X.509 certificates with proper extensions (BasicConstraints, KeyUsage, ExtendedKeyUsage for both serverAuth and clientAuth, and SubjectAlternativeName including service DNS names). Certificates use RSA-3072 keys with SHA-256 signing, which are FIPS-approved algorithms. The certificate storage follows XDG Base Directory specification with secure 0600 permissions. The SDK includes certificate lifecycle management (refresh, cleanup of expired/old certificates) which is important for long-running notebook sessions.

A notable architectural pattern is the vendored kuberay Python client (`src/codeflare_sdk/vendored/`), which is a fork of the upstream KubeRay Python client. This vendoring approach allows the SDK to evolve the client independently of upstream releases, though it creates a maintenance burden for keeping the vendored code in sync. The vendored client provides `RayClusterApi` and `RayjobApi` classes that wrap `kubernetes.client.CustomObjectsApi` with KubeRay-specific operations.
