## Purpose

**Short**: Manages the lifecycle, networking, authentication, and integration of Jupyter notebook workbenches on OpenShift AI.

**Detailed**: The kubeflow repository contains two complementary Kubernetes controllers that together manage Jupyter notebook workbenches in the Red Hat OpenShift AI (RHOAI) platform. The **notebook-controller** (upstream Kubeflow) handles core lifecycle management — creating StatefulSets, Services, and optional Istio VirtualServices for each Notebook CR, plus idle notebook culling via kernel/terminal activity polling. The **odh-notebook-controller** (ODH/RHOAI extension) layers on enterprise features: Gateway API HTTPRoute-based ingress, kube-rbac-proxy sidecar injection for fine-grained RBAC authentication, mutating/validating admission webhooks for notebook pod customization, and integrations with Data Science Pipelines (DSPA), MLflow, Feast, OpenShift ImageStreams, and cluster proxy configuration.

In RHOAI deployments, both controllers run side-by-side in the `opendatahub` namespace. The notebook-controller manages the core StatefulSet/Service lifecycle and culling, while the odh-notebook-controller handles admission webhooks, ingress routing, network policies, authentication sidecars, and platform integrations. Together they transform a simple Notebook CR into a fully functional, authenticated, and networked workbench with certificate trust chains, pipeline access, and feature store connectivity.

## Admission Webhooks

This component defines 3 webhook(s) (1 mutating, 1 validating, 1 conversion).

| Name | Type | Target Resources | Purpose |
|------|------|-----------------|---------|
| notebooks-validation.opendatahub.io | validating |  | Validates updates to Notebook resources, specifically preventing removal of the MLflow instance annotation while the notebook is running. This guards against a broken state where MLflow environment variables persist in the pod but the associated RoleBinding has already been deleted, leaving the MLflow client without permissions. |
| notebooks.opendatahub.io | mutating |  | Mutating admission webhook that intercepts Kubeflow Notebook create/update requests to inject operational infrastructure: a kube-rbac-proxy authentication sidecar, trusted CA certificate bundles, OpenShift cluster-wide proxy environment variables, pipeline runtime image mounts, Elyra/DSPA pipeline configuration secrets, Feast config volumes, MLflow tracking environment variables, and reconciliation-lock annotations. It also resolves container images from OpenShift ImageStreams and prevents unintended pod restarts by blocking webhook-only spec mutations on running notebooks until they are stopped or restarted. |

## Architectural Analysis

The kubeflow repository implements a **dual-controller architecture** that cleanly separates upstream Kubeflow concerns from RHOAI-specific enterprise features. The notebook-controller handles the fundamental lifecycle loop — creating a StatefulSet with a single replica, a ClusterIP Service on port 80→8888, and optionally an Istio VirtualService. It also owns the culling subsystem, which polls each notebook's Jupyter API (`/api/kernels`, `/api/terminals`) to detect idle notebooks and stop them by setting the `kubeflow-resource-stopped` annotation (which the reconciler translates to `replicas: 0`). This separation means the upstream controller can be updated independently from RHOAI-specific logic.

The odh-notebook-controller is architecturally more complex. Its **mutating admission webhook** is the central integration point — it intercepts every Notebook CREATE and UPDATE to inject kube-rbac-proxy sidecars, resolve ImageStream-based container images, mount CA certificate trust chains (aggregating from `odh-trusted-ca-bundle`, `kube-root-ca.crt`, and `openshift-service-ca.crt`), sync Elyra pipeline configuration secrets from DSPA CRs, mount Feast configuration, inject MLflow environment variables, and propagate cluster-wide proxy settings. The webhook also implements a reconciliation lock (`kubeflow-resource-stopped=odh-notebook-controller-lock`) that prevents the culling controller from stopping a notebook before the ODH controller finishes setting up network resources. A notable design pattern is the **update-pending mechanism**: when mutations would restart a running notebook, the webhook blocks the update and sets `notebooks.opendatahub.io/update-pending` to defer changes until the user explicitly stops or restarts the notebook.

The **Gateway API ingress pattern** is a key RHOAI 3.x architectural decision. HTTPRoutes are created in the controller's central namespace (not the user's namespace), with cross-namespace service references enabled by per-namespace ReferenceGrant resources. This means a single Gateway CR (`data-science-gateway` in `openshift-ingress`) serves all notebook traffic, with path-based routing (`/notebook/{namespace}/{name}`) directing to the appropriate backend. When authentication is enabled, the HTTPRoute points to the `{notebook-name}-kube-rbac-proxy` Service on port 8443 rather than the notebook Service on port 8888. The kube-rbac-proxy sidecar validates tokens via Kubernetes TokenReview/SubjectAccessReview APIs, checking that the user has `get` permission on the specific Notebook CR — providing fine-grained per-notebook access control. Legacy RHOAI 2.x resources (OAuthClients, oauth-proxy sidecars) are cleaned up via finalizers during notebook deletion, facilitating migration.

The controller manages a sophisticated **certificate trust chain** for notebook pods. The mutating webhook aggregates certificates from three ConfigMap sources into a per-namespace `workbench-trusted-ca-bundle` ConfigMap, which is mounted at `/etc/pki/tls/custom-certs` and referenced via six environment variables (`PIP_CERT`, `REQUESTS_CA_BUNDLE`, `SSL_CERT_FILE`, `PIPELINES_SSL_SA_CERTS`, `KF_PIPELINES_SSL_SA_CERTS`, `GIT_SSL_CAINFO`). The controller watches these source ConfigMaps and re-reconciles all affected notebooks when certificates change, ensuring notebook pods always have current trust chains without requiring pod restarts.

A **hermeticity gap** exists in the build pipeline: both Konflux Dockerfiles correctly set FIPS build flags (`CGO_ENABLED=1`, `GOEXPERIMENT=strictfipsruntime`, `-tags strictfipsruntime`) and use UBI9 base images with OpenSSL, but no cachi2/Hermeto prefetch markers or rpms.lock.yaml files are present. Go dependencies are locked via go.sum but fetched at build time rather than pre-fetched. This is a supply chain security consideration for the RHOAI 3.5 release.
