## Purpose

**Short**: Manages Jupyter notebook lifecycle on OpenShift, including pod orchestration, Gateway API ingress, authentication sidecar injection, and integration with data science platform services.

**Detailed**: The kubeflow repository is a downstream fork of the upstream Kubeflow project, maintained by Red Hat for use in Red Hat OpenShift AI (RHOAI). It contains two Kubernetes controllers that work together to manage notebook (workbench) instances:

1. **notebook-controller** (upstream Kubeflow): The core controller that reconciles `Notebook` custom resources into StatefulSets and Services. It manages the notebook pod lifecycle, tracks pod status, and optionally integrates with Istio for routing. It also includes an idle-notebook culling system that monitors Jupyter kernel/terminal activity and scales down inactive notebooks.

2. **odh-notebook-controller** (ODH/RHOAI extension): An OpenShift-aware controller that layers RHOAI-specific capabilities on top of the upstream controller. It creates Gateway API HTTPRoutes for external access, injects kube-rbac-proxy sidecars for authentication, manages per-notebook NetworkPolicies and RBAC bindings, integrates with Data Science Pipelines (DSPA) for Elyra notebook pipeline support, injects MLflow tracking configuration, mounts Feast feature store configs, resolves container images from OpenShift ImageStreams, and aggregates trusted CA certificate bundles. It implements both mutating and validating admission webhooks.

Both controllers run as separate Deployments in the platform namespace and operate on the same `kubeflow.org/v1 Notebook` CRD. The notebook-controller handles core StatefulSet/Service reconciliation while the odh-notebook-controller handles all RHOAI-specific extensions (ingress, auth, integrations).

## Architectural Analysis

The kubeflow repository implements a layered controller architecture where the upstream Kubeflow notebook-controller handles core pod lifecycle management (StatefulSet, Service) while the downstream odh-notebook-controller layers RHOAI-specific capabilities on top. Both controllers reconcile the same `Notebook` CRD but manage different aspects of the notebook's infrastructure. This separation preserves rebase compatibility with upstream Kubeflow while enabling extensive RHOAI-specific behavior.

The odh-notebook-controller's most sophisticated feature is its cross-namespace ingress management. HTTPRoutes are created in the **controller's namespace** (not the user's namespace), providing network isolation — users cannot directly modify their own routing. A `ReferenceGrant` in each user namespace explicitly authorizes the HTTPRoute to reference Services across the namespace boundary. This follows Gateway API's security model and is reference-counted: the ReferenceGrant is shared across all notebooks in a namespace and only deleted when the last notebook is removed, managed via finalizers rather than owner references (which cannot cross namespace boundaries).

The controller supports two ingress modes: authenticated (kube-rbac-proxy) and unauthenticated (direct). In authenticated mode, the controller creates a full sidecar stack per notebook: a `kube-rbac-proxy` container (8443/TCP), a ConfigMap with SAR authorization rules scoped to the specific notebook, a TLS Secret (auto-generated by OpenShift service-ca), a dedicated Service, a ClusterRoleBinding to `system:auth-delegator`, and an HTTPRoute pointing to the proxy service. The authorization rule enforces that the caller must have GET access to the specific `Notebook` resource — leveraging Kubernetes RBAC as the authorization backend. This is the RHOAI 3.x replacement for the RHOAI 2.x oauth-proxy pattern; the controller includes cleanup code for legacy OAuthClient resources.

The mutating webhook is the integration hub, performing 10+ distinct mutations on Notebook CREATE/UPDATE events: reconciliation locking (preventing pod startup until the controller finishes initial setup), ImageStream-based image resolution, CA certificate bundle injection (aggregating 3 sources into environment variables for pip, requests, git, and pipeline libraries), pipeline runtime image mounting, Elyra DSPA secret injection, Feast config mounting, MLflow environment variable injection, kube-rbac-proxy sidecar injection, cluster proxy environment variable injection, and running-notebook update prevention. The webhook and controller coordinate to prevent race conditions — for example, the webhook syncs pipeline-runtime-images and DSPA secrets before mounting them, handling the case where a namespace's first notebook would otherwise miss these resources.

Build-time FIPS compliance is well-implemented: both Konflux Dockerfiles use `CGO_ENABLED=1`, `GOEXPERIMENT=strictfipsruntime`, and `-tags strictfipsruntime` to produce dynamically-linked Go binaries that delegate to OpenSSL at runtime. The UBI9 base image provides OpenSSL. No non-FIPS crypto libraries or patterns were found in the source code. The controllers use controller-runtime's default TLS configuration, which inherits the FIPS-compliant crypto backend.
