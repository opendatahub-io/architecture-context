## Purpose

**Short**: Manages the lifecycle of Data Science Pipelines (Kubeflow Pipelines v2) custom resources and their associated Kubernetes infrastructure on Red Hat OpenShift AI.

**Detailed**: The Data Science Pipelines Operator (DSPO) is a controller-runtime-based Kubernetes operator that reconciles `DataSciencePipelinesApplication` (DSPA) custom resources. Each DSPA instance represents a complete data science pipeline stack, and the operator deploys and manages all infrastructure components required for pipeline authoring, execution, and metadata tracking.

The operator manages six core sub-components per DSPA instance: an API Server (Kubeflow Pipelines v2 backend), a Persistence Agent (syncs pipeline run metadata to the database), a Scheduled Workflow controller (cron-based pipeline triggering), an Argo Workflow Controller (pipeline step execution engine), ML Metadata services (gRPC server + Envoy proxy for artifact/model lineage tracking), and optional managed infrastructure including MariaDB for pipeline metadata storage and MinIO for artifact object storage. Each component is deployed via Go-templated Kubernetes manifests rendered with DSPA-specific parameters and applied through the manifestival library.

The operator implements comprehensive health checking, status management (10 condition types), Prometheus metrics (9 gauges), and security features including pod-to-pod TLS via OpenShift service-ca, kube-rbac-proxy sidecar injection for API server authentication, custom CA bundle aggregation, FIPS-compliant builds, and managed pipeline validation with OCI registry allowlisting. It integrates with external systems including Argo Workflows, Kubeflow Pipelines CRDs, Ray clusters, KServe inference services, and CodeFlare AppWrappers.

## Architectural Analysis

The DSPO follows a template-driven reconciliation pattern where all Kubernetes resources for managed components are defined as Go templates (`*.yaml.tmpl` files) under `config/internal/`. During reconciliation, the operator renders these templates with the `DSPAParams` struct (populated from the DSPA CR spec, config file, environment variables, and Kubernetes secrets) and applies them via the manifestival library. This approach provides excellent separation between resource definitions and configuration logic, making it straightforward to add new resources or modify existing ones without changing controller code. The template system also supports the FIPS-enabled launcher variant (`launcher-v2-fips`) through conditional template rendering based on the `FIPSEnabled` configuration flag.

The operator implements a strict prerequisite chain: database and object storage health checks must pass before any application components are deployed. This prevents cascading failures where an API server starts but cannot connect to its database. The health check system is thoughtfully designed with disableable checks (for air-gapped environments where external endpoints may not be reachable during deployment) and distinct error classification — X509 unknown authority errors produce actionable suggestions about configuring custom CA bundles.

The managed pipelines system is architecturally notable for its OCI-native design. Rather than shipping pipeline definitions in the operator image or a ConfigMap, pipelines are packaged as OCI container image layers. The operator uses `go-containerregistry` to resolve image manifests, validate against a registry allowlist, and extract `managed-pipelines.json` from image layers — complete with OCI whiteout marker support for proper layer semantics. Results are cached per content digest with a 10-minute TTL. The init container pattern (pipelines-components image mounted at `/config/managed-pipelines`) ensures pipeline definitions are available to the API server at startup. The validation distinguishes permanent errors (invalid image ref, registry not allowed, manifest malformed) from transient errors (network timeout), with permanent errors blocking deployment and transient errors triggering requeue.

The security architecture uses kube-rbac-proxy as the authentication/authorization layer for externally-exposed endpoints. When `enableRoute` is true for the API server, a kube-rbac-proxy sidecar container is injected into the deployment pod, listening on port 8443 and proxying authenticated requests to the API server on port 8888. Authentication is performed via Kubernetes TokenReview and SubjectAccessReview (SAR), ensuring that only users with appropriate RBAC permissions can access the pipeline API. The same pattern is applied to the MLMD Envoy proxy. Pod-to-pod TLS (enabled by default) uses OpenShift service-ca to auto-generate TLS certificates for internal services, with the operator watching for certificate creation events and re-reconciling to distribute certificates to dependent components.

The operator's cache management strategy is noteworthy: ConfigMap and Secret data payloads are stripped from the informer cache (to reduce memory) and read via direct API calls when needed. Pod watches are scoped by label selector (`data-science-pipelines=true`), and webhook configurations are filtered by name. This reduces the operator's memory footprint significantly compared to watching all resources cluster-wide. The `GOMEMLIMIT=3600MiB` environment variable provides an additional memory safety net.
