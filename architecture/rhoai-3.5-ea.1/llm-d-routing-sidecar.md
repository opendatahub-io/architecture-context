## Purpose

**Short**: A reverse proxy sidecar that orchestrates disaggregated prefill/decode (P/D) inference by routing requests between prefill and decode vLLM workers using the NIXL or LMCache KV-transfer protocols.

**Detailed**: The llm-d-routing-sidecar runs as a sidecar container alongside a vLLM decoder instance in a Kubernetes pod. It intercepts incoming OpenAI-compatible completion requests and, when a `x-prefiller-host-port` header is present, orchestrates a two-phase inference pipeline: first forwarding the request to a remote prefill worker to compute KV cache entries, then forwarding the modified request (with KV transfer parameters) to the local decoder for token generation. When no prefiller header is present, requests pass through directly to the local decoder.

The sidecar supports three P/D connector protocols: NIXLv2 (current default), NIXLv1 (deprecated), and LMCache (deprecated). It includes SSRF protection that can be enabled via feature flag, which watches Kubernetes `InferencePool` resources (from the Gateway API Inference Extension) to maintain a dynamic allowlist of valid prefill targets based on pod IPs matching the pool's label selectors.

**Note**: This repository is deprecated. The code has been moved to the [llm-d-inference-scheduler](https://github.com/llm-d/llm-d-inference-scheduler) repository under `cmd/pd_sidecar` and `pkg/sidecar`. No further development will occur in this repository.

## Architectural Analysis

The llm-d-routing-sidecar implements a clean separation between protocol handling and proxy mechanics. The core `Server` struct uses a `protocolRunner` function type to abstract the three P/D connector protocols (NIXLv2, NIXLv1, LMCache), making the protocol selection a configuration concern rather than an architectural one. Each protocol follows the same pattern: modify the incoming request for prefill (typically setting `stream=false` and `max_tokens=1`), forward to the remote prefiller, extract KV transfer metadata from the response, inject that metadata into the decode request, and forward to the local decoder. The key difference between NIXLv1 and NIXLv2 is the transfer metadata format: v1 uses four separate fields (`remote_block_ids`, `remote_engine_id`, `remote_host`, `remote_port`) while v2 consolidates them into a single `kv_transfer_params` object.

The SSRF protection mechanism is notable for its Kubernetes-native approach. Rather than maintaining a static allowlist, the `AllowlistValidator` uses Kubernetes informers to watch an `InferencePool` custom resource (from the Gateway API Inference Extension project, group `inference.networking.x-k8s.io/v1alpha2`). It extracts the pool's label selector, then watches all pods matching those labels to build a dynamic set of allowed IPs and hostnames. The allowlist is host-only (not port-specific), meaning any port on an allowed host is permitted. This design ensures the allowlist stays current as pods scale up/down without requiring explicit configuration changes.

The proxy uses an LRU cache (capacity 16) for `httputil.ReverseProxy` instances keyed by prefiller host:port. This avoids creating a new reverse proxy for every request to the same prefiller, which would be wasteful given that proxy setup includes TLS configuration. The TLS configuration is consistently applied across all three proxy contexts (server, decoder client, prefiller client) with the same set of FIPS-approved cipher suites and minimum TLS 1.2. However, the `InsecureSkipVerify` flags for both prefiller and decoder connections represent a deliberate trade-off for development/testing flexibility at the cost of security in production if misconfigured.

A design risk worth noting is that this repository is deprecated and the code has moved to `llm-d-inference-scheduler`. Any analysis of the RHOAI 3.5 deployment should verify which repository actually provides the container image being deployed. The StatefulSet manifest in `deploy/` deploys the sidecar as a standalone pod rather than as an actual sidecar container, suggesting the manifests are primarily for standalone testing rather than production use. In production, the sidecar is injected into decoder pods by the platform operator or inference scheduler.

The HTTP server configuration intentionally omits `ReadTimeout` and `WriteTimeout` (commented as suitable for LLM inference which can take hours for large contexts), setting only `IdleTimeout` (5 minutes) and `ReadHeaderTimeout` (30 seconds). This is appropriate for the LLM inference use case but means the server could hold connections indefinitely during inference.
