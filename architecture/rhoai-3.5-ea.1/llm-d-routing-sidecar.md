## Purpose

**Short**: A reverse proxy sidecar that implements disaggregated prefill/decode (P/D) routing for vLLM inference workloads in the llm-d inference stack.

**Detailed**: The llm-d-routing-sidecar is a Go-based HTTP reverse proxy designed to run as a sidecar container alongside a vLLM decoder instance. It intercepts incoming LLM inference requests (OpenAI-compatible `/v1/chat/completions` and `/v1/completions` endpoints) and orchestrates disaggregated prefill/decode workflows where the KV cache prefill phase is offloaded to a remote prefiller node while the local decoder handles token generation.

The sidecar supports multiple P/D connector protocols: NIXL v2 (default/recommended), NIXL v1 (deprecated), and LMCache (deprecated). When a request includes the `x-prefiller-host-port` header, the sidecar forwards the prompt to the specified prefiller for KV cache population, then sends the decode request to the local vLLM instance with the KV transfer parameters. Requests without this header are passed through directly to the local decoder.

The sidecar includes an optional SSRF (Server-Side Request Forgery) protection feature that watches Kubernetes `InferencePool` custom resources (from the Gateway API inference extension) and builds an allowlist of valid prefill targets based on pod IPs and names matching the pool's label selectors. This ensures that the `x-prefiller-host-port` header cannot be used to redirect requests to arbitrary hosts. **Note**: This repository is deprecated; all future development has moved to the `llm-d-inference-scheduler` repository under `cmd/pd_sidecar` and `pkg/sidecar`.

## Architectural Analysis

The llm-d-routing-sidecar implements a clean sidecar pattern for disaggregated LLM inference. It sits alongside a vLLM decoder pod and transparently intercepts OpenAI-compatible API requests, splitting them into prefill and decode phases when a routing header is present. The design is deliberately simple — the sidecar has no persistent state, no CRDs of its own, and no admission webhooks. All routing intelligence comes from the upstream `x-prefiller-host-port` header, which is set by the llm-d-inference-scheduler.

The three connector protocol implementations (LMCache, NIXL v1, NIXL v2) show the evolution of the P/D disaggregation approach. NIXL v2 (the current default) uses a structured `kv_transfer_params` JSON field to pass block IDs, engine IDs, and remote host/port between prefill and decode phases. The older NIXL v1 protocol uses flat top-level fields (`remote_block_ids`, `remote_engine_id`, etc.) and the even older LMCache protocol simply sets `max_tokens=1` on the prefill request without extracting transfer parameters. Both v1 and LMCache are deprecated.

The SSRF protection feature is architecturally interesting: it uses Kubernetes dynamic client watches (not controller-runtime) to monitor `InferencePool` CRs from the Gateway API inference extension. When an InferencePool is created or updated, the validator extracts its pod selector, starts a new pod informer for that selector, and builds an allowlist of pod IPs and hostnames. This is a host-only check — any port on an allowed host is permitted. The 30-second resync period ensures the allowlist stays reasonably current as pods scale up/down. The use of `unstructured.Unstructured` (dynamic client) rather than typed clients avoids importing the full Gateway API type system as a dependency.

The TLS configuration is thorough: the proxy server, decoder proxy, and prefiller proxy all use explicit cipher suite lists restricted to FIPS-compatible algorithms (AES-GCM and ChaCha20-Poly1305 with ECDHE key exchange). The minimum TLS version is 1.2 everywhere. However, the `InsecureSkipVerify` options for both prefiller and decoder connections could weaken security in production if enabled. The self-signed certificate fallback is useful for development but should be replaced with proper certificates (via `-cert-path`) in production deployments.

This repository is deprecated, with the code migrated to `llm-d-inference-scheduler` under `cmd/pd_sidecar` and `pkg/sidecar`. The RHOAI 3.5 EA release likely uses this standalone version, but future releases will use the integrated version from the inference scheduler repository.
