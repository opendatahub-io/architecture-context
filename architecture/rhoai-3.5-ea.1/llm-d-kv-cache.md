## Purpose

**Short**: A pluggable KV-cache management suite for LLM inference that enables cache-aware routing, cross-node cache coordination, and efficient storage offloading for vLLM-based serving platforms.

**Detailed**: The llm-d-kv-cache repository provides the KV-cache management layer for the llm-d distributed LLM inference platform. The primary component is the **KV-Cache Indexer**, a Go library that maintains a near-real-time, globally consistent view of KV-cache block residency across a fleet of vLLM pods. It subscribes to KV-Events emitted by model servers via ZeroMQ and exposes a fast scoring API that ranks pods by how much of a request's prefix they already have cached. This library is designed to be embedded in a host process (llm-d's EPP is the reference consumer), not run as a standalone service.

The repository also ships a **UDS Tokenizer Service** (Python gRPC sidecar), which provides tokenization and chat template rendering via Unix Domain Socket for the host process. Additionally, it includes the **llmd-fs-backend** storage connector (Python/C++/CUDA) for offloading KV-cache blocks between GPU memory and shared storage (POSIX filesystem, NFS, or S3), and a **PVC Evictor** utility that automatically manages disk space for offloaded KV-cache files on PersistentVolumeClaims.

## Architectural Analysis

The llm-d-kv-cache repository follows a **library-first architecture** where the core KV-cache indexing and scoring functionality is designed to be embedded in a host process rather than deployed as a standalone service. This is an unusual pattern for the RHOAI platform, where most components are operators or microservices. The implication is that the Go library's security and networking posture (FIPS, TLS, RBAC) is inherited from the host process — the llm-d EPP — rather than being self-contained. Analysis of this component alone cannot fully characterize its production security profile.

The **dual-key design** is a sophisticated architectural decision that solves a fundamental problem: scoring requires computing block keys from prompts (request keys), but eviction events from engines carry engine-specific keys. The indexer bridges these two hash spaces on ingestion by recording an engine-key-to-request-key mapping alongside the primary index entry. This keeps the read path (scoring) fast — it never touches engine keys — at the cost of maintaining a secondary lookup table. The pattern is well-suited to the access patterns: ingestion is bursty but scoring must be low-latency.

The **ZMQ event transport** is the primary integration point with vLLM, and it operates without encryption or authentication. Events flow from vLLM pods to the indexer via ZMQ PUB/SUB on port 5557/TCP. Security relies entirely on Kubernetes network policies and namespace isolation. The system supports two delivery modes — centralized (single SUB socket) and pod-discovery (per-pod subscriptions via K8s reconciler) — with pod-discovery being the recommended production mode for multi-replica deployments. This means the indexer host needs RBAC permissions to list/watch pods.

The **UDS tokenizer sidecar** is the only Konflux-built image shipped from this repository (`odh-llm-d-kv-cache-rhel9`). It communicates exclusively via Unix Domain Socket with 0o700 permissions, which provides strong pod-level isolation without requiring TLS. The gRPC server supports 100MB message sizes and includes Istio-compatible keepalive settings, suggesting production deployment behind a service mesh. The health probe on port 8082 is the only network-facing endpoint. gRPC reflection is disabled by default — a security-conscious choice documented in the code.

The **llmd-fs-backend** storage connector is architecturally interesting because it runs inside the vLLM process as a pip-installed package with native C++/CUDA extensions. It implements an async transfer pipeline with CPU staging buffers, read-preferring thread pools, and support for GPU Direct Storage (GDS). The connector supports both POSIX filesystem and S3/OBJ backends via the NIXL abstraction layer. The `metrics.py` module solves a practical problem — disambiguating Prometheus metric names when multiple OffloadingConnector instances run in the same vLLM process — by monkey-patching the metric initialization to append a connector-specific suffix.

The **PVC evictor** uses an N+2 multi-process architecture (N crawlers + activator + deleter) with hex-modulo-based file distribution for load balancing. This is a well-designed eviction system with hysteresis-based triggering (cleanup threshold vs target threshold) that prevents oscillation. The crawler processes use the same FileMapper path structure as the fs-backend connector, ensuring they correctly identify and prioritize eviction of KV-cache block files based on access time. The process communication uses Python multiprocessing primitives (Queue, Event) rather than network protocols, keeping the eviction system self-contained.
