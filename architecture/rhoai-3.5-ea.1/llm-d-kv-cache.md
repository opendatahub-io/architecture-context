## Purpose

**Short**: A pluggable KV-cache indexing and scoring library for KV-cache-aware routing in vLLM-based LLM inference platforms.

**Detailed**: The `llm-d-kv-cache` component provides a comprehensive suite of KV-cache management capabilities for the `llm-d` distributed LLM inference platform. Its primary function is to maintain a global, near-real-time view of KV-cache block locality across a fleet of vLLM pods and to score pods based on how well their cached KV blocks match incoming prompts. This enables intelligent routing: schedulers can direct inference requests to pods that already have relevant KV blocks cached, dramatically improving Time To First Token (TTFT) and overall throughput.

The component is architected as a Go library (consumed by the `llm-d-inference-scheduler`) with three major subsystems: (1) a **KV-block indexer** that tracks which blocks reside on which pods and on which storage tier (GPU, CPU, disk), (2) an **event processing pipeline** that ingests KVEvents from vLLM or SGLang engines via ZMQ and updates the index in near-real-time, and (3) a **tokenization system** that converts prompts to token IDs for block key computation. The tokenization system supports two modes: embedded Python tokenizers via CGo, or a standalone gRPC tokenizer sidecar service (the UDS Tokenizer). The repository also includes KV connector modules: a shared-storage filesystem backend for cross-node KV-cache offloading (`llmd_fs_backend`) and a PVC evictor utility for automatic disk space management.

## Architectural Analysis

The `llm-d-kv-cache` component follows a well-structured library architecture with clear separation of concerns across three main subsystems: indexing, event processing, and tokenization. The design prioritizes pluggability at every layer - storage backends (in-memory, Redis, Valkey, cost-aware memory), inference engine adapters (vLLM, SGLang), and tokenization modes (embedded CGo, external UDS gRPC) are all abstracted behind clean interfaces.

The indexing subsystem (`pkg/kvcache/kvblock/`) implements a sophisticated multi-tier architecture. The `InMemoryIndex` uses `hashicorp/golang-lru` with per-block pod caches, while the `CostAwareMemoryIndex` leverages `dgraph-io/ristretto` for memory-bounded eviction based on actual byte costs. The `RedisIndex` uses pipelined commands for single-RTT batch operations and includes a Lua script for atomic cleanup. All index implementations are wrapped with instrumentation (Prometheus metrics) and tracing (OpenTelemetry) via decorator patterns, demonstrating good observability design.

The event processing pipeline (`pkg/kvevents/`) is designed for high throughput with a sharded worker pool architecture. Messages are routed to workers via FNV-1a hashing on the pod ID (sharding key), ensuring all events from the same pod are processed in order. The engine adapters use positional field access with length guards rather than fixed structs, providing forward/backward compatibility as vLLM and SGLang evolve their event formats. The ZMQ subscriber implementation deliberately disables auto-reconnect to avoid data races, instead implementing its own retry loop.

The tokenization system has an interesting dual-mode design. The embedded mode uses CGo to call Python (vLLM's tokenizer) from Go, which avoids network overhead but requires Python development headers at build time. The UDS mode communicates with a separate Python gRPC server over Unix Domain Sockets, which simplifies the Go build but adds serialization overhead. The `CompositeTokenizer` implements a fallback chain pattern, trying each tokenizer until one succeeds. The UDS tokenizer service is container-aware, using cgroup detection for thread pool sizing, and implements the full OpenAI-compatible rendering pipeline via vLLM's `OpenAIServingRender`.

The block key computation algorithm (`pkg/kvcache/kvblock/token_processor.go`) faithfully mirrors vLLM's prefix-caching hash algorithm, using FNV-64a over CBOR-encoded token chunks with hash chaining. This ensures that the indexer's block keys exactly match what vLLM computes internally, which is critical for accurate cache hit detection. The multimodal support (`extra_keys.go`) extends this to handle image and other non-text modalities by tainting block hashes with multimodal metadata.

A notable architectural risk is the lack of encryption on the ZMQ transport between vLLM pods and the indexer. KVEvents contain block hashes and metadata but not actual model weights or inference data, so the exposure is limited to cache topology information. However, in multi-tenant environments, this could leak information about which models are being served and their usage patterns. The Redis/Valkey backend supports TLS via `rediss://` addressing, but no mTLS or authentication is configured by default.
