## Purpose

**Short**: Container image packaging the Llama Stack server with pre-configured AI providers for inference, safety, vector storage, and agentic workflows within Red Hat OpenShift AI.

**Detailed**: The llama-stack-distribution repository builds and ships a containerized Llama Stack server — Meta's open-source framework for building AI applications — packaged as a Red Hat-compatible container image based on UBI 9 (Python 3.12). The image bundles all necessary Python dependencies, pre-configured provider integrations, and an embedded embedding model (IBM Granite Embedding 125M English) so it can serve inference, safety, vector I/O, file management, tool runtime, batch processing, and agentic response APIs out of the box.

The repository does not contain any Llama Stack source code itself. Instead, it acts as a distribution layer: a `build.py` script generates the `Containerfile` by querying the upstream `llama-stack` CLI for provider dependencies, pins versions for reproducibility, and templates those into a multi-step container build. The runtime configuration (`build.yaml` / `config.yaml`) defines which providers are available (vLLM, Bedrock, Watsonx, Azure, Vertex AI, OpenAI for inference; Milvus, pgvector, Qdrant for vector I/O; TrustyAI FMS for safety) and how they are activated via environment variables. A companion vLLM CPU container image is also defined for local development and CI testing.

The distribution image is built by Konflux (via `Dockerfile.konflux`) for RHOAI releases and by GitHub Actions for upstream development. The Konflux build uses a pre-built base image from the AIPCC pipeline with pre-installed wheel releases, while the upstream build installs dependencies directly from PyPI and GitHub. The server exposes a single HTTP API on port 8321, supports optional OAuth2 JWT authentication, and integrates with PostgreSQL for persistent state storage.

## Architectural Analysis

The llama-stack-distribution repository is architecturally a **thin packaging layer** over the upstream Llama Stack framework. It contains no application logic of its own — no controllers, no custom API handlers, no business logic. Its value lies entirely in: (1) selecting and pinning the correct versions of upstream components and their transitive dependencies, (2) configuring the provider ecosystem via `build.yaml`, and (3) producing a reproducible, RHEL-compatible container image. This is a deliberate design choice — keeping the distribution layer minimal means upstream Llama Stack upgrades require only version bumps and configuration changes, not code modifications.

The **provider activation pattern** is notable. Rather than requiring separate deployments for each AI backend, the distribution uses a conditional environment variable pattern (`${env.VLLM_URL:+vllm-inference}`) that dynamically activates providers at runtime. When an environment variable is set, its corresponding provider becomes active; when unset, the provider is silently skipped. This allows a single container image to serve many different deployment topologies — from a minimal local setup (vLLM only) to a multi-cloud configuration (Bedrock + Watsonx + Azure + Vertex AI + OpenAI) — without rebuilding the image. The trade-off is a large image size (all provider dependencies are always installed) and a complex configuration surface (30+ environment variables).

The **dual build path** — Konflux (`Dockerfile.konflux`) for RHOAI releases and GitHub Actions (`distribution/Containerfile`) for upstream — introduces a significant divergence risk. The Konflux build uses a pre-built AIPCC base image with wheel releases and private GitLab PAT authentication, while the upstream build installs directly from PyPI and GitHub. The `build.py` script generates only the upstream `Containerfile`, not the Konflux Dockerfile, meaning the two can drift if dependencies change. The `konflux/cpu-ubi9.conf` file pins the base image and wheel release versions for the Konflux path, providing an independent version control mechanism. The `Containerfile.in` template approach (with `{dependencies}` and `{llama_stack_install_source}` placeholders) is a clean separation of build logic from dependency resolution.

The **storage architecture** deserves attention. The server requires PostgreSQL for production use, with two distinct backend types: `kv_postgres` (key-value store for metadata, prompts, agent state, batch jobs) and `sql_postgres` (relational store for inference logs, conversations, file metadata, agent responses). Both share the same PostgreSQL instance by default but use separate tables. The embedded Milvus Lite and FAISS providers are available as dependency-only options for development but are explicitly documented as incompatible with multi-worker deployments. This PostgreSQL dependency is the most significant operational requirement for deploying this component.

The **security model** is minimalist at the application level. Authentication is optional (activated by setting `AUTH_ISSUER`) and delegates to an external OIDC provider via JWKS-based JWT validation. The access control policy is basic: unauthenticated users can read unowned resources, authenticated users can create resources, and resource owners have full CRUD. There is no role-based access control, no multi-tenancy isolation, and no rate limiting at the application level. In the RHOAI deployment context, these concerns are expected to be handled by the platform's ingress stack (kube-rbac-proxy, Gateway API, network policies). The application itself listens on plaintext HTTP (port 8321) and relies on external TLS termination.
