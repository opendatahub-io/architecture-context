## Purpose

**Short**: OGX Distribution is a container image packaging of the OGX (formerly Llama Stack) AI inference and agent orchestration server for Red Hat OpenShift AI, bundling pre-configured inference providers, vector stores, tool runtimes, and storage backends.

**Detailed**: This repository produces a self-contained OGX server container image — `odh-ogx-core` — that provides a unified API gateway for AI inference, agentic responses, vector I/O, file storage, and tool orchestration. The image wraps the upstream [OGX project](https://github.com/opendatahub-io/ogx) (v1.0.2+rhaiv.0) with a curated set of providers and dependencies suitable for Red Hat OpenShift AI deployments.

The distribution supports multiple inference backends — including vLLM (primary, self-hosted), AWS Bedrock, IBM watsonx, Azure OpenAI, Google Vertex AI, and generic OpenAI-compatible endpoints — all activated via environment variables at runtime. For vector storage, it supports Milvus (inline and remote), FAISS (inline), pgvector, and Qdrant. It uses PostgreSQL for persistent storage of key-value data, inference logs, agent state, conversation history, and batch processing state.

The repository does not contain application source code; it is a distribution build system. The `distribution/build.py` script generates `Containerfile`, `install-deps.sh`, and `config.yaml` from the authoritative `build.yaml`. The Konflux pipeline (`Dockerfile.konflux`) builds the RHOAI-hardened image from a base image provided by the AIPCC team (`quay.io/aipcc/base-images/cpu`), while the upstream-community `Containerfile` builds from `ubi9/python-312`.

## Architectural Analysis

The OGX distribution repository follows a **distribution-as-code** pattern rather than a traditional application repository. It contains no application source code — the OGX server, its APIs, providers, and orchestration logic all come from the upstream `ogx` package (v1.0.2+rhaiv.0, from the `opendatahub-io/ogx` fork). This repository's value is in its build system (`distribution/build.py`), configuration authoring (`distribution/build.yaml`), and container image assembly.

The build system uses a split between `build.yaml` (authoritative, contains all providers including dependency-only ones like `inline::milvus`, `inline::faiss`, `inline::sentence-transformers`) and `config.yaml` (auto-generated, stripped of dependency-only providers). This split enables the image to ship with pre-installed dependencies for providers that are available but not default-enabled, allowing users to activate them by mounting a custom `config.yaml` at runtime. The `STRIPPED_PROVIDER_TYPES` set in `build.py` controls this split and includes validation to prevent drift between `build.yaml` and the strip list.

The **provider activation pattern** is notable: most remote providers are conditionally activated using OGX's `${env.VAR:+provider_id}` syntax, where the provider is only registered if the referenced environment variable is non-empty. This means a single container image supports deployment against vLLM, Bedrock, watsonx, Azure, Vertex AI, OpenAI, Milvus, pgvector, Qdrant, and S3 simultaneously — the operator or user simply sets the relevant environment variables to activate the desired backends. This is architecturally elegant but means the image bundles dependencies for all providers regardless of which are used.

The **dual Dockerfile strategy** deserves attention: `Dockerfile.konflux` builds from a curated AIPCC base image with pre-installed tools and RPM management (`fromager-rpm-check.py`, `selftest.py`, `scanlibs.py`, `public-repos.sh`), while `distribution/Containerfile` builds from vanilla `ubi9/python-312`. The Konflux Dockerfile runs as a non-root user (`CNB_USER_ID:CNB_GROUP_ID`) with proper security posture, while the community Containerfile has no USER directive (noted as an issue in `component-architecture.json`). The Konflux build supports three architectures: `linux/x86_64`, `linux-m2xlarge/arm64`, and `linux/ppc64le`.

The **supply chain hermeticity gap** is the most significant risk: no lock files exist for Python dependencies or RPMs, the push pipeline runs non-hermetically (`hermetic: false`), and dependencies are resolved live from PyPI with some critical packages pinned. While the pinned versions (boto3==1.35.88, pymilvus==2.6.9, tiktoken==0.12.0) reduce variability for known-problematic packages, the majority of the dependency tree (fastapi, httpx, transformers, torch, etc.) is unpinned. The RHAI private package index (`packages.redhat.com/api/pypi/public-rhai/rhoai/3.5-EA2/cpu-ubi9-test/simple`) is used as an extra index URL in `Dockerfile.konflux`, suggesting some packages are served from a controlled internal registry, but this is not comprehensive.

The repository has a thorough **CI testing strategy** with GitHub Actions: smoke tests validate the health endpoint, model listing, chat completion, and PostgreSQL persistence, while integration tests run the upstream ogx pytest suite against the distribution's running server. Multi-architecture testing (amd64 and arm64) is performed either locally (with vLLM CPU containers) or against MaaS endpoints when configured. The vLLM CPU CI helper image (`vllm/Containerfile`) pre-downloads models to avoid CI network dependencies, which is a pragmatic optimization for test reliability.
