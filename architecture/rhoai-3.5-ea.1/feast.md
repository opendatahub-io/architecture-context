## Purpose

**Short**: Feast is a feature store operator and runtime that manages the lifecycle of ML feature definitions, serving online features with low latency, and materializing features from offline to online stores.

**Detailed**: Feast (Feature Store) provides the infrastructure for managing and serving machine learning features in production. The RHOAI deployment consists of two primary Konflux-built components: the **feast-operator** (a Go-based Kubernetes operator) and the **feature-server** (a Python-based FastAPI/gRPC service). The operator watches `FeatureStore` custom resources and deploys multi-service pods containing the feature server processes (offline store, online store, registry, and UI), manages TLS, authorization (OIDC and Kubernetes RBAC), horizontal pod autoscaling, CronJobs for materialization, and integrates with ODH Notebooks.

The feature server provides REST and gRPC APIs for online feature retrieval, feature push operations, materialization, a registry REST API for metadata management, and optional MCP (Model Context Protocol) support for LLM integration. It supports pluggable online stores (Redis, PostgreSQL, DynamoDB, SQLite, Cassandra, Elasticsearch, and more), offline stores (Snowflake, BigQuery, DuckDB, Spark, etc.), and multiple registry backends. The Go feature server provides an alternative high-performance serving path with HTTP and gRPC APIs, supporting Redis, PostgreSQL, DynamoDB, and SQLite online stores.

## Architectural Analysis

The Feast component follows a **multi-language, operator-managed architecture** where the Go operator serves as the control plane and the Python feature server serves as the data plane. This separation allows the operator to handle complex lifecycle management (TLS, RBAC, scaling, materialization scheduling) while the feature server focuses on high-performance feature serving. The operator deploys all Feast services within a single multi-container Deployment per FeatureStore CR, which simplifies networking (services share localhost) but limits independent scaling of individual service types.

The **authorization architecture** is notably sophisticated with two distinct layers. At the platform level, the operator manages Kubernetes RBAC resources (Roles, RoleBindings, ClusterRoles) and integrates with OIDC providers. At the application level, the Python feature server implements its own permission-based RBAC system with fail-secure defaults (no permissions defined = deny all). The operator periodically reconciles these layers by fetching permission policies from the registry REST API using an intra-communication JWT token (HMAC-SHA256 signed, empty key), then translating Feast permission policies into Kubernetes RBAC bindings for auto-access. This polling-based approach (30-second reconciliation interval) introduces a window where RBAC changes lag behind permission policy updates.

The **TLS configuration** leverages OpenShift service serving certificates when available, auto-detected at reconciliation time. The operator annotates Services for certificate injection and creates a CA bundle ConfigMap. For the operator-to-registry internal communication path, TLS verification is intentionally skipped (`InsecureSkipVerify=true`) since the traffic is cluster-internal and the certificates are auto-provisioned. The Go feature server explicitly sets `MinVersion: tls.VersionTLS12` for HTTPS mode.

The **ODH Notebook integration** (NotebookConfigMapReconciler) is a secondary controller that watches for Notebook CRDs with the `opendatahub.io/feast-integration: "true"` label. It dynamically creates ConfigMaps containing Feast client configuration for notebooks, using owner references for automatic cleanup. The controller polls for Notebook CRD availability at startup (5-second interval) and gracefully handles the CRD being absent, which allows the operator to function on non-ODH clusters.

The **scaling architecture** supports both static replicas and HPA-based autoscaling, with validation rules ensuring that scaling requires database-backed persistence (not file-based SQLite) for both online and offline stores, plus a database-backed or remote registry. When scaling is enabled, the operator automatically injects soft topology spread constraints (zone-based) and pod anti-affinity to improve availability, unless the user provides custom constraints.
