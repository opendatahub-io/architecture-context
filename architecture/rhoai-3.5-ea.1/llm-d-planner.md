## Purpose

**Short**: llm-d Planner is an LLM deployment planning platform that guides users from business requirements to running llm-d deployments through GPU sizing, performance estimation, and Kubernetes configuration generation.

**Detailed**: llm-d Planner unifies three capabilities into a single platform: a conversational Planner workflow that converts natural-language business requirements into SLO-driven deployment recommendations, a Capacity Planner that estimates GPU memory requirements (model weights, KV cache, activation memory) for any HuggingFace model, and a GPU Recommender that estimates inference performance (TTFT, ITL, throughput) across GPU types using roofline modeling without requiring actual benchmarks.

The platform consists of a FastAPI backend that exposes REST APIs for intent extraction (via Ollama LLM), benchmark querying (via PostgreSQL), capacity planning, GPU estimation, multi-criteria recommendation scoring, and Kubernetes deployment (via KServe InferenceService). A Streamlit frontend provides a guided multi-step UI for requirements gathering, SLO configuration, recommendation review, and one-click deployment. An optional vLLM simulator enables GPU-free local development. For RHOAI integration, the backend can optionally sync benchmark data from the RHOAI Model Catalog API.

The deployment workflow generates production-ready KServe/vLLM YAML manifests (InferenceService, HPA, ServiceMonitor with Grafana dashboards) and can deploy directly to Kubernetes clusters. The recommendation engine scores configurations on four criteria (accuracy, price, latency, complexity) using benchmark data from PostgreSQL and roofline performance estimation from the BentoML llm-optimizer library.

## Architectural Analysis

**Multi-tier service architecture with LLM-augmented planning.** The llm-d Planner follows a classic three-tier pattern (UI → API → database) augmented with an LLM service for natural-language understanding. The Streamlit UI communicates exclusively with the FastAPI backend over HTTP, and the backend orchestrates all external integrations (Ollama, PostgreSQL, HuggingFace, Kubernetes API, Model Catalog). This clean separation means the UI has no direct dependencies on infrastructure services. The backend uses FastAPI's dependency injection pattern with lazily-initialized singletons for database connections, Kubernetes client managers (cached per-namespace, max 32), and the recommendation workflow pipeline.

**Recommendation engine with dual data sources.** The recommendation pipeline operates in two modes: benchmark-driven (querying PostgreSQL for real GuideLLM performance data) and estimation-driven (using BentoML's roofline model for models/GPUs without benchmarks). This dual approach is a pragmatic design — real benchmarks are preferred but expensive to produce, so roofline estimates fill gaps and expand the recommendation surface. Results from both sources are scored identically across four criteria (accuracy from benchmark CSVs, price from GPU catalog, latency from benchmark/estimate, complexity from GPU count) and ranked into five independent lists (best accuracy, lowest cost, lowest latency, simplest, balanced). The scorer uses non-linear formulas (e.g., cost scoring uses a power-0.7 curve) and task-specific bonuses (e.g., +20 for code models in code_completion use cases).

**YAML generation via Jinja2 templates targets KServe v1beta1.** Generated deployment manifests include three resources: a KServe InferenceService (with vLLM runtime, GPU requests, and model download from HuggingFace), an HPA (with queue-depth-based and CPU-based metrics, plus commented KEDA ScaledObject alternative), and a ServiceMonitor with an embedded Grafana dashboard ConfigMap. The templates support two modes: production (real vLLM with GPU scheduling via node selectors and tolerations) and simulator (vLLM-simulator image with no GPU requirements). The deployment ID is constrained to 44 characters to fit KServe naming limits.

**Security posture is minimal and development-oriented.** All 29 backend API endpoints are unauthenticated — there is no API authentication or authorization mechanism. CORS is set to `*` in the Kubernetes deployment. The only RBAC rule is a ClusterRole granting `nodes.list` for GPU discovery. The backend pod runs as non-root with security context restrictions (no privilege escalation, all capabilities dropped, RuntimeDefault seccomp). Secrets are stored as Opaque Kubernetes Secrets with placeholder values. The OpenShift Routes provide edge TLS termination, but internal service-to-service communication (UI→backend, backend→Ollama, backend→PostgreSQL) is plaintext HTTP. This security posture is consistent with an internal development/planning tool but would need hardening for multi-tenant production deployment.

**Optional RHOAI integration through Model Catalog sync.** When `PLANNER_BENCHMARK_SOURCE=model_catalog`, the backend starts a background thread that periodically fetches benchmark data from the RHOAI Model Catalog API over HTTPS (port 8443) and upserts it into PostgreSQL. This integration uses OpenShift's service CA infrastructure — the service CA certificate is injected via ConfigMap annotation and mounted into the backend pod. A cross-namespace NetworkPolicy in the `rhoai-model-registries` namespace explicitly allows ingress from the planner backend pod on port 8443. Authentication uses either a bearer token from the Kubernetes Secret or the ServiceAccount token mounted in the pod.
