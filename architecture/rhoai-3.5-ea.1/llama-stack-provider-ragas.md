## Purpose

**Short**: Implements Ragas evaluation metrics as an out-of-tree Llama Stack provider, enabling LLM/RAG evaluation via the Llama Stack Eval API with both inline (in-process) and remote (Kubeflow Pipelines) execution modes.

**Detailed**: This component bridges the [Ragas](https://github.com/explodinggradients/ragas) evaluation framework with the [Llama Stack](https://github.com/meta-llama/llama-stack) platform by implementing the Llama Stack `Eval` API as an external provider. It exposes RAG evaluation metrics — answer relevancy, context precision, faithfulness, and context recall — through the standard Llama Stack evaluation interface.

The provider ships in two modes. The **inline provider** runs Ragas evaluations in the same process as the Llama Stack server, wrapping the Llama Stack Inference API as LangChain-compatible LLM and embedding interfaces that Ragas expects. The **remote provider** submits evaluation jobs as Kubeflow Pipelines (KFP) workflows, where each pipeline step runs in a container on OpenShift, retrieves datasets from Llama Stack, executes Ragas evaluation, and writes results to S3. The remote provider uses a separate set of LLM/embedding wrappers based on the `llama-stack-client` SDK for use inside KFP containers.

Within RHOAI, the component is deployed via the Llama Stack Operator as a `LlamaStackDistribution` custom resource. The container image is built from `Dockerfile.konflux` using a UBI9 Python 3.12 base. The provider is registered as `inline::trustyai_ragas` and `remote::trustyai_ragas` in the Llama Stack distribution configuration. The TrustyAI service operator manages the base image reference for KFP pipeline components via a ConfigMap (`trustyai-service-operator-config`).

## Architectural Analysis

The llama-stack-provider-ragas component follows the Llama Stack external provider pattern, implementing the `Eval` and `BenchmarksProtocolPrivate` interfaces. The architecture is cleanly split into two execution modes that share a common configuration layer (`config.py`, `constants.py`, `compat.py`) but diverge in how they invoke the Ragas evaluation framework.

The **inline provider** uses an adapter pattern where `LlamaStackInlineLLM` and `LlamaStackInlineEmbeddings` wrap the Llama Stack Inference API (which itself wraps the actual LLM backend — Ollama, vLLM, etc.) into LangChain-compatible interfaces that Ragas expects. This creates a three-layer abstraction: Ragas → LangChain interfaces → Llama Stack Inference API → actual LLM backend. The inline provider runs evaluations as async tasks (`asyncio.create_task`) and tracks them in an in-memory dictionary keyed by job ID. This means job state is lost on server restart — there is no durable state backend for evaluation jobs.

The **remote provider** takes a fundamentally different approach: instead of wrapping the Inference API in-process, it submits a KFP pipeline with two components. The first component (`retrieve_data_from_llama_stack`) uses the `llama-stack-client` SDK to fetch the dataset from the Llama Stack server. The second component (`run_ragas_evaluation`) uses `LlamaStackRemoteLLM` and `LlamaStackRemoteEmbeddings` — a separate set of wrappers that use the `llama-stack-client` SDK directly rather than the in-process Inference API. This is necessary because KFP pipeline components run in separate containers. Results are written to S3 and later fetched by the provider when `job_result()` is called. The base image for KFP containers is resolved through a three-tier fallback: environment variable → Kubernetes ConfigMap (`trustyai-service-operator-config`) → hardcoded default.

The **compatibility layer** (`compat.py`) is architecturally notable — it provides backward compatibility between the legacy `llama_stack` package layout and the newer `llama_stack_api` package structure, using try/except import fallbacks for every API type. This suggests the component must support multiple versions of the Llama Stack SDK simultaneously, likely to bridge upstream development cadence with downstream RHOAI release cycles.

A notable security consideration is the KFP authentication model. The remote provider requires a bearer token for the Kubeflow Pipelines API, obtained either from an explicit environment variable (`KUBEFLOW_PIPELINES_TOKEN`) or by reading the local kubeconfig. The demo documentation notes that "the Llama Stack distribution service account does not have privileges to create pipeline runs," requiring a user-provisioned token — this is a gap in the RBAC model that could be addressed by granting the appropriate ClusterRole to the Llama Stack service account. The S3 credentials are mounted into KFP containers via `kubernetes.use_secret_as_env()`, which injects them as environment variables — standard KFP practice but worth noting for secret lifecycle management.
