## Purpose

**Short**: An out-of-tree Llama Stack evaluation provider that integrates Ragas (Retrieval-Augmented Generation Assessment) metrics into the Llama Stack eval API, supporting both inline (in-process) and remote (Kubeflow Pipelines) execution modes.

**Detailed**: This component implements the Ragas evaluation framework as an external provider for Meta's Llama Stack platform. It enables users to evaluate RAG (Retrieval-Augmented Generation) systems using Ragas metrics (answer relevancy, context precision, faithfulness, context recall) through Llama Stack's standard eval API (`client.alpha.eval.run_eval`). The provider wraps Ragas' LLM and embedding interfaces to delegate inference calls back to the Llama Stack server itself, creating a closed evaluation loop.

The provider ships in two execution modes. The **inline provider** (`inline::trustyai_ragas`) runs Ragas evaluation in the same process as the Llama Stack server, wrapping the server's own inference API for LLM and embedding calls. The **remote provider** (`remote::trustyai_ragas`) offloads evaluation to Kubeflow Pipelines (Data Science Pipelines), constructing a two-step KFP pipeline: (1) retrieve dataset from Llama Stack, (2) run Ragas evaluation using a dedicated container image. Results are persisted to S3 and fetched back on job completion. This remote mode is designed for production use on OpenShift AI / RHOAI clusters where Data Science Pipelines is available.

The component is packaged as a Python library (PyPI: `llama-stack-provider-ragas`) and as a container image (`odh-trustyai-ragas-lls-provider-dsp-rhel9`) built via Konflux. The container image is used as the base image for Kubeflow Pipeline components in remote evaluation mode. The provider is part of the TrustyAI ecosystem within RHOAI.

## Architectural Analysis

The llama-stack-provider-ragas component demonstrates a clean provider abstraction pattern within the Llama Stack ecosystem. Both inline and remote providers implement the same `Eval` and `BenchmarksProtocolPrivate` interfaces, meaning consumers interact identically regardless of execution mode. The mode selection is driven entirely by configuration — if `KUBEFLOW_LLAMA_STACK_URL` is set, the remote provider activates; otherwise, only the inline provider is available. This is handled via conditional provider spec registration in `provider.py`, with a runtime dependency check on `kfp`, `kubernetes`, and `s3fs` imports.

The wrapper layer (`wrappers_inline.py` and `wrappers_remote.py`) is the architectural bridge between Ragas and Llama Stack. Ragas expects LangChain-compatible `BaseRagasLLM` and `BaseRagasEmbeddings` interfaces, so the wrappers translate between Llama Stack's OpenAI-compatible completion/embedding APIs and LangChain's `LLMResult`/`Generation` types. The inline wrappers use the in-process Llama Stack inference API (async), while the remote wrappers use `LlamaStackClient`/`AsyncLlamaStackClient` HTTP clients to call a remote Llama Stack server from within KFP pipeline containers. Both are hardcoded to `RunConfig(max_workers=1)`, serializing all Ragas evaluation steps.

The Kubeflow Pipelines integration in the remote provider is notable for its base image resolution strategy. The `get_base_image()` function in `components.py` follows a three-tier fallback: (1) `KUBEFLOW_BASE_IMAGE` environment variable, (2) `trustyai-service-operator-config` ConfigMap in `redhat-ods-applications` or `opendatahub` namespaces, (3) default public image `quay.io/trustyai/llama-stack-provider-ragas:latest`. This design allows the TrustyAI operator to control which image version is used for evaluation pipelines without requiring provider reconfiguration, which is important for air-gapped or controlled environments.

A compatibility layer (`compat.py`) handles the migration between `llama_stack` and `llama_stack_api` package namespaces, using try/except import fallbacks. This supports both legacy Llama Stack layouts and the newer `llama_stack_api` module structure, indicating the upstream API is actively evolving. The `config.py` uses Pydantic models with `SecretStr` for the KFP token, ensuring it is redacted in logs.

One architectural limitation is that the remote provider's job tracking is in-memory (`self.evaluation_jobs` dict). If the Llama Stack server restarts, all in-progress job references are lost, though the actual KFP pipeline runs continue on the cluster. Results can still be retrieved from S3 if the job ID and S3 prefix are known, but the provider has no mechanism to recover orphaned jobs. The inline provider has the same limitation — evaluation jobs are tracked in an in-memory dict with auto-incrementing integer IDs.
