## Purpose

**Short**: A centralized library of reusable Kubeflow Pipelines (KFP) components and pre-built pipelines for AI/ML workflows on Red Hat OpenShift AI, covering data processing, training, evaluation, and deployment.

**Detailed**: The pipelines-components repository provides a structured, quality-controlled collection of KFP 2.x components and pipelines that run on the RHOAI platform. Components are atomic, reusable building blocks implemented as Python functions decorated with `@dsl.component`, while pipelines compose these components into end-to-end ML workflows using `@dsl.pipeline`. The repository spans four functional categories: data processing (S3 data loading, document text extraction, synthetic data generation), training (AutoML tabular/timeseries via AutoGluon, AutoRAG optimization via ai4rag, LLM fine-tuning via LoRA/OSFT/SFT), evaluation (LLM benchmarking via lm-eval), and deployment (model registry registration, Llama Stack request generation).

The repository also produces three container images via Konflux: (1) `odh-pipelines-components` — an init container that compiles managed pipelines and copies compiled YAML into a shared volume consumed by the KFP API server, (2) `odh-automl` — a runtime image pre-loaded with AutoGluon and its dependencies for tabular/timeseries ML pipelines, and (3) `odh-autorag` — a runtime image pre-loaded with docling, ai4rag, and Llama Stack client libraries for RAG optimization pipelines. A comprehensive `scripts/` directory provides tooling for pipeline compilation, metadata validation, base image governance, and CI enforcement.

## Architectural Analysis

The pipelines-components repository follows a cleanly layered architecture separating reusable components (`components/`) from composed pipelines (`pipelines/`), with a sophisticated tooling layer (`scripts/`) that enforces quality gates. Each component is a self-contained Python function decorated with `@dsl.component` that declares its own base image, dependencies, and I/O signature. Pipelines compose these components using `@dsl.pipeline` with explicit data flow via KFP artifacts and parameters. This separation enables independent component testing and versioning while allowing pipelines to be assembled from well-tested building blocks.

The runtime image strategy uses a two-tier approach: heavyweight pre-built images (`odh-automl`, `odh-autorag`) carry domain-specific ML frameworks that are too large to install at pipeline runtime, while lighter components use generic UBI9/Python base images and install dependencies inline via `packages_to_install` in the `@dsl.component` decorator. The AutoML and AutoRAG images pin to Red Hat-curated PyPI indices (`console.redhat.com/api/pypi/public-rhai/`), ensuring supply chain control. The environment variable override pattern (`RELATED_IMAGE_MPI_AUTOML_RUNTIME` / `RELATED_IMAGE_MPI_AUTORAG_RUNTIME`) in `utils/consts.py` follows the operator SDK disconnected environment pattern, allowing the platform operator to override base images at deployment time.

The managed pipeline initialization mechanism is architecturally notable. The `generate_managed_pipelines` script discovers all pipelines with `managed: true` in their `metadata.yaml`, compiles them to KFP YAML via the KFP compiler, and generates a `managed-pipelines.json` manifest. The `init_managed_pipelines` script then runs as an init container, copying compiled YAMLs into a shared volume (`/config/managed-pipelines/`) that the KFP API server reads on startup. This pattern ensures pipeline definitions are baked into the container image at build time — no runtime compilation needed — while remaining versioned alongside source code. The path-traversal check (`is_relative_to(APP_ROOT)`) in `init_managed_pipelines.py` prevents manifest injection attacks.

Fine-tuning pipelines demonstrate three distinct training architectures. LoRA uses the unsloth backend (single-node only, QLoRA 4/8-bit quantization), SFT uses instructlab-training with FSDP multi-GPU sharding (typically 4 workers x 1 GPU to avoid `/dev/shm` limitations), and OSFT uses mini-trainer for orthogonal subspace fine-tuning that preserves existing model capabilities. All three follow the same 4-stage pattern (dataset download -> train -> evaluate -> register) and share common utilities in `components/training/finetuning/shared/`. The "minimal" pipeline variants expose fewer parameters with sensible defaults for quick experimentation.

The CI/validation infrastructure is unusually thorough for a component library. An AST-based import guard (`.github/scripts/check_imports/`) prevents components from importing heavy dependencies at module load time (which would break KFP compilation). Base image governance uses an allowlist (`base_image_allowlist.yaml`) validated via regex patterns. Metadata schema validation ensures every component has standardized documentation. The compile check verifies every component and pipeline can be successfully compiled to KFP YAML. These checks run as GitHub Actions on PRs targeting `main` and as Tekton PipelineRuns via Konflux for RHOAI release branches.
