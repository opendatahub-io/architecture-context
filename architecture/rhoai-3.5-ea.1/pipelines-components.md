## Purpose

**Short**: A collection of reusable Kubeflow Pipelines (KFP) components and pipelines for AI/ML workflows including LLM fine-tuning (LoRA, OSFT, SFT), AutoML (tabular/time-series), AutoRAG optimization, synthetic data generation, and model evaluation—shipped as a Python package and container images for RHOAI managed pipelines.

**Detailed**: The pipelines-components repository is the centralized hub for reusable Kubeflow Pipelines components and end-to-end pipelines within the Red Hat OpenShift AI (RHOAI) platform. It organizes AI workflow building blocks into four categories: Data Processing (dataset download, synthetic data generation, document discovery/extraction/indexing), Training (LoRA, OSFT, SFT fine-tuning; AutoGluon tabular and time-series AutoML; AutoRAG optimization), Evaluation (lm-eval-based benchmark and custom holdout evaluation), and Deployment (Kubeflow Model Registry registration, OGX Responses API request body generation).

The repository produces three container images via Konflux: `odh-automl` (AutoGluon-based tabular/time-series training runtime), `odh-autorag` (RAG optimization and document processing runtime with Docling models), and `odh-pipelines-components` (init container that compiles and stages managed pipeline YAMLs into the KFP API server's shared volume). Components are implemented as `@dsl.component` Python functions using the KFP SDK v2 (≥2.15.2), with shared utility modules for dataset resolution, Kubernetes job management, model persistence, and metrics extraction. Pipelines follow a consistent 4-phase architecture (Data → Training → Evaluation → Registry) with Kubernetes secret injection, PVC workspace management, and GPU/accelerator configuration.

## Architectural Analysis

The pipelines-components repository follows a layered library architecture rather than a service architecture. All components are defined as `@dsl.component` Python functions using KFP SDK v2, which means they run as ephemeral containers inside KFP pipeline pods—not as long-running services. This design gives the repository an unusually flat deployment model: it produces container images consumed by KFP, but exposes no APIs, CRDs, or persistent services of its own.

**Component vs. Pipeline Separation**: The repository enforces a clean separation between reusable components (`components/` directory) and pipeline compositions (`pipelines/` directory). Components define individual tasks (e.g., "download a dataset from S3", "train with LoRA"), while pipelines wire components together with PVC configuration, secret injection, and resource allocation. This pattern enables pipeline variants (standard vs. minimal) that share the same underlying components but differ in parameter exposure and defaults. The minimal pipeline variants (lora_minimal, osft_minimal, sft_minimal) hardcode sensible defaults for ~80% of parameters, reducing the pipeline surface from ~112 to ~18 parameters.

**Three-Image Architecture**: The repository produces three Konflux container images, each serving a distinct architectural role. `odh-automl` and `odh-autorag` are pre-built runtime images with heavyweight ML dependencies (AutoGluon, Docling, ai4rag) that would be impractical to install at task startup via `packages_to_install`. These images are referenced via environment-overridable constants (`RELATED_IMAGE_ODH_AUTOML_IMAGE`, `RELATED_IMAGE_ODH_AUTORAG_IMAGE`) enabling OLM-based image pinning for air-gapped deployments. The third image, `odh-pipelines-components`, is an init container that bridges the gap between the repository and the KFP API server: it compiles pipeline Python files to KFP YAML and stages them at a well-known filesystem path.

**Managed Pipeline Lifecycle**: The managed pipeline mechanism is a noteworthy design pattern. Pipelines opt into management via `managed: true` in their `metadata.yaml`. The `generate_managed_pipelines.py` script uses AST-based static analysis (not dynamic import) to extract pipeline descriptions, avoiding the need to install all pipeline dependencies during compilation. The init container (`init_managed_pipelines.py`) validates paths with `is_relative_to()` to prevent path traversal attacks before copying YAMLs to the KFP shared volume. This lifecycle—develop → validate → compile → stage → discover—is fully automated in the Konflux build pipeline.

**Training Backend Diversity**: The fine-tuning pipelines reveal an interesting backend strategy. LoRA uses the Unsloth backend (single-node only, optimized for parameter-efficient fine-tuning), OSFT uses the mini-trainer backend (multi-node capable, continual learning), and SFT uses the instructlab-training backend (multi-node FSDP, full supervised fine-tuning). All three share common infrastructure via the `shared/` modules (data.py, setup.py, training.py, output.py), but each delegates actual training to different ClusterTrainingRuntime backends. The `wait_for_training_job()` function implements a robust polling pattern with log streaming, retry logic, and timeout management.

**Security Observations**: The Model Registry integration uses insecure HTTP (`is_secure=False`) by default, which is a notable gap in an otherwise TLS-aware codebase. The AutoRAG components implement SSL verification fallback (`verify=False`) for development convenience with self-signed certificates, but this pattern persists in production code without clear gating. The OGX integration provides proper TLS configuration with `REQUESTS_CA_BUNDLE`/`SSL_CERT_FILE` support and an explicit `OGX_TLS_INSECURE` flag for development environments. Kubernetes authentication uses bearer tokens from environment variables rather than in-cluster service account mounting, providing flexibility but requiring explicit secret management.
