## Purpose

**Short**: A Go-based CLI tool and data container that extracts, enriches, and catalogs metadata for Red Hat AI models and MCP (Model Context Protocol) servers from OCI container registries and HuggingFace.

**Detailed**: The model-metadata-collection component serves two distinct roles in the RHOAI platform. First, it provides a **build-time CLI tool** (`model-extractor`) that processes OCI container images from Red Hat's container registry, extracts model cards from container layers, enriches the extracted metadata with data from HuggingFace (API, YAML frontmatter, README parsing), generates standardized YAML catalogs for AI models and MCP servers, and produces metadata completeness reports. Second, it produces a **data-only container image** (`Dockerfile.konflux`) that packages the pre-generated catalog YAML files and sample benchmark data as static files. This container image is designed to be consumed by other platform components (such as the RHOAI dashboard) via volume mounts or init containers, providing a curated catalog of validated AI models, partner/community MCP servers, and their associated metadata.

The tool supports three categories of models (Red Hat, validated, other) and three tiers of MCP servers (Red Hat, partner, community). Metadata enrichment uses a multi-source strategy with priority ordering: HuggingFace YAML frontmatter > HuggingFace API > HuggingFace tags > modelcard YAML frontmatter > modelcard regex extraction > generated fallbacks. The enrichment pipeline also adds OCI artifact metadata (architectures, timestamps), vLLM recommended configurations, and tool-calling documentation to model READMEs.

## Architectural Analysis

The model-metadata-collection component follows a **batch processing pipeline** architecture rather than a service-oriented one. The Go CLI tool runs during CI/CD to scrape, enrich, and assemble metadata catalogs, which are then committed to the repository as static YAML files. The Konflux Dockerfile simply packages these pre-computed files into a minimal container image. This "pre-baked data" pattern is architecturally significant: it means the deployed container has zero runtime dependencies on external services (no HuggingFace API calls, no registry access at runtime), which improves reliability and startup time but introduces a coupling between the catalog freshness and the CI/CD pipeline execution frequency.

The metadata enrichment pipeline demonstrates a sophisticated **multi-source merge strategy** with explicit priority ordering. Each metadata field (name, provider, description, license, etc.) is tracked with its source (`modelcard.yaml`, `modelcard.regex`, `huggingface.yaml`, `huggingface.api`, `huggingface.tags`, `huggingface.regex`, `generated`), enabling full provenance tracking. The enrichment YAML files written alongside each model's metadata provide an audit trail of where each piece of data originated. This design decision enables the metadata-report tool to generate field-level completeness and source distribution reports, which is valuable for catalog quality assurance.

The model family matching system in `internal/config/model_families.go` is designed as a centralized registry of known model families (llama, granite, mistral, qwen, etc.) that prevents cross-family matching during enrichment. This is critical because model names from different families can share tokens (e.g., a granite-8b container shouldn't match a llama-8b HuggingFace entry). The matching algorithm uses token-level similarity scoring with substring boosting, and the model family list must be manually maintained as new model architectures are added. The version normalization logic (converting "3.1" to "3v1" and "granite-3-3" to "granite-3v3") is particularly intricate and ensures that version sub-components are not confused with other numeric tokens during matching.

The MCP server catalog pipeline was added more recently (visible in the commit history) and follows the same enrichment pattern as models but with OCI-level enrichment for architectures and timestamps rather than HuggingFace enrichment. MCP servers are categorized into three tiers (Red Hat, partner, community), each with separate index and catalog files, supporting the platform's tiered support model. The input YAML files for MCP servers include detailed tool definitions with parameters, deployment modes, and security indicators — a richer schema than the model catalog.

The build system uses two different Dockerfiles: `Dockerfile` (used for upstream ODH builds via GitHub Actions, targeting `quay.io/opendatahub`) and `Dockerfile.konflux` (used for downstream RHOAI builds via Tekton/Konflux). The Konflux Dockerfile uses `ubi9-minimal` as the base image and explicitly copies each catalog file individually rather than using wildcards, which provides better build cache behavior and explicit dependency tracking. Both Dockerfiles run the container as a non-root user (UID 1001) with `sleep infinity` as the default command, confirming the data-container pattern where the image exists solely to provide files via volume mounts.
