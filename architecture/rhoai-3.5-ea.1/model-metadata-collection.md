## Purpose

**Short**: A Go CLI tool that extracts, enriches, and catalogs metadata from Red Hat AI container images and MCP servers, producing static YAML catalogs shipped as a data-only container image.

**Detailed**: The model-metadata-collection component serves as the metadata pipeline for the RHOAI AI model catalog. It discovers AI models from HuggingFace collections (specifically Red Hat AI validated model collections), extracts model cards from OCI container image layers (modelcar format), enriches metadata from multiple sources (HuggingFace API, YAML frontmatter, container registry manifests), and produces structured YAML catalog files consumed by downstream RHOAI components (e.g., the dashboard model catalog UI).

The tool also processes MCP (Model Context Protocol) server metadata, reading structured YAML input files for Red Hat, partner, and community MCP servers, enriching them with OCI registry data (architectures, timestamps), and generating aggregated catalog files. Additionally, it generates metadata completeness reports analyzing field coverage and data source provenance.

The final container image is a minimal data-only container (UBI9-minimal base, no Go binary) that ships pre-generated YAML catalog files at `/app/data/`. This image is consumed by other RHOAI components via init containers or volume mounts to access model and MCP server metadata. The Go tool itself runs during CI/CD to regenerate the catalog data before the container image is built.

## Architectural Analysis

The model-metadata-collection component implements a multi-stage metadata pipeline with a clear data priority hierarchy. Metadata flows through four priority levels: (1) HuggingFace YAML frontmatter (highest), (2) container modelcard.md content, (3) HuggingFace API data, and (4) registry metadata and generated defaults (lowest). This design ensures that curated, structured data always takes precedence over inferred or auto-generated content. The priority logic is implemented in `internal/enrichment/enrichment.go` and `internal/enrichment/update.go`, where each metadata field tracks its source provenance for auditability.

The architecture uses a data-only container pattern, which is notable. The Go CLI tools (model-extractor, metadata-report) are build-time tools that run during CI/CD, not runtime services. The shipped container image contains only static YAML files on a minimal UBI9 base, running `sleep infinity`. This design eliminates runtime dependencies, attack surface, and resource consumption. Other RHOAI components access the catalog data by mounting volumes from this container or using init containers to copy files. The Konflux Dockerfile (`Dockerfile.konflux`) and upstream Dockerfile (`Dockerfile`) differ: Konflux copies `data/*.yaml` with wildcards, while the upstream Dockerfile explicitly names each file and also includes `sample-data/` as benchmarks.

The concurrent model processing system (`processModelsInParallelWithMetadata` in `cmd/model-extractor/main.go`) uses Go's goroutine-based parallelism with a configurable semaphore (default 5 concurrent jobs) to process OCI image layers efficiently. Each model's metadata is extracted independently, then enriched in a second pass with HuggingFace data. The similarity matching algorithm (`pkg/utils/text.go:CalculateSimilarity`) uses token-based comparison with model family validation (`internal/config/model_families.go`) to prevent cross-family mismatches (e.g., a Llama container being matched to a Granite HuggingFace entry).

The MCP server catalog pipeline is a newer addition (evidenced by recent commits). It follows a similar pattern: an index file references individual server YAML definitions, each server is optionally enriched with OCI registry data (architectures and timestamps), and the result is aggregated into a catalog YAML. Three tiers of MCP servers are supported: Red Hat, partner, and community, each with separate index and catalog files. The enrichment stage uses exponential backoff retry logic (`pkg/utils/retry.go`) with configurable timeouts for resilient registry interactions.

A notable design decision is the tool-calling and vLLM configuration enrichment, which appends structured documentation sections to model README content. This enrichment uses Go templates (`pkg/utils/templates/*.tmpl`) embedded in the binary via `embed.FS`, and includes validation to prevent duplicate sections on re-enrichment runs. The vLLM recommended configurations are loaded from static YAML files in `input/models/vllm-config/` and matched to models by exact HuggingFace model name lookup.
