## Purpose

**Short**: Payload processing plugins for the AI Gateway that enable request/response mutation, API translation, authentication injection, and content moderation for external LLM providers.

**Detailed**: AI Gateway Payload Processing implements a set of Body-Based Routing (BBR) plugins built on the Kubernetes Gateway API Inference Extension framework. It runs as an external processing (ext-proc) service attached to an Envoy-based Gateway, intercepting inference requests and responses to perform multi-stage transformations.

The service bridges the gap between a unified OpenAI-compatible client API and diverse external LLM provider APIs (OpenAI, Anthropic, Azure OpenAI, AWS Bedrock, Google Vertex AI). It resolves client-facing model names to provider-specific models via ExternalModel and ExternalProvider CRDs, translates request/response payloads between API formats, injects provider-specific authentication credentials from Kubernetes Secrets, and optionally enforces content moderation through NVIDIA NeMo Guardrails integration.

The component also includes two Kubernetes controllers that reconcile ExternalModel and ExternalProvider custom resources into Gateway API HTTPRoutes, Kubernetes Services (ExternalName), Istio ServiceEntries, and Istio DestinationRules — dynamically constructing the networking infrastructure needed to route inference traffic to external providers with TLS origination.

## Architectural Analysis

The AI Gateway Payload Processing component implements a clean plugin-based architecture built on top of the upstream Kubernetes Gateway API Inference Extension's Body-Based Routing (BBR) framework. The five plugins (model-provider-resolver, api-translation, apikey-injection, nemo-request-guard, nemo-response-guard) execute in a well-defined pipeline order, communicating through a shared CycleState that carries per-request context (provider, model, credential references) between stages. This design avoids tight coupling between plugins while maintaining a clear data flow.

The component straddles two architectural roles: it is both a runtime request processor (ext-proc service) and a Kubernetes controller plane. The ExternalModel and ExternalProvider controllers dynamically construct the networking infrastructure (HTTPRoutes, ExternalName Services, Istio ServiceEntries, DestinationRules) that the Gateway needs to route traffic to external providers. This means the component is not just a passive plugin — it actively manages the Gateway API routing topology. The controllers use owner references to ensure cleanup of managed resources when CRs are deleted.

A notable design decision is the dual ExternalModel schema: the controllers reconcile `inference.opendatahub.io/v1alpha1/ExternalModel`, while the model-provider-resolver plugin watches `maas.opendatahub.io/v1alpha1/ExternalModel` via unstructured client. This suggests the component is bridging a transition between the older MaaS (Models-as-a-Service) API group and the newer inference API group. The unstructured watch for the maas CRD is a pragmatic choice that avoids importing the maas type package.

The API translation layer is comprehensive, covering five provider API formats with bidirectional translation. The Anthropic translator is by far the most complex, handling system prompt extraction, tool call conversion, content block transformation, and usage token mapping. The Azure OpenAI and Bedrock translators are thin wrappers that rely on those providers' OpenAI-compatible endpoints, primarily handling path rewriting and stripping provider-specific response fields.

Security-wise, the credential management approach — watching labeled Secrets and caching them in thread-safe in-memory stores — is effective for the ext-proc use case where every request needs credential lookup. The label-based filtering (`inference.networking.k8s.io/bbr-managed=true`) limits the blast radius of the Secret watch. However, the NeMo Guardrails integration uses a plain HTTP client with no TLS configuration, which means the connection to the guardrails service relies entirely on network-level security (e.g., Istio mTLS) rather than application-level encryption.
