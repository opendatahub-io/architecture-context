## Purpose

**Short**: Envoy Body Based Routing (BBR) plugin service that intercepts AI Gateway traffic to perform model-to-provider resolution, API format translation, API key injection, and NeMo guardrail enforcement for external LLM providers.

**Detailed**: AI Gateway Payload Processing is a Go service that runs as an Envoy External Processing (ext_proc) server, connected to an AI Gateway via the Kubernetes Inference Gateway's pluggable BBR (Body Based Routing) framework. It processes incoming inference requests by resolving model names to external provider configurations, translating between OpenAI Chat Completions format and provider-native formats (Anthropic, Azure OpenAI, AWS Bedrock, Google Vertex AI), injecting provider-specific API keys from Kubernetes Secrets, and optionally enforcing content guardrails via NVIDIA NeMo Guardrails.

The service watches `ExternalModel` CRDs (from the `maas.opendatahub.io` API group) and labeled Kubernetes Secrets via controller-runtime reconcilers to maintain an in-memory cache of model-to-provider mappings and API credentials. At request time, it chains four primary plugins — model-provider-resolver, api-translation, apikey-injection, and nemo guards — that share per-request state through the BBR framework's CycleState mechanism. The service is deployed via a Helm chart that wraps the upstream `body-based-routing` chart and installs an Istio EnvoyFilter to attach the ext_proc to the AI Gateway's Envoy proxy.

This component is part of the RHOAI Model-as-a-Service (MaaS) stack, enabling users to define `ExternalModel` CRs that map client-facing model names to external provider endpoints (OpenAI, Anthropic, Azure, AWS Bedrock, Vertex AI), with automatic API translation and credential management handled transparently by the gateway.

## Architectural Analysis

The AI Gateway Payload Processing service is architecturally notable as a **data-plane extension** rather than a traditional Kubernetes controller or standalone API server. It runs as an Envoy External Processing (ext_proc) service attached to the AI Gateway's Envoy proxy via Istio EnvoyFilter. This design means the service sits in the hot path of every inference request, but only activates its plugin chain when it detects an ExternalModel by matching the request path to its in-memory model store. Internal model requests (those not matching an ExternalModel CR) pass through untouched, with no overhead beyond a CycleState lookup.

The **plugin chain architecture** follows a clear pipeline: `body-field-to-header` (upstream, extracts model name to header for routing) → `model-provider-resolver` (resolves model to provider info via CycleState) → `api-translation` (mutates request body and headers for the target provider) → `apikey-injection` (injects auth headers from cached Secrets). The CycleState mechanism enables loose coupling between plugins — the resolver writes provider, model, and credential references that downstream plugins consume without direct dependencies. The nemo guard plugins (`nemo-request-guard`, `nemo-response-guard`) are optional additions that can be inserted into the chain to enforce content safety rails via an external NeMo Guardrails service.

The **API translation layer** is the most complex part of the codebase. The Anthropic translator (`pkg/plugins/api-translation/translator/anthropic/`) performs full bidirectional translation between OpenAI Chat Completions and Anthropic Messages API, including system message extraction, tool use/tool_calls conversion, image content block translation, stop reason mapping, and usage metric normalization. The Vertex AI translator similarly handles native GenerateContent format conversion with function calling support. In contrast, the OpenAI, Azure OpenAI, Bedrock, and Vertex OpenAI translators are primarily path rewriters since those providers accept OpenAI-compatible request formats. Azure OpenAI and Vertex OpenAI also strip provider-specific response fields (e.g., `prompt_filter_results`, `usage.extra_properties`) via the `ResponseFieldStripper` utility.

The **reconciler-based caching pattern** is worth noting: both the model store and secret store use controller-runtime reconcilers to watch Kubernetes resources and maintain thread-safe in-memory maps. The `externalModelReconciler` watches `ExternalModel` CRDs using unstructured client (avoiding a hard import dependency on the MaaS API types), while the `secretReconciler` uses a label predicate (`inference.networking.k8s.io/bbr-managed=true`) to watch only relevant Secrets. This keeps the RBAC footprint minimal — the service needs read-only access to ExternalModel CRDs and labeled Secrets, nothing more. The stores use `sync.RWMutex` for concurrent read safety during request processing.

The component defines its own **CRD types** (`ExternalModel` and `ExternalProvider` under `inference.opendatahub.io/v1alpha1`) in the `api/` directory, but the reconciler actually watches ExternalModel CRDs from the `maas.opendatahub.io` group (the MaaS controller's API group). The `inference.opendatahub.io` CRDs appear to be a parallel/future API that is defined but not actively reconciled by this service. This dual-CRD situation suggests an ongoing API migration where the payload processing types may eventually replace or coexist with the MaaS types.
