## Purpose

**Short**: MLServer is a multi-model inference server implementing the V2 Inference Protocol (KFServing predict protocol v2) with REST, gRPC, and Kafka interfaces for serving machine learning models in RHOAI.

**Detailed**: MLServer is a Python-based inference serving runtime originally developed by Seldon Technologies and forked for Red Hat OpenShift AI (RHOAI). It provides a standards-compliant inference server that supports loading and serving multiple ML models simultaneously through both HTTP/REST (via FastAPI/Uvicorn) and gRPC interfaces, following the V2 Inference Protocol (also known as the Open Inference Protocol).

The server supports a pluggable runtime architecture where different ML framework backends (scikit-learn, XGBoost, LightGBM, ONNX, MLflow, HuggingFace, Catboost, Alibi-Explain, Alibi-Detect) can be loaded as runtime implementations. In the RHOAI distribution, the Konflux-built container image ships with four pre-installed runtimes: LightGBM, ONNX, scikit-learn, and XGBoost, secured by a trusted runtimes allowlist that prevents loading arbitrary code.

Key features include adaptive batching for throughput optimization, parallel inference workers to bypass the Python GIL, response caching, CloudEvents integration for event-driven architectures, OpenTelemetry tracing, Prometheus metrics, and streaming inference via Server-Sent Events (REST) and bidirectional streaming (gRPC). The server operates as a model serving component within the KServe inference stack deployed by RHOAI.

## Architectural Analysis

MLServer follows a clean layered architecture with a shared `DataPlane` handler that implements the V2 Inference Protocol business logic, consumed by three independent transport layers: REST (FastAPI), gRPC (grpc.aio), and Kafka (aiokafka). This shared handler ensures consistent behavior across all interfaces. The REST and gRPC servers run concurrently via asyncio, along with an optional dedicated metrics server on a separate port. The `generate` and `generate_stream` endpoints are aliases for `infer` and `infer_stream` respectively, providing compatibility with generative model naming conventions without duplicating logic.

The runtime security model is one of MLServer's most architecturally significant features for RHOAI. The trusted runtimes allowlist (`/etc/mlserver/trusted-runtimes.json`) is baked into the container image at build time and creates a clear PRODUCTION vs DEVELOPMENT mode distinction. In PRODUCTION mode (when the file exists), only pre-approved runtime import paths can be loaded, and dynamic environment loading (`environment_tarball`, `environment_path`) is blocked — preventing arbitrary code execution through model-settings.json manipulation. Wildcard CORS origins are also rejected in PRODUCTION mode. The allowlist is validated both at model settings parse time (early rejection) and again at runtime property access time (defense in depth), with import path canonicalization to prevent bypass via aliased module paths.

The parallel inference architecture uses Python multiprocessing to bypass the GIL, spawning a configurable pool of worker processes that each load the model independently. A `Dispatcher` routes inference requests to workers via multiprocessing queues, and a `WorkerRegistry` tracks loaded models so that crashed workers can be automatically restarted and re-loaded with all previously active models. Adaptive batching adds another performance layer, queuing individual requests and merging them into batches based on configurable size and time thresholds before dispatching to the model's `predict()` method.

The Konflux build pipeline (`Dockerfile.konflux`) is notably minimal — it installs MLServer and runtimes from PyPI using `pip install mlserver` and `pip install mlserver-${_runtime}` rather than building from source. The hermetic build is achieved through the Tekton PipelineRun configuration, which prefetches all pip dependencies from a Red Hat-managed PyPI index (`packages.redhat.com`) using hash-pinned requirements. The regular `Dockerfile` uses a multi-stage build with Poetry wheel building, which is the upstream build path. The image runs as UID 1000 with the `/opt/mlserver` directory made world-writable (sticky bit) for OpenShift random UID compatibility.

The CloudEvents middleware is always active and injects standard CloudEvents headers (`Ce-Id`, `Ce-Source`, `Ce-Type`, etc.) into every inference request and response. This enables event-driven ML pipeline integration and audit trail capabilities. The namespace is read from the Kubernetes service account mount at startup. The OpenTelemetry integration instruments both FastAPI and gRPC with auto-instrumentation, excluding health check endpoints from tracing to reduce noise.
