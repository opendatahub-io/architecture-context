## Purpose

**Short**: Kubernetes operator that manages distributed ML training jobs by reconciling TrainJob custom resources into JobSet workloads using pluggable TrainingRuntime templates.

**Detailed**: Kubeflow Trainer (V2) is the next-generation training operator for the Kubeflow ecosystem, replacing the previous training-operator (V1). It introduces a simplified API model centered around three CRDs: `TrainJob` (the user-facing training job request), `TrainingRuntime` (namespaced reusable training templates), and `ClusterTrainingRuntime` (cluster-scoped reusable templates). When a user creates a `TrainJob`, the controller resolves the referenced runtime template, applies ML-specific policies (PyTorch distributed, MPI, TorchTune fine-tuning), constructs a `JobSet` resource with the appropriate pod topology, and manages the lifecycle through to completion.

The RHOAI distribution extends upstream with several RHAI (Red Hat AI) features: progression tracking that polls training pod HTTP metrics endpoints to report real-time training progress (steps, epochs, ETA) via TrainJob annotations; NetworkPolicy creation for pod isolation during training; and pre-configured ClusterTrainingRuntimes for PyTorch distributed training across CUDA, ROCm, and CPU accelerator variants. The operator also manages gang-scheduling integration via Kubernetes scheduler-plugins (coscheduling) and Volcano PodGroups, and supports Kueue-based multi-cluster job management.

The operator deploys as a single controller-manager pod in the `opendatahub` namespace and manages resources cluster-wide. It uses self-managed certificate rotation for its validating webhook server and exposes Prometheus metrics over a secure HTTPS endpoint.

## Architectural Analysis

The Kubeflow Trainer V2 architecture represents a significant redesign from V1, moving from per-framework operators (PyTorchJob, MPIJob, TFJob) to a unified `TrainJob` API backed by pluggable `TrainingRuntime` templates. The core insight is separating the _what_ (training job specification) from the _how_ (runtime execution template). Users specify their training configuration in a TrainJob, reference a ClusterTrainingRuntime, and the controller constructs the underlying JobSet with all necessary ML-specific configuration. This template-based approach enables platform teams to ship pre-configured runtimes for common patterns (PyTorch distributed, MPI, TorchTune fine-tuning) while users only need to specify their training image, command, and resource requirements.

The plugin framework (`pkg/runtime/framework/`) is cleanly layered: `EnforceMLPolicyPlugin` handles ML-specific environment variable injection (torchrun args, MPI hostfiles), `EnforcePodGroupPolicyPlugin` creates scheduling artifacts (PodGroups), `PodNetworkPlugin` resolves pod DNS endpoints, `ComponentBuilderPlugin` materializes the final Kubernetes resources, and `TrainJobStatusPlugin` aggregates status from the underlying workload. This makes adding new ML frameworks or scheduling integrations straightforward without modifying controller logic. The JobSet plugin serves as the primary `ComponentBuilderPlugin`, translating runtime templates into JobSet apply configurations via server-side apply with field ownership.

The RHAI progression tracking system (`pkg/rhai/`) is a notable RHOAI-specific extension. It operates within the TrainJob reconciliation loop rather than as a separate controller, polling the primary training pod's HTTP `/metrics` endpoint to capture real-time training progress (current step, total steps, epoch, loss, ETA). The design uses `APIReader` for pod lookups (avoiding pod informer/watch overhead on the controller cache), stores progress in TrainJob annotations (avoiding CRD schema changes), and uses configurable poll intervals (default 30s, range 5s-300s). The NetworkPolicy created per TrainJob ensures that only the controller pods can access the metrics endpoint, providing a security boundary even though the metrics transport is plaintext HTTP. Final progress capture supports both HTTP polling and container termination message reading as fallback, with a grace period mechanism to handle pod termination races.

The webhook architecture uses self-managed certificate rotation via `open-policy-agent/cert-controller` rather than cert-manager, making the operator self-contained without external certificate management dependencies. The readiness probe gates on webhook certificate readiness, ensuring the controller doesn't accept traffic until it can serve webhook requests. HTTP/2 is disabled by default as a mitigation for CVE-2023-44487 and CVE-2023-39325 (HTTP/2 Rapid Reset attacks). The controller also enforces immutability constraints on TrainJob fields (runtimeRef, managedBy) and validates that PodTemplateOverrides can only be modified when the TrainJob is suspended and the underlying JobSet is inactive.

The RHOAI overlay provides 15 pre-configured ClusterTrainingRuntimes spanning three accelerator types (CUDA, ROCm, CPU) with multiple PyTorch/Training Hub version combinations. Images are parameterized through a `params.env` file and kustomize replacements, allowing the platform operator to inject version-specific images at deployment time. The OpenShift ImageStream resources enable the Workbench UI to discover available training images. The naming convention uses a "floating" pattern (e.g., `torch-distributed` points to latest CUDA) alongside explicit version-pinned runtimes (e.g., `torch-distributed-cuda130-torch210-py312`).
