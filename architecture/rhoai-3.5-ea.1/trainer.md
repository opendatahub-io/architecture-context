## Purpose

**Short**: Kubeflow Trainer is a Kubernetes operator that manages distributed machine learning training jobs on RHOAI, abstracting the complexity of multi-node training across PyTorch, MPI, and other ML frameworks.

**Detailed**: Kubeflow Trainer (formerly Kubeflow Training Operator) provides a declarative API for running distributed machine learning training workloads on Kubernetes. Users create `TrainJob` custom resources that reference `ClusterTrainingRuntime` or `TrainingRuntime` templates, and the operator reconciles them into `JobSet` resources with appropriate distributed training configuration.

The operator supports multiple ML frameworks through a plugin-based runtime system: PyTorch (torchrun/torchtune), MPI (OpenMPI/Intel MPI/MPICH), and plain ML workloads. It integrates with gang-scheduling systems (Volcano, Kubernetes Scheduler Plugins CoScheduling) to ensure coordinated pod placement for distributed training. The RHOAI distribution adds Red Hat AI (RHAI) features including real-time training progression tracking via HTTP metrics polling, NetworkPolicy-based security isolation for training pods, and pre-configured ClusterTrainingRuntimes with CUDA, ROCm, and CPU training images. It also provides a Rust-based data cache component using Apache Arrow Flight for high-performance distributed data access during training.

## Architectural Analysis

The Kubeflow Trainer implements a sophisticated plugin-based runtime framework that decouples ML-framework-specific logic from core reconciliation. The framework defines extension points (`EnforceMLPolicyPlugin`, `EnforcePodGroupPolicyPlugin`, `ComponentBuilderPlugin`, `PodNetworkPlugin`, `TrainJobStatusPlugin`, `CustomValidationPlugin`) that are executed in a defined sequence during reconciliation. Six plugins are registered: Torch, MPI, PlainML, JobSet, Volcano, and CoScheduling. This design allows adding new ML frameworks or scheduling integrations without modifying core controller logic. The JobSet plugin is the only `TrainJobStatusPlugin` — it maps JobSet/ReplicatedJob status back to TrainJob conditions.

The RHAI (Red Hat AI) extensions add three significant capabilities atop upstream Kubeflow Trainer. First, real-time progression tracking polls training metrics from the rank-0 pod's HTTP endpoint at configurable intervals (default 30s), storing progress data as JSON annotations on the TrainJob (`trainer.opendatahub.io/trainerStatus`). This avoids watching pods in the controller's informer cache and instead uses the APIReader for direct pod lookups. Second, per-TrainJob NetworkPolicies enforce pod isolation: only pods within the same TrainJob can communicate freely, and only the controller namespace can access the metrics port. Third, the RHOAI overlay ships 16 pre-configured ClusterTrainingRuntimes covering CUDA 12.8/13.0, ROCm 6.4, and CPU variants across torch-distributed and training-hub frameworks, along with OpenMPI CUDA support.

The data cache component uses a head-worker architecture built on Apache Arrow Flight gRPC. The head node partitions Iceberg table data across workers using a simple range-based algorithm. Workers load assigned files and serve them with a `cache_index` column for global row ordering. Health checks use simple HTTP endpoints that verify table queryability. This component currently has no Konflux Dockerfile, suggesting it may be in early development or not yet shipped as part of the RHOAI distribution. Its Dockerfile uses Debian base images (not UBI) and does not include FIPS build flags, which would need addressing before RHOAI GA.

The controller's certificate management is noteworthy — it uses `open-policy-agent/cert-controller` for automatic webhook certificate rotation, storing TLS certs in a Kubernetes Secret and updating the ValidatingWebhookConfiguration's CA bundle. The readiness probe gates on certificate availability, ensuring no webhook requests are served before certs are ready. HTTP/2 is disabled by default to mitigate the HTTP/2 Stream Cancellation and Rapid Reset CVEs (GHSA-qppj-fm5r-hxr3, GHSA-4374-p667-p6c8). The controller concurrency is configured at 5 for TrainJobs but only 1 for TrainingRuntime/ClusterTrainingRuntime, reflecting the expected volume difference.

The watcher pattern between controllers is also notable: TrainingRuntime and ClusterTrainingRuntime reconcilers implement a `TrainJobWatcher` interface. When a TrainJob is created, updated, or deleted, the TrainJob reconciler notifies the runtime reconcilers via buffered channels (capacity 10), enabling the runtime reconcilers to add or remove finalizers. This cross-controller coordination prevents deletion of runtimes that are still referenced by active TrainJobs without requiring direct cross-controller API calls.
