## Purpose

**Short**: MLServer is a multi-model inference server implementing the KFServing V2 Inference Protocol over REST, gRPC, and Kafka, serving as the Python model-serving backend for KServe in RHOAI.

**Detailed**: MLServer provides a production-ready inference server for machine learning models with full compliance to the KFServing V2 Dataplane specification. It supports multi-model serving within a single process, allowing users to host multiple models simultaneously. The server exposes three transport protocols: REST (via FastAPI/Uvicorn on port 8080), gRPC (via grpc.aio on port 8081), and Apache Kafka (for asynchronous event-driven inference). A separate Prometheus metrics server runs on port 8082.

The architecture supports parallel inference through a multiprocessing worker pool that bypasses the Python GIL, adaptive batching to group individual requests for efficient GPU/CPU utilization, and response caching for deduplication. MLServer ships with pre-built runtime plugins for scikit-learn, XGBoost, LightGBM, ONNX, MLflow, HuggingFace, Alibi-Detect, Alibi-Explain, and CatBoost. In the RHOAI distribution, the Konflux-built container image bundles four runtimes (sklearn, xgboost, lightgbm, onnx) and enforces a trusted runtime allowlist in production mode to prevent arbitrary code execution.

MLServer integrates with KServe as the default Python inference runtime. When a user creates an InferenceService with a Python-based model format, KServe deploys an MLServer container with the appropriate runtime pre-configured. The server discovers models from a configurable model repository directory (defaulting to `/mnt/models`), loading `model-settings.json` files that specify which runtime implementation to use.

## Architectural Analysis

MLServer's architecture follows a clean layered design: protocol adapters (REST, gRPC, Kafka) sit atop a unified DataPlane handler layer, which dispatches to model-specific runtimes through a registry. This separation means adding a new transport protocol requires only implementing a thin server adapter — the DataPlane, model lifecycle, and runtime logic remain unchanged. The handler layer applies cross-cutting concerns (CloudEvents middleware, response caching, metrics) uniformly across all protocols.

The parallel inference subsystem is architecturally significant. Rather than using threading (which is ineffective due to the Python GIL), MLServer spawns worker processes via `multiprocessing.spawn`. Each worker runs its own event loop and model instances. A round-robin dispatcher routes requests to workers through multiprocessing queues, with an `AsyncResponses` layer that maps each dispatched request to an `asyncio.Future` for non-blocking await in the main process. This design enables true CPU parallelism for NumPy/scikit-learn workloads while maintaining the async-first API surface. Worker failures are detected via SIGCHLD signals and trigger automatic restarts with model reloading.

The trusted runtime allowlist is a notable security feature for RHOAI deployments. In production mode, a JSON allowlist file is baked into the container image at `/etc/mlserver/trusted-runtimes.json` with read-only permissions (0444). When a `model-settings.json` specifies an `implementation` import path, MLServer validates it against this allowlist at two layers: once during settings validation (Pydantic validator) and again at access time (property getter), providing defense-in-depth against runtime code injection. Production mode also blocks custom Python environments (which could smuggle untrusted code) and wildcard CORS origins. The `/v2/runtimes` endpoint exposes the current security mode (PRODUCTION/DEVELOPMENT) and the active allowlist, enabling operators to audit deployed images.

The adaptive batching system collects individual inference requests into batches using an async queue with configurable `max_batch_size` and `max_batch_time` parameters. Each request gets an internal ID and a Future; when the batch threshold is met or the time window expires, the batch is sent to the model's `predict()` method in a single call. The response is then split back to individual requestors. This is critical for GPU-bound models where batching dramatically improves throughput. However, streaming inference (`predict_stream`) does not support adaptive batching — a warning is logged and requests fall through unbatched.

The RHOAI Konflux build pipeline is hermetic: the Tekton PipelineRun sets `hermetic: true` and uses Hermeto pip prefetch with a hash-pinned requirements file sourced from Red Hat's internal PyPI mirror. This ensures reproducible builds with verified dependency provenance. The base image (`quay.io/aipcc/base-images/cpu:3.5.0-ea.1`) is a UBI-based image providing Python 3.12 and system libraries. Multi-architecture builds (x86_64 and arm64) are configured. The image outputs to `quay.io/rhoai/odh-mlserver-rhel9`.
