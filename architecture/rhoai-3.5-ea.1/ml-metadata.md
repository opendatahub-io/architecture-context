## Purpose

**Short**: ML Metadata (MLMD) is a gRPC server and Python library for recording, retrieving, and querying metadata associated with ML workflows, including artifacts, executions, contexts, and their lineage relationships.

**Detailed**: ML Metadata is a Google-originated library (google/ml-metadata) forked and maintained by Red Hat Data Services for inclusion in Red Hat OpenShift AI (RHOAI). It provides a metadata store that tracks the full lifecycle of ML pipelines: the artifacts produced and consumed (datasets, models, metrics), the executions that transform them (training runs, evaluations, deployments), and the contexts that group them (pipelines, experiments, runs). The data model consists of typed nodes (ArtifactType, ExecutionType, ContextType), their instances (Artifact, Execution, Context), and edges (Event, Attribution, Association, ParentContext) that form a provenance graph.

The primary deployable artifact is a C++ gRPC server (`metadata_store_server`) that exposes the `MetadataStoreService` API defined in protobuf. This server supports three relational database backends: SQLite (in-memory or file-based), MySQL/MariaDB, and PostgreSQL. The server handles schema creation, migration (upgrade and downgrade), and provides optimistic concurrency control, pagination, filtering, and lineage graph traversal. A secondary artifact is a Python client library (`ml-metadata` PyPI package) with pybind11 bindings to the C++ core, allowing direct database connections or gRPC-based remote connections.

In RHOAI, the MLMD gRPC server is deployed as a sidecar or standalone service alongside ML pipeline components (particularly Kubeflow Pipelines), providing the metadata backbone that tracks pipeline runs, artifacts, and lineage. The server connects to an external database (typically MySQL or PostgreSQL) provisioned by the platform.

## Architectural Analysis

ML Metadata follows a clean layered architecture with well-defined separation of concerns. At the bottom layer, database-specific `MetadataSource` implementations (SQLite, MySQL, PostgreSQL) handle raw SQL execution. Above that, `QueryExecutor` and `QueryConfigExecutor` use template-driven SQL queries defined in `MetadataSourceQueryConfig` protobuf messages, abstracting away SQL dialect differences. The `RDBMSMetadataAccessObject` implements the core CRUD logic, while `MetadataStore` wraps everything in transaction-safe operations. The `MetadataStoreServiceImpl` gRPC layer is intentionally thin — each RPC method creates a new database connection, performs the operation, and returns. This per-request connection pattern (noted in the TODO at `metadata_store_service_impl.h:28`) means the server does not maintain a connection pool, which simplifies concurrency but may impact performance under load.

The data model is notable for its type system flexibility. Types (ArtifactType, ExecutionType, ContextType) define property schemas that can be evolved forward-compatibly — new properties can be added but existing property types cannot change. This is enforced by the `can_add_fields` and `can_omit_fields` flags on type upsert operations. The system also supports system-defined base types (DATASET, MODEL, METRICS for artifacts; TRAIN, TRANSFORM, PROCESS for executions) that enable tooling to interpret metadata semantically without knowing custom type schemas.

The schema migration system is sophisticated — the `MetadataSourceQueryConfig` proto includes a `migration_schemes` map keyed by schema version, with upgrade and downgrade query sequences and verification schemes. The server supports automatic upgrade migration on startup (via `--enable_database_upgrade`) and explicit downgrades (via `--downgrade_db_schema_version`). This enables rolling upgrades and rollbacks in production environments.

The RHOAI downstream fork (red-hat-data-services/ml-metadata) maintains the upstream Google codebase with minimal modifications. Recent commit history shows only automated UBI base image digest updates managed by Renovate, plus Tekton pipeline synchronization from the Konflux central repository. The Konflux Dockerfile (`Dockerfile.konflux`) is nearly identical to the community `Dockerfile.redhat`, differing mainly in pinned base image digests (SHA256) and build flag tuning. The multi-architecture build supports amd64, arm64, and ppc64le platforms as specified in the Tekton PipelineRun.

A significant architectural consideration is the BoringSSL dependency. The C++ binary links against Bazel-fetched BoringSSL rather than the system OpenSSL. While the UBI9-minimal runtime image includes FIPS-capable OpenSSL, the application binary uses its own BoringSSL build which is not configured for FIPS mode. This creates a potential FIPS compliance gap for the gRPC TLS layer — the server's TLS implementation runs through BoringSSL regardless of the host's FIPS settings. For FIPS-sensitive deployments, the workaround is to not enable TLS at the MLMD server level and instead rely on network-level encryption (e.g., Istio mTLS or a TLS-terminating proxy).
