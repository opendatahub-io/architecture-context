## Purpose

**Short**: ML Metadata (MLMD) is a gRPC server that records and retrieves metadata associated with ML workflows, storing artifacts, executions, contexts, and their lineage relationships in a relational database backend.

**Detailed**: ML Metadata is a library and server originally developed by Google as part of the TFX (TensorFlow Extended) ecosystem. It provides a schema-aware metadata store for tracking the provenance of ML artifacts (datasets, models, metrics), executions (training runs, transformations, evaluations), and contexts (pipelines, experiments) along with the events that connect them. The data model forms a directed acyclic graph that enables lineage tracing — answering questions like "which dataset was used to train this model?" or "what pipeline produced this artifact?"

In the RHOAI platform, the MLMD gRPC server is deployed as a backend service that Kubeflow Pipelines and Data Science Pipelines use to persist pipeline run metadata. The server accepts protobuf-serialized gRPC requests on port 8080 and delegates to a pluggable storage backend — supporting MySQL/MariaDB, PostgreSQL, and SQLite. The RHOAI distribution builds the server from C++ source using Bazel, packages it into a UBI9-minimal container image, and ships it via Konflux CI. The Python client library (published as `ml-metadata` on PyPI) can connect either directly to a database or to the gRPC server, providing a high-level API for ML orchestration systems.

The server is stateless itself — all state is stored in the configured database backend. It supports schema migration (upgrade and downgrade), optimistic concurrency control for concurrent writers, transaction retry with exponential backoff, and optional TLS for both client-server and server-database connections.

The service exposes 40+ RPC methods organized into these categories:

- `GetArtifactsByExternalIds`, `GetExecutionsByExternalIds`, `GetContextsByExternalIds` — Retrieve by external ID

**Event & Relationship Operations**:
- `PutEvents` — Record input/output events between artifacts and executions
- `PutExecution` — Atomically upsert an execution with its artifacts, events, and context associations
- `PutLineageSubgraph` — Atomically upsert a collection of executions, artifacts, contexts, and event edges
- `PutAttributionsAndAssociations` — Create context-artifact and context-execution relationships
- `PutParentContexts` — Create hierarchical context relationships
- `GetEventsByExecutionIDs`, `GetEventsByArtifactIDs` — Retrieve events
- `GetContextsByArtifact`, `GetContextsByExecution` — Navigate context relationships
- `GetArtifactsByContext`, `GetExecutionsByContext` — Navigate context memberships
- `GetParentContextsByContext`, `GetChildrenContextsByContext` — Navigate context hierarchy

**Lineage Queries**:
- `GetLineageSubgraph` — Graph traversal from starting nodes with configurable hops, direction, and ending conditions
- `GetLineageGraph` (deprecated) — Older lineage API

## Architectural Analysis

The ML Metadata server follows a clean layered architecture. At the top is the gRPC service implementation (`MetadataStoreServiceImpl`), which delegates to a `MetadataStore` C++ library. The store uses a `MetadataAccessObject` that abstracts database operations through a `QueryExecutor` interface with implementations for MySQL (`PostgreSQLQueryExecutor`), SQLite, and PostgreSQL. SQL queries are templated via `MetadataSourceQueryConfig` protobufs, making the query layer database-agnostic while still supporting database-specific optimizations and migration paths.

The data model is centered around three node types — Artifacts, Executions, and Contexts — connected by edges (Events, Attributions, Associations, ParentContexts). Each node type has a corresponding Type that defines its schema (properties and their value types). This type-instance pattern is similar to a class-object relationship and enables the system to enforce schema consistency while allowing schema evolution through backward-compatible type updates. The `PutExecution` method is particularly notable: it provides an atomic "publish" operation that creates/updates an execution along with all its artifacts, events, and context associations in a single transaction, which is critical for pipeline orchestrators that need to avoid partial metadata records.

The RHOAI distribution packages this as a multi-architecture container image (amd64, arm64, ppc64le) built on UBI9-minimal. The build process is notable for its complexity: Bazel compiles the C++ server from source including all dependencies (gRPC, protobuf, ZetaSQL, BoringSSL), then a multi-stage Docker build copies only the final binary into a minimal runtime image. The server runs as non-root (UID 65534) and exposes only port 8080 for gRPC traffic. The ZetaSQL dependency requires special attention — different RHEL/UBI versions require different ZetaSQL commits, and the `Dockerfile.redhat` patches the WORKSPACE file to switch commits for UBI9 compatibility.

A key architectural observation is that the MLMD server itself has no Kubernetes awareness — it is a pure gRPC service with no CRDs, no controller logic, and no direct Kubernetes API interaction. All Kubernetes integration (Deployment, Service, ConfigMap for the server config file, database connection secrets) is managed by the Data Science Pipelines operator. This separation of concerns makes the MLMD server highly portable but means its deployment topology is entirely determined by its deploying operator.

The Python client library supports two connection modes: direct database connection (using pybind11 bindings to the C++ library) and gRPC connection to the server. The gRPC client includes automatic retry with exponential backoff for `AbortedError` responses (up to 5 retries by default), which handles transient database contention gracefully. The library also supports SSL/TLS with custom CA certificates, client key/cert for mutual TLS, and configurable gRPC channel arguments like max message length.
