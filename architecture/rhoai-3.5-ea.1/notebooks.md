## Purpose

**Short**: Builds and ships container images for interactive data science workbenches (JupyterLab, Code-Server) and Elyra pipeline runtimes used by Red Hat OpenShift AI.

**Detailed**: The notebooks repository produces the complete set of container images that RHOAI users interact with when they launch a workbench or execute an Elyra pipeline. It builds a hierarchy of images spanning multiple IDE types (JupyterLab, Code-Server), ML frameworks (PyTorch, TensorFlow, LLM Compressor, TrustyAI), and hardware accelerators (CPU, NVIDIA CUDA, AMD ROCm). Each image variant has both a standard Dockerfile and a Konflux-hermetic Dockerfile (`Dockerfile.konflux.*`) for reproducible, air-gapped builds in the Konflux CI/CD system.

The repository also produces OpenShift `ImageStream` manifests (in `manifests/rhoai/` and `manifests/odh/`) that the rhods-operator/opendatahub-operator deploys to clusters. These ImageStreams carry rich annotations (`opendatahub.io/notebook-python-dependencies`, `opendatahub.io/recommended-accelerators`, etc.) that the ODH Dashboard reads to present workbench options to users. The manifest layer uses Kustomize with `configMapGenerator` and `replacements` to inject image digests and commit hashes into ImageStream tags at deployment time.

Supporting infrastructure includes a Python lockfile generation system (`scripts/pylocks_generator.py`) producing PEP 751 `pylock.toml` files, a Go tool (`scripts/buildinputs/`) that statically analyzes Dockerfiles to extract COPY/ADD dependencies for minimal build contexts, a comprehensive test suite (unit, container integration via testcontainers, browser via Playwright), and CVE management tooling.

## Architectural Analysis

The notebooks repository is architecturally unique within the RHOAI platform: it produces no running services or controllers, only container images and kustomize manifests. Its "deployment" is entirely mediated by other components -- the platform operator deploys ImageStreams, the Dashboard reads annotations, and the Notebook Controller handles pod lifecycle. This separation of concerns means the repository's architecture is best understood through its image hierarchy, build system, and manifest parameterization rather than through traditional service-oriented analysis.

The image hierarchy forms a layered build chain where each image inherits from the previous layer via `FROM`. The base layer provides the OS (UBI9/c9s) and Python interpreter. The `jupyter/minimal` layer adds JupyterLab and basic scientific packages. The `jupyter/datascience` layer adds the full data science stack (NumPy, Pandas, scikit-learn, Elyra, MLflow, etc.). Framework-specific layers (pytorch, tensorflow, trustyai, llmcompressor) build on datascience. This means a change to the minimal image's Dockerfile triggers rebuilds of all downstream images. The `runtimes/` images mirror the same flavor structure but exclude JupyterLab, producing headless images for Elyra pipeline execution.

The dual-Dockerfile pattern (`Dockerfile.cpu` vs `Dockerfile.konflux.cpu`) is a notable architectural choice. The Konflux variants implement fully hermetic builds: all RPMs are prefetched into `cachi2/output/deps/rpm/`, all Python wheels into `cachi2/output/deps/pip/`, Go modules into `cachi2/output/deps/gomod/`, and generic assets (GPG keys, code-server source, VS Code headers) into `cachi2/output/deps/generic/`. RPM GPG keys are imported explicitly. Python packages are installed with `uv pip install --no-index --find-links /cachi2/output/deps/pip`. A `check_dockerfile_alignment.sh` script enforces byte-identical parity between standard and Konflux Dockerfiles (post-hermetic additions). Lock files exist at three layers: `rpms.lock.yaml` for OS packages, `pylock.*.toml` (PEP 751) for Python, and `go.sum` for Go. This represents a mature supply chain hardening posture.

The code-server workbench has the most complex internal architecture due to its three-process stack (nginx + httpd + bash CGI). Since code-server lacks a Jupyter-compatible API, the CGI script `access.cgi` polls `localhost:8888/codeserver/healthz` for heartbeats and translates them into the `/api/kernels/` JSON format that the Notebook Controller's culler expects. The ARCHITECTURE.md notes this architecture is "fragile and is planned for replacement with a single Go reverse proxy." This is a significant technical debt item.

The manifest parameterization system is sophisticated but complex. ImageStream tags use placeholder values (e.g., `odh-workbench-jupyter-minimal-cpu-py312-ubi9-n_PLACEHOLDER`) that are replaced at deployment time through a multi-layer process: the operator reads `params-latest.env`, patches values from `RELATED_IMAGE_*` CSV environment variables, generates a ConfigMap via kustomize's `configMapGenerator`, and then kustomize's `replacements` section injects each ConfigMap key into the corresponding ImageStream field. This produces a 1250-line `kustomization.yaml` with approximately 400 replacement rules. The `generate_kustomization.py` tool auto-generates this file to prevent drift.
