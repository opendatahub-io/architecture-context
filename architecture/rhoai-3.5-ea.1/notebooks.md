## Purpose

**Short**: Builds container images for interactive data science workbenches (JupyterLab, Code-Server, RStudio) and pipeline runtime images (Elyra) for RHOAI/ODH platforms.

**Detailed**: The notebooks repository produces the complete set of container images that data scientists use as interactive development environments on Red Hat OpenShift AI. These images are organized in a layered hierarchy: base images (CPU, CUDA, ROCm) provide OS and GPU runtimes, minimal images add JupyterLab, datascience images add ML libraries, and specialized images add framework-specific packages (PyTorch, TensorFlow, TrustyAI, LLM Compressor). Three IDE types are supported: JupyterLab (primary), Code-Server (VS Code in browser), and RStudio Server.

In addition to interactive workbench images, the repository produces pipeline runtime images used by Elyra v4.3.1 to execute Jupyter notebooks as steps in Kubeflow Pipelines. These runtime images are lightweight variants without JupyterLab UI but with the same ML framework packages.

The repository does not deploy Kubernetes operators or services. Instead, it produces OpenShift ImageStream manifests consumed by the ODH/RHOAI platform operator, which deploys them to clusters. The ODH Dashboard discovers these ImageStreams via annotations to populate the workbench launcher UI. The notebook controller (kubeflow) manages pod lifecycle, injecting kube-rbac-proxy sidecars, CA certificate bundles, and Elyra pipeline configuration at runtime.

## Architectural Analysis

### Image Hierarchy and Build Strategy

The repository implements a sophisticated layered image hierarchy that maximizes layer sharing while supporting three GPU platforms (CPU, NVIDIA CUDA, AMD ROCm) and four ML frameworks (PyTorch, TensorFlow, TrustyAI, LLM Compressor). Each Dockerfile uses multi-stage builds: a `cpu-base`/`cuda-base`/`rocm-base` stage installs OS packages and tools, then successive stages (`jupyter-minimal` -> `jupyter-datascience` -> `jupyter-pytorch`) add framework-specific Python packages. This layering means PyTorch and TensorFlow images share the entire datascience layer, avoiding duplication of ~2GB of common ML libraries.

The build system supports two modes: standard Dockerfiles (Dockerfile.cpu, Dockerfile.cuda) for local development, and Konflux Dockerfiles (Dockerfile.konflux.*) for hermetic CI builds. Both variants reference `/cachi2/output/deps/` paths for prefetched dependencies, indicating the build system has converged on hermetic builds even for local development. The `PYLOCK_FLAVOR` build argument selects the correct lock file variant (cpu, cuda, rocm) for each image, enabling a single Dockerfile to produce architecture-appropriate builds.

### Idle Culling Emulation Architecture

A notable architectural pattern is the idle culling emulation for Code-Server and RStudio. The ODH notebook culler expects a Jupyter-compatible `/api/kernels/` endpoint returning JSON with `last_activity` timestamps and `execution_state` fields. Since Code-Server and RStudio lack this API, the repository implements a three-process stack: nginx (port 80) acts as reverse proxy with custom JSON access logging, httpd (port 8080) serves as a CGI gateway, and bash CGI scripts (`access.cgi`) translate IDE-specific health signals into Jupyter kernel format. For Code-Server, the script polls `/codeserver/healthz` and converts heartbeat timestamps; for RStudio, it parses nginx access logs and marks idle after 10 minutes of inactivity. The ARCHITECTURE.md file notes this approach is fragile and planned for replacement with a single Go reverse proxy.

### Dependency Management and Supply Chain Security

The dependency management approach uses a meta-package composition pattern where shared dependency sets (`odh-notebooks-meta-runtime-elyra-deps`, `odh-notebooks-meta-workbench-datascience-deps`, etc.) are defined as separate Python packages in the `dependencies/` directory. Each image's `pyproject.toml` references these meta-packages as dependencies, and `uv` resolves the full transitive closure into platform-specific lock files (`pylock.cpu.toml`, `pylock.cuda.toml`, `pylock.rocm.toml`). CVE constraints are managed centrally in `dependencies/cve-constraints.txt` and propagate to all images during lock file regeneration. This approach ensures consistent dependency versions across the entire image fleet while allowing per-platform overrides (e.g., protobuf==6.31.1 pinned for s390x UPB C extension segfaults).

### Multi-Architecture Considerations

The images support four architectures (x86_64, aarch64, ppc64le, s390x) with careful per-platform handling. Several packages are excluded on ppc64le and s390x via pyproject.toml platform markers (codeflare-sdk, torch, triton, accelerate). The mongocli Go binary creates a dummy placeholder on s390x. ROCm images use custom Python scripts (`de-vendor-torch.py`, `link-solibs.py`) to handle AMD GPU library linking differences. The Cachi2 hermetic build system uses `whl-cache` stages to pre-compile Python C extensions on architectures where pre-built wheels are unavailable (ppc64le, s390x).

### Manifest Parameterization Pipeline

The kustomize manifests implement a sophisticated three-version image tracking system for RHOAI (3.4/current, 2025.2/previous, 2025.1/older). At deploy time, the RHOAI operator substitutes `dummy` placeholders in `params-latest.env` with actual image digests from `RELATED_IMAGE_*` environment variables, enabling the operator to control which image version gets the "latest" tag without modifying manifests.
