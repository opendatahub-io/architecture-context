## Purpose

**Short**: Data Science Pipelines is a multi-component platform for defining, deploying, scheduling, and monitoring reusable end-to-end machine learning workflows on Kubernetes, built on Kubeflow Pipelines v2 and Argo Workflows.

**Detailed**: Data Science Pipelines provides the core ML pipeline orchestration capabilities for Red Hat OpenShift AI. It enables data scientists to define reproducible ML workflows as directed acyclic graphs (DAGs) of containerized steps, submit them for execution via a REST/gRPC API, and track their artifacts, parameters, and lineage through an integrated metadata service. The platform compiles user-defined pipeline specifications into Argo Workflow resources, which are then executed on the Kubernetes cluster.

The system consists of five shippable container images built via Konflux: an API server that exposes pipeline CRUD operations and coordinates execution, a driver that runs inside Argo task pods to handle DAG orchestration and input/output resolution, a launcher that executes user code and manages artifact upload/download, a persistence agent that watches Argo Workflows and reports their status back to the API server's database, and a scheduled workflow controller that creates recurring pipeline runs based on cron or periodic schedules.

Pipeline artifacts (models, datasets, metrics) are stored in S3-compatible object storage (SeaweedFS, MinIO, or cloud-native S3/GCS), while pipeline metadata (definitions, run history, experiment groupings) is persisted in a relational database (MySQL or PostgreSQL). ML Metadata (MLMD) provides execution lineage tracking through a separate gRPC service. The system supports both single-user and multi-user modes with Kubernetes-native RBAC authorization via SubjectAccessReview and TokenReview APIs.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| API Server | Go Service | Central REST/gRPC API for pipeline CRUD, run management, experiment tracking, artifact retrieval, and pipeline compilation |
| Driver | Go CLI (Argo task container) | Orchestrates DAG execution inside Argo task pods: resolves inputs, evaluates conditions, checks cache, generates pod spec patches for launchers |
| Launcher | Go CLI (user container sidecar) | Executes user code inside pipeline step containers: downloads input artifacts, runs user commands, uploads outputs, publishes execution metadata to MLMD |
| Persistence Agent | Go Service | Watches Argo Workflow and ScheduledWorkflow resources via Kubernetes informers and persists their status to the API server database |
| Scheduled Workflow Controller | Go Controller | Kubernetes controller that watches ScheduledWorkflow CRDs and creates Argo Workflow instances according to cron or periodic schedules |
| Frontend UI | TypeScript/React Service | Web interface for browsing pipelines, runs, experiments, and artifacts (port 3000) |

## Architectural Analysis

Data Science Pipelines implements a **compiler-driver-launcher** architecture that cleanly separates pipeline definition, orchestration, and execution concerns. The API server compiles user-defined PipelineSpec (protobuf-based IR from the KFP SDK) into Argo Workflow YAML at submission time, embedding driver invocations as Argo task templates. This compilation step translates the KFP component model (with its input/output type system, artifact lineage, and caching semantics) into Argo's native DAG execution model. The driver-launcher split mirrors a control plane / data plane separation within each pipeline task: the driver handles orchestration decisions (input resolution, cache checking, condition evaluation) while the launcher handles data plane concerns (artifact download/upload, user code execution, metadata publishing).

The **dual-binary launcher** strategy is architecturally notable: the Konflux Dockerfile builds both a statically-linked non-FIPS binary (`launcher-v2`, CGO_ENABLED=0) and a dynamically-linked FIPS binary (`launcher-v2-fips`, CGO_ENABLED=1 with strictfipsruntime). This allows the platform operator to select the appropriate binary at runtime based on FIPS requirements. All other components (API server, driver, persistence agent, scheduled workflow controller) are built exclusively with FIPS-compliant flags, reflecting their deployment as managed infrastructure rather than user-injected containers.

The **persistence agent** and **scheduled workflow controller** demonstrate classic Kubernetes informer-based patterns with work queues and exponential backoff retry. Both use projected ServiceAccount tokens with 1-hour expiration for API server authentication, implementing a token refresher that transparently handles rotation. The persistence agent's TTL-based workflow cleanup (default 86400s) is critical for preventing unbounded resource accumulation in long-running clusters. The scheduled workflow controller supports cron, periodic, and one-off schedules with concurrency control (MaxConcurrency 1-10) and catchup semantics, enabling complex recurring pipeline patterns.

The **caching system** operates at the container driver level by generating fingerprints from input parameters, artifact URIs, container images, commands, and PVC names. On cache hit, the driver short-circuits execution entirely—no user container is created, and output artifacts from the cached execution are linked to the new execution in MLMD. This provides significant cost savings for iterative ML development but requires careful consideration: PVC-mounted data changes aren't captured in the cache key, potentially causing stale results. The cache staleness controls (MAXIMUM_CACHE_STALENESS, DEFAULT_CACHE_STALENESS) provide administrators with configurable bounds.

The **multi-database support** (MySQL and PostgreSQL via GORM) with automatic schema migration and legacy schema detection reflects the component's evolution from KFP v1 to v2. The API server detects pre-2.15 schema versions at startup and performs in-place upgrades including column resizing, index recreation, and data backfill. This migration path is critical for RHOAI upgrades where existing pipeline data must be preserved. The Kubernetes-backed pipeline storage mode (using Pipeline/PipelineVersion CRDs instead of the database) is an alternative deployment model with an associated validating/mutating webhook, but the database-backed mode remains the default for RHOAI deployments.
