## Purpose

**Short**: Kubeflow Pipelines backend and frontend for orchestrating reusable end-to-end ML workflows on Kubernetes via Argo Workflows.

**Detailed**: Data Science Pipelines (DSP) is the RHOAI distribution of Kubeflow Pipelines v2, providing a complete ML pipeline orchestration platform. It enables users to define, compile, schedule, and monitor multi-step ML workflows that execute as Argo Workflows on Kubernetes. The platform consists of an API server exposing gRPC and REST endpoints for pipeline CRUD operations, a persistence agent that synchronizes Argo Workflow state back to the API server, a scheduled workflow controller for cron/periodic pipeline execution, a driver component that orchestrates DAG execution and caching decisions before containers run, and a launcher that manages artifact I/O and metadata recording inside task containers.

The system stores pipeline definitions and execution metadata in a relational database (MySQL or PostgreSQL), artifacts in S3-compatible object storage (MinIO, AWS S3, or GCS), and execution lineage metadata in ML Metadata (MLMD) via gRPC. A mutating webhook-based cache server intercepts pod creation to skip previously-computed pipeline steps, improving execution efficiency. The frontend provides a React-based UI for browsing experiments, pipelines, runs, and artifacts. The platform integrates with the broader RHOAI ecosystem through the `data-science-pipelines-operator` (DSPO) which manages deployment of all these components.

## Architecture Components

| Component | Type | Purpose |
|-----------|------|---------|
| API Server | Go Service (gRPC + REST) | Central control plane: pipeline CRUD, run management, experiment tracking, artifact serving, webhook handling |
| Driver | Go CLI (Argo step) | Pre-container orchestration: DAG traversal, input resolution, cache lookup, pod spec patching |
| Launcher | Go CLI (Argo step) | In-container execution: artifact download/upload, user code execution, metadata recording |
| Persistence Agent | Go Service (controller) | Watches Argo Workflows and ScheduledWorkflows, persists state to API server via gRPC |
| Scheduled Workflow Controller | Go Controller | Manages ScheduledWorkflow CRD, creates Argo Workflows or Runs on cron/periodic schedules |
| Cache Server | Go Service (webhook) | Mutating admission webhook that intercepts pod creation to inject cached execution results |
| Viewer Controller | Go Controller | Manages Viewer CRD for TensorBoard instances with backing Deployments and Services |
| Frontend | React/TypeScript SPA | Web UI for pipeline management, run monitoring, experiment browsing, artifact visualization |
| Python SDK | Python Library | Client SDK for defining, compiling, and submitting pipelines programmatically |

## Architectural Analysis

Data Science Pipelines follows a **two-phase execution model** that cleanly separates orchestration from computation. The **driver** runs as a pre-step in Argo Workflows to handle DAG traversal, input resolution, and caching decisions, producing a pod spec patch that configures the actual task container. The **launcher** then runs inside the task container to handle artifact I/O, user code execution, and metadata recording. This separation means the driver can make caching decisions before any compute resources are consumed — if a cache hit occurs, the pod runs a dummy busybox image with minimal resources instead of the actual computation.

The **cache architecture** operates at two levels. The v1 cache (cache-server webhook) intercepts pod creation at the Kubernetes admission level and modifies the pod spec in-place for Argo-native workflows. The v2 cache operates through the driver component, which queries the API server's task cache via gRPC before the task container starts. The launcher binary ships as **two binaries in a single image** — a FIPS-compliant dynamically-linked version and a non-FIPS statically-linked version — allowing the API server to select the appropriate binary based on the cluster's FIPS mode. This dual-binary pattern is unique among RHOAI components.

The **API server** is architecturally the most complex component, serving as the single point of coordination. It exposes a dual gRPC/REST API via grpc-gateway, manages pipeline storage in both database and Kubernetes CRD backends, handles admission webhooks for PipelineVersion resources, and provides the cache query interface for the driver. The server supports two database backends (MySQL and PostgreSQL) through a SQL dialect abstraction layer, and three object store backends (MinIO/S3, GCS, filesystem) through gocloud.dev. Authentication is pluggable: single-user mode uses a simple HTTP header (`kubeflow-userid`), while multi-user mode performs Kubernetes TokenReview and SubjectAccessReview for fine-grained RBAC.

The **TLS configuration** across components follows a consistent pattern using a shared `GetTLSConfig()` utility that loads the system CA pool and optionally appends a custom CA certificate. However, no components configure custom cipher suites or minimum TLS versions — they rely on Go's defaults, which in FIPS mode (strictfipsruntime) are restricted to FIPS-approved cipher suites via OpenSSL. The persistence agent and scheduled workflow controller both authenticate to the API server using Kubernetes service account tokens mounted at a custom path (`/var/run/secrets/kubeflow/tokens/`), with automatic token refresh on configurable intervals.

The **kustomize manifest structure** supports multiple deployment topologies: standalone single-user, multi-user with Istio, OpenShift with Routes, and platform-agnostic with cert-manager TLS. The multi-user overlay adds ClusterRoles, Istio AuthorizationPolicies, and per-namespace isolation. In RHOAI deployments, these manifests are not applied directly — instead, the data-science-pipelines-operator (DSPO) reads the component's configuration from a `DataSciencePipelinesApplication` CR and deploys the components with RHOAI-specific customizations including kube-rbac-proxy sidecars for authentication.
