## Purpose

**Short**: Kubernetes operator that manages MLflow tracking server deployments on OpenShift AI / Open Data Hub platforms using a Helm-based reconciliation pattern.

**Detailed**: The MLflow Operator is a controller-runtime based Kubernetes operator that watches cluster-scoped `MLflow` custom resources and reconciles them into fully functional MLflow tracking server deployments. Each MLflow CR declaratively specifies the desired state — container image, storage backends (SQLite, PostgreSQL, MySQL), artifact storage (local PVC, S3, GCS), replica count, network policies, and TLS configuration — and the operator renders an embedded Helm chart to produce the corresponding Kubernetes resources (Deployment, Service, PVC, NetworkPolicy, RBAC, ServiceMonitor).

The operator integrates with the RHOAI/ODH platform by creating Gateway API HTTPRoutes that register the MLflow service with the platform's shared data-science gateway, and by creating OpenShift ConsoleLinks for UI integration. It supports multi-tenant workspace isolation via the Kubernetes namespace-based workspace provider, where namespace owners can override artifact storage through namespace-scoped `MLflowConfig` CRDs. The operator also manages CA certificate bundle composition (platform + custom CAs) for TLS verification against external storage backends (PostgreSQL SSL, S3 with custom certificates).

The operator follows the RHOAI 3.x ingress pattern: services expose HTTPS on port 8443 and are fronted by Gateway API HTTPRoutes referencing the platform's `data-science-gateway`. Authentication is handled by the MLflow server's built-in Kubernetes auth middleware (`--app-name=kubernetes-auth`) which performs SelfSubjectAccessReview checks against RBAC pseudo-resources.

## Architectural Analysis

The MLflow Operator follows a distinctive **Helm-in-operator** pattern: rather than using code-generated Kubernetes objects or static manifests, the controller embeds a complete Helm chart (`charts/mlflow/`) and renders it in-process using the Helm v3 library. The `RenderChart` function translates the `MLflow` CR spec into Helm values, runs the Helm template engine, and applies the resulting unstructured objects via Server-Side Apply. This approach gives the operator the flexibility of Helm templates (conditional resources, computed values) while maintaining the reconciliation guarantees of a controller-runtime operator. The tradeoff is that the Helm chart must be embedded in the operator binary at build time (via `COPY --from=builder /workspace/charts/mlflow charts/mlflow` in the Dockerfile), making chart updates inseparable from operator releases.

The operator implements a graceful **feature detection pattern** for optional platform capabilities. At startup, it uses the Kubernetes discovery API to check for the existence of `ConsoleLink` (OpenShift console), `HTTPRoute` (Gateway API), and `ServiceMonitor` (Prometheus) CRDs. If a CRD is absent, the corresponding watch, cache entry, and reconciliation logic are all skipped. This allows the same operator binary to run on vanilla Kubernetes (without OpenShift or Gateway API) with reduced functionality, and on OpenShift/RHOAI with full platform integration.

Security is enforced at multiple layers. The MLflow server itself runs with `--app-name=kubernetes-auth`, which enables the built-in Kubernetes authorization middleware that performs `SelfSubjectAccessReview` checks against RBAC pseudo-resources (`experiments`, `datasets`, `registeredmodels`, etc.) in the `mlflow.kubeflow.org` API group. The `MLFLOW_SERVER_DISABLE_SECURITY_MIDDLEWARE` env var is hardcoded to `false` and blocked from user override via CEL validation rules on the CR spec. Network isolation is enforced by a per-instance `NetworkPolicy` with explicit allowlists for DNS, HTTPS, Kubernetes API, PostgreSQL, MySQL, and S3-compatible storage ports. The operator also enforces a restrictive pod security context: non-root, read-only root filesystem, `ALL` capabilities dropped, and `RuntimeDefault` seccomp profile.

The CA bundle management is notably thorough. The operator checks for the platform-managed `odh-trusted-ca-bundle` ConfigMap and optionally a user-specified CA ConfigMap, then deploys an init container and sidecar container (`ca-bundle-watcher`) that combine system CAs with all mounted ConfigMap certificates into a single PEM file. Multiple environment variables (`SSL_CERT_FILE`, `REQUESTS_CA_BUNDLE`, `CURL_CA_BUNDLE`, `AWS_CA_BUNDLE`, `PGSSLROOTCERT`, `MLFLOW_MYSQL_CA`) are set to point at this combined bundle, ensuring that all Python libraries used by MLflow (requests, boto3, psycopg2, pycurl) use the correct trust store. The sidecar polls for ConfigMap changes every 30 seconds and atomically regenerates the combined bundle.

The RBAC design includes three aggregate ClusterRoles (`mlflow-view`, `mlflow-edit`, `mlflow-integration`) that integrate with Kubernetes' built-in role aggregation system. This allows platform administrators to grant MLflow access through standard `admin`/`edit`/`view` roles without creating MLflow-specific bindings. The `mlflow-integration` role is purpose-built for service accounts of other components (e.g., notebook controllers, pipeline runners) that need programmatic access to MLflow's experiment tracking and AI Gateway features without full administrative privileges.
