## Purpose

**Short**: Kubernetes operator that manages MLflow tracking server deployments on OpenShift AI using a Helm-based reconciliation pattern.

**Detailed**: The MLflow Operator watches a cluster-scoped `MLflow` Custom Resource (`mlflow.opendatahub.io/v1`) and renders a bundled Helm chart to deploy and manage the full MLflow tracking server stack. The operator creates Deployments, Services, ServiceAccounts, ClusterRoles, ClusterRoleBindings, PersistentVolumeClaims, NetworkPolicies, HTTPRoutes (Gateway API), ConsoleLinks (OpenShift), and ServiceMonitors (Prometheus). It uses controller-runtime with Server-Side Apply for declarative resource management.

The operator supports configurable backend stores (SQLite, PostgreSQL, MySQL), artifact storage (local files, S3, GCS), workspace-aware multi-tenancy via the Kubernetes namespace model, and custom CA bundle injection for TLS connections to external services. On OpenShift, it automatically provisions TLS certificates via the service-ca operator and creates ConsoleLink resources for the OpenShift web console. The operator integrates with the RHOAI data-science gateway via Gateway API HTTPRoutes to expose MLflow through the platform's unified ingress.

A secondary CRD (`mlflow.kubeflow.org/v1 MLflowConfig`) provides namespace-scoped artifact storage configuration, allowing individual teams to override the default artifact root with their own S3-compatible bucket credentials.

## Architectural Analysis

The MLflow Operator follows a Helm-in-operator pattern: rather than managing individual Kubernetes resources through hand-crafted Go structs, it embeds a full Helm chart (`charts/mlflow/`) and renders it at reconcile time with values derived from the MLflow CR spec. This approach provides several advantages: the chart can also be deployed standalone for development, the template language handles conditional resource inclusion (e.g., PVC only when `storage.enabled`), and the resource definitions remain maintainable YAML rather than deeply nested Go object construction. The rendered manifests are applied via Server-Side Apply with `ForceOwnership`, giving the operator clean three-way merge semantics. A notable exception is PVCs, which are skipped if they already exist to avoid immutable spec errors.

The security model is unusually thorough for a component operator. The MLflow server itself enforces Kubernetes-native authorization via the `--app-name=kubernetes-auth` flag, which delegates access checks to `SelfSubjectAccessReview` against pseudo-resources in the `mlflow.kubeflow.org` API group (experiments, datasets, registeredmodels, gatewayendpoints, etc.). This means standard Kubernetes RBAC roles control who can read/write MLflow data, and the aggregate ClusterRoles (`mlflow-view`, `mlflow-edit`, `mlflow-integration`) automatically extend the built-in `admin`, `edit`, and `view` roles. The `MLFLOW_SERVER_DISABLE_SECURITY_MIDDLEWARE` env var is hardcoded to `"false"` and the CR-level CEL validation explicitly rejects attempts to set it, preventing accidental security bypass.

The CA bundle handling is sophisticated. When platform or custom CA ConfigMaps are present, the operator renders an init container and a sidecar (`ca-bundle-watcher`) that combine system CAs, platform CAs (`odh-trusted-ca-bundle`), and user-provided CAs into a single PEM file. The sidecar continuously watches for ConfigMap updates (30-second poll interval) and regenerates the combined bundle. Six environment variables (`SSL_CERT_FILE`, `REQUESTS_CA_BUNDLE`, `CURL_CA_BUNDLE`, `AWS_CA_BUNDLE`, `PGSSLROOTCERT`, `MLFLOW_MYSQL_CA`) are set to point to this combined bundle, ensuring all Python HTTP and database libraries use the correct trust store. On OpenShift, `PGSSLMODE=verify-full` is enforced for PostgreSQL connections and `MLFLOW_S3_IGNORE_TLS=false` prevents accidental TLS bypass for S3 storage.

The operator supports multi-instance deployments via a resource suffix mechanism. The CR is constrained to be named `mlflow` by a CEL validation rule, but the code includes a `getResourceSuffix()` function that returns `"-{name}"` for CRs not named `mlflow`, suggesting future multi-instance support is partially scaffolded. HTTPRoutes are created with path rewriting rules (`/mlflow/api/*` to `/api/*`, `/mlflow/v1/*` to `/v1/*`) to integrate cleanly behind the shared RHOAI gateway. The operator uses discovery API checks at startup to conditionally watch CRDs (ConsoleLink, HTTPRoute, ServiceMonitor), degrading gracefully on clusters without OpenShift or Gateway API.

The NetworkPolicy deployed by the Helm chart is restrictive by default: ingress is limited to the MLflow HTTPS port from any namespace (since gateway and Prometheus source namespaces vary), while egress allows DNS, HTTPS/443, Kubernetes API/6443, PostgreSQL/5432, MySQL/3306, and S3-compatible storage ports (9000, 8333, 8334). Users can extend egress via `networkPolicyAdditionalEgressRules` in the CR spec. The operator disables HTTP/2 on its own metrics endpoint (`NextProtos: ["http/1.1"]`) as a defense against HTTP/2 rapid-reset attacks, a pattern common in recent controller-runtime scaffolds.
