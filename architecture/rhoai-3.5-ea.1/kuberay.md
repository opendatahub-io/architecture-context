## Purpose

**Short**: KubeRay is a Kubernetes operator that manages the lifecycle of Ray clusters, Ray jobs, and Ray serving deployments on OpenShift/Kubernetes.

**Detailed**: KubeRay provides three Custom Resource Definitions — RayCluster, RayJob, and RayService — that allow users to declaratively create and manage Ray distributed computing clusters. The operator handles pod lifecycle management for head and worker nodes, automatic scaling, fault tolerance via Redis-backed GCS, and zero-downtime upgrades for serving workloads.

In the RHOAI distribution, KubeRay integrates with the platform's authentication and networking infrastructure. The operator injects `kube-rbac-proxy` sidecars into head pods for OIDC-based authentication, creates Gateway API HTTPRoute resources for secure dashboard access through the platform gateway, manages mTLS certificates via cert-manager for intra-cluster encryption, and deploys NetworkPolicies for pod-level network segmentation. These features are controlled via the `odh.ray.io/secure-trusted-network` annotation, which the mutating webhook automatically sets to `true` on OpenShift clusters.

The operator also integrates with batch scheduling systems (Volcano, Yunikorn, Kubernetes Scheduler Plugins) for gang scheduling of Ray worker pods, and exposes Prometheus metrics for cluster provisioning duration, job execution duration, and cluster metadata.

## Architectural Analysis

The KubeRay operator follows a classic controller-runtime pattern with six reconcilers managing different aspects of Ray cluster lifecycle. The core `RayClusterReconciler` handles pod and service lifecycle, while specialized controllers (`AuthenticationController`, `NetworkPolicyController`, `RayClusterMTLSController`) layer security features via the `odh.ray.io/secure-trusted-network` annotation. This annotation-driven activation pattern allows the same operator binary to run in both upstream Kubernetes (minimal security) and RHOAI (full security stack) environments without code branching.

The authentication architecture in RHOAI 3.x uses a multi-layer approach: the `AuthenticationController` creates Gateway API `HTTPRoute` resources in the **platform namespace** (not the cluster namespace) to route traffic through the RHOAI gateway, combined with `ReferenceGrant` resources that allow cross-namespace service references. The `kube-rbac-proxy` sidecar injected into head pods performs OIDC token validation via Kubernetes `SubjectAccessReview`, meaning authentication is enforced at the pod level rather than at the ingress layer. The mutating webhook enforces `enableIngress=false` on OpenShift to prevent users from creating unprotected Ingress/Route resources that would bypass the authentication stack. Both `IntegratedOAuth` and `OIDC` modes now converge on the same kube-rbac-proxy implementation — the legacy oauth-proxy pattern has been replaced.

The `RayServiceReconciler` implements a sophisticated zero-downtime upgrade mechanism using a pending/active cluster pattern. When the RayService spec changes, the controller creates a new "pending" RayCluster, deploys Serve applications to it, verifies readiness, then atomically switches the Kubernetes Service selector from the active to the pending cluster. This ensures no request drops during upgrades. The serve application config is cached in-memory to avoid redundant submissions to the Ray dashboard API.

The mTLS implementation via `RayClusterMTLSController` dynamically updates certificate SANs as pods are created or deleted, since pod IPs are not known at cluster creation time. This requires watching Pod events and re-reconciling certificates, adding complexity but ensuring that all intra-cluster communication can be authenticated. Certificates use a two-tier hierarchy: a self-signed root CA with 10-year validity, and per-pod certificates with 90-day validity and 15-day early renewal. The `NetworkPolicyController` complements mTLS by restricting ingress to head pods — only same-cluster pods, namespace peers (for client access), the operator pod, optional monitoring namespace, and the gateway namespace are allowed.

The operator's integration with batch schedulers (Volcano, Yunikorn, Scheduler Plugins) is abstracted through a `SchedulerManager` interface that hooks into the pod creation pipeline. Gang scheduling ensures all Ray worker pods are scheduled atomically, preventing resource deadlocks in multi-tenant clusters. The `ScaleExpectation` mechanism tracks pending pod creations to prevent the reconciler from racing with the Kubernetes scheduler.
