## Purpose

**Short**: Manages the lifecycle of Ray clusters, jobs, and services on Kubernetes and OpenShift.

**Detailed**: KubeRay is a Kubernetes operator that automates the deployment, scaling, and management of Ray clusters for distributed computing workloads. It provides three Custom Resource Definitions — RayCluster, RayJob, and RayService — enabling users to declaratively manage Ray infrastructure including head nodes, worker groups, autoscaling, and fault tolerance via GCS with external Redis storage.

On RHOAI (Red Hat OpenShift AI), the operator includes significant OpenShift-specific enhancements: Gateway API integration via HTTPRoute for centralized ingress, kube-rbac-proxy sidecar injection for OIDC-based authentication, cert-manager integration for mTLS between Ray nodes, NetworkPolicy enforcement for pod-level network isolation, SecurityContextConstraints for restricted pod execution, and webhook-based enforcement of secure defaults. The operator automatically detects OpenShift environments and applies these security hardening measures transparently.

The operator also integrates with batch scheduling systems (Volcano, YuniKorn, Kubernetes Scheduler Plugins) for gang scheduling support, and provides comprehensive Prometheus metrics for cluster provisioning, job execution, and service readiness monitoring.

## Architectural Analysis

The KubeRay operator follows a multi-controller architecture where each controller is responsible for a distinct aspect of the Ray cluster lifecycle. The main RayCluster controller orchestrates pod and service creation, while specialized controllers handle cross-cutting concerns: the Authentication controller manages HTTPRoute and auth sidecar injection, the mTLS controller manages cert-manager resources for inter-node encryption, and the NetworkPolicy controller creates pod-level network isolation. This separation of concerns enables each security feature to be independently enabled via annotations (e.g., `ray.io/enable-secure-trusted-network=true`).

A notable architectural decision is the condition-based coordination between controllers. The RayCluster controller checks the `AuthenticationReady` condition (set by the Authentication controller) before creating head pods with auth sidecars. This prevents a race condition where pods could be created before authentication resources (ServiceAccount, HTTPRoute, ConfigMap) are ready. The feature gate `RayClusterStatusConditions` (beta, default true) controls this behavior, allowing graceful rollout of the condition-based flow.

The RHOAI overlay introduces a significant security posture change through the mutating webhook. On OpenShift, the webhook automatically sets `ray.io/enable-secure-trusted-network=true` and forces `enableIngress=false` on every RayCluster creation/update. This ensures that all clusters use Gateway API access (via HTTPRoute + kube-rbac-proxy) rather than direct OpenShift Routes, enforcing centralized authentication through the platform's data-science-gateway. The webhook literally overrides user-specified `enableIngress: true` to `false`, making Gateway-only access non-negotiable on OpenShift.

The operator's ingress architecture on RHOAI 3.x creates HTTPRoutes in the platform namespace (not the cluster namespace) with path-based routing: `/ray/{namespace}/{cluster}` is rewritten to `/` before proxying to the kube-rbac-proxy sidecar on port 8443. A ReferenceGrant in each target namespace authorizes cross-namespace service references. This is a shared model — one ReferenceGrant named `kuberay-gateway-access` serves all clusters in a namespace, with reference counting during cleanup to avoid premature deletion.

The batch scheduler integration uses a clean factory pattern supporting three scheduler backends (Volcano, YuniKorn, Scheduler Plugins) through a common interface. Each implementation translates Ray cluster topology into scheduler-specific constructs (PodGroup for Volcano/Scheduler Plugins, task group annotations for YuniKorn). The `managedBy` field on RayCluster and RayJob enables delegation to external controllers like Kueue for multi-cluster workload management, making KubeRay composable with the broader Kubernetes scheduling ecosystem.
