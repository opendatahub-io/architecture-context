## Purpose

**Short**: Out-of-tree Llama Stack Eval API provider that enables LLM red-teaming and vulnerability scanning using the NVIDIA Garak framework, with support for inline execution, Kubeflow Pipeline (KFP) remote execution, and eval-hub K8s Job execution.

**Detailed**: This component implements Meta's Llama Stack `Eval` API to provide automated LLM security assessments using NVIDIA's Garak red-teaming framework (v0.14.1+rhaiv.8). It allows users to register benchmarks based on compliance frameworks (OWASP LLM Top 10, AVID, CWE, Quality) or intents-based risk assessments, then execute vulnerability scans against models served via OpenAI-compatible endpoints.

The provider supports three execution modes: (1) **Inline** -- runs Garak as a local subprocess with concurrent job management; (2) **Remote (KFP)** -- submits scans as Kubeflow Pipeline DAGs with a 6-step workflow (validate, resolve taxonomy, SDG generate, prepare prompts, scan, parse results); and (3) **eval-hub** -- integrates as a `FrameworkAdapter` for execution within the eval-hub sidecar pattern on Kubernetes Jobs.

Key capabilities include shield scanning (testing Llama Stack guardrails by routing probes through input/output shields before/after LLM inference), Synthetic Data Generation (SDG) using the sdg-hub library to generate adversarial prompts from harm taxonomies, Automated Red Teaming (ART) with escalating probe pipelines (direct prompts, DAN jailbreaks, multilingual translation, TAP attacks), and Context-Aware Scanning (CAS) using Garak's intent-based probe system. Results are reported using the AVID vulnerability format and TBSA (Tier-Based Security Aggregate) scoring.

## Architectural Analysis

### Execution Mode Architecture

The provider implements a strategy pattern with three distinct execution backends behind a common `GarakEvalBase` interface:

1. **Inline mode** (`GarakInlineEvalAdapter`): Runs Garak directly as a subprocess using `asyncio.create_subprocess_exec`. A `Semaphore(max_concurrent_jobs)` gates concurrency. Output files (report.jsonl, avid.jsonl, hitlog.jsonl, report.html) are uploaded to the Llama Stack Files API. Best for development and small-scale testing.

2. **Remote/KFP mode** (`GarakRemoteEvalAdapter`): Compiles scan configuration into a KFP pipeline DAG (`garak_scan_pipeline`) and submits it via `kfp.Client`. The 6-step pipeline runs entirely inside the garak provider container image: validate -> resolve_taxonomy -> sdg_generate -> prepare_prompts -> garak_scan -> parse_results. The parse_results step uploads artifacts back to the Llama Stack Files API. Pipeline pod image resolution follows a 3-tier fallback: explicit config -> K8s ConfigMap (`trustyai-service-operator-config`, key `garak-provider-image`) -> hardcoded default.

3. **eval-hub mode** (`GarakAdapter`): Implements the eval-hub `FrameworkAdapter` interface for execution as a K8s Job with sidecar communication. Supports both `simple` (subprocess) and `kfp` (pipeline) inner execution modes. Adds OCI artifact persistence and MLflow metric logging. S3 Data Connection credentials are read from environment variables injected by the DSPA operator.

### Shield Scanning Design

The `SimpleShieldOrchestrator` implements a proxy pattern where Garak's `function.Single` generator type is used to route every probe through the Llama Stack Safety API. The flow is: (1) run input shields on the probe prompt, (2) if no violation, forward to LLM via chat completions, (3) run output shields on the LLM response, (4) return violation message or LLM response. This tests guardrail effectiveness by checking if shields catch adversarial inputs/outputs.

### ART Intents Pipeline

The intents-based benchmark (`trustyai_garak::intents`) implements an escalating attack pipeline with increasing complexity:
- **SPOIntent**: Direct adversarial prompts (with DAN jailbreak augmentation)
- **SPOIntentUserAugmented / SystemAugmented / BothAugmented**: System/user prompt manipulation
- **TranslationIntent**: Multilingual translation attacks (default: Chinese)
- **TAPIntent**: Tree of Attacks with Pruning -- uses an attack model and evaluator model for adaptive jailbreaking

The judge detector (`MulticlassJudge`) classifies responses as complied/rejected/alternative/other with confidence scoring. Results generate heatmap and funnel chart visualizations in ART HTML reports.

### Configuration Merging

Benchmark configuration uses a 3-layer deep-merge precedence: benchmark_config (user-supplied at run_eval time) > profile defaults (from FRAMEWORK_PROFILES/SCAN_PROFILES) > provider defaults. The `deep_merge_dicts` function performs recursive merging, allowing users to override specific nested fields without replacing entire configuration blocks.

### Compatibility Layer

The `compat.py` module provides forward/backward compatibility for the Llama Stack API migration from `llama_stack` to `llama_stack_api` package. It also patches the `Job` model to allow extra fields, enabling the provider to attach metadata to job responses.

### Container Image Design

The Konflux Dockerfile uses `${BASE_IMAGE}` ARG (UBI9 base), installs the `[inline]` extra (which includes garak and sdg-hub), and sets XDG environment variables to `/tmp` subdirectories for container writability.
