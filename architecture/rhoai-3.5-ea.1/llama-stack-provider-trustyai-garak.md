## Purpose

**Short**: Out-of-tree Llama Stack evaluation provider and eval-hub adapter that performs automated LLM red-teaming and vulnerability scanning using NVIDIA Garak.

**Detailed**: This component provides automated vulnerability assessment, red-teaming, and compliance scanning for Large Language Models deployed on Red Hat OpenShift AI. It integrates NVIDIA's Garak red-teaming framework with two completely independent orchestration surfaces: the Llama Stack provider API and the RHOAI eval-hub platform.

The Llama Stack provider registers as an `Eval` API provider (both inline and remote variants) and exposes benchmark registration, scan execution, and result retrieval through the standard Llama Stack client API. The eval-hub adapter implements the `FrameworkAdapter` interface from the eval-hub SDK, reading job specifications from mounted ConfigMaps and reporting results back via sidecar callbacks.

Both integration surfaces share a common core of garak configuration resolution, command building, scan execution, and result parsing logic. The component supports multiple execution modes: inline (garak runs locally in the Llama Stack server process), simple (garak runs as a subprocess inside a K8s Job pod), and KFP (garak scans are orchestrated as six-step Kubeflow Pipelines). The KFP mode enables the "intents" feature — a sophisticated red-teaming workflow that generates synthetic adversarial prompts via SDG (Synthetic Data Generation), then tests the target model with probes of increasing complexity including tree-of-attacks with persuasion (TAPIntent).

## Architectural Analysis

This component exhibits a clean separation of concerns across two independent integration surfaces that share a common core. The Llama Stack provider (`inline/`, `remote/`) and the eval-hub adapter (`evalhub/`) have no import-time dependencies on each other — the `evalhub/` module does not import from `base_eval.py` or any Llama Stack types. Instead, both surfaces delegate to shared modules in `core/` for pipeline step logic, `garak_command_config.py` for configuration models, `result_utils.py` for report parsing, and `intents.py`/`sdg.py` for the intents workflow. This design allows the eval-hub adapter to function entirely without Llama Stack installed.

The configuration resolution follows a layered deep-merge pattern: predefined benchmark profiles (OWASP LLM Top 10, AVID taxonomy, CWE, intents) provide sensible defaults, user overrides are deep-merged on top, and the `intents_models` shorthand provides a simplified API for configuring the four model roles (judge, attacker, evaluator, SDG). API keys are never embedded in configuration — they use a placeholder system (`__FROM_ENV__`, `DUMMY`, `***`) that gets resolved at runtime via `resolve_api_key()`, which checks role-specific environment variables, volume-mounted secret files at `/mnt/model-auth`, and generic fallbacks. All configurations are redacted before persistence to S3, OCI, or Files API.

The KFP pipeline architecture mirrors across both integration surfaces with a six-step linear DAG: validate → resolve_taxonomy → sdg_generate → prepare_prompts → garak_scan → parse_results/write_kfp_outputs. The EvalHub KFP variant uses S3 (via Data Connection secrets) for artifact transfer, while the Llama Stack KFP variant uses the Files API. The pipeline supports three intents modes (default taxonomy + SDG, custom taxonomy + SDG, bypass SDG with pre-generated prompts) with mutual exclusivity validation between `policy_file_id`/`policy_s3_key` and `intents_file_id`/`intents_s3_key`.

The Rust `Cargo.toml` at the repository root exists solely for Konflux hermetic build requirements — it defines a "dummy-project" that forces pre-compilation of native Python extension dependencies (fancy-regex, pyo3, regex) that garak's transitive dependencies require. The `.konflux/main.rs` contains only `fn main() {}`. The `rpms.in.yaml` installs `rust`, `cargo`, `gcc`, and `ninja-build` as build-time OS packages to enable this native extension compilation.

The shield scanning subsystem (`shield_scan.py`) implements a `SimpleShieldOrchestrator` that intercepts garak probe prompts, runs them through Llama Stack input shields, passes surviving prompts to the target LLM, then runs responses through output shields. This is registered as a garak `function.Single` generator, allowing shield effectiveness testing without modifying garak's core. The orchestrator uses httpx with configurable TLS verification and supports custom CA certificate paths.
