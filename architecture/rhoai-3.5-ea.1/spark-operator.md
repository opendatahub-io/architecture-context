## Purpose

**Short**: Kubernetes operator that automates the submission, scheduling, monitoring, and lifecycle management of Apache Spark applications on Kubernetes clusters.

**Detailed**: The Spark Operator (originally from the Kubeflow project) manages Apache Spark workloads on Kubernetes through three Custom Resource Definitions: `SparkApplication` for one-time Spark job submissions, `ScheduledSparkApplication` for cron-scheduled recurring Spark jobs, and `SparkConnect` (alpha) for persistent Spark Connect server sessions. The operator handles the complete lifecycle of Spark applications including spark-submit execution, driver/executor pod monitoring, web UI service creation, Prometheus metrics integration, and automatic cleanup via TTL.

The operator is deployed as two separate components in RHOAI: a **controller** deployment that runs three reconciliation loops (SparkApplication, ScheduledSparkApplication, SparkConnect) and a **webhook** deployment that provides admission control for validating and defaulting Spark resources. It integrates with multiple batch schedulers (Volcano, YuniKorn, kube-scheduler plugins) for gang scheduling and supports dynamic executor allocation. In the RHOAI distribution, it deploys into the `redhat-ods-applications` namespace with network policies restricting inter-pod communication to Spark driver/executor traffic patterns.

## Architectural Analysis

The Spark Operator follows a clean split-deployment architecture where the **controller** and **webhook** run as separate Kubernetes Deployments. This is a deliberate design choice: the webhook needs to be highly available and fast (admission requests have a 10-second timeout), while the controller can tolerate brief downtime during leader election transitions. Both components share the same container image (`spark-operator` binary) but are started with different subcommands (`controller start` vs `webhook start`). The entrypoint script (`entrypoint.sh`) handles OpenShift's random UID assignment through `libnss_wrapper.so` and delegates to `tini` as PID 1 for proper signal handling.

The controller implements a sophisticated **state machine** for SparkApplication lifecycle management, with 14 distinct states (New → Submitted → Running → Succeeding → Completed, with branches for failure, suspension, invalidation, and pending-rerun). State transitions trigger resource creation/deletion: web UI services and ingresses are created when the application starts running, and cleaned up on termination. The `ScheduledSparkApplication` controller adds cron scheduling with three concurrency policies (Allow, Forbid, Replace) and history management that tracks past successful/failed run names. The `SparkConnect` controller (v1alpha1, alpha quality) manages a persistent Spark Connect server with a service exposing four ports (driver-rpc/7078, blockmanager/7079, web-ui/4040, spark-connect/15002).

The **batch scheduler integration** is particularly well-designed, using a registry pattern with a common `Interface` (Name, ShouldSchedule, Schedule, Cleanup). Three backends are supported: Volcano (creates PodGroup CRs with minMember/minResources), YuniKorn (annotates pods with task group JSON for queue-based scheduling), and kube-scheduler plugins (creates scheduling.x-k8s.io PodGroups). The scheduler is invoked before spark-submit and cleaned up after application termination. This design allows adding new scheduler backends without modifying controller logic.

A notable **FIPS compliance gap** exists: the operator binary is built with `CGO_ENABLED=0` (static linking), which means it uses Go's pure-Go crypto implementation rather than linking against OpenSSL. This will fail the check-payload gate in Konflux FIPS validation. The `crypto/md5` import in `pkg/util/sparkapplication.go` (used for content hashing) is another non-FIPS-approved algorithm. The webhook's TLS implementation disables HTTP/2 by default as a mitigation for CVE-2023-44487 (HTTP/2 Rapid Reset), and uses RSA 2048-bit self-signed certificates with a generous 10-year validity period. Certificate auto-renewal triggers when the cert would be invalid 180 days in the future.

The **network policy** in the RHOAI overlay is minimal but targeted: it allows inbound TCP traffic on ports 7078 (driver-rpc), 7079 (blockmanager), and 4040 (spark-ui) to any pod with the `sparkoperator.k8s.io/launched-by-spark-operator=true` label, from either same-namespace pods or OpenShift ingress namespace pods. This permits Spark driver-executor communication and optional external Spark UI access while restricting arbitrary inbound traffic to Spark pods.
