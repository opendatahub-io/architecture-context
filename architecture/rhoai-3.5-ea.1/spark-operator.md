## Purpose

**Short**: Kubernetes operator that manages Apache Spark application lifecycle, scheduled Spark jobs, and Spark Connect server sessions on OpenShift/Kubernetes clusters.

**Detailed**: The Spark Operator (originally Kubeflow Spark Operator) automates the submission, monitoring, and management of Apache Spark applications on Kubernetes. It watches for SparkApplication, ScheduledSparkApplication, and SparkConnect custom resources, translating declarative specifications into Spark driver/executor pod orchestration via `spark-submit`.

The operator runs as two separate deployment processes from a single binary: a **controller** process that reconciles CRDs and manages Spark job lifecycle (submission, monitoring, restart, cleanup), and a **webhook** process that validates and mutates SparkApplication resources and Spark pods at admission time. The webhook injects environment variables, volumes, sidecar containers, scheduling annotations, and Prometheus monitoring configuration into Spark pods. It also enforces optional ResourceQuota validation for Spark workloads.

The operator integrates with three batch schedulers (Volcano, Yunikorn, and kube-scheduler via scheduler-plugins) for gang scheduling of Spark applications, and provides Prometheus metrics for application state tracking and executor lifecycle monitoring. For RHOAI, it deploys into the `redhat-ods-applications` namespace with NetworkPolicy isolation for Spark pods and PodMonitor-based metrics collection.

## Architectural Analysis

The Spark Operator follows the Kubeflow convention of splitting the operator binary into two processes: a **controller** deployment and a **webhook** deployment. Both share the same Go binary (`/usr/bin/spark-operator`) invoked with different subcommands (`controller start` vs `webhook start`). This separation allows independent scaling and failure isolation -- the webhook must remain available for pod admission even if the controller is restarting, and vice versa. The controller uses leader election via ConfigMap-based leases to ensure only one active reconciler, while the webhook deployment handles its own leader election independently.

The operator's **unique architectural pattern** is its use of `spark-submit` as a local process execution rather than creating driver pods directly via the Kubernetes API. The controller invokes the `spark-submit` binary (bundled in the container image alongside a full PySpark 4.0.1 installation and Java 17 runtime) to submit Spark applications. This means the operator container is significantly larger than typical Go operators because it carries the entire Spark runtime. The driver and executor pods are then created by Spark's native Kubernetes scheduler integration, with the operator's mutating webhook intercepting pod creation to inject operator-managed configuration (volumes, env vars, scheduling annotations, Prometheus exporters, security contexts). This webhook-based mutation is critical -- without it, Spark pods would lack the configuration specified in the SparkApplication CRD.

The **SparkConnect controller** (v1alpha1) represents a newer pattern where the operator directly creates server pods and services rather than delegating to spark-submit. The SparkConnect server exposes four ports (driver-rpc/7078, blockmanager/7079, web-ui/4040, spark-connect/15002) and manages executor pods dynamically. This pattern bypasses the spark-submit flow entirely, creating pods and services through the Kubernetes API with owner references for garbage collection.

**FIPS compliance is a notable gap**: the Dockerfile.konflux builds with `CGO_ENABLED=0`, producing a statically-linked binary that uses Go's pure-Go crypto implementation rather than OpenSSL/BoringCrypto. This would fail the check-payload FIPS validation gate. The certificate management code generates RSA 2048-bit self-signed certificates with 10-year validity and auto-rotation at 180 days before expiry, which is functional but uses the non-FIPS Go crypto stack. The HTTP/2 disabling (defaulting to HTTP/1.1 only) is a deliberate CVE mitigation (GHSA-qppj-fm5r-hxr3, GHSA-4374-p667-p6c8) rather than a FIPS requirement.

The **batch scheduler integration** is well-architected with a pluggable registry pattern. Three schedulers are supported: Volcano (PodGroup CRD with queue/priority), Yunikorn (annotation-based gang scheduling with task groups), and kube-scheduler via scheduler-plugins (PodGroup CRD). Each scheduler implements a common `Interface` with `Schedule()` and `Cleanup()` methods. The Volcano integration notably supports both cluster mode (driver+executors in one PodGroup) and client mode (executor-only PodGroup), reflecting Spark's different deployment modes.
