## Purpose

**Short**: An MCP (Model Context Protocol) server that enables AI agents to interact with Red Hat OpenShift AI environments through programmatic tools, resources, and workflow prompts.

**Detailed**: RHOAI MCP Server is a Python-based MCP server built on the FastMCP framework that exposes the capabilities of the OpenShift AI Dashboard as programmatic tools for AI agents. It provides a plugin-based architecture where domain modules (projects, notebooks, inference, pipelines, connections, storage, training, model registry, quickstarts) each contribute MCP tools, resources, and prompts that are registered at startup via a pluggy hook system.

The server acts as a bridge between AI agents (such as Claude Desktop or Claude Code) and a Kubernetes/OpenShift cluster running Red Hat OpenShift AI. It uses the Kubernetes Python client with dynamic resource discovery to interact with both core Kubernetes resources (Namespaces, Secrets, PVCs, Pods) and RHOAI-specific Custom Resources (Notebooks, InferenceServices, DataSciencePipelinesApplications, TrainJobs, etc.). The server supports three transport modes (stdio for Claude Desktop, SSE and streamable-http for network access), multiple authentication modes (auto, kubeconfig, token), and includes safety features like read-only mode, dangerous operation gating, and credential masking.

Composite modules layer on top of domains to provide cross-cutting tools such as cluster exploration, training workflow orchestration, tool discovery/recommendation, and Neural Navigator model recommendation integration. The server also provides 18 MCP prompts that guide AI agents through multi-step workflows including model training, deployment, troubleshooting, and project setup.

## Architectural Analysis

The RHOAI MCP Server follows a clean plugin-based architecture that separates concerns into three layers: domain modules providing CRUD operations for specific Kubernetes resource types, composite modules that orchestrate multiple domains for complex workflows, and the core server infrastructure that handles MCP transport, authentication, and plugin lifecycle. The pluggy hook system enables loose coupling — each plugin declares its metadata, required CRDs, and health check logic independently, and plugins that fail health checks are gracefully degraded rather than preventing server startup. This design allows the server to operate in clusters where only a subset of RHOAI components are installed.

The server's approach to Kubernetes API interaction is notable for its use of the dynamic client with CRD caching. Rather than using generated typed clients for each CRD, the `K8sClient` class uses `kubernetes.dynamic.DynamicClient` with a `search()` + cache pattern that avoids repeated API discovery calls. This makes the server resilient to CRD version changes and allows it to work with any version of the RHOAI CRDs without code changes. However, this means type safety for CRD field access relies on runtime attribute access rather than compile-time checking.

The response formatting system is designed specifically for AI agent context window optimization. Every tool response includes a `_source` metadata dict that grounds the response to a specific Kubernetes resource (kind, apiVersion, name, namespace, uid), enabling agents to verify information provenance. The three verbosity levels (minimal, standard, full) allow agents to request token-efficient responses for status checks (~85% token savings with minimal) while still being able to drill into full detail when needed. Pagination is applied uniformly across all list operations with configurable limits.

The safety model implements defense-in-depth with three layers: (1) a global `read_only_mode` that blocks all write operations, (2) a `enable_dangerous_operations` gate specifically for delete operations, and (3) per-tool `confirm=True` parameters that require explicit confirmation for destructive actions. Credential masking is applied at the model layer — the `DataConnection.from_secret()` factory method masks AWS secret access keys (showing only first 4 + last 4 characters) before the data ever reaches the tool response. This prevents accidental credential exposure through AI agent conversations.

The Model Registry integration demonstrates a sophisticated service discovery pattern. The `ModelRegistryDiscovery` class first queries the `ModelRegistry` component CRD to find the registries namespace, then scans common namespace patterns, then finds matching services with port preference (8080 for direct REST > 8443 for kube-rbac-proxy > 443 for HTTPS). It also includes API type detection that probes endpoints to distinguish between the newer Model Catalog API and the standard Kubeflow Model Registry API. For outside-cluster access, the `PortForwardManager` creates and manages `oc port-forward` subprocesses with reference counting to share tunnels across multiple concurrent requests.

The Neural Navigator composite module integrates with an external ML recommendation engine to provide GPU-aware model recommendations. It translates natural language use case descriptions into structured deployment intents, then uses SLO-based ranking to recommend model configurations optimized for performance, cost, or balance. This is architecturally significant because it adds an AI-to-AI interaction pattern — an AI agent uses the MCP server which calls another AI service to get recommendations, creating a layered intelligence stack.
