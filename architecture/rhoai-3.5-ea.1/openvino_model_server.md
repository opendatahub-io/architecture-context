## Purpose

**Short**: OpenVINO Model Server (OVMS) is a high-performance C++ inference serving engine that hosts AI models and exposes them via gRPC and REST APIs compatible with TensorFlow Serving, KServe, and OpenAI protocols.

**Detailed**: OVMS provides a scalable model serving platform optimized for Intel architectures. It supports serving traditional ML/DL models (OpenVINO IR, ONNX, TensorFlow SavedModel, PaddlePaddle, TFLite) as well as large language models (LLMs) through continuous batching pipelines. The server implements three API families: TensorFlow Serving API (v1), KServe Predict Protocol (v2), and an OpenAI-compatible API (v3) supporting chat completions, text completions, embeddings, reranking, tokenization, image generation, and audio (speech-to-text, text-to-speech).

Within RHOAI, OVMS serves as the inference runtime for the OpenVINO ServingRuntime in KServe. It is deployed as a container within InferenceService pods, receiving model artifacts from KServe's storage initializer and serving predictions through the KServe data plane. The RHOAI build is compiled from source against UBI9 base images using Bazel, with OpenVINO, OpenVINO Tokenizers, and OpenVINO GenAI compiled from the `opendatahub-io` fork of the upstream repositories. The server also includes a MediaPipe graph execution engine for DAG-style pipelines and supports Python node execution within those graphs.

## Architectural Analysis

OpenVINO Model Server is a monolithic C++ inference server with a modular internal architecture. The `Server` class (singleton) orchestrates module lifecycle with a deliberate initialization order: the gRPC and HTTP servers start accepting connections before the model manager finishes loading all models. This design enables fast liveness probe responses while models load in the background — critical for Kubernetes health check deadlines.

The server implements three API families through a unified request routing layer. The TensorFlow Serving (v1) and KServe (v2) APIs handle traditional tensor-in/tensor-out inference, while the OpenAI-compatible (v3) API routes through MediaPipe graph calculators to support generative AI workloads. The v3 routing is particularly interesting: the `HttpRestApiHandler` dispatches all `/v3/*` requests to a shared `processV3()` method, which creates MediaPipe graph instances. Different calculator implementations handle specific endpoint types — `HttpLLMCalculator` for completions, `EmbeddingsCalculator` for embeddings, `RerankCalculator` for reranking, `S2TCalculator` and `T2SCalculator` for audio, and `HttpImageGenCalculator` for image generation. This graph-based architecture allows composing complex pipelines (e.g., tokenize → infer → detokenize) without modifying core server code.

The LLM pipeline architecture deserves special attention. OVMS supports four pipeline types: LM (legacy single-request), VLM (vision-language), LM_CB (continuous batching), and VLM_CB (VLM with continuous batching). The continuous batching variants use OpenVINO GenAI's `ContinuousBatchingPipeline`, which dynamically schedules multiple inference requests across a shared KV cache. This is the recommended production configuration for LLM serving, as it maximizes GPU/CPU utilization. The server also supports speculative decoding via draft models and EAGLE3 mode for accelerated generation.

A notable security consideration is that OVMS provides no built-in TLS. The gRPC server uses `InsecureServerCredentials()` and the Drogon HTTP server listens on plaintext HTTP. TLS termination is expected to happen at the ingress layer (KServe Gateway, nginx-mtls-auth sidecar, or Istio sidecar proxy). The `extras/nginx-mtls-auth/` directory provides an optional nginx sidecar configuration that adds mutual TLS with client certificate verification using TLS 1.2 with restricted cipher suites. For RHOAI deployments, TLS is typically handled by the platform's Gateway/Envoy ingress stack with kube-rbac-proxy authentication sidecars.

The build system is Bazel-based with significant complexity. The RHOAI Konflux build compiles OpenVINO, OpenVINO Tokenizers, and OpenVINO GenAI from source (using the `opendatahub-io` fork), rather than using pre-built binaries. Build flags include LTO (Link-Time Optimization) with `-flto=auto` and architecture targeting (`-march=haswell`). The multi-stage Dockerfile uses four stages: `base_build` (dependency compilation), `build` (OVMS + OpenVINO compilation), `capi-build` (C API shared library), and `release` (minimal UBI9-minimal runtime image). The release image runs as non-root user `ovms` (UID 5000).
