## Purpose

**Short**: OpenVINO Model Server (OVMS) is a high-performance C++ inference server that serves AI models over gRPC and REST APIs using the OpenVINO runtime, supporting TensorFlow Serving, KServe v2, and OpenAI-compatible protocols.

**Detailed**: OVMS provides a scalable model serving infrastructure optimized for Intel architectures. It loads AI models from local storage or cloud object stores (S3, Azure Blob, Google Cloud Storage, HuggingFace Hub) and exposes them for inference via multiple API protocols. The server supports classical vision/NLP models via TensorFlow Serving and KServe v2 APIs, as well as large language models (LLMs) via OpenAI-compatible chat/completion APIs with continuous batching and streaming.

The server is built on the OpenVINO inference runtime and includes support for MediaPipe graph-based serving pipelines, enabling complex multi-model workflows such as DAG (Directed Acyclic Graph) scheduling, custom node pipelines, and specialized AI capabilities including text embeddings, reranking, image generation, speech-to-text, and text-to-speech. The architecture uses a modular design with separate modules for gRPC serving, HTTP/REST serving (via the Drogon framework), model management, metrics collection, Python interpreter integration, and HuggingFace model pulling. In the RHOAI distribution, OVMS is deployed as a serving runtime container for KServe InferenceService workloads, providing the inference backend for data science model deployments.

## Architectural Analysis

OVMS follows a highly modular C++ architecture centered on a singleton `Server` class that orchestrates independently-startable `Module` instances. The module lifecycle (NOT_INITIALIZED → STARTED_INITIALIZE → INITIALIZED → STARTED_SHUTDOWN → SHUTDOWN) provides clean startup ordering and graceful shutdown. Nine modules are defined: Profiler, gRPC Server, HTTP Server, Servable Manager, HuggingFace Pull, Metrics, Python Interpreter, C API, and Servables Config Manager. This modular design allows selective feature enablement — for instance, the gRPC server is entirely skipped when `--port=0`, and the Python interpreter is disabled via build-time Bazel flags (`--config=mp_on_py_off`).

The serving architecture supports three distinct API families through a unified backend. TensorFlow Serving v1 compatibility (`/v1/models/...`) is implemented via `tfs_frontend/`, KServe v2 inference protocol (`/v2/models/...`) via `kfs_frontend/`, and OpenAI/Cohere-compatible generative AI APIs (`/v3/...`) via the MediaPipe graph execution engine with specialized calculator nodes (LLM, embeddings, rerank, image generation, speech-to-text, text-to-speech). This three-tier API design means OVMS can serve traditional ML models and LLMs simultaneously through the same server instance, which is architecturally significant for mixed-workload deployments.

The HTTP server uses the Drogon framework with a deliberate threading split: unary requests (standard inference) use Drogon's internal worker threads, while streaming requests (LLM text generation) are dispatched to a separate MediaPipe ThreadPool. This prevents slow streaming responses from starving the unary request pipeline. The gRPC server uses standard grpc++ with InsecureServerCredentials and configurable resource quotas (max threads, memory quota up to 2GB default). Both gRPC and HTTP support multiple bind addresses (comma-separated) and IPv6. Notably, OVMS provides NO built-in TLS — it relies entirely on external infrastructure (KServe Istio sidecar, kube-rbac-proxy, or nginx reverse proxy) for transport encryption, which is the standard pattern in the RHOAI platform.

The storage abstraction layer (`FileSystem` base class with `S3FileSystem`, `AzureFileSystem`, `GCSFileSystem`, `LocalFileSystem` implementations) provides uniform model loading from heterogeneous backends. The `HfPullModelModule` adds HuggingFace Hub support via libgit2 for Git LFS and a custom GGUF downloader for quantized model formats. Model versioning is handled by directory-based version numbering, with configurable version policies and runtime model reloading. For LLM workloads, the `GenAiServable` class implements a three-phase execution model (request parsing → input preparation with Jinja2 chat templates → execution scheduling with continuous batching), supporting both unary and streaming responses via SSE. The continuous batching implementation leverages OpenVINO GenAI's scheduling to maximize GPU/CPU utilization across concurrent requests.

From a security perspective, the API key authentication mechanism (`--api_key_file`) is a simple Bearer token check that protects REST endpoints but not gRPC endpoints. The nginx mTLS proxy pattern (available in `extras/nginx-mtls-auth/`) provides an optional TLS termination layer but is not included in the Konflux-built RHOAI image. In the RHOAI deployment model, authentication is enforced by the platform's kube-rbac-proxy sidecar and Gateway API infrastructure, making OVMS's lack of built-in TLS architecturally appropriate. The use of OpenSSL's MD5 for model file checksums is non-cryptographic and does not represent a FIPS compliance concern, though it could be replaced with SHA-256 for defense-in-depth.
