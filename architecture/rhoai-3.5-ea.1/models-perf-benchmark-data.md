## Purpose

**Short**: Provides a container image packaging pre-processed model performance benchmark data for the RHOAI Model Catalog, along with a Kafka processor that extracts and transforms that data from upstream Kafka topics.

**Detailed**: The `models-perf-benchmark-data` repository produces a minimal data container image built on UBI9 that bundles validated AI/ML model performance benchmark data (evaluation scores, performance metrics, validation results, and metadata) into a static filesystem at `/app/benchmarks`. This image is deployed as an init container in the `model-catalog` Deployment within the `odh-model-registries` namespace, where it seeds the Model Catalog with Red Hat-validated model performance data.

The repository also contains a `kafka-processor` tool — a Python CLI application that consumes CDC (Change Data Capture) messages from Kafka topics, processes them through an entity classification pipeline, applies post-processing transforms (e.g., computing overall accuracy from evaluation scores, combining hardware configuration fields), and writes the results as structured JSON/NDJSON files organized in a `provider/model` directory hierarchy. This processor is used offline during data preparation to pull benchmark data from internal Kafka clusters and produce the `data/` directory that is then baked into the container image.

The data itself is confidential (the repository must remain private) and includes performance benchmarks, evaluation scores, and validation status for AI models that Red Hat has tested against specific RHOAI and vLLM versions on specific hardware configurations.

## Architectural Analysis

The `models-perf-benchmark-data` component follows a data-as-container pattern common in Kubernetes: benchmark data is baked into a minimal container image at build time and delivered to the consuming application (model-catalog) at runtime via an init container. This pattern provides immutable, versioned data snapshots with the same lifecycle management (image tags, rollbacks, vulnerability scanning) as application code. The image is intentionally minimal — UBI9-minimal base with no package manager, no shell beyond what UBI provides, and no running processes. The `CMD ["sleep", "infinity"]` is a placeholder that is never executed in production; Kubernetes runs the init container's copy operation and then terminates it.

The Kafka processor (`kafka-processor/`) is architecturally separate from the shipped container — it is a data preparation pipeline that runs offline (locally or in CI/CD) to produce the `data/` directory. The pipeline implements a CDC (Change Data Capture) envelope parser that supports both a modern simple format and a legacy Debezium format, an entity type classification system that routes messages to the correct output files based on topic name and field heuristics, and a post-processing transform framework. The two built-in transforms — `OverallAccuracyTransform` (computes mean evaluation score across all benchmarks) and `HardwareConfigurationTransform` (combines hardware type and count into a human-readable string) — demonstrate a clean extensibility pattern with abstract base class, target file routing, and per-record vs. per-model granularity.

The entity processing pipeline organizes data into a `provider/model` filesystem hierarchy using HuggingFace repository names (e.g., `meta-llama/Llama-3.1-8B-Instruct`) which naturally creates a two-level directory structure. Entity types include model metadata (single JSON), performance metrics (append-only NDJSON), evaluations (append-only NDJSON), validations (merged into metadata.json), providers (single JSON per provider), and variant groups (top-level NDJSON). The processor supports CDC operations including CREATE, UPDATE, and DELETE, with proper idempotent deduplication by record ID.

A notable security consideration is that the repository is explicitly marked as confidential — it contains proprietary benchmark data for Red Hat-validated AI models. The repository must remain private, and the container image should be distributed only through authorized registries. The Kafka credentials for the upstream data source use SCRAM-SHA-512 over TLS, which is appropriate for production Kafka clusters.

The Dockerfile divergence between `Dockerfile` (UBI9-micro) and `Dockerfile.konflux` (UBI9-minimal) is intentional — the Konflux variant uses `ubi9-minimal` which includes `microdnf` and satisfies Konflux/check-payload requirements, while the development Dockerfile uses the even smaller `ubi9-micro`. The GitHub Actions workflow validates only the Konflux Dockerfile, confirming it is the production build path.
