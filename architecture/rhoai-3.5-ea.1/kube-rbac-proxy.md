## Purpose

**Short**: An HTTP reverse proxy that performs Kubernetes RBAC authorization via SubjectAccessReview before forwarding requests to an upstream service.

**Detailed**: kube-rbac-proxy is a small HTTP/HTTPS reverse proxy designed to run as a sidecar container alongside application containers in Kubernetes Pods. It intercepts incoming requests, authenticates the caller using either bearer tokens (via Kubernetes TokenReview), client TLS certificates (mTLS), or OpenID Connect (OIDC) JWTs, and then authorizes the request by submitting a SubjectAccessReview to the Kubernetes API server. Only requests from authenticated and authorized callers are forwarded to the upstream application.

In the RHOAI 3.x platform, kube-rbac-proxy serves as the primary authentication/authorization sidecar. The platform operator (rhods-operator) injects kube-rbac-proxy containers into component Pods to enforce RBAC on metrics endpoints, API endpoints, and other sensitive surfaces. It replaces the older oauth-proxy pattern used in RHOAI 2.x. The proxy typically listens on port 8443/TCP with TLS and forwards authorized traffic to the application container on a localhost port (e.g., 127.0.0.1:8081).

The proxy supports two authorization configuration formats: Format1 (top-level resourceAttributes with optional query/header-based rewrites) and Format2 (path-scoped endpoint rules with per-method resource attribute mappings). Format2 enables fine-grained authorization where different API paths can require different RBAC permissions. A hardcoded authorizer also permits the OpenShift Monitoring Prometheus service account (`system:serviceaccount:openshift-monitoring:prometheus-k8s`) to scrape `/metrics` without additional RBAC configuration.

## Architectural Analysis

kube-rbac-proxy follows a clean layered-filter architecture for request processing. The main HTTP handler composes four filter layers in sequence: path filtering (`WithAllowPaths`), authentication (`WithAuthentication`), authorization (`WithAuthorization`), and auth header injection (`WithAuthHeaders`). Each filter is a decorator over the next, creating a straightforward middleware chain. The `--ignore-paths` mechanism allows specific paths to bypass the authn/authz chain entirely, which is useful for health checks but carries security implications if misconfigured.

The authorization system supports two configuration formats that coexist in the same config file. Format1 defines a single set of resource attributes at the top level, optionally rewritten using query parameters or HTTP headers (e.g., for namespace-scoped metrics). Format2 adds path-scoped endpoint rules with per-HTTP-method resource attribute mappings, enabling fine-grained authorization where different API paths require different RBAC permissions. When Format2 endpoints are configured, they take precedence for matching paths; unmatched paths fall through to Format1. This dual-format approach was added to support components like TrustyAI that need different authorization rules per API endpoint.

The hardcoded metrics authorizer (`pkg/hardcodedauthorizer/metrics.go`) is a notable design decision — it unconditionally allows the OpenShift Monitoring Prometheus service account to read `/metrics` without requiring RBAC configuration. This is practical for operator-deployed sidecars where every component needs metrics scraping, but it creates a tight coupling to the OpenShift monitoring stack's service account naming convention. The code includes a TODO to remove this once the Cluster Monitoring Operator lands static authorizer configuration.

TLS certificate management uses a polling-based hot-reload mechanism (`pkg/tls/reloader.go`) that checks for certificate file changes at a configurable interval (default 1 minute). When changes are detected, the new certificate is loaded without requiring a process restart. The OIDC CA and client CA files use Kubernetes' `DynamicFileCAContent` for similar dynamic reloading. If no TLS certificates are provided, the proxy generates a self-signed certificate at startup — a useful development feature but one that should never occur in production RHOAI deployments. The token masking filter (`SanitizingFilter`) intercepts klog output to redact bearer tokens from TokenReview JSON payloads, preventing accidental token exposure in logs.
