## Purpose

**Short**: An HTTP reverse proxy sidecar that performs Kubernetes RBAC authorization via SubjectAccessReview before forwarding requests to an upstream service.

**Detailed**: kube-rbac-proxy is a small HTTP reverse proxy designed to run as a sidecar container alongside application Pods. It intercepts incoming requests, authenticates the caller (via bearer tokens using Kubernetes TokenReview, client TLS certificates, or OIDC JWT tokens), and then authorizes the request by performing a Kubernetes SubjectAccessReview against the API server. Only authenticated and authorized requests are forwarded to the upstream application.

In the RHOAI 3.x platform, kube-rbac-proxy serves as the primary authentication and authorization enforcement sidecar. The platform operator (rhods-operator) injects kube-rbac-proxy containers into component Pods to protect their HTTP endpoints — typically metrics endpoints and API endpoints. It replaces the oauth-proxy sidecar pattern used in RHOAI 2.x. The proxy listens on a TLS-secured port (typically 8443/TCP) and forwards authorized traffic to the application container listening on localhost.

The proxy also includes a hardcoded authorizer that always permits the OpenShift Monitoring Prometheus service account (`system:serviceaccount:openshift-monitoring:prometheus-k8s`) to scrape `/metrics` endpoints, ensuring cluster monitoring works without additional RBAC configuration. A token-masking log filter prevents bearer tokens from being leaked in log output.

## Architectural Analysis

kube-rbac-proxy follows a clean sidecar proxy pattern where all authentication and authorization concerns are extracted from application code and handled by a co-located proxy container. The architecture separates concerns into three distinct layers: authentication (pkg/authn), authorization (pkg/authz), and HTTP middleware (pkg/filters). The authentication layer supports three mutually exclusive strategies — Kubernetes token delegation (default), OIDC JWT validation, and X.509 client certificates — while the authorization layer uses a union authorizer that chains a hardcoded metrics authorizer, a static configuration-based authorizer, and a Kubernetes SubjectAccessReview authorizer.

The hardcoded metrics authorizer (`pkg/hardcodedauthorizer/metrics.go`) is a notable design decision specific to the OpenShift ecosystem. It unconditionally allows the `system:serviceaccount:openshift-monitoring:prometheus-k8s` service account to perform GET requests on `/metrics`, bypassing the normal SAR authorization flow. This ensures that cluster-level Prometheus monitoring always works, even if RBAC policies are misconfigured. The code includes a TODO comment about removing this once CMO (Cluster Monitoring Operator) implements static authorizer configuration, suggesting this is considered a temporary workaround that has become permanent.

The TLS implementation is thorough: the proxy supports configurable minimum TLS version (default 1.2), configurable cipher suites, hot-reloading of server certificates via a polling-based `CertReloader`, and dynamic client CA reloading for mTLS authentication. HTTP/2 can be explicitly disabled via `--http2-disable` as a mitigation for CVE-2023-44487 (HTTP/2 rapid reset attack). When disabled, the server explicitly sets `TLSNextProto` to an empty map and restricts ALPN to `http/1.1`. The proxy also supports h2c (HTTP/2 cleartext) connections to the upstream via `--upstream-force-h2c`, enabling it to proxy to gRPC servers that don't use TLS.

The token sanitization filter (`cmd/kube-rbac-proxy/app/sanitazion.go`) intercepts all klog output and masks bearer tokens in TokenReview objects before they reach log storage. This is a defense-in-depth measure against credential leakage through verbose logging (klog verbosity level 10 is commonly used in examples). The filter deserializes JSON log content containing TokenReview objects, replaces the `Spec.Token` field with `<masked>`, and re-serializes before logging.

From a build perspective, the RHOAI distribution uses a strict FIPS-compliant build configuration: `GOEXPERIMENT=strictfipsruntime` with `CGO_ENABLED=1` ensures the Go binary uses OpenSSL for all cryptographic operations via dynamic linking. The Konflux pipeline configures hermetic builds with gomod prefetch and produces multi-architecture images (x86_64, ppc64le, s390x, arm64). The upstream Dockerfile uses distroless base images, while the RHOAI Dockerfiles (Dockerfile.konflux and Dockerfile.redhat) use UBI9-based images for Red Hat certification and FIPS compliance.
