## Purpose

**Short**: CPU-optimized build of vLLM providing OpenAI-compatible and gRPC inference serving for large language models without GPU hardware.

**Detailed**: vLLM CPU is a downstream Red Hat build of the open-source vLLM project, packaged specifically for CPU-only inference on ppc64le and s390x architectures. It serves as a high-throughput, memory-efficient LLM inference engine that exposes an OpenAI-compatible REST API (FastAPI/Uvicorn) and an optional gRPC interface. The server supports text generation (chat completions, completions, responses), embeddings, tokenization, scoring, classification, speech-to-text, and real-time WebSocket endpoints.

Within the RHOAI platform, vLLM CPU is deployed as a serving runtime container image (`odh-vllm-cpu-rhel9`) that is referenced by KServe `InferenceService` custom resources. It enables model serving on clusters that lack GPU hardware or need to serve models on IBM Power (ppc64le) and IBM Z (s390x) mainframe architectures. The image is built via Konflux multi-arch pipelines and ships as part of the RHOAI model serving stack.

The codebase is a fork of the upstream vLLM project with Red Hat-specific modifications: UBI9-based container images, architecture-specific build scripts (`build_vllm_ppc64le.sh`, `build_vllm_s390x.sh`), Konflux/Tekton CI pipeline definitions, IBM DevPI wheel index integration for ppc64le dependencies, and custom CMake configuration for CPU-optimized kernels.

## Architectural Analysis

The vLLM CPU repository is a downstream fork of the upstream vLLM project (`vllm-project/vllm`), adapted specifically for CPU-only inference on non-x86 architectures. The primary architectural distinction from the upstream GPU-focused vLLM is the complete removal of CUDA dependencies and the addition of architecture-specific build scripts for IBM Power (ppc64le) and IBM Z (s390x). The Konflux build pipeline (`Dockerfile.konflux.cpu`) uses a multi-stage approach where the builder stage compiles native dependencies (OpenBLAS, numactl, LAPACK) from source via platform-specific shell scripts, then the final stage assembles a minimal runtime image.

The API server architecture follows the standard vLLM V1 engine design: a FastAPI application serves as the HTTP frontend, dispatching requests to the `AsyncLLM` engine running in-process. The engine manages model loading, KV-cache allocation, continuous batching, and token generation. The server exposes a comprehensive OpenAI-compatible API surface including chat completions, completions, embeddings, responses API, and a separate Anthropic-compatible messages endpoint. An optional gRPC entrypoint (`vllm.entrypoints.grpc_server`) provides an alternative serving interface via the `smg-grpc-servicer` package with server reflection enabled.

Security is handled at the application layer through optional Bearer token authentication (`AuthenticationMiddleware`) and optional TLS via Uvicorn's built-in SSL support. The `SSLCertRefresher` class provides certificate hot-reloading by watching certificate files for changes using the `watchfiles` library — this is critical for Kubernetes environments where cert-manager rotates certificates in mounted Secrets. The RHOAI-specific build sets `VLLM_USAGE_STATS_SERVER` to point to Red Hat's telemetry endpoint rather than the upstream vLLM stats server, and the Konflux Dockerfile sets `VLLM_NO_USAGE_STATS=1` to disable usage statistics by default.

The build pipeline is notably non-hermetic for this EA release — the Tekton PipelineRun explicitly sets `hermetic: false`, and there are no lock files for Python packages or RPMs. Dependencies are resolved at build time from PyPI, IBM DevPI (for ppc64le-specific wheels), and Fedora EPEL. The `GRPC_PYTHON_BUILD_SYSTEM_OPENSSL=1` environment variable ensures that the grpcio package links against the system OpenSSL from the UBI9 base image rather than bundling its own, which is important for FIPS compliance on OpenShift clusters. The image runs as non-root user 2000 in group 0, following OpenShift's arbitrary UID security model.

The container entrypoint (`python -m vllm.entrypoints.openai.api_server`) starts the OpenAI-compatible server by default. The `VLLM_USE_V1=1` environment variable enables the V1 engine (the current-generation architecture), and `VLLM_WORKER_MULTIPROC_METHOD=fork` configures multi-process workers to use forking rather than spawning. Performance tuning for CPU inference includes `LD_PRELOAD=/usr/lib64/libtcmalloc.so.4` (tcmalloc for memory allocation) in the Konflux image, and libomp for OpenMP threading in the UBI image.
