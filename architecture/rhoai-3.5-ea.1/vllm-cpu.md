## Purpose

**Short**: CPU-only build of the vLLM high-throughput inference engine, providing an OpenAI-compatible API server for serving large language models without GPU hardware.

**Detailed**: vllm-cpu is the CPU-optimized variant of the vLLM inference engine, packaged as a container image for Red Hat OpenShift AI (RHOAI). It provides an OpenAI-compatible REST API server built on FastAPI/Uvicorn that supports chat completions, text completions, embeddings, reranking, scoring, and other LLM inference tasks. The server is designed for deployment as a model serving runtime in KServe or standalone inference pods on OpenShift.

The CPU build targets ppc64le and s390x architectures via the Konflux build pipeline, and x86_64 via the standard UBI-based Dockerfile. It includes native C++ kernels compiled via CMake for CPU-optimized attention and quantization operations, while omitting CUDA/ROCm GPU acceleration. The component supports model formats compatible with Hugging Face Transformers, including LLaMA, Mistral, Qwen, GPT-NeoX, and many others, with features like LoRA adapters, speculative decoding, structured output enforcement, and tool calling.

In the RHOAI platform, vllm-cpu is typically deployed as a ServingRuntime container managed by KServe's InferenceService CRD, receiving traffic via the platform's Gateway API ingress infrastructure. Usage telemetry can be configured to report to the Red Hat AI Insights Stats endpoint.

## Architectural Analysis

vllm-cpu is a downstream fork of the upstream [vllm-project/vllm](https://github.com/vllm-project/vllm) repository, rebased onto a RHOAI-specific release branch. The key architectural difference from the upstream is the target device: the CPU variant omits CUDA/ROCm GPU kernels and instead builds only the C/C++ CPU kernels in `csrc/cpu/`. These kernels provide optimized attention mechanisms, activation functions, and quantization operations for x86_64 (with AVX2 and AVX512 specializations), ppc64le, and s390x architectures. The CMakeLists.txt (49,578 lines) contains the full build matrix for all target devices, with the `VLLM_TARGET_DEVICE=cpu` flag selecting the CPU-only build path.

The server architecture follows a clean separation between the FastAPI HTTP layer and the async inference engine. The entrypoint (`vllm.entrypoints.openai.api_server`) builds a FastAPI application with modular router registration — each API capability (chat completions, embeddings, pooling, speech-to-text, realtime WebSocket) is conditionally registered based on model-supported tasks. The engine itself (`vllm.v1.engine.async_llm.AsyncLLM`) runs as an in-process async engine with a watchdog task that monitors engine health and triggers graceful shutdown on errors. Multiprocessing support is available via a forkserver pattern for multi-worker deployments. Authentication is handled by a lightweight ASGI middleware that performs SHA256-based constant-time token comparison on `/v1/*` paths only, leaving health and metrics endpoints unauthenticated for Kubernetes probes and Prometheus scraping.

The RHOAI-specific customizations are primarily in the Dockerfiles and Tekton pipeline configurations. The `Dockerfile.cpu.ubi` defines the standard x86_64 build with a `payload/run.sh` installation script pattern, while `Dockerfile.konflux.cpu` targets ppc64le and s390x with architecture-specific build scripts (`build_vllm_ppc64le.sh`, `build_vllm_s390x.sh`) that compile dependencies like OpenBLAS, LAPACK, and numactl from source. The container runs as non-root user (UID 2000, GID 0) for OpenShift compatibility, pre-downloads tiktoken tokenizers for disconnected environments, and sets `VLLM_USE_V1=1` to enable the new V1 engine. The `Dockerfile.cpu.ubi` sets `VLLM_USAGE_STATS_SERVER=https://console.redhat.com/api/rhaiis-stats` to direct telemetry to the Red Hat AI Insights endpoint, while the Konflux Dockerfile sets `VLLM_NO_USAGE_STATS=1` to disable telemetry entirely — suggesting the telemetry configuration is applied differently per build variant.

A notable design consideration is the SSL certificate hot-reload capability (`SSLCertRefresher` in `vllm/entrypoints/ssl.py`), which uses the `watchfiles` library to monitor certificate files and reload the SSL context on changes. This is important for long-running inference pods in OpenShift where certificate rotation is managed by cert-manager or the service CA. The gRPC server (`vllm/entrypoints/grpc_server.py`) is available as an alternative interface using the `smg-grpc-servicer` package but currently only supports insecure ports — TLS for gRPC would need to be provided by a sidecar or service mesh.
