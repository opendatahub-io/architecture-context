## Purpose

**Short**: llm-d is a Kubernetes-native distributed LLM inference serving stack that provides optimized container images, deployment guides, and composable recipes for production-grade large language model serving across multiple hardware accelerators.

**Detailed**: llm-d is a meta-project that orchestrates the deployment of a distributed inference serving stack optimized for large language models (LLMs) on Kubernetes. Unlike typical source code repositories, llm-d does not contain application source code itself. Instead, it provides: (1) multi-platform Dockerfiles that build vLLM-based inference server images with GPU/accelerator-specific optimizations, RDMA networking libraries, and KV-cache transfer capabilities; (2) Kustomize-based deployment recipes and Helm value overlays for composing inference topologies; and (3) comprehensive deployment guides for well-tested serving patterns including optimized baselines, prefill/decode disaggregation, wide expert-parallelism, tiered prefix caching, and workload autoscaling.

The project integrates with the Kubernetes Gateway API Inference Extension (GAIE) for intelligent request routing via the llm-d Router (EPP), leverages NIXL for high-performance GPU-to-GPU KV-cache transfer, and supports multiple accelerator families including NVIDIA CUDA (A100/H100/H200/B200/GB200), AMD ROCm (MI250+), Intel XPU and Gaudi HPU, Google TPU (v5e/v6e/v7), and CPU. For RHOAI, llm-d provides the foundational container images and deployment patterns consumed by the platform's inference serving capabilities.

## Architectural Analysis

llm-d is architecturally unique among RHOAI components in that it is a **meta-project** rather than a traditional application repository. It contains no application source code — instead it orchestrates the assembly of a complex inference serving stack from upstream open-source components. The Dockerfiles are the most architecturally significant artifacts, functioning as build pipelines that compose vLLM, NIXL, NVSHMEM, UCX, DeepEP, FlashInfer, LMCache, and OpenTelemetry into optimized container images for five distinct accelerator families.

The **multi-stage CUDA Dockerfile** (592 lines) is the most complex artifact in the repository. It implements a builder/runtime separation where compiled C/C++ extensions (gdrcopy, UCX, NIXL, DeepEP, DeepGEMM, FlashInfer, LMCache) are built in a CUDA devel stage and copied as wheels or shared libraries into a minimal runtime stage. The runtime stage supports three vLLM installation modes: full editable install from source (development), precompiled binary install (default for faster builds), and full prebuilt wheel install (production). A dedicated `runtime-gb200` Docker target extends the base runtime with GB200-specific DeepEP, enabling the same Dockerfile to produce both standard and GB200-optimized images.

The **deployment architecture follows a layered composition model**. Base Kustomize recipes define the fundamental deployment templates (decode-only or prefill+decode), which are then specialized via accelerator-specific overlays (GPU, HPU, TPU, XPU, CPU, AMD) and platform-specific patches (GKE, CoreWeave, OpenShift). The EPP (Endpoint Picker Plugin) is configured via layered Helm values: base.values.yaml provides the scheduler image version, guide-specific values enable feature plugins (P/D disaggregation, prefix-cache scoring, flow control), and feature-specific values add monitoring or tracing. This composition model allows users to mix and match topologies, accelerators, and features without combinatorial explosion.

The **NVSHMEM patches** reveal important infrastructure hardening: the `cks_nvshmem3.3.20.patch` extends InfiniBand device detection beyond Mellanox (mlx5) hardware to support a configurable HCA interface name via the `NVSHMEM_IBGDA_HCA_IFNAME` environment variable — critical for deployments on non-Mellanox RDMA networks. The `nvshmem_zero_ibv_ah_attr` patches fix uninitialized struct memory that caused "Unable to create ah" errors on RoCE devices, demonstrating the project's deep engagement with RDMA networking edge cases that affect production stability.

The project's **security posture is minimal by design** — the model server containers expose unauthenticated HTTP endpoints on port 8000, relying on the Kubernetes network policy and the upstream Gateway provider for access control. This is intentional: llm-d operates at the infrastructure layer below authentication concerns, which are handled by the platform operator (RHOAI) or user-configured gateway providers. The absence of FIPS-specific build flags is expected since llm-d builds Python/C++ artifacts, not Go binaries — FIPS compliance is inherited from the UBI9 base image's system OpenSSL.

A notable **build hermeticity gap** exists: the project has no lock files (no go.sum, no uv.lock, no rpms.lock.yaml) and no Konflux Dockerfiles. All dependencies are pinned by version tag or commit SHA in the Dockerfiles and `docker/common-versions` file, but builds clone dependencies from git at build time, making them non-hermetic. The downstream RHOAI integration would need to introduce Konflux Dockerfiles with Hermeto prefetch to achieve supply chain compliance for productization.
