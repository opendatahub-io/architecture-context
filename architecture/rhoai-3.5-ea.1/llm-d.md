## Purpose

**Short**: llm-d is a high-performance distributed inference serving stack for Kubernetes that provides optimized container images, deployment recipes, and well-lit-path guides for serving large language models at scale across multiple hardware accelerators.

**Detailed**: llm-d is a CNCF sandbox project founded by Red Hat, Google Cloud, IBM Research, CoreWeave, and NVIDIA. This repository serves as the central integration point for the llm-d ecosystem — it does not contain the router, scheduler, or model server source code, but instead provides: (1) multi-accelerator container images that package vLLM with high-performance networking libraries (NVSHMEM, UCX, NIXL, DeepEP, FlashInfer, LMCache), (2) kustomize-based deployment recipes and Helm values files for Kubernetes, and (3) documented "well-lit path" guides that compose the llm-d Router, InferencePool CRD, and vLLM model servers into production-grade serving topologies.

The repository produces container images for five accelerator backends (NVIDIA CUDA, AMD ROCm, Intel Gaudi HPU, Intel XPU, CPU) and one utility image (RDMA tools). Each image bundles vLLM with optimized kernels, RDMA networking, and observability instrumentation. The guides directory provides tested, benchmarked deployment patterns covering intelligent routing, prefill/decode disaggregation, KV cache tiering, expert parallelism, flow control, and workload autoscaling — each with hardware-specific overlays and platform-specific patches (OpenShift, GKE, CoreWeave).

## Architectural Analysis

### Repository as Integration Hub

The llm-d repository is architecturally significant as the integration point for the entire llm-d ecosystem, despite containing no runtime source code for the Router or model server. Its primary outputs are container images and deployment recipes. The container images embed carefully pinned versions of vLLM (from a neuralmagic fork with cherry-picks), high-performance networking libraries (NVSHMEM, UCX, NIXL), and specialized compute kernels (DeepEP, FlashInfer, DeepGEMM) — all compiled from source inside multi-stage Docker builds. This approach trades build complexity for tight version control across the stack.

### Build Architecture and Multi-Accelerator Strategy

The Dockerfile architecture follows a consistent pattern across accelerator backends: a builder stage compiles all native dependencies from source, producing wheels and shared libraries, then a runtime stage installs them into a minimal base image. The CUDA Dockerfile is by far the most complex (600+ lines, 10+ compiled components) due to NVIDIA's rich GPU networking ecosystem. The GB200 variant demonstrates an interesting pattern — a Docker multi-stage target (`runtime-gb200`) that swaps out a single wheel (DeepEP) while inheriting everything else from the standard runtime. The build system uses `docker/common-versions` as a single source of truth for vLLM and CUDA versions, imported by the Makefile.

### Deployment Composition Model

Rather than a monolithic deployment, llm-d uses a compositional approach: guides combine kustomize bases, hardware-specific overlays, platform-specific patches, and Helm values files. The `guides/recipes/` directory provides reusable building blocks (model server base, gateway providers, scheduler configurations), while each well-lit path guide (optimized-baseline, pd-disaggregation, etc.) selects and composes these blocks. This enables permutative coverage across 6+ accelerator types, 4+ infrastructure providers (GKE, OpenShift, CKS, generic), and multiple serving topologies. The kustomize-first migration (announced in v0.7.0) consolidates earlier Helm-only guides into kustomize+Helm layering.

### Security Considerations

The images run as non-root user 2000 in the CUDA image (OpenShift compatible) but default to root in CPU, HPU, and ROCm images. No TLS is configured by default on the inference endpoint (port 8000) — TLS termination is expected at the Gateway/proxy layer. The WVA autoscaler guide configurations set `PROMETHEUS_TLS_INSECURE_SKIP_VERIFY=true`, which is a known security gap in production deployments. FIPS compliance is not explicitly addressed: the images use Python (vLLM), which delegates crypto to system OpenSSL, but there is no explicit FIPS mode configuration. The threat model document is a placeholder.

### Observability Design

Observability is deeply integrated: all images include OpenTelemetry SDK packages for distributed tracing, and vLLM natively exposes Prometheus metrics on port 8000/metrics. The EPP adds its own metrics endpoint on port 9090. This dual-metric architecture (model server metrics + router metrics) enables sophisticated autoscaling via the WVA, which queries Prometheus for KV cache utilization, queue depth, and active request counts to make global scaling decisions.
