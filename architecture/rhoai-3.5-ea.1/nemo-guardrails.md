## Purpose

**Short**: NeMo Guardrails is an open-source toolkit for adding programmable guardrails (input/output safety, topic control, jailbreak detection, hallucination checks) to LLM-based conversational systems, exposing an OpenAI-compatible HTTP API.

**Detailed**: NeMo Guardrails, originally developed by NVIDIA, provides a framework for controlling and securing interactions with Large Language Models. It implements a "rails" architecture where user inputs and LLM outputs pass through configurable safety checks before being returned to the caller. The system supports multiple rail types: input rails (validate user messages), output rails (validate LLM responses), dialog rails (control conversation flow), retrieval rails (validate RAG context), and tool input/output rails (validate tool calls and results).

The RHOAI distribution packages NeMo Guardrails as a containerized FastAPI server built on Red Hat UBI 9 with Python 3.12. The server exposes an OpenAI-compatible `/v1/chat/completions` endpoint plus a dedicated `/v1/guardrail/checks` endpoint for standalone message validation. It supports multiple LLM backends (OpenAI, NVIDIA NIM, Azure OpenAI, Anthropic, Cohere, vLLM, TRT-LLM) and includes a library of built-in guardrail implementations for content safety, sensitive data detection, hallucination checking, jailbreak detection, and topic safety. The RHOAI build filters out closed-source guardrails, keeping only the open-source subset.

The component integrates with OpenTelemetry for distributed tracing, supports both streaming and non-streaming responses, and can optionally use Redis for thread/conversation state persistence. Pre-trained NLP models (SpaCy, Sentence Transformers, FastEmbed) are baked into the container image at build time to support offline inference for guardrail evaluation.

## Architectural Analysis

NeMo Guardrails follows a layered pipeline architecture where each LLM interaction passes through a configurable chain of "rails" — input rails validate user messages before they reach the LLM, dialog rails control conversation flow, retrieval rails validate RAG context, and output rails validate LLM responses before they're returned. This design enables defense-in-depth: multiple independent safety checks can be composed declaratively using the Colang domain-specific language or via Python action classes.

The server implements an OpenAI-compatible API surface (`/v1/chat/completions`, `/v1/models`), making it a drop-in intermediary between any OpenAI-compatible client and backend LLM. The `GuardrailsChatCompletionRequest` schema extends the standard OpenAI request with a `guardrails` field for config selection, inline config, thread management, and generation options. This design means existing applications can adopt guardrails with minimal client-side changes. The `/v1/guardrail/checks` endpoint provides a standalone validation mode where messages are evaluated against rails without generating new LLM responses — useful for pre-flight checks or batch validation.

The RHOAI distribution applies a "profile filtering" mechanism at build time (`scripts/filter_guardrails.py` with `scripts/provider-list.yaml`) that removes closed-source guardrail integrations (ActiveFence, AI Defense, CrowdStrike, etc.), keeping only open-source implementations like content safety, self-check, sensitive data detection, hallucination detection, topic safety, regex, and LLaMA Guard. This ensures the shipped image contains only redistributable components. The `GUARDRAILS_PROFILE=opensource` build argument controls this filtering.

A notable architectural characteristic is the heavy use of pre-baked ML models in the container image. The Dockerfile runs `pre_download_required_models.py` during the build, downloading SpaCy models, Sentence Transformers, FastEmbed models, and NLTK data into `/app/.cache/`. This makes the container self-contained for inference — no network access is needed at runtime for guardrail evaluation. The SpaCy `en_core_web_lg` model is additionally locked via `artifacts.lock.yaml` and prefetched through Hermeto/cachi2, providing supply chain integrity for this large artifact. The trade-off is a significantly larger container image.

The multi-architecture build support (amd64 and ppc64le) is handled through conditional Dockerfile stages. The ppc64le path requires building PyTorch, tiktoken, ONNX Runtime, OpenBLAS, and hf-xet from source due to the lack of pre-built wheels for that architecture. This is implemented as 6 parallel builder stages (openblas, torch, tiktoken, onnxruntime, hf-xet, then final assembly) in `Dockerfile.konflux`. The amd64 path uses pre-built wheels and is significantly simpler. This multi-arch complexity is a maintenance risk but is required for RHOAI's ppc64le support.

The server has no built-in authentication — it forwards `Authorization` headers from incoming requests to upstream LLM providers but does not validate them itself. In the RHOAI deployment, authentication and TLS are expected to be provided by the platform's ingress layer (kube-rbac-proxy sidecar and/or Gateway API with TLS termination). The entrypoint script (`scripts/entrypoint.sh`) validates that configuration files exist at `/app/config/${CONFIG_ID}/` and starts the server with the chat UI disabled (`--disable-chat-ui`), appropriate for a headless API service in a platform deployment.
