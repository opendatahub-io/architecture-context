## Purpose

**Short**: Kubernetes operator that extends KServe with OpenShift/RHOAI-specific model serving capabilities including ingress, authentication, monitoring, NVIDIA NIM integration, and model registry support.

**Detailed**: The odh-model-controller is a controller-runtime operator that watches KServe custom resources (InferenceService, ServingRuntime, InferenceGraph, LLMInferenceService) and supplements KServe's behavior with platform-specific integrations for Red Hat OpenShift AI. It manages the full lifecycle of model serving infrastructure beyond what upstream KServe provides.

The controller creates and manages OpenShift Routes for inference endpoints, Kuadrant AuthPolicy resources for Gateway API-based authentication (RHOAI 3.x pattern), Istio EnvoyFilter resources for TLS bootstrap with Authorino, Prometheus ServiceMonitors and metrics dashboards for observability, network policies for traffic isolation, KEDA TriggerAuthentications for autoscaling, and storage-config secrets aggregating data connections for KServe model access.

Additionally, the operator manages NVIDIA NIM (NVIDIA Inference Microservices) integration through a dedicated Account CRD that handles API key validation, pull secret generation for NVIDIA Container Registry, and ServingRuntime template management. A separate model-serving-api server component provides a REST API for gateway discovery, enabling the RHOAI dashboard to offer self-service gateway selection for LLM deployments.

## Architectural Analysis

The odh-model-controller demonstrates a layered reconciliation architecture with clear separation between KServe InferenceService management (RHOAI 2.x patterns) and LLMInferenceService management (RHOAI 3.x patterns). The InferenceService controller orchestrates sub-reconcilers for Routes, ClusterRoleBindings, metrics Services, ServiceMonitors, and metrics dashboards — each implemented as a pluggable `SubResourceReconciler` with independent lifecycle management. The LLMInferenceService controller adds a Gateway API integration layer, managing AuthPolicy resources that target HTTPRoutes for Kuadrant-based authentication, while the separate GatewayReconciler bootstraps EnvoyFilter and AuthPolicy resources on Gateway objects independently of any specific model deployment.

The controller uses a delta-processor pattern throughout — a `DeltaProcessor` computes changes between desired and existing resource states using pluggable `ResourceComparator` functions, then applies create/update/delete operations accordingly. This pattern provides consistent change detection across all 12 resource types managed by the operator (Routes, ClusterRoleBindings, NetworkPolicies, ConfigMaps, Secrets, ServiceAccounts, Services, ServiceMonitors, PodMonitors, AuthPolicies, EnvoyFilters, TriggerAuthentications).

The NIM Account controller implements a handler-chain pattern where four handlers execute sequentially: ValidationHandler (API key validation against NVIDIA NGC), ConfigMapHandler (model metadata ConfigMap), TemplateHandler (OpenShift Template with ServingRuntime), and PullSecretHandler (NVIDIA registry pull secret). Each handler returns a `HandleResponse` indicating whether to continue, requeue, or fail — providing clean composition with per-handler error isolation. The air-gapped mode short-circuits API calls while maintaining expected resource references.

The model-serving-api server is architecturally notable for its security-first design: FIPS-compliant TLS configuration with restricted cipher suites (ECDHE+AES-GCM only, P-256/P-384 curves), hot-reload certificate management via fsnotify with debouncing for Kubernetes secret volume rotations, Bearer token delegation via SelfSubjectAccessReview for user-scoped gateway discovery, and comprehensive security headers (HSTS, X-Frame-Options, no-cache). The gateway discovery logic respects Gateway API namespace policies (All/Same/Selector) ensuring users only see gateways whose listeners permit their target namespace.

A notable migration pattern exists in the codebase: the MaaSRBACCleanupRunner (a `manager.Runnable` requiring leader election) removes legacy Roles and RoleBindings with `-model-post-access` suffixes created by the v3.3 controller. This demonstrates how the codebase handles breaking changes across versions — cleanup runs once at startup under leader election to avoid duplicate work, then the controller proceeds with normal reconciliation using the updated resource model.
