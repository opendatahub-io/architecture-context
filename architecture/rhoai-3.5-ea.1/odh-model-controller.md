## Purpose

**Short**: Kubernetes operator that automates model serving infrastructure management for KServe InferenceService, LLMInferenceService, and NVIDIA NIM deployments on OpenShift AI.

**Detailed**: The odh-model-controller is a controller-runtime based Kubernetes operator that eliminates manual steps when deploying AI/ML models on the Red Hat OpenShift AI platform. It watches KServe InferenceService and LLMInferenceService custom resources and automatically provisions networking (OpenShift Routes, Gateway API HTTPRoutes), monitoring (Prometheus ServiceMonitors, metrics dashboards), autoscaling (KEDA TriggerAuthentication), security (ClusterRoleBindings, AuthPolicies, EnvoyFilters), and certificate management (Ray TLS, CA bundle consolidation) resources.

The controller has three major subsystems: (1) **Serving controllers** that manage classic KServe InferenceService deployments with OpenShift Route-based networking, Prometheus monitoring integration, and KEDA autoscaling; (2) **LLM controllers** that manage next-generation LLMInferenceService deployments with Gateway API (HTTPRoute), Kuadrant AuthPolicy, and Istio EnvoyFilter integration; and (3) **NIM controller** that integrates with NVIDIA's NGC catalog to validate API keys, fetch model metadata, create ServingRuntime templates, and manage container registry pull secrets for NVIDIA NIM inference microservices.

The repository also ships a separate **model-serving-api** server binary — a lightweight HTTPS REST API that discovers Gateway API Gateways and returns filtered gateway references to the dashboard frontend, enabling users to select inference gateways when deploying models.

## Architectural Analysis

The odh-model-controller employs a **composite reconciler architecture** where the primary InferenceServiceReconciler delegates to specialized sub-reconcilers (Route, Metrics Service, ServiceMonitor, Dashboard ConfigMap, KEDA, ClusterRoleBinding, ServiceAccount) through a clean `SubResourceReconciler` interface. Each sub-reconciler implements `Reconcile()`, `Delete()`, and `Cleanup()` methods, enabling independent resource lifecycle management. This pattern is extended to the LLM subsystem via `LLMSubResourceReconciler` with the `KserveAuthPolicyReconciler` as its first implementation. The delta-based reconciliation pattern (desired vs. existing state comparison with typed comparators) provides efficient, predictable updates across all resource types.

The **LLM/Gateway API subsystem** represents the RHOAI 3.x networking evolution. The `GatewayReconciler` operates independently of the `LLMInferenceServiceReconciler` — it bootstraps EnvoyFilter and AuthPolicy resources on Gateways proactively, ensuring auth infrastructure exists before any services reference the gateway. This is a deliberate design choice: the gateway's TLS and auth configuration is a prerequisite for HTTPRoute-based routing, so it must be established ahead of service deployment. The controller uses sophisticated watch predicates to trigger only when gateway references actually change in LLMInferenceService specs, avoiding unnecessary reconciliation loops.

The **NIM integration** follows a handler-chain pattern where the Account controller sequentially invokes Validation → ConfigMap → Template → PullSecret handlers. Each handler returns a `HandleResponse` that controls whether to continue, requeue, or fail. The handler chain ensures that downstream resources (Templates, PullSecrets) are only created after upstream prerequisites (validated API key, fetched model metadata) succeed. The air-gapped mode short-circuits remote API calls entirely, using pre-configured templates — a critical feature for disconnected environments. The singleton-per-namespace constraint enforced by the Account validating webhook prevents misconfiguration.

The **model-serving-api** server is architecturally notable for its dual-auth pattern: it uses the caller's Bearer token for RBAC authorization (via `SelfSubjectAccessReview` checking `create` permission on `llminferenceservices`) while using its own ServiceAccount for listing Gateway resources. This separation ensures that users only see gateways in namespaces where they have deployment permissions, without requiring the user's token to have direct Gateway list access. The server uses a shared HTTP transport with per-request Bearer token injection to avoid connection pool churn across RBAC checks.

The **certificate management** layer is distributed across multiple controllers: the ConfigMap controller consolidates CA certificates from `odh-trusted-ca-bundle` and `openshift-service-ca.crt` into `odh-kserve-custom-ca-bundle`; the Secret controller references this bundle in `storage-config` for HTTPS S3 endpoints; the Pod controller and webhook collaborate on Ray TLS — the webhook mutates pods to inject init containers and volume mounts, while the Pod controller dynamically generates per-pod TLS certificates signed by a self-managed CA. Certificate rotation triggers cascading reconciliation: a CA bundle change deletes `storage-config` (forcing KServe regeneration), and a Ray CA secret update propagates to all user-namespace `ray-tls` secrets.
