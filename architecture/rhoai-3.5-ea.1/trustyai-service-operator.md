## Purpose

**Short**: The TrustyAI Service Operator manages the lifecycle of AI trustworthiness, evaluation, and guardrails services on RHOAI clusters.

**Detailed**: The TrustyAI Service Operator is a multi-controller Kubernetes operator that manages five distinct AI/ML-related services within the Red Hat OpenShift AI platform. It follows a plugin architecture where controllers are selectively enabled via command-line flags, allowing the same operator binary to serve different deployment profiles.

The operator manages five core workloads: (1) **TrustyAIService (TAS)** — the original bias/fairness monitoring service that integrates with KServe and ModelMesh to capture inference data and compute fairness metrics; (2) **LMEvalJob (LMES)** — a Kubernetes-native job execution engine for running LM-Evaluation-Harness benchmarks against language models with optional Kueue-based scheduling; (3) **EvalHub** — a multi-tenant evaluation framework providing a centralized API for managing evaluation providers (garak, lm-evaluation-harness, guidellm, lighteval) and benchmark collections; (4) **GuardrailsOrchestrator (GORCH)** — a deployment manager for guardrails orchestration services that auto-discovers KServe InferenceServices and configures detector routing with optional gateway sidecars; and (5) **NemoGuardrails** — a deployment manager for NVIDIA NeMo guardrails servers with CA bundle management.

Each controller creates its own set of Kubernetes resources (Deployments, Services, Routes, ConfigMaps, RBAC) and integrates with the platform through shared patterns including kube-rbac-proxy authentication, OpenShift Route creation, Prometheus ServiceMonitor scraping, and ConfigMap-based image parameterization. The operator is deployed by the platform operator (rhods-operator) via kustomize overlays that select which controllers to enable.

## Architectural Analysis

The TrustyAI Service Operator is a multi-tenant, multi-controller operator that consolidates five distinct AI trustworthiness and evaluation workloads into a single binary. This "mono-operator" pattern is unusual — most operators manage a single CRD — but it provides operational simplicity (one Deployment to manage) at the cost of a large RBAC footprint. The operator binary must hold permissions for all five controllers even when only a subset is enabled, though the kustomize component architecture mitigates this by assembling only the needed ClusterRoles per overlay.

The plugin registration architecture (`controllers/gorch.go`, `controllers/lmes.go`, etc.) is clean: each top-level file calls `registerService()` with a service name and setup function. The `--enable-services` flag selects which controllers to activate at startup. This pattern supports the kustomize overlay model where different distributions enable different controller subsets. The `config/components/` directory structure mirrors this modularity — each controller's CRD, RBAC, and supporting resources are packaged as a kustomize Component that can be independently composed.

The TAS controller is architecturally distinct from the other four controllers. It manages a long-running service deployment with kube-rbac-proxy auth, PVC or database storage, and deep integration with KServe/ModelMesh — patching InferenceService inference loggers and injecting environment variables into ModelMesh Deployments. It also integrates with Istio when present, creating DestinationRule and VirtualService resources. This makes TAS the most tightly coupled component to the platform's networking stack. The TAS controller also disables controller-runtime's default caching for ConfigMaps, Secrets, Pods, and Services (`client.CacheOptions.DisableFor`) to prevent OOM from cluster-wide informer floods — a pragmatic optimization for operators running in namespaces with many resources.

The LMES controller follows a fundamentally different pattern: instead of managing a long-lived Deployment, it creates ephemeral Pods that execute evaluation jobs. The driver binary (built from `cmd/lmes_driver/main.go`) is injected as an init container that copies itself into a shared volume, then coordinates the evaluation run from within the main container. Status polling uses `pods/exec` to invoke the driver's `--get-status` endpoint — an unusual but effective pattern for Pod-based job management that avoids the need for a persistent service. The optional JobManager controller wraps LMEvalJob as a Kueue GenericJob, enabling scheduling/preemption without modifying the LMES controller's Pod lifecycle.

The EvalHub controller introduces multi-tenancy to the operator: it watches Namespaces labeled with `evalhub.trustyai.opendatahub.io/tenant` and creates per-tenant ServiceAccounts, Roles, and RoleBindings. This allows evaluation jobs to be submitted in tenant namespaces while EvalHub itself runs in the operator namespace. The auth model differs from TAS and GORCH — instead of kube-rbac-proxy sidecars, EvalHub performs SubjectAccessReview checks internally, using an auth-reviewer ClusterRoleBinding. This is appropriate for a service that needs fine-grained per-request authorization rather than blanket endpoint protection.

A notable security design decision is the RHOAI vs ODH difference in LMES defaults: the RHOAI overlay sets `lmes-allow-online=false` and `lmes-allow-code-execution=false`, while ODH sets both to `true`. This reflects RHOAI's stricter security posture for enterprise customers — LMES evaluation jobs cannot download remote models/datasets or execute arbitrary code unless explicitly overridden per-job. The enforcement happens at the controller level (`controllers/lmes/lmevaljob_controller.go`) where protected environment variables are injected that override any user-specified settings.
