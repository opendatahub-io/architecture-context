## Purpose

**Short**: Kubernetes operator that manages AI trustworthiness, evaluation, and guardrails services for Red Hat OpenShift AI.

**Detailed**: The TrustyAI Service Operator is a multi-controller Kubernetes operator that manages five distinct AI/ML trustworthiness services within RHOAI. It uses a modular, service-registration architecture where each controller is independently enabled via command-line flags (`--enable-services TAS,LMES,GORCH,NEMO_GUARDRAILS,EVALHUB`), allowing fine-grained deployment composition through kustomize overlays.

The operator manages: (1) **TrustyAI Service (TAS)** — an AI explainability and fairness monitoring service that integrates with KServe InferenceServices and ModelMesh to capture model predictions and compute bias/fairness metrics; (2) **LM Evaluation Service (LMES)** — a Kubernetes-native evaluation framework that orchestrates language model evaluation jobs using lm-evaluation-harness and Unitxt, with optional Kueue integration for queue-based scheduling; (3) **Guardrails Orchestrator (GORCH)** — deploys the FMS Guardrails Orchestrator with auto-configuration from KServe InferenceServices, supporting built-in detectors and sidecar gateways; (4) **NeMo Guardrails** — manages NVIDIA NeMo Guardrails server deployments with CA bundle management and configurable guardrail configurations; (5) **EvalHub** — a centralized evaluation hub with multi-tenant RBAC, provider/collection management, MLflow integration, and Kueue workload failure detection.

Each controller creates Kubernetes resources dynamically (Deployments, Services, Routes, ConfigMaps, PVCs, ServiceMonitors) and uses embedded Go templates for resource generation. The operator integrates with OpenShift-specific features (Routes, Service Serving Certificates) and optionally with Istio service mesh for traffic management.

## Architectural Analysis

The TrustyAI Service Operator demonstrates a sophisticated modular controller architecture that packs five functionally distinct AI trustworthiness services into a single operator binary. The service-registration pattern (`registerService()` in `controllers/controllers.go`) with CLI flag-based enablement (`--enable-services TAS,LMES,GORCH,NEMO_GUARDRAILS,EVALHUB`) combined with kustomize components (one per controller) provides clean separation of concerns while sharing a single deployment, leader election, and ConfigMap. This is architecturally significant because it means the platform operator only needs to manage one Deployment to get all five capabilities, but the kustomize structure ensures only the needed CRDs and RBAC are installed for each overlay.

The controller implementations show two distinct resource-generation patterns: the TAS, GORCH, and NemoGuardrails controllers use an embedded Go template system (`//go:embed` with `.tmpl.yaml` files and a generic `ParseResource[T]()` function) for creating Kubernetes resources from YAML templates with Go template substitution. In contrast, the EvalHub controller constructs resources entirely in Go code. The LMES controller is unique in that it creates bare Pods (not Deployments) with a two-container init-container/main-container pattern — the driver binary is copied via init container into a shared EmptyDir volume, then the main container executes `python -m lm_eval` while the driver provides an HTTP status API on localhost:18080 for progress tracking. This pattern avoids the overhead of Job/Deployment controllers for ephemeral evaluation workloads but means the operator itself must implement job lifecycle management (the 6-state machine: New → Scheduled → Running → Complete/Cancelled/Suspended).

Authentication is consistently implemented via kube-rbac-proxy sidecars across TAS, GORCH, and NemoGuardrails controllers, with SubjectAccessReview-based authorization. The kube-rbac-proxy configuration enforces FIPS 140-2 compliant TLS (TLS 1.2 minimum with AES-GCM cipher suites). EvalHub takes a different approach, using OpenShift Service Serving Certificates directly on the application container (port 8443) with application-level SAR checks and multi-tenant namespace isolation via X-Tenant headers. This divergence likely reflects EvalHub's newer design optimizing for simpler deployment topology.

The operator has a notable memory optimization in `cmd/main.go` (lines 116-125): it disables the controller-runtime informer cache for ConfigMaps, Secrets, Pods, and Services to prevent OOM from cluster-wide informer cache flooding. This is critical because several controllers (TAS, GORCH) need to read ConfigMaps and Secrets across namespaces, which would otherwise cause the cached client to create cluster-wide informers. The trade-off is ~50ms API server reads instead of ~1ms cache reads, acceptable for the operator's reconciliation frequency (30-second default requeue for TAS, 10-second polling for LMES).

The Kueue integration spans two controllers: the Job Manager (`controllers/job_mgr`) implements the Kueue `GenericJob` interface to bridge LMEvalJob ↔ Kueue Workload, while the EvalHub controller independently watches Kueue Workloads to detect failed/inadmissible workloads and propagate failure status back to evaluation jobs. This dual integration ensures evaluation jobs benefit from cluster-wide resource scheduling while the evaluation hub maintains accurate status tracking.
