## Purpose

**Short**: A collection of text detection microservices for the FMS Guardrails Orchestrator, providing regex-based PII detection, HuggingFace model-based content classification, and LLM-as-a-judge evaluation capabilities.

**Detailed**: Guardrails Detectors is a repository of detector algorithms and microservices that integrate with the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator). The orchestrator invokes these detectors to analyze text generation input and output for safety, compliance, and quality concerns.

The repository produces three distinct detector types, each packaged as a separate container image: (1) **Built-in detectors** — lightweight, heuristic detection functions including regex-based PII detection (email, credit card, SSN, phone numbers, IP addresses), file-type validation (JSON, XML, YAML with optional schema validation), and user-defined custom Python detectors with static code analysis sandboxing; (2) **HuggingFace detectors** — a generic ML inference runtime compatible with `AutoModelForSequenceClassification`, `AutoModelForTokenClassification`, and `GraniteForCausalLM` models, supporting GPU acceleration via CUDA and multi-architecture builds (amd64, ppc64le, s390x); (3) **LLM Judge detectors** — an LLM-as-a-judge detector that delegates evaluation to an external OpenAI-compatible vLLM server using the `vllm_judge` library, supporting built-in metrics (safety, toxicity, accuracy, helpfulness, etc.) and custom natural-language criteria.

All three detector types expose a common REST API surface conforming to the IBM Detector API specification (`/api/v1/text/contents`) and are designed to be deployed as KServe InferenceServices on Red Hat OpenShift AI. The detectors are maintained by the TrustyAI team and are part of the RHOAI Guardrails feature.

## Architectural Analysis

The Guardrails Detectors repository follows a clean modular architecture with three independent detector types sharing a common FastAPI base layer. The `detectors/common/` package provides the `DetectorBaseAPI` class, Pydantic request/response schemas, and a `InstrumentedDetector` base class that standardizes Prometheus metrics (detection counts, request counts, error counts, and runtime duration) across all detector types. This shared instrumentation layer means all detectors emit consistent `trustyai_guardrails_*` metrics regardless of their internal detection logic.

The built-in detector is architecturally the simplest but most extensible. It uses a registry pattern (`BaseDetectorRegistry`) where each detector kind (regex, file_type, custom) registers its functions at startup. The custom detector subsystem is notable for its security design: it loads user-provided Python code from `custom_detectors/custom_detectors.py` but first passes it through AST-based static code analysis (`static_code_analysis()`) that blocks imports of `os`, `subprocess`, `sys`, `shutil` and calls to `eval`, `exec`, `open`, `compile`, `input`. Specific exceptions are carved out for `os.environ` and `os.getenv` access. The custom detector framework also supports `@non_blocking` and `@use_instruments` decorators that are injected into the user module namespace, allowing custom detectors to run asynchronously and register Prometheus metrics.

The HuggingFace detector is a self-contained ML inference runtime that auto-detects the model architecture at startup (`AutoConfig.from_pretrained`) and dispatches to the appropriate processing pipeline: `process_sequence_classification` for binary/multi-class classifiers (e.g., toxicity models), `process_token_classification` for NER-style models (e.g., PII entity detection), or `process_causal_lm` for generative models like Granite Guardian that produce safe/unsafe verdicts via text generation. The token classification pipeline is particularly sophisticated — it maps subword tokens back to character offsets using the fast tokenizer's offset mapping, groups adjacent same-label tokens into spans, and averages per-token probabilities into span-level scores. Multi-architecture support (ppc64le, s390x) is handled entirely in the Dockerfile via conditional PyTorch compilation from source with gcc-toolset-13 and OpenBLAS, which significantly increases build complexity but enables deployment on IBM Power and Z systems.

The LLM Judge detector takes a fundamentally different approach by delegating all inference to an external vLLM server. It uses the `vllm_judge` library's `Judge.from_url()` to connect to any OpenAI-compatible API endpoint and evaluates content against built-in metrics (safety, toxicity, accuracy, helpfulness, clarity, etc.) or custom natural-language criteria. This design trades self-containment for flexibility — any LLM served by vLLM can be used as a judge without model-specific code. The judge detector is the only one that exposes a `/api/v1/text/generation` endpoint for evaluating prompt+response pairs, in addition to the standard `/api/v1/text/contents` endpoint.

A key architectural observation is the inconsistency in Konflux build support across detector types: the built-in and HuggingFace detectors have `Dockerfile.konflux.*` variants (indicating they are built by the RHOAI Konflux pipeline), but the LLM Judge detector only has `Dockerfile.judge` (no Konflux variant). This suggests the LLM Judge detector may be in an earlier maturity stage and not yet shipped as a supported RHOAI component, or it may be deployed through a different mechanism. The Tekton pipeline definitions in `.tekton/` confirm only `odh-built-in-detector` and `odh-guardrails-detector-huggingface-runtime` are built by Konflux CI.
