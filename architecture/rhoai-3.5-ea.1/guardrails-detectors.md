## Purpose

**Short**: A collection of detector algorithm microservices for the FMS Guardrails Orchestrator, providing content safety detection via regex/heuristic, HuggingFace ML model, and LLM-as-a-judge approaches.

**Detailed**: The guardrails-detectors repository provides a set of text content detection microservices designed for use with the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator). These detectors analyze text content for safety concerns, PII, format validation, and arbitrary custom detection logic, returning structured detection results with position information and confidence scores.

The repository produces three distinct detector services, each packaged as a separate container image: (1) a **built-in detector** offering lightweight regex-based PII detection (emails, credit cards, SSNs, phone numbers, IP addresses), file format validation (JSON, YAML, XML with schema support), and user-defined custom Python detection functions; (2) a **HuggingFace runtime detector** that loads HuggingFace AutoModel-based ML models (sequence classification, token classification, and Granite causal LM) for content safety classification; and (3) an **LLM Judge detector** that integrates with vLLM-compatible inference servers to perform LLM-as-a-judge based content evaluation using configurable metrics (safety, toxicity, helpfulness, etc.).

All three detectors expose a common `/api/v1/text/contents` REST endpoint following the IBM Detector API specification, making them interchangeable within the Guardrails Orchestrator framework. They are intended to be deployed as KServe InferenceServices on OpenShift AI, with the HuggingFace and LLM Judge detectors typically running within the KServe serving infrastructure.

## Architectural Analysis

The guardrails-detectors repository follows a well-structured modular architecture with three detector types sharing a common library layer. The `detectors/common/` package provides the `DetectorBaseAPI` FastAPI base class with built-in exception handling, health endpoints, and Prometheus counter instrumentation (detections, requests, errors, runtime). Each detector type extends this base with its specific detection logic while maintaining API compatibility through the shared `/api/v1/text/contents` endpoint.

The built-in detector employs an extensible registry pattern (`BaseDetectorRegistry`) where detector types (regex, file_type, custom) are registered at startup. The custom detector subsystem is particularly noteworthy: it dynamically loads user-provided Python code from `custom_detectors/custom_detectors.py` at startup, but performs static code analysis (AST-based) to block dangerous imports (`os`, `subprocess`, `sys`, `shutil`) and function calls (`eval`, `exec`, `open`, `compile`, `input`). An exception is made for `os.environ` and `os.getenv` access. The custom detector framework also supports user-defined Prometheus instruments via the `@use_instruments` decorator and non-blocking background execution via the `@non_blocking` decorator — these are injected into the loaded module namespace automatically.

The HuggingFace detector is architecturally the most complex, supporting three distinct model architectures (sequence classification, token classification, and Granite causal LM) determined at startup by inspecting the model's `config.architectures` field. Token classification is a recent addition (commit `70ba220`) that provides per-token entity detection with span grouping. The detector handles multi-architecture builds (amd64, ppc64le, s390x) with PyTorch and OpenBLAS compiled from source for non-x86 architectures — this is reflected in the complex multi-stage Dockerfiles. CUDA support is present but optional; the detector falls back to CPU inference gracefully.

The LLM Judge detector is the lightest architecturally, acting essentially as a thin REST-to-REST proxy that translates the FMS Guardrails content analysis API into vllm_judge evaluation calls. It does not run models locally, instead delegating all inference to an external OpenAI-compatible vLLM server. This design means it has very low resource requirements but introduces a hard dependency on a running LLM inference service. Notably, only the built-in and HuggingFace detectors have Konflux Dockerfiles and Tekton pipelines; the LLM Judge detector only has a community Dockerfile, suggesting it may not yet be shipped as an official RHOAI component through the Konflux build pipeline.

All three detectors are designed for deployment as KServe InferenceServices using the `RawDeployment` mode (not serverless/Knative), with Istio sidecar injection for mTLS. The example manifests include `security.opendatahub.io/enable-auth: 'true'` annotations to enable platform authentication. Prometheus metrics are exposed through multiprocess-safe collectors on all services, with the metric prefix `trustyai_guardrails` providing per-detector-function counters for detections, requests, errors, and cumulative runtime latency.
