## Purpose

**Short**: Kubernetes operator that manages the lifecycle of distributed machine learning training jobs across six frameworks: PyTorch, TensorFlow, MPI, XGBoost, PaddlePaddle, and JAX.

**Detailed**: The Training Operator (also known as KFTO — Kubeflow Training Operator) is a controller-runtime-based Kubernetes operator that provides CRD-driven orchestration for distributed training workloads. It defines six Custom Resource Definitions — PyTorchJob, TFJob, MPIJob, XGBoostJob, PaddleJob, and JAXJob — each tailored to the coordination patterns of its respective ML framework.

For each job, the operator creates and manages replica pods (workers, masters, parameter servers, launchers) and headless services that enable inter-pod DNS resolution for distributed communication. The operator handles the full job lifecycle including pod creation with framework-specific environment variables (e.g., `TF_CONFIG` for TensorFlow, `MASTER_ADDR`/`WORLD_SIZE` for PyTorch), status tracking, failure detection with backoff limits, and cleanup policies. It supports optional gang scheduling through Volcano or scheduler-plugins to ensure all replicas are co-scheduled.

In the RHOAI distribution, the operator is deployed into the `opendatahub` namespace with the `kubeflow-` name prefix and is managed by the platform operator (rhods-operator). The RHOAI build uses FIPS-compliant Go compilation (`GOEXPERIMENT=strictfipsruntime`, `CGO_ENABLED=1`) and runs on a UBI9 minimal base image. The operator exposes Prometheus metrics, health/readiness probes, and validating admission webhooks with self-managed TLS certificates.

## Architectural Analysis

The Training Operator follows a **unified framework pattern** where a common controller base (`pkg/controller.v1/common/`) implements the generic job lifecycle — pod reconciliation, service management, status tracking, and cleanup — while framework-specific controllers inject distributed training semantics through the `ControllerInterface`. Each framework controller primarily implements `SetClusterSpec()` to inject framework-specific environment variables (e.g., `TF_CONFIG` JSON for TensorFlow, `MASTER_ADDR`/`WORLD_SIZE` for PyTorch) that enable the training pods to discover each other and coordinate. This design keeps the 6 controller implementations remarkably thin; most are under 300 lines of framework-specific logic.

The **MPI controller is the architectural outlier** — it breaks the unified pattern by managing its own launcher/worker lifecycle rather than delegating to the common pod reconciler. The MPI launcher requires a ConfigMap with `kubexec.sh` (a script that uses `kubectl exec` to invoke commands on worker pods), a dedicated ServiceAccount with pods/exec permissions, and a Role/RoleBinding scoped to the job's namespace. The init container pattern using `kubectl-delivery` (a separate container image that copies `kubectl` into a shared volume) is specific to MPI and adds deployment complexity.

**Gang scheduling integration** is well-abstracted through the `PodGroupControlInterface` with implementations for both Volcano and scheduler-plugins. The operator detects which scheduler is available at startup via Kubernetes RESTMapper CRD discovery. A notable design decision is that Volcano delays pod creation until the PodGroup reaches `Inqueue` phase (preventing wasted resources), while scheduler-plugins does not — pods are created immediately regardless of PodGroup status. This behavioral difference is transparent to job authors but significant for cluster operators managing GPU resources.

**PyTorch receives preferential treatment** in the codebase with two unique capabilities: NetworkPolicy creation (isolating job pods and whitelisting OpenShift monitoring on port 8080) and HPA-based elastic scaling via `ElasticPolicy`. The NetworkPolicy creation is notable because no other framework creates network isolation resources, suggesting PyTorch is the primary production workload in RHOAI. The elastic scaling support generates PyTorch Elastic (`torchrun`) environment variables (`PET_NNODES`, `PET_RDZV_*`) that enable dynamic worker scaling during training — a capability not available to other frameworks.

The operator uses **label-filtered informer caches** (`training.kubeflow.org/operator-name` label selector) to reduce memory consumption, ensuring the controller only watches resources it created rather than all pods/services in the cluster. This is particularly important in multi-tenant RHOAI deployments where thousands of unrelated pods may exist. The `GOMEMLIMIT=460MiB` environment variable in the deployment manifest further constrains Go runtime memory usage relative to the 512Mi container limit, preventing OOM kills from GC pressure.

The webhook architecture uses **self-managed certificates** via the `open-policy-agent/cert-controller` library rather than cert-manager. The rotator creates a self-signed CA, generates server certificates with DNS name `{service}.{namespace}.svc`, stores them in a Kubernetes Secret, and patches the `ValidatingWebhookConfiguration` with the CA bundle. The readiness probe gates on certificate availability, ensuring the operator doesn't accept traffic before webhooks are functional. MPI is the only framework without a validating webhook — it uses a scaffold (no-op) function, meaning invalid MPIJob specs are only caught at reconciliation time.
