## Purpose

**Short**: Kubernetes operator for managing distributed machine learning training jobs across multiple frameworks (PyTorch, TensorFlow, XGBoost, JAX, MPI, PaddlePaddle).

**Detailed**: The Kubeflow Training Operator (KFTO) is a controller-runtime-based Kubernetes operator that provides Custom Resource Definitions (CRDs) for six ML training frameworks: PyTorchJob, TFJob, XGBoostJob, JAXJob, MPIJob, and PaddleJob. Each CRD abstracts the complexity of distributed training by automatically managing pod lifecycle, inter-node networking via headless services, environment variable injection for framework-specific cluster configuration, and job status tracking.

The operator implements a plugin architecture where a shared `JobController` base class handles common reconciliation logic (pod/service creation, status updates, gang scheduling integration, expectation tracking) while framework-specific controllers extend it with custom cluster specification logic (e.g., PyTorch elastic rendezvous, TensorFlow TF_CONFIG, MPI hostfile generation). This design enables consistent behavior across all frameworks while allowing each to define its own distributed training topology.

In RHOAI, the training operator is deployed via kustomize overlays with FIPS-compliant builds (GOEXPERIMENT=strictfipsruntime, CGO_ENABLED=1), Prometheus monitoring via PodMonitor, and RBAC aggregation roles (training-edit, training-view) that integrate with OpenShift's role-based access control system. The operator supports optional gang scheduling via Volcano or scheduler-plugins PodGroups, and integrates with Kueue for multi-cluster job management via the `managedBy` field.

## Architectural Analysis

The training-operator employs a well-structured plugin architecture centered around the `JobController` base class in `pkg/controller.v1/common/`. This base class implements the full reconciliation lifecycle -- pod creation/deletion, service management, status tracking, expectation-based consistency, and gang scheduling integration -- as a reusable foundation. Each framework controller (PyTorch, TensorFlow, etc.) only needs to implement the `ControllerInterface` methods: `SetClusterSpec()` for environment variable injection, `IsMasterRole()` for success determination, and `GetDefaultContainerName()`/`GetDefaultContainerPortName()` for framework defaults. This minimizes code duplication across frameworks while keeping framework-specific logic isolated.

The networking model relies entirely on DNS-based pod discovery via headless services. For each training pod, the operator creates a corresponding headless `ClusterIP: None` service with the naming convention `{jobname}-{replicatype}-{index}`. Framework-specific environment variables (e.g., `MASTER_ADDR`, `TF_CONFIG`, `COORDINATOR_ADDRESS`) reference these DNS names, allowing training processes to discover each other without any external service discovery mechanism. PyTorch additionally uses an init container that blocks worker startup until the master pod's DNS name resolves, preventing race conditions during cluster formation. This pattern is simple but means all training communication flows over unencrypted HTTP within the cluster network -- there is no mTLS or encryption between training pods.

The gang scheduling integration is notable for its abstraction. The `PodGroupControlInterface` provides a common API over both Volcano (`scheduling.volcano.sh/v1beta1`) and scheduler-plugins (`scheduling.x-k8s.io/v1alpha1`) PodGroup implementations. The `DelayPodCreationDueToPodGroup` mechanism ensures pods are not created until the PodGroup reaches "Inqueue" status, preventing partial scheduling of distributed training jobs. The scheduling mode is selected at operator startup via the `--gang-scheduler-name` flag, and the operator validates the corresponding CRD exists before starting.

PyTorch elastic training support is the most sophisticated feature, integrating with PyTorch's Elastic Launch (`torchrun`/`torch.distributed.elastic`). When `ElasticPolicy` is specified, the controller injects rendezvous environment variables (`PET_RDZV_ENDPOINT`, `PET_RDZV_BACKEND`, `PET_NNODES` as a min:max range), enables HPA-based scaling via `autoscaling/v2` MetricSpec, and creates a `NetworkPolicy` allowing ingress from same-job pods plus OpenShift monitoring. The MPI controller is also noteworthy for its unique pattern: it creates a ConfigMap with `kubexec.sh` and `discover_hosts.sh` scripts, plus a ServiceAccount/Role/RoleBinding granting the launcher pod permission to list pods -- effectively bootstrapping a minimal RBAC context within the job's namespace.

The RHOAI integration layer is lightweight but effective. The `manifests/rhoai/` overlay adds RBAC aggregation labels (`rbac.authorization.k8s.io/aggregate-to-edit`, `aggregate-to-view`) to training-edit and training-view ClusterRoles, making training job permissions automatically available to OpenShift admin/edit/view users. The Konflux Dockerfile builds with full FIPS compliance (CGO_ENABLED=1, GOEXPERIMENT=strictfipsruntime), and the base UBI9 image provides the OpenSSL runtime needed for check-payload validation. The cache filtering in `main.go` (label selector on `kubeflow.org/operator-name`) is a critical production optimization -- without it, the operator's informer cache would load all Pods, Services, and ConfigMaps cluster-wide, causing unbounded memory growth.
