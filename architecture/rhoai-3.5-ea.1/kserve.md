## Purpose

**Short**: KServe is a Kubernetes-native model serving platform that provides CRD-based lifecycle management for inference services, including model deployment, autoscaling, ingress routing, and model caching.

**Detailed**: KServe is a component operator within the Red Hat OpenShift AI (RHOAI) platform that manages the full lifecycle of machine learning model serving workloads. It operates through multiple controllers, each managing a distinct CRD: the InferenceService controller (v1beta1) handles traditional model serving with predictor/transformer/explainer components, the LLMInferenceService controller (v1alpha2) manages LLM-specific workloads with disaggregated prefill/decode, Gateway API routing, and WVA-based autoscaling, and the LocalModel controller (v1alpha1) manages node-local model caching via DaemonSets and PersistentVolumes.

The repository also contains the KServe Python SDK, which provides a FastAPI+gRPC model serving framework supporting KServe v1, v2, and OpenAI-compatible inference protocols. The storage-initializer component downloads model artifacts from S3, GCS, Azure Blob, HuggingFace Hub, HDFS, and other backends. Additional components include the inference router (for InferenceGraph traffic splitting), the inference agent (for model downloading and syncing), and a queue-proxy extension (qpext) for Knative metrics.

KServe integrates deeply with the RHOAI platform through build-tagged code paths (the `distro` tag) that enable OpenShift-specific features: OpenShift Route support, service-ca certificate management, SecurityContextConstraints for multi-node workloads, Istio DestinationRules for TLS origination, and Kuadrant AuthPolicy integration for authentication.

## Architectural Analysis

KServe's architecture reflects a multi-generational design where the original InferenceService controller (v1beta1) coexists with the newer LLMInferenceService controller (v1alpha2), each running as a separate binary from independent Dockerfiles. The InferenceService controller supports multiple deployment modes -- Knative serverless, raw Kubernetes Deployments, and ModelMesh -- with ingress handled through VirtualServices, HTTPRoutes, or standard Kubernetes Ingresses depending on cluster capabilities. The LLMInferenceService controller, by contrast, is purpose-built for LLM workloads and exclusively uses Gateway API with InferencePools for routing, LeaderWorkerSets for multi-node tensor parallelism, and a sophisticated WVA (Workload Variant Autoscaling) system that publishes desired replica counts to Prometheus and feeds them back through HPA or KEDA.

The `distro` build tag system is the primary mechanism for platform differentiation. Go files tagged with `//go:build distro` are compiled only in RHOAI/ODH builds, while `//go:build !distro` files provide no-op defaults for upstream. This affects multiple cross-cutting concerns: TLS certificate provisioning (OpenShift service-ca vs self-signed), workload permissions (SecurityContextConstraints for multi-node), ingress preconditions (Kuadrant AuthPolicy CRD checks), and network configuration (Istio DestinationRules for TLS origination through service mesh). The controller Dockerfile passes `-tags distro,strictfipsruntime` to produce the RHOAI binary.

The LLMInferenceService controller implements a sophisticated configuration merging system. LLMInferenceServiceConfig CRs serve as templates (well-known defaults exist for single-node, multi-node, prefill, decode, and data-parallel topologies), and the controller merges these templates with user-specified overrides in a deterministic order. Once merged, config names are pinned in status annotations to ensure upgrade stability. The scheduler component (EPP - Endpoint Picker) is deployed per LLMInferenceService with its own ServiceAccount, RBAC Role, and ClusterRoleBinding to system:auth-delegator, demonstrating a security-conscious per-workload isolation pattern.

A notable FIPS compliance gap exists: while the four primary Go binaries (controller, llmisvc-controller, agent, router) are built with `CGO_ENABLED=1` and `GOEXPERIMENT=strictfipsruntime` for FIPS-compliant dynamic linking, the localmodel controller and localmodel-agent Dockerfiles use `CGO_ENABLED=0` (static linking) without FIPS flags. These binaries would use pure-Go crypto implementations rather than the OpenSSL backend required for FIPS certification. The storage-initializer is Python-based and inherits OpenSSL from the UBI9 base image, so its FIPS posture depends on the Python runtime's crypto provider configuration.

The InferencePool API migration (v1alpha2 to v1) in the LLMInferenceService controller demonstrates a careful backwards-compatible upgrade strategy: both v1 and v1alpha2 InferencePools are created simultaneously, the HTTPRoute backend reference is migrated to v1 only after confirming the Gateway accepts the v1 API group (via `ResolvedRefs=True`), and the migration is tracked via an annotation (`serving.kserve.io/inference-pool-migrated: "v1"`) that prevents rollback. This pattern avoids breaking existing clusters where the Gateway controller may not yet support the v1 InferencePool API.
