## Purpose

**Short**: KServe is a Kubernetes-native model serving platform that provides CRDs and controllers for deploying, scaling, and managing machine learning inference services on OpenShift.

**Detailed**: KServe is the core model-serving component of Red Hat OpenShift AI (RHOAI). It defines the `InferenceService` CRD as the primary abstraction for deploying ML models, supporting multiple deployment modes including raw Kubernetes Deployments, Knative Serving (serverless), and ModelMesh (multi-model serving). The controller watches InferenceService resources and reconciles the full stack: workload Deployments, Services, autoscaling (HPA/KEDA), ingress routing (Kubernetes Ingress, Istio VirtualService, or Gateway API HTTPRoute), and observability (OpenTelemetry collectors).

KServe also provides the `LLMInferenceService` CRD (v1alpha2) for LLM-optimized serving with features like disaggregated prefill/decode, multi-node inference via LeaderWorkerSet, Gateway API-based routing with InferencePool integration, and workload-variant autoscaling (WVA). The `InferenceGraph` CRD enables chaining multiple inference steps (sequence, ensemble, splitter, switch) into directed acyclic graphs. Local model caching (`LocalModelCache`/`LocalModelNamespaceCache`) pre-downloads models to node-local storage for faster startup. A comprehensive webhook system mutates pods to inject storage initializers, model agents, loggers, and batchers as sidecars or init containers.

The RHOAI distribution builds seven container images via Konflux, with FIPS-compliant Go binaries (strictfipsruntime) for the core controller, agent, router, and LLMISvc controller components. The Python-based storage initializer handles model downloads from S3, GCS, Azure, HuggingFace, PVC, HDFS, and OCI sources.

## Architectural Analysis

KServe is a multi-controller architecture with three separate manager binaries (`cmd/manager`, `cmd/llmisvc`, `cmd/localmodel`) that can be deployed independently or together. The main `kserve-controller-manager` handles the established InferenceService, TrainedModel, and InferenceGraph CRDs, while the `llmisvc-controller-manager` is a newer, dedicated controller for the LLMInferenceService CRD (v1alpha2). This separation reflects both organizational boundaries (LLMISvc was developed as a distinct feature) and the different dependency profiles — the LLMISvc controller needs Gateway API, LeaderWorkerSet, WVA, and InferencePool CRDs that the main controller does not.

The InferenceService controller implements a strategy pattern for deployment modes. The reconciler delegates to mode-specific reconcilers: `RawKubeReconciler` (Deployment + Service + Autoscaler), `KsvcReconciler` (Knative Service), or passes through to ModelMesh. Each mode creates different ingress resources — Kubernetes Ingress or Gateway API HTTPRoute for raw mode, Istio VirtualService for Knative mode. The ingress reconciler inspects the `enableGatewayAPI` config flag to decide between HTTPRoute and Kubernetes Ingress. This design allows the same InferenceService CRD to work across different infrastructure stacks without changing the user-facing API.

The webhook system is architecturally significant: the pod mutating webhook (`/mutate-pods`) runs a sequential chain of injectors that transform a bare model server pod into a fully-featured inference endpoint. The StorageInitializerInjector adds an init container for model download, the AgentInjector adds a sidecar for model pulling/logging/batching, the MetricsAggregateInjector configures Prometheus scraping, and the AcceleratorInjector handles GPU node selection. Each injector checks annotations and configuration to decide whether to activate, creating a composable system where features are opt-in per InferenceService. The `ReinvocationPolicy: IfNeeded` ensures that mutations from other webhooks (e.g., Istio sidecar injection) are compatible.

The LLMISvc controller introduces several architectural patterns not present in the main controller: disaggregated prefill/decode scheduling, InferencePool-based routing (migrating from v1alpha2 to v1), self-signed TLS for workload endpoints, and distribution-specific build tags (`//go:build distro` and `//go:build !distro`) for OCP-specific features like DestinationRule creation and SecurityContextConstraints. The `distro` build tag is set via `GOTAGS=distro` in the Konflux Dockerfile, ensuring the RHOAI build includes OCP-specific controller logic.

A notable FIPS compliance gap exists in the localmodel components: both `localmodel.Dockerfile.konflux` and `localmodel-agent.Dockerfile.konflux` build with `CGO_ENABLED=0` (static linking) and no `strictfipsruntime` tags, while the other four Go binaries (controller, agent, router, llmisvc) properly use `CGO_ENABLED=1` with `GOEXPERIMENT=strictfipsruntime`. This means the localmodel controller and node agent binaries use Go's pure-Go crypto implementation instead of OpenSSL, which would fail FIPS check-payload validation. This appears to be an oversight, as these are newer components (added for model caching) that haven't yet been hardened to the same level as the core components.
