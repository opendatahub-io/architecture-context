## Purpose

**Short**: REST API orchestration server that coordinates AI text generation with content safety guardrails by managing multiple detector, chunker, and LLM backend services.

**Detailed**: The FMS Guardrails Orchestrator acts as middleware between clients and language models to provide content safety detection, filtering, and secure text generation. It receives client requests for text generation or chat completions, optionally runs input text through configured content safety detectors (PII detection, harmful content filtering, etc.), forwards validated requests to backend generation services (TGIS, caikit-NLP, or OpenAI-compatible APIs), and then runs output text through additional detectors before returning results to the client.

The orchestrator supports both unary and streaming request patterns across multiple API versions (v1 legacy and v2 current). It manages concurrent connections to multiple downstream services — generation servers via gRPC (TGIS/NLP) or HTTP (OpenAI-compatible), detector services via HTTP REST, and chunker services via gRPC. Each detector can be independently configured with its own threshold, chunker assignment, and TLS settings.

The service is built on a high-performance async Rust stack using tokio, axum (HTTP), and tonic (gRPC), with comprehensive observability via OpenTelemetry (traces and metrics). It runs as a dual-server process with a main API server (default port 8033) and a separate health check server (default port 8034), supporting optional TLS and mTLS on the main API server.

## Architectural Analysis

The FMS Guardrails Orchestrator follows a sophisticated middleware pattern where it sits between clients and LLM backends, intercepting requests to apply safety guardrails. Its core design revolves around a **dual-server architecture** — a main API server on port 8033 handling all guardrails endpoints (with optional TLS/mTLS) and a separate lightweight health server on port 8034. This separation ensures health probes are never blocked by TLS handshake failures and can be accessed even when the main API is undergoing certificate rotation.

The **orchestration engine** uses a task-based execution model where each API endpoint maps to a handler implementing the `Handle<Task>` trait. Handlers compose primitive operations — chunk, detect, generate — into complex pipelines. The streaming handlers are particularly noteworthy: they use an actor-based `DetectionBatcherManager` pattern with mpsc channels to safely aggregate concurrent detection results from multiple detectors processing the same text stream. The `CompletionBatcher` and `MaxProcessedIndexBatcher` implementations provide pluggable batching strategies that enable ordered delivery of detection results interspersed with generation tokens. This design allows incremental safety checking of streaming LLM output without buffering the entire response.

The service supports **three distinct generation backends** — TGIS (IBM's text generation inference via gRPC proto), caikit-NLP (Red Hat's NLP runtime via gRPC proto), and OpenAI-compatible servers (via HTTP REST). The generation provider is configured at startup via the `generation.provider` field (tgis/nlp) or the `openai` config section. Detectors are always HTTP REST clients following the Detector API v1 specification, while chunkers are always gRPC clients using the caikit Chunkers proto. This mixed-protocol design reflects the ecosystem: generation and chunking services are typically caikit-based (gRPC), while detectors are standalone Python services with REST APIs.

The **TLS architecture** deserves attention. The server uses rustls with the ring crypto provider (installed explicitly at startup via `CryptoProvider::install_default()`). Each downstream service can have its own independent TLS configuration referenced by name from a shared `tls` config section, supporting different certificate chains for different backends. The `insecure` option and `NoVerifier` struct exist for development scenarios but represent a security risk if enabled in production. The `passthrough_headers` config and `rewrite_forwarded_access_header` option support integration with external auth systems — when enabled, X-Forwarded-Access-Token headers from the client are rewritten to standard Authorization Bearer headers before forwarding to downstream services.

The observability stack is comprehensive, using OpenTelemetry SDK for both traces and metrics with OTLP export. The tracing middleware tracks per-request spans with method, path, status code, duration, and separate counters for success/4xx/5xx/streaming responses. W3C Trace Context propagation (traceparent header) is injected into all downstream gRPC and HTTP calls, enabling end-to-end distributed tracing across the orchestrator and all backend services. The metrics reader uses a 3-second export interval, and the service suppresses verbose logging from infrastructure crates (ginepro, hyper, h2, tower, tonic, reqwest) via EnvFilter configuration.
