## Purpose

**Short**: A REST API orchestrator that coordinates AI text generation with content safety guardrails by managing detector, chunker, and LLM backend services.

**Detailed**: The FMS Guardrails Orchestrator (internally named `fms-guardrails-orchestr8`) serves as middleware between client applications and language models, enforcing content safety through a pluggable architecture of detector services, chunker services, and generation backends. It receives text generation or detection requests, routes content through configured safety detectors (for both input and output), applies text chunking for efficient processing, and coordinates with LLM backends (via TGIS gRPC, Caikit NLP gRPC, or OpenAI-compatible HTTP APIs) to produce safe text generation results.

The orchestrator exposes both v1 (classification-with-generation) and v2 (detection-centric) REST APIs, supporting unary and streaming response patterns. It supports the OpenAI chat completions and completions API formats when an OpenAI-compatible backend is configured, enabling drop-in integration with vLLM and similar inference servers. The service manages concurrent detection tasks across multiple detector instances, applies threshold-based filtering, and provides health monitoring for all downstream services.

The orchestrator is deployed as a standalone container on RHOAI, providing guardrails for AI workloads. It supports TLS (including mTLS) for both its own server endpoints and all downstream client connections, with configurable TLS per service. OpenTelemetry-based tracing and metrics are built in, supporting both gRPC and HTTP OTLP export protocols.

## Architectural Analysis

The FMS Guardrails Orchestrator follows a clean middleware architecture where the orchestrator sits between client applications and AI inference backends, intercepting text generation requests to apply safety guardrails. The design centers on a task-handler pattern: incoming HTTP requests are validated and converted into typed `Task` structs, which are dispatched through the `Orchestrator` via the `Handle<Task>` trait. Each handler (classification_with_gen, chat_completions_detection, text_content_detection, etc.) implements the orchestration logic for its specific use case, coordinating chunker, detector, and generation client calls.

The client abstraction layer is notable for its heterogeneity — it manages four fundamentally different client types (TGIS gRPC, Caikit NLP gRPC, HTTP detectors, OpenAI-compatible HTTP) through a type-erased `ClientMap` with runtime downcasting. The `GenerationClient` enum further abstracts over TGIS and NLP backends, providing a unified generation interface. This design enables runtime configuration of different backend combinations, but the type-erasure and downcasting pattern trades compile-time safety for flexibility. The retry logic with exponential backoff is only implemented for the NLP provider, not for TGIS or detector clients, which could lead to inconsistent resilience behavior.

The TLS architecture is comprehensive but contains a significant FIPS gap: the entire TLS stack (both server and all outbound clients) uses `rustls` with the `ring` crypto backend, which is a pure Rust/assembly implementation that does not use the FIPS-validated OpenSSL module on RHEL. While OpenSSL is installed in the container image (likely satisfying basic check-payload image scanning), the application's actual TLS operations bypass OpenSSL entirely. The `NoVerifier` struct in `src/utils/tls.rs` provides a way to completely disable certificate verification for insecure configurations — while useful for development, this should be carefully audited in production deployments.

The streaming architecture supports multiple patterns: SSE (Server-Sent Events) for v1 streaming classification and OpenAI chat completions, ND-JSON for v2 streaming content detection, and gRPC bidi streaming for chunker interactions. The streaming content detection endpoint is the most complex, accepting ND-JSON input streams and producing ND-JSON output streams while coordinating with chunker bidi-streaming and detector HTTP calls. The use of `tokio::sync::mpsc` channels and `ReceiverStream` bridges between async task boundaries and Axum's response streams.

The configuration system (`config.yaml`) provides a rich DSL for connecting to multiple detector, chunker, and generation services, each with independent TLS, timeout, and retry settings. Named TLS configurations can be shared across services, reducing configuration duplication. Header passthrough allows propagating authentication tokens from client requests to downstream services, with a specific `X-Forwarded-Access-Token` to `Authorization: Bearer` rewrite feature. The `api_token` field in service configs reads from environment variables at startup, providing a secure way to inject API keys without embedding them in configuration files.
