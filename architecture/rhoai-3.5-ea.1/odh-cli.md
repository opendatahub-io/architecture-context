## Purpose

**Short**: CLI tool for validating, diagnosing, and managing Red Hat OpenShift AI (RHOAI) deployments on Kubernetes clusters.

**Detailed**: `odh-cli` (shipped as `rhai-cli` / `kubectl-odh`) is a kubectl plugin that provides cluster administrators with comprehensive tooling for RHOAI lifecycle management. The CLI connects to Kubernetes clusters via standard kubeconfig authentication and queries the Kubernetes API server to perform cluster health assessments, upgrade readiness validation, workload backup, dependency management, component lifecycle control, operator log aggregation, and cluster migration orchestration.

The tool's primary use case is upgrade readiness assessment via its `lint` command, which runs 30+ version-aware diagnostic checks against live cluster state to identify blocking issues, deprecated configurations, and workload incompatibilities before upgrading between RHOAI versions (e.g., 2.x to 3.x). The lint framework implements a Kubernetes-native DiagnosticResult CR pattern with condition-based validation, impact classification (blocking/advisory), and machine-readable JSON/YAML output for CI/CD integration.

The CLI is distributed as a container image (`quay.io/rhoai/rhoai-upgrade-helpers-rhel9`) based on `ose-cli-rhel9`, bundling the Go binary alongside `oc`, `kubectl`, `jq`, `yq`, `python3`, and legacy upgrade helper scripts from the `rhoai-upgrade-helpers` submodule. It supports multi-architecture builds (amd64, arm64, ppc64le) with FIPS-compliant cryptography for air-gapped and regulated environments.

## Architectural Analysis

The odh-cli is architecturally a **pure Kubernetes client application** with zero server-side components. All data access is mediated through the Kubernetes API server via client-go, with a single exception: the `deps --refresh` command fetches dependency manifests from GitHub over HTTPS. This design makes the tool fully compatible with air-gapped environments when using the embedded (non-refresh) dependency manifest.

The **lint framework** is the most architecturally significant subsystem, implementing a miniature diagnostic engine with 30+ checks organized into four categories (Platform, Components, Dependencies, Workloads). Each check follows a `CanApply → Validate` lifecycle where `CanApply` uses semantic version comparison to determine applicability (e.g., "only on 2.x → 3.x upgrade path") and `Validate` returns a Kubernetes-native `DiagnosticResult` with condition-based findings. The framework enforces **sequential execution** (parallel check execution is explicitly prohibited) to guarantee deterministic output ordering for diff-based workflows and CI/CD integration. Checks use a **read-only client** (`client.Reader` interface) enforced at the type level, ensuring lint checks cannot mutate cluster state.

The CLI's **client throttling** configuration (QPS=50, Burst=100) is deliberately 10x higher than kubectl defaults (QPS=5, Burst=10), optimized for parallel worker patterns used in backup (fan-out resource discovery) and status (concurrent health checks across 8 sections). This is configurable via `--qps` and `--burst` flags for cluster-specific tuning.

The **backup command** implements a dependency resolution graph that discovers workloads (Notebooks, InferenceServices, RayClusters, etc.) and transitively resolves their dependencies (ConfigMaps, PVCs, Secrets) with configurable include/exclude filters and JSONPath field stripping for portable manifests. The resolver uses a registry pattern where each workload type registers its own dependency resolution strategy (e.g., DSPA resolves database secrets, Notebooks resolve connection secrets).

The **migrate subsystem** (list/prepare/run) is designed for orchestrated cluster migrations with a prepare-then-execute workflow, dry-run support, and user confirmation gates. Currently the primary migration implemented is Kueue RHBOK (Red Hat Build of Kueue) migration. The migration framework uses an action registry pattern analogous to the lint check registry, enabling new migrations to be added as discrete packages.

A notable **supply chain security** pattern exists in the deps install command: it validates OLM catalog sources against an allowlist (`redhat-operators`, `community-operators`, `certified-operators`, `redhat-marketplace`) and rejects installations targeting system namespaces (`kube-*`, `openshift-*`), preventing privilege escalation through attacker-controlled catalog sources or target namespaces.
