## Purpose

**Short**: CLI tool for validating, diagnosing, and managing Red Hat OpenShift AI (RHOAI) deployments on Kubernetes clusters.

**Detailed**: odh-cli (also known as `kubectl-odh` or `rhai-cli`) is a command-line interface that serves as both a kubectl plugin and a standalone container-based tool for RHOAI platform operators and administrators. It provides five primary capabilities: (1) **lint** — validates current cluster configuration and assesses upgrade readiness by running a comprehensive suite of checks across platform components, dependencies, and user workloads; (2) **get** — provides simplified access to common RHOAI resources (notebooks, inference services, serving runtimes, pipelines); (3) **deps** — displays operator dependency status by cross-referencing an embedded manifest from odh-gitops against OLM subscriptions on the cluster; (4) **components** — manages RHOAI component lifecycle (list, describe, enable, disable) through DataScienceCluster spec mutations; and (5) **migrate** — executes structured migrations with prepare (backup) and run phases, currently supporting Kueue-to-RHBOK migration.

The tool is designed to be run either as a native binary (`kubectl odh ...`) or from a container image (`quay.io/rhoai/rhoai-upgrade-helpers-rhel9`) that bundles additional utilities including `oc`, `kubectl`, `jq`, `yq`, and Python-based upgrade helper scripts. The container image is built by Konflux with FIPS-compliant Go binaries and targets multi-architecture platforms (amd64, arm64, ppc64le). The CLI uses the Kubernetes API extensively for read operations (listing resources, querying CRDs, checking OLM subscriptions) and limited write operations (patching DataScienceCluster spec, creating OLM subscriptions during migration).

## Architectural Analysis

The odh-cli follows a clean, well-structured Go CLI architecture built on Cobra with a consistent four-phase command lifecycle (AddFlags → Complete → Validate → Run). The separation between `cmd/` (thin Cobra wrappers) and `pkg/` (business logic) is rigorous — command files in `cmd/` contain only flag registration and error handling, delegating all logic to corresponding packages. This enables testability and reuse across commands.

The lint framework is the most architecturally sophisticated subsystem. It implements a registry-based check system where each check is a self-contained unit that declares its target versions, component scope, and severity via metadata. The check executor runs checks concurrently with error isolation — permission errors on individual resources degrade gracefully to "Unknown" results rather than failing the entire lint run. This is critical for a diagnostic tool that may run with limited RBAC permissions. The version-aware selector system (`check.Selector`) allows checks to be filtered by target upgrade version, enabling the same binary to validate different upgrade paths (2.x→3.x, 3.3→3.4, etc.).

A notable design decision is the heavy use of JQ expressions (via the `gojq` library) for Kubernetes resource field extraction. Rather than importing and depending on typed Go structs for every CRD the CLI interacts with (which would create tight coupling to specific CRD versions), the CLI uses unstructured resources with JQ queries like `.spec.components.kserve.managementState`. This provides version-resilient field access at the cost of compile-time type safety. The `pkg/util/jq` package wraps gojq with typed generics (`Query[T]`, `Predicate`, `Transform`) to recover some type safety at the call site.

The dependency management system (`pkg/deps`) uses an interesting hybrid approach: dependency manifests from odh-gitops are embedded at build time via `go:embed` (fetched by `make fetch-deps` from a pinned git commit) but can be refreshed at runtime via `--refresh` flag. This ensures the CLI works offline while allowing users to check against the latest manifest. The pinned commit SHA (`ODH_GITOPS_COMMIT` in Makefile) provides reproducible builds.

The migration framework (`pkg/migrate`) implements a structured action pattern with prepare (backup) and run (execute) phases, dry-run support, and step-level recording. The Kueue→RHBOK migration is the first implementation and demonstrates the pattern well: it backs up ClusterQueues/LocalQueues, installs the RHBOK operator via OLM, patches the DataScienceCluster to set Kueue to Unmanaged, and verifies resources are preserved. The action registry supports pattern-based filtering and version targeting, designed for future migration additions.

The output system uses a structured envelope pattern (`pkg/output/envelope.go`) with `apiVersion: cli.opendatahub.io/v1` for machine-readable JSON/YAML output, complemented by human-readable table output via the generic `Renderer[T]` printer system. All three output formats (table, JSON, YAML) are supported uniformly across commands via functional options. The table renderer supports JQ-based column formatters and struct-to-map auto-extraction via mapstructure, enabling declarative column definitions.
