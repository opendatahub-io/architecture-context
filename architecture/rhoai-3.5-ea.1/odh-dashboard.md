## Purpose

**Short**: The ODH Dashboard is the web-based management console for Red Hat OpenShift AI, providing a unified interface for managing data science projects, model serving, pipelines, notebooks, and platform configuration.

**Detailed**: The ODH Dashboard is a multi-component monorepo that produces the primary user-facing web application for the RHOAI platform. It consists of a React/TypeScript frontend built on PatternFly 6, a Node.js/Fastify backend that proxies Kubernetes API calls and serves static assets, and multiple Go-based Backend-for-Frontend (BFF) sidecar modules that provide domain-specific APIs for features like GenAI Studio, Model-as-a-Service (MaaS), MLflow experiment tracking, AutoML, AutoRAG, Eval Hub, and Model Registry.

The architecture uses Webpack Module Federation to compose the main dashboard shell with independently-built federated modules. Each module runs as a sidecar container in the same pod, serving its own frontend assets and BFF API on a unique port (8043-8743). The main dashboard container proxies requests to these sidecars via in-pod HTTPS. External traffic enters through an HTTPRoute (Gateway API) → kube-rbac-proxy (authentication) → backend/BFF container chain. The dashboard interacts with virtually every RHOAI platform component through Kubernetes CRDs, REST APIs, and service proxies, making it the central integration point for the platform's user experience.

## Architectural Analysis

The ODH Dashboard employs a sophisticated **module federation architecture** that enables independent development and deployment of feature-specific UI modules while maintaining a unified user experience. The main dashboard acts as the "shell" application — it renders the primary layout, navigation, and core pages, then dynamically loads federated modules (GenAI Studio, MaaS, MLflow, etc.) at runtime via Webpack Module Federation. Each federated module consists of a React frontend and a Go-based Backend-for-Frontend (BFF) that run as sidecar containers in the same pod.

**The sidecar BFF pattern is architecturally significant.** Rather than deploying each module as a separate microservice (which would multiply ingress complexity), all modules run as containers in a single Deployment. They share a TLS certificate (from OpenShift service-ca), are exposed through different ports on the same Service, and communicate via localhost. The main Node.js backend proxies requests to BFF sidecars using `@fastify/http-proxy`, forwarding the authenticated user's token via the `x-forwarded-access-token` header. This design minimizes network hops and avoids per-module HTTPRoute/Gateway configuration. However, it creates a "mega-pod" with 9+ containers (main dashboard + kube-rbac-proxy + 7 BFF sidecars), which has implications for resource scheduling, rolling updates, and blast radius — a crash in any BFF restarts the entire pod.

**Service account isolation** is carefully managed in the modular architecture overlay. The main dashboard and kube-rbac-proxy containers use the `odh-dashboard` ServiceAccount (with broad ClusterRole permissions), while all BFF sidecar containers use the `odh-dashboard-modules` ServiceAccount (with limited permissions — only MLflow CRD access, ingress discovery, and SubjectAccessReview creation). This is enforced by disabling `automountServiceAccountToken` and using projected volumes to mount the correct SA token into each container group. This prevents BFF modules from inheriting the dashboard's elevated permissions.

**Authentication flows through three layers.** External requests hit the platform Gateway (data-science-gateway HTTPRoute), which routes to the kube-rbac-proxy sidecar on port 8443. The proxy validates the user's OIDC token, performs a Kubernetes RBAC check (can the user list OpenShift projects?), and injects `X-Auth-Request-User` and `X-Auth-Request-Groups` headers before forwarding to the Node.js backend on port 8080. The backend extracts the user's forwarded access token and uses it for all K8s API calls, ensuring that K8s RBAC is enforced with the user's own permissions. For service proxies (pipelines, model registry, etc.), the user's token is forwarded to the upstream service.

**The dashboard functions as the platform's primary integration hub.** It watches CRDs from at least 15 different operators and proxies REST API calls to 8+ backend services. This creates a wide dependency surface — the dashboard must be updated whenever a new platform component adds CRDs or APIs. The `OdhDashboardConfig` CRD provides a feature flag mechanism with 40+ boolean flags that control which UI sections are visible, allowing gradual feature rollout and distribution-specific customization.

**Inter-BFF communication** is an emerging pattern. The GenAI BFF needs to call the MaaS BFF for certain cross-cutting operations (RHOAIENG-49012). This is configured via environment variables (`BFF_MAAS_SERVICE_NAME`, `BFF_MAAS_SERVICE_PORT`) and uses in-cluster HTTPS with the user's forwarded token. This creates an internal dependency graph between BFF modules that isn't visible in the kustomize manifests alone.
