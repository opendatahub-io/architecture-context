## Purpose

**Short**: The unified web UI for Red Hat OpenShift AI, providing a dashboard for managing notebooks, model serving, pipelines, model registries, and AI/ML workloads through a modular architecture with federated micro-frontends.

**Detailed**: odh-dashboard is a multi-container application that serves as the primary user interface for the RHOAI platform. It consists of a React/TypeScript frontend (the "host" application) and a Fastify-based Node.js backend that proxies Kubernetes API calls and serves static assets. The host application loads federated micro-frontend modules at runtime using Webpack Module Federation, enabling feature teams to independently develop and deploy UI extensions for capabilities like Gen AI, Model Registry, MLflow, MaaS (Models-as-a-Service), AutoML, AutoRAG, and Evaluation Hub.

Each federated module is deployed as a sidecar container in the same pod as the host application, running its own Go-based BFF (Backend-for-Frontend) server that handles domain-specific API calls, proxies to upstream ML services (LlamaStack, MLflow, Pipeline Server, MaaS Gateway, etc.), and serves the module's compiled frontend assets. The modular architecture enables RHOAI to ship new AI capabilities as independently built and versioned container images while maintaining a unified user experience. Authentication is enforced at the pod edge by kube-rbac-proxy, which validates user tokens before forwarding requests to the backend containers.

## Architectural Analysis

The odh-dashboard is architecturally notable for its Module Federation approach to extensibility. Rather than building a monolithic frontend, the host application (frontend/) defines a shell with shared dependencies (React, PatternFly, react-router) and dynamically loads federated remote modules at runtime. Each module (model-registry, gen-ai, maas, mlflow, eval-hub, automl, autorag) ships as an independent container image containing both a React frontend bundle and a Go BFF server. In production, all 8 containers (host + 7 modules) run as sidecars in a single pod, sharing a single Kubernetes Service with multiple ports (8443, 8043, 8143, 8243, 8343, 8543, 8643, 8743). The Node.js backend dynamically proxies requests to BFF sidecars based on the `federation-config` ConfigMap, which maps URL paths like `/_mf/genAi/*` to the appropriate sidecar port.

The authentication architecture implements a multi-layer chain. External traffic enters through the platform's Gateway API (`data-science-gateway` HTTPRoute), hits the `kube-rbac-proxy` sidecar on port 8443, which performs an OpenShift RBAC check (can the user `list projects`?), then injects identity headers (`x-auth-request-user`, `x-auth-request-groups`) and the original token as `x-forwarded-access-token`. The Node.js backend uses these headers for its own authorization decisions (admin checks via SelfSubjectAccessReview against `services.platform.opendatahub.io/auths`), and passes the token downstream to K8s API calls and BFF sidecars. The Go BFFs extract the token from the same header and forward it to upstream ML services, maintaining end-to-end user identity throughout the proxy chain.

Inter-BFF communication is a notable architectural pattern: the Gen-AI BFF makes direct HTTPS calls to the MaaS BFF on port 8243 for token operations, using the same `x-forwarded-access-token` header for auth. This creates a service mesh within the pod where BFFs can compose functionality without the user making additional round-trips through the ingress chain. The networking is secured by NetworkPolicy that restricts ingress to only the OpenShift ingress controller and same-namespace pods, while allowing broad egress to support the many upstream services the dashboard communicates with.

A key security consideration is the SSRF protection implemented in the AutoML and AutoRAG BFFs' S3 clients, which validate that S3 endpoints resolve to RFC-1918 private IPs only, rejecting loopback, link-local, and reserved ranges. This is critical because these BFFs accept user-provided S3 connection details. The `InsecureSkipVerify` flag is available across all BFFs but is gated behind the `DEV_MODE` environment variable and defaults to false, ensuring it cannot be accidentally enabled in production.

The deployment manifests use a layered kustomize structure where `core-bases/base` defines the foundational deployment (2 replicas with pod anti-affinity across zones), and `modular-architecture` patches in the 7 sidecar containers via JSON6902 patches. RHOAI and ODH overlays customize branding (deployment names, section titles, ConsoleLinks, app catalogs) while sharing the same base infrastructure. The `federation-config` ConfigMap is the runtime contract between the Node.js proxy layer and the sidecar BFFs — it defines which modules exist, their service ports, proxy path rewrites, and auth requirements. Additionally, the Perses observability dashboard and MLflow Embedded services are configured as proxy-only or external Module Federation entries in this same ConfigMap, demonstrating the extensibility of the federation pattern beyond the dashboard's own modules.
