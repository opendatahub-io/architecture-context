## Purpose

**Short**: An intelligent inference request scheduler that makes optimized routing decisions for LLM inference requests within the llm-d framework, implementing the Endpoint Picker (EPP) and a P/D routing sidecar.

**Detailed**: The llm-d-inference-scheduler provides two primary components for the llm-d inference framework: the Endpoint Picker (EPP) and the P/D routing sidecar. The EPP extends the Gateway API Inference Extension (GIE) project to schedule incoming inference requests arriving through a Kubernetes Gateway API gateway. It uses a pluggable architecture with configurable filters, scorers, profile handlers, and deciders to route requests to optimal model-serving pods based on KV-cache locality, load distribution, session affinity, context length, and role-based disaggregation across Encode/Prefill/Decode (E/P/D) pipeline stages.

The routing sidecar (pd-sidecar) runs alongside vLLM model-serving pods as an init container with `restartPolicy: Always` (native sidecar). It acts as an HTTP reverse proxy that intercepts inference requests and orchestrates the disaggregated inference pipeline — coordinating KV cache transfers between prefill and decode instances using pluggable connectors (NIXL v2, shared storage, SGLang). The sidecar also supports data-parallel inference across multiple GPU ranks and an experimental Encoder stage for multimodal content preprocessing.

Together, these components enable sophisticated LLM inference scheduling that balances cache efficiency, load distribution, and disaggregated execution — critical for operating large-scale LLM inference clusters on RHOAI.

## Architectural Analysis

The llm-d-inference-scheduler is architecturally split into two distinct runtime components — the EPP (Endpoint Picker) and the P/D routing sidecar — which work in concert but run as separate processes. The EPP acts as the "brain" that makes scheduling decisions, while the sidecar acts as the "hands" that orchestrate the actual disaggregated inference pipeline. This separation is deliberate: the EPP runs as a centralized scheduler (referenced by InferencePool), while the sidecar is distributed across every decode pod. The communication between them is implicit — the EPP sets HTTP headers (x-prefiller-host-port, x-encoder-hosts-ports, x-data-parallel-host-port) on the routed request, and the sidecar reads these headers to determine which disaggregation protocol to execute.

The plugin system is remarkably extensible, with 21+ registered plugins across 7 categories (filters, scorers, profile handlers, deciders, pre-request handlers, prepare-data, and data-layer). The architecture follows the GIE framework's lifecycle hooks pattern: PrepareData → Filter → Score → ProfileHandler.Pick → PreRequest → (execution) → ResponseComplete. The disaggregation profile handler (`disagg-profile-handler`) is particularly sophisticated — it orchestrates a multi-stage scheduling flow where decode endpoints are selected first (mandatory), then encode and prefill endpoints are conditionally selected based on decider plugins. This enables runtime-flexible E/P/D pipeline composition without code changes, purely through YAML configuration.

The KV-cache aware scheduling is the most complex subsystem. The `precise-prefix-cache-scorer` maintains a real-time in-memory KV-block index by subscribing to ZMQ events from vLLM engines. It tokenizes incoming requests, hashes them into KV-blocks, and scores endpoints by how many prefix blocks match their cached state. The speculative indexing feature is particularly noteworthy — after routing a request to an endpoint, it immediately writes predicted cache entries to the index, closing the "blind spot" between when a routing decision is made and when the actual KV events arrive. This is backed by a TTL cache (default 2s) that auto-evicts speculative entries that aren't confirmed by real events.

The sidecar's KV transfer protocols each address different distributed inference architectures: NIXL v2 (the default) uses a sequential prefill→decode flow with explicit kv_transfer_params passing; shared storage uses an optimistic "decode-first" approach with cache_hit_threshold fallback; SGLang uses concurrent prefill and decode with bootstrap-room-based coordination; and the EPD protocol fans out multimodal items to encoder pods in round-robin fashion. The sidecar also implements SSRF protection by watching InferencePool resources and maintaining an allowlist of pod IPs, preventing header-injection attacks that could redirect traffic to arbitrary hosts.

One notable design choice is the use of variable substitution (`${EPP_NAME}`, `${POOL_NAME}`, etc.) throughout the deployment manifests rather than standard kustomize parameterization. This suggests the manifests are consumed by a higher-level deployment tool or script that performs string substitution before kustomize processing, rather than using kustomize's native `vars` or `replacements` features. The deployment manifests are primarily for development and testing — in production RHOAI, the platform operator manages the actual deployment of these components.
