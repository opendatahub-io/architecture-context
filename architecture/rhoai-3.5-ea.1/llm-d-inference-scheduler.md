## Purpose

**Short**: Intelligent inference request routing and scheduling service for the llm-d framework, implementing Envoy ext-proc based endpoint selection and disaggregated prefill/decode proxy capabilities.

**Detailed**: The llm-d-inference-scheduler provides two primary components for optimizing LLM inference workload distribution. The Endpoint Picker Processor (EPP) acts as an Envoy external processing (ext-proc) service that intercepts inference requests via a Kubernetes Gateway API infrastructure and routes them to optimal model-serving endpoints using a pluggable scheduling framework with filters, scorers, and pickers. It extends the upstream Gateway API Inference Extension (GIE) project with llm-d-specific features such as Prefill/Decode (P/D) and Encoder/Prefill/Decode (E/P/D) disaggregation, prefix cache-aware scoring, KV cache utilization scoring, and flow control with fairness policies.

The PD Sidecar is a reverse proxy deployed alongside model-serving containers (vLLM, SGLang) that implements the disaggregated inference protocol. It receives routing decisions from the EPP via HTTP headers and orchestrates the KV cache transfer between prefill and decode stages using pluggable connector protocols (NIXL V2, Shared Storage, SGLang). The sidecar also supports data-parallel configurations where multiple model instances share a single endpoint. Together, these components enable intelligent, cache-aware, SLO-driven scheduling of LLM inference requests across heterogeneous GPU pools with support for disaggregated prefill/decode architectures.

## Architectural Analysis

The llm-d-inference-scheduler represents a sophisticated extension of the upstream Gateway API Inference Extension (GIE) project, adding llm-d-specific scheduling intelligence while maintaining compatibility with the GIE plugin framework. The architecture follows a clear separation of concerns: the EPP handles scheduling decisions (which endpoint to route to), while the Sidecar handles execution (how to coordinate multi-stage inference). This decoupling allows independent scaling and deployment of the scheduling and execution layers.

The plugin framework is one of the most architecturally significant aspects. With 40+ registered plugins spanning scheduling filters, scorers, pickers, flow control policies, data layer sources, and request handlers, the system achieves high configurability through the EndpointPickerConfig YAML format. The topological dependency sorting (via Kahn's algorithm in `data_graph.go`) ensures plugins that produce data are executed before plugins that consume it, enabling complex scheduling pipelines like "tokenize request → compute prefix hash → score by KV cache affinity → pick highest scorer" to be expressed declaratively. The two-phase configuration loading (parse + instantiate) with system defaults auto-injection makes this extensibility practical.

The disaggregated inference support is implemented across both components with three connector protocols (NIXL V2, Shared Storage, SGLang), each optimized for different deployment scenarios. NIXL V2 uses vLLM's native KV transfer parameter passing for sequential prefill-then-decode. Shared Storage leverages a shared filesystem with a cache-hit-threshold optimization that attempts decode-first and falls back to prefill only on cache miss. SGLang uses concurrent prefill and decode with a bootstrap protocol for KV cache coordination. The E/P/D extension adds multimodal encoding as a third stage with fanout distribution across encoder pods. The Sidecar's SSRF protection via InferencePool-derived pod allowlists is a notable security feature that prevents routing headers from being used to access arbitrary network endpoints.

The flow control subsystem adds enterprise-grade request management with pluggable fairness policies (global strict priority, round-robin), ordering policies (FCFS, Earliest Deadline First, SLO-deadline), saturation detection (concurrency-based and utilization-based), and eviction with priority-then-time ordering. The sharded registry with leasing enables distributed flow control across multiple EPP replicas. This is critical for production deployments where request queuing and fair resource allocation prevent individual tenants or models from monopolizing GPU resources.

A notable FIPS compliance gap exists: the EPP Konflux Dockerfile uses `CGO_ENABLED=1` (required for check-payload) but omits the `-tags strictfipsruntime` build tag that the Sidecar Dockerfile includes. This means the EPP binary is dynamically linked (passing check-payload validation) but may not use FIPS-validated cryptographic routines at runtime. Additionally, the default InsecureSkipVerify=true for model server metrics scraping is a security concern in production environments, though it is configurable.
