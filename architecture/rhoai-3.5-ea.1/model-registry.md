## Purpose

**Short**: Central repository for storing and managing ML model metadata, versions, artifacts, and serving environments via a REST API, with a web UI, model catalog, Kubernetes controller for KServe integration, and a storage initializer for model downloads.

**Detailed**: Model Registry (formerly Kubeflow Model Registry, now part of Kubeflow Hub) is a multi-component platform service that provides a centralized store for ML model metadata. It replaces the legacy ml-metadata C++ gRPC service with a native Go REST API backed by MySQL or PostgreSQL databases. The service manages registered models, model versions, model artifacts, serving environments, inference services, experiments, and experiment runs through an OpenAPI v3-defined REST API.

The component includes a Kubernetes controller that watches KServe InferenceService custom resources and synchronizes their state into the Model Registry, enabling bidirectional tracking between deployed inference endpoints and registered model metadata. A Container Storage Initializer (CSI) component integrates with KServe to download models from the registry using a custom `model-registry://` URI scheme.

The platform also ships a React/TypeScript web UI with a Go Backend-for-Frontend (BFF) that proxies requests to Model Registry instances and a Model Catalog service. The Model Catalog is a separate subcommand of the same binary that aggregates model metadata from external sources (HuggingFace, YAML catalogs) and MCP server catalogs, backed by its own PostgreSQL database with leader election for HA deployments. An async-upload Python job handles background model transfer operations between storage backends.

## Architectural Analysis

Model Registry exhibits a multi-faceted architecture that has evolved from its origins as a Kubeflow metadata store into a comprehensive ML model lifecycle management platform. The repository module path (`github.com/kubeflow/hub`) reflects its recent rebranding, while the container labels still reference `odh-model-registry`, showing the transition in progress. The Go module structure uses multiple `go.mod` files (`cmd/controller/go.mod`, `cmd/csi/go.mod`, `pkg/inferenceservice-controller/go.mod`, `pkg/openapi/go.mod`) to isolate dependency trees — the controller depends on heavy KServe/controller-runtime dependencies while the main API server stays lightweight with just GORM, chi router, and database drivers.

The database layer uses an "EmbedMD" pattern — the service embeds ML Metadata semantics (artifacts, contexts, executions, properties) into a relational schema managed via golang-migrate. This is significant because it replaces what was previously a separate C++ gRPC service (ml-metadata/MLMD) with native Go code, eliminating a cross-language process boundary. The schema supports both MySQL and PostgreSQL with parallel migration directories, and uses GORM for the ORM layer with auto-generated model structs (via gorm-gen). The OpenAPI code generation (openapi-generator for both server stubs and Go client) and goverter-based converter generation indicate a strong contract-first design approach.

The BFF (Backend-for-Frontend) pattern is notable — the Go BFF handles Kubernetes service discovery dynamically, finding Model Registry service endpoints by querying the Kubernetes API at request time rather than using static configuration. It also implements namespace-scoped access control by creating SubjectAccessReview resources against the Kubernetes API, implementing fine-grained RBAC at the UI layer. The three deployment modes (kubeflow, federated, standalone) reflect the component's need to operate in different platform contexts with different authentication and service discovery strategies.

The Model Catalog is architecturally distinct — it runs as a separate subcommand of the same binary but requires its own PostgreSQL database and implements leader election via `cirello.io/pglock` for HA deployments. It aggregates data from external sources (HuggingFace, YAML catalogs) and the emerging MCP server catalog, demonstrating the component's evolution beyond simple model metadata into a broader AI asset discovery platform. The leader election ensures that only one instance performs expensive external API polling and data ingestion while all instances serve read traffic.

A security concern is the hardcoded `InsecureSkipVerify: true` in the inference service controller's HTTP transport (`pkg/inferenceservice-controller/controller.go`), which disables TLS certificate verification when the controller communicates with Model Registry instances. While mitigated by the `SKIP_TLS_VERIFY` environment variable defaulting to false in the kustomize params, the code path that enables it does not validate the flag — it unconditionally sets `InsecureSkipVerify: true` on the transport. The BFF also supports `--insecure-skip-verify` but defaults to secure (false) and uses TLS 1.3 minimum when TLS is enabled, which is the correct approach.
