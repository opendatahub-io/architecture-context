## Purpose

**Short**: A central metadata repository for registering, versioning, and serving ML models, with a model catalog for curated model discovery and an async upload job for model transfer between storage backends.

**Detailed**: Model Registry is a multi-component system that provides a REST API for ML model lifecycle management — registering models, tracking versions, recording artifacts, managing serving environments and inference services. The core server is a Go-based REST API (OpenAPI v1alpha3) backed by MySQL or PostgreSQL, serving as the metadata store that replaces the legacy C++ MLMD gRPC service with a higher-level HTTP API.

The repository also contains a Model Catalog service that aggregates and indexes models from external catalog sources (e.g., HuggingFace) and MCP (Model Catalog Protocol) servers, providing curated discovery and browsing APIs. A Kubernetes controller watches KServe InferenceService CRDs and synchronizes their state with the Model Registry, enabling bidirectional tracking between deployed models and their registry metadata. A CSI storage initializer enables KServe inference pods to download model artifacts directly from the registry using `model-registry://` URIs. An async upload job handles background transfer of models between storage backends (S3, OCI, HTTP) with optional digital signing.

The system is designed for the Red Hat OpenShift AI (RHOAI) platform but also supports Open Data Hub (ODH) deployments. It integrates with Istio service mesh for mTLS and authorization, and with KServe for model serving orchestration.

## Architectural Analysis

Model Registry follows a multi-binary monorepo pattern: five distinct deployable components (`proxy`, `catalog`, `controller`, `csi`, `async-upload`) share a single Go module and Python package structure. The proxy and catalog services are subcommands of the same `model-registry` binary (via Cobra CLI), but are deployed as separate pods with independent lifecycles. This design enables code sharing (OpenAPI types, converter logic, database layer) while maintaining deployment isolation.

The embedded metadata ("embedmd") architecture is a significant design choice. Rather than depending on an external MLMD gRPC service (as the original Kubeflow Model Registry did), this implementation embeds a GORM-based SQL layer directly into the proxy binary. Database schema management uses `golang-migrate` with numbered migrations, and the datastore layer implements a pluggable connector pattern — though currently only "embedmd" is implemented. The proxy starts the HTTP server immediately and initializes the database connection asynchronously, allowing it to serve health endpoints before the database is ready. The `DynamicRouter` pattern enables swapping in the real API router only after the database connection succeeds, returning HTTP 503 for all data requests during initialization.

The catalog service introduces distributed coordination via PostgreSQL-based leader election (`pglock`). All replicas serve read-only REST API requests, but only the leader performs file watching, config reloads, and database writes. This dual-mode architecture (read-only standby vs. leader) simplifies horizontal scaling while maintaining data consistency. The leader election heartbeat (15s default) and lock duration (60s default) are configurable, with a validation constraint that heartbeat must be ≤ lockDuration/2. The catalog supports both traditional model catalogs and MCP (Model Catalog Protocol) servers, with a unified loader that coordinates both protocols through a composite pattern.

The controller-CSI pair is notable for its lightweight integration approach. Rather than implementing a full operator with CRD management, the controller simply watches existing KServe InferenceService CRDs and mirrors their state into the Model Registry via REST API calls. The CSI storage initializer implements KServe's `Provider` interface to resolve `model-registry://` URIs into concrete storage URIs (S3, GCS, Azure), then delegates the actual download to KServe's existing storage providers. This delegation pattern avoids reimplementing storage protocol handling.

A security consideration: the controller's OpenAPI client has `InsecureSkipVerify: true` hardcoded when communicating with the Model Registry API. While this is mitigated by Istio mTLS in production, it means the controller does not independently verify the identity of the Model Registry service at the TLS layer. The proxy server's database TLS configuration is more robust — it supports configurable cipher suites with an explicit rejection of insecure ciphers via `tls.InsecureCipherSuites()` check, and enforces `MinVersion: tls.VersionTLS12`.
