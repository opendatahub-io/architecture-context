## Purpose

**Short**: TrustyAI is a Java library and Quarkus REST service providing AI explainability, fairness metrics, and data drift detection for ML models served via KServe and ModelMesh on the RHOAI platform.

**Detailed**: TrustyAI Explainability is a multi-module Maven project that provides the core AI explainability capabilities for Red Hat OpenShift AI. The project consists of four main modules: `explainability-core` (the core Java library implementing LIME, SHAP, counterfactual, and time series saliency explainers along with fairness metrics, drift detection, and NLP evaluation metrics), `explainability-connectors` (gRPC and HTTP connectors implementing the KServe v1/v2 inference protocol for communicating with model servers), `explainability-arrow` (Apache Arrow-based data interchange for bridging Java and Python TrustyAI implementations), and `explainability-service` (a Quarkus REST service exposing the library's capabilities as HTTP endpoints with Prometheus metrics integration).

The service consumes inference payloads from ModelMesh (via a REST endpoint at `/consumer/kserve/v2`) and KServe (via Knative CloudEvents), stores and reconciles them into internal dataframes, and then provides on-demand and scheduled fairness metrics (Statistical Parity Difference, Disparate Impact Ratio), drift detection (KS test, Fourier MMD, mean shift), and model explanations (LIME, SHAP, counterfactual). Metrics are published to Prometheus on a configurable schedule. The service supports multiple storage backends: PVC-backed flat files, MinIO object storage, in-memory storage, and MariaDB/MySQL via Hibernate ORM. In the RHOAI distribution, local and global explainer endpoints are disabled by default, while fairness and drift endpoints remain enabled.

## Architectural Analysis

TrustyAI Explainability follows a clean multi-module Maven architecture that separates core algorithmic capabilities from protocol integration and service concerns. The `explainability-core` module is a pure library with no framework dependencies, making it reusable across different deployment contexts including the Python TrustyAI library (via the Arrow bridge module). The `explainability-connectors` module encapsulates all KServe/ModelMesh protocol details behind the `PredictionProvider` interface, allowing the core algorithms to remain transport-agnostic. This separation enables the same LIME or SHAP implementation to work against both gRPC and HTTP model servers without code changes.

The service module uses Quarkus's CDI-based dependency injection with `@LookupIfProperty` annotations to implement a strategy pattern for storage backends. The `StorageFormat` enum (MEMORY, PVC, MINIO, DATABASE) selects which `Storage` implementation is activated at startup, with corresponding `DataSource` implementations managing metadata and dataframe persistence. This architecture allows the same service binary to be deployed with different storage backends based solely on configuration, which is critical for the operator-managed deployment model where the operator sets `service.storage-format` via the `trustyai-config` ConfigMap.

The inference payload reconciliation system is a key architectural pattern. Both ModelMesh (`ModelMeshInferencePayloadReconciler`) and KServe (`KServeInferencePayloadReconciler`) produce partial payloads — inference inputs arrive separately from outputs. The reconcilers match input/output pairs by ID, parse them using the KServe v2 protocol, and convert them to TrustyAI's internal `Dataframe` representation. This decoupled ingestion model means the service can handle both synchronous REST payloads (ModelMesh) and asynchronous CloudEvents (KServe) through a common reconciliation pipeline.

The RHOAI distribution profile (`%rhoai` in `application.properties`) disables all local and global explainer endpoints while keeping fairness and drift metrics enabled. This is a deliberate product decision — explainer endpoints require outbound connections to model servers and are computationally expensive, while fairness/drift metrics operate purely on stored data. The `@EndpointDisabled` annotation pattern provides compile-time-safe feature flagging without requiring separate builds. The Prometheus metrics scheduler runs on a configurable interval (default 5s in dev, operator-configured in production) and publishes gauges with the `trustyai_` prefix, enabling platform-level monitoring dashboards.

The gRPC connectors use a semaphore-based concurrency limiter (default: 5 concurrent requests) and create fresh `ManagedChannel` instances per request with automatic cleanup. While this ensures resource isolation, it means no connection pooling — each explainer request establishes a new gRPC channel to the model server. The connectors currently use plaintext gRPC channels, which is appropriate for cluster-internal communication but means TLS enforcement for model server connections must be handled at the network/mesh layer.
