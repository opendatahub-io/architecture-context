## Purpose

**Short**: TrustyAI Explainability is a Java-based Quarkus REST service that provides fairness metrics, model drift detection, and explainable AI (XAI) algorithms for ML models deployed on KServe and ModelMesh.

**Detailed**: TrustyAI Explainability is the core AI fairness and explainability engine for Red Hat OpenShift AI. It receives inference data from model serving platforms (KServe v1/v2 and ModelMesh) via REST and CloudEvent endpoints, stores that data in configurable backends (PVC flat files, MinIO S3, or MariaDB/MySQL databases via Hibernate ORM), and computes fairness metrics (Disparate Impact Ratio, Statistical Parity Difference), drift metrics (Kolmogorov-Smirnov, Fourier MMD, Meanshift, Jensen-Shannon), and local/global explanations (LIME, SHAP, Counterfactual, Time Series Saliency, Partial Dependence Plots).

The service integrates with Prometheus for scheduled metric publication, enabling dashboard-driven monitoring of model fairness and data drift over time. It exposes a comprehensive REST API with OpenAPI documentation and supports gzip-compressed uploads for large datasets. In RHOAI deployments, the explainer endpoints (LIME, SHAP, Counterfactual, TSSaliency, Global) are disabled by default via Quarkus profile configuration, while fairness and drift endpoints remain enabled. The service is deployed per-namespace by the TrustyAI Operator (`trustyai-service-operator`), which manages its lifecycle, PVC provisioning, and integration with model serving infrastructure.

## Architectural Analysis

TrustyAI Explainability follows a modular Maven multi-module architecture with clear separation of concerns. The `explainability-core` module (197 Java files) contains all XAI algorithms and fairness/drift metrics as a reusable library, while `explainability-connectors` provides the model serving integration layer with both KServe v1/v2 HTTP and v2 gRPC protocols. The `explainability-service` module wraps these as a Quarkus REST service with configuration-driven feature toggling. This design allows the core algorithms to be used independently (e.g., in the TrustyAI Python library via `explainability-arrow`) without requiring the service runtime.

The service uses a strategy pattern for storage backends, selecting between PVC flat-file, MinIO S3, and Hibernate database storage via the `service.storage-format` configuration property. Each backend is a CDI bean activated by `@LookupIfProperty`, ensuring only the configured backend is instantiated. The inference payload reconciliation system (`InferencePayloadReconciler`) handles the async nature of model serving: inference requests and responses arrive as separate partial payloads that must be matched by ID before being stored as complete records. Separate reconcilers exist for KServe (CloudEvent-based) and ModelMesh (REST-based) patterns, with the ModelMesh reconciler performing additional model ID normalization (stripping `__isvc` suffixes).

The Prometheus integration is notable: the `PrometheusScheduler` runs periodic metric calculations at a configurable interval (default 5s in dev, configurable via `service.metrics-schedule`), computing all registered fairness and drift metrics against the stored inference data, and publishing results as Prometheus gauges with the `trustyai_` prefix. Clients register metric computation requests (specifying model, protected attributes, thresholds) via the REST API, and these are stored in-memory and recalculated each cycle. The `MetricsConfig` provides sensible defaults for metric thresholds (SPD: -0.1 to 0.1, DIR: 0.8 to 1.2, drift delta: 0.05).

A significant architectural decision is the profile-based feature gating: in RHOAI deployments, all explainer endpoints (LIME, SHAP, Counterfactual, TSSaliency, Global PDP) are disabled via `@EndpointDisabled` annotations driven by the `rhoai` Quarkus profile. This reduces attack surface and resource consumption for production deployments where only fairness monitoring and drift detection are required. The gzip request filter (`GzipRequestFilter`) specifically targets the `/data/upload` endpoint, supporting compressed uploads for large inference datasets. The service also logs errors to `/dev/termination-log` for Kubernetes pod termination message visibility, enabling operators to diagnose startup failures without log aggregation.

The gRPC prediction channel used by explainer endpoints creates a new `ManagedChannel` per prediction request with `usePlaintext()`, which is a security gap for production deployments. However, since explainers are disabled in the RHOAI profile, this is mitigated in practice. The concurrency is controlled by a `Semaphore` (default 5 concurrent requests) to prevent resource exhaustion during explanation computation, which involves multiple prediction calls to the model server.
