## Purpose

**Short**: MLflow is the experiment tracking, model registry, and artifact storage server for Red Hat OpenShift AI, providing a REST API and web UI for managing ML experiments, model versions, and artifacts with Kubernetes-native authentication and multi-tenant workspace support.

**Detailed**: MLflow is a comprehensive ML lifecycle management platform that serves as the experiment tracking and model registry component within RHOAI. The Red Hat fork (version 3.10.1+rhaiv.3) extends upstream MLflow with Kubernetes-native authentication via ServiceAccount tokens, multi-tenant workspace isolation mapped to Kubernetes namespaces, and hermetic Konflux-based container builds for all supported architectures (amd64, arm64, ppc64le, s390x).

The server runs as a hybrid Flask/FastAPI application behind Uvicorn, exposing protobuf-generated REST APIs (v2.0 and v3.0) for experiment management, run tracking, metric logging, model registration, and artifact storage. It supports PostgreSQL as its primary production database backend via SQLAlchemy, with Alembic-managed schema migrations (50 migration files). The artifact storage layer supports multiple backends (S3, GCS, Azure Blob, local filesystem) through a plugin-based repository registry.

For RHOAI deployment, the server is configured with `--app-name kubernetes-auth` for ServiceAccount-based authentication, `--enable-workspaces` for multi-tenant isolation, and `--workspace-store-uri kubernetes://` to map Kubernetes namespaces to MLflow workspaces via the external `mlflow-kubernetes-plugins` package. The React-based web UI is built and bundled into the container image during the multi-stage Docker build, including a federated module build for integration with the RHOAI dashboard.

## Architectural Analysis

The RHOAI MLflow deployment represents a significant extension of upstream MLflow with Kubernetes-native multi-tenancy. The most architecturally interesting aspect is the three-layer authentication and workspace stack: the `kubernetes-auth` app plugin wraps the standard Flask application with permission-based authorization; the Kubernetes request auth provider reads ServiceAccount tokens from the projected volume mount with a 60-second TTL cache; and the `mlflow-kubernetes-plugins` external package maps Kubernetes namespaces to MLflow workspaces via the `kubernetes://` URI scheme. This creates a system where each Kubernetes namespace can be an isolated MLflow workspace, with access controlled by Kubernetes RBAC rather than MLflow's built-in user management. The workspace context is propagated through Python ContextVars on a per-request basis, ensuring thread-safe isolation.

The server's dual Flask/FastAPI architecture is worth noting. The core protobuf-generated REST API runs on Flask (via generated handler registration from proto descriptors), while newer endpoints (OTLP trace ingestion, job management, gateway API) use native FastAPI routers. FastAPI wraps Flask via Starlette's WSGIMiddleware, allowing both frameworks to coexist on a single Uvicorn server. This means security middleware must be duplicated across both frameworks — Flask gets `security.init_security_middleware()` while FastAPI gets `init_fastapi_security()` — introducing a risk of security policy drift between the two layers. Both layers implement DNS rebinding protection (host header validation), CORS blocking for state-changing methods, and security header injection.

The envelope encryption system for gateway secrets (`mlflow/utils/crypto.py`) follows best practices: PBKDF2-HMAC-SHA256 with 600,000 iterations for key derivation, AES-256-GCM for both data encryption and key wrapping, and Additional Authenticated Data (AAD) combining secret ID and name to prevent ciphertext substitution attacks. Key rotation is atomic and idempotent. However, the Key Encryption Key (KEK) passphrase is managed via an environment variable (`MLFLOW_CRYPTO_KEK_PASSPHRASE`), which requires careful Secret management in Kubernetes.

The build pipeline demonstrates mature hermetic build practices with notable gaps. The ODH pipeline runs fully hermetic (`hermetic: 'true'`), while the RHOAI pipeline runs non-hermetic (`hermetic: 'false'`) due to yarn/npm prefetching limitations, though Cachi2 prefetch is still used for pip and RPM dependencies. The multi-stage Dockerfile efficiently separates the Node.js UI build, Python wheel compilation, and minimal runtime image, resulting in a UBI9-minimal-based final image with only the necessary runtime dependencies. The image runs as non-root (UID 1001) and exposes only port 5000.

The database layer supports four SQL backends (PostgreSQL, MySQL, SQLite, MSSQL) with 50 Alembic migration files tracking schema evolution. The workspace-aware store variants (`WorkspaceAwareSqlAlchemyStore`) add workspace column filtering to all queries, enabling true multi-tenant data isolation at the SQL level. Connection pool configuration is highly tunable via environment variables, supporting all SQLAlchemy pool classes. The `fs2db` migration tool provides a path from legacy FileStore to SQL backends, covering experiments, runs, metrics, traces, and model registry data.
