## Purpose

**Short**: MLflow is the experiment tracking, model registry, and artifact management server for Red Hat OpenShift AI, providing workspace-isolated ML lifecycle management with Kubernetes-native authentication.

**Detailed**: MLflow serves as the central platform for the complete machine learning lifecycle within RHOAI. It provides experiment tracking (metrics, parameters, artifacts), a model registry (versioned model storage with aliases and stages), artifact management (proxied access to S3, GCS, Azure Blob, and other backends), and an AI Gateway for LLM endpoint routing. The server is a hybrid Flask/FastAPI Python application that runs behind gunicorn or uvicorn, with a React frontend served as static assets.

In the RHOAI context, MLflow is deployed with workspace isolation enabled, where each Kubernetes namespace maps to an MLflow workspace. Authentication uses Kubernetes service account tokens via the `kubernetes-namespaced` auth plugin, which injects both `Authorization: Bearer <token>` and `X-MLFLOW-WORKSPACE: <namespace>` headers. This provides multi-tenant data isolation across experiments, runs, models, and artifacts without requiring a separate MLflow instance per team. The frontend supports both standalone deployment and OpenShift module federation, exposing experiment tracking, prompts management, and run visualization as federated micro-frontends consumable by the RHOAI dashboard.

The RHOAI build produces a single container image (`odh-mlflow-rhel9`) via Konflux, built on UBI9 with Python 3.12. The image includes the MLflow server, the pre-built React UI, and PostgreSQL client libraries for production database backends. The build uses Cachi2/Hermeto for hermetic dependency prefetching across pip, yarn, and RPM layers, and ships on four architectures (amd64, arm64, ppc64le, s390x).

## Architectural Analysis

MLflow in the RHOAI context represents a carefully adapted upstream project. The upstream MLflow codebase (maintained by Databricks) is a large, feature-rich platform, and the RHOAI fork (`red-hat-data-services/mlflow`) carries a minimal set of downstream modifications focused on three areas: (1) Konflux build integration, (2) RHOAI-specific dependency pinning via the AIPCC package index, and (3) frontend module federation for dashboard integration. The version string `3.10.1+rhaiv.3` follows PEP 440 local version conventions, indicating this is MLflow 3.10.1 with the third RHOAI integration release.

The server architecture is a hybrid Flask/FastAPI design where the FastAPI application wraps the Flask WSGI app via `WSGIMiddleware`. FastAPI handles OpenTelemetry trace ingestion (`/v1/traces`), the AI Gateway router, and job execution API, while Flask serves the bulk of the tracking/registry API via protobuf-serialized RPC endpoints. Security middleware is layered in a strict order: host validation (DNS rebinding protection), CORS blocking (origin validation for state-changing requests), and security headers (X-Content-Type-Options, X-Frame-Options). Workspace context resolution happens after security middleware, extracting the `X-MLFLOW-WORKSPACE` header and setting a request-scoped context variable.

The workspace isolation model is architecturally significant for RHOAI. Rather than running separate MLflow instances per namespace (which would be resource-intensive), a single MLflow server handles multiple workspaces. The `WorkspaceAwareSqlAlchemyStore` applies workspace filtering at the SQL query level — every experiment, run, trace, model, and gateway resource query includes a workspace column predicate. The workspace store itself (`mlflow/store/workspace/sqlalchemy_store.py`) validates names against Kubernetes naming conventions (lowercase alphanumeric, 2-63 chars, optional internal hyphens), enforcing a clean mapping between K8s namespaces and MLflow workspaces. Workspace deletion supports three modes (RESTRICT, CASCADE, SET_DEFAULT) with preflight conflict detection.

The Kubernetes authentication plugin (`kubernetes_request_auth_provider.py`) implements a dual-cache strategy with 60-second TTL for both service account tokens and namespace resolution. Token and namespace are resolved independently — the token can come from a mounted service account while the namespace comes from kubeconfig, enabling hybrid scenarios. The `kubernetes-namespaced` variant is the RHOAI production auth mode, automatically propagating the pod's namespace as the workspace header.

The frontend module federation setup is noteworthy. The React UI supports two build modes: standalone (via CRACO/CRA) for development and testing, and federated (via Webpack 5 ModuleFederationPlugin) for RHOAI dashboard integration. The federated build exposes three wrapper components (`MlflowExperimentWrapper`, `MlflowPromptsWrapper`, `MlflowRunTabsWrapper`) as remote modules. Shared dependencies include React 18 and PatternFly 6 as singletons, but critically NOT react-router — the host (RHOAI dashboard) uses react-router v7 while MLflow uses v6, so each maintains its own router instance. The wrappers provide a BrowserRouter (v6) with a configurable basename, allowing MLflow routes to coexist within the dashboard shell.

The build pipeline targets four architectures (amd64, arm64, ppc64le, s390x) via Konflux multi-arch builds. The Dockerfile uses a three-stage build: Node.js for UI compilation, Python for wheel building, and UBI9-minimal for the runtime image. Dependency hermeticity is partially achieved — RPM lock files, Python hashed requirements, and Yarn are all prefetched via Cachi2, but the overall Tekton pipeline still runs with `hermetic: 'false'` pending full yarn/npm prefetching support.
