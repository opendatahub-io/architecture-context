## Purpose

**Short**: ai4rag is a Python optimization engine that automatically discovers optimal hyperparameters for Retrieval-Augmented Generation (RAG) pipelines.

**Detailed**: ai4rag (AI for RAG) is an LLM- and vector-database-agnostic optimization engine that accepts a variety of RAG Templates and a search space definition, then returns an initialized RAG Template with optimal parameter values (called a RAG Pattern). It operates as a pure Python library — there are no Dockerfiles, container images, Kubernetes manifests, CRDs, or services deployed by this component.

The library orchestrates a complete RAG optimization workflow: it defines a search space over chunking methods, chunk sizes, embedding models, foundation models, retrieval strategies, and search modes; then uses hyperparameter optimization (GAM-based or random search) to evaluate RAG pattern configurations against benchmark data using metrics such as answer correctness, faithfulness, and context correctness. It integrates out of the box with Llama Stack for embeddings, vector storage (Milvus, Qdrant), and text generation, with an alternative OpenAI-compatible client path.

ai4rag is designed to be consumed as a dependency by other RHOAI platform components (e.g., Kubeflow Pipelines components) rather than running as a standalone service. It provides a `KFPEventHandler` for streaming results into pipeline steps, and a `LocalEventHandler` for development use.

## Architectural Analysis

ai4rag follows a clean layered architecture with well-defined abstractions at each level. The library separates concerns into four main layers: the experiment orchestration layer (`core/experiment/`), the hyperparameter optimization layer (`core/hpo/`), the RAG pipeline layer (`rag/`), and the evaluation layer (`evaluator/`). Each layer uses abstract base classes to define interfaces, allowing users to swap implementations (e.g., different vector stores, foundation models, or evaluators) without modifying the orchestration logic.

The optimization strategy is noteworthy: the default `GAMOptimizer` uses Generalized Additive Models (via `pygam`) as a surrogate model to predict promising parameter combinations after an initial random exploration phase. This is a lightweight form of Bayesian optimization that avoids the overhead of Gaussian Processes while still providing model-guided search. The optimizer supports warm-starting via `known_observations`, enabling continuation of experiments or transfer learning across similar datasets.

The search space architecture is particularly well-designed. `AI4RAGSearchSpace` extends a generic `SearchSpace` with RAG-specific constraint rules that prune invalid combinations (e.g., chunk size must be greater than 2× chunk overlap, window retrieval requires non-zero window size, hybrid search requires ranker parameters). This constraint system uses composable rule functions, making it easy to add new constraints. The search space also supports hybrid search modes with configurable ranker strategies (RRF, weighted, normalized) for Llama Stack vector stores.

The models pre-selection (MPS) module is an efficiency optimization that addresses the combinatorial explosion when many foundation and embedding models are available. Before running full HPO, MPS evaluates all model pairs on a small data sample using an in-memory ChromaDB store, then selects the top-performing models for the expensive full optimization. This two-phase approach significantly reduces total computation time when the search space includes many models.

The event handler abstraction (`BaseEventHandler`) provides a clean extension point for integrating ai4rag into different execution environments. The `LocalEventHandler` writes results to the filesystem for development, while `KFPEventHandler` aggregates results in memory for Kubeflow Pipelines integration. This pattern allows ai4rag to be embedded into various orchestration systems without modifying core logic. The event handler receives structured payloads with full experiment provenance (settings, scores, confidence intervals, per-question breakdowns), making results traceable and reproducible.

A notable design decision is the dual-client support: ai4rag supports both Llama Stack (`LlamaStackClient`) and OpenAI (`OpenAI`) clients for foundation models and embeddings. The `get_vector_store` factory uses a string prefix convention (`ls_` prefix for Llama Stack providers, `chroma` for ChromaDB) to dispatch to the appropriate vector store implementation. This design makes ai4rag provider-agnostic while maintaining type safety through the generic base classes.
