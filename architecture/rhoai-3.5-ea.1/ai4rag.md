## Purpose

**Short**: ai4rag is a provider-agnostic optimization engine that finds optimal hyperparameters for Retrieval-Augmented Generation (RAG) pipelines.

**Detailed**: ai4rag accepts a set of RAG pipeline templates and a search space definition, then uses hyperparameter optimization (HPO) to find the best-performing RAG pattern for a given knowledge base and benchmark dataset. The optimization targets quality metrics such as answer correctness, faithfulness, and context correctness, evaluated via the `unitxt` evaluation framework.

The library is designed around a provider-agnostic architecture. Abstract base classes define interfaces for foundation models (`BaseFoundationModel`), embedding models (`BaseEmbeddingModel`), vector stores (`BaseVectorStore`), evaluators (`BaseEvaluator`), and RAG templates (`BaseRAGTemplate`). Out of the box, ai4rag integrates with OGX (previously named Llama Stack) for foundation models, embeddings, and vector storage, and with ChromaDB for local in-memory vector indexing.

The experiment workflow consists of: (1) optional models pre-selection (MPS) that evaluates model pairs on a sample to prune the search space, (2) hyperparameter optimization using a Generalized Additive Model (GAM)-based optimizer or random search, and (3) per-iteration RAG pattern evaluation that involves document chunking, embedding, vector indexing, retrieval, generation, and metric evaluation. Results are streamed through an event handler interface for integration with orchestration platforms like Kubeflow Pipelines. ai4rag is consumed by the RHOAI platform as a library dependency within RAG optimization pipeline components.

## Architectural Analysis

ai4rag follows a clean provider-agnostic design built around Python abstract base classes. The core abstractions (`BaseFoundationModel`, `BaseEmbeddingModel`, `BaseVectorStore`, `BaseEvaluator`, `BaseRAGTemplate`) define contracts that isolate the optimization engine from specific infrastructure. This design allows swapping OGX for alternative model serving backends without modifying the experiment orchestration code. The `OGXFoundationModel` and `OGXEmbeddingModel` implementations demonstrate the concrete provider pattern, while `ChromaVectorStore` provides a local alternative to the remote OGX vector store for development and testing.

The hyperparameter optimization layer is particularly well-designed. The `GAMOptimizer` uses Generalized Additive Models (via `pygam`) as a surrogate model to predict performance of untested hyperparameter combinations, directing the search toward promising regions of the parameter space. This is more sophisticated than pure grid or random search — the optimizer evaluates an initial random set of configurations, fits a GAM to predict scores for remaining combinations, and greedily selects the highest-predicted configurations for actual evaluation. The `SearchSpace` and `Parameter` classes implement a constraint-based pruning system with composable rule functions (e.g., chunk size must exceed 2x chunk overlap, window retrieval parameters are only valid with the window method, hybrid search parameters require consistent ranker configuration). This prevents evaluating invalid configurations that would waste compute.

The Models Pre-Selector (MPS) phase is an important optimization: when the search space contains many foundation or embedding models, MPS evaluates all model pairs on a small data sample using ChromaDB (in-memory, fast) before the main HPO loop. This prunes the model space early, avoiding expensive full-dataset evaluations of poorly-performing models. The pre-selection uses a simple mean-based scoring approach — rank all model pairs by their mean metric score on the sample, then keep the top N.

The event handler pattern (`BaseEventHandler` → `LocalEventHandler`, `KFPEventHandler`) is the integration seam for platform consumption. `LocalEventHandler` writes pattern results to local JSON files for development. `KFPEventHandler` aggregates results in memory for Kubeflow Pipeline component output. This pattern allows RHOAI to embed ai4rag within pipeline steps and stream progress/results to the platform UI without modifying the core library. The `PatternPayload` TypedDict provides a stable schema (version "1.0") for pattern results including scores, settings, and per-question evaluation breakdowns.

The library uses concurrent RAG execution via `ThreadPoolExecutor` (default 10 threads) in `query_rag()` for parallel question evaluation, which significantly speeds up pattern evaluation when the model serving backend supports concurrent requests. Collection reuse logic in `ExperimentResults` avoids re-indexing documents when only retrieval or generation parameters change between iterations, which is a critical performance optimization given that embedding + indexing is typically the most expensive step.
