## Purpose

**Short**: A batch-oriented execution environment for running standardized language model evaluations using the EleutherAI lm-evaluation-harness framework, integrated with Red Hat's EvalHub evaluation platform.

**Detailed**: The lm-evaluation-harness component provides a containerized runtime for evaluating language models against a comprehensive library of over 6,500 benchmark tasks. It is a downstream fork of the open-source EleutherAI lm-evaluation-harness (v0.4.8), extended with a custom adapter layer (`main.py`) that integrates with the EvalHub evaluation platform. The adapter reads job specifications from a mounted JSON file, configures the evaluation harness to connect to the model under test via OpenAI-compatible completions APIs, executes the benchmarks, and reports results back to EvalHub through callback URLs.

The component runs as a one-shot Kubernetes Job — it is not a long-running service. Each job evaluates a single model against a single benchmark, producing structured results that are persisted as OCI artifacts to a container registry (via `skopeo` and the `olot` library) and optionally logged to MLflow. The component supports air-gapped/disconnected cluster operation through automatic detection of locally mounted tokenizer and dataset files under `/test_data`, enabling offline Hugging Face Hub mode without explicit configuration.

The image is built for four architectures (x86_64, arm64, ppc64le, s390x) via Konflux multi-arch pipelines, with platform-specific build stages for compiling PyArrow, PyTorch, and OpenBLAS from source on non-x86 architectures. It integrates the Unitxt evaluation framework for additional task flexibility.

## Architectural Analysis

The lm-evaluation-harness is architecturally a **thin adapter pattern** layered on top of the upstream EleutherAI evaluation framework. The core `lm_eval` library is a well-structured Python package with clear separation between the model abstraction layer (`lm_eval/api/model.py`), the task registry system (`lm_eval/tasks/`), and the evaluation engine (`lm_eval/evaluator.py`). The Red Hat-specific adapter (`main.py`) adds integration with EvalHub without modifying the core library, making upstream rebasing tractable. The `LMEvalAdapter` class extends `FrameworkAdapter` from the `eval-hub-sdk` and primarily translates between EvalHub's `JobSpec` format and `lm_eval`'s `simple_evaluate()` parameters.

A notable design decision is the exclusive use of the `local-completions` model backend, which connects to OpenAI-compatible API endpoints rather than loading models locally. This is a deliberate architectural choice for the RHOAI platform: model serving is handled by separate infrastructure (vLLM, TGI, or other inference servers deployed via KServe/ServingRuntime), and the evaluation job only needs to send prompts and receive completions. This separation of concerns keeps the evaluation image lightweight (no GPU drivers or model weights) and allows it to evaluate any model accessible via an OpenAI-compatible API. The `num_concurrent` parameter (bounded to max 128) enables parallel inference requests, which is critical for evaluation throughput when the model server supports concurrent requests.

The offline/air-gapped support (`_infer_auto_offline_from_local_test_data` and `configure_hf_offline_environment`) demonstrates careful engineering for disconnected cluster environments. Rather than requiring explicit configuration, the adapter automatically detects when a tokenizer path under `/test_data` exists alongside dataset directories containing `dataset_dict.json` files and enables offline mode. This detection runs before any HuggingFace library imports to ensure the offline environment variables take effect before those libraries initialize their connection logic. The `metrics/` directory at the repository root provides 54 offline-capable metric implementations, and the `__main__.py` module patches `evaluate.load()` to redirect metric loading to this local path when offline.

Security-wise, the adapter implements credential sanitization (`_sanitize_error_message`) that redacts Bearer tokens, API keys, passwords, and URL credentials before any error message reaches EvalHub callbacks — preventing credential leakage in error reports. The `verify_certificate` support with explicit symlink rejection (`raw_path.is_symlink()` check) and path traversal prevention via `_resolve_job_spec_path_for_read` (confining job spec reads to `/meta/`) shows defense-in-depth against CWE-22 (path traversal) and CWE-61 (symlink following). The container runs as non-root user `65532:65532` with no privilege escalation.

The multi-architecture build pipeline is substantial: the Konflux Dockerfile has 5 build stages, with separate stages for building PyArrow from source (ppc64le/s390x), PyTorch from source (ppc64le/s390x), and OpenBLAS from source (ppc64le). This complexity exists because upstream wheels for these packages are not available for non-x86 architectures, requiring full source compilation in the container build. The x86_64 and arm64 builds use pre-built wheels from PyPI, making them significantly faster.
