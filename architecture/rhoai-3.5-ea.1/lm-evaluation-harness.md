## Purpose

**Short**: A batch-oriented evaluation framework for benchmarking language models, packaged as a job container for RHOAI's TrustyAI evaluation pipeline.

**Detailed**: LM Evaluation Harness (lm-eval) is a standardized framework originally developed by EleutherAI for evaluating language model performance across a wide variety of benchmarks and tasks. The RHOAI distribution extends the upstream framework with an EvalHub adapter layer (`main.py`) that integrates it into Red Hat's model evaluation infrastructure.

The adapter receives evaluation job specifications as JSON (mounted via ConfigMap at `/meta/job.json`), configures the lm-eval framework to send inference requests to an OpenAI-compatible model endpoint (e.g., vLLM, TGI), executes the benchmark, and reports results back to the EvalHub service via callbacks. Results can optionally be persisted as OCI artifacts to a container registry and logged to MLflow. The component supports offline/disconnected evaluations using pre-cached datasets from S3, concurrent inference requests, and custom CA certificates for TLS verification.

The container image (`odh-ta-lmes-job-rhel9`) is built for four architectures (x86_64, arm64, ppc64le, s390x) using Konflux pipelines, with special multi-stage builds for PyTorch and Apache Arrow on non-x86 platforms. It includes 150+ bundled evaluation task definitions covering common NLP benchmarks (ARC, HellaSwag, MMLU, GSM8K, etc.) plus the Unitxt evaluation framework.

## Architectural Analysis

The LM Evaluation Harness in RHOAI represents a carefully layered architecture where the upstream EleutherAI framework serves as the core evaluation engine, wrapped by a thin but critical EvalHub adapter layer. The adapter (`main.py`) implements the `FrameworkAdapter` interface from `eval-hub-sdk`, which provides a standardized contract for evaluation frameworks in RHOAI's TrustyAI ecosystem. This separation allows the upstream framework to evolve independently while the adapter handles RHOAI-specific concerns: job spec parsing, status callbacks, OCI artifact persistence, MLflow integration, and credential management.

The model backend architecture is notably flexible. While the adapter always uses the `local-completions` backend (which sends text prompts to OpenAI-compatible HTTP endpoints), the upstream framework supports 20+ model backends including HuggingFace local inference, vLLM, SGLang, Anthropic, WatsonX, and more. The RHOAI adapter deliberately constrains this to HTTP-based remote inference, which aligns with the platform's architecture of separating model serving (KServe/vLLM) from evaluation workloads. The `TemplateAPI` base class in `api_models.py` provides sophisticated async HTTP request handling with concurrent requests (`num_concurrent`), retry logic with exponential backoff, custom SSL context management, and tokenized vs. string prompt modes.

The offline mode implementation (added in commits c714067c, b87d4f1e, cfafeaca) is architecturally significant for disconnected/air-gapped deployments. When `parameters.offline` is set in the job spec, the adapter sets `HF_HOME=/test_data`, `HF_HUB_OFFLINE=1`, `HF_DATASETS_OFFLINE=1`, `HF_EVALUATE_OFFLINE=1`, and `TRANSFORMERS_OFFLINE=1`, forcing all HuggingFace libraries to use only locally cached data. The `__main__.py` module also patches `evaluate.load()` to resolve metrics from a local `metrics/` directory instead of downloading them. The `/test_data` directory is populated by an init container (managed by the TrustyAI operator) that downloads data from S3 via `scripts/s3_downloader.py`.

The multi-architecture build strategy in `Dockerfile.konflux.lmes-job` is complex and notable. The image uses a 5-stage build: base builder, Arrow builder (for ppc64le/s390x where pre-built PyArrow wheels aren't available), PyTorch builder (for ppc64le/s390x), OpenBLAS builder (for ppc64le), and final image assembly. The x86_64 and arm64 builds simply install pre-built wheels from PyPI, while ppc64le and s390x compile PyTorch and Arrow from source with architecture-specific optimizations. This adds significant build time but ensures consistent behavior across all supported architectures. The container runs as non-root user 65532:65532 for security.

Security-wise, the adapter includes careful credential handling: model auth secrets can provide API keys, HuggingFace tokens (for gated datasets like GPQA), and custom CA certificates. The `verify_certificate` parameter in `api_models.py` explicitly rejects symlinks for CA cert paths and uses `resolve(strict=True)` to prevent path traversal attacks. Error messages are sanitized to avoid leaking raw exception details (commit c255efce), and gated dataset errors are detected and surfaced with specific guidance. The S3 downloader supports custom CA bundles via `AWS_CA_BUNDLE` but also allows disabling SSL verification via `S3_VERIFY_SSL`, which is an operator-level configuration concern.
