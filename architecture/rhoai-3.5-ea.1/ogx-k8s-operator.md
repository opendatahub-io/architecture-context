## Purpose

**Short**: Kubernetes operator that creates and manages Llama Stack server deployments for running AI agent distributions on OpenShift.

**Detailed**: The Llama Stack K8s Operator manages the lifecycle of Llama Stack server instances on Kubernetes and OpenShift clusters. It watches `LlamaStackDistribution` custom resources and reconciles a complete deployment stack for each instance, including Deployments, Services, NetworkPolicies, Ingress resources, PersistentVolumeClaims, PodDisruptionBudgets, HorizontalPodAutoscalers, ServiceAccounts, and RoleBindings.

The operator supports multiple distribution types (starter, remote-vllm, meta-reference-gpu, postgres-demo) through an embedded distribution image registry. It also supports image override via a ConfigMap-based configuration system, enabling RHOAI to inject its own validated distribution images. The operator provides comprehensive CA bundle management, including auto-detection of ODH trusted CA bundle ConfigMaps and explicit user-provided CA bundles. User configuration is injected via ConfigMaps mounted into the Llama Stack containers.

The controller uses kustomize internally to render base manifests and applies Go-based transformer plugins for namespace injection, name prefixing, field mutation, and NetworkPolicy rule generation. Feature flags (e.g., network policy enablement) are managed through an operator-scoped ConfigMap with automatic upgrade logic. The operator also performs legacy cleanup tasks (e.g., removing cluster-scoped RoleBindings from earlier versions) during startup.

## Architectural Analysis

The Llama Stack K8s Operator follows a standard kubebuilder/controller-runtime pattern with a single controller managing a single CRD (`LlamaStackDistribution`). The architecture is notable for its use of embedded kustomize rendering — rather than static Go struct construction, the controller renders manifests from embedded YAML files (`controllers/manifests/base/`) using the kustomize Go library, then applies a chain of Go-based transformer plugins for name prefixing, namespace injection, field mutation, and NetworkPolicy rule generation. This approach provides a clean separation between manifest templates and runtime transformation logic, making the manifests human-readable while allowing dynamic customization.

The operator's ConfigMap management is particularly well-designed. It uses two separate API clients: a cached client (controller-runtime's default) for operator-managed resources with the `app.kubernetes.io/managed-by: llama-stack-operator` label, and a direct (non-cached) API client for reading user-referenced ConfigMaps that lack operator labels. The cache is further optimized with label selectors (`llamastack.io/watch: "true"`) and stripped managed fields to reduce memory usage. User ConfigMaps that carry the watch label trigger immediate reconciliation via a Watches handler, while unlabeled ConfigMaps are picked up on the 5-minute periodic requeue cycle.

The CA bundle handling demonstrates defense-in-depth: the operator validates PEM structure and X.509 certificate format for all CA bundle entries, enforces size limits (10MB total, 1000 certificates max), filters out non-CERTIFICATE PEM blocks, and re-encodes certificates to normalize formatting. Multiple CA sources (explicit user-provided, auto-detected ODH platform bundle) are merged into a single operator-managed ConfigMap that is mounted into pods. This managed ConfigMap pattern avoids cross-namespace volume mount issues while maintaining proper owner reference chains for garbage collection.

The network security model is configurable but defaults to a secure posture: when NetworkPolicy is enabled (default), the transformer generates ingress rules that allow traffic only from the same namespace, the operator namespace, and any explicitly specified namespaces or label-matching namespaces. When `spec.network.exposeRoute` is true, an additional peer for OpenShift router namespaces is added, and a Kubernetes Ingress resource is created. The use of standard Kubernetes Ingress (rather than OpenShift Routes or Gateway API HTTPRoutes) is notable — it provides portability across Kubernetes distributions, though it means the operator does not leverage RHOAI 3.x Gateway API patterns for its managed resources.

The operator includes thoughtful upgrade path handling: startup cleanup removes legacy cluster-scoped RoleBindings from earlier versions, the operator config ConfigMap is automatically patched with watch labels and updated feature flag defaults on version changes, and the deployment strategy is automatically set to Recreate when persistent storage is configured (to avoid RWO PVC multi-attach deadlocks during rolling updates). Server-side apply is used for resource patching with ownership tracking, with a full replacement fallback for cases where legacy volumes cannot be removed via SSA.
