## Purpose

**Short**: Kubernetes operator that creates and manages OGX (formerly Llama Stack) AI inference servers as declarative custom resources.

**Detailed**: The OGX K8s Operator is a controller-runtime-based Kubernetes operator that reconciles `OGXServer` custom resources into fully managed inference server deployments. It automates the lifecycle of AI model serving infrastructure — provisioning Deployments, Services, PersistentVolumeClaims, NetworkPolicies, Ingresses, HorizontalPodAutoscalers, and PodDisruptionBudgets based on a single declarative CR.

The operator supports multiple AI provider backends (vLLM, OpenAI, Azure, AWS Bedrock, Google Vertex AI, IBM watsonx, Milvus, PGVector, Qdrant, and more) through a type-safe provider specification system. It generates OGX server configuration from the CR spec, handles distribution image resolution (with support for both embedded and Red Hat-shipped distributions), and provides legacy migration from the older `LlamaStackDistribution` v1alpha1 API to the current `OGXServer` v1beta1 API through an adoption mechanism for PVCs, Services, and Ingresses.

The operator is deployed as a component of both Open Data Hub (ODH) and Red Hat OpenShift AI (RHOAI), with kustomize overlays providing distribution-specific configuration including namespace, image references, and TLS certificate management via OpenShift's service-serving-cert-signer.

## Architectural Analysis

The OGX K8s Operator demonstrates a well-structured kustomize-based resource rendering pipeline that separates template definition from runtime parameterization. Rather than constructing Kubernetes objects in Go code, the operator embeds base kustomize manifests (`controllers/manifests/base/`) and applies a chain of transformer plugins at runtime — name prefix, namespace, JSON Pointer field mutations, and a NetworkPolicy transformer — to produce the final resource set. This approach keeps the manifest templates readable and auditable while still supporting per-instance customization. The Deployment name is notably excluded from the name prefix plugin to maintain backward compatibility with existing selectors.

The provider type system is one of the most sophisticated aspects of the API design. The `ProvidersSpec` hierarchy uses a consistent Remote/Inline split pattern across six API domains (inference, vector_io, tool_runtime, files, batches, responses), where each domain defines typed providers with derived IDs (e.g., `remote-vllm`, `remote-pgvector`, `inline-localfs`) alongside a `CustomProvider` escape hatch that requires `remote::` or `inline::` prefixed type strings. This allows users to configure well-known AI backends with strong type safety while preserving extensibility. The webhook validates provider ID uniqueness across all domains and ensures model resource references point to configured providers.

The legacy adoption system (`controllers/legacy_adoption.go`) manages a careful migration from `LlamaStackDistribution` (v1alpha1) to `OGXServer` (v1beta1). The adoption is annotation-driven (`ogx.io/adopt-storage`, `ogx.io/adopt-networking`), scales the legacy Deployment to zero before PVC transfer, validates owner references to prevent accidental takeover of unrelated resources, and intentionally avoids setting owner references on adopted PVCs so they outlive the new OGXServer CR. This is a safety-critical design decision — PVC deletion would cause data loss. The adoption cleanup path (removing adopted Services/Ingresses when the annotation is removed) demonstrates attention to the full lifecycle.

The operator's resource comparison logic (`pkg/compare/comparison.go`) prevents reconciliation thrashing by ignoring cluster-managed fields (ResourceVersion, UID, ManagedFields) and server-set Service fields (ClusterIP). The Deployment update path includes detection of legacy volume configurations (old CA bundle emptyDir pattern, stale user-config volumes) that require a full replace instead of server-side apply — a pragmatic solution to the SSA limitation of not being able to remove previously-applied fields.

The NetworkPolicy transformer plugin deserves attention: it constructs default ingress rules allowing traffic from all pods in the same namespace and from the operator namespace, with optional OpenShift router namespace peers for external access. Users can override with custom ingress/egress rules via `spec.network.policy`. The feature is enabled by default but can be disabled via a ConfigMap-based feature flag (`enableNetworkPolicy`), with proper cleanup of existing NetworkPolicy resources when disabled.
