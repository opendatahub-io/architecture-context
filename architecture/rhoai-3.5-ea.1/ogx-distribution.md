## Purpose

**Short**: Builds and packages the Red Hat OpenShift AI distribution of Meta's Llama Stack server as a production-ready container image with multi-provider inference, vector I/O, tool runtime, and observability capabilities.

**Detailed**: The ogx-distribution repository produces a container image that bundles the [Llama Stack](https://github.com/llamastack/llama-stack) framework — specifically the OpenDataHub fork at version `v0.7.1+rhaiv.1` — with all necessary Python dependencies, provider plugins, and runtime configuration for deployment on Red Hat OpenShift AI. The image exposes a unified REST API on port 8321 that speaks the OpenAI-compatible chat completions protocol, the Llama Stack native API for responses/batches/files/vector_io/tool_runtime, and a health endpoint. It acts as an inference gateway that can route requests to multiple backend providers including vLLM, AWS Bedrock, IBM WatsonX, Azure OpenAI, Google Vertex AI, and OpenAI.

The repository does not contain the Llama Stack source code itself — it is a *distribution packaging* repo. The `distribution/build.py` script generates a `Containerfile` from a template (`Containerfile.in`) by resolving Python dependencies via the `llama stack list-deps` CLI. The `build.yaml` file defines the full provider manifest (build-time), while `config.yaml` is a stripped version (runtime default) that excludes dependency-only providers (inline::sentence-transformers, inline::milvus, inline::faiss). Users can supply a custom `config.yaml` at runtime to activate any installed provider.

For RHOAI downstream builds, Konflux uses `Dockerfile.konflux` which builds on a different base image (`quay.io/aipcc/base-images/cpu`) and installs dependencies via pre-built wheel releases rather than pip from PyPI, providing a hardened supply chain. The repo also includes a secondary `vllm/Containerfile` that produces a vLLM CPU serving image with pre-baked HuggingFace models.

## Architectural Analysis

The ogx-distribution repository follows a **distribution packaging pattern** rather than containing the Llama Stack application source code itself. The actual Llama Stack server code lives in the `opendatahub-io/llama-stack` fork (at tag `v0.7.1+rhaiv.1`), while this repo manages: (1) the build-time dependency resolution and Containerfile generation, (2) the runtime configuration defining which providers are available, and (3) the Konflux pipeline definitions for producing hardened RHOAI images. This separation of concerns means the architecture of the *deployed* service is largely defined by the upstream Llama Stack framework, not by code in this repository.

The provider activation system uses a clever environment-variable-based conditional pattern in `config.yaml`. Provider entries use the syntax `${env.VAR:+provider-id}` — when the environment variable is set, the provider ID is expanded (activating the provider); when unset, the provider ID is empty and the provider is skipped. This allows a single container image to support many inference backends without requiring configuration changes beyond setting environment variables. The `build.yaml` vs `config.yaml` distinction is also notable: `build.yaml` is the full manifest (used for dependency resolution), while `config.yaml` is a stripped version that excludes dependency-only providers like `inline::milvus` and `inline::faiss` — providers whose Python packages are installed but which aren't activated by default because they have limitations (e.g., `inline::milvus` is incompatible with multi-worker deployments).

The dual-Dockerfile strategy reflects the split between upstream (community) and downstream (RHOAI) builds. The `distribution/Containerfile` is auto-generated from `Containerfile.in` by `build.py` and installs dependencies directly from PyPI — suitable for upstream CI and community builds. The `Dockerfile.konflux` uses a completely different base image (`quay.io/aipcc/base-images/cpu`) and installs from pre-built wheel releases via the AIPCC toolchain (`install-wheel-release.sh`), providing a hardened supply chain for RHOAI. Build args like `WHEEL_RELEASE_AARCH64` and `WHEEL_RELEASE_X86_64` (defined in `konflux/cpu-ubi9.conf`) pin exact wheel versions per architecture. The Konflux build also includes the `ibm-granite/granite-embedding-125m-english` embedding model and tiktoken cache baked into the image, avoiding runtime downloads.

The authentication architecture supports optional OAuth2/OIDC JWT validation configured via `AUTH_ISSUER`, `AUTH_JWKS_URI`, and `AUTH_AUDIENCE` environment variables. When enabled, the Llama Stack server validates incoming JWTs against the configured JWKS endpoint and enforces a resource-ownership access policy: all users can read system resources, authenticated users can create resources, and only owners can update/delete their own resources. TLS verification for the auth endpoint is configurable via `AUTH_VERIFY_TLS` (defaulting to `true`). When `AUTH_ISSUER` is not set, authentication is completely disabled.

The storage architecture uses PostgreSQL as the backing store for all persistent data, with two abstraction layers: a KV store (`kv_postgres`) for registry metadata, vector_io persistence, agent state, and prompt storage; and a SQL store (`sql_postgres`) for inference logs, conversations, response histories, file metadata, and batch operations. Both backends connect to the same PostgreSQL instance (configurable via `POSTGRES_*` environment variables). The SQL store is configured with write queuing (queue size 10000, 4 writers) for high-throughput inference logging.
