## Purpose

**Short**: Builds and packages the OGX AI inference gateway into container images for Red Hat OpenShift AI deployment.

**Detailed**: ogx-distribution is a container image packaging repository that produces the `odh-ogx-core` container image for RHOAI. OGX (formerly LLaMA Stack) is a multi-provider AI inference gateway that exposes a unified OpenAI-compatible API while routing requests to configurable inference backends including vLLM, AWS Bedrock, Google Vertex AI, Azure OpenAI, IBM watsonx, and OpenAI. The repository does not contain the OGX application code itself; instead, it defines the build process (`distribution/build.py`) that auto-generates Dockerfiles, dependency installation scripts, and runtime configuration from a declarative `build.yaml` manifest.

The distribution image ships with a comprehensive provider ecosystem covering inference, vector I/O (Milvus, pgvector, Qdrant, FAISS), file storage (local filesystem, S3), tool runtimes (Brave Search, Tavily Search, MCP, file search), batch processing, and agentic responses. Providers are conditionally enabled via environment variables at runtime, allowing a single image to support diverse deployment topologies. The image uses PostgreSQL as its persistent storage backend for key-value storage, SQL storage, inference logging, conversations, and metadata. Authentication is configurable via OAuth2/OIDC token validation with JWKS key rotation support.

A secondary `vllm/Containerfile` produces a vLLM CPU container image with pre-downloaded HuggingFace models baked in at build time, used exclusively for CI testing to avoid model downloads during pipeline runs. This image is not part of the RHOAI product.

## Architectural Analysis

The ogx-distribution repository follows a **build-time code generation** pattern: the source of truth is `distribution/build.yaml` (a declarative manifest of all providers, APIs, and configuration), and `distribution/build.py` generates three derivative artifacts: `distribution/Containerfile` (the upstream Dockerfile), `distribution/install-deps.sh` (the dependency installation script), and `distribution/config.yaml` (the runtime configuration with dependency-only providers stripped). The `Dockerfile.konflux` is a separate, manually maintained file that uses the RHOAI base image and PyPI index instead of building from source. Pre-commit hooks enforce that generated files stay in sync with `build.yaml`.

The **provider activation pattern** is noteworthy: providers are conditionally enabled at runtime via environment variables using OGX's `${env.VAR:+provider_id}` syntax. When the env var is unset or empty, the provider ID resolves to empty and OGX skips it. This allows a single container image to support many deployment topologies without rebuild. Three providers (`inline::sentence-transformers`, `inline::milvus`, `inline::faiss`) have their Python dependencies pre-installed but are stripped from the default `config.yaml` to avoid loading unnecessary modules at startup. Users can re-enable them by mounting a custom `config.yaml`.

The **dual-build strategy** (upstream Containerfile vs. Konflux Dockerfile) reflects the downstream/upstream split common in RHOAI components. The upstream Containerfile installs OGX directly from GitHub source (`git+https://github.com/opendatahub-io/ogx.git@v1.0.2+rhaiv.0`), while the Konflux Dockerfile installs pre-built packages from the RHOAI PyPI index. The Konflux build uses a base image (`quay.io/aipcc/base-images/cpu:3.5.0-ea.1`) that likely includes additional RHOAI-specific tooling (fromager RPM checks, scanlibs validation). The `versions.env` and `konflux/cpu-ubi9.conf` files parameterize the OGX version and base image for the Konflux build path.

The **security model** relies on optional OAuth2/OIDC token validation configured at runtime. When `AUTH_ISSUER` is not set, authentication is completely disabled. Access policies are simple: unowned resources are publicly readable, authenticated users can create resources, and owners can manage their own resources. The server itself listens on plaintext HTTP (port 8321); TLS termination is expected to be handled by the platform ingress layer (Gateway API / kube-rbac-proxy in RHOAI 3.x). This means sensitive credentials (API keys, database passwords) in HTTP headers and request bodies traverse the internal network in plaintext between the ingress proxy and the OGX pod.

The **build hermeticity is weak** on this branch: no lock files exist at any layer (RPM, Python, artifacts), and `install-deps.sh` installs approximately 50 Python packages with a mix of pinned and unpinned versions. The Konflux build mitigates this somewhat by using the RHOAI PyPI index (which likely has curated package versions), but without `rpms.lock.yaml` or `uv.lock`, reproducible builds are not guaranteed. The `renovate.json` file delegates to `konflux-central` for automated dependency updates.
