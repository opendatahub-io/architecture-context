## Purpose

**Short**: High-performance system for processing large-scale batch inference jobs in Kubernetes environments, providing an OpenAI-compatible API for submitting, tracking, and managing batch inference workloads.

**Detailed**: Batch Gateway is a distributed batch inference processing system designed to facilitate efficient processing of batch workloads alongside interactive inference workloads. It provides OpenAI-compatible `/v1/batches` and `/v1/files` REST APIs for submitting up to 50,000 requests per job, tracking progress in real time, and managing batch inference lifecycle. The system is composed of three independently deployable components: an API Server that handles REST API requests, a Batch Processor that pulls jobs from a priority queue and dispatches individual inference requests to downstream inference gateways, and a Garbage Collector that cleans up expired data.

The system uses PostgreSQL for persistent metadata storage, Redis or Valkey for priority queuing, event channels, and volatile status, and S3-compatible object storage or local filesystem for input/output file storage. The Processor implements model-aware scheduling that groups and orders requests by model and system prompt for optimal downstream utilization, with intelligent flow control via semaphore-based concurrency limiting at both global and per-model levels. Authentication and authorization are delegated to an external gateway layer (e.g., Kuadrant/Authorino), while multi-tenancy isolation is enforced at the data access layer by scoping all operations to the authenticated tenant. The system is deployed via Helm charts with Kubernetes Gateway API (HTTPRoute) integration for ingress routing.

## Architectural Analysis

The Batch Gateway follows a clean three-tier architecture separating API handling, asynchronous processing, and maintenance. The API Server is purely synchronous and stateless, receiving batch submissions, storing metadata in PostgreSQL, enqueueing jobs into a Redis/Valkey priority queue (scored by SLO deadline in microseconds), and storing input files in S3. The Processor runs a polling loop that dequeues jobs, downloads input files, builds binary execution plans sorted by system prompt hash (for downstream KV cache optimization), and dispatches individual requests to inference gateways. The GC runs on a configurable interval (default 1 hour) to clean up expired items. This separation enables independent scaling: API servers scale horizontally for throughput, processors scale for inference parallelism, and exactly one GC instance prevents concurrent delete races.

The concurrency model in the Processor is carefully layered. A worker-level semaphore limits how many jobs process simultaneously (default 1). A global semaphore (default 100 tokens) caps total in-flight inference requests across all workers. Per-model semaphores (default 10 tokens) prevent any single model from monopolizing the global capacity. Semaphore acquisition order is per-model first, then global, avoiding starvation. Each semaphore includes a guard callback that triggers graceful shutdown on double-release detection, a defensive pattern against concurrency bugs. The Processor also implements comprehensive crash recovery: on startup, it scans its working directory for leftover job artifacts and recovers them based on the job's last known phase (re-enqueue for early phases, upload partial results for late phases).

Multi-tenancy is enforced at the data access layer rather than at the API level. The API server extracts the tenant ID from a configurable header (default `X-MaaS-Username`, using the last header value for Envoy ext_authz compatibility) and stores it in the request context. All database queries include the tenant ID, and resource lookups that don't match the authenticated tenant return 404 (not 403) to prevent information leakage. File storage uses SHA256-hashed tenant IDs as folder names, providing deterministic and collision-resistant isolation within S3 buckets. Authentication itself is delegated to an external gateway layer (Kuadrant/Authorino), supporting API key, ServiceAccount token, and OpenShift user token modes.

The system's observability stack is comprehensive. The API server exposes three Prometheus metrics (request count, duration histogram, in-flight gauge) using route patterns rather than raw paths to avoid cardinality explosion. The Processor exposes 15+ metrics covering job processing, per-model execution, token usage, cancellation, and recovery. PrometheusRule templates define four alerts: high queue wait time (p95 > 5 min), high job failure rate (>10%), expired jobs detected (>3%), and worker saturation (>90%). OpenTelemetry tracing propagates trace context from API server to processor via batch job tags with an `otel:` prefix, enabling end-to-end distributed traces. The API server injects `x-request-id` headers and the processor adds `x-slo-ttft-ms` and `x-gateway-inference-objective` headers to inference requests for downstream scheduling integration.

Security follows defense-in-depth principles. Pod security contexts enforce non-root execution (UID 1001/65532), read-only root filesystem, dropped capabilities, and RuntimeDefault seccomp profile. Secrets are mounted read-only at `/etc/.secrets/` and read via `os.OpenInRoot()` to prevent path traversal. The API server sets security headers (X-Content-Type-Options, X-Frame-Options, X-XSS-Protection) on every response. Input validation includes strict JSON decoding, file size limits (200 MB default), and line count limits (50,000 default). TLS support includes optional server-side TLS 1.2+ on the API server, optional mTLS for processor-to-inference-gateway connections with custom CA and client certificate support, and optional TLS for Redis/PostgreSQL connections. The HTTP client implements status-code-aware retry logic with exponential backoff, honoring Retry-After headers for 429 responses and using 3x backoff multiplier for rate limiting.
