## Purpose

**Short**: OpenAI-compatible batch inference API gateway for llm-d that accepts, processes, and garbage-collects batch inference jobs across multiple LLM inference endpoints.

**Detailed**: The batch-gateway provides an OpenAI Batch API-compatible REST interface that enables users to submit large batches of inference requests (in JSONL format), which are then processed asynchronously against configurable LLM inference gateways. The system is composed of three independently deployable components: an API server that handles file uploads and batch job submission, a processor that dequeues jobs and dispatches individual inference requests with concurrency control, and a garbage collector that removes expired jobs and files.

The gateway supports multi-tenancy via header-based tenant isolation, pluggable storage backends (PostgreSQL or Redis/Valkey for metadata, S3 or local filesystem for files), and per-model or global inference gateway routing. It integrates with Kubernetes-native observability (Prometheus metrics, OpenTelemetry tracing, Grafana dashboards) and supports Gateway API HTTPRoutes for ingress. The batch processor implements sophisticated concurrency control with global and per-model semaphores, SLO-aware deadline management, and crash recovery with partial result preservation.

## Architectural Analysis

The batch-gateway implements a classic producer-consumer architecture with three independently scalable components communicating through shared state (Redis or PostgreSQL). The API server acts as the producer, accepting batch job requests and enqueueing them into a Redis sorted set (priority queue) with SLO-derived scores. The processor acts as the consumer, polling the priority queue and dispatching individual inference requests against configurable LLM inference gateways. The garbage collector operates independently, performing periodic sweeps of expired data.

The concurrency model is notably sophisticated for a batch processing system. The processor uses a two-level semaphore hierarchy — a global semaphore limiting total in-flight inference requests across all workers, and per-model semaphores preventing any single model from monopolizing resources. This design prevents request starvation while respecting upstream gateway capacity limits. The semaphore implementation includes a double-release guard that triggers graceful shutdown on semaphore over-release, indicating a defensive approach to concurrency bugs.

Multi-tenancy is enforced at multiple layers through a header-based tenant isolation pattern designed for Envoy ext_authz integration. The middleware extracts the tenant ID from the last value of a configurable header (defaulting to `X-MaaS-Username`), using "last value" semantics specifically to handle auth proxies that append validated identity headers. Every database query includes the tenant ID in its filter, and post-retrieval validation returns 404 (not 403) on tenant mismatch — an information-hiding pattern that prevents tenant enumeration attacks. File storage isolation uses SHA256-hashed tenant IDs as folder names, ensuring filesystem-safe and S3-compatible path segments.

The crash recovery system demonstrates production maturity. On startup, the processor discovers stale job directories from previous container crashes and performs phase-aware recovery: jobs that were finalizing get their partial results uploaded, jobs that were executing are re-enqueued, and jobs that were validating are transitioned to failed. Partial results are preserved even for cancelled, expired, or failed jobs — users can retrieve whatever completed before the failure. The recovery flow uses detached contexts that survive SIGTERM, ensuring terminal operations (file uploads, database writes) complete even during pod termination.

The system's storage abstraction layer is well-designed, with pluggable backends for both metadata (Redis/Valkey/PostgreSQL) and file storage (local filesystem/S3). Redis usage is particularly interesting — it serves four distinct roles through different data structures: hash sets for metadata, sorted sets for the priority queue, blocking lists for the event channel (cancellation signaling), and simple key-value pairs with TTL for progress status. This Redis-centric design enables low-latency operations but creates a single point of failure unless Redis is deployed with replication. The PostgreSQL alternative provides stronger durability guarantees but lacks the priority queue and event channel features, which remain Redis-backed.
