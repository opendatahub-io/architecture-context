## Purpose

**Short**: HPU (Intel Gaudi) hardware plugin for vLLM that enables high-performance large language model inference on Habana Gaudi accelerators.

**Detailed**: vllm-gaudi is a hardware plugin package for the vLLM inference engine that provides Intel Gaudi (Habana) accelerator support. It follows vLLM's hardware-pluggable architecture (RFC #11162 and #19161), registering custom platform, attention, model, and operation implementations via Python entry points. The plugin replaces GPU-specific code paths with HPU-optimized equivalents, including custom attention backends (paged attention, unified attention, MLA), FP8/GPTQ/AWQ quantization for Gaudi hardware, HPU graph compilation for reduced overhead, and distributed inference via HCCL (Habana Collective Communications Library).

The component ships as a container image (`odh-vllm-gaudi-rhel9`) built on UBI 9 with Habana Synapse AI 1.23.0 and PyTorch 2.9.0. It is consumed by RHOAI's model serving stack — when users deploy a ServingRuntime that targets Intel Gaudi hardware, this image provides the vLLM OpenAI-compatible API server with Gaudi acceleration. The image includes both the upstream vLLM core (pinned to v0.16.0) and the vllm-gaudi plugin, packaged together. It runs as a non-root user (UID 2000) with the entrypoint `python3 -m vllm.entrypoints.openai.api_server`, exposing an OpenAI-compatible REST API for chat completions, completions, and embeddings.

## Architectural Analysis

### Plugin Architecture and Registration System

vllm-gaudi follows vLLM's hardware-pluggable architecture with four distinct registration entry points defined in `setup.py` (lines 66-72). The `vllm.platform_plugins` entry point registers `HpuPlatform` as the device platform. Three `vllm.general_plugins` entries register utilities, operations, and models in a specific numbered order (01, 02, 03) to ensure proper initialization sequence. This design allows the plugin to override vLLM's internal behavior without modifying upstream code — each registration hooks into vLLM's `register_oot` (out-of-tree) pattern, replacing GPU-specific implementations with HPU equivalents. The plugin overrides 11 model architectures, 15+ operation layers (attention, MoE, LayerNorm, quantization), and the entire worker/scheduler pipeline.

### Attention Backend Strategy

The plugin provides four attention backends selected dynamically based on model architecture and configuration: HPUAttentionBackendV1 (standard paged attention), HPUUnifiedAttentionBackend (flash-like unified attention with better memory efficiency), HPUMLAAttentionBackend (Multi-Head Latent Attention for DeepseekV2-style models), and HPUUnifiedMLABackend. The unified attention path (`unified_attn` config flag) enables a significantly different execution model — it uses continuous batching with block-diagonal attention masks and supports defragmentation of KV cache blocks. The choice between backends is made at model initialization time via `HpuPlatform.get_attn_backend_cls()` (platform.py:44-67) and cannot be changed at runtime. The `extension/ops.py` file alone is ~1500 lines, containing the core attention kernel dispatch logic with multiple implementations (naive, FusedSDPA, flex attention, pipelined paged attention).

### Quantization Support

The plugin supports six quantization methods: FP8 (per-tensor and per-channel), INC (Intel Neural Compressor), GPTQ, AWQ, ModelOpt, and compressed-tensors. Each is registered via vLLM's quantization config system. The FP8 path is the most sophisticated, supporting both block-wise and channel-wise strategies with dynamic scale computation (`extension/scales.py`). GPTQ and AWQ use HPU-native `torch.ops.hpu.convert_from_uint4()` for efficient 4-bit dequantization. A notable design choice: when INC quantization is detected, the platform forces CPU-first weight loading (`platform.py:170-174`) to prevent OOM during the quantize-then-transfer pipeline.

### Build Architecture and FIPS Implications

The Dockerfile.konflux.gaudi uses a 3-stage multi-stage build: `gaudi-base` (UBI 9 + Habana drivers), `gaudi-pytorch` (PyTorch installation from Habana artifacts), and `vllm-openai` (vLLM + plugin installation). The build explicitly removes the OpenSSL FIPS provider (`rpm -e --nodeps openssl-fips-provider-so`) in the base stage, making this image incompatible with FIPS mode. The build is non-hermetic (`hermetic: false` in the Tekton pipeline) — it clones from GitHub and installs from PyPI at build time. The `.konflux/build-args.conf` provides version pinning for the main components (`VLLM_GAUDI_COMMIT=v0.16.0`, `VLLM_PROJECT_COMMIT=v0.16.0`), but transitive Python dependencies are not locked. The image runs as UID 2000 (non-root) with group 0 permissions for OpenShift compatibility.

### Distributed Inference and Disaggregated Serving

Multi-card inference uses HCCL (Habana's collective library) as the distributed backend, initialized in `hpu_worker.py`. The `HpuCommunicator` class wraps standard PyTorch distributed operations (all-reduce, all-gather) with HPU-specific mark_step synchronization. Expert parallelism for MoE models is supported via dedicated DP/EP groups in the communicator. The NIXL (Network Offload for Inference Latency) subsystem enables disaggregated serving where prefill and decode phases run on separate HPU instances with KV cache transferred over the network. The `hetero_hpu_nixl_connector.py` implements block size conversion for heterogeneous configurations (e.g., prefill with block_size=4, decode with block_size=16) and handles the HPU-specific challenge of virtual-to-physical address translation for DMA transfers (`hpu_nixl_connector.py` patches `torch.Tensor.data_ptr()`).

### Speculative Decoding (EAGLE)

Two EAGLE implementations are provided: standard (`hpu_eagle.py`) for paged attention mode and unified (`hpu_eagle_unified.py`) for unified attention mode. Both implement iterative draft token generation with custom attention metadata preparation. The unified variant handles rejected token bookkeeping differently — it computes token indices for parallel verification within the unified batch structure. EAGLE3 support adds auxiliary hidden state outputs that are combined with target hidden states before the draft model forward pass.
