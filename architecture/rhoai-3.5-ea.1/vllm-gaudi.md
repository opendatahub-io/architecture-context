## Purpose

**Short**: Intel Gaudi (Habana) hardware plugin for vLLM that enables high-performance LLM inference on HPU accelerators.

**Detailed**: The vLLM Gaudi plugin (`vllm_gaudi`) is a hardware abstraction layer that integrates Intel Gaudi AI accelerators with the vLLM inference engine. It provides HPU-specific implementations of attention mechanisms, quantization methods, model architectures, and distributed communication primitives. The plugin follows vLLM's out-of-tree (OOT) plugin architecture via Python entry points, registering platform-specific implementations for device management, attention backends, custom kernels, and model optimizations.

Within RHOAI, this component is shipped as a container image (`odh-vllm-gaudi-rhel9`) based on UBI 9 with the Habana Synapse AI runtime (v1.23.0) and PyTorch 2.9.0. The image serves as a model serving runtime for KServe/ServingRuntime CRDs, providing an OpenAI-compatible API server for LLM inference on Intel Gaudi 2 and Gaudi 3 hardware. It is deployed as a pod within a KServe InferenceService, not as a standalone operator or service.

The plugin supports advanced inference features including FP8/GPTQ/AWQ quantization, speculative decoding (Eagle), Mixture-of-Experts (MoE) with expert parallelism, Multi-head Latent Attention (MLA), KV cache offloading, LoRA adapters, and distributed inference via tensor/data/expert parallelism using the HCCL (Habana Collective Communications Library) backend.

## Architectural Analysis

### Plugin Architecture and Monkey-Patching Pattern

The vllm-gaudi plugin integrates with upstream vLLM through two complementary mechanisms: Python entry points for clean registration, and extensive monkey-patching for behavioral overrides. The entry points (`vllm.platform_plugins` and `vllm.general_plugins` in `setup.py`) register the `HpuPlatform` class and trigger the import of custom ops, utilities, and model overrides. However, many integrations rely on runtime monkey-patching — replacing vLLM's `AsyncScheduler` with `HPUAsyncScheduler` (in `utils.py:290`), patching `Attention.forward` (in `hpu_attention.py:61`), replacing `FusedMoE.forward` (in `hpu_fused_moe.py:410`), and overriding KV offloading handlers (in `cpu_hpu.py:376-379`). While this achieves integration without forking vLLM, it creates a fragile coupling where upstream vLLM changes can silently break the plugin. The `@ClassName.register_oot` decorator pattern (used for RoPE, attention backends, and quantization methods) is the more robust approach, and more of the monkey-patches should ideally migrate to this pattern over time.

### Quantization and Hardware Optimization

The plugin provides comprehensive quantization support spanning FP8 (per-tensor, per-channel, per-block), GPTQ (2/3/4/8-bit), AWQ (4-bit), ModelOpt FP8, and CompressedTensors. Each quantization method has Gaudi-specific optimizations: FP8 uses `hpu_ops.apply_fp8_linear_hpu()` for fused linear kernels, GPTQ/AWQ use `torch.ops.hpu.convert_from_uint4()` for hardware-accelerated dequantization, and the MoE variants use dedicated `VllmMixtureOfExpertsOp*` operators. A notable cross-platform concern is FP8 scale format: Gaudi2 uses FP8_E4M3_FNUZ while Gaudi3 uses FP8_E4M3_FN, requiring scale adjustment during weight loading (`hpu_compressed_tensors.py:951-994`). The `ConvertScaleToHwAligned()` utility ensures scales are aligned to HPU vector unit boundaries for kernel efficiency.

### Distributed Inference Architecture

The plugin supports three parallelism dimensions: tensor parallelism (TP), data parallelism (DP), and expert parallelism (EP) for MoE models. All use the HCCL backend (`hpu_communicator.py`), which wraps Habana's collective communications library. The `HPUDPMetadata` system (`hpu_dp_utils.py`) coordinates MoE expert dispatch across DP ranks, using `all_gather_into_tensor` to aggregate hidden states and top-k routing decisions. For disaggregated prefill/decode, the NIXL connector (`hetero_hpu_nixl_connector.py`) enables KV cache transfer between nodes via UCX/RDMA. The heterogeneous connector is particularly sophisticated — it handles different block sizes between prefill and decode nodes, different KV cache layouts (NHD vs HND), and MLA vs standard attention formats, with per-layer block window tracking for Llama 4 local attention.

### Memory Management and KV Cache

The HPU worker implements a sleep/wake mechanism (`hpu_worker.py:386-475`) that moves model weights to CPU and discards KV cache when idle, freeing HPU memory for other workloads. The KV cache offloading system (`cpu_hpu.py`) uses HPU streams and events for async block transfers between CPU and HPU memory, with stream pooling and event-based sequencing. The `InputBatch` class (`hpu_input_batch.py`) keeps most batch metadata on CPU to minimize HPU memory pressure, using `async_h2d_copy` for selective synchronization only when sampling parameters change. The default block size is 128 (enforced by `HpuPlatform.check_and_update_config`), and the only supported KV cache quantization dtype is `fp8_inc`.

### FIPS Non-Compliance as a Design Decision

The explicit removal of the OpenSSL FIPS provider (`rpm -e --nodeps openssl-fips-provider-so`) in the Dockerfile is a deliberate design decision, not an oversight. The Habana Synapse AI runtime and its dependencies require the standard OpenSSL provider, and the FIPS provider would interfere with their cryptographic operations. This means the vllm-gaudi container image cannot run in FIPS-enforced mode, which is a deployment constraint for environments requiring FIPS 140-2/3 compliance. Since TLS termination and authentication are handled by the kube-rbac-proxy sidecar (which runs in a separate FIPS-compliant container), the actual cryptographic exposure of the non-FIPS vLLM container is limited to model weight download (which can use a FIPS-compliant init container or PVC pre-staging).
