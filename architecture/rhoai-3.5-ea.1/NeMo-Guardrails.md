## Purpose

**Short**: NeMo Guardrails is an open-source toolkit for adding programmable guardrails (input/output safety checks, content moderation, topic control, jailbreak detection) to LLM-based conversational systems.

**Detailed**: NeMo Guardrails, originally developed by NVIDIA, provides a framework for controlling and securing LLM interactions. It implements a programmable "rails" system using the Colang domain-specific language (v1.0 and v2.x), which allows users to define conversation flows, content filters, and safety policies. The toolkit intercepts user prompts and model outputs, applying configurable guardrail checks including content safety, jailbreak detection, injection detection, topic safety, hallucination checks, fact-checking, and sensitive data detection.

The Red Hat downstream fork (`red-hat-data-services/NeMo-Guardrails`) ships as the `odh-trustyai-nemo-guardrails-server` container image, which runs a FastAPI-based HTTP server exposing an OpenAI-compatible chat completions API (`/v1/chat/completions`) and a dedicated guardrail checks endpoint (`/v1/guardrail/checks`). The RHOAI distribution builds using `Dockerfile.konflux` on UBI 9 / Python 3.12, pre-downloads embedding models (spaCy, FastEmbed, Sentence Transformers) at build time, and filters out closed-source guardrail modules via a profile-based selection mechanism. The server supports OpenTelemetry auto-instrumentation for observability integration and connects to upstream LLM providers (OpenAI, vLLM, NIM, Anthropic, Azure OpenAI, Cohere) to proxy and safeguard inference requests.

## Architectural Analysis

NeMo Guardrails implements a proxy-style architecture where the server sits between the client and the upstream LLM provider, intercepting both requests and responses to apply programmable safety checks. The core abstraction is the "rail" — a configurable check that can be applied at the input, output, tool-input, or tool-output phase of a conversation turn. Rails are defined using the Colang domain-specific language, which provides a declarative syntax for specifying conversation flows and safety policies.

The server maintains an in-memory cache of `LLMRails` instances keyed by configuration ID and model name. This means configuration changes require either restarting the server or enabling the `auto_reload` watchdog feature (which monitors the config directory for filesystem changes). The thread management system uses an abstract `DataStore` interface for persisting conversation state across requests, though the concrete implementation is injected externally — the base class only defines `get`/`set` operations.

A significant downstream customization is the guardrails profile filtering system (`scripts/filter_guardrails.py` and `scripts/provider-list.yaml`). At build time, the RHOAI image uses the `opensource` profile which strips out 15 closed-source guardrail modules (ActiveFence, AI Defense, AutoAlign, Clavata, CleanLab, CrowdStrike, Fiddler, GCP Moderate Text, Pangea, PatronusAI, PolicyAI, PrivateAI, Prompt Security, Trend Micro, and jailbreak_detection). This reduces the attack surface and avoids shipping modules that depend on proprietary third-party services. The remaining open-source guardrails include content safety, topic safety, self-check (input/output/facts), hallucination detection, injection detection, regex-based filtering, sensitive data detection, factchecking (AlignScore), GLiNER, Guardrails AI, LLaMA Guard, and attention-based checks.

The OpenTelemetry integration added to the downstream fork allows the entrypoint script to wrap the server with `opentelemetry-instrument` when `OTEL_EXPORTER_OTLP_ENDPOINT` is set, enabling automatic instrumentation of FastAPI routes and httpx outbound calls. This is critical for RHOAI observability since the guardrails server sits on the critical path of every LLM interaction. The tracing system also includes custom span extractors (v1 and v2) that can create detailed per-rail spans, and supports both filesystem and OpenTelemetry log adapters.

The multi-architecture build configuration (x86_64, arm64, ppc64le) in the Konflux pipeline is noteworthy for its complexity. The ppc64le architecture requires building PyTorch, tiktoken, onnxruntime, and OpenBLAS from source (stages 2-6 in `Dockerfile.konflux`), significantly increasing build times and introducing architecture-specific conditional logic throughout the Dockerfile. The entrypoint validates the config directory structure, requiring both `config.yaml` and `rails.co` files before starting, which ensures the server doesn't start in an unconfigured state.
