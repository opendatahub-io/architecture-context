## Purpose

**Short**: Argo Workflows is the workflow execution engine that powers Data Science Pipelines in Red Hat OpenShift AI, providing container-native workflow orchestration on Kubernetes.

**Detailed**: Argo Workflows is an open-source container-native workflow engine for orchestrating parallel jobs on Kubernetes. In the RHOAI platform, it serves as the execution backend for Data Science Pipelines (DSP), enabling users to define and run multi-step ML pipelines as Kubernetes-native workflows. The system manages the full lifecycle of workflow execution including scheduling, artifact management, retry logic, and status reporting.

The component consists of three main binaries: the **workflow-controller** (a Kubernetes controller that watches Workflow CRDs and orchestrates pod-based step execution), **argoexec** (a sidecar/init-container executor injected into workflow pods to manage artifact I/O and container lifecycle), and **argo** (a CLI and API server that provides the REST/gRPC API and web UI). For RHOAI, only the workflow-controller and argoexec images are built and shipped via Konflux; the argo-server/CLI is not part of the RHOAI distribution as DSP uses its own API layer.

The workflow controller uses leader election for high availability and creates pods dynamically to execute workflow steps. It supports multiple artifact storage backends (S3/Minio, GCS, Azure Blob, HDFS, Alibaba OSS), optional workflow archival to PostgreSQL or MySQL, and extensive workflow lifecycle management including cron scheduling, TTL-based cleanup, and artifact garbage collection.

## Architectural Analysis

The Argo Workflows codebase in this RHOAI fork is a downstream build of the upstream argoproj/argo-workflows project, with Red Hat-specific Dockerfiles and Konflux CI integration. The primary architectural characteristic is the separation of concerns between the workflow-controller (control plane) and argoexec (data plane): the controller watches Workflow CRDs and creates pods, while argoexec runs inside each workflow pod to manage the actual step execution, artifact I/O, and result reporting.

The argoexec component uses the "emissary" executor pattern, which is the only supported executor in modern Argo Workflows. The emissary approach wraps the user's container command: argoexec is set as the container entrypoint, and it in turn executes the user's command while managing signal forwarding, dependency ordering (for DAG-based workflows with container sets), stdout/stderr capture for script outputs, and artifact collection. Communication between argoexec and the controller happens asynchronously through WorkflowTaskResult CRDs rather than direct network calls, making the system resilient to controller restarts.

A notable FIPS consideration: the argoexec Konflux Dockerfile builds two binaries - `argoexec` (statically linked, non-FIPS, `CGO_ENABLED=0`) and `argoexec-fips` (dynamically linked, FIPS-compliant, `CGO_ENABLED=1 GOEXPERIMENT=strictfipsruntime`). Both are included in the image, and the DSPO or runtime environment selects which to use. The workflow-controller Konflux image only builds the FIPS variant. This dual-binary approach allows flexibility in environments where FIPS mode may be optional.

The TLS implementation has a security consideration worth noting: when auto-generating TLS certificates, the code sets `InsecureSkipVerify: true` in the TLS config (util/tls/tls.go:127). This is because the auto-generated certs are self-signed and not rooted in a trusted CA. In the RHOAI context, where the Argo Server is not deployed and the controller communicates with the Kubernetes API via in-cluster config (which handles its own TLS verification), this has limited exposure. However, the SSO configuration also supports `InsecureSkipVerify` for OIDC issuer connections, which could weaken TLS validation in SSO-enabled deployments.

The workflow controller uses Kubernetes leader election via Lease resources in the `coordination.k8s.io` API group, ensuring only one active controller instance processes workflows at a time. The controller supports configurable worker pools for different concerns (workflow reconciliation, TTL cleanup, pod cleanup, cron scheduling, archival), enabling tuning for different workload profiles. Default worker counts total 56 concurrent workers across all pools.
