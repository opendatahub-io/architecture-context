## Purpose

**Short**: Argo Workflows is the workflow execution engine used by Data Science Pipelines in RHOAI to orchestrate multi-step ML pipeline runs as Kubernetes-native workflows.

**Detailed**: Argo Workflows provides a Kubernetes-native workflow engine that defines and runs complex, multi-step computational workflows as CRDs. In the RHOAI context, it serves as the backend execution engine for Data Science Pipelines (DSP), where Kubeflow Pipelines compiles ML pipeline definitions into Argo Workflow CRs. The workflow controller watches these CRs and creates pods for each pipeline step, managing their lifecycle, dependency resolution, artifact passing, and retry logic.

The system consists of three main binaries: the **workflow-controller** that reconciles Workflow CRs and manages pod lifecycle; the **argoexec** executor that runs as a sidecar in workflow pods to manage container lifecycle, capture outputs, and handle artifact storage; and the **argo** CLI/server that provides a gRPC+REST API and web UI for workflow management. RHOAI ships only the workflow-controller and argoexec images via Konflux; the argo-server is typically not deployed directly but is managed through the Data Science Pipelines operator.

## Architecture Components

### argoexec

**Dockerfile**: `argo-argoexec/Dockerfile.konflux`
**Intent**: The workflow executor sidecar that runs inside every workflow pod. Uses the "emissary" execution model where it wraps the user's container command, manages inter-container dependencies via file-based signaling through `/var/run/argo/`, captures output parameters and artifacts, and reports results back to the controller via WorkflowTaskResult CRs.
**Image**: odh-data-science-pipelines-argo-argoexec
**Language**: Go
**Port**: None (runs as init/sidecar container, communicates via filesystem)

## Architectural Analysis

Argo Workflows in the RHOAI context serves as the backend execution engine for Data Science Pipelines, not as a standalone workflow platform. This means several of its capabilities (the argo-server UI, SSO, event-driven workflows, CLI) are largely unused — the DSP API Server and Dashboard provide the user-facing layer while Argo handles the "pod-level plumbing." The RHOAI Konflux build produces only two images (workflow-controller and argoexec), confirming that the argo-server/CLI is not part of the RHOAI distribution.

The **emissary executor model** is architecturally significant. Rather than using a separate sidecar container that monitors the main container via the Kubernetes API (the older "docker" and "k8sapi" executor modes), the emissary wraps the user's command by replacing the container entrypoint. It uses a shared emptyDir volume at `/var/run/argo/` for inter-container communication: the controller writes template definitions, the emissary writes combined stdout/stderr and exit codes, and the wait container reads results. This reduces Kubernetes API load compared to polling-based approaches but requires the executor image to be injected as an init container that copies the `argoexec` binary into the shared volume.

The **dual-binary FIPS strategy** in the argoexec Dockerfile is notable. The Konflux build produces both a FIPS-compliant (`argoexec-fips`, dynamically linked with CGO_ENABLED=1 and strictfipsruntime) and non-FIPS (`argoexec`, statically linked with CGO_ENABLED=0) binary. Both are copied into the final image, and the ENTRYPOINT defaults to the non-FIPS variant. This allows the deployer (DSPO) to select FIPS mode by overriding the entrypoint to `/bin/argoexec-fips` at runtime, without needing separate container images. The workflow-controller Konflux build, in contrast, produces only the FIPS variant — there is no non-FIPS fallback, which is the correct approach for a controller that runs on the platform itself rather than in user workload pods.

The **TLS configuration** has a few patterns worth noting for security review. The self-signed certificate generator in `util/tls/tls.go` sets `InsecureSkipVerify: true` on the TLS config (line 127), which is expected for a self-signed cert scenario but means the server won't validate its own certificate. The internal gRPC gateway loopback connection (argoserver.go:352) also uses InsecureSkipVerify when cloning the TLS config for the localhost dial. The SSO configuration allows `insecureSkipVerify` as a user-configurable option (config/sso.go:23), which could bypass OIDC provider certificate validation. These are all intentional design choices but represent potential security configuration risks if misused.

The **worker architecture** in the controller is highly configurable with separate worker pools for different concerns: 32 workflow reconcilers, 4 TTL cleanup workers, 4 pod cleanup workers, 8 cron workers, and 8 archive workers. This separation prevents slow archiving or cleanup operations from blocking workflow reconciliation. The controller uses Kubernetes leader election via Lease resources for HA deployments, with configurable lease duration, renew deadline, and retry period.
