## Purpose

**Short**: Platform operator that orchestrates the complete lifecycle of Red Hat OpenShift AI (RHOAI) — deploying, configuring, and managing all platform components, ingress infrastructure, authentication, monitoring, and cloud provider integrations.

**Detailed**: The rhods-operator (also known as opendatahub-operator downstream) is the central control plane for RHOAI. It manages the entire platform through two primary custom resources: `DSCInitialization` (platform-level initialization — namespaces, networking, auth, monitoring, gateway) and `DataScienceCluster` (component lifecycle — enabling/disabling individual AI/ML components like KServe, Dashboard, Pipelines, etc.).

The operator deploys a complete ingress stack based on Kubernetes Gateway API with Istio/Envoy data plane, including an OAuth2/OIDC authentication proxy (kube-auth-proxy), EnvoyFilter-based auth enforcement, and legacy URL redirect services. It manages 16+ individual component controllers that deploy downstream AI/ML operators and services via kustomize manifests fetched at pinned commits.

For RHOAI 3.x, the operator also includes a cloud manager subsystem (`cloudmanager` binary) that manages Helm-based dependency installation (Gateway API, cert-manager, LeaderWorkerSet, Sail/Istio) on non-OpenShift Kubernetes clusters (Azure AKS, CoreWeave). The operator provides comprehensive admission webhooks for singleton enforcement, hardware profile injection, connection management, and API deprecation guidance.

## Architectural Analysis

The rhods-operator is architecturally the most complex component in the RHOAI platform. It follows a **hierarchical controller pattern**: the DSCInitialization controller initializes platform-level infrastructure (namespaces, networking, auth, monitoring, gateway), and the DataScienceCluster controller manages individual component lifecycles. Each component has its own controller that renders kustomize manifests from `prefetched-manifests/` and applies them via ServerSideApply. This two-level hierarchy allows the operator to manage both platform-wide concerns and per-component concerns independently.

The **gateway/ingress architecture** is particularly sophisticated. RHOAI 3.x deploys a full Kubernetes Gateway API stack: a GatewayClass and Gateway CR served by an Istio/Envoy data plane, with EnvoyFilter CRs for ext_authz authentication enforcement and Lua-based token forwarding. The authentication proxy (kube-auth-proxy) supports three modes — integrated OpenShift OAuth, external OIDC, and no-auth — detected at runtime from cluster configuration. The ingress supports two operational modes: LoadBalancer (for custom domains with dual HTTPS listeners) and OcpRoute (for standard OpenShift routing with ClusterIP service). Legacy URL redirects are handled by a separate nginx-based Deployment + Route chain that 301-redirects old dashboard/gateway hostnames to the new Gateway API hostname.

The **component registry pattern** is a key design decision. All 16 component controllers register themselves in a central registry (`internal/controller/components/registry/`), and the DSC controller iterates this registry to create component CRs for enabled components. Each component controller follows a consistent action pipeline: initialize → releases → kustomize render → deploy → check deployments → garbage collect. This pattern enables adding new components with minimal boilerplate. Components interact with each other primarily through the GatewayConfig CR (for domain/OIDC configuration) and the Auth CR (for admin group membership).

The **cloud manager subsystem** is a separate binary (`cloudmanager`) packaged in the same container image but designed for non-OpenShift Kubernetes deployments. It manages Helm-based dependency installation (Gateway API CRDs, cert-manager, LeaderWorkerSet, Sail/Istio) via `AzureKubernetesEngine` and `CoreWeaveKubernetesEngine` CRs. The Helm charts are prefetched at build time into `prefetched-charts/`. A notable design pattern is the `SkipCRDIfPresent` hook that avoids ServerSideApply ownership conflicts when CRDs (like ServiceMonitor) are already managed by another operator.

The **webhook architecture** is extensive — 12+ webhook handlers covering validation, defaulting, mutation, and deprecation. The HardwareProfile injection webhook (1463 lines) is the most complex, supporting three workload types (Notebooks, InferenceServices, LLMInferenceServices) with container-aware resource injection, toleration merging, Kueue label management, and profile change detection. The deprecation webhook framework uses a generic `TypeValidator` pattern that blocks creation of deprecated API types with clear migration guidance.

A significant risk area is the **FIPS compliance gap**: the operator is built with `GOEXPERIMENT=strictfipsruntime` and `CGO_ENABLED=1` (FIPS-compliant binary), but the OLM CSV annotation declares `fips-compliant: "false"`. The EnvoyFilter's DestinationRule uses `InsecureSkipVerify` for the kube-auth-proxy connection (configurable via GatewayConfig). This disconnect between build-time FIPS compliance and the declarative annotation needs investigation.
