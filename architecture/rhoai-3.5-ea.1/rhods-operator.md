## Purpose

**Short**: Platform operator that deploys, configures, and manages the lifecycle of all Red Hat OpenShift AI components, services, and ingress infrastructure.

**Detailed**: The rhods-operator (also known as the opendatahub-operator in upstream) is the central orchestration operator for Red Hat OpenShift AI (RHOAI). It manages the complete platform lifecycle through two singleton cluster-scoped Custom Resources: `DSCInitialization` (DSCI) for infrastructure prerequisites (namespaces, RBAC, networking, gateway, authentication, monitoring) and `DataScienceCluster` (DSC) for component enablement and configuration (Dashboard, KServe, Kueue, Ray, Workbenches, Model Registry, Training Operator, TrustyAI, Data Science Pipelines, and more).

The operator implements a sophisticated action-based reconciliation framework where each component is managed by a dedicated controller that renders kustomize manifests from prefetched bundles. In RHOAI 3.x, the operator deploys a complete ingress stack built on Kubernetes Gateway API: a Gateway CR with Envoy data plane, EnvoyFilter CRs for authentication injection, a kube-auth-proxy deployment (OAuth2/OIDC), HTTPRoutes for per-component routing, and dashboard redirect infrastructure for legacy URL migration. The operator also includes a separate CloudManager binary for Azure AKS and CoreWeave cluster provisioning with Helm chart management.

The operator supports three platform modes: Self-Managed RHOAI, Managed RHOAI (addon), and upstream Open Data Hub, with platform-specific behavior for namespace naming, RBAC groups, monitoring configuration, and component defaults. It uses OLM (Operator Lifecycle Manager) for installation and includes conversion webhooks for v1→v2 API migration.

## Architectural Analysis

The rhods-operator is one of the most architecturally complex operators in the RHOAI platform, serving as the single control plane for the entire ML/AI stack. Its design reflects several deliberate architectural decisions that merit analysis.

**Action-Based Reconciliation Framework**: Rather than monolithic reconcile functions, each controller composes a pipeline of discrete action functions (`initialize → checkPreConditions → render → deploy → gc`). This pattern, implemented in `pkg/controller/reconciler/`, enables code reuse across 16+ component controllers while allowing per-component customization. Each action receives a `ReconciliationRequest` struct containing the reconciled resource, client, and accumulated resources. The kustomize rendering action (`pkg/controller/actions/render/kustomize/`) transforms component manifests from `prefetched-manifests/` using overlays, params.env image substitution, and label injection before the deploy action applies them via Server-Side Apply. This separation of rendering from deployment is a key enabler for the multi-distribution support (ODH vs RHOAI vs RHAII).

**Gateway API Ingress Architecture**: The RHOAI 3.x ingress stack is a significant departure from the 2.x Route-based model. The Gateway controller (`internal/controller/services/gateway/`) deploys a complete ingress chain: GatewayClass → Gateway → Envoy (OpenShift Gateway Controller) → EnvoyFilter (ext_authz + Lua) → kube-auth-proxy → backend HTTPRoutes. The EnvoyFilter uses Istio's `ext_authz` pattern to intercept every request, forward credentials to kube-auth-proxy for validation, then use a Lua filter to transform tokens (access_token → Bearer header for OAuth, id_token preservation for OIDC) and strip OAuth2 proxy cookies. This architecture provides a single authentication enforcement point for all platform components while supporting both integrated OpenShift OAuth and external OIDC providers. The DestinationRule uses `InsecureSkipVerify: true` for internal Service-CA certificates, which is acceptable for in-cluster communication but notable from a security audit perspective.

**Dual-Binary Architecture**: The Dockerfile.konflux produces two binaries — `manager` and `cloudmanager` — from the same source tree. The cloudmanager operates independently with its own CRDs (AzureKubernetesEngine, CoreWeaveKubernetesEngine), leader election, and Helm chart management. This separation prevents cloud-specific concerns from increasing the attack surface of the core platform operator while sharing the same Go module graph and action framework. The cloud manager bootstraps cert-manager, Gateway API CRDs, LeaderWorkerSet, and Sail operator via Helm charts, with hooks to work around known issues (e.g., OSSM-12397 for Istio webhook annotations).

**Component Registry Pattern**: Components are registered in a type-safe registry (`internal/controller/components/registry/`) that the DSC controller iterates during reconciliation. Each component implements a `ComponentHandler` interface with `NewCRObject()`, `UpdateDSCStatus()`, and `IsEnabled()` methods. This allows adding new components (e.g., Trainer, MLflowOperator, SparkOperator were added in v2) without modifying the core orchestration logic. The v1→v2 API conversion webhooks handle field renaming (DataSciencePipelines→AIPipelines) and new component introduction with `Removed` default state.

**Multi-Namespace Resource Management**: The operator manages resources across 5+ namespaces simultaneously: the operator namespace (redhat-ods-operator), the application namespace (configurable, default redhat-ods-applications), openshift-ingress (Gateway and auth proxy infrastructure), monitoring namespace, and conditional namespaces (models-as-a-service, kuadrant-system). The controller-runtime cache is configured with namespace-scoped selectors to avoid watching unnecessary resources, which is critical for memory efficiency in production clusters. The NetworkPolicy created in the application namespace allows traffic from monitoring, ingress, and labeled application namespaces while blocking everything else.
