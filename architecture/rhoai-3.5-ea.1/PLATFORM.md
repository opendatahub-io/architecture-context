## Platform Overview

Red Hat OpenShift AI (RHOAI) 3.5-ea.1 is a comprehensive AI/ML platform deployed on OpenShift Container Platform that provides end-to-end capabilities for data science workbench management, ML pipeline orchestration, model training, model serving, model evaluation, feature stores, experiment tracking, and AI safety guardrails. The platform ships 54 components producing 129 container images, coordinated by a single root operator (`rhods-operator`) that manages the lifecycle of 16 component controllers and 5 service controllers through an OLM-delivered ClusterServiceVersion.

The platform's ingress architecture underwent a fundamental shift in the 3.x series, moving from OpenShift Route-based ingress to Gateway API. The `rhods-operator` deploys a complete ingress chain: GatewayClass, Gateway (backed by Envoy via the OpenShift Gateway Controller), EnvoyFilter with `ext_authz` for centralized authentication, and kube-auth-proxy for OIDC/OAuth credential validation. All platform components expose themselves via HTTPRoute resources attached to this shared gateway, providing a single authentication enforcement point. This architecture supports both integrated OpenShift OAuth and external OIDC identity providers.

The security posture is anchored in several platform-wide patterns: FIPS 140-2 compliance via `GOEXPERIMENT=strictfipsruntime` for Go binaries and FIPS-validated crypto libraries for Python/Rust/C++ components, non-root container execution (241 of 264 user-specified Dockerfiles run as non-root), UBI9-based container images for Red Hat supply chain assurance, kube-rbac-proxy sidecars for Kubernetes RBAC-enforced authentication on operator metrics and service endpoints, and NetworkPolicy-based microsegmentation. The platform supports disconnected (air-gapped) installations via the RELATED_IMAGE pattern with OLM relatedImages and digest-pinned image references.

## Deployment Architecture

### Deployment Topology

The platform follows a hierarchical operator deployment model. The `rhods-operator` is installed via OLM into the `redhat-ods-operator` namespace and creates two singleton CRs: `DSCInitialization` (cluster infrastructure) and `DataScienceCluster` (component enablement). The DSCInitialization controller provisions the application namespace (`redhat-ods-applications`), deploys the Gateway API ingress stack into `openshift-ingress`, sets up monitoring infrastructure (Prometheus, Thanos, OpenTelemetry), creates platform RBAC (admin/allowed group ClusterRoles), and installs HardwareProfile CRDs. The DataScienceCluster controller then iterates through 16 registered component handlers, deploying each enabled component's operator or service into the application namespace.

User-facing workloads (notebooks, inference services, pipelines, training jobs, Ray clusters, Spark applications, MLflow instances, feature stores) are deployed into user-created data science project namespaces. Operators in the application namespace watch across all namespaces for their respective CRDs, while NetworkPolicies restrict traffic to only monitoring, ingress, and labeled application namespaces.

#### Manifest Consumption Pipeline

Component manifests flow from individual component repositories through a multi-stage pipeline: component repos maintain their own kustomize structures with base and overlay directories. The `get_all_manifests.sh` script in the rhods-operator repository clones each component repo at pinned commits and copies manifests into `prefetched-manifests/`. During operator reconciliation, the action-based framework renders these manifests through kustomize (applying distribution overlays, params.env image substitution, and label injection) and applies them to the cluster via Server-Side Apply. Konflux CI builds the operator image with pre-fetched manifests baked in, ensuring hermetic deployment without runtime git clones.

### Disconnected Support

RHOAI fully supports air-gapped (disconnected) installations through the RELATED_IMAGE pattern. The rhods-operator's ClusterServiceVersion declares all 129 container images as `relatedImages` entries with sha256 digest references, enabling OLM to catalog them for image mirroring. Administrators use `oc-mirror` to create an ImageContentSourcePolicy (ICSP) or ImageDigestMirrorSet (IDMS) that redirects image pulls from the public registry to an internal mirror. The operator's `params.env` files use `RELATED_IMAGE_*` environment variables that are resolved at reconciliation time, ensuring all downstream components pull from the mirrored registry. All image references throughout the platform use digests rather than tags, preventing cache poisoning and ensuring bit-for-bit reproducibility in disconnected environments.

## Platform Architectural Analysis

RHOAI 3.5-ea.1 exhibits a maturing but architecturally heterogeneous platform with several cross-cutting observations that emerge from analyzing all 54 components together.

**Operator proliferation and the single-root-operator pattern.** The platform's most distinctive architectural choice is having a single root operator (rhods-operator) that deploys 16 sub-operators, each of which manages its own CRDs and workloads. This creates a two-tier operator hierarchy: the root operator renders and applies sub-operator manifests via kustomize, while sub-operators independently reconcile user-facing resources. The benefit is a single OLM subscription and unified lifecycle management, but the cost is significant complexity in the root operator (managing 16 component controllers, 5 service controllers, 9 webhook systems, and cross-namespace resource management across 5+ namespaces). The action-based reconciliation framework (`initialize -> checkPreConditions -> render -> deploy -> gc`) mitigates this by providing a composable pipeline, but the rhods-operator remains the single point of failure for the entire platform.

**Authentication inconsistency across components.** Despite the Gateway API providing a centralized authentication enforcement point, individual components implement authentication differently depending on when they were added to the platform. Older components (dashboard, notebook controllers, KubeRay) use the kube-rbac-proxy sidecar pattern with SubjectAccessReview. The Gateway layer uses kube-auth-proxy (an oauth2-proxy fork) with OIDC/OAuth. EvalHub implements application-level SAR checks with X-Tenant headers. MLflow uses a custom Kubernetes ServiceAccount token auth plugin. MaaS integrates with Kuadrant/Authorino for API key and rate-limiting policy. This diversity means there are at least five distinct authentication code paths, each with different token formats, session management, and failure modes. The gateway's ext_authz provides a baseline, but components behind it often perform additional authorization that is not centrally auditable.

**The llm-d inference stack as a platform-within-a-platform.** The llm-d ecosystem (llm-d, llm-d-inference-scheduler, llm-d-kv-cache, llm-d-routing-sidecar) introduces architectural patterns not seen elsewhere in RHOAI: disaggregated prefill/decode (P/D) inference with KV-cache transfer via NIXL/LMCache protocols, Envoy ext-proc-based intelligent request scheduling with 40+ pluggable scheduler plugins, ZMQ-based event streaming for real-time KV-cache block indexing, and multi-accelerator support (CUDA, ROCm, Gaudi, CPU). This stack operates somewhat independently of the KServe control plane, using its own InferencePool and InferenceObjective CRDs from the Gateway API Inference Extension. The integration point is through the KServe LLMInferenceService CRD (v1alpha2), which reconciles into InferencePool + LeaderWorkerSet + WVA resources, bridging the KServe abstraction with the llm-d data plane.

**Module Federation as an extensibility strategy.** The odh-dashboard's Module Federation architecture represents a deliberate shift from monolithic to composable UI. Each federated module (model-registry, gen-ai, maas, mlflow, eval-hub, automl, autorag) ships as an independent container with both a React frontend and a Go BFF, communicating via a `federation-config` ConfigMap. This enables independent release cycles for UI features, but it also means a single dashboard pod runs 8 containers (host + 7 modules) sharing a Kubernetes Service with 8 different ports. Inter-BFF communication (e.g., gen-ai BFF calling maas BFF on port 8243) creates a service mesh within the pod that is not visible to Kubernetes-level observability tools.

**Training operator duplication.** The platform ships both `training-operator` (Kubeflow v1, managing PyTorchJob/TFJob/XGBoostJob/MPIJob/PaddlePaddleJob/JAXJob CRDs) and `trainer` (Kubeflow v2, managing TrainJob/ClusterTrainingRuntime/TrainingRuntime CRDs plus a Rust data-cache service). Both are deployed simultaneously, creating overlapping functionality for distributed training. The newer Trainer introduces JobSet-based orchestration, init containers for dataset/model download, and the Apache Arrow Flight-based data cache -- features absent in Training Operator. This appears to be a transitional state where both must coexist until Training Operator deprecation.

**FIPS compliance gaps.** While the platform declares FIPS compliance and most Go binaries are built with `GOEXPERIMENT=strictfipsruntime` and `CGO_ENABLED=1`, the KServe localmodel controller and localmodel-agent use `CGO_ENABLED=0` (static linking) without FIPS tags. This means these binaries use Go's pure-Go crypto instead of OpenSSL, which would fail FIPS check-payload validation. Similarly, upstream components like Argo Workflows, ml-metadata, and several Python-based services have varying levels of FIPS compliance that depend on the base image's OpenSSL configuration rather than explicit FIPS build flags.

**Evaluation and guardrails convergence.** The platform has accumulated three distinct guardrails approaches: FMS Guardrails Orchestrator (Rust-based, detector pipeline pattern), NeMo Guardrails (Python-based, Colang DSL), and AI Gateway Payload Processing (Go-based, Envoy ext_proc with NeMo integration). Similarly, evaluation spans EvalHub (centralized Go service), TrustyAI Service Operator's LMES controller (bare Pod pattern), llama-stack-provider-ragas (Llama Stack Eval API), and llama-stack-provider-trustyai-garak (red-teaming). These parallel systems reflect rapid feature addition from different teams but create integration surface area that users must navigate.
