## Platform Overview

Red Hat OpenShift AI (RHOAI) 3.5-ea.1 is a comprehensive AI/ML platform deployed on OpenShift Container Platform that provides end-to-end lifecycle management for machine learning and large language model workloads. The platform spans model development (Jupyter notebooks, CodeFlare SDK), training (Kubeflow Training Operator v1/v2, distributed PyTorch/MPI/JAX), pipeline orchestration (Data Science Pipelines built on Kubeflow Pipelines v2 + Argo Workflows), model serving (KServe with InferenceService and LLMInferenceService CRDs, vLLM, OpenVINO, MLServer), model management (Model Registry, MLflow), evaluation (EvalHub, TrustyAI, lm-evaluation-harness), and AI safety (FMS Guardrails Orchestrator, NeMo Guardrails, TrustyAI).

The platform is deployed via a single OLM-managed operator (`rhods-operator`) that uses a three-tier controller hierarchy: DSCInitialization creates platform infrastructure, DataScienceCluster manages 16 component controllers, and individual component controllers render kustomize manifests for each subsystem. Ingress is handled through Gateway API with automatic detection of LoadBalancer (xKS) or OcpRoute (OpenShift) modes, unified by an EnvoyFilter-based authentication stack supporting IntegratedOAuth, OIDC, and unauthenticated modes. A cloud manager binary provides separate operator lifecycle for non-OpenShift Kubernetes platforms (Azure AKS, CoreWeave).

The security posture centers on FIPS 140-2 compliance via `GOEXPERIMENT=strictfipsruntime` for Go binaries, UBI 9-based container images with non-root execution (USER 1001 or 65532), kube-rbac-proxy sidecars for Kubernetes-native authentication/authorization across all operator-managed services, and comprehensive NetworkPolicy isolation. The platform supports disconnected deployment through RELATED_IMAGE environment variables and OLM relatedImages, enabling air-gapped operation with image mirroring.

### Central Components

**rhods-operator** is the platform root — 14+ components directly depend on it for deployment, configuration, and lifecycle management. It orchestrates all component installations via kustomize manifest rendering and manages the shared gateway/ingress infrastructure.

**KServe** is the inference backbone with 8 reverse dependencies. It provides the InferenceService and LLMInferenceService CRDs that odh-model-controller, Model Registry, TrustyAI, odh-dashboard, and the llm-d stack all integrate with. Its dual-controller architecture (traditional InferenceService + LLMInferenceService) reflects the platform's evolution from classical ML to LLM workloads.

**kube-rbac-proxy** is the shared authentication/authorization primitive, used as a sidecar by odh-dashboard, Model Registry Operator, TrustyAI Service Operator, OGX distribution, and EvalHub. It provides Kubernetes-native RBAC enforcement via TokenReview/SubjectAccessReview without requiring Istio or external identity providers.

**odh-dashboard** is the primary user-facing entry point with 17 outgoing dependencies, integrating with virtually every platform subsystem through its micro-frontend architecture (7 federated UI plugins + Go BFF containers).

**Kueue** appears as a transversal dependency for resource quota management across Ray clusters (CodeFlare SDK), training jobs (Trainer), evaluation jobs (EvalHub), and Spark applications.

## Deployment Architecture

### Deployment Topology

The platform follows a hub-and-spoke deployment model. The **operator namespace** (`redhat-ods-operator`) contains the rhods-operator and its OLM subscription. The **platform namespace** (`redhat-ods-applications`) hosts all platform-level controllers, the dashboard, and shared infrastructure (auth proxy, monitoring agent). The **monitoring namespace** (`redhat-ods-monitoring`) contains the dedicated Prometheus and AlertManager instances. **User namespaces** are created per data science project, each hosting their own DSP instances, notebooks, InferenceServices, training jobs, MLflow instances, and Model Registry instances. The **ingress namespace** hosts the Istio/Gateway API infrastructure including the `data-science-gateway` Gateway resource.

Component operators run in `redhat-ods-applications` but create workloads (Deployments, StatefulSets, Jobs) in user namespaces via ownerReference-less resources, using label selectors and finalizers for lifecycle management. This cross-namespace management pattern is consistent across KServe, Data Science Pipelines, Model Registry Operator, MLflow Operator, TrustyAI Service Operator, and EvalHub.

### Disconnected Support

RHOAI supports air-gapped/disconnected deployment through a comprehensive image reference management system. All container image references are defined as `RELATED_IMAGE_*` environment variables in the operator's ClusterServiceVersion (CSV), enabling OLM to populate the `relatedImages` field for catalog mirroring. The operator resolves images at runtime from these environment variables rather than hardcoded references. Image mirroring is performed via `oc-mirror` or `oc adm catalog mirror`, and all production images use SHA256 digest pinning (not floating tags) to ensure reproducibility. The operator's `disconnected` feature flag is enabled by default, and the platform detects disconnected environments to disable telemetry and external network access where applicable.

## Platform Architectural Analysis

RHOAI 3.5-ea.1 reveals a platform undergoing a significant architectural transition across several axes simultaneously, creating both opportunity and tension.

**The LLM Serving Stack Bifurcation.** The most visible architectural tension is between the classical InferenceService path (KServe v1beta1, single-container Deployments, Knative optional) and the new LLMInferenceService path (KServe v1alpha2, LeaderWorkerSets, disaggregated prefill/decode, InferencePool + EPP routing). These two paths share a codebase but have fundamentally different resource graphs — LLMInferenceService creates InferencePools, VariantAutoscalings, LeaderWorkerSets, PodMonitors, ServiceMonitors, and DestinationRules that InferenceService never touches. The llm-d stack (vLLM container images, EPP scheduler, KV-cache indexer, routing sidecar, WVA autoscaler, batch-gateway) adds another layer of complexity. This represents a transition from "model serving as a Deployment wrapper" to "model serving as a distributed system" — but both paths must coexist during the transition.

**Authentication Fragmentation.** The platform uses at least five distinct authentication mechanisms: (1) kube-rbac-proxy sidecars for operator-managed services, (2) EnvoyFilter + kube-auth-proxy for gateway-level auth, (3) Authorino for MaaS API key management, (4) mTLS for Ray cluster communication, and (5) ServiceAccount RBAC for pod-to-API-server access. While each choice is reasonable in isolation, the aggregate creates operational complexity. The kube-rbac-proxy pattern is the most consistent — used by 6+ components with shared FIPS-compliant cipher suites — but its deployment varies (some inject via mutating webhook, others via static manifests). The transition from `oauth-proxy` to `kube-rbac-proxy` in KServe inference pods (preserving existing containers during upgrades) shows the difficulty of evolving auth mechanisms in a running platform.

**The Evaluation Ecosystem Explosion.** RHOAI 3.5 introduces a rich evaluation stack (EvalHub + 5+ evaluation backends: lm-evaluation-harness, RAGAS, Garak, GuideLLM, LightEval) managed by the TrustyAI Service Operator. The operator's multi-controller architecture (5 CRD controllers in one binary) is architecturally sound, but the evaluation ecosystem creates a complex dependency web: evaluation jobs run in tenant namespaces, need access to model endpoints (KServe), experiment tracking (MLflow), artifact storage (OCI registries), and job scheduling (Kueue). The EvalHub's failure reconciler — monitoring Jobs and Kueue Workloads for operator-only failures that crashed containers cannot self-report — is an elegant solution to an inherent distributed systems problem.

**Module Federation as Platform UI Strategy.** The odh-dashboard's evolution to a micro-frontend architecture (7 federated plugins in a single pod with 9+ containers) is architecturally ambitious. Each plugin (gen-ai, maas, model-registry, mlflow, eval-hub, automl, autorag) has its own Go BFF providing domain-specific API proxying and SubjectAccessReview authorization. The tradeoff is clear: strong domain isolation and independent deployability vs. operational complexity (a single pod with 9 containers is challenging to debug) and resource overhead. The inter-BFF communication pattern (e.g., gen-ai BFF calling maas BFF for token delegation) creates implicit coupling that may not be visible in standard dependency analysis.

**Cross-Cutting FIPS Gap.** While the platform maintains excellent FIPS compliance discipline (strictfipsruntime, UBI 9, FIPS cipher suites), there are known gaps: KServe localmodel components use CGO_ENABLED=0 (static linking without FIPS crypto), KServe uses InsecureSkipVerify in PodMonitor TLS config, and the dashboard has INSECURE_SKIP_VERIFY flags in Go BFFs. These gaps reflect the tension between security compliance and development velocity — each has a pragmatic justification but collectively weakens the FIPS posture.

**CRD API Version Maturity Spectrum.** The platform spans CRD API versions from v1 (mature: DSC, DSCI, InferencePool migration) through v1beta1 (stable: InferenceService) to v1alpha1 (experimental: most new CRDs). The InferencePool v1alpha2→v1 migration uses a one-way lock mechanism (annotation-based, no fallback once migrated) that could strand resources if the v1 version has unexpected issues. The DSC v1→v2 migration introduces API version guards in webhooks that prevent v1 updates when v2-only components are enabled — a necessary but potentially confusing restriction for operators familiar with v1.

**Language and Runtime Diversity.** The 57 components span 7+ languages: Go (31), Python (30), Rust (4), C++ (2), Java (1), TypeScript (5), and C (1). The Rust components (FMS Guardrails Orchestrator, guardrails-regex-detector, vllm-orchestrator-gateway) share an axum web framework but are otherwise independent. The Python components range from simple libraries (training-hub, CodeFlare SDK) to complex services (vLLM, MLflow, Feast). This diversity is natural for an AI/ML platform but creates operational challenges: different build systems (Go modules, pip/poetry, Cargo, Maven, npm), different observability patterns (Prometheus client libraries differ by language), and different TLS handling approaches.
