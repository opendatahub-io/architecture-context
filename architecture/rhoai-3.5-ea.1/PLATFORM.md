## Platform Overview

Red Hat OpenShift AI (RHOAI) is a comprehensive machine learning and artificial intelligence platform deployed on OpenShift Container Platform. It provides end-to-end capabilities spanning notebook-based development (Jupyter workbenches), distributed training (PyTorch, TensorFlow, JAX, MPI, XGBoost, Paddle), ML pipeline orchestration (Kubeflow Pipelines v2 with Argo Workflows), model serving (KServe with InferenceService and LLMInferenceService), model registry, experiment tracking (MLflow), feature stores (Feast), AI safety guardrails (FMS Guardrails Orchestrator, NeMo Guardrails, TrustyAI), LLM-optimized inference (llm-d disaggregated prefill/decode, vLLM variants), and Models-as-a-Service with subscription-based access control.

The platform is deployed via a central Kubernetes operator (`rhods-operator`) managed through OLM (Operator Lifecycle Manager). This operator reconciles two singleton cluster-scoped CRDs — `DSCInitialization` for infrastructure prerequisites (namespaces, RBAC, networking, gateway, authentication, monitoring) and `DataScienceCluster` for component enablement. The operator uses an action-based reconciliation framework where 16 component controllers render kustomize manifests from prefetched bundles. In RHOAI 3.x, the platform deploys a Gateway API-based ingress stack: a Gateway CR with Envoy data plane, EnvoyFilter CRs for authentication injection via ext_authz, a kube-auth-proxy deployment (OAuth2/OIDC), and HTTPRoutes for per-component routing.

The security posture combines multiple authentication patterns: Gateway-level OAuth2/OIDC enforcement via kube-auth-proxy and Envoy ext_authz for the core platform, kube-rbac-proxy sidecars with Kubernetes SubjectAccessReview for component-level RBAC, and Kuadrant/Authorino-based AuthPolicy with TokenRateLimitPolicy for the Models-as-a-Service subsystem. TLS is managed through OpenShift service-ca auto-generated certificates for internal services, with FIPS-compliant builds (GOEXPERIMENT=strictfipsruntime, CGO_ENABLED=1) for Go components. The platform supports disconnected/air-gapped deployment via RELATED_IMAGE environment variables and OLM relatedImages, and builds multi-architecture images (amd64, arm64, ppc64le, s390x) through Konflux CI/CD pipelines.

### Central Components

- **rhods-operator**: Central orchestration operator managing 16 component controllers, 5 service controllers (Gateway, Auth, Monitoring, CertConfigMapGenerator, Setup), and 9 webhook systems. All other platform components are deployed through its reconciliation framework.
- **KServe**: Core model serving infrastructure used by vLLM, OpenVINO, MLServer, guardrails, and the llm-d stack. Provides InferenceService and LLMInferenceService CRDs consumed by TrustyAI, Model Registry, and Model Controller.
- **Gateway API / kube-auth-proxy**: Platform ingress stack providing unified authentication for all HTTPS traffic through Envoy ext_authz and OAuth2/OIDC proxy.
- **kube-rbac-proxy**: Most widely used auth sidecar, injected into Dashboard, DSP API Server, notebook pods, Model Registry, and other components for Kubernetes RBAC enforcement.
- **Argo Workflows**: Pipeline execution engine used by Data Science Pipelines for ML workflow orchestration.

## Deployment Architecture

### Deployment Topology

The RHOAI platform deploys across multiple namespaces with clear separation of concerns:

**Operator namespace** (`redhat-ods-operator`): Contains the rhods-operator Deployment (3 replicas with leader election) that manages all other components. The operator watches DSCInitialization and DataScienceCluster singleton CRs and reconciles component controllers.

**Platform namespace** (`redhat-ods-applications`): The primary application namespace where all component operators and controllers are deployed — KServe controllers, DSP Operator, Notebook controllers, TrustyAI Operator, Training Operator, Trainer, KubeRay, Feast Operator, LlamaStack Operator, MLflow Operator, Spark Operator, Model Controller, WVA, Dashboard, and Model Registry Operator.

**Ingress namespace** (`openshift-ingress`): Houses the Gateway API infrastructure — the data-science-gateway Gateway CR with Envoy data plane, kube-auth-proxy deployment, EnvoyFilter CRs for ext_authz, dashboard redirect nginx, and associated Secrets/ConfigMaps.

**MaaS namespace** (`models-as-a-service`): Dedicated namespace for Models-as-a-Service resources — Tenant, MaaSModelRef, MaaSAuthPolicy, MaaSSubscription CRs and the MaaS controller.

**Policy namespace** (`kuadrant-system`): Kuadrant/Authorino policy enforcement for MaaS model endpoints.

**User namespaces**: Per-project workloads including Notebook StatefulSets, InferenceService Deployments, DSPA pipeline instances, RayCluster pods, TrainJob/PyTorchJob workers, and SparkApplication executors.

#### Manifest Consumption Pipeline

Component manifests flow from individual component repositories through a centralized collection process into the platform operator:

1. **Component repos**: Each component maintains kustomize manifests with base + overlay structure and params.env for image references.
2. **get_all_manifests.sh**: The rhods-operator build process runs this script to fetch component manifests from each repo's release branch, applying any required patches.
3. **prefetched-manifests/**: Fetched manifests are stored in the operator container image at `/opt/manifests/{component}/`.
4. **Operator reconciliation**: During runtime, component controllers read manifests from `/opt/manifests/`, apply kustomize transformations (image substitution from RELATED_IMAGE env vars, label injection, namespace prefixing), and deploy via Server-Side Apply.
5. **Cluster deployment**: Resources are created/updated in the target namespaces with field ownership tracked by the operator's service account.

### Disconnected Support

RHOAI supports air-gapped/disconnected deployment through several mechanisms:

- **RELATED_IMAGE pattern**: All component operators use `RELATED_IMAGE_*` environment variables to reference container images, allowing OLM and oc-mirror to discover and mirror all required images to a disconnected registry.
- **OLM relatedImages**: The rhods-operator ClusterServiceVersion (CSV) declares all component images in the `relatedImages` section, enabling automated catalog mirroring with `oc-mirror`.
- **Digest pinning**: Container image references in production builds use SHA256 digests rather than tags, ensuring reproducible deployments in disconnected environments.
- **params.env image substitution**: Component manifests use params.env files with image references that are substituted at deployment time, allowing registry overrides for mirrored images.
- **odh-trusted-ca-bundle**: A platform-wide ConfigMap aggregates custom CA certificates for environments with custom TLS inspection proxies.
- **Custom CA bundle injection**: Multiple components (DSP, Notebooks, Model Registry, TrustyAI) support mounting custom CA bundles for external TLS connections through corporate proxies.

## Platform Architectural Analysis

The RHOAI platform architecture reveals several cross-cutting patterns and tensions that emerge from analyzing all 51 components together.

**Authentication Architecture Fragmentation**: The platform employs three distinct authentication stacks that coexist but don't share a unified identity model. The core platform uses Gateway-level OAuth2/OIDC (kube-auth-proxy + Envoy ext_authz) for browser-based access. Individual components add kube-rbac-proxy sidecars for Kubernetes SAR-based authorization. MaaS introduces a third layer with Kuadrant/Authorino for API key-based and subscription-based access. This creates a situation where a single request to a MaaS model endpoint traverses three auth checks (Gateway OAuth → Authorino policy → optional component RBAC), while a direct InferenceService request might bypass Gateway auth entirely. The platform would benefit from a unified auth framework, but the current layered approach reflects the evolutionary addition of capabilities.

**FIPS Compliance Inconsistency**: While the majority of Go components correctly implement FIPS-compliant builds (CGO_ENABLED=1, GOEXPERIMENT=strictfipsruntime), seven components have documented FIPS gaps. The most critical is the fms-guardrails-orchestrator (Rust/Axum), which uses rustls and the ring crate — both of which use non-FIPS-certified cryptographic implementations. This means AI safety guardrails, a security-critical function, cannot be FIPS-certified. The KServe localmodel-controller uses CGO_ENABLED=0, meaning it statically links Go's stdlib crypto rather than system OpenSSL. The spark-operator compounds its CGO_ENABLED=0 issue by also importing crypto/md5. These gaps represent a systemic challenge: the platform declares `fips-compliant: "false"` in its OLM annotation, but the expectation is that all components should be FIPS-ready for government customers.

**Gateway API Migration In Progress**: RHOAI 3.x introduces Gateway API as the primary ingress mechanism, replacing the 2.x Route-based model. However, the migration is incomplete: the rhods-operator still creates legacy OpenShift Routes for backward compatibility (dashboard redirect), KServe InferenceServices can use Istio VirtualServices (Knative mode), OpenShift Routes (raw mode), or Gateway API HTTPRoutes (standard mode), and some components (EvalHub, TrustyAI Service) still expose Routes directly. The platform is in a transitional state where three ingress mechanisms coexist, increasing operational complexity.

**Operator-of-Operators Pattern**: The rhods-operator acts as a meta-operator, deploying and managing 16+ component operators. This creates a deep control hierarchy: rhods-operator → component operator (e.g., TrustyAI Operator) → component CRD (e.g., GuardrailsOrchestrator) → workload resources. While this provides centralized lifecycle management, it introduces challenges: upgrade ordering dependencies (platform operator must be upgraded before component operators), status aggregation complexity (DSC status must reflect the health of 16+ independent reconciliation loops), and manifest version coupling (all component manifests are baked into the operator image at build time via get_all_manifests.sh).

**Module Federation as Architecture Pattern**: The odh-dashboard's use of Webpack Module Federation with Go BFF sidecars is architecturally significant — it enables 7 independent feature teams to develop, build, and deploy UI capabilities as separate container images while maintaining a unified user experience. Each module (Gen AI Studio, MaaS, Model Registry, MLflow, EvalHub, AutoML, AutoRAG) is a complete micro-frontend with its own Go backend for domain-specific API calls. This pattern results in a single pod running 8 containers (host + 7 modules), which pushes pod resource limits and requires careful sidecar lifecycle management, but provides excellent team autonomy and independent release velocity.

**Disaggregated LLM Inference Stack**: The llm-d stack (llm-d-inference-scheduler, llm-d-kv-cache, KServe LLMInferenceService) implements a sophisticated disaggregated inference architecture that separates prefill and decode phases across different pod pools, with KV cache transfer between them. The Endpoint Picker (EPP) implements 40+ scheduling plugins for request routing based on queue depth, KV cache locality, session affinity, and load balancing. This is architecturally the most complex runtime component in the platform, requiring tight coordination between the EPP, PD Sidecar, KV Cache, and vLLM model servers.

**Build Hermeticity Gaps**: While most components use Konflux CI/CD pipelines, several lack full hermetic build support. Components without rpms.lock.yaml files cannot guarantee reproducible OS-level dependencies. The fms-guardrails-orchestrator and vllm-orchestrator-gateway (Rust) explicitly set `hermetic: false` in their Tekton PipelineRuns. Python-based components (vllm-cpu, vllm-gaudi, guardrails-detectors, notebooks, lm-evaluation-harness) face particular challenges with pip dependency resolution in hermetic builds, though some mitigate this with Hermeto/cachi2 prefetch integration.

**Database Proliferation**: The platform uses at least four different database technologies across components: MySQL/MariaDB (DSP metadata, Model Registry), PostgreSQL (MaaS API keys, Model Catalog), SQLite (embedded in various components), and the legacy ml-metadata C++ gRPC service (being replaced by Go-native Model Registry). This creates operational overhead for backup, monitoring, and high availability configuration. The Model Registry's migration from ml-metadata to a Go REST API with embedded database support is a positive consolidation effort.
