## Purpose

**Short**: A lightweight Rust HTTP service that detects PII and custom patterns in text using regular expressions, serving as a detection backend for the FMS Guardrails Orchestrator.

**Detailed**: The guardrails-regex-detector is a stateless HTTP microservice built in Rust using the axum web framework. It provides a single detection endpoint that accepts text content and a list of regex patterns (either built-in named detectors or arbitrary custom regular expressions), returning match locations and classification metadata for each detection.

The service ships with three built-in PII detectors — email address, Social Security Number (SSN), and credit card number — each implemented as hardcoded regex patterns with predefined detection types. Clients can also supply arbitrary regex strings, which are compiled and matched at request time. The service is designed to be called by the FMS Guardrails Orchestrator as part of a guardrails pipeline that screens LLM inputs and outputs for sensitive content.

The service is minimal by design: no persistent state, no external dependencies, no TLS termination, and no authentication. It relies on the deploying infrastructure (the Guardrails Orchestrator and platform networking) for security and traffic management. It listens on a configurable HTTP port (default 8080) and binds to all interfaces by default.

## Architectural Analysis

The guardrails-regex-detector is intentionally minimal — a single-purpose microservice with approximately 200 lines of Rust code across two source files. The architectural decision to use Rust (rather than Python, which is dominant in the ML/AI stack) reflects a focus on low-latency regex evaluation. The `regex` crate provides compiled DFA/NFA-based matching that is significantly faster than Python's `re` module for high-throughput scanning.

The service has zero external runtime dependencies — no database, no cache, no downstream API calls. All regex patterns (both built-in and custom) are compiled per-request in `handle_text_contents` (`src/detectors.rs:96-137`). This means there is no regex caching between requests; the built-in detectors (`email_address_detector`, `ssn_detector`, `credit_card_detector`) re-compile their hardcoded patterns on every invocation. For high-throughput scenarios, this is a potential optimization opportunity — `lazy_static` or `once_cell` (both already transitive dependencies via `tracing`) could cache compiled `Regex` instances for the built-in detectors.

The detection response format wraps results in a double array (`Json(json!([detections]))` at `src/detectors.rs:136`), which produces `[[{...}, {...}]]` rather than the flat `[{...}, {...}]` one might expect. This appears to be an intentional API contract matching what the Guardrails Orchestrator expects — the outer array likely maps to a per-content-item response structure, though in the current implementation all detections across all content items are flattened into a single inner array.

From a security perspective, the service accepts arbitrary regex patterns from clients (`src/detectors.rs:121-131`). There is no regex complexity validation or timeout mechanism — a maliciously crafted regex (e.g., catastrophic backtracking patterns) could cause CPU exhaustion. However, Rust's `regex` crate uses a Thompson NFA implementation that guarantees linear-time matching, which mitigates the classic ReDoS vulnerability. Custom regex compilation failures are handled gracefully with a warning log rather than a server error (`src/detectors.rs:130`).

The Dockerfile reveals a non-Konflux build pipeline: a multi-stage build using the upstream `rust:1.84.0` image for compilation, with the final image based on UBI 9 minimal. The installation of `compat-openssl11` (`Dockerfile:42`) suggests the image may need OpenSSL 1.1 compatibility for other components in the pod or for check-payload validation, even though the regex-detector binary itself does not use OpenSSL. The absence of a `USER` directive means the container runs as root by default — a security concern noted in `component-architecture.json`.
