## Purpose

**Short**: A lightweight HTTP server that detects PII and custom patterns in text using regex matching, serving as a detector plugin for the FMS Guardrails Orchestrator.

**Detailed**: The Guardrails Regex Detector is a Rust-based microservice that provides text content scanning via regular expression matching. It exposes a single POST endpoint (`/api/v1/text/contents`) that accepts text content and regex patterns, returning detection results with character offsets, matched text, detection type, and confidence scores.

The service ships with three built-in PII detectors — email address, Social Security Number (SSN), and credit card number — each implemented as pre-compiled regex patterns. In addition, callers can supply arbitrary custom regex patterns via the request payload. The detector is designed to integrate with the [FMS Guardrails Orchestrator](https://github.com/foundation-model-stack/fms-guardrails-orchestrator), which routes text through multiple detector backends to enforce content safety policies on LLM inputs and outputs.

The service is built on the Axum web framework with Tokio async runtime and deployed as a single stateless container on Red Hat UBI 9 minimal. It has no persistence, no authentication, and no TLS — it is expected to run behind the orchestrator within a trusted network boundary.

## Architectural Analysis

The Guardrails Regex Detector is deliberately minimal — a single-purpose microservice with no state, no authentication, no TLS, and no external dependencies beyond the Rust standard library and a few well-known crates. This simplicity is a strength: the attack surface is extremely small, the failure modes are limited, and the service can be horizontally scaled trivially as a stateless pod.

The service implements a plugin pattern for the FMS Guardrails Orchestrator. The orchestrator calls the detector's `/api/v1/text/contents` endpoint with text content and a list of regex identifiers. The detector supports two kinds of patterns: **built-in named patterns** (`email`, `ssn`, `credit-card`) which use hardcoded regex strings with curated detection types, and **custom patterns** where the caller supplies an arbitrary regex string that is compiled at request time. This dual-mode design allows the orchestrator to use well-tested PII patterns while also supporting ad-hoc content policies.

A notable architectural observation is that custom regex patterns are compiled on every request via `Regex::new()`. For high-throughput scenarios, this could become a performance bottleneck — pre-compiling and caching frequently-used custom patterns would improve latency. The built-in patterns also re-compile on each invocation rather than using `lazy_static` or `once_cell`, though the Rust `regex` crate's internal optimization mitigates this somewhat.

The security model relies entirely on network-level isolation. There is no authentication, no TLS, and no input validation beyond checking that the regex list is non-empty. The service trusts that only the orchestrator can reach it. In a Kubernetes deployment, this assumption would be enforced through NetworkPolicy or service mesh configuration. The absence of TLS means that text content (potentially containing PII) traverses the network in plaintext — acceptable within a pod network with mTLS (e.g., Istio service mesh) but a risk in flat network topologies.

The Dockerfile uses a multi-stage build with a Debian-based Rust builder (`rust:1.84.0`) and a UBI 9 minimal runtime image. The `compat-openssl11` package is installed for runtime compatibility. Notably, there is no Konflux-specific Dockerfile (`Dockerfile.konflux`) on this branch, which is unusual for a RHOAI component. This may indicate that Konflux integration is configured at the pipeline level or that a Konflux Dockerfile exists on a different branch.
