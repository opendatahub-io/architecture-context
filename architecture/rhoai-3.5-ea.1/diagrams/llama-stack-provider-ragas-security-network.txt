Secrets:
- (user-defined S3 credentials) (Opaque) - AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY,
  AWS_DEFAULT_REGION — Manual provisioning, no auto-rotation
- kubeflow-pipelines-token (Opaque) - KFP API auth token — Manual or kubeconfig extraction,
  no auto-rotation

  VLLM_API_TOKEN — Manual provisioning, no auto-rotation

Authentication & Authorization:
- Kubeflow Pipelines API:
  - Methods: POST (create_run), GET (get_run), POST (terminate_run)
  - Auth: Bearer Token (OpenShift OAuth)
  - Enforcement: KFP server
  - Token source: kubeconfig or KUBEFLOW_PIPELINES_TOKEN env var
- S3 Object Storage:
  - Methods: GET, PUT
  - Auth: AWS IAM credentials (AccessKey/SecretKey)
  - Enforcement: S3 service
  - Injected via Kubernetes Secret into KFP pipeline components
- Llama Stack Server:
  - Methods: POST (eval, inference, datasetio)
  - Auth: None (local API)
  - Enforcement: Llama Stack server
  - Provider runs in-process (inline) or as KFP component (remote)

FIPS Compliance:
- Base image: UBI 9 (registry.redhat.io/ubi9/python-312) — OpenSSL available
- Crypto library: cryptography==46.0.2 (FIPS-capable via OpenSSL backend on UBI)
- No custom TLS configuration — delegates to Llama Stack server and upstream libraries
- No non-FIPS crypto risks detected (no MD5, RC4, DES, or custom crypto)

Build Hermeticity:
- Python deps: Pinned with SHA256 hashes in requirements.txt (uv pip compile) + uv.lock
- OS packages: rpms.lock.yaml ABSENT — hermeticity gap
- Hermeto/cachi2 prefetch: NOT present — pip install fetches from PyPI at build time

Architectural Notes:
- In-memory job tracking: Both inline and remote providers use in-memory dicts for job state.
  Server restart loses all job references. KFP pipeline runs continue but provider cannot
  recover orphaned jobs.
- Base image resolution (remote mode): 3-tier fallback —
  (1) KUBEFLOW_BASE_IMAGE env var
  (2) trustyai-service-operator-config ConfigMap (redhat-ods-applications or opendatahub ns)
  (3) Default: quay.io/trustyai/llama-stack-provider-ragas:latest
