RBAC Summary:
- Workbench pods do NOT define their own RBAC resources
- RBAC is managed by the Notebook Controller:
  - Creates per-notebook ServiceAccounts
  - Creates per-notebook RoleBindings
  - Injects kube-rbac-proxy sidecar for authentication/authorization
- Workbench pods run as user 1001 (non-root)

Authentication Architecture:
- External access: OAuth/OIDC → Platform Gateway → kube-rbac-proxy (Bearer Token) → workbench
- kube-rbac-proxy: RBAC-based user authorization on port 8443/TCP
- Internal APIs (/api/kernels/): No auth (pod-internal only, consumed by culler via kube-rbac-proxy)
- code-server: Three-process stack (nginx → httpd → CGI) to fake Jupyter API compatibility

Secrets:
- DSPA connection secret (Opaque) - Pipeline endpoint credentials, mounted at /opt/app-root/runtimes/, provisioned by Notebook Controller, no auto-rotation
- CA certificate bundle (Opaque) - Custom CA certs, mounted at /etc/pki/tls/custom-certs/ca-bundle.crt, provisioned by Notebook Controller, no auto-rotation

FIPS Compliance:
- mongocli Go binary: CGO_ENABLED=1, -tags strictfipsruntime (FIPS-compliant Go crypto via OpenSSL)
- Python: Inherits system OpenSSL via UBI9 base image
- TLS: System OpenSSL configuration from UBI9/RHEL9
- Risk: Minimal/code-server images do NOT build mongocli (no FIPS Go binary); Python crypto depends on user code

Build Hermeticity:
- OS packages: rpms.lock.yaml (rpm-lockfile-prototype)
- Python: pylock.*.toml (PEP 751, via uv pip compile)
- Go: go.sum (go mod)
- Artifacts: artifacts.lock.yaml (Hermeto)
- Konflux Dockerfiles use --mount=/cachi2/output/deps/ for all dependency types
