RBAC Summary:
  Roles are managed by the deploying operator (rhods-operator), not bundled.
  - (operator-managed) ClusterRole:
    - API: authentication.k8s.io | Resources: tokenreviews | Verbs: create
    - API: authorization.k8s.io | Resources: subjectaccessreviews | Verbs: create
    - API: user.openshift.io | Resources: users | Verbs: get
    - API: oauth.openshift.io | Resources: oauthaccesstokens | Verbs: get, delete

Authentication Chain (Auth Mode, priority order):
  1. Kubernetes service account tokens (TokenReview API)
  2. OpenShift OAuth tokens (sha256~ format, User API validation)
  3. JWT bearer tokens (OIDC issuer verification)
  4. Basic auth (htpasswd file, bcrypt hashes)
  5. Stored session cookies (AES-encrypted, or Redis ticket)

Authorization Chain (RBAC Mode, evaluated in order):
  1. Hardcoded authorizer: prometheus-k8s SA → GET /metrics (always allowed)
  2. Static authorizer: config file pattern matching (user/verb/namespace/resource)
  3. SubjectAccessReview: Kubernetes API server RBAC delegation

Secrets:
  - cookie-secret (Opaque) - AES key for session cookie encryption, operator/manual, no auto-rotate
  - tls-cert/tls-key (kubernetes.io/tls) - HTTPS listener certs, cert-manager/operator, fsnotify hot-reload
  - client-secret (Opaque) - OAuth2 client secret for OIDC/OpenShift, manual/operator, no auto-rotate
  - htpasswd-file (Opaque) - Basic auth bcrypt password file, manual, fsnotify hot-reload

FIPS Compliance:
  - Build: GOEXPERIMENT=strictfipsruntime, CGO_ENABLED=1, UBI 9 base
  - Crypto: AES-CFB/GCM (session), HMAC-SHA256 (signing), TLS 1.2-1.3
  - No MD5/RC4/DES detected in codebase
  - InsecureSkipVerify available but requires explicit opt-in
