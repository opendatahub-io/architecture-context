RBAC Summary:
- MLServer does NOT define any RBAC resources
- Runs as a workload inside KServe-managed pods
- Uses KServe-defined service accounts
- All Kubernetes RBAC is managed by KServe operator

Authentication & Authorization:
- Application-level: None (all V2 endpoints are unauthenticated)
- Infrastructure-level: Auth delegated to KServe sidecar / kube-rbac-proxy / Istio
- Runtime Security: Trusted runtime allowlist in PRODUCTION mode blocks unauthorized runtimes
- Runtime Security: DEVELOPMENT mode allows all runtimes (not for production use)
- Endpoint /v2/runtimes exposes current security mode and active allowlist

Secrets:
- MLServer does NOT manage secrets directly
- Model storage credentials are injected by KServe
- TLS certificates are managed by KServe/Istio infrastructure

FIPS Compliance:
- Python application — inherits FIPS from UBI base image OpenSSL
- No TLS termination in application (handled upstream by KServe/Istio)
- No direct crypto library usage (uses Python stdlib ssl backed by system OpenSSL)
- No non-FIPS crypto risks detected (no MD5, RC4, or custom RNG)
- Batch processing CLI has --use-ssl flag but does not configure custom cipher suites

Build Hermeticity:
- Konflux build: hermetic=true
- Dependency prefetch: Hermeto pip prefetch with SHA-256 pinned requirements
- Source: Red Hat internal PyPI mirror (packages.redhat.com)
- Lock files: 11 Poetry lock files + requirements-cpu.txt (hash-pinned)
- Base image: quay.io/aipcc/base-images/cpu:3.5.0-ea.1 (UBI-based)
- Output: quay.io/rhoai/odh-mlserver-rhel9
