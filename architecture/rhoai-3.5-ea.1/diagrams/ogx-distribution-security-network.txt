RBAC Summary:
- No Kubernetes RBAC defined. This is a container image without direct
  Kubernetes API interaction. RBAC is managed by the platform operator
  deploying this image.

Authentication & Authorization:
- Mechanism: OAuth2/OIDC JWT validation (optional, via AUTH_ISSUER env)
- When AUTH_ISSUER is unset: authentication is COMPLETELY DISABLED
- Token validation: JWKS-based (AUTH_JWKS_URI), key rotation period configurable
- TLS verification for auth endpoints: configurable (AUTH_VERIFY_TLS)
- Access policy:
  - Unowned resources: publicly readable
  - Authenticated users: can create resources
  - Resource owners: can read/update/delete their own resources

Secrets (Environment Variables):
- VLLM_API_TOKEN (Bearer) - vLLM inference auth, manual provisioning, no rotation
- VLLM_EMBEDDING_API_TOKEN (Bearer) - vLLM embedding auth, manual, no rotation
- AWS_BEARER_TOKEN_BEDROCK (Bearer) - AWS Bedrock, 12h TTL auto-rotate
- WATSONX_API_KEY (API key) - IBM watsonx, manual, no rotation
- AZURE_API_KEY (API key) - Azure OpenAI, manual, no rotation
- OPENAI_API_KEY (API key) - OpenAI, manual, no rotation
- POSTGRES_PASSWORD (password) - PostgreSQL auth, manual, no rotation
- MILVUS_TOKEN (token) - Milvus auth, manual, no rotation
- PGVECTOR_PASSWORD (password) - pgvector auth, manual, no rotation
- QDRANT_API_KEY (API key) - Qdrant auth, manual, no rotation
- AWS_ACCESS_KEY_ID (IAM) - S3 access, manual, no rotation
- AWS_SECRET_ACCESS_KEY (IAM) - S3 access, manual, no rotation

FIPS Compliance:
- Python application on UBI 9 with system OpenSSL
- TLS delegated to upstream libraries (httpx, asyncpg, boto3, google-genai)
- No explicit non-FIPS crypto detected in repository code
- FIPS compliance depends on base image OpenSSL configuration
- Milvus supports custom CA bundle and client certificates

Build Hermeticity:
- NO lock files present (no rpms.lock.yaml, uv.lock, poetry.lock)
- ~50 Python packages installed with mixed version pinning
- Konflux build uses RHOAI PyPI index (some versioning guarantees)
- Build is NOT fully hermetic

Security Observations:
- Server listens on plaintext HTTP (8321/TCP)
- TLS termination expected at platform ingress layer
- Sensitive credentials (API keys, passwords) in HTTP headers/bodies
  traverse internal network in plaintext between ingress and OGX pod
- PostgreSQL connection uses plaintext by default
- Authentication is entirely optional (disabled when AUTH_ISSUER unset)
