## Diagram Descriptions

### Component Dependency Graph
Shows how RHOAI's 57 components depend on each other. Key hubs:
- **rhods-operator** (red): Platform root, deploys 14+ components
- **KServe** (blue): Inference backbone with 8 reverse dependencies
- **odh-dashboard** (green): Primary UI with 17 outgoing connections
- **kube-rbac-proxy** (purple): Shared auth sidecar for 8 components
- **Kueue** (gray): Transversal quota management

Color scheme: red = operator root, blue = serving stack, green = platform services, orange = model management + evaluation, purple = new platform services, gray = external platform dependencies.

### Platform Network Topology
Complete network architecture showing 6 namespaces, Gateway API ingress, service mesh configuration, and cross-namespace management patterns.

**Two formats provided**:
- **Mermaid (.mmd)**: Visual diagram with color-coded trust zones (external, ingress, operator, platform, monitoring, mesh, user, egress)
- **ASCII (.txt)**: Precise text format with per-component ports, protocols, auth mechanisms, ServiceAccounts, and RBAC summary

Both formats cover: ingress (Dashboard, KServe, MaaS, Notebooks), operator namespace, platform namespace (14 controllers + dashboard pod), monitoring namespace, service mesh, user namespaces (11 workload types), and 5 egress destinations.

### Cross-Component Workflows
Five sequence diagrams showing end-to-end flows:
1. **Model Deployment via Dashboard**: User → Dashboard → K8s API → KServe → storage-initializer → model pod → Gateway
2. **LLM Inference with Guardrails**: Client → Gateway → ai-gateway-payload-processing → EPP → vLLM → Guardrails → NeMo
3. **Data Science Pipeline Execution**: User → DSP API → Argo → Driver → Launcher → S3/MLMD
4. **Notebook-to-Training Pipeline**: Notebook → training-hub → Trainer → JobSet → Kueue → distributed pods
5. **Model Evaluation and Safety Scanning**: Dashboard → EvalHub → eval jobs → model endpoint → MLflow → OCI registry

### Platform Security Overview
Visual representation of RHOAI's defense-in-depth security architecture:
- **7 authentication mechanisms**: kube-rbac-proxy, IntegratedOAuth, OIDC, mTLS, bearer forwarding, API keys (Authorino), ServiceAccount RBAC
- **8 kube-rbac-proxy deployments**: Dashboard, Model Registry, TrustyAI, OGX, EvalHub, Feast, NeMo, Notebooks
- **6 most privileged ClusterRoles**: rhods-operator, kserve-manager, odh-model-controller, odh-dashboard, dsp-operator, training-operator
- **TLS configuration**: Webhooks (9443), metrics (8443), gateway (443), mesh (mTLS), Ray (client certs)
- **Container security**: Non-root (148 Dockerfiles), UBI 9 (87), FIPS builds, capability drops, multi-stage (372)
- **Known FIPS gaps**: 3 identified (KServe localmodel, PodMonitor TLS, dashboard BFFs)

### Platform Observability
Maps all monitoring and observability endpoints:
- **50 Prometheus scrape targets**: 14 operator metrics endpoints (all via kube-rbac-proxy :8443) + user namespace workload metrics
- **76 health probes**: Liveness (:8081 healthz) and readiness (:8081 readyz) for all controller-runtime operators; HTTP health for services
- **WVA autoscaling metrics**: KV-cache utilization, queue depth, and saturation metrics for llm-d stack
- **Training metrics**: Non-Prometheus HTTP polling by Trainer controller for real-time training progress

### Platform Maturity Dashboard
High-level metrics:
- 57 components across 7+ languages (Go 31, Python 30, TypeScript 5, Rust 4, C++ 2, Java 1, C 1)
- 126 shipped container images, 121 CRDs, 166 services, 443 controller watches
- API maturity spectrum from v1 (GA) through v1alpha1 (experimental)
- Deployment readiness: HA (leader election), disconnected (RELATED_IMAGE), multi-arch (4 archs), Gateway API (full), xKS (cloudmanager)
- Security posture: FIPS enabled (with gaps), mTLS MIXED, kube-rbac-proxy coverage
