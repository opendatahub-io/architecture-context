RBAC Summary:
  N/A — ai4rag is a Python library, not a Kubernetes workload.
  No ClusterRoles, Roles, or RoleBindings are defined.
  RBAC requirements are the responsibility of the consuming component
  (e.g., the KFP Pipeline step's ServiceAccount).

Service Mesh Configuration:
  N/A — ai4rag does not deploy any Kubernetes services.
  Service mesh (mTLS, PeerAuthentication) applies to the hosting pod,
  not to the library itself.

Secrets:
  No secrets managed directly by ai4rag.
  - OGX API Key: Passed via OgxClient(api_key=...) constructor
  - OGX Base URL: Passed via OgxClient(base_url=...) constructor
  Credential management is the caller's responsibility.
  The consuming KFP component typically sources these from
  Kubernetes Secrets mounted as environment variables.

Authentication:
  - OGX Server API: API Key (Bearer Token) — configured by caller
  - ChromaDB: No auth (in-memory, same process)
  - unitxt: No auth (in-process Python library)

FIPS Compliance:
  - ai4rag performs no direct cryptographic operations
  - hashlib.sha256 used for document deduplication (FIPS-approved)
  - TLS handled by ogx-client → httpx → platform OpenSSL
  - FIPS compliance depends on the container image's OpenSSL build
  - No container image produced by ai4rag itself

  - uv.lock present (658KB) — all Python deps pinned
  - No RPM or artifact lock files (expected: library, not container)
