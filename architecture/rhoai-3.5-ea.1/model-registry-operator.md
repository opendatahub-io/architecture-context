## Purpose

**Short**: Manages the lifecycle of Model Registry and Model Catalog custom resources, provisioning all necessary Kubernetes resources for model metadata storage and discovery.

**Detailed**: The Model Registry Operator is a Kubernetes operator built with Kubebuilder (controller-runtime) that manages `ModelRegistry` custom resources in the `modelregistry.opendatahub.io` API group. For each `ModelRegistry` CR, the operator provisions a complete stack of Kubernetes resources including Deployments, Services, ServiceAccounts, Roles, RoleBindings, NetworkPolicies, and optionally OpenShift Routes and PostgreSQL database instances.

The operator supports two API versions (v1alpha1 and v1beta1, with v1beta1 as the storage version) and implements automatic migration from OAuth proxy to kube-rbac-proxy authentication. It detects cluster capabilities at startup (OpenShift vs plain Kubernetes, BYOIDC mode, Auth API availability) and conditionally creates platform-specific resources. The operator also includes a `ModelCatalogReconciler` that manages a singleton model catalog service for browsing and discovering models, with its own PostgreSQL database, kube-rbac-proxy authentication, and configurable catalog sources via ConfigMaps. A storage version migration system handles CRD version upgrades using either the Kubernetes StorageVersionMigration API or a manual fallback strategy.

## Architectural Analysis

The Model Registry Operator follows a classic Kubebuilder operator pattern with two independent reconcilers sharing a single manager process. The `ModelRegistryReconciler` handles per-instance lifecycle (one reconciler invocation per `ModelRegistry` CR), while the `ModelCatalogReconciler` manages a singleton catalog service using a fixed reconcile key (`model-catalog`) to deduplicate events from multiple watched resources. This split is architecturally significant: the model registry is multi-tenant (many CRs, many namespaces), while the catalog is a platform-wide singleton.

The operator's approach to authentication has undergone a notable migration from OAuth proxy to kube-rbac-proxy. The v1beta1 webhook (`MigrateOAuthProxyToKubeRBACProxy`) automatically converts existing OAuth proxy configurations to kube-rbac-proxy on create/update, and the controller performs the same migration during reconciliation for CRs updated outside the webhook. This dual-path migration ensures all registries converge to kube-rbac-proxy regardless of how they were created. The kube-rbac-proxy configuration uses Kubernetes `SubjectAccessReview` to authorize users based on their ability to `get` the registry's Service resource, effectively delegating authorization to standard Kubernetes RBAC.

The cluster capability detection system (`DetectClusterCapabilities`) is well-designed for multi-platform support. It handles three distinct modes: traditional OpenShift (all APIs available), BYOIDC-enabled OpenShift 4.20+ (route API present but user API absent), and plain Kubernetes. The BYOIDC detection is particularly careful — it validates that `user.openshift.io` failures are specifically "not found" errors (expected) rather than transient failures (unexpected), preventing false-positive BYOIDC detection. In BYOIDC mode, the operator skips OpenShift Group creation and relies on external IdP group names in RoleBindings.

The template-based resource generation (`TemplateApplier`) uses Go's `text/template` with embedded YAML templates to generate Kubernetes resources. This approach provides flexibility (templates can conditionally include kube-rbac-proxy sidecars, SSL volumes, database-specific configurations) but trades off compile-time type safety. The `banzaicloud/k8s-objectmatcher` library provides intelligent diff-based updates, avoiding unnecessary API writes when resources haven't changed.

The model catalog's architecture includes two init containers (`catalog-data-init` and `benchmark-data-init`) that copy data from separate images into shared `emptyDir` volumes. The catalog supports multiple catalog source ConfigMaps — a managed default source, a user-editable source, and dynamically discovered ConfigMaps labeled with `opendatahub.io/catalog-source: "true"`. This layered configuration approach allows platform defaults to be updated by the operator while preserving user customizations.
