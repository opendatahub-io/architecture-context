## Purpose

**Short**: Manages the lifecycle of Model Registry instances and an optional Model Catalog service on Kubernetes/OpenShift clusters.

**Detailed**: The Model Registry Operator is a Kubebuilder-based controller-runtime operator that reconciles `ModelRegistry` custom resources (API group `modelregistry.opendatahub.io`). For each `ModelRegistry` CR, the operator provisions a complete deployment stack: a REST API server (the upstream Kubeflow Model Registry), an optional auto-provisioned PostgreSQL or user-provided PostgreSQL/MySQL database, a `kube-rbac-proxy` authentication sidecar with OpenShift Route, RBAC resources, NetworkPolicies, and ServiceAccounts.

The operator also includes a `ModelCatalog` controller that manages an optional model catalog service with its own PostgreSQL database, kube-rbac-proxy authentication, and dynamic catalog source discovery via labeled ConfigMaps. The operator supports automatic migration paths: Istio → OAuth Proxy → kube-rbac-proxy for authentication, and v1alpha1 → v1beta1 for CRD storage version migration using either the Kubernetes StorageVersionMigration API or a manual re-read/re-write fallback strategy.

The operator detects cluster capabilities at startup (OpenShift vs. vanilla Kubernetes, BYOIDC mode, platform API availability) and conditionally creates platform-specific resources such as OpenShift Routes, User Groups, and ClusterRoleBindings. It integrates with the broader RHOAI/ODH platform by watching `Auth` CRs for admin group configuration and using `DefaultModelRegistry` CRs as owner references for catalog resources.

## Architectural Analysis

The Model Registry Operator follows a well-structured Kubebuilder pattern with two independent controllers — `ModelRegistryReconciler` and `ModelCatalogReconciler` — that share a common template engine and cluster capability detection system. The operator's most distinctive architectural feature is its multi-layered authentication migration path: it transparently upgrades Istio-based authentication (v1alpha1) through OAuth Proxy to kube-rbac-proxy (v1beta1) via webhook defaulting and CR conversion, ensuring zero-downtime migration for existing deployments.

The template-driven resource generation system (Go `text/template` with 35+ YAML templates in `internal/controller/config/templates/`) provides significant flexibility but creates a gap between static manifest analysis and actual runtime behavior. Resources like Deployments, Services, Routes, NetworkPolicies, and RBAC bindings are rendered per-instance from templates with conditional logic (e.g., kube-rbac-proxy sidecar injection only when `spec.kubeRBACProxy` is set, PostgreSQL auto-provisioning only when `spec.postgres.generateDeployment` is true). This means the operator creates far more Kubernetes resources dynamically than what appears in its static kustomize manifests.

The cluster capability detection system (`kubebuilder.go`) is notable for its handling of OpenShift BYOIDC (Bring Your Own Identity Provider) mode, where the User API group is intentionally absent. The operator gracefully degrades: it still creates Routes and RoleBindings but skips Group creation, preventing reconciliation failures in BYOIDC environments. This detection uses API group discovery and classifies errors to distinguish between expected BYOIDC scenarios and transient failures.

The Model Catalog controller introduces an interesting deduplication pattern: all watched resources (Deployments, ConfigMaps, Secrets, etc.) map to a single fixed reconcile request (`model-catalog/{targetNamespace}`), preventing reconcile storms when multiple resources change simultaneously. It also implements dynamic catalog source discovery by watching ConfigMaps with the `opendatahub.io/catalog-source` label, validating their names as DNS labels, and mounting them as volumes into the catalog deployment.

A notable supply chain consideration: the `Dockerfile.konflux` runs `go mod download` directly without Hermeto/cachi2 prefetch, meaning Go dependency resolution depends on network access at build time. While `go.sum` provides integrity verification, this is less hermetic than a fully prefetched build. The FIPS compliance story is solid at the build-time layer (CGO_ENABLED=1, strictfipsruntime, UBI 9 base), and no non-FIPS crypto usage was found in production code paths.
