Network Policies & Security Configuration
═════════════════════════════════════════════════════════════════════════════════════

Service Mesh: Not required (training jobs communicate via ClusterIP services)

Network Isolation:
  - Training job pods communicate internally via ClusterIP services
  - No ingress required (jobs are batch workloads, not user-facing services)
  - Egress required for:
    - S3 storage (443/TCP HTTPS)
    - Container registries (443/TCP HTTPS)
    - PyPI / pip index (443/TCP HTTPS)
    - Model Registry API (8080/TCP HTTP)
    - MLFlow Server (5000/TCP HTTP)

Plaintext Communication (within cluster):
  - PyTorchJob: Master ←→ Workers (23456/TCP)
  - TFJob: Workers ←→ Parameter Servers (2222/TCP gRPC)
  - MPIJob: Workers ←→ Workers (dynamic TCP/UDP)
  - Justification: High-performance distributed training requires low-latency
    communication. Pod-to-pod network is trusted within Kubernetes.

Encrypted Communication:
  - User → Kubernetes API (6443/TCP HTTPS TLS 1.2+)
  - Kubernetes API → Webhook Server (9443/TCP HTTPS TLS 1.2+)
  - Operator → Kubernetes API (6443/TCP HTTPS TLS 1.2+)
  - Training pods → S3 storage (443/TCP HTTPS TLS 1.2+)
  - Training pods → Container registries (443/TCP HTTPS TLS 1.2+)
  - MPIJob launcher → Workers (22/TCP SSH)

Security Recommendations
═════════════════════════════════════════════════════════════════════════════════════

1. Credential Management:
   - Use AWS IAM Roles for Service Accounts (IRSA) instead of static credentials
   - Rotate SSH keys regularly for MPIJob workloads
   - Store credentials in external secret management systems (e.g., Vault)

2. Network Security:
   - Apply NetworkPolicies to restrict egress to only required destinations
   - Use private VPC endpoints for S3 access to avoid public internet egress
   - Consider service mesh (Istio) for mTLS pod-to-pod communication if compliance requires

3. RBAC Hardening:
   - Limit training-operator ClusterRole to specific namespaces if possible
   - Use namespace-scoped Roles for user access to training job CRs
   - Apply least-privilege principle for training job ServiceAccounts

4. Monitoring & Audit:
   - Enable Kubernetes audit logging for training job creation/deletion
   - Monitor S3 access patterns for anomalous data exfiltration
   - Alert on webhook certificate expiration (cert-manager auto-rotate)

5. Runtime Security:
   - Use Pod Security Standards (restricted profile) for training job pods
   - Scan training container images for vulnerabilities
   - Apply resource limits to prevent resource exhaustion attacks
